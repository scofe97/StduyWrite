# How to Read Code in the AI Era (6 Techniques)

- **원문**: How to Read Code in the AI Era (6 Techniques) — 영상 (영상 링크 추후 보강)
- **읽은 날**: 2026-06-15
- **한 줄 요지**: 코드 쓰기는 매달 싸지고, 낯선 코드를 빠르게 읽어 모델이 놓친 엣지 케이스를 잡는 능력이 희소해졌습니다.

---

## 왜 이 글을 모아두는가

Claude나 Cursor가 코드 초안을 대신 쓰는 시대에 "그럼 엔지니어는 무엇을 잘해야 하는가"를 이 영상은 읽기 역량으로 답합니다. 코드를 쓰는 비용은 매달 떨어지지만, 코드를 이해하고 모델이 빠뜨린 엣지 케이스를 잡아내는 일은 여전히 어렵다는 진단입니다. 문제는 대부분의 엔지니어가 코드를 쓰도록 훈련됐지 낯선 코드를 읽도록 훈련되지 않았다는 데 있습니다. 그래서 코드를 받으면 책처럼 위에서 아래로 훑다가, 몇 분이 지나도 무슨 일이 벌어지는지 모르는 상태에 빠집니다. 제가 운영하는 토큰 위생·코드리뷰 루틴과도 맞닿아 있어서, 낯선 코드를 빠르게 파악하는 절차의 원전으로 보관합니다.

## 핵심 전제

이 영상이 6가지 기법을 떠받치는 한 문장은 이렇습니다. 코드는 위에서 아래로 읽는 이야기가 아니라, 누가 무엇을 호출하는지의 그래프입니다. 그래서 파일 맨 위가 아니라 바깥 세계가 코드로 들어오는 지점에서 시작해, 호출을 따라 바깥으로 뻗어 나가야 합니다. 영상은 로그인 엔드포인트라는 흔한 예시로 이 관점을 보여줍니다. 파일 3개, 50~60줄 안쪽의 코드에 6가지 기법을 적용하는데, 어떤 실제 코드베이스에 들어가도 통하는 절차입니다.

## 코드를 읽는 6가지 기법

| # | 기법 | 핵심 |
|---|------|------|
| ① | 엔트리 포인트에서 시작 | 파일 맨 위가 아니라 요청을 처리하는 줄에서 출발. 예시는 `routes/auth.js`의 `app.post('/login')` |
| ② | 함수보다 테스트를 먼저 | happy path 테스트가 계약을 알려줌. 자격 증명을 넣으면 토큰이 나온다는 형상 |
| ③ | 함수가 아니라 데이터를 따라간다 | 중요한 변수(`user`)의 여정을 추적. 탄생→비교→토큰 적재→반환 |
| ④ | 첫 패스에서 불필요한 것 건너뛰기 | rate limiter·audit log는 흐름을 바꾸지 않으면 일단 통과 |
| ⑤ | happy path 후 실패 경로 하나만 | 보안 질문 2개로 코드가 틀리는 지점을 확인 |
| ⑥ | 추적 결과를 한 문장으로 메모 | 한 줄로 압축 못 하면 이해한 게 아니라 본 것일 뿐 |

**① 엔트리 포인트에서 시작합니다.** user 모델이나 미들웨어 폴더를 습관적으로 먼저 열지 않습니다. 들어오는 요청을 처리하는 코드 줄을 찾는데, 예시에서는 `routes/auth.js` 안의 `app.post('/login')`입니다. 바깥 세계가 이 코드로 진입하는 자리라서, 여기서부터 호출을 바깥으로 따라가면 됩니다.

**② 함수보다 테스트를 먼저 읽습니다.** 로그인 핸들러를 열기 전에 `auth.test.js`의 happy path 테스트부터 봅니다. 이메일과 비밀번호를 POST하고, 200 상태 코드와 응답 본문의 토큰을 기대하는 다섯 줄짜리 테스트가 계약을 알려줍니다. 자격 증명을 넣으면 토큰이 나온다는 형상입니다. 테스트가 전부를 말해주지는 않지만, 소스를 읽기 전 출발점이 됩니다.

**③ 함수가 아니라 데이터를 따라갑니다.** 핸들러를 열면 중요한 변수 하나, 즉 `user`의 여정을 지켜봅니다. 요청의 이메일로 사용자를 조회하면서 태어나고, 제출된 비밀번호가 `user.password_hash`와 `bcrypt.compare`로 비교되며, `user.id`가 만료 시각과 함께 서명된 JWT 토큰에 담기고, 그 토큰이 응답으로 반환됩니다. 이 핵심 흐름은 30초면 파악됩니다.

**④ 첫 패스에서는 필요 없는 것을 건너뜁니다.** rate limiter, audit log 호출, 부가 validator는 일단 지나칩니다. 요청을 바꾸거나 흐름을 막거나, 지금 쫓는 버그를 설명하지 않는 한 건너뛰어도 됩니다. 다만 미들웨어와 validator는 핸들러가 실행될지 자체를 결정하는 경우가 많아 영원히 무시할 수는 없고, 코드의 모양을 그리는 동안만 미뤄둡니다. 대부분의 엔지니어가 바로 여기서 무너집니다. 헬퍼를 만나는 순간 그 함수를 열어야 한다는 의무감에 빠져, 필요 없는 시간을 너무 많이 씁니다.

**⑤ happy path가 분명해지면 실패 경로를 정확히 하나만 읽습니다.** 로그인이라면 두 가지 질문을 던집니다. 첫째, 이메일이 존재할 때와 존재하지 않을 때 에러 메시지가 다른가입니다. 메시지가 다르면 공격자가 어느 계정이 실재하는지 쉽게 가려낼 수 있습니다. 둘째, 비밀번호가 틀린 응답이 사용자를 못 찾은 응답보다 오래 걸리는가입니다. 응답 시간이 다르면 메시지가 똑같아도 공격자가 그 간극을 재서 어떤 이메일이 등록됐는지 알아냅니다.

**⑥ 추적 결과를 자기 메모에 한 문장으로 적습니다.** 아무도 하지 않는 마지막 단계입니다. 로그인이라면 "이메일로 사용자를 찾고, 저장된 해시에 비밀번호를 대조하고, 사용자 ID로 JWT를 서명해 반환한다"가 그 한 줄입니다. 방금 읽은 것을 한 줄로 압축하지 못하면, 코드를 이해한 게 아니라 그냥 쳐다본 것입니다.

## 인상 깊은 점

happy path와 failure path의 대비가 또렷합니다. happy path는 코드가 하는 일을 보여주고, failure path는 코드가 틀리는 일을 보여줍니다. 그래서 정상 흐름만 읽고 끝내면 보안 결함처럼 코드가 잘못하는 지점을 영영 못 봅니다. 마지막 기법인 한 문장 압축도 같은 맥락에서 강력합니다. 압축할 수 없다는 것은 이해의 부재를 드러내는 신호라서, 읽기의 종료 조건을 스스로 검증하는 장치가 됩니다.

## 내 시스템과의 연결점

이 영상의 6번(한 문장 압축)은 제가 학습 노트마다 다는 "한 줄 요지"와 journal 루틴에 그대로 얹힙니다. 추적을 끝낸 뒤 한 줄로 못 줄이면 다시 읽는다는 종료 조건으로 쓸 수 있습니다. 2번(테스트를 먼저 읽기)은 testing-style 룰의 Given/When/Then 계약을 먼저 확인하는 습관과 호응하는데, 테스트가 곧 코드의 계약이라는 같은 전제를 공유합니다. 4번(첫 패스에서 건너뛰기)은 토큰 위생 규칙의 "전체 Read 금지, 필요한 라인만 읽기"와 같은 결입니다. 낯선 코드를 처음 훑을 때 헬퍼마다 파고들지 않고 모양부터 그린다는 점에서, 사람이 코드를 읽는 절차와 에이전트가 컨텍스트를 아끼는 절차가 맞물립니다.
