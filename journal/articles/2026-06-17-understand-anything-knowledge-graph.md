# Understand-Anything — 코드베이스를 지식 그래프로 읽는 도구

- **원문**: https://github.com/Egonex-AI/Understand-Anything (Egonex, MIT, v2.8.0)
- **읽은 날**: 2026-06-17
- **한 줄 요지**: 코드 구조의 *사실*은 파서(Tree-sitter)가 결정론적으로 뽑고, *해석*은 LLM이 맡는 분리가 핵심입니다. "그래프가 과시하지 않고 가르쳐야 한다"는 철학으로 낯선 코드베이스를 인터랙티브 지식 그래프로 바꿔줍니다.

---

## 왜 이 글을 모아두는가

낯선 코드베이스 온보딩을 다룬 도구인데, 바로 앞 저널의 ["AI 시대에 코드 읽는 6가지 기법"](2026-06-15-how-to-read-code-in-the-ai-era.md)과 같은 문제를 정반대 방향에서 풉니다. 그 영상은 *사람이* 호출 그래프를 머릿속에 그리는 절차였고, 이 도구는 그 그래프를 *기계가* 그려서 보여줍니다. 둘을 나란히 두면 "코드는 위→아래 이야기가 아니라 누가 무엇을 호출하는지의 그래프"라는 같은 전제를 사람·도구 양쪽에서 확인하게 됩니다.

보관하는 진짜 이유는 따로 있습니다. 이 도구의 설계 한 줄 — **"파서가 사실, LLM이 해석"** — 이 제가 운영하는 코드 분석 루틴(`explore`·`ddd-expert`)의 약점을 정확히 찌르기 때문입니다. 제 에이전트들은 grep + LLM 해석에만 의존해서 구조 자체를 환각할 여지가 있는데, 이 도구는 그 환각을 결정론적 골격으로 잡습니다. 이번에 이 원리를 ddd 가이드에 규칙으로 옮겨 적었고, 그 출처로 이 글을 남깁니다.

## 무엇인가

코드베이스를 클릭 가능한 노드(파일·함수·클래스)와 엣지(호출·import·테스트 관계)로 이뤄진 **지식 그래프**로 변환하는 오픈소스 Claude Code 플러그인입니다. "20만 줄짜리 코드에 새로 합류했을 때 어디부터 보나"가 출발 질문입니다. README의 한 문장이 철학을 압축합니다.

> "그래프가 코드가 얼마나 복잡한지 과시하는 게 아니라, 각 조각이 어떻게 맞물리는지 조용히 가르쳐야 한다 (Graphs that teach > graphs that impress)."

## 핵심 설계 — 하이브리드 분리

이 도구를 다른 코드 분석기와 가르는 한 줄은 두 책임을 섞지 않는다는 점입니다.

| 축 | 도구 | 역할 | 성질 |
|----|------|------|------|
| **구조** | Tree-sitter (WASM) | import·export·함수·클래스·호출지점 추출 | 결정론적·재현 가능 |
| **의미** | LLM 서브에이전트 6종 | 구조+소스를 읽어 요약·레이어·도메인 매핑 생성 | 해석·자연어 |

왜 이 분리가 중요한가. LLM에게 "이 코드의 구조를 알려줘"라고 시키면 그럴듯하지만 틀린 호출 관계를 지어낼 수 있습니다. 반면 파서는 추측하지 않고 소스 트리에서 사실만 뽑습니다. 그래서 사실은 파서가 확정하고, LLM은 그 확정된 골격 *위에서만* 해석을 얹습니다. 환각이 끼어들 자리를 구조적으로 좁히는 설계입니다.

## 어떻게 동작하는가 — 7단계 파이프라인

`/understand` 한 번에 다음이 순서대로 돕니다.

| 단계 | 에이전트/스크립트 | 산출 |
|------|------------------|------|
| scan | project-scanner | 언어·import 맵 (`scan-result.json`) |
| batch | Node 스크립트 | 병렬 분석용 의미 단위 묶음 (`batches.json`) |
| analyze | file-analyzer ×최대 5 동시 | GraphNode/Edge → merge → `assembled-graph.json` |
| review | assemble-reviewer | 그래프 정합성 검증 |
| architecture | architecture-analyzer | 레이어 식별 (`layers[]`) |
| tour | tour-builder | README·진입점부터 의존성 순 학습 경로 (`tour.json`) |
| validate & save | graph-reviewer | dangling edge 자동수정 → `knowledge-graph.json` |

commit hash를 비교해 **바뀐 파일만 재분석**하는 증분 업데이트를 지원하고, `--auto-update`로 post-commit 훅에 걸 수 있습니다. 노드 13종(file·function·class·endpoint·table·pipeline·schema 등)과 엣지 26종(imports·calls·tested_by·configures·deploys 등)으로 그래프를 구성합니다.

명령어: `/understand` `/understand-dashboard` `/understand-chat` `/understand-diff` `/understand-domain` `/understand-explain` `/understand-onboard` `/understand-knowledge`.

## 내가 쓸 만한 유스케이스

도구를 통째로 들이는 것보다, 내 워크플로의 어디에 끼우면 값을 하는지가 중요합니다. 우선순위 순으로 정리합니다.

**① 설계 원리 이식 — 가장 영속적 (실제로 이번에 함)**
"구조=AST로 확정, 해석=LLM"을 제 `ddd-routing.md`에 규칙으로 박았습니다. 이미 보유한 `ast_grep_search` MCP로 일부는 도구 없이도 즉시 실현됩니다. 도메인 의존 방향이나 Aggregate 경계를 진단할 때, LLM이 "A가 B를 호출한다"고 *추측*하기 전에 AST로 호출 사실을 먼저 확정하는 절차입니다. 이건 도구 설치와 무관하게 영구히 남는 이득입니다.

**② TPS 멀티모듈 온보딩 보조**
operator-api·executor·common-api로 갈라진 TPS는 정확히 "어디부터" 케이스입니다. 한 모듈에 `/understand`를 돌려 의존성 순 tour와 레이어 그래프를 뽑으면, 신규 모듈 진입이나 Bounded Context 경계 진단의 시각 보조가 됩니다. 단, 사내 코드라 분석 단계가 외부로 코드를 보내는지(서브에이전트=로컬 Claude인지) 먼저 확인해야 합니다.

**③ 학습 노트 자체를 그래프로**
runners-high 노트는 이미 카테고리·교차참조로 엮인 누적 구조입니다. `understand-knowledge`(Karpathy 패턴 wiki를 force-directed 그래프로 파싱)로 노트 간 관계를 시각화하면, 주제 축 재배치(reorg)를 결정할 때 어느 노트가 고립됐고 어디가 과밀한지 한눈에 보입니다. 단, Drive CloudStorage의 NFD·mmap 함정이 도구와 충돌하지 않는지가 변수입니다.

**들이지 말아야 할 것.** 플러그인 전체 설치는 비추입니다. `/understand`는 제 `deepinit`(계층형 AGENTS.md)와, `understand-knowledge`는 제 `wiki` 스킬과, project-scanner는 `explore` 에이전트와 기능이 겹칩니다. 게다가 file-analyzer ×5 동시 실행은 제 토큰 위생 규칙(동시 3개)과 충돌하고, 다크럭셔리 React 대시보드는 터미널·마크다운 중심인 제 동선과 결이 다릅니다. 그래서 결론은 "통째 도입 비추, 설계 패턴만 이식 + 필요 시 산출 JSON만 활용"입니다.

## 인상 깊은 점

가장 인상적인 건 "그래프가 가르쳐야 한다"는 절제입니다. 코드 시각화 도구는 보통 노드를 잔뜩 그려 복잡함을 과시하는 쪽으로 흐르는데, 이 도구는 명시적으로 그 반대를 목표로 둡니다. tour-builder가 README와 진입점부터 의존성 순으로 학습 경로를 만든다는 점도 같은 맥락입니다 — 그래프를 던져주고 끝내는 게 아니라 *어디부터 볼지*까지 정해줍니다. 앞 저널의 "엔트리 포인트에서 시작" 기법을 도구가 자동화한 셈입니다.

## 내 시스템과의 연결점

이 도구의 "파서가 사실, LLM이 해석" 분리는 제 MEMORY의 두 교훈 — "라이브러리 사실은 jar/소스가 1차 자료", "DB 스키마는 라이브가 1차 자료" — 과 정확히 같은 원리입니다. 셋 다 "LLM의 그럴듯한 답을 결정론적 1차 자료로 교차검증한다"로 수렴합니다. 이번에 이 원리를 `.claude/rules/ddd-routing.md`에 "AST 사실 우선" 절로 박아, 코드 구조 판단이 추측에서 출발하지 않도록 게이트를 추가했습니다. 앞 저널(코드 읽기 6기법)의 "③ 함수가 아니라 데이터를 따라간다"가 사람의 절차였다면, 이 도구의 엣지 그래프는 그 데이터 흐름을 기계가 그려준 형태라서, 사람·도구 양쪽에서 같은 읽기 모델을 공유합니다.

## 더 파볼 주제 (나중에 읽기용)

이 도구 자체보다, 도구가 채택한 *기법의 원전*을 따라가는 쪽이 학습 가치가 큽니다. 당장 실습할 건 아니고, 나중에 아래 검색어에서 출발하기 위한 포인터로 남겨둡니다. 가치순입니다.

**① 하이브리드 코드 분석 — 결정론 + 확률론.** "파서가 사실, LLM이 해석"은 이 도구의 발명이 아니라 프로그램 분석 학계의 오래된 긴장을 LLM 시대에 재배치한 것입니다. 검색 축: *정적 분석 vs 동적 분석*, **sound(거짓음성 없음) vs complete(거짓양성 없음)** 트레이드오프 — 이 도구는 sound한 골격 위에 unsound한 해석을 얹는 절충입니다. 그리고 **neuro-symbolic**(기호적 골격 + 신경망 해석) 접근의 한 사례로 읽으면, 제 "1차 자료 우선" 교훈이 사실 이 분야의 핵심 원리와 같다는 게 보입니다.

**② 코드의 그래프 표현론 — AST를 넘어서.** 이 도구는 호출/import 그래프를 쓰는데, 그 위에 더 깊은 표현이 있습니다. **Code Property Graph (CPG)** — AST + CFG(제어흐름) + PDG(데이터흐름)를 한 그래프로 합친 표현으로, Joern 같은 보안 분석 도구의 기반입니다. 이 도구의 "엣지 26종"이 한 일의 학술적 정본입니다. 함께: **tree-sitter의 incremental parsing** 알고리즘(편집된 부분만 재파싱 — 이 도구의 증분 업데이트가 차용). ①②는 제 DDD 의존성 진단(ArchUnit)·보안 분석·LLM 환각 통제가 한 점으로 수렴하는 이론 축이라, 읽으면 여러 작업의 근거가 동시에 단단해집니다.

**③ 코드베이스 온보딩의 인지과학.** "위→아래 읽기"가 왜 실패하고 "호출 그래프 추적"이 왜 통하는가. 검색 축: *program comprehension* 연구의 **top-down vs bottom-up 모델**, **beacon** 개념. 앞 저널(6기법)과 이 도구가 공유하는 전제의 학술적 근거입니다.

**④ LLM 에이전트 오케스트레이션 패턴.** 6에이전트 파이프라인(scanner→analyzer→reviewer→builder)은 제 OMC 멀티에이전트 환경과 같은 설계 공간입니다. 검색 축: **scatter-gather**, reviewer 분리, 증분 캐싱. 흥미롭지만 ①②만큼 제 작업에 직결되진 않습니다.
