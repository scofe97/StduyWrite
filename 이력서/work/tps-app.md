## 티켓 · 변경관리(CM) · 워크플로우

> 2024-04 ~ 2026-04 · 도메인 최대 비중 · 주요 티켓 TPS-2104~2111, IGMU-297/321/491/516

트럼본의 변경관리 핵심 도메인입니다. 금융권 고객사가 쓰는 시스템이라 "모든 상태 변화는 사후에 누가·언제·왜였는지 재구성할 수 있어야 한다"를 출발점으로 삼았고, 티켓 CRUD·검색·필터·연계·진행이력 전 구간을 프런트와 백엔드에서 단독 구축했습니다. 외부 PMS 와 연결티켓 관계를 별도 정책으로 분리해 마스터·서브 티켓 흐름을 명확히 했습니다.

2025년에는 티켓 화면을 v3 로 재아키텍처했습니다. 컴포넌트별로 흩어져 있던 변경관리(CM) 상태를 React Context 기반으로 묶어 단계 간 데이터 정합을 잡고, 백엔드 v3 API 로 데이터 계층을 마이그레이션했습니다. 워크플로우 컴포넌트를 그룹화해 복잡한 다단계 흐름을 한 화면에서 보이도록 개편했고, 승인/반려·티켓 생성 결재흐름의 프로세스 흐름도를 추가했습니다. 정적분석(Sparrow) 결과를 워크플로우 Step·티켓 리포트 컴포넌트로 연동하는 작업도 진행했습니다.

## 결재 · 승인 관리

> 2024-10 ~ 2026-04 · 주요 티켓 TPS-1736~1741(승인관리 신규), TPS-2012·2141(티켓·결재 연계)

결재를 비즈니스 트랜잭션으로 보고, 외부 시스템 장애가 결재 상태를 오염시키지 않도록 설계했습니다. 결재 완료 트랜잭션과 외부 호출 후속 트랜잭션을 분리해(2025-12 트랜잭션 이동/분리 작업), 외부 장애 시 결재 상태는 유지되고 후속 호출만 재시도되도록 했습니다.

승인관리 페이지를 신규 구축하면서(TPS-1736~1741) 권한별 결재 플로우, 단독결재 그룹처리, 승인워크플로우 병렬처리, 대결(代決) 로직, 결재 회수를 각각 별도 정책으로 분리해 결재선 변형마다 코드를 다시 짜지 않도록 했습니다. 결재 팝업에 Lexical 에디터를 탑재하고, 결재명 중복 방지와 결재 트리거 조회를 구현했습니다.

3.0.5 에서는 티켓과 결재를 단일 흐름으로 연계했습니다(TPS-2012/2141). 티켓 내부에서 결재를 처리하고, 승인·반려 결과에 따라 티켓 상태가 자동 전이되도록 두 도메인 상태를 동기화했습니다. 결재 컴포넌트 타입 필터링 정책과 검색용 필터링 메서드도 추가했습니다.

## 통합 · 인수테스트 시나리오 관리

> 2024-10 ~ 2025+ · 약 230건 · QA 티켓 프로젝트 TQ-

제품 기능으로서의 테스트 관리 시스템입니다(개발용 테스트가 아님). 변경관리에서 테스트는 별도 산출물이 아니라 티켓 흐름의 한 단계라는 기준으로, 테스트 의뢰부터 결과까지 같은 감사 추적선 위에 두었습니다.

테스트 시나리오 그룹과 시나리오를 별도 도메인으로 설계해 CRUD·검색·드래그 정렬·불러오기를 구현하고, 시나리오 테이블을 영속 계층(core-persistence)에 신규 설계했습니다. 통합테스트 의뢰/결과를 티켓 진행이력에 연계하고, JUnit·정적테스트 결과와 증빙 파일을 시나리오에 묶었습니다. 파일 증빙은 Minio 로 처리했고(공통 업로드/다운로드 훅), 시나리오 생성·수정·불러오기마다 감사로그를 자동 적재했습니다. 작업 중 레거시 폼(formik)을 react-hook-form 으로 마이그레이션해 폼 검증 일관성을 확보했습니다.
