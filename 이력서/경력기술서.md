## 트럼본 본체 (티켓·결재·워크플로우) | 오케스트로 | 2024.01 ~ 재직 중

### 프로젝트 소개
- 금융권·공공 고객사가 사용하는 차세대 DevSecOps 플랫폼의 변경관리 핵심 도메인. 흩어진 Jenkins·ArgoCD·SonarQube·GitLab 흐름을 티켓 기반 워크플로우로 정렬하고, 금감원 변경관리 규정에 맞춰 모든 배포를 감사 추적 가능하게 만드는 것이 비즈니스 가치입니다.

### 프로젝트 목표
- 외부 PMS·엑셀로 흩어져 있던 요구사항을 단일 출처로 통합
- 결재·티켓·워크플로우 상태 전이를 감사 가능하게 코드 레벨에서 보장
- 결재가 모두 승인된 뒤 외부 호출 실패로 발생하던 "승인됐지만 후속 처리는 안 된" 모호한 상태 제거
- 사용자가 한 화면에서 변경관리 전 과정을 완결할 수 있는 흐름 구축

### 사용 기술
- Java 17 | Spring Boot 3.x | MyBatis | WebClient | React 18 | TypeScript | MariaDB | Webhook | Scheduler

### 담당 업무

- 결재 관리 시스템 백엔드·프런트엔드 단독 신규 구축

### 문제 해결 경험

- **외부 PMS 연계로 인한 상태 불일치 제거**

  [업무 배경]
  - Jira·내부 PMS·엑셀로 요구사항이 흩어져 있어 동일 티켓이 시스템마다 다른 상태로 존재했습니다.
  - 금융권 변경관리 감사 시 상태 불일치가 위험 요소였습니다. "모든 상태 변화는 누가·언제·왜 일어났는지 사후에 재구성할 수 있어야 한다"는 기준을 도메인 설계의 출발점으로 삼았습니다.

  [해결 방안]
  - 외부 PMS와 양방향 동기화하는 연계 로직을 구현해 단일 출처를 확보했습니다.
  - 연결 티켓 관계와 라이프사이클을 별도 정책으로 분리해 마스터·서브 티켓 흐름을 명확히 했습니다. 같은 도메인 모델에 라이프사이클 정책을 섞으면 변형마다 코드를 다시 짜야 한다고 판단했기 때문입니다.

  [구현]
  - MyBatis 기반 백엔드에서 양방향 동기화 어댑터를 별도 모듈로 분리했습니다.
  - 상태 전이 지점마다 진행이력·감사로그를 자동 적재하는 트래킹 구조를 설계했습니다. 도메인 객체가 상태 전이를 발생시키면 도메인 이벤트가 발행되고, 별도 어드바이저가 감사로그를 적재하는 구조입니다.
  - 프런트엔드는 React 18 + TypeScript로 티켓 CRUD·검색·필터·연계·이력 전 구간을 단독 구축했습니다.

  [결과]
  - 3.0.4 외부 릴리즈에 안정 탑재. 결재 도메인 47회·티켓 도메인 24회 커밋으로 도메인 안정화 완료.
  - 사후 재구성 가능성을 코드 레벨에서 보장. 감사 시 상태 불일치 0 건.

- **결재 트랜잭션과 외부 호출 분리로 모호한 상태 제거**

  [업무 배경]
  - 결재가 모두 승인된 뒤 외부 API 호출이 실패하면 "승인됐지만 후속 처리는 안 된" 모호한 상태가 남았습니다.
  - 단순 해결책으로 결재 완료와 외부 호출을 단일 트랜잭션으로 묶으면 외부 장애가 결재 트랜잭션을 통째로 잠그는 역효과가 발생했습니다. "결재는 비즈니스 트랜잭션이며, 외부 시스템 장애가 결재 상태를 오염시켜선 안 된다"는 기준이 필요했습니다.

  [해결 방안]
  - 결재 완료 트랜잭션과 외부 호출 후속 트랜잭션을 분리했습니다. 외부 호출 실패는 결재 상태에 영향을 주지 않고, 후속 호출만 재시도 큐로 이동하는 구조입니다.
  - 권한별 결재 플로우·진행 단계 필터링·대결(代決) 로직·결재 회수 알람을 각각 별도 정책으로 분리해 결재선 변형마다 코드를 다시 짜지 않아도 되도록 했습니다.

  [구현]
  - WebClient 호출에 멱등 보장을 추가했습니다. 같은 요청이 중복 발송돼도 외부 시스템 상태가 일관되게 유지됩니다.
  - 외부 호출 실패·사용자 반려·시간 초과 세 경우를 자동 재상신과 자동 반려 흐름으로 코드화해 운영자 수기 작업을 제거했습니다.
  - Webhook 기반 알람으로 모든 상태 변화를 담당자에게 자동 전달했습니다.

  [결과]
  - 결재 상태와 외부 호출 상태가 독립적으로 추적됩니다.
  - 외부 시스템 장애가 결재 무결성을 오염시키는 사례 0 건.

- **티켓·결재 흐름 통합으로 화면 전환 비용 제거**

  [업무 배경]
  - 3.0.4까지는 티켓 화면과 결재 화면이 분리되어 사용자가 작업 도중 화면을 오가야 했습니다. 화면 전환 사이에 상태가 어긋나는 사례가 보고됐습니다.
  - "화면 전환은 비용이다. 사용자는 동일 작업 단위 안에서 화면을 이동하지 않아야 한다"는 기준이 고도화의 출발점이었습니다.

  [해결 방안]
  - 티켓 내부에서 결재를 처리할 수 있는 프로세스 흐름을 신규 구축했습니다.
  - 결재 승인·반려 결과에 따라 티켓 상태가 자동 전이되도록 두 도메인의 상태 머신을 동기화했습니다.

  [구현]
  - 티켓-결재 매핑별 이력·감사로그를 별도 적재해 어느 도메인의 변경이 다른 도메인의 상태를 바꿨는지 추적 가능하게 했습니다.
  - 워크플로우 컴포넌트를 그룹화해 단계별 진행 상태를 단일 화면에 노출하고, 티켓 보고서 다운로드 기능으로 외부 감사 자료 생성을 자동화했습니다.

  [결과]
  - 사용자 한 명이 한 화면에서 변경관리 전 과정을 완결 가능.
  - 외부 감사 자료 생성 수기 작업 제거.



## 트럼본 실행 엔진 (CI/CD 파이프라인 엔진) | 오케스트로 | 2024 ~ 재직 중

### 프로젝트 소개
- 외부 Jenkins 인스턴스 호출과 상태 추적을 담당하는 신규 실행 엔진. 트럼본 본체와 분리된 별도 모듈로, 헥사고날 아키텍처와 Kafka·Avro 기반으로 신규 구축했습니다.

### 프로젝트 목표
- 외부 Jenkins 장애가 내부 트랜잭션을 잠그는 v3 구조 제거
- 다중 Jenkins 인스턴스 환경에서 동시 디스패치 충돌 0 건 달성
- Jenkins REST API·Crumb 인증·Folder Plugin 경로의 일관성 확보
- Webhook 유실·외부 장기 무응답 등 좀비 Job 자동 종료

### 사용 기술
- Java 17 | Spring Boot 3.x | Jenkins REST API | Jenkins Pipeline | Kubernetes Plugin | Webhook | Kafka | Redpanda | Avro | Hexagonal Architecture | Pessimistic Locking | WireMock

### 담당 업무
- Jenkins 호출 추상화·인증 통합 모듈 설계·개발
- Jenkins 빌드 추적 키를 빌드 번호에서 큐 아이템 ID 기반 2단계 추적으로 전환
- Job 라이프사이클 6단계(접수·실행 요청·실행 확인·상태 추적·실패 복구·취소) 비동기 흐름 재설계
- 동시성 충돌 해결을 위한 인스턴스 식별 그룹화 값 객체 도입
- 다중 인스턴스 환경 디스패치 직렬화(비관적 락) 도입
- 동시성 제어 정책 문서 신규 작성·팀 내 의사결정 기준 명문화
- Jenkins 컨트롤러 Helm 차트 K8s 배포 + 동적 에이전트(Kubernetes Plugin) 환경 구성
- cloudbees-folder 플러그인 Jenkins LTS 기본 이미지 명시 설치 + Job 이관 파이프라인 자동화
- Jenkins Pipeline webhook 발행 스크립트 작성

### 문제 해결 경험

- **외부 Jenkins 동기 호출이 내부 트랜잭션을 잠그는 문제 해결**

  [업무 배경]
  - 기존 v3는 동기 HTTP 호출이 외부 Jenkins 장애를 그대로 전파해 내부 트랜잭션이 함께 멈추는 구조였습니다.
  - 한 인스턴스가 응답하지 않으면 전체 워크플로우가 정체됐습니다. "외부 시스템에 대한 동기 호출은 내부 트랜잭션을 잠그면 안 된다. 외부가 멈춰도 내부는 계속 돌아야 한다"는 기준이 필요했습니다.

  [해결 방안]
  - Job의 접수·실행 요청·실행 확인·상태 추적·실패 복구·취소까지 여섯 단계를 메시지 브로커 기반 비동기 흐름으로 재설계했습니다. 외부 호출은 트랜잭션 경계 밖으로 분리됩니다.
  - 헥사고날 아키텍처로 도메인 계층을 분리해 영속성·메시지 브로커 의존을 도메인 밖으로 밀어냈습니다.

  [구현]
  - GET은 Feign, POST는 RestTemplate으로 역할을 분리하되 인증·경로 처리는 단일 추상화로 묶었습니다. 호출 지점에서는 인증 디테일을 모릅니다.
  - Folder Plugin 계층 경로(`/job/{folder}/job/{name}/...`)와 Job 빌드 파라미터에서 외부 컨텍스트를 추출하는 어댑터를 별도 모듈로 분리했습니다.
  - Jenkins가 빌드 번호를 큐 진입이 아니라 executor 할당 시점에 부여하기 때문에, 트리거 직후 응답의 큐 아이템 ID를 1차 키로 잡고 큐 상태를 조회해 빌드 번호가 나타나면 보강하는 2단계 추적으로 전환했습니다. 동시 트리거 시 빌드 번호만으로 매칭이 깨지던 문제를 해결했습니다.
  - Jenkins 빌드 콜백을 webhook으로 수신하되 5분 폴링 fallback을 함께 두어 webhook 유실에도 상태 추적이 끊기지 않게 했습니다.
  - Jenkins가 24시간 이상 응답하지 않는 경우를 대비한 타임아웃 안전망을 도입해 좀비 Job을 자동 종료 상태로 전이시킵니다.

  [결과]
  - 외부 장애가 시스템 자원을 무한히 점유하는 사례 0 건.
  - 도메인 단위 테스트는 외부 의존 0으로 실행됩니다.

- **동일 Jenkins 인스턴스 동시 디스패치 충돌 해결**

  [업무 배경]
  - 같은 Jenkins 인스턴스를 가리키는 서로 다른 두 논리 그룹이 별도 흐름으로 처리되면서 해당 Jenkins 인스턴스의 대기 큐가 정원을 초과했습니다.
  - 논리 그룹 단위로 한도를 계산하니 물리 인스턴스에서는 합산 한도가 무력화됐습니다. "동시성 충돌은 항상 식별 단위의 문제다. 충돌의 진짜 단위(물리 Jenkins 인스턴스)와 다른 단위(논리 그룹)로 처리량 한도를 산출하면 충돌이 재발한다"는 기준이 필요했습니다.

  [해결 방안]
  - Jenkins 응답 헤더(`X-Jenkins`)와 노드 이름 지문으로 인스턴스 동일성을 식별하는 값 객체(Value Object)를 도입했습니다.
  - 디스패치 단계의 그룹화를 1차(논리 그룹) → 2차(물리 Jenkins 인스턴스) 두 단계로 확장해 같은 인스턴스에 속한 모든 작업의 대기 적재량을 합산하여 처리량 한도를 산출하도록 변경했습니다.

  [구현]
  - 동적 에이전트 환경에서 정적 executor 카운트가 0으로 보이는 케이스를 K8s 클러스터 상태로 보강 감지해 폴링 누락을 방지했습니다.
  - 헤더 노출이 없는 환경에서는 호스트·포트 조합으로 보조 식별하는 fallback을 두어 환경 차이를 흡수했습니다.

  [결과]
  - 두 논리 그룹이 각 4 건씩 동시 디스패치하는 부하 조건에서 Jenkins 큐 정원 초과 0 건.
  - 통합 테스트 신규 시나리오 전건 통과(2/2), QA 자동화 시나리오 4 건이 약 392초 안에 통과.

- **다중 애플리케이션 인스턴스 환경 디스패치 직렬화**

  [업무 배경]
  - 동일한 작업 그룹을 여러 애플리케이션 인스턴스가 동시에 디스패치할 때 한도 계산이 비결정적이었습니다.
  - 각 인스턴스가 자신의 시점에서만 한도를 보니 합산이 어긋났습니다. "분산 환경에서 한도 계산은 직렬화돼야 한다. 한도라는 개념 자체가 단일 진리값을 전제로 한다"는 기준이 필요했습니다.

  [해결 방안]
  - 디스패치 게이트에 비관적 락(`SELECT ... FOR UPDATE`)을 도입해 같은 그룹의 디스패치가 한 애플리케이션 인스턴스에서만 진행되도록 직렬화했습니다.
  - 락의 적용 범위를 그룹 단위로 좁혀 처리량 손실을 최소화했습니다.

  [구현]
  - 동시성 제어 정책 문서를 신규 작성해 낙관적 락·비관적 락·공유 락의 적용 책임을 분리하고 팀 내 의사결정 기준을 명문화했습니다. 새 동시성 이슈가 나올 때마다 같은 논의를 반복하지 않도록 했습니다.
  - 외부 호출 가상화는 WireMock으로 처리해 통합 테스트가 외부 의존 없이 실행됩니다.

  [결과]
  - 잔여 동시성 충돌 0으로 수렴.
  - 단위 테스트 34 건 전건 통과, 외부 호출 가상화 기반 통합 테스트 전체 통과.
