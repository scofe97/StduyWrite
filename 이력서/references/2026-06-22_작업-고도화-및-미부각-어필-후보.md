# 작업 고도화 후보 + 이미 한 것 중 미부각 어필 포인트 (2026-06-22)

> 면접에서 Jenkins·Java·Spring Boot 깊이를 강하게 어필하기 위한 작업 후보 모음.
> **두 축**: ① 고도화 후보(아직 안 함 — 테스트 후 반영) ② 미부각 어필(이미 함 — 바로 반영 가능).
> ⚠️ **정직성**: 축①은 playground/개인 GCP에서 직접 테스트·구현한 *뒤* "테스트/검증/설계" 톤으로만 이력서에 올린다. 하기 전엔 "했다"고 쓰지 않는다. 수치(STW ms·처리량 배수)는 실측값만.

## 축①. 작업 고도화 후보 — [이미 한 것] → [고도화] (면접 임팩트 순)

각 항목은 *새 주제가 아니라 기존 작업의 연장*이라, 면접에서 "이건 이렇게 했고, 더 나아가 이렇게 했다/할 수 있다"로 자연스럽게 이어진다.

### 1. ★ 실행 엔진 + JVM 성능 튜닝 (최상)
- **한 것**: 메시지 큐 기반 비동기 실행 엔진, 디스패치 한도 제어
- **고도화**: JMH 로 디스패치·소비 핫패스를 측정하고 GC(G1↔ZGC) 전환·힙 사이징으로 p99 지연·STW(일시정지)를 개선
- **왜 강한가**: "외부 의존이 큰 엔진이라 지연이 곧 장애"라는 *이미 한 블록의 문제의식*을 JVM 레벨로 끌고 감. JVM 을 "읽을 줄 안다"가 아니라 "튜닝할 줄 안다"로 격상. 수치로 다루는 백엔드는 드물다.
- **실현**: 학습노트(JMH·G1/ZGC·escape analysis) 깊음 → redpanda-playground 에서 측정만 하면 진짜가 됨
- **면접 한 줄**: "지연이 민감한 모듈이라 JMH 로 핫패스를 떠보고, GC 로그를 분석해 ZGC 로 STW 를 줄였습니다."

### 2. Outbox/멱등 → 메시징 일관성 "전체 그림" (상)
- **한 것**: Outbox 패턴, 멱등 가드, 재시도 토픽
- **고도화**: DLQ + 중복 제거 윈도우 + 재처리 정책까지 묶어 exactly-once 근접 *전체 설계*로
- **왜 강한가**: 흩어진 조각(Outbox·멱등·재시도)을 하나의 일관성 전략으로 승격. "부분 구현"이 아니라 "설계자"로 보임.
- **면접 한 줄**: "at-least-once 위에서 멱등·Outbox·DLQ 로 사실상 exactly-once 에 근접시켰습니다."

### 3. 운영표준화 → JCasC + ArgoCD (상)
- **한 것**: Jenkins 컨트롤러 Helm 형상관리, 멀티 인스턴스 운영
- **고도화**: JCasC(Configuration as Code)로 컨트롤러 *설정*까지 YAML 화 + ArgoCD 로 Git SSOT → 설정 드리프트 제거
- **왜 강한가**: 본인 ArgoCD 강점과 결합. "이미지·플러그인은 코드화했는데 설정 드리프트가 남았다 → JCasC 로 설정까지"라는 자연스러운 다음 단계.
- **실현**: playground 1~2일 PoC. ⚠️ JCasC 세부 스키마는 미검증 — 깊은 질문 시 "플러그인 문서 확인 필요"로 솔직히.

### 4. 디스패치 한도 → 적응형 백프레셔 (중상)
- **한 것**: DB·실제 큐 중 큰 값으로 임계값 제어(이중 안전장치)
- **고도화**: Jenkins 응답 지연·실패율에 따라 한도를 동적으로 조절하는 적응형 backpressure
- **왜 강한가**: 고정 임계값 → 실시간 상태 기반 적응형으로 한 차원 위. "backpressure" 용어로 분산 시스템 이해 어필.

### 5. queueId 추적 → durability/resume 보정 (중상)
- **한 것**: queueId 2단계 매칭, webhook+폴링 하이브리드
- **고도화**: Jenkins 컨트롤러 재시작 시 진행 중 빌드 상태 reconcile 보정 루프 (CPS durability)
- **왜 강한가**: "재시작 사이 webhook 유실"이라는 *이미 다룬 유실 문제*를 컨트롤러 장애까지 확장. ⚠️ CPS durability 세부는 미검증.

### 6. record 453개 → 불변 도메인 + 검증을 컴파일 타임으로 (중상)
- **한 것**: VO·DTO·쿼리 조건을 record 로 (operator 453개). compact 생성자 일부.
- **고도화**: record compact 생성자에 불변식 검증(범위·null·도메인 규칙)을 모아 *생성 시점에* 깨진 상태를 원천 차단 + sealed interface 로 상태 타입을 닫아 switch 표현식이 모든 경우를 강제하게.
- **왜 강한가**: "record 썼다"(문법)가 아니라 "불변성·전수 검사를 타입 시스템에 위임했다"(설계)로 격상. 자바 17+ 모던 기능을 *왜* 쓰는지 답할 수 있음.
- **실현**: 이미 record·switch 표현식 사용 중 → playground 모듈에 sealed 상태 + compact 검증 1세트 시연.
- **면접 한 줄**: "상태를 sealed 로 닫아, 새 상태가 생기면 컴파일러가 누락된 분기를 잡게 했습니다."

### 7. @Transactional 355곳 → 트랜잭션 경계·전파 설계 (중상)
- **한 것**: 서비스 전반 @Transactional (operator 355, executor 101). 이벤트 발행과 DB 변경 공존.
- **고도화**: 외부 호출(Jenkins·Feign)을 트랜잭션 밖으로 빼 커넥션 점유를 줄이고, @TransactionalEventListener(AFTER_COMMIT)로 "커밋된 뒤에만 이벤트 발행"을 보장 → 롤백 시 유령 이벤트 차단.
- **왜 강한가**: 트랜잭션을 "붙인다"가 아니라 "경계를 어디 긋느냐"로 다룸. 외부 I/O 와 DB 트랜잭션 분리는 실무 단골 함정.
- **실현**: 이미 이벤트 리스너 다수 보유 → AFTER_COMMIT 전환 1건 시연.
- **면접 한 줄**: "외부 호출을 트랜잭션 밖으로 빼고, 이벤트는 커밋 후에만 나가게 해 롤백과 발행의 정합을 맞췄습니다."

### 8. 단순 @Transactional → 메트릭 가시화(Micrometer) (중상)
- **한 것**: 디스패치·소비·복구 스케줄러 등 핵심 경로. *메트릭은 아직 없음*(grep 0).
- **고도화**: 디스패치 처리량·큐 적체·재시도율·DLT 유입을 Micrometer Counter/Timer 로 노출 → Grafana 대시보드(이미 운영 중)에 실행엔진 패널 추가.
- **왜 강한가**: 본인 관측성 인프라(Grafana·Prometheus)와 *앱 레벨 메트릭*을 잇는 그림. "지표 없이 추측"에서 "지표로 본다"로.
- **실현**: Micrometer 의존성 + @Timed 1세트, playground 에서 Prometheus 스크랩 확인.
- **면접 한 줄**: "엔진 핵심 경로에 메트릭을 심어 큐 적체와 재시도율을 대시보드에서 봤습니다."

### 9. 멱등 가드 → 분산 환경 멱등키/중복 윈도우 (중상)
- **한 것**: 작업 ID 기반 멱등 가드, RecoveryIdempotency E2E.
- **고도화**: 멀티 인스턴스에서도 안전하도록 멱등키를 DB 유니크 제약 또는 Redis SETNX 로 승격 + TTL 중복 제거 윈도우.
- **왜 강한가**: 단일 인스턴스 멱등 → 수평 확장 멱등으로. "스케일 아웃하면 깨지지 않나?"라는 압박 질문에 답이 됨.
- **실현**: ⚠️ Redis 미사용 → DB 유니크 제약 방식이 더 정직. playground 검증.

### 10. switch 표현식 → 상태 전이 명시 모델 (중상)
- **한 것**: switch 표현식 18곳, 결재·티켓 상태 분기.
- **고도화**: 흩어진 if/switch 상태 분기를 *전이 테이블*(현재상태 × 이벤트 → 다음상태)로 모아, 정의되지 않은 전이를 런타임 예외가 아니라 *표에 없으면 거부*로.
- **왜 강한가**: "상태 머신으로 모델링했다"는 이력서 문장에 실체를 부여. 전이 표는 면접에서 화이트보드로 그리기 좋음.
- **실현**: 이미 enum 상태 보유 → 전이 맵 1개로 리팩토링 시연.

### 11. WireMock E2E → 결정적 통합 테스트 피라미드 (중상)
- **한 것**: Jenkins fault injection·dispatch retry·idempotency 등 WireMock E2E *다수*.
- **고도화**: 외부 장애 시나리오(타임아웃·5xx·부분 응답)를 표로 정리해 계약 테스트화 + 실패 주입을 CI 게이트로.
- **왜 강한가**: "테스트 짰다"가 아니라 "외부 장애를 *재현 가능하게* 테스트한다". fault injection 은 면접관이 좋아하는 디테일.
- **실현**: 이미 fault injection E2E 보유 → 시나리오 매트릭스 문서화면 바로 어필.
- **면접 한 줄**: "Jenkins 5xx·타임아웃을 WireMock 으로 주입해, 재시도·복구 경로를 결정적으로 검증했습니다."

### 12. MyBatis+JPA+QueryDSL 혼용 → 의도적 선택 기준 (중상)

- **왜 강한가**: 세 ORM 혼용은 보통 "정리 안 됨"으로 보이지만, 선택 기준을 말하면 "도구를 안다"로 뒤집힘. 흔한 압박 질문.
- **실현**: 코드 분포는 이미 있음 → 기준 1장 정리(코드 변경 불필요).

### 13. AuditorAware → 감사 추적 무결성 강화 (중)

### 14. @Scheduled 복구 → 리더 선출/중복 실행 방지 (중)

### 15. Feign 11곳 → 회복탄력성(타임아웃·재시도·회로차단) (중)

## 축②. 이미 한 것 중 이력서에 덜 부각된 어필 포인트 (코드 근거 확인됨)

이미 이력서에 있는 것(비동기 엔진·디스패치·queueId·crumb·Outbox·멱등·cloudbees-folder·webhook hook) **외에**, 실제 코드에서 발굴한 Java·Spring·Jenkins 기술 깊이. **이건 이미 한 것이라 테스트 없이 바로 이력서/면접에 쓸 수 있다.**

### Java 깊이
- **record 기반 불변 설계** — VO·DTO·쿼리 조건을 record 로 (`executor/core-db/.../query/Resolved*.java`, `core-library/.../UserData.java`). switch 표현식 병행.
  → 어필: "불변 값 객체 중심으로 도메인·쿼리 조건을 설계해 동시성 안전성 확보"

### Spring 깊이
- **Spring Kafka @RetryableTopic + DLT** — 컨슈머 실패 시 재시도 토픽·DLT 자동화 (`engine/common/messaging/{Deploy,Build}CommandConsumer.java`)
  → 어필: "Spring Kafka 의 재시도 토픽·DLT 로 컨슈머 실패를 유실 없이 처리"
- **AuditorAware 감사 투명 주입** — 생성/수정자 감사 컬럼을 인프라 레벨에서 자동 주입 (`engine/jenkins/infrastructure/config/AuditConfig.java`)
  → 어필: "감사 추적을 비즈니스 코드와 분리해 인프라 레벨에서 투명하게 적재" (금융권 감사 맥락과 직결)
- **@Scheduled 복구 스케줄러** — submit·복구 폴링 (`engine/jenkins/infrastructure/scheduler/SubmitScheduler.java`)

### Jenkins / Feign 깊이
- **JenkinsAuthInterceptor (Feign RequestInterceptor)** — crumb 인증을 호출 직전 자동 주입 (`engine/jenkins/infrastructure/jenkins/JenkinsAuthInterceptor.java`)
  → 어필: "인증 디테일을 호출 지점에서 분리해, 호출 코드는 crumb 을 모르게 인터셉터로 캡슐화" (현재 이력서엔 "crumb 직접 처리"만 있고 *interceptor 패턴*은 미부각)

### 아키텍처 깊이
- **헥사고날 멀티 도메인 port/adapter 분리** — jenkins·argocd·artifact·manifest 각 도메인이 `domain/port/{in,out}` 으로 분리, 도메인은 Spring 의존 0 (`engine/*/domain/port/`)
  → 어필: "도메인을 외부 의존 0 으로 유지해 인프라 없이 단위 테스트가 도는 헥사고날 구조"

### 테스트 깊이
- **도메인 단위 + 통합(WireMock) given/when/then, 포트 mock 격리** — 이력서엔 "단위 테스트 34건"만 있고 *전략*은 미부각
  → 어필: "외부 호출을 WireMock 으로 가상화하고 포트를 mock 으로 격리해, 도메인 로직을 외부 의존 없이 검증"

## 활용 가이드

- **축② (미부각)**: 이미 한 것이라 바로 반영 가능. 단 현재 이력서가 50~80자로 압축돼 있어 *어디에 끼울지*는 별도 결정. 후보: 실행엔진 블록(Feign interceptor·헥사고날), 티켓 블록(AuditorAware 감사), 기술 태그 보강.
- **축① (고도화)**: 하나 골라 playground 에서 PoC → 끝나면 정직 톤으로 반영. **1순위 = JVM 성능 튜닝**(사용자 직접 언급 + 학습 깊이 + 차별화).
- 면접 구두 카드로도 활용: 본 문서 자체가 "이건 이렇게 했고, 더 파면 이렇게 된다"는 답변 시나리오.
