## A. 금융권 변경관리 ↔ CI/CD 게이트 (승인·감사추적·릴리즈)

### 업계 연결 구조 (확인된 사실)

규제 환경의 CI/CD 파이프라인은 ITIL 변경관리와 결합합니다. 핵심은 수동 검토를 자동화된 워크플로 게이트로 대체하되 통제는 유지하는 것입니다(출처: aws.plainenglish.io, isaca.org).

1. **변경 유형 3분류(ITIL)** — Standard(저위험·사전승인)·Normal·Emergency. 반복 가능한 배포는 Standard change로 사전 승인해 속도를 올립니다(atlassian.com/itsm/change-management).
2. **승인 게이트 + 직무분리(SoD)** — SOX 핵심: 개발한 사람이 배포할 수 없다. 현대 파이프라인은 PR 피어 리뷰 강제 + 파이프라인 자체를 "배포자"로 삼아 통제를 시스템화합니다(harness.io).
3. **불변 감사추적** — git commit → 최종 배포까지 모든 행위가 불변 로그를 남깁니다. 파이프라인이 commit 해시·변경 사유·스캔 결과·승인자를 자동 캡처합니다(harness.io).
4. **추적성** — 감사관은 모든 프로덕션 변경을 원래 승인된 티켓(Jira·ServiceNow)까지 역추적할 수 있어야 합니다.

### 한국 금융권 규제 매핑 (확인된 사실 — TROMBONE 금융 도메인 직결)

전자금융감독규정의 "프로그램 통제" 조항이 위 업계 패턴과 정확히 대응합니다:
- 프로그램 등록·변경·폐기 기준 수립 의무
- **직무분리**: 프로그램 릴리즈·실행파일 생성·운영시스템 등록은 프로그램 담당자가 아닌 자(예: 자료관리자)가 수행(law.go.kr 전자금융감독규정)
- 배포 전 자체 보안성 심의 의무
- CISO 정기 점검 + CEO 보고, 정보보호위원회 심의, FSS 검사

> **면접 카드 A**: "제가 만든 티켓·결재 시스템은 단순 워크플로 도구가 아니라, 전자금융감독규정의 프로그램 통제·직무분리 요건을 코드 배포 흐름에 강제하는 거버넌스 게이트였습니다. 결재 승인이 곧 change authorization 레코드이고, 결재자≠개발자를 시스템이 보장함으로써 직무분리를 사람 약속이 아닌 시스템 통제로 전환했습니다. Jenkins 실행 엔진은 '파이프라인이 배포자'가 되는 패턴으로, 개발자가 직접 프로덕션에 손대지 않게 하면서 commit→승인→배포 전 과정의 불변 감사추적을 자동 생성합니다."

## B. DORA 메트릭 ↔ 티켓·결재·파이프라인 데이터 연결

### DORA 5메트릭 (확인된 사실, 2024년 5개로 확장)

| 메트릭 | 정의 | 데이터 소스 |
|--------|------|-------------|
| Deployment Frequency | 프로덕션 배포 빈도 | VCS/CI·CD |
| Change Lead Time | commit→프로덕션까지 시간 | VCS + CI·CD |
| Failed Deployment Recovery Time(구 MTTR) | 변경 실패 복구 시간 | 인시던트 관리 |
| Change Failure Rate | 즉시 개입 필요 실패 비율 | CI·CD + 인시던트 |
| Deployment Rework Rate(2024 신규) | 인시던트로 인한 비계획 배포 비율 | 인시던트 + 배포 |

핵심 연구 결과: 속도와 안정성은 트레이드오프가 아니라 상관관계 — 고성과 팀은 5개를 동시에 잘합니다(dora.dev). 단 정확도는 "티켓 위생"에 좌우됩니다 — 상태 즉시 갱신, 인시던트를 배포 티켓에 링크(getdx.com/blog/dora-metrics-jira).

> **면접 카드 B**: "제 티켓·결재 도메인은 DORA 메트릭의 데이터 백본이 될 수 있습니다. 결재 승인 시각 → 배포 트리거 시각의 차이가 Change Lead Time의 거버넌스 구간이고, Jenkins 실행 엔진의 배포 이벤트가 Deployment Frequency 원천입니다. 실패 시 롤백 티켓을 원 배포 티켓에 링크하면 Change Failure Rate와 Recovery Time을 산출할 수 있습니다. 중요한 건 이게 티켓 위생에 달려 있다는 점인데, 제 시스템은 결재-배포-인시던트 링크를 데이터 모델로 강제했기 때문에 사람이 깜빡해도 데이터가 끊기지 않습니다."

⚠️ 정직 포인트: DORA는 트래픽 메트릭이 아니라 배포 거버넌스 메트릭입니다. 이 카드는 "대용량 트래픽 정량 메트릭" 공백을 우회해 강점인 거버넌스/티켓 도메인으로 정량 스토리를 만드는 정공법입니다. 단 실제 산출한 수치가 없다면 "이렇게 연결할 수 있다/설계했다"로 표현하고 측정 실적이 있는 양 과장하지 마십시오.

## C. GitOps/ArgoCD에서 변경승인 강제 (모두 공식문서·실무 출처)

1. **PR이 1차 승인 게이트** — Git 브랜치 보호 규칙(피어 리뷰 N개, code owner 승인, status check)으로 프로덕션 브랜치 머지 강제(argo-cd readthedocs).
2. **PreSync hook으로 2인 규칙(SoD)** — sync 실행자 ≠ 마지막 commit 작성자를 검증해 직무분리를 GitOps에 구현(oneuptime.com).
3. **Sync Window(AppProject)** — cron 기반 allow/deny로 배포 허용 시점 통제. 분기말 freeze 차단. allow/deny 동시면 deny 우선(argo-cd readthedocs sync_windows).
4. **manualSync: true** — 자동 sync는 막되 긴급 시 인가 운영자의 수동 배포만 허용.
5. **프로덕션 auto-sync 비활성** — OutOfSync로 두고 인가 사용자가 수동 트리거해야 배포.
6. **Policy-as-Code** — OPA/admission controller로 CVE 차단·승인된 레지스트리 제한(reddit r/ArgoCD, harness.io).

> **면접 카드 C**: "제 결재 도메인 경험을 GitOps로 옮기면, 결재 승인은 Git PR 브랜치 보호 + code owner 승인에, 결재자≠개발자 직무분리는 ArgoCD PreSync hook의 2인 규칙에, 변경관리의 배포 가능 시간대 통제는 AppProject sync window의 cron allow/deny에 각각 대응됩니다. 금융권 분기말 변경 freeze는 deny window로, 긴급 배포는 manualSync override로 모델링됩니다. 제가 결재 엔진에서 구현했던 승인·시점통제·직무분리·감사추적이 GitOps에서는 ArgoCD 네이티브 기능으로 1:1 매핑된다는 걸 설명할 수 있습니다."

## D. 재표현(rephrasing) 사전 — 면접에서 바로 쓸 용어 매핑

| 기존 표현 | 거버넌스/릴리즈 엔지니어링 언어 |
|---|---|
| 티켓·결재 시스템 | Change authorization / approval gate, system of record for change |
| 결재자 ≠ 기안자 | Segregation of Duties(SoD), two-person rule |
| 결재 이력·로그 | Immutable audit trail, self-documenting deployment evidence |
| Jenkins 파이프라인 실행 엔진 | Deployment orchestrator / "pipeline-as-deployer" |
| 변경 승인 후 배포 | Release gate, deployment promotion |
| 티켓↔배포 연결 | Traceability(change → deployment), DORA data backbone |
| 비관적 락 직렬화 | Concurrency control for deployment serialization |
| 배포 시간대 통제(있다면) | Release window / change freeze(ArgoCD sync window deny) |

## E. 예상 면접 질문 & 반박 대비

1. **"승인 게이트가 배포 속도를 떨어뜨리지 않나?"** → ITIL Standard change 사전승인으로 반복 배포는 게이트 없이 통과, Normal/Emergency만 결재. DORA 연구상 통제와 속도는 트레이드오프가 아님(atlassian, dora.dev).
2. **"수동 CAB는 구식 아닌가?"** → ServiceNow도 자동 변경을 위한 Change Enablement로 진화 중. 결재 상태 변경이 파이프라인 스테이지를 자동 트리거/거부하도록 연동 가능.
3. **"ArgoCD에서 직무분리를 어떻게 코드로 강제?"** → PreSync hook이 sync 실행자와 commit 작성자 동일 여부 검증(oneuptime.com).
4. **"감사 대응은?"** → commit 해시·승인자·시각·스캔결과가 불변 로그로 남아 티켓까지 역추적(harness.io).
