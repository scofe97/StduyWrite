## A. VM / Ansible 배포

### A-1. 신규 VM 부팅 직후 apt 락 충돌로 playbook 멈춤
- **증상**: 갓 만든 VM에 첫 playbook 돌리면 `Could not get lock /var/lib/dpkg/lock-frontend` 로 apt 태스크 실패.
- **원인**: cloud-init/`unattended-upgrades`가 부팅 직후 백그라운드로 dpkg 락을 잡고 있음. [경험: Ubuntu 24.04 kubeadm 함정 3번]
- **해결**: playbook 상단 pre_task로 `systemctl stop unattended-upgrades`, 또는 apt 모듈에 `lock_timeout` 부여 후 진행. `DEBIAN_FRONTEND=noninteractive` 환경변수로 대화형 프롬프트 차단. [경험]
- **재발방지**: bootstrap 역할(role)을 별도로 두고 "락 해제 → 패키지 → 설정" 순서를 고정. apt 태스크에 `until/retries`로 락 재시도. [통념]

### A-2. 무중단 배포 중 일부 노드 실패가 전체로 번짐
- **증상**: rolling 배포 도중 한 노드 헬스체크 실패인데 전 노드가 동시에 내려가 서비스 중단.
- **원인**: `serial` 미설정 → 기본 fork(5) 병렬로 전 호스트 동시 갱신. 헬스체크/드레인 게이트 없음. [WEB: Ansible delegation 문서 — serial이 동시 실행 호스트 수 제한]
- **해결**: `serial: 1`(또는 `25%`) + `max_fail_percentage`로 실패 임계 차단. LB 드레인은 `delegate_to: lb_host`(또는 `local_action`)로 풀에서 빼고 → 배포 → 헬스 확인 → 재투입. `run_once`로 공유 리소스 쓰기 직렬화. [WEB]
- **재발방지**: pre_task=드레인, post_task=헬스+재투입을 한 play 안에 계약으로 박음. 배치 크기·실패율을 변수화해 환경별 조정. [통념]

### A-3. playbook이 재실행 때마다 변경(changed) 발생 — 멱등성 깨짐
- **증상**: 같은 playbook 두 번 돌리면 매번 `changed`. drift 추적·CI 게이트가 무의미.
- **원인**: `command`/`shell` 직접 호출, `lineinfile` 대신 텍스트 append, 비결정적 템플릿(타임스탬프) 사용. [통념]
- **해결**: 전용 모듈(`apt`, `copy`, `template`, `systemd`) 우선. `shell`은 `creates`/`removes`/`changed_when`로 멱등 가드. 템플릿은 고정 출력. [통념]
- **재발방지**: `--check`(dry-run) + `--diff`를 CI에 넣어 2회차 changed=0을 회귀 게이트로. 25종 playbook 각각에 idempotence 테스트(molecule 등). [경험: Ansible 25종 playbook 운영 → 멱등성 회귀가 실질 리스크][추측: molecule 도입 여부는 본인 확인 필요]

## B. Kubernetes (kubeadm / Kubespray)

### B-1. Ubuntu 24.04에서 kubeadm init이 preflight에서 실패
- **증상**: `kubeadm init`이 `conntrack not found` preflight 에러로 중단.
- **원인**: Ubuntu 24.04 minimal 이미지에 `conntrack` 패키지 미포함. kube-proxy가 conntrack 의존. [경험: kubeadm 함정 2번]
- **해결**: `apt-get install -y conntrack` 선행. [경험]
- **재발방지**: 노드 준비 playbook에 conntrack·socat·ebtables 등 K8s 의존 패키지 일괄 설치 단계 박제. [경험]

### B-2. containerd 설치가 docker-ce를 밀어내 노드 런타임 깨짐
- **증상**: `apt-get install containerd` 후 docker가 사라지거나 containerd 버전 충돌.
- **원인**: Ubuntu 배포판 `containerd`와 Docker CE의 `containerd.io`가 동일 파일을 두고 충돌. [경험: kubeadm 함정 1번]
- **해결**: Docker CE 저장소의 `containerd.io`로 통일. containerd `config.toml` 대화형 프롬프트는 `rm /etc/containerd/config.toml` 후 `DEBIAN_FRONTEND=noninteractive dpkg --configure -a`로 복구. `SystemdCgroup = true` 명시. [경험: 함정 1·4번]
- **재발방지**: 런타임 소스를 한 군데로 고정(저장소 우선순위 pin). containerd 설정을 템플릿으로 관리해 패키지 재설정 프롬프트 자체를 회피. [통념]

### B-3. 노드가 NotReady + "cni plugin not initialized"
- **증상**: 워커가 조인은 됐는데 `NotReady`, kubelet 로그에 `network plugin returns error: cni plugin not initialized`. Pod이 `ContainerCreating`에 멈춤.
- **원인**: `/etc/cni/net.d/`에 CNI conflist 없음(=CNI 미설치/미배포), 또는 sandbox(`pause`) 이미지 pull 실패(폐쇄망), 또는 잘못된 임시파일(`.swp`)이 파싱 깨뜨림. [WEB: platform9/kubespray #13279/containerd]
- **해결**: ① CNI 매니페스트 적용 확인(`kubectl get pods -n kube-system`에서 calico/flannel daemonset). ② 해당 노드 CNI pod 재시작. ③ 그래도 안 되면 `/etc/cni/net.d/` 정리 + `ip link delete cni0` + containerd 재시작. ④ 폐쇄망은 pause 이미지를 사내 레지스트리에서 받도록 sandbox_image 설정. [WEB]
- **재발방지**: Kubespray라면 `kube_network_plugin`이 `cni`/`none`이 아닌지 확인(미설치 상태로 남음). pause 이미지를 Nexus 미러에 박제. 노드 조인 후 `Ready` 게이트를 배포 파이프라인 체크로. [WEB][경험: 폐쇄망 Nexus 프록시 운영]

### B-4. readOnlyRootFilesystem 적용 후 Pod이 /tmp 못 써서 크래시
- **증상**: 보안 강화로 `readOnlyRootFilesystem: true` 넣었더니 앱이 임시파일 쓰기 실패로 CrashLoopBackOff.
- **원인**: 루트 FS가 RO라 `/tmp`·캐시·로그 경로 쓰기 불가. JVM tmpdir, 멀티파트 업로드 등이 막힘. [경험: readOnlyRootFilesystem /tmp]
- **해결**: 쓰기가 필요한 경로만 `emptyDir` volume으로 마운트(`/tmp`, `/var/run`, 앱 캐시). JVM은 `-Djava.io.tmpdir=/tmp` + emptyDir. [경험][통념]
- **재발방지**: 컨테이너 보안 컨텍스트 표준 템플릿에 "RO root + 명시적 쓰기 volume 목록"을 묶어서 배포. 신규 워크로드는 이 베이스라인 상속. [통념]

### B-5. Kubespray 멀티 컨트롤플레인(master3/worker7) 운영 함정
- **증상**: control-plane 3대 중 일부 재부팅 후 etcd 쿼럼·인증서 이슈, 스케일아웃 시 join 실패.
- **원인**: etcd는 홀수 쿼럼(3=1대 장애 허용) 전제 — 2대 동시 다운 시 쓰기 불가. 인증서 만료, `inventory.ini` 그룹 매핑 실수. [경험: Kubespray master3/worker7][통념]
- **해결**: control-plane 동시 재부팅 금지(롤링). `kubeadm certs check-expiration`/Kubespray `upgrade-cluster.yml`로 인증서 갱신. 노드 추가는 `scale.yml`, 제거는 `remove-node.yml` 사용(수동 join 지양). [통념]
- **재발방지**: etcd 백업(snapshot) cron + 복구 리허설. 인벤토리를 SSOT로 버전관리. [통념]

## C. CNI / 컨테이너 네트워킹 (지원자 환경 = Calico 또는 Flannel on kubeadm GCP 3노드)

> 지원자 클러스터의 실제 CNI 종류는 본인 확인 필요. 아래는 Calico/Flannel/Cilium 별 핵심 함정. [추측: GCP kubeadm 기본은 Calico 또는 Flannel일 가능성 — 확인 권장]

### C-1. Pod IP 풀 고갈 — ContainerCreating에서 멈춤
- **증상**: 신규 Pod이 `ContainerCreating`/`Pending` + "failed to allocate IP".
- **원인**: 노드별 CIDR 블록 소진(node-level) 또는 클러스터 풀 자체 소진(cluster-level). stale 할당(Pod 없는데 IP 점유)도 누적. [WEB: cilium IPAM 글]
- **해결**: 노드별 IP 가용량 확인(Cilium은 `kubectl get ciliumnodes`). 워크로드 스케일다운, per-node CIDR 마스크 확대(예: /24→/22), 또는 클러스터 풀 확장. stale 할당 정리. [WEB]
- **재발방지**: 노드 수×Pod 밀도 기준으로 podCIDR 사이징을 초기 설계에 반영. IP 가용량 메트릭을 Prometheus 알람으로(가용 IP < 임계). [WEB][경험: 관측성 스택 보유]

### C-2. MTU 불일치 — 작은 패킷은 되는데 큰 전송/TLS가 멈춤
- **증상**: ping/짧은 요청은 정상인데 대용량 전송·TLS 핸드셰이크가 행(hang).
- **원인**: 오버레이(VXLAN/IPIP) 캡슐화 오버헤드를 MTU에 반영 안 함. 물리 1500이면 오버레이는 ~1450이어야 함(VXLAN 50바이트). GCP는 기본 MTU 1460이라 더 빡빡함. [WEB: flannel MTU 글][추측: GCP 1460 영향은 통념적 사실]
- **해결**: CNI ConfigMap의 MTU를 물리-오버헤드로 맞추고 daemonset 재시작. `nc -zuv <node> 8472`로 VXLAN 포트 차단 여부 확인. [WEB]
- **재발방지**: 클러스터 구축 시 노드 NIC MTU를 먼저 측정해 CNI MTU 자동 산정. GCP 1460 전제로 오버레이 MTU 박제. [WEB][추측]

### C-3. NetworkPolicy 적용 후 통신 두절/끊김
- **증상**: NetworkPolicy 도입하자 의도치 않은 Pod 간 연결 차단, 또는 정책 갱신 시 기존 연결이 TCP reset으로 끊김.
- **원인**: NetworkPolicy는 default-deny가 적용되면 명시 허용 외 전부 차단. Flannel은 NetworkPolicy 미지원(Calico/Cilium 필요). 정책 reprogram 시 기존 세션 리셋. [WEB: calico/linkedin][통념: Flannel 단독은 정책 미지원]
- **해결**: default-deny 전에 필수 통신(DNS=kube-dns 53, API server) egress 허용부터. Flannel만 쓰면 Calico policy-only 병행 또는 Cilium 전환. 단계적 적용(namespace 단위). [WEB][통념]
- **재발방지**: 정책을 코드로 관리(GitOps) + 스테이징에서 `cilium connectivity test`류 검증 후 prod. [WEB]

### C-4. Calico BGP 피어링 미성립
- **증상**: 노드 간 Pod 라우팅 안 됨, BGP 세션이 `idle`/`Active`에 머묾.
- **원인**: BGP 이웃 미성립(방화벽 179 차단, update-source 오설정), 또는 GCP 같은 클라우드 L2에서 BGP 직접 피어링 비호환. [WEB: noction/cisco BGP][통념: 클라우드는 IPIP/VXLAN 오버레이가 무난]
- **해결**: `calicoctl node status`로 피어 확인. 클라우드 환경은 BGP 대신 IPIP/VXLAN 오버레이 모드 권장. 온프레면 179 포트·라우터 피어링 점검. [WEB][통념]
- **재발방지**: 환경(클라우드 vs 온프레)에 맞는 encapsulation을 IPPool에 명시. GCP 3노드는 오버레이 고정. [추측: 본인 환경 모드 확인]

## D. Docker / 컨테이너 빌드·런타임

### D-1. 존재하지 않는 이미지 태그로 compose 전체 기동 실패
- **증상**: `docker compose up`이 한 서비스의 `manifest unknown`으로 전체 스택 안 뜸.
- **원인**: 검증 안 된 이미지 태그 지정(예: 오탈자 버전). [경험: docker-compose.md 주의문 — 존재하지 않는 태그 시 전체 compose 실패]
- **해결**: 배포 전 `docker manifest inspect <image:tag>`로 태그 존재 확인. [경험]
- **재발방지**: 이미지 버전을 변수/SSOT로 고정, CI에서 manifest inspect 게이트. [경험]

### D-2. 폐쇄망에서 이미지 pull 실패
- **증상**: 외부 레지스트리 접근 차단 환경에서 `pull access denied`/timeout.
- **원인**: 폐쇄망이라 Docker Hub·quay 직접 접근 불가. [경험: 폐쇄망 Nexus 프록시 운영]
- **해결**: Nexus를 프록시/미러 레지스트리로 두고 모든 pull을 사내 경유. containerd/docker에 mirror endpoint 등록, pause·CNI 이미지까지 미러. [경험][통념]
- **재발방지**: 신규 의존 이미지 도입 시 "Nexus에 먼저 캐시" 절차 박제. 미러 누락 이미지 알람. [경험]

### D-3. 컨테이너 쓰기 권한·tmpfs 문제 (B-4와 연동)
- **증상**: non-root + RO FS 컨테이너가 임시쓰기 실패.
- **원인/해결/재발방지**: B-4 참조(emptyDir/tmpfs 명시 마운트). [경험]

## E. Redpanda / 메시징 인프라 배포

### E-1. advertised listener 오설정으로 외부 클라이언트 접속 실패
- **증상**: 브로커 자체는 뜨는데 클라이언트가 부트스트랩 후 받은 주소로 재접속 실패(컨테이너 내부 IP/hostname 광고).
- **원인**: Redpanda/Kafka는 부트스트랩 응답으로 advertised listener 주소를 돌려줌. 이게 컨테이너 내부 주소(또는 NodePort 미반영)면 클라이언트가 닿지 못함. [경험: Redpanda advertised listener]
- **해결**: 외부 접근 경로(NodePort/노드 IP/도메인)를 advertised listener로 명시. 내부/외부 리스너 분리(internal vs external). [경험][통념]
- **재발방지**: 배포 매니페스트에 listener 매트릭스(internal·external 각각 advertised 주소)를 표로 고정. 환경별 값 템플릿화. [경험: redpanda 멀티 NodePort 운영]

## F. GitOps / ArgoCD

### F-1. Application이 계속 OutOfSync / drift 반복
- **증상**: 배포 후에도 ArgoCD가 OutOfSync로 표시, 또는 수동 변경이 자꾸 되돌려짐.
- **원인**: self-heal 비활성(drift 자동 교정 안 됨), 무시 설정(`ignoreDifferences`) 누락으로 컨트롤러가 채운 필드를 drift로 오인, 또는 여러 App이 같은 리소스 공유. [WEB: ArgoCD sync-options 공식문서]
- **해결**: 의도된 자동교정이면 `selfHeal: true`. 컨트롤러/웹훅이 변형하는 필드는 `RespectIgnoreDifferences` + `ignoreDifferences`로 제외. 공유 리소스는 `FailOnSharedResource=true`로 충돌 차단. [WEB]
- **재발방지**: 한 리소스=한 App 소유 원칙. mutating 필드는 ignoreDifferences에 SSOT로 등록. [WEB]

### F-2. Application이 Progressing/Syncing에 멈춤
- **증상**: 동기화가 "Syncing"에서 안 끝남.
- **원인**: `Prune=confirm`/`Delete=confirm`이 수동 승인 대기, 또는 sync wave/hook 순서 꼬임, 또는 헬스체크가 절대 Healthy 안 되는 CRD. [WEB: ArgoCD 문서 — confirm은 수동 승인까지 Syncing 유지]
- **해결**: 승인 대기면 UI/CLI/`deletion-approved` 어노테이션으로 승인. large 리소스는 `ServerSideApply=true`(annotation 한계 회피). 잘못된 wave 순서 교정. [WEB]
- **재발방지**: confirm 옵션 사용처를 명시적으로 문서화. CRD 헬스 룰을 커스텀 정의. [WEB]

## 면접 전달 팁 (구조화)
- 각 카드를 말할 때 **증상→원인→해결→재발방지** 4박자를 그대로 따라가면 "현상 추적 + 근본원인 + 재현방지"를 한 번에 보여줌 — SRE/플랫폼 면접관이 좋아하는 형태. [통념]
- 본인이 **실제로 겪은 것**(B-1·B-2·B-4, A-1, D-1·D-2, E-1)은 1인칭 사건으로, **환경상 겪을 수 있는 것**(C-1~C-4, F-1·F-2)은 "이 환경이면 이런 함정이 있어 이렇게 대비한다"로 톤 구분. 안 겪은 걸 겪은 척하지 않기. [경험 기반 정직성 원칙]
- 공백 영역(대용량 트래픽 정량 메트릭) 보강: C-1 IP 풀·MTU를 "관측성 스택으로 가용 IP/패킷드롭을 메트릭화"로 연결하면 정량 측정 약점을 일부 상쇄. [추측: 어필 전략]

## 확인 필요 (지원자 본인만 답 가능)
1. GCP 3노드 클러스터의 **실제 CNI**(Calico/Flannel/Cilium)와 encapsulation 모드(IPIP/VXLAN/none)? → C 카드 톤 확정.
2. Ansible 25종 playbook에 **molecule/idempotence 테스트** 도입 여부? → A-3 [추측] 제거 여부.
3. Redpanda를 **K8s NodePort**로 노출하는지 vs Docker compose인지(둘 다 문서에 있음) → E-1 컨텍스트.
4. ArgoCD sync 충돌을 **실제로** 겪었는지, 겪었다면 어떤 리소스에서 → F 카드를 [경험]으로 승격 가능.
