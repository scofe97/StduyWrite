## 1. 정량·구체: 모호한 형용사 대신 수치·사실로 임팩트를 보여준 문장

> "출시 후 예상 대비 10~100배 트래픽에서도 안정 운영"

### 발췌 2

> "Redis 구조 추가 개선/개편 — 사용성 최적화 및 인스턴스 축소로 전체 사용 비용 $n 감소에 기여 (기존 대비 n% 수준)"

- **출처**: 유용우, resume.yowu.dev — 야놀자 Redis 섹션

### 발췌 3

> "Redis 활용한 Response Cache 도입 및 리팩토링으로 기존 대비 Latency 90% 감소"

- **출처**: 유용우, resume.yowu.dev — 야놀자 국내 숙소 상세 개편 섹션
- **왜 읽히는가**: "빨라졌다" 대신 "Latency 90% 감소"라는 단일 숫자가 문장 전체를 떠받칩니다. 동사(도입·리팩토링)와 수치(90%)가 짝을 이뤄 인과관계가 명확합니다.

### 발췌 4

> "Coroutine 기반 병렬 조회 및 Redis 캐시 적용, 부분 실패 허용 설계로 안정성 확보"

- **출처**: 유용우, resume.yowu.dev — 네이버플러스 스토어 상품 태그 통합 API 섹션
- **왜 읽히는가**: 기법 세 가지(병렬 조회·캐시·부분 실패 허용)를 나열하되 각각이 하나의 설계 결정임을 알 수 있습니다. 같은 분량에 정보 밀도가 높습니다.

## 2. 동사·능동: "담당했다/기여했다" 대신 무엇을 했는지 동사로 미는 문장

### 발췌 5

> "5종 데이터 소스에서 개별 조회하던 상품 태그를 단일 통합 API로 설계·구현"

- **출처**: 유용우, resume.yowu.dev — 네이버플러스 스토어 상품 태그 통합 API 섹션
- **왜 읽히는가**: "설계·구현"이라는 두 동사가 문장 끝을 잡습니다. "담당했다"가 없습니다. 주어가 없어도 "내가 했다"가 자명합니다.

### 발췌 6

> "분산된 클라이언트 호출을 BFF 서버로 통합하여 데이터 경로 단일화"

- **출처**: 유용우, resume.yowu.dev — 네이버 쇼핑 메인 홈 화면 개편 섹션
- **왜 읽히는가**: 동사 "통합하여"가 원인, "단일화"가 결과를 담습니다. 명사형 마무리(단일화)가 선언적이어서 단정합니다.

### 발췌 7

> "홈을 일회성 페이지가 아닌 여러 도메인이 확장 가능한 서비스 API 플랫폼으로 구축"

- **출처**: 유용우, resume.yowu.dev — 네이버 쇼핑 메인 홈 화면 개편 섹션
- **왜 읽히는가**: "일회성 페이지가 아닌"이라는 부정 대비가 설계 의도를 드러냅니다. "구축"이라는 동사가 문장의 무게를 받습니다.

### 발췌 8

> "Java → Kotlin 전환 및 Coroutine 기반 비동기 처리 적용으로 null 안전성과 생산성 향상"

- **출처**: 유용우, resume.yowu.dev — 네이버 쇼핑 앱 서버 개발 섹션
- **왜 읽히는가**: 화살표(→)가 전환 방향을 시각적으로 보여줍니다. 기술 결정(Kotlin 전환)과 그 이유(null 안전성·생산성)가 한 문장에 연결됩니다.

## 5. 문학의 좋은 문장 원리가 이력서에서 어떻게 적용되는가

문학과 이력서는 장르가 다르지만, 좋은 문장이 작동하는 원리는 같습니다.

| 문학의 원리 | 이력서에서의 적용 |
|------------|----------------|
| **구체가 추상을 이긴다** — "그는 슬펐다" 대신 "그는 신발 끈을 세 번 묶었다" | 정량 메트릭 — "안정적이었다" 대신 "Latency 90% 감소", "예상 대비 10~100배 트래픽에서도 안정 운영" |
| **동사가 문장을 민다** — 형용사·부사에 기대지 않고 정확한 동사 하나로 장면을 만든다 | 능동 동사 — "담당", "기여", "관련" 같은 관계어 대신 "설계·구현", "통합", "전환", "구축" |
| **판단이 드러나야 인물이 보인다** — 사건의 나열이 아니라 선택의 이유가 캐릭터를 만든다 | 기준점·판단 — 기능 목록 대신 "왜 그 설계를 택했는가"를 한 줄로 붙인다 |
| **생략이 밀도를 높인다** — 말하지 않아도 되는 것을 지우면 남은 것이 강해진다 | 군더더기 제거 — "~하는 역할을 담당하였습니다" 대신 "설계·구현" 두 글자로 끊는다 |
| **대비가 의미를 만든다** — 전과 후, 안과 밖, 기대와 결과를 나란히 놓는다 | Before→After 구조 — "일회성 페이지가 아닌 확장 가능한 플랫폼", "PHP v1 → Node.js v2 + TDD" |

## 6. 지원자(심보현) 이력서의 Before 후보 — 개선 여지가 있는 문장

아래는 RESUME.md §3 Experience 카드에서 발췌한 현재 문장과 개선 방향입니다.

### Before 1 — 동사가 약한 문장

> "결재 완료 트랜잭션과 외부 호출 후속 트랜잭션을 분리하고, `WebClient` 호출에 멱등 보장을 추가했습니다."

- **문제**: "분리하고 … 추가했습니다"는 두 행위를 나열했지만 결과가 없습니다. 독자는 그래서 어떻게 됐는지를 모릅니다.
- **개선 방향**: 결과 한 줄을 붙입니다. 예: "결재 상태와 외부 호출 실패를 분리해, 외부 장애 시에도 결재 기록이 오염되지 않도록 설계했습니다." — 이유(분리)와 효과(오염 방지)가 한 문장에 담깁니다.

### Before 2 — 수치 없이 결과를 주장하는 문장

> "결과: 두 논리 그룹이 각 4 건씩 동시 디스패치하는 부하 조건에서 Jenkins 큐 정원 초과 0 건."

- **문제**: "0건"은 강하지만 "각 4건씩"이라는 조건이 작습니다. 이 조건이 실제로 얼마나 높은 부하인지 독자가 판단할 기준이 없습니다.
- **개선 방향**: Jenkins 인스턴스의 executor 수 또는 기존 초과 발생 빈도를 기준점으로 추가합니다. 예: "기존 방식에서는 동시 디스패치 시 큐 정원을 주기적으로 초과했으나, 인스턴스 식별 그룹화 도입 후 동일 조건에서 초과 0건으로 수렴했습니다."

### Before 3 — 판단 없이 기능을 나열하는 문장

> "권한별 결재 플로우와 진행 단계 필터링, 대결(代決) 로직, 결재 회수 알람을 각각 별도 정책으로 분리해 결재선 변형마다 코드를 다시 짜지 않아도 되도록 했습니다."

- **문제**: 분리의 목적(코드를 다시 짜지 않아도 된다)은 좋습니다. 그런데 "각각 별도 정책으로 분리"가 구체적이지 않아 어떤 구조인지 알 수 없습니다.
- **개선 방향**: 패턴 이름이나 구조를 한 단어로 고정합니다. 예: "결재 플로우·대결·알람을 각각 독립 정책 클래스로 분리해, 결재선 변형 시 해당 정책만 교체하면 나머지 코드는 그대로 유지됩니다."

### Before 4 — 주어 없이 흘러가는 문장

> "티켓 CRUD·검색·필터·연계·이력까지 전 구간을 백엔드(MyBatis)와 프런트엔드(React)에서 단독 신규 구축, 3.0.4 외부 릴리즈에 안정 탑재했습니다."

- **문제**: "단독"이라는 단어가 핵심인데 문장 중간에 묻혀 있습니다. 단독 구축의 무게가 나머지 기술 나열에 희석됩니다.
- **개선 방향**: "단독"을 문장 앞으로 가져옵니다. 예: "백엔드(MyBatis)와 프런트엔드(React) 전 구간을 단독으로 구축해, 티켓 CRUD·검색·필터·연계·이력까지 3.0.4 릴리즈에 안정 탑재했습니다." — 단독이라는 사실이 먼저 읽힙니다.

### Before 5 — 숫자가 있지만 맥락이 없는 문장

> "단일 모듈 기준 약 185 회 커밋을 진행하며 Jenkins 통합 전 구간을 직접 다뤘습니다."

- **문제**: 185회는 많은 것처럼 보이지만 기간과 대비 기준이 없어 독자가 무게를 가늠하기 어렵습니다. 커밋 수보다 그 커밋이 다룬 범위가 더 강한 신호입니다.
- **개선 방향**: 커밋 수 대신 담당 범위를 구체화합니다. 예: "접수·실행 요청·상태 추적·실패 복구·취소까지 6단계 전 구간을 단독으로 설계·구현했습니다." — 커밋 수를 지우고 단계 범위를 명시하면 담당 폭이 더 선명합니다.
