**티켓·워크플로우 시스템**

기준점: 요구사항은 한 시스템에서 흐르고, 모든 상태 전이는 감사 가능해야 한다.

- 문제: Jira·내부 PMS·엑셀로 요구사항이 흩어져 있어 동일 티켓이 시스템마다 다른 상태로 존재했습니다. 변경관리 감사 시 상태 불일치가 위험 요소였습니다.
- 해결: 외부 PMS 와 양방향 동기화하는 연계 로직을 구현해 단일 출처를 확보하고, 연결 티켓 관계와 라이프사이클을 별도 정책으로 분리해 마스터·서브 티켓 흐름을 명확히 했습니다.
- 해결: 상태 전이마다 진행이력과 감사로그를 자동 적재하는 트래킹 구조를 설계해 사후 재구성 가능성을 코드 레벨에서 보장했습니다.
- 결과: 백엔드(MyBatis)와 프런트엔드(React) 전 구간을 혼자 짰습니다. 티켓 조회·검색·연계·이력까지 묶어 3.0.4 릴리즈에 올렸습니다.

**결재관리 시스템**

기준점: 결재는 비즈니스 트랜잭션이며, 외부 시스템 장애가 결재 상태를 오염시켜선 안 된다.

- 문제: 결재가 모두 승인된 뒤 외부 API 호출이 실패하면 "승인됐지만 후속 처리는 안 된" 모호한 상태가 남았습니다. 단일 트랜잭션으로 묶으면 외부 장애가 결재 트랜잭션을 통째로 잠그는 역효과가 생겼습니다.
- 해결: 결재 트랜잭션과 외부 호출을 떼어내고 `WebClient` 호출에 멱등성을 걸었습니다. 외부가 무너져도 결재 기록은 그대로 남고, 실패한 호출만 재시도 큐로 빠집니다.
- 해결: 결재 플로우·대결(代決)·회수 알람을 각각 독립 정책으로 떼어냈습니다. 결재선이 바뀌면 해당 정책만 갈아 끼우면 됩니다.
- 결과: 결재 상태와 외부 호출 상태가 독립적으로 추적되며, Webhook 기반 알람으로 모든 상태 변화가 담당자에게 자동 전달됩니다.

**통합·인수테스트 시나리오 관리 시스템**

기준점: 변경관리에서 테스트는 별도 산출물이 아니라 티켓 흐름의 한 단계다. 테스트 의뢰부터 결과까지 같은 감사 추적선 위에 있어야 한다.

- 문제: 통합·인수테스트의 의뢰와 결과가 티켓 외부에서 관리되어, 어떤 변경에 어떤 테스트가 통과했는지 사후에 잇기 어려웠습니다. 금융권 감사에서는 테스트 증빙이 변경 이력과 끊기면 안 됩니다.
- 해결: 테스트 시나리오 그룹과 시나리오를 별도 도메인으로 떼어 조회·검색·드래그 정렬·불러오기를 짰고, 시나리오 테이블을 영속 계층에 새로 설계했습니다. 시나리오를 만들고 고치고 불러올 때마다 감사로그가 자동으로 쌓입니다.
- 해결: 통합테스트 의뢰/결과를 티켓 진행이력에 연계하고, JUnit·정적테스트 결과와 증빙 파일(파일 업로드)을 시나리오에 묶어 테스트 증빙을 변경 이력과 단일 추적선으로 연결했습니다.
- 결과: 테스트 관리 전 구간을 백엔드(workflow-api)·영속 계층(core-persistence)·프런트엔드에서 혼자 짰습니다. 작업 도중 레거시 폼(formik)을 react-hook-form 으로 옮겨 검증 방식을 하나로 통일했습니다.

**Skill Keywords**: Spring Boot, MyBatis, WebClient, React 18, TypeScript, MariaDB, Webhook, Minio, JUnit, react-hook-form

- 해결: 티켓 내부에서 결재를 처리할 수 있는 프로세스 흐름을 신규 구축하고, 결재 승인·반려 결과에 따라 티켓 상태가 자동 전이되도록 두 도메인 상태 머신을 동기화했습니다.
- 해결: 티켓-결재 매핑별 이력과 감사로그를 별도 적재해 어느 도메인의 변경이 다른 도메인의 상태를 바꿨는지 추적 가능하게 했습니다.

**티켓 v3 프론트엔드 재아키텍처**

기준점: 화면이 늘수록 상태가 흩어진다. 변경관리처럼 단계가 많은 화면은 상태 관리 구조 자체를 다시 잡아야 한다.

- 문제: 3.0.4 의 티켓 화면은 컴포넌트별로 상태가 산재해 단계 간 데이터 정합이 어려웠고, 백엔드 API 도 v3 로 재편되면서 프런트 연동을 전면 교체해야 했습니다.
- 해결: 변경관리(CM) 화면 상태를 Context 기반으로 재설계해 단계 간 데이터를 단일 소스로 묶고, 티켓 v3 API 로 데이터 계층을 마이그레이션했습니다. 나의 할일 화면도 v3 로 재구축했습니다.
- 해결: 워크플로우 컴포넌트를 그룹화해 화면을 개편하고, 변경관리 페이지의 UX 흐름을 단계별로 재설계했습니다.

**감사 보고서·UX 고도화**

기준점: 복잡한 워크플로우일수록 한 화면에서 전체 흐름이 보여야 하고, 감사 자료는 자동으로 나와야 한다.

- 문제: 단계가 많은 워크플로우는 진행 상태를 한눈에 파악하기 어려웠고, 외부 감사 자료를 수기로 정리하는 비용이 컸습니다.
- 해결: 티켓 보고서와 티켓 감사보고서 화면을 구현하고 다운로드 기능을 추가해 외부 감사 자료 생성을 자동화했습니다.

**결재 회복·자동 만료**

기준점: 외부 시스템·사용자·시간 모두 결재 흐름을 멈출 수 있다. 세 경우 모두 자동 회복 경로가 있어야 한다.

#### Jenkins 파이프라인 실행 엔진 신규 구축 (2024 ~ 진행 중)

> 외부 시스템(Jenkins)에 강하게 의존하는 모듈을 설계할 때 "외부 장애는 내부 트랜잭션을 잠그면 안 된다", "동시성은 식별 단위에서 봉합한다" 두 원칙을 끝까지 끌고 갔습니다. 접수부터 실행·추적·복구·취소까지 여섯 단계를 혼자 설계하고 구현했습니다.

**Jenkins 호출 추상화와 인증 통합**

기준점: 외부 호출 표면은 좁고 일관돼야 한다. 인증·경로·파라미터 처리 방식이 호출 지점마다 다르면 장기적으로 운영 부채가 된다.

- 문제: Jenkins REST API 는 Crumb 인증(CSRF 토큰 + 세션 쿠키)이 필요한데, Feign 은 GET 위주이고 POST 호출은 별도 처리가 필요했습니다. Folder Plugin 계층 경로도 호출마다 다르게 작성돼 일관성이 떨어졌습니다.
- 해결: GET 은 Feign, POST 는 `RestTemplate` 으로 역할을 분리하되 인증·경로 처리는 단일 추상화로 묶었습니다. 호출 지점에서는 인증 디테일을 모릅니다.
- 해결: Folder Plugin 계층 경로(`/job/{folder}/job/{name}/...`)와 Job 빌드 파라미터에서 외부 컨텍스트를 추출하는 어댑터를 별도 모듈로 분리해 다단계 폴더 구조의 Job 조회·생성·이관을 표준화했습니다.
- 해결: 동적 에이전트 환경에서 정적 executor 카운트가 0 으로 보이는 케이스를 K8s 클러스터 상태로 보강 감지해 폴링 누락을 방지했습니다.

**Job 라이프사이클 신규 구축**

기준점: 외부 시스템에 대한 동기 호출은 내부 트랜잭션을 잠그면 안 된다. 외부가 멈춰도 내부는 계속 돌아야 한다.

- 문제: 기존 v3 는 동기 HTTP 호출이 외부 Jenkins 장애를 그대로 전파해 내부 트랜잭션이 함께 멈추는 구조였습니다. 한 인스턴스가 응답하지 않으면 전체 워크플로우가 정체됐습니다.
- 해결: Job 의 접수·실행 요청·실행 확인·상태 추적·실패 복구·취소까지 여섯 단계를 메시지 브로커 기반 비동기 흐름으로 재설계했습니다. 외부 호출은 트랜잭션 경계 밖으로 분리됩니다.
- 해결: Jenkins 가 빌드 번호를 큐 진입이 아니라 executor 할당 시점에 부여한다는 점을 반영해, 실행 레코드 매칭 키를 빌드 번호에서 큐 아이템 ID 로 전환했습니다. 트리거 직후 응답의 큐 아이템을 1 차 키로 잡고, 큐 상태를 조회해 빌드 번호가 나타나면 보강하는 2 단계 추적으로 동시 트리거 시 매칭이 깨지는 문제를 해결했습니다.
- 해결: Jenkins 빌드 콜백을 webhook 으로 수신하되 5 분 폴링 fallback 을 함께 두어 webhook 유실에도 상태 추적이 끊기지 않게 했습니다.
- 해결: Jenkins 가 24 시간 이상 응답하지 않는 경우를 대비한 타임아웃 안전망을 도입해 좀비 Job 을 자동 종료 상태로 전이시킵니다. 외부 장애가 시스템 자원을 무한히 점유하지 않습니다.
- 해결: 헥사고날 아키텍처로 도메인 계층을 분리해 영속성·메시지 브로커 의존을 도메인 밖으로 밀어냈습니다. 도메인 단위 테스트는 외부 의존 0 으로 실행됩니다.

**동시성 충돌 해결: Jenkins 인스턴스 식별 그룹화**

기준점: 동시성 충돌은 항상 식별 단위의 문제다. 충돌의 진짜 단위(물리 Jenkins 인스턴스)와 다른 단위(논리 그룹)로 처리량 한도를 산출하면 충돌이 재발한다.

- 문제: 같은 Jenkins 인스턴스를 가리키는 서로 다른 두 논리 그룹이 별도 흐름으로 처리되면서 해당 Jenkins 인스턴스의 대기 큐가 정원을 초과했습니다. 논리 그룹 단위로 한도를 계산하니 물리 인스턴스에서는 합산 한도가 무력화됐습니다.
- 해결: Jenkins 응답 헤더(`X-Jenkins`)와 노드 이름 지문으로 인스턴스 동일성을 식별하는 값 객체(Value Object)를 도입했습니다. 헤더 노출이 없는 환경에서는 호스트·포트 조합으로 보조 식별합니다.
- 해결: 디스패치 단계의 그룹화를 1 차(논리 그룹) → 2 차(물리 Jenkins 인스턴스) 두 단계로 확장해 같은 인스턴스에 속한 모든 작업의 대기 적재량을 합산하여 처리량 한도를 산출하도록 변경했습니다.
- 결과: 같은 인스턴스에 두 그룹이 동시에 작업을 밀어넣어도 큐 정원은 한 번도 넘치지 않았습니다. 신규 통합 시나리오 2 건과 QA 자동화 4 건이 모두 통과했습니다(약 392 초).

**다중 인스턴스 환경 디스패치 직렬화**

기준점: 분산 환경에서 한도 계산은 직렬화돼야 한다. 한도라는 개념 자체가 단일 진리값을 전제로 한다.

- 문제: 동일한 작업 그룹을 여러 애플리케이션 인스턴스가 동시에 디스패치할 때 한도 계산이 비결정적이었습니다. 각 인스턴스가 자신의 시점에서만 한도를 보니 합산이 어긋났습니다.
- 해결: 디스패치 게이트에 비관적 락(SELECT ... FOR UPDATE)을 도입해 같은 그룹의 디스패치가 한 애플리케이션 인스턴스에서만 진행되도록 직렬화했습니다. 락의 적용 범위를 그룹 단위로 좁혀 처리량 손실을 최소화했습니다.
- 해결: 동시성 제어 정책 문서를 신규 작성해 낙관적 락, 비관적 락, 공유 락의 적용 책임을 분리하고 팀 내 의사결정 기준을 명문화했습니다. 새 동시성 이슈가 나올 때마다 같은 논의를 반복하지 않도록 했습니다.
- 결과: 남아 있던 동시성 충돌이 더는 재현되지 않았습니다. 단위 테스트 34 건과 외부 호출을 가상화(WireMock)한 통합 테스트가 모두 통과했습니다.

**Skill Keywords**: Java 17, Spring Boot, Jenkins REST API, Kafka, Redpanda, Avro, Hexagonal Architecture, Pessimistic Locking, WireMock, JUnit5



#### Jenkins 멀티 인스턴스 운영 표준화 (2024 ~ 진행 중)

> 외부 시스템에 강하게 의존하는 모듈은 운영 자체가 코드와 함께 형상관리되어야 합니다. 인스턴스 추가·플러그인 설치·콜백 등록이 수기 작업으로 처리되면 신규 인스턴스 합류 비용이 사람에 의존하게 됩니다.

**컨트롤러·플러그인·콜백 표준화**

기준점: 인프라 구성은 코드와 함께 형상관리되어야 한다.

- 문제: 다중 Jenkins 인스턴스가 늘면서 Job 이관·플러그인 설치·webhook 등록이 인스턴스마다 수기 작업으로 처리됐습니다. 신규 인스턴스를 합류시킬 때마다 사람에 의존한 절차가 반복됐습니다.
- 해결: Jenkins 컨트롤러를 Helm 차트로 K8s 에 배포하고, 동적 에이전트(Kubernetes Plugin) 환경과 `cloudbees-folder` 플러그인을 LTS 기본 이미지에 명시 설치해 컨트롤러 자체를 코드로 형상관리했습니다. `cloudbees-folder` 의 namespace-aware 특성을 활용해 폴더 단위로 Job 경로를 격리하고 자격증명을 폴더 레벨로 스코핑했습니다.
- 해결: Jenkins Pipeline 의 webhook 발행 스크립트(`webhook-listener.groovy`) 를 작성해 Job 완료 이벤트가 메시지 브로커로 직접 발행되도록 표준화했습니다. 인스턴스마다 콜백 수동 등록 작업이 사라졌습니다.
- 해결: Job 이관 파이프라인(`config.xml` 추출 → 폴더 생성 → 재전송) 을 자동화해 인스턴스 간 Job 이관을 단일 명령으로 처리합니다.
- 결과: 신규 Jenkins 인스턴스 합류와 Job 이관 절차가 코드 기반 자동화로 이동했습니다. 정량 메트릭은 사내 데이터로 추후 보강 예정입니다.

**Skill Keywords**: Jenkins, Helm, Kubernetes Plugin, cloudbees-folder, Groovy, Webhook, GitOps



#### TPS 관측성·GitOps 인프라 (2026, 진행 중)

> 개인 GCP 클러스터에서 먼저 검증한 관측성·GitOps 패턴을 사내 운영 환경(305P)에 옮기는 작업입니다. 사이드 프로젝트가 본업 인프라 의사결정의 위험을 미리 흡수하는 구조를 의도했습니다.

**관측성 스택 통합**

기준점: 관측성은 단일 진입점에서 봐야 한다. 메트릭과 로그가 서로 다른 Grafana 에 흩어지면 장애 대응 시 화면을 오가게 된다.

- 문제: Grafana 가 loki-stack 과 kube-prometheus-stack 두 곳에서 따로 떠 있어 대시보드와 데이터소스가 분산됐고, 로그 수집기 Promtail 이 EOL 되어 교체가 필요했습니다.
- 해결: loki-stack 의 Grafana 를 비활성화하고 kube-prometheus-stack 의 Grafana 에 Loki datasource 를 붙여 로그·메트릭을 한 화면으로 통합했습니다. Promtail 을 Grafana Alloy 로 전환해 ArgoCD Application 으로 등록하고, readOnlyRootFilesystem 환경의 `/tmp` 쓰기 오류를 해결했습니다.
- 결과: 이미지 태그·차트 버전을 클러스터 실버전과 정합하고, 폐쇄망 대비 Nexus 프록시 레지스트리 설정을 적용해 관측성 스택을 GitOps 로 형상관리했습니다.

**305P Jenkins app-of-apps GitOps 부트스트랩**

기준점: Jenkins 컨트롤러 자체가 코드로 형상관리되어야 신규 인스턴스 합류가 사람에 묶이지 않는다.

- 문제: 운영 환경(PPP)의 Jenkins 컨트롤러·플러그인·콜백이 수기 절차로 관리됐고, 폐쇄망 제약과 ArgoCD sync 충돌이 반복됐습니다.
- 해결: trb-oss app-of-apps 부트스트랩과 Jenkins 컨트롤러 StatefulSet 을 한 번에 구성하고, Job 완료 이벤트를 메시지 브로커로 발행하는 webhook listener(Groovy)를 자동 설치하도록 묶었습니다. 폐쇄망 대비 플러그인 해석 고정·Update Center 비활성, PVC storageClass 보존·releaseName 고정 등 ArgoCD sync 안정화 가드레일을 적용했습니다.
- 결과: 운영 대시보드도 코드로 옮겼습니다. operator/executor 의 ERROR 로그 대시보드(LogQL)와 message-lib Outbox 메트릭 대시보드를 Helm 차트로 작성해 Grafana sidecar 자동 import 로 관리합니다.

**Skill Keywords**: ArgoCD, GitOps, Grafana, Loki, Alloy, Prometheus, Helm, Jenkins, Groovy, LogQL, Kubernetes
