## 매칭 분석 (vs RESUME.md)

**강점 (이 공고에서 어필 가능한 내 마일스톤)**
- **Java + Spring Boot 풀스택**: RESUME §2 Skills "Java 17·21, Spring Boot 3.x" 와 필수 정확 매칭
- **Event 기반 서비스 운영**: 트럼본 실행 엔진의 헥사고날 + Kafka·Avro 비동기 흐름 — 우대사항 "Event 기반 서비스 운영" 정확 매칭. 면접에서 인스턴스 식별 그룹화 + 비관적 락 직렬화 사례로 깊이 어필 가능
- **외부 시스템 연동 + 트랜잭션 격리**: 결재 트랜잭션과 외부 호출 분리 사례(결재 도메인 47회 커밋) → PG 연동의 "결재 상태 오염 방지" 문제와 직접 대응. 거의 그대로 면접 답변 가능
- **MSA/컨테이너**: 개인 GCP 3노드 K8s 클러스터 운영(Side Project) + CKA 인증 → 필수 "MSA 운영 경험" + 우대 "K8S 이행 경험" 동시 충족
- **AWS**: SAA·CCP 보유 → 우대 "AWS 개발·상용 운영" 부분 충족 (단, 상용 운영은 GCP 위주라 보완 필요)
- **결재선 자동화 + 감사 추적**: 금융권 변경관리 경험 → 결제 시스템의 트랜잭션 무결성·감사 요구와 사고 방식이 일치

**공백 (보강하거나 학습 계획을 세워야 할 부분)**
- **PG 연동 직접 경험 없음** — Jenkins 외부 시스템 연동 경험을 "외부 시스템 호출 추상화 + 멱등 보장"으로 일반화해 설명. 면접 전에 PG 표준 인터페이스(나이스페이먼츠/토스페이먼츠 API 문서) 1회 훑기 권장
- **간편결제 시스템 경험 없음** — 동일 전략으로 추상화 설명, 또는 사이드 프로젝트로 작은 결제 모듈(테스트 결제) 추가 검토
- **대용량 트래픽 처리 정량 메트릭 부재** — RESUME §3에 트래픽 수치(MAU/TPS) 없음. 트럼본의 큐 정원 초과 0 건 외에 "동시 사용자 N명 처리" 같은 수치 보강 검토
- **E-Commerce·O2O 도메인 직접 경험 없음** — RESUME §5 슬롯에 "예약/결제 도메인 한 줄 정의 + 예상 기술부채" 작성으로 우회
- **상용 AWS 운영 경험**: 자격증만 있고 실 운영은 GCP. "GCP K8s 운영을 AWS EKS로 그대로 이전 가능" 한 줄 답변 준비

## 도메인 한 줄 + 예상 기술부채

**도메인**: 숙박·여행 O2O 플랫폼의 예약·결제 트랜잭션 인프라. 다양한 PG·결제수단 연동 + 안정적 거래 경험이 핵심 비즈니스 가치.

**예상 기술부채 3개**
1. **PG 연동 코드 중복·표준화 부재** — 결제수단별 어댑터가 PG마다 따로 작성됐을 가능성. 단일 추상화 + 멱등 패턴으로 정리 가능 (트럼본 Jenkins 통합 사례 그대로 적용)
2. **레거시 모놀리식 + MSA 혼재** — MSA "기반"으로 표현했지만 결제 트랜잭션은 단일 DB 잠금이 남아있을 가능성. 결재 트랜잭션·외부 호출 분리 패턴 적용 가능
3. **장애 대응 자동화 부족** — "장애 없는 거래 경험"이 미션으로 명시 = 현재 장애 처리에 수기 작업이 남아있다는 신호. 자동 재시도·자동 보상·webhook fallback 패턴 도입 가능
