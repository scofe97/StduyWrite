## 매칭 분석 (vs 지원자)
**강점 (어필 가능한 마일스톤)**
- **결재(변경관리) 도메인 ↔ 가맹점 정산 인접성**: 트럼본 결재·티켓의 상태 머신·감사 추적이 "가맹점 정산 개발·운영"과 정합. 정산 정확성·이력 추적 사고를 직접 어필
- **Kafka 실무 경험 = 우대 직접 매칭**: 카카오페이는 "유연한 대용량 정산 데이터 처리를 위해 Kafka 활용". 트럼본 Kafka/Redpanda + Avro + 이벤트 드리븐 경험이 우대 스택과 정확 매칭 — 5건 중 본인 Kafka 강점이 가장 직접적으로 살아나는 공고
- **Spring Boot 3.x + MySQL 모델링·튜닝**: 필수 정확 매칭
- **TDD 지향**: 본인 GWT 기반 테스트 작성 습관 + given-when-then 컨벤션으로 정합
- **동시성 직렬화 + 분산 트랜잭션**: 정산 정합성(이중 정산 방지)에 비관적 락·이벤트 처리 경험 정합
**공백**
- **Kotlin 주 언어 부담** — 정산 플랫폼 주 언어가 Kotlin. JVM 동일하나 실무 비중이 높아 코루틴·DSL·null 안전성 학습 필수
- **MongoDB·Redis 실무 경험 부족** — Redis는 알려진 공백, MongoDB도 무경험. 정산 데이터 스토어로 쓰이므로 학습 권장
- **대용량 정산 트래픽 정량 메트릭 부재** — "대용량 정산 데이터"에 비해 정량 부하 경험 약함
**매칭 신뢰도**: 상 — Kafka·이벤트 드리븐·결재 도메인 인접성이 카카오페이 정산 플랫폼과 강하게 정합. Kotlin·Redis/MongoDB 공백만 보강하면 5건 중 도메인+스택 동시 매칭이 가장 높은 후보

## 도메인 한 줄 + 예상 기술부채
**도메인**: 간편결제·송금·청구서·멤버십을 제공하는 핀테크. 정산 플랫폼은 가맹점 정산을 Kafka 기반 대용량 데이터로 처리하고 자동화·아키텍처를 개선.
**예상 기술부채 2~3개**
1. **정산 정합성·재처리(이중/누락 정산)** — 대용량 비동기 정산에서 메시지 중복·유실 시 금전 사고. 멱등 컨슈머 + 정산 체크포인트 + dead-letter 재처리 패턴 적용 가능 (Kafka 멱등 경험 전이)
2. **정산 운영 수작업 비용** — "운영 비용 최소화" 명시 = 수동 정산 보정 업무 잔존 추정. 자동 대사 + 예외 자동 분류 + 셀프힐링 배치 적용 가능
3. **이종 데이터 스토어(MongoDB/MySQL/Redis) 정합** — 스토어 간 정산 상태 불일치 위험. 단일 source of truth + 이벤트 기반 동기화 패턴 적용 가능
