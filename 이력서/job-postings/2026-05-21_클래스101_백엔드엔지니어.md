## 매칭 분석 (vs RESUME.md)

**강점 (이 공고에서 어필 가능한 내 마일스톤)**
- **데이터 모델·API 설계 주도 경험**: 트럼본 본체 도메인을 단독 신규 구축(결재 47회·티켓 24회 커밋)하면서 외부 PMS 양방향 동기화 단일 출처 확보 → 자격 "비즈니스 요건 이해 + 데이터 모델·API 설계" 직접 매칭
- **Java + Spring Boot 풀스택**: RESUME §2 Skills "Java 17·21, Spring Boot 3.x" 와 필수 정확 매칭
- **MSA 개발·운영**: 헥사고날 + Kafka·Avro 비동기 흐름 신규 구축 → 우대 "MSA 개발·운영" 직접 매칭
- **AWS/GCP 인프라 구축**: 개인 GCP 3노드 K8s 클러스터 자가호스팅 + CKA → 우대 "AWS/GCP 인프라 구축" 직접 매칭 (GCP는 직접 구축, AWS는 자격증)
- **결제 도메인 사고 회로 정합**: 외부 호출과 비즈니스 트랜잭션 분리 + 멱등 보장 + 결재 회복 자동화 → 우대 "커머스 개발·결제 연동" 의 결제 무결성 사고와 일치 (VAN 직접 경험은 없으나 외부 시스템 격리 패턴은 그대로 적용 가능)
- **DRI·주도성 컬처 정합**: 트럼본 두 트랙 단독 담당 + 동시성 제어 정책 문서 신규 작성 → "DRI 문화 (Last Man으로 책임)" 컬처와 정합
- **챕터 멘토링 환경 적합**: 정책 문서화 + 도메인 안정화 커밋 활동 → 챕터 리드의 코드 리뷰·기술 멘토링 문화에서 빠르게 성장 가능

**공백 (보강하거나 학습 계획을 세워야 할 부분)**
- **Kotlin 직접 경험 없음** — RESUME.md는 Java 17·21만 노출. Kotlin은 JVM 기반이라 러닝 커브 작음. Spring Boot 모듈 1개 Kotlin 컨버전 PoC 권장 (이미 §14 권고로 누적)
- **Node.js 직접 경험 없음** — 필수 스택 중 하나. Java/JVM 깊이로 어필 + "필요시 학습" 답변 준비
- **MongoDB 운영 경험 부재** — RESUME.md는 MariaDB만 노출. NoSQL/문서 DB 학습 + RDBMS와의 트랜잭션 모델 차이 정리
- **Redis 운영 경험 부족** — 이미 §14 권고로 누적 (퀸잇과 동일). Side Project에 Redis 추가 우선순위 상승
- **컨슈머 대용량 트래픽 정량 메트릭 부재** — 이미 §14 권고로 누적 (4건 연속). 트래픽 처리량·동시 사용자 수치 보강 우선순위 1위로 굳어짐
- **VAN 연동 직접 경험 없음** — Jenkins 외부 시스템 통합 경험을 "외부 시스템 호출 추상화 + 멱등 보장"으로 일반화 (여기어때와 동일 전략)
- **ffmpeg·동영상 인코딩 경험 없음** — Product Lifecycle 팀 지원 시 약점. Payment & Payout 또는 Auth & Account 팀 지원이 매칭 더 강함

## 도메인 한 줄 + 예상 기술부채

**도메인**: 크리에이터-수강생 매칭 + VOD/구독/커뮤니티 + 결제·정산 통합 에듀테크 플랫폼. 누적 550만+ 계정 + 다양한 상품 형태(VOD·멤버십·구독·키트·전자책·프로그램) 카탈로그.

**예상 기술부채 3개**
1. **다형 상품 카탈로그 모델링 복잡도** — Product Lifecycle 팀 신설 명시 = 상품 형태가 늘면서 카탈로그 데이터 모델이 응집력을 잃은 단계. 도메인 분리 + 이벤트 기반 카탈로그 변경 전파 패턴 도입 가능
2. **결제·정산 트랜잭션 외부 시스템 의존 누적** — Payment & Payout 팀이 정산·결제 전반을 책임 = VAN·PG·국세청·세무 시스템 등 외부 의존이 누적됐을 가능성. 외부 호출 트랜잭션 분리 + 멱등 보장 + 보상 트랜잭션 패턴 도입 가능
3. **다언어 백엔드(Java/Kotlin/Node.js) 운영 부담** — 표준화·관찰성·배포 도구 통일 비용. OpenTelemetry + 통합 로깅 + 공통 라이브러리 패턴 도입 가능
