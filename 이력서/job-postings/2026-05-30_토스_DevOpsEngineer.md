## 매칭 분석 (vs 지원자)
**강점 (어필 가능한 마일스톤)**
- **프로덕션 Kubernetes 운영 + CKA**: 개인 GCP 3노드 K8s 자가호스팅 → 필수 "프로덕션 K8s 운영·트러블슈팅" 직접 매칭
- **분산 시스템 체계적 해결**: 분산 트랜잭션·동시성 직렬화(비관적 락)·이벤트드리븐·Kafka/Redpanda → "복잡한 분산 시스템 이슈" 필수 정합
- **메트릭/모니터링 구축**: Grafana·Loki·Tempo·Prometheus 자가 관찰성 → "메트릭 시스템 구축·운영" 직접 매칭
- **CI/CD 서버 운영**: Jenkins 멀티 인스턴스 표준화 + Helm·ArgoCD 배포 → "컨테이너 CI/CD 서버 운영" 매칭
- **HA 아키텍처 사고**: 동시성 직렬화·분산 트랜잭션 경험 → "고가용성 아키텍처 구현" 사고 회로 정합

**공백**
- **Terraform 인프라 자동화 직접 경험 부족** (알려진 공백 최우선) — 자격요건 핵심인데 본인은 수기 kubeadm·Helm 중심. 가장 보완 시급한 갭
- **Istio(서비스 메시) 운영 경험 없음** — 필수 스택. K8s 네트워킹 이해는 있으나 Istio 직접 운영 미경험. 학습 필요
- **AWS 상용 운영 경험 부재** (자격증만) — 토스는 AWS 기반. GCP 자가호스팅 경험을 AWS로 전이 학습 필요
- **대규모 분산 메트릭 정량 경험 제한** — "대규모 분산 환경" 규모는 미경험. 개인 클러스터·금융 변경관리로 우회

**매칭 신뢰도**: 중상 — K8s·분산 시스템·메트릭·CI/CD는 강하게 매칭되나, 핵심 자격요건인 Terraform·Istio·AWS 3종이 모두 알려진 공백과 겹침. 핀테크 도메인(변경관리·결재) 친화성은 플러스.

## 도메인 한 줄 + 예상 기술부채
**도메인**: 핀테크 슈퍼앱 토스의 K8s·Istio 기반 서비스 클러스터 운영과 배포·관찰성 자동화로 개발 생산성·안정성을 높이는 DevOps.

**예상 기술부채 2~3개**
1. **수많은 마이크로서비스의 배포 표면 폭증** — 서비스 수가 많아 배포·롤백 표준이 흩어지면 사고 위험. 본인 Jenkins 표준화 + Helm/ArgoCD 경험으로 배포 혁신 기여 가능
2. **Istio 메시 복잡도·관측 비용** — 서비스 메시가 트래픽 라우팅·mTLS·관측을 떠안으며 운영 부담 누적. 본인 관찰성(Prometheus·Tempo) 경험으로 메시 텔레메트리 표준화 학습·기여
3. **Terraform state 드리프트·금융 규제 추적** — 핀테크 규제상 인프라 변경 감사 추적 필수. 본인 금융 변경관리·결재선 자동화 도메인 경험을 IaC 변경 거버넌스로 연결 가능
