## 매칭 분석 (vs 지원자)
**강점 (어필 가능한 마일스톤)**
- **Kotlin/Java 11+ + Spring Boot + JPA**: 필수 스택 핵심 3개 정확 매칭. Java 17/21 경험은 Java 11+ 하한 상회
- **RDBMS(MySQL) + 도메인 모델링**: 트럼본 결재·티켓 도메인을 MyBatis/JPA로 구현 → MySQL·JPA 요건 충족
- **클라우드 + Linux/Unix**: GCP 3노드 K8s 자가호스팅 → 클라우드·Linux 요건 강하게 충족 (우대 클라우드 경험도 충족)
- **백오피스 시스템 구축**: 트럼본 백오피스 단독 구축 → "백오피스 시스템 개발·운영" 정합
- **외부 연동(Open-API)**: Jenkins/Git provider 연동 등 외부 시스템 연동 경험과 구조적 유사
- **경력 1~10년 하한**: 3년차로 충족

**공백**
- **NoSQL·InMemoryStore 운영 경험 부족** — Redis 미경험(알려진 공백 5순위)과 직결. 카탈로그 캐시·재고 동시성에서 자주 요구
- **Node.js 미경험** (우대, JVM 외 언어 시급도 하향) — 필수 아님이라 영향 작음
- **B2C 패션 커머스(카탈로그·가격·재고) 도메인 경험 없음**

**매칭 신뢰도**: 상 — 필수 스택 대부분이 핵심 강점과 매칭되고 경력 하한도 넉넉. NoSQL/InMemoryStore만 보완 포인트

## 도메인 한 줄 + 예상 기술부채
**도메인**: 지그재그(카카오스타일) — 패션·뷰티 커머스 플랫폼. 카탈로그·카테고리·재고·가격·QnA 상품 도메인
**예상 기술부채 2~3개**
1. **재고(Inventory)·가격(Pricing) 동시성 정합성** — 다수 판매자·실시간 재고 차감에서 초과 판매 위험. 비관적 락 직렬화 경험으로 재고 차감 게이트 직렬화 가능
2. **카탈로그 ↔ 검색/노출 캐시 정합성(InMemoryStore)** — 상품 변경이 캐시에 늦게 반영되는 stale 문제. 도메인 이벤트 + 캐시 무효화 전략 적용 가능
3. **Open-API 외부 상품 연동의 멱등성·실패 복원** — 외부 연동 중복/유실 위험. Outbox + 멱등 키로 보강 가능 (Redis 학습이 전제)
