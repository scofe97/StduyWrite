## 매칭 분석 (vs RESUME.md)

**강점 (이 공고에서 어필 가능한 내 마일스톤)**
- **주도적 설계·구현 경험**: 트럼본 본체(티켓·워크플로우·결재) + 실행 엔진 모두 신규 구축 단독 담당 → "제품을 주도적으로 설계·구현" 우대 정확 매칭
- **분산 트랜잭션·트랜잭션 격리**: 결재 트랜잭션과 외부 호출 분리 + 인스턴스 식별 그룹화 + 비관적 락 직렬화 → "분산 환경 트랜잭션 처리" 우대 직접 매칭. 특히 **Commerce Platform Team** 핏에 매우 강력
- **RDBMS 성능·트랜잭션 이해**: 결재 47회·티켓 24회·실행 엔진 185회 커밋으로 도메인 안정화 + SELECT FOR UPDATE 비관적 락 정책 문서화 → 필수 "RDBMS 성능과 트랜잭션 모두 고려" 정확 매칭
- **이벤트 스트림 기반 아키텍처**: 트럼본 실행 엔진의 헥사고날 + Kafka·Avro 비동기 흐름 → **Product Platform Team**의 "이벤트 스트림 기반 아키텍처" 정확 매칭
- **도메인 구조 주도 설계**: 동시성 제어 정책 문서 신규 작성으로 팀 내 의사결정 기준 명문화 → "도메인 구조·시스템 설계 주도" 자격 매칭

**공백 (보강하거나 학습 계획을 세워야 할 부분)**
- **Kotlin 직접 경험 없음** — RESUME.md는 Java 17·21만 노출. Kotlin은 JVM 기반이라 러닝 커브 작음. 면접 전 코틀린 공식 문서 + Coroutines 핵심 1회 훑기 권장. "Java에서 Kotlin으로 점진 마이그레이션 PoC" 같은 답변 준비
- **커머스 도메인 직접 경험 없음** — 트럼본은 B2B DevSecOps라 커머스(주문·결제·쿠폰·정산) 도메인 모델링 직접 경험 없음. 결재·정산 흐름이 결제 도메인과 사고가 유사하다는 점으로 우회 + 우대사항 "도메인 오너십" 어필
- **Elasticsearch 직접 경험 없음** — Search Squad 지원 시 약점. Commerce Platform Team이나 Product Platform Team이 매칭 더 강함
- **Redis 운영 경험 부족** — Side Project(GCP K8s)에 Redis 추가 검토. 또는 캐싱 패턴 학습으로 우회
- **퀸잇·라포랩스 도메인 한 줄 정의 필요** — RESUME §5 슬롯에 "30대 여성 패션 커머스의 트래픽·재고·정산" 한 줄 작성 권장
- **컨슈머 트래픽 정량 메트릭 부재** — B2B 금융권 경험이라 컨슈머 트래픽 수치 없음. 큐 정원 초과 0 건 + 동시 디스패치 직렬화로 "동시성 압력 하 무결성" 으로 우회
