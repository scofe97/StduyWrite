## 매칭 분석 (vs 지원자)
**강점 (어필 가능한 마일스톤)**
- **Java/Kotlin + Spring Boot/Webflux**: 기술 스택 정면 매칭. 지원자 Java 17/21 + Spring Boot 3.x 주력과 일치. Kotlin은 JVM 기반이라 러닝 커브 작음
- **Kubernetes(CKA) + AWS**: 스택에 K8s·AWS 명시. 지원자의 개인 GCP 3노드 K8s 자가호스팅 + CKA가 "지속 확장 가능한 제품" 업무에 직접 어필 (AWS는 자격증 수준이나 K8s 실경험으로 보강)
- **헥사고날 + 아키텍처 설계**: "아키텍처 설계 및 지속적 확장" 업무에 트럼본 실행 엔진을 헥사고날로 단독 설계한 경험 어필
- **운영 솔루션·B2B SaaS 도메인 친화**: 트럼본 백오피스·결재(변경관리) 단독 구축 경험이 "병원 운영 체계·고객 관리 제품"의 운영 도구 개발 감각과 정합

**공백**
- **경력 5년 하한 vs 3년차** — 정량 경력 부족. 단독 구축 사례·아키텍처 깊이로 보완 어필
- **MongoDB 운영 경험 부족** — 스택에 MongoDB 명시. 지원자는 RDBMS 중심(MariaDB/MySQL). 문서 DB 학습 필요
- **미용 의료(헬스케어) 도메인 경험 없음** — B2B 병원 SaaS 특유 도메인. 일반화로 어필하되 격차 인정
- **Spring Webflux(리액티브) 직접 경험 미확인** — MVC 중심일 가능성. 리액티브 스택 학습 보강 검토

**매칭 신뢰도**: 중 — Java/Kotlin/Spring Boot + K8s가 강하게 일치하나, 경력 5년 하한·MongoDB·Webflux 리액티브가 변수

## 도메인 한 줄 + 예상 기술부채
**도메인**: 글로벌 미용·의료 정보 플랫폼 강남언니. C(시술 정보·예약)와 B2B SaaS(병원 운영·고객 관리 솔루션)를 함께 운영하는 헬스케어 플랫폼.

**예상 기술부채 2~3개**
1. **병원 운영 SaaS의 멀티테넌시 데이터 격리** — B2B SaaS는 병원별 데이터 격리·권한이 핵심. 지원자의 도메인 경계 설계(헥사고날) + 권한 모델 경험으로 테넌트 격리를 설계할 수 있음
2. **예약·고객 관리 상태 정합성** — 예약·시술·상담 상태 전이가 복잡. 비관적 락 동시성 직렬화·분산 트랜잭션 경험으로 정합성 가드를 적용 가능
3. **MVC/Webflux·MySQL/MongoDB 혼재 스택 일관성** — 리액티브·논블로킹과 동기 스택, RDB·문서 DB가 섞여 부채화. 점진적 경계 정리로 일관성 확보 가능
