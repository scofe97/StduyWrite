## 매칭 분석 (vs 지원자)
**강점 (어필 가능한 마일스톤)**
- **결재(변경관리) 도메인 ↔ 카드결제/정산 인접성**: 트럼본 결재·티켓 상태 머신·감사 추적이 카드결제 승인·정산 흐름과 정합. "안정성·검증 자세"라는 우대 인재상에 직접 부합
- **Java + Spring Boot 3.x 정확 매칭**: 필수 스택 정합. Kotlin은 학습 보강 필요하나 동일 JVM 생태계
- **분산 트랜잭션·동시성 직렬화**: 카드결제 승인-정산의 정합성 보장에 비관적 락·이벤트 드리븐 경험 정합
- **외부 시스템 연동 추상화 경험**: "카드사 직접 가맹 연동 PG 구축"에, 트럼본 Jenkins·외부 PMS 연동 추상화 경험 전이 가능
**공백**
- **TCP protocol 저수준 통신 경험** — 카드사 연동은 HTTP가 아닌 TCP 소켓·전문(電文) 기반인 경우 다수. 본인은 REST/Kafka 중심으로 저수준 소켓 통신 경험 약함. 학습 필수
- **암호화·금융 보안 실무** — 결제 전문 암호화(예: 키 관리, EMV) 경험 부족. 금융 규제·암호화 1회 정리 권장
- **Kotlin 실무 경험 부족** — 동일 JVM이나 코루틴·문법 보강 권장
**매칭 신뢰도**: 중 — Java/Spring·결재 도메인 인접성은 강하나, TCP 전문 통신 + 결제 암호화라는 카드결제 특유 저수준 요구가 명확한 공백. 면접 전 TCP 전문·암호화 학습 시 중상으로 상승

## 도메인 한 줄 + 예상 기술부채
**도메인**: 네이버 생태계의 카드결제·정산 인프라. 카드사 직접 가맹 연동 PG와 스노우·크림·네이버페이 카드 결제를 담당하는 결제 플랫폼.
**예상 기술부채 2~3개**
1. **카드사별 전문(TCP) 연동 분기 누적** — 카드사마다 전문 규격·세션 관리가 달라 분기 폭증. 프로토콜 어댑터 추상화 + 전문 매핑 테이블화 적용 가능
2. **승인-정산 정합성·재대사(reconciliation)** — 카드 승인과 정산 대사 불일치는 금전 사고. 멱등 + 일별 대사 배치 + 차이 알람 패턴 적용 가능
3. **암호화 키·세션 수명 관리** — 결제 보안 키 회전·세션 누수 위험. 키 관리 추상화 + 회전 자동화 적용 가능 (학습 전제)
