## 매칭 분석 (vs 지원자)
**강점 (어필 가능한 마일스톤)**
- **결재(변경관리) 도메인 ↔ 결제 승인·정산 도메인 인접성**: 트럼본 결재 상태 머신·감사 추적 경험이 결제 "승인·정산 핵심 로직"과 직접 정합. 결재 워크플로우의 상태 전이·멱등 사고를 결제 트랜잭션에 그대로 어필 가능
- **동시성 직렬화(비관적 락) 경험**: "선착순 결제 = 순간 수십 배 트래픽" 요구에 직접 매칭. 트럼본 실행 엔진의 동시성 직렬화·비관적 락 경험이 핵심 셀링 포인트
- **Java 17/21 + Spring Boot 3.x + JPA**: 필수 스택 정확 매칭. Kotlin은 학습 보강 필요하나 JVM 동일 생태계
- **이벤트 드리븐 + Kafka/Redpanda**: 결제·정산 비동기 처리에 정합 (공고에 Kafka 명시는 없으나 대용량 결제 인프라에서 통상 사용)
**공백**
- **Kotlin 실무 경험 부족** — 토스는 Kotlin 비중 높음. JVM 동일하나 코루틴·null 안전성 문법 1회 정리 권장
- **대용량 트래픽 정량 메트릭 부재** — "수십 배 트래픽" 요구에 비해 본인은 정량 부하 지표 경험이 알려진 공백. 개인 GCP 부하 테스트로 보강 권장
- **PG·금융 규제 도메인 무경험** — 결제대행 정산·세금계산서 등 PG 특유 도메인 학습 필요
**매칭 신뢰도**: 상 — 결재 도메인 인접성 + 동시성 직렬화 경험이 토스 핵심 요구(승인/정산/선착순)와 직접 정합. Kotlin·트래픽 메트릭 공백만 보강하면 강한 매칭

## 도메인 한 줄 + 예상 기술부채
**도메인**: 온라인·모바일 커머스의 결제대행(PG) 인프라. 계약·체크아웃·승인·정산 핵심 로직과 브랜드페이·결제 위젯을 제공하는 핀테크.
**예상 기술부채 2~3개**
1. **승인·정산 트랜잭션 격리·멱등 누수** — 결제 승인과 정산 계산이 분리 시스템이면 이중 정산·누락 위험. 결재 트랜잭션 분리 + 멱등 키 + 이벤트 재처리 패턴 적용 가능
2. **선착순 트래픽 스파이크 시 락 경합** — 비관적 락 과다 시 처리량 붕괴. 트럼본 동시성 직렬화 경험으로 큐 적재량 quota + 락 계층 재설계 가능
3. **OpenAPI 연동 표면 다양성** — 가맹점별 요구사항으로 연동 분기 누적. 외부 호출 단일 추상화 + 버전 게이트 패턴 적용 가능
