티켓 워크플로우 시스템

변경 요청을 접수해 단계별 승인을 거쳐 처리하는 티켓 워크플로우 시스템입니다. 배포처럼 승인이 필요한 작업이 책임자 결재 없이 진행되지 않도록 막는 것이 핵심입니다. 백엔드와 프런트 전 구간을 개발했습니다.

  • 단계와 분기가 많아 흐름 제어가 어렵던 변경관리 워크플로우의 복잡한 상태를, 단계별 전이 규칙과 진입 조건을 명시한 티켓 상태 머신으로 모델링해 허용되지 않은 전이를 코드 레벨에서 차단하고 모든 변경이 정의된 경로로만 진행되도록 도메인을 설계
  • 금융권 감사에서는 배포 하나하나가 어떤 변경 요청을 거쳐 어떤 승인을 받고 처리했는지 추적할 수 있어야 해서, 티켓 생성부터 워크플로우 전 단계의 상태 전이마다 이력을 자동 적재하고 이를 모아 감사 증빙 문서로 바로 출력할 수 있게 구성
  • 결재·파이프라인·PMS 등 여러 CI/CD 도구를 각각 담당하는 MSA 모듈과 티켓을 연계하면서, 모듈마다 메시지 큐 및 REST 로 연동하되 결합도와 장애 전파 범위를 고려해 모듈 특성에 맞는 통신 방식을 선택해 설계
  • 화면·API 개발과 레거시 리팩토링에서 같은 패턴이 반복되던 비용을, Claude Code 기반 개발 하네스(스킬·훅 자동화)를 직접 구성해 자동화하고 동일 패턴 구현 작업에 적용해 차기 버전 개발 기간을 약 한 달 단축

Jenkins 파이프라인 실행 엔진

빌드·배포·테스트로 이어지는 여러 Jenkins 파이프라인의 실행 순서와 흐름을 제어하는 모듈입니다. 작업 접수부터 실행·추적·취소까지 전 과정을 설계하고 구현했습니다.

  • 내부 서비스가 외부 Jenkins 를 REST 로 동기 호출하며 강하게 결합돼, Jenkins 가 느려지거나 멈추면 내부 처리까지 함께 멈추던 구조를, 메시지 큐 기반 비동기로 바꿔 결합도를 낮추고 외부 장애가 플랫폼 전체로 번지지 않게 차단
  • Jenkins 에 작업이 한꺼번에 몰려 대기 큐가 넘치던 상황을, 큐의 DB 적재 수와 Jenkins 실제 큐 길이 중 큰 값을 기준으로 임계값까지만 실행을 내보내고 애플리케이션이 전체 흐름을 제어·복구하도록 설계해 Jenkins 과부하를 방지
  • Jenkins REST 호출 내부 동작에 맞춰 클라이언트를 설계해, POST 에 필요한 Crumb(CSRF 토큰) 인증을 직접 처리하고 빌드 번호가 실행 시작 시점에 부여되는 점에 맞춰 큐 아이템 ID 로 1차 추적 후 빌드 번호로 보강하는 2단계 매칭으로 동시 트리거에도 정확히 식별
  • 빌드 상태를 폴링으로 계속 확인하면 인스턴스가 늘수록 부하가 커지는 점을 고려해, webhook 으로 빌드 완료 상태를 수신하는 방식을 기본으로 두고, webhook 유실에 대비해 낮은 주기의 폴링과 24시간 무응답 작업 자동 종료로만 보완하도록 설계
  • 메시지 재수신이나 재시도로 같은 작업이 두 번 실행될 위험을, 작업 ID 기반 멱등 가드로 이미 처리·진행 중인 작업을 걸러내고, 컨슈머 실패 시 재시도 토픽으로 메시지를 유실 없이 재처리하며, 상태 변경과 이벤트 발행을 Outbox 로 묶어 내부 상태와 발행 이벤트가 어긋나지 않도록 보장

Jenkins 멀티 인스턴스 운영 표준화

Jenkins 인스턴스가 늘면서 수동으로 반복하던 설치·설정·운영 작업을 코드로 자동화했으며, VM 과 K8s 두 환경의 Jenkins 를 각각 Ansible 과 Helm 으로 형상관리하고, Groovy 로 파이프라인 시작·종료 시점에 웹훅을 발행해 빌드 흐름을 플랫폼과 연계했습니다.

  • 고객사마다 요구하는 환경이 달라 VM 과 K8s 를 따로 또는 혼합해 사용하여, 이를 대응하기 위해 어느 형태의 Jenkins 든 VM 은 Ansible, K8s 는 Helm 으로 컨트롤러 설치부터 플러그인 구성까지 코드로 형상관리
  • 빌드 에이전트도 VM 정적 노드(Docker 데몬·레지스트리 인증서 수동 설정)와 K8s 동적 Pod(빌드마다 생성·회수)를 환경에 맞춰 함께 운영하고, 신규 인스턴스 합류 절차를 수작업 없이 코드 배포만으로 끝나도록 단순화
  • 인스턴스마다 콜백을 수동 등록하던 문제를, 파이프라인 시작·종료 시점에 빌드 상태를 플랫폼으로 알리는 웹훅을 Jenkins 생명주기 훅(Groovy)으로 발행하도록 개발해 등록 절차 없이 모든 인스턴스가 동일하게 연계되도록 표준화
  • 인스턴스를 늘리거나 옮길 때마다 수십 개 Job 을 일일이 다시 만들던 작업을, 폴더 단위로 설정까지 통째 복제하는 Groovy 스크립트로 자동화해 서버 간 Job 이관을 수작업 재등록 없이 명령 한 번으로 처리
