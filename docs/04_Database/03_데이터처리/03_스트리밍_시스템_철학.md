# Chapter 13: A Philosophy of Streaming Systems

---

## 📌 핵심 요약
> 이 장에서는 **스트림 처리 기반의 데이터 시스템 설계 철학**을 다룬다. 단일 도구로 모든 요구사항을 충족할 수 없으므로, **여러 특화된 시스템을 조합**하여 데이터 통합을 달성해야 한다. **이벤트 로그 기반의 비동기 데이터플로우**를 통해 분산 트랜잭션 없이도 **정합성(Integrity)**을 유지할 수 있으며, **End-to-End 관점**에서 시스템 전체의 정확성을 보장해야 한다. 최종적으로 **신뢰하되 검증하라(Trust, but Verify)** 원칙으로 감사 가능한 시스템을 구축해야 한다.

## 📖 본문 정리

### 1. 데이터 통합 (Data Integration)

#### 1.1 특화된 도구의 조합

복잡한 애플리케이션에서 데이터는 여러 다른 방식으로 사용된다:

```
┌─────────────────────────────────────────────────────────────────┐
│                    데이터 사용 패턴들                              │
├─────────────────────────────────────────────────────────────────┤
│  📊 OLTP Database     → 트랜잭션 처리, 실시간 읽기/쓰기            │
│  🔍 Search Index      → 키워드 검색, 전문 검색                     │
│  📈 Analytics/DW      → 분석 쿼리, 집계                           │
│  💾 Cache             → 빠른 읽기, 지연 시간 단축                   │
│  🤖 ML Models         → 추천, 분류, 예측                          │
│  📬 Notifications     → 변경 알림, 실시간 업데이트                  │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 마치 주방에서 각 도구가 특화된 역할을 하듯이(칼은 자르기, 냄비는 끓이기), 데이터 시스템도 각각의 강점이 있다. 만능 도구는 어떤 것도 잘 하지 못한다.

#### 1.2 파생을 통한 도구 조합

**System of Record (기록 시스템)**과 **Derived Data (파생 데이터)** 구분:

```
┌──────────────────────────────────────────────────────────────────┐
│                     데이터플로우 아키텍처                          │
│                                                                  │
│   ┌─────────────┐                                                │
│   │   User      │                                                │
│   │   Input     │                                                │
│   └──────┬──────┘                                                │
│          │                                                       │
│          ▼                                                       │
│   ┌─────────────┐     CDC/Event Log                              │
│   │  System of  │ ─────────────────┬────────┬────────┐           │
│   │   Record    │                  │        │        │           │
│   │ (Database)  │                  ▼        ▼        ▼           │
│   └─────────────┘           ┌──────────┐ ┌──────┐ ┌──────┐       │
│                             │  Search  │ │Cache │ │  ML  │       │
│                             │  Index   │ │      │ │Model │       │
│                             └──────────┘ └──────┘ └──────┘       │
│                                                                  │
│   ──────────────────────────────────────────────────────────     │
│   Key: 모든 파생 데이터는 단일 이벤트 로그에서 동일한 순서로 처리     │
└──────────────────────────────────────────────────────────────────┘
```

**핵심 원칙**:
- 데이터가 **먼저 기록되는 곳**(System of Record)을 명확히 정의
- 다른 표현들은 이 소스에서 **파생**됨
- **CDC(Change Data Capture)**를 통해 변경 사항을 캡처하고 전파

#### 1.3 파생 데이터 vs 분산 트랜잭션

| 관점 | 분산 트랜잭션 | 파생 데이터 시스템 |
|------|--------------|------------------|
| **순서 결정** | 락(Locks)을 통한 상호 배제 | 로그(Log)를 통한 순서화 |
| **원자성 보장** | Atomic Commit (2PC) | 결정론적 재시도 + 멱등성 |
| **읽기 일관성** | 즉시 최신 값 읽기 가능 | 비동기 업데이트 (지연 있음) |
| **장애 허용** | 참여자 하나만 실패해도 전체 실패 | 로컬 장애 격리 가능 |
| **성능** | 동기 조정으로 인한 오버헤드 | 비동기로 더 나은 성능 |

> 💬 **비유**: 분산 트랜잭션은 모든 사람이 동시에 손을 들어야 하는 만장일치 투표와 같고, 파생 데이터는 회의록을 작성해서 나중에 각자 읽는 것과 같다.

#### 1.4 전체 순서화의 한계

**Total Order가 어려운 상황들**:

```
┌─────────────────────────────────────────────────────────────────┐
│              전체 순서화(Total Ordering)의 한계                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1️⃣ 처리량 초과                                                  │
│     ┌─────────┐                                                 │
│     │ Single  │ ← 단일 노드 처리 한계 초과                        │
│     │ Leader  │    → 샤딩 필요 → 샤드 간 순서 모호                │
│     └─────────┘                                                 │
│                                                                 │
│  2️⃣ 지리적 분산                                                  │
│     ┌──────┐    Network    ┌──────┐                             │
│     │ DC1  │ ←─────────────→│ DC2  │                             │
│     │Leader│    Latency    │Leader│                             │
│     └──────┘               └──────┘                             │
│     → 각 DC별 리더 → DC 간 순서 불명확                            │
│                                                                 │
│  3️⃣ 마이크로서비스 독립성                                         │
│     ┌─────┐  ┌─────┐  ┌─────┐                                   │
│     │Svc A│  │Svc B│  │Svc C│                                   │
│     └──┬──┘  └──┬──┘  └──┬──┘                                   │
│        │        │        │                                      │
│     각 서비스가 독립적 상태 관리 → 서비스 간 순서 미정의            │
│                                                                 │
│  4️⃣ 오프라인 클라이언트                                          │
│     📱 ←──offline──→ 🖥️                                         │
│     클라이언트와 서버가 다른 순서로 이벤트 인식                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 1.5 인과관계 캡처 문제

**예시: 소셜 네트워크에서의 인과관계**

```
시나리오: Alice와 Bob이 헤어짐

Timeline:
1. Alice가 Bob을 친구 목록에서 삭제  ──→ Friendship Service
2. Alice가 친구들에게 Bob 험담 메시지 전송 ──→ Message Service
3. 알림 서비스가 메시지를 친구들에게 전송 ──→ Notification Service

문제:
┌─────────────────────────────────────────────────────────────────┐
│  Notification Service                                           │
│                                                                 │
│  메시지 이벤트 ────┐                                             │
│                   ├──→ JOIN ──→ 알림 발송                       │
│  친구 목록 ───────┘                                             │
│                                                                 │
│  만약 unfriend 이벤트가 message 이벤트보다 늦게 처리되면?         │
│  → Bob에게도 험담 메시지 알림이 전송됨! 😱                        │
└─────────────────────────────────────────────────────────────────┘
```

**해결 접근법들**:

| 접근법 | 설명 | 한계 |
|--------|------|------|
| **논리적 타임스탬프** | 조정 없이 전체 순서 제공 | 순서 어긋난 이벤트 처리 필요 |
| **인과 관계 식별자** | 이전 상태의 ID를 참조 | 추가 메타데이터 전파 필요 |
| **충돌 해결 알고리즘** | CRDT 등 자동 해결 | 외부 부작용에는 도움 안 됨 |

### 2. 배치와 스트림 처리

#### 2.1 파생 상태 유지

```
┌─────────────────────────────────────────────────────────────────┐
│                   파생 상태 유지 원칙                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  입력 (Immutable)                                                │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────┐                        │
│  │     결정론적 변환 함수               │                        │
│  │   (Deterministic Transform)         │                        │
│  │                                     │                        │
│  │  • 동일 입력 → 동일 출력             │                        │
│  │  • 부작용 없음 (Side-effect free)    │                        │
│  │  • 입력은 불변, 출력은 Append-only   │                        │
│  └─────────────────────────────────────┘                        │
│       │                                                         │
│       ▼                                                         │
│  출력 (파생 데이터)                                               │
│                                                                 │
│  장점:                                                          │
│  ✅ 장애 복구 용이 (재처리 가능)                                  │
│  ✅ 조직 내 데이터플로우 추론 용이                                │
│  ✅ 비동기로 장애 격리 가능                                       │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 수학 함수 f(x) = 2x처럼, 입력이 같으면 출력도 항상 같다. 어디서 계산하든, 몇 번 계산하든 결과는 동일하다.

#### 2.2 재처리를 통한 애플리케이션 진화

**철도 궤간 마이그레이션 비유**:

```
19세기 영국 철도 궤간(Gauge) 표준화 과정:

Phase 1: 혼합 궤간 (Dual Gauge)
┌────────────────────────────────────────┐
│  ═══════════════════════════════════   │  ← 기존 궤간
│  ───────────────────────────────────   │  ← 새 궤간 (3번째 레일 추가)
│  ═══════════════════════════════════   │
└────────────────────────────────────────┘
두 종류의 기차 모두 운행 가능!

Phase 2: 점진적 전환
┌────────────────────────────────────────┐
│  구 기차 → 새 기차로 교체 (점진적)       │
│  사용자는 서비스 중단 없이 이용          │
└────────────────────────────────────────┘

Phase 3: 완료
┌────────────────────────────────────────┐
│  불필요한 레일 제거                      │
└────────────────────────────────────────┘
```

**데이터 시스템에서의 적용**:

```
┌─────────────────────────────────────────────────────────────────┐
│                  점진적 스키마 마이그레이션                        │
│                                                                 │
│  이벤트 로그 (원본)                                               │
│       │                                                         │
│       ├──────────────→ Old View (기존 스키마)                    │
│       │                    │                                    │
│       │                    ▼                                    │
│       │               [90% 사용자]                               │
│       │                                                         │
│       └──────────────→ New View (새 스키마)                      │
│                            │                                    │
│                            ▼                                    │
│                       [10% 사용자 - 테스트]                       │
│                                                                 │
│  점진적으로 비율 조정 → 문제 발생 시 즉시 롤백 가능                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 2.3 배치와 스트림 통합

**Lambda vs Kappa Architecture**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Lambda Architecture                          │
│                                                                 │
│                    ┌───────────────┐                            │
│              ┌────→│  Batch Layer  │────┐                       │
│              │     │ (정확한 결과)  │    │                       │
│  이벤트 ─────┤     └───────────────┘    ├───→ 쿼리 결과          │
│              │     ┌───────────────┐    │                       │
│              └────→│ Speed Layer   │────┘                       │
│                    │ (빠른 근사치)  │                            │
│                    └───────────────┘                            │
│                                                                 │
│  문제점: 두 코드 경로 유지 필요, 복잡성 증가                       │
├─────────────────────────────────────────────────────────────────┤
│                    Kappa Architecture                           │
│                                                                 │
│                    ┌───────────────┐                            │
│  이벤트 ──────────→│ Single Stream │───────→ 쿼리 결과           │
│                    │   Processor   │                            │
│                    └───────────────┘                            │
│                           │                                     │
│                           ▼                                     │
│                    히스토리 재처리도 동일 엔진 사용                 │
│                                                                 │
│  장점: 단일 코드 경로, 일관성 보장                                │
└─────────────────────────────────────────────────────────────────┘
```

**통합을 위한 필수 기능**:

| 기능 | 설명 |
|------|------|
| **히스토리 재생** | 동일한 처리 엔진으로 과거 이벤트 재처리 |
| **Exactly-once** | 장애 발생해도 결과는 장애 없을 때와 동일 |
| **이벤트 시간 윈도우** | 처리 시간이 아닌 이벤트 시간 기준 |

### 3. 데이터베이스 Unbundling

#### 3.1 데이터베이스 기능의 분해

```
┌─────────────────────────────────────────────────────────────────┐
│               전통적 데이터베이스 vs Unbundled 접근               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  전통적 Monolithic DB              Unbundled 접근               │
│  ┌─────────────────────┐          ┌─────────────────────┐       │
│  │   Single Database   │          │    Event Log        │       │
│  │  ┌───────────────┐  │          │    (이벤트 허브)    │       │
│  │  │ Storage Engine│  │          └─────────┬───────────┘       │
│  │  │ Indexes       │  │                    │                   │
│  │  │ Replication   │  │          ┌─────────┼─────────┐         │
│  │  │ Query Engine  │  │          ▼         ▼         ▼         │
│  │  │ Transactions  │  │    ┌──────────┐┌───────┐┌────────┐     │
│  │  └───────────────┘  │    │ Search   ││ Cache ││Analytics│    │
│  └─────────────────────┘    │ Index    ││       ││  DW    │     │
│                             └──────────┘└───────┘└────────┘     │
│                                                                 │
│  장점:                      장점:                                │
│  • 단순한 운영              • 각 컴포넌트 독립적 확장             │
│  • 일관된 인터페이스         • 다양한 접근 패턴 최적화             │
│  • 강력한 보장              • 느슨한 결합으로 장애 격리            │
│                                                                 │
│  단점:                      단점:                                │
│  • 모든 요구사항 충족 어려움  • 운영 복잡성 증가                   │
│  • 확장 한계                • 학습 곡선                          │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 올인원 복합기 vs 전문 프린터+스캐너+팩스. 복합기는 편하지만, 각각의 기능은 전문 장비만 못하다.

#### 3.2 인덱스 생성 = 재처리

#### 3.3 연합(Federation) vs Unbundling

| 관점 | Federated Database | Unbundled Database |
|------|-------------------|-------------------|
| **초점** | 읽기 통합 | 쓰기 동기화 |
| **방식** | 통합 쿼리 인터페이스 | 이벤트 로그 + 멱등 쓰기 |
| **철학** | 관계형 전통 (고수준 추상화) | Unix 전통 (작은 도구 조합) |
| **예시** | Trino, PostgreSQL FDW | Debezium + Kafka + 스트림 처리 |
| **장점** | 쿼리 통합 용이 | 느슨한 결합, 장애 격리 |

```
┌─────────────────────────────────────────────────────────────────┐
│                    Unbundling의 장점                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  🔧 시스템 수준:                                                 │
│     • 비동기 이벤트 스트림 → 장애 격리                            │
│     • 컨슈머 느려지거나 실패해도 프로듀서 영향 없음                │
│     • 메시지 버퍼링으로 복구 시 데이터 손실 없음                   │
│                                                                 │
│  👥 조직 수준:                                                   │
│     • 팀별 독립적 개발/배포/유지보수                              │
│     • 잘 정의된 인터페이스 (이벤트 로그)                          │
│     • 특화된 역량에 집중 가능                                     │
│                                                                 │
│  ⚠️ 주의:                                                        │
│     • 필요하지 않은 규모에서는 단일 DB가 더 효율적                 │
│     • Unbundling은 깊이보다 넓이에 대한 것                        │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 데이터플로우 기반 애플리케이션 설계

#### 4.1 애플리케이션 코드 = 파생 함수

```
┌─────────────────────────────────────────────────────────────────┐
│                    파생 함수의 종류                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Secondary Index:                                               │
│     베이스 테이블 → 인덱스 필드 추출 → 정렬 → 인덱스 구조          │
│     (표준화된 함수, CREATE INDEX로 호출)                          │
│                                                                 │
│  Full-Text Search Index:                                        │
│     문서 → 언어 감지 → 단어 분리 → 어근 추출 → 동의어 처리         │
│          → 역인덱스 구조 생성                                    │
│     (도메인 특화 튜닝 필요)                                       │
│                                                                 │
│  ML Model:                                                      │
│     학습 데이터 → 피처 추출 → 통계 분석 → 모델                    │
│     새 입력 + 모델 → 예측 결과                                   │
│     (애플리케이션 특화 피처 엔지니어링 필요)                       │
│                                                                 │
│  Cache:                                                         │
│     원본 데이터 → UI에서 필요한 형태로 변환 → 캐시                 │
│     (UI 변경 시 캐시 정의도 변경 필요)                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 4.2 애플리케이션 코드와 상태의 분리

**전통적 웹 애플리케이션 모델**:

```
┌─────────────────────────────────────────────────────────────────┐
│                  Stateless 애플리케이션 모델                      │
│                                                                 │
│     ┌─────────┐    ┌─────────┐    ┌─────────┐                   │
│     │ App     │    │ App     │    │ App     │                   │
│     │ Server  │    │ Server  │    │ Server  │  ← Stateless      │
│     └────┬────┘    └────┬────┘    └────┬────┘                   │
│          │              │              │                        │
│          └──────────────┼──────────────┘                        │
│                         │                                       │
│                         ▼                                       │
│                  ┌─────────────┐                                 │
│                  │  Database   │  ← 공유 가변 변수처럼 동작       │
│                  │  (State)    │                                │
│                  └─────────────┘                                 │
│                                                                 │
│  "We believe in the separation of Church and state"             │
│   (Church = 람다 계산법 창시자 Alonzo Church)                     │
│   → 함수형 프로그래밍: 불변 상태와 로직 분리                       │
└─────────────────────────────────────────────────────────────────┘
```

**데이터플로우 관점의 재해석**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    데이터플로우 기반 모델                         │
│                                                                 │
│  전통적: DB를 수동적 변수로 취급                                  │
│     app.read(db)  →  값 반환                                    │
│     app.write(db) →  상태 변경                                  │
│                                                                 │
│  데이터플로우: 상태 변경과 코드의 상호작용                         │
│                                                                 │
│     상태 변경 A ──→ 코드 반응 ──→ 상태 변경 B ──→ 코드 반응       │
│                                                                 │
│  스프레드시트와 유사:                                            │
│     셀 A1 변경 → A1을 참조하는 B1 자동 재계산                     │
│                                                                 │
│  데이터 시스템에서:                                              │
│     레코드 변경 → 인덱스 자동 업데이트                            │
│                → 캐시 자동 갱신                                  │
│                → 뷰 자동 리프레시                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 4.3 스트림 프로세서 vs 마이크로서비스

**환율 조회 예시**:

```
┌─────────────────────────────────────────────────────────────────┐
│                마이크로서비스 방식 vs 데이터플로우 방식            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  마이크로서비스 방식:                                            │
│  ┌──────────────┐      RPC       ┌──────────────┐              │
│  │ Purchase     │ ──────────────→│ Exchange     │              │
│  │ Service      │    요청/응답    │ Rate Service │              │
│  └──────────────┘                └──────────────┘              │
│                                                                 │
│  • 동기 네트워크 요청                                            │
│  • 환율 서비스 장애 → 구매 서비스도 영향                          │
│  • 네트워크 지연 추가                                            │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  데이터플로우 방식:                                              │
│                                                                 │
│  ┌──────────────┐                                               │
│  │ Exchange     │                                               │
│  │ Rate Updates │ ─────┐                                        │
│  └──────────────┘      │                                        │
│                        ▼                                        │
│                 [환율 이벤트 스트림]                              │
│                        │                                        │
│  ┌──────────────┐      ▼                                        │
│  │ Purchase     │ ──→ 로컬 DB에                                 │
│  │ Service      │     환율 저장 ──→ 로컬 조회                    │
│  └──────────────┘                                               │
│                                                                 │
│  • 비동기 스트림 구독                                            │
│  • 로컬 DB에서 환율 조회 (네트워크 없음)                          │
│  • 환율 서비스 장애에도 기존 환율로 계속 동작                      │
│  • Stream Join: 구매 이벤트 ⋈ 환율 업데이트 이벤트               │
│                                                                 │
│  "The fastest network request is no network request at all!"    │
└─────────────────────────────────────────────────────────────────┘

### 5. 파생 상태 관찰 (Observing Derived State)

#### 5.1 Write Path와 Read Path

```
┌─────────────────────────────────────────────────────────────────┐
│                Write Path와 Read Path 개념                       │
│                                                                 │
│                                                                 │
│  데이터 수집                                                     │
│      │                                                          │
│      ▼                                                          │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                    WRITE PATH                              │  │
│  │  입력 → 변환 → 조인 → 필터 → 집계 → 모델 학습               │  │
│  │                                                            │  │
│  │  • Eager Evaluation (즉시 계산)                            │  │
│  │  • 데이터 도착 시 바로 처리                                 │  │
│  │  • 누가 읽을지 모르지만 미리 준비                           │  │
│  └────────────────────────┬──────────────────────────────────┘  │
│                           │                                     │
│                           ▼                                     │
│               ┌───────────────────────┐                         │
│               │    Derived Dataset    │ ← Write/Read 경계       │
│               │  (Index, Cache, View) │                         │
│               └───────────────────────┘                         │
│                           │                                     │
│  ┌────────────────────────▼──────────────────────────────────┐  │
│  │                     READ PATH                              │  │
│  │                                                            │  │
│  │  쿼리 → 파생 데이터 읽기 → 추가 처리 → 응답 생성            │  │
│  │                                                            │  │
│  │  • Lazy Evaluation (요청 시 계산)                          │  │
│  │  • 사용자 요청이 있을 때만 실행                             │  │
│  └────────────────────────────────────────────────────────────┘  │
│                           │                                     │
│                           ▼                                     │
│                      데이터 소비                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: Write Path는 레스토랑에서 미리 식재료를 손질하고 육수를 끓여두는 prep 작업이고, Read Path는 주문이 들어왔을 때 실제로 요리하는 것이다.

#### 5.2 경계 이동의 Trade-off

**검색 인덱스 예시**:

```
┌─────────────────────────────────────────────────────────────────┐
│              검색 시스템에서의 Write/Read 경계                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  옵션 1: 인덱스 없음 (경계를 읽기 쪽으로)                         │
│  ┌─────────────────┐                                            │
│  │ 문서 저장만     │ → 검색 시 전체 스캔 (grep)                  │
│  └─────────────────┘                                            │
│  Write 비용: 낮음 / Read 비용: 매우 높음                         │
│                                                                 │
│  옵션 2: 인덱스 사용 (균형)                                      │
│  ┌─────────────────┐                                            │
│  │ 문서 저장 +     │ → 검색 시 인덱스 룩업                       │
│  │ 인덱스 업데이트 │                                            │
│  └─────────────────┘                                            │
│  Write 비용: 중간 / Read 비용: 중간                              │
│                                                                 │
│  옵션 3: 모든 쿼리 결과 사전 계산 (경계를 쓰기 쪽으로)            │
│  ┌─────────────────┐                                            │
│  │ 문서 저장 +     │ → 검색 시 바로 결과 반환                    │
│  │ 모든 쿼리 결과  │                                            │
│  │ 미리 계산       │                                            │
│  └─────────────────┘                                            │
│  Write 비용: 불가능 (무한) / Read 비용: 낮음                      │
│                                                                 │
│  옵션 4: 인기 쿼리만 캐시 (Materialized View)                    │
│  ┌─────────────────┐                                            │
│  │ 인덱스 + 인기   │ → 인기 쿼리는 캐시, 나머지는 인덱스         │
│  │ 쿼리 결과 캐시  │                                            │
│  └─────────────────┘                                            │
│  Write 비용: 중간+ / Read 비용: 낮음~중간                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

│        │                                                        │
│        │ POST /transfer                                         │
│        │──────────────────►                                     │
│        │      ✕ 타임아웃                                        │
│        │                                                        │
│        │ "Are you sure you want to                              │
│        │  submit this form again?"                              │
│        │                                                        │
│        │ 사용자: "Yes!" (원하니까!)                              │
│        │                                                        │
│        │ POST /transfer (재시도)                                │
│        │──────────────────►                                     │
│        │                                                        │
│  결과: 두 번 이체됨!                                             │
│                                                                 │
│  TCP도, DB 트랜잭션도 이 문제를 막지 못함                         │
│  → End-to-End 솔루션 필요                                       │
└─────────────────────────────────────────────────────────────────┘
```

#### 6.2 End-to-End 중복 억제 솔루션

```sql
-- Request ID를 통한 멱등성 보장
ALTER TABLE requests ADD UNIQUE (request_id);

BEGIN TRANSACTION;

-- 요청 기록 (중복 시 실패)
INSERT INTO requests
  (request_id, from_account, to_account, amount)
  VALUES('0286FDB8-D7E1-423F-B40B-792B3608036C', 4321, 1234, 11.00);

-- 실제 이체 수행
UPDATE accounts SET balance = balance + 11.00 WHERE account_id = 1234;
UPDATE accounts SET balance = balance - 11.00 WHERE account_id = 4321;

COMMIT;
```

**핵심 원리**:

```
┌─────────────────────────────────────────────────────────────────┐
│                  End-to-End Request ID 패턴                      │
│                                                                 │
│  ┌──────────┐                                                   │
│  │ Browser  │ ← UUID 생성 (예: 폼 hidden field)                 │
│  │ [uuid-1] │                                                   │
│  └────┬─────┘                                                   │
│       │                                                         │
│       │ request_id=uuid-1                                       │
│       ▼                                                         │
│  ┌──────────┐                                                   │
│  │  Server  │ ← request_id 전달                                 │
│  └────┬─────┘                                                   │
│       │                                                         │
│       │ INSERT INTO requests (request_id, ...)                  │
│       ▼                                                         │
│  ┌──────────┐                                                   │
│  │ Database │ ← UNIQUE 제약조건으로 중복 거부                    │
│  └──────────┘                                                   │
│                                                                 │
│  재시도해도 동일한 request_id → INSERT 실패 → 트랜잭션 롤백       │
│  → 중복 실행 방지!                                               │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **End-to-End Argument** (Saltzer, Reed, Clark, 1984):
> "해당 기능은 통신 시스템의 끝점에 있는 애플리케이션의 지식과 도움으로만 완전하고 정확하게 구현될 수 있다."

### 7. 제약조건 강제 (Enforcing Constraints)

#### 7.1 유일성 제약과 합의

**유일성 제약 = 합의 필요**:

```
┌─────────────────────────────────────────────────────────────────┐
│                유일성 제약의 본질                                 │
│                                                                 │
│  동시 요청:                                                      │
│     Client A: 사용자명 "alice" 등록 요청                         │
│     Client B: 사용자명 "alice" 등록 요청                         │
│                                                                 │
│  시스템은 결정해야 함:                                           │
│     • 누구의 요청을 수락?                                        │
│     • 누구의 요청을 거부?                                        │
│                                                                 │
│  이것은 곧 합의(Consensus) 문제!                                 │
│                                                                 │
│  일반적 해결책:                                                  │
│  ┌─────────────────┐                                            │
│  │  Single Leader  │ ← 모든 결정을 단일 노드가                   │
│  └─────────────────┘                                            │
│                                                                 │
│  확장: 값 기준 샤딩                                              │
│     username.hash() → 특정 샤드                                 │
│     해당 샤드의 리더가 결정                                       │
│                                                                 │
│  ⚠️ 비동기 다중 리더 복제 불가능                                  │
│     다른 리더가 동시에 충돌 쓰기 수락 가능                        │
└─────────────────────────────────────────────────────────────────┘
```

#### 7.2 로그 기반 메시징에서의 유일성

```
┌─────────────────────────────────────────────────────────────────┐
│            Log 기반 유일성 제약 구현                              │
│                                                                 │
│  1. 요청을 이벤트로 변환하여 로그에 추가                          │
│     username.hash() → 특정 샤드                                 │
│                                                                 │
│  2. 스트림 프로세서가 순차 처리                                   │
│                                                                 │
│     ┌────────────────────────────────────────┐                  │
│     │ Log Shard (username hash = 42)        │                  │
│     │                                        │                  │
│     │ [req-1: alice] [req-2: alice] [req-3: bob]              │
│     └────────────────────────────────────────┘                  │
│                    │                                            │
│                    ▼                                            │
│     ┌────────────────────────────────────────┐                  │
│     │ Stream Processor                       │                  │
│     │                                        │                  │
│     │ Local DB: {alice: taken}              │                  │
│     │                                        │                  │
│     │ req-1: alice 사용 가능 → Success 출력  │                  │
│     │ req-2: alice 이미 사용 → Reject 출력   │                  │
│     │ req-3: bob 사용 가능 → Success 출력    │                  │
│     └────────────────────────────────────────┘                  │
│                    │                                            │
│                    ▼                                            │
│     ┌────────────────────────────────────────┐                  │
│     │ Output Stream                          │                  │
│     │ [success: req-1] [reject: req-2] [success: req-3]       │
│     └────────────────────────────────────────┘                  │
│                                                                 │
│  클라이언트는 Output Stream을 구독하여 결과 확인                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 7.3 멀티 샤드 요청 처리

**분산 트랜잭션 없이 원자성 달성**:

```
┌─────────────────────────────────────────────────────────────────┐
│           Multi-Shard 이체 처리 (분산 트랜잭션 없이)              │
│                                                                 │
│  시나리오: 계좌 A → 계좌 B 이체 + 수수료                          │
│                                                                 │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │ 1. 사용자 요청                                             │ │
│  │    request_id + source_account + dest + fees + amount      │ │
│  │                        │                                   │ │
│  │                        ▼                                   │ │
│  │    Source Account Shard (Log)                              │ │
│  └────────────────────────────────────────────────────────────┘ │
│                           │                                     │
│                           ▼                                     │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │ 2. Source Account Processor                                │ │
│  │    • 잔액 확인                                             │ │
│  │    • 충분하면 예약(reserve) 상태로 변경                     │ │
│  │    • 3개 이벤트 emit:                                      │ │
│  │      - Outgoing Payment → Source Shard                     │ │
│  │      - Incoming Payment → Dest Shard                       │ │
│  │      - Fee Payment → Fee Shard                             │ │
│  └────────────────────────────────────────────────────────────┘ │
│                           │                                     │
│           ┌───────────────┼───────────────┐                     │
│           ▼               ▼               ▼                     │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐            │
│  │Source Shard  │ │ Dest Shard   │ │ Fee Shard    │            │
│  │ Processor    │ │ Processor    │ │ Processor    │            │
│  │              │ │              │ │              │            │
│  │ Outgoing     │ │ Incoming     │ │ Incoming     │            │
│  │ Payment 처리 │ │ Payment 처리 │ │ Payment 처리 │            │
│  │ (request_id  │ │ (request_id  │ │ (request_id  │            │
│  │  중복 확인)  │ │  중복 확인)  │ │  중복 확인)  │            │
│  └──────────────┘ └──────────────┘ └──────────────┘            │
│                                                                 │
│  핵심:                                                          │
│  • 원자성은 초기 이벤트가 로그에 기록되는 것에서 옴               │
│  • 이후 모든 파생 이벤트는 결정론적으로 발생                      │
│  • request_id로 각 단계에서 중복 방지                            │
│  • 장애 시 재처리해도 동일 결과 (멱등성)                          │
└─────────────────────────────────────────────────────────────────┘
```

### 8. 적시성과 정합성 (Timeliness and Integrity)

#### 8.1 두 개념의 분리

```
┌─────────────────────────────────────────────────────────────────┐
│             Timeliness vs Integrity                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Timeliness (적시성)                                             │
│  ─────────────────                                              │
│  • 사용자가 최신 상태를 관찰하는 것                               │
│  • 위반 시: 일시적 불일치 (기다리면 해결)                         │
│  • 예: 방금 쓴 데이터가 아직 안 보임                             │
│  • CAP의 Consistency = Linearizability = 강한 Timeliness        │
│  • "Eventual Consistency"                                       │
│                                                                 │
│  Integrity (정합성)                                              │
│  ─────────────────                                              │
│  • 데이터 손실/오염 없음, 모순된 데이터 없음                      │
│  • 위반 시: 영구적 손상 (명시적 수리 필요)                        │
│  • 예: 인덱스에 레코드 누락, 잔액 불일치                          │
│  • ACID의 C (도메인 불변식)                                      │
│  • "Perpetual Inconsistency"                                    │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  비유:                                                          │
│  • Timeliness 위반: 신용카드 명세서에서 어제 구매가 안 보임       │
│    → 기다리면 나타남 (정상)                                      │
│                                                                 │
│  • Integrity 위반: 명세서 합계가 개별 거래 합과 다름              │
│    → 절대 자동으로 해결 안 됨 (심각한 문제)                       │
│                                                                 │
│  슬로건:                                                        │
│  "Timeliness 위반 = Eventual Consistency"                       │
│  "Integrity 위반 = Perpetual Inconsistency"                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 8.2 데이터플로우 시스템의 정확성

**이벤트 기반 시스템이 Integrity를 유지하는 방법**:

```
┌─────────────────────────────────────────────────────────────────┐
│           Dataflow System의 Integrity 보장 메커니즘              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1️⃣ 단일 메시지로 쓰기 작업 표현                                 │
│     • 쓰기 내용을 단일 이벤트로 원자적 기록                       │
│     • Event Sourcing과 잘 맞음                                  │
│                                                                 │
│  2️⃣ 결정론적 파생 함수                                          │
│     • 동일 입력 → 동일 출력                                     │
│     • 저장 프로시저와 유사                                       │
│     • 재처리해도 동일 결과                                       │
│                                                                 │
│  3️⃣ 클라이언트 생성 Request ID                                  │
│     • End-to-End 중복 억제                                      │
│     • 모든 처리 단계를 통과                                      │
│     • 멱등성 보장                                               │
│                                                                 │
│  4️⃣ 불변 메시지 + 재처리 가능                                   │
│     • 버그 발생 시 수정 후 재처리로 복구                          │
│     • 원본 이벤트는 삭제하지 않음                                │
│                                                                 │
│  결과: 분산 트랜잭션 없이도 Integrity 달성 가능!                  │
│        (단, Timeliness는 보장하지 않음)                          │
└─────────────────────────────────────────────────────────────────┘
```

#### 8.3 느슨하게 해석되는 제약조건

**비즈니스 현실에서의 제약조건**:

```
┌─────────────────────────────────────────────────────────────────┐
│              실제 비즈니스의 제약조건 해석                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  "재고보다 많이 팔 수 없다" 제약                                 │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 이론적: 엄격한 제약 → 재고 확인 후에만 판매                  ││
│  │                                                             ││
│  │ 현실: 지게차가 물건 파손하면? 어차피 사과해야 함              ││
│  │       → 초과 판매해도 사과하고 보상하면 됨                   ││
│  │       → 제약 위반 후 보상 (Compensating Transaction)        ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  항공권 오버부킹                                                 │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ "한 좌석에 한 명" 제약을 의도적으로 위반                     ││
│  │ 노쇼 예상 → 초과 예약                                       ││
│  │ 문제 발생 시 → 환불, 업그레이드, 다른 항공편 제공            ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  은행 계좌 잔액                                                  │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 잔액보다 많이 출금?                                         ││
│  │ → 초과 출금 수수료 부과                                     ││
│  │ → 일일 한도로 리스크 제한                                   ││
│  │ → 채권 추심 프로세스 존재                                   ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  핵심 인사이트:                                                  │
│  • 많은 제약은 즉시 강제할 필요 없음                             │
│  • 위반 후 사과(Compensating Transaction)가 허용됨              │
│  • Timeliness는 필요 없어도 Integrity는 여전히 필요              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 8.4 조정 회피 데이터 시스템

```
┌─────────────────────────────────────────────────────────────────┐
│          Coordination-Avoiding Data Systems                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  두 가지 관찰:                                                   │
│                                                                 │
│  1. 데이터플로우 시스템은 원자적 커밋/선형성/동기 조정 없이도     │
│     파생 데이터의 Integrity를 유지할 수 있다                      │
│                                                                 │
│  2. 많은 애플리케이션은 느슨한 제약으로도 충분하다               │
│     (위반 후 사과 가능, Integrity만 유지되면 됨)                  │
│                                                                 │
│  결론: 조정(Coordination) 없이도 많은 앱 요구사항 충족 가능!      │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Multi-DC, Multi-Leader 가능:                                   │
│                                                                 │
│  ┌──────────┐        비동기         ┌──────────┐               │
│  │   DC 1   │◄─────────────────────►│   DC 2   │               │
│  │  Leader  │       복제            │  Leader  │               │
│  └──────────┘                       └──────────┘               │
│       ↑                                  ↑                      │
│       │                                  │                      │
│    사용자 A                           사용자 B                   │
│                                                                 │
│  • 각 DC 독립 운영 가능                                          │
│  • 동기 크로스-리전 조정 불필요                                  │
│  • 약한 Timeliness 보장 (Linearizable 아님)                     │
│  • 강한 Integrity 보장                                          │
│                                                                 │
│  Trade-off:                                                     │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ Coordination ↑ → 불일치로 인한 사과 ↓                       ││
│  │             → 가용성/성능 저하로 인한 사과 ↑                 ││
│  │                                                             ││
│  │ Coordination ↓ → 불일치로 인한 사과 ↑                       ││
│  │             → 가용성/성능 저하로 인한 사과 ↓                 ││
│  │                                                             ││
│  │ 완전히 사과 없애는 건 불가능 → 최적 지점 찾기!               ││
│  └─────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────┘
```

### 9. 신뢰하되 검증하라 (Trust, but Verify)

#### 9.1 시스템 모델의 한계

```
┌─────────────────────────────────────────────────────────────────┐
│                    시스템 모델 가정들                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  전통적으로 가정하는 것들:                                        │
│  ✓ 프로세스는 크래시할 수 있다                                   │
│  ✓ 네트워크는 지연되거나 메시지 손실 가능                         │
│  ✓ 클럭은 틀릴 수 있다                                          │
│                                                                 │
│  하지만 암묵적으로 신뢰하는 것들:                                 │
│  ✗ fsync 후 디스크 데이터는 안전하다                             │
│  ✗ 메모리 데이터는 손상되지 않는다                               │
│  ✗ CPU 연산 결과는 정확하다                                     │
│                                                                 │
│  현실:                                                          │
│  • 디스크도 데이터 손상 가능 (bit rot)                           │
│  • 메모리도 손상 가능 (cosmic rays, hardware faults)            │
│  • 네트워크도 데이터 변조 가능                                   │
│  • 소프트웨어에는 버그가 있다                                    │
│                                                                 │
│  대규모에서는 "매우 드문" 일도 발생한다!                          │
└─────────────────────────────────────────────────────────────────┘
```

#### 9.2 감사 가능한 설계

```
┌─────────────────────────────────────────────────────────────────┐
│                    Auditability를 위한 설계                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  전통적 트랜잭션:                                                │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ UPDATE accounts SET balance = balance - 100 WHERE id = 1;  ││
│  │ UPDATE accounts SET balance = balance + 100 WHERE id = 2;  ││
│  │                                                             ││
│  │ → 왜 이 변경이 일어났는지 알 수 없음                         ││
│  │ → 애플리케이션 로직은 휘발성                                 ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  Event Sourcing:                                                │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ Event: {                                                    ││
│  │   type: "transfer",                                         ││
│  │   from: 1, to: 2, amount: 100,                              ││
│  │   reason: "payment for invoice #12345",                     ││
│  │   initiated_by: "user-789",                                 ││
│  │   timestamp: "2024-01-15T10:30:00Z"                         ││
│  │ }                                                           ││
│  │                                                             ││
│  │ → 불변 이벤트로 이유/주체/시점 모두 기록                     ││
│  │ → 결정론적 파생 → 동일 이벤트 재처리 = 동일 결과             ││
│  │ → 데이터 출처(Provenance) 추적 가능                         ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  장점:                                                          │
│  • 해시로 이벤트 저장소 무결성 검증 가능                          │
│  • 파생 상태는 재처리로 검증 가능                                │
│  • 시스템 동작 디버깅/추적 용이                                  │
│  • Time-travel debugging 가능                                   │
└─────────────────────────────────────────────────────────────────┘
```

### 면접 대비 질문

**Q1: 파생 데이터 시스템과 분산 트랜잭션의 차이점은?**

```
A: 목표는 유사하지만 메커니즘이 다릅니다.

   분산 트랜잭션:
   • 락으로 순서 결정, 2PC로 원자성 보장
   • 즉시 일관된 읽기 가능 (Linearizable)
   • 장애 시 전체 실패, 성능 오버헤드 큼

   파생 데이터:
   • 로그로 순서 결정, 멱등성으로 정확성 보장
   • 비동기 업데이트 (Eventual Consistency)
   • 장애 격리 가능, 더 나은 성능/가용성

   핵심: Timeliness를 포기하고 Integrity를 유지하면
        분산 트랜잭션 없이도 정확한 시스템 구축 가능
```

**Q2: End-to-End Argument가 데이터 시스템에서 중요한 이유는?**

```
A: 하위 계층의 보장만으로는 전체 정확성을 보장할 수 없기 때문입니다.

   예: TCP 중복 억제
   • TCP 연결 내에서만 동작
   • 연결 끊김 후 재시도 → 애플리케이션 레벨 중복 발생

   예: DB 트랜잭션
   • 클라이언트 타임아웃 시 커밋 여부 불명
   • 사용자의 의도적 재시도 → 중복 실행 가능

   해결책: 클라이언트에서 생성한 Request ID를
          전체 경로에 걸쳐 전달하여 End-to-End 멱등성 보장
```

**Q3: Timeliness와 Integrity의 차이는 무엇이고, 왜 분리해서 생각해야 하나요?**

```
A:
   Timeliness: 최신 상태를 즉시 관찰할 수 있는가
   • 위반 = "Eventual Consistency" (기다리면 해결)
   • 예: 방금 쓴 데이터가 아직 안 보임

   Integrity: 데이터가 손상/모순 없이 정확한가
   • 위반 = "Perpetual Inconsistency" (명시적 수리 필요)
   • 예: 인덱스 누락, 잔액 불일치

   분리가 중요한 이유:
   • 대부분의 앱에서 Integrity가 훨씬 중요
   • Timeliness 포기 → 조정 비용 대폭 감소
   • 비동기 시스템도 강한 Integrity 보장 가능
   • 비즈니스적으로 약간의 지연은 허용되는 경우 많음
```

**Q4: 데이터베이스 Unbundling의 장단점은?**

```
A:
   장점:
   • 각 컴포넌트 독립적 확장/최적화 가능
   • 느슨한 결합으로 장애 격리
   • 팀별 독립 개발/배포 가능
   • 다양한 접근 패턴에 최적화된 저장소 선택

   단점:
   • 운영 복잡성 증가 (여러 시스템 관리)
   • 학습 곡선
   • 조합 시 일관성 보장 어려움

   핵심 원칙:
   • 필요 없으면 하지 마라 (단일 DB로 충분하면 그냥 사용)
   • Unbundling은 깊이가 아닌 넓이에 대한 것
   • CDC + Event Log로 쓰기 동기화가 핵심
```
