# Chapter 11: Batch Processing

---

## 📌 핵심 요약

> **배치 처리(Batch Processing)**는 대량의 데이터를 입력받아 새로운 출력 데이터를 생성하는 오프라인 처리 방식이다. 입력을 **불변(immutable)**으로 취급하고 부작용을 피함으로써, 버그가 발생해도 코드를 롤백하고 재실행하면 정상 출력을 얻을 수 있는 **Human Fault Tolerance**를 제공한다. 분산 배치 처리는 **분산 파일시스템**(HDFS, S3), **작업 오케스트레이션**(YARN, Kubernetes), **처리 모델**(MapReduce, Dataflow Engine)의 3가지 핵심 요소로 구성된다.

## 📖 본문 정리

### 1. 데이터 처리 시스템의 분류

```
┌─────────────────────────────────────────────────────────────────┐
│                    데이터 처리 시스템 분류                        │
├─────────────────┬─────────────────┬─────────────────────────────┤
│   Online System │  Batch System   │     Stream System           │
├─────────────────┼─────────────────┼─────────────────────────────┤
│ 요청-응답 모델   │ 입력→처리→출력   │ 무한 입력 연속 처리          │
│ 빠른 응답시간    │ 높은 처리량      │ 낮은 지연시간               │
│ 고가용성 필요    │ 장애 허용        │ 실시간성 필요               │
│                 │                 │                             │
│ 예: 웹서버, API  │ 예: ETL, ML학습  │ 예: 실시간 분석             │
└─────────────────┴─────────────────┴─────────────────────────────┘
```

> 💬 **비유**: Online 시스템은 **패스트푸드점**(주문 즉시 제공), Batch 시스템은 **급식소**(대량 조리 후 일괄 제공), Stream 시스템은 **컨베이어 벨트 초밥집**(계속 흘러가며 원하는 것을 집음)과 같다.

### 2. 배치 처리의 핵심 특성

#### 불변 입력과 Human Fault Tolerance

```
┌──────────────────────────────────────────────────────────────┐
│                    배치 처리의 장점                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  입력 데이터 (불변)                                           │
│       │                                                      │
│       ▼                                                      │
│  ┌─────────────────┐                                         │
│  │   배치 작업      │──→ 출력 v1 (버그 발견!)                  │
│  │   (버그 있음)    │                                         │
│  └─────────────────┘                                         │
│       │                                                      │
│       │ 코드 수정 후 재실행                                    │
│       ▼                                                      │
│  ┌─────────────────┐                                         │
│  │   배치 작업      │──→ 출력 v2 (정상!) ✅                    │
│  │   (버그 수정)    │                                         │
│  └─────────────────┘                                         │
│                                                              │
│  ⚠️ OLTP DB와 다름: 잘못된 쓰기는 코드 롤백으로 복구 불가       │
└──────────────────────────────────────────────────────────────┘
```

**배치 처리의 이점**:
| 특성 | 설명 |
|------|------|
| **롤백 용이** | 출력 삭제 후 재실행으로 복구 (Time Travel) |
| **Agile 개발** | 실수해도 되돌릴 수 있어 빠른 개발 가능 |
| **다중 활용** | 동일 입력을 여러 작업에서 재사용 가능 |
| **자원 효율** | OLTP보다 훨씬 저렴한 리소스 사용 |

### 3. Unix 도구를 이용한 배치 처리

#### 로그 분석 예제

```bash
# 웹서버 로그에서 가장 많이 요청된 URL Top 5 찾기
cat /var/log/nginx/access.log |  # 1. 로그 읽기
  awk '{print $7}' |             # 2. URL 필드 추출 (Map)
  sort             |             # 3. 정렬
  uniq -c          |             # 4. 카운트 (Reduce)
  sort -r -n       |             # 5. 내림차순 정렬
  head -n 5                      # 6. 상위 5개
```

```
┌─────────────────────────────────────────────────────────────┐
│                    Unix 파이프라인 흐름                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  로그파일 → awk → sort → uniq -c → sort -rn → head         │
│     │        │      │        │         │         │          │
│     │        │      │        │         │         └─► Top 5  │
│     │        │      │        │         └─► 내림차순          │
│     │        │      │        └─► 카운팅                      │
│     │        │      └─► 알파벳순 정렬                        │
│     │        └─► URL 추출                                   │
│     └─► 레코드 분리                                          │
│                                                             │
│  💬 비유: 조립라인 - 각 공정이 전 공정의 출력을 입력으로 받음   │
└─────────────────────────────────────────────────────────────┘
```

#### 정렬 vs 인메모리 집계

```
┌─────────────────────────────────────────────────────────────┐
│              정렬 기반 vs 해시 테이블 기반 집계                 │
├──────────────────────────┬──────────────────────────────────┤
│    해시 테이블 방식        │        정렬 방식                  │
│    (Python dict)         │        (Unix sort)              │
├──────────────────────────┼──────────────────────────────────┤
│ 메모리: 고유 URL 수에 비례  │ 메모리: 청크 크기에 비례           │
│ 소규모 데이터에 적합        │ 대규모 데이터에 적합              │
│ 메모리 부족 시 실패         │ 디스크 스필 자동 처리             │
│                          │ (Mergesort 활용)                 │
├──────────────────────────┴──────────────────────────────────┤
│ GNU sort: 메모리 초과 시 자동 디스크 스필, 멀티코어 병렬 정렬   │
└─────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 해시 테이블은 **책상 위 서류 정리함**(공간 제한), 정렬 방식은 **창고와 연결된 서류 정리**(필요시 창고 활용)와 같다.

### 5. 분산 파일시스템 (DFS)

#### HDFS 아키텍처

```
┌───────────────────────────────────────────────────────────────────┐
│                      HDFS 아키텍처                                 │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│                    ┌─────────────────┐                            │
│                    │   NameNode      │  ← 메타데이터 관리           │
│                    │  (메타데이터)    │    (파일→블록 매핑)         │
│                    └────────┬────────┘                            │
│                             │                                     │
│        ┌────────────────────┼────────────────────┐                │
│        ▼                    ▼                    ▼                │
│  ┌───────────┐       ┌───────────┐       ┌───────────┐           │
│  │ DataNode1 │       │ DataNode2 │       │ DataNode3 │           │
│  │ ┌───┬───┐ │       │ ┌───┬───┐ │       │ ┌───┬───┐ │           │
│  │ │B1 │B2 │ │       │ │B1 │B3 │ │       │ │B2 │B3 │ │           │
│  │ └───┴───┘ │       │ └───┴───┘ │       │ └───┴───┘ │           │
│  └───────────┘       └───────────┘       └───────────┘           │
│                                                                   │
│  💬 각 블록(128MB)은 3개 노드에 복제됨 (Replication Factor=3)       │
└───────────────────────────────────────────────────────────────────┘
```

**DFS 특징**:
| 항목 | 로컬 FS | 분산 FS |
|------|---------|---------|
| **블록 크기** | 4KB (ext4) | 128MB (HDFS), 4MB (JuiceFS) |
| **복제** | RAID | 다중 노드 복제 / Erasure Coding |
| **메타데이터** | inode | NameNode / 메타데이터 서비스 |
| **캐싱** | 페이지 캐시 | DataNode 페이지 캐시 + 클라이언트 캐시 |

#### 오브젝트 스토어 (S3)

```
┌───────────────────────────────────────────────────────────────┐
│                 오브젝트 스토어 vs 분산 FS                      │
├────────────────────────┬──────────────────────────────────────┤
│     분산 파일시스템      │        오브젝트 스토어                 │
├────────────────────────┼──────────────────────────────────────┤
│ 디렉토리 계층 구조       │ 플랫 네임스페이스 (키-값)              │
│ 파일 핸들/seek 지원     │ GET/PUT만 지원                       │
│ 파일 수정 가능          │ 불변 (전체 재작성만 가능)              │
│ 원자적 rename 지원      │ rename = copy + delete              │
│ 빈 디렉토리 가능        │ 빈 디렉토리 없음                      │
│ 데이터 로컬리티 활용     │ 스토리지-컴퓨트 분리                   │
├────────────────────────┴──────────────────────────────────────┤
│  s3://my-bucket/2025/04/01/file.png                          │
│       └─버킷─┘  └───────키(경로처럼 보이지만 단순 문자열)────┘    │
└───────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: DFS는 **도서관 서가**(계층적 분류, 책 위치 이동 가능), 오브젝트 스토어는 **창고 보관함**(고유 번호로만 접근, 물건 교체는 전체 교체)과 같다.

### 8. MapReduce

#### MapReduce 실행 흐름

```
┌───────────────────────────────────────────────────────────────────┐
│                    MapReduce 4단계 실행 흐름                        │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  1. 레코드 분리 (Input Format)                                     │
│     파일 → 레코드들                                                │
│                                                                   │
│  2. Map 함수 (사용자 정의)                                         │
│     레코드 → (key, value) 쌍들                                     │
│                                                                   │
│  3. Shuffle & Sort (프레임워크 자동)                               │
│     모든 (key, value)를 key 기준 정렬                              │
│                                                                   │
│  4. Reduce 함수 (사용자 정의)                                      │
│     (key, [values...]) → 출력 레코드                              │
│                                                                   │
├───────────────────────────────────────────────────────────────────┤
│                         예시: URL 카운팅                           │
│                                                                   │
│  입력: 로그 라인들                                                 │
│    ↓                                                              │
│  Map: line → (URL, 1)                                            │
│    ↓                                                              │
│  Shuffle: 같은 URL끼리 모음                                        │
│    ↓                                                              │
│  Reduce: (URL, [1,1,1,...]) → (URL, count)                       │
└───────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: MapReduce는 **선거 개표**와 같다. Map은 각 투표용지에서 후보명을 읽는 것, Shuffle은 같은 후보끼리 분류하는 것, Reduce는 각 후보의 총 득표수를 세는 것이다.

### 9. Dataflow 엔진 (Spark, Flink)

#### MapReduce vs Dataflow 엔진

```
┌───────────────────────────────────────────────────────────────────┐
│               MapReduce vs Dataflow 엔진 비교                      │
├─────────────────────────┬─────────────────────────────────────────┤
│       MapReduce         │        Dataflow Engine                  │
│                         │        (Spark, Flink)                   │
├─────────────────────────┼─────────────────────────────────────────┤
│ Map → Shuffle → Reduce  │ 유연한 연산자 조합                       │
│ 고정된 2단계            │ (filter, join, groupBy, ...)           │
├─────────────────────────┼─────────────────────────────────────────┤
│ 중간 결과 DFS 저장       │ 메모리/로컬 디스크 저장                  │
├─────────────────────────┼─────────────────────────────────────────┤
│ 작업마다 새 JVM 시작     │ 프로세스 재사용                          │
├─────────────────────────┼─────────────────────────────────────────┤
│ 단계별 완료 후 다음 시작  │ 파이프라이닝 (스트리밍 실행)             │
├─────────────────────────┼─────────────────────────────────────────┤
│ 불필요한 정렬 발생       │ 필요한 곳에서만 정렬                     │
└─────────────────────────┴─────────────────────────────────────────┘
```

#### Dataflow 엔진의 최적화

```
┌───────────────────────────────────────────────────────────────┐
│              Dataflow 엔진 최적화 기법                          │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  1. 연산자 퓨전 (Operator Fusion)                              │
│     ┌──────┐   ┌──────┐   ┌──────┐                           │
│     │filter│ → │ map  │ → │filter│  ← 3개를 1개로 합침         │
│     └──────┘   └──────┘   └──────┘                           │
│                    ↓                                          │
│     ┌────────────────────────────────┐                        │
│     │  fused(filter → map → filter)  │                        │
│     └────────────────────────────────┘                        │
│                                                               │
│  2. 로컬리티 최적화                                             │
│     데이터 생산 태스크와 소비 태스크를 같은 노드에 배치           │
│     → 네트워크 I/O 감소                                        │
│                                                               │
│  3. 지연 정렬 (Lazy Sorting)                                   │
│     필요할 때만 정렬 수행                                       │
│                                                               │
│  4. 파이프라이닝                                                │
│     이전 단계 완료 전에 다음 단계 시작                           │
└───────────────────────────────────────────────────────────────┘
```

### 10. Shuffle 알고리즘

#### 분산 Shuffle 과정

```
┌───────────────────────────────────────────────────────────────────┐
│                      Shuffle 데이터 흐름                           │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│   Map Phase                              Reduce Phase             │
│                                                                   │
│  ┌──────────┐                          ┌──────────┐              │
│  │ Mapper 1 │─────┐              ┌─────│Reducer 1 │              │
│  │  (m1)    │     │              │     │   (r1)   │              │
│  └──────────┘     │              │     └──────────┘              │
│       │           │              │                                │
│       │     ┌─────┼──────────────┤                                │
│       │     │     │              │                                │
│  ┌────▼─────▼┐    │         ┌────▼─────┐                          │
│  │ m1→r1    │    │         │  r2      │                          │
│  │ m1→r2    │    │         │ ┌──────┐ │                          │
│  │ m1→r3    │    └─────────┼─│merge │ │                          │
│  └──────────┘              │ └──────┘ │                          │
│                            └──────────┘                          │
│  ┌──────────┐                                                     │
│  │ Mapper 2 │─── 같은 방식으로 r1, r2, r3에 파티션 ───            │
│  └──────────┘                                                     │
│                                                                   │
│  핵심: hash(key) % num_reducers → 어느 리듀서로 갈지 결정          │
└───────────────────────────────────────────────────────────────────┘
```

**Shuffle 단계**:
1. **Map 출력 파티셔닝**: `hash(key) % R`로 리듀서 결정
2. **로컬 정렬**: 메모리에서 정렬, 필요시 디스크 스필
3. **네트워크 전송**: 리듀서가 모든 매퍼에서 자신의 파티션 가져옴
4. **병합 정렬**: 여러 매퍼의 출력을 Mergesort로 병합

### 11. 분산 JOIN과 GROUP BY

#### Sort-Merge Join

```
┌───────────────────────────────────────────────────────────────────┐
│                    Sort-Merge Join 예시                            │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  Activity Events (Fact)           User Database (Dimension)       │
│  ┌────────────────────┐          ┌─────────────────────┐         │
│  │user_id│page_view   │          │user_id│birth_date   │         │
│  ├───────┼────────────┤          ├───────┼─────────────┤         │
│  │  123  │ /home      │          │  123  │ 1990-05-15  │         │
│  │  456  │ /products  │          │  456  │ 1985-12-20  │         │
│  │  123  │ /checkout  │          │  789  │ 1995-03-10  │         │
│  └───────┴────────────┘          └───────┴─────────────┘         │
│         │                                 │                       │
│         └────────────┬────────────────────┘                       │
│                      ▼                                            │
│               Map: user_id를 key로                                │
│                      │                                            │
│                      ▼                                            │
│                   Shuffle                                         │
│                      │                                            │
│                      ▼                                            │
│               같은 user_id가 같은 Reducer로                        │
│                      │                                            │
│                      ▼                                            │
│  Reducer 1: user_id=123                                          │
│  ┌─────────────────────────────────────────┐                     │
│  │ (123, birth_date: 1990-05-15)          │ ← 먼저 도착          │
│  │ (123, page_view: /home)                │                      │
│  │ (123, page_view: /checkout)            │                      │
│  └─────────────────────────────────────────┘                     │
│           ↓                                                       │
│  출력: (123, 1990-05-15, /home)                                   │
│        (123, 1990-05-15, /checkout)                              │
└───────────────────────────────────────────────────────────────────┘
```

**Secondary Sort**: 리듀서 내에서 레코드 순서 보장 (예: 사용자 정보가 먼저)

> 💬 **비유**: Sort-Merge Join은 **우체국 분류**와 같다. 각 우체국(Mapper)에서 우편번호별로 분류하고, 중앙 배송센터(Reducer)에서 같은 우편번호의 편지들을 모아 처리한다.

### 면접 질문 4선

1. **MapReduce의 Shuffle 과정에서 발생할 수 있는 성능 병목과 해결 방법은?**
   - 네트워크 I/O 병목: Combiner 사용, 압축
   - 데이터 스큐: Salting, Adaptive Query Execution
   - 디스크 I/O: 메모리 버퍼 증가, SSD 사용

2. **Spark의 Lazy Evaluation이 제공하는 이점은?**
   - 전체 실행 계획 확인 후 최적화 가능
   - 불필요한 연산 제거 (Predicate Pushdown)
   - 연산자 퓨전으로 중간 데이터 최소화
   - 파이프라이닝으로 메모리 효율 증가

3. **배치 처리에서 Exactly-Once 시맨틱을 어떻게 보장하는가?**
   - 입력 불변성 + 멱등성 출력
   - Flink: 체크포인트 기반 장애 복구
   - Spark: RDD Lineage 추적으로 재계산
   - 외부 시스템: 트랜잭션 또는 멱등 쓰기

4. **Data Lakehouse 아키텍처가 기존 Data Lake와 다른 점은?**
   - ACID 트랜잭션 지원 (Delta Lake, Iceberg)
   - 스키마 진화 및 Time Travel 지원
   - 인덱싱으로 쿼리 성능 향상
   - SQL 기반 분석과 ML 워크로드 통합
