# Chapter 12: Stream Processing

---

## 📌 핵심 요약

> **스트림 처리(Stream Processing)**는 무한히 흘러오는 이벤트 데이터를 연속적으로 처리하는 방식이다. 배치 처리가 유한한 입력을 대상으로 하는 반면, 스트림 처리는 **무한한(unbounded)** 데이터를 다룬다. **메시지 브로커**(AMQP/JMS 스타일, 로그 기반)를 통해 이벤트를 전달하고, **CDC(Change Data Capture)**로 데이터베이스 변경을 스트림으로 캡처하며, **윈도우 연산**, **스트림 조인**, **장애 복구**(마이크로배칭, 체크포인팅, 멱등성)를 통해 정확한 결과를 보장한다.

## 📖 본문 정리

### 1. 배치 처리 vs 스트림 처리

```
┌───────────────────────────────────────────────────────────────────┐
│                    배치 vs 스트림 처리 비교                         │
├─────────────────────────┬─────────────────────────────────────────┤
│      배치 처리           │          스트림 처리                     │
├─────────────────────────┼─────────────────────────────────────────┤
│ 유한한(bounded) 입력     │ 무한한(unbounded) 입력                   │
│ 전체 입력 처리 후 출력    │ 이벤트 도착 즉시 처리                    │
│ 작업 종료 시점 명확       │ 작업이 영원히 실행                       │
│ 정렬 가능                │ 정렬 불가 (전체 입력을 알 수 없음)         │
│ 지연 허용 (시간~일 단위)  │ 저지연 요구 (밀리초~초 단위)              │
├─────────────────────────┴─────────────────────────────────────────┤
│                                                                   │
│  배치: ─────[데이터]───── → [처리] → [출력]                        │
│              (유한)                                               │
│                                                                   │
│  스트림: ────→●→●→●→●→●→ [처리] → ●→●→●→                         │
│              (무한, 연속)                                          │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 배치 처리는 **책을 다 읽고 독후감 쓰기**, 스트림 처리는 **뉴스 속보를 실시간으로 전달**하는 것과 같다.

**스트림 처리가 필요한 이유**:
- 일별 배치는 결과 반영까지 하루가 걸림
- 사용자는 더 빠른 피드백을 원함
- 이벤트 발생 즉시 처리해야 하는 경우 존재 (사기 탐지, 트레이딩 등)

### 3. 메시징 시스템

#### 메시지 전달 시 고려사항

```
┌───────────────────────────────────────────────────────────────┐
│              메시징 시스템 설계 시 핵심 질문                     │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  Q1. Producer가 Consumer보다 빠르면?                           │
│  ┌────────────┬────────────┬────────────────┐                 │
│  │ 메시지 삭제 │ 큐에 버퍼링 │ 백프레셔 적용   │                 │
│  │ (손실 허용) │ (메모리/디스크)│ (생산자 차단) │                 │
│  └────────────┴────────────┴────────────────┘                 │
│                                                               │
│  Q2. 노드 장애 시 메시지 손실?                                  │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │ 내구성 보장 = 디스크 쓰기 + 복제 → 비용 증가             │  │
│  │ 손실 허용 = 더 높은 처리량, 낮은 지연시간                 │  │
│  └─────────────────────────────────────────────────────────┘  │
│                                                               │
│  💡 센서 데이터: 가끔 손실 괜찮음 (다음 값이 곧 옴)             │
│  💡 결제 이벤트: 손실 불가 (정확한 카운트 필요)                 │
└───────────────────────────────────────────────────────────────┘
```

#### Direct Messaging vs Message Broker

```
┌───────────────────────────────────────────────────────────────┐
│                    메시징 아키텍처 비교                         │
├─────────────────────────────┬─────────────────────────────────┤
│      Direct Messaging       │       Message Broker            │
├─────────────────────────────┼─────────────────────────────────┤
│ Producer → Consumer 직접 연결 │ Producer → Broker → Consumer   │
│                             │                                 │
│ UDP Multicast (금융)        │ RabbitMQ, ActiveMQ              │
│ ZeroMQ, nanomsg             │ Kafka, Amazon Kinesis           │
│ StatsD (메트릭)             │ Google Cloud Pub/Sub            │
│ Webhooks (HTTP 콜백)        │                                 │
├─────────────────────────────┼─────────────────────────────────┤
│ 장점: 낮은 지연시간          │ 장점: 내구성, 클라이언트 분리    │
│ 단점: 오프라인 Consumer      │ 단점: 추가 홉, 관리 오버헤드    │
│       메시지 처리 불가        │                                 │
└─────────────────────────────┴─────────────────────────────────┘
```

> 💬 **비유**: Direct Messaging은 **전화 통화**(양쪽 다 통화 가능 상태여야 함), Message Broker는 **우체통**(부재중에도 편지 보관)과 같다.

### 4. 다중 Consumer 패턴

```
┌───────────────────────────────────────────────────────────────┐
│                   다중 Consumer 패턴                           │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  (a) Load Balancing - 작업 분산                               │
│  ┌─────────┐                                                  │
│  │ Topic   │ ──→ Consumer 1 (m1, m3, m5...)                  │
│  │ m1→m6   │ ──→ Consumer 2 (m2, m4, m6...)                  │
│  └─────────┘                                                  │
│  각 메시지는 하나의 Consumer에게만 전달                         │
│                                                               │
│  (b) Fan-out - 브로드캐스트                                   │
│  ┌─────────┐ ──→ Consumer Group A (모든 메시지)               │
│  │ Topic   │ ──→ Consumer Group B (모든 메시지)               │
│  │ m1→m6   │ ──→ Consumer Group C (모든 메시지)               │
│  └─────────┘                                                  │
│  각 Consumer 그룹은 모든 메시지 수신                           │
│                                                               │
│  (c) Kafka Consumer Groups - 조합                             │
│  ┌─────────┐                                                  │
│  │ Topic   │ ──→ Group A: C1(m1,m3), C2(m2,m4) ← Load Balance│
│  │         │ ──→ Group B: C3(m1,m2,m3,m4)      ← Fan-out     │
│  └─────────┘                                                  │
└───────────────────────────────────────────────────────────────┘
```

#### Acknowledgment과 Redelivery

```
┌───────────────────────────────────────────────────────────────┐
│                   메시지 재전송 문제                            │
├───────────────────────────────────────────────────────────────┐
│                                                               │
│  Producer → m1 → m2 → m3 → m4 → m5 → Broker                  │
│                         │                                     │
│  Consumer 1: m1 ✓, m2 ✓, m4 처리 중                           │
│  Consumer 2: m3 처리 중 → 💥 Crash!                           │
│                         │                                     │
│                         ▼                                     │
│  m3 ACK 미수신 → Consumer 1에게 재전송                         │
│                                                               │
│  결과: Consumer 1이 m4, m3, m5 순서로 처리                     │
│        (원래 순서: m3, m4, m5)                                 │
│                                                               │
│  ⚠️ 순서 보장과 Load Balancing은 상충!                        │
│  해결: 파티션별 단일 Consumer 또는 순서 무관한 메시지 설계       │
└───────────────────────────────────────────────────────────────┘
```

**Dead Letter Queue (DLQ)**:
- 처리 실패한 메시지를 별도 큐로 이동
- 무한 재시도 방지
- 운영자가 수동으로 검토/수정

### 5. 로그 기반 메시지 브로커

#### AMQP/JMS vs Log-based 비교

```
┌───────────────────────────────────────────────────────────────────┐
│              메시지 브로커 스타일 비교                              │
├─────────────────────────────┬─────────────────────────────────────┤
│    AMQP/JMS 스타일          │      로그 기반 (Kafka)              │
│  (RabbitMQ, ActiveMQ)       │   (Kafka, Kinesis, Redpanda)       │
├─────────────────────────────┼─────────────────────────────────────┤
│ ACK 후 메시지 삭제           │ 메시지 보존 (설정된 기간/크기)       │
│ 메시지 단위 할당             │ 파티션 단위 할당                     │
│ 순서 보장 어려움 (재전송 시)  │ 파티션 내 순서 보장                  │
│ 과거 메시지 재소비 불가       │ 오프셋 조정으로 재소비 가능          │
│ 임시 메시징 사상             │ 영구 로그 사상                       │
├─────────────────────────────┼─────────────────────────────────────┤
│ 적합: 작업 큐, RPC 대체      │ 적합: 이벤트 소싱, CDC, 로그 처리    │
│      메시지별 처리 비용 높음  │      높은 처리량, 순서 중요          │
└─────────────────────────────┴─────────────────────────────────────┘
```

#### Kafka 로그 구조

```
┌───────────────────────────────────────────────────────────────────┐
│                    Kafka Topic-Partition 구조                      │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  Topic: user-events                                               │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │ Partition 0: [0][1][2][3][4][5][6][7]... → (무한 추가)       │  │
│  │                          ↑                                   │  │
│  │                   Consumer Offset                            │  │
│  └─────────────────────────────────────────────────────────────┘  │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │ Partition 1: [0][1][2][3][4][5]...                          │  │
│  └─────────────────────────────────────────────────────────────┘  │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │ Partition 2: [0][1][2][3]...                                │  │
│  └─────────────────────────────────────────────────────────────┘  │
│                                                                   │
│  특징:                                                            │
│  • 파티션 내 순서 보장 (Offset = 단조 증가 시퀀스 번호)            │
│  • 파티션 간 순서 보장 없음                                        │
│  • Consumer는 Offset만 저장 (메시지별 ACK 불필요)                  │
│  • 파티션 수 = 최대 병렬 Consumer 수                              │
└───────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: Kafka는 **무한히 늘어나는 노트**와 같다. 각 페이지(메시지)에 번호가 매겨져 있고, 여러 사람이 각자 "어디까지 읽었는지" 표시만 하면 된다. 노트 자체는 지우지 않는다.

#### Consumer Offset의 의미

```
┌───────────────────────────────────────────────────────────────┐
│                Consumer Offset과 DB 복제 비교                   │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  Kafka Consumer           │  DB Replication                   │
│  ────────────────────────────────────────────────────────────│
│  Consumer Offset          │  Log Sequence Number (LSN)        │
│  Message Broker           │  Leader DB                        │
│  Consumer                 │  Follower DB                      │
│                                                               │
│  동작:                                                         │
│  1. Consumer 장애 발생                                         │
│  2. 다른 Consumer가 마지막 기록된 Offset부터 재개               │
│  3. 아직 기록 안 된 메시지는 중복 처리 가능                     │
│                                                               │
│  💡 DB 복제와 동일한 원리 - 일관된 상태 복구 가능               │
└───────────────────────────────────────────────────────────────┘
```

#### 디스크 공간 관리

├───────────────────────────────────────────────────────────────┤

### 6. 데이터베이스와 스트림

#### 시스템 간 동기화 문제

```
┌───────────────────────────────────────────────────────────────┐
│                    Dual Write의 문제점                         │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  Client 1: X = A                  Client 2: X = B             │
│      │                                │                       │
│      ▼                                ▼                       │
│  ┌────────┐                      ┌────────┐                   │
│  │Database│ ① X=A  ③ X=B        │Database│                   │
│  │ 최종:B │◄──────────────────── │        │                   │
│  └────────┘                      └────────┘                   │
│      │                                │                       │
│      │ ④ X=A                         │ ② X=B                  │
│      ▼                                ▼                       │
│  ┌─────────┐                     ┌─────────┐                  │
│  │ Search  │ ← 최종: A (불일치!)  │ Search  │                  │
│  │ Index   │◄──────────────────── │ Index   │                  │
│  └─────────┘                     └─────────┘                  │
│                                                               │
│  ⚠️ 문제: 동시 쓰기 시 DB와 Search Index가 불일치             │
│  ⚠️ 문제: 한쪽 쓰기 실패 시 불일치 (원자적 커밋 필요)          │
└───────────────────────────────────────────────────────────────┘
```

#### Change Data Capture (CDC)

```
┌───────────────────────────────────────────────────────────────────┐
│                    CDC를 통한 동기화 해결                          │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  Client 1: X=A    Client 2: X=B                                   │
│       │               │                                           │
│       └───────┬───────┘                                           │
│               ▼                                                   │
│          ┌─────────┐                                              │
│          │ Database│  (단일 진실의 원천)                           │
│          │  X=B    │  (DB가 순서 결정)                            │
│          └────┬────┘                                              │
│               │                                                   │
│               ▼  CDC (변경 캡처)                                   │
│          ┌─────────┐                                              │
│          │ Change  │  [X=A] → [X=B]                               │
│          │  Log    │  (DB 쓰기 순서대로)                           │
│          └────┬────┘                                              │
│               │                                                   │
│      ┌────────┼────────┐                                          │
│      ▼        ▼        ▼                                          │
│  ┌───────┐┌───────┐┌───────┐                                      │
│  │Search ││ Cache ││ Data  │  모두 동일한 순서로 적용              │
│  │Index  ││       ││ WH    │  → 일관성 보장!                       │
│  │ X=B   ││ X=B   ││ X=B   │                                      │
│  └───────┘└───────┘└───────┘                                      │
│                                                                   │
│  구현: Debezium, Maxwell, Kafka Connect                           │
│  원리: DB Replication Log 파싱 (binlog, WAL)                       │
└───────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: CDC는 **CCTV 녹화**와 같다. 은행(DB)에서 일어난 모든 거래가 순서대로 기록되고, 다른 지점들(파생 시스템)은 그 녹화를 순서대로 재생하여 동일한 상태를 유지한다.

#### Log Compaction

```
┌───────────────────────────────────────────────────────────────┐
│                     Log Compaction 원리                        │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  원본 로그:                                                    │
│  [mew:1][purr:2][mew:3][scratch:5][purr:7][mew:9]             │
│                                                               │
│  Compaction 후:                                                │
│  [scratch:5][purr:7][mew:9]                                   │
│                                                               │
│  원리:                                                         │
│  • 같은 키에 대해 최신 값만 유지                                │
│  • 삭제 = tombstone (null 값) 기록                            │
│  • 디스크 공간 = 현재 데이터 크기에만 의존                      │
│                                                               │
│  장점:                                                         │
│  • 새 Consumer가 offset 0부터 시작해도 전체 상태 복구 가능     │
│  • 스냅샷 없이 데이터베이스 전체 복제 가능                      │
└───────────────────────────────────────────────────────────────┘
```

#### CDC vs Event Sourcing

```
┌───────────────────────────────────────────────────────────────┐
│                  CDC vs Event Sourcing 비교                    │
├──────────────────────────┬────────────────────────────────────┤
│         CDC              │         Event Sourcing             │
├──────────────────────────┼────────────────────────────────────┤
│ DB의 가변 데이터 모델     │ 불변 이벤트만 저장                  │
│ 낮은 수준 (행 변경)       │ 높은 수준 (사용자 의도)             │
│ DB 리플리케이션 로그 파싱  │ 애플리케이션이 이벤트 직접 기록     │
│ 기존 앱에 쉽게 추가       │ 앱 아키텍처 변경 필요              │
│ Log Compaction 가능      │ 전체 이벤트 히스토리 유지 필요      │
├──────────────────────────┴────────────────────────────────────┤
│                                                               │
│  CDC 이벤트 예시:                                              │
│  { op: "UPDATE", table: "users",                              │
│    before: {balance: 100}, after: {balance: 150} }            │
│                                                               │
│  Event Sourcing 이벤트 예시:                                   │
│  { type: "MoneyDeposited", userId: 123, amount: 50 }          │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 7. 상태, 스트림, 불변성

#### 상태와 스트림의 이중성

```
┌───────────────────────────────────────────────────────────────┐
│                  상태와 스트림의 수학적 관계                    │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│                      ∫                                        │
│  이벤트 스트림 ─────────────▶ 현재 상태                        │
│              (시간에 대한 적분)                                │
│                                                               │
│                      d/dt                                     │
│  현재 상태    ─────────────▶ 변경 스트림                       │
│              (시간에 대한 미분)                                │
│                                                               │
│  예시:                                                         │
│  • 계좌 잔액 = Σ(입금 이벤트) - Σ(출금 이벤트)                 │
│  • 좌석 현황 = Σ(좌석 상태 변경 이벤트)                        │
│  • 응답시간 그래프 = 개별 요청 응답시간들의 집계               │
│                                                               │
│  "모든 상태는 그것을 만든 이벤트들의 결과다"                    │
│                                                               │
│  💬 Jim Gray: "로그가 모든 정보를 담고 있다.                   │
│              DB는 성능을 위한 캐시일 뿐이다."                  │
└───────────────────────────────────────────────────────────────┘
```

#### 불변 이벤트의 장점

```
┌───────────────────────────────────────────────────────────────┐
│                   불변 이벤트의 이점                           │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  1. 감사 추적 (Audit Trail)                                   │
│     회계: 잘못된 거래를 삭제하지 않고 반대 거래 추가           │
│     → 모든 히스토리 보존                                      │
│                                                               │
│  2. 버그 복구                                                  │
│     버그 코드가 잘못된 데이터 기록                             │
│     → 이벤트 로그에서 복원 가능                               │
│                                                               │
│  3. 분석 가치                                                  │
│     장바구니 담기 → 삭제 이벤트                                │
│     • 주문 시스템: 삭제 = 정보 없음                           │
│     • 분석 시스템: 담았다 취소 = 유의미한 사용자 행동         │
│                                                               │
│  4. 다중 뷰 생성 (CQRS)                                       │
│     하나의 이벤트 로그 → 여러 읽기 최적화 뷰                   │
│     • 검색 인덱스                                             │
│     • 분석용 OLAP 큐브                                        │
│     • 실시간 대시보드                                         │
└───────────────────────────────────────────────────────────────┘
```

#### 불변성의 한계

- **데이터 양**: 업데이트가 빈번한 경우 로그가 과도하게 커짐
- **법적 삭제 요구**: GDPR의 삭제권 등
- **실제 삭제**: Datomic의 Excision, Fossil의 Shunning
- **Crypto-Shredding**: 암호화 후 키 폐기로 "삭제" 구현

├───────────────────────────────────────────────────────────────┤
│                                                               │
│  이벤트 시간 (Event Time):                                     │

│                                                               │

│  ┌─────────────────────────────────────────────────────────┐  │
│  │ 실제 요청:    ●──●──●──●──●──●──●  (일정한 속도)         │  │
│  │                                                         │  │
│  │ 처리기 재시작:        💥                                 │  │
│  │                       ▼                                 │  │
│  │ 처리 시간 기준:    ●  ●●●●●●  ●    (스파이크 발생!)       │  │
│  │                       백로그 처리                        │  │

│  💬 Star Wars 비유:                                           │
│     Episode IV(1977) → V(1980) → VI(1983)                    │
│     → I(1999) → II(2002) → III(2005)                         │
│     개봉일(처리시간) ≠ 스토리 순서(이벤트시간)                  │

```

#### Straggler Events 처리

```
┌───────────────────────────────────────────────────────────────┐
│                  늦게 도착하는 이벤트 처리                       │

│                                                               │
│  시나리오:                                                     │
│  • 10:37분 윈도우 집계 중                                      │

│  │    • "t 이전 이벤트는 더 이상 없다" 선언               │    │

#### 모바일 디바이스 시계 문제

┌───────────────────────────────────────────────────────────────┐
│               디바이스 시계 보정 기법                           │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  세 개의 타임스탬프 기록:                                      │
│                                                               │
│  t1: 이벤트 발생 시각 (디바이스 시계)                          │

│            └─────────┬─────────┘                                  │

### 12. 장애 허용

#### Microbatching vs Checkpointing

```
┌───────────────────────────────────────────────────────────────────┐
│                    장애 복구 메커니즘 비교                          │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  1. Microbatching (Spark Streaming)                               │
│     ┌────┐┌────┐┌────┐┌────┐┌────┐                               │
│     │ B1 ││ B2 ││ B3 ││ B4 ││ B5 │  (1초 단위 미니 배치)          │
│     └────┘└────┘└────┘└────┘└────┘                               │
│                    💥                                             │
│                 B3 실패 → B3만 재처리                             │
│                                                                   │
│     장점: 배치와 동일한 장애 복구 모델                             │
│     단점: 배치 크기 = 최소 지연 시간                              │
│                                                                   │
│  2. Checkpointing (Flink)                                         │
│     ───●───●───●───●───║───●───●───●───●───║───●───              │
│                        ↑                   ↑                      │
│                   Checkpoint 1        Checkpoint 2                │
│                        💥                                         │
│     장애 발생 → 마지막 Checkpoint부터 재개                        │
│                                                                   │
│     장점: 윈도우 크기와 무관                                       │
│     단점: Checkpoint 사이 이벤트는 재처리됨                        │
└───────────────────────────────────────────────────────────────────┘
```

#### Exactly-Once 시맨틱

```
┌───────────────────────────────────────────────────────────────────┐
│                    Exactly-Once 달성 방법                          │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  목표: 장애가 발생해도 각 이벤트가 "딱 한 번" 처리된 것처럼 보임    │
│                                                                   │
│  방법 1: Atomic Commit (트랜잭션)                                  │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │ 하나의 트랜잭션 내에서:                                     │   │
│  │ • 입력 메시지 ACK                                          │   │
│  │ • 상태 업데이트                                            │   │
│  │ • 출력 메시지 전송                                         │   │
│  │                                                            │   │
│  │ 구현: Kafka Transactions, Flink 2PC                        │   │
│  │ 제한: 이기종 시스템 간 트랜잭션은 어려움                    │   │
│  └────────────────────────────────────────────────────────────┘   │
│                                                                   │
│  방법 2: Idempotence (멱등성)                                      │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │ 같은 연산을 여러 번 수행해도 결과 동일                      │   │
│  │                                                            │   │
│  │ 예: DELETE FROM users WHERE id = 123  (멱등)              │   │
│  │     UPDATE counter SET value = value + 1 (비멱등)         │   │
│  │                                                            │   │
│  │ 비멱등 연산을 멱등하게 만들기:                              │   │
│  │ • 메시지 Offset을 함께 저장                                │   │
│  │ • 이미 처리된 Offset이면 스킵                              │   │
│  │                                                            │   │
│  │ 요구사항:                                                  │   │
│  │ • 동일 메시지, 동일 순서로 재생 (로그 기반 브로커)          │   │
│  │ • 결정적 처리                                              │   │
│  │ • 동시 업데이트 없음 (또는 Fencing)                        │   │
│  └────────────────────────────────────────────────────────────┘   │
└───────────────────────────────────────────────────────────────────┘
```

### 면접 질문 4선

1. **Kafka에서 Consumer Group과 Partition의 관계는? 최대 병렬 처리 수는?**
   - 하나의 파티션은 Consumer Group 내 하나의 Consumer에만 할당
   - 최대 병렬 처리 수 = 파티션 수
   - Consumer 수 > 파티션 수 → 일부 Consumer 유휴 상태
   - 해결: 파티션 수 충분히 설정, 또는 Consumer 수 조절

2. **Event Time과 Processing Time의 차이는? 왜 중요한가?**
   - Event Time: 이벤트 실제 발생 시간 (비즈니스 의미)
   - Processing Time: 처리 시스템의 로컬 시계 시간
   - Processing Time 기준 → 재처리 시 결과 달라짐 (비결정적)
   - Event Time 기준 → 결정적 결과, 재현 가능
   - Late Event 처리를 위해 Watermark 사용

3. **CDC와 Event Sourcing의 차이점은? 각각 언제 사용하는가?**
   - CDC: 기존 CRUD DB의 변경을 캡처, 낮은 수준(row 변경), 기존 앱에 쉽게 추가
   - Event Sourcing: 이벤트가 시스템의 진실의 원천, 높은 수준(비즈니스 이벤트), 앱 설계 변경 필요
   - CDC 사용: 기존 시스템 통합, 데이터 파이프라인
   - Event Sourcing 사용: 감사 추적 중요, 이벤트 기반 설계

4. **스트림 처리에서 Exactly-Once 시맨틱을 어떻게 달성하는가?**
   - 방법 1: 트랜잭션 (Kafka Transactions, Flink 2PC)
     - 입력 ACK + 상태 + 출력을 원자적으로 처리
   - 방법 2: 멱등성
     - Offset과 함께 결과 저장
     - 이미 처리된 Offset은 스킵
   - 방법 3: Checkpointing + 재처리
     - 마지막 Checkpoint부터 결정적으로 재실행
