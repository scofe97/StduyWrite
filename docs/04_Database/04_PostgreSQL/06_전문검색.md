# Chapter 6: Postgres for Full Text Search - 면접정리

## 핵심 개념 상세 설명

### 1. 전문 검색 기본 개념

```
LIKE vs Full Text Search
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 LIKE/ILIKE 】                                           │
│  SELECT * FROM posts WHERE content LIKE '%postgres%'        │
│                                                             │
│  문제점:                                                    │
│  • Full Table Scan (인덱스 활용 어려움)                     │
│  • 대소문자 구분 (ILIKE는 느림)                             │
│  • 단어 경계 무시 ('postgres'가 'postgressql'도 매칭)       │
│  • 어간(Stemming) 미지원 ('running' ≠ 'run')               │
│                                                             │
│  【 Full Text Search 】                                     │
│  SELECT * FROM posts                                        │
│  WHERE to_tsvector('english', content) @@                   │
│        to_tsquery('english', 'postgres');                   │
│                                                             │
│  장점:                                                       │
│  • GIN/GiST 인덱스 활용                                     │
│  • 언어 기반 정규화 (Stemming)                              │
│  • 불용어(Stop Words) 제거                                  │
│  • 관련도 순위 지원                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. tsvector와 tsquery

```
tsvector: 문서를 검색 가능한 형태로 변환
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  입력: "The quick brown foxes jumped over the lazy dogs"    │
│                                                             │
│  to_tsvector('english', ...)                                │
│  ↓                                                          │
│  결과: 'brown':3 'dog':10 'fox':4 'jump':5 'lazi':8         │
│        'quick':2                                            │
│                                                             │
│  처리 과정:                                                  │
│  1. 토큰화: 단어로 분리                                     │
│  2. 정규화: 소문자 변환                                     │
│  3. 불용어 제거: the, over 제거                             │
│  4. 어간 추출: foxes→fox, jumped→jump, lazy→lazi           │
│  5. 위치 저장: 각 렉셈(lexeme)의 위치 기록                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘

tsquery: 검색 조건 표현
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  to_tsquery('english', 'running')                           │
│  → 'run' (어간 추출됨)                                      │
│                                                             │
│  연산자:                                                     │
│  • & (AND): 'cat & dog' - 둘 다 포함                        │
│  • | (OR):  'cat | dog' - 하나라도 포함                     │
│  • ! (NOT): '!cat'      - 포함하지 않음                     │
│  • <-> (FOLLOWED BY): 'new <-> york' - 인접 단어            │
│  • <N> (N 단어 간격): 'quick <2> fox' - 2단어 이내          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 기본 검색
SELECT * FROM articles
WHERE to_tsvector('english', content) @@ to_tsquery('english', 'database');

-- 복합 조건
SELECT * FROM articles
WHERE to_tsvector('english', content) @@
      to_tsquery('english', 'postgres & performance');

-- 인접 검색
SELECT * FROM articles
WHERE to_tsvector('english', content) @@
      to_tsquery('english', 'new <-> york');
```

### 3. 검색 편의 함수

```sql
-- plainto_tsquery: 일반 텍스트를 AND 조건으로 변환
plainto_tsquery('english', 'fat cats')
→ 'fat' & 'cat'

-- phraseto_tsquery: 구문 검색 (인접)
phraseto_tsquery('english', 'fat cats')
→ 'fat' <-> 'cat'

-- websearch_to_tsquery: 웹 검색 스타일
websearch_to_tsquery('english', 'fat cats -dogs "new york"')
→ 'fat' & 'cat' & !'dog' & 'new' <-> 'york'

-- 웹 검색 문법:
-- 공백: AND
-- OR: 명시적 OR
-- -단어: NOT
-- "구문": 정확한 순서
```

### 4. 검색 결과 순위 (Ranking)

```
ts_rank 함수
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ts_rank(tsvector, tsquery, [normalization])                │
│                                                             │
│  가중치 (A, B, C, D):                                       │
│  • D (기본): 0.1                                            │
│  • C: 0.2                                                   │
│  • B: 0.4                                                   │
│  • A: 1.0                                                   │
│                                                             │
│  setweight(to_tsvector('english', title), 'A') ||           │
│  setweight(to_tsvector('english', content), 'D')            │
│                                                             │
│  → 제목 매치는 본문보다 10배 높은 점수                      │
│                                                             │
│  정규화 옵션:                                                │
│  • 0: 정규화 없음                                           │
│  • 1: 문서 길이로 나눔                                      │
│  • 2: 문서 길이의 로그로 나눔                               │
│  • 4: 단어 간 거리 고려                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 가중치 적용한 검색
SELECT
    title,
    ts_rank(
        setweight(to_tsvector('english', title), 'A') ||
        setweight(to_tsvector('english', content), 'B'),
        to_tsquery('english', 'database')
    ) AS rank
FROM articles
WHERE to_tsvector('english', title || ' ' || content) @@
      to_tsquery('english', 'database')
ORDER BY rank DESC;
```

### 5. 검색 결과 하이라이트 (ts_headline)

```sql
-- 검색어 하이라이팅
SELECT
    title,
    ts_headline('english', content, to_tsquery('english', 'postgres'),
        'StartSel=<b>, StopSel=</b>, MaxWords=35, MinWords=15'
    ) AS snippet
FROM articles
WHERE to_tsvector('english', content) @@ to_tsquery('english', 'postgres');

-- 결과 예시:
-- "...using <b>Postgres</b> for full-text search provides..."
```

```
ts_headline 옵션
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  StartSel, StopSel: 강조 마커 (기본: <b>, </b>)            │
│  MaxWords: 최대 단어 수                                     │
│  MinWords: 최소 단어 수                                     │
│  ShortWord: 짧은 단어 기준                                  │
│  HighlightAll: 모든 매치 강조                               │
│  MaxFragments: 최대 프래그먼트 수                           │
│  FragmentDelimiter: 프래그먼트 구분자                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6. 저장된 tsvector 컬럼

```sql
-- tsvector 컬럼 추가 (성능 최적화)
ALTER TABLE articles ADD COLUMN search_vector tsvector;

-- tsvector 값 채우기
UPDATE articles SET search_vector =
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(content, '')), 'B');

-- 트리거로 자동 업데이트
CREATE FUNCTION update_search_vector() RETURNS trigger AS $$
BEGIN
    NEW.search_vector :=
        setweight(to_tsvector('english', coalesce(NEW.title, '')), 'A') ||
        setweight(to_tsvector('english', coalesce(NEW.content, '')), 'B');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER articles_search_update
    BEFORE INSERT OR UPDATE ON articles
    FOR EACH ROW EXECUTE FUNCTION update_search_vector();
```

### 7. GIN vs GiST 인덱스

```
인덱스 비교
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 GIN (Generalized Inverted Index) 】                     │
│                                                             │
│  구조: 각 렉셈 → 포함하는 문서 목록                         │
│  'database' → [doc1, doc5, doc12, ...]                      │
│  'postgres' → [doc2, doc5, doc7, ...]                       │
│                                                             │
│  특징:                                                       │
│  • 정확한 매칭에 최적                                       │
│  • 검색 빠름                                                │
│  • 빌드/업데이트 느림                                       │
│  • 크기 큼                                                  │
│                                                             │
│  【 GiST (Generalized Search Tree) 】                       │
│                                                             │
│  구조: 손실 압축 (signature)                                │
│                                                             │
│  특징:                                                       │
│  • 빌드/업데이트 빠름                                       │
│  • 크기 작음                                                │
│  • 검색 시 False Positive 가능 → 재확인 필요                │
│  • 근접 검색 지원                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- GIN 인덱스 (권장)
CREATE INDEX idx_articles_search_gin
ON articles USING GIN(search_vector);

-- GiST 인덱스
CREATE INDEX idx_articles_search_gist
ON articles USING GiST(search_vector);

-- 인덱스 사용 확인
EXPLAIN ANALYZE
SELECT * FROM articles
WHERE search_vector @@ to_tsquery('english', 'database');
```

## 비교표

### LIKE vs Full Text Search

| 구분 | LIKE | Full Text Search |
|------|------|------------------|
| 인덱스 | trigram 필요 | GIN/GiST 지원 |
| 어간 추출 | 불가 | 지원 |
| 불용어 | 매칭됨 | 제거됨 |
| 순위 | 불가 | ts_rank 지원 |
| 복합 조건 | 단순 AND/OR | 구문, 근접 검색 |

### GIN vs GiST 인덱스

| 구분 | GIN | GiST |
|------|-----|------|
| 검색 속도 | 빠름 | 상대적 느림 |
| 빌드 속도 | 느림 | 빠름 |
| 인덱스 크기 | 큼 | 작음 |
| 정확도 | 정확 | False Positive 가능 |
| 근접 검색 | 미지원 | 지원 |
| 권장 용도 | 일반적 (기본) | 자주 업데이트되는 경우 |

## 면접 예상 질문 및 모범 답안

### Q1. PostgreSQL의 Full Text Search와 LIKE 검색의 차이점은?

**모범 답안:**

**LIKE 검색의 한계:**
- 패턴 매칭 기반으로 Full Table Scan 발생
- 대소문자 구분 (ILIKE는 성능 저하)
- 'running'과 'run'을 다른 단어로 인식
- 'the', 'a' 같은 불용어도 검색됨

**Full Text Search의 장점:**
1. **언어 처리:**
   - 어간 추출: foxes → fox, running → run
   - 불용어 제거: the, a, is 등

2. **인덱스 지원:**
   - GIN 인덱스로 O(log N) 검색

3. **고급 검색:**
   - AND/OR/NOT 조합
   - 구문 검색 (인접 단어)
   - 관련도 순위

```sql
-- LIKE: 단순 패턴 매칭
WHERE content LIKE '%postgres%'

-- FTS: 언어 기반 검색
WHERE to_tsvector('english', content) @@
      to_tsquery('english', 'postgres')
```

### Q2. tsvector와 tsquery의 역할을 설명해주세요.

**모범 답안:**

**tsvector:**
문서를 검색 가능한 형태로 변환한 벡터입니다.
- 토큰화: 텍스트를 단어로 분리
- 정규화: 소문자 변환, 어간 추출
- 불용어 제거
- 위치 정보 저장

```sql
SELECT to_tsvector('english', 'The cats are running');
-- 결과: 'cat':2 'run':4
```

**tsquery:**
검색 조건을 표현하는 쿼리입니다.
- 논리 연산자: & (AND), | (OR), ! (NOT)
- 인접 연산자: <-> (바로 옆), <N> (N 단어 이내)

```sql
SELECT to_tsquery('english', 'cat & dog');
-- 'cat' & 'dog'
```

**@@ 연산자:**
tsvector가 tsquery와 매칭되는지 확인합니다.

### Q3. ts_rank 함수와 가중치 시스템을 설명해주세요.

**모범 답안:**

ts_rank는 검색 결과의 관련도를 계산합니다.

**가중치 레벨 (A, B, C, D):**
- A: 1.0 (가장 중요)
- B: 0.4
- C: 0.2
- D: 0.1 (기본)

**setweight 함수로 가중치 적용:**
```sql
SELECT ts_rank(
    setweight(to_tsvector('english', title), 'A') ||
    setweight(to_tsvector('english', body), 'D'),
    to_tsquery('english', 'postgres')
) AS rank
FROM articles
ORDER BY rank DESC;
```

이렇게 하면 제목 매칭이 본문보다 10배 높은 점수를 받습니다.

**정규화 옵션:**
- 긴 문서가 불리하지 않도록 문서 길이로 나눌 수 있음
- `ts_rank(vector, query, 1)` - 길이로 나눔

### Q4. GIN과 GiST 인덱스의 차이점과 선택 기준은?

**모범 답안:**

**GIN (Generalized Inverted Index):**
- 각 렉셈에 대해 포함하는 문서 목록 저장
- 검색이 빠르고 정확
- 빌드/업데이트가 느리고 크기가 큼

**GiST (Generalized Search Tree):**
- 손실 압축(signature) 사용
- 빌드/업데이트가 빠르고 크기가 작음
- 검색 시 False Positive 가능 → 재확인 필요

**선택 기준:**

| 상황 | 권장 인덱스 |
|------|------------|
| 읽기 위주 | GIN |
| 쓰기 많음 | GiST |
| 정확도 중요 | GIN |
| 디스크 제약 | GiST |
| 일반적인 경우 | GIN (기본 권장) |

### Q5. 저장된 tsvector 컬럼을 사용하는 이유와 방법은?

**모범 답안:**

**이유:**
매 쿼리마다 `to_tsvector()`를 호출하면 CPU 오버헤드 발생. 사전에 계산하여 저장하면 검색 성능 향상.

**구현 방법:**

1. **컬럼 추가:**
```sql
ALTER TABLE articles ADD COLUMN search_vector tsvector;
```

2. **데이터 채우기:**
```sql
UPDATE articles SET search_vector =
    setweight(to_tsvector('english', title), 'A') ||
    setweight(to_tsvector('english', body), 'B');
```

3. **트리거로 자동 갱신:**
```sql
CREATE TRIGGER tsvector_update
    BEFORE INSERT OR UPDATE ON articles
    FOR EACH ROW EXECUTE FUNCTION tsvector_update_trigger(
        search_vector, 'english', title, body
    );
```

4. **인덱스 생성:**
```sql
CREATE INDEX idx_search ON articles USING GIN(search_vector);
```

**주의:** 저장 공간은 증가하지만 검색 성능은 크게 향상됨.

### Q6. websearch_to_tsquery 함수의 장점은?

**모범 답안:**

websearch_to_tsquery는 웹 검색 엔진 스타일의 문법을 지원합니다.

**지원 문법:**
```sql
websearch_to_tsquery('english', 'fat cats -dogs "new york"')
-- 결과: 'fat' & 'cat' & !'dog' & 'new' <-> 'york'
```

- 공백: AND
- `-단어`: NOT
- `"구문"`: 정확한 순서
- `OR`: 명시적 OR

**장점:**
1. 사용자 친화적 (구글 검색과 유사)
2. 특수 문자 자동 처리
3. 구문 검색 지원
4. SQL 인젝션 안전

**vs 다른 함수:**
- `to_tsquery`: 직접 연산자 작성 필요
- `plainto_tsquery`: 모든 단어 AND
- `phraseto_tsquery`: 구문만
- `websearch_to_tsquery`: 복합 문법 지원
