### 3가지 "사용하지 않아야 하는" 상황 상세

```
상황 1: 다른 DB가 명확히 더 적합
┌─────────────────────────────────────────────────────────────────┐
│  핵심 원칙:                                                      │
│  "다른 DB가 use case를 쉽게 지원하는데,                          │
│   Postgres를 억지로 끼워 맞추려고 팀의 삶을                       │
│   복잡하게 만들 필요는 없다."                                    │
├─────────────────────────────────────────────────────────────────┤
│  판단 기준:                                                      │
│  ├── Postgres로 구현 가능한가? → 불가능하면 다른 DB              │
│  ├── 구현 복잡도 vs 가치 비교                                    │
│  └── 복잡도 > 가치 → 다른 DB 선택                               │
└─────────────────────────────────────────────────────────────────┘

상황 2: 기존 시스템이 잘 작동
┌─────────────────────────────────────────────────────────────────┐
│  유지 조건:                                                      │
│  ├── ✅ 현재 DB가 역할을 잘 수행 중                              │
│  ├── ✅ 팀에서 불만이나 우려 없음                                │
│  ├── ✅ 성능/확장성 문제 없음                                    │
│  └── ✅ 유지보수 부담 감당 가능                                  │
├─────────────────────────────────────────────────────────────────┤
│  "유료 제품이라도 회사가 비용에 문제 없다면 그냥 지불하게 둬라.  │
│   이미 작동하는 것을 깨뜨리지 말라."                             │
└─────────────────────────────────────────────────────────────────┘

상황 3: 팀이 다른 DB에 훨씬 능숙
┌─────────────────────────────────────────────────────────────────┐
│  팀 전문성 활용의 이점:                                          │
│  ├── 더 빠른 개발 및 배포                                        │
│  ├── 더 적은 실수                                                │
│  ├── 프로덕션 문제 대응력 향상                                   │
│  └── (보너스) 새로운 기술 학습 기회                              │
├─────────────────────────────────────────────────────────────────┤
│  Postgres와 다른 DB 모두 적합할 때:                              │
│  → 팀 전문성 있는 DB 선택이 더 실용적                            │
└─────────────────────────────────────────────────────────────────┘
```

### "Just Use Postgres!"의 진정한 의미

```
오해:
┌─────────────────────────────────────────┐
│  "Just Use Postgres!"                   │
│  = "항상 Postgres만 사용하라"           │  ← X
└─────────────────────────────────────────┘

진정한 의미:
┌─────────────────────────────────────────────────────────────────┐
│  "Just Use Postgres!" =                                          │
│  ├── Postgres의 다양한 가능성을 알아라                           │
│  ├── 한 DB로 여러 use case를 커버할 수 있음을 알아라             │
│  ├── 불필요한 복잡성(polyglot persistence)을 피할 수 있음을 알아라│
│  └── 단, 맹목적 도입은 경계하라                                  │
└─────────────────────────────────────────────────────────────────┘

핵심 메시지:
├── 기술 자체보다 문제 해결이 목적
├── Postgres가 만능은 아님을 인정
└── 팀과 기존 시스템을 존중
```

### 기술 선택의 균형 원칙

```
균형 잡힌 의사결정 원칙:
┌──────────────┬────────────────────────────────────────────────┐
│ 실용주의     │ 기술 자체보다 문제 해결이 목적                 │
├──────────────┼────────────────────────────────────────────────┤
│ 겸손함       │ Postgres가 만능은 아님을 인정                  │
├──────────────┼────────────────────────────────────────────────┤
│ 팀 중심      │ 기술보다 사람의 역량이 중요                    │
├──────────────┼────────────────────────────────────────────────┤
│ 비용 의식    │ 전환 비용 vs 유지 비용 비교                    │
├──────────────┼────────────────────────────────────────────────┤
│ 열린 마음    │ 다른 기술에서도 배울 점 있음                   │
└──────────────┴────────────────────────────────────────────────┘
```

## 3. 면접 예상 질문 및 모범 답안

### Q1. PostgreSQL을 선택하지 않아야 하는 상황은 언제인가요?

**모범 답안:**

세 가지 주요 상황을 고려합니다.

**1. 다른 DB가 명확히 더 적합할 때:**
```
예시:
├── 순수 그래프 쿼리 중심 → Neo4j
├── 페타바이트급 분석 → ClickHouse
└── 실시간 스트리밍 → Kafka Streams
```

**2. 기존 시스템이 잘 작동할 때:**
"Postgres를 위한 Postgres" 도입은 불필요한 리스크입니다. 현재 DB가 성능, 확장성, 유지보수 측면에서 문제 없다면 유지하는 것이 합리적입니다.

**3. 팀이 다른 DB에 훨씬 능숙할 때:**
팀 전문성을 활용하면 더 빠른 개발, 적은 실수, 프로덕션 대응력 향상을 기대할 수 있습니다.

핵심은 **기술적 우월성보다 실무적 판단**이 중요하다는 것입니다.

---

### Q2. "Just Use Postgres"라는 관점에서, Polyglot Persistence와의 균형을 어떻게 보시나요?

**모범 답안:**

**Polyglot Persistence의 장단점:**
```
장점:
├── 각 use case에 최적화된 DB 사용
└── 특정 작업에서 성능 극대화

단점:
├── 운영 복잡성 증가
├── 데이터 동기화 문제
├── 팀의 학습 부담
└── 트랜잭션 일관성 보장 어려움
```

**균형점:**
Postgres의 확장성(JSONB, pgvector, TimescaleDB, PostGIS 등)을 활용하면 많은 경우 **단일 DB로 다양한 use case를 커버**할 수 있습니다. 불필요한 Polyglot Persistence를 피하면서도, 정말 특화 DB가 필요한 경우(순수 그래프 쿼리, 페타바이트 분석 등)에는 유연하게 선택하는 것이 바람직합니다.

---

### Q3. 기존에 사용하던 데이터베이스에서 PostgreSQL로 마이그레이션을 검토할 때 고려해야 할 사항은?

**모범 답안:**

**마이그레이션 전 체크리스트:**
```
1. 현재 시스템 평가
   ├── 현재 DB에 실제 문제가 있는가?
   ├── 성능/확장성 한계에 도달했는가?
   └── 유지보수 비용이 과도한가?

2. 마이그레이션 비용 분석
   ├── 스키마 변환 복잡도
   ├── 애플리케이션 코드 수정 범위
   ├── 다운타임 허용 범위
   └── 테스트 및 검증 기간

3. 팀 역량 평가
   ├── PostgreSQL 운영 경험
   ├── 학습 곡선 감당 가능 여부
   └── 장애 대응 능력

4. ROI 분석
   ├── 마이그레이션 비용 vs 장기 운영 비용
   └── 비즈니스 가치 vs 기술적 이점
```

"깨지지 않은 것을 고치지 마라"는 원칙을 기억하며, **실질적인 문제 해결이 목적**일 때만 마이그레이션을 진행해야 합니다.

### Q4. 팀에 PostgreSQL 경험이 없는 상황에서 새 프로젝트에 PostgreSQL을 도입하려 합니다. 어떻게 접근하시겠습니까?

**모범 답안:**

**단계적 접근:**
```
1단계: 적합성 평가
├── Use case가 PostgreSQL에 적합한가?
├── 팀이 능숙한 다른 DB가 적합하지 않은가?
└── PostgreSQL의 장기적 이점이 학습 비용을 정당화하는가?

2단계: 점진적 도입 (적합하다면)
├── 비핵심 서비스에서 먼저 시작
├── 충분한 학습 시간 확보
├── 외부 전문가 컨설팅 검토
└── 프로덕션 전 충분한 부하 테스트

3단계: 역량 구축
├── 팀 내 PostgreSQL 챔피언 육성
├── 모니터링/장애 대응 프로세스 수립
└── 점진적으로 핵심 서비스 확대
```

팀 전문성이 부족한 상황에서 급하게 도입하면 **프로덕션 장애 시 대응력이 떨어질 수 있으므로**, 충분한 준비 기간을 확보하는 것이 중요합니다.

---

### Q5. PostgreSQL의 한계점이나 약점을 어떻게 보시나요?

**모범 답안:**

PostgreSQL의 **현재 한계보다는 실무적 판단 기준**이 더 중요합니다.

**이유:**
```
PostgreSQL 생태계의 빠른 진화:
├── 오늘의 미지원 기능 → 내일의 기본 기능
├── 예: pgvector, TimescaleDB 등 확장으로 해결
└── 커뮤니티의 활발한 발전
```

**실무적 관점:**
기술적 한계 목록을 암기하는 것보다, **use case 적합성, 팀 역량, 기존 시스템 상태**를 종합적으로 고려하여 판단하는 것이 더 가치 있습니다.

물론 현재 시점에서 순수 그래프 쿼리, 페타바이트급 실시간 분석 등에서는 특화 솔루션이 우위에 있습니다. 하지만 이것도 PostgreSQL 생태계의 발전에 따라 변할 수 있으므로, **최신 동향을 지속적으로 파악**하는 것이 중요합니다.

### Q6. "기술 선택에서 실용주의"란 어떤 의미인가요? 실제 경험을 기반으로 설명해주세요.

**모범 답안:**

**실용주의의 핵심:**
```
기술 선택 = 문제 해결 수단
├── 목적: 비즈니스 가치 창출
├── 수단: 기술 (PostgreSQL, MongoDB, 등)
└── 기술 자체가 목적이 되면 안 됨
```

**실용주의적 의사결정 예시:**
```
상황: 문서 DB를 사용 중인 팀
├── 애플리케이션에 잘 맞음
├── 팀 전체가 해당 DB에 익숙
└── 성능/확장성 문제 없음

잘못된 선택: "PostgreSQL이 더 좋으니 교체하자"
올바른 선택: 기존 DB 최적화에 집중 → 인프라 비용 5배 절감
```

**실용주의 체크리스트:**
- 이 결정이 실제 문제를 해결하는가?
- 전환 비용 대비 가치가 있는가?
- 팀이 효율적으로 운영할 수 있는가?
- "멋져 보여서"가 아닌 "필요해서" 선택하는가?

## 4. 실무 체크리스트

### Postgres 도입 의사결정 체크리스트

**Use Case 적합성:**
- [ ] PostgreSQL이 이 use case를 자연스럽게 지원하는가?
- [ ] 다른 DB가 더 적합하지는 않은가?
- [ ] Postgres 도입으로 복잡도가 증가하지 않는가?

**기존 시스템 평가:**
- [ ] 현재 DB에 문제가 있는가?
- [ ] 마이그레이션 비용 대비 가치가 있는가?
- [ ] "깨지지 않은 것을 고치려는" 것은 아닌가?

**팀 역량 평가:**
- [ ] 팀이 PostgreSQL 경험이 있는가?
- [ ] 학습 곡선을 감당할 여유가 있는가?
- [ ] 다른 DB 전문성을 활용하는 게 더 효율적인가?

**장기적 관점:**
- [ ] PostgreSQL 생태계의 이점이 전환 비용을 정당화하는가?
- [ ] 향후 use case 확장 시 PostgreSQL가 유리한가?

### 마이그레이션 전 점검 사항

- [ ] 현재 시스템의 실제 문제점 문서화
- [ ] 예상 마이그레이션 비용 산정
- [ ] 다운타임 허용 범위 확인
- [ ] 롤백 계획 수립
- [ ] 팀 역량 갭 분석
- [ ] 프로덕션 전 충분한 테스트 계획

### 균형 잡힌 기술 선택 원칙

- [ ] 실용주의: 기술보다 문제 해결이 목적인가?
- [ ] 겸손함: Postgres가 만능이 아님을 인정하는가?
- [ ] 팀 중심: 기술보다 팀 역량을 우선하는가?
- [ ] 비용 의식: 전환 비용 vs 유지 비용을 비교했는가?
- [ ] 열린 마음: 다른 기술의 장점도 고려했는가?
