# Chapter 10: Postgres for Geospatial Data - 면접정리

## 핵심 개념 상세 설명

### 1. 지리공간 데이터 유형

```
지리공간 데이터 기본 유형
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 Point (점) 】                                           │
│  • 단일 위치: 레스토랑, 버스 정류장, 랜드마크               │
│  • 표현: (경도, 위도) 또는 (x, y)                          │
│                                                             │
│  【 LineString (선) 】                                       │
│  • 연결된 점들: 도로, 강, 철도, 등산로                      │
│  • 표현: [(x1,y1), (x2,y2), ..., (xn,yn)]                  │
│                                                             │
│  【 Polygon (다각형) 】                                      │
│  • 닫힌 영역: 도시 경계, 공원, 건물, 호수                   │
│  • 표현: 닫힌 LineString (첫 점 = 마지막 점)               │
│                                                             │
│  【 Multi* 유형 】                                           │
│  • MultiPoint, MultiLineString, MultiPolygon               │
│  • 여러 개의 기본 유형을 하나로 그룹화                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. geometry vs geography

```
PostGIS 데이터 타입 비교
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 geometry 】                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 평면 좌표계 (Cartesian)                          │   │
│  │  • 유클리드 수학 사용                               │   │
│  │  • SRID 명시 필요                                   │   │
│  │  • 계산 빠름                                        │   │
│  │  • 대거리에서 부정확 (지구 곡률 무시)               │   │
│  │  • 단위: SRID에 따름 (미터 또는 도)                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【 geography 】                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 구형 좌표계 (Spheroidal)                         │   │
│  │  • 지구 곡률 반영                                   │   │
│  │  • 기본 SRID 4326 (WGS 84)                         │   │
│  │  • 계산 느림                                        │   │
│  │  • 정확한 거리 계산                                 │   │
│  │  • 단위: 항상 미터                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  선택 기준:                                                  │
│  • 전 세계 범위, 정확도 중요 → geography                   │
│  • 지역 범위, 성능 중요 → geometry (적절한 SRID)           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3. SRID (Spatial Reference System Identifier)

```
주요 SRID
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 SRID 4326 (WGS 84) 】                                    │
│  • GPS 표준 좌표계                                          │
│  • 단위: 도(°)                                              │
│  • 범위: 경도 -180~180, 위도 -90~90                        │
│  • geography 타입 기본값                                    │
│  • 예: (-80.225, 25.782) = 마이애미                         │
│                                                             │
│  【 SRID 3857 (Web Mercator) 】                              │
│  • 웹 지도 표준 (Google Maps, OSM)                         │
│  • 단위: 미터                                               │
│  • 평면 투영 (극지방 왜곡)                                  │
│  • geometry 타입에서 주로 사용                              │
│  • 예: (-8930649, 2972156) = 마이애미                       │
│                                                             │
│  좌표 변환:                                                  │
│  ST_Transform(geom, target_srid)                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 테이블 생성 예시
CREATE TABLE locations (
    id SERIAL PRIMARY KEY,
    name TEXT,
    -- geometry: Web Mercator (미터 단위)
    geom geometry(Point, 3857),
    -- geography: WGS 84 (도 단위)
    geo geography(Point, 4326)
);

-- 데이터 삽입
INSERT INTO locations (name, geom, geo) VALUES (
    'Miami',
    ST_SetSRID(ST_MakePoint(-8930649, 2972156), 3857),
    ST_MakePoint(-80.225, 25.782)::geography
);
```

### 4. PostGIS 주요 함수

```
좌표 및 변환 함수
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ST_MakePoint(x, y)        Point 생성                       │
│  ST_SetSRID(geom, srid)    SRID 설정                        │
│  ST_Transform(geom, srid)  좌표계 변환                      │
│  ST_X(point)               X 좌표(경도) 추출                │
│  ST_Y(point)               Y 좌표(위도) 추출                │
│  ST_AsText(geom)           WKT 형식으로 변환                │
│  ST_GeomFromText(wkt)      WKT에서 geometry 생성            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 좌표 변환 예시
SELECT
    name,
    ST_AsText(ST_Transform(geom, 4326)) AS wgs84_coords
FROM locations;

-- 결과: POINT(-80.225 25.782)
```

### 5. 거리 및 공간 관계 함수

```
거리 계산 함수
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 ST_Distance(g1, g2) 】                                   │
│  • 두 지오메트리 간 최단 거리                               │
│  • 인덱스 사용 불가 ❌                                      │
│  • 모든 행에 대해 거리 계산                                 │
│                                                             │
│  【 ST_DWithin(g1, g2, distance) 】                          │
│  • 지정 거리 이내 여부 (boolean)                            │
│  • 인덱스 사용 가능 ✅                                      │
│  • 필터링 후 상세 계산                                      │
│                                                             │
│  권장: ST_DWithin으로 필터링 후 ST_Distance로 정렬          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 500m 이내 레스토랑 찾기 (최적화된 쿼리)
SELECT
    name,
    ST_Distance(hotel.geom, restaurant.geom) AS distance_m
FROM restaurants restaurant, hotels hotel
WHERE hotel.name = 'Grand Hotel'
  AND ST_DWithin(hotel.geom, restaurant.geom, 500)  -- 인덱스 활용
ORDER BY distance_m
LIMIT 10;
```

```
공간 관계 함수
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ST_Within(A, B)      A가 B 안에 완전히 포함 ✅ 인덱스      │
│  ST_Contains(A, B)    A가 B를 완전히 포함 ✅ 인덱스         │
│  ST_Intersects(A, B)  A와 B가 교차 (공유 영역) ✅ 인덱스    │
│  ST_Overlaps(A, B)    A와 B가 부분 겹침                     │
│  ST_Touches(A, B)     A와 B가 경계만 접촉                   │
│  ST_Equals(A, B)      A와 B가 동일                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 디즈니월드 내 테마파크 찾기
SELECT p.name
FROM parks p
JOIN boundaries b ON ST_Within(p.geom, b.geom)
WHERE b.name = 'Walt Disney World';
```

### 7. GiST 인덱스

```
GiST (Generalized Search Tree) for Spatial Data
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  구조: Bounding Box 기반 R-tree                             │
│                                                             │
│           [Root: 전체 영역]                                 │
│          /        |        \                                │
│    [R1: 북서]  [R2: 북동]  [R3: 남부]                       │
│      /   \       / \         / \                            │
│  [점들] [점들] [점들] [점들] [점들] [점들]                  │
│                                                             │
│  검색 과정:                                                  │
│  1. 쿼리 영역과 바운딩 박스 비교                            │
│  2. 겹치는 노드만 탐색                                      │
│  3. 리프 노드에서 정확한 검사                               │
│                                                             │
│  인덱스 인식 함수:                                          │
│  ✅ ST_DWithin, ST_Within, ST_Intersects, ST_Contains       │
│  ❌ ST_Distance (항상 계산 필요)                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- GiST 인덱스 생성
CREATE INDEX idx_locations_geom
ON locations USING GiST(geom);

-- 인덱스 사용 확인
EXPLAIN ANALYZE
SELECT * FROM locations
WHERE ST_DWithin(geom, ST_MakePoint(-80.2, 25.8)::geometry, 1000);

-- 결과: Bitmap Index Scan on idx_locations_geom
```

## 비교표

### geometry vs geography

| 특성 | geometry | geography |
|------|----------|-----------|
| 좌표계 | 평면 (Cartesian) | 구형 (Spheroidal) |
| 기본 SRID | 없음 (명시 필요) | 4326 (WGS 84) |
| 거리 단위 | SRID에 따름 | 항상 미터 |
| 정확도 | 대거리 부정확 | 정확 |
| 계산 속도 | 빠름 | 느림 |
| 함수 지원 | 모든 함수 | 일부 함수 |

### 인덱스 인식 함수

| 함수 | 인덱스 사용 | 용도 |
|------|------------|------|
| ST_DWithin | ✅ | 거리 이내 필터링 |
| ST_Within | ✅ | 포함 관계 |
| ST_Intersects | ✅ | 교차 여부 |
| ST_Contains | ✅ | 포함 관계 |
| ST_Distance | ❌ | 정확한 거리 (정렬용) |

## 면접 예상 질문 및 모범 답안

### Q1. geometry와 geography 타입의 차이점과 선택 기준은?

**모범 답안:**

**geometry:**
- 평면 좌표계 (유클리드 기하학)
- SRID를 명시해야 함
- 계산이 빠름
- 대거리에서 오차 발생 (지구 곡률 무시)

**geography:**
- 구형 좌표계 (지구 곡률 반영)
- 기본 WGS 84 (SRID 4326)
- 정확한 거리 계산
- 계산이 느림

**선택 기준:**

| 상황 | 권장 타입 |
|------|----------|
| 전 세계 범위 데이터 | geography |
| 정확한 거리 필수 | geography |
| 지역 범위 (도시/국가) | geometry + 적절한 SRID |
| 복잡한 공간 연산 | geometry |
| 성능 중요 | geometry |

```sql
-- 지역 데이터: geometry with Web Mercator
CREATE TABLE local_places (
    geom geometry(Point, 3857)
);

-- 글로벌 데이터: geography
CREATE TABLE global_places (
    geo geography(Point, 4326)
);
```

### Q2. SRID 4326과 3857의 차이점을 설명해주세요.

**모범 답안:**

**SRID 4326 (WGS 84):**
- GPS 표준 좌표계
- 단위: 도(°)
- 위도/경도로 표현
- geography 타입 기본값
- 예: (-122.4194, 37.7749) = 샌프란시스코

**SRID 3857 (Web Mercator):**
- 웹 지도 표준 (Google Maps, OpenStreetMap)
- 단위: 미터
- 평면 좌표로 표현
- 극지방에서 왜곡
- 예: (-13627665, 4548674) = 샌프란시스코

**사용 시점:**
- 4326: 데이터 저장, GPS 데이터, 정확한 거리
- 3857: 웹 지도 표시, 타일 계산, 빠른 연산

**변환:**
```sql
SELECT ST_Transform(geom, 4326) FROM places;  -- 3857 → 4326
```

### Q3. ST_Distance와 ST_DWithin의 차이점과 성능 고려사항은?

**모범 답안:**

**ST_Distance:**
- 두 지오메트리 간 정확한 거리 반환
- **인덱스 사용 불가**
- 모든 행에 대해 거리 계산
- 정렬에 사용

**ST_DWithin:**
- 지정 거리 이내인지 boolean 반환
- **인덱스 사용 가능** (GiST)
- 바운딩 박스로 빠르게 필터링
- WHERE 절에서 사용

**최적화된 쿼리 패턴:**
```sql
-- 좋은 예: ST_DWithin으로 필터링 후 ST_Distance로 정렬
SELECT name, ST_Distance(a.geom, b.geom) AS dist
FROM places a, my_location b
WHERE ST_DWithin(a.geom, b.geom, 1000)  -- 인덱스 활용
ORDER BY dist;

-- 나쁜 예: ST_Distance만 사용
SELECT name, ST_Distance(a.geom, b.geom) AS dist
FROM places a, my_location b
WHERE ST_Distance(a.geom, b.geom) < 1000  -- Full Scan!
ORDER BY dist;
```

### Q4. PostGIS에서 GiST 인덱스는 어떻게 동작하나요?

**모범 답안:**

GiST(Generalized Search Tree)는 공간 데이터용 인덱스로, R-tree 구조를 사용합니다.

**동작 원리:**
1. 각 지오메트리의 바운딩 박스(MBR) 계산
2. 바운딩 박스를 계층적 트리로 구성
3. 검색 시 쿼리 영역과 바운딩 박스 비교
4. 겹치는 노드만 탐색 (프루닝)
5. 리프 노드에서 정확한 공간 검사

**인덱스 인식 함수:**
- ✅ ST_DWithin, ST_Within, ST_Intersects, ST_Contains
- ❌ ST_Distance (바운딩 박스로 판단 불가)

**생성:**
```sql
CREATE INDEX idx_geom ON places USING GiST(geom);
```

### Q5. 근처 장소 검색 쿼리를 최적화하는 방법은?

**모범 답안:**

**1. GiST 인덱스 생성:**
```sql
CREATE INDEX idx_places_geom ON places USING GiST(geom);
```

**2. ST_DWithin으로 필터링:**
```sql
SELECT name, ST_Distance(geom, query_point) AS distance
FROM places
WHERE ST_DWithin(geom, query_point, 1000)  -- 1km 이내
ORDER BY distance
LIMIT 10;
```

**3. 자주 검색되는 컬럼에 B-tree 추가:**
```sql
CREATE INDEX idx_places_name ON places(name)
WHERE name IS NOT NULL;
```

**4. 부분 인덱스 활용:**
```sql
-- 활성 장소만 인덱싱
CREATE INDEX idx_active_places ON places USING GiST(geom)
WHERE status = 'active';
```

**5. EXPLAIN으로 확인:**
```sql
EXPLAIN ANALYZE SELECT ...
-- Bitmap Index Scan on idx_places_geom 확인
```

### Q6. OpenStreetMap 데이터를 PostgreSQL에서 어떻게 활용하나요?

**모범 답안:**

**1. 데이터 로드 (osm2pgsql):**
```bash
osm2pgsql -d postgres --create \
    florida-latest.osm.pbf
```

**2. 생성되는 테이블:**
- `planet_osm_point`: 점 (레스토랑, 정류장)
- `planet_osm_line`: 선 (도로, 강)
- `planet_osm_polygon`: 면 (건물, 공원)
- `planet_osm_roads`: 주요 도로 (렌더링용)

**3. 쿼리 예시:**
```sql
-- 특정 도시의 레스토랑
SELECT name, ST_AsText(ST_Transform(way, 4326))
FROM planet_osm_point
WHERE amenity = 'restaurant'
  AND ST_DWithin(way, city_center, 5000);
```

**4. 데이터 특성:**
- way 컬럼: geometry(3857)
- 태그 기반 속성 (amenity, tourism 등)
