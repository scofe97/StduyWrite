# Chapter 11: Postgres as a Message Queue - 면접정리

## 핵심 개념 상세 설명

### 1. Postgres를 메시지 큐로 사용하는 기준

```
3가지 핵심 기준
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 기준 1: 트랜잭션 원자성 필요 】                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  BEGIN;                                              │   │
│  │  INSERT INTO orders (...) VALUES (...);  -- 주문    │   │
│  │  SELECT mq.enqueue('order_created');     -- 메시지  │   │
│  │  COMMIT;                                             │   │
│  │  -- 둘 다 성공하거나 둘 다 실패                      │   │
│  └─────────────────────────────────────────────────────┘   │
│  → 비즈니스 로직과 메시지 발행을 원자적으로 처리           │
│                                                             │
│  【 기준 2: 메시지 볼륨이 Postgres로 감당 가능 】           │
│  • 단일 인스턴스로 처리 가능한 수준인가?                   │
│  • 읽기 부하 → 레플리카 추가                               │
│  • 쓰기 부하 → 스케일업 또는 분산 Postgres                 │
│                                                             │
│  【 기준 3: 이미 Postgres 사용 중 】                        │
│  • 추가 인프라 없이 아키텍처 단순화                        │
│  • 운영/모니터링 통합                                      │
│  • 학습 곡선 최소화                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 커스텀 큐 테이블 설계

```sql
-- 스키마 및 상태 타입
CREATE SCHEMA mq;

CREATE TYPE mq.status AS ENUM ('new', 'processing', 'completed');

-- 큐 테이블
CREATE TABLE mq.queue (
    id BIGSERIAL PRIMARY KEY,
    message JSON NOT NULL,           -- JSONB 아닌 JSON (원본 보존)
    created_at TIMESTAMPTZ DEFAULT NOW(),
    status mq.status NOT NULL DEFAULT 'new'
);
```

```
테이블 설계 포인트
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 id: BIGSERIAL 】                                        │
│  • 2^63개 고유 ID                                          │
│  • FIFO 순서 보장 (created_at과 함께)                      │
│                                                             │
│  【 message: JSON (not JSONB) 】                            │
│  • 메시지 원본 그대로 저장/반환                            │
│  • 쿼리/인덱싱 불필요 (큐 특성상)                          │
│  • JSONB는 전처리 오버헤드만 발생                          │
│                                                             │
│  【 status: ENUM 】                                         │
│  • new → processing → completed                            │
│  • 명시적 상태 관리                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3. FOR UPDATE SKIP LOCKED

```
동시성 처리의 핵심
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Consumer 1                    Consumer 2                   │
│      │                             │                        │
│      ├─ dequeue(1)                 │                        │
│      │  → msg_id=1 잠금            │                        │
│      │                             ├─ dequeue(1)            │
│      │                             │  → msg_id=1 SKIP       │
│      │                             │  → msg_id=2 반환       │
│      │                             │                        │
│      ├─ 처리 중...                 ├─ 처리 중...            │
│      │                             │                        │
│      ├─ mark_completed(1)          ├─ mark_completed(2)     │
│      │                             │                        │
│                                                             │
│  FOR UPDATE: 선택된 행에 행 수준 잠금                       │
│  SKIP LOCKED: 이미 잠긴 행은 건너뜀 (블로킹 없음!)          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- dequeue 함수
CREATE OR REPLACE FUNCTION mq.dequeue(messages_cnt INT)
RETURNS TABLE (msg_id BIGINT, message JSON, enqueued_at TIMESTAMPTZ) AS $
BEGIN
  RETURN QUERY
  WITH new_messages AS (
    SELECT id FROM mq.queue
    WHERE status = 'new'
    ORDER BY created_at
    FOR UPDATE SKIP LOCKED  -- 핵심!
    LIMIT messages_cnt
  )
  UPDATE mq.queue q
  SET status = 'processing'
  FROM new_messages
  WHERE q.id = new_messages.id
  RETURNING q.id, q.message, q.created_at;
END;
$ LANGUAGE plpgsql;
```

### 4. enqueue / mark_completed 함수

```sql
-- enqueue: 메시지 추가
CREATE OR REPLACE FUNCTION mq.enqueue(new_message JSON)
RETURNS VOID AS $
BEGIN
  INSERT INTO mq.queue (message)
  VALUES (new_message);

  -- 선택: 실시간 알림
  PERFORM pg_notify('queue_new_message', 'new');
END;
$ LANGUAGE plpgsql;

-- mark_completed: 처리 완료
CREATE OR REPLACE FUNCTION mq.mark_completed(
  message_ids BIGINT[],
  to_delete BOOLEAN DEFAULT FALSE
)
RETURNS VOID AS $
BEGIN
  IF to_delete THEN
    DELETE FROM mq.queue WHERE id = ANY(message_ids);
  ELSE
    UPDATE mq.queue SET status = 'completed'
    WHERE id = ANY(message_ids);
  END IF;
END;
$ LANGUAGE plpgsql;
```

### 5. LISTEN / NOTIFY

```
실시간 알림 패턴
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Producer                  Postgres                Consumer │
│      │                        │                        │    │
│      │                        │  LISTEN queue_channel  │    │
│      │                        │←───────────────────────│    │
│      │                        │  (구독 등록)           │    │
│      │                        │                        │    │
│      │  INSERT + pg_notify    │                        │    │
│      │───────────────────────→│                        │    │
│      │                        │  Async Notification    │    │
│      │                        │───────────────────────→│    │
│      │                        │                        │    │
│      │                        │                  dequeue    │
│      │                        │←───────────────────────│    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- Consumer 세션
LISTEN queue_new_message;

-- Producer 세션 (enqueue 함수에 포함)
PERFORM pg_notify('queue_new_message', 'new_message_payload');
```


### 6. 큐 최적화 전략

```
인덱싱 전략
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 저볼륨 】                                                │
│  PK 인덱스만으로 충분                                       │
│                                                             │
│  【 중볼륨 】                                                │
│  CREATE INDEX ON mq.queue (created_at);                     │
│                                                             │
│  【 고볼륨 】 부분 인덱스                                    │
│  CREATE INDEX ON mq.queue (created_at, status)              │
│  WHERE status = 'new';                                      │
│  → new 상태만 인덱싱 (처리 완료 제외)                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 파티셔닝 (고볼륨)
CREATE TABLE mq.queue (
    id BIGSERIAL,
    message JSON NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    status mq.status NOT NULL DEFAULT 'new',
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

-- 일별 파티션
CREATE TABLE mq.queue_2025_01_01 PARTITION OF mq.queue
FOR VALUES FROM ('2025-01-01') TO ('2025-01-02');
```

### 7. pgmq 확장

```sql
-- pgmq 설치
CREATE EXTENSION pgmq;

-- 큐 생성
SELECT pgmq.create('orders_queue');

-- 메시지 발송
SELECT pgmq.send(
  queue_name => 'orders_queue',
  msg => '{"order_id": 123, "action": "process"}'
);

-- 메시지 조회 (가시성 타임아웃)
SELECT * FROM pgmq.read(
  queue_name => 'orders_queue',
  vt => 120,  -- 2분 타임아웃
  qty => 5    -- 5개 조회
);

-- 처리 완료 (아카이브)
SELECT pgmq.archive('orders_queue', 1);

-- 즉시 삭제
SELECT pgmq.delete('orders_queue', 1);
```

```
pgmq 가시성 타임아웃 (Visibility Timeout)
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  1. Consumer가 read(vt=120)로 메시지 조회                   │
│  2. 메시지는 120초간 다른 Consumer에게 비가시               │
│  3. 처리 성공 → archive() 호출                              │
│  4. 처리 실패 (타임아웃) → 자동으로 다시 가시               │
│                                                             │
│  장점:                                                       │
│  • 장애 복구 자동화                                         │
│  • At-least-once 보장                                       │
│  • 수동 상태 관리 불필요                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 면접 예상 질문 및 모범 답안

### Q1. Postgres를 메시지 큐로 사용하는 것이 적합한 경우는?

**모범 답안:**

세 가지 기준으로 판단합니다:

**1. 트랜잭션 원자성이 필요한 경우:**
```sql
BEGIN;
INSERT INTO orders (...);       -- 비즈니스 로직
SELECT mq.enqueue(...);         -- 메시지 발행
COMMIT;                         -- 둘 다 성공 또는 실패
```
외부 MQ는 이런 원자성을 보장하기 어렵습니다.

**2. 메시지 볼륨이 적당한 경우:**
- 단일 Postgres로 처리 가능한 수준
- 초당 수천~수만 메시지 수준

**3. 이미 Postgres를 사용 중인 경우:**
- 추가 인프라 불필요
- 운영 복잡도 감소
- 통합 모니터링

**부적합한 경우:**
- 수백만 TPS 필요
- 복잡한 라우팅/토픽 필요
- 글로벌 분산 필요

### Q2. FOR UPDATE SKIP LOCKED의 역할과 동작 방식을 설명해주세요.

**모범 답안:**

`FOR UPDATE SKIP LOCKED`는 동시에 여러 Consumer가 큐를 처리할 때 핵심 메커니즘입니다.

**동작 방식:**
1. `FOR UPDATE`: 선택된 행에 행 수준 배타적 잠금
2. `SKIP LOCKED`: 이미 잠긴 행은 건너뜀

**예시:**
```sql
-- Consumer 1
SELECT * FROM queue WHERE status='new'
FOR UPDATE SKIP LOCKED LIMIT 1;
-- msg_id=1 잠금, 반환

-- Consumer 2 (동시)
SELECT * FROM queue WHERE status='new'
FOR UPDATE SKIP LOCKED LIMIT 1;
-- msg_id=1 스킵, msg_id=2 반환
```

**장점:**
- 블로킹 없음 (대기 없이 다음 메시지)
- 중복 처리 방지
- 병렬 처리 가능

**vs FOR UPDATE만:**
FOR UPDATE만 쓰면 잠긴 행을 기다리며 블로킹됨.

### Q3. LISTEN/NOTIFY의 장단점과 제한사항은?

**모범 답안:**

**장점:**
- 폴링 없이 실시간 알림
- 네트워크 오버헤드 감소
- 구현 단순

**제한사항:**

1. **백로그 없음:**
   - 구독 전 알림은 받을 수 없음
   - 해결: 연결 후 기존 메시지 조회

2. **Primary 전용:**
   - 레플리카에서 LISTEN 불가
   - 해결: 알림용 Primary 연결 유지

3. **페이로드 제한:**
   - 8000 bytes
   - 해결: 알림은 신호만, 데이터는 쿼리로

```sql
-- 권장 패턴
LISTEN queue_channel;
-- 알림 수신 시
SELECT * FROM mq.dequeue(10);  -- 실제 데이터 조회
```

### Q4. pgmq의 가시성 타임아웃(Visibility Timeout)은 무엇인가요?

**모범 답안:**

가시성 타임아웃은 메시지 처리 중 다른 Consumer에게 메시지를 숨기는 시간입니다.

**동작:**
```sql
-- vt=120: 120초간 비가시
SELECT * FROM pgmq.read('queue', vt => 120, qty => 1);
```

1. Consumer가 메시지 조회
2. 120초간 다른 Consumer에게 비가시
3. 처리 성공 → `pgmq.archive()` 호출
4. 처리 실패 (타임아웃) → 자동으로 다시 가시

**장점:**
- At-least-once 보장
- 장애 복구 자동화
- 수동 상태 관리 불필요

**설정 기준:**
- 예상 처리 시간의 2~3배
- 너무 짧으면: 중복 처리
- 너무 길면: 장애 복구 지연

### Q5. 커스텀 큐와 pgmq 중 어떤 것을 선택해야 하나요?

**모범 답안:**

**커스텀 큐 선택:**
- LISTEN/NOTIFY 필요
- 완전한 제어 필요
- 특수한 상태 관리
- 학습 목적

**pgmq 선택:**
- 빠른 구현 필요
- 가시성 타임아웃 필요
- 아카이브/히스토리 필요
- AWS SQS 유사 API 선호

**결정 플로우:**
```
가시성 타임아웃 필요? → pgmq
LISTEN/NOTIFY 필요? → 커스텀
빠른 구현 원함? → pgmq
완전한 제어 원함? → 커스텀
```

### Q6. 고볼륨 큐에서 성능을 최적화하는 방법은?

**모범 답안:**

**1. 부분 인덱스:**
```sql
CREATE INDEX ON mq.queue (created_at)
WHERE status = 'new';
-- new 상태만 인덱싱, completed 제외
```

**2. 파티셔닝:**
```sql
CREATE TABLE mq.queue (...) PARTITION BY RANGE (created_at);
-- 일별 파티션으로 VACUUM/DROP 효율화
```

**3. 배치 처리:**
```sql
SELECT mq.dequeue(100);  -- 한 번에 여러 메시지
SELECT mq.mark_completed(ARRAY[1,2,3,...]);
```

**4. 처리 완료 메시지 정리:**
- pgmq: 자동 아카이브
- 커스텀: pg_cron으로 주기적 DELETE/TRUNCATE

**5. VACUUM 정책:**
```sql
ALTER TABLE mq.queue SET (autovacuum_vacuum_scale_factor = 0.01);
-- 더 자주 VACUUM
```
