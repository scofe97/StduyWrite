# Chapter 4: Indexes - 면접정리

## 핵심 개념 상세 설명

### 1. 인덱스 기본 개념

```
인덱스의 역할
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  인덱스 없이 (Full Table Scan):                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Page 1 → Page 2 → Page 3 → ... → Page N            │   │
│  │  O(N) - 모든 행을 순차 스캔                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  인덱스 사용 (Index Scan):                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │       [Index Structure]                              │   │
│  │             ↓                                        │   │
│  │       직접 해당 행 접근                              │   │
│  │  O(log N) - 트리 탐색                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Trade-off:                                                 │
│  ├─ 장점: SELECT 성능 향상                                 │
│  └─ 단점: INSERT/UPDATE/DELETE 시 인덱스 유지 비용         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. B-tree 인덱스

```
B-tree 구조
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│                     [Root Node]                             │
│                    [50 | 100]                               │
│                   /     |     \                             │
│                  /      |      \                            │
│      [Leaf Nodes]   [Leaf]   [Leaf Nodes]                   │
│      [10,20,30,40] [60,70,80,90] [110,120,130]              │
│          ↓              ↓            ↓                      │
│       [Heap Tuples - 실제 데이터 위치]                      │
│                                                             │
│  특징:                                                       │
│  • 시간복잡도: O(log N)                                     │
│  • 균형 트리 유지                                           │
│  • 범위 검색에 효율적                                       │
│  • PostgreSQL 기본 인덱스 타입                              │
│                                                             │
│  지원 연산자:                                                │
│  <, <=, =, >=, >, BETWEEN, IN, LIKE 'prefix%'              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- B-tree 인덱스 생성 (기본)
CREATE INDEX idx_users_email ON users(email);

-- 정렬 방향 지정
CREATE INDEX idx_orders_date_desc ON orders(order_date DESC);

-- NULL 위치 지정
CREATE INDEX idx_products_price ON products(price NULLS LAST);
```

### 3. Hash 인덱스

```
Hash 인덱스 구조
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  hash(value) → bucket                                       │
│                                                             │
│  ┌───────────────────────────────────────────────────┐     │
│  │ Bucket 0: [value1 → row_ptr, value2 → row_ptr]    │     │
│  │ Bucket 1: [value3 → row_ptr]                      │     │
│  │ Bucket 2: [value4 → row_ptr, value5 → row_ptr]    │     │
│  │ ...                                               │     │
│  └───────────────────────────────────────────────────┘     │
│                                                             │
│  특징:                                                       │
│  • 시간복잡도: O(1) - 상수 시간                             │
│  • 등호(=) 연산만 지원                                      │
│  • 범위 검색 불가                                           │
│  • B-tree보다 작은 크기                                     │
│                                                             │
│  사용 시점:                                                  │
│  • UUID, 해시값 등 등호 검색만 필요한 경우                  │
│  • 카디널리티가 높은 컬럼                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- Hash 인덱스 생성
CREATE INDEX idx_users_uuid ON users USING hash(uuid);

-- 언제 Hash 선택?
-- 등호 검색만 사용하고, 범위 검색 없을 때
SELECT * FROM users WHERE uuid = 'abc-123';  -- Hash 적합
SELECT * FROM users WHERE created_at > '2025-01-01';  -- B-tree 필요
```

### 4. 복합 인덱스 (Composite Index)

```
복합 인덱스와 Leading Column
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  CREATE INDEX idx_composite ON orders(customer_id, date);  │
│                                                             │
│  인덱스 정렬 순서:                                          │
│  (customer_id=1, date=2025-01-01)                          │
│  (customer_id=1, date=2025-01-02)                          │
│  (customer_id=1, date=2025-01-03)                          │
│  (customer_id=2, date=2025-01-01)  ← customer_id 우선 정렬 │
│  (customer_id=2, date=2025-01-02)                          │
│                                                             │
│  사용 가능한 쿼리:                                          │
│  ✅ WHERE customer_id = 1                  (첫 컬럼만)      │
│  ✅ WHERE customer_id = 1 AND date > X     (모두 사용)      │
│  ❌ WHERE date > X                         (두 번째만)      │
│     → 인덱스 스캔 불가, Full Scan 발생                      │
│                                                             │
│  핵심 규칙: Leading Column 먼저!                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 복합 인덱스 설계 시 고려사항
-- 1. 선택도가 높은 컬럼을 앞에
-- 2. 등호 조건 컬럼을 범위 조건 컬럼 앞에

-- 좋은 예: status(등호)가 date(범위) 앞
CREATE INDEX idx_orders ON orders(status, order_date);
SELECT * FROM orders WHERE status = 'completed' AND order_date > '2025-01-01';

-- 나쁜 예: date가 앞이면 status만 사용하는 쿼리에서 비효율
CREATE INDEX idx_orders ON orders(order_date, status);
```

### 5. Covering Index (INCLUDE)

```
Covering Index 개념
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  일반 인덱스:                                                │
│  Index Scan → Heap Fetch (테이블 접근)                      │
│       ↓            ↓                                        │
│   인덱스에서    실제 row에서                                │
│   key 찾음      데이터 읽음                                 │
│                                                             │
│  Covering Index (Index-Only Scan):                          │
│  Index Scan → 완료! (테이블 접근 불필요)                    │
│       ↓                                                     │
│  인덱스에 필요한                                            │
│  모든 컬럼 포함                                              │
│                                                             │
│  성능 이점:                                                  │
│  • I/O 감소 (Heap 접근 생략)                                │
│  • 특히 디스크 기반 테이블에서 효과 큼                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- INCLUDE로 Covering Index 생성
CREATE INDEX idx_orders_covering ON orders(customer_id)
    INCLUDE (order_date, amount);

-- 이 쿼리는 Index-Only Scan 가능
SELECT customer_id, order_date, amount
FROM orders
WHERE customer_id = 123;

-- INCLUDE vs 복합 인덱스 키
-- INCLUDE: 검색에는 사용 안 됨, 조회만
-- 복합 키: 검색 조건으로 사용 가능
```

### 6. 부분 인덱스 (Partial Index)

```
부분 인덱스 개념
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  전체 인덱스:                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  모든 행 인덱싱 (active, inactive, deleted...)      │   │
│  │  인덱스 크기: 100MB                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  부분 인덱스 (WHERE status = 'active'):                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  active 행만 인덱싱                                  │   │
│  │  인덱스 크기: 10MB (90% 감소!)                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  사용 시점:                                                  │
│  • 특정 조건의 행만 자주 조회                               │
│  • 전체의 일부만 활성 상태                                  │
│  • NULL이 아닌 값만 조회                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 부분 인덱스 생성
CREATE INDEX idx_active_orders ON orders(order_date)
    WHERE status = 'active';

-- 이 쿼리에서 인덱스 사용됨
SELECT * FROM orders
WHERE status = 'active' AND order_date > '2025-01-01';

-- 주의: status = 'completed'에서는 인덱스 사용 안 됨
```

### 7. 함수 기반 인덱스 (Expression Index)

```sql
-- 대소문자 무시 검색
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

SELECT * FROM users WHERE LOWER(email) = 'user@example.com';

-- 날짜 추출
CREATE INDEX idx_orders_month ON orders(DATE_TRUNC('month', order_date));

SELECT * FROM orders
WHERE DATE_TRUNC('month', order_date) = '2025-01-01';

-- JSONB 필드
CREATE INDEX idx_profile_city ON users((profile->>'city'));

SELECT * FROM users WHERE profile->>'city' = 'Seoul';
```

```
함수 기반 인덱스 주의사항
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  중요: 쿼리의 표현식이 인덱스 정의와 정확히 일치해야 함!    │
│                                                             │
│  인덱스: LOWER(email)                                       │
│  ✅ WHERE LOWER(email) = 'x'                               │
│  ❌ WHERE email = 'X'                                       │
│  ❌ WHERE UPPER(email) = 'X'                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 8. EXPLAIN ANALYZE

```sql
-- 실행 계획 분석
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders WHERE customer_id = 123;

-- 출력 해석
-- Seq Scan on orders  (cost=0.00..1520.00 rows=100 width=120)
--                      (actual time=0.023..15.234 rows=98 loops=1)
--   Filter: (customer_id = 123)
--   Rows Removed by Filter: 49902
--   Buffers: shared hit=520

-- Index Scan using idx_customer on orders
--                      (cost=0.42..8.44 rows=100 width=120)
--                      (actual time=0.028..0.156 rows=98 loops=1)
--   Index Cond: (customer_id = 123)
--   Buffers: shared hit=5
```

```
EXPLAIN 출력 해석
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  cost=시작비용..총비용                                       │
│  • 시작비용: 첫 row 반환까지                                │
│  • 총비용: 모든 row 반환까지 (arbitrary unit)              │
│                                                             │
│  rows=예상 행 수                                             │
│  actual rows=실제 행 수 (ANALYZE 시)                        │
│                                                             │
│  Buffers:                                                    │
│  • shared hit: 메모리 캐시에서 읽음                         │
│  • shared read: 디스크에서 읽음                             │
│                                                             │
│  스캔 유형:                                                  │
│  • Seq Scan: 전체 테이블 스캔 (인덱스 미사용)               │
│  • Index Scan: 인덱스로 검색 후 테이블 접근                 │
│  • Index Only Scan: 인덱스만으로 완료 (Covering)            │
│  • Bitmap Index Scan: 비트맵으로 여러 행 수집 후 접근       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 비교표

### 인덱스 타입 비교

| 타입 | 시간복잡도 | 지원 연산 | 용도 |
|------|-----------|----------|------|
| B-tree | O(log N) | <, <=, =, >=, >, BETWEEN | 범용, 기본값 |
| Hash | O(1) | = | 등호 검색만 |
| GIN | O(log N) | 포함, 존재 | JSONB, 배열, FTS |
| GiST | O(log N) | 범위, 근접 | 지리공간, 범위 |
| BRIN | O(1) | 범위 | 대용량 순차 데이터 |

### 인덱스 설계 전략

| 전략 | 효과 | 사용 시점 |
|------|------|----------|
| 복합 인덱스 | 여러 조건 최적화 | 자주 함께 사용되는 조건 |
| Covering (INCLUDE) | I/O 감소 | SELECT 컬럼이 고정된 경우 |
| 부분 인덱스 | 크기 감소, 속도 향상 | 특정 조건만 조회 |
| 함수 기반 | 변환 후 검색 최적화 | LOWER, DATE_TRUNC 등 |

## 면접 예상 질문 및 모범 답안

### Q1. B-tree 인덱스의 구조와 동작 원리를 설명해주세요.

**모범 답안:**

B-tree(Balanced Tree)는 PostgreSQL의 기본 인덱스 타입으로, 균형 잡힌 트리 구조입니다.

**구조:**
- Root 노드 → Internal 노드 → Leaf 노드
- Leaf 노드에 실제 키 값과 테이블 행 포인터 저장
- 모든 Leaf 노드가 같은 깊이 (균형 유지)

**동작 원리:**
1. 검색: Root에서 시작해 비교하며 내려감
2. O(log N) 시간 복잡도
3. 범위 검색: Leaf 노드 간 연결로 순차 탐색

**지원 연산:**
- 등호: =
- 범위: <, <=, >=, >, BETWEEN
- LIKE 'prefix%' (앞부분 일치)

**선택 이유:**
범용적이고 대부분의 쿼리 패턴에 적합하기 때문에 기본값입니다.

### Q2. 복합 인덱스에서 컬럼 순서가 중요한 이유는?

**모범 답안:**

복합 인덱스는 첫 번째 컬럼(Leading Column)을 기준으로 정렬되고, 그 안에서 두 번째 컬럼으로 정렬됩니다.

**예시:**
```sql
CREATE INDEX idx ON orders(customer_id, order_date);
```

이 인덱스는 다음처럼 정렬됩니다:
```
(1, 2025-01-01), (1, 2025-01-02), (2, 2025-01-01), (2, 2025-01-02)
```

**인덱스 사용 가능 여부:**
- `WHERE customer_id = 1` ✅ (첫 컬럼 사용)
- `WHERE customer_id = 1 AND order_date > X` ✅ (모두 사용)
- `WHERE order_date > X` ❌ (두 번째만으로 검색 불가)

**설계 원칙:**
1. 선택도가 높은 컬럼을 앞에
2. 등호 조건 컬럼을 범위 조건 앞에
3. 자주 단독으로 사용되는 컬럼을 앞에

### Q3. Covering Index(INCLUDE)의 장점과 사용 시나리오를 설명해주세요.

**모범 답안:**

Covering Index는 쿼리에 필요한 모든 컬럼을 인덱스에 포함시켜 테이블 접근을 생략하는 기법입니다.

**장점:**
1. **I/O 감소**: Heap(테이블) 접근 불필요
2. **성능 향상**: 특히 디스크 I/O가 병목인 경우
3. **Index-Only Scan** 가능

**INCLUDE vs 복합 인덱스 키:**
```sql
-- INCLUDE: 검색에는 사용 안 됨, 조회만
CREATE INDEX idx ON orders(customer_id) INCLUDE (amount, status);

-- 복합 키: 검색 조건으로도 사용 가능
CREATE INDEX idx ON orders(customer_id, amount);
```

**사용 시나리오:**
- SELECT 컬럼이 고정된 자주 실행되는 쿼리
- 조회 컬럼이 인덱스 키로는 부적합한 경우 (예: TEXT)

### Q4. 부분 인덱스(Partial Index)는 어떤 상황에서 사용하나요?

**모범 답안:**

부분 인덱스는 테이블의 일부 행만 인덱싱하는 기법입니다.

**사용 시나리오:**

1. **상태 기반 조회:**
```sql
-- 활성 주문만 자주 조회
CREATE INDEX idx_active ON orders(order_date) WHERE status = 'active';
```

2. **NULL 제외:**
```sql
-- NULL이 아닌 이메일만 인덱싱
CREATE INDEX idx_email ON users(email) WHERE email IS NOT NULL;
```

3. **최근 데이터:**
```sql
-- 최근 1년 데이터만
CREATE INDEX idx_recent ON logs(created_at)
WHERE created_at > '2024-01-01';
```

**장점:**
- 인덱스 크기 대폭 감소
- 쓰기 성능 향상 (인덱싱 대상 감소)
- 쿼리 성능 향상 (작은 인덱스)

### Q5. EXPLAIN ANALYZE 결과를 어떻게 해석하나요?

**모범 답안:**

EXPLAIN ANALYZE는 실제 쿼리를 실행하고 상세 통계를 제공합니다.

**주요 지표:**

1. **cost**: 예상 비용 (낮을수록 좋음)
2. **actual time**: 실제 실행 시간 (ms)
3. **rows**: 예상 vs 실제 행 수
4. **Buffers**: 메모리/디스크 I/O

**스캔 타입 우선순위:**
```
Index Only Scan > Index Scan > Bitmap Index Scan > Seq Scan
```

**문제 징후:**
- 예상 rows와 실제 rows 큰 차이 → 통계 갱신 필요 (ANALYZE)
- Seq Scan on 대형 테이블 → 인덱스 필요
- Buffers: shared read 많음 → 메모리 부족

```sql
-- 통계 갱신
ANALYZE orders;
```

### Q6. 인덱스를 만들지 말아야 하는 경우는?

**모범 답안:**

인덱스는 항상 좋은 것이 아니며, 오버헤드가 있습니다.

**인덱스 피해야 하는 경우:**

1. **작은 테이블**: Full Scan이 더 효율적
2. **낮은 카디널리티**: boolean, status 등 (전체의 50%가 같은 값)
3. **자주 갱신되는 컬럼**: INSERT/UPDATE/DELETE 시 인덱스 유지 비용
4. **거의 조회 안 하는 컬럼**: 쓰기 비용만 증가

**인덱스 오버헤드:**
- 디스크 공간 사용
- 쓰기 시 인덱스 갱신
- VACUUM 대상 증가

**검증 방법:**
```sql
-- 인덱스 사용률 확인
SELECT indexrelname, idx_scan, idx_tup_read
FROM pg_stat_user_indexes
WHERE schemaname = 'public';

-- idx_scan = 0이면 사용 안 되는 인덱스
```

## 실무 체크리스트

```
□ EXPLAIN ANALYZE로 쿼리 분석 후 인덱스 추가
□ 복합 인덱스 컬럼 순서 최적화 (등호 → 범위, 고선택도 → 저선택도)
□ Covering Index로 자주 조회되는 컬럼 포함
□ 부분 인덱스로 활성 데이터만 인덱싱
□ 함수 기반 인덱스 시 쿼리 표현식 일치 확인
□ 정기적으로 사용되지 않는 인덱스 제거
□ VACUUM, ANALYZE 정책 설정
```
