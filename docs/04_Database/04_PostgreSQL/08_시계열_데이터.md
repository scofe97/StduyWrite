# Chapter 9: Postgres for Time Series - 면접정리

## 핵심 개념 상세 설명

### 1. 시계열 데이터 특성

```
시계열 데이터 특징
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 특성 1: 타임스탬프 필수 】                               │
│  모든 데이터 포인트에 시간 정보 포함                        │
│  • 언제 측정되었는지가 핵심                                 │
│  • TIMESTAMPTZ 타입 권장 (시간대 자동 변환)                 │
│                                                             │
│  【 특성 2: Append-Only 패턴 】                              │
│  한번 기록되면 수정하지 않음                                │
│  • INSERT 위주, UPDATE 거의 없음                           │
│  • 과거 데이터는 읽기 전용                                  │
│                                                             │
│  【 데이터 증가율 예시 】                                    │
│  1분 간격 × 1,000명 × 365일 = 5.26억 레코드/년             │
│  → 파티셔닝/압축 필수                                       │
│                                                             │
│  【 예시 】                                                  │
│  • 서버 메트릭 (CPU, Memory)                                │
│  • IoT 센서 데이터 (온도, 습도)                             │
│  • 금융 시계열 (주가, 환율)                                 │
│  • 사용자 활동 로그                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. PostgreSQL 파티셔닝

```
Range Partitioning by Time
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Parent Table: measurements                                 │
│       ↓                                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Jan 2025 │ Feb 2025 │ Mar 2025 │ ... │ Dec 2025   │   │
│  │ Partition│ Partition│ Partition│     │ Partition  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  장점:                                                       │
│  • 파티션 프루닝: 관련 파티션만 스캔                        │
│  • 파티션 단위 VACUUM, DROP                                 │
│  • 오래된 데이터 빠른 삭제 (DROP PARTITION)                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- 파티션 테이블 생성
CREATE TABLE measurements (
    device_id INT NOT NULL,
    recorded_at TIMESTAMPTZ NOT NULL,
    value NUMERIC NOT NULL
) PARTITION BY RANGE (recorded_at);

-- 월별 파티션 생성
CREATE TABLE measurements_2025_01
    PARTITION OF measurements
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

CREATE TABLE measurements_2025_02
    PARTITION OF measurements
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');

-- 파티션 프루닝 확인
EXPLAIN SELECT * FROM measurements
WHERE recorded_at BETWEEN '2025-01-15' AND '2025-01-20';
-- → measurements_2025_01 파티션만 스캔
```

### 3. TimescaleDB 확장

```
TimescaleDB Hypertable
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Hypertable (논리적 테이블)                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐       │   │
│  │  │Chunk 1 │ │Chunk 2 │ │Chunk 3 │ │Chunk N │       │   │
│  │  │Dec-Jan │ │Jan-Feb │ │Feb-Mar │ │  ...   │       │   │
│  │  └────────┘ └────────┘ └────────┘ └────────┘       │   │
│  │                                                     │   │
│  │  • 자동 청크(파티션) 생성                           │   │
│  │  • 시간 범위별 자동 분할                            │   │
│  │  • 수동 파티션 관리 불필요                          │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- TimescaleDB 설정
CREATE EXTENSION timescaledb;

-- 하이퍼테이블 생성
SELECT create_hypertable(
    'measurements',
    by_range('recorded_at', INTERVAL '1 month')
);

-- 청크 조회
SELECT show_chunks('measurements');
```

### 4. time_bucket 함수

```sql
-- 기본 사용법
SELECT
    time_bucket('10 minutes', recorded_at) AS bucket,
    AVG(heart_rate) AS avg_rate,
    MAX(heart_rate) AS max_rate
FROM measurements
WHERE recorded_at >= '2025-01-01'
GROUP BY bucket
ORDER BY bucket;

-- 시작점 지정
SELECT
    time_bucket('1 week', recorded_at, '2025-01-01'::timestamptz) AS week,
    SUM(amount) AS weekly_total
FROM sales
GROUP BY week;

-- 시간대 지정
SELECT
    time_bucket('1 day', recorded_at, 'Asia/Seoul') AS day,
    COUNT(*) AS count
FROM events
GROUP BY day;
```

```
time_bucket 파라미터
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  time_bucket(interval, timestamp, [origin], [timezone])     │
│                                                             │
│  interval: 버킷 크기                                        │
│  • '5 minutes', '1 hour', '1 day', '1 week'                │
│                                                             │
│  origin: 버킷 시작점 (선택)                                 │
│  • 기본: Unix epoch (1970-01-01)                           │
│  • 월요일 시작 주간: 원하는 월요일 날짜 지정               │
│                                                             │
│  timezone: 시간대 (선택)                                    │
│  • 'Asia/Seoul', 'America/New_York'                        │
│  • DST(서머타임) 자동 처리                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5. time_bucket_gapfill

```
데이터 갭 채우기
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  원본 데이터 (갭 존재):                                     │
│  07:25 → 127                                                │
│  07:29 → 121                                                │
│  (07:30, 07:31 누락)                                        │
│  07:32 → 122                                                │
│                                                             │
│  time_bucket_gapfill + LOCF:                                │
│  07:25 → 127                                                │
│  07:29 → 121                                                │
│  07:30 → 121 (이전 값 유지)                                 │
│  07:31 → 121 (이전 값 유지)                                 │
│  07:32 → 122                                                │
│                                                             │
│  time_bucket_gapfill + interpolate:                         │
│  07:29 → 121                                                │
│  07:30 → 121.33 (선형 보간)                                 │
│  07:31 → 121.67 (선형 보간)                                 │
│  07:32 → 122                                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- LOCF (Last Observation Carried Forward)
SELECT
    time_bucket_gapfill('1 minute', recorded_at) AS minute,
    LOCF(AVG(value)) AS filled_value
FROM measurements
WHERE recorded_at BETWEEN '2025-01-01 07:00' AND '2025-01-01 08:00'
GROUP BY minute;

-- 선형 보간
SELECT
    time_bucket_gapfill('1 minute', recorded_at) AS minute,
    interpolate(AVG(value)) AS interpolated_value
FROM measurements
WHERE recorded_at BETWEEN '2025-01-01 07:00' AND '2025-01-01 08:00'
GROUP BY minute;
```

### 6. 연속 집계 (Continuous Aggregates)

```sql
-- 연속 집계 생성
CREATE MATERIALIZED VIEW hourly_stats
WITH (timescaledb.continuous) AS
SELECT
    device_id,
    time_bucket('1 hour', recorded_at) AS hour,
    AVG(value) AS avg_value,
    MIN(value) AS min_value,
    MAX(value) AS max_value,
    COUNT(*) AS sample_count
FROM measurements
GROUP BY device_id, hour;

-- 자동 갱신 정책
SELECT add_continuous_aggregate_policy(
    'hourly_stats',
    start_offset => INTERVAL '3 hours',
    end_offset => INTERVAL '1 hour',
    schedule_interval => INTERVAL '1 hour'
);

-- 수동 갱신
CALL refresh_continuous_aggregate(
    'hourly_stats',
    '2025-01-01 00:00',
    '2025-01-02 00:00'
);
```

```
연속 집계 동작 방식
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  원본 데이터 (1분 간격)                                     │
│       ↓ 자동/수동 갱신                                      │
│  연속 집계 (1시간 집계)                                     │
│       ↓ 쿼리                                                │
│  빠른 응답 (사전 계산됨)                                    │
│                                                             │
│  장점:                                                       │
│  • 반복 집계 쿼리 최적화                                    │
│  • 하이퍼테이블로 자동 파티셔닝                             │
│  • 증분 갱신 (변경분만)                                     │
│  • 인덱스 생성 가능                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 8. B-tree vs BRIN 인덱스

```
시계열 인덱스 선택
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  【 B-tree 】                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  실제 값과 행 포인터 저장                            │   │
│  │  • 직접 행 접근 가능                                │   │
│  │  • 크기: 수천 KB                                    │   │
│  │  • 작은 범위 쿼리에 최적                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【 BRIN (Block Range Index) 】                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  페이지 범위별 min/max만 저장                        │   │
│  │  • 범위 검색으로 페이지 필터링                      │   │
│  │  • 크기: 수십 KB (매우 작음!)                       │   │
│  │  • 순차 저장된 데이터에 효과적                      │   │
│  │  • 시계열 데이터에 적합                             │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  선택 기준:                                                  │
│  • 특정 시점 조회 많음 → B-tree                            │
│  • 범위 분석 쿼리 → BRIN                                   │
│  • 저장 공간 제약 → BRIN                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```sql
-- B-tree 인덱스
CREATE INDEX idx_btree ON measurements (recorded_at, device_id);

-- BRIN 인덱스
CREATE INDEX idx_brin ON measurements USING brin (recorded_at);

-- 인덱스 크기 비교
SELECT
    indexname,
    pg_size_pretty(pg_relation_size(indexname::regclass)) AS size
FROM pg_indexes
WHERE tablename = 'measurements';
```

## 비교표

### B-tree vs BRIN 인덱스

| 특성 | B-tree | BRIN |
|------|--------|------|
| 인덱스 크기 | 크다 | 매우 작다 |
| 포인트 조회 | 빠름 | 느림 |
| 범위 조회 | 보통 | 효율적 |
| 순차 데이터 | 보통 | 최적 |
| 권장 용도 | 특정 시점 조회 | 범위 분석 |

### 갭 채우기 방법 비교

| 방법 | 용도 | 예시 |
|------|------|------|
| LOCF | 값이 유지되는 경우 | 재고량, 잔액 |
| interpolate | 연속 변화하는 값 | 온도, 속도 |

## 면접 예상 질문 및 모범 답안

### Q1. 시계열 데이터의 특성과 일반 테이블과의 차이점은?

**모범 답안:**

**시계열 데이터 특성:**
1. **타임스탬프 필수**: 모든 레코드에 시간 정보 포함
2. **Append-Only**: INSERT 위주, UPDATE 거의 없음
3. **대용량 증가**: 짧은 간격으로 지속적 생성
4. **시간 기반 쿼리**: 범위 조회, 집계가 주요 패턴

**일반 테이블과의 차이:**

| 특성 | 일반 테이블 | 시계열 테이블 |
|------|------------|--------------|
| 쓰기 패턴 | CRUD 모두 | INSERT 위주 |
| 쿼리 패턴 | 포인트 조회 | 범위 + 집계 |
| 인덱스 | B-tree | B-tree + BRIN |
| 최적화 | 정규화 | 파티셔닝, 압축 |

**설계 고려사항:**
- 파티셔닝으로 오래된 데이터 관리
- 보존 정책으로 자동 삭제
- 연속 집계로 쿼리 성능 향상

### Q2. TimescaleDB의 Hypertable과 일반 파티션 테이블의 차이점은?

**모범 답안:**

**일반 파티션 테이블:**
```sql
CREATE TABLE ... PARTITION BY RANGE (recorded_at);
-- 각 파티션 수동 생성 필요
CREATE TABLE ... PARTITION OF ... FOR VALUES FROM ... TO ...;
```
- 파티션을 수동으로 생성/관리
- 파티션 범위 직접 지정

**TimescaleDB Hypertable:**
```sql
SELECT create_hypertable('measurements', by_range('recorded_at'));
-- 데이터 삽입 시 청크 자동 생성
```

**차이점:**

| 기능 | 일반 파티션 | Hypertable |
|------|------------|------------|
| 파티션 생성 | 수동 | 자동 |
| 시계열 함수 | 없음 | time_bucket 등 |
| 연속 집계 | 없음 | 지원 |
| 보존 정책 | 직접 구현 | 내장 |
| 압축 | 없음 | 지원 |

### Q3. time_bucket 함수의 용도와 사용 방법을 설명해주세요.

**모범 답안:**

time_bucket은 시간을 지정된 간격으로 버킷화하는 함수입니다.

**용도:**
- 시간 단위 집계 (분/시간/일/주별)
- 리포트, 대시보드 데이터
- 이동 평균 계산

**기본 사용법:**
```sql
SELECT
    time_bucket('1 hour', recorded_at) AS hour,
    AVG(value) AS avg_value
FROM measurements
GROUP BY hour
ORDER BY hour;
```

**고급 옵션:**
```sql
-- 시작점 지정 (월요일 시작 주간)
time_bucket('1 week', recorded_at, '2025-01-06'::timestamptz)

-- 시간대 지정
time_bucket('1 day', recorded_at, 'Asia/Seoul')

-- 오프셋
time_bucket('1 day', recorded_at, origin, timezone, INTERVAL '6 hours')
```

### Q4. 연속 집계(Continuous Aggregates)의 장점과 사용 시점은?

**모범 답안:**

연속 집계는 사전 계산된 시계열 집계를 저장하는 Materialized View입니다.

**장점:**
1. **쿼리 성능**: 사전 계산으로 빠른 응답
2. **증분 갱신**: 변경된 데이터만 업데이트
3. **자동 파티셔닝**: Hypertable로 관리
4. **실시간 쿼리**: 최신 데이터 포함 옵션

**사용 시점:**
- 대시보드에서 반복 조회되는 집계
- 시간별/일별/월별 리포트
- 긴 기간의 집계 쿼리

**설정 예시:**
```sql
CREATE MATERIALIZED VIEW daily_stats
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 day', ts) AS day, AVG(value)
FROM measurements GROUP BY day;

SELECT add_continuous_aggregate_policy(
    'daily_stats',
    start_offset => INTERVAL '3 days',
    end_offset => INTERVAL '1 day',
    schedule_interval => INTERVAL '1 day'
);
```

### Q5. 시계열 데이터에서 BRIN 인덱스가 효과적인 이유는?

**모범 답안:**

BRIN(Block Range Index)은 페이지 범위별로 min/max 값만 저장하는 인덱스입니다.

**시계열에서 효과적인 이유:**

1. **순차 저장 패턴:**
   - 시계열 데이터는 시간순으로 INSERT
   - 물리적 저장 순서와 시간 순서가 일치
   - BRIN이 효과적으로 범위 필터링

2. **인덱스 크기:**
   ```
   B-tree: 수천 KB
   BRIN: 수십 KB (100배 이상 작음)
   ```

3. **범위 쿼리 패턴:**
   - 시계열은 대부분 시간 범위 조회
   - BRIN은 범위 쿼리에 최적화

**주의점:**
- 랜덤 삽입/업데이트 시 효과 감소
- 포인트 조회에는 B-tree가 나음

### Q6. 데이터 보존 정책(Retention Policy)은 왜 필요하고 어떻게 설정하나요?

**모범 답안:**

**필요성:**
- 시계열 데이터는 빠르게 증가
- 오래된 데이터는 가치 감소
- 스토리지 비용 관리
- 쿼리 성능 유지

**설정 방법:**

```sql
-- 원본 데이터: 30일 보존
SELECT add_retention_policy(
    'measurements',
    drop_after => INTERVAL '30 days',
    schedule_interval => INTERVAL '1 day'  -- 매일 실행
);

-- 집계 데이터: 1년 보존
SELECT add_retention_policy(
    'hourly_stats',
    drop_after => INTERVAL '1 year'
);
```

**전략:**
- 원본 데이터: 짧은 보존 (30일)
- 시간별 집계: 중간 보존 (1년)
- 일별/월별 집계: 장기 보존 (영구)

이렇게 하면 상세 데이터는 짧게, 집계 데이터는 길게 유지하여 효율적입니다.
