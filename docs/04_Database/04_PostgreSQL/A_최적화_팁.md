# Appendix A: Five Optimization Tips - 면접 정리

## 1. 핵심 개념 상세 설명

### EXPLAIN 명령어 체계

```
EXPLAIN 파라미터 구조:
┌─────────────────────────────────────────────────────────────────┐
│                      EXPLAIN 명령어                              │
├─────────────────────────────────────────────────────────────────┤
│  ANALYZE  │  실제 실행하여 런타임 통계 수집 (default: off)       │
│  COSTS    │  예상 비용 정보 표시 (default: on)                   │
│  BUFFERS  │  메모리 버퍼 사용 정보 (default: off)                │
│  TIMING   │  각 단계별 시간 측정 (default: on)                   │
│  FORMAT   │  출력 형식 - TEXT/JSON/XML/YAML                      │
└─────────────────────────────────────────────────────────────────┘

실무 권장 조합:
┌──────────────────────────────────────┐
│  EXPLAIN (ANALYZE, BUFFERS ON)       │  ← 느린 쿼리 분석
│  EXPLAIN (ANALYZE, COSTS OFF)        │  ← 간결한 출력
│  EXPLAIN (ANALYZE, FORMAT JSON)      │  ← 도구 연동
└──────────────────────────────────────┘
```

**버퍼 정보 해석**:
```
버퍼 유형과 의미:
├── shared hit: 101   → 메모리 캐시에서 읽음 (빠름, 최적)
├── shared read: 50   → 디스크에서 읽음 (느림)
└── shared written: 5 → 디스크에 씀

성능 진단:
├── hit 비율 높음 → 캐싱 효과적
├── read 비율 높음 → 메모리 부족 가능성
│                   → 해결: RAM 증설, shared_buffers 조정
└── 예상(cost) ≠ 실제(actual) → ANALYZE tablename; 실행
```

### 인덱스 균형 전략

```
인덱스 상태 스펙트럼:
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  Underindexing          적정 수준          Overindexing        │
│       ◄─────────────────────┼─────────────────────►           │
│                             │                                  │
│  문제:                      │                    문제:         │
│  - Seq Scan 발생            │                    - 쓰기 성능↓ │
│  - 읽기 성능↓               │                    - 스토리지↑  │
│                             │                    - 계획 오버헤드│
└────────────────────────────────────────────────────────────────┘

Underindexing 진단:
┌─────────────────────────────────────┐
│  EXPLAIN 결과에서:                   │
│  → Seq Scan on table_name           │  ← 인덱스 미사용
│  → Rows Removed by Filter: 2375     │  ← 많은 행 필터링
└─────────────────────────────────────┘
         │
         ▼ 해결
┌─────────────────────────────────────┐
│  CREATE INDEX idx_... ON table      │
│  USING GIN/BTREE/... (column);      │
└─────────────────────────────────────┘
         │
         ▼ 결과
┌─────────────────────────────────────┐
│  → Bitmap Index Scan                │
│  → Index Cond: (condition)          │  ← 인덱스 활용
└─────────────────────────────────────┘

Overindexing 비용:
┌──────────────────────┬─────────────────────────────────────────┐
│ 스토리지 비용        │ 각 인덱스가 별도 데이터 구조 유지        │
│ 쓰기 증폭            │ INSERT/UPDATE/DELETE 시 모든 인덱스 갱신 │
│ 플래닝 오버헤드      │ 더 많은 인덱스 고려로 계획 시간 증가     │
└──────────────────────┴─────────────────────────────────────────┘
```

### SELECT 최적화 원칙

```
SELECT * 의 비용:
┌────────────────────────────────────────────────────────────────┐
│  SELECT * FROM orders  (15개 컬럼, 5개만 필요한 경우)          │
│                                                                │
│  불필요한 처리:                                                │
│  ├── 1 row   × 10 extra columns = 10개 불필요 값              │
│  ├── 10 rows × 10 extra columns = 100개 불필요 값             │
│  └── 100 req/sec × 10 rows × 10 = 10,000개/초 낭비            │
│                                                                │
│  비용 발생 위치:                                               │
│  ├── DB     : 불필요한 컬럼 읽기 + 직렬화                     │
│  ├── Network: 추가 데이터 전송                                │
│  └── App    : 역직렬화 + 메모리 사용                          │
└────────────────────────────────────────────────────────────────┘

컬럼 타입별 영향:
┌──────────────────┬─────────────────┬─────────────────────────┐
│ 타입             │ 크기            │ 불필요 조회 시 영향     │
├──────────────────┼─────────────────┼─────────────────────────┤
│ int, timestamp   │ 고정 (4-8 bytes)│ 영향 적음               │
│ text, varchar    │ 가변 (KB~)      │ 영향 큼                 │
│ jsonb, vector    │ 가변 (KB~MB)    │ 영향 매우 큼            │
└──────────────────┴─────────────────┴─────────────────────────┘

count 함수 비교:
┌─────────────────────────────────────────────────────────────┐
│  count(*) vs count(column)                                  │
│                                                             │
│  count(*)       → width=0  → 컬럼 값 안 읽음 (최적)         │
│  count(column)  → width=4+ → 컬럼 값 읽음 (비최적)          │
│                                                             │
│  결론: 행 수만 세려면 count(*) 사용                         │
└─────────────────────────────────────────────────────────────┘
```

### 데이터베이스 연산 활용

```
데이터 처리 위치 결정:
┌─────────────────────────────────────────────────────────────┐
│  작업 유형                │  처리 위치     │  이유           │
├──────────────────────────┼───────────────┼─────────────────┤
│  정렬                    │  DB (ORDER BY)│  인덱스 활용     │
│  그룹핑/집계             │  DB (GROUP BY)│  데이터 근처    │
│  복잡한 집계             │  DB (윈도우)   │  최적화됨       │
│  데이터 집약적 로직      │  DB (함수)     │  네트워크↓      │
└──────────────────────────────────────────────────────────────┘

안티 패턴 vs 권장 패턴:
┌─────────────────────────────────────────────────────────────┐
│  Bad: 애플리케이션에서 정렬                                 │
│  rows = db.query("SELECT * FROM orders")                    │
│  sorted_rows = sorted(rows, key=lambda x: x.created_at)     │
├─────────────────────────────────────────────────────────────┤
│  Good: 데이터베이스에서 정렬                                │
│  rows = db.query("SELECT * FROM orders ORDER BY created_at")│
└─────────────────────────────────────────────────────────────┘

데이터베이스 연산 활용 예시:
├── ORDER BY              : DB 레벨 정렬
├── GROUP BY + 집계함수    : DB 레벨 그룹핑
├── 윈도우 함수           : ROW_NUMBER, RANK 등
├── CTE (WITH 절)         : 복잡한 쿼리 분리
└── 저장 프로시저/함수     : 데이터 집약적 비즈니스 로직
```

## 2. 비교표

### EXPLAIN 파라미터 비교

| 파라미터 | 기본값 | 용도 | 실무 사용 시점 |
|----------|--------|------|----------------|
| `ANALYZE` | off | 실제 실행 + 런타임 통계 | 느린 쿼리 분석 |
| `COSTS` | on | 예상 비용 표시 | 항상 (기본) |
| `BUFFERS` | off | 메모리/디스크 I/O 정보 | 캐싱 효율 분석 |
| `TIMING` | on | 단계별 시간 측정 | 항상 (기본) |
| `FORMAT` | TEXT | 출력 형식 | JSON (도구 연동) |

### 인덱싱 문제 비교

| 구분 | Underindexing | Overindexing |
|------|---------------|--------------|
| **증상** | Seq Scan 발생 | 쓰기 성능 저하 |
| **EXPLAIN 결과** | Filter로 많은 행 제거 | 사용 안 되는 인덱스 |
| **영향** | 읽기 성능 저하 | 쓰기 성능 저하 + 스토리지 증가 |
| **해결** | 적절한 인덱스 추가 | 미사용 인덱스 제거 |
| **확인 쿼리** | `pg_stat_user_tables` | `pg_stat_user_indexes` |

### 커넥션 풀링 유형 비교

| 구분 | Client-side | Server-side |
|------|-------------|-------------|
| **위치** | 애플리케이션 | 데이터베이스 앞단 |
| **예시** | HikariCP, SQLAlchemy pool | PgBouncer, Pgpool-II |
| **장점** | 설정 간단 | 여러 앱 커넥션 집계 |
| **단점** | 앱별 풀 분리 | 추가 인프라 필요 |
| **적합한 환경** | 단일 앱 | 마이크로서비스, 다중 앱 |

### Server-side 풀러 비교

| 풀러 | 특징 | 적합한 환경 |
|------|------|-------------|
| **PgBouncer** | 경량, 트랜잭션/세션/문장 풀링 | 대부분의 환경 |
| **Pgpool-II** | 풀링 + 로드밸런싱 + 복제 | HA 환경 |
| **Odyssey** | 멀티스레드, 고성능 | 고부하 환경 |

### SELECT 최적화 비교

| 패턴 | 성능 | 권장 |
|------|------|------|
| `SELECT *` | 불필요한 데이터 처리 | X |
| `SELECT col1, col2` | 필요한 데이터만 | O |
| `count(*)` | 컬럼 값 안 읽음 | O |
| `count(column)` | 컬럼 값 읽음 | X (행 수만 필요시) |

## 3. 면접 예상 질문 및 모범 답안

### Q1. EXPLAIN ANALYZE의 결과에서 예상 비용(cost)과 실제 비용(actual)이 크게 다를 때 어떻게 대응하시나요?

**모범 답안:**

예상과 실제 비용의 큰 차이는 **통계 정보 불일치**가 주 원인입니다.

**원인 분석:**
```
예상(cost) ≠ 실제(actual) 원인:
├── 통계 정보 오래됨 (autovacuum 주기 문제)
├── 대량 데이터 변경 후 ANALYZE 미실행
└── 데이터 분포 급격한 변화
```

**대응 방안:**
1. **즉각 대응**: `ANALYZE tablename;`으로 통계 갱신
2. **설정 조정**: `autovacuum_analyze_threshold`, `autovacuum_analyze_scale_factor` 튜닝
3. **확장 통계**: 상관관계 있는 컬럼에 `CREATE STATISTICS` 사용

실제 운영에서는 대량 데이터 마이그레이션 후 반드시 ANALYZE를 실행하고, 모니터링을 통해 통계 갱신 주기를 조정합니다.

### Q2. 인덱스를 추가했는데 EXPLAIN에서 여전히 Seq Scan이 나타나는 경우, 가능한 원인과 해결 방법은?

**모범 답안:**

**가능한 원인:**
```
인덱스 미사용 원인:
├── 1. 테이블 크기가 작음 → Seq Scan이 더 효율적
├── 2. 선택도(selectivity) 낮음 → 대부분의 행 반환
├── 3. 인덱스 타입 불일치 → B-tree로 JSONB 검색
├── 4. 함수/연산자 적용 → WHERE LOWER(name) = 'john'
└── 5. 통계 정보 부정확 → ANALYZE 필요
```

**해결 방법:**
1. **테이블 크기**: 데이터가 충분히 많으면 인덱스 사용 (정상)
2. **선택도**: WHERE 조건이 행의 대부분을 반환하면 인덱스 비효율
3. **인덱스 타입**: JSONB는 GIN, 공간 데이터는 GiST 사용
4. **함수 사용**: Expression Index 생성 (`CREATE INDEX ON t (LOWER(name))`)
5. **통계 갱신**: `ANALYZE tablename;` 실행

실무에서는 `SET enable_seqscan = off;`로 강제 인덱스 사용 후 비용 비교를 통해 원인을 파악합니다.

### Q3. PgBouncer의 트랜잭션 풀링과 세션 풀링의 차이점과 선택 기준을 설명해주세요.

**모범 답안:**

**풀링 모드 비교:**
```
┌─────────────────────────────────────────────────────────────┐
│  Session Pooling:                                           │
│  Client ←→ Connection (세션 전체 유지)                      │
│  - 세션 변수, Prepared Statement 사용 가능                  │
│  - 커넥션 재사용 효율 낮음                                  │
├─────────────────────────────────────────────────────────────┤
│  Transaction Pooling:                                       │
│  Client ←→ Connection (트랜잭션 동안만 유지)                │
│  - 트랜잭션 완료 후 즉시 반환                               │
│  - 커넥션 재사용 효율 높음                                  │
│  - 세션 변수, Prepared Statement 사용 불가                  │
└─────────────────────────────────────────────────────────────┘
```

**선택 기준:**
| 상황 | 권장 모드 | 이유 |
|------|----------|------|
| 일반 웹 애플리케이션 | Transaction | 높은 동시성, 짧은 트랜잭션 |
| Prepared Statement 필수 | Session | 트랜잭션 풀링에서 미지원 |
| SET 명령어 사용 | Session | 세션 변수 필요 |
| 커넥션 수 제한 심각 | Transaction | 최대 재사용 |

대부분의 경우 Transaction 풀링이 효율적이지만, 애플리케이션의 세션 상태 의존성을 먼저 확인해야 합니다.

### Q4. SELECT *를 피해야 하는 이유와 count(*)가 count(column)보다 효율적인 이유를 설명해주세요.

**모범 답안:**

**SELECT * 문제:**
```
비용 발생 지점:
1. DB 레벨: 불필요한 컬럼 읽기 + 직렬화
2. 네트워크: 추가 데이터 전송
3. 애플리케이션: 역직렬화 + 메모리 할당

특히 문제되는 경우:
├── JSONB, TEXT 등 가변 크기 컬럼 포함
├── 대용량 테이블에서 많은 행 조회
└── 높은 TPS 환경
```

**count(*) vs count(column):**
```sql
-- count(*): Visibility Map만 확인, 컬럼 값 읽지 않음
EXPLAIN SELECT count(*) FROM orders;
-- width=0 (데이터 읽지 않음)

-- count(column): 컬럼 값을 실제로 읽음
EXPLAIN SELECT count(order_id) FROM orders;
-- width=4 (4바이트씩 읽음)
```

`count(*)`는 NULL 체크 없이 행 존재만 확인하므로 더 효율적입니다. `count(column)`은 NULL 제외를 위해 컬럼 값을 읽어야 합니다.

### Q5. 정렬, 그룹핑, 집계를 애플리케이션 대신 데이터베이스에서 처리해야 하는 이유는?

**모범 답안:**

**DB 처리가 효율적인 이유:**
```
데이터베이스 장점:
├── 1. 인덱스 활용: ORDER BY가 인덱스 활용 가능
├── 2. 데이터 근접성: 디스크 I/O 최소화
├── 3. 네트워크 절감: 집계 결과만 전송
├── 4. 최적화 엔진: 수십 년간 최적화된 쿼리 플래너
└── 5. 메모리 효율: 스트리밍 처리 가능

애플리케이션 처리 문제:
├── 전체 데이터 네트워크 전송
├── 메모리에 전체 데이터 로드
└── 최적화되지 않은 정렬/집계 알고리즘
```

**예시 비교:**
```python
# Bad: 100만 행 전송 후 애플리케이션에서 정렬
rows = db.query("SELECT * FROM orders")  # 100만 행 전송
sorted_rows = sorted(rows)  # 메모리에서 정렬

# Good: DB에서 정렬 후 필요한 만큼만 전송
rows = db.query("SELECT * FROM orders ORDER BY created_at LIMIT 100")
```

단, 복잡한 비즈니스 로직이나 외부 API 호출이 필요한 경우는 애플리케이션에서 처리하는 것이 적합합니다.

### Q6. 프로덕션 환경에서 느린 쿼리를 분석할 때의 단계적 접근 방법을 설명해주세요.

**모범 답안:**

**단계적 분석 프로세스:**
```
1단계: 현상 파악
├── pg_stat_statements로 느린 쿼리 식별
├── auto_explain 또는 log_min_duration_statement 활용
└── 쿼리 빈도, 평균 실행 시간 확인

2단계: EXPLAIN 분석
├── EXPLAIN (ANALYZE, BUFFERS ON)으로 상세 분석
├── 실행 계획에서 병목 지점 식별
│   ├── Seq Scan → 인덱스 필요 여부 검토
│   ├── Nested Loop → 조인 순서, 인덱스 확인
│   └── Sort → 인덱스 ORDER BY 가능 여부
└── 버퍼 히트 비율 확인 (메모리 vs 디스크)

3단계: 원인별 대응
├── 인덱스 부족 → 적절한 인덱스 추가
├── 통계 불일치 → ANALYZE 실행
├── 메모리 부족 → work_mem, shared_buffers 조정
└── 쿼리 구조 문제 → 쿼리 리팩토링

4단계: 검증
├── 수정 후 EXPLAIN으로 개선 확인
├── 프로덕션 반영 전 테스트 환경에서 검증
└── 모니터링으로 지속적 관찰
```

실무에서는 `pg_stat_statements`와 `auto_explain`을 상시 활성화하여 사전 모니터링하고, 문제 발생 시 신속하게 원인을 파악할 수 있도록 합니다.

## 4. 실무 체크리스트

### 성능 문제 발생 시 점검 순서

- [ ] EXPLAIN (ANALYZE, BUFFERS ON)으로 실행 계획 분석
- [ ] 버퍼 히트 비율 확인 (shared hit vs shared read)
- [ ] 예상 비용과 실제 비용 차이 확인
- [ ] Seq Scan 발생 시 인덱스 필요 여부 검토
- [ ] 통계 정보 갱신 필요 여부 확인 (ANALYZE)

### 인덱스 관리 체크리스트

- [ ] 자주 실행되는 쿼리 패턴 파악
- [ ] 하나의 인덱스로 여러 쿼리 커버 가능한지 검토
- [ ] 미사용 인덱스 정기적 점검 (pg_stat_user_indexes)
- [ ] 인덱스 크기와 테이블 크기 비율 모니터링
- [ ] 쓰기 성능 저하 시 인덱스 수 검토

### 커넥션 관리 체크리스트

- [ ] 애플리케이션 레벨 커넥션 풀링 설정 확인
- [ ] 다중 앱 환경에서 PgBouncer 도입 검토
- [ ] max_connections 설정값과 실제 사용량 비교
- [ ] 커넥션 대기 시간 모니터링

### 쿼리 작성 체크리스트

- [ ] SELECT * 대신 필요한 컬럼만 명시
- [ ] 행 수만 필요하면 count(*) 사용
- [ ] 정렬, 그룹핑은 DB 레벨에서 처리
- [ ] 인덱스 활용 가능한 WHERE 조건 작성
- [ ] 대용량 조회 시 LIMIT 사용
