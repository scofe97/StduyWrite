# Chapter 5: Encoding and Evolution

---

## 📌 핵심 요약
> 애플리케이션은 지속적으로 변화하며, 이에 따라 데이터 형식도 진화해야 한다. **인코딩(Encoding)**은 메모리 내 데이터를 바이트 시퀀스로 변환하는 과정이고, **디코딩(Decoding)**은 그 역과정이다. 시스템의 원활한 진화를 위해 **하위 호환성(Backward Compatibility)**과 **상위 호환성(Forward Compatibility)**을 모두 유지해야 한다. JSON/XML 같은 텍스트 포맷은 널리 쓰이지만, Protocol Buffers와 Avro 같은 바이너리 스키마 기반 포맷이 더 컴팩트하고 명확한 호환성 규칙을 제공한다.

## 📖 본문 정리

### 1. 데이터 인코딩의 필요성

프로그램은 데이터를 두 가지 표현으로 다룬다:

```
┌─────────────────────────────────────────────────────────────────┐
│                        메모리 내 표현                             │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐               │
│  │ Objects │ │ Structs │ │  Lists  │ │  Trees  │               │
│  │(포인터) │ │(포인터) │ │(포인터) │ │(포인터) │               │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘               │
│       ↑ CPU 접근에 최적화된 구조                                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓ 인코딩 (Encoding)
                              ↑ 디코딩 (Decoding)
┌─────────────────────────────────────────────────────────────────┐
│                      바이트 시퀀스 표현                           │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 0x83 0xa8 0x75 0x73 0x65 0x72 0x4e 0x61 0x6d 0x65 ...    │   │
│  └──────────────────────────────────────────────────────────┘   │
│       ↑ 파일 저장, 네트워크 전송에 적합                           │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 인코딩은 한국어로 작성된 문서를 모스 부호로 변환하는 것과 같다. 전화선으로 전송하려면 모스 부호(바이트)로 바꿔야 하고, 받는 쪽에서는 다시 한국어로 해석(디코딩)해야 한다.

### 2. 호환성: 하위 호환성 vs 상위 호환성

#### 정의

| 호환성 유형 | 설명 | 구현 난이도 |
|------------|------|-----------|
| **하위 호환성 (Backward)** | 새 코드가 옛 데이터를 읽을 수 있음 | 쉬움 |
| **상위 호환성 (Forward)** | 옛 코드가 새 데이터를 읽을 수 있음 | 어려움 |

#### 롤링 업그레이드 시나리오

```
┌─────────────────────────────────────────────────────────────────┐
│                     Rolling Upgrade 진행 중                      │
│                                                                 │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐         │
│  │ Node A  │   │ Node B  │   │ Node C  │   │ Node D  │         │
│  │  v2.0   │   │  v2.0   │   │  v1.0   │   │  v1.0   │         │
│  │ (새코드) │   │ (새코드) │   │ (옛코드) │   │ (옛코드) │         │
│  └────┬────┘   └────┬────┘   └────┬────┘   └────┬────┘         │
│       │             │             │             │               │
│       └─────────────┴─────────────┴─────────────┘               │
│                           ↓                                     │
│              동일 데이터베이스 공유                                │
│                                                                 │
│  새 코드 → 옛 데이터 읽기: 하위 호환성 필요                         │
│  옛 코드 → 새 데이터 읽기: 상위 호환성 필요                         │
└─────────────────────────────────────────────────────────────────┘
```

#### 상위 호환성의 함정: 데이터 손실

```
┌─────────────────────────────────────────────────────────────────┐
│                    데이터 손실 시나리오                           │
│                                                                 │
│  Step 1: 새 코드가 새 필드 포함하여 저장                          │
│  ┌─────────────────────────────────────────┐                    │
│  │ { userName: "Martin",                   │                    │
│  │   favoriteNumber: 1337,                 │                    │
│  │   photoURL: "http://..." }  ← 새 필드   │                    │
│  └─────────────────────────────────────────┘                    │
│                         ↓                                       │
│  Step 2: 옛 코드가 읽음 (photoURL 모름)                          │
│  ┌─────────────────────────────────────────┐                    │
│  │ { userName: "Martin",                   │                    │
│  │   favoriteNumber: 1337 }                │                    │
│  │   (photoURL 무시됨)                      │                    │
│  └─────────────────────────────────────────┘                    │
│                         ↓                                       │
│  Step 3: 옛 코드가 수정 후 다시 저장                              │
│  ┌─────────────────────────────────────────┐                    │
│  │ { userName: "Martin K.",                │                    │
│  │   favoriteNumber: 1337 }                │                    │
│  │   ⚠️ photoURL 손실!                      │                    │
│  └─────────────────────────────────────────┘                    │
└─────────────────────────────────────────────────────────────────┘
```

**해결책**: 알 수 없는 필드를 보존하는 디코딩 로직 구현

### 3. 인코딩 포맷 비교

#### 3.1 언어 특화 포맷 (비권장)

| 언어 | 포맷 | 문제점 |
|------|------|--------|
| Java | `java.io.Serializable` | 언어 종속, 보안 취약, 성능 저하 |
| Python | `pickle` | 임의 클래스 인스턴스화 → 원격 코드 실행 위험 |
| Ruby | `Marshal` | 버전 호환성 미고려 |

> ⚠️ **보안 경고**: 신뢰할 수 없는 바이트 시퀀스 역직렬화는 **원격 코드 실행(RCE)** 공격으로 이어질 수 있다.

#### 3.2 JSON, XML, CSV

**장점**:
- 널리 지원됨
- 사람이 읽을 수 있음
- 언어 독립적

**단점**:

| 문제 | 설명 |
|------|------|
| 숫자 모호성 | 정수 vs 부동소수점 구분 불가 (JSON) |
| 정밀도 한계 | 2^53 초과 정수 정확도 손실 (JavaScript) |
| 바이너리 미지원 | Base64 인코딩 필요 → 33% 크기 증가 |
| 스키마 복잡성 | JSON Schema, XML Schema 학습 곡선 |

**JSON Schema 예시 - 정수 키 맵 정의**:
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "patternProperties": {
    "^[0-9]+$": {
      "type": "string"
    }
  },
  "additionalProperties": false
}
```

#### 3.3 바이너리 인코딩 비교

**예제 데이터**:
```json
{
    "userName": "Martin",
    "favoriteNumber": 1337,
    "interests": ["daydreaming", "hacking"]
}
```

**인코딩 크기 비교**:

```
┌──────────────────────────────────────────────────────────────┐
│                    인코딩 크기 비교                            │
│                                                              │
│  JSON (텍스트)          ████████████████████████████  81 bytes│
│  MessagePack (바이너리) ██████████████████████████   66 bytes│
│  Protocol Buffers       ████████████████             33 bytes│
│  Avro                   ███████████████              32 bytes│
│                                                              │
│  0        20        40        60        80       100         │
└──────────────────────────────────────────────────────────────┘
```

### 4. Protocol Buffers

#### 스키마 정의

```protobuf
syntax = "proto3";

message Person {
    string user_name = 1;       // 필드 태그 = 1
    int64 favorite_number = 2;  // 필드 태그 = 2
    repeated string interests = 3;  // 필드 태그 = 3
}
```

#### 바이너리 인코딩 구조

```
┌─────────────────────────────────────────────────────────────────┐
│                Protocol Buffers 인코딩 (33 bytes)               │
│                                                                 │
│  ┌────┬────┬────────────────────────────────────────────────┐  │
│  │Tag │Type│              Value                              │  │
│  │ 1  │str │ 06 4d 61 72 74 69 6e ("Martin")                 │  │
│  ├────┼────┼────────────────────────────────────────────────┤  │
│  │Tag │Type│              Value                              │  │
│  │ 2  │int │ b9 0a (1337, varint 인코딩)                      │  │
│  ├────┼────┼────────────────────────────────────────────────┤  │
│  │Tag │Type│              Value                              │  │
│  │ 3  │str │ 0b daydreaming                                  │  │
│  ├────┼────┼────────────────────────────────────────────────┤  │
│  │Tag │Type│              Value                              │  │
│  │ 3  │str │ 07 hacking (repeated = 같은 태그 반복)           │  │
│  └────┴────┴────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**핵심 특징**:
- **필드 태그**: 필드 이름 대신 숫자 사용 → 공간 절약
- **Varint 인코딩**: 작은 숫자는 적은 바이트 사용 (1337 = 2 bytes)
- **repeated**: 리스트를 같은 태그의 반복으로 표현

#### 스키마 진화 규칙

| 작업 | 하위 호환 | 상위 호환 | 주의사항 |
|------|----------|----------|---------|
| 필드 추가 (기본값 있음) | ✅ | ✅ | 새 태그 번호 사용 |
| 필드 삭제 | ✅ | ✅ | 태그 번호 재사용 금지 |
| 필드명 변경 | ✅ | ✅ | 태그만 보존되면 OK |
| 타입 변경 | ⚠️ | ⚠️ | 절삭(truncation) 위험 |

> 💬 **비유**: 필드 태그는 집 주소 같다. 이사(필드명 변경)해도 주소(태그)만 그대로면 우편물(데이터)이 배달된다. 하지만 주소를 바꾸면(태그 변경) 과거의 모든 우편물이 분실된다.

### 5. Avro

#### 스키마 정의 (Avro IDL)

```avro
record Person {
    string               userName;
    union { null, long } favoriteNumber = null;
    array<string>        interests;
}
```

#### JSON 표현

```json
{
    "type": "record",
    "name": "Person",
    "fields": [
        {"name": "userName", "type": "string"},
        {"name": "favoriteNumber", "type": ["null", "long"], "default": null},
        {"name": "interests", "type": {"type": "array", "items": "string"}}
    ]
}
```

#### 핵심 차이점: 태그 번호 없음

```
┌─────────────────────────────────────────────────────────────────┐
│                  Avro 인코딩 (32 bytes)                         │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ 0c 4d 61 72 74 69 6e │ 02 │ b9 14 │ 04 │ ... │ ... │     │  │
│  │  ↑ "Martin" (len=6)  │ ↑  │  ↑    │ ↑  │     │     │     │  │
│  │                      │tag│ 1337  │배열│     │     │     │  │
│  │                      │=1 │       │len │     │     │     │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  ⚠️ 필드 순서와 타입은 스키마에서만 알 수 있음                     │
└─────────────────────────────────────────────────────────────────┘
```

#### Writer's Schema vs Reader's Schema

```
┌─────────────────────────────────────────────────────────────────┐
│              Avro 스키마 해석 방식                               │
│                                                                 │
│  ┌─────────────────┐              ┌─────────────────┐          │
│  │ Writer's Schema │              │ Reader's Schema │          │
│  │ (인코딩 시 사용) │              │ (디코딩 시 사용) │          │
│  ├─────────────────┤              ├─────────────────┤          │
│  │ userName: str   │──────────────│ userName: str   │ ✅ 매칭   │
│  │ favoriteNum: i64│──────────────│ favoriteNum: i64│ ✅ 매칭   │
│  │ interests: arr  │──────────────│ interests: arr  │ ✅ 매칭   │
│  │ photoURL: str   │──────────X   │                 │ 무시됨    │
│  │                 │              │ email: str      │ 기본값    │
│  └─────────────────┘              └─────────────────┘          │
│                                                                 │
│  → 필드명으로 매칭, 순서 무관                                     │
│  → Writer에만 있는 필드: Reader가 무시                           │
│  → Reader에만 있는 필드: 기본값으로 채움                          │
└─────────────────────────────────────────────────────────────────┘
```

#### Writer's Schema를 어떻게 알 수 있나?

| 시나리오 | 해결책 |
|---------|--------|
| 대용량 파일 (수백만 레코드) | 파일 헤더에 스키마 포함 (Avro Object Container) |
| 데이터베이스 | 레코드마다 버전 번호 저장 → 스키마 레지스트리 조회 |
| 네트워크 연결 | 연결 설정 시 스키마 버전 협상 |

#### 동적 스키마 생성의 장점

```
┌─────────────────────────────────────────────────────────────────┐
│              DB 스키마 → Avro 스키마 자동 생성                    │
│                                                                 │
│  ┌───────────────────┐         ┌───────────────────┐           │
│  │   DB 테이블        │   →    │   Avro 스키마      │           │
│  ├───────────────────┤   자동   ├───────────────────┤           │
│  │ id: INT           │   생성   │ id: int           │           │
│  │ name: VARCHAR     │         │ name: string      │           │
│  │ email: VARCHAR    │         │ email: string     │           │
│  └───────────────────┘         └───────────────────┘           │
│                                                                 │
│  DB 스키마 변경 시:                                              │
│  - Protocol Buffers: 태그 번호 수동 관리 필요                     │
│  - Avro: 자동 생성 가능 (태그 없이 필드명으로 매칭)                │
└─────────────────────────────────────────────────────────────────┘
```

### 6. 스키마 기반 인코딩의 장점

| 장점 | 설명 |
|------|------|
| **컴팩트** | 필드명 생략으로 바이너리 JSON보다 작음 |
| **문서화** | 스키마 자체가 문서 역할 |
| **호환성 검증** | 배포 전 스키마 호환성 자동 검사 |
| **코드 생성** | 정적 타입 언어에서 컴파일 타임 타입 체크 |

> 💬 **비유**: 스키마는 건축 설계도와 같다. 설계도(스키마)가 있으면 건물(데이터)을 일관되게 짓고, 증축(필드 추가) 시 기존 구조와 호환되는지 미리 검증할 수 있다.

### 7. 데이터 흐름 모드

#### 7.1 데이터베이스를 통한 데이터 흐름

```
┌─────────────────────────────────────────────────────────────────┐
│                데이터베이스를 통한 데이터 흐름                     │
│                                                                 │
│  ┌─────────────┐                         ┌─────────────┐        │
│  │ 프로세스 A   │                         │ 프로세스 B   │        │
│  │ (쓰기)      │                         │ (읽기)      │        │
│  │             │                         │             │        │
│  │  인코딩 ──→ │                         │ ←── 디코딩  │        │
│  └──────┬──────┘                         └──────┬──────┘        │
│         │                                       │               │
│         ▼                                       ▲               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    Database                              │   │
│  │                                                          │   │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐          │   │
│  │  │5년전 │ │3년전 │ │1년전 │ │어제  │ │오늘  │          │   │
│  │  │v1.0  │ │v1.2  │ │v2.0  │ │v2.1  │ │v2.2  │          │   │
│  │  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘          │   │
│  │                                                          │   │
│  │  → "Data outlives code": 데이터는 코드보다 오래 산다       │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

**핵심 포인트**:
- DB에는 다양한 시대의 스키마로 인코딩된 데이터가 공존
- 읽을 때 최신 스키마로 해석 (스키마 진화)
- 아카이브/백업 시 최신 스키마로 통일하여 저장

#### 7.2 서비스를 통한 데이터 흐름 (REST/RPC)

```
┌─────────────────────────────────────────────────────────────────┐
│                    마이크로서비스 통신                           │
│                                                                 │
│  ┌─────────┐     HTTP/gRPC     ┌─────────┐                     │
│  │ Client  │ ←───────────────→ │ Server  │                     │
│  │         │                   │         │                     │
│  │ 요청 인코딩                  │ 요청 디코딩                    │
│  │ 응답 디코딩                  │ 응답 인코딩                    │
│  └─────────┘                   └─────────┘                     │
```

#### 7.4 워크플로우 엔진과 내구성 실행 (Durable Execution)

```
┌─────────────────────────────────────────────────────────────────┐
│                  결제 처리 워크플로우                             │
│                                                                 │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐     │
│  │  사기   │ →  │ 신용카드 │ →  │   은행   │ →  │  완료   │     │
│  │  탐지   │    │   차감   │    │   입금   │    │         │     │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘     │
│       ↓              ↓              ↓              ↓           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                  Durable Execution Log                   │   │
│  │  [Task 1: OK] [Task 2: OK] [Task 3: 실패] [재시도...]    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ⚡ 장애 발생 시:                                               │
│  - WAL에서 상태 복구                                            │
│  - 이미 완료된 RPC는 저장된 결과 반환 (재실행 안 함)              │
│  - 실패한 Task만 재시도                                         │
└─────────────────────────────────────────────────────────────────┘
```

**Temporal 워크플로우 예시 (Python)**:

```python
@workflow.defn
class PaymentWorkflow:
    @workflow.run
    async def run(self, payment: PaymentRequest) -> PaymentResult:
        # 각 activity는 자동으로 WAL에 기록됨
        is_fraud = await workflow.execute_activity(
            check_fraud,
            payment,
            start_to_close_timeout=timedelta(seconds=15),
        )
        if is_fraud:
            return PaymentResultFraudulent

        credit_card_response = await workflow.execute_activity(
            debit_credit_card,
            payment,
            start_to_close_timeout=timedelta(seconds=15),
        )
        # ...
```

**내구성 실행의 제약사항**:
- 외부 서비스는 **멱등성(Idempotency)** API 제공 필요
- 코드는 **결정론적(Deterministic)**이어야 함
- 랜덤, 시스템 시계 등 비결정론적 함수 사용 주의

**메시지 브로커의 장점**:

| 장점 | 설명 |
|------|------|
| 버퍼링 | 수신자 과부하/장애 시 메시지 보관 |
| 재전송 | 크래시 시 자동 재전송 → 메시지 손실 방지 |
| 디스커버리 불필요 | 직접 연결 대신 브로커 주소만 알면 됨 |
| 브로드캐스트 | 하나의 메시지를 여러 수신자에게 |
| 디커플링 | 송신자는 수신자를 알 필요 없음 |

## 💡 실무 적용 포인트

### Quick Reference

| 상황 | 권장 포맷 | 이유 |
|------|----------|------|
| 외부 API | JSON + OpenAPI | 널리 지원, 사람 읽기 가능 |
| 내부 마이크로서비스 | gRPC (Protocol Buffers) | 고성능, 강력한 타입 |
| 대용량 데이터 파이프라인 | Avro | 스키마 진화, 동적 스키마 |
| 실시간 게임/HFT | FlatBuffers/Cap'n Proto | Zero-copy 성능 |

### 주의사항

- ⚠️ **언어 내장 직렬화 사용 금지**: 보안 취약점, 호환성 문제
- ⚠️ **JSON 정수 정밀도**: 2^53 초과 숫자는 문자열로 처리
- ⚠️ **스키마 레지스트리 사용**: 프로덕션에서 스키마 버전 관리 필수
- ⚠️ **멱등성 API 설계**: RPC 재시도 시 중복 실행 방지

### 기술 면접 예상 질문

**Q1. Protocol Buffers와 Avro의 스키마 진화 방식의 차이점은?**
> A: Protocol Buffers는 필드 태그 번호를 사용하여 필드를 식별하므로 필드명을 자유롭게 변경할 수 있지만, 태그 번호는 절대 변경/재사용 불가. Avro는 태그 대신 필드명을 사용하고 Writer's Schema와 Reader's Schema를 비교하여 스키마 해석. Avro는 DB 스키마에서 자동 생성이 용이하지만, Protocol Buffers는 더 컴팩트한 인코딩 가능.

**Q2. 롤링 업그레이드 중 호환성을 유지하려면 어떤 호환성이 필요한가?**
> A: 하위 호환성(새 코드 → 옛 데이터)과 상위 호환성(옛 코드 → 새 데이터) 모두 필요. 업그레이드 중에는 새 버전과 옛 버전이 공존하므로, 양방향으로 데이터를 읽을 수 있어야 함. 특히 상위 호환성 유지 시 알 수 없는 필드를 보존하는 것이 중요.

**Q3. RPC가 로컬 함수 호출과 다른 점은?**
> A: (1) 네트워크 장애로 예측 불가한 실패 발생 (2) 타임아웃 시 성공/실패 알 수 없음 (3) 재시도 시 중복 실행 가능 → 멱등성 필요 (4) 지연 시간 변동 큼 (5) 모든 파라미터를 직렬화해야 함 (포인터 불가).

**Q4. Durable Execution이 exactly-once 시멘틱스를 보장하는 방법은?**
> A: 모든 RPC 호출과 상태 변경을 WAL에 기록. 장애 복구 시 WAL을 재생하되, 이미 완료된 RPC는 재실행하지 않고 저장된 결과를 반환. 단, 외부 서비스는 멱등성 API를 제공해야 하고, 워크플로우 코드는 결정론적이어야 함.

### 인코딩 기초
- [ ] 인코딩(직렬화)과 디코딩(역직렬화)의 개념을 설명할 수 있다

- [ ] JSON/XML의 장단점과 한계를 알고 있다 (숫자 정밀도, 바이너리)

- [ ] REST/gRPC의 차이와 적합한 사용 사례를 구분한다
- [ ] RPC의 본질적 한계를 이해한다

- [ ] 메시지 브로커의 Queue vs Topic 패턴을 구분한다
- [ ] 액터 모델의 특징과 장점을 알고 있다
