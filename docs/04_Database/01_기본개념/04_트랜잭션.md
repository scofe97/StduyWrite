# 트랜잭션 (Transactions)

> **통합 문서**: 분산 시스템 이론 (DDIA) + PostgreSQL 실무 관점 통합

---

## 📌 핵심 요약

트랜잭션은 여러 읽기/쓰기 작업을 하나의 논리적 단위로 묶어 **전체 성공(commit) 또는 전체 실패(abort)**를 보장하는 추상화 계층이다. ACID 속성, 다양한 격리 수준, 그리고 분산 트랜잭션을 위한 2PC 프로토콜을 다룬다.

---

## 🎯 학습 목표

- [ ] ACID의 각 속성이 무엇을 보장하는지 설명할 수 있다
- [ ] PostgreSQL의 MVCC 동작 원리를 이해할 수 있다
- [ ] Read Committed와 Snapshot Isolation의 차이를 설명할 수 있다
- [ ] Dirty Read, Lost Update, Write Skew를 구분할 수 있다
- [ ] 직렬화 가능 격리 수준의 세 가지 구현 방식을 비교할 수 있다
- [ ] 2PC(Two-Phase Commit)의 동작 원리와 한계를 설명할 수 있다

### 1. 트랜잭션 기본 개념

```
┌─────────────────────────────────────────────────────────────────────┐
│                    트랜잭션의 기본 개념                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   정상 실행:                                                         │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ BEGIN TRANSACTION                                       │       │
│   │   ├── Read(A)                                          │       │
│   │   ├── Write(A)                                         │       │
│   │   ├── Read(B)                                          │       │
│   │   └── Write(B)                                         │       │
│   │ COMMIT  ✅                                              │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   오류 발생 시:                                                      │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ BEGIN TRANSACTION                                       │       │
│   │   ├── Read(A)                                          │       │
│   │   ├── Write(A) ✅                                       │       │
│   │   ├── Read(B)                                          │       │
│   │   └── Write(B) ❌ ERROR!                                │       │
│   │ ROLLBACK → Write(A) 취소                                │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   All or Nothing: 전체 성공 또는 전체 실패                           │
└─────────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 은행 송금 - A 계좌 출금 + B 계좌 입금은 반드시 함께 성공하거나 함께 실패해야 함

### 2. ACID 속성

```
┌─────────────────────────────────────────────────────────────────────┐
│                         ACID 속성                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   A - Atomicity (원자성)                                            │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ "All or Nothing"                                        │       │
│   │ 트랜잭션 중 오류 발생 시 모든 변경사항 롤백               │       │
│   │ ⚠️ 동시성과 무관 (그건 Isolation)                        │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   C - Consistency (일관성)                                          │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ "불변식(Invariant) 유지"                                 │       │
│   │ 예: 모든 계좌의 합계 = 상수                              │       │
│   │ ⚠️ 애플리케이션의 책임 (DB 단독 보장 불가)               │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   I - Isolation (격리성)                                            │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ "동시 실행 트랜잭션 간 간섭 방지"                         │       │
│   │ 이상적: Serializability (직렬화 가능)                    │       │
│   │ 현실: 다양한 격리 수준 제공 (성능 트레이드오프)           │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   D - Durability (지속성)                                           │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ "커밋된 데이터는 영구 저장"                              │       │
│   │ 구현: fsync, WAL, 복제                                   │       │
│   │ ⚠️ 완벽한 지속성은 존재하지 않음                         │       │
│   └─────────────────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────────────────┘
```

#### "Consistency"의 5가지 다른 의미

| 컨텍스트 | 의미 |
|----------|------|
| ACID Consistency | 애플리케이션 불변식 유지 |
| Replica Consistency | 복제본 간 데이터 동기화 (Eventually Consistent) |
| Consistent Snapshot | 특정 시점의 일관된 스냅샷 |
| Consistent Hashing | 샤딩을 위한 해시 알고리즘 |
| CAP Consistency | Linearizability (선형화 가능성) |

### 3. PostgreSQL 트랜잭션 실무

#### 3.1 암시적 트랜잭션

모든 단일 SQL 문은 자동으로 트랜잭션 내에서 실행됨.

```sql
-- 단일 문장도 트랜잭션으로 처리됨
UPDATE products.catalog
SET stock_quantity = stock_quantity + 100
WHERE id = 1;
```

#### 3.2 명시적 트랜잭션

여러 문장을 하나의 원자적 작업으로 묶음.

```sql
BEGIN;  -- 트랜잭션 시작

INSERT INTO sales.orders (id, customer_id, total_amount)
VALUES ('19a0cffc-...', 1, 26.53);

INSERT INTO sales.order_items (order_id, product_id, quantity, price)
VALUES ('19a0cffc-...', 1, 1, 16.54),
       ('19a0cffc-...', 4, 1, 9.99);

UPDATE products.catalog
SET stock_quantity = stock_quantity - 1
WHERE id IN (1, 4);

COMMIT;  -- 성공 시 커밋 (실패 시 자동 롤백)
```

#### 3.3 PostgreSQL MVCC

**MVCC(다중 버전 동시성 제어)**:
- 읽기와 쓰기가 서로 블로킹하지 않음
- 동일 행 동시 수정 시에만 블로킹 발생
- **Readers never block Writers, Writers never block Readers**

### 4. 경쟁 조건(Race Condition) 유형

```
┌─────────────────────────────────────────────────────────────────────┐
│                    동시성 문제 유형                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. Dirty Read: 커밋되지 않은 데이터 읽기                            │
│     T1: Write(x=3) ─────────────────────▶ ABORT                    │
│     T2:          Read(x) → 3 (잘못된 값!)                           │
│                                                                     │
│  2. Dirty Write: 커밋되지 않은 데이터 덮어쓰기                        │
│     T1: Write(listing=Bryce) ─────────────▶ COMMIT                 │
│     T2:        Write(invoice=Aaliyah) ────▶ COMMIT                 │
│     결과: listing=Bryce, invoice=Aaliyah (불일치!)                  │
│                                                                     │
│  3. Read Skew (Non-repeatable Read): 읽기 중 데이터 변경            │
│     T1: Read(A) → $500 ─────────────── Read(B) → $400              │
│     T2:              Transfer $100 (A→B), COMMIT                   │
│     결과: $500 + $400 = $900 (실제는 $1000)                         │
│                                                                     │
│  4. Lost Update: 동시 수정으로 업데이트 손실                         │
│     T1: Read(x=42) → Write(x=43) ───────────▶ COMMIT               │
│     T2: Read(x=42) ──────────────▶ Write(x=43) → COMMIT            │
│     결과: x=43 (44가 되어야 함, 하나의 증가 손실)                    │
│                                                                     │
│  5. Write Skew: 읽기 결과 기반 쓰기 충돌                             │
│     T1: Read(on_call=2) → 내가 퇴근해도 됨 → off_call(me)           │
│     T2: Read(on_call=2) → 나도 퇴근해도 됨 → off_call(me)           │
│     결과: on_call=0 (최소 1명 필요한데 0명!)                         │
│                                                                     │
│  6. Phantom: 검색 조건에 맞는 새 행 추가                             │
│     T1: SELECT COUNT(*) WHERE room=123 → 0, INSERT                 │
│     T2: SELECT COUNT(*) WHERE room=123 → 0, INSERT                 │
│     결과: 같은 방에 두 개의 예약!                                    │
└─────────────────────────────────────────────────────────────────────┘
```

### 5. 격리 수준 (Isolation Levels)

#### 5.1 격리 수준별 보호 범위

| 격리 수준 | Dirty Read | Read Skew | Phantom | Lost Update | Write Skew |
|-----------|------------|-----------|---------|-------------|------------|
| Read Uncommitted | ✗ 가능 | ✗ 가능 | ✗ 가능 | ✗ 가능 | ✗ 가능 |
| **Read Committed** | ✅ 방지 | ✗ 가능 | ✗ 가능 | ✗ 가능 | ✗ 가능 |
| **Snapshot Isolation** | ✅ 방지 | ✅ 방지 | ✅ 방지 | ? | ✗ 가능 |
| **Serializable** | ✅ 방지 | ✅ 방지 | ✅ 방지 | ✅ 방지 | ✅ 방지 |

#### 5.2 데이터베이스별 용어

| 데이터베이스 | 용어 | 실제 의미 |
|--------------|------|----------|
| PostgreSQL | Repeatable Read | Snapshot Isolation |
| MySQL/InnoDB | Repeatable Read | 약한 MVCC (SI보다 약함) |
| Oracle | Serializable | Snapshot Isolation (진짜 Serializable 아님!) |
| SQL Server | Snapshot | Snapshot Isolation |

#### 5.3 PostgreSQL 격리 수준

| 현상 | Read Committed | Repeatable Read | Serializable |
|-----|----------------|-----------------|--------------|
| Dirty Read | ❌ 방지 | ❌ 방지 | ❌ 방지 |
| Non-repeatable Read | ⚠️ 가능 | ❌ 방지 | ❌ 방지 |
| Phantom Read | ⚠️ 가능 | ⚠️ 가능* | ❌ 방지 |

**기본값**: Read Committed (Oracle, PostgreSQL, SQL Server)

### 6. Lost Update 방지

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Lost Update 방지 기법                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   1. Atomic Operations (원자적 연산)                                │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ UPDATE counters SET value = value + 1 WHERE key = 'foo';│       │
│   │ → Read-Modify-Write를 DB 내부에서 원자적으로 수행       │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   2. Explicit Locking (명시적 잠금)                                 │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ SELECT * FROM figures WHERE id = 1234 FOR UPDATE;       │       │
│   │ -- 비즈니스 로직 수행                                    │       │
│   │ UPDATE figures SET position = 'c4' WHERE id = 1234;     │       │
│   │ COMMIT;                                                  │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   3. Automatic Detection (자동 감지)                                │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ PostgreSQL, Oracle, SQL Server: Lost Update 자동 감지   │       │
│   │ → 충돌 시 트랜잭션 중단, 재시도 필요                     │       │
│   │ ⚠️ MySQL/InnoDB: 자동 감지 없음!                        │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   4. Compare-and-Set (조건부 쓰기)                                  │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ UPDATE wiki SET content = 'new'                         │       │
│   │ WHERE id = 1234 AND content = 'old';                    │       │
│   │ -- 0 rows affected → 다른 누군가가 수정함, 재시도       │       │
│   └─────────────────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────────────────┘
```

### 7. Write Skew 방지

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Write Skew 방지 기법                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   1. SELECT ... FOR UPDATE (명시적 잠금)                            │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ BEGIN TRANSACTION;                                      │       │
│   │ SELECT * FROM doctors                                   │       │
│   │   WHERE on_call = true AND shift_id = 1234              │       │
│   │   FOR UPDATE;  -- 조회된 모든 행 잠금                   │       │
│   │                                                         │       │
│   │ UPDATE doctors SET on_call = false                      │       │
│   │   WHERE name = 'Aaliyah' AND shift_id = 1234;           │       │
│   │ COMMIT;                                                 │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   2. Materializing Conflicts (충돌 구체화)                          │
│   → 존재하지 않는 row에 대한 잠금 불가 문제 해결                     │
│   → 데이터 모델에 잠금 객체 인위적 추가                              │
│   ⚠️ 오류 발생 쉬움, 최후의 수단                                    │
│                                                                     │
│   3. Serializable Isolation (직렬화 가능 격리) - 가장 권장           │
└─────────────────────────────────────────────────────────────────────┘
```

### 8. Serializable 격리 수준 구현

#### 8.1 Actual Serial Execution (VoltDB, Redis)

```
모든 트랜잭션을 단일 스레드에서 순차 실행

장점: 구현 단순, 동시성 문제 없음
단점: 단일 코어 제한, 저장 프로시저 필수
적합: 인메모리 DB, 짧은 트랜잭션
```

#### 8.2 Two-Phase Locking (MySQL, SQL Server)

```
규칙:
- 읽기: Shared Lock 획득 (다른 읽기 허용, 쓰기 차단)
- 쓰기: Exclusive Lock 획득 (모든 접근 차단)
- 트랜잭션 끝까지 잠금 유지

단점: Writers block Readers, Deadlock 가능
```

#### 8.3 Serializable Snapshot Isolation - SSI (PostgreSQL)

```
Optimistic Concurrency Control:
"일단 진행하고, 커밋 시 충돌 검사"

Snapshot Isolation + 직렬화 충돌 감지

장점:
- Readers never block Writers (Snapshot Isolation 유지)
- 2PL보다 훨씬 나은 성능
- 분산 환경에서도 확장 가능

사용: PostgreSQL (Serializable), CockroachDB, FoundationDB
```

### 9. 분산 트랜잭션: Two-Phase Commit (2PC)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Two-Phase Commit (2PC)                           │
├─────────────────────────────────────────────────────────────────────┤
│   ⚠️ 2PL (Two-Phase Locking)과 다름!                                │
│                                                                     │
│   Coordinator            Participant A        Participant B         │
│       │                       │                     │               │
│       │──── Prepare ─────────▶│                     │               │
│       │──── Prepare ──────────────────────────────▶│               │
│       │◀─── Yes ──────────────│                     │               │
│       │◀─── Yes ─────────────────────────────────│               │
│       │   [Write decision to log: COMMIT]           │               │
│       │   ← Commit Point                            │               │
│       │──── Commit ──────────▶│                     │               │
│       │──── Commit ─────────────────────────────▶│               │
│                                                                     │
│   Phase 1 (Prepare): "커밋할 수 있어?"                               │
│   Phase 2 (Commit/Abort): "커밋해!" 또는 "중단해!"                   │
└─────────────────────────────────────────────────────────────────────┘
```

#### Coordinator 장애 문제

- **In-Doubt 상태**: Participant가 커밋/롤백 여부 모름
- **잠금 유지**: 다른 트랜잭션 차단
- **해결**: Coordinator 복구 또는 관리자 수동 개입

---

### 10. XA 트랜잭션의 문제점

1. **Coordinator가 Single Point of Failure**
2. **In-Doubt 동안 잠금 유지** → 애플리케이션 가용성 저하
3. **Lowest Common Denominator** → 시스템별 최적화 불가
4. **Orphaned In-Doubt 트랜잭션** → 수동 개입 필요

**대안**: 단일 데이터베이스 내부 분산 트랜잭션 (CockroachDB, TiDB, Spanner)

## 💡 실무 적용

### 격리 수준 선택 가이드

| 상황 | 권장 격리 수준 |
|------|----------------|
| 대부분의 OLTP | Read Committed (기본값) |
| 백업, 분석 쿼리 | Snapshot Isolation |
| 금융, 예약 시스템 | Serializable |
| 높은 동시성 + 정확성 | SSI (PostgreSQL Serializable) |

### Serializable 구현 비교

| 기법 | 처리량 | 지연 시간 | 확장성 | 사용 사례 |
|------|--------|-----------|--------|-----------|
| Serial Execution | 낮음 | 낮음 | 샤딩 필요 | VoltDB, Redis |
| 2PL | 중간 | 높음 | 제한적 | MySQL, SQL Server |
| SSI | 높음 | 낮음 | 좋음 | PostgreSQL, CockroachDB |

## ✅ 체크리스트

### ACID 기본 개념
- [ ] Atomicity가 동시성이 아닌 장애 복구와 관련됨을 이해한다
- [ ] ACID의 C가 애플리케이션 책임임을 안다
- [ ] "Consistency"의 5가지 다른 의미를 구분할 수 있다

### 격리 수준
- [ ] Dirty Read, Read Skew, Lost Update, Write Skew를 구분할 수 있다
- [ ] MVCC의 동작 원리와 가시성 규칙을 설명할 수 있다
- [ ] SELECT FOR UPDATE의 용도를 설명할 수 있다

### Serializable 구현
- [ ] Serial Execution, 2PL, SSI의 차이를 설명할 수 있다
- [ ] PostgreSQL의 SSI가 어떻게 동작하는지 이해한다

### 분산 트랜잭션
- [ ] 2PC의 두 Phase를 설명할 수 있다
- [ ] Coordinator 장애 시 In-Doubt 문제를 이해한다
- [ ] XA 트랜잭션의 문제점을 열거할 수 있다
