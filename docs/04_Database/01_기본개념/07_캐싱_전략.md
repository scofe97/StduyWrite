# 캐싱 전략 (Caching Strategies)

---

## 📌 핵심 요약

데이터베이스 부하를 줄이고 응답 속도를 개선하기 위한 캐싱 전략을 다룬다. Cache-Aside, Write-Through, Write-Behind 패턴과 Redis를 활용한 구현 방법을 학습한다.

---

- [ ] 캐시 무효화 전략을 이해할 수 있다
- [ ] Redis를 활용한 캐싱을 구현할 수 있다
- [ ] 캐시 관련 문제(Cache Stampede, Hot Key)를 해결할 수 있다

### 1. 캐싱이 필요한 이유

```
┌─────────────────────────────────────────────────────────────────────┐
│                    레이턴시 비교                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   작업                           지연 시간                          │
│   ────────────────────────────────────────────                     │
│   L1 캐시 참조                   ~1ns                               │
│   L2 캐시 참조                   ~4ns                               │
│   메인 메모리 참조               ~100ns                             │
│   Redis 조회                     ~100-500μs                         │
│   SSD 랜덤 읽기                  ~150μs                             │
│   PostgreSQL 단순 쿼리           ~1-10ms                            │

│   네트워크 왕복 (같은 DC)        ~500μs-1ms                         │
│                                                                     │
│   → Redis는 DB보다 10-100배 빠름                                    │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. 캐싱 패턴

#### 2.1 Cache-Aside (Lazy Loading)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Cache-Aside 패턴                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   읽기:                                                              │
│   1. 캐시 확인 → Hit → 반환                                          │
│                   │                                                 │
│                  Miss                                               │
│                   ↓                                                 │
│   2. DB 조회                                                         │
│   3. 캐시에 저장                                                     │
│   4. 반환                                                            │
│                                                                     │
│   쓰기:                                                              │
│   1. DB에 쓰기                                                       │
│   2. 캐시 무효화 (또는 업데이트)                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**장점**: 필요한 데이터만 캐싱, 구현 단순
**단점**: Cache Miss 시 지연, 캐시/DB 불일치 가능

```python
def get_user(user_id):
    # 1. 캐시 확인
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)

    # 2. DB 조회
    user = db.query("SELECT * FROM users WHERE id = %s", user_id)

    # 3. 캐시에 저장 (TTL 1시간)
    redis.setex(f"user:{user_id}", 3600, json.dumps(user))

    return user

def update_user(user_id, data):
    # 1. DB 업데이트
    db.execute("UPDATE users SET ... WHERE id = %s", user_id)

    # 2. 캐시 무효화
    redis.delete(f"user:{user_id}")
```

#### 2.2 Write-Through

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Write-Through 패턴                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   쓰기:                                                              │
│   1. 캐시에 쓰기                                                     │
│   2. 캐시가 DB에 동기적으로 쓰기                                     │
│   3. 완료 응답                                                       │
│                                                                     │
│   App → Cache → DB (동기)                                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**장점**: 캐시/DB 항상 일관성 유지
**단점**: 쓰기 지연 증가, 읽히지 않는 데이터도 캐싱

#### 2.3 Write-Behind (Write-Back)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Write-Behind 패턴                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   쓰기:                                                              │
│   1. 캐시에 쓰기                                                     │
│   2. 즉시 응답                                                       │
│   3. 백그라운드에서 비동기적으로 DB에 쓰기                           │
│                                                                     │
│   App → Cache → (async) → DB                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**장점**: 매우 빠른 쓰기 응답, 쓰기 배치 가능
**단점**: 복잡한 구현, 캐시 장애 시 데이터 손실 위험

### 3. 캐시 무효화 전략

#### 3.1 TTL (Time-To-Live)

```python
# 시간 기반 만료
redis.setex("user:123", 3600, user_data)  # 1시간 후 만료
redis.setex("session:abc", 86400, session_data)  # 24시간 후 만료

# 캐시 데이터 특성에 따른 TTL 설정
CACHE_TTL = {
    "user_profile": 3600,      # 1시간 (자주 변경되지 않음)
    "product_list": 300,       # 5분 (중간 빈도 변경)
    "stock_price": 5,          # 5초 (실시간 데이터)
    "session": 86400,          # 24시간
}
```

#### 3.2 Event-Driven Invalidation

```python
# 데이터 변경 시 캐시 무효화
def update_product(product_id, data):
    db.execute("UPDATE products SET ... WHERE id = %s", product_id)

    # 관련 캐시 모두 무효화
    redis.delete(f"product:{product_id}")
    redis.delete(f"product_list:category:{data['category']}")
    redis.delete("product_list:all")

# Pub/Sub로 분산 캐시 무효화
def publish_invalidation(key):
    redis.publish("cache_invalidation", key)

# 다른 서버에서 구독
def on_invalidation(message):
    local_cache.delete(message)
```

#### 3.3 Cache Tagging

```python
# 태그 기반 무효화 (관련 캐시 그룹 삭제)
def cache_with_tags(key, value, tags, ttl=3600):
    redis.setex(key, ttl, value)
    for tag in tags:
        redis.sadd(f"tag:{tag}", key)

def invalidate_by_tag(tag):
    keys = redis.smembers(f"tag:{tag}")
    if keys:
        redis.delete(*keys)
    redis.delete(f"tag:{tag}")

# 사용 예시
cache_with_tags("product:123", data, ["category:electronics", "brand:apple"])
invalidate_by_tag("category:electronics")  # 모든 electronics 제품 캐시 삭제
```

### 4. 캐시 관련 문제와 해결

#### 4.1 Cache Stampede (Thundering Herd)

```
문제: 캐시 만료 시 동시에 많은 요청이 DB로 몰림

┌─────────────────────────────────────────────────────────────────────┐
│   캐시 만료 시점                                                     │
│                                                                     │
│   Request 1 ──→ Cache Miss ──→ DB Query ──────┐                    │
│   Request 2 ──→ Cache Miss ──→ DB Query ──────│                    │
│   Request 3 ──→ Cache Miss ──→ DB Query ──────│── DB 과부하!       │
│   Request 4 ──→ Cache Miss ──→ DB Query ──────│                    │
│   ...                                                               │
└─────────────────────────────────────────────────────────────────────┘
```

**해결책 1: 락 사용**

```python
def get_with_lock(key, fetch_func, ttl=3600):
    # 캐시 확인
    value = redis.get(key)
    if value:
        return json.loads(value)

    # 락 획득 시도
    lock_key = f"lock:{key}"
    if redis.setnx(lock_key, 1):
        redis.expire(lock_key, 30)  # 락 TTL
        try:
            # DB 조회 후 캐시 저장
            value = fetch_func()
            redis.setex(key, ttl, json.dumps(value))
            return value
        finally:
            redis.delete(lock_key)
    else:
        # 락 실패 시 대기 후 재시도
        time.sleep(0.1)
        return get_with_lock(key, fetch_func, ttl)
```

**해결책 2: Probabilistic Early Expiration**

```python
def get_with_early_expiration(key, fetch_func, ttl=3600, beta=1):
    value, expiry = redis.get_with_ttl(key)
    if value:
        remaining = expiry - time.time()
        # 확률적으로 미리 갱신 (만료에 가까울수록 확률 증가)
        if remaining > 0:
            random_val = random.random()
            threshold = math.exp(-beta * remaining / ttl)
            if random_val < threshold:
                # 백그라운드에서 갱신
                threading.Thread(target=refresh_cache, args=(key, fetch_func, ttl)).start()
        return json.loads(value)

    return fetch_and_cache(key, fetch_func, ttl)
```

#### 4.2 Hot Key

```
문제: 특정 키에 요청이 집중되어 Redis 단일 노드 병목

해결책:
1. 복제: hot_key:1, hot_key:2, hot_key:3 (랜덤 선택)
2. 로컬 캐시: 애플리케이션 메모리에 추가 캐싱
3. Redis Cluster: 키별 샤딩
```

```python
# Hot Key 복제
def get_hot_key(key, replicas=3):
    replica_key = f"{key}:{random.randint(1, replicas)}"
    return redis.get(replica_key)

def set_hot_key(key, value, replicas=3, ttl=3600):
    for i in range(1, replicas + 1):
        redis.setex(f"{key}:{i}", ttl, value)
```

#### 4.3 Cache Penetration

```
문제: 존재하지 않는 데이터에 대한 반복 요청 → DB 계속 조회

해결책:
1. Null 캐싱: 없는 데이터도 캐싱 (짧은 TTL)
2. Bloom Filter: 존재 여부 사전 확인
```

```python
# Null 캐싱
def get_with_null_cache(key, fetch_func):
    value = redis.get(key)

    if value == "NULL":
        return None  # 캐시된 null
    if value:
        return json.loads(value)

    # DB 조회
    result = fetch_func()
    if result is None:
        redis.setex(key, 60, "NULL")  # 1분간 null 캐싱
    else:
        redis.setex(key, 3600, json.dumps(result))

    return result
```

### 5. Redis 캐싱 구현

#### 5.1 기본 패턴

```python
import redis
import json

redis_client = redis.Redis(host='localhost', port=6379, db=0)

class CacheService:
    def __init__(self, redis_client, default_ttl=3600):
        self.redis = redis_client
        self.default_ttl = default_ttl

    def get(self, key):
        value = self.redis.get(key)
        return json.loads(value) if value else None

    def set(self, key, value, ttl=None):
        self.redis.setex(key, ttl or self.default_ttl, json.dumps(value))

    def delete(self, key):
        self.redis.delete(key)

    def delete_pattern(self, pattern):
        """패턴 매칭으로 삭제 (주의: 프로덕션에서 KEYS 사용 지양)"""
        for key in self.redis.scan_iter(match=pattern):
            self.redis.delete(key)
```

#### 5.2 데코레이터 패턴

```python
import functools
import hashlib

def cached(ttl=3600, key_prefix=""):
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            # 캐시 키 생성
            key_parts = [key_prefix, func.__name__] + [str(a) for a in args]
            key_parts += [f"{k}={v}" for k, v in sorted(kwargs.items())]
            cache_key = ":".join(key_parts)

            # 캐시 확인
            cached_value = redis_client.get(cache_key)
            if cached_value:
                return json.loads(cached_value)

            # 함수 실행 및 캐싱
            result = func(*args, **kwargs)
            redis_client.setex(cache_key, ttl, json.dumps(result))
            return result
        return wrapper
    return decorator

# 사용 예시
@cached(ttl=3600, key_prefix="user")
def get_user_profile(user_id):
    return db.query("SELECT * FROM users WHERE id = %s", user_id)
```

### 6. PostgreSQL 쿼리 캐싱

#### 6.1 Materialized View

```sql
-- 복잡한 집계 쿼리 결과 캐싱
CREATE MATERIALIZED VIEW monthly_sales_summary AS
SELECT
    date_trunc('month', order_date) AS month,
    category,
    SUM(amount) AS total_sales,
    COUNT(*) AS order_count
FROM orders
JOIN products ON orders.product_id = products.id
GROUP BY month, category;

-- 인덱스 생성
CREATE INDEX ON monthly_sales_summary (month, category);

-- 조회 (캐시된 데이터)
SELECT * FROM monthly_sales_summary WHERE month = '2024-01-01';

-- 갱신 (스케줄러로 주기적 실행)
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_sales_summary;
```

#### 6.2 Prepared Statements

```python
# 쿼리 계획 캐싱
cursor.execute("PREPARE get_user AS SELECT * FROM users WHERE id = $1")

# 준비된 문장 실행 (계획 재사용)
```

## 💡 실무 적용

### 캐싱 전략 선택

| 상황 | 권장 패턴 |
|------|----------|
| 읽기 위주, 간단한 구현 | Cache-Aside |
| 항상 최신 데이터 필요 | Write-Through |
| 높은 쓰기 처리량 필요 | Write-Behind |
| 복잡한 집계 쿼리 | Materialized View |

### 캐시 계층

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Multi-Level Caching                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   L1: 로컬 캐시 (애플리케이션 메모리)                                │
│       → 가장 빠름, 용량 작음, 서버별 분리                           │
│                                                                     │
│   L2: 분산 캐시 (Redis)                                             │
│       → 빠름, 용량 중간, 서버 간 공유                               │
│                                                                     │
│   L3: 데이터베이스                                                   │
│       → 느림, 용량 큼, 영구 저장                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```
