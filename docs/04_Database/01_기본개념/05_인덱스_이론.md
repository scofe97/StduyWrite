# 인덱스 이론 (Index Theory)

---

## 📌 핵심 요약

데이터베이스 인덱스는 테이블 데이터에 대한 빠른 접근을 제공하는 자료구조다. B-tree, LSM-tree, Hash Index 등 다양한 인덱스 유형의 내부 구조와 트레이드오프를 이해한다.

---

## 🎯 학습 목표

- [ ] B-tree 인덱스의 구조와 동작 원리를 설명할 수 있다
- [ ] LSM-tree의 쓰기 최적화 원리를 이해할 수 있다
- [ ] Hash 인덱스의 특성과 제한사항을 알 수 있다
- [ ] 인덱스 유형별 적합한 워크로드를 판단할 수 있다

## 📖 본문

### 1. 인덱스의 기본 개념

```
┌─────────────────────────────────────────────────────────────────────┐
│                    인덱스의 역할                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Without Index:                                                    │
│   ┌─────────────────────────────────────────────────────────┐       │
│   │ Table Scan: O(N) - 모든 행 검사                          │       │
│   └─────────────────────────────────────────────────────────┘       │
│                                                                     │
│   With Index:                                                       │
│   ┌─────────────────────────────────────────────────────────┐       │

│                                                                     │
│   트레이드오프: 읽기 속도 ↑ / 쓰기 속도 ↓ / 저장 공간 ↑            │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. B-tree 인덱스

#### 2.1 구조

```
                         [50]                      ← Root (Level 0)
                        /    \
                  [20, 30]    [70, 80]             ← Branch (Level 1)
                  /  |  \      /  |  \
               [10] [25] [35] [60] [75] [90]       ← Leaf (Level 2)
                ↓    ↓    ↓    ↓    ↓    ↓
              Data  Data Data Data Data Data      ← Table Rows
```

#### 2.2 특징

| 특성 | 설명 |
|------|------|
| **균형 트리** | 모든 리프 노드가 같은 깊이 |
| **범위 검색** | 정렬된 순서로 저장, 범위 쿼리 효율적 |
| **시간 복잡도** | 검색/삽입/삭제 모두 O(log N) |
| **디스크 최적화** | 노드 크기 = 페이지 크기 (4KB, 8KB) |

#### 2.3 B-tree vs B+tree

| 구분 | B-tree | B+tree |
|------|--------|--------|
| 데이터 위치 | 모든 노드 | 리프 노드만 |
| 리프 노드 연결 | 없음 | Linked List |
| 범위 쿼리 | 느림 | 빠름 |
| 사용 DB | 일부 | PostgreSQL, MySQL 등 대부분 |

#### 2.4 B+tree 연산

```sql
-- 점 조회: O(log N)
SELECT * FROM users WHERE id = 100;

-- 범위 조회: O(log N + M), M = 결과 행 수
SELECT * FROM users WHERE id BETWEEN 100 AND 200;

-- 삽입: O(log N), 노드 분할 가능
INSERT INTO users VALUES (150, 'Alice');

-- 삭제: O(log N), 노드 병합 가능
DELETE FROM users WHERE id = 100;
```

### 3. LSM-tree (Log-Structured Merge-tree)

#### 3.1 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LSM-tree 구조                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Write Path:                                                       │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐            │
│   │  MemTable   │ → │  SSTable    │ → │  SSTable    │ → ...       │
│   │  (Memory)   │    │  Level 0   │    │  Level 1   │              │
│   └─────────────┘    └─────────────┘    └─────────────┘            │
│         ↓                  ↓                  ↓                     │
│      순차 쓰기          Compaction         Compaction               │
│                                                                     │
│   Read Path:                                                        │
│   Query → MemTable → Level 0 → Level 1 → ... → Level N             │
│           (최신 데이터 우선 검색)                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### 3.2 특징

| 특성 | 설명 |
|------|------|
| **쓰기 최적화** | 순차 쓰기로 랜덤 I/O 최소화 |
| **Write Amplification** | Compaction으로 같은 데이터 여러 번 쓰기 |
| **Read Amplification** | 여러 레벨 검색 필요 |
| **Space Amplification** | 임시 중복 데이터 존재 |

#### 3.3 사용 사례

| 데이터베이스 | 설명 |
|--------------|------|
| RocksDB | Facebook, 임베디드 스토리지 |
| LevelDB | Google, 경량 key-value |
| Cassandra | 분산 NoSQL |
| InfluxDB | 시계열 데이터 |

### 4. Hash 인덱스

#### 4.1 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Hash Index                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Key → hash(key) → Bucket → Value                                 │
│                                                                     │
│   hash("alice") = 3  →  Bucket[3] → Row Pointer                    │
│   hash("bob")   = 7  →  Bucket[7] → Row Pointer                    │
│   hash("carol") = 3  →  Bucket[3] → Row Pointer (충돌)             │
│                                                                     │
│   시간 복잡도: O(1) 평균, O(N) 최악 (해시 충돌)                      │
└─────────────────────────────────────────────────────────────────────┘
```

#### 4.2 특징

| 특성 | 설명 |
|------|------|
| **점 조회** | O(1), 매우 빠름 |
| **범위 조회** | 불가능 |
| **정렬** | 불가능 |
| **메모리 기반** | 대부분 인메모리에서만 사용 |

#### 4.3 PostgreSQL Hash Index

```sql
-- Hash 인덱스 생성
CREATE INDEX idx_hash ON users USING hash (email);

-- 효율적: 등가 비교
SELECT * FROM users WHERE email = 'alice@example.com';

-- 비효율적: 범위 조회 (Hash 인덱스 사용 불가)
SELECT * FROM users WHERE email > 'a%';
```

### 5. 인덱스 유형 비교

| 인덱스 | 점 조회 | 범위 조회 | 쓰기 성능 | 적합한 워크로드 |
|--------|---------|-----------|-----------|-----------------|
| B-tree | O(log N) | O(log N + M) | 중간 | 범용 (OLTP) |
| LSM-tree | O(log N) | O(log N + M) | 높음 | 쓰기 집약 |
| Hash | O(1) | 불가 | 높음 | Key-Value |
| Bitmap | - | - | 낮음 | OLAP, 낮은 카디널리티 |

### 6. 복합 인덱스 (Composite Index)

#### 6.1 컬럼 순서의 중요성

```sql
-- 복합 인덱스 생성
CREATE INDEX idx_composite ON orders (customer_id, order_date);

-- 인덱스 사용 가능 (선두 컬럼 포함)
SELECT * FROM orders WHERE customer_id = 100;
SELECT * FROM orders WHERE customer_id = 100 AND order_date > '2024-01-01';

-- 인덱스 사용 불가 (선두 컬럼 없음)
SELECT * FROM orders WHERE order_date > '2024-01-01';
```

#### 6.2 Covering Index

```sql
-- 모든 필요 컬럼이 인덱스에 포함되면 테이블 접근 불필요
CREATE INDEX idx_covering ON orders (customer_id, order_date, total_amount);

-- Index-Only Scan 가능
SELECT order_date, total_amount
FROM orders
WHERE customer_id = 100;
```

### 7. 특수 인덱스

#### 7.1 GIN (Generalized Inverted Index)

```sql
-- 배열, JSON, 전문 검색용
CREATE INDEX idx_gin ON articles USING gin (to_tsvector('english', content));
CREATE INDEX idx_gin_array ON products USING gin (tags);
```

#### 7.2 GiST (Generalized Search Tree)

```sql
-- 공간 데이터, 범위 타입용
CREATE INDEX idx_gist ON locations USING gist (geom);
```

#### 7.3 BRIN (Block Range Index)

```sql
-- 물리적으로 정렬된 대용량 테이블용 (시계열 등)
CREATE INDEX idx_brin ON logs USING brin (created_at);
-- 매우 작은 인덱스 크기, 정렬된 데이터에 효율적
```

## 💡 실무 적용

### 인덱스 선택 가이드

| 상황 | 권장 인덱스 |
|------|------------|
| 범용 OLTP | B-tree |
| 등가 조회만 | Hash |
| 쓰기 집약 | LSM-tree 기반 DB |
| 전문 검색 | GIN |
| 공간 데이터 | GiST |
| 시계열/로그 | BRIN |

### PostgreSQL 인덱스 확인

```sql
-- 테이블의 인덱스 확인
SELECT * FROM pg_indexes WHERE tablename = 'users';

-- 인덱스 사용 여부 확인
EXPLAIN ANALYZE SELECT * FROM users WHERE id = 100;
```
