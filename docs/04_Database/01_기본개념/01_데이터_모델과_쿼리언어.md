## 📖 본문 정리

### 1. 데이터 모델의 계층화 (Data Model Layering)

대부분의 애플리케이션은 여러 계층의 데이터 모델을 쌓아 올려 구축된다:

```
┌─────────────────────────────────────────────────────────────┐
│  Application Layer                                          │
│  현실 세계 → 객체/데이터 구조/API로 모델링                    │
├─────────────────────────────────────────────────────────────┤
│  Data Model Layer                                           │
│  객체 → JSON/XML/관계형 테이블/그래프로 표현                  │
├─────────────────────────────────────────────────────────────┤
│  Storage Engine Layer                                       │
│  데이터 → 메모리/디스크/네트워크 상의 바이트로 표현           │
├─────────────────────────────────────────────────────────────┤
│  Hardware Layer                                             │
│  바이트 → 전류, 빛, 자기장 등으로 표현                       │
└─────────────────────────────────────────────────────────────┘
```

각 계층은 **아래 계층의 복잡성을 숨기고** 깔끔한 데이터 모델을 제공한다. 이러한 추상화 덕분에 데이터베이스 벤더 엔지니어와 애플리케이션 개발자가 효과적으로 협업할 수 있다.

> 💬 **비유**: 데이터 모델 계층화는 **레고 블록**과 같다. 가장 아래에는 작은 기본 블록(바이트)이 있고, 그 위에 더 큰 조립 블록(테이블, 문서)을 쌓고, 최상위에 완성된 구조물(애플리케이션)을 만든다. 각 층은 아래 층의 세부 사항을 알 필요 없이 작업할 수 있다.

### 2. 선언적 쿼리 언어 (Declarative Query Languages)

| 특성 | 선언적 (Declarative) | 명령적 (Imperative) |
|------|---------------------|---------------------|
| 접근 방식 | **무엇**을 원하는지 명시 | **어떻게** 달성할지 명시 |
| 예시 | SQL, Cypher, SPARQL | Python, Java 알고리즘 |
| 장점 | 간결, 최적화 자동화 | 세밀한 제어 |
| 병렬화 | DB가 자동 처리 | 직접 구현 필요 |

**선언적 쿼리의 장점**:
- 쿼리 최적화기(Query Optimizer)가 인덱스와 조인 알고리즘을 자동 결정
- 쿼리 변경 없이 성능 개선 가능
- 병렬 실행을 DB가 알아서 처리

```sql
-- 선언적: 무엇을 원하는지만 명시
SELECT * FROM users WHERE age > 25 ORDER BY name;

-- 명령적이었다면: 어떻게 찾을지 직접 명시해야 함
-- for each row in users:
--   if row.age > 25: add to results
-- sort results by name
```

### 3. 관계형 모델 vs 문서 모델

#### 3.1 Object-Relational Mismatch (임피던스 불일치)

객체지향 언어로 개발할 때, 객체와 관계형 테이블 사이에 **변환 계층**이 필요하다.

```
┌─────────────────┐     임피던스 불일치     ┌─────────────────┐
│   Application   │ ←──────────────────→ │    Database     │
│   (Objects)     │        ORM           │   (Tables)      │
│                 │                      │                 │
│  user.positions │ ←─ 변환 필요 ─→      │  users 테이블   │
│  user.education │                      │  positions 테이블│
└─────────────────┘                      └─────────────────┘
```

> 💬 **비유**: 임피던스 불일치는 **언어 장벽**과 같다. 한국어(객체)를 영어(테이블)로 통역해야 하는데, 완벽한 번역은 어렵고 항상 뉘앙스가 손실된다.

**ORM의 장단점**:

| 장점 | 단점 |
|------|------|
| 보일러플레이트 코드 감소 | 두 모델을 모두 이해해야 함 |
| 쿼리 결과 캐싱 지원 | N+1 쿼리 문제 발생 가능 |
| 스키마 마이그레이션 도움 | 비효율적 스키마/쿼리 생성 가능 |

**N+1 쿼리 문제 예시**:
```
1. 댓글 N개 조회 (1 쿼리)
2. 각 댓글의 작성자 조회 (N 쿼리)
→ 총 N+1 쿼리 발생 (JOIN으로 1쿼리로 해결 가능)
```

#### 3.2 문서 모델의 장점: One-to-Many 관계

이력서(LinkedIn 프로필) 예시:

**관계형 모델** (Figure 3-1):
```
users 테이블 ──┬── positions 테이블 (FK: user_id)
               ├── education 테이블 (FK: user_id)
               └── contact_info 테이블 (FK: user_id)
```

**문서 모델** (JSON):
```json
{
  "user_id": 251,
  "first_name": "Barack",
  "last_name": "Obama",
  "positions": [
    {"job_title": "President", "organization": "United States of America"},
    {"job_title": "US Senator (D-IL)", "organization": "United States Senate"}
  ],
  "education": [
    {"school_name": "Harvard University", "start": 1988, "end": 1991}
  ],
  "contact_info": {
    "website": "https://barackobama.com"
  }
}
```

**문서 모델의 장점**:
- **데이터 지역성 (Locality)**: 모든 관련 데이터가 한 곳에 → 쿼리 단순화, 성능 향상
- **트리 구조 명시적 표현**: one-to-many 관계가 자연스럽게 표현됨
- **스키마 유연성**: schema-on-read 방식

> 💬 **비유**: 문서 모델은 **서류철**과 같다. 한 사람에 대한 모든 서류(이력서, 학력증명서, 경력증명서)를 한 폴더에 모아두면 찾기 쉽다. 관계형은 서류 종류별로 다른 캐비닛에 보관하는 것과 같다.

### 4. 정규화 vs 비정규화

#### 4.1 핵심 개념

| 특성 | 정규화 (Normalization) | 비정규화 (Denormalization) |
|------|----------------------|--------------------------|
| 저장 방식 | ID 참조 사용 | 실제 값 복사 |
| 쓰기 성능 | 빠름 (한 곳만 수정) | 느림 (여러 곳 수정) |
| 읽기 성능 | 느림 (JOIN 필요) | 빠름 (JOIN 불필요) |
| 저장 공간 | 적음 | 많음 |
| 일관성 | 자동 유지 | 수동 유지 필요 |

```sql
-- 정규화: region_id로 참조
SELECT users.*, regions.region_name
FROM users
JOIN regions ON users.region_id = regions.id
WHERE users.id = 251;

-- 비정규화: region_name 직접 저장
-- JOIN 불필요, 하지만 지역명 변경 시 모든 곳 수정 필요
```

#### 4.2 소셜 네트워크 타임라인 사례

X(Twitter)의 Materialized Timeline 구현:

```
┌──────────────────────────────────────────────────────────────┐
│  Timeline Entry (비정규화)                                    │
├──────────────────────────────────────────────────────────────┤
│  • post_id (저장) ← 변경 빈번한 데이터는 ID만 저장           │
│  • sender_id (저장)                                          │
│  • 실제 내용, 좋아요 수, 프로필 사진 → 읽을 때 조회 (Hydration)│
└──────────────────────────────────────────────────────────────┘
```

**Hydration**: ID를 사람이 읽을 수 있는 정보로 변환하는 과정 (application-level JOIN)

**트레이드오프 결정 요소**:
- 정보가 얼마나 자주 변경되는가?
- 읽기/쓰기 비용 (특히 outlier 고려)
- 구현 복잡도

> 💬 **비유**: 정규화는 **도서관 참조 시스템**과 같다. "ISBN: 978-0-321-12521-7"만 적어두면 책 제목이 바뀌어도 문제없다. 비정규화는 책 제목을 직접 적어두는 것 - 빨리 찾지만 제목 바뀌면 모든 곳을 고쳐야 한다.

### 5. Many-to-Many 관계

#### 5.1 관계형 모델: Associative Table (Join Table)

```
┌─────────┐     ┌─────────────┐     ┌──────────────┐
│  users  │────→│  positions  │←────│ organizations│
│         │     │ (join table)│     │              │
│ user_id │     │ user_id     │     │ org_id       │
│ name    │     │ org_id      │     │ name         │
└─────────┘     │ job_title   │     │ logo_url     │
                └─────────────┘     └──────────────┘
```

#### 5.2 문서 모델: ID 참조

```json
{
  "user_id": 251,
  "positions": [
    {"start": 2009, "end": 2017, "job_title": "President", "org_id": 513},
    {"start": 2005, "end": 2008, "job_title": "US Senator", "org_id": 514}
  ]
}
```

**양방향 쿼리 지원**:
- 특정 사람이 일한 모든 조직 찾기
- 특정 조직에서 일한 모든 사람 찾기
→ Secondary Index 필요

### 6. Star Schema와 Snowflake Schema (분석용)

데이터 웨어하우스에서 사용하는 스키마 구조:

```
                    ┌─────────────┐
                    │ dim_product │
                    └──────┬──────┘
                           │
┌─────────────┐     ┌──────┴──────┐     ┌─────────────┐
│  dim_store  │─────│ fact_sales  │─────│  dim_date   │
└─────────────┘     │ (중심 테이블)│     └─────────────┘
                    └──────┬──────┘
                           │
┌─────────────┐            │            ┌─────────────┐
│ dim_customer│────────────┴────────────│dim_promotion│
└─────────────┘                         └─────────────┘
```

| 스키마 | 특징 | 장점 |
|--------|------|------|
| **Star** | Dimension이 Fact에 직접 연결 | 쿼리 단순, 분석가 친화적 |
| **Snowflake** | Dimension이 sub-dimension으로 분해 | 더 정규화됨 |
| **OBT (One Big Table)** | Dimension을 Fact에 비정규화 | 저장 공간 ↑, 쿼리 속도 ↑ |

> 💬 **비유**: Star Schema는 **태양계**와 같다. 중심에 태양(Fact Table)이 있고, 주변에 행성들(Dimension Tables)이 빛살처럼 연결되어 있다.

### 7. 언제 어떤 모델을 사용할까?

#### 7.1 모델 선택 가이드

| 상황 | 추천 모델 | 이유 |
|------|----------|------|
| 트리 구조 (one-to-many) | 문서 모델 | 자연스러운 표현, 지역성 |
| 복잡한 many-to-many | 관계형 모델 | JOIN 지원, 정규화 용이 |
| 모든 것이 연결됨 | 그래프 모델 | 관계 탐색 최적화 |
| 이기종 데이터 | 문서 모델 | 스키마 유연성 |
| 분석/BI | 관계형 (Star/Snowflake) | SQL 생태계 |

#### 7.2 Schema-on-Read vs Schema-on-Write

```
Schema-on-Write (관계형)          Schema-on-Read (문서)
┌─────────────────────┐          ┌─────────────────────┐
│ 쓰기 시점에 검증     │          │ 읽기 시점에 해석     │
│ 컴파일 타임 타입체크 │   vs     │ 런타임 타입체크      │
│ 마이그레이션 필요    │          │ 애플리케이션 코드로  │
│                     │          │ 버전 핸들링          │
└─────────────────────┘          └─────────────────────┘
```

**Schema 변경 예시** (first_name 필드 추가):

```sql
-- Schema-on-Write: 마이그레이션 필요
ALTER TABLE users ADD COLUMN first_name text DEFAULT NULL;
UPDATE users SET first_name = split_part(name, ' ', 1);
```

```javascript
// Schema-on-Read: 읽을 때 처리
if (user && user.name && !user.first_name) {
    user.first_name = user.name.split(" ")[0];
}
```

#### 7.3 모델의 수렴 (Convergence)

현대 데이터베이스들은 서로의 장점을 흡수:
- 관계형 DB → JSON 지원 추가 (PostgreSQL jsonb)
- 문서 DB → JOIN, Secondary Index 지원 (MongoDB $lookup)
- 관계형-문서 하이브리드가 강력한 조합

### 8. 그래프 데이터 모델

#### 8.1 그래프가 적합한 경우

| 사용 사례 | Vertex (정점) | Edge (간선) |
|----------|--------------|-------------|
| 소셜 그래프 | 사람 | 친구 관계 |
| 웹 그래프 | 웹 페이지 | HTML 링크 |
| 도로/철도 | 교차로 | 도로/철로 |
| 지식 그래프 | 엔티티 (사람, 장소, 조직) | 관계 |

#### 8.2 Property Graph 모델

```
Vertex:                          Edge:
┌──────────────────────┐        ┌──────────────────────┐
│ unique_id            │        │ unique_id            │
│ label (타입)         │        │ tail_vertex (시작)   │
│ properties (K-V)     │        │ head_vertex (끝)     │
│ outgoing_edges       │        │ label (관계 타입)    │
│ incoming_edges       │        │ properties (K-V)     │
└──────────────────────┘        └──────────────────────┘
```

**SQL로 그래프 표현**:
```sql
CREATE TABLE vertices (
    vertex_id   integer PRIMARY KEY,
    label       text,
    properties  jsonb
);

CREATE TABLE edges (
    edge_id     integer PRIMARY KEY,
    tail_vertex integer REFERENCES vertices (vertex_id),
    head_vertex integer REFERENCES vertices (vertex_id),
    label       text,
    properties  jsonb
);

CREATE INDEX edges_tails ON edges (tail_vertex);
CREATE INDEX edges_heads ON edges (head_vertex);
```

#### 8.3 Cypher 쿼리 언어

```cypher
-- 데이터 생성
CREATE
  (usa      :Location {name:'United States',  type:'country'}),
  (idaho    :Location {name:'Idaho',          type:'state'}),
  (lucy     :Person   {name:'Lucy'}),
  (idaho) -[:WITHIN]-> (usa),
  (lucy)  -[:BORN_IN]-> (idaho)

-- 미국에서 태어나 유럽에 사는 사람 찾기
MATCH
  (person) -[:BORN_IN]->  () -[:WITHIN*0..]-> (:Location {name:'United States'}),
  (person) -[:LIVES_IN]-> () -[:WITHIN*0..]-> (:Location {name:'Europe'})
RETURN person.name
```

`-[:WITHIN*0..]->` : WITHIN 관계를 0번 이상 따라감 (재귀 탐색)

> 💬 **비유**: 그래프 쿼리는 **지하철 노선도 탐색**과 같다. "A역에서 B역까지 환승 몇 번이든 상관없이 갈 수 있는 경로"를 찾는 것처럼, 관계를 따라 원하는 목적지까지 탐색한다.

#### 8.4 SQL에서의 그래프 쿼리 (WITH RECURSIVE)

```sql
WITH RECURSIVE
  in_usa(vertex_id) AS (
      SELECT vertex_id FROM vertices
        WHERE label = 'Location' AND properties->>'name' = 'United States'
    UNION
      SELECT edges.tail_vertex FROM edges
        JOIN in_usa ON edges.head_vertex = in_usa.vertex_id
        WHERE edges.label = 'within'
  )
-- ... 31줄 (vs Cypher 4줄)
```

**비교**: Cypher 4줄 = SQL 31줄 → 적합한 모델/언어 선택이 중요!

#### 8.5 Triple-Store와 SPARQL

**Triple 형식**: (subject, predicate, object)

```turtle
# Turtle 형식
@prefix : <urn:example:>.
_:lucy     a       :Person.
_:lucy     :name   "Lucy".
_:lucy     :bornIn _:idaho.
_:idaho    a       :Location.
_:idaho    :name   "Idaho".
_:idaho    :within _:usa.
```

**SPARQL 쿼리**:
```sparql
PREFIX : <urn:example:>

SELECT ?personName WHERE {
  ?person :name ?personName.
  ?person :bornIn  / :within* / :name "United States".
  ?person :livesIn / :within* / :name "Europe".
}
```

#### 8.6 Datalog: 재귀적 관계형 쿼리

```prolog
% 사실 정의
location(1, "North America", "continent").
location(2, "United States", "country").
within(2, 1).  /* US is in North America */

% 재귀 규칙 정의
within_recursive(LocID, PlaceName) :- location(LocID, PlaceName, _).
within_recursive(LocID, PlaceName) :- within(LocID, ViaID),
                                      within_recursive(ViaID, PlaceName).

% 쿼리
us_to_europe(Person) :- migrated(Person, "United States", "Europe").
```

#### 8.7 GraphQL

**목적**: 클라이언트가 필요한 데이터만 정확히 요청

```graphql
query ChatApp {
  channels {
    name
    recentMessages(latest: 50) {
      timestamp
      content
      sender {
        fullName
        imageUrl
      }
      replyTo {
        content
        sender { fullName }
      }
    }
  }
}
```

**특징**:
- 클라이언트 주도 쿼리 → 서버 API 변경 없이 UI 요구사항 대응
- 재귀 쿼리 불가 (보안상 제한)
- OLTP 용도로 설계

### 9. Event Sourcing과 CQRS

#### 9.1 핵심 개념

```
┌─────────────────────────────────────────────────────────────┐
│                     Event Log (Source of Truth)             │
│  [RegistrationOpened] [SeatBooked] [SeatCancelled] [...]   │
└─────────────────────────────┬───────────────────────────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
     ┌────────────────┐ ┌──────────────┐ ┌────────────────┐
     │ Booking Status │ │  Dashboard   │ │ Badge Printer  │
     │ (Materialized  │ │ (Materialized│ │ (Materialized  │
     │    View)       │ │    View)     │ │    View)       │
     └────────────────┘ └──────────────┘ └────────────────┘
```

**Event Sourcing**: 모든 상태 변경을 불변 이벤트로 저장
**CQRS**: 쓰기 최적화(Event Log)와 읽기 최적화(Materialized Views) 분리

#### 9.2 이벤트 네이밍 규칙

```
❌ 현재형: "예약한다" (X)
✅ 과거형: "예약되었다" (SeatWasBooked)

→ 이벤트는 이미 발생한 사실의 기록
→ 취소해도 "예약했었다"는 사실은 변하지 않음
→ 취소는 별도 이벤트로 추가
```

#### 9.3 Command vs Event

```
┌──────────────┐     검증      ┌───────────────┐
│   Command    │ ───────────→ │    Event      │
│ (요청/명령)   │   성공 시     │ (확정된 사실)  │
│              │              │               │
│ "좌석 예약"   │ → 검증 후 → │ "좌석 예약됨"  │
└──────────────┘              └───────────────┘
```

#### 9.4 장점과 단점

| 장점 | 단점 |
|------|------|
| 의도가 명확히 전달됨 | 외부 정보 처리 주의 필요 |
| View 재생성 가능 (버그 수정 시) | GDPR 개인정보 삭제 어려움 |
| 다양한 읽기 최적화 View 생성 가능 | Side-effect 재실행 주의 |
| 새 기능 추가 용이 (새 이벤트 타입) | - |
| 감사 로그(Audit Log) 역할 | - |
| 높은 쓰기 처리량 | - |

> 💬 **비유**: Event Sourcing은 **회계 원장**과 같다. 잔액을 직접 수정하는 게 아니라 모든 거래를 기록하고, 잔액은 거래 내역에서 계산한다. 실수가 있으면 수정 거래를 추가한다.

### 10. DataFrame, 매트릭스, 배열

#### 10.1 DataFrame 특징

- R, Pandas, Spark에서 사용
- 관계형 테이블과 유사하지만 더 유연
- 데이터 랭글링(Wrangling)에 최적화
- 관계형 → 매트릭스 변환 지원

#### 10.2 관계형 → 매트릭스 변환 예시

```
관계형 테이블                      희소 매트릭스 (Sparse Matrix)
┌──────────┬──────────┬────────┐   ┌──────┬────────┬────────┬───────┐
│ user_id  │ movie_id │ rating │   │      │ Movie1 │ Movie2 │ Movie3│
├──────────┼──────────┼────────┤   ├──────┼────────┼────────┼───────┤
│ Alice    │ Movie1   │ 5      │   │Alice │   5    │   3    │   -   │
│ Alice    │ Movie2   │ 3      │   │Bob   │   4    │   -    │   2   │
│ Bob      │ Movie1   │ 4      │   │Carol │   -    │   5    │   4   │
│ Bob      │ Movie3   │ 2      │   └──────┴────────┴────────┴───────┘
│ Carol    │ Movie2   │ 5      │
│ Carol    │ Movie3   │ 4      │
└──────────┴──────────┴────────┘
```

**One-Hot Encoding** (범주형 데이터 변환):
```
장르: "comedy" → [1, 0, 0]
장르: "drama"  → [0, 1, 0]
장르: "horror" → [0, 0, 1]
```

### 면접 질문

<details>
<summary><strong>Q1: 관계형 모델과 문서 모델의 트레이드오프를 설명하세요.</strong></summary>

**관계형 모델**:
- 장점: JOIN 지원, 정규화로 데이터 일관성, 복잡한 쿼리 가능
- 단점: ORM 임피던스 불일치, 스키마 변경 비용

**문서 모델**:
- 장점: 스키마 유연성, 데이터 지역성, 객체 매핑 용이
- 단점: JOIN 지원 약함, 중첩 데이터 직접 참조 어려움

**선택 기준**: one-to-many 관계가 많고 트리 구조면 문서, many-to-many가 복잡하면 관계형
</details>

<details>
<summary><strong>Q2: 정규화와 비정규화의 트레이드오프를 설명하세요.</strong></summary>

**정규화**:
- 쓰기 빠름 (한 곳만 수정), 저장 공간 적음, 일관성 자동 유지
- 읽기 느림 (JOIN 필요)

**비정규화**:
- 읽기 빠름 (JOIN 불필요),
- 쓰기 느림 (여러 곳 수정), 저장 공간 많음, 일관성 수동 관리

**실무 예시**: SNS 타임라인에서 post_id만 저장(정규화)하고, 조회 시 hydration으로 내용 채움
</details>

<details>
<summary><strong>Q3: Event Sourcing의 장단점을 설명하세요.</strong></summary>

**장점**:
- 변경 의도가 명확 ("예약 취소됨" vs "active 컬럼 false로 변경")
- 버그 수정 후 View 재생성 가능
- 감사 로그 역할
- 높은 쓰기 처리량

**단점**:
- 외부 정보(환율 등) 처리 주의 필요
- GDPR 개인정보 삭제 어려움
- Side-effect(이메일 발송) 재실행 주의

**적합한 경우**: 복잡한 비즈니스 도메인, 감사 요구사항, 높은 쓰기 부하
</details>

<details>
<summary><strong>Q4: 그래프 DB가 관계형 DB보다 적합한 경우는?</strong></summary>

**그래프 DB 적합**:
- 관계가 핵심 데이터인 경우 (소셜 네트워크, 추천 시스템)
- 경로 탐색이 자주 필요 (친구의 친구, 최단 경로)
- 관계 깊이가 가변적 (N-hop traversal)
- 이질적 엔티티 간 다양한 관계 타입

**관계형 적합**:
- 고정된 스키마, 정형화된 관계
- 집계/분석 쿼리가 주요 사용 패턴
- 트랜잭션 ACID 보장이 중요

**예시**: "Lucy가 살고 있는 도시의 국가에서 태어난 사람 찾기" - 관계형은 31줄 SQL, Cypher는 4줄
</details>
