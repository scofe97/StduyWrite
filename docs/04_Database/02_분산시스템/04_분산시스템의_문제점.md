# Chapter 9: The Trouble with Distributed Systems

---

## 📌 핵심 요약
> 분산 시스템은 단일 컴퓨터와 근본적으로 다르다. **네트워크는 패킷을 잃거나 지연**시키고, **클럭은 동기화되지 않으며**, **프로세스는 언제든 멈출 수 있다**. 이러한 **부분 장애(Partial Failure)**는 비결정적이고 예측 불가능하다. 신뢰할 수 있는 시스템을 구축하려면 이런 현실을 인정하고, **타임아웃, 펜싱 토큰, 쿼럼** 같은 메커니즘으로 대응해야 한다.

## 📖 본문 정리

### 1. 부분 장애 (Partial Failures)

#### 단일 컴퓨터 vs 분산 시스템

```
┌─────────────────────────────────────────────────────────────────┐
│                     단일 컴퓨터 (Single Computer)                │
├─────────────────────────────────────────────────────────────────┤
│  • 결정적(Deterministic): 같은 연산 = 같은 결과                   │
│  • 이진적 상태: 완전히 동작 OR 완전히 고장                         │
│  • 하드웨어 문제 → 전체 시스템 장애 (커널 패닉, 블루스크린)         │
│  • 물리적 현실을 추상화하여 수학적 완벽함 제공                      │
└─────────────────────────────────────────────────────────────────┘
                              ↓ vs ↓
┌─────────────────────────────────────────────────────────────────┐
│                     분산 시스템 (Distributed System)             │
├─────────────────────────────────────────────────────────────────┤
│  • 비결정적(Nondeterministic): 때로는 성공, 때로는 실패           │
│  • 부분 장애: 일부는 동작, 일부는 고장                             │
│  • 성공 여부조차 알 수 없음 (응답 없음 = 성공? 실패?)              │
│  • 물리적 현실의 혼란스러움과 직접 대면                            │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 단일 컴퓨터는 "완벽하게 동작하거나 아예 꺼지는 전등"이고, 분산 시스템은 "일부 전구만 깜빡이고, 어느 전구가 문제인지도 모르는 크리스마스 트리 조명"이다.

#### 실제 발생하는 장애들 (Coda Hale 경험담)

| 장애 유형 | 설명 |
|-----------|------|
| 네트워크 파티션 | 단일 데이터센터 내에서도 발생 |
| PDU 장애 | 전력 분배 장치 고장 |
| 스위치 장애 | 네트워크 스위치 고장 |
| 전체 랙 전원 차단 | 실수로 랙 전체 전원 off |
| 데이터센터 백본 장애 | DC 간 연결 끊김 |
| 전체 DC 정전 | 데이터센터 전체 전원 상실 |
| 물리적 사고 | 트럭이 HVAC 시스템 충돌 |

### 2. 신뢰할 수 없는 네트워크 (Unreliable Networks)

#### 비동기 패킷 네트워크의 문제점

```
┌──────────┐                                      ┌──────────┐
│  Client  │                                      │  Server  │
└────┬─────┘                                      └────┬─────┘
     │                                                 │
     │ ─────① 요청 전송────────→ ?                     │
     │        (패킷 손실? 큐에서 대기?)                 │
     │                                                 │
     │        ② 서버 상태 불명                         │
     │        (크래시? GC 중? 처리 완료?)              │
     │                                                 │
     │ ← ─────③ 응답 불명 ─────────                    │
     │        (응답 손실? 지연 중?)                     │
     │                                                 │
     ▼                                                 ▼
  타임아웃!                                       (실제로는 처리 완료됨)
  "요청 실패"로 판단
```

**응답이 없을 때 알 수 없는 것들:**
1. 요청이 손실되었는가?
2. 요청이 큐에서 대기 중인가?
3. 서버가 크래시되었는가?
4. 서버가 일시적으로 응답 불능인가? (GC pause)
5. 응답이 손실되었는가?
6. 응답이 지연되고 있는가?

> 💬 **비유**: 편지를 보내고 답장을 기다리는 것과 같다. 답장이 안 오면 편지가 분실됐는지, 상대방이 바쁜지, 답장이 분실됐는지 알 수 없다.

#### TCP의 한계

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP가 제공하는 것                             │
├─────────────────────────────────────────────────────────────────┤
│  ✅ 손실된 패킷 재전송                                           │
│  ✅ 순서가 바뀐 패킷 재정렬                                       │
│  ✅ 체크섬으로 손상 감지                                          │
│  ✅ 혼잡 제어 (Congestion Control)                               │
└─────────────────────────────────────────────────────────────────┘
                              BUT
┌─────────────────────────────────────────────────────────────────┐
│                    TCP가 제공하지 못하는 것                       │
├─────────────────────────────────────────────────────────────────┤
│  ❌ 최종 전달 보장 (케이블 뽑히면 TCP도 무력)                     │
│  ❌ 원본 패킷 vs ACK 손실 구분                                    │
│  ❌ 애플리케이션 레벨 처리 완료 보장                               │
│  ❌ 연결 끊김 시 얼마나 처리됐는지 확인                            │
└─────────────────────────────────────────────────────────────────┘
```

**중요**: TCP ACK는 커널이 패킷을 받았다는 것만 의미. 애플리케이션이 처리했다는 보장은 **애플리케이션 레벨 응답**으로만 확인 가능.

#### 네트워크 장애 실제 사례

| 원인 | 설명 |
|------|------|
| 동물 | 소가 광섬유 밟음, 비버가 케이블 씹음, 상어 공격 |
| 인간 | 설정 오류, 케이블 절도, 사보타주 |
| 장비 | 스위치 장애, 라우터 문제, NIC 비대칭 장애 |
| 환경 | 소프트웨어 업그레이드 중 토폴로지 재구성 |

**비대칭 장애 예시**: 네트워크 인터페이스가 인바운드 패킷은 모두 드롭하면서 아웃바운드는 정상 전송 [22]

#### 네트워크 파티션 (Netsplit)

```
┌─────────────────────────────────────────────────────────────────┐
│                    네트워크 파티션 상황                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌───────┐       ┌───────┐       ┌───────┐                    │
│   │ Node  │ ←───→ │ Node  │   ✗   │ Node  │                    │
│   │   A   │       │   B   │       │   C   │                    │
│   └───────┘       └───────┘       └───────┘                    │
│       ↑               ↑               ↑                        │
│       └───────────────┘               │                        │
│             통신 가능            통신 불가                       │
│                                                                 │
│   특수 케이스: A↔B ✓, B↔C ✓, A↔C ✗ (부분 연결)                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**주의**: 네트워크 파티션과 샤딩 파티셔닝(Chapter 7)은 다른 개념!

#### 장애 감지 (Detecting Faults)

```
┌─────────────────────────────────────────────────────────────────┐
│              장애 감지가 필요한 상황                             │
├─────────────────────────────────────────────────────────────────┤
│  • 로드 밸런서: 죽은 노드에 요청 보내지 않기                      │
│  • 리더 선출: 리더 장애 시 팔로워 승격                           │
│  • 분산 트랜잭션: 참여자 장애 처리                               │
└─────────────────────────────────────────────────────────────────┘

명시적 피드백을 받을 수 있는 경우:
┌─────────────────────────────────────────────────────────────────┐
│ ✅ RST/FIN 패킷    | 프로세스 크래시, 포트 미사용               │
│ ✅ 스크립트 알림    | OS는 동작, 프로세스만 종료 (HBase 방식)   │
│ ✅ 스위치 관리 인터페이스 | 하드웨어 링크 장애 감지             │
│ ✅ ICMP Unreachable | 라우터가 IP 도달 불가 판단               │
└─────────────────────────────────────────────────────────────────┘

하지만 대부분의 경우:
┌─────────────────────────────────────────────────────────────────┐
│ ❌ 아무 응답 없음 → 타임아웃에 의존해야 함                       │
└─────────────────────────────────────────────────────────────────┘
```

#### 타임아웃과 무한 지연 (Timeouts and Unbounded Delays)

**타임아웃 딜레마:**

| 짧은 타임아웃 | 긴 타임아웃 |
|--------------|------------|
| ✅ 장애 빠르게 감지 | ✅ 잘못된 판정 감소 |
| ❌ 일시적 지연을 장애로 오판 | ❌ 장애 대응 지연 |
| ❌ 조기 선언 → 작업 중복 실행 | ❌ 사용자 오래 대기 |
| ❌ 부하 전이 → 연쇄 장애 | |

**이상적 타임아웃 공식** (실제로는 불가능):

```
가정:
  d = 네트워크 최대 지연 (bounded)
  r = 노드 최대 처리 시간 (bounded)

타임아웃 = 2d + r

현실:
  - 비동기 네트워크 = 무한 지연 가능 (unbounded)
  - 서버 처리 시간 = 보장 불가 (GC, 스와핑 등)
  → "올바른" 타임아웃 값은 존재하지 않음
```

#### 네트워크 혼잡과 큐잉

```
┌─────────────────────────────────────────────────────────────────┐
│                    네트워크 지연의 주요 원인                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1️⃣ 스위치 큐 (Switch Queue)                                   │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  Port 1 ──┐                                             │   │
│   │  Port 2 ──┼──→ [Queue] ──→ Port 3 (목적지)              │   │
│   │  Port 4 ──┘       ↑                                     │   │
│   │                   │                                     │   │
│   │              큐가 가득 차면                               │   │
│   │              패킷 DROP!                                  │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│   2️⃣ OS 수신 큐: CPU/스레드가 바쁘면 대기                       │
│   3️⃣ VM 버퍼: 다른 VM 실행 중이면 네트워크 소비 불가            │
│   4️⃣ TCP 송신 버퍼: 혼잡 제어로 인한 전송 지연                  │
│   5️⃣ TCP 재전송: 패킷 손실 시 재전송 대기                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 고속도로 정체와 같다. 평소에는 빠르게 이동하지만, 교통량이 많아지면 병목 지점에서 큐가 쌓이고 지연이 급격히 증가한다.

#### TCP vs UDP

| 특성 | TCP | UDP |
|------|-----|-----|
| 재전송 | ✅ 손실 시 자동 재전송 | ❌ 없음 |
| 순서 보장 | ✅ | ❌ |
| 흐름 제어 | ✅ | ❌ |
| 지연 변동성 | 높음 (재전송 대기) | 낮음 |
| 적합한 용도 | 데이터 무결성 중요 | 지연 민감 (VoIP, 화상회의) |

**UDP 선택 이유**: 지연된 데이터는 가치 없음 → 재전송보다 무시하고 다음 진행
```
VoIP 예: 패킷 손실 → 짧은 무음 → 계속 진행
        "다시 말씀해 주시겠어요?" (인간 레이어에서 재시도)
```

#### 동기식 vs 비동기식 네트워크

```
┌─────────────────────────────────────────────────────────────────┐
│                    회선 교환 (Circuit Switching)                 │
│                    - 전통적 전화망 -                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [발신자] ═══════════════════════════════════════ [수신자]     │
│            고정 대역폭 할당 (예: 64kbps)                         │
│            통화 종료까지 독점 사용                               │
│                                                                 │
│   장점: 지연 시간 예측 가능 (bounded delay)                      │
│   단점: 유휴 시에도 대역폭 점유 → 낭비                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    패킷 교환 (Packet Switching)                  │
│                    - 인터넷, 데이터센터 -                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [발신자] ──┬──┬──┬──→ [스위치] ──→ [수신자]                   │
│             패킷들이 가용 대역폭 경쟁                            │
│                                                                 │
│   장점: 회선 활용률 극대화, 버스트 트래픽에 적합                  │
│   단점: 큐잉으로 인한 무한 지연 가능 (unbounded)                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**왜 패킷 교환을 선택했나?**
- 웹 요청, 이메일, 파일 전송 = **버스트(Bursty) 트래픽**
- 회선 교환은 버스트에 비효율적 (대역폭 추측 필요)
- TCP가 가용 대역폭에 맞춰 동적 조절

**핵심 통찰**: 가변 지연은 자연법칙이 아니라 **비용/이익 트레이드오프**의 결과

```
정적 자원 할당 → 지연 보장 가능 → 활용률 낮음 → 비쌈
동적 자원 공유 → 지연 보장 불가 → 활용률 높음 → 쌈
```

#### Phi Accrual Failure Detector

```
고정 타임아웃 대신 적응형 접근:
┌─────────────────────────────────────────────────────────────────┐
│  1. 응답 시간 분포 지속적 측정                                   │
│  2. 지터(Jitter) 추적                                           │
│  3. 관측된 분포에 따라 타임아웃 자동 조절                         │
└─────────────────────────────────────────────────────────────────┘
사용 예: Akka, Cassandra
```

### 3. 신뢰할 수 없는 클럭 (Unreliable Clocks)

#### 클럭이 필요한 상황들

| 용도 | 예시 |
|------|------|
| **지속 시간 측정** | 타임아웃, 응답 시간, QPS |
| **특정 시점** | 기사 발행일, 알림 예약, 캐시 만료, 로그 타임스탬프 |

#### 분산 시스템에서 시간이 어려운 이유

```
┌───────────────────────────────────────────────────────────────────┐
│                    문제 1: 통신 지연                              │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│   Node A (시간: 100ms)                     Node B                 │
│        │                                      │                   │
│        │ ────── 메시지 전송 ──────→           │                   │
│        │        (네트워크 지연: ?ms)          │                   │
│        │                                      │ (시간: ?ms)       │
│        │                                      │                   │
│   메시지 수신 시점이 발신 시점보다 늦지만,                         │
│   "얼마나" 늦은지는 알 수 없음                                    │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────────┐
│                    문제 2: 클럭 동기화 불완전                      │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│   각 머신의 클럭 = 독립적인 쿼츠 오실레이터                        │
│   → 각자의 속도로 drift (빨라지거나 느려짐)                        │
│                                                                   │
│   NTP로 동기화해도:                                               │
│   - 네트워크 지연만큼의 오차 발생                                 │
│   - 인터넷 NTP: 최소 35ms 오차, 혼잡 시 100ms+                    │
│   - 로컬 NTP: 수 ms 오차                                         │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

#### Time-of-Day Clock vs Monotonic Clock

```
┌─────────────────────────────────────────────────────────────────┐
│                    Time-of-Day Clock                             │
│                    (벽시계, Wall Clock)                          │
├─────────────────────────────────────────────────────────────────┤
│  API: clock_gettime(CLOCK_REALTIME), System.currentTimeMillis() │
│                                                                 │
│  반환값: Unix epoch(1970-01-01 UTC) 이후 초/밀리초               │
│                                                                 │
│  특성:                                                          │
│  ✅ NTP 동기화 → 다른 머신과 (이상적으로) 같은 의미              │
│  ❌ 뒤로 점프할 수 있음 (NTP 조정, 윤초)                         │
│  ❌ DST로 점프 (UTC 사용으로 회피 가능)                          │
│  ❌ 경과 시간 측정에 부적합                                      │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    Monotonic Clock                               │
│                    (단조 증가 클럭)                              │
├─────────────────────────────────────────────────────────────────┤
│  API: clock_gettime(CLOCK_MONOTONIC), System.nanoTime()         │
│                                                                 │
│  반환값: 부팅 이후 나노초 등 (절대값 무의미)                      │
│                                                                 │
│  특성:                                                          │
│  ✅ 항상 앞으로만 진행 (뒤로 점프 없음)                          │
│  ✅ 경과 시간 측정에 적합 (스톱워치처럼)                         │
│  ❌ 절대값은 머신마다 다름 → 비교 불가                           │
│  ⚠️ NTP가 속도 조절(slewing)은 할 수 있음 (±0.05%)              │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**:
> - Time-of-Day Clock = 손목시계 (현재 몇 시인지 알려줌, 하지만 배터리 교체하면 시간 재설정 필요)
> - Monotonic Clock = 스톱워치 (몇 분 경과했는지 측정, 절대 시간은 모름)

#### 클럭 동기화의 어려움

| 문제 | 설명 |
|------|------|
| 쿼츠 드리프트 | 200ppm = 30초 동기화 주기에 6ms, 하루에 17초 |
| 강제 리셋 | NTP와 차이가 크면 시간이 갑자기 점프 |
| 방화벽 | NTP 트래픽 차단 시 드리프트 누적 |
| 잘못된 NTP 서버 | 수 시간 오차 보고하는 서버 존재 |
| 윤초 | 분이 59초 or 61초 → 많은 시스템 크래시 유발 |
| VM 클럭 | VM 일시정지 시 수 초 뒤처짐 가능 |
| 사용자 조작 | 게임 치팅용 시간 조작 |

**고정밀 동기화 (금융권 MiFID II)**:
- 요구사항: UTC 대비 100μs 이내
- 필요 장비: GPS 수신기 또는 원자 시계, PTP (Precision Time Protocol)
- GPS 재밍 위험 존재

#### 타임스탬프로 이벤트 순서 정하기의 위험

```
┌─────────────────────────────────────────────────────────────────┐
│                  LWW(Last Write Wins) 문제                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Client A              Node 1          Node 3         Node 2   │
│       │                   │               │              │      │
│       │──── x=1 ────→     │               │              │      │
│       │               [42.004초]          │              │      │
│       │                   │ ─────복제────→│              │      │
│       │                                   │              │      │
│   Client B                                │              │      │
│       │────────────── x++ (x=2) ────→     │              │      │
│       │                              [42.003초]          │      │
│       │                                   │ ───복제───→  │      │
│       │                                                  │      │
│       │                                                  │      │
│       │                   Node 2에서 비교:                │      │
│       │                   x=1 (42.004초) vs x=2 (42.003초)│      │
│       │                   → x=1이 "최신"으로 판정!       │      │
│       │                   → x=2 증가분 손실! 💀           │      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**LWW의 문제점:**
1. 느린 클럭을 가진 노드의 쓰기가 무시됨
2. 인과 관계 있는 순차 쓰기 vs 진짜 동시 쓰기 구분 불가
3. 같은 타임스탬프 충돌 시 임의 해결 → 인과성 위반

**해결책**: 물리적 클럭(Physical Clock) 대신 **논리적 클럭(Logical Clock)** 사용
- 증가하는 카운터 기반
- 실제 시간 측정하지 않음
- 이벤트의 상대적 순서만 결정
- 자세한 내용은 "Detecting Concurrent Writes" 참조

#### 클럭 읽기의 신뢰 구간 (Confidence Interval)

```
┌─────────────────────────────────────────────────────────────────┐
│              클럭 값 ≠ 정확한 시점                               │
│              클럭 값 = 시간 범위 (신뢰 구간)                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   예: 95% 신뢰도로 현재 시간은 10.3초 ~ 10.5초 사이              │
│                                                                 │
│   ├─────────[━━━━━━━━━━━━]─────────┤                           │
│   10.3초    실제 시간 범위    10.5초                            │
│                                                                 │
│   ±100ms 오차라면 마이크로초 자릿수는 의미 없음                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**대부분의 시스템**: 오차 범위를 노출하지 않음

**예외:**
- Google Spanner의 **TrueTime API**: [earliest, latest] 반환
- Amazon의 **ClockBound**: 신뢰 구간 제공

#### Spanner의 글로벌 스냅샷

```
┌─────────────────────────────────────────────────────────────────┐
│                 Spanner의 TrueTime 활용                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   트랜잭션 A: [A_earliest ──────── A_latest]                    │
│   트랜잭션 B:                      [B_earliest ──── B_latest]   │
│                                                                 │
│   구간이 겹치지 않으면:                                          │
│   A_earliest < A_latest < B_earliest < B_latest                 │
│   → B는 확실히 A 이후에 발생                                     │
│                                                                 │
│   구간이 겹치면:                                                 │
│   → 순서 불확실                                                  │
│                                                                 │
│   Spanner 전략:                                                  │
│   커밋 전에 신뢰 구간 길이만큼 대기 (wait out the uncertainty)   │
│   → 이후 트랜잭션과 구간이 겹치지 않도록 보장                    │
│                                                                 │
│   원자 시계/GPS로 불확실성 최소화 (~7ms)                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 프로세스 일시정지 (Process Pauses)

#### 리스 (Lease) 기반 리더십의 문제

```java
while (true) {
    request = getIncomingRequest();

    // 리스 만료 10초 전에 갱신
    if (lease.expiryTimeMillis - System.currentTimeMillis() < 10000) {
        lease = lease.renew();
    }

    if (lease.isValid()) {
        process(request);  // ← 여기서 15초 멈추면?
    }
}
```

**문제점:**
1. 동기화된 클럭에 의존 (만료 시간은 다른 머신에서 설정)
2. 시간 체크와 요청 처리 사이에 일시정지 가능

#### 일시정지의 원인들

```
┌─────────────────────────────────────────────────────────────────┐
│                    프로세스 일시정지 원인                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1️⃣ 스레드 경합                                                 │
│      락, 큐 등 공유 자원 대기 시간 증가                          │
│      (CPU 코어 증가 시 오히려 악화 가능)                         │
│                                                                 │
│   2️⃣ GC (Garbage Collection)                                    │
│      Stop-the-world 일시정지                                    │
│      과거: 수 분, 현재: 수 ms (개선됨)                           │
│                                                                 │
│   3️⃣ VM 일시중지/재개                                           │
│      Live Migration (다른 호스트로 이동)                        │
│      노트북 덮개 닫기                                           │
│                                                                 │
│   4️⃣ 컨텍스트 스위칭                                            │
│      OS의 다른 스레드 실행                                      │
│      하이퍼바이저의 다른 VM 실행 (steal time)                   │
│                                                                 │
│   5️⃣ 동기식 디스크 I/O                                          │
│      느린 디스크 작업 대기                                      │
│      네트워크 파일시스템 추가 지연                               │
│      Java classloader의 lazy loading                           │
│                                                                 │
│   6️⃣ 페이지 스와핑 (Paging)                                     │
│      메모리 → 디스크 스왑 시 I/O 대기                           │
│      Thrashing: 대부분의 시간을 스왑에 사용                     │
│      서버에서는 보통 비활성화                                   │
│                                                                 │
│   7️⃣ SIGSTOP 시그널                                             │
│      Ctrl-Z로 프로세스 정지                                     │
│      SIGCONT까지 CPU 할당 없음                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 프로세스는 언제든 "깜빡 잠들 수 있고", 깨어났을 때 자신이 얼마나 잤는지 모른다. 그 사이 세상은 계속 돌아가고, 심지어 자신을 "죽었다"고 선언했을 수도 있다.

#### Hard Real-Time 시스템

| 일반 시스템 | Hard Real-Time 시스템 |
|------------|---------------------|
| 대부분 빠르면 충분 | 데드라인 필수 준수 |
| GC 허용 | GC 제한/금지 |
| 일반 OS 사용 | RTOS (Real-Time OS) |
| 동적 메모리 할당 | 제한적 사용 |
| 처리량 우선 | 응답 시간 우선 |
| 경제적 | 매우 비쌈 |

**사용 분야**: 항공기, 로켓, 자동차, 로봇 제어, 에어백

**주의**: "Real-Time"은 "빠름"이 아니라 "예측 가능한 응답 시간 보장"을 의미

#### GC 영향 완화 전략

| 전략 | 설명 |
|------|------|
| 최신 GC 사용 | ZGC, Shenandoah, G1 등 |
| 오브젝트 풀링 | 객체 재사용, off-heap 저장 |
| GC 없는 언어 | Rust (ownership), Swift (ARC) |
| GC 시점 조절 | 요청 처리 전에 노드에서 트래픽 제거 후 GC |
| 주기적 재시작 | Full GC 전에 프로세스 재시작 (롤링) |

### 5. 지식, 진실, 거짓말 (Knowledge, Truth, and Lies)

#### 노드가 알 수 있는 것

```
┌─────────────────────────────────────────────────────────────────┐
│              분산 시스템 노드의 지식 한계                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   노드 A가 노드 B에 대해 알 수 있는 것:                          │
│   ✅ B로부터 받은 메시지 내용                                    │
│   ✅ 메시지를 받았다는 사실                                      │
│                                                                 │
│   노드 A가 알 수 없는 것:                                        │
│   ❌ B의 현재 상태                                               │
│   ❌ B의 데이터                                                  │
│   ❌ B가 정상 동작 중인지                                        │
│   ❌ 응답 없음이 네트워크 문제인지 B 문제인지                    │
│                                                                 │
│   결론: 분산 시스템에서 확신할 수 있는 것은 거의 없다            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 다수결의 원칙 (The Majority Rules)

**시나리오 1: 비대칭 네트워크 장애**
```
┌─────────────────────────────────────────────────────────────────┐
│   반쯤 연결된 노드 (Semi-disconnected Node)                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [Node A]  ←── 수신 OK ──  [다른 노드들]                       │
│      │                          │                               │
│      └── 송신 실패 ──→          │                               │
│           (드롭/지연)           │                               │
│                                 ↓                               │
│                        "A는 죽었다" 판정                         │
│                                                                 │
│   Node A 입장: "나 살아있는데?!" 😱                              │
│   하지만 아무도 들을 수 없음...                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**시나리오 2: 일시정지 후 좀비**
```
┌─────────────────────────────────────────────────────────────────┐
│   1분간 일시정지된 노드                                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   t=0     Node A 일시정지 시작 (GC, VM suspend 등)              │
│   t=30s   다른 노드들: "A 응답 없음"                             │
│   t=45s   다른 노드들: "A는 죽었다" 판정 → 새 리더 선출          │
│   t=60s   Node A 재개: "어? 방금 뭐 있었나?"                    │
│           → A는 자신이 여전히 리더라고 생각                      │
│           → Split Brain! 🧟                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 장례식에서 관 뚜껑을 열고 벌떡 일어나는 사람. 다른 사람들은 놀라고, 본인은 왜 모두가 검은 옷을 입었는지 모른다.

**교훈: 개별 노드는 자신의 판단을 신뢰할 수 없다**

**쿼럼 (Quorum) 방식:**
```
┌─────────────────────────────────────────────────────────────────┐
│              쿼럼 기반 의사결정                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   단일 노드 결정 (위험):                                         │
│   Node A: "나는 리더다" → 장애 시 시스템 멈춤                    │
│                                                                 │
│   쿼럼 결정 (안전):                                              │
│   과반수 노드: "Node A는 죽었다" → A는 죽은 것으로 취급          │
│   A가 실제로 살아있어도 쿼럼 결정에 따라야 함                    │
│                                                                 │
│   장점:                                                          │
│   - 3노드 중 1노드 장애 허용                                     │
│   - 5노드 중 2노드 장애 허용                                     │
│   - 동시에 두 개의 상충되는 과반수 불가능                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 분산 락과 리스 (Distributed Locks and Leases)

**리스의 위험한 사용:**
- 리더 선출 (split brain 방지)
- 특정 리소스의 단독 접근 권한
- 작업 중복 실행 방지

```
│                                   │                    │ 💥     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 펜싱 토큰 (Fencing Token)

```
┌─────────────────────────────────────────────────────────────────┐
│                    펜싱 토큰으로 좀비 차단                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Lock Service            Client 1/2           Storage          │
│       │                       │                   │             │
│       │ ── token=33 ──→ Client 1                  │             │
│       │                       │                   │             │
│       │                  [일시정지...]             │             │
│       │                       │                   │             │
│       │ ── token=34 ──→ Client 2                  │             │
│       │                       │                   │             │
│       │              Client 2: 쓰기(token=34) ──→ │ ✓           │
│       │                       │                   │ max=34      │
│       │                       │                   │             │
│       │              [Client 1 재개]              │             │
│       │              Client 1: 쓰기(token=33) ──→ │             │
│       │                       │                   │ 33 < 34     │
│       │                       │                   │ → 거부! ✓   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

규칙: Storage는 이전에 본 최대 토큰보다 작은 토큰의 쓰기 거부
```

**펜싱 토큰의 다른 이름:**
- Chubby: Sequencer
- Kafka: Epoch Number
- Paxos: Ballot Number
- Raft: Term Number

**구현 방법:**
- ZooKeeper: `zxid` 또는 `cversion` 사용
- etcd: `revision` + `lease ID`
- Hazelcast: `FencedLock` API
- 조건부 쓰기: S3 Conditional Writes, Azure Conditional Headers

#### 복제본에서의 펜싱

```
┌─────────────────────────────────────────────────────────────────┐
│            Leaderless 복제 DB에서 펜싱 토큰 활용                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   타임스탬프 구성: [펜싱토큰][실제시간]                          │
│                                                                 │
│   Client 1 (token=33): 타임스탬프 = 33.xxxxx                    │
│   Client 2 (token=34): 타임스탬프 = 34.xxxxx                    │
│                                                                 │
│   → 34.xxxxx는 항상 33.xxxxx보다 큼                             │
│   → LWW에서 Client 2의 쓰기가 항상 우선                         │
│                                                                 │
│   쿼럼 쓰기 시:                                                  │
│   ┌────────┐  ┌────────┐  ┌────────┐                           │
│   │Replica1│  │Replica2│  │Replica3│ (오프라인)                 │
│   └───┬────┘  └───┬────┘  └────────┘                           │
│       │           │                                             │
│   C2: 34.xx ✓     34.xx ✓                                       │
│   C1: 33.xx 거부  33.xx 거부    33.xx 성공 (but minority)       │
│                                                                 │
│   결과: 쿼럼 읽기 시 34.xx 값 반환 (더 높은 타임스탬프)          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 비잔틴 장애 (Byzantine Faults)

```
┌─────────────────────────────────────────────────────────────────┐
│                    신뢰 수준별 장애 유형                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1️⃣ 정직하지만 신뢰할 수 없음 (Honest but Unreliable)          │
│      - 느리거나 응답 없음 (장애)                                 │
│      - 상태가 오래됨 (GC pause, 네트워크 지연)                   │
│      - 하지만 응답한다면 프로토콜을 따름                         │
│      → 이 책에서 다루는 범위                                     │
│                                                                 │
│   2️⃣ 비잔틴 장애 (Byzantine Faults)                             │
│      - 노드가 "거짓말"할 수 있음                                 │
│      - 잘못된/손상된 응답 전송                                   │
│      - 악의적 행동 가능                                          │
│      → 훨씬 어려운 문제                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**비잔틴 장군 문제 (Byzantine Generals Problem):**
- n명의 장군이 합의해야 함
- 일부는 배신자 (거짓 메시지 전송)
- 누가 배신자인지 사전에 알 수 없음

**비잔틴 내결함성이 필요한 경우:**

| 분야 | 이유 |
|------|------|
| 항공우주 | 방사선으로 인한 메모리/CPU 오류 |
| 블록체인/암호화폐 | 서로 신뢰하지 않는 참여자들 |
| P2P 네트워크 | 중앙 권위 없음 |

**대부분의 서버 시스템에서 불필요한 이유:**
- 데이터센터 노드는 조직이 통제
- 방사선 수준 낮음
- 비용이 매우 높음
- 멀티테넌트 격리는 방화벽/가상화/접근제어로 충분

**약한 형태의 "거짓말" 방어:**

| 방어 수단 | 대상 |
|-----------|------|
| 애플리케이션 레벨 체크섬 | 네트워크 패킷 손상 |
| 입력 검증/이스케이핑 | SQL 인젝션, XSS |
| 다중 NTP 서버 | 잘못된 시간 보고 |
| TLS 암호화 | 전송 중 변조 |

### 6. 시스템 모델과 현실 (System Model and Reality)

#### 타이밍 가정에 따른 시스템 모델

```
┌─────────────────────────────────────────────────────────────────┐
│                    동기식 모델 (Synchronous)                     │
├─────────────────────────────────────────────────────────────────┤
│  가정: 네트워크 지연, 프로세스 일시정지, 클럭 오차 모두 bounded  │
│  현실성: ❌ 비현실적 (unbounded 지연/일시정지 실제로 발생)       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    부분 동기식 모델 (Partially Synchronous)      │
├─────────────────────────────────────────────────────────────────┤
│  가정: 대부분 동기식처럼 동작하나, 가끔 bound 초과               │
│  현실성: ✅ 현실적 (대부분의 실제 시스템)                        │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    비동기식 모델 (Asynchronous)                  │
├─────────────────────────────────────────────────────────────────┤
│  가정: 타이밍에 대한 어떤 가정도 금지, 클럭도 없음               │
│  현실성: 매우 제한적 (일부 알고리즘만 가능)                      │
└─────────────────────────────────────────────────────────────────┘
```

#### 노드 장애에 대한 시스템 모델

```
┌─────────────────────────────────────────────────────────────────┐
│                    Crash-Stop (Fail-Stop)                        │
├─────────────────────────────────────────────────────────────────┤
│  가정: 노드는 크래시로만 실패, 크래시 후 영원히 복귀 안 함       │
│  특성: 가장 단순한 모델                                          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    Crash-Recovery                                │
├─────────────────────────────────────────────────────────────────┤
│  가정: 크래시 후 언젠가 복귀 가능                                │
│  특성: 안정적 저장소(디스크)는 보존, 메모리는 손실               │
│  현실성: ✅ 가장 현실적인 모델                                   │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    Degraded Performance / Gray Failure           │
├─────────────────────────────────────────────────────────────────┤
│  현상: 노드가 느려짐 (헬스체크는 통과, 실제 작업은 못함)         │
│       - 드라이버 버그로 네트워크 속도 급감                       │
│       - 메모리 압박으로 GC만 수행                                │
│       - SSD 성능 불안정                                          │
│  특성: 깔끔한 장애보다 처리하기 더 어려움 (Limping Node)         │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    Byzantine (Arbitrary) Faults                  │
├─────────────────────────────────────────────────────────────────┤
│  가정: 노드가 무엇이든 할 수 있음 (속이기, 거짓 응답)            │
│  특성: 가장 어려운 모델, 대부분의 경우 불필요                    │
└─────────────────────────────────────────────────────────────────┘
```

#### Safety vs Liveness 속성

```
┌─────────────────────────────────────────────────────────────────┐
│                    Safety 속성                                   │
│                    "나쁜 일이 일어나지 않는다"                   │
├─────────────────────────────────────────────────────────────────┤
│  예시:                                                          │
│  - Uniqueness: 두 요청이 같은 펜싱 토큰을 받지 않음             │
│  - Monotonic: 완료 순서에 따라 토큰이 증가                      │
│  - Serializability: 트랜잭션이 순차 실행된 것처럼 동작          │
│                                                                 │
│  특성:                                                          │
│  - 위반 시점을 특정할 수 있음                                   │
│  - 한번 위반되면 되돌릴 수 없음 (피해 발생)                     │
│  - 모든 상황에서 항상 보장되어야 함                             │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    Liveness 속성                                 │
│                    "좋은 일이 결국 일어난다"                     │
├─────────────────────────────────────────────────────────────────┤
│  예시:                                                          │
│  - Availability: 요청하면 결국 응답 받음                        │
│  - Eventual Consistency: 결국 모든 복제본이 일치                │
│  - Termination: 알고리즘이 결국 종료                            │
│                                                                 │
│  특성:                                                          │
│  - 특정 시점에 아직 만족 안 될 수 있음 (미래 희망)              │
│  - 조건부 허용 가능 (과반수 정상, 네트워크 복구 시)             │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**:
> - Safety: "차가 충돌하지 않는다" (한번 충돌하면 되돌릴 수 없음)
> - Liveness: "차가 결국 목적지에 도착한다" (지금은 막혀있어도 언젠가...)

### 7. 형식 검증과 테스팅 (Formal Methods and Testing)

#### 검증 방법 비교

| 방법 | 설명 | 장단점 |
|------|------|--------|
| **Model Checking** | TLA+, FizzBee 등으로 알고리즘 명세 작성 후 모든 상태 탐색 | ✅ 구현 전 버그 발견, ❌ 무한 상태 공간 |
| **Fault Injection** | 실제 시스템에 장애 주입 (Chaos Monkey, Jepsen) | ✅ 실제 코드 테스트, ❌ 모든 시나리오 불가 |
| **DST** | 결정적 시뮬레이션으로 많은 실행 경로 탐색 | ✅ 재현 가능, ❌ 결정성 확보 어려움 |

#### Model Checking (TLA+ 등)

```
┌─────────────────────────────────────────────────────────────────┐
│                    Model Checking 프로세스                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 알고리즘을 명세 언어로 작성 (TLA+, FizzBee, Gallina)       │
│   2. 불변 조건(Invariant) 정의                                  │
│   3. Model Checker가 가능한 모든 상태 탐색                      │
│   4. 불변 조건 위반 여부 확인                                   │
│                                                                 │
│   한계:                                                          │
│   - 무한 상태 공간 → 제한된 범위만 검증                         │
│   - 명세 ≠ 구현 (동기화 필요)                                   │
│                                                                 │
│   사용 예: CockroachDB, TiDB, Kafka                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Fault Injection

```
┌─────────────────────────────────────────────────────────────────┐
│                    Fault Injection 유형                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   네트워크: 패킷 드롭, 지연, 파티션                              │
│   프로세스: kill, SIGSTOP (pause)                               │
│   디스크: umount, 손상                                          │
│   시스템: 전원 차단, VM 일시중지                                │
│                                                                 │
│   도구:                                                          │
│   - Jepsen: 분산 시스템 테스트 프레임워크                       │
│   - Chaos Monkey: Netflix의 프로덕션 장애 주입                  │
│   - tc (Traffic Control): Linux 네트워크 조작                   │
│                                                                 │
│   Jepsen 발견 버그: 많은 유명 시스템의 심각한 결함 발견         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Deterministic Simulation Testing (DST)

```
┌─────────────────────────────────────────────────────────────────┐
│                    DST의 핵심 아이디어                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   비결정적 요소를 제어:                                          │
│   - 네트워크 통신 → 모의 네트워크                               │
│   - I/O 작업 → 모의 파일시스템                                  │
│   - 클럭 → 모의 시간 (빠르게 진행 가능)                         │
│   - 스레드 스케줄링 → 단일 스레드 또는 제어된 스케줄링          │
│                                                                 │
│   장점:                                                          │
│   - 실제 코드 테스트 (모델이 아님)                              │
│   - 버그 재현 가능 (같은 시드 = 같은 결과)                      │
│   - 실제 시간보다 빠르게 실행 가능                              │
│   - 더 많은 코드 경로 탐색                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**DST 구현 수준:**

| 수준 | 방식 | 예시 |
|------|------|------|
| **Application Level** | 비동기 라이브러리를 결정적으로 설계 | FoundationDB (Flow), TigerBeetle |
| **Runtime Level** | 런타임 패치로 결정성 주입 | madsim (Rust), FrostDB (Go) |
| **Machine Level** | 하이퍼바이저 레벨에서 모든 비결정적 호출 대체 | Antithesis |

#### 결정성의 힘 (The Power of Determinism)

```
┌─────────────────────────────────────────────────────────────────┐
│                    결정성이 활용되는 곳                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. Event Sourcing: 이벤트 재생으로 상태 재구성                │
│   2. Workflow Engines: 결정적 워크플로우로 내구성 실행          │
│   3. State Machine Replication: 동일 트랜잭션 순서 → 동일 상태 │
│   4. DST: 결정적 시뮬레이션으로 버그 재현                       │
│                                                                 │
│   비결정성 제거 주의점:                                          │
│   - 해시 테이블 순회 순서                                       │
│   - 리소스 제한 (메모리 할당 실패, 스택 오버플로)               │
│   - 난수 생성                                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 면접 예상 질문

**Q1: 분산 시스템에서 타임아웃을 어떻게 설정해야 하나요?**
> 정답은 없다. 너무 짧으면 일시적 지연을 장애로 오판하여 불필요한 페일오버와 연쇄 장애 유발. 너무 길면 실제 장애 대응이 늦어진다. 실험적으로 네트워크 RTT 분포를 측정하고, Phi Accrual Failure Detector처럼 적응형 타임아웃을 고려해야 한다.

**Q2: 펜싱 토큰이란 무엇이며 왜 필요한가요?**
> 락/리스 획득 시 부여되는 단조 증가하는 숫자. 일시정지나 네트워크 지연으로 인해 이전 리스 보유자가 여전히 유효하다고 착각하고 쓰기를 시도할 수 있다. Storage가 최대 토큰보다 작은 쓰기를 거부함으로써 좀비와 지연된 요청을 차단한다.

**Q3: 왜 분산 시스템에서 wall-clock 타임스탬프로 이벤트 순서를 정하면 안 되나요?**
> NTP 동기화가 완벽하지 않아 노드 간 클럭 편차 발생. 인과적으로 나중에 발생한 이벤트가 더 이른 타임스탬프를 가질 수 있어 LWW에서 데이터 손실. 논리적 클럭(Lamport/Vector clock)이나 신뢰 구간 기반 접근(Spanner TrueTime)을 사용해야 한다.

**Q4: Safety와 Liveness 속성의 차이는 무엇인가요?**
> Safety: "나쁜 일이 일어나지 않음" - 위반 시점 특정 가능, 되돌릴 수 없음, 항상 보장 필요. Liveness: "좋은 일이 결국 일어남" - 현재 미충족이어도 미래 희망, 조건부 허용 가능 (과반수 정상일 때). 분산 알고리즘은 Safety는 항상, Liveness는 조건부로 보장한다.
