# Chapter 7: Sharding

---

## 📌 핵심 요약
> 이 장에서는 **대용량 데이터를 여러 노드에 분산 저장하는 샤딩(Sharding)** 기법을 다룬다. 복제(Replication)가 동일한 데이터의 복사본을 여러 노드에 저장하는 것이라면, 샤딩은 데이터를 **작은 조각(shard)으로 나누어 서로 다른 노드에 저장**하는 것이다. 키 범위 샤딩, 해시 샤딩, 일관된 해싱 등 다양한 샤딩 전략과 핫스팟 문제, 리밸런싱, 요청 라우팅, 보조 인덱스 처리 방법을 학습한다.

## 📖 본문 정리

### 1. 샤딩 개요

#### 1.1 복제 vs 샤딩

```
┌─────────────────────────────────────────────────────────────────────┐
│                    분산 데이터베이스의 두 가지 전략                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  복제 (Replication)              샤딩 (Sharding)                    │
│  ┌───────────────┐               ┌───────────────┐                  │
│  │  Full Data    │               │  Shard A      │                  │
│  │  ┌─────────┐  │               │  [Records     │                  │
│  │  │A B C D E│  │               │   1-1000]     │                  │
│  │  └─────────┘  │               └───────────────┘                  │
│  └───────────────┘               ┌───────────────┐                  │
│  ┌───────────────┐               │  Shard B      │                  │
│  │  Full Data    │               │  [Records     │                  │
│  │  ┌─────────┐  │  (Copy)       │   1001-2000]  │                  │
│  │  │A B C D E│  │               └───────────────┘                  │
│  │  └─────────┘  │               ┌───────────────┐                  │
│  └───────────────┘               │  Shard C      │                  │
│                                  │  [Records     │                  │
│  동일한 데이터 복사본              │   2001-3000]  │                  │
│  → 가용성, 내결함성               └───────────────┘                  │
│                                                                     │
│                                  다른 데이터 조각                     │
│                                  → 확장성, 처리량                    │
└─────────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 복제는 같은 책을 여러 도서관에 비치하는 것이고, 샤딩은 백과사전을 A-F, G-L, M-R, S-Z 권으로 나누어 다른 서가에 보관하는 것이다.

#### 1.2 샤딩 + 복제 조합

```
┌─────────────────────────────────────────────────────────────────────┐
│                 샤딩과 복제의 조합 (Single-Leader)                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Node 1                Node 2                Node 3                │
│   ┌──────────┐          ┌──────────┐          ┌──────────┐         │
│   │ Shard A  │ ──────▶  │ Shard A  │ ──────▶  │ Shard A  │         │
│   │ (Leader) │          │(Follower)│          │(Follower)│         │
│   ├──────────┤          ├──────────┤          ├──────────┤         │
│   │ Shard B  │ ◀──────  │ Shard B  │          │ Shard B  │         │
│   │(Follower)│          │ (Leader) │ ──────▶  │(Follower)│         │
│   ├──────────┤          ├──────────┤          ├──────────┤         │
│   │ Shard C  │          │ Shard C  │ ◀──────  │ Shard C  │         │
│   │(Follower)│          │(Follower)│          │ (Leader) │         │
│   └──────────┘          └──────────┘          └──────────┘         │
│                                                                     │
│   각 노드는 일부 샤드의 리더이면서 다른 샤드의 팔로워                    │
└─────────────────────────────────────────────────────────────────────┘
```

#### 1.3 용어 정리

| 시스템 | 용어 |
|--------|------|
| Kafka | Partition |
| CockroachDB | Range |
| HBase, TiDB | Region |
| Bigtable, YugabyteDB, ScyllaDB | Tablet |
| Riak | Vnode |
| Cassandra | Token-range |
| Couchbase | vBucket |
| PostgreSQL | Partition (단일 머신) / Shard (분산) |

### 2. 샤딩의 장단점

#### 2.1 장점

| 장점 | 설명 |
|------|------|
| **수평 확장성** | 데이터/트래픽 증가 시 노드 추가로 대응 (Scale-out) |
| **처리량 증가** | 쓰기 부하를 여러 노드에 분산 |
| **병렬 처리** | 여러 샤드에서 동시에 쿼리 실행 |
| **비용 효율** | 고가의 대형 서버 대신 저렴한 소형 서버 다수 사용 |

#### 2.2 단점

| 단점 | 설명 |
|------|------|
| **복잡성 증가** | 파티션 키 선택, 라우팅 로직 필요 |
| **조인 어려움** | 샤드 간 조인은 비효율적 |
| **분산 트랜잭션** | 여러 샤드에 걸친 트랜잭션은 느리고 복잡 |
| **핫스팟 위험** | 불균등한 부하 분산 가능성 |
| **스키마 변경 어려움** | 샤딩 스키마 변경은 대규모 데이터 이동 필요 |

> 💬 **비유**: 샤딩은 고속도로를 여러 차선으로 나누는 것과 같다. 차량(데이터)을 분산시킬 수 있지만, 차선 변경(샤드 간 이동)은 비용이 크고, 한 차선에 차가 몰리면(핫스팟) 전체가 막힌다.

### 3. 멀티테넌시를 위한 샤딩

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Cell-Based Architecture                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Cell 1 (US Region)        Cell 2 (EU Region)                     │
│   ┌───────────────────┐     ┌───────────────────┐                  │
│   │  ┌─────────────┐  │     │  ┌─────────────┐  │                  │
│   │  │ App Server  │  │     │  │ App Server  │  │                  │
│   │  └──────┬──────┘  │     │  └──────┬──────┘  │                  │
│   │         │         │     │         │         │                  │
│   │  ┌──────▼──────┐  │     │  ┌──────▼──────┐  │                  │
│   │  │  Database   │  │     │  │  Database   │  │                  │
│   │  │ (Tenants    │  │     │  │ (Tenants    │  │                  │
│   │  │  A, B, C)   │  │     │  │  X, Y, Z)   │  │                  │
│   │  └─────────────┘  │     │  └─────────────┘  │                  │
│   └───────────────────┘     └───────────────────┘                  │
│                                                                     │
│   Cell 간 독립 운영 → 장애 격리 (Fault Isolation)                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### 멀티테넌시 샤딩의 이점

| 이점 | 설명 |
|------|------|
| **리소스 격리** | 한 테넌트의 부하가 다른 테넌트에 영향 X |
| **권한 격리** | 접근 제어 버그로 인한 데이터 유출 방지 |
| **장애 격리** | Cell-based 아키텍처로 장애 범위 제한 |
| **개별 백업/복원** | 테넌트별 독립적인 백업 및 복원 가능 |
| **규정 준수** | GDPR 등 데이터 삭제 요청 시 샤드 단위 처리 |
| **데이터 거주지** | 특정 지역에 데이터 저장 (Data Residency) |
| **점진적 스키마 롤아웃** | 테넌트별로 스키마 마이그레이션 점진 적용 |

### 4. 키-값 데이터의 샤딩 전략

#### 4.1 키 범위 샤딩 (Key Range Sharding)

```
┌─────────────────────────────────────────────────────────────────────┐
│                 키 범위 샤딩 (백과사전 비유)                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Vol.1      Vol.2      Vol.3      Vol.4      Vol.5                │
│   ┌────┐     ┌────┐     ┌────┐     ┌────┐     ┌────┐               │
│   │A-C │     │D-F │     │G-L │     │M-R │     │S-Z │               │
│   │    │     │    │     │    │     │    │     │    │               │
│   │ 📖 │     │ 📖 │     │ 📖 │     │ 📖 │     │ 📖 │               │
│   └────┘     └────┘     └────┘     └────┘     └────┘               │
│                                                                     │
│   키의 연속 범위를 각 샤드에 할당                                      │
│   장점: 범위 쿼리 효율적                                              │
│   단점: 연속 키 쓰기 시 핫스팟 발생 가능                               │
└─────────────────────────────────────────────────────────────────────┘
```

**장점**:
- 범위 스캔(Range Scan) 효율적
- 샤드 내 키가 정렬되어 있어 연속 데이터 조회 빠름
- 샤드 수가 데이터 볼륨에 따라 자동 조정

**단점**:
- 연속 키(예: 타임스탬프) 쓰기 시 핫스팟 발생
- 샤드 분할(Split)이 비용이 큼

**핫스팟 해결 예시**:
```
❌ 문제: timestamp를 키로 사용
   → 현재 월의 샤드에 모든 쓰기 집중

✅ 해결: sensor_id + timestamp 복합 키
   → 센서별로 쓰기가 분산됨
   → 단점: 시간 범위 조회 시 센서별로 쿼리 필요
```

#### 4.2 해시 샤딩 (Hash Sharding)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    해시 함수를 통한 균등 분산                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Input Keys           Hash Function           Output (0-65535)    │
│   ┌──────────┐         ┌──────────┐           ┌──────────┐        │
│   │ "user_1" │ ──────▶ │          │ ──────▶   │  12847   │        │
│   │ "user_2" │ ──────▶ │  hash()  │ ──────▶   │  45231   │        │
│   │ "user_3" │ ──────▶ │          │ ──────▶   │  8924    │        │
│   └──────────┘         └──────────┘           └──────────┘        │
│                                                                     │
│   비슷한 입력 → 균등하게 분산된 출력                                   │
│                                                                     │
│   Shard 0: hash 0-16383                                            │
│   Shard 1: hash 16384-32767                                        │
│   Shard 2: hash 32768-49151                                        │
│   Shard 3: hash 49152-65535                                        │
└─────────────────────────────────────────────────────────────────────┘
```

**사용되는 해시 함수**:
- MongoDB: MD5
- Cassandra, ScyllaDB: Murmur3
- 주의: Java `hashCode()`, Ruby `Object#hash`는 프로세스마다 다른 값 → 샤딩 부적합

### 5. 해시 샤딩의 세 가지 방식

#### 5.1 Mod N 방식 (비추천)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    hash(key) % N 의 문제점                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Before (N=3)                    After (N=4)                       │
│   ┌─────────────────────┐         ┌─────────────────────┐          │
│   │ hash % 3            │         │ hash % 4            │          │
│   │                     │         │                     │          │
│   │ hash=0 → Node 0     │         │ hash=0 → Node 0 ✓   │          │
│   │ hash=1 → Node 1     │         │ hash=1 → Node 1 ✓   │          │
│   │ hash=2 → Node 2     │         │ hash=2 → Node 2 ✓   │          │
│   │ hash=3 → Node 0     │   ──▶   │ hash=3 → Node 3 ✗   │  이동!   │
│   │ hash=4 → Node 1     │         │ hash=4 → Node 0 ✗   │  이동!   │
│   │ hash=5 → Node 2     │         │ hash=5 → Node 1 ✗   │  이동!   │
│   │ hash=6 → Node 0     │         │ hash=6 → Node 2 ✗   │  이동!   │
│   │ hash=7 → Node 1     │         │ hash=7 → Node 3 ✗   │  이동!   │
│   │ hash=8 → Node 2     │         │ hash=8 → Node 0 ✗   │  이동!   │
│   │ hash=9 → Node 0     │         │ hash=9 → Node 1 ✗   │  이동!   │
│   └─────────────────────┘         └─────────────────────┘          │
│                                                                     │
│   노드 1개 추가 시 대부분의 키가 이동해야 함! → 비효율적                 │
└─────────────────────────────────────────────────────────────────────┘
```

#### 5.2 고정 샤드 수 방식 (Fixed Number of Shards)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    고정 샤드 수 방식                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   1000개 샤드를 미리 생성, 10개 노드에 분배                            │
│                                                                     │
│   Before (10 nodes)                After (11 nodes)                 │
│   ┌────────────────────┐           ┌────────────────────┐          │
│   │ Node 0: 100 shards │           │ Node 0: 91 shards  │ ← 9 이동 │
│   │ Node 1: 100 shards │           │ Node 1: 91 shards  │ ← 9 이동 │
│   │ Node 2: 100 shards │           │ Node 2: 91 shards  │ ← 9 이동 │
│   │ ...                │   ──▶     │ ...                │          │
│   │ Node 9: 100 shards │           │ Node 9: 91 shards  │ ← 9 이동 │
│   └────────────────────┘           │ Node 10: 90 shards │ ← 새 노드│
│                                    └────────────────────┘          │
│                                                                     │
│   key → shard = hash(key) % 1000 (변하지 않음)                       │
│   변하는 것: shard → node 매핑만 변경                                 │
│                                                                     │
│   사용: Citus, Riak, Elasticsearch, Couchbase                       │
└─────────────────────────────────────────────────────────────────────┘
```

**장점**: 리밸런싱 시 샤드 단위로 이동 (분할 불필요)
**단점**: 샤드 수를 초기에 잘못 설정하면 재샤딩 비용 큼

#### 5.3 해시 범위 샤딩 (Hash Range Sharding)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    해시 범위 샤딩                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Hash Space (16-bit: 0-65535)                                     │
│   ├────────────────────────────────────────────────────────────┤   │
│   0        16384      32768      49152      65535              │   │
│   │         │          │          │          │                 │   │
│   │ Shard 0 │ Shard 1  │ Shard 2  │ Shard 3  │                 │   │
│                                                                     │
│   필요 시 샤드 분할 가능:                                             │
│                                                                     │
│   Split Shard 1:                                                    │
│   ├─────────────────────────────────────────────────────────────┤  │
│   0     16384   24576   32768      49152      65535                │
│   │       │       │       │          │          │                  │
│   │Shard 0│Shard1a│Shard1b│ Shard 2  │ Shard 3  │                  │
│                                                                     │
│   사용: YugabyteDB, DynamoDB, MongoDB                               │
└─────────────────────────────────────────────────────────────────────┘
```

### 6. 일관된 해싱 (Consistent Hashing)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    일관된 해싱의 특성                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   두 가지 핵심 속성:                                                  │
│   1. 각 샤드에 키가 대략 균등하게 분배                                  │
│   2. 샤드 수 변경 시 최소한의 키만 이동                                 │
│                                                                     │
│   Cassandra/ScyllaDB 방식:                                          │
│                                                                     │
│   Hash Ring (0-1023)                                               │
│              0                                                      │
│          ╱       ╲                                                  │
│        900        100                                               │
│       ╱              ╲                                              │
│      │    Node 1      │                                             │
│      │   (0-339)      │                                             │
│     800              200                                            │
│      │                │                                             │
│      │    Node 2      │                                             │
│      │  (340-679)     │                                             │
│     700              300                                            │
│       ╲              ╱                                              │
│        600        400                                               │
│          ╲   │   ╱                                                  │
│           500│                                                      │
│              │                                                      │
│         Node 3                                                      │
│        (680-1023)                                                   │
│                                                                     │
│   알고리즘 종류:                                                      │
│   - Cassandra/ScyllaDB: 랜덤 경계를 가진 해시 범위                     │
│   - Rendezvous Hashing (HRW): 가장 높은 점수의 노드 선택               │
│   - Jump Consistent Hash: 빠른 계산, 균등 분배                        │
└─────────────────────────────────────────────────────────────────────┘
```

### 7. 핫스팟과 스큐 문제

```
┌─────────────────────────────────────────────────────────────────────┐
│                    핫스팟 (Hot Spot) 문제                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   정상 상태:                    스큐 발생:                            │
│   ┌─────┐ ┌─────┐ ┌─────┐     ┌─────┐ ┌─────┐ ┌─────┐             │
│   │ 33% │ │ 33% │ │ 33% │     │ 10% │ │ 10% │ │ 80% │ ← Hot!      │
│   │     │ │     │ │     │     │     │ │     │ │ 🔥  │             │
│   └─────┘ └─────┘ └─────┘     └─────┘ └─────┘ └─────┘             │

```

#### 자동 리밸런싱의 위험한 시나리오

```
Timeline:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[T0] Node A 과부하 → 응답 지연
      │
[T1]  ├─▶ 다른 노드: "Node A가 죽었나?" (잘못된 판단)
      │
[T2]  ├─▶ 자동 리밸런싱 시작
      │   Node A의 샤드들을 다른 노드로 이동
      │
[T3]  ├─▶ 네트워크 부하 증가
      │   다른 노드들도 느려짐
      │
[T4]  ├─▶ 추가 노드들도 "죽은 것으로" 판단됨
      │
[T5]  └─▶ 캐스케이드 장애 발생! 🔥
          (Cascading Failure)
```

### 10. 보조 인덱스와 샤딩

#### 10.1 로컬 보조 인덱스 (Local Secondary Index)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    로컬 보조 인덱스 (Document-Partitioned)            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Shard 0 (IDs 0-499)              Shard 1 (IDs 500-999)           │
│   ┌───────────────────────┐        ┌───────────────────────┐       │
│   │ Primary Data:         │        │ Primary Data:         │       │
│   │ ID:191 color:red      │        │ ID:768 color:red      │       │
│   │ ID:306 color:black    │        │ ID:891 color:black    │       │
│   │                       │        │                       │       │
│   │ Local Index:          │        │ Local Index:          │       │
│   │ color:red   → [191]   │        │ color:red   → [768]   │       │
│   │ color:black → [306]   │        │ color:black → [891]   │       │
│   └───────────────────────┘        └───────────────────────┘       │
│                                                                     │
│   "color = red" 검색 시:                                            │
│   ┌─────────┐                                                      │
│   │ Client  │─────┬─────────▶ Shard 0 → [191]                      │
│   └─────────┘     │                                                │
│                   └─────────▶ Shard 1 → [768]                      │
│                                                                     │
│   결과 병합: [191, 768]                                             │
│                                                                     │
│   장점: 쓰기 시 단일 샤드만 업데이트                                   │
│   단점: 읽기 시 모든 샤드 조회 필요 (Scatter-Gather)                   │
│                                                                     │
│   사용: MongoDB, Riak, Cassandra, Elasticsearch, VoltDB             │
└─────────────────────────────────────────────────────────────────────┘
```

#### 10.2 글로벌 보조 인덱스 (Global Secondary Index)

```
┌─────────────────────────────────────────────────────────────────────┐
│                    글로벌 보조 인덱스 (Term-Partitioned)              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Primary Key Index                 Global Secondary Index          │
│   ┌─────────────────┐               ┌─────────────────┐            │
│   │ Shard 0         │               │ Index Shard 0   │            │
│   │ IDs 0-499       │               │ (colors a-r)    │            │
│   │                 │               │                 │            │
│   │ ID:191 red      │               │ color:red →     │            │
│   │ ID:306 black    │               │   [191, 768]    │            │
│   ├─────────────────┤               │ color:black →   │            │
│   │ Shard 1         │               │   [306, 891]    │            │
│   │ IDs 500-999     │               ├─────────────────┤            │
│   │                 │               │ Index Shard 1   │            │
│   │ ID:768 red      │               │ (colors s-z)    │            │
│   │ ID:891 black    │               │                 │            │
│   └─────────────────┘               │ color:silver →  │            │
│                                     │   [...]         │            │
│                                     └─────────────────┘            │
│                                                                     │
│   "color = red" 검색:                                               │
│   Client → Index Shard 0 → [191, 768] (단일 샤드 조회!)             │
│                                                                     │
│   장점: 읽기 시 단일 인덱스 샤드만 조회                                │
│   단점: 쓰기 시 여러 인덱스 샤드 업데이트 필요                          │
│         (분산 트랜잭션 필요 또는 비동기 반영)                          │
│                                                                     │
│   사용: CockroachDB, TiDB, YugabyteDB, DynamoDB                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### 로컬 vs 글로벌 인덱스 비교

| 특성 | 로컬 인덱스 | 글로벌 인덱스 |
|------|-------------|---------------|
| **쓰기** | 단일 샤드 (빠름) | 여러 샤드 (느림) |
| **읽기** | 모든 샤드 조회 (Scatter) | 단일 샤드 조회 (빠름) |
| **일관성** | 즉시 일관성 | 비동기 시 지연 가능 |
| **복잡성** | 단순 | 분산 트랜잭션 필요 |
| **적합** | 쓰기 > 읽기 | 읽기 > 쓰기 |

### 기술 면접 예상 질문

**Q1: 키 범위 샤딩과 해시 샤딩의 차이점과 각각의 적합한 사용 사례는?**
```
키 범위 샤딩:
- 연속 키를 같은 샤드에 저장
- 범위 쿼리 효율적 (예: 시계열 데이터)
- 핫스팟 위험 (연속 쓰기 시)

해시 샤딩:
- 해시값 기반 균등 분산
- 핫스팟 방지
- 범위 쿼리 비효율적

선택 기준:
- 범위 쿼리 필요 → 키 범위
- 균등 분산 필요 → 해시
- 둘 다 필요 → 해시 범위 + 복합 키
```

**Q2: 일관된 해싱(Consistent Hashing)이 필요한 이유와 동작 원리는?**
```
필요한 이유:
- mod N 방식은 노드 추가/제거 시 대부분의 키 이동
- 일관된 해싱은 최소한의 키만 이동

동작 원리:
1. 해시 공간을 원(ring)으로 표현
2. 각 노드와 키를 해시하여 링에 배치
3. 키는 시계 방향으로 가장 가까운 노드에 할당
4. 노드 추가/제거 시 인접 노드만 영향

변형:
- Virtual nodes: 불균형 해결
- Jump consistent hash: 더 균등한 분배
```

**Q3: 로컬 보조 인덱스와 글로벌 보조 인덱스의 트레이드오프는?**
```
로컬 보조 인덱스:
- 쓰기: 빠름 (단일 샤드)
- 읽기: 느림 (모든 샤드 Scatter-Gather)
- 일관성: 즉시

글로벌 보조 인덱스:
- 쓰기: 느림 (여러 샤드, 분산 트랜잭션)
- 읽기: 빠름 (단일 샤드)
- 일관성: 비동기 시 지연 가능

선택 기준:
- 읽기 >> 쓰기: 글로벌
- 쓰기 >> 읽기: 로컬
- 강한 일관성 필요: 로컬 (또는 글로벌 + 분산 트랜잭션)
```

**Q4: 샤딩된 데이터베이스에서 핫스팟이 발생했을 때 해결 방법은?**
```
1. 핫키 분산:
   - 랜덤 접미사 추가 (예: key_00 ~ key_99)
   - 쓰기 분산, 읽기 시 병합 필요

2. 전용 샤드 할당:
   - 핫키만을 위한 별도 샤드/노드 할당

3. 캐싱:
   - 읽기 핫스팟: Redis 등 캐시 레이어 추가

4. 애플리케이션 레벨 샤딩:
   - 핫키 목록 관리
   - 동적 샤딩 정책 적용

5. 클라우드 서비스:
   - DynamoDB Adaptive Capacity
   - 자동 핫샤드 분할
```
