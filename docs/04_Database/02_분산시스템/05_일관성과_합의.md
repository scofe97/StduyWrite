# Chapter 10: Consistency and Consensus

---

## 📌 핵심 요약
> 분산 시스템에서 **강한 일관성(Strong Consistency)**을 제공하려면 **선형성(Linearizability)**이라는 명확한 정의가 필요하다. 선형성은 "데이터가 하나만 존재하고, 모든 연산이 원자적으로 실행되는 것처럼 보이게 하는 것"이다. 이를 내결함성 있게 구현하려면 **합의 알고리즘(Consensus)**이 필요하다. Raft, Paxos, Zab 같은 합의 알고리즘은 노드 장애에도 불구하고 모든 노드가 동일한 결정에 도달하도록 보장한다.

## 📖 본문 정리

### 1. 일관성의 두 가지 철학

```
┌─────────────────────────────────────────────────────────────────┐
│                    최종 일관성 (Eventual Consistency)            │
├─────────────────────────────────────────────────────────────────┤
│  • 복제본이 여러 개라는 사실이 애플리케이션에 노출됨              │
│  • 불일치와 충돌을 애플리케이션이 직접 처리                       │
│  • Multi-leader, Leaderless 복제에서 주로 사용                   │
│  • 오프라인 앱, 지역 분산 시스템에 적합                          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    강한 일관성 (Strong Consistency)              │
├─────────────────────────────────────────────────────────────────┤
│  • 시스템이 단일 노드처럼 동작                                   │
│  • 애플리케이션이 복제를 신경 쓸 필요 없음                       │
│  • 성능 비용 발생, 일부 장애에 더 취약                           │
│  • 빠르고 안정적인 네트워크 환경에 적합                          │
└─────────────────────────────────────────────────────────────────┘
```

### 2. 선형성 (Linearizability)

#### 정의와 핵심 개념

> **선형성**: 데이터의 복사본이 **하나만 존재**하고, 모든 연산이 **원자적**으로 실행되는 것처럼 보이게 하는 것

다른 이름들:
- Atomic Consistency
- Strong Consistency
- Immediate Consistency
- External Consistency

```
┌─────────────────────────────────────────────────────────────────┐
│                    선형성의 핵심: 최신성 보장 (Recency)          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   한 클라이언트가 쓰기를 완료하면,                               │
│   이후 모든 클라이언트는 반드시 그 값을 읽을 수 있어야 함        │
│                                                                 │
│   → 오래된 캐시나 지연된 복제본에서 읽으면 안 됨                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 비선형 시스템 예시 (스포츠 웹사이트)

```
┌─────────────────────────────────────────────────────────────────┐
│   Aaliyah와 Bryce가 같은 방에서 경기 결과 확인                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Aaliyah ──────→ [최신 복제본] ──→ "경기 종료! A팀 승리!"      │
│       │                                                         │
│       │ "우리 팀 이겼어!"                                       │
│       ↓                                                         │
│   Bryce ────────→ [지연된 복제본] ──→ "경기 진행 중..."         │
│                                                                 │
│   Bryce는 Aaliyah가 말한 후에 새로고침했으므로,                  │
│   적어도 Aaliyah만큼 최신 결과를 기대함                          │
│   → 선형성 위반! ❌                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 마치 두 사람이 같은 게시판을 보는데, 한 사람은 업데이트된 공지를 보고 다른 사람에게 알려줬는데, 그 사람이 게시판을 보니 아직 이전 공지가 있는 상황.

#### 선형 시스템의 요구사항

```
┌─────────────────────────────────────────────────────────────────┐
│                    선형성 판정 규칙                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   연산 유형:                                                     │
│   • read(x) ⇒ v : 레지스터 x를 읽어서 값 v 반환                 │
│   • write(x, v) ⇒ r : 레지스터 x에 값 v 쓰기, 응답 r 반환       │
│   • cas(x, v_old, v_new) ⇒ r : Compare-and-Set                  │
│                                                                 │
│   규칙:                                                          │
│   1. 쓰기 전에 완료된 읽기 → 반드시 이전 값 반환                │
│   2. 쓰기 후에 시작된 읽기 → 반드시 새 값 반환                  │
│   3. 쓰기와 동시에 실행되는 읽기 → 이전 또는 새 값 가능         │
│                                                                 │
│   핵심 추가 규칙:                                                │
│   어떤 읽기가 새 값을 반환하면,                                  │
│   이후 모든 읽기(같은 클라이언트든 다른 클라이언트든)도          │
│   반드시 새 값을 반환해야 함                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**시각화:**

```
시간 ──────────────────────────────────────────────────────────→

Client C: ════════ write(x, 1) ════════
                        │
                   (원자적 전환 시점)
                        ↓
Client A: ══ read() ══     ══ read() ══
              ↓                 ↓
             0 (이전)          1 (새값)
                               ↑
                    이 시점 이후 모든 읽기는 1을 반환해야 함

Client B:        ══ read() ══
                      ↓
                     1 (A가 1을 읽은 후에 시작했으므로)
```

#### Linearizability vs Serializability

| 특성 | Linearizability | Serializability |
|------|-----------------|-----------------|
| **대상** | 단일 객체(레지스터)의 읽기/쓰기 | 여러 객체에 대한 트랜잭션 |
| **보장** | **최신성**: 이후 연산은 이전 연산 결과를 반영 | **격리**: 트랜잭션이 순차 실행된 것처럼 동작 |
| **Write Skew** | 방지 못함 (단일 객체 보장) | 방지 가능 (다중 객체 트랜잭션) |
| **Stale Read** | 허용 안 함 | 허용 가능 (순서만 맞으면 됨) |

**Strict Serializability (Strong-1SR)**:
- Serializability + Linearizability 모두 제공
- 예: Spanner, FoundationDB

```
┌─────────────────────────────────────────────────────────────────┐
│                    혼동 주의!                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Sequential Consistency: 또 다른 개념 (여기서 다루지 않음)      │
│                                                                 │
│   CockroachDB: Serializability ✓, Strict Serializability ✗     │
│   Spanner: Strict Serializability ✓                             │
│   FoundationDB: Strict Serializability ✓                        │
│                                                                 │
│   일관성 모델과 격리 수준은 대체로 독립적으로 선택 가능          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 선형성이 필요한 경우

**1. 잠금과 리더 선출**

```
┌─────────────────────────────────────────────────────────────────┐
│                    리스(Lease) 기반 리더 선출                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Node A ─────────┐                                             │
│   Node B ─────────┼──→ [Lease 서비스] ──→ 하나만 성공           │
│   Node C ─────────┘                                             │
│                                                                 │
│   요구사항: 두 노드가 동시에 리스를 획득하면 안 됨               │
│   → 선형성 필요!                                                 │
│                                                                 │
│   사용 예: ZooKeeper, etcd                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**2. 제약 조건과 유일성 보장**

```
┌─────────────────────────────────────────────────────────────────┐
│                    유일성 제약 조건                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   • 사용자명/이메일 고유성                                       │
│   • 파일 경로 고유성                                             │
│   • 은행 잔고가 음수가 되지 않도록                               │
│   • 재고보다 많이 판매하지 않도록                                │
│   • 같은 좌석 중복 예약 방지                                     │
│                                                                 │
│   → 모든 노드가 동일한 최신 값에 동의해야 함                    │
│   → 선형성 또는 그에 준하는 보장 필요                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**3. 크로스 채널 타이밍 의존성**

```
┌─────────────────────────────────────────────────────────────────┐
│                    비디오 트랜스코딩 예시                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Web Server                                                    │
│       │                                                         │
│       ├──① 비디오 업로드 ──→ [File Storage]                    │
│       │                           │                             │
│       └──② 트랜스코딩 요청 ──→ [Message Queue]                 │
│                                    │                             │
│   Transcoder ←────③ 메시지 수신 ───┘                            │
│       │                                                         │
│       └──④ 비디오 가져오기 ──→ [File Storage]                  │
│                                                                 │
│   문제: 메시지 큐(②→③)가 파일 복제(①→④)보다 빠르면             │
│         트랜스코더가 이전 버전 또는 빈 파일을 가져올 수 있음     │
│                                                                 │
│   해결: File Storage가 선형성을 제공하면 문제 없음               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **비유**: 택배 추적 알림이 실제 배송보다 빨리 도착하는 상황. "배송 완료" 알림을 받았는데 문 앞에 택배가 없다.

#### 선형 시스템 구현

```
┌─────────────────────────────────────────────────────────────────┐
│                    복제 방식별 선형성                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Single-Leader Replication:                                    │
│   ├─ 리더에서 읽기/쓰기 → 잠재적으로 선형화 가능 ✓              │
│   ├─ 주의: 리더가 맞는지 확실해야 함 (split brain 문제)         │
│   └─ 비동기 복제 → 페일오버 시 데이터 손실 가능 ✗               │
│                                                                 │
│   Consensus Algorithms (Raft, Paxos, Zab):                      │
│   ├─ 자동 리더 선출과 페일오버                                   │
│   ├─ Split brain 방지 설계                                      │
│   └─ 선형화 가능 ✓ (단, 읽기도 쿼럼 확인 필요)                  │
│                                                                 │
│   Multi-Leader Replication:                                     │
│   └─ 일반적으로 선형화 불가 ✗ (비동기 복제, 충돌)               │
│                                                                 │
│   Leaderless Replication:                                       │
│   ├─ 쿼럼 읽기/쓰기로도 선형화 보장 안 됨 ✗                     │
│   └─ LWW 타임스탬프 기반 → 클럭 스큐로 비선형화                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Leaderless에서의 비선형 실행 예시:**

```
┌─────────────────────────────────────────────────────────────────┐
│   w + r > n 쿼럼 조건을 만족해도 비선형 가능                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Writer: x=0 → x=1 쓰기 중                                     │
│                                                                 │
│   Replica 1: [x=0 → x=1]                                        │
│   Replica 2: [x=0 → x=1]                                        │
│   Replica 3: [x=0 ─────]                                        │
│                                                                 │
│   Client A (r=2): Replica 1(x=1) + Replica 3(x=0) → x=1 반환   │
│   Client B (r=2): Replica 2(x=0) + Replica 3(x=0) → x=0 반환   │
│                                                                 │
│   B의 요청이 A의 완료 후에 시작됐는데 더 오래된 값 반환!        │
│   → 선형성 위반 ❌                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 선형성의 비용 (The Cost of Linearizability)

#### 네트워크 파티션과 가용성

```
┌─────────────────────────────────────────────────────────────────┐
│                    Multi-Region 배포 시나리오                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [Region A]                    [Region B]                      │
│   ┌─────────┐      ✗ 네트워크    ┌─────────┐                   │
│   │ Leader  │  ←── 파티션 ───→  │Follower │                    │
│   │Database │                    │Database │                    │
│   └─────────┘                    └─────────┘                    │
│       ↑                              ↑                          │
│   [Client A]                    [Client B]                      │
│                                                                 │
│   Multi-Leader: 각 리전이 독립적으로 쓰기 가능 → 가용성 ✓       │
│   Single-Leader: Region B는 리더에 접근 불가 → 가용성 ✗        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### CAP 정리

```
┌─────────────────────────────────────────────────────────────────┐
│                    CAP 정리의 실제 의미                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   잘못된 해석: "C, A, P 중 2개 선택"                            │
│                                                                 │
│   올바른 해석:                                                   │
│   - 네트워크 파티션(P)은 피할 수 없는 현실                      │
│   - 파티션 발생 시, C(선형성) 또는 A(가용성) 중 선택            │
│                                                                 │
│   CP: 파티션 시 일부 노드가 요청 거부 (일관성 유지)             │
│   AP: 파티션 시에도 모든 노드가 요청 처리 (비선형)              │
│                                                                 │
│   더 나은 표현: "파티션 시 일관성 또는 가용성"                   │
│   (Either Consistent or Available when Partitioned)             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**CAP의 한계:**

| 문제점 | 설명 |
|--------|------|
| 좁은 범위 | 선형성만 고려, 약한 일관성 모델 미포함 |
| 비현실적 가용성 정의 | 실제 고가용성 시스템과 다름 |
| 파티션만 고려 | 네트워크 지연, 노드 장애 등 미포함 |
| 이분법적 | CP도 AP도 아닌 시스템 설명 불가 |

**PACELC 원칙:**
```
P(Partition) 시:  A(Availability) vs C(Consistency) 선택
E(Else, 정상):    L(Latency) vs C(Consistency) 선택
```

#### 선형성과 네트워크 지연

```
┌─────────────────────────────────────────────────────────────────┐
│                    CPU 메모리도 비선형!                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   CPU Core 1               CPU Core 2                           │
│   ┌─────────┐             ┌─────────┐                          │
│   │ Cache   │             │ Cache   │                          │
│   └────┬────┘             └────┬────┘                          │
│        │                       │                                │
│        └───────────┬───────────┘                                │
│                    ↓                                             │
│              [Main Memory]                                       │
│                                                                 │
│   Core 1이 메모리에 쓰기 → Core 2가 즉시 읽지 못할 수 있음      │
│   (Memory Barrier/Fence 없으면)                                  │
│                                                                 │
│   이유: 캐시로 인한 여러 복사본 + 비동기 업데이트                │
│   목적: 성능! (CAP가 아님)                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Attiya-Welch 증명:**
> 선형성을 원하면, 읽기/쓰기 응답 시간은 **네트워크 지연의 불확실성에 비례**할 수밖에 없다.

→ 지연에 민감한 시스템에서는 약한 일관성 모델이 유리할 수 있음

### 4. ID 생성기와 논리 클럭 (ID Generators and Logical Clocks)

#### 분산 ID 생성의 도전

```
┌─────────────────────────────────────────────────────────────────┐
│                    단일 노드 Auto-Increment                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   장점:                                                          │
│   ✅ 컴팩트 (64비트)                                             │
│   ✅ ID 순서 = 생성 순서 (인과성 보장)                          │
│   ✅ 구현 간단 (CPU atomic increment)                           │
│                                                                 │
│   문제:                                                          │
│   ❌ 단일 장애점 (SPOF)                                          │
│   ❌ 다른 리전에서 ID 가져오기 느림                              │
│   ❌ 높은 쓰기 처리량에서 병목                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 분산 ID 생성 방식 비교

| 방식 | 장점 | 단점 |
|------|------|------|
| **Sharded ID** | 확장성 | 순서 보장 안 됨 (짝수/홀수) |
| **Block Allocation** | 중앙 부하 감소 | 순서 보장 안 됨 |
| **UUID v4** | 로컬 생성, 통신 불필요 | 128비트, 순서 무작위 |
| **Wall-clock 기반** | 대략적 순서 | 클럭 스큐로 순서 오류 |

**Wall-clock 기반 ID 예시:**
- UUID v7
- Twitter Snowflake
- ULID
- MongoDB ObjectID

```
┌─────────────────────────────────────────────────────────────────┐
│   Snowflake 형식 (64비트)                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌───────────────┬──────────┬────────────┐                    │
│   │  Timestamp    │ Machine  │  Sequence  │                    │
│   │  (41 bits)    │ (10 bits)│  (12 bits) │                    │
│   └───────────────┴──────────┴────────────┘                    │
│                                                                 │
│   → 대략적인 시간순 정렬 가능                                   │
│   → 하지만 클럭 스큐로 인해 선형성 보장 안 됨                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Lamport 타임스탬프

```
┌─────────────────────────────────────────────────────────────────┐
│                    Lamport 클럭 알고리즘                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   타임스탬프 = (counter, node_id)                               │
│                                                                 │
│   규칙:                                                          │
│   1. 이벤트 발생 시: counter++ 후 사용                          │
│   2. 메시지 수신 시: local_counter = max(local, received) + 1   │
│                                                                 │
│   비교:                                                          │
│   1. counter 값 비교                                             │
│   2. counter 같으면 node_id로 타이브레이커                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**예시:**

```
Aaliyah (counter=0)    Caleb (counter=0)    Bryce (counter=0)
        │                     │                    │
        │ ─── "질문" ───────────────────────────→ │
        │ counter=1                                │
        │                     │ ─── "답변" ──────→ │
        │                     │ counter=1          │
        │                     │                    │
        │                     │                    │ counter = max(1,1) + 1 = 2
        │                     │                    │ ← "응답"
        │                     │                    │

순서: (1, "Aaliyah") < (1, "Caleb") < (2, "Bryce")
```

**Lamport 클럭의 특성:**
- ✅ 인과성 보장: A가 B 전에 발생 → A의 타임스탬프 < B의 타임스탬프
- ❌ 선형성 미보장: 서로 모르는 두 이벤트의 실제 순서는 알 수 없음
- ❌ 물리적 시간과 무관

#### 하이브리드 논리 클럭 (HLC)

```
┌─────────────────────────────────────────────────────────────────┐
│              Hybrid Logical Clock                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   물리 클럭 + Lamport 클럭의 장점 결합                          │
│                                                                 │
│   특성:                                                          │
│   ✅ 초/마이크로초 단위 시간 값 유지                             │
│   ✅ Lamport처럼 다른 노드의 더 큰 값 수용                       │
│   ✅ 매 타임스탬프 생성 시 증가 (단조 증가 보장)                │
│   ✅ NTP 조정으로 물리 클럭이 뒤로 점프해도 안전                 │
│                                                                 │
│   결과: happens-before 관계와 일치하는 대략적 물리 시간         │
│                                                                 │
│   사용 예: CockroachDB                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Lamport/HLC vs Vector Clock

| 특성 | Lamport/HLC | Vector Clock |
|------|-------------|--------------|
| 크기 | 작음 (하나의 카운터) | 큼 (노드당 하나의 카운터) |
| 동시성 감지 | 불가 | 가능 |
| 인과성 보장 | ✅ | ✅ |
| 용도 | 스냅샷 격리 트랜잭션 ID | 동시 쓰기 감지 |

#### 선형화 가능한 ID 생성기

**필요한 이유 - SNS 개인정보 시나리오:**

```
┌─────────────────────────────────────────────────────────────────┐
│   User A: 계정을 비공개로 설정 → 민감한 사진 업로드             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [Laptop]                        [Phone]                       │
│      │                               │                          │
│      │ ①"비공개 설정"                │                          │
│      └──→ [Accounts DB]             │                          │
│             TS = 100                 │                          │
│                                      │ ②"사진 업로드"           │
│                                      └──→ [Photos DB]           │
│                                             TS = 99 (느린 클럭!) │
│                                                                 │
│   Viewer (비친구)의 읽기 (TS=99.5):                             │
│   • 계정: 공개로 보임 (100 > 99.5)                              │
│   • 사진: 보임 (99 < 99.5)                                      │
│   → 비공개로 설정한 사진이 노출! 💀                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**해결책: 선형화 가능한 ID 생성기**

```
구현 방법:
1. 단일 노드 카운터 + 영속화 + 복제 (Single-Leader)
2. 배치 ID 할당으로 최적화 (디스크 쓰기 감소)
3. Spanner 방식: TrueTime 신뢰 구간만큼 대기

예: TiDB/TiKV의 Timestamp Oracle
```

**제약:**
- 샤딩 불가 (순서 보장 못함)
- 다중 리전 분산 어려움
- 병목 가능성 (단, 작업이 단순하므로 높은 처리량 가능)

### 5. 합의 (Consensus)

#### 합의가 필요한 상황

```
┌─────────────────────────────────────────────────────────────────┐
│   단일 노드에서는 쉽지만, 내결함성을 원하면 어려운 것들          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 선형화 가능한 데이터베이스                                  │
│      • 단일 리더에서 읽기/쓰기 → 리더 장애 시 페일오버?        │
│      • Split brain 방지는?                                      │
│                                                                 │
│   2. 선형화 가능한 ID 생성기                                     │
│      • 단일 노드 카운터 → 그 노드가 크래시하면?                 │
│                                                                 │
│   3. Compare-and-Set (CAS)                                      │
│      • 락/리스 획득 경쟁 해결                                   │
│      • 파일/사용자 유일성 보장                                  │
│      • 단일 노드에서는 CPU 명령 하나 → 내결함성?                │
│                                                                 │
│   → 모두 합의(Consensus) 문제로 귀결됨                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 주요 합의 알고리즘

| 알고리즘 | 특징 |
|----------|------|
| **Viewstamped Replication** | 1988년, 원조 |
| **Paxos** | 1998년, 이론적 기초 |
| **Raft** | 2014년, 이해하기 쉬움 |
| **Zab** | ZooKeeper에서 사용 |

모두 **non-Byzantine 모델**: 네트워크 지연/드롭, 노드 크래시/재시작은 허용하지만 악의적 행동은 가정 안 함

#### FLP 불가능성 결과

```
┌─────────────────────────────────────────────────────────────────┐
│   FLP 결과 (Fischer, Lynch, Paterson, 1985)                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   "노드가 크래시할 수 있는 환경에서                              │
│    항상 합의에 도달하는 알고리즘은 존재하지 않는다"              │
│                                                                 │
│   그런데 우리는 합의 알고리즘을 사용하고 있다?                   │
│                                                                 │
│   해석:                                                          │
│   • FLP는 "항상 종료됨"을 보장하지 못한다는 것                  │
│   • 비동기 모델 + 결정적 알고리즘 가정                          │
│   • 타임아웃을 사용하면 (부분 동기 모델) 해결 가능               │
│   • 난수를 사용해도 해결 가능                                   │
│                                                                 │
│   실제로는 대부분의 경우 합의에 도달함                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 합의의 다양한 표현

```
┌─────────────────────────────────────────────────────────────────┐
│   놀라운 사실: 다음 문제들은 모두 동등하다!                      │
│   (하나의 해법을 다른 것으로 변환 가능)                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 단일 값 합의 (Single-Value Consensus)                      │
│   2. Compare-and-Set (CAS)                                      │
│   3. 공유 로그 (Total Order Broadcast)                          │
│   4. 원자적 커밋 (Atomic Commitment)                            │
│   5. Fetch-and-Add (2노드 한정)                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 단일 값 합의 (Single-Value Consensus)

```
┌─────────────────────────────────────────────────────────────────┐
│                    합의 속성                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. Uniform Agreement (균등 합의)                              │
│      두 노드가 다른 값을 결정할 수 없음                         │
│                                                                 │
│   2. Integrity (무결성)                                         │
│      한번 결정하면 마음을 바꿀 수 없음                          │
│                                                                 │
│   3. Validity (유효성)                                          │
│      결정된 값은 누군가가 제안한 값이어야 함                    │
│      (항상 null을 결정하는 알고리즘 방지)                       │
│                                                                 │
│   4. Termination (종료)                                         │
│      크래시하지 않은 모든 노드는 결국 결정해야 함               │
│      (Liveness 속성, 과반수 정상 가정)                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Compare-and-Set을 합의로

```
초기값 = null

Node A: CAS(expected=null, new="A") → 성공 → 결정값 = "A"
Node B: CAS(expected=null, new="B") → 실패 (이미 "A")
Node C: CAS(expected=null, new="C") → 실패 (이미 "A")

→ 모든 노드가 "A"에 동의
```

#### 공유 로그 (Total Order Broadcast)

```
┌─────────────────────────────────────────────────────────────────┐
│                    공유 로그 속성                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   연산:                                                          │
│   • append(value): 로그에 값 추가 요청                          │
│   • read(): 로그 항목 읽기                                      │
│                                                                 │
│   속성:                                                          │
│   1. Eventual Append: 요청하면 결국 추가됨                      │
│   2. Reliable Delivery: 항목 손실 없음                          │
│   3. Append-Only: 한번 읽은 항목은 불변                         │
│   4. Agreement: 모든 노드가 같은 순서로 같은 항목을 봄          │
│   5. Validity: 읽은 값은 누군가 추가 요청한 값                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**공유 로그로 합의 구현:**
```
각 노드가 제안할 값을 로그에 추가 요청
→ 첫 번째로 읽히는 항목의 값이 결정된 값
→ 모든 노드가 같은 순서로 읽으므로 동일한 결정
```

**합의로 공유 로그 구현:**
```
슬롯 1, 2, 3, ... 각각에 대해 별도의 합의 실행
→ 각 슬롯에 어떤 값이 들어갈지 결정
→ 연속된 결정된 슬롯들을 로그 항목으로 추가
```

#### 공유 로그의 활용

```
┌─────────────────────────────────────────────────────────────────┐
│                    공유 로그 활용 사례                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. State Machine Replication                                  │
│      • 로그 항목 = 결정적 연산                                  │
│      • 모든 복제본이 같은 순서로 실행 → 같은 상태              │
│      • Event Sourcing의 원리                                    │
│                                                                 │
│   2. Serializable Transactions                                  │
│      • 로그 항목 = 저장 프로시저                                │
│      • 모든 노드가 같은 순서로 실행 → 직렬화 가능              │
│                                                                 │
│   3. Single-Value Consensus                                     │
│      • 로그의 첫 번째 항목 = 결정된 값                         │
│                                                                 │
│   4. Fencing Token 생성                                         │
│      • 로그 항목의 순번 = 펜싱 토큰 (zxid in ZooKeeper)        │
│                                                                 │
│   5. Atomic Fetch-and-Add                                       │
│      • 로그 항목 = 더할 숫자                                    │
│      • 카운터 값 = 지금까지 로그 항목의 합                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 원자적 커밋 (Atomic Commitment)

```
┌─────────────────────────────────────────────────────────────────┐
│                    원자적 커밋 속성                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. Uniform Agreement: 한 노드가 커밋하면 다른 노드가 어보트 불가│
│   2. Integrity: 한번 커밋/어보트하면 변경 불가                   │
│   3. Validity: 모든 노드가 커밋 투표 → 커밋해야 함              │
│   4. Non-triviality: 모든 노드가 커밋 투표 + 타임아웃 없으면 커밋│
│   5. Termination: 크래시 안 한 노드는 결국 커밋 또는 어보트     │
│                                                                 │
│   합의와의 차이:                                                 │
│   • 합의: 어떤 값이든 결정 가능                                 │
│   • 원자적 커밋: 어보트 투표가 있으면 반드시 어보트             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**합의로 원자적 커밋 구현:**
```
1. 각 노드가 모든 노드에게 투표(커밋/어보트) 전송
2. 모든 노드에서 커밋 투표 받음 → "커밋" 제안
3. 어보트 투표 받거나 타임아웃 → "어보트" 제안
4. 합의 알고리즘으로 결정
5. 결정에 따라 커밋 또는 어보트
```

#### Fetch-and-Add의 합의 능력

```
┌─────────────────────────────────────────────────────────────────┐
│   Fetch-and-Add로 합의?                                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   시도:                                                          │
│   • 카운터 = 0으로 초기화                                       │
│   • 각 노드가 fetch-and-add 실행                                │
│   • 0을 읽은 노드가 승자                                        │
│                                                                 │
│   문제:                                                          │
│   • 다른 노드들은 누가 0을 읽었는지 모름                        │
│   • 0을 읽은 노드가 결과 발표 전에 크래시하면?                  │
│   • 다른 노드들은 영원히 기다림 → Termination 위반              │
│                                                                 │
│   예외: 2개 노드만 참여하면 해결 가능                            │
│   → Consensus Number = 2                                        │
│                                                                 │
│   비교: CAS, 공유 로그 → Consensus Number = ∞                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 합의 알고리즘의 작동 원리

```
┌─────────────────────────────────────────────────────────────────┐
│               합의 알고리즘 공통 구조                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   핵심 아이디어: Single-Leader + 자동 페일오버                   │
│                                                                 │
│   1. Epoch Number (Paxos: ballot, Raft: term, Zab: epoch)       │
│      • 각 epoch 내에서 리더는 유일                              │
│      • 새 리더 선출 시 epoch 증가                               │
│      • 두 리더가 충돌하면 높은 epoch가 승리                     │
│                                                                 │
│   2. 두 번의 투표                                                │
│      ① 리더 선출 투표 (새 epoch에 대한 동의)                    │
│      ② 로그 항목 추가 투표 (리더의 제안에 대한 동의)            │
│                                                                 │
│   3. 쿼럼 겹침 보장                                              │
│      • 리더 선출 쿼럼 ∩ 제안 투표 쿼럼 ≠ ∅                     │
│      • 최소 1개 노드가 양쪽 투표에 참여                         │
│      • 더 높은 epoch가 선출됐는지 감지 가능                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**2PC vs 합의 알고리즘:**

| 특성 | 2PC | 합의 알고리즘 |
|------|-----|---------------|
| 선출 | 코디네이터가 고정 | 아무 노드나 선출 시작 가능 |
| 투표 | 모든 참여자 동의 필요 | 쿼럼(과반수) 동의면 충분 |
| 장애 시 | 코디네이터 장애 → 블로킹 | 새 리더 자동 선출 |

#### 합의의 장단점

**장점:**
```
✅ 자동 페일오버로 고가용성
✅ Split brain 방지
✅ 커밋된 데이터 손실 없음
✅ Chapter 9의 모든 문제에도 데이터 무결성 보장
```

**단점:**
```
❌ 최소 과반수 노드 필요 (3노드 → 1장애 허용, 5노드 → 2장애 허용)
❌ 노드 추가해도 처리량 증가 안 함 (오히려 느려짐)
❌ 네트워크 파티션 시 소수 측은 진행 불가
❌ 타임아웃 튜닝 어려움 (지역 분산 환경)
❌ 네트워크 문제에 민감 (Raft의 특정 엣지 케이스)
```

#### 활용 사례

**1. 설정 관리**
```
프로세스 시작 → 설정 로드 → 변경 알림 구독
→ 설정 변경 시 즉시 반영 또는 재시작
```

**2. 작업 노드 할당**
```
• 리더/프라이머리 선출
• 샤드 → 노드 매핑
• 리밸런싱, 페일오버 자동화
```

**3. 서비스 디스커버리**
```
서비스 시작 → 엔드포인트 등록 → 다른 서비스가 조회
→ 인스턴스 추가/제거 시 알림

주의: 서비스 디스커버리에는 선형성이 과도할 수 있음
→ DNS 기반 캐싱이 더 적합할 수도 있음
→ ZooKeeper Observer: 비선형 읽기로 가용성/처리량 향상
```

**코디네이션 서비스의 이점:**
```
• 3-5개 노드로 수천 개 노드 시스템 조정 가능
• 합의 알고리즘을 직접 구현할 필요 없음
• Apache Curator 같은 고수준 라이브러리 활용 가능
• 느리게 변하는 메타데이터에 적합 (분당/시간당 변경)
```

### 면접 예상 질문

**Q1: Linearizability와 Serializability의 차이점은 무엇인가요?**
> Linearizability는 단일 객체에 대한 읽기/쓰기의 최신성 보장으로, 한 연산이 완료된 후 시작된 연산은 최소한 그 결과를 반영해야 한다. Serializability는 여러 객체에 대한 트랜잭션의 격리 속성으로, 트랜잭션들이 어떤 순서로 순차 실행된 것처럼 동작해야 한다. 둘은 독립적으로 제공 가능하며, 둘 다 제공하면 Strict Serializability다.

**Q2: CAP 정리의 실제 의미는 무엇인가요?**
> "C, A, P 중 2개 선택"이 아니라, 네트워크 파티션(P)은 피할 수 없는 현실이므로 파티션 발생 시 선형성(C)과 가용성(A) 중 하나를 포기해야 한다는 것. 정상 상태에서는 둘 다 가능. 또한 CAP의 가용성 정의는 실제 시스템과 다르고, 선형성만 다루므로 약한 일관성 모델은 설명 못함.

**Q3: 왜 Leaderless 복제에서 쿼럼 읽기/쓰기로도 선형성을 보장할 수 없나요?**
> w + r > n을 만족해도 네트워크 지연으로 인해 두 클라이언트가 서로 다른 복제본 조합을 읽을 수 있다. 한 클라이언트가 새 값을 읽은 후에 다른 클라이언트가 요청을 시작해도, 지연된 복제본에서 오래된 값을 읽을 수 있다. 동기식 read repair와 정교한 타임스탬프 처리로 개선 가능하지만 성능 비용이 있다.

**Q4: 합의와 원자적 커밋의 차이점은 무엇인가요?**
> 합의는 제안된 값 중 어떤 것이든 결정할 수 있다. 원자적 커밋은 한 참여자라도 어보트 투표를 하면 반드시 어보트해야 하는 추가 제약이 있다. 그러나 둘은 동등한 문제로, 합의 알고리즘으로 원자적 커밋을 구현할 수 있고 그 역도 가능하다.
