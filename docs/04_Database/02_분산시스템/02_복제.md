# Chapter 6: Replication

---

## 📌 핵심 요약
> **복제(Replication)**는 동일한 데이터를 네트워크로 연결된 여러 머신에 복사본으로 유지하는 것이다. 복제의 목적은 **고가용성**, **내구성**, **지연 시간 감소**, **읽기 처리량 확장**이다. 복제 방식은 크게 **단일 리더(Single-Leader)**, **다중 리더(Multi-Leader)**, **리더리스(Leaderless)** 세 가지로 나뉜다. 각 방식은 일관성, 가용성, 성능 사이의 트레이드오프가 다르며, 비동기 복제 시 발생하는 **복제 지연(Replication Lag)** 문제와 **충돌 해결(Conflict Resolution)**이 핵심 과제이다.

## 📖 본문 정리

### 1. 복제의 목적

```
┌─────────────────────────────────────────────────────────────────┐
│                    복제(Replication)의 목적                      │
│                                                                 │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐             │
│  │ 고가용성    │  │   내구성    │  │ 지연시간    │             │
│  │ (Availability)│ │ (Durability) │ │  (Latency)  │             │
│  │             │  │             │  │             │             │
│  │ 노드 장애시 │  │ 데이터 손실 │  │ 지리적으로  │             │
│  │ 서비스 지속 │  │ 방지       │  │ 가까운 복제 │             │
│  └─────────────┘  └─────────────┘  └─────────────┘             │
│                                                                 │
│  ┌─────────────────────────────────────────────────┐           │
│  │              확장성 (Scalability)                │           │
│  │                                                  │           │
│  │  읽기 요청을 여러 복제본에 분산 → 처리량 증가     │           │
│  └─────────────────────────────────────────────────┘           │
└─────────────────────────────────────────────────────────────────┘
```

#### 복제 vs 백업

| 구분 | 복제 (Replication) | 백업 (Backup) |
|------|-------------------|---------------|
| **목적** | 실시간 데이터 동기화 | 과거 상태 복원 |
| **데이터** | 최신 상태 반영 | 특정 시점 스냅샷 |
| **삭제 복구** | ❌ 삭제도 전파됨 | ✅ 이전 상태 복원 가능 |
| **용도** | 고가용성, 확장성 | 재해 복구, 실수 복구 |

> 💬 **비유**: 복제는 여러 비서가 동시에 같은 메모를 작성하는 것이고, 백업은 중요 문서를 금고에 보관하는 것이다. 비서들의 메모에서 실수로 내용을 지우면 모든 비서 메모에서 사라지지만, 금고의 문서는 그대로 남아있다.

### 2. 단일 리더 복제 (Single-Leader Replication)

#### 기본 구조

```
┌─────────────────────────────────────────────────────────────────┐
│                   Single-Leader Replication                      │
│                                                                 │
│  ┌─────────────┐                                                │
│  │   Client    │                                                │
│  │  (Write)    │                                                │
│  └──────┬──────┘                                                │
│         │ 쓰기 요청                                              │
│         ▼                                                       │
│  ┌─────────────┐     복제 로그      ┌─────────────┐            │
│  │   Leader    │ ─────────────────→ │  Follower 1 │            │
│  │  (Primary)  │ ─────────────────→ │  Follower 2 │            │
│  │             │ ─────────────────→ │  Follower 3 │            │
│  └─────────────┘                    └─────────────┘            │
│         ↑                                  ↑                    │
│         │                                  │                    │
│  ┌──────┴──────┐                   ┌──────┴──────┐             │
│  │   Client    │                   │   Client    │             │
│  │  (Read)     │                   │  (Read)     │             │
│  └─────────────┘                   └─────────────┘             │
│                                                                 │
│  • 쓰기: 오직 Leader만 가능                                     │
│  • 읽기: Leader 또는 Follower 가능                              │
└─────────────────────────────────────────────────────────────────┘
```

**사용 예**: PostgreSQL, MySQL, MongoDB, DynamoDB, Kafka, CockroachDB (Raft), etcd

#### 동기식 vs 비동기식 복제

```
┌─────────────────────────────────────────────────────────────────┐
│              동기식 vs 비동기식 복제 타임라인                     │
│                                                                 │
│  Client    Leader     Follower 1    Follower 2                 │
│    │         │            │              │                      │
│    │──Write→│            │              │                      │
│    │         │──Replicate→│              │                      │
│    │         │            │              │                      │
│    │         │←───ACK────│              │  ← 동기식 (대기)      │
│    │         │                          │                      │
│    │         │───Replicate──────────────→│  ← 비동기식 (대기X)  │
│    │         │                          │                      │
│    │←──OK───│                          │                      │
│    │         │                          │                      │
│                                         │                      │
│              ┌──────────────────────────┘                      │
│              │ 비동기 Follower는 나중에                         │
│              │ 처리 (지연 발생 가능)                            │
│              └──────────────────────────                       │
└─────────────────────────────────────────────────────────────────┘
```

| 복제 방식 | 장점 | 단점 |
|----------|------|------|
| **동기식** | 데이터 일관성 보장, 장애 시 데이터 손실 없음 | Follower 응답 대기 → 지연, 가용성 저하 |
| **비동기식** | 빠른 응답, 높은 가용성 | 데이터 손실 가능, 일관성 저하 |
| **반동기식 (Semi-sync)** | 최소 2개 노드에 데이터 보장 | 일부 노드만 동기화 |

> ⚠️ **반동기식**: 하나의 Follower만 동기식, 나머지는 비동기식. 동기식 Follower가 느려지면 다른 Follower가 동기식으로 승격.

#### 새 Follower 설정 (무중단)

```
┌─────────────────────────────────────────────────────────────────┐
│                 새 Follower 설정 프로세스                        │
│                                                                 │
│  Step 1: Leader의 일관된 스냅샷 생성                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 스냅샷 @ LSN 12345                                       │   │
│  │ (Log Sequence Number / binlog coordinates / GTID)        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         ↓                                       │
│  Step 2: 스냅샷을 새 Follower에 복사                            │
│                         ↓                                       │
│  Step 3: Leader에 연결하여 스냅샷 이후 변경사항 요청              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ "LSN 12345 이후의 모든 변경사항을 보내주세요"              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         ↓                                       │
│  Step 4: 백로그 처리 완료 → Caught Up (따라잡음)                │
│                         ↓                                       │
│  Step 5: 실시간 복제 스트림 수신 시작                           │
└─────────────────────────────────────────────────────────────────┘
```

**도구 예시**: Percona XtraBackup (MySQL), WAL-G (PostgreSQL), Litestream (SQLite)

#### 노드 장애 처리

**Follower 장애 (Catch-up Recovery)**:
- 로컬 로그에서 마지막 처리 트랜잭션 확인
- Leader에 연결하여 누락된 변경사항 요청
- 백로그 적용 후 정상 복제 재개

**Leader 장애 (Failover)**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Leader Failover 프로세스                      │
│                                                                 │
│  Step 1: 장애 감지 (Timeout, 보통 30초)                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Leader가 응답 없음 → Dead로 간주                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         ↓                                       │
│  Step 2: 새 Leader 선출                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ • 선거 (다수결) 또는 Controller 노드가 지정              │   │
│  │ • 가장 최신 데이터를 가진 Follower 선택                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         ↓                                       │
│  Step 3: 시스템 재구성                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ • 클라이언트 → 새 Leader로 쓰기 요청 라우팅              │   │
│  │ • 기존 Leader 복구 시 → Follower로 강등                  │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

**Failover의 위험성**:

| 문제 | 설명 | 실제 사례 |
|------|------|----------|
| **데이터 손실** | 비동기 복제 시 미복제 쓰기 유실 | - |
| **외부 시스템 불일치** | Auto-increment ID 재사용 | GitHub: MySQL + Redis 불일치로 개인정보 노출 |
| **Split Brain** | 두 노드가 모두 Leader로 동작 | 데이터 손상, 충돌 |
| **타임아웃 설정** | 너무 짧으면 불필요한 Failover | 부하/네트워크 문제 시 상황 악화 |

> 💬 **STONITH (Shoot The Other Node In The Head)**: Split Brain 방지를 위해 이전 Leader를 강제 종료하는 메커니즘. "Fencing"이라고도 함.

#### 복제 로그 구현 방식

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **Statement-based** | SQL 문장 그대로 전송 | 컴팩트 | 비결정적 함수(NOW, RAND) 문제 |
| **WAL Shipping** | 물리적 WAL 전송 | 정확한 복제 | 버전 호환성 문제 |
| **Logical (Row-based)** | 행 단위 변경 이벤트 | 버전 독립적, CDC 용이 | 약간 더 큰 크기 |

**Logical Log 예시**:
```
INSERT: {table: users, id: 123, name: "Alice", email: "alice@ex.com"}
UPDATE: {table: users, id: 123, email: "alice@new.com"}
DELETE: {table: users, id: 123}
COMMIT
```

### 3. 복제 지연 문제 (Replication Lag)

#### Eventual Consistency (최종 일관성)

```
┌─────────────────────────────────────────────────────────────────┐
│                    Eventual Consistency                          │
│                                                                 │
│  시간 →                                                         │
│  ┌────────────────────────────────────────────────────────┐    │
│  │ Leader:    [A] → [B] → [C] → [D]                       │    │
│  │ Follower:  [A] → [A] → [B] → ... → [D] (지연 후 일치)  │    │
│  └────────────────────────────────────────────────────────┘    │
│                                                                 │
│  정의: 쓰기를 멈추고 기다리면, 결국 모든 복제본이 일치함          │
│  문제: "결국(eventually)"이 언제인지 정해지지 않음               │
│       - 정상: 밀리초                                            │
│       - 장애/과부하: 분~시간                                    │
└─────────────────────────────────────────────────────────────────┘
```

#### 3가지 일관성 문제와 해결책

**1. Read-After-Write Consistency (자기 쓰기 읽기)**

```
┌─────────────────────────────────────────────────────────────────┐
│                  Read-After-Write 문제                          │
│                                                                 │
│  User     Leader      Follower                                  │
│   │                                                             │
│   │──Update Profile─→│                                         │
│   │←────OK──────────│                                          │
│   │                                                             │
│   │──────────────Read Profile────────────→│                    │
│   │←──────────Old Profile (아직 미복제)───│                    │
│   │                                                             │
│   │  😡 "내 변경이 저장 안됐어!"                                │
│                                                                 │
│  해결책:                                                        │
│  • 자신이 수정 가능한 데이터는 Leader에서 읽기                   │
│  • 최근 1분 내 수정된 데이터는 Leader에서 읽기                   │
│  • 클라이언트가 마지막 쓰기 타임스탬프 기억 → 해당 시점          │
│    이후 데이터가 있는 복제본에서만 읽기                          │
└─────────────────────────────────────────────────────────────────┘
```

**2. Monotonic Reads (단조 읽기)**

```
┌─────────────────────────────────────────────────────────────────┐
│                    Monotonic Reads 문제                         │
│                                                                 │
│  User 2345                                                      │
│    │                                                            │
│    │──Read──→ Follower A (최신) → 댓글 보임                     │
│    │                                                            │
│    │──Read──→ Follower B (지연) → 댓글 안 보임!                 │
│    │                                                            │
│    │  😵 "방금 본 댓글이 사라졌어!"                             │
│    │  (시간이 거꾸로 가는 것처럼 느껴짐)                         │
│                                                                 │
│  해결책:                                                        │
│  • 같은 사용자는 항상 같은 복제본에서 읽기                       │
│  • User ID 해시로 복제본 선택                                   │
└─────────────────────────────────────────────────────────────────┘
```

**3. Consistent Prefix Reads (일관된 접두사 읽기)**

```
┌─────────────────────────────────────────────────────────────────┐
│                Consistent Prefix Reads 문제                      │
│                                                                 │
│  Mr. Poons: "몇 초 앞을 볼 수 있나요, Mrs. Cake?"              │
│  Mrs. Cake: "보통 10초 정도요, Mr. Poons."                      │
│                                                                 │
│  관찰자가 다른 샤드에서 읽으면:                                  │
│  ┌────────────────────────────────────────────────────────┐    │
│  │ Mrs. Cake: "보통 10초 정도요, Mr. Poons."  (먼저 도착) │    │
│  │ Mr. Poons: "몇 초 앞을 볼 수 있나요?"      (나중 도착) │    │
│  └────────────────────────────────────────────────────────┘    │
│                                                                 │
│  😲 Mrs. Cake가 질문 전에 대답?! (인과관계 위반)               │
│                                                                 │
│  해결책:                                                        │
│  • 인과적으로 연관된 쓰기는 같은 샤드에 저장                     │
│  • 인과 의존성 추적 알고리즘 사용                               │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 다중 리더 복제 (Multi-Leader Replication)

#### 사용 사례

```
┌─────────────────────────────────────────────────────────────────┐
│               Multi-Leader: 지역 분산 배포                       │
│                                                                 │
│  ┌──────────────────┐         ┌──────────────────┐             │
│  │   Region: US     │         │  Region: Europe  │             │
│  │                  │         │                  │             │
│  │  ┌────────────┐  │ ←─────→ │  ┌────────────┐  │             │
│  │  │  Leader A  │  │ 비동기   │  │  Leader B  │  │             │
│  │  └──────┬─────┘  │ 복제    │  └──────┬─────┘  │             │
│  │         │        │         │         │        │             │
│  │   ┌─────┴─────┐  │         │   ┌─────┴─────┐  │             │
│  │   │ Followers │  │         │   │ Followers │  │             │
│  │   └───────────┘  │         │   └───────────┘  │             │
│  └──────────────────┘         └──────────────────┘             │
│                                                                 │
│  장점:                                                          │
│  • 지역 내 쓰기 지연 감소 (Leader가 가까움)                      │
│  • 지역 장애 시 다른 지역에서 계속 운영                          │
│  • 네트워크 문제에 더 강함                                      │
│                                                                 │
│  단점:                                                          │
│  • 충돌 해결 필요 (같은 데이터 동시 수정 시)                     │
│  • 강한 일관성 보장 불가 (은행 잔고, 유니크 제약)                │
└─────────────────────────────────────────────────────────────────┘
```

#### Sync Engine과 Local-First 소프트웨어

```

│  │ Mobile  │     │ Laptop  │     │ Desktop │                   │
│  │ (Leader)│     │ (Leader)│     │ (Leader)│                   │
│  │         │     │         │     │         │                   │
│  │ 로컬DB  │     │ 로컬DB  │     │ 로컬DB  │                   │
│  └────┬────┘     └────┬────┘     └────┬────┘                   │
│       │              │              │                          │
│       └──────────────┼──────────────┘                          │
│                      ↓                                         │
│              ┌─────────────┐                                   │
│              │   Server    │                                   │
│              │ (Sync Hub)  │                                   │
│              └─────────────┘                                   │
│                                                                 │
│  예: Google Docs, Figma, Linear, 캘린더 앱                      │
│  기술: CRDTs, Operational Transformation                       │
│  라이브러리: Automerge, Yjs, PouchDB/CouchDB                    │
└─────────────────────────────────────────────────────────────────┘
```

#### 복제 토폴로지

```
┌─────────────────────────────────────────────────────────────────┐
│                  Multi-Leader 토폴로지                          │
│                                                                 │
│  (a) Circular            (b) Star              (c) All-to-All  │
│                                                                 │
│      ┌───┐                 ┌───┐                ┌───┐          │
│      │ A │←─┐              │ A │                │ A │←─────┐   │
│      └─┬─┘  │              └─┬─┘                └─┬─┘  ↑   │   │
│        │    │                │                    │    │   │   │
│        ▼    │                ▼                    ▼    │   │   │
│      ┌───┐  │            ┌───┼───┐            ┌───┐  │   │   │
│      │ B │──┤            │   │   │            │ B │←─┼───┤   │
│      └─┬─┘  │            ▼   ▼   ▼            └─┬─┘  │   │   │
│        │    │          ┌───┬───┬───┐            │    │   │   │
│        ▼    │          │ B │ C │ D │            ▼    │   │   │
│      ┌───┐  │          └───┴───┴───┘          ┌───┐  │   │   │
│      │ C │──┘                                 │ C │──┴───┘   │
│      └───┘                                    └─┬─┘          │

│                                                                 │
│  문제: (a),(b) 단일 장애점 존재                                 │
│        (c) 메시지 추월 문제 (순서 뒤바뀜)                       │
└─────────────────────────────────────────────────────────────────┘

### 5. 충돌 해결 (Conflict Resolution)

#### 충돌 발생 시나리오

```
┌─────────────────────────────────────────────────────────────────┐
│                    동시 쓰기 충돌                                │
│                                                                 │
│  User 1                     User 2                              │
│    │                          │                                 │
│    │  Title: A → B            │  Title: A → C                   │
│    │       ↓                  │       ↓                         │
│    │  ┌────────┐              │  ┌────────┐                     │
│    │  │Leader 1│              │  │Leader 2│                     │
│    │  │Title=B │              │  │Title=C │                     │
│    │  └───┬────┘              │  └───┬────┘                     │
│    │      │                   │      │                          │
│    │      └───────┬───────────┘      │                          │
│    │              ↓                   │                          │
│    │        ⚠️ CONFLICT!              │                          │
│    │        B인가? C인가?             │                          │
└─────────────────────────────────────────────────────────────────┘
```

#### 충돌 해결 전략

| 전략 | 설명 | 장단점 |
|------|------|--------|
| **회피** | 같은 레코드는 항상 같은 Leader로 | 간단하지만 Leader 변경 시 깨짐 |
| **LWW (Last Write Wins)** | 타임스탬프가 큰 값 채택 | 데이터 손실 발생 |
| **수동 해결** | 모든 값(siblings) 저장 → 사용자/앱이 병합 | 복잡, API 변경 필요 |
| **자동 병합** | CRDT, OT 알고리즘 사용 | 데이터 타입별 알고리즘 필요 |

#### CRDT vs Operational Transformation

```
┌─────────────────────────────────────────────────────────────────┐
│          "ice"에 동시에 "n"과 "!"를 삽입하는 예시               │
│                                                                 │
│  초기 상태: "ice"                                               │
│  Replica A: 앞에 "n" 삽입 → "nice"                             │
│  Replica B: 뒤에 "!" 삽입 → "ice!"                             │
│  목표: "nice!" (두 변경 모두 반영)                              │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ OT (Operational Transformation)                         │   │
│  │                                                          │   │
│  │ • 연산: insert("n", index=0), insert("!", index=3)       │   │
│  │ • A 적용 후 B 적용 시: index=3 → 변환 → index=4          │   │
│  │ • 결과: "nice!"                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ CRDT (Conflict-free Replicated Data Type)               │   │
│  │                                                          │   │
│  │ • 각 문자에 고유 ID 부여: i(1A), c(2A), e(3A)            │   │
│  │ • 삽입: n(1B) after nil, !(4B) after 3A                  │   │
│  │ • ID로 위치 결정 → 인덱스 변환 불필요                     │   │
│  │ • 결과: "nice!"                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  사용 예:                                                       │
│  • OT: Google Docs                                             │
│  • CRDT: Redis Enterprise, Riak, Automerge, Yjs                │
└─────────────────────────────────────────────────────────────────┘
```

#### 쇼핑 카트 병합 문제 (Amazon 사례)

```
┌─────────────────────────────────────────────────────────────────┐
│                  쇼핑 카트 병합 Anomaly                          │

### 6. 리더리스 복제 (Leaderless Replication)

#### 기본 개념

```
┌─────────────────────────────────────────────────────────────────┐
│                   Leaderless Replication                         │
│                  (Dynamo-style Datastores)                       │
│                                                                 │
│  ┌──────────┐                                                   │
│  │  Client  │                                                   │
│  └────┬─────┘                                                   │
│       │                                                         │
│       │ 쓰기/읽기를 여러 노드에 동시 전송                        │
│       │                                                         │
│       ├────────────────┬────────────────┐                       │
│       ▼                ▼                ▼                       │
│  ┌─────────┐      ┌─────────┐      ┌─────────┐                 │
│  │Replica 1│      │Replica 2│      │Replica 3│                 │
│  │  (OK)   │      │  (OK)   │      │ (Down)  │                 │
│  └─────────┘      └─────────┘      └─────────┘                 │
│                                                                 │
│  • Leader 없음 → Failover 불필요                                │
│  • 일부 노드 실패해도 계속 동작                                  │
│  • 예: Riak, Cassandra, ScyllaDB, (원조) Amazon Dynamo          │
└─────────────────────────────────────────────────────────────────┘
```

**주의**: DynamoDB는 Dynamo와 다름! DynamoDB는 Single-Leader (Multi-Paxos 기반)

#### 쿼럼 (Quorum)

```
┌─────────────────────────────────────────────────────────────────┐
│                    Quorum 읽기/쓰기                             │
│                                                                 │
│  n = 복제본 총 개수                                             │
│  w = 쓰기 성공에 필요한 응답 수                                  │
│  r = 읽기 성공에 필요한 응답 수                                  │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │         w + r > n  →  최신 값 읽기 보장                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  예: n=5, w=3, r=3                                             │
│                                                                 │
│  쓰기 시:        읽기 시:         겹침 보장:                    │
│  ┌───┐          ┌───┐            ┌───┐                         │
│  │ 1 │ ✓ w     │ 1 │            │ 1 │ ✓ w                      │
│  │ 2 │ ✓ w     │ 2 │ ✓ r       │ 2 │ ✓ w, ✓ r ← 겹침!        │
│  │ 3 │ ✓ w     │ 3 │ ✓ r       │ 3 │ ✓ w, ✓ r ← 겹침!        │
│  │ 4 │         │ 4 │ ✓ r       │ 4 │        ✓ r               │
│  │ 5 │ (down)  │ 5 │            │ 5 │                          │
│  └───┘          └───┘            └───┘                         │
│                                                                 │
│  읽은 3개 중 최소 1개는 최신 값을 가짐!                          │
└─────────────────────────────────────────────────────────────────┘
```

**일반적인 설정**: n = 홀수 (3 또는 5), w = r = (n+1)/2

#### 누락된 쓰기 복구 메커니즘

| 메커니즘 | 설명 | 특징 |
|---------|------|------|
| **Read Repair** | 읽기 시 오래된 복제본 발견 → 최신 값 쓰기 | 자주 읽는 데이터에 효과적 |
| **Hinted Handoff** | 장애 노드 대신 다른 노드가 쓰기 보관 → 복구 시 전달 | 읽지 않는 데이터도 복구 |
| **Anti-Entropy** | 백그라운드 프로세스가 복제본 간 차이 동기화 | 순서 보장 X, 지연 있음 |

#### Quorum의 한계

**w + r > n 이어도 일관성 문제 발생 가능**:

- 장애 복구 시 오래된 데이터로 복원
- 리밸런싱 중 쿼럼 중복 깨짐
- 읽기와 쓰기 동시 발생 시 타이밍 문제
- 일부 복제본에만 쓰기 성공 (롤백 없음)
- 클럭 동기화 문제 (LWW 사용 시)

### 7. 동시 쓰기 감지 (Detecting Concurrent Writes)

#### Happens-Before 관계

```
┌─────────────────────────────────────────────────────────────────┐
│                    Happens-Before 관계                          │
│                                                                 │
│  두 연산 A, B의 관계:                                           │
│                                                                 │
│  1) A happens-before B: B가 A를 알고 있음 (인과관계)            │
│     A ────────→ B                                               │
│                                                                 │
│  2) B happens-before A: A가 B를 알고 있음                       │
│     B ────────→ A                                               │
│                                                                 │
│  3) A와 B가 동시(concurrent): 서로 모름 (충돌!)                 │
│     A ←───?───→ B                                               │
│                                                                 │
│  핵심: 물리적 시간이 아닌 "알고 있음"으로 판단                   │
└─────────────────────────────────────────────────────────────────┘
```

> 💬 **상대성 이론과의 연관**: 빛의 속도보다 빠르게 정보가 전달될 수 없듯이, 네트워크 지연으로 인해 두 이벤트가 서로를 알 수 없으면 "동시"로 간주한다.

#### Version Number를 이용한 동시성 감지

```
┌─────────────────────────────────────────────────────────────────┐
│                쇼핑 카트 예시 (단일 복제본)                      │
│                                                                 │
│  Client 1                    Server                 Client 2    │
│     │                          │                        │       │
│  ① │──set(cart, [milk])──────→│ v1: [milk]            │       │
│     │←─────────────v1──────────│                        │       │
│     │                          │                        │       │
│  ② │                          │←──set(cart, [eggs])───│       │
│     │                          │ v2: [eggs]            │       │
│     │                          │ siblings: [milk]       │       │
│     │                          │───v2, [milk],[eggs]──→│       │
│     │                          │                        │       │
│  ③ │──set(cart, [milk,flour]) │                        │       │
│     │      with v1            →│ v3: [milk,flour]      │       │
│     │                          │ 덮어씀: v1 [milk]      │       │
│     │                          │ 유지: v2 [eggs]        │       │
│     │←──v3, [milk,flour],     │                        │       │
│     │       [eggs]─────────────│                        │       │
│     │                          │                        │       │
│  ④ │                          │←──set([eggs,milk,ham])│       │
│     │                          │      with v2          │       │
│     │                          │ v4: [eggs,milk,ham]   │       │
│     │                          │───v4, [milk,flour],  →│       │
│     │                          │    [eggs,milk,ham]    │       │
│     │                          │                        │       │
│  ⑤ │──set([milk,flour,       │                        │       │
│     │    eggs,bacon]) v3─────→│ v5: [milk,flour,      │       │
│     │                          │      eggs,bacon]      │       │
│     │                          │ 유지: [eggs,milk,ham] │       │
│     │                          │                        │       │
│  최종: siblings = {[milk,flour,eggs,bacon], [eggs,milk,ham]}   │
└─────────────────────────────────────────────────────────────────┘
```

**알고리즘 규칙**:
1. 서버는 키마다 버전 번호 유지, 쓰기마다 증가
2. 클라이언트 읽기 시: 모든 siblings + 최신 버전 번호 반환
3. 클라이언트 쓰기 시: 이전 읽기의 버전 번호 + 병합된 값 전송
4. 서버 쓰기 수신 시: 해당 버전 이하 값 덮어쓰기, 더 높은 버전은 유지

#### Version Vector (다중 복제본)

```
┌─────────────────────────────────────────────────────────────────┐
│                      Version Vector                              │
│                                                                 │
│  단일 버전 번호로는 다중 복제본에서 동시성 판단 불가              │
│  → 복제본별로 버전 번호 유지                                     │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Version Vector = { Replica1: 5, Replica2: 3, Replica3: 7 } │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  비교 규칙:                                                     │
│  • VV_A의 모든 항목이 VV_B 이하 → A happens-before B           │
│  • VV_A의 일부가 더 크고 일부가 더 작음 → 동시(Concurrent)       │
│                                                                 │
│  예:                                                            │
│  VV_A = {R1:3, R2:5} vs VV_B = {R1:4, R2:6}                    │
│  → A의 모든 값 ≤ B의 값 → A happens-before B                   │
│                                                                 │
│  VV_A = {R1:5, R2:3} vs VV_B = {R1:4, R2:6}                    │
│  → R1: A>B, R2: A<B → Concurrent (충돌!)                       │
└─────────────────────────────────────────────────────────────────┘
```

**Riak의 Causal Context**: Version Vector를 문자열로 인코딩하여 클라이언트가 읽기/쓰기 시 전달

## 💡 실무 적용 포인트

### Quick Reference

| 요구사항 | 권장 복제 방식 |
|---------|--------------|
| 강한 일관성 필요 | Single-Leader (동기식) |
| 읽기 확장성 필요 | Single-Leader + 읽기 복제본 |
| 지역 분산 (쓰기 지연 최소화) | Multi-Leader 또는 Leaderless |
| 오프라인 지원/협업 앱 | Multi-Leader (Sync Engine) |
| 최대 가용성/장애 허용 | Leaderless (Quorum) |

### 주의사항

- ⚠️ **비동기 복제의 데이터 손실**: Failover 시 미복제 쓰기 유실 가능
- ⚠️ **Split Brain 방지**: STONITH/Fencing 메커니즘 필수
- ⚠️ **LWW의 데이터 손실**: 동시 쓰기 시 무작위로 하나 선택, 나머지 폐기
- ⚠️ **Quorum ≠ 완벽한 일관성**: 엣지 케이스 존재 (복구, 리밸런싱 등)
- ⚠️ **클럭 동기화**: LWW 사용 시 NTP 동기화 필수

### 기술 면접 예상 질문

**Q1. 동기식 복제와 비동기식 복제의 트레이드오프는?**
> A: 동기식은 데이터 일관성과 내구성을 보장하지만 지연 시간이 증가하고 Follower 장애 시 쓰기 불가. 비동기식은 빠른 응답과 높은 가용성을 제공하지만 Leader 장애 시 데이터 손실 가능. 실무에서는 반동기식(Semi-sync)으로 절충 - 최소 하나의 Follower만 동기화.

**Q2. Quorum에서 w + r > n이 최신 값 읽기를 보장하는 이유는?**
> A: 쓰기 성공에 w개 노드, 읽기에 r개 노드가 필요하고, w + r > n이면 쓰기 집합과 읽기 집합이 반드시 겹침. 겹치는 노드 중 최소 하나는 최신 값을 가지므로, 읽기 응답 중 가장 큰 버전의 값이 최신. 단, 리밸런싱, 복구, 동시 읽기/쓰기 등 엣지 케이스에서는 깨질 수 있음.

**Q3. CRDT와 OT의 차이점은?**
> A: OT는 연산의 인덱스를 변환하여 충돌 해결 (Google Docs), CRDT는 각 요소에 고유 ID를 부여하여 변환 없이 병합. OT는 서버 중심 조정이 필요하고, CRDT는 탈중앙화 가능하여 Local-First 앱에 적합. 두 방식 모두 텍스트, 리스트, 맵 등 다양한 데이터 타입 지원.

**Q4. Leaderless 복제가 Single-Leader보다 장애에 강한 이유는?**
> A: (1) Failover 프로세스가 없어 Leader 선출 지연 없음 (2) 요청이 여러 노드에 병렬로 가므로 일부 노드 느림/장애 시 빠른 노드 응답 사용 (Request Hedging) (3) 정상/장애 상황을 구분하지 않아 Gray Failure에도 강함. 단점은 충돌 해결 필요, 강한 일관성 보장 어려움.

## ✅ 체크리스트

### 복제 기초
- [ ] 복제의 4가지 목적 (가용성, 내구성, 지연, 확장성)을 설명할 수 있다
- [ ] 복제와 백업의 차이를 이해한다
- [ ] 동기식/비동기식/반동기식 복제의 트레이드오프를 안다

### Single-Leader
- [ ] Leader-Follower 아키텍처의 쓰기/읽기 흐름을 설명할 수 있다
- [ ] 새 Follower 설정 과정 (스냅샷 + 로그)을 이해한다
- [ ] Failover의 단계와 위험성 (Split Brain, 데이터 손실)을 안다
- [ ] WAL Shipping과 Logical Log의 차이를 설명할 수 있다

### Replication Lag
- [ ] Eventual Consistency의 의미와 한계를 이해한다
- [ ] Read-after-write, Monotonic reads, Consistent prefix reads를 구분한다
- [ ] 각 일관성 문제의 해결책을 설명할 수 있다

### Multi-Leader
- [ ] Multi-Leader 사용 사례 (지역 분산, Sync Engine)를 안다
- [ ] 복제 토폴로지 (Circular, Star, All-to-all)의 장단점을 비교한다
- [ ] Sync Engine과 Local-First 소프트웨어 개념을 이해한다

### 충돌 해결
- [ ] LWW의 작동 방식과 데이터 손실 위험을 안다
- [ ] CRDT와 OT의 차이를 설명할 수 있다
- [ ] 쇼핑 카트 병합 문제와 해결책을 이해한다

### Leaderless
- [ ] Quorum 공식 (w + r > n)의 의미를 설명할 수 있다
- [ ] Read Repair, Hinted Handoff, Anti-Entropy를 구분한다
- [ ] Quorum 일관성의 한계를 알고 있다

### 동시성 감지
- [ ] Happens-Before 관계와 동시성(Concurrent)을 정의할 수 있다
- [ ] Version Number로 동시 쓰기를 감지하는 알고리즘을 이해한다
- [ ] Version Vector가 필요한 이유를 설명할 수 있다
