# 01. Setting Up Your Go Environment 면접 정리

---

## 1. 핵심 개념 상세 설명

### 1.1 Go 프로그램의 특징

Go 프로그램은 컴파일 시 단일 네이티브 바이너리(single native binary)로 생성됩니다. 이것은 Java나 Python과 근본적으로 다른 특성입니다. Java는 JVM(Java Virtual Machine)이 필요하고, Python은 인터프리터가 필요하지만, Go 바이너리는 운영체제 위에서 직접 실행됩니다. 이 특성은 배포를 극도로 단순화합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Go vs 다른 언어 실행 모델                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   Go                           Java                              │
│   ┌─────────────┐              ┌─────────────┐                  │
│   │  Go Binary  │              │  .class     │                  │
│   │  (단일 파일) │              │  (바이트코드) │                  │
│   └──────┬──────┘              └──────┬──────┘                  │
│          │                            │                          │
│          ▼                            ▼                          │
│   ┌─────────────┐              ┌─────────────┐                  │
│   │     OS      │              │     JVM     │                  │
│   └─────────────┘              └──────┬──────┘                  │
│                                       │                          │
│                                       ▼                          │
│                                ┌─────────────┐                  │
│                                │     OS      │                  │
│                                └─────────────┘                  │
│                                                                   │
│   Python                       JavaScript (Node.js)              │
│   ┌─────────────┐              ┌─────────────┐                  │
│   │   .py 파일   │              │   .js 파일   │                  │
│   └──────┬──────┘              └──────┬──────┘                  │
│          │                            │                          │
│          ▼                            ▼                          │
│   ┌─────────────┐              ┌─────────────┐                  │
│   │ Python 인터프리터│         │  Node.js    │                  │
│   └──────┬──────┘              │   Runtime   │                  │
│          │                      └──────┬──────┘                  │
│          ▼                            │                          │
│   ┌─────────────┐                     ▼                          │
│   │     OS      │              ┌─────────────┐                  │
│   └─────────────┘              │     OS      │                  │
│                                └─────────────┘                  │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

컨테이너 환경에서 이 특성은 더욱 빛을 발합니다. Go 애플리케이션은 scratch 이미지나 distroless 이미지 위에서 실행할 수 있어 컨테이너 크기를 수 MB 수준으로 줄일 수 있습니다. 반면 Java 애플리케이션은 JRE가 포함된 베이스 이미지가 필요하여 수백 MB가 되기도 합니다. 작은 이미지는 빠른 배포, 적은 공격 표면, 낮은 저장 비용으로 이어집니다.

### 1.2 Go Module 시스템

Go 1.11부터 도입된 모듈 시스템은 Go의 의존성 관리 방식입니다. `go mod init` 명령으로 모듈을 초기화하면 `go.mod` 파일이 생성되며, 이 파일에는 모듈 이름과 최소 Go 버전, 그리고 의존성 정보가 기록됩니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Go Module 파일 구조                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   project/                                                       │
│   ├── go.mod         ◄── 모듈 정의 (수동 편집 지양)               │
│   │   ┌──────────────────────────────────────────┐              │
│   │   │ module github.com/user/myproject        │              │
│   │   │                                          │              │
│   │   │ go 1.21                                  │              │
│   │   │                                          │              │
│   │   │ require (                                │              │
│   │   │     github.com/gin-gonic/gin v1.9.1     │              │
│   │   │     google.golang.org/grpc v1.58.0      │              │
│   │   │ )                                        │              │
│   │   └──────────────────────────────────────────┘              │
│   │                                                              │
│   ├── go.sum         ◄── 의존성 체크섬 (자동 생성, 커밋 필수)      │
│   │                                                              │
│   ├── main.go                                                    │
│   └── ...                                                        │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

모듈 이름은 일반적으로 저장소 경로를 사용합니다. 예를 들어 `github.com/username/projectname` 형식입니다. 이렇게 하면 다른 프로젝트에서 이 모듈을 import할 때 Go 도구가 자동으로 저장소를 찾을 수 있습니다. go.sum 파일은 의존성의 체크섬을 저장하여 동일한 버전의 의존성이 항상 동일한 내용임을 보장합니다.

### 1.3 Semicolon Insertion Rule

Go 컴파일러는 lexer 단계에서 자동으로 세미콜론을 삽입합니다. 개발자가 명시적으로 세미콜론을 작성할 필요가 없어 코드가 깔끔해지지만, 이 규칙을 이해하지 못하면 예상치 못한 컴파일 에러가 발생합니다.

규칙은 단순합니다. 특정 토큰(식별자, 리터럴, break, continue, return, ++, --, ), }) 뒤에 개행이 오면 세미콜론이 자동 삽입됩니다. 이 규칙 때문에 여는 중괄호 `{`는 반드시 함수 선언과 같은 줄에 있어야 합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                 Semicolon Insertion Rule                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   작성한 코드:                     컴파일러가 보는 코드:           │
│   ┌──────────────────────┐        ┌──────────────────────┐      │
│   │ func main()          │   →    │ func main();         │      │
│   │ {                    │        │ {                    │      │
│   │     fmt.Println()    │        │     fmt.Println();   │      │
│   │ }                    │        │ };                   │      │
│   └──────────────────────┘        └──────────────────────┘      │
│           ↓                                   ↓                  │
│        ❌ 컴파일 에러!              세미콜론이 삽입되어           │
│                                    함수 본문이 분리됨            │
│                                                                   │
│   올바른 코드:                                                    │
│   ┌──────────────────────┐                                       │
│   │ func main() {        │   ◄── 여는 중괄호가 같은 줄에         │
│   │     fmt.Println()    │                                       │
│   │ }                    │                                       │
│   └──────────────────────┘                                       │
│           ↓                                                      │
│        ✅ 정상 컴파일                                            │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

이 규칙은 Go의 포맷팅 강제 정책과 맞물려 있습니다. Go는 K&R 스타일(여는 중괄호가 선언과 같은 줄)을 강제하며, 이에 대한 논쟁의 여지를 없앴습니다.

### 1.4 Go 도구 체인

Go는 풍부한 내장 도구를 제공합니다. `go build`는 컴파일러, `go fmt`는 코드 포맷터, `go vet`는 정적 분석기입니다. 이 도구들은 Go 개발 워크플로우의 핵심이며, 대부분의 Go 프로젝트에서 동일하게 사용됩니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Go 개발 워크플로우                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   코드 작성                                                      │
│       │                                                          │
│       ▼                                                          │
│   ┌─────────────┐                                               │
│   │   go fmt    │  ◄── 코드 포맷팅 (탭, 공백, 정렬 통일)         │
│   └──────┬──────┘                                               │
│          │                                                       │
│          ▼                                                       │
│   ┌─────────────┐                                               │
│   │   go vet    │  ◄── 정적 분석 (논리적 오류 탐지)              │
│   └──────┬──────┘                                               │
│          │                                                       │
│          ▼                                                       │
│   ┌─────────────┐                                               │
│   │  go build   │  ◄── 컴파일 (문법 오류 탐지)                   │
│   └──────┬──────┘                                               │
│          │                                                       │
│          ▼                                                       │
│   ┌─────────────┐                                               │
│   │  go test    │  ◄── 테스트 실행                              │
│   └──────┬──────┘                                               │
│          │                                                       │
│          ▼                                                       │
│      바이너리 생성                                               │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

`go fmt`는 코드 스타일을 강제 통일합니다. 들여쓰기는 탭, 여는 중괄호는 선언과 같은 줄 등 규칙이 정해져 있어 팀 내 코드 스타일 논쟁이 원천적으로 불가능합니다. `go vet`는 컴파일은 되지만 논리적으로 잘못된 코드를 탐지합니다. 예를 들어 `fmt.Printf`에서 포맷 문자열과 인자 개수가 맞지 않는 경우를 찾아냅니다.

### 1.5 Go Compatibility Promise

Go 팀은 Go 1.x 버전에서 하위 호환성을 보장한다고 약속했습니다. Go 1.0에서 작성된 코드는 Go 1.21에서도 변경 없이 컴파일되고 동작해야 합니다. 이 약속은 언어 스펙과 표준 라이브러리에 적용되지만, go 명령어 자체의 플래그나 동작은 변경될 수 있습니다.

이 호환성 약속은 기업 환경에서 Go 채택의 중요한 요인입니다. 프레임워크나 언어 업그레이드로 인한 대규모 코드 수정 없이 보안 패치와 성능 개선을 받을 수 있기 때문입니다.

## 3. 면접 예상 질문 및 모범 답안

### Q1. Go 프로그램이 단일 바이너리로 컴파일되는 것의 장단점을 설명해주세요.

**모범 답안:**

Go가 단일 네이티브 바이너리로 컴파일되는 것은 여러 장점이 있습니다.

첫째, 배포가 극도로 단순해집니다. 바이너리 파일 하나만 복사하면 실행 가능하므로, 런타임 설치, 의존성 관리, 버전 충돌 같은 문제가 없습니다. "내 컴퓨터에서는 되는데" 문제를 원천적으로 방지합니다.

둘째, 컨테이너 환경에서 이미지 크기를 최소화할 수 있습니다. scratch나 distroless 같은 최소 이미지를 베이스로 사용할 수 있어 수 MB 수준의 컨테이너 이미지가 가능합니다. 작은 이미지는 빠른 배포, 적은 보안 공격 표면, 낮은 저장 비용으로 이어집니다.

셋째, 시작 시간이 빠릅니다. JVM처럼 런타임을 초기화하는 오버헤드가 없어 cold start가 빠르며, 이는 서버리스 환경에서 특히 유리합니다.

단점도 있습니다. 바이너리 크기가 상대적으로 큽니다. Go 런타임과 가비지 컬렉터가 바이너리에 포함되기 때문입니다. 또한 크로스 컴파일은 쉽지만, 각 플랫폼(OS/아키텍처 조합)별로 별도의 바이너리를 빌드해야 합니다. 마지막으로 동적 링킹의 유연성이 없어 라이브러리 업데이트 시 재컴파일이 필요합니다.

---

### Q2. go fmt가 코드 스타일을 강제하는 것의 의의를 설명해주세요.

**모범 답안:**

go fmt가 코드 스타일을 강제하는 것은 Go의 핵심 철학인 단순성과 실용성을 반영합니다.

첫째, 개발자 간 스타일 논쟁을 원천 차단합니다. "탭 vs 스페이스", "여는 중괄호 위치", "import 정렬 순서" 같은 논쟁에 시간을 낭비하지 않습니다. Go에서는 탭, K&R 스타일, 알파벳 순 import가 표준이며 예외가 없습니다.

둘째, 코드 리뷰가 본질에 집중할 수 있습니다. 스타일에 대한 피드백이 불필요하므로 리뷰어는 로직과 설계에만 집중할 수 있습니다.

셋째, 도구 개발이 용이해집니다. 모든 Go 코드가 동일한 포맷을 따르므로, 코드 분석 도구, 리팩토링 도구, 코드 생성 도구가 일관된 출력을 보장할 수 있습니다.

넷째, 팀 온보딩이 빨라집니다. 새로운 개발자가 팀의 코드 스타일 가이드를 배울 필요가 없습니다. 모든 Go 프로젝트가 동일한 스타일을 따르기 때문입니다.

실무에서는 저장 시 자동 포맷팅을 설정하거나, CI 파이프라인에서 `go fmt` 결과와 커밋된 코드가 동일한지 검사하여 일관성을 보장합니다.

### Q3. Semicolon Insertion Rule을 설명하고, 이로 인해 발생할 수 있는 문제를 설명해주세요.

**모범 답안:**

Go 컴파일러는 lexer 단계에서 자동으로 세미콜론을 삽입합니다. 특정 토큰 뒤에 개행이 오면 세미콜론이 삽입되는데, 그 토큰은 식별자(변수명, 타입명), 리터럴(숫자, 문자열), 특정 키워드(break, continue, fallthrough, return), 연산자(++, --), 닫는 괄호(], ), })입니다.

이 규칙 때문에 발생하는 가장 흔한 문제는 여는 중괄호 위치입니다. 함수 선언 다음 줄에 여는 중괄호를 두면 컴파일 에러가 발생합니다. `func main()` 뒤에 개행이 오면 `func main();`로 변환되어 함수 본문이 없는 것으로 처리되기 때문입니다.

비슷한 문제가 if, for, switch 문에서도 발생합니다. 조건문 뒤에 바로 개행하면 안 됩니다. 또한 return 문에서 반환값을 다음 줄에 쓰면 빈 return으로 처리됩니다.

이 규칙은 Go가 특정 코드 스타일(K&R 스타일)을 강제하는 근거이기도 합니다. go fmt가 코드를 자동으로 이 스타일로 변환하므로, 개발자가 이 규칙을 외우지 않아도 됩니다. 하지만 규칙을 이해하면 이상한 컴파일 에러를 만났을 때 원인을 빠르게 파악할 수 있습니다.

---

### Q4. go build와 go vet의 차이점을 설명해주세요.

**모범 답안:**

go build와 go vet는 모두 코드 품질을 검사하지만 역할이 다릅니다.

go build는 컴파일러로, 문법적으로 올바른지 검사합니다. 타입 불일치, 정의되지 않은 변수 사용, 잘못된 import 등 컴파일 에러를 탐지합니다. 문법적으로 틀린 코드는 go build를 통과할 수 없습니다.

go vet는 정적 분석 도구로, 문법적으로는 올바르지만 논리적으로 문제가 있을 가능성이 높은 코드를 탐지합니다. 대표적인 예로 `fmt.Printf`에서 포맷 문자열의 placeholder 개수와 인자 개수가 맞지 않는 경우가 있습니다. 이 코드는 컴파일은 되지만 런타임에 문제가 발생합니다. go vet는 이런 문제를 컴파일 시점에 잡아냅니다.

go vet가 탐지하는 다른 예로는 도달 불가능한 코드, 잘못된 build tag, 불필요한 할당, Printf 계열 함수의 형식 오류, 잠재적 race condition 패턴 등이 있습니다.

실무에서는 go fmt → go vet → go build 순서로 실행하여 코드 품질을 단계적으로 검증합니다. CI 파이프라인에서 go vet 실패 시 빌드를 중단하도록 설정하는 것이 권장됩니다.

---

### Q5. Go Compatibility Promise가 무엇이고, 실무에서 어떤 의미가 있는지 설명해주세요.

**모범 답안:**

Go Compatibility Promise는 Go 팀이 Go 1.x 버전에서 하위 호환성을 보장하겠다는 약속입니다. Go 1.0에서 작성되어 정상 동작하는 코드는 Go 1.21에서도 변경 없이 컴파일되고 동일하게 동작해야 합니다. 이 약속은 언어 스펙과 표준 라이브러리에 적용됩니다.

단, go 명령어 자체의 플래그나 동작은 이 약속에 포함되지 않습니다. 예를 들어 go mod 명령은 Go 1.11에서 도입되었고, 이후 버전에서 동작이 변경되기도 했습니다. 또한 명백한 버그 수정이나 보안 취약점 패치는 하위 호환성보다 우선됩니다.

실무에서 이 약속의 의미는 큽니다. 첫째, Go 버전 업그레이드가 안전합니다. 언어나 프레임워크 메이저 버전 업그레이드로 인한 대규모 코드 수정 없이 보안 패치와 성능 개선을 받을 수 있습니다. 둘째, 장기 프로젝트 유지보수가 용이합니다. 5년 전에 작성한 코드도 최신 Go 버전에서 동작합니다. 셋째, 의존성 문제가 줄어듭니다. 라이브러리들이 Go 버전 호환성으로 충돌할 일이 적습니다.

---

### Q6. go.mod와 go.sum 파일의 역할을 설명해주세요.

**모범 답안:**

go.mod 파일은 Go 모듈의 정의 파일입니다. 모듈 이름(module path), 최소 지원 Go 버전, 직접 의존성(direct dependencies)이 기록됩니다. Python의 requirements.txt나 Node.js의 package.json과 유사한 역할입니다.

go.sum 파일은 의존성의 체크섬(해시값)을 저장합니다. 이 파일은 의존성의 무결성을 보장합니다. 동일한 버전의 모듈이라도 내용이 변경되면 체크섬이 달라지므로, 공급망 공격이나 우발적 변경을 탐지할 수 있습니다.

두 파일 모두 버전 관리 시스템에 커밋해야 합니다. go.mod는 프로젝트의 의존성을 명시하고, go.sum은 재현 가능한 빌드(reproducible build)를 보장합니다. 다른 개발자나 CI 환경에서 go build를 실행하면 go.sum의 체크섬과 비교하여 동일한 의존성이 사용되었는지 검증합니다.

실무에서 주의할 점은 go.mod를 직접 편집하지 않는 것입니다. go get, go mod tidy 같은 명령어를 통해 관리하는 것이 권장됩니다. 또한 go.sum에서 체크섬 불일치 에러가 발생하면 네트워크 문제보다는 의존성 변조 가능성을 먼저 의심해야 합니다.

### Go 개발 환경 설정 체크리스트

- [ ] Go 설치 완료 (`go version` 확인)

- [ ] Delve (디버거) 설치 완료

- [ ] `go mod init <module-path>` 실행
- [ ] go.mod 파일에 올바른 모듈 경로 설정
- [ ] Makefile 생성 (fmt, vet, build, test 타겟)
- [ ] .gitignore에 바이너리 파일 제외 규칙 추가
- [ ] go.mod, go.sum 버전 관리에 포함

### 개발 워크플로우 체크리스트

- [ ] 커밋 전 `go vet ./...` 실행

- [ ] 의존성 추가 시 `go get` 사용
- [ ] 정기적으로 `go mod tidy`로 의존성 정리
