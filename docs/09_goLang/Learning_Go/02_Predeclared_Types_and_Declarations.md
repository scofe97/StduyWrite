# 02. Predeclared Types and Declarations 면접 정리

## 1. 핵심 개념 상세 설명

### 1.1 Zero Value

Go의 Zero Value는 변수가 선언되었지만 명시적으로 값이 할당되지 않았을 때 자동으로 부여되는 기본값입니다. C나 C++에서는 초기화되지 않은 변수가 이전 메모리에 남아있던 쓰레기 값(garbage value)을 가질 수 있어 심각한 버그의 원인이 되지만, Go는 모든 변수에 예측 가능한 Zero Value를 보장합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    타입별 Zero Value                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   수치 타입                      불리언/문자열                    │
│   ┌─────────────┐               ┌─────────────┐                 │
│   │ int      → 0│               │ bool   → false│                │
│   │ int8     → 0│               │ string → ""   │                │
│   │ int16    → 0│               └─────────────┘                 │
│   │ int32    → 0│                                                │
│   │ int64    → 0│               참조/포인터 타입                  │
│   │ uint     → 0│               ┌─────────────┐                 │
│   │ float32  → 0│               │ pointer → nil│                │
│   │ float64  → 0│               │ slice   → nil│                │
│   │ complex64→0+0i│             │ map     → nil│                │
│   └─────────────┘               │ chan    → nil│                │
│                                  │ func    → nil│                │
│   문자 타입                      │ interface→nil│                │
│   ┌─────────────┐               └─────────────┘                 │
│   │ byte    → 0 │                                                │
│   │ rune    → 0 │               구조체 타입                      │
│   └─────────────┘               ┌─────────────────┐             │
│                                  │ struct → 모든    │             │
│                                  │ 필드가 Zero Value│             │
│                                  └─────────────────┘             │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

Zero Value는 단순히 기본값이 아니라 Go의 설계 철학을 반영합니다. 변수 선언 시 `var count int`라고 쓰면, count가 0으로 초기화됨이 보장됩니다. 이는 "변수를 선언하고 나중에 값을 할당하겠다"는 의도를 명확히 표현하는 방법입니다.

### 1.2 명시적 타입 변환

Go는 자동 타입 변환(implicit type conversion)을 허용하지 않습니다. 다른 언어에서는 `int + float`가 자동으로 float 연산이 되지만, Go에서는 컴파일 에러가 발생합니다. 모든 타입 변환은 명시적으로 작성해야 합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    명시적 타입 변환                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   다른 언어 (암시적 변환)              Go (명시적 변환만)          │
│   ┌──────────────────────┐           ┌──────────────────────┐   │
│   │ int x = 10;          │           │ var x int = 10       │   │
│   │ double y = 3.14;     │           │ var y float64 = 3.14 │   │
│   │ double z = x + y;    │  ←─────→  │ z := float64(x) + y  │   │
│   │ // x가 자동으로 double로│         │ // 반드시 변환 명시    │   │
│   └──────────────────────┘           └──────────────────────┘   │
│                                                                   │
│   타입 변환 문법: 타입명(값)                                      │
│   ┌──────────────────────────────────────────────────────────┐  │
│   │ float64(intVar)    // int → float64                      │  │
│   │ int(floatVar)      // float64 → int (소수점 버림)         │  │
│   │ string(runeVar)    // rune → string                      │  │
│   │ []byte(stringVar)  // string → []byte                    │  │
│   └──────────────────────────────────────────────────────────┘  │
│                                                                   │
│   ⚠️ 같은 크기라도 변환 필요                                     │
│   ┌──────────────────────────────────────────────────────────┐  │
│   │ var a int32 = 10                                         │  │
│   │ var b int64 = 20                                         │  │
│   │ // c := a + b        // 컴파일 에러!                      │  │
│   │ c := int64(a) + b    // 명시적 변환 필요                   │  │
│   └──────────────────────────────────────────────────────────┘  │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

이 설계 결정의 이유는 명확합니다. 암시적 타입 변환은 예상치 못한 정밀도 손실이나 오버플로우를 야기할 수 있습니다. Go는 개발자가 타입 변환의 결과를 명확히 인지하도록 강제합니다. 코드를 읽을 때도 어떤 타입 변환이 일어나는지 즉시 파악할 수 있어 유지보수성이 높아집니다.

### 1.3 Untyped 상수와 리터럴

Go의 리터럴(10, 3.14, "hello" 등)은 Untyped입니다. 이는 리터럴이 특정 타입에 종속되지 않고, 할당되는 변수의 타입에 따라 유연하게 적용됨을 의미합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Untyped 리터럴의 유연성                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   리터럴 10 (Untyped)                                            │
│        │                                                         │
│        ├──→ var a int = 10       ✓ int로 할당                   │
│        ├──→ var b int64 = 10     ✓ int64로 할당                 │
│        ├──→ var c float64 = 10   ✓ float64로 할당 (10.0)        │
│        ├──→ var d byte = 10      ✓ byte로 할당                  │
│        └──→ var e string = 10    ✗ 컴파일 에러 (호환 불가)       │
│                                                                   │
│   Typed 상수 vs Untyped 상수                                     │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ const untypedX = 10     // Untyped: 여러 타입에 할당 가능  │ │
│   │ const typedX int = 10   // Typed: int에만 할당 가능        │ │
│   │                                                            │ │
│   │ var a float64 = untypedX  // ✓ 가능                       │ │
│   │ var b float64 = typedX    // ✗ 컴파일 에러                 │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   리터럴 기본 타입 (타입 추론 시 사용)                            │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ 정수 리터럴     → int                                      │ │
│   │ 부동소수점 리터럴 → float64                                 │ │
│   │ 룬 리터럴       → rune                                     │ │
│   │ 문자열 리터럴    → string                                  │ │
│   │ 불리언 리터럴    → bool                                    │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

Untyped 상수는 수학적 상수를 정의할 때 특히 유용합니다. `const Pi = 3.14159265358979`처럼 선언하면, 이 상수는 float32에도, float64에도 할당할 수 있습니다. 단, 범위를 벗어나거나 호환되지 않는 타입에 할당하면 컴파일 에러가 발생합니다.

### 1.4 var와 := 선택

Go는 변수 선언에 두 가지 방법을 제공합니다. `var` 키워드를 사용하는 방법과 `:=` 짧은 선언을 사용하는 방법입니다. 이 둘은 단순한 문법 차이가 아니라 의도를 표현하는 방식이 다릅니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    var vs := 선택 가이드                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   var 사용 (명시적 의도 표현)                                     │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ // 1. Zero Value가 의도된 값일 때                          │ │
│   │ var count int      // 0으로 시작하는 카운터                 │ │
│   │ var message string // 빈 문자열로 시작                      │ │
│   │                                                            │ │
│   │ // 2. 패키지 레벨 변수                                      │ │
│   │ var config Config  // 패키지 레벨에서는 := 사용 불가        │ │
│   │                                                            │ │
│   │ // 3. 리터럴 기본 타입과 다른 타입이 필요할 때               │ │
│   │ var port uint16 = 8080  // int가 아닌 uint16 필요          │ │
│   │ var size byte = 255     // int가 아닌 byte 필요            │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   := 사용 (간결한 코드)                                          │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ // 1. 함수 반환값 할당                                      │ │
│   │ result, err := doSomething()                               │ │
│   │                                                            │ │
│   │ // 2. 리터럴 기본 타입을 사용할 때                           │ │
│   │ name := "Alice"     // string                              │ │
│   │ count := 42         // int                                 │ │
│   │ ratio := 3.14       // float64                             │ │
│   │                                                            │ │
│   │ // 3. 표현식 결과 할당                                      │ │
│   │ sum := a + b                                               │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   ⚠️ := 주의사항                                                 │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ x := 10                                                    │ │
│   │ x, y := 20, "hello"  // x는 재할당, y는 새 선언             │ │
│   │ // 최소 1개의 새 변수가 있어야 := 사용 가능                  │ │
│   │                                                            │ │
│   │ // 패키지 레벨에서 := 불가                                  │ │
│   │ // x := 10  // 컴파일 에러                                  │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

`var count int`라고 쓰면 "count는 정수이고, 0부터 시작할 것"이라는 의도를 명확히 전달합니다. 반면 `count := 0`은 그냥 초기값을 지정한 것인지, 0이 특별한 의미를 가지는지 불분명합니다. Go의 관용적 코드에서는 이러한 미묘한 차이를 통해 코드의 의도를 전달합니다.

### 1.5 사용하지 않는 변수 규칙

Go는 선언된 지역 변수가 최소 한 번이라도 읽히지 않으면 컴파일 에러를 발생시킵니다. 이는 다른 언어에서는 경고(warning) 수준이지만, Go에서는 에러입니다.

이 규칙의 목적은 코드 품질을 강제하는 것입니다. 사용하지 않는 변수는 리팩토링 과정에서 남은 잔재이거나, 로직 오류의 징후일 수 있습니다. Go는 이런 코드가 존재하는 것 자체를 허용하지 않습니다.

단, 이 규칙은 지역 변수에만 적용됩니다. 패키지 레벨 변수나 상수는 이 검사를 받지 않습니다. 이는 패키지가 다른 곳에서 사용될 수 있기 때문입니다. 그러나 Go 커뮤니티에서는 패키지 레벨 변수 자체를 최소화하는 것을 권장합니다.

### 1.6 byte와 rune

byte와 rune은 각각 uint8과 int32의 별칭(alias)입니다. 타입 별칭이란 같은 기저 타입에 다른 이름을 부여한 것으로, 코드의 의도를 명확히 하기 위해 사용됩니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    byte와 rune의 용도                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   byte (uint8의 별칭)                 rune (int32의 별칭)        │
│   ┌───────────────────────┐          ┌───────────────────────┐  │
│   │ 용도: 바이트 데이터     │          │ 용도: 유니코드 코드포인트│  │
│   │ 범위: 0 ~ 255          │          │ 범위: -2^31 ~ 2^31-1   │  │
│   │                        │          │                        │  │
│   │ 사용 예:               │          │ 사용 예:               │  │
│   │ - 파일 I/O            │          │ - 문자열 처리          │  │
│   │ - 네트워크 패킷        │          │ - 유니코드 문자 처리    │  │
│   │ - 바이너리 데이터      │          │ - 문자 리터럴          │  │
│   └───────────────────────┘          └───────────────────────┘  │
│                                                                   │
│   코드 예시:                                                      │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ // byte 사용: 바이너리 데이터 처리                          │ │
│   │ data := []byte{0x48, 0x65, 0x6c, 0x6c, 0x6f}               │ │
│   │                                                            │ │
│   │ // rune 사용: 유니코드 문자 처리                            │ │
│   │ var emoji rune = '😀'  // U+1F600                          │ │
│   │ for _, r := range "한글" {                                 │ │
│   │     fmt.Printf("%c: U+%04X\n", r, r)                       │ │
│   │ }                                                          │ │
│   │ // 한: U+D55C                                              │ │
│   │ // 글: U+AE00                                              │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

byte를 사용하면 "이 변수는 바이트 데이터를 담고 있다"는 의도가 전달되고, rune을 사용하면 "이 변수는 유니코드 문자를 담고 있다"는 의도가 전달됩니다. uint8이나 int32를 직접 사용해도 기술적으로 동일하지만, 코드의 가독성과 의도 전달 측면에서 byte와 rune을 사용하는 것이 권장됩니다.

## 2. 비교표

### 2.1 정수 타입 비교

| 타입 | 크기 (비트) | 범위 | 용도 |
|------|------------|------|------|
| `int8` | 8 | -128 ~ 127 | 작은 범위 signed 정수 |
| `int16` | 16 | -32,768 ~ 32,767 | 중간 범위 signed 정수 |
| `int32` | 32 | -2^31 ~ 2^31-1 | 큰 범위 signed 정수, rune 별칭 |
| `int64` | 64 | -2^63 ~ 2^63-1 | 매우 큰 범위 signed 정수 |
| `int` | 32/64 | 플랫폼 의존 | **기본 정수 타입** |
| `uint8` | 8 | 0 ~ 255 | 바이트 데이터, byte 별칭 |
| `uint16` | 16 | 0 ~ 65,535 | 포트 번호 등 |
| `uint32` | 32 | 0 ~ 2^32-1 | unsigned 32비트 |
| `uint64` | 64 | 0 ~ 2^64-1 | unsigned 64비트 |
| `uint` | 32/64 | 플랫폼 의존 | unsigned 기본 |

### 2.2 부동소수점 타입 비교

| 타입 | 크기 (비트) | 정밀도 | 권장 사용 |
|------|------------|--------|----------|
| `float32` | 32 | 약 6-7자리 | 호환성 필요 시에만 (드묾) |
| `float64` | 64 | 약 15-16자리 | **기본 권장** |

### 2.3 변수 선언 방식 비교

| 방식 | 문법 | 사용 위치 | 사용 시점 |
|------|------|----------|----------|
| 완전한 형태 | `var x int = 10` | 패키지/함수 | 타입과 값 모두 명시 필요 시 |
| 타입 추론 | `var x = 10` | 패키지/함수 | 타입은 추론, 값은 명시 |
| Zero Value | `var x int` | 패키지/함수 | Zero Value 의도 명시 |
| 짧은 선언 | `x := 10` | 함수 내부만 | 간결한 로컬 변수 선언 |
| 선언 블록 | `var (...)` | 패키지/함수 | 관련 변수 그룹화 |

### 2.4 문자열 리터럴 비교

| 종류 | 문법 | 특징 | 사용 예 |
|------|------|------|---------|
| Interpreted | `"..."` | 이스케이프 시퀀스 해석 | `"Hello\nWorld"` |
| Raw | `` `...` `` | 있는 그대로 (개행 포함) | 정규식, SQL 쿼리 |

## 3. 면접 예상 질문 및 모범 답안

### Q1. Go의 Zero Value 개념을 설명하고, 왜 이런 설계를 했는지 말해주세요.

**모범 답안:**

Zero Value는 Go에서 변수가 선언되었지만 명시적으로 값이 할당되지 않았을 때 자동으로 부여되는 기본값입니다. 수치 타입은 0, bool은 false, string은 빈 문자열, 포인터와 슬라이스, 맵 등 참조 타입은 nil이 Zero Value입니다.

이 설계의 첫 번째 이유는 안전성입니다. C나 C++에서는 초기화되지 않은 변수가 이전 메모리의 쓰레기 값을 가질 수 있어, 예측 불가능한 동작이나 보안 취약점의 원인이 됩니다. Go는 모든 변수에 예측 가능한 값을 보장하여 이 문제를 원천적으로 방지합니다.

두 번째 이유는 코드 명확성입니다. `var count int`라고 선언하면, 개발자는 count가 0으로 시작됨을 확신할 수 있습니다. 이는 코드의 의도를 명확히 전달하며, 특히 카운터나 누적 변수 등 0에서 시작하는 것이 자연스러운 경우에 유용합니다.

세 번째로 실용적인 이점이 있습니다. 슬라이스나 맵의 Zero Value인 nil도 append나 range 같은 기본 연산이 가능하여, 별도의 초기화 없이 바로 사용할 수 있는 경우가 많습니다.

### Q2. Go가 자동 타입 변환을 허용하지 않는 이유를 설명해주세요.

**모범 답안:**

Go가 자동 타입 변환(암시적 타입 변환)을 허용하지 않는 이유는 코드의 명확성과 안전성을 위해서입니다.

첫째, 암시적 타입 변환은 데이터 손실을 숨깁니다. float64에서 int로 변환하면 소수점 이하가 잘리고, 큰 정수 타입에서 작은 정수 타입으로 변환하면 오버플로우가 발생합니다. 자동 변환은 이런 손실이 코드에서 보이지 않게 하여 버그를 유발합니다. Go는 `int(floatVar)`처럼 명시적 변환을 요구하여 개발자가 변환 결과를 인지하도록 합니다.

둘째, 코드 가독성이 향상됩니다. 자동 타입 변환 규칙은 언어마다 다르고 복잡합니다. C에서 signed와 unsigned 정수를 섞어 연산하면 예상치 못한 결과가 나올 수 있습니다. Go에서는 모든 타입 변환이 코드에 명시되므로, 읽는 사람이 정확히 어떤 일이 일어나는지 알 수 있습니다.

셋째, Go의 철학인 "이해 가능성이 간결함보다 중요하다(comprehensibility over conciseness)"를 반영합니다. 몇 글자를 더 쓰더라도 코드의 의도와 동작이 명확한 것이 좋다고 판단한 것입니다.

실무에서는 `float64(intVar)`와 같이 타입명을 함수처럼 사용하여 변환합니다. 같은 크기의 정수 타입끼리도(int32와 int64 등) 명시적 변환이 필요합니다.

### Q3. Untyped 상수가 무엇이고, 어떤 상황에서 유용한지 설명해주세요.

**모범 답안:**

Untyped 상수는 타입이 지정되지 않은 상수로, 사용되는 맥락에 따라 다양한 타입으로 해석될 수 있습니다. `const x = 10`처럼 타입을 명시하지 않고 선언한 상수가 Untyped 상수입니다.

Untyped 상수가 유용한 첫 번째 상황은 수학적 상수를 정의할 때입니다. `const Pi = 3.14159265358979`처럼 선언하면, 이 상수는 float32 변수에도, float64 변수에도 할당할 수 있습니다. 반면 `const Pi float64 = ...`처럼 Typed로 선언하면 float32에 할당할 수 없습니다.

두 번째로 다양한 정수 타입과 함께 사용되는 상수에 유용합니다. `const MaxRetries = 3`처럼 선언하면, int, int32, uint, byte 등 다양한 정수 타입에 할당할 수 있습니다. 이는 라이브러리 코드에서 특히 유용합니다.

세 번째로 Untyped 상수는 컴파일 타임에 더 높은 정밀도로 계산됩니다. Untyped 상수끼리의 연산은 CPU의 부동소수점 정밀도 제한을 받지 않고 임의 정밀도로 계산됩니다. 최종적으로 변수에 할당될 때 해당 타입의 정밀도로 변환됩니다.

단, Untyped 상수라도 타입 호환이 안 되거나 오버플로우가 발생하면 컴파일 에러가 납니다. `var s string = 10`이나 `var b byte = 1000`은 컴파일 에러입니다.

---

### Q4. var와 := 중 어떤 것을 사용해야 하는지 상황별로 설명해주세요.

**모범 답안:**

var와 :=는 단순한 문법 차이가 아니라 코드의 의도를 표현하는 방식입니다. 상황에 따라 적절한 것을 선택해야 합니다.

var를 사용해야 하는 상황은 세 가지입니다. 첫째, Zero Value가 의도된 값일 때입니다. `var count int`는 "count는 0부터 시작한다"는 의도를 명확히 전달합니다. `count := 0`은 그냥 초기값인지, 0이 특별한 의미인지 불분명합니다. 둘째, 패키지 레벨 변수를 선언할 때입니다. :=는 함수 내부에서만 사용 가능하므로, 패키지 레벨에서는 반드시 var를 사용합니다. 셋째, 리터럴의 기본 타입과 다른 타입이 필요할 때입니다. `var port uint16 = 8080`처럼 int가 아닌 특정 타입이 필요하면 var로 타입을 명시합니다.

:=를 사용해야 하는 상황도 있습니다. 함수 반환값을 받을 때 `result, err := doSomething()`처럼 사용하면 간결합니다. 리터럴의 기본 타입을 그대로 사용할 때도 `name := "Alice"`처럼 :=가 자연스럽습니다.

주의할 점은 :=의 재할당 동작입니다. `x := 10` 후에 `x, y := 20, "hello"`를 하면 x는 재할당되고 y만 새로 선언됩니다. 최소 하나의 새 변수가 있어야 :=를 사용할 수 있습니다. 이 동작이 의도치 않은 shadowing을 유발할 수 있으므로 주의가 필요합니다.

---

### Q5. Go에서 사용하지 않는 변수가 컴파일 에러인 이유와, 이 규칙의 예외를 설명해주세요.

**모범 답안:**

Go에서 선언된 지역 변수가 한 번도 읽히지 않으면 컴파일 에러가 발생합니다. 이 규칙의 목적은 코드 품질을 강제하는 것입니다.

사용하지 않는 변수는 여러 문제의 징후일 수 있습니다. 리팩토링 과정에서 남은 잔재일 수 있고, 로직 오류로 인해 할당은 했지만 사용하지 않는 것일 수 있습니다. 또한 불필요한 변수는 코드를 읽는 사람을 혼란스럽게 합니다. Go는 이런 코드가 존재하는 것 자체를 허용하지 않습니다.

다른 언어에서는 이런 경우 경고(warning)를 발생시키지만, Go는 에러로 처리합니다. 이는 Go의 설계 철학인 "경고는 없다, 모든 것은 에러 아니면 정상"을 반영합니다.

예외가 있습니다. 첫째, 패키지 레벨 변수는 이 검사를 받지 않습니다. 패키지가 다른 곳에서 import되어 사용될 수 있기 때문입니다. 둘째, 상수도 검사를 받지 않습니다. 사용되지 않는 상수는 바이너리에 포함되지 않으므로 해가 없습니다. 셋째, 블랭크 식별자 `_`에 할당하는 것은 허용됩니다. `_, err := doSomething()`처럼 반환값 중 일부만 필요할 때 사용합니다.

### Q6. byte와 rune의 차이점과 각각 언제 사용해야 하는지 설명해주세요.

**모범 답안:**

byte와 rune은 각각 uint8과 int32의 타입 별칭입니다. 기술적으로 동일한 타입이지만 코드에서 다른 의도를 표현합니다.

byte는 0부터 255까지의 값을 담을 수 있으며, 바이트 데이터를 처리할 때 사용합니다. 파일 I/O, 네트워크 패킷, 바이너리 데이터 등 실제 바이트 스트림을 다룰 때 `[]byte`를 사용합니다. 문자열을 바이트로 변환할 때도 `[]byte(str)` 형태로 사용합니다.

rune은 유니코드 코드포인트를 표현하는 데 사용합니다. 유니코드 문자는 1바이트에서 4바이트까지 다양한 크기를 가지므로, 단일 문자를 다룰 때 rune을 사용합니다. 문자열을 range로 순회하면 각 문자가 rune으로 반환됩니다. 한글, 이모지, 특수 문자 등 ASCII가 아닌 문자를 처리할 때 특히 중요합니다.

실무에서 주의할 점은 문자열의 길이입니다. `len("한글")`은 6을 반환하는데, 이는 바이트 수입니다. 문자 수를 얻으려면 `utf8.RuneCountInString("한글")`을 사용하면 2를 반환합니다. 문자열을 인덱스로 접근하면 바이트를 얻고, range로 순회하면 rune을 얻습니다.

## 4. 실무 체크리스트

### 타입 선택 체크리스트

- [ ] 일반 정수 연산에는 int 사용
- [ ] 바이너리/네트워크 프로토콜에는 명시적 크기 정수 사용 (int32, uint16 등)
- [ ] 바이트 데이터에는 byte (uint8) 사용
- [ ] 유니코드 문자에는 rune (int32) 사용
- [ ] 부동소수점에는 float64 사용 (특별한 이유 없으면 float32 사용 안 함)
- [ ] 금전 계산에 부동소수점 사용 안 함 (decimal 라이브러리 사용)

### 변수 선언 체크리스트

- [ ] Zero Value가 의도된 값일 때 `var x int` 형태 사용
- [ ] 타입 명시가 필요할 때 `var x uint16 = 8080` 형태 사용
- [ ] 일반적인 함수 내 선언에 `:=` 사용
- [ ] 패키지 레벨 변수에 `var` 사용 (`:=` 불가)
- [ ] 관련 변수들은 선언 블록으로 그룹화

### 코드 품질 체크리스트

- [ ] 선언된 모든 지역 변수가 최소 한 번 읽힘
- [ ] 타입 변환이 명시적으로 표현됨
- [ ] 변수명이 camelCase를 따름
- [ ] 짧은 스코프에서는 짧은 변수명 사용
- [ ] 상수는 관련 있는 것끼리 const 블록으로 그룹화
