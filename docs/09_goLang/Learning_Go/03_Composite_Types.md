# 03. Composite Types 면접 정리

---

## 1. 핵심 개념 상세 설명

### 1.1 Array vs Slice

Go에서 Array와 Slice는 둘 다 순서가 있는 요소들의 컬렉션이지만, 근본적으로 다른 특성을 가집니다. Array는 고정 크기이며 크기가 타입의 일부입니다. 즉, `[3]int`와 `[4]int`는 완전히 다른 타입입니다. 이 특성 때문에 함수 파라미터로 사용하기 어렵고, 런타임에 크기를 결정할 수 없습니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Array vs Slice                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   Array (고정 크기)                                              │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │ var arr [4]int = [4]int{1, 2, 3, 4}                     │   │
│   │                                                          │   │
│   │ 메모리: [1][2][3][4]  ← 연속된 메모리 블록               │   │
│   │                                                          │   │
│   │ • 크기가 타입의 일부: [3]int ≠ [4]int                    │   │
│   │ • 컴파일 타임에 크기 결정                                 │   │
│   │ • 함수에 전달 시 전체 복사                                │   │
│   │ • 비교 가능 (==, !=)                                     │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                   │
│   Slice (가변 길이)                                              │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │ var slice []int = []int{1, 2, 3, 4}                     │   │
│   │                                                          │   │
│   │ Slice Header (24 bytes):                                 │   │
│   │ ┌─────────┬─────────┬─────────┐                         │   │
│   │ │ ptr     │ len = 4 │ cap = 4 │                         │   │
│   │ └────┬────┴─────────┴─────────┘                         │   │
│   │      │                                                   │   │
│   │      ▼                                                   │   │
│   │ Backing Array: [1][2][3][4]                             │   │
│   │                                                          │   │
│   │ • 크기가 타입이 아님: []int는 모든 길이 허용              │   │
│   │ • 런타임에 크기 변경 가능 (append)                        │   │
│   │ • 함수에 전달 시 헤더만 복사 (backing array 공유)         │   │
│   │ • nil과만 비교 가능                                      │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

Slice는 세 부분으로 구성됩니다. backing array를 가리키는 포인터, 현재 요소 수를 나타내는 length, 할당된 공간을 나타내는 capacity입니다. 이 구조 덕분에 Slice는 가변 길이를 지원하면서도 효율적으로 동작합니다. 함수에 Slice를 전달하면 24바이트 헤더만 복사되고, backing array는 공유됩니다.

### 1.2 Slice의 메모리 공유와 위험성

Slice의 가장 중요한 특성 중 하나는 메모리 공유입니다. Slice를 슬라이싱하여 새 Slice를 만들면, 두 Slice는 동일한 backing array를 공유합니다. 한 Slice를 통해 요소를 수정하면 다른 Slice에도 영향을 미칩니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Slice 메모리 공유 문제                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   x := []string{"a", "b", "c", "d"}                             │
│   y := x[:2]    // ["a", "b"]                                   │
│   z := x[1:3]   // ["b", "c"]                                   │
│                                                                   │
│   메모리 상태:                                                   │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │ Backing Array: ["a"]["b"]["c"]["d"]                     │   │
│   │                  ↑    ↑    ↑    ↑                        │   │
│   │                  │    │    │    │                        │   │
│   │ x ──────────────►│────│────│────│  len=4, cap=4         │   │
│   │ y ──────────────►│────│         │  len=2, cap=4         │   │
│   │ z ─────────────────►──│────│     │  len=2, cap=3         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                   │
│   문제 발생:                                                     │
│   x[1] = "Y"   // x = ["a", "Y", "c", "d"]                      │
│                // y = ["a", "Y"]          ← 영향받음!           │
│                // z = ["Y", "c"]          ← 영향받음!           │
│                                                                   │
│   더 심각한 문제 - append:                                       │
│   y = append(y, "!")                                             │
│   // x = ["a", "b", "!", "d"]  ← x[2]가 덮어써짐!               │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

이 문제를 해결하는 방법은 두 가지입니다. 첫째, copy 함수를 사용하여 독립적인 복사본을 만듭니다. 둘째, Full Slice Expression `[start:end:max]`를 사용하여 capacity를 제한합니다. capacity가 length와 같으면 append 시 새 backing array가 할당되어 원본에 영향을 주지 않습니다.

### 1.3 make 함수와 Slice 초기화

make 함수는 Slice, Map, Channel을 초기화하는 데 사용됩니다. Slice의 경우 길이와 용량을 지정할 수 있습니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    make 함수 사용 패턴                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   패턴 1: make([]T, length)                                      │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ x := make([]int, 5)                                       │ │
│   │ // len=5, cap=5, 값=[0, 0, 0, 0, 0]                       │ │
│   │                                                            │ │
│   │ 용도: 인덱스로 직접 접근하여 값 설정                        │ │
│   │       x[0] = 10                                           │ │
│   │       x[1] = 20                                           │ │
│   │                                                            │ │
│   │ ⚠️ 주의: append 사용 시                                   │ │
│   │ x = append(x, 100)  // [0, 0, 0, 0, 0, 100] ← 뒤에 추가! │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   패턴 2: make([]T, 0, capacity) ← 권장                          │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ x := make([]int, 0, 5)                                    │ │
│   │ // len=0, cap=5, 값=[]                                    │ │
│   │                                                            │ │
│   │ 용도: append로 요소 추가 (capacity 재할당 최소화)          │ │
│   │       x = append(x, 10)  // [10]                          │ │
│   │       x = append(x, 20)  // [10, 20]                      │ │
│   │                                                            │ │
│   │ 장점: 미리 메모리 할당으로 성능 향상                        │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   Slice 선언 의사결정:                                           │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ 상황                        │ 선언 방식                    │ │
│   │ ─────────────────────────────────────────────────────────│ │
│   │ 비어있거나 nil 가능         │ var s []int                 │ │
│   │ 초기값 있음                 │ s := []int{1, 2, 3}         │ │
│   │ 크기 알고, append 사용      │ s := make([]int, 0, n)      │ │
│   │ 크기 알고, 인덱스 접근      │ s := make([]int, n)         │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

make([]T, 5) 패턴을 사용하고 append를 호출하면 흔한 실수가 발생합니다. 이미 5개의 zero value 요소가 있으므로 append는 6번째 위치에 추가합니다. 크기를 알고 append를 사용할 계획이라면 make([]T, 0, 5)를 사용해야 합니다.

### 1.4 Map의 특성과 Comma Ok Idiom

Map은 key-value 쌍을 저장하는 해시 테이블 기반 자료구조입니다. nil map과 빈 map의 차이, 그리고 키 존재 여부를 확인하는 comma ok idiom이 중요합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Map 특성과 Comma Ok Idiom                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   nil Map vs 빈 Map                                              │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ var nilMap map[string]int     // nil, 읽기 OK, 쓰기 panic │ │
│   │ emptyMap := map[string]int{}  // non-nil, 읽기/쓰기 OK    │ │
│   │ madeMap := make(map[string]int) // non-nil                │ │
│   │                                                            │ │
│   │ nilMap["key"]        // 0 반환 (panic 아님)               │ │
│   │ nilMap["key"] = 1    // panic: assignment to nil map      │ │
│   │ emptyMap["key"] = 1  // OK                                │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   Comma Ok Idiom                                                 │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ m := map[string]int{                                      │ │
│   │     "exists": 0,                                          │ │
│   │ }                                                         │ │
│   │                                                            │ │
│   │ // 문제: 키 없음 vs 값이 0인지 구분 불가                   │ │
│   │ v := m["missing"]  // 0                                   │ │
│   │ v := m["exists"]   // 0                                   │ │
│   │                                                            │ │
│   │ // 해결: comma ok idiom                                   │ │
│   │ v, ok := m["exists"]   // v=0, ok=true                    │ │
│   │ v, ok := m["missing"]  // v=0, ok=false                   │ │
│   │                                                            │ │
│   │ // 일반적인 패턴                                          │ │
│   │ if val, ok := m[key]; ok {                                │ │
│   │     // 키가 존재할 때의 처리                               │ │
│   │ } else {                                                  │ │
│   │     // 키가 없을 때의 처리                                 │ │
│   │ }                                                         │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

Map의 키로 사용할 수 있는 타입은 비교 가능한(comparable) 타입이어야 합니다. Slice, Map, 함수는 키로 사용할 수 없습니다. 하지만 Struct는 모든 필드가 비교 가능하면 키로 사용할 수 있습니다.

### 1.5 String, Rune, Byte의 관계

Go의 문자열은 UTF-8 인코딩된 바이트 시퀀스입니다. 이 특성을 이해하지 못하면 다국어 문자열을 다룰 때 버그가 발생합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    String, Rune, Byte 관계                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   s := "Hello 한글"                                              │
│                                                                   │
│   []byte 관점 (UTF-8 인코딩):                                    │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ H   e   l   l   o   ' ' 한        글                      │ │
│   │ 48  65  6C  6C  6F  20  ED95 9C   EA B8 80                │ │
│   │ 1B  1B  1B  1B  1B  1B  3 bytes   3 bytes                 │ │
│   │                                                            │ │
│   │ len(s) = 12  ← 바이트 수                                  │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   []rune 관점 (유니코드 코드포인트):                             │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ 'H'  'e'  'l'  'l'  'o'  ' '  '한'  '글'                 │ │
│   │ U+48 U+65 U+6C U+6C U+6F U+20 U+D55C U+AE00               │ │
│   │                                                            │ │
│   │ len([]rune(s)) = 8  ← 문자 수                             │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   주의사항:                                                      │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ // 인덱스 접근은 바이트 단위!                              │ │
│   │ s[6]  // 237 (한의 첫 바이트), '한'이 아님!               │ │
│   │                                                            │ │
│   │ // 슬라이싱도 바이트 단위!                                 │ │
│   │ s[6:9]  // "한" (3바이트가 한 글자)                       │ │
│   │ s[6:8]  // 잘못된 UTF-8 (깨진 문자)                       │ │
│   │                                                            │ │
│   │ // 문자 단위 순회는 range 사용                             │ │
│   │ for i, r := range s {                                     │ │
│   │     // i: 바이트 인덱스, r: rune (문자)                   │ │
│   │ }                                                         │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

문자열을 문자 단위로 처리해야 할 때는 `[]rune`으로 변환하거나 `range`를 사용합니다. range로 문자열을 순회하면 각 반복에서 바이트 인덱스와 rune(유니코드 코드포인트)을 얻습니다.

### 1.6 Struct와 Anonymous Struct

Struct는 서로 다른 타입의 필드를 묶어 사용자 정의 타입을 만듭니다. Go에서 클래스 대신 사용되는 핵심 구조입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Struct 정의와 사용                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│   Named Struct (일반적인 사용)                                   │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ type Person struct {                                      │ │
│   │     Name string                                           │ │
│   │     Age  int                                              │ │
│   │     Tags []string                                         │ │
│   │ }                                                         │ │
│   │                                                            │ │
│   │ // 필드명 명시 (권장)                                      │ │
│   │ p := Person{                                              │ │
│   │     Name: "Alice",                                        │ │
│   │     Age:  30,                                             │ │
│   │ }                                                         │ │
│   │                                                            │ │
│   │ // 순서대로 (비권장 - 필드 추가 시 깨짐)                    │ │
│   │ p := Person{"Alice", 30, nil}                             │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   Anonymous Struct (일회성 사용)                                 │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ // 사용 시나리오 1: JSON 언마샬링                          │ │
│   │ var response struct {                                     │ │
│   │     Status  int    `json:"status"`                        │ │
│   │     Message string `json:"message"`                       │ │
│   │ }                                                         │ │
│   │ json.Unmarshal(data, &response)                           │ │
│   │                                                            │ │
│   │ // 사용 시나리오 2: 테이블 기반 테스트                      │ │
│   │ tests := []struct {                                       │ │
│   │     input    string                                       │ │
│   │     expected int                                          │ │
│   │ }{                                                        │ │
│   │     {"hello", 5},                                         │ │
│   │     {"world", 5},                                         │ │
│   │ }                                                         │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
│   Struct 비교와 변환 규칙                                        │
│   ┌───────────────────────────────────────────────────────────┐ │
│   │ type A struct { Name string; Age int }                    │ │
│   │ type B struct { Name string; Age int }                    │ │
│   │                                                            │ │
│   │ a := A{Name: "X", Age: 1}                                 │ │
│   │ b := B{Name: "X", Age: 1}                                 │ │
│   │                                                            │ │
│   │ // a == b        // 컴파일 에러 (다른 타입)               │ │
│   │ // b = a         // 컴파일 에러                           │ │
│   │ b = B(a)         // OK (명시적 변환)                      │ │
│   │                                                            │ │
│   │ // 필드 이름, 순서, 타입이 모두 같아야 변환 가능            │ │
│   └───────────────────────────────────────────────────────────┘ │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

Anonymous struct는 named type과 자유롭게 비교하고 대입할 수 있습니다. 필드 구조만 같으면 됩니다. 이 특성은 테스트 코드나 임시 데이터 구조에서 유용합니다.

## 2. 비교표

### 2.1 Array, Slice, Map 비교

| 특성 | Array | Slice | Map |
|------|-------|-------|-----|
| **크기** | 고정 (타입의 일부) | 가변 (동적 확장) | 가변 (동적 확장) |
| **Zero Value** | 모든 요소 zero value | nil | nil |
| **nil 읽기** | N/A | OK (빈 결과) | OK (zero value 반환) |
| **nil 쓰기** | N/A | panic | panic |
| **비교 (==)** | 가능 | nil만 가능 | nil만 가능 |
| **함수 전달** | 전체 복사 | 헤더만 복사 (공유) | 참조 전달 |
| **키 타입** | 정수 인덱스 | 정수 인덱스 | comparable 타입 |

### 2.2 Slice 선언 방식 비교

| 선언 방식 | 결과 | 용도 |
|----------|------|------|
| `var s []int` | nil, len=0, cap=0 | 비어있거나 nil 가능할 때 |
| `s := []int{}` | non-nil, len=0, cap=0 | JSON에서 빈 배열 필요 시 |
| `s := []int{1,2,3}` | len=3, cap=3 | 초기값이 있을 때 |
| `s := make([]int, 5)` | len=5, cap=5, [0,0,0,0,0] | 인덱스로 직접 접근 |
| `s := make([]int, 0, 5)` | len=0, cap=5 | append 사용 예정 |

### 2.3 문자열 관련 타입 비교

| 연산 | string | []byte | []rune |
|------|--------|--------|--------|
| **len()** | 바이트 수 | 바이트 수 | 문자 수 |
| **인덱스 접근** | 바이트 | 바이트 | 문자 |
| **슬라이싱** | 바이트 단위 | 바이트 단위 | 문자 단위 |
| **range** | (인덱스, rune) | (인덱스, byte) | (인덱스, rune) |
| **용도** | 텍스트 저장 | 바이너리/I/O | 문자 조작 |

## 3. 면접 예상 질문 및 모범 답안

### Q1. Go에서 Array 대신 Slice를 사용하는 이유를 설명해주세요.

**모범 답안:**

Go에서 Array보다 Slice를 선호하는 이유는 세 가지입니다.

첫째, Array의 크기가 타입의 일부이기 때문입니다. `[3]int`와 `[4]int`는 완전히 다른 타입이라서 함수 파라미터로 Array를 받으면 특정 크기의 Array만 받을 수 있습니다. 반면 Slice는 `[]int` 하나로 모든 길이의 정수 시퀀스를 처리할 수 있습니다.

둘째, Array는 컴파일 타임에 크기가 결정되어야 합니다. 변수를 사용하여 크기를 지정할 수 없어 런타임에 크기가 결정되는 상황에 대응할 수 없습니다. Slice는 make 함수나 append로 런타임에 크기를 조절할 수 있습니다.

셋째, Array는 함수에 전달할 때 전체가 복사됩니다. 1000개 요소의 Array를 전달하면 1000개 모두 복사됩니다. Slice는 헤더(포인터, length, capacity)만 복사되고 backing array는 공유되므로 효율적입니다.

Array가 사용되는 경우는 크기가 알고리즘에 정의된 경우입니다. 예를 들어 SHA256 해시 함수의 결과는 항상 32바이트이므로 `[32]byte`를 사용합니다. 또한 Slice의 backing store로 암묵적으로 Array가 사용됩니다.

### Q2. Slice의 length와 capacity의 차이를 설명하고, capacity 관리가 중요한 이유를 말해주세요.

**모범 답안:**

Length는 Slice에 현재 저장된 요소의 개수이고, Capacity는 Slice가 재할당 없이 저장할 수 있는 최대 요소 개수입니다. Length는 `len()` 함수로, Capacity는 `cap()` 함수로 확인합니다.

Capacity 관리가 중요한 이유는 성능 때문입니다. append로 요소를 추가할 때 length가 capacity를 초과하면 Go 런타임은 새로운 backing array를 할당하고, 기존 데이터를 복사한 다음, 새 요소를 추가합니다. 이 과정은 메모리 할당과 복사 비용이 발생합니다.

예를 들어 1000개의 요소를 추가할 것을 알면서 capacity 없이 시작하면, capacity가 1, 2, 4, 8, 16... 으로 증가하면서 약 10번의 재할당이 발생합니다. 하지만 `make([]int, 0, 1000)`으로 시작하면 재할당이 전혀 없습니다.

`make([]int, 0, n)`과 `make([]int, n)`의 차이도 중요합니다. 전자는 length가 0이어서 append로 요소를 추가하면 처음부터 채워집니다. 후자는 length가 n이어서 이미 n개의 zero value가 있고, append하면 n+1번째부터 추가됩니다. 용도에 맞게 선택해야 합니다.

### Q3. Slice를 슬라이싱할 때 발생할 수 있는 문제와 해결 방법을 설명해주세요.

**모범 답안:**

Slice를 슬라이싱하면 새 Slice 헤더가 생성되지만, backing array는 원본과 공유됩니다. 이로 인해 두 가지 문제가 발생합니다.

첫째, 한 Slice를 통한 수정이 다른 Slice에 영향을 미칩니다. `x[1] = "new"`를 하면 같은 backing array를 공유하는 모든 Slice에서 해당 위치가 변경됩니다. 이것이 의도된 동작인지 아닌지 주의해야 합니다.

둘째, subslice에 append할 때 원본이 덮어써질 수 있습니다. subslice의 capacity가 원본과 공유되므로, append가 원본의 뒷부분을 덮어쓸 수 있습니다. 이것은 찾기 어려운 버그를 만듭니다.

해결 방법은 세 가지입니다. 첫째, copy 함수를 사용하여 독립적인 복사본을 만듭니다. `copy(dest, src)`는 backing array를 공유하지 않는 새 Slice를 만듭니다. 둘째, Full Slice Expression `x[start:end:max]`를 사용합니다. 세 번째 인자로 capacity를 length와 같게 제한하면, append 시 새 backing array가 할당됩니다. 셋째, Go 1.21의 slices.Clone 함수를 사용하면 간편하게 독립적인 복사본을 만들 수 있습니다.

### Q4. Map에서 comma ok idiom이 필요한 이유와 사용 방법을 설명해주세요.

**모범 답안:**

Map에서 존재하지 않는 키로 값을 읽으면 해당 값 타입의 zero value가 반환됩니다. 문제는 키가 존재하고 값이 zero value인 경우와 키가 존재하지 않는 경우를 구분할 수 없다는 것입니다.

예를 들어 `map[string]int`에서 `m["key"]`가 0을 반환했을 때, 키가 없어서 0인지, 키가 있고 값이 0인지 알 수 없습니다. 이것이 comma ok idiom이 필요한 이유입니다.

사용 방법은 두 개의 반환값을 받는 것입니다. `value, ok := m[key]` 형태로 사용하면, ok는 키 존재 여부를 나타내는 boolean입니다. ok가 true이면 키가 존재하고, false이면 키가 없습니다.

일반적인 패턴은 if 문과 함께 사용하는 것입니다. `if val, ok := m[key]; ok { ... }` 형태로 키가 존재할 때만 특정 로직을 실행할 수 있습니다. 이 패턴은 값만 필요하고 존재 여부 확인이 필요하지 않을 때는 `val := m[key]`로 간단히 쓸 수 있고, 존재 여부만 확인할 때는 `_, ok := m[key]`로 값을 무시할 수 있습니다.

### Q5. Go 문자열이 UTF-8이라는 것이 실무에서 어떤 의미가 있는지 설명해주세요.

**모범 답안:**

Go 문자열이 UTF-8 인코딩된 바이트 시퀀스라는 것은 문자열 처리 시 바이트와 문자의 차이를 인식해야 한다는 의미입니다.

가장 중요한 차이는 `len()` 함수입니다. `len("Hello")`는 5를 반환하지만, `len("한글")`은 6을 반환합니다. 한글 한 글자가 UTF-8에서 3바이트이기 때문입니다. 문자 수를 얻으려면 `utf8.RuneCountInString()` 함수나 `len([]rune(s))`를 사용해야 합니다.

인덱스 접근도 주의가 필요합니다. `s[0]`은 첫 번째 문자가 아니라 첫 번째 바이트를 반환합니다. ASCII 문자열에서는 문제가 없지만, 한글이나 이모지가 포함되면 잘못된 결과가 나옵니다. 슬라이싱도 마찬가지로 `s[0:3]`이 첫 세 문자가 아니라 첫 세 바이트입니다.

문자 단위로 처리하려면 세 가지 방법이 있습니다. 첫째, `[]rune`으로 변환하면 문자 단위로 인덱싱할 수 있습니다. 둘째, `range`로 순회하면 각 반복에서 문자(rune)를 얻습니다. 셋째, `unicode/utf8` 패키지의 함수들을 사용할 수 있습니다.

실무에서는 영문만 처리하는 경우에도 미래에 다국어 지원이 필요할 수 있으므로, 처음부터 UTF-8을 고려한 코드를 작성하는 것이 좋습니다.

### Q6. nil slice와 빈 slice의 차이점과 각각 언제 사용해야 하는지 설명해주세요.

**모범 답안:**

nil slice는 `var s []int`로 선언된 초기화되지 않은 Slice이고, 빈 slice는 `s := []int{}`로 생성된 길이가 0인 Slice입니다. 둘 다 `len(s) == 0`이고, append와 range가 정상 동작합니다. 하지만 `s == nil` 비교 결과가 다릅니다.

nil slice를 사용해야 하는 경우가 더 많습니다. Go의 관용적 코드에서 "값이 없음"을 표현할 때 nil을 사용합니다. 함수가 결과가 없을 때 nil을 반환하면, 호출자는 `if result == nil` 또는 `if len(result) == 0`으로 확인할 수 있습니다. 메모리도 nil slice는 할당이 없어 더 효율적입니다.

빈 slice를 사용해야 하는 경우는 JSON 직렬화입니다. nil slice를 JSON으로 변환하면 `null`이 되고, 빈 slice는 `[]`가 됩니다. API 응답에서 빈 배열을 보내야 할 때는 `[]int{}`를 사용합니다. 예를 들어 `{"items": null}` 대신 `{"items": []}`를 원하면 빈 slice를 사용해야 합니다.

실무 가이드라인은 기본적으로 nil slice를 선호하고, JSON에서 빈 배열이 필요한 경우에만 빈 slice 리터럴을 사용하는 것입니다. 빈 slice를 반환하는 것보다 nil을 반환하는 것이 더 관용적인 Go 코드입니다.

## 4. 실무 체크리스트

### Slice 사용 체크리스트

- [ ] 크기를 모르면 nil slice (`var s []int`) 사용
- [ ] append 사용 예정이면 `make([]T, 0, capacity)` 사용
- [ ] 인덱스 접근 예정이면 `make([]T, length)` 사용
- [ ] subslice 반환 시 copy 또는 Full Slice Expression 사용
- [ ] 함수에서 Slice를 수정할 때 원본 영향 여부 문서화

### Map 사용 체크리스트

- [ ] 선언만 하고 초기화 안 하면 nil (쓰기 시 panic)
- [ ] zero value와 키 없음 구분 필요 시 comma ok idiom 사용
- [ ] 반복 순회 순서가 보장되지 않음 인지
- [ ] 동시성 환경에서 sync.Map 또는 뮤텍스 사용

### 문자열 처리 체크리스트

- [ ] `len()`은 바이트 수, 문자 수가 아님
- [ ] 인덱스 접근은 바이트 단위
- [ ] 문자 단위 처리 시 `[]rune` 변환 또는 `range` 사용
- [ ] 다국어 지원 필요 시 UTF-8 고려한 설계
