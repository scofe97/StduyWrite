# Chapter 8: Generics - 면접 정리

## 핵심 개념 상세 설명

### 1. 제네릭이 필요한 이유

Go 1.18 이전에는 여러 타입에 동일한 로직을 적용하려면 두 가지 선택지가 있었습니다. 첫째, 타입마다 별도의 코드를 작성하는 것으로 코드 중복이 발생합니다. 둘째, 인터페이스와 any를 사용하는 것으로 타입 안전성이 상실됩니다.

```
┌────────────────────────────────────────────────────────────────┐
│                 제네릭 이전의 문제점                            │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  방법 1: 타입별 구현              방법 2: 인터페이스 + any      │
│  ─────────────────────           ───────────────────────       │
│  type IntTree struct { ... }     type Tree struct {             │
│  type StringTree struct { ... }      val Orderable              │
│  type Float64Tree struct { ... }     left, right *Tree          │
│                                  }                              │
│                                                                 │
│  문제: 코드 중복                  문제: 타입 안전성 상실        │
│       유지보수 어려움                  런타임 panic 가능        │
│                                                                 │
│  ──────────────────────────────────────────────────────────    │
│                                                                 │
│  제네릭 해결책                                                  │
│  ─────────────                                                  │
│  type Tree[T any] struct {                                      │
│      val T                                                      │
│      left, right *Tree[T]                                       │
│  }                                                              │
│                                                                 │
│  장점: 단일 구현 + 컴파일 타임 타입 체크                        │
└────────────────────────────────────────────────────────────────┘
```

제네릭은 타입 파라미터를 통해 코드 중복을 줄이면서 컴파일 타임에 타입 안전성을 보장합니다.

### 2. 제네릭 기본 문법

제네릭 타입과 함수는 대괄호 안에 타입 파라미터와 제약을 선언합니다. 타입 파라미터 이름은 관례적으로 대문자 한 글자(T, K, V 등)를 사용합니다.

```
┌────────────────────────────────────────────────────────────────┐
│              제네릭 문법 구조                                   │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  타입 선언:                                                     │
│  type Stack[T any] struct { vals []T }                          │
│            ↑  ↑                                                 │
│            │  └── 제약 (constraint)                             │
│            └── 타입 파라미터                                    │
│                                                                 │
│  함수 선언:                                                     │
│  func Map[T1, T2 any](s []T1, f func(T1) T2) []T2              │
│          ↑           ↑                                          │
│          │           └── 파라미터와 반환 타입에서 사용          │
│          └── 복수 타입 파라미터                                 │
│                                                                 │
│  메서드 선언:                                                   │
│  func (s *Stack[T]) Push(val T) { ... }                         │
│              ↑                                                  │
│              └── receiver에서 타입 파라미터 사용               │
│                  (추가 타입 파라미터 불가!)                     │
└────────────────────────────────────────────────────────────────┘
```

제네릭 타입 사용 시 타입 인자를 명시합니다: `var intStack Stack[int]`. 컴파일러가 추론 가능하면 생략할 수 있습니다.

### 3. 타입 제약 (Type Constraints)

타입 제약은 타입 파라미터가 지원해야 하는 기능을 정의합니다. Go는 세 가지 내장 제약을 제공합니다.

```
┌────────────────────────────────────────────────────────────────┐
│                타입 제약 종류                                   │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  any                                                            │
│  ─────                                                          │
│  - 모든 타입 허용                                               │
│  - ==, != 등 연산자 사용 불가                                   │
│  - interface{} 와 동일                                          │
│                                                                 │
│  comparable                                                     │
│  ──────────                                                     │
│  - == 와 != 연산자 사용 가능                                    │
│  - map의 키 타입으로 사용 가능                                  │
│  - 주의: 인터페이스 타입은 런타임 panic 가능                    │
│                                                                 │
│  커스텀 인터페이스                                              │
│  ──────────────                                                 │
│  type Stringer interface { String() string }                    │
│  - 메서드 구현 요구                                             │
│  - 제네릭 인터페이스도 가능: Differ[T any]                      │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

### 4. Type Terms와 ~ 연산자

타입 요소(Type Terms)는 연산자 지원을 위해 허용되는 타입을 직접 나열합니다. `|`로 여러 타입을 결합합니다.

```
┌────────────────────────────────────────────────────────────────┐
│           Type Terms와 ~ 연산자                                 │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  기본 Type Terms:                                               │
│  type Integer interface {                                       │
│      int | int8 | int16 | int32 | int64                        │
│  }                                                              │
│                                                                 │
│  type MyInt int                                                 │
│  divAndRemainder[MyInt](10, 3)  // 에러! MyInt ≠ int            │
│                                                                 │
│  ~ 연산자 (underlying type 포함):                               │
│  type Integer interface {                                       │
│      ~int | ~int8 | ~int16 | ~int32 | ~int64                   │
│  }                                                              │
│                                                                 │
│  divAndRemainder[MyInt](10, 3)  // OK! MyInt의 underlying = int │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  type MyInt int                                          │   │
│  │       │                                                  │   │
│  │       ▼                                                  │   │
│  │  underlying type: int                                    │   │
│  │                                                          │   │
│  │  int  제약  ──→  MyInt 불일치 ❌                         │   │
│  │  ~int 제약  ──→  MyInt 일치 ✅ (underlying이 int)        │   │
│  └─────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────┘
```

Go 1.21의 cmp.Ordered는 비교 연산자를 지원하는 모든 타입을 포함합니다: `~int | ~float64 | ~string | ...`.

### 5. 타입 추론

컴파일러가 함수 인자에서 타입을 추론할 수 있으면 타입 인자를 생략할 수 있습니다. 반환 타입만 제네릭인 경우에는 모든 타입 인자를 명시해야 합니다.

```go
// 추론 가능: 인자에서 타입 결정
words := []string{"a", "b", "c"}
lengths := Map(words, func(s string) int { return len(s) })
// Map[string, int] 자동 추론

// 추론 불가: 반환 타입만 제네릭
func Convert[T1, T2 Integer](in T1) T2 { return T2(in) }
b := Convert[int, int64](a)  // 모든 타입 인자 명시 필요
```

### 6. 제네릭과 성능

Go의 제네릭은 컴파일 시 underlying type별로 함수를 생성합니다. 모든 포인터 타입은 단일 함수를 공유하며, 이때 런타임 타입 조회가 추가됩니다.

```
┌────────────────────────────────────────────────────────────────┐
│              제네릭 컴파일 전략                                 │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Go 제네릭 컴파일                                               │
│       │                                                         │
│       ├──→ Underlying Type별 함수 생성                         │
│       │    (int, string, float64 각각)                         │
│       │                                                         │
│       └──→ 모든 포인터 타입은 단일 함수 공유                   │
│            (런타임 타입 조회 추가 → 일부 성능 저하)            │
│                                                                 │
│  벤치마크 결과 (trivial 함수, Go 1.20):                        │
│  - 인터페이스 파라미터: 기준                                   │
│  - 제네릭 파라미터: 약 30% 느림                                │
│                                                                 │
│  권장사항:                                                      │
│  - 성능 목적으로 인터페이스를 제네릭으로 변환하지 말 것        │
│  - 제네릭은 유지보수성과 타입 안전성이 목적                    │
│  - 프로파일링 후 실제 병목만 최적화                            │
└────────────────────────────────────────────────────────────────┘
```

### 7. comparable의 함정

인터페이스 타입은 comparable을 만족하지만, 내부에 비교 불가능한 타입을 담으면 런타임에 panic이 발생합니다.

```
┌────────────────────────────────────────────────────────────────┐
│            comparable 주의사항                                  │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  comparable 제약                                                │
│       │                                                         │
│       ├──→ 구체 타입?                                          │
│       │         │                                               │
│       │    Yes  ├──→ 비교 가능? (int, string 등)               │
│       │         │        │                                      │
│       │         │   Yes ─┴──→ ✅ 컴파일 OK, 런타임 OK          │
│       │         │   No  ────→ ❌ 컴파일 에러 (slice 등)        │
│       │         │                                               │
│       │    No   └──→ 인터페이스 타입                           │
│       │                   │                                     │
│       │              ⚠️ 컴파일 OK                               │
│       │                   │                                     │
│       │              런타임에 underlying 검사                   │
│       │                   │                                     │
│       │              비교 가능 ──→ ✅ OK                        │
│       │              비교 불가 ──→ 💥 panic                     │
│                                                                 │
│  예시:                                                          │
│  var a Thinger = ThingerSlice{1,2,3}  // slice는 비교 불가      │
│  Comparer(a, a)  // 컴파일 OK, 런타임 panic!                    │
└────────────────────────────────────────────────────────────────┘
```

### 8. 제네릭의 제한사항

Go 제네릭은 의도적으로 단순하게 설계되어 일부 기능이 없습니다.

메서드에 추가 타입 파라미터를 선언할 수 없습니다. 예를 들어 `func (s Slice[T]) Map[E any](f func(T) E)` 형태는 불가능합니다. 대안으로 일반 함수를 사용합니다.

연산자 오버로딩이 없어서 사용자 정의 타입에 `[]`나 `range`를 적용할 수 없습니다. 가변 타입 파라미터(`...T`)도 지원하지 않습니다.

## 비교표

### 제네릭 vs 인터페이스 선택

| 상황 | 권장 | 이유 |
|------|------|------|
| 동일 로직, 다른 타입 | 제네릭 | 코드 중복 제거, 타입 안전성 |
| 다른 구현, 공통 동작 | 인터페이스 | 다형성 필요 |
| 성능 중요 (trivial 함수) | 인터페이스 | 제네릭이 약간 느림 |
| 컴파일 타임 타입 체크 필요 | 제네릭 | 런타임 에러 방지 |
| 메서드에서 타입 변환 필요 | 함수 | 메서드 타입 파라미터 불가 |

### 타입 제약 비교

| 제약 | 허용 타입 | 사용 가능 연산자 |
|------|----------|----------------|
| any | 모든 타입 | 없음 |
| comparable | 비교 가능 타입 | ==, != |
| Ordered (cmp) | 정렬 가능 타입 | <, <=, >, >= |
| 커스텀 Type Terms | 명시된 타입만 | 해당 타입의 연산자 |

### Type Term 문법

| 문법 | 의미 | 예시 |
|------|------|------|
| `int` | 정확히 int 타입만 | `type MyInt int` 불포함 |
| `~int` | underlying type이 int인 모든 타입 | `type MyInt int` 포함 |
| `int \| string` | int 또는 string | 두 타입 모두 허용 |
| `~int \| ~string` | underlying이 int 또는 string | 파생 타입 모두 포함 |

## 면접 예상 질문 및 모범 답안

### Q1. Go에서 제네릭이 필요한 이유와 인터페이스 기반 접근법의 한계를 설명하세요.

**모범 답안:**

제네릭 도입 전에는 여러 타입에 동일한 로직을 적용하려면 두 가지 방법이 있었습니다.

첫째, 타입마다 별도의 코드를 작성하는 것입니다. IntTree, StringTree처럼 타입별로 구현하면 코드 중복이 발생하고 유지보수가 어렵습니다.

둘째, 인터페이스와 any를 사용하는 것입니다. 공통 인터페이스를 정의하고 any 타입을 사용하면 코드 중복은 줄지만 타입 안전성이 상실됩니다. 예를 들어 int를 담는 Tree에 string을 삽입해도 컴파일 에러가 발생하지 않고, 런타임에 panic이 발생합니다.

제네릭은 이 두 문제를 모두 해결합니다. `Tree[T any]`처럼 타입 파라미터를 사용하면 단일 구현으로 여러 타입을 지원하면서, 컴파일 타임에 타입 체크가 이루어집니다. `Tree[int]`에 string을 삽입하려 하면 컴파일 에러가 발생합니다.

표준 라이브러리에서도 math.Max/Min이 float64만 지원하거나, sort.Slice가 reflection을 사용하는 등의 제약이 있었는데, 제네릭으로 이런 문제들이 해결되었습니다.

### Q2. Go의 ~ 연산자가 필요한 이유와 사용법을 설명하세요.

**모범 답안:**

~ 연산자는 underlying type 기반 매칭을 위해 사용합니다.

Go에서 `type MyInt int`로 새 타입을 선언하면 MyInt와 int는 다른 타입입니다. Type Terms로 `int`만 명시하면 MyInt는 해당 제약을 만족하지 않습니다.

```go
type Integer interface {
    int | int64  // 정확히 int, int64만 허용
}

type MyInt int
divAndRemainder[MyInt](10, 3)  // 에러: MyInt는 int가 아님
```

~ 연산자를 붙이면 해당 타입을 underlying type으로 가지는 모든 타입을 허용합니다.

```go
type Integer interface {
    ~int | ~int64  // underlying이 int 또는 int64인 모든 타입
}

type MyInt int
divAndRemainder[MyInt](10, 3)  // OK: MyInt의 underlying은 int
```

이 기능은 사용자 정의 타입에서 특히 유용합니다. 예를 들어 cmp.Ordered는 `~int | ~float64 | ~string | ...` 형태로 정의되어 있어서, ID나 Score 같은 사용자 정의 타입도 정렬 함수에 사용할 수 있습니다.

### Q3. Go에서 comparable 제약을 사용할 때 주의할 점을 설명하세요.

**모범 답안:**

comparable 제약의 주요 함정은 인터페이스 타입과 관련됩니다.

comparable은 == 연산자를 사용할 수 있는 타입을 의미합니다. 구체적인 타입(int, string 등)은 컴파일 타임에 비교 가능 여부를 검사할 수 있습니다. slice나 map처럼 비교 불가능한 타입을 직접 사용하면 컴파일 에러가 발생합니다.

문제는 인터페이스 타입입니다. 인터페이스 타입은 comparable을 만족합니다. 하지만 인터페이스가 내부적으로 비교 불가능한 타입(slice 등)을 담고 있으면 런타임에 panic이 발생합니다.

```go
type Thinger interface { Thing() }
type ThingerSlice []int  // slice는 비교 불가

func (t ThingerSlice) Thing() {}

func Comparer[T comparable](t1, t2 T) { ... }

var a Thinger = ThingerSlice{1, 2, 3}
Comparer(a, a)  // 컴파일 OK, 런타임 panic!
```

대응 방법은 두 가지입니다. 첫째, 가능하면 인터페이스 대신 구체적인 타입을 사용합니다. 둘째, 인터페이스를 사용해야 하면 내부에 비교 불가능한 타입이 들어가지 않도록 주의합니다.

### Q4. Go 제네릭의 성능 특성과 인터페이스와의 차이를 설명하세요.

**모범 답안:**

Go 제네릭은 컴파일 시 특수화(specialization)를 수행합니다. underlying type별로 별도의 함수 코드를 생성합니다. 예를 들어 `Max[int]`와 `Max[int64]`는 별도의 기계어 코드가 됩니다.

하지만 모든 포인터 타입은 단일 함수를 공유합니다. 이 경우 런타임에 실제 타입을 조회하는 과정이 추가됩니다. 벤치마크에 따르면 Go 1.20 기준으로 trivial한 함수에서 제네릭이 인터페이스 파라미터보다 약 30% 느릴 수 있습니다.

따라서 성능 최적화 목적으로 인터페이스를 제네릭으로 바꾸는 것은 권장되지 않습니다. 제네릭의 주요 이점은 유지보수성과 타입 안전성입니다. 성능이 중요하다면 프로파일링 후 실제 병목 지점만 최적화해야 합니다.

한편, 정렬 같은 복잡한 연산에서는 제네릭이 reflection 기반 인터페이스보다 빠를 수 있습니다. sort.Slice는 reflection을 사용하는데, slices.Sort는 제네릭을 사용하여 더 효율적입니다.

---

### Q5. Go 제네릭에서 지원하지 않는 기능과 그 대안을 설명하세요.

**모범 답안:**

Go 제네릭은 의도적으로 단순하게 설계되어 몇 가지 기능이 없습니다.

첫째, 메서드에 추가 타입 파라미터를 선언할 수 없습니다. `func (s Slice[T]) Map[E any](f func(T) E)` 형태는 불가능합니다. 대안으로 일반 함수를 사용합니다: `filtered := Filter(slice, pred); mapped := Map(filtered, fn)`.

둘째, 연산자 오버로딩이 없습니다. 사용자 정의 타입에 `[]`나 `range`를 적용할 수 없습니다. 대안으로 메서드를 제공합니다: `Get(i)`, `Set(i, v)`, `Iterator()` 등.

셋째, 가변 타입 파라미터가 없습니다. `func Tuple[T1, T2, T3... any]()` 형태는 불가능합니다. 대안으로 타입 개수별로 별도 타입을 정의하거나, any slice를 사용합니다.

넷째, 타입별 특화 구현(Specialization)이 없습니다. `Max[int]`와 `Max[float64]`에 다른 최적화를 적용할 수 없습니다. 대안으로 별도의 함수를 제공합니다.

이런 제한은 Go의 단순성 철학을 반영합니다. 복잡한 제네릭 시스템보다 읽기 쉬운 코드를 우선시합니다.

---

### Q6. slices, maps, cmp 패키지의 제네릭 함수 활용법을 설명하세요.

**모범 답안:**

Go 1.21부터 표준 라이브러리에 제네릭 기반 유틸리티 패키지가 추가되었습니다.

slices 패키지는 슬라이스 조작 함수를 제공합니다. `slices.Contains(s, v)`로 요소 포함 여부를 확인하고, `slices.Index(s, v)`로 위치를 찾습니다. `slices.Sort(s)`는 정렬, `slices.BinarySearch(s, v)`는 이진 검색을 수행합니다. `slices.Insert`, `slices.Delete`로 요소를 추가/삭제할 수 있습니다.

maps 패키지는 맵 조작 함수를 제공합니다. `maps.Clone(m)`으로 얕은 복사, `maps.Equal(m1, m2)`로 비교, `maps.DeleteFunc(m, fn)`으로 조건부 삭제가 가능합니다.

cmp 패키지는 비교 함수를 제공합니다. `cmp.Compare(a, b)`는 -1, 0, 1을 반환하고, `cmp.Less(a, b)`는 a < b 여부를 반환합니다. cmp.Ordered 인터페이스는 모든 정렬 가능 타입을 포함합니다.

직접 구현하는 대신 표준 라이브러리를 사용하면 테스트된 코드를 활용할 수 있고, 코드 일관성도 높아집니다. 예를 들어 Contains 함수를 직접 작성하지 않고 `slices.Contains`를 사용합니다.

## 실무 체크리스트

### 제네릭 사용 판단
- [ ] 타입만 다르고 로직이 동일한 코드 중복이 있는가?
- [ ] 컴파일 타임 타입 체크가 필요한가?
- [ ] 인터페이스보다 제네릭이 적합한 상황인가?

### 타입 제약 설계
- [ ] any 대신 더 구체적인 제약이 가능한가?
- [ ] comparable 사용 시 인터페이스 타입 주의
- [ ] underlying type 지원 필요 시 ~ 연산자 사용
- [ ] cmp.Ordered 같은 표준 제약 활용

### 구현 패턴
- [ ] 메서드가 아닌 함수로 Map/Filter/Reduce 구현
- [ ] 비교 함수 타입으로 유연성 확보: `OrderableFunc[T]`
- [ ] zero value 반환 패턴: `var zero T; return zero, false`

### 표준 라이브러리 활용
- [ ] slices 패키지: Contains, Sort, BinarySearch 등
- [ ] maps 패키지: Clone, Equal, DeleteFunc 등
- [ ] cmp 패키지: Compare, Less, Ordered

### 성능 고려
- [ ] 성능 목적으로 인터페이스를 제네릭으로 변환하지 않기
- [ ] 프로파일링 후 실제 병목만 최적화
- [ ] trivial 함수에서는 인터페이스가 더 빠를 수 있음
