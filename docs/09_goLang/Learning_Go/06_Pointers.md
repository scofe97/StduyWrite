# Chapter 6: Pointers - 면접 정리

## 핵심 개념 상세 설명

### 1. 포인터 기초와 연산자

Go의 포인터는 메모리 주소를 저장하는 변수입니다. 포인터를 다루기 위해 세 가지 핵심 연산자가 존재합니다. `&`(Address Operator)는 값의 메모리 주소를 반환하고, `*`(Indirection/Dereference Operator)는 포인터가 가리키는 실제 값을 반환합니다. `*T`는 T 타입을 가리키는 포인터 타입을 나타냅니다.

```
┌─────────────────────────────────────────────────────────────┐
│                     메모리 구조                              │
├─────────────────────────────────────────────────────────────┤
│  주소 0x1000:  x = 10        (int)                          │
│  주소 0x1008:  pointerX = 0x1000  (*int, x를 가리킴)        │
│                                                              │
│  &x          → 0x1000  (x의 주소)                           │
│  *pointerX   → 10      (pointerX가 가리키는 값)             │
│  *pointerX = 20        → x가 20으로 변경됨                  │
└─────────────────────────────────────────────────────────────┘
```

포인터의 Zero Value는 `nil`입니다. nil 포인터를 역참조하면 panic이 발생하므로 반드시 nil 체크가 필요합니다.

### 2. 포인터 생성 방법과 제약

포인터를 생성하는 방법은 크게 세 가지입니다. 첫째, `&` 연산자를 사용하여 기존 변수의 주소를 얻는 방법이 가장 일반적입니다. 둘째, `new(T)` 함수를 사용하면 T 타입의 zero value를 가리키는 포인터를 생성하지만, 거의 사용되지 않습니다. 셋째, 구조체 리터럴에 `&`를 적용하여 직접 포인터를 생성할 수 있습니다.

중요한 제약사항으로 리터럴이나 상수의 주소는 직접 얻을 수 없습니다. `p := &10`은 컴파일 에러입니다. 이를 해결하려면 변수로 먼저 저장하거나, 제네릭 헬퍼 함수를 사용합니다.

```go
func makePointer[T any](t T) *T {
    return &t
}
p := makePointer(10)  // *int
```

### 3. Go는 Call by Value 언어

Go는 순수한 Call by Value 언어입니다. 함수에 인자를 전달할 때 항상 값이 복사됩니다. 포인터를 전달해도 포인터 값(메모리 주소) 자체가 복사됩니다. 이로 인해 함수 내에서 포인터 자체를 재할당해도 원본에 영향을 주지 않습니다.

```
┌────────────────────────────────────────────────────────────────┐
│            포인터 전달과 재할당의 동작                          │
├────────────────────────────────────────────────────────────────┤
│  main():                                                        │
│    f := &Foo{x: 10}     f ──────────┐                          │
│                                      ↓                          │
│  inner(f *Foo):                   [Foo{x:10}] (0x1000)         │
│    g = f (복사)         g ──────────┘                          │
│                                                                 │
│  g = &Foo{x: 30}        g ──→ [Foo{x:30}] (0x2000, 새로운 객체)│
│                         f ──→ [Foo{x:10}] (0x1000, 원본 그대로)│
│                                                                 │
│  원본 수정하려면: *g = ... 또는 g.field = ... 사용              │
└────────────────────────────────────────────────────────────────┘
```

### 4. 포인터는 Mutability를 의미

Go에는 `const`나 `immutable` 키워드가 없습니다. 대신 값 타입 파라미터는 원본을 수정할 수 없고(immutable처럼 동작), 포인터 파라미터는 원본을 수정할 수 있습니다(mutable). 따라서 API를 설계할 때 포인터 파라미터는 "이 함수가 원본을 수정할 수 있다"는 신호입니다.

### 5. 포인터는 최후의 수단

Go에서는 기본적으로 값을 사용하고, 포인터는 필요한 경우에만 사용해야 합니다. 값을 사용하면 데이터 흐름이 명확해지고, 불변성이 보장되며, 가비지 컬렉터 부담이 줄어듭니다.

```
┌────────────────────────────────────────────────────────────────┐
│              포인터 사용 결정 트리                              │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  함수 파라미터가 필요한가?                                      │
│           │                                                     │
│           ▼                                                     │
│  원본 수정이 필요한가? ──No──→ 값 사용 ✅                       │
│           │                                                     │
│          Yes                                                    │
│           ▼                                                     │
│  nil이 의미 있는 값인가? ──Yes──→ 포인터 사용                   │
│           │                                                     │
│          No                                                     │
│           ▼                                                     │
│  데이터 크기가 10MB 이상인가? ──Yes──→ 포인터 사용              │
│           │                                                     │
│          No                                                     │
│           ▼                                                     │
│      값 사용 ✅ (또는 벤치마크로 결정)                          │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

포인터 파라미터가 필요한 경우: (1) `json.Unmarshal`처럼 인터페이스를 받는 함수, (2) 버퍼 재사용으로 메모리 할당 최소화, (3) 상태를 수정해야 하는 I/O 작업, (4) 동시성에서 공유 상태 관리.

### 6. Map과 Slice의 내부 구현

Map은 내부적으로 포인터로 구현되어 있어 함수에 전달하면 원본 데이터를 수정할 수 있습니다. 키-값 추가, 수정, 삭제 모두 원본에 반영됩니다.

Slice는 세 개의 필드를 가진 구조체로 구현됩니다: length, capacity, 그리고 실제 데이터를 가리키는 포인터. 함수에 전달되면 이 구조체가 복사됩니다.

```
┌────────────────────────────────────────────────────────────────┐
│                  Slice 내부 구조                                │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  s := []int{1, 2, 3}                                           │
│                                                                 │
│  ┌──────────────────┐                                          │
│  │  Slice Header    │                                          │
│  │  len: 3          │                                          │
│  │  cap: 3          │     ┌───┬───┬───┐                        │
│  │  ptr ───────────────→ │ 1 │ 2 │ 3 │  (메모리 블록)          │
│  └──────────────────┘     └───┴───┴───┘                        │
│                                                                 │
│  함수에 전달 시:                                                │
│  - 요소 수정 s[0] = 100  → 원본 반영됨 (같은 메모리 블록)       │
│  - append s = append(s, 4) → 원본 반영 안됨 (len만 복사본 변경) │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

### 7. Stack vs Heap과 Escape Analysis

스택은 함수 호출 시 자동으로 관리되는 메모리 영역으로, 할당과 해제가 매우 빠릅니다. 힙은 동적으로 할당되는 메모리 영역으로, 가비지 컬렉터가 관리합니다.

Go 컴파일러는 Escape Analysis를 통해 변수를 스택에 할당할지 힙에 할당할지 결정합니다. 포인터가 함수 외부로 반환되거나, 인터페이스에 할당되거나, 클로저에서 캡처되면 힙으로 escape합니다.

```
┌────────────────────────────────────────────────────────────────┐
│              Stack vs Heap 비교                                 │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│       Stack (빠름)              │        Heap (느림)            │
│  ───────────────────────────────┼─────────────────────────────  │
│  - 컴파일 타임에 크기 결정      │  - 런타임에 동적 크기         │
│  - 함수 종료 시 자동 해제       │  - GC가 관리                  │
│  - 순차적 메모리 접근           │  - 랜덤 메모리 접근           │
│  - 포인터 반환 없음             │  - 포인터 반환 시             │
│                                                                 │
│  func stackAlloc() int {        │  func heapAlloc() *int {      │
│      x := 10                    │      x := 10                  │
│      return x  // 값 반환       │      return &x // 포인터 반환 │
│  }                              │  }                            │
│                                                                 │
│  확인: go build -gcflags="-m" main.go                          │
└────────────────────────────────────────────────────────────────┘
```

### 8. 가비지 컬렉터 튜닝

Go의 GC는 낮은 지연시간(500µs 미만)을 목표로 설계되었습니다. GOGC와 GOMEMLIMIT 환경변수로 GC 동작을 튜닝할 수 있습니다.

GOGC는 GC 목표 비율을 설정합니다. 기본값 100은 현재 힙의 100% 증가 시 GC를 실행합니다. GOGC=50이면 더 자주 GC가 실행되고(메모리 사용 줄임, CPU 사용 늘림), GOGC=200이면 덜 자주 실행됩니다.

GOMEMLIMIT은 메모리 상한을 설정합니다. 컨테이너 환경에서 특히 유용합니다.

### 값 전달 vs 포인터 전달 성능

| 데이터 크기 | 값 전달 시간 | 포인터 전달 시간 | 권장 방법 |
|------------|-------------|-----------------|----------|
| 100 Byte | ~10ns | ~30ns | 값 |
| 10 KB | ~100ns | ~30ns | 값 |
| 100 KB | ~50µs | ~5µs | 상황에 따라 |
| 10 MB | ~1.5ms | ~0.5ms | 포인터 |

### Map vs Slice 함수 전달 동작

| 연산 | Map | Slice |
|------|-----|-------|
| 요소 수정 | 원본 반영 ✅ | 원본 반영 ✅ |
| 요소 추가 | 원본 반영 ✅ | 원본 반영 ❌ |
| 요소 삭제 | 원본 반영 ✅ | 해당 없음 |
| 길이 변경 | 원본 반영 ✅ | 원본 반영 ❌ |

### Escape Analysis 규칙

| 상황 | 할당 위치 |
|------|----------|
| 지역 변수, 크기 고정 | 스택 |
| 포인터 반환 | 힙 |
| 인터페이스에 할당 | 힙 (대부분) |
| 클로저에서 캡처 | 힙 (대부분) |
| slice append로 grow | 힙 |

## 면접 예상 질문 및 모범 답안

### Q1. Go에서 포인터를 사용해야 하는 경우와 피해야 하는 경우를 설명하세요.

**모범 답안:**

Go에서는 기본적으로 값을 사용하고, 포인터는 필요한 경우에만 사용해야 합니다.

포인터를 사용해야 하는 경우는 네 가지입니다. 첫째, 함수 내에서 원본 데이터를 수정해야 할 때입니다. 둘째, nil이 의미 있는 값일 때, 예를 들어 JSON에서 null과 zero value를 구분해야 할 때 포인터 필드를 사용합니다. 셋째, json.Unmarshal처럼 인터페이스를 받는 함수에 전달할 때입니다. 넷째, 매우 큰 데이터(메가바이트 단위)를 전달할 때 복사 비용을 줄이기 위해 사용합니다.

포인터를 피해야 하는 경우도 있습니다. 10KB 미만의 작은 데이터는 값 전달이 오히려 빠릅니다. 데이터 불변성이 중요한 경우 값을 사용하면 함수가 원본을 수정할 수 없습니다. 또한 데이터 흐름을 명확하게 하고 싶을 때 값을 사용하면 입력과 출력이 명확해집니다.

값을 사용하면 스택 할당 가능성이 높아져 가비지 컬렉터 부담이 줄어들고, 순차적 메모리 접근으로 CPU 캐시 효율이 높아집니다. 이를 "Mechanical Sympathy"라고 합니다.

### Q2. Go에서 nil 포인터를 함수 내에서 non-nil로 만들 수 없는 이유를 설명하세요.

**모범 답안:**

Go는 순수한 Call by Value 언어이기 때문입니다. 함수에 포인터를 전달하면 포인터 값(메모리 주소) 자체가 복사됩니다.

```go
func failedUpdate(g *int) {
    x := 10
    g = &x    // g는 복사본이므로 원본 f에 영향 없음
}

func main() {
    var f *int  // nil
    failedUpdate(f)
    fmt.Println(f)  // 여전히 nil!
}
```

failedUpdate 함수 내의 g는 f의 복사본입니다. g = &x는 복사본 g를 새 주소로 재할당할 뿐, 원본 f에는 영향을 주지 않습니다.

이를 해결하려면 두 가지 방법이 있습니다. 첫째, 포인터를 반환하는 방법입니다: `func update() *int { x := 10; return &x }`. 둘째, 이중 포인터를 사용하는 방법입니다: `func update(g **int) { x := 10; *g = &x }`. 하지만 이중 포인터는 코드가 복잡해지므로, 값을 반환하는 첫 번째 방법이 Go에서 권장됩니다.

### Q3. Go에서 Map과 Slice가 함수에 전달될 때 다르게 동작하는 이유를 설명하세요.

**모범 답안:**

Map과 Slice는 내부 구현이 다르기 때문입니다.

Map은 내부적으로 해시 테이블에 대한 포인터로 구현되어 있습니다. map 변수는 실제로 구조체를 가리키는 포인터입니다. 따라서 함수에 map을 전달하면 포인터가 복사되고, 이 복사된 포인터도 같은 해시 테이블을 가리킵니다. 그래서 함수 내에서 map에 키-값을 추가, 수정, 삭제하면 모두 원본에 반영됩니다.

Slice는 세 개의 필드를 가진 구조체로 구현됩니다: length, capacity, 그리고 실제 데이터를 가리키는 포인터. 함수에 slice를 전달하면 이 구조체가 복사됩니다. 복사된 구조체의 포인터는 같은 메모리 블록을 가리키므로 요소 수정은 원본에 반영됩니다. 하지만 append로 요소를 추가하면 복사본의 length만 증가하고, 원본의 length는 변경되지 않습니다. capacity를 초과하면 새 메모리가 할당되어 원본과 완전히 분리됩니다.

이런 이유로 함수가 slice를 수정해야 한다면 수정된 slice를 반환하거나, slice 포인터를 전달해야 합니다.

### Q4. Escape Analysis가 무엇이고, Go에서 왜 중요한지 설명하세요.

**모범 답안:**

Escape Analysis는 Go 컴파일러가 변수를 스택에 할당할지 힙에 할당할지 결정하는 분석입니다.

스택 할당은 힙 할당보다 훨씬 빠릅니다. 스택은 함수 호출 시 자동으로 할당되고 함수 종료 시 자동으로 해제되므로 가비지 컬렉터가 관여하지 않습니다. 또한 스택의 데이터는 메모리에서 연속적이므로 CPU 캐시 효율이 높습니다.

컴파일러는 다음 조건에서 변수를 힙으로 escape시킵니다: (1) 포인터가 함수 외부로 반환될 때, (2) 변수가 인터페이스에 할당될 때(런타임에 크기를 알 수 없으므로), (3) 클로저에서 변수를 캡처할 때, (4) slice가 append로 grow할 때.

`go build -gcflags="-m"` 명령으로 어떤 변수가 escape하는지 확인할 수 있습니다. 성능이 중요한 코드에서는 불필요한 escape를 줄여 GC 부담을 낮추고 성능을 개선할 수 있습니다.

실용적인 팁으로, 값을 반환하면 스택에 남고, 포인터를 반환하면 힙으로 escape합니다. 작은 데이터는 값으로 다루는 것이 좋습니다.

### Q5. GOGC와 GOMEMLIMIT 환경변수의 역할과 튜닝 방법을 설명하세요.

**모범 답안:**

GOGC와 GOMEMLIMIT은 Go 가비지 컬렉터의 동작을 제어하는 환경변수입니다.

GOGC는 GC 목표 비율을 설정합니다. 기본값은 100이며, 이는 현재 힙 크기의 100%가 증가하면 GC를 실행한다는 의미입니다. 공식은 `다음 GC 트리거 = 현재 힙 + (현재 힙 × GOGC/100)`입니다. GOGC=50이면 GC가 더 자주 실행되어 메모리 사용량은 줄지만 CPU 사용량이 늘어납니다. GOGC=200이면 GC가 덜 자주 실행되어 메모리를 더 사용하지만 CPU 오버헤드가 줄어듭니다. GOGC=off는 GC를 비활성화하지만, 장기 실행 프로그램에서는 메모리 누수 위험이 있어 권장되지 않습니다.

GOMEMLIMIT은 Go 런타임이 사용할 수 있는 메모리 상한을 설정합니다. 예: GOMEMLIMIT=4GiB. 컨테이너 환경에서 메모리 제한이 있을 때 특히 유용합니다. 이 한도에 가까워지면 GC가 더 공격적으로 실행됩니다.

권장 사용법은 둘 다 설정하는 것입니다: `GOGC=100 GOMEMLIMIT=4GiB ./myprogram`. GC 동작을 디버깅하려면 `GODEBUG=gctrace=1`을 사용합니다.

### Q6. Go에서 JSON null과 zero value를 구분해야 할 때 어떻게 처리하나요?

**모범 답안:**

Go에서 기본 타입의 zero value(int의 0, string의 "")와 JSON의 null을 구분하려면 포인터 필드를 사용합니다.

```go
type Person struct {
    Name string  `json:"name"`
    Age  *int    `json:"age,omitempty"`  // 포인터로 선언
}
```

Age가 `*int`이면 세 가지 상태를 표현할 수 있습니다: (1) nil이면 JSON에서 해당 필드가 없거나 null, (2) 0을 가리키면 명시적으로 0이 설정됨, (3) 다른 값을 가리키면 해당 값이 설정됨.

사용할 때는 nil 체크가 필요합니다:
```go
if person.Age != nil {
    fmt.Println("Age:", *person.Age)
} else {
    fmt.Println("Age not specified")
}
```

JSON 처리가 아닌 일반 코드에서는 comma ok idiom을 사용하는 것이 더 명확합니다:
```go
func GetValue() (int, bool) {
    return value, hasValue
}

if val, ok := GetValue(); ok {
    // val 사용
}
```

포인터 필드는 코드를 복잡하게 만들고 nil 역참조 위험이 있으므로, JSON 직렬화/역직렬화가 필요한 API 경계에서만 사용하는 것이 좋습니다.

### 포인터 사용 판단
- [ ] 함수가 원본을 수정해야 하는가? → 포인터 사용
- [ ] nil이 의미 있는 값인가? → 포인터 사용
- [ ] 데이터 크기가 10MB 이상인가? → 포인터 고려
- [ ] 위 조건에 해당하지 않으면 → 값 사용

### nil 포인터 안전성
- [ ] 포인터 역참조 전 nil 체크 수행
- [ ] nil 포인터를 함수 내에서 재할당해도 원본은 변하지 않음 인지
- [ ] 포인터 반환 함수에서 error와 함께 nil 반환 패턴 사용

### Map/Slice 함수 전달
- [ ] Map 전달 시 원본이 수정됨을 인지
- [ ] Slice 요소 수정은 원본 반영, append는 미반영 인지
- [ ] Slice 수정이 필요하면 반환 또는 포인터 전달

### 메모리 최적화
- [ ] 값 타입 사용으로 스택 할당 유도
- [ ] `go build -gcflags="-m"`으로 Escape Analysis 확인
- [ ] 버퍼 재사용 패턴으로 GC 부담 감소
- [ ] 연속된 메모리 배치로 캐시 효율 향상 ([]Data vs []*Data)

### GC 튜닝 (프로덕션)
- [ ] GOMEMLIMIT으로 메모리 상한 설정 (컨테이너 환경)
