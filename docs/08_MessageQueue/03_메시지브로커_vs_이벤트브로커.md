# 메시지 브로커 vs 이벤트 브로커

## 1. 개요

메시징 플랫폼은 크게 **두 가지 종류**로 나뉜다:

| 유형 | 설명 | 대표 제품 |
|------|------|----------|
| **메시지 브로커** | 메시지를 받아 처리 후 삭제 | Redis Queue, RabbitMQ |
| **이벤트 브로커** | 메시지를 보존하며 재처리 가능 | Kafka, AWS Kinesis |

**핵심 관계**:
- 메시지 브로커는 이벤트 브로커 역할을 할 수 **없음**
- 이벤트 브로커는 메시지 브로커 역할을 할 수 **있음**

---

## 2. 메시지 브로커 (Message Broker)

### 2.1 특징

```
Producer → [Queue] → Consumer
                 ↓
            메시지 삭제
```

- 메시지를 받아서 **적절히 처리하면 즉시 삭제**
- 대규모 메시지 기반 **미들웨어 아키텍처**에서 사용
- 서비스/애플리케이션 간 **효율적인 데이터 연결**

### 2.2 동작 방식 (RabbitMQ 예시)

```
1. Producer가 메시지를 Queue에 전송
2. Broker가 메시지를 Queue 형태로 저장 (FIFO)
3. Consumer가 메시지 요청
4. Broker가 메시지 전달 + Queue에서 삭제
```

### 2.3 Smart Broker / Dumb Consumer

- **Broker가 메시지 관리**를 담당
- Consumer는 단순히 요청만 하면 됨
- 메시지 삭제 책임은 Broker에 있음

### 2.4 대표 제품

| 제품 | 특징 |
|------|------|
| **RabbitMQ** | AMQP 프로토콜, 복잡한 라우팅 |
| **Redis Queue** | 인메모리, 빠른 처리 |
| **ActiveMQ** | JMS 표준, 엔터프라이즈 |

---

## 3. 이벤트 브로커 (Event Broker)

### 3.1 특징

```
Producer → [Log/Topic] → Consumer
                ↓
          메시지 보존 (인덱스 관리)
```

- **이벤트를 삭제하지 않고 보존**
- 인덱스(오프셋)를 통해 개별 액세스 관리
- 업무상 필요한 시간 동안 이벤트 보존

### 3.2 동작 방식 (Kafka 예시)

```
1. Producer가 메시지를 Topic에 전송
2. Broker가 메시지를 Log 형태로 디스크에 저장
3. 각 메시지는 Offset(인덱스)을 가짐
4. Consumer가 자신의 Offset을 기억하며 메시지 소비
5. 메시지는 삭제되지 않음 (보존 기간까지)
```

### 3.3 Dumb Broker / Smart Consumer

- **Consumer가 Offset을 관리**
- Broker는 단순히 로그 저장만 담당
- Consumer가 어디까지 읽었는지 직접 추적

### 3.4 이벤트 보존의 장점

| 장점 | 설명 |
|------|------|
| **단일 진실 공급원** | 한 번 발생한 이벤트를 데이터 소스로 활용 |
| **장애 복구** | 장애 발생 지점부터 재처리 가능 |
| **실시간 스트림** | 대용량 스트림 데이터 효과적 처리 |
| **이벤트 리플레이** | 과거 이벤트 재생 가능 |

### 3.5 대표 제품

| 제품 | 특징 |
|------|------|
| **Apache Kafka** | 업계 표준, 높은 처리량 |
| **AWS Kinesis** | AWS 관리형 서비스 |
| **RedPanda** | Kafka 호환, C++ 구현 |

## 4. 핵심 비교

### 4.1 데이터 처리 방식

| 항목 | 메시지 브로커 | 이벤트 브로커 |
|------|--------------|--------------|
| **처리 후** | 즉시 삭제 | 보존 (TTL까지) |
| **저장 방식** | Queue (메모리) | Log (디스크) |
| **재처리** | 불가능 | 가능 |
| **Offset 관리** | Broker | Consumer |

### 4.2 아키텍처 역할

```
┌─────────────────────────────────────────────────┐
│              메시지 브로커                        │
│    - 작업 큐 (Task Queue)                        │
│    - Point-to-Point 통신                         │
│    - 요청/응답 패턴                              │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│              이벤트 브로커                        │
│    - 이벤트 소싱 (Event Sourcing)                │
│    - 이벤트 스트리밍                             │
│    - 마이크로서비스 이벤트 버스                   │
│    - CDC (Change Data Capture)                   │
│    + 메시지 브로커 역할도 가능                    │
└─────────────────────────────────────────────────┘
```

### 4.3 적합한 사용 사례

| 사용 사례 | 추천 |
|----------|------|
| 작업 분산 처리 | 메시지 브로커 |
| 복잡한 라우팅 | 메시지 브로커 (RabbitMQ) |
| 이벤트 리플레이 필요 | 이벤트 브로커 |
| 대용량 스트리밍 | 이벤트 브로커 |
| 장애 복구 중요 | 이벤트 브로커 |
| 마이크로서비스 이벤트 버스 | 이벤트 브로커 |

---

## 5. 선택 가이드

```
Q: 메시지를 재처리해야 하는가?
├─ YES → 이벤트 브로커 (Kafka, Kinesis)
└─ NO ─→ Q: 복잡한 라우팅이 필요한가?
         ├─ YES → 메시지 브로커 (RabbitMQ)
         └─ NO ─→ Q: 처리량이 매우 높은가?
                  ├─ YES → 이벤트 브로커 (Kafka)
                  └─ NO ─→ 메시지 브로커 (Redis Queue)
```

## 6. 면접 포인트

### Q: 메시지 브로커와 이벤트 브로커의 차이점은?

**핵심 답변**:
> "메시지 브로커는 메시지를 처리 후 즉시 삭제하는 반면, 이벤트 브로커는 메시지를 보존하여 재처리가 가능합니다. 이벤트 브로커는 메시지 브로커 역할도 수행할 수 있지만, 그 반대는 불가능합니다."

**추가 설명**:
- RabbitMQ는 Smart Broker/Dumb Consumer 패턴
- Kafka는 Dumb Broker/Smart Consumer 패턴
- 이벤트 브로커는 장애 복구, 이벤트 소싱에 적합

### Q: Kafka가 RabbitMQ보다 유리한 상황은?

**답변**:
> "이벤트 리플레이가 필요하거나, 대용량 스트리밍 처리, 마이크로서비스 이벤트 버스, 장애 시 재처리가 중요한 경우 Kafka가 유리합니다."

## 7. 참고

- 메시지 브로커 대표: RabbitMQ, Redis Queue, ActiveMQ
- 이벤트 브로커 대표: Apache Kafka, AWS Kinesis, RedPanda
- Kafka 클러스터 구축 시 이벤트 기반 마이크로서비스 아키텍처 발전에 유리
