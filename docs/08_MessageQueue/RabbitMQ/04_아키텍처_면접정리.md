# RabbitMQ 아키텍처 - 면접 정리

## 핵심 한 줄 정리

> **RabbitMQ는 Producer → Exchange → Binding → Queue → Consumer 구조로, Exchange의 라우팅 규칙에 따라 메시지를 유연하게 분배하는 Push 기반 메시지 브로커입니다.**

---

## 면접 필수 암기

### Q1. RabbitMQ 메시지 흐름을 설명해주세요.

**답변 (30초):**
> Producer가 메시지를 **Exchange**로 발행합니다. Exchange는 **Binding 규칙**과 **Routing Key**를 기반으로 적절한 Queue로 메시지를 라우팅합니다. Consumer는 Queue를 구독하여 메시지를 받습니다.
>
> ```
> Producer → Exchange → (Binding) → Queue → Consumer
> ```
>
> Kafka와 달리 **Exchange라는 라우팅 계층**이 있어서, Producer가 Queue를 직접 알 필요 없이 복잡한 라우팅 로직을 구현할 수 있습니다.

---

### Q2. Connection과 Channel의 차이는?

**답변:**

| 항목 | Connection | Channel |
|------|------------|---------|
| **물리적/논리적** | TCP 연결 (물리) | 가상 연결 (논리) |
| **생성 비용** | 높음 | 낮음 |
| **권장 수** | 애플리케이션당 1개 | **스레드당 1개** |
| **스레드 안전** | Yes | **No** |

> **왜 Channel을 쓰나요?**
> TCP 연결은 비용이 높습니다. Channel은 하나의 TCP 위에 여러 가상 연결을 만들어, **적은 리소스로 높은 동시성**을 달성합니다. 단, Channel은 스레드 안전하지 않으므로 **스레드당 하나씩** 사용해야 합니다.

---

### Q3. RabbitMQ 클러스터에서 데이터는 어떻게 분산되나요?

**답변:**
> ```
> 복제되는 것:
> - 메타데이터: Exchange, Binding, Virtual Host, 사용자 (모든 노드)
>
> 복제 안 되는 것 (기본):
> - Queue 데이터: 단일 노드에만 존재 (Classic Queue)
> ```
>
> **고가용성이 필요하면:**
> - **Quorum Queue**: Raft 복제로 여러 노드에 데이터 복제
> - **Stream**: 마찬가지로 Raft 복제

---

### Q4. Prefetch(QoS)가 무엇이고 왜 필요한가요?

**답변:**
> Prefetch는 Consumer가 **한 번에 받을 수 있는 미확인 메시지 수**를 제한합니다.
>
> ```java
> channel.basicQos(10);  // 최대 10개까지만 미리 받음
> ```
>
> **왜 필요한가?**
> RabbitMQ는 Push 모델입니다. 제한이 없으면 Consumer가 처리 못하는 속도로 메시지가 밀려옵니다. Prefetch로 **Consumer의 처리 능력에 맞게 흐름을 제어**합니다.
>
> - 값이 너무 작으면: 처리량 저하 (자주 요청해야 함)
> - 값이 너무 크면: 메모리 부담, 장애 시 재처리 메시지 증가

---

### Q5. Virtual Host(vhost)는 무엇인가요?

**답변:**
> Virtual Host는 **논리적 격리 단위**입니다. 하나의 RabbitMQ 서버에서 여러 독립적인 환경을 운영할 수 있습니다.
>
> ```
> /production  - 프로덕션 Exchange, Queue, 사용자
> /staging     - 스테이징 Exchange, Queue, 사용자
> /development - 개발 Exchange, Queue, 사용자
> ```
>
> 각 vhost는 **완전히 격리**됩니다. 다른 vhost의 Exchange나 Queue에 접근할 수 없습니다. 멀티테넌트 환경이나 환경 분리에 유용합니다.

## 깊은 이해를 위한 설명

### Exchange 타입별 라우팅

**Direct (정확 매칭):**
```
Routing Key: "error"
→ Binding이 "error"인 Queue만 받음
```

**Fanout (브로드캐스트):**
```
Routing Key: 무시됨
→ 바인딩된 모든 Queue가 받음
```

**Topic (패턴 매칭):**
```
Routing Key: "order.created.kr"

Binding "order.#"      → 매칭 (# = 0개 이상 단어)
Binding "order.*"      → 미매칭 (* = 정확히 1단어)
Binding "order.created.*" → 매칭
```

**Headers (속성 매칭):**
```
Message Headers: {type: "report", format: "pdf"}

x-match: all, {type: "report", format: "pdf"} → 매칭
x-match: any, {type: "report", format: "xlsx"} → 매칭 (하나만 일치해도)
```

### Acknowledgement의 중요성

```
메시지 유실 방지 체인:

1. Publisher Confirms
   Producer → Broker: "받았어?" → "응, 받았어(confirm)"

2. Persistent Message
   Broker → Disk: 메시지를 디스크에 저장

3. Consumer Ack
   Consumer → Broker: "처리 완료(ack)" → 메시지 삭제
```

**Ack 모드:**
| 모드 | 동작 | 위험도 |
|------|------|--------|
| auto-ack | 전송 즉시 삭제 | 높음 (유실 가능) |
| **manual-ack** | 명시적 Ack 후 삭제 | 낮음 (권장) |

### Memory/Disk Alarm

```
메모리 사용률:
40% (정상) → 75% (경고) → 95% (차단)
                           ↓
                    Publisher 차단됨
                    (백프레셔)
```

> `vm_memory_high_watermark = 0.4` (기본값)
> 메모리의 40% 이상 사용 시 Publisher를 차단하여 시스템 보호

## 코드 레벨 이해

### 기본 패턴

```java
// 1. Connection (애플리케이션 시작 시 1회)
ConnectionFactory factory = new ConnectionFactory();
factory.setHost("localhost");
Connection connection = factory.newConnection();

// 2. Channel (작업마다)
Channel channel = connection.createChannel();

// 3. Exchange 선언
channel.exchangeDeclare("orders", "topic", true);

// 4. Queue 선언
channel.queueDeclare("order-processor", true, false, false, null);

// 5. Binding
channel.queueBind("order-processor", "orders", "order.created.*");

// 6. 발행
channel.basicPublish("orders", "order.created.kr", null, message.getBytes());

// 7. 소비
channel.basicConsume("order-processor", false, (tag, delivery) -> {
    // 처리
    channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
}, tag -> {});
```

### 포트 정리

| 포트 | 용도 |
|------|------|
| **5672** | AMQP (클라이언트) |
| **15672** | Management UI |
| **25672** | 클러스터 내부 통신 |
| **5552** | Streams 전용 프로토콜 |
| **1883** | MQTT |

## 면접 예상 꼬리 질문

### Q. Kafka 토픽과 RabbitMQ Exchange의 차이는?

**답변:**
> | | Kafka Topic | RabbitMQ Exchange |
> |--|------------|-------------------|
> | **역할** | 저장 + 라우팅 | **라우팅만** |
> | **데이터 저장** | 토픽에 저장 | Queue에 저장 |
> | **라우팅 유연성** | 파티션 키만 | **4가지 타입** |
> | **패턴 매칭** | 없음 | Topic/Headers |
>
> RabbitMQ는 Exchange(라우팅)와 Queue(저장)가 **분리**되어 더 유연한 라우팅이 가능합니다.

### Q. RabbitMQ에서 순서 보장은 어떻게 되나요?

**답변:**
> **단일 Queue 내에서는 FIFO 순서 보장**됩니다.
>
> 그러나:
> - 여러 Consumer가 경쟁 소비하면 **처리 완료 순서는 보장 안 됨**
> - Exchange가 여러 Queue로 분배하면 **Queue 간 순서 보장 안 됨**
>
> 엄격한 순서가 필요하면 **하나의 Queue + 하나의 Consumer** 또는 **Streams의 오프셋 기반 소비**를 사용합니다.

### Q. Dead Letter Queue(DLQ)는 어떻게 동작하나요?

**답변:**
> 메시지가 **처리 실패**했을 때 별도의 Queue로 이동시키는 기능입니다.
>
> **DLQ로 이동하는 경우:**
> 1. Consumer가 `nack` (requeue=false)
> 2. 메시지 TTL 만료
> 3. Queue 길이 초과
>
> ```java
> args.put("x-dead-letter-exchange", "dlx");
> args.put("x-dead-letter-routing-key", "failed");
> ```
>
> Kafka에는 이런 기본 기능이 없어서 **직접 구현**해야 합니다.

### Q. Kafka와 RabbitMQ의 확장성 차이는?

**답변:**
> **Kafka:**
> - 파티션 추가로 **자연스럽게 수평 확장**
> - 각 파티션이 독립적으로 노드에 분산
>
> **RabbitMQ:**
> - Queue는 기본적으로 **단일 노드**
> - Quorum Queue로 복제는 가능하지만, **리더가 병목**
> - 확장을 위해 **여러 Queue로 샤딩** 필요 (수동)
>
> 대용량 처리에는 Kafka가 더 자연스럽게 확장됩니다.
