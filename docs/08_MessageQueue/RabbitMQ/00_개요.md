# RabbitMQ 개요

## 1. RabbitMQ란?

RabbitMQ는 2007년 Pivotal Software에서 출시한 오픈소스 메시지 브로커입니다. Erlang으로 작성되었으며, AMQP(Advanced Message Queuing Protocol)를 구현한 대표적인 메시지 큐 시스템입니다.

### 핵심 특징

| 항목 | 설명 |
|------|------|
| **출시** | 2007년 |
| **언어** | Erlang |
| **프로토콜** | AMQP 0-9-1, AMQP 1.0, MQTT, STOMP, Streams |
| **라이선스** | MPL 2.0 (Mozilla Public License) |
| **최신 버전** | 4.x (2024년~) |

---

## 2. 아키텍처

### 기본 구성요소

```
Producer → Exchange → Binding → Queue → Consumer
```

1. **Producer**: 메시지를 생성하여 Exchange로 전송
2. **Exchange**: 라우팅 규칙에 따라 메시지를 Queue로 분배
3. **Binding**: Exchange와 Queue를 연결하는 규칙
4. **Queue**: 메시지를 저장하는 버퍼
5. **Consumer**: Queue에서 메시지를 소비

### Exchange 타입

| 타입 | 설명 | 사용 사례 |
|------|------|----------|
| **Direct** | Routing Key가 정확히 일치하는 Queue로 전달 | 특정 작업 라우팅 |
| **Fanout** | 연결된 모든 Queue로 브로드캐스트 | 로그 브로드캐스트 |
| **Topic** | 패턴 매칭 (*, #) 기반 라우팅 | 유연한 다중 구독 |
| **Headers** | 헤더 속성 기반 라우팅 | 복잡한 라우팅 조건 |

## 3. 메시지 전달 방식: Push vs Pull

RabbitMQ는 기본적으로 **Push 기반** 메시지 전달을 사용합니다.

```
Kafka/RedPanda (Pull):
  Consumer가 Broker에서 데이터를 가져감
  → Consumer가 처리 속도 제어

RabbitMQ (Push):
  Broker가 Consumer에게 메시지를 밀어줌
  → Prefetch limit으로 과부하 방지
  → 더 낮은 지연시간 달성 가능
```

### Prefetch Count

Consumer가 한 번에 받을 수 있는 미확인 메시지 수를 제한:

```java
channel.basicQos(10); // 최대 10개의 미확인 메시지
```

## 4. 큐 타입 (2025년 기준)

RabbitMQ는 여러 종류의 큐 타입을 제공합니다:

| 큐 타입 | 도입 버전 | 특징 | 상태 |
|---------|----------|------|------|
| **Classic Queue** | 초기 | 단일 노드, 빠른 처리 | 지원 (HA 제거됨) |
| **Classic Mirrored Queue** | 과거 | 복제 기반 HA | **4.0에서 제거** |
| **Quorum Queue** | 3.8 (2019) | Raft 기반 복제, 고가용성 | **권장** |
| **Stream** | 3.9 (2021) | Kafka 스타일 로그 기반 | **권장** |

---

## 5. RabbitMQ 4.0 주요 변경사항 (2024년)

### 제거된 기능
- Classic Mirrored Queues 완전 제거
- 기존 HA 정책 미지원

### 추가/강화된 기능
- **Khepri**: Raft 기반 메타데이터 저장소 정식 지원
- **AMQP 1.0**: 코어 프로토콜로 승격
- **Quorum Queues**: 우선순위 지원 추가

## 6. 사용 사례

### 적합한 경우
- 마이크로서비스 간 비동기 통신
- 작업 큐 (Task Queue) / 백그라운드 작업
- IoT 메시징 (MQTT 네이티브 지원)
- 복잡한 라우팅이 필요한 경우
- 다중 프로토콜 지원이 필요한 경우

### 부적합한 경우
- 초대용량 스트리밍 (→ Kafka/RedPanda)
- 장기 메시지 보존 + 재처리가 핵심인 경우 (→ Kafka)
- 극한의 처리량이 필요한 경우 (→ Kafka/RedPanda)
