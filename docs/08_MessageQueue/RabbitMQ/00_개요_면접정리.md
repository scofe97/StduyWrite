# RabbitMQ 개요 - 면접 정리

## 핵심 한 줄 정리

> **RabbitMQ는 AMQP 프로토콜을 구현한 메시지 브로커로, Exchange를 통한 유연한 라우팅과 다중 프로토콜 지원이 특징입니다.**

---

## 면접 필수 암기

### Q1. RabbitMQ란 무엇인가요?

**답변 (30초):**
> RabbitMQ는 2007년에 출시된 오픈소스 메시지 브로커입니다. Erlang으로 작성되었으며, AMQP 프로토콜을 구현합니다. 가장 큰 특징은 Exchange를 통한 유연한 메시지 라우팅과 AMQP, MQTT, STOMP 등 다중 프로토콜 지원입니다. Kafka와 달리 Push 기반으로 동작하여 저지연 메시징에 유리합니다.

---

### Q2. RabbitMQ의 메시지 흐름을 설명해주세요.

**답변:**
> Producer가 메시지를 Exchange로 발행하면, Exchange는 Binding 규칙에 따라 적절한 Queue로 메시지를 라우팅합니다. Consumer는 Queue를 구독하여 메시지를 받습니다.
>
> ```
> Producer → Exchange → (Binding) → Queue → Consumer
> ```
>
> 이 구조의 장점은 Producer가 Queue를 직접 알 필요 없이, Exchange의 라우팅 규칙만으로 유연하게 메시지를 분배할 수 있다는 것입니다.

---

### Q3. Exchange 타입 4가지를 설명해주세요.

**답변:**

| 타입 | 동작 | 사용 사례 |
|------|------|----------|
| **Direct** | Routing Key 정확히 일치 | 특정 서비스로 요청 전달 |
| **Fanout** | 모든 바인딩된 Queue로 브로드캐스트 | 로그 수집, 알림 발송 |
| **Topic** | 패턴 매칭 (`*`, `#`) | 이벤트 필터링 구독 |
| **Headers** | 헤더 속성 기반 라우팅 | 복잡한 조건부 라우팅 |

**Topic 패턴 예시:**
- `order.*` → `order.created` 매칭, `order.created.kr` 미매칭
- `order.#` → `order.created`, `order.created.kr` 모두 매칭

---

### Q4. RabbitMQ는 Push 기반인데, Kafka의 Pull 기반과 차이점은?

**답변:**
> **Push (RabbitMQ):** Broker가 Consumer에게 메시지를 밀어줍니다. 즉시 전달되므로 저지연에 유리하지만, Consumer가 처리 못하면 과부하 발생 가능합니다. 이를 방지하기 위해 `prefetch_count`로 한 번에 받을 미확인 메시지 수를 제한합니다.
>
> **Pull (Kafka):** Consumer가 원하는 속도로 메시지를 가져갑니다. 백프레셔가 자연스럽게 처리되고, 배치 처리에 유리합니다. 하지만 폴링 간격만큼 지연이 발생합니다.

---

### Q5. RabbitMQ의 큐 타입을 설명해주세요.

**답변:**
> RabbitMQ 4.0 기준으로 주요 큐 타입은 세 가지입니다:
>
> 1. **Classic Queue**: 단일 노드, 고성능, 데이터 안전성 낮음
> 2. **Quorum Queue**: Raft 기반 복제, 고가용성, 데이터 안전성 높음 (권장)
> 3. **Stream**: Kafka 스타일 append-only 로그, 메시지 재생 가능
>
> Classic Mirrored Queue는 4.0에서 제거되었으므로, 고가용성이 필요하면 Quorum Queue를 사용해야 합니다.

## 깊은 이해를 위한 설명

### RabbitMQ를 선택해야 하는 상황

RabbitMQ는 **전통적인 메시지 큐 패턴**에 최적화되어 있습니다. 다음 상황에서 Kafka보다 적합합니다:

1. **복잡한 라우팅이 필요할 때**
   - Topic Exchange로 `order.*.kr` 같은 패턴 매칭
   - Headers Exchange로 메시지 속성 기반 라우팅
   - Kafka는 토픽/파티션 구조로 이런 유연성이 부족

2. **다양한 프로토콜을 지원해야 할 때**
   - IoT 디바이스 (MQTT)
   - 웹 클라이언트 (STOMP/WebSocket)
   - 엔터프라이즈 시스템 (AMQP 1.0)

3. **작업 큐(Task Queue) 패턴**
   - 여러 Worker가 하나의 Queue에서 경쟁적으로 작업 가져가기
   - 메시지 처리 후 삭제 (Kafka는 기본적으로 보존)

4. **우선순위 큐, Dead Letter Queue 필요**
   - RabbitMQ 네이티브 지원
   - Kafka는 직접 구현해야 함

### RabbitMQ의 한계

- **처리량**: Kafka 대비 1/10 수준 (5만-10만 msg/s vs 100만+)
- **메시지 보존**: 기본적으로 소비 후 삭제 (Streams 제외)
- **수평 확장**: Queue는 기본적으로 단일 노드에 존재 (Quorum Queue로 복제 가능하지만 Kafka만큼 자연스럽지 않음)

## 코드 레벨 이해

### Connection과 Channel의 차이

```java
// Connection: TCP 연결 (비용 높음, 애플리케이션당 1개)
Connection connection = factory.newConnection();

// Channel: 가상 연결 (비용 낮음, 스레드당 1개)
Channel channel = connection.createChannel();
```

**왜 Channel을 쓰나요?**
> TCP 연결은 생성 비용이 높습니다. 하지만 하나의 연결만으로는 동시성이 부족합니다. Channel은 하나의 TCP 연결 위에 여러 가상 연결을 만들어, 적은 리소스로 높은 동시성을 달성합니다.

### Prefetch 설정

```java
channel.basicQos(10); // 미확인 메시지 최대 10개
```

**왜 필요한가요?**
> Push 기반이므로 Consumer가 처리 못하면 메시지가 쌓입니다. `prefetch_count`를 설정하면 해당 수만큼만 미리 받고, Ack 후에 다음 메시지를 받습니다. 이것이 RabbitMQ의 흐름 제어 방식입니다.

## 면접 예상 꼬리 질문

### Q. RabbitMQ에서 메시지 유실을 방지하려면?

**답변:**
1. **Publisher Confirms**: 메시지가 Broker에 도착했는지 확인
2. **Durable Queue**: 브로커 재시작 후에도 큐 유지
3. **Persistent Message**: 메시지를 디스크에 저장
4. **Consumer Ack**: 처리 완료 후 명시적으로 Ack
5. **Quorum Queue**: Raft 복제로 노드 장애에도 데이터 보존

### Q. RabbitMQ 클러스터에서 Queue는 어떻게 동작하나요?

**답변:**
> 기본적으로 Queue 데이터는 **단일 노드에만 존재**합니다. 메타데이터(Exchange, Binding)만 모든 노드에 복제됩니다. 고가용성이 필요하면 **Quorum Queue**를 사용해야 하며, 이는 Raft 합의로 여러 노드에 데이터를 복제합니다.

### Q. RabbitMQ와 Kafka 중 어떤 것을 선택해야 하나요?

**답변:**
> - **RabbitMQ**: 복잡한 라우팅, 다중 프로토콜, 작업 큐, 저지연 필요 시
> - **Kafka**: 대용량 스트리밍, 메시지 재처리, 이벤트 소싱, 로그 수집 시
>
> 최근에는 RabbitMQ Streams가 Kafka 스타일 기능을 제공하므로, 두 패러다임이 필요하면 RabbitMQ 하나로 해결할 수도 있습니다.
