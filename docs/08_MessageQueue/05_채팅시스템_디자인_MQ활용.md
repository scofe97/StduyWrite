# 채팅 시스템 디자인과 메시지 큐 활용

## 1. 개요

채팅 애플리케이션(카카오톡, WhatsApp 등)의 시스템 디자인에서 메시지 큐가 어떻게 활용되는지 알아본다.

### 1.1 채팅 시스템 트래픽 특성

| 특성 | 설명 |
|------|------|
| **트래픽 양** | 매우 많음 (페이스북+왓츠앱: 하루 600억 개) |
| **데이터 접근** | 최근 메시지 위주 조회 (오래된 채팅 잘 안봄) |
| **Read/Write 비율** | 거의 1:1 (대화가 오가는 구조) |

---

## 2. 통신 방식 선택

### 2.1 HTTP의 한계

```
A (클라이언트) ─── HTTP Request ───→ 서버
                                    │
                                    ✗ 서버가 B에게 먼저 요청 불가
```

- HTTP는 **클라이언트가 시작**하는 커넥션
- 서버가 클라이언트에게 **먼저 Push 불가**
- A가 B에게 메시지 보내면, B는 어떻게 알 수 있을까?

### 2.2 해결 방법 비교

#### Polling (폴링)

```
B: 새 메시지 있나요? → 서버: 없어요
B: 새 메시지 있나요? → 서버: 없어요
B: 새 메시지 있나요? → 서버: 있어요! (A가 보낸 메시지)
```

**단점**:
- 대부분 "없어요" 응답 → 불필요한 요청
- Polling 간격만큼 **레이턴시 발생**
- 서버 부하 증가

#### Long Polling (롱 폴링)

- 요청 후 응답을 **지연**시키는 방식
- 새 메시지가 올 때까지 연결 유지
- Polling보다 효율적이나 여전히 한계 존재

#### WebSocket (정답)

```
┌─────────┐                    ┌─────────┐
│ Client  │ ←── Open Connection ──→ │ Server  │
└─────────┘     (양방향 통신)      └─────────┘
```

- **오픈 커넥션 유지**
- **양방향 소통 가능** (서버 → 클라이언트 Push 가능)
- 채팅 앱에 **최적의 선택**

## 3. 시스템 아키텍처

### 3.1 서버 구성

```
┌─────────────────────────────────────────────────────┐
│                    클라이언트                         │
└──────────────┬────────────────────┬─────────────────┘
               │                    │
          WebSocket              HTTP
               │                    │
               ▼                    ▼
┌──────────────────────┐  ┌──────────────────────┐
│     Chat Server      │  │     API Server       │
│  (WebSocket 유지)     │  │  (로그인, 프로필 등)   │
└──────────────────────┘  └──────────────────────┘
```

- **Chat Server**: WebSocket 오픈 커넥션 전담
- **API Server**: 일반 HTTP 요청 처리 (로그인, 프로필 변경 등)

### 3.2 전체 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  User A (Online) ←─WebSocket─→ Chat Server 1               │
│                                      │                      │
│                                      ▼                      │
│                              ┌─────────────┐                │
│                              │ Message Queue │               │
│                              │ (User별 Queue)│               │
│                              └──────┬──────┘                │
│                                     │                       │
│                    ┌────────────────┼────────────────┐      │
│                    ▼                ▼                ▼      │
│            ┌───────────┐    ┌───────────┐    ┌───────────┐  │
│            │    DB     │    │   Push    │    │Chat Server│  │
│            │  저장     │    │Notification│   │     3     │  │
│            └───────────┘    └───────────┘    └─────┬─────┘  │
│                                                    │        │
│                                              WebSocket      │
│                                                    ▼        │
│                                          User B (Online)    │
│                                                             │
│  User C (Offline) ←── Push Notification                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 4. 메시지 전송 플로우

### 4.1 1:1 채팅

```
1. User A가 메시지 작성, 보내기 클릭
2. WebSocket을 통해 Chat Server로 전송
3. Chat Server가 User B의 Queue에 메시지 삽입
4. Message Queue가 구독자들에게 알림:
   ├─ Chat Server 3: B에게 WebSocket으로 메시지 Push
   ├─ DB Service: 메시지 저장
   └─ Push Service: B가 오프라인이면 Push 알림
```

### 4.2 그룹 채팅

```
1. User A가 그룹 채팅에 메시지 전송
2. Chat Server가 그룹 DB 조회 → 멤버 목록 확인 (A, B, C)
3. 수신자들의 Queue에 각각 메시지 삽입 (Fan-out)
4. 나머지는 1:1 채팅과 동일
```

**그룹 채팅 제한**:
- 그룹 당 최대 200명 가정
- Fan-out 비용 고려 필요
- 대규모 그룹은 다른 설계 필요

## 5. 메시지 큐의 역할

### 5.1 디커플링 효과

**MQ 없이**:
```
Chat Server → DB 저장
           → 메시지 전달
           → Push 알림
           → 이메일 발송

(Chat Server에 모든 의존성 집중)
```

**MQ 도입 후**:
```
Chat Server → [Message Queue] → DB Service (Subscribe)
                             → Chat Server (Subscribe)
                             → Push Service (Subscribe)
                             → Email Service (Subscribe)

(Chat Server는 MQ만 알면 됨)
```

### 5.2 비동기 처리

- 메시지 전송 → Queue에 넣고 **즉시 완료**
- DB 저장, Push 등은 **비동기로 처리**
- 사용자 경험 향상 (빠른 응답)

### 5.3 확장성

- Chat Server 수평 확장 용이
- 각 서비스 **독립적으로 스케일**
- 장애 격리

## 6. 데이터베이스 선택

### 6.1 요구사항

- 트래픽 양이 매우 많음
- Read/Write 비율 1:1
- 다른 테이블과 Join 거의 없음
- 시간 순서대로 Range Scan 필요

### 6.2 선택: Key-Value Store

| 선택 | 이유 |
|------|------|
| **Relational DB 부적합** | 데이터 많아지면 느려짐, Join 불필요 |
| **Key-Value Store 적합** | 스케일 용이, Read Latency 낮음 |

**사례**:
- Facebook, Discord → **Cassandra** 사용

### 6.3 키 설계

```
메시지 ID를 시간 순서대로 증가하게 설계
→ Key Range Scan 시 채팅 순서대로 조회 가능
```

## 7. 면접 포인트

### Q: 채팅 시스템에서 HTTP 대신 WebSocket을 사용하는 이유는?

**핵심 답변**:
> "HTTP는 클라이언트가 시작하는 커넥션이라 서버가 먼저 클라이언트에게 메시지를 Push할 수 없습니다. WebSocket은 오픈 커넥션을 유지하여 양방향 소통이 가능하므로 채팅 앱에 적합합니다."

### Q: 채팅 시스템에서 메시지 큐를 사용하는 이유는?

**답변**:
> "메시지 큐를 통해 Chat Server와 다른 서비스들(DB, Push, Email 등)을 디커플링할 수 있습니다. 이를 통해 각 서비스를 독립적으로 확장하고, 비동기 처리로 사용자 응답 속도를 높일 수 있습니다."

### Q: 그룹 채팅의 Fan-out 문제는 어떻게 해결하나요?

**답변**:
> "그룹 크기를 제한(예: 200명)하여 Fan-out 비용을 관리합니다. 대규모 그룹이 필요한 경우 별도의 Broadcast 서버나 다른 아키텍처를 고려해야 합니다."

### Q: 채팅 데이터에 Cassandra를 사용하는 이유는?

**답변**:
> "채팅 데이터는 다른 테이블과 Join이 거의 없고, 시간 순서대로 Range Scan이 필요합니다. Cassandra는 스케일이 용이하고 Read Latency가 낮으며, Key 설계를 통해 효율적인 Range Scan이 가능합니다."
