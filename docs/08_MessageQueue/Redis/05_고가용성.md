# Redis 고가용성 (High Availability)

## 1. 개요

Redis는 **두 가지 고가용성 솔루션**을 제공한다:

| 솔루션 | 목적 | 특징 |
|--------|------|------|
| **Redis Sentinel** | 자동 장애 조치 | 단일 마스터, 모니터링 |
| **Redis Cluster** | 샤딩 + 고가용성 | 분산 저장, 자동 장애 조치 |

### 1.1 선택 가이드

```
┌─────────────────────────────────────────────────────────────┐
│                      선택 가이드                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Q: 데이터가 단일 노드에 들어가는가?                          │
│  ├─ YES → Sentinel 사용                                    │
│  └─ NO ─→ Cluster 사용 (샤딩 필요)                         │
│                                                             │
│  Q: 수평 확장이 필요한가?                                    │
│  ├─ YES → Cluster 사용                                     │
│  └─ NO ─→ Sentinel 사용                                    │
│                                                             │
│  Q: 운영 복잡도를 최소화하고 싶은가?                          │
│  ├─ YES → Sentinel 사용                                    │
│  └─ NO ─→ Cluster 사용                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. Redis Sentinel

### 2.1 개요

Redis Sentinel은 **Redis Cluster를 사용하지 않을 때** 고가용성을 제공하는 시스템이다.

### 2.2 핵심 기능

| 기능 | 설명 |
|------|------|
| **모니터링** | Master/Replica 상태 지속 감시 |
| **알림** | 장애 발생 시 시스템/API 알림 |
| **자동 장애 조치** | Master 장애 시 Replica 승격 |
| **설정 제공자** | 클라이언트에게 현재 Master 주소 제공 |

### 2.3 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                    Sentinel 아키텍처                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│     ┌──────────┐   ┌──────────┐   ┌──────────┐            │
│     │Sentinel 1│   │Sentinel 2│   │Sentinel 3│            │
│     └────┬─────┘   └────┬─────┘   └────┬─────┘            │
│          │              │              │                   │
│          └──────────────┼──────────────┘                   │
│                         │ 모니터링                          │
│                         ▼                                  │
│                  ┌─────────────┐                           │
│                  │   Master    │                           │
│                  └──────┬──────┘                           │
│                         │ 복제                              │
│            ┌────────────┴────────────┐                     │
│            ▼                         ▼                     │
│     ┌─────────────┐          ┌─────────────┐              │
│     │  Replica 1  │          │  Replica 2  │              │
│     └─────────────┘          └─────────────┘              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.4 Quorum (정족수)

```
Sentinel 수: 3
Quorum: 2 (과반수)

장애 조치 조건:
- 최소 2개 Sentinel이 Master 장애 동의
- 과반수 Sentinel이 온라인 상태

권장 Sentinel 수: 3 또는 5 (홀수)
```

### 2.5 장애 조치 과정

```
1. 주관적 다운 (SDOWN)
   - 개별 Sentinel이 Master 응답 없음 감지

2. 객관적 다운 (ODOWN)
   - Quorum 이상의 Sentinel이 SDOWN 동의

3. Leader 선출
   - Sentinel 중 Leader 선출 (Raft)

4. Replica 선택
   - 가장 최신 데이터를 가진 Replica 선택

5. 승격
   - 선택된 Replica를 Master로 승격

6. 재구성
   - 다른 Replica들이 새 Master를 바라보도록 설정
   - 클라이언트에게 새 Master 주소 알림
```

### 2.6 설정 예제

**sentinel.conf**:
```
# Sentinel 포트
port 26379

# Master 모니터링 (이름, 호스트, 포트, Quorum)
sentinel monitor mymaster 127.0.0.1 6379 2

# Master 다운 판단 시간 (밀리초)
sentinel down-after-milliseconds mymaster 30000

# 장애 조치 타임아웃
sentinel failover-timeout mymaster 180000

# 동시 동기화 Replica 수
sentinel parallel-syncs mymaster 1
```

### 2.7 클라이언트 연결

```python
import redis
from redis.sentinel import Sentinel

# Sentinel 연결
sentinel = Sentinel([
    ('sentinel1', 26379),
    ('sentinel2', 26379),
    ('sentinel3', 26379)
], socket_timeout=0.1)

# Master 연결 (자동 장애 조치)
master = sentinel.master_for('mymaster', socket_timeout=0.1)
master.set('key', 'value')

# Replica 연결 (읽기 전용)
slave = sentinel.slave_for('mymaster', socket_timeout=0.1)
value = slave.get('key')
```

## 3. Redis Cluster

### 3.1 개요

Redis Cluster는 **데이터 샤딩**과 **고가용성**을 동시에 제공하는 분산 시스템이다.

### 3.2 핵심 특징

| 특징 | 설명 |
|------|------|
| **자동 샤딩** | 16384개 해시 슬롯으로 데이터 분산 |
| **자동 장애 조치** | Sentinel 없이 자체 처리 |
| **수평 확장** | 노드 추가로 용량/처리량 확장 |
| **다중 키 제한** | 같은 슬롯의 키만 다중 키 연산 가능 |

### 3.3 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                    Cluster 아키텍처                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  슬롯 0-5460          슬롯 5461-10922      슬롯 10923-16383 │
│  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐ │
│  │  Master A   │      │  Master B   │      │  Master C   │ │
│  │   (6379)    │      │   (6380)    │      │   (6381)    │ │
│  └──────┬──────┘      └──────┬──────┘      └──────┬──────┘ │
│         │                    │                    │        │
│         ▼                    ▼                    ▼        │
│  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐ │
│  │  Replica A  │      │  Replica B  │      │  Replica C  │ │
│  │   (6382)    │      │   (6383)    │      │   (6384)    │ │
│  └─────────────┘      └─────────────┘      └─────────────┘ │
│                                                             │
│                    Gossip Protocol                          │
│            (노드 간 상태 정보 교환)                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 해시 슬롯

```
슬롯 계산: CRC16(key) mod 16384

예:
  "user:1000" → CRC16 → 슬롯 12345 → Master C
  "user:2000" → CRC16 → 슬롯 3456 → Master A

해시 태그 (같은 슬롯 강제):
  "user:{1000}:profile"  → 슬롯 X
  "user:{1000}:settings" → 슬롯 X (동일)
  → {1000} 부분만 해시 계산
```

### 3.5 클러스터 생성

```bash
# 노드 시작 (cluster-enabled yes)
redis-server --port 6379 --cluster-enabled yes --cluster-config-file nodes-6379.conf

# 클러스터 생성
redis-cli --cluster create \
  127.0.0.1:6379 127.0.0.1:6380 127.0.0.1:6381 \
  127.0.0.1:6382 127.0.0.1:6383 127.0.0.1:6384 \
  --cluster-replicas 1

# 클러스터 상태 확인
redis-cli -c -p 6379 cluster info
redis-cli -c -p 6379 cluster nodes
```

### 3.6 클라이언트 연결

```python
from redis.cluster import RedisCluster

# 클러스터 연결
rc = RedisCluster(
    host='127.0.0.1',
    port=6379,
    decode_responses=True
)

# 자동으로 올바른 노드로 라우팅
rc.set('key1', 'value1')
rc.set('key2', 'value2')

# 해시 태그로 같은 슬롯 보장
rc.set('user:{100}:name', 'Alice')
rc.set('user:{100}:email', 'alice@example.com')

# 파이프라인 (같은 슬롯만)
pipe = rc.pipeline()
pipe.set('item:{shop}:1', 'A')
pipe.set('item:{shop}:2', 'B')
pipe.execute()
```

### 3.7 제한사항

```
❌ 다중 키 연산 제한
   MGET, MSET 등은 같은 슬롯 키만 가능
   → 해시 태그 사용으로 해결

❌ 트랜잭션 제한
   MULTI/EXEC는 같은 슬롯 키만 가능

❌ Lua 스크립트 제한
   스크립트 내 키가 같은 슬롯에 있어야 함

❌ Pub/Sub 확장성
   일반 Pub/Sub는 모든 노드에 브로드캐스트
   → Sharded Pub/Sub (Redis 7.0+) 사용
```

## 4. Sentinel vs Cluster 비교

| 항목 | Sentinel | Cluster |
|------|----------|---------|
| **목적** | 고가용성 | 샤딩 + 고가용성 |
| **데이터 분산** | 없음 (단일 마스터) | 있음 (16384 슬롯) |
| **수평 확장** | 불가 | 가능 |
| **다중 키 연산** | 제한 없음 | 같은 슬롯만 |
| **운영 복잡도** | 낮음 | 높음 |
| **최소 노드** | Master 1 + Replica 1 + Sentinel 3 | Master 3 + Replica 3 |
| **장애 조치** | Sentinel이 수행 | 노드 자체 수행 |

## 5. 복제 (Replication)

### 5.1 동작 방식

```
┌─────────┐                    ┌─────────┐
│ Master  │ ───── 복제 ─────► │ Replica │
└─────────┘                    └─────────┘
     │                              │
  쓰기 허용                      읽기 전용
```

### 5.2 Redis 8.x 복제 개선

```
기존: 단일 복제 스트림

Redis 8.0+: 이중 복제 스트림
  - 스트림 1: 스냅샷 전송
  - 스트림 2: 변경 사항 동시 전송
  → 최대 18% 빠른 복제
```

### 5.3 복제 설정

```redis
# Replica 설정
REPLICAOF 192.168.1.100 6379

# 복제 상태 확인
INFO replication
```

## 6. 영속성 (Persistence)

### 6.1 RDB (Snapshotting)

```
장점:
- 컴팩트한 단일 파일
- 빠른 재시작

단점:
- 스냅샷 사이 데이터 유실 가능
- fork() 비용

설정:
save 900 1     # 900초 동안 1개 변경 시
save 300 10    # 300초 동안 10개 변경 시
save 60 10000  # 60초 동안 10000개 변경 시
```

### 6.2 AOF (Append Only File)

```
장점:
- 더 나은 내구성
- 데이터 유실 최소화

단점:
- 파일 크기 증가
- 재시작 시간 증가

설정:
appendonly yes
appendfsync everysec  # 매초 동기화 (권장)
# appendfsync always  # 매 쓰기마다 (느림)
# appendfsync no      # OS에 맡김 (빠름)
```

### 6.3 권장 설정

```
# 최대 내구성: RDB + AOF 동시 사용
save 900 1
save 300 10
save 60 10000
appendonly yes
appendfsync everysec
```

## 7. Kubernetes 배포

### 7.1 Redis Sentinel on K8s

```yaml
# StatefulSet으로 배포
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: redis-sentinel
spec:
  serviceName: redis-sentinel
  replicas: 3
  selector:
    matchLabels:
      app: redis-sentinel
  template:
    spec:
      containers:
      - name: sentinel
        image: redis:8
        command: ["redis-sentinel"]
        args: ["/etc/redis/sentinel.conf"]
```

### 7.2 Redis Cluster on K8s

```yaml
# Bitnami Helm Chart 사용 권장
helm install redis-cluster bitnami/redis-cluster \
  --set cluster.nodes=6 \
  --set cluster.replicas=1
```

## 8. 모니터링

### 8.1 핵심 메트릭

| 메트릭 | 설명 | 임계값 |
|--------|------|--------|
| **used_memory** | 메모리 사용량 | maxmemory의 80% |
| **connected_clients** | 연결된 클라이언트 | 설정의 80% |
| **instantaneous_ops_per_sec** | 초당 연산 | 기준선 대비 |
| **keyspace_hits/misses** | 캐시 히트율 | > 90% 권장 |
| **replication_lag** | 복제 지연 | < 1초 |

### 8.2 명령어

```redis
# 서버 정보
INFO

# 메모리 상태
INFO memory

# 복제 상태
INFO replication

# 클라이언트 목록

# 느린 쿼리
```
