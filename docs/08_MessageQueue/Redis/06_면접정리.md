# Redis 메시지 큐 면접 정리

## 1. Redis 기본

### Q: Redis란 무엇인가요?

> **Redis는 인메모리 데이터 구조 저장소**로, 데이터베이스, 캐시, 메시지 브로커로 사용됩니다. C언어로 작성되어 초저지연(~0.1ms)을 제공하며, 다양한 데이터 구조(String, List, Set, Sorted Set, Hash, Stream 등)를 지원합니다.

**추가 설명**:
- Redis 8.x에서 최대 87% 빠른 명령어 처리, 2x 처리량 향상
- 2025년 AGPLv3 라이선스로 오픈소스 복귀

---

### Q: Redis로 메시지 큐를 구현하는 방법은?

> Redis는 **4가지 방식**으로 메시지 큐를 구현할 수 있습니다:
>
> 1. **Pub/Sub**: 실시간 브로드캐스트, 메시지 비보존 (Fire-and-Forget)
> 2. **List**: LPUSH/BRPOP으로 FIFO 큐, 단순 작업 큐에 적합
> 3. **Sorted Set**: Score 기반 우선순위 큐, 지연 큐
> 4. **Streams**: Kafka 스타일, Consumer Group, 메시지 보존, ACK 지원

**선택 기준**:
| 요구사항 | 추천 |
|---------|------|
| 메시지 유실 허용, 실시간 | Pub/Sub |
| 단순 FIFO | List |
| 우선순위/지연 | Sorted Set |
| 보존, Consumer Group | Streams |

---

## 2. Redis Streams

### Q: Redis Streams의 특징은?

> Redis Streams는 **Kafka 스타일의 로그 기반 메시징**을 제공합니다. 메시지가 명시적으로 삭제될 때까지 보존되며, Consumer Group을 통해 분산 처리가 가능합니다. ACK 메커니즘으로 메시지 처리를 보장할 수 있습니다.

**핵심 명령어**:
- `XADD`: 메시지 추가
- `XREAD`: 메시지 읽기
- `XREADGROUP`: Consumer Group으로 읽기
- `XACK`: 처리 확인

---

### Q: Consumer Group이란?

> **Consumer Group**은 여러 Consumer가 하나의 Stream을 분산 처리할 수 있게 해주는 기능입니다. 각 메시지는 그룹 내 하나의 Consumer에게만 전달되며, PEL(Pending Entries List)로 처리 중인 메시지를 추적합니다.

```
Stream ──► Group ──┬── Consumer A (메시지 1, 4)
                   ├── Consumer B (메시지 2, 5)
                   └── Consumer C (메시지 3, 6)
```

**장점**:
- 분산 처리로 처리량 향상
- ACK로 정확히 한 번 처리 보장
- 장애 시 다른 Consumer가 인수 가능 (XCLAIM)

---

### Q: XACK의 중요성은?

> **XACK**는 메시지 처리 완료를 확인하는 명령어입니다. ACK하지 않은 메시지는 PEL에 남아있어 다시 전달될 수 있습니다. 메모리 효율성과 정확한 처리 보장을 위해 처리 후 반드시 ACK해야 합니다.

```redis
XACK mystream mygroup 1526569495631-0
```

---

## 3. Redis Pub/Sub

### Q: Pub/Sub의 한계는?

> Redis Pub/Sub는 **Fire-and-Forget** 방식으로, 여러 한계가 있습니다:
>
> 1. **메시지 비보존**: 구독자 없으면 메시지 유실
> 2. **ACK 없음**: 전달 확인 불가
> 3. **재처리 불가**: 놓친 메시지 복구 불가
> 4. **클러스터 확장성**: 모든 노드에 브로드캐스트 (음의 확장성)

**대안**: 신뢰성이 필요하면 **Streams** 사용

---

### Q: Sharded Pub/Sub란?

> **Sharded Pub/Sub**는 Redis 7.0에서 도입된 기능으로, 클러스터 확장성 문제를 해결합니다. 채널이 특정 슬롯에 할당되어 해당 Shard 내에서만 메시지가 전파됩니다.

```redis
SSUBSCRIBE news.sports   # Sharded 구독
SPUBLISH news.sports "Goal!"  # Sharded 발행
```

**장점**: 노드 증가 시 선형 확장 가능

---

## 4. Redis List Queue

### Q: BRPOP vs RPOP의 차이는?

> **BRPOP**은 블로킹 연산으로, 리스트가 비어있으면 데이터가 올 때까지 대기합니다. **RPOP**은 즉시 반환하여 리스트가 비어있으면 nil을 반환합니다.

```redis
BRPOP queue 5   # 최대 5초 대기
RPOP queue      # 즉시 반환
```

**BRPOP 장점**: 폴링 없이 CPU 효율적

---

### Q: Reliable Queue 패턴이란?

> 기본 RPOP은 Worker 크래시 시 메시지가 유실됩니다. **Reliable Queue**는 LMOVE(또는 BRPOPLPUSH)를 사용해 메시지를 처리 중 리스트로 원자적으로 이동시킵니다.

```
1. LMOVE tasks processing RIGHT LEFT  # 원자적 이동
2. 작업 처리
3. LREM processing 1 "task_data"      # 완료 시 제거
4. (크래시 시) processing에서 복구 가능
```

---

## 5. Redis Sorted Set Queue

### Q: Sorted Set으로 우선순위 큐를 어떻게 구현하나요?

> Score를 우선순위로 사용합니다. 낮은 Score가 높은 우선순위입니다.

```redis
ZADD tasks 1 "critical_task"   # 최우선
ZADD tasks 5 "normal_task"     # 보통
ZADD tasks 10 "low_task"       # 낮음

ZPOPMIN tasks  # 가장 높은 우선순위 작업 반환
```

---

### Q: Sorted Set으로 지연 큐를 어떻게 구현하나요?

> Score를 **실행 예정 Unix 타임스탬프**로 사용합니다.

```redis
# 5분 후 실행
ZADD delayed_queue 1704067500 '{"action":"send_email"}'

# 현재 시간 이전 작업 조회
ZRANGEBYSCORE delayed_queue -inf 1704067200 LIMIT 0 10
```

**장점**: O(log N)으로 효율적인 범위 쿼리

---

### Q: 대기열 시스템에서 Kafka 대신 Redis Sorted Set을 사용하는 이유는?

> 대기열 시스템에서 Redis Sorted Set이 유리한 이유:
>
> 1. **순번 조회**: ZRANK로 O(log N)에 순번 확인 (Kafka는 불가)
> 2. **중간 이탈**: ZREM으로 대기열 이탈 가능 (Kafka는 불가)
> 3. **중복 방지**: 고유 멤버로 자동 멱등성
> 4. **단순성**: 운영 복잡도 낮음

---

## 6. 고가용성

### Q: Sentinel과 Cluster의 차이는?

| 항목 | Sentinel | Cluster |
|------|----------|---------|
| **목적** | 고가용성 | 샤딩 + 고가용성 |
| **데이터 분산** | 없음 | 16384 슬롯 |
| **수평 확장** | 불가 | 가능 |
| **운영 복잡도** | 낮음 | 높음 |

**선택 기준**:
- 데이터가 단일 노드에 들어가면 → Sentinel
- 수평 확장 필요하면 → Cluster

---

### Q: Sentinel의 Quorum이란?

> **Quorum**은 Master 장애를 확정하기 위해 필요한 최소 Sentinel 동의 수입니다. 일반적으로 과반수(예: 3개 중 2개)로 설정합니다.

**장애 조치 과정**:
1. 개별 Sentinel이 SDOWN(주관적 다운) 감지
2. Quorum 이상이 동의하면 ODOWN(객관적 다운)
3. Leader Sentinel이 Replica를 Master로 승격

---

### Q: Redis Cluster의 해시 슬롯이란?

> Redis Cluster는 **16384개 해시 슬롯**으로 데이터를 분산합니다. 키의 CRC16 해시값을 16384로 나눈 나머지로 슬롯을 결정합니다.

```
슬롯 = CRC16(key) mod 16384
```

**해시 태그**: 같은 슬롯 강제
```
user:{1000}:profile → 슬롯 X
user:{1000}:settings → 슬롯 X (동일)
```

---

## 7. Redis vs Kafka vs RabbitMQ

### Q: 언제 Redis를 선택해야 하나요?

> **Redis 선택 시점**:
> - 이미 Redis를 캐시로 사용 중
> - 초저지연이 필수 (~0.1ms)
> - 단순한 큐 패턴
> - 소~중규모 워크로드
> - 빠른 개발/프로토타이핑

**Redis가 부적합한 경우**:
- 대용량 이벤트 스트리밍 → Kafka
- 복잡한 라우팅 → RabbitMQ
- 장기 메시지 보존 → Kafka

---

### Q: Redis Streams vs Kafka 비교?

| 항목 | Redis Streams | Kafka |
|------|--------------|-------|
| 처리량 | 수천~수만/s | 수십만/s |
| 지연시간 | ~0.15ms | ~수십ms |
| 메시지 보존 | 인메모리 | 디스크 |
| 파티셔닝 | 없음 | 있음 |
| Consumer Group | 있음 | 있음 |
| 운영 복잡도 | 낮음 | 높음 |

---

## 8. 실전 질문

### Q: Redis에서 메시지 유실을 방지하는 방법은?

> 1. **Streams 사용**: ACK 메커니즘으로 처리 보장
> 2. **Reliable Queue 패턴**: LMOVE로 원자적 이동
> 3. **AOF 영속성**: appendfsync everysec 설정
> 4. **Sentinel/Cluster**: 고가용성 구성

---

### Q: Redis 성능 최적화 방법은?

> 1. **파이프라이닝**: 여러 명령을 배치로 전송
> 2. **블로킹 명령**: BRPOP으로 폴링 방지
> 3. **Lua 스크립트**: 원자적 다중 연산
> 4. **적절한 데이터 구조**: 용도에 맞는 구조 선택
> 5. **메모리 관리**: maxmemory, TTL 설정

---

### Q: Redis 8.x의 주요 변경사항은?

> 1. **I/O 스레딩**: 최대 112% 처리량 향상
> 2. **새로운 복제 메커니즘**: 18% 빠른 복제
> 3. **Query Engine**: 16x 쿼리 처리 능력
> 4. **새 데이터 구조**: Vector Set, JSON, Time Series 등
> 5. **Streams 개선**: XACKDEL, XDELEX 등

---
