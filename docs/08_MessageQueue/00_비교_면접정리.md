# Kafka vs RedPanda vs RabbitMQ 비교 - 면접 정리

## 핵심 한 줄 정리

> **Kafka는 대용량 스트리밍의 업계 표준, RedPanda는 Kafka 호환 + 단순 운영, RabbitMQ는 유연한 라우팅 + 다중 프로토콜의 전통적 메시지 브로커입니다.**

---

## 면접 필수 암기

### Q1. 세 시스템을 한 문장으로 비교해주세요.

**답변 (30초):**
> - **Kafka**: 2011년 출시된 **업계 표준** 이벤트 스트리밍 플랫폼. 대용량 처리, 메시지 보존, 풍부한 에코시스템이 강점.
> - **RedPanda**: 2020년 출시된 **Kafka 호환** C++ 구현. JVM/ZK 없이 더 적은 리소스로 유사 성능, 운영 단순화가 강점.
> - **RabbitMQ**: 2007년 출시된 **전통적 메시지 브로커**. Exchange 기반 유연한 라우팅, 다중 프로토콜(AMQP, MQTT) 지원이 강점.

---

### Q2. 아키텍처의 핵심 차이는?

**답변:**

| 항목 | Kafka | RedPanda | RabbitMQ |
|------|-------|----------|----------|
| **언어** | Java/Scala | C++ | Erlang |
| **메시지 모델** | **Pull** (로그) | **Pull** (로그) | **Push** (큐) |
| **데이터 구조** | Append-only 로그 | Append-only 로그 | Queue + Streams |
| **라우팅** | 토픽/파티션 | 토픽/파티션 | **Exchange** (4타입) |
| **메타데이터** | KRaft | 내장 Raft | Khepri |

> **Pull vs Push**: Kafka/RedPanda는 Consumer가 가져가고, RabbitMQ는 Broker가 밀어줍니다.

---

### Q3. 성능 비교는 어떻게 되나요?

**답변:**

| 메트릭 | Kafka | RedPanda | RabbitMQ |
|--------|-------|----------|----------|
| **처리량** | 600+ MB/s | 600+ MB/s | 30-100 MB/s |
| **메시지/초** | 100만+ | 100만+ | 5-10만 (Queue) |
| **P99 지연** | ~5ms | ~2-3ms | <1ms (저부하) |
| **확장성** | 매우 높음 | 높음 | 중간 |

> RabbitMQ는 처리량은 낮지만 **저부하에서 가장 낮은 지연시간**을 제공합니다.

---

### Q4. 고가용성은 어떻게 구현되나요?

**답변:**

| 시스템 | 방식 | 도입 시점 |
|--------|------|----------|
| **Kafka** | KRaft + ISR 복제 | KRaft: 2025년 완성 |
| **RedPanda** | 내장 Raft | 2020년부터 |
| **RabbitMQ** | Quorum Queues (Raft) | **2019년** |

> 세 시스템 모두 현재는 **Raft 계열 합의 알고리즘**을 사용합니다. RabbitMQ가 가장 먼저 Raft를 도입했고, Kafka가 가장 나중에 완성했습니다.

---

### Q5. 언제 어떤 시스템을 선택해야 하나요?

**답변:**

| 요구사항 | 추천 | 이유 |
|---------|------|------|
| 대용량 이벤트 스트리밍 | **Kafka** | 검증된 확장성, 에코시스템 |
| Kafka 대체 + 운영 단순화 | **RedPanda** | 호환성 + 적은 노드 |
| 복잡한 라우팅 | **RabbitMQ** | Exchange 4타입 |
| 다중 프로토콜 (MQTT, AMQP) | **RabbitMQ** | 네이티브 지원 |
| 작업 큐 (Task Queue) | **RabbitMQ** | Push 모델, Ack |
| 메시지 재처리 필요 | Kafka/RedPanda | 로그 기반 보존 |
| 이벤트 소싱 | Kafka/RedPanda | 영구 저장 |

## 깊은 이해를 위한 설명

### 패러다임의 근본적 차이

**이벤트 스트리밍 (Kafka, RedPanda):**
```
Producer → [Log: M1, M2, M3, M4, M5] → Consumer
                     ↑
              오프셋으로 재읽기 가능
              메시지 보존 (삭제 안 됨)
```

**메시지 큐 (RabbitMQ Queue):**
```
Producer → [Queue: M1, M2, M3] → Consumer
                     ↓
              소비 후 삭제
              재처리 불가 (DLQ로 이동)
```

**하이브리드 (RabbitMQ Streams):**
```
RabbitMQ 3.9+:
├── Queue: 전통적 메시지 큐 패턴
└── Streams: Kafka 스타일 로그 패턴
    → 두 패러다임 모두 지원
```

### 2024-2025년 주요 변화

**Kafka 4.0 (2025.03):**
- ZooKeeper **완전 제거**, KRaft만 지원
- 100만+ 파티션 지원

**RabbitMQ 4.0 (2024):**
- Classic Mirrored Queue **제거**
- Khepri 메타데이터 저장소 정식 지원

**RedPanda:**
- Tiered Storage 강화
- Serverless 옵션 추가

> 세 시스템 모두 **외부 의존성 제거**와 **Raft 기반 메타데이터**로 수렴 중입니다.

### 선택 결정 플로우

```
1. 복잡한 라우팅이 필요한가?
   └─ YES → RabbitMQ

2. 메시지 재처리/장기 보존이 핵심인가?
   └─ YES → Kafka 또는 RedPanda

3. MQTT/AMQP 프로토콜이 필요한가?
   └─ YES → RabbitMQ

4. 기존 Kafka 인프라가 있는가?
   └─ YES → Kafka 유지 또는 RedPanda 마이그레이션

5. 운영 단순화가 최우선인가?
   └─ YES → RedPanda
   └─ NO → Kafka (에코시스템)
```

## 코드 레벨 비교

### 동일한 기능의 다른 구현

**메시지 발행:**
```java
// Kafka / RedPanda (동일 코드)
producer.send(new ProducerRecord<>("topic", "key", "value"));

// RabbitMQ
channel.basicPublish("exchange", "routing.key", null, "value".getBytes());
```

**메시지 소비:**
```java
// Kafka / RedPanda
consumer.subscribe(List.of("topic"));
ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));

// RabbitMQ
channel.basicConsume("queue", false, (tag, delivery) -> {
    // 처리
    channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
}, tag -> {});
```

> Kafka/RedPanda는 **Pull**, RabbitMQ는 **Push(callback)** 방식입니다.

## 면접 예상 꼬리 질문

### Q. 세 시스템을 모두 쓰는 경우가 있나요?

**답변:**
> 네, **하이브리드 아키텍처**가 있습니다:
>
> ```
> IoT 디바이스 ──MQTT──▶ RabbitMQ ──▶ Kafka ──▶ 데이터 레이크
>                      (프로토콜 변환)  (대용량 저장)
> ```
>
> - RabbitMQ: 다양한 프로토콜 수집, 라우팅
> - Kafka: 대용량 저장, 스트림 처리
>
> 각 시스템의 강점을 활용하는 것이 핵심입니다.

### Q. RabbitMQ Streams가 있으면 Kafka가 필요 없나요?

**답변:**
> **상황에 따릅니다.**
>
> **Streams로 충분한 경우:**
> - 중간 규모 처리량
> - 큐 + 스트림 두 패러다임 필요
> - 기존 RabbitMQ 인프라 활용
>
> **Kafka가 더 나은 경우:**
> - 대규모 스트리밍 (수백만 msg/s)
> - Kafka Connect, ksqlDB 에코시스템 필요
> - 검증된 대규모 프로덕션 경험
>
> Streams는 2021년 도입으로 **Kafka만큼 성숙하지 않습니다**.

### Q. 마이크로서비스에서 어떤 것을 추천하나요?

**답변:**
> **일반적 추천: RabbitMQ**
>
> 이유:
> - 서비스 간 **요청-응답** 패턴에 적합
> - **복잡한 라우팅** (서비스별 큐)
> - **작업 큐** (백그라운드 작업)
> - Push 모델로 **즉시 처리**
>
> **Kafka가 더 나은 경우:**
> - 이벤트 소싱 아키텍처
> - 서비스 간 **이벤트 공유** (여러 서비스가 같은 이벤트 구독)
> - **감사 로그** 필요

### Q. 비용 관점에서 비교하면?

**답변:**

| 항목 | Kafka | RedPanda | RabbitMQ |
|------|-------|----------|----------|
| **라이선스** | 무료 (Apache 2.0) | 제한적 (BSL) | 무료 (MPL) |
| **노드 수** | 많음 | **적음** | 중간 |
| **운영 인력** | 높음 | **낮음** | 중간 |
| **관리형 비용** | Confluent (높음) | RedPanda Cloud | CloudAMQP |

> **총 소유 비용(TCO)**: RedPanda가 노드 절감으로 유리할 수 있지만, Kafka의 풍부한 인력 풀과 문서가 숨은 비용을 줄여줄 수 있습니다.

### Q. 2026년 트렌드는 어떻게 예상하나요?

**답변:**
> 1. **Raft 기반 통합**: 세 시스템 모두 Raft 계열로 수렴 완료
> 2. **Tiered Storage**: 클라우드 스토리지 연동 표준화
> 3. **Serverless**: 사용량 기반 과금 옵션 확대
> 4. **하이브리드**: 큐 + 스트림 경계 모호화 (RabbitMQ Streams 성숙)
>
> **Kafka는 여전히 표준**, RedPanda는 **틈새 시장 확대**, RabbitMQ는 **하이브리드 강점** 유지 예상.

│                 메시지 브로커 선택 가이드                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Kafka                                                      │
│  ├─ 언제: 대용량 스트리밍, 이벤트 소싱, 검증된 솔루션 필요    │
│  ├─ 강점: 에코시스템, 확장성, 커뮤니티                       │
│  └─ 약점: 운영 복잡도, 리소스 사용량                         │
│                                                             │
│  RedPanda                                                   │
│  ├─ 언제: Kafka 대체, 운영 단순화, 비용 절감                 │
│  ├─ 강점: Kafka 호환, 적은 노드, 자동 튜닝                   │
│  └─ 약점: BSL 라이선스, 성숙도, 벤치마크 과장                │
│                                                             │
│  RabbitMQ                                                   │
│  ├─ 언제: 복잡한 라우팅, 다중 프로토콜, 작업 큐              │
│  ├─ 강점: Exchange 라우팅, MQTT/AMQP, 하이브리드             │
│  └─ 약점: 처리량 한계, 확장 어려움                          │
│                                                             │
│  핵심 질문: "메시지를 보존해야 하나요?"                      │
│  ├─ YES + 대용량 → Kafka/RedPanda                          │
│  ├─ YES + 중소규모 → RabbitMQ Streams                      │
│  └─ NO + 라우팅 필요 → RabbitMQ Queue                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
