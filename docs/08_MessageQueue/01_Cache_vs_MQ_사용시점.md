# Cache vs MQ - 언제 무엇을 사용할까?

> 출처: [코딩하는 기술사 - 성능튜닝의 양대산맥, Cache와 MQ](https://www.youtube.com/@codingtechnician)

## 1. 핵심 개념

### 면접 질문

> "트래픽이 폭주해서 서버가 느려졌습니다. Cache와 MQ 중 무엇을 도입하시겠습니까?"

### 잘못된 답변

```
❌ "레디스가 빠르니까 레디스 캐시를 도입하겠습니다."
❌ "카프카를 도입하겠습니다."
```

**문제점:**
1. 특정 솔루션 이름(Redis, Kafka)을 바로 언급 → **개념을 먼저 설명해야 함**
2. 원인 분석 없이 솔루션 제시 → **병목 지점 파악이 우선**

### 올바른 답변

```
✅ "먼저 모니터링 툴을 통해 병목 지점이 어디인지 파악해서 전략을 수립하겠습니다.
   - 조회 쪽에 부하가 크다면 → Cache(예: Redis)를 도입해서 DB 부하를 줄이겠습니다.
   - 트래픽 폭주로 처리가 지연된다면 → MQ(예: Kafka)를 둬서 작업을 비동기로 전환하겠습니다."
```

---

## 2. Cache와 MQ의 본질적 차이

| 구분 | Cache | Message Queue |
|------|-------|---------------|
| **주 목적** | **읽기** 성능 향상 | **쓰기/처리** 성능 향상 |
| **핵심 원리** | **재사용** (똑같은 일을 반복하지 마) | **비동기** (나중에 처리) |
| **배치 위치** | DB **앞단** | API와 Worker **사이** |
| **대표 솔루션** | Redis, Memcached | Kafka, RabbitMQ |

---

## 3. Cache (캐시)

### 핵심 원리: "똑같은 일을 반복해서 하지 마"

자주 찾는 데이터나 복잡한 계산 결과를 **미리 저장**해서 재사용합니다.

### 비유: 뷔페 vs 주문 요리

```
주문 요리 (캐시 없음):
  손님 요청 → 요리사가 새로 조리 → 시간 소요

뷔페 (캐시 있음):
  미리 만들어둔 음식 → 바로 가져감 → 빠름
```

### 캐시 전략 (Cache-Aside Pattern)

```
┌─────────────────────────────────────────────────┐
│                  요청 처리 흐름                  │
├─────────────────────────────────────────────────┤
│                                                 │
│  1. 캐시에 데이터가 있는지 확인                   │
│     │                                           │
│     ├─ Cache HIT → 캐시에서 바로 반환           │
│     │                                           │
│     └─ Cache MISS → DB 조회                     │
│                      → 캐시에 저장              │
│                      → 반환                     │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 언제 Cache를 도입할까?

```
✅ SELECT 조회가 많아서 DB가 느려졌을 때
✅ 동일한 데이터를 반복 조회할 때
✅ 계산 비용이 높은 결과를 재사용할 때
```

### Cache 도입 시 고려사항

| 고려사항 | 설명 |
|---------|------|
| **데이터 정합성** | DB와 캐시 값이 다를 수 있음 |
| **만료 시간 (TTL)** | 언제 캐시를 갱신할지 정책 필요 |
| **캐시 갱신 전략** | Write-through, Write-back 등 |
| **캐시 스탬피드** | 캐시 만료 시 동시 요청 폭주 |

## 4. Message Queue (MQ)

### 핵심 원리: "나중에 한다" + "디커플링"

요청을 즉시 처리하지 않고 **대기열에 넣어두고** 비동기로 처리합니다.

### 비유: 맛집 대기표

```
대기표 없음 (동기 처리):
  손님 → 주문 → 음식 나올 때까지 대기 → 다음 손님 못 받음

대기표 있음 (비동기 처리):
  손님 → 대기표 받음 → 다음 손님 받음 → 순서대로 처리
```

### MQ 처리 흐름

```
┌─────────────────────────────────────────────────────────────┐
│                    MQ 비동기 처리 흐름                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Client                                                     │
│     │                                                       │
│     │ 1. 요청                                               │
│     ▼                                                       │
│  ┌─────┐                                                    │
│  │ API │ ── 2. 즉시 MQ에 넣고                               │
│  └──┬──┘    3. 바로 응답 반환 (사용자는 빠르게 응답 받음)      │
│     │                                                       │
│     ▼                                                       │
│  ┌─────┐                                                    │
│  │  MQ │ ◀─────────────────────┐                           │
│  └──┬──┘                       │                           │
│     │                          │ 4. Worker가 메시지 소비     │
│     ▼                          │                           │
│  ┌────────┐                    │                           │
│  │ Worker │ ───────────────────┘                           │
│  └────────┘                                                │
│     │                                                       │
│     │ 5. 실제 처리 (DB 저장, 이메일 발송, 외부 API 호출 등)   │
│     ▼                                                       │
│  ┌────┐                                                    │
│  │ DB │                                                    │
│  └────┘                                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 언제 MQ를 도입할까?

```
✅ 주문, 결제 등 복잡한 처리(INSERT, UPDATE)가 느릴 때
✅ 트래픽 폭주로 처리가 밀릴 때
✅ 시스템 간 결합도를 낮추고 싶을 때
✅ 외부 API 호출, 이메일 발송 등 시간이 오래 걸리는 작업
```

### MQ 도입 시 고려사항

| 고려사항 | 설명 |
|---------|------|
| **복잡도 증가** | 비동기 처리로 전체 흐름 설계 필요 |
| **지연된 결과** | 처리가 밀리면 사용자가 결과를 늦게 받음 |
| **실패 처리** | Dead Letter Queue, 재시도 전략 필요 |
| **순서 보장** | 순서가 중요하면 추가 설계 필요 |

## 5. 아키텍처 배치

### Cache 위치

```
User → API → [Cache] → DB
              ↑
         DB 앞단에 배치
```

### MQ 위치

```
User → API → [MQ] → Worker → DB/외부시스템
              ↑
         API와 Worker 사이에 배치
```

## 6. 면접 후속 질문

### Q1. 비동기(MQ) 처리하면 사용자에게 완료 응답은 어떻게 주나요?

**답변:**
> 1. **폴링(Polling)**: 클라이언트가 주기적으로 상태 조회
> 2. **WebSocket/SSE**: 서버에서 완료 시 푸시 알림
> 3. **Callback URL**: 완료 시 지정된 URL로 호출

### Q2. 캐시 스탬피드(Cache Stampede) 현상은 무엇인가요?

**답변:**
> 캐시가 만료되는 순간 **동시에 많은 요청**이 DB로 몰리는 현상입니다.
>
> **해결책:**
> - Lock을 걸어 하나의 요청만 DB 조회
> - 캐시 만료 전에 미리 갱신 (Background refresh)
> - TTL에 랜덤 값 추가 (Jitter)

### Q3. Redis는 캐시인가요? MQ인가요?

**답변:**
> Redis는 **둘 다 가능**합니다.
> - **캐시**: 기본적인 Key-Value 저장소로 캐시 용도
> - **MQ**: List(LPUSH/RPOP), Stream, Pub/Sub으로 MQ 구현 가능
>
> 단, 전문 MQ(Kafka, RabbitMQ)만큼의 기능(메시지 보존, 컨슈머 그룹 등)은 제한적입니다.

## 7. 정리 비교표

| 항목 | Cache | Message Queue |
|------|-------|---------------|
| **목적** | 읽기 성능 향상 | 쓰기/처리 성능 향상 + 디커플링 |
| **핵심 원리** | 재사용 | 비동기, 버퍼링 |
| **위치** | DB 앞단 | API ↔ Worker 사이 |
| **대표 솔루션** | Redis, Memcached | Kafka, RabbitMQ |
| **고려사항** | 데이터 정합성, TTL | 복잡도, 지연 응답 |
