# 대기열 시스템에서 Kafka가 아닌 Redis를 사용하는 이유

> 출처: [코딩하는 기술사 - 대규모 예매 시스템에서 대기열에 Kafka가 아닌 Redis를 사용하는 이유](https://www.youtube.com/@codingtechnician)

## 1. 핵심 질문

> "대기열을 MQ(Kafka) 대신 Redis Sorted Set을 사용하는 이유는?"

### 배경

선착순 예매 시스템(KTX, 콘서트 티켓 등)에서 **대기열**이 필요합니다.

- MQ(Kafka): 선입선출(FIFO) 처리 가능
- Redis Sorted Set: 선입선출(FIFO) 처리 가능

**그렇다면 왜 MQ를 안 쓰고 Redis를 쓸까요?**

---

## 2. 핵심 답변

> **"보여줘야 하기 때문"**

### 대기열의 필수 UX

사용자에게 **"내 앞에 몇 명이 남았는지"** 보여주는 것이 핵심입니다.

```
┌─────────────────────────────────────┐
│      현재 대기 순번: 1,234번        │
│      예상 대기 시간: 약 5분         │
│      ▓▓▓▓▓▓▓▓░░░░░░░░ 45%          │
└─────────────────────────────────────┘
```

이 **실시간 피드백**이 대기열의 필수 기능입니다.

여기서부터 **MQ와 Redis의 길이 갈립니다.**

---

## 3. MQ(Kafka)의 문제점

### 문제 1: 순번 확인 불가

MQ는 본질적으로 **파이프라인** 구조입니다.

```
MQ (Kafka):
  메시지 → [  →  →  →  →  ] → 컨슘
           ↑
     내부를 랜덤 조회 불가
```

**비유: 터널**
> 터널에 차가 줄 서서 들어가지만, **내가 터널 안에서 몇 번째인지 바로 확인하기 어렵습니다.**

Kafka의 **오프셋**은 "내가 어디까지 읽었냐"의 개념이지, **특정 사용자의 위치를 즉시 찾는 기능이 아닙니다.**

> **결론: MQ는 대기열의 "상태 조회"에 적합하지 않습니다.**

---

### 문제 2: 중복 클릭/새로고침 처리

대기열에서 흔한 장애 원인:
- 사용자가 **빨리 하려고 클릭 연타**
- **F5 새로고침** 반복

```
MQ:
  클릭 → 메시지 추가
  클릭 → 메시지 추가 (중복!)
  클릭 → 메시지 추가 (중복!)

결과: 동일 사용자가 여러 번 대기열에 등록됨
```

MQ에서 중복 제거를 하려면 **컨슈머 단계에서 별도 로직**이 필요합니다.

---

### 문제 3: 중간 취소 처리

사용자가 대기하다가 **취소**하면, 대기열에서 빼야 합니다.

```
MQ:
  [A] [B] [C] [D] [E]
        ↑
      C가 취소하고 싶음
      → 중간 삭제 불가!
```

MQ는 **앞에서부터 순서대로 꺼내는 구조**라 중간 삭제가 어렵습니다.

구현하려면:
1. 별도 **취소 목록** 저장소 필요
2. 컨슈머에서 **취소 여부 확인** 로직 필요

---

## 4. Redis Sorted Set의 장점

### Redis Sorted Set = "랭킹 명단"

Redis Sorted Set(ZSET)은 **큐가 아니라 랭킹 명단**에 가깝습니다.

```
Redis Sorted Set:
  ┌──────────────────────────────────┐
  │ Member (유저ID)  │ Score (시간)  │
  ├──────────────────────────────────┤
  │ user_001        │ 1706000001    │
  │ user_002        │ 1706000002    │
  │ user_003        │ 1706000003    │
  │ ...             │ ...           │
  └──────────────────────────────────┘
```

대기열 자체를 **데이터 구조로 들고 있어서** 특정 유저가 몇 번째인지 **바로 알 수 있습니다.**

---

### 장점 1: 순번 조회 (ZRANK)

```bash
# 특정 유저의 순번 조회
ZRANK waiting_queue user_123
# 결과: 456 (457번째)
```

사용자가 **새로고침해도** 처음부터 다시 줄 서는 게 아니라, **줄 자체가 서버에 남아 있습니다.**

---

### 장점 2: 멱등성 보장 (중복 방지)

```bash
# 동일 유저, 동일 스코어로 추가
ZADD waiting_queue 1706000001 user_123
ZADD waiting_queue 1706000001 user_123  # 아무리 여러 번 해도
ZADD waiting_queue 1706000001 user_123  # 하나로 유지됨!
```

**멱등성(Idempotency)**: 같은 요청을 여러 번 수행해도 결과가 동일

> **Sorted Set의 동일 멤버 + 동일 스코어 추가는 멱등적입니다.**
> 
> 대기열에서 멱등성이 보장되면 **트래픽이 훨씬 안정됩니다.**

---

### 장점 3: 중간 삭제 (ZREM)

```bash
# 대기열 중간에 있는 유저 삭제
ZREM waiting_queue user_456
```

**한 줄 명령어**로 중간 취소가 가능합니다.

---

### 장점 4: 상위 N명 추출 (ZPOPMIN)

```bash
# 처음부터 100명 추출 (추출 후 삭제)
ZPOPMIN waiting_queue 100
```

---

## 5. Redis Sorted Set 대기열 운영 명령어

| 기능 | 명령어 | 설명 |
|------|--------|------|
| **대기열 등록** | `ZADD` | 유저를 대기열에 추가 |
| **순번 조회** | `ZRANK` | 특정 유저의 순번 확인 |
| **내 뒤 인원** | `ZCOUNT` | 특정 범위의 인원 수 |
| **중간 삭제** | `ZREM` | 대기열에서 특정 유저 제거 |
| **상위 N명 추출** | `ZPOPMIN` | 처음부터 N명 꺼내기 (원자적) |
| **전체 대기 인원** | `ZCARD` | 대기열 총 인원 수 |

│     ▼                                                           │

│     │                                                           │
│     │ 4. Worker가 처리                                          │
│     ▼                                                           │
│  ┌───────────────────────────────────┐                         │
│  │    DB / 외부 시스템 / 알림        │                          │

### 역할 분담

| 위치 | 솔루션 | 역할 | 중요 포인트 |
|------|--------|------|------------|
| **앞단** | Redis Sorted Set | 대기열 관리 | 상태 조회, 실시간 피드백 |
| **뒷단** | MQ (Kafka) | 후처리 | 비동기, 처리량, 버퍼링 |

> **앞단은 대기열 조회/관리가 중요 → Redis**
> **뒷단은 처리량/비동기가 중요 → MQ**

## 7. MQ vs Redis 대기열 비교

| 항목 | MQ (Kafka) | Redis Sorted Set |
|------|-----------|------------------|
| **순번 조회** | ❌ 어려움 | ✅ ZRANK |
| **중복 방지** | ❌ 별도 로직 필요 | ✅ 멱등성 보장 |
| **중간 취소** | ❌ 어려움 | ✅ ZREM |
| **실시간 상태** | ❌ 미지원 | ✅ 즉시 조회 |
| **대용량 처리** | ✅ 최적화 | △ 중간 규모 |
| **메시지 보존** | ✅ 영구 저장 | △ 메모리 기반 |
| **비동기 처리** | ✅ 최적화 | △ 가능하지만 제한적 |

## 9. 면접 포인트

### Q. 대기열에 MQ 대신 Redis를 쓰는 이유는?

**모범 답변:**
> 대기열의 핵심 UX는 사용자에게 **"내 순번이 몇 번째인지"** 보여주는 것입니다.
>
> **MQ의 한계:**
> 1. 내부 랜덤 조회 불가 → 순번 확인 어려움
> 2. 중복 클릭 시 메시지 중복 → 별도 처리 필요
> 3. 중간 취소 → 구조적으로 어려움
>
> **Redis Sorted Set의 장점:**
> 1. `ZRANK`로 즉시 순번 조회
> 2. 동일 멤버 추가는 멱등적 → 중복 자동 방지
> 3. `ZREM`으로 중간 삭제 가능
>
> 따라서 **대기열 관리(앞단)는 Redis**, **후처리(뒷단)는 MQ**로 역할을 분담하는 것이 일반적입니다.

### Q. 그럼 MQ는 언제 쓰나요?

**답변:**
> MQ는 **비동기 처리와 버퍼링**에 사용합니다. 대기열 통과 후 **DB 저장, 알림 발송, 외부 API 호출** 등 무거운 작업을 MQ에 넣어 비동기로 처리합니다. 앞단에서 빠르게 응답하고, 뒷단에서 순차적으로 처리하는 구조입니다.
