# RedPanda 성능 및 벤치마크

## 1. RedPanda의 성능 주장

RedPanda는 다음과 같은 성능 우위를 주장합니다:

| 주장 | 내용 |
|------|------|
| **지연시간** | Kafka 대비 10배 낮은 P99.99 테일 레이턴시 |
| **리소스 효율** | 동일 워크로드에 1/3 노드 필요 |
| **처리량** | 100만+ 메시지/초 |

---

## 2. 공식 벤치마크 (RedPanda)

### 테스트 환경

```
도구: OpenMessaging Benchmark (Linux Foundation)
노드: i3en.6xlarge (AWS)
- 24 vCPU
- 192GB RAM
- NVMe SSD

클라이언트: m5n.8xlarge x 4
- 32 vCPU
- 128GB RAM
- 25Gbps Network

버전: Kafka 3.2.0, RedPanda 22.2.2
```

### 결과 (RedPanda 주장)

| 메트릭 | Kafka | RedPanda | 차이 |
|--------|-------|----------|------|
| P99.99 지연 | 수십 ms | 수 ms | **~10x** |
| 1GB/s 필요 노드 | 9대 | 3대 | **3x** |
| 처리량 | 비슷 | 비슷 | - |

## 3. 독립 벤치마크 (Confluent/Jack Vanlightly)

### 테스트 환경

```
노드: i3en.6xlarge x 3 (동일)
TLS: 동일 조건 적용
클라이언트: 동일 설정
```

### 발견된 문제점

#### 1. 프로듀서 수 증가 시 성능 저하

```
프로듀서 4개 → 50개 변경 시:

Kafka: 성능 유지
RedPanda: 성능 저하 관찰
```

#### 2. 장기 운영 시 성능 저하

```
12시간+ 연속 운영 시:

RedPanda의 성능이 점진적으로 저하
특히 데이터 보존 한계 도달 시 레이턴시 급증
```

#### 3. 워크로드 의존성

```
RedPanda가 우위인 경우:
- 적은 수의 프로듀서
- 큰 메시지 배치
- 단기 테스트

Kafka가 우위이거나 동등한 경우:
- 많은 수의 프로듀서
- 작은 메시지
- 장기 운영
```

### 결론

> "RedPanda's claims regarding Kafka and even their own performance are greatly exaggerated."
> "RedPanda can outshine only when it gets the right workload."
> — Jack Vanlightly (Confluent)

## 4. 실제 성능 특성

### 장점이 명확한 경우

| 시나리오 | 이유 |
|---------|------|
| **낮은 지연 요구** | GC 없음, Thread-per-core |
| **적은 노드로 운영** | 높은 단일 노드 효율 |
| **빠른 리더 선출** | Raft 기반 빠른 합의 |
| **예측 가능한 성능** | JVM 튜닝 불필요 |

### 주의가 필요한 경우

| 시나리오 | 이유 |
|---------|------|
| **매우 많은 프로듀서** | 성능 저하 가능 |
| **극한의 처리량** | Kafka와 유사하거나 열세 |
| **장기간 연속 운영** | 성능 일관성 검증 필요 |
| **복잡한 워크로드** | 직접 벤치마크 필수 |

## 6. 지연시간 비교

### P99 지연시간

| 부하 | Kafka | RedPanda | RabbitMQ |
|------|-------|----------|----------|
| 낮음 (30MB/s) | ~5ms | ~2-3ms | <1ms |
| 중간 (200MB/s) | ~5-10ms | ~3-5ms | 불가 |
| 높음 (600MB/s) | ~10-20ms | ~5-10ms | 불가 |

### P99.99 지연시간 (테일)

```
RedPanda 주장: Kafka 대비 10x 낮음
독립 검증: 워크로드에 따라 다름 (2-5x 범위)
```

## 7. 리소스 효율성

### CPU 사용률

```
Kafka:
- JVM 오버헤드
- GC 사이클로 인한 스파이크
- Thread Pool 경합

RedPanda:
- 네이티브 실행
- GC 없음
- Thread-per-core로 균등 분산
```

### 메모리 사용

| 항목 | Kafka | RedPanda |
|------|-------|----------|
| 힙 메모리 | 6-31GB 권장 | 해당 없음 |
| 페이지 캐시 | OS 관리 | 직접 관리 |
| 총 메모리 | 높음 | 중간 |

### 디스크 I/O

```
둘 다 Sequential I/O 최적화
RedPanda: io_uring 지원으로 추가 최적화 가능
```

## 8. 자체 벤치마크 권장사항

### 왜 직접 벤치마크해야 하는가?

1. **워크로드 특성**: 실제 사용 패턴이 벤치마크와 다름
2. **하드웨어 차이**: 클라우드/온프레미스 환경 차이
3. **설정 차이**: 튜닝에 따른 성능 변화
4. **버전 차이**: 지속적인 성능 개선

### 벤치마크 도구

```bash
# OpenMessaging Benchmark
git clone https://github.com/openmessaging/benchmark
cd benchmark

# Kafka 벤치마크
bin/benchmark --drivers kafka.yaml --workloads workload.yaml

# RedPanda 벤치마크 (동일 도구 사용)
bin/benchmark --drivers redpanda.yaml --workloads workload.yaml
```

### 테스트 시나리오 권장

| 시나리오 | 설정 |
|---------|------|
| **기본** | 100 프로듀서, 100 컨슈머, 1KB 메시지 |
| **고처리량** | 10 프로듀서, 10 컨슈머, 배치 최대화 |
| **저지연** | 단일 프로듀서, 배치 비활성화 |
| **장기 운영** | 24시간+ 연속 테스트 |
| **장애 복구** | 노드 장애 시뮬레이션 |

## 9. 성능 튜닝

### RedPanda 자동 튜닝

```bash
# 시스템 자동 최적화
rpk redpanda tune all

# 개별 튜닝
rpk redpanda tune cpu
rpk redpanda tune net
rpk redpanda tune disk
```

### 주요 설정

```yaml
# redpanda.yaml
redpanda:
  # CPU 설정
  smp: 4  # 사용할 코어 수
  
  # 메모리 설정
  memory: 8G
  
  # 배치 설정
  kafka_batch_max_bytes: 1048576
  
  # 복제 설정
  default_topic_replications: 3
  
  # 로그 설정
  log_segment_size: 134217728  # 128MB
```

## 10. 결론

### RedPanda 선택 시 기대 효과

```
✅ 운영 복잡도 감소 (단일 바이너리, 자동 튜닝)
✅ 더 적은 노드로 유사 성능
✅ 더 낮은 P99 지연시간 (대부분의 워크로드)
✅ 빠른 시작/복구 시간
```

### 주의사항

```
⚠️ 공식 벤치마크 수치를 그대로 신뢰하지 말 것
⚠️ 실제 워크로드로 직접 테스트 필수
⚠️ 장기 운영 안정성 검증 필요
⚠️ 특수 워크로드에서는 Kafka가 더 나을 수 있음
```
