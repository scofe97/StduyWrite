# RedPanda 성능 벤치마크 - 면접 정리

## 핵심 한 줄 정리

> **RedPanda는 특정 워크로드에서 Kafka보다 낮은 지연시간과 적은 노드로 운영 가능하지만, 공식 벤치마크는 과장되었다는 독립 분석이 있으므로 실제 워크로드로 직접 테스트가 필수입니다.**

---

## 면접 필수 암기

### Q1. RedPanda의 성능 주장은?

**답변:**
> RedPanda 공식 벤치마크 주장:
> - Kafka 대비 **10배 낮은 P99.99 테일 레이턴시**
> - 동일 워크로드에 **1/3 노드**로 운영 가능
> - **100만+ msg/s** 처리량
>
> 그러나 이 수치는 **특정 조건에서의 결과**이며, 독립 검증에서는 다른 결과가 나오기도 합니다.

---

### Q2. 독립 분석 결과는 어땠나요?

**답변:**
> Confluent의 Jack Vanlightly 분석:
>
> | 발견 | 내용 |
> |------|------|
> | **프로듀서 증가** | 4개 → 50개 시 성능 저하 |
> | **장기 운영** | 12시간+ 시 성능 점진적 저하 |
> | **보존 한계** | 데이터 보존 한계 도달 시 레이턴시 급증 |
> | **결론** | "주장이 **과장됨**, 특정 워크로드에서만 우위" |
>
> **핵심**: 벤치마크 조건에 따라 결과가 크게 달라집니다.

---

### Q3. 실제 성능 특성은 어떤가요?

**답변:**
> **RedPanda가 유리한 경우:**
> - 적은 수의 프로듀서
> - 큰 메시지 배치
> - 저지연이 중요한 워크로드
> - 단기 테스트
>
> **Kafka가 유사하거나 더 나은 경우:**
> - 많은 수의 프로듀서 (50+)
> - 작은 메시지
> - 장기 연속 운영
> - 극한의 처리량 요구

---

### Q4. 처리량/지연시간 실제 수치는?

**답변:**

**처리량 (대략적 수치):**
| 시스템 | 단일 노드 | 클러스터 (3노드) |
|--------|----------|-----------------|
| Kafka | 200-400 MB/s | 600+ MB/s |
| RedPanda | 300-500 MB/s | 600+ MB/s |

**지연시간:**
| 부하 | Kafka P99 | RedPanda P99 |
|------|-----------|--------------|
| 저부하 | ~5ms | ~2-3ms |
| 고부하 | 10-20ms | 5-10ms |

> 저부하에서 RedPanda가 유리하지만, **고부하에서는 차이가 줄어듭니다**.

---

### Q5. 왜 직접 벤치마크가 필요한가요?

**답변:**
> 1. **워크로드 특성**: 메시지 크기, 프로듀서/컨슈머 수, 배치 설정이 다름
> 2. **하드웨어 차이**: 클라우드/온프레미스, 디스크 종류
> 3. **설정 차이**: 튜닝에 따라 성능이 크게 변화
> 4. **버전 차이**: 지속적인 성능 개선
>
> **결론**: 공식 벤치마크나 블로그 수치를 그대로 믿지 말고, **프로덕션과 유사한 환경**에서 테스트하세요.

## 깊은 이해를 위한 설명

### 성능 차이의 원인

**RedPanda가 빠른 이유:**
```
1. GC 없음
   - JVM: Stop-the-world GC → 예측 불가능한 지연 스파이크
   - C++: 수동 메모리 관리 → 일관된 지연시간

2. Thread-per-core
   - 락 경합 없음 → CPU 효율 극대화

3. Zero-copy I/O
   - 불필요한 메모리 복사 제거

4. Raft 복제
   - ISR 동기화 오버헤드 없음
```

**Kafka가 선방하는 이유:**
```
1. 성숙한 최적화
   - 10년+ 프로덕션 경험으로 다양한 최적화

2. 배치 처리 효율
   - 대용량 배치에서 JVM 오버헤드 상쇄

3. ISR 유연성
   - 느린 복제본 제외로 성능 유지
```

### 벤치마크 함정 주의

**공식 벤치마크의 조건:**
```
환경:
- 고성능 인스턴스 (i3en.6xlarge)
- NVMe SSD
- 25Gbps 네트워크
- 적은 프로듀서 (4개)
- 30분 warm-up 후 측정
```

**실제 프로덕션과의 차이:**
```
- 다양한 인스턴스 타입
- 네트워크 제약
- 수십~수백 프로듀서
- 24/7 연속 운영
- 다양한 메시지 크기
```

### 언제 RedPanda 성능이 빛나는가?

**최적 시나리오:**
```
✅ 금융 거래 시스템 (저지연 필수)
✅ 실시간 분석 파이프라인 (일관된 P99)
✅ 소규모~중규모 클러스터 (노드 절약)
✅ 스타트업/새 프로젝트 (운영 단순화)
```

**주의가 필요한 시나리오:**
```
⚠️ 수백 개 프로듀서 (성능 저하 가능)
⚠️ 매우 긴 데이터 보존 (레이턴시 증가)
⚠️ 극한의 처리량 (Kafka와 유사)
⚠️ 대규모 프로덕션 첫 도입 (검증 필요)
```

## 코드 레벨 이해

### 벤치마크 도구

```bash
# OpenMessaging Benchmark (업계 표준)
git clone https://github.com/openmessaging/benchmark
cd benchmark

# RedPanda 벤치마크
bin/benchmark \
  --drivers redpanda.yaml \
  --workloads workload.yaml

# Kafka 벤치마크 (동일 조건)
bin/benchmark \
  --drivers kafka.yaml \
  --workloads workload.yaml
```

### 권장 테스트 시나리오

```yaml
# workload.yaml 예시
scenarios:
  - name: baseline
    producers: 10
    consumers: 10
    messageSize: 1KB
    duration: 30m
    
  - name: high_producer
    producers: 100
    consumers: 10
    messageSize: 1KB
    duration: 30m
    
  - name: long_running
    producers: 10
    consumers: 10
    messageSize: 1KB
    duration: 24h  # 장기 테스트
```

```bash
# RedPanda 자동 튜닝
rpk redpanda tune all

# 수동 설정 예시
rpk redpanda config set \
  --config kafka_batch_max_bytes=1048576

rpk cluster status

```

## 면접 예상 꼬리 질문

### Q. 10배 빠르다는 주장을 어떻게 봐야 하나요?

**답변:**
> **P99.99 (테일 레이턴시)**에서 10배 차이가 날 수 있습니다. 이는 **GC 스파이크** 때문입니다:
>
> - Kafka: 99.99%에서 GC로 인한 스파이크 발생
> - RedPanda: GC 없어서 일관된 지연
>
> 그러나 **P50, P99**에서는 차이가 훨씬 작습니다 (2-5배). 대부분의 애플리케이션은 P99까지만 중요하므로, "10배"는 **마케팅적 표현**에 가깝습니다.

### Q. 노드 1/3로 충분하다는 건 사실인가요?

**답변:**
> **조건부로 사실**입니다.
>
> RedPanda가 더 효율적인 이유:
> - C++ 네이티브 → 낮은 메모리 사용
> - Thread-per-core → 높은 CPU 효율
>
> 그러나:
> - 프로덕션에서는 **버퍼 여유** 필요
> - 장애 대비 **추가 용량** 필요
> - 실제로는 **50-70% 노드 절감**이 현실적
>
> 1/3은 **최적 조건에서의 수치**입니다.

### Q. 성능이 안 좋을 때 어디를 봐야 하나요?

**답변:**
> 1. **프로듀서 수**: 너무 많으면 성능 저하
> 2. **배치 크기**: 너무 작으면 오버헤드 증가
> 3. **복제 팩터**: 높을수록 쓰기 지연 증가
> 4. **디스크 I/O**: NVMe vs HDD 차이 큼
> 5. **네트워크**: 노드 간 지연 확인
> 6. **데이터 보존**: 한계 근처에서 성능 저하
>
> ```bash
> # 모니터링
> rpk cluster health
> curl localhost:9644/metrics | grep -E "(latency|throughput)"
> ```

### Q. Kafka와 RedPanda 중 어떤 것이 더 빠른가요?

**답변 (정석):**
> "**워크로드에 따라 다릅니다.**"
>
> - 저지연, 적은 프로듀서 → **RedPanda** 유리
> - 고처리량, 많은 프로듀서 → **Kafka**와 유사하거나 Kafka 유리
> - 장기 운영 안정성 → **Kafka** (더 많은 검증)
>
> **결론**: 벤치마크 수치보다 **실제 워크로드 테스트**가 중요합니다. 두 시스템 모두 프로덕션급 성능을 제공하며, 선택은 **성능 외 요소**(운영 복잡도, 에코시스템, 라이선스)도 고려해야 합니다.
