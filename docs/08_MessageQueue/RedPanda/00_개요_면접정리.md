# RedPanda 개요 - 면접 정리

## 핵심 한 줄 정리

> **RedPanda는 C++로 작성된 Kafka 호환 스트리밍 플랫폼으로, JVM과 ZooKeeper 없이 단일 바이너리로 동작하며 더 적은 리소스로 유사한 성능을 제공합니다.**

---

## 면접 필수 암기

### Q1. RedPanda란 무엇인가요?

**답변 (30초):**
> RedPanda는 2020년에 출시된 **Kafka API 100% 호환** 스트리밍 플랫폼입니다. C++로 작성되어 **JVM이 필요 없고**, 내장 Raft로 **ZooKeeper도 필요 없습니다**. 기존 Kafka 클라이언트를 코드 변경 없이 그대로 사용할 수 있어서, Kafka 대비 **운영 복잡도를 낮추면서 유사하거나 더 나은 성능**을 목표로 합니다. Schema Registry와 HTTP Proxy도 내장되어 있습니다.

---

### Q2. Kafka 대신 RedPanda를 쓰는 이유는?

**답변:**

| Kafka의 과제 | RedPanda의 해결 |
|-------------|----------------|
| JVM 필요 (GC 지연) | **C++ 네이티브** (GC 없음) |
| ZooKeeper 필요 (4.0 이전) | **내장 Raft** (처음부터 불필요) |
| 복잡한 JVM 튜닝 | **자동 튜닝** (rpk tune) |
| Schema Registry 별도 설치 | **내장** |
| REST Proxy 별도 설치 | **Pandaproxy 내장** |

> 한 마디로: "**Kafka의 기능 + 더 단순한 운영 + 더 적은 리소스**"

---

### Q3. Kafka 호환성은 어느 정도인가요?

**답변:**
> **100% Kafka Protocol 호환**입니다. 다음이 그대로 동작합니다:
> - Kafka 클라이언트 (Java, Python, Go 등)
> - Kafka Connect
> - Kafka Streams
> - ksqlDB
> - MirrorMaker
>
> ```java
> // 코드 변경 없이 주소만 변경
> props.put("bootstrap.servers", "redpanda:9092");
> ```
>
> 단, Kafka의 일부 고급 기능이나 내부 API는 차이가 있을 수 있으므로 프로덕션 전 테스트가 필요합니다.

---

### Q4. RedPanda의 단점이나 주의점은?

**답변:**
> 1. **라이선스**: BSL 1.1로, **경쟁 SaaS 서비스 제공 금지** (4년 후 Apache 2.0 전환)
> 2. **성숙도**: 2020년 출시로 Kafka(2011년) 대비 역사가 짧음
> 3. **에코시스템**: Kafka 에코시스템 대비 커뮤니티 작음
> 4. **벤치마크 주의**: 공식 벤치마크가 과장되었다는 독립 분석 존재
> 5. **장기 운영 검증**: 일부 워크로드에서 12시간+ 운영 시 성능 저하 보고

---

### Q5. RedPanda의 주요 내장 기능은?

**답변:**
> | 기능 | Kafka | RedPanda |
> |------|-------|----------|
> | Schema Registry | Confluent 별도 설치 | **내장** |
> | HTTP Proxy | REST Proxy 별도 | **Pandaproxy 내장** |
> | Admin Console | AKHQ 등 별도 | **Console 내장** |
> | Tiered Storage | Confluent 유료 | **내장** |
>
> RedPanda는 "**all-in-one**" 철학으로, 별도 컴포넌트 설치 없이 단일 바이너리로 모든 기능을 제공합니다.

## 깊은 이해를 위한 설명

### 왜 C++인가?

**JVM의 한계:**
```
Kafka (JVM):
├── GC Stop-the-world → 예측 불가능한 지연 스파이크
├── JVM 힙 메모리 관리 → 복잡한 튜닝 필요
├── Thread Pool 공유 → 락 경합
└── 오버헤드 → 더 많은 리소스 필요
```

**C++/Seastar의 장점:**
```
RedPanda (C++):
├── GC 없음 → 일관된 지연시간
├── 메모리 직접 관리 → 효율적
├── Thread-per-core → 락 경합 제거
└── 네이티브 → 적은 리소스
```

> Seastar는 ScyllaDB(Cassandra C++ 대체)에서 검증된 고성능 프레임워크입니다.

### Kafka와 RedPanda의 메타데이터 관리 비교

```
Kafka 진화:
  ~2024: ZooKeeper (외부 의존성)
  2025~: KRaft (내장 Raft, ZK 제거)

RedPanda:
  2020~: 내장 Raft (처음부터 독립적)
```

> RedPanda는 Kafka가 2025년에 KRaft로 달성한 것을 **2020년부터 이미 제공**했습니다.

### 언제 RedPanda를 선택해야 하나?

**RedPanda가 적합한 경우:**
- 기존 Kafka를 **더 적은 노드로 운영**하고 싶을 때
- **운영 복잡도를 줄이고** 싶을 때 (단일 바이너리, 자동 튜닝)
- **저지연**이 중요할 때 (GC 없음)
- 새 프로젝트에서 Kafka 에코시스템을 쓰되 **단순하게 시작**하고 싶을 때

**Kafka가 더 나은 경우:**
- **대규모 프로덕션 검증**이 필수일 때
- 특수한 Kafka 내부 API에 의존할 때
- **커뮤니티 지원**과 풍부한 문서가 중요할 때
- BSL 라이선스가 **비즈니스에 맞지 않을 때**

## 코드 레벨 이해

### rpk CLI 기본 사용

```bash
# 토픽 생성
rpk topic create orders --partitions 3 --replicas 3

# 토픽 목록
rpk topic list

# 메시지 발행
echo '{"order_id": 123}' | rpk topic produce orders

# 메시지 소비
rpk topic consume orders --from-beginning

# 시스템 튜닝 (자동)
rpk redpanda tune all

# 클러스터 상태
rpk cluster health
```

### Docker로 빠른 시작

```bash
docker run -d --name=redpanda \
  -p 9092:9092 \  # Kafka API
  -p 8081:8081 \  # Schema Registry
  -p 8082:8082 \  # HTTP Proxy
  -p 9644:9644 \  # Admin API
  docker.redpanda.com/redpandadata/redpanda:latest \
  redpanda start --smp 1 --memory 1G
```

### 기존 Kafka 코드 그대로 사용

```java
// 변경 전 (Kafka)
props.put("bootstrap.servers", "kafka:9092");

// 변경 후 (RedPanda) - 주소만 변경!
props.put("bootstrap.servers", "redpanda:9092");

// 나머지 코드 동일
KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.send(new ProducerRecord<>("topic", "key", "value"));
```

## 면접 예상 꼬리 질문

### Q. RedPanda가 Kafka보다 빠르다는 게 사실인가요?

**답변:**
> **조건부로 사실**입니다.
>
> RedPanda 공식 벤치마크에서는 "**10배 낮은 테일 레이턴시, 1/3 노드**"를 주장합니다. 그러나 **Confluent의 독립 분석**에서는:
> - 특정 워크로드에서만 우위
> - 프로듀서 수 증가 시 성능 저하
> - 12시간+ 장기 운영 시 성능 불안정
>
> **결론**: 공식 벤치마크를 그대로 믿지 말고, **실제 워크로드로 직접 테스트**해야 합니다.

### Q. Kafka에서 RedPanda로 마이그레이션하려면?

**답변:**
> 1. **클라이언트 코드 변경 없음**: bootstrap.servers 주소만 변경
> 2. **데이터 마이그레이션**: MirrorMaker2로 토픽 복제
> 3. **설정 검토**: 대부분 호환되지만 일부 차이 확인 필요
> 4. **단계적 전환**: 트래픽 점진적 이동 권장
>
> Kafka API 호환이므로 마이그레이션 난이도는 **낮은 편**입니다.

### Q. BSL 라이선스의 제약은 정확히 무엇인가요?

**답변:**
> **허용:**
> - 개발, 테스트, 프로덕션 사용
> - 소스 코드 수정
> - 내부 서비스에서 메시지 브로커로 사용
>
> **금지:**
> - RedPanda를 기반으로 **경쟁 SaaS 서비스** 제공
> - 예: "Managed RedPanda" 클라우드 서비스 판매
>
> **시간 제한:**
> - 4년 후 Apache 2.0으로 자동 전환
>
> 대부분의 기업 사용에는 **문제없습니다**. AWS MSK 같은 서비스를 직접 만들려는 게 아니라면요.

### Q. RedPanda Cloud와 직접 운영의 차이는?

**답변:**
> **RedPanda Cloud 옵션:**
> - **Dedicated**: 전용 클러스터 (Confluent Cloud 대응)
> - **BYOC**: 사용자 VPC에 배포하고 RedPanda가 관리
> - **Serverless**: 사용량 기반 과금
>
> **직접 운영 대비 장점:**
> - 운영 부담 없음
> - 자동 업그레이드, 모니터링
>
> **직접 운영 장점:**
> - 비용 통제
> - 완전한 제어권
> - 라이선스 제약 없음 (BSL 적용 안 됨)
