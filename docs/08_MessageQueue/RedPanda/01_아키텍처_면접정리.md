# RedPanda 아키텍처 - 면접 정리

## 핵심 한 줄 정리

> **RedPanda는 Thread-per-core 아키텍처와 Seastar 프레임워크를 사용하여 각 CPU 코어가 독립적으로 동작하며, 내장 Raft로 메타데이터와 데이터를 모두 관리합니다.**

---

## 면접 필수 암기

### Q1. Thread-per-core 아키텍처란?

**답변 (30초):**
> 각 CPU 코어에 **전용 스레드**가 할당되어 독립적으로 동작합니다. 코어 간 **메모리 공유가 없어서 락 경합이 발생하지 않습니다**. Kafka의 Thread Pool 모델은 공유 메모리로 인한 락 경합이 있지만, RedPanda는 이를 **구조적으로 제거**했습니다.
>
> ```
> Kafka:   [Thread Pool] ←→ [Shared Memory] ←→ Lock 경합
> RedPanda: [Core0|Mem] [Core1|Mem] [Core2|Mem] → 독립, 락 없음
> ```

---

### Q2. Seastar 프레임워크란?

**답변:**
> Seastar는 ScyllaDB에서 개발한 **고성능 C++ 프레임워크**입니다.
>
> **핵심 특징:**
> - **Thread-per-core**: 코어별 독립 실행
> - **비동기 I/O**: 모든 I/O가 논블로킹
> - **Zero-copy**: 불필요한 메모리 복사 제거
> - **Shard-nothing**: 코어 간 상태 공유 없음
>
> ScyllaDB(Cassandra C++ 대체)에서 **검증된 기술**입니다.

---

### Q3. RedPanda의 Raft 사용 방식은?

**답변:**
> RedPanda는 **모든 분산 합의에 Raft**를 사용합니다:
>
> | 용도 | Kafka | RedPanda |
> |------|-------|----------|
> | 메타데이터 | KRaft (4.0+) | **내장 Raft** |
> | 파티션 복제 | ISR 기반 | **내장 Raft** |
>
> ```
> Partition Raft Group:
> ├── Leader (Node 1)
> ├── Follower (Node 2)
> └── Follower (Node 3)
> ```
>
> Kafka의 ISR(In-Sync Replicas)과 달리, **Quorum 합의**로 더 예측 가능한 동작을 제공합니다.

---

### Q4. RedPanda vs Kafka의 아키텍처 차이 요약?

**답변:**

| 구성요소 | Kafka | RedPanda |
|---------|-------|----------|
| **언어** | Java/Scala | C++ |
| **런타임** | JVM | 네이티브 |
| **스레딩** | Thread Pool (공유) | **Thread-per-core (독립)** |
| **GC** | 있음 (지연 유발) | **없음** |
| **메타데이터** | KRaft | **내장 Raft** |
| **데이터 복제** | ISR | **Raft** |
| **외부 의존성** | 없음 (4.0+) | **처음부터 없음** |

---

### Q5. RedPanda의 포트 구성은?

**답변:**

| 포트 | 용도 |
|------|------|
| **9092** | Kafka API (클라이언트) |
| **8081** | Schema Registry |
| **8082** | HTTP Proxy (Pandaproxy) |
| **9644** | Admin API |
| **33145** | 노드 간 내부 RPC |

> Kafka의 9092와 동일한 포트를 사용하여 **클라이언트 설정 변경 최소화**.

## 깊은 이해를 위한 설명

### Thread-per-core vs Thread Pool

**Kafka (Thread Pool):**
```
┌─────────────────────────────────┐
│         Shared Memory           │
│    ┌───┬───┬───┬───┬───┐       │
│    │T1 │T2 │T3 │T4 │...│       │
│    └─┬─┴─┬─┴─┬─┴─┬─┴───┘       │
│      └───┴───┴───┘              │
│         Lock 경합               │
│         ↓                       │
│   Context Switching 오버헤드     │
└─────────────────────────────────┘
```

**RedPanda (Thread-per-core):**
```
┌─────────────────────────────────┐
│  ┌───────┐ ┌───────┐ ┌───────┐ │
│  │Core 0 │ │Core 1 │ │Core 2 │ │
│  │┌─────┐│ │┌─────┐│ │┌─────┐│ │
│  ││ Mem ││ ││ Mem ││ ││ Mem ││ │
│  │└─────┘│ │└─────┘│ │└─────┘│ │
│  │┌─────┐│ │┌─────┐│ │┌─────┐│ │
│  ││ I/O ││ ││ I/O ││ ││ I/O ││ │
│  │└─────┘│ │└─────┘│ │└─────┘│ │
│  └───────┘ └───────┘ └───────┘ │
│                                 │
│   No Shared State = No Locks   │
└─────────────────────────────────┘
```

> **핵심**: 락 경합이 구조적으로 불가능하여 **일관된 지연시간**을 제공합니다.

### 파티션과 Shard의 관계

```
Topic: orders (3 partitions)

Node 1 (4 cores):
┌────────────────────────────────────┐
│ Core 0: orders-0 shard            │
│ Core 1: orders-1 shard            │
│ Core 2: orders-2 shard (follower) │
│ Core 3: (다른 토픽)                │
└────────────────────────────────────┘

파티션이 특정 코어에 "고정(pinned)"됨
→ CPU affinity로 캐시 효율 극대화
```

### ISR vs Raft 복제

**Kafka ISR:**
```
1. Leader가 메시지 수신
2. Follower들이 복제 (비동기)
3. "동기화된" Follower만 ISR에 포함
4. min.insync.replicas 만족 시 ack
5. ISR 목록이 동적으로 변화 (지연된 복제본 제외)
```

**RedPanda Raft:**
```
1. Leader가 메시지 수신
2. Leader가 모든 Follower에 복제 요청
3. 과반수(Quorum) 확인 시 커밋
4. ISR 같은 동적 목록 없음 (항상 Quorum 필요)
```

> **차이점**: Kafka ISR은 **유연한 성능 조절**, Raft는 **엄격한 일관성**.

## 코드 레벨 이해

### 리소스 설정

```yaml
# redpanda.yaml
redpanda:
  # CPU 코어 수 (thread-per-core)
  smp: 4
  
  # 메모리 할당
  memory: 8G
  
  # 예약 메모리 (OS용)
  reserve_memory: 1G
```

### 자동 튜닝

```bash
# 시스템 자동 최적화
rpk redpanda tune all

# 개별 튜닝
rpk redpanda tune cpu    # CPU governor 설정
rpk redpanda tune net    # 네트워크 버퍼 최적화
rpk redpanda tune disk   # I/O 스케줄러 설정
rpk redpanda tune aio    # 비동기 I/O 설정
```

### 클러스터 상태 확인

```bash
# 클러스터 건강 상태
rpk cluster health

# 파티션 상태
rpk cluster partitions list

# Raft 그룹 상태
rpk cluster status
```

## 면접 예상 꼬리 질문

### Q. Thread-per-core의 단점은 없나요?

**답변:**
> **단점:**
> - 코어 수만큼만 병렬성 확보 (Thread Pool은 더 많은 스레드 가능)
> - 특정 파티션에 부하 집중 시 해당 코어만 포화
> - 코어 간 작업 재분배가 자동으로 안 됨
>
> **해결책:**
> - 파티션을 충분히 분산
> - 핫 파티션 모니터링
> - 필요시 수동 리밸런싱

### Q. RedPanda에서 리더 선출은 어떻게 되나요?

**답변:**
> Raft 프로토콜을 따릅니다:
> 1. Leader의 heartbeat가 timeout 동안 없으면
> 2. Follower가 Candidate로 전환
> 3. 투표 요청 (RequestVote)
> 4. **과반수 득표 시 새 Leader**
> 5. 새 Leader가 heartbeat 시작
>
> Kafka의 Controller 기반 선출보다 **분산되어 있고 빠릅니다**.

### Q. ZooKeeper 없이 어떻게 클러스터 조정을 하나요?

**답변:**
> **Controller Raft Group**이 ZooKeeper 역할을 대신합니다:
> - 토픽/파티션 메타데이터 관리
> - 브로커 멤버십 관리
> - 설정 저장
>
> 이 Controller도 Raft로 복제되어 **고가용성**을 보장합니다. Kafka가 KRaft로 2025년에 달성한 것을 RedPanda는 **2020년부터 제공**했습니다.

### Q. Tiered Storage는 어떻게 동작하나요?

**답변:**
> 오래된 데이터를 **자동으로 S3/GCS로 오프로드**합니다:
>
> ```yaml
> cloud_storage_enabled: true
> cloud_storage_bucket: "my-bucket"
> retention_local_target_bytes: 10GB  # 로컬 보존량
> ```
>
> **동작:**
> 1. 최신 데이터: 로컬 NVMe (빠른 접근)
> 2. 오래된 데이터: S3 (저렴한 저장)
> 3. Consumer 요청 시: 자동으로 S3에서 fetch
>
> Confluent에서는 **유료 기능**이지만, RedPanda는 **내장**.
