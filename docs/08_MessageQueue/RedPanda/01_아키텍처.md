# RedPanda 아키텍처

## 1. 전체 아키텍처 개요

RedPanda는 Kafka와 유사한 분산 로그 시스템이지만, 내부 구현이 완전히 다릅니다.

```
┌─────────────────────────────────────────────────────────────┐
│                    RedPanda Cluster                          │
│                                                             │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐       │
│  │   Node 1    │   │   Node 2    │   │   Node 3    │       │
│  │             │   │             │   │             │       │
│  │ ┌─────────┐ │   │ ┌─────────┐ │   │ ┌─────────┐ │       │
│  │ │ Raft    │ │   │ │ Raft    │ │   │ │ Raft    │ │       │
│  │ │ Groups  │ │   │ │ Groups  │ │   │ │ Groups  │ │       │
│  │ └─────────┘ │   │ └─────────┘ │   │ └─────────┘ │       │
│  │             │   │             │   │             │       │
│  │ ┌─────────┐ │   │ ┌─────────┐ │   │ ┌─────────┐ │       │
│  │ │Partitions│ │   │ │Partitions│ │   │ │Partitions│ │    │
│  │ └─────────┘ │   │ └─────────┘ │   │ └─────────┘ │       │
│  └─────────────┘   └─────────────┘   └─────────────┘       │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐ │
│  │                Internal RPC (Raft)                     │ │
│  └───────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

## 2. Thread-per-Core 아키텍처

### 기존 Kafka 모델

```
Kafka (JVM):
┌─────────────────────────────────┐
│         Shared Memory           │
│    ┌───┬───┬───┬───┬───┐       │
│    │T1 │T2 │T3 │T4 │...│       │
│    └───┴───┴───┴───┴───┘       │
│         ↑   ↑   ↑              │
│         └───┴───┘              │
│        Lock Contention         │
└─────────────────────────────────┘
```

### RedPanda 모델 (Seastar)

```
RedPanda (C++/Seastar):
┌─────────────────────────────────┐
│  ┌───────┐ ┌───────┐ ┌───────┐ │
│  │Core 0 │ │Core 1 │ │Core 2 │ │
│  │ ┌───┐ │ │ ┌───┐ │ │ ┌───┐ │ │
│  │ │Mem│ │ │ │Mem│ │ │ │Mem│ │ │
│  │ └───┘ │ │ └───┘ │ │ └───┘ │ │
│  │ ┌───┐ │ │ ┌───┐ │ │ ┌───┐ │ │
│  │ │I/O│ │ │ │I/O│ │ │ │I/O│ │ │
│  │ └───┘ │ │ └───┘ │ │ └───┘ │ │
│  └───────┘ └───────┘ └───────┘ │
│                                 │
│  No Shared State = No Locks    │
└─────────────────────────────────┘
```

### 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **Shard-Nothing** | 각 코어가 독립적인 메모리/I/O 관리 |
| **No Locks** | 코어 간 공유 상태 없음 → 락 경합 제거 |
| **CPU Affinity** | 파티션이 특정 코어에 고정 |
| **Async I/O** | 모든 I/O가 비동기 |

## 3. Raft 합의 프로토콜

### 메타데이터와 데이터 모두 Raft 사용

```
Kafka:
- 메타데이터: ZooKeeper (과거) / KRaft (현재)
- 데이터: ISR 기반 복제

RedPanda:
- 메타데이터: 내장 Raft (controller)
- 데이터: 내장 Raft (각 파티션)
```

### Raft Group 구조

```
Topic: orders (3 partitions, RF=3)

Partition 0 Raft Group:
├── Leader: Node 1
├── Follower: Node 2
└── Follower: Node 3

Partition 1 Raft Group:
├── Leader: Node 2
├── Follower: Node 1
└── Follower: Node 3

Partition 2 Raft Group:
├── Leader: Node 3
├── Follower: Node 1
└── Follower: Node 2
```

### Controller Raft Group

클러스터 메타데이터를 관리하는 특수 Raft 그룹:

```
Controller Partition:
├── 토픽 구성
├── 파티션 할당
├── ACL
├── 설정
└── Consumer Group 오프셋
```

## 4. 스토리지 아키텍처

### 로그 구조

```
/var/lib/redpanda/data/
└── kafka/
    └── orders/
        ├── 0/                    # Partition 0
        │   ├── 0-1-v1.log       # Segment 파일
        │   ├── 0-1-v1.base_index
        │   └── ...
        ├── 1/                    # Partition 1
        └── 2/                    # Partition 2
```

### 세그먼트 관리

```
Segment Lifecycle:
Active Segment → Closed Segment → Compacted/Deleted

┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│   Active     │   │   Closed     │   │   Closed     │
│   Segment    │   │   Segment    │   │   Segment    │
│   (쓰기중)    │   │   (읽기전용)  │   │   (삭제대기) │
└──────────────┘   └──────────────┘   └──────────────┘
```

### Tiered Storage

```
┌─────────────────────────────────────────────┐
│              Hot Data (Local)               │
│     최신 데이터, 빠른 접근                    │
├─────────────────────────────────────────────┤
│              Cold Data (S3/GCS)             │
│     오래된 데이터, 저비용 저장                │
└─────────────────────────────────────────────┘

설정:
cloud_storage_enabled: true
retention_local_target_bytes: 10GB  # 로컬 보존
```

## 6. 메모리 관리

### JVM vs RedPanda

| 항목 | Kafka (JVM) | RedPanda (C++) |
|------|-------------|----------------|
| GC | Stop-the-world GC 발생 | GC 없음 |
| 힙 사이즈 | 튜닝 필요 | 해당 없음 |
| 메모리 할당 | JVM 관리 | Seastar 관리 |
| Page Cache | OS 의존 | 직접 관리 |

### 메모리 구성

```bash
# 권장 설정
rpk redpanda config set --config rpk.tune_aio_events true
rpk redpanda config set --config rpk.tune_coredump true

# 메모리 할당 (예: 16GB 중 12GB를 RedPanda에)
redpanda start --memory 12G
```

## 7. 복제 메커니즘

### ISR vs Raft

```
Kafka ISR (In-Sync Replicas):
- Leader가 ISR 목록 관리
- 동기화 지연 시 ISR에서 제거
- min.insync.replicas로 내구성 제어

RedPanda Raft:
- Quorum 기반 합의
- 과반수 확인 시 커밋
- 더 예측 가능한 동작
```

### 쓰기 경로

```
Producer
    │
    ▼
┌─────────┐
│ Leader  │ ──────► WAL 기록
└────┬────┘
     │
     ├──────────────┬──────────────┐
     ▼              ▼              ▼
┌─────────┐   ┌─────────┐   ┌─────────┐
│Follower1│   │Follower2│   │Follower3│
└────┬────┘   └────┬────┘   └────┬────┘
     │              │              │
     └──────────────┴──────────────┘
                    │
              Quorum ACK
                    │
                    ▼
            Leader Commit
                    │
                    ▼
            Producer ACK
```

## 8. 리밸런싱

### 파티션 이동

```bash
# 파티션 재할당
rpk cluster partitions move <topic> <partition> \
  --to <node-id>

# 자동 리밸런싱
rpk cluster partitions balance
```

### 리더 선출

Raft 기반 자동 리더 선출:

```
Node 1 (Leader) 장애
        │
        ▼
Node 2, Node 3 election timeout
        │
        ▼
새 Leader 선출 (더 높은 term)
        │
        ▼
클라이언트 자동 재연결
```

## 10. Kafka 대비 아키텍처 차이 요약

| 구성요소 | Kafka | RedPanda |
|---------|-------|----------|
| **언어** | Java/Scala | C++ |
| **런타임** | JVM | Native |
| **스레딩** | Thread Pool | Thread-per-core |
| **메타데이터** | KRaft/ZK | 내장 Raft |
| **데이터 복제** | ISR | Raft |
| **Schema Registry** | 별도 컴포넌트 | 내장 |
| **HTTP Proxy** | REST Proxy (별도) | Pandaproxy (내장) |
| **설정** | 많은 튜닝 필요 | 자동 튜닝 |
