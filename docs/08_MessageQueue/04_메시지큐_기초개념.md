# 메시지 큐 기초 개념

## 1. 메시지 큐란?

### 1.1 정의

메시지 큐(Message Queue)는 **애플리케이션 간 비동기 통신**을 가능하게 하는 아키텍처 기법이다.

```
┌──────────┐     ┌─────────────┐     ┌──────────┐
│ Producer │ ──→ │ Message Queue │ ──→ │ Consumer │
└──────────┘     └─────────────┘     └──────────┘
    보내는 쪽         택배함           받는 쪽
```

### 1.2 동기 vs 비동기 통신

| 방식 | 설명 | 비유 |
|------|------|------|
| **동기 (Synchronous)** | 보내는 순간 받아야 함 | 무전기 |
| **비동기 (Asynchronous)** | 나중에 받아도 됨 | 택배함/이메일 |

**동기 통신의 문제점**:
- 수신자가 꺼져 있으면 데이터 유실
- 수신자 과부하 시 전송 실패
- 보내는 쪽이 받는 쪽 상태를 신경 써야 함

---

## 2. 핵심 구성요소

### 2.1 Producer (생산자)

- 메시지를 **보내는** 애플리케이션
- 메시지를 큐에 넣고 자기 일에 집중
- Consumer 상태를 **신경 쓸 필요 없음**

### 2.2 Consumer (소비자)

- 메시지를 **받는** 애플리케이션
- 가능할 때 큐에서 메시지를 가져감
- 필요한 메시지만 **선택적으로 소비**

### 2.3 Broker (브로커)

- Producer와 Consumer 사이의 **중개자**
- 메시지 **저장 및 전달** 담당
- 지정된 시점까지 메시지 **보관**

---

## 3. 메시지 전달 패턴

### 3.1 Point-to-Point (일대일)

```
Producer ─── [Queue] ─── Consumer
```

- 하나의 메시지는 **하나의 Consumer**만 소비
- 웹폼 제출 → 처리 시스템
- 주문 → 결제 처리

### 3.2 Publish-Subscribe (발행-구독)

```
              ┌── Consumer A
Producer ─── [Topic] ─── Consumer B
              └── Consumer C
```

- 하나의 메시지를 **여러 Consumer**가 소비
- 새 주문 → 재고, 배송, 알림, 정산 모두 수신
- SNS 포스트 → 피드, 알림, 통계, 로그 모두 수신

### 3.3 Request-Reply (요청-응답)

```
Producer ─── [Request Queue] ─── Consumer
    ↑                                │
    └───── [Reply Queue] ←──────────┘
```

- 요청 보내고 **별도 큐로 응답** 수신
- 검색 요청 → 결과 취합 → 응답

## 4. 메시지 큐의 장점

### 4.1 디커플링 (Decoupling)

**문제 상황**:
```
채팅 서버 → DB 저장
         → 메시지 전달
         → 푸시 알림
         → 이메일 발송

(모든 의존성이 채팅 서버에 집중)
```

**메시지 큐 도입 후**:
```
채팅 서버 → [Message Queue] → DB 서비스
                           → 전달 서비스
                           → 푸시 서비스
                           → 이메일 서비스

(채팅 서버는 MQ만 알면 됨)
```

### 4.2 확장성 (Scalability)

- Producer/Consumer를 **독립적으로 확장**
- 트래픽 급증 시 Consumer만 늘리면 됨
- 각 서비스를 **다른 언어/기술**로 구현 가능

### 4.3 안정성 (Reliability)

- 메시지 **유실 방지** (브로커가 보관)
- Consumer 장애 시에도 **메시지 보존**
- 장애 복구 후 **재처리 가능**

### 4.4 유연성 (Flexibility)

- 받는 쪽 추가/삭제가 보내는 쪽에 **영향 없음**
- 시스템 구조 변경이 **용이**
- 비즈니스 로직 **분리**

---

## 5. 적합한 사용 사례

### 5.1 메시지 큐가 유용한 경우

| 사용 사례 | 설명 |
|----------|------|
| **이메일 발송** | 즉시 응답 불필요, 비동기 처리 |
| **로그 수집** | 대량 데이터 안정적 수집 |
| **온라인 결제** | 안전한 메시지 전달 필요 |
| **이미지 처리** | 시간 걸리는 작업 분산 |
| **마이크로서비스** | 서비스 간 독립성 확보 |
| **알림 시스템** | 다수 대상 비동기 전달 |

### 5.2 메시지 큐가 부적합한 경우

| 사용 사례 | 이유 |
|----------|------|
| **단순한 시스템** | 오버헤드만 증가 |
| **즉각 응답 필요** | 비동기 특성상 지연 발생 |
| **강한 일관성 필요** | 트랜잭션 처리 복잡 |

## 6. 이메일 비유로 이해하기

```
┌─────────────────────────────────────────────────────┐
│                   이메일 = 메시지 큐                  │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ✉️ 이메일 작성/발송    =    Producer 메시지 전송     │
│  📥 받은편지함          =    Message Queue           │
│  👀 메일 확인           =    Consumer 메시지 소비     │
│                                                     │
│  특징:                                              │
│  - 보내고 바로 다른 일 가능 (비동기)                  │
│  - 받는 사람이 나중에 확인해도 됨                     │
│  - 메일이 쌓여있어도 유실 안됨                        │
│  - 언제 답장 받을지 걱정 안해도 됨                    │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 7. 마이크로서비스에서의 활용

### 7.1 복잡한 의존성 문제

SNS 서비스 예시:
```
사용자 포스트 서비스 → 피드 갱신 서비스
                   → 알림 서비스
                   → 통계 서비스
                   → 로그 서비스

팔로우 서비스 ────→ (동일한 의존성들)
메시징 서비스 ────→ (동일한 의존성들)
프로필 서비스 ────→ (동일한 의존성들)
```

### 7.2 메시지 큐 도입 효과

```
모든 서비스 → [Message Queue] → 관심 있는 서비스가 구독

- 아키텍처 단순화
- 지역적 장애에 강함
- 수평적 확장 용이
```

## 8. 면접 포인트

### Q: 메시지 큐를 사용하는 이유는?

**핵심 답변**:
> "메시지 큐는 비동기 통신을 통해 시스템을 디커플링하고, 데이터 유실을 방지하며, 수평적 확장을 용이하게 합니다."

### Q: 동기 통신 대신 메시지 큐를 선택해야 하는 상황은?

**답변**:
> "즉각적인 응답보다 안전한 메시지 전달이 중요하거나, 다수의 서비스에 데이터를 전달해야 하거나, 시스템 간 독립성과 확장성이 필요한 경우 메시지 큐를 선택합니다."

### Q: Producer-Consumer 패턴의 장점은?

**답변**:
> "Producer는 Consumer 상태를 신경 쓰지 않고 메시지를 전송할 수 있고, Consumer는 자신의 처리 속도에 맞춰 메시지를 소비할 수 있습니다. 이를 통해 시스템 간 결합도를 낮추고 독립적인 확장이 가능합니다."

---

## 9. 참고

- **메시지(Message)**: 시스템 간 전달되는 데이터 (JSON, 파일, 메타데이터 등)
- **큐(Queue)**: 메시지가 순서대로 저장되는 구조 (FIFO)
- **토픽(Topic)**: Kafka 등에서 메시지를 그룹화하는 단위
