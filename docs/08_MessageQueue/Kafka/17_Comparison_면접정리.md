## 1. Data Inside vs Data Outside (Pat Helland)

### 1.1 개념 소개

Pat Helland의 논문 "Data on the Outside vs. Data on the Inside"(2005)는 분산 시스템에서 데이터를 이해하는 핵심 프레임워크를 제공합니다. 이 개념은 Kafka의 역할을 이해하는 데 중요한 기반이 됩니다.

**Data Inside vs Data Outside**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Data Inside vs Data Outside                   │
│                                                                  │
│   Data Inside (내부 데이터)            Data Outside (외부 데이터)│
│   ┌───────────────────────┐          ┌───────────────────────┐  │
│   │                       │          │                       │  │
│   │  범위: 단일 서비스    │          │  범위: 서비스 경계    │  │
│   │        내부           │          │        밖 (통신)      │  │
│   │                       │          │                       │  │
│   │  변경: 자유롭게       │          │  변경: 불변           │  │
│   │        수정 가능      │          │        (Immutable)    │  │
│   │                       │          │                       │  │
│   │  시간: 현재 상태만    │          │  시간: 히스토리가     │  │
│   │        중요           │          │        중요           │  │
│   │                       │          │                       │  │
│   │  계약: 내부 구현      │          │  계약: 인터페이스     │  │
│   │        자유           │          │        엄격           │  │
│   │                       │          │                       │  │
│   │  예시:                │          │  예시:                │  │
│   │  - RDBMS 테이블      │          │  - 메시지/이벤트     │  │
│   │  - 내부 캐시         │          │  - API 응답          │  │
│   │  - 로컬 변수         │          │  - 로그 레코드       │  │
│   │                       │          │                       │  │
│   └───────────────────────┘          └───────────────────────┘  │
│              │                                   │               │
│              ▼                                   ▼               │
│   ┌───────────────────────┐          ┌───────────────────────┐  │
│   │    "개인 노트"        │          │    "공식 계약서"      │  │
│   │    언제든 수정 가능   │          │    서명 후 변경 불가  │  │
│   │    최신 버전만 필요   │          │    모든 버전 기록     │  │
│   └───────────────────────┘          └───────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 특성 비교

| 특성 | Data Inside | Data Outside |
|------|-------------|--------------|
| **범위** | 단일 서비스 내부 | 서비스 간 통신 |
| **변경성** | 자유롭게 변경 가능 (Mutable) | 불변 (Immutable) |
| **시간 관점** | 현재 상태 중심 | 이벤트 히스토리 중요 |
| **스키마** | 내부 구현 자유 | 계약/인터페이스 엄격 |
| **트랜잭션** | ACID 보장 가능 | 결국적 일관성 (Eventual Consistency) |
| **질의 방식** | 복잡한 쿼리 가능 | 순차적 읽기 |

### 1.3 Kafka의 위치

Kafka는 **Data Outside**를 다루는 시스템입니다.

**Kafka가 Data Outside인 이유**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Kafka as Data Outside                         │
│                                                                  │
│   Service A                 Kafka                   Service B    │
│   ┌─────────┐              ┌─────────┐              ┌─────────┐ │
│   │         │              │ 불변의  │              │         │ │
│   │ Data    │──(이벤트)───▶│ 이벤트  │──(이벤트)───▶│ Data    │ │
│   │ Inside  │              │ 로그    │              │ Inside  │ │
│   │         │              │         │              │         │ │
│   │ (DB)    │              │ (토픽)  │              │ (DB)    │ │
│   └─────────┘              └─────────┘              └─────────┘ │
│                                                                  │
│   Kafka의 특성:                                                  │
│   1. 불변 (Append-only): 한번 쓰면 변경 불가                     │
│   2. 히스토리 유지: 설정된 보존 기간 동안 모든 이벤트 유지       │
│   3. 계약 기반: Schema Registry로 스키마 관리                    │
│   4. 서비스 간 통신: 표준화된 데이터 교환 방법                   │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 전통적 메시징 시스템 vs Kafka

### 2.1 근본적 차이

전통적 메시징 시스템(ActiveMQ, RabbitMQ 등)과 Kafka는 설계 철학부터 다릅니다.

**아키텍처 비교**:

```
┌─────────────────────────────────────────────────────────────────┐
│              Traditional Messaging vs Kafka                      │
│                                                                  │
│   Traditional Messaging (RabbitMQ, ActiveMQ)                    │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   Producer ──▶ [ Queue ] ──▶ Consumer                   │   │
│   │                   │                                     │   │
│   │                   └─▶ 소비 후 삭제                      │   │
│   │                                                         │   │
│   │   특징:                                                 │   │
│   │   - 메시지 큐 (FIFO)                                   │   │
│   │   - 소비 후 삭제 (한번만 읽기)                         │   │
│   │   - ACK 기반 전달 보장                                  │   │
│   │   - 복잡한 라우팅 (Exchange, Binding)                  │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│   Apache Kafka                                                   │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   Producer ──▶ [     Log     ] ──▶ Consumer A          │   │
│   │                 │    │    │                             │   │
│   │                 ▼    ▼    ▼                             │   │
│   │              offset 0, 1, 2, ... ──▶ Consumer B        │   │
│   │                                                         │   │
│   │   특징:                                                 │   │
│   │   - 분산 로그 (Append-only)                            │   │
│   │   - 보존 기간까지 유지 (여러 번 읽기 가능)             │   │
│   │   - 오프셋 기반 소비                                    │   │
│   │   - 파티션 기반 병렬 처리                               │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 메시지 보존

**소비 후 메시지 처리 비교**:

| 구분 | 전통적 메시징 | Kafka |
|------|--------------|-------|
| **메시지 보존** | 소비 후 즉시 삭제 | 설정된 기간 동안 보존 |
| **재소비** | 불가능 (메시지 사라짐) | 오프셋 리셋으로 언제든 가능 |
| **멱등 처리** | 어려움 (재처리 불가) | 재처리로 쉽게 구현 |
| **감사 로그** | 별도 구현 필요 | 기본 제공 (로그 자체) |
| **디버깅** | 메시지 확인 어려움 | 언제든 메시지 조회 가능 |

**시나리오: 장애 후 재처리**:

```
전통적 메시징 (RabbitMQ):
┌────────────────────────────────────────────────────┐
│ Consumer가 메시지 3 처리 중 장애 발생               │
│                                                    │
│ Queue: [msg1] [msg2] [msg3] [msg4]                 │
│                        ↑                           │
│                    처리 중 장애                    │
│                                                    │
│ ACK 전 장애 → 메시지 3 재전달 (한 번만)            │
│ ACK 후 장애 → 메시지 3 손실 (처리 안 됨)           │
│                                                    │
│ 문제: 메시지 1, 2는 이미 삭제되어 재처리 불가      │
└────────────────────────────────────────────────────┘

Kafka:
┌────────────────────────────────────────────────────┐
│ Consumer가 offset 2 처리 중 장애 발생              │
│                                                    │
│ Log: [0] [1] [2] [3] [4] ...                       │
│           ↑                                        │
│       offset 2                                     │
│                                                    │
│ 장애 복구 후:                                       │
│ - 마지막 커밋된 offset(1)부터 재시작               │
│ - 또는 offset 0부터 전체 재처리 가능               │
│                                                    │
│ 모든 메시지가 보존되어 있어 언제든 재처리 가능     │
└────────────────────────────────────────────────────┘
```

### 2.3 소비 모델

**전통적 메시징의 소비 모델**:

```
Producer ──▶ Queue ──▶ Consumer (단일 소비자만 읽음)
                  │
                  └─▶ 다른 Consumer는 메시지를 받을 수 없음
```

**Kafka의 소비 모델**:

```
                        ┌──▶ Consumer Group A (Consumer 1, 2)
                        │    - 파티션별 병렬 처리
                        │
Producer ──▶ Topic ─────┼──▶ Consumer Group B (Consumer 3)
             (Log)      │    - 독립적으로 전체 메시지 소비
                        │
                        └──▶ Consumer Group C (Consumer 4, 5, 6)
                             - 또 다른 독립적 처리

모든 Consumer Group이 같은 메시지를 각자의 속도로 소비
```

### 2.4 확장성과 순서 보장

| 특성 | 전통적 메시징 | Kafka |
|------|--------------|-------|
| **수평 확장** | 제한적 (클러스터링 복잡) | 파티션으로 무한 확장 |
| **순서 보장** | 큐 전체에서 보장 | 파티션 내에서만 보장 |
| **컨슈머 그룹** | 제한적 지원 | 네이티브 지원 |
| **백프레셔** | ACK 기반 흐름 제어 | 컨슈머가 속도 제어 (Pull 모델) |
| **처리량** | 중간 (수만 msg/s) | 높음 (수백만 msg/s) |

### 2.5 라우팅

**전통적 메시징의 라우팅 (RabbitMQ 예시)**:

```
Exchange Types:
├─ Direct: routing key 일치 시 전달
├─ Topic: pattern matching으로 전달
├─ Fanout: 모든 바인딩된 큐에 전달
└─ Headers: 헤더 속성으로 라우팅
```

**Kafka의 라우팅**:

```
단순한 토픽 기반:
├─ Producer: 특정 토픽에 발행
├─ Consumer: 특정 토픽 구독
└─ 복잡한 라우팅이 필요하면 Kafka Streams로 처리
```

Kafka는 복잡한 브로커 라우팅 대신 **클라이언트 측 처리**를 권장합니다.

## 3. REST API vs Kafka

### 3.1 통신 패턴 비교

REST API와 Kafka는 서로 다른 문제를 해결합니다. 둘 중 하나가 우월한 것이 아니라 **적합한 상황이 다릅니다**.

**통신 패턴 비교**:

```
┌─────────────────────────────────────────────────────────────────┐
│              REST (Synchronous) vs Kafka (Asynchronous)          │
│                                                                  │
│   REST API (동기식):                                             │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   Client ──(요청)──▶ Server                             │   │
│   │      │                  │                               │   │
│   │      │    대기 중...    │                               │   │
│   │      │                  │                               │   │
│   │      ◀──(응답)─────────┘                                │   │
│   │                                                         │   │
│   │   특징: 요청-응답 완료까지 블로킹                       │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│   Kafka (비동기식):                                              │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   Producer ──(발행)──▶ Kafka Topic                      │   │
│   │      │                      │                           │   │
│   │      ▼                      ▼                           │   │
│   │   계속 진행             Consumer가 나중에 처리          │   │
│   │                                                         │   │
│   │   특징: 발행 후 즉시 진행, 처리는 비동기적으로          │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 특성 비교

| 특성 | REST API | Kafka |
|------|----------|-------|
| **통신 방식** | 동기 (Synchronous) | 비동기 (Asynchronous) |
| **결합도** | 강결합 (호출자↔피호출자) | 약결합 (발행↔구독) |
| **장애 전파** | 즉시 전파 (연쇄 장애) | 버퍼링으로 격리 |
| **스케일링** | 요청 증가 시 부하 증가 | 토픽이 버퍼 역할 |
| **재시도** | 클라이언트가 구현 | 자동 재처리 가능 |
| **디스커버리** | 서비스 레지스트리 필요 | 토픽 이름만 알면 됨 |
| **응답 시간** | 즉각적 | 처리 후 별도 확인 필요 |

### 3.3 사용 시나리오

**REST가 적합한 경우**:

```
✅ REST 선택:

1. 즉각적 응답 필요
   └─ 예: 계좌 잔액 조회, 사용자 프로필 조회

2. 단순 CRUD 작업
   └─ 예: 리소스 생성/조회/수정/삭제

3. 동기적 트랜잭션
   └─ 예: 결제 승인 결과 즉시 확인

4. 요청-응답 패턴
   └─ 예: 검색 쿼리 → 결과 반환
```

**Kafka가 적합한 경우**:

```
✅ Kafka 선택:

1. 이벤트 기반 아키텍처
   └─ 예: 주문 생성 → 재고 감소 → 배송 요청 → 알림 발송

2. 다수의 소비자 (Fan-out)
   └─ 예: 하나의 이벤트를 분석, 알림, 감사 등 여러 시스템이 처리

3. 순서 보장이 중요한 처리
   └─ 예: 거래 내역, 상태 변경 히스토리

4. 시스템 간 디커플링
   └─ 예: 마이크로서비스 간 통신, 장애 격리

5. 비동기 배치 처리
   └─ 예: 로그 집계, 메트릭 수집, ETL
```

### 3.4 혼합 사용 패턴

실제로는 REST와 Kafka를 **함께 사용**하는 경우가 많습니다.

```
┌─────────────────────────────────────────────────────────────────┐
│              REST + Kafka 혼합 패턴                              │
│                                                                  │
│   사용자 요청 흐름:                                              │
│                                                                  │
│   Mobile App                                                     │
│       │                                                          │
│       ▼                                                          │
│   ┌─────────┐    REST     ┌─────────┐                           │
│   │   API   │◀───────────▶│  Order  │                           │
│   │ Gateway │             │ Service │                           │
│   └─────────┘             └────┬────┘                           │
│                                │                                 │
│                                │ Kafka                           │
│                                ▼                                 │
│                         ┌───────────┐                           │
│                         │   Topic   │                           │
│                         │ (orders)  │                           │
│                         └─────┬─────┘                           │
│                               │                                  │
│           ┌───────────────────┼───────────────────┐             │
│           ▼                   ▼                   ▼             │
│   ┌─────────────┐     ┌─────────────┐     ┌─────────────┐       │
│   │  Inventory  │     │   Payment   │     │ Notification│       │
│   │   Service   │     │   Service   │     │   Service   │       │
│   └─────────────┘     └─────────────┘     └─────────────┘       │
│                                                                  │
│   REST: 사용자 요청/응답                                         │
│   Kafka: 서비스 간 이벤트 전파                                   │
└─────────────────────────────────────────────────────────────────┘
```

## 4. 관계형 데이터베이스 vs Kafka

### 4.1 역할의 차이

RDBMS와 Kafka는 **보완적인 관계**이지, 대체 관계가 아닙니다.

**역할 비교**:

```
┌─────────────────────────────────────────────────────────────────┐
│              RDBMS vs Kafka: 역할의 차이                         │
│                                                                  │
│   RDBMS (관계형 데이터베이스)                                    │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   "사진첩"                                              │   │
│   │   - 최신 상태를 저장                                    │   │
│   │   - 언제든 수정/삭제 가능                               │   │
│   │   - 복잡한 쿼리로 조회                                  │   │
│   │                                                         │   │
│   │   users 테이블:                                         │   │
│   │   | id | name  | balance |                              │   │
│   │   | 1  | Alice | 1000    | ← 현재 잔액 (과거 무시)     │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│   Kafka                                                          │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   "비디오 녹화"                                         │   │
│   │   - 모든 변경을 기록                                    │   │
│   │   - 한번 쓰면 변경 불가                                 │   │
│   │   - 순차적으로 읽기                                     │   │
│   │                                                         │   │
│   │   balance_changes 토픽:                                 │   │
```

### 4.2 데이터 모델 비교

| 특성 | 관계형 DB | Kafka |
|------|----------|-------|
| **데이터 모델** | 테이블, 행, 열 | 토픽, 파티션, 레코드 |

| **데이터 보존** | 무기한 | 보존 정책에 따름 |
| **인덱싱** | B-tree, Hash 등 | 오프셋 인덱스만 |
| **스키마** | 엄격한 스키마 강제 | 유연한 스키마 (Schema Registry 권장) |

## 면접 예상 질문

### Q1: Kafka와 RabbitMQ의 가장 큰 차이점은?

**모범 답변**:

가장 큰 차이점은 **메시지 보존 정책**입니다.

**RabbitMQ (전통적 메시징)**:
- 소비자가 메시지를 읽고 ACK하면 **즉시 삭제**
- 한번 소비된 메시지는 다시 읽을 수 없음
- 재처리가 필요하면 Producer가 다시 보내야 함

**Kafka**:
- 소비 여부와 관계없이 **설정된 보존 기간 동안 유지**
- 여러 Consumer Group이 같은 메시지를 독립적으로 소비 가능
- 오프셋 리셋으로 언제든 재처리 가능

**이로 인한 차이점**:
1. **재처리**: Kafka는 장애 복구나 버그 수정 후 재처리가 쉬움
2. **다수 소비자**: Kafka는 하나의 메시지를 여러 시스템이 독립적으로 소비
3. **감사 로그**: Kafka는 기본적으로 모든 이벤트 히스토리 유지
4. **디버깅**: Kafka는 과거 메시지를 조회하여 문제 분석 가능

**선택 기준**:
- 빠른 메시지 처리, 복잡한 라우팅 → RabbitMQ
- 이벤트 저장, 재처리, 다수 소비자 → Kafka

### Q2: REST 대신 Kafka를 사용해야 하는 경우는?

**모범 답변**:

Kafka를 REST 대신 사용해야 하는 주요 경우:

1. **여러 서비스가 같은 이벤트 필요** (Fan-out):
   - 주문 생성 이벤트를 재고, 결제, 알림, 분석 시스템이 모두 처리
   - REST: 각 서비스에 개별 호출 필요
   - Kafka: 한번 발행으로 모든 서비스가 구독

2. **순서 보장이 중요**:
   - 거래 내역, 상태 변경 히스토리
   - REST: 순서 보장 어려움
   - Kafka: 파티션 내 순서 보장

3. **장애 시 버퍼링 필요**:
   - 대상 서비스 장애 시 메시지 손실 방지
   - REST: 클라이언트가 재시도 로직 구현
   - Kafka: 토픽이 버퍼 역할, 복구 후 자동 처리

4. **비동기 처리가 적합한 경우**:
   - 로그 수집, 메트릭 전송, 배치 처리
   - REST: 응답 대기로 지연
   - Kafka: 발행 후 즉시 진행

5. **시스템 간 디커플링**:
   - 서비스 간 직접 의존성 제거
   - REST: A→B 강결합
   - Kafka: A→Topic←B 약결합

**주의**: 즉각적 응답이 필요한 경우(잔액 조회 등)는 여전히 REST가 적합합니다.

### Q3: Data Inside vs Outside의 개념을 설명하세요

**모범 답변**:

Pat Helland의 개념으로, 분산 시스템에서 데이터의 두 가지 유형을 구분합니다.

**Data Inside (내부 데이터)**:
- **범위**: 단일 서비스 내부
- **변경성**: 자유롭게 수정 가능 (Mutable)
- **시간 관점**: 현재 상태만 중요
- **스키마**: 내부 구현, 언제든 변경 가능
- **예시**: 서비스의 내부 DB, 로컬 캐시, 변수

**Data Outside (외부 데이터)**:
- **범위**: 서비스 경계 밖, 서비스 간 통신
- **변경성**: 불변 (Immutable), 한번 전송되면 변경 불가
- **시간 관점**: 이벤트 히스토리가 중요
- **스키마**: 계약/인터페이스, 함부로 변경 불가
- **예시**: 메시지, 이벤트, API 응답

**Kafka의 위치**:
Kafka는 **Data Outside**를 다루는 시스템입니다.
- 불변의 이벤트 로그 저장 (Append-only)
- 서비스 간 데이터 교환의 표준화된 방법 제공
- Schema Registry로 계약 관리

**비유**:
- Data Inside = 개인 노트 (언제든 수정, 최신 버전만 필요)
- Data Outside = 공식 계약서 (서명 후 변경 불가, 모든 버전 기록)

---

### Q4: Event Sourcing에서 Kafka의 역할은?

**모범 답변**:

Kafka는 Event Sourcing 패턴에서 **Event Store** 역할을 합니다.

**Event Store로서의 Kafka**:

1. **모든 상태 변경을 이벤트로 저장**:
   - 현재 상태가 아닌 상태 변경 이벤트를 저장
   - 예: `OrderCreated`, `ItemAdded`, `OrderConfirmed`

2. **불변 로그**:
   - 이벤트는 한번 쓰면 변경/삭제 불가 (Append-only)
   - 완전한 감사 로그(Audit Trail) 역할

3. **재생(Replay) 가능**:
   - 이벤트를 처음부터 재생하여 현재 상태 복원
   - 새로운 뷰(Read Model) 생성 가능
   - 버그 수정 후 재처리

4. **시점 복구**:
   - 특정 시점까지의 이벤트만 재생하여 과거 상태 복원
   - Point-in-time Recovery

**Event Sourcing의 장점**:
- 완전한 히스토리 유지
- 디버깅 용이 (과거 상태 재현)
- 새로운 분석 요구사항에 유연하게 대응

**주의사항**:
- Log Compaction 사용 시 주의 (중간 이벤트 삭제 가능)
- 대용량 이벤트 시 스냅샷 전략 필요

---

### Q5: CDC가 필요한 이유는?

**모범 답변**:

**CDC (Change Data Capture)**가 필요한 핵심 이유는 **이중 쓰기(Dual Write) 문제를 해결**하기 위함입니다.

**이중 쓰기의 문제**:
```
애플리케이션이 DB와 Kafka에 모두 쓰는 경우:
1. DB에 INSERT (성공)
2. Kafka에 PRODUCE (실패 - 네트워크 오류)

결과: DB에는 있지만 Kafka에는 없음 → 데이터 불일치
```

**CDC의 해결책**:
- 애플리케이션은 DB에만 쓰기
- Debezium(CDC 도구)이 DB의 WAL을 감시
- 변경사항을 자동으로 Kafka에 발행

**CDC의 장점**:

1. **데이터 일관성 보장**:
   - DB 트랜잭션이 커밋되면 확실히 Kafka로 전달
   - 원자적(Atomic) 연산

2. **기존 애플리케이션 수정 불필요**:
   - 애플리케이션 코드 변경 없이 이벤트 발행
   - 레거시 시스템에 이벤트 기반 아키텍처 적용

3. **실시간 데이터 동기화**:
   - DB 변경이 밀리초 단위로 Kafka에 반영
   - 캐시, 검색 엔진, 분석 시스템 동기화

4. **스키마 변경 감지**:
   - DDL 변경도 이벤트로 캡처
   - 스키마 진화 추적

## 기술 선택 가이드

### 의사결정 트리

```
데이터 통신 필요
    │
    ├─ 즉각적 응답 필요?
    │   │
    │   ├─ Yes → 복잡한 쿼리 필요?
    │   │         │
    │   │         ├─ Yes → RDBMS + REST API
    │   │         └─ No  → REST API
    │   │
    │   └─ No  → 다수 소비자?
    │            │
    │            ├─ Yes → Kafka
    │            └─ No  → 순서 보장 필요?
    │                      │
    │                      ├─ Yes → Kafka
    │                      └─ No  → 전통적 MQ (RabbitMQ)
```

### 상황별 권장 기술

| 상황 | 권장 기술 | 이유 |
|------|----------|------|
| 사용자 잔액 조회 | REST API | 즉각적 응답 필요 |
| 주문 이벤트 처리 | Kafka | 다수 시스템이 소비, 순서 보장 |
| 실시간 대시보드 | Kafka + Streams | 스트림 집계 |
| 복잡한 보고서 | RDBMS | SQL 조인 필요 |
| 마이크로서비스 통신 | Kafka | 약결합, 확장성 |
| 단순 작업 큐 | RabbitMQ | 라우팅 유연성 |

## 실무 체크리스트

### Kafka 도입 검토

- [ ] 다수 소비자가 같은 데이터 필요?
- [ ] 순서 보장이 중요?
- [ ] 재처리/디버깅을 위한 히스토리 필요?
- [ ] 시스템 간 디커플링 필요?

### 주의할 실수

- [ ] 모든 것을 Kafka로 해결하려 하지 않기 (REST가 적합한 경우도 있음)
- [ ] Kafka를 DB처럼 사용하지 않기 (무기한 보존, 복잡한 쿼리 부적합)
- [ ] 동기식 패턴을 Kafka에 적용하지 않기 (Request-Reply 복잡)
- [ ] Schema 관리 필수 (Schema Registry 활용)
- [ ] CDC 없이 이중 쓰기 피하기
