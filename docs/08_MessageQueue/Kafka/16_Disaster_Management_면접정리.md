# Chapter 16: 재해 관리 (Disaster Management) - 면접 정리

## 1. 장애 유형 (What Could Possibly Go Wrong?)

### 1.1 개요

Kafka와 같은 분산 시스템에서는 다양한 유형의 장애가 발생할 수 있습니다. 재해 관리의 핵심은 이러한 장애 유형을 이해하고, 각 유형에 맞는 대응 전략을 수립하는 것입니다. 장애는 기술적 요인뿐만 아니라 인적 요인에 의해서도 증폭될 수 있습니다.

**장애 유형 분류**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Kafka Failure Types                           │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │               Technical Failures                         │    │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐│    │
│  │  │ Network  │  │ Compute  │  │ Storage  │  │   DC     ││    │
│  │  │          │  │          │  │          │  │          ││    │
│  │  │ 클라이언트│  │ 브로커   │  │ 디스크   │  │ 전체DC  ││    │
│  │  │ 연결불가 │  │ 다운     │  │ 장애     │  │ 장애     ││    │
│  │  └──────────┘  └──────────┘  └──────────┘  └──────────┘│    │
│  └─────────────────────────────────────────────────────────┘    │
│                            │                                     │
│                            ▼                                     │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                Human Factors (증폭 요인)                 │    │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐   │    │
│  │  │ 잘못된 설정  │  │ 유지보수 오류│  │부적절한 절차 │   │    │
│  │  └──────────────┘  └──────────────┘  └──────────────┘   │    │
│  └─────────────────────────────────────────────────────────┘    │
│                            │                                     │
│                            ▼                                     │
│                   ┌──────────────┐                              │
│                   │ 서비스 영향  │                              │
│                   └──────────────┘                              │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 네트워크 장애 (Network Failures)

분산 시스템에서 네트워크 문제는 **거의 확실히 발생**합니다. 클라이언트와 브로커 간, 또는 브로커와 브로커 간 연결이 끊어질 수 있습니다.

**시나리오별 영향과 대응**:

| 시나리오 | 영향 | 대응 옵션 |
|---------|------|----------|
| **클라이언트-브로커 연결 불가** | 메시지 전송/수신 불가 | 대기, 버퍼링, 드롭, 크래시 |
| **브로커 간 네트워크 문제** | 복제 지연, 리더 선출 영향 | 연결 복구 대기 |

**프로듀서 대응 옵션**:

```
프로듀서의 네트워크 장애 대응:

1. 블로킹 (대기)
   └─ buffer.memory 설정 내에서 대기 후 TimeoutException

2. 버퍼링 (메모리 저장)
   └─ 연결 복구 시 전송
   └─ 메모리 제한 주의

3. 메시지 드롭
   └─ 중요하지 않은 메트릭 데이터 등
   └─ 데이터 손실 허용 시

4. 크래시 후 사용자 알림
   └─ 수동 개입이 필요한 크리티컬 시스템
```

**핵심 포인트**:
- Kafka의 비동기 아키텍처와 영속성 모델 덕분에 네트워크 복구 후 메시지를 재생산/재소비할 수 있습니다
- 데이터 손실이나 불일치 상태를 **두려워할 필요 없습니다**
- 단, **실시간 크리티컬 애플리케이션**은 짧은 중단도 허용 불가하므로 별도 전략 필요

### 1.3 컴퓨트 장애 (Compute Failures)

브로커 프로세스나 호스트 자체의 장애입니다. Kafka의 복제(Replication) 메커니즘이 이를 처리합니다.

**장애 허용 설정**:

```
┌─────────────────────────────────────────────────────────────────┐
│                 Fault Tolerance Configuration                    │
│                                                                  │
│   Replication Factor = 3                                         │
│   min.insync.replicas = 2                                        │
│                                                                  │
│   정상 상태:                                                      │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐                         │
│   │Broker 1 │  │Broker 2 │  │Broker 3 │                         │
│   │ Leader  │  │ ISR     │  │ ISR     │                         │
│   └─────────┘  └─────────┘  └─────────┘                         │
│       ↑            ↑            ↑                               │
│       └────────────┴────────────┘                               │
│              ISR = 3 (정상)                                      │
│                                                                  │
│   1개 브로커 장애 (허용됨):                                       │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐                         │
│   │Broker 1 │  │Broker 2 │  │Broker 3 │                         │
│   │ Leader  │  │ ISR     │  │  DOWN   │                         │
│   └─────────┘  └─────────┘  └─────────┘                         │
│       ↑            ↑                                            │
│       └────────────┘                                            │
│        ISR = 2 (min.insync.replicas 충족)                        │
│        → acks=all 쓰기 가능                                      │
│                                                                  │
│   2개 브로커 장애 (문제 발생):                                    │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐                         │
│   │Broker 1 │  │Broker 2 │  │Broker 3 │                         │
│   │ Leader  │  │  DOWN   │  │  DOWN   │                         │
│   └─────────┘  └─────────┘  └─────────┘                         │
│       ↑                                                         │
│       └ ISR = 1 (min.insync.replicas 미달)                       │
│       → acks=all 쓰기 불가 (NotEnoughReplicasException)          │
│       → acks=1 쓰기는 가능                                       │
│       → 컨슈머는 계속 작동                                       │
└─────────────────────────────────────────────────────────────────┘
```

**설정별 장애 허용**:

| 설정 | 장애 허용 | 적합한 케이스 |
|------|----------|--------------|
| RF=3, min.insync.replicas=2 | 1개 브로커 | 일반적인 프로덕션 |
| RF=4, min.insync.replicas=2 | 2개 브로커 | 미션 크리티컬 시스템 |
| RF=5, min.insync.replicas=3 | 2개 브로커 | 초고가용성 요구 |

**브로커 복구 절차**:

```
브로커 복구 단계:

1. 새 하드웨어에 Kafka 설치
   └─ 같은 버전, 같은 설정

2. ⚠️ 동일한 broker.id로 설정 (핵심!)
   └─ broker.id가 다르면 새 브로커로 인식
   └─ 기존 파티션 할당 정보 유지됨

3. 브로커 시작
   └─ 컨트롤러가 인식

4. 자동 데이터 복제 시작
   └─ 리더로부터 데이터 동기화
   └─ UnderReplicatedPartitions 감소

5. ISR 합류 후 리더 역할 재개
   └─ 선호 리더 선출 (auto.leader.rebalance.enable=true)
```

**주의**: Replication Factor 증가는 스토리지 비용이 비례하여 증가합니다. 또한 브로커 장애만 보호하며, 인적 오류 같은 더 흔한 장애는 보호하지 못합니다.

### 1.4 스토리지 장애 (Storage Failures)

디스크 관련 장애는 데이터 손실로 이어질 수 있어 특별한 주의가 필요합니다.

**스토리지 장애 유형**:

| 유형 | 영향 | 대응 |
|------|------|------|
| **디스크 풀 (Full)** | 모든 브로커 동시 발생 시 영구 데이터 손실 | 예방 필수 (모니터링, 알람) |
| **디스크 영구 장애** | 해당 파티션 복제본 손실 | 다른 브로커에서 복제 |
| **중앙 스토리지 장애** | 복구 후 일부 데이터 손실 가능 | 로컬 스토리지 권장 |

**디스크 풀 예방**:

```
디스크 풀 예방 전략:

1. 모니터링
   ├─ 디스크 사용률 < 60% 유지
   ├─ 임계값 초과 시 알람
   └─ 추세 분석 (증가율 모니터링)

2. 보존 정책 조정
   ├─ retention.ms 단축
   ├─ retention.bytes 설정
   └─ 긴급 시 토픽별 조정 가능

3. 시스템 관리자 팁
   └─ 볼륨에 10GB 여유 파일 유지 (dd if=/dev/zero of=reserve bs=1G count=10)
   └─ 긴급 시 이 파일 삭제로 공간 확보
```

**다중 로그 디렉토리 (RAID 미사용 시)**:

```properties
# broker.properties
log.dirs=/data/kafka-logs1,/data/kafka-logs2,/data/kafka-logs3
```

- 각 디렉토리가 별도 물리 디스크에 위치
- 하나의 디렉토리 실패 시 다른 디렉토리는 **계속 작동**
- 실패한 디스크 교체 후 자동 복제 시작
- RAID보다 세밀한 장애 격리 가능

### 1.5 데이터센터 장애 (Data Center Failures)

전체 데이터센터의 장애는 가장 큰 규모의 장애입니다. **Stretched Cluster**로 대응할 수 있습니다.

**Stretched Cluster 아키텍처**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Stretched Cluster                             │
│                                                                  │
│   DC1 (AZ-1)              DC2 (AZ-2)              DC3 (AZ-3)     │
│   ┌─────────────┐        ┌─────────────┐        ┌─────────────┐ │
│   │ Broker 1    │        │ Broker 2    │        │ Broker 3    │ │
│   │ Broker 4    │        │ Broker 5    │        │ Broker 6    │ │
│   │             │        │             │        │             │ │
│   │ KRaft Node  │        │ KRaft Node  │        │ KRaft Node  │ │
│   └──────┬──────┘        └──────┬──────┘        └──────┬──────┘ │
│          │                      │                      │        │
│          └──────────────────────┴──────────────────────┘        │
│                    저지연 네트워크 연결 (< 30ms)                  │
│                                                                  │
│   Topic: orders (RF=3)                                          │
│   ├─ Partition 0: Leader(DC1), Replica(DC2), Replica(DC3)       │
│   ├─ Partition 1: Leader(DC2), Replica(DC3), Replica(DC1)       │
│   └─ Partition 2: Leader(DC3), Replica(DC1), Replica(DC2)       │
│                                                                  │
│   → DC1 전체 장애 시에도 DC2, DC3에서 계속 운영                  │
└─────────────────────────────────────────────────────────────────┘
```

**Stretched Cluster 핵심 원칙**:

| 원칙 | 설명 | 이유 |
|------|------|------|
| **최소 3개 DC** | 2개만으로는 불충분 | 조정 클러스터 과반수(Quorum) 확보 불가 |
| **네트워크 지연 < 30ms** | 리전 간이 아닌 AZ 간 배포 | 높은 지연은 성능 저하 |
| **broker.rack 설정** | 복제본을 DC 간 균등 분배 | 단일 DC 장애 시 데이터 보호 |
| **client.rack 설정** | 컨슈머가 같은 DC의 복제본에서 읽기 | 크로스 DC 트래픽 비용 절감 |

**설정 예시**:

```properties
# broker.properties (각 DC별)
broker.rack=dc1  # 또는 dc2, dc3

# consumer.properties
client.rack=dc1  # 컨슈머가 위치한 DC
```

**경고**: Stretched Cluster는 복잡성과 비용이 크게 증가합니다. 데이터센터 전체 장애 시 Kafka보다 더 큰 문제(네트워크, 다른 서비스 등)가 있을 수 있으므로 **신중히 결정**해야 합니다.

## 2. Kafka 백업 (Backing Up Kafka)

### 2.1 백업의 어려움

Kafka 백업은 전통적인 데이터베이스 백업과 다릅니다. Kafka의 특성상 일관성 있는 백업을 만들기 어렵습니다.

**백업이 어려운 이유**:

```
┌─────────────────────────────────────────────────────────────────┐
│                 Kafka Backup Challenges                          │
│                                                                  │
│  1. 타이밍 문제                                                  │
│     ├─ 여러 브로커/파티션의 스냅샷 시점 불일치                   │
│     ├─ 백업 중에도 데이터 계속 생성                              │
│     └─ 일관성 있는 특정 시점 포착 불가                           │
│                                                                  │
│  2. 중복 데이터                                                  │
│     ├─ RF=3이면 동일 데이터 3번 백업                             │
│     └─ 스토리지 비용 증가                                        │
│                                                                  │
│  3. 연속 생산/소비                                               │
│     ├─ 백업 완료 후에도 새 데이터 생성                           │
│     └─ 백업 시점과 복구 시점 사이 데이터 손실                    │
│                                                                  │
│  4. 오프셋 미저장                                                │
│     ├─ 컨슈머 오프셋은 __consumer_offsets 토픽에 저장            │
│     ├─ 데이터 백업만으로는 컨슈머 위치 복구 불가                 │
│     └─ 복구 후 어디서부터 처리할지 불명확                        │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 백업이 필요 없을 수도 있는 경우

모든 Kafka 클러스터가 백업이 필요한 것은 아닙니다.

**백업이 필요 없는 경우**:

| 케이스 | 설명 |
|--------|------|
| **원본 시스템에서 재생성 가능** | DB → Kafka CDC 파이프라인 (DB에서 다시 추출 가능) |
| **단순 메시징 용도** | 데이터가 수 초만 필요하고 이미 처리된 경우 |
| **로그/메트릭 데이터** | 손실 허용 가능한 텔레메트리 데이터 |

### 2.3 백업 접근법 비교

| 접근법 | 장점 | 단점 |
|--------|------|------|
| **파일시스템 스냅샷** | 익숙한 절차, 도구 풍부 | 타이밍 문제, 중복 데이터, 일관성 없음 |
| **Kafka Connect → S3** | 연속 백업, 증분 방식 | 컨슈머 오프셋 미저장, 복원 복잡 |
| **Tiered Storage** | Kafka 내장 기능, 자동화 | Compacted 토픽 미지원, 오프셋 토픽 미지원 |
| **Kannika (상용)** | 완전한 백업/복원, 특정 시점 복구 | 상용 솔루션, 비용 |

## 3. MirrorMaker로 클러스터 미러링

### 3.1 MirrorMaker 개요

MirrorMaker는 Kafka Connect 기반의 클러스터 미러링 도구입니다. 서로 다른 클러스터 간 데이터를 복제하여 재해 복구(DR), 지리적 분산, 데이터 집계 등을 구현합니다.

**MirrorMaker 아키텍처**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    MirrorMaker 2 Architecture                    │
│                                                                  │
│   Source Cluster                      Target Cluster            │
│   ┌─────────────────┐                ┌─────────────────┐        │
│   │                 │                │                 │        │
│   │  orders (토픽)  │────────────────▶│ source.orders  │        │
│   │                 │                │  (Remote Topic) │        │
│   │  users (토픽)   │────────────────▶│ source.users   │        │
│   │                 │                │  (Remote Topic) │        │
│   │  __consumer_    │                │ __consumer_     │        │
│   │    offsets      │──(변환)────────▶│   offsets      │        │
│   │                 │                │                 │        │
│   └─────────────────┘                └─────────────────┘        │
│                                                                  │
│                    MirrorMaker 2 Connectors                     │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │  ┌─────────────────┐  ┌─────────────────┐  ┌─────────┐ │   │
│   │  │MirrorSource     │  │MirrorCheckpoint │  │Mirror   │ │   │
│   │  │Connector        │  │Connector        │  │Heartbeat│ │   │
│   │  │                 │  │                 │  │Connector│ │   │
│   │  │ 토픽 데이터     │  │ 오프셋 동기화   │  │ 연결    │ │   │
│   │  │ ACL 복제       │  │ (자동 변환)     │  │ 상태    │ │   │
│   │  └─────────────────┘  └─────────────────┘  └─────────┘ │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

**MirrorMaker 커넥터 역할**:

| 커넥터 | 역할 | 대상 |
|--------|------|------|
| **MirrorSourceConnector** | 토픽 데이터 및 ACL 미러링 | 토픽 메시지, ACL 정책 |
| **MirrorCheckpointConnector** | 컨슈머 오프셋 동기화 | 소스 오프셋 → 타겟 오프셋 변환 |
| **MirrorHeartbeatConnector** | 연결 상태 확인 | heartbeats 토픽으로 모니터링 |

### 3.2 Active-Passive 클러스터

Active-Passive 구성은 한쪽 클러스터만 활성 상태로 운영하고, 다른 쪽은 대기(Standby) 상태로 유지합니다.

**Active-Passive 아키텍처**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Active-Passive Cluster                        │
│                                                                  │
│   평상시 (정상 운영):                                            │
│   ┌───────────────────────┐      ┌───────────────────────┐      │
│   │    Active Cluster     │      │   Passive Cluster     │      │
│   │                       │      │                       │      │
│   │  ┌─────────────────┐  │      │  ┌─────────────────┐  │      │
│   │  │    Producers    │  │      │  │   (대기 중)     │  │      │
│   │  │        ↓        │  │      │  │                 │  │      │
│   │  │     Kafka       │─────────▶│     Kafka       │  │      │
│   │  │        ↓        │  MM2   │  │   (Mirror)      │  │      │
│   │  │    Consumers    │  │      │  │                 │  │      │
│   │  └─────────────────┘  │      │  └─────────────────┘  │      │
│   └───────────────────────┘      └───────────────────────┘      │
│                                                                  │
│   장애 시 (Failover):                                            │
│   ┌───────────────────────┐      ┌───────────────────────┐      │
│   │    Active Cluster     │      │   Passive → Active    │      │
│   │                       │      │                       │      │
│   │  ┌─────────────────┐  │      │  ┌─────────────────┐  │      │
│   │  │     DOWN        │  │      │  │    Producers    │  │      │
│   │  │                 │  │      │  │        ↓        │  │      │
│   │  │                 │  │   ←──── │     Kafka       │  │      │
│   │  │                 │  │  DNS   │        ↓        │  │      │
│   │  │                 │  │  전환   │    Consumers    │  │      │
│   │  └─────────────────┘  │      │  └─────────────────┘  │      │
│   └───────────────────────┘      └───────────────────────┘      │
└─────────────────────────────────────────────────────────────────┘
```

**Active-Passive 장단점**:

| 장점 | 단점 |
|------|------|
| 구현 단순 | 장애 복구 후 재구축 필요 |
| 데이터센터 마이그레이션에 유용 | 페일백(Failback) 복잡 |
| MirrorMaker 설정 단순 | Passive 클러스터 리소스 낭비 |

**주의**: 매 페일오버 후 이전 Active 클러스터를 **삭제하고** 새 Passive 클러스터를 생성해야 합니다. 단순히 MirrorMaker 방향만 바꾸면 데이터 불일치 발생 가능합니다.

### 3.3 Active-Active 클러스터

Active-Active 구성은 양쪽 클러스터가 모두 활성 상태로 운영되며, 서로의 데이터를 미러링합니다.

**Active-Active 아키텍처**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Active-Active Cluster                         │
│                                                                  │
│   Cluster A (DC-1)                    Cluster B (DC-2)          │
│   ┌───────────────────────┐          ┌───────────────────────┐  │
│   │                       │          │                       │  │
│   │  Local Topics:        │          │  Local Topics:        │  │
│   │  ├─ orders           │◀────────▶│  ├─ orders           │  │
│   │  ├─ users            │   MM2    │  ├─ users            │  │
│   │  └─ payments         │  (양방향) │  └─ payments         │  │
│   │                       │          │                       │  │
│   │  Remote Topics:       │          │  Remote Topics:       │  │
│   │  ├─ dc-b.orders      │          │  ├─ dc-a.orders      │  │
│   │  ├─ dc-b.users       │          │  ├─ dc-a.users       │  │
│   │  └─ dc-b.payments    │          │  └─ dc-a.payments    │  │
│   │                       │          │                       │  │
│   │  ┌─────────────────┐ │          │  ┌─────────────────┐  │  │
│   │  │ Local Producers │ │          │  │ Local Producers │  │  │
│   │  │ Local Consumers │ │          │  │ Local Consumers │  │  │
│   │  └─────────────────┘ │          │  └─────────────────┘  │  │
│   └───────────────────────┘          └───────────────────────┘  │
│                                                                  │
│   장애 시 (DC-1 Down):                                           │
│   ┌───────────────────────┐          ┌───────────────────────┐  │
│   │   Cluster A (DOWN)    │          │   Cluster B (Active)  │  │
│   │                       │          │                       │  │
│   │         ✕             │    DNS   │  모든 트래픽 처리      │  │
│   │                       │    전환   │  dc-a.* 토픽에서      │  │
│   │                       │   ────▶  │  데이터 계속 읽기     │  │
│   └───────────────────────┘          └───────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**Remote Topics**:

MirrorMaker 2에서 도입된 **Remote Topics**는 소스 클러스터 이름이 접두사로 붙은 읽기 전용 토픽입니다.

| 특성 | 설명 |
|------|------|
| **명명 규칙** | `{source-cluster}.{topic-name}` (예: dc-a.orders) |
| **읽기 전용** | ACL로 쓰기 차단, 로컬 Producer 쓰기 불가 |
| **무한 루프 방지** | 이미 미러링된 토픽(접두사 있음)은 다시 미러링하지 않음 |
| **자동 생성** | MirrorMaker가 자동으로 생성 |

**중복 처리 방지**:

Active-Active 환경에서는 같은 메시지가 양쪽 클러스터에 존재합니다. 중복 처리를 방지해야 합니다.

```
중복 처리 방지 전략:

1. 멱등성(Idempotence) 설계
   ├─ 결제: 트랜잭션 ID로 중복 체크
   ├─ 주문: 주문 ID로 중복 체크
   └─ 상태 변경: 버전 번호 또는 타임스탬프 비교

2. Consumer Group 분리
   ├─ DC-1 Consumer는 로컬 orders + dc-b.orders 구독
   ├─ DC-2 Consumer는 로컬 orders + dc-a.orders 구독
   └─ 각 DC에서 처리한 메시지 추적

3. 트랜잭션 ID 기반 중복 제거
   └─ 메시지에 고유 ID 포함, 처리 전 중복 확인
```

**Active-Active 장점**:

| 장점 | 설명 |
|------|------|
| 즉시 페일오버 | 한 DC 장애 시 다른 DC에서 즉시 처리 |
| 재구축 불필요 | 장애 복구 후 자동 동기화 |
| 리소스 활용 | 양쪽 모두 활성 (유휴 리소스 없음) |
| 지역 지연 최소화 | 사용자가 가까운 DC에 연결 |

### 3.4 Hub-and-Spoke 토폴로지

Hub-and-Spoke 토폴로지는 중앙(Hub) 클러스터와 여러 분산(Spoke) 클러스터로 구성됩니다.

**Hub-and-Spoke 아키텍처**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Hub-and-Spoke Topology                        │
│                                                                  │
│                       ┌───────────────────┐                     │
│                       │   Hub (본사)       │                     │
│                       │                   │                     │
│                       │  ┌─────────────┐  │                     │
│                       │  │   Kafka     │  │                     │
│                       │  │  Cluster    │  │                     │
│                       │  └─────────────┘  │                     │
│                       │                   │                     │
│                       │  ┌─────────────┐  │                     │
│                       │  │  Analytics  │  │                     │
│                       │  │   System    │  │                     │
│                       │  └─────────────┘  │                     │
│                       └─────────┬─────────┘                     │
│                                 │                               │
│           ┌─────────────────────┼─────────────────────┐         │
│           │                     │                     │         │
│           ▼                     ▼                     ▼         │
│   ┌───────────────┐     ┌───────────────┐     ┌───────────────┐ │
│   │ Spoke 1 (지점)│     │ Spoke 2 (지점)│     │ Spoke 3 (지점)│ │
│   │               │     │               │     │               │ │
│   │  Kafka Mini   │     │  Kafka Mini   │     │  Kafka Mini   │ │
│   │               │     │               │     │               │ │
│   │ ↑ 매출 데이터 │     │ ↑ 매출 데이터 │     │ ↑ 매출 데이터 │ │
│   │ ↓ 가격 정보   │     │ ↓ 가격 정보   │     │ ↓ 가격 정보   │ │
│   └───────────────┘     └───────────────┘     └───────────────┘ │
│                                                                  │
│   데이터 흐름:                                                   │
│   Spoke → Hub: 매출, 재고, 고객 데이터 (집계 분석용)             │
│   Hub → Spoke: 가격, 프로모션, 정책 (배포용)                     │
└─────────────────────────────────────────────────────────────────┘
```

**사용 사례**:

| 사례 | 설명 |
|------|------|
| **슈퍼마켓 체인** | 본사 ↔ 각 매장 (POS 데이터 집계, 가격 정보 배포) |
| **크루즈 선박** | 항구에서 동기화, 바다에서 자율 운영 |
| **정부 기관** | 인터넷 연결 없이도 로컬 운영, 연결 시 동기화 |
| **제조업** | 공장별 IoT 데이터 수집, 중앙 분석 |

**Hub-and-Spoke 특징**:

| 특징 | 설명 |
|------|------|
| **연결 끊어져도 로컬 운영 가능** | Spoke가 독립적으로 Producer/Consumer 처리 |
| **연결 복구 시 자동 동기화** | MirrorMaker가 밀린 데이터 전송 |
| **중앙 집중 분석 가능** | Hub에서 모든 Spoke 데이터 집계 |
| **양방향 데이터 흐름 지원** | Hub → Spoke (배포), Spoke → Hub (수집) |

**주의**: MirrorMaker는 **비동기**로 동작합니다. 연결이 끊어진 동안의 데이터는 복구 후에 전송되지만, 네트워크 장애 직전의 최신 데이터는 손실될 수 있습니다.

## 4. 아키텍처 선택 가이드

어떤 재해 복구 아키텍처를 선택할지는 요구사항에 따라 다릅니다.

**선택 의사결정 트리**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    DR Architecture Selection                     │
│                                                                  │
│   ① 클러스터 미러링 필요?                                        │
│      │                                                          │
│      ├─ No → 단일 클러스터 (Stretched Cluster 고려)             │
│      │                                                          │
│      └─ Yes → ② 두 DC 모두 활성 필요?                           │
│               │                                                 │
│               ├─ No → Active-Passive                            │
│               │       ├─ DR 용도                                │
│               │       └─ 마이그레이션 용도                       │
│               │                                                 │
│               └─ Yes → ③ 분산된 지점 다수?                      │
│                        │                                        │
│                        ├─ No → Active-Active                    │
│                        │       ├─ 양쪽 DC 활성                  │
│                        │       └─ 무중단 운영 필요               │
│                        │                                        │
│                        └─ Yes → Hub-and-Spoke                   │
│                                 ├─ 본사-지점 구조               │
│                                 └─ 연결 끊어져도 운영            │
└─────────────────────────────────────────────────────────────────┘
```

**아키텍처별 비교**:

| 아키텍처 | 복잡도 | RTO | RPO | 비용 | 적합한 케이스 |
|----------|--------|-----|-----|------|--------------|
| **Stretched Cluster** | 높음 | 낮음 | 0 | 높음 | 같은 리전 내 AZ 분산 |
| **Active-Passive** | 낮음 | 중간 | 분 단위 | 중간 | DR, 마이그레이션 |
| **Active-Active** | 높음 | 낮음 | 초 단위 | 높음 | 무중단 운영 |
| **Hub-and-Spoke** | 중간 | 중간 | 분~시간 | 중간 | 분산 지점 구조 |

> **RTO** (Recovery Time Objective): 복구 목표 시간
> **RPO** (Recovery Point Objective): 복구 목표 시점 (허용 가능한 데이터 손실)

## 면접 예상 질문

### Q1: Kafka에서 브로커 장애를 어떻게 처리하나요?

**모범 답변**:

Kafka는 **복제(Replication)**를 통해 브로커 장애를 처리합니다.

**핵심 설정**:
- `replication.factor`: 파티션당 복제본 수 (일반적으로 3)
- `min.insync.replicas`: 최소 동기화 복제본 수 (일반적으로 2)

**장애 허용**:
- RF=3, min.insync.replicas=2 설정 시 **1개 브로커 장애** 허용
- `acks=all` 설정의 Producer는 ISR 수가 min.insync.replicas 미만이면 쓰기 불가
- Consumer는 리더만 있으면 읽기 가능

**브로커 복구 절차**:
1. 새 하드웨어에 Kafka 설치
2. **동일한 broker.id로 설정** (핵심!)
3. 브로커 시작하면 컨트롤러가 인식
4. 자동으로 리더로부터 데이터 동기화 시작
5. ISR 합류 후 리더 역할 재개 가능

**주의할 점**:
- Replication Factor 증가는 스토리지 비용 비례 증가
- 브로커 장애는 보호하지만 인적 오류는 보호하지 못함

---

### Q2: Active-Passive와 Active-Active 클러스터의 차이점은?

**모범 답변**:

**Active-Passive**:
- 한 클러스터만 트래픽 처리 (Active)
- 다른 클러스터는 대기 상태로 데이터만 미러링 (Passive)
- 장애 시 Passive → Active로 전환
- 구현이 단순하지만 페일백이 복잡
- 매 페일오버 후 이전 Active 클러스터 재구축 필요

**Active-Active**:
- 양쪽 클러스터 모두 트래픽 처리
- MirrorMaker로 양방향 미러링
- Remote Topics로 무한 루프 방지 (dc-a.orders, dc-b.orders)
- 한 DC 장애 시 다른 DC에서 즉시 처리 계속
- 장애 복구 후 자동 동기화
- 중복 처리 방지를 위한 멱등성 설계 필요

**선택 기준**:
- 운영 복잡도 감수 가능 + 무중단 필수 → Active-Active
- 단순한 DR 용도 + 분 단위 RPO 허용 → Active-Passive

---

### Q3: MirrorMaker 2의 Remote Topics란?

**모범 답변**:

**Remote Topics**는 MirrorMaker 2에서 도입된 개념으로, 소스 클러스터에서 미러링된 토픽에 소스 클러스터 이름이 접두사로 붙은 것입니다.

**예시**:
- 소스 클러스터 "dc-a"의 "orders" 토픽
- 타겟 클러스터에서는 "dc-a.orders"로 생성

**특징**:
1. **명명 규칙**: `{source-cluster}.{topic-name}`
2. **읽기 전용**: ACL로 로컬 쓰기 차단
3. **무한 루프 방지**: 이미 접두사가 붙은 토픽(dc-a.orders)은 다시 미러링하지 않음
4. **자동 생성**: MirrorMaker가 소스 토픽 감지 시 자동 생성

**장점**:
- 로컬 토픽과 미러링된 토픽 명확히 구분
- Active-Active 환경에서 무한 미러링 루프 방지
- Consumer가 양쪽 클러스터의 데이터를 모두 읽을 수 있음

---

### Q4: Stretched Cluster가 최소 3개 DC가 필요한 이유는?

**모범 답변**:

Stretched Cluster가 3개 DC가 필요한 이유는 **조정 클러스터의 쿼럼(Quorum)** 때문입니다.

**쿼럼의 원리**:
- ZooKeeper 또는 KRaft Controller는 **과반수**(majority)가 있어야 결정 가능
- 2개 DC에 균등 배치하면 하나의 DC 장애 시 과반수 확보 불가

**2개 DC의 문제**:

```
2개 DC 구성:
DC-A: 2노드 (ZK/KRaft)
DC-B: 2노드 (ZK/KRaft)
총: 4노드, 쿼럼: 3노드 필요

DC-A 장애 → 남은 노드: 2개 → 쿼럼 미달 → 클러스터 정지
```

**3개 DC의 해결책**:

```
3개 DC 구성:
DC-A: 2노드
DC-B: 2노드
DC-C: 1노드
총: 5노드, 쿼럼: 3노드 필요

DC-A 장애 → 남은 노드: 3개 → 쿼럼 충족 → 클러스터 계속 운영
```

**대안**:
- 2개 DC만 있다면 3번째 DC에 조정 노드만 배치 (데이터 없이)
- 또는 Active-Passive/Active-Active 아키텍처 선택

---

### Q5: Kafka 백업이 어려운 이유와 대안은?

**모범 답변**:

**백업이 어려운 이유**:

1. **타이밍 문제**: 여러 브로커/파티션의 스냅샷 시점이 불일치하여 일관성 있는 백업이 어려움
2. **중복 데이터**: RF=3이면 동일 데이터가 3번 백업되어 비효율적
3. **연속 생산/소비**: 백업 중에도 데이터가 계속 생성되어 백업 완료 시점과 복구 시점 사이 데이터 손실
4. **오프셋 미저장**: 데이터만 백업하면 Consumer가 어디서부터 처리할지 알 수 없음

**대안**:

1. **백업이 필요 없는 경우 판단**:
   - 원본 시스템(DB)에서 재생성 가능하면 백업 불필요
   - 단순 메시징 용도로 이미 처리된 데이터

2. **MirrorMaker 활용**:
   - 별도 클러스터로 실시간 미러링
   - 백업보다 유연하고 오프셋 동기화 포함

3. **Tiered Storage (Kafka 3.9+)**:
   - 오래된 데이터를 S3 등으로 자동 이동
   - 단, Compacted 토픽, __consumer_offsets 미지원

4. **상용 솔루션 (Kannika 등)**:
   - 특정 시점 복구(Point-in-Time Recovery) 지원
   - 완전한 백업/복원 기능

---

### Q6: Hub-and-Spoke 토폴로지의 사용 사례는?

**모범 답변**:

Hub-and-Spoke는 **중앙(Hub) 클러스터**와 **여러 분산(Spoke) 클러스터**로 구성된 토폴로지입니다.

**사용 사례**:

1. **슈퍼마켓 체인**:
   - Hub(본사): 전체 매장 데이터 집계, 분석 시스템
   - Spoke(매장): POS 시스템, 로컬 재고 관리
   - Spoke → Hub: 매출, 재고 데이터
   - Hub → Spoke: 가격 정보, 프로모션

2. **크루즈 선박**:
   - Hub(항구 서버): 예약 시스템, 고객 데이터
   - Spoke(선박): 선상 서비스, 결제 시스템
   - 바다에서는 자율 운영, 항구에서 동기화

3. **정부 기관**:
   - Hub(중앙 정부): 정책 데이터, 통계 집계
   - Spoke(지방 기관): 로컬 민원 처리
   - 인터넷 연결 없이도 로컬 운영 필요

4. **제조업**:
   - Hub(본사): 전체 생산 분석, 품질 관리
   - Spoke(공장): IoT 센서 데이터, 로컬 모니터링

**핵심 특징**:
- 연결이 끊어져도 Spoke가 독립적으로 운영
- 연결 복구 시 MirrorMaker가 자동 동기화
- 양방향 데이터 흐름 지원 (수집 + 배포)

## 실무 체크리스트

### 장애 대비

- [ ] Replication Factor와 min.insync.replicas 적절히 설정
- [ ] 디스크 사용률 모니터링 (< 60%)
- [ ] 브로커 복구 절차 문서화 (동일 broker.id 사용)
- [ ] 다중 로그 디렉토리 설정 (RAID 미사용 시)

### Stretched Cluster

- [ ] 최소 3개 DC/AZ 확보
- [ ] 네트워크 지연 < 30ms 확인
- [ ] broker.rack 설정으로 복제본 분산
- [ ] client.rack 설정으로 비용 최적화

### MirrorMaker

- [ ] 적절한 아키텍처 선택 (Active-Passive, Active-Active, Hub-and-Spoke)
- [ ] Remote Topics 명명 규칙 이해
- [ ] Active-Active 시 중복 처리 방지 전략
- [ ] MirrorMaker 지연 모니터링

### 백업 전략

- [ ] 백업 필요성 평가 (원본 재생성 가능 여부)
- [ ] 오프셋 동기화 방안 고려
- [ ] 복구 절차 테스트
