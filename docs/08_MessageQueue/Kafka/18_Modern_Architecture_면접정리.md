## 1. Data Mesh의 핵심으로서의 Kafka

### 1.1 전통적 데이터 관리의 문제점

전통적인 데이터 관리 방식에서는 **중앙 데이터 엔지니어링 팀**이 모든 데이터 통합을 담당합니다. 이 방식은 조직이 성장하면서 심각한 병목 현상을 일으킵니다.

**전통적 데이터 관리의 구조**:

```
┌─────────────────────────────────────────────────────────────────┐
│              Traditional Data Management (문제점)                │
│                                                                  │
│   영업팀 (CRM)        생산팀 (ERP)        재무팀 (회계)         │
│   ┌─────────┐         ┌─────────┐         ┌─────────┐          │
│   │ 스키마  │         │ 불완전한│         │ 복잡한  │          │
│   │ 변경    │         │ 데이터  │         │ 정규화  │          │
│   │ (알림X) │         │ 추출    │         │ 데이터  │          │
│   └────┬────┘         └────┬────┘         └────┬────┘          │
│        │                   │                   │               │
│        └───────────────────┼───────────────────┘               │
│                            ▼                                    │
│                   ┌─────────────────┐                          │
│                   │   중앙 데이터팀 │                          │
│                   │                 │                          │
│                   │ - 모든 도메인   │                          │
│                   │   전문성 부족   │                          │
│                   │ - ETL 파이프라인│                          │
│                   │   빈번한 장애   │                          │
│                   │ - 병목 현상     │                          │
│                   └────────┬────────┘                          │
│                            ▼                                    │
│                   ┌─────────────────┐                          │
│                   │ Data Warehouse  │                          │
│                   │ (신뢰할 수 없는 │                          │
│                   │  리포트)        │                          │
│                   └─────────────────┘                          │
│                                                                  │
│   결과: 영업팀 불만 ("리포트가 틀렸어!")                        │

│        경영진 좌절 ("의사결정을 못하겠어!")                     │
└─────────────────────────────────────────────────────────────────┘
```

**핵심 문제점**:

| 부서 | 문제 |
|------|------|
| **영업팀** | CRM 스키마 변경 시 데이터팀에 알리지 않음 |
| **생산팀** | 시스템 문제로 불완전한 데이터 추출 |
| **데이터팀** | 모든 도메인 전문성 부족, ETL 파이프라인 빈번한 장애 |
| **결과** | 신뢰할 수 없는 리포트, 모든 팀의 불만 |

**원칙별 상세 설명**:

| 원칙 | 설명 | 책임 주체 |

## 2. 코어 시스템에서 데이터 해방

### 2.1 코어 시스템의 특성

대기업의 코어 시스템(은행 코어뱅킹, SAP ERP, 메인프레임 등)은 비즈니스에 필수적이지만 현대적 요구사항에 적응하기 어렵습니다.

**코어 시스템의 특성**:

| 특성 | 설명 |
|------|------|
| **안정성** | 수십 년간 검증된 안정적인 운영 |
| **복잡한 데이터 구조** | 수백 개의 정규화된 테이블 |
| **변경 어려움** | 변경 시 막대한 테스트와 비용 필요 |
| **폐쇄적** | 외부 시스템에 데이터 노출 어려움 |

### 2.2 CDC를 통한 데이터 해방

**CDC (Change Data Capture)**를 사용하면 코어 시스템을 수정하지 않고 데이터를 "해방"할 수 있습니다.

### 2.3 구현 옵션 비교

| 접근법 | 장점 | 단점 | 적합한 케이스 |
|--------|------|------|--------------|
| **Raw CDC → Kafka** | 구현 단순 | 복잡한 정규화 데이터, 도메인 전문지식 필요 | 내부 전문가 소비자만 있을 때 |
| **Kafka Streams 변환** | 통합된 데이터 형식 | 복잡한 조인, 운영 부담 | 여러 테이블 조인 필요 |
| **Secondary DB + Outbox** | SQL로 쉬운 조인 | 추가 DB 운영 필요 | 복잡한 조인이 많을 때 |

**주의**: "1000줄의 조인을 200개 테이블에 걸쳐 Kafka Streams에서 어떻게 구현하나요?"라는 질문의 최선의 답은 **"하지 마세요"**입니다. 복잡한 조인은 관계형 DB에서 수행 후 **Outbox 패턴**으로 Kafka에 전달하는 것이 낫습니다.

## 3. IIoT (Industrial Internet of Things)에서의 Kafka

### 3.1 IIoT 사용 사례

공장, 차량, 발전소, 풍력 터빈 등에서 생성되는 산업 IoT 데이터를 처리합니다.

**IIoT 사용 사례**:

| 사용 사례 | 설명 | Kafka의 역할 |
|----------|------|-------------|
| **Predictive Maintenance** | 장비 상태 모니터링, 고장 예측 | 센서 데이터 실시간 스트리밍 |
| **Factory Automation** | 생산 라인 IoT 센서 통합 | 기기 간 실시간 통신 |
| **Supply Chain** | 차량, 창고, 공급업체 데이터 통합 | 실시간 추적 및 동적 조정 |
| **Energy Management** | 터빈, 발전기 데이터 수집 | 에너지 생산 최적화 |
| **Real-time Analytics** | 생산 병목, 품질 문제 감지 | 실시간 데이터 분석 지원 |

### 3.2 MQTT와 Kafka 통합

```
│   │   └───────────┘ └───────────┘ └───────────┘            │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

**MQTT vs Direct Kafka 선택 기준**:

| 고려사항 | MQTT 사용 | Direct Kafka |
|----------|----------|--------------|
| **디바이스 리소스** | 제한적 (마이크로컨트롤러) | 충분함 (서버급) |
| **연결 안정성** | 불안정 (간헐적 연결, 배터리) | 안정적 (24/7 연결) |
| **보안** | 신뢰할 수 없는 디바이스 | 신뢰할 수 있는 환경 |
| **프로토콜 크기** | 경량 필요 | 크기 무관 |

### 3.3 다중 Kafka 클러스터 고려사항

IIoT 환경에서는 다중 Kafka 클러스터가 필요할 수 있습니다.

**다중 클러스터가 필요한 경우**:

| 이유 | 설명 |
|------|------|
| **성능 격리** | 실시간 처리 vs 배치 분석 워크로드 분리 |
| **데이터 보안** | HR 데이터 vs 운영 데이터 분리 |
| **지리적 분산** | 북미 vs 유럽 (GDPR 등 규정 준수) |
| **확장성** | 트랜잭션 vs 분석 워크로드 분리 |

**주의**: 단일 클러스터가 기본 권장입니다. 다중 클러스터는 **관리 복잡성 증가**를 수반하므로 신중하게 결정해야 합니다.

## 4. Kafka가 적합하지 않은 경우

Kafka는 **만능 솔루션이 아닙니다**. 적합하지 않은 상황을 이해하는 것이 중요합니다.

### 4.1 Kafka는 관계형 데이터베이스가 아니다

**RDBMS vs Kafka 비교**:

| 특성 | RDBMS | Kafka |
|------|-------|-------|
| **ACID** | 완전 지원 | 부분 지원 (Isolation 제한) |
| **쿼리** | 복잡한 SQL 조인 | 오프셋 기반 순차 읽기 |
| **데이터 형태** | 정규화 (Normalized) | 비정규화 (Denormalized) |
| **통신** | 동기 | 비동기 |
| **데이터 변경** | Update, Delete 가능 | Append-only |

**핵심**: Kafka는 **서비스 간 데이터 교환**, RDBMS는 **서비스 내부 데이터 저장**에 사용합니다. Data Mesh에서도 많은 데이터베이스가 여전히 필요합니다.

### 4.2 Kafka는 동기 통신 인터페이스가 아니다

**안티패턴 vs 권장 패턴**:

```
❌ 안티패턴 (동기식 Kafka 사용):
┌────────────────────────────────────────────────────────────┐
│                                                            │
│   User ──▶ Frontend ──▶ Backend ──▶ Kafka ──▶ Service     │
│     │                       │          │          │        │
│     │                       │          │          │        │
│     │       대기 중...      │   대기   │   대기   │        │
│     │                       │          │          │        │
│     ◀───────────────────────────────────────────────       │
│                         응답 (수 초)                       │
│                                                            │
│   문제:                                                    │
│   - 사용자가 긴 대기 시간 경험                             │
│   - Kafka 장애 시 사용자 경험 저하                         │
│   - Kafka의 비동기 특성에 반함                             │
└────────────────────────────────────────────────────────────┘

✅ 권장 패턴 (비동기 Kafka + 동기 응답):
┌────────────────────────────────────────────────────────────┐
│                                                            │
│   User ──▶ Frontend ──▶ Backend ──▶ DB (동기)             │
│     │                       │          │                   │
│     │                       │          │                   │
│     ◀──────────────────── "요청 접수됨" ✅                │
│                                        │                   │
│                                        ▼                   │
│                                   CDC ──▶ Kafka            │
│                                              │              │
│                                              ▼              │
│                                          Service           │
│                                              │              │
│                                              ▼              │
│   User ◀── 푸시 알림 ◀── Backend ◀── Kafka (완료)         │
│                                                            │
│   장점:                                                    │
│   - 즉각적인 "요청 접수" 응답                              │
│   - 비동기로 처리, 완료 시 별도 알림                       │
│   - Kafka 장애가 사용자 경험에 영향 최소화                 │
└────────────────────────────────────────────────────────────┘
```

### 4.3 Kafka는 파일 교환 플랫폼이 아니다

**파일 교환 시 권장 방법**:

| 데이터 유형 | 권장 접근법 |
|------------|------------|
| **PDF 내용** | JSON으로 기계 판독 가능한 데이터 전송 → 별도 PDF 생성 서비스 |
| **대용량 파일** | Object Storage(S3) 저장 → Kafka로 파일 위치 알림 |
| **대용량 SOAP** | 의미 단위로 분할 → 이벤트당 하나의 메시지 |

```
✅ 권장 패턴:

Producer                     Kafka                      Consumer
    │                          │                           │
    │ 파일 저장                │                           │
    ├────────▶ S3/MinIO       │                           │
    │                          │                           │
    │ 메타데이터 발행          │                           │
    ├─────────────────────────▶│                           │
    │ {"file_id": "abc",       │                           │
    │  "s3_path": "s3://..."}  │                           │
    │                          │                           │
    │                          │ 메타데이터 수신           │
    │                          ├──────────────────────────▶│
    │                          │                           │
    │                          │        파일 다운로드      │
    │        S3/MinIO ◀────────────────────────────────────┤
```

### 4.4 소규모 애플리케이션에는 의문스럽다

**Kafka 도입 의사결정**:

```
Kafka 도입 고려
    │
    ├─ 시간당 메시지 > 1,000개?
    │   │
    │   ├─ No ─▶ 향후 1년간 데이터 증가 예상?
    │   │         │
    │   │         ├─ No ─▶ 여러 팀이 사용?
    │   │         │         │
    │   │         │         ├─ No ─▶ PostgreSQL + 테이블 로그로 충분
    │   │         │         └─ Yes ─▶ Kafka 적합
    │   │         └─ Yes ─▶ Kafka 적합
    │   └─ Yes ─▶ Kafka 적합
```

**Kafka가 불필요한 경우**:
- 시간당 1,000개 미만의 메시지
- 향후 1년간 데이터 증가 예상 없음
- 단일 팀만 데이터 사용
- 모놀리식 애플리케이션 내부 통신

**대안**: PostgreSQL + 테이블 기반 이벤트 로그로 충분

### 4.5 Kafka는 좋은 아키텍처를 대체하지 않는다

**핵심 교훈**:

```
┌─────────────────────────────────────────────────────────────────┐
│              Kafka Won't Fix Bad Architecture                    │
│                                                                  │
│   ❌ Kafka가 해결하지 못하는 것:                                 │
│   ├─ 잘못된 데이터 모델                                         │
│   ├─ 불분명한 요구사항                                          │
│   ├─ 도메인 이해 부족                                           │
│   └─ "Garbage In → Garbage Out"                                 │
│                                                                  │
│   ✅ 올바른 접근:                                                │
│   1. 시스템 요구사항 이해                                        │
│   2. 아키텍처 계획                                               │
│   3. Kafka 적합성 평가                                           │
│   4. 구현                                                        │
│                                                                  │
│   "Think First, Plan, Then Execute"                             │
└─────────────────────────────────────────────────────────────────┘
```

## 면접 예상 질문

### Q1: Data Mesh란 무엇이며 Kafka가 어떻게 지원하나요?

**모범 답변**:

**Data Mesh**는 Zhamak Dehghani가 제안한 분산 데이터 관리 패러다임으로, 중앙 집중식 데이터 팀의 병목을 해결합니다.

**4대 원칙**:
1. **Domain Ownership**: 각 부서가 자신의 데이터 소유 및 품질 책임
2. **Data as a Product**: 다른 부서를 "고객"으로, 데이터를 "제품"으로 취급
3. **Self-Service Infrastructure**: 긴 프로세스 없이 데이터 발행/소비 가능
4. **Decentralized Governance**: 보안, 표준화, 상호운용성 정책 수립

**Kafka의 역할**:
- **데이터 교환 플랫폼**: 부서 간 실시간 데이터 발행/소비
- **Schema Registry**: 스키마 계약 관리, 호환성 검증
- **접근 제어**: ACL을 통한 거버넌스 통합
- **셀프서비스**: 토픽 생성, 스키마 등록 등 자동화 지원

**비유**: Data Mesh는 푸드코트와 같습니다. 중앙 관리팀은 건물 인프라와 위생 규정(플랫폼, 거버넌스)만 관리하고, 각 레스토랑(도메인 팀)은 자신의 메뉴(데이터)의 품질을 책임집니다.

---

### Q2: CDC를 통한 데이터 해방의 장단점은?

**모범 답변**:

**CDC (Change Data Capture)**는 코어 시스템의 데이터베이스 변경 로그를 캡처하여 Kafka로 전달하는 방식입니다.

**장점**:
1. **코어 시스템 수정 불필요**: 애플리케이션 코드 변경 없이 데이터 추출
2. **실시간 데이터**: 변경 즉시 Kafka로 전달
3. **데이터 일관성**: 트랜잭션 로그 기반으로 정확한 데이터 캡처
4. **이중 쓰기 방지**: DB만 쓰고 CDC가 자동으로 Kafka에 전달

**단점**:
1. **복잡한 정규화 데이터**: Raw CDC 데이터는 정규화된 형태로 복잡
2. **도메인 전문지식 필요**: 테이블 구조와 비즈니스 의미 이해 필요
3. **변환 작업**: 소비자가 쉽게 사용할 수 있도록 변환 필요
4. **복잡한 조인**: 수백 개 테이블 조인은 Kafka Streams에서 어려움

**권장**:
- 복잡한 조인이 필요하면 **Secondary DB + Outbox 패턴** 사용
- 단순 테이블이면 **Raw CDC → Kafka** 직접 사용

---

### Q3: Kafka가 적합하지 않은 상황 3가지를 말씀해주세요

**모범 답변**:

1. **복잡한 SQL 조인이 필요한 경우**:
   - Kafka는 오프셋 기반 순차 읽기만 지원
   - 복잡한 JOIN, GROUP BY, 서브쿼리 불가
   - RDBMS가 적합

2. **동기식 Request-Response 통신**:
   - Kafka는 비동기 메시징 시스템
   - 사용자가 즉각적인 응답 필요한 경우(잔액 조회 등) 부적합
   - REST API가 적합

3. **대용량 파일 교환**:
   - PDF, 이미지 등 대용량 바이너리 파일 전송에 비효율
   - Object Storage(S3)에 저장하고 Kafka로 메타데이터만 전달

**추가**:
- 소규모 애플리케이션 (시간당 1,000개 미만 메시지, 단일 팀)
- ACID 트랜잭션이 필요한 경우 (완전한 Isolation 필요)

---

### Q4: IIoT에서 MQTT 대신 Kafka를 직접 사용하면 안 되는 이유는?

**모범 답변**:

IIoT 엣지 디바이스에서 Kafka를 직접 사용하기 어려운 이유:

1. **제한된 디바이스 리소스**:
   - 마이크로컨트롤러는 메모리, CPU가 제한적
   - Kafka 클라이언트는 상대적으로 무거움
   - MQTT는 매우 경량 프로토콜 (~10KB)

2. **불안정한 연결**:
   - 공장/차량 환경에서 간헐적 연결
   - 배터리 기반 디바이스
   - MQTT는 QoS 레벨로 불안정한 연결 처리에 최적화

3. **보안 문제**:
   - 신뢰할 수 없는 디바이스에 Kafka 인증 정보 노출 위험
   - MQTT Broker가 게이트웨이 역할로 보안 격리

**권장 아키텍처**:
- 엣지: 센서 → 마이크로컨트롤러 → MQTT Broker
- 중앙: MQTT Broker → Kafka Connector → Kafka → 분석/알림

---

### Q5: Kafka가 좋은 아키텍처를 대체할 수 없다는 의미는?

**모범 답변**:

Kafka는 **도구(Tool)**이지 **만능 솔루션(Silver Bullet)**이 아닙니다.

**Kafka가 해결하지 못하는 것**:

1. **잘못된 데이터 모델**:
   - "Garbage In, Garbage Out"
   - Kafka는 메시지 내용에 관여하지 않음
   - 잘못된 데이터가 들어오면 잘못된 데이터가 나감

2. **불분명한 요구사항**:
   - "일단 Kafka 도입하면 뭔가 좋아지겠지" 접근은 실패
   - 명확한 사용 사례와 요구사항 정의 필수

3. **도메인 이해 부족**:
   - 비즈니스 로직 이해 없이 기술만으로 문제 해결 불가
   - 올바른 토픽 설계, 파티셔닝 전략 필요

4. **아키텍처 문제**:
   - 강결합된 모놀리식을 Kafka로 연결해도 본질적 문제 해결 안됨
   - 먼저 도메인 분리, 책임 정의가 필요

**올바른 접근**:
1. 시스템 요구사항 이해
2. 아키텍처 계획
3. Kafka 적합성 평가
4. 구현

"**Think First, Plan, Then Execute**"
