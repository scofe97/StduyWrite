## 1. 스트림 처리의 본질적 이해

### 1.1 배치 처리와 스트림 처리의 근본적 차이

배치 처리와 스트림 처리는 데이터를 처리하는 시점과 결과의 성격에서 근본적인 차이가 있습니다.

**배치 처리**는 데이터를 축적한 후 특정 시점에 일괄 처리합니다. "어젯밤 11시 59분까지의 모든 거래를 집계하여 일일 매출 보고서를 생성한다"가 전형적인 예입니다. 배치 처리의 결과는 최종적이고 확정적(Definite)입니다. 데이터의 경계가 명확하기 때문에 모든 데이터를 고려한 정확한 결과를 얻을 수 있습니다.

**스트림 처리**는 데이터가 도착하는 즉시 처리합니다. "현재까지의 누적 매출을 실시간으로 대시보드에 표시한다"가 전형적인 예입니다. 스트림 처리의 결과는 중간 결과(Intermediate)이며, 최종적 일관성(Eventually Consistent)을 가집니다. 새로운 데이터가 계속 도착하므로 결과는 계속 갱신됩니다.

```
배치 처리 모델:
┌─────────────────────────────────────────────────┐
│  데이터 축적 (00:00 ~ 23:59)                    │
└─────────────────────────────────────────────────┘
                    ↓
            ┌──────────────┐
            │ 새벽 4시     │
            │ 일괄 처리    │
            └──────────────┘
                    ↓
            ┌──────────────┐
            │ 최종 결과    │
            │ (확정적)     │
            └──────────────┘

스트림 처리 모델:
이벤트 → 처리 → 중간결과 → 이벤트 → 처리 → 갱신된 결과 → ...
  ↑                                              ↑
  실시간                                    최종적 일관성
```

이 차이가 중요한 이유는 사용 사례에 따라 적합한 처리 방식이 다르기 때문입니다. 월간 결산 보고서처럼 정확성이 중요하고 지연이 허용되는 경우 배치가 적합합니다. 사기 탐지처럼 즉각적인 반응이 필요한 경우 스트림이 필수입니다.

### 1.2 스트림 처리 프레임워크 선택

Kafka 기반 스트림 처리를 위한 주요 프레임워크들의 특성을 이해해야 적절한 선택을 할 수 있습니다.

**Kafka Streams**는 Kafka 네이티브 자바 라이브러리입니다. 별도의 클러스터 없이 애플리케이션에 라이브러리로 포함하여 사용합니다. Consumer Group 메커니즘을 활용하여 확장하고, Spring Boot와 자연스럽게 통합됩니다. Kafka만 사용하고 자바/코틀린 팀이라면 가장 적합한 선택입니다.

**Apache Flink**는 범용 스트림 처리 엔진입니다. Kafka 외에도 다양한 소스/싱크를 지원하고, Flink SQL로 선언적 처리가 가능합니다. 별도의 Flink 클러스터가 필요하지만, 더 풍부한 기능(정교한 시간 처리, 복잡한 이벤트 처리 등)을 제공합니다.

**Faust Streaming**은 Python 기반 스트림 처리 라이브러리입니다. Kafka Streams의 핵심 개념을 Python으로 구현했습니다. Python 데이터 과학 팀이나 ML 파이프라인에서 유용합니다.

## 2. Kafka Streams의 프로세서 모델

### 2.1 프로세서 토폴로지의 개념

Kafka Streams 애플리케이션은 프로세서들의 연결인 토폴로지(Topology)로 구성됩니다. 토폴로지는 방향성 비순환 그래프(DAG)로, 데이터가 프로세서에서 프로세서로 흘러갑니다.

토폴로지는 세 종류의 프로세서로 구성됩니다. **Source 프로세서**는 Kafka 토픽에서 데이터를 읽어 토폴로지에 공급합니다. **Stream 프로세서**는 데이터를 변환, 필터링, 집계합니다. **Sink 프로세서**는 처리된 데이터를 Kafka 토픽에 씁니다.

```
간단한 토폴로지 예시:

Source (prices) → Filter (가격 > 100) → Sink (expensive-products)

복잡한 토폴로지 예시:

Source (north) ──┐
                 ├── Merge → GroupByKey → Count → Sink (counts)
Source (south) ──┘
```

### 2.2 무상태 프로세서 (Stateless Processors)

무상태 프로세서는 이전 메시지의 정보 없이 각 메시지를 독립적으로 처리합니다.

**filter**는 조건에 맞는 메시지만 통과시킵니다. 가격이 100원 이상인 상품만 선택하는 것이 예입니다.

**map**은 메시지의 키와 값을 변환합니다. 여기서 중요한 점은 키를 변경하면 Repartitioning이 발생한다는 것입니다. Repartitioning은 메시지가 새 키에 맞는 파티션으로 재분배되는 것으로, 추가적인 네트워크 I/O와 중간 토픽이 필요합니다.

**mapValues**는 값만 변환하고 키는 유지합니다. 키가 변경되지 않으므로 Repartitioning이 발생하지 않아 성능상 유리합니다. 가능하면 map 대신 mapValues를 사용해야 합니다.

**merge**는 두 스트림을 하나로 합칩니다. SQL의 UNION과 유사하게 동작합니다.

**split**은 하나의 스트림을 조건에 따라 여러 브랜치로 분기합니다.

### 2.3 상태 유지 프로세서 (Stateful Processors)

상태 유지 프로세서는 이전 메시지들의 정보를 State Store에 저장하고 활용합니다.

**count**는 키별 메시지 수를 집계합니다. 내부적으로 State Store에 각 키의 카운트를 저장하고, 새 메시지가 도착할 때마다 증가시킵니다.

**aggregate**는 사용자 정의 집계 로직을 수행합니다. 초기값(Initializer)과 누적 로직(Aggregator)을 정의합니다.

**reduce**는 동일 타입의 값들을 하나로 축소합니다. aggregate의 특수한 경우로, 입력과 출력 타입이 동일합니다.

상태 유지 프로세서의 핵심은 State Store입니다. 기본적으로 RocksDB(내장 키-값 저장소)를 사용하며, 모든 상태 변경은 Changelog 토픽에 기록되어 장애 복구에 사용됩니다.

```
상태 유지 프로세서의 내부 동작:

메시지 도착 → State Store 조회 → 상태 갱신 → Changelog 토픽 기록
                     ↓
              (로컬 RocksDB)
```

## 3. KStream과 KTable: 두 가지 데이터 추상화

### 3.1 KStream: 이벤트 스트림

KStream은 무한히 계속되는 레코드의 스트림을 표현합니다. 각 레코드는 독립적인 이벤트로, 같은 키를 가진 여러 레코드가 모두 의미 있습니다.

은행 거래 내역이 전형적인 예입니다. 계좌 A에서 100원 출금, 같은 계좌 A에서 200원 출금, 이 두 이벤트는 모두 별개의 거래로 의미가 있습니다.

```
KStream 예시 (거래 스트림):
(계좌A, 출금 100원) → (계좌A, 입금 500원) → (계좌A, 출금 200원)
     ↑                      ↑                      ↑
  모든 이벤트가 별개의 의미를 가짐
```

### 3.2 KTable: 현재 상태 테이블

KTable은 키별 최신 값을 유지하는 변경 로그(Changelog)를 표현합니다. 같은 키의 새 레코드는 이전 레코드를 대체합니다.

계좌 잔액이 전형적인 예입니다. 계좌 A의 잔액이 1000원에서 900원으로, 다시 1400원으로 변경되면, 현재 관심 있는 것은 최종 잔액인 1400원입니다.

```
KTable 예시 (계좌 잔액):
(계좌A, 1000원) → (계좌A, 900원) → (계좌A, 1400원)
                                         ↑
                               최종 상태만 유지
```

### 3.3 상호 변환과 용도

KStream에서 KTable로 변환은 `groupByKey().aggregate()` 또는 `toTable()`로 수행합니다. 이는 이벤트들을 집계하여 현재 상태를 만드는 것입니다.

KTable에서 KStream으로 변환은 `toStream()`으로 수행합니다. 이는 상태 변경을 이벤트 스트림으로 변환하는 것입니다.

선택 기준은 명확합니다. 개별 이벤트가 모두 의미 있다면 KStream, 키별 최신 상태만 중요하다면 KTable입니다.

## 4. 스트리밍 조인: 실시간 데이터 결합

### 4.1 조인의 세 가지 유형

스트림 처리에서 조인은 배치 처리의 조인과 근본적으로 다릅니다. 데이터가 언제 도착할지 모르기 때문에, 조인 트리거 시점과 시간 범위가 중요합니다.

**Stream-Table Join**은 스트림의 각 레코드를 테이블의 현재 상태와 조인합니다. 스트림에 새 레코드가 도착할 때만 조인이 트리거됩니다. 테이블이 변경되어도 기존 스트림 레코드와 재조인하지 않습니다.

전형적인 사용 사례는 데이터 보강(Enrichment)입니다. 주문 이벤트(스트림)에 고객 정보(테이블)를 추가하는 것이 예입니다.

```
Stream-Table Join:
주문 스트림 ─────┐
                 │→ Join → 보강된 주문
고객 테이블 ─────┘
(스트림 도착 시 트리거)
```

**Table-Table Join**은 두 테이블의 현재 상태를 조인합니다. 어느 쪽 테이블이 변경되어도 조인이 트리거됩니다. 결과도 테이블로, 현재 조인 상태를 유지합니다.

마스터 데이터 결합이 전형적인 예입니다. 고객 테이블과 주소 테이블을 조인하여 완전한 고객 정보 테이블을 유지하는 것입니다.

**Stream-Stream Join**은 두 스트림을 조인합니다. 양쪽 스트림 어느 쪽에 새 레코드가 도착해도 조인이 트리거됩니다. 하지만 무한 스트림을 전부 저장할 수 없으므로, 반드시 시간 윈도우가 필요합니다.

상관관계 분석이 전형적인 예입니다. 가격 변경 이벤트와 판매 이벤트를 1분 윈도우 내에서 조인하여 가격 변경이 판매에 미치는 영향을 분석하는 것입니다.

### 4.2 Co-partitioning 요구사항

모든 조인에는 Co-partitioning 요구사항이 있습니다. 조인되는 양쪽 데이터가 동일한 파티션 수를 가지고, 동일한 파티셔닝 전략으로 키가 분배되어야 합니다.

이 요구사항이 필요한 이유는 Kafka Streams가 파티션 단위로 처리하기 때문입니다. 파티션 0을 처리하는 Task는 양쪽 토픽의 파티션 0만 볼 수 있습니다. 같은 키가 다른 파티션에 있으면 조인이 불가능합니다.

Co-partitioning이 깨지는 경우, `selectKey()`나 `map()`으로 키를 변경한 후 조인하면 Kafka Streams가 자동으로 Repartitioning을 수행합니다. 이는 중간 토픽 생성과 추가 네트워크 I/O를 수반합니다.

## 5. 시간과 윈도우: 스트림 처리의 핵심

### 5.1 시간의 종류 이해

스트림 처리에서 "시간"은 여러 의미를 가질 수 있습니다.

**Event Time**은 이벤트가 실제로 발생한 시간입니다. 애플리케이션이 메시지에 명시적으로 타임스탬프를 포함해야 합니다. 가장 의미 있는 시간이지만, 이벤트가 순서대로 도착하지 않을 수 있습니다.

**Create Time(Ingestion Time)**은 Producer가 메시지를 생성한 시간입니다. Kafka Producer가 자동으로 설정하는 기본값입니다. Event Time을 설정하지 않으면 이 시간이 사용됩니다.

**Log Append Time**은 Broker가 메시지를 로그에 기록한 시간입니다. Broker 설정으로 활성화해야 합니다. 모든 메시지에 일관된 시간을 부여하지만, 실제 이벤트 발생 시간과 차이가 있을 수 있습니다.

**Stream Time**은 Kafka Streams 내부에서 사용하는 시간으로, 지금까지 관찰한 메시지 중 가장 큰 타임스탬프입니다. 윈도우 닫힘과 늦은 데이터 판단에 사용됩니다.

### 5.2 윈도우 유형별 특성

**Tumbling Window**는 고정 크기의 비중첩 윈도우입니다. 매 정각, 매 1분 단위처럼 명확한 경계를 가집니다. 각 이벤트는 정확히 하나의 윈도우에 속합니다.

```
Tumbling Window (1분):
|--Window 1--|--Window 2--|--Window 3--|
00:00~00:59  01:00~01:59  02:00~02:59

사용 사례: 분당 요청 수, 시간대별 매출
```

**Hopping Window**는 고정 크기이지만 중첩될 수 있는 윈도우입니다. 윈도우 크기와 이동 간격(hop)을 별도로 지정합니다. 각 이벤트는 여러 윈도우에 속할 수 있습니다.

```
Hopping Window (크기: 3분, hop: 1분):
|----Window 1----|
     |----Window 2----|
          |----Window 3----|
00:00    01:00    02:00    03:00

사용 사례: 5분 이동 평균을 1분마다 계산
```

**Sliding Window**는 이벤트 도착에 따라 동적으로 생성되는 윈도우입니다. 윈도우 크기만 지정하고, 새 이벤트가 도착할 때마다 해당 이벤트 시간을 기준으로 윈도우가 생성됩니다.

```
Sliding Window (5분):
이벤트 A(02:00) 도착 → 윈도우 [01:55~02:00]
이벤트 B(02:03) 도착 → 윈도우 [01:58~02:03]

사용 사례: "최근 5분간의 이상 거래 건수"
```

**Session Window**는 비활성 간격(inactivity gap)으로 구분되는 동적 윈도우입니다. 이벤트가 계속 도착하면 세션이 연장되고, 지정된 간격 동안 이벤트가 없으면 세션이 종료됩니다.

```
Session Window (비활성 간격: 5분):
|--Session 1--|          |--Session 2--|
  A B C          (gap)     D E F
 (3분 내)       (7분 공백)  (2분 내)

사용 사례: 사용자 세션 분석, 활동 기간 측정
```

### 5.3 늦은 데이터(Late Data) 처리

스트림 처리에서 이벤트가 순서대로 도착하지 않는 것은 흔한 일입니다. 네트워크 지연, 재처리 등으로 이벤트 시간보다 늦게 도착할 수 있습니다.

**Grace Period**는 윈도우가 닫힌 후에도 얼마 동안 늦은 데이터를 허용할지 지정합니다. 예를 들어, 1분 Tumbling Window에 30초 Grace Period를 설정하면, 윈도우가 닫힌 후 30초까지 도착하는 늦은 데이터도 해당 윈도우에 포함됩니다.

Grace Period 이후에 도착하는 데이터는 기본적으로 버려집니다. 이를 처리하려면 별도의 늦은 데이터 처리 로직을 구현해야 합니다.

## 6. Kafka Streams의 확장성과 내결함성

### 6.1 Task와 파티션 기반 병렬 처리

Kafka Streams의 병렬 처리 단위는 Task입니다. Task는 입력 토픽의 파티션과 1:1로 매핑됩니다. 12개의 파티션이 있다면 12개의 Task가 생성됩니다.

Task는 Stream Thread에 할당되고, Thread는 애플리케이션 인스턴스에 할당됩니다. `num.threads` 설정으로 인스턴스당 Thread 수를 조절할 수 있습니다.

```
파티션 12개, 인스턴스 3개, 각 2 Thread:

인스턴스 1:  Thread 1 (Task 0,1), Thread 2 (Task 2,3)
인스턴스 2:  Thread 1 (Task 4,5), Thread 2 (Task 6,7)
인스턴스 3:  Thread 1 (Task 8,9), Thread 2 (Task 10,11)
```

최대 병렬 처리 수준은 입력 토픽의 파티션 수로 제한됩니다. 12개 파티션이면 최대 12개 Task가 병렬로 실행됩니다. 더 많은 인스턴스나 Thread를 추가해도 일부는 유휴 상태가 됩니다.

### 6.2 State Store의 복구와 Standby Replica

상태 유지 프로세서가 있는 Task가 다른 인스턴스로 이동하면, 해당 인스턴스에서 State Store를 복구해야 합니다. 이는 Changelog 토픽을 처음부터 읽어 상태를 재구축하는 과정으로, 대용량 상태의 경우 시간이 오래 걸릴 수 있습니다.

**Standby Replica**는 이 복구 시간을 줄이기 위한 기능입니다. `num.standby.replicas=1`을 설정하면, 각 Task의 State Store 복제본을 다른 인스턴스에 미리 유지합니다. 장애 발생 시 Standby가 즉시 활성화되어 복구 시간이 크게 단축됩니다.

### 6.3 Repartitioning과 Subtopology

키를 변경하는 연산(`selectKey()`, `map()`) 후 상태 유지 연산(집계, 조인)을 수행하면 Repartitioning이 발생합니다. 이는 토폴로지를 두 개의 Subtopology로 분리하고, 그 사이에 중간 토픽을 생성합니다.

```
Repartitioning 발생:

Subtopology 1: Source → selectKey → Repartition Topic
Subtopology 2: Repartition Topic → aggregate → Sink

Repartitioning 방지 (mapValues 사용):

단일 Topology: Source → mapValues → aggregate → Sink
```

Repartitioning은 추가적인 네트워크 I/O, 디스크 I/O, 지연을 발생시키므로, 가능하면 `mapValues()`를 사용하여 키 변경을 피해야 합니다.

## 7. 면접 핵심 질문과 모범 답변

### Q1. KStream과 KTable의 차이점은 무엇인가요?

**모범 답변**: KStream과 KTable은 스트림 데이터를 바라보는 두 가지 관점입니다.

KStream은 이벤트의 연속으로, 각 레코드가 독립적인 의미를 가집니다. 같은 키를 가진 여러 레코드가 모두 의미 있습니다. 은행 거래 내역이 전형적인 예로, 같은 계좌에서의 여러 거래는 모두 별개의 이벤트입니다.

KTable은 키별 최신 상태를 유지하는 테이블입니다. 같은 키의 새 레코드는 이전 레코드를 대체합니다. 계좌 잔액이 전형적인 예로, 관심 있는 것은 현재 잔액이지 모든 변경 이력이 아닙니다.

내부적으로 KTable은 State Store(RocksDB)를 사용하여 각 키의 현재 값을 저장하고, 변경 사항을 Changelog 토픽에 기록합니다. KStream은 상태를 유지하지 않고 레코드를 흘려보냅니다.

### Q2. Stream-Stream Join에 반드시 시간 윈도우가 필요한 이유는 무엇인가요?

**모범 답변**: 두 가지 근본적인 이유가 있습니다.

첫째, 저장 공간의 제한입니다. 스트림은 무한히 계속됩니다. 조인을 위해 한쪽 스트림의 모든 레코드를 저장할 수 없습니다. 시간 윈도우를 지정하면 해당 윈도우 기간의 레코드만 버퍼링하면 됩니다.

둘째, 의미론적 이유입니다. 1년 전 가격 변경과 오늘 판매를 조인하는 것은 의미가 없을 수 있습니다. 시간 윈도우는 "관련 있는" 이벤트의 시간 범위를 정의합니다. 예를 들어, "가격 변경 후 1분 이내의 판매만 상관관계가 있다"고 정의할 수 있습니다.

### Q3. Tumbling Window와 Session Window의 차이와 각각의 사용 사례를 설명해주세요.

**모범 답변**: Tumbling Window는 고정 크기, 비중첩, 연속 윈도우입니다. 0분~1분, 1분~2분처럼 시간이 일정하게 분할됩니다. 각 이벤트는 정확히 하나의 윈도우에 속합니다.

Session Window는 비활성 간격(inactivity gap)으로 구분되는 동적 윈도우입니다. 이벤트가 계속 도착하면 세션이 연장되고, 지정된 간격 동안 이벤트가 없으면 세션이 종료됩니다. 윈도우 크기가 가변적입니다.

사용 사례를 보면, Tumbling Window는 "분당 요청 수", "시간대별 매출"처럼 고정된 시간 단위의 집계에 적합합니다. Session Window는 "사용자별 세션 시간", "활동 기간 분석"처럼 사용자의 연속 활동을 추적할 때 적합합니다.

예를 들어, 사용자가 10:00에 로그인하고 10:05, 10:07에 클릭하고, 10:30에 다시 클릭했다면, 5분 비활성 간격의 Session Window는 두 개의 세션(10:00~10:07, 10:30)을 생성합니다.

### Q4. Kafka Streams에서 상태(State)는 어떻게 관리되고 장애 복구는 어떻게 이루어지나요?

**모범 답변**: Kafka Streams의 상태 관리는 State Store와 Changelog 토픽의 조합으로 이루어집니다.

State Store는 기본적으로 RocksDB를 사용하는 로컬 키-값 저장소입니다. 각 Task가 자신의 State Store를 가지고, 해당 파티션의 상태를 관리합니다. 집계, 조인 등 상태 유지 연산의 결과가 여기에 저장됩니다.

Changelog 토픽은 State Store의 모든 변경을 기록하는 Kafka 토픽입니다. Log Compaction이 적용되어 각 키의 최신 값만 유지됩니다.

장애 복구 시, 새로 할당된 Task는 Changelog 토픽을 읽어 State Store를 재구축합니다. 이 과정을 빠르게 하기 위해 `num.standby.replicas` 설정으로 Standby Replica를 미리 유지할 수 있습니다. Standby는 Changelog를 실시간으로 복제하여, 장애 시 즉시 활성화됩니다.

### Q5. Repartitioning이란 무엇이고 왜 피해야 하나요?

**모범 답변**: Repartitioning은 메시지의 키가 변경되어 새 키에 맞는 파티션으로 재분배되는 것입니다. `selectKey()`, `map()` 등으로 키를 변경한 후 집계나 조인을 수행하면 자동으로 발생합니다.

Repartitioning을 피해야 하는 이유는 세 가지입니다. 첫째, 중간 토픽이 생성되어 추가 저장 공간을 소비합니다. 둘째, 네트워크를 통해 데이터가 재전송되어 I/O 부하가 증가합니다. 셋째, 중간 단계가 추가되어 처리 지연이 발생합니다.

방지 방법은 키 변경이 꼭 필요한지 검토하고, 가능하면 `map()` 대신 `mapValues()`를 사용하는 것입니다. mapValues는 키를 변경하지 않으므로 Repartitioning이 발생하지 않습니다.

## 8. 실무 체크리스트

- [ ] 처리 요구사항에 따라 적절한 윈도우 유형 선택 (Tumbling, Hopping, Sliding, Session)
- [ ] 조인 전 Co-partitioning 요구사항 확인 (파티션 수, 파티셔닝 전략)
- [ ] 가능하면 mapValues 사용하여 Repartitioning 방지
- [ ] num.standby.replicas=1 설정으로 상태 복구 시간 단축
- [ ] Grace Period 설정으로 늦은 데이터 처리 전략 수립
- [ ] 파티션 수 = 최대 병렬 처리 수준 고려한 토픽 설계
- [ ] 처리 지연, 상태 크기, 처리량 등 주요 JMX 메트릭 모니터링
