# Chapter 5: The Docker Engine

---

## 📌 핵심 요약
> Docker Engine은 컨테이너를 실행하고 관리하는 **서버 측 컴포넌트**다. 원래는 모놀리식이었으나, 현재는 **모듈화된 구조**로 분리되어 있다. 핵심 컴포넌트는 **containerd**(고수준 런타임), **runc**(저수준 런타임), **shim**(중간 계층)이며, 이들이 협력하여 컨테이너를 생성하고 관리한다. 이 구조 덕분에 **데몬을 재시작해도 컨테이너가 영향받지 않는다** (Daemonless Containers).

---

이 챕터를 읽고 나면:
- [ ] Docker Engine의 모듈화된 아키텍처를 설명할 수 있다
- [ ] containerd와 runc의 역할 차이를 구분할 수 있다
- [ ] 컨테이너 생성 시 각 컴포넌트의 동작 흐름을 설명할 수 있다
- [ ] shim의 역할과 이점을 이해할 수 있다
- [ ] "Daemonless Containers"의 의미를 설명할 수 있다
- [ ] OCI 표준이 Docker Engine에 미친 영향을 이해할 수 있다

### 5. runc (저수준 런타임)

```
┌─────────────────────────────────────────────────────────┐
│                      runc                                │
│              (발음: "run see")                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   정의: OCI runtime-spec의 참조 구현 (Reference Impl.)   │
│                                                         │
│   역할:                                                  │
│   ┌─────────────────────────────────────────────┐      │
│   │  • OS 커널과 직접 인터페이스                  │      │
│   │  • namespaces 생성 (프로세스 격리)            │      │
│   │  • cgroups 생성 (리소스 제한)                 │      │
│   │  • 컨테이너 생성 후 즉시 종료                 │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
│   특징:                                                  │
│   • libcontainer 기반                                   │
│   • 경량 CLI 래퍼                                       │
│   • 단독 사용 가능하나 기능 제한적                       │
│   • Docker, Kubernetes 모두 사용                        │
│                                                         │
│   계층 표현:                                             │
│   ┌─────────────────────────────────────────────┐      │
│   │  "OCI Layer에서 동작" = Low-Level Runtime    │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
│   GitHub: github.com/opencontainers/runc/releases       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

> 💬 **비유**: runc는 실제로 벽돌을 쌓는 건설 노동자다. 설계도(OCI 번들)를 받아서 실제 건물(컨테이너)을 짓고, 완성되면 현장을 떠난다.

### 6. containerd (고수준 런타임)

```
┌─────────────────────────────────────────────────────────┐
│                    containerd                            │
│              (발음: "container dee")                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   정의: 고수준 런타임 (High-Level Runtime)               │
│                                                         │
│   역할:                                                  │
│   ┌─────────────────────────────────────────────┐      │
│   │  컨테이너 라이프사이클 관리                   │      │
│   │  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐  │      │
│   │  │Start│→│ Run │→│Pause│→│Stop │→│Delete│ │      │
│   │  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘  │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
│   추가 기능 (모듈형):                                    │
│   ┌─────────────────────────────────────────────┐      │
│   │  • 이미지 관리 (push/pull)                   │      │
│   │  • 네트워크 관리                             │      │
│   │  • 볼륨 관리                                 │      │
│   │  • OCI 번들 생성                             │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
│   관계:                                                  │
│   ┌─────────────────────────────────────────────┐      │
│   │  containerd ─────▶ shim ─────▶ runc         │      │
│   │  (관리)           (중개)        (실행)       │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
│   출처: Docker → CNCF 기부 → Graduated 프로젝트         │
│   사용처: Docker, Kubernetes, Firecracker, Fargate      │
│                                                         │
│   GitHub: github.com/containerd/containerd/releases     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

> 💬 **비유**: containerd는 현장 감독(Site Manager)과 같다. 전체 프로젝트를 관리하고, 실제 작업은 runc(노동자)에게 위임한다.

### 8. Shim의 역할

```
┌─────────────────────────────────────────────────────────┐
│                    Shim이란?                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   정의: containerd와 OCI 계층 사이의 중간 컴포넌트       │
│                                                         │
│   ┌─────────────────────────────────────────────┐      │
│   │            containerd                        │      │
│   │                 │                            │      │
│   │                 ▼                            │      │
│   │  ┌─────────────────────────────────────┐    │      │
│   │  │            Shim                      │    │      │
│   │  │   containerd-shim-runc-v2           │    │      │
│   │  │                                     │    │      │
│   │  │   • 컨테이너 부모 프로세스           │    │      │
│   │  │   • STDIN/STDOUT 스트림 유지         │    │      │
│   │  │   • 컨테이너 상태 보고               │    │      │
│   │  │   • 경량 프로세스                    │    │      │
│   │  └──────────────┬──────────────────────┘    │      │
│   │                 │                            │      │
│   │                 ▼                            │      │
│   │         runc (또는 다른 OCI 런타임)          │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Shim의 3대 이점:**

| 이점 | 설명 |
|------|------|
| **Daemonless Containers** | Docker 데몬 재시작해도 컨테이너 영향 없음 |
| **효율성 향상** | runc 종료 후 경량 shim만 유지 |
| **플러그인 가능** | runc 대신 다른 저수준 런타임 사용 가능 (예: Wasm) |

```
┌─────────────────────────────────────────────────────────┐
│              Daemonless Containers                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   기존 (shim 없이):                                      │
│   ┌─────────────────────────────────────────────┐      │
│   │  Daemon ──▶ Container                        │      │
│   │     │                                        │      │
│   │     ❌ Daemon 재시작 → 컨테이너도 영향!        │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
│   현재 (shim 있음):                                      │
│   ┌─────────────────────────────────────────────┐      │
│   │  Daemon ─X─▶ containerd ──▶ shim ──▶ Container│     │
│   │     │              │          │               │     │
│   │  재시작!        독립!      부모 유지!          │     │
│   │                                               │     │
```

```
┌─────────────────────────────────────────────────────────┐
│              각 컴포넌트의 역할 요약                      │

│ Docker Daemon│ API 서비스, 인증, 보안, 네트워킹          │

├──────────────┼──────────────────────────────────────────┤
│ containerd   │ 이미지 관리, 컨테이너 라이프사이클        │
│              │ OCI 번들 생성, High-Level Runtime         │
├──────────────┼──────────────────────────────────────────┤
│ shim         │ 컨테이너 부모 프로세스, 상태 보고         │
│              │ STDIN/STDOUT 유지, daemonless 지원        │
├──────────────┼──────────────────────────────────────────┤
│ runc         │ 커널과 인터페이스, namespaces/cgroups     │
│              │ 컨테이너 생성 후 종료, Low-Level Runtime   │
└──────────────┴──────────────────────────────────────────┘
```

## 💡 실무 적용 포인트

### Quick Reference

| 상황 | 관련 컴포넌트 |
|------|--------------|
| 컨테이너가 시작되지 않음 | runc, containerd 로그 확인 |
| 데몬 업데이트 필요 | shim 덕분에 컨테이너 영향 없음 |
| OCI 호환성 문제 | runc 버전 확인 |
| 이미지 관리 문제 | containerd 상태 확인 |

### 면접 질문

**Q1: containerd와 runc의 차이점은 무엇인가요?**
```
A: 둘 다 런타임이지만 수준이 다릅니다:

   containerd (High-Level Runtime):
   - 컨테이너 라이프사이클 관리 (start, stop, delete)
   - 이미지 관리, OCI 번들 생성
   - gRPC API 제공

   runc (Low-Level Runtime):
   - OS 커널과 직접 인터페이스
   - namespaces, cgroups 생성
   - 컨테이너 생성 후 즉시 종료
   - OCI runtime-spec의 참조 구현

   관계: containerd가 runc를 호출하여 실제 컨테이너 생성
```

**Q2: "Daemonless Containers"란 무엇이고 왜 중요한가요?**
```
A: Docker 데몬을 재시작, 업데이트해도 실행 중인 컨테이너에
   영향을 주지 않는 것을 의미합니다.

   가능한 이유:
   - shim이 컨테이너의 부모 프로세스 역할
   - containerd와 데몬의 분리
   - runc는 생성 후 종료되므로 의존성 없음

   중요한 이유:
   - 운영 환경에서 Docker 업데이트 시 다운타임 없음
   - 컨테이너 안정성 향상
   - 유지보수 용이성 증가
```

**Q3: Docker Engine이 모놀리식에서 모듈화로 변한 이유는?**
```
A: 세 가지 주요 문제점 때문입니다:

   1. 성능 저하: 모든 기능이 하나의 바이너리에 있어 느림
   2. 생태계 요구: 다른 프로젝트가 컴포넌트를 재사용하기 어려움
   3. 혁신 어려움: 한 부분 수정이 전체에 영향

   해결책:
   - 전문화된 작은 도구로 분리
   - containerd, runc 등 독립 프로젝트로 기부
   - Kubernetes, Firecracker 등에서 재사용 가능
```

**Q4: 컨테이너 생성 시 runc가 종료되는 이유는?**
```
A: 효율성과 안정성을 위해서입니다.

   동작 방식:
   1. containerd가 runc 프로세스를 fork
   2. runc가 커널과 통신하여 컨테이너 생성
   3. 컨테이너가 runc의 자식 프로세스로 시작
   4. runc 종료 → shim이 새 부모 프로세스가 됨

   이점:
   - runc 프로세스 유지 불필요 (리소스 절약)
   - 경량 shim만 유지 (효율성)
   - runc 업데이트가 실행 중 컨테이너에 영향 없음
```
