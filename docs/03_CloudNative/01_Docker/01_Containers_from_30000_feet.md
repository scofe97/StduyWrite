# Chapter 1: Containers from 30,000 feet

---

## 📌 핵심 요약
> 컨테이너는 **VM의 한계를 극복**하기 위해 등장했다. VM이 각각 독립된 OS를 필요로 하는 반면, 컨테이너는 **호스트 OS 커널을 공유**하여 더 가볍고, 빠르고, 이식성이 높다. Docker는 복잡했던 Linux 컨테이너를 **대중화**시킨 핵심 기술이며, 현재 Wasm과 AI 생태계와도 통합되며 진화하고 있다.

---

## 🎯 학습 목표
이 챕터를 읽고 나면:
- [ ] 물리 서버 → VM → 컨테이너로 이어지는 인프라 진화 과정을 설명할 수 있다
- [ ] VM과 컨테이너의 핵심 차이점을 이해할 수 있다
- [ ] Docker의 역할과 의의를 설명할 수 있다
- [ ] Windows/Linux/Mac에서의 컨테이너 동작 방식을 구분할 수 있다
- [ ] Wasm과 컨테이너의 관계를 이해할 수 있다

## 📖 본문 정리

### 1. 인프라 진화의 역사

#### 1.1 The Bad Old Days: 물리 서버 시대

```
┌─────────────────────────────────────────────────────────┐
│                    물리 서버 시대                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   [Server 1]      [Server 2]      [Server 3]           │
│   ┌─────────┐     ┌─────────┐     ┌─────────┐          │
│   │  App A  │     │  App B  │     │  App C  │          │
│   ├─────────┤     ├─────────┤     ├─────────┤          │
│   │   OS    │     │   OS    │     │   OS    │          │
│   ├─────────┤     ├─────────┤     ├─────────┤          │
│   │Hardware │     │Hardware │     │Hardware │          │
│   └─────────┘     └─────────┘     └─────────┘          │
│                                                         │
│   사용률: 5~10%    사용률: 5~10%   사용률: 5~10%         │
│                                                         │
│   💰 비용 낭비 + 🌍 환경 자원 낭비                        │
└─────────────────────────────────────────────────────────┘
```

**문제점:**
- 1개 애플리케이션 = 1개 서버 (강제)
- 성능 예측 어려움 → 과도한 스펙의 서버 구매
- 서버 활용률 5~10% → **자본 및 환경 자원 낭비**

> 💬 **비유**: 10명이 탈 수 있는 버스에 1명만 태우고 다니는 것과 같다.

---

#### 1.2 Hello VMware: 가상화 시대

```
┌─────────────────────────────────────────────────────────┐
│                    VM (가상화) 시대                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│           [Single Physical Server]                      │
│   ┌───────────────────────────────────────────┐        │
│   │  ┌─────────┐ ┌─────────┐ ┌─────────┐     │        │
│   │  │  App A  │ │  App B  │ │  App C  │     │        │
│   │  ├─────────┤ ├─────────┤ ├─────────┤     │        │
│   │  │  OS 1   │ │  OS 2   │ │  OS 3   │     │        │
│   │  └────┬────┘ └────┬────┘ └────┬────┘     │        │
│   │       └───────────┼───────────┘          │        │
│   │              Hypervisor                   │        │
│   ├───────────────────────────────────────────┤        │
│   │              Host OS                      │        │
│   ├───────────────────────────────────────────┤        │
│   │              Hardware                     │        │
│   └───────────────────────────────────────────┘        │
│                                                         │
│   ✅ 서버 통합 성공! 하지만...                           │
└─────────────────────────────────────────────────────────┘
```

**VMware의 기여:**
- 단일 서버에서 여러 애플리케이션 실행 가능
- 기존 자산의 가치 극대화
- **Game Changer!**

**VMwarts (VM의 한계):**

| 문제점 | 설명 |
|--------|------|
| **OS 오버헤드** | 각 VM마다 완전한 OS 필요 → CPU, RAM 낭비 |
| **패치 부담** | 모든 VM + OS를 개별 패치해야 함 |
| **모니터링 복잡성** | 각 VM + OS를 개별 모니터링 |
| **느린 부팅** | 전체 OS 부팅 필요 (분 단위) |
| **이식성 제한** | VM 이미지가 크고 이동이 어려움 |

> 💬 **비유**: VM은 아파트를 통째로 빌리는 것. 주방, 화장실, 거실이 다 필요 없어도 다 가져야 한다.

#### 1.3 Hello Containers: 컨테이너 시대

```
┌─────────────────────────────────────────────────────────┐
│                    Container 시대                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│           [Single Physical/Virtual Host]                │
│   ┌───────────────────────────────────────────┐        │
│   │  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ │        │
│   │  │App A│ │App B│ │App C│ │App D│ │App E│ │        │
│   │  └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘ │        │
│   │     └───────┴───────┼───────┴───────┘    │        │
│   │              Container Runtime            │        │
│   ├───────────────────────────────────────────┤        │
│   │         Shared Host OS Kernel             │ ← 핵심!│
│   ├───────────────────────────────────────────┤        │
│   │              Hardware                     │        │
│   └───────────────────────────────────────────┘        │
│                                                         │
│   VM 10개 실행 가능한 호스트 → 컨테이너 50개 실행 가능!    │
└─────────────────────────────────────────────────────────┘
```

**컨테이너의 핵심 특징:**
- **커널 공유**: 모든 컨테이너가 호스트 OS 커널 공유
- **경량화**: OS 오버헤드 없음
- **빠른 시작**: 초 단위 부팅
- **높은 이식성**: 어디서든 동일하게 실행

**Linux 컨테이너 핵심 기술:**

| 기술 | 역할 |
|------|------|
| **Kernel Namespaces** | 프로세스 격리 (PID, Network, Mount 등) |
| **Control Groups (cgroups)** | 리소스 제한 (CPU, Memory, I/O) |
| **Capabilities** | 권한 세분화 |

> 💬 **비유**: 컨테이너는 호텔 방을 빌리는 것. 로비, 수영장, 레스토랑은 공유하면서 내 방만 독립적으로 사용.

### 2. VM vs Container 비교

```
┌─────────────────────────────────────────────────────────┐
│                   VM vs Container                        │
├────────────────────────┬────────────────────────────────┤
│          VM            │          Container             │
├────────────────────────┼────────────────────────────────┤
│                        │                                │
│  ┌──────────────────┐  │  ┌──────────────────┐         │
│  │      App         │  │  │      App         │         │
│  ├──────────────────┤  │  ├──────────────────┤         │
│  │    Guest OS      │  │  │   (No Guest OS)  │         │
│  │   (Full OS)      │  │  │                  │         │
│  ├──────────────────┤  │  └────────┬─────────┘         │
│  │   Hypervisor     │  │           │                   │
│  └────────┬─────────┘  │           │                   │
│           │            │           │                   │
│  ┌────────┴─────────┐  │  ┌────────┴─────────┐         │
│  │     Host OS      │  │  │Container Runtime │         │
│  ├──────────────────┤  │  ├──────────────────┤         │
│  │    Hardware      │  │  │     Host OS      │         │
│  └──────────────────┘  │  ├──────────────────┤         │
│                        │  │    Hardware      │         │
│                        │  └──────────────────┘         │
├────────────────────────┼────────────────────────────────┤
│  💾 GB 단위 이미지     │  📦 MB 단위 이미지              │
│  ⏱️ 분 단위 부팅       │  ⚡ 초 단위 부팅                │
│  🔧 하드웨어 수준 격리  │  🔧 프로세스 수준 격리          │
│  🖥️ 다양한 OS 지원     │  🐧 호스트 커널 의존            │
└────────────────────────┴────────────────────────────────┘
```

| 항목 | VM | Container |
|------|-----|-----------|
| **격리 수준** | 하드웨어 가상화 | OS 수준 가상화 |
| **OS** | 각 VM에 완전한 OS | 호스트 커널 공유 |
| **이미지 크기** | GB 단위 | MB 단위 |
| **부팅 시간** | 분 단위 | 초 단위 |
| **밀도** | 낮음 (호스트당 ~10개) | 높음 (호스트당 ~50+개) |
| **이식성** | 낮음 | 높음 |
| **리소스 효율** | 낮음 | 높음 |

### 3. Docker의 등장

```
┌─────────────────────────────────────────────────────────┐
│                    Docker의 역할                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   Before Docker:                                        │
│   ┌─────────────────────────────────────────────┐      │
│   │  Linux 컨테이너 = 복잡 + 어려움 + 전문가 전용  │      │
│   │                                             │      │
│   │  Namespaces + Cgroups + Capabilities + ...  │      │
│   │  = 😵 일반 개발자 접근 불가                   │      │
│   └─────────────────────────────────────────────┘      │
│                        │                                │
│                        ▼                                │
│   After Docker:                                         │
│   ┌─────────────────────────────────────────────┐      │
│   │            docker run nginx                 │      │
│   │                   ✅                         │      │
│   │         누구나 컨테이너 사용 가능!            │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
│   Docker = 컨테이너의 대중화를 이끈 "마법"              │
└─────────────────────────────────────────────────────────┘
```

**Docker의 기여:**
- 복잡한 Linux 컨테이너 기술을 **단순한 CLI**로 추상화
- 이미지 빌드, 공유, 배포의 **표준화**
- 개발자 친화적인 워크플로우 제공

> 💬 **비유**: Docker는 복잡한 자동차 엔진을 운전대, 페달, 기어로 추상화한 것. 엔진 작동 원리를 몰라도 운전할 수 있다.

### 4. 플랫폼별 컨테이너 지원

```
┌─────────────────────────────────────────────────────────┐
│              플랫폼별 컨테이너 지원 현황                   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │   Windows   │  │    Linux    │  │     Mac     │     │
│  ├─────────────┤  ├─────────────┤  ├─────────────┤     │
│  │             │  │             │  │             │     │
│  │ ✅ Windows  │  │ ✅ Linux    │  │ ❌ Mac      │     │
│  │  Containers │  │  Containers │  │  Containers │     │
│  │             │  │             │  │             │     │
│  │ ✅ Linux    │  │             │  │ ✅ Linux    │     │
│  │  (via WSL2) │  │             │  │  (via VM)   │     │
│  │             │  │             │  │             │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 핵심 원칙: 커널 공유
> **컨테이너는 호스트 커널을 공유한다!**
> - Linux 앱 → Linux 커널 필요
> - Windows 앱 → Windows 커널 필요

| 플랫폼 | Windows 컨테이너 | Linux 컨테이너 | 비고 |
|--------|-----------------|---------------|------|
| **Windows** | ✅ 네이티브 | ✅ WSL 2 | Win 10/11, Server |
| **Linux** | ❌ | ✅ 네이티브 | 기본 환경 |
| **Mac** | ❌ | ✅ VM 기반 | Docker Desktop, Podman |

**현실:**
- 거의 모든 컨테이너는 **Linux 컨테이너**
- Linux 컨테이너가 더 작고, 빠르고, 도구가 풍부함
- 이 책의 모든 예제는 Linux 컨테이너 기반

### 면접 질문

**Q1: VM과 컨테이너의 가장 큰 차이점은 무엇인가요?**
```
A: VM은 각각 완전한 Guest OS를 가지며 하이퍼바이저 위에서 실행됩니다.
   컨테이너는 호스트 OS 커널을 공유하여 OS 오버헤드가 없습니다.

   결과적으로 컨테이너는:
   - 더 작고 (MB vs GB)
   - 더 빠르고 (초 vs 분)
   - 더 효율적입니다 (호스트당 더 많은 인스턴스)
```

**Q2: 왜 Linux 컨테이너는 Windows에서 직접 실행할 수 없나요?**
```
A: 컨테이너는 호스트의 커널을 공유하기 때문입니다.
   Linux 컨테이너는 Linux 커널 기능(namespaces, cgroups)이 필요하므로
   Windows에서는 WSL 2를 통해 Linux 커널을 에뮬레이션해야 합니다.
```

**Q3: Docker가 컨테이너 기술에 기여한 가장 큰 점은 무엇인가요?**
```
A: Docker는 복잡한 Linux 컨테이너 기술을 단순한 CLI로 추상화하여
   일반 개발자도 컨테이너를 쉽게 사용할 수 있게 했습니다.
   또한 이미지 빌드, 공유, 배포의 표준을 만들어 생태계를 구축했습니다.
```

**Q4: 컨테이너와 Wasm의 관계는 어떻게 되나요?**
```
A: Wasm은 컨테이너보다 더 작고, 빠르고, 이식성이 높은 바이너리 포맷입니다.
   현재는 생태계가 미성숙하지만, 미래에는 VM, Container, Wasm이
   각각의 영역에서 공존할 것으로 예상됩니다.
   Docker 생태계는 이미 Wasm 지원을 시작했습니다.
```
