# Chapter 2: Docker and Container-related Standards and Projects

---

## 📌 핵심 요약
> Docker는 **Docker, Inc.** (회사)와 **Docker 플랫폼** (기술) 두 가지를 의미한다. 플랫폼은 **CLI(클라이언트)**와 **Engine(서버)**로 구성되며, 복잡한 내부 구조를 단순한 명령어로 추상화한다. 컨테이너 생태계는 **OCI** (저수준 표준), **CNCF** (프로젝트 호스팅), **Moby Project** (플랫폼 빌딩 도구)라는 세 축으로 거버넌스된다.

---

- [ ] Docker, Inc.의 역사와 Docker라는 용어의 두 가지 의미를 설명할 수 있다
- [ ] Docker 플랫폼의 고수준 아키텍처(CLI + Engine)를 이해할 수 있다

- [ ] CNCF의 역할과 프로젝트 성숙도 단계를 이해할 수 있다
- [ ] Moby Project의 목적을 설명할 수 있다

### 3. Docker 플랫폼 아키텍처

#### 3.1 고수준 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│              Docker 플랫폼 고수준 아키텍처               │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌─────────────────┐                                   │
│   │  Developer/User │                                   │
│   └────────┬────────┘                                   │
│            │ docker run, build, push...                 │
│            ▼                                            │
│   ┌─────────────────┐          ┌─────────────────┐     │
│   │   Docker CLI    │   API    │  Docker Engine  │     │
│   │   (Client)      │ ──────▶  │   (Server)      │     │
│   │                 │  REST    │                 │     │
│   └─────────────────┘          └─────────────────┘     │
│                                        │                │
│                                        ▼                │
│                               ┌─────────────────┐      │
│                               │   Containers    │      │
│                               │   🐳 🐳 🐳      │      │
│                               └─────────────────┘      │
│                                                         │
│   💡 CLI와 Engine은 같은 호스트 또는 네트워크로 연결     │
└─────────────────────────────────────────────────────────┘
```

| 컴포넌트 | 역할 | 위치 |
|----------|------|------|
| **CLI (Client)** | 사용자 명령어 → API 요청 변환 | 로컬 또는 원격 |
| **Engine (Server)** | 컨테이너 실행 및 관리 | 호스트 시스템 |

### 5. OCI (Open Container Initiative)

#### 5.1 OCI 탄생 배경

```
┌─────────────────────────────────────────────────────────┐
│                    OCI 탄생 배경                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   초기 상황:                                            │
│   ┌─────────────────────────────────────────────┐      │
│   │  Docker가 컨테이너 생태계를 지배             │      │
│   │              │                               │      │
│   │              ▼                               │      │
│   │  CoreOS 회사: "Docker 독점이 싫다!" 😤       │      │
│   │              │                               │      │
│   │              ▼                               │      │
│   │  appc 표준 + rkt(rocket) 런타임 개발         │      │
│   └─────────────────────────────────────────────┘      │
│                        │                                │
│                        ▼                                │
│   문제 발생:                                            │
│   ┌─────────────────────────────────────────────┐      │
│   │                                             │      │
│   │    Docker 방식  vs  appc 방식               │      │
│   │         ⚔️ 경쟁 표준 ⚔️                      │      │
│   │                                             │      │
│   │  → 혼란 발생, 사용자 채택 저하               │      │
│   └─────────────────────────────────────────────┘      │
│                        │                                │
│                        ▼                                │
│   해결:                                                 │
│   ┌─────────────────────────────────────────────┐      │
│   │  주요 플레이어들이 모여 OCI 설립 (2015)      │      │
│   │                                             │      │
│   │  • 벤더 중립적 (vendor-neutral)             │      │
│   │  • 경량 거버넌스                            │      │
│   │  • appc 프로젝트 아카이브                   │      │
│   │  • 모든 저수준 표준을 OCI 하에 통합          │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

> 💬 **비유 (철도 표준)**: 철도 궤간(레일 간격)이 표준화되자, 기업가들은 자신이 만든 기차, 객차, 신호 시스템이 표준 레일에서 작동할 것임을 확신할 수 있었다. OCI가 컨테이너 생태계에 같은 역할을 했다.

**Docker의 OCI 표준 구현:**

| OCI 표준 | Docker 구현 |
|----------|-------------|

| **runtime-spec** | runc → OCI 호환 런타임 사용 |
| **distribution-spec** | Docker Hub → OCI 호환 레지스트리 |

### 6. CNCF (Cloud Native Computing Foundation)

```
┌─────────────────────────────────────────────────────────┐
│                         CNCF                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   설립: 2015년                                          │
│   미션: "클라우드 네이티브 컴퓨팅을 어디서나 가능하게"    │
│                                                         │
│   역할: 표준 X → 프로젝트 호스팅 및 성숙화 지원 ✓        │
│                                                         │
├─────────────────────────────────────────────────────────┤
│                 프로젝트 성숙도 단계                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│      Sandbox          Incubating         Graduated      │
│   ┌───────────┐    ┌───────────┐    ┌───────────┐      │
│   │  🧪       │ ─▶ │  🌱       │ ─▶ │  🎓       │      │
│   │ 초기 단계 │    │ 성장 단계 │    │ 성숙 단계 │      │
│   │           │    │           │    │           │      │
│   │ • 좋은    │    │ • 커뮤니티│    │ • 프로덕  │      │
│   │   아이디어│    │   성장 중 │    │   션 준비 │      │
│   │ • 도움    │    │ • 거버넌스│    │ • 강력한  │      │
│   │   필요    │    │   개선 중 │    │   거버넌스│      │
│   └───────────┘    └───────────┘    └───────────┘      │
│                                                         │
│   성숙도 요구사항:                                       │
│   • 거버넌스 강화                                        │
│   • 문서화                                              │
│   • 보안 감사                                           │
│   • 기여 추적                                           │
│   • 커뮤니티 참여                                        │
└─────────────────────────────────────────────────────────┘
```

**주요 CNCF 프로젝트:**

| 프로젝트 | 역할 | 상태 |
|----------|------|------|
| **Kubernetes** | 컨테이너 오케스트레이션 | Graduated |
| **containerd** | 컨테이너 런타임 | Graduated |
| **Prometheus** | 모니터링 | Graduated |
| **Cilium** | 네트워킹/보안 | Graduated |
| **Notary** | 이미지 서명/신뢰 | Incubating |

> 💬 **비유**: CNCF는 스타트업 인큐베이터와 같다. 좋은 아이디어를 가진 프로젝트가 성장하고 성숙해질 수 있도록 공간, 구조, 지원을 제공한다.

### 면접 질문

**Q1: Docker라는 용어의 두 가지 의미를 설명해주세요.**
```
A: Docker는 두 가지를 의미합니다:

1. Docker, Inc.: 캘리포니아 Palo Alto에 본사를 둔 기술 회사로,
   Solomon Hykes가 설립했습니다. 원래 dotCloud라는 PaaS 회사였으나
   2013년 Docker에 집중하기 위해 리브랜딩했습니다.

2. Docker Platform: 컨테이너를 생성, 관리, 오케스트레이션하는
   기술 플랫폼입니다. CLI(클라이언트)와 Engine(서버)으로 구성됩니다.
```

**Q2: OCI는 왜 만들어졌고, 어떤 표준을 관리하나요?**
```
A: OCI는 Docker 독점에 대한 우려로 CoreOS가 appc 표준을 만들면서
   경쟁 표준이 생기자, 혼란을 방지하기 위해 2015년에 설립되었습니다.

   관리하는 3대 표준:
   1. image-spec: 컨테이너 이미지 포맷
   2. runtime-spec: 컨테이너 런타임 동작 방식
   3. distribution-spec: 이미지 배포/레지스트리 API

   OCI 덕분에 다양한 도구들이 상호 호환됩니다.
```

**Q3: CNCF의 역할은 무엇이고, 프로젝트 성숙도 단계를 설명해주세요.**
```
A: CNCF는 클라우드 네이티브 프로젝트를 호스팅하고 성숙시키는 역할을 합니다.
   (표준을 만드는 것이 아님)

   성숙도 3단계:
   1. Sandbox: 초기 단계, 좋은 아이디어지만 지원 필요
   2. Incubating: 성장 중, 거버넌스 개선 중
   3. Graduated: 프로덕션 준비 완료, 강력한 거버넌스

   Kubernetes, containerd, Prometheus 등이 Graduated 프로젝트입니다.
```

**Q4: Docker 플랫폼은 어떤 프로젝트들의 기술로 구성되어 있나요?**
```
A: Docker 플랫폼은 여러 프로젝트의 기술을 조합합니다:

   • Docker 자체: Daemon, CLI
   • Moby Project: BuildKit (이미지 빌드)
   • CNCF: containerd (컨테이너 라이프사이클), Notary (이미지 서명)
   • OCI: runc (저수준 런타임)

   이는 Docker가 개방형 생태계와 협력하여 발전함을 보여줍니다.
```

## ✅ 체크리스트

### Docker 이해
- [ ] Docker, Inc.와 Docker Platform의 차이를 설명할 수 있다
- [ ] dotCloud에서 Docker, Inc.로의 전환 배경을 알고 있다
- [ ] Docker CLI와 Engine의 역할을 구분할 수 있다
- [ ] Docker Engine 내부의 주요 컴포넌트를 나열할 수 있다

### OCI 이해
- [ ] OCI 설립 배경(CoreOS, appc, 경쟁 표준 문제)을 설명할 수 있다
- [ ] OCI 3대 표준(image, runtime, distribution)을 설명할 수 있다
- [ ] 철도 궤간 비유로 표준화의 중요성을 설명할 수 있다

### CNCF 이해
- [ ] CNCF의 역할(프로젝트 호스팅, 성숙화)을 설명할 수 있다
- [ ] Sandbox → Incubating → Graduated 단계를 설명할 수 있다
- [ ] containerd, Kubernetes 등 주요 CNCF 프로젝트를 알고 있다

### Moby Project 이해
- [ ] Moby Project의 목적(플랫폼 빌더용 도구)을 설명할 수 있다
- [ ] Docker 플랫폼이 Moby, CNCF, OCI 기술을 조합함을 이해한다
