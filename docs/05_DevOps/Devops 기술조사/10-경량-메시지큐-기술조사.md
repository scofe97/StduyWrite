## 2. 경량 메시지 큐 후보

### 2.1 NATS

#### 개요
- **개발사**: Synadia (CNCF 프로젝트)
- **언어**: Go
- **특징**: 초경량, 클라우드 네이티브 설계

#### 기술 스펙
| 항목 | 내용 |
|------|------|
| 메모리 사용량 | ~10-20MB |
| 처리량 | 수십만 msg/s |
| 지연시간 | < 1ms |
| 영속성 | JetStream 추가 시 지원 |
| 프로토콜 | TCP, WebSocket |

#### Spring Boot 통합
```gradle
implementation 'io.nats:jnats:2.17.+'
```

```java
@Configuration
public class NatsConfig {
    @Bean
    public Connection natsConnection() throws IOException {
        Options options = new Options.Builder()
            .server("nats://localhost:4222")
            .build();
        return Nats.connect(options);
    }
}

@Service
public class NatsPublisher {
    private final Connection nats;

    public void publish(String subject, String message) {
        nats.publish(subject, message.getBytes(StandardCharsets.UTF_8));
    }
}

@Service
public class NatsSubscriber {
    @PostConstruct
    public void subscribe() {
        Dispatcher dispatcher = nats.createDispatcher(msg -> {
            String data = new String(msg.getData(), StandardCharsets.UTF_8);
            // 메시지 처리
        });
        dispatcher.subscribe("events.>");
    }
}
```

#### Docker 배포
```yaml
# docker-compose.yml
services:
  nats:
    image: nats:latest
    ports:
      - "4222:4222"   # Client
      - "8222:8222"   # Monitoring
    command: ["--jetstream"]  # 영속성 활성화
```

#### 장점
- 설정 최소화, 바로 사용 가능
- 클러스터링 간단
- 다양한 메시징 패턴 지원 (Pub/Sub, Request/Reply, Queue Groups)

#### 단점
- 기본 영속성 없음 (JetStream 필요)
- Kafka 대비 생태계 작음

### 2.2 Redis Streams

#### 개요
- **개발사**: Redis Ltd.
- **특징**: Redis 5.0+에 포함된 스트림 데이터 타입
- **장점**: 이미 Redis 사용 중이면 추가 인프라 불필요

#### 기술 스펙
| 항목 | 내용 |
|------|------|
| 메모리 사용량 | Redis 인스턴스에 포함 (~50MB+) |
| 처리량 | 수만 msg/s |
| 영속성 | RDB/AOF 지원 |
| Consumer Group | 지원 |

#### Spring Boot 통합
```gradle
implementation 'org.springframework.boot:spring-boot-starter-data-redis'
```

```java
@Configuration
public class RedisStreamConfig {
    @Bean
    public RedisTemplate<String, String> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, String> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(new StringRedisSerializer());
        return template;
    }
}

@Service
public class RedisStreamPublisher {
    private final StringRedisTemplate redisTemplate;

    public void publish(String streamKey, Map<String, String> message) {
        redisTemplate.opsForStream().add(streamKey, message);
    }
}

@Component
public class RedisStreamConsumer implements StreamListener<String, MapRecord<String, String, String>> {
    @Override
    public void onMessage(MapRecord<String, String, String> message) {
        Map<String, String> value = message.getValue();
        // 메시지 처리
    }
}
```

#### Docker 배포
```yaml
services:
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    command: redis-server --appendonly yes
```

#### 장점
- 캐시 + 큐 통합 운영
- Spring Data Redis로 쉬운 통합
- Consumer Group으로 분산 처리

#### 단점
- 메모리 기반 (대용량 시 비용 증가)
- 메시지 리플레이 제한적

### 2.3 ZeroMQ (0MQ)

#### 개요
- **특징**: 브로커리스 (Brokerless) 메시징 라이브러리
- **언어 바인딩**: Java, Python, C++, Go 등 다수

#### 기술 스펙
| 항목 | 내용 |
|------|------|
| 메모리 사용량 | 애플리케이션 임베디드 |
| 처리량 | 2.8M msg/s (10byte) |
| 지연시간 | ~33μs |
| 영속성 | 없음 |
| 브로커 | 불필요 |

#### Java 사용 예시
```gradle
implementation 'org.zeromq:jeromq:0.5.+'
```

```java
// Publisher
try (ZContext context = new ZContext()) {
    ZMQ.Socket publisher = context.createSocket(SocketType.PUB);
    publisher.bind("tcp://*:5555");

    publisher.send("topic message");
}

// Subscriber
try (ZContext context = new ZContext()) {
    ZMQ.Socket subscriber = context.createSocket(SocketType.SUB);
    subscriber.connect("tcp://localhost:5555");
    subscriber.subscribe("topic");

    String message = subscriber.recvStr();
}
```

#### 장점
- 최고 성능, 최저 지연
- 별도 서버 불필요
- 다양한 패턴 (PUB/SUB, PUSH/PULL, REQ/REP)

#### 단점
- Spring 통합 라이브러리 부족
- 브로커 기능 직접 구현 필요
- 운영 모니터링 어려움

### 2.4 ActiveMQ Artemis

#### 개요
- **개발사**: Apache
- **특징**: ActiveMQ 차세대 버전, 임베디드 모드 지원

#### 기술 스펙
| 항목 | 내용 |
|------|------|
| 메모리 사용량 | ~30-50MB |
| 프로토콜 | AMQP, STOMP, MQTT, OpenWire |
| 영속성 | 지원 |
| JMS | 완벽 지원 |

#### Spring Boot 통합
```gradle
implementation 'org.springframework.boot:spring-boot-starter-artemis'
```

```java
// application.yml
spring:
  artemis:
    mode: embedded  # 임베디드 모드
    embedded:
      enabled: true
      queues: my-queue

@Service
public class ArtemisPublisher {
    private final JmsTemplate jmsTemplate;

    public void send(String queue, String message) {
        jmsTemplate.convertAndSend(queue, message);
    }
}

@Component
public class ArtemisConsumer {
    @JmsListener(destination = "my-queue")
    public void receive(String message) {
        // 메시지 처리
    }
}
```

#### 장점
- Spring Boot 공식 지원
- JMS 표준 준수
- 임베디드 모드로 별도 서버 불필요

#### 단점
- NATS/Redis 대비 무거움
- 클러스터링 복잡

### 2.5 NSQ

#### 개요
- **개발사**: Bitly
- **언어**: Go
- **특징**: 분산 실시간 메시징 플랫폼

#### 기술 스펙
| 항목 | 내용 |
|------|------|
| 메모리 사용량 | ~30MB |
| 처리량 | 수십만 msg/s |
| 영속성 | 디스크 백업 지원 |
| 단일 장애점 | 없음 |

#### 장점
- 수평 확장 용이
- 관리 UI 내장 (nsqadmin)
- Go 클라이언트 우수

#### 단점
- Java 클라이언트 미성숙
- Spring 통합 부족

## 3. Kafka 대비 경량 MQ 한계점

### 3.1 기능 비교표

| 기능 | Kafka | NATS | Redis Streams | ZeroMQ | Artemis |
|------|-------|------|---------------|--------|---------|
| **메시지 영속성** | ✅ 무제한 | △ JetStream | △ 메모리 | ❌ | ✅ |
| **메시지 리플레이** | ✅ offset 기반 | △ 제한적 | △ 제한적 | ❌ | ❌ |
| **파티셔닝** | ✅ | △ | ❌ | ❌ | ❌ |
| **Consumer Group** | ✅ | ✅ | ✅ | ❌ | ✅ |
| **Exactly-once** | ✅ | ❌ | ❌ | ❌ | △ |
| **순서 보장** | ✅ 파티션 내 | ❌ | ✅ 단일 | ❌ | ✅ |
| **처리량** | 수백만/s | 수십만/s | 수만/s | 수백만/s | 수만/s |
| **데이터 보존** | 일/주/월 | 제한적 | 메모리 | 없음 | 설정 |

### 3.2 Kafka 필수 기능 상세

#### 메시지 리플레이 (Replay)
```
Kafka: Consumer가 offset을 조정해 과거 메시지 재처리 가능
       → 장애 복구, 데이터 재처리에 필수

경량 MQ: 소비된 메시지는 삭제됨 → 재처리 불가
```

**영향**:
- 버그 수정 후 데이터 재처리 불가
- 감사 로그 구현 어려움

#### 파티셔닝 (Partitioning)
```
Kafka: 토픽을 N개 파티션으로 분할
       → N개 Consumer가 병렬 처리
       → 선형적 수평 확장

경량 MQ: 단일 큐 또는 수동 샤딩 필요
```

**영향**:
- 처리량 한계 존재
- 확장 시 아키텍처 변경 필요

#### Exactly-Once 시맨틱
```
Kafka: 트랜잭션으로 정확히 한 번 처리 보장
       → 금융, 결제 시스템에 필수

경량 MQ: At-least-once 또는 At-most-once
       → 중복 처리 또는 유실 가능
```

**영향**:
- 멱등성 처리 로직 필수
- 데이터 정합성 위험

#### 멀티 컨슈머 독립 소비
```
Kafka: 같은 토픽을 여러 서비스가 각자 속도로 소비
       Service A: offset 1000 (실시간)
       Service B: offset 500 (배치)

경량 MQ: Fan-out 별도 구현 필요
```

**영향**:
- 서비스별 토픽 분리 또는 복제 필요
- 아키텍처 복잡도 증가

### 3.3 Kafka 필수 판단 기준

| 요구사항 | Kafka 필수 | 경량 MQ 가능 |
|----------|-----------|-------------|
| 일 수백만 건 이상 처리 | ✅ | |
| 메시지 재처리/감사 로그 | ✅ | |
| 이벤트 소싱 아키텍처 | ✅ | |
| DB CDC (Change Data Capture) | ✅ | |
| 데이터 파이프라인 (ETL) | ✅ | |
| 단순 작업 큐/알림 | | ✅ |
| 마이크로서비스 간 단순 통신 | | ✅ |
| 웹훅 처리 | | ✅ |
| CI/CD 파이프라인 트리거 | | ✅ |

## 4. 기술 선정 가이드

### 4.1 의사결정 플로우차트

```
메시지 재처리 필요?
├─ Yes → Kafka
└─ No
   └─ 일 처리량 100만 건 이상?
      ├─ Yes → Kafka 또는 NATS + JetStream
      └─ No
         └─ 이미 Redis 사용 중?
            ├─ Yes → Redis Streams
            └─ No
               └─ JMS 표준 필요?
                  ├─ Yes → ActiveMQ Artemis
                  └─ No → NATS
```

### 4.2 Jenkins/GitLab 미들웨어 통신 추천

#### 아키텍처 예시
```
┌─────────────┐     Webhook      ┌─────────────────┐
│   GitLab    │ ───────────────→ │  Spring Boot    │
└─────────────┘                  │  (API Server)   │
                                 └────────┬────────┘
                                          │ Publish
                                          ▼
                                 ┌─────────────────┐
                                 │      NATS       │
                                 │   (Message Q)   │
                                 └────────┬────────┘
                                          │ Subscribe
              ┌───────────────────────────┼───────────────────────────┐
              ▼                           ▼                           ▼
     ┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
     │  Spring Boot    │         │  Spring Boot    │         │  Spring Boot    │
     │  (Build Worker) │         │  (Test Worker)  │         │ (Deploy Worker) │
     └────────┬────────┘         └─────────────────┘         └─────────────────┘
              │ Trigger
              ▼
     ┌─────────────────┐
     │    Jenkins      │
     └─────────────────┘
```

#### 추천 조합

| 우선순위 | 솔루션 | 이유 |
|----------|--------|------|
| 1 | **NATS** | 최경량, 간단한 설정, 충분한 성능 |
| 2 | **Redis Streams** | Redis 이미 사용 시, 인프라 단순화 |
| 3 | **ActiveMQ Artemis** | JMS 필요 시, 임베디드 가능 |

## 5. 운영 고려사항

### 5.1 모니터링

#### NATS
```bash
# 모니터링 엔드포인트
curl http://localhost:8222/varz   # 서버 상태
curl http://localhost:8222/connz  # 연결 정보
curl http://localhost:8222/subsz  # 구독 정보
```

#### Redis Streams
```bash
# Redis CLI
XINFO STREAM my-stream
XINFO GROUPS my-stream
XINFO CONSUMERS my-stream my-group
```

### 5.2 고가용성

#### NATS 클러스터
```yaml
# docker-compose.yml
services:
  nats-1:
    image: nats:latest
    command: ["--cluster", "nats://0.0.0.0:6222", "--routes", "nats://nats-2:6222,nats://nats-3:6222"]
  nats-2:
    image: nats:latest
    command: ["--cluster", "nats://0.0.0.0:6222", "--routes", "nats://nats-1:6222,nats://nats-3:6222"]
  nats-3:
    image: nats:latest
    command: ["--cluster", "nats://0.0.0.0:6222", "--routes", "nats://nats-1:6222,nats://nats-2:6222"]
```

#### Redis Sentinel
```yaml
services:
  redis-master:
    image: redis:7-alpine
  redis-sentinel:
    image: redis:7-alpine
    command: redis-sentinel /etc/redis/sentinel.conf
```

### 5.3 백업/복구

| 솔루션 | 백업 방법 |
|--------|----------|
| NATS JetStream | `nats stream backup` CLI |
| Redis | RDB 스냅샷, AOF 로그 |
| Artemis | Journal 파일 백업 |

## 6. 결론

### 최종 추천

**Jenkins/GitLab 미들웨어 통신 용도**:

```
✅ NATS (1순위)
   - 최경량, Docker 컨테이너 하나로 운영
   - JetStream으로 영속성 추가 가능
   - Spring Boot 통합 간단

✅ Redis Streams (2순위)
   - 이미 Redis 사용 중이면 최적
   - 캐시 + 큐 통합 운영
```

### Kafka 전환 시점

다음 요구사항 발생 시 Kafka 도입 검토:
- 일 처리량 100만 건 초과
- 메시지 재처리/감사 로그 필수
- 이벤트 소싱 아키텍처 도입
- 실시간 데이터 파이프라인 구축
