# 모노레포 및 멀티모듈 관리

> 작성일: 2025-01-14
> 목적: 모노레포 도구 비교, 멀티모듈 CI/CD 전략, 선택적 배포 방법 정리

## 1. 모노레포 vs 폴리레포

### 1.1 개념 비교

| 구분 | 모노레포 (Monorepo) | 폴리레포 (Polyrepo) |
|-----|---------------------|---------------------|
| **정의** | 여러 프로젝트를 하나의 저장소에서 관리 | 프로젝트마다 별도 저장소 |
| **코드 공유** | 직접 참조 가능 | 패키지 발행/설치 필요 |
| **의존성** | 단일 버전 관리 | 각자 관리 |
| **CI/CD** | 영향받는 부분만 빌드 가능 | 저장소별 독립 파이프라인 |
| **팀 협업** | 전체 코드 가시성 | 저장소별 분리 |

### 1.2 구조 예시

```
# 모노레포 구조
monorepo/
├── apps/
│   ├── web/              # 웹 애플리케이션
│   ├── mobile/           # 모바일 앱
│   └── api/              # API 서버
├── packages/
│   ├── ui/               # 공유 UI 컴포넌트
│   ├── utils/            # 공유 유틸리티
│   └── config/           # 공유 설정
├── tools/
│   └── scripts/          # 빌드/배포 스크립트
├── package.json
├── turbo.json           # Turborepo 설정
└── nx.json              # 또는 Nx 설정
```

### 1.3 모노레포 사용 기업

| 기업 | 도구 | 규모 |
|-----|------|------|
| Google | Bazel | 20억+ 라인 |
| Meta | Buck | 수천만 라인 |
| Microsoft | 커스텀 | Windows 코드베이스 |
| Stripe | Bazel | 300+ 서비스 |
| Airbnb | Nx | 다수 프로젝트 |

## 2. 모노레포 도구 비교

### 2.1 주요 도구

| 도구 | 언어 지원 | 특징 | 적합한 경우 |
|-----|----------|------|-----------|
| **Nx** | JS/TS (다른 언어도 가능) | 풀 스위트, 강력한 그래프 분석 | 엔터프라이즈, Angular |
| **Turborepo** | JS/TS | 빠른 빌드, 간단한 설정 | 스타트업, React/Next.js |
| **Bazel** | 다중 언어 | 최대 확장성, 원격 실행 | 대규모 다중 언어 프로젝트 |
| **Lerna** | JS/TS | npm 패키지 관리 중심 | 라이브러리 배포 |
| **Pants** | Python, Go, Java 등 | 다중 언어, Bazel 대안 | Python 중심 프로젝트 |

### 2.2 Nx 상세

```json
// nx.json
{
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "cache": true
    },
    "test": {
      "cache": true
    }
  },
  "namedInputs": {
    "default": ["{projectRoot}/**/*"],
    "production": ["default", "!{projectRoot}/**/*.spec.ts"]
  },
  "affected": {
    "defaultBase": "main"
  }
}
```

**주요 기능**:
- **Project Graph**: 의존성 시각화 (`nx graph`)
- **Affected 명령**: 변경된 프로젝트만 빌드/테스트
- **Remote Caching**: Nx Cloud로 팀 전체 캐시 공유
- **Task Orchestration**: 병렬 실행 및 의존성 순서 관리

```bash
# 영향받는 프로젝트만 빌드
nx affected:build --base=main

# 영향받는 프로젝트만 테스트
nx affected:test --base=main

# 특정 프로젝트와 의존성 빌드
nx build my-app --with-deps
```

### 2.3 Turborepo 상세

```json
// turbo.json
{
  "$schema": "https://turbo.build/schema.json",
  "globalDependencies": ["**/.env.*local"],
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", ".next/**"]
    },
    "test": {
      "dependsOn": ["build"],
      "cache": true
    },
    "lint": {
      "cache": true
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

**주요 기능**:
- **turbo prune**: Docker 빌드용 서브셋 추출
- **Remote Caching**: Vercel 또는 자체 서버
- **Pipeline 정의**: 태스크 의존성 및 캐시 설정

```bash
# 영향받는 패키지만 빌드
turbo run build --filter=...[origin/main]

# 특정 앱과 의존성 빌드
turbo run build --filter=web...

# Docker용 프루닝
turbo prune --scope=web --docker
```

### 2.4 Bazel 상세

```python
# BUILD.bazel
load("@rules_nodejs//nodejs:defs.bzl", "nodejs_binary")
load("@npm//:defs.bzl", "npm_link_all_packages")

npm_link_all_packages(name = "node_modules")

nodejs_binary(
    name = "app",
    data = [
        ":node_modules",
        "//packages/utils:lib",
    ],
    entry_point = "src/index.js",
)
```

**특징**:
- **Hermetic Builds**: 완전히 재현 가능한 빌드
- **Remote Execution**: 서버 팜에서 빌드 실행
- **언어 무관**: Java, Go, Python, C++ 등 모두 지원

## 3. 모노레포 CI/CD 전략

### 3.1 Affected/Changed Detection

변경된 파일을 분석하여 영향받는 프로젝트만 빌드/테스트:

```yaml
# GitHub Actions with Nx
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  main:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 전체 히스토리 필요

      - uses: nrwl/nx-set-shas@v4
        id: setSHAs

      - name: Install dependencies
        run: npm ci

      - name: Run affected builds
        run: npx nx affected -t build --base=${{ steps.setSHAs.outputs.base }}

      - name: Run affected tests
        run: npx nx affected -t test --base=${{ steps.setSHAs.outputs.base }}
```

### 3.2 캐싱 전략

```yaml
# Turborepo Remote Caching
- name: Setup Turbo cache
  uses: actions/cache@v4
  with:
    path: .turbo
    key: turbo-${{ runner.os }}-${{ github.sha }}
    restore-keys: |
      turbo-${{ runner.os }}-

# 원격 캐시 사용
- name: Build with remote cache
  run: turbo run build --remote-only
  env:
    TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
    TURBO_TEAM: ${{ vars.TURBO_TEAM }}
```

**캐시 키 포함 항목**:
- OS 버전
- Node.js 버전
- 락 파일 해시
- 빌드 대상 해시

### 3.3 병렬 실행 전략

```yaml
# 매트릭스 빌드로 병렬화
strategy:
  matrix:
    package: [web, api, mobile]

steps:
  - name: Build ${{ matrix.package }}
    run: turbo run build --filter=${{ matrix.package }}
```

## 4. 선택적 배포 (Selective Deployment)

### 4.1 변경 감지 기반 배포

```yaml
# GitHub Actions - 변경된 앱만 배포
jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      web: ${{ steps.filter.outputs.web }}
      api: ${{ steps.filter.outputs.api }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            web:
              - 'apps/web/**'
              - 'packages/ui/**'
              - 'packages/utils/**'
            api:
              - 'apps/api/**'
              - 'packages/utils/**'

  deploy-web:
    needs: detect-changes
    if: needs.detect-changes.outputs.web == 'true'
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying web app..."

  deploy-api:
    needs: detect-changes
    if: needs.detect-changes.outputs.api == 'true'
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying API..."
```

### 4.2 Nx를 활용한 선택적 배포

```yaml
# 영향받는 앱만 빌드하고 배포
- name: Get affected apps
  id: affected
  run: |
    AFFECTED=$(npx nx print-affected --type=app --base=origin/main | jq -c '.projects')
    echo "apps=$AFFECTED" >> $GITHUB_OUTPUT

- name: Deploy affected apps
  run: |
    for app in $(echo '${{ steps.affected.outputs.apps }}' | jq -r '.[]'); do
      echo "Deploying $app..."
      npx nx deploy $app
    done
```

### 4.3 태그 기반 선택적 배포

```yaml
# 태그 형식: app-name/v1.0.0
on:
  push:
    tags:
      - '*/v*.*.*'

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Extract app name and version
        id: extract
        run: |
          TAG=${GITHUB_REF#refs/tags/}
          APP_NAME=${TAG%/v*}
          VERSION=${TAG#*/v}
          echo "app=$APP_NAME" >> $GITHUB_OUTPUT
          echo "version=$VERSION" >> $GITHUB_OUTPUT

      - name: Deploy ${{ steps.extract.outputs.app }}
        run: |
          echo "Deploying ${{ steps.extract.outputs.app }} version ${{ steps.extract.outputs.version }}"
```

## 5. Docker 빌드 최적화

### 5.1 Turborepo prune 활용

```dockerfile
# Dockerfile
FROM node:20-alpine AS builder

# turbo prune으로 필요한 파일만 복사
WORKDIR /app
COPY . .
RUN npx turbo prune --scope=web --docker

# 의존성 설치
FROM node:20-alpine AS installer
WORKDIR /app

# 프루닝된 락파일로 의존성 설치
COPY --from=builder /app/out/json/ .
COPY --from=builder /app/out/package-lock.json ./package-lock.json
RUN npm ci

# 소스 복사 및 빌드
COPY --from=builder /app/out/full/ .
RUN npx turbo run build --filter=web

# 실행 이미지
FROM node:20-alpine AS runner
WORKDIR /app

COPY --from=installer /app/apps/web/dist ./dist
COPY --from=installer /app/node_modules ./node_modules

CMD ["node", "dist/index.js"]
```

### 5.2 레이어 캐싱 최적화

```dockerfile
# 의존성 레이어 분리
COPY package.json package-lock.json ./
COPY apps/web/package.json ./apps/web/
COPY packages/ui/package.json ./packages/ui/
COPY packages/utils/package.json ./packages/utils/

# 의존성 설치 (이 레이어가 캐시됨)
RUN npm ci --workspace=web

# 소스 복사 (변경 시에만 재빌드)
COPY apps/web ./apps/web
COPY packages ./packages

RUN npm run build --workspace=web
```

## 6. 멀티모듈 버전 관리

### 6.1 독립 버전 관리

각 패키지가 독립적인 버전을 가짐:

```json
// packages/ui/package.json
{
  "name": "@myorg/ui",
  "version": "2.1.0"
}

// packages/utils/package.json
{
  "name": "@myorg/utils",
  "version": "1.5.3"
}

// apps/web/package.json
{
  "name": "@myorg/web",
  "version": "3.0.0",
  "dependencies": {
    "@myorg/ui": "workspace:*",
    "@myorg/utils": "workspace:*"
  }
}
```

### 6.2 통합 버전 관리

모든 패키지가 동일한 버전을 공유:

```json
// lerna.json
{
  "version": "2.5.0",
  "packages": ["packages/*", "apps/*"]
}
```

### 6.3 Changesets를 활용한 버전 관리

```bash
# 변경사항 추가
npx changeset add

# 버전 업데이트
npx changeset version

# 패키지 발행
npx changeset publish
```

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: changesets/action@v1
        with:
          publish: npm run release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

## 7. 실제 성과 사례

### 7.1 Stripe

- **전환**: 300+ 서비스를 모노레포로 통합
- **도구**: Bazel
- **결과**: CI 시간 45분 → 7분 이하로 단축

### 7.2 스타트업 사례

- **전환**: 15+ 프로젝트를 Turborepo로 통합
- **결과**:
  - 빌드 시간 20분 → 2분
  - 영향받는 마이크로서비스만 배포
  - 클라우드 비용 절감

## 8. 도구 선택 가이드

```
┌─────────────────────────────────────────────────────────────────┐
│                    모노레포 도구 선택                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  프로젝트 규모는?                                                │
│  ├── 소규모 (3개 미만) → npm/yarn workspaces만으로 충분         │
│  ├── 중규모 (3-20개)                                           │
│  │   ├── React/Next.js 중심? → Turborepo                       │
│  │   └── Angular/Enterprise? → Nx                              │
│  └── 대규모 (20개 이상)                                         │
│      ├── 단일 언어? → Nx                                        │
│      └── 다중 언어? → Bazel                                     │
│                                                                 │
│  팀 경험 수준은?                                                 │
│  ├── 초급 → Turborepo (간단한 설정)                             │
│  ├── 중급 → Nx (풍부한 문서)                                    │
│  └── 고급 → Bazel (높은 학습 곡선)                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
