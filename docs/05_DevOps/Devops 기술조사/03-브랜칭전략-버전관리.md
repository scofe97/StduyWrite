## 1. 브랜칭 전략 비교

### 1.1 Trunk-Based Development (TBD)

**개념**: 모든 개발자가 단일 브랜치(main/trunk)에서 작업하고 자주 통합

```
main ────●────●────●────●────●────●────●────●────●──→
          │              │              │
          └── short-lived feature branches (< 1-2일)
```

**특징**:
- 짧은 수명의 피처 브랜치 (1-2일 이내)
- 잦은 통합 (하루 여러 번)
- Feature Flag로 미완성 기능 숨김
- CI/CD와 자연스럽게 연계

**사용 기업**:
| 기업 | 배포 빈도 |
|-----|----------|
| Netflix | 하루 수천 번 |
| Airbnb | 하루 50회 이상 |
| Google | 지속적 배포 |
| Facebook/Meta | 몇 분마다 |

**장점**:
- 머지 충돌 최소화
- 빠른 피드백 루프
- 코드가 항상 배포 가능 상태
- 간단한 히스토리

**단점**:
- 강력한 자동화 테스트 필수
- Feature Flag 관리 오버헤드
- 미숙한 팀에게는 위험할 수 있음

---

### 1.2 GitFlow

**개념**: 엄격한 브랜치 구조로 병렬 개발과 릴리스 관리

```
main     ────────●─────────────────────●─────────────●──→
                 │                     │             │
release          │     ●───────●───────┘             │
                 │     │                             │
develop  ────●───●─────●───────●───────●─────────────●──→
             │   │     │       │       │
feature   ───●   │     │       │───────┘
                 │     │
hotfix           │     ●───────────────────────────────→
```

**브랜치 역할**:
| 브랜치 | 목적 | 수명 |
|-------|------|------|
| `main` | 프로덕션 릴리스 히스토리 | 영구 |
| `develop` | 다음 릴리스 통합 | 영구 |
| `feature/*` | 새 기능 개발 | 기능 완료까지 |
| `release/*` | 릴리스 준비 | 릴리스까지 |
| `hotfix/*` | 프로덕션 긴급 수정 | 수정 완료까지 |

**장점**:
- 명확한 릴리스 관리
- 병렬 개발 용이
- 여러 버전 유지보수 가능
- 코드 리뷰 프로세스 명확

**단점**:
- 복잡한 브랜치 그래프
- 머지 충돌 빈번
- 피드백 루프 느림
- CI/CD 파이프라인 복잡

---

### 1.3 GitHub Flow

**개념**: GitFlow의 단순화된 버전

```
main     ────●────●────●────●────●────●────●──→
             │    │    │
feature   ───●    │    │
                  │    │
feature   ────────●    │
                       │
feature   ─────────────●
```

**워크플로우**:
1. `main`에서 피처 브랜치 생성
2. 커밋 및 푸시
3. Pull Request 생성
4. 코드 리뷰 및 논의
5. 머지 후 즉시 배포

**적합한 경우**:
- 지속적 배포 환경
- 웹 애플리케이션
- 단일 프로덕션 버전

### 1.4 브랜칭 전략 선택 가이드

```
┌─────────────────────────────────────────────────────────────────┐
│                    브랜칭 전략 선택 기준                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  지속적 배포(CD) 가능?                                           │

│  │         └── 중간 → GitHub Flow                               │
│  └── No                                                         │
│       ├── 다중 버전 유지보수 필요?                                │
│       │   └── Yes → GitFlow                                     │
│       └── 정기 릴리스 사이클?                                    │
│           └── Yes → GitFlow                                     │
│                                                                 │
│  규제/컴플라이언스 요구사항?                                      │
│  ├── 높음 → GitFlow (명확한 승인 프로세스)                       │
│  └── 낮음 → TBD / GitHub Flow                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2. Semantic Versioning (SemVer)

### 2.1 버전 번호 구조

```
MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]

예시: 2.1.3-beta.1+build.456

┌─────┬─────┬─────┬──────────┬────────────┐
│  2  │  1  │  3  │  beta.1  │  build.456 │
├─────┴─────┴─────┴──────────┴────────────┤
│ MAJOR: 호환성 깨지는 변경               │
│ MINOR: 하위 호환 새 기능                 │
│ PATCH: 하위 호환 버그 수정               │
│ PRERELEASE: 사전 릴리스 식별자           │
│ BUILD: 빌드 메타데이터                   │
└─────────────────────────────────────────┘
```

### 2.2 버전 증가 규칙

| 변경 유형 | 버전 증가 | 예시 |
|----------|----------|------|
| 버그 수정, 패치 | PATCH | 1.0.0 → 1.0.1 |
| 새 기능 추가 (하위 호환) | MINOR | 1.0.1 → 1.1.0 |
| 호환성 깨지는 변경 | MAJOR | 1.1.0 → 2.0.0 |

## 3. Git 태그 기반 배포

### 3.1 Git 태그 유형

| 유형 | 설명 | 명령어 |
|-----|------|--------|
| **Lightweight** | 단순 포인터 | `git tag v1.0.0` |
| **Annotated** | 메타데이터 포함 (권장) | `git tag -a v1.0.0 -m "Release 1.0.0"` |
| **Signed** | GPG 서명 포함 | `git tag -s v1.0.0 -m "Release 1.0.0"` |

### 3.2 태그 기반 CI/CD 트리거

```yaml
# GitHub Actions - 태그 푸시 시 배포
name: Release

on:
  push:
    tags:
      - 'v*.*.*'  # v1.0.0, v2.1.3 등

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Get version from tag
        id: version
        run: echo "VERSION=${GITHUB_REF#refs/tags/v}" >> $GITHUB_OUTPUT

      - name: Build with version
        run: |
          echo "Building version ${{ steps.version.outputs.VERSION }}"
          docker build -t myapp:${{ steps.version.outputs.VERSION }} .

      - name: Push to registry
        run: |
          docker push myapp:${{ steps.version.outputs.VERSION }}
          docker tag myapp:${{ steps.version.outputs.VERSION }} myapp:latest
          docker push myapp:latest
```

### 3.3 자동 버전 관리 도구

| 도구 | 특징 | 지원 플랫폼 |
|-----|------|-----------|
| **semantic-release** | 완전 자동화, Conventional Commits | GitHub, GitLab, npm |
| **GitVersion** | .NET 생태계, 다양한 워크플로우 | Azure DevOps, GitHub |
| **release-please** | Google 개발, 변경 로그 자동 생성 | GitHub |
| **standard-version** | npm 패키지용, CHANGELOG 생성 | 모든 Git 저장소 |

## 4. Conventional Commits

### 4.1 커밋 메시지 형식

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### 4.2 커밋 타입

| 타입 | 설명 | 버전 증가 |
|-----|------|----------|
| `feat` | 새 기능 | MINOR |
| `fix` | 버그 수정 | PATCH |
| `docs` | 문서 변경 | - |
| `style` | 포매팅 변경 | - |
| `refactor` | 리팩토링 | - |
| `perf` | 성능 개선 | PATCH |
| `test` | 테스트 추가/수정 | - |
| `chore` | 빌드/설정 변경 | - |
| `BREAKING CHANGE` | 호환성 깨지는 변경 | MAJOR |

### 4.3 예시

```bash
# 새 기능 추가 (MINOR 증가)
feat(auth): add OAuth2 login support

# 버그 수정 (PATCH 증가)
fix(api): resolve null pointer in user service

# 호환성 깨지는 변경 (MAJOR 증가)
feat(api)!: change response format for user endpoint

BREAKING CHANGE: The user endpoint now returns data in a different format.
```

## 5. semantic-release 설정

### 5.1 기본 설정

```json
// .releaserc.json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    "@semantic-release/npm",
    "@semantic-release/github",
    "@semantic-release/git"
  ]
}
```

### 5.2 GitHub Actions 워크플로우

```yaml
name: Release

on:
  push:
    branches: [main]

permissions:
  contents: write
  issues: write
  pull-requests: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
          persist-credentials: false

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci

      - name: Release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
        run: npx semantic-release
```

## 6. 릴리스 트레인 (Release Train)

### 6.1 개념

정해진 일정에 따라 주기적으로 릴리스하는 방법론:

```
┌─────────────────────────────────────────────────────────────────┐
│                     릴리스 트레인 예시                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Sprint 1      Sprint 2      Sprint 3      Sprint 4            │
│  ─────────     ─────────     ─────────     ─────────           │
│                                                                 │
│     │             │             │             │                 │
│     ▼             ▼             ▼             ▼                 │
│  ┌─────┐      ┌─────┐      ┌─────┐      ┌─────┐               │
│  │ Dev │  →   │ QA  │  →   │Stag │  →   │Prod │               │
│  └─────┘      └─────┘      └─────┘      └─────┘               │
│                                                                 │
│  Release v1.0  Release v1.1  Release v1.2  Release v1.3        │
│                                                                 │
│  "기차가 역을 떠나면, 준비되지 않은 기능은 다음 기차를 타야 함"     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 릴리스 트레인 베스트 프랙티스

1. **고정된 릴리스 일정**: 매주 목요일 10시
2. **프리즈 기간**: 릴리스 2일 전 코드 프리즈
3. **Feature Flag 활용**: 미완성 기능 숨김
4. **자동화된 품질 게이트**: 모든 테스트 통과 필수
5. **롤백 계획**: 항상 즉시 롤백 가능하도록 준비

### 6.3 Microsoft의 사례

- Trunk-based 개발 + 3주 스프린트 릴리스
- Feature Flag로 기능 배포와 노출 분리
- 메인 브랜치에서 지속적 개발, 릴리스 시점에 배포

## 7. 멀티 팀 릴리스 관리

### 7.1 Agile Release Train (ART)

```
┌─────────────────────────────────────────────────────────────────┐
│                    Agile Release Train                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │  Team A  │  │  Team B  │  │  Team C  │  │  Team D  │       │
│  │ (Auth)   │  │  (Cart)  │  │ (Payment)│  │ (Search) │       │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘       │
│       │             │             │             │              │
│       └─────────────┴─────────────┴─────────────┘              │
│                           │                                    │
│                           ▼                                    │
│              ┌───────────────────────┐                         │
│              │    Release Manager    │                         │
│              │  (릴리스 조율/관리)    │                         │
│              └───────────────────────┘                         │
│                           │                                    │
│                           ▼                                    │
│              ┌───────────────────────┐                         │
│              │    Unified Release    │                         │
│              │       (v2.5.0)        │                         │
│              └───────────────────────┘                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 릴리스 조율 원칙

1. **동기화된 케이던스**: 모든 팀이 같은 주기로 작업
2. **디커플링**: 한 팀의 지연이 다른 팀 릴리스를 막지 않도록
3. **통합 테스트**: 릴리스 전 전체 시스템 통합 검증
4. **명확한 의존성 관리**: 팀 간 의존성 명시적 추적
