# MinIO AGPL 라이선스 검토 및 폐쇄망 대응 가이드

## 1. 문서 목적

본 문서는 트럼본(Trombone) 솔루션에서 MinIO를 첨부파일 저장 용도로 사용할 때 AGPL 라이선스 위반 여부를 판단하고, 폐쇄망 환경에서 감사 및 법무 대응을 위한 기준을 정리한 내부/대외 공용 가이드이다.

**대상 독자**: 개발/플랫폼 팀, 아키텍트, 보안 감사 대응 담당자, 법무 협업 시 참고 자료

---

## 2. MinIO 라이선스 개요

MinIO는 듀얼 라이선스 모델을 사용한다.

| 라이선스 | 대상 | 비고 |
|----------|------|------|
| **AGPL v3** | Community Edition | 무료, 소스 공개 의무 존재 |
| **Commercial License** | Enterprise / AIStor | 유료, 소스 공개 의무 없음 |

AGPL은 GPL 계열 중에서도 **네트워크 서비스 제공 시 소스 공개 의무**를 명시적으로 포함하는 가장 강력한 카피레프트 라이선스이다. 핵심적으로 이해해야 할 점은 다음과 같다.

- 인터넷 연결 여부와 **무관**하게 적용된다
- 내부망, 폐쇄망, 전용망 환경 **모두 포함**된다
- "수정 여부"보다 **"제공 방식"**이 핵심 판단 기준이다

## 3. AGPL v3 라이선스 상세 분석

### 3.1. AGPL v3 vs GPL v3 비교

AGPL v3은 GPL v3에 **Section 13(네트워크 상호작용 조항)** 하나만 추가한 것이다. 나머지 조문은 동일하다.

| 항목 | GPL v3 | AGPL v3 |
|------|--------|---------|
| 카피레프트 강도 | 강함 | **가장 강함** |
| 배포 시 소스 공개 | 필수 | 필수 |
| 네트워크/SaaS 소스 공개 | **불필요** | **필수** |
| 특허 허여 | 있음 | 있음 |
| Anti-Tivoization | 있음 | 있음 |
| 상호 호환성 | 있음 (Section 13) | 있음 (Section 13) |

### 3.2. Section 13: 네트워크 상호작용 조항

Section 13은 **"네트워크 카피레프트"**를 구현한 조항이다. GPL의 SaaS 허점(loophole)을 막기 위해 만들어졌다.

**핵심 요구사항**: 프로그램을 수정하여 네트워크를 통해 사용자에게 제공하는 경우, 해당 수정 버전의 소스 코드를 무료로 네트워크 서버를 통해 제공해야 한다.

이 조항을 이해하기 위한 핵심 뉘앙스가 있다.

**수정(Modification)이 트리거 조건이다.** Section 13의 소스 코드 공개 의무는 AGPL 소프트웨어를 **수정한 경우에만** 발생한다. 수정 없이 원본 그대로 서버에서 실행하는 경우, Section 13은 소스 공개를 요구하지 않는다는 것이 다수 법률 전문가의 해석이다.

**"수정"의 범위는 논쟁 중이다.** 가장 관대한 해석은 소스 코드를 변경하여 새 바이너리를 만드는 것만 수정으로 본다. 가장 엄격한 해석은 다른 소프트웨어와의 결합까지 수정으로 본다. 듀얼 라이선스 모델을 사용하는 벤더(MinIO 포함)는 엄격한 해석을 선호하는 경향이 있다.

### 3.3. 파생 저작물(Derivative Work) vs 집합물(Aggregate)

AGPL v3은 GPL v3에서 상속받은 **"집합물(Aggregate)"** 개념을 명시적으로 정의한다.

> 독립적이고 분리된 저작물들과 함께 하나의 저장 매체나 배포 매체에 컴파일된 것으로, 그 성질상 해당 저작물의 확장이 아니며 더 큰 프로그램을 형성하도록 결합되지 않은 것을 "집합물"이라 한다.

**집합물에 포함되는 것은 AGPL이 다른 부분에 적용되는 것을 유발하지 않는다.** 즉, MinIO와 트럼본이 각각 독립적인 프로그램으로 동작하고, 표준 API(S3)를 통해서만 통신한다면 집합물로 볼 수 있다.

그러나 "파생 저작물" vs "집합물"의 구분은 오픈소스 라이선스에서 **가장 논쟁이 많고 법적으로 아직 확립되지 않은 영역** 중 하나이다. 마이크로서비스 아키텍처에서 서비스 간 통신(API, 메시지 큐, RPC)이 파생 저작물을 구성하는지 여부는 **사안별 판단**이 필요하며 법적으로 미확정 상태이다.

## 4. SaaS 허점(SaaS Loophole)과 AGPL의 탄생 배경

### 4.1. GPL의 한계

GPL에서 "배포(distribution)"는 소프트웨어를 물리적으로 전달하거나 다운로드 가능하게 만드는 것을 의미한다. **서버에서 소프트웨어를 실행하여 네트워크 서비스를 제공하는 것은 배포에 해당하지 않는다.** 이것은 GPL v3의 의도적인 설계 결정이었다.

1990년대 후반~2000년대에 웹 기반 애플리케이션이 부상하면서 이 한계가 드러났다. Google, Amazon, Facebook 같은 회사들이 GPL 소프트웨어를 서버에서 대규모로 수정하여 사용하면서도, "배포"하지 않는다는 이유로 소스 코드를 한 줄도 공유하지 않을 수 있었다.

### 4.2. AGPL의 해결 방식

AGPL은 트리거 조건을 "배포(distribution)"에서 **"수정 + 네트워크 가용성(modification + network availability)"**으로 변경했다. 수정된 소프트웨어를 네트워크를 통해 사용자가 접근할 수 있게 만드는 것 자체가 소스 공개 의무를 발생시킨다.

### 4.3. 듀얼 라이선스 전략의 도구

AGPL의 실질적인 상업적 용도는 **듀얼 라이선스의 수단**이다. 저작권자가 AGPL로 공개하면 경쟁사는 두 가지 선택을 강요받는다.

1. 자신의 소스 코드를 공개하거나
2. 상용 라이선스를 구매하거나

MinIO, MongoDB(SSPL 이전), Neo4j, Grafana 등이 이 모델을 사용한다/사용했다.

## 5. MinIO 라이선스 변경 이력

### 5.1. 연대기

| 시점 | 사건 |
|------|------|
| **2019년 이전** | Apache 2.0 라이선스 |
| **2019년 10월** | 일부 컴포넌트 AGPL v3 전환 시작 |
| **2021년 5월 11일** | **AGPL v3 전면 전환 완료** (서버, 클라이언트, 게이트웨이 모두) |
| **2022년 7월** | Nutanix 라이선스 위반 공개 비난 |
| **2022년 11월** | Nutanix, Objects 3.6 릴리즈에서 MinIO 완전 제거 |
| **2023년 3월** | Weka 라이선스 위반 공개 비난 |
| **2025년 6월** | Community Edition에서 Web UI 제거 (관리 기능이 Enterprise 전용으로 전환) |
| **2025년 12월 3일** | **Community Edition 유지보수 모드 전환** (신규 기능 없음, PR 미수용, 보안 패치만 선별 적용) |

### 5.2. 전환 과정

MinIO는 모든 기여 코드를 분석하여 다음과 같이 분류했다.

- 저작권 가치가 있는 외부 기여: AGPL v3 릴리즈에 포함하되, 원본은 Apache 2.0으로 별도 `-contrib` 파일에 보존
- 저작권 가치가 없는 기여: AGPL v3 릴리즈에 포함
- 시간이 지나면서 재작성되거나 제거된 기여

### 5.3. 커뮤니티 반응

반응은 **상당히 부정적**이었다.

- 기존 사용자들의 법적 문제를 야기할 수 있다는 우려
- "사전 통보나 커뮤니티 의견 수렴 없이 갑작스럽게 변경되었다"는 비판 (GitHub Discussion #12157)
- 단독 저작권자가 아닌 프로젝트에서 기여 코드를 재라이선스할 수 있는지에 대한 법적 의문 제기
- Hacker News에서도 큰 논란 (HN #26919510)

### 5.4. 2025년의 상업화 전환

MinIO의 궤적은 오픈코어 비판자들이 경고하는 **"bait-and-switch(미끼 전환)"** 패턴의 교과서적 사례이다.

1. **Apache 2.0** (초기) - 완전 허용적, 커뮤니티 성장
2. **AGPL v3** (2021년) - 카피레프트, 듀얼 라이선스
3. **UI 제거** (2025년 6월) - Community Edition 기능 축소
4. **유지보수 모드** (2025년 12월) - 신규 기능 중단, 사전 빌드 바이너리 미제공

현재 MinIO는 사용자를 다음으로 유도한다.

- **AIStor Free**: 무료 라이선스 키로 전체 기능 사용 가능한 독립 버전
- **AIStor Enterprise**: 연간 최소 $96,000에서 1PB 기준 $244,032까지

## 6. 폐쇄망과 AGPL에 대한 오해와 실제

### 6.1. 흔한 오해

| 오해 | 실제 |
|------|------|
| "폐쇄망이니까 라이선스 상관없다" | 폐쇄망이라도 AGPL 적용 대상이 될 수 있다 |
| "외부 인터넷 연결이 없으니 AGPL 적용 안 된다" | AGPL의 기준은 인터넷이 아니라 네트워크를 통한 제3자 제공 여부이다 |
| "내부 사용이면 소스 공개 안 해도 된다" | 대체로 맞지만, "내부"의 경계가 모호할 수 있다 |
| "수정하지 않으면 괜찮다" | 수정하지 않으면 Section 13이 트리거되지 않는다는 해석이 지배적이다 |
| "AGPL 컴포넌트와 직접 상호작용하는 사용자만 해당된다" | API 호출, 백그라운드 작업, 제품을 통해 노출되는 내부 도구도 해당될 수 있다 |
| "규모가 작으면 문제없다" | 투자자와 감사인은 규모와 관계없이 오픈소스 사용을 조사한다 |

### 6.2. 법률 전문가의 해석

**내부 사용에 대한 지배적 해석**: AGPL 소프트웨어를 내부 사설 네트워크에서 사용하는 것은 소스 코드 공유 요구 사항을 **트리거하지 않는다**는 것이 다수 법률 전문가의 해석이다.

그러나 "내부"와 "외부"의 경계가 모호해지는 경우가 있다.

- 파트너 접근
- 고객 포탈
- 외주 업체 접근
- 외부에 노출된 API

이러한 경우에는 별도의 법적 검토가 필요하다.

### 6.3. AGPL의 핵심 판단 질문

> **해당 소프트웨어가 네트워크를 통해 제3자가 사용하는 형태로 제공되는가?**

이 질문에 "예"라고 답해야 하는 상황이라면, AGPL 의무가 발생할 수 있다.

## 7. 법적 판례 및 분쟁 사례

### 7.1. 주요 판례

| 사건 | 관할 | 의의 |
|------|------|------|
| **Artifex v. Hancom (2017)** | 미국 (N.D. Cal.) | 한국 기업 한글과컴퓨터가 GPL 듀얼 라이선스인 Ghostscript를 한컴오피스에 포함하면서 상용 라이선스 구매도, 소스 공개도 하지 않았다. 법원은 **서명하지 않은 GPL도 라이선스이자 계약으로 집행 가능**하다고 판결했다. |
| **GlobaLeaks/Hermes v. ANAC (이탈리아)** | 이탈리아 | 최초의 AGPL 준수 사건. 이탈리아 반부패 기관이 AGPL 소프트웨어를 EUPL로 재공개하고 출처를 변경했다. 법원 합의로 라이선스가 복원되었다. |
| **Neo4j v. PureThink/Graph Foundation** | 미국 (N.D. Cal., 현재 9th Circuit 항소) | **가장 중요한 AGPL 관련 사건.** AGPL + Commons Clause 제한이 부과될 수 있는지가 쟁점이다. 하급심 판결이 유지되면 GPL 코드에 추가 제한을 부과/제거할 수 있는지에 대한 이해를 "근본적으로 변경"할 수 있다. |
| **MinIO v. Nutanix (2022)** | 공개 분쟁 | MinIO가 Nutanix의 라이선스를 종료했고, Nutanix는 결국 MinIO 코드를 전부 제거했다. |
| **MinIO v. Weka (2023)** | 공개 분쟁 | MinIO가 Weka의 라이선스를 취소했고, Weka는 Apache 2.0 버전의 코드만 사용했으며 Apache 2.0은 "취소 불가"라고 주장했다. |

### 7.2. 핵심 관찰

**AGPL은 아직까지 정식 재판(trial)에서 완전히 심리된 적이 없다.** 이 판례법의 부재가 상당한 법적 불확실성을 만들고 있다. Neo4j 사건(현재 9th Circuit 항소 중)이 최초의 주요 선례가 될 수 있다.

## 8. 주요 기업의 AGPL 대응 정책

### 8.1. Google - 완전 금지

Google은 가장 공개적으로 문서화된 AGPL 정책을 가지고 있다.

> "AGPL 라이선스 코드는 Google에서 사용해서는 안 된다."

- google3 코드베이스에 AGPL 코드 체크인 금지
- Google 제품에 AGPL 사용 금지
- 워크스테이션/노트북/휴대폰에 AGPL 프로그램 설치 금지 (Open Source Programs Office의 명시적 승인 없이)
- 이유: Google의 핵심 제품이 네트워크 서비스이므로, 엔지니어가 실수로 AGPL에 의존하면 결과가 심각하기 때문

### 8.2. SK Telecom - 명시적 금지

SK Telecom은 AGPL과 SSPL 라이선스 소프트웨어의 **제품 및 서비스 개발 사용을 금지**한다. 이유: "네트워크 서비스가 배포로 간주되어, 핵심 서버의 연결된 소프트웨어 소스 코드도 AGPL로 공개해야 하므로, 핵심 서버 프로그램에 수용할 수 없는 리스크가 된다."

### 8.3. 일반적인 기업 관행

대부분의 기업은 오픈소스 정책을 공개하지 않는다. 라이선스 전문가 Heather Meeker에 따르면, 많은 기업이 AGPL을 **"중지(stop)" 목록**(사안별 승인 필요)에 올려두고 있으며, 완전 금지와는 다르다. 제품 가치가 준수 노력을 정당화하면 채택이 가능한 구조이다.

## 9. 한국 기업의 오픈소스 라이선스 환경

### 9.1. 주요 기관 및 시스템

| 기관/시스템 | 역할 |
|-------------|------|
| **한국저작권위원회 OLIS** (olis.or.kr) | 라이선스 분류, 비교 가이드, 준수 체크리스트, **코드아이(Code Eye)** 스캐닝 서비스 제공 |
| **NIPA 공개SW 포털** (oss.kr) | 오픈소스 가이드 및 위반 사례 제공 |
| **OpenChain** | 한국어 OpenChain 준수 가이드 제공 |

OLIS는 라이선스를 3가지로 분류한다.

- **강한 카피레프트**: GPL, AGPL (수정/결합 시 전체 소스 공개)
- **약한 카피레프트**: LGPL, MPL (해당 파일/라이브러리만 공개)
- **퍼미시브**: MIT, Apache 2.0, BSD (고지만 하면 됨)

### 9.2. 한국 관련 위반 사례

| 사건 | 내용 | 결과 |
|------|------|------|
| **Artifex v. 한글과컴퓨터 (2017)** | 한컴오피스에 GPL 듀얼 라이선스 Ghostscript를 상용 라이선스 없이 포함 | 미국 연방법원에서 GPL 집행 가능 판결. 한컴이 라이선스 합의 |
| **엘림넷 vs 하이온넷** | 두 회사가 실제로는 GPL 소프트웨어인 것을 독점 소유 주장 | FSF와 GNU Korea 개입. 두 회사 모두 공개 사과 및 GPL 소스 공개 (2006년 11월) |
| **한국 동영상 플레이어 GPL 위반** | KMPlayer, GomPlayer, PotPlayer 등이 FFmpeg(GPL/LGPL) 컴포넌트를 소스 공개 없이 사용 | 커뮤니티의 지속적 압박. 한국 미디어 소프트웨어의 광범위한 비준수 사례로 주목 |
| **SFC v. 14개 기업** | OLIS가 **한국 기업이 포함된 최초의 집단 GPL 대응 사례**로 기술 | 상세 미공개 |

### 9.3. 한국 기업 환경의 감사 시나리오

1. **출시 전 감사**: 소프트웨어 업데이트나 수정 전 라이선스 준수 감사 수행 필수
2. **코드아이 스캐닝**: 한국저작권위원회가 무료 스캐닝 서비스 제공
3. **M&A 실사**: 한국 M&A 거래에서 오픈소스 라이선스 준수 검토 증가 추세
4. **정부 조달**: 한국 정부 IT 조달 계약에서 오픈소스 준수 문서 요구 증가
5. **확인 항목**: 소스 코드 공개 의무, 저작권 고지 의무, 특허 면책 의무, 사용 허가 고지 의무 등 80종 이상의 라이선스 의무

## 10. 트럼본에서의 MinIO 사용 방식 분석

### 10.1. 현재 사용 패턴

| 항목 | 내용 |
|------|------|
| **사용 목적** | 첨부파일 저장소 |
| **역할** | S3-compatible Object Storage |
| **통신 방식** | 표준 S3 API |
| **핵심 기능과의 관계** | 트럼본 핵심 기능과 **분리됨** |
| **교체 가능성** | MinIO <-> Ceph <-> Cloud Object Storage 교체 가능 |

이 구조는 AGPL 관점에서 상대적으로 **매우 유리한 형태**이다. MinIO가 트럼본의 "파생 저작물(Derivative Work)"이 아닌 "집합물(Aggregate)"의 독립 컴포넌트로 해석될 가능성이 높다.

### 10.2. AGPL 위반 판단의 핵심

> **MinIO가 트럼본의 결합 저작물(Derivative Work)인가?**

현재 구조에서의 해석:

- MinIO = **외부 인프라 컴포넌트** (데이터베이스와 유사한 위치)
- 트럼본 = 업무 로직/워크플로우/PMS 애플리케이션
- 결합도 = **낮음** (표준 S3 API 통신)
- AGPL 전염 가능성 = **낮음**

## 11. 리스크 급상승 조건

아래 중 **하나라도 해당되면** AGPL 리스크가 급격히 높아진다. 감사나 법적 분쟁에서 "사실상 하나의 제품"으로 해석될 여지가 커진다.

### 11.1. 번들링(Bundling)

- 트럼본 설치 패키지에 MinIO가 **기본 포함**되어 있는 경우
- Helm Chart에서 MinIO가 **default enable**로 설정된 경우
- Docker Compose에서 MinIO가 필수 의존성으로 정의된 경우

### 11.2. 문서 표현

다음과 같은 표현이 공식 문서에 포함된 경우:

- "트럼본은 MinIO 기반입니다"
- "기본 스토리지는 MinIO입니다"
- "MinIO가 필요합니다" (대체 불가 뉘앙스)

### 11.3. 기술적 의존

- MinIO 고유 기능(policy, lifecycle, bucket rule 등)에 의존하는 경우
- MinIO SDK를 직접 import하여 사용하는 경우 (S3 표준 SDK 대신)
- MinIO의 내부 API를 호출하는 경우

## 12. 위반 판단 주체: 누가 문제를 제기하는가

### 12.1. MinIO 벤더 직접 문제 제기

- 이론적으로 가능하나 실제로는 드물다
- 폐쇄망에서 기술적 탐지는 불가능하다
- **문서, 계약서, 마케팅 자료**에서 단서를 확보한다
- Nutanix, Weka 사례처럼 규모가 큰 경우 주로 발생한다

### 12.2. 고객사 내부 감사 (가장 현실적)

한국 기업 환경에서 가장 현실적인 시나리오이다.

- 오픈소스 사용 목록 제출 요구
- AGPL 포함 사유 설명 요청
- 번들 여부, 구조 설명 요구
- 코드아이 스캐닝 결과와의 대조

### 12.3. 계약 분쟁 시 법적 쟁점화

- 유지보수/계약 분쟁 중 라이선스 위반 주장이 부수적 쟁점으로 등장
- **설치 구조와 문서 표현이 증거**로 사용됨

## 13. MinIO 대안 비교

### 13.1. S3 호환 스토리지 비교

| 솔루션 | 라이선스 | S3 호환성 (ceph s3-tests) | 성숙도 | 적합한 상황 |
|--------|----------|---------------------------|--------|-------------|
| **MinIO** | AGPL v3 (유지보수 모드) | 321/576 통과 | 높음 (하락 중) | 에지, 단일 노드, CI/CD 캐싱 |
| **Ceph RGW** | LGPL 2.1/3.0 | **576/576 통과** (최고) | 매우 높음 | 페타바이트 규모, 멀티 프로토콜 |
| **SeaweedFS** | **Apache 2.0** | 56/576 통과 | 중간 | 대규모 데이터셋, 아카이브, 백업 |
| **Garage** | AGPL v3 | 제한적 | 중간 | 10TB 미만, 에지, 지리적 분산 |
| **RustFS** | **Apache 2.0** | 미측정 | 신규 | 성능 중심 (4KB 객체 기준 MinIO 대비 2.3배 빠르다고 주장) |

### 13.2. 선택 가이드

**Ceph RGW 선택 시기**: 페타바이트 규모의 포괄적 스토리지 인프라가 필요할 때. CERN, Bloomberg, DreamWorks 등이 사용한다. S3 호환성이 가장 높다. LGPL은 약한 카피레프트로 AGPL보다 훨씬 관대하다. 단, 학습 곡선이 가파르다.

**SeaweedFS 선택 시기**: Apache 2.0(카피레프트 없음)이 필요할 때. 배포가 단순하고, 콘텐츠 배포/아카이브/백업 용도에 적합하다. 소규모 파일 성능이 좋다.

**RustFS 선택 시기**: Apache 2.0이 필요하면서 고성능이 우선일 때. 데이터 레이크, AI, 빅데이터 환경에 적합하다. 다만 새로운 프로젝트이므로 안정성 검증이 필요하다.

## 면접 대응 핵심 포인트

### Q: AGPL이 무엇이고 GPL과 어떻게 다른가?

A: AGPL v3는 GPL v3에 Section 13(네트워크 상호작용 조항)을 추가한 것이다. GPL은 소프트웨어를 "배포"할 때만 소스 공개 의무가 발생하지만, AGPL은 수정된 소프트웨어를 네트워크를 통해 사용자에게 제공하는 것만으로도 소스 공개 의무가 발생한다. 이것은 1990년대 후반 웹 서비스의 부상으로 드러난 GPL의 "SaaS 허점"을 막기 위해 만들어졌다.

### Q: 폐쇄망에서 AGPL 소프트웨어를 사용하면 문제가 되는가?

A: 지배적인 법률 해석에 따르면 조직 내부 사설 네트워크에서의 사용은 Section 13의 소스 공개 의무를 트리거하지 않는다. 그러나 "내부"의 경계가 모호해지는 경우(파트너 접근, 고객 포탈 등)에는 별도 검토가 필요하다. 중요한 점은 폐쇄망이라는 이유만으로 AGPL이 적용되지 않는 것은 아니라는 것이다. AGPL의 판단 기준은 인터넷 연결 여부가 아니라 네트워크를 통한 제3자 제공 여부이다.

### Q: MinIO를 사용할 때 AGPL 전염을 피하려면 어떻게 해야 하는가?

A: 핵심은 MinIO를 독립적인 인프라 컴포넌트로 유지하는 것이다. 구체적으로 S3 표준 API만 사용하고(MinIO 고유 API 사용 금지), MinIO SDK 대신 범용 S3 SDK를 사용하며, 설치 패키지에 번들하지 않고, 문서에서 "S3-compatible storage"로 표현하고, 다른 S3 호환 스토리지로 교체 가능한 구조를 유지해야 한다. 이렇게 하면 집합물(Aggregate)로 해석될 가능성이 높아져 AGPL 전염이 적용되지 않는다.

### Q: Google이 AGPL을 금지하는 이유는?

A: Google의 핵심 제품이 네트워크 서비스이기 때문이다. 엔지니어가 실수로 AGPL 코드에 의존하면 해당 서비스의 전체 소스 코드를 공개해야 할 수 있는 리스크가 있다. Google은 이 결과가 너무 심각하다고 판단하여 "공격적으로 광범위한 금지" 정책을 유지한다. 한국에서는 SK Telecom도 동일한 이유로 AGPL과 SSPL을 금지한다.

### Q: AGPL 관련 실제 법적 분쟁 사례가 있는가?

A: 대표적으로 MinIO vs Nutanix(2022), MinIO vs Weka(2023) 분쟁이 있다. Nutanix는 MinIO를 자사 제품에 포함하면서 적절한 소스 공개와 출처 표시를 하지 않았고, 결국 MinIO 코드를 전부 제거했다. 한국 관련으로는 Artifex vs 한글과컴퓨터(2017) 사건이 있는데, GPL이 서명 없이도 라이선스이자 계약으로 집행 가능하다는 판례를 남겼다. 다만 AGPL은 아직 정식 재판에서 완전히 심리된 적이 없어 법적 불확실성이 존재한다.
