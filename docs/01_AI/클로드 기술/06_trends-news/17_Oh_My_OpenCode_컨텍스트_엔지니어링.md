# Oh-My-OpenCode 뜯어 보고 컨텍스트 엔지니어링의 미래 확인하기

> 단순한 프롬프팅이 아닌, 컨텍스트 엔지니어링 차원의 구조적 혁신

---

## 배경

지금 개발자들 사이에서 Opencode가 큰 화제다. 무료로 풀린 고성능 모델과 강력한 플러그인 생태계가 결합되면서 '클로드 코드 해지' 움직임이 가속화되고 있다.

특히 한국인 개발자 YeonGyu Kim 님이 만든 **'Oh My OpenCode'** 플러그인이 여러 AI 모델을 팀처럼 오케스트레이션하는 멀티 에이전트 구축 사례로 큰 호응을 받고 있다.

코드를 뜯어보니 단순한 프롬프팅이 아닌, **컨텍스트 엔지니어링** 차원의 구조적 혁신이 숨어있었다.

---

## 1. 기존 AI 코드 에이전트의 구조적 한계

대부분의 AI 코딩 도구는 단일 에이전트(Single Agent)가 기획자, 개발자, 디버거, 리서처 역할을 **순차 실행(Serial Execution)** 방식으로 수행한다.

이 과정에서 아래와 같은 한계가 존재한다:

| 한계 | 설명 |
|------|------|
| **컨텍스트 윈도우 소모** | 역할 전환마다 집중력 분산, 윈도우가 급격히 소모 |
| **컨텍스트 오버로드** | 할루시네이션 발생 또는 작업 중단 |
| **단일 모델 취약점** | 한 모델의 약점이 전체 작업 품질을 좌우 |

---

## 2. Oh-My-OpenCode의 핵심: 오케스트레이터 기반 팀 구조

이 플러그인의 진짜 혁신은 **Sisyphus**라는 매니저 에이전트가 전문화된 하위 에이전트들에게 **병렬 실행(Parallel Execution)**으로 작업을 위임한다는 점이다.

```
                     ┌──────────────┐
                     │  Sisyphus    │  ← 오케스트레이터 (매니저)
                     │  (매니저)     │
                     └──────┬───────┘
                            │ 병렬 위임
              ┌─────────────┼─────────────┐──────────────┐
              │             │             │              │
       ┌──────▼──────┐ ┌───▼────┐ ┌──────▼──────┐ ┌─────▼──────┐
       │  Frontend   │ │Librarian│ │   Oracle    │ │  Document  │
       │  Engineer   │ │(리서처) │ │ (아키텍트)   │ │  Writer    │
       └─────────────┘ └────────┘ └─────────────┘ └────────────┘
        Gemini-3-Pro   Sonnet-4.5    GPT-5         Gemini-3-Flash
```

### 핵심 특징

- **컨텍스트 격리**: 각 에이전트의 컨텍스트가 코드 레벨에서 격리되어 단일 작업에만 집중. 이 부분이 **Context Rot**을 방지하는 차원에서 매우 중요하다.
- **모델 최적 배치**: 작업 특성에 맞는 모델을 선택적으로 할당

| 에이전트 | 역할 | 할당 모델 | 이유 |
|----------|------|-----------|------|
| **Oracle** | 아키텍처 설계 | GPT-5 | 고수준 추론 |
| **Librarian** | 증거 기반 리서치 | Claude Sonnet 4.5 | 정확한 코드 분석 |
| **Frontend Engineer** | 창의적 UI 생성 | Gemini-3-Pro | 창의적 생성 |
| **Document Writer** | 문서 작성 | Gemini-3-Flash | 빠른 텍스트 생성 |

---

## 3. Sisyphus 오케스트레이터의 설계 철학

Sisyphus는 단순한 역할 분담이 아닌, **프로그래밍으로 강제되는 작업 흐름(Workflow)**을 구현한다.

### Phase 구조

```
Phase 0: Intent Gate     → 사용자 의도 파악 및 분류
Phase 1: Planning        → 작업 분해 및 에이전트 할당
Phase 2: Parallel Exec   → 하위 에이전트 동시 실행
Phase 3: Completion      → 결과 통합 및 검증
```

### 병렬 실행 강제

- `createSisyphusAgent` 함수가 동적으로 Phase 0~3까지 프롬프트를 조립
- 코드에 `// CORRECT: Always background, always parallel` 주석과 함께 `background_task` 호출 패턴을 강제 주입
- **순차 실행(Serial Execution)을 구조적으로 차단**하고 모든 하위 작업을 동시 실행하도록 설계

## 4. Librarian 에이전트: 증거 기반 리서치의 구현

할루시네이션을 원천 차단하기 위한 가장 정교한 설계가 Librarian이다.

### 핵심 메커니즘

| 메커니즘 | 설명 |
|----------|------|
| **출처 강제** | 모든 주장에 GitHub Permalink 필수. "공식 문서 3번째 줄, GitHub 이슈 #1234, 소스코드 47번째 라인" |
| **요청 분석 블록** | Literal Request(문자 그대로의 요청)와 Actual Need(실제 필요)를 분리해 명시 |
| **Type 분류 검색** | Type A/B/C/D로 분류하여 GitHub Issues, 공식 문서, 소스코드를 병렬 검색 |
| **시간 필터** | 2024년 이전 정보 자동 거부, 2025년 최신 문서 우선 검색 강제 |

> **왜 중요한가?** 단순히 "검색해줘"가 아니라, 검증 가능한 증거를 기반으로만 답변하도록 구조적으로 강제한다. 할루시네이션이 발생할 여지 자체를 제거하는 설계다.

## 5. 프로그래밍으로 강제되는 완성도

가장 인상적인 부분은 **행동을 코드로 강제하는 메커니즘**이다.

### Todo Continuation Enforcer

에이전트가 작업 완료를 착각하는 순간을 방지한다.

```
session.idle 이벤트 감지
  → 남은 TODO 체크
  → "남은 작업이 있습니다. 계속하세요" 시스템 메시지 자동 주입
```

### Ralph Loop

에이전트가 명시적으로 `<promise>DONE</promise>` 태그를 출력할 때까지 반복 실행을 강제한다.

> **"생각"이 아닌 "증명"으로 완료를 판단한다.**

이것은 앞선 글([16번 — AI 루프와 채점 시스템](16_AI_루프와_채점_시스템.md))에서 다룬 "채점 시스템"의 구체적 구현 사례이기도 하다.

## 6. LSP 통합: IDE가 보는 방식으로 코드베이스 이해

최근 Claude Code에도 도입된 방식인데, 일반적인 grep 기반 검색과 달리 실제 **Language Server Protocol(LSP)** 클라이언트를 구현했다.

| 기능 | 설명 |
|------|------|
| **LSPClient 클래스** | typescript-language-server 같은 언어 서버와 직접 통신 |
| **프로토콜 구현** | Content-Length 헤더 + JSON-RPC 메시지 처리 |
| **도구 노출** | diagnostics, definition, references를 에이전트 도구로 직접 노출 |

VSCode/IntelliJ와 동일한 방식으로 코드를 이해한다. grep은 텍스트를 찾지만, LSP는 **의미**를 찾는다.

## 7. 계층적 컨텍스트 자동 주입 (Hierarchical Context Injection)

개발자가 매번 컨텍스트를 설명하지 않아도 AI가 프로젝트의 암묵지를 이해한다.

### 작동 방식

`findAgentsMdUp` 함수가 디렉토리 트리를 역방향 탐색한다.

예를 들어, `src/components/auth/LoginForm.tsx` 수정 시:

```
수집되는 컨텍스트:
├── src/AGENTS.md                      ← 전체 아키텍처 규칙
├── src/components/AGENTS.md           ← UI 컴포넌트 패턴
└── src/components/auth/AGENTS.md      ← 인증 도메인 보안 가이드
```

아키텍처 규칙, UI 패턴, 보안 가이드를 **코드 작성 전에** 에이전트에게 주입한다.

## 정리: 컨텍스트 엔지니어링의 핵심 패턴

Oh-My-OpenCode에서 발견한 패턴을 정리하면:

| # | 패턴 | 설명 | 효과 |
|---|------|------|------|
| 1 | **컨텍스트 격리** | 에이전트별 독립된 컨텍스트 윈도우 | Context Rot 방지 |
| 2 | **모델 최적 배치** | 작업 특성에 맞는 모델 할당 | 각 모델의 강점 극대화 |
| 3 | **병렬 실행 강제** | 코드 레벨에서 순차 실행 차단 | 처리 속도 극대화 |
| 4 | **증거 기반 답변** | 출처 없는 주장 구조적 차단 | 할루시네이션 제거 |
| 5 | **행동 강제** | TODO Enforcer, Ralph Loop | "완료 착각" 방지 |
| 6 | **의미 기반 검색** | LSP로 코드 구조 이해 | 정확한 코드 분석 |
| 7 | **계층적 주입** | 디렉토리별 AGENTS.md 자동 수집 | 암묵지 전달 |

### 왜 이것이 "컨텍스트 엔지니어링"인가?

단순히 "좋은 프롬프트를 쓰는 것"이 아니다. **컨텍스트를 구조적으로 설계하고, 코드로 강제하고, 모델 간 최적 분배하는 것**이다. 프롬프트 엔지니어링이 "무엇을 말할까"의 문제라면, 컨텍스트 엔지니어링은 "어떤 정보를 어떤 에이전트에게 어떤 타이밍에 줄까"의 문제다.

Cursor나 Claude Code 대비, 여러 모델의 강점을 동시에 조합하고 컨텍스트를 구조적으로 관리하며 작동 방식을 코드로 강제하는 **엔지니어링의 승리**를 보여준다. 커뮤니티 기반으로 빠르게 확산되는 지금, 이 접근법이 업계 표준이 될 가능성을 주목할 필요가 있다.
