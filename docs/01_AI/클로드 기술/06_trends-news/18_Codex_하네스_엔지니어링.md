# Codex 에이전트에게 100만 줄을 맡긴 OpenAI, 사람이 한 건 딱 5가지였습니다

> 에이전트가 일할 환경을 설계하는 능력이 곧 엔지니어링 실력이 된다

> 원문: https://lnkd.in/gRj6-5G3

---

## 배경

"하네스 엔지니어링"이라는 말이 갑자기 보이기 시작했다. 솔직히 처음엔 또 새 유행어인가 싶었는데, OpenAI가 공개한 블로그를 읽고 생각이 바뀌었다.

추론 모델이 **두뇌**라면, 하네스는 **손발**이다. 파일을 읽고 코드를 고치고 테스트를 돌리는 모든 행위가 하네스 안에서 일어난다.

OpenAI 팀은 2025년 8월 빈 저장소에서 시작해 **Codex 에이전트로만 100만 줄짜리 제품**을 만들었다. 수작업 대비 10분의 1 시간이 걸렸다고 밝혔다.

---

## 1. 에이전트가 못 보는 지식은 존재하지 않는 것과 동일하다

Google Docs 기획 문서도, Slack 합의사항도, 암묵지도 에이전트에겐 전부 보이지 않는다. 3개월 뒤 합류하는 신입과 동일한 상황이다.

그래서 **모든 결정을 저장소 안 마크다운과 ExecPlan(자기 완결형 설계 문서)으로 밀어넣었다.**

| 원칙 | 설명 |
|------|------|
| **자기 완결성** | 초보자가 읽고 끝까지 구현 가능해야 통과 기준 충족 |
| **단일 프롬프트** | 7시간 이상 연속 작업 사례 존재 |
| **저장소 = 유일한 진실** | 외부 문서 의존 제거 |

> **핵심 통찰**: 에이전트에게 "더 똑똑해져라"가 아니라, "볼 수 있는 곳에 모든 정보를 놓아라"가 답이다.

---

## 2. "더 노력해"가 아니라 "무슨 능력이 빠졌나"를 물어야 한다

초기 에이전트 속도가 느렸던 원인은 모델이 아니라 **환경**이었다. 실패할 때마다 이렇게만 질문했다:

> "어떤 능력이 빠져 있고, 에이전트가 읽고 검증할 수 있게 만들 방법은 뭔가?"

### "지루한(boring) 기술"이 에이전트에게 유리한 이유

- API 안정성이 높다
- 학습 데이터에서의 표현도가 높다 (에이전트가 잘 안다)
- 예: 외부 `p-limit` 대신 자체 동시성 헬퍼를 만들어 OpenTelemetry 100% 연동

---

## 3. 문서가 아니라 기계적 강제가 일관성을 지킨다

**불변 규칙만 기계로 강제**하고 구현은 에이전트 자율에 맡겼다.

```
강제하는 것 (기계적)          자율에 맡기는 것
─────────────────────────    ─────────────────────
데이터 경계에서 파싱 필수     라이브러리 선택
의존성 방향 검증              구현 방식
레이어 구조                   세부 로직
```

- 데이터 경계에서 파싱을 필수로 했지만 라이브러리 선택은 자유 → Codex가 알아서 **Zod**를 골랐다
- 도메인마다 **Providers → Service → Runtime → UI** 고정 레이어
- 의존성 방향을 **커스텀 린터로 검증** (린터 자체도 Codex가 작성)

> 이것은 [16번 — AI 루프와 채점 시스템](16_AI_루프와_채점_시스템.md)에서 다룬 "채점 도구와 채점 기준"의 실전 적용이다.

## 4. 에이전트에게 눈을 달아주면 자는 동안에도 일한다

**Chrome DevTools Protocol**을 연결해 DOM 스냅샷과 스크린샷을 Codex에 제공했다.

### 관측 스택 구조

```
작업 시작
  → git worktree 생성
  → 임시 관측 스택 기동
    ├── Victoria Logs (LogQL) → 에이전트가 직접 로그 조회
    └── Victoria Metrics (PromQL) → 메트릭 확인
  → 작업 전후 DOM 스냅샷 비교
  → 수정 루프 (깨끗해질 때까지 반복)
작업 종료
  → 관측 스택 자동 정리
```

- 단일 실행이 **6시간 넘게** 한 작업에 집중하는 경우도 정기적으로 관찰
- 사람이 자는 동안에도 에이전트가 시각적 피드백을 받으며 작업을 계속한다

## 5. 프로젝트 구조가 담긴 지도를 주되, 1,000페이지 매뉴얼은 주지 마라

**거대한 AGENTS.md 하나에 전부 담는 방식은 실패했다.**

matklad의 `ARCHITECTURE.md` 개념이 여기서 빛났다. 전체 구조를 짧게 조감하되 자주 바뀌지 않는 것만 담는 원칙이다.

| 문서 | 역할 | 원칙 |
|------|------|------|
| **ARCHITECTURE.md** | 코드맵 (조감도) | 자주 바뀌지 않는 경계만 명시 |
| **ExecPlan** | 실행 계획 (구현 상세) | 자기 완결형, 초보자도 따라할 수 있는 수준 |

## 정리: 하네스 엔지니어링의 5가지 원칙

| # | 원칙 | 핵심 |
|---|------|------|
| 1 | **보이게 하라** | 모든 지식을 저장소 안 마크다운으로 |
| 2 | **능력을 채워라** | 실패 원인은 모델이 아니라 환경 |
| 3 | **기계로 강제하라** | 문서 대신 린터와 테스트로 일관성 확보 |
| 4 | **눈을 달아줘라** | DOM 스냅샷, 로그, 메트릭으로 시각적 피드백 |
| 5 | **지도를 주되 매뉴얼은 주지 마라** | ARCHITECTURE.md로 경계만 명시 |

### 더 큰 그림

> 단순히 코드를 잘 쓰는 시대는 끝나고 있다. **에이전트가 일할 환경을 설계하는 능력**이 곧 엔지니어링 실력이 된다.

에이전트만으로 만든 시스템이 수년간 일관성을 유지할 수 있는지는 아직 아무도 모른다. OpenAI 스스로도 "배우는 중"이라고 밝혔다. 그러나 방향은 명확하다. 코드를 직접 쓰는 것에서, **코드를 쓰는 환경을 설계하는 것**으로 엔지니어링의 무게 중심이 이동하고 있다.
