# OpenAI Codex 앱이 나왔지만, 개발자들은 터미널로 돌아가고 있다

> 코드를 한 줄씩 읽는 시대는 끝났고, 변화를 감독하는 시대가 왔다

---

## 배경

OpenAI에서 Mac 전용 Codex 앱을 공개했다. worktree 내장, 병렬 에이전트, diff 중심 인터페이스. Agentic Coding 시대를 다시 선언한 셈이다.

그런데 재밌는 건, **이 앱이 풀려는 문제를 이미 다른 조합으로 해결하고 있는 개발자들이 꽤 있다**는 것이다.

---

## 1. 왜 diff 중심 UI인가

Codex 앱은 코드 에디터가 아니다. VS Code를 연결해서 띄워줄 뿐, 본체에서는 **diff만 보여준다.** 이게 의도된 설계인데, 요즘 개발 방식이 그렇게 바뀌고 있기 때문이다.

OpenClaw 개발자 Peter Steinberger는 최근 인터뷰에서 이렇게 말했다:

> "나는 읽지 않은 코드를 배포합니다(I ship code I don't read)."

충격적으로 들리지만 맥락을 알면 이해가 된다. AI 에이전트가 생성한 코드를 한 줄씩 읽는 대신, **변경된 부분만 확인하고 테스트가 통과하는지 검증**하는 방식으로 일한다. 한 달에 6,600개 이상의 커밋을 혼자 처리하면서.

| 변화 | 과거 | 현재 |
|------|------|------|
| **확인 방식** | 코드 전체 읽기 | diff로 변경점 확인 |
| **검증 방식** | 수동 리뷰 | 테스트 자동화 |
| **개발자 역할** | 코드 작성자 | 변경 감독자 |

---

## 2. 왜 이렇게 바뀌었을까

이유는 단순하다. **AI가 생성하는 코드 양이 사람이 읽을 수 있는 속도를 이미 넘어섰다.**

한 연구에 따르면 AI 도입 이후:
- PR 크기가 평균 **18% 증가**
- 변경 실패율도 **30% 상승**

더 많은 코드가 더 빠르게 생성되니까, 전체를 읽는 건 병목이 되어버렸다.

### 새로운 검증 방식

```
코드 한 줄씩 읽기 (X)

대신:
1. diff로 "뭐가 바뀌었나" 빠르게 파악
2. 테스트 자동화로 "동작하는가" 확인
3. 아키텍처 수준에서 "설계가 맞나" 판단
```

Claude Code 개발자 Boris도 비슷한 얘기를 한다:

> "AI에게 자기 작업을 검증할 방법을 주면 결과물 품질이 2~3배 올라갑니다."

중요한 건 코드를 읽는 게 아니라, **검증 루프를 만드는 것**이다.

---

## 3. 터미널 조합이 뜨고 있다

Codex 앱이 이런 워크플로우를 GUI로 제공한다면, 실무 개발자들은 **터미널에서 비슷한 환경을 이미 만들어 쓰고 있다.**

| 도구 | 역할 | 특징 |
|------|------|------|
| **ghostty** | GPU 가속 터미널 | Zig로 제작, 가볍고 빠르고 네이티브 느낌 |
| **lazygit** | 터미널 Git UI | 키보드만으로 stage, commit, diff 처리 |
| **git worktree** | 병렬 브랜치 작업 | 같은 저장소에서 여러 브랜치를 별도 디렉토리로 동시 작업 |

Boris 본인도 터미널에서 5개의 Claude Code 세션을 동시에 돌리는데, 각각 별도의 git checkout에서 작업한다. 복잡한 세팅 없이, 기본에 충실하게.

> **현실적 이슈**: CPU와 메모리 문제가 꽤 빈번하게 나온다. 브라우저 탭 많이 띄우는 사람은 더욱 그렇다. 그래서 최대한 터미널 환경으로만 진행하게 된다.

---

## 4. ghostty + Fish 4.1 조합이 게임 체인저인 이유

ghostty 최신 버전과 Fish 4.1 이상을 함께 쓰면, **터미널에서도 클릭으로 커서 이동이 가능**하다. 터미널의 고질적인 단점이 드디어 해결된 것이다.

Mitchell Hashimoto가 트윗으로 알렸는데, Kitty 이후 처음으로 이 기능을 지원하는 터미널이라고 한다.

> 직접 써보니까 진짜 편하다. 기존 코드 에디터를 무리해서 열어두지 않게 되었다.

---

## 5. 메모리 부담 줄이기

RAM 가격이 오른 시대에, 메모리 효율은 지나칠 문제가 아니다.

| 조합 | 메모리 부담 |
|------|------------|
| VS Code + Claude Code 여러 개 | 높음 (금방 한계) |
| ghostty + lazygit + Claude Code | 낮음 (네이티브 수준) |
| Zed + 터미널 | 중간 (Rust 기반으로 가벼움) |

여러 에이전트를 병렬로 돌릴 때 이 차이가 꽤 크다.

## 정리

### Agentic Coding의 핵심 원칙

```
코드를 읽는 것 (X)
  → diff로 변화를 확인하고
  → 테스트로 검증하고
  → 승인하거나 되돌리는 것 (O)
```

도구가 Codex 앱이든 터미널 조합이든, 이 원칙은 같다.

### 왜 터미널이 다시 뜨는가

| 이유 | 설명 |
|------|------|
| **가벼움** | 메모리 부담 최소화 |
| **빠름** | GPU 가속, 네이티브 성능 |
| **병렬 적합** | 여러 에이전트 동시 실행에 최적 |
| **커서 이동** | ghostty + Fish 4.1로 고질적 단점 해결 |

터미널 중심 개발이 다시 뜰 줄은 몰랐다. Vim을 고집하는 분들의 영역이라고 생각했다. 그런데 AI 에이전트 시대가 되니까, 오히려 터미널이 합리적인 선택이 됐다.

> **결론**: 도구가 중요한 게 아니다. 개인에게 맞는 도구가 최고다. 다만 이러한 개발자들의 움직임이 시사하는 바는 크다. **코드를 한 줄씩 읽는 시대는 끝났고, 변화를 감독하는 시대가 왔다.**
