# Claude Code, 드디어 치매 탈출? — Auto Memory 기능 등장

> CLAUDE.md와 다른, Claude가 스스로 쓰는 업무 일지

---

## 문제: 타임루프 NPC

Claude Code를 매일 쓰는 사람이라면 한 번쯤 이런 경험이 있다.

버그를 디버깅하는데 A 방법 시도 → 실패 → B 방법 시도 → 실패 → 그러더니 "A 방법을 시도해보겠습니다". 컨텍스트가 길어지면 앞에서 뭘 했는지 까먹고, 타임루프물에 갇힌 NPC처럼 같은 행동을 반복한다.

물론 이걸 해결하던 사람들도 있었다. CLAUDE.md에 "이 방법은 안 됨"을 수동으로 기록하거나, 세션마다 교훈을 정리해서 넘겨주거나. 영화 메멘토 주인공이 몸에 문신을 새기듯, 까먹을 걸 알기에 매번 직접 기억을 만들어주는 방식이었다.

그런데 어느 날, 버그 디버깅을 마치고 나니 갑자기 Claude가 이러더라.

> "메모리에 기록했습니다."

Claude Code에 **Auto Memory**라는 기능이 추가되어 있었다.

---

## Auto Memory란?

프로젝트별로 메모리 공간이 자동 생성되고, Claude가 알아서 작업 중 배운 것들을 기록한다. 다음 세션이 시작되면 이 메모리를 자동으로 읽고 시작한다.

단, 세션 전체를 통째로 기억하는 게 아니다. **200줄짜리 단기 기억에 뭘 남길지 스스로 판단한다.** "이 실수는 다음에도 할 수 있겠다" 싶은 것만 골라서 기록하는 구조다.

### 저장 위치

```
~/.claude/projects/<project-hash>/memory/
├── MEMORY.md          # 메인 메모리 (매 세션 자동 로드, 200줄 제한)
├── debugging.md       # 주제별 상세 노트 (선택)
├── patterns.md        # 주제별 상세 노트 (선택)
└── ...
```

- `MEMORY.md`가 매 세션 시작 시 시스템 프롬프트에 자동 주입된다
- 200줄 이후는 잘린다 → 핵심만 남겨야 한다
- 주제별 파일은 MEMORY.md에서 링크로 참조한다

---

## 어떻게 작동하는가?

Claude의 시스템 프롬프트에 다음 지침이 내장되어 있다:

> "As you work, consult your memory files to build on previous experience. When you encounter a mistake that seems like it could be common, check your auto memory for relevant notes — and if nothing is written yet, record what you learned."

**요약: 작업하면서 배운 걸 기록해라. 다음에 같은 실수 하지 마라.**

이것이 시스템 레벨에서 주입된다.

### 세부 규칙 (시스템 프롬프트 원문 기반)

| 규칙 | 설명 |
|------|------|
| **자동 로드** | `MEMORY.md`가 매 세션 시작 시 시스템 프롬프트에 포함 |
| **200줄 제한** | 200줄 이후는 잘림 → 간결하게 유지해야 함 |
| **주제별 분리** | 상세 내용은 별도 파일(debugging.md 등)로 분리, MEMORY.md에서 링크 |
| **자동 갱신** | 틀리거나 오래된 메모리는 알아서 업데이트하거나 삭제 |
| **의미 기반 정리** | 시간순이 아닌 주제별로 조직화 |

### 기록 대상 vs 비대상

| 기록 O | 기록 X |
|--------|--------|
| 여러 세션에서 확인된 안정적 패턴/컨벤션 | 세션별 임시 컨텍스트 (현재 작업, 진행 중인 상태) |
| 핵심 아키텍처 결정, 중요 파일 경로 | 불완전하거나 검증 안 된 정보 |
| 사용자의 워크플로우/도구/커뮤니케이션 선호도 | CLAUDE.md 지침과 중복되는 내용 |
| 반복되는 문제에 대한 해결책, 디버깅 인사이트 | 파일 하나 읽고 내린 추측 |

### 사용자 명시 요청

사용자가 직접 "이것 기억해줘"라고 말하면 → 여러 세션 확인 없이 즉시 저장한다. "이거 잊어줘"라고 하면 → 해당 항목을 찾아서 삭제한다.

## CLAUDE.md와 뭐가 다른가?

한 줄 정리:

- **CLAUDE.md** = 내가 쓰는 규칙서 → "이렇게 해라"
- **Auto Memory** = Claude가 쓰는 업무 일지 → "저번에 이래서 실패했다"

| 구분 | CLAUDE.md | Auto Memory |
|------|-----------|-------------|
| **작성자** | 사람 | Claude (자동) |
| **성격** | 규칙, 지침, 설정 | 경험, 교훈, 패턴 |
| **갱신** | 수동 편집 | Claude가 자동 갱신 |
| **범위** | 프로젝트 전체 규칙 | 작업 중 발견한 사실 |
| **예시** | "커밋 시 티켓번호 필수" | "Avro enum에 메서드 추가 불가 → 유틸 클래스로 분리" |

**상호 보완 관계**: CLAUDE.md는 "어떻게 행동하라"를 정의하고, Auto Memory는 "이전에 뭘 배웠다"를 축적한다. 둘 다 시스템 프롬프트에 주입되지만 역할이 다르다.

## 왜 임팩트가 큰가?

### 1. 세션 간 학습 연속성

세션이 바뀌어도 기억이 이어진다. 어제 삽질해서 알아낸 "이 라이브러리는 v3에서 API가 바뀌었다"를 오늘 세션에서 다시 삽질하지 않는다.

### 2. 멀티 에이전트 공유 기억

하나의 큰 작업을 여러 에이전트로 나눠서 돌릴 때, 한 에이전트가 배운 교훈이 같은 프로젝트의 다른 에이전트에게도 공유된다. A 에이전트가 삽질해서 알아낸 걸, B 에이전트가 같은 삽질을 안 해도 된다.

### 3. 수동 기록 부담 감소

이전에는 "이 방법 안 됨"을 CLAUDE.md에 사람이 직접 적어야 했다. 이제 Claude가 스스로 적는다. 사람은 규칙(CLAUDE.md)에만 집중하면 된다.

## 실제 Auto Memory 예시

실제로 축적된 메모리의 모습:

```markdown
# Auto Memory

## Key Patterns

### Redpanda Spring Boot PoC
- Avro enum은 메서드 추가 불가 → 비즈니스 로직은 별도 유틸리티 클래스로 분리
- Redpanda는 Schema Registry 내장 → 별도 컨테이너 불필요
- specific.avro.reader=true → SpecificRecord, false → GenericRecord

## Lessons Learned
- Google Drive 파일 이동 시 copy+delete가 mv보다 안전 (클라우드 동기화 충돌 방지)
- 학습 문서 검증 시 실제 프로젝트(build.gradle)와 대조 필수
- enable.idempotence=true → PID+시퀀스로 중복 감지, acks=all 강제
```

패턴(반복 사용하는 사실)과 교훈(실수에서 배운 것)이 주제별로 정리되어 있다.

## 메모리 검색

Auto Memory는 단순 로드뿐 아니라 검색도 가능하다. 시스템 프롬프트에 검색 방법이 포함되어 있다:

```bash
# 주제 파일 검색
Grep pattern="<검색어>" path="~/.claude/projects/<hash>/memory/" glob="*.md"

# 세션 로그 검색 (최후 수단 — 파일 크고 느림)
Grep pattern="<검색어>" path="~/.claude/projects/<hash>/" glob="*.jsonl"
```

좁은 검색어(에러 메시지, 파일 경로, 함수명)를 사용하라고 권장한다. 넓은 키워드는 비효율적이다.

## 한계와 주의점

| 한계 | 설명 |
|------|------|
| **200줄 제한** | MEMORY.md가 200줄을 넘으면 잘린다 → 정기적 정리 필요 |
| **판단 의존** | Claude가 "뭘 기록할지" 스스로 판단 → 중요한 걸 빠뜨릴 수 있음 |
| **검증 부재** | 기록된 내용이 정확한지 사람이 가끔 확인해야 함 |
| **프로젝트 범위** | 프로젝트별로 격리됨 → 프로젝트 A의 교훈이 프로젝트 B에는 안 간다 |
