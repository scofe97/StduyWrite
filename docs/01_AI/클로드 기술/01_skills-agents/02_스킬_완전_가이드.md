## 1. 소개

### Skill이란?

**Skill**은 Claude에게 특정 작업이나 워크플로우를 처리하는 방법을 가르치는 명령어 세트입니다. 간단한 폴더 형태로 패키징되며, Claude를 커스터마이징하는 가장 강력한 방법 중 하나입니다.

매번 대화에서 선호도, 프로세스, 도메인 전문성을 다시 설명하는 대신, Skill을 통해 한 번 가르치면 매번 그 혜택을 받을 수 있습니다.

### Skill이 효과적인 경우

- **반복 가능한 워크플로우**: 스펙에서 프론트엔드 디자인 생성, 일관된 방법론으로 리서치 수행
- **팀 스타일 가이드를 따르는 문서 작성**
- **다단계 프로세스 오케스트레이션**
- **MCP 통합과 함께 사용**: 원시 도구 접근을 신뢰할 수 있는 최적화된 워크플로우로 변환

### 이 가이드에서 배울 내용

- Skill 구조에 대한 기술 요구사항 및 모범 사례
- 독립형 Skill과 MCP 강화 워크플로우 패턴
- 다양한 사용 사례에서 효과적인 패턴
- Skill 테스트, 반복 및 배포 방법

### 대상 독자

- Claude가 특정 워크플로우를 일관되게 따르길 원하는 **개발자**
- 특정 워크플로우를 원하는 **파워 유저**
- 조직 전체에서 Claude 작업 방식을 표준화하려는 **팀**

### 예상 소요 시간

skill-creator를 사용하여 첫 번째 작동하는 Skill을 구축하고 테스트하는 데 약 **15-30분** 소요됩니다.

## 2. 기본 개념

### Skill의 구성 요소

Skill은 다음을 포함하는 폴더입니다:

| 파일/폴더 | 필수 여부 | 설명 |
|-----------|----------|------|
| `SKILL.md` | **필수** | YAML frontmatter가 포함된 Markdown 명령어 |
| `scripts/` | 선택 | 실행 가능한 코드 (Python, Bash 등) |
| `references/` | 선택 | 필요에 따라 로드되는 문서 |
| `assets/` | 선택 | 출력에 사용되는 템플릿, 폰트, 아이콘 |

### 핵심 설계 원칙

#### 1. Progressive Disclosure (점진적 공개)

Skill은 3단계 시스템을 사용합니다:

| 단계 | 위치 | 설명 |
|------|------|------|
| **1단계** | YAML frontmatter | 항상 Claude의 시스템 프롬프트에 로드됨. Skill 사용 시점 판단에 필요한 최소 정보 |
| **2단계** | SKILL.md 본문 | Claude가 관련 있다고 판단할 때 로드. 전체 명령어와 가이드 |
| **3단계** | 연결된 파일 | Skill 디렉토리 내 추가 파일. 필요시에만 탐색 및 발견 |

이 점진적 공개 방식은 **토큰 사용량을 최소화**하면서 전문 지식을 유지합니다.

#### 2. Composability (구성 가능성)

Claude는 **여러 Skill을 동시에 로드**할 수 있습니다. Skill은 유일한 기능이라고 가정하지 않고, 다른 Skill과 잘 작동해야 합니다.

#### 3. Portability (이식성)

Skill은 **Claude.ai, Claude Code, API에서 동일하게 작동**합니다. 환경이 Skill에 필요한 종속성을 지원한다면, 한 번 만들면 모든 표면에서 수정 없이 작동합니다.

### MCP 빌더를 위한: Skills + Connectors

#### Kitchen 비유

| MCP (연결성) | Skills (지식) |
|--------------|---------------|
| Claude를 서비스(Notion, Asana, Linear 등)에 연결 | Claude가 서비스를 효과적으로 사용하는 방법 교육 |
| 실시간 데이터 접근 및 도구 호출 제공 | 워크플로우와 모범 사례 캡처 |
| Claude가 **할 수 있는 것** | Claude가 **어떻게 해야 하는지** |

**MCP**는 전문 주방(도구, 재료, 장비 접근)을 제공하고, **Skills**는 레시피(가치 있는 것을 만드는 단계별 명령어)를 제공합니다.

#### MCP 사용자에게 이것이 중요한 이유

**Skill 없이:**
- 사용자가 MCP에 연결하지만 다음에 무엇을 해야 할지 모름
- "통합으로 X를 어떻게 하나요?"라는 지원 티켓
- 매 대화가 처음부터 시작
- 사용자마다 다른 프롬프팅으로 일관성 없는 결과
- 실제 문제는 워크플로우 가이드인데 커넥터 탓

**Skill과 함께:**
- 필요할 때 사전 구축된 워크플로우가 자동 활성화
- 일관되고 신뢰할 수 있는 도구 사용
- 모든 상호작용에 모범 사례 내장
- 통합에 대한 낮은 학습 곡선

## 3. 기획 및 설계

### 사용 사례로 시작하기

코드 작성 전에 Skill이 가능하게 해야 할 **2-3개의 구체적인 사용 사례**를 식별합니다.

#### 좋은 사용 사례 정의 예시

```
Use Case: 프로젝트 스프린트 계획
Trigger: 사용자가 "이 스프린트 계획 도와줘" 또는 "스프린트 작업 생성"이라고 말함
Steps:
1. MCP를 통해 현재 프로젝트 상태 가져오기
2. 팀 속도 및 용량 분석
3. 작업 우선순위 제안
4. Linear에 적절한 라벨과 추정치로 작업 생성
Result: 작업이 생성된 완전히 계획된 스프린트
```

#### 스스로에게 물어볼 질문

- 사용자가 무엇을 달성하고 싶어하는가?
- 이를 위해 어떤 다단계 워크플로우가 필요한가?
- 어떤 도구가 필요한가 (내장 또는 MCP)?
- 어떤 도메인 지식이나 모범 사례가 포함되어야 하는가?

### 일반적인 Skill 사용 사례 카테고리

#### Category 1: 문서 및 자산 생성

**용도**: 문서, 프레젠테이션, 앱, 디자인, 코드 등 일관된 고품질 출력 생성

**실제 예시**: frontend-design skill
> "높은 디자인 품질로 독특하고 프로덕션 수준의 프론트엔드 인터페이스 생성. 웹 컴포넌트, 페이지, 아티팩트, 포스터, 애플리케이션 구축 시 사용."

**핵심 기술**:
- 내장된 스타일 가이드 및 브랜드 표준
- 일관된 출력을 위한 템플릿 구조
- 최종화 전 품질 체크리스트
- 외부 도구 불필요 - Claude의 내장 기능 사용

#### Category 2: 워크플로우 자동화

**용도**: 일관된 방법론의 이점이 있는 다단계 프로세스, 여러 MCP 서버 간 조정 포함

**실제 예시**: skill-creator skill
> "새 Skill 생성을 위한 대화형 가이드. 사용 사례 정의, frontmatter 생성, 명령어 작성, 검증을 안내."

**핵심 기술**:
- 검증 게이트가 있는 단계별 워크플로우
- 일반 구조를 위한 템플릿
- 내장된 검토 및 개선 제안
- 반복적 개선 루프

#### Category 3: MCP 강화

**용도**: MCP 서버가 제공하는 도구 접근을 강화하는 워크플로우 가이드

**실제 예시**: sentry-code-review skill (Sentry)
> "Sentry의 MCP 서버를 통한 오류 모니터링 데이터를 사용하여 GitHub Pull Request에서 감지된 버그를 자동으로 분석하고 수정."

**핵심 기술**:
- 여러 MCP 호출을 순서대로 조정
- 도메인 전문 지식 내장
- 사용자가 달리 지정해야 할 컨텍스트 제공
- 일반적인 MCP 문제에 대한 오류 처리

### 성공 기준 정의

#### 정량적 지표

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| 관련 쿼리에서 Skill 트리거 | 90% | 10-20개 테스트 쿼리 실행, 자동 로드 vs 명시적 호출 추적 |
| X번의 도구 호출로 워크플로우 완료 | - | Skill 활성화/비활성화 상태에서 동일 작업 비교 |
| 워크플로우당 실패한 API 호출 | 0 | 테스트 실행 중 MCP 서버 로그 모니터링 |

#### 정성적 지표

- 사용자가 다음 단계에 대해 Claude에게 프롬프트할 필요 없음
- 사용자 수정 없이 워크플로우 완료
- 세션 간 일관된 결과

### 기술 요구사항

#### 파일 구조

```
your-skill-name/
├── SKILL.md           # 필수 - 메인 Skill 파일
├── scripts/           # 선택 - 실행 가능한 코드
│   ├── process_data.py
│   └── validate.sh
├── references/        # 선택 - 문서
│   ├── api-guide.md
│   └── examples/
└── assets/            # 선택 - 템플릿 등
    └── report-template.md
```

#### 핵심 규칙

**SKILL.md 네이밍:**
- 정확히 `SKILL.md`여야 함 (대소문자 구분)
- 변형 불가 (SKILL.MD, skill.md 등)

**Skill 폴더 네이밍:**
- kebab-case 사용: `notion-project-setup` ✅
- 공백 불가: `Notion Project Setup` ❌
- 언더스코어 불가: `notion_project_setup` ❌
- 대문자 불가: `NotionProjectSetup` ❌

**README.md 금지:**
- Skill 폴더 내에 README.md 포함하지 않음
- 모든 문서는 SKILL.md 또는 references/에 작성

### YAML Frontmatter: 가장 중요한 부분

YAML frontmatter는 Claude가 Skill을 로드할지 결정하는 방법입니다.

#### 최소 필수 형식

```yaml
---
name: your-skill-name
description: What it does. Use when user asks to [specific phrases].
---
```

#### 필드 요구사항

| 필드 | 필수 | 요구사항 |
|------|------|----------|
| `name` | **필수** | kebab-case만, 공백/대문자 없음, 폴더명과 일치해야 함 |
| `description` | **필수** | **무엇을 하는지** + **언제 사용하는지** 모두 포함, 1024자 미만, XML 태그 불가 |
| `license` | 선택 | 오픈소스용 (MIT, Apache-2.0 등) |
| `compatibility` | 선택 | 1-500자, 환경 요구사항 표시 |
| `metadata` | 선택 | 커스텀 키-값 쌍 (author, version, mcp-server 등) |

#### 보안 제한사항

**Frontmatter에서 금지:**
- XML 꺾쇠 괄호 (`<` `>`)
- 이름에 "claude" 또는 "anthropic" (예약됨)

**이유**: Frontmatter는 Claude의 시스템 프롬프트에 나타납니다. 악의적인 콘텐츠가 명령어를 주입할 수 있습니다.

### 효과적인 Skill 작성

#### Description 필드

**구조**: `[무엇을 하는지]` + `[언제 사용하는지]` + `[핵심 기능]`

**좋은 예시:**

```yaml
# 좋음 - 구체적이고 실행 가능
description: Figma 디자인 파일을 분석하고 개발자 핸드오프 문서를 생성합니다.
사용자가 .fig 파일을 업로드하거나, "디자인 스펙", "컴포넌트 문서",
"디자인-투-코드 핸드오프"를 요청할 때 사용합니다.

# 좋음 - 트리거 문구 포함
description: 스프린트 계획, 작업 생성, 상태 추적을 포함한 Linear 프로젝트
워크플로우를 관리합니다. 사용자가 "스프린트", "Linear 작업", "프로젝트 계획"을
언급하거나 "티켓 생성"을 요청할 때 사용합니다.
```

**나쁜 예시:**

```yaml
# 너무 모호
description: 프로젝트를 도와줍니다.

# 트리거 누락
description: 정교한 다중 페이지 문서 시스템을 생성합니다.

# 너무 기술적, 사용자 트리거 없음
description: 계층적 관계가 있는 Project 엔티티 모델을 구현합니다.
```

#### 메인 명령어 작성

Frontmatter 후에 Markdown으로 실제 명령어를 작성합니다.

**권장 구조:**

```markdown
---
name: your-skill
description: [...]
---

# Your Skill Name

## Instructions

### Step 1: [첫 번째 주요 단계]
무엇이 일어나는지 명확한 설명.

Example:
```bash
python scripts/fetch_data.py --project-id PROJECT_ID
```
Expected output: [성공이 어떻게 보이는지 설명]

(필요에 따라 더 많은 단계 추가)

## Examples

### Example 1: [일반적인 시나리오]
User says: "새 마케팅 캠페인 설정해줘"
Actions:
1. MCP를 통해 기존 캠페인 가져오기
2. 제공된 매개변수로 새 캠페인 생성
Result: 확인 링크와 함께 캠페인 생성됨

(더 많은 예시 추가)

## Troubleshooting

### Error: [일반적인 오류 메시지]
Cause: [왜 발생하는지]
Solution: [어떻게 수정하는지]

(더 많은 오류 케이스 추가)
```

#### 명령어 작성 모범 사례

**1. 구체적이고 실행 가능하게**

```markdown
# ✅ 좋음
`python scripts/validate.py --input {filename}`을 실행하여 데이터 형식을 확인합니다.
검증 실패 시 일반적인 문제:
- 필수 필드 누락 (CSV에 추가)
- 잘못된 날짜 형식 (YYYY-MM-DD 사용)

# ❌ 나쁨
진행하기 전에 데이터를 검증하세요.
```

**2. 오류 처리 포함**

```markdown
## Common Issues

### MCP Connection Failed
"Connection refused" 오류 발생 시:
1. MCP 서버 연결 확인: Settings > Extensions
2. API 키 유효성 확인
3. 재연결 시도: Settings > Extensions > [Your Service] > Reconnect
```

**3. 번들된 리소스 명확히 참조**

```markdown
쿼리 작성 전, `references/api-patterns.md`에서 다음을 참조:
- Rate limiting 가이드
- Pagination 패턴
- 오류 코드 및 처리
```

**4. Progressive disclosure 사용**

SKILL.md는 핵심 명령어에 집중합니다. 상세 문서는 `references/`로 이동하고 링크합니다.

## 4. 테스트 및 반복

### 테스트 접근 방식

| 방법 | 설명 | 용도 |
|------|------|------|
| Claude.ai에서 수동 테스트 | 쿼리를 직접 실행하고 동작 관찰 | 빠른 반복, 설정 불필요 |
| Claude Code에서 스크립트 테스트 | 변경 사항에 걸쳐 반복 가능한 검증을 위한 테스트 케이스 자동화 | |
| Skills API를 통한 프로그래밍 테스트 | 정의된 테스트 세트에 대해 체계적으로 실행되는 평가 스위트 구축 | |

> **Pro Tip**: 확장 전에 단일 작업에서 반복하세요. 가장 효과적인 Skill 제작자는 Claude가 성공할 때까지 단일 도전적인 작업에서 반복한 다음, 승리한 접근 방식을 Skill로 추출합니다.

### 권장 테스트 접근 방식

#### 1. 트리거 테스트

**목표**: Skill이 적절한 시점에 로드되는지 확인

```
Should trigger:
- "새 ProjectHub 워크스페이스 설정 도와줘"
- "ProjectHub에서 프로젝트 만들어야 해"
- "Q4 계획을 위한 ProjectHub 프로젝트 초기화"

Should NOT trigger:
- "샌프란시스코 날씨 어때?"
- "Python 코드 작성 도와줘"
- "스프레드시트 만들어줘"
```

#### 2. 기능 테스트

**목표**: Skill이 올바른 출력을 생성하는지 확인

```
Test: 5개 작업으로 프로젝트 생성
Given: 프로젝트 이름 "Q4 Planning", 5개 작업 설명
When: Skill이 워크플로우 실행
Then:
  - ProjectHub에 프로젝트 생성됨
  - 올바른 속성으로 5개 작업 생성됨
  - 모든 작업이 프로젝트에 연결됨
  - API 오류 없음
```

#### 3. 성능 비교

**목표**: Skill이 기준선 대비 결과를 개선하는지 증명

```
Without skill:
- 사용자가 매번 명령어 제공
- 15번의 주고받는 메시지
- 재시도 필요한 3번의 실패한 API 호출
- 12,000 토큰 소비

With skill:
- 자동 워크플로우 실행
- 명확화 질문 2개만
- 실패한 API 호출 0
- 6,000 토큰 소비
```

### skill-creator Skill 사용하기

skill-creator는 Claude.ai 플러그인 디렉토리 또는 Claude Code 다운로드를 통해 사용 가능합니다.

**기능:**
- 자연어 설명에서 Skill 생성
- 적절한 형식의 SKILL.md와 frontmatter 생성
- 트리거 문구와 구조 제안
- 일반적인 문제 플래그 (모호한 설명, 누락된 트리거, 구조 문제)
- 잠재적 과소/과다 트리거 위험 식별
- Skill의 명시된 목적에 기반한 테스트 케이스 제안

**사용 방법:**
```
"skill-creator Skill을 사용해서 [사용 사례]를 위한 Skill 구축을 도와줘"
```

### 피드백 기반 반복

Skill은 살아있는 문서입니다. 다음을 기반으로 반복 계획을 세우세요:

#### Under-triggering 신호

- Skill이 로드되어야 할 때 로드되지 않음
- 사용자가 수동으로 활성화
- 언제 사용해야 하는지에 대한 지원 질문

**해결책**: Description에 더 많은 세부 사항과 뉘앙스 추가 - 특히 기술 용어에 대한 키워드 포함

#### Over-triggering 신호

- 관련 없는 쿼리에 Skill 로드
- 사용자가 비활성화
- 목적에 대한 혼란

**해결책**: 부정 트리거 추가, 더 구체적으로

#### 실행 문제

- 일관성 없는 결과
- API 호출 실패
- 사용자 수정 필요

**해결책**: 명령어 개선, 오류 처리 추가

### 오픈 표준

Anthropic은 Agent Skills를 오픈 표준으로 발표했습니다. MCP처럼 Skill은 도구와 플랫폼 간에 이식 가능해야 합니다 - 동일한 Skill이 Claude 또는 다른 AI 플랫폼을 사용하든 작동해야 합니다.

### API를 통한 Skill 사용

프로그래밍 사용 사례 (애플리케이션, 에이전트, 자동화 워크플로우 구축)의 경우 API가 Skill 관리 및 실행에 대한 직접 제어를 제공합니다.

**주요 기능:**
- `/v1/skills` 엔드포인트로 Skill 목록 조회 및 관리
- Messages API 요청에 `container.skills` 매개변수로 Skill 추가
- Claude Console을 통한 버전 관리
- 커스텀 에이전트 구축을 위한 Claude Agent SDK와 함께 작동

| 사용 사례 | 최적 표면 |
|----------|----------|
| 최종 사용자가 Skill과 직접 상호작용 | Claude.ai / Claude Code |
| 개발 중 수동 테스트 및 반복 | Claude.ai / Claude Code |
| 개별, 임시 워크플로우 | Claude.ai / Claude Code |
| Skill을 프로그래밍 방식으로 사용하는 애플리케이션 | API |
| 대규모 프로덕션 배포 | API |
| 자동화 파이프라인 및 에이전트 시스템 | API |

> **참고**: API의 Skill은 Code Execution Tool 베타가 필요합니다.

### 권장 접근 방식

1. **GitHub에 호스팅**
   - 오픈소스 Skill을 위한 공개 저장소
   - 설치 지침이 있는 명확한 README
   - 예시 사용 및 스크린샷

2. **MCP 저장소에 문서화**
   - MCP 문서에서 Skill 링크
   - 두 가지를 함께 사용하는 가치 설명
   - 빠른 시작 가이드 제공

3. **설치 가이드 생성**

```markdown
## [서비스] Skill 설치

1. Skill 다운로드:
   - 저장소 클론: `git clone https://github.com/yourcompany/skills`
   - 또는 Releases에서 ZIP 다운로드

2. Claude에 설치:
   - Claude.ai > Settings > Skills 열기
   - "Upload skill" 클릭
   - Skill 폴더 선택 (압축)

3. Skill 활성화:
   - [서비스] Skill 토글 켜기
   - MCP 서버 연결 확인

4. 테스트:
   - Claude에게: "[서비스]에서 새 프로젝트 설정해줘"
```

## 6. 패턴 및 트러블슈팅

### 접근 방식 선택: 문제 우선 vs 도구 우선

| 접근 방식 | 설명 |
|----------|------|
| **문제 우선** | "프로젝트 워크스페이스 설정 필요" → Skill이 올바른 순서로 올바른 MCP 호출 오케스트레이션 |
| **도구 우선** | "Notion MCP 연결됨" → Skill이 Claude에게 최적의 워크플로우와 모범 사례 교육 |

### 패턴 1: 순차적 워크플로우 오케스트레이션

**사용 시점**: 사용자가 특정 순서의 다단계 프로세스가 필요할 때

```markdown
## Workflow: 신규 고객 온보딩

### Step 1: 계정 생성
Call MCP tool: `create_customer`
Parameters: name, email, company

### Step 2: 결제 설정
Call MCP tool: `setup_payment_method`
Wait for: 결제 방법 확인

### Step 3: 구독 생성
Call MCP tool: `create_subscription`
Parameters: plan_id, customer_id (Step 1에서)

### Step 4: 환영 이메일 발송
Call MCP tool: `send_email`
Template: welcome_email_template
```

**핵심 기술**:
- 명시적 단계 순서
- 단계 간 종속성
- 각 단계에서 검증
- 실패 시 롤백 명령어

### 패턴 2: 다중 MCP 조정

**사용 시점**: 워크플로우가 여러 서비스에 걸쳐 있을 때

```markdown
### Phase 1: 디자인 내보내기 (Figma MCP)
1. Figma에서 디자인 자산 내보내기
2. 디자인 사양 생성
3. 자산 매니페스트 생성

### Phase 2: 자산 저장 (Drive MCP)
1. Drive에 프로젝트 폴더 생성
2. 모든 자산 업로드
3. 공유 가능한 링크 생성

### Phase 3: 작업 생성 (Linear MCP)
1. 개발 작업 생성
2. 작업에 자산 링크 첨부
3. 엔지니어링 팀에 할당

### Phase 4: 알림 (Slack MCP)
1. #engineering에 핸드오프 요약 게시
2. 자산 링크 및 작업 참조 포함
```

**핵심 기술**:
- 명확한 단계 분리
- MCP 간 데이터 전달
- 다음 단계로 이동 전 검증
- 중앙 집중식 오류 처리

### 패턴 3: 반복적 개선

**사용 시점**: 출력 품질이 반복으로 개선될 때

```markdown
## 반복적 보고서 생성

### 초기 초안
1. MCP를 통해 데이터 가져오기
2. 첫 번째 초안 보고서 생성
3. 임시 파일에 저장

### 품질 검사
1. 검증 스크립트 실행: `scripts/check_report.py`
2. 문제 식별:
   - 누락된 섹션
   - 일관성 없는 형식
   - 데이터 검증 오류

### 개선 루프
1. 식별된 각 문제 해결
2. 영향받는 섹션 재생성
3. 재검증
4. 품질 임계값 충족까지 반복

### 최종화
1. 최종 형식 적용
2. 요약 생성
3. 최종 버전 저장
```

### 패턴 4: 컨텍스트 인식 도구 선택

**사용 시점**: 동일한 결과, 컨텍스트에 따라 다른 도구

```markdown
## 스마트 파일 저장

### 결정 트리
1. 파일 유형 및 크기 확인
2. 최적 저장 위치 결정:
   - 대용량 파일 (>10MB): 클라우드 스토리지 MCP 사용
   - 협업 문서: Notion/Docs MCP 사용
   - 코드 파일: GitHub MCP 사용
   - 임시 파일: 로컬 스토리지 사용

### 저장 실행
결정에 따라:
- 적절한 MCP 도구 호출
- 서비스별 메타데이터 적용
- 접근 링크 생성

### 사용자에게 컨텍스트 제공
해당 저장소가 선택된 이유 설명
```

### 패턴 5: 도메인별 인텔리전스

**사용 시점**: Skill이 도구 접근 이상의 전문 지식을 추가할 때

```markdown
## 컴플라이언스와 결제 처리

### 처리 전 (컴플라이언스 검사)
1. MCP를 통해 거래 세부 정보 가져오기
2. 컴플라이언스 규칙 적용:
   - 제재 목록 확인
   - 관할권 허용 확인
   - 위험 수준 평가
3. 컴플라이언스 결정 문서화

### 처리
IF 컴플라이언스 통과:
  - 결제 처리 MCP 도구 호출
  - 적절한 사기 검사 적용
  - 거래 처리
ELSE:
  - 검토 플래그
  - 컴플라이언스 케이스 생성

### 감사 추적
- 모든 컴플라이언스 검사 로그
- 처리 결정 기록
- 감사 보고서 생성
```

### 트러블슈팅

#### Skill이 업로드되지 않음

| 오류 | 원인 | 해결책 |
|------|------|--------|
| "Could not find SKILL.md in uploaded folder" | 파일이 정확히 SKILL.md로 명명되지 않음 | SKILL.md로 이름 변경 (대소문자 구분) |
| "Invalid frontmatter" | YAML 형식 문제 | `---` 구분자 확인, 따옴표 닫힘 확인 |
| "Invalid skill name" | 이름에 공백 또는 대문자 | kebab-case 사용 |

#### Skill이 트리거되지 않음

**증상**: Skill이 자동으로 로드되지 않음

**수정**: Description 필드 수정
- 너무 일반적인가?
- 사용자가 실제로 말할 트리거 문구 포함?
- 관련 파일 유형 언급?

**디버깅 접근**: Claude에게 "[skill name] Skill을 언제 사용하겠어?"라고 물어보세요.

#### Skill이 너무 자주 트리거됨

**증상**: 관련 없는 쿼리에 Skill 로드

**해결책**:

1. **부정 트리거 추가**
```yaml
description: CSV 파일에 대한 고급 데이터 분석. 통계 모델링, 회귀,
클러스터링에 사용. 간단한 데이터 탐색에는 사용하지 않음
(대신 data-viz Skill 사용).
```

2. **더 구체적으로**
```yaml
# 너무 광범위
description: 문서를 처리합니다.

# 더 구체적
description: 계약 검토를 위한 PDF 법률 문서를 처리합니다.
```

3. **범위 명확화**
```yaml
description: 이커머스용 PayFlow 결제 처리. 온라인 결제 워크플로우에
특별히 사용, 일반 금융 쿼리에는 사용하지 않음.
```

#### MCP 연결 문제

**증상**: Skill은 로드되지만 MCP 호출 실패

**체크리스트**:
1. MCP 서버 연결 확인 (Settings > Extensions)
2. 인증 확인 (API 키 유효성, 권한/스코프, OAuth 토큰)
3. MCP 독립적 테스트 ("Skill 없이 [서비스] MCP로 프로젝트 가져와줘")
4. 도구 이름 확인 (대소문자 구분)

#### 명령어가 따라지지 않음

**증상**: Skill 로드되지만 Claude가 명령어를 따르지 않음

**일반적인 원인 및 해결책**:

1. **명령어가 너무 장황**: 간결하게, 글머리 기호와 번호 목록 사용
2. **명령어가 묻힘**: 중요한 명령어를 상단에, ## Important 헤더 사용
3. **모호한 언어**:
   ```markdown
   # 나쁨
   적절하게 검증하세요

   # 좋음
   CRITICAL: create_project 호출 전 확인:
   - 프로젝트 이름이 비어있지 않음
   - 최소 한 명의 팀 멤버 할당
   - 시작 날짜가 과거가 아님
   ```

4. **모델 "게으름"**: 명시적 격려 추가
   ```markdown
   ## Performance Notes
   - 철저히 하세요
   - 속도보다 품질이 중요
   - 검증 단계를 건너뛰지 마세요
   ```

#### 대용량 컨텍스트 문제

**증상**: Skill이 느리거나 응답 품질 저하

**원인**:
- Skill 콘텐츠가 너무 큼
- 너무 많은 Skill이 동시에 활성화
- Progressive disclosure 대신 모든 콘텐츠 로드

**해결책**:
1. **SKILL.md 크기 최적화**: 상세 문서를 references/로 이동, 인라인 대신 링크, SKILL.md를 5,000 단어 미만으로 유지
2. **활성화된 Skill 줄이기**: 20-50개 이상 동시 활성화 평가, 선택적 활성화 권장, 관련 기능을 위한 Skill "팩" 고려

## 부록 B: YAML Frontmatter 참조

### 필수 필드

```yaml
---
name: skill-name-in-kebab-case
description: 무엇을 하고 언제 사용하는지. 특정 트리거 문구 포함.
---
```

### 모든 선택 필드

```yaml
name: skill-name
description: [필수 설명]
license: MIT                    # 오픈소스용 라이선스
allowed-tools: "Bash(python:*) Bash(npm:*) WebFetch"  # 도구 접근 제한
metadata:                       # 커스텀 필드
  author: Company Name
  version: 1.0.0
  mcp-server: server-name
  category: productivity
  tags: [project-management, automation]
  documentation: https://example.com/docs
  support: support@example.com
```

### 보안 참고사항

**허용:**
- 모든 표준 YAML 유형 (문자열, 숫자, 불리언, 리스트, 객체)
- 커스텀 메타데이터 필드
- 긴 description (최대 1024자)

**금지:**
- XML 꺾쇠 괄호 (`<` `>`) - 보안 제한
- YAML에서 코드 실행 (안전한 YAML 파싱 사용)
- "claude" 또는 "anthropic" 접두사가 있는 Skill 이름 (예약됨)
