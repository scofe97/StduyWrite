# 에이전트 R&R 설정 가이드

> 에이전트를 지휘하기 위한 Role & Responsibility 설정법

---

## 핵심 요약

- **R&R (Role & Responsibility)** 설정이 에이전트 활용의 핵심
- 프로젝트 **지휘자** 마인드셋 필요
- Role별 **세부 규칙 주입**으로 범용 도구 대비 5~10배 생산성 향상
- **Arena(영역)** 설정으로 작업 충돌 방지

---

## R&R 설정의 중요성

### 경력보다 중요한 것

> "경력의 문제가 아니야. 주니어든 뭐든 중요한 건 프로젝트 지휘자라는 느낌으로 접근하는 것"

- 주니어/시니어 상관없이 **지휘자 마인드셋** 필요
- "이 작업은 누구(어떤 Role)한테 맡겨야 하나?" 판단 능력

---

## 범용 도구 vs 맞춤형 설정

### 범용 도구의 한계

```
범용 도구 사용 시 에이전트 동작:
1. 코드 패턴 학습 (시간 소요)
2. 훅 위치 파악
3. 컴포넌트 구조 분석
4. 규칙 추론
5. 그제서야 작업 시작
```

**문제점**: 매번 학습 과정 반복, 기존 패턴과 불일치 위험

### 맞춤형 설정의 장점

```
맞춤형 설정 시:
1. 미리 주입된 규칙 참조
2. 즉시 작업 시작
3. 일관된 결과물
```

**결과**: 생산성 **5~10배** 향상

## Role별 세부 규칙 주입

### UI 에이전트 설정 예시

| 항목 | 설정 내용 |
|------|----------|
| **브랜드 톤** | 회사 브랜드 가이드라인 |
| **컬러 프리셋** | 사용 가능한 색상 팔레트 |
| **디자인 토큰** | 토큰 파일 위치, 사용법 |
| **컴포넌트 패턴** | 기존 컴포넌트 구조 |
| **개발 컨벤션** | 네이밍, 폴더 구조 규칙 |

### 설정 문서 구조 예시

```markdown
# UI Agent Rules

## 브랜드 톤
- Primary Color: #1E40AF
- Secondary Color: #64748B
- Font: Pretendard

## 디자인 토큰 위치
- `src/styles/tokens.css`
- `src/styles/variables.scss`

## 컴포넌트 규칙
- Atomic Design 패턴 적용
- `src/components/{atoms|molecules|organisms}/`

## 개발 컨벤션
- 컴포넌트: PascalCase
- 스타일: CSS Modules
- 상태관리: Zustand
```

## Arena(영역) 설정

### 작업 영역 분리의 중요성

> "Arena를 잘 설정해야 한다. 못 한다면 에이전트를 잘 못 쓰고 있는 것"

### Arena 분리 예시

```
Project Arena Map
├── UI Arena
│   ├── 컴포넌트 개발
│   ├── 스타일링
│   └── 애니메이션
├── Backend Arena
│   ├── API 엔드포인트
│   ├── 비즈니스 로직
│   └── 데이터 처리
├── QA Arena
│   ├── 테스트 작성
│   ├── 테스트 실행
│   └── 버그 리포팅
└── DevOps Arena
    ├── CI/CD
    ├── 인프라
    └── 모니터링
```

### Arena 설정 원칙

1. **최대한 독립적으로** - 영역 간 교차 최소화
2. **명확한 경계** - 어떤 파일/폴더가 어느 Arena인지 정의
3. **충돌 방지** - 같은 파일을 여러 에이전트가 동시 수정하지 않도록

## 실전 적용

### CLAUDE.md 활용

```markdown
# Agent Roles Configuration

## UI Agent
- Arena: src/components/, src/styles/
- Rules: @docs/ui-conventions.md
- Design System: @docs/design-tokens.md

## Backend Agent
- Arena: src/api/, src/services/
- Rules: @docs/api-conventions.md
- Database: @docs/schema.md

## QA Agent
- Arena: tests/, __tests__/
- Rules: @docs/testing-guide.md
- Coverage: minimum 80%
```

### Role 전환 프롬프트

```
지금부터 너는 UI 엔지니어야.
다음 규칙을 따라:
- 디자인 시스템: docs/design-system.md 참조
- 컴포넌트 패턴: Atomic Design
- 스타일: CSS Modules 사용
- 브랜드 컬러: Primary #1E40AF
```

## 협의체 구조

여러 에이전트가 **협의체처럼 작동**:

```
Project Council
├── UI Agent: "이 컴포넌트 수정했습니다"
├── Backend Agent: "API 스펙 변경 알림"
├── QA Agent: "테스트 실패 보고"
└── Coordinator (You): 의사결정 및 조율
```

---

## 생산성 향상 공식

```
기존 생산성 × 에이전트 수 × R&R 최적화 = 최종 생산성

예시:
- 기존: 1x
- 에이전트 5개: 5x
- R&R 최적화: 2x
- 최종: 10x
```
