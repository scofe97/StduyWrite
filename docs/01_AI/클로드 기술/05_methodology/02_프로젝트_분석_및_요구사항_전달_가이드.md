# AI 코딩 에이전트 협업 베스트 프랙티스

> 리서치/계획/구현 워크플로우를 위한 폴더 구조 및 실무 가이드

## 개요

이 문서는 [복잡한 코드베이스에서 AI 활용하기](./01_복잡한_코드베이스에서_AI_활용하기.md)에서 소개한 **리서치/계획/구현 워크플로우**의 베스트 프랙티스를 정리합니다.

---

## 1. 권장 폴더 구조

### 최적의 구조

```
project-root/
│
├── CLAUDE.md                       # 🧠 프로젝트 두뇌 (필수)
│
├── .cursorrules                    # Cursor IDE 규칙 (선택)
│
├── specs/                          # 📋 AI 협업 스펙 폴더
│   │
│   ├── research/                   # 📚 리서치 (코드베이스 분석)
│   │   ├── _archive/               # 완료된 리서치 보관
│   │   ├── feature-user-auth.md
│   │   └── bug-payment-error.md
│   │
│   ├── plans/                      # 📝 구현 계획
│   │   ├── _archive/               # 완료된 계획 보관
│   │   ├── feature-user-auth.md
│   │   └── refactor-api-layer.md
│   │
│   └── decisions/                  # 🎯 기술 결정 기록 (ADR)
│       ├── 001-use-mapstruct.md
│       └── 002-adopt-hexagonal.md
│
└── src/                            # 소스 코드
```

### 폴더별 역할

| 폴더 | 역할 | 보관 기간 |
|------|------|-----------|
| `CLAUDE.md` | 프로젝트 전체 컨텍스트, 아키텍처, 규칙 | **영구** (지속 업데이트) |
| `specs/research/` | 기능별 코드 분석 결과 | 구현 완료 후 아카이브 |
| `specs/plans/` | 단계별 구현 계획 | 구현 완료 후 아카이브 |
| `specs/decisions/` | 중요한 기술 결정 (ADR) | **영구** |

## 3. 리서치/계획/구현 워크플로우

### 3단계 프로세스

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   📚 1단계: 리서치        📝 2단계: 계획         💻 3단계: 구현          │
│   ┌─────────────┐        ┌─────────────┐        ┌─────────────┐        │
│   │ 코드베이스  │   →    │ 실행 계획   │   →    │    코드     │        │
│   │   분석      │        │   수립      │        │    작성     │        │
│   └─────────────┘        └─────────────┘        └─────────────┘        │
│         ↓                      ↓                      ↓                │
│   specs/research/        specs/plans/            커밋/PR               │
│                                                                         │
│   ⭐⭐⭐ 최고 레버리지      ⭐⭐ 높은 레버리지       ⭐ 기본              │
│   (여기서 실수하면        (여기서 실수하면        (수정 비용 낮음)       │
│    수천 줄 낭비)           수백 줄 낭비)                                │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 세션 분리 전략

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        컨텍스트 활용률 관리                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  세션 1 (리서치)          세션 2 (계획)          세션 3 (구현)          │
│  ┌─────────────┐         ┌─────────────┐        ┌─────────────┐        │
│  │ 40-60%      │         │ 40-60%      │        │ 40-60%      │        │
│  │ 활용률 유지  │         │ 활용률 유지  │        │ 활용률 유지  │        │
│  └─────────────┘         └─────────────┘        └─────────────┘        │
│        ↓                       ↓                      ↓                │
│  research/*.md 저장      plans/*.md 저장         코드 커밋             │
│        ↓                       ↓                      ↓                │
│     세션 종료     →      새 세션에서 참조  →    새 세션에서 참조        │
│                                                                         │
│  💡 핵심: 각 단계 결과를 파일로 저장 → 다음 세션에서 로드               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 4. 단계별 프롬프트 템플릿

### 📚 1단계: 리서치 프롬프트

```
@CLAUDE.md

[기능명/버그명]을 분석해주세요.

분석할 내용:
1. 관련 코드 파일 및 위치
2. 현재 구현 방식 및 흐름
3. 유사 기능 참고점
4. 수정 시 영향 범위
5. 주의해야 할 패턴/규칙

결과를 specs/research/[name].md로 저장해주세요.
```

### 📝 2단계: 계획 프롬프트

```
@CLAUDE.md
@specs/research/[name].md

리서치를 바탕으로 구현 계획을 수립해주세요.

계획에 포함할 내용:
1. 변경할 파일 목록
2. 단계별 구현 순서 (Step 1, 2, 3...)
3. 각 단계별 상세 내용
4. 테스트 계획
5. 리스크 및 대응

결과를 specs/plans/[name].md로 저장해주세요.
```

### 💻 3단계: 구현 프롬프트

```
@CLAUDE.md
@specs/plans/[name].md

계획에 따라 [Step N]을 진행해주세요.

※ 한 번에 한 Step씩 진행
※ 완료 후 다음 Step 요청
```

## 7. 세션별 컨텍스트 관리

### ✅ 좋은 예

```
# 세션 시작 시 필요한 파일만 로드
@CLAUDE.md
@specs/plans/feature-auth.md

Step 2를 진행해주세요.
```

### ❌ 나쁜 예

```
# 모든 것을 한 번에 요청
전체 프로젝트를 분석하고 인증 기능을 처음부터 끝까지 구현해주세요.
```

### 작업 규모별 세션 분리

| 작업 규모 | 권장 세션 수 | 방식 |
|-----------|-------------|------|
| 간단한 버그 수정 | 1 세션 | 바로 수정 |
| 중간 규모 기능 | 2-3 세션 | 리서치 → 계획 → 구현 |
| 대규모 기능 | 3-5 세션 | 리서치 → 계획 → Step별 구현 |
| 리팩토링 | 3+ 세션 | 영향 분석 → 계획 → 단계별 수정 |

### 문서 상태 관리

```
specs/
├── research/
│   ├── feature-auth.md          # 🔄 진행 중
│   ├── feature-payment.md       # 🔄 진행 중

│       └── 2025-01/

│
└── plans/
    ├── feature-auth.md          # 🔄 진행 중

### 문서 보관 규칙

| 문서 유형 | 완료 후 처리 | 이유 |
|-----------|-------------|------|
| research/ | `_archive/YYYY-MM/`로 이동 | 나중에 유사 기능 참고 |
| plans/ | `_archive/YYYY-MM/`로 이동 | 의사결정 히스토리 |
| decisions/ | 영구 보관 | ADR은 삭제하지 않음 |

## 9. 핵심 원칙 요약

### 레버리지 피라미드

```
                    ▲
                   /│\
                  / │ \
                 /  │  \      ⭐⭐⭐ 리서치 리뷰
                /   │   \     (오류 시 수천 줄 낭비)
               /────┼────\
              /     │     \
             /      │      \   ⭐⭐ 계획 리뷰
            /       │       \  (오류 시 수백 줄 낭비)
           /────────┼────────\
          /         │         \
         /          │          \ ⭐ 코드 리뷰
        /           │           \(수정 비용 낮음)
       ─────────────┴─────────────

    💡 높은 곳에서 시간을 투자할수록 효율적
```

### 5가지 핵심 원칙

1. **CLAUDE.md는 필수** - 모든 세션의 시작점
2. **리서치 먼저** - 코드 작성 전 분석 필수
3. **계획 수립** - Step 단위로 나누어 관리
4. **세션 분리** - 컨텍스트 40-60% 유지
5. **문서 저장** - 모든 결과물은 파일로 보존
