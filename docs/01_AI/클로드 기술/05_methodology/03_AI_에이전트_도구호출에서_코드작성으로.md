# AI 에이전트: 도구 호출에서 코드 작성으로 전환하는 이유

> 발표자: 리타 코즐로프 (Rita Kozlov) - WorkOS  
> 원본 영상: [YouTube](https://www.youtube.com/watch?v=L2j3tYTtJwk)

---

## 개요

MCP(Model Context Protocol)는 강력한 에이전트 데모를 대거 선보였지만, **실제 운영 환경**에서는 단순히 도구를 호출하는 것 이상의 기능이 필요합니다.

이 발표에서는 MCP 기반 에이전트가 **실험 단계를 넘어 기업 환경**에서 실제 사용될 때 어떤 변화가 발생하는지 살펴봅니다.

---

## 핵심 통찰

### LLM은 도구 호출보다 코드 작성이 더 뛰어나다

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   기존 방식: 도구 반복 호출                                  │
│   ─────────────────────                                     │
│   LLM → Tool1 호출 → 결과 → LLM → Tool2 호출 → 결과 → ...   │
│   (매번 LLM 왕복, 토큰 소비 증가)                            │
│                                                             │
│   새로운 방식: 코드 생성 + 실행                              │
│   ─────────────────────────                                 │
│   LLM → 코드 생성 → 샌드박스 실행 → 결과                     │
│   (한 번에 처리, 효율적)                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 왜 도구 호출 방식이 한계에 도달하는가?

### 기존 도구 호출 방식의 문제점

| 문제 | 설명 |
|------|------|
| **토큰 비용 증가** | 매 도구 호출마다 LLM 왕복 필요 |
| **지연 시간** | 반복적인 호출로 응답 시간 증가 |
| **비결정성** | 같은 입력에도 다른 결과 가능 |
| **복잡성 증가** | 에이전트가 복잡해질수록 호출 체인 관리 어려움 |
| **확장성 한계** | 대규모 작업에서 성능 저하 |

### 실제 예시

```
# 기존: 도구 반복 호출
LLM: "파일 목록을 가져와야겠다" → list_files() 호출
LLM: "첫 번째 파일을 읽어야겠다" → read_file(file1) 호출
LLM: "두 번째 파일을 읽어야겠다" → read_file(file2) 호출
... (N번 반복)

# 개선: 코드 생성
LLM: 코드 생성 →
```python
files = list_files()
contents = [read_file(f) for f in files]
return analyze(contents)
```
→ 샌드박스에서 한 번에 실행
```

---

## 코드 생성 방식의 장점

### 1. 효율성 (Efficiency)

```
토큰 사용량
     ↑
     │  ████████████████  도구 호출 방식
     │  ████              코드 생성 방식
     │
     └──────────────────────────────────→ 작업 복잡도
```

- **토큰 사용량 대폭 감소**
- **응답 시간 단축**
- **비용 절감**

### 2. 결정성 (Determinism)

| 방식 | 결정성 |
|------|--------|
| 도구 반복 호출 | 낮음 (매번 LLM 판단 필요) |
| 코드 생성 + 실행 | 높음 (생성된 코드는 동일하게 실행) |

### 3. 확장성 (Scalability)

- 복잡한 작업도 단일 코드 블록으로 처리
- 병렬 처리 가능
- 대규모 데이터 처리에 적합

## 아키텍처 변화

### 기존 아키텍처: Tool Calling

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│   LLM   │ ←→  │  Tool1  │     │  Tool2  │
│         │ ←→  │         │     │         │
│         │ ←→  └─────────┘     └─────────┘
│         │ ←→  ┌─────────┐
│         │     │  Tool3  │
└─────────┘     └─────────┘

(매번 LLM 경유)
```

### 새로운 아키텍처: Code Generation + Sandbox

```
┌─────────┐     ┌──────────────────────────────┐
│   LLM   │ →   │         Sandbox              │
│         │     │  ┌──────────────────────┐    │
│ (코드   │     │  │   Generated Code     │    │
│  생성)  │     │  │   ─────────────────  │    │
│         │     │  │   Tool1.call()       │    │
│         │     │  │   Tool2.call()       │    │
└─────────┘     │  │   process(results)   │    │
                │  └──────────────────────┘    │
                │              ↓               │
                │         [Result]             │
                └──────────────────────────────┘
```

### 핵심 변화

| 요소 | 기존 | 새로운 방식 |
|------|------|------------|
| **도구 접근** | LLM이 직접 호출 | 코드에서 API처럼 호출 |
| **실행 환경** | 분산된 도구들 | 샌드박스 내 통합 실행 |
| **제어 흐름** | LLM이 매번 결정 | 생성된 코드가 제어 |

## "실제 에이전트"의 모습

### 데모 vs 프로덕션

| 측면 | 데모 | 프로덕션 |
|------|------|----------|
| **복잡도** | 단순한 시나리오 | 복잡한 워크플로우 |
| **오류 처리** | 기본적 | 강력한 복구 메커니즘 |
| **보안** | 고려 안 함 | 필수 (샌드박싱, 권한) |
| **확장성** | 단일 사용자 | 다중 사용자, 고트래픽 |
| **비용** | 고려 안 함 | 최적화 필수 |

### 프로덕션 레벨 에이전트 특징

```
┌─────────────────────────────────────────────────────────────┐
│                    Production Agent                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 코드 생성 기반 실행                                      │
│  ✅ 샌드박스 환경에서 안전하게 실행                          │
│  ✅ 도구를 API처럼 활용                                      │
│  ✅ 효율적인 토큰 사용                                       │
│  ✅ 결정적인 실행 결과                                       │
│  ✅ 확장 가능한 아키텍처                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## MCP와 같은 표준의 중요성

### 장기적 상호 운용성

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   표준 없이:                                                 │
│   ─────────                                                 │
│   Agent A ←→ [Custom Protocol] ←→ Tool Set A               │
│   Agent B ←→ [Different Protocol] ←→ Tool Set B            │
│   (서로 호환 불가)                                           │
│                                                             │
│   MCP 표준 사용:                                             │
│   ─────────────                                             │
│   Agent A ←→ [MCP] ←→ Tool Set A                            │
│   Agent B ←→ [MCP] ←→ Tool Set A (공유 가능!)               │
│   Agent B ←→ [MCP] ←→ Tool Set B                            │
│   (상호 운용 가능)                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### MCP의 역할

| 역할 | 설명 |
|------|------|
| **표준화** | 도구 호출 방식 통일 |
| **상호 운용성** | 다양한 에이전트/도구 간 호환 |
| **생태계** | 도구 재사용 및 공유 가능 |
| **확장성** | 새로운 도구 쉽게 추가 |

## 기업이 해결해야 할 다음 과제

### ID, 권한 부여, 거버넌스

에이전트가 **사용자, 조직, 워크플로를 대신하여 작동**하기 시작하면서:

```
┌─────────────────────────────────────────────────────────────┐
│                    Enterprise Challenges                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  🔐 Identity (신원)                                          │
│  ─────────────                                              │
│  - 에이전트는 누구를 대신하는가?                             │
│  - 에이전트의 신원 증명 방법                                 │
│                                                             │
│  🔑 Authorization (권한 부여)                                │
│  ─────────────────────────                                  │
│  - 에이전트가 무엇을 할 수 있는가?                           │
│  - 어떤 리소스에 접근 가능한가?                              │
│  - 권한 범위 제한                                            │
│                                                             │
│  📋 Governance (거버넌스)                                    │
│  ─────────────────────                                      │
│  - 에이전트 행동 감사                                        │
│  - 규정 준수 (Compliance)                                    │
│  - 책임 추적                                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 기업 ID는 핵심 인프라

> "에이전트 시스템이 사용자, 조직, 워크플로를 대신하여 작동하기 시작하면서 **기업 ID는 더 이상 부차적인 고려 사항이 아닌 핵심 인프라**가 됩니다."

| 과제 | 필요한 것 |
|------|----------|
| **에이전트 인증** | 에이전트 전용 토큰/인증서 |
| **위임된 권한** | OAuth 스타일 권한 위임 |
| **감사 로그** | 모든 에이전트 행동 기록 |
| **정책 관리** | 에이전트별 권한 정책 |

## 핵심 요약

### 아키텍처 진화

```
Stage 1: Demo          Stage 2: Production      Stage 3: Enterprise
─────────────          ───────────────────      ───────────────────
단순 도구 호출          코드 생성 + 샌드박스      + ID/권한/거버넌스

LLM ←→ Tools          LLM → Code → Sandbox     + Authentication
                                               + Authorization
                                               + Audit
```

### 핵심 메시지

1. **도구 반복 호출은 한계가 있다** - 복잡해질수록 비효율적
2. **코드 생성이 더 효율적** - 토큰 절약, 결정성, 확장성
3. **MCP 같은 표준이 중요** - 장기적 상호 운용성
4. **기업 환경은 더 많은 것을 요구** - ID, 권한, 거버넌스

- [ ] MCP 호환 도구 구성
- [ ] 에이전트 인증/인가 체계 수립
