# AI 글쓰기 인간화 가이드

## 왜 필요한가

AI는 "누구의 취향도 타지 않는 글"을 쓴다. 중립적이고, 일반적이고, 추상적이다. 읽는 사람이 "이거 AI가 쓴 거 아니야?"라고 느끼는 이유는 글에 개성이 없기 때문이다.

사람이 쓴 글은 다르다. 개인적인 사례가 들어가고, 독특한 어투가 있고, 주관적인 의견이 담긴다. "나는 이렇게 생각한다"가 있어야 글에 체온이 느껴진다.

이 가이드는 AI가 생성한 초안을 전문가가 직접 쓴 것처럼 자연스럽게 다듬는 기법을 정리한다.

---

## 7대 핵심 지시사항

### 1. 분량 채우기 방지

AI는 요청된 분량을 맞추려고 같은 말을 돌려서 반복한다. 소제목만 바꿔서 비슷한 내용을 늘어놓거나, "즉", "다시 말해" 뒤에 앞 문장을 재탕한다.

**지시**: "반복 없이 핵심만 전달하라. 단어 하나에도 전문성을 담아라. 분량이 부족하면 새로운 관점이나 사례를 추가하라."

### 2. 도입부 강화

AI의 도입부는 대부분 "오늘날 ~는 매우 중요합니다"로 시작한다. 누구나 아는 뻔한 말로 시작하면 독자는 첫 문단에서 이탈한다.

**지시**: "독자가 첫 문장에서 멈추게 하라. 구체적인 숫자, 질문, 혹은 반직관적 사실로 시작하라. '~의 중요성' 같은 진부한 도입은 금지."

### 3. 용어 선택

AI는 영어 단어를 과도하게 사용한다. "인사이트", "솔루션", "임팩트"처럼 한국어로 충분히 표현 가능한 단어까지 영어로 쓴다.

**지시**: "한국인이 일상에서 쓰는 단어를 우선 사용하라. '인사이트' 대신 '깨달음' 또는 '발견', '솔루션' 대신 '해결책'. 전문 용어(API, CI/CD 등)는 한글 대체어가 어색한 경우에만 영어를 유지하라."

### 4. 자연스러운 표현

AI는 "~할 수 있습니다", "~하는 것이 중요합니다"를 반복한다. 모든 문장이 같은 어미로 끝나면 기계적으로 느껴진다.

**지시**: "문장 어미를 다양하게 섞어라. '~다', '~이다', '~한다', '~일까?' 등을 번갈아 사용. 같은 어미가 3번 연속 나오면 안 된다. 한국인 전문가가 블로그에 쓰는 말투를 따라라."

### 5. CTA(행동 유도) 개선

AI의 CTA는 "지금 바로 시작해보세요!"처럼 공허하다. 왜 시작해야 하는지, 어떻게 시작하는지가 빠져 있다.

**지시**: "행동을 유도할 때는 동기부여(왜?)와 장벽 낮추기(얼마나 쉬운지)를 함께 제시하라. '5분이면 되는 첫 단계'처럼 구체적으로."

### 6. 구조 개선

AI는 불릿 포인트를 남발한다. 모든 내용을 "- 항목 1, - 항목 2"로 나열하면 성의 없어 보인다. 키워드만 던지고 설명이 없는 목록은 검색 엔진 최적화용 글처럼 느껴진다.

**지시**: "핵심 개념은 문단으로 설명하라. 불릿은 3개 이상의 동급 항목을 나열할 때만 사용. 각 불릿에도 '왜'를 포함한 설명을 붙여라."

### 7. 맥락 연결성

AI는 요소를 나열할 때 글의 주제와 연결하지 않는다. "Docker의 장점: 이식성, 격리성, 확장성"이라고만 쓰면 왜 이 글에서 Docker를 언급하는지 맥락이 끊긴다.

**지시**: "모든 요소를 소개할 때 '이 글의 주제/목적과 어떻게 연결되는지' 한 문장을 추가하라. 독자가 '이걸 왜 여기서 말하지?'라고 느끼면 안 된다."

## 금지 표현 목록

### AI 특유 강조어 (과도한 수식)
- "매우", "굉장히", "정말로", "엄청나게"
- "획기적인", "혁신적인", "강력한"
- "따라서", "그러므로" (문단 시작 남발)

### 공허한 공식체
- "당사는", "저희는"
- "~하는 것이 중요합니다"
- "~할 수 있습니다" (반복)
- "~에 대해 알아보겠습니다"

### 분량 채우기 패턴
- 소제목만 바꿔서 같은 내용 반복
- "즉,", "다시 말해" 뒤에 앞 문장 재탕
- "마지막으로 정리하면" + 앞 내용 요약 반복

### 진부한 도입
- "오늘날 ~는 매우 중요합니다"
- "~에 대해 들어보셨나요?"
- "많은 사람들이 ~를 궁금해합니다"

## 4가지 변환 원칙

### 1. 기계적 표현 제거 → 유연한 문장

| Before (AI) | After (인간화) |
|-------------|---------------|
| "이것은 매우 중요한 기능입니다" | "이 기능이 없으면 배포가 멈춘다" |
| "다양한 방법이 있습니다" | "내가 써본 방법 3가지를 공유한다" |
| "~하는 것을 권장합니다" | "~하지 않으면 나중에 고생한다" |

### 2. 문장 구조 최적화 → 간결 + 부드러운 흐름

긴 문장을 짧게 쪼개되, 문장 간 자연스러운 연결을 유지한다. "그리고", "또한"으로 기계적으로 잇지 말고, 앞 문장의 끝이 다음 문장의 시작으로 이어지도록 배치한다.

### 3. 공감과 신뢰감 → 감정 + 질문형 + 예시

독자에게 말을 거는 문장을 섞는다. "이런 경험 없는가?", "처음엔 나도 그랬다" 같은 표현이 공감을 만든다. 추상적 설명 대신 구체적 예시를 하나 넣는 것만으로 신뢰가 올라간다.

### 4. 전문성 + 친근함 균형

전문 용어를 쓰되 설명을 곁들인다. "멱등성(같은 요청을 여러 번 보내도 결과가 같다는 뜻)"처럼 괄호 한 줄이면 된다. 전문가처럼 보이면서도 초보자가 읽을 수 있는 글이 가장 좋은 글이다.

## 적용 프롬프트 템플릿

### 기본 변환 프롬프트

```
다음 글을 한국인 전문가가 직접 쓴 것처럼 다듬어줘.

규칙:
- "매우", "굉장히", "정말로" 같은 AI 특유 강조어 제거
- 같은 어미가 3번 연속 나오지 않게 어미 다양화
- 불릿 나열보다 문단형 설명 우선
- 모든 주장에 구체적 예시 또는 근거 하나씩 추가
- "~하는 것이 중요합니다" 대신 왜 중요한지를 보여주는 문장으로 교체
- 영어 단어는 한국어 대체어가 자연스러우면 한국어로 전환

[글 내용]
```

### 기술 문서용 변형

```
다음 기술 문서를 실무 경험이 있는 시니어 개발자가 쓴 것처럼 다듬어줘.

추가 규칙:
- 전문 용어(API, CI/CD 등)는 영어 유지하되, 처음 등장 시 한국어 설명 괄호 추가
- "~할 수 있습니다" → "~한다" (단정적 서술)
- 코드 예시가 있으면 왜 이렇게 쓰는지 한 줄 설명 추가
- "Best Practice" 같은 표현 대신 "실무에서 검증된 방식"

[기술 문서 내용]
```

### 학습 문서용 변형

```
다음 학습 자료를 경험 많은 선배 개발자가 후배에게 설명하는 톤으로 다듬어줘.

추가 규칙:
- 개념 설명 시 비유나 실제 사례 하나씩 추가
- "이 개념은 중요합니다" → 왜 중요한지 실무 상황과 연결
- 독자에게 질문을 던지는 문장 중간중간 삽입 ("여기서 문제가 뭘까?")
- 문단 끝에 핵심 요약 한 줄 (소제목이 아닌 자연스러운 마무리)

[학습 자료 내용]
```
