# 복잡한 코드베이스에서 AI 활용하기

> 원문: [Advanced Context Engineering for Coding Agents](https://github.com/humanlayer/advanced-context-engineering-for-coding-agents/blob/main/ace-fca.md)

## 개요

AI 코딩 도구가 실제 프로덕션 코드베이스에서 어려움을 겪는다는 것은 널리 알려진 사실입니다. [스탠포드의 AI 개발자 생산성 연구](https://www.youtube.com/watch?v=tbDDYKRFjhk)에 따르면:

1. AI 도구가 생성한 "추가 코드"의 상당 부분이 지난주에 작성된 저품질 코드를 다시 수정하는 데 사용됨
2. 코딩 에이전트는 새 프로젝트나 작은 변경에는 뛰어나지만, 대규모 기존 코드베이스에서는 오히려 개발자 생산성을 **저하**시킬 수 있음

일반적인 반응은 "절대 안 될 것"이라는 비관론과 "언젠가 더 똑똑한 모델이 나오면..."이라는 신중한 태도 사이에 있습니다.

하지만 수개월간의 실험 끝에 발견한 사실:

> **핵심 컨텍스트 엔지니어링 원칙을 적용하면 오늘날의 모델로도 충분히 멀리 갈 수 있다.**

---

## 실제로 가능한 것들

### 구체적 사례: BAML 프로젝트

- **300,000 라인의 Rust 코드베이스**를 Claude Code로 처리
- 일주일 분량의 작업을 **하루 만에** 완료
- 전문가 리뷰를 통과하는 **코드 품질** 유지

저자는 Rust 초보이면서도 BAML 코드베이스를 처음 접한 지 1시간 만에 버그 수정 PR을 제출했고, 다음 날 아침 승인받았습니다. 이후 7시간 동안 **35,000 라인의 코드**를 작성하여 취소 지원과 WASM 컴파일 기능을 추가했습니다.

## 핵심 개념: 빈번한 의도적 압축 (Frequent Intentional Compaction)

### 문제: 순진한(Naive) 컨텍스트 관리 방식

대부분의 사람들은 코딩 에이전트를 챗봇처럼 사용합니다:
- 2시간 동안 AI와 대화하며 원하는 것을 지정
- 최종 코드만 커밋하고 모든 프롬프트는 버림

이는 마치 Java 개발자가 컴파일된 JAR 바이너리만 체크인하고 소스 코드를 버리는 것과 같습니다.

### 컨텍스트 윈도우 이해하기

```
┌─────────────────────────────────────────┐
│           컨텍스트 윈도우                │
│  ┌─────────────────────────────────┐    │
│  │ 시스템 프롬프트                  │    │
│  ├─────────────────────────────────┤    │
│  │ 코드베이스 정보                  │    │
│  ├─────────────────────────────────┤    │
│  │ 사용자 메시지                    │    │
│  ├─────────────────────────────────┤    │
│  │ 이전 대화 기록                   │    │  ← 시간이 지날수록 증가
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

**핵심 통찰:**
- 컨텍스트 윈도우는 **고정 크기**
- 대화가 길어질수록 이전 정보가 밀려남
- 80% 이상 활용률에서 성능이 **급격히 저하**됨

### 해결책: 컨텍스트 활용률 40-60% 유지

```
성능
  ↑
  │     ████████████████████
  │   ██                    ██
  │  █                        █
  │ █                          █
  │█                            █
  └───────────────────────────────→ 컨텍스트 활용률
       40%        60%        80%
       
       └──── 최적 구간 ────┘
```

**실천 방법:**
- 작업을 작은 단위로 나누기
- 각 단계의 결과물을 별도 파일로 저장
- 새 세션 시작 시 필요한 정보만 로드

## 리서치/계획/구현 워크플로우

### 3단계 프로세스

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│   1. 리서치        2. 계획           3. 구현                 │
│   ┌─────────┐     ┌─────────┐       ┌─────────┐             │
│   │코드베이스│ →   │실행 계획│   →   │   코드  │             │
│   │  분석   │     │  작성   │       │   작성  │             │
│   └─────────┘     └─────────┘       └─────────┘             │
│        ↓               ↓                 ↓                   │
│   research.md     plan.md           PR 생성                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1단계: 리서치 (Research)

**목적:** 코드베이스 이해 및 관련 파일 식별

```markdown
# research.md 예시

## 목표
- 사용자 인증 로직 분석
- 관련 파일 및 함수 식별

## 발견 사항
- 인증 로직: `src/auth/AuthService.java`
- 토큰 처리: `src/security/TokenManager.java`
- 관련 테스트: `test/auth/AuthServiceTest.java`

## 주요 코드 흐름
1. 로그인 요청 → AuthController.login()
2. 사용자 검증 → AuthService.authenticate()
3. 토큰 생성 → TokenManager.generateToken()
```

### 2단계: 계획 (Plan)

**목적:** 구체적인 실행 계획 수립

```markdown
# plan.md 예시

## 변경 목표
OAuth2 소셜 로그인 추가

## 단계별 계획

### Step 1: 의존성 추가
- `build.gradle`에 OAuth2 클라이언트 추가

### Step 2: 설정 클래스 생성
- `OAuth2Config.java` 생성
- 필요 Bean 정의

### Step 3: 서비스 계층 수정
- `AuthService.java`에 OAuth2 처리 메서드 추가
- 기존 로직과 통합

### Step 4: 테스트 작성
- 단위 테스트 추가
- 통합 테스트 추가
```

### 3단계: 구현 (Implement)

**목적:** 계획에 따른 코드 작성

- 계획서를 참조하며 코드 작성
- 각 단계 완료 후 검증
- 테스트 실행 및 확인

## 인간 레버리지 (Human Leverage)

### 레버리지 포인트의 중요성

```
영향력
  ↑
  │
  │ 리서치 오류     ████████████████████████  → 수천 라인의 잘못된 코드
  │                 
  │ 계획 오류       ████████████████        → 수백 라인의 잘못된 코드
  │
  │ 코드 오류       ████████                → 몇 라인의 잘못된 코드
  │
  └─────────────────────────────────────────→ 
```

**핵심 원칙:**
- 나쁜 코드 한 줄은... 그냥 나쁜 코드 한 줄
- 나쁜 **계획** 한 줄은 수백 라인의 나쁜 코드로 이어질 수 있음
- 나쁜 **리서치** 한 줄은 수천 라인의 나쁜 코드로 이어질 수 있음

### 고레버리지 리뷰 포인트

```
┌────────────────────────────────────────────────────────────┐
│                     리뷰 우선순위                          │
│                                                            │
│  ⭐⭐⭐ 리서치 리뷰 (최고 레버리지)                         │
│  - 코드베이스 이해가 정확한가?                             │
│  - 올바른 파일/함수를 식별했는가?                          │
│                                                            │
│  ⭐⭐ 계획 리뷰 (높은 레버리지)                             │
│  - 접근 방식이 올바른가?                                   │
│  - 누락된 단계가 있는가?                                   │
│                                                            │
│  ⭐ 코드 리뷰 (기본 레버리지)                               │
│  - 코드 품질 확인                                          │
│  - 버그 탐지                                               │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 코드 리뷰의 진정한 목적

### 정신적 정렬 (Mental Alignment)

코드 리뷰의 가장 중요한 목적은 **팀원들이 코드 변경 사항과 이유에 대해 동일한 이해를 갖도록** 하는 것입니다.

**AI가 더 많은 코드를 생성할수록:**
- 코드베이스의 더 많은 부분이 특정 엔지니어에게 **낯설게** 됨
- 팀 내 **정신적 정렬**이 더욱 중요해짐

### 스펙 기반 개발의 장점

```
전통적 접근                    스펙 기반 접근
─────────────                  ──────────────
2,000줄 코드 리뷰              200줄 계획서 리뷰
(매우 피곤함)                  (관리 가능)

코드 스펠렁킹 1시간+            리서치 문서 10분 읽기
(비효율적)                     (효율적)
```

## 실제 적용 팁

### 1. 컨텍스트 효율성 유지

```bash
# 새 세션 시작 시
1. 필요한 문서만 로드
2. 이전 리서치/계획 참조
3. 구체적인 작업 범위 지정
```

### 2. 아티팩트 생성

각 단계에서 영구적인 아티팩트를 생성:

```
project/
├── specs/
│   ├── research/
│   │   ├── auth-analysis.md
│   │   └── api-structure.md
│   └── plans/
│       ├── oauth-implementation.md
│       └── refactoring-plan.md
└── src/
    └── ...
```

### 3. 반복적 개선

```
리서치 → 리뷰 → 수정
    ↓
  계획 → 리뷰 → 수정
    ↓
  구현 → 리뷰 → 수정
    ↓
  완료
```

## 주의사항과 한계

### 여전히 어려운 것들

- 모든 문제를 7시간 안에 해결할 수는 없음
- 깊은 의존성 트리 분석이 필요한 경우 여전히 어려움
- **코드베이스 전문가 최소 1명**이 필요함

### 실패 사례

저자는 Hadoop 의존성을 Parquet Java에서 제거하려는 시도에서 실패했습니다:
- 리서치 단계에서 의존성 트리를 충분히 깊게 탐색하지 못함
- 코드베이스 전문가 없이 진행함

## 결론

### 달성 가능한 것들

| 목표 | 달성 여부 |
|------|----------|
| 브라운필드(기존) 코드베이스에서 작동 | ✅ |
| 복잡한 문제 해결 | ✅ |
| 저품질 코드(슬롭) 없음 | ✅ |
| 팀 정신적 정렬 유지 | ✅ |

### 핵심 요약

1. **컨텍스트 엔지니어링**은 오늘날 모델에서 최대 성능을 끌어내는 핵심
2. **빈번한 의도적 압축**으로 컨텍스트 활용률 40-60% 유지
3. **리서치/계획/구현** 워크플로우로 구조화된 접근
4. **고레버리지 포인트**에서 인간 리뷰 집중
5. **스펙 기반 개발**로 팀 정렬 유지

### 마지막 조언

> "마법의 프롬프트 하나로 모든 문제를 해결하려는 것은 환상입니다. 
> 깊이 몰입하여 작업에 참여해야 합니다. 그렇지 않으면 제대로 작동하지 않습니다."
