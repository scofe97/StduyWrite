> 에이전트가 일할 환경을 설계하는 3대 축: Context, Constraints, Entropy Management

> 관련 문서: [18_Codex_하네스_엔지니어링](../06_trends-news/18_Codex_하네스_엔지니어링.md) (OpenAI 원문 정리)

## 1. Context Engineering vs Harness Engineering

OpenAI 블로그만 보면 "문서 잘 정리하면 되는 거 아냐?"라는 오해가 생긴다. Martin Fowler와 mtrajan의 분석이 이 구분을 명확하게 해준다.

| 구분 | Context Engineering | Harness Engineering |
|------|-------------------|-------------------|
| **질문** | "에이전트가 뭘 알아야 하나?" | "시스템이 뭘 막고, 측정하고, 교정해야 하나?" |
| **범위** | 정보 제공 (docs, AGENTS.md, 메모리) | 구조적 제약 + 피드백 루프 |
| **실패 유형** | 잘못된 출력 (1회성) | 시간이 지나며 쌓이는 품질 저하 |
| **해결책** | 문서 개선, 프롬프트 튜닝 | 린터, CI 게이트, 자동 정리 에이전트 |

Context Engineering은 Harness Engineering의 **부분집합**이다. 문서를 잘 정리하는 것(context)은 필요조건이지만, 기계적 강제(constraints)와 엔트로피 관리(entropy management)가 없으면 시간이 지나며 코드베이스가 무너진다.

> 핵심: 단일 실패는 context 문제, 점진적 저하는 harness 문제.

## 2. 3대 축 상세

### 2-1. Context Engineering — 에이전트가 알아야 할 것

OpenAI가 실천한 것:
- **docs/ 디렉토리**를 system of record로 취급. 외부 문서(Google Docs, Slack) 의존 제거.
- **AGENTS.md는 ~100줄의 지도(포인터)**. 깊은 내용은 docs/에 위임.
- **ExecPlan**: 자기 완결형 실행 계획. 초보자가 읽고 끝까지 구현 가능해야 통과.

왜 이렇게 하는가: 에이전트는 3개월 뒤 합류하는 신입과 같다. 암묵지는 존재하지 않는 것과 동일하다. 모든 결정과 맥락이 저장소 안에 있어야 에이전트가 자율적으로 작업할 수 있다.

### 2-2. Architectural Constraints — 하면 안 되는 것을 강제

OpenAI가 실천한 것:
- **불변 규칙만 기계로 강제**, 구현 방식은 자율에 맡김
- 강제: 데이터 경계 파싱, 의존성 방향, 레이어 구조
- 자율: 라이브러리 선택, 세부 구현 로직
- **커스텀 린터**가 아키텍처 드리프트를 CI에서 차단 (린터 자체도 Codex가 작성)
- **린터 에러 메시지 = 컨텍스트 교육** — 모든 실패 메시지가 "왜"와 "어떻게 고치나"를 포함

왜 이렇게 하는가: 코드 리뷰로 일관성을 유지하는 것은 사람에게도 어렵고, 에이전트에게는 불가능하다. 기계가 검증할 수 있는 것은 기계에 맡기고, 사람은 판단이 필요한 곳에 집중해야 한다.

### 2-3. Entropy Management — 시간이 지나며 쌓이는 엔트로피 제거

OpenAI가 실천한 것:
- **배경 에이전트**가 문서 진부화를 스캔하고 정리 PR을 자동 생성
- **품질 기준을 코드로 인코딩** (golden rules, quality grades, recurring checks)
- 기준 통과한 교정 PR은 **자동 머지** — "코드 품질 가비지 컬렉션"
- 측정 가능한 요구사항 설정 (예: "시작 800ms 이내")

왜 이렇게 하는가: 에이전트가 만든 코드도 시간이 지나면 기술 부채가 쌓인다. 사람이 수동으로 정리하는 것은 지속 불가능하므로, 엔트로피 제거 자체를 자동화해야 한다.

## 3. 내 환경 GAP 분석

### 이미 잘 하고 있는 것

| Harness 원칙 | 현재 구현 | 평가 |
|-------------|----------|------|
| Context Engineering | CLAUDE.md(진입점) + specs/(상세) + .claude/skills/(도메인 지식) | 강함 |
| AGENTS.md 포인터 역할 | ~19줄, 스킬 해결 우선순위 + CLAUDE.md 참조 | OpenAI 권장과 일치 |
| 제약된 해결 공간 | dev-standards.md(외과적 변경), workflow.md(요구사항 명확화) | 부합 |
| 기계 가독성 | 스킬별 SKILL.md 자체 문서화, 카테고리 구조 | 강함 |
| 반복적 개선 | CLAUDE.md "실수 개선 프로세스" — 원인분석→수정→재발방지→문서화 | 부합 |
| 컨텍스트 관리 | workflow.md의 5파일→에이전트 위임, 20회→compact 규칙 | 강함 |

### 도출된 GAP 3가지

| # | 원칙 | GAP | 적용 방법 |
|---|------|-----|----------|
| G1 | Mechanical Enforcement | 규칙이 문서에만 존재, 기계적 강제 없음 | Git hook으로 기계화 |
| G2 | Entropy Management | 문서 진부화 탐지 프로세스 없음 | 월 1회 스케줄 추가 |
| G3 | 린터 에러 = 교육 | 규칙 위반 시 에이전트가 참조할 가이드 부재 | 수정 가이드 패턴 추가 |

## 4. 적용 결과

### G1. 규칙 기계화 — Git Hooks

`.git/hooks/commit-msg`: 커밋 메시지 형식 검증
- `타입: 설명` 형식만 허용 (feat/fix/refactor/docs/test/chore/style)
- `Co-Authored-By` 패턴 차단 (프로젝트 규칙)
- TPS 형식 `[이슈번호][프로젝트명]`도 허용

`.git/hooks/pre-commit`: 커밋 전 안전 검증
- `.env`, `credentials`, `secret`, `.pem`, `.key` 등 민감 파일 스테이징 차단
- 변경 파일 15개 초과 시 경고 (커밋 분할 권장)

**원칙**: 문서에 적혀있던 규칙을 기계가 강제하도록 전환. 에이전트와 사람 모두에게 동일하게 적용.

### G2. 문서 진부화 탐지 — 월 1회 스케줄

`schedules.json`에 `docs-freshness-check` 추가 (매월 15일):
1. SKILL.md 내 참조 경로가 실제로 존재하는지 확인
2. CLAUDE.md 프로젝트 테이블 경로 유효성 확인
3. .claude/rules/*.md 규칙과 실제 패턴 일치 여부 샘플 체크
4. 3개월 이상 수정되지 않은 SKILL.md 목록 출력

**원칙**: OpenAI의 "배경 에이전트가 문서 진부화 스캔" 개념을, 기존 학습 스케줄러에 1개 스케줄로 구현.

### G3. 린터 에러 = 교육 — 수정 가이드

`dev-standards.md` 각 섹션(보안/테스트/문서화/외과적 변경)에 위반 시 수정 가이드 추가:

```
### 위반 시 수정 가이드
- **증상**: 어떤 증상이 보이는가
- **원인**: 왜 이 실수가 발생하는가
- **수정**: 구체적으로 어떻게 고치는가
- **예방**: 재발 방지를 위해 무엇을 확인하는가
```

**원칙**: OpenAI의 "모든 린터 에러 메시지가 가이드 역할" 개념을, 규칙 문서 자체에 적용. 에이전트가 규칙을 위반했을 때 스스로 교정할 수 있는 정보를 제공.

## 5. 적용하지 않은 것과 이유

| 항목 | 이유 |
|------|------|
| 별도 docs/ 디렉토리 신설 | specs/ + skills/로 동일 역할 수행 중 |
| AGENTS.md 대폭 확장 | 현재 ~19줄 포인터 역할 = OpenAI 권장과 일치 |
| 자동 머지 | 1인 개발 환경에서 PR 자동화는 오버헤드 |
| Chrome DevTools 통합 | 학습 프로젝트에서는 불필요 |
| 배경 에이전트 자동 PR | 1인 환경에서는 스케줄 기반 알림이 더 적합 |
| 정량적 품질 기준 | 현재 정성적 원칙으로 충분, 필요 시 추후 추가 |
