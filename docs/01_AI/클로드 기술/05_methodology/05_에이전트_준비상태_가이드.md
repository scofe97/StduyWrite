> 발표자: Eno Reyes - Factory AI CTO  
> 출처: AI Engineer 컨퍼런스  
> 원본 영상: [YouTube](https://www.youtube.com/watch?v=ShuJ_CN6zr4)

---

## 개요

에이전트가 소프트웨어 엔지니어링을 혁신하고 있지만, **데모에서는 훌륭하게 작동하던 에이전트가 실제 운영 환경에서는 불안정하게 작동**하는 경우가 많습니다.

이 격차는 **모델의 품질 문제가 아니라 환경 준비 상태의 문제**입니다.

---

## 1. 에이전트 실패의 원인

### 에이전트가 효과적으로 작동하려면 필요한 것

| 필요 조건 | 설명 |
|-----------|------|
| **빠른 피드백 루프** | 실행 결과를 즉시 확인 가능 |
| **명확한 지침** | 문서화된 규칙과 가이드 |
| **예측 가능한 환경** | 일관된 개발 환경 |

### 에이전트의 발목을 잡는 요소

```
┌─────────────────────────────────────────────────────────────┐
│                에이전트 실패의 숨은 원인                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ❌ 환경 변수 누락                                           │
│     "이 API 키는 어디서 가져오지?"                           │
│                                                             │
│  ❌ 문서화되지 않은 종속성                                   │
│     "이 라이브러리는 수동 설치가 필요한데..."                 │
│                                                             │
│  ❌ 암묵적 지식 (Tribal Knowledge)                           │
│     "그건 모두가 그냥 알고 있는 거야"                        │
│                                                             │
│  ❌ 일관성 없는 개발 환경                                    │
│     "내 컴퓨터에서는 되는데..."                              │
│                                                             │
│  ❌ 느린 빌드/테스트                                         │
│     피드백 루프 지연 → 에이전트 효율 저하                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. 에이전트 준비 상태 (Agent Readiness)

### 8가지 평가 범주

코드베이스가 에이전트를 수용할 준비가 되었는지 판단하는 8가지 범주:

```
┌─────────────────────────────────────────────────────────────┐
│                  Agent Readiness 8 Categories                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1️⃣  스타일 검증 (Style Validation)                         │
│  2️⃣  빌드 시스템 (Build System)                             │
│  3️⃣  개발 환경 (Development Environment)                    │
│  4️⃣  관찰 가능성 (Observability)                            │
│  5️⃣  테스트 인프라 (Testing Infrastructure)                 │
│  6️⃣  문서화 (Documentation)                                 │
│  7️⃣  종속성 관리 (Dependency Management)                    │
│  8️⃣  CI/CD 파이프라인 (CI/CD Pipeline)                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 각 범주별 상세 가이드

### 1️⃣ 스타일 검증 (Style Validation)

**목적**: 에이전트가 일관된 코드 스타일을 유지하도록 함

| 항목 | 설명 | 도구 예시 |
|------|------|----------|
| **린터 설정** | 코드 스타일 자동 검증 | ESLint, Pylint |
| **포맷터 설정** | 자동 코드 포맷팅 | Prettier, Black |
| **규칙 문서화** | .eslintrc, pyproject.toml | 설정 파일 커밋 |

```bash
# 스타일 검증 명령어 (에이전트가 사용)
npm run lint
npm run format:check
```

**체크리스트**:
- [ ] 린터 설정 파일 존재
- [ ] 포맷터 설정 파일 존재
- [ ] pre-commit 훅 설정
- [ ] CI에서 스타일 검증

---

### 2️⃣ 빌드 시스템 (Build System)

**목적**: 빠르고 일관된 빌드 환경 제공

| 항목 | 권장 사항 |
|------|----------|
| **빌드 시간** | 5분 이내 |
| **증분 빌드** | 변경된 파일만 재빌드 |
| **캐싱** | 의존성 캐싱 활성화 |
| **단일 명령어** | `npm run build` 하나로 완료 |

```bash
# 이상적인 빌드 명령어
npm run build        # 전체 빌드
npm run build:dev    # 개발 빌드 (빠름)
npm run build:prod   # 프로덕션 빌드
```

**체크리스트**:
- [ ] 단일 빌드 명령어
- [ ] 빌드 시간 5분 이내
- [ ] 캐싱 설정
- [ ] 빌드 오류 메시지 명확

---

### 3️⃣ 개발 환경 (Development Environment)

**목적**: 누구나 (에이전트 포함) 동일한 환경에서 작업

| 항목 | 권장 사항 |
|------|----------|
| **환경 변수** | .env.example 제공 |
| **컨테이너화** | Docker/devcontainer 사용 |
| **버전 관리** | .nvmrc, .python-version |
| **원클릭 셋업** | 단일 명령어로 환경 구성 |

```bash
# 이상적인 환경 설정
git clone [repo]
cp .env.example .env
docker-compose up -d
npm install
npm run dev
```

**체크리스트**:
- [ ] .env.example 파일 존재
- [ ] Docker/devcontainer 설정
- [ ] 버전 고정 (.nvmrc 등)
- [ ] README에 셋업 가이드

---

### 4️⃣ 관찰 가능성 (Observability)

**목적**: 에이전트가 실행 결과를 즉시 확인 가능

| 항목 | 권장 사항 |
|------|----------|
| **로깅** | 구조화된 로그 출력 |
| **오류 메시지** | 명확하고 해결 가능한 메시지 |
| **상태 확인** | 헬스체크 엔드포인트 |
| **디버깅** | 디버그 모드 지원 |

```bash
# 에이전트가 확인하는 것
npm run start       # 로그 출력 확인
npm run lint        # 오류 메시지 확인
npm run test        # 테스트 결과 확인
```

**체크리스트**:
- [ ] 구조화된 로그
- [ ] 명확한 오류 메시지
- [ ] 헬스체크 엔드포인트
- [ ] 디버그 모드

---

### 5️⃣ 테스트 인프라 (Testing Infrastructure)

**목적**: 에이전트가 변경 후 검증 가능

| 항목 | 권장 사항 |
|------|----------|
| **단위 테스트** | 빠른 피드백 (< 1분) |
| **통합 테스트** | 주요 흐름 검증 |
| **테스트 명령어** | 단일 명령어로 실행 |
| **테스트 커버리지** | 최소 70% 이상 |

```bash
# 에이전트가 사용하는 테스트 명령어
npm run test           # 전체 테스트
npm run test:unit      # 단위 테스트만 (빠름)
npm run test:coverage  # 커버리지 확인
```

**체크리스트**:
- [ ] 테스트 스위트 존재
- [ ] 단위 테스트 1분 이내
- [ ] CI에서 테스트 실행
- [ ] 테스트 커버리지 리포트

---

### 6️⃣ 문서화 (Documentation)

**목적**: 에이전트가 컨텍스트를 빠르게 파악

| 문서 | 내용 |
|------|------|
| **README.md** | 프로젝트 개요, 셋업 방법 |
| **CLAUDE.md** | AI 에이전트 전용 가이드 |
| **CONTRIBUTING.md** | 기여 가이드라인 |
| **아키텍처 문서** | 시스템 구조 설명 |

```markdown
# CLAUDE.md 핵심 내용

## 프로젝트 개요
[한 문단]

## 빌드 & 테스트 명령어
npm run build
npm run test

## 아키텍처
[구조 설명]

## 코딩 규칙
[규칙 목록]

## 주의사항
[알아야 할 것들]
```

**체크리스트**:
- [ ] README.md 존재 및 최신화
- [ ] CLAUDE.md 또는 에이전트 가이드
- [ ] 아키텍처 문서
- [ ] API 문서 (해당 시)

### 7️⃣ 종속성 관리 (Dependency Management)

**목적**: 일관된 종속성, 예측 가능한 동작

| 항목 | 권장 사항 |
|------|----------|
| **잠금 파일** | package-lock.json 커밋 |
| **버전 고정** | 정확한 버전 지정 |
| **자동 업데이트** | Dependabot/Renovate |
| **취약점 스캔** | npm audit 자동화 |

```bash
# 종속성 관리 명령어
npm ci                 # 잠금 파일 기반 설치 (권장)
npm audit              # 취약점 확인
npm outdated           # 업데이트 확인
```

**체크리스트**:
- [ ] 잠금 파일 커밋
- [ ] 버전 고정
- [ ] 취약점 스캔 자동화
- [ ] 정기 업데이트 프로세스

---

### 8️⃣ CI/CD 파이프라인 (CI/CD Pipeline)

**목적**: 자동화된 검증 및 배포

| 단계 | 포함 항목 |
|------|----------|
| **빌드** | 컴파일, 번들링 |
| **테스트** | 린트, 단위/통합 테스트 |
| **보안** | 취약점 스캔 |
| **배포** | 스테이징, 프로덕션 |

```yaml
# 이상적인 CI 파이프라인
name: CI
on: [push, pull_request]

jobs:
  build:
    steps:
      - npm ci
      - npm run lint
      - npm run typecheck
      - npm run test
      - npm run build
```

**체크리스트**:
- [ ] CI 파이프라인 존재
- [ ] PR마다 자동 실행
- [ ] 실패 시 명확한 피드백
- [ ] 배포 자동화

## 5. 쉽게 개선할 수 있는 부분 (Quick Wins)

### 즉시 적용 가능한 개선

```
┌─────────────────────────────────────────────────────────────┐
│                        Quick Wins                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1️⃣  CLAUDE.md 작성 (1시간)                                 │
│      → 에이전트 생산성 즉시 향상                             │
│                                                             │
│  2️⃣  .env.example 추가 (10분)                               │
│      → 환경 변수 누락 문제 해결                              │
│                                                             │
│  3️⃣  단일 셋업 스크립트 (30분)                              │
│      → ./setup.sh 하나로 환경 구성                           │
│                                                             │
│  4️⃣  린터 설정 커밋 (15분)                                  │
│      → 일관된 코드 스타일                                    │
│                                                             │
│  5️⃣  빌드/테스트 명령어 통일 (15분)                         │
│      → npm run build, npm run test                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 우선순위 결정

```
영향력
  ↑
  │  ┌─────────────┐
  │  │ CLAUDE.md   │  ← 가장 효과적
  │  │ 문서화      │
  │  └─────────────┘
  │        ┌─────────────┐
  │        │ 개발 환경   │
  │        │ 셋업 간소화  │
  │        └─────────────┘
  │              ┌─────────────┐
  │              │ 테스트      │
  │              │ 인프라      │
  │              └─────────────┘
  └───────────────────────────────────→ 구현 난이도
```

## 6. Factory의 실제 사례

### 엔터프라이즈 환경에서의 자율 에이전트

Factory AI가 발견한 실제 운영 환경의 교훈:

| 문제 | 해결책 |
|------|--------|
| 환경 변수 누락 | 완전한 .env.example + 검증 스크립트 |
| 암묵적 지식 | CLAUDE.md에 모든 것 문서화 |
| 느린 피드백 루프 | 테스트 분할, 캐싱 최적화 |
| 일관성 없는 환경 | Docker 기반 개발 환경 |

### 성공 사례의 공통점

```
✅ 원클릭 환경 셋업
✅ 5분 이내 빌드
✅ 1분 이내 단위 테스트
✅ 명확한 오류 메시지
✅ 완전한 문서화
```
