# 바이브 모델링 설계 사례연구

> 출처: [코딩하는오후 YouTube](https://www.youtube.com/@codingpm)
> 영상: 바이브 모델링 설계 사례연구 (Spring Modulith와 개발공정을 고려한)

---

## 바이브 모델링이란?

**AI와 협업하여 도메인의 핵심 개념과 행위적 흐름을 동기화하는 설계 방법론**

- 설계자의 의도와 AI의 해석 사이 간극을 줄이는 것이 목적
- 바이브 코딩과 다름: 코드가 아닌 **유스케이스와 액티비티 다이어그램** 도출이 목표

```
바이브 코딩    : 실행 가능한 코드를 빠르게 얻는 것
바이브 모델링  : 비즈니스 정렬, 개발자가 브레이크다운할 수 있는 설계 산출물
```

---

## 3단계 설계 패턴

### 전체 흐름

```
1단계 Vibe Sync ──────────► 유스케이스 도출
        ↓
2단계 Dynamic Skeleton ───► 액티비티 도출
        ↓
3단계 Vibe Iteration ─────► 속성 구체화 (반복)
```

---

### 1단계: Vibe Sync (느낌 동기화)

**목적**: AI와 도메인 페르소나, 톤앤매너 맞추기

**하는 일**:
- 서비스가 지향하는 사용자 경험의 "온도" 전달
- 핵심 시나리오에 대한 느낌 공유

**예시: "사용자 관리" 도메인**

| 느낌 | 중요 속성 |
|------|----------|
| 금융 서비스 | 본인인증, 보안등급, 거래한도 |
| 쇼핑 멤버십 | 포인트, 등급, 쿠폰 |

**결과**: AI가 용어를 스스로 선택
- User / Member / Player 중 적절한 것 선택

**주의**: 모든 기능을 한 번에 맞추려 하지 말 것. **핵심 시나리오만** 진행

---

### 2단계: Dynamic Skeleton (흐름 도출)

**목적**: 컨텍스트별 액티비티 흐름 파악

**하는 일**:
- 바운더리별 흐름 대화하며 풀어나가기
- **테이블 설계에 매몰되지 않기** (AI가 자꾸 테이블 설계하려 함)

**예시: "메뉴 관리"**

```
1단계에서: "이게 백오피스 메뉴야 vs 식당 메뉴야" 온도 맞춤
2단계에서: 권한에 따라 보이기/안보이기 제약조건 추가하며 흐름 완성
```

**효과**:
- 비즈니스 로직 누락 방지
- 예외 상황을 모델 단계에서 미리 방어

**산출물**: 액티비티 다이어그램 (UML)

**주의**: 동시에 여러 컨텍스트 처리하지 말 것. **한 번에 하나씩**

---

### 3단계: Vibe Iteration (반복 구체화)

**목적**: 액티비티를 통해 속성 구체화

**하는 일**:
- 각 액티비티의 속성 정의
- 기능이 너무 많아지면 **프롬프트로 제지**

**효과**:
- 비즈니스 컨텍스트를 Entity 집합이 아닌 **행위 중심**으로 유지
- 도메인을 행위로 판단할 수 있게 됨

---

## 용어 정의

### 모델 (Model)

```
객체가 가진 상태와 행위 (책임)
→ Aggregate Root로 도출
```

### 액티비티 (Activity)

```
도메인 내에서 일어나는 행위와 흐름
→ 서비스 인터페이스로 도출
```

### 액티비티와 모델의 연결

```
유스케이스 ──► 액티비티 ──► 서비스 인터페이스
                    │
                    ▼
               Aggregate Root ──► 엔티티/값객체
```

---

## 구현 단계 주의사항

### 1. 빈약한 도메인 방지

**문제**: AI가 습관적으로 getter/setter만 있는 빈약한 도메인 생성

```java
// ❌ AI가 만들려는 빈약한 도메인
@Entity
@Getter @Setter
public class User {
    private Long id;
    private String name;
    // 로직 없음
}

// ✅ 풍부한 도메인
@Entity
public class User {
    private Long id;
    private String name;
    private UserStatus status;

    public void activate() {
        if (this.status == UserStatus.PENDING) {
            this.status = UserStatus.ACTIVE;
        }
    }
}
```

**해결**: 액티비티를 계속 유지하면서 이터레이션 진행

---

### 2. 액티비티 원자성 보장

**문제**: 하나의 액티비티가 너무 많은 Aggregate Root를 건드림

**해결**:
- CQRS로 Command/Query 분리
- 단일 유스케이스 = 단일 트랜잭션 범위

```java
// ❌ 여러 Aggregate 건드림
public interface UserActivity {
    void createUserAndAssignRole(User user, Role role, Department dept);
}

// ✅ 잘게 분리
public interface CreateUserActivity {
    User createUser(CreateUserRequest request);
}

public interface AssignRoleActivity {
    void assignRole(Long userId, Long roleId);
}
```

---

### 3. 추상화 수준 유지

**문제**: AI가 테이블 스키마로 흘러감

**징후**: 프롬프트 응답에 `CREATE TABLE`, `FK` 등 등장

**해결**:
- 서비스 인터페이스는 **액티비티의 언어** 사용
- 스키마 설계로 흘러가면 즉시 제지

## 컨텍스트 오염 문제

### 발생 원인

```
"메뉴" 단어의 해석 차이:
- 설계자: 백오피스 네비게이션 메뉴
- AI: 식당 메뉴 (판매 가격, 옵션 등)
```

### 해결 방법

| 단계 | 오염 상황 | 보완 방법 |
|------|----------|----------|
| Sync | 도메인 성격 미싱크 | 도메인 성격 명시적 정의 |
| Skeleton | 테이블 설계로 빠짐 | 흐름 중심 대화 유지 |
| Iteration | 기능 과다 추가 | 제지 프롬프트 |

### 최후의 수단

> 바이브가 너무 오염되면 **채팅 삭제 후 새 채팅**으로 시작

## 바이브 코딩 vs 바이브 모델링

| 구분 | 바이브 코딩 | 바이브 모델링 |
|------|------------|--------------|
| **목표** | 실행 가능한 코드 | 유스케이스 + 액티비티 |
| **산출물** | 소스 코드 | UML 다이어그램 |
| **판단 기준** | 동작 여부 | 설계자의 직관/직감 |
| **적합한 영역** | 구현 | 도메인 설계 |

### 알고리즘 설계에서의 바이브

**바이브 모델링은 알고리즘 설계에 부적합**

- 알고리즘: 수치적, 명확한 제약조건 필요
- 바이브: "Happy Path" 중심으로 흘러가기 쉬움
- 알고리즘 시 **의사 코드 + 테스트 케이스 선설계** 권장

## Spring Modulith 연동 공정

### 5단계 개발 공정

| 단계 | 패키지 | 역할 | 노출 범위 |
|------|--------|------|----------|
| 1 | `domain` | Aggregate Root, Entity | 모듈 외부 |
| 2 | `application` | Activity 인터페이스 | 모듈 외부 |
| 3 | `internal` | Service 구현체 | 모듈 내부 |
| 4 | `port` | 외부 인프라 연결 | 모듈 내부 |
| 5 | `adapter` | Port 구현체 | 모듈 내부 |

### 바이브 모델링 → 구현 흐름

```
바이브 1단계 (Sync)
    ↓
유스케이스 다이어그램 (PlantUML)
    ↓
바이브 2단계 (Skeleton)
    ↓
액티비티 다이어그램 (PlantUML)
    ↓
바이브 3단계 (Iteration)
    ↓
Aggregate Root + 속성 정의
    ↓
5단계 개발 공정 적용
    ↓
Spring Modulith 모듈 구현
```

## 실전 프롬프트 가이드

### 첫 프롬프트

```
나는 [시스템명] 바이브 모델링을 하려고 해.
가이드라인은 다음과 같아:
[가이드라인 전문 붙여넣기]
```

### 유스케이스 도출

```
[시스템]의 핵심 유스케이스를 도출해줘.
PlantUML 유스케이스 다이어그램으로 그려줘.
```

### 액티비티 도출

```
[유스케이스명]에 대해서 액티비티 다이어그램 그려줘.
PlantUML로 그려줘.
```

### 제지 프롬프트

```
테이블 설계로 가지 마.
액티비티 흐름에 집중해줘.
```

## 핵심 인사이트

### 1. 느낌 전달이 핵심

> "바이브 모델링은 코드의 오류를 잡는 것이 아니라 **설계자의 의도를 디버깅**하는 작업이다."

### 2. 직관과 직감을 믿기

- 도메인 모델: 느낌으로 직관적 판단 가능
- 알고리즘: 논리적 검증 필요 (바이브 부적합)

### 3. 반복 공정의 중요성

> "앞에서 완벽하게 안 해도 뒤에서 알아차려서 수정 가능하게 설계"
