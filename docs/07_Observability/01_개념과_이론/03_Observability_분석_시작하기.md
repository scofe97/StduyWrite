# Chapter 3: Getting Started with Observability Analysis

## 📌 핵심 요약

> **Core Analysis Loop(핵심 분석 루프)**는 Observability 데이터를 활용한 **가설 기반 디버깅 기법**입니다. 전통적인 "알려진 조건에서 디버깅"(직관, 경험, 런북 의존)과 달리, **제1원칙에서 디버깅**하면 시스템에 익숙하지 않은 사람도 체계적으로 문제를 해결할 수 있습니다. 이 접근법은 **Wide Structured Events**가 필수이며, 자동화된 도구(예: Honeycomb BubbleUp)로 분석을 가속화할 수 있습니다.

## 📖 본문 정리

### 1. 알려진 조건에서 디버깅 (Debugging from Known Conditions)

```
💬 비유: 같은 동네에서 30년간 살았던 주민이 "저 골목으로 가면 돼"라고
       직관적으로 길을 알려주는 것처럼, 시니어 엔지니어는 경험과 직관으로
       문제를 진단합니다. 하지만 새로운 동네에서는 GPS가 필요합니다.
```

#### 전통적 디버깅 방식

```
┌─────────────────────────────────────────────────────────────┐
│           전통적 디버깅: 알려진 조건 기반                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   시니어 엔지니어의 "마법":                                  │
│   ┌─────────────────────────────────────────────────┐      │
│   │  🧙 시스템과의 친밀한 경험                       │      │
│   │       ↓                                         │      │
│   │  🔮 "올바른 질문"을 직관적으로 암                │      │
│   │       ↓                                         │      │
│   │  🎯 어디를 봐야 하는지 본능적으로 암             │      │
│   └─────────────────────────────────────────────────┘      │
│                                                             │
│   이 마법을 포착하려는 시도:                                 │
│   - 런북 (Runbooks) 작성                                    │
│   - 대시보드 무한 생성                                       │
│   - 근본 원인 문서화                                         │
│                                                             │
│   ⚠️ 문제: 현대 분산 시스템은 같은 방식으로 두 번 실패하지 않음│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 런북(Runbooks)의 한계

| 런북이 유용한 경우 | 런북이 실패하는 경우 |
|-------------------|---------------------|
| 기술 부채 임시 해결 | 분산 시스템의 긴 꼬리 문제 |
| 반복되는 단일 이슈 | 5가지 불가능한 조건의 조합 |
| 다음 스프린트까지 임시 조치 | 몇 년에 한 번 발생하는 대규모 장애 |

```
⚠️ 런북에 대한 주의사항:

✅ 적절한 문서화:
   - 서비스 소유 팀
   - 온콜 엔지니어 연락처
   - 에스컬레이션 포인트
   - 의존성 서비스 목록
   - 좋은 쿼리/대시보드 링크

❌ 위험한 문서화:
   - 모든 가능한 시스템 오류와 해결책을 담은 "살아있는 문서"
   - 빠르게 변하는 시스템에서는 금방 낡아버림
   - 틀린 문서는 문서가 없는 것보다 더 위험함

💡 대안:
   계측(Instrumentation) 자체가 최고의 문서
   - 의도 (엔지니어가 수집하기로 결정한 차원)
   - 실시간, 최신 프로덕션 상태 정보
```

#### 전통적 방식의 문제점

```
┌─────────────────────────────────────────────────────────────┐
│              전통적 디버깅 워크플로우의 한계                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 시스템의 모든 부분을 친밀하게 이해해야 함               │
│      (직접 경험, 문서, 런북 통해)                            │
│                                                             │
│   2. 대시보드를 보고...                                     │
│                                                             │
│   3. 답을 "직관"해야 함? 🤔                                 │
│      또는 근본 원인을 "추측"하고 확인 증거 찾기              │
│                                                             │
│   ⚠️ Observability 데이터를 수집해도 여전히 이 방식 사용 가능:│
│   - Wide events → tail -f | grep (알려진 문자열 검색)       │
│   - 쿼리 결과 → 무한한 대시보드로 스트리밍                   │
│   - 대시보드 간 시각적 패턴 매칭                             │
│                                                             │
│   💡 데이터 수집이 아니라 "접근 방식"이 중요!                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 제1원칙에서 디버깅 (Debugging from First Principles)

```
💬 비유: 과학자가 가설을 세우고 실험으로 검증하듯이,
       제1원칙 디버깅은 가정 없이 데이터로 증명합니다.
       "나는 이게 문제라고 생각해"가 아니라
       "데이터가 이것이 문제라고 보여줘"입니다.
```

#### 제1원칙(First Principle)이란?

> **"다른 가정에서 추론되지 않은, 시스템에 대한 기본 가정"**
> 철학에서는 "어떤 것이 알려지게 되는 첫 번째 기초"

| 전통적 디버깅 | 제1원칙 디버깅 |
|--------------|---------------|
| 직관에 의존 | 관찰에 기반 |
| 과거 경험 활용 | 현재 데이터 분석 |
| 패턴 매칭 | 가설 검증 |
| 시스템 전문가 필요 | 누구나 가능 |
| 확장 불가능 | 체계적, 반복 가능 |

#### 제1원칙 디버깅의 이점

```
┌─────────────────────────────────────────────────────────────┐
│              제1원칙 디버깅의 민주화 효과                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   전통적 방식:                                              │
│   ┌─────────────────────────────────────────────────┐      │
│   │  성공 = f(시스템 친숙도, 경험, 직관)            │      │
│   │                                                 │      │
│   │  🧙 시스템 위저드만 빠르게 해결                 │      │
│   │  👶 주니어는 오래 걸리거나 실패                 │      │
│   └─────────────────────────────────────────────────┘      │
│                                                             │
│   제1원칙 방식:                                             │
│   ┌─────────────────────────────────────────────────┐      │
│   │  성공 = f(호기심, 성실함, 관찰력)               │      │
│   │                                                 │      │
│   │  🔍 가장 호기심 많은 사람이 성공                │      │
│   │  📊 데이터를 꼼꼼히 보는 사람이 성공            │      │
│   │  👶 시스템 지식이 적어도 디버깅 가능!          │      │
│   └─────────────────────────────────────────────────┘      │
│                                                             │
│   💡 핵심: "마법" 대신 "방법론적, 반복 가능, 검증 가능"     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3. Core Analysis Loop (핵심 분석 루프)

```
💬 비유: 탐정이 범죄 현장에서 증거를 수집하고,
       용의자를 좁혀가며, 알리바이를 검증하고,
       확실한 증거가 나올 때까지 반복하는 것과 같습니다.
```

#### 4단계 프로세스

```
┌─────────────────────────────────────────────────────────────┐
│                   Core Analysis Loop                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│         ┌─────────────────────────────────────┐            │
│         │  1️⃣ 조사 촉발점으로 시작            │            │
│         │     (알림, 고객 불만 등)            │            │
│         └────────────────┬────────────────────┘            │
│                          │                                  │
│                          ▼                                  │
│         ┌─────────────────────────────────────┐            │
│         │  2️⃣ 알고 있는 것이 사실인지 검증    │            │
│         │     (성능 변화가 실제로 발생?)      │            │
│         │     → 그래프 곡선 변화 확인         │            │
│         └────────────────┬────────────────────┘            │
│                          │                                  │
│                          ▼                                  │
│         ┌─────────────────────────────────────┐            │
│         │  3️⃣ 변화를 유발하는 차원 검색       │            │
│         │     - 샘플 행에서 이상치 검사       │            │
│         │     - 다양한 차원으로 슬라이싱      │            │
│         │     - 특정 값으로 필터링            │            │
│         └────────────────┬────────────────────┘            │
│                          │                                  │
│                          ▼                                  │
│         ┌─────────────────────────────────────┐            │
│         │  4️⃣ 충분한 정보를 알게 됐는가?     │            │
│         │     ✅ 예 → 완료!                   │            │
│         │     ❌ 아니오 → 필터링 후 3번으로   │            │
│         └─────────────────────────────────────┘            │
│                          │                                  │
│                          │ (반복)                           │
│                          ▼                                  │
│                    ┌───────────┐                           │
│                    │  🎯 해결!  │                           │
│                    └───────────┘                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 각 단계 상세

| 단계 | 목적 | 방법 |
|------|------|------|
| **1단계** | 조사 시작점 설정 | 알림, 고객 불만 등 촉발 이벤트 확인 |
| **2단계** | 현상 검증 | 성능 변화가 그래프 곡선에 나타나는지 확인 |
| **3단계** | 원인 탐색 | 샘플 검사, 슬라이싱, 필터링으로 패턴 발견 |
| **4단계** | 반복 또는 완료 | 충분하면 종료, 아니면 필터링 후 반복 |

#### 3단계의 세부 기법

```
┌─────────────────────────────────────────────────────────────┐
│              3단계: 변화를 유발하는 차원 검색                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  방법 A: 샘플 행 검사 (Sample Row Examination)              │
│  ┌─────────────────────────────────────────────────┐       │
│  │  변화가 나타나는 영역의 샘플 행 추출            │       │
│  │  → 열에서 이상치(outlier)가 있는지 확인         │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
│  방법 B: 다양한 차원으로 슬라이싱 (Slice & Dice)            │
│  ┌─────────────────────────────────────────────────┐       │
│  │  - 행들을 다양한 차원으로 잘라보기              │       │
│  │  - 특정 차원에서 뚜렷한 행동 차이 발견?         │       │
│  │  - 예: status_code로 GROUP BY 실험              │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
│  방법 C: 필터링 (Filtering)                                 │
│  ┌─────────────────────────────────────────────────┐       │
│  │  특정 차원이나 값으로 필터링                    │       │
│  │  → 잠재적 이상치를 더 잘 드러냄                 │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### Brute-Force 방법

```
💡 사전 지식이나 지혜 없이도 가능:

   모든 가용 차원을 순환하며
   → 어떤 차원이 이상 그래프와 상관/설명하는지 식별
   → 시스템에 대한 사전 지식 불필요!

   ⚠️ 단점: 시간이 오래 걸림
   ✅ 해결: 자동화!
```

### 4. Core Analysis Loop 자동화

```
💬 비유: 탐정이 모든 증거를 일일이 분석하는 대신,
       포렌식 팀이 빠르게 지문 대조, DNA 분석을 해주는 것처럼,
       컴퓨터가 무거운 계산을 대신합니다.
```

#### 자동화 원리

```
┌─────────────────────────────────────────────────────────────┐
│                 자동화된 Core Analysis Loop                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  목표: 정상 시스템 노이즈 속에서 중요한 신호 찾기            │
│                                                             │
│  방법:                                                      │
│  ┌─────────────────────────────────────────────────┐       │
│  │  1. 관심 영역 격리 (이상 영역)                  │       │
│  │                                                 │       │
│  │  2. 모든 차원의 값 검색:                        │       │
│  │     - 격리된 영역 내부 (이상)                   │       │
│  │     - 격리된 영역 외부 (베이스라인)             │       │
│  │                                                 │       │
│  │  3. 두 영역 간 차이(diff) 계산                  │       │
│  │                                                 │       │
│  │  4. 차이가 큰 순서로 정렬                       │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
│  결과:                                                      │
│  "이상 영역과 나머지 시스템이 어떻게 다른지"                │
│  → 하나의 편차든 수십 개의 편차든 한눈에 파악               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 자동화 예시: 지연 스파이크 분석

```
요청 지연 스파이크 격리 → 자동화된 분석 결과:

┌─────────────────────────────────────────────────────────────┐
│  차원                  │ 이상 영역 │ 베이스라인 │ 차이    │
├─────────────────────────────────────────────────────────────┤
│  request.endpoint      │           │            │         │
│  (value: batch)        │   100%    │    20%     │ +80%    │
├─────────────────────────────────────────────────────────────┤
│  handler_route         │           │            │         │
│  (value: /1/markers/)  │   100%    │    10%     │ +90%    │
├─────────────────────────────────────────────────────────────┤
│  request.header.       │           │            │         │
│  user_agent (populated)│    97%    │   100%     │ -3%     │
└─────────────────────────────────────────────────────────────┘

💡 해석: batch 엔드포인트, /1/markers/ 경로가
        이상 영역에서 특히 높은 비율로 나타남
        → 이 조건이 지연을 유발할 가능성 높음
```

### 5. Honeycomb BubbleUp 예시

```
💬 비유: 현미경으로 세포를 관찰할 때 염색을 하면
       특정 구조가 더 잘 보이듯이, BubbleUp은
       이상 영역을 "염색"하여 차이를 시각화합니다.
```

#### BubbleUp 워크플로우

```
┌─────────────────────────────────────────────────────────────┐
│                   BubbleUp 분석 과정                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 히트맵에서 관심 영역 시각화                        │
│  ┌─────────────────────────────────────────────────┐       │
│  │  성능 히트맵                                    │       │
│  │  ████████████████████████████████████████████  │       │
│  │  ████████████████████████▓▓▓▓████████████████  │       │
│  │  ████████████████████████▓▓▓▓████████████████  │       │
│  │  ████████████████████████████████████████████  │       │
│  │                         ↑                      │       │
│  │                    스파이크/이상               │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
│  Step 2: 이상 영역 주위에 박스 그리기 (클릭 & 드래그)        │
│                                                             │
│  Step 3: BubbleUp 자동 계산                                 │
│  - 박스 내부 (이상) vs 박스 외부 (베이스라인)               │
│  - 모든 차원의 값 비교                                      │
│  - 차이 비율로 정렬                                         │
│                                                             │
│  Step 4: 결과 시각화                                        │
│  ┌─────────────────────────────────────────────────┐       │
│  │  🟧 주황색 막대 = 이상 영역 (선택된 박스)        │       │
│  │  🟦 파란색 막대 = 베이스라인 (나머지)           │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘

```
💬 비유: 과거에는 AI가 "패턴 인식"만 했다면,

#### AIOps에서 Generative AI로

| 구분 | AIOps (2017~) | Generative AI (현재) |
|------|---------------|---------------------|
| 목적 | 알림 노이즈 감소, 이상 탐지 | 분석 자동화, 맥락 이해 |
| 방식 | ML로 패턴 학습 | LLM으로 의미 이해 |
| 한계 | 변화하는 베이스라인 모델링 어려움 | 환각(Hallucination) 가능성 |
| 장점 | 대규모 데이터셋에서 패턴 식별 | 인간 수준의 맥락 추가 가능 |

## 💡 실무 적용 포인트

### 면접 질문 예상

#### Q1: Core Analysis Loop의 4단계를 설명해주세요.

```
A: Core Analysis Loop는 가설 기반 디버깅 기법으로 4단계로 구성됩니다:

   1단계: 조사 촉발점으로 시작
   - 알림, 고객 불만 등 조사를 시작하게 된 이유 확인
   - "무엇이 느리다"는 알지만 "왜"는 모르는 상태

   2단계: 알고 있는 것이 사실인지 검증
   - 성능 변화가 실제로 발생하는지 데이터로 확인
   - 그래프에서 곡선 변화 식별

   3단계: 변화를 유발하는 차원 검색
   - 샘플 행에서 이상치 검사
   - 다양한 차원으로 슬라이싱 (예: status_code로 GROUP BY)
   - 특정 값으로 필터링하여 이상치 노출

   4단계: 충분한 정보인지 판단
   - 예: 완료!
   - 아니오: 필터링 후 3단계로 돌아가 반복

   핵심: 사전 지식 없이도 체계적으로 문제 해결 가능
```

#### Q2: "알려진 조건에서 디버깅"과 "제1원칙에서 디버깅"의 차이는?

```
A: 두 접근법의 핵심 차이점:

   알려진 조건에서 디버깅:
   - 시스템에 대한 친밀한 경험 필요
   - 직관과 패턴 매칭에 의존
   - 런북, 대시보드로 지식 포착 시도
   - 시니어 엔지니어에게 유리
   - 확장 불가능 (모든 사람이 전문가가 될 수 없음)

   제1원칙에서 디버깅:
   - 사전 가정 없이 시작
   - 데이터로 가설 검증
   - 관찰과 실험으로 결론 도출
   - 호기심 많은 사람에게 유리
   - 민주화된 디버깅 (누구나 가능)

   실무 영향:
   - 전통적: 같은 방식으로 두 번 실패하지 않는 현대 시스템에 부적합
   - 제1원칙: 처음 보는 문제도 체계적으로 해결 가능

   결론: 복잡한 분산 시스템에서는 제1원칙 접근이 필수
```

#### Q3: BubbleUp 같은 자동화 도구의 원리는 무엇인가요?

```
A: BubbleUp은 Core Analysis Loop를 자동화합니다:

   원리:
   1. 관심 영역 격리 (이상 영역 선택)
   2. 모든 차원의 값을 두 영역에서 검색:
      - 격리된 영역 내부 (이상)
      - 격리된 영역 외부 (베이스라인)
   3. 두 영역 간 차이(diff) 계산
   4. 차이가 큰 순서로 정렬

   결과:
   - "이 영역이 나머지 시스템과 어떻게 다른가"를 한눈에 파악
   - 주황색 막대: 이상 영역의 비율
   - 파란색 막대: 베이스라인의 비율

   실제 예시:
   - global.availability_zone = us-east-1a
   - 이상 영역에서 98%, 베이스라인에서 17%
   - 즉시 특정 AZ에서 문제가 집중됨을 파악

   핵심 가치:
   - 수동으로 수십 개 차원을 검사하는 시간 절약
   - 사전 지식 없이도 패턴 발견
   - 시스템에 "지능"을 주입할 필요 없음 (브루트포스 계산)
```

#### Q4: Wide Structured Events가 Core Analysis Loop에 필수인 이유는?

```
A: Wide Structured Events 없이는 분석 자체가 불가능합니다:

   Metrics의 한계:
   - 넓은 컨텍스트 부족
   - 슬라이스 & 다이스 불가능
   - 줌인/줌아웃 불가능

   Logs의 조건부 가능:
   - Request ID, Trace ID 등 헤더 올바르게 추가해야 함
   - 대량의 후처리로 이벤트 재구성 필요
   - 읽기 시점 집계 + 복잡한 커스텀 쿼리 필요
   - 결국 Wide Structured Events를 재구성하는 것

   Wide Structured Events의 장점:
   - 모든 컨텍스트가 하나의 이벤트에 포함
   - 상관관계 재구성 불필요
   - 어떤 차원으로든 즉시 슬라이싱 가능
   - High Cardinality, High Dimensionality 지원

   결론:
   도구가 아무리 좋은 자동화를 제공해도,
   기본 데이터 형식이 Wide Structured Events가 아니면
   수동 Core Analysis Loop조차 수행 불가능
```

## 📊 핵심 용어 정리

| 용어 | 정의 |
|------|------|
| **First Principle** | 다른 가정에서 추론되지 않은 기본 가정 |
| **Core Analysis Loop** | 가설 기반 디버깅을 위한 4단계 반복 프로세스 |
| **BubbleUp** | Honeycomb의 자동화된 Core Analysis Loop 기능 |
| **Baseline** | 정상 상태의 시스템 성능 (비교 대상) |
| **Anomaly** | 비정상적인 성능을 보이는 격리된 영역 |
| **Brute-Force** | 모든 차원을 순회하며 상관관계 찾는 방법 |
| **Slice & Dice** | 다양한 차원으로 데이터를 잘라서 분석 |
| **Wide Structured Event** | 모든 컨텍스트를 포함한 넓은 구조화된 이벤트 |
| **Hypothesis-Driven** | 가설을 세우고 데이터로 검증하는 접근법 |
| **Democratized Debugging** | 전문가가 아니어도 디버깅 가능한 상태 |
