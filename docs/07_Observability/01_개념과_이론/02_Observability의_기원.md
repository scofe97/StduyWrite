# Chapter 2: The Origins of Observability in Software

## 📌 핵심 요약

> **Observability의 기원**은 Facebook의 Scuba 도구 경험에서 비롯되었습니다. Charity Majors와 Christine Yen은 이 경험을 바탕으로 **6가지 기능적 요구사항**을 정의하고 Honeycomb을 설립했습니다. 현재 Observability는 인프라 비용의 **15-25%**를 차지하지만, 대부분의 조직은 "Observability 가격에 모니터링 결과"를 받고 있습니다. 핵심은 **빠른 피드백 루프**를 통해 복잡한 시스템을 이해하는 것이며, **Multiple Pillars → Unified Storage** 모델로의 전환이 그 해답입니다.

### 4. 빠른 피드백 루프의 가치

```
💬 비유: 운동할 때 거울을 보며 자세를 교정하는 것처럼,
       빠른 피드백은 즉각적인 개선을 가능하게 합니다.
       12-24시간 후에야 결과를 볼 수 있다면,
       대부분의 엔지니어는 이미 다른 문제로 넘어갔을 것입니다.
```

#### 피드백 루프가 중요한 이유

```
┌─────────────────────────────────────────────────────────────┐
│                  피드백 루프의 복합 효과                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   빠른 피드백 → 더 많은 질문 → 더 나은 이해 → 더 빠른 개선    │
│        ↑                                         │          │
│        └─────────────────────────────────────────┘          │
│                    (증폭 피드백 루프)                        │
│                                                             │
│   피드백 루프 예시:                                          │
│   - 배포 (Deploys)                                          │
│   - 코드 리뷰 (Code Review)                                 │
│   - 계측 (Instrumentation)                                  │
│                                                             │
│   💡 핵심 인사이트:                                          │
│   "Observability는 피드백 루프의 피드백 루프다"               │
│   "복잡한 시스템의 sense-making apparatus"                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 피드백 속도에 따른 결과

| 피드백 속도 | 결과 |
|------------|------|
| **초 단위** | 질문 → 답변 → 새 질문 → 사고 흐름 유지 |
| **분 단위** | 일부 질문 가능, 복잡한 조사 어려움 |
| **12-24시간** | 대부분 포기, 다음 문제로 이동 |

### 6. Observability의 6가지 기능적 요구사항

```
💬 비유: 레스토랑을 열 때 "맛있는 음식"이라는 막연한 목표가 아니라
       "신선한 재료, 숙련된 셰프, 적절한 온도, 빠른 서빙" 등
       구체적인 요구사항을 정의하듯이, Observability도
       명확한 기술적 요구사항이 필요합니다.
```

#### 6가지 요구사항 상세

```
┌─────────────────────────────────────────────────────────────┐
│           Observability 6가지 기능적 요구사항                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1️⃣ High Cardinality 데이터 분석 능력                       │
│     ├─ User ID, Transaction ID, IP, Container ID 등         │
│     └─ 세분화된 시스템 세부 정보에서 공통점 찾기              │
│                                                             │
│  2️⃣ High Dimensionality (초고차원 컨텍스트)                 │
│     ├─ 어떤 속성이 필요할지 미리 알 수 없음                  │
│     └─ 가능한 모든 데이터를 풍부하게 수집                    │
│                                                             │
│  3️⃣ 통합 데이터 저장소 + 초고속 분석                        │
│     ├─ 단일 저장소에 모든 데이터 통합                        │
│     ├─ 어떤 차원으로 슬라이싱해도 빠른 조회                  │
│     └─ Columnar Database 채택                               │
│                                                             │
│  4️⃣ 도구 간 상관관계 분석 불필요                            │
│     ├─ 서로 다른 도구의 데이터 타입을 공통 타입으로 분해      │
│     └─ "Arbitrarily-wide structured event" 탄생             │
│                                                             │
│  5️⃣ 탐색적, 인터랙티브 시각적 인터페이스                    │
│     ├─ 사전 제작된 정적 대시보드에 의존하지 않음             │
│     ├─ 줌인/줌아웃, 슬라이스/다이스 자유롭게                 │
│     └─ 동적 탐색으로 빠른 답변 도출                          │
│                                                             │
│  6️⃣ 실시간에 가까운 처리                                   │
│     ├─ 쿼리 결과: 초 단위                                   │
│     ├─ 데이터 수집 후 분석 가능: 초 단위                    │
│     └─ 전체 분석 사이클: 실시간                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 기술적 선택

| 요구사항 | 기술적 해결책 |
|----------|-------------|
| High Cardinality | Columnar Storage Engine |
| High Dimensionality | Wide Structured Events |
| 통합 저장소 | 단일 데이터 저장소 |
| 빠른 분석 | Columnar Database (DuckDB, ClickHouse, Snowflake) |
| 인터랙티브 UI | 탐색적 시각화 인터페이스 |
| 실시간 | 스트리밍 인제스션 + 빠른 쿼리 엔진 |

### 7. Multiple Pillars vs Unified Storage 모델

```
┌─────────────────────────────────────────────────────────────┐
│        Multiple Pillars vs Unified Storage 비교             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Multiple Pillars (다중 기둥)                              │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐       │
│   │ Metrics │  │  Logs   │  │ Traces  │  │Profiling│       │
│   │  TSDB   │  │Log Aggr.│  │Trace DB │  │Prof. DB │       │
│   └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘       │
│        │            │            │            │             │
│        └────────────┼────────────┼────────────┘             │
│                     │ 수동 링크  │                          │
│                     ▼            ▼                          │
│              ┌────────────────────────┐                    │
│              │  💰 각 도구별 저장 비용  │                    │
│              │  💰 도구 간 링크 비용    │                    │
│              └────────────────────────┘                    │
│                                                             │
│   ─────────────────────────────────────────────────────    │
│                                                             │
│   Unified Storage (통합 저장소)                             │
│   ┌─────────────────────────────────────────────────┐      │
│   │              통합 텔레메트리 저장소               │      │
│   │  ┌─────────────────────────────────────────┐   │      │
│   │  │   Wide Structured Events                │   │      │
│   │  │   (Metrics + Logs + Traces + ...)       │   │      │
│   │  └─────────────────────────────────────────┘   │      │
│   │                                                 │      │
│   │  💰 한 번만 저장 비용 지불                       │      │
│   │  🔍 Trace/Log/Metric 뷰로 자유롭게 시각화        │      │
│   └─────────────────────────────────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 두 모델 비교

| 구분 | Multiple Pillars | Unified Storage |
|------|------------------|-----------------|
| **저장소** | 도구별 분리 (10-20개) | 단일 통합 저장소 |
| **비용** | N배 (도구 수만큼) | 1배 |
| **상관관계** | 수동 링크 필요 | 자동 (데이터 내 관계 보존) |
| **쿼리** | 도구별 개별 쿼리 | 통합 쿼리/조인 가능 |
| **시각화** | 도구별 대시보드 | 동일 데이터의 다양한 뷰 |

### 8. 사회기술적(Sociotechnical) 이점

```
💬 비유: 같은 악보를 보고 연주하는 오케스트라처럼,
       통합된 데이터로 전체 팀이 같은 관점에서 시스템을 이해합니다.
```

#### Observability가 가능하게 하는 것

```
┌─────────────────────────────────────────────────────────────┐
│              Observability의 사회기술적 이점                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  🔍 Known-Unknowns (알고 있는 미지):                        │
│     → 알고 있지만 검증이 필요한 문제 디버깅                  │
│                                                             │
│  🔮 Unknown-Unknowns (모르는 미지):                         │
│     → 이전에 본 적 없는 문제, 예상치 못한 질문               │
│     → 사전 예측 없이도 데이터를 따라가면 답을 찾음           │
│                                                             │
│  📊 컨텍스트 보존:                                          │
│     → 다른 모니터링 도구가 버리는 연결 고리 유지             │
│     → 슬라이스/다이스, 개별 요청 줌인, 트렌드 줌아웃         │
│                                                             │
│  👁️ 다양한 시각화:                                          │
│     → 같은 데이터를 Log, Trace, Metric으로 시각화           │
│     → 조사 목적에 맞는 뷰 선택                               │
│                                                             │
│  🎯 관점의 전환:                                            │
│     기존: 소프트웨어가 "어떻게 운영되는지" 보여주기          │
│     신규: 소프트웨어를 "어떻게 개발할지" 이해하기            │
│                                                             │
│     기존: 인프라 중심, 컴포넌트 상태 기반 임계값             │
│     신규: 코드 중심, 개별 사용자 요청 기반 경험              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 모니터링 vs Observability

```
┌─────────────────────────────────────────────────────────────┐
│                    금융 비유로 본 차이                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
```

## 💡 실무 적용 포인트

### 면접 질문 예상

#### Q1: Observability 비용이 인프라의 15-25%를 차지하는데, 왜 문제인가요?

```
A: 비용 자체가 문제가 아니라 "비용 대비 가치"가 문제입니다.

   현재 상황:
   - 비용: 15년간 매년 40% 증가 (수백 배 상승)
   - 결과: 10-15년 전과 거의 동일한 기능
   - 도구: 동일한 모니터링 도구에 "Observability" 리브랜딩

   문제의 본질:
   "Observability 가격에 모니터링 결과를 받고 있다"

   해결책:
   1. Multiple Pillars → Unified Storage 모델 전환
   2. 탐색적 디버깅 가능한 도구 도입
   3. 빠른 피드백 루프 구축
   4. 데이터 중복 저장 제거 (도구 통합)

   기대 결과:
   - 10x 빠른 배포
   - 50% 작은 팀
   - 고객 만족도 및 수익 증가
```

#### Q2: "빠른 피드백 루프"가 왜 Observability의 핵심인가요?

```
A: 피드백 루프는 소프트웨어 개발의 품질과 속도를 결정합니다.

   피드백 루프의 종류:
   - 배포 (Deploys)
   - 코드 리뷰 (Code Review)
   - 계측 (Instrumentation)

   Observability의 역할:
   "Observability는 피드백 루프의 피드백 루프"
   "복잡한 시스템의 sense-making apparatus"

   속도에 따른 차이:
   - 초 단위: 질문 → 답변 → 새 질문 → 사고 흐름 유지
   - 12-24시간: 대부분 포기, 다음 문제로 이동

   실제 사례:
   한 고객사는 고세분화 데이터를 데이터 웨어하우스에서
   야간 처리 후에만 제공했습니다. 12-24시간 지연으로 인해
   대부분의 엔지니어가 질문을 포기했고,
   팀은 "어둠 속에서 비행"하는 상태가 되었습니다.

   핵심 원리:
   빠른 피드백 → 더 많은 질문 → 더 나은 이해 → (증폭 효과)
```

#### Q3: Observability의 6가지 기능적 요구사항을 설명해주세요.

```
A: Honeycomb 창업자들이 정의한 6가지 요구사항:

   1. High Cardinality 분석
      - User ID, Transaction ID 등 고유 식별자 분석
      - 세분화된 시스템 세부 정보에서 패턴 발견

   2. High Dimensionality (초고차원)
      - 어떤 속성이 필요할지 미리 알 수 없음
      - 가능한 모든 데이터를 풍부하게 수집

   3. 통합 저장소 + 초고속 분석
      - 단일 저장소에 모든 데이터 통합
      - Columnar Database로 빠른 분석

   4. 도구 간 상관관계 불필요
      - 다른 데이터 타입을 공통 타입으로 분해
      - "Arbitrarily-wide structured event" 사용

   5. 탐색적, 인터랙티브 UI
      - 정적 대시보드 대신 동적 탐색
      - 줌인/줌아웃, 슬라이스/다이스

   6. 실시간 처리
      - 쿼리 결과: 초 단위
      - 데이터 수집 → 분석 가능: 초 단위

   이 요구사항들이 모여 "어떤 시스템 상태든 이해할 수 있는"
   진정한 Observability를 가능하게 합니다.
```

#### Q4: Multiple Pillars 모델에서 Unified Storage 모델로 전환해야 하는 이유는?

```
A: 비용, 효율성, 정확성 세 가지 측면에서 이점이 있습니다.

   1. 비용 절감
      - Multiple Pillars: 10-20개 도구 = 10-20배 비용
      - Unified: 한 번만 저장 = 1배 비용

   2. 효율성 향상
      - Multiple Pillars: 도구 간 수동 상관관계 분석
      - Unified: 자동으로 관계 보존, 통합 쿼리/조인

   3. 정확성 개선
      - Multiple Pillars: 인위적 링크로 인한 오류 발생
      - Unified: 데이터 내 관계가 그대로 보존

   4. 유연성
      - Multiple Pillars: 도구별 고유 대시보드
      - Unified: 동일 데이터를 Log/Trace/Metric 뷰로 시각화

   핵심 통찰:
   "Observability 데이터는 그냥 데이터다"
   → 다른 데이터 문제처럼 접근해야 함
   → 분리된 도구에 데이터가 흩어져 있으면
      간단한 데이터 문제도 볼 수 없음
```

## 📊 핵심 용어 정리

| 용어 | 정의 |
|------|------|
| **High Cardinality** | 고유 값이 매우 많은 데이터 (예: User ID, Transaction ID) |
| **High Dimensionality** | 많은 속성/차원을 가진 데이터 |
| **Columnar Database** | 열 기반 저장으로 분석에 최적화된 DB |
| **Wide Structured Event** | 임의로 넓은 구조화된 이벤트 데이터 |
| **Multiple Pillars** | Metrics, Logs, Traces 분리 저장 모델 |
| **Unified Storage** | 모든 텔레메트리를 단일 저장소에 통합 |
| **Feedback Loop** | 출력이 입력에 영향을 주는 순환 구조 |
| **Sense-making Apparatus** | 복잡한 시스템을 이해하게 하는 장치 |
| **Known-Unknowns** | 알고 있지만 검증이 필요한 문제 |
| **Unknown-Unknowns** | 존재 자체를 모르는 문제 |
