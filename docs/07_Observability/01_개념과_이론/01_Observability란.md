# Chapter 1: What Is Observability?

## 📌 핵심 요약

> **Observability(관측 가능성)**는 소프트웨어 시스템의 **외부 출력(external outputs)**을 통해 **내부 상태(internal state)**를 얼마나 잘 추론하고 디버깅할 수 있는지를 측정하는 **소프트웨어 의존성(dependability)의 속성**입니다. 전통적인 모니터링(metrics, logs, traces의 분리된 "세 기둥")에서 **통합된 텔레메트리 데이터 저장소(unified telemetry datastore)**로의 기술적 패러다임 전환이 높은 수준의 Observability를 가능하게 합니다.

## 📖 본문 정리

### 1. Observability의 수학적 정의

```
💬 비유: 자동차 계기판으로 엔진 내부 상태를 추론하는 것처럼,
       시스템의 "외부 출력"만으로 "내부 상태"를 파악하는 능력입니다.
```

#### Rudolf E. Kálmán의 정의 (1960)

> **"Observability는 시스템의 외부 출력에 대한 지식으로부터 내부 상태를 얼마나 잘 추론할 수 있는지를 측정하는 것이다."**

| 구분 | 제어 이론 (Control Theory) | 소프트웨어 시스템 |
|------|---------------------------|------------------|
| 대상 | 물리적 시스템 | 가상 소프트웨어 시스템 |
| 목표 | 특정 최종 상태 달성 | 디버깅 및 문제 해결 |
| 도구 | 센서, 선형대수 방정식 | 텔레메트리, 분석 도구 |
| 특성 | 수학적, 형식적 | 유연하고 적응적 |

### 2. 소프트웨어 의존성(Dependability)의 속성

```
💬 비유: 집의 안전성을 평가할 때 구조, 방화, 보안, 유지보수성을
       종합적으로 고려하듯이, 소프트웨어도 여러 속성을 함께 평가합니다.
```

#### J.C. Laprie의 의존성 프레임워크 (1994) + Observability

```
┌─────────────────────────────────────────────────────────────┐
│              소프트웨어 의존성 (Dependability)               │

│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │ Availability│  │ Reliability │  │   Safety    │         │
│  │  (가용성)    │  │  (신뢰성)   │  │   (안전성)  │         │
│  │ 사용 준비    │  │ 서비스 연속 │  │ 재앙적 결과 │         │
│  │   상태      │  │    제공     │  │    방지     │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
│                                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │  Security   │  │  Integrity  │  │Maintaintic │         │

│  │              [현대 시스템의 필수 속성]              │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 장애 대응 기법 (Fault Handling Techniques)

| 기법 | 설명 | Observability 관련성 |
|------|------|---------------------|
| **Fault Prevention** | 장애 도입/발생 방지 | 사전 분석 지원 |
| **Fault Tolerance** | 활성 장애에도 서비스 제공 | 장애 감지 필수 |
| **Fault Removal** | 장애 수 및 심각도 감소 | 근본 원인 식별 필요 |
| **Fault Forecasting** | 현재/미래 장애 예측 | 패턴 분석 기반 |

### 3. 전통 시스템 vs 현대 분산 시스템

```
┌───────────────────────────────────────────────────────────────┐
│                    시스템 복잡도 진화                          │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│   과거 (모놀리식)              현재 (분산 시스템)              │
│   ┌─────────────┐             ┌─────────────────────┐        │
│   │             │             │  ┌──┐ ┌──┐ ┌──┐    │        │
│   │   단일      │             │  │μS│ │μS│ │μS│    │        │
│   │   서버      │    ──→      │  └──┘ └──┘ └──┘    │        │
│   │             │             │    ↓   ↓   ↓       │        │
│   │   ⬆️ ⬇️     │             │  ┌──┐ ┌──┐ ┌──┐    │        │
│   │ UP / DOWN   │             │  │DB│ │캐시│ │SaaS│   │        │
│   └─────────────┘             │  └──┘ └──┘ └──┘    │        │
│                               └─────────────────────┘        │
│   가용성: 이진적               가용성: 부분 장애 상태          │
│   (UP or DOWN)                (무한한 조합 가능)              │
│                                                               │
│   장애: 예측 가능              장애: 예측 불가능               │
│   모니터링: 충분               모니터링: 불충분                │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

#### 현대 시스템에서 "설명 불가능한 느림"의 원인 예시

```
💬 비유: 100개의 기어가 연결된 시계에서 하나의 기어가 살짝 느려지면
       전체 시계가 미묘하게 영향받는 것처럼, 분산 시스템의 장애는
       수많은 요인의 조합으로 발생합니다.
```

| 원인 | 영향 | 전통 모니터링으로 감지? |
|------|------|----------------------|
| 알 수 없는 파일의 기본 설정이 us-east-1 지정 | 북미 외 사용자 지연 증가 | ❌ 집계 지표로는 불가 |
| 로드밸런서 부분 성능 저하 | 1/100 요청만 지연 | ⚠️ 99% 가용성으로 표시 |
| Progressive Delivery로 다중 버전 운영 | 신규 버전만 성능 저하 | ❌ 버전별 분리 불가 |
| 외부 API 의존성 성능 저하 | 정상 여부 판단 불가 | ❌ 접근 권한 없음 |

### 4. Observability 수준 평가하기

```
💬 비유: 의사가 환자를 진단할 때, 숙련된 의사만 "느낌"으로 아는 것이
       아니라 누구나 검사 결과를 보고 판단할 수 있어야 하듯이,
       시스템도 경험에 상관없이 누구나 디버깅할 수 있어야 합니다.
```

#### Observability 평가의 3가지 축

```
┌─────────────────────────────────────────────────────────────┐
│                 Observability 평가 축                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1️⃣ 장애 감지 (Detection)                                  │
│     ├─ 고객보다 먼저 문제를 알 수 있는가?                     │
│     └─ 모니터링 시스템이 중요한 이슈를 놓치지 않는가?          │
│                                                             │
│  2️⃣ 원인 분류 (Triage)                                     │
│     ├─ 문제의 정확한 발생 위치를 파악할 수 있는가?            │
│     └─ 연쇄 장애 시 시작점을 식별할 수 있는가?                │
│                                                             │
│  3️⃣ 원인 이해 (Reasoning)                                  │
│     ├─ 경험에 상관없이 누구나 진단할 수 있는가?               │
│     └─ 데이터 기반 추론이 가능한가? (직관이 아닌)             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 높은 Observability 시스템의 특징

| 특징 | 설명 |
|------|------|
| **내부 작동 이해** | 애플리케이션의 내부 동작을 이해할 수 있음 |
| **새로운 상태 이해** | 이전에 보지 못한 시스템 상태도 이해 가능 |
| **외부 도구만으로 이해** | 외부 도구로 관찰하고 질의하여 내부 상태 파악 |
| **사전 지식 불필요** | 새 코드 배포 없이 상태 이해 가능 |

#### 자가 진단 질문 (Litmus Test)

```
높은 Observability를 갖춘 시스템에서 가능한 것들:

✅ 조사 막다른 길 없이 개방형 질문에 지속적 답변
✅ 특정 사용자가 겪는 경험을 언제든 이해
✅ 집계 뷰부터 개별 요청까지 모든 단면 확인
✅ 예상치 못한 동작을 보이는 사용자들의 공통 속성 비교
✅ 의심스러운 속성을 모든 요청에서 검색
✅ 가장 많은 부하를 생성하는 사용자 순위 식별
✅ 최근 성능에 영향을 주기 시작한 사용자 식별
✅ 142번째로 느린 사용자의 요청을 분리하여 원인 파악
✅ 99.99% 백분위수 이상에 숨겨진 타임아웃 발견
✅ 사전 예측 없이도 질문에 답변 가능
✅ 이전에 한 번도 본 적 없는 문제도 답변 가능
✅ 수 초 내에 답변을 얻어 사고 흐름 유지
✅ 복잡하고 깊이 숨겨진 장애도 수 분 내 분리
```

### 5. 모니터링 vs Observability

```
┌─────────────────────────────────────────────────────────────┐
│              모니터링 vs Observability 비교                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│        모니터링                    Observability             │
│   ┌─────────────────┐         ┌─────────────────┐          │
│   │  임계값 기반    │         │  탐색적 분석    │          │
│   │    반응형      │         │    선제적       │          │
│   │                │         │                │          │
│   │  "무엇이       │         │  "왜 그런지    │          │
│   │   문제인가?"   │         │   이해하기"    │          │
│   │                │         │                │          │
│   │  알려진 장애   │         │  미지의 장애   │          │
│   │    감지       │         │    탐색        │          │
│   └─────────────────┘         └─────────────────┘          │
│                                                             │
│   집계된 시스템 상태            개별 트랜잭션 수준            │

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 업계 현실: 용어의 혼란

```
⚠️ 마케팅 vs 실제:
   - 일부 벤더: "Observability = 모니터링의 새 이름일 뿐"
   - 다른 벤더: "Observability = logs + metrics + traces"

💡 저자의 입장:
   도구 분류가 아닌 "결과(outcomes)"로 Observability를 판단해야 함
   → 어떤 도구든 높은 수준의 Observability를 가능하게 하면 OK
```

### 6. 기술적 패러다임 전환 (Step Change)

```
💬 비유: 면도기 날이 1개 → 2개 → 7개로 늘어난 것은 점진적 개선이지만,
       전기 면도기의 등장은 패러다임 전환입니다.
       Observability에서 "통합 데이터 저장소"가 바로 그런 전환입니다.
```

```
│              │   - 집계 ↔ 개별 요청 탐색     │               │

│                                                             │

```

#### 패러다임 전환 비교

| 구분 | Multiple Pillars | Unified |
|------|------------------|---------|
| 데이터 저장 | 분리된 저장소 | 통합 저장소 |
| 상관관계 분석 | 수동 (엔지니어) | 자동 (시스템) |
| 컨텍스트 전환 | 빈번함 | 최소화 |
| 복잡한 아키텍처 대응 | 비효율적 | 효율적 |
| Observability 수준 | 낮음 ~ 중간 | 높음 |

### 면접 질문 예상

#### Q1: Observability와 모니터링의 차이점은 무엇인가요?

```
A: 모니터링은 사전에 정의된 임계값과 지표를 기반으로 시스템이
   "정상인지 비정상인지"를 판단하는 반응형(reactive) 접근입니다.

   반면, Observability는 시스템의 외부 출력을 통해 내부 상태를
   이해하고 디버깅할 수 있는 능력을 측정하는 시스템 속성입니다.

   핵심 차이:
   - 모니터링: "무엇이 문제인가?" (알려진 장애 감지)
   - Observability: "왜 그런가?" (미지의 장애 탐색)

   모니터링은 Observability를 달성하기 위한 하나의 도구일 뿐이며,
   높은 수준의 Observability를 위해서는 탐색적 분석이 가능해야 합니다.
```

#### Q2: "세 기둥(Three Pillars)" 접근법의 한계는 무엇인가요?

```
A: Logs, Metrics, Traces를 별도의 도구와 저장소에서 관리하는
   "세 기둥" 접근법은 역사적으로 의미가 있었지만, 현대 분산 시스템에서는
   여러 한계가 있습니다:

   1. 수동 상관관계 분석 필요
      - 도구 간 컨텍스트 전환으로 인한 인지 부하
      - 데이터 간 연결을 엔지니어가 직접 수행

   2. 오류 발생 가능성
      - 시간 범위 불일치, 누락된 연결고리

   3. 비효율성
      - 복잡한 아키텍처에서 상관관계 파악이 어려움
      - 디버깅 시간 증가

   해결책: 통합 텔레메트리 저장소로 자동 상관관계 분석 지원
```

#### Q3: 시스템의 Observability 수준을 어떻게 평가하나요?

```
A: Observability는 수치로 측정되지 않고 정성적으로 평가됩니다.
   세 가지 축으로 평가할 수 있습니다:

   1. 장애 감지 (Detection)
      - 고객보다 먼저 문제를 인지하는가?
      - 중요한 이슈를 놓치지 않는가?

   2. 원인 분류 (Triage)
      - 문제의 정확한 발생 위치를 파악할 수 있는가?
      - Alert Storm에서 시작점을 식별할 수 있는가?

   3. 원인 이해 (Reasoning)
      - 경험과 무관하게 누구나 진단할 수 있는가?
      - 데이터 기반 추론이 가능한가? (직관이 아닌)

   추가로, 다음 질문에 "예"라고 답할 수 있다면 높은 Observability:
   - 이전에 본 적 없는 문제도 디버깅 가능한가?
   - 사전 예측 없이도 질문에 답변 가능한가?
   - 수 초 내에 답변을 얻을 수 있는가?
```

#### Q4: Observability가 소프트웨어 의존성(Dependability)의 속성인 이유는?

```
A: J.C. Laprie의 1994년 프레임워크는 소프트웨어 의존성을
   Availability, Reliability, Safety, Security, Integrity,
   Maintainability로 정의했습니다.

   현대 분산 시스템에서는 Observability가 추가되어야 하는 이유:

   1. 장애의 복잡성 증가
      - 부분 장애 상태(partial failure)가 일반화
      - 이진적(UP/DOWN)이 아닌 무한한 조합의 상태

   2. 장애 대응 기법과의 연관성
      - Fault Detection: Observability 없이 감지 불가
      - Fault Removal: 근본 원인 식별 필요
      - Fault Forecasting: 패턴 분석 기반

   3. 의존성의 다른 속성들과의 상호작용
      - 높은 Availability를 유지하려면 빠른 장애 감지 필요
      - Maintainability를 위해 시스템 이해가 선행되어야 함

   따라서 Observability는 다른 의존성 속성들을 지원하는
   기반(foundation) 역할을 합니다.
```
