# Chapter 5: Acting On and Debugging SLO-Based Alerts

## 📌 핵심 요약

> **SLO 기반 알림의 핵심**: Error Budget을 소진하기 **전에** 예측하여 알림을 발생시키는 것이 핵심입니다. 전통적인 Threshold 알림은 문제 해결 가이드를 제공하지 못하지만, SLO 기반 알림은 **예측(Forecasting)**과 **Event 데이터**를 활용하여 더 정확하고 액션 가능한 알림을 제공합니다.

## 📖 본문 정리

### 1. Error Budget 기본 개념

```
💬 비유: Error Budget은 "월간 용돈"과 같습니다.
       용돈이 바닥나기 전에 지출 속도를 파악하고,
       월말에 돈이 떨어지지 않도록 미리 조절해야 합니다.
```

#### Error Budget이란?

**Error Budget** = 비즈니스가 감내할 수 있는 최대 시스템 비가용 시간

| SLO 목표 | 연간 허용 다운타임 | 월간 허용 다운타임 |
|----------|-------------------|-------------------|
| 99.9% | 8시간 45분 57초 | 43분 50초 |
| 99.95% | 4시간 22분 58초 | 21분 54초 |
| 99.99% | 52분 35초 | 4분 23초 |

#### Error Budget 계산 방식

```
┌─────────────────────────────────────────────────────────────┐
│                   Error Budget 계산                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Time-Based (시간 기반):                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 허용 다운타임 = (1 - SLO%) × 총 시간               │   │
│  │ 예: (1 - 0.999) × 43,800분/월 = 43.8분/월          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Event-Based (이벤트 기반):                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 허용 실패 = 총 요청 수 × (1 - SLO%)                │   │
│  │ 예: 100,000,000 요청 × 0.001 = 100,000 실패 허용   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  남은 Error Budget = 허용 실패 - 소진된(실패한) 요청 수    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### Error Budget 소진 시 발생하는 일

```
Error Budget 소진 시 조치:
┌─────────────────────────────────────────────────────────────┐
│  ⚠️ Feature Freeze (기능 개발 중단)                         │
│  ⚠️ 안정성 개선 작업만 수행                                 │
│  ⚠️ Production 변경 최소화                                  │
│  ⚠️ 인시던트 복구에만 집중                                  │
└─────────────────────────────────────────────────────────────┘

→ 목표: Error Budget이 완전히 소진되기 **전에** 대응하는 것!
```

### 2. 시간 프레이밍: Sliding Window vs Fixed Window

```
💬 비유: Fixed Window는 "월급날 리셋되는 통장",
       Sliding Window는 "항상 최근 30일 지출을 추적하는 가계부"입니다.
       고객의 기억은 월말에 리셋되지 않습니다!
```

#### 두 Window 방식 비교

```
┌─────────────────────────────────────────────────────────────┐
│         Fixed Window vs Sliding Window                      │
├────────────────────────┬────────────────────────────────────┤
│    Fixed Window        │       Sliding Window               │
├────────────────────────┼────────────────────────────────────┤
│  ┌───┬───┬───┬───┐    │    ←──── 30일 ────→               │
│  │1월│2월│3월│4월│    │   [    항상 이동    ]              │
│  └───┴───┴───┴───┘    │        ↓                          │
│    ↑                   │   [    항상 이동    ]              │
│  월초에 Budget 리셋    │        ↓                          │
│                        │   [    항상 이동    ]              │
├────────────────────────┼────────────────────────────────────┤
│ • 달력 기준 (1일~30일) │ • 현재 기준 trailing 30일         │
│ • 월말에 Budget 리셋   │ • 점진적 회복/소진                 │
│ • 고객 경험과 불일치   │ • 인간 감정/기억과 일치            │
│ • 극적인 리셋 발생     │ • 부드러운 변화                    │
└────────────────────────┴────────────────────────────────────┘
```

#### Sliding Window를 선택해야 하는 이유

```
고객 시나리오:
┌─────────────────────────────────────────────────────────────┐
│  31일: 대규모 장애 발생 → 환불 처리                         │
│                                                             │
│  Fixed Window 관점:                                         │
│  ├─ 1일 (다음 달): "Error Budget 리셋됐으니 괜찮아!"       │
│  └─ 2일: 또 장애 발생 → "기술적으로는 SLO 준수"            │
│                                                             │
│  고객 관점:                                                  │
│  └─ "이틀 연속 장애? 용납 불가!" 😤                        │
│                                                             │
│  결론: SLO는 법적 구속이 아닌 고객 만족도 측정 도구        │
│       인간의 감정은 월말에 리셋되지 않는다!                 │
└─────────────────────────────────────────────────────────────┘
```

#### 권장 Window 기간

| 기간 | 장단점 | 권장 여부 |
|------|--------|-----------|
| **7-14일** | 고객 기억/제품 사이클과 불일치 | ❌ 너무 짧음 |
| **30일** | 가장 실용적, 균형 잡힌 기간 | ✅ **권장** |
| **90일** | 하루에 90% 소진해도 SLO 충족 가능 | ❌ 너무 김 |

### 3. 예측 알림 방식 세 가지

#### 3.1 Threshold Crossing Alerts (임계값 교차 알림)

```
┌─────────────────────────────────────────────────────────────┐
│              Threshold Crossing Alert                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Error Budget (%)                                           │
│  100% ┤                                                     │
│       │ ╲                                                   │
│       │   ╲                                                 │
│   30% ┤─────╲─────────────── ← Threshold (임계값)          │
│       │       ╲                                             │
│       │         ╲  ← 알림 발생!                            │
│    0% ┼─────────────────────────→ Time                     │
│                                                             │
│  방식: 남은 Budget이 특정 % 이하로 떨어지면 알림           │
│  예시: "남은 Budget이 30% 이하면 알림"                     │
│                                                             │
│  ⚠️ 단점: 결국 goalpost를 옮긴 것일 뿐                    │
│          30%가 새로운 "empty"가 됨                         │
│          불필요한 Feature Freeze 시간 발생                 │
└─────────────────────────────────────────────────────────────┘
```

#### 3.2 Relative Burn Alerts (상대적 소진율 알림)

```
💬 비유: "평소 속도의 몇 배로 돈을 쓰고 있는가?"를 추적하는 것입니다.
       1배속은 정상, 2배속은 주의, 4배속은 긴급!
```

```
┌─────────────────────────────────────────────────────────────┐
│               Relative Burn Alert                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Burn Rate                                                  │
│       │                                                     │
│   4x ─┤─────────────────────── ← 🚨 Page (긴급)           │
│       │        ╱╲                                          │
│   2x ─┤───────╱──╲───────────── ← ⚠️ Non-paging          │
│       │      ╱    ╲                                        │
│   1x ─┤─────╱──────╲────────── ← 정상 burn rate           │
│       │   ╱          ╲                                     │
│    0 ─┼─────────────────────────→ Time                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**계산 예시**:
```
조건:
- 월간 요청: 100,000,000개
- SLO 목표: 99.9%
- 허용 실패: 100,000개/월
- 15분 윈도우 수: 2,880개 (30일 기준)
- 15분당 허용 실패: ~3.4개

상대적 Burn Alert 계산:
┌────────────┬───────────────────┬─────────────────────────┐
│ Burn Rate  │ 15분당 허용 실패   │ 알림 수준               │
├────────────┼───────────────────┼─────────────────────────┤
│ 1x         │ 3.4개             │ 정상                    │
│ 3x         │ 10개              │ ⚠️ 주의                │
│ 20x        │ 68개              │ 🚨 긴급 Page           │
└────────────┴───────────────────┴─────────────────────────┘
```

**핵심 차이점**: 단순 에러율(%)이 아닌 **전체 SLO 기간의 예상 요청량**을 고려

#### 3.3 Predictive Burn Alerts (예측적 소진 알림)

```
💬 비유: "현재 지출 속도라면 월말 전에 통장이 바닥날까?"를 예측하는 것입니다.
       과거 데이터를 기반으로 미래를 추정합니다.
```

```
┌─────────────────────────────────────────────────────────────┐
│              Predictive Burn Alert                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Error Budget (%)                                           │
│  100% ┤                                                     │
│       │ ╲                                                   │
│       │   ╲  Baseline Window                               │
│       │     ╲ (과거 데이터)                                │
│       │       ╲                                             │
│       │         ╲........ ← 예측선 (외삽)                  │
│       │           ╲.......                                 │
│    0% ┼─────────────╲.....──→ 🚨 Budget 소진 예측 시점    │
│       │              ↑                                      │
│       │         Lookahead Window                           │
│       │         (미래 예측 기간)                           │
│                                                             │
│  핵심: Baseline 데이터로 Lookahead 기간을 외삽(extrapolate) │
└─────────────────────────────────────────────────────────────┘
```

### 4. Baseline Window와 Lookahead Window

#### Lookahead Window (미래 예측 기간)

```
┌─────────────────────────────────────────────────────────────┐
│              Lookahead Window 선택 가이드                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  시나리오 1: 느린 성능 저하                                 │
│  ├─ 현재 99.88% → 목표 99.9%                               │
│  ├─ 1개월 후 SLO 위반 예측                                 │
│  └─ 대응: 다음 영업일에 조사 (긴급 아님)                   │
│                                                             │
│  시나리오 2: 급격한 장애                                    │
│  ├─ 현재 98% → 1시간 내 SLO 위반 예측                      │
│  ├─ 수 시간 내 월간/분기/연간 Budget 소진 가능             │
│  └─ 대응: 즉시 On-call 엔지니어 호출 🚨                    │
│                                                             │
│  → Lookahead 기간에 따라 알림 긴급도가 다름                │
└─────────────────────────────────────────────────────────────┘
```

#### Baseline Window와 Lookahead Window의 관계

```
┌─────────────────────────────────────────────────────────────┐
│         실용적 외삽 비율: 최대 4배                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  규칙: Baseline × 4 = 최대 신뢰할 수 있는 Lookahead        │
│                                                             │
│  ┌────────────────┬────────────────────────────────────┐   │
│  │ Baseline       │ Lookahead (최대)                   │   │
│  ├────────────────┼────────────────────────────────────┤   │
│  │ 15분           │ 1시간                              │   │
│  │ 30분           │ 2시간                              │   │
│  │ 1시간          │ 4시간                              │   │
│  │ 6시간          │ 24시간 (1일)                       │   │
│  │ 1일            │ 4일                                │   │
│  └────────────────┴────────────────────────────────────┘   │
│                                                             │
│  ⚠️ 계절성 보정 없이 4배 이상 외삽하면 부정확              │
│     (피크/오프피크, 주중/주말, 월초/월말 등)               │
└─────────────────────────────────────────────────────────────┘
```

#### 30일 Sliding Window에서의 조정

```
┌─────────────────────────────────────────────────────────────┐
│         4일 예측 시 Window 조정 예시                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ←────────────── 30일 Sliding Window ──────────────→       │
│                                                             │
│  ┌──────────────────────┬─────────┬──────────────────┐     │
│  │   Adjusted Lookback  │ Baseline│   Lookahead      │     │
│  │      (26일)          │  (1일)  │    (4일)         │     │
│  └──────────────────────┴─────────┴──────────────────┘     │
│                                                             │
│  계산: 26일 과거 데이터 + 1일 Baseline × 4 = 30일 예측     │
│                                                             │
│  → 4일 후의 30일 Sliding Window 상태를 예측               │
└─────────────────────────────────────────────────────────────┘
```

### 5. Short-term vs Context-aware Burn Alerts

#### 5.1 Short-term (Ahistorical) Burn Alerts

```
💬 비유: "최근 6시간 지출만 보고 월말 상태를 예측"
       과거 기록 없이, 최근 데이터만으로 선형 외삽
```

**계산 방식**:
```
┌─────────────────────────────────────────────────────────────┐
│            Short-term Burn Alert 계산                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  조건:                                                       │
│  • SLO: 99% (30일 moving window)                           │
│  • 월간 예상 요청: 43,800개                                │
│  • Error Budget: 438개 (1%)                                │
│                                                             │
│  관측 데이터:                                                │
│  • 지난 6시간: 50개 요청 중 25개 실패 (50% 실패율)         │
│                                                             │
│  ❌ 선형 외삽 (잘못된 방식):                                │
│     6시간에 25실패 → 24시간에 100실패 (438 미만 → OK?)     │
│                                                             │
│  ✅ 비례 외삽 (올바른 방식):                                │
│     24시간 예상 요청: 43,800 / 30 = 1,460개                │
│     50% 실패율 유지 시: 1,460 × 0.5 = 730 실패            │
│     730 > 438 → 🚨 약 반나절 내 Budget 소진!              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**특징**:
- 과거 이력 없이 최근 Baseline만 사용
- 계산이 단순하고 빠름
- 트래픽 변동을 고려하지 못할 수 있음

#### 5.2 Context-aware (Historical) Burn Alerts

```
💬 비유: "최근 26일간의 지출 내역 + 최근 1일 추세를 합쳐서 예측"
       전체 SLO 기간의 누적 데이터를 활용
```

**계산 방식**:
```
┌─────────────────────────────────────────────────────────────┐
│          Context-aware Burn Alert 계산                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  조건:                                                       │
│  • SLO: 99% (30일 moving window)                           │
│  • 월간 예상 요청: 43,800개                                │
│  • Error Budget: 438개                                     │
│                                                             │
│  누적 데이터:                                                │
│  • 지난 26일: 37,960개 요청 중 285개 실패                  │
│  • 지난 1일 (Baseline): 1,460개 요청 중 130개 실패         │
│                                                             │
│  계산 과정:                                                  │
│  1. 26일 전 시작: 100% Budget 남음                         │
│  2. 1일 전 시점: 65% Budget 남음 (285/438 소진)            │
│  3. 현재: 35% Budget 남음 (285+130 = 415 소진)             │
│  4. 1일 동안 30% 소진 → 2일 내 Budget 완전 소진 예측       │
│                                                             │
│  → 🚨 알림 발생!                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Go 코드 예시 (간략화)**:
```go
func isBurnViolation(now time.Time, config *BurnAlertConfig,
                     slo *SLO, timeseries *TimeseriesMap) bool {
    // 1. Lookahead 기간의 1/4을 Baseline으로 사용
    baselineOffset := config.ExhaustionMinutes / 4

    // 2. 조정된 Lookback Window 계산
    adjustedWindow := now.AddDate(0, 0, -slo.TimePeriodDays).
                      Add(config.ExhaustionMinutes)

    // 3. Lookback 기간의 누적 데이터 수집
    var runningTotal, runningFails int64
    var projectedTotal, projectedFails int64

    for _, entry := range timeseries {
        if entry.InBaselineWindow {
            // Baseline 데이터는 4배 가중치
            projectedTotal += 4 * entry.Total
            projectedFails += 4 * entry.Fails
        }
        runningTotal += entry.Total
        runningFails += entry.Fails
    }

    // 4. 예측값 계산 및 위반 여부 판단
    projectedTotal += runningTotal
    projectedFails += runningFails
    allowedFails := projectedTotal * slo.BudgetPPM / 1e6

    return projectedFails >= allowedFails
}
```

#### Short-term vs Context-aware 비교

| 구분 | Short-term | Context-aware |
|------|------------|---------------|
| **데이터 범위** | Baseline만 사용 | 전체 SLO 기간 + Baseline |
| **계산 비용** | 낮음 | 높음 (AWS Lambda $5,000/일 경험담) |
| **정확도** | 상대적으로 낮음 | 높음 |
| **남은 Budget 고려** | ❌ 고려 안 함 | ✅ 남은 Budget에 따라 민감도 조절 |
| **적합한 상황** | 빠른 감지 필요 | 정밀한 예측 필요 |

### 6. Multiple Burn Alert 설정

```
💬 비유: 단기 일기예보와 장기 일기예보를 모두 확인하는 것처럼,
       여러 시간대의 Burn Alert를 함께 설정해야 합니다.
```

#### 왜 여러 개의 Burn Alert가 필요한가?

```

│           Multiple Burn Alert 시나리오                      │

│                                                             │
│  상황: 시스템 장애 발생                                      │
│                                                             │

│                                                             │
│  1일 Burn Alert (6시간 Baseline):                          │
│  └─ ✅ "1일 기준으로는 아직 괜찮음"                        │
│                                                             │
│  이유: 서로 다른 Baseline을 사용하기 때문                   │

│  • 최근 6시간 전체: 평균적으로 안정적                      │
```

### 8. Time-Series vs Event Data for SLOs

```
💬 비유: Time-Series는 "매 분마다 교실 전체가 조용한지 시끄러운지 판단",
       Event Data는 "각 학생이 조용한지 시끄러운지 개별 판단"입니다.
       부분 장애 시 Event Data가 훨씬 정확합니다.
```

#### 두 데이터 타입 비교

```
┌─────────────────────────────────────────────────────────────┐
│         Time-Series vs Event-Based SLO 계산                 │
├────────────────────────┬────────────────────────────────────┤
│    Time-Series         │       Event-Based                  │
├────────────────────────┼────────────────────────────────────┤
│                        │                                    │
│  평가 단위: 시간 구간  │  평가 단위: 개별 요청              │
│  (1분, 5분 등)         │  (각 사용자 트랜잭션)              │
│                        │                                    │
│  판단: 이 1분은        │  판단: 이 요청은                   │
│       Good/Bad?        │       Good/Bad?                    │
│                        │                                    │
│  예시:                 │  예시:                             │
│  p95 < 300ms?          │  이 요청 duration < 300ms?         │
│  → 전체 분이 Good/Bad  │  → 이 요청만 Good/Bad              │
│                        │                                    │
└────────────────────────┴────────────────────────────────────┘
```

#### Stringent SLO에서의 문제점

```
┌─────────────────────────────────────────────────────────────┐
│         99.99% SLO에서 Time-Series의 한계                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  SLO: 99.99% (4-9s)                                        │
│  월간 Error Budget: 4.32분                                  │
│                                                             │
│  Time-Series 시나리오 (1분 단위 평가):                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 분석 대기: 1분이 끝날 때까지 기다려야 함            │   │
│  │                                                     │   │
│  │ 94%만 300ms 미만인 경우:                            │   │
│  │ → 전체 1분이 "Bad"로 판정                          │   │
│  │ → 1분 = 4.32분 Budget의 23% 소진                    │   │
│  │                                                     │   │
│  │ 5분 윈도우 (AWS CloudWatch 기본):                   │   │
│  │ → 1번 실패 = SLO 위반!                             │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Event-Based 시나리오:                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 94%가 300ms 미만인 경우:                            │   │
│  │ → 6%만 "Bad"로 계산                                │   │
│  │ → 전체 분이 아닌 실패한 요청만 Budget에서 차감     │   │
│  │                                                     │   │
│  │ 더 정확하고, 더 많은 대응 시간 확보!               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 부분 장애(Brownout) 시나리오

```
┌─────────────────────────────────────────────────────────────┐
│         Blackout vs Brownout 비교                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  현대 분산 시스템에서:                                       │
│  • 100% Blackout (완전 장애): 드물다                        │
│  • Brownout (부분 장애): 일반적                             │
│                                                             │
│  수학적 등가:                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  99.99% SLO의 1% Brownout                          │   │
│  │         ≈                                          │   │
│  │  99% SLO의 100% Blackout                           │   │
│  │                                                     │   │
│  │  두 경우 모두: ~7시간+ 대응 시간 확보 가능          │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  → Event-Based 측정이 부분 장애를 더 정확히 반영           │
│  → 더 많은 대응 시간 = 더 달성 가능한 SLO 목표            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 💡 실무 적용 포인트

### 면접 질문 예상

#### Q1: Error Budget이 소진되면 어떻게 해야 하나요?

```
A: Error Budget 소진 시 대응 전략:

   즉시 조치:
   1. Feature Freeze - 새 기능 배포 중단
   2. 안정성 개선 작업에만 집중
   3. Production 변경 최소화
   4. 인시던트 원인 분석 및 복구

   예방적 접근:
   • Budget이 0이 되기 전에 예측 알림 설정
   • 30%, 20%, 10% 등 단계별 경고
   • Burn Rate 모니터링으로 조기 감지

   장기적 개선:
   • 반복되는 장애 패턴 분석
   • Architecture 개선
   • Chaos Engineering 도입 검토
   • SLO 목표치 재조정 (필요시)
```

#### Q2: Sliding Window와 Fixed Window 중 어떤 것을 선택해야 하나요?

```
A: Sliding Window를 권장합니다.

   Fixed Window의 문제점:
   • 월말에 Budget이 급격히 리셋됨
   • "31일 장애 → 1일 또 장애"도 기술적으로 SLO 충족
   • 고객 경험/기억과 불일치

   Sliding Window의 장점:
   • 점진적인 회복과 소진
   • 인간의 감정/기억 패턴과 일치
   • 더 현실적인 고객 만족도 측정

   실무 팁:
   • 30일 Sliding Window가 가장 실용적
   • 7-14일은 너무 짧음 (제품 사이클과 불일치)
   • 90일은 너무 김 (하루에 90% 소진해도 통과 가능)
```

#### Q3: Short-term과 Context-aware Burn Alert 중 어떤 것을 사용해야 하나요?

```
A: 상황에 따라 다르지만, 일반적으로 둘 다 사용을 권장합니다.

   Short-term Burn Alert:
   • 장점: 계산 비용 낮음, 빠른 감지
   • 단점: 과거 이력 무시, 트래픽 변동 미반영
   • 적합: 급격한 장애 즉시 감지

   Context-aware Burn Alert:
   • 장점: 높은 정확도, 남은 Budget 고려
   • 단점: 계산 비용 높음 (AWS Lambda 비용 폭탄 경험담)
   • 적합: 정밀한 예측, 스프린트 계획 반영

   권장 구성:
   ┌──────────────┬─────────────┬──────────────┐
   │ 긴급 알림    │ Short-term  │ 2-4시간 예측 │
   │ 계획 알림    │ Context-aw. │ 1-4일 예측   │
   └──────────────┴─────────────┴──────────────┘
```

#### Q4: Time-series와 Event 데이터 중 SLO에 어떤 것이 더 적합한가요?

```
A: Event 데이터가 더 적합합니다.

   Time-series의 한계:
   • Stringent SLO (99.99%+)에서 측정 어려움
   • 1분 평가 대기 동안 Budget 소진 가능
   • 부분 장애(Brownout) 과대 평가
   • "이 1분 전체가 Good/Bad" 이진 판단

   Event 데이터의 장점:
   • 개별 요청 단위로 Good/Bad 판단
   • 부분 장애를 정확히 반영
   • 더 많은 대응 시간 확보
   • 실제 고객 경험에 더 가까움

   예시 비교:
   94%가 정상인 1분 구간에서:
   • Time-series: 전체 1분 = Bad (100% 손실)
   • Event: 6%만 Bad (6% 손실)

   → 현대 분산 시스템에서는 Event 기반 SLO가 필수
```

## 📝 핵심 공식 정리

```
┌─────────────────────────────────────────────────────────────┐
│                    SLO 핵심 공식                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Error Budget = 총 요청 수 × (1 - SLO%)                    │
│                                                             │
│  남은 Budget = Error Budget - 실패한 요청 수               │
│                                                             │
│  Burn Rate = 실제 실패율 / 허용 실패율                     │
│                                                             │
│  Relative Burn = 관측된 실패 / (Budget × Baseline비율)     │
│                                                             │
│  Lookahead ≤ Baseline × 4 (계절성 미보정 시)               │
│                                                             │
│  조정된 Lookback = SLO 기간 - Lookahead                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
