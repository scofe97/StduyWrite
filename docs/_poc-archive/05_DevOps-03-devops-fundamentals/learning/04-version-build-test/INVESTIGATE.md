## Q2. Semantic Versioning 규칙과 Breaking Change 판단

**질문**: 아래 변경 사항들 각각에 SemVer 버전 번호를 어떻게 올려야 하는지 결정하고, 근거를 설명하라.

```
현재 버전: 2.4.1

a) GET /users/:id 응답에 createdAt 필드 추가
b) POST /users의 name 필드를 firstName + lastName으로 분리
c) JWT 만료 시간을 1시간 → 24시간으로 변경 (문서화된 동작이 아니었음)
d) 내부 캐싱 버그 수정 (외부 API 동작 변화 없음)
e) Node.js 최소 지원 버전을 18 → 20으로 상향
```

**탐구 포인트**:
- "하위 호환 불가"의 기준을 누가 정하는가? API 소비자의 기대가 문서화된 계약과 다를 때 어떻게 판단하는가?
- `0.y.z` 버전에서는 MINOR와 MAJOR의 경계가 달라지는가?
- 라이브러리와 REST API의 SemVer 적용 방식에 차이가 있는가?

## Q5. Mock vs Stub vs Fake 차이와 사용 시점

```
a) 이메일 서비스를 호출했는지 "검증"하고 싶다
b) 외부 결제 API가 항상 성공을 "반환"하도록 고정하고 싶다
c) 인메모리 HashMap으로 DB 저장소를 "대체"하고 싶다
```
