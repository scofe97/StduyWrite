## Q1. PaaS vs IaaS 선택 기준

**질문**: 팀이 새 웹 서비스를 출시하려 한다. CTO가 "PaaS 쓸지 EC2 직접 운영할지 결정해줘"라고 했을 때, 어떤 기준으로 판단하겠는가? 구체적인 질문 목록과 판단 근거를 설명하라.

**핵심 포인트**:
- 팀의 운영 역량(Linux 서버 운영 경험, On-call 인원)이 IaaS를 감당할 수 있는가를 먼저 확인한다
- 트래픽 규모와 비용 예측성 — 상시 트래픽이 있으면 EC2 고정 비용이 PaaS 종량제보다 저렴할 수 있다
- 특수 요구사항(특정 리눅스 커널, GPU, 커스텀 네트워크 구성)이 있으면 IaaS 외엔 선택지가 없다
- MVP·사이드 프로젝트는 PaaS로 시작해서 병목이 생기면 마이그레이션하는 전략이 합리적이다
- 선택 기준은 기술 선호가 아니라 팀 역량 × 요구사항 × 비용의 교집합이다

**심화 질문**: PaaS 위에서 시작했다가 IaaS로 마이그레이션해야 하는 시점은 언제인가? 마이그레이션 과정에서 가장 큰 리스크는 무엇인가?

## Q2. Reverse Proxy가 필요한 이유

**질문**: "앱이 이미 8080 포트에서 잘 동작하는데, 왜 nginx를 앞에 두는가? 그냥 EC2 Security Group에서 8080을 열면 안 되는가?"라는 질문에 답하라.

**핵심 포인트**:
- TLS 종료(termination)를 nginx에서 처리하면 앱 코드가 인증서를 몰라도 된다 — 앱은 평문 HTTP만 처리하면 충분
- 80/443 포트는 root 권한이 필요한데, 앱을 root로 실행하면 보안 위험이 커진다. nginx는 root로 시작했다가 곧바로 권한을 낮추는 방식을 사용한다
- 하나의 서버에서 여러 도메인(app1.example.com, app2.example.com)을 다른 포트의 앱으로 라우팅할 수 있다 (Virtual Host)
- 정적 파일(CSS, JS, 이미지)은 nginx가 직접 서빙하면 앱 프로세스를 거치지 않아 훨씬 빠르다
- Rate limiting, 요청 로깅, 버퍼링, 압축(gzip)을 앱 수정 없이 nginx 설정만으로 적용할 수 있다

**심화 질문**: nginx 대신 Caddy를 선택하는 이유는 무엇인가? 자동 HTTPS(Let's Encrypt 연동)를 nginx와 Caddy 각각에서 어떻게 설정하는지 비교하라.

## Q3. systemd의 역할과 프로세스 관리

**질문**: EC2에서 앱을 `nohup ./my-app &`으로 실행하는 방식과 systemd 서비스로 등록하는 방식의 차이를 설명하라. systemd를 사용하는 이유는 무엇인가?

**핵심 포인트**:
- `nohup &`은 터미널이 닫혀도 프로세스가 살아있지만, 서버 재부팅 시 자동으로 다시 시작되지 않는다
- systemd는 부팅 순서(After=network.target)를 명시할 수 있어 앱이 필요한 의존 서비스보다 먼저 뜨는 문제를 방지한다
- `Restart=always`로 크래시 시 자동 복구를 선언적으로 설정할 수 있다 — `nohup`은 별도 감시 스크립트가 필요하다
- `journalctl -u my-app -f`로 systemd가 관리하는 모든 로그를 중앙에서 조회한다. nohup은 stdout을 파일로 리다이렉트하는 임시방편이 필요하다
- `ProtectSystem=strict`, `NoNewPrivileges=yes` 같은 보안 지시자로 프로세스 권한을 샌드박스 수준으로 제한할 수 있다

**심화 질문**: `systemctl status` 출력에서 "active (running)"과 "active (exited)"의 차이는 무엇인가? Type=oneshot 서비스는 언제 사용하는가?

## Q4. Health Check 설계

**질문**: `/health` 엔드포인트를 설계할 때 고려해야 할 사항은 무엇인가? 단순히 200 OK만 반환하는 것과 DB 연결 상태까지 확인하는 것의 트레이드오프를 설명하라.

**핵심 포인트**:
- Liveness(앱 프로세스가 살아있는가)와 Readiness(트래픽을 받을 준비가 됐는가)를 분리하면 로드밸런서가 더 정확하게 판단할 수 있다
- DB 연결 확인을 포함하면 DB 장애 시 앱 인스턴스 전체가 health check에 실패해 불필요한 재시작 루프가 발생할 수 있다 — 외부 의존성 실패는 앱 자체의 실패가 아니다
- 헬스체크 엔드포인트 자체가 무거운 쿼리를 실행하면 높은 빈도(15초 간격)의 프로브가 DB 부하가 된다
- 응답에 버전 정보, 가동 시간(uptime), 의존성별 상태를 포함하면 디버깅에 유용하다
- 헬스체크 로그는 nginx access log에서 제외(`access_log off`)하지 않으면 로그 노이즈가 심해진다

**심화 질문**: Kubernetes의 livenessProbe와 readinessProbe, startupProbe는 각각 어떤 상황에서 사용하는가? startupProbe가 없을 때 느리게 시작하는 앱에 어떤 문제가 생기는가?

## Q5. 무중단 배포의 최소 조건

**질문**: EC2 한 대에서 앱을 새 버전으로 교체할 때 다운타임 없이 배포하려면 최소한 무엇이 필요한가? 단일 인스턴스에서 무중단이 가능한가?

**핵심 포인트**:
- 단일 인스턴스에서는 완전한 무중단이 불가능하다 — systemd reload는 수 밀리초의 gap이 생기며, 이 시간에 들어오는 요청은 실패한다
- 진정한 무중단 배포는 최소 2개 인스턴스가 필요하다 (로드밸런서 앞에 인스턴스 2개, 하나씩 교체)
- Rolling 배포 전략: 신 버전 인스턴스가 헬스체크를 통과한 후에 구 버전 인스턴스에서 트래픽을 빼는 순서가 중요하다
- `fly.toml`의 `strategy = "rolling"`은 이 과정을 플랫폼이 자동으로 처리한다
- 단일 인스턴스에서의 현실적 타협: 배포를 새벽 트래픽 최저 시간대에 진행하거나, nginx upstream을 소켓 파일로 교체하는 방식으로 gap을 최소화한다

**심화 질문**: Blue-Green 배포와 Canary 배포의 차이를 설명하고, 각각 언제 더 적합한지 트레이드오프를 비교하라.

## Q6. DevOps 진화 단계와 현재 조직 수준 평가

**질문**: DevOps 성숙도 모델에서 9단계(수동 FTP → AI-Ops)를 설명하고, 현재 자신이 속한 조직(또는 가장 최근에 경험한 환경)이 어느 단계에 있는지 근거를 들어 평가하라. 다음 단계로 가기 위한 병목은 무엇인가?

**핵심 포인트**:
- 단계 평가 기준: 배포 빈도(얼마나 자주 배포하는가), 배포 소요 시간, 장애 복구 시간(MTTR), 변경 실패율 — DORA 메트릭으로 객관화 가능
- CI(자동 테스트)가 없는 상태에서 CD를 도입하면 오히려 버그 배포 속도만 빨라진다 — 순서가 중요하다
- 컨테이너 도입 없이 Kubernetes를 도입하려는 시도는 복잡성만 추가되고 이점을 못 얻는 안티패턴이다
- 기술 성숙도보다 조직 문화(배포에 대한 두려움, 책임 분리 구조)가 실제 병목인 경우가 많다
- Platform Engineering 단계는 개발자가 인프라를 신경 쓰지 않고 배포할 수 있는 추상화 레이어를 제공하는 것이 목표다

**심화 질문**: DORA(DevOps Research and Assessment) 메트릭의 4가지 지표(배포 빈도, 리드타임, 변경 실패율, MTTR)를 설명하고, Elite 등급과 Low 등급의 수치 차이를 비교하라.

## Q7. 배포 실패 시나리오 대응

**질문**: 프로덕션 배포 직후 에러율이 갑자기 5%에서 30%로 치솟았다. 이 상황에서 첫 5분, 다음 15분 동안 어떻게 행동하겠는가? 롤백 결정 기준은 무엇인가?

**핵심 포인트**:
- 첫 5분은 원인 분석보다 서비스 복구가 우선이다 — 롤백이 가능한 상황이라면 먼저 실행하고 원인은 나중에 분석한다
- 롤백 결정 기준: 에러율 임계치(예: 1% 초과), 핵심 기능 비정상, 데이터 정합성 문제 중 하나라도 해당하면 즉시 롤백
- 롤백과 동시에 팀에 상황을 공유한다 — 혼자 조용히 해결하려다 골든타임을 놓치는 것이 가장 흔한 실수다
- 에러 로그에서 공통 패턴 파악: 특정 엔드포인트만 실패하는가, 전체가 실패하는가, DB 타임아웃인가 코드 예외인가
- 배포 후 15분은 반드시 모니터링 앞에 대기한다 — "배포 완료"를 선언하고 자리를 뜨는 것은 위험한 습관이다

**심화 질문**: 롤백이 불가능한 상황(DB 스키마 마이그레이션이 이미 실행된 경우)에서 배포 실패가 발생하면 어떻게 대응하는가? 이를 방지하기 위한 DB 마이그레이션 전략을 설명하라.
