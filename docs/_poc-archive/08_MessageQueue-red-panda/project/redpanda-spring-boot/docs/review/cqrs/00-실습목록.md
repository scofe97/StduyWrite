## 각 실습의 구현 범위

### 1. 이벤트 설계 & Avro 스키마
- 도메인 이벤트 정의: PostCreated, PostLiked, UserFollowed, UserUnfollowed
- 이벤트 설계 4원칙 적용: 과거 시제, 충분한 정보, 멱등성 고려, 도메인 언어
- Avro .avsc 스키마 파일 작성 (Schema Registry 등록)
- 기존 ch03 Avro 패턴 재활용

### 2. Command Side 구현
- CommandHandler 클래스 (CreatePostCommand, FollowUserCommand 등)
- CommandHandler → KafkaTemplate.send()로 이벤트 발행
- REST API: POST 요청 → 202 Accepted (비동기 처리)
- Command 모델과 Query 모델 분리 확인

### 3. Event Store 토픽 설계
- 토픽 생성: social.events.posts, social.events.follows (실습 6에서 1:1 분리)
- Partition key = aggregateId (같은 Aggregate의 이벤트 순서 보장)
- retention.ms=-1 (무한 보존, Event Store 역할)
- Kafka/Redpanda가 Event Store로 적합한 이유 6가지 확인

### 4. Kafka Streams Topology
- StreamsBuilder로 토폴로지 정의
- KTable 구성: posts-store, follows-store, timeline-store
- State Store 설정 (RocksDB 기반, Changelog 토픽 자동 생성)
- 이벤트 → State Store 물화 로직

### 5. Interactive Query API
- ReadOnlyKeyValueStore 접근
- PostQueryService: State Store에서 직접 조회
- PostQueryController: REST GET /posts/{id}, /users/{id}/posts
- CRUD JPA 조회와의 차이점 확인

### 6. 토픽 1:1 분리 + Serde 팩토리 + 토픽 상수 관리
- 1토픽N이벤트 → 1토픽1이벤트 분리 (2개→4개 토픽)
- Query Side에서 merge()로 분리된 토픽 재결합
- CqrsSerdeFactory: 두 토폴로지의 Avro Serde 생성 중복 제거
- CqrsTopics: 토픽명 상수 중앙 관리 (Config/Handler/Topology 3곳 중복 해소)
- CqrsTopicConfig: eventStoreTopic() 헬퍼로 공통 설정 추출

### 7. 빌더 패턴 전환 + HTTP 폴더 정리
- PostView, FollowsView에 @Builder + @Builder.Default 추가
- 토폴로지의 newPostView(), newFollowsView()를 빌더로 전환
- mutate 코드(likePost, follow/unfollow)는 setter 유지
- HTTP 파일을 http/cqrs/ 폴더로 이동, Command 중복 제거
- 02-command-side.http를 canonical로, 03~05는 검증/쿼리만

### 8. Timeline Materialized View
- follows-store에서 팔로잉 목록 조회
- posts KStream과 follows-store join
- timeline-store에 사용자별 타임라인 자동 구성
- 새 팔로우/언팔로우 시 타임라인 실시간 반영

### 9. Event Replay & Rebuild
- 새 Consumer Group 생성 (auto.offset.reset=earliest)
- 전체 이벤트 리플레이로 Materialized View 재구축
- EventReplayService 구현
- 새 분석용 View 추가 시나리오 (예: 인기 게시물 랭킹)

### 10. CQRS vs CRUD 성능 비교
- 같은 쿼리 시나리오 정의 (타임라인 조회, 게시물 상세)
- CRUD 방식: ch02 JPA Repository 조회 (11번 DB 쿼리, 수십~수백 ms)
- CQRS 방식: State Store 직접 조회 (1번 조회, 1~5 ms)
- 쓰기 성능: DB 저장 대기(10~50ms) vs 이벤트 발행(1~2ms)
- 종합 비교 테이블 작성 (11개 항목)
