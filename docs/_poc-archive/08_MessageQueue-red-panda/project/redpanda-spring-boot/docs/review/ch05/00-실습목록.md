## 각 실습의 구현 범위

### 1. 에러 분류 + 예외 필터링
- 커스텀 예외 클래스: `TransientException`, `NonTransientException` (또는 기존 예외 활용)
- `ch06/config/DlqKafkaConfig` — addNotRetryableExceptions로 Non-Transient 즉시 DLT
- addRetryableExceptions 방식과의 차이 이해 (화이트리스트 vs 블랙리스트)
- **DeserializationException 즉시 DLT 등록** (Confluent 사례: poison pill 파티션 블로킹 방지)
- 테스트: Transient → 재시도 후 DLT, Non-Transient → 즉시 DLT

### 2. DefaultErrorHandler 심화
- `ExponentialBackOffWithMaxRetries` 적용 (1s → 2s → 4s → 8s, max 10s)
- ContainerFactory별 다른 핸들러: "strict" factory (1회 재시도 500ms) vs "lenient" factory
- Bean 자동 적용 원리 확인 (KafkaAutoConfiguration 동작)
- **[Bean 충돌 방지]** Ch03 `KafkaErrorConfig`가 전역 DefaultErrorHandler Bean 등록 중 → ch06은 `@Qualifier("ch06ErrorHandler")` + ch06 전용 ContainerFactory에만 수동 설정 (Ch09 이슈 재발 방지)

### 3. 커스텀 DLT 메타데이터
- `DeadLetterPublishingRecoverer` 서브클래스 — `createProducerRecord` 오버라이드
- 헤더: X-Original-Topic, X-Original-Partition, X-Original-Offset, X-Original-Timestamp
- 헤더: X-Error-Message, X-Error-Timestamp, X-Retry-Count
- DLT Consumer에서 헤더 파싱 검증

### 4. ErrorHandlingDeserializer
- Avro 환경 적응: `ErrorHandlingDeserializer`로 `KafkaAvroDeserializer` 래핑
- `springDeserializerExceptionValue` 헤더로 원본 바이트 접근
- 잘못된 Avro 바이트 전송 시 Consumer 크래시 없이 DLT 라우팅 검증
- 주의: Ch03/Ch09 기존 ConsumerFactory와의 공존 (ch06 전용 팩토리)

### 5. @RetryableTopic 심화
- 토픽 전략 3가지 구현 비교:
  - `SUFFIX_WITH_INDEX_VALUE` (Ch03 C6에서 사용)
  - `SUFFIX_WITH_DELAY_VALUE` (지연 시간 suffix)
  - `SINGLE_TOPIC` (단일 retry 토픽)
- `include`/`exclude` 예외 필터로 선별적 재시도
- 프로덕션 `NewTopic` Bean으로 retry/DLT 토픽 명시적 생성
- `rpk topic list`로 실제 생성된 토픽 확인
- **[@DltHandler 폴백 테스트]** @DltHandler에서 의도적 예외 발생 → DefaultErrorHandler 폴백 동작 검증 (학습 문서 line 405~413)
- **[수동 ACK 호환성]** @RetryableTopic + ack-mode: manual 조합 시 retry 리스너 ACK 모드 충돌 여부 확인 (학습 문서 line 445)
- **[리서치]** @RetryableTopic + KafkaTransactionManager 비호환: Spring Kafka GitHub #2934 공식 제한사항 — 트랜잭션 환경에서는 DefaultErrorHandler 사용 필수 (Ch09가 이미 올바른 선택)

### 6. DLT 모니터링 + 재처리 API
- `DltMessage` JPA 엔티티 (id, originalTopic, key, value, errorMessage, status, receivedAt)
- `DltStatus` enum (PENDING, REPROCESSED, REPROCESS_FAILED, DISCARDED)
- `DltMonitoringService` — @KafkaListener로 DLT 수신 → DB 저장
- `DltController` REST API:
  - GET `/api/dlt` — 목록 조회 (토픽/상태 필터, 페이징)
  - POST `/api/dlt/{id}/reprocess` — 원본 토픽으로 재전송
  - POST `/api/dlt/reprocess-batch` — 기간별 일괄 재처리
  - POST `/api/dlt/{id}/discard` — 사유 기록 후 폐기
- **[리서치]** DLT 보존 정책: `rpk topic create --config retention.ms=604800000` (7일 보존)
- **[리서치]** Multi-level DLT: @DltHandler에서 복구 불가 판정 시 `.DLT.POISON` 토픽으로 2차 격리 (IBM EDA 패턴)
- **[리서치]** 재처리 멱등성: Ch08 `ProcessedEvent` 패턴 재활용 — correlationId 기반 중복 replay 방지

### 7. 트랜잭션 환경의 DLQ
- Ch09 `TripKafkaConfig` 확장: Kafka TX 안에서 DLQ 전송이 원자적인지 검증
- `isolation.level=read_committed` Consumer가 abort된 DLQ 메시지를 읽지 않는지 확인
- 트랜잭션 없음 vs 있음 비교 시나리오 (메시지 유실 vs 원자적 처리)

### 8. 통합 테스트 + 비교 리뷰
- 로컬 인프라 (docker-compose: Redpanda + PostgreSQL)
- 시나리오:
  - Transient 에러 → 재시도 후 성공 (또는 DLT)
  - Non-Transient 에러 → 즉시 DLT
  - 역직렬화 에러 → ErrorHandlingDeserializer → DLT
  - DLT 재처리 API → 원본 토픽 재전송 → 정상 처리
  - @DltHandler 실패 → DefaultErrorHandler 폴백 경로
  - DLT 재처리 시 멱등성 보장 (correlationId 중복 필터)
- 비교 리뷰 문서: DefaultErrorHandler vs @RetryableTopic 케이스별 선택 기준
  - Ch03(기본) → Ch05(심화) 경험 대비
  - 학습 문서 line 376~403 표 기반 실전 판단 기준
  - **[리서치]** 의사결정 트리: 트랜잭션 사용? → 순서 중요? → 처리량 우선? 기반 선택
