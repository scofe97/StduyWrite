## 각 실습의 구현 범위

### 1. Linear Chain 구현
- 3단계 파이프라인: raw-orders → validated-orders → enriched-orders → fulfilled-orders
- 각 단계를 @KafkaListener + @SendTo로 연결
- 단계별 변환 로직 (검증: 필수 필드 체크, 보강: 고객 정보 조회, 이행: 재고 차감)
- NewTopic 빈 3개 선언

### 2. Fan-Out 구현
- 1:N 브로드캐스트: 하나의 이벤트 → 여러 토픽으로 동시 발행
- KafkaTemplate.executeInTransaction()으로 원자적 다중 전송
- 각 Consumer가 독립적으로 처리 (알림, 분석, 감사 로그 등)

### 3. Fan-In 구현
- N:1 집계: 여러 토픽의 결과를 하나로 합류
- correlation-id 기반 그룹핑 (같은 주문의 결과를 묶기)
- 상태 저장소: ConcurrentHashMap(단일 인스턴스) 또는 Redis(다중 인스턴스)
- 타임아웃: 일정 시간 내 모든 결과 미도착 시 불완전 집계로 처리

### 4. Filter-Branch 구현
- 조건부 라우팅: 이벤트 속성에 따라 다른 토픽으로 분기
- Strategy 패턴: RoutingStrategy 인터페이스 + 구현체로 라우팅 규칙 분리
- @KafkaListener + KafkaTemplate.send() 조합
- KStream.split() 대안 비교

### 5. Diamond 구현
- Fan-Out(분기) + Fan-In(합류) 결합
- correlation-id 헤더 전파로 분산된 결과 추적
- 합류 시점 판단: 모든 브랜치 완료 vs 타임아웃
- 부분 완료 처리 전략 (fallback 값, 재시도)

### 6. ReplyingKafkaTemplate
- ReplyingKafkaTemplate 빈 설정 (reply 토픽, 타임아웃)
- sendAndReceive()로 동기식 요청-응답
- REPLY_PARTITION 헤더: 다중 인스턴스 환경에서 응답이 올바른 인스턴스로 전달
- @SendTo vs ReplyingKafkaTemplate 사용 시점 비교

### 7. 파이프라인 관측성
- 커스텀 헤더 전파: X-Trace-Id, X-Pipeline-Stage, X-Created-At
- ProducerInterceptor로 헤더 자동 주입
- MDC 연동: Consumer에서 헤더 → MDC 매핑 (로그 추적)
- TTL 헤더로 순환 방지 (stage 카운터 초과 시 DLQ)
- @RetryableTopic + DLQ 통합 모니터링
