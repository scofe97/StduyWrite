## 진행 순서 및 이유

| 순서 | 실습 | 이유 |
|------|------|------|
| 1 | 비동기 커맨드 발행 | 요청 발행부터 시작해야 전체 비동기 흐름의 시작점을 이해할 수 있음 |
| 2 | 완료 이벤트 리스너 | 발행한 커맨드의 결과를 수신하는 쌍을 구현 — 요청-응답 흐름 완성 |

| 4 | 타임아웃 처리 | 완료 이벤트가 오지 않는 경우의 안전장치 — 운영 환경 필수 요소 |
| 5 | 통합 테스트 | 1~4번 구현을 시나리오로 검증, Thread.sleep 방식과의 정확성 차이 체감 |

## 각 실습의 구현 범위

### 1. 비동기 커맨드 발행
- `AsyncCommand.java`: correlationId, commandType, payload 필드를 가진 커맨드 모델
- `AsyncCommandPublisher.java`: `KafkaTemplate.send("async.command", correlationId, command)` 구현
- `Ch08TopicConfig.java`: `async.command`, `async.completion` 토픽 NewTopic 빈 설정
- `src/main/avro/ch08/AsyncCommand.avsc`: Avro 스키마 (correlationId, commandType, payload, requestedAt)

### 2. 완료 이벤트 리스너
- `CompletionListener.java`: `@KafkaListener(topics="async.completion", groupId="completion-group")` 구현
- `AsyncCompletion.java`: correlationId, status(SUCCESS/FAILURE/TIMEOUT), result, completedAt 모델
- `src/main/avro/ch08/AsyncCompletion.avsc`: Avro 스키마 (correlationId, status enum, result, completedAt)
- 수신한 완료 이벤트를 CorrelationManager에 전달하여 대기 중인 요청 해소

### 3. Correlation 관리
- `CorrelationManager.java`: `ConcurrentHashMap<String, CompletableFuture<AsyncCompletion>>` 기반 매핑
- `PendingRequestStore.java`: 진행 중 요청 목록 관리, TTL 기반 만료 처리
- `AsyncCommandPublisher`에서 발행 전 correlationId로 CompletableFuture 등록
- `CompletionListener`에서 수신 후 correlationId로 CompletableFuture 완료

### 4. 타임아웃 처리
- `TimeoutHandler.java`: `@Scheduled`로 PendingRequestStore를 주기적으로 스캔
- 설정된 TTL(기본 10분) 초과 요청 감지 후 TIMEOUT 상태로 CompletableFuture 완료
- 타임아웃 발생 시 `async.completion` 토픽에 TIMEOUT 이벤트 발행 (감사 로그 유지)
- 타임아웃 후 PendingRequestStore에서 해당 요청 제거 (메모리 누수 방지)

### 5. 통합 테스트
- `Ch08AsyncEventTest.java`: Spring Kafka Test 기반 통합 테스트
- 시나리오 1: 빠른 완료(3초) — Thread.sleep 대비 정확한 시점에 SUCCESS 수신 확인
- 시나리오 2: 느린 완료(30초) — hardcoded 10초 success 대신 실제 완료 시점 반영 확인
- 시나리오 3: 타임아웃 — TimeoutHandler가 TIMEOUT 이벤트를 발행하고 리소스 정리 확인
