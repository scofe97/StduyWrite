## 왜 이렇게 구현했는가

### NewTopic 빈 등록 → KafkaAdmin이 시작 시 자동 생성

```java
@Configuration
public class KafkaTopicConfig {

    @Bean
    public NewTopic ordersTopic() {
        return TopicBuilder.name("orders")
                .partitions(3)
                .replicas(1)
                .build();
    }
}
```

Spring Boot Auto-Configuration이 `KafkaAdmin` 빈을 자동으로 등록한다. `KafkaAdmin`은 컨텍스트 내의 **모든 `NewTopic` 빈을 감지**하고, 애플리케이션 시작 시 브로커에 해당 토픽이 없으면 생성한다.

```
KafkaTopicConfig → NewTopic Bean 등록
      ↓
KafkaAdmin(Auto-Config) → 시작 시 NewTopic 목록 조회
      ↓
브로커에 AdminClient.createTopics() 호출
      ↓
토픽 없으면 생성 / 이미 있으면 무시
```

별도의 초기화 스크립트나 CLI 명령 없이 애플리케이션이 스스로 필요한 인프라를 구성한다는 점에서 **Infrastructure as Code** 패턴의 경량 구현이다.

### 이미 존재하는 토픽은 건드리지 않음

`KafkaAdmin`은 토픽이 이미 존재하면 생성 요청을 건너뛴다. 이 동작의 함의:

| 시나리오 | 결과 |
|---------|------|
| 토픽 미존재 | 생성 (partitions, replicas 적용) |
| 토픽 존재, 설정 동일 | 아무것도 하지 않음 |
| 토픽 존재, **파티션 수 증가** | 증가 적용 가능 (KafkaAdmin 설정에 따라) |
| 토픽 존재, **파티션 수 감소** | **불가** (브로커가 거부) |
| 토픽 삭제 후 재생성 | 불가 (KafkaAdmin은 삭제 기능 없음) |

파티션 감소는 이미 저장된 메시지의 위치가 바뀌어 Consumer의 offset이 무효화되므로 Kafka 브로커 자체가 허용하지 않는다. 이것은 KafkaAdmin의 제약이 아니라 Kafka 설계 원칙이다.

### Infrastructure as Code 관점

토픽 설정을 Java 코드로 관리하면 세 가지 이점이 생긴다:

1. **버전 관리**: 토픽의 파티션 수, 복제 인수가 Git 이력에 남는다. "누가, 언제, 왜 바꿨는지" 추적 가능.
2. **코드 리뷰**: 토픽 변경이 PR에 포함되어 팀 리뷰를 거친다.
3. **배포 자동화**: 애플리케이션 배포만으로 토픽 준비까지 완료된다. 별도 인프라 스크립트 실행 단계가 불필요.

단점: 토픽 설정이 애플리케이션 코드에 묶여 있어 운영팀이 독립적으로 변경하기 어렵다.

### Testcontainers에서도 동일하게 동작

```java
// AbstractKafkaTest의 @DynamicPropertySource
@DynamicPropertySource
static void kafkaProperties(DynamicPropertyRegistry registry) {
    registry.add("spring.kafka.bootstrap-servers", kafka::getBootstrapServers);
}
```

Testcontainers가 실행한 빈 Redpanda 브로커에 `KafkaAdmin`이 연결되어 `NewTopic` 빈들을 그대로 생성한다. 테스트 환경과 프로덕션 환경에서 **동일한 코드 경로**로 토픽이 생성된다는 점에서 테스트 신뢰도가 높다.

수동으로 `kafka-topics.sh --create`를 실행하거나 docker-compose에 init 스크립트를 추가할 필요 없다.

### 프로덕션에서 Terraform/별도 AdminClient로 관리하는 경우

`NewTopic` 빈 방식의 한계로 인해 프로덕션에서는 별도 도구를 선호하는 경우가 많다:

| 도구 | 장점 | 적합한 환경 |
|------|------|-----------|
| **NewTopic 빈** (현재) | 간단, 코드 내장, 자동 실행 | 개발/테스트, 소규모 |
| Terraform (Kafka 프로바이더) | 선언적, plan/apply 워크플로우, 드리프트 감지 | 프로덕션 |
| 별도 AdminClient 애플리케이션 | 세밀한 제어 (retention, compaction 등) | 복잡한 토픽 정책 |
| Confluent Control Center / Redpanda Console | GUI 기반 관리 | 운영팀 자율 관리 |

`NewTopic` 빈은 partitions와 replicas 외에 `configs()` 메서드로 추가 설정(retention.ms, cleanup.policy 등)도 지정할 수 있지만, 기존 토픽의 설정 변경은 지원하지 않으므로 운영 복잡도가 높아지면 한계가 드러난다.

## 토픽 유형별 설정 차이

`KafkaTopicConfig`의 토픽들은 모두 기본 설정(delete policy, 기본 retention)을 사용하지만, 프로덕션에서는 토픽 유형에 따라 설정이 달라진다:

| 유형 | Cleanup Policy | Retention | 핵심 설정 | 이 프로젝트 예시 |
|------|:-------------:|:---------:|----------|---------------|
| **Entity Topic** | `compact` | 무기한 | `min.cleanable.dirty.ratio`, tombstone 보존 | (해당 없음) |
| **Event Topic** | `delete` | 시간/크기 기반 | `retention.ms`, `compression.type` | `chapter2.orders`, forward/failure 토픽 |
| **Command Topic** | `delete` | 짧게 (1일) | 단일 컨슈머 전제 | `chapter2.sendto-input` |
| **DLT** | `delete` | 짧게 | 파티션 1, 분석 후 삭제 | `chapter2.error-test-dlt` |

**Entity Topic**은 `compact` 정책으로 같은 키의 최신 메시지만 유지한다. DB 테이블의 행과 대응되며, tombstone(null payload)으로 삭제를 표현한다. 이 프로젝트에는 Entity Topic이 없지만, 프로덕션에서 고객 상태(`order.customers.state`) 같은 토픽에 사용한다.

**Event Topic**은 발생한 사실을 불변 레코드로 기록한다. `chapter2.orders`가 이에 해당한다. 이벤트는 나중에 필드를 추가할 수 없으므로, 발행 시점에 필요한 정보를 최대한 포함해야 한다.

**Command Topic**은 수행 요청이다. `chapter2.sendto-input`이 이에 해당하며, 리스너가 입력을 받아 처리 후 `chapter2.sendto-output`(Event)으로 결과를 발행한다. Kafka는 pub-sub 모델이므로, 커맨드가 주된 통신 수단이라면 Kafka 외의 선택지(REST, gRPC)도 고려해야 한다.

### 학습용 vs 프로덕션 네이밍

| 관점 | 학습용 (현재) | 프로덕션 |
|------|-------------|---------|
| 패턴 | `chapter{n}.{purpose}` | `{team}.{domain}.{entity}` |
| 예시 | `chapter2.orders` | `order-team.commerce.orders` |
| 목적 | 챕터별 격리 | 소유팀/도메인 식별 |

학습용 네이밍은 챕터 간 토픽 충돌을 방지하기 위한 것이다. 프로덕션에서는 팀과 도메인을 식별할 수 있는 네이밍 컨벤션을 사용하고, GitOps 도구(Jikkou 등)로 네이밍 규칙을 자동 검증한다.

> 상세: learning `02-fundamentals/11-topic-design.md` §7 (토픽 유형 심화), §2 (네이밍 컨벤션)

## 핵심 학습 포인트

1. **NewTopic 빈 → KafkaAdmin 자동 감지** — 시작 시 토픽 자동 생성, 별도 스크립트 불필요
2. **존재하면 무시** — 멱등성 있는 초기화, 재시작해도 기존 토픽 보존
3. **파티션 감소 불가** — Kafka 설계 원칙, KafkaAdmin 제약이 아님
4. **Testcontainers 호환** — 빈 브로커에도 동일하게 동작, 테스트 환경 설정 불필요
5. **IaC 경량 구현** — Git 이력 + 코드 리뷰로 토픽 변경 추적
6. **프로덕션 한계** — 복잡한 정책이나 기존 토픽 수정은 Terraform/AdminClient로 보완 필요
7. **토픽 유형 인식** — Entity/Event/Command/DLT 유형에 따라 cleanup policy와 retention이 달라짐
