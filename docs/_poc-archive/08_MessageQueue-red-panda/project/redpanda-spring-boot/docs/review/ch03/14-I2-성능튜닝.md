## 왜 이렇게 구현했는가

### 성능 튜닝의 본질: 트레이드오프

성능 튜닝은 "빠르게"가 아니라 **처리량(throughput), 지연 시간(latency), 리소스 사용량** 간의 균형을 조정하는 작업이다. 하나를 올리면 다른 하나가 내려간다.

### Producer 튜닝 설정

#### batch-size + linger.ms: 처리량 vs 지연 시간

Producer는 메시지를 즉시 보내지 않고 **배치로 모아서** 전송한다. 이 두 설정이 배치를 제어한다.

| 설정 | 역할 | 기본값 |
|------|------|--------|
| `batch-size` | 배치 최대 크기 (바이트) | 16384 (16KB) |
| `linger.ms` | 배치가 안 찼어도 이 시간 후 전송 | 0 (즉시) |

**동작**: `batch-size`가 차거나 `linger.ms`가 경과하면 전송한다. 둘 중 먼저 발생하는 조건이 트리거.

```
linger.ms=0:   메시지 → [즉시 전송] → 지연 최소, 네트워크 왕복 많음
linger.ms=10:  메시지 → [10ms 모음] → 배치 전송 → 왕복 줄어듦, 10ms 지연 추가
linger.ms=100: 메시지 → [100ms 모음] → 큰 배치 → 처리량 최대, 100ms+ 지연
```

| 시나리오 | batch-size | linger.ms | 이유 |
|---------|-----------|----------|------|
| 실시간 주문 | 1KB~16KB | 0~5 | 지연 최소화 |
| 로그 집계 | 32KB~64KB | 50~100 | 처리량 우선 |
| IoT 센서 데이터 | 64KB+ | 100~500 | 네트워크 효율 |

#### compression-type: CPU vs 네트워크

| 알고리즘 | 압축 속도 | 압축률 | CPU 부담 | 적합한 경우 |
|---------|----------|--------|---------|------------|
| `lz4` | **가장 빠름** | 보통 | 낮음 | 낮은 지연 + 적당한 압축 |
| `snappy` | 빠름 | 보통 | 낮음 | 범용 |
| `gzip` | 느림 | **가장 높음** | 높음 | 네트워크 대역폭이 병목 |
| `zstd` | 보통 | 높음 | 보통 | 압축률과 속도 균형 |

텍스트 데이터(JSON, 로그)는 70~80% 압축률이 나온다. 이미 압축된 데이터(이미지, 바이너리)는 효과가 적다.

#### buffer-memory: 전송 대기 버퍼

Producer가 브로커로 보내기 전에 메시지를 저장하는 메모리. 기본값 32MB. 버퍼가 가득 차면 `send()`가 블록된다(`max.block.ms` 동안). 처리량이 높으면 64MB 이상으로 늘린다.

### Consumer 튜닝 설정

#### max-poll-records: poll() 크기 vs rebalancing 위험

| 값 | 처리량 | 위험 |
|----|--------|------|
| 작게 (100~500) | 낮음 | 안전 — 처리 시간이 `max.poll.interval.ms`를 초과할 가능성 낮음 |
| 크게 (1000~5000) | 높음 | 위험 — 처리 시간 초과 시 rebalancing 발생 |

**안전 마진 계산**: 메시지당 처리 10ms, `max.poll.interval.ms=300000`(5분) → 최대 30000건 가능하지만, 안전 마진을 두어 500~1000으로 설정.

#### fetch-min-size + fetch-max-wait

브로커가 데이터를 반환하는 조건. `fetch-min-size` 바이트가 쌓이거나 `fetch-max-wait` 시간이 지나면 반환한다.

| 시나리오 | fetch-min-size | fetch-max-wait | 이유 |
|---------|---------------|----------------|------|
| 실시간 처리 | 1 | 100 | 즉시 가져옴 |
| 배치 효율 | 1024 | 500~1000 | 충분히 모아서 가져옴 |

#### concurrency: Consumer 스레드 수

`concurrency`는 한 애플리케이션 내에서 동시에 실행되는 Consumer 스레드 수다. **파티션 수 이하**로 설정해야 모든 스레드가 일을 한다.

```
파티션 3개, concurrency=3: 각 스레드가 1개 파티션 담당 (최적)
파티션 3개, concurrency=5: 2개 스레드가 idle (낭비)
파티션 3개, concurrency=1: 1개 스레드가 3개 파티션 모두 처리 (병목)
```

### Producer 배치와 Consumer 배치는 독립이다

혼동하기 쉬운 포인트: "Producer가 배치로 보내면 Consumer도 배치로 받는 것 아닌가?"

**아니다. 둘은 완전히 독립적이다.**

| | Producer 배치 | Consumer 배치 |
|---|---|---|
| 무엇? | **네트워크 전송** 최적화 | **@KafkaListener** 수신 방식 |
| 설정 | `batch-size`, `linger.ms` | `factory.setBatchListener(true)` |
| 기본 동작 | **항상 배치** (크기만 조절) | **단건** (OrderEvent 1건씩) |
| 관계 | 서로 독립 | 서로 독립 |

Producer는 여러 메시지를 묶어서 브로커로 전송하지만, 브로커는 이를 풀어서 **개별 메시지(offset)로 저장**한다. Consumer는 poll()로 가져올 때 자기 설정에 따라 단건 또는 List로 받는다.

```
Producer: 1000건을 배치로 전송 → 브로커에 1000건 개별 저장(offset 1000개)

Consumer A (단건):  poll() → 500건 → @KafkaListener(OrderEvent event) 를 500번 호출
Consumer B (배치):  poll() → 500건 → @KafkaListener(List<OrderEvent> events) 를 1번 호출
```

같은 토픽, 같은 메시지인데 Consumer 설정에 따라 수신 방식이 다르다. Producer가 배치로 보냈든 단건으로 보냈든 Consumer에게는 영향이 없다.

**linger.ms=0이면 단건 전송인가?** 아니다. `linger.ms=0`은 "기다리지 않겠다"이지 "1건씩 보내겠다"가 아니다. 비동기 send()가 네트워크 I/O보다 빨리 호출되면 버퍼에 자연스럽게 쌓여서 배치가 된다. 진짜 단건 전송은 `send().get()`(동기)으로만 가능하다.

### 적용한 설정 요약

```yaml
# Producer
batch-size: 16384        # 16KB — 기본값 유지 (학습 환경)
compression-type: lz4    # 빠른 압축
buffer-memory: 33554432  # 32MB — 기본값 유지
linger.ms: 10            # 10ms 모아서 전송

# Consumer
max-poll-records: 500    # 안전한 기본값
fetch-min-size: 1        # 즉시 가져옴
fetch-max-wait: 500      # 500ms 대기
concurrency: 3           # chapter2.orders 파티션 3개에 맞춤
```

## 핵심 학습 포인트

1. **batch-size + linger.ms** — 처리량 vs 지연 시간의 핵심 트레이드오프
2. **compression-type** — CPU vs 네트워크, `lz4`가 범용적으로 가장 적합
3. **max-poll-records** — 크게 잡으면 처리량 증가하지만 rebalancing 위험 증가
4. **concurrency ≤ 파티션 수** — 초과하면 idle 스레드 발생, 낭비
5. **튜닝 = 트레이드오프** — "무조건 빠르게"가 아니라 상황에 맞는 균형점 찾기
6. **설정 후 측정** — 설정만 바꾸고 끝내지 않고, 실제 throughput/latency를 측정해서 검증
