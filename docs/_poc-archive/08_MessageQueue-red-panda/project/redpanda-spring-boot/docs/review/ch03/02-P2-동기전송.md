# P2: 동기 전송 (Synchronous Send)

## 목적

기존 `sendOrder()`는 비동기 전송이다. `CompletableFuture`를 반환하고, 호출자는 원하면 나중에 결과를 확인한다. 동기 전송은 `.get()`으로 브로커 ACK를 기다린 후에야 다음 코드로 진행한다.

```
비동기: send() -> 즉시 반환 -> 후속 로직 실행 -> (나중에) 콜백에서 결과 확인
동기:   send().get() -> 브로커 ACK 대기 -> 결과 확인 -> 후속 로직 실행
```

---

## 변경 파일

### OrderProducer.java - `sendOrderSync()` 추가

```java
private static final long SYNC_TIMEOUT_SECONDS = 10;

public SendResult<String, OrderEvent> sendOrderSync(OrderEvent event) {
    try {
        SendResult<String, OrderEvent> result = kafkaTemplate
                .send("chapter2.orders", event.getOrderId(), event)
                .get(SYNC_TIMEOUT_SECONDS, TimeUnit.SECONDS);

        log.info("Sync sent: topic={}, partition={}, offset={}",
                result.getRecordMetadata().topic(),
                result.getRecordMetadata().partition(),
                result.getRecordMetadata().offset());

        return result;
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        throw new RuntimeException("Interrupted while sending message: " + event.getOrderId(), e);
    } catch (ExecutionException e) {
        throw new RuntimeException("Failed to send message: " + event.getOrderId(), e.getCause());
    } catch (TimeoutException e) {
        throw new RuntimeException("Timeout sending message: " + event.getOrderId(), e);
    }
}
```

## 설계 결정과 이유

### SendResult을 직접 반환하는 이유

비동기 `sendOrder()`는 `CompletableFuture<SendResult>`을 반환한다. 동기 `sendOrderSync()`는 `.get()`으로 기다린 후 `SendResult` 자체를 반환한다. 호출자가 Future를 다룰 필요 없이 즉시 결과를 사용할 수 있다.

### InterruptedException을 별도로 catch하는 이유

Codex 피드백(#1)에서 지적된 패턴이다. `InterruptedException`은 스레드 인터럽트 신호이므로, `Thread.currentThread().interrupt()`로 인터럽트 상태를 복구해야 한다. 이를 하지 않으면 상위 코드가 인터럽트 발생을 감지할 수 없다.

```java
// BAD: 인터럽트 상태가 사라짐
catch (ExecutionException | InterruptedException | TimeoutException e) {
    throw new RuntimeException(e);
}

// GOOD: 인터럽트 상태 복구 후 예외 전파
catch (InterruptedException e) {
    Thread.currentThread().interrupt();  // 복구
    throw new RuntimeException(e);
}
```

### ExecutionException에서 e.getCause()를 전파하는 이유

`ExecutionException`은 래핑 예외다. 실제 원인은 `getCause()`에 있다. `e` 자체를 전파하면 로그에 "ExecutionException" 한 겹이 더 쌓여서 디버깅이 어려워진다.

### 타임아웃을 상수로 뺀 이유

`SYNC_TIMEOUT_SECONDS = 10`으로 상수화했다. 매직 넘버를 코드에 직접 쓰면:
- 여러 곳에서 동기 전송을 쓸 때 타임아웃이 불일치할 수 있다
- 값을 변경할 때 모든 호출 지점을 찾아야 한다

프로덕션이라면 `@Value("${kafka.sync.timeout:10}")`로 외부 설정으로 빼는 것이 더 좋다.

### whenComplete 콜백을 쓰지 않은 이유

비동기 `sendOrder()`는 `whenComplete`로 성공/실패 로깅을 한다. 동기 버전에서는 `.get()` 이후 직접 로깅하고, 실패는 catch 블록에서 예외로 전파한다. 동기 흐름에서 콜백은 불필요하다 - 결과가 즉시 있으므로.

### 비동기 vs 동기 - 언제 어떤 것을 쓰는가

| 상황 | 선택 | 이유 |
|------|------|------|
| 로그 수집, 이벤트 스트리밍 | 비동기 | 처리량 우선, 개별 실패 허용 |
| REST API 응답이 전송 성공에 의존 | **동기** | 실패 시 즉시 에러 응답 필요 |
| 순서 의존적 전송 (A 성공 후 B) | **동기** | A 실패 시 B를 보내면 안 됨 |
| DB 트랜잭션과 묶어야 할 때 | **동기** | 전송 실패 시 트랜잭션 롤백 필요 |
