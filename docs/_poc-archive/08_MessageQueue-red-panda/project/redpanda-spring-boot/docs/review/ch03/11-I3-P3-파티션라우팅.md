## 왜 이렇게 구현했는가

### 메시지 키와 파티션의 관계

Kafka의 기본 파티셔너(DefaultPartitioner)는 `murmur2(key) % numPartitions`으로 파티션을 결정한다. 이 해시 함수는 결정적(deterministic)이므로 **같은 key는 항상 같은 파티션**에 라우팅된다.

| key 값 | 파티션 결정 방식 | 결과 |
|--------|----------------|------|
| non-null | `murmur2(key) % partitions` | 항상 동일 파티션 |
| null | round-robin (sticky partitioner) | 파티션 골고루 분배 |

### orderId를 key로 사용하는 이유

같은 주문의 이벤트(생성, 수정, 취소)가 **같은 파티션**에 들어가면 **파티션 내 순서가 보장**된다. Consumer가 단일 파티션을 처리할 때 이벤트 순서가 유지되므로, 주문 상태 변경이 올바른 순서로 적용된다.

```
key="ORD-001" → 파티션 2
  offset 0: ORDER_CREATED
  offset 1: ORDER_MODIFIED   ← 순서 보장
  offset 2: ORDER_CANCELLED  ← 순서 보장
```

다른 주문(ORD-002)은 다른 파티션으로 갈 수 있으므로, 주문 간 순서는 보장되지 않는다. 이것이 Kafka의 설계 의도 — **파티션 내 순서 보장, 파티션 간 순서 미보장**.

### 파티션 수동 지정이 필요한 경우

`send(topic, partition, key, value)`로 파티션을 명시하면 **key의 해시를 무시**하고 지정한 파티션에 저장한다.

| 방식 | 사용 시점 |
|------|----------|
| key 기반 자동 라우팅 (기본) | 대부분의 경우. key 설계만 잘하면 충분 |
| 파티션 수동 지정 | 특정 파티션에 우선순위 데이터를 모으거나, 테스트 용도 |

실무에서 파티션을 수동 지정하는 경우는 드물다. key 설계로 라우팅을 제어하는 것이 표준 패턴이다.

### 파티션 수 변경 시 주의점

파티션 수가 변경되면 `murmur2(key) % newPartitions` 결과가 달라져서, **같은 key가 다른 파티션으로 라우팅**될 수 있다. 이미 적재된 메시지는 이동하지 않으므로, 같은 key의 이벤트가 여러 파티션에 분산된다.

→ 프로덕션에서 파티션 수 증가는 신중하게 결정해야 한다. 순서 보장이 깨질 수 있다.

## 핵심 학습 포인트

1. **같은 key → 같은 파티션** — murmur2 해시의 결정적 특성, 파티션 내 순서 보장의 기반
2. **null key → round-robin** — 파티션 골고루 분배, 순서 보장 불필요할 때 사용
3. **파티션 수동 지정 시 key 해시 무시** — send(topic, partition, key, value)의 partition이 우선
4. **key 설계 = 순서 보장 단위** — orderId를 key로 쓰면 같은 주문의 이벤트 순서 보장
5. **파티션 수 변경 = 라우팅 변경** — 프로덕션에서 파티션 증가는 순서 보장 관점에서 주의
