## 각 실습의 구현 범위

### A1. Linear Chain 구현
- 3단계 파이프라인: raw-orders → validated-orders → enriched-orders → fulfilled-orders
- 각 단계를 @KafkaListener + @SendTo로 연결
- 단계별 변환 로직 (검증: 필수 필드 체크, 보강: 고객 정보 조회, 이행: 재고 차감)
- NewTopic 빈 3개 선언

### A2. Fan-Out 구현
- 1:N 브로드캐스트: 하나의 이벤트 → 여러 토픽으로 동시 발행
- KafkaTemplate.executeInTransaction()으로 원자적 다중 전송
- 각 Consumer가 독립적으로 처리 (알림, 분석, 감사 로그 등)

### A3. Fan-In 구현
- N:1 집계: 여러 토픽의 결과를 하나로 합류
- correlation-id 기반 그룹핑 (같은 주문의 결과를 묶기)
- 상태 저장소: ConcurrentHashMap(단일 인스턴스) 또는 Redis(다중 인스턴스)
- 타임아웃: 일정 시간 내 모든 결과 미도착 시 불완전 집계로 처리

### A4. Filter-Branch 구현
- 조건부 라우팅: 이벤트 속성에 따라 다른 토픽으로 분기
- Strategy 패턴: RoutingStrategy 인터페이스 + 구현체로 라우팅 규칙 분리
- @KafkaListener + KafkaTemplate.send() 조합
- KStream.split() 대안 비교

### A5. Diamond 구현
- Fan-Out(분기) + Fan-In(합류) 결합
- correlation-id 헤더 전파로 분산된 결과 추적
- 합류 시점 판단: 모든 브랜치 완료 vs 타임아웃
- 부분 완료 처리 전략 (fallback 값, 재시도)

### A6. 파이프라인 관측성
- 커스텀 헤더 전파: X-Trace-Id, X-Pipeline-Stage, X-Created-At
- ProducerInterceptor로 헤더 자동 주입
- MDC 연동: Consumer에서 헤더 → MDC 매핑 (로그 추적)
- TTL 헤더로 순환 방지 (stage 카운터 초과 시 DLQ)
- @RetryableTopic + DLQ 통합 모니터링

### B1. ReplyingKafkaTemplate 기본 설정 + 동기 Request-Reply
- Reply Container 빈 설정 (ConcurrentMessageListenerContainer)
- ReplyingKafkaTemplate 빈 생성 (replyTimeout, sharedReplyTopic)
- 호출자: sendAndReceive()로 동기 요청-응답
- 응답자: @KafkaListener + @SendTo로 요청 처리 및 응답 반환
- correlation-id, reply-topic 헤더 동작 원리 이해

### B2. 비동기 변형 (CompletableFuture + 다중 요청)
- CompletableFuture 패턴: sendAndReceive() 반환값을 비동기로 처리
- WebFlux 통합: Mono.fromFuture()로 리액티브 파이프라인 연결
- allOf()로 다중 요청 동시 전송 + 전체 응답 대기

### B3. 다중 인스턴스 Reply 라우팅
- 문제: 같은 Consumer Group 공유 시 응답이 잘못된 인스턴스에 전달
- 해결법 1: REPLY_PARTITION 헤더 + setSharedReplyTopic(true)
- 해결법 2: 인스턴스별 고유 reply 토픽 (hostname/UUID 기반)
- 두 방식의 적합 시점 비교 (파티션 수 제약 vs 토픽 수 증가)

### B4. 에러 전파
- 서버 측: @SendTo 응답자에서 예외 발생 시 reply 헤더에 예외 정보 포함
  - EXCEPTION_FQCN, EXCEPTION_MESSAGE, EXCEPTION_STACKTRACE 헤더
- 호출자 측: reply 헤더에서 에러 감지 → 정상/에러 분기 처리
- ReplyExceptionHandler 커스터마이징

### B5. AggregatingReplyingKafkaTemplate (Scatter-Gather)
- Scatter-Gather 패턴: 하나의 요청 → 다중 파티션/인스턴스에 분산 → 응답 집계
- releaseStrategy 구현 (응답 N개 도착 시 해제, 또는 타임아웃)
- 호출자: 다중 파티션에 요청 분산 + 집계된 응답 수신
- 부분 응답 처리 (타임아웃 내 일부만 도착한 경우)

### C1. Event Envelope
- ProducerInterceptor 구현: onSend()에서 CloudEvents 헤더 자동 주입
  - ce-type, ce-source, ce-id, ce-time, ce-specversion
- interceptor.classes 속성에 FQCN 등록 (또는 ProducerPostProcessor)
- Consumer에서 @Header("ce-type") 등으로 메타데이터 추출
- 이벤트 타입별 라우팅에 ce-type 헤더 활용

### C2. Dual Write → Outbox 진화
- 단계 1: 단순 Dual Write (DB 저장 + KafkaTemplate.send) — 문제점 확인
- 단계 2: TransactionSynchronization.afterCommit()으로 순서 보장
- 단계 3: OutboxEvent 엔티티 + @TransactionalEventListener로 Outbox 패턴
- 단계 4: (선택) Debezium CDC 설정으로 폴링 없는 Outbox

### C3. Event Gateway
- EventGatewayController: REST 엔드포인트 → Kafka 이벤트 발행
- 이벤트 검증: @Valid + CloudEvents 헤더 매핑
- 이벤트 변환: REST DTO → Avro/도메인 이벤트
- Rate Limiting: @RateLimiter (Resilience4j) 또는 Bucket4j 적용

### C4. Event Router
- 방식 1: @KafkaListener + @KafkaHandler — 타입별 다형성 라우팅
- 방식 2: KStream.split() — 스트림 기반 조건부 라우팅 (Predicate)
- 방식 3: @SendTo — 반환값 기반 정적 라우팅
- 세 방식의 적합 시점 비교 문서

### C5. Claim Check
- S3StorageService: putObject/getObject 래퍼 (AWS SDK v2)
- ClaimCheckProducer: 메시지 크기 > 500KB → S3 업로드 + 참조 키만 Kafka로 전송
- ClaimCheckConsumer: 참조 키 수신 → S3에서 원본 다운로드 → 처리
- build.gradle S3 의존성 설정

### C6. Event Splitter + Translator
- Splitter: 주문 이벤트 → 주문 항목별 개별 이벤트로 분할 (1→N)
- Translator: Avro 스키마 진화(호환 변환) 활용한 자동 변환
- Kafka Streams mapValues()로 도메인 간 스키마 변환
- @KafkaListener 기반 변환 서비스 (간단한 경우)

### C7. Sink Consumer
- S3 Sink: 배치 리스너(BatchMessageListener) + 주기적 flush (시간/건수 기준)
- Elasticsearch Sink: BulkRequest 배치 색인 + 실패 항목 재시도
- 배치 모드 ContainerFactory 설정 (ConcurrentKafkaListenerContainerFactory.setBatchListener)
- 외부 시스템 장애 시 재시도/DLQ 처리

### C8. 패턴 통합 파이프라인
- 전체 흐름: REST 수신(Gateway) → 검증(Filter) → 라우팅(Router) → 변환(Translator) → 적재(Sink)
- Kafka Streams Topology로 파이프라인 구성
- TopologyTestDriver로 단위 테스트
- Testcontainers + Redpanda 통합 테스트

### D1. 3-tier 통신 아키텍처 리뷰 + Docker Compose 환경 구성
- Spring Boot / Redpanda / Jenkins 세 컴포넌트의 역할과 통신 관계 리뷰
- 동기(REST, gRPC) vs 비동기(Message Queue) 통신 패턴 비교
- 하이브리드 접근법: 외부→REST, 내부→Redpanda, 고성능→gRPC
- 개발 환경 Docker Compose 구성 (Spring Boot + Redpanda + Jenkins)
