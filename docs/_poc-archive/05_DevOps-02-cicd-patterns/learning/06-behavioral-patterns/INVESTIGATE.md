# Ch06. 행동 디자인 패턴 — 탐구 질문

이 질문들은 LEARN.md의 개념을 실제 운영 환경 맥락에서 심화한다.
각 질문에 답하기 전에 직접 코드를 작성하거나 다이어그램을 그려보길 권한다.
정답보다 "왜 이 설계를 선택했는가"를 설명할 수 있는 것이 목표다.

---

## Q1. Observer 패턴에서 webhook 실패 시 재시도 전략은 어떻게 설계해야 하는가?

LEARN.md의 webhook observer는 `Promise.allSettled`로 실패를 기록하지만 재시도하지 않는다.
Slack webhook이 일시적으로 다운되면 그 알림은 영영 사라진다.

탐구 방향:
- 지수 백오프(exponential backoff) + jitter를 적용하면 재시도 폭풍(retry storm)을 어떻게 방지하는가?
- 재시도 횟수와 알림 지연 사이의 트레이드오프는 무엇인가? (PagerDuty는 즉각성이 핵심)
- 재시도 실패가 지속되면 Dead Letter Queue(DLQ)에 보관하는 방법을 설계해보라
- Redis나 DB에 발행 이벤트를 저장하는 Outbox 패턴과 Observer 패턴을 어떻게 결합할 수 있는가?
- 구독자별로 재시도 정책을 달리 적용해야 할 때(Slack은 3회, PagerDuty는 10회) Observer 인터페이스를 어떻게 확장하는가?

힌트: `axios-retry` 라이브러리의 `retryCondition`과 `retryDelay` 옵션을 살펴보라.

## Q2. Strategy 패턴과 if-else 분기 중 어느 쪽이 장기적으로 유지보수하기 쉬운가?

"패턴은 복잡도를 추가한다"는 반론이 있다. Strategy 패턴은 클래스 수가 늘어나고, 작은 팀에는 오버엔지니어링일 수 있다.

탐구 방향:
- 배포 전략이 2개일 때와 5개일 때 if-else와 Strategy의 코드 크기를 비교해보라
- if-else의 분기는 테스트 커버리지를 어떻게 복잡하게 만드는가? (각 분기마다 독립 테스트 필요)
- Strategy 패턴에서 전략 선택 로직(`select_strategy`) 자체는 어디에 있어야 하는가? 팩토리 메서드인가, 설정 파일인가, 환경변수인가?
- 전략 객체를 YAML 설정으로 선언하고 런타임에 로드하면 코드 수정 없이 전략을 바꿀 수 있다. 이 방식의 장단점은 무엇인가?
- Argo Rollouts나 Flagger는 Strategy 패턴을 CRD(Custom Resource Definition) 수준에서 구현한다. 이들의 설계에서 무엇을 배울 수 있는가?

## Q3. Command 패턴으로 파이프라인 롤백을 어떻게 구현하는가?

LEARN.md의 `Pipeline._rollback()`은 단순히 `undo()`를 역순으로 호출한다.
실제 파이프라인에서는 `undo()`가 실패할 수도 있고, 일부 단계는 되돌릴 수 없다.

탐구 방향:
- Docker 이미지 push는 undo가 가능한가? (레지스트리에 이미 올라간 이미지)
- `undo()` 자체가 실패할 때의 처리 전략을 설계해보라 (재시도? 알림? 수동 개입 요청?)
- 파이프라인 히스토리를 DB에 저장하면 어떤 이점이 있는가? (이벤트 소싱과의 연결)
- Command 패턴으로 파이프라인 실행 로그를 구조화된 감사 로그(audit log)로 만들 수 있는가?
- `execute()`가 멱등성을 보장해야 하는 이유는 무엇이고, 어떻게 구현하는가? (재시도 시 중복 실행 방지)

## Q4. GitHub Actions matrix strategy의 비용을 어떻게 최적화하는가?

LEARN.md의 3 × 3 × 2 matrix는 최대 18개 runner를 동시에 사용한다.
GitHub Actions의 과금은 runner 분 × 플랫폼 배율(Linux=1, Windows=2, macOS=10)로 계산된다.

탐구 방향:
- macOS runner는 Linux의 10배 비용이다. matrix에서 macOS를 언제 포함하고 언제 제외해야 하는가?
- PR 빌드와 main 브랜치 빌드에 다른 matrix를 적용하는 전략(PR은 핵심 조합만, main은 전체)을 설계해보라
- `paths` 트리거를 matrix와 조합하면 어떻게 변경된 컴포넌트에 해당하는 matrix만 실행할 수 있는가?
- self-hosted runner를 사용하면 비용 구조가 어떻게 달라지는가? 유지보수 비용과의 트레이드오프는?
- matrix 조합 결과를 캐싱하면 중복 실행을 줄일 수 있다. `actions/cache`의 캐시 키 설계 전략을 수립해보라

## Q5. Chain of Responsibility로 배포 승인 체인을 설계할 때 어떤 엣지 케이스를 고려해야 하는가?

LEARN.md의 승인 체인은 위험도 점수를 기준으로 핸들러가 결정된다.
실제 운영에서는 시간, 승인자 부재, 긴급 배포 등 예외 상황이 빈번하다.

탐구 방향:
- 승인자가 응답하지 않을 때(timeout) 자동으로 다음 승인자에게 에스컬레이션하는 로직을 설계해보라
- 긴급 배포(hotfix)에서 체인을 단축하는 "bypass" 메커니즘은 어떻게 감사(audit) 가능하게 만드는가?
- 위험도 점수 계산 자체를 Command나 Strategy 패턴과 조합하여 플러그인 형태로 만들 수 있는가?
- 승인 체인에 SLA(예: 30분 내 승인 없으면 자동 반려)를 적용하면 파이프라인 설계에 어떤 변화가 생기는가?
- LEARN.md의 Chain은 단방향이다. 승인 후 재검토 요청(request changes)이 필요한 경우 양방향 체인을 어떻게 모델링하는가?

## Q6. Observer와 Strategy를 조합하면 어떤 시너지가 생기는가?

LEARN.md 마지막 섹션에서 두 패턴의 조합을 간략히 소개했다.
이 조합이 단순 패턴 중첩이 아닌 새로운 설계 가능성을 열어준다는 점을 탐구한다.

탐구 방향:
- "배포 환경에 따라 알림 전략이 달라진다"는 요구사항을 Observer + Strategy 없이 구현하면 코드가 어떻게 생겼는가? (비교를 위해 직접 작성해보라)
- Observer의 이벤트 스키마(event object)를 Strategy 선택의 입력으로 사용하면 어떤 정보를 이벤트에 포함해야 하는가?
- 세 번째 패턴 Command를 추가하면 "알림 전송"이라는 행동 자체를 큐에 넣고 재시도할 수 있다. 이 3중 조합의 구조를 다이어그램으로 그려보라
- 행동 패턴들을 조합할 때 발생하는 복잡도 증가를 어떤 기준으로 정당화하는가? ("이 복잡도는 어떤 변화를 단순하게 만드는가?"를 기준으로 판단해보라)
- 실제 오픈소스 CI/CD 도구(Jenkins, Tekton, Argo Workflows)에서 이 패턴들의 흔적을 찾아보라. 어떤 형태로 구현되어 있는가?

## 심화 실험

위 질문들을 모두 탐구했다면 다음 구현을 시도해본다.

1. **재시도 가능한 Observer**: Q1에서 설계한 지수 백오프 재시도 로직을 webhook-observer.js에 추가하라. DLQ는 간단한 파일 또는 Redis list로 구현한다. 재시도 3회 실패 시 콘솔 경고와 함께 DLQ에 적재되는지 확인한다.

2. **설정 기반 Strategy**: Q2에서 논의한 YAML 설정으로 배포 전략을 선택하는 `StrategyLoader`를 구현하라. `pyyaml`로 설정을 읽고 Strategy 객체를 생성한다. 설정 예시: `strategy: canary`, `canary_weight: 10`, `interval_minutes: 5`.

3. **비용 시뮬레이터**: Q4에서 분석한 matrix 비용 계산기를 Python으로 작성하라. matrix 조합과 각 runner의 예상 실행 시간을 입력받아 월간 예상 비용을 출력한다. Linux/Windows/macOS 배율(1/2/10)을 적용하고, PR 빌드(핵심 조합)와 main 빌드(전체 조합)를 구분하여 비교한다.

4. **Chain 승인 타임아웃**: Q5에서 논의한 승인자 무응답 처리를 구현하라. `asyncio.wait_for`로 승인 대기에 타임아웃을 적용하고, 타임아웃 발생 시 다음 핸들러로 에스컬레이션한다. 모든 에스컬레이션 이유를 구조화된 로그로 남긴다.
