# Ch09. 탐구 질문 — 측정, 감사, 성숙도 모델

이 챕터의 개념을 실무에 적용하기 전에 스스로 답해볼 질문들이다. 각 질문은 단순한 정의 암기가 아니라 트레이드오프와 실제 적용 맥락을 탐구하도록 설계되었다.

---

## Q1. DORA 메트릭 수집 자동화 방법

**질문**: 배포 빈도, 리드 타임, CFR, MTTR을 수동 집계 없이 자동으로 수집하려면 어떤 아키텍처가 필요한가?

**탐구 포인트**:
- 리드 타임 계산에서 "최초 커밋"을 어떻게 정의하는가? 피처 브랜치의 첫 커밋인가, main에 병합된 커밋인가? 정의에 따라 수치가 크게 달라진다.
- CI/CD 플랫폼(GitHub Actions, Jenkins)마다 이벤트 모델이 다르다. 플랫폼 독립적인 수집 계층을 설계하려면 어떤 추상화가 필요한가?
- MTTR 계산을 위해 "장애 감지 시각"을 어디서 가져오는가? PagerDuty, Opsgenie, 내부 인시던트 시스템 중 어느 것과 연동해야 하는가?
- 수집된 데이터가 부정확하면 잘못된 의사결정을 유발한다. 메트릭 정확도를 검증하는 방법은 무엇인가?

**실험 아이디어**: GitHub Actions에서 배포 워크플로우 완료 시 Prometheus Push Gateway로 메트릭을 전송하는 단계를 추가하고, 1주일 후 수집된 데이터와 수동 집계 결과를 비교해 보라.

## Q2. Quality Gate 실패 시 파이프라인 대응 전략

**질문**: SonarQube Quality Gate가 실패했을 때 파이프라인을 즉시 중단하는 것이 항상 옳은가? 상황에 따른 대응 전략을 어떻게 설계해야 하는가?

**탐구 포인트**:
- 핫픽스(hotfix) 상황에서 Quality Gate 실패로 배포가 막힌다면 어떻게 해야 하는가? "긴급 우회(emergency bypass)" 메커니즘을 설계할 때 남용 방지를 어떻게 보장하는가?
- 신규 코드 기준과 전체 코드 기준을 다르게 적용하는 것이 합리적인가? 레거시 코드베이스에서 80% 커버리지 기준을 일괄 적용하면 어떤 일이 발생하는가?
- Quality Gate 조건을 팀이 임의로 완화할 수 있다면 게이트의 의미가 있는가? 조건 변경에 대한 거버넌스는 어떻게 설계하는가?
- SAST(정적 분석) 결과의 오탐(false positive)이 많을 때 개발자 신뢰도를 유지하면서도 보안 기준을 지키려면 어떤 접근이 필요한가?

**실험 아이디어**: 고의로 커버리지를 낮춘 브랜치를 생성해 Quality Gate 실패 시나리오를 만들고, 팀의 실제 대응 패턴을 관찰해 보라. 우회 시도가 있다면 그 이유를 파악하는 것이 더 중요한 정보다.

## Q3. 성숙도 모델의 실제 적용과 한계

**질문**: CI/CD 성숙도 모델을 조직에 도입할 때 어떤 함정이 있으며, 모델 자체의 한계는 무엇인가?

**탐구 포인트**:
- 성숙도 레벨이 높다고 반드시 좋은 소프트웨어를 만들고 있다는 의미인가? 성숙도 점수를 KPI로 삼으면 어떤 역효과가 발생할 수 있는가?
- 배포 빈도가 높지 않아야 적절한 도메인(규제 산업, 임베디드 시스템)에서 Level 5를 목표로 하는 것이 올바른가?
- 성숙도 모델은 기술 실천에 집중하는 경향이 있다. 조직 문화, 심리적 안전감, 팀 자율성 같은 비기술 요소는 어떻게 평가하는가?
- 외부 감사나 경영진 보고를 위해 성숙도 점수를 "관리"하려는 동기가 생긴다면 어떻게 대응하는가?

**실험 아이디어**: 팀 내 다른 구성원들이 동일한 체크리스트를 독립적으로 평가하게 하고 결과를 비교해 보라. 항목 해석의 차이가 클수록 평가 기준이 모호하다는 신호다.

## Q4. Lead Time 단축을 위한 병목 식별

**질문**: 리드 타임이 기대보다 길다면 어디서 시간이 소모되는지 어떻게 파악하고 우선순위를 정하는가?

**탐구 포인트**:
- 리드 타임을 구성 요소로 분해할 수 있는가? (코딩 시간 + PR 대기 시간 + 리뷰 시간 + CI 실행 시간 + 배포 대기 시간) 각 구간 중 어느 곳이 가장 긴가?
- 코드 리뷰 대기 시간이 병목이라면 기술적 해결책(자동 리뷰어 배정, 리뷰 SLA)과 문화적 해결책(리뷰 우선순위 합의) 중 무엇이 더 효과적인가?
- CI 파이프라인 실행 시간이 30분 이상이라면 어떤 최적화 전략을 적용할 수 있는가? 캐싱, 병렬화, 테스트 분할 중 ROI가 가장 높은 것은 무엇인가?
- 특정 파일이나 모듈 변경이 유독 리드 타임을 높인다면 그 패턴에서 무엇을 알 수 있는가? (예: 모노리스의 공유 모듈, 잦은 충돌 영역)

**실험 아이디어**: 지난 20개 PR을 분석해 커밋 시각, PR 생성 시각, 승인 시각, 머지 시각, 배포 시각을 각각 기록하고 가장 큰 갭이 어느 구간인지 확인해 보라.

## Q5. Change Failure Rate 감소와 테스트 전략의 관계

**질문**: CFR을 낮추기 위한 테스트 전략을 설계할 때 커버리지 숫자에 집착하는 것 외에 어떤 접근이 실질적으로 효과적인가?

**탐구 포인트**:
- 커버리지 80%와 커버리지 60%인 두 팀 중 CFR이 낮은 팀이 어디인지 예측할 수 있는가? 커버리지 숫자와 CFR 사이의 실제 상관관계는 어느 정도인가?
- 프로덕션 장애를 유발한 버그가 테스트로 잡힐 수 있었는가? 사후 분석(postmortem)에서 "테스트가 있었다면 막을 수 있었던" 케이스를 추적하면 어떤 패턴이 나오는가?
- 프로퍼티 기반 테스트(Property-Based Testing), 뮤테이션 테스트(Mutation Testing)는 일반 단위 테스트와 어떤 점에서 다른 종류의 버그를 잡는가?
- 카나리 배포나 피처 플래그로 변경의 영향 범위를 제한하는 것이 CFR을 낮추는 데 테스트 강화보다 더 효과적인 경우는 언제인가?

**실험 아이디어**: 최근 프로덕션 장애 3건을 선택해 각 장애가 어떤 테스트 전략으로 사전에 탐지될 수 있었는지 역추적해 보라. 테스트 타입(단위/통합/E2E)과 탐지 가능 여부를 매핑하면 투자 우선순위가 드러난다.

## Q6. 조직 문화와 CI/CD 성숙도의 상관관계

**질문**: 동일한 도구와 프로세스를 도입해도 어떤 팀은 성과를 내고 어떤 팀은 그렇지 못하다. CI/CD 성숙도에서 문화가 기술보다 중요한 이유는 무엇인가?

**탐구 포인트**:
- DORA 연구에서 기술 실천(자동화, 테스트)과 문화 지표(심리적 안전감, 학습 문화) 중 어느 쪽이 높은 소프트웨어 전달 성과와 더 강하게 상관되는가?
- "배포가 무섭다"는 감정이 팀 내에 있다면 이것은 기술 문제인가, 문화 문제인가? 배포에 대한 두려움을 측정하는 방법은 무엇인가?
- 포스트모템을 비난 없이(blameless) 진행하는 문화가 MTTR에 미치는 영향은 무엇인가? 비난 문화에서는 장애 정보가 어떻게 왜곡되는가?
- CI/CD 성숙도 향상을 "엔지니어링 팀의 과제"로만 다루면 왜 실패하기 쉬운가? 제품 관리자, 운영팀, 경영진이 각각 어떤 역할을 해야 하는가?

**실험 아이디어**: 팀에게 "지금 당장 프로덕션에 배포하는 것이 편한가?"라는 질문을 익명으로 설문해 보라. 불편함의 이유를 분류하면 기술적 원인(느린 파이프라인, 테스트 부족)과 문화적 원인(승인 절차, 책임 소재)으로 나뉜다. 어느 쪽이 더 많은지가 개선 방향을 결정한다.

## 심화 탐구 — 측정의 역설

**질문**: 측정이 오히려 행동을 왜곡하는 사례는 무엇이며, "좋은 메트릭"과 "나쁜 메트릭"을 어떻게 구별하는가?

굿하트의 법칙(Goodhart's Law)은 "측정이 목표가 되는 순간 좋은 측정 지표가 아니게 된다"고 말한다. DORA 메트릭도 예외가 아니다. 배포 빈도를 높이기 위해 의미 없는 변경을 쪼개서 배포하거나, CFR을 낮추기 위해 장애를 "계획된 다운타임"으로 분류하는 일이 실제로 발생한다.

**탐구 포인트**:
- 배포 빈도를 높이기 위해 기능을 인위적으로 작게 쪼개는 것이 실제로 리스크를 줄이는가, 아니면 숫자만 좋아 보이게 하는가?
- 메트릭을 수집하고 공개할 때 팀 간 경쟁이 생기면 어떤 부작용이 나타나는가? "우리 팀 CFR이 저 팀보다 낮다"는 비교가 협업에 미치는 영향은?
- 메트릭 정의를 팀이 직접 결정하는 것과 외부에서 부과하는 것 중 어느 쪽이 더 의미 있는 개선으로 이어지는가?
- "측정할 수 없으면 개선할 수 없다"는 명제와 "측정 자체가 행동을 왜곡한다"는 명제를 어떻게 조화시킬 수 있는가?

**실험 아이디어**: 팀에게 DORA 메트릭 목표치(예: 배포 빈도 주 3회 이상)를 공개적으로 부여하고 한 달간 추적해 보라. 목표 달성을 위해 어떤 행동 변화가 나타나는지 관찰하고, 그 변화가 실제 품질 향상으로 이어졌는지 평가해 보라. 숫자와 실질 개선 사이의 간극이 메트릭 설계의 완성도를 알려준다.
