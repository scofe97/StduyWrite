# 05. CLI 토큰 분석

Claude Code는 사용한 만큼 비용이 발생하지만, 기본적으로 "얼마나 사용했는지" 알려주지 않습니다. OMC CLI는 세션 로그를 분석하여 토큰 사용량, 비용, 에이전트별 분석을 제공합니다. 비용 최적화의 첫 번째 단계는 측정이며, 측정할 수 없으면 개선할 수 없습니다.

---

## 목표

- [ ] OMC CLI 주요 명령어(stats, agents, tui, backfill)를 사용할 수 있다
- [ ] 모델별 비용 구조를 이해하고 비용 최적화 전략을 설명할 수 있다
- [ ] 에이전트별 비용 분석 결과를 해석할 수 있다

---

## 1. 주요 명령어

### omc stats - 토큰 사용 통계

세션별, 날짜별 토큰 사용량을 집계합니다.

```bash
omc stats

# 출력 예시:
# 📊 Token Usage Statistics
# ─────────────────────────
# Total Input:  1,234,567 tokens
# Total Output:   456,789 tokens
# Total:       1,691,356 tokens
#
# 📅 By Date:
# 2026-02-09: 500,000 tokens
# 2026-02-08: 691,356 tokens
```

```bash
# 기간 필터링
omc stats --days 7     # 최근 7일
omc stats --days 30    # 최근 30일

# JSON 출력 (스크립트 연동용)
omc stats --json
```

### omc agents - 에이전트별 비용 분석

어떤 에이전트가 가장 많은 비용을 사용했는지 분석합니다. 비용 최적화의 핵심 데이터입니다.

```bash
omc agents

# 출력 예시:
# 🤖 Agent Breakdown by Cost
# ──────────────────────────
# 1. architect    $2.34  (45%)  ████████░░
# 2. explorer     $1.23  (24%)  ████░░░░░░
# 3. executor     $0.89  (17%)  ███░░░░░░░
# 4. planner      $0.72  (14%)  ██░░░░░░░░
```

이 결과에서 architect(Opus)가 전체 비용의 45%를 차지하고 있다면, architect 호출을 줄이거나 `architect-low`(Haiku)로 대체할 수 있는 작업이 있는지 검토합니다.

### omc tui - 대화형 대시보드

터미널에서 대화형 UI로 상세 분석을 수행합니다.

```bash
omc tui
```

제공하는 기능:
- 세션별 필터링
- 시간대별 사용량 그래프
- 상세 로그 조회
- 에이전트별 드릴다운

### omc backfill - 과거 데이터 채우기

이전 세션의 토큰 데이터를 분석합니다. 설치 전의 세션도 소급 분석이 가능합니다.

```bash
omc backfill
```

## 2. 비용 구조

### 모델별 단가

| 모델 | 입력 토큰 (per 1M) | 출력 토큰 (per 1M) | 특징 |
|------|-------------------|-------------------|------|
| **Opus** | $15 | $75 | 최고 추론 능력, 최고 비용 |
| **Sonnet** | $3 | $15 | 균형잡힌 성능/비용 |
| **Haiku** | $0.25 | $1.25 | 최저 비용, 빠른 응답 |

### 비용 시나리오

일반적인 작업 세션에서 발생하는 비용을 추정합니다.

| 시나리오 | 토큰 사용 | Opus만 | 혼합 (Ecomode) | 절감 |
|---------|----------|--------|----------------|------|
| 코드 탐색 (100K 입력) | 100K | $1.50 | $0.025 (Haiku) | **98%** |
| 기능 구현 (500K 입력, 100K 출력) | 600K | $15.00 | $3.00 (Sonnet) | **80%** |
| 아키텍처 설계 (200K 입력, 50K 출력) | 250K | $6.75 | $6.75 (Opus 필수) | 0% |

코드 탐색을 Haiku로 대체하는 것만으로 98%의 비용을 절감할 수 있습니다. 이것이 3-Tier 라우팅과 Ecomode의 핵심 가치입니다.

---

## 3. 비용 최적화 전략

### 전략 1: Haiku 우선 사용

간단한 작업은 Haiku 에이전트를 활용합니다.

| 작업 | Opus | Haiku | 절감 |
|------|------|-------|------|
| 파일 탐색 | `architect` | `explore` | 60x |
| 간단한 수정 | `executor-high` | `executor-low` | 60x |
| 문서 작성 | `writer` (기본 Haiku) | 이미 최적 | - |

### 전략 2: 컨텍스트 관리

불필요한 파일을 컨텍스트에 포함하지 않습니다. 컨텍스트 크기는 입력 토큰에 직접 영향을 미치므로, 관련 없는 파일을 읽는 것은 비용 낭비입니다.

### 전략 3: 작업 분할

큰 작업을 작은 단위로 나누면 각 세션의 컨텍스트가 작아져 비용이 줄어듭니다. 또한 작업이 실패했을 때 롤백 범위도 줄어듭니다.

### 전략 4: 캐시 활용

반복적으로 같은 파일을 읽는 패턴을 피합니다. cache_read_input_tokens는 할인된 가격이 적용되지만, 아예 읽지 않는 것이 가장 좋습니다.

---

## 4. 데이터 소스

OMC CLI는 Claude Code의 세션 로그 파일에서 데이터를 추출합니다.

| 항목 | 위치 |
|------|------|
| 세션 로그 | `~/.claude/projects/{project-id}/` |
| 형식 | JSONL (한 줄 = 한 이벤트) |
| 포함 정보 | 토큰 수, 모델, 에이전트 이름, 도구 사용 기록 |

## 체크포인트

다음 질문에 면접에서 답변하듯이 설명할 수 있는지 확인하세요.

1. **`omc agents` 결과에서 architect가 전체 비용의 60%를 차지합니다. 어떻게 최적화하겠습니까?**
2. **Opus, Sonnet, Haiku의 비용 차이는 몇 배이며, 이것이 실무에서 어떤 의미를 가지나요?**
3. **비용 최적화에서 "측정이 먼저"인 이유는 무엇인가요?**

<details>
<summary>모범 답안 확인</summary>

**1. architect 비용 60% 최적화**

architect(Opus)가 호출되는 상황을 분류합니다. 간단한 코드 구조 질문은 `architect-low`(Haiku)로 대체합니다. 파일 탐색 목적의 architect 호출은 `explore`(Haiku)로 전환합니다. 실제로 복잡한 아키텍처 분석이 필요한 경우만 architect(Opus)를 유지합니다. Ecomode를 활성화하여 자동으로 적절한 티어가 선택되도록 합니다. 변경 후 `omc agents`로 비율 변화를 추적합니다.

**2. 모델별 비용 차이의 의미**

입력 토큰 기준으로 Opus($15/1M)는 Sonnet($3/1M)의 5배, Haiku($0.25/1M)의 60배입니다. 출력 토큰도 동일한 비율입니다. 실무에서 이것은, 코드 탐색 작업 하나를 Opus로 처리하면 Haiku로 60번 처리할 수 있는 비용이라는 뜻입니다. 하루에 수백 번 이루어지는 탐색/조회 작업을 모두 Opus로 처리하면 비용이 60배로 불어나므로, 3-Tier 라우팅으로 자동 최적화하는 것이 필수적입니다.

**3. 측정이 먼저인 이유**

최적화 대상을 정확히 식별해야 효과적인 개선이 가능하기 때문입니다. "비용이 높다"는 막연한 인식만으로는 어디를 개선해야 할지 알 수 없습니다. `omc agents`로 에이전트별 비용 비중을 측정하면, 실제로 비용을 지배하는 에이전트가 무엇인지 데이터로 확인됩니다. 비용의 80%를 차지하는 상위 20% 에이전트를 최적화하는 것이 전체를 균등하게 개선하는 것보다 훨씬 효과적입니다.

</details>
