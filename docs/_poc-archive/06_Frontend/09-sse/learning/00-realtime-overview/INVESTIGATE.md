# 00. 실시간 통신 기술 개론 - 조사 (INVESTIGATE)

## 학습 목표
웹 실시간 통신 기술의 역사적 발전 과정을 이해하고, 폴링/롱폴링/SSE/WebSocket의 동작 원리와 트레이드오프를 면접에서 설명할 수 있다.

---

## Q1. 웹 실시간 통신의 역사

### 생각해볼 질문
- **1990년대 웹에서 "실시간"처럼 보이게 하려면 어떻게 했을까요?**
- **`<iframe>`, `<frame>` 태그가 실시간 통신에 어떻게 사용되었나요?**
- **"딸깍딸깍" 소리는 왜 났을까요?**

### 힌트
- 페이지 전체를 새로고침하지 않고 일부만 업데이트하려면?
- 숨겨진 프레임에 새 페이지를 로드하면?
- HTTP/1.0은 매 요청마다 새 연결

## Q2. 폴링 (Polling)

### 생각해볼 질문
- **폴링의 동작 원리를 설명해보세요.**
- **폴링 주기를 짧게 하면 무슨 문제가 생기나요?**
- **폴링 주기를 길게 하면 무슨 문제가 생기나요?**
- **폴링이 적합한 상황은 언제인가요?**

### 힌트
```javascript
// 이 코드가 어떻게 동작하는지 분석해보세요
setInterval(async () => {
  const response = await fetch('/api/data');
  const data = await response.json();
  updateUI(data);
}, 5000);  // 5초마다
```

## Q3. 롱폴링 (Long Polling)

### 생각해볼 질문
- **롱폴링이 일반 폴링과 다른 점은 무엇인가요?**
- **서버가 응답을 "홀딩"한다는 것은 무슨 의미인가요?**
- **타임아웃 설정이 왜 필요한가요?**
- **롱폴링의 한계점은 무엇인가요?**

### 힌트
- 서버가 새 데이터가 생길 때까지 응답을 보류
- 30~60초 후 타임아웃으로 빈 응답
- 응답 받자마자 즉시 새 요청

---

## Q4. 기술 선택 기준

### 생각해볼 질문
- **단방향 vs 양방향 통신이 필요할 때 각각 어떤 기술을 선택해야 할까요?**
- **HTTP/2가 SSE에 미친 영향은 무엇인가요?**
- **방화벽/프록시 환경에서 WebSocket은 왜 문제가 될 수 있나요?**
- **모바일 환경에서 고려해야 할 점은?**

### 힌트
- HTTP/1.1의 동시 연결 제한 (도메인당 6개)
- HTTP/2 멀티플렉싱 (128개 스트림)
- 모바일 Safari 백그라운드 30초 제한
