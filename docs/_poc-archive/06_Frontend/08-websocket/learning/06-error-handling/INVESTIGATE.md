# INVESTIGATE: 에러 처리

## 학습 목표
WebSocket에서 발생할 수 있는 다양한 에러 상황을 이해하고 적절히 처리하는 방법을 학습합니다.

## 소크라테스 질문

### Q1. onError vs onClose

> `onError`와 `onClose`의 차이점은 무엇인가?

**생각해볼 점:**
- `onError`가 호출된 후 `onClose`도 호출되는가?
- `onError`에서 받을 수 있는 정보는 무엇인가?
- 모든 종료 상황에서 `onError`가 먼저 호출되는가?

```typescript
useWebSocket(url, {
  onError: (event) => { /* ??? */ },
  onClose: (event) => { /* ??? */ },
});
```

### Q2. 에러 유형 분류

> WebSocket에서 발생할 수 있는 에러 유형은 어떤 것들이 있는가?

**생각해볼 점:**
- 연결 실패 (서버 다운, 네트워크 문제)
- 연결 중 끊김 (네트워크 불안정)
- 프로토콜 에러 (잘못된 데이터 형식)
- 애플리케이션 에러 (서버에서 보내는 에러 메시지)

---

### Q3. HTTP 폴링 Fallback

> WebSocket 실패 시 HTTP 폴링으로 Fallback하는 것이 언제 필요한가?

**생각해볼 점:**
- WebSocket을 지원하지 않는 환경이 있는가?
- 방화벽/프록시가 WebSocket을 차단하는 경우는?
- Fallback 구현 시 어떤 점을 고려해야 하는가?

---

### Q4. 사용자 알림

> 에러 발생 시 사용자에게 어떻게 알려야 하는가?

**생각해볼 점:**
- 모든 에러를 사용자에게 표시해야 하는가?
- 에러 메시지는 어떻게 표현해야 하는가? (기술적 vs 친화적)
- 에러 알림의 위치와 지속 시간은?
