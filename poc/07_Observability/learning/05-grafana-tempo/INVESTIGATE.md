# Ch05. Grafana Tempo - 점검 질문

## Q1. 로그와 메트릭만으로는 답하기 어렵고, 트레이스로만 답할 수 있는 질문은 무엇인가?

**핵심 포인트**:
- "전체 요청 중 어느 구간이 느렸는가" — 메트릭은 집계값, 로그는 서비스별 단편
- "에러를 유발한 upstream은 어디인가" — 요청 경로의 인과 관계는 trace만 보여 줌
- 개별 요청 단위의 경로 추적이 필요한 모든 질문

## Q2. Tempo가 span 속성을 인덱싱하지 않는 설계를 택한 이유를 비용과 운영 관점에서 설명하라.

**핵심 포인트**:
- 인덱싱 없으므로 저장 비용이 데이터 크기에만 비례
- 객체 스토리지 활용 → 인덱스 DB(Cassandra, ES) 운영 불필요
- Grafana 스택의 일관된 철학: "인덱싱 최소화, 비용과 단순성 택함"

## Q3. Jaeger와 Tempo의 핵심 차이를 탐색 흐름 관점에서 설명하라.

**핵심 포인트**:
- Jaeger: 서비스명/태그로 독립 검색 → trace 조회 (자체 완결)
- Tempo: Loki 로그 탐색 → trace_id 추출 → Tempo trace 조회 (스택 협력)
- 선택 기준: Grafana 스택 여부 + trace 독립 검색 필요성

## Q4. TraceQL이 해결하는 문제와 한계를 설명하라.

**핵심 포인트**:
- trace_id를 모르는 상태에서 span 속성 기반 탐색 가능
- 스캔 기반이므로 대량 데이터에서 인덱스 검색보다 느릴 수 있음
- 실무에서는 Loki 연동 탐색의 보완 수단으로 사용하는 경우가 많음

## Q5. Span, Trace, Attributes의 관계를 실제 요청 예시로 설명하라.

**핵심 포인트**:
- Span = 개별 작업 단위(HTTP 호출, DB 쿼리), 부모-자식 관계로 연결
- Trace = 같은 trace_id를 공유하는 span들의 집합 = 하나의 요청 경로
- Attributes = span에 붙는 키-값 속성, 풍부할수록 탐색이 쉬움

## Q6. Observability 스택에서 메트릭→로그→트레이스 탐색 흐름을 장애 시나리오로 설명하라.

**핵심 포인트**:
- 메트릭 알림(P95 상승) → Loki 에러 로그 조회 → trace_id 추출 → Tempo에서 병목 span 식별
- 세 신호가 trace_id로 연결되지 않으면 각각 따로 봐야 함 → 분석 시간 증가
- Tempo는 이 연결의 마지막 조각을 채우는 역할

## Q7. Tempo가 적합한 환경과 부적합한 환경을 구체적으로 구분하라.

**핵심 포인트**:
- 적합: Grafana 스택 사용자, 비용 민감 대규모 환경, trace 열람이 주 패턴
- 부적합: trace 독립 검색이 핵심(Jaeger 유리), 비-Grafana 환경
- 핵심 판단: "이미 Grafana 스택을 쓰는가?" + "trace 검색 패턴이 무엇인가?"
