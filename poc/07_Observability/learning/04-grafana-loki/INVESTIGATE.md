# Ch04. Grafana Loki - 점검 질문

## Q1. ELK 스택의 로그 전문 인덱싱이 대규모 환경에서 어떤 문제를 만드는가?

**핵심 포인트**:
- 인덱스가 원본만큼 커져서 저장 비용 폭발
- 샤드 관리, JVM 튜닝 등 운영 복잡도
- 실제 운영 질문 대부분은 서비스명+심각도 필터인데, 전문 인덱싱은 그에 비해 과잉 투자

## Q2. Loki가 라벨만 인덱싱하는 설계를 택한 이유를 비용과 운영 관점에서 설명하라.

**핵심 포인트**:
- 인덱스 크기를 극적으로 줄여 저장 비용 절감
- 객체 스토리지 백엔드로 상태 관리 부담 최소화
- Prometheus 라벨 모델에서 영감 — K8s 환경과 자연스러운 결합

## Q3. 스트림, 청크, 인덱스의 관계를 쿼리 실행 흐름으로 설명하라.

**핵심 포인트**:
- 동일 라벨 조합 = 하나의 스트림, 스트림의 로그가 청크에 압축 저장
- 인덱스는 라벨→청크 매핑만 보관
- 쿼리: 라벨로 스트림 특정 → 해당 청크 읽기 → 본문 순차 스캔

## Q4. 고카디널리티 라벨이 왜 Loki의 핵심 전제를 무너뜨리는가?

**핵심 포인트**:
- request_id 같은 값은 요청마다 새 스트림 생성 → 스트림 수 폭발
- 인덱스가 커져서 "인덱스를 작게 유지한다"는 전제 붕괴
- 대안: Structured Metadata 또는 로그 본문에 남기고 LogQL 런타임 파싱으로 필터

## Q5. LogQL의 Stream Selector와 Pipeline이 쿼리 성능에 미치는 영향을 설명하라.

**핵심 포인트**:
- Stream Selector가 인덱스를 타고 대상 청크를 좁힘 → 좁을수록 빠름
- Pipeline은 청크 내 본문을 순차 스캔 → 스캔 범위가 넓으면 느림
- 라벨 설계가 곧 쿼리 성능을 결정하는 이유

## Q6. Loki가 적합한 환경과 부적합한 환경을 구체적 예시와 함께 설명하라.

**핵심 포인트**:
- 적합: K8s/마이크로서비스(라벨 모델 일치), 비용 민감(객체 스토리지), Grafana 스택(LGTM 상관분석)
- 부적합: SIEM(넓은 범위 전문 검색), 비정형 로그 복잡 집계
- 핵심: "어떤 질문을 로그에 자주 던지는가"로 도구 선택이 달라져야 함

## Q7. Observability 스택에서 Loki의 역할을 장애 분석 워크플로우로 설명하라.

**핵심 포인트**:
- 메트릭 알림 → Loki 에러 로그 조회 → trace_id 추출 → Tempo에서 병목 span 식별
- Loki는 "무슨 일이 있었는가?"를 좁히는 출발점
- 세 신호(메트릭/로그/트레이스) 연결이 장애 분석 시간을 단축
