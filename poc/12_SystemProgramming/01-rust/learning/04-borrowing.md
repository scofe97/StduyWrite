# 04. 참조와 빌림(Borrowing)

빌림(Borrowing)은 소유권을 이전하지 않고 값에 접근하는 방법입니다. 불변 참조(&T)는 여러 개 동시에 존재할 수 있지만, 가변 참조(&mut T)는 하나만 허용됩니다. 이 규칙은 컴파일 타임에 데이터 레이스를 방지합니다. 슬라이스는 컬렉션의 일부에 대한 참조로, 소유권 없이 데이터에 접근할 수 있게 해줍니다.

## 목표

이 챕터를 완료하면 다음을 할 수 있어야 합니다:

- [ ] 불변 참조(&T)와 가변 참조(&mut T)의 차이를 설명하고 사용할 수 있다
- [ ] 빌림 규칙(1 mut OR many immut)을 설명하고, 위반 시 컴파일 에러를 해결할 수 있다
- [ ] 댕글링 참조(dangling reference)가 불가능한 이유를 설명할 수 있다

- [ ] NLL(Non-Lexical Lifetimes)이 무엇인지 설명할 수 있다

## 명령어 요약

| 개념 | 설명 |
|------|------|
| `&T` | 불변 참조 (읽기 전용) |
| `&mut T` | 가변 참조 (읽기/쓰기) |
| **빌림 규칙** | 1) 하나의 가변 참조 OR 여러 불변 참조 2) 참조는 항상 유효해야 함 |
| `let r = &s;` | s를 불변으로 빌림 |
| `let r = &mut s;` | s를 가변으로 빌림 (s는 mut이어야 함) |
| `&[T]` | 슬라이스 (배열/벡터의 일부에 대한 참조) |
| `&str` | 문자열 슬라이스 |
| `&s[0..5]` | 인덱스 0부터 4까지의 슬라이스 |
| `&s[..]` | 전체 슬라이스 |
| **NLL** | Non-Lexical Lifetimes, 참조의 실제 사용 범위 기준으로 빌림 검사 |

## 체크포인트

1. **동시에 &mut와 &를 가질 수 없는 이유는?**
   - 데이터 레이스 방지: &mut로 수정 중일 때 &로 읽으면 일관성 없는 데이터를 읽을 수 있음
   - 컴파일 타임에 동시 접근을 막아서 런타임 에러 방지

2. **NLL(Non-Lexical Lifetimes)이란?**
   - Rust 2018부터 도입된 빌림 검사 개선
   - 이전: 참조의 스코프(중괄호) 끝까지 빌림 유지
   - 이후: 참조의 마지막 사용 시점까지만 빌림 유지 (더 유연함)

3. **Java 참조와 Rust 참조의 근본 차이는?**
   - Java: 모든 객체 참조는 가변 가능, 동시 접근 제한 없음 (런타임에 문제 발생 가능)
   - Rust: &와 &mut 구분, 컴파일 타임에 동시 접근 차단, 소유권 시스템과 통합

4. **다음 코드가 컴파일되는 이유는?**
   ```rust
   let mut s = String::from("hello");
   let r1 = &s;
   let r2 = &s;
   println!("{} {}", r1, r2);
   let r3 = &mut s; // OK
   ```
   - NLL 덕분: r1, r2는 println! 이후 사용되지 않으므로, r3 생성 시점에는 빌림이 끝남

5. **함수가 참조를 반환할 때 주의할 점은?**
   - 로컬 변수의 참조를 반환하면 댕글링 참조 발생 (컴파일 에러)
   - 매개변수로 받은 참조를 반환하거나, 힙에 할당된 데이터(Box 등)를 반환해야 함

6. **&String과 &str의 차이는?**
   - &String: String에 대한 참조, String 메서드 사용 가능
   - &str: 문자열 슬라이스, 더 범용적 (String도 &str로 강제 변환 가능)
   - 매개변수는 &str을 선호 (유연성)

7. **슬라이스는 소유권을 가지는가?**
   - 아니오, 슬라이스는 참조의 일종으로 소유권이 없음
   - 원본 데이터가 변경되거나 해제되면 슬라이스도 무효화됨
