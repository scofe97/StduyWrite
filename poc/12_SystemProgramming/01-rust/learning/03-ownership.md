# 03. 소유권(Ownership)

소유권은 Rust의 가장 독특한 기능으로, 가비지 컬렉터 없이 메모리 안전성을 보장합니다. 모든 값은 정확히 하나의 소유자를 가지며, 소유자가 스코프를 벗어나면 값이 자동으로 해제됩니다. 이동 의미론(Move Semantics)은 소유권 이전을 명시적으로 만들어 데이터 레이스와 이중 해제를 컴파일 타임에 방지합니다.

## 명령어 요약

| 개념 | 설명 |
|------|------|
| **소유권 규칙** | 1) 각 값은 정확히 하나의 소유자를 가짐 2) 소유자가 스코프를 벗어나면 값이 drop됨 3) 한 번에 하나의 소유자만 가능 |
| `let s2 = s1;` | 소유권 이동 (s1은 무효화) |
| `let s2 = s1.clone();` | 깊은 복사 (힙 데이터 복제) |
| `Copy` 트레이트 | 스택 데이터 복사 (i32, bool 등), 이동 대신 복사 발생 |
| `Clone` 트레이트 | 명시적 깊은 복사, `.clone()` 호출 필요 |
| `Drop` 트레이트 | 소유자가 스코프를 벗어날 때 자동 호출 (RAII) |
| `String::from("hello")` | 힙 할당 문자열, 소유권 이동 |
| `"hello"` | 문자열 리터럴, 타입은 `&str`, 바이너리에 하드코딩 |
| `&str` | 문자열 슬라이스, 소유권 없음, 불변 참조 |

## 체크포인트

1. **let s1 = String::from("hello"); let s2 = s1; 후 s1 사용 시 에러 이유는?**
   - s1에서 s2로 소유권이 이동했기 때문. Rust는 이중 해제(double free)를 방지하기 위해 s1을 무효화함

2. **i32는 왜 이동이 아닌 복사가 발생하는가?**
   - i32는 Copy 트레이트를 구현함. 스택에만 존재하고 크기가 작아 복사 비용이 저렴하기 때문

3. **Drop 트레이트와 Copy 트레이트를 동시에 구현할 수 없는 이유는?**
   - Copy는 비트 단위 복사를 의미하므로, 복사 후 둘 다 drop되면 이중 해제 발생
   - Drop이 필요하다는 것은 힙 리소스 등 특별한 정리가 필요하다는 의미이므로 단순 복사 불가

4. **String은 왜 이동이 발생하는가?**
   - String은 힙에 데이터를 저장함. 포인터+길이+용량이 스택에 있고, 실제 문자열은 힙에 있음
   - 복사 시 힙 데이터까지 복사하는 것은 비용이 크므로, 기본적으로 이동(포인터만 복사 후 원본 무효화)

5. **&str은 왜 Copy인가?**
   - &str은 참조(포인터+길이)일 뿐, 소유권이 없음. 참조 자체는 스택에 있고 크기가 작으므로 Copy 가능

6. **함수에 String을 전달하면 소유권은?**
   - 소유권이 함수로 이동. 함수 종료 시 drop됨
   - 소유권을 유지하려면 참조(&String 또는 &str) 전달하거나, 반환받아야 함

7. **String과 &str 중 어떤 것을 매개변수로 받아야 하는가?**
   - 일반적으로 &str (더 유연함, String도 &str로 변환 가능)
   - 소유권이 필요하거나 문자열을 수정해야 하면 String
