# 05. 라이프타임(Lifetimes)

라이프타임은 참조가 유효한 범위를 나타내는 제네릭 파라미터입니다. 컴파일러는 대부분의 경우 라이프타임을 추론하지만, 참조 간의 관계가 모호할 때는 명시적 어노테이션이 필요합니다. 라이프타임은 댕글링 참조를 방지하고, 참조가 항상 유효한 데이터를 가리키도록 보장합니다.

## 명령어 요약

| 개념 | 설명 |
|------|------|
| `'a` | 라이프타임 파라미터 (제네릭처럼 사용) |
| `fn foo<'a>(x: &'a str) -> &'a str` | 입력과 출력 참조의 라이프타임 연결 |
| `struct S<'a> { field: &'a str }` | 구조체 필드의 라이프타임 |
| `'static` | 프로그램 전체 실행 시간 동안 유효한 라이프타임 |
| **생략 규칙 1** | 각 참조 매개변수는 고유한 라이프타임을 가짐 |
| **생략 규칙 2** | 입력 라이프타임이 하나면, 모든 출력 참조에 할당 |
| **생략 규칙 3** | 입력에 &self 또는 &mut self가 있으면, self의 라이프타임을 모든 출력에 할당 |

## 체크포인트

1. **fn longest(x: &str, y: &str) -> &str 이 컴파일 실패하는 이유는?**
   - 컴파일러가 반환값의 라이프타임을 추론할 수 없음 (x와 y 중 어느 것의 라이프타임인지 모호)
   - 해결: `fn longest<'a>(x: &'a str, y: &'a str) -> &'a str`

2. **라이프타임 생략 규칙 3가지는?**
   1. 각 참조 매개변수는 고유한 라이프타임 파라미터를 가짐
   2. 입력 라이프타임이 정확히 하나면, 그 라이프타임이 모든 출력 참조에 할당됨
   3. 입력에 &self 또는 &mut self가 있으면, self의 라이프타임이 모든 출력 참조에 할당됨

3. **'static은 언제 사용하는가?**
   - 문자열 리터럴 (`"hello"`): 바이너리에 하드코딩되어 프로그램 전체 동안 유효
   - 전역 상수 (`const`, `static`)
   - 소유권이 있는 데이터를 반환할 때 (예: `Box::leak()`)
   - 주의: 'static을 남용하면 메모리 누수 가능

4. **라이프타임은 실행 시간에 영향을 주는가?**
   - 아니오, 라이프타임은 컴파일 타임 개념으로 런타임 오버헤드가 전혀 없음
   - 제로 비용 추상화(zero-cost abstraction)의 예

5. **다음 코드에서 라이프타임 어노테이션이 필요한가?**
   ```rust
   fn first_word(s: &str) -> &str {
       &s[..1]
   }
   ```
   - 필요 없음. 생략 규칙 2번 적용 (입력 라이프타임이 하나이므로 출력에 자동 할당)

6. **구조체에 참조 필드가 있으면 왜 라이프타임이 필요한가?**
   - 구조체 인스턴스가 참조하는 데이터보다 오래 살 수 없도록 보장하기 위해
   - 예: `struct S<'a> { x: &'a str }` → S 인스턴스는 x가 가리키는 데이터보다 오래 살 수 없음

7. **여러 라이프타임 파라미터('a, 'b)는 언제 사용하는가?**
   - 입력 참조들의 라이프타임이 독립적일 때
   - 예: `fn foo<'a, 'b>(x: &'a str, y: &'b str) -> &'a str` → 반환값은 x의 라이프타임만 따름

8. **라이프타임과 제네릭의 차이는?**
   - 제네릭: 타입 파라미터 (`T`), 다양한 타입 지원
   - 라이프타임: 참조의 유효 범위 파라미터 (`'a`), 안전성 보장
   - 둘 다 컴파일 타임에 단형화(monomorphization)됨
