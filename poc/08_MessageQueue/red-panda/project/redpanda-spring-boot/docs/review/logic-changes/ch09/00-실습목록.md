## 진행 순서 및 이유

| 순서 | 실습 | 이유 |
|------|------|------|
| 1 | 인메모리 상태 저장소 | DB 없이 상태를 관리하는 기반 — 나머지 모든 실습이 StateStore를 사용 |
| 2 | 상태 변경 이벤트 발행 | 상태 변경을 토픽에 기록해야 복구와 감사 추적이 가능 — Event Sourcing의 핵심 |
| 3 | 상태 복구 Consumer | 인메모리의 약점(재시작 시 손실)을 토픽 재생으로 해결 — 실용적 완성 |
| 4 | Log Compaction 토픽 설정 | 복구용 토픽이 무한 성장하지 않도록 키 기반 압축 설정 — 운영 필수 |
| 5 | 스냅샷 전략 | 이벤트가 많을 때 전체 재생 시간을 줄이는 최적화 — 대규모 시스템 대비 |

## 각 실습의 구현 범위

### 1. 인메모리 상태 저장소
- `StateStore.java`: `ConcurrentHashMap<String, StateChangeEvent>` 기반 상태 저장
- `get(dmndId)`, `put(dmndId, state)`, `remove(dmndId)` 메서드 구현
- `putIfAbsent()` 활용으로 중복 초기화 방지 (원자 연산)
- `@Scheduled` 기반 TTL 정리 — COMPLETE/FAILURE 상태 24시간 후 자동 제거

### 2. 상태 변경 이벤트 발행
- `StateChangeEvent.java`: dmndId, fromStatus, toStatus, reason, timestamp 모델
- `StateChangeProducer.java`: `KafkaTemplate.send("pipeline.status.update", dmndId, event)` 구현
- `src/main/avro/ch09/StateChangeEvent.avsc`: Avro 스키마 (dmndId, fromStatus, toStatus, reason, timestamp)
- `Ch09TopicConfig.java`: `pipeline.status.update` 토픽 설정 (cleanup.policy 포함)
- StateStore 업데이트 직후 토픽 발행 — 인메모리와 이벤트 로그 동기화

### 3. 상태 복구 Consumer
- `StateRecoveryConsumer.java`: `@PostConstruct`에서 토픽 처음부터 재생 (`seekToBeginning`)
- `ConsumerRecord` 순서대로 `StateStore.put()` 호출하여 상태 복원
- 재생 완료 후 Consumer 종료 — 복구 전용 Consumer와 운영 Consumer 분리
- 복구 완료 후 서비스 정상 동작 시작 (`ReadinessIndicator` 연동)

### 4. Log Compaction 토픽 설정
- `CompactedTopicConfig.java`: `cleanup.policy=compact` 설정한 NewTopic 빈
- `min.cleanable.dirty.ratio=0.1`: 압축 주기 설정
- `delete.retention.ms`: 삭제된 키 보관 기간 설정
- 동일 dmndId의 최신 StateChangeEvent만 보관 → 토픽 크기 제어

### 5. 스냅샷 전략
- `SnapshotStrategy.java`: 주기적으로 StateStore 전체 상태를 별도 저장소에 직렬화
- 스냅샷 저장 시 현재 토픽 offset을 checkpoint로 함께 저장
- 재시작 시 최신 스냅샷 로드 → checkpoint offset 이후 이벤트만 재생
- 스냅샷 저장 실패 시 전체 토픽 재생으로 폴백 (안전성 우선)
