## 진행 순서 및 이유

| 순서 | 실습 | 이유 |
|------|------|------|
| 1 | 파티션 전략 | Consumer Group이 파티션을 어떻게 나누는지 이해해야 나머지 실습의 동작이 설명됨 |
| 2 | 커스텀 파티셔너 | 기본 전략을 이해한 뒤 특정 키를 특정 파티션으로 고정하는 커스터마이징 학습 |
| 3 | 순서 보장 Consumer | 파티션 할당 방식을 알아야 "같은 파티션 내 순서 보장"의 의미를 정확히 파악 가능 |
| 4 | Cooperative Sticky 리밸런싱 | Consumer 설정을 완성한 뒤, 리밸런싱 전략을 변경해 가용성 차이를 체험 |
| 5 | 순서 보장 데모 + 테스트 | 앞 4개 실습의 결과물을 하나의 시나리오로 묶어 "Consumer 다운 → 복구" 전체 흐름 검증 |

## 각 실습의 구현 범위

### 1. 파티션 전략 (`PartitionStrategy`)

- `PartitionStrategy.java`에서 RoundRobin, Range, Sticky 세 가지 할당 전략의 동작 방식을 설명하는 TODO 구현
- 파티션 수 > Consumer 수, 파티션 수 = Consumer 수, 파티션 수 < Consumer 수 각 케이스에서 할당 결과를 로그로 출력
- `Ch11TopicConfig.java`에서 토픽 파티션 수(3개) 설정 확인 및 의도 주석 작성

### 2. 커스텀 파티셔너 (`CustomPartitioner`)

- `CustomPartitioner.java`에서 메시지 키(예: orderId)를 해싱하여 특정 파티션으로 라우팅하는 로직 구현
- 동일 키는 항상 동일 파티션으로 가는지 검증하는 단순 출력 또는 assert 작성
- `Ch11TopicConfig.java`의 `ProducerConfig.PARTITIONER_CLASS_CONFIG` 설정과 연동 확인

### 3. 순서 보장 Consumer (`SinglePartitionConsumer`)

- `SinglePartitionConsumer.java`에서 `@KafkaListener`에 특정 파티션(topicPartitions)을 고정 지정하는 방식 구현
- `group.id` + `partition` 고정 조합으로 "하나의 Consumer가 하나의 파티션만 전담"하는 패턴 작성
- 수신 메시지의 offset을 로그로 출력하여 순서가 보장되는지 확인

### 4. Cooperative Sticky 리밸런싱 (`CooperativeStickyConfig`)

- `CooperativeStickyConfig.java`에서 `partition.assignment.strategy = CooperativeStickyAssignor` 설정 구현
- `session.timeout.ms = 6000`, `heartbeat.interval.ms = 2000` 설정으로 빠른 장애 감지 구성
- Eager 리밸런싱(기본값)과 Cooperative 리밸런싱의 동작 차이를 주석으로 비교 설명

### 5. 순서 보장 데모 + 테스트 (`OrderGuaranteeDemo`, `Ch11PartitionTest`)

- `OrderGuaranteeDemo.java`에서 Producer가 동일 키 메시지 10개를 발행하고, Consumer가 파티션 순서대로 수신하는 시나리오 실행
- Consumer 강제 중단 → 리밸런싱 → 재시작 후 마지막 커밋 Offset부터 재개되는 흐름 로그로 확인
- `Ch11PartitionTest.java`에서 파티션 배분, 순서 보장, 리밸런싱 후 메시지 유실 없음을 assert로 검증
