## 각 실습의 구현 범위

### 1. 토픽 설계 (우선순위별 5개 토픽)
- `Ch05TopicConfig.java`에 5개 토픽 빈 정의: `cspnf.retry`, `timeout.process`, `fail.retry`, `rspnd.process`, `pndng.process`
- 각 토픽의 파티션 수를 우선순위에 따라 다르게 설정 (CSPNF: 5개, TIMEOUT: 2개, FAIL: 3개, RSPND: 2개, PNDNG: 5개)
- `StatusChangeEvent.avsc` 스키마 확인 (messageId, status, eventType, timestamp, retryCount 필드)
- retention.ms를 토픽 중요도에 따라 차등 설정 (CSPNF: 24시간, PNDNG: 48시간)

### 2. 상태 변경 이벤트 발행
- `StatusChangeProducer.java`에 `publishStatusEvent(messageId, status)` 메서드 구현
- 상태값(CSPNF/TIMEOUT/FAIL/RSPND/PNDNG)에 따라 적절한 토픽으로 라우팅하는 `determineTopicByStatus()` 구현
- `kafkaTemplate.send(topic, messageId, event)` 호출 시 messageId를 파티션 키로 사용
- `whenComplete` 콜백으로 발행 성공/실패 로깅

### 3. @KafkaListener 이벤트 소비
- `StatusChangeListener.java`에 5개 @KafkaListener 메서드 구현 (각 토픽별 독립 Consumer)
- 각 Consumer에 적절한 concurrency 설정 (CSPNF: 3, TIMEOUT: 2, FAIL: 2, RSPND: 2, PNDNG: 3)
- `AckMode.MANUAL`로 수동 오프셋 커밋 (`Acknowledgment.acknowledge()`)
- 처리 성공 후에만 acknowledge 호출하여 at-least-once 보장

### 4. 멱등 처리 + DLT
- `IdempotentStatusProcessor.java`에 현재 메시지 상태를 확인하고 이미 처리된 경우 스킵하는 로직 구현
- 상태 확인 패턴: `if (expectedStatus.equals(message.getStatus())) { /* 처리 */ }` — 중복 호출 시 조건 불만족으로 안전하게 스킵
- `RetryWithDltConfig.java`에 `DefaultErrorHandler` 설정: 재시도 간격(1초), 최대 재시도 횟수(3회) 이후 DLT 전송
- DLT 토픽 네이밍 규칙 확인 (예: `cspnf.retry.DLT`)

### 5. 통합 테스트
- `Ch05SchedulerToEventTest.java`에서 `StatusChangeProducer`로 이벤트 발행 후 `StatusChangeListener`가 처리하는 전체 흐름 검증
- 멱등성 테스트: 동일한 messageId로 이벤트 2회 발행 → 처리 결과가 동일한지 확인
- Consumer Lag 확인: 발행 후 일정 시간 내 Lag이 0으로 수렴하는지 검증
- `@EmbeddedKafka` 또는 `@ActiveProfiles("local")` + docker-compose 환경에서 실행
