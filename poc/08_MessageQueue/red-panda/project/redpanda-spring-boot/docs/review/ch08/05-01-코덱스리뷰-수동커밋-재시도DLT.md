## 1. 수동 커밋 (AckMode.MANUAL) 전환

### 변경 내용

| 파일 | 변경 |
|------|------|
| `SagaKafkaConfig` | `AckMode.RECORD` → `AckMode.MANUAL` |
| `InventoryService` | 3개 리스너에 `Acknowledgment ack` 파라미터 + `ack.acknowledge()` |
| `PaymentService` | 2개 리스너에 `Acknowledgment ack` + 멱등성 체크 후 ack |
| `ShippingService` | 1개 리스너에 `Acknowledgment ack` |
| `OrderService` | 4개 리스너에 `Acknowledgment ack` |

### ack 시점 설계 원칙

| 리스너 유형 | ack 시점 | 이유 |
|-------------|----------|------|
| **정방향 (forward-flow)** | try/catch 양쪽에서 ack | 실패 이벤트를 발행했으므로 "처리 완료" |
| **상태 업데이트** | 메서드 끝에서 ack | DB 업데이트 성공 = 처리 완료 |
| **보상 (compensation)** | 성공 시만 ack, 실패 시 throw | 보상은 반드시 성공해야 → 재전달 |

```java
// 보상 리스너 패턴
try {
    releaseInventory(event.orderId(), event.correlationId());
    ack.acknowledge();  // 보상 성공 → 수동 커밋
} catch (Exception e) {
    // ack 하지 않음 → 트랜잭션 abort → 메시지 재전달
    throw e;  // → DefaultErrorHandler가 재시도 → DLT
}
```

## 2. DefaultErrorHandler + Dead Letter Topic (DLT)

### 왜 @RetryableTopic이 아닌가

**핵심 발견: `@RetryableTopic` + `KafkaTransactionManager` = 공식 비호환**

| 항목 | @RetryableTopic (논블로킹) | DefaultErrorHandler (블로킹) |
|------|---------------------------|------------------------------|
| 동작 방식 | 별도 retry 토픽 + 별도 Consumer 컨테이너 | 같은 스레드에서 즉시 재시도 |
| Kafka TX 호환 | 불가 — 별도 컨테이너가 TX 참여 불가 | 호환 — 같은 컨테이너 안에서 재시도 |
| 순서 보장 | 불가 — retry 토픽에서 순서 깨짐 | 보장 — 블로킹이므로 순서 유지 |
| SAGA 적합성 | 부적합 | **적합** |

Spring Kafka 공식 문서에서 "Non-blocking retries CANNOT combine with Container Transactions" 명시.

### 구현

```java
// SagaKafkaConfig.java
@Bean("ch03ErrorHandler")
public DefaultErrorHandler ch03ErrorHandler(
        @Qualifier("ch03KafkaTemplate") KafkaTemplate<String, Object> kafkaTemplate) {
    DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(kafkaTemplate);
    FixedBackOff backOff = new FixedBackOff(1_000L, 3L);  // 3회 재시도, 1초 간격

    DefaultErrorHandler errorHandler = new DefaultErrorHandler(recoverer, backOff);
    errorHandler.addNotRetryableExceptions(IllegalStateException.class);
    return errorHandler;
}
```

### DLT 토픽 (5개)

| DLT 토픽 | 원본 토픽 | 리스너 유형 |
|----------|----------|------------|
| `chapter3.payment-failed.DLT` | payment-failed | 보상 (InventoryService) + 상태 (OrderService) |
| `chapter3.shipping-failed.DLT` | shipping-failed | 보상 (PaymentService) + 상태 (OrderService) |
| `chapter3.payment-refunded.DLT` | payment-refunded | 보상 (InventoryService) |
| `chapter3.shipping-requested.DLT` | shipping-requested | 상태 (OrderService) |
| `chapter3.inventory-reservation-failed.DLT` | inventory-reservation-failed | 상태 (OrderService) |

### 재시도 설계 원칙

| 원칙 | 설명 |
|------|------|
| **모든 컨슈머에 retry+DLT 불필요** | 비즈니스 중요도와 오류 유형으로 판단 |
| **재시도 대상 제외** | `IllegalStateException` (비즈니스 규칙 위반) — 재시도해도 결과 동일 |
| **보상 > 정방향** | 보상 실패 = 데이터 불일치 → 더 많은 재시도 + 엄격한 DLT 알림 (프로덕션) |
| **Exponential Backoff + Jitter** | Thundering Herd 방지 (프로덕션 권장) |

### 프로덕션 권장 패턴

```
현재 (학습용):
  모든 리스너 → 단일 ContainerFactory → FixedBackOff(1s, 3회) → DLT

프로덕션 권장:
  정방향 리스너 → forwardContainerFactory → FixedBackOff(1s, 3회) → DLT
  보상 리스너   → compensationContainerFactory → ExponentialBackOff(10회, jitter) → DLT + 알림
  DLT Consumer → Slack/PagerDuty 알림 + 수동 재처리 대시보드
```

## 2-1. @RetryableTopic은 언제 쓰는가? (비블로킹 재시도)

ch03에서 `@RetryableTopic`이 Kafka 트랜잭션과 비호환임을 확인했다. 그렇다면 이 기능은 어떤 상황에 적합한가?

### @RetryableTopic vs DefaultErrorHandler 선택 기준

| 기준 | @RetryableTopic (비블로킹) | DefaultErrorHandler (블로킹) |
|------|---------------------------|------------------------------|
| **처리 방식** | 실패 메시지 → retry topic 이동, 파티션 계속 소비 | 실패 메시지 같은 스레드에서 재처리, 파티션 블록 |
| **처리량 영향** | 거의 없음 | 80% 감소 가능 |
| **메시지 순서** | 보장 안 됨 (나중 메시지가 먼저 처리 가능) | 파티션 내 순서 보장 |
| **Kafka 트랜잭션** | **호환 불가** (공식 문서 명시) | 호환됨 |
| **Batch Listener** | **호환 불가** | 호환됨 |
| **재시도 간격** | 수십 초 ~ 수 분 (long backoff 적합) | 수백 ms ~ 수 초 (short backoff) |
| **DLT 지원** | `@DltHandler` 내장 | `DeadLetterPublishingRecoverer` 별도 구성 |
| **토픽 생성** | retry topic 자동 생성 필요 | 불필요 |
| **설정 난이도** | 애노테이션 기반, 간단 | 빈 설정 필요, 중간 |

### 선택 체크리스트 (순서대로 질문)

```
Q1. KafkaTransactionManager(Exactly-Once)를 사용하는가?
    → YES: DefaultErrorHandler 필수 ★ ch03은 여기에 해당

Q2. Batch Listener를 사용하는가?
    → YES: DefaultErrorHandler 필수

Q3. 같은 파티션 내 메시지 간에 비즈니스 순서 의존성이 있는가?
    (예: 계좌 입금→출금 순서, 상태머신 전이 순서)
    → YES: DefaultErrorHandler 권장

Q4. 실패 원인이 모든 메시지에 공통적인가? (DB 전체 다운 등)
    → YES: DefaultErrorHandler 권장 (비블로킹의 이점 없음)

Q5. 재시도 간격이 수 분 이상의 긴 backoff가 필요한가?
    (예: rate limit 429, 외부 API 일시적 장애)
    → YES: @RetryableTopic 강력 권장

Q6. 파티션 내 메시지들이 서로 독립적으로 처리 가능한가?
    → YES + 높은 처리량 필요: @RetryableTopic 적합
```

### @RetryableTopic이 적합한 실제 시나리오 4가지

**A. 알림/이메일 시스템**
```
시나리오: 주문 완료 → 이메일 발송, SMTP 일시적 오류로 실패
특징: 고객 A 이메일 실패가 고객 B 이메일을 막으면 안 됨
결론: 각 메시지가 독립적 + 순서 무관 → @RetryableTopic
```

**B. 이벤트 도착 순서 불일치 (핵심 사용 사례)**
```
시나리오: user-profile-updated가 user-created보다 먼저 도착
         → DB에 아직 user가 없어서 update 실패
결론: update를 retry topic에 넣고 파티션 계속 소비
     → create 처리 후 update 재처리 시 성공
```

**C. 외부 API 연동 (rate limit)**
```
시나리오: Slack/SMS API에서 429 Too Many Requests 반환
특징: 수 분 후 재시도하면 성공, 다른 메시지는 즉시 처리 가능
결론: 긴 backoff(1분~5분) + 파티션 블록 없음 → @RetryableTopic 최적
```

**D. 독립적 재고 업데이트**
```
시나리오: 상품 A 재고 업데이트 실패 (DB timeout), 상품 B는 정상
특징: 상품 간 의존성 없음, A 실패가 B를 막으면 안 됨
결론: @RetryableTopic으로 A만 retry topic 이동, B는 즉시 처리
```

### DefaultErrorHandler가 적합한 시나리오 (ch03 포함)

| 시나리오 | 이유 |
|----------|------|
| **SAGA Choreography (ch03)** | Kafka 트랜잭션 사용 → @RetryableTopic 불가 |
| **금융 계좌 이벤트** | 입금→출금 순서 역전 시 잔액 오류 → 순서 필수 |
| **다운스트림 전체 장애** | DB 서버 다운 → 모든 메시지 동일 실패 → 비블로킹 이점 없음 |
| **Batch 처리** | 일일 정산 등 배치 → @RetryableTopic 호환 불가 |

### @RetryableTopic 실무 설정 (참고: velog/@2eunpal)

```java
@RetryableTopic(
    attempts = "5",                    // 4-5회 권장 (총 15-30초)
    backoff = @Backoff(
        delay = 1000,                  // 초기 1초
        multiplier = 2.0,              // 지수 증가: 1s → 2s → 4s → 8s
        maxDelay = 10000               // 최대 10초
    ),
    timeout = "60000",                 // 60초 초과 시 즉시 DLT
    include = {SocketTimeoutException.class},   // 재시도 대상
    exclude = {NullPointerException.class}      // 재시도 제외 (비즈니스 오류)
)
@KafkaListener(topics = "order-events")
public void processOrder(OrderEvent event) {
    orderService.process(event);
}

@DltHandler
public void handleDlt(OrderEvent event,
        @Header(KafkaHeaders.EXCEPTION_MESSAGE) String error) {
    failedEventRepository.save(event, error);  // DB 저장
    alertService.sendAlert("처리 실패", event);  // 알림 발송
}
```

**실무 황금비율**: 80%는 재시도로 자동 해결, 20%는 DLT → 수동 재처리

### Hybrid 패턴 (블로킹 + 비블로킹 조합)

Spring Kafka는 두 방식을 함께 사용하는 것도 지원한다:

```
1단계: DefaultErrorHandler로 짧은 블로킹 retry (2-3회, 수백ms)
       → 일시적 네트워크 글리치는 여기서 해결

2단계: 그래도 실패하면 @RetryableTopic으로 비블로킹 장기 retry (수 분 간격)
       → 외부 서비스 장애 시 파티션 블록 없이 대기
```

단, 이 조합도 **Kafka 트랜잭션 환경에서는 사용 불가** (2단계 @RetryableTopic이 TX 비호환이므로).

### 왜 정확히 트랜잭션과 비호환인가? (Spring Kafka GitHub #2918, #2934)

`@RetryableTopic` + `KafkaTransactionManager` 조합 시 **트랜잭션 커밋 의미가 완전히 달라진다**:

```
■ 블로킹 retry + 트랜잭션 (DefaultErrorHandler)
  리스너 예외 발생
  → 트랜잭션 롤백 (outgoing 메시지도 함께 취소)
  → 새 트랜잭션으로 DLT 전송
  → ✅ 원자성 보장: "처리 실패 = 모든 것 취소"

■ 비블로킹 retry + 트랜잭션 (@RetryableTopic)
  리스너 예외 발생
  → retry topic으로 메시지 전송 (같은 트랜잭션 안에서)
  → 트랜잭션 커밋됨 (retry 전송 + consumer offset + outgoing 메시지 모두)
  → ❌ 원자성 깨짐: "처리 실패했는데 outgoing 메시지는 커밋됨"
```

**핵심 문제**: 비블로킹 retry는 예외를 `DeadLetterPublishingRecoverer`가 catch하여 retry topic으로 보내고 "성공"으로 처리한다. 컨테이너 입장에서는 정상 반환이므로 트랜잭션이 커밋된다. 따라서 리스너 안에서 `kafkaTemplate.send()`로 보낸 다른 메시지들도 함께 커밋되어, **실패한 처리의 부분 결과가 영구 저장**된다.

SAGA 예시로 설명하면:
```
InventoryService.onOrderCreated 실행 중:
  1. 재고 예약 (DB) ← JPA TX
  2. kafkaTemplate.send("inventory-reserved") ← Kafka TX
  3. 외부 API 호출 중 예외 발생!

블로킹 retry: TX 롤백 → #2의 "inventory-reserved" 메시지도 취소 ✅
비블로킹 retry: TX 커밋 → #2의 "inventory-reserved" 메시지가 전달됨 ❌
                 → PaymentService가 결제 진행 → 하지만 재고 예약은 실패 → 불일치!
```

이것이 Spring Kafka 팀이 "Non-Blocking Retries CANNOT combine with Container Transactions"라고 명시한 이유다.

**결론**: 트랜잭션이 필요하면 `DefaultErrorHandler` (블로킹), 트랜잭션 없이 높은 처리량이 필요하면 `@RetryableTopic` (비블로킹). 두 요구사항을 동시에 만족하는 방법은 없다.

| 멱등성 | 같은 이벤트 2번 전달 시 중복 보상 방지 |
| 부분 재고 실패 | 2개 상품 중 1개 실패 → 성공한 1개 롤백 |
| 경계값 | available=2, 주문=2 (정확히 맞는 경우) |

## 핵심 학습 포인트

1. **`@RetryableTopic` + `KafkaTransactionManager` = 비호환** — 논블로킹 재시도는 별도 컨테이너를 생성하므로 Kafka 트랜잭션에 참여할 수 없다. 트랜잭션 환경에서는 `DefaultErrorHandler`(블로킹 재시도)가 유일한 선택
2. **ProducerFactory는 반드시 같은 인스턴스 공유** — KafkaTemplate과 KafkaTransactionManager가 다른 ProducerFactory를 사용하면 트랜잭션이 격리되어 exactly-once 깨짐. `@Qualifier` 파라미터 주입으로 보장
3. **재시도 설계는 비즈니스 중요도 기반** — 모든 컨슈머에 동일한 retry+DLT 불필요. 보상 리스너는 더 많은 재시도 + 엄격한 알림, 정방향은 이미 자체 에러 처리
4. **`addNotRetryableExceptions`로 불필요한 재시도 방지** — 비즈니스 규칙 위반(재고 부족, 엔티티 미존재)은 재시도해도 결과 동일
5. **테스트 격리는 `@BeforeEach`에서 명시적으로** — JUnit5 테스트 순서 미보장 + Spring Context 캐싱 = 이전 테스트 부작용이 남음
