## 왜 이렇게 구현했는가

### 1. ch02와 완전 분리 (ch03 전용 빈)

ch02는 비트랜잭션 Producer를 사용한다. ch03만 트랜잭션을 쓰려면 별도의 ProducerFactory가 필요하다. `transactional.id`를 설정하면 해당 ProducerFactory의 모든 send()는 반드시 트랜잭션 컨텍스트 안에서 실행되어야 하므로, ch02와 공유할 수 없다.

### 2. KafkaTransactionManager가 필요한 이유

이 매니저 없이 transactional KafkaTemplate만 있으면:
- send()마다 개별 미니 트랜잭션 생성
- Consumer 오프셋이 Kafka TX에 포함 안 됨 → exactly-once 불완전

이 매니저가 있으면:
- 리스너가 메시지를 받을 때 Kafka TX 시작
- 리스너 내 모든 send()가 같은 TX에 포함
- 정상 반환 → TX 커밋 + sendOffsetsToTransaction (오프셋도 원자적)
- 예외 → TX abort + 오프셋 미커밋 → 재전달

### 3. isolation.level=read_committed

| 설정 | 동작 |
|------|------|
| `read_uncommitted` (기본) | abort된 메시지도 보임 → 데이터 불일치 |
| `read_committed` | commitTransaction() 완료된 메시지만 보임 → 원자성 보장 |

트랜잭션 Producer를 쓰면 Consumer도 반드시 `read_committed`로 설정해야 한다. 그렇지 않으면 abort된 메시지를 읽고 처리하는 오류가 발생한다.

### 4. @Value로 프로퍼티 주입

application.yml의 `bootstrap-servers`, `schema.registry.url`, `auto.register.schemas`를 @Value로 주입하여 프로파일별 설정이 자동 적용된다. 하드코딩하면 staging/prod에서 변경이 불가능해진다.

### 알려진 제한사항: JPA + Kafka 트랜잭션 원자성

현재 서비스의 forward-flow 리스너는 다음 패턴을 사용한다:

```java
@Transactional  // JPA TX
@KafkaListener(...)  // KafkaTransactionManager → Kafka TX
public void onEvent(...) {
    try {
        // DB 업데이트 + 성공 이벤트 send
    } catch (Exception e) {
        // 실패 이벤트 send (같은 Kafka TX)
    }
}
```

**문제**: catch 블록이 예외를 삼키면 JPA TX도 커밋된다. try 블록에서 부분 DB 쓰기가 있었다면 불일치 상태가 된다.

**프로덕션 해결법**:
1. **Transactional Outbox 패턴** — 이벤트를 DB outbox 테이블에 저장 → CDC로 Kafka에 발행 (EDA ch06에서 다룸)
2. **ChainedKafkaTransactionManager** — JPA + Kafka TX를 체인으로 연결 (Spring Kafka 지원)
3. **서비스 분리** — DB 로직과 이벤트 발행을 별도 메서드/클래스로 분리

이 PoC에서는 #4의 학습 범위에 집중하고, Outbox 패턴은 별도 챕터에서 다룬다.

### @ConditionalOnMissingBean 자동 구성 차단 이슈

ch03 `SagaKafkaConfig`가 `ConsumerFactory`, `ProducerFactory`, `KafkaTemplate`, `KafkaTransactionManager` 빈을 등록하면서 Spring Boot 자동 구성이 연쇄적으로 차단되었다. Ch09에서 발생한 것과 동일한 패턴이다.

**차단된 자동 구성 빈 (5개)**:

| 자동 구성 빈 | 차단 원인 (ch03 빈) | 영향 |
|---|---|---|
| `kafkaConsumerFactory` | ch03ConsumerFactory (ConsumerFactory) | BatchKafkaConfig 주입 실패 |
| `kafkaProducerFactory` | ch03ProducerFactory (ProducerFactory) | KafkaTemplate 생성 실패 |
| `kafkaTemplate` | ch03KafkaTemplate (KafkaTemplate) | OrderProducer, EnhancedKafkaTemplate, KafkaErrorConfig 주입 실패 |
| `kafkaListenerContainerFactory` | 위 빈들 부재로 연쇄 실패 | ch02 @KafkaListener 컨테이너 미생성 |
| `transactionManager` (JPA) | ch03KafkaTransactionManager (TransactionManager) | @Transactional 동작 불가 |

**핵심 원리**: Spring Boot 자동 구성은 `@ConditionalOnMissingBean(타입.class)`로 조건부 생성한다. 같은 타입의 빈이 **하나라도** 존재하면 자동 구성을 건너뛴다. ch03 빈의 이름이 다르더라도(`ch03ConsumerFactory` vs `kafkaConsumerFactory`), **타입이 같으면** 자동 구성이 차단된다.

**해결: `Ch02KafkaConfig.java` 신규 생성**

자동 구성이 생성했을 5개 빈을 명시적으로 생성하고 `@Primary`로 표시:

```java
@Configuration
public class Ch02KafkaConfig {
    @Bean @Primary
    public DefaultKafkaConsumerFactory<?, ?> kafkaConsumerFactory(KafkaProperties props) { ... }

    @Bean @Primary
    public DefaultKafkaProducerFactory<?, ?> kafkaProducerFactory(KafkaProperties props) { ... }

    @Bean @Primary
    public KafkaTemplate<?, ?> kafkaTemplate(...) { ... }

    @Bean @Primary
    public ConcurrentKafkaListenerContainerFactory<?, ?> kafkaListenerContainerFactory(...) { ... }

    @Bean @Primary
    public PlatformTransactionManager transactionManager(EntityManagerFactory emf) { ... }
}
```

**와일드카드(`<?, ?>`) 반환 타입이 중요한 이유**: Spring Boot 자동 구성도 `DefaultKafkaConsumerFactory<?, ?>`로 선언한다. 와일드카드 타입이면 `ConsumerFactory<String, OrderEvent>` 등 구체 타입 주입에도 매칭된다. `ConsumerFactory<String, Object>`로 선언하면 `<String, OrderEvent>`에 매칭되지 않아 주입 실패한다.

**교훈**: 멀티 챕터 프로젝트에서 챕터 전용 빈을 등록할 때는, 자동 구성을 차단하지 않는지 반드시 확인해야 한다. `debug=true`로 CONDITIONS EVALUATION REPORT를 확인하면 어떤 자동 구성이 차단됐는지 알 수 있다.

## 핵심 학습 포인트

1. **transactional.id 설정만으로 트랜잭션 활성화** — Producer에 transactional.id를 설정하면 모든 send()는 반드시 트랜잭션 안에서 실행되어야 한다
2. **KafkaTransactionManager가 exactly-once의 핵심** — 이것 없이는 오프셋과 메시지 발행이 별도로 커밋되어 중복 처리 가능
3. **read_committed는 트랜잭션 Producer의 필수 짝** — Consumer가 abort된 메시지를 읽으면 트랜잭션의 의미가 없다
4. **JPA + Kafka TX는 자동으로 원자적이지 않다** — 두 시스템의 커밋은 독립적이며, 완전한 원자성은 Outbox 패턴이나 ChainedTransactionManager가 필요하다
5. **ch02와 빈 분리는 @Qualifier로** — 같은 타입의 빈이 여러 개일 때 Spring은 이름으로 매칭을 시도하지만, 명시적 @Qualifier가 안전하다
