# P4: 헤더 포함 전송 (Custom Headers)

## 목적

`KafkaTemplate.send(topic, key, value)`는 헤더를 보낼 수 없다. `ProducerRecord`를 직접 생성하면 커스텀 헤더를 추가할 수 있다. 분산 추적(Correlation ID), 메시지 출처 식별, 라우팅 힌트 등에 사용한다.

---

## 변경 파일

### OrderProducer.java - `sendOrderWithHeaders()` 추가

```java
public CompletableFuture<SendResult<String, OrderEvent>> sendOrderWithHeaders(
        OrderEvent event, Map<String, String> headers) {
    RecordHeaders recordHeaders = new RecordHeaders();
    headers.forEach((key, value) ->
            recordHeaders.add(key, value.getBytes(StandardCharsets.UTF_8)));

    ProducerRecord<String, OrderEvent> record = new ProducerRecord<>(
            "chapter2.orders",
            null,           // partition (자동 할당)
            event.getOrderId(),  // key
            event,               // value
            recordHeaders        // headers
    );

    return kafkaTemplate.send(record)
            .whenComplete((result, ex) -> { ... });
}
```

### OrderConsumer.java - 커스텀 헤더 수신

```java
@Header(value = "X-Correlation-Id", required = false) String correlationId,
@Header(value = "X-Source", required = false) String source,
```

추가된 필드:
```java
@Getter
private String lastReceivedCorrelationId;

@Getter
private String lastReceivedSource;
```

### ProducerConsumerTest.java - 테스트 2개 추가

2. **커스텀_헤더_없이_전송하면_null로_수신된다**: `required = false` 동작 검증

## 설계 결정과 이유

### ProducerRecord를 직접 생성한 이유

`KafkaTemplate`의 편의 메서드들은 헤더를 지원하지 않는다:

```java
// 헤더 불가
kafkaTemplate.send("topic", key, value);

// 헤더 가능 - ProducerRecord 직접 생성
ProducerRecord<String, OrderEvent> record = new ProducerRecord<>(
        "topic", null, key, value, headers);
kafkaTemplate.send(record);
```

`ProducerRecord` 생성자의 `partition` 파라미터에 `null`을 전달하면 파티셔너가 key 기반으로 자동 할당한다.

### 헤더 값을 byte[]로 변환하는 이유

Kafka 헤더의 값 타입은 `byte[]`이다. String을 그대로 넣을 수 없으므로 `StandardCharsets.UTF_8`로 인코딩한다. Consumer 쪽에서 Spring이 자동으로 `byte[]` → `String` 디코딩해준다 (`@Header`가 타입 변환 처리).

### required = false를 설정한 이유

기존 메시지(P1, P2로 보낸 것)에는 커스텀 헤더가 없다. `required = true`(기본값)이면 해당 헤더가 없는 메시지를 수신할 때 `MessageHandlingException`이 발생한다.

```java
// BAD: 헤더 없는 메시지 수신 시 예외 발생
@Header("X-Correlation-Id") String correlationId

// GOOD: 헤더 없으면 null, 예외 없음
@Header(value = "X-Correlation-Id", required = false) String correlationId
```

이 Consumer는 헤더가 있는 메시지와 없는 메시지 모두 처리해야 하므로 `required = false`가 필수다.

### 헤더 네이밍 규칙: X- 접두사

`X-Correlation-Id`, `X-Source`처럼 `X-` 접두사를 사용했다. HTTP에서 유래한 관례로, **커스텀(비표준) 헤더임을 표시**하기 위해 사용한다.

단, HTTP 표준(RFC 6648, 2012)에서는 `X-` 접두사를 **더 이상 권장하지 않는다**. 비표준 헤더가 나중에 표준이 되면 이름을 바꿔야 하기 때문이다. Kafka에서는 공식 표준이 없으므로 팀/조직 내 컨벤션을 정하면 된다:

| 방식 | 예시 | 장단점 |
|------|------|--------|
| `X-` 접두사 | `X-Correlation-Id` | 직관적, 전통적 |
| 네임스페이스 | `myapp.correlation-id` | 충돌 방지, 체계적 |
| 단순 이름 | `correlation-id` | 간결, 표준화 가능 |

실무에서는 `X-` 또는 `{서비스명}.{헤더명}` 형태를 많이 사용한다.

### Kafka 기본 헤더 목록

Spring Kafka가 자동으로 설정하는 헤더:

| 헤더 | 값 | 제어 가능 |
|------|-----|----------|
| `kafka_receivedTopic` | 토픽명 | 브로커 설정 |
| `kafka_receivedPartitionId` | 파티션 번호 | 브로커 설정 |
| `kafka_offset` | 오프셋 | 브로커 설정 |
| `kafka_receivedTimestamp` | 타임스탬프 | Producer CreateTime / 브로커 LogAppendTime 선택 가능 |
| `kafka_receivedMessageKey` | 메시지 키 | Producer에서 설정 |
| `kafka_groupId` | Consumer Group ID | `@KafkaListener(groupId=...)` |
| `__TypeId__` | 역직렬화 타입 힌트 | `JsonSerializer` 사용 시 자동, Avro에선 없음 |

타임스탬프 타입은 토픽 설정 `message.timestamp.type`으로 제어:
- `CreateTime` (기본): Producer가 전송한 시점
- `LogAppendTime`: 브로커가 저장한 시점

### @Header, @Payload 외 사용 가능한 어노테이션/파라미터

| 어노테이션/타입 | 용도 |
|----------------|------|
| `@Payload` | 메시지 본문 (value) |
| `@Header("name")` | 개별 헤더 접근 |
| `@Headers` | 모든 헤더를 `Map<String, Object>`로 수신 |
| `ConsumerRecord<K,V>` | Kafka 원시 레코드 전체 (topic, partition, offset, key, value, headers 모두 포함) |
| `Acknowledgment` | 수동 ACK |
| `Consumer<K,V>` | Kafka Consumer 인스턴스 직접 접근 (pause/resume 등) |

`ConsumerRecord`를 사용하면 `@Header` 없이도 모든 메타데이터에 접근 가능하지만, 파라미터별로 분리하는 것이 가독성과 테스트에 유리하다.
