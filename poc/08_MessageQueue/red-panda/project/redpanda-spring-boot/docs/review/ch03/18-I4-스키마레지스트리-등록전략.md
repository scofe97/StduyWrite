# I4: 스키마 레지스트리 등록 전략 리뷰

## 구현 요약

| 항목 | 내용 |
|------|------|
| 목적 | 환경별 스키마 등록 방식 분리 (dev: 자동, staging/prod: CI/CD 수동) |
| 변경 파일 | `build.gradle`, `application.yml` |
| 추가 플러그인 | `com.github.imflog.kafka-schema-registry-gradle-plugin:2.1.0` |
| 새 Gradle task | `schemaRegistryTestCompatibility`, `schemaRegistryRegister` |
| 새 프로파일 | `staging` (`auto.register.schemas=false`) |

---

## 왜 이렇게 구현했는가

### 자동 등록의 동작 원리

현재 프로젝트의 스키마는 **Gradle Plugin이 아니라 `KafkaAvroSerializer`가 메시지 전송 시점에 자동 등록**한 것이다.

```
테스트 실행 or 앱 기동
→ Producer가 OrderEvent 메시지 전송
→ KafkaAvroSerializer가 직렬화 시도
→ "이 스키마가 Registry에 있나?" 확인
→ 없으면 자동 등록 (HTTP POST to /subjects/{name}/versions)
→ 있으면 스키마 ID만 조회
→ 메시지 앞에 [magic byte(1) + 스키마 ID(4바이트)] 붙여서 전송
```

즉 `auto.register.schemas=true` 상태에서 테스트를 한 번이라도 실행하면 이미 Registry에 스키마가 들어가 있다. **등록 주체가 애플리케이션(Producer)**이라는 것이 핵심이다.

### 왜 수동 등록(CI/CD)이 필요한가

자동 등록은 **"아무 Producer나 스키마를 등록할 수 있다"**는 뜻이다. 개발에는 편리하지만 프로덕션에서는 위험하다.

| | 자동 등록 (KafkaAvroSerializer) | 수동 등록 (Gradle Plugin / CI) |
|---|---|---|
| **등록 주체** | 모든 Producer 인스턴스 | CI/CD 파이프라인만 |
| **등록 시점** | 메시지 전송 시 (런타임) | 빌드/배포 시 (빌드타임) |
| **코드 리뷰** | 거치지 않음 | MR/PR 리뷰 필수 |
| **호환성 검증** | 없음 (Registry 설정에 의존) | `testCompatibility`로 사전 검증 |
| **감사 추적** | 불가 (누가 등록했는지 모름) | Git 커밋 + CI 로그로 추적 |
| **롤백** | 어려움 (배포 프로세스 밖) | 쉬움 (Git revert → CI 재실행) |

**구체적 시나리오:**

```
[자동 등록 사고]
1. 개발자 A가 OrderEvent에 required 필드 추가
2. 로컬에서 테스트 → 자동 등록으로 prod Registry에 v2 등록
3. 기존 Consumer들이 v2를 모르므로 역직렬화 실패
4. 장애 발생 — 누가, 언제, 왜 등록했는지 추적 불가

[수동 등록 방어]
1. 개발자 A가 OrderEvent에 required 필드 추가
2. MR 생성 → CI가 testCompatibility 실행 → INCOMPATIBLE → MR 차단
3. 개발자가 optional + default null로 수정
4. CI 통과 → 코드 리뷰 → 머지 → CI가 schemaRegistryRegister 실행
5. 안전하게 v2 등록
```

### 문제: 자동 등록의 위험 (정리)

`auto.register.schemas=true`(현재 local/dev 설정)이면 Producer가 첫 메시지 전송 시 Schema Registry에 스키마를 자동 등록한다. 개발에는 편리하지만 프로덕션에서는 위험하다.

| 위험 | 설명 |
|------|------|
| 거버넌스 부재 | 누가, 언제, 왜 변경했는지 추적 불가 — Git 히스토리와 분리 |
| 호환성 파괴 | Consumer가 이전 스키마 사용 중인데 호환 불가 스키마가 등록됨 |
| 롤백 어려움 | 배포 프로세스 없이 등록되었으므로 원인 파악/되돌리기 어려움 |

### 해결: 환경별 전략 분리

| 환경 | `auto.register.schemas` | 등록 방법 | Registry 접근 |
|------|------------------------|-----------|-------------|
| **local** | `true` | 자동 (편의성) | Read/Write |
| **dev** | `true` | 자동 (편의성) | Read/Write |
| **staging** | `false` | CI/CD 파이프라인 | **Read-only** |
| **prod** | `false` | CI/CD 파이프라인 | **Read-only** |

## Gradle Schema Registry Plugin

### 설정 시 주의: pluginManagement 저장소

플러그인 추가 후 빌드하면 다음 에러가 발생한다:

```
Could not find io.confluent:kafka-schema-registry:7.6.0.
  Searched in: https://plugins.gradle.org/m2/
Could not find com.github.ImFlog.avro:avro:1.11.2.
  Searched in: https://plugins.gradle.org/m2/
```

**원인**: `plugins {}` 블록의 플러그인 의존성과 `repositories {}` 블록의 프로젝트 의존성은 **별도 스코프**로 해결된다.

| 스코프 | 해결 저장소 | 설정 위치 |
|--------|-----------|----------|
| 프로젝트 의존성 (`implementation`) | `build.gradle`의 `repositories {}` | build.gradle |
| **플러그인 의존성** (`plugins {}`) | **`settings.gradle`의 `pluginManagement {}`** | settings.gradle |

프로젝트의 `repositories`에 Confluent Maven이 이미 있어서 `kafka-avro-serializer:7.6.0` 등은 잘 해결되고 있었다. 하지만 플러그인의 전이 의존성은 별도 경로로 해결하므로, Gradle Plugin Portal에서만 검색하다가 실패한 것이다.

```
com.github.imflog:kafka-schema-registry-gradle-plugin:2.1.0
  ├── io.confluent:kafka-schema-registry:7.6.0   ← Confluent 저장소에만 있음
  └── com.github.ImFlog.avro:avro:1.11.2         ← JitPack에만 있음
```

**해결**: `settings.gradle`에 `pluginManagement`로 플러그인 전용 저장소 추가:

```groovy
// settings.gradle
pluginManagement {
    repositories {
        gradlePluginPortal()
        maven { url 'https://packages.confluent.io/maven/' }
        maven { url 'https://jitpack.io' }
    }
}
```

> **학습 포인트**: Gradle의 `repositories`는 프로젝트 의존성 전용이다. 플러그인이 외부 저장소의 라이브러리에 의존하면 `pluginManagement`에 별도로 저장소를 등록해야 한다.

### 플러그인 역할

Schema Registry의 **REST API를 호출하는 클라이언트 도구**다. Redpanda가 Git을 감시하거나 파일을 자동 탐지하는 것이 아니라, CI가 명시적으로 Gradle task를 실행하여 Registry에 등록한다.

### 설정 구조

```groovy
schemaRegistry {
    url = project.findProperty('schemaRegistryUrl') ?: 'http://localhost:18081'

    register {
        // src/main/avro/*.avsc 파일을 동적 스캔
        fileTree('src/main/avro').matching { include '*.avsc' }.each { file ->
            def subjectName = file.name.replace('.avsc', '-value')
            subject(subjectName, file.path, 'AVRO')
        }
    }

    compatibility {
        // 동일 subject 목록으로 호환성만 검증
        fileTree('src/main/avro').matching { include '*.avsc' }.each { file ->
            def subjectName = file.name.replace('.avsc', '-value')
            subject(subjectName, file.path, 'AVRO')
        }
    }
}
```

### fileTree 동적 스캔의 의미

`.avsc` 파일을 추가/삭제하면 **build.gradle 수정 없이** 자동으로 등록 대상이 변경된다. subject 이름은 `{파일명}-value` 컨벤션을 따른다.

현재 프로젝트의 6개 .avsc 파일:

| 파일 | Subject 이름 |
|------|-------------|
| `OrderEvent.avsc` | `OrderEvent-value` |
| `TransactionEvent.avsc` | `TransactionEvent-value` |
| `MessageStatus.avsc` | `MessageStatus-value` |
| `PipelineExecuteRequest.avsc` | `PipelineExecuteRequest-value` |
| `PipelineExecuteResponse.avsc` | `PipelineExecuteResponse-value` |
| `PipelineStatusUpdate.avsc` | `PipelineStatusUpdate-value` |

### Gradle task 사용법

```bash
# 호환성 검증만 (MR/PR 파이프라인에서 사용)
./gradlew schemaRegistryTestCompatibility -PschemaRegistryUrl=http://localhost:18081

# 스키마 등록 (develop/main 머지 후 사용)
./gradlew schemaRegistryRegister -PschemaRegistryUrl=http://localhost:18081
```

`-PschemaRegistryUrl`로 환경별 Registry URL을 CI에서 주입할 수 있다.

## 브랜치 전략과 Schema Registry

Schema Registry는 **stateful 공유 자원**이다. 여러 브랜치가 동시에 스키마를 변경하면 충돌할 수 있으므로, **검증과 등록을 분리**한다.

```
feature/add-field ──→ testCompatibility만 (등록 안 함)
develop           ──→ dev Registry에 등록
main              ──→ prod Registry에 등록
```

### CI/CD 파이프라인 예시 (GitLab CI)

```yaml
stages:
  - validate
  - register

# 모든 MR: 호환성 검증만 (등록하지 않음)
schema-validate:
  stage: validate
  script:
    - ./gradlew schemaRegistryTestCompatibility
        -PschemaRegistryUrl=$DEV_REGISTRY_URL
  rules:
    - if: $CI_MERGE_REQUEST_IID

# develop 머지 후: dev 환경에 등록
schema-register-dev:
  stage: register
  script:
    - ./gradlew schemaRegistryRegister
        -PschemaRegistryUrl=$DEV_REGISTRY_URL
  rules:
    - if: $CI_COMMIT_BRANCH == "develop"

# main 머지 후: prod 환경에 등록
schema-register-prod:
  stage: register
  script:
    - ./gradlew schemaRegistryRegister
        -PschemaRegistryUrl=$PROD_REGISTRY_URL
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
```

**핵심**: feature 브랜치에서 등록하면 다른 브랜치가 머지할 때 예상치 못한 호환성 실패가 발생한다. `testCompatibility`는 실제 등록 없이 현재 Registry 상태에 대해 검증만 수행하므로 안전하다.

## staging 프로파일 설정

```yaml
spring:
  config:
    activate:
      on-profile: staging
  kafka:
    producer:
      properties:
        auto.register.schemas: false    # CI/CD에서만 등록
        use.latest.version: true        # 이미 등록된 최신 버전 사용
```

| 설정 | 값 | 의미 |
|------|-----|------|
| `auto.register.schemas` | `false` | Producer가 스키마를 자동 등록하지 않음 |
| `use.latest.version` | `true` | Registry에 등록된 최신 버전을 자동으로 사용 |

`use.latest.version=true`가 없으면, 앱이 사용하는 스키마 ID와 Registry의 최신 버전이 다를 때 오류가 발생할 수 있다.

## 호환 불가 스키마 변경 대처법

BACKWARD 호환이 깨지는 변경이 필요할 때:

| 방법 | 적합한 상황 | 위험도 |
|------|------------|--------|
| **1. 호환 변환** (deprecate + add) | 변경을 optional 추가로 표현 가능 | 낮음 |
| **2. 새 토픽 + 이중 쓰기** | 구조 자체가 완전히 달라짐 | 중간 |
| **3. Subject 삭제 후 재등록** | 토픽에 기존 데이터 없음 | 높음 |
| **4. 호환성 NONE 임시 설정** | dev 환경에서 빠른 반복 | 최고 |

**실무 기준**: 방법 1을 우선 시도하고, 정말 불가능할 때만 방법 2를 사용한다.

| 설정 | local | dev | staging |
|------|-------|-----|---------|
| `auto.register.schemas` | `true` | `true` | `false` |
| `use.latest.version` | - | - | `true` |
| `bootstrap-servers` | `localhost:19092` | K8s IP | 환경변수 |
| `schema.registry.url` | `localhost:18081` | K8s IP | 환경변수 |
| 스키마 등록 주체 | Producer (자동) | Producer (자동) | **CI/CD** |

## 핵심 학습 포인트

1. **자동 등록은 개발용** — 프로덕션에서는 거버넌스/호환성/롤백 문제로 반드시 수동 등록
2. **Gradle Plugin = REST API 클라이언트** — Registry가 Git을 감시하는 것이 아님
3. **fileTree 동적 스캔** — .avsc 파일 추가 시 build.gradle 수정 불필요
4. **검증과 등록 분리** — feature 브랜치는 testCompatibility만, develop/main에서만 register
5. **use.latest.version** — 수동 등록 환경에서 앱이 최신 스키마를 자동으로 참조
6. **호환 불가 변경** — deprecate+add가 최우선, 새 토픽 이중 쓰기가 차선
