# C2: 메타데이터 접근 (@Header)

## 목적

`@KafkaListener`에서 메시지 본문(value) 외에 topic, partition, offset, timestamp, key 같은 메타데이터를 `@Header` 어노테이션으로 접근한다. 디버깅, 모니터링, 분산 추적에 필수적인 정보다.

---

## 변경 파일

### OrderConsumer.java

기존 `consume(OrderEvent event, Acknowledgment ack)` 시그니처에 `@Header` 파라미터 추가.

```java
@KafkaListener(topics = "chapter2.orders", groupId = "order-consumer-group")
public void consume(
        @Payload OrderEvent event,
        @Header(KafkaHeaders.RECEIVED_TOPIC) String topic,
        @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
        @Header(KafkaHeaders.OFFSET) long offset,
        @Header(KafkaHeaders.RECEIVED_TIMESTAMP) long timestamp,
        @Header(KafkaHeaders.RECEIVED_KEY) String key,
        Acknowledgment ack
) {
    log.info("Received: orderId={}, topic={}, partition={}, offset={}, timestamp={}, key={}",
            event.getOrderId(), topic, partition, offset, timestamp, key);

    lastReceivedEvent = event;
    lastReceivedPartition = partition;
    lastReceivedOffset = offset;
    latch.countDown();
    ack.acknowledge();
}
```

추가된 필드:
```java
@Getter
private int lastReceivedPartition = -1;

@Getter
private long lastReceivedOffset = -1;
```

### ProducerConsumerTest.java - 메타데이터 검증 테스트 추가

```java
@Test
void Consumer는_메타데이터를_수신할_수_있다() throws Exception {
    orderConsumer.resetLatch(1);

    // 1건 전송
    SendResult<String, OrderEvent> sendResult =
            orderProducer.sendOrder(event).get(10, TimeUnit.SECONDS);

    boolean received = orderConsumer.getLatch().await(30, TimeUnit.SECONDS);

    assertThat(received).isTrue();
    // Producer가 보낸 partition == Consumer가 수신한 partition
    assertThat(orderConsumer.getLastReceivedPartition())
            .isEqualTo(sendResult.getRecordMetadata().partition());
    assertThat(orderConsumer.getLastReceivedOffset())
            .isGreaterThanOrEqualTo(0);
}
```

## 설계 결정과 이유

### @Payload를 명시적으로 붙인 이유

파라미터가 `event` 하나일 때는 Spring이 자동으로 메시지 본문으로 인식한다. 하지만 `@Header`가 여러 개 붙으면 **어떤 파라미터가 본문인지 명시**하는 것이 가독성에 좋다. `@Payload` 없이도 동작하지만, 파라미터가 7개인 메서드에서 역할을 명확히 구분하기 위해 붙였다.

### KafkaHeaders 상수를 사용한 이유

```java
// BAD: 문자열 직접 사용 - 오타 위험, IDE 자동완성 불가
@Header("kafka_receivedPartitionId") int partition

// GOOD: 상수 사용 - 컴파일 타임 검증, IDE 자동완성
@Header(KafkaHeaders.RECEIVED_PARTITION) int partition
```

`KafkaHeaders`는 Spring Kafka가 제공하는 상수 클래스다. 문자열을 직접 쓰면 오타 시 런타임에야 발견된다.

### partition과 offset을 필드에 저장한 이유

테스트에서 **Producer가 보낸 partition과 Consumer가 받은 partition이 같은지** 교차 검증하기 위해서다. 같은 메시지이므로 반드시 같아야 한다. 이 검증으로 메타데이터가 정확히 전달되는지 확인할 수 있다.

### lastReceivedPartition 초기값을 -1로 설정한 이유

Kafka 파티션은 0부터 시작한다. 초기값이 0이면 "아직 수신하지 않음"과 "파티션 0에서 수신"을 구분할 수 없다. -1은 유효하지 않은 파티션 번호이므로 "미수신" 상태를 명확히 표현한다.

### 사용 가능한 주요 KafkaHeaders

| 상수 | 타입 | 설명 |
|------|------|------|
| `RECEIVED_TOPIC` | String | 수신된 토픽명 |
| `RECEIVED_PARTITION` | int | 수신된 파티션 번호 |
| `OFFSET` | long | 메시지 오프셋 |
| `RECEIVED_TIMESTAMP` | long | 브로커가 기록한 타임스탬프 (epoch millis) |
| `RECEIVED_KEY` | String | 메시지 키 |
| `GROUP_ID` | String | Consumer Group ID |

커스텀 헤더도 `@Header(value = "X-Custom", required = false)` 형태로 조회할 수 있다. `required = false`는 해당 헤더가 없을 수 있는 경우 NPE를 방지한다. (P4 실습에서 활용 예정)
