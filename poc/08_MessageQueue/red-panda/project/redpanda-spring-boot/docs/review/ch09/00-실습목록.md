## 각 실습의 구현 범위

### 1. 상태 모델 + Command 설계 (완료)
- `SagaStep` enum (FLIGHT_BOOKING, HOTEL_BOOKING, COMPLETED)
- `SagaStatus` enum (STARTED, IN_PROGRESS, COMPENSATING, COMPLETED, COMPENSATED, FAILED)
- `SagaState` 엔티티 (sagaId, tripId, 여행 정보, 단계별 결과, 실패 정보)
- `SagaStateRepository` (JPA, stalled SAGA 조회 쿼리 포함)
- Command 3종: BookFlight, CancelFlight, BookHotel
- Event 5종: FlightBooked, FlightBookingFailed, FlightCancelled, HotelBooked, HotelBookingFailed
- Avro 스키마 8개 (`src/main/avro/ch04/`, namespace: `com.study.redpanda.avro.trip`)
- `TripSagaEventMapper` — 도메인 record ↔ Avro SpecificRecord 변환

### 2. Orchestrator 정상 플로우
- `TripSagaOrchestrator` — startSaga() + 성공 리스너 2개
- startSaga: SagaState 생성 → BookFlightCommand 발행
- onFlightBooked: 상태 업데이트 → BookHotelCommand 발행
- onHotelBooked: SagaStatus.COMPLETED
- Kafka 설정 (ch04 전용 Avro Producer/Consumer/KafkaTemplate)
- `SagaController` — sagaId/tripId 기반 상태 조회 REST API

### 3. Command Handler 구현
- `FlightCommandHandler` — BookFlight/CancelFlight 처리
- `HotelCommandHandler` — BookHotel 처리
- 각 핸들러: Command 수신 → 도메인 로직 → 결과 이벤트 발행

### 4. 보상 트랜잭션
- onFlightBookingFailed: 보상 불필요 → 바로 FAILED
- onHotelBookingFailed: COMPENSATING → CancelFlightCommand 발행
- onFlightCancelled: 보상 완료 → COMPENSATED
- 보상 실패 처리 전략 (재시도 → DLQ → 수동 개입)

### 5. Kafka 트랜잭션 (CTP)
- Transactional Producer 설정 (transactional.id, idempotence)
- KafkaTransactionManager를 ListenerContainerFactory에 연결
- Consumer isolation.level=read_committed 설정
- DB @Transactional + Kafka TX 체이닝 구현

### 6. 타임아웃 + 장애 복구
- SagaState에 stepStartedAt 필드 추가
- 타임아웃 감지 스케줄러 (@Scheduled)
- stalled SAGA 복구 스케줄러
- 멱등성 보장 (2계층):
  - Orchestrator 측: 복구 재전송 시 동일 Command 보장 (sagaId + currentStep 기반)
  - Handler 측: sagaId + commandType 기반 중복 처리 방지

### 7. 통합 테스트
- 정상 플로우: 항공 → 호텔 → COMPLETED
- Step 1 실패: 항공 실패 → FAILED (보상 불필요)
- Step 2 실패: 호텔 실패 → 항공 취소 → COMPENSATED
- 타임아웃: 서비스 응답 지연 → 보상 트리거
- Testcontainers (Redpanda + PostgreSQL)

### 8. Choreography vs Orchestration 비교 리뷰
- Ch08 vs Ch09 구현 경험 비교 (코드 복잡도, 추적성, 테스트 용이성)
- 안티패턴 3가지 (God Orchestrator, 동기 호출 혼합, 보상 누락) 체크리스트
- DB 기반 vs Event Sourcing 상태 관리 선택 근거
- 대안 패턴(WorkflowStep + WebClient, Spring State Machine)과의 트레이드오프

### 9. Transactional Outbox 패턴
- 실습 #5의 DB+Kafka Dual-Write 간극을 Outbox 패턴으로 근본 해결
- `OutboxEvent` 엔티티 + `OutboxEventRepository` (폴링 + 정리 쿼리)
- `OutboxPublisher`: 500ms 폴링, JSON→Avro→`executeInTransaction()`, 5분 정리
- `TripSagaOrchestrator` / `TripSagaRecoveryScheduler`: kafkaTemplate → outbox 교체
- 폴링 CDC vs Debezium CDC 비교, Handler 변경 불필요 근거
