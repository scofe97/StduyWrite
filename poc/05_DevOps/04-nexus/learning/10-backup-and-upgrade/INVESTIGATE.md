# Ch10: 백업, 복구, 업그레이드 — 심화 탐구

## Q1: Blob Store 백업 중 쓰기가 발생하면 정합성 문제가 생기는가?

생길 수 있다. Nexus가 실행 중인 상태에서 Blob Store를 백업하면, 백업 시작 시점과 종료 시점 사이에 새 blob이 추가되거나, Cleanup Task가 blob을 삭제할 수 있다.

tar나 rsync는 파일을 순차적으로 읽으므로, 백업 도중 변경된 파일은 불완전하게 캡처될 가능성이 있다.

가장 흔한 불일치 시나리오는 이렇다. 백업이 A 디렉토리까지 처리했을 때 새 아티팩트가 B 디렉토리에 배포되면, DB Export에는 해당 아티팩트의 메타데이터가 있지만 Blob 백업에는 파일이 없는 상태가 만들어진다.

복구 후 이 아티팩트에 접근하면 "Asset Blob Unreadable" 에러가 발생한다. 반대 경우인 orphan blob은 디스크만 낭비할 뿐 기능적 문제는 없으므로 상대적으로 무해하다.

실무적 대응책은 세 가지다.

**첫째, cold backup.** Nexus를 멈추고 백업하는 것이 가장 확실하다. `docker compose stop nexus`로 멈추고, `tar czf`로 전체 데이터를 묶고, `docker compose up -d nexus`로 재시작하면 된다. 30GB 규모라면 5-10분이면 충분하다.

**둘째, 파일시스템 스냅샷.** LVM 스냅샷이나 ZFS 스냅샷을 활용하면 hot backup에서도 정합성을 보장할 수 있다. 스냅샷은 COW(Copy-on-Write) 방식이므로 생성 자체는 밀리초 단위로 완료된다.

클라우드 환경이라면 EBS Snapshot(AWS)이나 Persistent Disk Snapshot(GCP)이 동일한 역할을 한다.

**셋째, 순서 전략.** DB Export를 먼저 실행하고 Blob을 백업하는 것이다. DB에 있는데 Blob에 없는 불일치는 "Export 후 ~ Blob 백업 완료 전"에 배포된 아티팩트에서만 발생하므로, 이 창(window)을 최소화하면 피해를 줄일 수 있다.

## Q2: OrientDB에서 H2로 마이그레이션한 Nexus의 DB 백업 방식 차이는?

Nexus 3.70 이전에는 OrientDB를 내장 DB로 사용했고, 3.70+부터 H2로 전환이 진행 중이다.

OrientDB는 `nexus-data/db/` 디렉토리에 여러 `.pcl`, `.cfl` 파일로 데이터를 저장한다. hot backup 시 파일 잠금 문제가 발생할 수 있다.

OrientDB는 자체적으로 WAL을 사용하는데, 파일을 직접 복사하면 WAL과 데이터 파일의 불일치가 생길 수 있다. DB Export Task가 내부 API를 통해 일관된 스냅샷을 내보내므로, 파일 직접 복사보다 Export Task가 안전하다.

Export 결과는 JSON 형태의 덤프 파일이므로, 사람이 읽을 수도 있고 다른 도구로 분석할 수 있다는 부가적 장점이 있다.

H2는 단일 `.mv.db` 파일에 모든 데이터를 저장하므로 파일 복사가 더 단순해 보인다. 하지만 실행 중에 복사하면 MVStore 엔진의 페이지 쓰기와 불일치가 생길 수 있다.

복사 도중 일부 페이지만 새 데이터이고 나머지는 이전 데이터인 상태가 되면 파일이 손상된 것으로 인식된다. H2도 마찬가지로 DB Export Task를 통한 백업이 권장된다.

OrientDB → H2 마이그레이션은 업그레이드 중 자동으로 수행된다. 수만 개의 컴포넌트가 있는 환경에서는 30분~1시간이 소요되기도 한다.

마이그레이션 중 강제 종료하면 DB가 손상될 수 있으므로, 반드시 전체 백업 후 충분한 시간을 두고 업그레이드해야 한다.

마이그레이션이 완료된 뒤 기존 OrientDB 파일은 더 이상 사용되지 않으므로, 디스크 절약을 위해 삭제해도 무방하다. 단, 롤백 가능성을 위해 백업은 보관할 것.

## Q3: Docker volume 백업 시 컨테이너를 멈춰야 하는가?

반드시 멈춰야 하는 건 아니지만, 멈추는 게 안전하다.

Docker named volume은 결국 호스트 파일시스템의 특정 경로(`/var/lib/docker/volumes/<volume-name>/_data`)에 있는 디렉토리이므로, 컨테이너가 실행 중이어도 접근 자체는 가능하다.

문제는 Nexus가 DB에 쓰기를 하는 도중에 파일을 복사하면 "torn write"가 발생할 수 있다는 점이다. 파일의 절반만 새 데이터이고 나머지는 이전 데이터인 상태로 복사되면, 복구 시 DB 파일이 손상된 것으로 인식된다.

Blob Store 파일(아티팩트 바이너리)은 한 번 쓰면 수정되지 않는(immutable) 특성이 있어서 torn write 위험이 낮다. 하지만 `.properties` 파일이나 DB 파일은 수시로 업데이트되므로 위험하다.

멈추지 않고 안전하게 백업하는 대안으로는, Docker의 `--volumes-from` 옵션으로 별도 컨테이너에서 read-only로 볼륨을 마운트하는 방법이 있다.

하지만 이것도 파일시스템 레벨의 원자성을 보장하지는 않는다. read-only 마운트는 "이 컨테이너에서는 쓸 수 없다"일 뿐, 원본 컨테이너의 쓰기를 막는 게 아니기 때문이다.

확실한 방법은 `docker compose stop nexus`로 멈추고 백업한 뒤 재시작하는 것이다. 30GB 규모의 nexus-data라면 tar 생성에 5-10분이면 된다.

이 다운타임이 허용되지 않는 환경이라면, LVM/ZFS 스냅샷이나 클라우드 디스크 스냅샷을 사용해야 한다.

## Q4: 업그레이드 후 롤백이 불가능한 경우는?

DB 스키마가 이전 버전과 호환되지 않는 방향으로 변경된 경우 "단순 태그 롤백"이 불가능하다.

새 버전이 테이블 컬럼을 추가하거나 데이터 형식을 변환하면, 이전 버전이 해당 스키마를 이해하지 못해서 시작에 실패한다. 에러 메시지는 보통 "Unknown column" 또는 "Schema version mismatch" 형태로 나타난다.

Nexus의 대표적인 사례가 OrientDB → H2 마이그레이션이다. 이 전환이 완료된 후에는 OrientDB 기반의 이전 버전으로 되돌릴 수 없다. DB 엔진 자체가 바뀌었기 때문이다.

마이너 버전 업그레이드에서도 DB 스키마 변경이 포함될 수 있다. Nexus의 릴리즈 노트에 "Database migration included"라는 문구가 있으면, 해당 업그레이드는 되돌리기 어렵다고 봐야 한다.

이런 상황에 대비하는 유일한 방법은 업그레이드 전 전체 백업이다. 롤백이 필요하면 이전 이미지 태그 + 이전 백업 데이터를 함께 복원해야 한다.

구체적 절차는 이렇다: (1) 현재 컨테이너 중지 → (2) 현재 데이터 볼륨 보관 → (3) 백업 데이터를 새 볼륨에 복원 → (4) 이전 이미지 태그로 컨테이너 시작.

업그레이드 후 최소 1-2주간 이전 백업을 보존하는 것이 권장된다. 그 기간 동안 문제가 없으면 삭제해도 무방하다.

프로덕션 업그레이드 전 스테이징 환경에서 먼저 테스트하는 것도 빠뜨리면 안 된다. 스테이징 Nexus에 프로덕션 데이터의 복사본을 넣고 업그레이드를 시도해보면, 마이그레이션 소요 시간과 잠재적 문제를 미리 파악할 수 있다.

## Q5: RPO와 RTO를 설계할 때 고려할 현실적 요소는?

RPO(Recovery Point Objective)와 RTO(Recovery Time Objective)는 이론적으로는 "짧을수록 좋다"이지만, 현실에서는 비용과 복잡도의 트레이드오프가 있다.

RPO 1시간이면 매 시간 백업을 돌려야 하고, 그만큼 스토리지와 네트워크 비용이 증가한다.

RTO 30분이면 자동화된 복구 스크립트와 모니터링이 필수이며, 이를 구축하고 유지하는 인력 비용이 든다.

Nexus의 경우 현실적인 질문은 "Nexus가 N시간 동안 다운되면 무슨 일이 벌어지는가?"이다.

CI 빌드가 로컬 캐시(`~/.m2`, `node_modules`)로 돌아갈 수 있다면 실제 영향은 제한적이고, RTO를 4시간으로 잡아도 괜찮을 수 있다. 반면 컨테이너 기반 CI에서 매번 깨끗한 환경으로 시작하는 경우, Nexus 다운이 곧 전체 개발팀의 작업 중단을 의미한다.

RPO도 마찬가지다. SNAPSHOT 위주의 아티팩트는 다시 빌드하면 복구되므로 RPO 24시간이면 충분하다.

하지만 RELEASE 아티팩트가 포함된 환경에서는 RPO를 더 짧게 잡아야 한다. RELEASE는 재빌드해도 동일한 바이너리가 나온다는 보장이 없으니까. 빌드 시점의 의존성 버전, 환경변수, 심지어 시간에 따라 결과물이 달라질 수 있다.

비용 효율적인 설계 지침을 정리하면 이렇다. 소규모(~30명)는 RPO 24시간 + RTO 4시간. 중규모(30~100명)는 RPO 4~8시간 + RTO 1~2시간. 대규모(100명+)는 RPO 1시간 + RTO 30분으로, 이 수준이면 Nexus Pro HA를 검토하는 것이 인력 비용 대비 효율적이다.

## Q6: 멀티 노드 환경에서 백업 동기화 전략은?

Nexus Pro의 HA 클러스터에서는 여러 노드가 공유 Blob Store(NFS 또는 S3)와 공유 DB(외부 PostgreSQL)를 사용한다. 이 경우 백업 전략이 단일 인스턴스와 근본적으로 달라진다.

공유 Blob Store가 S3라면 S3의 Cross-Region Replication이나 Versioning으로 백업을 대체할 수 있다. 별도의 Blob Store 백업이 불필요해지는 셈이다.

S3 Versioning을 켜두면 삭제된 아티팩트도 버전 이력으로 남아 있어 복원이 가능하다. 비용은 버전 수에 비례해서 증가하므로, Lifecycle Policy로 오래된 버전을 자동 삭제하는 규칙을 함께 설정해야 한다.

NFS라면 NFS 서버 측에서 스냅샷을 찍는 방식이 적합하다.

외부 PostgreSQL은 `pg_dump`로 DB 백업을 하되, Blob Store 백업과 시점을 최대한 맞춰야 한다. `pg_dump`는 일관된 스냅샷을 제공하므로 hot backup에서도 DB 정합성은 보장된다.

하지만 DB 스냅샷 시점과 Blob Store 백업 시점이 차이 나면 불일치가 발생할 수 있다. 이상적으로는 `pg_dump`를 먼저 실행하고, 완료 직후 S3 스냅샷을 트리거하는 순서다.

OSS(단일 노드)에서 "멀티 노드 비슷한 효과"를 내고 싶다면, Nexus를 두 대 운영하면서 하나는 읽기(proxy/group), 하나는 쓰기(hosted)로 역할을 분리하는 방법이 있다.

쓰기 노드가 다운되어도 읽기 노드에서 이미 캐싱된 의존성은 제공할 수 있으므로, 빌드가 완전히 멈추는 것은 방지된다.

## 심화 질문

> 재해 복구(DR) 사이트를 구성할 때 Nexus 데이터 복제 방법은?

DR 사이트 구성의 핵심은 "주 사이트 전체가 불능이 되어도 보조 사이트에서 서비스를 이어갈 수 있는가"이다.

**첫째, 스토리지 레벨 복제.** S3 Cross-Region Replication이나 NFS 미러링으로 Blob Store를 실시간 복제하고, DB는 PostgreSQL Streaming Replication으로 복제본을 유지한다. RPO가 가장 짧지만(수 초~수 분), 인프라 비용이 높다.

**둘째, 백업 전송 방식.** 주기적으로 백업 파일을 DR 사이트로 전송하고, 대기 상태의 Nexus를 두는 구조다. RPO는 백업 주기에 의존하며(1시간~24시간), 비용은 낮지만 전환 시간이 길다. DR 전환을 분기별로 모의 훈련하는 것이 중요하다.

**셋째, Nexus Pro Replication.** Pro 버전은 리포지토리 간 복제를 지원해서, 주 사이트의 아티팩트가 DR 사이트에 자동으로 복제된다. 가장 깔끔한 방법이지만 라이선스 비용이 수반된다.

OSS에서 현실적인 DR 전략은, S3 Blob Store + Cross-Region Replication으로 아티팩트를 보호하고, DB 백업을 매 시간 S3에 업로드하며, DR 사이트에 Nexus 설정이 완료된 Docker 환경을 미리 준비해두는 것이다.

장애 시 DB 복원 + Blob Store S3 엔드포인트 변경으로 30분 내 복구가 가능하다. DNS 전환(Route 53 failover 등)까지 자동화하면 개발자 측에서는 URL 변경 없이 DR 사이트를 사용할 수 있다.
