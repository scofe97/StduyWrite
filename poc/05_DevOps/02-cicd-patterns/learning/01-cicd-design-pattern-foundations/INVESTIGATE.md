## Q1. GoF 패턴 중 CI/CD에 가장 자주 적용되는 3개와 이유는 무엇인가?

**핵심 포인트**

- **Factory Method**: CI/CD에서 가장 흔한 문제인 "환경별 파이프라인 중복"을 직접 해결한다. dev/staging/prod 파이프라인이 공통 구조를 공유하면서도 환경별 설정(네임스페이스, 레플리카 수, 승인 여부)만 다를 때 Factory로 생성 로직을 한 곳에 집중시킨다.
- **Strategy**: 배포 방식은 시스템이 성숙하면서 반드시 바뀐다. 초기엔 단순 Rolling 배포로 시작하다가 Blue-Green, Canary로 진화한다. Strategy 패턴으로 배포 로직을 인터페이스 뒤에 숨기면 교체 시 파이프라인 핵심 로직을 건드리지 않아도 된다.
- **Observer**: 빌드/배포 이벤트를 여러 채널(Slack, PagerDuty, 이메일)에 전달해야 하는 요구사항은 시간이 지날수록 채널이 추가되는 방향으로 변한다. Observer로 이벤트 발행-구독을 분리하면 새 채널 추가 시 기존 파이프라인 코드는 변경 없이 새 구독자만 등록하면 된다.
- 이 세 패턴이 자주 선택되는 공통 이유는 CI/CD의 세 가지 핵심 변화 지점(환경 구성 / 배포 전략 / 알림 채널)과 정확히 매핑되기 때문이다.
- 패턴 선택의 역순은 위험하다. "Factory를 써야 하니까 구조를 맞추자"가 아니라 "이 중복 문제를 해결할 패턴이 무엇인가"라는 방향으로 접근해야 한다.

**심화 질문**: Factory와 Strategy를 함께 적용하면 어떤 구조가 만들어지는가? 환경별로 다른 배포 전략을 Factory가 주입하는 설계를 코드로 표현해 보라.

## Q2. Test Pyramid과 Test Diamond 중 어떤 기준으로 선택해야 하는가?

**핵심 포인트**

- **비즈니스 로직의 위치**: 단위 테스트로 검증 가능한 순수 함수 형태의 비즈니스 로직이 풍부하면 Pyramid가 유리하다. 반면 로직 대부분이 서비스 간 데이터 흐름에 있다면 통합 테스트가 더 많은 가치를 제공한다.
- **시스템 특성**: 마이크로서비스처럼 서비스 경계가 명확하면 Pyramid, API 게이트웨이나 레거시 시스템 통합처럼 서비스 간 계약이 복잡하면 Diamond가 더 현실적이다.
- **파이프라인 속도 목표**: 단위 테스트 1,000개는 수 초 안에 끝나지만, 통합 테스트 1,000개는 수십 분이 걸린다. Diamond를 선택할 때는 파이프라인 전체 실행 시간을 반드시 측정하고 허용 범위를 사전에 정의해야 한다.
- **팀 역량**: Pyramid를 제대로 실현하려면 코드가 테스트 가능한 형태(의존성 주입, 순수 함수)로 작성되어야 한다. 레거시 코드베이스에서 이것이 어렵다면 Diamond가 더 현실적인 출발점이 될 수 있다.
- **하이브리드도 유효하다**: 핵심 도메인 로직에는 Pyramid를, 외부 시스템 통합 레이어에는 Diamond를 적용하는 혼합 전략도 실무에서 자주 선택된다.
- 두 모델 중 하나가 절대적으로 옳은 것은 아니다. 선택 기준이 명확하지 않을 때는 현재 버그가 어느 레이어에서 주로 발생하는지를 먼저 분석하는 것이 실용적이다.

**심화 질문**: 현재 프로젝트의 테스트 비율을 측정했을 때 Pyramid인가 Diamond인가? 그 결과가 파이프라인 실행 시간과 어떻게 연결되는지 분석해 보라.

## Q3. Pipeline as Code에 SRP를 어떻게 적용하는가?

**핵심 포인트**

- SRP(단일 책임 원칙)는 "변경 이유가 하나여야 한다"는 원칙이다. 파이프라인 단계에 적용하면 빌드, 테스트, 보안 스캔, 배포가 각각 독립된 단계로 분리되어야 한다는 의미다.
- **분리의 실익**: 보안 스캔 도구를 바꿀 때 배포 로직을 건드리지 않아도 된다. 빌드가 실패하면 보안 스캔 단계까지 도달하지 않으므로 실패 원인이 즉시 명확해진다.
- **단계 간 책임 경계 예시**: `build` 단계는 아티팩트 생성만, `test` 단계는 품질 검증만, `security-scan` 단계는 취약점 탐지만, `deploy` 단계는 인프라 변경만 담당한다.
- **위반 증상**: 단계 이름이 `build-and-test-and-scan`처럼 "and"로 연결되어 있거나, 하나의 셸 스크립트가 100줄 이상으로 여러 작업을 수행한다면 SRP 위반 신호다.
- SRP를 파이프라인에 적용하면 병렬 실행도 자연스럽게 가능해진다. 책임이 분리된 단계들은 서로 의존하지 않으므로 동시 실행이 안전하고, 전체 파이프라인 소요 시간이 줄어든다.
- **트레이드오프 인식**: 단계를 너무 잘게 나누면 컨테이너 초기화 오버헤드가 쌓인다. SRP는 "의미 있는 단위"로 분리하는 것이지, 모든 명령어를 별도 단계로 만드는 것이 아니다.

**심화 질문**: 파이프라인 단계에 SRP를 적용했을 때 단계 수가 늘어나면 오버헤드(컨테이너 시작 시간 등)가 증가할 수 있다. 이 트레이드오프를 어떻게 관리하는가?

## Q4. Creational 패턴이 멀티 환경 파이프라인에 왜 필요한가?

**핵심 포인트**

- 멀티 환경 파이프라인의 핵심 문제는 "공통점이 많고 차이점이 적은 구성들을 어떻게 관리하는가"다. Creational 패턴은 생성 로직을 캡슐화해서 이 문제를 해결한다.
- **Factory Method**: 환경 이름 하나를 받아서 해당 환경의 파이프라인 구성 객체를 반환한다. 공통 로직은 상위에, 환경별 차이는 팩토리 내부에 격리된다. 새 환경 추가 시 기존 코드를 수정하지 않아도 된다(OCP).
- **Abstract Factory**: 단순히 배포 설정뿐 아니라 빌드 설정, 모니터링 설정, 알림 설정까지 환경별로 묶어서 일관되게 생성해야 할 때 사용한다. "dev 환경 전체 패키지"를 하나의 팩토리가 책임진다.
- **Builder**: 파이프라인 단계 조합이 프로젝트 타입(백엔드/프론트엔드/모바일)에 따라 다르고, 단계 순서가 유동적일 때 유용하다. 플루언트 인터페이스로 단계를 조립하므로 가독성도 높아진다.
- Creational 패턴 없이 멀티 환경을 관리하면 결국 환경 수만큼 파이프라인 파일이 생기고, 새 환경 추가 시 복사-붙여넣기가 반복된다. 이것이 Snowflake Pipeline 안티패턴의 시작점이다.
- **Fail Fast 설계**: Factory에 알 수 없는 환경명이 전달되면 즉시 예외를 발생시키는 것이 안전하다. 잘못된 환경으로 배포가 진행되는 것보다 초기에 실패하는 쪽이 비용이 훨씬 낮다.

**심화 질문**: Abstract Factory와 Builder를 함께 사용해야 하는 상황은 언제인가? 두 패턴이 협력하는 설계를 설명해 보라.

## Q5. Observer 패턴은 CI/CD에서 Webhook 알림에 어떻게 적용되는가?

**핵심 포인트**

- Observer 패턴의 구조는 Subject(이벤트 발행자)와 Observer(구독자) 분리다. CI/CD에서 Subject는 파이프라인 실행 엔진이고, Observer는 Slack 봇, PagerDuty, 이메일 서버 등이다.
- **핵심 이점**: 파이프라인 엔진은 어떤 채널이 이벤트를 받는지 알 필요가 없다. 새 알림 채널을 추가할 때 파이프라인 코드를 수정하지 않고 새 Observer만 등록하면 된다(OCP 준수).
- **실제 구현 형태**: GitHub Actions의 `on: workflow_run` 트리거, GitLab CI의 `trigger` 키워드, Jenkins의 Post-build Actions가 모두 Observer 패턴의 구현체다.
- **Webhook의 역할**: Webhook은 Observer 패턴의 HTTP 기반 구현이다. 파이프라인이 이벤트를 HTTP POST로 발행하면, 등록된 Webhook URL들이 동시에 알림을 받는다.
- **Observer 실패 격리**: Observer 실패가 Subject에 영향을 미쳐서는 안 된다. Slack 알림이 실패해도 배포 파이프라인이 실패로 처리되면 안 된다. 비동기 처리와 예외 격리가 필수다.
- **순서 보장 필요 시**: 일반 Observer는 순서를 보장하지 않는다. 알림 간 순서가 중요하다면(예: 배포 시작 → 배포 완료 순서 보장) 이벤트 큐나 Choreography 기반 Saga 패턴으로 보완해야 한다.

**심화 질문**: Observer 패턴을 CI/CD에 적용할 때 알림 순서 보장이 필요한 상황이 있는가? 순서 보장이 필요하다면 어떤 방식으로 구현할 수 있는가?

## Q6. 파이프라인 코드의 기술 부채를 어떻게 관리하는가?

**핵심 포인트**

- **가시화가 먼저**: 파이프라인 기술 부채는 애플리케이션 코드 부채보다 늦게 발각된다. 파이프라인 YAML 파일도 정적 분석 도구(yamllint, actionlint)와 코드 리뷰 대상에 포함해야 한다.
- **부채의 주요 유형**: (1) 복사-붙여넣기 중복(DRY 위반), (2) 비밀값 하드코딩, (3) 하나의 단계가 너무 많은 일을 함(SRP 위반), (4) 핀 없이 `latest` 태그 사용으로 인한 재현 불가 빌드, (5) 테스트 없는 파이프라인 스크립트.
- **점진적 개선**: 파이프라인 전체를 한 번에 리팩터링하려 하면 실패한다. 변경이 빈번한 파이프라인 단계부터 DRY 원칙을 적용하고, 새 기능 추가 시 보이스카우트 룰(캠프장 규칙: 왔을 때보다 깨끗하게)을 적용한다.
- **패턴 적용은 부채 예방 수단**: Factory, Strategy, Observer 패턴을 처음부터 적용하면 나중에 환경 추가, 전략 교체, 채널 확장 시 부채가 쌓이지 않는다. 패턴은 단순 구조 개선이 아니라 미래 변경 비용을 줄이는 투자다.
- **측정 지표**: 파이프라인 코드 중복률(공통 로직이 몇 군데 복사되어 있는가), 단계당 평균 스크립트 길이, 환경 추가 시 필요한 코드 변경량 등을 추적하면 부채 수준을 정량화할 수 있다.
- **리팩터링 안전망**: 파이프라인 코드 리팩터링 전에 기존 동작을 문서화(스냅샷 테스트, 실행 로그 기록)해 두면, 리팩터링 후 동작이 변하지 않았음을 검증할 수 있다.
- **부채 우선순위**: 보안 관련 부채(하드코딩된 시크릿, 고정되지 않은 액션 버전)는 DRY 위반보다 먼저 해결해야 한다. 공급망 공격(supply chain attack) 벡터가 될 수 있기 때문이다.

**심화 질문**: 파이프라인 코드 리팩터링 시 가장 큰 위험은 무엇인가? 리팩터링이 기존 배포 동작을 깨뜨리지 않았음을 어떻게 검증하는가?

## Q7. 여러 GoF 패턴을 함께 적용할 때 조합 전략은 무엇인가?

**핵심 포인트**

- 실무 파이프라인은 단일 패턴으로 해결되지 않는다. Factory가 환경별 Strategy 인스턴스를 생성하고, Observer가 각 단계 완료를 구독하며, Chain of Responsibility가 품질 게이트를 구성하는 식으로 여러 패턴이 협력한다.
- **자연스러운 조합 쌍**: Factory + Strategy는 "환경별로 다른 배포 전략을 생성"에서 자주 묶인다. Factory가 어떤 Strategy를 생성할지 결정하고, 파이프라인은 Strategy 인터페이스만 알면 된다. Observer + Command는 "단계 완료 이벤트를 구독해 보상 트랜잭션을 큐에 넣는" 구조에서 협력한다.
- **패턴 조합의 기준**: 패턴을 조합할 때는 각 패턴이 해결하는 문제가 서로 다른지 먼저 확인해야 한다. 같은 문제를 두 패턴이 중복 해결하면 복잡도만 올라간다.
- **계층별 패턴 배치**: Creational 패턴은 파이프라인 초기화 레이어(구성 생성)에, Structural 패턴은 파이프라인 실행 레이어(단계 조합)에, Behavioral 패턴은 파이프라인 동작 레이어(전략, 이벤트, 게이트)에 배치하면 구조가 정리된다.
- **점진적 도입**: 처음부터 모든 패턴을 적용하면 과설계가 된다. Factory로 환경 중복을 먼저 제거하고, 배포 전략 교체 요구사항이 생기면 Strategy를 추가하는 식으로 문제가 실제로 발생할 때 패턴을 도입하는 편이 실용적이다.
- **패턴 조합 문서화**: 여러 패턴이 협력하는 구조는 코드만으로는 의도가 전달되지 않는다. 아키텍처 문서나 ADR(Architecture Decision Record)에 "왜 이 패턴 조합을 선택했는가"를 기록해 두면 팀원 온보딩 비용이 낮아진다.

**심화 질문**: Factory + Strategy + Observer를 모두 적용한 파이프라인 설계를 다이어그램으로 그려보라. 각 패턴이 어느 레이어에 위치하고, 서로 어떻게 협력하는지 설명해 보라.

두 번째 질문: 이 조합에 Chain of Responsibility를 추가한다면 어느 지점에 삽입하는 것이 자연스러운지 이유와 함께 답해 보라.
