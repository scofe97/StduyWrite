# Ch07. INVESTIGATE — 도메인 주도 CI/CD와 규제 준수

탐구 질문은 LEARN.md의 내용을 단순히 확인하는 것을 넘어,
실제 도입 시 직면하는 판단 문제를 다룬다.

---

## Q1. Bounded Context와 파이프라인 경계는 항상 일치해야 하는가?

DDD의 Bounded Context 경계와 파이프라인 경계가 1:1로 대응하는 것이 이상적이지만,
현실에서는 그렇지 않은 경우가 많다. 예를 들어 "결제" Context가 결제처리·환불·정산 세 개의
서비스로 구성된 경우, 파이프라인을 Context 단위(1개)로 묶을지 서비스 단위(3개)로 분리할지
판단이 필요하다.

**탐구 포인트**:
- 공유 커널(Shared Kernel) 패턴에서 공통 라이브러리 변경이 두 Context의 파이프라인을
  연쇄 트리거해야 하는가? 연쇄 트리거가 없으면 소비자가 깨지고, 있으면 불필요한 배포가 늘어난다.
- 파이프라인이 지나치게 세밀해지면(서비스 100개 = 파이프라인 100개) 유지보수 비용이 폭증한다.
  재사용 가능 워크플로우(`workflow_call`)로 공통 스텝을 추출하면 파이프라인 수가 늘어도
  수정 지점은 단일하게 유지된다.
- 규제 경계(PCI-DSS 범위 내 vs 외부 서비스)가 DDD Context 경계와 다를 때,
  어느 경계를 파이프라인 분리 기준으로 삼아야 하는가?

**조사 방향**: Conway's Law("시스템 구조는 조직의 커뮤니케이션 구조를 반영한다")와
파이프라인 설계의 관계. 팀 구조를 먼저 바꾸지 않고 파이프라인 구조만 바꾸면 어떤 마찰이 생기는가?

## Q2. RBAC와 ABAC 중 어느 것이 더 안전한가?

LEARN.md에서 규제 환경에서 ABAC를 권장했지만, ABAC는 복잡성이 높아 잘못 구현하면
오히려 보안 허점이 된다. 정책 조건이 많아질수록 특정 조합에서 의도치 않은 허용이 발생할
가능성도 높아진다. "더 안전한 방식"은 정책 강도가 아니라 정책을 얼마나 체계적으로
테스트하느냐에 달려 있다.

**탐구 포인트**:
- OPA `rego`로 ABAC 정책을 작성할 때, 경계 케이스(contractor의 production 승인 시도,
  clearance_level 미달 사용자의 critical 환경 접근)에 대한 단위 테스트를 어떻게 구성하는가?
  `opa test`가 정책 변경의 회귀를 자동으로 잡아낼 수 있는가?
- RBAC + 컨텍스트 조건(IP 대역, 근무 시간, MFA 여부)을 조합하는 "경량 ABAC"가
  완전한 ABAC보다 실용적인 경우는 언제인가? 팀 규모와 정책 복잡도 사이의 임계점은 어디인가?
- 정책 YAML 변경 자체도 변경 관리 대상이다. ABAC 정책 파일 PR은 일반 코드 PR과 동일한
  리뷰어 자격 요건을 가져야 하는가, 아니면 보안팀 전용 리뷰어가 필요한가?

**조사 방향**: OPA Rego 단위 테스트 작성법을 공식 문서로 학습하고, `allow`/`deny` 규칙
각각에 대한 positive/negative 테스트 케이스를 직접 작성해볼 것.

## Q3. Compliance as Code를 도입할 때 가장 큰 장벽은 무엇인가?

기술적 구현보다 조직적 장벽이 더 높다. 규제 감사관은 수십 년간 PDF 체크리스트와
서명 문서를 증적으로 받아왔다. "파이프라인 실행 로그를 보면 됩니다"라는 답변이
즉각 수용되지 않는 경우가 많다.

**탐구 포인트**:
- 감사관 설득의 핵심은 동등성 증명이다. "수동 체크리스트의 각 항목이 파이프라인의 어느
  스텝에 대응하는지" 매핑 문서를 제시하면 설득력이 높아진다. 이 매핑 문서 자체도
  버전 관리되어야 한다.
- 오탐(false positive)으로 인한 파이프라인 차단이 반복되면 개발팀이 검사를 무력화
  (`# trivy:ignore`)하려는 압력이 생긴다. 예외 파일(`.trivyignore`, `.gitleaksignore`)의
  변경은 보안팀 필수 리뷰어가 승인해야 한다는 정책을 파이프라인으로 강제할 수 있는가?
- 레거시 전환 전략: 신규 서비스부터 적용하고, 레거시는 "warn-only" 모드(차단 없이 로그만
  기록)로 시작하여 6개월 내 enforce 모드로 전환하는 단계적 접근이 현실적이다.

**조사 방향**: SOC 2 Type II 인증에서 자동화된 증적이 어떻게 평가되는지 조사.
감사 시 제출 증적의 형태(스크린샷 vs API 출력 vs 서명 PDF)가 어떻게 다른지 파악할 것.

## Q4. 감사 추적의 불변성을 기술적으로 완전히 보장할 수 있는가?

"불변 저장소"는 저장된 이후의 수정을 막지만, 저장되기 전 데이터 조작은 막지 못한다.
따라서 불변성 보장은 저장소 수준과 전송 수준을 함께 고려해야 한다.

**탐구 포인트**:
- S3 Object Lock `COMPLIANCE` 모드는 루트 계정도 삭제할 수 없다. 그러나 AWS 계정 자체가
  침해되면 어떻게 되는가? 감사 로그를 격리된 Audit Account에 보관하고 본 계정에는
  쓰기 전용 크로스 계정 권한만 부여하는 패턴이 왜 더 강한 보장을 제공하는가?
- 실시간 스트리밍(Kinesis Firehose → S3)과 배치 업로드의 차이: 배치 방식에서는
  에이전트가 로그를 업로드하기 전에 수정할 수 있는 시간 창(time window)이 존재한다.
  실시간 스트리밍은 이 창을 없애지만, 스트리밍 서비스 장애 시 로그 유실 위험이 있다.
- SHA-256 체크섬은 레코드 변조를 탐지하지만 레코드 삭제는 탐지하지 못한다.
  이전 레코드의 해시를 현재 레코드에 포함하는 체인 구조(블록체인 원리)를 적용하면
  삭제도 탐지할 수 있는가?

**조사 방향**: NIST SP 800-92 "Guide to Computer Security Log Management"에서
로그 보호 권고사항을 찾고, AWS Organizations Audit Account 패턴과 비교 정리할 것.

## Q5. Separation of Duties를 자동화할 때 발생하는 엣지 케이스는?

"코드 작성자 ≠ 승인자"라는 규칙은 두 명 이상의 팀원이 있을 때만 물리적으로 적용 가능하다.
소규모 팀, 긴급 상황, 플랫폼 제약은 SoD 자동화에서 반복적으로 등장하는 예외 케이스다.

**탐구 포인트**:
- 소규모 팀(2~3명)에서 한 명이 부재 중이면 SoD 강제로 배포가 불가능해진다.
  SoD 완화(waiver) 프로세스를 설계할 때, 완화 자체가 감사 추적에 기록되고
  72시간 내 사후 검토가 의무화되어야 규제 요건을 충족한다. Jira 자동 티켓으로 연동할 수 있는가?
- 긴급 배포(break-glass) 절차에서 SoD를 우회한 경우, 파이프라인이 "긴급 모드 활성화됨"을
  별도 플래그로 감사 로그에 기록하고 보안팀에 즉시 알림을 보내는 구조를 어떻게 구현하는가?
- GitHub Environment Protection의 자기 승인 차단이 없는 플랫폼(GitLab, Bitbucket)에서는
  ABAC 정책의 `subject.id != resource.last_committer_id` 조건으로 SoD를 강제해야 한다.
  이 조건 검증에 어떤 VCS API를 활용하는가?

**조사 방향**: ISO 27001 A.6.1.2(직무 분리)와 파이프라인 자동화의 교차 지점을 조사하고,
인증 심사에서 SoD 증적으로 인정되는 형태(설정 스크린샷, 자동 거부 로그)를 정리할 것.

## Q6. 규제 준수와 배포 속도는 본질적으로 상충하는가?

직관적으로는 검사가 많을수록 느려진다고 생각하기 쉽다. 그러나 수동 체크리스트 방식이
오히려 더 느린 이유를 이해하면 자동화의 방향이 명확해진다. 트레이드오프가 아닌
"자동화 수준의 문제"로 재정의할 수 있다.

**탐구 포인트**:
- 금융권에서 하루 수십 번 배포하면서 PCI-DSS를 유지하는 팀의 핵심은 "범위(scope) 최소화"다.
  카드 데이터를 직접 처리하지 않는 서비스를 PCI-DSS 범위에서 제외하면 가벼운 파이프라인을
  유지할 수 있다. Bounded Context 분리가 규제 범위 최소화에 직접 기여하는 구조다.
- 피처 플래그를 사용하면 코드 배포와 기능 활성화를 분리할 수 있다. 규제 감사에서 "코드 배포"와
  "플래그 토글(설정 변경)"을 어떻게 구분하는가? PCI-DSS v4.0 Req 6.4는 이 두 가지를
  명시적으로 구분하지 않는다는 점을 어떻게 해석할 것인가?
- DORA "리드 타임"을 규제 환경에서 측정할 때, 의무적인 대기 시간(48시간 Wait timer)을
  포함해야 하는가? 포함하면 팀의 실제 기술 역량이 가려진다.
- 컴플라이언스 오버헤드 측정: "컴플라이언스 게이트 소요 시간 / 전체 배포 리드 타임" 비율이
  20%를 초과하면 자동화 수준을 재검토할 신호다. 파이프라인 단계별 시간을 어떻게 기록하는가?

**조사 방향**: "Accelerate" (Forsgren 외, 2018)의 고성과 팀 특성이 규제 환경에도 적용되는지
검토. Google DORA "State of DevOps" 보고서에서 금융·의료 산업 팀의 배포 빈도 데이터를
비규제 팀과 비교할 것.
