## Q1. Monitoring vs Observability — 무엇이 다른가?

**질문**: "모니터링이 충분한데 왜 관측성이 별도로 필요한가?"

**탐구 방향**:
- 모니터링은 사전에 정의한 지표가 임계값을 넘는지 감시한다. CPU 80%, 에러율 5% 같은 알려진 실패 패턴에 대응하는 구조다.
- 관측성은 시스템이 충분한 데이터를 외부로 방출하는 속성(property)이다. 어떤 질문이 들어오든 내부 상태를 추론할 수 있어야 한다는 의미다.
- 분산 마이크로서비스에서는 서비스 A의 응답 지연이 B의 DB 쿼리 때문인지, C의 외부 API 때문인지 사전에 예측하기 어렵다. 이때 Traces가 없으면 "어디서 느려졌는가"에 답하기 불가능하다.
- 핵심 차이: 모니터링은 "알려진 것을 추적", 관측성은 "알려지지 않은 것을 탐색 가능하게" 만드는 시스템 특성이다.

**검증 질문**: 카나리 배포 중 p99 응답 시간이 갑자기 올랐다. 모니터링만으로 원인을 찾을 수 있는가? 어떤 도구가 추가로 필요한가?

**참고 관점**: 관측성 성숙도 모델은 보통 3단계로 구분한다. 1단계는 Metrics만 있는 상태, 2단계는 Metrics + Logs가 연계된 상태, 3단계는 Metrics + Logs + Traces가 TraceID로 연결된 상태다. 대부분의 조직이 2단계에 머물고 3단계로 전환하는 데 가장 큰 투자가 필요하다.

## Q2. USE 메서드와 RED 메서드 — 언제 어떤 것을 쓰는가?

**질문**: "같은 시스템을 보는데 왜 두 가지 방법론이 따로 있는가?"

**탐구 방향**:
- **USE 메서드** (Brendan Gregg 제안): Utilization(사용률), Saturation(포화도), Errors(에러). 인프라/리소스 관점이다. CPU가 얼마나 바쁜지, 큐가 얼마나 쌓이는지, 하드웨어 에러가 있는지를 본다.
  - 적용 시나리오: Node Exporter로 서버 메트릭을 볼 때, 디스크/네트워크/메모리 병목을 진단할 때.
- **RED 메서드** (Tom Wilkie 제안): Rate(초당 요청 수), Errors(에러 비율), Duration(응답 시간). 서비스/사용자 관점이다.
  - 적용 시나리오: 사용자가 "느리다"고 할 때 어떤 서비스의 어떤 엔드포인트가 문제인지 찾을 때.
- 두 방법론은 보완 관계다. USE로 인프라 병목을 찾고, RED로 서비스 품질을 측정한다.

**참고 관점**: Google SRE에서는 Four Golden Signals라는 세 번째 방법론도 사용한다. Latency(지연), Traffic(트래픽), Errors(에러), Saturation(포화도)으로 구성된다. RED 메서드에 Saturation을 추가한 형태로, 서비스 관점과 리소스 관점을 함께 본다는 점에서 RED와 USE의 중간 지점에 해당한다.

**검증 질문**: DB 서버가 느리다고 할 때 USE 메서드로 확인할 지표 3가지와 PromQL 쿼리를 작성해보라.

## Q3. SLI/SLO 설계와 에러 버짓 계산

**질문**: "SLO 99.9%와 99.99%의 실질적 차이는 무엇인가? 에러 버짓을 어떻게 활용하는가?"

**탐구 방향**:
- 월 30일 기준 에러 버짓:
  - 99.9% → 0.1% × 30일 × 24시간 × 60분 = **43.8분**
  - 99.99% → 0.01% × 43,200분 = **4.3분**
  - 차이는 10배지만, 4.3분은 단순 배포 하나로도 소진될 수 있다.
- 에러 버짓 활용 원칙: 버짓이 50% 이상 남아 있으면 새 기능 배포 가능. 50% 미만이면 배포 속도를 줄이고 신뢰성 작업 우선. 버짓 소진 시 다음 기간까지 배포 동결.
- SLI를 설계할 때는 사용자가 실제로 느끼는 품질을 측정해야 한다. "서버가 살아있는가(uptime)"보다 "사용자 요청 중 성공 비율(availability)"이 더 나은 SLI다.

**참고 관점**: 에러 버짓 소진 속도(Burn Rate)를 알림으로 걸 수 있다. 예를 들어 1시간 안에 에러 버짓의 5%가 소진되면 즉각 페이저를 울린다. 단순 에러율 임계값보다 SLO 달성 여부에 직접 연계된 알림이기 때문에 실무에서 훨씬 의미 있는 신호를 제공한다.

**검증 질문**: SLO 99.5%로 설정된 서비스가 이번 달 30분 다운됐다. 에러 버짓은 얼마나 남았는가? (30일 기준, 에러 버짓 총량을 먼저 계산할 것)

## Q4. Prometheus Pull vs Push 모델 — 트레이드오프

**질문**: "Prometheus는 왜 Push가 아닌 Pull 방식을 선택했는가? Push가 더 나은 경우는 언제인가?"

**탐구 방향**:
- **Pull 방식의 장점**:
  - Prometheus가 타겟을 직접 조회하므로 타겟이 죽었는지 즉시 감지 가능 (`up` 메트릭이 0이 됨).
  - 타겟이 Prometheus 주소를 알 필요 없다. 서비스 코드에 Prometheus 의존성이 없다.
  - 설정이 Prometheus 쪽에 집중되어 관리가 단순하다.
- **Push 방식이 유리한 경우**:
  - 배치 작업이나 람다 함수처럼 수명이 짧아 Prometheus가 폴링할 시간이 없는 경우. 이 때 Pushgateway를 사용한다.
  - 방화벽으로 외부에서 내부로 접근이 불가능한 환경.
  - 에이전트가 여러 백엔드로 동시에 데이터를 보내야 하는 경우(OpenTelemetry Collector의 Push 방식).
- Prometheus도 Pushgateway를 통해 Push를 지원하지만, 이는 예외 경로로 취급한다.

**참고 관점**: OpenTelemetry Collector는 Pull과 Push를 모두 지원하는 중간 허브 역할을 한다. 앱은 OTLP(Push)로 Collector에 보내고, Collector는 Prometheus 형식으로 expose하거나 다른 백엔드로 전달한다. 이 구조 덕분에 앱 코드를 변경하지 않고도 백엔드를 교체할 수 있다.

**검증 질문**: 5분마다 실행되는 크론 작업의 실행 결과(성공/실패, 소요 시간)를 Prometheus로 모니터링하려면 어떻게 해야 하는가?

## Q5. 분산 트레이싱에서 Context Propagation

**질문**: "TraceID가 서비스 간에 어떻게 전달되는가? 전달에 실패하면 무슨 일이 생기는가?"

**탐구 방향**:
- W3C TraceContext 표준은 `traceparent` HTTP 헤더를 정의한다. 형식: `version-traceId-parentSpanId-flags` (예: `00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01`).
- 서비스 A가 B를 호출할 때 이 헤더를 포함시킨다. B는 헤더를 파싱해 같은 TraceID로 새 Span을 생성하고, A의 SpanID를 부모로 설정한다. 이 과정이 Context Propagation이다.
- OpenTelemetry의 `getNodeAutoInstrumentations()`는 HTTP 클라이언트(axios, fetch 등)를 자동 계측해서 이 헤더를 자동으로 주입/추출한다.
- 전달 실패 시: B가 생성하는 Span이 고아(orphan) Span이 된다. Jaeger에서 연결이 끊어진 별도의 Trace로 표시되어 전체 흐름을 볼 수 없다.
- 비동기 메시지(Kafka, RabbitMQ) 경우: 메시지 헤더에 TraceContext를 포함시켜야 한다. OpenTelemetry의 메시지 계측 라이브러리가 이를 처리한다.

**참고 관점**: Sampling 전략도 Context Propagation과 함께 고려해야 한다. 모든 요청을 트레이싱하면 성능 부하가 크다. Head-based Sampling은 요청 시작 시 확률적으로 추적 여부를 결정하고, Tail-based Sampling은 요청이 완료된 후 에러가 있거나 느린 요청만 선별해 저장한다. 후자가 더 유용하지만 Collector 측에서 버퍼링이 필요해 구현이 복잡하다.

**검증 질문**: gRPC 서비스 간 호출에서 Context Propagation을 구현하려면 HTTP 헤더 대신 무엇을 사용해야 하는가?

## Q6. Alert Fatigue 방지 전략

**질문**: "알림이 너무 많으면 오히려 위험하다. 어떤 전략으로 줄이는가?"

**탐구 방향**:
- **증상 기반 알림**: "CPU 90% 초과" 대신 "사용자 영향이 있는 에러율 5% 초과"로 전환한다. 원인(CPU)이 아닌 증상(에러율)을 기반으로 해야 온콜 엔지니어가 즉시 행동할 수 있다.
- `for` 지시어 활용: Prometheus에서 `for: 5m`은 조건이 5분 연속 충족될 때만 알림을 발화한다. 순간적인 스파이크를 걸러낸다.
- **알림 그룹화**: Alertmanager의 `group_by`로 같은 서비스의 여러 알림을 하나의 메시지로 묶는다. 5개 인스턴스가 동시에 죽으면 5개 알림 대신 1개의 묶음 알림이 온다.
- **알림 억제(Inhibition)**: 상위 알림(클러스터 다운)이 발화 중이면 하위 알림(개별 서비스 다운)을 억제한다. 원인 하나로 파생된 수십 개의 알림이 쏟아지는 것을 방지한다.
- **Silence(유지보수 모드)**: 예정된 배포나 점검 시간에 알림을 임시 억제한다.
- **알림 리뷰 문화**: 주간 알림 리뷰에서 "지난 주에 발화했지만 아무 조치도 안 했던 알림"을 찾아 삭제하거나 임계값을 조정한다. 조치 없이 넘어간 알림은 노이즈다.

**참고 관점**: 알림의 품질을 측정하는 지표로 Signal-to-Noise Ratio(SNR)를 쓴다. 발화한 알림 중 실제 조치가 필요했던 비율이 높을수록 좋은 알림 체계다. 팀이 알림에 반응하는 시간(MTTA, Mean Time To Acknowledge)이 늘어나고 있다면 SNR이 낮아지고 있다는 신호로 볼 수 있다.

**검증 질문**: 배포 직후 에러율이 일시적으로 3% 올라갔다가 2분 만에 정상화됐다. `for: 5m` 설정이 있었다면 알림이 발화했겠는가? 이 설정이 적절한 이유는?
