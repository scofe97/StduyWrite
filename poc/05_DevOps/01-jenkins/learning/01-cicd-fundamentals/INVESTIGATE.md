## Q1. CI와 CD의 차이, Continuous Delivery vs Continuous Deployment

**질문**: CI와 CD의 차이를 설명하고, Continuous Delivery와 Continuous Deployment의 차이를 구체적으로 설명하시오.

**핵심 포인트**:
- CI는 코드를 공유 저장소에 자주 통합하면서 자동 빌드+테스트로 문제를 조기 발견하는 프랙티스라는 점
- CI의 전제 조건 3가지: 단일 저장소, 자동 빌드, 자동 테스트
- Delivery는 프로덕션 배포 가능 상태까지 자동화하되 실제 배포는 수동 승인이라는 점
- Deployment는 승인 단계 없이 모든 검증 통과 시 프로덕션까지 자동 배포라는 점
- 모든 조직이 Deployment를 하지 않는 이유: 규제 요건, 비즈니스 타이밍, 기술적 성숙도

**심화 질문**: 현재 팀이 Continuous Delivery를 하고 있다면, Continuous Deployment로 전환하기 위해 어떤 선행 조건이 필요한가? 테스트 커버리지, 모니터링, 자동 롤백 각각의 관점에서 설명하시오.

## Q2. DORA 메트릭과 CI/CD의 관계

**질문**: 수동 배포 대비 CI/CD의 구체적인 이점을 DORA 메트릭 관점에서 설명하시오.

**핵심 포인트**:
- DORA의 네 가지 메트릭을 각각 정의할 수 있어야 함: Deployment Frequency, Lead Time for Changes, Change Failure Rate, MTTR
- Elite 팀과 Low 팀의 차이가 수백 배에 달한다는 연구 결과
- CI/CD가 각 메트릭을 어떻게 개선하는지 구체적으로 연결: 자동 배포 → Frequency 증가, 파이프라인 → Lead Time 단축, 자동 테스트 → Failure Rate 감소, 자동 롤백 → MTTR 단축
- 배포 빈도를 높이면 오히려 위험이 줄어드는 역설적 관계 설명

**심화 질문**: DORA 메트릭 중 하나만 개선해야 한다면 어떤 것을 선택하겠는가? 그 이유는 무엇이며, 해당 메트릭을 개선하면 다른 메트릭에 어떤 영향을 미치는가?

## Q3. Jenkins vs GitHub Actions 선택 기준

**질문**: Jenkins를 선택해야 하는 상황과 GitHub Actions를 선택해야 하는 상황을 비교하시오.

**핵심 포인트**:
- Jenkins가 유리한 경우: 온프레미스 환경(보안/규제), 레거시 시스템 통합, 극도로 복잡한 파이프라인, 기존 Jenkins 인프라가 큰 조직
- GitHub Actions가 유리한 경우: GitHub 기반 프로젝트, 빠른 시작이 필요한 경우, 운영 부담을 줄이고 싶은 경우, 오픈소스 프로젝트
- Vendor Lock-in 관점: GitHub Actions는 GitHub에 종속되고, Jenkins는 특정 벤더에 종속되지 않음
- 운영 비용 관점: Jenkins는 서버 운영 비용 + 인력, GitHub Actions는 러너 비용 + 사용량 과금
- 두 도구의 장점을 모두 활용하는 하이브리드 전략의 가능성

**심화 질문**: 현재 Jenkins를 사용 중인 팀이 GitHub Actions로 마이그레이션을 검토하고 있다. 마이그레이션 판단 기준과, 마이그레이션 시 가장 큰 리스크는 무엇인가?

## Q4. 테스트 단계 배치 순서와 그 이유

**질문**: CI/CD 파이프라인에서 테스트 단계를 어떤 순서로 배치해야 하는지, 그 이유와 함께 설명하시오.

**핵심 포인트**:
- 테스트 피라미드 개념: 빠르고 저렴한 테스트를 먼저, 느리고 비싼 테스트를 나중에
- 구체적 순서: Lint → Unit Test → Build → Integration Test → Deploy to Staging → E2E Test → Deploy to Production
- 이 순서의 이유: "비용이 낮은 검증을 먼저"라는 원칙. Lint에서 잡을 수 있는 문제를 E2E까지 가서 발견하면 시간 낭비
- 각 단계의 실행 시간 차이: Lint(초 단위) vs E2E(분~시간 단위)
- 빌드가 Unit Test 뒤에 오는 이유: 빌드 실패 전에 코드 품질 문제를 먼저 잡을 수 있음 (단, 컴파일 언어의 경우 빌드가 먼저 올 수 있음)

**심화 질문**: E2E 테스트가 30분 이상 걸리는 대규모 프로젝트에서, 파이프라인 전체 실행 시간을 줄이기 위한 전략은 무엇인가? 테스트 병렬화, 영향 범위 기반 선택적 테스트 실행 등을 포함하여 설명하시오.

## Q5. 배포 공포(Deployment Fear)의 원인과 해결

**질문**: "배포 공포(Deployment Fear)"가 발생하는 원인과 CI/CD가 이를 어떻게 해결하는지 설명하시오.

**핵심 포인트**:
- 배포 공포의 정의: 배포할 때마다 장애 가능성에 대한 심리적 부담으로, 배포 빈도를 줄이려는 경향
- 악순환 구조: 배포 공포 → 배포 빈도 감소 → 한 번의 변경량 증가 → 장애 확률 증가 → 더 큰 배포 공포
- 수동 배포에서 Human Error가 공포를 강화하는 메커니즘: 체크리스트 누락, 환경 불일치 등
- CI/CD가 해결하는 방식: 자동화로 Human Error 제거, 작은 단위 배포로 위험 감소, 자동 롤백으로 심리적 안전망 제공
- 선순환 구조: 자동화 → 안전한 배포 → 배포 빈도 증가 → 변경량 감소 → 더 안전한 배포

**심화 질문**: 배포 공포가 팀 문화에 이미 깊이 자리잡은 상황에서, CI/CD 도구를 도입하는 것만으로 충분한가? 기술적 해결책 외에 조직 문화 차원에서 어떤 변화가 필요한가?

## Q6. 환경 일관성과 아티팩트 관리

**질문**: "동일한 아티팩트가 모든 환경을 통과해야 한다"는 원칙이 왜 중요한지, 이를 위반하면 어떤 문제가 발생하는지 설명하시오.

**핵심 포인트**:
- 환경별로 새로 빌드하면 발생하는 문제: 의존성 버전 차이, 빌드 도구 버전 차이, 환경변수 차이로 서로 다른 바이너리가 생성
- "staging에서 됐는데 production에서 안 된다" 현상의 근본 원인
- 올바른 방식: CI에서 한 번만 빌드, 동일 아티팩트를 dev → staging → production으로 승격(promote)
- 환경별로 달라져야 하는 것은 설정(config)뿐이어야 한다는 원칙
- Docker가 이 원칙을 실현하는 데 기여한 이유: 모든 의존성을 이미지에 포함

**심화 질문**: 컨테이너 없이(예: JAR 파일 배포) 환경 일관성을 보장하려면 어떤 방법이 있는가? Artifact Repository(Nexus, Artifactory)의 역할과 연결하여 설명하시오.

## Q7. CI/CD 도입의 선행 조건

**질문**: 수동으로 배포하고 있는 팀에 CI/CD를 처음 도입한다면, 어떤 순서로 진행하겠는가?

**핵심 포인트**:
- 한 번에 모든 것을 자동화하려 하면 실패한다는 점 (점진적 도입)
- 권장 순서: (1) 자동 빌드 → (2) 자동 단위 테스트 → (3) 자동 통합 테스트 → (4) 자동 스테이징 배포 → (5) 자동 프로덕션 배포
- 각 단계에서 필요한 전제 조건: 버전 관리, 빌드 스크립트, 테스트 코드, 환경 설정 분리
- CI 없이 CD를 도입하면 안 되는 이유: 검증 없이 자동 배포하면 장애를 자동으로 배포하는 것
- 팀의 수용도(Buy-in)가 도구 선택보다 중요하다는 점

**심화 질문**: 테스트 코드가 전혀 없는 레거시 프로젝트에 CI를 도입하려 한다. 테스트를 먼저 작성해야 하는가, CI 인프라를 먼저 구축해야 하는가? 그 이유는 무엇인가?
