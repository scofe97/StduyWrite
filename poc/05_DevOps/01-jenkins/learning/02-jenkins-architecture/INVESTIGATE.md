## Q1. Controller-Agent 분리의 필요성

**Jenkins Controller와 Agent를 분리해야 하는 이유를 보안, 확장성, 리소스 관점에서 각각 설명하시오.**

### 핵심 포인트

- **리소스**: Controller의 JVM과 빌드 프로세스가 메모리를 공유하면, 대규모 빌드가 Controller를 OOM으로 죽일 수 있다. UI 응답 지연과 스케줄링 장애로 이어진다.
- **보안**: 빌드 스크립트는 외부 코드를 실행하므로, Controller에서 직접 실행하면 `JENKINS_HOME/secrets/`에 접근하여 모든 크레덴셜을 탈취할 수 있다. Agent로 격리하면 빌드가 접근할 수 있는 범위를 Agent의 파일시스템으로 한정한다.
- **확장성**: 단일 서버는 수직 확장만 가능하지만, Agent를 분리하면 필요에 따라 Agent를 수평으로 추가할 수 있다. Kubernetes 환경에서는 빌드마다 Pod을 동적으로 생성하고 삭제하여 리소스를 효율적으로 사용한다.
- **환경 격리**: 서로 다른 기술 스택(Java, Node.js, Python)의 빌드가 서로 영향을 주지 않도록 Agent별로 환경을 분리한다.

### 심화 질문

- Flyweight Execution과 Heavyweight Execution의 차이는 무엇이며, `agent none`을 사용했을 때 Controller에서 실행되는 것은 무엇인가?
- 임시 에이전트(Ephemeral Agent)가 상시 에이전트(Permanent Agent)보다 보안적으로 유리한 이유는 무엇인가?

## Q2. Controller 장애와 복구

**Jenkins Controller가 장애를 일으키면 실행 중인 빌드는 어떻게 되는가? 복구 절차를 단계별로 설명하시오.**

### 핵심 포인트

- **실행 중 빌드**: Agent에서 실행 중인 빌드 프로세스는 Agent의 OS 프로세스이므로 즉시 죽지 않는다. 그러나 빌드가 완료되어도 결과를 Controller에 보고할 수 없어 로그와 아티팩트가 유실된다.
- **대기 중 빌드**: 빌드 큐는 Controller 메모리에 있으므로, 큐에 대기 중이던 모든 빌드가 사라진다.
- **새 빌드 불가**: 웹훅, 크론 트리거 등 모든 빌드 요청을 처리할 주체가 없다.
- **UI 접근 불가**: 대시보드 자체가 Controller가 서빙하므로 상태 확인이 불가능하다.
- **복구 절차**: (1) `JENKINS_HOME` 상태 확인 → (2) 정상이면 Controller 프로세스 재시작 → (3) 손상되었으면 백업에서 `config.xml`, `jobs/`, `secrets/` 복원 → (4) 재시작 후 미완료 빌드를 수동 재트리거.
- **예방**: JCasC(Configuration as Code)로 설정을 코드화하면, Controller를 처음부터 재구축하는 시간을 대폭 단축할 수 있다.

### 심화 질문

- Active-Passive HA 구성에서 `JENKINS_HOME`을 NFS에 두면 어떤 문제가 발생할 수 있는가? (파일 잠금, 성능)
- Jenkins가 데이터베이스 대신 파일시스템에 상태를 저장하는 설계가 HA에 미치는 영향은 무엇인가?

## Q3. 플러그인 시스템의 명과 암

**Jenkins 플러그인 시스템의 장점과 문제점을 설명하고, 프로덕션 환경에서 안전한 플러그인 관리 전략을 제시하시오.**

### 핵심 포인트

- **장점**: Core를 최소화하고 필요한 기능만 플러그인으로 추가하므로, 시스템이 가볍고 유연하다. 1,800개 이상의 생태계로 거의 모든 시나리오를 커버한다.
- **의존성 충돌**: 플러그인 A와 B가 같은 라이브러리의 다른 버전을 요구하면 클래스로딩 수준에서 충돌이 발생한다. Jenkins의 클래스 격리가 완벽하지 않기 때문이다.
- **업데이트 연쇄 반응**: 하나의 플러그인 업데이트가 의존 플러그인의 최소 버전을 올려, 줄줄이 업데이트가 필요해지고, 이 과정에서 호환성이 깨질 수 있다.
- **관리 전략**: (1) `plugins.txt`로 버전 고정하여 재현 가능한 환경 유지, (2) 최소 필수 플러그인만 설치하여 충돌 확률과 공격 표면 최소화, (3) 스테이징 환경에서 먼저 테스트 후 프로덕션 반영.
- **보안 관점**: 플러그인이 많을수록 취약점 발생 확률이 높아지며, 의존성 문제로 보안 패치 적용이 지연되는 상황이 실제로 빈번하다.

### 심화 질문

- `plugins.txt` 없이 Jenkins를 운영하면 어떤 문제가 발생하는가? (환경 재현성, 드리프트)
- 보안 취약점이 발견된 플러그인의 업데이트가 의존성 문제로 불가능할 때 어떻게 대응하는가?

## Q4. JENKINS_HOME 백업 전략

**JENKINS_HOME 디렉토리에서 백업이 필수인 경로와 재생성 가능한 경로를 구분하고, 그 이유를 설명하시오.**

### 핵심 포인트

- **백업 필수**:
  - `config.xml`: 전역 보안 설정, 도구 설정, 환경변수가 포함된다. 잃으면 전체 시스템 설정을 처음부터 수동으로 재구성해야 한다.
  - `jobs/`: 잡 정의(`config.xml`)와 빌드 히스토리(`builds/`)를 포함한다. 잡 정의를 잃으면 모든 파이프라인을 재생성해야 하며, 빌드 기록(트렌드, 감사 로그)도 영구 유실된다.
  - `secrets/`: `master.key`와 `hudson.util.Secret`이 포함된다. 이 파일을 잃으면 Jenkins에 저장된 모든 크레덴셜(Git 토큰, AWS 키, DB 비밀번호 등)을 복호화할 수 없어, 모든 크레덴셜을 다시 등록해야 한다.
- **백업 권장**: `nodes/`(Agent 설정), `users/`(사용자 정보, API 토큰). 잃으면 수동으로 재등록 가능하지만 번거롭다.
- **백업 불필요**: `workspace/`는 빌드 시 자동으로 체크아웃하여 생성된다. 백업하면 저장 공간만 낭비된다.
- **핵심 원리**: "재생성할 수 없는 데이터"만 백업한다. `workspace/`는 Git에서 다시 받으면 되지만, `secrets/`의 암호화 키는 어디에서도 복구할 수 없다.

### 심화 질문

- `secrets/master.key`를 잃었을 때 크레덴셜 복구가 불가능한 기술적 이유는 무엇인가? (AES 암호화 키 특성)
- Jenkinsfile을 SCM에 저장하는 것과 `jobs/config.xml`에 저장하는 것의 차이는 무엇이며, 백업 전략에 어떤 영향을 주는가?

## Q5. 인증(Authentication)과 인가(Authorization)

**Jenkins의 인증과 인가의 차이를 설명하고, Matrix-based Security의 동작 방식을 구체적으로 설명하시오.**

### 핵심 포인트

- **인증(Authentication)**: "이 사용자가 누구인지" 확인하는 과정이다. Jenkins 내부 DB, LDAP, SAML, OAuth 중 선택하며, 대부분의 기업 환경에서는 LDAP 또는 SAML을 사용하여 기존 계정 시스템과 연동한다.
- **인가(Authorization)**: "이 사용자가 무엇을 할 수 있는지" 결정하는 과정이다. 인증을 통과한 것과 권한이 있는 것은 별개다. 로그인은 가능하지만 특정 잡을 빌드할 권한은 없을 수 있다.
- **Matrix-based Security**: 행(Row)에 사용자 또는 그룹을, 열(Column)에 권한 카테고리(Overall, Job, View, Agent 등)를 배치한 2차원 매트릭스로 관리한다. 체크박스 방식으로 세밀한 권한 제어가 가능하다. 예: `developer` 그룹에는 Job Read + Job Build만 부여하고, `admin` 그룹에는 전체 권한을 부여.
- **Project-based 확장**: Matrix-based Security만으로는 "팀 A는 팀 A의 잡만 볼 수 있고, 팀 B의 잡은 보이지 않는다"는 격리가 불가능하다. Project-based Matrix Authorization을 사용하면 잡 단위로 별도 권한을 설정하여 팀 간 격리를 구현할 수 있다.
- **Agent 보안**: Agent가 Controller의 파일시스템에 접근하지 못하도록 Agent→Controller Access Control을 반드시 활성화해야 한다. 이를 통해 악의적 빌드 스크립트가 `secrets/`에 접근하는 것을 차단한다.

### 심화 질문

- Role-based Strategy 플러그인이 Matrix-based Security보다 대규모 환경에서 유리한 이유는 무엇인가?
- Jenkins 크레덴셜 스토어의 암호화 방식(AES-128-CBC)의 한계는 무엇이며, Controller 파일시스템 접근 권한과 어떤 관계가 있는가?

## Q6. 라벨 기반 라우팅과 실행 전략

**Jenkins에서 라벨 기반 Agent 라우팅이 필요한 이유를 설명하고, `agent any`, `agent { label }`, `agent none`의 차이와 각각의 사용 시나리오를 제시하시오.**

### 핵심 포인트

- **라벨의 필요성**: 다양한 기술 스택(Java, Node.js, Python, .NET 등)의 빌드가 공존하는 환경에서, 적합한 도구가 설치된 Agent에서만 빌드가 실행되도록 보장하기 위해 라벨 기반 라우팅을 사용한다. Python 빌드가 Java Agent에서 실행되면 실패한다.
- **`agent any`**: 사용 가능한 아무 Agent에서 실행한다. 빌드 환경에 특별한 요구사항이 없는 간단한 작업에 적합하다. 위험: 의도하지 않은 Agent에서 실행될 수 있다.
- **`agent { label 'java17 && linux' }`**: 지정된 라벨 조합과 일치하는 Agent에서만 실행한다. 논리 연산자(`&&`, `||`)를 사용하여 정밀한 매칭이 가능하다.
- **`agent none`**: Pipeline 레벨에서 Agent를 지정하지 않고, 각 stage에서 개별적으로 Agent를 선택한다. 빌드 → 테스트 → 배포 각 단계가 서로 다른 환경을 요구할 때 사용한다. Controller에서는 Flyweight(오케스트레이션)만 실행된다.
- **Docker Agent**: `agent { docker { image 'node:18' } }`는 Agent 위에서 Docker 컨테이너를 띄워 그 안에서 빌드를 실행한다. Agent에 도구를 설치하지 않아도 되므로 환경 관리가 단순해진다.

### 심화 질문

- Kubernetes Plugin 환경에서 라벨 대신 Pod Template을 사용하는 방식은 어떤 장점이 있는가?
- `agent none` + stage별 Agent 지정 패턴에서 stage 간 데이터(워크스페이스)를 공유하려면 어떻게 해야 하는가? (`stash`/`unstash`)
