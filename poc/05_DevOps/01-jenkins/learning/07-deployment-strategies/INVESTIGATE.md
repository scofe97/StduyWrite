## Q1. Blue-Green, Canary, Rolling Update 배포 전략을 비교하고 각각 적합한 상황을 설명하시오.

**질문 의도**: 세 가지 배포 전략의 동작 원리와 트레이드오프를 이해하고 있는지 확인

### 핵심 포인트
- **Blue-Green**은 두 개의 동일한 환경을 운영하여 트래픽을 한 번에 전환하는 전략이다. 롤백이 단순한 스위칭이므로 수초 내에 가능하지만, 인프라 비용이 2배 필요하다. DB 스키마 변경 시 양쪽 버전 호환성을 보장해야 하므로 Expand-Contract 마이그레이션 패턴을 사용한다.
- **Canary**는 트래픽의 일부(1~5%)만 새 버전으로 보내고, 메트릭을 확인하며 점진적으로 비율을 높이는 전략이다. 실제 프로덕션 트래픽으로 검증할 수 있지만, 트래픽 분할 인프라(Istio, NGINX weighted routing)와 실시간 메트릭 비교 시스템이 필요하다.
- **Rolling Update**는 기존 인스턴스를 하나씩 새 버전으로 교체하는 전략이다. Kubernetes의 기본 전략이며 추가 리소스가 최소화되지만, 롤백 시 교체에 걸린 시간만큼 다시 소요된다. 배포 중 두 버전이 혼재한다.
- 선택 기준은 "롤백 속도 vs 리소스 비용 vs 구현 복잡도"의 트레이드오프이다. 미션 크리티컬 서비스는 Blue-Green, 대규모 사용자 서비스는 Canary, 내부 도구는 Rolling Update가 적합하다.

### 심화 질문
- Blue-Green 배포에서 DB 마이그레이션을 안전하게 수행하는 Expand-Contract 패턴을 구체적으로 설명하시오.
- Canary 배포에서 "이 버전이 안전하다"를 판단하는 기준은 무엇이며, 자동 승격(auto-promote)과 수동 승격의 장단점은 무엇인가?
- 10개의 Pod를 Rolling Update하는 중 6번째에서 장애가 감지되었다면, 이미 교체된 5개의 Pod는 어떻게 처리하는가?

## Q2. GitOps에서 Jenkins의 역할과 ArgoCD의 역할을 구분하고, 왜 Jenkins가 직접 K8s에 배포하지 않는 것이 좋은지 설명하시오.

**질문 의도**: GitOps 원칙과 Push/Pull 모델의 보안 차이를 이해하고 있는지 확인

### 핵심 포인트
- **Jenkins의 역할은 CI**(빌드, 테스트, 이미지 푸시, 매니페스트 저장소에 이미지 태그 업데이트 커밋)이다. Jenkins는 `kubectl`을 사용하지 않으며, 클러스터 접근 권한이 필요 없다.
- **ArgoCD의 역할은 CD**(매니페스트 저장소의 변경을 감지하여 클러스터에 동기화)이다. ArgoCD는 클러스터 내부에서 실행되며, Git 저장소만 읽는다.
- Jenkins가 직접 배포하지 않는 이유는 세 가지이다. 첫째, **보안**: Push 모델은 외부(Jenkins)에 클러스터 접근 권한을 두므로 공격 표면이 넓다. Pull 모델은 클러스터 내부에서 외부(Git)를 읽으므로 안전하다. 둘째, **감사 추적**: Git 커밋 히스토리가 곧 배포 이력이므로 "누가, 언제, 무엇을" 변경했는지 완전한 추적이 가능하다. 셋째, **Drift 감지**: ArgoCD는 주기적으로 클러스터 상태와 Git 상태를 비교하여, 누군가 `kubectl`로 직접 수정한 변경도 감지하고 원상 복구한다.
- Git 저장소가 "단일 진실의 원천(Single Source of Truth)"이 되므로, 클러스터의 현재 상태를 항상 Git에서 확인할 수 있다.

### 심화 질문
- ArgoCD가 아닌 Flux를 사용하는 경우 Jenkins와의 연동 방식에 차이가 있는가?
- GitOps 환경에서 긴급 롤백이 필요할 때 `git revert`와 ArgoCD UI에서의 수동 Sync 중 어느 방법을 선택해야 하며, 그 이유는 무엇인가?
- 소스코드 저장소와 매니페스트 저장소를 분리하는 이유는 무엇이며, 모노레포로 합치면 어떤 문제가 발생하는가?

## Q3. 프로덕션 배포 후 자동 롤백을 구현하는 방법을 헬스체크 관점에서 설명하시오.

**질문 의도**: 자동 롤백의 판단 기준과 Kubernetes 프로브 활용 능력을 확인

### 핵심 포인트
- 자동 롤백은 세 단계로 구성된다. **1단계: Kubernetes Readiness Probe**는 Pod가 트래픽을 받을 준비가 되었는지 확인한다. 새 버전의 Pod가 Readiness Probe를 통과하지 못하면 Service 엔드포인트에서 제거되어 트래픽을 받지 않는다. `kubectl rollout status`는 이 상태를 모니터링한다.
- **2단계: 스모크 테스트**는 배포 직후 핵심 비즈니스 플로우(로그인, API 호출, 주문 생성 등)를 실행하여 기능적 정상 동작을 확인한다. 프로브가 통과해도 비즈니스 로직에 버그가 있을 수 있기 때문이다.
- **3단계: 메트릭 기반 판단**은 배포 후 5~15분간 에러율, p99 레이턴시, 비즈니스 메트릭을 모니터링한다. 핵심은 절대값이 아니라 **"기존 대비 변화율"**로 판단하는 것이다. 에러율이 기존 대비 5배 증가하면 롤백을 트리거한다.
- Helm의 `--atomic` 플래그는 배포 실패 시 자동으로 이전 릴리스로 롤백하므로, 별도의 롤백 로직 없이도 안전한 배포가 가능하다.

### 심화 질문
- Readiness Probe에 DB 연결 확인, 캐시 워밍업 완료, 의존 서비스 연결 확인을 모두 포함하면 어떤 부작용이 발생할 수 있는가? (힌트: 연쇄 장애)
- "에러율 기존 대비 5배"라는 기준을 정할 때, 배포 직후의 메트릭 노이즈(웜업 기간)를 어떻게 처리하는가?
- Blue-Green 배포에서 Green 환경의 스모크 테스트가 실제 사용자 데이터에 영향을 주지 않도록 하려면 어떻게 설계해야 하는가?

## Q4. Feature Flag가 배포 전략에서 어떤 역할을 하는지, 롤백과의 차이를 설명하시오.

**질문 의도**: 배포와 릴리스의 분리 개념, Feature Flag의 실무 활용을 이해하고 있는지 확인

### 핵심 포인트
- Feature Flag는 **"배포"와 "릴리스"를 분리**하는 기법이다. 코드는 프로덕션에 배포하지만, 새 기능은 플래그로 비활성화한 상태를 유지한다. 릴리스는 플래그를 켜는 시점에 발생한다.
- **롤백과의 핵심 차이**: 롤백은 전체 배포를 되돌리므로 다른 기능에도 영향을 줄 수 있다. Feature Flag는 문제가 있는 기능만 비활성화하므로 나머지 기능은 영향을 받지 않는다. 수행 시간도 롤백(수초~수분)보다 플래그 토글(밀리초)이 훨씬 빠르다.
- Feature Flag의 활용 패턴은 네 가지이다. Release Flag(장기 브랜치 방지), Experiment Flag(A/B 테스트), Ops Flag(부하 시 기능 비활성화), Permission Flag(사용자 그룹별 기능 제어).
- **Canary와의 차이**: Canary는 인프라 레벨에서 트래픽을 분할(다른 Pod로 라우팅)하지만, Feature Flag는 애플리케이션 레벨에서 코드 경로를 분할(같은 Pod 내에서 분기)한다. 두 기법을 조합하면 더 세밀한 릴리스 제어가 가능하다.

### 심화 질문
- Feature Flag를 오래 방치하면 어떤 문제가 발생하며, "기술 부채로서의 Feature Flag"를 관리하는 전략은 무엇인가?
- Feature Flag가 켜진 상태와 꺼진 상태 모두를 테스트해야 한다면, 테스트 경우의 수가 기하급수적으로 늘어나는 문제를 어떻게 해결하는가?
- LaunchDarkly 같은 외부 SaaS Feature Flag 서비스가 장애를 일으키면, 애플리케이션의 모든 기능이 비활성화될 수 있는가? 이를 방지하는 설계는?

## Q5. Kubernetes 환경에서 Jenkins 파이프라인이 배포할 때 kubectl 크레덴셜을 안전하게 관리하는 방법을 설명하시오.

**질문 의도**: CI/CD 파이프라인의 보안 관행과 최소 권한 원칙의 적용 능력을 확인

### 핵심 포인트
- **가장 위험한 방식**: kubeconfig 파일을 Jenkins 서버에 직접 저장하거나, `cluster-admin` 권한의 ServiceAccount를 사용하는 것이다. Jenkins가 해킹되면 클러스터 전체가 노출된다.
- **권장 방식 1 (직접 배포 시)**: 전용 ServiceAccount를 생성하고, RBAC으로 필요한 네임스페이스와 리소스(Deployment, Service, ConfigMap)에 대한 최소 권한만 부여한다. 해당 토큰을 Jenkins Credentials에 Secret File로 저장하고, `withCredentials` 블록 내에서만 사용한다.
- **권장 방식 2 (GitOps)**: Jenkins가 클러스터에 직접 접근하지 않도록 아키텍처를 변경한다. Jenkins는 매니페스트 저장소에 커밋만 하고, ArgoCD가 클러스터 내부에서 배포를 수행한다. 이 방식이 보안상 가장 안전하다.
- **고급 방식**: HashiCorp Vault와 연동하여 동적 크레덴셜을 발급받는다. 크레덴셜이 짧은 TTL을 가지므로 유출되어도 피해가 제한된다. OIDC 연동으로 Jenkins의 ID를 기반으로 Kubernetes 인증을 처리하는 방식도 있다.
- 어떤 방식이든 **파이프라인 로그에 크레덴셜이 노출되지 않도록** `set +x`나 Jenkins의 Secret Masking 기능을 활용해야 한다.

### 심화 질문
- ServiceAccount 토큰이 만료되지 않는 Kubernetes 버전(1.23 이전)에서의 보안 위험과, 1.24 이후 도입된 Bound ServiceAccount Token의 차이를 설명하시오.
- Jenkins가 멀티 클러스터 환경에 배포해야 할 때, 각 클러스터의 크레덴셜을 어떻게 관리하는 것이 안전한가?
- Jenkins 에이전트가 Kubernetes Pod로 실행되는 경우, Pod의 ServiceAccount에 배포 권한을 부여하는 방식과 외부 크레덴셜을 주입하는 방식 중 어느 것이 더 안전한가?

## Q6. 마이크로서비스 환경에서 여러 서비스를 동시에 배포해야 할 때, 배포 순서와 호환성을 어떻게 관리하는가?

**질문 의도**: 실무에서 자주 발생하는 다중 서비스 배포의 복잡성 이해도를 확인

### 핵심 포인트
- 마이크로서비스 간 API 계약이 있으므로, **하위 호환성(Backward Compatibility)**을 유지하는 것이 원칙이다. 새 버전이 이전 버전의 API 요청도 처리할 수 있어야 한다. 그래야 배포 순서에 관계없이 안전하다.
- **API 버저닝**(v1, v2 공존)으로 하위 호환성을 보장하고, 이전 버전 API는 Deprecation 기간을 거쳐 제거한다.
- **의존성 방향의 역순으로 배포**하는 것이 안전하다. 예를 들어 A가 B를 호출하면, B를 먼저 배포한 후 A를 배포한다. B가 새 API를 제공해야 A가 사용할 수 있기 때문이다.
- **Contract Testing**(Pact 등)으로 서비스 간 API 호환성을 배포 전에 검증한다. CI 단계에서 계약 위반을 감지하면 배포를 차단한다.
- Feature Flag를 활용하면 코드는 먼저 배포하되, 새 API를 호출하는 로직은 모든 서비스 배포가 완료된 후에 플래그로 활성화할 수 있다.

### 심화 질문
- DB 스키마 변경이 포함된 배포에서, 새 스키마와 이전 코드 버전이 동시에 동작해야 하는 상황을 어떻게 처리하는가?
- 서비스 메시(Istio)의 트래픽 미러링 기능을 활용하여 새 버전의 호환성을 사전에 검증하는 방법을 설명하시오.
- 20개의 마이크로서비스가 있는 환경에서 배포 순서를 자동으로 결정하는 시스템을 어떻게 설계하겠는가?
