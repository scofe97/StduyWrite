## Q1. Multibranch Pipeline의 동작 원리를 설명하고, 일반 Pipeline Job과의 차이를 설명하시오.

**핵심 포인트:**

- **동작 원리**: Branch Source Plugin이 주기적으로(기본 1분) SCM을 스캔하여 `Jenkinsfile`이 존재하는 모든 브랜치를 탐지하고, 각 브랜치마다 독립적인 하위 Job을 자동으로 생성한다. 브랜치가 삭제되면 Orphaned Item Strategy에 따라 해당 Job도 자동 정리된다.
- **일반 Pipeline Job과의 차이**: 일반 Pipeline Job은 하나의 브랜치(또는 고정된 SCM 설정)에 대해 단일 Job을 수동으로 생성하는 반면, Multibranch Pipeline은 저장소 전체를 대상으로 브랜치 수만큼의 Job을 자동으로 관리한다. 따라서 새 브랜치를 만들 때 수동으로 Job을 설정할 필요가 없다.
- **PR 빌드**: GitHub/GitLab Branch Source Plugin과 연동하면 Pull Request가 열릴 때 소스를 타겟 브랜치에 머지한 상태로 빌드를 수행하여, 머지 전에 빌드 실패 여부를 확인할 수 있다. 빌드 결과는 PR 페이지에 상태 체크로 자동 표시된다.
- **Jenkinsfile 기반 일관성**: 모든 브랜치가 동일한 `Jenkinsfile`을 사용하므로 파이프라인 정의가 코드로 관리되며, 브랜치마다 파이프라인 설정이 달라지는 Configuration Drift를 방지한다.
- **자동화의 가치**: 10명 이상의 팀에서 동시에 20개 이상의 feature 브랜치가 활성화되는 상황에서, 수동 Job 관리는 누락과 불일치를 유발한다. Multibranch Pipeline은 이를 구조적으로 해결한다.

**심화 질문:**
- Multibranch Pipeline에서 `Jenkinsfile`의 위치를 루트가 아닌 다른 경로로 지정할 수 있는가? 어떤 상황에서 그런 설정이 필요한가?
- Orphaned Item Strategy를 "즉시 삭제"로 설정하면 어떤 문제가 발생할 수 있는가?
- 모노레포에서 Multibranch Pipeline을 사용할 때, 변경된 모듈만 빌드하려면 어떻게 설계해야 하는가?

## Q2. 파이프라인에서 병렬 실행을 사용할 때의 이점과 주의사항을 설명하시오.

**핵심 포인트:**

- **이점 - 시간 단축**: 독립적인 작업(단위 테스트, 통합 테스트, 보안 스캔, 린트)을 동시에 실행하면 전체 소요 시간이 "가장 오래 걸리는 작업의 시간"으로 수렴한다. 순차 실행 시 합산 18분이 병렬 실행 시 10분으로 줄어드는 식이다.
- **이점 - 빠른 피드백**: 개발자가 코드를 push한 후 결과를 기다리는 시간이 짧아져 개발 사이클이 빨라진다. 10분 대기와 20분 대기의 차이는 생산성에 직접적인 영향을 준다.
- **주의사항 - 워크스페이스 충돌**: 병렬 스테이지가 동일한 워크스페이스를 공유하면 파일 쓰기 레이스 컨디션이 발생할 수 있다. 각 병렬 스테이지에 별도의 `agent`를 지정하거나 `dir()` 블록으로 작업 디렉토리를 분리해야 한다.
- **주의사항 - 리소스 소비**: 4개의 병렬 스테이지가 각각 에이전트를 필요로 하면 에이전트 풀에서 4개의 슬롯을 동시에 점유한다. 에이전트 수가 제한된 환경에서는 다른 Job의 실행이 지연될 수 있으므로, 리소스 용량 계획이 필요하다.
- **failFast 전략**: `failFast true`는 하나의 스테이지 실패 시 나머지를 즉시 중단하여 빠른 피드백을 제공한다. 반면 `failFast false`(기본값)는 모든 스테이지를 완료까지 실행하여 전체 실패 목록을 한 번에 확인할 수 있게 한다. 상황에 따라 선택해야 한다.

**심화 질문:**
- 병렬 스테이지 중 일부만 선택적으로 failFast를 적용할 수 있는가?
- 병렬 실행 중 공유 리소스(데이터베이스, 외부 API)에 동시 접근할 때 발생할 수 있는 문제와 해결 방법은?
- 병렬 테스트의 결과를 하나의 리포트로 합치려면 어떤 접근이 필요한가?

## Q3. 프로덕션 배포 전 수동 승인 게이트를 구현하는 방법과 이유를 설명하시오.

**핵심 포인트:**

- **구현 방법**: `input` 스텝을 사용하여 파이프라인을 일시 중지하고 지정된 사용자의 승인을 기다린다. `submitter` 파라미터로 승인 권한을 특정 사용자나 그룹으로 제한하고, `timeout`으로 최대 대기 시간을 설정한다.
- **필요한 이유**: 프로덕션 배포는 되돌리기 어렵고 사용자에게 직접적인 영향을 미치는 작업이다. 자동화된 테스트가 모든 문제를 감지하지 못하므로, 스테이징에서의 수동 QA 검증 후 사람의 판단을 개입시켜 안전망을 확보한다.
- **에이전트 점유 방지**: `input`이 `steps` 블록 안에 있으면 대기 시간 동안 에이전트가 점유된다. `agent none`을 지정하거나 `input`을 스테이지 레벨의 `when` 조건과 조합하여 에이전트 없이 대기하는 패턴을 사용해야 한다.
- **timeout의 중요성**: 승인자가 부재중이거나 잊어버린 경우 파이프라인이 무한히 대기하면서 에이전트를 점유할 수 있다. `timeout(time: 2, unit: 'HOURS')`와 같이 적절한 타임아웃을 설정하여 리소스 낭비를 방지한다.
- **감사 추적**: `input`의 `parameters` 옵션으로 승인 사유를 입력받으면, 누가 언제 왜 배포를 승인했는지 기록이 남아 감사(audit) 요구사항을 충족한다.

**심화 질문:**
- 승인 게이트 없이 완전 자동화된 프로덕션 배포(Continuous Deployment)로 전환하려면 어떤 전제 조건이 필요한가?
- 여러 단계의 승인이 필요한 경우(예: QA팀 승인 → 운영팀 승인) 파이프라인을 어떻게 설계하겠는가?
- 야간이나 주말에 긴급 배포가 필요할 때 승인 게이트가 병목이 되는 문제를 어떻게 해결하겠는가?

## Q4. when {} 조건을 활용하여 브랜치 전략(Git Flow)에 맞는 파이프라인을 설계하시오.

**핵심 포인트:**

- **Git Flow 매핑**: Git Flow의 각 브랜치 유형에 따라 다른 파이프라인 동작을 `when { branch }` 조건으로 구현한다. `feature/*`는 빌드+테스트, `develop`은 Dev 환경 자동 배포, `release/*`는 Staging 배포+승인, `main`은 Production 배포, `hotfix/*`는 긴급 패치 경로를 탄다.
- **when 조건 종류의 활용**: `branch`는 브랜치 이름 매칭, `tag`는 릴리스 태그 감지, `changeset`은 변경 파일 경로 기반 조건, `expression`은 Groovy 표현식으로 복잡한 로직 구현, `allOf`/`anyOf`는 조건 조합에 사용한다.
- **beforeAgent 최적화**: GPU 노드나 대용량 빌드 서버처럼 비싼 리소스가 필요한 스테이지에서는 `beforeAgent true`를 설정하여 에이전트 할당 전에 조건을 평가함으로써 불필요한 리소스 점유를 방지한다.
- **changeset 활용**: 모노레포에서 `changeset "backend/**"`와 `changeset "frontend/**"`를 사용하면 변경된 부분만 빌드하여 CI 시간과 리소스를 절약할 수 있다.
- **설계 예시**:
  - `feature/*` → Test + Lint
  - `develop` → Test + Build + Deploy to Dev
  - `release/*` → Test + Build + Deploy to Staging + Approval
  - `main` → Deploy to Production (release 브랜치에서 머지된 검증 완료 코드)
  - `hotfix/*` → Test + Build + Deploy to Staging + Approval + Deploy to Production

**심화 질문:**
- Git Flow 대신 GitHub Flow(main + feature 브랜치만)를 사용하는 경우 파이프라인이 어떻게 단순화되는가?
- `when { changeset }` 사용 시 첫 빌드에서는 이전 커밋이 없어 변경 파일을 감지할 수 없는 문제를 어떻게 해결하는가?
- 하나의 Jenkinsfile에 모든 브랜치 조건을 넣으면 복잡도가 올라가는데, 이를 관리하기 위한 전략은 무엇인가?

## Q5. Matrix 빌드가 필요한 상황과 구현 방법을 설명하시오.

**핵심 포인트:**

- **필요한 상황**: 라이브러리나 SDK처럼 여러 런타임 버전(JDK 11/17/21)이나 OS(Ubuntu/Windows/macOS)에서 동작을 보장해야 하는 프로젝트에서 필수적이다. 수동으로 각 조합을 스테이지로 작성하면 축이 추가될 때마다 코드가 기하급수적으로 늘어나지만, Matrix는 축 선언만으로 모든 조합을 자동 생성한다.
- **구현 방법**: `matrix` 블록 안에 `axes`로 축을 정의하고, 각 축에 `values`를 나열한다. Jenkins가 자동으로 모든 조합(카르테시안 곱)을 생성하여 병렬로 실행한다.
- **excludes 활용**: 지원 종료된 환경이나 불가능한 조합(예: macOS + ARM에서 Windows 전용 테스트)을 제외할 수 있다. 이를 통해 불필요한 빌드를 줄이고 리소스를 절약한다.
- **에이전트 매핑**: 각 셀에 적합한 에이전트를 `agent { label "${OS} && jdk-${JDK_VERSION}" }` 형태로 동적으로 할당한다. 이를 위해 에이전트 노드에 적절한 라벨이 미리 설정되어 있어야 한다.
- **실무 고려사항**: Matrix의 각 셀이 독립적인 에이전트를 필요로 하므로, 3x3 Matrix는 최대 9개의 에이전트를 동시에 사용한다. 에이전트 풀 크기를 고려한 용량 계획이 필요하며, 클라우드 에이전트(EC2, Kubernetes Pod)를 활용하면 탄력적으로 대응할 수 있다.

**심화 질문:**
- Matrix 빌드 중 특정 조합에서만 추가 스테이지를 실행하려면 어떻게 설계하는가?
- Matrix 빌드의 결과를 하나의 호환성 매트릭스 리포트로 시각화하는 방법은?
- GitHub Actions의 matrix strategy와 Jenkins Matrix의 차이점은 무엇인가?

## Q6. Upstream/Downstream 트리거를 사용한 파이프라인 체이닝의 설계 원칙과 주의사항을 설명하시오.

**핵심 포인트:**

- **개념**: Upstream 프로젝트(예: 공통 라이브러리)의 빌드가 성공하면 Downstream 프로젝트(예: 이를 사용하는 서비스들)의 빌드를 자동으로 트리거하는 패턴이다. 마이크로서비스 아키텍처에서 공유 라이브러리 변경의 영향을 자동으로 검증할 수 있다.
- **구현 방법**: Upstream 파이프라인의 `post { success { } }` 블록에서 `build job: 'downstream-job'`을 호출하거나, Downstream 파이프라인의 `triggers` 블록에서 `upstream(upstreamProjects: 'upstream-job')`을 선언한다.
- **wait 파라미터**: `build job: 'service-a', wait: false`는 Downstream 빌드를 트리거만 하고 결과를 기다리지 않는다. `wait: true`는 결과를 기다리므로 Downstream 실패 시 Upstream도 실패로 표시된다. 사용 목적에 따라 선택한다.
- **순환 의존성 방지**: A가 B를 트리거하고 B가 A를 트리거하면 무한 루프가 발생한다. 의존성 그래프가 DAG(비순환 방향 그래프)를 유지하도록 설계해야 한다.
- **threshold 설정**: `upstream` 트리거의 `threshold` 파라미터로 어떤 결과(SUCCESS, UNSTABLE, FAILURE)에서 Downstream을 트리거할지 제어한다. 일반적으로 SUCCESS에서만 트리거하지만, UNSTABLE(테스트 일부 실패)에서도 트리거할지는 프로젝트 정책에 따라 결정한다.

**심화 질문:**
- 10개 이상의 마이크로서비스가 공통 라이브러리에 의존하는 상황에서, 모든 서비스를 동시에 트리거하면 발생할 수 있는 문제와 해결 방법은?
- Upstream/Downstream 대신 이벤트 기반(예: 메시지 큐) 파이프라인 트리거를 사용하는 것이 적합한 상황은?
- 다이아몬드 의존성(A → B, A → C, B → D, C → D)에서 D가 두 번 빌드되는 것을 방지하는 방법은?
