## Q1. Pipeline as Code와 팀 협업

**Pipeline as Code가 기존 Freestyle Job 대비 갖는 이점을 팀 협업 관점에서 설명하시오.**

### 핵심 포인트

- **변경 추적**: Freestyle Job은 UI에서 설정을 변경해도 누가 언제 왜 바꿨는지 기록이 남지 않는다. Jenkinsfile은 Git 커밋 히스토리로 모든 변경의 저자, 시점, 이유(커밋 메시지)를 추적할 수 있다.
- **코드 리뷰**: 빌드 파이프라인 변경도 PR 프로세스에 포함되므로, 팀원이 파이프라인 변경을 리뷰하고 위험한 설정 변경(예: 프로덕션 배포 조건 완화)을 사전에 발견할 수 있다.
- **재현성**: Jenkins 서버 장애나 마이그레이션 시 Freestyle Job 설정은 수동 복구가 필요하지만, Jenkinsfile은 Git에서 clone하면 즉시 복원된다.
- **온보딩**: 신규 입사자가 저장소의 Jenkinsfile을 읽으면 CI/CD 흐름을 코드 수준에서 이해할 수 있다. git blame으로 각 설정의 맥락까지 파악 가능하다.
- **일관성**: 하나의 Jenkinsfile이 모든 브랜치에 적용되므로, 환경별로 개별 Job을 만들어 관리하는 설정 파편화 문제가 사라진다.

### 심화 질문

- Jenkinsfile을 팀에서 공유할 때 Shared Library와의 관계는 무엇인가? 공통 로직을 어떻게 중앙화하는가?
- Jenkinsfile 변경이 특정 브랜치에만 적용되어야 할 때(예: feature 브랜치에서 파이프라인 실험) 어떻게 안전하게 테스트하는가?

## Q2. Declarative vs Scripted Pipeline

**Declarative Pipeline과 Scripted Pipeline의 차이를 설명하고, 각각 적합한 상황을 제시하시오.**

### 핵심 포인트

- **구조적 차이**: Declarative는 `pipeline {}` 블록 안에 정해진 구조(agent, stages, post)를 따르는 DSL이다. Scripted는 `node {}` 안에서 순수 Groovy 코드를 자유롭게 작성한다.
- **사전 검증**: Declarative는 Jenkins가 실행 전에 문법 구조를 검증하여 잘못된 파이프라인이 실행되는 것을 방지한다. Scripted는 실행 시점에만 오류를 발견할 수 있어 디버깅 비용이 높다.
- **에러 핸들링**: Declarative는 `post {}` 블록으로 성공/실패 후처리를 선언적으로 정의한다. Scripted는 `try-catch-finally`를 직접 구현해야 한다.
- **stage 재시작**: Declarative는 실패한 특정 stage부터 재시작이 가능하지만, Scripted는 전체 파이프라인을 처음부터 다시 실행해야 한다.
- **Declarative 적합 상황**: 일반적인 CI/CD 흐름(빌드 → 테스트 → 분석 → 배포)처럼 순차적이고 예측 가능한 파이프라인. 대부분의 프로젝트가 여기에 해당한다.
- **Scripted 적합 상황**: 런타임에 동적으로 스테이지를 생성해야 하거나, 외부 API 응답에 따라 복잡한 분기가 필요한 경우. 또는 기존 Groovy 라이브러리를 직접 호출해야 하는 경우.
- **90/10 법칙**: 실무에서 90%는 Declarative로 충분하고, 나머지 10%의 복잡한 로직은 `script {}` 블록으로 Groovy 코드를 삽입하여 해결한다. 전체를 Scripted로 작성할 필요는 거의 없다.

### 심화 질문

- Declarative Pipeline 안에서 `script {}` 블록을 과도하게 사용하면 사실상 Scripted Pipeline과 다를 바 없어진다. 이 경계를 어떻게 관리할 것인가?
- Declarative Pipeline의 구조적 제약이 오히려 장점이 되는 구체적 사례를 설명할 수 있는가?

## Q3. 크레덴셜 관리

**Jenkinsfile에서 `credentials()`를 사용하는 이유와 평문 비밀값을 사용하면 안 되는 이유를 설명하시오.**

### 핵심 포인트

- **Git 히스토리 영속성**: Jenkinsfile에 비밀값을 평문으로 작성하면 Git 히스토리에 영구히 남는다. 이후 커밋으로 삭제해도 과거 커밋에서 여전히 조회 가능하다. 완전한 제거를 위해서는 `git filter-branch`나 BFG Repo-Cleaner 같은 히스토리 재작성이 필요하며, 이는 팀 전원에게 영향을 준다.
- **로그 마스킹**: `credentials()` 함수로 바인딩된 값은 Jenkins 콘솔 로그에서 자동으로 `****`로 마스킹된다. 평문 값은 빌드 로그에 그대로 출력되어, 로그 접근 권한이 있는 누구든 비밀값을 볼 수 있다.
- **중앙 관리**: Jenkins Credentials Store에서 비밀값을 중앙 관리하면, 비밀값 변경 시 Jenkinsfile 수정 없이 Jenkins UI에서만 업데이트하면 된다. 여러 파이프라인이 동일한 credential ID를 참조하므로 일관성이 보장된다.
- **스코프 제어**: `credentials()`는 파이프라인 전체에, `withCredentials`는 특정 블록 안에서만 비밀값을 노출한다. 필요한 범위에서만 최소한으로 접근할 수 있어 최소 권한 원칙(Principle of Least Privilege)을 구현한다.
- **접근 제어**: Jenkins Credentials는 폴더 단위로 접근 권한을 설정할 수 있다. 특정 팀의 파이프라인에서만 접근 가능한 비밀값을 설정하여 조직 단위 보안을 구현한다.

### 심화 질문

- `credentials()`과 `withCredentials`의 스코프 차이를 구체적으로 설명하고, 각각 언제 사용하는 것이 적절한가?
- Jenkins Credentials Store 외에 HashiCorp Vault 같은 외부 비밀 관리 시스템과 연동하는 이유는 무엇인가? Jenkins 자체 저장소만으로 부족한 상황은 언제인가?

## Q4. Post Actions

**Post actions의 종류와 각각의 실무 활용 패턴을 설명하시오.**

### 핵심 포인트

- **always**: 빌드 성공/실패와 무관하게 반드시 실행된다. 워크스페이스 정리(`cleanWs()`), 테스트 결과 수집(`junit`), 아티팩트 아카이브가 대표적 용도다. always가 없으면 실패한 빌드의 워크스페이스가 디스크를 계속 점유하여 에이전트의 저장 공간이 고갈된다.
- **success**: 파이프라인이 성공했을 때만 실행된다. Slack/Teams 성공 알림, 다음 파이프라인 트리거(CD 연계), 배포 완료 알림 등에 사용한다.
- **failure**: 파이프라인이 실패했을 때만 실행된다. Slack 실패 알림(@here 멘션으로 즉각 대응 유도), 디버그 로그 아카이브, Jira 이슈 자동 생성 등에 사용한다. 빠른 장애 인지를 위해 가장 중요한 post 조건이다.
- **unstable**: 테스트 일부 실패 등으로 빌드가 불안정 상태가 되었을 때 실행된다. 테스트 리포트를 담당자에게 발송하거나, 코드 커버리지 하락 알림을 보내는 데 활용한다.
- **changed**: 이전 빌드 결과와 현재 결과가 달라졌을 때 실행된다. "빌드가 다시 정상으로 돌아왔습니다" 또는 "이전까지 성공하던 빌드가 실패하기 시작했습니다" 같은 상태 전환 알림에 사용한다.
- **aborted**: 사용자가 빌드를 수동으로 중단하거나 타임아웃으로 중단되었을 때 실행된다. 점유하고 있던 외부 리소스(테스트 DB, 임시 인프라 등)를 해제하는 데 사용한다.
- **실행 순서**: 결과별 조건(success, failure 등)이 먼저 실행되고, 그 후에 always가 실행된다. 이 순서를 이해해야 리소스 정리 로직을 올바르게 배치할 수 있다.

### 심화 질문

- `post` 블록은 파이프라인 레벨뿐 아니라 개별 stage 레벨에서도 정의할 수 있다. 각각 어떤 상황에서 사용하는 것이 적절한가?
- Scripted Pipeline에서 동일한 후처리 로직을 구현하려면 어떻게 해야 하는가? Declarative의 `post`가 제공하는 구조적 이점은 무엇인가?

## Q5. script {} 블록의 사용과 남용

**Declarative Pipeline에서 `script {}` 블록을 사용해야 하는 상황과 남용했을 때의 문제를 설명하시오.**

### 핵심 포인트

- **script {} 블록의 역할**: Declarative Pipeline의 구조적 제약 안에서 순수 Groovy 코드를 실행할 수 있는 탈출구다. 반복문, 조건 분기, 변수 조작, 외부 API 호출 등 Declarative DSL로 표현할 수 없는 로직에 사용한다.
- **적절한 사용 상황**:
  - 런타임에 동적으로 병렬 스테이지를 생성해야 할 때 (예: `services.json`에서 서비스 목록을 읽어 각각 빌드)
  - 외부 API 응답 값에 따라 복잡한 분기를 처리해야 할 때
  - 빌드 결과에 따라 다른 잡을 조건부로 트리거해야 할 때
  - 리스트를 순회하며 반복 작업을 수행해야 할 때
- **남용의 문제**:
  - **사전 검증 무력화**: `script {}` 블록 안의 Groovy 코드는 Jenkins의 사전 문법 검증 대상이 아니다. 실행 시점에만 오류를 발견할 수 있어 디버깅 비용이 증가한다.
  - **시각화 손실**: Blue Ocean 등 UI 도구가 `script {}` 블록 내부의 실행 흐름을 시각적으로 표현하지 못한다. 파이프라인 가시성이 떨어진다.
  - **stage 재시작 불가**: `script {}` 블록이 과도하게 사용된 stage는 재시작이 안정적으로 동작하지 않을 수 있다.
  - **가독성 저하**: Declarative Pipeline의 장점인 선언적 구조가 무너지면서 코드를 읽는 사람이 파이프라인 흐름을 파악하기 어려워진다. 사실상 Scripted Pipeline과 다를 바 없어진다.
- **대안 우선 적용**: `script {}` 블록을 작성하기 전에 Declarative의 선언적 기능(`when`, `environment`, `post`, `matrix`)으로 해결할 수 있는지 먼저 검토해야 한다. 선언적 기능으로 불가능한 경우에만 `script {}`를 사용한다.

### 심화 질문

- 팀에서 `script {}` 블록의 사용을 제한하려면 어떤 가이드라인을 수립해야 하는가? 코드 리뷰에서 어떤 기준으로 판단하는가?
- `script {}` 블록에 들어가는 로직이 복잡해질 경우, Shared Library로 추출하는 것이 더 나은 선택인 이유는 무엇인가?

## Q6. 파이프라인 스테이지 순서 설계

**CI/CD 파이프라인에서 Build, Test, Analysis, Docker Build, Deploy 스테이지를 이 순서로 배치하는 이유를 "Fail Fast" 원칙과 연결하여 설명하시오.**

### 핵심 포인트

- **Fail Fast 원칙**: 실패할 가능성이 있는 작업 중 비용이 낮은 것을 먼저 실행하여, 불필요한 리소스 소모를 최소화하는 원칙이다. 컴파일(수초)이 실패하면 테스트(수분)를 실행할 필요가 없고, 테스트가 실패하면 Docker 이미지 빌드(수분)와 배포(수분~수십분)를 실행할 이유가 없다.
- **피드백 루프 최소화**: 개발자에게 실패를 최대한 빨리 알려줘야 수정 비용이 낮다. 컴파일 에러를 30초 만에 알 수 있는데, Docker 빌드까지 기다린 후 10분 후에 알리면 개발자는 이미 다른 작업을 시작한 상태다.
- **단계별 분리의 가치**: 빌드 실패인지, 테스트 실패인지, 코드 품질 문제인지를 한 눈에 구분할 수 있다. 하나의 스테이지에 모든 것을 넣으면 콘솔 로그를 파싱해야 원인을 찾을 수 있다.
- **의존 관계 명시**: SonarQube는 테스트 커버리지 데이터(JaCoCo)를 분석하므로 테스트 이후에 배치한다. Docker Build는 빌드 아티팩트(JAR)가 필요하므로 Build 이후에 배치한다. 스테이지 순서가 곧 데이터 흐름을 나타낸다.
- **게이트 역할**: 각 스테이지는 다음 스테이지로 진입하기 위한 품질 관문(Quality Gate) 역할을 한다. 컴파일 통과 → 테스트 통과 → 품질 기준 충족 → 이미지 생성 → 배포 승인. 이 관문들이 불량 코드가 프로덕션에 도달하는 것을 계층적으로 방어한다.

### 심화 질문

- 파이프라인 전체 실행 시간이 30분을 넘길 때, 스테이지 순서를 유지하면서 속도를 개선할 수 있는 방법은 무엇인가? (힌트: 병렬 실행, 캐싱)
- "모든 검증을 통과한 코드만 배포한다"는 원칙과 "빠른 배포"라는 요구가 충돌할 때, 어떻게 균형을 잡는가?
