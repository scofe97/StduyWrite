# 내부 클래스에서 람다까지: Deep Investigation

> 본 문서의 학습 자료를 읽은 뒤, 실무와 면접에서 깊이 있게 다뤄야 할 질문들

## Q1. 람다가 캡처하는 변수가 effectively final이어야 하는 메모리 모델 근거는 무엇인가

### 왜 이 질문이 중요한가
람다의 변수 캡처 제약은 단순한 문법 규칙이 아니라 JMM(Java Memory Model)의 가시성 보장과 직결된다. 이 근거를 모른다면 "컴파일 에러가 나서 final을 붙인다"는 수준에 그치고, 멀티스레드 환경에서의 람다 사용 시 미묘한 버그를 놓치게 된다.

### 답변
람다는 클로저(closure)이지만 Java는 값 캡처(capture by value)만 허용한다. 람다가 지역 변수를 캡처할 때 그 변수의 현재 값을 복사해서 람다 인스턴스 내부 필드로 저장한다. 이 복사본이 원본과 항상 동일하다는 보장을 위해 effectively final 조건이 필요하다. 만약 변수가 변경 가능하다면 람다가 읽는 복사본과 외부에서 변경되는 원본 사이에 가시성 불일치가 발생한다.

JMM 관점에서 지역 변수는 스택에 할당되고 각 스레드가 독립된 스택을 가진다. 람다가 별도 스레드에서 실행될 때 지역 변수를 직접 참조하면 이미 스택 프레임이 사라진 후일 수 있다. 값 복사 방식은 이 문제를 우회하지만, 복사 시점 이후 원본이 변경된다면 복사본의 신선도(freshness)를 보장할 방법이 없다. effectively final은 "복사 시점과 사용 시점의 값이 항상 동일"함을 컴파일러가 정적으로 검증하는 장치다.

인스턴스 변수나 클래스 변수는 이 제약을 받지 않는다. 힙에 존재하기 때문에 람다가 `this` 참조(익명 클래스의 경우)나 외부 인스턴스 참조를 통해 직접 접근할 수 있고, 가시성은 `volatile`이나 `synchronized` 같은 동기화 수단으로 별도 관리한다.

```java
// 컴파일 에러: count는 effectively final이 아님
int count = 0;
Runnable r = () -> System.out.println(count); // OK — 변경 없으면 effectively final
count++; // 이 줄이 있으면 위 람다에서 컴파일 에러 발생

// 힙 객체는 참조만 final이면 됨 — 내부 상태 변경은 허용
List<String> list = new ArrayList<>(); // list 자체는 effectively final
Runnable r2 = () -> list.add("item");  // 컴파일 OK
```

실무에서 AtomicInteger나 배열을 "우회 수단"으로 사용하는 패턴을 자주 보는데, 이는 effectively final 제약을 피하되 가시성 보장을 프로그래머가 직접 책임지는 방식이다. 특히 병렬 스트림에서 이런 우회 패턴을 쓰면 경쟁 조건이 발생하므로, 올바른 해결책은 collect나 reduce 같은 스트림 API를 활용하는 것이다.

## Q2. 익명 클래스와 람다의 바이트코드 차이는 무엇이며 실무에 어떤 영향을 주는가

### 왜 이 질문이 중요한가
람다가 "단순히 익명 클래스의 문법적 설탕"이라고 오해하는 개발자가 많다. 바이트코드 수준의 차이를 알면 람다의 성능 특성, 직렬화 동작, 클래스 로딩 비용을 정확히 예측할 수 있어 라이브러리 설계나 성능 최적화 시 올바른 선택을 한다.

### 답변
익명 클래스는 컴파일 시 별도의 `.class` 파일(`Outer$1.class`)을 생성한다. 각 인스턴스 생성마다 `new` 바이트코드가 실행되고 클래스 로더가 해당 클래스를 로드한다. 반면 람다는 `invokedynamic` 명령어 하나로 변환된다. JVM이 처음 해당 지점을 실행할 때 bootstrap method(`LambdaMetafactory.metafactory`)를 호출해 람다를 구현하는 클래스를 동적으로 생성하고 이후 호출은 캐시된 구현체를 재사용한다.

```
// 익명 클래스 바이트코드
new #2           // 클래스 로드 + 인스턴스 생성
invokespecial #3 // 생성자 호출

// 람다 바이트코드
invokedynamic #4 // bootstrap method 호출 (첫 실행 시에만 실제 작업 수행)
```

이 차이가 실무에 주는 영향은 세 가지다. 첫째, 클래스 파일 수가 줄어 JAR 크기와 클래스 로딩 시간이 감소한다. 대규모 코드베이스에서 익명 클래스를 람다로 전환하면 초기 구동 시간이 단축된다. 둘째, `this` 의미가 다르다. 익명 클래스 내부의 `this`는 익명 클래스 인스턴스를 가리키지만, 람다 내부의 `this`는 람다를 감싸는 외부 클래스 인스턴스를 가리킨다. 셋째, 직렬화 동작이 다르다. 익명 클래스는 `Serializable`을 구현하면 직렬화되지만, 람다의 직렬화는 특별한 위험을 가지며 별도로 주의해야 한다.

```java
// this의 차이
class Outer {
    Runnable anon = new Runnable() {
        public void run() {
            System.out.println(this); // Outer$1 인스턴스
        }
    };
    Runnable lambda = () -> {
        System.out.println(this); // Outer 인스턴스
    };
}
```

성능 측면에서 람다가 항상 빠르지는 않다. 변수를 캡처하는 람다는 매번 새 인스턴스를 생성하지만, 캡처하지 않는 람다(stateless lambda)는 JVM이 싱글턴으로 최적화할 수 있다.
