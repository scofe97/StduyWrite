# 효율적 동시성
---
> JVM이 멀티코어 하드웨어의 메모리 모델 차이를 어떻게 추상화하는지, 그리고 동시성 성능을 높이기 위해 락을 어떻게 최적화하는지 이해한다. 하드웨어 캐시 일관성 문제, JMM의 설계 의도, 락 최적화 기법을 다룬다.

## 1. 하드웨어 효율성과 일관성

멀티프로세서 시스템에서 각 CPU 코어는 연산 속도와 메모리 IO 속도의 격차를 줄이기 위해 자신만의 캐시 계층(L1/L2/L3)을 갖는다. 문제는 여러 코어가 동일한 메인 메모리 영역을 각자의 캐시에 복사하여 사용할 때 발생한다. 한 코어가 값을 수정하면 다른 코어의 캐시에는 아직 이전 값이 남아 있다. 이것이 **캐시 일관성(Cache Coherence)** 문제다.

이를 해결하기 위해 프로세서는 정해진 프로토콜을 따라 캐시를 관리한다. **MESI 프로토콜**이 가장 널리 사용되며, 캐시 라인을 네 가지 상태로 관리한다:

- **Modified**: 이 코어만 수정한 데이터, 메인 메모리와 다르다
- **Exclusive**: 이 코어만 보유 중이며 메인 메모리와 같다
- **Shared**: 여러 코어가 읽기 전용으로 공유 중이다
- **Invalid**: 다른 코어가 수정하여 이 캐시 라인은 무효다

캐시 외에도 프로세서는 **비순차 실행(Out-of-Order Execution)** 최적화를 통해 성능을 높인다. 입력 코드 순서와 다른 순서로 명령어를 실행하되, 최종 결과는 순차 실행과 동일하게 재구성한다. JIT 컴파일러의 **명령어 재정렬(Instruction Reordering)**도 같은 목적으로 수행된다.

## 2. 자바 메모리 모델(JMM)

자바 메모리 모델(Java Memory Model, JMM)은 Java 프로그램이 다양한 하드웨어와 운영체제의 메모리 모델 차이로부터 보호받도록 정의된 명세다. C/C++이 하드웨어 메모리 모델을 개발자가 직접 다뤄야 하는 것과 달리, JMM은 이를 추상화하여 이식성과 안전성을 보장한다.

### 2-1. 메인 메모리와 작업 메모리

JMM은 모든 변수(인스턴스 변수, 정적 변수, 배열 요소)를 **메인 메모리(Main Memory)**에 저장하고, 각 스레드는 자신만의 **작업 메모리(Working Memory)**를 갖는다. 작업 메모리는 메인 메모리에서 복사한 변수 사본을 포함하며, 스레드는 직접 메인 메모리가 아닌 자신의 작업 메모리에서만 변수를 읽고 쓴다.

스레드 간 변수 교환은 메인 메모리를 경유해야만 가능하다. 이 구조가 가시성 문제와 원자성 문제의 근원이다.

### 2-2. 원자성, 가시성, 실행 순서

JMM이 동시성에서 보장해야 하는 세 가지 특성이다:

**원자성(Atomicity)**: JMM이 직접 보장하는 원자적 연산은 읽기, 적재, 할당, 사용, 저장, 쓰기 여섯 가지다. `int` 읽기/쓰기는 원자적이지만 `long`/`double`은 32비트 플랫폼에서 두 번의 32비트 연산으로 나뉠 수 있다. `i++`는 읽기-증가-쓰기 세 단계로 구성되어 원자적이지 않다.

**가시성(Visibility)**: 한 스레드가 공유 변수를 수정하면 다른 스레드가 즉시 수정 결과를 볼 수 있어야 한다. `volatile`은 수정 즉시 메인 메모리에 동기화하고, 사용 시마다 메인 메모리에서 새로 읽도록 강제하여 가시성을 보장한다. `synchronized` 블록은 락 해제 전 반드시 메인 메모리에 동기화한다.

**실행 순서(Ordering)**: 현재 스레드에서는 모든 연산이 순서대로 실행되는 것처럼 보이지만, 다른 스레드에서 관찰하면 명령어 재정렬로 인해 순서가 다를 수 있다. `volatile`의 메모리 배리어는 재정렬을 방지하고, `synchronized`는 락을 통해 재정렬을 억제한다.

### 2-3. volatile 변수

`volatile`은 가볍고 효율적인 동기화 메커니즘이다. 두 가지를 보장한다:

- **가시성**: 수정 즉시 메인 메모리에 플러시하고, 읽기 시 항상 메인 메모리에서 최신값을 가져온다
- **명령어 재정렬 방지**: `volatile` 변수 접근 앞뒤로 메모리 배리어(Memory Barrier)를 삽입하여 재정렬을 막는다

단, `volatile`은 원자성을 보장하지 않는다. `i++` 같은 복합 연산은 여전히 경쟁 조건이 발생한다.

```java
// 가시성 문제 — 일반 변수
boolean flag = false;
// Thread A: flag = true  → Thread B의 캐시에는 아직 false

// volatile로 해결
volatile boolean flag = false;
// Thread A: flag = true  → 즉시 메인 메모리 반영 → Thread B 즉시 인식
```

## 3. 락 최적화

Java의 `synchronized`는 JDK 1.6 이전까지 OS 뮤텍스를 직접 사용하는 **중량 락(Heavyweight Lock)**이었다. 락 획득마다 커널 모드 전환이 발생하여 비용이 컸다. JDK 1.6부터 핫스팟은 경합 수준에 따라 락을 자동으로 업그레이드하는 **적응형 락(Adaptive Locking)**을 도입했다.

### 3-1. 편향 락(Biased Lock)

락이 거의 항상 동일한 스레드에 의해 획득되는 패턴을 최적화한다. 처음 락을 획득한 스레드 ID를 객체 헤더의 Mark Word에 기록하여, 이후 동일 스레드가 락을 요청할 때 CAS 연산 없이 즉시 진입한다. 다른 스레드가 경합하면 편향을 취소하고 경량 락으로 전환한다. JDK 15부터 deprecated되었으며 JDK 21에서 제거되었다.

### 3-2. 경량 락(Lightweight Lock)

여러 스레드가 서로 다른 시간대에 락을 획득하는 교대 패턴(경합이 적은 경우)에서 사용된다. 커널 뮤텍스 대신 **CAS(Compare-And-Swap)** 연산으로 객체 헤더를 업데이트하여 락을 획득한다. 두 스레드가 동시에 CAS를 시도하면 한쪽이 실패하고, 실패한 쪽은 스핀 락으로 전환한다.

### 3-3. 중량 락(Heavyweight Lock)

실제 경합이 발생하면 OS 뮤텍스를 사용하는 중량 락으로 승격된다. 대기 스레드는 커널 수준 대기 큐에 블록되어 CPU를 사용하지 않는다. 경량 락에서 스핀이 일정 횟수 이상 실패하거나, 두 스레드 이상이 동시에 대기하면 중량 락으로 전환된다.

### 3-4. 스핀 락과 적응형 스핀

스레드가 락을 기다리는 동안 컨텍스트 스위치 없이 CPU를 점유하며 반복 시도하는 방식이다. 락 보유 시간이 매우 짧으면 컨텍스트 스위치 비용보다 스핀 비용이 작다. **적응형 스핀**은 이전 스핀 시도의 성공 이력을 보고 스핀 횟수를 동적으로 조절한다.

락 상태 전이 흐름은 다음과 같다:

```
객체 생성(무락)
    ↓ (처음 동기화 진입)
편향 락 — 동일 스레드 재진입: 비용 없음
    ↓ (다른 스레드 경합 발생)
경량 락 — CAS 연산으로 획득 시도
    ↓ (동시 경합 발생)
스핀 락 — CPU 점유하며 재시도
    ↓ (스핀 실패 반복)
중량 락 — OS 뮤텍스, 커널 대기 큐
```

## 4. 락 제거(Lock Elision)

이스케이프 분석을 통해 특정 객체가 메서드 경계를 벗어나지 않음이 확인되면, JIT 컴파일러는 해당 객체의 락 연산을 완전히 제거한다. 다른 스레드가 접근할 수 없는 객체의 동기화는 의미가 없기 때문이다.

```java
public String concat(String s1, String s2) {
    StringBuffer sb = new StringBuffer();
    sb.append(s1);   // synchronized append
    sb.append(s2);   // synchronized append
    return sb.toString();
}
// sb는 이 메서드를 벗어나지 않으므로
// JIT 컴파일러가 synchronized를 모두 제거한다
```

## 5. 락 조대화(Lock Coarsening)

여러 개의 락 획득/해제 연산이 연속되면 JIT 컴파일러는 이를 하나의 큰 락 범위로 병합한다. 락의 잦은 획득/해제 오버헤드를 줄이기 위한 최적화다.

```java
// 최적화 전: 락 3번 획득/해제
for (int i = 0; i < 100; i++) {
    synchronized (lock) {
        list.add(i);
    }
}

// 락 조대화 후: 락 1번 획득/해제
synchronized (lock) {
    for (int i = 0; i < 100; i++) {
        list.add(i);
    }
}
```

락 조대화와 락 제거는 개발자가 명시적으로 요청하지 않아도 JIT 컴파일러가 자동으로 적용한다. 불필요하게 세밀하게 쪼갠 `synchronized` 블록이나 탈출하지 않는 공유 객체의 동기화를 걱정하지 않아도 되는 이유가 여기 있다.
