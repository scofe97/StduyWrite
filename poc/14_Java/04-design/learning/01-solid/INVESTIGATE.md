# SOLID 원칙: Deep Investigation

> 본 문서의 학습 자료를 읽은 뒤, 실무와 면접에서 깊이 있게 다뤄야 할 질문들

## Q1. SRP에서 "책임"의 범위를 어떻게 정하는가?

### 왜 이 질문이 중요한가

"책임"은 직관적으로 이해하기 쉽지만 실무에서 경계를 긋기가 어렵다. 너무 잘게 나누면 클래스가 폭발적으로 늘어나고, 너무 크게 잡으면 SRP를 위반한다. 면접에서 "SRP를 어떻게 적용했나요?"라는 질문에 단순히 "하나의 기능만 담당하게 했어요"라고 답하면 원칙의 본질을 이해하지 못한 것으로 평가받는다.

### 답변

SRP에서 "책임"은 기능이 아니라 **변경의 이유(reason to change)**이며, 더 정확히는 **변경을 요구하는 이해관계자(actor)**를 의미한다. 로버트 마틴은 이를 "하나의 클래스는 하나의 액터에 대해서만 책임을 져야 한다"고 재정의했다.

실무에서 책임 경계를 잡는 방법은 "이 코드가 바뀌는 이유가 누구의 요청인가?"를 묻는 것이다. 예를 들어 `Employee` 클래스에 `calculatePay()`와 `reportHours()`가 함께 있다면, 전자는 재무팀의 요청으로 바뀌고 후자는 인사팀의 요청으로 바뀐다. 두 액터가 다르므로 두 책임이다.

```java
// 위반: 두 액터(재무팀, 인사팀)의 요청이 한 클래스에 혼재
class Employee {
    double calculatePay() { /* 재무팀 규칙 */ }
    void reportHours()    { /* 인사팀 규칙 */ }
}

// 준수: 액터별로 분리
class PayCalculator   { double calculate(Employee e) { /* 재무팀 */ } }
class HourReporter    { void report(Employee e)       { /* 인사팀 */ } }
```

과도한 분리의 함정도 있다. "한 메서드만 있으면 SRP 준수"라는 착각이 대표적이다. 같은 액터의 요청으로 함께 바뀌는 메서드들은 같은 클래스에 있어야 응집도가 높아진다. SRP의 목표는 클래스 수를 줄이는 것이 아니라 **변경이 퍼지는 범위를 최소화**하는 것이다.

## Q2. OCP를 과도하게 적용하면 생기는 문제는 무엇인가?

### 왜 이 질문이 중요한가

OCP는 "모든 코드에 적용하라"가 아니다. 추상화 지점을 잘못 선택하면 오히려 코드가 복잡해지고 이해하기 어려워진다. OCP의 한계를 이해하는 것이 올바른 적용의 전제조건이다.

### 답변

OCP를 과도하게 적용하면 세 가지 문제가 생긴다.

첫째, **불필요한 추상화 계층**이 쌓인다. 변경될 가능성이 낮은 코드에도 인터페이스를 만들면, 구현체를 찾기 위해 인터페이스 → 구현체를 계속 따라가야 한다. "모든 클래스에 인터페이스를 만들어야 한다"는 잘못된 믿음에서 비롯된다.

둘째, **YAGNI(You Ain't Gonna Need It) 위반**이다. 확장 가능성을 위해 미리 인터페이스를 만들었지만 결국 구현체가 하나밖에 없는 경우, 추상화가 오히려 복잡성을 높인다.

셋째, **잘못된 추상화가 기술 부채**가 된다. 확장 지점을 잘못 예측하면, 실제 변경이 일어날 때 인터페이스 자체를 바꿔야 하는 상황이 생긴다. 이는 OCP를 적용하지 않은 것보다 더 큰 비용이다.

올바른 접근은 "세 번 반복되면 추상화한다(Rule of Three)"이다. 변경이 실제로 발생한 뒤에 패턴을 파악하고 추상화하는 것이, 변경을 예측하고 미리 추상화하는 것보다 훨씬 안전하다.

```java
// 과도한 OCP: 구현체가 하나뿐인데 인터페이스를 만든 경우
interface UserNameFormatter { String format(String name); }
class DefaultUserNameFormatter implements UserNameFormatter {
    public String format(String name) { return name.trim().toLowerCase(); }
}
// → 그냥 정적 메서드 하나로 충분하다

// 적절한 OCP: 실제로 확장이 필요한 지점
interface DiscountPolicy { double apply(double price); } // 정책이 자주 추가됨
```
