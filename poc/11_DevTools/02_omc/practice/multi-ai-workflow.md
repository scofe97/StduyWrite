# Claude + Codex 교차 검증 워크플로우

## 개요

Claude(주 실행자)가 코드를 작성하고, Codex(보조 검증자)가 독립적으로 교차 검증하는 Dual-Agent 워크플로우.
"자기 코드의 맹점은 외부 AI가 발견한다"는 원칙에 기반한다.

## 세션 관리

### 시작
```bash
multi-ai
```

### 레이아웃
```
┌──────────────────┬─────────────────┐
│     Claude       │      Codex      │
│   (주 실행자)     │   (교차 검증)    │
│     60%          │      40%        │
└──────────────────┴─────────────────┘
```

### 패인 전환
| 키 | 동작 |
|----|------|
| `Ctrl+B → ←` | Claude 패인으로 이동 |
| `Ctrl+B → →` | Codex 패인으로 이동 |
| `Ctrl+B → z` | 현재 패인 전체화면 토글 |
| `Ctrl+B → q` | 패인 번호 표시 |

### 세션 종료/재연결
```bash
# 세션 분리 (백그라운드 유지)
Ctrl+B → d

# 재연결
multi-ai   # 기존 세션 있으면 자동 attach

# 완전 종료
tmux kill-session -t multi-ai
```

## 교차 검증 프롬프트 템플릿

### 1. 코드 리뷰
```
다음 코드를 리뷰해줘. 버그, 엣지 케이스, 성능 문제를 찾아줘.
특히 원저자가 놓쳤을 수 있는 맹점에 집중해줘.

[코드 붙여넣기]

컨텍스트: [이 코드의 목적과 사용 환경 간략 설명]
```

### 2. 아키텍처 리뷰
```
다음 설계/아키텍처를 독립적으로 평가해줘.
- 확장성 문제는 없는지
- 더 단순한 대안이 있는지
- 숨겨진 커플링이나 의존성 문제는 없는지

[설계 설명 또는 코드 붙여넣기]
```

### 3. 보안 검증
```
다음 코드의 보안 취약점을 검사해줘.
OWASP Top 10 기준으로 점검하고, 구체적인 공격 시나리오와 수정안을 제시해줘.

[코드 붙여넣기]

기술 스택: [사용 기술 명시]
```

### 4. 문서 검증
```
다음 기술 문서의 정확성을 검증해줘.
- 기술적으로 틀린 설명이 있는지
- 빠진 중요한 내용이 있는지
- 오해를 유발할 수 있는 표현이 있는지

[문서 붙여넣기]
```

## 역할 분담 가이드

### Codex 투입 시점

| 상황 | Codex 투입 | 이유 |
|------|-----------|------|
| 보안 민감 코드 (인증, 결제) | O (필수) | 보안 맹점은 치명적 |
| 아키텍처 결정 | O (권장) | 대안 관점 확보 |
| 프로덕션 배포 전 | O (권장) | 최종 안전망 |
| 복잡한 알고리즘 | O (권장) | 엣지 케이스 발견 |
| 단순 CRUD | X | 오버헤드 > 이익 |
| 설정 파일 변경 | X | 단순 작업 |
| 문서 오타 수정 | X | 단순 작업 |

### 작업 흐름

```
1. Claude에서 코드/설계 작성
2. 결과물을 Codex 패인에 복사
3. 교차 검증 프롬프트로 리뷰 요청
4. Codex 피드백 확인
5. Claude에서 피드백 반영 여부 판단 및 수정
```

---

## 팁

- Codex에게는 **컨텍스트를 충분히** 제공해야 유의미한 피드백을 받을 수 있다
- 두 AI의 의견이 충돌하면 **근거(이유)를 비교**하여 판단한다
- Codex 리뷰는 **참고 의견**이지 절대적 기준이 아니다 — 최종 판단은 사람이 한다
