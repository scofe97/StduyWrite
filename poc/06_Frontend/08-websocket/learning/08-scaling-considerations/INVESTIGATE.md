# 08. 스케일링 고려사항 - 조사 (INVESTIGATE)

## 학습 목표
WebSocket 기반 시스템의 스케일링 과제를 이해하고, 대규모 환경에서의 아키텍처 설계 및 한계를 면접에서 설명할 수 있다.

---

## Q1. WebSocket 스케일링의 도전

### 생각해볼 질문
- **HTTP API 서버와 WebSocket 서버의 스케일링 방식은 어떻게 다른가요?**
- **Stateful 연결이 로드밸런싱에 미치는 영향은?**
- **서버 A에 연결된 사용자에게 서버 B의 이벤트를 어떻게 전달하나요?**

### 힌트
- HTTP: 어느 서버로 가도 무관 (Stateless)
- WebSocket: 특정 서버에 연결 유지 (Stateful)
- 서버 간 메시지 브로드캐스트 필요

---

## Q2. 로드밸런서 설정

### 생각해볼 질문
- **WebSocket 연결에 적합한 로드밸런싱 전략은?**
- **L4 vs L7 로드밸런서의 차이는?**
- **Sticky Session이 필요한 이유는?**

### 힌트
- L4: TCP 레벨 (IP, 포트)
- L7: HTTP 레벨 (헤더, 쿠키, URL)
- WebSocket 핸드셰이크는 HTTP로 시작

---

## Q3. 메시지 브로드캐스트

### 생각해볼 질문
- **채팅방에 100만 명이 접속해 있고, 서버가 10대라면?**
- **한 서버에서 보낸 메시지를 다른 서버의 사용자에게 어떻게 전달하나요?**
- **Redis Pub/Sub는 이 문제를 어떻게 해결하나요?**

### 힌트
- 서버 간 통신 레이어 필요
- Redis, Kafka, RabbitMQ
- Pub/Sub 패턴

---

## Q4. 재접속 폭풍 대응

### 생각해볼 질문
- **네트워크 장애로 10만 연결이 끊기면 무슨 일이 일어나나요?**
- **Socket.IO의 폴백 전략은 어떻게 동작하나요?**
- **클라이언트 측에서 할 수 있는 대응은?**

### 힌트
- 지수 백오프 (Exponential Backoff)
- Jitter (랜덤 지연)
- 연결 풀 제한

---

## Q5. 언제 WebSocket을 선택하지 말아야 하는가

### 생각해볼 질문
- **단방향 상태 확인만 필요할 때 WebSocket이 과도한 이유는?**
- **50만+ 동시접속 환경에서 폴링이 더 나을 수 있는 이유는?**
- **기업 네트워크 환경에서 WebSocket의 문제는?**

### 힌트
- 연결 유지 비용 vs 요청 비용
- Stateful vs Stateless
- 방화벽/프록시 차단
