# 08. 대규모 트래픽 아키텍처 - 조사 (INVESTIGATE)

## 학습 목표
대규모 트래픽 환경(50만+ 동시접속)에서의 실시간 통신 기술 선택 기준을 이해하고, "연결 비용 vs 요청 비용" 트레이드오프를 면접에서 설명할 수 있다.

---

## Q1. WebSocket의 숨겨진 비용

### 생각해볼 질문
- **50만 명이 동시 접속하면 서버에 무슨 일이 일어날까요?**
- **"연결을 유지한다"는 것은 어떤 리소스를 소비하나요?**
- **아무 데이터도 안 보내도 비용이 발생할까요?**

### 힌트
- 소켓 객체는 메모리에 상주
- 로드밸런서의 Max Connection 설정
- 커널의 파일 디스크립터 한계

---

## Q2. 재접속 폭풍 (Reconnection Storm)

### 생각해볼 질문
- **네트워크가 1초 동안 깜빡이면 무슨 일이 일어날까요?**
- **50만 명이 동시에 재접속을 시도하면?**
- **이것이 왜 DDoS 공격과 비슷한가요?**

### 힌트
- WebSocket 연결 끊김 감지
- 자동 재연결 로직
- 핸드셰이크 비용

---

## Q3. 폴링의 재발견

### 생각해볼 질문
- **대규모 환경에서 폴링이 WebSocket보다 유리할 수 있는 이유는?**
- **"요청 한 건의 무게"는 무엇을 의미하나요?**
- **대기열 조회 API vs 상품 검색 API의 요청 무게 차이는?**

### 힌트
- Stateless vs Stateful
- Redis 단순 조회 vs RDB 복잡 쿼리
- 스케일 아웃의 용이성

---

## Q4. 어댑티브 폴링 (Adaptive Polling)

### 생각해볼 질문
- **모든 사용자가 같은 주기로 폴링하면 무슨 문제가 있나요?**
- **서버가 폴링 주기를 제어하면 어떤 이점이 있나요?**
- **Jitter란 무엇이고 왜 필요한가요?**

### 힌트
- TTL (Time-To-Live) 응답
- 대기 순번에 따른 동적 조절
- 요청 분산

---

## Q5. SRT/KTX 대기열 시스템 분석

### 생각해볼 질문
- **SRT 예매 대기열은 왜 WebSocket이 아닌 폴링을 사용할까요?**
- **JSONP 방식을 사용하는 이유는?**
- **Sticky Session이 왜 필요한가요?**

### 힌트
- 네퍼널(Netfunnel) 상용 솔루션
- 크로스 도메인 문제
- 로드밸런싱과 상태 관리

---

## 핵심 공식

```
연결 유지 비용  vs  자주 묻는 비용
    (Stateful)         (Stateless)
```
