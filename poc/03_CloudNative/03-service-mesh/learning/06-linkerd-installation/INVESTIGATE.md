# Ch06. Linkerd 설치와 메시 구성 — Deep Dive Questions

## Q1. 프로덕션 환경에서 신뢰 앵커(trust anchor)를 교체할 때 어떤 중단 없이 수행할 수 있는가?

### 왜 중요한가?

신뢰 앵커는 Linkerd 메시 전체의 신뢰 체계 최상위에 있는 루트 인증서다. 모든 사이드카 프록시가 이 신뢰 앵커를 기준으로 상대방의 인증서를 검증한다. 신뢰 앵커가 만료되거나 교체 필요성이 생기면, 잘못된 절차로 인해 메시 전체의 mTLS 통신이 중단될 수 있다. 이 작업은 Linkerd 운영에서 가장 위험도가 높은 절차 중 하나다.

### 분석

신뢰 앵커 교체의 핵심 원리는 "새 신뢰 앵커를 기존 앵커와 함께 번들(bundle)로 배포하는 중간 단계"다. 모든 프록시가 새 신뢰 앵커를 신뢰하도록 업데이트된 이후에야 기존 앵커를 제거할 수 있다. 이 과정을 "trust anchor rotation"이라고 하며, 최소 두 단계가 필요하다.

1단계: 새 신뢰 앵커 생성 후 기존 앵커와 결합한 번들을 Linkerd에 적용한다. `linkerd upgrade --identity-trust-anchors-file=bundle.pem` 명령으로 컨트롤 플레인을 업데이트하면, 모든 새로 주입되는 프록시는 두 신뢰 앵커를 모두 신뢰한다. 기존 프록시는 이미 실행 중이므로 롤링 재시작이 필요하다.

2단계: 모든 프록시가 새 번들로 업데이트됐음을 확인한 후, 기존 신뢰 앵커만 제거하고 새 신뢰 앵커만 남긴다. 이 시점에서 발급자 인증서(issuer certificate)도 새 신뢰 앵커로 서명된 것으로 교체해야 한다.

자동화 측면에서 cert-manager와 trust-manager 조합이 가장 견고한 접근이다. cert-manager는 발급자 인증서 자동 갱신을 담당하고, trust-manager는 신뢰 앵커 번들을 모든 네임스페이스에 전파한다. 수동 교체는 단계 사이의 타이밍을 잘못 맞추면 인증서 불일치로 mTLS 핸드셰이크가 실패하므로, 자동화가 사실상 필수다.

주의할 점은 프록시 롤링 재시작 속도다. 수백 개의 Deployment가 있는 대규모 클러스터에서 모든 Pod를 재시작하려면 수십 분이 걸릴 수 있다. 이 기간 동안 기존 신뢰 앵커와 새 신뢰 앵커가 혼재하므로, 번들 상태를 유지해야 한다.

### 실무 적용

신뢰 앵커 유효 기간을 10년으로 충분히 길게 설정하고, 발급자 인증서는 1년 주기로 자동 갱신하는 패턴이 운영 부담을 줄이는 현실적 방법이다. 신뢰 앵커 만료 날짜를 팀 캘린더와 알림 시스템에 등록해두고, 만료 90일 전부터 교체 프로세스를 시작하는 것이 안전하다. `linkerd check` 명령의 `--proxy` 플래그로 모든 프록시의 인증서 상태를 한 번에 확인할 수 있다.

## Q2. GitOps 환경에서 Helm과 CLI 기반 설치의 차이는 단순한 도구 선택 이상의 의미를 가지는가?

### 왜 중요한가?

Linkerd 설치 방법으로 CLI(`linkerd install`)와 Helm 차트 두 가지가 있다. 표면적으로는 동일한 결과물을 생성하지만, GitOps 워크플로우(ArgoCD, Flux)와 통합하거나 멀티 환경(dev/staging/prod) 배포를 관리할 때 두 접근법의 차이는 운영 방식 전체에 영향을 미친다. 도구 선택이 아닌 철학의 차이를 이해하는 것이 중요하다.

### 분석

CLI 기반 설치(`linkerd install | kubectl apply -f -`)의 문제는 생성되는 매니페스트가 명령 실행 시점의 상태 스냅샷이라는 점이다. 이 스냅샷을 Git에 커밋해 GitOps로 관리할 수 있지만, 다음 버전 업그레이드 시 새 매니페스트를 생성해 diff를 관리해야 한다. 이 과정에서 설정 드리프트(config drift)가 발생하기 쉽고, 환경별 차이를 관리하기 어렵다.

Helm 차트는 values.yaml 파일로 환경별 설정을 선언적으로 관리한다. Linkerd의 공식 Helm 차트는 `linkerd-crds`, `linkerd-control-plane`, 선택적으로 `linkerd-viz` 세 차트로 구성된다. CRD를 별도 차트로 분리한 것은 Helm의 CRD 업그레이드 제한(Helm은 기존 CRD를 업데이트하지 않음) 문제를 우회하기 위한 설계다.

ArgoCD와의 통합에서 주의해야 할 점이 있다. Linkerd의 proxy-injector는 MutatingWebhookConfiguration을 사용해 Pod 생성 시 사이드카를 주입한다. ArgoCD가 직접 관리하는 네임스페이스에 주입이 활성화되면, ArgoCD 자체의 Pod에도 Linkerd 사이드카가 주입될 수 있다. 이 경우 ArgoCD와 Linkerd 컨트롤 플레인 사이의 부트스트랩 순서 문제가 생긴다. 일반적으로 `linkerd.io/inject: disabled` 어노테이션을 ArgoCD 네임스페이스에 적용해 이 문제를 회피한다.

또한 Helm을 통한 업그레이드 시 롤링 업데이트와 컨트롤 플레인 컴포넌트 간 버전 호환성을 주의해야 한다. Linkerd는 컨트롤 플레인과 데이터 플레인(프록시) 사이에 n-1 버전 호환성을 보장하지만, 컨트롤 플레인 컴포넌트들이 업그레이드 중 혼재하는 상태를 최소화하는 것이 좋다.

### 실무 적용

GitOps 환경에서는 Helm + values.yaml 파일의 환경별 오버레이(kustomize 또는 Helm의 `-f` 플래그)가 관리 복잡도를 줄이는 효과적인 패턴이다. 프로덕션 values.yaml은 `ha.enabled: true`, 충분한 리소스 요청/제한, 신뢰 앵커 설정을 포함하고, 개발 환경은 최소 설정으로 유지한다. Linkerd 버전 업그레이드는 별도 PR로 관리해 리뷰 프로세스를 거치게 하면 실수를 줄일 수 있다.

## Q3. 사이드카 주입 실패를 디버깅할 때 체계적으로 접근하는 방법은 무엇인가?

### 왜 중요한가?

Linkerd 사이드카 주입은 Kubernetes MutatingWebhookConfiguration을 통해 Pod 생성 시점에 자동으로 이루어진다. 이 메커니즘이 실패하면 사이드카가 없는 Pod가 배포되고, mTLS와 관측성이 조용히 비활성화된다. 더 나쁜 경우는 주입 자체가 오류를 일으켜 Pod 생성이 차단되는 상황이다. 두 시나리오 모두 명확한 오류 메시지 없이 발생하는 경우가 있어 원인을 찾기 어렵다.

### 분석

주입 실패의 원인은 크게 세 범주로 나뉜다. 첫째는 웹훅 연결 문제다. proxy-injector 웹훅 서버에 API 서버가 연결할 수 없으면 주입이 실패한다. API 서버는 클러스터 내부 서비스 IP로 웹훅 서버에 접근하므로, 네트워크 정책이 이 트래픽을 차단하고 있는지 확인해야 한다. 특히 API 서버가 웹훅 서버에 접근할 수 있는 CIDR을 네트워크 정책의 `from` 절에 명시해야 한다.

둘째는 인증서 검증 문제다. 웹훅 설정의 `caBundle` 필드에 등록된 CA 인증서와 proxy-injector 서버의 TLS 인증서가 일치해야 한다. cert-manager를 사용하지 않고 수동으로 인증서를 관리한다면, 인증서 갱신 후 웹훅 설정의 `caBundle`을 업데이트하지 않아 불일치가 생길 수 있다.

셋째는 주입 조건 불일치다. 주입은 네임스페이스 또는 Pod 어노테이션(`linkerd.io/inject: enabled`)이 설정된 경우에만 발생한다. 네임스페이스에 어노테이션이 있어도 Pod에 `linkerd.io/inject: disabled`가 있으면 주입이 건너뛰어진다. 반대로 주입이 기대되는 Pod에 어노테이션이 없는 경우도 흔한 실수다.

진단 순서는 다음과 같다. `kubectl get events -n <namespace>`로 Pod 생성 이벤트 확인 → `kubectl describe pod <pod>` 에서 Annotations 섹션 확인 (linkerd 어노테이션이 있는지) → `kubectl logs -n linkerd -l linkerd.io/control-plane-component=proxy-injector`로 웹훅 서버 로그 확인 → `linkerd check`로 전체 시스템 상태 점검.

### 실무 적용

주입 실패를 조기에 감지하려면 배포 파이프라인에 `linkerd check --proxy -n <namespace>` 단계를 추가하는 것이 효과적이다. 이 명령은 지정 네임스페이스의 모든 Pod가 올바르게 주입됐는지 확인한다. 또한 새 네임스페이스 생성 시 자동으로 주입 어노테이션을 붙이는 admission webhook 또는 OPA/Kyverno 정책을 통해 누락을 방지할 수 있다. 주입이 필요 없는 네임스페이스(kube-system, cert-manager 등)는 명시적으로 `linkerd.io/inject: disabled`를 설정해 의도를 명확히 한다.

## Q4. 무중단 업그레이드(zero-downtime upgrade)를 위한 Linkerd 버전 관리 전략은 무엇인가?

### 왜 중요한가?

서비스 메시는 모든 서비스 간 트래픽의 경로에 있으므로, 메시 자체를 업그레이드하는 작업은 일반 애플리케이션 업그레이드보다 훨씬 신중해야 한다. Linkerd는 컨트롤 플레인과 데이터 플레인(사이드카 프록시)이 분리되어 있어, 두 컴포넌트의 버전 호환성과 업그레이드 순서가 중요하다. 잘못된 순서로 업그레이드하면 프록시가 컨트롤 플레인과 통신하지 못하는 상황이 발생한다.

### 분석

Linkerd의 버전 호환성 정책은 컨트롤 플레인과 데이터 플레인 사이에 n-1 버전 호환성을 보장한다. 즉, 컨트롤 플레인이 2.15라면 데이터 플레인은 2.14 또는 2.15 프록시와 함께 동작한다. 이 정책 덕분에 "컨트롤 플레인을 먼저 업그레이드, 데이터 플레인을 이후 롤링 업그레이드"하는 안전한 순서가 가능하다.

업그레이드 절차를 단계별로 보면, 먼저 `linkerd upgrade` 명령 또는 Helm upgrade로 컨트롤 플레인을 새 버전으로 업데이트한다. 이 시점에서 기존 프록시(구 버전)는 여전히 이전 컨트롤 플레인과 동일하게 동작한다. 다음으로 `linkerd check`를 실행해 컨트롤 플레인이 정상적으로 동작하는지 확인한다. 이후 각 네임스페이스의 Deployment를 롤링 재시작해 새 버전의 사이드카가 주입된 Pod를 배포한다. `kubectl rollout restart deployment -n <namespace>` 명령이 이 작업을 수행한다.

롤링 재시작의 속도 조절도 고려해야 한다. 수십 개의 Deployment를 동시에 재시작하면 클러스터에 순간적인 부하가 발생할 수 있다. Kubernetes의 기본 롤링 업데이트 전략(`maxSurge`, `maxUnavailable`)을 활용해 속도를 제어하거나, 네임스페이스 단위로 순차적으로 재시작하는 것이 안전하다.

엣지 케이스로 StatefulSet과 DaemonSet의 업그레이드가 있다. StatefulSet은 롤링 재시작이 일반 Deployment보다 느리고, Pod가 하나씩 순서대로 재시작된다. DaemonSet은 각 노드의 Pod가 순서대로 재시작되므로, 노드 수에 비례해 시간이 걸린다.

### 실무 적용

업그레이드를 스테이징 환경에서 먼저 검증한 후 프로덕션에 적용하는 것이 기본 원칙이다. 스테이징에서 업그레이드 후 `linkerd check --proxy`와 핵심 서비스의 헬스 체크를 자동화 테스트로 실행해 회귀를 감지한다. 프로덕션 업그레이드는 트래픽이 낮은 시간대에 시작하고, 각 단계 사이에 충분한 관측 시간을 두는 것이 좋다. 특히 인증서 갱신 주기와 업그레이드 시점이 겹치지 않도록 일정을 조율한다.

## Q5. cert-manager를 Linkerd의 인증서 관리에 통합할 때 어떤 의존성 순서 문제가 발생하는가?

### 왜 중요한가?

cert-manager는 Kubernetes에서 인증서 라이프사이클을 자동화하는 사실상의 표준 도구다. Linkerd의 발급자 인증서(issuer certificate)와 신뢰 앵커를 cert-manager로 관리하면 수동 갱신의 번거로움이 사라지지만, 설치 순서와 부트스트랩 의존성을 잘못 설정하면 순환 의존성 문제가 생긴다. 이 문제를 정확히 이해하지 못하면 클러스터 초기 설치나 재설치 시 예상치 못한 장애를 만나게 된다.

### 분석

핵심 부트스트랩 순서는 다음과 같아야 한다. cert-manager가 먼저 실행되어야 Linkerd가 사용하는 Certificate 리소스를 처리할 수 있다. Linkerd 컨트롤 플레인은 Certificate 리소스가 실제 인증서 시크릿으로 변환된 이후에 시작돼야 한다. 순환 의존성은 Linkerd의 사이드카가 cert-manager Pod에 주입되는 경우에 발생한다. Linkerd 컨트롤 플레인이 준비되지 않은 상태에서 cert-manager Pod가 주입 시도를 하면, cert-manager도 Linkerd도 제대로 시작하지 못하는 교착 상태가 된다.

해결책은 두 가지다. 첫째, cert-manager 네임스페이스에 `linkerd.io/inject: disabled` 어노테이션을 적용해 cert-manager Pod에 Linkerd 사이드카가 주입되지 않도록 한다. 이렇게 하면 cert-manager가 Linkerd와 독립적으로 동작할 수 있다. 둘째, Linkerd와 cert-manager의 설치 순서를 명시적으로 보장한다. ArgoCD를 사용한다면 App-of-Apps 패턴에서 `syncWave` 어노테이션으로 순서를 제어한다.

cert-manager와 trust-manager의 조합도 중요하다. cert-manager가 인증서를 발급하면 그 결과는 특정 네임스페이스의 Secret에 저장된다. trust-manager는 이 인증서를 `Bundle` 리소스를 통해 여러 네임스페이스에 ConfigMap으로 전파한다. Linkerd는 이 ConfigMap을 신뢰 앵커 소스로 사용할 수 있어, 신뢰 앵커 변경이 자동으로 전체 클러스터에 반영된다.

인증서 갱신 타이밍도 고려해야 한다. cert-manager는 인증서 유효 기간의 2/3 시점에 갱신을 시도한다. 갱신된 인증서는 새 Secret으로 저장되고, Linkerd identity 서버는 이를 감지해 새 발급자 인증서로 전환한다. 이 전환 중에 기존 발급자 인증서로 서명된 클라이언트 인증서와 새 발급자로 서명된 인증서가 공존하는 짧은 시간이 있으며, Linkerd는 이 상황을 번들 신뢰 앵커를 통해 처리한다.

### 실무 적용

cert-manager 통합의 검증은 `cmctl status certificate -n linkerd linkerd-identity-issuer` 명령으로 인증서 상태를 확인하고, `linkerd check` 로 전체 통합 상태를 점검하는 두 단계로 한다. 인증서 갱신이 정상적으로 이루어지는지 확인하려면 cert-manager의 `certificate_expiry_seconds` 메트릭을 Prometheus에서 모니터링하는 것이 좋다. 갱신 실패 시 알림을 설정해 수동 개입이 필요한 시점을 조기에 감지해야 한다.

## Q6. 네임스페이스 단위 주입과 Pod 단위 주입 중 어떤 전략이 운영 단순성과 보안 사이에서 더 나은 균형을 제공하는가?

### 왜 중요한가?

Linkerd 사이드카 주입은 네임스페이스 어노테이션(`linkerd.io/inject: enabled` on namespace)으로 해당 네임스페이스의 모든 Pod에 일괄 적용하거나, Pod 또는 Deployment 어노테이션으로 개별 적용할 수 있다. 이 결정은 단순한 설정 선택이 아니라, 조직의 보안 정책과 운영 거버넌스 방식을 반영한다.

### 분석

네임스페이스 단위 주입은 "기본 활성화(opt-out)" 모델이다. 네임스페이스에 어노테이션을 붙이면 별도 설정 없이 새로 배포되는 모든 Pod에 사이드카가 주입된다. 이 접근법의 장점은 개발자가 명시적으로 주입을 요청하지 않아도 메시의 혜택을 자동으로 누린다는 것이다. 운영팀이 인프라 레이어에서 정책을 일관되게 적용할 수 있다.

Pod 단위 주입은 "기본 비활성화(opt-in)" 모델이다. 각 Deployment가 명시적으로 주입을 요청해야 한다. 이 접근법은 메시에 참여하는 워크로드를 세밀하게 제어할 수 있고, 레거시 서비스나 특수 목적 서비스를 메시에서 제외하는 것이 자연스럽다. 그러나 개발자가 어노테이션을 추가하지 않으면 주입이 누락되는 실수가 발생한다.

실제 운영에서는 두 방식을 혼합하는 패턴이 일반적이다. 프로덕션 워크로드를 담는 네임스페이스는 네임스페이스 단위 활성화, 시스템 네임스페이스(kube-system, cert-manager 등)는 비활성화, 특정 Pod만 제외할 경우 Pod 단위 비활성화 어노테이션을 사용한다.

보안 관점에서 네임스페이스 단위 활성화가 더 안전한 기본값이다. 개별 서비스가 메시를 "잊고" 배포하는 실수를 방지하고, 메시의 mTLS 커버리지를 감사(audit)할 때 네임스페이스 목록만 확인하면 된다. 반면 모든 네임스페이스에 주입이 활성화되어 있다면 의도하지 않은 서비스가 포함될 위험이 있으므로, 주입 제외 목록을 명시적으로 관리해야 한다.

### 실무 적용

권장 전략은 "프로덕션 네임스페이스는 네임스페이스 단위 활성화, 인프라 네임스페이스는 명시적 비활성화"다. 새 네임스페이스 생성 시 자동으로 주입 어노테이션을 붙이는 Kyverno 또는 OPA 정책을 적용하면 일관성을 유지할 수 있다. `linkerd check --proxy --namespace <ns>` 명령을 CI/CD 파이프라인에 포함시켜 배포 후 주입 상태를 자동 검증하는 것이 운영 안전망이 된다.

## Q7. Linkerd 설치 후 헬스 체크가 통과했지만 실제 트래픽에서 문제가 발생하는 상황은 어떻게 진단하는가?

### 왜 중요한가?

`linkerd check` 명령이 모든 항목에서 통과를 보고하더라도, 실제 워크로드 트래픽에서 문제가 발생하는 경우가 있다. 컨트롤 플레인 자체는 정상이지만 특정 서비스 간 통신에서 오류가 나타나는 시나리오는 설치 완료와 운영 정상 사이의 간극을 보여준다. 이 진단 역량은 Linkerd를 신뢰할 수 있는 인프라로 운영하기 위한 필수 조건이다.

### 분석

컨트롤 플레인 체크와 데이터 플레인 실제 동작 사이의 간극은 주로 프로토콜 감지 실패, 네트워크 정책 충돌, 그리고 opaque port 설정 누락에서 발생한다.

프로토콜 감지 실패는 특히 MySQL, PostgreSQL, Redis 같은 비HTTP 프로토콜에서 나타난다. 프록시가 프로토콜을 감지하기 위해 첫 바이트를 기다리는 동안 타임아웃이 발생해 연결이 느려지거나 끊긴다. `linkerd viz tap` 명령으로 실시간 트래픽을 관찰하면 어떤 프로토콜이 어떻게 처리되는지 확인할 수 있다.

네트워크 정책과의 충돌은 Linkerd 프록시가 인터셉트하는 포트(4143, 4191)를 네트워크 정책이 차단할 때 발생한다. Linkerd 사이드카의 init 컨테이너가 iptables 규칙을 설정해 트래픽을 리다이렉트하므로, 기존 네트워크 정책과의 상호작용을 명확히 이해해야 한다. `kubectl exec -n <ns> <pod> -c linkerd-proxy -- curl localhost:4191/metrics`로 프록시 메트릭을 직접 확인하는 것이 첫 번째 진단 단계다.

인증서 관련 문제는 `linkerd viz tap -n <ns> deploy/<name>` 출력에서 TLS 오류로 나타난다. 특히 레거시 서비스가 mTLS를 기대하지 않는 방식으로 연결을 처리할 때(예: TLS를 자체 처리하는 서비스) Linkerd의 mTLS와 충돌이 생길 수 있다.

### 실무 적용

설치 후 검증 체크리스트를 별도로 관리하는 것이 좋다. `linkerd check` 외에 실제 서비스 간 요청을 보내는 연기 테스트(smoke test)를 포함시키고, Linkerd 대시보드(`linkerd viz dashboard`)에서 골든 시그널(레이턴시, 오류율, 처리량)이 정상 범위인지 확인한다. 새 서비스를 메시에 추가할 때는 해당 서비스의 통신 패턴(프로토콜, 포트, 연결 수명)을 사전에 문서화하고, opaque ports 설정이 필요한지 검토하는 체크리스트를 팀 내에서 공유한다.
