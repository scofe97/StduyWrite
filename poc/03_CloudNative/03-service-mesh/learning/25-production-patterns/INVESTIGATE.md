# Ch25. 프로덕션 패턴 — Deep Dive Questions

## Q1. 카나리 분석에서 어떤 메트릭을 선택하면 실제로 의미 있는 배포 안전성을 확보할 수 있는가?

### 왜 중요한가?

Flagger나 Argo Rollouts로 카나리 배포를 자동화할 때 핵심 결정은 "어떤 메트릭이 나빠지면 자동 롤백할 것인가"다. 잘못된 메트릭을 선택하면 두 가지 문제가 생긴다. 문제있는 버전이 프로덕션에 배포되거나(False Negative), 멀쩡한 버전이 반복적으로 롤백되어 배포가 불가능해진다(False Positive). 메트릭 선택은 기술적 판단이면서 동시에 서비스 특성에 대한 깊은 이해를 요구하는 도메인 판단이기도 하다.

### 분析

카나리 분석에 흔히 사용하는 메트릭은 세 종류다. 에러율(5xx/요청 수), 레이턴시(p99), 처리량(RPS)이다. 이 세 가지를 단순 조합하면 기본적인 안전망이 된다. 하지만 이것만으로는 중요한 문제를 놓친다.

에러율 임계치 설정이 까다롭다. 새벽 시간대는 트래픽이 적어 에러 샘플이 부족하다. 요청 10개 중 1개가 실패하면 에러율 10%지만, 통계적으로 의미 있는 수치가 아닐 수 있다. Flagger는 `threshold` 외에 `interval`과 `iterations` 설정으로 통계적 안정성을 높이지만, 적은 트래픽 환경에서는 여전히 불안정하다. 이런 경우 절대 에러 수 임계치(`errorCount > 5`)와 에러율을 AND 조건으로 사용하는 것이 낫다.

비즈니스 메트릭을 카나리 분석에 포함하는 것이 이상적이다. 전자상거래에서 "체크아웃 완료율"이 기술 메트릭보다 실제 영향을 더 직접적으로 반영한다. 결제 처리 시간이 늘어도 HTTP 200을 반환하면 기술 메트릭은 정상이지만 비즈니스 영향은 크다. Flagger의 `custom metric` 또는 Webhook을 활용해 Prometheus에서 비즈니스 메트릭을 쿼리하는 방법이 있다.

A/B 테스트와 카나리의 혼동도 주의해야 한다. 카나리는 "새 버전이 기존 버전보다 나쁘지 않은가"를 검증한다. A/B 테스트는 "버전 A와 B 중 어느 것이 비즈니스 지표를 더 높이는가"를 검증한다. 같은 인프라를 쓰지만 목적과 판단 기준이 다르다. 카나리 분석에 A/B 테스트 지표를 넣으면 판단이 흐려진다.

### 실무 적용

서비스별 카나리 메트릭 정의서를 작성한다. 각 서비스에 대해 (1) 절대 허용 불가 조건(에러율 임계치), (2) 성능 회귀 조건(레이턴시 p99 > 기준값 * 1.2), (3) 비즈니스 핵심 지표(서비스별 정의)를 문서화한다. 이 정의서는 Flagger의 `MetricTemplate`과 `Canary` 리소스에 코드화되고, PR 리뷰 대상이 된다. 신규 서비스 배포 전 메트릭 정의서 작성이 완료 기준에 포함되어야 한다.

## Q2. 재시도 폭풍(Retry Storm)은 왜 발생하며, 서비스 메시 수준에서 이를 예방하는 근본적 접근은 무엇인가?

### 왜 중요한가?

서비스 메시의 재시도 정책은 일시적 장애를 투명하게 복구해주는 편리한 기능이다. 하지만 잘못 구성된 재시도 정책은 부분적 장애를 전체 시스템 장애로 증폭시키는 재시도 폭풍을 유발한다. 한 서비스가 느려지면 업스트림이 재시도하고, 재시도된 요청이 이미 과부하된 서비스를 더 과부하시키는 양성 피드백 루프가 형성된다. 이 현상은 2012년 Amazon과 2015년 Netflix 사고에서 실제로 발생했다.

### 분析

재시도 폭풍의 메커니즘을 수식으로 보면 이해가 빠르다. 서비스 B가 초당 100 RPS를 처리 가능하고, 서비스 A가 100 RPS로 호출한다고 가정하자. B가 느려져 에러율 30%가 발생하면, A의 재시도 정책(최대 3회)이 발동한다. A는 실패한 30 RPS에 대해 재시도하므로 실제 B로 향하는 요청은 130 RPS가 된다. 130 RPS가 들어오면 B는 더 느려지고 에러율이 높아지며, 재시도는 더 늘어난다. 이 루프가 수십 초 안에 B를 완전히 죽인다.

Istio의 `retryOn` 설정에서 어떤 에러 코드에 재시도할지가 중요하다. `5xx` 전체가 아니라 `gateway-error,reset,connect-failure`처럼 네트워크 수준 에러에만 재시도하고, `retriable-status-codes: 503`처럼 특정 상태 코드만 지정하면 의도치 않은 재시도를 줄인다. 특히 `5xx` 전체 재시도는 비멱등 요청에 위험하다. POST 요청이 서버에서 처리된 후 응답 반환 중에 연결이 끊기면, 클라이언트는 실패로 판단하고 재시도하지만 실제로 요청은 두 번 처리된다.

재시도 폭풍 예방의 근본 접근은 세 가지다. 첫째, 지수 백오프(Exponential Backoff)와 지터(Jitter)를 사용한다. Envoy의 재시도 간격이 고정이면 동기화된 재시도 버스트가 발생한다. 지터를 추가해 재시도 시점을 분산시켜야 한다. 둘째, 재시도 예산(Retry Budget)을 설정한다. Linkerd는 기본적으로 재시도 예산 개념을 사용한다. 전체 요청의 최대 20%(기본값)만 재시도하도록 제한해 재시도 폭풍을 구조적으로 차단한다. 셋째, 서킷 브레이커와 재시도를 조합한다. 서킷이 Open 상태에서는 재시도하지 않는다.

### 실무 적용

재시도 정책 리뷰 체크리스트를 만든다. (1) 비멱등 요청(POST, PATCH)에는 재시도를 비활성화하거나 명시적으로 활성화한 이유를 기록, (2) 재시도 횟수는 최대 3회, (3) 백오프는 지수 + 지터 사용, (4) 재시도 예산은 20% 이하로 설정. Istio VirtualService의 `retries` 설정은 코드 리뷰 필수 항목으로 지정하고, 신규 서비스 추가 시 이 정책이 명시적으로 정의되어 있는지 자동으로 검사하는 린팅 규칙을 추가한다.

## Q3. Rate Limiting의 세분화 수준(글로벌 vs 서비스별 vs 사용자별)을 어떻게 결정하는가?

### 왜 중요한가?

서비스 메시에서 Rate Limiting은 과부하 방어의 핵심이다. 하지만 Rate Limiting을 어느 수준에서 적용하느냐에 따라 보호 효과와 운영 복잡성이 달라진다. 너무 넓게 설정하면(글로벌 RPS만 제한) 특정 사용자의 폭탄 요청이 전체 사용자에게 영향을 준다. 너무 세밀하게 설정하면(사용자별, 엔드포인트별) 관리 포인트가 폭발적으로 증가한다. 이 균형점을 찾는 기준이 필요하다.

### 분析

Rate Limiting 세분화는 두 차원에서 생각할 수 있다. "무엇을 기준으로 제한하는가"(IP, 사용자, 테넌트, API 키)와 "어디에 제한을 두는가"(Gateway 레이어, 서비스 레이어, 인스턴스 레이어)다.

Istio의 로컬 Rate Limiting(`EnvoyFilter`로 `local_rate_limit` 필터 설정)은 각 Envoy 인스턴스가 독립적으로 카운터를 관리한다. 구성이 단순하고 외부 의존성이 없지만, 인스턴스가 3개면 실제 글로벌 제한은 설정값의 3배가 된다. 수평 확장에 따라 제한값이 달라지는 문제가 있다.

글로벌 Rate Limiting은 외부 Rate Limit 서비스(Redis 기반)를 사용한다. Envoy의 `rateLimit` 필터가 모든 요청에 대해 외부 서비스에 허용 여부를 묻는다. 정확한 글로벌 제어가 가능하지만, 이 외부 서비스가 병목 또는 단일 실패 지점이 된다. Redis 클러스터의 가용성이 Rate Limiting의 가용성을 결정한다.

세분화 수준 결정의 실용적 기준은 "이 수준의 제한 위반이 얼마나 자주 발생하는가"다. IP 기반 제한 위반이 거의 없다면 사용자별 제한을 추가해도 실질적 보호 효과가 없다. 먼저 모니터링으로 실제 Rate Limiting 트리거 패턴을 파악하고, 문제가 되는 차원(특정 API 엔드포인트, 특정 테넌트)에 집중적으로 세분화하는 것이 효율적이다.

인프라 비용도 고려해야 한다. 사용자별 Rate Limiting은 사용자 수에 비례해 Redis의 키 수와 메모리가 늘어난다. 수백만 사용자 서비스에서 사용자별 슬라이딩 윈도우 Rate Limiting을 구현하면 Redis 클러스터 비용이 상당하다.

### 실무 적용

Rate Limiting 설계는 세 단계로 진행한다. 1단계: Envoy 로컬 Rate Limiting으로 인스턴스별 최대 부하를 설정(서킷 브레이커의 보완). 2단계: Gateway 레이어에서 IP 기반 Rate Limiting으로 DDoS 기초 방어. 3단계: 비즈니스 요구가 명확한 경우에만(멀티테넌트, API 상용화) 테넌트/API 키 기반 글로벌 Rate Limiting 추가. 이 순서로 단계적으로 도입해 운영 복잡성과 보호 효과를 균형 있게 가져간다.

## Q4. 프로덕션에서 카오스 엔지니어링을 실행하는 것은 정당화될 수 있는가, 아니면 스테이징으로 충분한가?

### 왜 중요한가?

카오스 엔지니어링(Chaos Engineering)은 의도적으로 장애를 주입해 시스템의 회복탄력성을 검증하는 접근이다. 서비스 메시가 올바르게 구성되어 있다면 — 재시도, 타임아웃, 서킷 브레이커, 페일오버가 제대로 작동한다면 — 카오스 테스트에서 사용자 영향 없이 시스템이 회복되어야 한다. 하지만 이 검증을 프로덕션에서 해야 하는가, 스테이징으로 충분한가는 조직마다 다르다.

### 분析

스테이징이 프로덕션과 완전히 동일할 수 없다는 것이 카오스 엔지니어링을 프로덕션에서 해야 한다는 주장의 핵심이다. 스테이징의 트래픽 패턴, 데이터 분포, 외부 의존성, 부하 수준이 프로덕션과 다르면 스테이징에서의 카오스 테스트 결과가 프로덕션의 실제 동작을 보장하지 않는다. Netflix가 Chaos Monkey를 프로덕션에 적용한 것은 이 이유에서다.

반대 입장도 있다. 성숙한 스테이징 환경(프로덕션 트래픽 복제, 동일 인프라 스펙)을 운영하면 프로덕션 카오스 테스트 없이도 신뢰성을 확보할 수 있다. 프로덕션 카오스 테스트는 실제 사용자 영향 위험이 있고, 이 위험을 감수할 정도의 비즈니스 가치가 있어야 한다. 금융 서비스나 의료 시스템에서 프로덕션 카오스 테스트는 규제 및 계약 위반이 될 수 있다.

서비스 메시 관점에서 카오스 테스트의 구체적 목표를 정하는 것이 중요하다. "서킷 브레이커가 올바르게 작동하는가"는 Istio의 `VirtualService`에 `fault injection`(`abort`, `delay`)을 적용해 스테이징에서 충분히 검증 가능하다. "프로덕션 트래픽 패턴에서 카오스 주입 시 카나리 분석이 자동으로 롤백을 결정하는가"는 프로덕션에 가까운 환경이 필요하다.

Litmus Chaos, Chaos Mesh 같은 도구는 Kubernetes 네이티브 카오스 실험을 지원한다. Istio와 통합하면 메시 레이어 장애(Envoy 사이드카 종료, xDS 연결 끊김 등)도 주입 가능하다. 이 도구들은 폭발 반경을 제한하는 기능(특정 네임스페이스, 특정 레이블의 Pod만 영향)을 제공해 프로덕션 실행의 위험을 줄인다.

### 실무 적용

카오스 엔지니어링 성숙도 모델을 단계적으로 적용한다. 1단계: 스테이징에서 모든 메시 재시도/타임아웃/서킷 브레이커 설정을 장애 주입으로 검증. 2단계: 프로덕션 트래픽의 1% 미만을 처리하는 비핵심 서비스에서 제한적 카오스 테스트. 3단계: Game Day(정기 카오스 훈련일)를 지정해 팀 전체가 참여하는 계획된 카오스 테스트. 각 단계에서 학습 내용을 문서화하고 메시 설정을 개선한다.

## Q5. Flagger의 Custom Metric Webhook은 어떤 시나리오에서 진가를 발휘하며, 잘못 구현하면 어떤 문제가 생기는가?

### 왜 중요한가?

Flagger는 카나리 분석을 위해 Prometheus 메트릭 외에 외부 Webhook을 호출하는 기능을 제공한다. 이 Webhook으로 Datadog, New Relic, 또는 자체 분석 시스템에서 데이터를 가져와 배포 진행 여부를 결정할 수 있다. 기술 메트릭을 넘어 비즈니스 지표를 카나리 분석에 통합하는 핵심 확장 포인트다. 하지만 Webhook 구현이 잘못되면 배포 파이프라인이 잘못된 이유로 차단되거나, 반대로 문제있는 버전이 통과할 수 있다.

### 분析

Flagger의 Webhook 타입은 여러 가지다. `pre-rollout`, `rollout`, `confirm-rollout`, `confirm-promotion`, `post-rollout`이 있고, 카나리 분석용은 주로 `metric`과 `rollout` Webhook을 사용한다. `metric` Webhook은 Flagger가 주기적으로 호출하고 성공/실패 여부를 분석 결과로 사용한다. HTTP 200 = 분석 통과, 그 외 = 분석 실패로 단순하게 작동한다.

Custom Metric Webhook이 진가를 발휘하는 시나리오는 다음과 같다. 첫째, SLO 기반 카나리 분석이다. Sloth나 Pyrra 같은 SLO 도구에서 현재 에러 예산 소진율을 쿼리하고, 카나리 배포 중 에러 예산이 가속 소진되면 자동 롤백한다. 순수 기술 메트릭이 아닌 SLO 렌즈로 배포 품질을 평가하는 것이다. 둘째, 외부 모니터링 시스템 통합이다. 조직이 Prometheus가 아닌 Datadog을 주 모니터링 도구로 쓴다면, Datadog API로 카나리 메트릭을 가져오는 Webhook이 Prometheus MetricTemplate보다 자연스럽다.

Webhook 구현의 흔한 실수는 응답 지연이다. Flagger의 Webhook 타임아웃 기본값은 1분이다. 외부 분석 시스템 호출이 30초 이상 걸리면 Webhook이 타임아웃으로 실패 처리된다. 분석 실패가 반복되면 Flagger는 카나리를 롤백한다. 실제 메트릭이 양호해도 Webhook 지연 때문에 롤백이 발생하는 False Positive가 된다. Webhook 구현 시 캐싱이나 비동기 처리로 응답 시간을 줄여야 한다.

Webhook의 멱등성도 중요하다. Flagger가 Webhook을 여러 번 호출해도 동일한 결과를 반환해야 한다. 상태를 유지하는 Webhook(예: "이 배포 ID에 대해 몇 번째 호출인가를 카운트")은 재시도 시 잘못된 결과를 줄 수 있다. Webhook은 주어진 시점의 스냅샷 메트릭을 반환하는 Stateless 구현이 안전하다.

### 실무 적용

Custom Metric Webhook 개발 시 응답 시간 SLO를 먼저 정의한다(예: p95 < 5초). 이 SLO를 충족하지 못하는 외부 시스템 호출은 캐시 레이어를 추가한다. Webhook 자체에 대한 모니터링(호출 횟수, 응답 시간, 성공률)도 Prometheus로 노출해 Flagger와 별개로 Webhook의 건강 상태를 확인할 수 있어야 한다. Webhook이 항상 성공을 반환하도록 설정해두고 카나리 분석 결과를 로그로만 기록하는 "관찰 전용 모드"로 시작해, 동작이 안정화된 후 실제 분석에 적용하는 단계적 접근을 권장한다.

## Q6. 서비스 메시의 타임아웃 계층 구조를 어떻게 설계해야 하는가?

### 왜 중요한가?

분산 시스템에서 타임아웃은 장애 전파를 막는 첫 번째 방어선이다. 서비스 메시는 여러 레이어에서 타임아웃을 설정할 수 있다. Envoy의 요청 타임아웃, 연결 타임아웃, 유휴 타임아웃, 그리고 쿠버네티스의 Liveness Probe 타임아웃, Readiness Probe 타임아웃까지 있다. 이 타임아웃들이 조화롭게 설계되지 않으면 예상치 못한 동작이 나타난다. 전형적인 문제는 "상위 서비스의 타임아웃이 하위 서비스보다 짧아서 정상 응답이 돌아와도 상위가 이미 타임아웃 처리를 해버리는 것"이다.

### 분析

타임아웃 계층 설계의 기본 원칙은 "하위 서비스의 타임아웃 합계 < 상위 서비스의 타임아웃"이다. 서비스 A → B → C 호출 체인에서 C의 타임아웃 = 2초, B의 타임아웃 = 3초(C 호출 2초 + 여유 1초), A의 타임아웃 = 5초(B 호출 3초 + 재시도 1회 + 여유)로 설계한다. 이 계층이 깨지면 상위에서 타임아웃이 발생해 에러를 반환했는데 하위 서비스는 계속 처리를 진행하는 낭비가 발생한다.

Istio의 VirtualService 타임아웃과 Envoy의 Upstream 연결 타임아웃은 다른 개념이다. VirtualService 타임아웃은 요청 처리 전체 시간이고, Upstream 연결 타임아웃은 TCP 연결을 맺는 시간이다. 연결 타임아웃은 보통 짧게(수백 ms) 설정하고, 요청 타임아웃은 서비스 특성에 맞게(수초~수십초) 설정한다.

gRPC에서 Deadline Propagation은 타임아웃 계층을 자동으로 관리하는 기능이다. 클라이언트가 5초 Deadline을 설정하면 이 Deadline이 호출 체인 전체에 전파된다. 중간 서비스는 남은 Deadline 내에서 처리해야 하므로 자연스럽게 계층이 형성된다. Envoy는 gRPC Deadline을 HTTP `grpc-timeout` 헤더로 변환하므로, gRPC 서비스에서 Deadline을 올바르게 설정하면 메시 레이어의 개별 타임아웃 설정 부담이 줄어든다.

### 실무 적용

서비스 의존성 그래프를 그리고 각 엣지에 현재 설정된 타임아웃을 표시한다. 그래프에서 "상위 타임아웃 < 하위 타임아웃 + 예상 레이턴시" 조건을 위반하는 엣지를 찾는다. 이 작업은 `istioctl analyze`로 기본 설정 오류를 감지하는 것에 더해 수동으로 의미론적 검증을 해야 한다. 분기별로 이 그래프를 업데이트하고 팀 전체가 리뷰하는 것이 타임아웃 설계를 최신 상태로 유지하는 실용적인 방법이다.
