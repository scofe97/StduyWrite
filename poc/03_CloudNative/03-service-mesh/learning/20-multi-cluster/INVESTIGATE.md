# Ch20. 멀티클러스터 서비스 메시 — Deep Dive Questions

## Q1. 멀티클러스터 환경에서 Split-Brain이 발생했을 때, 서비스 메시는 어떻게 일관성을 유지해야 하는가?

### 왜 중요한가?

Split-Brain은 네트워크 파티션이 발생해 두 클러스터가 서로를 인식하지 못하는 상태를 말한다. 이때 각 클러스터는 자신이 "정상"이라고 판단하고 독립적으로 트래픽을 처리한다. 서비스 메시 없이도 이 상황은 위험하지만, 메시가 있으면 문제가 더 복잡해진다. 메시의 컨트롤 플레인이 원격 클러스터의 엔드포인트 정보를 더 이상 수신하지 못하면, 이전에 캐시된 정보로 계속 트래픽을 보낼지, 아니면 페일오버를 결정할지 판단해야 한다.

이 결정이 잘못되면 두 가지 문제 중 하나가 발생한다. 죽은 엔드포인트로 트래픽이 계속 유입되어 대량 오류가 발생하거나, 반대로 살아있는 원격 클러스터를 장애로 오인해 로컬 클러스터에 트래픽이 과부하된다. 어느 쪽도 SLO를 지키기 어렵다.

### 분석

Istio의 멀티클러스터 모델에서는 각 클러스터의 Istiod가 원격 클러스터의 API 서버에서 서비스/엔드포인트 정보를 수집한다. 네트워크 파티션이 발생하면 이 Watch 연결이 끊기고, Istiod는 마지막으로 알려진 상태를 사용한다. Envoy 사이드카는 업데이트를 받지 못한 채 구식 엔드포인트 목록으로 트래픽을 보내게 된다. 이 시나리오에서 Outlier Detection이 핵심 안전망 역할을 한다. 연속 5xx가 임계치를 넘으면 해당 엔드포인트를 회로 차단기로 제거하는데, 이는 컨트롤 플레인이 아닌 데이터 플레인 수준의 자율 결정이다.

Linkerd의 ServiceMirror 컨트롤러는 원격 클러스터의 서비스를 로컬 클러스터에 미러링한다. 연결이 끊기면 미러된 서비스의 엔드포인트가 비워진다. 이는 즉각적인 페일오버를 뜻하지만, TrafficSplit을 사용해 가중치를 조정하지 않았다면 로컬 클러스터가 100% 트래픽을 갑자기 받게 된다. 사전에 `locality.kubernetes.io/zone` 레이블과 Failover 정책을 설정해두는 것이 이 상황에서 완충 역할을 한다.

핵심 질문은 "얼마나 오래된 정보까지 신뢰할 것인가"이다. CAP 정리로 보면 멀티클러스터 메시는 파티션 허용성(P)을 포기할 수 없으므로, 가용성(A)과 일관성(C) 사이에서 선택해야 한다. 대부분의 메시는 기본적으로 가용성을 택한다(구식 엔드포인트라도 계속 시도). 금융 시스템처럼 일관성이 중요하다면 이 기본값을 재고해야 한다.

### 실무 적용

Split-Brain 시나리오를 위한 Runbook을 미리 작성해두어야 한다. 구체적으로는 (1) 메시 컨트롤 플레인의 연결 상태 메트릭 알림 설정, (2) 파티션 감지 시 자동으로 TrafficSplit 가중치를 로컬로 전환하는 스크립트, (3) 복구 후 미러 엔드포인트 재동기화 절차가 필요하다. Chaos Engineering으로 정기적으로 클러스터 간 네트워크를 끊어보고 실제 동작을 관찰하는 훈련도 이 시나리오 이해에 필수적이다.

## Q2. 클러스터 간 레이턴시가 SLO에 미치는 영향을 어떻게 정량화하고, 메시 라우팅 정책으로 어디까지 완화할 수 있는가?

### 왜 중요한가?

서울 리전과 도쿄 리전 사이의 RTT는 평균 30-40ms다. 이 숫자 하나가 p99 레이턴시 SLO 설계를 완전히 바꾼다. 로컬 호출은 1ms 이하인데, 멀티클러스터 호출 경로에 이 레이턴시가 끼어들면 기존 SLO가 무너진다. 문제는 개발 단계에서는 이 영향이 보이지 않는다는 점이다. 로컬 개발 환경은 단일 클러스터이고, 스테이징도 같은 리전에 있는 경우가 대부분이다.

### 분석

레이턴시 영향을 정량화하는 첫 단계는 서비스 의존성 그래프를 그리고 각 엣지에 클러스터 경계 여부를 표시하는 것이다. Jaeger나 Tempo 같은 분산 추적 도구에서 클러스터 간 스팬을 필터링하면 실제 크로스-클러스터 레이턴시 분포를 확인할 수 있다. 여기서 p50이 아니라 p99와 p999를 봐야 한다. 네트워크는 평균이 좋아도 꼬리 레이턴시가 SLO를 위반하는 경우가 흔하다.

서비스 메시의 Locality-Aware Routing은 이 문제의 첫 번째 대응이다. Istio의 `localityLbSetting`은 같은 리전/존의 엔드포인트를 우선하고, 가중치 기반으로 원격 클러스터로 오버플로우를 조절한다. 하지만 이것만으로는 충분하지 않다. 동기 호출 체인이 깊다면(A→B→C→D가 모두 동기) 클러스터 경계를 한 번 넘는 것보다 훨씬 큰 레이턴시 누적이 발생한다. 이른바 팬아웃 지연(Fanout Latency)이다.

Hedged Request(Envoy의 `hedge_on_per_try_timeout`)는 꼬리 레이턴시를 낮추는 기법이다. 일정 시간 안에 응답이 없으면 다른 엔드포인트로 동일 요청을 병렬 발송하고 먼저 오는 응답을 사용한다. 대역폭을 더 쓰지만 p99를 낮출 수 있다. 단, 쓰기 요청에는 절대 사용해선 안 된다. 멱등성이 보장된 읽기 요청에만 적용해야 한다.

### 실무 적용

SLO를 재정의할 때 클러스터 경계를 기준으로 두 개의 SLO를 만드는 방법을 고려해볼 만하다. "동일 클러스터 p99 < 50ms", "크로스-클러스터 p99 < 150ms"처럼 경계를 명시하면 모니터링 알림 설계도 자연스럽게 따라온다. 메시의 Destination Rule에 레이턴시 기반 Outlier Detection을 설정해두면 일시적으로 느려진 원격 엔드포인트를 자동으로 회피하는 효과도 얻을 수 있다.

## Q3. 멀티클러스터에서 루트 CA를 공유할 때 발생하는 보안 위험은 무엇이며, 중간 CA 아키텍처가 실제로 그 위험을 줄이는가?

### 왜 중요한가?

mTLS 기반 서비스 메시에서 클러스터 간 상호 인증이 작동하려면 두 클러스터의 워크로드가 서로의 인증서를 신뢰해야 한다. 가장 단순한 방법은 루트 CA를 공유하는 것이다. 하지만 루트 CA는 PKI의 가장 민감한 자산이다. 루트 CA의 개인키가 노출되면 그 CA를 신뢰하는 모든 인증서가 위협받는다. 멀티클러스터 환경에서 모든 클러스터가 같은 루트를 공유한다면, 클러스터 하나가 침해되었을 때 폭발 반경이 전체 메시로 확장된다.

### 분석

중간 CA(Intermediate CA) 아키텍처는 이 위험을 계층화한다. 오프라인 루트 CA가 각 클러스터별 중간 CA 인증서에 서명하고, 실제 워크로드 인증서는 중간 CA가 발급한다. 클러스터 A가 침해되면 클러스터 A의 중간 CA를 CRL(Certificate Revocation List)에 추가하거나 해당 중간 CA 인증서를 폐기하면 클러스터 B는 영향받지 않는다. 루트 CA 자체는 오프라인이므로 네트워크 공격 표면이 없다.

Istio에서는 `cacerts` 시크릿에 루트 CA 또는 중간 CA의 키/인증서를 제공한다. Cert-Manager + Vault 조합을 사용하면 중간 CA 자동 갱신과 클러스터별 격리를 구현할 수 있다. Vault의 PKI 시크릿 엔진에서 클러스터별 중간 CA를 별도 마운트 경로로 관리하면, 각 클러스터는 자신의 중간 CA 경로에만 접근 권한을 갖는다.

하지만 중간 CA 아키텍처가 모든 위험을 해소하지는 않는다. 인증서 유효성 검사 시 SPIFFE의 SVID(SPIFFE Verifiable Identity Document) 포맷에서 TrustDomain을 확인해야 한다. Istio는 기본적으로 클러스터 간에 TrustDomain을 같게 설정하는데, 이러면 침해된 클러스터의 워크로드가 정상 클러스터에서 유효한 정체성으로 인정될 수 있다. TrustDomain을 클러스터별로 분리하고 `MeshConfig.trustDomainAliases`를 명시적으로 관리해야 이 허점을 막을 수 있다.

### 실무 적용

실제 구현에서는 루트 CA를 HSM(Hardware Security Module)에 보관하고, 중간 CA 갱신 작업은 승인 워크플로우를 통해서만 진행하도록 자동화 파이프라인을 구성한다. 인증서 만료 90일 전부터 알림을 보내고, 갱신은 배포 파이프라인에 통합한다. cert-manager의 `Certificate` 리소스와 Vault의 `Issuer`를 연결하면 이 과정을 대부분 자동화할 수 있다.

## Q4. Flat 네트워크와 Non-Flat 네트워크 멀티클러스터 모델의 비용 구조 차이는 무엇인가?

### 왜 중요한가?

서비스 메시를 멀티클러스터로 확장할 때 네트워크 아키텍처 선택이 클라우드 비용에 직접 영향을 준다. Flat 네트워크는 두 클러스터의 Pod IP가 서로 직접 라우팅 가능한 구조다. Non-Flat 네트워크는 각 클러스터가 독립된 Pod CIDR을 가지며, 클러스터 간 통신은 전용 Gateway를 통해 이루어진다. 이 차이는 단순한 아키텍처 선호의 문제가 아니라, 클라우드 비용, 운영 복잡성, 보안 경계 설계에 복합적으로 영향을 미친다.

### 분석

Flat 네트워크는 VPC Peering 또는 Transit Gateway를 사용한다. AWS에서 VPC Peering은 AZ 내 트래픽은 무료지만 AZ 간, 리전 간 트래픽에는 데이터 전송 요금이 붙는다. 멀티클러스터 서비스 메시에서 Pod 간 직접 통신이 이루어지면 이 요금이 예측하기 어렵게 쌓인다. 장점은 레이턴시가 낮고 Gateway 단일 실패 지점이 없다는 점이다.

Non-Flat 네트워크에서는 Istio의 East-West Gateway 또는 Linkerd의 Gateway가 클러스터 경계의 단일 진입/출구 역할을 한다. 트래픽이 Gateway를 통과하므로 비용 측정이 명확하다. Gateway 인스턴스 비용과 데이터 전송 비용이 분리되어 예산 관리가 쉽다. 단, Gateway가 병목이 될 수 있고, 고가용성을 위해 최소 2개 이상을 운영해야 하므로 고정 비용이 발생한다.

실제 비용 비교는 트래픽 패턴에 따라 달라진다. 크로스-클러스터 트래픽이 많다면 Non-Flat의 Gateway 처리 비용이 더 비쌀 수 있다. 반면 트래픽이 적고 예측 가능하다면 Gateway의 고정 비용이 더 크다. Cloud Provider가 제공하는 네트워크 비용 계산기로 현재 트래픽 패턴을 기준으로 시뮬레이션해보는 것이 의사결정의 출발점이다.

### 실무 적용

두 모델을 선택하기 전 현재 클러스터 간 예상 트래픽 볼륨을 측정해야 한다. 기존 로그나 메트릭에서 서비스 간 통신 빈도를 추출하고, 클러스터 경계를 넘을 것으로 예상되는 트래픽의 비율을 계산한다. 이 수치를 각 Cloud Provider의 데이터 전송 요금표와 곱하면 월간 네트워크 비용 추정치가 나온다. 여기에 Gateway 인스턴스 비용을 더해 Non-Flat 모델과 비교하면 된다.

## Q5. 멀티클러스터에서 분산 추적의 Gap은 무엇이며, 이를 완전히 해소하는 것이 가능한가?

### 왜 중요한가?

분산 추적은 서비스 메시의 핵심 가시성 기능이다. 단일 클러스터에서는 Jaeger나 Zipkin이 스팬을 수집하고 TraceID로 전체 요청 경로를 연결한다. 멀티클러스터로 확장하면 이 연결이 끊어진다. 클러스터 A의 스팬과 클러스터 B의 스팬이 같은 TraceID를 공유하더라도, 두 클러스터가 서로 다른 Jaeger 인스턴스를 사용한다면 하나의 트레이스로 볼 수 없다. 이 Gap은 장애 분석 시 원인 파악을 어렵게 만든다.

### 분석

Gap을 해소하는 접근법은 세 가지다. 첫째, 중앙 집중형 추적 백엔드를 사용한다. 모든 클러스터의 OTel Collector가 하나의 Tempo 또는 Jaeger 클러스터로 스팬을 전송한다. 구조가 단순하지만 추적 백엔드 자체가 단일 실패 지점이 되고, 대규모 트래픽에서 수집 파이프라인이 병목이 될 수 있다.

둘째, 연합형(Federation) 추적을 사용한다. 각 클러스터가 독립 추적 백엔드를 운영하고, 쿼리 레이어에서 TraceID로 여러 백엔드를 동시에 조회해 스팬을 병합한다. Grafana Tempo의 Multi-Tenant 모드나 Jaeger의 Remote Storage 플러그인이 이 역할을 할 수 있다. 복잡하지만 백엔드 격리가 유지된다.

셋째, Baggage Propagation을 강화한다. W3C Trace Context 헤더(`traceparent`, `tracestate`)가 클러스터 경계를 넘어 전파되는지 확인해야 한다. 서비스 메시의 Envoy 사이드카는 기본적으로 이 헤더를 전달하지만, 어플리케이션이 헤더를 전파하지 않으면 스팬 체인이 끊어진다. 특히 메시지 큐를 통한 비동기 통신에서 TraceID 전파는 자동으로 이루어지지 않아 수동 구현이 필요하다.

완전한 해소는 현실적으로 어렵다. 비동기 처리, 배치 작업, 타사 서비스 호출 등에서 항상 TraceID 연결이 끊길 가능성이 있다. 목표는 완전한 연결이 아니라, 중요한 요청 경로(Critical Path)에서의 연결을 보장하는 것이다.

### 실무 적용

OpenTelemetry Collector를 각 클러스터에 DaemonSet으로 배포하고, 중앙 Collector 클러스터로 스팬을 집계하는 파이프라인을 구성한다. `service.cluster.name` 속성을 모든 스팬에 추가하면 Grafana 대시보드에서 클러스터별 필터링이 가능하다. 멀티클러스터 요청 경로를 대표하는 E2E 테스트를 작성하고, 해당 테스트의 TraceID가 두 클러스터 모두에서 스팬을 생성하는지 주기적으로 검증하는 것이 관찰 가능성 품질을 유지하는 실용적인 방법이다.

## Q6. 멀티클러스터 서비스 디스커버리에서 DNS 기반과 메시 기반 접근의 근본적인 트레이드오프는 무엇인가?

### 왜 중요한가?

멀티클러스터에서 서비스 A가 다른 클러스터의 서비스 B를 호출하려면 B의 주소를 알아야 한다. 가장 단순한 방법은 DNS로 B의 ClusterIP 또는 LoadBalancer IP를 노출하는 것이다. 하지만 DNS는 상태를 모르고, 헬스체크를 하지 않으며, 빠른 페일오버를 지원하지 않는다. 메시 기반 디스커버리는 엔드포인트 상태를 실시간으로 추적하고 라우팅 정책을 세밀하게 제어할 수 있다. 이 차이가 실제 가용성에 어떤 영향을 주는지 이해하는 것이 중요하다.

### 분석

DNS 기반 접근(CoreDNS + External DNS)은 구성이 단순하다. 클러스터 외부에 `serviceB.cluster-b.example.com`을 등록하고 TTL을 짧게 설정하면 기본적인 페일오버가 가능하다. 하지만 DNS TTL은 최소 30초 수준이고, 캐싱 레이어가 여러 단계에 걸쳐 있어 실제 전파 시간은 훨씬 길다. 장애 발생 후 DNS 레코드가 업데이트되기 전까지 클라이언트는 죽은 엔드포인트를 계속 시도한다.

메시 기반 디스커버리에서 Istio의 `ServiceEntry`나 Linkerd의 `ServiceMirror`는 원격 서비스의 엔드포인트를 실시간으로 추적한다. Envoy의 EDS(Endpoint Discovery Service)를 통해 초 단위로 엔드포인트 목록이 갱신된다. 이는 DNS보다 훨씬 빠른 페일오버를 가능하게 한다. 하지만 메시 컨트롤 플레인이 다운되면 마지막 알려진 엔드포인트로만 라우팅이 고정된다(ADS 연결 끊김 시 Envoy는 기존 설정 유지).

두 접근의 핵심 차이는 "누가 헬스체크를 하는가"다. DNS는 애플리케이션 또는 로드밸런서가 헬스체크를 담당한다. 메시는 데이터 플레인의 Outlier Detection이 헬스체크 역할을 하고, 컨트롤 플레인이 엔드포인트 목록을 관리한다. 이 두 메커니즘이 조화롭게 작동해야 빠르고 정확한 페일오버가 된다.

### 실무 적용

멀티클러스터 초기 구축 단계에서는 DNS 기반과 메시 기반을 레이어로 쌓는 하이브리드 접근이 안전하다. DNS는 클러스터 진입점(External LoadBalancer)을 가리키고, 메시는 클러스터 내부 라우팅을 담당한다. 이렇게 하면 메시 컨트롤 플레인 장애 시에도 DNS 레이어에서 기본적인 라우팅이 유지된다. 메시에 완전히 의존하기 전에 컨트롤 플레인 HA 구성이 선행되어야 한다.
