# Ch15. Istio 보안 — Deep Dive Questions

## Q1. STRICT mTLS로 마이그레이션하는 과정에서 어떤 연결이 끊기는가?

### 왜 중요한가?
PeerAuthentication을 `PERMISSIVE`에서 `STRICT`로 전환하면 메시 내 모든 통신이 mTLS를 강제한다. 이 전환이 "보안을 강화한다"는 점은 명확하지만, 그 과정에서 예상치 못한 연결 단절이 발생할 수 있다. 어떤 연결이 영향받는지 미리 파악하지 않으면, 전환 후 서비스 장애를 진단하는 데 오랜 시간이 걸린다.

### 분析
사이드카가 없는 Pod에서 사이드카가 있는 서비스로의 연결이 첫 번째 피해자다. Kubernetes Job, CronJob, 또는 `sidecar.istio.io/inject: "false"` 레이블이 붙은 Pod는 Envoy 사이드카가 없으므로 mTLS를 수행할 수 없다. `STRICT` 전환 후 이런 Pod들이 메시 내 서비스를 호출하면 연결이 거부된다. 어떤 Pod가 사이드카 없이 실행 중인지 전환 전에 반드시 목록화해야 한다.

모니터링 시스템도 영향을 받는다. Prometheus가 메시 내 서비스를 직접 스크래핑하는 구성이라면, Prometheus Pod에 사이드카가 없을 경우 스크래핑이 실패한다. Istio는 이 문제를 위해 Prometheus 스크래핑 시 mTLS를 우회하는 특수 설정(`prometheusPort.protocol: HTTP`)을 제공하지만, 이를 미리 설정하지 않으면 전환 즉시 메트릭 수집이 끊긴다.

헬스 체크도 STRICT 전환의 영향을 받는다. Kubernetes kubelet이 수행하는 HTTP livenessProbe와 readinessProbe는 mTLS 인증서를 보내지 않는다. Istio는 기본적으로 kubelet 헬스 체크를 mTLS에서 제외하는 처리를 하지만, 특정 Istio 버전이나 구성에서 이 예외가 올바르게 동작하지 않는 경우가 보고됐다. 전환 전 헬스 체크 동작을 명시적으로 검증해야 한다.

인증서 갱신 타이밍 문제도 있다. `STRICT` 전환 직후, 일부 Envoy 인스턴스는 아직 새 인증서를 받지 못한 상태일 수 있다. istiod가 새 mTLS 정책을 모든 Envoy에 배포하는 데 시간이 걸리기 때문이다. 이 과도기에 일부 서비스 간 연결이 일시적으로 실패할 수 있다. 특히 클러스터 규모가 크면 이 과도기가 수 분에 달할 수 있다.

### 실무 적용
STRICT mTLS 전환 절차: (1) `PERMISSIVE` 상태에서 `istio_requests_total{connection_security_policy="none"}` 메트릭으로 mTLS 없이 통신하는 연결 목록 수집 (이것이 STRICT 후 끊길 연결들이다), (2) 사이드카 없는 Pod 파악(`kubectl get pods -A -l '!sidecar.istio.io/inject'` 또는 실제 컨테이너 수로 확인), (3) 해당 Pod들에 사이드카를 추가하거나, `PeerAuthentication`에 포트별 예외를 추가, (4) STRICT 전환은 네임스페이스 단위로 점진적으로 적용하고, 각 네임스페이스 전환 후 24시간 모니터링, (5) 전환 직후 `connection_security_policy` 메트릭이 `mutual_tls`로만 나오는지 확인.

## Q2. ext_authz(외부 인가)의 레이턴시 오버헤드는 허용 가능한 수준인가? 어떻게 측정하고 최적화하는가?

### 왜 중요한가?
`ExtensionProvider`로 설정하는 외부 인가(ext_authz)는 모든 요청을 외부 인가 서비스(Open Policy Agent, 커스텀 인가 서버)에 전달해 허용/거부 판단을 받는 패턴이다. 세밀한 인가 로직을 중앙에서 관리할 수 있는 장점이 있지만, 모든 요청에 추가 RPC 호출이 발생한다. 이 오버헤드가 서비스 레이턴시에 미치는 영향을 정확히 이해해야 도입 여부를 결정할 수 있다.

### 分析
ext_authz 레이턴시의 구성 요소는 네트워크 왕복 시간(RTT), 인가 서버 처리 시간, Envoy 연결 관리 오버헤드다. 인가 서버가 같은 클러스터 내에 있다면 RTT는 수 밀리초 수준이지만, 인가 서버의 처리 시간은 정책 복잡도에 따라 크게 달라진다. OPA는 간단한 정책에서 마이크로초 수준이지만, 복잡한 Rego 정책과 대규모 데이터 조회가 결합되면 수십 밀리초가 걸릴 수 있다.

HTTP 기반 ext_authz와 gRPC 기반의 차이도 중요하다. gRPC 기반 인가는 연결 재사용이 효율적이고, HTTP/2 멀티플렉싱으로 연결 오버헤드를 줄인다. HTTP 기반은 각 요청마다 HTTP/1.1 연결을 열거나 keepalive를 사용한다. 초당 수천 건의 요청이 있다면 gRPC 기반이 훨씬 효율적이다.

캐싱 전략이 레이턴시 최적화의 핵심이다. 동일 principal(서비스 계정)의 동일 자원에 대한 인가 결과는 시간적으로 변하지 않는 경우가 많다. OPA는 결정 캐시를 지원하고, Envoy의 ext_authz 필터도 응답 캐싱 설정이 있다. 그러나 캐시 유효 시간(TTL) 결정이 어렵다. 짧으면 캐시 효과가 없고, 길면 정책 변경이 즉시 반영되지 않아 보안 허점이 생긴다.

인가 서버 자체의 가용성도 고려해야 한다. `failOpen` vs `failClose` 설정에 따라, 인가 서버가 다운되면 모든 요청이 허용될 수도, 차단될 수도 있다. 프로덕션에서는 인가 서버 자체의 고가용성(HA)이 ext_authz 채택의 전제 조건이다.

### 실무 적용
ext_authz 도입 전 레이턴시 측정 방법: (1) 스테이징에서 ext_authz 없이 baseline 레이턴시 측정, (2) ext_authz 활성화 후 동일 테스트로 P50/P95/P99 레이턴시 비교, (3) 오버헤드가 SLO 예산 안에 들어오는지 확인. 최적화 순서: 인가 서버를 동일 네임스페이스에 DaemonSet 또는 다중 Replica로 배포 → gRPC 프로토콜 사용 → 결정 캐시 활성화 → 정책 단순화로 처리 시간 단축. 레이턴시 예산이 10ms 이하인 서비스에서는 ext_authz 대신 AuthorizationPolicy 기반 선언적 정책을 우선 검토해야 한다.

## Q3. AuthorizationPolicy 순서와 병합 규칙에서 예상치 못한 접근 허용이 발생하는 패턴은 무엇인가?

### 왜 중요한가?
AuthorizationPolicy는 작업(action)으로 `ALLOW`, `DENY`, `CUSTOM`을 지원한다. 여러 정책이 같은 워크로드에 적용될 때, Istio는 특정 우선순위 규칙으로 이를 평가한다. 이 규칙을 잘못 이해하면, 의도한 것보다 더 많은 접근을 허용하거나 차단하는 정책을 작성할 수 있다. 보안 정책의 버그는 기능 버그보다 발견이 늦고 영향이 크다.

### 분析
Istio AuthorizationPolicy 평가 순서는 문서화되어 있지만 직관적이지 않다. 평가 순서는: (1) `CUSTOM` 정책 먼저 평가(실패 시 거부), (2) `DENY` 정책 평가(일치 시 거부), (3) `ALLOW` 정책이 없으면 암시적 허용 없음 — DENY 정책만 있으면 나머지는 허용, (4) `ALLOW` 정책이 있으면 명시적 허용 필요.

가장 흔한 실수는 `DENY` 정책을 추가하면서 `ALLOW` 정책 없이 "나머지는 허용"을 기대하는 경우다. 규칙은 반대다. `ALLOW` 정책이 하나라도 있으면, `ALLOW`에 명시되지 않은 접근은 암시적으로 거부된다. 반대로 `DENY` 정책만 있으면, `DENY`에 해당하지 않는 접근은 암시적으로 허용된다. 이 두 경우의 기본값이 반대라는 점이 혼란의 근원이다.

네임스페이스 정책과 워크로드 정책이 함께 있을 때도 주의가 필요하다. 네임스페이스 수준(selector 없음)의 `ALLOW` 정책과 특정 워크로드의 `ALLOW` 정책이 공존하면, 두 정책의 합집합이 허용된다. 워크로드별로 더 엄격한 정책을 적용하려면, 네임스페이스 정책과 워크로드 정책의 관계를 명확히 설계해야 한다.

`source.principals`와 `source.namespaces`의 조합 매칭도 함정이 있다. 두 조건을 같은 `from` 블록에 넣으면 AND 조건이지만, 별도 `from` 블록에 넣으면 OR 조건이 된다. "A 네임스페이스의 B 서비스 계정만 허용"을 표현하려면 두 조건이 같은 `from` 블록에 있어야 하는데, 실수로 별도 블록에 넣으면 "A 네임스페이스의 모든 서비스 계정 OR 모든 네임스페이스의 B 서비스 계정"이 허용된다.

### 실무 적용
정책 작성 후 `istioctl x authz check <pod>` 명령으로 특정 요청이 어떤 정책에 의해 허용/거부되는지 사전 검증해야 한다. 보안 정책은 `deny-all` 기본 정책으로 시작하고, 필요한 접근만 명시적으로 `ALLOW`로 열어주는 화이트리스트 방식이 최안전하다: `action: DENY` + 모든 source/operation을 `*`로 설정한 정책을 기본으로 두고, 서비스별 `ALLOW` 정책을 추가하는 구조다. 정책 변경 시 스테이징에서 전체 접근 경로를 테스트한 후 프로덕션에 적용해야 한다.

## Q4. JWT 클레임 기반 라우팅에는 어떤 보안 취약점이 잠재되어 있는가?

### 왜 중요한가?
Istio는 JWT 클레임을 기반으로 트래픽을 라우팅할 수 있다. 예를 들어 JWT의 `role` 클레임이 `admin`이면 v2로, `user`이면 v1으로 라우팅하는 설정이 가능하다. 편리한 기능이지만, JWT 클레임 기반 라우팅과 인가(authorization)를 혼동하면 보안 허점이 생긴다. 라우팅과 인가는 서로 다른 목적으로 설계되었다.

### 分析
JWT 클레임 기반 라우팅은 `RequestAuthentication`으로 JWT를 검증한 후, 클레임을 Envoy 내부 메타데이터로 변환하고, VirtualService의 `headers` 매칭으로 라우팅 결정에 사용하는 흐름이다. 그러나 이 라우팅은 `AuthorizationPolicy`가 아니므로, 접근 허용/거부를 결정하지 않는다. 라우팅 규칙에 매칭되지 않는 요청은 기본 라우팅 규칙으로 처리되거나 오류를 반환할 수 있지만, 차단된다는 보장이 없다.

`RequestAuthentication`의 동작 방식에 대한 오해가 취약점의 원인이다. `RequestAuthentication`은 JWT가 있으면 검증하지만, JWT가 없는 요청은 통과시킨다. JWT 없이 admin 기능에 직접 접근하는 요청은 `RequestAuthentication`을 통과한다. JWT가 없으면 클레임 기반 라우팅 조건에 매칭되지 않아 다른 경로로 라우팅될 수 있다. 진정한 인가는 `AuthorizationPolicy`로 "JWT의 `role` 클레임이 `admin`인 요청만 허용"을 명시해야 한다.

헤더 기반 라우팅과의 혼용도 위험하다. 일부 구현에서 JWT 클레임을 HTTP 헤더로 변환해 라우팅하는 패턴을 사용한다. 그런데 클라이언트가 직접 해당 헤더(`X-User-Role: admin`)를 설정해 보내면, JWT 없이도 admin 경로로 라우팅될 수 있다. Envoy는 내부 메타데이터로 변환된 JWT 클레임과 클라이언트가 보낸 헤더를 구분할 수 있지만, 이 구분이 올바르게 설정되어 있는지 확인해야 한다.

토큰 만료 처리도 확인이 필요하다. JWT가 만료된 후에도 캐시된 클레임 정보가 일정 시간 동안 라우팅 결정에 사용될 수 있다. Envoy는 JWT 만료를 실시간으로 확인하지만, 클록 스큐(clock skew)가 있는 환경에서는 만료된 토큰이 잠시 유효한 것으로 처리될 수 있다.

### 실무 적용
JWT 기반 접근 제어 패턴: `RequestAuthentication`은 토큰 서명 검증만 담당하고, 실제 접근 제어는 `AuthorizationPolicy`로 처리하는 역할 분리가 필수다. `AuthorizationPolicy`에서 `request.auth.claims[role]` 조건으로 JWT 클레임을 확인하면, JWT 없는 요청은 클레임 조건에 매칭되지 않아 거부된다. 또한 인가 정책 테스트 시 (1) 유효한 JWT + 올바른 클레임, (2) 유효한 JWT + 잘못된 클레임, (3) JWT 없음, (4) 만료된 JWT, (5) 서명이 잘못된 JWT 다섯 가지 경우를 모두 테스트해야 한다.

## Q5. ambient 모드의 L4 전용 보안으로 충분한 경우와 부족한 경우를 어떻게 구분하는가?

### 왜 중요한가?
ambient 모드에서 ztunnel은 L4 mTLS를 제공하고, L7 정책은 waypoint를 통해서만 적용된다. waypoint 없이 ztunnel만 사용하는 구성은 전송 암호화와 서비스 계정 기반 ID를 제공하지만, HTTP 헤더 수준의 세밀한 인가는 불가능하다. 어떤 서비스가 L4 보안으로 충분하고 어떤 서비스가 L7 보안이 필요한지 판단 기준이 있어야 waypoint 배포를 최소화하면서 보안 요건을 충족할 수 있다.

### 分析
L4 보안만으로 충분한 경우를 먼저 살펴보면, 서비스 간 통신이 단순한 TCP 기반이고, 접근 제어가 "어떤 서비스가 접근 가능한가"(서비스 계정 기반)로 표현될 수 있을 때다. 예를 들어 데이터베이스 서비스에 특정 애플리케이션 서비스만 접근할 수 있게 하는 정책은 L4 수준 source principal 기반으로 표현 가능하다. ztunnel의 AuthorizationPolicy는 L4 속성(source principal, destination principal, port)에 기반한 정책을 지원한다.

L7 보안이 필요한 경우는 보다 명확하다. HTTP 메서드에 따른 접근 제어(GET은 허용, POST는 거부), URL 경로 기반 접근 제어(`/admin` 경로는 관리자만), HTTP 헤더 기반 정책(JWT 클레임 검증), 요청 본문 기반 정책은 L7 수준에서만 가능하다. 이런 정책이 필요한 서비스에는 waypoint를 반드시 배포해야 한다.

gRPC 서비스의 경우 판단이 미묘하다. gRPC는 HTTP/2 기반이므로, 특정 gRPC 메서드(`/package.Service/Method`)에 대한 접근 제어는 HTTP/2 경로 기반 정책으로 표현할 수 있어 L7 waypoint가 필요하다. 그러나 단순히 "이 서비스 계정만 이 gRPC 서비스에 접근 가능"이라면 L4로 충분하다.

보안 요구사항과 성능 트레이드오프를 명시적으로 결정해야 한다. waypoint는 추가 hop과 리소스 오버헤드를 의미한다. 모든 서비스에 waypoint를 배포하면 ambient 모드의 리소스 절감 효과가 반감된다. 보안 요건 분석으로 waypoint가 필요한 서비스를 최소화하고, 나머지는 L4 ztunnel만으로 운영하는 것이 ambient 모드의 설계 의도에 맞다.

### 실무 적용
서비스별 보안 요건 분류 방법: 각 서비스의 API 스펙을 검토하고, 접근 제어가 (1) "누가" 접근하는가(서비스 계정)만 필요한지, (2) "무엇을"(HTTP 메서드, 경로, 헤더, JWT)까지 제어해야 하는지 분류한다. (1)이면 L4만으로 충분하고, (2)이면 waypoint가 필요하다. 이 분류 결과를 인프라 문서에 기록하고, 새 서비스 도입 시 보안 요건 평가 프로세스를 통해 waypoint 배포 여부를 결정하는 체계를 갖추어야 한다.

## Q6. 상호 인증(mutual authentication)과 인가(authorization)를 혼동하면 어떤 보안 허점이 생기는가?

### 왜 중요한가?
mTLS는 "상호 인증"을 제공한다. 클라이언트와 서버 모두 인증서로 서로를 인증하므로, 클라이언트가 누구인지 신뢰할 수 있다. 그러나 "인증됐다"는 것이 "인가됐다"는 것을 의미하지 않는다. 이 두 개념을 혼동하면, mTLS만 설정하고 인가 정책을 설정하지 않아 메시 내 모든 서비스가 서로에게 무제한 접근할 수 있는 상태가 된다.

### 分析
Istio에서 mTLS(PeerAuthentication)가 활성화되면, 서비스 간 통신이 암호화되고 양측이 SPIFFE ID를 교환한다. 이때 "A 서비스가 B 서비스임을 증명했다"는 것은 확인되지만, "A 서비스가 C 서비스에 접근할 권한이 있는가"는 별도 인가 정책 없이는 제어되지 않는다. mTLS만 설정된 클러스터에서는 메시 내 모든 서비스가 다른 모든 서비스에 접근 가능하다. 내부 위협(compromised service)이 수평 이동(lateral movement)하는 것을 막을 수 없다.

서비스 메시의 "신뢰 경계" 오해가 이 문제의 근원이다. "mTLS를 설정했으니 서비스 간 통신은 안전하다"는 믿음에서 인가 정책을 생략하는 경우가 있다. 그러나 mTLS는 암호화와 ID 검증을 제공하는 것이지, 최소 권한 원칙(principle of least privilege)을 자동으로 구현하지 않는다. 공격자가 메시 내 하나의 서비스를 탈취하면, 인가 정책 없이는 모든 다른 서비스에 자유롭게 접근할 수 있다.

`AuthorizationPolicy`의 `AUDIT` 액션도 혼동을 만들 수 있다. `AUDIT`는 요청을 기록하지만 허용/거부 결정을 하지 않는다. `AUDIT` 정책만 있고 `ALLOW`/`DENY` 정책이 없으면, 모든 접근이 허용되고 기록만 된다. "정책이 있으니 제어된다"는 착각이다.

제로 트러스트 네트워크의 실제 구현은 mTLS + AuthorizationPolicy의 조합이다. 인증(mTLS)으로 "당신이 누구인지" 확인하고, 인가(AuthorizationPolicy)로 "당신이 무엇을 할 수 있는지" 제어한다. 두 요소가 모두 있어야 제로 트러스트가 달성된다.

### 실무 적용
메시 보안 성숙도를 평가하는 질문: "모든 서비스 간 통신이 AuthorizationPolicy로 명시적으로 허용됐는가, 아니면 암시적으로 허용됐는가?" 암시적 허용이 있다면 아직 제로 트러스트가 달성되지 않은 것이다. 권장 접근법은: (1) 메시 전체에 `deny-all` AuthorizationPolicy 적용(action: DENY, rules 없음 — 모든 트래픽 거부), (2) 서비스 의존성 맵을 기반으로 필요한 서비스 간 통신만 `ALLOW` 정책 추가, (3) 정기적으로 실제 통신 패턴(Kiali 서비스 그래프)과 인가 정책을 비교해 불일치 탐지. 이 과정은 서비스 의존성 문서화를 강제하는 부수 효과도 있다.
