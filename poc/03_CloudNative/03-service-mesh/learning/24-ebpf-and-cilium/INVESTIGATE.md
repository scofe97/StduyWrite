# Ch24. eBPF와 Cilium Service Mesh — Deep Dive Questions

## Q1. eBPF Verifier의 한계가 복잡한 프로그램 개발에 어떤 영향을 미치는가?

### 왜 중요한가?

eBPF 프로그램은 커널 공간에서 실행되므로 안전성이 필수다. eBPF Verifier는 프로그램이 커널을 손상시키거나 무한 루프에 빠지지 않도록 로딩 시점에 정적 검증을 수행한다. Cilium처럼 복잡한 네트워크 정책을 eBPF로 구현할 때 이 Verifier의 한계가 개발과 운영에 실질적인 제약을 준다. 이 제약을 이해하지 못하면 Cilium이 특정 기능을 지원하지 않는 이유, 또는 커널 버전 요구사항이 왜 계속 올라가는지를 파악하기 어렵다.

### 분석

Verifier는 프로그램을 DAG(Directed Acyclic Graph)로 분석하며 복잡도를 제한한다. 커널 5.2 이전에는 최대 4,096개 인스트럭션만 허용했다. 커널 5.2 이후 100만 개로 늘었지만, Verifier가 분석하는 상태 수가 폭발적으로 증가할 수 있다. 반복문이 제한되는 이유도 여기 있다. 커널 5.3에서 bounded loops가 도입되었지만, 루프 횟수를 Verifier가 정적으로 파악할 수 있는 경우만 허용된다.

Cilium의 eBPF 프로그램은 이 제약을 우회하기 위해 여러 기법을 사용한다. 첫째, 긴 프로그램을 여러 BPF 프로그램으로 분리하고 `bpf_tail_call`로 체이닝한다. 각 프로그램이 4,096 인스트럭션 제한(구버전 커널 대응) 안에 들어오게 쪼개는 것이다. 둘째, 복잡한 로직은 BPF Map에 미리 컴파일된 결정 테이블로 저장하고, BPF 프로그램은 테이블을 조회만 한다. 이 접근은 Verifier를 통과하기 쉽지만, Map 업데이트 로직이 복잡해진다.

이 제약의 실질적 영향은 기능 구현 시 커널 버전 의존성이 생긴다는 점이다. "CiliumNetworkPolicy에서 DNS 기반 필터링을 사용하려면 커널 5.3+ 권장"이라는 문서 문구는 Bounded Loops 지원이 필요하기 때문이다. 커널을 업그레이드할 수 없는 환경에서는 일부 Cilium 기능을 사용하지 못한다.

### 실무 적용

Cilium을 사용하는 노드의 커널 버전을 모니터링하고, Cilium 릴리스 노트에서 "requires kernel X.X" 표기를 정기적으로 확인한다. `cilium-agent` 로그에서 eBPF 프로그램 로드 실패 메시지(`verifier error`)가 발생하면, 해당 기능의 커널 요구사항을 먼저 확인한다. 새 기능 도입 전 `cilium status --verbose`와 `cilium debuginfo`로 현재 환경에서 지원되는 기능 목록을 파악하는 습관이 필요하다.

## Q2. WireGuard와 mTLS는 동일한 암호화 목적을 달성하는가, 아니면 근본적으로 다른 보안 모델인가?

### 왜 중요한가?

Cilium은 WireGuard를 사용한 투명한 암호화를 제공한다. Istio와 Linkerd는 mTLS를 사용한다. 두 방법 모두 "트래픽이 암호화된다"는 결과를 내지만, 보안 모델은 다르다. 이 차이를 이해하지 못하면 "WireGuard 쓰니까 mTLS 필요 없다"라는 잘못된 결론에 이르거나, 반대로 "mTLS가 있으니 WireGuard는 중복이다"라는 판단을 내릴 수 있다.

### 분析

mTLS의 핵심은 상호 인증(Mutual Authentication)이다. 서비스 A와 B가 각자 인증서를 교환하고 검증한다. "나는 서비스 A이고, 너는 정말 서비스 B가 맞는가?"를 확인한다. SPIFFE/SVID로 발급된 인증서에는 서비스 아이덴티티가 포함되어 있어, 인증서 검증이 곧 서비스 아이덴티티 검증이 된다. 정책 엔진(AuthorizationPolicy)은 이 아이덴티티를 기반으로 "서비스 A는 서비스 B의 `/api` 경로에 접근 가능하다"를 결정한다.

WireGuard는 노드 간 암호화 터널을 생성한다. Pod A(노드 1)에서 Pod B(노드 2)로 트래픽이 흐를 때 노드 1과 노드 2 사이 구간이 암호화된다. WireGuard는 노드 수준 아이덴티티를 사용한다. Pod A가 Pod C로 위장할 수 없지만, 노드 수준에서 인증할 뿐 Pod 수준의 서비스 아이덴티티를 검증하지는 않는다.

두 모델의 차이를 요약하면 이렇다. WireGuard는 "올바른 노드에서 오는 트래픽인가(노드 인증 + 암호화)"를 보장하고, mTLS는 "올바른 서비스에서 오는 트래픽인가(서비스 인증 + 암호화)"를 보장한다. 따라서 서비스 수준의 세밀한 접근 제어(Zero Trust의 "Least Privilege")가 필요하다면 WireGuard만으로는 부족하다. mTLS가 제공하는 서비스 아이덴티티 기반 정책 적용이 필요하다.

Cilium은 이 두 가지를 레이어로 조합할 수 있다. WireGuard로 노드 간 암호화를 제공하고, Cilium의 L7 정책 엔진으로 서비스 수준 접근 제어를 적용한다. Cilium의 L7 정책은 mTLS와 달리 인증서 기반 아이덴티티가 아닌 Kubernetes 아이덴티티(ServiceAccount, 레이블)를 사용한다.

### 실무 적용

보안 요구사항 문서에서 "암호화"와 "상호 인증"을 분리해서 정의한다. PCI-DSS 요구사항 4.2.1("강력한 암호화로 전송 중 PAN 보호")은 WireGuard로 충족 가능하다. Zero Trust 아키텍처의 "모든 접근을 서비스 아이덴티티 기반으로 검증"은 mTLS 또는 Cilium의 Kubernetes 아이덴티티 기반 L7 정책으로 충족한다. 이 두 요구사항을 혼동하지 않도록 보안 팀과 명시적으로 논의해야 한다.

## Q3. Cilium CNI를 선택하면 어느 정도의 Lock-In이 발생하며, 이를 최소화할 수 있는가?

### 왜 중요한가?

Cilium은 CNI(Container Network Interface)이면서 동시에 서비스 메시 기능을 제공한다. CNI는 쿠버네티스 클러스터의 가장 기저에 있는 네트워킹 레이어다. CNI를 교체하는 것은 클러스터 전체 노드를 재구성해야 할 만큼 큰 작업이다. "Cilium으로 서비스 메시까지 통합"의 장점은 명확하지만, 이 선택이 미래의 유연성을 얼마나 제한하는지 냉정하게 평가해야 한다.

### 분析

Lock-In은 두 레이어에서 발생한다. 첫째, CNI 레이어의 Lock-In이다. Cilium의 `CiliumNetworkPolicy`는 표준 `NetworkPolicy`를 확장한 Cilium 전용 CRD다. DNS 기반 필터링, L7 정책, FQDN 기반 Egress 제어 등 Cilium 고급 기능을 사용하면 이 CRD가 클러스터 전반에 퍼진다. 다른 CNI(Calico, Flannel)로 교체하면 이 CRD가 모두 무효화된다.

둘째, 서비스 메시 레이어의 Lock-In이다. Cilium Service Mesh를 사용하면 `CiliumClusterwideNetworkPolicy`, Envoy 없이 eBPF로 처리되는 L7 정책 등이 추가된다. 사이드카 기반 메시(Istio)로 마이그레이션하려면 메시 설정을 모두 다시 작성해야 한다.

Lock-In을 최소화하는 전략은 표준 API를 최대한 사용하는 것이다. 기본 `NetworkPolicy`만 사용하고 `CiliumNetworkPolicy`를 사용하지 않으면 CNI 교체가 쉬워진다. 서비스 메시는 Gateway API로 트래픽 정책을 작성하면 메시 교체 비용이 줄어든다. 하지만 표준만 사용하면 Cilium을 선택하는 이유(DNS 필터링, 성능 최적화)를 포기하는 것이기도 하다. 이는 "차별화 기능 활용"과 "이동성 유지" 사이의 트레이드오프다.

현실적으로 CNI는 클러스터 생애주기에서 한 번 선택하면 잘 바꾸지 않는다. 중요한 것은 CNI 선택을 가볍게 보지 않고, 장기적인 네트워킹 요구사항을 충분히 검토한 후 결정하는 것이다.

### 실무 적용

새 클러스터 설계 시 "5년 후에도 이 CNI를 쓸 것인가?"를 팀 내에서 논의한다. Cilium을 선택한다면 어떤 기능을 쓸 것인지 명시하고, 그 기능이 표준 API로 대체 가능한지 기록해둔다. 이 문서가 나중에 마이그레이션 범위를 평가할 때 기준이 된다. 또한 클러스터 IaC(Terraform, Pulumi) 코드에 CNI 선택 이유와 날짜를 주석으로 남기면 미래의 의사결정자에게 컨텍스트를 제공할 수 있다.

## Q4. eBPF 프로그램을 디버깅하는 것이 왜 어렵고, 실제 트러블슈팅에서 어떤 도구를 사용하는가?

### 왜 중요한가?

eBPF 프로그램은 커널 공간에서 실행되므로 전통적인 디버깅 방법(GDB, printf 디버깅)을 사용할 수 없다. Cilium이 네트워크 문제를 일으킬 때 — 패킷이 드롭되거나, 연결이 예상치 않게 차단되거나, 레이턴시가 갑자기 높아질 때 — 원인이 Cilium의 eBPF 프로그램에 있는지 확인하기 어렵다. 이 가시성 부재가 운영 팀에게 큰 부담이 된다.

### 분析

Cilium이 제공하는 일차 디버깅 도구는 `cilium monitor`다. 이 명령은 eBPF 프로그램이 패킷을 처리하는 이벤트를 실시간으로 스트리밍한다. 패킷이 드롭된 경우 드롭 이유(`DROP_REASON_POLICY`, `DROP_NO_ENDPOINT` 등)가 표시된다. Cilium 1.14부터는 Hubble이 이 데이터를 더 구조적으로 제공한다. `hubble observe --verdict DROPPED`로 드롭된 패킷만 필터링해 원인을 파악할 수 있다.

BPF Map 상태를 직접 조회하는 것도 중요한 디버깅 방법이다. `cilium bpf policy get <endpoint-id>`로 특정 Pod에 적용된 정책을 확인하고, `cilium bpf ct list global`로 커넥션 추적 테이블을 볼 수 있다. 패킷이 차단되는 이유가 정책인지, 커넥션 상태 불일치인지를 이 도구로 구분한다.

더 저수준의 디버깅이 필요할 때는 `bpftool`을 사용한다. `bpftool prog show`로 로드된 eBPF 프로그램 목록을 확인하고, `bpftool prog dump xlated id <prog-id>`로 eBPF 바이트코드를 사람이 읽을 수 있는 형태로 출력한다. `bpftrace`를 사용하면 실행 중인 eBPF 프로그램의 특정 지점에서 변수 값을 추적하는 ad-hoc 스크립트를 작성할 수 있다.

eBPF 디버깅의 근본적인 어려움은 "어디서 잘못됐는가"를 좁히는 과정이다. 레이어(L3/L4/L7), 방향(Ingress/Egress), 엔드포인트(Source/Destination) 조합이 많기 때문에, 체계적인 이분 탐색(Binary Search) 접근이 필요하다. "이 트래픽이 source Pod에서 나가고 있는가? → 노드 네트워크를 통과하는가? → destination 노드에 도달하는가? → destination Pod에 전달되는가?"를 단계별로 검증한다.

### 실무 적용

Cilium 트러블슈팅 Runbook을 팀 위키에 작성한다. 증상별(연결 거부, 타임아웃, 높은 레이턴시)로 사용할 명령어 시퀀스를 정리한다. `cilium connectivity test`는 기본 연결 검증에 유용하며, CI/CD에 통합해 메시 정책 변경 후 자동으로 실행하면 배포 전 문제를 조기에 발견할 수 있다.

## Q5. 프로덕션에서 eBPF를 위한 커널 업그레이드 계획은 어떻게 수립해야 하는가?

### 왜 중요한가?

Cilium의 새 기능은 종종 최신 커널을 요구한다. Cilium 1.15의 WireGuard + Encryption Context는 커널 5.14+를, Multi-Protocol Load Balancing은 5.10+를 필요로 한다. 프로덕션 Kubernetes 노드의 커널을 업그레이드하는 것은 계획 없이 진행하면 서비스 중단을 유발할 수 있다. 하지만 업그레이드를 미루면 보안 취약점(CVE)이 누적되고 새 기능을 사용할 수 없다.

### 분析

프로덕션 커널 업그레이드의 핵심 도전은 "노드를 재부팅하지 않고 커널을 바꿀 수 없다"는 사실이다. kpatch(RHEL) 같은 라이브 패치 도구가 있지만, 새 커널 버전으로의 업그레이드가 아닌 보안 패치에만 적합하다. Kubernetes에서 커널 업그레이드는 Drain → 재부팅(또는 노드 교체) → Uncordon 절차를 따른다.

롤링 노드 업그레이드 전략은 Kubernetes의 `maxUnavailable` 설정과 맞물린다. `PodDisruptionBudget`을 올바르게 설정하지 않으면 노드 Drain 중 서비스가 중단된다. 특히 StatefulSet Pod, 긴 커넥션을 유지하는 서비스, 기동 시간이 긴 애플리케이션이 있는 경우 Drain 타임아웃을 충분히 설정해야 한다.

클라우드 환경에서는 노드를 교체하는 방식이 더 안전하다. 새 커널 버전의 새 노드 그룹을 만들고, 기존 노드 그룹의 트래픽을 점진적으로 새 그룹으로 이동한 후 기존 그룹을 삭제한다. Terraform이나 Karpenter를 사용하면 이 과정을 자동화할 수 있다. 이 방법은 재부팅 없이 새 커널 버전을 적용하는 것과 같은 효과를 낸다.

커널 업그레이드 전 검증 단계가 필수다. 스테이징 환경에서 새 커널 버전으로 Cilium을 운영하며 `cilium connectivity test` 전체 통과를 확인하고, 주요 서비스의 E2E 테스트를 실행한다. 특히 eBPF 프로그램이 새 커널에서 Verifier를 통과하는지 확인해야 한다. 커널 버전에 따라 Verifier 동작이 달라지는 경우가 있기 때문이다.

### 실무 적용

연간 커널 업그레이드 계획을 수립한다. 연초에 해당 연도의 LTS 커널 버전(보통 Ubuntu 또는 RHEL의 지원 버전)을 목표로 정하고, 분기별로 스테이징에서 검증한다. 프로덕션 업그레이드는 트래픽이 낮은 시간대(새벽)에 진행하고, 노드 그룹 교체 방식을 사용한다. 업그레이드 후 48시간 동안 Cilium 관련 메트릭(드롭 수, 정책 위반, BPF Map 사용량)을 집중 모니터링한다.
