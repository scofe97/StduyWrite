# Ch05. Linkerd 아키텍처 — Deep Dive Questions

## Q1. Buoyant Enterprise for Linkerd(BEL) 라이선스 변경이 오픈소스 생태계에 미친 실질적 영향은 무엇인가?

### 왜 중요한가?

2023년 Buoyant가 Linkerd의 안정 릴리스(stable release)를 BEL 상용 라이선스로 전환한 결정은 CNCF 프로젝트 역사에서 드문 사례다. 이전까지 Linkerd는 Apache 2.0으로 완전 개방되어 있었고, 사용자는 비용 없이 프로덕션에서 사용할 수 있었다. 이 변경은 단순한 라이선스 문제가 아니라, 오픈소스 지속 가능성과 상업적 이해관계 사이의 긴장을 가장 잘 보여주는 사례 중 하나다.

실무자 입장에서 이 변경은 두 가지 현실적 질문을 낳는다. 첫째, edge 릴리스(비안정 버전)만 무료로 사용할 수 있다면 프로덕션 적용이 가능한가? 둘째, 대안으로 Istio나 Cilium으로 이동하는 비용이 BEL 라이선스 비용보다 낮은가?

### 분석

라이선스 구조를 정확히 이해하는 것이 출발점이다. Buoyant는 edge 채널(최신 커밋 기반)은 Apache 2.0으로 유지하고, stable 채널(장기 지원, 보안 패치 포함)만 BEL로 전환했다. 즉, 무료 사용자는 최신 버전을 빠르게 따라가야 하는 운영 부담을 안게 된다. 패치 주기를 따라가지 못하는 조직에서는 사실상 유료 전환을 강제받는 구조다.

CNCF 내부 반응도 주목할 만하다. CNCF는 졸업 프로젝트에 OSI 승인 라이선스를 요구하지만, Linkerd의 코어 코드베이스 자체는 여전히 Apache 2.0이기 때문에 기술적으로 CNCF 정책 위반이 아니다. 그러나 "안정 바이너리를 상업화"한다는 접근 방식은 커뮤니티의 신뢰 기반을 흔들었고, Linkerd 채택을 검토하던 조직들이 Istio로 방향을 선회하는 계기가 됐다. 실제로 2023~2024년 기간 동안 Istio의 커뮤니티 기여자 수와 다운로드 수가 눈에 띄게 증가했다.

한편 Buoyant의 논리도 무시하기 어렵다. Linkerd의 핵심 기여자가 대부분 Buoyant 직원이었고, 지속적인 개발을 위한 수익 모델 없이는 프로젝트 자체가 위험해진다는 주장은 현실적이다. HashiCorp의 BSL 전환, Elastic의 라이선스 변경과 같은 맥락에서 오픈소스 지속 가능성 문제는 업계 전반의 과제다.

### 실무 적용

신규 서비스 메시 도입을 검토할 때, 라이선스 조건을 계약 단계에서 확인하는 것이 필수다. BEL은 30노드 이하 클러스터에 대해 무료 티어를 제공하므로, 소규모 팀이라면 실질적 비용 없이 stable 채널을 사용할 수 있다. 대규모 프로덕션 환경에서는 Linkerd edge 채널 운영 오버헤드(잦은 업그레이드)와 BEL 비용, 그리고 Istio 전환 비용을 정량적으로 비교하는 분석이 선행돼야 한다.

## Q2. Linkerd가 Envoy 대신 Rust로 자체 프록시(linkerd2-proxy)를 구현한 결정은 어떤 트레이드오프를 수반하는가?

### 왜 중요한가?

서비스 메시의 데이터 플레인은 모든 서비스 간 트래픽이 통과하는 핵심 경로다. 프록시의 언어 선택은 성능, 메모리 안전성, 확장성, 운영 복잡도에 직접 영향을 미친다. Istio, Consul, Kuma는 모두 Envoy를 데이터 플레인으로 선택했지만, Linkerd는 Rust로 처음부터 linkerd2-proxy를 구현했다. 이 선택의 배경과 결과를 이해하면 기술 결정의 본질을 볼 수 있다.

### 분석

Envoy를 선택하지 않은 이유는 Envoy의 범용성 자체에 있다. Envoy는 API Gateway, 사이드카 프록시, 엣지 프록시 등 다양한 역할을 지원하도록 설계됐기 때문에, 그 설정 복잡도와 API 표면적이 크다. xDS API를 통한 동적 설정은 강력하지만, 컨트롤 플레인이 Envoy의 모든 기능을 올바르게 구성해야 한다는 부담이 따른다. Linkerd 팀은 "서비스 메시 사이드카"라는 단일 역할에 최적화된 프록시가 더 단순하고 안전하다는 판단을 내렸다.

Rust 선택의 핵심 이점은 메모리 안전성이다. C++로 작성된 Envoy는 역사적으로 메모리 관련 CVE가 다수 발생했다. Rust는 컴파일 타임에 메모리 오류(버퍼 오버플로, use-after-free)를 방지하므로, 모든 트래픽이 통과하는 프록시에서 공격 표면을 줄이는 효과가 있다. 또한 Tokio 비동기 런타임과 Tower 미들웨어 스택을 활용해 낮은 레이턴시와 높은 처리량을 달성했다. Linkerd의 벤치마크에서 linkerd2-proxy는 Envoy 대비 p99 레이턴시에서 일관되게 낮은 수치를 보였다.

그러나 트레이드오프도 분명하다. linkerd2-proxy는 Envoy보다 확장 포인트가 훨씬 적다. Envoy는 WebAssembly(Wasm) 필터, C++ 필터, Lua 스크립트를 통해 커스텀 로직을 주입할 수 있지만, linkerd2-proxy는 이런 확장 메커니즘이 없다. 따라서 특수한 인증 로직, 요청 변환, 커스텀 프로토콜 지원이 필요한 환경에서 Linkerd는 한계를 드러낸다. 또한 Rust 기반 프록시를 운영할 수 있는 엔지니어 풀이 C++ Envoy 전문가 풀보다 작다는 현실적 문제도 있다.

### 실무 적용

linkerd2-proxy의 한계는 "Linkerd가 할 수 없는 것"을 먼저 파악하는 것으로 이어진다. JWT 검증, 요청 본문 변환, gRPC 트랜스코딩 같은 기능이 필요하다면 Linkerd만으로는 부족하고 별도의 API Gateway가 필요하다. 반면 단순한 mTLS, 레이턴시 기반 로드 밸런싱, 기본 재시도 정책만 필요한 환경에서는 linkerd2-proxy의 낮은 오버헤드가 명확한 장점이 된다.

## Q3. Linkerd 컨트롤 플레인이 단일 장애점(SPOF)이 되지 않으려면 어떤 설계가 필요한가?

### 왜 중요한가?

서비스 메시의 역설은 안정성을 제공하는 컴포넌트 자체가 안정적이어야 한다는 점이다. 컨트롤 플레인이 다운되면 새로운 정책을 배포할 수 없고, 인증서 갱신이 중단되며, 새로운 사이드카 주입이 실패할 수 있다. 그러나 기존에 실행 중인 데이터 플레인은 마지막으로 받은 설정으로 계속 동작한다. 이 "컨트롤 플레인 없이도 동작"하는 특성의 경계를 정확히 이해하는 것이 운영의 핵심이다.

### 분석

Linkerd 컨트롤 플레인의 주요 컴포넌트는 destination(서비스 디스커버리, 정책), identity(인증서 발급), proxy-injector(사이드카 주입 웹훅), 그리고 선택적 viz 컴포넌트다. 이 중 destination과 identity는 데이터 플레인의 지속적 동작에 직접 영향을 미친다.

데이터 플레인 복원력 관점에서, linkerd2-proxy는 destination 서버와의 연결이 끊겨도 마지막으로 캐시한 엔드포인트 목록으로 트래픽을 계속 처리한다. 단, 새로운 서비스 엔드포인트 변경(Pod 추가/제거)은 반영되지 않으므로 스케일 아웃 중에 컨트롤 플레인이 다운되면 부하가 고르게 분산되지 않는 문제가 생긴다. 인증서 측면에서는 identity 서버가 다운되면 기존 인증서는 만료될 때까지 유효하지만, 갱신이 불가능하다. Linkerd의 기본 인증서 유효 기간이 24시간이므로, identity 서버 다운이 24시간을 넘으면 mTLS 연결이 실패하기 시작한다.

고가용성(HA) 구성은 Linkerd CLI와 Helm 모두 지원한다. `--ha` 플래그를 사용하면 컨트롤 플레인 컴포넌트가 3개 이상의 replica로 배포되고, PodDisruptionBudget이 적용되며, anti-affinity 규칙으로 여러 노드에 분산된다. 그러나 identity 컴포넌트는 신뢰 앵커(trust anchor)와 발급자 인증서를 마운트하는 시크릿에 의존하므로, 시크릿 자체의 가용성도 HA의 일부다. etcd 기반 Kubernetes API server가 다운되면 Linkerd HA 구성도 무의미해진다는 점에서, 컨트롤 플레인 HA는 Kubernetes 클러스터 HA의 전제 위에서만 의미를 가진다.

### 실무 적용

프로덕션 환경에서 Linkerd를 운영할 때 컨트롤 플레인 컴포넌트를 전용 노드 풀(node pool)에 배치하는 것을 검토해볼 만하다. 워크로드와 컨트롤 플레인이 같은 노드를 공유하면, 워크로드의 리소스 경합이 컨트롤 플레인 응답성을 떨어뜨릴 수 있다. 또한 identity 서버 가용성을 직접 모니터링하는 알림을 설정하고, 인증서 만료까지 남은 시간을 추적하는 것이 예상치 못한 mTLS 장애를 예방하는 현실적 방법이다.

## Q4. linkerd2-proxy의 Tokio/Tower 기반 비동기 아키텍처는 어떤 시나리오에서 한계를 드러내는가?

### 왜 중요한가?

linkerd2-proxy는 Rust의 비동기 에코시스템인 Tokio(런타임)와 Tower(미들웨어)를 기반으로 구축됐다. 이 아키텍처는 낮은 레이턴시와 높은 처리량을 동시에 달성하는 데 효과적이지만, 특정 시나리오에서는 기대와 다른 동작을 보일 수 있다. 어떤 상황에서 이 설계가 한계를 드러내는지 이해하는 것은 장애 대응에 직결된다.

### 분석

Tokio의 작업 스케줄링은 이벤트 루프 기반이기 때문에, CPU를 오래 점유하는 동기 작업(CPU-bound work)이 비동기 태스크 안에 섞이면 전체 이벤트 루프가 블로킹된다. linkerd2-proxy는 네트워크 I/O 집중 워크로드(HTTP/gRPC 프록시)에 최적화되어 있으므로, 일반적인 사용 패턴에서는 문제가 없다. 그러나 대용량 요청 본문을 버퍼링하거나 TLS 핸드셰이크가 집중되는 상황에서는 레이턴시 스파이크가 나타날 수 있다.

Tower의 미들웨어 스택은 타입 시스템을 활용한 정적 구성(static composition)을 사용한다. 이는 런타임 오버헤드를 줄이는 장점이 있지만, 동적으로 미들웨어를 추가하거나 제거하는 것이 불가능하다. Envoy의 동적 필터 체인과 비교하면, linkerd2-proxy는 배포 없이 동작을 변경할 수 있는 범위가 제한된다. 정책 변경은 컨트롤 플레인을 통해 가능하지만, 완전히 새로운 처리 로직을 주입하는 것은 프록시 자체를 교체해야 한다.

메모리 소비 패턴도 주목할 필요가 있다. linkerd2-proxy는 커넥션 풀을 유지하고 각 연결에 대한 상태를 메모리에 보관한다. 연결 수가 수천 개에 달하는 환경에서는 사이드카 프록시의 메모리 사용량이 예상을 초과할 수 있다. Linkerd의 기본 메모리 리소스 요청은 보수적으로 설정되어 있어, 실제 부하 테스트를 통한 튜닝이 필요하다.

### 실무 적용

linkerd2-proxy의 성능 한계를 파악하려면 실제 트래픽 패턴을 시뮬레이션하는 부하 테스트가 필수다. `linkerd diagnostics proxy-metrics` 명령으로 프록시 내부 메트릭을 확인하고, 특히 `tcp_open_connections`와 `request_duration_ms` 분포를 관찰한다. 연결 수가 급격히 증가하는 패턴이 있다면, 업스트림 서비스의 커넥션 풀 설정을 검토해 불필요한 연결 생성을 줄이는 것이 우선이다.

## Q5. Linkerd의 "zero config"철학은 복잡한 트래픽 정책이 필요한 환경에서 어떤 설계 긴장을 일으키는가?

### 왜 중요한가?

Linkerd의 핵심 가치 제안은 "설정 없이 설치 후 바로 mTLS와 관측성을 얻는다"는 단순성이다. 이 철학은 초기 도입 장벽을 낮추지만, 동시에 "충분히 복잡한 정책"이 필요해질 때 Linkerd가 선택지가 아닌 제약이 될 수 있음을 내포한다. 이 긴장을 인식하지 못한 채 Linkerd를 도입하면 나중에 메시 전환이라는 큰 비용을 치르게 된다.

### 분석

Linkerd가 기본적으로 제공하는 트래픽 정책은 재시도, 타임아웃, 회로 차단기(circuit breaker), 그리고 ServiceProfile 기반 라우팅이다. 이 범위 안에서는 설정이 간결하고 예측 가능하다. 그러나 조직의 요구사항이 헤더 기반 라우팅(header-based routing), 가중치 미러링(traffic shadowing), 요청 변환, JWT 클레임 기반 정책으로 확장될 때 Linkerd의 한계가 드러난다.

ServiceProfile은 Linkerd의 핵심 트래픽 설정 리소스지만, HTTPRoute(Gateway API)로 점진적으로 마이그레이션이 진행 중이다. 이 전환 과정은 현재 진행형이며, 두 API가 공존하는 기간 동안 문서와 예제가 혼재한다는 혼란이 있다. "zero config"를 표방하지만 실제 프로덕션에서는 ServiceProfile 또는 HTTPRoute를 세밀하게 구성해야 하는 상황이 자주 생긴다.

Istio와 비교하면 이 긴장은 더 명확하다. Istio의 VirtualService는 정규식 기반 경로 매칭, 헤더 조작, 결함 주입(fault injection), 미러링을 네이티브로 지원한다. Linkerd에서 동일한 기능을 구현하려면 별도의 API Gateway(예: Nginx Ingress, Traefik)를 도입하거나, 애플리케이션 레벨에서 처리해야 한다. "단순함을 위해 기능을 포기"한 설계 결정이 조직의 성숙도에 따라 장점에서 단점으로 바뀌는 지점이 여기다.

### 실무 적용

Linkerd 도입 결정 전에 "18개월 후 우리에게 필요한 트래픽 정책이 무엇인가?"를 먼저 정의해야 한다. 현재 필요하지 않더라도 카나리 배포, A/B 테스트, 지역 기반 라우팅이 로드맵에 있다면, Linkerd만으로는 부족할 가능성이 높다. 이 경우 Linkerd + API Gateway 조합, 또는 처음부터 Istio를 선택하는 것이 장기적으로 더 효율적일 수 있다. "단순함"은 유지 비용이지만, "기능 부족"은 마이그레이션 비용이다.

## Q6. Linkerd의 멀티클러스터 아키텍처에서 게이트웨이 기반 접근법의 보안 모델은 어떻게 작동하는가?

### 왜 중요한가?

멀티클러스터 환경에서 서비스 메시의 신뢰 경계는 단일 클러스터보다 복잡해진다. Linkerd의 멀티클러스터 구현은 각 클러스터에 게이트웨이를 두고 클러스터 간 트래픽을 이 게이트웨이를 통해 라우팅한다. 이 설계가 신뢰 경계를 어떻게 정의하고, 어떤 조건에서 mTLS가 단대단(end-to-end)으로 보장되는지 이해하는 것은 보안 아키텍처의 핵심이다.

### 분석

Linkerd 멀티클러스터의 핵심은 서비스 미러링(service mirroring)이다. 소스 클러스터의 service-mirror 컨트롤러가 대상 클러스터의 서비스를 감시하고, 로컬 클러스터에 미러 서비스를 생성한다. 애플리케이션은 이 미러 서비스로 요청을 보내고, 실제 트래픽은 대상 클러스터의 멀티클러스터 게이트웨이로 전달된다.

보안 측면에서 주목할 점은 게이트웨이 간 mTLS가 별도로 작동한다는 것이다. 각 클러스터는 독립적인 신뢰 앵커(trust anchor)를 가질 수 있으며, 멀티클러스터 링크를 구성할 때 대상 클러스터의 신뢰 앵커를 소스 클러스터에 등록한다. 이 과정에서 크레덴셜은 Kubernetes Secret으로 관리되므로, Secret의 접근 권한 제어가 곧 클러스터 간 신뢰 관리의 핵심이 된다.

게이트웨이 기반 접근법의 제약은 게이트웨이가 트래픽의 복호화 지점이 된다는 것이다. 즉, 소스 Pod → 소스 게이트웨이 구간은 Linkerd mTLS로 보호되지만, 게이트웨이는 트래픽을 복호화하고 대상 클러스터 게이트웨이와 새로운 mTLS 연결을 맺는다. 엄밀한 의미에서 단대단 암호화는 아니며, 게이트웨이 자체가 신뢰 경계 내에 있어야 한다는 전제가 있다.

### 실무 적용

멀티클러스터 게이트웨이를 전용 노드에 배치하고, 네트워크 정책으로 게이트웨이 포트(기본 4143)에 대한 접근을 제한하는 것이 기본 강화 방법이다. 클러스터 간 크레덴셜(링크 시크릿)은 별도의 시크릿 관리 솔루션(Vault, External Secrets Operator)으로 관리하는 것이 하드코딩보다 안전하다. 또한 각 클러스터의 신뢰 앵커 로테이션 일정을 통일하면 클러스터 간 인증 실패를 예방할 수 있다.

## Q7. Linkerd가 HTTP/2 멀티플렉싱을 기본 적용하는 방식은 기존 HTTP/1.1 서비스와 어떤 호환성 문제를 일으킬 수 있는가?

### 왜 중요한가?

Linkerd는 Pod 간 통신에 자동으로 HTTP/2 프로토콜 업그레이드를 적용한다. 이는 멀티플렉싱을 통한 연결 효율성 향상을 제공하지만, 원래 HTTP/1.1로 통신하도록 설계된 서비스와의 상호작용에서 예상치 못한 동작을 일으킬 수 있다. 특히 커넥션 기반 상태를 유지하는 레거시 서비스가 있는 환경에서는 이 동작을 정확히 이해해야 한다.

### 분석

Linkerd의 프로토콜 감지(protocol detection)는 첫 번째 바이트를 분석해 HTTP/1.1인지 HTTP/2인지 판단한다. HTTP로 감지되면 linkerd2-proxy는 자동으로 HTTP/2 업그레이드를 시도한다. 이 과정은 애플리케이션에 투명하게 이루어지지만, 일부 시나리오에서는 문제를 일으킨다.

대표적인 문제는 "프로토콜 감지 지연(protocol detection timeout)"이다. 클라이언트가 첫 번째 바이트를 즉시 보내지 않는 프로토콜(서버가 먼저 데이터를 보내는 MySQL, SMTP 등)에서 linkerd2-proxy는 기본 10초 동안 클라이언트 바이트를 기다리다가 타임아웃 처리한다. 이런 서비스는 `opaque ports` 설정으로 프로토콜 감지를 건너뛰어야 한다.

HTTP/1.1의 keep-alive 연결을 사용하는 서비스도 주의가 필요하다. Linkerd는 클라이언트와 프록시 사이, 프록시와 서버 사이의 연결을 각각 독립적으로 관리하므로, 클라이언트가 특정 서버 연결에 "고정"되는 동작(세션 기반 서버 선택)이 의도대로 작동하지 않을 수 있다. 이는 세션 어피니티(session affinity)가 필요한 레거시 애플리케이션에서 특히 문제가 된다.

### 실무 적용

Linkerd 주입 전에 각 서비스의 프로토콜과 연결 패턴을 명세화해야 한다. MySQL, Redis, MongoDB 등 바이너리 프로토콜 서비스는 반드시 `config.linkerd.io/opaque-ports` 어노테이션으로 지정한다. 세션 어피니티가 필요한 서비스는 Kubernetes Service의 `sessionAffinity: ClientIP` 설정이 Linkerd 환경에서 기대대로 동작하는지 검증이 필요하다. 이 경우 애플리케이션 레벨의 세션 관리로 전환하는 것이 더 명확한 해결책이다.
