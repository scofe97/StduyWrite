# Ch08. Linkerd 보안과 정책 — Deep Dive Questions

## Q1. 기존 허용 중심 네트워크에서 default-deny 정책으로 마이그레이션할 때 어떤 장애 시나리오가 발생하며, 이를 안전하게 전환하는 순서는 무엇인가?

### 왜 중요한가?

Linkerd의 Server와 AuthorizationPolicy 리소스를 사용해 default-deny 모델을 구현하면 명시적으로 허용된 트래픽만 통과한다. 그러나 기존 시스템은 묵시적 허용(모든 내부 트래픽 허용)을 전제로 설계된 경우가 많고, 실제로 어떤 서비스가 어떤 서비스와 통신하는지 완벽하게 문서화되어 있지 않다. 이 상태에서 갑자기 default-deny를 활성화하면 예상치 못한 서비스가 차단되어 장애가 발생한다.

### 분석

안전한 마이그레이션의 핵심 원리는 "차단 전에 관찰"이다. Linkerd의 Server 리소스에는 `accessPolicy` 필드가 있으며, 기본값은 `deny`지만 `audit` 모드를 사용하면 정책을 평가하되 실제로 차단하지 않고 로그와 메트릭만 기록한다. 이 감사 모드를 통해 어떤 트래픽이 허용 정책 없이 통과하려 하는지 사전에 파악할 수 있다.

마이그레이션 단계는 다음과 같다. 1단계는 현재 통신 패턴 매핑이다. `linkerd viz tap` 명령과 Linkerd 메트릭의 소스/목적지 레이블을 분석해 서비스 간 의존 관계를 그래프로 작성한다. 2단계는 Server 리소스를 audit 모드로 배포하고, 정책 위반 메트릭(`policy.linkerd.io/auth_denied_total`)을 모니터링한다. 3단계는 관찰된 트래픽 패턴을 기반으로 AuthorizationPolicy를 작성한다. 4단계는 Server를 deny 모드로 전환하고, 차단된 트래픽이 없는지 확인한다.

가장 흔한 장애 시나리오는 헬스 체크 프로브다. Kubernetes의 liveness/readiness probe는 kubelet에서 직접 Pod에 접근하므로, 이 트래픽도 정책의 적용을 받는다. kubelet의 IP 주소는 노드 IP이고, Linkerd의 인증서 기반 정책은 mTLS 클라이언트 인증을 사용하므로, kubelet 헬스 체크는 mTLS를 수행하지 않아 default-deny에서 차단된다. 이를 허용하려면 unauthenticated 출처에서의 헬스 체크 포트 접근을 별도로 허용하는 정책이 필요하다.

다른 흔한 실수는 Prometheus 스크레이핑이다. Linkerd viz의 Prometheus가 각 프록시의 메트릭 엔드포인트(포트 4191)를 스크레이핑할 때도 정책의 적용을 받는다. Prometheus의 서비스 어카운트를 메트릭 포트에 대해 명시적으로 허용해야 한다.

### 실무 적용

default-deny 마이그레이션은 서비스 단위로 진행하는 것이 안전하다. 중요도가 낮은 서비스부터 시작해 정책 작성 프로세스를 검증하고, 경험이 쌓이면 핵심 서비스로 확장한다. 각 서비스에 대한 AuthorizationPolicy는 해당 서비스 팀이 소유하고 관리하도록 책임을 분산하면 전체 정책 부하를 줄일 수 있다. 정책 검증은 배포 파이프라인의 일부로 자동화해 새 서비스가 추가될 때 통신 패턴이 정책에 반영됐는지 확인한다.

## Q2. Post-quantum 암호화를 Linkerd mTLS에 적용하면 실질적인 성능 오버헤드는 얼마이며, 현재 시점에서 적용이 필요한가?

### 왜 중요한가?

양자 컴퓨터가 현재의 RSA와 타원곡선(ECC) 암호화를 파괴할 수 있다는 위협 아래, NIST가 2024년에 포스트 퀀텀 암호화 표준을 확정했다. 서비스 메시의 mTLS는 서비스 간 모든 통신을 보호하므로, 이 암호화 전환이 서비스 메시 레이어에서 어떤 의미를 갖는지 이해할 필요가 있다. 그러나 "지금 당장 필요한가"라는 질문에는 신중한 분석이 필요하다.

### 분석

Linkerd의 mTLS는 현재 TLS 1.3을 기반으로 하며, 키 교환에 X25519(ECDH)와 인증에 Ed25519를 사용한다. 이 알고리즘들은 양자 컴퓨터에 취약하다. 포스트 퀀텀 대안으로는 NIST가 표준화한 ML-KEM(Kyber의 후속), ML-DSA(Dilithium의 후속), SLH-DSA(SPHINCS+의 후속)가 있다.

성능 오버헤드는 알고리즘에 따라 다르다. ML-KEM은 키 캡슐화 크기가 커서(~1KB vs ECDH의 ~32 bytes) 핸드셰이크 시 추가 바이트 전송이 필요하다. 처리 속도 면에서 ML-KEM은 X25519와 비슷하거나 오히려 빠른 경우도 있다. 반면 ML-DSA는 서명 크기가 상당히 크고(~2.4KB) 검증 속도도 ECDSA보다 느리다. 서비스 메시 환경에서 가장 영향을 받는 부분은 TLS 핸드셰이크 시 인증서 크기 증가로 인한 네트워크 지연이다.

실질적인 성능 영향을 평가하려면 서비스 메시의 연결 패턴을 이해해야 한다. 장기 연결(long-lived connection)을 사용하는 gRPC 서비스는 핸드셰이크가 연결 수립 시 한 번만 발생하므로 오버헤드가 희석된다. 반면 짧은 HTTP 요청이 많은 REST API는 연결당 핸드셰이크 비율이 높아 오버헤드가 더 크게 느껴질 수 있다.

현재 시점(2025년 기준)에서 포스트 퀀텀 전환의 긴급성은 "Harvest Now, Decrypt Later" 위협 모델에 달려 있다. 공격자가 현재 암호화된 트래픽을 저장해뒀다가 미래의 양자 컴퓨터로 복호화하는 시나리오가 현실적인 위협이라면, 특히 기밀성이 중요한 트래픽에 대해 포스트 퀀텀 전환을 검토해야 한다. 그러나 linkerd2-proxy는 현재 포스트 퀀텀 암호화를 기본적으로 지원하지 않으므로, 이 기능이 필요하다면 Linkerd 버전 로드맵을 추적하거나 커스텀 빌드를 검토해야 한다.

### 실무 적용

대부분의 조직에서 현재 시점의 우선순위는 포스트 퀀텀 전환보다 기본 mTLS의 올바른 구성과 신뢰 앵커 관리다. 포스트 퀀텀 전환은 TLS 라이브러리(rustls in linkerd2-proxy)가 이 알고리즘을 지원하는 시점에 자연스럽게 따라가는 것이 현실적이다. 지금 할 수 있는 준비는 인증서 유효 기간을 짧게 유지하고(자동 갱신), 암호화 알고리즘을 한 곳에서 관리하는 구조를 만드는 것이다.

## Q3. 클러스터 외부 워크로드(external workload)를 Linkerd 메시에 통합할 때 신뢰 경계는 어떻게 설정해야 하는가?

### 왜 중요한가?

Linkerd 2.14부터 클러스터 외부의 VM이나 베어메탈 서비스를 메시에 포함시키는 ExternalWorkload 리소스가 추가됐다. 이 기능은 하이브리드 환경(클러스터 내 서비스 + 레거시 VM 서비스)에서 일관된 mTLS와 정책을 적용할 수 있게 한다. 그러나 외부 워크로드의 인증서 발급과 신뢰 경계 설정은 클러스터 내부 서비스와 다른 도전을 가져온다.

### 분석

클러스터 내부 워크로드는 Linkerd identity 서버가 Kubernetes 서비스 어카운트 토큰을 검증한 후 인증서를 발급한다. 이 메커니즘은 Kubernetes API server의 신뢰를 기반으로 하므로, 클러스터 내에서는 자동으로 안전하다. 외부 워크로드는 이 메커니즘을 사용할 수 없다. 대신 외부 워크로드의 linkerd2-proxy가 클러스터 identity 서버에 직접 접근해 인증서를 요청해야 한다.

이 접근에서 신뢰 경계의 핵심 질문은 "외부 워크로드가 어떻게 자신의 신원을 증명하는가"다. 현재 Linkerd의 ExternalWorkload 구현에서 외부 워크로드는 Kubernetes ServiceAccount 토큰을 직접 사용하거나, 별도의 부트스트랩 인증 메커니즘을 필요로 한다. 이 초기 신원 증명 단계가 취약하면 악의적인 외부 시스템이 메시에 합류할 수 있다.

네트워크 측면에서 외부 워크로드가 Linkerd identity 서버에 접근하려면 클러스터 내부 서비스에 접근할 수 있는 네트워크 경로가 필요하다. 이는 VPN, AWS PrivateLink, Google Cloud Private Service Connect 같은 프라이빗 연결을 통해 구성하는 것이 일반적이다. 공용 인터넷을 통해 identity 서버에 접근하는 구성은 신뢰 앵커와 identity 서버 자체가 공개 노출된다는 의미이므로 권장되지 않는다.

외부 워크로드의 인증 정책은 내부 워크로드와 동일한 AuthorizationPolicy로 적용된다. 단, 외부 워크로드의 서비스 어카운트 ID(SPIFFE URI)가 정책에 올바르게 반영돼야 한다. `policy.linkerd.io/authenticated-identity` 조건에서 외부 워크로드의 ID를 정확히 지정하지 않으면 접근 제어가 기대대로 동작하지 않는다.

### 실무 적용

외부 워크로드 통합은 소수의 서비스부터 파일럿으로 시작해 인증서 발급, 갱신, 정책 적용이 올바르게 작동하는지 충분히 검증한 후 확장하는 것이 안전하다. 외부 워크로드의 네트워크 경로를 전용 서브넷으로 분리하고, 방화벽 규칙으로 identity 서버 포트(8080)에 대한 접근을 외부 워크로드 IP 범위로만 제한한다. 외부 워크로드의 인증서 상태를 클러스터 내부 프록시와 동일한 모니터링 체계에 포함시켜 만료 전 알림을 받도록 설정한다.

## Q4. Linkerd 정책 디버깅 도구가 제한적인 이유는 무엇이며, 실무에서 어떤 방법으로 정책 동작을 검증하는가?

### 왜 중요한가?

AuthorizationPolicy가 의도대로 동작하는지 확인하는 것은 보안 운영의 핵심이다. 그러나 정책 구성의 복잡성과 Linkerd의 디버깅 도구 성숙도 사이의 간격이 있어, 정책이 예상대로 적용되고 있는지 확인하는 것이 생각보다 어렵다. 이 간격을 이해하고 실무적 검증 방법을 갖추는 것이 보안 운영 역량의 일부다.

### 분석

Linkerd의 정책 가시성 도구는 주로 두 가지다. 첫째는 `linkerd policy` CLI 서브커맨드로, 특정 서비스에 적용된 Server와 AuthorizationPolicy를 조회할 수 있다. 그러나 이 도구는 "현재 적용된 정책이 무엇인가"를 보여줄 뿐, "왜 특정 요청이 허용/차단됐는가"를 직접 설명하지 않는다.

둘째는 Linkerd 프록시 메트릭의 `policy.linkerd.io/auth_denied_total`과 `policy.linkerd.io/auth_allowed_total` 카운터다. 이 메트릭으로 차단 발생 여부를 감지할 수 있지만, 어떤 클라이언트가 어떤 이유로 차단됐는지의 세부 정보는 프록시 로그에서만 확인할 수 있다.

프록시 로그 레벨을 높이는 방법이 있다. `linkerd inject` 또는 `config.linkerd.io/proxy-log-level: debug` 어노테이션으로 특정 Pod의 프록시 로그 레벨을 높이면 정책 평가 로그를 볼 수 있다. 그러나 debug 로그는 지나치게 상세해 신호 대 노이즈 비율이 낮고, 프로덕션에서 장시간 활성화하면 로그 저장 비용이 급증한다.

실제 정책 검증에서 가장 신뢰할 수 있는 방법은 테스트 트래픽을 생성하고 결과를 관찰하는 것이다. `kubectl run` 으로 테스트 Pod를 생성하고, curl 또는 grpc-client로 대상 서비스에 요청을 보내 허용/차단 여부를 확인한다. 허용돼야 하는 요청이 차단됐다면 `linkerd viz tap` 으로 실시간 트래픽을 보면서 어느 계층에서 거부가 발생하는지 추적한다.

Kubernetes audit log도 유용한 보조 수단이다. AuthorizationPolicy 리소스의 생성/변경 이벤트를 audit log에서 추적하면 정책 변경 이력을 관리할 수 있다. 이는 "언제 누가 어떤 정책을 변경했는가"라는 컴플라이언스 요구사항에 대응하는 방법이다.

### 실무 적용

정책 변경은 항상 스테이징 환경에서 먼저 검증하고, 검증 결과를 PR에 포함시키는 프로세스를 수립한다. 자동화된 정책 테스트 스위트를 구축하는 것이 이상적이며, 허용돼야 하는 연결(positive test)과 차단돼야 하는 연결(negative test)을 모두 포함한다. 이 테스트는 CI/CD 파이프라인의 일부로 실행해 정책 변경이 기존 동작을 깨뜨리지 않는지 자동으로 검증한다.

## Q5. Zero trust 아키텍처와 네트워크 세분화(network segmentation)는 상호 보완적인가, 아니면 하나가 다른 하나를 대체할 수 있는가?

### 왜 중요한가?

"Zero trust를 구현하면 방화벽과 네트워크 세분화가 필요 없다"는 주장이 있다. Linkerd의 mTLS와 AuthorizationPolicy가 신원 기반 접근 제어를 제공하므로, IP 기반 네트워크 정책은 불필요하다는 논리다. 그러나 이 주장은 위협 모델을 단순화한 것이다. 두 접근법의 역할과 한계를 명확히 이해하면 더 견고한 보안 설계가 가능하다.

### 분석

네트워크 세분화의 핵심 가치는 "깊이 있는 방어(defense in depth)"다. Kubernetes NetworkPolicy 또는 eBPF 기반 네트워크 정책은 IP/포트 레벨에서 트래픽을 제어한다. 이 레이어는 Linkerd의 인증서 인프라가 손상된 경우에도 추가적인 보호를 제공한다. 예를 들어 공격자가 특정 서비스 어카운트의 인증서를 탈취했다면, Linkerd의 AuthorizationPolicy는 이를 합법적 클라이언트로 허용할 것이다. 그러나 이 클라이언트가 원래 허용된 네트워크 경로 밖에 있다면 NetworkPolicy가 연결 자체를 차단한다.

반면 Linkerd의 신원 기반 정책은 네트워크 세분화가 제공하지 못하는 것을 제공한다. IP 주소는 동적으로 변하고 NAT 환경에서 신뢰하기 어렵다. Kubernetes에서 Pod IP는 재시작마다 변경되므로 IP 기반 허용 목록은 유지 관리가 어렵다. 반면 Linkerd의 SPIFFE 기반 신원은 서비스 어카운트에 연결되어 있어 Pod가 재시작되어도 변하지 않는다.

실제 위협 시나리오에서 두 레이어의 가치를 보면 더 명확하다. 네트워크 세분화만 있는 경우: 내부 Pod가 침해되면 같은 네트워크 세그먼트의 서비스에 임의로 접근할 수 있다. Linkerd 정책만 있는 경우: 인증서 탈취나 identity 서버 손상 시 정책이 무의미해진다. 두 레이어가 모두 있는 경우: 하나가 손상되더라도 다른 레이어가 추가 보호를 제공한다.

현실적인 조언은 "Linkerd가 NetworkPolicy를 대체한다"가 아니라 "Linkerd와 NetworkPolicy는 서로 다른 위협 벡터를 막는다"는 것이다. Linkerd는 "A 서비스가 B 서비스에 접근해야 하는가"라는 애플리케이션 레벨 정책을, NetworkPolicy는 "A 네임스페이스의 IP 범위가 B 네임스페이스에 접근해야 하는가"라는 네트워크 레벨 정책을 담당한다.

### 실무 적용

최소한의 NetworkPolicy(네임스페이스 격리)와 Linkerd AuthorizationPolicy(서비스 간 최소 권한)를 함께 적용하는 계층화된 접근이 실무 권장 사항이다. NetworkPolicy로 네임스페이스 간 기본 격리를 설정하고, Linkerd 정책으로 허용된 네임스페이스 내에서도 서비스 수준의 세밀한 접근 제어를 추가한다. 두 정책 레이어를 별도의 팀(네트워크 팀 vs 플랫폼 팀)이 관리하면 책임이 명확해지고, 한 레이어의 실수가 다른 레이어에서 발견될 수 있다.

## Q6. Linkerd의 mTLS 신원(SPIFFE/SVID)이 서비스 어카운트 기반인 경우, 같은 서비스 어카운트를 공유하는 여러 서비스가 가진 보안 함의는 무엇인가?

### 왜 중요한가?

Linkerd는 SPIFFE(Secure Production Identity Framework For Everyone) 표준에 따라 각 워크로드에 신원을 부여한다. 이 신원은 Kubernetes 서비스 어카운트를 기반으로 하며, 형식은 `system:serviceaccount:<namespace>:<service-account-name>`이다. 여러 Deployment가 동일한 서비스 어카운트를 사용한다면, 이 모든 Deployment는 Linkerd 관점에서 동일한 신원을 갖는다. 이 사실이 정책 설계에 어떤 영향을 미치는지 이해해야 한다.

### 분석

Kubernetes의 기본 동작은 서비스 어카운트를 명시적으로 지정하지 않으면 `default` 서비스 어카운트를 사용한다. 한 네임스페이스에 10개의 Deployment가 있고 모두 `default` 서비스 어카운트를 사용한다면, Linkerd AuthorizationPolicy에서 이 10개 서비스를 개별적으로 구분할 수 없다. 서비스 A를 허용하는 정책은 동일한 서비스 어카운트를 사용하는 서비스 B도 허용한다.

최소 권한 원칙을 제대로 구현하려면 각 서비스가 전용 서비스 어카운트를 사용해야 한다. 이는 추가적인 운영 부담(서비스 어카운트 생성 및 관리)이지만, 정책의 세밀도를 서비스 수준으로 유지하는 유일한 방법이다. 새 서비스를 배포할 때 전용 서비스 어카운트 생성을 체크리스트에 포함시키거나, Kyverno 정책으로 `default` 서비스 어카운트 사용을 금지하는 규칙을 적용하는 것이 좋다.

또 다른 함의는 서비스 어카운트 토큰의 범위다. Kubernetes 1.24부터 서비스 어카운트 토큰은 기본적으로 자동 마운트되지 않는다(automountServiceAccountToken: false가 기본). Linkerd identity 서버가 서비스 어카운트 토큰을 검증해 인증서를 발급하므로, 토큰이 마운트되어 있어야 한다. 보안을 위해 토큰 자동 마운트를 비활성화한 클러스터에서 Linkerd를 사용한다면 이 설정과 충돌이 생길 수 있다.

SPIFFE ID의 범위가 서비스 어카운트 수준이라는 사실은 멀티 테넌트 클러스터에서 특히 중요하다. 다른 팀의 서비스가 동일한 네임스페이스에 배포되고 동일한 서비스 어카운트를 공유한다면, Linkerd 정책은 팀 간 격리를 보장하지 못한다. 이 경우 네임스페이스 단위 격리와 각 팀별 전용 서비스 어카운트의 조합이 필요하다.

### 실무 적용

신규 서비스 배포 표준에 "서비스당 전용 서비스 어카운트"를 포함시키고, Helm 차트 또는 Kustomize 기반 배포 템플릿에 서비스 어카운트 생성을 기본으로 포함한다. 기존 서비스가 `default` 서비스 어카운트를 공유하는 경우, 전용 서비스 어카운트로 마이그레이션하고 관련 AuthorizationPolicy를 업데이트하는 작업을 기술 부채로 등록해 체계적으로 처리한다. 정기적으로 `kubectl get sa -A`와 서비스별 서비스 어카운트 매핑을 감사해 공유 서비스 어카운트 사용 현황을 파악한다.

## Q7. Linkerd의 보안 정책이 멀티 테넌트 클러스터에서 충분한 격리를 보장할 수 있는가?

### 왜 중요한가?

여러 팀 또는 고객의 워크로드가 하나의 Kubernetes 클러스터를 공유하는 멀티 테넌트 환경에서, 테넌트 간 격리는 보안의 핵심 요구사항이다. Linkerd의 AuthorizationPolicy가 이 격리를 제공할 수 있는지, 아니면 추가적인 격리 메커니즘이 필요한지 명확히 판단해야 한다.

### 분석

Linkerd의 격리 메커니즘은 기본적으로 네임스페이스 수준이다. Server와 AuthorizationPolicy는 네임스페이스 범위 리소스이므로, 각 네임스페이스(테넌트)는 자신만의 정책을 가진다. 그러나 Linkerd 컨트롤 플레인은 클러스터 전체를 관리하므로, 컨트롤 플레인 자체는 공유된다. 컨트롤 플레인이 손상되면 모든 테넌트의 인증서와 정책에 영향을 미친다.

신뢰 앵커도 클러스터 전체가 공유하는 단일 루트다. 즉, 한 테넌트의 서비스가 발급받은 인증서는 다른 테넌트 서비스의 인증서와 동일한 신뢰 앵커로 서명된다. 이론적으로 테넌트 A가 테넌트 B의 서비스에 mTLS 연결을 시도할 수 있고(두 인증서 모두 같은 신뢰 앵커에서 신뢰받으므로), 차단은 AuthorizationPolicy에서만 이루어진다. AuthorizationPolicy가 잘못 설정되거나 누락된 경우 격리가 깨진다.

강한 테넌트 격리가 필요하다면 물리적 클러스터 분리 또는 가상 클러스터(vcluster)가 더 적절하다. 각 테넌트가 별도의 Linkerd 컨트롤 플레인과 신뢰 앵커를 가지면 격리가 훨씬 강해진다. 그러나 이는 운영 복잡도와 비용의 증가를 의미한다.

현실적인 멀티 테넌트 Linkerd 운영에서는 테넌트별 네임스페이스 + NetworkPolicy(IP 레벨 격리) + AuthorizationPolicy(신원 기반 격리) + RBAC(정책 변경 권한 제어)의 조합이 허용 가능한 보안 수준을 제공한다. 이 조합은 실수나 설정 누락 시 여러 레이어에서 보호한다.

### 실무 적용

클러스터를 공유하는 테넌트의 신뢰 수준(동일 조직 내 팀 vs 외부 고객)에 따라 격리 수준을 결정해야 한다. 동일 조직 내 팀이라면 논리적 격리(네임스페이스 + 정책)로 충분할 수 있지만, 외부 고객 데이터를 처리하는 멀티 테넌트 SaaS라면 물리적 격리나 강한 하드 멀티 테넌시 메커니즘이 필요하다. 보안 요구사항을 먼저 명확히 한 후, Linkerd가 그 요구사항을 충족하는지 판단하는 순서가 맞다.
