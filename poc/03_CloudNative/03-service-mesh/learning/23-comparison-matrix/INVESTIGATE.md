# Ch23. Linkerd vs Istio vs Cilium 비교 — Deep Dive Questions

## Q1. 벤치마크 수치는 얼마나 신뢰할 수 있는가 — 합성 테스트와 실제 워크로드의 차이

### 왜 중요한가?

서비스 메시를 선택할 때 흔히 벤치마크 보고서를 참고한다. "Linkerd가 Istio보다 레이턴시 p99에서 X% 낮다"는 식의 수치가 의사결정에 영향을 준다. 하지만 이 수치를 그대로 믿는 것은 위험하다. 벤치마크는 누가, 어떤 환경에서, 어떤 설정으로 측정하느냐에 따라 결과가 크게 달라진다. 특히 서비스 메시 벤치마크는 측정 도구 자체도 메시의 영향을 받기 때문에, 테스트 설계가 잘못되면 메시가 아닌 다른 요소를 측정하는 함정에 빠진다.

### 분석

합성 벤치마크(Synthetic Benchmark)는 통제된 환경에서 재현 가능한 결과를 만들지만, 현실을 단순화한다. `wrk`나 `fortio`로 고정 RPS를 발생시키는 테스트는 실제 서비스의 버스티(Bursty) 트래픽 패턴을 반영하지 못한다. 실제 서비스는 아침 출근 시간, 점심, 이벤트 등 예측 가능한 피크와 예측 불가능한 스파이크가 혼재한다. 이 패턴이 레이턴시 꼬리(tail latency)에 미치는 영향은 고정 RPS 테스트에서 나타나지 않는다.

측정 방법론에도 함정이 있다. Coordinated Omission 문제는 부하 생성기가 응답을 기다리는 동안 다음 요청을 보내지 않아 실제 레이턴시가 과소평가되는 현상이다. `hdrhistogram`을 사용하는 부하 생성기(예: `wrk2`)만이 이 문제를 올바르게 처리한다. Istio와 Linkerd 벤치마크 중 상당수가 이 문제를 고려하지 않아 결과가 왜곡된다.

Cilium의 eBPF 기반 데이터 플레인은 커널 레이어에서 동작하므로, 사이드카 모델(Linkerd, Istio)과 측정 기준이 다르다. "오버헤드"를 측정할 때 사이드카 모델은 Pod 간 추가 홉이 생기지만, eBPF 모델은 XDP(eXpress Data Path) 처리 시간을 오버헤드로 본다. 이 둘을 같은 척도로 비교하면 의미 없는 숫자가 나온다.

벤치마크를 신뢰하려면 다음을 확인해야 한다. (1) 누가 측정했는가(벤더 vs 중립적 제3자), (2) 실제 워크로드와 유사한 패턴인가, (3) Coordinated Omission을 처리했는가, (4) 측정 환경의 노이즈(동일 노드 다른 Pod)를 통제했는가.

### 실무 적용

직접 측정이 최선이다. 현재 서비스의 프로덕션 트래픽 패턴(RPS 분포, 페이로드 크기, 연결 수명 등)을 파악하고, 스테이징 환경에서 이 패턴을 재현한 후 각 메시를 설치해 비교한다. 최소 72시간 이상 측정하고, 피크 타임과 비피크 타임을 구분해 분석한다. 이렇게 해도 프로덕션과 완전히 같지는 않지만, 합성 벤치마크보다 훨씬 신뢰할 수 있는 데이터가 된다.

## Q2. Buoyant Enterprise Linkerd(BEL) 라이선스 변경이 Linkerd 채택에 실제로 어떤 영향을 미쳤는가?

### 왜 중요한가?

2023년 Buoyant가 Linkerd 2.13 이후 버전의 안정적 릴리스에 상용 라이선스(BEL)를 적용했다. 이 결정은 CNCF 프로젝트 중에서도 이례적인 것이었고, 오픈소스 커뮤니티에서 큰 논란을 일으켰다. 무료로 사용 가능한 것은 엣지(Edge) 릴리스뿐이며, 엣지 릴리스는 프로덕션 안정성 보장이 없다. 이 변화가 도입을 고려하는 조직에게 무엇을 의미하는지 이해해야 한다.

### 분석

Linkerd의 라이선스 변화를 "오픈소스 배신"으로 보는 시각과 "지속 가능한 비즈니스 모델"로 보는 시각이 공존한다. 전자의 입장은 CNCF 졸업 프로젝트가 상용 라이선스를 요구하는 것은 생태계 신뢰를 훼손한다는 것이다. 후자는 Envoy와 달리 Linkerd의 데이터 플레인(Linkerd2-proxy)이 Rust로 독자 개발되어 유지 비용이 크고, 상용 지원 없이는 프로젝트 지속이 어렵다는 현실을 지적한다.

실제 채택 영향을 보면, 대기업이 신규 Linkerd 도입을 주저하는 경향이 생겼다. 라이선스 비용 자체보다는 "언제 또 변경될지 모른다"는 불확실성이 더 큰 장벽이다. 반면 이미 Linkerd를 사용 중인 소규모 팀은 엣지 릴리스를 계속 사용하거나 BEL 계약을 맺는 방향으로 갈렸다.

이 사태가 Istio와 Cilium에 반사 이익을 주었다는 분석도 있다. Istio는 Google, Red Hat, IBM이 뒤에 있고 Apache 2.0 라이선스를 유지한다. Cilium은 Isovalent(CNCF eBPF 기반)가 상용 버전을 제공하지만 코어 Cilium은 Apache 2.0이다. 라이선스 리스크를 중요하게 보는 조직이라면 이 차이가 의사결정에 영향을 준다.

오픈소스 지속 가능성 문제는 Linkerd만의 이슈가 아니다. HashiCorp의 Terraform BSL 전환, Redis의 라이선스 변경도 같은 맥락이다. 인프라 도구를 선택할 때 단순 기술 비교뿐 아니라 프로젝트의 거버넌스 구조와 재정 모델을 함께 평가하는 것이 필요하다.

### 실무 적용

신규 서비스 메시 도입 시 "이 프로젝트가 5년 후에도 독립적으로 존재할 것인가"를 평가 기준에 추가한다. CNCF 졸업 상태, 컨트리뷰터 다양성(단일 기업 의존도), 라이선스 이력, 재정 모델을 체크리스트로 만들어 평가한다. 이미 Linkerd를 사용 중이라면 BEL 비용 대비 Istio 마이그레이션 비용을 비교해 전략을 결정한다.

## Q3. eBPF 기반 메시가 요구하는 커널 버전이 엔터프라이즈 환경에서 실제로 진입 장벽이 되는가?

### 왜 중요한가?

Cilium Service Mesh는 eBPF를 사용하므로 Linux 커널 4.9 이상을 요구하고, 고급 기능(WireGuard 암호화, BPF-based load balancing)은 5.10+ 이상을 권장한다. 클라우드 네이티브 팀 관점에서는 당연한 요구사항처럼 보이지만, 엔터프라이즈 환경에서는 다르다. RHEL 7은 커널 3.10 기반이다. 5년 이상 된 온프레미스 서버, PCI-DSS 준수 환경, 커스텀 커널 패치를 사용하는 환경에서 커널 버전 요구사항은 현실적인 장벽이다.

### 분석

클라우드 환경에서는 이 문제가 덜하다. EKS는 Amazon Linux 2 기반으로 커널 5.4+ 제공, GKE는 Container-Optimized OS로 최신 커널을 유지, AKS도 Ubuntu 기반으로 5.15+ 제공한다. 관리형 쿠버네티스를 사용하는 팀이라면 커널 버전이 크게 문제가 되지 않는다.

문제는 하이브리드 환경과 온프레미스다. RHEL 8/9로 업그레이드하면 커널 4.18/5.14를 쓸 수 있지만, 엔터프라이즈 RHEL 업그레이드는 수개월의 인증 및 검증 과정을 거친다. 보안 규정상 커널 버전 업그레이드가 변경 관리 위원회(CAB) 승인을 요구하는 조직도 있다. 이런 환경에서 Cilium 도입을 결정하면 커널 업그레이드 프로젝트를 선행해야 한다.

커널 버전 외에도 eBPF는 `CONFIG_BPF_SYSCALL`, `CONFIG_BPF_JIT`, `CONFIG_CGROUP_BPF` 등 커널 컴파일 옵션이 활성화되어야 한다. 일부 커스텀 커널 또는 보안 강화 커널(SELinux strict mode, gVisor 등)에서는 이 옵션이 비활성화되어 있거나 eBPF 프로그램 로딩이 제한될 수 있다. Cilium 설치 전 `cilium-preflight-check`를 실행해 환경 적합성을 반드시 확인해야 한다.

### 실무 적용

Cilium 도입 타당성을 평가하는 체크리스트를 만든다. (1) 전체 노드의 커널 버전 분포 조회(`kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}: {.status.nodeInfo.kernelVersion}{"\n"}{end}'`), (2) `cilium-preflight`를 테스트 노드에서 실행해 필수 커널 기능 지원 여부 확인, (3) 보안/컴플라이언스 팀과 커널 버전 요구사항 협의. 이 과정에서 커널 업그레이드가 필요하다면, Cilium 도입 타임라인에 커널 업그레이드 기간을 추가해야 한다.

## Q4. 서비스 메시 시장은 결국 하나의 표준으로 수렴될 것인가, 아니면 영구적으로 분화된 상태를 유지할 것인가?

### 왜 중요한가?

컨테이너 오케스트레이션은 Kubernetes가 사실상 표준이 되었다. 서비스 메시도 같은 경로를 밟을까? 아니면 서로 다른 사용 사례(클라우드 네이티브 vs 온프레미스, 소규모 vs 대규모, 단순 mTLS vs 복잡한 트래픽 관리)에 맞게 복수의 솔루션이 공존할까? 이 질문은 지금 특정 메시에 투자하는 것이 "표준에 베팅하는가, 틈새에 베팅하는가"를 판단하는 데 중요하다.

### 분석

수렴을 지지하는 근거는 Gateway API의 등장이다. Gateway API는 서비스 메시의 인터페이스를 표준화하려는 시도이며, Istio, Linkerd, Cilium 모두 Gateway API를 지원한다. 어플리케이션 팀이 `HTTPRoute`, `GRPCRoute`를 사용해 트래픽 정책을 정의하면, 특정 메시 구현에 종속되지 않는다. 이는 컨테이너 런타임에서 CRI가 표준화된 것과 유사한 패턴이다.

분화를 지지하는 근거는 데이터 플레인의 근본적 차이다. 사이드카 모델과 eBPF 모델은 아키텍처가 다르고, 각각이 최적인 시나리오가 다르다. 사이드카는 L7 정책을 Pod 수준에서 세밀하게 적용할 수 있고, eBPF는 L3/L4에서 고성능을 낸다. 이 차이가 특정 워크로드(고빈도 소량 패킷 vs 복잡한 HTTP 정책)에서 의미 있기 때문에, 하나의 솔루션이 모든 경우를 최적으로 처리하기 어렵다.

현실적으로는 3-4개 프로젝트가 경쟁하는 상태가 5-7년 더 지속될 가능성이 높다. Kubernetes 생태계에서 CNI도 Calico, Flannel, Cilium 등이 공존하며 각자의 위치를 유지하고 있다. 메시도 비슷한 패턴을 따를 것으로 보인다.

### 실무 적용

"표준이 될 메시를 기다리자"는 전략은 현실적이지 않다. 지금 당장 필요하다면 현재 가장 성숙하고 Gateway API 지원이 완전한 솔루션을 선택한다. 하지만 메시에 깊이 종속되는 기능(예: Istio 전용 `EnvoyFilter`)은 최소화하고, Gateway API의 표준 리소스를 최대한 사용해 향후 마이그레이션 비용을 줄인다.

## Q5. "무료" 오픈소스 서비스 메시의 숨겨진 비용은 무엇인가?

### 왜 중요한가?

Istio, Linkerd, Cilium은 모두 무료다. 라이선스 비용이 없다. 하지만 소프트웨어 비용이 전체 TCO(Total Cost of Ownership)의 일부에 불과하다는 것은 잘 알려진 사실이다. 서비스 메시는 특히 운영 복잡성이 높아, "무료"로 도입했다가 예상치 못한 비용이 발생하는 사례가 많다. 도입 전에 이 숨겨진 비용을 명시적으로 계산해야 한다.

### 분석

첫 번째 숨겨진 비용은 학습 곡선이다. Istio의 개념(VirtualService, DestinationRule, ServiceEntry, EnvoyFilter 등)을 팀 전체가 이해하는 데 수개월이 걸린다. 이 기간에 운영 팀은 새로운 장애 모드를 처음 접하고 원인 파악에 더 오랜 시간을 쓴다. Forrester 연구에 따르면 서비스 메시 도입 첫 6개월간 엔지니어링 생산성이 일시적으로 15-20% 감소하는 경향이 있다.

두 번째는 인프라 비용이다. Istio의 컨트롤 플레인(Istiod)은 대규모 클러스터에서 상당한 CPU/메모리를 소비한다. Envoy 사이드카는 Pod당 추가 메모리(50-100MB)를 사용하고, CPU 오버헤드도 발생한다. 1,000 Pod 클러스터라면 사이드카만으로 50-100GB의 추가 메모리가 필요하다. 이는 추가 노드 비용으로 이어진다.

세 번째는 디버깅 복잡성이다. 메시가 없을 때는 네트워크 문제가 서비스 로그나 `tcpdump`로 진단 가능했다. 메시가 있으면 Envoy 로그, xDS 설정, 인증서 상태, Pilot 상태 등 여러 레이어를 봐야 한다. 장애 시 MTTR(Mean Time to Recovery)이 길어지고, 이는 SLA 위반 비용으로 연결될 수 있다.

네 번째는 업그레이드 비용이다. 메시는 쿠버네티스 버전과의 호환성을 유지하며 정기적으로 업그레이드해야 한다. Istio는 매 쿠버네티스 마이너 버전마다 지원 매트릭스가 바뀌고, 사이드카 업그레이드는 롤링 재시작을 요구한다. 이 과정이 연 2-3회 필요하다.

### 실무 적용

도입 제안서에 TCO 계산서를 포함한다. 라이선스 비용(0원) 외에 (1) 엔지니어 교육 비용(인당 5-10일 * 팀 규모), (2) 추가 인프라 비용(사이드카 메모리 * Pod 수 * 노드 단가), (3) 운영 인력 비용(메시 운영 담당 0.5 FTE 추정), (4) 업그레이드 주기당 엔지니어링 일수를 항목으로 넣는다. 이 계산을 상용 메시(Consul Enterprise, Buoyant BEL)의 라이선스 비용과 비교하면 실제 의사결정에 도움이 된다.

## Q6. Gateway API 표준이 메시 선택의 영향을 실질적으로 줄일 수 있는가, 아니면 여전히 구현 종속성이 남는가?

### 왜 중요한가?

Gateway API는 Kubernetes SIG-Network에서 개발 중인 다음 세대 트래픽 관리 API다. Ingress API의 한계를 극복하고, 서비스 메시까지 포괄하는 표준을 목표로 한다. "메시 독립적인 트래픽 정책"이 가능해진다면, 특정 메시 선택이 중요성이 줄어든다. 하지만 표준 API가 있어도 구현체마다 동작이 다른 경우는 흔하다. 실제로 Gateway API가 메시 선택의 영향을 줄여주는지 따져봐야 한다.

### 분析

Gateway API의 `HTTPRoute`, `GRPCRoute`, `TCPRoute`는 기본적인 트래픽 라우팅을 표준화한다. 가중치 기반 라우팅, 헤더 매칭, 경로 기반 라우팅은 세 메시 모두 표준 스펙에 따라 동작한다. 이 범위에서는 메시 교체가 실제로 쉬워진다.

하지만 고급 기능에서는 구현 종속성이 남는다. Istio의 `EnvoyFilter`로만 가능한 L7 정책(예: 커스텀 Lua 필터, 세밀한 재시도 정책)은 Gateway API로 표현할 수 없다. Linkerd의 서킷 브레이커 정책도 Linkerd 전용 CRD(`ServiceProfile`)를 사용한다. Cilium의 L3/L4 네트워크 정책은 `CiliumNetworkPolicy`로 관리한다.

Gateway API의 GAMMA(Gateway API for Mesh Management and Administration) 이니셔티브는 메시 내부 통신(East-West)에도 Gateway API를 적용하려는 시도다. 아직 실험적 단계이며, 완전 표준화까지는 시간이 필요하다. 현재 시점에서 Gateway API만으로 모든 메시 기능을 추상화하는 것은 불가능하다.

실용적인 관점에서 "어느 정도까지 표준을 사용하면 이동이 쉬워지는가"를 기준으로 설계해야 한다. 기본 라우팅은 Gateway API, 고급 정책은 메시 전용 CRD를 사용하되, 메시 전용 기능을 사용하는 레이어를 명확히 분리하면 나중에 교체할 때 변경 범위를 최소화할 수 있다.

### 실무 적용

신규 서비스 개발 시 트래픽 관리 정책을 Gateway API 리소스(`HTTPRoute` 등)로 먼저 작성하고, 메시 전용 기능이 필요할 때만 메시 CRD를 추가한다. 이렇게 작성된 정책은 나중에 `helm diff`로 메시 교체 영향 범위를 쉽게 파악할 수 있다. 팀 내 트래픽 정책 리뷰 가이드라인에 "Gateway API로 표현 가능한가?"를 첫 번째 체크 항목으로 추가한다.
