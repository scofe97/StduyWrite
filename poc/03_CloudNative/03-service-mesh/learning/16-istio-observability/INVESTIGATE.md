# Ch16. Istio 관측성 — Deep Dive Questions

## Q1. 메트릭 레이블이 늘어날 때 카디널리티 폭발이 발생하는 지점은 어디인가?

### 왜 중요한가?
Istio는 각 요청에 대해 `istio_requests_total` 같은 메트릭을 생성하고, 여러 레이블(source_app, destination_service, response_code, request_protocol 등)을 붙인다. Prometheus는 레이블 조합 수만큼 시계열을 생성한다. 레이블 값의 경우의 수가 늘어날수록 시계열 수가 기하급수적으로 증가하는 카디널리티 폭발이 발생한다. 이 시점을 예측하지 못하면 Prometheus 메모리가 폭증해 OOM 장애가 발생한다.

### 분析
레이블 카디널리티 문제의 전형적 원인은 고유 값이 많은 레이블을 추가하는 경우다. 사용자 ID, 요청 ID, 세션 ID 같은 레이블은 각 요청마다 다른 값을 가지므로, 이를 메트릭 레이블로 추가하면 시계열 수가 요청 수만큼 증가한다. Istio 기본 레이블은 이런 고유 값을 포함하지 않도록 설계되어 있지만, `Telemetry` CRD로 커스텀 레이블을 추가할 때 이 원칙을 위반하기 쉽다.

기본 레이블만으로도 카디널리티가 높아지는 경우도 있다. `destination_service_name`은 클러스터의 서비스 수만큼 값을 가진다. 500개 서비스가 있고, 각 서비스별로 5개 source_app이 통신하며, 20개 response_code 조합이 있다면, `istio_requests_total`의 시계열 수는 500 × 5 × 20 = 50,000개다. 여기에 `request_protocol`(HTTP, gRPC, TCP) 레이블까지 더하면 150,000개로 늘어난다. 이 수준에서는 Prometheus 성능이 저하되기 시작한다.

Istio는 카디널리티를 줄이기 위해 `Telemetry` CRD의 `metrics.overrides`를 제공한다. 불필요한 레이블을 비활성화하거나, 특정 메트릭 자체를 비활성화할 수 있다. 예를 들어 `destination_principal` 레이블이 필요 없다면 비활성화해 시계열 수를 줄일 수 있다. 또한 TCP 메트릭이 필요 없는 HTTP 전용 서비스에서 TCP 메트릭을 비활성화하는 것도 효과적이다.

Prometheus 측에서는 `metric_relabel_configs`로 수집 시점에 레이블을 드롭하거나 집계할 수 있다. 이미 카디널리티가 높은 메트릭에서 불필요한 레이블을 스크래핑 시점에 제거하면, Prometheus 내부 시계열 수를 줄일 수 있다. 단, 드롭한 레이블로 나중에 분석할 수 없다는 점을 감수해야 한다.

### 실무 적용
카디널리티 모니터링: `prometheus_tsdb_head_series` 메트릭으로 현재 시계열 수를 추적하고, 빠르게 증가하는 추세를 조기에 발견해야 한다. `topk(10, count by (__name__)({job="istio-proxy"}))` 쿼리로 시계열이 가장 많은 메트릭을 찾고, 해당 메트릭의 레이블 카디널리티를 분석한다. 새 커스텀 레이블 추가 시 "이 레이블이 가질 수 있는 고유 값의 최대 수가 몇 개인가?"를 먼저 계산하고, 100개 이상이라면 레이블 추가를 재고해야 한다.

## Q2. 분산 추적의 샘플링 비율을 어떻게 결정해야 하는가? 1%와 100% 사이의 트레이드오프는 무엇인가?

### 왜 중요한가?
분산 추적에서 모든 요청을 추적(100% 샘플링)하면 완전한 가시성을 얻지만, 추적 데이터 볼륨이 트래픽 볼륨에 비례해 증가한다. 1% 샘플링은 스토리지 비용을 줄이지만, 희귀한 오류나 레이턴시 스파이크를 추적 데이터로 포착할 확률이 낮아진다. 올바른 샘플링 비율은 단일 수치가 아니라 서비스 특성과 운영 목표에 따라 달라진다.

### 分析
헤드 기반 샘플링(head-based sampling)과 테일 기반 샘플링(tail-based sampling)의 차이를 먼저 이해해야 한다. Istio의 기본 샘플링은 헤드 기반으로, 요청이 시작될 때 샘플링 여부를 결정한다. 이 방식은 단순하지만, 샘플링 결정 시점에 요청이 오류를 반환할지 알 수 없다. 결과적으로 오류 요청이 샘플링되지 않을 수 있다. 1% 샘플링에서 오류율이 0.1%라면, 오류 추적을 포착할 확률이 낮다.

테일 기반 샘플링은 요청이 완료된 후 결과(오류 여부, 레이턴시)를 보고 샘플링 여부를 결정한다. OpenTelemetry Collector의 `tail_sampling` 프로세서를 사용하면, "오류가 있는 스팬은 100% 샘플링, 정상 요청은 1% 샘플링" 같은 정책을 구현할 수 있다. 이 방식이 진단 가치는 높지만, Collector에서 요청 완료를 기다려야 하므로 메모리 버퍼가 필요하고 인프라 복잡도가 증가한다.

서비스 초당 요청 수(RPS)에 따라 적합한 샘플링 비율이 다르다. RPS가 10인 서비스에서 1% 샘플링은 6분에 한 번 추적 데이터를 수집하는 것과 같다. 이 정도면 추적 데이터로 의미 있는 분석이 어렵다. 반대로 RPS가 10,000인 서비스에서 100% 샘플링은 초당 10,000개 스팬을 추적 백엔드에 전송한다. Jaeger나 Tempo의 수신 용량을 확인해야 한다.

샘플링 비율이 레이턴시 측정에 미치는 영향도 있다. 추적 데이터 전송 자체가 레이턴시를 추가한다. Envoy는 스팬을 생성하고 Jaeger 에이전트에 UDP로 전송하는데, 이 오버헤드는 수 마이크로초 수준으로 무시 가능하다. 그러나 추적 백엔드 사이드카(Jaeger agent)가 포화 상태면 UDP 패킷이 드롭되어 추적 데이터가 손실될 수 있다.

### 실무 적용
샘플링 전략 권장안: (1) 개발/스테이징 환경: 100% 샘플링 (모든 문제를 포착), (2) 프로덕션 저트래픽 서비스(RPS < 100): 10-100% 샘플링, (3) 프로덕션 고트래픽 서비스(RPS > 1,000): 1-5% 샘플링 + 테일 기반 오류 샘플링. Istio의 `Telemetry` CRD로 서비스별 샘플링 비율을 다르게 설정할 수 있다. `meshConfig.defaultConfig.tracing.sampling` 글로벌 설정보다 `Telemetry` CRD 설정이 우선하므로, 중요 서비스에 더 높은 샘플링을 적용할 수 있다.

## Q3. Kiali와 Grafana 중 어느 대시보드를 언제 선택해야 하는가?

### 왜 중요한가?
Kiali와 Grafana는 서비스 메시 관측성을 위해 자주 함께 사용되지만, 설계 목적이 다르다. 둘 다 있으면 팀원마다 다른 도구를 열게 되어, 인시던트 대응 시 어디를 봐야 하는지 혼란이 생긴다. 각 도구가 실제로 무엇을 잘하고, 어떤 시나리오에서 어느 도구가 더 적합한지 명확히 알아야 팀이 일관된 방식으로 메시를 운영할 수 있다.

### 分析
Kiali는 서비스 토폴로지와 메시 구성에 특화되어 있다. Kiali의 핵심 기능은 서비스 간 트래픽 흐름을 실시간 그래프로 시각화하는 것이다. 어떤 서비스가 어떤 서비스와 통신하는지, 그 연결의 오류율과 레이턴시가 얼마인지를 한눈에 파악할 수 있다. 또한 Istio CRD(VirtualService, DestinationRule)의 유효성을 검증하고, 잘못 구성된 정책을 경고해준다. 새 서비스 의존성 파악, 카나리 배포 진행 상황 모니터링, Istio 설정 문제 진단에는 Kiali가 적합하다.

Grafana는 메트릭 기반 심층 분석에 강하다. 시계열 쿼리(PromQL)를 자유롭게 작성해 복잡한 메트릭 분석이 가능하고, 알람 설정, 패널 커스터마이징, 여러 데이터 소스(Prometheus, Loki, Tempo) 통합이 Grafana의 장점이다. Istio 공식 Grafana 대시보드는 서비스별 RPS, 레이턴시 분포, 오류율을 시계열로 보여준다. 용량 계획, SLO 달성률 추적, 성능 회귀 분석에는 Grafana가 더 적합하다.

인시던트 대응 시나리오에서의 차이가 선택 기준이 된다. "서비스 A가 느려졌다"는 신고를 받았을 때: Kiali를 열면 서비스 A와 연결된 의존 서비스의 오류율을 그래프로 즉시 확인할 수 있어, "B 서비스에서 오는 호출이 느리다"는 원인을 빠르게 파악한다. Grafana를 열면 PromQL로 정확한 레이턴시 분포와 시계열 변화를 분석해, 언제부터 느려졌는지와 패턴을 확인한다. 초기 원인 파악에는 Kiali, 상세 분석과 근거 수집에는 Grafana를 사용하는 워크플로우가 자연스럽다.

두 도구의 데이터 신선도도 다르다. Kiali는 실시간 메트릭을 표시하고(기본 1분 갱신), Grafana는 Prometheus 스크래핑 주기(기본 15초)에 따라 데이터가 업데이트된다. 급격한 변화를 추적하려면 Grafana가 더 적합하다.

### 실무 적용
팀 내 도구 사용 규칙을 명확히 정하는 것이 혼란을 줄인다. 권장 분업: (1) 서비스 의존성 이해, Istio 설정 검증, 트래픽 흐름 확인 → Kiali, (2) 시계열 메트릭 분석, SLO 대시보드, 알람 설정, 용량 계획 → Grafana, (3) 분산 추적 검색 → Jaeger/Tempo. 인시던트 런북에 각 증상별로 "Kiali에서 X를 확인, 이후 Grafana에서 Y 쿼리 실행" 형태로 도구 순서를 명시하면 인시던트 대응 시간을 단축할 수 있다.

## Q4. 분산 추적의 헤더 전파를 놓치면 어떤 일이 생기는가? 라이브러리 호환성 문제는 어떻게 접근하는가?

### 왜 중요한가?
Istio 사이드카는 인바운드 요청에서 `x-request-id`, `x-b3-traceid` 같은 추적 헤더를 자동으로 생성하지만, 이 헤더를 아웃바운드 요청에 전파하는 것은 애플리케이션 코드가 해야 한다. 애플리케이션이 헤더를 전파하지 않으면, 추적 데이터가 서비스 경계에서 끊겨 분산 추적의 핵심 기능(전체 요청 흐름 추적)이 동작하지 않는다. 이 요구사항이 서비스 코드에 어떤 부담을 주는지, 어떻게 관리할 수 있는지 알아야 한다.

### 分析
헤더 전파를 누락했을 때의 증상은 Jaeger/Tempo에서 추적이 중간에 끊기는 것이다. Service A → Service B → Service C 흐름에서, B가 헤더를 전파하지 않으면 A→B 스팬과 B→C 스팬이 별개의 트레이스로 기록된다. 장애 발생 시 전체 요청 경로를 단일 트레이스로 조회할 수 없어, 원인 추적이 어려워진다.

전파해야 하는 헤더 목록이 추적 시스템에 따라 다르다. B3 형식(Zipkin)은 `x-b3-traceid`, `x-b3-spanid`, `x-b3-parentspanid`, `x-b3-sampled`, `x-b3-flags`를 전파해야 한다. W3C TraceContext 형식(OpenTelemetry 표준)은 `traceparent`와 `tracestate`만 전파하면 된다. Istio 1.12 이후 W3C TraceContext가 기본 형식으로 채택되는 추세이지만, 기존 서비스가 B3를 사용한다면 두 형식을 모두 전파해야 하는 과도기가 생긴다.

서비스 언어별 라이브러리 호환성이 실제 문제다. Java Spring Boot는 Spring Cloud Sleuth(최신에서는 Micrometer Tracing)가 헤더 전파를 자동으로 처리한다. Go의 경우 OpenTelemetry SDK를 사용하면 되지만, HTTP 클라이언트 미들웨어에 명시적으로 propagator를 설정해야 한다. Python requests 라이브러리는 기본적으로 추적 헤더를 전파하지 않으므로, OpenTelemetry instrumentor를 추가해야 한다.

gRPC 서비스에서의 헤더 전파는 HTTP와 다르다. gRPC는 HTTP 헤더 대신 메타데이터(metadata)를 사용한다. gRPC 서버로 들어온 추적 컨텍스트를 gRPC 클라이언트 호출에 전파하려면, 언어별 gRPC OpenTelemetry interceptor를 사용해야 한다. 이 interceptor가 없으면 gRPC 서비스 경계에서 추적이 끊긴다.

### 실무 적용
헤더 전파를 서비스 코드에 맡기는 대신, OpenTelemetry SDK를 공통 라이브러리화하고 서비스 템플릿에 포함시키는 것이 조직적 해결책이다. Java Spring Boot 기반 조직이라면 `spring-boot-starter-opentelemetry` 를 공통 BOM에 포함하고, HTTP 클라이언트와 gRPC interceptor 설정을 자동 구성해 신규 서비스가 별도 설정 없이 헤더 전파를 수행하도록 한다. 전파 여부 검증은 `istioctl x describe pod <pod>` 또는 Kiali에서 추적이 서비스 경계를 넘어 연결되는지 확인하는 방법으로 수행할 수 있다.

## Q5. 액세스 로그를 대규모로 활성화하면 성능에 어떤 영향이 있는가?

### 왜 중요한가?
Istio 사이드카는 각 요청에 대해 액세스 로그를 생성할 수 있다. 초당 수천 건의 요청이 있는 서비스에서 모든 요청을 로깅하면, 로그 볼륨이 서비스의 데이터 볼륨을 초과할 수 있다. 이 로그를 수집, 전송, 저장하는 파이프라인의 부하도 증가한다. 액세스 로그의 실제 가치가 이 비용을 정당화하는지 판단해야 한다.

### 分析
Envoy 액세스 로그 생성 자체의 CPU 오버헤드는 요청당 수 마이크로초 수준으로 미미하다. 그러나 로그를 직렬화하고 파일 디스크립터에 쓰는 I/O 오버헤드가 실제 병목이 될 수 있다. Envoy는 기본적으로 표준 출력(stdout)에 액세스 로그를 쓰고, 로그 수집 에이전트(Fluent Bit, Fluentd)가 이를 읽어간다. 초당 10,000건 로그라면 로그 에이전트의 처리 용량을 확인해야 한다.

네트워크 전송 비용도 고려해야 한다. 로그를 중앙 집중식 스토리지(Elasticsearch, Loki)로 전송할 때, 로그 볼륨이 클러스터 내부 네트워크 대역폭을 상당 부분 소비할 수 있다. 특히 크로스 AZ 로그 전송은 데이터 전송 비용으로 직결된다. 로그를 압축해 전송하거나, 같은 AZ의 수집 엔드포인트를 사용하는 방식으로 비용을 줄일 수 있다.

액세스 로그의 실용적 가치는 샘플링된 분산 추적으로 대체 가능한 경우가 많다. 오류 진단에 필요한 요청 상세 정보(헤더, 응답 코드, 레이턴시)는 추적 데이터에도 있다. 오류 요청만 액세스 로그를 남기는 선택적 로깅이 비용 대비 가치를 높인다. Envoy의 `%RESPONSE_CODE%` 조건으로 5xx 응답에만 상세 로그를 남기도록 액세스 로그 포맷을 커스터마이징할 수 있다.

`Telemetry` CRD의 `accessLogging` 설정으로 서비스별, 네임스페이스별로 액세스 로그를 다르게 설정할 수 있다. 중요 서비스는 상세 로그를, 내부 헬스 체크 트래픽은 로그를 비활성화하는 방식으로 로그 볼륨을 제어한다. 특히 `/health`, `/ready`, `/metrics` 경로는 1분에 수십 번 호출되므로, 이 경로의 로그를 제외하면 로그 볼륨이 크게 줄어든다.

### 실무 적용
액세스 로그 전략: (1) 기본은 오류(5xx) + 느린 요청(P99 이상) 조건부 로깅, (2) 헬스 체크, 메트릭 스크래핑 경로 제외, (3) 로그 포맷을 JSON으로 설정해 구조화 쿼리 가능하게, (4) 로그 보존 기간을 서비스 SLA 기반으로 결정 (7일 이상은 대부분 과도함), (5) 로그 볼륨 증가 추세를 `container_network_transmit_bytes_total{container="istio-proxy"}` 메트릭으로 추적. 규제 요구사항으로 전체 로깅이 필요하다면, 로그를 Loki에 저장하고 cold storage에 아카이빙하는 파이프라인을 구성하는 것이 비용 효율적이다.

## Q6. OpenTelemetry Collector가 단일 장애점이 되는 것을 어떻게 방지하는가?

### 왜 중요한가?
OpenTelemetry Collector는 메트릭, 추적, 로그를 수집해 여러 백엔드로 라우팅하는 중간 집계 계층이다. 단일 Collector 인스턴스에 클러스터 전체 원격 측정 데이터가 집중되면, 그 Collector가 장애를 일으킬 때 관측성 데이터 전체가 손실된다. 서비스 장애 시 관측성 데이터도 함께 손실되면, 장애 원인을 파악하기 어려워지는 최악의 타이밍에 관측성이 사라진다.

### 分析
OpenTelemetry Collector를 DaemonSet, Deployment, StatefulSet 중 어느 방식으로 배포하느냐에 따라 가용성 특성이 다르다. DaemonSet으로 배포하면 각 노드에 Collector가 실행되어 노드 장애 시 영향이 해당 노드로 국한된다. 그러나 노드당 Collector가 실행되므로 전체 리소스 사용량이 증가하고, 각 Collector의 백엔드 연결이 독립적으로 관리된다.

Collector를 계층화(Gateway-Agent 패턴)하는 것이 대규모 환경의 표준 구성이다. 각 노드에 경량 에이전트(DaemonSet)가 데이터를 수집하고, 중앙 게이트웨이 Collector(Deployment)가 집계, 변환, 라우팅을 담당한다. 에이전트는 단순 포워딩만 하므로 경량이고, 게이트웨이는 수평 확장 가능하다. 게이트웨이 장애 시 에이전트는 데이터를 버퍼에 일시 저장하고 게이트웨이 복구 후 재전송한다. 이 버퍼 크기(`buffer_size`)와 재전송 타임아웃(`timeout`)을 적절히 설정해야 데이터 손실을 최소화할 수 있다.

게이트웨이 Collector의 고가용성은 여러 Replica + LoadBalancer Service로 구현한다. 단, 추적 데이터는 동일 트레이스의 스팬이 같은 Collector 인스턴스로 전송되어야 tail 샘플링이 올바르게 동작한다. tail 샘플링을 사용한다면 `load_balancing` 익스포터를 사용해 traceID 기반으로 동일 Collector로 라우팅해야 한다.

백엔드 장애 시 Collector의 동작도 설계 대상이다. Jaeger가 다운됐을 때 Collector가 무한히 재시도하며 메모리를 소진하는 것을 방지하려면, `retry_on_failure`와 `sending_queue.queue_size` 설정으로 재시도 한계와 큐 크기를 제한해야 한다. 큐가 꽉 찼을 때 새 데이터를 버리거나(drop) 오래된 데이터를 버리거나(evict) 선택해야 하는데, 관측성 데이터는 최신 데이터가 더 가치 있으므로 오래된 데이터를 먼저 버리는 전략이 일반적이다.

### 실무 적용
Collector 고가용성 구성: (1) 에이전트(DaemonSet) + 게이트웨이(Deployment, 최소 3 Replica) 계층화, (2) 게이트웨이 앞에 LoadBalancer Service 배치, (3) `retry_on_failure: {enabled: true, max_elapsed_time: 5m}` 설정으로 무한 재시도 방지, (4) `sending_queue: {enabled: true, num_consumers: 10, queue_size: 1000}` 설정으로 백엔드 장애 시 버퍼링, (5) Collector 자체의 메트릭(`otelcol_processor_dropped_metric_points`)을 Prometheus로 모니터링해 데이터 손실을 탐지. Collector가 내보내는 자체 메트릭을 Prometheus가 수집하도록 설정하면, Collector 가 포화 상태에 가까워질 때 사전에 대응할 수 있다.
