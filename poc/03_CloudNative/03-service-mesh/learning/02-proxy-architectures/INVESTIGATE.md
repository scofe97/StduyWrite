# Ch02. 프록시 아키텍처 비교 — Deep Dive Questions

## Q1. Envoy가 왜 서비스 메시 표준 데이터 플레인이 되었는가? 기술적 설계 결정 중 어떤 것이 이 위치를 만들었고, 그 선택이 가져온 트레이드오프는 무엇인가?

### 왜 중요한가?

Nginx, HAProxy, Linkerd proxy가 모두 훌륭한 프록시인데 왜 Istio, Consul Connect, AWS App Mesh, Kuma 등 대부분의 메시 컨트롤 플레인이 Envoy를 데이터 플레인으로 선택했는지를 이해하는 것은 단순한 역사 공부가 아니다. Envoy의 설계 철학을 이해하면 왜 특정 기능이 그 방식으로 동작하는지, 왜 특정 문제가 발생하는지를 더 깊이 파악할 수 있다.

또한 이 질문은 "표준이 된 것"과 "최선인 것"이 항상 일치하지 않는다는 더 큰 교훈을 담고 있다. Envoy가 표준이 된 이유 중 상당 부분은 기술적 우월성이 아니라 Lyft의 오픈소스 타이밍, Google의 지지, xDS API의 범용성에 있다.

### 분석

Envoy의 핵심 설계 결정은 네 가지다.

**첫째, xDS(eXtensible Discovery Service) API.** Envoy는 설정을 파일에서 읽지 않고 gRPC 스트림을 통해 컨트롤 플레인으로부터 동적으로 수신한다. 이 덕분에 재시작 없이 설정을 변경할 수 있다. HAProxy도 런타임 API를 가지지만 xDS의 범용성과 생태계 지원에는 미치지 못한다. Istio, Envoy Gateway, AWS 등 다양한 컨트롤 플레인이 xDS를 구현할 수 있었던 것은 이 API가 개방형 표준으로 설계되었기 때문이다.

**둘째, L7 First 설계.** Nginx가 L4 HTTP 서버로 시작해 L7 기능을 추가한 반면, Envoy는 처음부터 HTTP/2, gRPC, Thrift 등 다양한 L7 프로토콜을 1등 시민으로 취급했다. 이는 마이크로서비스 시대의 프록시로서 Nginx보다 자연스러운 출발점이었다.

**셋째, 필터 체인 아키텍처.** HTTP 요청은 여러 필터(라우팅, 인증, 속도 제한, 로깅)를 순서대로 통과한다. 이 아키텍처 덕분에 기능 추가가 코어를 건드리지 않고 가능하다. WASM 필터 지원이 이 설계의 자연스러운 확장이다.

그러나 이 설계가 만들어내는 트레이드오프도 있다. Envoy는 C++로 작성되어 메모리 사용량이 상대적으로 높고(최소 50MB), 설정 구조가 복잡하다. Envoy의 설정 YAML은 학습 곡선이 가파르고, 잘못된 설정이 미묘한 문제를 일으키기 쉽다.

### 실무 적용

Envoy를 직접 설정하는 경우보다 Istio나 Envoy Gateway를 통해 간접적으로 사용하는 경우가 대부분이다. 그러나 디버깅 시에는 Envoy의 내부 구조를 알아야 한다. `kubectl exec -it <pod> -c istio-proxy -- curl localhost:15000/config_dump`로 Envoy 설정 전체를 볼 수 있고, 예상치 못한 라우팅 동작의 원인을 찾을 때 이 덤프를 해석할 수 있어야 한다.



## Q2. Rust로 프록시를 작성하는 것(Linkerd proxy, ztunnel)이 C++ Envoy 대비 실질적 이점을 제공하는가? 메모리 안전성 보장이 프록시 맥락에서 얼마나 중요한가?

### 왜 중요한가?

Linkerd v2는 Go로 작성된 프록시(linkerd-proxy)를 Rust로 교체했고, Istio의 Ambient mesh ztunnel도 Rust로 작성되었다. 이 선택은 "Rust의 메모리 안전성"이라는 마케팅적 이유만이 아닌, 실제 공학적 판단에 기반한 것이어야 한다. 그 판단이 무엇인지, 그리고 실제로 약속된 이점이 실현되는지를 검토할 필요가 있다.

특히 프록시는 신뢰 경계(trust boundary)에 위치하므로 CVE 하나가 전체 메시의 보안을 무너뜨릴 수 있다. 메모리 안전성이 보안 측면에서 실질적으로 어떤 의미인지를 구체적으로 이해해야 한다.

### 분석

C++의 메모리 관련 버그(use-after-free, buffer overflow, dangling pointer)는 역사적으로 CVE의 약 70%를 차지한다(Microsoft, Google 보안팀 연구 기준). 프록시처럼 외부 입력을 파싱하고 처리하는 소프트웨어에서 이런 버그는 특히 위험하다. 악의적으로 조작된 HTTP 요청으로 프록시 프로세스 메모리를 읽거나 실행 흐름을 바꿀 수 있기 때문이다.

Rust는 컴파일 타임에 이런 버그를 원천 차단한다. `unsafe` 블록 없이는 메모리 오류가 발생하지 않음을 컴파일러가 보증한다. 이는 런타임 검사나 가비지 컬렉터 없이 달성되므로, C++와 동등한 성능을 유지하면서 안전성을 얻는다.

그러나 실제 이점은 맥락에 따라 다르다. Linkerd proxy의 메모리 사용량이 Envoy 대비 낮은 것은 Rust의 언어 특성이기도 하지만, 더 좁은 기능 범위(Linkerd는 HTTP/2와 gRPC만 집중)의 영향도 크다. 순수하게 "Rust vs C++"의 언어 차이만으로 성능 차이를 설명하기는 어렵다.

생산성 측면에서 Rust의 학습 곡선은 가파르다. 소유권 모델은 프록시처럼 I/O 집약적인 비동기 코드에서 복잡해진다. Envoy의 C++ 코드베이스에 기여하는 엔지니어 풀이 Rust보다 크다는 현실적 제약도 있다.

### 실무 적용

사용자 관점에서 프록시가 Rust인지 C++인지는 대부분의 경우 투명하다. 차이가 드러나는 시점은 CVE가 발생했을 때다. Rust 프록시에서 메모리 관련 CVE가 나오기는 어렵지만, 로직 버그나 `unsafe` 코드의 CVE는 여전히 가능하다. Rust가 모든 보안 문제를 해결하지는 않는다.

선택 기준: 최소 footprint와 보안이 최우선이라면 Linkerd(Rust proxy). 기능 완성도와 생태계 통합이 우선이라면 Envoy(C++)를 기반으로 하는 메시를 선택하는 것이 현실적이다.



## Q3. WASM 필터는 Envoy 확장성을 어떻게 바꾸는가? 그리고 그 접근법이 내장 필터(built-in filter)나 ext_proc 대비 어떤 상황에서 선택되어야 하는가?

### 왜 중요한가?

Envoy의 WASM 필터 지원은 "사용자 정의 로직을 프록시에 삽입"하는 새로운 방식을 제공한다. 이전에는 Envoy를 포크하거나 Lua 스크립트를 쓰거나 ext_proc(외부 프로세스)를 사용해야 했다. WASM은 이 세 가지 선택지의 중간 어딘가에 위치한다. 어디에 위치하는지, 언제 적합한지를 판단할 수 있어야 한다.

특히 WASM의 보안 모델은 흥미롭다. 샌드박스 환경에서 실행되므로 버그가 있는 필터가 Envoy 프로세스 전체를 무너뜨리기 어렵다. 이 속성이 실제 운영에서 어떤 의미를 갖는지를 검토해야 한다.

### 분석

Envoy 필터 확장의 세 가지 방식을 비교해보자.

**내장 필터(built-in C++ filter)**: Envoy 소스 코드에 포함된다. 최고 성능, 모든 Envoy API 접근 가능. 단점은 Envoy를 직접 빌드해야 하고, 업스트림 기여 없이는 유지보수 부담이 크다. 소수의 조직만 현실적으로 선택 가능하다.

**WASM 필터**: 별도 컴파일된 WASM 모듈을 Envoy가 로드한다. 언어 독립적(Rust, C++, Go의 TinyGo로 작성 가능). 샌드박스 격리로 안전하고, 동적 로드가 가능하다. 단점은 WASM 런타임 오버헤드(요청당 약 5-20μs 추가), 메모리 접근 제한(WASM 모듈은 Envoy 내부 상태에 완전히 접근 불가), 그리고 WASM ABI의 안정성 문제(Proxy-WASM 스펙이 아직 발전 중)다.

**ext_proc(External Processing)**: HTTP 요청/응답을 외부 gRPC 서비스로 전달해 처리한다. 가장 유연하고, 어떤 언어로도 작성 가능하며, 복잡한 비즈니스 로직 삽입에 적합하다. 단점은 네트워크 홉이 추가되어 레이턴시 오버헤드가 크고, 외부 서비스 장애가 트래픽에 영향을 준다.

선택 기준은 단순하다: 요청당 처리 시간이 마이크로초 단위로 중요하면 내장 필터, 요청당 수 밀리초까지 허용되면 WASM, 요청당 수십 밀리초까지 허용되고 복잡한 로직이 필요하면 ext_proc.

### 실무 적용

2026년 현재 WASM 필터의 실용적 사용 사례: JWT 검증 커스텀 로직, 요청 본문 기반 라우팅, 커스텀 헤더 조작, API 키 변환. 이런 경우 WASM이 내장 필터보다 배포 유연성이 좋고 ext_proc보다 오버헤드가 낮다.

단, WASM 필터를 프로덕션에 배포하기 전에 반드시 해당 Envoy 버전의 Proxy-WASM 지원 수준을 확인해야 한다. 지원 범위가 버전마다 다르고, 아직 실험적 기능으로 분류된 API가 존재한다.



## Q4. ztunnel의 보안 모델은 무엇이고, 노드 수준 프록시가 Pod 수준 사이드카보다 신뢰 모델에서 어떤 약점을 가지는가?

### 왜 중요한가?

Ambient mesh의 ztunnel은 하나의 프로세스가 노드의 모든 Pod 트래픽을 처리한다. 이것은 사이드카 모델에서 각 Pod가 자신의 프록시를 갖는 것과 근본적으로 다른 신뢰 구조다. 보안 감사나 컴플라이언스 검토에서 이 차이는 중요하게 다루어질 수 있다.

특히 멀티테넌트 클러스터(하나의 클러스터에 여러 팀이나 고객의 워크로드가 함께 실행되는 환경)에서 이 문제는 더욱 날카로워진다. ztunnel이 테넌트 A의 암호화 키를 사용해 테넌트 B의 트래픽을 처리할 때 어떤 격리 보장이 있는가?

### 분석

사이드카 모델의 신뢰 경계는 Pod다. 각 Pod의 Envoy 사이드카는 해당 Pod의 SPIFFE 인증서만 보유하고, 다른 Pod의 트래픽을 처리하지 않는다. Pod가 침해되어도 해당 Pod의 사이드카만 영향을 받는다.

ztunnel의 신뢰 경계는 노드다. ztunnel은 노드의 모든 Pod 인증서를 관리하거나 접근해야 한다(mTLS 처리를 위해). Istio의 구현에서 ztunnel은 워크로드 API를 통해 필요한 인증서를 Just-in-Time으로 요청한다 — 모든 인증서를 미리 캐싱하지 않는다. 그러나 ztunnel 프로세스 자체가 침해된다면 해당 노드의 모든 워크로드 트래픽에 접근할 수 있다.

이를 완화하는 Istio의 접근법: ztunnel은 privileged 없이 실행되고, 최소 권한 원칙을 적용한다. Node level의 RBAC으로 ztunnel이 접근할 수 있는 시크릿을 제한한다. 그러나 이것이 사이드카 모델의 Pod 수준 격리만큼 강한 보안 경계를 제공하는지는 환경과 위협 모델에 따라 다르게 평가될 수 있다.

멀티테넌트 시나리오에서 권장사항: 서로 다른 보안 요구사항을 가진 테넌트는 별도 노드 풀로 격리하거나, 사이드카 모드를 유지하는 것이 더 안전하다.

### 실무 적용

보안 감사에서 ztunnel의 신뢰 모델을 설명해야 하는 상황이 온다면, 핵심 포인트는 두 가지다. 첫째, ztunnel의 폭발 반경은 노드지만, 이는 DaemonSet 자체의 속성이며 사이드카보다 물리적으로 더 안전한 환경(호스트 네임스페이스 분리)에서 실행된다. 둘째, SPIFFE 기반 인증서 관리는 ztunnel과 사이드카 모두 동일하게 적용된다.

엄격한 멀티테넌시 요구사항이 있는 환경에서는 Ambient mesh 채택 전 Istio의 보안 문서와 최신 CVE 이력을 반드시 검토해야 한다.



## Q5. 고처리량(high-throughput) 시나리오에서 프록시 오버헤드는 어떻게 측정해야 하고, 어떤 최적화 기법이 실질적으로 효과가 있는가?

### 왜 중요한가?

"프록시 오버헤드"를 논의할 때 가장 흔한 실수는 단일 숫자로 표현하는 것이다. 오버헤드는 RPS(초당 요청 수), 페이로드 크기, 연결 패턴(단기 연결 vs 연결 재사용), 그리고 CPU 코어 수에 따라 크게 달라진다. 이 복잡성을 이해하지 못하면 벤치마크 결과를 오해하거나, 최적화 작업에서 엉뚱한 곳을 건드리게 된다.

특히 초당 수만 건 이상의 요청을 처리하는 서비스에서 프록시 오버헤드는 인프라 비용과 직결된다. CPU 사용량 20% 증가가 클러스터 전체로 곱해지면 수십 개의 노드에 해당하는 비용이 된다.

### 분석

프록시 오버헤드의 주요 구성 요소를 분해해보자.

**연결 처리 비용**: 새로운 TCP 연결 수립(TLS handshake 포함)은 기존 연결 재사용 대비 10-100배 비용이 높다. HTTP/1.1에서 connection pooling 없이 매 요청마다 새 연결을 만드는 패턴이 있다면, 프록시 오버헤드의 대부분이 여기서 발생한다. 이 경우 프록시 튜닝보다 애플리케이션의 연결 관리 개선이 효과적이다.

**헤더 처리 비용**: HTTP/2는 헤더 압축(HPACK)을 제공해 반복적인 헤더 전송 비용을 줄인다. Envoy는 HTTP/1.1을 수신해 HTTP/2로 업그레이드하는 경우 이 이점을 활용한다. 마이크로서비스 간 통신에서 HTTP/2를 기본으로 설정하면 네트워크 오버헤드와 프록시 처리 비용 모두 줄어든다.

**필터 체인 길이**: Envoy의 HTTP 필터 수가 많을수록 요청당 처리 시간이 증가한다. 불필요한 필터(사용하지 않는 Wasm 필터, 과도한 로깅 필터)를 제거하면 성능이 개선된다. 특히 고처리량 경로에서는 액세스 로그를 샘플링하거나 비동기 전송으로 구성해야 한다.

실제 효과가 있는 최적화: Connection pool 크기 증가(circuit_breakers 설정), HTTP/2 활성화(upstream/downstream 모두), 불필요한 telemetry 필터 제거, Envoy worker 스레드 수를 CPU 코어 수에 맞게 조정.

### 실무 적용

고처리량 시나리오의 프록시 오버헤드를 측정하는 올바른 접근: 먼저 `envoy/stats` 엔드포인트(포트 15000)에서 `downstream_cx_*`와 `upstream_cx_*` 메트릭을 수집해 연결 패턴을 파악한다. 그 다음 CPU 프로파일링(`perf`, `pprof`)으로 실제 핫스팟을 찾는다. 가정이 아닌 데이터를 기반으로 최적화 방향을 결정해야 한다.

"프록시가 느리다"는 증상이 나타날 때, 실제로는 프록시가 아니라 DNS 조회, 인증서 검증 캐시 미스, 또는 upstream 연결 풀 고갈이 원인인 경우가 많다.
