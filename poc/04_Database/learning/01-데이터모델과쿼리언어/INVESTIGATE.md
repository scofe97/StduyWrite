## Q1. 관계형 모델과 문서 모델, 실무에서 어떤 기준으로 선택하는가?

### 왜 이 질문이 중요한가
기술 면접에서 단골 질문이며, 실제 프로젝트 설계 단계에서 DB 선택을 결정짓는 핵심 판단이다. 잘못된 선택은 프로젝트 전체의 쿼리 패턴, 성능, 유지보수 비용에 영향을 미친다.

### 답변

"one-to-many냐 many-to-many냐"라는 단순 기준만으로는 부족하다. 실무에서는 다음 네 가지 축을 함께 고려해야 한다.

**축 1: 데이터 접근 패턴**
- 한 엔티티의 모든 정보를 한 번에 조회하는 패턴이 주라면 → 문서 모델이 유리하다
- 여러 엔티티를 JOIN하여 복합 쿼리를 실행하는 패턴이 주라면 → 관계형이 유리하다
- 예: 제품 상세 페이지(문서형 적합) vs 주문-결제-배송 조합 조회(관계형 적합)

**축 2: 스키마 변경 빈도**
- 필드가 자주 추가/변경되는 도메인(콘텐츠 관리, 사용자 프로필) → 문서 모델의 schema-on-read가 유리
- 스키마가 안정적이고 데이터 정합성이 중요한 도메인(금융, 재고) → 관계형의 schema-on-write가 안전

**축 3: 트랜잭션 요구사항**
- 복수 엔티티에 걸친 ACID 트랜잭션이 필요하면 관계형이 자연스럽다
- MongoDB도 4.0부터 멀티 도큐먼트 트랜잭션을 지원하지만, 본질적으로 문서 단위 원자성에 최적화되어 있다

**축 4: 팀의 기술 역량**
- SQL 경험이 풍부한 팀이라면 관계형에서 시작하는 것이 생산성 면에서 유리하다
- PostgreSQL의 jsonb를 활용하면 관계형의 안정성과 문서 모델의 유연성을 동시에 얻을 수 있다

### 실무 적용
가장 안전한 시작점은 PostgreSQL + jsonb 조합이다. 핵심 비즈니스 엔티티는 정규화된 테이블로 설계하고, 유연한 메타데이터나 설정 정보는 jsonb 컬럼에 저장한다. 순수 문서형 DB는 데이터 접근 패턴이 문서 단위 조회로 확정된 이후에 도입하는 것이 리스크가 적다.

## Q2. 임피던스 불일치를 해결하는 현실적인 방법은 무엇인가?

### 왜 이 질문이 중요한가
ORM을 쓰든 안 쓰든, 객체와 테이블 사이의 변환 비용은 모든 백엔드 프로젝트에서 발생한다. 이 문제를 이해하지 못하면 N+1 쿼리, 불필요한 데이터 로딩, 복잡한 매핑 코드 같은 함정에 빠진다.

### 답변

임피던스 불일치의 근본 원인은 객체 모델(트리/그래프)과 관계형 모델(평면 테이블)의 구조적 차이에 있다. 이를 해결하는 접근법은 크게 네 가지다.

**접근법 1: ORM 프레임워크 (JPA/Hibernate, SQLAlchemy)**
- 장점: 보일러플레이트 코드 감소, 스키마 마이그레이션 도구 제공
- 함정: N+1 문제, Lazy Loading 예측 불가, 생성 SQL 비효율
- 대응: `@BatchSize`, `JOIN FETCH`, 쿼리 로그 모니터링

**접근법 2: 쿼리 빌더 (JOOQ, Kysely, QueryDSL)**
- SQL을 직접 작성하되 타입 안전성을 확보하는 방식
- ORM의 마법 없이 실행되는 SQL을 명확히 제어할 수 있다
- 복잡한 쿼리가 많은 프로젝트에 적합

**접근법 3: Active Record / Data Mapper 패턴**
- Active Record: 도메인 객체가 직접 DB 접근 로직 포함 (Rails, Django)
- Data Mapper: 도메인 객체와 DB 접근 로직을 분리 (Hibernate)
- 도메인 복잡도가 높으면 Data Mapper가 유지보수에 유리

**접근법 4: 문서 모델 자체로 전환**
- 임피던스 불일치가 심한 도메인이라면 MongoDB 같은 문서형 DB를 고려
- 객체 → JSON 변환이 자연스러워 매핑 비용이 거의 없다
- 단, many-to-many 관계가 많은 도메인에서는 오히려 복잡해질 수 있다

### 실무 적용
N+1 문제는 "항상 발생할 수 있다"는 전제로 시작해야 한다. 개발 환경에서 SQL 로그를 활성화하고, 쿼리 수가 예상보다 많으면 즉시 Batch Fetching이나 JOIN으로 전환한다. ORM을 사용하되, 복잡한 조회는 네이티브 쿼리나 쿼리 빌더로 작성하는 혼합 전략이 현실적이다.

## Q3. 소셜 네트워크에서 그래프 DB가 관계형 DB보다 유리한 이유는 무엇인가?

### 왜 이 질문이 중요한가
"관계(relationship)"라는 이름을 가진 관계형 DB가 왜 관계 탐색에 약한지를 이해하면, 데이터 모델과 쿼리 엔진의 본질적인 설계 차이를 파악할 수 있다. 면접에서도 그래프 DB의 존재 이유를 설명하는 데 자주 등장한다.

### 답변

관계형 DB에서 "친구의 친구의 친구"를 찾으려면 자기 조인(self-join)을 3번 수행해야 한다. 홉(hop) 수가 가변적이면 WITH RECURSIVE를 사용해야 하는데, 이는 성능 예측이 어렵고 쿼리가 복잡해진다.

```sql
-- 관계형: 3-hop 탐색에 31줄 필요 (WITH RECURSIVE)
WITH RECURSIVE friends_of(person_id, depth) AS (
    SELECT friend_id, 1 FROM friendships WHERE person_id = ?
    UNION ALL
    SELECT f.friend_id, fof.depth + 1
    FROM friendships f JOIN friends_of fof ON f.person_id = fof.person_id
    WHERE fof.depth < 3
)
SELECT DISTINCT person_id FROM friends_of;
```

```cypher
// 그래프: 동일한 탐색을 1줄로
MATCH (me:Person {id: ?})-[:FRIEND*1..3]->(fof) RETURN DISTINCT fof
```

핵심 차이는 **인접 노드 접근 방식**에 있다.
- 관계형 DB: 인접 노드를 찾으려면 인덱스를 통한 테이블 조회가 필요하다 (O(log n) per hop)
- 그래프 DB: 각 노드가 인접 노드에 대한 직접 포인터를 저장한다 (O(1) per hop). 이를 "index-free adjacency"라고 부른다.

이 차이는 홉 수가 늘어날수록 극적으로 벌어진다. 2-hop까지는 관계형도 충분히 빠르지만, 5-hop 이상의 깊은 탐색에서는 그래프 DB가 수십 배 이상 빠를 수 있다.

### 실무 적용
소셜 네트워크, 추천 시스템, 사기 탐지처럼 "관계 자체가 데이터"인 도메인에서는 그래프 DB를 검토할 가치가 있다. 단, 전체 데이터에 대한 집계 쿼리(총 사용자 수, 평균 친구 수 등)는 관계형이 더 효율적이다. 대부분의 시스템에서는 핵심 데이터를 관계형에 저장하고, 관계 탐색 기능만 그래프 DB로 분리하는 Polyglot Persistence 전략이 현실적이다.

## Q4. 정규화와 비정규화의 경계선은 어디인가? 실무 판단 기준은?

### 왜 이 질문이 중요한가
"정규화가 좋다"는 교과서적 답변은 실무와 괴리가 있다. 대규모 트래픽을 처리하는 시스템에서는 비정규화가 필수적인 경우가 많고, 언제 비정규화할지의 판단이 시스템 성능을 좌우한다.

### 답변

판단 기준을 세 가지 관점에서 정리한다.

**관점 1: 읽기/쓰기 비율 (Read/Write Ratio)**
- 읽기 >> 쓰기인 서비스(뉴스 피드, 제품 카탈로그) → 비정규화가 유리하다
- 쓰기가 빈번한 서비스(재고 관리, 결제) → 정규화를 유지해야 일관성이 보장된다
- X(Twitter)의 홈 타임라인은 읽기가 쓰기보다 수백 배 많으므로, Materialized Timeline을 비정규화하여 미리 생성해둔다

**관점 2: 데이터 변경 전파 비용**
- 사용자 이름처럼 드물게 변하는 데이터는 비정규화해도 전파 비용이 낮다
- 가격이나 재고처럼 실시간으로 변하는 데이터는 정규화 상태를 유지하고 읽기 시점에 JOIN하는 것이 안전하다

**관점 3: Eventual Consistency 허용 여부**
- 비정규화된 데이터가 원본과 불일치하는 시간 간격을 비즈니스가 허용하는가?
- SNS 타임라인에서 프로필 사진이 5분 동안 옛날 사진으로 보이는 것은 괜찮지만, 은행 계좌 잔액이 일시적으로 틀리면 문제가 된다

### 실무 적용
"정규화로 시작하고, 병목이 확인되면 비정규화한다"가 가장 안전한 접근법이다. 비정규화를 적용할 때는 (1) 어떤 데이터를 복사하는지, (2) 원본 변경 시 어떻게 동기화하는지, (3) 동기화 실패 시 어떤 영향이 있는지를 문서화해두어야 나중에 디버깅이 가능하다.

## Q5. Event Sourcing은 언제 도입하고, 언제 피해야 하는가?

### 왜 이 질문이 중요한가
Event Sourcing은 강력하지만 복잡도가 높은 패턴이다. 도입 시점을 잘못 판단하면 단순한 CRUD 서비스에 불필요한 아키텍처 복잡성을 더하게 된다.

### 답변

**도입이 적합한 경우:**
- 감사 로그(audit trail)가 법적으로 요구되는 금융/의료 도메인
- 복잡한 비즈니스 워크플로우가 있고, "왜 이 상태에 도달했는지"를 추적해야 하는 경우
- 같은 이벤트로부터 다양한 읽기 모델(대시보드, 알림, 보고서)을 생성해야 하는 경우
- 쓰기 처리량이 높고, 이벤트 로그의 append-only 특성이 성능에 유리한 경우

**도입을 피해야 하는 경우:**
- 단순 CRUD 서비스 (게시판, 설정 관리 등). 복잡도 대비 이득이 없다
- GDPR 삭제 권리처럼 "과거 기록을 지워야 하는" 요구사항이 강한 경우
- 팀이 Event Sourcing 경험이 없고, 학습 시간이 부족한 경우
- 이벤트 스키마 진화를 관리할 인프라(스키마 레지스트리 등)가 갖춰지지 않은 경우

**주의할 함정:**
- 이벤트 재생 시 외부 API 호출(이메일 발송, 결제)이 재실행되지 않도록 멱등성 처리가 필수
- 이벤트 스토어가 커지면 스냅샷(snapshot)으로 재생 시간을 단축해야 한다
- Materialized View 재구축에 시간이 걸리므로, View 장애 복구 전략을 미리 설계해야 한다

### 실무 적용
Event Sourcing을 전체 시스템에 적용하기보다, 감사가 중요한 특정 도메인(결제, 주문 상태 변경)에만 부분적으로 도입하는 방식이 현실적이다. 나머지 영역은 전통적인 CRUD로 유지하면 복잡도를 관리할 수 있다.
