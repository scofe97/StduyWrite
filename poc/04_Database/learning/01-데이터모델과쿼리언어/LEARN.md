# Ch.01 - 데이터 모델과 쿼리 언어

> **이론 매핑**: `docs/04_Database/01_기본개념/01_데이터_모델과_쿼리언어.md`

---

## 핵심 요약
> 데이터 모델은 소프트웨어가 문제를 표현하는 근본적인 방식이다. 관계형, 문서형, 그래프 모델은 각각 다른 종류의 데이터 관계를 효율적으로 표현하며, 어떤 모델을 선택하느냐에 따라 쿼리 패턴과 시스템 아키텍처가 결정된다. 이 장에서는 각 모델의 구조적 차이와 트레이드오프를 이해하고, 상황에 맞는 데이터 모델을 선택하는 판단력을 기른다.

---

## 학습 목표
1. 데이터 모델 계층화(Layering) 개념을 설명할 수 있다
2. 관계형 모델과 문서 모델의 장단점을 비교할 수 있다
3. 임피던스 불일치(Object-Relational Mismatch) 문제를 이해하고 해결 방안을 제시할 수 있다
4. 정규화와 비정규화의 트레이드오프를 판단할 수 있다
5. 그래프 데이터 모델과 Cypher 쿼리 언어의 기본 구조를 파악할 수 있다
6. 상황에 맞는 데이터 모델을 선택할 수 있다

## 본문

### 1. 데이터 모델의 계층화

소프트웨어 시스템은 여러 층의 데이터 모델을 쌓아 올려 구축된다. 애플리케이션 개발자는 현실 세계를 객체나 API로 모델링하고, 그 아래 계층에서는 이를 JSON, 테이블, 그래프 등으로 표현한다. 스토리지 엔진은 다시 이 데이터를 디스크나 메모리 상의 바이트로 변환한다.

```
┌──────────────────────────────────────┐
│ Application: 현실 세계 → 객체, API   │
├──────────────────────────────────────┤
│ Data Model: 객체 → 테이블/문서/그래프 │
├──────────────────────────────────────┤
│ Storage Engine: 데이터 → 바이트      │
├──────────────────────────────────────┤
│ Hardware: 바이트 → 전류, 자기장      │
└──────────────────────────────────────┘
```

이 구조가 중요한 이유는 각 계층이 아래 계층의 복잡성을 숨기기 때문이다. 데이터베이스 엔지니어는 바이트 레벨의 저장 방식을 신경 쓰고, 애플리케이션 개발자는 데이터 모델 수준에서 사고한다. 이러한 추상화가 없다면 모든 개발자가 디스크 I/O까지 고려해야 할 것이다.

---

### 2. 선언적 쿼리 vs 명령적 쿼리

SQL이 오랜 기간 지배적인 쿼리 언어로 유지되는 이유는 "선언적(declarative)" 특성에 있다. 선언적 쿼리는 **무엇**을 원하는지만 명시하고, **어떻게** 찾을지는 데이터베이스 옵티마이저에게 맡긴다.

| 특성 | 선언적 (SQL, Cypher) | 명령적 (Python, Java) |
|------|---------------------|----------------------|
| 접근 방식 | "무엇"을 원하는지 명시 | "어떻게" 달성할지 명시 |
| 최적화 | DB가 자동 결정 | 개발자가 직접 구현 |
| 병렬화 | DB가 알아서 처리 | 직접 구현 필요 |

이 차이가 실무에서 중요한 이유는, 선언적 쿼리에서는 인덱스 추가나 옵티마이저 업그레이드만으로 쿼리 변경 없이 성능이 개선될 수 있기 때문이다. 명령적 코드로 데이터를 가져오고 있었다면, 성능 개선을 위해 코드 자체를 다시 작성해야 한다.

```sql
-- 선언적: 무엇을 원하는지만 명시
SELECT * FROM users WHERE age > 25 ORDER BY name;

-- 명령적이었다면 개발자가 루프, 정렬까지 직접 구현해야 함
```

### 3. 관계형 모델 vs 문서 모델

#### 3.1 임피던스 불일치 (Object-Relational Mismatch)

객체지향 언어로 개발할 때, 객체와 관계형 테이블 사이에는 근본적인 구조 차이가 있다. 객체는 중첩된 트리 구조를 자연스럽게 표현하지만, 관계형 테이블은 평평한(flat) 행과 열로 데이터를 나눈다. 이 간극을 메우기 위해 ORM(Object-Relational Mapping)이 등장했다.

```
┌─────────────────┐                    ┌─────────────────┐
│   Application   │  ← ORM 변환 →     │    Database     │
│   user.orders   │                    │  users 테이블   │
│   user.address  │                    │  orders 테이블  │
└─────────────────┘                    └─────────────────┘
```

ORM이 보일러플레이트 코드를 줄여주지만, N+1 쿼리 문제처럼 예상치 못한 성능 저하를 유발할 수 있다. N+1 문제란 부모 엔티티 1개를 조회한 뒤 자식 엔티티 N개를 각각 별도 쿼리로 조회하는 현상으로, JOIN 한 번이면 해결될 것을 N+1번의 쿼리로 처리하게 된다.

#### 3.2 문서 모델의 강점: 데이터 지역성

이력서처럼 한 사람에 대한 모든 정보(경력, 학력, 연락처)를 한꺼번에 조회하는 패턴에서는 문서 모델이 유리하다. 모든 관련 데이터가 하나의 JSON 문서에 모여 있으므로 쿼리가 단순해지고, 디스크 I/O도 줄어든다.

```json
{
  "user_id": 251,
  "name": "Barack Obama",
  "positions": [
    {"title": "President", "org": "United States of America"},
    {"title": "US Senator", "org": "United States Senate"}
  ],
  "education": [
    {"school": "Harvard University", "start": 1988, "end": 1991}
  ]
}
```

관계형 모델이었다면 users, positions, education 세 테이블을 JOIN해야 동일한 결과를 얻는다. 반면 문서 모델은 복잡한 many-to-many 관계를 표현하기 어렵고, 중첩된 데이터를 직접 참조하는 것이 불편하다.

#### 3.3 모델 선택 기준

| 상황 | 추천 모델 | 이유 |
|------|----------|------|
| 트리 구조 (one-to-many) | 문서 모델 | 자연스러운 표현, 지역성 |
| 복잡한 many-to-many | 관계형 모델 | JOIN 지원, 정규화 용이 |
| 모든 것이 관계로 연결 | 그래프 모델 | 관계 탐색 최적화 |
| 유연한 스키마 필요 | 문서 모델 | schema-on-read |
| 분석/BI | 관계형 (Star Schema) | SQL 생태계, 집계 |

현대 데이터베이스들은 서로의 장점을 흡수하고 있다. PostgreSQL은 jsonb를 지원하고, MongoDB는 $lookup으로 JOIN을 제공한다. 관계형-문서 하이브리드 접근이 실무에서 강력한 조합이 되고 있다.

### 4. 정규화 vs 비정규화

정규화는 데이터를 ID 참조로 연결하여 중복을 제거하는 방식이고, 비정규화는 실제 값을 복사하여 읽기 성능을 높이는 방식이다.

| 특성 | 정규화 | 비정규화 |
|------|--------|---------|
| 쓰기 성능 | 빠름 (한 곳만 수정) | 느림 (여러 곳 수정) |
| 읽기 성능 | 느림 (JOIN 필요) | 빠름 (JOIN 불필요) |
| 일관성 | 자동 유지 | 수동 유지 필요 |
| 저장 공간 | 적음 | 많음 |

X(Twitter)의 타임라인 구현이 좋은 예시다. 타임라인 항목에 post_id만 저장(정규화)하고, 조회 시점에 실제 내용을 채워넣는(Hydration) 방식을 사용한다. 프로필 사진이나 좋아요 수처럼 자주 변하는 데이터는 ID만 저장하고, 표시할 때 최신 값을 조회하는 것이다.

정규화와 비정규화 중 어느 것을 선택할지는 세 가지 질문으로 판단할 수 있다: (1) 해당 정보가 얼마나 자주 변경되는가? (2) 읽기/쓰기 비율은 어떤가? (3) 데이터 불일치의 비즈니스 영향이 얼마나 큰가?

---

### 5. Schema-on-Read vs Schema-on-Write

관계형 데이터베이스는 Schema-on-Write 방식으로, 데이터를 저장할 때 스키마를 강제한다. 스키마 변경이 필요하면 ALTER TABLE과 마이그레이션을 수행해야 한다.

문서 데이터베이스는 Schema-on-Read 방식으로, 저장할 때는 아무 형태나 허용하고, 읽을 때 애플리케이션 코드에서 해석한다. 이는 정적 타입 언어 vs 동적 타입 언어의 차이와 유사하다.

```sql
-- Schema-on-Write: 마이그레이션 필요
ALTER TABLE users ADD COLUMN first_name text;
UPDATE users SET first_name = split_part(name, ' ', 1);
```

```javascript
// Schema-on-Read: 읽을 때 처리
if (user.name && !user.first_name) {
    user.first_name = user.name.split(" ")[0];
}
```

Schema-on-Read는 데이터 구조가 자주 바뀌거나, 외부 시스템에서 들어오는 비정형 데이터를 다룰 때 유리하다. 반면 데이터 정합성이 중요한 금융 시스템 같은 경우에는 Schema-on-Write가 안전하다.

### 6. 그래프 데이터 모델

그래프 모델은 엔티티 간의 "관계"가 핵심인 도메인에서 빛을 발한다. 소셜 네트워크, 추천 시스템, 사기 탐지, 지식 그래프 등이 대표적인 사용 사례다.

Property Graph 모델에서는 정점(Vertex)과 간선(Edge)이 각각 고유 ID, 레이블, 속성(key-value)을 가진다.

```
Vertex:                     Edge:
┌──────────────────┐       ┌──────────────────┐
│ id, label        │       │ id, label        │
│ properties (K-V) │       │ tail → head      │
│ in/out edges     │       │ properties (K-V) │
└──────────────────┘       └──────────────────┘
```

Cypher 쿼리 언어를 사용하면, "미국에서 태어나 유럽에 사는 사람"을 4줄로 찾을 수 있다. 동일한 쿼리를 SQL의 WITH RECURSIVE로 작성하면 31줄이 필요하다. 이 차이는 적합한 모델과 언어를 선택하는 것이 얼마나 중요한지를 보여준다.

```cypher
MATCH
  (person) -[:BORN_IN]->  () -[:WITHIN*0..]-> (:Location {name:'United States'}),
  (person) -[:LIVES_IN]-> () -[:WITHIN*0..]-> (:Location {name:'Europe'})
RETURN person.name
```

`-[:WITHIN*0..]->` 구문은 WITHIN 관계를 0번 이상 재귀적으로 따라간다는 의미다. 도시 → 주 → 국가 → 대륙처럼 계층적 위치 관계를 자유롭게 탐색할 수 있다.

### 7. Event Sourcing과 CQRS

전통적인 CRUD 방식은 현재 상태만 저장하지만, Event Sourcing은 모든 상태 변경을 불변(immutable) 이벤트로 기록한다. "좌석이 예약되었다", "좌석이 취소되었다"처럼 이미 발생한 사실을 순서대로 쌓는 방식이다.

```
Event Log (Source of Truth):
[RegistrationOpened] → [SeatBooked] → [SeatCancelled] → [...]
        │                    │                │
        ▼                    ▼                ▼
  Booking View         Dashboard View    Badge Printer View
  (Materialized)       (Materialized)    (Materialized)
```

CQRS(Command Query Responsibility Segregation)는 쓰기(Event Log)와 읽기(Materialized View)를 분리하는 패턴으로, Event Sourcing과 자주 함께 사용된다.

이 방식의 장점은 (1) 변경 의도가 명확히 기록되고, (2) 버그 수정 후 View를 재생성할 수 있으며, (3) 감사 로그 역할을 겸한다는 것이다. 단점으로는 GDPR 개인정보 삭제가 어렵고, 이메일 발송 같은 부수 효과(side-effect)를 이벤트 재생 시 주의해야 한다.

## 핵심 정리
| 개념 | 한 줄 요약 |
|------|-----------|
| 데이터 모델 계층화 | 각 계층이 아래 계층의 복잡성을 숨겨 추상화를 제공한다 |
| 선언적 쿼리 | "무엇"만 명시하고 "어떻게"는 DB에 맡겨 최적화 자유도를 확보한다 |
| 임피던스 불일치 | 객체와 테이블 간 구조 차이로 ORM이 필요하지만 N+1 등 함정이 있다 |
| 정규화/비정규화 | 읽기-쓰기 비율과 데이터 변경 빈도에 따라 선택한다 |
| 문서 모델 | one-to-many 관계와 유연한 스키마에 적합하다 |
| 그래프 모델 | 관계 탐색이 핵심인 도메인(소셜, 추천)에서 강력하다 |
| Event Sourcing | 상태 변경을 불변 이벤트로 기록하여 감사와 재생이 가능하다 |
