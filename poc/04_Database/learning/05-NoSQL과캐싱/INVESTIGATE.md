## Q1. CAP 정리에 대한 흔한 오해는 무엇이며, 실제로는 어떻게 적용해야 하는가?

### 왜 이 질문이 중요한가
CAP 정리는 분산 시스템 면접의 단골 주제이지만, 잘못 이해하고 있는 경우가 많다. 정확한 의미를 파악해야 DB 선택과 아키텍처 설계에 올바르게 적용할 수 있다.

### 답변

**오해 1: "CP/AP/CA 중 하나를 선택해야 한다"**
CA는 분산 시스템에서 성립하지 않는다. 네트워크 파티션은 항상 발생할 수 있으므로, P를 포기하는 것은 "분산하지 않겠다"는 의미다. 실질적 선택은 CP와 AP 사이에서만 이루어진다.

**오해 2: "시스템 전체가 CP 또는 AP이다"**
같은 시스템 내에서도 데이터 종류에 따라 다른 선택을 할 수 있다. 재고 데이터는 CP(일관성 우선)로, 사용자 프로필은 AP(가용성 우선)로 설정하는 것이 가능하다. Cassandra에서는 쿼리별로 consistency level을 `ONE`, `QUORUM`, `ALL` 중 선택할 수 있다.

**오해 3: "파티션이 없으면 C와 A를 동시에 가질 수 있다"**
이것은 사실이다. CAP 정리는 파티션이 발생했을 때만 적용된다. 정상 상태에서는 C와 A를 모두 제공할 수 있으며, 대부분의 시간은 정상 상태이다.

**오해 4: "Consistency = Strong Consistency"**
CAP의 C는 Linearizability를 의미한다. Eventual Consistency, Causal Consistency, Sequential Consistency 등 다양한 일관성 수준이 존재하며, 대부분의 시스템은 Linearizability보다 약한 수준에서 실용적인 일관성을 제공한다.

**Eric Brewer(CAP 정리 제안자)의 후속 논문(2012):**
CAP을 "이분법적 선택"으로 단순화하지 말고, 일관성-가용성의 스펙트럼 위에서 워크로드별로 조절해야 한다고 수정했다.

### 실무 적용
DB를 "CP인가 AP인가?"로 분류하기보다, "파티션 발생 시 이 서비스에서 일관성과 가용성 중 무엇이 더 중요한가?"를 질문해야 한다. 결제 시스템은 CP, 소셜 피드는 AP가 적합하다는 식으로 데이터 도메인별로 판단한다.

## Q2. 캐시 무효화 전략을 어떻게 설계해야 데이터 불일치를 최소화할 수 있는가?

### 왜 이 질문이 중요한가
Phil Karlton의 명언 "컴퓨터 과학에서 어려운 두 가지: 캐시 무효화와 이름 짓기"가 있을 만큼, 캐시 무효화는 분산 시스템에서 가장 까다로운 문제 중 하나다. 잘못된 무효화 전략은 오래된 데이터 노출, 불필요한 DB 조회, 또는 캐시 스탬피드로 이어진다.

### 답변

**전략 1: TTL 기반 (Time-Based)**
```
장점: 구현이 가장 단순하다
단점: TTL 동안 옛 데이터가 노출된다
적합: 실시간성이 덜 중요한 데이터 (추천 결과, 통계 등)
```

TTL을 너무 짧게 설정하면 캐시 적중률(hit rate)이 떨어져 캐싱의 의미가 없어진다. 너무 길면 데이터 신선도가 떨어진다. 데이터의 변경 빈도와 비즈니스 허용 지연을 기준으로 설정해야 한다.

**전략 2: 이벤트 기반 무효화 (Event-Driven)**
```
장점: 데이터 변경 즉시 캐시 갱신 가능
단점: 어떤 캐시 키가 영향받는지 추적해야 한다
적합: 실시간성이 중요한 데이터 (가격, 재고 등)
```

구현 시 주의할 점: 데이터 변경 시 "캐시 업데이트"가 아닌 "캐시 삭제"를 해야 한다. 캐시 업데이트는 DB 쓰기와 캐시 쓰기 사이에 경쟁 조건이 발생할 수 있기 때문이다.

```
위험한 패턴 (캐시 업데이트):
T1: DB 업데이트(v2) → 네트워크 지연 → 캐시 업데이트(v2)
T2: DB 업데이트(v3) → 캐시 업데이트(v3)
결과: DB는 v3인데 T1이 늦게 도착하여 캐시가 v2로 덮어쓰일 수 있다

안전한 패턴 (캐시 삭제):
T1: DB 업데이트(v2) → 캐시 삭제
T2: DB 업데이트(v3) → 캐시 삭제
다음 읽기 시 DB에서 최신 v3을 가져와 캐시에 저장
```

**전략 3: CDC 기반 무효화 (Change Data Capture)**
DB의 변경 로그(WAL)를 구독하여 캐시를 갱신한다. Debezium + Kafka 조합이 대표적이다. 애플리케이션 코드에 캐시 무효화 로직을 넣지 않아도 되므로 관심사 분리(SoC)가 깔끔하다. 다만 CDC 파이프라인 자체의 지연(보통 수 밀리초~수 초)이 존재한다.

### 실무 적용
TTL + 이벤트 기반을 조합하는 것이 가장 실용적이다. 이벤트로 즉시 무효화하되, 이벤트 누락에 대비하여 TTL을 안전망(safety net)으로 설정한다. TTL이 5분이면 최악의 경우에도 5분 후에는 최신 데이터로 갱신된다.

## Q3. Redis vs Memcached, 어떤 기준으로 선택하는가?

### 왜 이 질문이 중요한가
캐시 시스템으로 Redis와 Memcached가 양대 산맥이다. 두 시스템의 차이를 이해해야 프로젝트 요구사항에 맞는 선택을 할 수 있다.

### 답변

| 기준 | Redis | Memcached |
|------|-------|-----------|
| 자료구조 | String, Hash, List, Set, Sorted Set, Stream | String만 |
| 지속성 | RDB 스냅샷, AOF 로그 | 없음 (순수 캐시) |
| 복제 | 마스터-슬레이브 복제 | 없음 |
| 클러스터 | Redis Cluster (자동 샤딩) | 클라이언트 측 샤딩 |
| 메모리 효율 | 상대적으로 낮음 (오버헤드) | 높음 (slab allocator) |
| 멀티스레드 | 6.0부터 I/O 멀티스레딩 | 멀티스레드 |
| Pub/Sub | 지원 | 미지원 |
| Lua 스크립팅 | 지원 | 미지원 |

**Redis를 선택해야 하는 경우:**
- 캐싱 외에 Sorted Set(리더보드), List(큐), Pub/Sub 등 다양한 자료구조가 필요한 경우
- 캐시 데이터의 영속성(persistence)이 필요한 경우
- 복제와 고가용성이 필요한 경우
- 단일 캐시 시스템으로 다양한 용도를 커버하고 싶은 경우

**Memcached를 선택해야 하는 경우:**
- 순수 캐싱만 필요하고, 최대한 단순한 시스템을 원하는 경우
- 메모리 효율이 중요한 대규모 캐싱 (Memcached의 slab allocator가 메모리 단편화에 강하다)
- 멀티스레드 성능이 중요한 경우 (높은 코어 수 활용)

### 실무 적용
대부분의 새 프로젝트에서는 Redis가 사실상 표준이다. 다양한 자료구조, 복제, 클러스터, Pub/Sub 등 Redis만으로 캐싱 외 여러 문제를 해결할 수 있기 때문이다. Memcached는 이미 구축된 대규모 캐시 인프라가 있거나, 순수 캐싱 성능만 극한으로 필요한 특수한 경우에 고려한다.

## Q4. 캐시와 DB 사이의 일관성을 어떻게 보장하는가?

### 왜 이 질문이 중요한가
Cache-Aside 패턴에서 캐시와 DB 사이에 일관성이 깨지는 시나리오는 다양하다. 이 문제를 무시하면 사용자가 오래된 데이터를 보거나, 더 심각하게는 결제 금액이 잘못 표시될 수 있다.

### 답변

**시나리오 1: DB 쓰기 성공, 캐시 삭제 실패**
```
1. DB UPDATE 성공
2. Redis DELETE 실패 (네트워크 오류)
3. 캐시에 옛 데이터가 남아 있음
```
해결: TTL로 결국 만료시키거나, 재시도 큐를 사용하여 캐시 삭제를 보장한다.

**시나리오 2: 동시 갱신으로 인한 경쟁 조건**
```
T1: DB 업데이트(v2) → (지연) → 캐시 삭제
T2: 캐시 Miss → DB 읽기(v2) → 캐시 저장(v2)
T3: DB 업데이트(v3) → 캐시 삭제
T1의 캐시 삭제가 T2의 캐시 저장 이후에 도착하면:
  캐시가 삭제되어 다음 읽기에서 v3을 가져옴 (정상)

하지만 T2의 캐시 저장이 T3의 캐시 삭제 이후에 도착하면:
  캐시에 v2가 남아 있음 (불일치!)
```

**Double-Delete 패턴:**
```python
def update_with_double_delete(key, update_func):
    redis.delete(key)                # 1차 삭제
    update_func()                     # DB 업데이트
    time.sleep(0.5)                   # 잠시 대기 (진행 중인 읽기 완료 대기)
    redis.delete(key)                 # 2차 삭제
```
완벽하지는 않지만, 경쟁 조건의 시간 창을 줄여준다.

**궁극적 해결: CDC 기반**
DB의 WAL을 구독하여 캐시를 갱신하면, DB에 반영된 최종 상태만 캐시에 반영된다. 경쟁 조건이 원천적으로 해결된다.

### 실무 적용
대부분의 서비스에서 "캐시 삭제 + TTL"이면 충분하다. 캐시와 DB 사이의 일시적 불일치(수 초~수 분)가 비즈니스에 허용 가능한 경우가 많기 때문이다. 불일치가 허용되지 않는 데이터(결제 금액, 재고 수량)는 캐시하지 않고 항상 DB에서 직접 읽는 것이 가장 안전하다.

## Q5. Redis를 캐시 외에 어떤 용도로 활용할 수 있는가?

### 왜 이 질문이 중요한가
Redis를 "캐시"로만 인식하면 활용 범위의 절반을 놓치게 된다. Redis의 다양한 자료구조를 이해하면, 별도의 시스템 없이 분산 락, 메시지 큐, 실시간 랭킹 등을 구현할 수 있다.

### 답변

**1. 분산 락 (Distributed Lock)**
```python
# Redlock 알고리즘 기반
lock_acquired = redis.set("lock:order:123", "owner_id", nx=True, ex=30)
if lock_acquired:
    try:
        process_order(123)
    finally:
        redis.delete("lock:order:123")
```
`SET key value NX EX 30`: 키가 없을 때만 설정하고, 30초 후 자동 만료. 분산 환경에서 동시 접근을 제어할 때 사용한다.

**2. Rate Limiting (속도 제한)**
```python
def is_rate_limited(user_id, limit=100, window=60):
    key = f"rate:{user_id}:{int(time.time()) // window}"
    count = redis.incr(key)
    if count == 1:
        redis.expire(key, window)
    return count > limit
```
Sliding Window 방식으로 API 호출 횟수를 제한한다.

**3. 실시간 리더보드 (Sorted Set)**
```
ZADD leaderboard 1500 "player1"
ZADD leaderboard 2300 "player2"
ZADD leaderboard 1800 "player3"

ZREVRANGE leaderboard 0 9 WITHSCORES  # 상위 10명
ZRANK leaderboard "player1"            # 순위 조회
```
수백만 명의 순위를 O(log n)으로 실시간 갱신하고 조회할 수 있다.

**4. 메시지 큐 (List/Stream)**
```
LPUSH task_queue "task_data"    # 작업 추가
BRPOP task_queue 30              # 30초 대기하며 작업 가져오기
```
간단한 작업 큐를 Redis List로 구현할 수 있다. Redis Streams를 사용하면 Kafka 수준의 Consumer Group과 메시지 확인(acknowledgment)을 지원한다.

**5. 세션 스토어 (Hash)**
```
HSET session:abc user_id 123 role "admin" expires 1708000000
HGETALL session:abc
```
여러 서버가 세션을 공유해야 하는 환경에서, Redis Hash로 세션 데이터를 저장하면 어떤 서버가 요청을 받아도 동일한 세션에 접근할 수 있다.

### 실무 적용
Redis를 도입할 때 캐싱만 고려하지 말고, 분산 락, Rate Limiting, 세션 관리 등의 요구사항이 있는지 함께 검토한다. 하나의 Redis 인스턴스로 여러 문제를 해결하면 인프라 복잡도를 줄일 수 있다. 단, Redis가 단일 장애점(SPOF)이 되지 않도록 Redis Sentinel이나 Redis Cluster로 고가용성을 확보해야 한다.
