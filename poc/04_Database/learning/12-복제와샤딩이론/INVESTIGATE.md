## Q1. CAP 정리를 실무에서 어떻게 적용해야 하는가?

### 왜 이 질문이 중요한가

CAP 정리는 분산 시스템 면접에서 가장 빈번하게 등장하는 주제이지만, "C, A, P 중 2개 선택"이라는 잘못된 해석이 만연하다. 실무에서 데이터베이스나 메시지 큐를 선택할 때 CAP 레이블(CP/AP)에만 의존하면 잘못된 아키텍처 결정을 내릴 수 있다. 정확한 이해가 기술 선택의 품질을 결정한다.

### 답변

CAP 정리의 정확한 의미는 "네트워크 파티션이 발생했을 때, 선형성(Linearizability)과 가용성(Availability) 중 하나를 포기해야 한다"는 것이다. 핵심 오해와 실제를 정리하면 다음과 같다.

**오해 1: "3개 중 2개를 선택한다"**
파티션은 선택이 아니라 현실이다. 네트워크는 반드시 실패하므로, 실제 선택은 "파티션 시 C 또는 A" 둘 중 하나이다.

**오해 2: "CP 시스템은 항상 가용하지 않다"**
CAP에서 말하는 가용성은 "모든 정상 노드가 모든 요청에 응답"이라는 이론적 정의이다. 실제 시스템의 99.99% SLA와는 다른 개념이다. CP 시스템도 파티션이 없는 정상 상태에서는 높은 가용성을 제공한다.

**오해 3: "시스템 전체가 CP 또는 AP이다"**
하나의 시스템 내에서도 기능별로 다른 선택이 가능하다. 예를 들어 결제 시스템은 CP(잔고 일관성), 상품 추천은 AP(최신이 아니어도 무방)로 설계할 수 있다.

CAP보다 실용적인 프레임워크가 **PACELC**이다.

| 시스템 | Partition 시 | 정상(Else) 시 |
|--------|-------------|--------------|
| Cassandra | A 선택 | L(Latency) 선택 |
| CockroachDB | C 선택 | C 선택 (지연 감수) |
| DynamoDB | A 선택 (기본) | L 선택 (기본), C 선택 가능 (strong consistent read) |

### 실무 적용

데이터베이스 선택 시 "CP인가 AP인가"보다 다음 질문이 더 유용하다.
- 이 데이터의 불일치가 비즈니스에 어떤 영향을 주는가?
- 파티션 발생 확률은 얼마인가? (단일 데이터센터 내에서는 드물다)
- 지연 시간 요구사항은 어느 정도인가?

## Q2. Raft vs Paxos - 어떤 합의 알고리즘을 선택해야 하는가?

### 왜 이 질문이 중요한가

etcd(Raft), ZooKeeper(Zab/Paxos 변형), CockroachDB(Raft), Spanner(Paxos) 등 실무에서 사용하는 분산 시스템마다 다른 합의 알고리즘을 채택한다. 이들의 차이를 이해해야 시스템의 장애 특성과 성능 한계를 예측할 수 있다.

### 답변

**Paxos (1998, Leslie Lamport)**

Paxos는 합의 알고리즘의 이론적 기초이다. 핵심 아이디어는 Proposer가 값을 제안하고, Acceptor 과반수가 동의하면 결정되는 것이다. 2단계 프로토콜(Prepare-Accept)로 동작한다.

- Prepare 단계: Proposer가 고유한 번호(ballot)로 Acceptor들에게 "이 번호로 제안해도 되는가?" 질문
- Accept 단계: 과반수가 허락하면 값을 제안, 과반수가 수락하면 결정

문제는 원본 Paxos 논문이 단일 값 합의만 다루었다는 점이다. 연속적인 결정(로그 복제)을 위해서는 Multi-Paxos가 필요한데, 이에 대한 명확한 명세가 없어 구현마다 해석이 달랐다. Chubby 개발자가 "Paxos 논문과 실제 구현 사이에는 큰 간극이 있다"고 말한 것은 유명하다.

**Raft (2014, Ongaro & Ousterhout)**

Raft는 "이해하기 쉬운 합의 알고리즘"을 목표로 설계되었다. Paxos와 같은 문제를 해결하지만, 리더 선출, 로그 복제, 안전성을 명확하게 분리했다.

- Leader Election: term(epoch) 기반, 랜덤 타임아웃으로 충돌 방지
- Log Replication: 리더가 로그 항목을 팔로워에게 복제, 과반수 확인 후 커밋
- Safety: 가장 최신 로그를 가진 노드만 리더가 될 수 있음

| 비교 항목 | Paxos | Raft |
|----------|-------|------|
| 이해 난이도 | 어려움 (논문 해석 분분) | 상대적으로 쉬움 |
| 리더 | 선택적 (리더리스 가능) | 필수 (강한 리더) |
| 로그 복제 | 별도 확장 필요 (Multi-Paxos) | 내장 |
| 구현 명세 | 모호 | 명확 |
| 성능 최적화 | 리더리스로 지연 줄일 수 있음 (EPaxos) | 리더 병목 존재 |
| 사용 시스템 | Spanner, Chubby | etcd, CockroachDB, TiKV |

**Multi-Paxos vs Raft의 실질적 차이는 작다.** 둘 다 과반수 기반이며, 안정 상태에서의 동작도 유사하다. 주요 차이는 리더 선출 메커니즘과 로그 관리 방식에 있다.

### 실무 적용

대부분의 경우 합의 알고리즘을 직접 구현하지 않고 etcd나 ZooKeeper를 사용한다. 시스템 선택 시 알고리즘 자체보다 구현의 성숙도, 커뮤니티 지원, 운영 도구가 더 중요하다. 다만, 장애 상황에서 디버깅하려면 사용 중인 알고리즘의 리더 선출 로직과 로그 커밋 조건을 이해하고 있어야 한다.

## Q3. 복제 지연(Replication Lag)을 실전에서 어떻게 다루는가?

### 왜 이 질문이 중요한가

비동기 복제를 사용하는 거의 모든 시스템(PostgreSQL Streaming Replication, MySQL/MariaDB 복제)에서 복제 지연은 피할 수 없다. 이론적으로는 "최종 일관성"이라고 말하지만, 사용자가 자신이 방금 수정한 데이터를 보지 못하면 버그로 인식한다. 복제 지연을 어떻게 관리하느냐가 사용자 경험을 결정한다.

### 답변

복제 지연 문제의 유형별 실전 해결 전략을 정리한다.

**1. Read-After-Write (자기 쓰기 읽기)**

가장 흔한 문제이다. 사용자가 프로필을 수정한 직후 새로고침하면 이전 데이터가 보이는 현상이다.

실전 해결 패턴:
- **리더 읽기 라우팅**: 사용자가 자신의 데이터를 읽을 때는 리더에서 읽기. 예를 들어 `GET /users/me`는 리더, `GET /users/{other_id}`는 팔로워로 라우팅한다.
- **타임스탬프 기반**: 클라이언트가 마지막 쓰기 타임스탬프를 쿠키/헤더로 전달하고, 해당 시점 이후 데이터가 반영된 팔로워에서만 읽는다.
- **세션 고정(Session Affinity)**: 같은 세션의 요청을 같은 노드로 고정한다. 단, 노드 장애 시 세션 재배치가 필요하다.

**2. Monotonic Reads (단조 읽기)**

같은 사용자가 여러 팔로워에서 읽으면 "시간 역행"이 발생한다. User ID 해시로 팔로워를 고정하면 해결되지만, 해당 팔로워 장애 시 다른 팔로워로 전환해야 하므로 일시적으로 문제가 재발할 수 있다.

**3. 복제 지연 모니터링**

PostgreSQL에서는 `pg_stat_replication`의 `replay_lag` 컬럼으로 팔로워별 지연을 확인한다. 지연이 임계치(예: 5초)를 넘으면 해당 팔로워를 로드밸런서에서 제외하는 것이 일반적인 운영 패턴이다.

**4. 동기식 복제로의 전환**

중요한 데이터(결제, 재고)에 대해서는 동기식 복제를 사용하거나, 적어도 반동기식(Semi-sync)으로 최소 1개 팔로워에 데이터가 도달하도록 보장한다. 이는 Ch.13(PostgreSQL)과 Ch.14(MariaDB)에서 구체적으로 다룬다.

### 실무 적용

복제 지연 전략을 선택할 때 핵심 기준은 "이 데이터의 불일치가 사용자에게 보이는가?"이다. 사용자에게 직접 보이는 데이터(프로필, 주문 상태)는 Read-After-Write를 보장하고, 내부 분석이나 캐시 갱신처럼 약간의 지연이 허용되는 경우는 최종 일관성으로 충분하다.

## Q4. Consistent Hashing은 실제로 어떻게 구현되며, 어떤 변형이 있는가?

### 왜 이 질문이 중요한가

Consistent Hashing은 분산 캐시(Memcached, Redis Cluster), 분산 DB(Cassandra, DynamoDB), 로드밸런서 등 광범위하게 사용된다. 원본 알고리즘의 한계를 이해하고 실전에서 사용되는 변형을 알아야 적절한 샤딩 전략을 설계할 수 있다.

### 답변

**원본 Consistent Hashing (Karger et al., 1997)**

해시 공간을 링으로 표현하고, 노드와 키를 모두 해시하여 링 위에 배치한다. 각 키는 시계 방향으로 가장 가까운 노드에 할당된다. 노드 추가/제거 시 인접 노드만 영향받으므로 이동하는 키가 최소화된다.

문제는 노드 수가 적을 때 부하가 불균등하게 분배된다는 것이다. 노드 3개가 링 위에 나란히 배치되면 한 노드가 링의 대부분을 담당할 수 있다.

**Virtual Nodes (Vnodes)**

각 물리 노드를 링 위에 여러 지점(가상 노드)으로 배치하는 해결책이다. 예를 들어 물리 노드 3개에 각각 256개의 가상 노드를 배치하면 768개 지점이 링 위에 분산되어 부하가 균등해진다. Cassandra, Riak이 이 방식을 사용한다.

**Jump Consistent Hash (Google, 2014)**

링을 사용하지 않고 수학적으로 균등 분배를 보장하는 알고리즘이다. 코드가 5줄 정도로 짧고 실행 속도가 빠르지만, 노드를 순번(0, 1, 2, ...)으로만 식별할 수 있어 노드 제거가 불편하다는 제약이 있다.

**Rendezvous Hashing (HRW)**

각 키에 대해 모든 노드의 "점수"를 계산하고 가장 높은 점수의 노드에 할당하는 방식이다. 구현이 직관적이고 노드 추가/제거가 자유롭지만, 노드 수가 많으면 모든 노드의 점수를 계산해야 하므로 O(n)이 된다.

| 변형 | 균등 분배 | 노드 추가/제거 | 계산 비용 |
|------|----------|---------------|----------|
| 원본 + Vnodes | 좋음 | 자유 | O(log n) |
| Jump Consistent Hash | 최적 | 끝에서만 제거 가능 | O(ln n) |
| Rendezvous Hashing | 좋음 | 자유 | O(n) |

### 실무 적용

대부분의 경우 데이터베이스나 캐시 시스템이 내장한 구현을 그대로 사용한다. 직접 구현해야 하는 경우(예: 애플리케이션 레벨 샤딩), 노드 수가 적으면(<100) Rendezvous Hashing이 간단하고, 노드 수가 많으면 Virtual Nodes 방식의 Consistent Hashing이 효율적이다.

## Q5. 동시 쓰기 감지는 왜 물리적 시간이 아닌 논리적 관계로 판단하는가?

### 왜 이 질문이 중요한가

분산 시스템에서 "동시"라는 개념은 일상적 직관과 다르다. 물리적으로 같은 시각에 발생하지 않았더라도 서로의 존재를 모르면 "동시"로 간주한다. 이 개념을 잘못 이해하면 LWW(Last Write Wins) 같은 전략의 데이터 손실 위험을 간과하게 된다.

### 답변

**왜 물리적 시간을 신뢰할 수 없는가?**

분산 시스템의 각 노드는 독립적인 시계를 가진다. NTP로 동기화하더라도 밀리초~수백 밀리초 수준의 오차(clock skew)가 존재한다. 이 오차 범위 안에서 발생한 두 이벤트의 선후관계를 물리적 시간으로 판단하면 잘못된 결과가 나올 수 있다.

예를 들어 Node A의 시계가 10ms 앞서 있으면, Node B에서 Node A보다 나중에 발생한 이벤트가 더 작은 타임스탬프를 가질 수 있다. LWW를 사용하면 실제로 나중에 쓴 데이터가 사라지는 것이다.

**Happens-Before 관계**

Leslie Lamport가 1978년 정의한 개념이다. 두 이벤트 A, B의 관계는 세 가지 중 하나이다.

1. A happens-before B: B가 A의 결과를 알고 있음 (인과관계)
2. B happens-before A: A가 B의 결과를 알고 있음
3. A와 B가 동시(concurrent): 서로의 존재를 모름

핵심은 **물리적 시간이 아니라 "정보의 흐름"으로 판단**한다는 점이다. 빛의 속도보다 빠르게 정보가 전달될 수 없듯이, 네트워크 지연으로 인해 두 이벤트가 서로를 알 수 없으면 "동시"로 간주한다.

**Version Vector로 동시성 감지**

단일 복제본에서는 단순한 버전 번호로 동시 쓰기를 감지할 수 있지만, 다중 복제본에서는 Version Vector가 필요하다. 복제본별로 버전 번호를 유지하고, 두 벡터를 비교해서 하나가 다른 하나의 모든 요소보다 작거나 같으면 happens-before, 일부가 크고 일부가 작으면 concurrent로 판단한다.

```
VV_A = {R1:5, R2:3} vs VV_B = {R1:4, R2:6}
R1: A > B, R2: A < B → Concurrent (충돌!)

VV_A = {R1:3, R2:5} vs VV_B = {R1:4, R2:6}
모든 요소에서 A <= B → A happens-before B
```

### 실무 적용

LWW를 사용하는 시스템(DynamoDB 기본 설정, Cassandra)에서는 동시 쓰기 시 데이터 손실이 발생할 수 있다는 점을 반드시 인지해야 한다. 데이터 손실이 허용되지 않는 경우(쇼핑 카트, 문서 편집), CRDT나 애플리케이션 레벨 병합 로직을 구현하거나, 애초에 Single-Leader 복제로 충돌 자체를 회피하는 것이 안전하다.
