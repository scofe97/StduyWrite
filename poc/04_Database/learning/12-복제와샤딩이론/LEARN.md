# Ch.12 - 복제와 샤딩 이론

## 핵심 요약

> 데이터베이스 클러스터링을 이해하려면 세 가지 이론적 기둥을 먼저 세워야 한다. **복제(Replication)**는 동일한 데이터를 여러 노드에 복사해 가용성과 읽기 성능을 높이고, **샤딩(Sharding)**은 데이터를 조각내어 쓰기 부하를 분산시킨다. 그리고 **일관성과 합의(Consistency & Consensus)**는 분산된 노드들이 "하나의 시스템처럼" 동작하기 위한 약속이다. 이 세 개념은 Ch.13(PostgreSQL 클러스터링)과 Ch.14(MariaDB 클러스터링)의 모든 아키텍처 결정을 뒷받침하는 토대이므로, 여기서 확실히 정리하고 넘어가야 한다.

1. Single-Leader, Multi-Leader, Leaderless 세 가지 복제 전략의 트레이드오프를 비교할 수 있다
2. Key-range 샤딩과 Hash 샤딩의 차이를 설명하고 적절한 상황을 판단할 수 있다
3. CAP 정리의 실제 의미를 오해 없이 설명할 수 있다
4. Raft와 Paxos 합의 알고리즘의 핵심 아이디어를 이해할 수 있다
5. 복제 지연(Replication Lag) 문제의 유형과 해결 전략을 설명할 수 있다

## 본문

### 1. 복제(Replication) - 데이터를 복사하는 세 가지 방법

복제란 동일한 데이터를 네트워크로 연결된 여러 머신에 유지하는 것이다. 백업과 혼동하기 쉽지만, 복제는 **실시간 동기화**이고 백업은 **특정 시점 스냅샷**이라는 점에서 근본적으로 다르다. 복제본에서 데이터를 삭제하면 모든 복제본에서 삭제되지만, 백업은 과거 상태를 그대로 보존한다.

복제의 목적은 네 가지로 정리된다.

| 목적 | 설명 |
|------|------|
| 고가용성 | 노드 장애 시에도 서비스 지속 |
| 내구성 | 디스크 손실에 대비한 데이터 보호 |
| 지연 시간 감소 | 지리적으로 가까운 복제본에서 읽기 |
| 읽기 확장성 | 읽기 요청을 여러 복제본에 분산 |

#### 1.1 Single-Leader 복제

가장 직관적인 방식이다. 하나의 리더(Primary)만 쓰기를 받고, 팔로워(Replica)들은 리더의 변경 로그를 받아 동기화한다. PostgreSQL의 Streaming Replication, MySQL/MariaDB의 기본 복제가 이 방식을 사용한다.

```
Client(Write) ──→ [Leader] ──복제로그──→ [Follower 1]
                     │                    [Follower 2]
                     │                    [Follower 3]
Client(Read) ───→ [Leader 또는 Follower]
```

동기식과 비동기식 복제의 차이가 핵심 트레이드오프를 만든다.

| 방식 | 장점 | 단점 |
|------|------|------|
| 동기식 | 데이터 손실 없음, 일관성 보장 | 팔로워 장애 시 쓰기 차단 |
| 비동기식 | 빠른 응답, 높은 가용성 | 리더 장애 시 데이터 유실 가능 |
| 반동기식(Semi-sync) | 최소 2개 노드에 데이터 보장 | 부분적 일관성만 제공 |

반동기식은 하나의 팔로워만 동기식으로 운영하고 나머지는 비동기식으로 두는 절충안이다. 동기 팔로워가 응답이 느려지면 다른 팔로워를 동기 대상으로 승격시킨다.

**리더 장애 처리(Failover)**는 Single-Leader의 가장 어려운 부분이다. 리더가 죽으면 팔로워 중 하나를 새 리더로 선출해야 하는데, 이때 여러 위험이 존재한다.

| 위험 | 설명 |
|------|------|
| 데이터 손실 | 비동기 복제 시 미복제 쓰기 유실 |
| Split Brain | 두 노드가 동시에 리더로 동작 |
| 외부 시스템 불일치 | ID 재사용으로 인한 데이터 충돌 (GitHub 사례) |
| 잘못된 타임아웃 | 과부하를 장애로 오판하여 불필요한 Failover 발생 |

Split Brain 방지를 위해 STONITH(Shoot The Other Node In The Head)라는 메커니즘으로 이전 리더를 강제 종료한다. 이를 "Fencing"이라고도 부른다.

#### 1.2 Multi-Leader 복제

여러 데이터센터에 각각 리더를 두는 방식이다. 각 리전의 리더가 독립적으로 쓰기를 받고, 리더 간 비동기 복제로 동기화한다.

```
[Region A: Leader A] ←──비동기 복제──→ [Region B: Leader B]
       │                                      │
  [Followers]                            [Followers]
```

지리적으로 분산된 서비스에서 쓰기 지연을 줄이고, 한 리전 장애 시에도 다른 리전이 독립 운영되는 것이 장점이다. 그러나 같은 데이터를 동시에 수정할 때 **충돌(Conflict)**이 발생한다는 것이 핵심 단점이다.

충돌 해결 전략은 다음과 같다.

| 전략 | 방식 | 트레이드오프 |
|------|------|-------------|
| 회피 | 같은 레코드를 항상 같은 리더로 라우팅 | 리더 변경 시 깨짐 |
| LWW (Last Write Wins) | 타임스탬프가 큰 값 채택 | 데이터 손실 발생 |
| 수동 해결 | 모든 버전(siblings) 저장, 사용자/앱이 병합 | 복잡하고 API 변경 필요 |
| 자동 병합 | CRDT, Operational Transformation 사용 | 데이터 타입별 알고리즘 필요 |

**CRDT(Conflict-free Replicated Data Type)**는 각 요소에 고유 ID를 부여해 인덱스 변환 없이 병합할 수 있는 데이터 구조이다. Automerge, Yjs 같은 라이브러리가 이를 구현하며, Google Docs의 OT(Operational Transformation)와 달리 서버 중심 조정 없이 P2P 환경에서도 작동한다.

#### 1.3 Leaderless 복제

리더 없이 클라이언트가 여러 노드에 동시에 읽기/쓰기를 보내는 방식이다. Amazon Dynamo 논문에서 유래했으며, Cassandra, ScyllaDB, Riak이 이 방식을 사용한다.

핵심 개념은 **쿼럼(Quorum)**이다.

```
n = 복제본 총 개수
w = 쓰기 성공에 필요한 응답 수
r = 읽기 성공에 필요한 응답 수

조건: w + r > n → 읽기 집합과 쓰기 집합이 반드시 겹침
                → 겹치는 노드 중 최소 하나는 최신 값 보유
```

일반적으로 n=3 또는 5(홀수), w=r=(n+1)/2로 설정한다.

Failover 과정이 없어 리더 선출 지연이 없고, 일부 노드 장애에도 요청 처리가 가능하다는 것이 장점이다. 하지만 w+r>n을 만족해도 리밸런싱, 복구, 동시 읽기/쓰기 등 엣지 케이스에서 일관성이 깨질 수 있다는 한계가 있다.

누락된 쓰기를 복구하는 메커니즘은 세 가지다.

| 메커니즘 | 동작 | 특징 |
|---------|------|------|
| Read Repair | 읽기 시 오래된 복제본 발견하면 최신 값 쓰기 | 자주 읽는 데이터에 효과적 |
| Hinted Handoff | 장애 노드 대신 다른 노드가 쓰기 보관 | 장애 복구 시 전달 |
| Anti-Entropy | 백그라운드에서 복제본 간 차이 동기화 | 순서 보장 없음, 지연 있음 |

### 2. 복제 지연(Replication Lag) 문제

비동기 복제를 사용하면 팔로워가 리더보다 뒤처지는 구간이 생긴다. 이를 "최종 일관성(Eventual Consistency)"이라고 부르는데, 문제는 "결국(eventually)"이 언제인지 보장되지 않는다는 점이다. 정상 시에는 밀리초이지만, 장애나 과부하 시에는 분~시간까지 늘어날 수 있다.

세 가지 대표적인 일관성 문제가 있다.

**Read-After-Write Consistency**: 사용자가 자신의 데이터를 수정한 직후 읽었을 때 이전 값이 보이는 문제다. 해결책은 자신이 수정 가능한 데이터는 리더에서 읽거나, 클라이언트가 마지막 쓰기 타임스탬프를 기억해서 해당 시점 이후 데이터가 반영된 복제본에서만 읽는 것이다.

**Monotonic Reads**: 같은 사용자가 여러 번 읽을 때 최신 데이터를 봤다가 다음 요청에서 오래된 데이터를 보는 "시간 역행" 문제다. 같은 사용자는 항상 같은 복제본에서 읽도록 User ID 해시로 복제본을 고정하면 해결된다.

**Consistent Prefix Reads**: 인과관계가 있는 데이터가 순서대로 보이지 않는 문제다. 예를 들어 질문보다 답변이 먼저 보이는 현상이 발생할 수 있다. 인과적으로 연관된 쓰기를 같은 샤드에 저장하거나 인과 의존성 추적 알고리즘을 사용해서 해결한다.

### 3. 샤딩(Sharding) - 데이터를 나누는 전략

샤딩은 데이터를 작은 조각(shard)으로 나누어 서로 다른 노드에 저장하는 것이다. 복제가 "같은 책을 여러 도서관에 비치"하는 것이라면, 샤딩은 "백과사전을 A-F, G-L, M-R, S-Z 권으로 나누어 다른 서가에 보관"하는 것이다.

실전에서는 샤딩과 복제를 함께 사용한다. 각 샤드가 자체적으로 복제본을 가지며, 한 노드가 일부 샤드의 리더이면서 다른 샤드의 팔로워가 된다.

#### 3.1 Key-range 샤딩

키의 연속 범위를 각 샤드에 할당하는 방식이다.

```
Shard 0: A-F | Shard 1: G-L | Shard 2: M-R | Shard 3: S-Z
```

범위 쿼리(Range Scan)가 효율적이라는 것이 최대 장점이다. 시계열 데이터에서 특정 기간 조회가 빈번한 경우 적합하다. 그러나 연속 키(예: 타임스탬프)로 쓰기가 집중되면 핫스팟이 발생한다. 이를 피하려면 `sensor_id + timestamp` 같은 복합 키를 사용해야 하지만, 그러면 시간 범위 조회 시 센서별로 쿼리해야 하는 트레이드오프가 생긴다.

#### 3.2 Hash 샤딩

키를 해시 함수에 통과시켜 해시값으로 샤드를 결정하는 방식이다. 비슷한 입력값도 해시 후에는 균등하게 분산되므로 핫스팟 문제를 크게 줄인다. 하지만 해시 후 키의 순서가 사라지므로 범위 쿼리가 비효율적이다.

해시 샤딩의 세 가지 변형이 있다.

| 방식 | 원리 | 장단점 |
|------|------|--------|
| Mod N | hash(key) % N | 노드 추가 시 대부분의 키 이동 - 비추천 |
| 고정 샤드 수 | 샤드를 미리 많이 생성, 노드에 분배 | 리밸런싱 시 샤드 단위 이동, 초기 설정 중요 |
| 해시 범위 | 해시 공간을 범위로 분할, 필요 시 분할 | 동적 분할 가능, DynamoDB/MongoDB 사용 |

**일관된 해싱(Consistent Hashing)**은 노드 추가/제거 시 최소한의 키만 이동하도록 설계된 알고리즘이다. 해시 공간을 링으로 표현하고, 키는 시계 방향으로 가장 가까운 노드에 할당된다. Cassandra와 ScyllaDB가 이 방식을 사용한다.

#### 3.3 핫스팟 문제와 해결

해시 샤딩을 써도 특정 키에 트래픽이 몰리면 핫스팟이 발생한다. 소셜 미디어에서 유명인의 포스트가 대표적인 예시이다. 해결책은 애플리케이션 레벨에서 핫키에 랜덤 접미사(00~99)를 붙여 100개 샤드로 분산시키는 것인데, 읽을 때는 100개 샤드를 모두 조회 후 병합해야 하므로 쓰기 분산과 읽기 복잡성 사이의 트레이드오프가 존재한다.

#### 3.4 요청 라우팅

클라이언트의 요청을 올바른 샤드로 전달하는 세 가지 방식이 있다.

1. **임의 노드 접근**: 아무 노드에 요청 → 해당 노드가 올바른 노드로 포워딩
2. **라우팅 티어**: 전용 라우팅 계층이 샤드 매핑 정보를 갖고 올바른 노드로 전달
3. **클라이언트 직접 라우팅**: 클라이언트가 샤드 매핑을 알고 직접 올바른 노드에 요청

샤드-노드 매핑 정보는 ZooKeeper나 etcd 같은 코디네이션 서비스가 관리한다. 노드가 등록/해제될 때 매핑이 업데이트되고, 라우팅 티어나 클라이언트가 이를 구독(Subscribe)하여 최신 상태를 유지한다.

### 4. 일관성 모델과 CAP 정리

분산 시스템에서 "일관성"은 두 가지 철학으로 나뉜다.

**최종 일관성(Eventual Consistency)**: 복제본이 여러 개라는 사실이 애플리케이션에 노출된다. 불일치와 충돌을 애플리케이션이 직접 처리해야 하지만, 가용성과 성능이 높다.

**강한 일관성(Strong Consistency)**: 시스템이 단일 노드처럼 동작한다. 애플리케이션이 복제를 신경 쓸 필요가 없지만, 성능 비용이 발생하고 일부 장애에 더 취약하다.

#### 4.1 선형성(Linearizability)

강한 일관성의 가장 엄격한 형태가 선형성이다. "데이터의 복사본이 하나만 존재하고, 모든 연산이 원자적으로 실행되는 것처럼 보이게 하는 것"이 핵심 정의이다.

선형성이 필요한 대표적 상황은 세 가지다.

| 상황 | 이유 |
|------|------|
| 리더 선출 / 분산 잠금 | 두 노드가 동시에 리더가 되면 Split Brain 발생 |
| 유일성 제약 | 사용자명, 좌석 예약 등 중복 방지 |
| 크로스 채널 타이밍 | 메시지 큐가 파일 복제보다 빠르면 데이터 불일치 |

복제 방식별 선형성 지원 여부는 다음과 같다.

| 복제 방식 | 선형성 | 조건 |
|----------|--------|------|
| Single-Leader | 가능 | 리더에서 읽기, Split Brain 방지 필수 |
| 합의 알고리즘 (Raft/Paxos) | 가능 | 읽기도 쿼럼 확인 필요 |
| Multi-Leader | 불가 | 비동기 복제, 충돌 존재 |
| Leaderless | 불가 | 쿼럼 조건 만족해도 타이밍 문제 |

#### 4.2 CAP 정리의 실제 의미

흔히 "C, A, P 중 2개를 선택"이라고 알려져 있지만, 이는 부정확하다. 올바른 해석은 이렇다.

- 네트워크 파티션(P)은 **피할 수 없는 현실**이다
- 파티션 발생 시, **선형성(C) 또는 가용성(A)** 중 하나를 포기해야 한다
- 정상 상태에서는 둘 다 제공할 수 있다

CAP의 한계도 알아야 한다. CAP는 선형성만 다루고, 약한 일관성 모델은 설명하지 못한다. 또한 네트워크 지연이나 노드 장애 같은 현실적 문제를 포함하지 않는다. PACELC가 더 현실적인 프레임워크인데, 정상 상태(Else)에서도 **지연(Latency) vs 일관성(Consistency)** 선택이 있다는 점을 추가한다.

```
Partition 시: Availability vs Consistency
Else(정상):   Latency vs Consistency
```

### 5. 합의(Consensus) - 분산 노드가 동의하는 방법

합의란 "여러 노드가 하나의 값에 동의하는 것"이다. 단일 노드에서는 사소한 문제이지만, 노드가 장애를 겪을 수 있는 분산 환경에서는 놀랍도록 어렵다.

합의가 필요한 대표적 상황은 리더 선출, 선형화 가능한 ID 생성, Compare-and-Set(CAS) 연산이다. 이들은 모두 동등한 문제로, 하나의 해법을 다른 것으로 변환할 수 있다.

#### 5.1 합의의 네 가지 속성

| 속성 | 설명 |
|------|------|
| Uniform Agreement | 두 노드가 다른 값을 결정할 수 없음 |
| Integrity | 한번 결정하면 번복 불가 |
| Validity | 결정된 값은 누군가가 제안한 값이어야 함 |
| Termination | 정상 노드는 결국 결정에 도달해야 함 (과반수 정상 가정) |

FLP 불가능성 결과(1985)에 따르면, 노드가 크래시할 수 있는 환경에서 항상 합의에 도달하는 알고리즘은 이론적으로 존재하지 않는다. 하지만 이는 순수 비동기 모델과 결정적 알고리즘에 한정된 이야기이며, 타임아웃(부분 동기 모델)이나 난수를 도입하면 실전에서는 대부분 합의에 도달한다.

#### 5.2 Raft와 Paxos

주요 합의 알고리즘들의 공통 구조는 "Single-Leader + 자동 페일오버"이다.

1. **Epoch Number**: 각 epoch 내에서 리더는 하나뿐이다. 새 리더 선출 시 epoch가 증가하며, 두 리더가 충돌하면 높은 epoch가 승리한다. Paxos에서는 ballot, Raft에서는 term, Zab에서는 epoch라고 부른다.

2. **두 번의 투표**: 먼저 리더 선출 투표를 하고, 그 후 리더의 제안에 대해 로그 항목 추가 투표를 한다.

3. **쿼럼 겹침**: 리더 선출 쿼럼과 제안 투표 쿼럼이 반드시 겹치도록 보장하여, 더 높은 epoch가 선출됐는지 감지할 수 있다.

2PC(Two-Phase Commit)와의 차이를 비교하면 합의 알고리즘의 강점이 드러난다.

| 특성 | 2PC | 합의 알고리즘 |
|------|-----|---------------|
| 코디네이터 | 고정 (단일 장애점) | 아무 노드나 선출 시작 가능 |
| 투표 | 모든 참여자 동의 필요 | 쿼럼(과반수) 동의면 충분 |
| 장애 시 | 코디네이터 죽으면 블로킹 | 새 리더 자동 선출 |

#### 5.3 공유 로그(Total Order Broadcast)

합의와 동등한 또 다른 표현이 공유 로그이다. 모든 노드가 **같은 순서로 같은 항목**을 보는 로그를 구현할 수 있다면, 이는 합의를 해결한 것과 같다.

공유 로그의 핵심 활용 사례는 State Machine Replication이다. 로그 항목을 결정적 연산으로 정의하고, 모든 복제본이 같은 순서로 실행하면 동일한 상태에 도달한다. 이것이 Raft 기반 시스템(etcd, CockroachDB)의 작동 원리이다.

#### 5.4 코디네이션 서비스

ZooKeeper, etcd, Consul은 합의 알고리즘을 내장한 코디네이션 서비스이다. 직접 합의 알고리즘을 구현하는 것은 어렵고 위험하므로, 이러한 서비스를 활용해 리더 선출, 분산 잠금, 설정 관리, 서비스 디스커버리를 구현한다.

3~5개 노드로 구성된 코디네이션 서비스가 수천 개 노드의 분산 시스템을 조정할 수 있다는 점이 핵심 가치이다. Kubernetes가 etcd를 사용하고, Spark/Flink가 ZooKeeper를 사용하는 이유이기도 하다.

### 6. 복제 + 샤딩 + 합의의 조합

실제 분산 데이터베이스는 이 세 가지를 조합해서 사용한다.

```
┌──────────────────────────────────────────────────────┐
│                  분산 DB 아키텍처                       │
│                                                        │
│  샤딩: 데이터를 N개 샤드로 분할                        │
│     ↓                                                  │
│  복제: 각 샤드를 M개 복제본으로 복제                   │
│     ↓                                                  │
│  합의: 각 샤드 그룹 내에서 리더 선출 + 로그 복제       │
│     ↓                                                  │
│  코디네이션: 샤드-노드 매핑 관리 (etcd/ZooKeeper)     │
└──────────────────────────────────────────────────────┘
```

CockroachDB를 예로 들면, 데이터를 Range(샤드)로 나누고, 각 Range를 3개 복제본으로 복제하며, 복제본 간 합의는 Raft를 사용한다. TiDB도 유사하게 TiKV의 Region(샤드)을 Raft로 복제한다.

Ch.13에서 다룰 PostgreSQL 클러스터링과 Ch.14의 MariaDB 클러스터링도 이 세 가지 이론 위에 구축된다. Patroni + etcd는 합의 알고리즘 기반 리더 선출을, Galera Cluster는 가상 동기식 Multi-Master 복제를 구현하는 방식이 핵심 차이인데, 그 배경이 바로 이 장에서 다룬 이론이다.

## 핵심 정리

| 개념 | 한 줄 요약 |
|------|-----------|
| Single-Leader | 하나의 리더가 쓰기 담당, 팔로워는 읽기 전용 - 가장 단순하지만 Failover가 어렵다 |
| Multi-Leader | 여러 리더가 독립 쓰기 가능 - 지역 분산에 유리하지만 충돌 해결이 필수이다 |
| Leaderless | 쿼럼 기반 읽기/쓰기 - Failover 불필요하지만 강한 일관성 보장이 어렵다 |
| Key-range 샤딩 | 범위 쿼리에 효율적이나 핫스팟 위험이 있다 |
| Hash 샤딩 | 균등 분산에 유리하나 범위 쿼리가 비효율적이다 |
| Consistent Hashing | 노드 변경 시 최소한의 키만 이동하는 해시 분배 알고리즘이다 |
| CAP 정리 | 파티션 시 일관성과 가용성 중 선택 - "3개 중 2개"가 아니다 |
| 선형성 | 모든 연산이 원자적으로 실행되는 것처럼 보이는 가장 강한 일관성이다 |
| Raft/Paxos | Epoch + 쿼럼 기반 리더 선출과 로그 복제로 합의를 달성한다 |
| Total Order Broadcast | 모든 노드가 같은 순서로 같은 로그를 보는 것 - 합의와 동등하다 |
| 코디네이션 서비스 | ZooKeeper/etcd - 합의를 직접 구현하지 않고 활용하는 인프라이다 |
