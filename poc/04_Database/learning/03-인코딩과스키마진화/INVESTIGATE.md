## Q1. Protocol Buffers vs Avro, 어떤 상황에서 어느 것을 선택하는가?

### 왜 이 질문이 중요한가
마이크로서비스 간 통신이나 대용량 데이터 파이프라인을 설계할 때, 인코딩 포맷 선택은 시스템 전체의 성능과 운영 복잡도에 영향을 미친다. 두 포맷의 실질적 차이를 이해해야 근거 있는 선택이 가능하다.

### 답변

**Protocol Buffers가 유리한 경우:**
- gRPC 기반 마이크로서비스 통신: gRPC는 Protobuf를 기본 직렬화 포맷으로 사용하며, 서비스 정의(.proto 파일)에서 클라이언트/서버 코드를 자동 생성한다
- 정적 타입 언어 중심 프로젝트(Java, Go, C++): 코드 생성을 통한 컴파일 타임 타입 체크가 강력하다
- 스키마가 사전에 잘 정의된 환경: 태그 번호 관리가 명확한 규칙으로 운영된다

**Avro가 유리한 경우:**
- Hadoop/Spark 기반 데이터 파이프라인: Avro Object Container File은 스키마를 파일 헤더에 포함하여 자기 서술적(self-describing)이다
- DB 스키마에서 자동 생성이 필요한 ETL: 태그 번호 없이 필드명만으로 매칭하므로 DB 스키마 변경에 자동 대응한다
- 동적 타입 언어 중심 프로젝트: 스키마를 런타임에 해석하므로 코드 생성 없이 사용 가능하다

**크기 비교:**
동일 데이터에서 Avro가 약 1바이트 더 작다(32 vs 33). 이 차이는 레코드 수가 수억 건일 때 의미가 있을 수 있지만, 포맷 선택의 주된 기준은 아니다.

**스키마 진화 방식의 차이:**
- Protobuf: 태그 번호로 필드를 식별하므로, 필드명은 자유롭게 바꿀 수 있지만 태그 번호는 절대 변경/재사용 불가
- Avro: 필드명으로 매칭하므로, 필드명 변경은 aliases를 통해 처리하고, 필드 추가/삭제는 기본값 설정으로 호환성 유지

### 실무 적용
마이크로서비스 간 동기 통신에는 gRPC + Protobuf, 비동기 이벤트 스트리밍(Kafka)에는 Avro + Schema Registry가 가장 검증된 조합이다. 두 포맷을 한 시스템에서 혼용하는 것도 가능하다.

## Q2. 스키마 진화 전략: 무중단 배포에서 호환성을 어떻게 보장하는가?

### 왜 이 질문이 중요한가
프로덕션 시스템에서 스키마 변경은 피할 수 없다. 잘못된 스키마 변경은 서비스 장애로 이어질 수 있으므로, 안전한 진화 전략을 체계적으로 이해해야 한다.

### 답변

**안전한 스키마 변경 (호환성 유지):**
1. 새 optional 필드 추가 (기본값 포함)
2. optional 필드 삭제 (단, 태그/필드명은 영구 예약)
3. 필드명 변경 (Protobuf에서는 태그 보존 시 가능)

**위험한 스키마 변경 (호환성 깨짐 가능):**
1. required 필드 추가 → 옛 코드가 이 필드를 보내지 않아 에러 발생
2. 필드 타입 변경(int32 → string) → 디코딩 실패
3. 태그 번호 재사용(Protobuf) → 옛 데이터가 잘못 해석됨

**Confluent Schema Registry의 호환성 모드:**

| 모드 | 의미 | 사용 시점 |
|------|------|----------|
| BACKWARD | 새 스키마로 옛 데이터를 읽을 수 있음 | Consumer 먼저 업그레이드 |
| FORWARD | 옛 스키마로 새 데이터를 읽을 수 있음 | Producer 먼저 업그레이드 |
| FULL | 양방향 호환 | 독립 배포 필요 시 (가장 안전) |
| NONE | 검증 없음 | 개발 환경 전용 |

**안전한 배포 순서 (Kafka 예시):**
```
1. Consumer를 새 스키마 호환 버전으로 업그레이드 (BACKWARD 호환)
2. Schema Registry에 새 스키마 등록 (호환성 자동 검증)
3. Producer를 새 스키마로 업그레이드
4. 충분한 시간 후 옛 스키마 데이터가 모두 소비되면 완료
```

### 실무 적용
Schema Registry의 호환성 모드를 FULL로 설정하는 것이 가장 안전하다. 새 필드는 항상 optional + 기본값으로 추가하고, 삭제된 필드의 태그 번호는 `reserved` 키워드로 영구 예약한다. CI/CD 파이프라인에 스키마 호환성 검증 단계를 추가하면, 비호환 변경이 프로덕션에 배포되는 것을 차단할 수 있다.

## Q3. Zero-Downtime 마이그레이션은 어떻게 수행하는가?

### 왜 이 질문이 중요한가
데이터베이스 스키마 변경은 서비스 중단의 대표적인 원인이다. 대형 테이블의 ALTER TABLE은 수 시간이 걸릴 수 있으며, 그 동안 테이블이 잠길 수 있다. 무중단 마이그레이션 기법은 실무에서 반드시 알아야 하는 기술이다.

### 답변

**단계적 마이그레이션 (Expand-Contract 패턴):**

```
Phase 1: Expand (확장)
- 새 컬럼 추가 (nullable, 기본값 포함)
- 새 코드는 양쪽 컬럼 모두 쓰기 (dual-write)
- 옛 코드는 옛 컬럼만 사용 (영향 없음)

Phase 2: Migrate (이전)
- 백그라운드 잡으로 옛 데이터를 새 컬럼으로 복사
- 점진적으로 배치 처리 (서비스 영향 최소화)

Phase 3: Contract (축소)
- 모든 코드가 새 컬럼을 사용하는지 확인
- 옛 컬럼 삭제 (또는 deprecated 마킹)
```

**PostgreSQL에서의 안전한 마이그레이션:**
- `ALTER TABLE ... ADD COLUMN ... DEFAULT NULL`: 즉시 완료(메타데이터만 변경)
- `ALTER TABLE ... ADD COLUMN ... DEFAULT 'value'`: PostgreSQL 11+에서 즉시 완료
- `ALTER TABLE ... ALTER COLUMN ... TYPE`: 대형 테이블에서 주의 필요 (전체 테이블 재작성)
- `CREATE INDEX CONCURRENTLY`: 테이블 잠금 없이 인덱스 생성

**대형 테이블 타입 변경이 필요한 경우:**
1. 새 컬럼을 원하는 타입으로 추가
2. 트리거로 기존 컬럼 변경 시 새 컬럼도 동기화
3. 백그라운드 잡으로 기존 데이터 복사
4. 애플리케이션이 새 컬럼을 사용하도록 전환
5. 옛 컬럼 삭제

### 실무 적용
마이그레이션 도구(Flyway, Alembic, Rails Migrations)를 사용하되, 각 마이그레이션 스크립트가 서비스 무중단 상태에서 실행 가능한지 반드시 검증해야 한다. "이 DDL이 테이블 락을 잡는가?"를 항상 확인하고, 대형 테이블에 대한 마이그레이션은 별도 운영 계획을 세워야 한다.

## Q4. RPC의 본질적 한계와 gRPC가 이를 어떻게 보완하는가?

### 왜 이 질문이 중요한가
마이크로서비스 아키텍처에서 서비스 간 통신 방식 선택은 시스템 안정성에 직접적인 영향을 미친다. REST vs gRPC 논쟁의 본질을 이해하려면, RPC 자체의 한계부터 파악해야 한다.

### 답변

**RPC의 본질적 한계 (로컬 함수 호출과의 차이):**
1. 네트워크 실패: 요청이 도착했는지, 응답이 돌아오는지 보장되지 않는다
2. 타임아웃: 성공/실패를 알 수 없는 상태가 발생한다
3. 재시도 시 중복: 같은 요청이 두 번 실행될 수 있다
4. 지연 변동: 밀리초 ~ 수 초까지 불예측적이다
5. 직렬화 비용: 모든 파라미터를 바이트로 변환해야 한다

**gRPC가 제공하는 보완:**
- Protocol Buffers 기반 직렬화: JSON보다 2~3배 작고, 역직렬화 속도가 빠르다
- HTTP/2 기반: 멀티플렉싱(하나의 커넥션에 여러 요청), 헤더 압축, 서버 스트리밍 지원
- 양방향 스트리밍: 실시간 데이터 전송에 적합하다
- Deadline/Timeout 전파: 요청 체인 전체의 타임아웃을 제어한다
- 코드 생성: .proto 파일에서 클라이언트/서버 코드 자동 생성

**하지만 gRPC도 해결하지 못하는 것:**
- 네트워크 파티션 자체는 해결 불가 → 재시도 + 멱등성이 필수
- 브라우저 직접 호출이 제한적 → gRPC-Web이나 REST 게이트웨이 필요
- 디버깅: 바이너리 프로토콜이므로 curl로 직접 테스트하기 어렵다

### 실무 적용
외부 API(브라우저, 모바일)에는 REST + JSON, 내부 서비스 간 통신에는 gRPC + Protobuf가 가장 일반적인 조합이다. 두 방식을 동시에 지원하려면 gRPC Gateway(envoy proxy)로 REST ↔ gRPC 변환을 자동화할 수 있다.
