# Ch.03 - 인코딩과 스키마 진화

> **이론 매핑**: `docs/04_Database/01_기본개념/03_인코딩과_진화.md`

---

## 핵심 요약
> 애플리케이션은 끊임없이 변하고, 데이터 형식도 함께 진화해야 한다. 인코딩은 메모리 내 데이터를 네트워크나 디스크에 저장할 수 있는 바이트 시퀀스로 변환하는 과정이다. JSON 같은 텍스트 포맷은 사람이 읽기 쉽지만, Protocol Buffers나 Avro 같은 바이너리 스키마 기반 포맷이 더 컴팩트하고 명확한 호환성 규칙을 제공한다. 시스템의 무중단 업그레이드를 위해서는 하위 호환성과 상위 호환성을 모두 이해하고 유지해야 한다.

---

## 학습 목표
1. 인코딩(직렬화)과 디코딩(역직렬화)의 개념과 필요성을 설명할 수 있다
2. 하위 호환성(Backward)과 상위 호환성(Forward)의 차이를 구분할 수 있다
3. JSON, Protocol Buffers, Avro의 인코딩 방식과 크기 차이를 비교할 수 있다
4. 스키마 진화(Schema Evolution)가 어떻게 호환성을 유지하는지 설명할 수 있다
5. 데이터 흐름의 세 가지 모드(DB, RPC, 메시지 브로커)를 이해할 수 있다

---

## 본문

### 1. 인코딩의 필요성

프로그램은 데이터를 두 가지 형태로 다룬다. 메모리 내에서는 객체, 리스트, 해시맵 등 CPU 접근에 최적화된 포인터 기반 구조를 사용한다. 하지만 데이터를 파일에 저장하거나 네트워크로 전송하려면, 포인터가 의미를 잃으므로 자기 완결적인(self-contained) 바이트 시퀀스로 변환해야 한다.

이 변환 과정을 인코딩(encoding, 직렬화, marshalling)이라 하고, 반대 과정을 디코딩(decoding, 역직렬화, unmarshalling)이라 한다.

```
메모리: Objects, Structs, Lists (포인터 기반)
           ↓ 인코딩 / ↑ 디코딩
바이트: 0x83 0xa8 0x75 0x73 0x65 0x72 ... (파일/네트워크용)
```

언어 내장 직렬화(Java Serializable, Python pickle)는 사용하지 않는 것이 원칙이다. 보안 취약점(원격 코드 실행), 언어 종속성, 성능 저하, 버전 호환성 미보장 등 문제가 많기 때문이다.

---

### 2. 호환성: 하위 호환성과 상위 호환성

분산 시스템에서 모든 노드를 동시에 업데이트하는 것은 불가능하다. 롤링 업그레이드(rolling upgrade) 중에는 새 버전과 옛 버전의 코드가 동시에 실행된다.

| 호환성 유형 | 의미 | 구현 난이도 |
|------------|------|-----------|
| 하위 호환성 (Backward) | 새 코드가 옛 데이터를 읽을 수 있음 | 비교적 쉬움 |
| 상위 호환성 (Forward) | 옛 코드가 새 데이터를 읽을 수 있음 | 더 어려움 |

```
Rolling Upgrade 중:
Node A (v2.0) ──→ 공유 DB ←── Node C (v1.0)
Node B (v2.0) ──→         ←── Node D (v1.0)

v2.0이 새 필드를 추가한 데이터를 저장 → v1.0이 읽을 수 있어야 함 (상위 호환성)
v1.0이 저장한 옛 데이터를 v2.0이 읽어야 함 (하위 호환성)
```

상위 호환성에서 주의할 함정이 있다. 옛 코드가 새 필드를 모르는 상태에서 데이터를 읽고 다시 저장하면, 알지 못하는 새 필드가 누락될 수 있다. 해결책은 디코딩 시 알 수 없는 필드를 보존(preserve)하는 로직을 구현하는 것이다.

---

### 3. 인코딩 포맷 비교

동일한 데이터를 다양한 포맷으로 인코딩하면 크기 차이가 분명하다.

```json
{
    "userName": "Martin",
    "favoriteNumber": 1337,
    "interests": ["daydreaming", "hacking"]
}
```

```
JSON (텍스트)          ████████████████████████████  81 bytes
MessagePack (바이너리) ██████████████████████████   66 bytes
Protocol Buffers       ████████████████             33 bytes
Avro                   ███████████████              32 bytes
```

JSON이 81바이트인 반면 Protocol Buffers와 Avro는 32~33바이트로, 약 60% 크기 절감이 가능하다. 이 차이는 수백만 건의 레코드를 전송하거나 저장할 때 대역폭과 저장 비용에 직접 반영된다.

#### 3.1 JSON의 한계

JSON은 호환성과 가독성이 강점이지만 몇 가지 한계가 있다:
- 정수와 부동소수점을 구분하지 않는다
- JavaScript에서 2^53을 초과하는 정수는 정밀도가 손실된다
- 바이너리 데이터를 Base64로 인코딩하면 크기가 33% 증가한다
- 스키마가 선택적이므로 데이터 검증이 느슨하다

외부 API나 사람이 읽어야 하는 데이터에는 JSON이 적합하지만, 내부 서비스 간 통신이나 대용량 데이터 파이프라인에서는 바이너리 포맷이 효율적이다.

---

### 4. Protocol Buffers

Protocol Buffers(Protobuf)는 Google이 개발한 바이너리 인코딩 포맷이다. 스키마 정의에서 각 필드에 **태그 번호**를 부여하고, 인코딩 시 필드 이름 대신 태그 번호만 사용하여 공간을 절약한다.

```protobuf
message Person {
    string user_name = 1;       // 태그 1
    int64 favorite_number = 2;  // 태그 2
    repeated string interests = 3;  // 태그 3
}
```

인코딩된 바이트에는 필드 이름이 없고, 태그 번호와 타입 정보, 값만 존재한다. 작은 정수는 varint 인코딩으로 1~2바이트로 표현된다(1337 = 2바이트).

#### 스키마 진화 규칙

| 작업 | 하위 호환 | 상위 호환 | 주의사항 |
|------|----------|----------|---------|
| 필드 추가 (기본값 있음) | O | O | 새 태그 번호 사용 |
| 필드 삭제 | O | O | 태그 번호 재사용 금지 |
| 필드명 변경 | O | O | 태그만 보존되면 OK |
| 타입 변경 | 주의 | 주의 | 절삭(truncation) 위험 |

태그 번호는 집 주소와 같다. 필드 이름이 바뀌어도(이사) 태그 번호(주소)가 그대로면 데이터는 올바르게 전달된다. 하지만 태그 번호를 바꾸면 과거의 모든 데이터가 손상되므로, 한 번 할당된 태그 번호는 절대 변경하거나 재사용하지 않는다.

---

### 5. Avro

Avro는 Apache에서 개발한 바이너리 인코딩 포맷으로, Hadoop 생태계에서 널리 사용된다. Protocol Buffers와의 가장 큰 차이는 **태그 번호가 없다**는 점이다.

```avro
record Person {
    string               userName;
    union { null, long } favoriteNumber = null;
    array<string>        interests;
}
```

인코딩 시 필드 순서대로 값만 나열하므로, 디코딩하려면 인코딩에 사용된 스키마(Writer's Schema)를 알아야 한다.

#### Writer's Schema vs Reader's Schema

Avro의 핵심 개념은 Writer's Schema(데이터 쓸 때 사용한 스키마)와 Reader's Schema(데이터 읽을 때 사용하는 스키마)를 분리하는 것이다.

```
Writer's Schema              Reader's Schema
├── userName: str  ─────────  userName: str    (매칭)
├── favoriteNum: i64 ───────  favoriteNum: i64 (매칭)
├── interests: arr ─────────  interests: arr   (매칭)
├── photoURL: str  ─────X    (없음)            → Writer만 있으면 무시
│                            email: str        → Reader만 있으면 기본값
```

필드명으로 매칭하므로 필드 순서가 달라도 괜찮다. Writer에만 있는 필드는 Reader가 무시하고, Reader에만 있는 필드는 기본값으로 채운다.

#### Avro의 강점: 동적 스키마 생성

DB 테이블 스키마에서 자동으로 Avro 스키마를 생성할 수 있다. Protocol Buffers는 태그 번호를 수동으로 관리해야 하지만, Avro는 필드명 매칭이므로 이 과정이 자동화된다. Hadoop에서 DB 데이터를 추출(ETL)할 때 이 특성이 유용하다.

---

### 6. 스키마 기반 인코딩의 장점

바이너리 스키마 기반 인코딩이 JSON보다 유리한 이유는 네 가지다:

| 장점 | 설명 |
|------|------|
| 컴팩트 | 필드명을 생략하여 JSON보다 작다 |
| 문서화 | 스키마 자체가 API 문서 역할을 한다 |
| 호환성 검증 | 배포 전 스키마 변경의 호환성을 자동 검사할 수 있다 |
| 코드 생성 | 정적 타입 언어에서 컴파일 타임 타입 체크가 가능하다 |

Confluent Schema Registry를 사용하면, Kafka 프로듀서가 메시지를 보낼 때 자동으로 스키마 호환성을 검증한다. 비호환 스키마 변경은 배포 전에 차단되므로, 런타임 오류를 사전에 방지할 수 있다.

### 7. 데이터 흐름의 세 가지 모드

#### 7.1 데이터베이스를 통한 흐름

데이터베이스에는 수년에 걸쳐 다양한 스키마 버전으로 인코딩된 데이터가 공존한다. "Data outlives code"라는 표현처럼, 5년 전에 저장된 데이터를 오늘의 코드로 읽어야 하는 상황이 일상적이다. 아카이브/백업 시에는 모든 데이터를 최신 스키마로 통일하여 저장하는 것이 바람직하다.

#### 7.2 서비스 간 통신 (REST/RPC)

마이크로서비스 간 통신에서는 서버가 먼저 업그레이드되고 클라이언트가 나중에 따라오는 것이 일반적이다.
- 요청(Request): 하위 호환성 필요 (새 서버가 옛 클라이언트의 요청을 이해)
- 응답(Response): 상위 호환성 필요 (옛 클라이언트가 새 서버의 응답을 이해)

RPC는 로컬 함수 호출과 달리 네트워크 실패, 타임아웃, 지연 시간 변동이 있다. 재시도 시 중복 실행을 방지하기 위해 멱등성(idempotency) 설계가 필수다.

#### 7.3 메시지 브로커를 통한 흐름

메시지 브로커(Kafka, RabbitMQ)는 Producer와 Consumer를 디커플링한다.

```
Queue 패턴 (1:1):  Producer → Queue → Consumer (하나만 수신)
Topic 패턴 (1:N):  Publisher → Topic → Subscriber 1, 2, 3 (모두 수신)
```

메시지 브로커의 장점은 버퍼링(수신자 장애 시 메시지 보관), 재전송(크래시 시 자동 재전송), 디커플링(송신자와 수신자가 서로 몰라도 됨)이다. 메시지 인코딩에도 스키마 진화 규칙이 적용되므로, Producer와 Consumer의 독립적인 배포가 가능하다.

### 8. 워크플로우 엔진과 Durable Execution

Temporal 같은 워크플로우 엔진은 여러 단계의 비즈니스 로직(사기 탐지 → 카드 차감 → 은행 입금)을 하나의 워크플로우로 정의하고, 각 단계의 결과를 WAL에 기록한다.

장애가 발생하면 WAL에서 상태를 복구하되, 이미 완료된 RPC는 재실행하지 않고 저장된 결과를 반환한다. 이를 통해 "exactly-once" 시멘틱에 가까운 실행을 보장한다.

단, 워크플로우 코드는 결정론적(deterministic)이어야 한다. 랜덤 값이나 시스템 시계를 직접 사용하면 재생 시 다른 결과가 나올 수 있으므로, 워크플로우 엔진이 제공하는 API를 통해 사용해야 한다.

## 핵심 정리
| 개념 | 한 줄 요약 |
|------|-----------|
| 인코딩/디코딩 | 메모리 내 구조를 바이트로, 바이트를 메모리 내 구조로 변환하는 과정이다 |
| 하위 호환성 | 새 코드가 옛 데이터를 읽을 수 있음 (비교적 구현이 쉽다) |
| 상위 호환성 | 옛 코드가 새 데이터를 읽을 수 있음 (알 수 없는 필드 보존이 핵심) |
| Protocol Buffers | 태그 번호 기반으로 필드명 변경에 자유롭고 컴팩트하다 |
| Avro | 필드명 기반으로 동적 스키마 생성에 유리하고 가장 컴팩트하다 |
| 스키마 레지스트리 | 프로덕션에서 스키마 버전을 관리하고 호환성을 자동 검증한다 |
| Durable Execution | WAL에 RPC 결과를 기록하여 장애 복구 시 exactly-once에 가까운 보장을 제공한다 |
