# Go GC vs Java GC 학습 문서

## 1. 한 줄 요약
- Go GC는 "단순 운영 + 낮은 지연시간 기본값"에 강점이 있고, Java GC는 "다양한 수집기 + 깊은 튜닝"에 강점이 있다.

## 2. 핵심 비교

| 항목 | Go | Java |
|------|----|------|
| 기본 철학 | 기본값 중심, 운영 단순화 | 워크로드별 선택/튜닝 최적화 |
| GC 선택지 | 사실상 런타임 기본 전략 중심 | Serial, Parallel, G1, ZGC, Shenandoah 등 다양 |
| 튜닝 난이도 | 낮음 (`GOGC`, `GOMEMLIMIT` 중심) | 높음 (힙/세대/수집기별 옵션 다수) |
| 지연시간 특성 | 짧은 pause 유지에 집중 | 수집기 선택에 따라 다양 |
| 처리량 특성 | 안정적 기본 성능 | 튜닝 시 매우 높은 처리량 가능 |
| 운영 복잡도 | 낮음 | 중간~높음 |

## 3. 메모리 수집 모델 관점
- Go:
  - 동시성(concurrent) 중심의 mark/sweep 계열.
  - 애플리케이션 정지 시간을 줄이는 방향으로 설계.
- Java:
  - 전통적으로 세대 기반(Young/Old) 전략이 강함.
  - G1/ZGC/Shenandoah 등 목적이 다른 GC를 워크로드에 맞게 선택 가능.

## 4. 튜닝 포인트
- Go에서 먼저 보는 값:
  - `GOGC`: GC 빈도/메모리 사용량 균형 조절.
  - `GOMEMLIMIT`: 메모리 상한 기반 제어.
- Java에서 먼저 보는 값:
  - `-Xms`, `-Xmx`: 힙 크기 경계.
  - GC 선택: `G1`, `ZGC`, `Shenandoah`, `Parallel` 등.
  - 서비스 목표(SLO): pause time 우선인지, throughput 우선인지 먼저 결정.

## 5. 실무 선택 가이드
- Go가 유리한 경우:
  - 빠른 개발/배포, 단순 운영이 중요할 때.
  - GC 튜닝 인력 없이 안정적인 지연시간이 필요할 때.
- Java가 유리한 경우:
  - 대규모 트래픽에서 성능 최적화 여지가 중요할 때.
  - 서비스 특성에 맞춘 세밀한 GC 전략이 필요할 때.

## 6. 면접/학습 체크 질문
1. "낮은 지연시간"과 "높은 처리량"이 충돌할 때 어떤 지표를 우선할 것인가?
2. Go에서 `GOGC`를 낮추면 메모리와 CPU는 어떻게 바뀌는가?
3. Java에서 G1 대신 ZGC를 고려해야 하는 서비스 특성은 무엇인가?
4. 운영팀 역량(튜닝 가능 인력)이 GC 선택에 미치는 영향은 무엇인가?

## 7. 결론
- Go: 기본값 품질이 높고 운영이 단순하다.
- Java: 선택지와 튜닝 폭이 넓어 고성능 최적화에 유리하다.
- 따라서 GC 비교의 핵심은 "언어 우열"이 아니라 "서비스 목표와 운영 역량의 적합성"이다.
