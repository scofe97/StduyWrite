# Spring Boot + Redpanda 구현: 인프로세스 DAG 실행기

---

> 2-토픽 패턴이 분산 시스템 레벨의 DAG 오케스트레이션이라면, 인프로세스(in-process) <u>DAG 실행기는 **단일 JVM 안에서 DAG 노드를 병렬 처리**하는 접근법</u>입니다. 별도 Dispatcher 프로세스 없이 Consumer 하나가 위상 정렬(topological sort)과 `CompletableFuture` 기반으로 의존 관계를 해소하며 실행합니다.

## DAGExecutor 설계

`DAGExecutor`는 노드 목록과 의존성 맵을 입력받아 위상 정렬 순서로 실행합니다. 의존성이 없는 노드들은 동시에 실행하고, 각 노드의 완료 시점에 의존하는 후속 노드를 해제합니다:

```java
/**
 * 인프로세스 DAG 실행기 — 노드 목록과 의존성 맵을 받아 위상 정렬 순서로 실행한다.
 * CompletableFuture 체이닝으로 의존 관계를 표현하며, 의존성 없는 노드는 동시에 실행한다.
 */
@Component
public class DAGExecutor {

    private final ExecutorService threadPool;

    public DAGExecutor(@Value("${pipeline.dag.thread-pool-size:4}") int poolSize) {
        this.threadPool = Executors.newFixedThreadPool(poolSize);
    }

    public CompletableFuture<Map<String, Object>> execute(
            List<DAGNode> nodes
            , Map<String, List<String>> dependencies) {

        // 노드별 Future를 저장 — 후속 노드가 선행 노드의 Future를 참조한다
        Map<String, CompletableFuture<Object>> futures = new HashMap<>();

        for (DAGNode node : nodes) {
            List<String> deps = dependencies.getOrDefault(node.getId(), List.of());

            CompletableFuture<Object> future;
            if (deps.isEmpty()) {
                // 의존성 없음 → 즉시 실행
                future = CompletableFuture.supplyAsync(node::execute, threadPool);
            } else {
                // 의존성 있음 → 선행 노드 전부 완료 후 실행
                CompletableFuture<?>[] depFutures = deps.stream()
                        .map(futures::get)
                        .toArray(CompletableFuture[]::new);

                future = CompletableFuture.allOf(depFutures)
                        .thenApplyAsync(v -> node.execute(), threadPool);
            }
            futures.put(node.getId(), future);
        }

        // 모든 노드 완료 후 결과를 Map<nodeId, result>로 수집한다
        return CompletableFuture.allOf(futures.values().toArray(new CompletableFuture[0]))
                .thenApply(v -> futures.entrySet().stream()
                        .collect(Collectors.toMap(
                                Map.Entry::getKey
                                , e -> e.getValue().join()
                        )));
    }
}
```

이 구현의 핵심은 두 가지입니다. 첫째, 노드 등록 순서를 선행 노드 먼저로 유지하면 위상 정렬 없이도 `CompletableFuture` 체이닝만으로 의존성을 표현할 수 있습니다. 둘째, `thenApplyAsync`에 `threadPool`을 명시하여 모든 노드가 동일한 스레드풀에서 실행됨을 보장합니다.



## pause/resume 배리어 통합

인프로세스 DAG 실행기를 Kafka Consumer와 통합할 때 핵심 과제는 **묶음 경계 감지**입니다. `batchId`(또는 `groupId`)가 변경되는 시점을 감지하면 현재 묶음의 모든 DAG 노드가 완료될 때까지 새 메시지 소비를 차단해야 합니다. `pause()`/`resume()` API가 이 역할을 수행합니다:

```java
/**
 * pause/resume 배리어 기반 DAG Consumer — batchId 변경 시점(묶음 경계)을 감지하여
 * 현재 묶음의 모든 노드가 완료될 때까지 새 메시지 소비를 차단한다.
 */
@Component
@RequiredArgsConstructor
public class BatchBarrierDAGConsumer {

    private final KafkaListenerEndpointRegistry registry;
    private final DAGExecutor dagExecutor;
    private final ExecutorService threadPool;

    private String currentBatchId = null;
    private final List<CompletableFuture<Object>> pendingNodes = new ArrayList<>();

    private static final String LISTENER_ID = "batch-barrier-dag-listener";

    @KafkaListener(
        id = LISTENER_ID,
        topics = PipelineTopics.TASKS,
        groupId = "batch-barrier-dag-consumer",
        containerFactory = "batchBarrierContainerFactory"
    )
    public void consume(ConsumerRecord<String, byte[]> record, Acknowledgment ack) {
        String batchId = extractBatchId(record);

        // 묶음 경계 감지: 이전 묶음과 다른 batchId가 도착하면 배리어를 건다
        if (currentBatchId != null && !currentBatchId.equals(batchId)) {
            pauseConsumer();            // 새 메시지 소비 차단
            awaitCurrentBatch();        // 진행 중인 노드 전부 완료 대기
            ack.acknowledge();          // 이전 묶음 오프셋 커밋
            pendingNodes.clear();
            currentBatchId = batchId;
            resumeConsumer();           // 소비 재개
        }

        if (currentBatchId == null) {
            currentBatchId = batchId;
        }

        // DAG 노드 비동기 실행 — 실패 시 exceptionally 체인으로 DLQ 전송
        CompletableFuture<Object> node = CompletableFuture.supplyAsync(
            () -> processRecord(record), threadPool
        ).exceptionally(ex -> {
            publishToDlq(record, ex);   // DLQ로 원본 메시지 + 에러 정보 전송
            return null;
        });
        pendingNodes.add(node);
    }

    // 현재 묶음의 모든 비동기 노드가 완료될 때까지 블로킹한다
    private void awaitCurrentBatch() {
        CompletableFuture.allOf(pendingNodes.toArray(new CompletableFuture[0])).join();
    }

    private void pauseConsumer() {
        MessageListenerContainer container = registry.getListenerContainer(LISTENER_ID);
        if (container != null && container.isRunning()) {
            container.pause();
        }
    }

    private void resumeConsumer() {
        MessageListenerContainer container = registry.getListenerContainer(LISTENER_ID);
        if (container != null && container.isPauseRequested()) {
            container.resume();
        }
    }
}
```

- `awaitCurrentBatch()`에서 `allOf().join()`이 블로킹하는 동안 `poll()`이 호출되지 않습니다.
-  `pause()`는 비동기 처리 모델(리액티브, 콜백 기반)에서 그 사이에 새 메시지가 소비되는 것을 방지하는 추가 안전 장치로 작동합니다.



## 두 구현의 비교

2-토픽 패턴과 인프로세스 DAG 실행기는 같은 문제를 다른 레벨에서 해결합니다. 둘 중 어느 것을 선택할지는 처리량, 확장성, 운영 복잡도를 종합적으로 고려해야 합니다:

| 속성          | 2-토픽 DAG 패턴                 | 인프로세스 DAG 실행기     |
| ------------- | ------------------------------- | ------------------------- |
| 구현 복잡도   | 높음 (Dispatcher + Worker 분리) | 낮음 (단일 Consumer)      |
| 수평 확장     | 가능 (Worker 인스턴스 추가)     | 제한적 (단일 파티션 필요) |
| 장애 복구     | 브로커 오프셋 기반 재개         | 인메모리 상태 유실 위험   |
| 적합 시나리오 | 고처리량 다단계 오케스트레이션  | 소규모 배치, 단순 DAG     |
| 외부 의존성   | Redpanda (3개 토픽)             | Redpanda (1개 토픽)       |
