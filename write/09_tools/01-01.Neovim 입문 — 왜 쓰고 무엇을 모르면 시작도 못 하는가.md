# Neovim 입문 — 왜 쓰고 무엇을 모르면 시작도 못 하는가
---
> 이 절은 플러그인 한 줄 깔기 전에 읽어야 할 개념을 담는다. 모달 편집과 "연산자+모션"의 사고방식을 모르고 LazyVim을 켜면, 화려한 UI만 보고 IntelliJ보다 불편하다는 결론으로 직행한다.

## 왜 굳이 Neovim인가

IntelliJ는 코드를 *읽고 고치는* 데 강하다. 반면 Neovim은 *손이 키보드에서 떨어지지 않는* 편집 모델을 강제한다. 두 도구는 경쟁 관계가 아니라 다른 축에 있다. 그래서 본 가이드의 최종 결론도 "둘 다 쓴다"이다. 다만 Neovim을 쓰려면 IDE 메뉴 클릭으로 해결하던 동작을 *문장으로 짜는* 사고가 먼저 익숙해져야 한다.

세 가지 이유로 도입한다.

첫째, 터미널 안에서 끝난다. tmux 안에서 git commit, 서버 SSH 안에서 설정 파일 편집, Claude Code 옆 패널 글쓰기 — 환경을 바꾸지 않고 같은 키바인딩을 유지할 수 있다.

둘째, 편집이 언어가 된다. "단어 3개 지우기"가 메뉴 클릭이 아니라 `d3w`라는 *명령*이 된다. 명령은 조합과 반복이 된다. 익숙해지면 IDE의 마우스 동선보다 빠르다.

셋째, 설정이 코드다. `~/.config/nvim`은 git으로 관리할 수 있고 새 맥북에서도 복원된다. IntelliJ는 같은 일을 하지만 XML이 자주 깨진다.

## 모달 편집 — 이걸 모르면 다음 절을 읽어도 의미가 없다

Neovim은 *모드*에 따라 같은 키가 다른 일을 한다. IntelliJ는 어디서든 글자를 누르면 입력이지만, Neovim은 모드가 입력인지 명령인지를 가른다. 모드를 모르면 `j` 한 번 눌렀을 때 커서가 내려가는지 글자가 박히는지 예측할 수 없다.

기본 4개 모드만 우선 외운다.

```mermaid
stateDiagram-v2
    [*] --> Normal
    Normal --> Insert: i / a / o
    Insert --> Normal: Esc
    Normal --> Visual: v / V / Ctrl-v
    Visual --> Normal: Esc
    Normal --> Command: :
    Command --> Normal: Enter / Esc
    note right of Normal
        이동·삭제·복사·붙여넣기
        명령을 짠다
    end note
    note right of Insert
        IDE 처럼 글자를 친다
    end note
```

| 모드 | 진입 키 | 무엇을 하는가 |
|------|---------|-------------|
| **Normal** | `Esc` (시작 시 기본) | 이동·삭제·복사·붙여넣기 — 명령을 *짠다* |
| **Insert** | `i`, `a`, `o` | 글자를 *친다*. IDE처럼 동작하는 유일한 모드 |
| **Visual** | `v`, `V`, `Ctrl-v` | 범위를 *선택*한다. 일반/줄/사각 블록 세 종류 |
| **Command** | `:` | `:w` 저장, `:q` 종료 같은 메타 명령 |

가장 흔한 실수: Insert 모드인 채로 `dd`를 눌러서 `dd`가 그대로 박힘. 항상 `Esc`로 빠져나와 Normal에서 명령을 짠다는 감각이 몸에 붙어야 한다.

## 연산자 + 모션 — Neovim의 진짜 언어

Normal 모드의 명령은 대부분 **연산자(operator) + 모션(motion)** 의 결합이다. "무엇을 할지" + "어디까지" 가 한 문장이 된다.

```
d w   → delete word         (지운다 + 단어 끝까지)
c i "  → change inside "    (바꾼다 + 따옴표 안쪽)
y a p  → yank around paragraph (복사한다 + 문단 전체)
```

연산자(`d`, `c`, `y`, `>`, `<`, `=`)와 모션(`w`, `e`, `b`, `0`, `$`, `gg`, `G`, `f<char>`)을 외우면 조합 수가 폭발한다. 외운 단어 20개로 가능한 명령이 수백 개가 된다는 점이 모달 편집의 핵심이다.

특히 **텍스트 오브젝트**(`i`/`a` + 객체)는 IntelliJ에 정확한 대응이 없는 무기다.

| 명령 | 결과 |
|------|------|
| `ciw` | 커서가 닿은 단어를 통째로 바꾼다 |
| `ci"` | 따옴표 *안쪽* 내용만 바꾼다 (따옴표는 유지) |
| `ca{` | 중괄호 블록 *전체*를 (괄호 포함) 잘라낸다 |
| `dap` | 빈 줄 기준 문단 한 덩어리를 지운다 |
| `vit` | 현재 HTML/XML 태그 안쪽을 비주얼 선택 |

`ci"`는 `"이전 내용"` 안의 문자열만 정확히 갈아끼우는 일을 한 번에 한다. IntelliJ에서는 마우스로 끌어 선택 후 타이핑하는 동작이 키 3개로 끝난다.

## 반복의 두 무기 — `.` 와 매크로

같은 편집을 두 번째 할 때 다시 짜지 않는다.

`.` (dot)은 *직전 변경*을 그 자리에서 다시 실행한다. `ciw`로 단어를 바꾼 뒤 다음 단어에 커서를 옮기고 `.`만 누르면 같은 변경이 반복된다. 검색 + `.` 조합이 nvim의 "Find and Replace"를 대체하는 가장 흔한 패턴이다.

매크로(`q`)는 *여러 단계*를 녹화한다. `qa`로 레지스터 a에 녹화 시작 → 편집 동작 수행 → `q`로 종료 → `@a`로 재생. 50번 반복하고 싶으면 `50@a`. 정규식이 안 닿는 변형 작업(예: "각 줄 끝에 `,` 붙이되 빈 줄은 건너뛰기")을 손으로 한 번 시연하고 50번 돌리는 식이다.

## `:help` — Neovim에서 가장 자주 열게 되는 창

다른 에디터의 "문서를 찾아보는" 경험과 다르다. `:help <키워드>`는 동작·옵션·플러그인까지 모두 같은 문법으로 검색한다. 모르는 명령이 보이면 그 자리에서 `:help dap-mappings`처럼 친다. 답은 거의 항상 안에 있다.

특히 처음에는 다음 3개만 외워도 충분하다.

- `:help index` — 모든 키 매핑의 정식 목록
- `:help text-objects` — `iw`/`a"`/`ip` 같은 객체 한눈에
- `:Tutor` — 30분짜리 내장 튜토리얼. 한 번은 끝까지 돌린다

## 이걸 모르면 막히는 지점

- Normal 모드 진입을 잊고 명령이 본문에 박힘 → `Esc` 누르는 습관이 모든 출발점이다.
- 마우스로 드래그 선택하던 습관을 못 끊으면 텍스트 오브젝트의 가치가 평생 안 보인다 — 첫 일주일은 마우스 금지가 권장된다.
- `:help`를 안 열고 구글에서 찾으면 nvim 0.5 이전(Vimscript) 문서가 섞여서 잘못된 답에 도달한다. 1차 출처는 항상 `:help`.
