# 데이터 모델과 쿼리 언어

---

> 데이터 모델은 코드 아키텍처보다 한 단계 위에서 사고를 가른다. 같은 도메인을 관계형 테이블로 푸느냐, 문서로 푸느냐, 그래프로 푸느냐에 따라 자연스럽게 표현되는 질문과 어색해지는 질문이 갈린다. 본 챕터는 세 모델의 강점·약점, 정규화·비정규화 트레이드오프, 그래프 쿼리 언어, 이벤트 소싱이 어떤 자리에 어울리는지를 정리한다.



## 데이터 모델은 왜 사고를 가르는가

> 같은 현실을 어떤 모델로 옮기느냐가 그 위에서 가능한 질문과 가능한 변경을 결정한다.

소프트웨어는 여러 추상 계층을 쌓아 만든다. 애플리케이션은 객체와 데이터 구조로 현실을 표현하고, 그 객체는 JSON·관계형 테이블·그래프 같은 데이터 모델로 옮겨진다. 데이터 모델은 다시 메모리·디스크·네트워크의 바이트로 인코딩되고, 마지막에는 전류와 자기장이 된다. 각 계층은 아래 계층의 복잡성을 가리고 위 계층에 깔끔한 인터페이스를 준다.

데이터 모델 선택이 중요한 이유는 그 모델이 잘 표현하는 질문과 표현이 어색한 질문을 가르기 때문이다. 친구의 친구 N 단계를 묻는 질문은 그래프에서 자연스럽지만 SQL 로는 재귀 CTE 가 길게 따라온다. 사용자 프로필처럼 한 사람에 속한 학력·경력·연락처 묶음은 문서 모델이 자연스럽지만 정규화된 다섯 테이블 JOIN 은 늘 보일러플레이트가 된다. "어떤 모델이 옳은가" 보다 "이 워크로드에 어느 모델이 어울리는가" 를 묻는 편이 운영 관점에서 정확하다.



## 선언적 vs 명령적 — SQL 이 가져온 것

> SQL 같은 선언적 언어는 "무엇을" 묻고, 알고리즘 코드 같은 명령적 언어는 "어떻게" 답할지 명시한다.

```sql
SELECT * FROM users WHERE age > 25 ORDER BY name;
```

이 쿼리는 결과 조건만 말한다. 옵티마이저가 인덱스를 어떻게 쓸지, 정렬을 어떤 알고리즘으로 할지, 병렬 실행 여부를 결정한다. 같은 일을 명령형 코드로 짜면 루프와 정렬 알고리즘을 직접 두는 형태가 되어 DB 가 통계 정보로 자동 최적화하는 자유가 사라진다.

선언적 언어의 두 번째 가치는 시간이 갈수록 빛난다. 하드웨어가 바뀌고 인덱스가 추가되고 통계가 갱신되면 같은 SQL 이 다른 실행 계획으로 자동 전환된다. 명령형 코드는 그때마다 사람이 다시 짜야 한다. SQL 이 50 년 가까이 살아남은 이유의 절반은 이 선언성에 있다.



## 관계형 vs 문서 — Object-Relational 임피던스 미스매치

> 객체로 짠 코드와 정규화된 테이블 사이에는 늘 변환 계층이 끼어든다. ORM 이 풀려는 문제이자 종종 또 다른 문제를 만드는 자리이기도 하다.

이력서를 예로 들어 보자. 한 사람에게는 여러 경력과 여러 학력 그리고 연락처가 있다. 관계형 모델로 풀면 `users`, `positions`, `educations`, `contact_info` 네 테이블이 외래키로 묶이고, 문서 모델로 풀면 한 JSON 안에 모두 중첩된다.

```json
{
  "user_id": 251,
  "first_name": "Barack",
  "positions": [
    {"title": "President", "org": "United States"},
    {"title": "Senator", "org": "US Senate"}
  ],
  "educations": [{"school": "Harvard", "start": 1988, "end": 1991}]
}
```

문서 모델의 가치는 데이터 지역성(locality) 에 있다. 한 사람의 모든 정보가 한 곳에 모여 있어 한 번의 조회로 끝난다. 트리 구조의 one-to-many 관계가 자연스럽게 표현된다. 단점은 many-to-many 관계가 들어오면 무너진다는 점이다. 같은 조직에서 일한 모든 사람을 찾으려면 모든 문서를 훑어야 한다.

관계형 모델은 그 반대다. 정규화된 테이블 위 JOIN 은 임의 방향으로 데이터를 묶어 낸다. 단점은 객체와 테이블 사이의 변환(임피던스 미스매치) 이 늘 따라온다는 점이다. ORM 은 이 변환을 자동화하지만 N+1 쿼리, 비효율 스키마 같은 새 함정을 만들어 낸다(자세한 ORM 측면은 [`./spring/querydsl/`](./spring/querydsl/) 묶음 참고).

현대 DB 는 두 모델이 서로 흡수하는 방향으로 수렴하고 있다. PostgreSQL 의 `jsonb` 컬럼은 관계형 위에 문서 표현을 얹고, MongoDB 의 `$lookup` 은 문서 모델 위에서 JOIN 비슷한 동작을 제공한다. 운영 선택은 "관계형 vs 문서" 가 아니라 "어느 쪽이 주축이고 어느 쪽이 보조인가" 의 문제로 옮겨 갔다.



## 정규화 vs 비정규화

> 같은 정보를 한 곳에만 두느냐(정규화) 여러 곳에 복사해 두느냐(비정규화) 의 선택이다. 읽기·쓰기 비용이 정반대로 갈린다.

정규화는 ID 참조로 데이터를 한 곳에 둔다. 그 정보가 바뀔 때 한 곳만 고치면 되니 일관성이 자동으로 유지된다. 대신 읽을 때 JOIN 이 따라온다. 비정규화는 자주 함께 읽히는 정보를 미리 복사해 둔다. 읽기는 JOIN 없이 한 번에 끝나지만 원본이 바뀔 때 모든 사본을 함께 갱신해야 한다.

| 항목 | 정규화 | 비정규화 |
|------|--------|---------|
| 저장 방식 | ID 참조 | 값 복사 |
| 쓰기 비용 | 낮음 (한 곳) | 높음 (여러 곳) |
| 읽기 비용 | 높음 (JOIN) | 낮음 (단일 조회) |
| 디스크 | 적음 | 많음 |
| 일관성 | 자동 | 수동 (애플리케이션 책임) |

X(트위터) 의 머터리얼라이즈드 타임라인이 운영의 한 단면이다. 트윗 한 건이 올라오면 팔로워의 타임라인 캐시에 미리 복사해 둔다(비정규화). 다만 내용·좋아요 수처럼 자주 바뀌는 필드는 ID 만 두고 읽을 때 별도 조회로 채운다(hydration). 두 전략을 한 자리에 섞어 쓴다.

선택 기준은 단순하다. **정보가 얼마나 자주 바뀌는가** 와 **읽기·쓰기 비율이 얼마나 치우쳐 있는가** 다. 거의 안 바뀌고 압도적으로 자주 읽히는 데이터는 비정규화가 유리하다. 자주 갱신되는 데이터는 정규화 + 캐시(자세한 캐시 패턴은 [`./01-07.캐싱 전략.md`](./01-07.캐싱%20전략.md)) 조합이 안전하다.



## Many-to-Many — Join Table 과 ID 참조

> 한 사람이 여러 조직에서 일했고, 한 조직에 여러 사람이 속한다. 관계형은 join table 로, 문서는 ID 참조 + secondary index 로 푼다.

관계형 모델의 표준 답은 중간 테이블이다. `positions` 테이블이 `user_id` 와 `org_id` 외래키 두 개를 갖고, 양 끝의 `users` 와 `organizations` 와 JOIN 한다. 양방향 질문(특정 사람의 모든 조직 / 특정 조직의 모든 사람) 모두 같은 테이블 위에서 답한다.

문서 모델은 한쪽에 ID 만 박아 둔다. 사용자 문서에 일한 조직의 ID 배열을 두면 사람 → 조직 방향 쿼리는 단순하지만, 조직 → 사람 방향 쿼리는 모든 사용자 문서를 훑어 봐야 한다. MongoDB 같은 시스템은 secondary index 로 이 역방향 쿼리를 받쳐 준다.

Many-to-many 가 도메인의 핵심 관계라면 관계형 모델이 자연스러운 답이다. 트리 구조에 가끔 cross-link 가 있는 정도라면 문서 모델 + ID 참조도 충분히 동작한다.



## 그래프 모델 — 관계가 데이터의 본질일 때

> 모든 것이 모든 것과 연결되는 도메인이 있다. 소셜 네트워크, 추천, 지식 그래프, 도로망. 관계형으로도 풀 수 있지만 쿼리가 길어진다.

그래프는 정점(vertex) 와 간선(edge) 으로 구성된다. Property Graph 모델은 두 가지 모두에 라벨과 키-값 속성을 붙일 수 있게 한다. Neo4j 의 Cypher 가 이 모델의 대표 쿼리 언어다.

```cypher
MATCH (person)-[:BORN_IN]->()-[:WITHIN*0..]->(:Location {name:'United States'}),
      (person)-[:LIVES_IN]->()-[:WITHIN*0..]->(:Location {name:'Europe'})
RETURN person.name
```

"미국에서 태어나 유럽에 사는 사람" 을 찾는 쿼리다. `WITHIN*0..` 는 "WITHIN 관계를 0 번 이상 따라가라" 는 가변 길이 패턴이다. Idaho → United States 같은 직접 관계, Idaho → 행정구역 → United States 같은 두 단계 관계, 더 깊은 단계 모두 자동으로 매칭된다.

같은 쿼리를 SQL 의 `WITH RECURSIVE` 로 짜면 30 줄 가까이 된다. 관계형 DB 도 그래프 쿼리를 못 짜는 것은 아니지만, 관계 탐색이 도메인의 중심이라면 그래프 모델이 길이와 가독성에서 명확히 앞선다. 반대로 집계·분석이 주축이면 관계형이 여전히 유리하다.

Triple-Store(예: Apache Jena) 는 변형이다. 모든 데이터를 (subject, predicate, object) 세 자리 튜플로 정규화하고 SPARQL 로 쿼리한다. 의미 웹과 RDF 에서 표준이다. GraphQL 은 같은 이름이 들어가지만 그래프 *데이터베이스* 가 아니라 클라이언트 주도 *API* 쿼리 언어다. 백엔드는 보통 관계형 DB 위에서 동작한다.



## Star Schema — 분석을 위한 관계형 변형

> OLAP 워크로드는 OLTP 와 다른 모양의 스키마를 요구한다. 가운데 fact 테이블 하나에 dimension 들이 빛살처럼 붙는 별 모양이 표준이다.

```
              dim_product
                  |
dim_store ─── fact_sales ─── dim_date
                  |
            dim_customer
```

`fact_sales` 테이블은 매출 한 건 한 건의 측정값(수량·금액) 과 dimension 외래키들을 담는다. 차원 테이블들은 카테고리 정보(상품명·매장 위치·날짜의 분기 등) 를 정규화 또는 비정규화로 보관한다. dimension 이 다시 sub-dimension 으로 쪼개지면 Snowflake Schema, 모든 dimension 을 fact 안에 비정규화로 박으면 OBT(One Big Table) 가 된다.

OLAP 환경에서 Star Schema 가 표준이 된 이유는 단순함이다. 분석 쿼리 대부분이 fact 와 한두 개 dimension 만 JOIN 해서 답이 나온다. SQL 이 짧아지고 쿼리 옵티마이저가 잘 푼다. 컬럼 저장 엔진([`./01-02.저장소와 검색.md`](./01-02.저장소와%20검색.md) 의 컬럼 저장 절) 위에서 더 빛난다.



## Schema-on-Write vs Schema-on-Read

> 스키마를 어느 시점에 강제하느냐의 차이다. 두 접근 모두 결국 스키마를 강제하지만 강제 시점이 다르다.

Schema-on-Write(관계형) 는 INSERT/UPDATE 시점에 타입과 제약을 검사한다. 컴파일 타임 타입 체크와 비슷한 안정성을 준다. 단점은 스키마 변경에 마이그레이션이 따라온다는 점이다. 컬럼 추가는 보통 안전하지만 컬럼 삭제·타입 변경은 운영에서 신중을 요한다.

Schema-on-Read(문서) 는 쓰기 시점에 강제하지 않고 읽는 코드가 알아서 해석한다. 같은 컬렉션에 v1 과 v2 형식 문서가 공존할 수 있고, 읽는 코드는 둘 다 처리해야 한다.

```javascript
// 새 코드가 옛 데이터를 읽음 — schema-on-read
if (user && user.name && !user.first_name) {
    user.first_name = user.name.split(" ")[0];
}
```

마이그레이션 비용을 미루는 대신 코드가 형식 분기를 가져야 하므로 결국 어딘가에서 비용을 치른다. 인코딩과 스키마 진화 측면은 [`./01-03.인코딩과 진화.md`](./01-03.인코딩과%20진화.md) 가 자세히 다룬다.



## Event Sourcing 과 CQRS — 상태 대신 변화의 로그

> 현재 상태를 직접 저장하는 대신 모든 변경을 불변 이벤트로 기록한다. 회계 원장이 잔액이 아닌 거래 내역을 기록하는 것과 같다.

```
Event Log: [SeatReserved] [SeatCancelled] [SeatReserved] [PaymentReceived]
                              │
                ┌─────────────┼─────────────┐
                ▼             ▼             ▼
          BookingStatus  Dashboard   BadgePrinter
          (View)         (View)      (View)
```

이벤트 로그가 진실의 원천이고 다양한 조회 형태(view) 가 그 위에 머터리얼라이즈된다. 이를 CQRS(Command Query Responsibility Segregation) 라고 부른다. 쓰기는 이벤트로 단순하게, 읽기는 워크로드별로 최적화된 view 로 분리한다.

장점이 셋이다.

1. 의도가 명확하다. "active 컬럼 false" 가 아니라 "예약 취소됨(SeatCancelled)" 으로 기록되어 무엇이 왜 일어났는지 그대로 남는다.
2. 새 view 를 추가하기 쉽다. 이벤트 로그를 처음부터 다시 재생하면 된다. 버그를 수정한 view 도 같은 방식으로 다시 만든다.
3. 감사 로그가 자동으로 따라온다. 모든 변경이 영구 기록이므로 별도 audit table 이 필요 없다.

함정도 분명하다. GDPR 의 "잊혀질 권리" 가 이벤트 로그의 불변성과 정면충돌한다. 외부 효과(이메일 발송 등) 가 일어난 이벤트를 재생할 때 중복 실행을 막는 멱등성 설계가 필요하다. 이벤트 명명도 중요한데, 과거형(SeatReserved) 으로 적어 "이미 일어난 사실" 임을 분명히 한다. "예약한다" 같은 명령형은 명령(Command) 의 명명이지 이벤트 명명이 아니다.

이벤트 소싱은 운영 부담이 크고 학습 곡선이 가파르므로, 모든 도메인이 이걸로 가야 하는 것은 아니다. 복잡한 비즈니스 도메인 + 감사 요구사항 + 높은 쓰기 부하가 모이는 자리에서 가치가 가장 크다. 메시징 인프라 위 구현 측면은 [`../05_messaging/`](../05_messaging/) 가 다룬다.



## DataFrame — 분석 코드의 모델

> R, Pandas, Spark 에서 쓰는 표 형태 자료구조다. 관계형 테이블과 비슷하지만 컬럼 추가·변환이 자유롭고 행렬과의 변환도 자연스럽다.

DataFrame 의 가치는 데이터 랭글링(wrangling) 의 표현력이다. 사용자×영화 평점 테이블을 사용자×영화 매트릭스로 pivot 하거나, 범주형 컬럼을 one-hot encoding 으로 펼치는 작업이 표현 한 줄로 끝난다. 머신러닝 파이프라인의 입력 형식이라 사실상 데이터 사이언스의 표준이다.

DB 의 관계형 모델과 DataFrame 의 차이는 의도의 차이다. 관계형은 영구 저장과 동시 쿼리에 최적화되어 있고, DataFrame 은 일시적 분석 변환에 최적화되어 있다. 같은 SELECT/JOIN/GROUP BY 의미를 표현 가능하지만 DataFrame 은 행렬 변환·통계 요약·시각화와 직결된다.



## 모델 선택 가이드

> 워크로드의 모양으로 결정하는 편이 안전하다.

| 도메인 모양 | 권장 모델 | 대표 시스템 |
|------------|----------|-------------|
| CRUD + 트랜잭션, 정형 | 관계형 | PostgreSQL, MySQL |
| One-to-many 트리, 유연한 스키마 | 문서 | MongoDB, CouchDB |
| 관계 탐색이 핵심 | 그래프 | Neo4j |
| 분석·BI | 관계형 (Star Schema) + 컬럼 저장 | Snowflake, ClickHouse |
| 감사·복잡 도메인·높은 쓰기 | Event Sourcing + CQRS | EventStoreDB, Kafka |
| 데이터 사이언스 | DataFrame | Pandas, Spark |

운영에서는 한 모델만 고집하기보다 polyglot persistence 가 흔하다. 트랜잭션은 PostgreSQL, 캐시는 Redis, 검색은 Elasticsearch, 분석은 Snowflake 식이다. 각 모델이 잘 푸는 자리에만 그 모델을 쓴다는 발상이다. 자세한 NoSQL 비교는 [`./01-06.NoSQL 비교.md`](./01-06.NoSQL%20비교.md) 가 다룬다.
