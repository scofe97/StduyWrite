# window 함수 없는 JPA QueryDSL의 ROW_NUMBER 대체

---

> 운영 코드베이스에서 자주 만나는 시나리오 한 편을 통째로 풀어 본다. 네이티브 SQL의 CTE + `ROW_NUMBER()`로 짜둔 쿼리를 JPA QueryDSL로 옮길 때 마주치는 한계와, 그 한계를 sub-select와 정렬 트릭으로 어떻게 의미 동등하게 우회하는지 살핀다. 02-04가 단일 패턴 모음이라면 본 챕터는 *한 쿼리의 분해와 재조립* 을 다룬다.

---

## 본 챕터의 위치

> 02-04의 부록과 같은 결이다. 표준 학습이 끝난 뒤, 운영 코드의 비표준 SQL을 JPA로 옮겨야 할 때 본다.

JPA의 JPQL은 1.x 시절 표준의 한계를 그대로 들고 있어서 window 함수(`ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)`), 공통 테이블 표현식(CTE, `WITH x AS (...)`), `LATERAL` 조인 등을 지원하지 않는다. Hibernate 6 이후 일부 dialect에서 native SQL 기반의 window 함수가 통하긴 하지만 *JPQL 레벨* 에서는 여전히 표현할 길이 없다. QueryDSL은 JPQL 위에 얹혀 있어 같은 제약을 그대로 물려받는다.

본 챕터는 이 제약을 *피해 가는* 패턴을 한 케이스로 익힌다. 케이스는 TPS operator-api의 "내 결재함" 조회 쿼리다. 사용자가 *지금 처리해야 하는 결재 단계의 순번* 을 컬럼으로 채우고, 그 컬럼이 비어있지 않은 행을 상단으로 정렬한다. 네이티브 SQL에서는 `CUR_STEP` CTE + `ROW_NUMBER`로 푸는 전형적인 문제다.

---

## 시나리오 — "내 결재함" 한 줄 요약

> 조건이 겹쳐서 복잡해 보이지만 모든 조건은 *왜 그렇게 정렬되는가* 한 가지를 위해 동작한다.

요구사항을 한 문장으로 줄이면 이렇다.

> userId가 상신자이거나 결재선에 포함된 결재 실행 목록을 페이지 단위로 조회한다. 사용자가 *지금 처리할 단계의 atrzSn* 이 채워진 행을 상단에 두고, 그 뒤로 사용자 정렬을 적용한다.

세부 규칙은 다음과 같다.

| 규칙 | 의미 |
|------|------|
| 가시성 | `rgtrId = userId` (상신자) OR 권한 매칭으로 결재선에 포함 (수신자) |
| 권한 매칭 | 직접 결재자(`APRVR_SE_CD='01'`) OR 역할 결재자(`'03'` + AT_007 활성 + AT_008/AT_012/CM_004 경유) |
| 현재 단계 | 진행 중 단계(`dmndDt IS NOT NULL AND cmptnDt IS NULL`) 중 권한 매칭되고, `ordrDsgnYn='Y'` 우선·`atrzSn` 오름 |
| 정렬 | 현재 단계가 채워진 행을 우선, 그 뒤 사용자 입력 정렬 |
| 페이징 | `offset`/`limit` |
| 결과 | 결재 실행 단위 1행 (`groupBy("atrzExcnId")`) |

네이티브 SQL이라면 `CUR_STEP` CTE를 `ROW_NUMBER() OVER (PARTITION BY atrzExcnId ORDER BY CASE ordrDsgnYn WHEN 'Y' THEN 0 ELSE 1 END, atrzSn)`로 매기고 `rn = 1`만 남기는 형태가 자연스럽다. JPA QueryDSL에는 이 도구가 없다.

---

## 한계 정리 — JPA QueryDSL이 못 하는 것

> JPQL 표준의 빈자리와 QueryDSL이 그 위에서 못 하는 일을 분리해 본다.

1. **`ROW_NUMBER()` 등 window 함수 표준 미지원.** JPQL 명세에 없다. Hibernate 6.x 이후 native query에서는 동작하지만 JPQL → QueryDSL 경로에서는 잡히지 않는다.
2. **`WITH` (CTE) 미지원.** Spring Data JPA의 `@Query`도 JPQL이면 마찬가지다. 회피하려면 native query로 내려야 한다.
3. **`LATERAL` 조인 미지원.** 행 단위 sub-select를 join처럼 쓰는 길이 막혀 있다.
4. **`SELECT` 절의 sub-select 결과를 같은 쿼리의 `ORDER BY`에서 식별자 alias로 재사용 불가.** 같은 sub-select를 두 번 빌드해 넣어야 한다.

QueryDSL이 6.12든 7.x든, JPA 위에 있는 한 위 네 가지 제약은 동일하다. native SQL로 내리면 풀리지만 그 순간 QueryDSL의 정적 메타모델 이점이 사라지므로 다른 트레이드오프가 생긴다.

---

## 우회 전략의 골격

> 단일 sub-select + tie-breaker 정렬로 의미만 같게 만든다.

핵심 아이디어 두 가지로 압축된다.

1. **`ROW_NUMBER` 대신 `JPAExpressions` 단일 sub-select.** `select ... from ... where ... order by ... limit 1` 형태로 *현재 단계의 atrzSn* 한 값을 뽑는다. 이 값을 컬럼(`curAtrzSn`)으로 노출한다.
2. **`PARTITION BY` 대신 상관 서브쿼리.** sub-select의 `where`에 `aprvExcn.atrzExcnId = ctx.atrzExcnId` 같은 상관 조건을 넣어 *바깥 결재 실행마다 1행* 이 평가되게 만든다. 의미상 결재 실행 단위로 나눈 partition이 된다.
3. **`rn = 1` 대신 정렬 tie-breaker.** sub-select 결과가 `null`인 행을 뒤로 보내는 CASE 정렬을 *사용자 정렬보다 앞에* 끼워넣는다. window 함수 없이도 "현재 단계가 있는 행을 상단" 의도를 살린다.

이 셋만 손에 잡고 가면 나머지는 표준 QueryDSL 문법이다.

## 구현 — 단계별로 분해

> 운영 코드에서 가져온 형태를 6단계로 잘라 본다. 각 단계가 위 골격의 어느 부분을 담당하는지 명시한다.

### 1. 요청 해석 — 화이트리스트 정렬

```java
ResolvedListQuery<MyToListColumn> resolved = resolveListQuery(LIST_QUERY_RESOLVER, request);
```

`MyToListColumn`은 정렬 가능한 컬럼만 enum으로 묶어 화이트리스트로 둔다. 사용자가 임의 컬럼명을 보내도 enum 매칭에서 떨어진다. 이 단계는 본 챕터의 주제는 아니지만 *동적 정렬을 안전하게 받는 표준 패턴* 이므로 짚어 둔다.

### 2. 컨텍스트와 가시성 조건

```java
MyToListQueryContext ctx = defaultQueryContext();
Predicate predicate = buildPredicate(resolved, ctx, basePredicate(ctx, userId));
```

`basePredicate`가 본 챕터의 첫 핵심이다. 가시성 두 갈래를 OR로 묶는다.

```java
private Predicate basePredicate(MyToListQueryContext ctx, String userId) {
    BooleanExpression isSubmitter = ctx.aprvExcn().getString("rgtrId").eq(userId);
    BooleanExpression isReceiver = atrzPermissionExists(ctx, userId);
    return isSubmitter.or(isReceiver);
}
```

`isReceiver`는 `EXISTS` 서브쿼리를 한 번 더 감싼다. 직접 결재자와 역할 결재자(AT_007 활성 + AT_008/AT_012/CM_004 매핑)를 OR로 묶어 사용자가 결재선에 포함되었는지 본다. 02-04의 "상관 서브쿼리" 패턴이 그대로 들어온다.

### 3. CUR_ATRZ_SN — window 함수의 의미 동등 표현

`CUR_STEP` CTE를 단일 sub-select로 압축한 핵심 메서드다.

```java
private SubQueryExpression<Integer> curAtrzSnSubquery(MyToListQueryContext ctx, String userId) {
    PathBuilder<ApprovalProgressEntity> p = new PathBuilder<>(ApprovalProgressEntity.class, "pCur");
    PathBuilder<ApprovalTargetBasicEntity> s = new PathBuilder<>(ApprovalTargetBasicEntity.class, "sCur");
    NumberPath<Integer> pAtrzSn = p.get("id").getNumber("atrzSn", Integer.class);
    NumberPath<Integer> sAtrzSn = s.get("id").getNumber("atrzSn", Integer.class);

    PathBuilder<ApprovalTargetApproverEntity> aprvrStep =
            new PathBuilder<>(ApprovalTargetApproverEntity.class, "aprvrStep");
    BooleanExpression stepPermitted = JPAExpressions.selectOne()
            .from(aprvrStep)
            .where(
                aprvrStep.get("id").getString("atrzId").eq(p.get("id").getString("atrzId")),
                aprvrStep.get("id").getNumber("vsrn", Integer.class).eq(p.get("id").getNumber("vsrn", Integer.class)),
                aprvrStep.get("id").getNumber("atrzSn", Integer.class).eq(pAtrzSn),
                directApproverMatch(aprvrStep, userId).or(roleApproverMatch(aprvrStep, userId))
            )
            .exists();

    return JPAExpressions
            .select(pAtrzSn)
            .from(p)
            .innerJoin(s).on(
                s.get("id").getString("atrzId").eq(p.get("id").getString("atrzId")),
                s.get("id").getNumber("vsrn", Integer.class).eq(p.get("id").getNumber("vsrn", Integer.class)),
                sAtrzSn.eq(pAtrzSn)
            )
            .where(
                p.get("id").getString("atrzExcnId").eq(ctx.aprvExcn().getString("atrzExcnId")),
                p.getDateTime("dmndDt", LocalDateTime.class).isNotNull(),
                p.getDateTime("cmptnDt", LocalDateTime.class).isNull(),
                ctx.aprvExcn().getString("atrzPrgrsSttsCd").in("WAIT", "EXCN"),
                stepPermitted
            )
            .orderBy(
                new CaseBuilder()
                    .when(s.getString("ordrDsgnYn").eq("Y")).then(0)
                    .otherwise(1).asc(),
                pAtrzSn.asc()
            )
            .limit(1L);
}
```

다섯 가지를 하나씩 짚는다.

1. **PathBuilder로 별칭을 직접 짓는다.** `pCur`, `sCur`, `aprvrStep`. 같은 엔티티(`ApprovalProgressEntity`)가 바깥 쿼리와 sub-query 양쪽에 등장하기 때문에 Q-class 정적 인스턴스로는 이름이 충돌한다. 02-04의 "PathBuilder 변형"이 *왜* 필요한지 본 케이스에서 또렷이 드러난다.
2. **상관 조건이 partition을 대체한다.** `p.atrzExcnId = ctx.aprvExcn().atrzExcnId`가 바깥 결재 실행마다 sub-select를 다시 평가하게 만든다. 의미상 `PARTITION BY atrzExcnId`다.
3. **`limit 1`이 `ROW_NUMBER() = 1`을 대체한다.** ORDER BY로 우선순위를 정한 뒤 한 행만 뽑으면 `rn = 1`과 같다.
4. **`ordrDsgnYn='Y'` 우선 정렬은 `CASE WHEN`으로.** QueryDSL의 `CaseBuilder`가 그대로 SQL `CASE`로 풀린다. window 함수 안의 `ORDER BY CASE`와 동일하다.
5. **권한 매칭이 sub-select 안에 또 sub-select.** `stepPermitted`는 EXISTS다. 본 단계(atrzId/vsrn/atrzSn)에 사용자가 결재자로 지정되어 있는지 확인한다. 이 안에 또 EXISTS가 들어 있는 `roleApproverMatch`까지 합치면 EXISTS가 *3중 중첩* 된다. 가독성은 떨어지지만 의미는 명확하다.

### 4. 정렬 tie-breaker — null을 뒤로

```java
NumberExpression<Integer> curAtrzSn = Expressions.asNumber(curAtrzSnSubquery(ctx, userId));

OrderSpecifier<?>[] sortOrders = buildOrderSpecifiers(resolved, ctx);
OrderSpecifier<?>[] orders = withTieBreaker(curAtrzSn, sortOrders);
```

```java
private OrderSpecifier<?>[] withTieBreaker(NumberExpression<Integer> curAtrzSn, OrderSpecifier<?>[] base) {
    OrderSpecifier<?> nullsLast = new CaseBuilder()
            .when(curAtrzSn.isNull()).then(1)
            .otherwise(0)
            .asc();
    OrderSpecifier<?>[] result = new OrderSpecifier<?>[base.length + 1];
    result[0] = nullsLast;
    System.arraycopy(base, 0, result, 1, base.length);
    return result;
}
```

핵심은 *사용자 정렬보다 앞* 에 끼워넣는다는 점이다. `result[0] = nullsLast`. 사용자 정렬이 무엇이든 *현재 단계가 채워진 행이 항상 상단* 임을 보장한다.

### 5. 프로젝션 — 같은 sub-select를 두 번

```java
return applyJoins(
    queryFactory.select(Projections.fields(
        ApprovalMyToDetailDto.class,
        ctx.aprvExcn().getString("atrzExcnId").as("atrzExcnId"),
        // ... 표시용 컬럼들 ...
        ExpressionUtils.as(curAtrzSnSubquery(ctx, userId), Expressions.path(Integer.class, "curAtrzSn")),
        // ... 남은 컬럼들 ...
    ))
    .from(ctx.aprvExcn()),
    ctx
)
    .where(predicate)
    .groupBy(ctx.aprvExcn().getString("atrzExcnId"))
    .orderBy(orders)
    .offset(resolved.offset())
    .limit(resolved.limit())
    .fetch();
```

여기서 *눈에 띄는 비효율* 이 있다.

> 같은 `curAtrzSnSubquery(ctx, userId)`가 select 절과 order by 절에 *두 번* 빌드된다.

JPQL에는 select alias를 order by에서 다시 참조하는 표준 길이 없다. 사용자 정의 alias를 받아주는 dialect도 있지만 표준은 아니다. QueryDSL 빌더 단계에서 같은 sub-select를 두 번 만들어 SQL로 내린다.

옵티마이저가 동일 sub-select를 *공통 부분식 제거(Common Subexpression Elimination)* 로 합쳐 주면 한 번만 실행되지만, MySQL/MariaDB 옵티마이저는 보장이 약하다. PostgreSQL은 비교적 더 잘 합친다. 운영 환경에서 EXPLAIN을 떠 봐야 한다.

### 6. ThreadLocal userId — 표현식 내부 사용

`bindUserId(userId)` / `clearUserId()` 패턴은 02-04에서 다루지 않은 부수 패턴이다. 일부 표현식(예: `submitReceiveCdExpr`)이 매번 메서드 시그니처로 userId를 받는 대신 ThreadLocal에서 꺼낸다. 표현식 빌더가 인자를 줄여 가독성을 얻는 대신 *암시적 의존* 이라는 비용을 진다. `try/finally`로 누수만 방지하면 동작한다.

## 같은 sub-select 두 번 — 어떤 트레이드오프인가

> 표준 학습에서는 본 적 없는 패턴이다. 비용과 이득을 따져 본다.

### 비용

1. **SQL 텍스트 길이 증가.** 동일 sub-select가 두 번 나간다. 텍스트가 두 배가 된다.
2. **옵티마이저에 의존.** PostgreSQL이라면 동일 sub-select를 합쳐 한 번만 평가할 확률이 높다. MySQL 5.7은 합치지 않을 가능성이 크다. MySQL 8.x는 그 중간이다.
3. **EXPLAIN 가독성 저하.** 실행 계획에 같은 sub-select가 두 번 표시되면 분석이 어려워진다.

### 이득

1. **JPQL/QueryDSL 한 줄로 컴파일.** native query로 내리지 않아도 된다. 정적 타입 검사와 메타모델 이점을 그대로 유지한다.
2. **사용자 정렬과 자동 결합.** `withTieBreaker`가 사용자 정렬보다 앞에만 끼워 넣으면 끝이다. native SQL이라면 정렬 컬럼을 매번 SQL 텍스트에 박아야 한다.
3. **테스트 용이성.** QueryDSL 레벨에서 검증한다. `@DataJpaTest`로 검증 흐름이 끊기지 않는다.

### 언제 native로 내려가야 하나

세 가지 신호 중 둘 이상이면 native query를 고민한다.

1. EXPLAIN에서 같은 sub-select가 두 번 평가되고 결재 실행 수가 1만 건을 넘는다.
2. tie-breaker 외에도 partition 단위 집계(rolling sum, lag 등)가 추가로 필요해진다.
3. CTE를 여러 단계로 쌓아야 가독성이 살아난다.

이때는 `@Query(value = "...", nativeQuery = true)`로 내리고 QueryDSL은 *다른 부분* 만 맡긴다.

## 흔한 함정

> 본 패턴을 옮길 때 자주 밟는 곳을 모은다.

### 1. select 절 alias를 order by에서 재참조

```java
// 동작하지 않는다 (JPQL 표준 위반)
.orderBy(Expressions.numberPath(Integer.class, "curAtrzSn").asc())
```

`curAtrzSn`은 select 절에 `as("curAtrzSn")`로 노출했지만 JPQL에서는 order by가 그 alias를 참조하지 못한다. 같은 sub-select를 다시 빌드해 넘기는 길밖에 없다. dialect에 따라 통하는 경우도 있지만 *이식성 깨지는 비표준 의존* 이다.

### 2. `groupBy`와 select 절의 미집계 컬럼

```java
.groupBy(ctx.aprvExcn().getString("atrzExcnId"))
```

select 절에는 `atrzGdncWords`, `regDt`처럼 group 키가 아닌 컬럼이 함께 있다. SQL 표준 ONLY_FULL_GROUP_BY 모드에서는 에러다. MySQL의 *비표준 group by* 동작에 의존한다. 도메인 PK로 묶었으니 의미적으로는 결정적이지만, PostgreSQL로 이식할 때 깨진다. PostgreSQL이라면 *모든 select 컬럼을 group by에 넣거나*, `DISTINCT ON (atrzExcnId)` 또는 window 함수로 다시 짜야 한다.

### 3. EXISTS 중첩 깊이

`atrzPermissionExists` 안에 `roleApproverMatch`가 들어가고 그 안에 또 AT_008/AT_012/CM_004 EXISTS 셋이 들어간다. 이 구조가 `curAtrzSnSubquery`의 `stepPermitted` 안에도 같은 모양으로 들어간다. 같은 권한 매칭 로직이 *바깥 가시성* 과 *내부 현재 단계 판정* 두 곳에 중복된다.

권한 매칭 캐시를 임시 테이블이나 view로 미리 만들 수 있다면 EXISTS 중첩을 한 번으로 줄일 수 있다. 운영 부하 임계점 위에서는 검토 가치가 있다.

### 4. PathBuilder의 컬럼명 오타

```java
p.getString("aterzExcnId")  // 컴파일 통과, 런타임 에러
```

02-04에서 짚었듯 PathBuilder는 컴파일 검증이 약하다. 본 패턴은 PathBuilder가 *필수* 다(같은 엔티티에 다른 alias가 필요하므로). 단위 테스트 또는 빈번한 통합 테스트로 보완해야 한다.

### 5. `ThreadLocal` 누수

```java
bindUserId(userId);
try {
    // ...
} finally {
    clearUserId();
}
```

`try/finally`가 빠지면 다음 요청이 잘못된 userId로 동작한다. WAS 스레드 풀이 재사용되므로 누수 위험이 실제로 있다. 본 패턴을 도입할 때 *finally 강제* 를 코드 리뷰 체크리스트에 넣어야 한다.
