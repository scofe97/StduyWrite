# 캐싱 전략

---

> 캐시는 DB 부하를 줄이고 응답 시간을 한 자릿수 빠르게 만든다. 그 대가는 일관성 관리 비용이다. 본 챕터는 Cache-Aside·Write-Through·Write-Behind 세 패턴의 트레이드오프, TTL 과 이벤트 기반 무효화, Cache Stampede·Hot Key·Cache Penetration 의 운영 함정과 그 답, 그리고 PostgreSQL 의 머터리얼라이즈드 뷰가 캐시 자리에 어떻게 들어오는지를 정리한다.



## 왜 캐시를 두는가 — 지연 시간의 자릿수 차이

> 메모리 접근과 디스크 접근의 비용 차이가 자릿수 단위로 갈린다. 자주 읽는 데이터를 메모리 가까이 두는 발상이 캐시의 출발점이다.

운영 환경의 흔한 지연 시간을 세워 보면 차이가 분명하다.

| 작업 | 지연 시간 |
|------|----------|
| L1 CPU 캐시 참조 | ~1 ns |
| 메인 메모리 참조 | ~100 ns |
| Redis 단일 조회 (같은 데이터센터) | ~100~500 µs |
| SSD 랜덤 읽기 | ~150 µs |
| PostgreSQL 단순 쿼리 | 1~10 ms |
| PostgreSQL 복잡 쿼리 | 10~1000 ms |

PostgreSQL 단순 쿼리 한 건이 Redis 조회 한 건보다 10~100 배 느리다. 트래픽이 늘어나면 DB 가 먼저 병목이 되고, 같은 쿼리를 반복하는 자리에 캐시를 두는 패턴이 자연스럽게 따라온다.

캐시의 비용도 분명하다. 메모리는 디스크보다 비싸고, 캐시와 원본 사이의 일관성 관리가 새 비즈니스 로직이 된다. 결국 캐시는 "어떤 데이터를·얼마나 오래·어떻게 갱신할 것인가" 의 세 질문에 답을 내야 한다.



## Cache-Aside — 가장 흔한 출발점

> 애플리케이션이 캐시와 DB 를 모두 보고, 캐시 미스 시에만 DB 를 본다. 구현이 단순하고 자리를 가리지 않는다.

```
읽기:                          쓰기:
1. 캐시 확인 → Hit → 반환      1. DB 갱신
   ↓ Miss                     2. 캐시 무효화 (또는 갱신)
2. DB 조회
3. 캐시 저장 (TTL)
4. 반환
```

```python
def get_user(user_id):
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)

    user = db.query("SELECT * FROM users WHERE id = %s", user_id)
    redis.setex(f"user:{user_id}", 3600, json.dumps(user))
    return user

def update_user(user_id, data):
    db.execute("UPDATE users SET ... WHERE id = %s", user_id)
    redis.delete(f"user:{user_id}")  # 무효화
```

장점은 두 가지다. 자주 읽히는 데이터만 캐시에 들어가므로 메모리가 효율적이고, 캐시 장애 시 애플리케이션이 직접 DB 로 폴백할 수 있어 회복 탄력성이 좋다. 단점은 첫 조회 지연(cold start) 과 짧은 일관성 창이다. DB 갱신과 캐시 무효화 사이에 다른 요청이 옛 캐시를 채워 넣는 경합이 일어날 수 있다.

쓰기 시 캐시를 *무효화* 할지 *갱신* 할지의 선택도 운영 결정이다. 무효화는 단순하지만 다음 읽기에서 한 번 더 DB 를 친다. 갱신은 즉시 일관성을 주지만 갱신값을 계산하는 비용이 추가된다. 거의 모든 경우 무효화가 안전한 첫 선택이다.



## Write-Through — 캐시가 DB 의 앞단

> 모든 쓰기가 캐시를 거쳐 DB 까지 동기적으로 전파된다. 캐시와 DB 가 항상 같다.

```
App → Cache → DB (동기)
```

장점은 일관성 보장이다. 캐시에 있는 값은 항상 DB 와 같다. 장애 시점도 명확해 디버깅이 쉽다. 단점은 쓰기 지연이 늘어난다는 점과 읽히지 않는 데이터까지 캐시에 채워진다는 점이다. Cache-Aside 가 "쓸 만한 것만" 캐시에 두는 데 비해 Write-Through 는 모든 쓰기를 캐시에 박는다.

운영 자리는 데이터 일관성이 응답 속도보다 중요한 환경이다. 금융 잔액 조회, 재고 카운터처럼 캐시 미스가 곧 비즈니스 오류가 되는 자리에서 가치가 있다.



## Write-Behind (Write-Back) — 비동기 디퍼

> 쓰기를 캐시에 먼저 쓰고 즉시 응답한 뒤 백그라운드에서 DB 에 비동기 반영한다.

```
App → Cache → 즉시 응답
        └─→ (배치 비동기) → DB
```

장점은 압도적 쓰기 처리량이다. 쓰기를 묶어 배치로 DB 에 보내므로 DB 부하가 줄고 사용자 응답 시간이 짧다. 단점도 분명하다. 캐시 노드가 죽으면 DB 에 반영되지 않은 변경이 사라질 수 있고, 캐시와 DB 의 일관성 창이 길어 다른 시스템이 DB 를 읽으면 옛 값을 본다.

운영 자리는 손실 가능성을 감수할 수 있는 카운터·통계·로그 종류다. "조회수" 같은 메트릭은 1% 손실해도 큰 문제가 안 된다. 결제·재고처럼 정확성이 핵심인 자리에는 안 맞는다.



## TTL — 시간 기반 만료

> 모든 캐시 항목에 만료 시간을 둬 옛 데이터가 영원히 남는 사고를 막는다.

TTL 값은 데이터 특성에 따라 다르게 잡는다.

```python
CACHE_TTL = {
    "user_profile": 3600,    # 1시간 — 자주 안 바뀜
    "product_list": 300,     # 5분 — 중간 빈도
    "stock_price": 5,        # 5초 — 실시간
    "session": 86400,        # 24시간
}
```

TTL 만으로 일관성을 보장하지는 못한다. TTL 이 1 시간이면 데이터 변경 후 최대 1 시간 동안 옛 캐시가 남는다. 그 일관성 창이 비즈니스가 받아 줄 수 있는지가 TTL 결정의 기준이다.

TTL + 이벤트 기반 무효화를 함께 쓰는 패턴이 운영의 표준이다. 정상 흐름은 명시적 무효화로 즉시 일관성을 주고, 무효화가 누락된 자리는 TTL 이 안전망 역할을 한다.



## 이벤트 기반 무효화

> DB 변경이 일어났을 때 관련 캐시를 함께 지운다. 명시적이고 즉각적이다.

```python
def update_product(product_id, data):
    db.execute("UPDATE products SET ... WHERE id = %s", product_id)
    redis.delete(f"product:{product_id}")
    redis.delete(f"product_list:category:{data['category']}")
    redis.delete("product_list:all")
```

함정은 **무효화 누락** 이다. 한 데이터에 의존하는 캐시가 여러 키에 흩어져 있으면 어느 키를 지워야 할지 잊기 쉽다. 운영 코드 베이스가 커질수록 이 누락이 늘어난다.

태그 기반 무효화가 한 가지 답이다. 캐시 항목 저장 시 관련 "태그" 들을 함께 기록하고, 무효화는 태그 단위로 한다.

```python
def cache_with_tags(key, value, tags, ttl=3600):
    redis.setex(key, ttl, value)
    for tag in tags:
        redis.sadd(f"tag:{tag}", key)

def invalidate_by_tag(tag):
    keys = redis.smembers(f"tag:{tag}")
    if keys:
        redis.delete(*keys)
    redis.delete(f"tag:{tag}")

cache_with_tags("product:123", data, ["category:electronics", "brand:apple"])
invalidate_by_tag("category:electronics")  # 모든 electronics 캐시 삭제
```

분산 환경에서는 Redis Pub/Sub 으로 캐시 무효화 메시지를 모든 노드에 브로드캐스트하는 구성이 흔하다. 로컬 인메모리 캐시(L1) 와 Redis(L2) 를 함께 쓰는 multi-level 구조에서 특히 가치가 있다.



## Cache Stampede — 캐시 만료 순간의 폭주

> 인기 키의 TTL 이 만료되는 순간 수백 개 요청이 동시에 캐시 미스로 떨어져 DB 가 폭격당하는 문제다.

```
캐시 만료 시점:
Request 1 → Miss → DB Query ─┐
Request 2 → Miss → DB Query ─┤
Request 3 → Miss → DB Query ─┼─ DB 과부하!
Request 4 → Miss → DB Query ─┤
...                          ─┘
```

답은 두 갈래다.

**락 기반** 은 첫 미스만 DB 에 가도록 잠근다.

```python
def get_with_lock(key, fetch, ttl=3600):
    value = redis.get(key)
    if value:
        return json.loads(value)

    lock_key = f"lock:{key}"
    if redis.set(lock_key, 1, nx=True, ex=30):
        try:
            value = fetch()
            redis.setex(key, ttl, json.dumps(value))
            return value
        finally:
            redis.delete(lock_key)
    else:
        time.sleep(0.1)
        return get_with_lock(key, fetch, ttl)
```

다른 요청들은 락 보유자가 끝날 때까지 기다리고 그 사이 채워진 캐시를 다음 시도에서 읽는다.

**Probabilistic Early Expiration** 은 만료 시점에 가까워질수록 확률적으로 미리 갱신한다.

```python
def get_with_early_expiration(key, fetch, ttl=3600, beta=1):
    value, remaining = redis.get_with_ttl(key)
    if value:
        if random.random() < math.exp(-beta * remaining / ttl):
            # 백그라운드 갱신
            threading.Thread(target=refresh_cache, args=(key, fetch, ttl)).start()
        return json.loads(value)
    return fetch_and_cache(key, fetch, ttl)
```

만료가 멀면 갱신 확률이 거의 0 이지만 가까워질수록 점점 올라가, 동기 캐시 미스가 발생하기 전에 일부 요청이 백그라운드 갱신을 미리 시작한다. 결과적으로 만료 순간의 폭주가 부드러운 곡선으로 바뀐다.

운영에서는 인기 키 한 자릿수에만 적용해도 효과가 크다. 모든 키에 적용하면 갱신 트래픽이 늘어 역효과가 난다.



## Hot Key — 한 키에 트래픽 집중

> 인기 상품·인기 사용자처럼 특정 키 하나에 요청이 몰려 단일 Redis 노드가 병목이 되는 현상이다.

답은 세 가지다.

**키 복제** 는 같은 데이터를 N 개 키에 복제해 두고 읽기를 랜덤 분산한다.

```python
def get_hot_key(key, replicas=3):
    return redis.get(f"{key}:{random.randint(1, replicas)}")
```

쓰기 비용이 N 배 늘지만 hot key 의 읽기 처리량은 N 배 늘어난다. 읽기·쓰기 비율이 압도적으로 읽기 쪽인 자리에 맞다.

**로컬 캐시 추가** 는 애플리케이션 메모리에 한 단계 더 두는 길이다. Redis 한 노드의 처리량 한계를 우회한다. 단점은 노드 간 일관성 창이 늘어난다는 점이다.

**Redis Cluster 샤딩** 은 키 자체를 클러스터의 여러 노드로 분산한다. hot key 가 여전히 한 노드에 머무르므로 본질적 답은 아니지만 인기 키가 여럿일 때 효과가 있다.



## Cache Penetration — 존재하지 않는 데이터의 반복 요청

> 존재하지 않는 ID 를 반복 조회하면 항상 캐시 미스라 DB 가 매번 검색에 나선다. 악의적 트래픽일 수도 있고 클라이언트 버그일 수도 있다.

답은 두 갈래다.

**Null 캐싱** 은 "없음" 자체를 짧은 TTL 로 캐시에 박는다.

```python
def get_with_null_cache(key, fetch):
    value = redis.get(key)
    if value == "NULL":
        return None
    if value:
        return json.loads(value)

    result = fetch()
    if result is None:
        redis.setex(key, 60, "NULL")  # 1분 null 캐싱
    else:
        redis.setex(key, 3600, json.dumps(result))
    return result
```

**Bloom Filter** 는 키가 존재할 가능성을 사전에 걸러낸다([`./01-02.저장소와 검색.md`](./01-02.저장소와%20검색.md) 의 Bloom Filter 절 참고). 비트맵에 모든 유효 키를 미리 등록해 두고, 조회 키가 비트맵에 없으면 DB 조회 자체를 생략한다. 메모리가 적어 거대한 키 공간에도 적용 가능하다는 강점이 있다.



## PostgreSQL Materialized View — DB 안의 캐시

> 복잡한 집계 쿼리 결과를 DB 안에 캐시 테이블로 보존한다. Redis 와 다른 결의 캐시다.

```sql
CREATE MATERIALIZED VIEW monthly_sales_summary AS
SELECT
    date_trunc('month', order_date) AS month,
    category,
    SUM(amount) AS total_sales,
    COUNT(*) AS order_count
FROM orders
JOIN products ON orders.product_id = products.id
GROUP BY month, category;

CREATE INDEX ON monthly_sales_summary (month, category);

-- 조회 — 캐시된 결과
SELECT * FROM monthly_sales_summary WHERE month = '2026-04-01';

-- 갱신 (스케줄러로 주기 실행)
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_sales_summary;
```

가치는 분명하다. 분당 수천 번 실행되는 집계 쿼리를 한 번의 `REFRESH` 로 대체한다. SQL 위에서 그대로 동작하므로 애플리케이션 코드 수정이 거의 없다. `CONCURRENTLY` 옵션을 쓰면 갱신 중에도 조회가 막히지 않는다.

단점은 신선도다. `REFRESH` 가 한 시간에 한 번 돈다면 그 사이 변경은 반영되지 않는다. 점진적 갱신을 원하면 트리거 + 별도 테이블을 직접 짜거나, TimescaleDB 같은 시계열 확장의 continuous aggregate 같은 도구를 써야 한다.



## 다단 캐시 — L1·L2·L3

> 자리마다 다른 캐시를 쌓아 비용·지연·용량을 절충한다.

| 단계 | 위치 | 지연 | 용량 | 일관성 |
|------|------|------|------|--------|
| L1 | 애플리케이션 메모리 | 수십 ns | 작음(MB) | 노드별 분리 |
| L2 | Redis (분산) | 100~500 µs | 중간(GB) | 노드 간 공유 |
| L3 | DB | 1~10 ms | 큼(TB) | 진실의 원천 |

L1 은 같은 프로세스 안의 LRU 맵 정도다. Java 의 Caffeine, Go 의 ristretto 같은 라이브러리가 표준이다. 가장 빠르지만 노드별로 분리되어 일관성이 약하다.

L2 는 Redis 같은 분산 캐시다. 노드 간 공유되어 일관성이 더 강하지만 네트워크 비용이 따라온다.

운영에서는 두 단계를 함께 두고 L1 무효화를 Redis Pub/Sub 으로 브로드캐스트하는 구성이 흔하다. L1 미스 → L2 조회 → 그래도 미스면 L3(DB) 조회 → 양쪽 모두 채움 같은 흐름이다.



## 캐싱 전략 선택 가이드

| 자리 | 권장 패턴 |
|------|----------|
| 일반 읽기 위주 OLTP | Cache-Aside |
| 강한 일관성 요구(잔액·재고) | Write-Through 또는 캐시 미사용 |
| 카운터·통계 같은 손실 허용 | Write-Behind |
| 복잡한 집계 쿼리 | Materialized View |
| 인기 키 트래픽 폭주 | Hot Key 복제 + Cache-Aside |
| 만료 순간 폭주 | Cache-Aside + Probabilistic Early Expiration |
| 존재하지 않는 키 공격 | Null 캐싱 또는 Bloom Filter |



## 캐시를 쓸 때 함께 점검할 것

> 운영에서 캐시를 도입할 때 자주 빠지는 함정 모음.

캐시 키 명명 규칙을 첫날부터 정한다. `user:123`, `product:456:reviews` 같은 컬론 구분 표기가 표준이다. 키 충돌과 디버깅 비용을 줄여 준다.

TTL 을 명시 없이 기본값에 맡기지 않는다. Redis 는 TTL 없이 저장한 키가 영원히 남으므로 메모리 누수가 된다. 모든 `SET` 에 명시적 TTL 을 둔다.

Redis `maxmemory-policy` 의 의도를 운영 시작 전에 정한다. 기본값 `noeviction` 은 메모리 한도 도달 시 *새 쓰기를 거부* 해 사실상 장애가 된다. 캐시 용도라면 `allkeys-lru`(모든 키를 LRU 로 evict) 또는 `volatile-ttl`(TTL 있는 키만 만료 가까운 순으로 evict) 이 일반 답이고, 캐시·세션·큐가 한 인스턴스에 섞여 있다면 키 종류별로 인스턴스를 분리하는 편이 안전하다.

캐시 키 패턴을 한 자리에 모은다. 무효화 로직이 캐시 키 형태에 강하게 의존하므로, 키 생성을 분산시키면 무효화 누락이 늘어난다. 캐시 헬퍼 클래스 한 곳이 모든 키를 만들도록 강제한다.

캐시 메트릭(히트율·미스율·만료율) 을 모니터링한다. 히트율이 50% 를 못 넘으면 캐시 효용이 의심스럽고, TTL 또는 캐시 대상을 다시 점검해야 한다.

캐시 장애 시 폴백을 미리 정한다. Redis 가 죽었을 때 모든 요청이 DB 로 떨어지면 DB 가 함께 죽을 수 있다. Circuit breaker 와 응답 타임아웃을 짧게 둬 부분 장애가 전체 장애로 번지는 것을 막는다.
