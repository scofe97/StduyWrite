# 지리공간 데이터

---

> PostGIS 익스텐션은 PostgreSQL 을 사실상의 GIS 표준으로 만들었다. 점·선·면 같은 공간 데이터 타입, 공간 연산자·함수, GiST 인덱스가 한 묶음으로 들어 있다. 본 챕터는 `geometry` 와 `geography` 타입의 차이, SRID, 자주 쓰는 공간 함수, 공간 인덱스를 정리한다.



## 지리공간 데이터의 기본 유형

> 모든 공간 객체가 점·선·면(또는 그 집합) 의 조합이다.

| 타입 | 의미 | 예 |
|------|------|------|
| Point | 단일 위치 | 매장·정류장·랜드마크 |
| LineString | 연속된 점 | 도로·철도·강 |
| Polygon | 닫힌 영역 | 행정구역·건물·공원 |
| MultiPoint·MultiLineString·MultiPolygon | 위 셋의 컬렉션 | 군도·도로망·여러 영역 |

PostGIS 는 이 모두를 자료형으로 제공한다. WKT(Well-Known Text) 또는 WKB(Well-Known Binary) 형식으로 입력한다.

```sql
CREATE EXTENSION postgis;

-- WKT 로 지정
SELECT ST_GeomFromText('POINT(127.0 37.5)', 4326);
SELECT ST_GeomFromText('POLYGON((0 0, 10 0, 10 10, 0 10, 0 0))', 4326);
```



## geometry vs geography — 결정 기준

> PostGIS 의 두 핵심 타입이다. 이름이 비슷하지만 운영 결과가 크게 다르다.

| 항목 | geometry | geography |
|------|----------|-----------|
| 좌표계 | 평면 | 구면(지구) |
| 거리 계산 | 빠름, 부정확 | 느림, 정확 |
| 함수 지원 | 풍부 | 제한적 |
| 적합 자리 | 작은 영역 (도시 하나 등) | 큰 영역 (대륙·전 세계) |

**geometry** 는 평면 좌표계 위에서 동작한다. 모든 점이 평평한 표면에 있다고 가정하므로 계산이 빠르다. 작은 영역(서울·도쿄 같은 도시 단위) 에서는 평면 가정이 충분히 정확하다. 함수 종류가 압도적으로 많아 운영의 표준이다.

**geography** 는 지구의 곡률을 고려한다. 거리·면적 계산이 실제 지구 표면 거리에 맞다. 단점은 계산이 느리고 PostGIS 함수의 일부만 geography 를 지원한다. 큰 영역(대륙 단위·전 세계) 에서 정확성이 핵심일 때 답이다.

운영 환경 대부분이 geometry + 적절한 SRID 조합으로 충분하다. geography 가 정말 필요한 자리는 글로벌 지도·항공 거리 계산 같은 자리뿐이다.



## SRID — 좌표계 식별자

> 같은 좌표 (127.0, 37.5) 가 어느 좌표계에서 해석되느냐에 따라 의미가 다르다. SRID 가 그 좌표계를 가리키는 번호다.

자주 쓰는 SRID 가 두 가지다.

**SRID 4326 (WGS84)** — GPS 표준. 위도·경도 (lat·lon) 를 그대로 쓴다. 단위는 도(degree). 거리 계산이 부정확하지만 입력·저장의 표준이다.

**SRID 3857 (Web Mercator)** — Google Maps·OpenStreetMap 같은 웹 지도의 좌표계. 단위는 미터. 거리·면적 계산이 의미 있지만 극지방으로 갈수록 왜곡이 커진다.

```sql
-- WGS84 로 입력 후 Web Mercator 로 변환
SELECT ST_Transform(
  ST_GeomFromText('POINT(127.0 37.5)', 4326),
  3857
);
```

운영 패턴은 보통 *WGS84 로 저장, Web Mercator 로 계산* 이다. 입력은 GPS 표준에 맞추고 거리·면적 계산이 필요할 때 미터 단위로 변환한다.

한국 한정 좌표계도 자주 등장한다. **SRID 5179** (UTM-K) 가 국토지리정보원 표준이다. 한국 정부 데이터를 처리할 때 이 좌표계가 답이다.



## 자주 쓰는 공간 함수

> PostGIS 함수가 수백 개라 모두 다룰 수 없다. 운영에서 매일 쓰는 것들이다.

**거리 계산**

```sql
-- 두 점 사이 거리 (geometry, 좌표계 단위)
SELECT ST_Distance(
  ST_GeomFromText('POINT(127.0 37.5)', 4326),
  ST_GeomFromText('POINT(127.1 37.6)', 4326)
);

-- geography 로 변환 후 미터 단위
SELECT ST_DistanceSphere(
  ST_GeomFromText('POINT(127.0 37.5)', 4326),
  ST_GeomFromText('POINT(127.1 37.6)', 4326)
);

-- 또는 geography 타입 직접 사용
SELECT ST_Distance(
  'POINT(127.0 37.5)'::geography,
  'POINT(127.1 37.6)'::geography
);
```

**범위 검색** — 운영에서 가장 자주 쓰는 자리다.

```sql
-- 특정 점에서 1 km 안의 매장
SELECT * FROM stores
WHERE ST_DWithin(
  location::geography,
  ST_GeogFromText('POINT(127.0 37.5)'),
  1000  -- 미터
);

-- 폴리곤 안의 점
SELECT * FROM stores
WHERE ST_Within(location, area_polygon);
```

**`ST_DWithin`** 이 핵심 함수다. 이름 그대로 "거리 D 안에 있는가" 를 검사하고, GiST 인덱스를 활용한다. 운영의 표준 패턴이다.

**공간 관계** — 두 객체 사이 관계.

```sql
ST_Contains(a, b)    -- a 가 b 를 완전히 포함
ST_Intersects(a, b)  -- 교차 (겹침)
ST_Touches(a, b)     -- 경계만 닿음
ST_Within(a, b)      -- a 가 b 안
ST_Equals(a, b)      -- 동일
```

**도형 연산**

```sql
ST_Buffer(geom, 100)         -- 100 단위 버퍼
ST_Intersection(a, b)        -- 교집합 도형
ST_Union(geom_set)           -- 합집합
ST_Centroid(polygon)         -- 무게중심
ST_Area(polygon)             -- 면적
ST_Length(linestring)        -- 길이
```



## GiST 인덱스 — 공간 검색의 표준

> 공간 데이터의 모든 인덱스가 GiST 다.

```sql
CREATE INDEX idx_stores_location ON stores USING gist (location);
```

GiST 는 R-Tree 변형으로, 공간 객체를 그 *외접 사각형(bounding box)* 으로 트리에 인덱싱한다. 검색 시 외접 사각형으로 1 차 필터를 거치고 매칭된 객체만 정확한 도형 비교를 한다.

운영의 한 가지 패턴이 **인덱스 인식 함수와 그렇지 않은 함수의 구분** 이다.

| 인덱스 인식 ✓ | 인덱스 미인식 ✗ |
|---------------|----------------|
| `ST_DWithin` | `ST_Distance < N` |
| `ST_Intersects` | `ST_Disjoint` |
| `ST_Contains` | `ST_Equals` (대부분 환경) |
| `&&` (외접 사각형 겹침) | 좌표 직접 비교 |

같은 의미 같지만 인덱스 활용 측면에서 차이가 크다. `ST_Distance(a, b) < 1000` 보다 `ST_DWithin(a, b, 1000)` 이 한 자릿수 빠른 자리가 흔하다. EXPLAIN 으로 확인하는 운영 습관이 답이다.



## 운영에서 자주 부딪히는 함정

**SRID 불일치** — 두 객체의 SRID 가 다르면 함수가 에러를 낸다. 입력 시 SRID 를 명시하고, 변환은 `ST_Transform` 으로 명시적으로 한다.

**ST_Distance 의 좌표 단위** — geometry 의 거리는 좌표계 단위(WGS84 면 도) 다. 미터를 기대했다가 작은 숫자가 나오는 사고가 흔하다. 미터 단위가 필요하면 geography 또는 ST_Transform 으로 미터 좌표계로 옮긴 후 계산한다.

**잘못된 도형 (invalid geometry)** — 자기 교차 폴리곤 같은 잘못된 도형은 함수 결과가 예측 불가다. `ST_IsValid` 로 검증하고 `ST_MakeValid` 로 수정한다.

**큰 폴리곤의 성능** — 수만 개 점으로 구성된 도형(예: 복잡한 행정구역) 의 비교는 느리다. `ST_Simplify` 로 단순화한 사본을 함께 두고 1 차 필터로 활용하는 패턴이 흔하다.



## 면접 대비 체크리스트

1. geometry 와 geography 의 차이가 거리 계산·성능·함수 지원에 어떻게 이어지는가? 어느 자리에 어느 것이 어울리는가?
2. SRID 4326 과 3857 의 차이가 무엇이고, 운영에서 *저장은 4326, 계산은 3857* 패턴이 흔한 이유는?
3. `ST_DWithin` 과 `ST_Distance(...) < N` 이 같은 의미 같지만 운영 결과가 다른 이유는?
4. GiST 인덱스가 공간 데이터를 어떻게 인덱싱하는가? 외접 사각형 1 차 필터의 의미는?
5. 인덱스 인식 함수와 그렇지 않은 함수의 차이를 EXPLAIN 으로 어떻게 확인하는가?
6. SRID 불일치 사고를 피하는 운영 패턴은?
7. 큰 폴리곤의 성능 문제와 그 답(`ST_Simplify`) 은?
