# 성능 모니터링

---

> 운영 PostgreSQL 의 건강 상태를 어떻게 측정하고 무엇을 보는가. `pg_stat_*` 시스템 뷰가 핵심 도구다. `pg_stat_statements` 익스텐션이 쿼리 단위 통계를, `pg_stat_user_tables`·`pg_stat_user_indexes` 가 객체 단위 통계를, `pg_stat_activity` 가 현재 동작을 보여 준다. 본 챕터는 자주 보는 메트릭과 외부 모니터링 도구를 정리한다.



## pg_stat_statements — 쿼리 단위 통계

> 모든 쿼리의 실행 통계를 누적한다. 운영의 첫 번째 모니터링 도구다.

```sql
-- 사전 로드 필요 (postgresql.conf)
-- shared_preload_libraries = 'pg_stat_statements'
-- (수정 후 PostgreSQL 재시작)

CREATE EXTENSION pg_stat_statements;
```

**가장 자주 쓰는 쿼리 — 가장 시간을 많이 쓰는 Top 20**

```sql
SELECT
  substr(query, 1, 100) AS query,
  calls,
  round(total_exec_time::numeric, 2) AS total_ms,
  round(mean_exec_time::numeric, 2) AS mean_ms,
  round((100 * total_exec_time / sum(total_exec_time) OVER ())::numeric, 2) AS pct
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

세 가지 다른 정렬을 함께 본다.

**`total_exec_time`** — 전체 자원을 가장 많이 쓴 쿼리. 전체 부하 최적화의 첫 후보.

**`calls`** — 가장 자주 실행된 쿼리. 한 번에 빠르더라도 100 만 번 호출되면 합계가 크다.

**`mean_exec_time`** — 평균이 가장 느린 쿼리. 사용자 경험 측면의 최적화 후보.

세 정렬이 다른 답을 줄 수 있다. 한 쿼리가 평균은 빠른데 호출 빈도가 압도적이라 전체 시간을 지배할 수 있고, 반대도 가능하다.

**통계 리셋** — 측정 윈도우를 정해 그 안의 동작을 본다.

```sql
SELECT pg_stat_statements_reset();
-- 일정 시간 운영 후
SELECT ... FROM pg_stat_statements ...;
```

운영 패턴은 보통 *평소 동작 baseline 측정 → 변경 적용 → 다시 측정 → 비교* 다. 인덱스 추가·설정 변경의 효과를 정량적으로 본다.



## pg_stat_user_tables — 테이블 단위 통계

> 어떤 테이블이 어떻게 쓰이는지 보여 준다.

```sql
SELECT
  relname,
  seq_scan,        -- 풀 스캔 횟수
  idx_scan,        -- 인덱스 스캔 횟수
  n_live_tup,      -- 살아 있는 튜플 수
  n_dead_tup,      -- 죽은 튜플 수 (VACUUM 대상)
  last_vacuum,     -- 마지막 VACUUM
  last_autovacuum  -- 마지막 autovacuum
FROM pg_stat_user_tables
ORDER BY seq_scan DESC
LIMIT 20;
```

**높은 seq_scan / 낮은 idx_scan 비율** — 인덱스가 부족하거나 옵티마이저가 안 쓰는 신호다. EXPLAIN 으로 확인해 인덱스를 추가하거나 통계를 갱신한다.

**높은 n_dead_tup / n_live_tup 비율** — 부풀어 오른 테이블(table bloat). MVCC 의 죽은 튜플이 누적된 상태다. `VACUUM` 또는 `pg_repack` 으로 정리한다.

**오래된 last_autovacuum** — autovacuum 이 잘 동작하지 않는다는 신호. autovacuum 설정 검토 또는 수동 VACUUM.



## pg_stat_user_indexes — 인덱스 사용 통계

> 어떤 인덱스가 실제로 쓰이는지 보여 준다.

```sql
-- 안 쓰이는 인덱스 (idx_scan = 0)
SELECT
  s.schemaname,
  s.relname AS table_name,
  s.indexrelname AS index_name,
  pg_size_pretty(pg_relation_size(s.indexrelid)) AS size,
  s.idx_scan
FROM pg_stat_user_indexes s
WHERE s.idx_scan = 0
  AND s.indexrelname NOT LIKE '%pkey'
ORDER BY pg_relation_size(s.indexrelid) DESC;
```

미사용 인덱스는 두 가지 비용을 만든다 — 디스크 공간과 INSERT/UPDATE 비용. 정기적으로 점검해 제거한다.

함정 한 가지 — 통계가 PostgreSQL 재시작 후 0 부터 다시 누적된다. 며칠 운영 후 측정해야 의미 있는 데이터가 나온다.



## pg_stat_activity — 현재 동작

> 지금 이 순간 어떤 쿼리가 돌고 있는지 본다.

```sql
SELECT
  pid,
  usename,
  application_name,
  state,
  wait_event_type,
  wait_event,
  query_start,
  now() - query_start AS duration,
  substr(query, 1, 100) AS query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY query_start;
```

**state 값** — `active` (실행 중), `idle in transaction` (트랜잭션 중인데 쿼리 안 보냄), `idle` (트랜잭션 없음).

**`idle in transaction` 이 위험 신호** 다. 애플리케이션이 트랜잭션을 시작하고 안 끝낸 상태다. 그 트랜잭션이 잡고 있는 락 때문에 다른 트랜잭션이 대기한다. 길면 timeout 정책으로 자동 끊는다.

```sql
-- 5 분 이상 idle in transaction 인 세션 종료
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle in transaction'
  AND now() - query_start > interval '5 minutes';
```

운영에서는 `idle_in_transaction_session_timeout` 설정으로 자동화한다.

**wait_event** — 현재 쿼리가 무엇을 기다리는지. `Lock` (다른 트랜잭션의 락 대기), `IO` (디스크 I/O), `Client` (클라이언트 응답 대기). 이 값이 디버깅 힌트가 된다.



## 디스크와 캐시 모니터링

> PostgreSQL 의 메모리 사용과 디스크 I/O 패턴을 본다.

**Buffer Cache Hit Ratio** — 메모리에서 읽은 비율. 95% 이상이 일반적 목표다.

```sql
SELECT
  sum(blks_hit) AS hit,
  sum(blks_read) AS read,
  round(sum(blks_hit) * 100.0 / NULLIF(sum(blks_hit) + sum(blks_read), 0), 2) AS hit_ratio
FROM pg_stat_database;
```

낮은 hit ratio (90% 미만) 는 `shared_buffers` 메모리 부족 신호. 일반적으로 시스템 메모리의 25% 정도가 권장된다.

**테이블·인덱스 크기**

```sql
-- 가장 큰 테이블 Top 20
SELECT
  schemaname,
  relname,
  pg_size_pretty(pg_total_relation_size(schemaname||'.'||relname)) AS total_size,
  pg_size_pretty(pg_relation_size(schemaname||'.'||relname)) AS table_size,
  pg_size_pretty(pg_indexes_size(schemaname||'.'||relname)) AS indexes_size
FROM pg_stat_user_tables
ORDER BY pg_total_relation_size(schemaname||'.'||relname) DESC
LIMIT 20;
```

`pg_total_relation_size` 는 테이블 + 인덱스 + TOAST 합계. 디스크 사용량 분석의 표준 도구다.



## 복제 상태 — pg_stat_replication

> Standby 가 어디까지 따라왔는지 본다.

```sql
SELECT
  client_addr,
  state,           -- 'streaming', 'catchup', etc.
  sent_lsn,        -- 마스터가 보낸 위치
  flush_lsn,       -- standby 가 디스크에 쓴 위치
  replay_lsn,      -- standby 가 적용한 위치
  pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS lag
FROM pg_stat_replication;
```

**Replication lag** 가 핵심 메트릭이다. 복제 지연이 크면 read replica 에서 옛 데이터를 본다([`../../04_distributed/01-03.복제.md`](../../04_distributed/01-03.복제.md) 의 read-your-writes 절 참고).

운영에서는 lag 의 절대값보다 *증가 추세* 가 더 중요하다. 안정 상태에서는 lag 가 일정 범위 안에서 흔들리고, 마스터 부하 폭증·standby 디스크 가득참 같은 사고에서는 lag 가 단조 증가한다.



## 외부 모니터링 도구 — Prometheus + Grafana

> `pg_stat_*` 뷰는 운영 결정의 데이터 원천이지만 매번 직접 쿼리하지는 않는다. 외부 모니터링이 그 데이터를 시각화한다.

표준 조합은 **postgres_exporter** + **Prometheus** + **Grafana** 다.

```
PostgreSQL ─→ postgres_exporter ─→ Prometheus ─→ Grafana
   pg_stat_*    /metrics 노출      시계열 저장    대시보드
```

postgres_exporter 가 `pg_stat_*` 뷰를 주기적으로 쿼리해 Prometheus 형식 메트릭으로 노출한다. Prometheus 가 그것을 시계열로 저장하고, Grafana 가 시각화한다.

**자주 보는 대시보드 항목**

- 초당 트랜잭션 (TPS)
- 활성 커넥션 수
- 캐시 히트 비율
- 슬로 쿼리 (mean_exec_time 상위)
- 테이블 크기 추세
- 복제 지연
- Autovacuum 실행 상태
- 디스크 사용량

**알람 임계값 (예시)**

| 지표 | 경고 | 위험 |
|------|------|------|
| 캐시 히트 비율 | < 95% | < 90% |
| 활성 커넥션 / max_connections | > 70% | > 90% |
| 복제 지연 | > 1 GB | > 10 GB |
| Idle in transaction (5 분 이상) | > 0 | > 5 |
| Long-running query (10 분 이상) | > 0 | > 5 |

운영 환경마다 임계값이 달라야 한다. 본인 시스템의 baseline 을 며칠 측정한 후 그 위에서 임계값을 정한다.



## 모니터링 체크리스트

> 매일 / 매주 점검 항목.

**매일**
- [ ] 슬로 쿼리 Top 10 — `pg_stat_statements`
- [ ] 활성 커넥션 수 — 일별 변동 확인
- [ ] 복제 지연 — 안정 범위 안인지
- [ ] 디스크 사용량 — 80% 미만 유지

**매주**
- [ ] 미사용 인덱스 — `pg_stat_user_indexes`
- [ ] Bloat 가능성 — n_dead_tup / n_live_tup 비율
- [ ] Autovacuum 동작 — last_autovacuum 시점
- [ ] 새로 추가된 슬로 쿼리 — pg_stat_statements 비교

**매월 / 분기**
- [ ] 통계 갱신 (`ANALYZE`)
- [ ] Bloat 정리 (`VACUUM FULL` 또는 `pg_repack`)
- [ ] 인덱스 재검토 — 추가·제거
- [ ] 백업 복구 테스트



## 면접 대비 체크리스트

1. `pg_stat_statements` 의 세 가지 정렬(`total_exec_time`·`calls`·`mean_exec_time`) 이 다른 답을 줄 수 있는 이유는?
2. `pg_stat_user_tables` 의 `n_dead_tup` 가 의미하는 것은? 그것이 누적되면 어떤 운영 사고가 따라오는가?
3. 안 쓰이는 인덱스를 어떻게 찾고 왜 제거해야 하는가?
4. `idle in transaction` 상태가 위험한 이유는? 운영에서 어떻게 자동화하는가?
5. Buffer Cache Hit Ratio 의 일반적 목표값과 그것이 낮을 때의 답은?
6. 복제 지연을 모니터링할 때 절대값보다 추세를 보는 이유는?
7. 표준 모니터링 스택(postgres_exporter + Prometheus + Grafana) 이 어떻게 동작하는가?
