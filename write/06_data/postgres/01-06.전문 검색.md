# 전문 검색

---

> PostgreSQL 의 전문 검색은 단순 LIKE 의 한 자릿수 빠른 대안이다. 단어 단위 토큰화, 어간 추출(stemming), 가중치, 순위 매김(ranking) 같은 기능이 한 엔진 안에 들어 있어 별도 검색 엔진 없이도 대부분의 검색 워크로드를 받아 준다. 본 챕터는 tsvector·tsquery, GIN 인덱스, ts_rank, 한국어 검색의 운영 측면을 정리한다.



## LIKE 의 한계

> 운영 코드 베이스의 첫 검색은 거의 항상 LIKE 다. 그 한계가 어디서 보이는지 명확히 하면 전문 검색의 가치가 분명해진다.

```sql
-- LIKE 검색
SELECT * FROM articles WHERE content LIKE '%postgres%';
```

함정이 넷이다.

**인덱스 활용 어려움** — `LIKE 'prefix%'` 는 B-tree 가 받쳐 주지만 `LIKE '%middle%'` 은 풀 스캔이다. 양쪽 와일드카드가 운영의 흔한 패턴이라 인덱스가 사실상 무의미해진다.

**대소문자 구분** — `LIKE` 는 대소문자를 구분하고, `ILIKE` 는 안 구분하지만 더 느리다.

**단어 경계 무시** — `'postgres'` 검색이 `'postgresql'` 도 매칭한다. 의도한 단어를 정확히 찾기 어렵다.

**어간 미지원** — `'running'` 으로 검색해도 `'run'`·`'ran'` 은 안 잡힌다.

`pg_trgm` 익스텐션 + GIN 인덱스가 LIKE 의 인덱스 한계를 푸는 한 자리 답이다. 양쪽 와일드카드 LIKE 가 인덱스를 타고, 비슷한 단어 검색(fuzzy matching) 까지 가능하다. 단어 분석(어간·불용어) 이 필요하면 다음 절의 tsvector 가 답이다.



## tsvector 와 tsquery — 전문 검색의 기본 단위

> 텍스트를 검색 가능한 형태로 *전처리* 한 결과가 tsvector, 검색어를 표현한 것이 tsquery 다.

```sql
-- 텍스트를 tsvector 로 변환
SELECT to_tsvector('english', 'The quick brown foxes are jumping over the lazy dog');
-- 결과: 'brown':3 'dog':10 'fox':4 'jump':6 'lazi':9 'quick':2

-- 검색어를 tsquery 로
SELECT to_tsquery('english', 'fox & dog');
-- 결과: 'fox' & 'dog'

-- @@ 연산자로 매칭 검사
SELECT * FROM articles
WHERE to_tsvector('english', content) @@ to_tsquery('english', 'postgres & fast');
```

`to_tsvector` 가 하는 일이 셋이다. **토큰화** — 공백·구두점으로 단어를 나눈다. **불용어 제거** — `the`·`is`·`are` 같은 검색에 의미 없는 단어를 뺀다. **어간 추출** — `jumping` → `jump`, `foxes` → `fox` 처럼 단어를 기본형으로 줄인다. 결과 tsvector 가 단어 → 위치 매핑 형태로 정렬되어 검색에 효율적이다.

`to_tsquery` 는 검색어에 같은 처리를 해서 tsvector 와 매칭 가능한 형태로 만든다. `&`(AND), `|`(OR), `!`(NOT), `<->` (직접 인접) 같은 연산자를 지원한다.

**`websearch_to_tsquery`** 는 사용자 친화적 입력을 받는다. 따옴표·공백·OR 같은 표현을 자동으로 tsquery 로 번역한다.

```sql
SELECT websearch_to_tsquery('english', '"hello world" or postgres -slow');
-- 결과: 'hello' <-> 'world' | 'postgr' & !'slow'
```

운영 사용자 검색 입력에서는 `websearch_to_tsquery` 가 거의 항상 답이다. 사용자가 따옴표·OR·`-` 같은 표현을 자연스럽게 쓸 수 있다.



## 저장된 tsvector 컬럼 — 운영 패턴

> 매번 `to_tsvector(content)` 를 실행하지 않고 미리 계산해 컬럼에 저장한다.

```sql
-- 저장된 tsvector 컬럼 (PostgreSQL 12+ 의 generated column)
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    title TEXT,
    content TEXT,
    search_vector TSVECTOR GENERATED ALWAYS AS (
      setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
      setweight(to_tsvector('english', coalesce(content, '')), 'B')
    ) STORED
);

-- GIN 인덱스
CREATE INDEX idx_articles_search ON articles USING gin (search_vector);

-- 검색
SELECT * FROM articles
WHERE search_vector @@ websearch_to_tsquery('english', 'postgres performance');
```

`setweight` 가 핵심이다. 컬럼별로 가중치를 매겨 제목 검색이 본문 검색보다 더 높은 점수를 받게 한다. 가중치는 A(1.0), B(0.4), C(0.2), D(0.1) 네 단계다.

`STORED` generated column 의 가치는 명확하다. 입력 시 자동으로 계산되어 디스크에 저장되고, 검색은 그 컬럼을 직접 참조하므로 매번 토큰화 비용이 사라진다. 트리거를 직접 짤 필요도 없다.



## ts_rank — 검색 결과 순위

> 매칭된 행이 많을 때 어느 행을 먼저 보여 줄지 결정한다.

```sql
SELECT
  title,
  ts_rank(search_vector, query) AS rank
FROM articles, websearch_to_tsquery('english', 'postgres performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 20;
```

`ts_rank` 가 점수를 매기는 기준은 단어 빈도, 위치, 가중치(A/B/C/D)다. `ts_rank_cd`(cover density 변형) 는 매칭된 단어들이 얼마나 가까이 있는지도 본다 — 두 단어가 서로 가까이 등장한 행이 높은 점수를 받는다.

운영에서는 보통 `ts_rank_cd` 가 의미적으로 더 적절한 결과를 준다. "postgres performance" 검색에서 두 단어가 한 문단 안에 함께 나오는 글이 그 두 단어가 글 양 끝에 떨어져 나오는 글보다 관련성이 높다.

```sql
-- 가중치 적용 ts_rank_cd
SELECT
  title,
  ts_rank_cd('{0.1, 0.2, 0.4, 1.0}'::float4[], search_vector, query) AS rank
FROM articles, websearch_to_tsquery('english', 'postgres') query
WHERE search_vector @@ query
ORDER BY rank DESC;
-- 가중치 배열은 [D, C, B, A] 순서
```



## ts_headline — 검색 결과 하이라이트

> 매칭된 부분을 강조 표시한 텍스트를 돌려준다.

```sql
SELECT
  title,
  ts_headline(
    'english',
    content,
    websearch_to_tsquery('english', 'postgres'),
    'StartSel=<mark>, StopSel=</mark>, MaxWords=50, MinWords=20'
  ) AS snippet
FROM articles
WHERE search_vector @@ websearch_to_tsquery('english', 'postgres');
```

옵션이 풍부하다. `StartSel`·`StopSel` 로 강조 태그, `MaxWords`·`MinWords` 로 스니펫 길이, `ShortWord` 로 무시할 짧은 단어 길이 등. 검색 UI 의 표준 도구다.

함정 — `ts_headline` 은 인덱스를 쓰지 않고 *원본 텍스트* 를 다시 처리한다. 매칭된 결과 행에만 적용하므로 `WHERE` 로 행을 먼저 좁힌 뒤 `ts_headline` 을 호출하는 패턴이 거의 항상 답이다. 큰 결과 집합 전체에 적용하면 느려진다.



## GIN vs GiST — 전문 검색 인덱스

> 두 인덱스가 같은 자리에 쓰일 수 있지만 운영 결과가 다르다.

| 항목 | GIN | GiST |
|------|-----|------|
| 검색 속도 | 빠름 | 중간 |
| 인덱스 크기 | 큼 | 작음 |
| 갱신 속도 | 느림 | 빠름 |
| Lossy | No | Yes |

**GIN 이 권장** — 검색 빈도가 갱신 빈도보다 훨씬 높은 일반 자리(블로그·문서 검색·제품 카탈로그)에서 GIN 이 거의 항상 답이다. 검색 정확도가 높고 속도가 빠르다.

**GiST 가 어울리는 자리** — 갱신이 초당 수천 건 이상으로 잦거나 인덱스 크기가 핵심 제약(메모리 제한 환경)인 자리. 다만 GiST 는 lossy 라 인덱스 매칭 후 실제 행 검증이 한 번 더 들어간다.

운영의 첫 답은 GIN 이다. 갱신이 너무 잦아 GIN 이 못 따라가는 사고가 실제로 보일 때만 GiST 를 검토한다.



## 한국어 검색 — 형태소 분석

> 영어와 다르게 한국어는 단어 경계가 명확하지 않다. 기본 dictionary 로는 의미 있는 토큰화가 안 된다.

PostgreSQL 의 영어 dictionary 는 `english` 다. 한국어 dictionary 는 PostgreSQL 코어에 없고 외부 익스텐션이 답이다.

**`pg_bigm`** — bigram(두 글자) 기반. 별도 형태소 분석기 없이도 LIKE 비슷한 부분 일치 검색이 가능하다.

```sql
CREATE EXTENSION pg_bigm;
CREATE INDEX idx_articles_bigm ON articles USING gin (content gin_bigm_ops);
SELECT * FROM articles WHERE content LIKE '%포스트그레%';
```

**`mecab-ko` 기반 익스텐션** — 형태소 분석기를 PostgreSQL 안에 통합한다. 의미 단위 토큰화가 가능해 검색 정확도가 높지만 설치·운영이 복잡하다.

운영 환경 대부분은 형태소 분석이 필요하면 Elasticsearch + Nori 같은 외부 검색 엔진으로 빠지는 답이 더 흔하다. PostgreSQL 안에서 한국어 형태소 검색을 운영 수준으로 받쳐 주려면 별도 학습·운영 비용이 큰 편이다.
