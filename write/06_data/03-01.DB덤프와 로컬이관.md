# DB 덤프와 로컬이관
---
> 개발 DB를 로컬 Docker Compose 로 옮기는 일은 배를 띄우는 일이고, 그 뒤에 따라오는 데이터 신뢰성·개인정보·재현성·팀 표준화는 항로를 정하는 일이다. 처음에는 덤프로 빠르게 이관하되, 장기적으로는 seed 와 마스킹으로 가져가는 단계별 정책이 정답이다.



## 1. 왜 로컬 DB 인가

> 공유 dev DB 에 모두가 붙는 구조는 빠른 시작에는 좋지만, 한 사람이 데이터를 망가뜨리면 모두가 같이 멈춘다.

세 가지 사고 패턴이 반복된다. 

1. 누군가 마이그레이션 스크립트를 검증 없이 돌려서 컬럼이 사라지는 경우다. 
2. 신규 기능이 만든 행이 다른 사람의 테스트 fixture 를 깨는 경우다. 
3. VPN 이 끊기면 부팅 자체가 안 된다.

GitLab 도 이 문제를 인지하고 개발자가 로컬에서 기능을 검증할 수 있도록 별도의 development seed 파일과 Data Seeder 를 운영한다. seed 파일은 로컬 기능 검증용 데이터를 채우는 도구이며, Data Seeder 는 특정 user/group namespace 에 테스트 데이터를 주입하는 harness 로 설명된다(GitLab 공식 문서).

원격 dev DB 의존을 끊으면 얻는 것은 세 가지다.

1. 데이터 오염이 일어나도 자기 컨테이너만 영향을 받는다. `docker compose down -v` 한 번으로 원상복구된다.
2. 네트워크·VPN 없이도 부팅이 된다. 비행기 안에서도 작업할 수 있다는 뜻이 아니라, 회사 네트워크 정책이 흔들려도 개발이 안 멈춘다는 뜻이다.
3. 스키마 마이그레이션을 실제로 적용해 보고 회귀를 확인할 안전한 자리가 생긴다. 운영에 올리기 전 마지막 점검대 역할을 한다.



## 2. MariaDB 덤프 도구 선택

> `mariadb-dump` 와 `mysqldump` 는 같은 뿌리이지만, MariaDB 11 부터는 같은 이름이 아니다.

MariaDB 공식 문서 기준 덤프 도구의 정식 이름은 `mariadb-dump` 다. 

- 과거에는 `mysqldump` symlink 가 함께 제공됐지만, MariaDB 11.0 부터 공식 Docker 이미지에서 이 symlink 가 제거됐다(Docker Hub 공식 이미지 페이지). 컨테이너 내부에서 도구를 부를 때는 가능하면 `mariadb-dump` 를 쓰는 쪽이 안전하다.
- 로컬 macOS 에서 Homebrew 의 `mariadb@10.6` 같이 구버전을 깔아 둔 경우엔 `mysqldump` 가 여전히 동작한다. 이 글에서는 컨테이너 외부(호스트)에서 `mysqldump` 가 동작하는 환경을 전제로 한다. 
- 컨테이너 내부에서 `docker exec` 로 도구를 부를 때는 이름이 다를 수 있다는 점을 기억해 두면 된다.



## 3. 옵션 풀이 — 각 옵션이 없으면 어떤 사고가 나는가

> 옵션을 외우는 것이 목적이 아니라, *없으면 어떤 사고가 나는지* 를 아는 것이 목적이다. 모두 한 번씩은 누군가가 데이고 나서 추가된 옵션이다.

| 옵션 | 의미 | 빠지면 생기는 사고 |
|------|------|----------------|
| `--single-transaction` | InnoDB 한정, 트랜잭션 일관 시점으로 덤프 | 큰 테이블 중간에 쓰기가 끼면 일부 row 만 반영돼 외래키 불일치 |
| `--quick` | row 를 메모리에 모으지 않고 스트리밍 | 큰 테이블을 한 번에 올려 메모리 OOM, 또는 client 가 buffer 채우는 동안 server 가 holds locks |
| `--routines` | 저장 프로시저·함수 포함 | 복원 후 "함수 없음" 으로 비즈니스 로직이 깨짐. 발견은 보통 운영 직전에 |
| `--triggers` | 트리거 포함 (기본은 활성이지만 명시) | 트리거 누락이 INSERT 후 파생 row 미생성으로 이어져 통계가 어긋남 |
| `--events` | 이벤트 스케줄러 객체 포함 | 주기 작업이 누락돼 "왜 이 row 가 안 생기지" 추적이 길어짐 |
| `--hex-blob` | BLOB 컬럼을 16진수 리터럴로 직렬화 | 바이너리가 SQL 텍스트로 깨져 복원이 실패하거나, 운 나쁘면 *조용히 손상* |
| `--default-character-set=utf8mb4` | 덤프·복원 양쪽 클라이언트 문자셋 고정 | 한글·이모지가 `?` 또는 mojibake 로 망가짐. 운영에서는 멀쩡한데 로컬에서만 깨지는 미스테리 |
| `--databases <DB>` | `CREATE DATABASE` + `USE` 구문 포함 | 복원 시 어느 DB 로 들어가야 하는지 모호. entrypoint mount 방식에서 특히 사고 |
| `--no-data` | 데이터 제외, DDL·루틴만 | (옵션) 디버깅·스키마 비교 용. 평상시는 빼지 않는다 |

### 3.1 CLI 예시 — 권장 기본형

평상시 쓰는 한 줄. 환경 변수 `MYSQL_PWD` 로 비밀번호를 넘겨 `ps aux` 노출을 피한다.

```bash
MYSQL_PWD="$DEV_PASS" mysqldump \
  -h "$DEV_HOST" -P "$DEV_PORT" -u "$DEV_USER" \
  --single-transaction \
  --quick \
  --routines --triggers --events \
  --hex-blob \
  --default-character-set=utf8mb4 \
  --databases "$DEV_DB" \
  > ./dump/dev-db_$(date +%Y%m%d_%H%M).sql
```

- 옵션 한 줄에 하나씩 둔 이유는 *나중에 한 줄 추가/삭제* 가 깨끗하게 보이게 하기 위함이다. 한 줄에 모두 몰면 옵션 추가 PR diff 가 어색하게 보인다.

### 3.2 최소 옵션 — 빠른 한 번용(권장하지 않음, 비교용)

```bash
# ❌ 평상시 쓰지 말 것. 발견이 늦은 사고를 만든다.
mysqldump -h "$DEV_HOST" -u "$DEV_USER" -p"$DEV_PASS" "$DEV_DB" \
  > ./dump/dangerous.sql
```

이 형태가 만들 사고는 다음과 같다:

1. `-p"$DEV_PASS"` 가 `ps aux` 에 노출 — 다인 사용 호스트에서는 즉시 누출.
2. `--single-transaction` 누락 — 큰 테이블 중간에 쓰기가 끼면 외래키 불일치.
3. `--routines/--triggers/--events` 누락 — 복원 직후 멀쩡해 보이다가 특정 INSERT 에서 트리거 누락이 드러남.
4. `--default-character-set` 누락 — 환경에 따라 한글이 깨짐.
5. `--databases` 없이 단일 DB 만 지정 — `CREATE DATABASE`/`USE` 미포함이라 entrypoint mount 시 어느 DB 로 들어갈지 모호.

§3 의 표는 이 다섯 가지 사고를 *옵션으로 막은* 결과다.

### 3.3 DDL/루틴만 — 스키마 비교용

```bash
MYSQL_PWD="$DEV_PASS" mysqldump \
  -h "$DEV_HOST" -P "$DEV_PORT" -u "$DEV_USER" \
  --no-data \
  --routines --triggers --events \
  --default-character-set=utf8mb4 \
  --databases "$DEV_DB" \
  > ./dump/schema-only.sql
```

운영과 dev 의 스키마 drift 를 확인할 때 쓴다. `diff <(...) <(...)` 로 두 환경의 `--no-data` 덤프를 비교하면 ALTER 누락을 일찍 잡을 수 있다.

### 3.4 압축 + 격리 환경으로 파이프

대용량 덤프는 디스크에 잠시도 떨어뜨리지 않고 SSH 파이프로 격리 환경에 바로 적재한다. 원본이 작업 PC 에 남지 않는다.

```bash
MYSQL_PWD="$DEV_PASS" mysqldump \
  -h "$DEV_HOST" -P "$DEV_PORT" -u "$DEV_USER" \
  --single-transaction --quick \
  --routines --triggers --events \
  --hex-blob --default-character-set=utf8mb4 \
  --databases "$DEV_DB" \
  | gzip -9 \
  | ssh masking-host \
      'cat > /var/dumps/raw_$(date +%Y%m%d_%H%M).sql.gz'
```

마스킹 작업이 격리 호스트에서 끝난 다음, 마스킹된 결과만 사내 NAS·S3 로 배포한다. 마스킹 정책 상세는 [03-02 §5](./03-02.DB덤프와%20로컬이관%20(심화).md) 참고.

### 3.5 정책 — 평상시는 한 가지로

세 가지 형태가 있지만 일상적으로는 3.1 한 가지만 쓴다. 3.2 는 *왜 안 되는지를 보여 주는 비교용*, 3.3 은 *스키마 drift 확인이 필요할 때만*, 3.4 는 *대용량·민감 데이터를 다룰 때만* 꺼낸다. 동반 스크립트 `operator/core-db/docker/sync-dev-to-local.sh` 는 3.1 형태를 그대로 구현한다.

### 3.6 대소문자 정책 — `lower-case-table-names`

테이블명이 `TB_TRB_TJ_010` 처럼 대문자로 저장되는지, `tb_trb_tj_010` 처럼 소문자로 저장되는지는 *서버 변수* `lower-case-table-names` 가 결정한다. 0/1/2 세 값이 있다.

| 값 | 동작 | 흔한 환경 |
|----|------|----------|
| `0` | 입력한 대소문자 *그대로* 저장·조회. 대소문자 구분 | Linux MariaDB 기본 |
| `1` | 저장 시 모두 *소문자로 변환*. 조회 시도 소문자로 강제 | Docker MariaDB 이미지·macOS 기본 |
| `2` | 저장은 입력 그대로, 조회는 소문자 강제 (대소문자 무구분 비교) | macOS 네이티브 MariaDB |

운영(Linux MariaDB) 이 `0` 이면 로컬도 `0` 으로 맞춘다. 그렇지 않으면 dump 복원 시 `Table 'TPS.tb_xxx' doesn't exist` 가 난다 — 운영에서 `TB_*` 로 저장된 테이블을 로컬이 `tb_*` 로 받아들이지 않거나, 응용 코드의 `SELECT * FROM TB_TRB_TJ_010` 이 매치 실패한다.

```yaml
# docker-compose.yml
command:
  - --lower-case-table-names=0   # 대문자 보존
```

주의 — `lower-case-table-names` 는 **데이터 디렉토리 초기화 시점에 고정** 된다. 이미 한 번 부팅된 컨테이너의 값을 바꾸려면 볼륨 wipe 가 필요하다.

```bash
docker compose down -v   # 볼륨 제거
docker compose up -d     # 새 값으로 재초기화
```

볼륨이 살아 있는 채로 값을 바꾸면 부팅 자체가 실패하거나, *최악의 경우* InnoDB 사전이 손상된다. `--reset` 워크플로(§5.2 참고) 가 필요한 이유다.

컬럼명·alias 는 별개다. MariaDB/MySQL 의 *컬럼 식별자는 OS 와 무관하게 대소문자 무구분* — `USER_NM` 과 `user_nm` 은 같은 컬럼이다. 따라서 컬럼 대소문자를 바꾸기 위해 별도 변수를 설정할 필요는 없다.

### 3.7 부분/증분 dump — 일부 테이블만 최신화

전체 dump 가 6분, 26MB 정도여도 *바뀐 테이블 몇 개만 다시 받고 싶을 때* 가 있다. 두 가지 방향이 있다.

#### 3.7.1 특정 테이블만 dump

`--databases` 대신 *데이터베이스 이름 뒤에 테이블 이름 나열* 한다. 이 형태는 `CREATE DATABASE` 와 `USE` 구문을 포함하지 않으므로 적재 시 대상 DB 를 명시한다.

```bash
# TB_TRB_TJ_010 과 TB_TRB_TK_011 만 받기
MYSQL_PWD="$DEV_PASS" mysqldump \
  -h "$DEV_HOST" -P "$DEV_PORT" -u "$DEV_USER" \
  --single-transaction --quick \
  --hex-blob --default-character-set=utf8mb4 \
  --no-create-info \
  "$DEV_DB" TB_TRB_TJ_010 TB_TRB_TK_011 \
  > ./dump/partial.sql
```

- `--no-create-info` 가 핵심이다. 기존 테이블을 *드롭하지 않고* 데이터만 다시 받는다. CREATE 가 같이 들어오면 ALTER 가 의도치 않게 적용된다.
- 데이터만 받았으니 적재 전에 *해당 테이블만 비우거나* `--replace` 로 PK 충돌을 덮어쓰는 전략을 택한다(아래 3.7.2).

#### 3.7.2 `--where` 절로 변경분만 (증분 dump)

`updated_at`, `MDFD_DATE` 같은 변경 시각 컬럼이 있다면 그 시점 이후 row 만 dump 한다(MySQL 공식 문서, Stack Overflow `incremental backup` 패턴).

```bash
# 직전 동기화 이후 변경된 row 만 받기
LAST_SYNC='2026-05-13 20:00:00'
MYSQL_PWD="$DEV_PASS" mysqldump \
  -h "$DEV_HOST" -P "$DEV_PORT" -u "$DEV_USER" \
  --single-transaction --quick \
  --hex-blob --default-character-set=utf8mb4 \
  --no-create-info \
  --replace \
  --where="MDFD_DATE >= '$LAST_SYNC'" \
  "$DEV_DB" TB_TRB_TJ_010 \
  > ./dump/incr.sql
```

- `--replace` 가 핵심. `INSERT` 대신 `REPLACE INTO` 가 생성되어 같은 PK 가 이미 있으면 덮어쓴다.
- 한계 — 이 방식은 *삭제* 를 잡지 못한다. 운영에서 row 가 사라졌어도 로컬에 그대로 남는다. 삭제까지 따라가려면 *soft delete* 컬럼(`DEL_YN` 등) 이 필요하거나, 주기적 풀 dump 와 섞어 쓴다.
- `--where` 절은 *모든 dump 대상 테이블에 같은 조건* 으로 적용된다. 테이블마다 다른 시각 컬럼이면 테이블별로 dump 를 분리한다.

#### 3.7.3 제외할 테이블 — `--ignore-table`

큰 로그 테이블 한두 개만 빼고 dump 하고 싶을 때.

```bash
MYSQL_PWD="$DEV_PASS" mysqldump \
  -h "$DEV_HOST" -P "$DEV_PORT" -u "$DEV_USER" \
  --single-transaction --quick \
  --hex-blob --default-character-set=utf8mb4 \
  --routines --triggers --events \
  --ignore-table="$DEV_DB.TB_TRB_LOG_BIG" \
  --ignore-table="$DEV_DB.TB_TRB_AUDIT_HIST" \
  --databases "$DEV_DB" \
  > ./dump/no-logs.sql
```

`--ignore-table` 인자는 *DB명.테이블명* 형식이 필요하다. 옵션을 여러 번 반복해서 누적한다.

#### 3.7.4 워크플로 권장

| 상황 | 방식 |
|------|------|
| 매일 또는 매주 정기 동기화 | 전체 dump (`--databases`) + `--reset` |
| 특정 테이블만 빠르게 갱신 (분 단위) | 3.7.1 부분 dump → 해당 테이블 `TRUNCATE` 후 적재 |
| 큰 테이블의 *최근 변경분만* | 3.7.2 `--where` + `--replace` |
| 로그·감사 테이블이 dump 를 비대화 | 3.7.3 `--ignore-table` 로 제외 |

### 3.8 성능 최적화 옵션

평상시 전체 dump 가 분 단위로 끝나면 굳이 손대지 않는다. dump 가 *수십 분~수 시간* 으로 늘면 검토할 옵션들이다.

| 옵션 | 효과 | 주의 |
|------|------|------|
| `--extended-insert` (기본 ON) | 여러 row 를 한 `INSERT` 에 묶어 dump 크기·복원 시간 감소 | diff 가 어려움. 사람이 직접 보려면 `--skip-extended-insert` |
| `--net-buffer-length=16777216` | 클라이언트 통신 버퍼 키워 다중 row INSERT 효율 ↑ (16MB 예시) | 서버 `max_allowed_packet` 도 충분히 커야 함 |
| `--max-allowed-packet=1G` | 큰 BLOB row 가 들어 있는 테이블에서 필수 | 작은 DB 엔 무의미 |
| `--order-by-primary` | PK 순으로 row 정렬해 dump. 복원 시 인덱스 빌드가 빨라짐 | dump 시간은 약간 늘어남 |
| `--compact` | 주석·`SET` 헤더 등 부가 정보 제거 | 디버깅이 어려움. CI 자동화용 |
| `--skip-add-locks` | 테이블별 `LOCK TABLES` 생략 | 복원 중 다른 세션 영향 ↑. 단독 사용 환경에서만 |
| `--tab=DIR` | 테이블별 `.sql`(스키마) + `.txt`(데이터, TSV) 분리 출력 | 병렬 적재 가능, 단 `mariadb`/`mysql` 클라이언트 권한 필요 |

복원 측 안전장치 일시 해제(`FOREIGN_KEY_CHECKS=0`, `UNIQUE_CHECKS=0`, `AUTOCOMMIT=0`)는 dump 자체보다 *복원* 의 속도를 결정한다 — [03-02 §4](./03-02.DB덤프와%20로컬이관%20(심화).md) 참고.

#### 3.8.1 권장 튜닝 우선순위

성능 문제를 만났을 때 손대는 순서는 다음과 같다.

1. **`--quick` 가 켜져 있는가** — 기본 ON 이지만 누락된 형태로 굳어진 스크립트가 있다.
2. **`--single-transaction` + `--quick` 조합** — 메모리·락 부담 최소화.
3. **`--max-allowed-packet` 키우기** — BLOB 가 있다면 1GB 까지.
4. **`--net-buffer-length` 키우기** — 다중 row INSERT 효율.
5. **`--ignore-table` 로 큰 테이블 제외** — 가장 빠른 효과.
6. **`--tab` 으로 병렬 적재** — dump 시간보다 *복원 시간* 이 문제라면.
7. **복원 측 `FOREIGN_KEY_CHECKS=0` 등 일시 해제** — 03-02 §4.

`--tab` 까지 가야 하는 경우는 드물다. 보통 5번까지에서 답이 나온다.



`--single-transaction` 은 InnoDB 같은 트랜잭션 테이블에 유용하며, 큰 테이블에는 `--quick` 과 함께 쓰는 것이 권장된다. 단, 덤프 중 `ALTER TABLE`·`DROP TABLE`·`TRUNCATE TABLE` 같은 DDL 이 같이 돌면 일관성이 깨지거나 실패할 수 있다(MariaDB 공식 문서). 평일 낮 운영 DB 를 떠야 한다면 DBA 와 미리 합의해 DDL 창을 피한다.

`--routines` 와 `--triggers` 의 누락은 발견이 늦다. 복원 직후엔 멀쩡해 보이다가, 특정 INSERT 가 들어왔을 때 트리거가 만들어야 할 파생 row 가 안 생긴다. 운영을 따라 만든 환경에서 행위만 다른 셈이라, "내 PC 에서는 됩니다" 의 전형이다.

문자셋 옵션은 명시하지 않아도 기본값이 `utf8mb4` 지만, 클라이언트 OS·터미널·서버 버전 차이로 인해 한글이 망가지는 사례가 꾸준히 보고된다(MariaDB 문서). 명시하는 비용이 0 에 가까우니 항상 박아 둔다.

비밀번호는 `-p` 또는 `--password=` argv 로 넘기면 `ps aux` 에 노출된다. 보통은 `MYSQL_PWD` 환경변수로 넘기거나, `~/.my.cnf` 의 `[client]` 섹션에 두고 권한을 600 으로 잠근다. 이 글의 동반 스크립트(`operator/core-db/docker/sync-dev-to-local.sh`) 는 `MYSQL_PWD` 방식을 쓴다.



## 4. Docker Compose 로 받기

> MariaDB 공식 이미지의 `MARIADB_*` 환경변수와 `/docker-entrypoint-initdb.d` 동작 두 가지가 거의 모든 로컬 DB 사고의 원인이자 해결이다.

다음은 최소 골격이다:

```yaml
services:
  mariadb:
    image: mariadb:10.11
    container_name: trb-305p-mariadb-local
    ports:
      - "13306:3306"
    environment:
      MARIADB_ROOT_PASSWORD: root
      MARIADB_DATABASE: TPS
      MARIADB_USER: trbuser
      MARIADB_PASSWORD: cloud1234
      TZ: Asia/Seoul
    volumes:
      - mariadb-local-data:/var/lib/mysql
      - ./.dumps/305p-dev:/docker-entrypoint-initdb.d:ro
    command:
      - --character-set-server=utf8mb4
      - --collation-server=utf8mb4_unicode_ci
      - --lower-case-table-names=0
      - --log-bin-trust-function-creators=1
    healthcheck:
      test: ["CMD", "healthcheck.sh", "--connect", "--innodb_initialized"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

volumes:
  mariadb-local-data:
```

이 골격에서 자주 데이는 지점은 세 가지다.

1. 호스트 포트를 운영 기본 3306 그대로 두면 다른 로컬 DB(또는 dev DB 의 같은 포트) 와 혼동해 잘못된 곳에 쿼리를 날리는 사고가 난다. 13306 처럼 분리하면 "어디에 붙은 거지" 의문을 줄일 수 있다.
2. `lower-case-table-names=0` 은 운영(Linux MariaDB) 의 대소문자 보존 정책과 맞춘 값이다 — 운영이 `0`(대문자 보존) 인 305P dev 의 dump 를 그대로 받기 위함이다(상세는 §3.6).
3. `log-bin-trust-function-creators=1` 은 복원 시 함수·트리거 DEFINER 검증을 우회한다 — 운영 DB 객체의 DEFINER 가 로컬에 없을 때 복원이 실패하는 사고를 피한다.

`MARIADB_*` 환경변수는 **데이터 디렉토리가 비어 있을 때만** 적용된다(Docker Hub 공식 이미지 문서). `MARIADB_AUTO_UPGRADE` 를 제외한 모든 초기화 변수가 그렇다. 즉 `MARIADB_DATABASE` 를 바꿨는데 반영이 안 된다면 십중팔구 볼륨이 살아 있어서다. 이 동작은 **§5 복원 방식** 의 entrypoint mount 가 "최초 1회만 실행되는" 것과 같은 메커니즘이다.

healthcheck 는 예전처럼 `mysqladmin ping` 에 기대지 말고 공식 `healthcheck.sh --connect --innodb_initialized` 를 쓴다(MariaDB 공식 문서). `ping` 만으로는 InnoDB 가 아직 복구 중인 시점에 *살아 있다* 고 잘못 보고하는 사례가 있다. 자세한 비교는 [03-02 §3](./03-02.DB덤프와%20로컬이관%20(심화).md) 에서 다룬다.



## 5. 복원 두 방식 — 언제 어느 쪽을 쓰는가

> 운영에서는 두 가지 방식을 섞어 쓴다. 골자는 *컨테이너 외부에서 stdin pipe* 와 *entrypoint mount* 의 차이다.

### 5.1 컨테이너 외부에서 stdin pipe

```bash
docker compose up -d mariadb
docker exec -i trb-305p-mariadb-local \
  mariadb -uroot -proot TPS \
  < ./.dumps/305p-dev/01_full.sql
```

장점은 *언제든 다시* 적용할 수 있다는 것이다. 컨테이너를 죽이지 않고 덤프만 갈아치울 수 있다. 단점은 사람이 명령을 빠뜨릴 수 있다는 점과, 복원 도중 다른 사람이 쿼리를 날리면 어색한 중간 상태가 노출된다는 점이다.

### 5.2 entrypoint mount

```yaml
volumes:
  - ./.dumps/305p-dev:/docker-entrypoint-initdb.d:ro
```

장점은 *부팅하면 자동* 으로 시드된다는 것이다. CI 에서 일회용 DB 를 띄울 때 깔끔하다. 단점은 **최초 볼륨 생성 시 단 한 번만** 실행된다는 점이다(Docker Hub 공식 이미지 문서). `01_full.sql` 을 갈아치워도 컨테이너가 자동으로 다시 적용하지 않는다. 갱신하려면 `docker compose down -v` 로 볼륨을 지운 다음 `up -d` 로 재기동해야 한다.

어느 쪽을 쓸지는 한 기준으로 가른다: *덤프가 자주 갱신되는가, 아니면 한 번 박아 두고 안정적으로 굴리는가*. 사람이 매일 한두 번 dev → local 동기화를 한다면 5.1 이 편하다. CI 에서 PR 별로 일회용 DB 를 띄운다면 5.2 가 자연스럽다. 동반 스크립트는 둘을 섞어 `--reset` 플래그로 5.2 동작(볼륨 wipe + 재시드) 을 묶었고, 평상시는 5.1 처럼 dump 만 갱신할 수도 있게 했다.



## 6. 기업은 어떻게 하는가 — 한 단락 요약

> 정답은 하나가 아니다. 운영 모델은 *seed·덤프·마스킹된 덤프·원격 dev 환경·stub/mock* 다섯 가지 중 둘 이상의 조합이며, Cloud Dev Environment 와 Preview Environment 가 추상 한 단계 위에 얹힌다.

대부분의 팀은 자기 DB 는 dump 또는 seed 로, 외부 API 는 stub·mock 으로, 메시지 브로커는 로컬 컨테이너로 — *의존 종류별로 다른 도구* 를 매핑한다. dump 와 mock 은 경쟁이 아니라 *각자 다른 의존을 끊는 도구* 다. 다섯 가지 모델 비교, Cloud Dev / Preview Environment 추상, 의존 종류별 대체 도구 매핑은 짝 문서에서 다룬다.

> 짝 문서: [03-03. 로컬 개발환경 운영 모델](./03-03.로컬%20개발환경%20운영%20모델.md) — GitLab seed, Shopify Local/Cloud, Toss mock server, Phauer seeding+stub, Cloud Dev / Preview Environment 비교



## 7. 단계별 적용 — 우리는 어디부터 가는가

> 한 번에 seed 까지 가지 말고, 4단계로 점진적으로 올린다. 각 단계의 목표는 *다음 단계로 갈 명분을 만드는 것* 이다.

1. **1단계 — 빠른 이관.** 개발 DB 를 통째로 덤프해 로컬에 복원한다. 이 단계의 목표는 *오늘부터 공유 dev DB 의존을 끊는 것* 이다. 자동화·마스킹·seed 는 아직 신경 쓰지 않는다.
2. **2단계 — 자동화.** 동기화 스크립트, `Makefile` 또는 짧은 sh 한 개, healthcheck 와 charset 설정을 정리한다. 목표는 *다음에도 같은 방식으로 재현 가능* 하게 만드는 것이다. 동료가 같은 절차로 같은 결과를 얻을 수 있어야 한다.
3. **3단계 — 마스킹 정책.** 어느 컬럼이 위험한지 식별하고, 마스킹 SQL 을 정의한다. 마스킹된 덤프만 팀에 배포하고 원본은 즉시 삭제한다. 위험 컬럼 식별 가이드는 [03-02 §5](./03-02.DB덤프와%20로컬이관%20(심화).md) 에 있다.
4. **4단계 — seed 전환.** 덤프 전체 대신 *공통 코드 + 메뉴/권한 + 테스트 사용자* 정도만 seed SQL 로 두고, 장애 재현이 필요한 경우에만 별도 masked dump 를 복원한다. 덤프는 *도구* 로 격하되고, seed 가 *표준* 이 된다.

지금 우리 단계는 1~2 사이다. 3 으로 가는 트리거는 보통 컴플라이언스 요구나 보안 사고이고, 4 로 가는 트리거는 신규 개발자 온보딩 빈도가 증가하는 시점이다.



## 8. 면접 대비 요약

### 한 줄 정의

DB 덤프와 로컬이관은 *공유 dev DB 의존을 끊어 데이터 오염·네트워크 의존·재현성 문제를 해결하기 위한 단계별 정책* 이다. 도구는 `mariadb-dump` + Docker Compose, 방향성은 *덤프 → 마스킹 → seed* 로 진화한다.

### 핵심 포인트 5가지

1. `--single-transaction`, `--quick`, `--routines`, `--triggers`, `--hex-blob`, `--default-character-set=utf8mb4` 가 빠지면 각각 외래키 불일치·OOM·로직 누락·통계 어긋남·바이너리 손상·문자셋 깨짐 사고가 난다.
2. MariaDB 공식 Docker 이미지의 `MARIADB_*` 환경변수와 `/docker-entrypoint-initdb.d` 마운트는 *최초 볼륨 생성 시 단 한 번만* 실행된다. 변경이 반영 안 되면 볼륨부터 본다.
3. 복원 방식은 stdin pipe(자주 갱신용) 와 entrypoint mount(부팅 자동 시드용) 두 가지를 섞어 쓴다.
4. 단계별 적용은 *빠른 이관 → 자동화 → 마스킹 → seed* 의 4단계로 가져간다. 한 번에 끝내려 하면 실패한다.
5. 어느 정책이든 *마스킹되지 않은 운영 데이터가 개발자 PC 에 남지 않게 하는 것* 이 가장 중요한 한 가지 원칙이다.

### 자주 묻는 질문

**Q: 왜 mysqldump 대신 mariadb-dump 인가?**
A: MariaDB 11.0 부터 공식 Docker 이미지에서 `mysqldump` symlink 가 제거됐다. 호환은 유지되지만 정식 이름은 `mariadb-dump` 이고, 컨테이너 내부에서는 가능하면 `mariadb-dump` 를 부르는 쪽이 안전하다.

**Q: `--single-transaction` 만으로 충분한가?**
A: InnoDB 트랜잭션 테이블에는 유용하지만, 덤프 중 DDL(`ALTER TABLE` 등) 이 같이 돌면 일관성이 깨질 수 있다. 운영을 떠야 한다면 DBA 와 DDL 창을 미리 합의한다.

**Q: 덤프 파일을 Git 에 커밋해도 되나?**
A: 안 된다. 운영 데이터에는 평문/암호화 컬럼이 섞여 있고, Git 이력은 사실상 영구 보존이다. `.dumps/` 디렉토리는 gitignore 에 등록하고, 마스킹된 덤프만 사내 NAS·Nexus raw repository·S3 등에 보관한다. 파일명은 `서비스명_환경_YYYYMMDD_HHMM.sql.gz` 처럼 *생성일·기준 환경·마스킹 여부* 가 드러나게 둔다.

**Q: entrypoint mount 로 시드를 걸어 뒀는데 SQL 을 바꿔도 안 먹는다.**
A: 정상 동작이다. `/docker-entrypoint-initdb.d` 는 최초 볼륨 생성 시 한 번만 실행된다. `docker compose down -v && up -d` 로 볼륨을 새로 만들어야 다시 실행된다. 자세한 함정은 [03-02 §2](./03-02.DB덤프와%20로컬이관%20(심화).md).

**Q: 로컬에서만 한글이 깨진다.**
A: 클라이언트·서버·덤프 셋 중 하나라도 utf8mb4 가 아닐 때 발생한다. 덤프는 `--default-character-set=utf8mb4`, 서버는 `--character-set-server=utf8mb4 --collation-server=utf8mb4_unicode_ci`, 클라이언트는 `[client] default-character-set=utf8mb4` 로 세 곳 모두 정렬한다.
