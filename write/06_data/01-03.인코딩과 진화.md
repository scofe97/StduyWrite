# 인코딩과 진화

---

> 코드는 매주 바뀌지만 데이터는 5 년 전에 쓰인 그대로 디스크에 남아 있다. 이 비대칭이 인코딩 포맷 선택의 핵심 질문을 만든다. 어떤 형식으로 직렬화할 것인가, 그리고 그 형식이 시간이 지나면서 어떻게 호환성을 유지하는가. 본 챕터는 JSON·Protocol Buffers·Avro 의 트레이드오프, 스키마 진화의 두 방향(forward·backward), 그리고 데이터가 흐르는 세 가지 모드(DB·서비스·메시지) 의 인코딩 함정을 정리한다.



## 메모리 표현 vs 바이트 표현

> 객체와 포인터로 이뤄진 메모리 구조는 같은 프로세스 안에서만 의미가 있다. 디스크에 쓰거나 네트워크로 보내려면 바이트 시퀀스로 변환해야 한다.

이 변환이 인코딩(encoding) 또는 직렬화(serialization) 다. 역방향은 디코딩(decoding) 또는 역직렬화(deserialization) 라 부른다. 같은 데이터를 두 표현 사이에서 움직이는 변환 비용이 시스템 성능과 호환성을 결정한다.

운영 환경의 첫 갈래는 **언어 내장 직렬화** 다. Java 의 `Serializable`, Python 의 `pickle`, Ruby 의 `Marshal` 같은 길이다. 한 줄이면 끝나서 매력적이지만 실무에서는 거의 권장되지 않는다. 언어 종속이라 다른 언어가 못 읽고, 임의 클래스를 인스턴스화하므로 신뢰할 수 없는 데이터를 받으면 원격 코드 실행으로 이어진다. 버전 호환성도 보통 고려가 빈약하다. 이 모든 함정 때문에 운영 인코딩은 거의 항상 언어 중립 포맷을 쓴다.



## 호환성의 두 방향

> 시스템이 살아 있는 동안 코드는 계속 바뀐다. 그래서 인코딩 포맷은 두 방향의 호환성을 모두 책임져야 한다.

```
       옛 코드 ←─────────────→ 새 코드
        ↓                        ↓
       옛 데이터 ←─────────────→ 새 데이터
```

**Backward compatibility (하위 호환)** 는 새 코드가 옛 데이터를 읽을 수 있어야 한다는 약속이다. 새 컬럼이 없는 옛 데이터를 새 코드가 만났을 때 기본값으로 채우거나 명시적으로 처리할 수 있어야 한다. 보통 어렵지 않다. 옛 데이터의 구조는 새 코드를 짤 때 이미 알려져 있기 때문이다.

**Forward compatibility (상위 호환)** 는 옛 코드가 새 데이터를 읽을 수 있어야 한다는 약속이다. 새 코드가 추가한 새 필드를 옛 코드가 만났을 때 무시하면서도 깨지지 않아야 한다. 더 어려운 이유는 옛 코드를 짤 때 새 필드의 존재를 몰랐기 때문이다.

롤링 업그레이드(rolling upgrade) 환경이 두 방향을 모두 강제한다. 노드 4 대 중 2 대만 v2 이고 나머지는 v1 인 상태가 잠깐 존재한다. 그동안 v2 노드가 v1 데이터를 읽고(backward) v1 노드가 v2 데이터를 읽는다(forward). 한쪽 호환성만 보장하면 업그레이드 도중 오류가 터진다.

상위 호환성의 함정 한 가지는 데이터 손실이다. v2 가 새 필드 `photoURL` 을 포함해 저장한 레코드를 v1 이 읽으면 그 필드를 모르므로 무시한다. v1 이 그 레코드를 수정해 다시 저장하면 `photoURL` 이 사라진다. 이를 막으려면 디코딩 시 알 수 없는 필드를 그대로 보존해 다시 인코딩할 수 있어야 한다. Protocol Buffers 와 Avro 모두 이 동작을 지원한다.



## 텍스트 포맷 — JSON·XML·CSV

> 가장 흔한 인코딩이다. 사람이 읽을 수 있고 거의 모든 언어가 지원한다. 운영 한계도 분명하다.

JSON 의 첫 함정은 **숫자의 모호성** 이다. JSON 은 정수와 부동소수점을 구분하지 않는다. JavaScript 의 `Number` 타입은 64 비트 부동소수점이라 2^53 을 넘는 정수의 정확도가 깨진다. Twitter ID 같은 큰 정수가 클라이언트로 갈 때 문자열로 직렬화하는 우회가 표준이 된 이유다.

두 번째 함정은 **바이너리 데이터의 부재** 다. 이미지나 압축 파일 같은 바이너리는 Base64 로 인코딩해야 하는데, 그 결과 크기가 33% 늘어난다.

세 번째 함정은 **스키마의 부재** 다. JSON Schema 가 있긴 하지만 학습 곡선이 가파르고 도구 지원이 일정하지 않다. 결과적으로 JSON 데이터는 "어떤 필드가 있는지" 가 코드 안의 가정으로만 존재한다. 같은 API 를 호출하는 두 클라이언트가 서로 다른 가정을 갖기 쉽다.

이런 함정에도 JSON 이 외부 API 의 사실상 표준인 이유는 사람이 읽을 수 있고 도구 지원이 압도적이기 때문이다. 같은 데이터가 서비스 경계를 넘으면서 사람이 디버깅할 일이 많은 자리에서는 여전히 JSON 이 안전한 첫 선택이다.



## 바이너리 스키마 — Protocol Buffers·Avro·Thrift

> 같은 데이터를 더 작게, 그리고 컴파일 타임에 검증 가능하게 인코딩하려는 시도다. 스키마 정의가 인코딩의 첫 단계가 된다.

같은 페이로드(`{userName: "Martin", favoriteNumber: 1337, interests: ["daydreaming", "hacking"]}`) 의 인코딩 크기를 비교하면 차이가 분명하다.

| 포맷 | 크기 |
|------|------|
| JSON | 81 bytes |
| MessagePack | 66 bytes |
| Protocol Buffers | 33 bytes |
| Avro | 32 bytes |

Protocol Buffers 와 Avro 가 텍스트 포맷의 절반 미만으로 줄어드는 이유는 두 가지다. 필드 이름을 바이트에 박지 않는다(스키마가 정의하므로) 와 작은 정수에 varint 인코딩을 쓴다. 1337 같은 작은 수는 8 바이트가 아니라 2 바이트면 충분하다는 발상이다.



### Protocol Buffers — 필드 태그 번호

```protobuf
syntax = "proto3";

message Person {
    string user_name = 1;
    int64 favorite_number = 2;
    repeated string interests = 3;
}
```

핵심은 각 필드에 박힌 **태그 번호**(`= 1`, `= 2`) 다. 인코딩된 바이트에는 필드 이름이 빠지고 태그 번호만 들어간다. 태그 번호는 필드의 영구 식별자다.

스키마 진화 규칙이 이 태그 번호 위에서 정의된다. 새 필드는 새 태그 번호로 추가하면 forward·backward 호환을 모두 만족한다. 옛 코드는 모르는 태그를 만나면 그냥 무시한다(forward). 새 코드는 옛 데이터에 그 태그가 없으면 기본값으로 채운다(backward). 필드명은 자유롭게 바꿔도 된다. 태그 번호만 보존되면 호환이 유지된다.

절대 깨면 안 되는 규칙이 두 가지 있다. **태그 번호 재사용 금지** 와 **`required` 필드 변경 금지** 다. 한 번 쓴 태그 번호는 그 필드를 삭제해도 영원히 다른 필드가 못 가져간다. 옛 데이터에 남아 있는 그 태그를 잘못된 타입으로 디코딩하면 바이트가 어긋나면서 데이터가 깨진다.



### Avro — 필드 이름 매칭, 스키마 두 벌

Avro 는 다르게 푼다. 인코딩된 바이트에는 필드 이름도 태그 번호도 없다. 필드들이 정의된 순서대로 값만 나열된다. 그래서 디코딩하려면 인코딩 시점에 쓴 스키마(writer's schema) 가 따로 필요하다.

```avro
record Person {
    string               userName;
    union { null, long } favoriteNumber = null;
    array<string>        interests;
}
```

읽을 때는 두 스키마를 비교한다. **Writer's schema** 는 인코딩 시점, **reader's schema** 는 디코딩 시점이다. Avro 라이브러리가 둘을 매칭해 reader 가 모르는 필드는 무시하고, reader 가 추가한 필드는 기본값으로 채운다. 필드 이름이 매칭 키라 두 스키마의 필드 순서가 달라도 된다.

이 모델의 가치는 **DB 스키마 변경에 자동 따라간다** 는 점에 있다. RDBMS 의 컬럼이 추가·삭제될 때마다 Avro 스키마를 자동 생성하면 되고, 태그 번호 같은 사람의 관리 대상이 없다. CDC 파이프라인(Kafka Connect Debezium 등) 이 Avro 를 선호하는 이유다.

Writer's schema 를 어떻게 전달할지가 운영 질문이다. 큰 파일에는 헤더에 스키마를 박는다(Avro Object Container Format). 메시지 큐에서는 Confluent Schema Registry 같은 별도 서비스가 스키마를 보관하고 메시지에는 스키마 ID 만 박는다. RPC 에서는 연결 설정 시점에 스키마를 협상한다.



### 셋 중 무엇을 고를까

| 자리 | 권장 |
|------|------|
| 외부 API (사람이 디버깅) | JSON + OpenAPI |
| 내부 마이크로서비스 RPC | gRPC + Protocol Buffers |
| Kafka 같은 이벤트 스트림 | Avro + Schema Registry |
| 게임·HFT 등 고성능 | FlatBuffers / Cap'n Proto (zero-copy) |

기준은 단순하다. 스키마가 사람의 손으로 자주 진화하면 Protocol Buffers, DB 스키마처럼 자동 생성이 잦으면 Avro, 사람이 직접 디버깅할 일이 많으면 JSON 이다. 한 시스템 안에서 자리에 따라 다른 포맷을 섞어 써도 무방하다.



## 데이터 흐름 — 세 가지 모드

> 인코딩의 함정은 데이터가 어디로 흐르느냐에 따라 모양이 다르다.

### 1. DB 를 통한 흐름 — "Data outlives code"

DB 에는 5 년 전 v1.0 코드가 쓴 데이터, 1 년 전 v2.0 코드가 쓴 데이터, 어제 v2.5 코드가 쓴 데이터가 한 테이블에 공존한다. 코드가 6 개월마다 바뀌어도 데이터의 유효 수명은 그보다 훨씬 길다. 이를 "data outlives code" 라고 부른다.

이 비대칭이 backward·forward 호환성을 모두 강제한다. 새 코드가 옛 데이터를 읽고(backward), 어떤 이유로 옛 코드가 새 데이터를 만지면(forward), 모두 안전해야 한다. 마이그레이션 한 번으로 옛 데이터를 새 형식으로 통일하는 길도 있지만, 큰 테이블에서는 시간이 오래 걸리고 운영 부담이 크다. 점진적 진화를 받쳐 주는 인코딩이 안전한 선택이다.

### 2. 서비스 간 흐름 — REST·gRPC

마이크로서비스 환경에서는 클라이언트와 서버가 독립적으로 배포된다. 보통 서버를 먼저 올리고 클라이언트를 나중에 올리는 순서를 따른다. 그 사이 시간에 새 서버 + 옛 클라이언트 조합이 동작해야 하므로 요청은 backward 호환(새 서버가 옛 요청 형식을 받음), 응답은 forward 호환(옛 클라이언트가 새 응답 형식을 처리) 이 필요하다.

REST 위에 OpenAPI 스키마를 붙이거나, gRPC 위에 Protocol Buffers 를 박아 두면 호환성 검증이 빌드 시점에 자동으로 들어온다. 운영에서는 스키마 정의가 변경 PR 의 리뷰 포인트가 된다.

RPC 의 본질적 한계도 같이 본다. 로컬 함수 호출과 다른 점이 다섯이다. 네트워크 장애로 결과를 모를 수 있고, 타임아웃은 성공·실패 어느 쪽도 확정하지 못하며, 재시도가 중복 실행을 일으킬 수 있고, 지연 시간이 ms 단위에서 s 단위까지 출렁이며, 모든 인자를 직렬화해야 하므로 포인터를 못 보낸다. RPC API 는 보통 멱등성(idempotency) 을 받쳐 주는 형태로 설계해야 안전하다.

### 3. 메시지 브로커를 통한 흐름

Kafka·RabbitMQ 같은 메시지 브로커는 송신자와 수신자를 시간·공간으로 분리한다. 송신자는 수신자가 누군지 몰라도 되고, 수신자는 송신자가 살아 있는지 몰라도 된다. 브로커가 메시지를 잠시 보관하고 재전송한다.

이 모델에서 스키마 진화는 더 까다롭다. 한 토픽에 여러 버전의 메시지가 시간 순으로 쌓이고, 컨슈머는 자기 코드 버전대로 그것들을 읽는다. 게다가 컨슈머 자체가 여러 팀이라 동시에 같이 업그레이드하는 일이 거의 없다.

이 자리에서 Avro + Schema Registry 가 사실상 표준이 된 이유가 있다. 모든 메시지는 스키마 ID 와 함께 발행되고, 컨슈머는 ID 로 writer's schema 를 조회해 자기 reader's schema 와 매칭한다. 호환성 검증이 발행 시점에 자동으로 들어와 깨진 메시지가 토픽에 흘러드는 사고를 막는다(자세한 메시징 측면은 [`../05_messaging/`](../05_messaging/) 묶음 참고).



## Durable Execution — 워크플로우의 인코딩 측면

> Temporal·Cadence 같은 시스템은 모든 RPC 호출과 결과를 WAL 에 인코딩해 둔다. 그 덕에 워커가 죽었다 살아나도 실행이 정확히 한 번처럼 동작한다.

```python
@workflow.defn
class PaymentWorkflow:
    @workflow.run
    async def run(self, payment: PaymentRequest) -> PaymentResult:
        is_fraud = await workflow.execute_activity(
            check_fraud, payment,
            start_to_close_timeout=timedelta(seconds=15),
        )
        if is_fraud:
            return PaymentResultFraudulent
        # ... 각 activity 가 자동으로 WAL 에 기록됨
```

핵심은 워크플로우 코드 자체가 결정론적(deterministic) 이어야 한다는 점이다. 같은 입력이 들어왔을 때 같은 RPC 호출 시퀀스가 나오지 않으면 재생 시 흐름이 어긋난다. 시스템 시계, 랜덤, 환경 변수 같은 비결정론적 함수는 직접 쓰지 않고 워크플로우 SDK 가 제공하는 결정론적 버전을 쓴다.

외부 서비스도 멱등성을 가져야 한다. WAL 재생 중 같은 결제 요청이 두 번 나가면 사용자 입장에서 중복 청구가 된다. 보통 클라이언트가 발행한 idempotency key 를 서버가 기억해 같은 키의 두 번째 요청을 무시하는 식으로 푼다.



## 실무 주의 사항

> 운영에서 자주 부딪히는 함정 모음.

언어 내장 직렬화는 거의 모든 자리에서 피한다. 보안 함정과 호환성 함정이 모두 따라온다.

JSON 의 큰 정수(2^53 초과) 는 문자열로 직렬화한다. JavaScript 클라이언트가 정확도를 잃는다.

Protocol Buffers 의 태그 번호는 영구 식별자다. 한 번 쓴 번호는 그 필드를 삭제해도 다시 쓰지 않는다. `reserved` 키워드로 재사용을 명시적으로 막을 수 있다.

Avro 사용 시 Schema Registry 는 거의 필수다. 메시지마다 스키마를 함께 보내면 페이로드가 부풀고, 호환성 검증을 빌드 시점에 강제하기도 어렵다.

RPC API 는 멱등성 키를 첫날부터 설계에 포함시킨다. 나중에 추가하려고 하면 모든 호출처 코드를 고쳐야 한다.
