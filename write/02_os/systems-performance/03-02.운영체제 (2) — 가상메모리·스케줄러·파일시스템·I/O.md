# 운영체제 (2) — 가상메모리·스케줄러·파일시스템·I/O
---
> 이 노트는 3장의 배경 후반부로, 커널이 *자원과 I/O를 어떻게 관리하는가* 를 잡습니다. 03-01 이 실행·제어 흐름(커널·시스템 콜·인터럽트·프로세스·스택)이었다면, 여기서는 가상 메모리·스케줄러·파일시스템(VFS·I/O 스택)·캐싱·네트워킹·디바이스 드라이버·멀티프로세서·선점·자원 관리를 봅니다. 각 주제는 뒤 장(6~10장)에서 더 깊이 다뤄지며, 이 노트는 그 지도입니다.

성능 분석가가 "제한된 메모리가 성능에 어떻게 영향을 주나, 파일시스템이 I/O를 어떻게 처리하나"를 물을 때 필요한 자원 측 지식이 여기 모입니다. 03-01 의 실행 흐름 위에서, 커널이 메모리·CPU 시간·디스크·네트워크라는 *자원* 을 스레드들에 어떻게 나눠 주는지가 핵심입니다.

> 이 노트의 주제들은 각각 한 장씩으로 깊어집니다 — 스케줄러는 6장(CPU), 가상 메모리는 7장(Memory), 파일시스템은 8장, 디스크는 9장, 네트워킹은 10장입니다. 여기서는 성능 분석에 필요한 *멘탈 모델* 까지만 잡습니다. 커널 구현·발전사는 03-03 이 이어받습니다.



## 1. 가상 메모리 — 거의 무한한 사적 메모리

> 가상 메모리는 메인 메모리의 추상으로, 프로세스·커널에 거의 무한한 *사적* 메모리 뷰를 줍니다. 멀티태스킹(서로 다툴 걱정 없는 사적 주소 공간)과 오버서브스크립션(메인 메모리를 디스크로 투명 매핑)을 지원합니다.

가상 메모리는 메인 메모리의 추상으로, 프로세스와 커널에 거의 무한한 *사적* 메인 메모리 뷰를 줍니다(64비트 기준 — 32비트는 4GB 한계). 멀티태스킹을 지원해 프로세스·커널이 경합 걱정 없이 자기 주소 공간에서 동작하게 하고, 오버서브스크립션을 지원해 OS가 필요에 따라 가상 메모리를 메인 메모리와 보조 저장(디스크) 사이로 투명하게 매핑하게 합니다. 가상 메모리는 프로세서와 OS의 지원으로 가능하며, 실제 메모리가 아니라 대부분의 OS는 *처음 쓰일(written) 때만* 실제 메모리에 매핑합니다(demand).

#### 메모리 관리 — 스와핑과 페이징

가상 메모리는 보조 저장으로 메인 메모리를 늘리지만, 커널은 가장 활성인 데이터를 메인 메모리에 두려 합니다. 두 방식이 있습니다.

| 방식 | 단위 | 비고 |
|------|------|------|
| 프로세스 스와핑(swapping) | 프로세스 전체 | 원래 Unix 방식, 성능 손실이 큼 |
| 페이징(paging) | 작은 페이지(예: 4KB) | 더 효율적, BSD가 paged 가상 메모리로 도입 |

둘 다 least recently used(또는 not recently used) 메모리를 보조 저장으로 옮기고, 다시 필요할 때만 메인 메모리로 되돌립니다. **Linux에서 "swapping"은 페이징을 가리킵니다** — Linux 커널은 (옛) Unix식 프로세스 전체 스와핑을 지원하지 않습니다(7장).



## 2. 스케줄러 — CPU 시간을 나눈다

> Unix는 시분할 시스템으로, 스케줄러가 스레드(태스크)를 CPU에 매핑해 실행 시간을 나눕니다. 우선순위로 중요한 일을 먼저 돌리고, CPU-bound 워크로드의 우선순위를 낮춰 I/O-bound(저지연이 바람직한)를 먼저 돌립니다.

Unix 계열은 시분할(time-sharing) 시스템으로, 여러 프로세스가 실행 시간을 나눠 동시에 돕니다. 프로세서·CPU에 대한 스케줄링은 커널의 핵심 컴포넌트인 *스케줄러* 가 하며, 스레드(Linux는 태스크)를 CPU에 매핑합니다. 기본 의도는 CPU 시간을 활성 프로세스·스레드에 나누고, *우선순위* 개념으로 더 중요한 일을 먼저 돌리는 것입니다. 스케줄러는 ready-to-run 스레드를 전통적으로 우선순위별 *런큐(run queue)* 로 추적합니다(현대 커널은 CPU별 큐·다른 자료구조도 씀). 실행을 원하는 스레드가 가용 CPU보다 많으면 저우선 스레드가 차례를 기다리며, 대부분의 커널 스레드는 유저 프로세스보다 높은 우선순위로 돕니다.

워크로드는 둘로 나뉩니다.

| 유형 | 특성 | 한계 자원 |
|------|------|----------|
| CPU-bound | 무거운 계산(과학·수학 분석), 긴 실행 시간 | CPU |
| I/O-bound | I/O 위주·적은 계산(웹·파일 서버·대화형 셸), 저지연 바람직 | 스토리지·네트워크 |

Unix 이래 흔한 정책은 *CPU-bound 워크로드를 식별해 우선순위를 낮추는* 것입니다 — 최근 계산 시간(on-CPU) 대 실제 경과 시간의 비가 높은(계산 비율 높은) 프로세스의 우선순위를 낮춰, 저지연이 더 바람직한 I/O-bound(짧게 도는, 대개 I/O·사람 대화형)를 먼저 돌립니다. 현대 커널은 여러 *스케줄링 클래스·정책*(Linux)을 지원해, 실시간 스케줄링(커널 스레드 포함 모든 비핵심 일보다 높은 우선순위)을 선점 지원과 함께 제공해 예측 가능한 저지연 스케줄링을 줍니다(6장).



## 3. 파일시스템 — 전역 네임스페이스와 VFS

> 파일시스템은 데이터를 파일·디렉토리로 조직하고 POSIX 기반 인터페이스로 접근하게 합니다. OS는 루트("/")부터 트리로 전역 네임스페이스를 주고, 파일시스템은 마운트로 트리에 합류합니다. VFS는 파일시스템 유형을 추상화하는 커널 인터페이스입니다.

파일시스템은 데이터를 파일·디렉토리로 조직하고, 보통 POSIX 표준 기반의 파일 인터페이스로 접근하게 합니다. 커널은 여러 파일시스템 유형·인스턴스를 지원하며, 파일시스템 제공은 OS의 가장 중요한 역할 중 하나입니다. OS는 루트("/")에서 시작하는 top-down 트리로 *전역 파일 네임스페이스* 를 주고, 파일시스템은 *마운트*(자기 트리를 마운트 포인트 디렉토리에 붙임)로 합류합니다 — 덕분에 사용자는 밑에 깔린 파일시스템 유형과 무관하게 투명하게 탐색합니다. 대부분은 스토리지(디스크)에 내용을 저장하고, `/proc`·`/dev` 처럼 커널이 동적 생성하는 것도 있습니다. 커널은 프로세스를 네임스페이스 일부로 격리하는 방법(`chroot`, Linux의 mount namespace — 컨테이너에 흔히 씀)도 제공합니다(11장).

#### VFS와 I/O 스택

**VFS(가상 파일시스템)** 는 파일시스템 유형을 추상화하는 커널 인터페이스로, Sun이 UFS와 NFS가 쉽게 공존하도록 개발했습니다. 새 파일시스템 유형을 커널에 더하기 쉽게 하고, 전역 네임스페이스를 제공해 유저 프로그램이 여러 유형을 투명하게 접근하게 합니다.

스토리지 기반 파일시스템에서 유저 SW부터 스토리지 장치까지의 경로가 **I/O 스택** 입니다(전체 SW 스택의 부분집합). 일반 I/O 스택은 위에서 아래로 — 애플리케이션 → 시스템 콜 → VFS → 파일시스템 → 블록 장치 인터페이스 → 스토리지 장치 — 이며, 파일시스템을 우회해 블록 장치로 바로 가는 경로도 있습니다(관리 도구·DB가 가끔 씀). 파일시스템은 8장, 스토리지 장치는 9장에서 다룹니다. I/O 스택과 그 위에서 디스크 I/O를 흡수하는 캐시 층(§4)을 한 장으로 정리하면 다음과 같습니다.

## 4. 캐싱 — 디스크 I/O를 피한다

> 디스크 I/O는 역사적으로 지연이 커, SW 스택의 여러 층이 읽기를 캐싱하고 쓰기를 버퍼링해 회피합니다. 클라이언트·앱·웹서버·DB·디렉토리·메타데이터·OS 버퍼·페이지 캐시·장치·디스크 컨트롤러 등 여러 캐시가 순서대로 점검됩니다.

디스크 I/O는 역사적으로 지연이 커, SW 스택의 많은 층이 읽기 캐싱·쓰기 버퍼링으로 회피합니다. 점검되는 순서대로 본 캐시 층입니다.

| 층 | 캐시 | 예 |
|----|------|-----|
| 1~5 | 클라이언트·앱·웹서버·캐싱 서버·DB 캐시 | 브라우저 캐시·Apache·memcached·MySQL 버퍼 |
| 6~7 | 디렉토리·파일 메타데이터 캐시 | dcache·inode 캐시 |
| 8~9 | OS 버퍼 캐시·파일시스템 1차 캐시 | buffer cache·page cache·ZFS ARC |
| 10~11 | 파일시스템 2차·장치 캐시 | ZFS L2ARC·ZFS vdev |
| 12~15 | 블록·디스크 컨트롤러·스토리지 어레이·온디스크 캐시 | buffer cache·RAID 카드 캐시 |

예로 *buffer cache* 는 최근 쓴 디스크 블록을 담는 메인 메모리 영역으로, 요청 블록이 있으면 디스크 I/O의 높은 지연을 피해 즉시 제공합니다. 어떤 캐시가 있는지는 시스템·환경마다 다릅니다.



## 5. 네트워킹과 디바이스 드라이버

> 커널은 내장 네트워크 프로토콜 스택(TCP/IP 스택)을 제공해 분산 시스템에 참여합니다. 유저 앱은 소켓으로 접근합니다. 디바이스 드라이버는 커널이 물리 장치와 통신하는 SW로, character(raw)·block 인터페이스를 제공합니다.

#### 네트워킹

현대 커널은 내장 네트워크 프로토콜 스택(흔히 쓰는 TCP·IP 때문에 *TCP/IP 스택* 이라 부름)을 제공해, 시스템이 네트워크로 통신하고 분산 환경에 참여하게 합니다. 유저 앱은 프로그래밍 가능한 엔드포인트 *소켓* 으로 네트워크에 접근하고, 네트워크에 연결되는 물리 장치가 *네트워크 인터페이스*(보통 NIC에 제공)입니다. 프로토콜 자체는 자주 안 바뀌지만(채택 느는 QUIC는 예외, 10장), TCP 옵션·혼잡 제어 알고리즘 같은 개선·옵션은 더 자주 바뀌고 보통 커널 지원이 필요합니다(유저 공간 구현 예외). 새 NIC 지원은 새 드라이버를 요합니다(10장).

#### 디바이스 드라이버

커널은 다양한 물리 장치와 *디바이스 드라이버*(장치 관리·I/O용 커널 SW, 보통 HW 벤더가 제공)로 통신합니다. 일부 커널은 재시작 없이 적재·해제되는 pluggable 드라이버를 지원합니다.

| 인터페이스 | 특성 |
|-----------|------|
| character(raw) 장치 | 버퍼 없는 순차 접근, 단일 문자까지 임의 I/O 크기(키보드·시리얼 포트) |
| block 장치 | 블록 단위 I/O(역사적으로 512바이트), 블록 오프셋 기반 랜덤 접근 |

원래 Unix에서 block 장치 인터페이스는 buffer cache로 성능을 높였고, Linux에서 이 buffer cache는 이제 page cache의 일부입니다.



## 6. 멀티프로세서·IPI·선점

> 멀티프로세서 지원은 여러 CPU로 일을 병렬 실행하며, 보통 모든 CPU를 동등하게 보는 SMP로 구현됩니다(NUMA는 소켓별 메모리). CPU 간 조율은 IPI로 빠르게 하고, 선점은 고우선 스레드가 커널을 인터럽트하게 해 실시간 시스템을 가능케 합니다.

#### 멀티프로세서와 IPI

멀티프로세서 지원은 여러 CPU 인스턴스로 일을 병렬 실행하며, 보통 모든 CPU를 동등하게 보는 **SMP(대칭 멀티프로세싱)** 로 구현됩니다 — 병렬 스레드 간 메모리·CPU 공유 때문에 기술적으로 어려웠습니다. 멀티프로세서 시스템엔 소켓(물리 프로세서)마다 다른 메모리 뱅크가 붙는 **NUMA(non-uniform memory access)** 구조도 있어 성능 과제를 냅니다(6·7장). CPU들이 조율해야 할 때(예: 메모리 변환 항목의 캐시 일관성 — 캐시된 항목이 stale임을 다른 CPU에 알림), 한 CPU가 **IPI(inter-processor interrupt, SMP call·CPU cross call)** 로 다른 CPU(들)에 즉시 그 일을 요청합니다 — 빠르게 실행되도록 설계돼 다른 스레드 방해를 최소화합니다. IPI는 선점에도 쓰입니다.

#### 선점(preemption)

커널 선점 지원은 고우선 유저 레벨 스레드가 커널을 인터럽트하고 실행하게 합니다 — 정해진 시간 제약 안에 일을 처리하는 실시간 시스템(항공기·의료 장치)을 가능케 합니다. 선점을 지원하면 *fully preemptible* 이라 하지만, 실제로는 인터럽트 못 하는 작은 임계 경로가 남습니다. Linux는 또 *voluntary kernel preemption*(커널 코드의 논리적 정지점에서 선점 점검·수행)을 지원해, full preemption의 복잡성을 피하면서 흔한 워크로드에 저지연 선점을 줍니다.

| Kconfig 옵션 | 동작 |
|--------------|------|
| `CONFIG_PREEMPT_VOLUNTARY` | 논리적 정지점에서 선점(흔히 활성) |
| `CONFIG_PREEMPT` | 임계 구역 외 모든 커널 코드 선점 가능 |
| `CONFIG_PREEMPT_NONE` | 선점 비활성 — 지연을 키우는 대신 처리량 향상 |



## 7. 자원 관리와 관측성

> OS는 CPU·메모리·디스크·네트워크 접근을 미세 조정하는 자원 제어를 제공해, 다른 앱·여러 테넌트(클라우드)의 성능을 관리합니다. Linux는 cgroups로 발전했습니다. OS는 또 시스템 활동을 관측·분석하는 도구를 포함합니다.

#### 자원 관리(resource controls)

OS는 CPU·메모리·디스크·네트워크 접근을 미세 조정하는 설정 가능한 제어 — *자원 제어* — 를 제공해, 다른 앱을 돌리거나 여러 테넌트를 호스팅하는(클라우드) 시스템의 성능을 관리합니다. 프로세스(또는 그룹)별로 고정 한도를 두거나, 여분을 공유하는 유연한 방식을 씁니다. 초기 Unix·BSD는 기본적인 프로세스별 제어(`nice` CPU 우선순위, `ulimit` 자원 한도)가 있었습니다. Linux는 **cgroups(control groups)** 로 발전해 — 2.6.24(2008)에 통합되고 이후 여러 제어가 더해졌으며 — 4.5(2016)에 개선된 단일 계층 **cgroup v2** 가 나왔습니다. OS 가상화 테넌트 성능 관리가 한 사용 예입니다(11장).

#### 관측성(observability)

OS는 커널·라이브러리·프로그램으로 이뤄지며, 그 프로그램에는 시스템 활동을 관측하고 성능을 분석하는 도구가 있습니다(보통 `/usr/bin`·`/usr/sbin`). 서드파티 도구도 추가 관측성을 줄 수 있습니다. 관측 도구와 그 토대는 4장에서 다룹니다.



## 학습 점검

> 이 노트의 핵심을 스스로 떠올려 봅니다. 답이 막히면 해당 섹션으로 돌아가 확인합니다.

- 가상 메모리가 멀티태스킹과 오버서브스크립션을 어떻게 지원하는지, "Linux의 swapping=페이징"이 무슨 뜻인지 설명해 봅니다. (→ §1)
- 스케줄러가 CPU-bound 워크로드의 우선순위를 낮추는 이유와, I/O-bound가 왜 저지연이 더 바람직한지 말해 봅니다. (→ §2)
- VFS가 왜 필요했고(NFS·UFS 공존), I/O 스택의 위에서 아래 순서(앱→syscall→VFS→파일시스템→블록→스토리지)를 떠올려 봅니다. (→ §3)
- 캐시 층이 "점검되는 순서"로 나열된 이유와, buffer cache가 디스크 I/O를 어떻게 피하는지 설명해 봅니다. (→ §4)
- character(raw)와 block 장치 인터페이스의 차이를 말해 봅니다. (→ §5)
- IPI가 무엇이고 왜 빠르게 실행되도록 설계됐는지, voluntary vs full preemption의 차이를 설명해 봅니다. (→ §6)
- cgroups가 무엇을 하고, 클라우드 테넌트 성능 관리에 어떻게 쓰이는지 떠올려 봅니다. (→ §7)
