# 이미지 취약점 (2) — 스캐닝·CI/CD·제로데이
---
> 취약점이 무엇인지 알았다면(08-01), 이제 내 이미지에 그것이 들어 있는지 자동으로 찾아내야 합니다. 컨테이너에서 패치 과정은 완전히 새로 설계됐습니다. 컨테이너를 불변(immutable)으로 다루면 이미지 하나만 스캔해 그로부터 나온 수백 개 인스턴스의 취약점을 모두 파악할 수 있고, 스캔을 CI/CD 파이프라인 왼쪽으로 당겨(shift-left) 더 싸고 빠르게 이슈를 제거할 수 있습니다. 그래도 막을 수 없는 마지막 범주, 제로데이까지 봅니다.

이 노트는 Chapter 8 의 후반부 — 스캐닝 도구의 동작과 한계, 불변 컨테이너, 정기 스캔, CI/CD 통합, 이미지 갱신, 제로데이 — 를 다룹니다. 짝 노트(08-01)가 "취약점이라는 대상"을 다뤘다면, 이 노트는 "그것을 찾아 막는 행동"입니다.

전통적 호스트 기반 배포에서는 한 호스트의 모든 SW 가 같은 패키지를 공유했고, 보안 팀이 그 패키지를 정기적으로 패치하는 책임을 졌습니다. 컨테이너 기반 배포에서는 각 이미지가 자기 의존성을 품으므로, 취약점 관리를 개발 쪽으로 당겨 개발자가 직접 이미지를 다시 빌드해 고칠 수 있게 됩니다. 이 변화를 따라가 보겠습니다.

> 전제: 무엇이 들어 있는지 식별하는 것이 스캔의 첫 단계입니다 — SBOM 생성(07장)과 같은 작업입니다. 그래서 trivy 같은 스캐너가 SBOM 생성 기능까지 겸합니다.



## 1. 취약점 스캐닝 — 무엇이 들어 있는지 식별

> 이 장에서 다루는 스캐닝은 외부에서 살아 있는 시스템을 찔러 보는 포트 스캐너(nmap·nessus)가 아니라, root 파일시스템에 설치된 SW 를 *검사* 하는 도구입니다. SBOM 을 생성하는 작업과 같은 접근을 씁니다 — 이미지에 든 구성 요소를 이해하는 것이 스캔의 필수 단계이기 때문입니다.

SBOM 이 (아직) 없는 이미지의 취약점을 식별하려면, 먼저 무슨 SW 가 있는지 확인해야 합니다. SW 는 여러 경로로 설치됩니다.

| 설치 경로 | 예 |
|----------|----|
| 리눅스 배포판 root 파일시스템 | 배포판 자체에 취약점이 있을 수 있음 |
| 시스템 패키지 관리자 | `rpm`·`apk` 로 설치한 OS 패키지 |
| 언어별 패키지 관리자 | `pip`·RubyGems 등으로 설치한 의존성 |
| 직접 설치 | `wget`·`curl`·FTP 로 가져온 SW |

> 일부 스캐너는 패키지 관리자에 질의해 설치 SW 목록을 얻습니다. 그런 도구를 쓴다면 SW 를 *직접 설치하지 말아야* 합니다 — 그렇게 깔린 것은 스캔되지 않기 때문입니다.

호스트에 직접 패키지를 까는 일은 새롭지 않습니다. 시스템 관리자가 보안을 위해 패치해 온 것이 바로 그 패키지들입니다. 흔히 각 호스트에 SSH 로 들어가 패치 패키지를 설치하는 방식이었습니다. 클라우드 네이티브 시대에는 이를 지양합니다. 머신 상태를 수동으로 바꾸면 같은 상태로 자동 재생성할 수 없기 때문입니다. 대신 업데이트된 패키지로 새 머신 이미지를 빌드하거나, 프로비저닝 자동화 스크립트를 고치는 편이 낫습니다.



## 2. 이미지를 스캔한다 — 컨테이너가 아니라

> 배포가 취약한 SW 를 담은 컨테이너를 돌리고 있는지 알려면 그 안의 모든 의존성을 스캔해야 합니다. 오늘날 배포에서는 같은 이미지로부터 수백 개 컨테이너 인스턴스가 뜨는 일이 흔합니다. 인스턴스마다 스캔하면 같은 의존성을 수백 번 보게 되니 비효율적입니다. 컨테이너가 파생된 *이미지* 를 스캔하는 편이 훨씬 효율적입니다.

다만 이 접근은 한 가지 전제에 기댑니다 — 컨테이너가 이미지에 있던 SW *만* 돌리고 그 외엔 아무것도 돌리지 않는다는 것, 즉 **컨테이너 안 코드가 불변(immutable)** 이라는 전제입니다. 왜 컨테이너를 불변으로 다루는 것이 좋은지 봅니다.



## 3. 불변 컨테이너와 drift 방지

> 컨테이너가 시작된 뒤 파일시스템에 SW 를 추가로 내려받는 것을 (대개) 막을 장치는 없습니다. 컨테이너 초창기에는 이미지를 다시 빌드하지 않고 최신 SW 로 갱신하는 방법으로 흔히 쓰였습니다. 이 발상이 떠오른다면 곧바로 기억에서 지워야 합니다. 여러 이유로 나쁜 관행으로 여겨집니다.

런타임에 코드를 내려받으면 다음 문제가 생깁니다.

| 문제 | 내용 |
|------|------|
| 재현 불가 | 인스턴스마다 다른 버전을 돌릴 수 있고, 어느 인스턴스가 무엇을 도는지 알기 어려움. 저장된 코드 없이는 동일 복제가 곤란 → 현장 이슈 재현 곤란 |
| SBOM 무의미 | 내용을 바꾸면 이미지의 SBOM 이 실제 컨테이너를 더 이상 표현하지 못함. 라이선스·취약점 결과가 부정확해짐 |
| 출처(provenance) 무력화 | 언제 어디서든 코드를 내려받을 수 있다면 SW 출처를 통제·보장하기 어려움. 이미지의 provenance attestation 이 실제 도는 코드를 반영하지 못해 사실상 무의미해짐 |

이미지를 빌드해 레지스트리에 저장하는 것은 CI/CD 파이프라인에서 자동화하기 쉽고, 같은 파이프라인에 취약점 스캔·공급망 검증 같은 보안 점검을 끼워 넣기도 쉽습니다.

많은 프로덕션 배포가 강제 없이 관행으로만 컨테이너를 불변으로 다룹니다. 스캔 시점 이미지에 없던 실행 파일이 컨테이너에서 실행되는 것을 막아 불변성을 자동 강제하는 도구도 있습니다. 이를 **drift 방지(drift prevention)** 라 하며 Chapter 15 에서 더 다룹니다.

불변성을 얻는 또 다른 방법은 컨테이너를 **읽기 전용 파일시스템** 으로 돌리는 것입니다. 앱 코드가 쓰기 가능한 로컬 저장소를 필요로 하면 쓰기 가능한 임시 파일시스템을 마운트할 수 있는데, 이 임시 파일시스템에만 쓰도록 앱을 수정해야 할 수 있습니다.



## 4. 정기 스캔 — 코드가 안 변해도

> 컨테이너를 불변으로 다루면 이미지마다 한 번만 스캔해 그 안의 모든 취약점을 찾을 수 있습니다. 하지만 한 시점에 한 번 스캔하는 것으로는 부족합니다. 전 세계 보안 연구자들이 기존 코드에서 그동안 발견되지 않던 취약점을 계속 찾아내기 때문입니다. 때로는 수년간 잠복해 있던 이슈를 찾기도 합니다.

대표 사례가 **Heartbleed** 입니다. 널리 쓰이는 OpenSSL 패키지의 심각한 취약점으로, TLS 연결을 유지하는 heartbeat 요청·응답 흐름의 문제를 악용했습니다. 2014년 4월 드러난 이 취약점은, 공격자가 큰 버퍼에 작은 데이터를 요청하는 조작된 heartbeat 요청을 보낼 수 있게 했습니다. OpenSSL 코드에 길이 검사가 없어, 응답은 작은 데이터에 이어 활성 메모리에 마침 들어 있던 내용으로 나머지 버퍼를 채워 돌려줬습니다. 그 메모리에 든 민감 데이터가 공격자에게 반환됐고, 비밀번호·주민번호·의료기록 유출 사고가 Heartbleed 로 추적됐습니다.

Heartbleed 만큼 심각한 사례는 드물지만, 서드파티 의존성을 쓴다면 *언젠가는* 그 안에서 신규 취약점이 드러난다고 가정하는 편이 합리적입니다. 그것이 언제일지는 알 수 없습니다. 내 코드가 안 변해도 의존성 안에서 새 취약점이 드러날 수 있습니다.

> 정기 재스캔은 스캐너가 가장 최신 취약점 지식(NVD·기타 보안 권고)에 비추어 이미지 내용을 점검하게 합니다. 흔한 방식은 새 이미지를 빌드 시 스캔하는 것에 더해, **배포된 모든 이미지를 24시간마다 재스캔** 하는 것입니다.



## 5. 스캔 도구와 결과가 다른 이유

> Trivy·Grype 같은 오픈소스부터 JFrog·Palo Alto·Aqua 의 상용 솔루션까지 컨테이너 이미지 스캐너는 많습니다. Docker Hub(Docker Scout)·Harbor·주요 퍼블릭 클라우드 레지스트리는 스캔을 내장 기능으로 제공합니다. Kubernetes 에서는 Trivy operator 가 배포를 자동 스캔하고 Prometheus 메트릭을 내보내 대시보드에서 확인하게 합니다.

대부분의 도구가 SBOM·VEX 포맷을 입력으로 받아 스캔을 빠르게 하고, CVE 가 단지 존재하는지를 넘어 *악용 가능한지* 까지 반영해 더 적절한 결과를 줍니다. 그런데 스캐너마다 결과가 상당히 다릅니다. 그 이유를 아는 것이 도구 선택의 근거가 됩니다.

| 원인 | 내용 |
|------|------|
| 정보 출처 | 배포판별 보안 권고를 안 쓰고 NVD 만 의존하면 그 배포판 기반 이미지에 거짓 양성이 많아짐. Red Hat 은 OVAL(수정 있는 것만)·Security Data API(미수정 포함)로 둘 이상 |
| 출처의 노후화 | Alpine 이 `alpine-secdb` 갱신을 멈추고 `aports` 메타데이터로 전환 — 구조 변경 시 스캐너 대규모 업데이트 필요 |
| won't-fix 취약점 | 메인테이너가 안 고치기로 한 취약점을 결과에 보일지 말지는 철학적 문제. Aqua 는 사용자 선택 옵션 제공 — "정답인 결과 집합"은 없음 |
| 서브패키지 | `bind` 의 `docs` 서브패키지만 깔렸는데 전체가 깔렸다고 가정 → 거짓 양성 |
| 패키지명 차이 | Debian `shadow` 의 바이너리는 `login`·`passwd`·`uidmap` — 미반영 시 거짓 음성 |
| VEX 입력 | VEX 를 반영하면 악용 불가 취약점을 걸러 거짓 양성 감소 |

## 6. 정적 링크·멀티플랫폼·부가 기능

> 정적 링크된 standalone 바이너리(`scratch` 이미지)는 패키지 메타데이터가 없어 스캔이 까다롭습니다. 동적 라이브러리는 패키지 관리자가 깔아 버전 정보를 메타데이터로 남기지만, 단일 실행 파일을 복사만 하면 그 정보가 없습니다.

다행히 방법이 있습니다. 실행 파일은 ELF 심벌·섹션 헤더 등에 포함 모듈·라이브러리의 단서를 많이 담습니다. `syft` 같은 스캐너는 이를 분석해 빌드된 모듈·라이브러리를 열거한 SBOM 을 만듭니다. 더 정확하게는 컴파일러가 구성 요소를 끌어오며 SBOM 을 생성하는 것입니다 — 최근 Go 는 모듈 정보를 컴파일된 실행 파일의 `.note.go.buildinfo` 섹션에 박아 둡니다.

#### 멀티플랫폼 이미지

이미지는 여러 CPU 아키텍처용 버전을 담을 수 있고(Ch 6), 이미지 매니페스트가 포함 플랫폼을 나열합니다. 플랫폼마다 SW 가 다르므로 ARM 변형의 취약점이 Intel 변형과 다를 수 있습니다. 스캐너는 기본적으로 로컬 아키텍처 변형만 가져오지만, 파라미터로 다른 플랫폼을 지정할 수 있습니다.

```bash
$ trivy image --platform linux/arm64 <image>
```

#### 부가 스캔 기능

많은 스캐너가 이미지 취약점 탐지를 넘어 다음도 점검합니다.

| 점검 대상 | 내용 |
|----------|------|
| 알려진 멀웨어 | 시그니처·휴리스틱(YARA) 기반 탐지 |
| setuid 비트 실행 파일 | 권한 상승 통로(Ch 2) |
| root 실행 설정 | 이미지가 root 로 돌도록 구성됨 |
| 시크릿 자격증명 | 토큰·비밀번호 |
| 민감 데이터 | 신용카드·주민번호 등 |
| 정책 위반 | 노후 의존성·비준수 라이선스 |
| 빈약한 이미지 설정 | `HEALTHCHECK` 누락 등 |
| SBOM 대조 | 기대 구성 요소와 비교 → 악성 구성 요소 추가 탐지 |

> 스캐너에도 버그나 데이터 피드 결함으로 인한 거짓 양성·음성이 있습니다. 그래도 스캐너가 없는 것보다는 낫습니다. 정기적으로 쓸 스캐너가 없으면 내 SW 가 쉬운 악용에 노출됐는지 알 길이 없습니다. Shellshock 은 수십 년 묵은 코드에서 발견됐습니다 — 시간은 이 문제의 치유자가 아닙니다.



## 7. CI/CD 파이프라인 통합 — shift-left

> 파이프라인을 왼쪽(코드 작성)에서 오른쪽(프로덕션 배포)으로 보면, 이슈는 가능한 한 *왼쪽에서 일찍* 제거하는 편이 빠르고 쌉니다. 배포 후 버그를 찾아 고치는 것이 개발 중보다 훨씬 비싼 것과 같은 이치입니다. 컨테이너는 이미지마다 의존성을 품어 앱별 버전 충돌 걱정이 없으므로, 취약점 관리를 파이프라인 왼쪽으로 당길(shift-left) 수 있습니다.

취약점 관리를 더 이상 보안 팀이 수동으로 하지 않습니다. 취약점을 고쳐야 하면 개발자가 패치된 버전을 포함하도록 앱 이미지를 다시 빌드합니다. 스캔을 끼워 넣을 수 있는 자리는 여러 곳입니다.

| 위치 | 내용 |
|------|------|
| 개발자 스캔 | 데스크톱에 쉽게 깔리는 스캐너로 로컬 빌드를 스캔 → push 전에 수정 기회 |
| 빌드 시 스캔 | 이미지 빌드 직후 스캔. 일정 심각도 이상이면 빌드를 *실패* 시켜 배포를 막음 |
| 레지스트리 스캔 | push 후 정기 스캔. SBOM 이 있으면 변경 없는 이미지를 다시 검사할 필요 없이 목록만 취약점 DB 와 대조 → 빠름 |

> 배포 시점까지 스캔을 미루지 않는 것이 좋습니다 — 같은 이미지에서 뜨는 모든 인스턴스를 매번 스캔하게 되기 때문입니다. 불변 전제라면 컨테이너가 아니라 *이미지* 를 스캔해야 합니다.

#### 취약한 이미지가 도는 것을 막기 — admission control

스캐너로 이미지의 취약점 유무를 아는 것과, 취약한 이미지가 배포되지 않게 하는 것은 별개입니다. 후자는 admission control 단계(07-02)에서 합니다. 스캔을 통과한 이미지만 배포되게 하는 점검이 없으면 스캐너를 우회하기 쉽습니다.

Kubernetes 에서는 Kyverno 나 OPA Gatekeeper 로 커스텀 admission 점검을 강제할 수 있습니다 — 이미지가 취약점 스캔을 통과했는지 확인하고, 취약점 점수 임계값을 강제(예: High·Critical CVE 있는 이미지 차단)하며, 신뢰 공급자 서명 여부(07-02)도 점검합니다.



## 8. 이미지 갱신과 제로데이

> 취약하다는 것을 알고 수정 버전이 있다면, 이미지를 다시 빌드해 업데이트된 패키지를 쓰고 재배포해야 합니다. 얼마나 빨리 해야 할까요? 가이드라인은 보통 수 주를 권하지만(FedRAMP 30일, CISA 2019 지침은 Critical 15일), Mandiant 가 본 2023년 평균 악용 시간은 단 5일이었습니다. 전통적 30~45일 "패치 주기" 동안 시도될 수 있는 공격 수를 생각하면 아찔합니다.

신규 수정 취약점이 공개된 시점부터 다음 단계를 거칩니다.

1. 이미지·SBOM 스캔이 업데이트가 필요함을 포착합니다.
2. 코드·lockfile·스크립트를 패치 버전으로 갱신합니다.
3. 컨테이너 이미지를 다시 빌드·테스트합니다.
4. 업데이트 버전을 프로덕션에 배포합니다(rolling upgrade 로 무중단 교체).

GitHub Dependabot 같은 자동화가 취약 패키지 갱신 PR 을 제안해 돕습니다. 이 단계가 진행되는 동안에도 배포는 여전히 취약하므로, 업데이트 코드를 빨리 배포할수록 심각한 침해 가능성이 낮아집니다. 배포 준비를 기다리는 동안 eBPF 로 기존 컨테이너의 취약점을 완화하는 새 접근도 있습니다(Ch 15).

#### 제로데이(0-day) 취약점

> 신규 취약점이 발견된 뒤 수정본이 공개되기까지는 시간이 걸립니다. 수정본 공개 후 0일이 지났다는 뜻에서, 수정 전 취약점을 제로데이라 부릅니다. Mandiant 보고서에 따르면 2023년 악용된 취약점의 70%가 제로데이였습니다.

서드파티 라이브러리에 공격자가 악용할 버그가 있을 수 있듯, 우리 팀이 쓰는 애플리케이션 코드도 마찬가지입니다. 동료 리뷰·정적 분석·테스트가 도움이 되지만 일부 이슈는 빠져나갑니다. 좋은 소식은 취약점이 공개되지 않았다면 세상 대부분의 잠재 공격자도 우리만큼이나 그것을 모른다는 점입니다. 나쁜 소식은 정교한 공격자·국가 조직이 아직 공개되지 않은 취약점 라이브러리를 갖고 있다고 봐야 한다는 점입니다(Snowden 폭로로 확인됨).

아직 공개되지 않은 취약점은 어떤 취약점 DB 대조로도 식별할 수 없습니다. 악용 유형·심각도에 따라 sandboxing(Ch 10)이 앱과 데이터를 보호할 수 있고, 제로데이 방어의 최선은 **런타임에 비정상 행위를 탐지·차단** 하는 것입니다(Ch 15).



## 9. 학습 점검

> 이 노트의 핵심을 스스로 떠올려 봅니다. 답이 막히면 해당 섹션으로 돌아가 확인합니다.

- 컨테이너가 아니라 *이미지* 를 스캔해야 효율적인 이유, 그리고 그 전제가 무엇인지 말해 봅니다. (→ §2)
- 런타임에 코드를 내려받으면 안 되는 세 가지 이유(재현·SBOM·provenance)를 떠올려 봅니다. (→ §3)
- 코드가 안 변해도 정기 스캔이 필요한 이유를 Heartbleed 로 설명해 봅니다. (→ §4)
- 같은 이미지를 스캔해도 도구마다 결과가 다른 원인을 세 가지 이상 들어 봅니다. (→ §5)
- shift-left 가 무엇이고, 컨테이너 구조가 그것을 어떻게 가능하게 하는지 말해 봅니다. (→ §7)
- 제로데이가 일반 취약점과 다른 점, 그리고 DB 대조로 잡을 수 없을 때의 방어 전략을 설명해 봅니다. (→ §8)
