# cgroup 파일시스템 실습 — /sys/fs/cgroup으로 K8s 자원 제어 추적
---
> Pod 매니페스트의 `requests`/`limits` 한 줄이 worker 노드의 어떤 파일에 어떤 값으로 쓰이는가. 그리고 그 파일이 어떻게 throttling과 OOMKill을 강제하는가. 인터페이스 이론([`./01-02.cgroup v2 깊이.md`](./01-02.cgroup v2 깊이.md))을 실습으로 검증하는 장이다.



## 학습 목표
> cgroup v2 트리에서 Pod별 디렉토리를 찾아가 K8s 매니페스트와 1:1로 매핑할 수 있게 만든다.

이 장에서 확인할 목표는 다음과 같다:

1. `/sys/fs/cgroup` 트리 구조를 읽고 system 데몬과 Pod의 cgroup 위치 차이를 설명할 수 있다.
2. `memory.max`, `cpu.max`, `cpu.weight`, `memory.min`이 K8s `requests`/`limits`와 어떻게 매핑되는지 추적할 수 있다.
3. OOM이 단일 동작이 아니라 reclaim → kill 2단계 메커니즘임을 인터페이스 파일로 확인할 수 있다.
4. QoS 클래스별 OOM score 차이를 실측해 노드 압박 시 죽는 순서를 예측할 수 있다.



## 1. cgroup의 본질: "권고가 아닌 강제"

cgroup(Control Groups)은 리눅스 커널이 프로세스 그룹의 자원 사용량을 추적하고 한계를 강제하는 기능이다. 자주 오해하는 부분은 "cgroup은 모니터링 도구"라는 인식인데, 실제로는 모니터링과 enforcement를 같이 한다. `memory.max`에 적힌 값은 hint가 아니라 커널이 SIGKILL로 강제하는 상한이다.

cgroup이 다루는 자원은 네 가지다:

| 자원 | 성격 | 한도 초과 시 |
|------|------|------------|
| CPU | 압축 가능 | throttling(일시 정지) |
| Memory | 비압축 | OOMKill |
| Disk I/O | 압축 가능 | throttling |
| Network | 별도(tc/eBPF로 보조) | drop/queueing |

cgroup이 없는 호스트에서는 한 프로세스가 폭주하면 같은 머신의 다른 프로세스를 굶게 만든다. 이 현상을 "noisy neighbor" 또는 resource starvation이라고 부른다. cgroup은 그룹별로 상한을 걸어 이런 간섭을 차단한다.



## 2. cgroup vs namespace — 다른 일을 한다

같은 컨테이너 기술의 두 기둥이지만 역할이 다르다.

- **namespace**: 가시성 격리. 컨테이너 안에서는 다른 컨테이너의 PID/네트워크/마운트가 보이지 않는다.
- **cgroup**: 자원 한계. 컨테이너가 쓸 수 있는 CPU/메모리/IO의 상한과 보장을 강제한다.

컨테이너 = namespace(격리) + cgroup(한계) + overlay FS(파일시스템). 셋 중 하나만 빠져도 우리가 아는 컨테이너가 아니다. 이 장은 두 번째인 cgroup만 다룬다.



## 3. 압축 가능 자원 vs 비압축 자원

CPU와 Memory는 한도 초과 처리 방식이 정반대다. 이 차이가 인터페이스 파일의 모양에도 그대로 드러난다.

### CPU — 시간으로 나눌 수 있다

CPU 시간은 ms 단위로 잘게 쪼갤 수 있어, 한도를 넘기면 다음 할당 주기까지 대기시키면 된다(throttling). 이를 관리하는 스케줄러가 CFS(Completely Fair Scheduler)다. 인터페이스는 다음과 같다:

```bash
cat /sys/fs/cgroup/<...>/cpu.stat
# usage_usec 12345678
# user_usec  9876543
# system_usec 2469135
# nr_periods 1234
# nr_throttled 56
# throttled_usec 7890
```

`nr_throttled`와 `throttled_usec`은 throttling이 실제로 일어났는지를 알려주는 카운터다. CPU는 시간을 나누면 되므로 "죽이는" 카운터가 없다.

### Memory — 회수할 수 없다

메모리는 한 번 할당되면 시간으로 나누는 개념이 없다. 한도를 넘어 더 할당해야 하는 상황에서 커널이 할 수 있는 일은 두 가지뿐이다: 회수 가능한 메모리(page cache 등)를 비우거나, 아니면 프로세스를 죽이는 것. 그래서 인터페이스에는 "죽임"을 세는 카운터가 있다:

```bash
cat /sys/fs/cgroup/<...>/memory.events
# low 0
# high 0
# max 12
# oom 3
# oom_kill 3
```

`oom_kill`이 3이면 이 cgroup에서 OOM 킬이 세 번 일어났다는 뜻이다. CPU에는 이런 카운터가 없는 이유는 CPU를 "죽이는" 일이 일어나지 않기 때문이다. **인터페이스 파일의 모양은 자원의 성격을 그대로 반영한다.**



## 4. /sys/fs/cgroup 트리 워크스루

cgroup v2의 단일 트리는 `/sys/fs/cgroup`에 마운트된다. worker 노드에 SSH로 들어가 한 단계씩 따라가 보자.

```bash
$ ls /sys/fs/cgroup
cgroup.controllers  cpu.stat            memory.stat
cgroup.procs        init.scope          system.slice
cgroup.subtree_control  io.stat         user.slice
cpu.pressure       memory.pressure     kubepods.slice
```

루트에서 두 가지 큰 갈래가 보인다:

- `system.slice` — kubelet, containerd, sshd 같은 시스템 데몬
- `kubepods.slice` — 이 노드가 실행하는 Pod들

이 분리가 중요하다. 한 Pod의 메모리 누수가 같은 노드의 kubelet을 죽이면 노드 전체가 NotReady가 되니, 두 영역의 cgroup을 나눠 시스템 컴포넌트의 안전을 확보하는 구조다.

```bash
$ cat /sys/fs/cgroup/system.slice/kubelet.service/memory.max
max
$ cat /sys/fs/cgroup/system.slice/kubelet.service/cpu.max
max 100000
```

kubelet은 `max`로 잡혀 있다. 이건 "한도 없음"이라는 의미고, 시스템 데몬으로서 호스트 자원 전체를 쓸 수 있어야 하기 때문이다.

`kubepods.slice` 안에는 QoS 클래스별로 디렉토리가 있다:

```
kubepods.slice/
├── kubepods-besteffort.slice/        # BestEffort Pod 모음
├── kubepods-burstable.slice/         # Burstable Pod 모음
└── kubepods-pod<UID>.slice/          # Guaranteed Pod (별도 그룹화 없이 직접)
```

Guaranteed가 별도 그룹화 없이 `kubepods.slice` 바로 아래에 들어가는 점이 의미심장하다. "전용 자원을 보장한다"는 약속을 디렉토리 구조에서부터 표현한다. BestEffort/Burstable은 같은 등급끼리 묶여 있어 그 그룹 안에서 자원을 다투게 된다.



## 5. K8s 매니페스트 → cgroup 파일 매핑

YAML 한 줄과 cgroup 파일 한 줄을 맞붙여 보자.

| YAML 필드 | cgroup v2 파일 | 의미 |
|-----------|--------------|------|
| `requests.cpu` | `cpu.weight` | 경합 시 보장되는 CPU 비율(상대값, 1~10000) |
| `requests.memory` | `memory.min` | 절대 회수되지 않는 최소 메모리 |
| `limits.cpu` | `cpu.max` | "quota period" 형태 상한, 초과 시 throttling |
| `limits.memory` | `memory.max` | 메모리 상한, 초과 시 OOMKill |

`requests`는 스케줄러의 입력이자 보장의 하한이고, `limits`는 런타임의 강제 상한이다. 둘은 cgroup의 다른 파일에 쓰인다.



## 6. 실습: Pod 배포 후 cgroup 파일 검증

Burstable Pod 하나를 띄워서 위 매핑이 실제로 일어나는지 확인하자.

```yaml
# burstable-demo.yaml
apiVersion: v1
kind: Pod
metadata:
  name: burstable-demo
spec:
  containers:
    - name: app
      image: nginx:1.27
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "200m"
          memory: "170Mi"
```

```bash
kubectl apply -f burstable-demo.yaml
kubectl get pod burstable-demo -o jsonpath='{.metadata.uid}{"\n"}'
# e8a1f2c4-7b6d-4f1a-9c3e-1234567890ab
```

worker 노드로 들어가 Pod UID로 cgroup 디렉토리를 찾는다. systemd는 디렉토리 이름에서 `-`를 `_`로 바꾸므로 변환이 필요하다:

```bash
# UID 'e8a1f2c4-7b6d-4f1a-9c3e-1234567890ab' →
# 디렉토리 'kubepods-burstable-pode8a1f2c4_7b6d_4f1a_9c3e_1234567890ab.slice'

cd /sys/fs/cgroup/kubepods.slice/kubepods-burstable.slice/
ls | grep e8a1f2c4
# kubepods-burstable-pode8a1f2c4_7b6d_4f1a_9c3e_1234567890ab.slice

cd kubepods-burstable-pode8a1f2c4_7b6d_4f1a_9c3e_1234567890ab.slice/
ls
# cri-containerd-<container-id>.scope/
# memory.max  cpu.max  cpu.weight  ...
```

매니페스트와 파일 값을 직접 비교한다:

```bash
$ cat memory.max
178257920
# 170 * 1024 * 1024 = 178,257,920 → 정확히 170 MiB(이진 단위)

$ cat cpu.max
20000 100000
# quota=20000us, period=100000us → 20000/100000 = 0.2 CPU = 200m

$ cat cpu.weight
4
# requests.cpu=100m을 weight로 환산. 경합 시 가중치.

$ cat memory.min
134217728
# 128 * 1024 * 1024 = 128 MiB → requests.memory와 일치
```

`memory.max`는 항상 바이트 단위이며, K8s의 `Mi`(MiB)는 1024^2를 따른다(`M`은 1000^2이지만 K8s는 보통 `Mi`를 권장). 메모리는 page 단위(보통 4KiB)로 정렬되므로 임의의 바이트 수가 그대로 쓰이지는 않고 4096의 배수로 라운딩될 수 있다.

이 시점에 **컨테이너의 자원 제어가 어디에 어떻게 새겨졌는지가 단일 디렉토리에 모두 드러난다**. 이게 cgroup v2가 v1과 비교해 가장 크게 좋아진 점이다(v1은 자원별로 트리가 따로 있었다).



## 7. OOM kill 메커니즘 — 2단계로 일어난다

`memory.max`를 넘긴다고 즉시 SIGKILL이 가는 게 아니다. 커널은 두 단계를 밟는다.

**Step 1 — Reclaim(회수 시도).**
넘기는 순간, 커널은 이 cgroup 안에서 회수 가능한 메모리를 먼저 찾는다. 대표적으로 page cache(파일 읽기 캐시), clean한 익명 페이지(swap이 켜져 있을 때) 등이 대상이다. 회수가 충분하면 새 할당이 허용되고 OOMKill은 일어나지 않는다.

**Step 2 — OOM killer.**
회수가 실패하면(예: swap이 꺼져 있고 page cache도 쥐어짤 게 없으면) cgroup OOM killer가 발동해 SIGKILL을 보낸다. 종료 코드는 137(=128+9). 이때 죽는 건 이 cgroup 안의 프로세스고, 같은 노드의 다른 Pod는 영향을 받지 않는다.

이 cgroup-단위 OOM과 노드 전체 메모리 부족으로 발생하는 system OOM은 다르다. 후자는 `dmesg`에서 oom_score가 가장 높은 프로세스를 골라 죽이며, 노드 전체에 영향을 준다. 평소에 보는 OOMKilled의 대다수는 cgroup OOM이다.



## 8. 실습: 의도적으로 OOM 트리거하기

128Mi 한도에 200Mi를 요구하는 stress Pod로 Step 2를 직접 만들어 보자.

```yaml
# oom-demo.yaml
apiVersion: v1
kind: Pod
metadata:
  name: oom-demo
spec:
  restartPolicy: Never
  containers:
    - name: stress
      image: polinux/stress
      command: ["stress"]
      args: ["--vm", "1", "--vm-bytes", "200M", "--vm-hang", "1"]
      resources:
        limits:
          memory: "128Mi"
```

```bash
kubectl apply -f oom-demo.yaml
kubectl wait --for=condition=Ready pod/oom-demo --timeout=10s || true
kubectl get pod oom-demo
# NAME      READY   STATUS      RESTARTS   AGE
# oom-demo  0/1     OOMKilled   0          5s

kubectl get pod oom-demo -o jsonpath='{.status.containerStatuses[0].lastState.terminated.exitCode}{"\n"}'
# 137
```

worker 노드의 커널 로그를 보면 cgroup OOM이 트리거된 흔적이 남는다:

```bash
$ dmesg | grep -i oom | tail
[ 1234.5678 ] stress invoked oom-killer: gfp_mask=0x..., order=0, oom_score_adj=...
[ 1234.5680 ] memory: usage 131072kB, limit 131072kB, failcnt ...
[ 1234.5683 ] Memory cgroup out of memory: Killed process 12345 (stress) total-vm:..., anon-rss:130048kB
```

세 줄에서 읽을 점:

- "invoked oom-killer" — Step 2 진입
- "memory cgroup out of memory" — system OOM이 아니라 **cgroup OOM**임을 명시
- "anon-rss:130048kB" — 종료 시점 RSS가 약 127 MiB로 한도(128 MiB) 직전까지 갔다

Pod 단위로 발생한 OOM이라 같은 노드의 다른 Pod는 멀쩡하다. 이게 cgroup이 enforcement를 해주는 가장 직접적인 효과다.



## 9. OOM score adjustment — 누가 먼저 죽는가

노드 전체가 메모리 압박을 받을 때(=system OOM 또는 노드 eviction), 커널은 어느 프로세스를 죽일지 골라야 한다. 기준이 `/proc/<pid>/oom_score`이며, 이 점수는 메모리 사용량과 `oom_score_adj`(조정값)로 계산된다. 점수가 높을수록 먼저 죽는다.

Kubernetes는 QoS 클래스에 따라 `oom_score_adj`를 다르게 부여한다:

| 분류 | oom_score_adj | 의미 |
|------|--------------|------|
| BestEffort | +1000 | 가장 먼저 희생 |
| Burstable | 1 ~ 999 (동적) | request/노드 메모리 비율로 결정. 비율이 클수록 낮음(보호) |
| Guaranteed | -997 | 강한 보호 |
| Critical(kubelet, kube-proxy) | -999 | 시스템 전체가 위태로울 때만 |
| Exempt(init, systemd) | -1000 | 절대 보호. 죽이면 커널 패닉 |

미세한 점수 차이는 "사용자 앱 보호 vs 시스템 안정"의 절충이다. Guaranteed가 -997이고 Critical이 -999인 건, 정말 마지막 순간엔 사용자 앱이 시스템 데몬보다 먼저 죽도록 설계됐다는 뜻이다. 노드가 죽는 것보다 한 Pod가 죽는 게 낫기 때문이다.



## 10. 실습: QoS별 OOM score 비교

세 가지 QoS Pod를 띄워 노드에서 점수를 비교하자.

```yaml
# qos-trio.yaml
---
apiVersion: v1
kind: Pod
metadata: {name: qos-besteffort}
spec:
  containers:
    - name: app
      image: nginx:1.27
---
apiVersion: v1
kind: Pod
metadata: {name: qos-burstable}
spec:
  containers:
    - name: app
      image: nginx:1.27
      resources:
        requests: {cpu: "100m", memory: "128Mi"}
        limits:   {cpu: "200m", memory: "256Mi"}
---
apiVersion: v1
kind: Pod
metadata: {name: qos-guaranteed}
spec:
  containers:
    - name: app
      image: nginx:1.27
      resources:
        requests: {cpu: "200m", memory: "256Mi"}
        limits:   {cpu: "200m", memory: "256Mi"}
```

```bash
kubectl apply -f qos-trio.yaml

for p in qos-besteffort qos-burstable qos-guaranteed; do
  echo -n "$p: "
  kubectl get pod $p -o jsonpath='{.status.qosClass}{"\n"}'
done
# qos-besteffort: BestEffort
# qos-burstable:  Burstable
# qos-guaranteed: Guaranteed
```

worker 노드에서 각 컨테이너 PID의 `oom_score_adj`를 확인한다:

```bash
# 컨테이너 PID 찾기 (containerd 기준 예시)
crictl inspect $(crictl ps --name app -q) | jq '.info.pid'

# 각 PID에 대해
cat /proc/<pid>/oom_score_adj
# qos-besteffort:  1000
# qos-burstable:    967   (request/총메모리 비율로 동적 계산)
# qos-guaranteed:  -997
# kubelet PID:     -999
```

이 숫자가 곧 노드가 메모리 압박에 빠졌을 때의 사망 순서다. BestEffort부터 Burstable, 그 다음 Guaranteed 순으로 죽고, kubelet 같은 시스템 데몬은 가장 마지막이다. 운영 관점에서는 미션 크리티컬 워크로드(DB, 인증 서버)는 Guaranteed로 띄우는 게 옳고, 배치/실험은 BestEffort로 두면 노드 압박 시 자연스레 첫 희생자가 된다.
