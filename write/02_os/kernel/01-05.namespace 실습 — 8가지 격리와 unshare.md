# namespace 실습 — 8가지 격리와 unshare
---
> 컨테이너의 격리는 마법이 아니라 `clone` 시스템 콜에 `CLONE_NEW*` 플래그를 묶어 던진 결과다. 이 문서는 `unshare` 한 줄로 같은 결과를 손으로 만들어 보며, 8개 namespace를 "2-3-2" 구조로 외우는 방법까지 정리한다.



## 학습 목표
> Linux namespace의 정의와 K8s namespace와의 차이를 단단히 잡고, 격리 메커니즘을 손으로 재현해 본다.

이 장에서 확인할 목표는 다음과 같다:

1. Linux namespace와 Kubernetes namespace가 이름만 같은 다른 개념임을 설명할 수 있다.
2. `unshare`로 PID·UTS·MNT·NET 격리를 직접 만들고, `/proc/self/ns/*`의 inode 번호로 호스트와 다른 namespace에 들어왔음을 검증할 수 있다.
3. K8s가 사용하는 7개 namespace를 호스트(2)–Pod(3)–컨테이너(2) 그루핑으로 설명할 수 있다.



## 1. Linux namespace는 K8s namespace가 아니다

이름이 같지만 두 개념은 층위가 다르다. Kubernetes의 namespace는 `default`, `kube-system` 같은 **리소스 그룹**이다. Pod, Service, ConfigMap 같은 오브젝트를 묶어 RBAC와 쿼터를 적용하기 위한 **논리적 폴더**고, 클러스터 사용자가 직접 다룬다.

이 장에서 다루는 Linux namespace는 그보다 훨씬 아래에 있다. **커널이 프로세스에게 독립된 시야를 주는 격리 메커니즘**이며, 시스템 콜 `clone`/`unshare`/`setns`로 만들어진다. K8s에서 Pod를 만들면 결국 워커 노드의 커널이 이 Linux namespace를 새로 깎는다.

같은 단어이지만, 클러스터 운영자가 보는 namespace와 커널 개발자가 보는 namespace는 완전히 다른 대상이라는 점을 먼저 잡고 시작한다.



## 2. 영상이 던진 다섯 질문

원본 영상은 다음 다섯 질문으로 시작한다. 다섯 답은 모두 하나의 메커니즘으로 수렴한다.

1. 같은 Pod 안 사이드카 컨테이너에 왜 `localhost`로 닿는가
2. 다른 컨테이너의 프로세스가 왜 안 보이는가
3. Pod 안 컨테이너가 죽고 재생성돼도 IP가 왜 유지되는가
4. 컨테이너 안에서 `ps`를 치면 왜 항상 PID 1이 자기 프로세스인가
5. 컨테이너에서 `hostname`이 왜 노드 이름이 아니라 Pod 이름인가

각 질문은 NET, PID, NET(공유), PID, UTS namespace로 차례대로 답이 된다. 다섯 가지가 하나의 도구로 풀린다는 사실이 namespace를 배우는 동기가 된다.



## 3. namespace의 정의와 호스트의 초기 namespace

**namespace는 같은 커널을 공유하면서 각 프로세스에 독립된 시야를 제공하는 커널 기능이다**. 프로세스 A는 자신만 존재한다고 느끼고, 프로세스 B도 자신만 존재한다고 느낀다. 실제로는 한 커널 위에서 같이 돌아가지만 서로를 보지 못한다.

호스트의 시스템 데몬은 모두 같은 초기 namespace에 들어 있다. `/proc/<pid>/ns/<type>`은 그 프로세스가 속한 namespace의 inode 번호를 가리키는 심볼릭 링크다.

```bash
$ readlink /proc/1/ns/pid
pid:[4026531836]

$ readlink /proc/$(pgrep kubelet)/ns/pid
pid:[4026531836]

$ readlink /proc/$(pgrep containerd)/ns/pid
pid:[4026531836]
```

세 데몬 모두 `4026531836`이라는 같은 inode를 가진다. **같은 방에 있다는 뜻**이고, 따라서 서로 보이고 권한이 있다면 `kill`도 가능하다. 컨테이너라는 건 이 공유 공간에서 프로세스 하나를 떼어내 별도 namespace로 옮겨놓는 행위다.



## 4. PID namespace의 세 가지 특성

PID namespace는 영상에서 가장 깊게 파고드는 부분이다. 세 가지 특성을 알면 컨테이너의 라이프사이클이 한 번에 이해된다.

### 4.1 이중 매핑

같은 프로세스가 호스트 시점 PID와 namespace 내부 PID를 동시에 가진다. 호스트에서 보면 PID 23,400인 프로세스가 컨테이너 안에서는 PID 1로 보인다.

이게 가능한 이유는 커널의 `struct pid` 구조체가 namespace별로 PID를 따로 관리하기 때문이다. 한 프로세스의 `task_struct`는 자기가 속한 모든 PID namespace 각각에서 별개의 번호를 부여받는다. 어느 시점에서 보느냐에 따라 같은 프로세스의 번호가 달라진다.

### 4.2 일방 통행

호스트는 namespace 안을 본다. 안에 있는 프로세스는 호스트 PID로 식별 가능하고 권한이 충분하면 `kill`도 가능하다. 반면 namespace 안에서는 바깥이 보이지 않는다.

```bash
# 컨테이너 안에서
$ kill -0 2153
bash: kill: (2153) - No such process
```

호스트 PID 2153인 containerd가 namespace 내부에서는 존재 자체가 없다. 시야가 한 방향으로만 뚫려 있다.

### 4.3 PID 1 생명줄

namespace의 PID 1이 종료되면 그 namespace 전체가 함께 소멸한다. 이는 단순한 정리가 아니라 커널이 의도적으로 강제하는 동작이다. 커널 소스 `kernel/pid_namespace.c`의 `zap_pid_ns_processes()`가 PID namespace 안의 모든 프로세스에 SIGKILL을 보낸다.

> 컨테이너 메인 프로세스가 죽으면 컨테이너 전체가 죽는 이유가 여기에 있다. Docker나 K8s가 그렇게 만든 게 아니라, 커널이 PID namespace의 1번을 그렇게 다룬다.

`kill`은 보통 한 프로세스를 종료하지만 `zap`이라는 단어는 "한꺼번에 강하게 정리한다"는 뉘앙스다. 커널 코드에서 흔하게 보이는 표현이며, PID namespace가 살아 있을 수 있는 조건이 PID 1의 생존이라는 사실을 강조한다.



## 5. hostPID:true의 보안 함의

Pod 매니페스트의 `spec.hostPID: true`는 컨테이너를 **호스트의 PID namespace에 합류시키는** 옵션이다. 합류한 컨테이너는 호스트와 같은 방에 들어가게 되고, 따라서 같은 노드의 kubelet, containerd, sshd를 전부 볼 수 있다. 권한만 충분하면 `kill`로 강제 종료까지 가능하다.

이 옵션이 활성화된 컨테이너에 공격자가 침투하면, 그 시점부터 노드의 시스템 데몬이 공격면이 된다. 그래서 K8s의 Pod Security Standards는 `baseline`/`restricted` 프로파일에서 `hostPID`를 차단하며, 운영 환경에서는 사용을 금한다.

> 디버깅 목적으로 임시 사용하는 경우에도 PSA 예외 namespace 안에서만 허용하고, 작업 종료 후 즉시 회수하는 게 바람직하다.



## 6. 8개 namespace 카탈로그

Linux 커널이 제공하는 namespace는 8개다. K8s는 그중 `time`을 제외한 7개를 사용한다.

| # | namespace | clone 플래그 | 격리 대상 | 도입 |
|---|-----------|-------------|----------|------|
| 1 | PID | `CLONE_NEWPID` | 프로세스 ID 트리 (PID 1, ppid 등) | 2.6.24 |
| 2 | NET | `CLONE_NEWNET` | 네트워크 스택 (인터페이스, 라우팅, 소켓) | 2.6.24 |
| 3 | MNT | `CLONE_NEWNS` | 마운트 테이블 | 2.4.19 |
| 4 | UTS | `CLONE_NEWUTS` | hostname, domainname | 2.6.19 |
| 5 | IPC | `CLONE_NEWIPC` | 공유 메모리, 세마포어, 메시지 큐 | 2.6.19 |
| 6 | USER | `CLONE_NEWUSER` | UID/GID 매핑 | 3.8 |
| 7 | CGROUP | `CLONE_NEWCGROUP` | cgroup 계층 시야 | 4.6 |
| 8 | TIME | `CLONE_NEWTIME` | 시계(clock offset) | 5.6 |

마운트 namespace의 플래그가 `CLONE_NEWMOUNT`가 아니라 `CLONE_NEWNS`인 점이 특이하다. 2002년 마운트가 **첫 번째이자 유일한** namespace였을 때의 명명을 그대로 유지한 것이고, 이후 다른 namespace가 추가됐어도 하위 호환성 때문에 바꿀 수 없었다. 커널 코드에 자주 등장하는 역사적 화석이다.

CGROUP namespace는 cgroup 자체와 다르다. cgroup은 CPU/메모리 같은 자원을 **제한**하는 기능이고, CGROUP namespace는 그 cgroup 계층의 **시야**를 격리한다. 컨테이너 안에서 `/sys/fs/cgroup`을 봤을 때 자기 cgroup이 루트처럼 보이게 만드는 것이 namespace 쪽이다.



## 7. 외우는 법: 2-3-2 그루핑

7개를 한 번에 외우려면 잘 안 된다. 격리 경계가 어디에 그어지는지를 기준으로 묶으면 머리에 남는다.

```
호스트 노드
├── [USER, CGROUP]              ← 2개. 노드 보안·통제
│
├── Pod (Pause 컨테이너가 잡고 있음)
│   ├── [NET, UTS, IPC]         ← 3개. Pod 안 컨테이너끼리 공유
│   │
│   ├── 컨테이너 A
│   │   └── [PID, MNT]          ← 2개. 컨테이너마다 따로
│   │
│   └── 컨테이너 B
│       └── [PID, MNT]          ← 2개. 컨테이너마다 따로
```

| 층위 | namespace | 의미 |
|------|----------|------|
| 호스트(2) | USER, CGROUP | 컨테이너 탈출 시 호스트로의 권한 상승을 막고, cgroup 시야를 격리 |
| Pod(3) | NET, UTS, IPC | Pod 단위로 단일 IP/hostname/공유 메모리 — Pod 안 컨테이너가 `localhost`로 통신 가능 |
| 컨테이너(2) | PID, MNT | 컨테이너마다 자기 PID 1과 root 파일시스템 |

이 구조가 다섯 가지 질문의 답을 한 번에 설명한다. 같은 Pod의 사이드카에 `localhost`로 닿는 건 NET namespace 공유 때문이고, 컨테이너 안 PID 1이 자기 프로세스인 건 PID namespace가 컨테이너 단위로 분리되기 때문이다.



## 8. IPC namespace — 공유 메모리와 세마포어

IPC namespace는 공유 메모리(shared memory), 세마포어, 메시지 큐를 격리한다. 같은 Pod의 컨테이너끼리는 IPC namespace를 공유하므로 커널 메모리의 한 영역을 함께 읽고 쓸 수 있다. 이게 "zero-copy" 통신이라 불리는 형태다.

리눅스의 기본 원칙은 격리다. 보통 프로세스 A는 프로세스 B의 메모리를 들여다볼 수 없고, 데이터를 주고받으려면 네트워크 소켓이나 파이프로 커널을 두 번 거쳐 복사해야 한다. 데이터가 클수록 이 복사가 비싸진다. 공유 메모리는 커널 RAM의 한 구역을 양쪽 프로세스가 직접 매핑하게 만들어 복사 횟수를 0으로 만든다.

대신 race condition이 생긴다. A와 B가 같은 위치에 동시에 쓰면 데이터가 깨진다. 이를 막는 동기화 장치가 세마포어다.

```
1. 커널이 세마포 값을 1(사용 가능)로 둔다
2. A가 값을 확인하고 0(사용 중)으로 바꾼 뒤 메모리에 쓴다
3. B가 동시에 시도하면 값이 0인 걸 보고 대기한다
4. A가 끝나며 값을 1로 되돌리면 B가 잠금을 잡고 작업한다
```

운영자가 직접 세마포어를 다룰 일은 거의 없지만, "왜 IPC namespace가 Pod 단위로 공유되는가"의 근거가 된다. Pod 안 컨테이너끼리 빠르게 통신해야 한다는 의도와, 그 통신이 안전하려면 동기화 장치를 같이 보고 있어야 한다는 요구가 맞물린 결과다.



## 9. 실습 1 — `unshare`로 PID namespace 만들기

여기서부터는 손으로 따라간다. 컨테이너 런타임 없이 `unshare` 한 줄로 격리를 만든다.

```bash
$ sudo unshare --pid --mount --fork --mount-proc bash
# 새 PID namespace 안 셸로 들어왔다

# echo $$ 는 현재 셸의 PID. namespace 안이라 1번이 된다.
$ echo $$
1

# ps -ef 는 procfs를 읽어 프로세스 목록을 만든다.
# 격리된 procfs를 새로 마운트했으므로 호스트 프로세스는 보이지 않는다.
$ ps -ef
UID   PID  PPID  C STIME TTY    TIME CMD
root    1     0  0 12:34 pts/0  00:00:00 bash
root    7     1  0 12:34 pts/0  00:00:00 ps -ef

# 호스트의 containerd PID(예: 2153)에 신호를 보내려 해도
# 이 namespace 안에는 그 PID가 존재하지 않는다.
$ kill -0 2153
bash: kill: (2153) - No such process

# 자기 namespace의 inode 번호 — 호스트와 다르다.
$ readlink /proc/$$/ns/pid
pid:[4026532674]

$ exit
# PID 1(이 셸)이 종료되면 namespace 전체가 zap_pid_ns_processes로 정리된다.
```

`--mount-proc`이 왜 필요한지가 함정이다. PID namespace를 새로 만들었어도 `/proc`이 호스트의 procfs를 그대로 가리키고 있으면 `ps`가 호스트 프로세스 목록을 그대로 보여준다. `procfs`는 커널이 메모리 위에 실시간으로 만드는 가상 파일시스템이고, 어느 PID namespace 시점으로 만들지를 마운트 시점에 결정한다. `--mount-proc`은 새 namespace에 맞는 procfs를 `/proc`에 다시 마운트해 주는 옵션이고, 이게 없으면 PID 격리가 보이지 않는다.

`--fork` 옵션도 빠뜨리면 안 된다. 이걸 빼면 현재 셸이 새 namespace로 옮겨가지 않고 자식만 옮겨가서, 부모 셸은 호스트에 남고 흐름이 어긋난다.



## 10. 실습 2 — UTS + MNT namespace로 컨테이너 흉내

```bash
$ sudo unshare --uts --mount --fork bash

# UTS namespace가 분리됐으므로 hostname을 바꿔도 호스트에 영향이 없다.
$ hostname my-nginx-container
$ hostname
my-nginx-container

# MNT namespace 안에서 tmpfs를 마운트한다.
$ mount -t tmpfs tmpfs /mnt
$ echo "DB_PASSWORD=secret1234" > /mnt/secret.txt
$ cat /mnt/secret.txt
DB_PASSWORD=secret1234

$ exit

# 호스트 셸로 돌아왔다. 호스트의 hostname은 그대로다.
$ hostname
control-plane

# 그리고 호스트의 /mnt에는 secret.txt가 존재하지 않는다.
$ cat /mnt/secret.txt
cat: /mnt/secret.txt: No such file or directory
```

`exec` 키워드를 통해 셸을 갈아끼우면 프롬프트가 즉시 새 hostname으로 바뀌는데, 이게 `kubectl exec`로 Pod에 들어갔을 때 프롬프트가 Pod 이름인 이유와 같은 메커니즘이다.



## 11. 실습 3 — NET namespace

```bash
$ sudo unshare --net --fork bash

# 새 네트워크 namespace는 인터페이스가 비어 있다.
$ ip addr show
1: lo: <LOOPBACK> mtu 65536 ...   # 루프백조차 down 상태

$ ping -c1 8.8.8.8
ping: connect: Network is unreachable

$ readlink /proc/$$/ns/net
net:[4026532666]                  # 호스트와 다른 inode
$ exit
```

새로 만든 네트워크 namespace는 외부로 나가는 인터페이스가 아예 없다. 막 생성된 Pod의 모습이 정확히 이거다. 이 상태에서 호스트와 연결하려면 가상 케이블 한 쌍(veth pair)을 만들어, 한쪽 끝은 Pod의 net namespace에 두고 다른 쪽 끝을 호스트의 브리지에 꽂아야 한다. 이 일을 K8s에서는 CNI 플러그인(Calico, Cilium 등)이 한다.

`unshare --net`으로 만든 namespace는 셸 종료와 함께 사라진다. 영구적으로 유지하려면 `ip netns add <name>`을 사용한다:

```bash
$ sudo ip netns add eth_ns
$ ip netns list
eth_ns
$ sudo ip netns exec eth_ns ip addr show
$ sudo ip netns delete eth_ns
```

이 형태가 CNI 디버깅이나 직접 라우팅 테스트에 유용하다. veth pair 생성·연결과 Pod 트래픽 경로의 자세한 그림은 [`../../09_cloud/kubernetes/04-02.Pod 네트워크와 Linux 기반.md`](../../09_cloud/kubernetes/04-02.Pod 네트워크와 Linux 기반.md) §4를 참고한다.



## 12. K8s에서 같은 일이 어떻게 일어나는가

`unshare`는 학습용 짧은 길이고, K8s는 같은 도착점으로 가는 긴 길을 쓴다. `kubectl run nginx` 한 줄이 커널에 도달하는 경로는 다음과 같다.

```
kubectl run
  → kube-apiserver (etcd에 Pod 오브젝트 저장)
  → kube-scheduler (노드 결정)
  → kubelet (워커 노드에서 변경 감지)
  → containerd (CRI gRPC)
  → containerd-shim (컨테이너 라이프사이클 부모)
  → runc (OCI 런타임)
  → clone(CLONE_NEWPID|CLONE_NEWNET|CLONE_NEWNS|CLONE_NEWUTS|CLONE_NEWIPC|...)
  → 커널이 새 namespace 생성 + 그 안에서 첫 프로세스 실행
```

`unshare bash` 한 줄과 비교하면 단계가 6개 더 많지만, **시스콜 경계를 넘어 커널이 만나는 함수는 같다**. 도구가 무엇이든(docker, containerd, podman, runc, unshare) 결국 `clone` 또는 `unshare` 시스콜을 거쳐 커널 안에서 `copy_namespaces()` → `create_new_namespaces()`로 들어간다. 이 지점이 namespace가 실제로 태어나는 순간이다.

K8s의 Pod라는 추상은 커널에는 없다. 커널이 아는 건 프로세스와 namespace뿐이다. 여러 컨테이너가 같은 IP와 IPC를 공유하도록 묶어 주는 것이 Pod sandbox라는 K8s 개념이고, 그 sandbox를 물리적으로 잡고 있는 게 Pause 컨테이너다. Pause는 새로 깎인 NET/UTS/IPC namespace 안에서 PID 1로 살아 있으면서 시그널만 기다린다. 다른 namespace 멤버(PID 1이 살아 있어야 namespace가 유지된다는 §4.3 규칙)가 모두 사라져도 Pause가 namespace를 붙잡고 있으므로 Pod의 IP가 보존된다.

> Pause 컨테이너의 동작과 CNI 연결 흐름은 [`../../09_cloud/kubernetes/04-02.Pod 네트워크와 Linux 기반.md`](../../09_cloud/kubernetes/04-02.Pod 네트워크와 Linux 기반.md) §2에서 상세히 다룬다. 본 문서는 커널 시점의 namespace 메커니즘에 집중한다.



## 다섯 질문에 대한 답

서두의 다섯 질문을 다시 보면 답이 한 줄로 떨어진다.

| 질문 | 답 |
|------|-----|
| Pod 안에서 사이드카에 localhost로 닿는 이유 | NET namespace 공유 (Pod 단위) |
| 다른 컨테이너의 프로세스가 안 보이는 이유 | PID namespace 분리 (컨테이너 단위) |
| 컨테이너 재생성 후에도 IP가 유지되는 이유 | Pause가 NET namespace를 붙잡고 있음 |
| 컨테이너 안 PID가 1인 이유 | PID namespace의 첫 프로세스가 1번 |
| 컨테이너의 hostname이 Pod 이름인 이유 | UTS namespace 격리 (Pod 단위) |
