# 파이프라인 템플릿 형식 조사 및 2026-03 권장안
---
> 이 문서는 `pipeline-api` 코드베이스에서 실제로 사용 중인 템플릿 형식을 정리하고, 2026년 3월 전후 Jenkins 공식 문서 기준으로 어떤 형식이 기본 권장안에 가까운지 조사한 결과를 정리한 문서다.

## 1. 조사 기준
이 문서는 두 가지 기준을 함께 사용했다. 첫째는 현재 코드베이스에서 어떤 템플릿 파일 확장자가 실제로 사용되는지에 대한 로컬 코드 조사다. 둘째는 2026년 4월 12일에 확인한 Jenkins 공식 문서와 Jenkins 공식 플러그인 페이지를 바탕으로, 2026년 3월 시점에도 유효하다고 볼 수 있는 권장 패턴을 정리한 것이다.

즉 "현재 이 프로젝트가 무엇을 쓰고 있는가"와 "2026년 3월 기준으로 Jenkins 생태계에서 무엇이 기본 선택지인가"를 분리해서 읽는 편이 정확하다.

## 2. 이 프로젝트에서 실제로 쓰는 템플릿 형식
Jenkins 파이프라인 생성 관점에서 보면 이 프로젝트는 사실상 `ftl`만 템플릿 파일로 사용한다. `FreemarkerService`가 `configuration.getTemplate(...)`로 읽는 대상이 모두 FreeMarker 템플릿이며, 여기서 최종 Groovy 스크립트나 Jenkins `config.xml` 문자열을 생성한다.

대표 파일은 다음과 같다:

- `src/main/resources/template/jenkins/v3/TriggerGroovyScript.ftl`
- `src/main/resources/template/jenkins/v3/JenkinsJobConfigXml.ftl`
- `src/main/resources/template/jenkins/v3/JenkinsJobConfigXmlWithTicket.ftl`
- `src/main/resources/template/jenkins/v3/GitBaseJobConfigXml.ftl`
- `src/main/resources/template/jenkins/v3/GitBaseJobConfigXmlWithTicket.ftl`
- `src/main/resources/template/jenkins/v3/SonarqubeGroovyScript.ftl`
- `src/main/resources/template/jenkins/v3/UnitTestGroovyScript.ftl`
- `src/main/resources/template/jenkins/v3/SparrowGroovyScript.ftl`

코드 연결점은 다음과 같다:

- `pipeline-api/.../FreemarkerService.java`
- `pipeline-api/.../JenkinsService.java`

의미를 나누면 아래와 같다:

- `.ftl` -> 최종 Jenkins Groovy 스크립트 생성
- `.ftl` -> 최종 Jenkins `config.xml` 생성

즉 이 프로젝트에서 `ftl`은 "직접 실행되는 파일"이 아니라 "실행될 Groovy 또는 Jenkins XML을 만들어 내는 원본 템플릿"이다.

## 3. `ftl` 말고 다른 템플릿 파일이 있는가
있다. 다만 Jenkins 파이프라인 생성용은 아니다.

현재 리소스 디렉터리 기준으로 확인되는 다른 템플릿성 파일은 다음과 같다:

- SonarQube 리포트용 `.html`
  - `src/main/resources/template/sonarqube/staticAnalysisResult.html`
  - `src/main/resources/template/sonarqube/staticAnalysisIssue.html`
  - `src/main/resources/template/sonarqube/fragment/*.html`
- GitLab 문서 산출용 `.xlsx`
  - `src/main/resources/template/gitlab/gitlab-member-registration-template.xlsx`

반면 Jenkins 파이프라인 생성 쪽에서는 `.mustache`, `.hbs`, `.jinja`, `.vm`, `.tpl`, `.yaml` 기반 템플릿 엔진을 쓰는 흔적은 찾지 못했다. Jenkins 관련 템플릿은 `v2`, `v3` 모두 사실상 `.ftl` 계열로 통일돼 있다.

## 4. 실제 사용 중인 `ftl` 파일 예시
현재 코드베이스에서 Jenkins 파이프라인 관련으로 실제 사용 중인 `ftl` 파일은 크게 두 부류다. 하나는 Groovy 스크립트를 만드는 템플릿이고, 다른 하나는 Jenkins `config.xml`을 만드는 템플릿이다.

### 4.1 Groovy 스크립트 생성용 `ftl`
예를 들어 `TriggerGroovyScript.ftl`은 트리거 Job이 하위 Job을 호출하는 Groovy를 만든다. 실제 템플릿 일부는 아래와 같다:

```ftl
def buildTrigger(bizNm) {
    def jenkinsUrl = "${jenkinsUrl}"
    def separateId = "${wrkflwExcnNo}" + "-" + "${wrkflwInptNo}" + "-" + "${trigrSn}"

    bizBuild = build job: "${envrnCd}/${bizNm}", propagate: false, wait: true, quietPeriod: 5,
        parameters: [string(name: 'SEPARATE_ID', value: "${separateId}"),
                     string(name: 'IMAGE_TAG', value: "${separateId}")]
}
```

이 템플릿은 실제 실행 단계도 동적으로 만든다.

```ftl
<#if pplnOrdVoList??>
    <#list pplnOrdVoList as PplnOrdVo>
        stage("${taskCd}/${envrnCd}/${PplnOrdVo.bizNm}") {
            steps {
                script {
                    buildTrigger("${PplnOrdVo.bizNm}")
                }
            }
        }
    </#list>
</#if>
```

즉 `pplnOrdVoList`에 어떤 하위 파이프라인이 들어오느냐에 따라 최종 Jenkins Groovy가 달라진다.

다른 Groovy 계열 `ftl`도 실제로 사용 중이다:

- `UnitTestGroovyScript.ftl`
- `SonarqubeGroovyScript.ftl`
- `SparrowGroovyScript.ftl`

예를 들어 `UnitTestGroovyScript.ftl`은 아래처럼 테스트용 Jenkins 파이프라인 Groovy를 만든다.

```ftl
stage('Clone Source Code') {
    steps {
        script {
            git branch: "${brnchNm}",
                credentialsId: "${repositoryCredential}",
                url: "${repositoryUrl}"
        }
    }
}
```

즉 이 프로젝트에서 `ftl`은 단순 문자열 파일이 아니라 "실행될 Jenkins Pipeline Groovy 원본 틀"이다.

### 4.2 Jenkins `config.xml` 생성용 `ftl`
Jenkins Job 정의 자체는 XML이다. 이 프로젝트는 XML 파일을 정적으로 두지 않고, XML도 `ftl`로 생성한다.

대표 파일은 아래와 같다:

- `JenkinsJobConfigXml.ftl`
- `JenkinsJobConfigXmlWithTicket.ftl`
- `GitBaseJobConfigXml.ftl`
- `GitBaseJobConfigXmlWithTicket.ftl`

`JenkinsJobConfigXml.ftl` 일부는 아래와 같다:

```ftl
<?xml version='1.1' encoding='UTF-8'?>
<flow-definition plugin="workflow-job@1385.vb_58b_86ea_fff1">
    <properties>
<#if jenkinsJobParamVoList?has_content>
        <hudson.model.ParametersDefinitionProperty>
            <parameterDefinitions>
<#list jenkinsJobParamVoList as jenkinsJobParamVo>
                <hudson.model.StringParameterDefinition>
                    <name>${jenkinsJobParamVo.paramKey}</name>
                    <defaultValue>${jenkinsJobParamVo.paramValue}</defaultValue>
                </hudson.model.StringParameterDefinition>
</#list>
            </parameterDefinitions>
        </hudson.model.ParametersDefinitionProperty>
</#if>
    </properties>
    <definition class="org.jenkinsci.plugins.workflow.cps.CpsFlowDefinition">
        <script><![CDATA[${jenkinsFile}]]></script>
        <sandbox>true</sandbox>
    </definition>
</flow-definition>
```

이 템플릿은 `jenkinsFile` 변수에 들어온 Groovy 문자열을 Jenkins Job의 `<script><![CDATA[...]]></script>` 안에 넣는다. 즉 "Groovy를 실행하는 Jenkins Job XML"을 만드는 역할이다.

`GitBaseJobConfigXml.ftl`은 스크립트를 Jenkins 안에 직접 넣지 않고 Git 저장소에서 읽게 만드는 XML을 만든다.

```ftl
<definition class="org.jenkinsci.plugins.workflow.cps.CpsScmFlowDefinition">
    <scm class="hudson.plugins.git.GitSCM">
        <userRemoteConfigs>
            <hudson.plugins.git.UserRemoteConfig>
                <url>${intgrtdResponseVo.cloneUrl}</url>
                <credentialsId>${credentialId}</credentialsId>
            </hudson.plugins.git.UserRemoteConfig>
        </userRemoteConfigs>
    </scm>
    <scriptPath>${intgrtdResponseVo.fileName}</scriptPath>
</definition>
```

즉 XML 형식은 분명 존재한다. 다만 이 프로젝트에서는 `.xml` 템플릿 파일로 두지 않고, `.ftl` 안에서 XML을 생성한다.

## 5. XML 형식이 실제로 있는가
있다. 다만 두 종류를 구분해야 한다.

### 5.1 Jenkins Job 정의 XML
Jenkins Job 생성과 수정 시 Jenkins REST API의 `config.xml`을 사용한다. 실제 코드에서도 아래 경로로 요청한다.

- `JenkinsFeignClient.java`
  - `POST {pipelineStruct}/config.xml`
  - `GET {pipelineStruct}/config.xml`

즉 Jenkins Job 정의는 실제로 XML 형식이다. 단지 이 프로젝트는 그 XML을 정적 `.xml` 파일로 관리하지 않고 `JenkinsJobConfigXml.ftl`, `GitBaseJobConfigXml.ftl` 같은 FreeMarker 템플릿으로 생성한다.

### 5.2 애플리케이션 리소스 XML
리소스 디렉터리에는 Jenkins와 무관한 일반 XML 파일도 많다. 예를 들면 아래와 같다:

- `logback-spring.xml`
- `mybatis-config.xml`
- `mapper/*.xml`

이 파일들은 존재하지만 Jenkins 파이프라인 템플릿은 아니다. 따라서 "파이프라인 템플릿 포맷"을 설명할 때는 Jenkins `config.xml`과 MyBatis Mapper XML을 같은 부류로 보면 안 된다.

## 6. Jenkins 공식 문서 기준의 주요 선택지
2026년 3월 전후의 Jenkins 공식 문서와 공식 플러그인 페이지를 기준으로 보면, 파이프라인 정의와 Jenkins 구성 관점의 대표 선택지는 아래처럼 나뉜다.

### 4.1 `Jenkinsfile`
Jenkins 공식 문서는 Pipeline as Code의 기본 단위로 `Jenkinsfile`을 제시한다. 공식 문서는 Pipeline as Code를 "소스 저장소에 저장하고 버전 관리되는 파이프라인 코드"로 설명하며, 프로젝트는 저장소 루트에 `Jenkinsfile`을 둬야 한다고 안내한다.

이 형식은 특히 Multibranch Pipeline과 함께 사용할 때 Jenkins가 브랜치별 Job을 자동으로 발견하고 관리할 수 있다는 점이 핵심이다.

### 4.2 Shared Libraries
Jenkins 공식 문서는 파이프라인 공통 로직 재사용 방식으로 Shared Libraries를 제공한다. 공통 스텝과 DSL을 `src/`, `vars/`, `resources/` 구조로 분리해 재사용하고, 각 저장소의 `Jenkinsfile`은 얇게 유지하는 방식이다.

즉 `Jenkinsfile`이 엔트리포인트라면 Shared Libraries는 재사용 가능한 공통 모듈이라고 보면 된다.

### 4.3 Job DSL
Job DSL 플러그인은 Jenkins Job 자체를 코드로 생성하는 도구다. 공식 플러그인 페이지도 이 도구가 XML Job 설정을 생성하는 역할이라고 설명한다. 다만 같은 페이지에서 복잡한 자동화 프로세스는 Jenkins Pipeline이 더 나은 선택인 경우가 많다고 분명히 적고 있다.

즉 Job DSL은 "파이프라인 내용"보다는 "Job 프로비저닝"에 더 가깝다.

### 4.4 JCasC
Configuration as Code 플러그인은 Jenkins 전역 설정을 YAML로 관리하는 방식이다. 공식 플러그인 페이지는 이것을 Jenkins UI 설정을 사람이 읽기 쉬운 선언형 파일로 바꾸는 방식이라고 설명한다.

이건 파이프라인 스크립트 포맷이라기보다 Jenkins 컨트롤러와 글로벌 설정을 코드로 관리하는 선택지다.

### 4.5 Jenkins Templating Engine
공식 플러그인 페이지 기준으로 JTE는 중앙 템플릿 기반 파이프라인 거버넌스를 제공한다. 다만 2026년 4월 12일 확인 시점에 이 플러그인은 `up for adoption` 상태였고, 설치 비율도 낮았다.

따라서 존재하는 선택지이긴 하지만, 기본 권장안으로 보기는 어렵다.

## 7. 2026년 3월 기준 가장 무난한 베스트 형식
이 질문의 "베스트 형식"을 "새로 설계할 때 가장 기본 선택지로 추천할 수 있는 형식"으로 해석하면, 가장 무난한 권장안은 **SCM에 저장된 `Jenkinsfile` + Multibranch Pipeline + Shared Libraries 조합**이다.

그 이유는 세 가지다:

- Jenkins 공식 문서가 Pipeline as Code의 기본 단위로 `Jenkinsfile`을 직접 제시한다.
- 공식 문서가 공통 로직 재사용 방식으로 Shared Libraries를 제공한다.
- Multibranch Pipeline이 브랜치 발견과 Job 관리를 자동화해 수작업 Job 생성을 줄여 준다.

즉 2026년 3월 기준의 기본 권장 패턴은 아래처럼 이해하면 된다:

1. 각 저장소 루트에 `Jenkinsfile`을 둔다.
2. 공통 배포/빌드 로직은 Shared Library로 뺀다.
3. Jenkins는 Multibranch Pipeline으로 저장소와 브랜치를 자동 인식하게 한다.

이 조합은 Jenkins가 공식적으로 가장 자연스럽게 설명하는 구조이고, 특정 플러그인 하나에 과하게 의존하지 않는다는 점에서도 안정적이다.

## 8. 그럼 `ftl -> config.xml/Groovy 생성` 방식은 나쁜가
나쁘다고 보기는 어렵다. 다만 성격이 다르다.

현재 프로젝트의 `ftl` 방식은 일반 사용자 저장소에 `Jenkinsfile`을 두는 모델이 아니라, 플랫폼 서버가 Jenkins Job 정의를 중앙에서 생성하고 갱신하는 모델이다. 이런 구조에서는 FreeMarker로 `config.xml`과 Groovy를 조합하는 방식이 실용적일 수 있다.

하지만 2026년 3월 기준의 "일반적인 Jenkins 모범 선택"이라고 묻는다면, 중앙 서버가 `config.xml`을 직접 생성하는 방식보다 `Jenkinsfile` 기반 Pipeline as Code가 더 기본 권장안에 가깝다. 이건 공식 문서의 중심축이 `Jenkinsfile` 쪽에 있기 때문이다.

즉 해석은 다음처럼 나누는 편이 맞다:

- 현재 프로젝트 아키텍처 안에서는 `ftl` 방식이 합리적일 수 있다.
- Jenkins 생태계 전체의 기본 권장안은 `Jenkinsfile` 중심이다.

## 9. 목적별 추천안
실무에서는 하나만 "정답"으로 두기보다 목적별로 나누는 편이 정확하다:

| 목적 | 2026-03 기준 추천 형식 | 이유 |
| :--- | :--- | :--- |
| 저장소별 CI/CD 파이프라인 정의 | `Jenkinsfile` | Jenkins 공식 Pipeline as Code 기본 형식이다. |
| 공통 스텝 재사용 | Shared Libraries (`vars/`, `src/`) | 공식 재사용 메커니즘이다. |
| Jenkins 전역 설정 관리 | JCasC YAML | 컨트롤러 설정을 선언형으로 관리한다. |
| 대량 Job 생성/초기 부트스트랩 | Job DSL | Job 정의 생성에 적합하다. |
| 중앙 플랫폼이 동적 Job XML/Groovy 생성 | `ftl` 등 서버 템플릿 | 현재 프로젝트처럼 플랫폼이 Jenkins를 직접 관리할 때 실용적이다. |

따라서 "가장 베스트 형식"을 하나만 고르면 `Jenkinsfile`이라고 답하는 편이 맞다. 다만 "현재 시스템 같은 중앙 제어형 플랫폼에서 가장 실용적인 형식"을 묻는다면 `ftl -> Groovy/XML 생성`도 충분히 성립한다.

## 10. 결론
이 프로젝트에서 Jenkins 파이프라인 템플릿은 사실상 `.ftl`로 통일돼 있다. 다른 템플릿성 파일은 `.html`, `.xlsx` 정도가 있지만 Jenkins Job 생성용 대체 템플릿 엔진은 확인되지 않았다.

2026년 3월 기준 Jenkins 공식 방향에 가장 가까운 기본 권장안은 `Jenkinsfile` 기반 Pipeline as Code다. 여기에 Shared Libraries를 붙여 공통화를 하고, Jenkins 전역 설정은 JCasC YAML로 관리하는 조합이 가장 무난하다.

반면 현재 프로젝트처럼 플랫폼 API가 Jenkins Job을 중앙에서 만들어 주는 구조에서는 `ftl`로 Groovy와 `config.xml`을 생성하는 방식이 "표준 기본형"은 아니어도 "구조적으로 타당한 구현 방식"이라고 해석할 수 있다.
