# 시작하기

---

> PostgreSQL 은 1996 년부터 살아 있는 오픈소스 RDBMS 다. SQL 표준 준수와 ACID 보장 같은 RDBMS 기본기 위에 JSON·시계열·지리공간·전문 검색·메시지 큐 같은 확장이 한 엔진 안에 모여 있다. 본 챕터는 본격 학습 전 진입점 — 어떤 자리에 PostgreSQL 이 어울리고, 어떻게 띄우며, psql 의 기본 명령으로 무엇을 확인할 수 있는지 — 를 정리한다.



## PostgreSQL 의 위치

> "Beyond SQL" 이라는 표현이 자주 쓰인다. 표준 SQL 을 완전히 지원하면서 그 위에 확장을 더해 한 엔진이 여러 워크로드를 받아 준다는 뜻이다.

| 자리 | 사용 적합 | 이유 |
|------|----------|------|
| OLTP 메인 DB | ✓ | ACID, 강한 일관성, 풍부한 데이터 타입 |
| 작은~중간 OLAP | ✓ | CTE·윈도우 함수, FDW 로 외부 결합 |
| JSON 문서 저장 | ✓ | jsonb 타입, GIN 인덱스 |
| 전문 검색 | ✓ | tsvector, GIN 인덱스 |
| 시계열 (수십~수백 GB) | ✓ | TimescaleDB 확장, BRIN 인덱스 |
| 지리공간 | ✓ | PostGIS 확장 (GIS 표준) |
| 메시지 큐 (저~중 처리량) | ✓ | SKIP LOCKED, LISTEN/NOTIFY |
| 큰 OLAP (TB+) | ✗ | Snowflake/ClickHouse 가 한 자릿수 빠름 |
| 거대 KV 캐시 | ✗ | Redis 가 한 자릿수 빠름 |

운영 결정의 첫 발은 "PostgreSQL 한 대로 시작" 이 거의 항상 답이다. 한 자리가 한계에 부딪힐 때 그 자리만 특화 시스템(Elasticsearch·ClickHouse·Redis) 으로 빼내는 unbundling 패턴([`../02-03.스트리밍 시스템 철학.md`](../02-03.스트리밍%20시스템%20철학.md) 참고) 이 운영 흐름의 표준이다.



## Docker 로 띄우기

> 학습 환경에서 가장 빠른 길이다.

```bash
docker run -d \
  --name postgres \
  -e POSTGRES_PASSWORD=mypassword \
  -e POSTGRES_DB=mydb \
  -p 5432:5432 \
  -v postgres_data:/var/lib/postgresql/data \
  postgres:16
```

세 가지 결정만 짚고 넘어간다. **`-v postgres_data:/var/lib/postgresql/data`** 가 데이터 영속성의 핵심이다. 컨테이너가 죽어도 named volume 의 데이터는 살아남는다. 이 한 줄이 빠지면 컨테이너 재생성 시 모든 데이터가 사라진다. **`POSTGRES_PASSWORD`** 는 학습 환경에서만 평문으로 둔다. 운영에서는 secret manager 위에 둔다. **`postgres:16`** 의 메이저 버전을 명시한다. `latest` 태그는 호환성 사고의 흔한 원인이라 운영에서도 명시적 버전이 표준이다.

접속은 두 갈래다.

```bash
# 컨테이너 안의 psql
docker exec -it postgres psql -U postgres -d mydb

# 호스트의 psql (설치 필요)
psql -h localhost -p 5432 -U postgres -d mydb
```



## psql 핵심 명령

> psql 은 단순 CLI 가 아니라 운영의 표준 도구다. 메타 명령(`\` 로 시작) 이 GUI 없이도 모든 정보를 노출한다.

| 명령 | 동작 |
|------|------|
| `\l` | 데이터베이스 목록 |
| `\c <db>` | 다른 DB 로 연결 |
| `\dn` | 스키마 목록 |
| `\dt` | 테이블 목록 |
| `\d <table>` | 테이블 구조 + 인덱스 |
| `\df <pattern>` | 함수 목록 |
| `\du` | 역할(role) 목록 |
| `\timing on` | 쿼리 시간 측정 켜기 |
| `\x` | 결과 가로/세로 토글 |
| `\e` | 외부 에디터로 쿼리 작성 |
| `\watch 5` | 직전 쿼리를 5 초마다 반복 |
| `\?` | 메타 명령 도움말 |
| `\h <SQL>` | SQL 명령 도움말 |
| `\q` | 종료 |

`\timing on` 과 `\x` 는 운영 디버깅에 매일 쓴다. 쿼리 실행 시간이 즉시 보이고, 컬럼이 많은 결과를 세로로 펼치면 가독성이 압도적이다.

GUI 도구(pgAdmin·DBeaver·TablePlus) 도 있지만 운영 서버에 직접 연결할 때는 거의 항상 psql 이 답이다. SSH 터널 위에서 동작하고, 화면 캡처가 텍스트라 공유가 쉬우며, 자동화 스크립트와 같은 인터페이스다.



## generate_series — 학습용 데이터 생성

> 더미 데이터가 필요한 자리에서 표준 도구다. SQL 안에서 시퀀스를 만든다.

```sql
-- 1 부터 100 까지
SELECT generate_series(1, 100);

-- 2 씩 증가
SELECT generate_series(0, 100, 2);

-- 날짜 시퀀스 (1 시간 간격)
SELECT generate_series(
  '2026-01-01 00:00:00'::timestamp,
  '2026-01-02 00:00:00'::timestamp,
  '1 hour'::interval
);

-- 100 만 행 더미 데이터
INSERT INTO logs (created_at, message)
SELECT
  now() - (random() * interval '30 days'),
  'log-' || g
FROM generate_series(1, 1000000) g;
```

학습 환경에서 쿼리 성능을 측정하려면 충분한 양의 더미 데이터가 필요하다. `generate_series` 가 이 자리에서 빠르고 간결한 답이다. 100 만 행 INSERT 가 몇 초 안에 끝나서 인덱스 효과·쿼리 계획 변화를 직접 확인할 수 있다.
