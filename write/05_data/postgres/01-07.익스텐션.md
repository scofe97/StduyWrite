# 익스텐션

---

> PostgreSQL 의 가장 큰 강점 중 하나가 익스텐션 시스템이다. 데이터 타입·함수·인덱스 타입·PL 언어를 한 명령으로 추가할 수 있어, 한 엔진이 시계열·지리공간·전문 검색·암호화 같은 자리를 모두 받아 준다. 본 챕터는 익스텐션 관리 명령, 자주 쓰는 익스텐션, FDW(외부 테이블), Postgres 호환 솔루션을 정리한다.



## 익스텐션 관리 명령

> 익스텐션은 한 명령으로 설치하고 끈다.

```sql
-- 사용 가능한 익스텐션 목록
SELECT * FROM pg_available_extensions;

-- 설치된 익스텐션 목록
\dx
SELECT * FROM pg_extension;

-- 익스텐션 설치
CREATE EXTENSION pgcrypto;

-- 특정 스키마에 설치
CREATE EXTENSION postgis SCHEMA gis;

-- 익스텐션 업그레이드
ALTER EXTENSION postgis UPDATE TO '3.4.0';

-- 제거
DROP EXTENSION pgcrypto;
```

운영의 한 가지 함정은 **권한** 이다. 대부분 익스텐션 설치는 슈퍼유저 권한이 필요하다. 클라우드 PostgreSQL(AWS RDS·GCP Cloud SQL) 은 슈퍼유저 접근을 막아 두므로 *지원하는 익스텐션 목록* 이 미리 정해져 있고 그 외에는 설치 불가다. 운영 환경 결정 시 필요한 익스텐션이 클라우드 제공 목록에 있는지 미리 확인해야 한다.



## 자주 쓰는 익스텐션 — 카테고리별

> 익스텐션이 수십 개라 모두 다룰 수 없다. 카테고리별 대표만 정리한다.

**모니터링 / 통계**
- `pg_stat_statements` — 모든 쿼리의 실행 통계 수집. 운영 필수 ([`./02-03.성능 모니터링.md`](./02-03.성능%20모니터링.md))
- `auto_explain` — 느린 쿼리의 EXPLAIN 을 자동 로그
- `pg_buffercache` — buffer cache 의 현재 상태 조회

**암호화 / 보안**
- `pgcrypto` — 해시·암호화·UUID 생성 함수 (다음 절에서 자세히)
- `pg_audit` — 상세 감사 로그
- `passwordcheck` — 약한 비밀번호 거부

**검색 / 텍스트**
- `pg_trgm` — Trigram 기반 유사도 검색, LIKE 인덱싱
- `unaccent` — 발음 부호 제거 (résumé → resume)
- `fuzzystrmatch` — Soundex·Levenshtein 거리

**데이터 타입**
- `hstore` — 키-값 쌍 (jsonb 의 옛 버전, 신규 자리에는 jsonb 권장)
- `uuid-ossp` — UUID 생성 (PostgreSQL 13+ 의 `gen_random_uuid()` 가 표준이 됨)
- `citext` — 대소문자 무시 텍스트

**도메인 특화**
- `PostGIS` — 지리공간 ([`./01-09.지리공간 데이터.md`](./01-09.지리공간%20데이터.md))
- `TimescaleDB` — 시계열 ([`./01-08.시계열 데이터.md`](./01-08.시계열%20데이터.md))
- `pgvector` — 벡터 검색 (AI 임베딩)

**FDW (외부 테이블)**
- `postgres_fdw` — 다른 PostgreSQL 인스턴스
- `mysql_fdw` — MySQL
- `file_fdw` — CSV·텍스트 파일

운영에서 첫날부터 거의 항상 켜는 익스텐션은 `pg_stat_statements` 와 `pgcrypto` 다.



## pgcrypto — 암호화 표준 도구

> 해시·대칭 암호화·UUID 생성 함수가 한 묶음이다.

```sql
CREATE EXTENSION pgcrypto;

-- 비밀번호 해시 (bcrypt)
SELECT crypt('mypassword', gen_salt('bf'));
-- 결과: $2a$06$N9qo8uLOickgx2ZMRZoMye...

-- 비밀번호 검증
SELECT password_hash = crypt('mypassword', password_hash) AS valid
FROM users WHERE id = 1;

-- 대칭 암호화
SELECT pgp_sym_encrypt('sensitive data', 'mykey');
SELECT pgp_sym_decrypt(encrypted_data, 'mykey') FROM secrets;

-- UUID 생성
SELECT gen_random_uuid();
```

비밀번호 해시는 운영에서 가장 자주 쓰는 자리다. `crypt()` 함수는 단방향 해시고, `gen_salt('bf')` 가 bcrypt salt 를 만든다. 검증 시에도 같은 salt 가 자동으로 들어가므로 단순 등호 비교 한 줄로 끝난다.

함정 한 가지 — 대칭 암호화의 키를 *DB 안에 저장하지 않는다*. DB 자체가 유출되면 데이터와 키가 함께 유출되어 의미가 없다. 키는 환경 변수·secret manager 같은 외부 자리에 둔다.

PostgreSQL 13 부터는 `gen_random_uuid()` 가 코어에 들어와 `uuid-ossp` 익스텐션이 거의 필요 없게 됐다. 신규 코드는 코어 함수가 답이다.



## pg_stat_statements — 운영 필수 모니터링

> 모든 SQL 의 실행 통계를 수집한다. 어떤 쿼리가 느린지, 어떤 쿼리가 자주 실행되는지를 한 자리에 모은다.

```sql
-- postgresql.conf 에서 사전 로드 필요
-- shared_preload_libraries = 'pg_stat_statements'
-- pg_stat_statements.track = all
-- (수정 후 PostgreSQL 재시작)

CREATE EXTENSION pg_stat_statements;

-- 가장 느린 쿼리 Top 20
SELECT
  substr(query, 1, 100) AS query,
  calls,
  round(total_exec_time::numeric, 2) AS total_ms,
  round(mean_exec_time::numeric, 2) AS mean_ms,
  round((100 * total_exec_time / sum(total_exec_time) OVER ())::numeric, 2) AS pct
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

`pg_stat_statements` 가 받쳐 주는 운영 질문이 셋이다. **가장 시간을 많이 쓰는 쿼리는 무엇인가** (`total_exec_time`), **가장 자주 실행되는 쿼리는** (`calls`), **평균 응답이 가장 느린 쿼리는** (`mean_exec_time`). 세 질문이 다른 답을 줄 수 있어 함께 본다.

자세한 운영 측면은 [`./02-03.성능 모니터링.md`](./02-03.성능%20모니터링.md) 가 다룬다.



## Foreign Data Wrapper (FDW) — 외부 데이터를 테이블처럼

> 다른 DB·파일·API 를 PostgreSQL 안에서 SELECT 가능한 테이블로 노출한다.

```sql
CREATE EXTENSION postgres_fdw;

-- 외부 PostgreSQL 서버 정의
CREATE SERVER remote_db FOREIGN DATA WRAPPER postgres_fdw
OPTIONS (host 'remote.example.com', dbname 'analytics', port '5432');

-- 사용자 매핑
CREATE USER MAPPING FOR myuser SERVER remote_db
OPTIONS (user 'analytics_user', password 'secret');

-- 외부 스키마의 테이블을 한 번에 import
IMPORT FOREIGN SCHEMA public
FROM SERVER remote_db
INTO remote_schema;

-- 일반 SELECT 처럼 사용
SELECT * FROM remote_schema.events WHERE created_at > now() - interval '1 day';

-- 로컬 테이블과 JOIN
SELECT u.name, e.event_type
FROM users u
JOIN remote_schema.events e ON u.id = e.user_id;
```

운영 자리는 분명하다. 분석용 데이터를 별도 분석 DB 에 두고, 운영 DB 에서 가끔 결합 쿼리가 필요할 때 FDW 가 답이다. 완전한 ETL 파이프라인을 구축할 정도는 아닌 자리에서.

성능 함정도 분명하다. 옵티마이저가 외부 통계를 모르거나 푸시다운(predicate push-down) 이 안 되면 모든 데이터를 가져와 로컬에서 필터링한다. 큰 외부 테이블 + 복잡한 쿼리는 FDW 가 적절치 않다. 그 자리에서는 ETL 또는 CDC 가 답이다.



## Postgres 호환 솔루션

> "PostgreSQL API 는 그대로지만 내부 엔진은 다른" 시스템들이 늘어났다. 운영 결정 시 자주 검토 대상이다.

| 솔루션 | 특징 | 자리 |
|--------|------|------|
| AWS Aurora PostgreSQL | 스토리지·컴퓨트 분리, 자동 확장 | AWS 환경 OLTP |
| GCP AlloyDB | 컬럼 스토어 통합, OLTP+OLAP | OLTP+분석 혼합 |
| Citus (Azure Cosmos for PG) | 분산 SQL, 샤딩 자동 | 거대 OLTP |
| CockroachDB | 분산·다지역, 강한 일관성 | 글로벌 분산 |
| YugabyteDB | 분산 SQL, 다지역 복제 | 글로벌 OLTP |
| TimescaleDB | 시계열 특화 | 시계열 거대 데이터 |
| Greenplum | MPP OLAP | 데이터 웨어하우스 |
| Redshift | OLAP 최적화 (PG 호환은 약함) | AWS 분석 |

호환성 수준이 달라 *원조 PostgreSQL 의 모든 기능이 그대로 동작하지는 않는다*. Aurora 가 가장 호환성이 높고, 다지역 분산 시스템(CockroachDB·Yugabyte) 은 일부 함수·익스텐션이 안 동작한다. 운영 결정 전 반드시 자기 코드 베이스가 쓰는 기능들의 호환 여부를 검증해야 한다.

표준 PostgreSQL 위 자체 운영 vs 호환 솔루션의 결정 기준은 단순하다. **확장성** 이 핵심 요구면 호환 솔루션, **유연성과 비용** 이 핵심이면 표준 PostgreSQL. 대부분의 운영 환경이 아직 표준 PostgreSQL 한 대로 충분하다.
