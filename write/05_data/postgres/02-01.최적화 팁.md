# 최적화 팁

---

> 운영에서 가장 자주 부딪히는 성능 자리들이다. EXPLAIN 해석, 흔한 인덱스 사고, 커넥션 풀링, SELECT 최적화 패턴을 한 자리에 모은다. [`./01-04.인덱스.md`](./01-04.인덱스.md) 가 인덱스 설계를 다루고 [`./02-03.성능 모니터링.md`](./02-03.성능%20모니터링.md) 가 모니터링을 다룬다면, 본 챕터는 *느린 쿼리를 만났을 때 무엇을 보고 무엇을 고칠지* 에 집중한다.



## EXPLAIN ANALYZE 의 핵심 옵션

> 운영 디버깅의 표준 도구다. 옵션 조합이 보여 주는 정보가 다르다.

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE, FORMAT TEXT)
SELECT * FROM orders
WHERE customer_id = 100 AND order_date > '2026-01-01';
```

| 옵션 | 추가 정보 | 운영 가치 |
|------|----------|----------|
| `ANALYZE` | 실제 실행 시간·행 수 | 추정 vs 실제 비교 |
| `BUFFERS` | 메모리 hit / 디스크 read | I/O 병목 판정 |
| `VERBOSE` | 출력 컬럼·스키마 | 옵티마이저 판단 추적 |
| `SETTINGS` | 비기본 설정 표시 | 설정 영향 디버깅 |
| `WAL` | WAL 생성 통계 | 쓰기 부하 분석 |

운영에서 가장 자주 쓰는 조합은 `(ANALYZE, BUFFERS)` 다. 실행 시간과 I/O 패턴을 한 번에 본다.

**핵심 해석 포인트** 가 셋이다.

**Estimated vs Actual rows** — 추정 행 수와 실제 행 수가 한 자릿수 이상 차이 나면 옵티마이저가 잘못된 가정을 한다는 신호다. `ANALYZE table_name` 으로 통계를 갱신하면 종종 해결된다. 그래도 안 풀리면 더 많은 통계 샘플(`ALTER TABLE ... ALTER COLUMN ... SET STATISTICS 1000`) 또는 다중 컬럼 통계(`CREATE STATISTICS`) 를 검토한다.

**Buffers: shared hit / read** — hit 은 메모리에서, read 는 디스크에서. read 가 큰 쿼리는 디스크 I/O 가 병목이다. 캐시 적중률(`shared hit / (hit + read)`) 이 95% 미만이면 `shared_buffers` 메모리 부족이 의심된다.

**노드 종류** — `Seq Scan` (풀 스캔) vs `Index Scan` (인덱스) vs `Bitmap Heap Scan` (다행 인덱스 활용) vs `Index Only Scan` (인덱스만으로). 각 노드의 의미를 알면 무엇을 고칠지가 분명해진다.



## 옵티마이저가 인덱스를 안 쓰는 다섯 가지 이유

> "인덱스를 만들었는데 EXPLAIN 에서 Seq Scan 이 나온다" 는 흔한 운영 질문이다. 답이 다섯 갈래다.

**통계가 오래됨** — `ANALYZE table_name` 한 줄로 해결되는 경우가 의외로 많다. 대량 INSERT/UPDATE 후에 통계가 안 갱신된 자리.

**함수가 컬럼을 감쌈** — `WHERE LOWER(email) = ...` 는 일반 인덱스가 안 통한다. 표현식 인덱스가 답이다([`./01-04.인덱스.md`](./01-04.인덱스.md) 참고).

**OR 조건** — `WHERE a = 1 OR b = 2` 는 옵티마이저가 두 인덱스를 따로 쓰고 BitmapOr 로 합치거나, 안 되면 풀 스캔으로 떨어진다. 같은 컬럼의 OR 은 `IN` 으로 다시 쓰면 인덱스를 탄다.

**결과 행 수가 너무 많음** — 테이블의 30% 이상을 가져온다면 옵티마이저가 풀 스캔이 더 빠르다고 판단한다. 이건 정상 동작이고 문제가 아니다.

**`random_page_cost` 가 환경에 안 맞음** — 옛 HDD 시대 기본값(4.0) 이 SSD 환경에서는 너무 높다. SSD 환경에서는 `random_page_cost = 1.1` 같이 낮춰 옵티마이저가 인덱스를 더 자주 쓰도록 유도한다.

```sql
-- 통계 갱신
ANALYZE orders;

-- 더 많은 통계 샘플
ALTER TABLE orders ALTER COLUMN customer_id SET STATISTICS 1000;
ANALYZE orders;

-- random_page_cost 조정 (postgresql.conf)
random_page_cost = 1.1
```



## SELECT 최적화 — 자주 쓰는 패턴

**`SELECT *` 피하기** — 필요한 컬럼만 명시한다. 컬럼이 적으면 디스크 I/O 가 줄고, Index-Only Scan 이 가능해질 수 있다.

**`COUNT(*)` 의 함정** — PostgreSQL 의 `COUNT(*)` 는 모든 행을 세어야 하므로 큰 테이블에서 느리다. 정확한 카운트가 꼭 필요하지 않으면 추정값을 쓴다.

```sql
-- 추정 카운트 (즉시)
SELECT reltuples::BIGINT AS estimate
FROM pg_class WHERE relname = 'orders';

-- 정확한 카운트가 필요하면 캐시 테이블 또는 트리거로 유지
```

**`OFFSET` 의 함정** — 큰 OFFSET 은 모든 앞 행을 스킵해야 하므로 느리다. 키 기반 페이지네이션(keyset pagination) 이 답이다.

```sql
-- 느린 패턴
SELECT * FROM posts ORDER BY id LIMIT 20 OFFSET 100000;

-- 빠른 패턴 (keyset)
SELECT * FROM posts WHERE id > $last_id ORDER BY id LIMIT 20;
```

**`EXISTS` vs `IN` vs `JOIN`** — 같은 의미를 표현하는 세 가지 길이고, 옵티마이저가 보통 같은 계획을 만든다. 의미가 분명한 쪽을 쓴다. NULL 처리가 다를 수 있어 검증이 필요하다.

**`UPDATE` 가 느릴 때** — UPDATE 는 새 행 버전을 만드는 작업이라 인덱스 갱신이 모두 따라온다. 같은 트랜잭션에서 여러 UPDATE 는 한 번의 큰 UPDATE 로 합치는 게 보통 답이다.



## 데이터베이스 연산 활용

> 애플리케이션 메모리에서 가공할 일을 DB 안에서 해결하는 게 거의 항상 빠르다.

**서버 사이드 집계** — Python·Java 코드에서 1000 행을 가져와 합계를 계산하지 않고 `SELECT SUM(...)` 으로 끝낸다. 네트워크 비용이 사라진다.

**Window function 으로 순위·이동평균** — 애플리케이션 코드에서 정렬 후 처리하지 않고 [`./01-03.모던 SQL.md`](./01-03.모던%20SQL.md) 의 윈도우 함수가 답이다.

**JSON 변환을 DB 에서** — `SELECT json_agg(row_to_json(t)) FROM ... t` 같은 쿼리로 DB 가 직접 JSON 을 만든다. 애플리케이션의 직렬화 비용이 줄어든다.

**`UPSERT` 로 race condition 회피** — `INSERT ... ON CONFLICT` 가 SELECT + INSERT/UPDATE 패턴의 race condition 을 한 SQL 로 푼다.



## 커넥션 풀링 — PgBouncer

> PostgreSQL 은 한 커넥션마다 프로세스를 띄운다. 커넥션이 늘면 메모리·CPU 비용이 빠르게 증가하므로 풀링이 필수다.

```
애플리케이션 ─→ PgBouncer (풀러) ─→ PostgreSQL
   100 client                      10 connection
```

PgBouncer 의 풀 모드 세 가지가 있다.

**Session pooling** — 클라이언트가 연결한 동안 한 백엔드 커넥션을 점유한다. 가장 안전하지만 풀링 효과가 작다.

**Transaction pooling** — 트랜잭션 단위로 커넥션을 빌려준다. 풀링 효과가 크지만 prepared statement·세션 변수 같은 자리에서 함정이 있다. 운영의 표준 모드다.

**Statement pooling** — 한 쿼리마다 커넥션을 바꾼다. 트랜잭션 사용 불가라 거의 안 쓴다.

운영 환경 설정의 핵심은 **풀 크기 결정** 이다. PostgreSQL 의 `max_connections` (보통 100~300) 와 PgBouncer 의 `default_pool_size` (보통 20~50) 를 결합한다. 애플리케이션 인스턴스 수 × 인스턴스당 커넥션 수가 PgBouncer 풀 크기 이하여야 한다.

함정 한 가지가 **Transaction pooling 의 prepared statement** 다. PostgreSQL 14 이전에는 prepared statement 가 세션에 묶여 있어 transaction pooling 모드에서 동작 안 했다. 14 부터는 글로벌 prepared statement 가 추가되어 사용 가능한데, 클라이언트 라이브러리도 같은 기능을 지원해야 한다.



## 인덱스 균형 — 너무 많지도 너무 적지도

> 인덱스가 부족하면 검색이 느리지만 너무 많아도 쓰기·디스크가 무너진다.

자주 보는 사고가 *모든 컬럼에 인덱스* 다. 실제로 안 쓰이는 인덱스가 누적되어 INSERT 처리량이 한 자릿수 떨어진다. 답은 정기 점검이다.

```sql
-- 안 쓰이는 인덱스 (idx_scan = 0)
SELECT s.schemaname, s.relname, s.indexrelname,
       pg_size_pretty(pg_relation_size(s.indexrelid)) AS size,
       s.idx_scan
FROM pg_stat_user_indexes s
WHERE s.idx_scan = 0
  AND s.indexrelname NOT LIKE '%pkey'  -- 기본 키 제외
ORDER BY pg_relation_size(s.indexrelid) DESC;

-- 중복 인덱스 (같은 컬럼들)
SELECT pg_size_pretty(SUM(pg_relation_size(idx))::BIGINT) AS size,
       (array_agg(idx))[1] AS idx1,
       (array_agg(idx))[2] AS idx2
FROM (
  SELECT indexrelid::regclass AS idx,
         (indrelid::TEXT || E'\n' || indclass::TEXT || E'\n' ||
          indkey::TEXT || E'\n' || COALESCE(indexprs::TEXT, '')) AS KEY
  FROM pg_index
) sub
GROUP BY KEY HAVING COUNT(*) > 1;
```

**불필요한 인덱스 = DROP**, **중복 인덱스 = 하나만 남기기**. 정기 점검 항목이다.



## 빠른 디버깅 체크리스트

> 느린 쿼리를 만났을 때 순서대로 확인할 것들이다.

1. `EXPLAIN ANALYZE` 로 실행 계획 확인
2. `Estimated rows` vs `Actual rows` 비교 — 큰 차이면 통계 갱신
3. `Buffers: shared read` 확인 — 큰 값이면 디스크 I/O 병목
4. `Seq Scan` 이 나오면 인덱스 부재 또는 옵티마이저 판단 검토
5. 함수가 컬럼을 감싸지 않았는지 확인
6. OR 조건이 IN 으로 바뀔 수 있는지 확인
7. `LIMIT ... OFFSET` 이 있다면 keyset pagination 으로 변환
8. `pg_stat_statements` 에서 같은 쿼리의 운영 통계 확인
