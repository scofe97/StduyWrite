# 시계열 데이터

---

> 시계열은 시간 축이 데이터 모델의 1 급 시민인 자리다. IoT 센서, 시스템 메트릭, 금융 거래, 사용자 활동 로그가 모두 같은 모양이다 — 시간 컬럼 + 측정값 + (선택) 차원 컬럼. 본 챕터는 PostgreSQL 의 파티셔닝, BRIN 인덱스, TimescaleDB 익스텐션을 통한 시계열 운영을 정리한다.



## 시계열의 본질

> 시계열 워크로드의 특징이 일반 OLTP 와 다르다.

| 항목 | OLTP | 시계열 |
|------|------|--------|
| 쓰기 패턴 | 임의 행 갱신 | append-only, 시간순 |
| 읽기 패턴 | 키 단건 조회 | 시간 범위 집계 |
| 데이터 양 | GB~TB | TB~PB (빠른 누적) |
| 보존 정책 | 영구 | 점차 오래된 것 삭제·요약 |

이 특성이 운영 결정의 모든 차원을 가른다. append-only 라 잠금 경합이 없고, 시간순 정렬이 자연스러워 BRIN 인덱스가 빛나며, 자동 파티셔닝이 거대 테이블 관리를 단순화한다.

운영의 첫 결정은 *어느 도구를 쓸 것인가* 다. 일반 PostgreSQL 의 declarative partitioning 으로 충분한 자리가 있고, TimescaleDB 같은 익스텐션이 답인 자리가 있다. 데이터 규모와 쿼리 패턴이 결정 기준이다.



## 일반 PostgreSQL — Declarative Partitioning

> 큰 테이블을 파티션 키 기반으로 작은 자식 테이블들로 분할한다. PostgreSQL 10 부터 표준 기능이다.

```sql
-- 부모 테이블 정의 (range 파티셔닝)
CREATE TABLE sensor_data (
    id BIGSERIAL,
    sensor_id INTEGER,
    measured_at TIMESTAMPTZ NOT NULL,
    value DOUBLE PRECISION,
    PRIMARY KEY (id, measured_at)
) PARTITION BY RANGE (measured_at);

-- 월별 파티션 자동 생성
CREATE TABLE sensor_data_2026_01 PARTITION OF sensor_data
  FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');

CREATE TABLE sensor_data_2026_02 PARTITION OF sensor_data
  FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');

-- 인덱스도 자동으로 각 파티션에 생성
CREATE INDEX idx_sensor_data_sensor ON sensor_data (sensor_id, measured_at);
```

장점이 셋이다. **쿼리 가지치기(partition pruning)** — `WHERE measured_at >= '2026-02-01'` 같은 쿼리가 한 파티션만 스캔한다. **고립된 유지보수** — 옛 파티션을 통째로 DROP 하면 즉시 디스크 공간이 회수된다. DELETE 와 다르게 VACUUM 부담도 없다. **인덱스 효율** — 각 파티션의 인덱스가 작아 메모리에 통째로 올라간다.

함정은 **파티션 자동 생성** 이다. 일반 PostgreSQL 은 새 시간 범위 파티션을 자동으로 안 만든다. 매월 운영자가 수동으로 만들거나(또는 cron 으로 SQL 실행), `pg_partman` 같은 익스텐션을 함께 쓴다. 이 자동화 부담이 TimescaleDB 도입 동기 중 하나다.



## TimescaleDB — 시계열 특화 익스텐션

> PostgreSQL 위에서 시계열 운영을 자동화하는 익스텐션이다. Hypertable·자동 청크·연속 집계·압축이 핵심 기능이다.

```sql
CREATE EXTENSION timescaledb;

-- 일반 테이블을 hypertable 로
CREATE TABLE sensor_data (
    sensor_id INTEGER,
    measured_at TIMESTAMPTZ NOT NULL,
    value DOUBLE PRECISION
);

SELECT create_hypertable('sensor_data', 'measured_at',
  chunk_time_interval => INTERVAL '1 day');
```

**Hypertable** 은 사용자에게 단일 테이블처럼 보이지만 내부적으로는 청크(chunk) 라는 자식 테이블들로 자동 분할된다. INSERT 시 적절한 청크가 자동 생성되어 운영자의 파티션 관리 부담이 사라진다.

**Continuous Aggregate** — 자주 쓰는 집계를 자동으로 머터리얼라이즈하고 점진 갱신한다.

```sql
CREATE MATERIALIZED VIEW hourly_avg
WITH (timescaledb.continuous) AS
SELECT
  time_bucket('1 hour', measured_at) AS bucket,
  sensor_id,
  AVG(value) AS avg_value
FROM sensor_data
GROUP BY bucket, sensor_id;

-- 갱신 정책 (5 분마다, 최근 1 시간만)
SELECT add_continuous_aggregate_policy('hourly_avg',
  start_offset => INTERVAL '2 hours',
  end_offset => INTERVAL '1 hour',
  schedule_interval => INTERVAL '5 minutes');
```

옛 머터리얼라이즈드 뷰의 한계(전체 재계산) 가 사라진다. 새 데이터만 점진 추가되어 갱신이 한 자릿수 빠르다.

**Compression** — 옛 청크를 자동 압축한다.

```sql
ALTER TABLE sensor_data SET (
  timescaledb.compress,
  timescaledb.compress_segmentby = 'sensor_id'
);

SELECT add_compression_policy('sensor_data', INTERVAL '7 days');
-- 7 일 이상 된 청크 자동 압축
```

압축 후에도 SELECT 는 그대로 동작한다. 디스크 사용량이 5~10 배 줄지만 INSERT 는 안 됨 — 압축된 청크는 read-only 가 된다. 시계열의 append-only 특성과 맞아 자연스러운 패턴이다.

**Data Retention Policy** — 옛 데이터 자동 삭제.

```sql
SELECT add_retention_policy('sensor_data', INTERVAL '90 days');
-- 90 일 지난 청크 자동 DROP
```



## time_bucket — 시간 기반 그룹화

> SQL 의 `date_trunc` 보다 유연하다. 임의 간격(15 분·5 분 같은) 으로 시간을 묶을 수 있다.

```sql
-- 5 분 단위 평균
SELECT time_bucket('5 minutes', measured_at) AS bucket,
       AVG(value)
FROM sensor_data
WHERE measured_at >= now() - interval '1 hour'
GROUP BY bucket
ORDER BY bucket;

-- 시작 시간 오프셋 (예: 매일 오전 9 시 시작 일 단위)
SELECT time_bucket('1 day', measured_at, INTERVAL '9 hours') AS bucket
FROM sensor_data;
```

`date_trunc('hour', ...)` 는 시간 단위만 지원하지만 `time_bucket` 은 임의 interval 을 받는다. 운영 시각화 대시보드에서 표준 도구다.

**`time_bucket_gapfill`** — 데이터가 없는 시간 구간을 채운다.

```sql
SELECT time_bucket_gapfill('5 minutes', measured_at) AS bucket,
       COALESCE(AVG(value), 0) AS avg,
       interpolate(AVG(value)) AS interpolated  -- 선형 보간
FROM sensor_data
WHERE measured_at BETWEEN '2026-05-01' AND '2026-05-02'
GROUP BY bucket
ORDER BY bucket;
```

5 분 동안 데이터가 없어도 그 시간 슬롯이 결과에 등장한다. 차트가 끊어 보이는 사고를 막는 자리에 어울린다.



## BRIN 인덱스 — 시계열의 자연스러운 답

> 자연 정렬된 거대 테이블의 표준 인덱스다.

시계열의 INSERT 는 거의 항상 시간순이라 디스크 페이지가 시간 순으로 쌓인다. BRIN 이 이 특성을 활용해 페이지 범위(기본 128 페이지) 의 최솟값·최댓값만 저장한다.

```sql
-- B-tree 인덱스: 1 TB 테이블에서 100 GB+ 가 보통
CREATE INDEX idx_data_btree ON sensor_data (measured_at);

-- BRIN 인덱스: 같은 테이블에서 수 MB
CREATE INDEX idx_data_brin ON sensor_data USING brin (measured_at);
```

선택 기준은 단순하다. **자연 정렬된 시계열** 이면 BRIN 이 거의 항상 답이다. 인덱스 크기가 한 자릿수 작아 캐시에 통째로 올라가고, 갱신 비용도 거의 없다. 단점은 정확도가 B-tree 보다 약하다는 점이지만 시간 범위 쿼리에는 충분히 빠르다.

**자연 정렬이 깨지는 자리** 를 조심한다. 백필(backfill) 로 옛 시간 데이터를 INSERT 하면 BRIN 의 페이지 범위 가정이 깨져 검색 성능이 무너진다. 백필 후에는 `REINDEX` 가 답이다.



## 시계열 데이터 모델 — Wide vs Narrow

> 시계열을 어떤 테이블 모양으로 둘지의 결정이다.

**Wide 모델** — 한 행에 여러 측정값.

```sql
CREATE TABLE sensor_data_wide (
    measured_at TIMESTAMPTZ,
    sensor_id INTEGER,
    temperature DOUBLE PRECISION,
    humidity DOUBLE PRECISION,
    pressure DOUBLE PRECISION
);
```

**Narrow 모델** — 한 측정값마다 한 행.

```sql
CREATE TABLE sensor_data_narrow (
    measured_at TIMESTAMPTZ,
    sensor_id INTEGER,
    metric TEXT,         -- 'temperature' | 'humidity' | 'pressure'
    value DOUBLE PRECISION
);
```

선택 기준은 명확하다. **측정값 종류가 고정** 이고 자주 바뀌지 않으면 Wide(컬럼 한 자리, 압축 효율). **측정값 종류가 가변** 이거나 새 측정값이 자주 추가되면 Narrow(스키마 변경 없이 metric 컬럼 값만 추가).

InfluxDB·Prometheus 같은 시계열 DB 는 Narrow 모델을 채택한다. PostgreSQL 운영에서는 도메인에 따라 갈라진다.
