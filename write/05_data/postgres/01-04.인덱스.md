# 인덱스

---

> [`../01-05.인덱스 이론.md`](../01-05.인덱스%20이론.md) 가 인덱스 형태의 일반 이론을 다뤘다면, 본 챕터는 PostgreSQL 의 *구체 운영* 측면을 정리한다. 인덱스 타입 선택, EXPLAIN 으로 검증, 운영에서 부딪히는 함정과 답. 일반 패턴은 fundamentals 챕터를 참고하고 본 챕터는 PostgreSQL 만의 구체에 집중한다.



## PostgreSQL 의 인덱스 타입

> 한 엔진이 여섯 종류의 인덱스를 제공한다. 자리에 맞는 것을 고르는 것이 운영의 첫 결정이다.

| 타입 | 자료구조 | 자리 |
|------|---------|------|
| B-tree | B+Tree | 등가·범위·정렬 (기본) |
| Hash | 해시 테이블 | 등가만 (B-tree 가 거의 항상 더 나음) |
| GIN | 역색인 | 배열·jsonb·전문 검색 |
| GiST | 일반화 트리 | 공간·범위 타입 |
| SP-GiST | 공간 분할 트리 | 비균형 데이터 (GiST 의 변형) |
| BRIN | 블록 범위 | 거대 시계열 (자연 정렬) |

기본은 B-tree 다. 운영의 90% 가 B-tree 로 끝난다. 다른 타입은 그 자리를 받치기 어려운 워크로드(JSON 검색·전문 검색·공간·시계열) 에서 선택한다.

```sql
-- 명시적 타입 지정
CREATE INDEX idx_users_email ON users (email);  -- 기본: B-tree
CREATE INDEX idx_products_tags ON products USING gin (tags);
CREATE INDEX idx_locations_geom ON locations USING gist (geom);
CREATE INDEX idx_logs_created ON logs USING brin (created_at);
```



## B-Tree 의 운영 측면

> PostgreSQL 의 기본 인덱스다. PRIMARY KEY 와 UNIQUE 제약이 자동으로 B-tree 인덱스를 만든다.

복합 인덱스의 컬럼 순서가 가장 자주 운영 함정이 된다.

```sql
CREATE INDEX idx_orders ON orders (customer_id, order_date);

-- 인덱스 사용 ✓
SELECT * FROM orders WHERE customer_id = 100;
SELECT * FROM orders WHERE customer_id = 100 AND order_date > '2026-01-01';

-- 인덱스 사용 불가 ✗ (선두 컬럼 customer_id 가 없음)
SELECT * FROM orders WHERE order_date > '2026-01-01';
```

선두 컬럼 규칙은 [`../01-05.인덱스 이론.md`](../01-05.인덱스%20이론.md) 가 자세히 다룬다. 운영 함정은 EXPLAIN 으로 직접 확인하는 것이 답이다.

**INCLUDE 컬럼** 은 PostgreSQL 11 이후 가능한 커버링 인덱스 기능이다.

```sql
CREATE INDEX idx_orders_covering ON orders (customer_id, order_date)
INCLUDE (total_amount, status);

-- Index-Only Scan 이 가능해질 수 있음
SELECT total_amount, status FROM orders
WHERE customer_id = 100 AND order_date > '2026-01-01';
```

함정은 **visibility map** 이다. PostgreSQL 의 MVCC 때문에 인덱스 항목이 실제로 보이는 행을 가리키는지 확인하려고 종종 힙 페이지를 한 번 더 읽어야 한다. `VACUUM` 이 visibility map 을 갱신하므로 자주 갱신되는 테이블에서는 Index-Only Scan 이 의도와 달리 일반 Index Scan 으로 떨어진다.



## 부분 인덱스 — Partial Index

> PostgreSQL 의 가장 강력한 기능 중 하나다. 조건에 맞는 행만 인덱싱해 인덱스 크기와 갱신 비용을 줄인다.

```sql
-- 활성 사용자만 인덱싱
CREATE INDEX idx_active_users_email ON users (email)
WHERE is_active = true;

-- 미완료 주문만 인덱싱
CREATE INDEX idx_pending_orders ON orders (created_at)
WHERE status IN ('PENDING', 'PROCESSING');

-- NULL 이 아닌 값만 인덱싱
CREATE INDEX idx_orders_promo ON orders (promo_code)
WHERE promo_code IS NOT NULL;
```

비활성 사용자가 99% 이고 활성이 1% 인 1 억 행 테이블이라면, 일반 인덱스는 1 억 항목이지만 부분 인덱스는 100 만 항목이다. 인덱스가 100 배 작아져 메모리에 통째로 올라간다.

함정은 **쿼리 조건이 인덱스 정의 조건을 포함해야** 인덱스를 쓴다는 점이다. 옵티마이저가 그 포함 관계를 증명할 수 있어야 한다. `is_active = true` 와 `is_active IS TRUE` 가 다른 표현으로 인식되어 매칭이 빠지는 미묘한 사고가 가능하다.



## 표현식 인덱스 — Expression Index

> 컬럼 자체가 아닌 *함수의 결과* 를 인덱싱한다. 함수 호출이 들어간 쿼리의 인덱스 사용 가능 여부가 이 형태로 갈린다.

```sql
-- 인덱스 사용 불가 (일반 인덱스는 LOWER(email) 을 못 알아챔)
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';

-- 표현식 인덱스로 해결
CREATE INDEX idx_email_lower ON users (LOWER(email));
```

운영 자리는 대소문자 무시 검색, JSON 경로 추출, 날짜 트런케이션, 정규화 비교 등이다. 단점은 인덱스 정의에 함수가 박혀 있어 함수 변경이 인덱스 재생성으로 이어지고, 옵티마이저가 표현식 매칭을 잘 못 알아채는 경우가 있다는 점이다.

부분 인덱스와 결합도 자주 쓴다.

```sql
-- 활성 사용자의 이메일을 대소문자 무시로 인덱싱
CREATE INDEX idx_active_users_email_lower ON users (LOWER(email))
WHERE is_active = true;
```



## GIN·GiST·BRIN — 특수 인덱스

> B-tree 가 못 푸는 워크로드의 답이다. 자세한 일반 설명은 [`../01-05.인덱스 이론.md`](../01-05.인덱스%20이론.md) 에 있고, 본 절은 PostgreSQL 운영 측면만 다룬다.

**GIN** — Generalized Inverted Index. "한 행이 여러 키" 를 갖는 자료(배열·jsonb·tsvector) 에 어울린다.

```sql
CREATE INDEX idx_tags ON products USING gin (tags);
CREATE INDEX idx_meta ON events USING gin (metadata);
CREATE INDEX idx_fts ON articles USING gin (to_tsvector('english', content));
```

GIN 의 함정은 **갱신 비용** 이다. 한 행 갱신 시 그 행이 가진 모든 키마다 인덱스를 갱신해야 한다. PostgreSQL 은 이를 모았다가 일괄 처리하는 `fastupdate` 옵션을 기본 켠다. 처리량이 좋아지지만 검색 시 *대기 큐* 도 함께 봐야 해 검색 지연이 변동적이다. 운영에서 검색 지연이 핵심이면 `fastupdate = off` 를 검토한다.

**GiST** — Generalized Search Tree. 거리·범위 같은 비-동등 비교에 어울린다. PostGIS 의 공간 인덱스, `tsrange` 같은 범위 타입 인덱스가 표준 사용처다.

```sql
CREATE INDEX idx_geom ON locations USING gist (geom);
CREATE INDEX idx_period ON reservations USING gist (period);
```

**BRIN** — Block Range Index. 거대한 테이블이 *물리적으로 정렬되어 있을 때* 빛난다. 각 페이지 범위(기본 128 페이지) 의 최솟값·최댓값만 저장한다.

```sql
CREATE INDEX idx_logs_brin ON logs USING brin (created_at);
```

BRIN 의 가치는 **인덱스 크기** 다. 1 TB 테이블의 BRIN 이 수 MB 정도다. B-tree 의 1/100 ~ 1/1000 수준이라 캐시 효율이 압도적이다. 단점은 *정렬되지 않은 테이블에서는 거의 쓸모가 없다* 는 점이다. 시계열 로그처럼 INSERT 순서가 곧 정렬 순서인 자리에 한정된다.



## EXPLAIN ANALYZE — 검증의 표준 도구

> 인덱스를 만들었다고 옵티마이저가 반드시 쓰는 게 아니다. EXPLAIN 으로 직접 확인한다.

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)
SELECT * FROM orders WHERE customer_id = 100;
```

출력에서 핵심 정보가 셋이다.

**Plan 노드 종류** — `Index Scan`·`Index Only Scan` 이면 인덱스 사용, `Seq Scan` 이면 풀 스캔. `Bitmap Index Scan` + `Bitmap Heap Scan` 의 조합도 흔한데, 여러 행을 한 번에 가져올 때의 패턴이다.

**Buffers** — `shared hit` 는 메모리 캐시 적중, `shared read` 는 디스크에서 읽은 페이지 수. read 가 큰 쿼리는 디스크 I/O 가 병목이다.

**Actual time** — 실제 실행 시간. `cost` 는 옵티마이저의 추정이고 `actual` 이 실측이다. 두 값이 크게 다르면 통계가 오래됐거나 옵티마이저가 잘못된 가정을 한다는 신호다. `ANALYZE table_name` 으로 통계를 갱신하면 종종 해결된다.

옵티마이저가 인덱스를 안 쓰는 흔한 이유 다섯 가지가 있다. 통계가 오래됨(`ANALYZE` 로 해결), 함수가 컬럼을 감싼 쿼리(표현식 인덱스 필요), OR 조건(BitmapOr 또는 풀 스캔), 결과 행 수가 많음(전체 스캔이 더 빠름), `random_page_cost` 같은 설정 값이 운영 환경에 안 맞음. 하나하나 체크리스트로 다룬다.



## 인덱스 설계 — 운영 원칙 모음

> 인덱스를 추가할 때 따라오는 비용을 의식한다.

**쓰기 비용** — INSERT/UPDATE/DELETE 마다 모든 관련 인덱스를 갱신한다. 한 테이블에 인덱스 5~6 개를 두면 쓰기 처리량이 눈에 띄게 떨어진다. PostgreSQL 의 **HOT update** 가 이 비용을 일부 완화한다 — 갱신된 컬럼이 *어떤 인덱스에도 포함되지 않으면* 새 행 버전을 같은 페이지에 쓰고 인덱스 갱신을 건너뛴다. 반대로 자주 갱신되는 컬럼을 인덱스에 무심코 끼워 넣으면 HOT 이 깨져 같은 UPDATE 가 한 자릿수 느려진다. 인덱스 컬럼 선정 시 갱신 빈도를 함께 본다.

**디스크 비용** — B-tree 인덱스는 보통 테이블 크기의 30~50% 다. GIN 은 키 분포에 따라 테이블 크기를 넘기기도 한다.

**유지보수 비용** — PostgreSQL 의 B-tree 는 시간이 지나면서 부풀어 오른다(bloat). 주기적 `REINDEX CONCURRENTLY` 또는 `pg_repack` 으로 정리해 준다.

**미사용 인덱스 제거** — `pg_stat_user_indexes` 의 `idx_scan` 값이 0 인 인덱스는 안 쓰이고 있다는 신호다. 정기적으로 검토해 제거한다.

```sql
-- 안 쓰이는 인덱스 찾기
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;
```

**`CREATE INDEX CONCURRENTLY`** — 운영 테이블에 인덱스를 추가할 때 거의 항상 이 옵션이 답이다. 일반 `CREATE INDEX` 는 테이블 전체에 쓰기 락을 잡는다. CONCURRENTLY 는 락을 잡지 않지만 시간이 두 배 정도 걸린다. 운영 시간에 인덱스 추가는 무조건 CONCURRENTLY 로 한다.

2. 복합 인덱스의 선두 컬럼 규칙과 INCLUDE 컬럼의 차이는? Index-Only Scan 의 PostgreSQL 한정 함정(visibility map) 은?
3. 부분 인덱스가 1 억 행 테이블에서 어떤 비용을 줄이는가? 옵티마이저가 매칭 실패하는 함정 패턴은?
4. 표현식 인덱스가 필요한 자리(LOWER·jsonb 추출 등) 와 그 운영 함정은?
5. GIN 의 `fastupdate` 옵션이 처리량과 검색 지연을 어떻게 trade-off 하는가?
6. BRIN 이 시계열 로그에서 한 자릿수 작은 인덱스를 만드는 원리는? 정렬되지 않은 테이블에서 왜 쓸모가 없는가?
7. 옵티마이저가 인덱스를 안 쓰는 다섯 가지 이유와 각각의 답은?
