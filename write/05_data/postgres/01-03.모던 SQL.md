# 모던 SQL

---

> SQL:1999 이후 표준에 들어온 기능들이 PostgreSQL 에 거의 모두 구현되어 있다. CTE 로 복잡한 쿼리를 단계로 쪼개고, 윈도우 함수로 행 사이의 관계를 표현하며, LATERAL 로 행별 서브쿼리를 짜고, UPSERT 로 INSERT 와 UPDATE 를 한 문장에 합친다. 본 챕터는 운영에서 자주 쓰는 모던 SQL 도구를 정리한다.



## CTE — 복잡한 쿼리의 단계 분리

> Common Table Expression. `WITH` 절로 임시 결과 집합을 만들어 본 쿼리에서 재사용한다.

서브쿼리 중첩이 깊어질 때 CTE 가 답이다. 같은 쿼리를 단계로 쪼개 가독성을 올린다.

```sql
WITH high_value_customers AS (
  SELECT customer_id, SUM(total_amount) AS total
  FROM orders
  WHERE order_date >= '2026-01-01'
  GROUP BY customer_id
  HAVING SUM(total_amount) > 1000000
),
recent_orders AS (
  SELECT * FROM orders
  WHERE order_date >= now() - interval '7 days'
)
SELECT c.name, h.total, r.order_date
FROM high_value_customers h
JOIN customers c ON c.id = h.customer_id
LEFT JOIN recent_orders r ON r.customer_id = h.customer_id;
```

PostgreSQL 12 부터 CTE 의 동작이 한 가지 바뀌었다. 옛 버전은 CTE 가 항상 *materialize* 되어(임시 테이블처럼 디스크에 적재) 옵티마이저가 본 쿼리와 합쳐 최적화하지 못했다. 12 부터는 *기본적으로 inline* 되어 다른 서브쿼리와 비슷하게 옵티마이저가 자유롭게 다룬다. 명시적으로 강제하려면 `WITH cte AS MATERIALIZED (...)` 또는 `AS NOT MATERIALIZED (...)` 를 쓴다.



## 재귀 CTE — 트리·그래프 쿼리

> CTE 가 자기 자신을 참조하면 재귀가 된다. 부모-자식 트리 탐색의 표준 도구다.

```sql
-- 카테고리 트리 탐색
WITH RECURSIVE category_tree AS (
  -- 기준점 (root)
  SELECT id, name, parent_id, 0 AS depth, name::text AS path
  FROM categories
  WHERE parent_id IS NULL

  UNION ALL

  -- 재귀 단계
  SELECT c.id, c.name, c.parent_id, ct.depth + 1, ct.path || ' > ' || c.name
  FROM categories c
  JOIN category_tree ct ON c.parent_id = ct.id
)
SELECT * FROM category_tree ORDER BY path;
```

발상은 단순하다. 첫 SELECT 가 base case 를 만들고, UNION ALL 이후의 SELECT 가 base case 결과를 참조해 다음 단계를 만든다. 더 이상 새 행이 안 나올 때까지 반복한다.

운영에서 자주 보는 자리는 카테고리 트리, 조직도, 의존 그래프, 그래프 최단 경로다. Cypher 같은 그래프 쿼리 언어가 더 간결하지만([`../01-01.데이터 모델과 쿼리 언어.md`](../01-01.데이터%20모델과%20쿼리%20언어.md) 참고) PostgreSQL 안에서도 충분히 풀 수 있다.

함정 한 가지가 무한 루프다. 데이터에 사이클이 있으면 재귀가 안 끝난다. 운영에서는 `depth` 컬럼을 두고 `WHERE depth < 10` 같은 가드를 둔다.



## 윈도우 함수 — 행 사이의 관계

> GROUP BY 가 그룹 *결과* 만 돌려주는 데 비해, 윈도우 함수는 각 행을 그대로 두면서 그 행과 다른 행의 관계를 한 컬럼으로 더한다.

```sql
SELECT
  product_id,
  sale_date,
  amount,
  -- 같은 상품의 누적 합
  SUM(amount) OVER (PARTITION BY product_id ORDER BY sale_date) AS running_total,
  -- 같은 상품 안에서의 순위
  ROW_NUMBER() OVER (PARTITION BY product_id ORDER BY amount DESC) AS rank
FROM sales;
```

`OVER (...)` 절이 핵심이다. `PARTITION BY` 로 윈도우를 어떻게 나눌지, `ORDER BY` 로 안에서의 순서를 정한다. 같은 행 집합 위에서 여러 윈도우 함수가 다른 OVER 절로 동작할 수 있다.

순위 함수가 자주 헷갈리는 자리다.

| 함수 | 동점 처리 |
|------|----------|
| `ROW_NUMBER()` | 동점이 있어도 1, 2, 3, 4, ... 무조건 다른 번호 |
| `RANK()` | 동점은 같은 순위, 다음은 건너뜀 (1, 1, 3, 4) |
| `DENSE_RANK()` | 동점은 같은 순위, 다음은 안 건너뜀 (1, 1, 2, 3) |

운영 자리에서 가장 흔한 윈도우 함수는 `LAG`/`LEAD` (이전/다음 행 값), `FIRST_VALUE`/`LAST_VALUE` (윈도우의 첫/마지막), `NTILE(N)` (N 분위수 그룹) 이다.

```sql
-- 일별 매출 + 전일 대비 증감
SELECT
  sale_date,
  daily_total,
  LAG(daily_total) OVER (ORDER BY sale_date) AS prev_total,
  daily_total - LAG(daily_total) OVER (ORDER BY sale_date) AS change
FROM daily_sales;
```



## 윈도우 프레임 — ROWS BETWEEN

> 윈도우 함수의 동작 범위를 행 단위로 정밀하게 정한다. 이동 평균·이동 합계의 표준 도구다.

```sql
SELECT
  sale_date,
  amount,
  -- 7 일 이동 평균 (이전 6 일 + 현재)
  AVG(amount) OVER (
    ORDER BY sale_date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS moving_avg_7d
FROM daily_sales;
```

프레임 옵션이 다양하다. `ROWS` 는 물리적 행 단위, `RANGE` 는 ORDER BY 컬럼의 *값 범위* 단위다. `UNBOUNDED PRECEDING` 은 윈도우 시작, `UNBOUNDED FOLLOWING` 은 윈도우 끝, `CURRENT ROW` 는 현재 행이다.

`ORDER BY` 가 있는데 프레임이 명시 안 된 윈도우는 PostgreSQL 의 기본값이 `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` 다. 같은 ORDER BY 값을 가진 행들이 모두 한 묶음으로 들어가, 누적 합 같은 자리에서 의도치 않은 결과가 나오기 쉽다. ROW 기반 누적이 필요하면 `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` 를 명시한다.



## LATERAL — 행별 서브쿼리

> 일반 JOIN 의 서브쿼리는 외부 행을 참조 못 한다. LATERAL JOIN 은 그 제약을 푼다.

```sql
-- 각 사용자의 최신 주문 3 건을 같이 가져오기
SELECT u.name, o.*
FROM users u
LEFT JOIN LATERAL (
  SELECT * FROM orders
  WHERE customer_id = u.id  -- 외부 행 u 를 참조
  ORDER BY order_date DESC
  LIMIT 3
) o ON true;
```

`LATERAL` 키워드가 핵심이다. 서브쿼리가 같은 FROM 절의 앞 테이블 행을 참조해 행마다 다른 결과 집합을 돌려준다. 일반 JOIN 으로는 짤 수 없는 "각 N 마다 Top K" 패턴의 표준 답이다.

운영 자리는 분명하다. 카테고리별 베스트셀러 5 개, 사용자별 최근 활동 10 건, 매장별 인기 상품 3 개. 서브쿼리가 행마다 다른 결과를 돌려야 하는 자리에서 LATERAL 이 답이다.



## UPSERT — INSERT ... ON CONFLICT

> "있으면 UPDATE, 없으면 INSERT" 가 단일 SQL 로 끝난다. 흔히 부딪히는 race condition 을 자연스럽게 해결한다.

```sql
INSERT INTO products (id, name, price, stock)
VALUES (1, 'Laptop', 999, 10)
ON CONFLICT (id) DO UPDATE
SET name = EXCLUDED.name,
    price = EXCLUDED.price,
    stock = products.stock + EXCLUDED.stock;
```

`ON CONFLICT` 절이 충돌 시 동작을 정한다. `DO NOTHING` 은 충돌 시 무시, `DO UPDATE` 는 갱신이다. `EXCLUDED` 는 INSERT 하려던 새 행을 가리키는 가상 테이블이다. `products.stock + EXCLUDED.stock` 처럼 옛 값과 새 값을 결합한 갱신도 가능하다.

UPSERT 가 race condition 을 푸는 자리는 분명하다. 일반 코드 베이스의 "SELECT 후 INSERT 또는 UPDATE" 패턴은 두 단계 사이에 다른 트랜잭션이 INSERT 할 수 있어 lost update 위험이 있다. UPSERT 는 한 SQL 안에서 충돌 검사와 갱신이 원자적으로 일어나 그 위험이 없다.



## 흔히 쓰는 모던 SQL 도구 모음

> 위 절에 못 들어간 자주 쓰는 기능들이다.

**`RETURNING` 절** — INSERT/UPDATE/DELETE 의 결과를 즉시 SELECT 한다.

```sql
INSERT INTO orders (...) VALUES (...) RETURNING id, created_at;
UPDATE products SET stock = stock - 1 WHERE id = 5 RETURNING stock;
DELETE FROM logs WHERE created_at < now() - interval '1 year' RETURNING id;
```

**`FILTER` 절** — 집계 함수에 조건을 건다.

```sql
SELECT
  COUNT(*) AS total,
  COUNT(*) FILTER (WHERE status = 'completed') AS completed,
  COUNT(*) FILTER (WHERE status = 'cancelled') AS cancelled
FROM orders;
```

**Array 와 unnest** — PostgreSQL 의 배열 타입과 그것을 행으로 펼치는 함수.

```sql
SELECT id, unnest(tags) AS tag FROM products;
```

**`DISTINCT ON`** — 한 컬럼 기준 첫 행만 가져온다. PostgreSQL 특화 기능이다.

```sql
SELECT DISTINCT ON (customer_id) *
FROM orders
ORDER BY customer_id, order_date DESC;
-- 각 고객의 최신 주문 한 건씩
```
