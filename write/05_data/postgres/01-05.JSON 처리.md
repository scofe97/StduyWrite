# JSON 처리

---

> PostgreSQL 9.4 부터 jsonb 타입이 들어왔다. 관계형 DB 에 문서 모델을 자연스럽게 결합한 자리로, 한 엔진이 정형·반정형 데이터를 모두 다룰 수 있게 됐다. 본 챕터는 json 과 jsonb 의 차이, 운영에서 쓰는 연산자와 함수, GIN 인덱스 옵션, JSON Path 를 정리한다.



## json vs jsonb — 거의 항상 jsonb

> 두 타입이 비슷해 보이지만 운영 결과가 다르다.

| 항목 | json | jsonb |
|------|------|-------|
| 저장 형식 | 텍스트 그대로 | 분해된 바이너리 |
| 검증 | 입력 시 유효성만 | 유효성 + 정규화 |
| 키 순서 | 보존 | 보존 안 됨 |
| 중복 키 | 보존 | 마지막만 보존 |
| 공백 | 보존 | 제거 |
| 접근 시 | 매번 파싱 | 사전 파싱 |
| 인덱싱 | 거의 불가 | GIN 지원 |

운영의 답은 거의 항상 jsonb 다. 빠른 검색, 인덱스 지원, 일관된 형식이 모두 jsonb 쪽이다. json 이 나을 자리는 *입력 그대로 보존* 이 절대 요구일 때(예: 로그 원본 저장, 외부 API 응답 그대로 저장) 뿐이다.

```sql
-- jsonb 컬럼 정의
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    event_type TEXT,
    data JSONB,
    created_at TIMESTAMPTZ DEFAULT now()
);

INSERT INTO events (event_type, data) VALUES
  ('login', '{"user_id": 1, "ip": "1.2.3.4", "device": {"type": "mobile"}}'),
  ('purchase', '{"user_id": 2, "amount": 100, "items": [{"id": 5}, {"id": 7}]}');
```



## 핵심 연산자

> jsonb 연산자가 풍부하다. 운영에서 매일 쓰는 것들이다.

**`->`** vs **`->>`** — 가장 자주 헷갈리는 자리다.

```sql
-- -> 는 jsonb 를 돌려준다 (하위 객체 추출)
SELECT data -> 'device' FROM events;
-- 결과: {"type": "mobile"}

-- ->> 는 텍스트를 돌려준다 (값 자체 추출)
SELECT data ->> 'event_type' FROM events;
-- 결과: 'login'

-- 체이닝
SELECT data -> 'device' ->> 'type' FROM events;
-- 결과: 'mobile' (텍스트)
```

규칙은 단순하다. 다음 단계에 또 객체 접근이 따라오면 `->`, 마지막 단계에서 *값* 을 꺼낼 때는 `->>`. 타입 캐스팅이 필요한 자리에서는 `->>` 후 `::INT` 같이 명시한다.

**`#>`** vs **`#>>`** — 깊은 경로 추출이다.

```sql
SELECT data #>> '{device, type}' FROM events;
-- 결과: 'mobile'

SELECT data #> '{items, 0}' FROM events WHERE event_type = 'purchase';
-- 결과: {"id": 5}
```

배열 인덱스를 경로에 직접 쓸 수 있다(`{items, 0}` 은 items 배열의 첫 원소).

**`@>`** (포함 연산자) — jsonb 검색의 가장 강력한 도구다.

```sql
-- data 가 {"user_id": 1} 을 포함하는 행
SELECT * FROM events WHERE data @> '{"user_id": 1}';

-- 중첩 객체도 포함 검사
SELECT * FROM events WHERE data @> '{"device": {"type": "mobile"}}';

-- 배열 포함
SELECT * FROM events
WHERE data @> '{"items": [{"id": 5}]}';
```

`@>` 의 의미는 "왼쪽 jsonb 가 오른쪽 jsonb 의 모든 키-값을 포함하는가" 다. GIN 인덱스가 이 연산자를 직접 받쳐 주므로 운영에서 가장 자주 쓰는 검색 패턴이다.

**`?`·`?|`·`?&`** — 키 존재 검사.

```sql
SELECT * FROM events WHERE data ? 'user_id';        -- 'user_id' 키가 있는가
SELECT * FROM events WHERE data ?| ARRAY['a', 'b']; -- 둘 중 하나라도
SELECT * FROM events WHERE data ?& ARRAY['a', 'b']; -- 둘 다
```



## 수정 함수 — `||`·`-`·`jsonb_set`

> jsonb 값을 부분적으로 갱신한다.

```sql
-- || 로 병합 (같은 키는 오른쪽이 이김)
UPDATE events SET data = data || '{"verified": true}' WHERE id = 1;

-- - 로 키 제거
UPDATE events SET data = data - 'old_field' WHERE id = 1;

-- jsonb_set 으로 깊은 경로 수정
UPDATE events
SET data = jsonb_set(data, '{device, version}', '"2.0"', true)
WHERE id = 1;

-- 배열 원소 추가 (jsonb_insert 또는 - 와 ||)
UPDATE events
SET data = jsonb_set(
  data,
  '{items}',
  data->'items' || '[{"id": 99}]'::jsonb
)
WHERE id = 2;
```

함정 한 가지 — jsonb 갱신은 *전체 값을 다시 쓰는* 형태로 동작한다. 큰 jsonb 의 일부만 바뀌어도 디스크에 전체가 다시 쓰인다. 자주 갱신되는 부분은 jsonb 안이 아니라 별도 컬럼으로 빼는 게 운영 답이 된다.



## GIN 인덱싱 — jsonb_ops vs jsonb_path_ops

> jsonb 검색의 핵심 도구다. 두 가지 옵션이 운영 결과를 가른다.

**`jsonb_ops`** (기본) — 모든 키와 값을 인덱싱한다. 키 존재 검사(`?`)·포함(`@>`)·경로 쿼리(`@?`·`@@`) 모두 지원.

**`jsonb_path_ops`** — 경로(키 + 값 결합) 만 인덱싱한다. 인덱스 크기가 한 자릿수 작고 검색이 빠르지만 `?` 연산자 같은 키 존재 검사는 지원하지 않는다.

```sql
-- 일반적 자리
CREATE INDEX idx_events_data ON events USING gin (data);

-- 검색 패턴이 거의 @> 만이면 path_ops 가 최적
CREATE INDEX idx_events_data ON events USING gin (data jsonb_path_ops);
```

선택 기준은 단순하다. 검색 패턴이 다양하면 jsonb_ops, 거의 `@>` 검색만이면 jsonb_path_ops. 운영에서는 후자가 압도적으로 흔한 케이스다.

특정 키만 인덱싱도 가능하다. 표현식 인덱스 형태다.

```sql
-- user_id 만 자주 검색되면
CREATE INDEX idx_events_user ON events ((data->>'user_id'));

-- 함수 + 부분 인덱스 결합
CREATE INDEX idx_events_purchase_user ON events ((data->>'user_id'))
WHERE event_type = 'purchase';
```



## JSON Path — SQL/JSON 표준

> PostgreSQL 12 부터 SQL/JSON 표준의 JSON Path 를 지원한다. 복잡한 검색을 간결하게 표현한다.

```sql
-- 모바일 디바이스로 로그인한 행
SELECT * FROM events
WHERE data @? '$.device.type ? (@ == "mobile")';

-- 100 이상의 amount 가 있는 행
SELECT * FROM events
WHERE data @? '$.amount ? (@ >= 100)';

-- jsonb_path_query 로 특정 경로 추출
SELECT jsonb_path_query(data, '$.items[*].id') FROM events
WHERE event_type = 'purchase';
```

`@?` 는 path 가 매칭되는지, `@@` 는 boolean 결과가 true 인지 확인한다. 일반 `->`·`->>` 보다 표현력이 풍부하지만 운영 코드 베이스의 대부분이 단순 패턴이라 학습 우선순위는 그리 높지 않다.



## jsonb 를 언제 쓸지의 결정

> 같은 데이터를 정규화 테이블로 둘지, jsonb 한 컬럼으로 둘지의 결정이다.

**jsonb 가 어울리는 자리**:
- 스키마가 자주 바뀌거나 다양함 (사용자별 커스텀 필드, 이벤트 페이로드)
- 한 객체의 모든 정보를 함께 조회 (검색 인덱스 같은 자리)
- 반정형 외부 데이터 그대로 저장 (API 응답·웹훅 페이로드)
- 깊은 중첩이 자연스러움 (UI 설정, 폼 정의)

**정규화 테이블이 어울리는 자리**:
- 스키마가 안정적
- 여러 객체에 걸친 JOIN 이 흔함
- 강한 타입 검증과 제약조건이 필요
- 고성능 분석 쿼리

운영 환경의 답은 보통 *혼합* 이다. 핵심 비즈니스 데이터는 정규화 테이블로 두고, 사용자 정의 필드·이벤트 페이로드·UI 설정 같은 자리만 jsonb 컬럼으로 둔다.

함정 한 가지 — jsonb 안에 *외래 키* 를 넣으면 안 된다. 정규화의 referential integrity 가 jsonb 안에서는 강제되지 않는다. 다른 테이블의 행을 참조해야 하면 jsonb 가 아니라 별도 컬럼 + FK 가 답이다.
