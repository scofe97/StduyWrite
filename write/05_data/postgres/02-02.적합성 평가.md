# 적합성 평가 — Just Use Postgres?

---

> "Just Use Postgres" 라는 표현이 운영 환경에서 자주 들린다. 한 RDBMS 가 OLTP·검색·JSON·시계열·지리공간·메시지 큐를 모두 받아 주니 단일 시스템 운영이 가능하다는 발상이다. 본 챕터는 이 발상의 의미·한계와, 다른 시스템이 더 나은 자리, 그리고 마이그레이션 검토 시 고려사항을 정리한다.



## "Just Use Postgres" 의 진짜 의미

> 단일 시스템으로 모든 것을 한다는 의미가 아니다. *처음부터 단일 시스템으로 시작한다* 는 의미다.

운영 시스템 설계의 흔한 함정이 *처음부터 polyglot persistence* 다. 신규 프로젝트가 OLTP 는 PostgreSQL, 캐시는 Redis, 검색은 Elasticsearch, 분석은 ClickHouse, 메시지 큐는 Kafka 로 시작한다. 다섯 시스템의 백업·모니터링·HA·지식이 첫날부터 따라온다.

"Just Use Postgres" 는 이 함정에 대한 답이다. **처음에는 PostgreSQL 한 대로 시작** 한다. 검색이 한계에 부딪히면 Elasticsearch 를 빼고, 분석 쿼리가 운영을 방해하면 분석 시스템을 빼고, 처리량이 부족하면 메시지 큐를 외부로 옮긴다. 한 자리씩 *필요할 때* 쪼갠다. 데이터베이스 unbundling 이라고 부르고([`../02-03.스트리밍 시스템 철학.md`](../02-03.스트리밍%20시스템%20철학.md) 참고) 운영의 표준 패턴이다.

이 발상의 가치는 분명하다. **운영 단순함** 으로 시작하고, **실제 한계가 보일 때만 복잡도를 받는다**. 가설로 한계를 예측하지 않고 측정으로 결정한다. 작은 팀에서 이 패턴이 거의 항상 답이다.



## PostgreSQL 한 대로 어디까지 가는가

> 운영 데이터를 보면 답이 명확하다. 거의 모든 자리에서 한참 멀리 갈 수 있다.

| 워크로드 | 단일 PostgreSQL 한도 (대략) |
|---------|---------------------------|
| OLTP 처리량 | 수만 TPS (튜닝 + 좋은 하드웨어) |
| 데이터 크기 | TB 단위 |
| 동시 커넥션 | 수백 (PgBouncer 위에서 수만 클라이언트) |
| 분석 쿼리 | 수억 행까지 |
| 전문 검색 | 수천만 문서 |
| 메시지 처리량 | 분당 수만 건 |

이 숫자가 *대부분 운영 시스템의 실제 요구를 압도* 한다. 수억 사용자 규모 SaaS 가 아니라면 PostgreSQL 한 대 + 적절한 캐시면 거의 모든 자리에서 충분하다.

운영의 함정 한 가지가 *조기 최적화* 다. 미래의 가상 트래픽을 가정해 첫날부터 분산 시스템으로 가는 결정. 운영 부담은 즉시 발생하지만 그 트래픽이 실제로 오는지는 모른다. PostgreSQL 한 대가 한계에 부딪히는 시점이 와도, 그 시점이 올 때까지의 운영 단순함이 가져다 준 가치가 보통 더 크다.



## 다른 시스템이 더 나은 자리

> "Just Use Postgres" 도 한계가 있다. 다음 조건이 모이면 다른 시스템이 답이다.

**거대 OLAP — Snowflake·BigQuery·ClickHouse**

수 TB 이상 데이터, 복잡한 집계 쿼리, 분석가 다수 동시 사용. PostgreSQL 의 행 지향 저장이 컬럼 지향 OLAP 에 비해 한 자릿수 느리다. 같은 쿼리가 PostgreSQL 30 분, ClickHouse 10 초가 흔하다.

**거대 처리량 메시지 큐 — Kafka·Pulsar**

초당 수십만 건 이상, 여러 컨슈머 그룹이 다른 속도로 같은 토픽을 읽음, retention 수일~수년. PostgreSQL 의 SKIP LOCKED + LISTEN/NOTIFY 패턴이 분당 수만 건까지는 받아 주지만 그 이상은 Kafka 가 답이다.

**전 세계 분산·강한 일관성 — CockroachDB·Spanner·Yugabyte**

다지역 배포, 지역 간 강한 일관성, 자동 리밸런싱. PostgreSQL 의 streaming replication 으로는 다지역 강한 일관성이 어렵다. 글로벌 SaaS 의 핵심 시스템이라면 NewSQL 이 답이다.

**거대 KV 캐시 — Redis·Memcached**

인메모리, 1 ms 미만 지연, 자료구조 풍부함. PostgreSQL 도 캐시처럼 쓸 수 있지만 Redis 가 한 자릿수 빠르고 자료구조 지원이 압도적이다.

**거대 그래프 — Neo4j·Neptune**

수십~수백 단계 가변 깊이 탐색, 관계 자체가 분석 대상. PostgreSQL 의 재귀 CTE 도 그래프 쿼리를 풀 수 있지만 그래프 DB 가 표현력·성능에서 압도적이다.

**검색 엔진 — Elasticsearch·OpenSearch**

수억 문서, 한국어·일본어 형태소 분석, 패싯·집계 검색, 분산 인덱싱. PostgreSQL 의 GIN + tsvector 가 수천만 문서까지는 받쳐 주지만 그 이상은 검색 엔진이 답이다.

**벡터 검색 (대규모) — Pinecone·Weaviate·Milvus**

수천만~수억 벡터, 빠른 ANN(근사 최근접 이웃) 검색. pgvector 익스텐션이 수백만 벡터까지 충분하지만 그 이상은 벡터 DB 전용 시스템이 답이다.



## 마이그레이션 검토 — 옮기기 전에 무엇을 보는가

> 다른 DB 에서 PostgreSQL 로(또는 그 반대) 옮길 때의 결정 프레임이다.

**1. 진짜 한계인가** — 현재 시스템의 한계가 정말 *DB 자체* 의 한계인지 확인한다. 인덱스 부재·잘못된 쿼리·하드웨어 부족이 원인이면 마이그레이션이 답이 아니다. EXPLAIN·`pg_stat_statements`·하드웨어 메트릭으로 먼저 검증한다.

**2. 마이그레이션 비용 견적** — 데이터 이전, 스키마 변환, 애플리케이션 코드 수정, 테스트 비용이 모두 들어간다. 1 인 월 단위가 아니라 팀 분기 단위다.

**3. 새 시스템의 운영 비용** — 백업·모니터링·HA·재해 복구를 다시 짜야 한다. 팀에 그 시스템 운영 경험이 없으면 학습 곡선이 따라온다.

**4. 점진 전환 가능성** — 한 번에 옮기는 big-bang 마이그레이션은 거의 항상 사고로 이어진다. dual-write·shadow read·점진 트래픽 전환 같은 패턴으로 위험을 분산한다.

**5. 롤백 계획** — 새 시스템에서 문제가 발견되면 옛 시스템으로 즉시 돌아갈 수 있어야 한다. 이 계획 없이 시작하면 고립된 채 디버깅한다.

운영 결정의 한 줄 — *마이그레이션은 마지막 답* 이다. 현재 시스템 안에서의 답을 다 시도한 후에 검토한다.



## 팀 PostgreSQL 경험이 없을 때

> 신규 프로젝트에 PostgreSQL 도입을 검토 중인데 팀 경험이 약한 자리. 운영의 흔한 상황이다.

답은 단계적이다.

**1. 학습 환경에서 실습** — Docker 로 띄운 PostgreSQL 로 구체 워크로드 PoC. 1~2 주.

**2. 핵심 운영 패턴 학습** — 백업·복구, 인덱스 설계, EXPLAIN 해석, MVCC + VACUUM. 본 카테고리의 다른 챕터들이 그 자리다.

**3. 비즈니스 영향 적은 자리부터** — 내부 도구·로그 저장 같은 자리부터 도입한다. 운영 사고가 비즈니스를 직접 흔들지 않는다.

**4. 관리형 서비스 검토** — AWS RDS·GCP Cloud SQL·Aurora 같은 관리형 서비스가 운영 부담을 한 자릿수 줄여 준다. 팀이 자체 운영 가능해질 때까지의 답으로 적절하다.

**5. 외부 전문가 / 컨설팅** — 도입 초기 한두 달의 짧은 컨설팅이 큰 투자 대비 수익이 좋다. 흔한 함정과 운영 패턴을 압축적으로 학습한다.



## PostgreSQL 의 약점 — 솔직히

> "Just Use Postgres" 가 만능이 아닌 자리들이다.

**대규모 분석** — 컬럼 저장이 없어 OLAP 워크로드에서 한 자릿수 느리다. cstore_fdw 같은 익스텐션이 있지만 안정성이 낮다.

**수평 확장** — 단일 마스터 + 다수 read replica 가 표준이다. 쓰기 처리량은 단일 마스터에 묶인다. Citus·Aurora 같은 확장이 답이지만 표준 PostgreSQL 자체는 수평 확장이 약하다.

**복잡한 다지역 배포** — Streaming replication 이 동기 복제도 가능하지만 다지역 강한 일관성은 어렵다. 비동기 복제로 인한 지연이 항상 따라온다.

**스토어드 프로시저** — Oracle·SQL Server 의 PL/SQL 만큼 풍부하지 않다. PL/pgSQL 은 단순한 자리에는 충분하지만 복잡한 비즈니스 로직을 옮기기에는 약하다.

**복제 모니터링** — `pg_stat_replication` 으로 볼 수 있지만 MySQL·Oracle 의 도구만큼 친절하지 않다. 운영자 학습이 필요하다.

**소수 동시성 패턴** — XA 분산 트랜잭션이 가능하지만 운영에서 드물게 쓴다. 마이크로서비스 간 트랜잭션은 Saga 같은 패턴으로 우회한다.

운영 결정 시 이 약점들을 알고 시작한다. 약점이 자기 워크로드의 핵심 요구가 아니라면 PostgreSQL 이 여전히 답이다.



## 면접 대비 체크리스트

1. "Just Use Postgres" 가 *단일 시스템으로 모든 것* 이 아니라 *PostgreSQL 로 시작* 의 의미라는 주장의 의미는?
2. 단일 PostgreSQL 의 운영 한도가 *대부분 시스템의 실제 요구를 압도* 한다는 데이터 근거는?
3. PostgreSQL 보다 다른 시스템이 답인 자리 다섯 가지를 들 수 있는가?
4. 마이그레이션 검토 시 다섯 가지 결정 기준은? "현재 시스템 안에서의 답을 먼저" 의 의미는?
5. 팀 PostgreSQL 경험이 없을 때 단계적 도입 패턴은?
6. PostgreSQL 의 솔직한 약점 다섯 가지와 운영 영향은?
7. Polyglot persistence 의 비용과 unbundling 패턴이 그 비용을 어떻게 우회하는가?
