# 메시지 큐

---

> "Kafka 대신 PostgreSQL 을 메시지 큐로 써도 되는가" 라는 질문이 있다. 답은 자리에 따라 다르다. 트랜잭션 원자성이 핵심이고 처리량이 분당 수만 건 이하라면 PostgreSQL 이 충분히 답이다. `SKIP LOCKED` 와 `LISTEN/NOTIFY` 가 이 자리의 핵심 도구다. 본 챕터는 운영 결정 기준, 큐 테이블 설계, `pgmq` 익스텐션을 정리한다.



## PostgreSQL 을 메시지 큐로 쓸지의 결정

> 거의 항상 답은 "Kafka 가 더 나음" 이지만, 세 조건이 모이면 PostgreSQL 이 더 좋은 답이 된다.

| 조건 | PostgreSQL 큐 | 외부 큐 (Kafka·SQS) |
|------|--------------|---------------------|
| 트랜잭션 원자성 (DB 변경 + 메시지 발행) | ✓ 자연스러움 | △ Outbox 패턴 필요 |
| 처리량 | < 1 만 건/초 | 수십만~수백만 건/초 |
| 운영 시스템 수 | 적게 | DB + 큐 + 모니터링 |
| 메시지 보존 | 영구 가능 | retention 정책 |
| 정확한 순서 보장 | 자연스러움 | 파티션 단위 |

세 조건 — **트랜잭션 원자성·낮은 처리량·운영 단순함** — 이 모이면 PostgreSQL 큐가 답이다.

대표 자리가 **Outbox 패턴** 이다. 사용자가 주문하면 두 가지 일이 동시에 일어나야 한다 — orders 테이블에 INSERT 와 알림 메시지 발행. 두 일이 별도 시스템(DB + Kafka) 이면 *분산 트랜잭션* 이 필요해 운영이 복잡해진다. 두 일을 같은 DB 안의 트랜잭션으로 묶고, 별도 워커가 outbox 테이블에서 메시지를 꺼내 외부로 발행하는 패턴이 표준 답이다([`../../04_messaging/`](../../04_messaging/) 참고).



## 큐 테이블 설계 — SKIP LOCKED

> 핵심 메커니즘은 `SELECT ... FOR UPDATE SKIP LOCKED` 다. 여러 워커가 같은 테이블에서 메시지를 가져갈 때 충돌 없이 동작한다.

```sql
CREATE TABLE message_queue (
    id BIGSERIAL PRIMARY KEY,
    payload JSONB NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
        -- 'pending', 'processing', 'completed', 'failed'
    created_at TIMESTAMPTZ DEFAULT now(),
    processed_at TIMESTAMPTZ,
    retry_count INTEGER DEFAULT 0,
    error_message TEXT
);

CREATE INDEX idx_queue_pending ON message_queue (created_at)
WHERE status = 'pending';
```

부분 인덱스가 핵심 최적화다. `WHERE status = 'pending'` 인덱스가 처리 대기 메시지만 인덱싱하므로, 큐가 누적되어도 인덱스 크기가 안 부풀어 오른다.

워커의 메시지 가져오기 패턴이 다음과 같다.

```sql
BEGIN;
  -- 한 건 잠그고 가져옴
  UPDATE message_queue
  SET status = 'processing'
  WHERE id = (
    SELECT id FROM message_queue
    WHERE status = 'pending'
    ORDER BY created_at
    LIMIT 1
    FOR UPDATE SKIP LOCKED  -- ★
  )
  RETURNING *;
COMMIT;
```

**`SKIP LOCKED`** 가 핵심이다. 다른 워커가 잠근 행은 건너뛰고 다음 가용한 행을 잡는다. 락 대기 없이 워커들이 병렬로 동작한다. 이 한 줄이 PostgreSQL 메시지 큐의 운영 가능성을 만든다.

처리 완료는 단순한 UPDATE 다.

```sql
UPDATE message_queue
SET status = 'completed', processed_at = now()
WHERE id = $1;

-- 또는 실패 시 재시도
UPDATE message_queue
SET status = 'pending',
    retry_count = retry_count + 1,
    error_message = $2
WHERE id = $1 AND retry_count < 5;
```



## LISTEN/NOTIFY — 폴링 없는 알림

> SKIP LOCKED 만 쓰면 워커가 *주기적으로 폴링* 해야 한다. LISTEN/NOTIFY 가 그 폴링을 줄인다.

```sql
-- 발행자
NOTIFY message_queue_channel, 'new_message';

-- 또는 트리거로 자동
CREATE FUNCTION notify_new_message() RETURNS TRIGGER AS $$
BEGIN
  PERFORM pg_notify('message_queue_channel', NEW.id::TEXT);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER message_queue_notify
AFTER INSERT ON message_queue
FOR EACH ROW EXECUTE FUNCTION notify_new_message();
```

```python
# 컨슈머 (Python psycopg2 예)
import psycopg2
conn = psycopg2.connect(...)
conn.set_isolation_level(0)
cur = conn.cursor()
cur.execute("LISTEN message_queue_channel;")

while True:
    if select.select([conn], [], [], 5) == ([], [], []):
        # 5 초 폴링 fallback
        process_pending_messages()
    else:
        conn.poll()
        while conn.notifies:
            notify = conn.notifies.pop(0)
            process_message(notify.payload)
```

LISTEN/NOTIFY 의 함정 두 가지가 있다. **트랜잭션 커밋 시점에만 전달** — 발행자의 트랜잭션이 커밋되어야 NOTIFY 가 실제로 전달된다. **메시지 손실 가능** — 컨슈머가 LISTEN 안 하고 있는 동안의 NOTIFY 는 사라진다. 그래서 운영 패턴은 *NOTIFY 로 즉시성을 받쳐 주고, 폴링으로 손실 방지* 의 조합이다.



## 큐 운영 함정 — Stale Message·Long-Running Worker

> 운영에서 자주 부딪히는 자리들이다.

**Stale processing** — 워커가 'processing' 상태로 잠그고 죽으면 그 메시지가 영원히 처리 안 된다. 답은 *타임아웃 기반 회수* 다.

```sql
-- 5 분 이상 processing 인 메시지를 pending 으로 되돌림
UPDATE message_queue
SET status = 'pending'
WHERE status = 'processing'
  AND processed_at IS NULL
  AND updated_at < now() - interval '5 minutes';
```

이 로직을 별도 cron 잡으로 주기적 실행한다.

**Long-running worker** — 한 메시지 처리가 오래 걸리면 다른 워커들이 처리 못 하는 동안 큐가 쌓인다. 답은 *짧은 처리 + idempotent 설계* 다. 메시지 한 건이 1 초 안에 끝나도록 작업을 잘게 쪼개고, 같은 메시지가 두 번 처리되어도 결과가 같도록 한다.

**큐 길이 모니터링** — 처리 안 된 메시지 누적은 시스템 이상 신호다.

```sql
-- 모니터링 쿼리
SELECT status, count(*), min(created_at) AS oldest
FROM message_queue
WHERE status IN ('pending', 'processing')
GROUP BY status;
```

이 결과를 Prometheus·Grafana 같은 도구로 시각화하고 임계 초과 시 알람을 띄운다.



## pgmq — PostgreSQL 메시지 큐 익스텐션

> 위의 모든 패턴을 라이브러리로 묶은 익스텐션이다. SKIP LOCKED·재시도·DLQ·통계가 한 묶음이다.

```sql
CREATE EXTENSION pgmq;

-- 큐 생성
SELECT pgmq.create('my_queue');

-- 메시지 발행
SELECT pgmq.send('my_queue', '{"order_id": 123}');

-- 메시지 가져오기 (visibility timeout 30 초)
SELECT * FROM pgmq.read('my_queue', 30, 1);

-- 처리 완료
SELECT pgmq.delete('my_queue', $msg_id);

-- DLQ 로 이동 (재시도 한도 초과 시)
SELECT pgmq.archive('my_queue', $msg_id);
```

**Visibility timeout** 이 핵심 메커니즘이다. 메시지를 읽으면 그 시간 동안 다른 컨슈머에게 안 보인다. 그 안에 처리 완료(delete) 못 하면 자동으로 다시 가시화된다. AWS SQS·RabbitMQ 의 표준 동작이고 pgmq 가 같은 의미로 구현한다.

운영에서 직접 큐 테이블을 짜는 것보다 pgmq 가 거의 항상 답이다. 검증된 코드, 표준 인터페이스, 통계 기능이 모두 들어 있다. 다만 클라우드 PostgreSQL 일부가 pgmq 를 지원 안 할 수 있어 사전 확인이 필요하다.



## PostgreSQL 큐 vs Kafka·SQS — 결정 가이드

> 운영 결정의 한 줄 요약.

**PostgreSQL 큐가 답인 자리**:
- 작은~중간 규모 SaaS 백엔드
- 트랜잭션 원자성이 핵심 (Outbox)
- 운영 시스템 수를 줄이고 싶음
- 처리량 < 1 만 건/초
- 정확한 순서 보장이 단순해야 함

**Kafka 가 답인 자리**:
- 대규모 처리량 (수십만 건/초+)
- 여러 컨슈머 그룹이 같은 메시지를 다른 속도로 읽음
- 메시지 영구 보관 + 재처리
- 분산 시스템 핵심 인프라

**SQS 가 답인 자리**:
- AWS 환경 + 처리량 변동이 큼
- 운영 부담 최소화 (관리형)
- 표준 큐 의미면 충분

상당수 작은 서비스가 *처음부터 Kafka* 로 가는 사고가 흔하다. PostgreSQL 큐로 시작해 한계가 분명할 때 Kafka 로 옮기는 패턴이 거의 항상 운영 비용 측면에서 유리하다.
