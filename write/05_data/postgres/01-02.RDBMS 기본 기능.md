# RDBMS 기본 기능

---

> PostgreSQL 의 기본기 — 객체 계층, 트랜잭션·MVCC, 제약조건, 함수·트리거, 뷰, 역할 기반 접근 제어. 본 챕터는 운영에서 매일 쓰는 기능들의 PostgreSQL 특화 측면을 정리한다. 트랜잭션 이론은 [`../01-04.트랜잭션과 격리 수준.md`](../01-04.트랜잭션과%20격리%20수준.md) 가 다루므로 본 챕터는 PostgreSQL 의 구체 동작에 집중한다.



## 객체 계층 — Cluster · Database · Schema

> 다른 RDBMS 와 다르게 PostgreSQL 은 *세 단계 계층* 을 갖는다. 이 차이가 멀티테넌시 설계의 첫 결정에 들어온다.

```
Cluster (서버 인스턴스)
├── Database mydb
│   ├── Schema public
│   │   ├── Table users
│   │   └── Table orders
│   └── Schema reporting
│       └── View daily_sales
└── Database analytics
    └── Schema staging
```

**Cluster** 는 한 PostgreSQL 서버 프로세스다. 여러 데이터베이스가 그 안에 산다. **Database** 는 격리 단위다. 다른 DB 의 객체에 직접 접근 못 한다(FDW 또는 dblink 가 필요). **Schema** 는 한 DB 안의 네임스페이스다. 같은 DB 의 다른 스키마에는 자유롭게 접근 가능하다.

운영 결정의 자리는 멀티테넌시 모델에서 들어온다. **Database per tenant** 는 격리가 가장 강하지만 cluster 자원을 많이 쓴다. **Schema per tenant** 는 절충 — 같은 DB 안에서 스키마로 분리해 격리는 약하지만 자원은 효율적이다. **Single schema + tenant_id 컬럼** 은 가장 단순하지만 격리가 약하다([`../../05_data/02-04.샤딩.md`](../../05_data/02-04.샤딩.md) 의 멀티테넌시 절 참고).



## 트랜잭션과 ACID — PostgreSQL 의 구체 동작

> PostgreSQL 은 ACID 4 글자 모두를 강하게 보장한다. 그 위에 MVCC 가 동작한다.

기본 격리 수준은 **READ COMMITTED** 다. 이 자리에서 lost update 와 write skew 가 가능하므로([`../01-04.트랜잭션과 격리 수준.md`](../01-04.트랜잭션과%20격리%20수준.md)), 운영 코드가 이를 직접 막아야 한다.

```sql
-- 명시적 트랜잭션
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- 격리 수준 변경
BEGIN ISOLATION LEVEL SERIALIZABLE;
  -- ...
COMMIT;

-- SAVEPOINT 로 부분 롤백
BEGIN;
  INSERT INTO orders VALUES (...);
  SAVEPOINT before_items;
  INSERT INTO order_items VALUES (...);
  ROLLBACK TO SAVEPOINT before_items;
  -- orders 는 살아남고, items 만 롤백
COMMIT;
```

PostgreSQL 의 한 가지 특이점이 **암시적 트랜잭션** 이다. 단일 SQL 문장도 자동으로 트랜잭션 안에서 동작한다. `UPDATE table SET ...` 한 줄이 실패해도 부분 적용되지 않는다.

`autocommit` 모드가 기본이다. psql 에서 `BEGIN` 없이 SQL 을 실행하면 각 문장이 별도 트랜잭션으로 자동 커밋된다. 명시적 트랜잭션이 필요한 자리에서는 `BEGIN` 으로 시작한다.



## MVCC — VACUUM 운영

> MVCC 의 동작 원리(`xmin`/`xmax`, 스냅샷, "Readers never block Writers")는 [`../01-04.트랜잭션과 격리 수준.md`](../01-04.트랜잭션과%20격리%20수준.md) 의 MVCC 절이 자세히 다룬다. 본 절은 그 위에서 PostgreSQL 의 운영 비용 — 즉 dead tuple 회수 — 만 본다.

MVCC 비용이 PostgreSQL 운영에 들어오는 자리는 **dead tuple** 이다. 더 이상 어떤 트랜잭션도 보지 못하는 옛 행 버전이 디스크에 누적되므로 **VACUUM** 이 주기적으로 청소해 줘야 한다. `autovacuum` 이 자동으로 돌지만 대량 UPDATE/DELETE 직후에는 수동 `VACUUM ANALYZE` 가 필요한 경우가 있다.

`VACUUM FULL` 은 디스크 공간을 즉시 회수하지만 테이블 전체 락을 잡으므로 운영 시간에 안 쓴다. `VACUUM` 은 비차단이라 평소 운영에서 안전하다. 한 단계 더 강한 도구가 `pg_repack` — VACUUM FULL 의 효과를 비차단으로 얻는 외부 도구다.

운영에서 흔히 부딪히는 함정 한 가지가 **long-running transaction** 이다. 한 트랜잭션이 한 시간 동안 열려 있으면 그 시점 이후의 모든 dead tuple 이 회수되지 못한다(`xmin horizon` 이 그 트랜잭션에 막힘). `pg_stat_activity` 의 `xact_start` 가 오래된 세션을 정기적으로 점검해야 테이블 bloat 가 폭주하는 사고를 막는다.



## 제약조건 — DB 가 도와주는 불변식

> 데이터 무결성을 코드 외에서도 강제한다. 운영의 안전망이다.

```sql
CREATE TABLE accounts (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    age INTEGER CHECK (age >= 0 AND age <= 150),
    balance NUMERIC(15,2) DEFAULT 0,
    referrer_id INTEGER REFERENCES accounts(id),
    CHECK (balance >= 0)  -- 테이블 레벨 CHECK
);
```

| 제약 | 보장 |
|------|------|
| `NOT NULL` | 값이 반드시 있음 |
| `UNIQUE` | 값이 중복되지 않음 |
| `PRIMARY KEY` | NOT NULL + UNIQUE + 자동 인덱스 |
| `FOREIGN KEY` | 참조 무결성 |
| `CHECK` | 사용자 정의 조건 |
| `EXCLUDE` | 임의 연산자로 충돌 방지 (예: 시간 범위 겹침) |

`EXCLUDE` 가 PostgreSQL 의 강점이다. 회의실 예약처럼 "같은 회의실의 같은 시간 범위가 겹치면 안 됨" 을 DB 가 직접 강제한다.

```sql
CREATE TABLE reservations (
    room_id INTEGER,
    period TSRANGE,
    EXCLUDE USING gist (room_id WITH =, period WITH &&)
);
```

함수 호출 한 번 없이 시간 범위 충돌을 막는다. 다른 RDBMS 에서는 트리거 + 추가 락으로 짜야 하는 자리다.



## 함수·트리거·뷰

> 비즈니스 로직 일부를 DB 안에 둔다. 운영 결정 — 어디까지를 DB 안에 둘지 — 가 따라온다.

**함수(Function)** 는 PL/pgSQL·PL/Python·SQL 같은 언어로 작성한다. 운영에서 자주 보는 자리는 트리거 안의 짧은 로직, 복잡한 집계 함수, 도메인 특화 변환이다. 비즈니스 로직 전부를 함수로 옮기는 것은 권장되지 않는다 — 버전 관리·테스트·디버깅이 애플리케이션 코드보다 어렵기 때문이다.

**트리거(Trigger)** 는 INSERT/UPDATE/DELETE 시 자동 실행되는 함수다. 감사 로그·자동 갱신·제약 강제 같은 자리에 어울린다. 단점은 *눈에 안 보임* — 트리거의 존재를 모르고 INSERT 한 사람이 예상치 못한 동작을 만나는 사고가 흔하다. 트리거를 쓰면 코드 베이스 어딘가에 명시적 문서가 따라와야 한다.

**뷰(View)** 는 가상 테이블이다. 매번 쿼리할 때 정의된 SQL 이 실행된다. 복잡한 JOIN 을 한 곳에 모아 재사용할 수 있다.

**머터리얼라이즈드 뷰(Materialized View)** 는 결과를 디스크에 저장하는 뷰다. 무거운 집계 쿼리를 캐시하는 자리에 어울리고, `REFRESH MATERIALIZED VIEW CONCURRENTLY` 가 운영의 거의 표준이다(갱신 중에도 조회를 막지 않음). DDL/사용 패턴과 캐시 신선도 트레이드오프는 [`../01-07.캐싱 전략.md`](../01-07.캐싱%20전략.md) 의 Materialized View 절에 정리되어 있다.



## 역할 기반 접근 제어 (RBAC)

> 사용자(role) 단위로 권한을 나눠 보안을 강제한다.

PostgreSQL 의 모든 user 와 group 이 사실은 동일한 *role* 이다. 둘의 차이는 LOGIN 권한 유무뿐이다.

```sql
CREATE ROLE app_user WITH LOGIN PASSWORD 'secret';
CREATE ROLE readonly WITH LOGIN PASSWORD 'readsecret';

GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;

-- 역할 상속
CREATE ROLE analyst;
GRANT readonly TO analyst;
```

운영의 한 패턴이 **최소 권한 원칙** 이다. 애플리케이션이 쓰는 role 에는 그 애플리케이션이 정말 필요로 하는 권한만 준다. 슈퍼유저(`postgres`) 는 운영자만 쓰고 애플리케이션은 그 권한을 절대 안 갖는다.

**Row Level Security(RLS)** 가 한 단계 더 강한 도구다. 같은 테이블의 행 단위로 누가 어떤 행을 볼 수 있는지를 정책으로 강제한다.

```sql
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON orders
  USING (tenant_id = current_setting('app.tenant_id')::INTEGER);
```

멀티테넌시 환경에서 애플리케이션이 `SET app.tenant_id = 5` 만 호출하면 그 세션은 tenant 5 의 행만 본다. 코드에서 WHERE 절을 빠뜨려도 데이터가 새지 않는다는 안전망이다.
