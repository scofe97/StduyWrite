# ORM 개념
---
> ORM 은 객체와 관계형 테이블 사이의 불일치 (Object-Relational Impedance Mismatch) 를 다루는 도구다. JDBC 의 반복을 줄이고, 객체 그래프와 테이블 그래프를 자동 매핑하지만, 그 추상화가 *마법으로 보일 때* 가장 위험하다.

## 1. ORM 이 푸는 문제

> SQL 직접 작성의 반복과 객체-테이블 불일치를 동시에 해결한다.

JDBC 직접 사용의 문제:

1. 같은 패턴 (Connection → PS → RS → Mapping) 반복.
2. 객체 그래프를 SQL 결과로 재구성하는 코드가 흩어짐.
3. 변경 감지가 수동 — UPDATE 문 직접 작성.

ORM 의 해결:

1. **객체 ↔ 행 매핑 자동화** — `@Entity` 만 박으면.
2. **객체 그래프 = 외래키 그래프** — `@OneToMany`, `@ManyToOne` 으로 관계 표현.
3. **변경 감지** — 객체 필드 변경이 자동 UPDATE.
4. **언어 통일** — SQL 대신 JPQL/Criteria/QueryDSL.

## 2. JPA 와 Hibernate

> JPA 는 인터페이스 (Java EE 스펙), Hibernate 는 구현체. Spring Boot 는 기본 Hibernate.

| 위치 | 역할 |
|------|------|
| JPA (`jakarta.persistence`) | 표준 인터페이스·애너테이션 (`@Entity`, `EntityManager`) |
| Hibernate | 가장 흔한 구현체. 추가 기능 (`@Type`, `@Filter` 등) |
| Spring Data JPA | Repository 패턴으로 더 추상화 |

코드는 가능하면 JPA 표준만 사용 → 다른 구현체로 교체 자유도 확보.

## 3. JPA 가 동작하는 위치

> 애플리케이션과 JDBC 사이 — JDBC 를 직접 부르는 대신 JPA 가 부른다.

```
[Application]
    ↓ Entity, EntityManager
[JPA / Hibernate]
    ↓ SQL
[JDBC Driver]
    ↓ Network
[Database]
```

JPA 는 JDBC 를 대체하지 않고 *그 위* 에 얹힌다. 커넥션 풀·트랜잭션 등은 JDBC/Spring 의 것을 그대로 쓴다.

## 4. JPA 사용 이유

> 생산성과 유지보수의 두 축 — 다만 SQL 을 모르고 쓰면 함정이 크다.

### 4-1. 생산성

```java
// JDBC
String sql = "INSERT INTO member (name, age) VALUES (?, ?)";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setString(1, member.getName());
    ps.setInt(2, member.getAge());
    ps.executeUpdate();
}

// JPA
em.persist(member);
```

한 줄로 끝. INSERT SQL 자동 생성, 파라미터 자동 바인딩.

### 4-2. 유지보수

스키마 변경 시 `@Column` 매핑만 바꾸면 끝. SQL 문자열 곳곳을 찾아 바꿀 필요 없음.

### 4-3. 함정

JPA 가 *어떤 SQL 을 만들고 언제 실행하는지* 를 이해하지 못하면 N+1 (`./03-03.프록시와 N+1.md`), Lazy 함정, 트랜잭션 경계 오용 같은 문제가 누적된다. ORM 은 SQL 을 *숨기는* 것이 아니라 *생성하는* 도구다.

## 5. JPA 동작 — 저장과 조회

> 가장 단순한 두 동작이 JPA 의 핵심 추상을 보여준다.

### 5-1. 저장 (persist)

```java
@Transactional
public Long save(Member m) {
    em.persist(m);
    return m.getId();   // INSERT 후 PK 가 채워져 있음
}
```

`persist` 는 즉시 INSERT 가 아니라 *영속성 컨텍스트에 넣음*. 실제 SQL 은 flush 시점 (보통 COMMIT 직전).

### 5-2. 조회 (find)

```java
Member m = em.find(Member.class, 1L);
m.setName("changed");   // SET 만 호출, save 없음
// 트랜잭션 종료 시 자동 UPDATE
```

`find` 는 1차 캐시 → DB 순으로 조회. 조회된 객체 변경은 자동으로 dirty checking 으로 UPDATE 생성. 자세한 영속성 컨텍스트 동작은 `./01-02.JPA 시작과 영속성 컨텍스트.md`.

## 6. JPA 와 상속

> 객체 상속을 테이블로 표현하는 세 가지 방법.

| 전략 | 동작 | 적합 |
|------|------|------|
| 조인 전략 (`JOINED`) | 부모·자식 각각 테이블, JOIN 으로 조회 | 정규화 우선 |
| 단일 테이블 (`SINGLE_TABLE`) | 모든 자식을 한 테이블에 (NULL 컬럼 많음) | 조회 성능 우선 (기본) |
| 구현 클래스 (`TABLE_PER_CLASS`) | 자식마다 별도 테이블 (부모 컬럼 중복) | 거의 안 씀 |

상세는 `./02-03.상속과 값 타입.md`.
