# 낙관적 비관적 락 (@Version, @Lock, LockModeType)
---
> 같은 데이터를 동시에 변경하는 두 트랜잭션의 충돌을 어떻게 다룰 것인가 — 낙관적은 "충돌은 드물 것" 가정, 비관적은 "충돌이 잦으므로 미리 막자" 가정.

## 1. 낙관적 락 (Optimistic Lock)

> 락을 잡지 않고 진행, 커밋 시점에 충돌 검증 — `@Version` 컬럼 사용.

```java
@Entity
public class Product {
    @Id @GeneratedValue
    private Long id;
    private String name;
    private int stock;

    @Version
    private Long version;   // JPA 가 자동 관리
}
```

흐름:

1. `Product p = em.find(Product.class, 1L);` — version = 5 로 로딩.
2. 다른 트랜잭션이 같은 행 UPDATE → version = 6.
3. 우리 트랜잭션이 커밋 → `UPDATE product SET ... version = 6 WHERE id = 1 AND version = 5`.
4. 영향 행 수 = 0 → 다른 트랜잭션이 먼저 변경 → `OptimisticLockException` 발생.

이 동작이 의미하는 것: 우리는 락을 잡지 않았지만 *충돌이 있었다면 감지 가능*.

### 1-1. 적합 도메인

1. 충돌 빈도가 낮음 (전체 사용자의 1% 미만이 같은 행 동시 변경).
2. 재시도 비용이 작음.
3. 처리량이 중요.

전형적: 사용자 프로필 수정, 게시글 편집.

### 1-2. 부적합

1. 충돌 빈도 높음 (재고 차감·결제 같은 핫스팟).
2. 재시도 시 외부 시스템 호출이 다시 일어남 (이메일·결제 게이트웨이).

## 2. 비관적 락 (Pessimistic Lock)

> 미리 락을 잡고 시작 — `SELECT FOR UPDATE`.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Product findByIdForUpdate(@Param("id") Long id);
```

흐름:

1. `productRepository.findByIdForUpdate(1L)` — DB 에 X-lock.
2. 다른 트랜잭션이 같은 행 SELECT FOR UPDATE → 대기.
3. 우리 트랜잭션 커밋 → 락 해제 → 다른 트랜잭션 진행.

DB 레벨의 락이라 충돌 자체가 일어나지 않는다.

### 2-1. 적합 도메인

1. 충돌 빈도 높음.
2. 재시도 비용 큼 (외부 시스템 호출 포함).
3. 일관성이 처리량보다 중요.

전형적: 재고 차감·계좌 이체.

### 2-2. 부적합

1. 동시 요청이 많은 핫스팟 (대기 큐 폭증).
2. 트랜잭션이 길음 (락 보유 시간 길어짐).

## 3. LockModeType 의 종류

> JPA 의 표준 락 모드 — DB 별 구현 차이 가능.

| LockModeType | SQL | 의미 |
|--------------|-----|------|
| `OPTIMISTIC` | (없음, 커밋 시 version 검증) | 일반 낙관적 락 |
| `OPTIMISTIC_FORCE_INCREMENT` | UPDATE version | 변경 없어도 version 증가 (관련 Entity 변경 추적용) |
| `PESSIMISTIC_READ` | `LOCK IN SHARE MODE` | S-lock |
| `PESSIMISTIC_WRITE` | `FOR UPDATE` | X-lock |
| `PESSIMISTIC_FORCE_INCREMENT` | `FOR UPDATE` + version 증가 | 둘 다 |

대부분 `OPTIMISTIC` 또는 `PESSIMISTIC_WRITE` 선택.

## 4. 낙관적 락의 재시도 패턴

> `OptimisticLockException` 을 받으면 자동 재시도 — Spring Retry 또는 직접 구현.

```java
@Retryable(
    retryFor = OptimisticLockException.class,
    maxAttempts = 3,
    backoff = @Backoff(delay = 100)
)
@Transactional
public void updateStock(Long productId, int delta) {
    Product p = productRepository.findById(productId).orElseThrow();
    p.setStock(p.getStock() + delta);
}
```

`@Retryable` 이 예외 잡아 재시도. 새 트랜잭션·새 SELECT 로 재시도.

여기서 질문 하나 — 재시도가 무한 루프되지 않을까? `maxAttempts` 가 안전망. 3번 실패하면 진짜 충돌 — 사용자에게 알림.

## 5. 둘의 결정 매트릭스

> 같은 도메인이라도 액션별로 다른 선택이 자연스럽다.

| 액션 | 권장 | 이유 |
|------|------|------|
| 사용자 프로필 수정 | 낙관적 | 충돌 드물, 처리량 우선 |
| 게시글 편집 | 낙관적 (또는 lastModified 비교) | 충돌 드물, 재시도 쉬움 |
| 재고 차감 | 비관적 또는 분산 락 | 충돌 잦, 정확성 우선 |
| 계좌 이체 | 비관적 | 동시 이체 시 일관성 필수 |
| 좋아요 증감 | 낙관적 또는 원자 UPDATE | 충돌 드물, INCR/DECR 패턴 |

좋아요 같은 단순 카운터는 `UPDATE ... SET count = count + 1 WHERE id = ?` 로 *락 없는 원자 갱신* 이 더 좋다. JPA 대신 JPQL UPDATE 직접 사용.

```sql
UPDATE post SET like_count = like_count + 1 WHERE id = :id
```

## 6. 분산 환경의 락

> JPA 의 락은 *한 DB 안* 에서만 동작. 여러 서비스가 같은 외부 자원을 다툴 때는 Redis 분산 락 같은 외부 메커니즘.

```
[Service A] ─┐
              ├─ Redis 분산 락 (SET NX EX)
[Service B] ─┘
```

`SET resource:lock owner-id NX EX 10` 로 lease 기반 락. 락 보유 중 서비스가 죽어도 TTL 이 만료되며 자동 해제.
