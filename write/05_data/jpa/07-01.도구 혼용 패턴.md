# 도구 혼용 패턴 — Spring Data JPA + QueryDSL 어댑터 vs 직접 사용

---

> Spring Data JPA 와 QueryDSL 을 한 코드베이스에서 함께 쓸 때 *어댑터로 인터페이스를 유지할지* 또는 *서비스가 직접 두 리포지토리를 호출할지* 가 강의 2편 8장의 트레이드오프 주제다. 본 노트는 두 선택을 비교하고 *어느 선택이 어느 상황에 맞는지* 의 기준을 정리한다.

## 1. 어댑터 선택 — DI·OCP 유지

> 서비스가 자기 도메인의 `ItemRepository` 인터페이스만 알고, 그 뒤에 Spring Data JPA + QueryDSL 구현체를 두는 패턴이다.

```
[ItemService]
     │ depends on
     ↓
[ItemRepository (도메인 인터페이스)]
     ↑ implements
[JpaItemRepositoryV2 (어댑터)]
     │ uses
     ├── [SpringDataJpaItemRepository extends JpaRepository]
     └── [ItemQueryRepository — QueryDSL]
```

이 구조의 장점은 두 가지다.

- `ItemService` 가 *영속성 기술을 모른다*. Spring Data JPA 든 MyBatis 든 어댑터만 바꿔 끼우면 된다.
- 인터페이스가 *도메인 어휘* 로 표현된다. `findById`, `findAll(cond)` 같은 메서드명이 도메인 의미 그대로 남는다.

DI 와 OCP 를 살리는 표준적인 헥사고날 변형이다. *기존 인터페이스를 유지하면서 영속성을 갈아끼울 가능성* 이 있는 프로젝트에 어울린다.

### 1-1. 어댑터의 비용

대신 비용도 따른다.

- 클래스 수가 늘어난다 — 도메인 인터페이스 + 어댑터 + Spring Data JPA 리포지토리 + QueryDSL 리포지토리.
- 어댑터 코드가 *얇은 위임 메서드의 반복* 으로 채워진다. `save`, `findById`, `deleteById` 같은 메서드를 일일이 위임한다.
- 새 메서드 추가 시 *도메인 인터페이스 → 어댑터 → 구현* 까지 세 곳을 손대야 한다.

추상화에도 *유지보수 비용* 이 따른다. 강의는 이 비용이 *영속성 교체 가능성* 의 가치를 넘으면 안 된다고 짚는다.

## 2. 직접 사용 선택 — 구조 단순화

> 서비스가 Spring Data JPA 리포지토리와 QueryDSL 리포지토리를 *직접* 주입받아 호출한다.

```
[ItemServiceV2]
     │ depends on
     ├── [ItemRepositoryV2 extends JpaRepository<Item, Long>]
     └── [ItemQueryRepositoryV2 — QueryDSL]
```

`ItemRepositoryV2` 는 도메인 인터페이스가 아니라 *Spring Data JPA 리포지토리 그 자체* 다. `ItemQueryRepositoryV2` 는 QueryDSL 로 복잡한 검색을 담당한다.

```java
@Service
@RequiredArgsConstructor
@Transactional
public class ItemServiceV2 implements ItemService {

    private final ItemRepositoryV2 itemRepositoryV2;
    private final ItemQueryRepositoryV2 itemQueryRepositoryV2;

    @Override
    public Item save(Item item) {
        return itemRepositoryV2.save(item);  // Spring Data JPA 직접
    }

    @Override
    public List<Item> findItems(ItemSearchCond cond) {
        return itemQueryRepositoryV2.findAll(cond);  // QueryDSL 직접
    }
}
```

이 구조의 장점은 명확하다.

- 어댑터 한 층이 사라진다. 클래스 수가 줄고 위임 메서드 반복이 없다.
- *기본 CRUD 는 Spring Data JPA, 복잡 쿼리는 QueryDSL* 의 책임 분리가 *그대로 보인다*. 어댑터 뒤에 숨지 않는다.

단점은 DI·OCP 측면이다.

- `ItemService` 가 Spring Data JPA 와 QueryDSL 두 기술에 *직접 의존* 한다. 영속성 교체 시 서비스 코드도 손대야 한다.
- 도메인 인터페이스가 없어 *영속성 교체* 라는 시나리오 자체를 사실상 포기한다.

## 3. 두 책임의 분리 — 공통

> 어느 선택이든 *기본 CRUD* 와 *복잡 쿼리* 는 분리하는 편이 자연스럽다.

강의가 권하는 두 리포지토리 분리는 두 선택 모두에서 유효하다.

| 리포지토리 | 역할 | 도구 |
|----------|------|------|
| `ItemRepositoryV2` | 기본 CRUD + 단순 조회 (`findById`, `findAll`, `save`, `delete`) | Spring Data JPA (`JpaRepository`) |
| `ItemQueryRepositoryV2` | 동적 조건 검색·통계·집계 | QueryDSL (`JPAQueryFactory`) |

이 분리의 이유는 도구의 *강점이 다르기 때문* 이다. 단순 CRUD 는 Spring Data JPA 의 쿼리 메소드와 메서드 이름 규약이 가장 짧다. 동적 조건은 QueryDSL 의 타입 안전 메타모델이 강하다. 한 리포지토리에서 두 도구를 섞으면 *어느 쪽도 잘 안 보인다*.

`MemberRepositoryCustom` + `MemberRepositoryImpl` 패턴 ([./03-05](03-05.커스텀%20리포지토리%20패턴.md)) 으로 *한 리포지토리에 합치는 변형* 도 있다. 둘을 하나의 빈으로 호출자에게 보여주려면 그 쪽이, 두 책임을 명확히 분리해 보여주려면 본 절의 분리 방식이 자연스럽다.

## 4. 어느 선택이 어느 상황에 맞는가

> 단일 정답은 없다. *영속성 교체 가능성* 과 *어댑터 비용* 을 저울에 올린다.

| 신호 | 권장 |
|------|------|
| 영속성 기술이 정해져 있고 가까운 시일 내 바꿀 계획 없음 | 직접 사용 |
| 도메인 인터페이스가 *도메인 어휘* 자체로 의미 있음 (예: 헥사고날) | 어댑터 |
| 메서드가 5개 이하의 단순 도메인 | 직접 사용 (어댑터 비용이 가치 초과) |
| 영속성 두 종류가 공존 (RDB + 검색엔진 + 캐시) | 어댑터 (도메인 인터페이스가 한 자리에 모임) |
| 팀에 JPA·QueryDSL 학습 곡선 부담 | 직접 사용 (구조가 그대로 보임) |
| 도메인 인터페이스를 따로 정의하는 *합의된 코드 컨벤션* 있음 | 어댑터 (합의가 비용을 정당화함) |

강의 본문의 결론은 다음 한 문장으로 요약된다.

> *추상화도 비용이 든다. 인터페이스도 비용이 든다. 이 추상화 비용을 넘어설 만큼 효과가 있을 때 추상화를 도입하는 것이 실용적이다.*

## 5. 정리

두 선택의 본질은 *구조의 안정성 vs 단순한 구조와 개발의 편리성* 의 트레이드오프다.

- 어댑터는 *DI·OCP* 를 살리지만 *유지보수 비용* 을 더한다.
- 직접 사용은 *비용을 줄이지만* DI·OCP 를 포기한다.

이분법으로 보일 수 있지만 실제로는 *프로젝트 상황* 에 맞춰 선택하는 문제다. 영속성 교체 가능성이 낮고 도메인이 단순하면 직접 사용이 자연스럽고, 도메인 어휘가 의미 있고 영속성 다양화가 예상되면 어댑터가 자연스럽다. 강의가 강조한 *현재 상황에 맞는 선택을 하는 개발자* 가 이 자리에서 평가받는다.

기본 CRUD + 복잡 쿼리의 분리는 두 선택 모두에서 유효하다. Spring Data JPA + QueryDSL 의 통합 패턴은 [../querydsl/03-07](../querydsl/03-06.스프링%20데이터%20페이징%20통합.md) 가 다룬다.
