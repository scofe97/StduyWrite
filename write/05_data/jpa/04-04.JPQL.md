# JPQL — 기본 문법
---
> JPQL 은 SQL 과 비슷하지만 *Entity 와 필드* 를 대상으로 한다. DB 테이블·컬럼이 아니라 도메인 모델의 어휘로 쿼리한다는 점이 핵심.

## 1. JPQL vs SQL

> SQL 이 테이블·컬럼을 본다면, JPQL 은 Entity·필드를 본다.

```sql
-- SQL
SELECT * FROM member WHERE age > 20;

-- JPQL
SELECT m FROM Member m WHERE m.age > 20;
```

차이:

1. **FROM 절** — Entity 이름 (대소문자 구분).
2. **별칭 필수** — `Member m` 의 `m`.
3. **필드 접근** — `m.age` (컬럼명 아님).
4. **대소문자** — Entity 이름·필드 이름은 대소문자 구분, 키워드는 무관.

JPQL 의 결과는 *Entity 또는 그 값* 으로 매핑된다.

## 2. TypedQuery vs Query

> 결과 타입을 컴파일 타임에 알면 TypedQuery, 모르면 Query.

```java
// TypedQuery — 타입 안전
TypedQuery<Member> q = em.createQuery("SELECT m FROM Member m", Member.class);
List<Member> result = q.getResultList();

// Query — 타입 미지정 (캐스팅 필요)
Query q = em.createQuery("SELECT m FROM Member m");
List<Member> result = (List<Member>) q.getResultList();   // 안 권장
```

거의 항상 TypedQuery. Query 는 동적 SELECT 절 (필드 수가 가변) 에만.

## 3. 결과 조회

> 단건은 `getSingleResult`, 다건은 `getResultList`.

```java
// 다건
List<Member> members = q.getResultList();   // 빈 결과 시 빈 리스트

// 단건
Member m = q.getSingleResult();   // 결과 없으면 NoResultException, 여러 개면 NonUniqueResultException
```

`getSingleResult` 의 예외 처리가 번거로워서 Spring Data 의 `Optional<T> findOne()` 이 더 친화적.

## 4. 조인 — 묵시적 vs 명시적

> JPQL 은 묵시적 조인 (`m.team.name`) 을 허용하지만 위험.

### 4-1. 묵시적 조인

```sql
SELECT m.team.name FROM Member m WHERE m.age > 20
```

`m.team.name` 한 줄로 JOIN 이 자동 생성된다. 가독성은 좋지만:

1. **INNER JOIN 강제** — LEFT JOIN 필요해도 INNER 만.
2. **JOIN 조건 추가 불가** — `ON team.active = true` 같은 추가 조건 불가능.

### 4-2. 명시적 조인

```sql
SELECT m FROM Member m JOIN m.team t WHERE t.active = true
SELECT m FROM Member m LEFT JOIN m.team t
```

권장. 조인 종류·조건을 명시적으로.

## 5. 검색 — 페치 조인

> `JOIN FETCH` 로 연관 Entity 도 함께 조회 (`./03-03.프록시와 N+1.md`).

```sql
SELECT m FROM Member m JOIN FETCH m.team
SELECT t FROM Team t JOIN FETCH t.members      -- 컬렉션 fetch
```

## 6. 페이징

> `setFirstResult`, `setMaxResults`.

```java
List<Member> result = em.createQuery("SELECT m FROM Member m", Member.class)
    .setFirstResult(10)
    .setMaxResults(20)
    .getResultList();
```

DB 종류에 무관 — JPA 가 알아서 LIMIT/OFFSET, ROWNUM, TOP 등으로 변환.

## 7. CASE 식

> SQL 의 CASE 와 동일.

```sql
-- 기본 CASE
SELECT
    CASE WHEN m.age < 20 THEN '미성년'
         WHEN m.age < 60 THEN '성인'
         ELSE '시니어'
    END
FROM Member m

-- 단순 CASE
SELECT
    CASE m.status
        WHEN 'ACTIVE' THEN 1
        WHEN 'INACTIVE' THEN 0
        ELSE -1
    END
FROM Member m
```

`COALESCE`, `NULLIF` 도 JPQL 표준.

## 8. JPQL 의 한계와 대안

> 동적 쿼리·복잡한 조건이 잦으면 JPQL 문자열 조작이 끔찍해진다.

```java
StringBuilder jpql = new StringBuilder("SELECT m FROM Member m WHERE 1=1");
List<Object> params = new ArrayList<>();
if (name != null) { jpql.append(" AND m.name = ?"); params.add(name); }
if (minAge != null) { jpql.append(" AND m.age >= ?"); params.add(minAge); }
// 안티패턴
```

대안: **QueryDSL** — 타입 안전, 컴파일 타임 검증, 동적 쿼리 자연스러움.

```java
// QueryDSL
queryFactory.selectFrom(member)
    .where(
        name != null ? member.name.eq(name) : null,
        minAge != null ? member.age.goe(minAge) : null
    )
    .fetch();
```

자세한 QueryDSL 은 `../querydsl/README.md`.
