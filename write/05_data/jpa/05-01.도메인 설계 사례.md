# 도메인 설계 사례 — 주문 도메인 분석과 구현
---
> 주문 도메인은 JPA 학습의 표준 사례 — 회원·주문·주문상품·배송·카테고리·아이템의 여섯 Entity 가 연관관계, 값 타입, CASCADE, 상속을 모두 사용한다.

## 1. 도메인 요구사항

> 가벼운 쇼핑몰 — 회원이 상품을 주문하고 배송 받는다.

기능 목록:

1. 회원 가입·조회.
2. 상품 등록·수정.
3. 주문·주문 취소·주문 내역 조회.

## 2. 도메인 모델

> 여섯 Entity 와 그 관계.

```
[Member] ─1:N─> [Order] ─1:N─> [OrderItem] ─N:1─> [Item (abstract)]
                  │                                  ├─ Book
                  └1:1─> [Delivery]                  ├─ Album
                                                     └─ Movie
[Category] ─N:N─> [Item]  (중간 Entity 로 풀어냄)
```

| Entity | 책임 |
|--------|------|
| Member | 회원 정보, Address 임베디드 |
| Order | 주문, Member·OrderItem·Delivery 와 연관 |
| OrderItem | 주문 한 줄, Order·Item 과 연관, 수량·가격 |
| Item | 상품 (추상) — Book/Album/Movie 상속 |
| Delivery | 배송 정보, Address 임베디드, 상태 |
| Category | 카테고리, Item 과 N:N (중간 Entity 로 풀어냄) |

## 3. Entity 구현 — 연관관계와 편의 메서드

```java
@Entity
@Table(name = "orders")   // 'order' 는 예약어
public class Order {

    @Id @GeneratedValue
    @Column(name = "order_id")
    private Long id;

    @ManyToOne(fetch = LAZY)
    @JoinColumn(name = "member_id")
    private Member member;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL)
    private List<OrderItem> orderItems = new ArrayList<>();

    @OneToOne(fetch = LAZY, cascade = CascadeType.ALL)
    @JoinColumn(name = "delivery_id")
    private Delivery delivery;

    private LocalDateTime orderDate;

    @Enumerated(EnumType.STRING)
    private OrderStatus status;   // ORDER, CANCEL

    // ===== 연관관계 편의 메서드 =====
    public void setMember(Member member) {
        this.member = member;
        member.getOrders().add(this);
    }

    public void addOrderItem(OrderItem orderItem) {
        this.orderItems.add(orderItem);
        orderItem.setOrder(this);
    }

    public void setDelivery(Delivery delivery) {
        this.delivery = delivery;
        delivery.setOrder(this);
    }

    // ===== 생성 메서드 =====
    public static Order create(Member member, Delivery delivery, OrderItem... orderItems) {
        Order order = new Order();
        order.setMember(member);
        order.setDelivery(delivery);
        for (OrderItem oi : orderItems) {
            order.addOrderItem(oi);
        }
        order.status = OrderStatus.ORDER;
        order.orderDate = LocalDateTime.now();
        return order;
    }

    // ===== 비즈니스 로직 =====
    public void cancel() {
        if (delivery.getStatus() == DeliveryStatus.COMPLETED) {
            throw new IllegalStateException("배송 완료된 주문은 취소 불가");
        }
        this.status = OrderStatus.CANCEL;
        for (OrderItem oi : orderItems) {
            oi.cancel();
        }
    }

    public int getTotalPrice() {
        return orderItems.stream().mapToInt(OrderItem::getTotalPrice).sum();
    }
}
```

### 3-1. 연관관계 편의 메서드

> 양쪽 동기화를 한 메서드로 — 실수 방지 (`./02-02.연관관계 매핑.md §3-2`).

```java
public void setMember(Member member) {
    this.member = member;
    member.getOrders().add(this);   // 양쪽 모두 설정
}
```

### 3-2. 생성 메서드

> 응집된 생성 책임 — 외부에서 setter 남발 막음.

`Order.create(...)` 정적 팩토리 메서드가 모든 필수 의존성을 강제. `new Order()` 후 개별 set 하는 코드 금지.

### 3-3. 비즈니스 로직을 Entity 안으로

> `cancel()`, `getTotalPrice()` 가 Order 안에 — Service 가 비대해지지 않게.

Service 가 Entity 의 setter 를 직접 호출하는 안티패턴 (Anemic Domain Model) 을 피한다. *비즈니스 규칙은 Entity 가 안다*.

## 4. 값 타입 (Address)

> Member 와 Delivery 가 모두 주소를 가짐 — Embeddable 로 공유.

```java
@Embeddable
public record Address(String city, String street, String zipcode) {}

@Entity
public class Member {
    @Embedded
    private Address address;
}

@Entity
public class Delivery {
    @Embedded
    private Address address;
}
```

record 사용으로 자동 불변. 변경하려면 새 인스턴스: `member.setAddress(new Address("Busan", ...))`.

## 5. CASCADE 와 고아 객체

> Order ─> OrderItem 의 생명주기 통합.

```java
@OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
private List<OrderItem> orderItems = new ArrayList<>();
```

효과:

1. `Order` 영속화 시 `OrderItem` 도 자동 persist.
2. `Order` 삭제 시 `OrderItem` 도 자동 delete.
3. `orderItems.remove(oi)` 만 해도 자동 delete.

조건: OrderItem 은 *Order 의 일부* — 다른 부모를 가질 수 없음. 이게 보장될 때만 CASCADE.ALL + orphanRemoval.

## 6. 테이블·컬럼명 생성 전략

> Spring Boot 의 기본 — 카멜 → 스네이크 자동 변환.

```yaml
spring:
  jpa:
    hibernate:
      naming:
        physical-strategy: org.hibernate.boot.model.naming.CamelCaseToUnderscoresNamingStrategy
```

`orderItems` → `order_items`, `memberAddress` → `member_address` 자동. 명시적 `@Column(name = ...)` 은 예외 케이스에만.

## 7. 상속 매핑 — Item

```java
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(name = "dtype")
public abstract class Item {
    @Id @GeneratedValue
    private Long id;
    private String name;
    private int price;
    private int stockQuantity;
}

@Entity @DiscriminatorValue("B")
public class Book extends Item { private String author; private String isbn; }

@Entity @DiscriminatorValue("A")
public class Album extends Item { private String artist; private String etc; }

@Entity @DiscriminatorValue("M")
public class Movie extends Item { private String director; private String actor; }
```

SINGLE_TABLE 선택: 자식 컬럼 차이 크지 않고 조회 성능 우선. 자세한 상속 매핑은 `./02-03.상속과 값 타입.md`.
