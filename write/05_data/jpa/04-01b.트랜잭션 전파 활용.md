# 트랜잭션 전파 활용 — 회원과 로그 시나리오

---

> 회원 가입과 이력 로그를 묶을 때 *어디까지 같이 롤백할지* 가 트랜잭션 전파의 실제 문제다. 본 노트는 강의 11장의 *복구 REQUIRED 실패 → REQUIRES_NEW 해결* 흐름을 단계별로 정리한다.

## 1. 도메인 — 회원 가입과 이력 로그

> 두 리포지토리가 한 서비스에서 함께 호출된다. 정합성 요구가 절의 출발점이다.

회원 서비스의 가입 메서드는 두 가지를 한다.

1. `MemberRepository.save(member)` — 회원 정보 저장.
2. `LogRepository.save(logMessage)` — 회원 가입 이력 로그 저장.

```java
@Service
public class MemberService {

    @Transactional
    public void joinV1(String username) {
        Member member = new Member(username);
        Log logMessage = new Log(username);

        memberRepository.save(member);
        logRepository.save(logMessage);  // 여기서 실패하면?
    }
}
```

요구사항이 두 갈래로 갈린다.

- **버전 A**: 로그 저장이 실패하면 회원 가입도 같이 롤백한다 (정합성 우선).
- **버전 B**: 로그 저장이 실패해도 회원 가입은 유지한다 (가용성 우선).

A 는 `REQUIRED` 전파로 자연스럽게 풀린다. B 는 단순해 보이지만 *실패의 함정* 이 있다. 본 노트의 주제는 B 다.

## 2. 단일 트랜잭션으로 묶기 — REQUIRED

> 세 곳 모두 `@Transactional` 을 두면 한 물리 트랜잭션에 합쳐진다.

```java
@Service
public class MemberService {
    @Transactional
    public void joinV1(String username) { ... }
}

@Repository
public class MemberRepository {
    @Transactional
    public void save(Member member) { ... }
}

@Repository
public class LogRepository {
    @Transactional
    public void save(Log logMessage) { ... }
}
```

스프링의 기본 전파 옵션은 `REQUIRED` 다. 동작은 다음과 같다.

- 외부 트랜잭션이 있으면 *기존 트랜잭션에 참여* 한다 (논리 트랜잭션).
- 외부 트랜잭션이 없으면 *새로 생성* 한다 (물리 트랜잭션).

`MemberService.joinV1()` 진입 시 물리 트랜잭션이 시작된다. `MemberRepository.save()` 와 `LogRepository.save()` 는 *논리 트랜잭션* 으로 참여한다. 셋이 한 커넥션, 한 물리 트랜잭션을 공유한다.

### 2-1. 정상 케이스

세 곳 모두 정상이면 외부 트랜잭션 (`joinV1`) 의 커밋 시점에 물리 커밋이 일어난다. 두 리포지토리는 신규 트랜잭션이 아니어서 *커밋을 호출하지 않는다*.

### 2-2. 로그 저장 실패 케이스

`LogRepository.save()` 가 런타임 예외를 던지면 흐름은 다음과 같다.

1. `LogRepository` 트랜잭션 AOP 가 예외를 받는다.
2. 신규 트랜잭션이 아니므로 *물리 롤백을 호출하지 않는다*. 대신 트랜잭션 동기화 매니저에 `rollbackOnly` 를 표시한다.
3. 예외가 `MemberService` 로 전파된다.
4. `MemberService` 도 예외를 그대로 던진다.
5. `MemberService` 트랜잭션 AOP 는 신규 트랜잭션이므로 *물리 롤백을 호출한다*.

결과적으로 회원과 로그 모두 롤백된다. *버전 A 요구사항* 이 충족된다.

## 3. 복구 REQUIRED — 흔한 실패

> 로그 실패를 잡아서 정상 흐름으로 바꾸면 *전체 커밋* 이 될까? 답은 아니다.

버전 B 를 위해 `MemberService` 에서 로그 예외를 잡는다.

```java
@Transactional
public void joinV2(String username) {
    Member member = new Member(username);
    Log logMessage = new Log(username);

    memberRepository.save(member);
    try {
        logRepository.save(logMessage);
    } catch (RuntimeException e) {
        log.info("log 저장에 실패했습니다. logMessage={}", logMessage);
        log.info("정상 흐름 변환");
    }
}
```

직관적으로는 *예외를 잡았으니 커밋된다* 고 기대한다. 결과는 다르다. `UnexpectedRollbackException` 이 던져진다.

### 3-1. 왜 실패하는가

`LogRepository.save()` 가 예외를 던지는 순간 *내부 트랜잭션 AOP 가 먼저 동작* 한다.

1. `LogRepository` 트랜잭션 AOP 가 예외를 받는다.
2. 신규 트랜잭션이 아니므로 물리 롤백은 안 한다. *하지만* 트랜잭션 동기화 매니저에 `rollbackOnly = true` 를 *이미 표시* 한다.
3. 예외가 `MemberService` 로 던져진다.
4. `MemberService` 가 예외를 잡아 정상 흐름으로 만든다.
5. `MemberService` 트랜잭션 AOP 는 정상 응답이므로 *커밋을 호출* 한다.
6. 커밋 시점에 트랜잭션 매니저가 `rollbackOnly` 를 체크한다. 표시되어 있다.
7. 물리 트랜잭션을 *롤백* 하고, `UnexpectedRollbackException` 을 던진다.

핵심은 *예외를 잡았어도 `rollbackOnly` 표시는 지워지지 않는다* 는 점이다. 내부 트랜잭션의 의사가 외부 트랜잭션을 통해 강제 롤백을 일으킨다.

```
[MemberService.joinV2]
  ├── memberRepository.save()  → 논리 참여, OK
  ├── logRepository.save()  → 논리 참여
  │     └── 예외 → rollbackOnly = true 표시 후 throw
  ├── catch (RuntimeException) → 정상 흐름
  └── 외부 트랜잭션 커밋 시도
        └── rollbackOnly == true → 물리 롤백
        └── UnexpectedRollbackException 던짐
```

논리 트랜잭션 중 하나라도 롤백되면 *전체 물리 트랜잭션이 롤백된다*. 강의에서 "실무에서 많은 개발자가 이 방법을 사용해서 실패한다" 고 박은 자리다.

## 4. REQUIRES_NEW 로 분리

> 로그 저장만 *별도의 물리 트랜잭션* 으로 두면, 로그 롤백이 회원 가입에 영향을 주지 않는다.

`LogRepository.save()` 의 전파 옵션을 바꾼다.

```java
@Repository
public class LogRepository {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void save(Log logMessage) { ... }
}
```

`REQUIRES_NEW` 는 *항상 새로운 물리 트랜잭션* 을 생성한다. 별도의 DB 커넥션을 쓴다는 의미다.

### 4-1. 분리 후의 흐름

`MemberService.joinV2()` (예외 잡기 포함) 가 호출될 때 동작은 다음과 같다.

1. `MemberService` 가 외부 물리 트랜잭션 (커넥션 con1) 을 시작한다.
2. `MemberRepository.save()` 는 `REQUIRED` — con1 에 참여한다.
3. `LogRepository.save()` 는 `REQUIRES_NEW` — *새 커넥션 con2 와 새 물리 트랜잭션* 을 시작한다.
4. `LogRepository` 에서 예외 발생 → con2 의 *물리 트랜잭션이 즉시 롤백* 된다. con2 는 그대로 끝난다.
5. 예외가 `MemberService` 로 전파된다.
6. `MemberService` 가 잡아서 정상 흐름으로 변환한다.
7. `MemberService` 트랜잭션 AOP 가 커밋을 호출한다. con1 에 `rollbackOnly` 가 없으므로 *물리 커밋* 이 일어난다.

결과적으로 회원 데이터는 저장되고 로그만 롤백된다. *버전 B 요구사항* 이 충족된다.

```
[con1: MemberService 외부 트랜잭션]
  ├── memberRepository.save(con1) → 참여, OK
  ├── logRepository.save() ──→ [con2: REQUIRES_NEW]
  │                                ├── 예외 발생 → con2 물리 롤백
  │                                └── con2 종료
  ├── catch → 정상 흐름
  └── con1 커밋 → 물리 커밋
```

### 4-2. 주의점 — 커넥션 두 개

`REQUIRES_NEW` 는 *DB 커넥션을 두 개* 점유한다. 외부 트랜잭션이 끝날 때까지 외부 커넥션은 계속 잡혀 있고, 그 안에서 새 커넥션이 추가로 할당된다.

이 점이 트래픽 많은 환경에서는 문제가 된다. 풀 크기가 100 이라면, `REQUIRES_NEW` 가 흔한 코드 패턴이 될 때 사실상 50 동시 요청에서 한계에 닿는다. 풀 사이즈 산정 시 이 비용을 반드시 잡아야 한다.

대안 두 가지가 있다.

- **메시지 큐로 분리**: 로그는 트랜잭션 밖에서 메시지로 발행하고, 별도 컨슈머가 저장한다. 메인 트랜잭션과 완전히 분리되며 커넥션도 한 개로 끝난다.
- **다른 메서드로 호출 분리**: 외부 트랜잭션을 *나간 다음* 별도 메서드로 로그를 저장한다. AOP 자기 호출 함정을 피하려면 [`./04-01.스프링 트랜잭션.md`](./04-01.스프링%20트랜잭션.md) §AOP 의 자기 호출 절을 본다.

## 5. 정리

전파 활용의 두 갈래가 핵심이다.

- *함께 롤백* — `REQUIRED` 한 줄로 끝난다. 한 물리 트랜잭션 안에서 자연스럽게 묶인다.
- *부분 롤백 (로그는 실패해도 회원은 유지)* — `REQUIRES_NEW` 가 필요하다. 단순히 예외를 잡는 것만으론 `rollbackOnly` 표시 때문에 *외부도 같이 롤백* 된다.

`rollbackOnly` 표시는 *내부가 외부의 결정을 강제하는 메커니즘* 이다. 이 표시는 한번 박히면 *예외를 잡아도 지워지지 않는다*. 부분 롤백이 필요하면 *물리 트랜잭션 자체를 분리* 해야 한다.
