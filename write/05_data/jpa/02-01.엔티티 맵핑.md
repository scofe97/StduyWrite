# 엔티티 맵핑 (애너테이션 + 기본 키 전략)
---
> 객체와 테이블·컬럼을 잇는 첫 단계 — `@Entity`, `@Table`, `@Column`, `@Id` 의 다섯 애너테이션이 90% 의 매핑을 덮는다.

## 1. `@Entity` 와 `@Table`

> 클래스를 영속 객체로, 테이블 이름을 명시한다.

```java
@Entity
@Table(name = "members", indexes = {
    @Index(name = "idx_email", columnList = "email")
})
public class Member {
    @Id @GeneratedValue
    private Long id;
    // ...

    protected Member() {}   // JPA 가 리플렉션으로 호출 — public 또는 protected 필수
}
```

규칙:

1. `@Entity` 가 붙으면 기본 생성자 (no-args) 가 필수 — public 또는 protected.
2. final 클래스 금지 — Hibernate 가 프록시 생성을 위해 상속.
3. `@Table` 생략 시 클래스 이름이 테이블 이름.

## 2. `@Column` 과 기본 매핑

> 필드와 컬럼의 매핑 — 이름·null·길이·unique 등 세부 제어.

```java
@Column(name = "user_name", nullable = false, length = 50, unique = true)
private String userName;

@Column(precision = 10, scale = 2)
private BigDecimal price;
```

생략하면 카멜케이스 → 스네이크케이스 자동 변환 (Spring Boot 의 기본 NamingStrategy).

| 속성 | 의미 |
|------|------|
| `name` | 컬럼 이름 (생략 시 필드명 변환) |
| `nullable` | NULL 허용 (기본 true) |
| `length` | 문자 길이 (기본 255) |
| `unique` | UNIQUE 제약 |
| `insertable` | INSERT 시 포함 (기본 true) |
| `updatable` | UPDATE 시 포함 (기본 true) |

`insertable=false, updatable=false` 는 *읽기 전용 컬럼* (예: DB 자동 채움 컬럼) 에 사용.

## 3. 특수 타입 매핑

> 기본 타입 외의 매핑 — Enum·날짜·LOB·Transient.

| 애너테이션 | 용도 |
|-----------|------|
| `@Enumerated(EnumType.STRING)` | Enum 을 문자열로 (`ORDINAL` 절대 쓰지 말 것) |
| `@Temporal(TemporalType.DATE)` | `java.util.Date` 의 정밀도. Java 8+ `LocalDate` 는 불필요 |
| `@Lob` | BLOB/CLOB — 큰 바이너리/텍스트 |
| `@Transient` | DB 에 매핑하지 않음 (계산용 임시 필드) |

여기서 질문 하나 — `@Enumerated(EnumType.ORDINAL)` 이 왜 금지인가? Enum 의 *순서가 PK 처럼 저장* 된다. 나중에 Enum 값을 중간에 추가하면 옛 데이터가 전혀 다른 의미가 된다.

```java
// 위험
enum Status { ACTIVE, INACTIVE }  // ACTIVE=0, INACTIVE=1
// 나중에
enum Status { PENDING, ACTIVE, INACTIVE }  // 옛 0(ACTIVE) 이 이제 PENDING!
```

항상 `STRING` 으로.

## 4. 기본 키 매핑

> `@Id` + `@GeneratedValue` 의 조합 — 생성 전략 4가지.

```java
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
```

| 전략 | 동작 | 적합 DB |
|------|------|---------|
| `IDENTITY` | DB 의 AUTO_INCREMENT 위임 | MySQL, MariaDB |
| `SEQUENCE` | DB 의 Sequence 객체 사용 | PostgreSQL, Oracle |
| `TABLE` | 별도 키 테이블 + 행 카운터 | 호환성 최후 수단 (성능 나쁨) |
| `AUTO` | JPA 가 DB 보고 결정 | 권장 (DB 이식성) |

### 4-1. IDENTITY 의 성능 특성

> IDENTITY 는 *INSERT 가 일어나야 ID 가 결정* 된다 — JPA 의 쓰기 지연이 작동 안 함.

```java
@Transactional
public void batch() {
    em.persist(new Member("a"));   // 즉시 INSERT — ID 필요하므로
    em.persist(new Member("b"));   // 즉시 INSERT
    em.persist(new Member("c"));   // 즉시 INSERT
}
```

대량 INSERT 가 잦으면 SEQUENCE 가 유리 (미리 ID 발급받아 batch).

### 4-2. SEQUENCE 의 ID 미리 발급

```java
@Entity
@SequenceGenerator(name = "member_seq", sequenceName = "member_sequence",
                   allocationSize = 50)
public class Member {
    @Id
    @GeneratedValue(strategy = SEQUENCE, generator = "member_seq")
    private Long id;
}
```

`allocationSize=50` 이면 한 번에 50개씩 받아 메모리에 캐시. INSERT 마다 DB 호출 없이 ID 발급.

## 5. 스키마 자동 생성 (`ddl-auto`)

> Hibernate 가 Entity 정의로 CREATE TABLE 을 자동 생성 — 개발용.

| 값 | 동작 |
|----|------|
| `none` | 아무것도 안 함 (운영 권장) |
| `validate` | 시작 시 Entity vs DB 스키마 검증, 다르면 에러 |
| `update` | Entity 에 새 컬럼이 있으면 ALTER TABLE 추가 (위험) |
| `create` | 시작 시 DROP + CREATE |
| `create-drop` | 시작 시 CREATE, 종료 시 DROP (테스트용) |

운영은 **항상 `none` 또는 `validate`** + Flyway/Liquibase 같은 마이그레이션 도구. `update` 는 의도치 않은 변경이 운영 DB 에 적용될 위험.
