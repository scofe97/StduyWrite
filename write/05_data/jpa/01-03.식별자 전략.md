# 식별자 전략 — 자연 키 vs 대리 키
---
> Entity 의 PK 를 비즈니스 의미가 있는 컬럼 (자연 키) 으로 할지, 의미 없는 인공 컬럼 (대리 키) 으로 할지의 결정 — 이 한 가지가 모델의 유연성·성능을 결정한다.

## 1. 자연 키 (Natural Key)

> 비즈니스 도메인에서 이미 유일한 값 — 주민번호·이메일·ISBN.

```java
@Entity
public class Book {
    @Id
    private String isbn;   // 비즈니스 키
    private String title;
}
```

장점:

1. 추가 컬럼 없음 — 저장 공간 절약.
2. 도메인 의미가 PK 에 박힘 — 디버깅 시 의미 보임.

단점:

1. **변경 가능성** — 이메일 변경, 회사 이전, 정책 변경. 변경 시 FK 도 다 따라 바뀌어야 함.
2. **노출 위험** — URL 에 노출되면 보안 (주민번호) 또는 정보 누출 (회사 내부 코드).
3. **결합 키** — 자연 키가 복합인 경우 모든 FK 도 복합 컬럼.

## 2. 대리 키 (Surrogate Key)

> 의미 없는 인공 컬럼 — AUTO_INCREMENT·UUID·Sequence.

```java
@Entity
public class Book {
    @Id @GeneratedValue(strategy = IDENTITY)
    private Long id;        // 대리 키
    @Column(unique = true)
    private String isbn;    // 비즈니스 키지만 PK 아님
    private String title;
}
```

장점:

1. **불변** — 비즈니스 의미와 분리, 변경 없음.
2. **단순** — 항상 단일 컬럼.
3. **노출 안전** — URL `/books/12345` 에 도메인 정보 없음.

단점:

1. 추가 컬럼.
2. 자연 키 UNIQUE 제약을 따로 둬야 함 — 빠뜨리면 중복 데이터.

## 3. 권장 — 거의 항상 대리 키

> 자연 키의 변경 위험이 너무 크다 — 대리 키 + UNIQUE 제약이 표준.

대리 키를 권장하는 결정적 근거:

1. **변경 비용** — 자연 키 변경 시 모든 참조 테이블의 FK 도 함께 변경. ORM·메모리 캐시·외부 시스템까지 영향.
2. **JPA 와의 궁합** — `@GeneratedValue` 가 자연스러움.
3. **모델 단순화** — 모든 Entity 의 PK 가 같은 타입 (`Long` 또는 `UUID`) 이면 일관성.

자연 키가 정말 불변임이 보장될 때만 (예: 표준 코드 `ISO 4217` 통화 코드) 자연 키 고려.

## 4. ID 자체도 Value Object

> `Long` 직접 노출하지 말고 의미 있는 ID 타입으로 — 도메인 어휘를 타입 시스템에 박는다.

```java
public record MemberId(Long value) {}
public record OrderId(Long value) {}

// 나쁨
void grantReward(Long memberId, Long orderId);   // 순서 바꿔도 컴파일 통과

// 좋음
void grantReward(MemberId memberId, OrderId orderId);   // 순서 틀리면 컴파일 에러
```

JPA `@IdClass` 또는 `@EmbeddedId` 로 구현. record 와 결합하면 가장 깔끔.

## 5. UUID vs AUTO_INCREMENT

> 대리 키 안에서도 선택 — 단일 DB 면 AUTO_INCREMENT, 분산이면 UUID.

| 측면 | AUTO_INCREMENT | UUID |
|------|----------------|------|
| 크기 | 8바이트 (BIGINT) | 16바이트 |
| 생성 위치 | DB | 애플리케이션 또는 DB |
| 순차성 | 시간순 (인덱스 친화) | 무작위 (인덱스 분산) |
| 분산 환경 | DB 의존 (충돌 위험) | 충돌 없음 |
| URL 노출 | 다음 ID 추측 가능 (보안 위험) | 추측 불가 |

UUID v7 (시간순 정렬되는 UUID) 가 두 장점을 합치는 최근 선택. PostgreSQL 17+ 기본 지원.
