# 연관관계 매핑 — 단방향, 양방향, 다양한 관계
---
> 연관관계는 JPA 가 가장 강력하지만 가장 자주 실수하는 영역이다. 객체 그래프와 외래키 그래프의 *방향성 불일치* 가 모든 함정의 출처다.

## 1. 객체와 테이블의 차이

> 테이블은 외래키 하나로 양쪽 조회가 가능, 객체는 참조를 가진 쪽에서만 조회 가능.

```sql
CREATE TABLE member (id INT PRIMARY KEY, team_id INT);
CREATE TABLE team (id INT PRIMARY KEY, name VARCHAR(50));
```

이 외래키 `member.team_id` 하나로:
- `SELECT * FROM member WHERE team_id = ?` (팀의 멤버 조회)
- `SELECT * FROM team WHERE id = ?` (멤버의 팀 조회)

둘 다 가능. 그러나 객체는:

```java
class Member { Team team; }   // member.getTeam() 가능
class Team   { /* members 없음 */ }   // team.getMembers() 불가능
```

객체 참조는 *한 방향* 만 가능. 양방향 조회를 원하면 양쪽에 참조를 둬야 함.

## 2. 단방향 연관관계

> 한쪽만 참조를 가짐 — 가장 단순하고 안전.

```java
@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;

    @ManyToOne(fetch = LAZY)
    @JoinColumn(name = "team_id")
    private Team team;
}

@Entity
public class Team {
    @Id @GeneratedValue
    private Long id;
    private String name;
    // members 필드 없음
}
```

| 애너테이션 | 의미 |
|-----------|------|
| `@ManyToOne` | 다 (Member) → 일 (Team) |
| `@JoinColumn(name="team_id")` | 외래키 컬럼 이름 |
| `fetch = LAZY` | 지연 로딩 (권장 — `./03-03.프록시와 N+1.md`) |

여기서 질문 하나 — `@JoinColumn` 을 생략하면? 기본 이름 (`team_id`) 으로 자동 생성. 의도가 분명할 땐 생략 가능하지만 명시가 권장.

## 3. 양방향 연관관계

> 양쪽 모두 참조 — 객체 그래프 탐색이 자유롭지만 *주인* 개념이 필요.

```java
@Entity
public class Member {
    @ManyToOne(fetch = LAZY)
    @JoinColumn(name = "team_id")
    private Team team;     // 외래키를 가진 쪽 — 주인
}

@Entity
public class Team {
    @OneToMany(mappedBy = "team", fetch = LAZY)
    private List<Member> members = new ArrayList<>();   // 주인의 반대편
}
```

### 3-1. 양방향 연관관계의 주인

> 외래키가 있는 쪽이 주인 — 주인만이 외래키 INSERT/UPDATE 권한을 가진다.

| 측면 | 주인 (Member.team) | 반대편 (Team.members) |
|------|---------------------|------------------------|
| 외래키 관리 | O | X (`mappedBy` 로 위임) |
| `@JoinColumn` | O | X |
| `mappedBy` | X | O |

규칙은 단순하지만 자주 어긴다. 반대편에 값을 넣고 *주인은 빈 채로 두면* DB 에 외래키가 안 들어간다.

```java
// 잘못
team.getMembers().add(member);    // 반대편만 추가
em.persist(member);                // member.team_id 는 NULL!

// 올바름
member.setTeam(team);              // 주인에도 설정
team.getMembers().add(member);     // 반대편도 일관성 위해
em.persist(member);
```

### 3-2. 연관관계 편의 메서드

> 양쪽 동기화를 한 메서드로 — 실수 방지.

```java
public class Member {
    public void setTeam(Team team) {
        this.team = team;
        team.getMembers().add(this);   // 반대편도 자동 추가
    }
}
```

`setTeam` 만 호출하면 양쪽 동기화 끝. 단, 이전 팀에서 제거하는 로직도 필요할 수 있음.

## 4. 다양한 연관관계

> 다중성·방향성의 네 가지 조합.

| 다중성 | 단방향 | 양방향 |
|--------|--------|--------|
| N:1 (`@ManyToOne`) | 가장 흔함 | + `@OneToMany(mappedBy)` |
| 1:N (`@OneToMany`) | 비추 — 외래키가 자식에 있어 INSERT/UPDATE 분리 | 위와 동일 |
| 1:1 (`@OneToOne`) | 외래키 어느 쪽이든 가능 | + 반대편에 `@OneToOne(mappedBy)` |
| N:N (`@ManyToMany`) | 비추 — 중간 테이블 제어 불가, 항상 `@OneToMany` 두 개로 풀 것 | 비추 |

### 4-1. `@ManyToOne` 단방향이 표준

> 외래키가 자식에 있는 N:1 단방향이 가장 흔하고 가장 안전.

```java
class Order { @ManyToOne Customer customer; }   // ✅
```

양방향이 필요해질 때 추가. 처음부터 양방향으로 만들면 *주인 혼동* 의 함정.

### 4-2. `@ManyToMany` 절대 직접 쓰지 말 것

```java
// 안티패턴
class Member { @ManyToMany List<Role> roles; }
class Role   { @ManyToMany List<Member> members; }
```

문제:

1. 중간 테이블 (`member_role`) 의 *추가 컬럼* (가입일 등) 을 표현 못 함.
2. 두 객체의 상태 변화가 중간 테이블의 INSERT/DELETE 와 *비동기적*.
3. 디버깅 어려움.

해결:

```java
class Member { @OneToMany(mappedBy="member") List<MemberRole> memberRoles; }
class Role   { @OneToMany(mappedBy="role") List<MemberRole> memberRoles; }
class MemberRole {
    @ManyToOne Member member;
    @ManyToOne Role role;
    private LocalDateTime joinedAt;
}
```

중간 Entity 를 명시 — 모든 N:N 은 이 형태로.

## 5. 권장 — 단순한 단방향에서 시작

> 처음부터 양방향이나 복잡한 다중성을 선언하지 않는다. 필요해질 때 추가.

JPA 의 흔한 실수는 *처음부터 모든 관계를 양방향으로 박는 것*. 결과는:

1. 주인 혼동.
2. JSON 직렬화 시 무한 루프 (양쪽 참조).
3. 변경 영향 폭증.

원칙: **단방향이 우선, 양방향은 정말 필요할 때**. 거의 모든 경우 자식 → 부모의 `@ManyToOne` 단방향으로 충분.
