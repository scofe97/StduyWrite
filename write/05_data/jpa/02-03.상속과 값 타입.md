# 상속 매핑과 값 타입
---
> 객체의 상속과 값 타입을 테이블 모델로 옮기는 두 패턴 — 모두 잘못 쓰면 N+1·동등성·공유참조 함정에 빠진다.

## 1. 상속 매핑의 세 전략

> 같은 객체 상속을 셋 중 어느 방식으로 테이블에 옮길지의 결정.

```java
@Entity
@Inheritance(strategy = InheritanceType.JOINED)
@DiscriminatorColumn(name = "dtype")
public abstract class Item {
    @Id @GeneratedValue
    private Long id;
    private String name;
    private int price;
}

@Entity @DiscriminatorValue("ALBUM")
public class Album extends Item { private String artist; }

@Entity @DiscriminatorValue("BOOK")
public class Book extends Item { private String author; }
```

### 1-1. JOINED — 정규화 우선

```
[item]                 [album]              [book]
id, name, price, dtype id, artist           id, author
```

자식 테이블이 부모 테이블의 PK 로 JOIN.

장점:
1. 정규화. NULL 컬럼 없음.
2. 자식별 컬럼 명확.

단점:
1. 조회 시 JOIN 필요 — 느림.
2. INSERT 두 번 (item + album).

### 1-2. SINGLE_TABLE — 조회 성능 우선 (기본)

```
[item]
id, name, price, dtype, artist, book_author  -- 자식 컬럼이 모두 한 테이블
```

장점:
1. JOIN 없음 — 빠른 조회.
2. INSERT 한 번.

단점:
1. 자식 고유 컬럼이 다른 자식 행에서는 NULL.
2. 테이블 비대.
3. 자식 컬럼에 `NOT NULL` 못 검.

### 1-3. TABLE_PER_CLASS — 거의 안 씀

```
[album]                [book]
id, name, price, artist id, name, price, author  -- 부모 컬럼 중복
```

부모 테이블이 없고 자식마다 모든 컬럼. JOIN 도 UNION 도 필요해 비효율.

### 1-4. 선택 기준

| 상황 | 권장 |
|------|------|
| 단순·소규모 | SINGLE_TABLE (기본) |
| 자식별 데이터 차이가 크고 조회 성능 덜 중요 | JOINED |
| 자식 종류 변할 가능성 있음 | JOINED (확장 유연) |

## 2. 값 타입 (Value Object)

> Entity 가 아닌 *값 그 자체로 식별되는* 객체 — `Address`, `Money` 같은 것.

### 2-1. 임베디드 타입 (`@Embeddable`)

```java
@Embeddable
public class Address {
    private String city;
    private String street;
    private String zipcode;

    protected Address() {}
    public Address(String city, String street, String zipcode) { ... }
}

@Entity
public class Member {
    @Embedded
    private Address address;
}
```

DB 에는 컬럼이 펼쳐져 저장:

```
[member]
id, name, address_city, address_street, address_zipcode
```

장점:

1. 객체 응집 — Address 의 동작 (`getFullAddress()`) 을 한 클래스에.
2. 재사용 — 여러 Entity 에 같은 Address 임베디드.

### 2-2. `@AttributeOverride` — 컬럼 이름 재정의

```java
@Entity
public class Order {
    @Embedded
    private Address shippingAddress;

    @Embedded
    @AttributeOverrides({
        @AttributeOverride(name = "city",    column = @Column(name = "billing_city")),
        @AttributeOverride(name = "street",  column = @Column(name = "billing_street")),
        @AttributeOverride(name = "zipcode", column = @Column(name = "billing_zipcode"))
    })
    private Address billingAddress;
}
```

한 Entity 에 같은 Embeddable 을 두 번 쓸 때 컬럼 충돌 방지.

## 3. 값 타입 공유 참조 문제

> 값 타입은 *불변* 으로 다뤄야 한다 — 공유 참조의 변경이 의도 밖 부작용을 만든다.

```java
Address addr = new Address("Seoul", "...", "...");
member1.setAddress(addr);
member2.setAddress(addr);   // 같은 객체 공유

addr.setCity("Busan");      // member1, member2 모두 Busan 으로!
```

JPA 가 두 Entity 의 변경을 모두 감지해 UPDATE 두 번. 의도와 다른 결과.

### 3-1. 해결법

| 방법 | 설명 |
|------|------|
| 값 복사 | `member2.setAddress(new Address(addr.getCity(), addr.getStreet(), addr.getZipcode()))` |
| 불변 객체 | setter 제거, 모든 필드 final |
| `record` 사용 (Java 14+) | 자동으로 불변 |

권장은 **불변 객체** 또는 **record**. setter 가 없으면 공유해도 안전.

```java
public record Address(String city, String street, String zipcode) {}

member1.setAddress(addr);
member2.setAddress(addr);
// 둘 다 같은 객체 가리켜도 변경 불가 — 안전
// 변경하려면 새 인스턴스: new Address("Busan", ...)
```

## 4. 값 타입 컬렉션 (`@ElementCollection`)

> 값 타입의 컬렉션 — 별도 테이블로 풀려나간다.

```java
@Entity
public class Member {
    @ElementCollection
    @CollectionTable(
        name = "member_favorites",
        joinColumns = @JoinColumn(name = "member_id")
    )
    private Set<String> favoriteFoods = new HashSet<>();
}
```

```
[member_favorites]
member_id, favorite_food   -- PK = (member_id, favorite_food)
```

### 4-1. 단점

1. **변경 시 전체 삭제 후 재삽입** — Hibernate 의 흔한 구현.
2. **추적 어려움** — 식별자 없음.

규모가 커지면 **별도 Entity 로 승격** 이 답:

```java
// 더 나은 방법
@Entity
public class FavoriteFood {
    @Id @GeneratedValue
    private Long id;
    @ManyToOne
    private Member member;
    private String name;
}
```
