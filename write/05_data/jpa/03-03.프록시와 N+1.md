# 프록시와 N+1 문제 (즉시/지연 로딩, CASCADE, 고아 객체)
---
> JPA 의 프록시 메커니즘이 지연 로딩을 가능하게 만든다. 그러나 같은 메커니즘이 N+1 문제의 출처가 된다 — JPA 가 자동으로 만들어내는 SQL 의 *개수* 를 통제하지 못하면 운영에서 부하가 폭발한다.

## 1. 프록시 — 지연 로딩의 토대

> JPA 가 Entity 의 *대리 객체* 를 만들어 진짜 데이터 접근을 미룬다.

```java
Member m = em.getReference(Member.class, 1L);   // SELECT 안 함 — 프록시 반환
m.getName();   // 이 시점에 SELECT 실행
```

프록시의 특징:

1. **상속 기반** — Hibernate 가 런타임에 Member 의 서브클래스 생성.
2. **첫 메서드 호출 시 초기화** — DB 에서 실제 데이터 로딩.
3. **`getId()` 는 초기화 안 함** — id 는 프록시 생성 시점에 이미 알고 있음.

여기서 질문 하나 — `em.find()` 와 `em.getReference()` 의 차이? `find` 는 즉시 SELECT, `getReference` 는 프록시만. 다음에 필드 접근이 *확실히* 일어날 거면 `find`, *연관관계 설정용* 으로만 쓰일 거면 `getReference`.

```java
// 단순 FK 연결 — getReference 가 적절
Order order = new Order();
order.setMember(em.getReference(Member.class, memberId));   // SELECT 안 감
em.persist(order);
```

## 2. 즉시 로딩 (`EAGER`) vs 지연 로딩 (`LAZY`)

> 연관 Entity 를 *언제 가져올 것인가* 의 결정 — 거의 항상 LAZY.

```java
@Entity
public class Order {
    @ManyToOne(fetch = LAZY)
    private Member member;        // 권장

    @OneToMany(mappedBy = "order")   // 기본 LAZY
    private List<OrderItem> items;
}
```

### 2-1. 기본값

| 관계 | 기본 |
|------|------|
| `@ManyToOne` | EAGER (위험) |
| `@OneToOne` | EAGER (위험) |
| `@OneToMany` | LAZY |
| `@ManyToMany` | LAZY |

**모든 연관관계에 명시적으로 `fetch = LAZY` 를 박을 것**. 기본값 EAGER 는 의도치 않은 SQL 폭발의 출처.

### 2-2. 왜 EAGER 가 위험한가

```java
List<Order> orders = orderRepository.findAll();   // SELECT order
for (Order o : orders) {
    // EAGER 면 매 order 마다 SELECT member — N+1
}
```

JPQL `findAll()` 이 *연관 EAGER 까지 고려 못 함* — 각 Order 마다 추가 SELECT. 이게 N+1 의 첫 패턴.

## 3. N+1 문제

> 1개의 메인 쿼리 + N개의 추가 쿼리 — JPA 의 가장 흔한 성능 함정.

### 3-1. 발생 패턴

```java
List<Member> members = memberRepository.findAll();   // 1: SELECT member
for (Member m : members) {
    System.out.println(m.getTeam().getName());        // N: SELECT team WHERE id = ? (매번)
}
```

LAZY 라도 *접근 시점에* 매번 SELECT. 100명이면 1 + 100 = 101 쿼리.

### 3-2. 해결방안 1 — Fetch Join (가장 흔함)

```java
@Query("SELECT m FROM Member m JOIN FETCH m.team")
List<Member> findAllWithTeam();
```

한 쿼리로 Member + Team 모두. N+1 사라짐.

### 3-3. 해결방안 2 — `@EntityGraph`

```java
@EntityGraph(attributePaths = {"team"})
@Override
List<Member> findAll();
```

JPQL 안 쓰고 애너테이션으로. 같은 효과.

### 3-4. 해결방안 3 — `@BatchSize`

```java
@Entity
public class Team {
    @OneToMany(mappedBy = "team")
    @BatchSize(size = 100)
    private List<Member> members;
}
```

지연 로딩되지만 *접근 시 IN 절로 한 번에* 100개씩. 1 + N → 1 + (N/100) 으로 줄어듦.

### 3-5. 해결방안 4 — 서브쿼리

```java
@Query("SELECT m FROM Member m WHERE m IN " +
       "(SELECT m2 FROM Member m2 JOIN m2.team t WHERE t.active = true)")
List<Member> findActive();
```

복잡한 조건과 결합할 때.

### 3-6. 어떤 방법을 쓸까

| 상황 | 권장 |
|------|------|
| 단순 N:1 fetch | Fetch Join |
| 1:N 컬렉션 fetch + 페이징 | `@BatchSize` (Fetch Join + Page 는 위험) |
| 여러 연관 fetch | `@EntityGraph` |
| 동적 조건 fetch | QueryDSL 또는 Criteria |

## 4. CASCADE — 영속성 전이

> 부모 Entity 의 영속화·삭제가 자식에게도 전파.

```java
@Entity
public class Parent {
    @OneToMany(mappedBy = "parent", cascade = CascadeType.PERSIST)
    private List<Child> children = new ArrayList<>();
}
```

| CascadeType | 의미 |
|-------------|------|
| `PERSIST` | persist 전파 |
| `REMOVE` | remove 전파 |
| `MERGE` | merge 전파 |
| `ALL` | 모두 |

```java
Parent p = new Parent();
p.getChildren().add(new Child());
em.persist(p);   // CASCADE.PERSIST 면 Child 도 자동 persist
```

### 4-1. CASCADE 의 위험

`CascadeType.ALL` 은 편하지만 *부모 remove 시 자식 모두 삭제*. 의도가 분명한 경우에만:

1. 부모-자식이 *단독 소유* 관계 (게시글-댓글).
2. 자식이 다른 부모를 가질 수 없음.

여러 부모가 같은 자식을 참조하는 경우 CASCADE 금지.

## 5. 고아 객체 (`orphanRemoval = true`)

> 자식 컬렉션에서 제거하면 자동 DELETE.

```java
@OneToMany(mappedBy = "parent", orphanRemoval = true)
private List<Child> children = new ArrayList<>();

parent.getChildren().remove(0);   // 첫 자식이 컬렉션에서 빠짐 → 자동 DELETE
```

### 5-1. CASCADE.ALL + orphanRemoval = 생명주기 통합

```java
@OneToMany(mappedBy = "parent", cascade = CascadeType.ALL, orphanRemoval = true)
private List<Child> children = new ArrayList<>();
```

이 조합이 의미하는 것: *Child 가 Parent 의 일부* 라는 선언. Parent 생명주기 안에서만 Child 가 존재.

DDD 의 Aggregate Root 패턴과 자연스러운 매핑. 단, 다른 부모가 Child 를 참조할 가능성이 0% 일 때만.
