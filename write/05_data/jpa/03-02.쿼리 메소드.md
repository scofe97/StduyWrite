# 쿼리 메소드 (메서드 이름, @Query, 페이징, fetch join, Lock)
---
> Spring Data JPA 의 쿼리는 세 방식 — 메서드 이름 기반·`@Query`·NamedQuery. 각각의 적용 범위가 다르고, 그 위에 페이징·fetch join·Lock 같은 기능이 얹힌다.

## 1. 메서드 이름 기반 쿼리

> 메서드 이름의 키워드를 Spring Data 가 파싱해 쿼리 자동 생성.

```java
public interface MemberRepository extends JpaRepository<Member, Long> {
    List<Member> findByName(String name);
    List<Member> findByNameAndAge(String name, int age);
    List<Member> findByAgeGreaterThan(int age);
    Optional<Member> findFirstByEmailOrderByCreatedAtDesc(String email);
}
```

### 1-1. 핵심 키워드

| 키워드 | 의미 |
|--------|------|
| `findBy`, `getBy`, `readBy`, `queryBy` | SELECT |
| `existsBy` | EXISTS (boolean 반환) |
| `countBy` | COUNT (long 반환) |
| `deleteBy`, `removeBy` | DELETE |
| `findFirst<N>`, `findTop<N>` | LIMIT N |
| `And`, `Or` | WHERE 조건 |
| `Is`, `Equals` | = |
| `IsNot`, `Not` | != |
| `IsNull`, `IsNotNull` | IS NULL |
| `IsTrue`, `IsFalse` | boolean |
| `Like`, `NotLike`, `StartingWith`, `EndingWith`, `Containing` | LIKE 패턴 |
| `LessThan`, `GreaterThan`, `Between` | 비교 |
| `In`, `NotIn` | IN |
| `OrderBy<Field>Asc/Desc` | 정렬 |
| `IgnoreCase` | 대소문자 무관 |

### 1-2. 한계

메서드 이름이 길어지면 가독성 폭락. 조건 3개 이상이면 `@Query` 로 전환.

```java
// 가독성 한계
List<Member> findByNameAndAgeGreaterThanAndStatusInOrderByCreatedAtDesc(...);
```

## 2. `@Query` — JPQL 직접 작성

> 메서드 이름으로 표현하기 복잡한 쿼리는 JPQL 을 직접.

```java
@Query("SELECT m FROM Member m WHERE m.name = :name AND m.age > :age")
List<Member> findActiveMembers(@Param("name") String name, @Param("age") int age);
```

복잡한 JOIN, 서브쿼리, GROUP BY 도 가능.

### 2-1. DTO 반환

```java
@Query("SELECT new com.example.MemberDto(m.id, m.name, t.name) " +
       "FROM Member m JOIN m.team t WHERE m.age > :age")
List<MemberDto> findMembersWithTeamName(@Param("age") int age);
```

`new` 키워드 + DTO 생성자 호출. SELECT 컬럼 수와 DTO 생성자 시그니처가 일치해야 함.

### 2-2. Native SQL

```java
@Query(value = "SELECT * FROM member WHERE name = :name", nativeQuery = true)
List<Member> findByNameNative(@Param("name") String name);
```

DB 특화 함수가 필요할 때. 단, JPQL 의 객체 매핑·이식성을 잃음.

## 3. NamedQuery (`@NamedQuery`)

> Entity 에 쿼리를 박아두는 방식 — 거의 안 씀.

```java
@Entity
@NamedQuery(name = "Member.findByName",
            query = "SELECT m FROM Member m WHERE m.name = :name")
public class Member { ... }

// Repository
List<Member> findByName(@Param("name") String name);   // 메서드 이름이 NamedQuery 이름과 같음
```

장점: 시작 시점에 쿼리 검증.
단점: Entity 가 무거워짐.

현실적으로 `@Query` 로 대체됨. 레거시에서만 발견.

## 4. 페이징과 정렬

> `Pageable` 파라미터로 페이징, `Sort` 로 정렬.

```java
public interface MemberRepository extends JpaRepository<Member, Long> {
    Page<Member> findByAgeGreaterThan(int age, Pageable pageable);
    Slice<Member> findByActive(boolean active, Pageable pageable);
    List<Member> findByName(String name, Sort sort);
}

// 호출
Pageable pageable = PageRequest.of(0, 20, Sort.by("createdAt").descending());
Page<Member> page = memberRepository.findByAgeGreaterThan(20, pageable);
page.getContent();         // List<Member>
page.getTotalElements();   // 전체 행 수 (COUNT 쿼리 추가 실행)
page.hasNext();
```

### 4-1. Page vs Slice

| 타입 | totalCount 쿼리 | 적합 |
|------|-----------------|------|
| `Page` | 실행 (느림) | "총 N 페이지" UI |
| `Slice` | 실행 안 함 | 무한 스크롤 |

`Slice` 가 가벼움. 총 페이지 수가 정말 필요할 때만 `Page`.

## 5. JPQL fetch join

> N+1 의 가장 흔한 해결 — 연관 객체를 한 번에 JOIN.

```java
@Query("SELECT m FROM Member m JOIN FETCH m.team WHERE m.age > :age")
List<Member> findWithTeam(@Param("age") int age);
```

`JOIN FETCH` 는 Member + Team 을 한 쿼리로 가져옴. 자세한 N+1 대응은 `./03-03.프록시와 N+1.md`.

### 5-1. fetch join 의 함정

```java
@Query("SELECT m FROM Member m JOIN FETCH m.orders")
Page<Member> findWithOrdersPaged(Pageable pageable);    // 위험!
```

컬렉션 fetch join + 페이징 = 메모리에서 페이징. 데이터가 많으면 OOM. 대안:

1. `@BatchSize` — 컬렉션은 IN 절로 한 번에.
2. 서브쿼리로 PK 만 페이징 후 fetch.

## 6. Lock

> 비관적/낙관적 락을 메서드 단위로 지정.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT m FROM Member m WHERE m.id = :id")
Member findByIdForUpdate(@Param("id") Long id);

@Lock(LockModeType.OPTIMISTIC)
@Query("SELECT m FROM Member m WHERE m.id = :id")
Member findByIdOptimistic(@Param("id") Long id);
```

| LockModeType | 의미 |
|--------------|------|
| `PESSIMISTIC_READ` | `SELECT ... LOCK IN SHARE MODE` |
| `PESSIMISTIC_WRITE` | `SELECT ... FOR UPDATE` |
| `OPTIMISTIC` | `@Version` 컬럼 검증 |
| `OPTIMISTIC_FORCE_INCREMENT` | 변경 없어도 version 증가 |

상세는 `./04-02.낙관적 비관적 락.md`.

## 7. JPA Hint

> 옵티마이저나 캐시에 힌트 — 거의 안 씀.

```java
@QueryHints(value = @QueryHint(name = "org.hibernate.readOnly", value = "true"))
@Query("SELECT m FROM Member m WHERE m.id = :id")
Member findByIdReadOnly(@Param("id") Long id);
```

`readOnly = true` 는 스냅샷 저장 안 함 — 변경 감지가 안 되는 대신 메모리 절약.
