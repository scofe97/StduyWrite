# 스프링 트랜잭션 (추상화, 동기화, AOP, 자동 등록, 전파)
---
> 스프링 트랜잭션의 추상은 다섯 층으로 쌓여 있다 — `PlatformTransactionManager` 인터페이스, 동기화 메커니즘, AOP 프록시, 자동 등록, 전파 옵션. 이 다섯이 같이 동작해 `@Transactional` 한 줄이 가능하다.

## 1. 트랜잭션 추상화 — `PlatformTransactionManager`

> JDBC·JPA·JTA 모두 같은 인터페이스로 — 코드는 DB 종류 무관.

```java
public interface PlatformTransactionManager {
    TransactionStatus getTransaction(TransactionDefinition def);
    void commit(TransactionStatus status);
    void rollback(TransactionStatus status);
}
```

구현체:

| 구현 | 적합 |
|------|------|
| `DataSourceTransactionManager` | 순수 JDBC, MyBatis |
| `JpaTransactionManager` | JPA |
| `JtaTransactionManager` | 분산 트랜잭션 (XA) |

Spring Boot 가 의존성 보고 자동 선택. 직접 빈 선언할 일은 거의 없음.

## 2. 트랜잭션 동기화 — 스레드 로컬 컨텍스트

> Connection 을 메서드 인자로 안 넘기고도 *같은 트랜잭션 안* 임을 어떻게 알까?

스프링은 `TransactionSynchronizationManager` 의 `ThreadLocal<Connection>` 으로 현재 스레드의 Connection 을 추적한다.

```
[Service.methodA()  — 트랜잭션 시작]
   ↓ Connection 을 ThreadLocal 에 박음
[Repository.find()]
   ↓ DataSource.getConnection() 호출
   ↓ Spring 이 가로채 ThreadLocal 의 Connection 반환  ← 같은 Connection!
[Repository.update()]
   ↓ 위와 동일
[Service.methodA() — 커밋]
   ↓ ThreadLocal 의 Connection commit
```

이 덕분에 Repository 코드는 *트랜잭션을 인지하지 않고도* 같은 트랜잭션 안에서 동작한다.

### 2-1. 스레드 로컬의 주의

웹 환경에서 한 요청 = 한 스레드라는 전제. 비동기·코루틴 환경에서는 *컨텍스트 전파* 가 필요 (Reactor 의 `Context`, Kotlin 의 `CoroutineContext`).

#### parallelStream 안에서 트랜잭션이 깨지는 함정

`@Transactional` 메서드 안에서 `parallelStream` 으로 작업을 나누면 *트랜잭션이 따라가지 않는다*. 흔히 만나는 실패다.

```java
@Transactional
public void saveAll(List<MyEntity> entities) {
    entities.parallelStream().forEach(e -> {
        repository.save(e);
        if (e.isInvalid()) throw new RuntimeException();  // 전체 롤백 기대
    });
}
```

예외가 나도 *일부 데이터가 남는다*. 원인은 트랜잭션 동기화가 `ThreadLocal` 위에 있다는 점이다.

1. `@Transactional` 은 *메서드를 호출한 스레드* 의 `ThreadLocal` 에 커넥션을 박는다.
2. `parallelStream` 은 작업을 `ForkJoinPool` 의 *다른 스레드들* 로 흩뿌린다.
3. 그 워커 스레드들은 *자기 ThreadLocal 이 비어 있어* 트랜잭션 밖에서 동작한다. 각자 새 커넥션을 잡고 *개별 auto-commit* 으로 저장한다.

결과적으로 메인 스레드의 트랜잭션이 롤백돼도 워커 스레드들이 이미 커밋한 데이터는 남는다. 부가로 *커넥션 풀 고갈* 위험도 있다 — 워커마다 커넥션을 잡으면 풀 크기를 순식간에 넘는다.

대응은 셋 중 하나다.

- **순차 처리로 바꾼다.** 트랜잭션 안에서는 `stream()` (순차) 을 쓴다. 같은 스레드라 트랜잭션이 유지된다.
- **트랜잭션을 워커 단위로 분리한다.** 각 작업을 별도 `@Transactional` 메서드로 만들어 *작업마다 독립 트랜잭션* 으로 처리한다. 단 전체 원자성은 포기한다.
- **배치 처리로 옮긴다.** 병렬화가 성능 목적이면 `saveAll` + `rewriteBatchedStatements` ([../jdbc/05-01 §2](../jdbc/05-01.Spring%20DB%20JPA%20설정%20최적화.md)) 로 순차 배치가 더 단순하고 안전한 경우가 많다.

같은 함정이 `@Async`, 별도 `Thread`, `CompletableFuture` 에서도 똑같이 일어난다. *트랜잭션은 스레드를 건너지 못한다* 가 핵심이다.

### 2-2. 트랜잭션 매니저 12 단계 흐름

> `@Transactional` 한 줄이 가능한 이유는 트랜잭션 매니저·동기화 매니저·리포지토리가 *한 커넥션을 공유* 하기 때문입니다. 강의 1편 4장의 단계별 흐름을 정리합니다.

```
[Service]                                          [TransactionSynchronizationManager]
    │                                                            │
    │ 1. getTransaction(definition)                              │
    │    ────────────────────────→ [TransactionManager]          │
    │                                  │                         │
    │                                  │ 2. dataSource.getConnection()
    │                                  ↓                         │
    │                              [DataSource] → con0           │
    │                                  │                         │
    │                                  │ 3. con0.setAutoCommit(false)
    │                                  │ 4. 트랜잭션 시작        │
    │                                  │                         │
    │                                  │ 5. bindResource(con0) ──→ ThreadLocal에 con0 박힘
    │                                                            │
    │ 6. service 비즈니스 로직 실행                              │
    │     ↓                                                      │
    │ [Repository.update()]                                      │
    │     │ 7. DataSourceUtils.getConnection(ds)                 │
    │     │    ───────────────────────────────────────────────→ ThreadLocal에서 con0 꺼냄
    │     │ 8. con0 으로 SQL 실행                                │
    │     ↓                                                      │
    │ 9. 비즈니스 로직 종료                                       │
    │                                                            │
    │ 10. commit(status)                                         │
    │    ────────────────────────→ [TransactionManager]          │
    │                                  │ 11. ThreadLocal에서 con0 꺼냄
    │                                  │     con0.commit()       │
    │                                  │ 12. unbindResource()    │
    │                                  │     con0.close() ──→ 풀로 반환
```

핵심 결론 세 가지입니다.

- 리포지토리는 `DataSource` 가 아니라 `DataSourceUtils.getConnection(ds)` 를 통해 커넥션을 얻습니다. JdbcTemplate·JPA 가 내부적으로 이렇게 호출합니다.
- 같은 트랜잭션 안에서 호출된 리포지토리는 *서로 모르고도* 같은 커넥션을 공유합니다. 동기화 매니저의 `ThreadLocal<Connection>` 이 그 역할입니다.
- 커밋·롤백은 *트랜잭션을 시작한 메서드* (외부) 가 책임집니다. 안에서 호출된 리포지토리·서비스는 *물리 트랜잭션을 끝내지 않습니다* — 이 분리가 §7 의 전파 메커니즘으로 이어집니다.

## 3. `@Transactional` AOP

> 메서드 호출을 가로채 트랜잭션을 시작·종료 — 스프링의 가장 흔한 AOP 사용.

```java
@Service
public class OrderService {
    @Transactional
    public void process(Order o) {
        // 메서드 진입 시 트랜잭션 시작
        // 메서드 정상 종료 시 commit
        // RuntimeException 시 rollback
    }
}
```

### 3-1. AOP 의 자기 호출 함정

```java
@Service
public class OrderService {
    public void publicMethod() {
        this.processInTx();   // 같은 객체 메서드 직접 호출 — AOP 적용 안 됨!
    }

    @Transactional
    public void processInTx() {
        // 위에서 호출되면 트랜잭션 안 열림
    }
}
```

이유: AOP 는 *프록시 객체* 를 통해 동작. `this` 는 프록시가 아닌 원본 객체라 인터셉터를 거치지 않는다.

해결:

1. 자기 호출을 다른 빈으로 분리.
2. `ApplicationContext` 에서 자기 자신을 다시 받아 호출 (안티패턴 권장).
3. AspectJ 컴파일 시점 위빙 (드물게).

가장 자연스러운 해결은 1번 — 책임을 다른 빈으로.

## 4. 트랜잭션 AOP 의 초기화 시점

> Spring Boot 의 자동 트랜잭션 매니저 등록.

`spring-boot-starter-data-jpa` 가 포함되어 있으면 자동:

1. `DataSource` 자동 등록 (HikariCP).
2. `EntityManagerFactory` 자동 등록.
3. `JpaTransactionManager` 자동 등록.
4. `@EnableTransactionManagement` 자동 적용.

별도 설정 없이 `@Transactional` 즉시 사용 가능.

## 5. 예외와 롤백

> *RuntimeException* 만 자동 롤백 — Checked Exception 은 커밋.

```java
@Transactional
public void process() throws IOException {   // Checked
    throw new IOException();
    // 트랜잭션 COMMIT — 데이터 변경 살아남음!
}
```

이 동작이 의도와 다르면 `rollbackFor` 명시:

```java
@Transactional(rollbackFor = Exception.class)   // 모든 예외에 롤백
public void process() throws IOException { ... }
```

JPA 환경의 대부분 예외는 RuntimeException 이라 기본 동작으로 충분. Spring Data 도 RuntimeException 으로 자동 변환.

## 6. readOnly = true

> 변경이 없는 메서드는 명시 — Hibernate 가 스냅샷 저장 안 함.

```java
@Transactional(readOnly = true)
public Member findById(Long id) {
    return memberRepository.findById(id).orElseThrow();
}
```

효과:

1. **메모리 절약** — 변경 감지 스냅샷 없음.
2. **flush 안 함** — JPQL 호출 시 flush 생략.
3. **DB 측 readOnly 힌트** — 일부 DB 는 readonly 트랜잭션 최적화.

Spring Data 의 `SimpleJpaRepository` 가 클래스 레벨에 이미 `@Transactional(readOnly = true)`. 조회만 하는 서비스는 클래스 레벨 readOnly + 변경 메서드에만 `@Transactional`.

## 7. 트랜잭션 전파 (Propagation)

> 트랜잭션 중인 메서드가 또 다른 `@Transactional` 메서드를 호출하면 어떻게 될까?

| Propagation | 동작 |
|-------------|------|
| `REQUIRED` (기본) | 진행 중이면 참여, 없으면 시작 |
| `REQUIRES_NEW` | 항상 새로 시작 — 부모와 별도 |
| `SUPPORTS` | 있으면 참여, 없으면 비트랜잭션 |
| `NOT_SUPPORTED` | 있으면 일시 중지, 비트랜잭션으로 |
| `MANDATORY` | 반드시 진행 중이어야 — 없으면 예외 |
| `NEVER` | 진행 중이면 예외 |
| `NESTED` | 중첩 트랜잭션 (savepoint 기반) |

### 7-1. REQUIRED (기본)

```java
@Transactional
public void outer() {
    inner();   // 같은 트랜잭션에 참여
}

@Transactional
public void inner() {
    // outer 의 트랜잭션 안에서 동작 — 별도 commit 없음
}
```

`inner` 에서 예외 발생하면 *전체 롤백* (outer 변경도 포함).

### 7-2. REQUIRES_NEW

```java
@Transactional
public void outer() {
    try {
        inner();   // 별도 트랜잭션
    } catch (Exception e) {
        // inner 만 롤백, outer 는 살아남음
    }
}

@Transactional(propagation = REQUIRES_NEW)
public void inner() {
    // 부모와 별도 트랜잭션
}
```

용도: 감사 로그·알림 발송 같은 *부모 실패와 무관해야* 하는 부수 작업.

여기서 질문 하나 — REQUIRES_NEW 에서 inner 가 부모를 호출하면? 데드락 가능. inner 의 트랜잭션이 부모의 락을 기다리는데 부모도 inner 끝을 기다림. 부모-자식 락 충돌은 REQUIRES_NEW 의 흔한 함정.

### 7-3. NESTED (Savepoint)

```java
@Transactional
public void outer() {
    try {
        inner();
    } catch (Exception e) {
        // inner 만 롤백 (savepoint 기준)
    }
}

@Transactional(propagation = NESTED)
public void inner() { ... }
```

부모 트랜잭션 안에 savepoint 를 만들고, 실패 시 그 지점까지만 롤백. JDBC 의 `Connection.setSavepoint()` 위에서 동작. JPA 환경에서는 흔히 안 씀.

### 7-4. 물리 트랜잭션 vs 논리 트랜잭션

> `REQUIRED` 의 정확한 동작을 이해하려면 *물리* 와 *논리* 의 분리를 알아야 합니다.

용어를 먼저 정의합니다.

- **물리 트랜잭션** — DB 커넥션 수준의 진짜 트랜잭션. `con.setAutoCommit(false)` 부터 `con.commit()` 또는 `con.rollback()` 까지의 한 사이클입니다.
- **논리 트랜잭션** — 스프링이 `@Transactional` 마다 만드는 *추상 트랜잭션*. 실제 DB 커밋을 호출할 수도, 안 할 수도 있습니다.

`REQUIRED` 에서 외부와 내부가 같은 물리 트랜잭션을 공유할 때, 동작은 *어느 쪽이 신규 트랜잭션인가* 로 갈립니다.

| 시점 | 외부 (`isNewTransaction = true`) | 내부 (`isNewTransaction = false`) |
|------|--------------------------------|----------------------------------|
| `getTransaction()` | 물리 트랜잭션 시작 | 기존 트랜잭션에 참여 (`Participating in existing transaction`) |
| `commit()` | DB 에 물리 커밋 | 아무것도 안 함 |
| `rollback()` | DB 에 물리 롤백 | `rollbackOnly = true` 표시만 |

### 7-5. 외부 롤백·내부 롤백 시나리오

> 강의 2편 10장의 두 시나리오가 위 표의 *결과* 입니다.

**시나리오 1 — 외부 롤백 (내부는 커밋)**:

```java
TransactionStatus outer = txManager.getTransaction(new DefaultTransactionAttribute());
TransactionStatus inner = txManager.getTransaction(new DefaultTransactionAttribute());

txManager.commit(inner);   // 신규 트랜잭션 아니므로 물리 커밋 안 함
txManager.rollback(outer); // 신규 트랜잭션이므로 물리 롤백
```

내부가 커밋해도 물리 커밋이 일어나지 않아, 외부의 롤백이 전체에 적용됩니다. 결과적으로 *내부에서 저장한 데이터도 함께 롤백* 됩니다.

**시나리오 2 — 내부 롤백 (외부는 커밋)**:

```java
TransactionStatus outer = txManager.getTransaction(new DefaultTransactionAttribute());
TransactionStatus inner = txManager.getTransaction(new DefaultTransactionAttribute());

txManager.rollback(inner); // rollbackOnly = true 표시
txManager.commit(outer);   // 신규지만 rollbackOnly 가 박혀 있음
                           // → 물리 롤백 + UnexpectedRollbackException 던짐
```

내부가 롤백한 사실은 `rollbackOnly` 표시로 외부에 강제로 전파됩니다. 외부가 커밋을 호출해도 `rollbackOnly == true` 이면 물리 롤백이 일어나고, 트랜잭션 매니저가 `UnexpectedRollbackException` 을 던집니다.

스프링 로그로도 흔적이 보입니다.

```
Participating transaction failed - marking existing transaction as rollback-only
Global transaction is marked as rollback-only but transactional code requested commit
```

### 7-6. 정리 — REQUIRED 의 한 줄

`REQUIRED` 의 핵심은 *논리 트랜잭션 중 하나라도 롤백되면 전체 물리 트랜잭션이 롤백된다* 입니다. 부분 롤백을 원하면 물리 트랜잭션 자체를 분리해야 합니다 — `REQUIRES_NEW` 또는 메시지 큐로의 분리입니다. 구체 시나리오는 [`./04-01b.트랜잭션 전파 활용.md`](./04-01b.트랜잭션%20전파%20활용.md) 가 다룹니다.
