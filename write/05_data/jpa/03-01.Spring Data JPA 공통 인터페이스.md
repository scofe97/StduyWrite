# Spring Data JPA 공통 인터페이스
---
> Spring Data JPA 는 JPA 의 반복 코드를 한 번 더 줄인다. `Repository` 인터페이스만 선언하면 Spring 이 구현체를 자동 생성 — 그 구현체가 `SimpleJpaRepository` 다.

## 1. JpaRepository 의 계층

> Spring Data 의 Repository 인터페이스는 다섯 계층으로 점층적 추상.

```
Repository<T, ID>                       -- 빈 마커
  └─ CrudRepository<T, ID>              -- 기본 CRUD
       └─ PagingAndSortingRepository    -- 페이징·정렬
            └─ JpaRepository<T, ID>     -- JPA 특화 (flush, batch)
```

일반 사용은 `JpaRepository` 만 상속하면 된다. 더 가벼운 추상이 필요하면 `CrudRepository`.

```java
public interface MemberRepository extends JpaRepository<Member, Long> {
    // 메서드 없어도 동작
}
```

기본 제공 메서드:

| 메서드 | 동작 |
|--------|------|
| `save(entity)` | INSERT 또는 UPDATE |
| `saveAll(entities)` | 배치 |
| `findById(id)` | 단건 조회 (Optional) |
| `findAll()`, `findAll(Sort)`, `findAll(Pageable)` | 전체·정렬·페이징 |
| `existsById(id)` | 존재 여부 |
| `count()` | 전체 행 수 |
| `delete(entity)`, `deleteById(id)`, `deleteAll()` | 삭제 |
| `flush()` | 즉시 SQL 전송 |
| `saveAndFlush(entity)` | save + flush |

## 2. 구현체 `SimpleJpaRepository`

> Spring Data 가 런타임에 등록하는 실제 구현체 — 그 동작을 이해하면 함정이 보인다.

```java
@Repository
@Transactional(readOnly = true)
public class SimpleJpaRepository<T, ID> implements JpaRepository<T, ID> {

    @Override
    @Transactional
    public <S extends T> S save(S entity) {
        if (entityInformation.isNew(entity)) {
            em.persist(entity);
            return entity;
        } else {
            return em.merge(entity);
        }
    }
    // ...
}
```

핵심 동작:

1. **클래스 레벨 `@Transactional(readOnly = true)`** — 모든 조회 메서드가 자동으로 readOnly.
2. **메서드 레벨 `@Transactional`** — save/delete 가 메서드 단위 트랜잭션.
3. **save 의 분기** — *새 Entity* 면 persist, *기존* 이면 merge.

## 3. 새 Entity 판단 전략

> `isNew()` 가 어떻게 판단하느냐가 save 의 동작을 결정한다.

기본 전략:

1. **`@Id` 필드가 null** → 새 Entity → `persist`.
2. **`@Id` 가 primitive (`long`)** → 0 이면 새 Entity → `persist`.
3. **그 외** → 기존 → `merge`.

여기서 질문 하나 — UUID PK 를 클라이언트에서 미리 발급해 두는 경우는? 그러면 `@Id` 가 항상 채워져 있어 *항상 merge* 가 호출된다. merge 는 *SELECT 후 UPDATE* 라서 INSERT 면 되는데 SELECT 가 한 번 더 간다.

해결: `Persistable<ID>` 구현으로 직접 `isNew()` 정의.

```java
@Entity
public class Member implements Persistable<UUID> {
    @Id
    private UUID id;

    @Transient
    private boolean isNew = true;

    @Override public UUID getId() { return id; }
    @Override public boolean isNew() { return isNew; }

    @PostLoad @PostPersist
    void markNotNew() { isNew = false; }
}
```

`@PostLoad`/`@PostPersist` 가 영속화 후 `isNew = false` 로 표시. 이후 save 호출은 merge 가 아닌 *UPDATE 만* (변경 감지로).

## 4. `@Repository` 와 예외 변환

> Repository 가 던지는 DB 예외가 Spring 의 표준 `DataAccessException` 으로 변환된다.

```
[JDBC SQLException]  →  [Spring DataAccessException 계층]
  - DuplicateKeyException
  - DataIntegrityViolationException
  - EmptyResultDataAccessException
  - OptimisticLockingFailureException
  - PessimisticLockingFailureException
```

이 변환의 효과:

1. **DB 종류에 무관한 예외 처리** — MySQL 의 1062 (duplicate) 가 PostgreSQL 의 23505 와 같은 예외로.
2. **RuntimeException 화** — 체크 예외 강제 catch 없음.

`SimpleJpaRepository` 는 클래스 레벨에 `@Repository` 가 붙어 있어 이 변환이 자동 적용된다.

## 5. 메서드 명명 규칙

> Repository 인터페이스에 메서드 시그니처만 적으면 Spring 이 쿼리 자동 생성.

```java
public interface MemberRepository extends JpaRepository<Member, Long> {
    List<Member> findByName(String name);
    List<Member> findByAgeGreaterThan(int age);
    Optional<Member> findByEmailIgnoreCase(String email);
    long countByActiveTrue();
}
```
