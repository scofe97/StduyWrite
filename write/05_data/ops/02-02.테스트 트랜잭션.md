# 테스트 트랜잭션
---
> 테스트가 DB 에 데이터를 박으면 다음 테스트에 영향을 준다. `@Transactional` + 자동 롤백 패턴이 이 문제를 한 줄로 풀어준다.

## 1. 테스트 격리의 문제

> 한 테스트의 부수 효과가 다음 테스트의 입력이 되면 *순서 의존 테스트* 가 된다.

```java
@Test
void test1() {
    memberRepository.save(new Member("alice"));
    assertThat(memberRepository.findAll()).hasSize(1);
}

@Test
void test2() {
    memberRepository.save(new Member("bob"));
    assertThat(memberRepository.findAll()).hasSize(1);  // 실패! alice 가 남아 있음
}
```

해결의 세 가지 방향:

1. **매 테스트 후 `DELETE FROM ...`** — 번거롭고 빠뜨리기 쉬움.
2. **임베디드 DB 의 재시작** — 느림.
3. **트랜잭션 롤백** — 빠르고 자동.

## 2. `@Transactional` + 자동 롤백

> 테스트 메서드에 `@Transactional` 을 붙이면 Spring 이 *기본 동작으로* 메서드 끝에 ROLLBACK 한다.

```java
@SpringBootTest
@Transactional
class MemberRepositoryTest {

    @Autowired MemberRepository memberRepository;

    @Test
    void save() {
        Member m = new Member("alice");
        memberRepository.save(m);
        assertThat(memberRepository.findById(m.getId())).isPresent();
    }
    // 메서드 끝나면 자동 ROLLBACK — DB 깨끗
}
```

여기서 질문 하나 — 운영 코드의 `@Transactional` 은 커밋이 기본인데 테스트는 왜 롤백이 기본일까? 테스트의 *기본 의도가 격리* 이기 때문이다. 운영의 `@Transactional` 은 비즈니스 트랜잭션이라 커밋이 자연스럽고, 테스트는 한 번 끝나면 흔적이 없어야 다음 테스트와 격리된다.

## 3. 데이터 롤백 정책

> 롤백 여부를 명시적으로 제어할 수 있다.

| 설정 | 동작 |
|------|------|
| `@Transactional` (기본) | 테스트 후 ROLLBACK |
| `@Rollback(false)` 또는 `@Commit` | 테스트 후 COMMIT |

```java
@Test
@Commit
void persistAndKeep() {
    // 디버깅 용도 — DB 에 데이터를 남겨 직접 확인
    memberRepository.save(new Member("debug"));
}
```

운영 환경 마이그레이션 테스트나 데이터 시드 생성에 `@Commit` 을 쓴다. 일반 테스트는 기본 동작 유지.

## 4. 트랜잭션 안의 함정

> 테스트 트랜잭션이 *Service 내부의 새 트랜잭션* 과 만나면 의도와 다른 결과.

```java
@Service
public class OrderService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void process(Order o) {
        // 새 트랜잭션 — 부모 롤백과 무관
    }
}

@Test
@Transactional
void test_orderService() {
    orderService.process(new Order(...));
    // 테스트 ROLLBACK 시점에 process 의 변경은 이미 별도 트랜잭션으로 COMMIT 된 상태
    // → DB 에 데이터 남음
}
```

`REQUIRES_NEW` 같은 전파 옵션이 테스트 롤백을 우회한다. 자세한 전파는 `../jpa/04-01.스프링 트랜잭션.md`.

대응: 해당 테스트는 별도 정리 (`@AfterEach` 의 `DELETE`) 또는 Testcontainers 로 매번 새 DB.

## 5. JPA 와 함께 쓸 때의 주의

> `EntityManager` 의 1차 캐시가 테스트의 가시성을 흐릴 수 있다.

```java
@Test
@Transactional
void update() {
    Member m = memberRepository.findById(1L).get();
    m.setName("changed");
    // flush 안 했지만 1차 캐시에 변경 있음

    Member same = memberRepository.findById(1L).get();
    assertThat(same.getName()).isEqualTo("changed");  // 통과 — 1차 캐시 동일성
}
```

이 테스트는 통과하지만, *실제 DB 에 UPDATE 가 갔는지* 는 검증 못 한다. 명시적 flush 필요:

```java
@Test
@Transactional
void update_with_flush() {
    Member m = memberRepository.findById(1L).get();
    m.setName("changed");

    entityManager.flush();
    entityManager.clear();    // 1차 캐시 비움

    Member fresh = memberRepository.findById(1L).get();
    assertThat(fresh.getName()).isEqualTo("changed");  // DB 에서 다시 조회
}
```

JPA 의 변경 감지가 의도대로 동작하는지 검증해야 할 때의 표준 패턴.
