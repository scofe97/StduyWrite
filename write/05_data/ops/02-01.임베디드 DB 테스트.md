# 임베디드 DB 테스트
---
> 테스트가 실 DB 에 의존하면 환경 차이·동시 실행·정리 비용이 늘어난다. 임베디드 DB 는 JVM 안에서 동작하는 DB 로, 테스트 단위마다 깨끗한 상태를 자동으로 만들어준다.

## 1. 임베디드 DB 가 푸는 문제

> 테스트의 외부 의존성을 JVM 안으로 끌어와 *결정적·격리된 환경* 을 만든다.

실 DB 의존의 문제:

1. **환경 설치** — 각 개발자가 같은 DB 버전·설정을 갖추기 어렵다.
2. **테스트 간 오염** — 한 테스트의 데이터가 다음 테스트에 영향.
3. **CI 의존성** — CI 서버가 DB 를 띄워야 함.

임베디드 DB (H2·HSQLDB·Derby) 는 JVM 시작 시 메모리에 DB 를 띄우고, 종료 시 사라진다. 외부 의존이 0.

| DB | 특징 |
|----|------|
| H2 | 가장 빠름, MySQL/PostgreSQL 호환 모드 지원, Spring Boot 기본 |
| HSQLDB | 오래된 안정성, Spring 의 옛 기본 |
| Apache Derby | JDK 번들 (`derby.jar`) |

## 2. Spring Boot 의 자동 활용

> 의존성에 `h2` 를 추가하고 `application.yml` 에 H2 URL 만 두면 끝.

```gradle
testRuntimeOnly 'com.h2database:h2'
```

```yaml
# application-test.yml
spring:
  datasource:
    url: jdbc:h2:mem:testdb;MODE=MySQL;DB_CLOSE_DELAY=-1
    driver-class-name: org.h2.Driver
    username: sa
    password:
  jpa:
    hibernate:
      ddl-auto: create-drop
```

- `mem:testdb` — 메모리에 떠 있는 DB.
- `MODE=MySQL` — MySQL 의 SQL 방언 흉내. 실 DB 와 차이 줄임.
- `DB_CLOSE_DELAY=-1` — 마지막 커넥션이 닫혀도 DB 유지 (테스트 끝까지).
- `ddl-auto: create-drop` — 시작 시 스키마 생성, 종료 시 삭제.

## 3. SQL 스크립트로 초기화

> `schema.sql` · `data.sql` 로 시작 시점의 스키마·데이터를 박는다.

```
src/test/resources/
├── schema.sql    # CREATE TABLE 등
└── data.sql      # INSERT 등
```

Spring Boot 가 자동으로 실행. JPA 의 `ddl-auto` 와 함께 쓰면 둘 다 적용된다.

```yaml
spring:
  sql:
    init:
      mode: always       # always | embedded | never
      schema-locations: classpath:schema.sql
      data-locations: classpath:data.sql
```

`mode: embedded` 가 기본 — 임베디드 DB 일 때만 실행.

## 4. 임베디드 DB 의 한계

> 실 DB 와의 차이가 테스트의 거짓 통과·거짓 실패를 만든다.

| 차이점 | 결과 |
|--------|------|
| SQL 방언 | H2 에서 동작한 쿼리가 MySQL/PG 에서 실패 |
| 인덱스 동작 | 옵티마이저가 달라 EXPLAIN 결과 다름 |
| MVCC·락 동작 | InnoDB 의 Gap Lock 같은 디테일 없음 |
| 전체 텍스트 검색·JSON 함수 | DB 별 함수 비호환 |

여기서 질문 하나 — 그러면 임베디드 DB 가 잘못된 선택인가? 아니다. *비즈니스 로직 검증* 에는 충분히 유효하다. 단, *DB 특화 동작 검증* 에는 실 DB (Testcontainers) 가 필요하다.

## 5. Testcontainers — 실 DB 와의 절충

> 도커 컨테이너로 실제 DB 를 띄워 테스트 — 임베디드의 외부 의존 없음 + 실 DB 의 정확성.

```gradle
testImplementation 'org.testcontainers:mysql'
```

```java
@Testcontainers
@SpringBootTest
class OrderRepositoryTest {
    @Container
    static MySQLContainer<?> mysql = new MySQLContainer<>("mysql:8.0");

    @DynamicPropertySource
    static void registerProps(DynamicPropertyRegistry r) {
        r.add("spring.datasource.url", mysql::getJdbcUrl);
        r.add("spring.datasource.username", mysql::getUsername);
        r.add("spring.datasource.password", mysql::getPassword);
    }

    // 테스트 메서드들
}
```

테스트 시작 시 도커가 실 MySQL 컨테이너를 띄우고, 끝나면 정리. 외부 의존 없이 실 DB 정확성 확보.

운영 환경에 가까운 검증이 필요한 통합 테스트에 표준 선택.
