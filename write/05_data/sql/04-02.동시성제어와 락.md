# 동시성제어와 락
---
> 동시 트랜잭션이 같은 데이터를 만나면 *이상 현상* 이 발생한다. 락은 그 이상 현상을 막는 도구지만, 너무 많이 잠그면 성능이 무너지고 너무 적게 잠그면 데이터가 깨진다. 격리 수준의 결정이 이 트레이드오프의 다른 표현이다.

## 1. 동시성 이상 현상

> 격리 수준의 정의가 어떤 이상 현상을 막을 것인가의 선언이다.

| 이상 현상 | 의미 |
|-----------|------|
| Dirty Read | 다른 트랜잭션이 *아직 커밋 안 한* 변경을 본다 |
| Non-Repeatable Read | 같은 행을 두 번 읽었는데 값이 다르다 (사이에 다른 트랜잭션이 커밋) |
| Phantom Read | 같은 범위를 두 번 읽었는데 행이 추가/삭제됐다 |
| Lost Update | 두 트랜잭션이 같은 행을 갱신, 한쪽 변경이 사라진다 |
| 모순성·연쇄복귀 | 부분 커밋된 데이터를 다른 트랜잭션이 봐서 후속 결정이 오염 |

격리 수준 4단계와 이상 현상의 대응:

| 격리 수준 | Dirty | Non-Repeatable | Phantom |
|-----------|-------|----------------|---------|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED | 차단 | 발생 | 발생 |
| REPEATABLE READ | 차단 | 차단 | 발생* |
| SERIALIZABLE | 차단 | 차단 | 차단 |

*InnoDB 의 REPEATABLE READ 는 Gap Lock 으로 Phantom 도 사실상 차단함.

자세한 격리 이론은 `../theory/01-04.트랜잭션과 격리 수준.md`.

## 2. 락의 종류

> 락은 *무엇을 잠그느냐* 와 *어떻게 잠그느냐* 의 두 차원이다.

### 2-1. 공유 락 (S-Lock) vs 배타 락 (X-Lock)

| 락 종류 | 다른 S | 다른 X |
|---------|--------|--------|
| Shared (S) | 공존 가능 | 차단 |
| Exclusive (X) | 차단 | 차단 |

여러 트랜잭션이 동시에 읽기는 가능 (S+S), 쓰기 중에는 다른 모든 접근 차단 (X 가 모든 것을 막음).

### 2-2. 잠금 단위 — 행·갭·범위

| 단위 | 의미 |
|------|------|
| Row Lock | 한 행 |
| Gap Lock | 행과 행 사이의 빈 공간 — Phantom 방지 |
| Next-Key Lock | Row Lock + 앞의 Gap Lock — InnoDB 의 기본 |

`SELECT * FROM orders WHERE id BETWEEN 10 AND 20 FOR UPDATE` 의 흐름:

1. id 10~20 의 행에 X-lock.
2. 9~10, 20~21 의 갭에 Gap Lock — 그 범위에 새 행이 INSERT 되는 것을 막음.

이게 InnoDB 가 REPEATABLE READ 에서 Phantom 을 막는 메커니즘.

여기서 핵심 하나 — InnoDB 의 레코드 락은 *테이블의 행이 아니라 인덱스의 레코드* 를 잠근다. `WHERE id = 5` 가 PK 인덱스를 타면 그 인덱스 엔트리에 락이 걸린다. 그래서 *조건 컬럼에 인덱스가 없으면* 옵티마이저가 풀스캔을 하며 *스캔한 모든 행* 에 락을 건다. 한 행만 갱신할 의도였는데 테이블 전체가 잠기는 사고가 여기서 난다. 락 범위를 좁히려면 조건 컬럼에 인덱스를 두는 것이 첫 단추다.

### 2-3. 자동 증가 락 (Auto-Increment Lock)

> `AUTO_INCREMENT` 값을 채번할 때 InnoDB 가 거는 테이블 수준 락 — 순차성을 보장하되 트랜잭션과 다른 규칙으로 동작한다.

`INSERT` 나 `REPLACE` 가 `AUTO_INCREMENT` 컬럼 값을 받아야 할 때, InnoDB 는 *다음 값을 채번하는 동안* 테이블 수준의 짧은 락을 건다. 두 트랜잭션이 동시에 INSERT 해도 채번 값이 겹치지 않게 하는 장치다.

다른 락과 구별되는 두 성질이 있다.

1. **트랜잭션 끝까지 유지되지 않는다.** 일반 락은 COMMIT/ROLLBACK 까지 유지되지만, 자동 증가 락은 *채번 직후 즉시* 풀린다. 그래야 다음 INSERT 가 곧바로 채번할 수 있다.
2. **롤백해도 값이 복구되지 않는다.** 트랜잭션이 ROLLBACK 되어도 이미 채번한 `AUTO_INCREMENT` 값은 되돌아오지 않는다. 그래서 *PK 값에 구멍 (gap)* 이 생긴다. 1, 2, 4 처럼 3 이 빠질 수 있다. PK 가 연속이어야 한다는 가정을 두면 안 되는 이유다.

채번 방식은 `innodb_autoinc_lock_mode` 로 조정한다.

| 모드 | 값 | 동작 |
|------|-----|------|
| Traditional | 0 | 모든 INSERT 가 문장 끝까지 테이블 락 유지 — 순차성 강하나 동시성 낮음 |
| Consecutive | 1 (8.0 이전 기본) | 단순 INSERT 는 락 빨리 풂, 행 수 모르는 INSERT (`INSERT ... SELECT`) 만 테이블 락 |
| Interleaved | 2 (8.0 기본) | 락 최소화, 동시성 최고. 단 한 문장 안에서 채번 값이 *비연속* 일 수 있음 |

8.0 의 기본인 Interleaved 모드는 동시성을 위해 *값의 연속성을 포기* 한다. 복제 환경에서 statement 기반 복제를 쓰면 채번 순서가 달라질 수 있어, row 기반 복제와 함께 쓰는 것이 안전하다.

## 3. 공유락·배타락의 사용 규칙

> Two-Phase Locking (2PL) — 트랜잭션이 락을 한 번 풀기 시작하면 더 잡지 않는다.

```
Growing phase: 락 획득만
Shrinking phase: 락 해제만 (보통 COMMIT/ROLLBACK 시점에 일괄)
```

이 규칙이 깨지면 *덜 직렬화된* 결과가 나올 수 있다. InnoDB 는 Strong Strict 2PL — 모든 락을 COMMIT/ROLLBACK 까지 유지.

## 4. 데드락

> 두 트랜잭션이 서로의 락을 기다리며 영원히 멈춤.

```
T1: locks A, wants B
T2: locks B, wants A
→ 데드락
```

InnoDB 의 처리:

1. **자동 감지** — 락 그래프에서 사이클 발견.
2. **희생자 선택** — *변경 행 수가 적은* 트랜잭션을 강제 ROLLBACK.
3. **에러 반환** — `Deadlock found when trying to get lock; try restarting transaction` (SQLState 40001).

애플리케이션은 이 에러를 받으면 *재시도* 해야 한다. 자동 재시도가 표준 패턴.

### 4-1. 데드락 줄이는 방법

1. **락 순서 통일** — 모든 트랜잭션이 같은 순서로 잠금.
2. **트랜잭션 짧게** — 락 보유 시간 최소화.
3. **인덱스로 정확히** — 인덱스 누락 → 풀스캔 → 더 많은 행에 락.
4. **`SELECT FOR UPDATE` 신중히** — 정말 필요한 경우만.

## 5. 실전 — 낙관적 락 vs 비관적 락의 결정

> 같은 동시성 문제를 두 가지 방식으로 풀 수 있다 — 충돌 빈도가 선택을 정한다.

| 방식 | 동작 | 적합 |
|------|------|------|
| 비관적 락 | `SELECT FOR UPDATE` 로 미리 락 | 충돌 자주, 재시도 비용 큼 |
| 낙관적 락 | `@Version` 컬럼, 커밋 시 충돌 확인 | 충돌 드물게, 처리량 우선 |

비관적은 안전하지만 처리량 낮고, 낙관적은 처리량 높지만 충돌 시 재시도 필요. 같은 도메인이라도 *재고 차감* 은 비관적이 자연스럽고 *프로필 수정* 은 낙관적이 자연스럽다.
