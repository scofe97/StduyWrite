# MySQL 기초와 InnoDB
---
> MySQL 은 단일 DB 가 아니라 *MySQL 엔진 + 교체 가능한 스토리지 엔진* 의 조합이다. 이 구조 위에서 InnoDB 가 사실상 표준이 된 이유 — 트랜잭션·외래키·MVCC — 을 짚어두면 운영 의사결정이 분명해진다.

## 1. MySQL 의 두 층 구조

> MySQL 엔진은 SQL 파싱·옵티마이저·캐시 등을 담당하고, 스토리지 엔진은 실제 데이터 저장·인덱스를 담당한다.

```
[Client]
  ↓ SQL
[MySQL 엔진 — Parser, Optimizer, Query Cache]
  ↓ Handler API
[Storage Engine — InnoDB / MyISAM / Memory / Archive ...]
  ↓
[디스크]
```

이 분리 덕분에 한 테이블은 InnoDB, 다른 테이블은 MyISAM 으로 둘 수 있다 (권장하지는 않음). 운영 환경에서는 거의 모든 테이블이 InnoDB.

### 1-1. InnoDB vs MyISAM

| 측면 | InnoDB | MyISAM |
|------|--------|--------|
| 트랜잭션 | 지원 (ACID) | 미지원 |
| 외래키 | 지원 | 미지원 |
| 락 단위 | 행 (row) | 테이블 |
| 크래시 복구 | 자동 (redo log) | 수동 (`REPAIR TABLE`) |
| 풀텍스트 | 지원 (5.6+) | 지원 (오래된) |
| 적합 | 일반 OLTP | 읽기 전용 분석 (드물게) |

새 테이블은 거의 항상 InnoDB. MyISAM 은 레거시 호환 외에는 선택할 이유가 없다.

## 2. Character Set 과 Collation

> 같은 문자열이라도 인코딩·정렬 규칙이 다르면 다른 결과가 나온다 — 한국어 운영의 첫 함정.

```sql
CREATE DATABASE myapp CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

- **utf8mb4** — 4바이트 UTF-8, 이모지 포함 모든 유니코드. 그냥 `utf8` 은 3바이트라 이모지 깨짐. 새 DB 는 반드시 `utf8mb4`.
- **utf8mb4_unicode_ci** — 대소문자 무관 비교, 유니코드 표준 정렬. 한국어·다국어 정렬에 자연스럽다.
- **utf8mb4_bin** — 바이트 단위 비교. 대소문자 구분 필요할 때.

여기서 질문 하나 — 기존 DB 가 `utf8` 인데 `utf8mb4` 로 옮기려면? `ALTER DATABASE`, `ALTER TABLE CONVERT TO CHARACTER SET utf8mb4` 의 두 단계. 인덱스 크기가 증가할 수 있어 컬럼 길이가 인덱스 한계(767바이트, 또는 innodb_large_prefix 로 3072) 를 넘으면 사전에 컬럼 짧게 조정.

### 2-1. 콜레이션 3종 비교

> 콜레이션은 *문자열을 어떻게 비교하고 정렬할지* 정의한 규칙이다. 같은 `utf8mb4` 라도 세 콜레이션이 비교 결과를 다르게 만든다.

`utf8mb4_unicode_ci`, `utf8mb4_general_ci`, `utf8mb4_bin` 세 가지가 운영에서 자주 만나는 선택지다.

| 구분 기준 | `utf8mb4_general_ci` | `utf8mb4_unicode_ci` | `utf8mb4_bin` |
|----------|---------------------|---------------------|---------------|
| 대소문자 | 무시 (`A` = `a`) | 무시 (`A` = `a`) | 구분 (`A` ≠ `a`) |
| 전각·반각 | 구분 | 무시 (`Ａ` = `A`) | 구분 |
| 이모지·악센트 | 부분 구분 | 표준 규칙으로 비교 | 완전 구분 (바이트) |
| 비교 방식 | 문자 개별 비교 | 문자 간 관계까지 고려 | 바이트 단위 비교 |
| 속도·정확도 | 빠름·정확도 낮음 | 느림·정확도 높음 | 가장 단순·완전 구분 |

세 콜레이션의 성격은 다음과 같다.

- `general_ci` 는 문자를 *하나씩* 비교해 빠르지만, 유니코드 표준 정렬 규칙을 일부만 따른다. 다국어 정렬 정확도가 낮다.
- `unicode_ci` 는 문자 *사이의 관계* (악센트·전각 등) 까지 유니코드 표준대로 비교한다. 느리지만 한국어·다국어 정렬이 자연스럽다.
- `bin` 은 바이트 값을 그대로 비교한다. 대소문자·이모지를 전부 다른 문자로 보므로 *완전한 구분* 이 필요한 경우 (토큰·해시·코드 값) 에 쓴다.

여기서 질문 하나 — 대소문자를 구분해야 하는 컬럼 (예: Base64 토큰) 에 `ci` 콜레이션을 쓰면? `'aB'` 와 `'Ab'` 가 같은 값으로 취급되어 UNIQUE 제약·조회가 깨진다. 이런 컬럼만 `bin` 또는 `_cs` (case-sensitive) 콜레이션으로 따로 지정한다.

## 3. MySQL 드라이버와 Connection Pool

> 자바 환경의 표준 — `mysql-connector-j` (MySQL Connector/J) + HikariCP.

```gradle
runtimeOnly 'com.mysql:mysql-connector-j'
```

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/myapp?useSSL=false&serverTimezone=UTC
    username: appuser
    password: secret
    hikari:
      maximum-pool-size: 10
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
```

Connection Pool 의 자세한 설계는 `../jdbc/01-01.커넥션 풀과 DataSource.md` 가 다룬다.

## 4. InnoDB 의 핵심 구조

> InnoDB 가 단순한 저장소가 아니라 MVCC·트랜잭션·크래시 복구를 모두 책임진다.

| 구성 요소 | 역할 |
|----------|------|
| Buffer Pool | 데이터·인덱스 페이지 메모리 캐시 (보통 RAM 의 70~80%) |
| Redo Log | "곧 디스크에 쓸 변경" 의 미리쓰기 로그 (WAL, `../theory/03-04.WAL 패턴.md`) |
| Undo Log | MVCC 의 이전 버전·롤백용 |
| Doublewrite Buffer | torn page 방지 |
| Clustered Index | PK 기반 정렬 저장 (테이블 자체가 PK 인덱스) |

이 구조의 효과:

1. **빠른 쓰기** — 디스크 랜덤 쓰기 대신 redo log 순차 쓰기 + buffer pool 갱신.
2. **MVCC** — 같은 행의 여러 버전을 undo log 에 유지, 격리 수준 구현.
3. **크래시 복구** — redo log 재생으로 자동.

상세 MVCC 동작은 `./04-01.InnoDB MVCC.md` 가 다룬다.

## 5. DB 가 병목이 되는 흔한 이유

> 대용량 시스템에서 DB 가 느려지는 패턴은 거의 다섯 가지로 정리된다.

1. **인덱스 누락** — Full table scan. `EXPLAIN` 으로 확인 (`./03-01.인덱스 실전.md`).
2. **N+1** — 1 쿼리 + N 추가 쿼리. JPA·ORM 의 흔한 함정 (`../jpa/03-03.프록시와 N+1.md`).
3. **락 경합** — 같은 행에 동시 쓰기. 격리 수준·락 전략 (`./04-02.동시성제어와 락.md`).
4. **Connection Pool 부족** — 동시 요청 > 풀 크기 → 대기 큐. `../jdbc/01-01.커넥션 풀과 DataSource.md`.
5. **큰 결과셋 전송** — `SELECT *` 후 클라이언트에서 필터. 필요한 컬럼만, LIMIT 활용.

다섯 중 어느 것이든 *측정 먼저* — 추측으로 인덱스 추가하지 말고 `EXPLAIN` 과 슬로우 쿼리 로그를 본다.

3. `utf8` 과 `utf8mb4` 의 차이는 무엇이고, 새 DB 가 반드시 `utf8mb4` 여야 하는 이유는 무엇인가? — §2
