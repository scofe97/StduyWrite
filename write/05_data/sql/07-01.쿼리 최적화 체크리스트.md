# 쿼리 최적화 체크리스트

---

> 자주 마주치는 쿼리 개선 패턴을 한 자리에 모은다. 인덱스 자체의 함정은 [03-01 §5](03-01.인덱스%20실전%20—%20EXPLAIN%2C%20IOT%2C%20페이지네이션.md) 가, 조인·서브쿼리의 문법은 [01-02](01-02.JOIN%2C%20SUBQUERY%2C%20CTE.md) 가 다룬다. 본 노트는 그 둘을 *개선 체크리스트* 로 묶는 진입점이다.

> 개선 효과의 수치는 *환경 의존* 이다. 데이터 분포·인덱스·DB 버전에 따라 크게 달라진다. 수치를 외우기보다 `EXPLAIN` 으로 측정하고 적용하는 습관이 핵심이다.

## 1. 컬럼과 필터

> 가져올 데이터와 거를 데이터를 먼저 줄인다 — 가장 효과가 큰 한 끗.

### 1-1. 선택적 컬럼 명시 (`SELECT *` 회피)

```sql
-- 비효율
SELECT * FROM orders WHERE customer_id = 1001;

-- 개선
SELECT id, amount, status FROM orders WHERE customer_id = 1001;
```

`SELECT *` 는 필요 없는 컬럼까지 네트워크로 전송하고, *Covering Index 의 기회* 를 없앤다. 필요한 컬럼만 명시하면 인덱스만으로 결과를 만들 수 있는 경우 ([03-01 §3-1](03-01.인덱스%20실전%20—%20EXPLAIN%2C%20IOT%2C%20페이지네이션.md)) 디스크 접근이 사라진다.

### 1-2. `WHERE` 로 거르고 `HAVING` 은 집계에만

```sql
-- 비효율: 전부 그룹화한 뒤 HAVING 으로 거름
SELECT dept_id, COUNT(*) FROM employees
GROUP BY dept_id HAVING dept_id > 10;

-- 개선: 그룹화 전 WHERE 로 행을 먼저 줄임
SELECT dept_id, COUNT(*) FROM employees
WHERE dept_id > 10 GROUP BY dept_id;
```

`WHERE` 는 그룹화 *전* 행을 거르고, `HAVING` 은 그룹화 *후* 그룹을 거른다. 행 필터를 `WHERE` 로 옮기면 그룹화 대상 자체가 줄어든다. `HAVING` 은 `COUNT(*) > 10` 같은 *집계 결과 필터* 에만 쓴다 ([01-01 §5](01-01.SQL%20기초%20—%20DDL%2C%20DML%2C%20함수.md)).

### 1-3. 불필요한 `DISTINCT` 제거

`DISTINCT` 는 중복 제거를 위해 정렬·해시 비용을 치른다. 결과에 중복이 없음이 보장되면 (PK 조회, 적절한 조인) 빼는 것이 맞다. 습관적으로 붙인 `DISTINCT` 가 큰 결과셋에서 비싼 연산이 된다.

## 2. 조인과 서브쿼리

> 같은 결과를 여러 방법으로 만들 수 있다 — 옵티마이저가 잘 푸는 형태를 고른다.

### 2-1. 서브쿼리를 조인으로

```sql
-- 서브쿼리
SELECT * FROM employees
WHERE dept_id IN (SELECT id FROM departments WHERE budget > 1000000);

-- 조인
SELECT e.* FROM employees e
JOIN departments d ON e.dept_id = d.id
WHERE d.budget > 1000000;
```

옵티마이저가 서브쿼리를 풀어내지 못하는 경우가 있다. 같은 결과를 조인으로 만들 수 있으면 조인이 보통 더 빠르다 ([01-02 §3](01-02.JOIN%2C%20SUBQUERY%2C%20CTE.md)).

### 2-2. `IN` 절은 인덱스를 타게

`WHERE id IN (1, 2, 3)` 은 `id` 에 인덱스가 있으면 빠르다. 단 `IN` 의 값이 *서브쿼리* 이고 그게 큰 결과를 내면 조인으로 바꾸는 편이 낫다. JPA 환경에서 `IN` 절 파라미터 개수가 매번 달라 실행계획 캐시가 갈리는 문제는 `in_clause_parameter_padding` 으로 다룬다 ([jdbc/05-01 §4-1](../jdbc/05-01.Spring%20DB%20JPA%20설정%20최적화.md)).

### 2-3. 존재 검사는 `EXISTS`

```sql
-- DISTINCT 로 중복 제거하며 존재 검사 (비효율)
SELECT DISTINCT c.* FROM customers c
JOIN orders o ON c.id = o.customer_id;

-- EXISTS — 첫 매칭에서 멈춤
SELECT c.* FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.id);
```

"주문이 하나라도 있는 고객" 처럼 *존재 여부* 만 알면 되는 경우 `EXISTS` 가 첫 매칭에서 멈춘다. `JOIN + DISTINCT` 는 모든 매칭을 만든 뒤 중복을 제거하므로 더 비싸다.

### 2-4. 조인 조건의 `OR` 제거

```sql
-- OR — 양쪽 인덱스를 못 쓸 때가 많음
SELECT * FROM t1 JOIN t2 ON t1.a = t2.a OR t1.b = t2.b;

-- UNION 으로 분리 — 각 인덱스 활용
SELECT * FROM t1 JOIN t2 ON t1.a = t2.a
UNION
SELECT * FROM t1 JOIN t2 ON t1.b = t2.b;
```

`OR` 조건은 각각의 인덱스가 있어도 옵티마이저가 둘 다 못 쓰는 경우가 많다. `UNION` 으로 분리하면 각 조건이 자기 인덱스를 탄다.

## 3. 집합 연산

> `UNION` 의 숨은 비용을 안다.

```sql
-- UNION — 중복 제거 위해 정렬
SELECT id FROM a UNION SELECT id FROM b;

-- UNION ALL — 중복 제거 안 함 (빠름)
SELECT id FROM a UNION ALL SELECT id FROM b;
```

`UNION` 은 결과를 정렬해 중복을 제거한다. 두 집합에 중복이 없음을 알면 `UNION ALL` 이 거의 항상 더 빠르다 ([01-02 §2](01-02.JOIN%2C%20SUBQUERY%2C%20CTE.md)).

## 4. 표현식

> 인덱스를 무력화하는 표현식을 피한다.

### 4-1. 컬럼에 함수·연산 적용 금지

```sql
-- 인덱스 무력화 — 함수 결과로 비교
SELECT * FROM orders WHERE DATE(created_at) = '2026-05-24';

-- 인덱스 활용 — 범위로 변환
SELECT * FROM orders
WHERE created_at >= '2026-05-24 00:00:00'
  AND created_at <  '2026-05-25 00:00:00';
```

`WHERE DATE(created_at) = ...` 처럼 *컬럼에 함수를 씌우면* 인덱스를 못 탄다. 함수 결과에 대한 인덱스가 따로 없는 한 풀스캔이 된다. 같은 조건을 *범위* 로 바꾸면 인덱스를 탄다. `WHERE price * 1.1 > 100` 같은 연산도 같은 함정이다. 자세한 인덱스 무력화 함정은 [03-01 §5](03-01.인덱스%20실전%20—%20EXPLAIN%2C%20IOT%2C%20페이지네이션.md) 가 다룬다.

### 4-2. 불필요한 수학 연산 제거

`WHERE salary > 1000 * 12` 같은 상수 연산은 미리 계산해 `WHERE salary > 12000` 으로 둔다. 옵티마이저가 상수 폴딩을 해주는 경우가 많지만, 컬럼이 연산에 끼면 (`salary / 12 > 1000`) 인덱스를 못 타므로 *컬럼을 연산에서 빼는* 형태로 다시 쓴다.

## 5. 적용 순서 — 측정 먼저

> 위 패턴들을 무작정 적용하지 않는다. 순서가 있다.

1. **`EXPLAIN` 으로 현재 계획 확인** — `type: ALL` (풀스캔), `Using filesort`, `Using temporary` 가 보이는 쿼리를 먼저 잡는다 ([03-01 §1](03-01.인덱스%20실전%20—%20EXPLAIN%2C%20IOT%2C%20페이지네이션.md)).
2. **슬로우 쿼리 로그로 느린 쿼리 식별** — 추측 대신 실제 느린 쿼리부터.
3. **위 패턴 적용 후 다시 `EXPLAIN`** — 개선 효과를 측정으로 확인.

추측으로 인덱스를 더 만들거나 쿼리를 비트는 것이 답이 아닌 경우가 대부분이다. 측정이 먼저다.

> 본문을 읽었다면 다음 질문에 답할 수 있어야 한다.

1. `SELECT *` 가 Covering Index 의 기회를 없앤다는 게 무슨 뜻인가? — §1-1
2. `WHERE` 와 `HAVING` 중 행 필터를 어디에 두어야 하며, 그 이유는 무엇인가? — §1-2
3. 존재 여부만 확인할 때 `JOIN + DISTINCT` 보다 `EXISTS` 가 빠른 이유는 무엇인가? — §2-3
4. `WHERE DATE(created_at) = ?` 가 인덱스를 못 타는 이유와, 같은 조건을 인덱스 친화적으로 바꾸는 방법은 무엇인가? — §4-1
5. 쿼리 최적화를 시작하는 올바른 순서는 무엇이며, 왜 측정이 먼저인가? — §5
