# 정규화와 비정규화
---
> 정규화는 "중복을 없애 갱신 이상을 막는다" 의 결정, 비정규화는 "조회 성능을 위해 중복을 받아들인다" 의 결정. 둘은 반대가 아니라 트레이드오프의 양 끝이다.

## 1. 정규화

> 같은 사실이 여러 곳에 저장되면 변경 시 한 곳을 빠뜨려 모순이 생긴다 — 이 모순을 미리 막는 절차.

| 정규형 | 규칙 | 위반의 결과 |
|--------|------|------------|
| 1NF | 한 셀에 단일 값 (반복 그룹 금지) | 검색·정렬 불가 |
| 2NF | PK 의 일부에만 종속되는 컬럼 분리 | 부분 갱신 이상 |
| 3NF | PK 외의 컬럼에 종속되는 컬럼 분리 | 이행 갱신 이상 |
| BCNF | 모든 결정자가 후보키 | 드문 갱신 이상 |

실무는 보통 3NF 까지가 기본 목표. BCNF·4NF·5NF 는 학술적이거나 대규모 시스템 한정.

### 1-1. 정규화의 효과

1. **갱신 이상 방지** — 회원 이메일 변경 시 한 행만 바꿈.
2. **저장 공간 절약** — 같은 데이터 중복 제거.
3. **모델 명확성** — 각 테이블의 책임이 분명.

## 2. 비정규화

> "조회마다 5개 JOIN 이 필요하다" 의 비용이 크면 *의도적으로 중복* 을 받아들인다.

비정규화는 결정이지 실수가 아니다. 다음 신호가 보이면 검토:

1. **조회가 쓰기보다 압도적으로 많은** 도메인 (SNS 피드·게임 랭킹).
2. **여러 JOIN 이 한 쿼리에 필요** + 응답 시간 SLA 가 ms 단위.
3. **읽기 모델과 쓰기 모델이 자연스럽게 갈리는** 경우 (CQRS, `../jpa/01-01.ORM 개념.md`).

### 2-1. 비정규화의 비용

1. **갱신 이상 위험** — 한 사실을 여러 행에 박았으면 모두 갱신해야 함.
2. **저장 비용 증가** — 중복 데이터.
3. **일관성 책임이 애플리케이션으로** — DB 가 보장 못 함.

이 비용을 받아들일 가치가 있는지 *측정 후* 결정. 추측으로 비정규화 시작하면 운영 부담만 늘어남.

## 3. 히스토리성 데이터는 정규화 대상이 아니다

> 시간에 따라 *그 시점의 사실* 을 박제해야 하는 데이터는 정규화 원칙의 예외다.

예시:

- **주문의 가격** — 주문 시점의 상품 가격을 박제. 상품 가격이 나중에 바뀌어도 주문 가격은 변하지 않아야 함.
- **주문의 배송 주소** — 주문 시점 주소. 사용자가 주소 바꿔도 옛 주문의 배송 주소는 유지.
- **감사 로그** — 변경 이력 자체가 의미.

이런 데이터는 정규화하면 안 된다. `order.product_id` 만 두고 가격을 JOIN 으로 가져오면 가격 변경 시 옛 주문이 깨진다.

```sql
-- 잘못된 정규화
CREATE TABLE orders (
    id INT PRIMARY KEY,
    product_id INT,    -- 가격은 JOIN 으로 — 가격 변경 시 옛 주문 깨짐
    quantity INT
);

-- 올바른 비정규화
CREATE TABLE orders (
    id INT PRIMARY KEY,
    product_id INT,
    product_name VARCHAR(100),     -- 주문 시점 박제
    product_price DECIMAL(10,2),   -- 주문 시점 박제
    quantity INT
);
```

## 4. 실전 — SNS 피드 모델링

> 정규화·비정규화의 선택이 가장 명확하게 갈리는 도메인 중 하나.

SNS 의 피드 (팔로우한 사람의 게시글 모음) 를 두 방식으로 풀 수 있다.

### Pull 모델 (정규화)

```sql
SELECT p.*
FROM posts p
JOIN follows f ON p.author_id = f.followee_id
WHERE f.follower_id = ?
ORDER BY p.created_at DESC LIMIT 20;
```

조회마다 JOIN. 한 사용자가 1000명 팔로우 + 각자 매일 10 게시글이면 매 피드 조회가 무거워진다.

### Push 모델 (비정규화)

```sql
-- 게시글 작성 시 모든 팔로워의 피드 테이블에 사전 복사
INSERT INTO feed (user_id, post_id, created_at)
SELECT f.follower_id, ?, NOW() FROM follows f WHERE f.followee_id = ?;

-- 조회는 단일 테이블 스캔
SELECT * FROM feed WHERE user_id = ? ORDER BY created_at DESC LIMIT 20;
```

조회는 빠르지만 *쓰기가 폭발* 한다. 100만 팔로워의 셀럽이 게시글 1개 쓰면 100만 행 INSERT.

자세한 비교는 `./05-01.Fan-out 피드 시스템.md` 가 다룬다.

## 면접에서 받을 만한 질문

> 본문을 읽었다면 다음 질문에 답할 수 있어야 한다.

1. 정규화가 막으려는 갱신 이상이란 무엇이며, 실무가 보통 3NF 까지를 목표로 하는 이유는 무엇인가? — §1
2. 비정규화가 실수가 아니라 결정인 이유는 무엇이고, 어떤 신호가 보일 때 검토하는가? — §2
3. 주문 가격을 `product_id` 로 JOIN 해서 가져오면 무엇이 깨지는가? 히스토리성 데이터를 정규화하면 안 되는 이유는 무엇인가? — §3
4. SNS 피드의 Pull 모델과 Push 모델의 비용은 각각 어디에 쏠리는가? — §4
5. 비정규화를 하면 일관성 보장 책임이 어디로 옮겨가는가? — §2-1

각 질문에 막히면 본문 해당 절을 다시 본다.
