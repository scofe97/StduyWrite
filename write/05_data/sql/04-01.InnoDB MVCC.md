# InnoDB MVCC
---
> MVCC (Multi-Version Concurrency Control) 의 일반 이론은 `../theory/01-04.트랜잭션과 격리 수준.md` 가 다룬다. 본 문서는 InnoDB 가 그 이론을 구체적으로 어떻게 구현하는가 — 숨겨진 컬럼·Undo Log Chain·Read View — 만 다룬다.

## 1. InnoDB 의 숨겨진 컬럼

> 모든 InnoDB 행에 사용자가 안 보이는 시스템 컬럼이 셋 박혀 있다.

| 컬럼 | 의미 |
|------|------|
| `DB_TRX_ID` | 이 행을 마지막으로 갱신한 트랜잭션 ID (6바이트) |
| `DB_ROLL_PTR` | Undo Log 의 이전 버전을 가리키는 포인터 (7바이트) |
| `DB_ROW_ID` | PK 가 없을 때 자동 생성되는 row ID (6바이트, PK 있으면 없음) |

이 세 컬럼이 MVCC 의 토대다. 한 행의 *현재 상태* 와 *이전 버전 체인* 을 잇는다.

## 2. Undo Log Chain — 버전 체인

> 행을 갱신하면 이전 값이 Undo Log 로 박제되고, `DB_ROLL_PTR` 가 그것을 가리킨다.

```
[현재 행 (Clustered Index)]                [Undo Log]
PK=1, name="C", trx_id=300, roll_ptr=→ → [name="B", trx_id=200, roll_ptr=→]
                                                                          ↓
                                          [name="A", trx_id=100, roll_ptr=NULL]
```

`UPDATE` 가 일어날 때마다:

1. 새 값을 현재 행에 덮어쓴다.
2. 옛 값은 Undo Log 에 박제.
3. `DB_TRX_ID` 를 현재 트랜잭션 ID 로.
4. `DB_ROLL_PTR` 가 방금 박제한 Undo Log 를 가리킴.

이 체인이 두 가지 일을 한다.

1. **롤백** — `ROLLBACK` 시 체인을 따라가 옛 값 복원.
2. **MVCC SELECT** — 다른 트랜잭션이 옛 버전을 봐야 할 때 체인을 따라감.

## 3. Read View — "지금 내가 볼 수 있는 버전"

> 트랜잭션이 SELECT 를 할 때 InnoDB 가 만드는 스냅샷 메타데이터.

Read View 는 다음 네 가지를 박는다.

| 필드 | 의미 |
|------|------|
| `m_ids` | View 생성 시점에 활성(active) 인 트랜잭션 ID 목록 |
| `min_trx_id` | `m_ids` 의 최솟값 |
| `max_trx_id` | View 생성 시점에 다음 발급될 트랜잭션 ID |
| `creator_trx_id` | View 를 만든 트랜잭션 ID 자신 |

행의 `DB_TRX_ID` 가 Read View 와 만나면 다음 규칙으로 가시성을 판단한다.

1. `DB_TRX_ID == creator_trx_id` → 자기가 쓴 거 → **보임**.
2. `DB_TRX_ID < min_trx_id` → View 생성 전 커밋된 트랜잭션 → **보임**.
3. `DB_TRX_ID >= max_trx_id` → View 생성 후 시작된 트랜잭션 → **안 보임**.
4. `min_trx_id ≤ DB_TRX_ID < max_trx_id` 이면서 `m_ids` 에 포함 → 아직 활성 → **안 보임**.
5. 위 외 → 커밋된 거 → **보임**.

안 보이면 `DB_ROLL_PTR` 따라가 Undo Log 의 이전 버전을 본다. 그 버전도 안 보이면 계속 따라감.

### 3-1. Read View 생성 시점

> 격리 수준이 Read View 의 생성 시점을 결정한다.

| 격리 수준 | Read View 생성 시점 |
|----------|---------------------|
| READ UNCOMMITTED | 안 만듦 (현재 값 그대로) |
| READ COMMITTED | 매 SELECT 마다 새로 만듦 |
| REPEATABLE READ | 트랜잭션의 첫 SELECT 에서 한 번, 끝까지 유지 |
| SERIALIZABLE | REPEATABLE READ + 모든 SELECT 가 락 획득 |

InnoDB 의 기본은 REPEATABLE READ. 한 트랜잭션 내에서 같은 SELECT 가 항상 같은 결과를 봐야 하는 이유 — Read View 가 시작 시점에 고정되기 때문.

## 4. Clustered Index 와 Secondary Index 의 MVCC

> Secondary Index 에는 `DB_TRX_ID` 가 없다 — 그래서 MVCC 판단을 위해 항상 PK 로 가서 Clustered Index 를 확인해야 한다.

```sql
SELECT name FROM users WHERE email = 'alice@x.com';
```

1. Secondary index `(email)` 에서 `alice@x.com` → PK=1001.
2. Clustered index 에서 PK=1001 의 행과 `DB_TRX_ID` 조회.
3. Read View 로 가시성 판단.
4. 안 보이면 Undo Log 체인 따라감.

여기서 질문 하나 — 그러면 Covering Index 는 MVCC 와 잘 안 맞나? 그렇다. Secondary Index 만으로 결과를 만드는 Covering 은 *현재 시점 데이터* 만 본다. InnoDB 는 이런 경우 Read Uncommitted 같은 효과를 막기 위해 추가 검증 (또는 Clustered 까지 가는 보수적 동작) 을 한다.

## 5. SELECT 의 두 종류

> 같은 `SELECT` 라도 락을 잡느냐 안 잡느냐로 갈린다.

| 형태 | 락 | 가시성 |
|------|-----|--------|
| `SELECT` (일반) | 없음 (consistent read) | Read View 기준 |
| `SELECT ... LOCK IN SHARE MODE` | S-lock | 현재 커밋된 최신 |
| `SELECT ... FOR UPDATE` | X-lock | 현재 커밋된 최신 |

`LOCK IN SHARE MODE` 와 `FOR UPDATE` 는 *Read View 를 우회* 한다. 락을 잡고 현재 최신 값을 본다. 이게 `../jpa/04-02.낙관적 비관적 락.md` 의 `PESSIMISTIC_READ`/`PESSIMISTIC_WRITE` 와 직접 대응된다.

## 6. UPDATE 의 동작

> UPDATE 는 SELECT 와 다른 가시성 규칙 — *현재 커밋된 최신* 을 본다.

REPEATABLE READ 에서 트랜잭션 A 가 시작 → 다른 트랜잭션 B 가 같은 행 UPDATE 후 커밋 → A 가 그 행 SELECT 하면 *옛 값* 을 본다 (Read View). 그러나 A 가 그 행을 UPDATE 하려 하면 *B 가 갱신한 새 값* 을 보고 갱신한다 (현재 커밋 최신).

이 동작이 *팬텀 락 + 갭 락* 등 InnoDB 의 복잡한 락 동작의 근거가 된다. 자세한 락은 `./04-02.동시성제어와 락.md`.

> 본문을 읽었다면 다음 질문에 답할 수 있어야 한다.

1. InnoDB 의 숨겨진 시스템 컬럼 셋(`DB_TRX_ID`·`DB_ROLL_PTR`·`DB_ROW_ID`)은 각각 무엇이며 MVCC 의 토대가 되는 이유는 무엇인가? — §1
2. Undo Log Chain 은 어떻게 구성되며, 롤백과 MVCC SELECT 두 곳에서 어떻게 쓰이는가? — §2
3. Read View 의 가시성 판단 규칙은 행의 `DB_TRX_ID` 를 어떻게 비교하는가? 안 보이면 어디로 가는가? — §3
4. 격리 수준에 따라 Read View 생성 시점이 어떻게 달라지며, REPEATABLE READ 에서 같은 SELECT 가 같은 결과를 보는 이유는 무엇인가? — §3-1
5. 일반 `SELECT` 와 `SELECT ... FOR UPDATE` 의 가시성 차이는 무엇이고, 후자는 무엇과 대응되는가? — §5
