# SQL 기초 — DDL, DML, 함수
---
> SQL 은 데이터를 정의하는 DDL, 다루는 DML, 가공하는 함수의 세 갈래로 갈린다. 이 셋이 SQL 의 거의 모든 일상 작업을 차지한다.

## 1. 데이터 타입과 제약

> 컬럼의 타입과 제약이 데이터의 첫 안전망 — 잘 정해두면 애플리케이션 코드가 줄어든다.

문자형은 `VARCHAR(n)` 와 `CHAR(n)` 로 나뉜다. `VARCHAR(10)` 에 5글자를 저장하면 5바이트만 쓰고, `CHAR(10)` 은 항상 10바이트를 채운다. 가변 길이가 표준 선택이고, 고정 길이가 필요한 코드 값(국가코드 `KR`)에만 `CHAR` 를 쓴다.

숫자형은 `INT`, `BIGINT`, `DECIMAL(p,s)` 가 중심이다. `DECIMAL(5,2)` 는 전체 5자리 중 소수점 2자리 — 즉 `999.99` 까지. 금액 계산처럼 부동소수점 오차를 허용하지 못하는 곳은 `DECIMAL`, 그 외는 `INT`/`BIGINT`.

날짜형은 `DATE`(날짜만), `TIME`(시간만), `DATETIME` 또는 `TIMESTAMP`(둘 다)다. 표준은 `TIMESTAMP` + UTC 저장.

제약은 다섯 가지가 거의 모든 경우를 덮는다.

| 제약 | 의미 |
|------|------|
| `NOT NULL` | NULL 금지 |
| `UNIQUE` | 중복 금지 (NULL 은 허용) |
| `PRIMARY KEY` | NOT NULL + UNIQUE, 행의 식별자 |
| `FOREIGN KEY` | 다른 테이블 PK 만 허용 |
| `DEFAULT` | NULL 시 기본값 |
| `CHECK` | 값의 범위·종류 검증 (`CHECK(gender IN ('m','f'))`) |

## 2. DDL — 테이블 정의·변경

> 테이블의 모양을 결정하는 명령들 — 한 번 정해지면 바꾸기 비싸므로 신중히.

```sql
CREATE TABLE member (
    id          INT AUTO_INCREMENT,
    user_id     VARCHAR(16) NOT NULL UNIQUE,
    user_name   VARCHAR(20),
    user_pwd    VARCHAR(16),
    gender      CHAR(1) CHECK(gender IN ('m', 'f')),
    join_date   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT pk_member PRIMARY KEY (id)
);

ALTER TABLE member ADD COLUMN phone_number VARCHAR(20);
ALTER TABLE member MODIFY phone_number VARCHAR(20) NOT NULL;
ALTER TABLE member DROP COLUMN phone_number;

DROP TABLE member;
```

운영 환경의 `ALTER` 는 락·복제 비용이 크므로 `Expand/Contract` 패턴(`../theory/01-04.트랜잭션과 격리 수준.md`)으로 점진 변경한다.

## 3. DML — 조회·삽입·수정·삭제

> CRUD 네 명령이 SQL 작업의 95% 를 차지한다.

```sql
-- SELECT
SELECT id, user_name FROM member WHERE gender = 'f' ORDER BY join_date DESC LIMIT 10;

-- INSERT
INSERT INTO member (user_id, user_name, gender) VALUES ('alice', 'Alice', 'f');

-- UPDATE
UPDATE member SET user_name = 'Alice Kim' WHERE user_id = 'alice';

-- DELETE
DELETE FROM member WHERE id = 1001;
```

여기서 질문 하나 — `UPDATE`/`DELETE` 에서 `WHERE` 를 빠뜨리면? 모든 행이 영향받는다. 운영 환경에서는 `BEGIN; UPDATE ...; -- 확인 후 COMMIT;` 패턴으로 트랜잭션 안에서 검증하고 커밋한다.

## 4. SQL 함수

> 함수는 컬럼 값을 가공하거나 검증한다 — 숫자·문자·날짜·논리·NULL 처리의 다섯 갈래.

| 분류 | 대표 함수 |
|------|-----------|
| 숫자 | `ABS`, `ROUND`, `CEIL`, `FLOOR`, `MOD`, `POWER` |
| 문자 | `UPPER`, `LOWER`, `SUBSTR`, `TRIM`, `LENGTH`, `CONCAT`, `REPLACE` |
| 날짜 | `NOW`, `CURRENT_DATE`, `DATE_FORMAT`, `DATEDIFF`, `DATE_ADD` |
| 논리 | `CASE WHEN ... THEN ... ELSE ... END` |
| NULL | `COALESCE`, `NULLIF`, `IFNULL` (MySQL), `NVL` (Oracle) |

NULL 처리 함수의 차이를 명확히 한다.

- `COALESCE(a, b, c)` — 첫 NULL 아닌 값. 표준 SQL.
- `IFNULL(a, b)` — a 가 NULL 이면 b. MySQL 만.
- `NULLIF(a, b)` — a == b 면 NULL. "0 을 NULL 로 바꾸기" 패턴에 유용.
- `CASE` — 가장 일반적, 모든 DBMS 지원.

```sql
SELECT
  user_id,
  COALESCE(phone_number, email, 'no-contact') AS contact,
  CASE
    WHEN age < 20 THEN 'teen'
    WHEN age < 60 THEN 'adult'
    ELSE 'senior'
  END AS age_group
FROM member;
```

## 5. 집계와 그룹

> `GROUP BY` 는 행을 그룹으로 묶고, 집계 함수는 그룹마다 단일 값을 만든다.

| 함수 | 의미 |
|------|------|
| `COUNT(*)` | 행 수 (NULL 포함) |
| `COUNT(col)` | NULL 제외 행 수 |
| `SUM`, `AVG`, `MIN`, `MAX` | 합·평균·최소·최대 |

```sql
SELECT gender, COUNT(*) AS cnt, AVG(age) AS avg_age
FROM member
WHERE join_date > '2026-01-01'
GROUP BY gender
HAVING COUNT(*) > 10
ORDER BY cnt DESC;
```

`WHERE` 는 그룹 전 행 필터, `HAVING` 은 그룹 후 그룹 필터. 둘이 같이 있으면 항상 `WHERE` → `GROUP BY` → `HAVING` 순.

### 5-1. 윈도우 함수 (OVER)

> 행을 그룹화하지 않고 *각 행 옆에* 집계값을 붙인다.

```sql
SELECT
  user_id,
  score,
  RANK() OVER (PARTITION BY team ORDER BY score DESC) AS rank_in_team,
  AVG(score) OVER (PARTITION BY team) AS team_avg
FROM members;
```

각 행이 자기 팀 안에서의 순위와 팀 평균을 함께 갖는다. 같은 결과를 `GROUP BY` + `JOIN` 으로 만들면 쿼리가 두세 배 길어진다.

1. `VARCHAR(10)` 과 `CHAR(10)` 의 저장 방식 차이는 무엇이고, 언제 `CHAR` 를 선택하는가? — §1
2. 금액 계산에 `DECIMAL` 을 쓰고 `FLOAT`/`DOUBLE` 을 피하는 이유는 무엇인가? — §1
3. `WHERE` 와 `HAVING` 의 차이는 무엇이며, 둘이 함께 있을 때 실행 순서는 어떻게 되는가? — §5
4. `COALESCE`, `IFNULL`, `NULLIF` 의 동작 차이는 무엇이고 이식성 측면에서 무엇을 우선하는가? — §4
5. 윈도우 함수가 `GROUP BY` + `JOIN` 조합보다 나은 경우는 어떤 상황인가? — §5-1
