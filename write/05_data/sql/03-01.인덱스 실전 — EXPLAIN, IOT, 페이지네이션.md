# 인덱스 실전 — EXPLAIN, IOT, 페이지네이션
---
> 인덱스의 이론은 `../theory/01-05.인덱스 이론.md` 가 다룬다. 본 문서는 그 이론을 MySQL/InnoDB 환경에서 *측정하고 적용하는* 실전 측만 다룬다.

## 1. EXPLAIN — 옵티마이저의 결정 읽기

> 인덱스 추가/삭제의 첫 단계는 측정 — `EXPLAIN` 이 그 측정 도구다.

```sql
EXPLAIN SELECT * FROM orders WHERE customer_id = 1001 AND status = 'pending';
```

핵심 컬럼:

| 컬럼 | 의미 | 좋은 값 |
|------|------|---------|
| `type` | 접근 방법 | `const` > `eq_ref` > `ref` > `range` > `index` > `ALL` (왼쪽일수록 좋음) |
| `key` | 실제 사용한 인덱스 | NULL 이면 풀스캔 |
| `rows` | 검사할 예상 행 수 | 작을수록 좋음 |
| `Extra` | 추가 정보 | `Using index` (Covering) 좋음, `Using filesort` `Using temporary` 나쁨 |

`type: ALL` 이 나오면 풀테이블 스캔 — 인덱스가 안 쓰이고 있다는 신호. 인덱스가 있는데도 `ALL` 이면 옵티마이저가 풀스캔이 더 싸다고 판단한 것. 데이터 분포 (`ANALYZE TABLE`) 또는 인덱스 카디널리티 확인.

### 1-1. EXPLAIN ANALYZE (실제 측정)

```sql
EXPLAIN ANALYZE SELECT ...;
```

`EXPLAIN` 은 옵티마이저의 *예상*, `EXPLAIN ANALYZE` 는 실제 실행 + 측정. 운영 환경의 정확한 비용을 알려면 후자.

## 2. Heap-Organized vs Index-Organized Table (IOT)

> 테이블이 디스크에 어떻게 저장되는가 — MySQL/InnoDB 는 IOT, PostgreSQL 은 Heap.

| 측면 | Heap-Organized (PostgreSQL) | Index-Organized (InnoDB) |
|------|------------------------------|---------------------------|
| 물리적 정렬 | 삽입 순서 (또는 임의) | PK 기준 정렬 |
| PK 조회 | 인덱스 → ROW_ID → 행 (2단계) | 인덱스 = 행 (1단계) |
| Secondary Index | row_id (또는 ctid) 가리킴 | PK 값 가리킴 |
| 풀스캔 | 빠름 | 약간 느림 (PK 인덱스 따라감) |

### 2-1. InnoDB 의 PK 와 Secondary Index

> Secondary Index 가 row_id 가 아니라 PK 를 저장한다 — 이게 PK 선택의 무게를 키운다.

```
[Secondary Index on email]
"alice@x.com" → PK=1001
"bob@x.com"   → PK=1042
"carol@x.com" → PK=1015

[Clustered Index (= 테이블)]
PK=1001 → (alice, 30, ...)
PK=1015 → (carol, 25, ...)
PK=1042 → (bob, 28, ...)
```

`SELECT * FROM users WHERE email = 'alice@x.com'` 의 흐름:

1. Secondary index 에서 `alice@x.com` 검색 → PK=1001 발견.
2. Clustered index 에서 PK=1001 검색 → 행 데이터.

PK 가 UUID 같은 큰 값이면 Secondary index 마다 그 PK 가 통째로 저장된다. PK 선택이 모든 인덱스의 크기에 영향을 준다.

## 3. 랜덤 액세스의 비용

> 인덱스에서 행을 찾는 두 번째 단계 — Secondary Index → Clustered Index 의 랜덤 IO.

```sql
SELECT * FROM orders WHERE customer_id = 1001;
```

`customer_id` 인덱스로 1000 개 행을 찾았다면, 그 1000 개의 PK 로 Clustered Index 에서 1000 번 랜덤 IO. 이 비용이 풀스캔보다 커지면 옵티마이저가 풀스캔을 선택한다 (보통 테이블의 ~20% 이상).

### 3-1. Covering Index — 랜덤 액세스 회피

> 인덱스만 보고도 결과를 만들 수 있으면 Clustered Index 접근이 사라진다.

```sql
-- 인덱스: (customer_id, status, amount)
SELECT customer_id, status, amount FROM orders WHERE customer_id = 1001;
```

`EXPLAIN Extra: Using index` 가 나오면 Covering. SELECT 절의 모든 컬럼이 인덱스에 포함되어야 함.

여기서 질문 하나 — 모든 쿼리에 Covering Index 를 만들어야 하나? 아니다. 인덱스 자체의 쓰기 비용·저장 비용이 있다. 정말 자주 호출되는 핵심 쿼리에만.

## 4. 페이지네이션 최적화

> `LIMIT 10000, 20` 같은 깊은 페이지가 느린 이유와 대안.

### 4-1. Offset 기반 (느림)

```sql
SELECT * FROM posts ORDER BY created_at DESC LIMIT 10000, 20;
```

DB 가 10020 개를 정렬해 가져온 뒤 앞 10000 개를 버린다. offset 이 커질수록 비용이 선형 증가.

### 4-2. Cursor 기반 (빠름)

```sql
-- 이전 페이지의 마지막 created_at 을 cursor 로
SELECT * FROM posts
WHERE created_at < '2026-05-23 10:00:00'
ORDER BY created_at DESC LIMIT 20;
```

인덱스를 따라 한 번에 20 개만. offset 과 무관하게 일정 비용.

### 4-3. Covering Index 와 결합

```sql
-- 인덱스: (created_at, id) — covering
SELECT id FROM posts
WHERE created_at < ?
ORDER BY created_at DESC LIMIT 20;
```

ID 만 먼저 가져와 그 ID 로 본 데이터를 조회. 깊은 페이지에서 압도적으로 빠르다.

## 5. 인덱스가 안 쓰이는 흔한 함정

> 인덱스가 있어도 쿼리가 풀스캔이 되는 패턴들.

1. **컬럼에 함수 적용** — `WHERE DATE(created_at) = '2026-05-23'` → 함수 결과 인덱스 없으면 풀스캔.
2. **컬럼에 연산** — `WHERE price * 1.1 > 100` → 같은 문제.
3. **타입 변환** — `WHERE varchar_col = 123` → varchar 가 int 로 변환되며 인덱스 무력화.
4. **앞 와일드카드** — `WHERE name LIKE '%alice'` → 인덱스 못 씀. `'alice%'` 는 가능.
5. **OR 조건** — `WHERE a = 1 OR b = 2` → 각각의 인덱스가 있어도 못 쓸 때 많음. `UNION` 으로 풀어내거나.
6. **복합 인덱스의 앞 컬럼 누락** — 인덱스 `(a, b)` 에 `WHERE b = ?` 만 있으면 못 씀.

각 함정의 해결은 *쿼리를 인덱스 친화적으로 다시 쓰는 것* 이다. 인덱스를 더 만드는 것이 답이 아닌 경우가 대부분.

## 면접에서 받을 만한 질문

> 본문을 읽었다면 다음 질문에 답할 수 있어야 한다.

1. `EXPLAIN` 의 `type` 컬럼에서 `ALL` 이 나오면 무엇을 뜻하며, 인덱스가 있는데도 `ALL` 이면 무엇을 확인하는가? — §1
2. InnoDB(IOT)의 Secondary Index 가 row_id 가 아니라 PK 를 저장하는 사실이 PK 선택에 어떤 무게를 더하는가? — §2-1
3. Covering Index 가 랜덤 액세스를 없애는 원리는 무엇이며, 모든 쿼리에 만들면 안 되는 이유는 무엇인가? — §3-1
4. Offset 기반 페이지네이션이 깊은 페이지에서 느린 이유와 Cursor 기반의 대안은 무엇인가? — §4
5. 인덱스가 있어도 풀스캔이 되는 함정 여섯 가지 중 셋 이상을 들고, 해결의 일반 원칙은 무엇인가? — §5

각 질문에 막히면 본문 해당 절을 다시 본다.
