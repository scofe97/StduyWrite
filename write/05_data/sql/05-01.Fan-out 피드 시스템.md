# Fan-out 피드 시스템
---
> SNS 피드 같이 "팔로우한 사람들의 게시글 모음" 을 어떻게 구성할 것인가 — Pull 과 Push 의 결정이 시스템 전체 비용을 결정한다.

## 1. Pull 모델 (Fan-out on Read)

> 조회 시점에 팔로우 관계를 따라 게시글을 모은다 — 페이스북이 채택한 모델.

```sql
SELECT p.*
FROM posts p
JOIN follows f ON p.author_id = f.followee_id
WHERE f.follower_id = ?
ORDER BY p.created_at DESC LIMIT 20;
```

장점:

1. **저장 공간 절약** — 게시글 한 번만 저장.
2. **쓰기 단순** — INSERT 한 줄.
3. **팔로우 변경 즉시 반영** — JOIN 으로 매번 새로 계산.

단점:

1. **조회 비용 폭발** — 1000명 팔로우 + 각자 매일 10개 = 매 피드 조회가 10000+ 행 JOIN.
2. **인기 게시글 캐싱 어려움** — 사용자마다 결과가 달라 캐싱 키가 폭증.
3. **셀럽 효과** — 한 셀럽의 게시글이 100만 명에게 조회 시 100만 번 JOIN.

## 2. Push 모델 (Fan-out on Write)

> 게시글 작성 시 모든 팔로워의 피드 테이블에 미리 복사 — 트위터가 채택한 모델.

```sql
-- 게시글 작성
INSERT INTO posts (id, author_id, content) VALUES (?, ?, ?);

-- 모든 팔로워의 피드 테이블에 사전 복사 (비동기)
INSERT INTO feed (user_id, post_id, created_at)
SELECT f.follower_id, ?, NOW() FROM follows f WHERE f.followee_id = ?;
```

조회는 단순:

```sql
SELECT p.* FROM feed f
JOIN posts p ON f.post_id = p.id
WHERE f.user_id = ?
ORDER BY f.created_at DESC LIMIT 20;
```

장점:

1. **조회 빠름** — 한 사용자의 피드 행만 본다.
2. **수평 분산 자연스러움** — `feed` 테이블을 `user_id` 로 샤딩 (`../theory/02-04.샤딩.md`).

단점:

1. **쓰기 폭발** — 100만 팔로워의 셀럽이 게시글 1개 쓰면 100만 행 INSERT.
2. **저장 공간** — 같은 게시글이 100만 행에 복제.
3. **팔로우 변경 비용** — 언팔로우 시 옛 피드 청소.

## 3. 정합성과 성능의 트레이드오프

> 두 모델은 같은 문제의 다른 답 — 셋 중 어느 것을 우선할지에 따라 갈린다.

| 우선순위 | 적합 모델 |
|----------|----------|
| 조회 성능 | Push |
| 저장 공간 | Pull |
| 쓰기 처리량 | Pull |
| 팔로우 변경 즉시성 | Pull |

여기서 질문 하나 — 두 모델을 섞으면? 그렇다, 그게 실전이다.

## 4. 하이브리드 — 셀럽 분리

> 일반 사용자는 Push, 셀럽은 Pull — 트위터의 실제 운영 방식.

규칙:

1. **팔로워 수 < 임계치** (예: 10만) → Push.
2. **팔로워 수 >= 임계치** → Pull (자기 게시글은 피드 테이블에 안 박음).
3. **피드 조회 시** → 사용자의 피드 테이블 (Push 결과) + 팔로우한 셀럽들의 게시글 (Pull) 을 머지.

```sql
-- 1. Push 결과
SELECT p.* FROM feed f JOIN posts p ON f.post_id = p.id
WHERE f.user_id = ?
ORDER BY f.created_at DESC LIMIT 20;

-- 2. Pull (이 사용자가 팔로우한 셀럽들)
SELECT p.* FROM posts p
WHERE p.author_id IN (SELECT followee_id FROM follows WHERE follower_id = ? AND followee_is_celebrity = TRUE)
ORDER BY p.created_at DESC LIMIT 20;

-- 3. 두 결과 머지 후 시간순 재정렬
```

이 구조의 효과는 두 모델의 약점을 서로 보완한다는 점이다. 셀럽이 게시글 한 번 쓸 때 100만 행 INSERT 폭발이 사라지고, 일반 사용자의 빠른 피드 조회도 유지된다.

## 5. 운영 주의점

> Fan-out 시스템은 분산 + 비동기 + 캐시가 모두 만나는 곳이라 운영 함정이 많다.

1. **순서 일관성** — 두 모델 머지 시 게시글 시간순 정확히 유지. created_at 의 시계 동기화 필수.
2. **삭제 전파** — 게시글 삭제 시 모든 피드 테이블에서 제거. 보통 soft delete + 조회 시 필터.
3. **읽기 캐시** — Redis 의 Sorted Set 으로 사용자별 피드를 캐싱 (`../redis/03-01.리더보드.md` 와 같은 패턴).
4. **백필** — 새 팔로우 시점에 옛 게시글 일부를 피드에 박을지 결정.

## 면접에서 받을 만한 질문

> 본문을 읽었다면 다음 질문에 답할 수 있어야 한다.

1. Pull 모델(Fan-out on Read)의 조회 비용이 폭발하는 이유와 셀럽 효과는 무엇인가? — §1
2. Push 모델(Fan-out on Write)의 쓰기가 폭발하는 상황은 무엇이며, 수평 분산이 자연스러운 이유는 무엇인가? — §2
3. 조회 성능·저장 공간·쓰기 처리량·팔로우 변경 즉시성 각각에 어느 모델이 유리한가? — §3
4. 하이브리드(셀럽 분리) 모델은 팔로워 수 임계치로 무엇을 가르며, 조회 시 두 결과를 어떻게 합치는가? — §4
5. Fan-out 시스템의 운영 함정 네 가지(순서·삭제·캐시·백필) 중 셋 이상을 들 수 있는가? — §5

각 질문에 막히면 본문 해당 절을 다시 본다.
