# JOIN, SUBQUERY, CTE
---
> 여러 테이블의 데이터를 한 결과로 모으는 세 가지 방법 — JOIN(횡 결합), SUBQUERY(쿼리 안의 쿼리), CTE(`WITH`, 임시 명명 결과). 같은 결과를 셋 다로 만들 수 있지만 가독성·성능·재귀 지원이 다르다.

## 1. JOIN — 횡 결합

> 두 테이블의 행을 조건으로 묶는다 — 결과의 폭은 두 테이블의 컬럼 수 합이다.

| 종류 | 동작 |
|------|------|
| `INNER JOIN` | 양쪽 모두에 매칭되는 행만 |
| `LEFT OUTER JOIN` | 왼쪽 전체 + 오른쪽 매칭 (없으면 NULL) |
| `RIGHT OUTER JOIN` | 반대 |
| `FULL OUTER JOIN` | 양쪽 전체 (MySQL 미지원, `UNION` 으로 우회) |
| `CROSS JOIN` | 카르테시안 곱 (모든 조합) |
| `SELF JOIN` | 같은 테이블끼리 (계층 데이터·중복 검사) |

```sql
-- INNER: 주문 + 고객
SELECT o.id, o.amount, c.name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id;

-- LEFT: 모든 고객 + (있으면) 주문 수
SELECT c.id, c.name, COUNT(o.id) AS order_count
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name;

-- SELF: 같은 부서 동료 찾기
SELECT a.name AS me, b.name AS colleague
FROM employees a
JOIN employees b ON a.dept_id = b.dept_id AND a.id <> b.id;
```

여기서 질문 하나 — `LEFT JOIN` 후 `WHERE right_table.col = X` 를 쓰면? `INNER JOIN` 과 같아진다. NULL 행이 WHERE 에서 모두 걸러지기 때문. 진짜 LEFT 의도였으면 조건을 `ON` 절로 옮긴다.

## 2. UNION — 종 결합

> JOIN 이 컬럼을 옆으로 붙인다면, UNION 은 행을 위아래로 붙인다.

```sql
SELECT id, name FROM active_users
UNION         -- 중복 제거
SELECT id, name FROM archived_users;

SELECT id, name FROM active_users
UNION ALL     -- 중복 포함 (빠름)
SELECT id, name FROM archived_users;
```

`UNION` 은 정렬·중복제거 비용이 있다. 중복이 없음을 알면 `UNION ALL` 이 거의 항상 더 빠르다.

## 3. SUBQUERY — 쿼리 안의 쿼리

> 한 쿼리의 결과를 다른 쿼리의 입력으로 — 위치에 따라 세 종류.

| 위치 | 종류 | 동작 |
|------|------|------|
| `SELECT` 절 | 스칼라 서브쿼리 | 행당 단일 값 반환 |
| `FROM` 절 | 인라인 뷰 | 가상 테이블처럼 사용 |
| `WHERE` 절 | 일반 서브쿼리 | 비교·존재 검사 |

```sql
-- 스칼라
SELECT
  c.name,
  (SELECT COUNT(*) FROM orders o WHERE o.customer_id = c.id) AS order_count
FROM customers c;

-- 인라인 뷰
SELECT t.dept_id, t.avg_age
FROM (SELECT dept_id, AVG(age) AS avg_age FROM employees GROUP BY dept_id) t
WHERE t.avg_age > 35;

-- WHERE 절 — Nested Subquery
SELECT * FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

SELECT * FROM employees
WHERE dept_id IN (SELECT id FROM departments WHERE budget > 1000000);
```

서브쿼리는 가독성은 좋지만 옵티마이저가 풀어내지 못하는 경우가 있다. 같은 결과를 JOIN 으로 만들 수 있으면 JOIN 이 보통 더 빠르다.

### 3-1. 상관 서브쿼리

> 바깥 쿼리의 컬럼을 참조하는 서브쿼리 — 행마다 다시 실행된다.

```sql
SELECT e.id, e.salary,
  (SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id) AS dept_avg
FROM employees e;
```

문법은 자연스럽지만 N 행이면 N 번 실행. 큰 테이블에서는 윈도우 함수(`AVG OVER (PARTITION BY ...)`) 로 풀면 한 번 스캔으로 끝난다.

## 4. VIEW — 저장된 쿼리

> 쿼리에 이름을 붙여 테이블처럼 다룬다 — 복잡한 JOIN 을 한 번 정의해 두고 재사용.

```sql
CREATE VIEW active_customer_orders AS
SELECT c.id, c.name, COUNT(o.id) AS order_count
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE c.is_active = TRUE
GROUP BY c.id, c.name;

SELECT * FROM active_customer_orders WHERE order_count > 5;
```

VIEW 는 매 조회 시 원본 쿼리를 실행한다. 결과를 저장하지 않는다 (그건 Materialized View — PostgreSQL 에서 별도 명령). 성능 캐싱이 필요하면 별도 정책 (Materialized View, 캐시 테이블) 을 검토한다.

## 5. CTE (`WITH`) — 명명된 임시 결과

> 쿼리를 단계로 쪼개 가독성을 높인다 — SUBQUERY 의 가독성 한계를 넘는 도구.

```sql
WITH dept_avg AS (
  SELECT dept_id, AVG(salary) AS avg_salary
  FROM employees
  GROUP BY dept_id
),
high_paid AS (
  SELECT e.id, e.name, e.dept_id, e.salary, d.avg_salary
  FROM employees e
  JOIN dept_avg d ON e.dept_id = d.dept_id
  WHERE e.salary > d.avg_salary
)
SELECT * FROM high_paid;
```

`WITH` 가 만든 임시 이름은 *한 쿼리 안에서만* 유효하다. 같은 결과를 SUBQUERY 로 짜면 가독성이 급격히 떨어진다.

### 5-1. 재귀 CTE (`WITH RECURSIVE`)

> 자기 자신을 참조해 트리·그래프 데이터를 푸는 표준 도구.

```sql
WITH RECURSIVE org_tree AS (
  -- Anchor: 최상위
  SELECT id, name, parent_id, 0 AS depth
  FROM employees
  WHERE parent_id IS NULL

  UNION ALL

  -- Recursive: 자식들
  SELECT e.id, e.name, e.parent_id, t.depth + 1
  FROM employees e
  JOIN org_tree t ON e.parent_id = t.id
)
SELECT * FROM org_tree ORDER BY depth, name;
```

재귀 CTE 의 실행은 (1) Anchor 1회 실행, (2) Recursive 부분을 결과가 비워질 때까지 반복. 무한 루프 방지를 위해 `depth < N` 같은 종료 조건을 두는 것이 안전하다.

## 면접에서 받을 만한 질문

> 본문을 읽었다면 다음 질문에 답할 수 있어야 한다.

1. `LEFT JOIN` 후 오른쪽 테이블 컬럼을 `WHERE` 에서 비교하면 왜 `INNER JOIN` 과 같아지는가? 진짜 LEFT 의도라면 조건을 어디로 옮기는가? — §1
2. `UNION` 과 `UNION ALL` 의 차이는 무엇이고, 언제 `UNION ALL` 이 거의 항상 더 빠른가? — §2
3. 상관 서브쿼리가 큰 테이블에서 느린 이유는 무엇이며, 어떤 도구로 한 번 스캔으로 바꿀 수 있는가? — §3-1
4. VIEW 는 결과를 저장하는가? 성능 캐싱이 필요할 때 무엇을 검토하는가? — §4
5. 재귀 CTE 의 실행 단계(Anchor·Recursive)는 어떻게 동작하며 무한 루프를 어떻게 막는가? — §5-1

각 질문에 막히면 본문 해당 절을 다시 본다.
