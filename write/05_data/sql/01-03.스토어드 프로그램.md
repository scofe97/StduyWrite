# 스토어드 프로그램 (프로시저, 함수, 커서, 트리거)
---
> 비즈니스 로직을 DB 안에 두는 도구들 — 강력하지만 *애플리케이션과 DB 의 경계* 를 흐리므로 신중히 도입한다.

## 1. 도입 결정의 일반 원칙

> "이 로직을 DB 안에 둘 가치가 있는가" 의 답은 보통 "아니오" 다 — 다음 세 경우에만 "예".

1. **대량 데이터 조작** — 수백만 행 변환을 애플리케이션이 가져갔다 다시 보내는 비용이 큰 경우.
2. **DB 만이 알 수 있는 정합성** — 두 테이블 간 트리거 기반 자동 동기화.
3. **레거시 통합** — 옛 시스템과의 인터페이스가 SP 호출로 정해져 있는 경우.

위 셋 외에는 애플리케이션 코드에 두는 것이 디버깅·테스트·버전 관리 측면에서 압도적이다.

## 2. 스토어드 프로시저 (Stored Procedure)

> 명령들의 묶음을 이름으로 호출 — 반환값은 없거나 OUT 파라미터로.

```sql
DELIMITER //
CREATE PROCEDURE transfer_funds(
    IN from_id INT,
    IN to_id INT,
    IN amount DECIMAL(10,2)
)
BEGIN
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        ROLLBACK;
        RESIGNAL;
    END;

    START TRANSACTION;
    UPDATE accounts SET balance = balance - amount WHERE id = from_id;
    UPDATE accounts SET balance = balance + amount WHERE id = to_id;
    COMMIT;
END //
DELIMITER ;

CALL transfer_funds(1, 2, 100.00);
```

호출은 `CALL`. 트랜잭션·예외 처리·반복문(`WHILE`, `LOOP`)이 모두 가능.

## 3. 스토어드 함수 (Stored Function)

> 프로시저와 비슷하지만 *값을 반환* 하고 *SELECT 안에서 호출 가능*.

```sql
DELIMITER //
CREATE FUNCTION age_group(birth_date DATE) RETURNS VARCHAR(10)
DETERMINISTIC
BEGIN
    DECLARE age INT;
    SET age = TIMESTAMPDIFF(YEAR, birth_date, CURDATE());
    IF age < 20 THEN RETURN 'teen';
    ELSEIF age < 60 THEN RETURN 'adult';
    ELSE RETURN 'senior';
    END IF;
END //
DELIMITER ;

SELECT name, age_group(birth_date) FROM users;
```

`DETERMINISTIC` 은 "같은 입력에 항상 같은 출력" 의 명시 — 옵티마이저가 결과를 캐시할 수 있다.

## 4. 커서 (CURSOR) — 행 단위 처리

> 결과 집합을 한 행씩 처리 — 일반 SQL 의 *집합 단위 처리* 와 반대.

```sql
DELIMITER //
CREATE PROCEDURE process_orders()
BEGIN
    DECLARE done INT DEFAULT FALSE;
    DECLARE oid INT;
    DECLARE cur CURSOR FOR SELECT id FROM orders WHERE status = 'pending';
    DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;

    OPEN cur;
    read_loop: LOOP
        FETCH cur INTO oid;
        IF done THEN LEAVE read_loop; END IF;
        -- 각 주문 처리
    END LOOP;
    CLOSE cur;
END //
DELIMITER ;
```

여기서 질문 하나 — 커서를 써야만 풀리는 문제가 얼마나 있나? 거의 없다. 집합 단위 SQL (`UPDATE ... WHERE ...`) 이 대부분 풀어낸다. 커서는 *행마다 외부 시스템 호출* 같은 진짜 필요한 경우에만.

## 5. 트리거 (TRIGGER)

> 특정 이벤트(INSERT/UPDATE/DELETE) 가 일어나면 자동으로 실행되는 로직.

```sql
CREATE TRIGGER audit_member_changes
AFTER UPDATE ON member
FOR EACH ROW
BEGIN
    INSERT INTO member_audit (member_id, old_name, new_name, changed_at)
    VALUES (OLD.id, OLD.user_name, NEW.user_name, NOW());
END;
```

`OLD`, `NEW` 키워드로 변경 전/후 값에 접근. `BEFORE INSERT` 같은 시점 지정도 가능.

### 5-1. 트리거의 위험

> 트리거는 "자동 실행" 이 장점이자 함정이다.

1. **추적 어려움** — UPDATE 한 줄이 트리거 → 다른 트리거 → 다시 다른 트리거 연쇄.
2. **디버깅 비용** — 애플리케이션 로그에는 트리거가 안 보임.
3. **이식성** — DBMS 별 문법 차이.

대안은 애플리케이션 측의 이벤트 발행(`../jpa/04-01.스프링 트랜잭션.md` 또는 Outbox 패턴, `../theory/03-03.스트리밍 시스템 철학.md`)이다. 트리거의 자동성을 *코드가 명시적으로 발행* 하는 형태로 옮긴다.
