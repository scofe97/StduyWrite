# 자바 예외와 SQL — 체크/언체크와 throws 의존 전파

---

> JDBC 가 던지는 `SQLException` 은 체크 예외입니다. 체크 예외는 *메서드 시그니처에 의존을 끌고 다닌다* 는 점에서 데이터 접근 계층의 가장 큰 설계 압박입니다. 본 노트는 그 메커니즘과 *왜 런타임으로 변환해야 하는지* 를 정리합니다.

## 1. 예외 계층

> 자바 예외는 `Throwable` 을 정점으로 두 갈래로 갈립니다.

```
Object
  └── Throwable
        ├── Error          — 시스템 오류 (OOM 등), 잡지 않음
        └── Exception      — 애플리케이션 영역
              ├── (체크 예외)  IOException, SQLException, ...
              └── RuntimeException  — 언체크 예외
                    ├── NullPointerException
                    ├── IllegalArgumentException
                    └── ...
```

세 그룹의 의미가 다릅니다.

- `Error` 는 *복구 불가능* 한 시스템 실패입니다. `OutOfMemoryError`, `StackOverflowError` 가 여기 속합니다. 애플리케이션 코드에서 잡으면 안 됩니다.
- `Exception` 의 직접 자식은 *체크 예외* 입니다. 컴파일러가 `throws` 또는 `try-catch` 를 강제합니다.
- `RuntimeException` 과 그 자식은 *언체크 예외* (런타임 예외) 입니다. 컴파일러가 강제하지 않습니다.

상위 예외를 `catch` 로 잡으면 자식까지 같이 잡힙니다. 그래서 `Throwable` 이나 `Exception` 같은 광범위한 잡기는 `Error` 까지 같이 삼킬 위험이 있어 보통 피합니다.

## 2. 예외의 두 기본 규칙

> 예외는 *처리하거나 던지거나* 둘 중 하나입니다.

```
호출자 ─→ 메서드
              │
              ├── 예외 발생
              │       ├── 잡아서 처리 (try-catch)  →  이후 정상 흐름
              │       └── 처리 못 함 → throws 로 던짐
              │
              └── 호출자가 다시 같은 선택을 합니다.
```

규칙 두 가지를 기억합니다.

1. 잡거나 던지거나 둘 중 하나입니다. 무시는 안 됩니다.
2. *지정한 타입* 만 잡는 게 아니라 *그 자식* 까지 같이 잡힙니다. `throws` 도 마찬가지입니다.

이 두 규칙이 결합하면, *체크 예외는 시그니처를 통해 호출 체인을 따라 올라간다* 는 결과가 나옵니다.

## 3. 체크 예외의 의존 전파 문제

> `SQLException` 같은 체크 예외는 *호출 체인 전체의 시그니처* 에 박힙니다.

다음 시나리오를 봅니다.

```java
public class MemberRepository {
    public Member findById(Long id) throws SQLException {
        // JDBC 호출
    }
}

public class MemberService {
    public Member find(Long id) throws SQLException {  // 강제됨
        return memberRepository.findById(id);
    }
}

public class MemberController {
    public Response get(Long id) throws SQLException {  // 강제됨
        return memberService.find(id);
    }
}
```

`MemberRepository.findById` 가 `throws SQLException` 을 선언한 순간, 서비스도 컨트롤러도 같은 선언을 강제받습니다. 컴파일러가 그렇게 요구하기 때문입니다.

문제는 두 가지입니다.

- **인터페이스 오염**: 서비스 인터페이스에 `SQLException` 이 박힙니다. `SQLException` 은 *JDBC 라는 특정 기술* 의 어휘입니다. JPA 로 갈아끼우려면 인터페이스 시그니처 자체를 바꿔야 합니다.
- **무의미한 throws 반복**: 컨트롤러까지 따라 올라간 `SQLException` 은 *애초에 처리할 수 없는 예외* 입니다. 컨트롤러는 데이터베이스 오류를 사용자에게 의미 있게 보여줄 수 없습니다. 그런데도 시그니처에 매번 적어야 합니다.

`NetworkClient.connect()` 가 `ConnectException` 을 던진다면 같은 일이 일어납니다. 서비스 시그니처는 점점 길어집니다.

```java
public Member doWork() throws SQLException, ConnectException, ... { ... }
```

이런 시그니처는 *변경에 약한 코드* 의 신호입니다.

## 4. 체크 vs 언체크 — 언제 무엇을 쓰는가

> 기본은 *언체크 예외* 입니다. 체크 예외는 *반드시 잡아서 처리해야 하는 비즈니스 사건* 에만 씁니다.

강의에서 정리한 가이드는 다음과 같습니다.

- 기본적으로 *언체크 (런타임) 예외* 를 씁니다.
- *체크 예외* 는 *비즈니스 로직상 의도적으로 던지는 예외* 에만 씁니다.

체크 예외가 어울리는 자리는 *호출자가 반드시 다른 흐름을 타야 하는* 사건입니다.

- 계좌 이체 실패 (잔액 부족)
- 결제 시 포인트 부족
- 로그인 ID/비밀번호 불일치

이런 사건은 호출자가 *예외 처리를 깜빡하면* 큰 문제가 됩니다. 컴파일러가 강제해 주는 안전망이 의미를 가집니다.

반면 `SQLException` 처럼 *복구 불가능하고 호출자가 할 수 있는 게 없는* 예외는 언체크로 풀어야 합니다. 강의 6장의 `DataAccessException` 추상화가 정확히 이 변환을 다룹니다. 본 묶음의 [03-01.스프링 예외 추상화](03-01.스프링%20예외%20추상화.md) 가 이어 받습니다.

## 5. 예외 변환의 핵심 패턴

> 체크 예외를 *리포지토리 안에서* 런타임으로 변환해 던지면, 호출 체인이 깨끗해집니다.

```java
public class JdbcMemberRepository implements MemberRepository {
    public Member findById(Long id) {  // throws 없음
        try {
            // JDBC 호출
        } catch (SQLException e) {
            throw new MyDbException(e);  // 런타임 예외로 변환
        }
    }
}
```

변환의 두 원칙입니다.

- **원본 예외를 인자로 넘깁니다** (`new MyDbException(e)`). 스택 트레이스가 살아 있어야 디버깅이 가능합니다. 인자 없이 `new MyDbException()` 만 던지면 원인이 사라집니다.
- **변환된 예외는 *기술 중립적* 인 추상 타입이어야 합니다**. 직접 정의한 `MyDbException` 보다 스프링이 제공하는 `DataAccessException` 계층이 권장됩니다 ([03-01](03-01.스프링%20예외%20추상화.md)).

서비스 인터페이스의 시그니처는 단순해집니다.

```java
public interface MemberRepository {
    Member findById(Long id);  // throws 없음 — 기술 중립
}
```

JDBC 든 JPA 든 MyBatis 든, 구현이 달라져도 이 인터페이스는 그대로입니다.

## 6. 정리

체크 예외와 언체크 예외의 핵심 차이는 *호출 체인의 시그니처에 영향을 주는가* 입니다.

- 체크 예외는 *컴파일러가 처리를 강제* 하지만, *시그니처에 의존을 끌고 다닙니다*.
- 언체크 예외는 *강제는 없지만 시그니처가 깨끗* 합니다.

데이터 접근 계층은 *복구 불가능한 예외* 가 대부분이라 언체크가 자연스럽습니다. 스프링이 `DataAccessException` 계층으로 변환을 자동화한 이유가 여기 있습니다 — [03-01](03-01.스프링%20예외%20추상화.md) 가 그 메커니즘을 다룹니다.
