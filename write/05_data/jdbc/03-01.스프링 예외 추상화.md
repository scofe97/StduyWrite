# 스프링 예외 추상화 (SQLExceptionTranslator, DataAccessException)

---

> 스프링은 JDBC `SQLException` 같은 *특정 기술 종속 체크 예외* 를 `DataAccessException` 계층의 *런타임 예외* 로 변환합니다. 이 변환 덕분에 서비스 계층이 JDBC를 모르고도 데이터 접근 실패를 다룰 수 있습니다.

## 1. 왜 추상화가 필요한가

> 서비스 계층이 `SQLException` 을 throws 절에 들고 다니면, JDBC 를 JPA 로 갈아낄 때 서비스 시그니처까지 바꿔야 합니다.

리포지토리에서 JDBC `SQLException` 이 던져진다고 가정합니다. 서비스가 이를 throws 로 받으면 다음 일이 벌어집니다.

1. 서비스 인터페이스에 `throws SQLException` 이 박힙니다.
2. 인터페이스가 *JDBC 라는 특정 기술* 에 오염됩니다.
3. JPA·MyBatis 로 갈아끼우려면 인터페이스 자체를 수정해야 합니다.

근본 원인은 `SQLException` 이 *체크 예외* 라는 점에 있습니다. 체크 예외는 메서드 시그니처에 의존을 끌고 다니기 때문에, 구현 기술 변경에 인터페이스가 흔들립니다. 자세한 체크/언체크 비교는 1편 5장 (자바 예외 이해) 의 별도 노트에서 다룰 예정입니다.

해결의 방향은 두 단계입니다.

- 리포지토리 안에서 `SQLException` 을 *런타임 예외* 로 변환해서 던집니다.
- 변환된 예외는 *DB 기술과 무관한 추상 타입* 이어야 합니다. 그래야 서비스가 어떤 예외인지 모르고도 무시할 수 있습니다.

## 2. 스프링 데이터 접근 예외 계층

> 스프링은 `DataAccessException` 을 최상위로 하는 런타임 예외 계층을 제공합니다.

```
RuntimeException
    └── DataAccessException
            ├── NonTransientDataAccessException  (재시도해도 실패)
            │       ├── BadSqlGrammarException
            │       ├── DataIntegrityViolationException
            │       │       └── DuplicateKeyException
            │       └── ...
            └── TransientDataAccessException  (재시도하면 성공 가능)
                    ├── QueryTimeoutException
                    ├── PessimisticLockingFailureException
                    └── ...
```

두 갈래의 의미가 다릅니다.

- `NonTransient` 는 *일시적이지 않은* 실패를 말합니다. 같은 SQL을 다시 실행해도 같은 결과입니다. SQL 문법 오류, 유니크 제약 위배가 여기 속합니다.
- `Transient` 는 *일시적인* 실패를 말합니다. 락 경합이나 쿼리 타임아웃은 재시도하면 성공할 수도 있습니다.

이 분기 덕분에 호출자는 *재시도해도 되는 실패인지* 한 단계 위에서 판단할 수 있습니다. 재시도 로직을 짤 때 `if (ex instanceof TransientDataAccessException)` 로 가르면 됩니다.

## 3. SQLExceptionTranslator — 변환의 실제

> `SQLException.getErrorCode()` 의 DB 종속 코드를 받아 적절한 `DataAccessException` 하위 타입으로 변환합니다.

직접 변환을 하려면 다음과 같은 코드가 됩니다.

```java
@Test
void sqlExceptionErrorCode() {
    String sql = "select bad grammar";
    try {
        Connection con = dataSource.getConnection();
        PreparedStatement stmt = con.prepareStatement(sql);
        stmt.executeQuery();
    } catch (SQLException e) {
        int errorCode = e.getErrorCode();
        // H2 의 경우 42122
        log.info("errorCode={}", errorCode);
    }
}
```

매번 DB 의 에러 코드를 보고 `if-else` 로 분기하는 일은 비현실적입니다. DB 가 바뀌면 코드도 바뀝니다. 스프링은 이 분기를 대신해주는 `SQLExceptionTranslator` 를 제공합니다.

```java
@Test
void exceptionTranslator() {
    String sql = "select bad grammar";
    try {
        Connection con = dataSource.getConnection();
        PreparedStatement stmt = con.prepareStatement(sql);
        stmt.executeQuery();
    } catch (SQLException e) {
        SQLExceptionTranslator exTranslator =
                new SQLErrorCodeSQLExceptionTranslator(dataSource);
        DataAccessException resultEx =
                exTranslator.translate("select", sql, e);
        // resultEx 의 실제 타입은 BadSqlGrammarException
        assertThat(resultEx.getClass()).isEqualTo(BadSqlGrammarException.class);
    }
}
```

`translate()` 의 인자 셋은 다음과 같습니다.

- 첫 인자: 사람이 읽을 설명. 로그·예외 메시지에 박힙니다.
- 둘째 인자: 실행한 SQL. 추적용입니다.
- 셋째 인자: 잡힌 원본 `SQLException`.

반환 타입은 추상 `DataAccessException` 이지만, 실제 인스턴스는 구체 타입 (`BadSqlGrammarException` 등) 입니다.

## 4. sql-error-codes.xml — DB 별 코드 매핑

> 어떤 에러 코드를 어떤 예외로 변환할지는 XML 한 곳에 모여 있습니다.

`org.springframework.jdbc.support.sql-error-codes.xml` 의 구조는 다음과 같습니다.

```xml
<bean id="H2" class="org.springframework.jdbc.support.SQLErrorCodes">
    <property name="badSqlGrammarCodes">
        <value>42000,42001,42101,42102,42111,42112,42121,42122,42132</value>
    </property>
    <property name="duplicateKeyCodes">
        <value>23001,23505</value>
    </property>
</bean>

<bean id="MySQL" class="org.springframework.jdbc.support.SQLErrorCodes">
    <property name="badSqlGrammarCodes">
        <value>1054,1064,1146</value>
    </property>
    <property name="duplicateKeyCodes">
        <value>1062</value>
    </property>
</bean>
```

이 파일을 열어 보면 H2·MySQL·Oracle·PostgreSQL 같은 주요 DB 10여 종이 미리 등록되어 있습니다. 새로 추가된 DB 라면 같은 형식으로 등록해서 확장할 수도 있습니다. 다만 실무에서 직접 손댈 일은 드뭅니다.

## 5. 서비스 계층에서의 변화

> 추상 예외 도입 전후에 서비스 코드 시그니처가 어떻게 달라지는지 비교합니다.

추상화 *전* — 서비스가 JDBC를 압니다.

```java
public class MemberService {
    public void bizLogic(...) throws SQLException {  // JDBC 의존
        memberRepository.update(...);
    }
}
```

추상화 *후* — 서비스는 예외 자체를 명시하지 않습니다.

```java
public class MemberService {
    public void bizLogic(...) {  // 런타임 예외 — throws 불필요
        memberRepository.update(...);
    }
}
```

리포지토리는 내부에서 `SQLExceptionTranslator` 를 통해 변환된 `DataAccessException` 을 던집니다. 서비스가 그 예외를 *반드시* 다뤄야 할 의무는 없으므로, 무시하고 더 위 계층 (글로벌 `@ControllerAdvice` 같은 자리) 으로 흘려보낼 수 있습니다.

## 6. JdbcTemplate 과의 관계

> `JdbcTemplate` 은 이 변환을 자동으로 합니다.

`JdbcTemplate.queryForObject()` 같은 메서드를 호출하면, 내부에서 `SQLException` 발생 시 `SQLExceptionTranslator.translate()` 가 자동 호출됩니다. 호출자는 `DataAccessException` 만 받습니다. 변환을 직접 신경 쓰지 않아도 되는 구조입니다.

`JdbcTemplate` 의 이 동작 덕분에 *예외 변환 코드를 매 메서드에 반복하지 않아도* 됩니다. 자세한 `JdbcTemplate` 사용은 [`./02-01.JdbcTemplate.md`](./02-01.JdbcTemplate.md) 가 다룹니다.

## 7. 정리

스프링이 데이터 접근 예외 추상화로 푼 문제는 세 가지입니다.

- `SQLException` 같은 체크 예외가 서비스 인터페이스를 오염시키는 문제를 *런타임 예외 변환* 으로 끊어냅니다.
- DB 마다 다른 에러 코드를 *XML 매핑* 으로 흡수해 호출자가 코드를 직접 분기하지 않습니다.
- `Transient` 와 `NonTransient` 의 두 갈래로 *재시도 판단* 의 자리를 마련합니다.

JPA 를 쓰면 Hibernate 내부 예외도 같은 `DataAccessException` 계층으로 변환됩니다. 서비스 코드는 JDBC·JPA 어느 쪽이든 같은 예외 어휘로 다룰 수 있습니다.
