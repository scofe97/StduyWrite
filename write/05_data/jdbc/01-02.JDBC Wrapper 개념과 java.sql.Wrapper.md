# JDBC Wrapper 개념과 java.sql.Wrapper
---
> 순수 JDBC는 한 줄짜리 쿼리 주변에 자원 관리 코드가 30줄 가까이 따라붙습니다. wrapper는 이 반복을 감춰 주는 계층인데, "wrapper"라는 한 단어가 넓은 의미(추상화 계층)와 좁은 의미(`java.sql.Wrapper` 인터페이스)로 갈립니다. 이 문서는 둘을 구분하고, 실무에서 wrapper가 왜 여러 겹 쌓이는지를 정리합니다.



## 1. 왜 wrapper가 필요한가

> 순수 JDBC로 사용자 한 명을 조회한다고 가정하면, 비즈니스 로직은 사실상 "SELECT name FROM users WHERE id = ?" 한 줄이지만 그 주변에 30줄 가까운 부속 코드가 따라붙습니다.

```java
// 1. 자원 변수 선언 — finally에서 close() 하려면 try 바깥 스코프 필요
//    null 초기화: 위쪽 라인이 예외 던지면 일부 변수는 미할당 상태
Connection conn = null;
PreparedStatement pstmt = null;
ResultSet rs = null;

try {
    // 2. 커넥션 획득 — 실제 TCP 소켓 열고 인증
    conn = DriverManager.getConnection(URL, USER, PASSWORD);

    // 3. PreparedStatement 준비 — SQL 인젝션 방지 + 실행 계획 캐싱
    pstmt = conn.prepareStatement("SELECT name FROM users WHERE id = ?");

    // 4. 파라미터 바인딩 (인덱스는 1부터)
    pstmt.setInt(1, 10);

    // 5. 쿼리 실행 — ResultSet은 DB 커서 핸들, 반드시 close 필요
    rs = pstmt.executeQuery();

    // 6. 결과 읽기 — 초기 커서는 첫 행 앞, next()로 한 칸 이동
    if (rs.next()) {
        String name = rs.getString("name");
        // 비즈니스 로직
    }

} catch (SQLException e) {
    // SQLException은 checked — 반드시 처리 또는 throws
    logger.error("DB 에러 발생", e);

} finally {
    // 자원 해제: 역순(rs→pstmt→conn) + null 체크 + close 예외 격리
    // → try-with-resources(Java 7+)가 이 전체를 자동화함
    if (rs != null)    try { rs.close();    } catch (SQLException e) {}
    if (pstmt != null) try { pstmt.close(); } catch (SQLException e) {}
    if (conn != null)  try { conn.close();  } catch (SQLException e) {}
}
```

문제는 단순히 길다는 게 아니라 누락 시 시스템이 죽는다는 점입니다. `finally`에서 `close()`를 빠뜨리거나 예외가 끼어들어 자원이 안 닫히면 커넥션이 풀에 반환되지 않고, 누적되다 풀이 고갈되면 애플리케이션 전체가 멈춥니다. MyBatis나 JdbcTemplate을 쓰는 입장에선 거의 잊고 있는 위험인데, 그 잊을 수 있음이 바로 wrapper가 만들어준 결과입니다.



## 2. "Wrapper"라는 단어의 두 가지 용법

> 같은 단어가 넓은 의미와 좁은 의미로 갈리므로 먼저 구분합니다.

### 넓은 의미 — 디자인 패턴·추상화 계층으로서의 wrapper

JDBC의 반복 코드를 캡슐화해서 사용을 편하게 만든 모든 라이브러리·프레임워크를 통칭합니다. Spring `JdbcTemplate`, Apache Commons DbUtils, MyBatis, JPA, HikariCP가 다 여기에 들어갑니다. 각 계층이 아래 계층을 wrapping하는 구조입니다.

```java
// JdbcTemplate — 위 30줄이 이것 한 줄로 압축됨
String name = jdbcTemplate.queryForObject(
    "SELECT name FROM users WHERE id = ?",
    String.class,
    10
);
```

### 좁은 의미 — `java.sql.Wrapper` 인터페이스 (JDBC 4.0+)

JDBC 표준이 직접 정의한 인터페이스입니다. `Connection`, `Statement`, `ResultSet`, `DataSource`가 전부 이걸 상속합니다.

```java
public interface Wrapper {
    <T> T unwrap(Class<T> iface) throws SQLException;
    boolean isWrapperFor(Class<?> iface) throws SQLException;
}
```

HikariCP가 `Connection`을 감싸 `HikariProxyConnection`을 돌려주는데, 가끔 MySQL의 `JdbcConnection`이나 Oracle의 LOB API처럼 벤더 특화 기능에 직접 닿아야 할 때가 있습니다. 단순 캐스팅은 프록시 한 겹밖에 못 벗기지만 `unwrap()`은 여러 겹을 재귀적으로 뚫고 들어갑니다.

```java
if (connection.isWrapperFor(com.mysql.cj.jdbc.JdbcConnection.class)) {
    var mysqlConn = connection.unwrap(com.mysql.cj.jdbc.JdbcConnection.class);
    // 벤더 특화 API
}
```



## 3. 대표 Wrapper 역할

> 실무에서는 한 가지 관심사만 책임지는 wrapper가 여러 겹 쌓입니다.

| 계층 | 라이브러리 | 감추는 것 | 환경에서의 위치 |
|------|-----------|----------|----------------|
| ORM / SQL Mapper | MyBatis, JPA | SQL ↔ 객체 매핑 | 현재 주력 |
| Template | JdbcTemplate, DbUtils | try-finally, 예외 변환 | MyBatis 밑에 깔림 |
| Pool | HikariCP | close() = 풀 반환, 재사용 | Spring Boot 기본 |
| 관측 | p6spy, datasource-proxy | SQL 로깅, 슬로우쿼리 감지 | 디버깅용으로 끼움 |
| Spring proxy | `LazyConnectionDataSourceProxy`, `TransactionAwareDataSourceProxy` | 커넥션 획득 지연, 트랜잭션 동기화 | `@Transactional`이 의존 |

`MyBatis → Spring TransactionManager → HikariCP → MySQL Driver`처럼 wrapper가 4~5겹 쌓이는 게 정상입니다. 각 층이 한 가지 관심사(SQL 매핑, 트랜잭션, 풀링, 프로토콜)만 책임지도록 분리된 결과이고, `unwrap()`은 이 양파를 안전하게 까기 위한 표준 도구입니다.
