# 커넥션 풀과 DataSource
---
> 매 요청마다 DB 커넥션을 새로 만들면 TCP·인증·세션 설정의 비용이 ms 단위로 누적됩니다. 커넥션 풀은 이 비용을 한 번에 박제하고 재사용합니다.

## 0. JDBC 표준과 라이프사이클

> 풀과 DataSource 의 자리를 알려면 *JDBC 가 무엇을 표준화했는지* 부터 짚어야 합니다.

JDBC 는 1997년에 등장한 자바의 표준 데이터 접근 API입니다. 표준화하기 전에는 두 가지 문제가 있었습니다.

1. DB 종류가 바뀌면 *애플리케이션 코드까지* 바꿔야 했습니다.
2. 개발자는 DB 마다 다른 *커넥션 연결·SQL 전달·결과 응답* 방법을 새로 배워야 했습니다.

JDBC 는 세 표준 인터페이스로 이 문제를 풀었습니다.

| 인터페이스 | 역할 |
|----------|------|
| `java.sql.Connection` | DB 연결 |
| `java.sql.Statement` (와 `PreparedStatement`) | SQL 담는 객체 |
| `java.sql.ResultSet` | SQL 결과 응답 |

각 DB 벤더는 *이 표준을 구현한 라이브러리* 를 제공합니다. 이것이 JDBC 드라이버입니다. H2 드라이버는 `org.h2.jdbc.JdbcConnection` 같은 구현체를 가집니다. 애플리케이션 코드는 표준 인터페이스만 알면 되고, *드라이버 jar 만 교체* 해 DB 를 바꿀 수 있습니다.

### 0-1. DriverManager 의 호출 흐름

`DriverManager.getConnection(url, user, pw)` 한 줄이 실행될 때 일어나는 일은 다음과 같습니다.

```
[Application]
    │ 1. DriverManager.getConnection(jdbc:h2:..., sa, ...)
    ↓
[DriverManager]
    │ 2. 등록된 드라이버 목록을 순회하며 acceptsURL(url) 호출
    │    └── H2 드라이버: "jdbc:h2:..." 매칭 → OK
    │    └── MySQL 드라이버: 매칭 X
    ↓
[H2 Driver]
    │ 3. connect(url, props) 호출 → JdbcConnection 인스턴스 생성
    ↓
[Application]  ←  Connection 반환
```

핵심은 *URL prefix 로 드라이버를 선택* 한다는 점입니다. `jdbc:h2:...`, `jdbc:mysql:...`, `jdbc:postgresql:...` 각각의 prefix 가 어느 드라이버를 쓸지 결정합니다.

### 0-2. 커넥션·Statement·ResultSet 라이프사이클

가장 단순한 JDBC 호출 한 회는 다음 5단계로 흐릅니다.

```java
Connection conn = DriverManager.getConnection(url, user, pw);  // 1. 획득
PreparedStatement ps = conn.prepareStatement(sql);              // 2. SQL 준비
ps.setLong(1, id);                                              // 3. 파라미터 바인딩
ResultSet rs = ps.executeQuery();                               // 4. 실행
while (rs.next()) { /* 매핑 */ }                                // 5. 결과 순회
rs.close(); ps.close(); conn.close();                           // 6. 해제 (역순)
```

세 가지 주의점이 있습니다.

- **자원 해제는 역순.** `ResultSet → Statement → Connection` 순으로 닫습니다. 반대로 닫으면 *닫힌 부모* 에 매달린 자식 자원이 누수됩니다.
- **`close()` 누락이 곧 커넥션 누수.** `try-with-resources` 가 사실상 강제입니다. 풀을 쓰는 환경에서는 더 치명적입니다 — 풀이 점점 비어 가다가 한순간에 정지합니다.
- **`SQLException` 은 체크 예외다.** 매 호출이 `try-catch` 또는 `throws` 를 강제합니다. 이 의존 전파 문제는 [00-01 자바 예외와 SQL](00-01.자바%20예외와%20SQL.md) 가 다룹니다.

### 0-3. 표준화의 한계

JDBC 는 *프로토콜* 을 표준화했지 *SQL 자체* 까지 통일하진 않았습니다. 페이징 (`LIMIT/OFFSET` vs `ROWNUM` vs `OFFSET FETCH`), 함수, 데이터 타입은 DB 마다 다릅니다. JDBC 코드를 그대로 옮겨도 SQL 은 DB 에 맞게 손봐야 합니다. JPA 가 이 SQL 차이까지 흡수한 이유가 여기에 있습니다.

이 토대 위에서 *커넥션 획득 비용* 이 다음 절의 주제입니다.



## 1. 커넥션이 비싼 이유

> 커넥션 한 번 만드는 데 들어가는 일들의 합이 ms 단위 비용을 만듭니다.

새 커넥션 생성의 단계:

1. **TCP 3-way handshake** (네트워크 RTT × 1.5).
2. **인증** — 사용자명·비밀번호 전송, 권한 검증.
3. **세션 초기화** — character set, time zone, isolation level.
4. **준비된 statement 캐시 비움** — 새 커넥션은 캐시 0.

이 합계가 10~100ms. 매 HTTP 요청마다 일어나면 응답 시간 자체가 그것에 묶입니다. 풀은 이 비용을 *애플리케이션 시작 시* 미리 지불하고 재사용합니다.

## 2. DataSource — 커넥션 획득의 추상화

> `DriverManager.getConnection(...)` 의 직접 호출을 추상화 — 풀·로깅·메트릭이 끼어들 자리.

```java
public interface DataSource {
    Connection getConnection() throws SQLException;
    Connection getConnection(String user, String pw) throws SQLException;
    // ...
}
```

구현체:

| 구현 | 특징 |
|------|------|
| `DriverManagerDataSource` | 풀 없음, 매번 새 커넥션 (테스트·데모용) |
| HikariCP | 가장 빠름, Spring Boot 기본 |
| Apache DBCP2 | 오래된 표준, 풀 동작 안정적 |
| Tomcat JDBC | Tomcat 내장, 별도 설치 불필요 |
| Oracle UCP | Oracle DB 특화 |

추상화의 효과: 애플리케이션 코드는 어떤 풀을 쓰는지 모르고도 동작합니다. `application.yml` 변경만으로 풀 교체 가능.

## 3. HikariCP — 사실상 표준

> 가장 빠르고 가장 가벼운 풀 — Spring Boot 2.0+ 의 기본.

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/myapp
    username: appuser
    password: secret
    hikari:
      maximum-pool-size: 10
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
      leak-detection-threshold: 60000
```

핵심 파라미터:

| 파라미터 | 의미 | 권장 |
|---------|------|------|
| `maximum-pool-size` | 풀의 최대 커넥션 수 | 보통 (CPU 코어 × 2) ~ 30, 너무 키우지 말 것 |
| `minimum-idle` | 항상 유지할 최소 유휴 커넥션 | maximum-pool-size 와 같게 (Hikari 권장) |
| `connection-timeout` | 풀에서 커넥션 못 받으면 기다리는 시간 (ms) | 30000 |
| `idle-timeout` | 유휴 커넥션 폐기 시간 (ms) | 600000 |
| `max-lifetime` | 한 커넥션의 최대 수명 (ms) | 1800000 (DB 의 wait_timeout 보다 짧게) |
| `leak-detection-threshold` | 커넥션 누수 의심 임계치 (ms) | 60000 (디버그용) |

여기서 질문 하나 — `maximum-pool-size` 를 100 으로 키우면 처리량이 늘어날까요? 보통 아닙니다. DB 측의 CPU·디스크가 병목이라 풀 크기가 어느 정도를 넘으면 처리량은 줄고 락 경합·대기만 증가합니다. HikariCP 공식 가이드는 작게 두는 것을 권장합니다.

## 4. Spring Boot 의 자동 등록

> `spring-boot-starter-jdbc` 가 포함되어 있고 `spring.datasource.*` 가 있으면 DataSource 빈이 자동 등록.

```java
// 자동 등록되는 빈
@Configuration
public class DataSourceAutoConfiguration {
    // HikariDataSource 가 기본
}
```

코드에서는 `DataSource` 만 주입받으면 됩니다.

```java
@Repository
public class MemberRepository {
    private final DataSource dataSource;

    public MemberRepository(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    public Member findById(Long id) throws SQLException {
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement("SELECT * FROM member WHERE id = ?")) {
            ps.setLong(1, id);
            try (ResultSet rs = ps.executeQuery()) {
                if (rs.next()) {
                    return mapRow(rs);
                }
                return null;
            }
        }
    }
}
```

`try-with-resources` 가 핵심 — 사용 끝난 Connection 이 풀로 *반환* 됩니다 (실제 close 가 아님). 누수 시 풀 고갈로 후속 요청이 모두 timeout.

## 5. 운영 주의점

> 커넥션 풀은 *애플리케이션 측 자원* 이면서 *DB 측 자원* — 둘의 한계를 모두 봐야 합니다.

1. **풀 크기 vs DB max_connections** — 애플리케이션 인스턴스 × 풀 크기 < DB max_connections. 안 그러면 DB 측에서 거부.
2. **max-lifetime < DB wait_timeout** — DB 가 먼저 끊은 커넥션을 풀이 모를 때 `Communications link failure` 에러. max-lifetime 을 DB 측 timeout 보다 짧게.
3. **누수 감지** — `leak-detection-threshold` 로 60초 이상 보유 시 경고. 트랜잭션 누수의 1차 신호.
4. **메트릭 노출** — Micrometer + HikariCP 메트릭으로 active/idle/wait 시각화. 풀 고갈 사전 감지.
