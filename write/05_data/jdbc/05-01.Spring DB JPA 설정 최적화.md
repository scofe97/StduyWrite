# Spring DB · JPA 설정 최적화

---

> 커넥션 풀과 JPA 의 *개념* 은 [01-01](01-01.커넥션%20풀과%20DataSource.md) 과 [jpa/01-02](../jpa/01-02.JPA%20시작과%20영속성%20컨텍스트.md) 가 다룹니다. 본 노트는 그 위에서 *실제 운영 설정값* 을 어떻게 잡는지 — `application.yml` 한 장으로 모읍니다.

## 1. HikariCP 커넥션 풀 설정

> 풀 크기와 타임아웃 세 값이 운영 안정성의 8할을 결정합니다.

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 16
      minimum-idle: 16
      connection-timeout: 3000
      idle-timeout: 0
      max-lifetime: 170000
      connection-init-sql: SET NAMES utf8mb4
```

### 1-1. 풀 크기 — `(core * 2) + spindle`

풀 크기는 무작정 키운다고 처리량이 늘지 않습니다. HikariCP 가 인용하는 공식은 다음과 같습니다.

```
pool size = (CPU 코어 수 × 2) + 디스크 스핀들 수
```

SSD·클라우드 환경에서는 스핀들 개념이 흐려지므로 *코어 × 2* 를 기준으로 잡고 부하 테스트로 조정합니다. 개발 환경은 2 정도, 운영 환경은 16 정도가 흔한 출발점입니다. DB 측 CPU·디스크가 병목이라 풀이 일정 크기를 넘으면 처리량은 줄고 대기·락 경합만 늘어납니다.

### 1-2. `minimum-idle` 을 `maximum-pool-size` 와 같게

HikariCP 는 `minimum-idle` 과 `maximum-pool-size` 를 *같은 값* 으로 두기를 권장합니다. 둘이 다르면 풀이 부하에 따라 커넥션을 만들고 줄이는데, 이 *동적 조정 자체가 지연* 을 만듭니다. 갑자기 트래픽이 몰리면 커넥션을 새로 만드는 비용 (TCP·인증·세션) 이 응답 시간에 그대로 실립니다. 고정 크기 풀이 예측 가능합니다.

### 1-3. 타임아웃 세 값

세 타임아웃의 역할이 다릅니다.

- `connection-timeout` (3000ms) — 풀에서 커넥션을 못 받으면 기다리는 한계. 넘으면 예외. 짧게 잡아 *빨리 실패* 하게 합니다.
- `idle-timeout` (0) — 유휴 커넥션을 폐기하는 시간. `minimum-idle == maximum-pool-size` 인 고정 풀에서는 0 으로 두어 *유휴 커넥션을 유지* 합니다.
- `max-lifetime` (170000ms) — 한 커넥션의 최대 수명. *DB 의 `wait_timeout` 보다 짧게* 잡습니다. DB 가 먼저 끊은 커넥션을 풀이 모르고 쓰면 `Communications link failure` 가 납니다.

### 1-4. `connection-init-sql`

커넥션이 풀에 들어올 때 한 번 실행할 SQL 입니다. `SET NAMES utf8mb4` 로 문자셋을 고정하면 드라이버·서버 간 인코딩 불일치를 막습니다. 콜레이션 자체는 [sql/02-01 §2-1](../sql/02-01.MySQL%20기초와%20InnoDB.md) 이 다룹니다.

## 2. JDBC URL 파라미터

> URL 의 쿼리 파라미터 몇 개가 배치 성능과 안정성을 좌우합니다.

```
jdbc:mysql://host:3306/db?rewriteBatchedStatements=true&serverTimezone=UTC&useSSL=false&connectTimeout=3000&socketTimeout=60000
```

- `rewriteBatchedStatements=true` — 여러 INSERT 를 *하나의 멀티 밸류 INSERT* 로 묶어 보냅니다. JPA `saveAll` 이나 배치 INSERT 의 처리량이 크게 오릅니다. 이 옵션이 없으면 배치를 써도 한 건씩 나갑니다.
- `serverTimezone=UTC` — 서버 타임존을 UTC 로 고정. 애플리케이션·DB 의 타임존 불일치로 인한 시각 오류를 막습니다.
- `useSSL=false` — 내부망 개발 환경에서 SSL 핸드셰이크 비용 제거. 운영의 외부 구간은 SSL 유지가 원칙입니다.
- `connectTimeout` / `socketTimeout` — 커넥션 수립·소켓 읽기의 한계 시간. 무한 대기로 스레드가 묶이는 것을 막습니다.

## 3. JPA · 하이버네이트 설정

> ORM 의 동작을 운영에 맞게 조이는 설정들입니다.

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate
    open-in-view: false
    properties:
      hibernate:
        format_sql: true
        use_sql_comments: true
```

### 3-1. `ddl-auto`

| 값 | 동작 | 적합 |
|----|------|------|
| `validate` | 스키마가 엔티티와 맞는지 검증만 | 운영 (권장) |
| `none` | 아무것도 안 함 | 운영 (마이그레이션 도구 별도 사용 시) |
| `create-drop` | 시작 시 생성, 종료 시 삭제 | 테스트 |
| `update` | 차이만 반영 | 위험 — 운영 금지 (의도치 않은 변경) |

운영은 `validate` 또는 `none` 입니다. `update` 는 스키마를 *암묵적으로* 바꾸므로 운영에서 쓰면 안 됩니다. 스키마 변경은 Flyway·Liquibase 같은 마이그레이션 도구로 명시적으로 합니다.

### 3-2. OSIV 끄기 — `open-in-view: false`

OSIV (Open Session In View) 가 켜져 있으면 영속성 컨텍스트가 *컨트롤러 응답까지* 열려 있습니다. View 에서 Lazy 로딩이 되는 편리함이 있지만, *커넥션을 요청 끝까지 점유* 해 부하 분산이 어려워집니다. 운영은 OSIV 를 끄고 DTO 변환을 Service 계층에서 마칩니다. 자세한 영속성 컨텍스트 동작은 [jpa/01-02](../jpa/01-02.JPA%20시작과%20영속성%20컨텍스트.md) 가 다룹니다.

### 3-3. 쿼리 로깅

`show_sql` 은 `System.out` 으로 찍어 비권장이고, 로거를 통한 출력을 씁니다. `format_sql=true` 로 보기 좋게, `use_sql_comments=true` 로 JPQL 주석을 SQL 에 붙여 *어느 JPQL 에서 나온 SQL인지* 추적합니다. 이 옵션들은 *개발 환경에서만* 켭니다. 운영에서 켜면 로그 폭주로 성능이 떨어집니다.

## 4. 성능 옵션

> 실행계획 캐시 적중률과 트랜잭션 처리량을 끌어올리는 옵션들입니다.

```yaml
spring:
  jpa:
    properties:
      hibernate:
        criteria:
          literal_handling_mode: bind
        query:
          in_clause_parameter_padding: true
        connection:
          provider_disables_autocommit: true
  datasource:
    hikari:
      auto-commit: false
```

### 4-1. `in_clause_parameter_padding`

`IN` 절의 파라미터 개수를 *2의 거듭제곱* 으로 패딩합니다. `IN (?, ?, ?)` (3개) 를 4개로 채워 `IN (?, ?, ?, ?)` 로 만듭니다. 파라미터 개수가 다를 때마다 다른 쿼리로 인식되어 실행계획 캐시가 갈리는 문제를, 개수를 *2·4·8·16…* 의 몇 단계로 묶어 캐시 적중률을 올립니다.

### 4-2. `literal_handling_mode: bind`

쿼리의 상수값도 파라미터로 바인딩합니다. `WHERE status = 'ACTIVE'` 의 `'ACTIVE'` 같은 리터럴을 `?` 로 바꿔, 상수만 다른 쿼리들이 *같은 실행계획* 을 공유하게 합니다.

### 4-3. Auto-commit 비활성화

`auto-commit=false` + `provider_disables_autocommit=true` 의 조합이 핵심입니다. JPA 는 트랜잭션 시작 시 커넥션의 auto-commit 을 끄는데, 이 과정에서 *커넥션을 미리 가져오는* 비용이 듭니다. 두 옵션으로 *지연 커넥션 획득* 을 활성화하면, 실제 쿼리가 나갈 때까지 커넥션을 잡지 않습니다. 출처 글은 이 조합으로 70~110% 성능 향상을 보고합니다. 다만 수치는 *환경 의존* 이므로 자기 환경에서 측정 후 적용합니다.

## 5. 정리

설정 최적화는 세 계층에서 일어납니다.

- **커넥션 풀** — 고정 크기 (`min == max`), `max-lifetime < DB wait_timeout`, 빠른 실패 타임아웃.
- **JPA 동작** — `ddl-auto: validate`, OSIV off, 로깅은 개발만.
- **성능 옵션** — `in_clause_parameter_padding`·`literal_handling_mode: bind` 로 실행계획 캐시 적중, auto-commit off 로 지연 커넥션 획득.

모든 수치는 *출발점* 입니다. HikariCP 메트릭 (active/idle/wait) 과 슬로우 쿼리 로그로 측정하며 조정하는 것이 설정 최적화의 본질입니다.
