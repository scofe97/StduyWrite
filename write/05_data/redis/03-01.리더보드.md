# Redis 리더보드
---
> 게임·경연·실시간 인기글 같은 랭킹 시스템 — RDB 로 풀면 매번 정렬 비용이 폭발한다. Redis 의 Sorted Set 은 이 문제를 O(log N) 으로 자연스럽게 푼다.

## 1. RDB 로 푸는 경우의 한계

> "SELECT * FROM users ORDER BY score DESC LIMIT 10" 의 비용은 데이터가 늘수록 폭발한다.

100만 사용자의 점수가 있는 테이블에서 상위 10명 조회는 다음과 같은 비용을 갖는다.

1. score 컬럼에 인덱스 — 인덱스 정렬 O(N log N).
2. 매 점수 변경마다 인덱스 재정렬.
3. 동시 업데이트 시 락 경합.

실시간 갱신이 요구되면 캐싱으로도 어렵다. 매 1초 갱신해야 한다면 매 초 정렬 쿼리가 돈다.

## 2. Redis Sorted Set 으로 풀기

> 모든 점수 변경·조회가 O(log N) — 100만 사용자라도 µs 단위.

```
ZADD leaderboard 1500 "alice"
ZADD leaderboard 1200 "bob"
ZADD leaderboard 1800 "carol"

ZREVRANGE leaderboard 0 9 WITHSCORES    # 상위 10명
ZRANK leaderboard "bob"                  # 특정 사용자의 순위
ZINCRBY leaderboard 100 "alice"          # 점수 +100
```

Sorted Set 은 내부적으로 skip list + hash table 의 조합으로, 모든 핵심 연산이 O(log N) 이다.

여기서 질문 하나 — 동률 처리는? `ZADD` 의 score 가 같으면 멤버 이름의 사전순으로 정렬된다. 더 정밀한 동률 처리가 필요하면 score 를 `floor(score) * 1000000 + (max_timestamp - now())` 같은 합성 점수로 만든다.

## 3. 실전 패턴 — 시간 기반 리더보드

> "일간 · 주간 · 월간 리더보드" 는 키 자체에 기간을 박는다.

```
ZADD leaderboard:2026-05-23 100 "alice"
ZADD leaderboard:2026-W21    100 "alice"
ZADD leaderboard:2026-05     100 "alice"
```

각 키에 TTL 을 걸어두면 자동 소멸한다. 추가로 `ZUNIONSTORE` 로 여러 일간 점수를 합쳐 주간 리더보드를 동적으로 만들 수도 있다.

## 4. 운영 주의점

> 리더보드가 커지면 메모리·네트워크 비용이 빠르게 증가한다.

1. **메모리** — Sorted Set 한 항목당 약 100 바이트. 1억 사용자면 10GB.
2. **네트워크** — `ZREVRANGE 0 99999` 는 한 번에 100KB 이상 전송.
3. **순위 페이징** — 클라이언트에서 페이지 처리 (예: `ZREVRANGE k 100 199`).
4. **정확도 vs 메모리** — Top-N 만 필요하면 `ZREMRANGEBYRANK k 0 -N-1` 로 하위 정리.
