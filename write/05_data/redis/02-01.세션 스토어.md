# Redis 세션 스토어
---
> 분산 환경에서 세션을 어느 서버에 둘 것인가의 문제 — Sticky Session 의 한계가 명확해지면 Redis 같은 외부 세션 스토어가 답이 된다.

## 1. Session 이 푸는 문제

> HTTP 는 Stateless 인데 우리는 "방금 로그인한 사용자다" 라는 사실을 기억해야 한다 — 이 기억의 위치가 세션이다.

서블릿 환경에서 `HttpSession` 은 자동으로 `JSESSIONID` 쿠키를 만들고 서버 메모리에 세션 객체를 둔다. 단일 서버에서는 문제가 없지만 서버가 둘 이상이면 다음 문제가 발생한다.

1. WEB1 에서 로그인한 사용자가 WEB2 로 갔을 때 — WEB2 는 세션을 모른다.
2. WEB1 이 죽으면 그 위 모든 세션이 사라진다.
3. Scale-out 이 세션 동기화 부담으로 막힌다.

## 2. 분산 환경의 세 가지 대응

> Sticky Session · 세션 클러스터링 · 외부 세션 스토어 — 셋 중 어느 것을 쓸 것인가의 결정.

| 방식 | 동작 | 비용 |
|------|------|------|
| Sticky Session | 로드 밸런서가 한 사용자를 항상 같은 서버로 | 그 서버 죽으면 세션 손실, scale 비대칭 |
| 세션 클러스터링 | 서버끼리 세션 복제 | 네트워크 비용, 일관성 지연 |
| 외부 세션 스토어 | Redis 같은 외부 저장소 | 외부 의존성 |

Redis 가 세션 스토어의 표준으로 자리잡은 이유는 다음 세 가지다.

1. 빠른 접근 시간 — 매 요청마다 세션 조회·갱신.
2. 영속성 불필요 — 세션은 일시적 데이터.
3. TTL 지원 — `EXPIRE` 로 세션 만료 자동 처리.

## 3. Spring Session 통합

> Spring Session + Redis 의 조합이 가장 흔하다 — 의존성 추가와 설정 몇 줄로 자동 통합.

```gradle
implementation 'org.springframework.session:spring-session-data-redis'
implementation 'org.springframework.boot:spring-boot-starter-data-redis'
```

```yaml
spring:
  session:
    store-type: redis
    timeout: 30m
  data:
    redis:
      host: localhost
      port: 6379
```

`@EnableRedisHttpSession` 을 설정 클래스에 붙이면 `HttpSession` 의 모든 호출이 자동으로 Redis 로 라우팅된다. 기존 코드 (`session.setAttribute(...)` 등) 변경 없이 분산 세션이 완성된다.

여기서 질문 하나 — Spring Session 없이 RedisTemplate 으로 직접 다루면? 가능하지만 다음을 직접 구현해야 한다. 세션 ID 발급, 쿠키 생성, 만료 처리, 직렬화. Spring Session 은 이 모두를 위임받는다.

## 4. 세션 데이터의 특징과 Redis 의 적합성

> 세션 데이터의 네 가지 특징이 Redis 의 강점과 완벽히 맞아떨어진다.

1. **빠른 접근 시간 필요** — Redis 의 인메모리 접근.
2. **영속성 불필요** — Redis 의 RDB/AOF 없이도 충분.
3. **자주 변경** — 매 요청마다 마지막 접근 시간 갱신.
4. **TTL 명확** — 세션 타임아웃과 일치.

RDB 에 세션을 두면 위 네 가지가 모두 비효율적이다. 그래서 세션 스토어 결정은 거의 항상 Redis (또는 비슷한 인메모리 솔루션) 다.
