# Redis 캐시 레이어
---
> 캐시 전략의 이론은 `../theory/01-07.캐싱 전략.md` 가 다룬다. 본 문서는 그 이론을 Spring Boot + Redis 에 적용하는 구현 측만 다룬다.

## 1. Spring Boot Redis 설정

> Lettuce 가 기본 클라이언트 — 비동기·논블로킹 IO 로 처리량 우위.

```gradle
implementation 'org.springframework.boot:spring-boot-starter-data-redis'
```

```yaml
spring:
  data:
    redis:
      host: localhost
      port: 6379
      lettuce:
        pool:
          max-active: 8
          max-idle: 8
```

기본 설정만으로 `RedisTemplate` 과 `StringRedisTemplate` 빈이 자동 등록된다. Jedis 가 필요한 경우 `lettuce` 를 의존성에서 제외하고 `jedis` 를 명시 추가.

## 2. ValueOperations 사용

> 직접 RedisTemplate 호출 — 모든 자료구조 접근의 가장 일반적 경로.

```java
@Service
public class CacheService {
    private final RedisTemplate<String, Object> redisTemplate;

    public void put(String key, Object value, Duration ttl) {
        redisTemplate.opsForValue().set(key, value, ttl);
    }

    public Object get(String key) {
        return redisTemplate.opsForValue().get(key);
    }
}
```

자료구조별 진입점: `opsForValue()` (String), `opsForList()`, `opsForHash()`, `opsForSet()`, `opsForZSet()`.

## 3. Spring Cache 애너테이션

> `@Cacheable` · `@CacheEvict` · `@CachePut` — 메서드 단위로 캐시를 선언적으로 적용.

```java
@Service
public class ProductService {
    @Cacheable(value = "products", key = "#id")
    public Product findById(Long id) {
        return productRepository.findById(id).orElseThrow();
    }

    @CacheEvict(value = "products", key = "#product.id")
    public void update(Product product) {
        productRepository.save(product);
    }
}
```

`@EnableCaching` + `CacheManager` 빈으로 Redis 백엔드 연결.

```java
@Bean
public CacheManager cacheManager(RedisConnectionFactory cf) {
    RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
        .entryTtl(Duration.ofMinutes(10))
        .serializeValuesWith(SerializationPair.fromSerializer(new GenericJackson2JsonRedisSerializer()));
    return RedisCacheManager.builder(cf).cacheDefaults(config).build();
}
```

## 4. 캐시 패턴 — Spring Boot 관점

> `../theory/01-07.캐싱 전략.md` 의 네 패턴이 Spring Boot 에서 어떻게 표현되는가.

| 패턴 | Spring Boot 표현 | 적합 |
|------|------------------|------|
| Look Aside (Cache-Aside) | `@Cacheable` 기본 동작 | 읽기 많은 일반 캐시 |
| Read Through | `CacheLoader` 패턴 (수동) | 캐시 미스 시 자동 로드 |
| Write Through | `@CachePut` + 동기 DB 갱신 | 데이터 일관성 중시 |
| Write Around | DB 만 갱신, 캐시는 만료 대기 | 쓰기 후 즉시 읽지 않는 경우 |

여기서 질문 하나 — `@Cacheable` 만 쓰면 어떤 패턴이 되나? Look Aside 다. 메서드 진입 시 캐시 조회, 미스면 메서드 실행, 결과를 캐시에 저장하고 반환. 가장 흔한 선택.

## 5. 운영 주의점

> 캐시는 정답이 아니라 트레이드오프 — 다음 함정들을 인지하고 대응.

1. **Cache Stampede** — 인기 키 만료 순간 다수 요청이 동시에 DB 로 (`../theory/01-07.캐싱 전략.md` 의 대응 패턴).
2. **Hot Key** — 한 키에 부하 집중 — 키 분할 또는 로컬 캐시 1차.
3. **직렬화** — `JdkSerializer` 기본은 비효율. `GenericJackson2JsonRedisSerializer` 또는 Protobuf 권장.
4. **TTL 정책** — 일괄 만료보다 jitter 추가 (`ttl + random(0, 60s)`) 로 Stampede 완화.
