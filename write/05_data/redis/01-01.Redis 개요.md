# Redis 개요
---
> Redis 는 모든 데이터를 메모리에 두는 key-value 저장소다. 빠른 속도는 디스크를 안 쓴다는 사실에서 나오고, 그 대가로 영속성을 어떻게 보장할지가 매번 결정해야 할 문제로 남는다.

## 1. Redis 가 푸는 문제

> 디스크 기반 DB 의 ms 단위 응답으로 풀 수 없는 작업 — 세션·캐시·랭킹·실시간 큐 — 을 µs 단위로 처리한다.

Redis 의 출발점은 메모리 접근이 디스크 접근보다 수천 배 빠르다는 단순한 사실이다. 이 사실 위에서 다음 세 가지 결정이 따라 나온다.

1. **자료구조 서버** — 단순 key-value 가 아니라 String · List · Hash · Set · Sorted Set · Stream 을 그대로 다룬다.
2. **싱글 스레드 처리** — 동시성 제어 비용을 없애 µs 응답을 가능하게 한다.
3. **영속성은 옵션** — 메모리만 쓰는 캐시로도, RDB/AOF 로 디스크에 박는 DBMS 로도 쓸 수 있다.

여기서 질문 하나 — 그러면 항상 Redis 를 캐시 앞단에 둬야 할까? 그렇지 않다. 캐시가 필요한 접근 패턴 (자주 읽고 가끔 쓰는, ms 단위 응답이 SLA) 이 있을 때만 의미가 있다. 상세는 `../theory/01-07.캐싱 전략.md` 가 다룬다.

## 2. 영속성 — RDB 와 AOF

> 메모리 기반 DB 의 영속성은 디스크 스냅샷(RDB) 과 명령 로그(AOF) 두 방식으로 풀린다.

| 방식 | 동작 | 장점 | 단점 |
|------|------|------|------|
| RDB | 주기적으로 메모리 상태를 단일 파일로 압축 저장 | 복구 빠름, 백업/이관 단순 | 마지막 스냅샷 이후 데이터 손실 가능 |
| AOF | 모든 쓰기 명령을 로그로 누적 기록 | 손실 적음 (fsync 정책에 따라) | 파일 크기 증가, 복구 느림 |

운영에서는 두 방식을 함께 켜는 것이 표준이다. AOF 로 손실을 최소화하고, RDB 로 빠른 복구·백업을 확보한다. 자세한 fsync 정책과 복구 시나리오는 `./04-01.백업과 복구.md` 가 다룬다.

## 3. 싱글 스레드와 클러스터

> 한 노드는 싱글 스레드로 단순함을 얻고, 처리량은 클러스터로 수평 확장한다.

Redis 의 명령 처리는 싱글 스레드다. 한 명령이 끝나야 다음 명령이 실행된다. 이 모델이 가져오는 두 가지 효과는 다음과 같다.

1. **원자성** — 모든 명령이 atomic. `INCR` 같은 카운터가 락 없이 정확히 동작.
2. **간단한 사고 모델** — 락·동기화 디버깅이 없다.

그러나 한 노드가 모든 데이터를 담을 수 없을 때는 클러스터링이 필요하다. Redis Cluster 는 16384 개의 해시 슬롯으로 데이터를 분산하고, 마스터-레플리카 구조로 가용성을 확보한다.

여기서 질문 하나 — CPU 코어가 여러 개인 서버에서 싱글 스레드가 낭비 아닌가? 한 서버에 여러 Redis 인스턴스를 띄워 코어를 다 쓰는 것이 표준이다. 인스턴스마다 자기 메모리를 가지므로 NUMA 친화적이기도 하다.

## 4. 사용 사례

> Redis 는 다섯 가지 워크로드에서 압도적으로 자주 쓰인다.

1. **세션 스토어** — 분산 환경에서 세션 공유 (`./02-01.세션 스토어.md`).
2. **캐시 레이어** — Cache-Aside 패턴 (`./02-02.캐시 레이어.md`).
3. **리더보드·랭킹** — Sorted Set 기반 (`./03-01.리더보드.md`).
4. **Pub/Sub 메시징** — 실시간 채팅·알림 (`./03-02.Pub Sub 채팅.md`).
5. **Rate Limiting · 분산 락** — `INCR` + TTL, `SET NX EX` 패턴.

각 사례가 Redis 의 어떤 자료구조와 만나는지는 `./01-02.자료구조.md` 가 다룬다.
