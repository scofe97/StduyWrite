# Redis 자료구조
---
> Redis 가 다른 key-value 저장소와 갈리는 지점은 자료구조다. value 가 단순 바이트가 아니라 String · List · Hash · Set · Sorted Set · Bitmap · HyperLogLog · Stream 으로 서버 안에서 의미를 가진다. 적절한 자료구조 선택이 명령 복잡도와 메모리 사용을 동시에 결정한다.

## 1. Strings

> 가장 단순한 형태 — 모든 데이터를 바이트로 다룬다.

| 명령 | 동작 |
|------|------|
| `SET k v` / `GET k` | 단일 값 저장/조회 |
| `INCR k` / `DECR k` | 정수로 취급해 +1/-1 (원자적) |
| `MSET k1 v1 k2 v2` / `MGET k1 k2` | 다중 저장/조회 |
| `SETEX k 60 v` | TTL 포함 저장 |

바이너리 안전이라 JPEG 같은 파일도 저장 가능하다. 다만 메모리 비용을 생각하면 큰 객체는 보통 외부 저장소에 두고 Redis 에는 키만 둔다.

## 2. Lists

> linked-list 기반 — Queue 와 Stack 의 토대.

`LPUSH/RPUSH` 로 양 끝에 추가, `LPOP/RPOP` 로 꺼낸다. 양 끝 연산은 O(1) 이지만 중간 인덱스 접근은 O(N) 이다.

| 명령 | 동작 |
|------|------|
| `LPUSH k v` / `RPUSH k v` | 왼쪽/오른쪽 추가 |
| `LPOP k` / `RPOP k` | 왼쪽/오른쪽 제거 |
| `LRANGE k 0 -1` | 전체 범위 조회 |
| `LLEN k` | 길이 |
| `BLPOP k 5` | 블로킹 — 5초 대기 후 timeout |

`BLPOP` 의 블로킹 동작이 Redis 를 가벼운 작업 큐로 쓰게 만든다.

## 3. Hashes

> Field-Value 쌍의 묶음 — 객체 모델링에 자연스럽다.

```
HSET user:1001 name "Alice" age 30 email "alice@example.com"
HGET user:1001 email
HMGET user:1001 name age
HINCRBY user:1001 age 1
```

객체 하나를 String 으로 JSON 직렬화해 저장하는 것과 비교했을 때, Hash 는 특정 필드만 수정/조회가 가능해 네트워크 비용이 줄어든다.

## 4. Sets

> 중복 없는 집합 — 정렬 없음.

`SADD/SMEMBERS/SISMEMBER/SREM` 로 다룬다. 집합 연산(`SINTER` · `SUNION` · `SDIFF`)이 서버 안에서 일어나 강력하다.

응용 예: "오늘 방문한 유니크 사용자 집합" `SADD visitors:2026-05-23 user_id` 패턴.

## 5. Sorted Sets

> Score 가 붙은 정렬된 집합 — 랭킹의 토대.

`ZADD k score member` 로 추가, `ZRANGE`/`ZREVRANGE` 로 정렬된 결과를 본다. 모든 연산이 O(log N).

| 명령 | 동작 |
|------|------|
| `ZADD k 100 "alice"` | 멤버에 점수 부여 |
| `ZREVRANGE k 0 9 WITHSCORES` | 상위 10명 (점수 내림차순) |
| `ZRANK k "alice"` | 멤버의 현재 순위 |
| `ZINCRBY k 5 "alice"` | 점수 +5 |

리더보드·실시간 인기글·우선순위 큐의 자연스러운 토대다. 자세한 응용은 `./03-01.리더보드.md` 가 다룬다.

## 6. Bitmaps · HyperLogLog · Streams

> 특수 목적 자료구조 — 메모리를 극단적으로 아끼거나 로그/이벤트 스트림 처리.

| 자료구조 | 용도 |
|----------|------|
| Bitmap | 1억 사용자의 출석 여부를 12MB 로 표현 (`SETBIT k userId 1`) |
| HyperLogLog | 카디널리티 추정 — 1억 항목 카운팅을 12KB 로 (`PFADD`/`PFCOUNT`) |
| Stream | Kafka 비슷한 append-only log (`XADD`/`XREAD`/`XGROUP`) |

여기서 질문 하나 — HyperLogLog 는 정확한 카운트인가? 아니다. ~0.81% 표준 오차의 확률적 카운팅이다. 정확도가 필요 없는 "유니크 방문자 수" 같은 추정에 적합하다.
