# Redis Pub Sub 채팅
---
> Redis Pub/Sub 은 가장 가벼운 메시징 메커니즘 — 메시지 보존 없는 단방향 fire-and-forget. 채팅·알림·실시간 대시보드 같은 손실 허용 시나리오에 적합하고, 보장이 필요한 곳은 Kafka 같은 별도 솔루션이 답이다.

## 1. Redis Pub/Sub 의 특징

> Subscribe 한 시점부터 들어오는 메시지만 받는다 — 발행자와 구독자의 시간 결합이 강하다.

1. **Subscribe 시점 기준** — 가입 전에 발행된 메시지는 영원히 못 본다.
2. **메시지 보존 없음** — 발행 즉시 모든 구독자에게 전달, 디스크 저장 없음.
3. **재전송 없음** — 구독자가 잠시 끊겼다 돌아오면 그 사이 메시지는 누락.

이 특성이 단순함을 만들지만, 반드시 전달되어야 하는 메시지에는 적합하지 않다.

## 2. Kafka 와의 비교

> 같은 "메시징" 단어지만 보장 모델이 다르다.

| 측면 | Redis Pub/Sub | Kafka |
|------|---------------|-------|
| 메시지 보존 | 없음 | retention 기간만큼 |
| 늦게 가입 | 옛 메시지 못 봄 | 처음부터 재생 가능 |
| 보장 | At-most-once | At-least-once (또는 Exactly-once) |
| 지연 | µs | ms |
| 운영 복잡도 | 거의 없음 | 높음 |

Redis Pub/Sub 은 지연 우선 + 손실 허용 시나리오에서 압도적이다. 채팅방의 실시간 메시지 같은 곳. Kafka 는 보장 필수 시나리오 — 결제 이벤트·주문 처리 등. 자세한 Kafka 측은 `../../04_messaging/README.md`.

## 3. 명령어

> 발행/구독은 두 개의 명령으로 충분하다.

| 명령 | 동작 |
|------|------|
| `SUBSCRIBE channel` | 채널 구독 시작 |
| `UNSUBSCRIBE channel` | 채널 구독 해제 |
| `PUBLISH channel message` | 채널에 메시지 발행 |
| `PSUBSCRIBE pattern` | 패턴 기반 구독 (예: `news.*`) |

`PSUBSCRIBE news.*` 는 `news.sports`, `news.weather` 같은 여러 채널을 한 번에 구독한다.

## 4. Spring Boot 채팅방 구현 패턴

> Spring Data Redis 의 `MessageListener` + `RedisMessageListenerContainer` 조합.

```java
@Configuration
public class RedisConfig {
    @Bean
    public RedisMessageListenerContainer container(
            RedisConnectionFactory cf, MessageListenerAdapter listener) {
        RedisMessageListenerContainer container = new RedisMessageListenerContainer();
        container.setConnectionFactory(cf);
        container.addMessageListener(listener, new PatternTopic("chat.*"));
        return container;
    }
}

@Component
public class ChatMessageListener implements MessageListener {
    @Override
    public void onMessage(Message message, byte[] pattern) {
        String channel = new String(message.getChannel());
        String body = new String(message.getBody());
        // WebSocket 으로 클라이언트에 전달
    }
}
```

채팅 메시지 흐름은 일반적으로 다음과 같다.

1. 사용자가 WebSocket 으로 메시지 전송.
2. 서버가 `PUBLISH chat.roomId "{message}"` 발행.
3. 같은 채팅방을 구독한 모든 서버 인스턴스의 listener 가 받음.
4. 각 서버가 자기에게 연결된 WebSocket 클라이언트에 전송.

이 구조가 수평 확장된 채팅 서버의 표준 패턴이다.

## 5. 운영 주의점

> Pub/Sub 은 손실 허용이라는 전제 위에서 운영해야 한다.

1. **구독자 다운** — 재시작 후 그 사이 메시지 누락. 중요 메시지는 별도 저장.
2. **느린 구독자** — Redis 의 client output buffer 초과 시 강제 연결 끊김.
3. **메시지 영구 보존 필요** → Redis Streams (`XADD`/`XREAD`) 또는 Kafka 로 전환.
