# Redis 백업과 복구 (RDB · AOF)
---
> 메모리 기반 DB 의 영속성은 두 가지 방향에서 푼다 — 주기적인 스냅샷(RDB) 과 모든 명령의 로그(AOF). 둘 중 하나만 쓰는 결정은 거의 잘못이고, 운영 환경은 거의 항상 둘을 함께 켠다.

## 1. RDB (Snapshotting)

> 특정 시점의 메모리 전체를 단일 파일로 압축 저장 — "어제 9시 상태의 스냅샷" 같은 형태.

장점:

1. **복구 빠름** — 단일 파일이라 `redis-server` 시작 시 한 번에 로딩.
2. **백업 단순** — 파일 하나 복사로 끝.
3. **장기 보관 적합** — 압축률 좋아 디스크 비용 낮음.

단점:

1. **마지막 스냅샷 이후 데이터 손실** — `save 900 1` (15분에 한 번) 정책이면 최대 15분치 손실.
2. **fork 비용** — 스냅샷 만들 때 자식 프로세스 생성, 메모리 큰 인스턴스는 CPU·메모리 spike.

`save 900 1`, `save 300 10`, `save 60 10000` 의 형태로 "N초간 M개 이상 변경이면 스냅샷" 트리거를 설정.

## 2. AOF (Append Only File)

> 모든 쓰기 명령을 로그로 누적 — "어제 9시부터 지금까지 일어난 모든 명령".

장점:

1. **손실 최소** — `appendfsync everysec` (기본) 이면 최대 1초치만 손실.
2. **재실행 가능** — 명령 로그라 시점 복구 가능.

단점:

1. **파일 크기 큼** — 모든 명령 누적. AOF rewrite 로 압축 필요.
2. **복구 느림** — 모든 명령 재실행.

### 2-1. fsync 정책

> 메모리 → 디스크 flush 의 빈도가 손실 윈도우를 결정한다.

| 정책 | 동작 | 손실 윈도우 |
|------|------|-------------|
| `always` | 매 명령마다 fsync | 0 (가장 안전, 가장 느림) |
| `everysec` | 매 초마다 fsync | 최대 1초 |
| `no` | OS 에 위임 | 수십 초 (가장 빠름, 가장 위험) |

`everysec` 가 거의 모든 운영 환경의 기본 선택.

## 3. RDB + AOF 함께 운영

> "AOF 로 손실 최소화, RDB 로 빠른 복구" — 두 방식이 만나는 표준 운영.

Redis 4.0+ 의 hybrid persistence (`aof-use-rdb-preamble yes`) 가 둘을 통합한다. AOF 파일 시작부에 RDB 스냅샷을 박고 그 뒤를 명령 로그로 이어간다. 결과는 다음과 같다.

1. 복구 시: RDB 부분으로 빠르게 base 상태 복원, AOF 부분으로 마지막 변경까지 재생.
2. 디스크 사용: RDB 압축 + 작은 AOF 부분.
3. 손실 윈도우: AOF 의 fsync 정책 따름.

## 4. Replication (복제)

> 백업이 아니라 가용성 — 마스터가 죽었을 때 레플리카가 즉시 대체.

```
[Master]  --- async --> [Replica 1]
                    \-> [Replica 2]
```

마스터에서 일어난 모든 쓰기가 비동기로 레플리카에 전파된다. 레플리카는 읽기 부하 분산에도 쓰인다.

설정:

```
# replica.conf
replicaof master-host 6379
```

여기서 질문 하나 — Replication 이 있으면 RDB/AOF 가 필요 없나? 아니다. 레플리카는 마스터의 현재 상태를 따라간다. 마스터에서 `FLUSHALL` 같은 실수가 일어나면 레플리카에도 즉시 전파된다. 백업은 시간 차원의 안전망이고, 복제는 가용성 차원의 안전망이다.

## 5. Sentinel (자동 페일오버)

> Replication 위의 자동 모니터링·페일오버 — 마스터 다운 감지 시 레플리카를 자동 승격.

```
[Sentinel 1] [Sentinel 2] [Sentinel 3]   ← 셋 이상 권장 (quorum)
       ↓ 모니터링 ↓
   [Master] --> [Replica 1] [Replica 2]
```

Sentinel 의 책임:

1. 마스터·레플리카 상태 모니터링.
2. 마스터 다운 합의 (quorum 기반).
3. 레플리카 중 새 마스터 선출 + 다른 레플리카들의 master 재지정.
4. 클라이언트에게 새 마스터 알림.

Redis Cluster 모드에서는 Sentinel 없이 노드 간 gossip 으로 같은 역할을 수행한다.

## 6. 운영 체크리스트

> 운영 환경에서 다음 다섯이 모두 켜져 있어야 한다.

1. RDB + AOF 함께 (hybrid).
2. `appendfsync everysec`.
3. 레플리카 최소 1 대.
4. Sentinel 3 대 또는 Cluster 모드.
5. RDB 파일 외부 백업 (S3 등) — 주 1회 이상.
