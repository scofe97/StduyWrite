# 스프링 데이터 페이징 통합 — Custom Repository + Pageable

---

> Spring Data JPA `JpaRepository` + 사용자 정의 리포지토리 + `Pageable` 을 한 흐름으로 합치는 표준 패턴이다. 본 노트는 *진입점* 역할 — 명명 규약은 [03-01](03-01.커스텀%20리포지토리%20패턴.md), 카운트 최적화 세부는 [01-06](01-06.페이징과%20fetch%20join%20함정.md) 가 다룬다.

## 1. 통합 패턴의 그림

> 세 인터페이스와 한 구현 클래스가 *Spring Data 의 명명 규약* 으로 자동으로 합쳐진다.

```
[MemberRepository (Spring Data JpaRepository)]
        │ extends
        ├── JpaRepository<Member, Long>        ─→ 기본 CRUD
        └── MemberRepositoryCustom              ─→ 사용자 정의 검색·페이징
                ↑ implements
        [MemberRepositoryImpl]                  ─→ JPAQueryFactory 사용
```

호출자는 `MemberRepository` *하나만* 본다. Spring Data 가 *런타임에 두 구현을 합쳐* 한 빈을 제공한다.

```java
@Service
public class MemberService {
    private final MemberRepository memberRepository;  // 통합 빈

    public Page<MemberTeamDto> search(MemberSearchCondition cond, Pageable pageable) {
        return memberRepository.searchPageComplex(cond, pageable);  // 사용자 정의 호출
    }

    public Optional<Member> findById(Long id) {
        return memberRepository.findById(id);  // Spring Data 기본 호출
    }
}
```

명명 규약을 어기면 합쳐지지 않는다 — [03-01 §명명 규약을 어기면 어떻게 되는가](03-01.커스텀%20리포지토리%20패턴.md) 가 자세히 다룬다.

## 2. 세 인터페이스 작성

> 사용자 정의 인터페이스에 *페이징 메서드 두 개* 를 둔다. 단순 방식과 분리 방식이다.

### 2-1. 사용자 정의 인터페이스

```java
public interface MemberRepositoryCustom {
    List<MemberTeamDto> search(MemberSearchCondition condition);
    Page<MemberTeamDto> searchPageSimple(MemberSearchCondition condition, Pageable pageable);
    Page<MemberTeamDto> searchPageComplex(MemberSearchCondition condition, Pageable pageable);
}
```

검색 메서드와 페이징 메서드를 같은 인터페이스에 둔다. 도메인이 더 커지면 [03-01 §Fragment 패턴](03-01.커스텀%20리포지토리%20패턴.md) 처럼 쪼갠다.

### 2-2. Spring Data 리포지토리

```java
public interface MemberRepository
        extends JpaRepository<Member, Long>, MemberRepositoryCustom {

    List<Member> findByUsername(String username);  // 쿼리 메소드
}
```

여기서 두 부모를 모두 extends 한다. `MemberRepositoryCustom` 을 빠뜨리면 빈 합치기가 동작하지 않는다.

## 3. 페이징 두 방식 — 단순 vs 분리

> 강의 6장의 핵심 비교. 노트 [01-06](01-06.페이징과%20fetch%20join%20함정.md) 가 *왜 분리해야 하는지* 의 함정을 다루고, 본 절은 *구체 코드 패턴* 을 담는다.

### 3-1. 단순 방식 — 한 쿼리에서 카운트 같이 (deprecated)

옛 방식은 `fetchResults()` 로 콘텐츠와 카운트를 한 번에 가져왔다. QueryDSL 5.x 시기 강의 코드는 이 형태다.

```java
public Page<MemberTeamDto> searchPageSimple(MemberSearchCondition condition, Pageable pageable) {
    QueryResults<MemberTeamDto> results = queryFactory
            .select(new QMemberTeamDto(member.id, member.username, member.age, team.id, team.name))
            .from(member)
            .leftJoin(member.team, team)
            .where(usernameEq(condition.getUsername()),
                    teamNameEq(condition.getTeamName()),
                    ageGoe(condition.getAgeGoe()),
                    ageLoe(condition.getAgeLoe()))
            .offset(pageable.getOffset())
            .limit(pageable.getPageSize())
            .fetchResults();  // deprecated in 5.x+

    return new PageImpl<>(results.getResults(), pageable, results.getTotal());
}
```

문제 두 가지가 있다.

- `fetchResults()` 는 QueryDSL 5.x 부터 `@Deprecated` 표시되어 있다. 6.x 이상에서는 권장되지 않는다.
- 콘텐츠와 카운트를 *완전히 같은 쿼리* 로 강제한다. 카운트 쿼리에 페치 조인이 끼면 의도치 않은 비용이 든다.

본 묶음의 표준은 다음 *분리 방식* 이다.

### 3-2. 분리 방식 — 콘텐츠와 카운트 별도 (권장)

```java
public Page<MemberTeamDto> searchPageComplex(MemberSearchCondition condition, Pageable pageable) {
    List<MemberTeamDto> content = queryFactory
            .select(new QMemberTeamDto(member.id, member.username, member.age, team.id, team.name))
            .from(member)
            .leftJoin(member.team, team)
            .where(usernameEq(condition.getUsername()),
                    teamNameEq(condition.getTeamName()),
                    ageGoe(condition.getAgeGoe()),
                    ageLoe(condition.getAgeLoe()))
            .offset(pageable.getOffset())
            .limit(pageable.getPageSize())
            .fetch();

    Long total = queryFactory
            .select(member.count())
            .from(member)
            .leftJoin(member.team, team)
            .where(usernameEq(condition.getUsername()),
                    teamNameEq(condition.getTeamName()),
                    ageGoe(condition.getAgeGoe()),
                    ageLoe(condition.getAgeLoe()))
            .fetchOne();

    return new PageImpl<>(content, pageable, total);
}
```

장점은 두 가지다.

- 카운트 쿼리에서 *필요 없는 조인을 빼서* 비용을 줄일 수 있다 (예: 정렬용 조인 제거).
- `fetchResults` 의 deprecated 함정을 피한다.

## 4. 카운트 생략 최적화

> 마지막 페이지 검증이 필요 없는 경우 카운트 쿼리 자체를 생략한다.

`PageableExecutionUtils.getPage` 가 카운트를 *Supplier* 로 받아 *필요할 때만* 실행한다.

```java
public Page<MemberTeamDto> searchPageOptimized(MemberSearchCondition condition, Pageable pageable) {
    List<MemberTeamDto> content = queryFactory
            .select(new QMemberTeamDto(...))
            .from(member)
            .leftJoin(member.team, team)
            .where(...)
            .offset(pageable.getOffset())
            .limit(pageable.getPageSize())
            .fetch();

    JPAQuery<Long> countQuery = queryFactory
            .select(member.count())
            .from(member)
            .leftJoin(member.team, team)
            .where(...);

    return PageableExecutionUtils.getPage(content, pageable, countQuery::fetchOne);
}
```

카운트가 생략되는 두 경우는 [01-06 §PageableExecutionUtils로 카운트 최적화](01-06.페이징과%20fetch%20join%20함정.md) 에 정리되어 있다.

## 5. 정리

Spring Data JPA + QueryDSL 페이징 통합의 핵심은 *명명 규약으로 두 구현을 합친다* 는 점에 있다.

- `MemberRepository extends JpaRepository, MemberRepositoryCustom` 의 두 부모 선언이 출발점이다.
- `MemberRepositoryImpl implements MemberRepositoryCustom` 의 `Impl` 접미사가 합치기의 트리거다.
- 페이징은 *분리 방식* 을 기본으로, *마지막 페이지 검증 없음* 시 `PageableExecutionUtils.getPage` 로 카운트를 생략한다.

본 절의 코드 패턴은 강의 6장의 원형이다. 운영 코드에서 마주치는 변형 (Fragment 분리, `QuerydslRepositorySupport` 회피) 은 [03-01](03-01.커스텀%20리포지토리%20패턴.md) 이 다루고, 카운트 최적화의 세부는 [01-06](01-06.페이징과%20fetch%20join%20함정.md) 이 다룬다.
