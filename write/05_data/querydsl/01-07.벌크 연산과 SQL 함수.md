# 벌크 연산과 SQL 함수

---

> 쿼리 한 번으로 다건 수정·삭제를 처리하는 **벌크 연산**, 그리고 Dialect 에 등록된 함수를 호출하는 **SQL function** 두 가지를 다룹니다. 둘 다 영속성 컨텍스트를 우회하므로 사용 후 처리가 중요합니다.

## 1. 벌크 연산 — `update().execute()` / `delete().execute()`

> JPQL 의 벌크 연산을 QueryDSL 의 type-safe 방식으로 표현합니다.

### 1-1. 대량 UPDATE

조건에 맞는 회원의 이름을 한 번에 바꿉니다.

```java
long count = queryFactory
        .update(member)
        .set(member.username, "비회원")
        .where(member.age.lt(28))
        .execute();
```

`execute()` 의 반환값은 영향을 받은 *행 수* 입니다. 위 SQL 은 한 번의 UPDATE 로 28세 미만 회원 전체의 이름을 일괄 변경합니다.

### 1-2. 기존 값 기준 계산

기존 컬럼 값에 더하거나 곱하는 표현도 자연스럽게 작성됩니다.

```java
long count = queryFactory
        .update(member)
        .set(member.age, member.age.add(1))
        .execute();
```

`add(1)` 외에 `multiply(2)` 같은 산술 표현도 같은 형태로 가능합니다. 실제 실행되는 SQL 은 다음과 같습니다.

```sql
update member
set    age = age + 1
```

### 1-3. 대량 DELETE

```java
long count = queryFactory
        .delete(member)
        .where(member.age.gt(18))
        .execute();
```

조건에 맞는 행 전체를 한 번의 DELETE 로 처리합니다. 조건이 없으면 테이블 전체가 삭제되므로 `where` 누락은 사고로 직결됩니다.

## 2. 영속성 컨텍스트 우회 문제

> 벌크 연산은 JPQL 배치와 동일하게 *영속성 컨텍스트를 무시* 합니다. 1차 캐시와 DB 가 어긋날 수 있습니다.

같은 트랜잭션 안에서 다음 흐름을 봅니다.

```java
Member m = queryFactory.selectFrom(member).where(member.id.eq(1L)).fetchOne();
// m.username = "alice", 영속성 컨텍스트에 1차 캐시

queryFactory.update(member)
        .set(member.username, "비회원")
        .where(member.id.eq(1L))
        .execute();
// DB 의 username 은 "비회원", 영속성 컨텍스트의 m.username 은 여전히 "alice"

Member again = em.find(Member.class, 1L);
// 1차 캐시에서 꺼냄 — username 은 "alice" (DB 와 불일치)
```

벌크 연산 후 같은 엔티티를 다시 조회하면 *오염된 1차 캐시* 가 반환됩니다. 안전한 대응은 둘 중 하나입니다.

- `em.flush()` + `em.clear()` 로 컨텍스트를 비우고 다시 읽기
- 벌크 연산을 별도 트랜잭션 (또는 별도 메서드) 으로 분리해 컨텍스트 자체를 새로 만들기

실무에서는 *벌크 후 즉시 `clear`* 가 가장 흔한 패턴입니다.

## 3. SQL function 호출 — `Expressions.stringTemplate`

> JPA 와 마찬가지로 Dialect 에 등록된 함수만 호출할 수 있습니다.

### 3-1. 표준 함수 호출

회원 이름의 "member" 문자열을 "M" 으로 치환하는 예입니다.

```java
String result = queryFactory
        .select(Expressions.stringTemplate(
                "function('replace', {0}, {1}, {2})",
                member.username, "member", "M"))
        .from(member)
        .fetchFirst();
```

`function('함수명', 인자들)` 형식은 Hibernate 가 표준화한 호출 규약입니다. QueryDSL 은 이 문자열 템플릿을 그대로 SQL 에 박아 넣습니다.

### 3-2. 소문자 비교

`lower` 같은 ANSI 표준 함수는 두 방식 모두 가능합니다.

```java
// 방식 1 — stringTemplate 으로 직접 호출
.where(member.username.eq(
        Expressions.stringTemplate("function('lower', {0})", member.username)))

// 방식 2 — QueryDSL 내장 메서드
.where(member.username.eq(member.username.lower()))
```

두 결과는 같습니다. `lower`·`upper`·`concat` 같은 ANSI 표준 함수는 QueryDSL 이 이미 메서드로 제공하므로 `stringTemplate` 을 굳이 쓸 이유가 없습니다.

`stringTemplate` 이 필요한 자리는 *Dialect 가 등록한 비표준 함수* (예: MySQL 의 `JSON_EXTRACT`, PostgreSQL 의 `to_tsvector`) 입니다. 이런 경우는 다음 절차를 거칩니다.

1. 커스텀 Dialect 클래스에 함수를 등록합니다 (`registerFunction`).
2. QueryDSL 에서 `Expressions.stringTemplate("function('함수명', {0})", path)` 로 호출합니다.

## 4. 정리

벌크 연산과 SQL function 의 공통점은 *영속성 컨텍스트를 우회* 한다는 점입니다.

- 벌크 연산은 *다건을 한 번에* 처리할 수 있어 빠르지만, 컨텍스트와 DB 가 어긋납니다. 사용 후 `clear` 가 안전합니다.
- SQL function 은 *Dialect 의존* 입니다. DB 가 바뀌면 함수도 같이 점검해야 합니다.

`Expressions.stringTemplate` 의 더 일반적인 활용 (서브쿼리 alias, CASE 표현식 합성) 은 [`./02-03.정렬·집계·프로젝션 보충.md`](./02-03.정렬·집계·프로젝션%20보충.md) 가 다룹니다.
