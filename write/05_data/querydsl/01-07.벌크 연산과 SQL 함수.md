# 벌크 연산과 SQL 함수

---

> 쿼리 한 번으로 다건 수정·삭제를 처리하는 **벌크 연산**, 그리고 Dialect 에 등록된 함수를 호출하는 **SQL function** 두 가지를 다룹니다. 둘 다 영속성 컨텍스트를 우회하므로 사용 후 처리가 중요합니다.

## 1. 먼저 — 단건 변경은 QueryDSL 이 하지 않는다

> 이 챕터를 읽기 전에 짚을 전제가 있습니다. **한 건의 INSERT/UPDATE/DELETE 는 QueryDSL 의 일이 아닙니다.** 그건 JPA(EntityManager) 의 영속성 컨텍스트가 담당합니다. QueryDSL 의 `update()/delete()` 는 *조건에 맞는 다건을 한 방에* 처리하는 **벌크 전용** 도구입니다.

JPA 에서 한 건을 다루는 정석은 다음과 같습니다.

| 작업 | 정석 방식 | 핵심 |
|------|----------|------|
| **INSERT** | `em.persist(entity)` / `repository.save(entity)` | QueryDSL 에는 INSERT 자체가 **없다** |
| **UPDATE (단건)** | 엔티티 조회 → 필드 변경 → 트랜잭션 커밋 | **변경 감지(dirty checking)** — `save()` 호출조차 불필요 |
| **DELETE (단건)** | `em.remove(entity)` / `repository.delete(entity)` | 영속 상태 엔티티를 제거 표시 |

```java
// UPDATE 단건 — 조회 후 변경 메서드만 호출하면 커밋 시점에 UPDATE 가 자동 발행된다 (변경 감지)
Member m = em.find(Member.class, 1L);
m.changeAge(31);              // save() 호출 없음
// 트랜잭션 커밋 → flush → "update member set age=? ... where member_id=?" 1건

// DELETE 단건
Member m = em.find(Member.class, 1L);
em.remove(m);

// INSERT — QueryDSL 은 못 한다. persist 로 한다.
em.persist(Member.builder().username("eve").age(20).build());
```

**왜 QueryDSL 로 단건 UPDATE 를 하면 안 되나요?** 

- §3 에서 보듯 QueryDSL 벌크는 영속성 컨텍스트를 *우회* 합니다. 단건을 굳이 `queryFactory.update()...execute()` 로 처리하면, 같은 트랜잭션의 1차 캐시가 옛 값을 그대로 들고 있어 바로 뒤의 조회가 *DB 와 어긋난 값* 을 돌려줍니다. 
- 단건은 변경 감지가 1차 캐시·DB 를 함께 갱신해 주므로 훨씬 안전합니다.

**그러면 QueryDSL 벌크는 언제 쓰나요?** 

- "28세 미만 회원 전원의 등급을 한 번에 내려라" 처럼 *대상이 다건이고, 한 건씩 조회→변경하면 N 번의 UPDATE 가 나가 비효율* 일 때입니다. 이때만 §2 의 벌크가 정답입니다.

| 상황 | 도구 | 발행 쿼리 |
|------|------|-----------|
| 한 건 변경 | 변경 감지 / `em.remove` / `persist` | 1건 (캐시·DB 동기) |
| 다건 일괄 변경 | QueryDSL `update()/delete().execute()` | 1건 (캐시 우회 — `clear` 필요) |
| 한 건씩 다건 변경 | ❌ 안티패턴 | N건 (느림) |



## 2. 벌크 연산 — `update().execute()` / `delete().execute()`

> JPQL 의 벌크 연산을 QueryDSL 의 type-safe 방식으로 표현합니다.

### 2-1. 대량 UPDATE

조건에 맞는 회원의 이름을 한 번에 바꿉니다.

```java
long count = queryFactory
        .update(member)
        .set(member.username, "비회원")
        .where(member.age.lt(28))
        .execute();
```

- `execute()` 의 반환값은 영향을 받은 *행 수* 입니다. 위 SQL 은 한 번의 UPDATE 로 28세 미만 회원 전체의 이름을 일괄 변경합니다.

### 2-2. 기존 값 기준 계산

기존 컬럼 값에 더하거나 곱하는 표현도 자연스럽게 작성됩니다.

```java
long count = queryFactory
        .update(member)
        .set(member.age, member.age.add(1))
        .execute();
```

- `add(1)` 외에 `multiply(2)` 같은 산술 표현도 같은 형태로 가능합니다. 실제 실행되는 SQL 은 다음과 같습니다.

```sql
update member
set    age = age + 1
```

### 2-3. 대량 DELETE

```java
long count = queryFactory
        .delete(member)
        .where(member.age.gt(18))
        .execute();
```

- 조건에 맞는 행 전체를 한 번의 DELETE 로 처리합니다. 조건이 없으면 테이블 전체가 삭제되므로 `where` 누락은 사고로 직결됩니다.



## 3. 영속성 컨텍스트 우회 문제

> 벌크 연산은 JPQL 배치와 동일하게 *영속성 컨텍스트를 무시* 합니다. 1차 캐시와 DB 가 어긋날 수 있습니다.

### 3-1. 왜 무시하는가 — 두 흐름이 다른 길로 간다

JPA 의 정상 변경(`persist`·`remove`·변경 감지)은 *엔티티 객체* 를 다룹니다. 객체의 상태를 1차 캐시에 쌓아 두었다가 flush 시점에 그 변경분을 SQL 로 번역해 한꺼번에 내보냅니다. 즉 **1차 캐시가 곧 "무엇을 DB 에 쓸지" 의 원본** 입니다. 

- (이 정상 흐름의 상세는 [`../jpa/01-02.JPA 시작과 영속성 컨텍스트.md`](../jpa/01-02.JPA%20시작과%20영속성%20컨텍스트.md) 가 다룹니다.)

벌크 연산은 길 자체가 다릅니다. `queryFactory.update()...execute()` 는 *엔티티 객체를 거치지 않고* 처음부터 **하나의 JPQL UPDATE/DELETE 문장** 이고, 그 문장은 곧장 SQL UPDATE/DELETE 로 번역돼 DB 에 직접 나갑니다. 

- 이 경로에는 1차 캐시를 들여다보는 단계가 *애초에 없습니다*. 
- 한 문장으로 수천 행을 바꾸는 게 목적인데, 바뀌는 행마다 대응하는 엔티티 객체를 캐시에서 찾아 동기화하려 들면 벌크의 성능 이점이 사라지기 때문입니다.

그래서 결과가 갈립니다. DB 의 값은 벌크 문장대로 바뀌지만, *이미 1차 캐시에 로드돼 있던 엔티티* 는 벌크가 건드리지 않으므로 옛 값 그대로 남습니다. 캐시와 DB 가 어긋나는 건 버그가 아니라 *설계상 당연한 결과* 입니다.

> **"1차 캐시를 갱신하면서 JPA 가 알아서 벌크로 합쳐 주면 안 되나요?"** — 그러려면 *바뀌는 모든 행에 대응하는 엔티티를 1차 캐시에 먼저 로드* 해야 합니다. "28세 미만 회원 전원의 age +1" 을 하려면 그 회원을 전부 메모리에 올린 뒤 한 건씩 변경 감지를 돌려야 하죠. 대상이 100만 건이면 100만 객체를 힙에 쌓다 OOM 으로 죽습니다. 그 순간 *메모리에 안 올리고 DB 에서 한 방에* 라는 벌크의 존재 이유가 사라집니다. 그래서 JPA 는 두 길을 **의도적으로 분리** 해 둡니다 — 적은 엔티티는 변경 감지(캐시 동기), 대량은 벌크(캐시 우회, 대신 빠르고 가벼움). 우회는 *대가* 가 아니라 벌크를 벌크답게 만드는 *전제* 입니다.

### 3-2. 어긋남이 드러나는 순간

같은 트랜잭션 안에서 다음 흐름을 봅니다.

```java
Member m = queryFactory.selectFrom(member).where(member.id.eq(1L)).fetchOne();
// m.username = "alice", 영속성 컨텍스트에 1차 캐시

queryFactory.update(member)
        .set(member.username, "비회원")
        .where(member.id.eq(1L))
        .execute();
// DB 의 username 은 "비회원", 영속성 컨텍스트의 m.username 은 여전히 "alice"

Member again = em.find(Member.class, 1L);
// 1차 캐시에서 꺼냄 — username 은 "alice" (DB 와 불일치)
```

벌크 연산 후 같은 엔티티를 다시 조회하면 *오염된 1차 캐시* 가 반환됩니다. 안전한 대응은 둘 중 하나입니다.

- `em.flush()` + `em.clear()` 로 컨텍스트를 비우고 다시 읽기
- 벌크 연산을 별도 트랜잭션 (또는 별도 메서드) 으로 분리해 컨텍스트 자체를 새로 만들기

실무에서는 *벌크 후 즉시 `clear`* 가 가장 흔한 패턴입니다.



## 4. SQL function 호출 — `Expressions.stringTemplate`

> QueryDSL 은 SQL 함수를 직접 아는 게 아니라, **Hibernate(JPA) 가 등록해 둔 함수 레지스트리** 를 거쳐 호출합니다. 그래서 *Dialect 에 등록된 함수만* 부를 수 있습니다.

### 4-1. 호출이 SQL 로 변환되는 경로

`Expressions.stringTemplate("function('replace', {0}, {1}, {2})", ...)` 한 줄은 다음 3단계를 거칩니다.

```
QueryDSL stringTemplate           → JPQL/HQL function('replace', ...)
  → Hibernate 함수 레지스트리 조회 (replace 가 등록돼 있나?)
    → Dialect 가 DB 문법으로 변환  → SQL  replace(username, 'member', 'M')
```

- 핵심은 *가운데 단계* 입니다. Hibernate 의 함수 레지스트리에 `replace` 가 없으면 `function('replace', ...)` 는 파싱 단계에서 터집니다. 
- 그래서 "Dialect 에 등록된 함수만" 이라는 제약이 생깁니다. `{0}`, `{1}` 은 *위치 바인딩* 으로, 뒤에 넘긴 인자(`member.username`, `"member"`, `"M"`) 가 순서대로 채워집니다.

### 4-2. 표준 함수 호출

회원 이름의 "member" 문자열을 "M" 으로 치환하는 예입니다.

```java
String result = queryFactory
        .select(Expressions.stringTemplate(
                "function('replace', {0}, {1}, {2})",
                member.username, "member", "M"))
        .from(member)
        .fetchFirst();
```

`function('함수명', 인자들)` 형식은 Hibernate 가 표준화한 호출 규약입니다. QueryDSL 은 이 문자열 템플릿을 그대로 HQL 에 박고, Hibernate 가 레지스트리를 거쳐 SQL 로 바꿉니다.

> 반환 타입에 맞춰 템플릿 메서드를 고릅니다. 문자열은 `stringTemplate`, 숫자는 `numberTemplate(Long.class, ...)`, 불리언은 `booleanTemplate`, 날짜는 `dateTemplate` 입니다. 타입을 틀리면 프로젝션·비교에서 캐스팅 오류가 납니다.

### 4-3. 소문자 비교 — 내장 메서드가 있으면 그걸 써라

`lower` 같은 ANSI 표준 함수는 두 방식 모두 가능합니다.

```java
// 방식 1 — stringTemplate 으로 직접 호출
.where(member.username.eq(
        Expressions.stringTemplate("function('lower', {0})", member.username)))

// 방식 2 — QueryDSL 내장 메서드
.where(member.username.eq(member.username.lower()))
```

- 두 결과는 *같은 SQL* (`where username = lower(?)`) 입니다. `lower`·`upper`·`concat`·`substring`·`trim` 같은 ANSI 표준 함수는 QueryDSL 이 이미 타입 안전한 메서드로 제공하므로, `stringTemplate` 을 굳이 쓸 이유가 없습니다. 
- 내장 메서드는 *오타가 컴파일 에러로 잡히고* 반환 타입도 자동으로 맞춰지지만, 문자열 템플릿은 `"fucntion('lower'...)"` 같은 오타가 런타임에야 터집니다.

### 4-4. 비표준 함수 — Hibernate 6 에서 직접 등록하기

`stringTemplate` 이 *정말 필요한* 자리는 Dialect 에 기본 등록돼 있지 않은 **비표준 함수** (예: PostgreSQL 의 `to_tsvector`, MySQL 의 `JSON_EXTRACT`) 입니다. 이런 함수는 먼저 Hibernate 레지스트리에 등록해야 `function('...')` 호출이 통과합니다.

> ⚠️ **버전 주의**: Hibernate 5 의 `MetadataBuilder#applySqlFunction` / 커스텀 Dialect 의 `registerFunction(...)` 는 **Hibernate 6 에서 제거** 됐습니다. 이 프로젝트(Spring Boot 3.2 = Hibernate 6.4)에서는 동작하지 않습니다. 6 의 정석은 `FunctionContributor` 입니다.

```java
// 1) FunctionContributor 구현 — 레지스트리에 함수를 추가한다.
package com.runnershigh.querydsl.config;

import org.hibernate.boot.model.FunctionContributions;
import org.hibernate.boot.model.FunctionContributor;
import org.hibernate.type.StandardBasicTypes;

public class CustomFunctionContributor implements FunctionContributor {
    @Override
    public void contributeFunctions(FunctionContributions functions) {
        // "array_length(arr, dim)" 같은 PostgreSQL 함수를 등록하는 예
        functions.getFunctionRegistry()
                .registerPattern(
                        "array_length"                  // HQL 에서 부를 이름
                        , "array_length(?1, ?2)"        // 실제 SQL 패턴 (?1, ?2 = 인자 위치)
                        , functions.getTypeConfiguration()
                                .getBasicTypeRegistry()
                                .resolve(StandardBasicTypes.INTEGER)  // 반환 타입
                );
    }
}
```

```
# 2) META-INF/services/org.hibernate.boot.model.FunctionContributor 파일에
#    구현 클래스 FQN 한 줄을 적어 SPI 로 등록한다.
com.runnershigh.querydsl.config.CustomFunctionContributor
```

```java
// 3) 이제 QueryDSL 에서 호출할 수 있다.
queryFactory
        .select(Expressions.numberTemplate(Integer.class,
                "function('array_length', {0}, {1})", somePath, 1))
        .from(...)
        .fetch();
```

- `registerPattern` 은 "이름 → SQL 문자열 패턴" 매핑이라 가장 간단합니다. 더 정교한 인자 검증·오버로드가 필요하면 `registerPattern` 대신 `register(name, SqmFunctionDescriptor)` 로 디스크립터를 직접 구현합니다. 
- 표준 함수라면 애초에 Dialect 가 등록해 두므로 이 과정 자체가 불필요합니다.

> 함수 등록은 *애플리케이션 부팅 시 한 번* 일어납니다. `FunctionContributor` 외에 커스텀 `Dialect` 의 `initializeFunctionRegistry()` 를 오버라이드하는 길도 있지만, Dialect 를 통째로 바꾸지 않는 한 `FunctionContributor` + SPI 등록이 침습이 가장 적습니다.

## 5. 정리

벌크 연산과 SQL function 의 공통점은 *영속성 컨텍스트를 우회* 한다는 점입니다.

- 벌크 연산은 *다건을 한 번에* 처리할 수 있어 빠르지만, 컨텍스트와 DB 가 어긋납니다. 사용 후 `clear` 가 안전합니다.
- SQL function 은 *Dialect 의존* 입니다. DB 가 바뀌면 함수도 같이 점검해야 합니다.

`Expressions.stringTemplate` 의 더 일반적인 활용 (서브쿼리 alias, CASE 표현식 합성) 은 [`./02-03.정렬·집계·프로젝션 보충.md`](./02-03.정렬·집계·프로젝션%20보충.md) 가 다룹니다.
