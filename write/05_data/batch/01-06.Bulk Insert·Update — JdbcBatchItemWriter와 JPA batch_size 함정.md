# Bulk Insert·Update — JdbcBatchItemWriter와 JPA batch_size 함정

---

> Spring Batch 의 Writer 가 진짜 *batch* 인지는 *내부에서 어떻게 SQL 을 묶는가* 와 *JDBC 드라이버가 batch 를 진짜 한 번에 보내는가* 두 단계로 갈립니다. `JdbcBatchItemWriter` 는 *명시적으로 batch* 라 단순하지만, `JpaItemWriter` 는 *Hibernate 설정과 입력 데이터 모양에 따라 batch 가 깨집니다*. 본 편은 *언제 진짜 묶음 SQL 이 나가는지* 와 *왜 JPA 의 batch 가 자주 안 되는지* 의 원리를 봅니다.



## JDBC batch 가 진짜로 작동하는가

> JDBC 의 `executeBatch` 는 *N 개 PreparedStatement 실행을 모아 한 번에 드라이버에 넘기는* API 입니다. 단, 드라이버가 *진짜 한 번의 네트워크 왕복* 으로 보낼지는 드라이버·드라이버 옵션에 따라 갈립니다.

```java
PreparedStatement ps = connection.prepareStatement(
        "INSERT INTO trades (trade_id, amount) VALUES (?, ?)");
for (Trade t : items) {
    ps.setLong(1, t.getId());
    ps.setBigDecimal(2, t.getAmount());
    ps.addBatch();
}
int[] result = ps.executeBatch();
```

코드만 보면 *N 개를 묶어 보내는* 모양이지만 드라이버 내부는 다릅니다.

| 드라이버 | 디폴트 동작 | 진짜 batch 옵션 |
|----------|------------|----------------|
| MySQL Connector/J | 한 건씩 N 번 전송 | `rewriteBatchedStatements=true` |
| PostgreSQL JDBC | 한 건씩 N 번 전송 | `reWriteBatchedInserts=true` (INSERT 한정) |
| Oracle JDBC | 진짜 묶어 전송 | 추가 옵션 불필요 |
| MariaDB | 한 건씩 또는 묶음 (버전·설정에 따라) | `useBatchMultiSend` 또는 `rewriteBatchedStatements` |

**MySQL·PostgreSQL 의 디폴트는 *N 번의 네트워크 왕복* 입니다.** JDBC API 가 `executeBatch` 라도 드라이버가 *내부에서 한 건씩 풀어* 보냅니다. 옵션을 켜야 *INSERT VALUES (?,?), (?,?), ...* 같은 *멀티 VALUES 한 문장* 으로 다시 작성해 보냅니다.

운영에서 *Writer 가 충분히 빠르지 않다* 는 신고는 이 옵션 누락이 가장 흔합니다. 진단 명령은 *DataSource URL 의 옵션 확인* 또는 *네트워크 패킷 캡처로 INSERT 문장 수 세기* 입니다.



## JdbcBatchItemWriter — 명시적 batch

> `JdbcBatchItemWriter` 는 *내부에서 `NamedParameterJdbcTemplate.batchUpdate`* 를 호출합니다. chunk size 만큼 모은 items 를 한 번의 `batchUpdate` 로 보냅니다.

```java
@Bean
public JdbcBatchItemWriter<ProcessedCustomer> jdbcWriter(DataSource dataSource) {
    return new JdbcBatchItemWriterBuilder<ProcessedCustomer>()
            .dataSource(dataSource)
            .sql("INSERT INTO processed_customers " +
                 "(customer_id, full_name, email, credit_score, processed_date) " +
                 "VALUES (:customerId, :fullName, :email, :creditScore, :processedDate)")
            .beanMapped()
            .build();
}
```

`.beanMapped()` 는 `BeanPropertyItemSqlParameterSourceProvider` 를 붙입니다. 객체의 *게터 이름* (`getCustomerId`, `getFullName`) 으로 SQL 의 *명명 파라미터* (`:customerId`, `:fullName`) 가 자동 매핑됩니다.

객체 게터에 의존하지 않는 *명시적 매핑* 이 필요하면 `itemPreparedStatementSetter` 로 직접 짭니다.

```java
@Bean
public JdbcBatchItemWriter<Order> orderWriter(DataSource dataSource) {
    return new JdbcBatchItemWriterBuilder<Order>()
            .dataSource(dataSource)
            .sql("UPDATE orders SET status = ?, processed_at = ?, processor_id = ? " +
                 "WHERE order_id = ?")
            .itemPreparedStatementSetter((order, ps) -> {
                ps.setString(1, order.getStatus().name());
                ps.setTimestamp(2, Timestamp.valueOf(LocalDateTime.now()));
                ps.setString(3, order.getProcessorId());
                ps.setLong(4, order.getOrderId());
            })
            .build();
}
```

이 모델의 *진짜 batch 여부* 는 *DataSource URL 옵션* 에 달려 있습니다. `JdbcBatchItemWriter` 가 *코드 레벨에서는 묶어* 보내지만, 드라이버가 풀어 보내면 *N 번의 INSERT* 로 끝납니다.

### chunk size 와 batch 크기

`JdbcBatchItemWriter` 는 *chunk size 만큼 한 번에* batch 합니다. chunk size 가 100 이면 100 개씩 batch update 한 번. chunk size 가 너무 작으면 *batch 의 이점이 줄어들고*, 너무 크면 *한 트랜잭션이 길어져 락 보유 시간 증가* 와 *실패 시 롤백 비용 증가* 가 옵니다. 일반적 시작점은 *100~1000* 입니다. 정확한 값은 *벤치마킹* 으로 정합니다.



## JpaItemWriter — Hibernate 설정에 의존

> `JpaItemWriter` 는 *내부에서 `EntityManager.persist`* (또는 `merge`) *를 한 건씩 N 번 호출* 합니다. 그 N 번이 *진짜 N 번의 INSERT* 가 되느냐, *Hibernate 가 묶어 한 번의 batch INSERT* 가 되느냐는 *Hibernate 설정* 에 달려 있습니다.

```java
@Bean
public JpaItemWriter<Customer> jpaWriter(EntityManagerFactory emf) {
    return new JpaItemWriterBuilder<Customer>()
            .entityManagerFactory(emf)
            .build();
}
```

Hibernate 가 batch INSERT 를 내려면 세 가지 설정이 같이 켜져야 합니다.

```yaml
spring:
  jpa:
    properties:
      hibernate:
        jdbc:
          batch_size: 100
        order_inserts: true
        order_updates: true
```

| 설정 | 의미 |
|------|------|
| `hibernate.jdbc.batch_size` | 한 batch 에 묶을 SQL 개수 (디폴트 없음 = batch 안 함) |
| `hibernate.order_inserts` | 같은 엔티티 타입의 INSERT 들을 연속으로 정렬 |
| `hibernate.order_updates` | UPDATE 도 같은 정렬 |

`batch_size` 만 켜고 `order_inserts` 를 안 켜면 *서로 다른 엔티티 타입이 섞일 때마다 batch 가 깨집니다*. Customer · Order · Customer · Order 순으로 persist 되면 Hibernate 가 *4번의 단일 INSERT* 를 보냅니다. `order_inserts=true` 면 Customer · Customer · Order · Order 로 정렬해 *2번의 batch INSERT* 로 묶입니다.

### IDENTITY 전략의 함정

JPA Entity 가 `@GeneratedValue(strategy = GenerationType.IDENTITY)` 인 경우 **batch INSERT 가 작동하지 않습니다.** Hibernate 가 *INSERT 직후 generated id 를 받아야* 하는데, batch INSERT 는 *id 를 한꺼번에 받아오는 표준 방법이 없기* 때문입니다.

대안은 *`SEQUENCE` 전략* (PostgreSQL·Oracle) 또는 *`TABLE` 전략* (모든 DB) 입니다. MySQL 은 SEQUENCE 가 없으므로 *애플리케이션에서 id 를 생성* 하고 `IDENTITY` 를 안 쓰는 패턴이 batch INSERT 와 호환됩니다.

이 함정이 *JPA 로는 진짜 batch 가 어렵다* 는 평가의 한 축입니다. 대규모 INSERT 가 핵심이면 `JdbcBatchItemWriter` 가 더 단순한 답입니다.

### 1차 캐시와 메모리 누수

`JpaItemWriter` 는 *persist 후 flush* 만 하고 *clear* 는 안 합니다. chunk 가 끝나면 트랜잭션이 커밋되며 EntityManager 가 닫혀 1차 캐시도 같이 사라지지만, *chunk size 가 크면 chunk 도중 메모리에 N 개 엔티티가 쌓입니다*. 100 만 건 chunk 라면 OOM 위험.

이 자리에서 *명시적 `flush()` + `clear()`* 가 필요할 수 있습니다. `StepExecutionListener` 의 `afterChunk` 에서 호출하거나, *커스텀 Writer 로 감싸* 처리합니다.



## JdbcBatchItemWriter vs JpaItemWriter — 선택 기준

| 기준 | JdbcBatchItemWriter | JpaItemWriter |
|------|--------------------|--------------|
| 진짜 batch | 드라이버 옵션만 켜면 진짜 batch | Hibernate 설정 3개 + IDENTITY 회피 + 정렬 |
| 도메인 모델 의존도 | 낮음. SQL 직접 작성 | 높음. Entity 매핑이 SSOT |
| 연관관계 처리 | 직접 SQL 분리 | `@OneToMany` cascade 등 자동 |
| 변경 감지 (dirty checking) | 없음. UPDATE SQL 직접 | 자동 (1차 캐시 비교) |
| 메모리 | items 묶음만 | 1차 캐시 + items |

규칙은 단순합니다.

- *대규모 INSERT·UPDATE 가 메인* 이고 *진짜 batch 가 중요* 하면 `JdbcBatchItemWriter`
- *기존 JPA 도메인 모델을 그대로* 쓰면서 *적당한 성능* 이면 `JpaItemWriter`

운영 잡의 95% 이상이 *INSERT·UPDATE 가 main* 이면 SQL 을 직접 짜는 비용이 낮습니다. *기존 도메인 모델을 재사용하는 가치* 가 SQL 직접 작성 비용보다 작아지는 분기점이 *대량 처리* 입니다.



## CompositeItemWriter — 두 곳에 동시 쓰기

> 한 chunk 를 *DB 에 INSERT 하면서 동시에 로그 파일에 기록* 하는 패턴이 필요할 때 `CompositeItemWriter` 를 씁니다.

```java
@Bean
public CompositeItemWriter<Trade> compositeWriter(
        JdbcBatchItemWriter<Trade> jdbcWriter,
        FlatFileItemWriter<Trade> logWriter) {
    CompositeItemWriter<Trade> composite = new CompositeItemWriter<>();
    composite.setDelegates(List.of(jdbcWriter, logWriter));
    return composite;
}
```

두 Writer 가 *같은 chunk* 를 받습니다. 한쪽이 예외를 던지면 *chunk 트랜잭션이 롤백* 됩니다. 단, *파일 쓰기는 트랜잭션 안 됩니다*. 파일 쓰기가 성공하고 DB 쓰기가 롤백되면 *파일에는 N 건 기록, DB 에는 0 건* 의 비대칭이 남습니다.

이 자리의 답은 *외부 시스템에 직접 쓰지 말고 Outbox 테이블에 기록* 입니다. 외부 발행이 *별도 폴러* 로 분리되어 chunk 트랜잭션과 분리됩니다.



## 성능 진단 체크리스트

운영 Writer 가 느릴 때 확인하는 순서입니다.

**1. DataSource URL 옵션 확인** — `rewriteBatchedStatements=true` (MySQL) 또는 `reWriteBatchedInserts=true` (PostgreSQL) 가 있는가.

**2. chunk size 적정성** — 너무 작으면 batch 이점이 줄고, 너무 크면 트랜잭션 부담. 100~1000 범위에서 벤치마크.

**3. Hibernate 설정 (JPA 라면)** — `batch_size`, `order_inserts`, `order_updates` 셋이 다 켜져 있는가. `@GeneratedValue(IDENTITY)` 가 batch 를 막고 있지 않은가.

**4. 인덱스** — 대량 INSERT 가 *기존 인덱스 갱신 비용* 때문에 느릴 수 있음. 운영 측면에서 *INSERT 전 인덱스 drop → INSERT → 재생성* 패턴이 적합한 경우가 있음.

**5. Constraint 와 트리거** — FK·CHECK 제약, 트리거가 *건당 추가 비용* 을 만듭니다. 일시적 비활성화가 가능한지 검토.

**6. 네트워크** — DB 가 멀리 있으면 *batch 한 번* 의 왕복이 *100ms* 가 될 수 있음. 같은 가용 영역에 두는 것이 가장 큰 개선.
