# 스케줄링과 운영 — @Scheduled·Quartz·Argo Workflows 연결

---

> Spring Batch 의 공식 문서가 *Spring Batch does not include its own scheduling engine* 라고 명시합니다. *언제 잡을 시작할지* 와 *여러 잡 사이의 의존* 은 *외부 스케줄러* 의 책임입니다. 본 편은 *왜 Spring Batch 가 스케줄링을 안 가지는가* 에서 출발해, *Spring `@Scheduled` → Quartz → Argo Workflows* 로 이어지는 3단 진화와 *각 단계가 어떤 잡 운영에 맞는지* 를 정리합니다. ETL 파이프라인 설계도 이 운영 인프라 경계 안에서 다룹니다.



## Spring Batch 가 스케줄링을 안 가지는 이유

Spring Batch 는 *Job 의 정의·실행·메타데이터 관리* 만 책임집니다. *"매일 새벽 2시에 시작"*, *"잡 A 가 끝나면 잡 B 시작"*, *"실패 시 알림" *같은 결정은 다른 레이어의 일입니다. 분리의 이유 두 가지입니다.

**1. 잡 정의와 실행 결정은 다른 라이프사이클** — Job 의 코드는 *개발팀 책임* 이고, *언제 어떤 잡을 도리는지* 는 *운영팀 정책* 입니다. 한 코드베이스에 둘이 묶이면 *스케줄 바꾸려고 코드 배포* 가 발생합니다.

**2. 의존성 그래프는 잡 안에 안 들어옴** — *데이터 추출 잡 → 변환 잡 → 적재 잡* 같은 ETL 흐름에서 잡 A 의 *코드* 가 잡 B 의 *시작 조건* 을 모릅니다. 두 잡 사이의 의존을 표현하는 자리는 *워크플로우 엔진* 입니다.

이 사고가 *Spring Batch 가 단일 잡의 깊이* 를, *외부 스케줄러가 잡 사이의 흐름* 을 책임지는 분담을 만듭니다.



## JobLauncher — 잡을 시작하는 인터페이스

> 모든 스케줄러는 결국 `JobLauncher.run(job, parameters)` 또는 `JobOperator.start(jobName, properties)` 를 호출합니다. 그 호출이 *언제 일어나는가* 가 스케줄링의 본질입니다.

```java
@Controller
public class JobLauncherController {

    @Autowired
    JobLauncher jobLauncher;

    @Autowired
    Job job;

    @RequestMapping("/launch")
    public void handle() throws Exception {
        jobLauncher.run(job, new JobParameters());
    }
}
```

이 컨트롤러는 *HTTP 요청* 을 *잡 시작 트리거* 로 만든 형태입니다. 스케줄러는 *시간 조건이 맞을 때 이 컨트롤러를 호출하는* 외부 시스템이라고 볼 수도 있습니다.

`JobOperator` 는 더 풍부한 운영 API 를 제공합니다.

```java
@Service
public class BatchOperationsService {
    private final JobOperator jobOperator;
    private final JobRepository jobRepository;

    public long startJob(String jobName, Properties parameters) throws Exception {
        return jobOperator.start(jobName, parameters);
    }

    public long restartJob(long executionId) throws Exception {
        return jobOperator.restart(executionId);
    }

    public void stopJob(long executionId) throws Exception {
        jobOperator.stop(executionId);
    }

    public void abandonJob(long executionId) throws Exception {
        jobOperator.abandon(executionId);
    }
}
```

`start` / `restart` / `stop` / `abandon` 의 4가지 운영 액션이 외부 시스템에 제공할 인터페이스의 표준 형태입니다. *운영 대시보드* 가 있다면 이 메서드들 위에 UI 를 얹는 게 자연스럽습니다.



## 1단계: Spring `@Scheduled` — 가장 단순한 시작

> 같은 애플리케이션 안에서 *시간 조건이 맞으면 JobLauncher 호출* 하는 가장 가벼운 형태입니다. *잡이 1~2 개고 의존이 단순할 때* 의 답입니다.

```java
@Component
public class JobScheduler {
    @Autowired
    private JobLauncher jobLauncher;
    @Autowired
    private Job dailyReportJob;

    @Scheduled(cron = "0 0 2 * * *")  // 매일 새벽 2시
    public void runDailyReport() throws Exception {
        JobParameters params = new JobParametersBuilder()
                .addString("date", LocalDate.now().minusDays(1).toString())
                .toJobParameters();
        jobLauncher.run(dailyReportJob, params);
    }
}
```

이 모델의 *한계* 는 세 가지입니다.

**1. 잡 간 의존을 코드로 표현해야 함** — *잡 A 완료를 기다려 잡 B 시작* 을 `@Scheduled` 로 표현하려면 *잡 A 끝에서 잡 B 를 직접 호출* 하거나 *상태 폴링* 을 짜야 합니다. 코드가 *워크플로우 엔진의 흉내* 가 됩니다.

**2. 애플리케이션이 떠 있어야 동작** — 배포 중에는 *스케줄도 같이 멈춥니다*. 잡이 새벽 2시고 배포가 1시 55분에 끝나면 *그날치 잡이 누락* 됩니다.

**3. 다중 인스턴스에서 동시 실행 위험** — 여러 인스턴스가 떠 있으면 *모두 동시에* 잡을 띄웁니다. *분산 락* (ShedLock 같은 라이브러리) 으로 막아야 합니다.

`@Scheduled` 의 자세한 옵션과 한계는 [`../../11_spring/05_aop/01-02.스프링 스케줄링 — @Scheduled에서 Quartz까지.md`](../../11_spring/05_aop/01-02.스프링%20스케줄링%20—%20%40Scheduled에서%20Quartz까지.md) 에서 다룹니다.



## 2단계: Quartz — 상태 보존 스케줄러

> Quartz 는 *스케줄 상태를 DB 에 저장* 합니다. 애플리케이션이 죽었다 살아나도 *놓친 트리거를 보정* 할 수 있고 (`misfireInstruction`), *클러스터 모드* 에서 *여러 인스턴스 중 하나만* 실행을 보장합니다.

```xml
<bean id="jobDetail" class="org.springframework.scheduling.quartz.JobDetailBean">
    <property name="jobClass" value="org.springframework.batch.samples.misc.quartz.JobLauncherDetails"/>
    <property name="group" value="quartz-batch"/>
    <property name="jobDataAsMap">
        <map>
            <entry key="jobName" value="footballJob"/>
            <entry key="jobLocator" value-ref="jobRegistry"/>
            <entry key="jobLauncher" value-ref="jobLauncher"/>
        </map>
    </property>
</bean>
```

Quartz 의 `JobDetail` 안에서 Spring Batch 의 `JobLauncher` 를 호출하는 *2층 구조* 입니다. Quartz 는 *언제* 를 결정하고, Spring Batch 가 *무엇을 어떻게* 를 처리합니다.

Quartz 가 풀어주는 자리들입니다.

| 문제 | Quartz 의 답 |
|------|------------|
| 잡이 자주 멈춰 트리거 누락 | `misfireInstruction` 으로 정책별 보정 (즉시 실행 / 다음 트리거 / 무시) |
| 다중 인스턴스 동시 실행 | 클러스터 모드 + DB 락 |
| 스케줄을 코드 배포 없이 바꿈 | `QuartzScheduler.scheduleJob` 의 동적 등록 API |
| 복잡한 cron + 휴일·예외 | Calendar 추상화 |

Quartz 의 한계는 *잡 사이의 의존 그래프* 입니다. *잡 A 완료 → 잡 B 시작* 같은 *데이터 흐름* 은 *트리거 시간이 아니라 이전 잡의 상태에 의존* 합니다. Quartz 의 트리거는 *시간 기반* 이라 이 모델이 어색합니다.



## 3단계: Argo Workflows·Airflow — 워크플로우 엔진

> *여러 잡 사이의 DAG (Directed Acyclic Graph)*, *조건 분기*, *파라미터 전달*, *실패 시 재시도 정책*, *시각화 UI* 까지 책임지는 게 워크플로우 엔진입니다. Argo Workflows (K8s 네이티브) 와 Airflow 가 주류입니다.

```yaml
# Argo Workflows DAG 예시
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  name: etl-pipeline
spec:
  entrypoint: main
  templates:
  - name: main
    dag:
      tasks:
      - name: extract
        template: launch-batch-job
        arguments:
          parameters:
          - name: job-name
            value: extractJob
      - name: transform
        depends: extract
        template: launch-batch-job
        arguments:
          parameters:
          - name: job-name
            value: transformJob
      - name: load
        depends: transform
        template: launch-batch-job
        arguments:
          parameters:
          - name: job-name
            value: loadJob
  - name: launch-batch-job
    inputs:
      parameters:
      - name: job-name
    container:
      image: my-batch-app:latest
      command: ["java", "-jar", "/app.jar"]
      args: ["--spring.batch.job.name={{inputs.parameters.job-name}}"]
```

이 모델에서 *Spring Batch 의 잡은 K8s Pod 한 개* 가 됩니다. Argo 가 Pod 를 띄우고, Pod 안에서 *CommandLineJobRunner* (또는 Spring Boot 의 `spring.batch.job.name`) 로 잡 하나만 돌고 끝납니다.

이 분담의 장점입니다.

**Spring Batch 는 단일 잡 안의 깊이만** — chunk·재시작·skip·메타테이블. *외부와 무관한* 잡 내부의 표준.

**Argo·Airflow 는 잡 바깥의 흐름만** — DAG 의존, 파라미터 전달 (앞 잡 결과를 다음 잡 입력으로), 실패 시 재시도 / 알림, 시각화 UI.

**잡과 잡 사이의 데이터** — *큐* (Kafka), *오브젝트 스토리지* (S3·GCS), *공유 DB* 중 하나를 통합니다. Argo·Airflow 가 *데이터를 직접 옮기진 않습니다*. 잡이 *S3 에 파일을 두고* 다음 잡이 *그 S3 경로를 입력으로 받는* 형태가 표준.



## ETL 파이프라인 — 세 잡으로 표현

> ETL (Extract · Transform · Load) 은 *3단 데이터 파이프라인* 의 일반 명명입니다. Spring Batch + 워크플로우 엔진의 조합이 정확히 이 구조를 받쳐줍니다.

```
[원본 DB / API]
     ↓ (Extract Job)
[S3 / 임시 테이블]
     ↓ (Transform Job)
[정제된 형식의 S3 / 임시 테이블]
     ↓ (Load Job)
[목적 DB / 데이터 웨어하우스]
```

각 단계가 *독립된 Spring Batch Job* 입니다. *각 잡은 자체적으로 chunk·재시작·skip 보장* 을 가지고, *워크플로우 엔진이 셋을 순차 호출* 합니다.

세 잡으로 *분리하는 이유* 가 있습니다.

**재실행 단위 분리** — Transform 만 로직 변경됐을 때 *Transform 잡만 다시 도는* 게 자연스럽습니다. 셋이 한 잡이면 *Extract 부터 다시* 합니다.

**자원 프로파일 분리** — Extract 는 *네트워크 IO* 가 메인, Transform 은 *CPU 가 메인*, Load 는 *DB IO 가 메인*. 셋에 다른 자원을 줘야 효율적입니다. K8s 의 Pod 단위로 자원 limit 이 분리됩니다.

**관측성 분리** — 각 잡의 *처리 건수·소요 시간* 이 독립 메트릭으로 노출됩니다. *왜 어제보다 느린가* 를 단계별로 진단할 수 있습니다.



## 운영 관측성 — 잡이 도는 것을 어떻게 보는가

> 잡이 동작 중인지·실패했는지·얼마나 걸리는지를 *관측* 할 수 없으면 운영이 안 됩니다. Spring Batch 의 *메타테이블* 이 진실 공급원이지만, 그것만으로는 *운영 대시보드* 가 안 됩니다.

세 가지 관측 채널이 표준입니다.

**1. Spring Batch 의 메타테이블** — `BATCH_JOB_EXECUTION`, `BATCH_STEP_EXECUTION` 의 STATUS·EXIT_CODE·READ_COUNT·WRITE_COUNT. *SQL 로 직접 조회* 가능. `JobExplorer` API 도 있습니다 ([`./01-03`](01-03.JobRepository와%20메타테이블%20—%20JobInstance·JobExecution·StepExecution.md)).

**2. Micrometer 메트릭** — Spring Batch 5.x 가 *자동으로* Micrometer 메트릭을 노출합니다. `spring.batch.job.active`, `spring.batch.job.completed`, `spring.batch.step.execution.duration` 같은 이름. Prometheus 로 긁어 Grafana 대시보드.

**3. 로그·트레이스** — chunk 의 처리 시간, 외부 호출 분포 같은 디테일은 *로그* 와 *Distributed Tracing* (OpenTelemetry) 가 받습니다. `01-04` 의 재시작 위치 추적도 결국 *로그에 남은 chunk 커밋 시각* 이 가장 빠른 진단입니다.

운영 환경에서는 *세 채널을 모두* 켜는 게 표준입니다. 메타테이블이 *과거 잡의 진실*, Micrometer 가 *실시간 추이*, 로그·트레이스가 *디테일* 입니다.



## 실패 알림과 재시도 정책

잡 실패 시 *누구에게 어떻게 알릴지* 와 *자동 재시도를 얼마나 할지* 의 정책 자리입니다.

**자동 재시도** — 단순 시간 기반 (`@Scheduled` 또는 Quartz) 이면 *재시도 정책이 별도 코드* 입니다. Argo·Airflow 는 *DAG 단위 재시도* 와 *백오프* 를 디폴트 기능으로 가집니다. Argo 의 `retryStrategy` 또는 Airflow 의 `retries`, `retry_delay` 가 그 자리입니다.

**알림** — Spring Batch 의 `JobExecutionListener.afterJob` 에서 *Slack·이메일·PagerDuty* 로 발행. 또는 워크플로우 엔진의 *실패 알림* 통합 (Argo 의 Sensor + Slack template).

**알림의 조건** — *매번 실패할 때마다* 가 아니라 *재시도 후에도 실패할 때* 또는 *어제와 다른 패턴의 실패* 가 운영자에게 의미 있습니다. *알림 피로* 를 피하는 게 운영 안정성의 핵심.



## 운영 인프라 경계 — Spring Batch 가 어디서 끝나는가

본 시리즈 첫 편 [`./01-01`](01-01.Spring%20Batch%20골격%20—%20Job·Step·Chunk·Tasklet.md) 에서 시작한 *Spring Batch 의 책임 범위* 가 본 편에서 명확히 닫힙니다.

| 책임 | 위치 |
|------|------|
| Job·Step 정의, chunk 처리, 재시작, skip·retry, 메타테이블 기록 | Spring Batch |
| 잡을 *언제* 시작할지 | `@Scheduled` / Quartz / 워크플로우 엔진 |
| 잡들 *사이의* 흐름·의존 | 워크플로우 엔진 (Argo / Airflow) |
| 잡 *바깥의* 데이터 저장소 | S3·DB·Kafka |
| 잡의 *실행 환경* (자원·격리) | K8s Pod / VM |
| 잡 *결과의* 알림·대시보드 | Micrometer + Prometheus + Grafana + Slack |

이 분담이 *Spring Batch 를 단일 잡에 집중* 하게 만들고, *각 외부 시스템을 표준 도구* 로 재사용 가능하게 만듭니다.
