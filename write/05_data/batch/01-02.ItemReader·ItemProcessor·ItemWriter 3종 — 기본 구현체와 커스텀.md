# ItemReader·ItemProcessor·ItemWriter 3종 — 기본 구현체와 커스텀

---

> Chunk Step 안에서 한 사이클을 만드는 부품 세 가지의 역할과 카탈로그를 정리합니다. Reader 가 한 건씩 *읽고*, Processor 가 한 건씩 *변환·필터링* 하고, Writer 가 모인 묶음을 *쓰는* 흐름입니다. 핵심은 *세 인터페이스 시그니처가 단순하다* 는 점입니다. 기본 구현체 카탈로그를 알아두면 90% 의 ETL 케이스는 한 줄도 안 짜고 끝납니다. 커스텀이 필요한 자리는 명확히 갈립니다.



## 인터페이스 3개의 시그니처

> 세 인터페이스는 메서드 한 개씩입니다. 짧고 명확합니다.

```java
public interface ItemReader<T> {
    T read() throws Exception, UnexpectedInputException,
                    ParseException, NonTransientResourceException;
}

public interface ItemProcessor<I, O> {
    @Nullable O process(I item) throws Exception;
}

public interface ItemWriter<T> {
    void write(Chunk<? extends T> chunk) throws Exception;
}
```

Reader 는 *한 건만 리턴* 합니다. 더 읽을 게 없으면 `null` 을 리턴해 Step 종료를 알립니다 (Tasklet 이 `RepeatStatus.FINISHED` 로 끝나는 것과 같은 역할).

Processor 는 *입력 타입 I 와 출력 타입 O 가 다를 수 있습니다*. CSV 의 `Customer` 를 받아 DB INSERT 용 `CustomerEntity` 로 바꾸는 자리에 정확히 맞습니다. `process` 가 `null` 을 리턴하면 그 건은 *필터링* 되고, Writer 까지 가지 않습니다. StepExecution 의 `filterCount` 가 1 증가합니다.

Writer 는 *묶음 단위* 로 받습니다. Reader 가 한 건씩 채워 쌓은 `Chunk<T>` 를 통째로 받아 한 번에 씁니다. *INSERT N 개를 batch 로 묶어 보내는* 자리가 여기입니다.



## Reader 카탈로그 — 입력 소스별

> Reader 는 *어디서 읽는가* 로 갈립니다. 파일·DB·메시지 큐가 주력 셋이고, 각자 *어떻게 커서를 유지하는가* 가 핵심 차이입니다.

| 소스 | 구현체 | 핵심 동작 |
|------|--------|----------|
| 플랫 파일 (CSV·TSV·고정폭) | `FlatFileItemReader<T>` | 한 줄씩 읽어 `LineMapper` 로 도메인 객체로 변환 |
| XML | `StaxEventItemReader<T>` | StAX 이벤트 기반으로 한 fragment 씩 읽기 |
| JSON | `JsonItemReader<T>` | Jackson 기반, 배열의 한 요소씩 |
| JDBC (커서) | `JdbcCursorItemReader<T>` | DB 커서 유지하며 한 행씩, RowMapper 로 변환 |
| JDBC (페이징) | `JdbcPagingItemReader<T>` | 페이지 단위 SQL 재실행, 커서 유지 안 함 |
| JPA (커서) | `JpaCursorItemReader<T>` | EntityManager 의 `Query.getResultStream()` 활용 |
| JPA (페이징) | `JpaPagingItemReader<T>` | `setFirstResult` + `setMaxResults` 반복 |
| Kafka | `KafkaItemReader<K,V>` | 토픽 파티션 별 offset 진행 |
| 커스텀 | `ItemReader<T>` 직접 구현 | 외부 API·gRPC 등 |

JDBC 의 *커서 vs 페이징* 선택이 면접 단골입니다. **커서** 는 DB 가 결과셋 위치를 들고 있어 *한 번의 SQL* 로 끝나지만, 잡 실행 동안 *DB 커넥션을 점유* 합니다. **페이징** 은 매번 새 SQL 을 던지므로 *커넥션 점유 시간이 짧지만* `OFFSET` 이 커질수록 느려집니다. 100 만 건의 짧은 잡이면 커서, 1 억 건의 긴 잡이면 페이징이 답입니다.

### FlatFileItemReader 예시

```java
@Bean
@StepScope
public FlatFileItemReader<Customer> customerReader(
        @Value("#{jobParameters['inputFile']}") String inputFile) {
    return new FlatFileItemReaderBuilder<Customer>()
            .name("customerReader")
            .resource(new FileSystemResource(inputFile))
            .delimited()
                .names("name", "city", "age")
            .fieldSetMapper(new BeanWrapperFieldSetMapper<>() {{
                setTargetType(Customer.class);
            }})
            .build();
}
```

`@StepScope` 가 있어야 `jobParameters` 가 *Step 실행 시점* 에 주입됩니다. 빈 등록 시점 (애플리케이션 시작 시) 에는 jobParameters 가 존재하지 않으니 필수입니다.

`FlatFileItemReaderBuilder` 의 `.delimited()` 가 `DelimitedLineTokenizer` 를, `.fixedLength()` 가 `FixedLengthTokenizer` 를 자동으로 붙입니다. 직접 `LineMapper` 를 짜는 자리는 *한 파일에 여러 레코드 형태가 섞여 있는* 경우입니다 (`PatternMatchingCompositeLineMapper` 로 첫 칸 prefix 별로 분기).

### JdbcCursorItemReader 예시

```java
@Bean
public JdbcCursorItemReader<Trade> tradeReader(DataSource dataSource) {
    return new JdbcCursorItemReaderBuilder<Trade>()
            .name("tradeReader")
            .dataSource(dataSource)
            .sql("SELECT trade_id, version, company_name, issue_date, amount " +
                 "FROM trades ORDER BY trade_id")
            .rowMapper(new TradeRowMapper())
            .build();
}
```

`ORDER BY` 가 *재시작 시 같은 순서* 를 보장하는 핵심입니다. ORDER BY 가 없으면 PostgreSQL·MySQL 모두 정렬 순서를 보장하지 않으므로 재시작 시 어디까지 처리했는지 추적이 무너집니다. 자세한 재시작 메커니즘은 `01-04` 에서 다룹니다.



## Processor 카탈로그 — 변환·필터링·체이닝

> Processor 는 *선택* 입니다. Reader 가 읽은 객체를 Writer 가 그대로 쓰는 ETL 도 자주 있습니다. Processor 가 들어오는 자리는 세 가지입니다.

**변환** — 타입 I 에서 O 로 바꿉니다. CSV 의 `RawCustomer` 를 DB INSERT 용 `CustomerEntity` 로 바꾸는 자리. Reader 가 *읽기 모델* 을, Processor 가 *쓰기 모델* 로의 매핑을 책임지면 Reader·Writer 의 도메인 결합이 풀립니다.

**필터링** — `process` 가 `null` 을 리턴하면 그 건은 Writer 까지 가지 않습니다. *나이 18 세 미만 제외* 같은 단일 조건은 이 자리입니다. 단언만 한 줄짜리이면 람다도 됩니다.

```java
@Bean
public ItemProcessor<Customer, Customer> adultsOnly() {
    return customer -> customer.getAge() >= 18 ? customer : null;
}
```

**검증** — 잘못된 입력에 예외를 던집니다. `BeanValidatingItemProcessor` 가 Jakarta Bean Validation (`@NotNull`, `@Email` 같은 어노테이션) 을 그대로 활용해 *어노테이션 기반 검증* 을 붙입니다. 검증 실패는 예외로 표현되어 `skip` 정책에 걸립니다.

### CompositeItemProcessor — 체이닝

여러 Processor 를 *체인* 으로 묶어 *변환 → 검증 → 필터* 순으로 흘릴 수 있습니다.

```java
@Bean
public CompositeItemProcessor<Customer, CustomerEntity> chained(
        ItemProcessor<Customer, ValidatedCustomer> validator,
        ItemProcessor<ValidatedCustomer, CustomerEntity> mapper) {
    CompositeItemProcessor<Customer, CustomerEntity> composite =
            new CompositeItemProcessor<>();
    composite.setDelegates(List.of(validator, mapper));
    return composite;
}
```

한 단계의 출력이 다음 단계의 입력으로 갑니다. 한 단계라도 `null` 을 리턴하면 그 건은 거기서 끝나고 다음 단계로 안 갑니다. 변환·검증·필터링이 한 클래스에 뭉치는 *비대한 Processor* 를 피하는 도구입니다.



## Writer 카탈로그 — 출력 타깃별

> Writer 는 *Chunk 한 묶음을 한 번에* 받습니다. *INSERT 100 개를 한 batch 로 묶어 보내는* 자리가 여기입니다. 단건 INSERT 를 100 번 도는 흉내를 내면 chunk 모델의 성능 이점이 사라집니다.

| 타깃 | 구현체 | 핵심 동작 |
|------|--------|----------|
| 플랫 파일 | `FlatFileItemWriter<T>` | `LineAggregator` 로 객체 → 한 줄, 파일 끝에 append |
| XML | `StaxEventItemWriter<T>` | 한 fragment 씩 XML 로 |
| JSON | `JsonFileItemWriter<T>` | Jackson 기반 |
| JDBC | `JdbcBatchItemWriter<T>` | `NamedParameterJdbcTemplate` 의 `executeBatch` 로 묶어 INSERT/UPDATE |
| JPA | `JpaItemWriter<T>` | `EntityManager.persist` 또는 `merge` 반복 |
| MongoDB | `MongoItemWriter<T>` | `MongoOperations.insert/save` 묶음 |
| Kafka | `KafkaItemWriter<K,V>` | `KafkaTemplate.send` 묶음 |
| Composite | `CompositeItemWriter<T>` | 한 묶음을 두 Writer 에 동시에 (예: DB + 로그 파일) |

### JdbcBatchItemWriter 가 진짜 batch 인지

`JdbcBatchItemWriter` 의 이름이 그렇듯, 내부에서 `NamedParameterJdbcTemplate.batchUpdate` 를 호출합니다. JDBC 드라이버가 `executeBatch` 를 지원하면 *한 번의 네트워크 왕복* 으로 N 개 INSERT 가 나갑니다.

```java
@Bean
public JdbcBatchItemWriter<Trade> tradeWriter(DataSource dataSource) {
    return new JdbcBatchItemWriterBuilder<Trade>()
            .dataSource(dataSource)
            .sql("UPDATE trades SET version = :version, " +
                 "company_name = :companyName, amount = :amount " +
                 "WHERE trade_id = :tradeId")
            .beanMapped()
            .build();
}
```

`.beanMapped()` 가 `BeanPropertyItemSqlParameterSourceProvider` 를 붙여 *객체의 게터 이름* 으로 SQL 파라미터를 채웁니다. 컬럼명을 일일이 매핑하지 않아도 됩니다. JPA 와 결합한 경우 *JPA batch_size 설정* 과의 함정이 따로 있고, 그 내용은 `01-06` 에서 따로 다룹니다.

### JpaItemWriter 의 한계

`JpaItemWriter` 는 *내부에서 `EntityManager.persist` 또는 `merge` 를 한 건씩 호출* 합니다. 진짜 batch INSERT 가 나가려면 *Hibernate 의 `hibernate.jdbc.batch_size`* 와 *batch 정렬 옵션* (`order_inserts`, `order_updates`) 이 같이 설정돼야 합니다. 설정 없이는 JPA 가 1 차 캐시 비우는 시점마다 1 건씩 INSERT 를 날립니다. `JdbcBatchItemWriter` 가 *명시적으로 batch* 라면, `JpaItemWriter` 는 *Hibernate 설정에 의존* 한다는 차이가 핵심입니다.



## ItemStream — 재시작을 받쳐주는 인터페이스

> Reader·Writer 중 *재시작 시 어디부터 다시 해야 하는지* 를 알아야 하는 부품은 `ItemStream` 을 같이 구현합니다.

```java
public interface ItemStream {
    void open(ExecutionContext ctx) throws ItemStreamException;
    void update(ExecutionContext ctx) throws ItemStreamException;
    void close() throws ItemStreamException;
}
```

`open` 은 Step 시작 시 호출되며 *이전 실행에서 멈춘 지점* 을 `ExecutionContext` 에서 꺼내 복구합니다. `update` 는 chunk 커밋 직전 호출되어 *현재 진행 위치* 를 저장합니다. `close` 는 Step 종료 시 자원 해제입니다.

대부분의 *기본 Reader 구현체* (`FlatFileItemReader`, `JdbcCursorItemReader`, `JpaPagingItemReader` 등) 가 `ItemStream` 을 같이 구현하고 있어 재시작이 *자동으로* 동작합니다. **커스텀 Reader 를 짤 때 `ItemReader` 만 구현하고 `ItemStream` 을 빼면 재시작 시 처음부터 다시 돕니다.** 면접에서 *왜 커스텀 Reader 가 재시작 안 되나* 라는 질문의 답이 여기입니다.

자세한 재시작 메커니즘과 `ExecutionContext` 시리얼라이제이션은 `01-04` 에서 다룹니다.



## 언제 커스텀을 만드는가

> 카탈로그에 *없는 입력 소스* 또는 *없는 출력 타깃* 일 때만 커스텀입니다. 그 외에는 기본 구현체에 *RowMapper·LineMapper·SqlParameterSourceProvider* 만 갈아 끼우면 끝납니다.

세 가지 자리가 커스텀의 정당한 영역입니다.

**외부 API·gRPC 호출** — REST API 의 페이지네이션을 따라가며 한 페이지씩 읽는 Reader. `RestClient` 와 결합해 `read()` 안에서 *내부 버퍼* 를 들고 한 건씩 꺼내고, 버퍼가 비면 다음 페이지를 가져오는 패턴이 표준입니다.

**큐 폴링** — RabbitMQ·SQS 같은 *Kafka 가 아닌* 메시지 큐. 한 건 받아 리턴, 큐가 비면 `null` 로 종료.

**상태가 있는 Reader** — 외부 시스템에서 *세션을 열고 데이터를 받는* 형태 (예: SFTP 디렉토리 순회). 이 경우 반드시 `ItemStream` 까지 같이 구현해 재시작 가능하게 만듭니다.

### 커스텀 Reader 의 기본 골격

```java
public class CustomItemReader<T> implements ItemReader<T>, ItemStream {

    private List<T> buffer = new ArrayList<>();
    private int cursor = 0;

    @Override
    public T read() {
        if (cursor >= buffer.size()) {
            refillBuffer();        // 외부에서 다음 페이지 가져오기
            if (buffer.isEmpty()) {
                return null;       // 더 읽을 게 없으면 종료 신호
            }
            cursor = 0;
        }
        return buffer.get(cursor++);
    }

    @Override
    public void open(ExecutionContext ctx) {
        if (ctx.containsKey("cursor")) {
            cursor = ctx.getInt("cursor");
        }
    }

    @Override
    public void update(ExecutionContext ctx) {
        ctx.putInt("cursor", cursor);
    }

    @Override
    public void close() { /* 리소스 해제 */ }
}
```

`open` 에서 *이전 실행의 cursor 위치* 를 복구하고, `update` 에서 *현재 cursor* 를 저장합니다. 이 두 메서드를 빠뜨리면 잡이 재시작될 때 처음부터 다시 시작합니다.



## 운영 카운터 — StepExecution 의 readCount·writeCount·filterCount

> 세 부품의 동작은 *카운터로 관측* 됩니다. 메타테이블에 기록되어 운영 시 어디서 무엇이 막혔는지 추적합니다.

| 카운터 | 의미 |
|--------|------|
| `readCount` | Reader 가 성공적으로 읽은 건수 |
| `filterCount` | Processor 가 `null` 리턴으로 필터링한 건수 |
| `writeCount` | Writer 가 성공적으로 쓴 건수 |
| `readSkipCount` | Reader 에서 예외 발생 후 skip 정책으로 넘어간 건수 |
| `processSkipCount` | Processor 에서 예외 후 skip 된 건수 |
| `writeSkipCount` | Writer 에서 예외 후 skip 된 건수 |
| `commitCount` | chunk 커밋 횟수 (`readCount / chunkSize` 근사) |
| `rollbackCount` | chunk 롤백 횟수 |

`readCount = writeCount + filterCount + skipCount` 가 *대체로* 성립합니다. 정확한 차이는 Processor 가 *분기를 만드는* 경우 (한 입력에서 여러 출력이 나오는 변형) 에서 깨지지만, 표준 ETL 에서는 이 등식이 *디버깅 기준* 입니다. `readCount` 는 1000 인데 `writeCount` 가 0 이면 Processor 가 전부 필터링했거나 Writer 가 전부 예외를 냈다는 신호입니다.
