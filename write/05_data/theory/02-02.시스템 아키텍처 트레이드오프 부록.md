# 시스템 아키텍처 트레이드오프 부록 — 비기능 요구사항

---

> 시스템이 *무엇을* 하느냐(기능 요구) 만큼이나 *어떻게* 동작하느냐(비기능 요구) 가 운영을 가른다. 본 챕터는 성능·신뢰성·확장성·유지보수성 네 축을 정리한다. 응답 시간은 평균이 아닌 백분위수로 측정하고, 신뢰성은 장애 허용으로 달성하며, 확장성은 일차원 레이블이 아닌 "어떤 부하 증가에 어떤 길로 답하는가" 의 질문이다.



## 사례 — SNS 홈 타임라인이 던지는 질문

> 추상적인 비기능 요구를 구체 사례로 끌어 내린다. 트위터 규모의 타임라인이 좋은 출발점이다.

가정해 보자. 하루 5 억 포스트(초당 5,800 건, 피크 15 만 건/초). 평균 사용자가 200 명 팔로우, 200 명 팔로워. 동시 접속자 1,000 만 명. 사용자가 자기 홈 타임라인을 5 초마다 새로고침한다.

가장 단순한 답은 매번 SQL 로 푸는 길이다.

```sql
SELECT posts.*, users.* FROM posts
  JOIN follows ON posts.sender_id = follows.followee_id
  JOIN users   ON posts.sender_id = users.id
  WHERE follows.follower_id = ?
  ORDER BY posts.timestamp DESC LIMIT 1000
```

5 초 폴링이면 초당 200 만 쿼리, 200 명 팔로우라면 포스트 조회가 초당 4 억 회. 단일 DB 로는 답이 없다.

운영의 답은 **머터리얼라이즈드 타임라인 + Fan-out** 이다. 새 포스트가 올라오는 순간 모든 팔로워의 타임라인 캐시에 미리 복사한다. 읽기는 캐시에서 즉시 끝난다. 비용 구조가 뒤집힌다 — 읽기는 거의 공짜, 쓰기가 초당 100 만 건. 읽기·쓰기 비율이 압도적으로 읽기 쪽인 SNS 워크로드에 맞는 결정이다.

극단 케이스가 따라온다. 셀럽 계정 하나의 포스트가 1 억 개 타임라인 갱신을 일으키면 fan-out 이 무너진다. 답은 하이브리드다. 일반 사용자는 fan-out, 셀럽은 별도 저장 후 읽기 시점에 병합한다. 이 결정 하나가 비기능 요구(성능·확장성) 의 거의 모든 갈래를 보여 준다.



## 성능 측정 — 응답 시간과 처리량

> 두 메트릭이 다른 질문에 답한다.

**응답 시간(Response Time)** 은 한 요청이 시작에서 끝까지 걸리는 시간이다. 사용자 경험의 직접 지표다. **처리량(Throughput)** 은 단위 시간당 처리한 요청 수다. 비용·용량 계획의 지표다.

둘은 서로 영향을 준다. 시스템이 한가할 때는 응답 시간이 일정하다가, 처리량이 시스템 한계에 가까워지면 큐잉 지연이 폭발해 응답 시간이 비선형으로 치솟는다. 고속도로의 정체 곡선과 같은 모양이다.

**Metastable Failure** 는 이 비선형의 어두운 면이다. 부하 증가 → 응답 시간 증가 → 클라이언트 타임아웃 → 자동 재시도 → 부하 더 증가 의 루프가 한 번 시작되면 외부 개입 없이는 안 멈춘다. 답은 네 가지 갈래다. **Exponential Backoff** 로 재시도 간격을 늘리고, **Circuit Breaker** 로 오류 서비스 호출을 일시 차단하며, **Load Shedding** 으로 과부하 시 선제적으로 요청을 거부하고, **Backpressure** 로 클라이언트에게 속도를 줄이라고 신호한다. 운영 시스템에 네 가지 모두 필요한 자리가 흔하다.



## 백분위수 — 평균이 숨기는 것

> 응답 시간 분포는 거의 항상 비대칭이다. 평균은 그 분포의 형태를 숨긴다.

응답 시간 분포는 보통 오른쪽 꼬리가 긴 모양이다. 대부분의 요청은 50ms 안에 끝나는데 1% 정도가 1 초 이상 걸리는 식이다. 평균을 보면 "그럭저럭 빠르다" 같은 답이 나오지만 p99 를 보면 사용자 일부가 끔찍한 경험을 하고 있다는 사실이 드러난다.

| 메트릭 | 의미 | 운영 용도 |
|--------|------|----------|
| Mean(평균) | 모든 값 산술 평균 | 처리량 한계 추정 |
| p50(median) | 절반이 이보다 빠름 | 일반적 사용자 경험 |
| p95 | 95% 가 이보다 빠름 | 대부분의 사용자 경험 |
| p99 | 99% 가 이보다 빠름 | 중요 고객 경험 |
| p999 | 99.9% 가 이보다 빠름 | 최악 케이스 |

Amazon 이 p999 까지 관리하는 이유는 분명하다. 가장 느린 요청을 받은 고객이 곧 가장 데이터가 많은 고객이고, 가장 가치 있는 고객일 가능성이 크다. p99 만 봐서는 그 1% 의 손해를 못 본다.

**Tail Latency Amplification** 이 분산 환경에서 더 무섭다. 한 사용자 요청이 4 개 백엔드를 병렬 호출한다고 하자. 각 백엔드의 p99 가 100ms 라면 전체 요청이 *모든 백엔드의 응답을 기다리므로* 어느 하나라도 p99 에 걸리면 전체 응답이 100ms 가 된다. 4 개 모두 정상 응답할 확률이 0.99⁴ ≈ 96% 라 약 4% 사용자가 p99 응답을 본다. 호출 fan-out 이 클수록 꼬리가 정상으로 옮겨 온다.

**SLO** 는 이 백분위수 위에 세우는 운영 목표다. "p99 응답 시간이 1초 이하, 가용성 99.9%" 같은 형태다. **SLA** 는 SLO 미달 시 계약상 책임이 따라오는 약속이다. 보통 외부 고객용 계약에만 SLA 가 붙고, 내부 팀 간 약속은 SLO 로 다룬다.

운영 한 가지 함정 — **백분위수를 평균내면 안 된다**. 두 노드의 p99 가 각각 100ms 라고 해서 합산 시스템의 p99 가 100ms 가 아니다. 히스토그램을 합산해야 정확하다. HdrHistogram·t-digest·DDSketch 같은 라이브러리가 이 합산을 효율적으로 해 준다.



## 신뢰성 — Fault 와 Failure

> 둘은 같지 않다. 운영 설계는 Fault 가 Failure 로 번지지 않도록 만드는 작업이다.

| 용어 | 정의 | 예시 |
|------|------|------|
| Fault | 시스템 일부의 오작동 | 디스크 1 개 고장 |
| Failure | 시스템 전체가 서비스 불가 | 서비스 다운 |
| SPOF | 장애 시 전체 실패 유발하는 단일 점 | 단일 마스터 DB |

**장애 허용(Fault Tolerance)** 은 특정 유형의 특정 수의 fault 를 받아 줘도 failure 가 안 일어나도록 설계하는 것이다. 모든 fault 를 허용할 수는 없다(데이터센터 전소, 지구 멸망). 어디까지를 받을지가 운영 결정이다.

**Chaos Engineering** 은 이 설계가 실제로 동작하는지 검증하는 운영 관행이다. 의도적으로 fault 를 주입(노드 죽이기, 네트워크 지연 추가) 해 가설이 맞는지 확인한다. Netflix 의 Chaos Monkey 가 시작이고 이제는 표준 도구가 됐다.



## 하드웨어와 소프트웨어 장애

> 두 종류의 장애가 다른 모양으로 온다.

| 부품 | 연간 장애율 | 비고 |
|------|----------|------|
| HDD | 2~5% | 1 만대 클러스터 → 매일 1 대 장애 |
| SSD | 0.5~1% | 수정 불가 오류 연 1 회/드라이브 |
| CPU | 0.1% | 잘못된 계산 결과 가능 |
| RAM | 1%+ (수정 불가 오류) | ECC 도 완벽 아님 |
| 데이터센터 | 드물지만 치명 | 화재·홍수·태양 폭풍 |

하드웨어 장애의 답은 **중복성** 이다. RAID(디스크 중복), 이중 전원, 다중 데이터센터(AZ), Rolling Upgrade(노드 하나씩 재시작) 가 표준 도구다. 통계적으로 예측 가능하므로 용량 계획이 가능하다.

소프트웨어 장애는 다르다. 같은 버그를 모든 노드가 공유하므로 동시에 무너진다. 2012 년 윤초 버그가 Linux 커널 자바 앱들을 동시에 멈췄고, SSD 펌웨어 버그가 정확히 32,768 시간(약 3.7 년) 후 모든 SSD 를 동시에 죽였다. **Cascading Failure** 도 같은 결의 사고다 — 한 컴포넌트 문제가 의존 컴포넌트로 전파되어 전체가 무너진다.

운영 데이터가 더 솔직하다. **인간이 일으킨 장애가 70~90%** 다. 운영자가 잘못된 설정을 배포하거나, 마이그레이션 중 실수하거나, 모니터링 alert 을 무시한다. 답은 사람을 비난하지 않는 것이다. **Blameless Postmortem** 은 "왜 그 실수가 가능했는가" 를 묻는다. 시스템 설계와 조직 우선순위가 진짜 원인이고 그것을 고쳐야 같은 실수가 다시 안 난다.



## 확장성 — 일차원 레이블이 아니다

> "이 시스템은 확장 가능하다" 는 무의미한 문장이다. 어떤 부하의 어떤 차원이 어떻게 늘 때 어떤 답을 갖는가 가 진짜 질문이다.

부하 기술의 차원은 워크로드마다 다르다. 초당 요청 수, 일일 데이터 유입량, 읽기/쓰기 비율, 캐시 적중률, 동시 접속자 수, 데이터 크기 같은 변수들이 모인다. SNS 라면 "팔로워 분포의 꼬리 두께" 도 핵심 변수가 된다(셀럽의 팔로워 1 억 명이 fan-out 모델을 깨므로).

확장 아키텍처는 세 갈래로 정리된다.

**Shared-Memory(Scale Up)** 은 한 머신에 더 많은 CPU·RAM·디스크를 박는다. 단순하다. 비용이 비선형으로 증가하고 단일 머신 한계가 분명하다. 100GB RAM 머신이 1TB RAM 머신보다 10 배 싸지 않다.

**Shared-Disk** 는 여러 컴퓨트 노드가 같은 SAN 을 본다. 전통적 데이터 웨어하우스 패턴이다. 잠금 경합이 확장의 한계가 된다.

**Shared-Nothing(Scale Out)** 은 각 노드가 자기 CPU·RAM·디스크를 갖고 데이터를 샤딩으로 분산한다([`./02-04.샤딩.md`](./02-04.샤딩.md) 가 다룸). 선형 확장이 가능하지만 분산의 모든 비용(네트워크·일관성·조정) 이 따라온다.

확장성 원칙 세 가지가 있다. 첫째는 작고 독립적인 컴포넌트로 분해(마이크로서비스·샤딩·스트림 처리). 둘째는 필요 이상으로 복잡하게 만들지 않기. 단일 머신 DB 로 충분하면 분산을 피하고, 예측 가능 부하면 수동 스케일링도 충분하다. 셋째는 부하가 10 배 늘 때마다 아키텍처를 재검토하는 것이다. 한 자리수 성장은 보통 같은 아키텍처가 받지만, 10 배 성장은 거의 항상 다른 아키텍처를 요구한다.



## 유지보수성 — Operability·Simplicity·Evolvability

> 소프트웨어 비용의 대부분이 초기 개발이 아니라 그 이후 유지보수에 들어간다. 이를 줄이는 세 가지 설계 축이 있다.

**Operability(운영성)** 는 운영 팀이 일하기 쉬운 정도다. 좋은 운영성을 가진 시스템은 주요 메트릭 모니터링을 받쳐 주고, 개별 머신 의존을 끊어 유지보수 중에도 서비스가 계속 돌며, 운영 모델이 명확히 문서화되고, 합리적인 기본값에 관리자 오버라이드 가능성이 함께 있고, 동작이 예측 가능해 놀랍지 않다.

**Simplicity(단순성)** 는 새로 합류한 엔지니어가 빠르게 이해할 수 있는 정도다. 복잡성에는 숨겨진 가정과 의도치 않은 결과가 따라온다. 추상화가 복잡성 관리의 도구다 — SQL 이 디스크 구조를 가리고, 트랜잭션이 동시성을 가리고, 컨테이너가 OS 차이를 가린다. 추상화 한 단계가 곧 새 학습 비용이라는 점도 같이 본다.

**Evolvability(발전성)** 는 요구사항이 바뀔 때 시스템이 따라가기 쉬운 정도다. 비가역적 결정을 줄이는 게 핵심이다. 설정 변경은 가역, 코드 롤백도 가역, DB 스키마 변경은 비가역에 가까움, 데이터 마이그레이션은 거의 완전히 비가역이다. 같은 요구를 비가역에서 가역으로 옮길 수 있다면(예: 마이그레이션 대신 dual-write + 점진 전환) 그게 거의 항상 운영 비용을 줄인다.



## 면접 대비 체크리스트

1. 응답 시간(Response Time)·서비스 시간(Service Time)·지연 시간(Latency) 의 차이를 정의할 수 있는가?
2. 평균이 아닌 백분위수로 성능을 보는 이유는? p99 와 p999 사이의 차이를 한 시나리오로 댈 수 있는가?
3. Tail Latency Amplification 이 분산 환경에서 어떻게 동작하는가? Fan-out 이 클수록 어떻게 되는가?
4. Metastable Failure 와 그 답 네 가지(Backoff·Circuit Breaker·Load Shedding·Backpressure) 는?
5. Fault 와 Failure 의 차이가 운영 설계에 어떤 결과를 가져오는가?
6. 하드웨어 장애와 소프트웨어 장애가 통계 분포 측면에서 다른 이유는? 어느 쪽이 더 무서운가?
7. "인간 오류" 가 장애의 70~90% 라는 데이터를 어떻게 해석해야 하는가? Blameless Postmortem 의 발상은?
8. Scale Up·Shared-Disk·Scale Out 세 갈래의 트레이드오프와 자기 시스템에 어느 것이 어울리는가?
9. 확장성 원칙 세 가지(분해·과잉 회피·10 배 재검토) 중 어느 것이 자기 팀에서 가장 약한가?
10. Operability·Simplicity·Evolvability 의 정의와 비가역성이 evolvability 에 어떤 비용을 만드는지?
