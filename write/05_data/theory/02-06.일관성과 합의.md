# 일관성과 합의

---

> 분산 시스템에서 *강한 일관성* 의 정확한 정의는 선형성(linearizability) 다. "데이터가 하나만 존재하고 모든 연산이 원자적으로 보이게 한다" 는 약속이다. 노드 장애를 견디면서 이 약속을 구현하려면 합의(consensus) 알고리즘이 필요하다. Raft·Paxos·Zab 이 그 자리에 있다. 본 챕터는 일관성 모델의 스펙트럼, 합의가 푸는 문제, 코디네이션 서비스(ZooKeeper·etcd) 의 운영 역할을 정리한다.



## 일관성의 두 갈래 — Eventual 과 Strong

> 분산 시스템 일관성 모델은 약함과 강함의 스펙트럼이다. 어디에 머물지가 가용성·지연 시간 비용을 결정한다.

가장 약한 모델은 **eventual consistency** 다. "쓰기 후 충분히 기다리면 모든 복제본이 같은 값을 본다" 는 약속만 한다. Cassandra·DynamoDB 의 기본이 이 위에서 동작하고, 가용성·지연 시간이 가장 좋다. 단점은 사용자가 보는 이상 현상이다 — read-your-writes 위반, monotonic read 위반 같은 것들([`./02-03.복제.md`](./02-03.복제.md) 참고).

가장 강한 모델이 **선형성(linearizability)** 이다. 시스템이 마치 하나의 머신처럼 동작하는 것처럼 보이게 만든다. 어떤 두 연산도 시간 순으로 비교 가능하고, 한 연산이 시작 전에 완료된 다른 연산의 결과를 못 보면 정합성이 깨진다.

그 사이에 여러 모델이 있다. **인과적 일관성(causal consistency)** 은 인과 관계가 있는 연산 순서만 보장한다 — "댓글이 글보다 늦게 보이지 않는다". **순차 일관성(sequential consistency)** 은 모든 연산에 *어떤* 전역 순서가 있고 모든 노드가 같은 순서로 본다는 약속이다 — 시간 순일 필요는 없다. **세션 일관성** 은 한 사용자 세션 안에서만 일관성을 보장한다.

선택 기준은 단순하다. 가용성과 지연 시간이 핵심이면 eventual, 정확성이 핵심이면 linearizable, 둘 사이의 절충이면 causal. 운영 환경 대부분이 인과적 일관성 + 보강 패턴(예: 사용자 세션 핀·CRDT) 으로 동작한다.



## 선형성 — 정의와 비용

> "어떤 시점에 어떤 값이 있는지 모두가 동의한다" 는 약속이다. 멀티 코어 프로그래밍의 atomic 변수 의미를 분산 환경으로 끌어 올린 모델이다.

엄밀한 정의는 외부 관찰자 관점에서 동작한다. 모든 연산이 *어떤 순간에* 원자적으로 일어난 것처럼 보여야 하고, 그 순서가 실제 시간 순과 일치해야 한다(real-time order 보장). A 의 쓰기가 끝난 *후* B 의 읽기가 시작됐다면 B 는 A 의 결과 또는 더 새로운 결과를 봐야 한다.

선형성을 깨는 흔한 자리들이 있다. **CDN 캐시** 가 옛 버전을 보여 주고, **DB 복제 지연** 이 팔로워에서 옛 데이터를 노출하며, **다중 리더 환경** 에서 두 리더가 다른 값을 동시에 받는다. 이 모든 자리에서 선형성을 회복하려면 단일 리더로 돌아가거나 합의 알고리즘 위에 시스템을 올려야 한다.

선형성의 비용은 분명하다. **CAP 정리** 가 그 비용을 표현한다 — 네트워크 분할 시 가용성과 일관성 중 하나를 선택해야 한다. 선형성을 고집하면 분할된 소수 측이 응답을 멈춰야 한다(불가용). DDIA 의 **PACELC** 는 분할이 없을 때도 일관성을 위해 지연 시간을 희생해야 한다는 점을 추가한다 — Even when no partition, choose between Latency and Consistency.

운영에서는 선형성이 정말 필요한 자리에만 그 비용을 받아 준다. 결제·재고 차감·리더 선출 같은 자리. 사용자 프로필 조회·검색 결과·소셜 피드 같은 자리는 인과적 또는 eventual 로 가는 게 운영 비용 대비 가치가 더 크다.



## 논리 클럭 — Lamport 와 Vector 와 HLC

> 시계 동기화([`./02-05.분산 시스템의 문제점.md`](./02-05.분산%20시스템의%20문제점.md) 의 시계 절) 에 의존 안 하고도 인과 관계를 추적할 수 있다. 그 도구가 논리 클럭이다.

**Lamport 클럭** 은 각 노드가 자기 카운터를 들고, 메시지 송수신마다 카운터를 증가시킨다. 메시지를 받을 때 받은 timestamp 와 자기 카운터 중 더 큰 값을 선택해 +1 한다. 이렇게 하면 인과 관계가 있는 두 사건의 timestamp 가 자연스럽게 정렬된다. 단점은 **인과 무관 사건과 동시 사건을 구분하지 못한다** 는 점이다 — A 가 1, B 가 2 라고 해서 A 가 B 의 원인이라고 단정 못 한다.

**Vector 클럭** 이 그 차이를 푼다. 각 노드가 *모든 노드의* 카운터를 벡터로 들고 다닌다. 두 timestamp 를 비교할 때 모든 차원에서 한쪽이 크거나 같으면 인과 관계, 그렇지 않으면 동시. DynamoDB 의 동시 쓰기 감지([`./02-03.복제.md`](./02-03.복제.md) 의 Version Vector 절) 가 이 도구 위에서 동작한다.

**Hybrid Logical Clock(HLC)** 은 더 새로운 변형이다. 물리 시계와 논리 카운터를 결합해, 시계가 동기화되어 있을 때는 물리 시계 의미를 유지하고 그렇지 않을 때는 논리 카운터로 인과 관계를 보장한다. CockroachDB·MongoDB 가 이 위에서 분산 트랜잭션을 짠다. 시계 오차 범위가 작은 클라우드 환경에서 표준 도구가 되어 가는 중이다.



## 합의 — 무엇을 푸는 문제인가

> 분산 시스템의 모든 어려운 문제가 결국 같은 문제로 귀결된다. 이를 **합의(consensus)** 라고 부른다.

합의는 "여러 노드가 *같은 값* 에 동의한다" 는 약속이다. 단순해 보이지만 노드 장애·네트워크 분할이 있을 때 보장이 안 된다. 어떤 노드는 값 A 에 투표했는데 그 응답이 사라지고, 다른 노드는 값 B 에 투표하는 식이다.

여러 분산 문제가 사실 합의의 다른 표현이다.

| 문제 | 합의로 표현 |
|------|------------|
| 리더 선출 | 누가 리더인가 에 모두 동의 |
| 분산 락 획득 | 누가 락을 가지는가 에 동의 |
| 분산 트랜잭션 커밋 | 커밋·롤백 어느 결정인가 에 동의 |
| 멤버십 관리 | 클러스터에 어떤 노드가 있는가 에 동의 |
| 유일 ID 할당 | 누가 어느 ID 를 가지는가 에 동의 |

이 모든 자리에서 같은 알고리즘이 동작한다. **Total Order Broadcast** — 모든 노드에 같은 순서로 메시지를 전달하는 메커니즘 — 이 합의와 *동등* 한 문제임이 증명되어 있다. 한쪽이 풀리면 다른 쪽도 자동으로 풀린다.

합의 알고리즘의 표준이 된 도구는 두 갈래다.

**Paxos** 는 1989 년 Leslie Lamport 가 제안한 원조다. 정확하지만 이해하기 어렵다는 평이 표준이라 운영 구현이 드물다. Google Chubby·Spanner 가 변형을 쓴다.

**Raft** 는 2014 년 Stanford 가 "이해하기 쉬운 합의" 를 목표로 설계한 알고리즘이다. 리더 선출, 로그 복제, 안전성 세 단계로 명확히 갈라 설명한다. etcd·CockroachDB·TiKV·HashiCorp Consul 이 모두 Raft 위에 있다.

**Zab(ZooKeeper Atomic Broadcast)** 은 ZooKeeper 의 합의 프로토콜이다. Paxos 변형이지만 ZooKeeper 의 특수한 요구(primary-backup 모델) 에 맞춰져 있다.

세 알고리즘 모두 다음 약속을 공유한다. 노드 N 개 클러스터에서 N/2+1 노드가 살아 있고 서로 통신 가능하면 합의에 도달한다. 그보다 적으면 진행하지 않는다(가용성 희생). FLP 불가능성 정리([`./02-05.분산 시스템의 문제점.md`](./02-05.분산%20시스템의%20문제점.md) 참고) 의 우회로 부분 동기 가정을 사용한다 — 보통은 동기적이지만 가끔 임의로 길어질 수 있다는 가정.



## 코디네이션 서비스 — ZooKeeper·etcd

> 합의 알고리즘 위에 만들어진 운영 도구다. 분산 시스템의 "메타데이터" 를 안전하게 보관한다.

ZooKeeper 와 etcd 가 받쳐 주는 핵심 기능은 다섯 가지다.

**선형화 가능한 KV 저장소** — 작은 메타데이터(설정·리더 정보) 를 강한 일관성으로 읽고 쓴다. **분산 락** — 여러 노드가 같은 자원에 동시 접근 못 하도록 잠근다. **리더 선출** — 클라이언트들이 합의로 한 노드를 리더로 뽑는다. **멤버십 관리** — 클러스터에 어떤 노드가 살아 있는지 추적한다. **변경 알림(watch)** — 키의 값이 바뀌면 구독자에게 즉시 알린다.

운영 환경에서 이 다섯 기능이 자주 함께 쓰인다. Kubernetes 가 etcd 위에서 cluster state 를 보관하고, 모든 컨트롤러가 etcd watch 로 변경을 알아챈다. Kafka 의 컨트롤러 선출이 ZooKeeper(또는 KRaft) 위에서 동작한다. 운영 시스템 거의 모든 자리에 코디네이션 서비스가 한 자리는 차지한다.

**왜 직접 구현 안 하는가** 라는 질문의 답은 분명하다 — 합의는 정말 어렵다. TLA+ 검증을 안 받은 합의 구현은 거의 항상 살아남기 어려운 버그를 남긴다. 운영에서 검증된 ZooKeeper·etcd 를 한 자리 두고 다른 시스템들이 그 위에 올라타는 패턴이 표준이 된 이유다.

**성능 함정** 이 한 가지 있다. 코디네이션 서비스는 합의 비용 때문에 처리량이 일반 KV 저장소보다 한참 낮다(초당 수만 건 수준). 메타데이터 보관에는 충분하지만 정상 트래픽 데이터를 직접 보관하면 안 된다. Kafka 의 메시지 자체는 ZooKeeper 가 아니라 broker 에 들어간다.



## 면접 대비 체크리스트

1. Eventual·causal·linearizable 일관성을 한 시나리오씩 댈 수 있는가? 운영 시스템 대부분이 어디에 머무는가?
2. 선형성의 정의가 real-time order 를 포함하는 이유는? 같은 약속을 정합성으로 흉내낼 수 없는 이유는?
3. CAP 와 PACELC 의 차이가 무엇인가? 정상 상태에서도 일관성 비용을 평가해야 한다는 주장의 의미는?
4. Lamport 클럭과 Vector 클럭의 표현력 차이는? 무엇을 구분 못 하고 무엇을 구분 가능한가?
5. Hybrid Logical Clock 이 시계 동기화 가정 없이도 시간 순서를 거의 정확히 추적하는 원리는?
6. 합의 문제와 Total Order Broadcast 가 *동등* 한 문제임의 의미는? 한쪽 알고리즘이 다른 쪽을 풀어 주는 이유는?
7. Raft 가 Paxos 대비 운영 채택률이 높은 이유는? 세 가지 단계(리더 선출·로그 복제·안전성) 의 의미는?
8. FLP 불가능성 정리를 실용 알고리즘들이 어떻게 우회하는가?
9. ZooKeeper·etcd 가 받쳐 주는 다섯 기능은? 정상 트래픽 데이터를 거기 두면 안 되는 이유는?
