# 샤딩

---

> 데이터가 한 머신에 안 들어갈 때 여러 머신으로 쪼갠다. 복제가 같은 데이터의 사본을 여러 노드에 두는 것이라면, 샤딩은 데이터를 작은 조각(shard) 으로 나눠 *서로 다른* 노드에 둔다. 두 기법은 거의 항상 함께 쓰인다 — 각 샤드를 다시 복제로 보호한다.



## 복제와 샤딩의 차이

> 같은 분산이지만 푸는 문제가 다르다.

복제는 가용성·지연 시간·읽기 처리량을 위한 *중복* 이고, 샤딩은 저장 용량과 쓰기 처리량을 위한 *분할* 이다. 운영 시스템 대부분이 둘을 결합한다 — 데이터를 N 개 샤드로 나누고, 각 샤드를 다시 M 개 복제본으로 둔다. 결과적으로 N×M 개 노드가 동작한다.

```
샤딩 + 복제 (예: 4 샤드 × 3 복제본 = 12 노드)
─────────────────────────────────────────────
Shard A: leader_A, follower_A1, follower_A2
Shard B: leader_B, follower_B1, follower_B2
Shard C: leader_C, follower_C1, follower_C2
Shard D: leader_D, follower_D1, follower_D2
```

샤딩의 강점은 분명하다. 한 샤드의 부하가 한 노드 한도 안에서만 동작하면 되므로 시스템 전체 처리량이 노드 수에 거의 선형으로 늘어난다. 약점도 분명하다 — 여러 샤드를 동시에 보는 쿼리(글로벌 집계, 크로스 샤드 트랜잭션) 가 어려워진다.



## 샤딩 전략 — 키 범위 vs 해시

> 어떤 기준으로 키를 샤드에 배정할지가 운영의 모든 결정을 따라온다.

**키 범위 샤딩(Key Range Sharding)** 은 키의 정렬 순서를 기준으로 범위를 잘라 각 범위를 한 샤드에 배정한다. 백과사전이 알파벳 범위로 나뉘어 있는 것과 같은 발상이다.

```
샤드 1: A~F
샤드 2: G~M
샤드 3: N~T
샤드 4: U~Z
```

장점은 **범위 쿼리** 가 효율적이다. `WHERE name BETWEEN 'C' AND 'E'` 같은 쿼리가 한 샤드만 보면 끝난다. 단점은 **불균형 부하** 다. 사용자 활동이 시간대별로 달라 특정 범위에 쏠리면 그 샤드만 폭발한다(예: 시간 키로 샤딩하면 항상 최신 시간 샤드에 쓰기가 몰림).

**해시 샤딩(Hash Sharding)** 은 키에 해시를 적용해 그 결과로 샤드를 정한다. `hash(key) % N` 형태가 가장 단순하지만 N 이 바뀌면 거의 모든 키가 다른 샤드로 이동하는 함정이 있어 운영에서는 안 쓴다.

장점은 **균등 분산** 이다. 해시가 키 분포를 무작위화하므로 hot key 자체를 막지는 못해도 핫 샤드는 줄어든다. 단점은 **범위 쿼리 비효율** 이다. 인접 키가 다른 샤드에 흩어져 있어 범위 쿼리가 모든 샤드를 봐야 한다.

운영에서 자주 보는 변형이 두 가지 있다. MongoDB 의 **hashed shard key** 는 위 모델 그대로다. Cassandra 의 **composite key** 는 (파티션 키, 클러스터링 키) 형태로, 파티션 키로 샤드 분산을 결정하고 같은 샤드 안에서는 클러스터링 키로 정렬을 유지한다. 시계열 워크로드에 자주 등장하는 패턴이다 — `(user_id, timestamp)` 같은 식으로 사용자별 샤드 안에서 시간 정렬을 받는다.



## 일관된 해싱 (Consistent Hashing)

> 노드가 추가·삭제될 때 거의 모든 키가 재배치되는 함정을 줄인다. DynamoDB·Cassandra·Memcached 의 표준 도구다.

발상은 단순하다. 해시 공간을 0~2³² 같은 큰 원으로 보고, 노드와 키 모두 이 원 위에 해시된다. 키는 시계 방향으로 가장 가까운 노드가 담당한다.

```
원 위의 노드 배치 (간략):
       Node A (해시 100)
      ╱
     ╱
Node B (해시 250)  ←→ 키 200 → Node B (시계방향 첫 번째)
     ╲
      ╲
       Node C (해시 800)
```

노드가 추가되면 그 노드가 해시 위치에 들어가고, 그 직전 노드가 담당하던 키 일부만 새 노드로 이동한다. 노드 N 개 중 1 개를 추가할 때 평균 1/N 만큼만 재배치된다. 단순 모듈로 해싱이 거의 모든 키를 옮기는 것에 비해 압도적이다.

실무에서는 **가상 노드(virtual nodes)** 를 함께 쓴다. 한 물리 노드가 원 위에 여러 가상 위치를 가져, 노드 추가·삭제 시 부하 분산이 더 균등해진다. 같은 발상으로 노드 성능 차이도 가상 노드 개수로 표현 가능하다 — 강한 노드는 가상 노드 200 개, 약한 노드는 100 개.



## 핫스팟과 스큐 문제

> 균등 분산이 보장 안 되는 자리들이다. 운영에서 가장 자주 부딪히는 함정이다.

핫스팟(hot spot) 은 한 샤드에 트래픽이 집중되어 그 샤드만 병목이 되는 현상이다. 셀럽 사용자, 인기 상품, 글로벌 시간 키 같은 자리에서 자연스럽게 발생한다. 답은 **샤드 키 자체를 분산** 시키는 것이다 — 셀럽 ID 에 무작위 prefix(0~99) 를 붙이고 100 개 샤드로 흩어 보낸다. 읽기 시 100 번 조회 후 결합하는 비용이 따라오지만 쓰기 핫스팟은 사라진다.

스큐(skew) 는 좀 더 일반적이다. 어느 샤드의 데이터 양이나 부하가 다른 샤드보다 한참 크다. 핵심 측정값은 **load imbalance ratio** — 가장 큰 샤드와 평균의 비율이다. 1.5 를 넘으면 운영 점검이 필요하다.

답은 두 갈래다. 정적 샤드 수를 충분히 크게 잡고(노드 수보다 한 자릿수 큰 수), 부하 측정 후 샤드 재배치(rebalancing) 를 한다. 또는 동적 샤딩 — 한 샤드의 크기·부하가 임계를 넘으면 자동으로 둘로 쪼개고, 너무 작으면 인접 샤드와 합친다(MongoDB·HBase 의 자동 샤딩).



## 리밸런싱 — 자동 vs 수동

> 노드가 추가되거나 부하가 바뀔 때 데이터 분포를 다시 맞춘다. 누가 결정하느냐가 운영의 갈래다.

**자동 리밸런싱** 은 시스템이 스스로 노드 추가·삭제를 감지해 데이터를 옮긴다. 운영자 개입이 줄지만 의도치 않은 시점에 큰 데이터 이동이 일어나 운영 부담이 된다. 디스크 I/O 와 네트워크가 갑자기 폭증하면서 정상 트래픽 처리가 영향을 받는다. 운영 시간 외에 자동 리밸런싱이 시작되면 다음 날 사용자 응답 시간 이상으로 알게 되는 사고가 흔하다.

**수동 리밸런싱** 은 운영자가 명시적 명령을 내려야 데이터가 움직인다. 통제력은 좋지만 운영자가 잊으면 노드 부하 불균형이 그대로 유지된다.

운영의 절충은 보통 "자동 결정, 수동 트리거" 다. 시스템이 어느 데이터를 어디로 옮길지 계획만 세우고 운영자가 명령으로 실행을 시작한다. 그 명령을 트래픽이 적은 시간대에 내리는 운영 규약이 따라온다.

리밸런싱 방식의 한 갈래로 **샤드 수 고정** 이 있다. 처음에 충분히 많은 샤드를 만들어 두고(예: 1000 개), 노드 수가 변할 때 노드당 샤드 수만 조정한다. 한 샤드가 한 노드에 묶이지 않으므로 노드 추가·삭제가 단순한 재할당으로 끝난다. Cassandra·Riak 가 이 모델이다.



## 요청 라우팅

> 클라이언트가 어느 샤드로 요청을 보내야 하는지 어떻게 알 것인가. 세 가지 답이 있다.

**클라이언트 주도** — 클라이언트가 샤드 메타데이터를 들고 직접 적절한 노드를 호출한다. 네트워크 홉이 최소지만 메타데이터 갱신이 모든 클라이언트에 닿아야 하는 운영 부담이 따라온다.

**프록시 라우팅** — 모든 요청이 라우팅 프록시를 거치고, 프록시가 적절한 샤드로 전달한다. 클라이언트는 단순해지지만 프록시가 새 단일 점이 되고 한 홉이 더 들어간다. Vitess(MySQL) 가 이 모델이다.

**노드 간 전달** — 클라이언트가 임의 노드로 보내고, 그 노드가 책임 샤드로 내부적으로 전달한다. Cassandra 가 이 모델이다. 클라이언트가 가장 단순하지만 내부 네트워크 홉이 늘어난다.

샤드 메타데이터를 어디에 보관할지도 결정해야 한다. ZooKeeper·etcd 같은 코디네이션 서비스([`./02-06.일관성과 합의.md`](./02-06.일관성과%20합의.md) 참고) 가 표준 답이다. 메타데이터 변경이 모든 라우팅 노드로 빠르게 전파되어야 하므로 강한 일관성이 핵심이다.



## 보조 인덱스의 두 갈래

> 샤딩 키 외의 컬럼으로 검색하려면 보조 인덱스가 필요하다. 그 인덱스가 어떻게 분산되느냐로 두 모델이 갈린다.

**로컬 보조 인덱스(local secondary index, document-partitioned)** 는 각 샤드가 자기 데이터에 대한 인덱스만 갖는다. MongoDB·Riak 의 기본 모델이다. 인덱스 갱신이 같은 샤드 안에서 끝나 쓰기 비용이 낮다. 단점은 검색이 모든 샤드를 봐야 한다는 점이다 — scatter/gather 패턴이라 부른다.

**글로벌 보조 인덱스(global secondary index, term-partitioned)** 는 인덱스 자체를 *인덱스 키* 로 샤딩한다. 한 인덱스 검색이 한 샤드만 보면 끝난다. 단점은 인덱스 갱신이 여러 샤드를 건드린다는 점이다 — 한 행 변경이 인덱스 샤드의 분산 트랜잭션을 일으킨다. DynamoDB·Cassandra 가 이 모델을 지원한다.

선택 기준은 명확하다. 검색 빈도가 쓰기 빈도보다 압도적으로 높으면 글로벌 인덱스, 그 반대면 로컬 인덱스다. 운영에서는 종종 보조 인덱스를 별도 검색 시스템(Elasticsearch) 으로 빼는 답도 등장한다.



## 멀티테넌시를 위한 샤딩

> SaaS 환경에서 고객별 데이터를 어떻게 분리할지의 결정이다.

세 가지 모델이 있다. **단일 DB + tenant_id 컬럼** — 모든 고객이 한 DB 의 같은 테이블을 공유하고 `WHERE tenant_id = ?` 로 분리한다. 단순하지만 한 고객의 큰 쿼리가 다른 고객을 영향한다. **DB per tenant** — 고객마다 별도 DB. 격리는 완벽하지만 운영 비용이 고객 수에 비례한다. **샤딩 키 = tenant_id** — 같은 DB 안에서 tenant_id 로 샤딩. 일반 쿼리는 단일 샤드에서 끝나고 격리도 합리적이다.

세 번째 모델이 가장 흔하다. tenant_id 가 자연스러운 핫스팟 방지 차원이라 데이터 분산이 좋고, 한 고객 데이터를 통째로 마이그레이션하는 운영 시나리오도 단순하다.



## 면접 대비 체크리스트

1. 복제와 샤딩이 푸는 문제가 어떻게 다른가? 운영 시스템에서 둘을 어떻게 결합하는가?
2. 키 범위 샤딩과 해시 샤딩의 트레이드오프를 한 시나리오로 설명할 수 있는가?
3. 일관된 해싱이 단순 모듈로 해싱 대비 가지는 이득은? 가상 노드의 역할은?
4. 핫스팟이 발생하는 자연스러운 자리(셀럽·인기 상품·시간 키) 와 그 답(prefix 분산) 은?
5. 자동 리밸런싱과 수동 리밸런싱의 트레이드오프, 그리고 "자동 결정 + 수동 트리거" 절충의 의미는?
6. 요청 라우팅 세 모델(클라이언트·프록시·노드 간 전달) 의 장단점은?
7. 로컬 보조 인덱스와 글로벌 보조 인덱스의 선택 기준은? Scatter/gather 패턴의 비용은?
8. 멀티테넌시 SaaS 의 세 가지 데이터 격리 모델 중 어느 것이 어디에 어울리는가?
