# EmbeddedKafka·Testcontainers로 메시징 테스트

메시징은 단위 테스트로 충분히 검증되지 않는 영역이 많다. Producer 직렬화기와 Consumer 역직렬화기가 wire format 에서 만나야 비로소 결합 결함이 드러나고, DLQ 라우팅·재시도·idempotent producer 같은 인프라 기능은 실제 브로커 위에서만 동작한다. spring-kafka-test 의 `@EmbeddedKafka` 와 Testcontainers 의 `KafkaContainer` 가 두 가지 길이며, 가벼움과 운영 등가성의 트레이드오프가 명확하다. 이 챕터는 두 도구의 사용처, `KafkaTestUtils` 핵심 API, DLQ 라우팅 검증과 Test Double 패턴, `@RetryableTopic` 컴파일 타임 가드를 정리한다.



## 메시징 테스트가 잡는 결함

다음 결함은 단위 테스트로 거의 잡을 수 없다.

- Producer/Consumer 직렬화기 호환성 — Avro `SpecificRecord` ↔ `KafkaAvroDeserializer` 의 schema id 일치, JSON 의 type info, 깨진 wire format 의 처리
- `ErrorHandlingDeserializer` 의 무한 루프 차단 — poison pill 메시지가 DLQ 로 즉시 격리되는가
- `@RetryableTopic` 의 `kafkaTemplate` 속성 누락 — 잘못된 KafkaTemplate 으로 retry 메시지가 발행되어 ClassCastException 무한 루프
- `@KafkaListener` 의 컨테이너 팩토리 선택 — 여러 팩토리가 있을 때 `containerFactory` 명시
- 트랜잭션 producer 와 consumer offset 의 결합 — `read_committed` 격리, exactly-once 흐름
- DLT 헤더(`KafkaHeaders.DLT_ORIGINAL_TOPIC`, `DLT_EXCEPTION_FQCN`) 의 정확성

이런 결함은 실제 브로커가 떠 있어야 드러난다. 단위 테스트로는 mock KafkaTemplate 까지가 한계이며, 그 너머는 통합 테스트의 영역이다.



## EmbeddedKafka vs KafkaContainer

두 도구가 같은 의도(브로커 띄우기)를 다른 방식으로 푼다.

| 축 | `@EmbeddedKafka` | `KafkaContainer`(Testcontainers) |
|----|------------------|-----------------------------------|
| 구현 | JVM 안의 Kafka broker | Docker 컨테이너의 Kafka |
| 기동 시간 | 빠름 (수 초) | 상대적으로 느림 (10초 안팎) |
| 환경 의존 | JVM 만 있으면 동작 | Docker 필요 |
| 운영 등가성 | Kafka 의 일부 동작이 다를 수 있음 | 운영과 동일 이미지 사용 가능 |
| 멀티 broker | 가능하지만 설정 복잡 | 자연스럽게 가능 |
| 트랜잭션 | 동작하지만 일부 경계에서 차이 | 운영과 동일 |
| Schema Registry | 별도 설정 필요 | `ConfluentSchemaRegistryContainer` 와 결합 |

라이브러리·자체 기능 검증에는 `@EmbeddedKafka` 가 충분하다. 운영에 가까운 카오스/복구 시나리오는 `KafkaContainer` 가 더 안전하다. 실무에서는 한 모듈 안에 두 종류가 공존하는 경우가 많다. 라이브러리 자체 IT 는 `@EmbeddedKafka`, 애플리케이션 풀 E2E 는 외부 또는 Testcontainers Kafka 를 쓰는 식의 분장이 합리적이다.



## @EmbeddedKafka 기본 셋업

```java
@SpringBootTest(classes = OutboxPollerIT.TestConfig.class)
@EmbeddedKafka(partitions = 1, controlledShutdown = true)
@Testcontainers
class OutboxPollerIT {

    @Autowired EmbeddedKafkaBroker embeddedKafka;
    @Autowired EntityManager entityManager;
}
```

`@EmbeddedKafka` 는 클래스 레벨에서 broker 를 띄운다. `partitions` 는 테스트가 다룰 토픽의 파티션 수, `controlledShutdown = true` 는 종료 시점에 가벼운 정리를 수행한다. `topics = {"a", "b"}` 옵션으로 미리 토픽을 생성할 수 있다. `@Autowired EmbeddedKafkaBroker` 로 broker 인스턴스를 받아 producer/consumer 설정에 활용한다.

실제 producer/consumer 는 `KafkaTestUtils` 가 만든다.

```java
Map<String, Object> producerProps = KafkaTestUtils.producerProps(embeddedKafka);
producerProps.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
producerProps.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, ByteArraySerializer.class);

DefaultKafkaProducerFactory<String, byte[]> producerFactory =
    new DefaultKafkaProducerFactory<>(producerProps, new StringSerializer(), new ByteArraySerializer());
try (Producer<String, byte[]> producer = producerFactory.createProducer()) {
    producer.send(new ProducerRecord<>("topic", "key", payload)).get();
}

Map<String, Object> consumerProps = KafkaTestUtils.consumerProps("group-id-" + System.nanoTime(), "true", embeddedKafka);
consumerProps.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "earliest");

try (Consumer<String, byte[]> consumer = new DefaultKafkaConsumerFactory<>(
        consumerProps, new StringDeserializer(), new ByteArrayDeserializer()).createConsumer()) {

    embeddedKafka.consumeFromAnEmbeddedTopic(consumer, "topic");
    ConsumerRecord<String, byte[]> record = KafkaTestUtils.getSingleRecord(
        consumer, "topic", Duration.ofSeconds(15));
}
```

핵심 API 는 다음과 같다.

- `KafkaTestUtils.producerProps(broker)`/`consumerProps(group, autoCommit, broker)` — bootstrap-servers 등 기본값 셋업
- `embeddedKafka.consumeFromAnEmbeddedTopic(consumer, topic)` — 컨슈머가 토픽을 구독하게 함
- `KafkaTestUtils.getSingleRecord(consumer, topic, timeout)` — 한 레코드를 timeout 안에 받지 못하면 실패
- `KafkaTestUtils.getRecords(consumer, timeout, minRecords)` — 여러 레코드 일괄 수신

groupId 에 `System.nanoTime()` 같은 unique suffix 를 붙이면 테스트 간 offset leak 을 막는다. 같은 그룹으로 두면 이전 테스트의 commit 된 offset 이 다음 테스트 결과를 가린다.



## DLQ 라우팅과 ErrorHandlingDeserializer

깨진 wire format(잘못된 magic byte 또는 알 수 없는 schema id) 의 메시지를 컨슈머가 받으면 deserialization 단계에서 예외가 발생한다. 기본 컨슈머는 그 메시지에서 진행을 못 해 무한 루프에 빠진다. Spring Kafka 의 `ErrorHandlingDeserializer` 는 deserialize 실패를 헤더에 기록하고 listener 호출을 우회시켜, `DefaultErrorHandler` 가 `DeadLetterPublishingRecoverer` 로 메시지를 DLQ 토픽으로 발행하게 한다.

검증 시나리오는 다음과 같다.

1. consumer factory 의 value deserializer 를 `ErrorHandlingDeserializer` + 실패하는 delegate 로 구성
2. broken payload 를 producer 로 발행
3. listener 가 호출되지 않고 DLQ 토픽에 같은 payload 가 흘러가는지 확인
4. DLQ 레코드의 `KafkaHeaders.DLT_ORIGINAL_TOPIC` 헤더가 원본 토픽명을 담고 있는지 확인

```java
byte[] brokenPayload = new byte[]{0x00, 0x00, 0x00, 0x00, (byte) 0x99, 'x', 'y'};

producer.send(new ProducerRecord<>("test-business-topic", "key1", brokenPayload)).get();

ConsumerRecord<String, byte[]> dlqRecord = KafkaTestUtils.getSingleRecord(
    dlqConsumer, "tps.v305p.dlq", Duration.ofSeconds(15));

assertThat(dlqRecord).isNotNull();
assertThat(headerValue(dlqRecord, KafkaHeaders.DLT_ORIGINAL_TOPIC))
    .isEqualTo("test-business-topic");
```

문제는 실제 deserialize 실패를 일으키는 셋업이 까다롭다는 점이다. `KafkaAvroDeserializer` + dummy schema-registry URL 조합은 HTTP timeout 을 30 초 이상 기다려 IT 가 사실상 막힌다. 해결은 다음 절의 Test Double 이다.



## Test Double — FailingDeserializer

목표는 "deserialize 실패 흐름이 ErrorHandlingDeserializer → DefaultErrorHandler → DLQ 로 흘러가는가" 를 검증하는 것이지, schema-registry 가 정확히 어떻게 실패하는가가 아니다. value deserializer 의 delegate 만 즉시 `SerializationException` 을 던지는 stub 으로 교체하면 검증 의도는 동일하면서 시간은 ms 단위로 줄어든다.

```java
@DynamicPropertySource
static void registerProps(DynamicPropertyRegistry registry) {
    registry.add(
        "spring.kafka.consumer.properties.spring.deserializer.value.delegate.class",
        () -> "org.okestro.tps.messaging.config.FailingDeserializer"
    );
}
```

`FailingDeserializer` 는 `Deserializer<Object>` 를 구현해 `deserialize` 호출 시 즉시 `SerializationException` 을 던지는 단순 클래스다. 운영에서는 KafkaAvroDeserializer 가 그 자리에 들어간다. 테스트는 운영 동작을 정확히 흉내 내는 대신, 운영이 만들어 낼 결과(SerializationException 발생) 를 압축해 만든다.

이 패턴은 일반화된다. 외부 의존이 검증 의도의 본질이 아닐 때, 의도한 결과만 만들어 내는 stub 또는 fake 로 교체하면 빠르고 결정적이다. Mock 과 다른 점은 단순 호출 검증이 아니라 동작 흐름을 흉내 낸다는 점이다.



## @RetryableTopic 컴파일 타임 가드

`@RetryableTopic` 은 컨슈머 메서드의 retry/DLT 토픽을 자동 생성한다. 직렬화 흐름에서 함정이 있다. retry 메시지를 발행할 KafkaTemplate 을 어노테이션의 `kafkaTemplate` 속성으로 명시하지 않으면, Spring Boot 가 기본 등록한 `KafkaTemplate<String, byte[]>` 가 선택된다. 컨슈머가 받은 Avro `SpecificRecord` 가 byte[] 직렬화기로 흐르면서 ClassCastException 이 발생하고, 이 예외가 다시 retry 토픽으로 흘러 무한 루프가 발생한다.

이 함정을 빌드 단계에서 잡는 가드 테스트가 가능하다. 클래스패스를 스캔해 모든 `@RetryableTopic` 어노테이션을 모으고, `kafkaTemplate` 속성이 정해진 빈 이름(`retryKafkaTemplate`) 을 가리키지 않으면 위반으로 보고한다.

```java
class RetryableTopicConfigurationGuardTest {

    private static final String EXPECTED_TEMPLATE = "retryKafkaTemplate";
    private static final List<String> SCAN_PACKAGES = List.of(
        "org.okestro.tps.common.messaging",
        "org.okestro.tps.example"
    );

    @Test
    @DisplayName("executor/engine 의 @RetryableTopic 사용 메서드는 모두 kafkaTemplate=\"retryKafkaTemplate\" 을 지정해야 한다")
    void allRetryableTopicAnnotations_referenceRetryKafkaTemplate() throws Exception {
        List<Class<?>> classes = scanAll(SCAN_PACKAGES);
        List<String> violations = new ArrayList<>();
        for (Class<?> clazz : classes) {
            for (Method method : clazz.getDeclaredMethods()) {
                RetryableTopic retryable = method.getAnnotation(RetryableTopic.class);
                if (retryable == null) continue;
                if (!EXPECTED_TEMPLATE.equals(retryable.kafkaTemplate())) {
                    violations.add(clazz.getName() + "#" + method.getName()
                        + " kafkaTemplate=\"" + retryable.kafkaTemplate() + "\"");
                }
            }
        }
        assertThat(violations)
            .as("@RetryableTopic 은 kafkaTemplate=\"%s\" 를 명시해야 한다 (Avro 직렬화 회로 보호)", EXPECTED_TEMPLATE)
            .isEmpty();
    }
}
```

이 가드는 컨테이너를 띄우지 않으므로 단위 테스트 속도로 동작하며, 누군가 `@RetryableTopic` 을 새로 추가하면서 `kafkaTemplate` 속성을 빠뜨리면 빌드가 깨진다. ArchUnit 도 비슷한 일을 할 수 있지만, 어노테이션 속성값까지 검사하려면 reflection 가드가 더 직관적이다.



## EmbeddedKafka + Testcontainers DB 결합

라이브러리/애플리케이션 한 클래스가 Outbox 폴러처럼 DB 와 Kafka 를 둘 다 다룬다면 두 인프라를 같이 띄워야 한다. 메시징 라이브러리의 `OutboxPollerIT` 가 이 결합을 보여 준다.

```java
@SpringBootTest(classes = OutboxPollerIT.TestConfig.class)
@EmbeddedKafka(partitions = 1, controlledShutdown = true)
@Testcontainers
class OutboxPollerIT {

    @Container
    static final MariaDBContainer<?> MARIA_DB = new MariaDBContainer<>("mariadb:10.11")
        .withDatabaseName("tps_test")
        .withInitScript("testcontainers-init.sql");

    @DynamicPropertySource
    static void registerProps(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", MARIA_DB::getJdbcUrl);
        registry.add("spring.datasource.username", MARIA_DB::getUsername);
        registry.add("spring.datasource.password", MARIA_DB::getPassword);
        registry.add("spring.jpa.hibernate.ddl-auto", () -> "none");
    }
}
```

DB 는 Testcontainers MariaDB(운영과 동일 이미지), Kafka 는 EmbeddedKafka(라이브러리 검증의 본질이 broker 동작이므로) 라는 분장이다. 한 테스트 안에서 OutboxPoller 는 DB 의 PENDING 행을 읽고 EmbeddedKafka 토픽으로 발행한다. 검증은 두 측면을 모두 본다. DB 측에서는 행의 status 가 SENT 로 바뀌었는지, 재시도 시 status PENDING 으로 복귀하면서 retry_count 가 증가했는지. Kafka 측에서는 헤더 4 종(`ce_specversion`, `ce_id`, `ce_source`, `ce_correlationid`) 이 정확한 값으로 들어갔는지.



## 함정과 회피

`@EmbeddedKafka` 는 클래스마다 broker 를 새로 띄우는 게 기본이다. 같은 어노테이션 조합이면 컨텍스트 캐시로 broker 가 공유되지만, 어노테이션 속성이 한 글자라도 다르면 새 broker 가 뜬다. 빌드 시간이 길면 한 번 어노테이션 통일을 검토한다.

`groupId` 를 고정하면 테스트 순서에 따라 결과가 달라진다. `KafkaTestUtils.consumerProps("group-" + System.nanoTime(), "true", broker)` 처럼 unique suffix 를 붙이는 것이 안전하다.

`Awaitility` 를 추가 의존으로 쓸 수 있다. `await().atMost(5, SECONDS).untilAsserted(() -> ...)` 식으로 메시지 도착을 기다린다. `KafkaTestUtils.getSingleRecord(consumer, topic, duration)` 이 같은 의도를 라이브러리 안에서 처리하므로, 의존을 늘리지 않으려면 그쪽이 충분하다.

DLQ 토픽 이름은 라이브러리 또는 애플리케이션이 설정으로 정한다. 검증 시 하드코딩된 이름이 운영 설정과 어긋나지 않는지 한 번 확인한다.

스키마 레지스트리가 본질이 아니면 띄우지 않는다. Confluent SchemaRegistryContainer 를 띄우면 IT 한 케이스가 수십 초로 늘어난다. Test Double 로 대체할 수 있는지 먼저 검토한다.

`@RetryableTopic` 의 retry 토픽은 자동 생성된다. `topics =` 속성에 retry/DLT 토픽까지 명시하지 않아도 동작하지만, EmbeddedKafka 환경에서는 명시하는 편이 디버깅이 쉽다.



## TPS 사례 — 세 가지 패턴이 함께 산다

메시징 라이브러리 측은 세 가지 패턴이 공존한다.

`OutboxRetryPolicyTest`(단위) 는 컨텍스트 없이 `Clock.fixed` 로 정책을 검증한다. 백오프 곡선·max 임계 같은 분기 검증이 본질이라, 인프라가 필요 없다.

`OutboxPollerIT`(통합) 는 EmbeddedKafka + Testcontainers MariaDB 조합으로 실제 폴링·발행·SENT 마킹·재시도·DEAD 마킹을 모두 검증한다. `pollAndPublish()` 와 `cleanupSentEvents()` 를 수동 호출해 스케줄러 자동 실행을 배제하고 결정성을 확보한다.

`KafkaErrorIntegrationIT`(통합) 는 EmbeddedKafka + `FailingDeserializer` Test Double 로 DLQ 흐름만 좁게 검증한다. JPA·DataSource 자동 설정을 제외해 컨텍스트가 가볍다.

executor 측의 `RetryableTopicConfigurationGuardTest`(가드) 는 컨테이너 없이 reflection 으로 어노테이션 속성을 점검한다. 한 번의 사고(`kafkaTemplate` 누락 → ClassCastException 무한 루프) 를 영구 차단하는 자동화된 회귀 보호다.

```java
@SpringBootTest(classes = KafkaErrorIntegrationIT.TestConfig.class)
@EmbeddedKafka(partitions = 1, controlledShutdown = true,
    topics = {"test-business-topic", "tps.v305p.dlq"})
class KafkaErrorIntegrationIT {

    @DynamicPropertySource
    static void registerProps(DynamicPropertyRegistry registry) {
        registry.add(
            "spring.kafka.consumer.properties.spring.deserializer.value.delegate.class",
            () -> "org.okestro.tps.messaging.config.FailingDeserializer"
        );
    }

    @Configuration
    @EnableAutoConfiguration(exclude = {
        OutboxAutoConfiguration.class,
        MessagingTracingAutoConfiguration.class,
        KafkaJsonConsumerConfig.class,
        KafkaRetryTemplateConfig.class,
        TopicConfig.class,
        HibernateJpaAutoConfiguration.class,
        DataSourceAutoConfiguration.class
    })
    static class TestConfig {
        @Bean StubListener stubListener() { return new StubListener(); }
    }
}
```

이 결합의 장점은 검증 의도가 가장 좁은 인프라 조합으로 분리된다는 점이다. DLQ 흐름은 EmbeddedKafka + Test Double 로, Outbox 폴링은 EmbeddedKafka + Testcontainers MariaDB 로, 정책 검증은 단위로. 어느 케이스도 필요 이상의 인프라를 띄우지 않는다.

> 챕터 마무리 점검. 답을 입으로 한 번 말해 본다.

1. `@EmbeddedKafka` 와 `KafkaContainer` 의 결정 기준 한 문장은?
   - 답 요지: 라이브러리·자체 기능 검증은 `@EmbeddedKafka`(JVM 안 broker), 운영 등가성·카오스 시나리오는 `KafkaContainer`(Docker 컨테이너). 한 모듈 안에 둘이 공존해도 자연스럽다.
2. `KafkaTestUtils.consumerProps` 의 groupId 에 `System.nanoTime()` 같은 unique suffix 를 붙이는 이유는?
   - 답 요지: 같은 그룹으로 두면 이전 테스트가 commit 한 offset 이 다음 테스트 결과를 가린다. 테스트 간 offset leak 차단.
3. DLQ 라우팅 검증에서 schema-registry 를 띄우지 않고 `FailingDeserializer` 를 쓰는 이유는?
   - 답 요지: 검증 의도는 "deserialize 실패 흐름이 ErrorHandlingDeserializer → DefaultErrorHandler → DLQ 로 흘러가는가" 이지 schema-registry 가 어떻게 실패하는가가 아니다. Test Double 로 의도한 결과만 만들면 ms 단위로 줄어든다.
4. `@RetryableTopic` 에 `kafkaTemplate` 속성을 빠뜨리면 어떤 사고가 나는가?
   - 답 요지: 기본 `KafkaTemplate<String, byte[]>` 이 선택되어 Avro `SpecificRecord` 가 byte[] 직렬화기로 흐른다. ClassCastException 이 retry 토픽으로 발행되며 무한 루프가 발생한다.
5. ArchUnit 대신 reflection 가드를 쓰는 자리는?
   - 답 요지: 어노테이션 속성값(예: `kafkaTemplate = "retryKafkaTemplate"`) 처럼 문자열까지 검사해야 할 때. ArchUnit 은 패키지 의존·명명·어노테이션 부착에 강하고, 속성값은 reflection 이 더 직관적이다.
