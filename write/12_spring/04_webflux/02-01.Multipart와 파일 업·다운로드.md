# Multipart와 파일 업·다운로드

---

> multipart/form-data 본문은 단일 객체 직렬화로 표현 안 되는 첫 케이스다. 파일과 메타데이터를 같은 요청에 묶어 보내는 일이 흔하므로 한 번 익혀 두면 자주 쓴다. 본 챕터는 업로드의 `MultipartBodyBuilder` 패턴부터, 다운로드의 메모리 결정까지 다룬다.



## 왜 multipart인가

> 한 번의 HTTP 요청에 여러 종류의 데이터(파일 + JSON 메타 + 폼 필드)를 동시에 담는다.

multipart/form-data는 RFC 7578이 정의한 본문 형식이다. boundary 토큰으로 본문을 여러 파트로 나누고, 각 파트가 자기 헤더와 본문을 가진다. 일반적인 사용 패턴은 다음 셋이다.

1. **파일 + 메타데이터 동시 전송**. 첨부 파일을 올리면서 그 파일의 카테고리·태그·설명을 같이 보낸다.
2. **여러 파일 동시 업로드**. 같은 이름(`files`)으로 여러 파트를 추가.
3. **폼 필드와 파일 혼합**. HTML `<form enctype="multipart/form-data">`이 보내는 형식.

WebClient에서 multipart는 두 도구를 묶어 만든다. 본문 빌더 `MultipartBodyBuilder`와 본문 inserter `BodyInserters.fromMultipartData`다.



## 가장 단순한 형태 — 파일 한 개

> 5줄로 끝난다. 이 패턴이 출발점이다.

```java
MultipartBodyBuilder builder = new MultipartBodyBuilder();
builder.part("file", new ClassPathResource("/sample.pdf"))
        .contentType(MediaType.APPLICATION_PDF);

client.post().uri("/upload")
        .body(BodyInserters.fromMultipartData(builder.build()))
        .retrieve()
        .toBodilessEntity();
```

`MultipartBodyBuilder.part(name, value)`는 파트를 추가한다. 두 번째 인자는 다양한 타입을 받는다.

| 타입 | 동작 |
|------|------|
| `Resource` (`ClassPathResource`, `FileSystemResource`, `ByteArrayResource`) | 파일 파트 |
| `String` | 폼 필드 |
| `Object` | JSON 직렬화된 메타 파트 |
| `Mono<T>`, `Flux<T>` | 비동기 파트 (스트리밍) |

`.contentType(...)`은 파트의 `Content-Type` 헤더를 설정한다. 파일이라면 MIME 타입을, JSON이라면 `application/json`을 박는다. 생략하면 WebClient가 추측한다(이미지 파일이라도 `application/octet-stream`이 되는 경우가 많아 명시 권장).

⚠️ 호출 코드에서 `Content-Type` 헤더를 직접 박지 말 것. WebClient가 boundary 토큰을 포함한 `multipart/form-data; boundary=...` 헤더를 자동 생성한다. 직접 박은 헤더는 boundary가 없어 서버가 본문을 파싱하지 못한다.



## 메모리 안에 있는 바이트 배열을 보내는 패턴

> DB에서 꺼낸 파일을 그대로 multipart로 보내는 경우다. `ByteArrayResource`로 감싸지만 한 가지 트릭이 필요하다.

DB에서 꺼낸 첨부 파일은 `byte[]` 형태다. 이걸 `Resource`로 감싸 `MultipartBodyBuilder.part`에 넘긴다.

```java
byte[] data = attachment.getBytes();
String fileName = attachment.getFileName();
ByteArrayResource resource = new ByteArrayResource(data);

builder.part("file", resource).contentType(MediaType.APPLICATION_PDF);
```

위 코드는 동작하지만 한 가지 문제가 있다. multipart 파트의 `Content-Disposition: form-data; name="file"; filename="???"` 헤더에서 `filename` 부분이 비거나 임의 값이 된다. 서버가 파일명을 받아 저장하는 경우 이 누락이 치명적이다.

`ByteArrayResource`의 `getFilename()`은 기본 구현이 `null`을 반환한다. WebClient가 multipart 파트의 `filename` 속성을 채우려면 `Resource.getFilename()`이 의미 있는 값을 반환해야 한다. 익명 서브클래스 트릭으로 override한다.

```java
ByteArrayResource resource = new ByteArrayResource(data) {
    @Override
    public String getFilename() {
        return fileName;
    }
};

builder.part("file", resource).contentType(MediaType.APPLICATION_PDF);
```

이 패턴이 TPS `ApprovalUrlAdapter`에서 그대로 쓰인다.

```java
// ApprovalUrlAdapter.java:129-134
ByteArrayResource resource = new ByteArrayResource(data) {
    @Override
    public String getFilename() {
        return fileName;
    }
};

builder.part("files", resource).contentType(mediaType).filename(fileName);
```

`builder.part(...).filename(fileName)`도 같이 호출하므로 사실상 두 군데에서 파일명이 박힌다. `.filename(...)`이 multipart 파트 헤더의 `filename` 속성을 직접 설정하므로 `getFilename()` override는 보조 역할이다. 둘 다 박는 패턴이 안전하다(어느 쪽이 우선 적용되는지 코덱 구현에 따라 미세하게 다를 수 있다).



## JSON 메타 파트 + 파일 파트

> 가장 흔한 케이스다. 한 요청에 첨부 파일과 메타 JSON을 같이 보낸다.

```java
MultipartBodyBuilder builder = new MultipartBodyBuilder();

// 파일 파트
builder.part("file", new FileSystemResource(localPath))
        .contentType(MediaType.APPLICATION_PDF);

// 메타 JSON 파트
ContractMeta meta = new ContractMeta("contract-2026-q2", List.of("legal", "draft"));
builder.part("meta", meta, MediaType.APPLICATION_JSON);

client.post().uri("/contracts")
        .body(BodyInserters.fromMultipartData(builder.build()))
        .retrieve()
        .toBodilessEntity();
```

`builder.part(name, object, mediaType)` 형태로 호출하면 객체가 JSON으로 자동 직렬화된 뒤 파트로 들어간다. 서버 쪽 Spring 컨트롤러는 `@RequestPart("meta") ContractMeta meta`로 받을 수 있다.



## 폼 필드 평탄화 — TPS의 `flattenAndAppend`

> nested Map/List를 multipart 폼 필드로 변환하는 패턴이다. 표준은 아니지만 운영 코드에서 자주 본다.

multipart는 본질적으로 키-값 파트의 나열이다. nested 객체를 그대로 넣으면 JSON 파트가 되어 서버가 통째로 받지만, 어떤 서버는 폼 필드로 평탄화된 형태를 원한다(`address.city=Seoul`, `tags[0]=draft` 같은 형태).

TPS 어댑터는 nested Map/Collection을 dot/bracket 표기로 평탄화한다.

```java
// ApprovalUrlAdapter.java:222-246
private void flattenAndAppend(MultipartBodyBuilder builder, String parentKey, Object value) {
    if (value instanceof Map<?, ?> mapValue) {
        for (Map.Entry<?, ?> entry : mapValue.entrySet()) {
            String childKey = String.valueOf(entry.getKey());
            Object childValue = entry.getValue();
            flattenAndAppend(builder, parentKey + "." + childKey, childValue);
        }
    } else if (value instanceof Collection<?> colValue) {
        int index = 0;
        for (Object item : colValue) {
            flattenAndAppend(builder, parentKey + "[" + index++ + "]", item);
        }
    } else if (value != null && value.getClass().isArray()) {
        int len = Array.getLength(value);
        for (int i = 0; i < len; i++) {
            Object item = Array.get(value, i);
            flattenAndAppend(builder, parentKey + "[" + i + "]", item);
        }
    } else {
        String strValue = String.valueOf(value);
        if (!strValue.isBlank()) {
            builder.part(parentKey, strValue);
        }
    }
}
```

다음 입력이 들어오면

```java
Map.of("user", Map.of("name", "Alice", "tags", List.of("a", "b")))
```

다음 폼 필드들로 변환된다.

```
user.name=Alice
user.tags[0]=a
user.tags[1]=b
```

이 패턴의 한계는 두 가지다.

1. **표준이 없다**. 서버가 같은 평탄화 규칙을 알고 있어야 한다. dot 표기/bracket 표기는 사실상 사내 약속이다.
2. **타입이 사라진다**. 모든 값이 `String`이 된다. 숫자·boolean·null 구분이 서버에서 다시 파싱 비용을 만든다.

이 패턴이 정당화되는 경우는 외부 API가 그 형식을 요구할 때다. 새 API라면 메타 파트를 JSON으로 보내고 서버가 그대로 deserialize하는 편이 단순하다.



## 빈 컬렉션·빈 Map의 처리

> nested 평탄화에서 빈 컬렉션을 어떻게 보낼지가 미묘하다. 무 처리·빈 토큰·빈 객체 토큰 중 선택해야 한다.

TPS 어댑터의 처리를 보면 다음 분기가 있다.

```java
// ApprovalUrlAdapter.java:144-155
if (value == null) {
    // null인 경우 Part를 추가하지 않음
} else if (value instanceof Collection && ((Collection<?>) value).isEmpty()) {
    builder.part(key, "[]");
} else if (value instanceof Map && ((Map<?, ?>) value).isEmpty()) {
    builder.part(key, "{}");
} else if (value.getClass().isArray() && Array.getLength(value) == 0) {
    builder.part(key, "[]");
} else {
    flattenAndAppend(builder, key, value);
}
```

세 가지 결정이 보인다.

1. **null은 파트 자체를 생략**. 서버 쪽에서 누락을 null로 해석하는 약속.
2. **빈 컬렉션은 `"[]"` 문자열 파트**. JSON 빈 배열로 인식하도록 약속.
3. **빈 Map은 `"{}"` 문자열 파트**. JSON 빈 객체.

이 흐름은 평탄화 규칙과 마찬가지로 사내 약속이다. 새 코드를 짠다면 메타 파트를 JSON으로 한 번에 보내는 편이 모호함이 없다.



## 파일 다운로드 — 세 가지 형태

> 응답 본문이 파일이라면 세 가지 받는 방법이 있다. 메모리·스트리밍·임시 파일 중 선택.

### 1) 작은 파일 — `bodyToMono(byte[])`

```java
Mono<byte[]> data = client.get().uri("/files/{id}", 42L)
        .retrieve()
        .bodyToMono(byte[].class);
```

작은 파일(<1MB)이라면 가장 단순하다. `maxInMemorySize` 한도가 본문 전체보다 커야 한다. 파일이 클 가능성이 있다면 다음 형태로 옮긴다.

### 2) 중간 크기 파일 — `Mono<ResponseEntity<Resource>>`

```java
Mono<ResponseEntity<Resource>> response = client.get().uri("/files/{id}", 42L)
        .retrieve()
        .toEntity(Resource.class);

response.flatMap(entity -> {
    Resource body = entity.getBody();
    HttpHeaders headers = entity.getHeaders();
    String fileName = headers.getContentDisposition().getFilename();
    // body.getInputStream() 또는 body.readableChannel()로 처리
    return Mono.empty();
});
```

`Resource`로 받으면 응답 헤더(`Content-Disposition`의 파일명, `Content-Type`)와 함께 본문을 받을 수 있다. 메모리에 한 번 올라간다.

### 3) 큰 파일 — `bodyToFlux(DataBuffer)` + 파일에 쓰기

```java
Path target = Paths.get("/tmp/download-" + id);
Flux<DataBuffer> body = client.get().uri("/files/{id}", id)
        .retrieve()
        .bodyToFlux(DataBuffer.class);

DataBufferUtils.write(body, target, StandardOpenOption.CREATE)
        .doOnNext(DataBufferUtils::release)   // 버퍼 해제 책임
        .then();
```

청크 단위로 받으면서 파일에 그대로 쓴다. 메모리 사용량이 한 청크 크기 + 디스크 IO 버퍼 정도로 제한된다. 거대한 파일(>100MB) 다운로드는 이 형태가 표준이다.

⚠️ `DataBuffer`는 Netty 버퍼 풀에서 빌려오는 자원이다. `DataBufferUtils::release`로 명시적으로 해제하지 않으면 메모리 누수가 생긴다. `DataBufferUtils.write`는 내부에서 release를 처리하므로 별도 호출이 필요 없다(다만 `doOnNext(DataBufferUtils::release)`를 추가해도 무해).



## 함정 — 자주 만나는 세 가지

### 1. multipart 파트의 파일명이 비어 있다

`ByteArrayResource`의 `getFilename()`이 기본 `null`을 반환하기 때문이다. 익명 서브클래스로 override하거나 `builder.part(...).filename(...)`을 명시한다.

### 2. `Content-Type` 헤더를 직접 박았더니 본문이 도달 못한다

multipart의 `Content-Type` 헤더는 boundary 토큰을 포함한다. 호출자가 `multipart/form-data` 헤더를 직접 박으면 boundary 없이 들어가 서버가 본문을 분해 못한다. multipart 본문은 `body(BodyInserters.fromMultipartData(...))`만 호출하고 `contentType()`은 부르지 않는다.

### 3. `bodyToMono(byte[])`로 큰 파일을 받았더니 `DataBufferLimitException`

`maxInMemorySize` 기본 256KB를 넘는다. 빌더에서 한도를 키우거나 청크 다운로드(`bodyToFlux(DataBuffer)`)로 옮긴다.



## 면접에서 받을 만한 질문

> 챕터 마무리 점검.

1. multipart 본문에서 `Content-Type` 헤더를 호출자가 직접 박으면 안 되는 이유는?
   - 답 요지: boundary 토큰이 매 요청마다 다르게 생성되어야 하고 그 생성을 WebClient가 한다. 직접 박은 헤더는 boundary가 없어 서버 파싱 실패.
2. `ByteArrayResource`를 익명 서브클래스로 감싸 `getFilename()`을 override하는 이유는?
   - 답 요지: multipart 파트의 `filename` 속성이 채워지도록 한다. 기본 구현은 `null`을 반환해 파일명이 비어 들어간다.
3. nested Map을 multipart 폼 필드로 보낼 때 평탄화 규칙은 표준인가?
   - 답 요지: 표준이 없다. dot/bracket 표기는 사내 약속이다. 새 API라면 메타 파트를 JSON으로 보내는 편이 모호함이 없다.
4. 큰 파일을 다운로드하는 표준 패턴은?
   - 답 요지: `bodyToFlux(DataBuffer)` + `DataBufferUtils.write`. 청크 단위로 받으면서 파일에 그대로 쓴다. 메모리 peak이 청크 크기 정도로 제한된다.
5. `DataBuffer`를 직접 다룰 때 책임지는 것은?
   - 답 요지: 명시적 release. Netty 버퍼 풀에서 빌려오는 자원이므로 처리 후 `DataBufferUtils.release`를 호출하지 않으면 메모리 누수가 누적된다.
