# 3단계 점검 — 운영 인터페이스 핵심 질문

---

> 다루는 문서 - `03-01.REST API와 웹 통합`, `03-02.접근 제어와 인증`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다.



## Q1: Nexus가 offset/limit이 아닌 continuationToken을 쓰는 이유는?

내부 DB(과거 OrientDB, 현재 H2)에서 offset 기반 페이지네이션은 깊은 페이지로 갈수록 성능이 급격히 나빠진다. offset이 커질수록 그만큼의 행을 스캔하고 버려야 하기 때문이다. continuationToken은 마지막 조회 위치를 토큰에 인코딩하므로 데이터가 아무리 많아도 일정한 성능을 보장한다. 대신 임의 페이지 점프 불가, 순방향 순회만 가능이라는 제약이 따른다. UI 설계는 페이지 번호 대신 "더 보기" 또는 무한 스크롤로 가야 한다. 토큰은 불투명(opaque)하므로 파싱하지 말고 그대로 다음 요청에 넘긴다.

상세: `03-01.md` §3.3, §5.

### 심화

페이지 번호 UI가 반드시 필요한 상황에서, 클라이언트가 토큰을 캐싱해 페이지 번호를 흉내 내는 접근의 한계는 무엇인가?



## Q2: 업로드 응답이 204 No Content이고 컴포넌트 ID를 즉시 받지 못하는 설계의 함의는?

업로드는 멀티파트 기반이며 응답 본문이 비어 있다. 방금 올린 컴포넌트의 ID가 필요하면 검색 API로 다시 조회해야 한다. CI 자동화에서는 deploy → 검색 → 후속 작업으로 두 번의 호출을 묶어 처리하게 된다. 이 비효율을 피하려면 좌표(GAV 또는 path)로 직접 추적하는 패턴을 권장한다. 즉 ID에 의존하지 않고 좌표를 신뢰하는 워크플로를 설계한다.

상세: `03-01.md` §3.6, §4.2.

### 심화

`/search/assets`가 `downloadUrl`을 직접 반환하는 점이 후속 다운로드 단계를 어떻게 단축하는가?



## Q3: FormData를 쓸 때 `Content-Type`을 수동으로 설정하면 안 되는 이유는?

`multipart/form-data`는 본문 안에서 각 파트를 구분하기 위해 임의의 boundary 문자열을 사용한다. 브라우저가 FormData를 직렬화할 때 이 boundary를 자동 생성해 `Content-Type: multipart/form-data; boundary=...` 헤더에 포함시킨다. 코드에서 `Content-Type`을 수동으로 지정하면 boundary가 누락되어 서버가 본문을 파싱하지 못해 422 에러로 떨어진다. 한 번 일부러 넣어 보고 422를 받으면 진짜 이해된다.

상세: `03-01.md` §6.2.

### 심화

Node 환경의 `node-fetch`나 `axios`로 멀티파트를 보낼 때 boundary가 자동 생성되지 않는 라이브러리도 있는데, 이때는 어떻게 처리해야 하는가?



## Q4: CORS 우회의 세 패턴 중 백엔드 프록시가 가장 안전한 이유는?

백엔드 프록시는 프론트엔드가 자기 서버의 `/api/nexus/*`로 요청하고 서버가 Nexus에 대신 요청하는 구조다. 브라우저 입장에서 같은 출처이므로 CORS 자체가 발생하지 않는다. 더 중요한 점은 Nexus 인증 정보가 서버 쪽에서만 관리되어 프론트엔드 코드에 Basic Auth가 노출되지 않는다는 것이다. nginx CORS 헤더 주입은 편리하지만 `Allow-Origin: *`와 인증 정보 동시 사용이 불가하다는 제약이 있고, 동일 도메인 서빙은 라우팅 구성이 필요하다.

상세: `03-01.md` §6.1.

### 심화

`Allow-Origin: *`와 인증 헤더가 함께 사용 불가한 이유는 어떤 보안 모델에서 비롯되는가?



## Q5: Realm 순서에서 Local Authenticating을 항상 첫 번째로 두는 이유는?

LDAP이나 SAML이 첫 Realm이면 모든 인증 시도가 외부 서버로 먼저 간다. 외부 서버가 다운되면 로컬 admin 계정도 로그인할 수 없는 잠긴 상태가 된다. 운영자가 비상 시 들어와 설정을 고칠 통로가 사라지는 셈이다. Local Authenticating Realm을 가장 앞에 두면 외부 인프라 장애와 무관하게 로컬 admin이 항상 들어올 수 있다. Local Authorizing Realm은 외부 인증 사용자에게 내부 Role을 매핑하는 역할이므로 LDAP 사용 시 함께 활성화한다.

상세: `03-02.md` §2.2.

### 심화

Docker Token Realm을 비활성화한 채 Docker 리포지토리를 사용하면 `docker login`은 성공하는데 `docker push`가 실패하는 현상이 생기는 원인은 무엇인가?



## Q6: 권한 문자열 `nx-repository-view-maven2-maven-releases-read`에서 `read`와 `browse`의 차이는?

`read`는 에셋 콘텐츠를 다운로드할 수 있는 권한이고, `browse`는 리포지토리 구조를 탐색(목록 조회)할 수 있는 권한이다. 파일을 다운로드하려면 보통 둘 다 필요하므로 함께 부여하는 것이 일반적이다. CI에서 의존성을 받아오는 시나리오라면 `read` + `browse` 조합이 최소 권한이고, 단순히 UI에서 어떤 아티팩트가 있는지만 확인하는 사용자라면 `browse`만으로 충분할 수 있다.

상세: `03-02.md` §5.2.

### 심화

같은 패턴의 와일드카드 표현 `nx-repository-view-maven2-*-read`가 운영에서 가지는 위험은 무엇인가?



## Q7: Content Selector로 "특정 경로만 차단"이 어려운 이유는?

CSEL에는 NOT 연산자가 없다. Content Selector는 본질적으로 "허용" 방식으로 동작하므로, 차단하려면 차단 대상을 제외한 모든 경로를 명시적으로 허용해야 한다. 정규식 lookahead로 우회할 수 있지만 유지보수가 어려워 권장하지 않는다. 차단이 목적이면 Content Selector 대신 리포지토리를 분리하거나 Routing Rule(proxy 한정)을 함께 쓰는 것이 실용적이다.

상세: `03-02.md` §6.

### 심화

리포지토리를 분리하는 대신 CSEL로 경로 단위 권한을 쪼개는 접근이 적합한 시점은 언제인가?



## Q8: LDAP 그룹 변경이 Nexus에 즉시 반영되지 않는 이유와 긴급 차단 방법은?

Nexus는 성능을 위해 LDAP 조회 결과를 캐시한다. 매 요청마다 LDAP 서버에 가면 부하가 크고 빌드가 느려지기 때문이다. 캐시 TTL(보통 수 분~수십 분) 동안은 그룹 변경이 보이지 않는다. 퇴사자 LDAP 비활성화를 했는데 Nexus 캐시가 살아 있으면 그동안 접근이 가능해진다. 긴급 시 가장 확실한 방법은 Nexus에서 해당 사용자를 직접 비활성화(Status: Disabled)하는 것이다. REST API `PUT /service/rest/v1/security/users/{userId}`에 `status: disabled`를 넣으면 즉시 차단된다. 퇴사 프로세스에 이 단계를 반드시 포함시킨다.

상세: `03-02.md` §9.4.

### 심화

LDAP 캐시 TTL을 5분으로 짧게 설정하는 대안의 트레이드오프는 무엇인가?
