# 5단계 점검 — 운영 신뢰성 핵심 질문

---

> 다루는 문서 - `05-01.정리 정책과 스토리지 관리`, `05-02.백업·복구·업그레이드`, `05-03.모니터링과 트러블슈팅`, `05-04.프로덕션 운영 패턴`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다.



## Q1: Cleanup Task만 실행하면 디스크가 줄지 않는 이유는?

Cleanup은 soft delete 단계만 처리한다. DB 메타데이터를 제거하고 Blob Store의 `.properties` 파일에 `deleted=true` 마크를 붙일 뿐, `.bytes` 파일은 디스크에 그대로 남는다. 디스크 공간은 Compact Blob Store Task가 `deleted=true` 항목의 `.bytes`를 실제로 삭제할 때(hard delete) 회수된다. 두 단계로 나눈 이유는 복구 가능성 — soft delete 상태라면 잘못된 정리가 발생해도 DB 메타데이터 복원으로 되살릴 수 있다. 그래서 Cleanup 후에는 반드시 같은 날 또는 다음날 새벽에 Compact가 돌도록 스케줄을 묶는다. PostgreSQL의 VACUUM이 dead tuple을 정리하는 것과 같은 패턴이다.

상세: `05-01.md` §4.

### 심화

soft delete와 Compact 사이 간격을 너무 짧게 잡으면 어떤 안전선이 사라지는가?



## Q2: Docker 이미지 정리에서 Cleanup을 두세 번 반복 실행해야 하는 경우는?

manifest list(멀티 아키텍처)는 manifest list → platform manifest → layer라는 3단 참조 구조다. manifest list만 삭제하면 하위 platform manifest가 그제서야 orphan이 되고, 다음 정리 사이클에서야 layer가 orphan이 된다. 한 번의 Cleanup으로 전체 체인이 정리되지 않으므로 Cleanup 후 디스크가 예상만큼 줄지 않으면 한 번 더 실행해 본다. Docker 이미지가 많은 환경에서는 Cleanup을 일주일에 2~3회로 스케줄하는 것이 실무적 권장이다.

상세: `05-01.md` §5.

### 심화

orphan layer가 디스크만 낭비할 뿐 기능적 문제는 적은데도 즉시 정리해야 하는 운영적 이유는 무엇인가?



## Q3: hot backup의 정합성 불일치를 줄이는 순서 전략은?

DB Export를 먼저 실행하고 그 직후 Blob Store를 백업한다. 이 순서에서 발생할 수 있는 불일치는 "Export 후 ~ Blob 백업 완료 전" 창에 배포된 아티팩트뿐이다. DB에 메타데이터가 있는데 Blob 파일이 없는 불일치(복구 후 404)와, Blob에 파일이 있는데 DB에 메타데이터가 없는 불일치(orphan blob)를 비교하면 후자가 훨씬 덜 치명적이다. 완벽한 정합성이 필요하면 cold backup을 쓰고, 다운타임이 안 되면 LVM/ZFS 스냅샷이나 EBS Snapshot으로 COW 기반 정합 스냅샷을 뜬다.

상세: `05-02.md` §2, §7.2.

### 심화

LVM/ZFS 스냅샷이 cold backup과 같은 정합성을 보장하면서도 다운타임을 만들지 않는 원리는 무엇인가?



## Q4: 업그레이드 후 단순 이미지 태그 롤백이 실패하는 시나리오는?

새 버전이 DB 스키마를 변경한 경우다. 새 버전이 컬럼을 추가하거나 데이터 형식을 변환하면 이전 버전이 그 스키마를 읽지 못해 시작이 실패한다. 에러는 보통 "Unknown column" 또는 "Schema version mismatch" 형태로 나타난다. Nexus의 대표 사례가 OrientDB → H2 마이그레이션이며, 이 전환이 끝난 후 OrientDB 기반 이전 버전으로는 되돌릴 수 없다. 마이너 버전 업그레이드라도 릴리스 노트의 "Database migration included" 문구가 있으면 단순 태그 롤백이 불가능하다고 가정한다. 안전한 롤백은 이전 이미지 태그 + 이전 백업 데이터를 함께 복원하는 것이고, 그래서 업그레이드 직전 백업이 절대적으로 필요하다.

상세: `05-02.md` §4.3.

### 심화

업그레이드 직전 백업 파일에 어떤 명명 규칙을 쓰면 롤백 시 어떤 백업으로 돌아갈지를 즉시 판단할 수 있는가?



## Q5: Prometheus 메트릭 엔드포인트가 기본 비활성인 이유는?

보안과 성능 양쪽이다. 메트릭 엔드포인트가 JVM 힙·HTTP 요청 수·Blob Store 용량 등 내부 상태를 노출하므로 외부에 그대로 공개하면 인프라 구조를 추론할 단서가 된다. 또한 수집 자체가 미세한 오버헤드를 발생시킨다. 활성화는 `nexus.properties`에 `nexus.metrics.enabled=true`를 명시적으로 추가해야 한다. 프로덕션에서는 켜되 Basic Auth를 반드시 설정하고 Reverse Proxy ACL로 Prometheus 서버만 접근 가능하도록 제한한다.

상세: `05-03.md` §3.1.

### 심화

scrape_interval을 30초보다 짧게 가져갈 때 Nexus에 어떤 부하가 누적되며, 그 손익분기점은 어떻게 가늠하는가?



## Q6: 컨테이너 메모리 = JVM Heap + DirectMemory + α 공식의 α는 무엇인가?

α는 Metaspace + Thread Stacks + OS Overhead의 합이다. Metaspace는 OSGi 번들 수에 비례해 보통 256~512MB, Thread Stacks는 스레드당 1MB × 200~400 스레드 = 200~400MB, OS Overhead는 커널 매핑·JNI·native 코드용으로 300~500MB다. 실무 간이 공식으로는 `Container Memory Limit ≈ Xmx × 2 + 512MB`다. Heap 2GB 환경이면 `2G(Heap) + 2G(Direct) + 0.3G(Metaspace) + 0.3G(Threads) + 0.4G(OS) = 5.0GB`이고, GC 직전 Heap 피크와 Direct Memory 피크가 동시에 일어날 수 있어 20% 여유를 더해 6GB로 잡는 것이 안전하다.

상세: `05-03.md` §5.2.

### 심화

`MaxDirectMemorySize`를 명시하지 않을 때 OOM Killer가 발생해도 JVM 로그에 `OutOfMemoryError`가 안 남는 이유는 어디서 비롯되는가?



## Q7: Nexus 응답 불가 시 5분 타임박스 진단 순서는?

다섯 단계로 나눠 1분씩 쓴다. (1) 프로세스 생존 — `/status/writable` HTTP 코드와 `docker stats`로 살아 있는지/느린지 구분. (2) 최근 에러 로그 — `docker logs nexus --tail 50`에서 OutOfMemoryError(힙)·Cannot acquire lock(DB)·BlobStoreException(디스크) 같은 단서로 80%가 가려진다. (3) JVM 상태 — `jcmd $(pgrep -f nexus) GC.heap_info`로 Old Gen 90% 이상이면 Full GC 루프 의심. (4) 디스크/Task — `df -h /nexus-data`와 `/service/rest/v1/tasks`. (5) 트래픽 패턴 — request.log 마지막 100줄에서 IP별 폭주 여부 확인. 5단계로 안 풀리면 thread dump(`jcmd ... Thread.print`)로 데드락을 본다.

상세: `05-03.md` §6.

### 심화

(2) 단계에서 발견되는 `Cannot acquire lock`이 DB 잠금 문제일 때, 같은 호스트의 다른 프로세스가 DB 파일을 잡고 있는 시나리오와 Nexus 내부 데드락 시나리오를 어떻게 구분하는가?



## Q8: Active-Passive 모델로 충분하다는 판단의 근거는?

Nexus의 내부 데이터베이스(OrientDB·H2) 제약 때문에 Active-Active가 기술적으로 불가능에 가깝다. 임베디드 DB는 단일 프로세스를 전제하고, 외부 PostgreSQL을 쓰는 최신 버전에서도 캐시 레이어와 Blob Store 메타데이터 관리가 단일 writer를 전제로 설계돼 있다. Active-Active를 구현하려면 분산 락·캐시 무효화 전파·충돌 해소가 필요하다. 더 본질적인 판단은 "아티팩트 저장소에 그 수준 처리량이 정말 필요한가?"다. 대부분의 조직에서 장애 시 요구되는 건 처리량 분산이 아니라 빠른 복구이므로 Active-Passive가 적절한 선택이다. 읽기 트래픽 분산이 필요하면 Pro의 읽기 replica를 쓰거나, OSS에서 Reverse Proxy가 GET을 여러 인스턴스로 분산하되 같은 Proxy 리포지토리를 바라보게 구성한다.

상세: `05-04.md` §2.1.

### 심화

OSS 환경에서 "용도별 분리 인스턴스"(Maven 전용·Docker 전용)가 가용성을 어떻게 향상시키며, 어떤 운영 부담을 새로 만들어 내는가?



## Q9: IaC drift를 가장 선제적으로 잡는 방법은?

audit.log 모니터링이다. 주기적 비교 스크립트(API → JSON → diff)는 drift가 발생한 후에 잡고, Terraform plan은 다음 실행 시점에야 잡는다. audit.log는 변경 발생 직후 이벤트가 남으므로 감지 지연이 가장 짧다. 다만 가장 근본적인 해결은 admin UI 접근을 최소 인원으로 제한하고 모든 변경을 Git PR → 파이프라인 → API 호출 경로로 강제하는 것이다. 기술적 감지보다 프로세스적 차단이 우선이다.

상세: `05-04.md` §4.3.

### 심화

긴급 상황에서 UI 수동 변경을 허용해야 한다면, 그 변경이 사후에 IaC 정의 파일에 자동 반영되게 하는 워크플로는 어떻게 설계하는가?



## Q10: Smart Proxy의 캐시 일관성을 releases와 snapshots에서 다르게 잡는 이유는?

releases는 immutable이라 같은 좌표가 절대 바뀌지 않는다. TTL을 길게(24시간 이상) 잡아도 stale 위험이 없다. snapshots는 같은 좌표에 새 빌드가 계속 올라오므로 TTL을 짧게(1시간 이하) 설정해야 클라이언트가 최신 버전을 받는다. 강제 동기화가 필요한 보안 패치 전파 시점에는 invalidate API를 호출하는데, 무효화는 "다음 요청 시 원본에서 다시 가져오겠다"이지 즉시 삭제가 아니다. 개발자 로컬 캐시(`~/.m2`)까지 정리해야 완전한 제거가 된다.

상세: `05-04.md` §7.

### 심화

Smart Proxy가 단순 CDN과 다르게 "쓰기"도 가능한데, 한 리전에서 발생한 hosted 쓰기가 다른 리전에 동기화되는 시점은 어떻게 결정되는가?
