# 2단계 점검 — 리포지토리 모델 핵심 질문

---

> 다루는 문서 - `02-01.리포지토리 포맷과 구성`, `02-02.프록시와 캐싱 전략`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다.



## Q1: RELEASE의 Write Policy를 `ALLOW_ONCE`로 두는 이유는?

같은 GAV로 두 번째 배포를 거부함으로써 아티팩트 불변성을 기술적으로 강제하기 위함이다. 프로덕션에 배포된 `1.2.3`이 어제와 오늘 다른 바이너리라면 재현 가능한 빌드라는 개념이 무너진다. SNAPSHOT은 개발 편의를 위해 `ALLOW`로 반복 덮어쓰기를 허용하는 대신, 내부적으로 타임스탬프 버전을 붙여 이전 스냅샷도 보존한다. 한 리포지토리에 한 Write Policy만 적용되므로 RELEASE와 SNAPSHOT을 같은 hosted에 섞어선 안 된다.

상세: `02-01.md` §2.2.

## Q2: GAV 좌표가 디렉토리 경로로 변환되는 규칙을 설명할 수 있는가?

`groupId`의 `.`을 `/`로 치환하고, 그 아래에 `artifactId`/`version`/`{artifactId}-{version}.{ext}` 구조로 저장된다. 예컨대 `com.example:payment-service:1.2.3`은 `com/example/payment-service/1.2.3/payment-service-1.2.3.jar`이 된다. 이 규칙을 알면 빌드 도구가 못 찾는 아티팩트를 브라우저나 `curl`로 직접 조회해 hosted/proxy 어느 단계에서 끊기는지 즉시 진단할 수 있다.

상세: `02-01.md` §2.1.

### 심화

같은 디렉토리에 함께 놓이는 `maven-metadata.xml`의 역할은 무엇이며, SNAPSHOT 환경에서 이 파일이 어떤 정보를 추가로 담는가?



## Q3: npm `.npmrc`에서 install 경로와 publish 경로를 분리하는 이유는?

group 리포지토리는 읽기 전용이라 `npm publish`를 받을 수 없기 때문이다. 따라서 `registry`는 group(npm-group)을 가리키고, `@scope:registry`는 hosted(npm-hosted)를 직접 가리켜야 한다. `_auth` 토큰의 URL 경로가 등록한 registry와 한 글자라도 다르면 npm이 prefix 매칭에 실패해 인증 정보를 못 찾으므로, hosted와 group 양쪽 경로에 각각 `_auth`를 등록해야 한다.

상세: `02-01.md` §3.1.

### 심화

CI 환경에서 `.npmrc`에 비밀번호를 하드코딩하지 않는 안전한 패턴은 무엇인가?



## Q4: Docker 리포지토리가 별도 포트를 요구하는 이유와 우회 방법은?

Docker CLI는 Registry URL에 경로를 포함할 수 없다. `docker push host/repo/myimage`까지는 되지만 `host/repository/docker-hosted/myimage`는 불가하다. 그래서 hosted/proxy/group마다 8082, 8083, 8084 같은 별도 포트를 할당한다. 우회 패턴은 nginx 같은 reverse proxy로 `docker-hosted.example.com → 8082`처럼 호스트명 라우팅을 두는 것이다. 이때 `client_max_body_size 0`을 빠뜨리지 않는 것이 핵심이다.

상세: `02-01.md` §4.2.

### 심화

Docker Bearer Token Realm을 활성화하지 않으면 `docker login` 자체가 실패하는데, Realm은 어떤 인증 흐름을 추가하는가?



## Q5: Content Max Age가 RELEASE에서 -1(무기한)인 근거는?

Maven RELEASE 아티팩트는 immutable이라는 계약을 갖는다. `commons-lang3:3.14.0`은 한번 릴리스되면 같은 좌표로 절대 다시 올라올 수 없으므로, 캐시된 바이너리가 stale이 될 가능성 자체가 없다. 무기한 캐시는 외부 저장소가 사라지거나 다운돼도 영향을 받지 않는다는 부수 효과까지 준다. SNAPSHOT proxy는 정반대로 유한한 Max Age가 필수인데, 그러지 않으면 "최신 개발본을 쓴다"는 SNAPSHOT의 목적이 깨진다.

상세: `02-02.md` §2.1, §2.4.

### 심화

외부 SNAPSHOT proxy 자체를 가급적 피하라는 권장이 있는데, 그 대안인 SNAPSHOT pinning은 무엇인가?



## Q6: Negative cache가 빌드 실패를 만드는 함정은?

내부 라이브러리 `com.mycompany:auth-sdk:2.0.0`을 동료가 곧 deploy할 예정인데, 다른 동료가 그 직전에 빌드해 404를 받으면 negative cache에 1440분간 저장된다. 이후 deploy가 끝나도 TTL이 만료될 때까지 Nexus는 계속 "없다"고 응답한다. 해결은 세 갈래다 — TTL을 60–120분으로 짧게, deploy 직후 해당 경로 캐시 수동 무효화, CI 파이프라인에서 invalidate API 자동 호출. 다만 hosted에 올라간 아티팩트는 negative cache와 무관하다는 점도 함께 기억한다.

상세: `02-02.md` §2.3.

### 심화

negative cache가 proxy에만 적용되고 hosted에는 적용되지 않는다는 사실이 의미하는 운영상의 함의는 무엇인가?



## Q7: Auto-blocking이 빌드 시간을 줄이는 메커니즘은?

원격 저장소가 다운된 상태에서 cache miss가 나면 Auto-blocking이 없을 때는 `connection timeout × retry`만큼 매번 대기해야 한다. 의존성 200개에서 매 요청이 50초 대기하면 빌드는 사실상 멈춘다. Auto-blocking이 켜지면 연속 실패가 임계값을 넘는 순간 proxy가 blocked로 전환되고, 이후 cache miss는 즉시 404를 받는다. fail fast로 빌드 실패가 빨라지고, 이미 캐시된 아티팩트는 정상적으로 서빙된다는 점이 핵심이다. 모니터링에 blocked 알림을 연결해두면 운영자가 장애 사실을 빠르게 알 수 있다.

상세: `02-02.md` §3.

### 심화

blocked 상태가 자동 복구되지 않고 수동 해제가 필요한 시나리오는 어떤 경우인가?



## Q8: dependency confusion 공격을 group 멤버 순서와 Routing Rule이 함께 막는 방식은?

공격자는 회사 내부 groupId(`com.mycompany`)와 동일한 패키지를 Maven Central에 올린다. group이 proxy를 hosted보다 먼저 탐색하면 외부 악성 패키지가 우선 반환된다. 1차 방어는 group 멤버 순서를 hosted → proxy로 두는 것이고, 2차 방어는 proxy의 Routing Rule에 BLOCK 모드로 `com/mycompany/.*`를 등록해 외부에서 해당 네임스페이스를 가져오는 경로 자체를 차단하는 것이다. npm은 scope 소유권 검증이 있어 위험이 낮지만 Maven은 groupId 소유권 검증이 없어 Routing Rule이 더 중요하다.

상세: `02-01.md` §7.2 + `01-01.md` §3.3.

ALLOW 모드 화이트리스트가 보안상 더 강력하지만 운영에서 잘 안 쓰이는 이유는 무엇인가?
