# 4단계 점검 — 통합 활용 핵심 질문

---

> 다루는 문서 - `04-01.CI/CD 파이프라인 연동`, `04-02.Docker Registry로서의 Nexus`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다.



## Q1: `mvn deploy`와 `nexusArtifactUploader`의 결정적 차이는?

`mvn deploy`는 Maven 빌드 라이프사이클의 일부로 동작하며 POM 메타데이터(의존성·부모 POM·플러그인 설정)와 체크섬·classifier까지 함께 올린다. Nexus가 이 메타데이터를 인덱싱해 다른 Maven 프로젝트가 의존성으로 참조할 수 있게 된다. `nexusArtifactUploader`는 Jenkins 플러그인이 REST API를 직접 호출해 파일만 올리는 방식이다. POM 없이 동작하므로 Go·Rust·Python 빌드 결과물을 raw 리포지토리에 올릴 때 적합하다. 판단 기준은 단순하다 — "이 아티팩트를 다른 Maven 프로젝트에서 의존성으로 쓸 것인가?" 그렇다면 `mvn deploy`다.

상세: `04-01.md` §2.2~2.3.

## Q2: Jenkins `credentials()` 마스킹만 믿으면 안 되는 이유는?

마스킹은 Jenkins 콘솔 로그에서만 적용된다. 프로세스 메모리, 파일시스템, 환경변수 노출 명령(`env | sort`, `cat /proc/self/environ`)에서는 평문이 그대로 보인다. 더 위험한 시나리오는 외부 기여자가 보낸 PR이 Jenkinsfile을 수정해 환경변수를 외부로 전송하는 코드를 심는 것이다. 방어는 두 겹이다 — Multibranch Pipeline의 "Trust" 설정으로 외부 기여자의 Jenkinsfile 변경을 차단하고, PR 빌드에는 프로덕션 시크릿을 주입하지 않는다. HashiCorp Vault 같은 외부 시크릿 매니저로 TTL과 감사 로그를 추가하면 안전선이 한층 더 늘어난다.

상세: `04-01.md` §2.4.

## Q3: npm 배포에서 `.npmrc`의 `registry`와 `publishConfig.registry`를 분리하는 이유는?

`.npmrc`의 `registry`는 install(다운로드) 경로이고, `package.json`의 `publishConfig.registry`는 publish(업로드) 경로다. group 리포지토리는 읽기 전용이라 publish를 받지 않으므로 publish 경로는 hosted를 직접 가리켜야 한다. 둘을 혼동하면 publish는 성공하는데 install이 실패하는 상황이 벌어진다. CI에서는 `.npmrc`를 동적으로 생성하고 업로드 후 삭제하는 패턴이 가장 안전하다.

상세: `04-01.md` §4.

### 심화

`.gitignore`에 `.npmrc`를 추가했는데도 이미 tracking 중이면 적용되지 않는다. 이미 커밋된 `.npmrc`를 안전하게 제거하는 절차는?



## Q4: SNAPSHOT → RELEASE 프로모션 두 패턴의 트레이드오프는?

리포지토리 간 복사(Pro Staging)는 staging의 동일 바이너리를 release로 이동시키므로 빌드 재현성이 보장된다. SNAPSHOT에서 검증한 그 바이너리가 그대로 RELEASE가 된다는 신뢰 모델이다. 버전 변경 후 재빌드는 OSS의 흔한 패턴인데, `-SNAPSHOT`을 떼고 다시 빌드해 releases에 올린다. 간단하지만 "SNAPSHOT에서 테스트한 바이너리와 RELEASE 바이너리가 정말 같은가?"에 100% 확신할 수 없다는 약점이 있다. 빌드 환경의 사소한 차이(시스템 시간, JDK 패치 버전, 빌드 머신 변경)가 결과물의 미세한 차이를 만들 수 있기 때문이다.

상세: `04-01.md` §6.

## Q5: Docker 리포지토리마다 별도 포트가 필요한 기술적 이유는?

Maven은 URL 경로(`/repository/maven-releases/`)로 리포지토리를 구분하지만 Docker는 그렇지 않다. Docker 클라이언트가 `GET /v2/my-app/manifests/1.0`을 보낼 때 HTTP Host 헤더에는 `localhost:8082`만 들어가고 URL 경로에는 리포지토리 이름이 포함되지 않는다. 같은 포트로 들어온 요청을 hosted로 보낼지 group으로 보낼지 Nexus가 구분할 방법이 없으므로 포트가 곧 리포지토리 식별자가 된다. nginx 서브도메인 라우팅으로 표준 443 포트만 노출하면서 백엔드에서 포트별로 분기하는 구성이 운영을 단순화한다.

상세: `04-02.md` §4.

### 심화

Nexus Pro의 "하나의 HTTPS 커넥터에서 경로 기반 라우팅" 기능이 OSS의 포트 분리 한계를 어떻게 우회하는가?



## Q6: docker-proxy로 Docker Hub rate limit을 어떻게 효과적으로 줄이는가?

docker-proxy가 이미지를 한 번 캐싱하면 이후 팀원의 pull 요청은 Nexus 로컬 캐시에서 처리되므로 Docker Hub에 외부 요청이 나가지 않는다. 팀원이 50명이어도 같은 이미지에 대해 외부 요청은 1회뿐이다. proxy에 Docker Hub 인증을 등록하면 익명(100회/6시간) 대신 인증(200회/6시간) 한도가 적용되고, Pro/Team 계정이면 5,000회/일까지 올라간다. 추가로 Cleanup Policy에서 "Last Downloaded 14일"을 걸어 최근 사용 이미지만 유지하고 멀티 아키텍처 이미지의 plat-별 layer 캐시 누락도 함께 고려한다.

상세: `04-02.md` §3.2, §10.2.

### 심화

amd64 서버에서 pull한 멀티 아키텍처 이미지의 arm64 layer가 캐싱되지 않아 arm64 개발자가 다시 외부 요청을 발생시키는 함정을 어떻게 모니터링하는가?



## Q7: Bearer Token Realm을 비활성화한 채 Docker 리포지토리를 사용할 때의 증상은?

`docker login`은 성공하는데 `docker push`가 `unauthorized` 에러로 실패하는 혼란스러운 증상이 나타난다. Docker 클라이언트가 `WWW-Authenticate` 헤더에서 Bearer 흐름을 받아 토큰을 요청하는데, 토큰 발급을 처리할 Realm이 없으면 push 단계에서 막히기 때문이다. `Administration → Security → Realms`에서 Docker Bearer Token Realm을 Active로 옮기는 것이 첫 번째 의심 항목이다. `curl -v http://localhost:8082/v2/`로 `WWW-Authenticate` 헤더가 정상적으로 내려오는지 확인하면 진단이 빠르다.

상세: `04-02.md` §7.

### 심화

`WWW-Authenticate` 헤더의 `realm`, `service`, `scope` 필드가 Docker 클라이언트의 토큰 요청 흐름에서 각각 어떤 역할을 하는가?



## Q8: nginx 앞단에서 `proxy_buffering off`와 `proxy_request_buffering off`를 켜는 이유는?

Docker layer는 수백 MB에서 수 GB에 달할 수 있다. nginx가 layer 데이터를 버퍼에 모았다가 전달하면 메모리를 과도하게 사용하고 큰 layer에서는 버퍼가 디스크로 떨어져 추가 I/O가 발생한다. 스트리밍 모드로 바로 흘려 보내면 nginx 메모리 부담이 크게 줄고 push/pull 성능도 안정된다. 함께 챙길 항목은 `client_max_body_size`(2GB 이상), `proxy_read_timeout`(600초 이상), `chunked_transfer_encoding on`이다. 이 셋이 빠지면 대용량 layer 업로드가 중간에 끊기는 증상으로 나타난다.

상세: `04-02.md` §6.

### 심화

`client_max_body_size`가 부족할 때 발생하는 `413 Request Entity Too Large`를 Docker 클라이언트가 모호한 메시지로 표시하는데, 어떤 로그 위치를 봐야 빠르게 진단할 수 있는가?
