# 1단계 점검 — SonarQube 입문 핵심 질문

---

> 다루는 문서 - `01-01.정적 분석과 SonarQube 모델`, `01-02.운영 환경과 분석 흐름`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다. 표·다이어그램은 본문에 있으므로 여기서는 통찰만 정리한다.



## Q1: 컴파일러와 모든 테스트가 통과하는데도 SonarQube가 잡아내는 결함은 어떤 종류인가?

세 종류가 동시에 빠진다. 컴파일러는 타입 시스템 안의 모순만 본다. 단위 테스트는 개발자가 의식적으로 작성한 시나리오만 본다. 이 두 검사를 모두 통과해도 검증 없이 외부 입력을 SQL에 결합한 코드, 도달 불가능한 분기, 같은 로직이 세 군데에 복제된 구조, 200줄을 넘는 단일 메서드는 그대로 남는다. SonarQube는 이런 "관습 위반"과 "위험한 의도"를 AST 위에서 데이터·제어 흐름을 따라가며 잡는다. 결함을 없애는 도구가 아니라, 결함이 발견되는 시점을 빌드까지 왼쪽으로 옮기는 도구다.

상세: `01-01.md` §1.

### 심화

같은 결함을 운영 단계에서 발견했을 때의 비용이 빌드 단계에 비해 100배에 달한다는 IBM 연구의 곡선이 SaaS 시대에도 유효한가, 아니면 다른 비용 함수가 작동하는가?



## Q2: PostgreSQL을 권장하고 H2를 평가용으로만 두는 이유는?

H2는 동시성 모델과 백업 모델 둘 다 운영에 부적합하다. SonarQube는 Compute Engine과 Web Server가 동시에 같은 DB에 쓰고, 분석 이력과 이슈 라이프사이클이 트랜잭션 안전성에 의존한다. H2는 단일 프로세스 임베디드 DB이므로 동시 쓰기와 일관된 백업 스냅샷을 보장하기 어렵다. 또 운영 중에 H2 파일이 손상되면 회복 도구가 빈약하다. PostgreSQL/Oracle/SQL Server는 동시성·트랜잭션·백업 도구가 모두 성숙해 있으므로 운영 인스턴스의 기본 선택지가 된다. SonarSource가 H2를 "평가/시범 운영 전용"이라고 명시하는 이유다.

상세: `01-01.md` §2, `01-02.md` §1.

### 심화

큰 인스턴스에서 PostgreSQL → Oracle 같은 DB 횡 이동이 공식 지원되지 않는데, 어떤 운영 시나리오에서 이 이동이 강제되며 어떤 마이그레이션 전략이 가능한가?



## Q3: Quality Profile과 Quality Gate의 책임이 어떻게 다른가?

Quality Profile은 "어떤 규칙으로 검사할 것인가"를 정한다. 언어별 Rule들의 묶음이고, 프로젝트는 언어마다 하나의 Profile을 사용한다. Quality Gate는 "검사 결과의 어떤 수치 이상이면 실패로 본다"를 정한다. 측정값(metric)에 대한 조건들의 묶음이다. 검사기와 채점표의 차이라고 보면 된다. Profile을 바꾸면 발견되는 Issue 자체가 달라지고, Gate를 바꾸면 같은 Issue 수에서 통과/실패 판정만 달라진다. 둘을 한 개념으로 묶지 않은 것이 SonarQube 모델의 핵심 책임 분리다.

상세: `01-01.md` §3, §4.

### 심화

Profile을 표준화하지 않은 채 팀별로 자유롭게 Gate만 합의하는 운영 모델은 어디서 깨지는가? 반대로 Profile을 표준화하고 Gate를 팀별로 두는 모델은 어떤 부작용이 있는가?



## Q4: Clean as You Code가 "기존 코드 부채를 무시한다"는 비판에 어떻게 답하는가?

CAYC는 부채를 무시하는 게 아니라 부채 청산의 강제 시점을 옮기는 정책이다. 레거시에 SonarQube를 처음 붙이면 Issue가 수천 건 잡히는 게 보통이고, 이걸 한꺼번에 청산하라고 강요하면 정상 기능 개발이 마비되거나 의무감으로 빠르게 닫는 과정에서 회귀가 들어온다. CAYC는 게이트 조건을 New Code에 몰아 두고, 신규 코드만 깨끗하면 통과시킨다. 시간이 지나 코드의 상당 부분이 새로 쓰인 코드로 치환되면 결과적으로 전체 코드가 깨끗해진다. 부채는 사람이 따라올 수 있는 속도로만 청산된다는 현실을 받아들인 정책이다.

상세: `01-01.md` §5.

### 심화

리팩터링이 "기존 코드를 다시 쓰는" 활동이라는 점을 고려하면, CAYC의 New Code 정의가 리팩터링을 인센티브화하는가, 디스인센티브화하는가? 어느 쪽으로 작동하는지에 따라 게이트 설계는 어떻게 달라져야 하는가?



## Q5: MQR 모드가 한 Rule을 여러 품질 차원에 매핑하는 이유는?

같은 결함이 한 종류의 영향만 끼치는 경우가 드물기 때문이다. "암호화되지 않은 통신 채널 사용"은 보안 차원에서는 High 수준 영향이지만, 신뢰성 차원에서도 Medium 수준 영향을 가진다. Standard 모드는 이걸 Vulnerability/Critical 한 라벨로 압축했지만, MQR 모드는 security: High, reliability: Medium처럼 두 영향도를 동시에 매긴다. 이렇게 하면 보안 전담자와 신뢰성 전담자가 같은 결함을 자기 관점에서 우선순위 매기는 게 가능해진다. severity 단계가 Blocker/High/Medium/Low/Info로 갱신된 것도 차원별 영향도를 일관되게 매기기 위한 변경이다.

상세: `01-01.md` §3.2.

### 심화

기존 인스턴스를 MQR로 전환하면 과거 Issue의 severity가 재계산된다. 대규모 인스턴스에서 이 전환을 안전하게 진행하려면 어떤 시점·통신·롤백 전략이 필요한가?



## Q6: SonarScanner CLI 대신 Sonar Scanner for Gradle을 쓰면 무엇이 달라지는가?

Gradle 통합 Scanner는 Gradle 프로젝트 메타에서 분석에 필요한 정보를 자동 추출한다. 소스 디렉터리, 테스트 디렉터리, 언어, 컴파일된 바이너리 위치, 의존성 클래스패스가 그것이다. 일부 Rule(특히 SAST 계열)은 바이트코드 분석을 요구하므로 클래스패스가 정확해야 분석 품질이 떨어지지 않는다. CLI를 쓰면 위 정보를 `sonar-project.properties`에 손으로 명시해야 하는데, 멀티모듈 Gradle 빌드에서는 이게 사실상 불가능하다. CLI는 빌드 시스템 통합 Scanner가 없는 폴리글랏 프로젝트의 마지막 선택지로 본다.

상세: `01-02.md` §3.

### 심화

같은 모듈을 Gradle Scanner와 CLI로 각각 분석하면 결과가 어떻게 달라지는가? Rule 적용은 같다고 가정할 때 어디서 차이가 발생하며, 그 차이가 운영적으로 어떤 위험을 만드는가?



## Q7: User Token / Project Analysis Token / Global Analysis Token 중 CI 파이프라인에 어떤 것을 줘야 최소 권한인가?

Project Analysis Token이 기본이다. 한 프로젝트에 대한 분석 업로드 권한만 가지므로 토큰이 유출돼도 그 프로젝트 외에는 영향이 없다. User Token은 사용자의 모든 권한을 그대로 가지므로 CI에 두면 사용자 권한이 통째로 노출되고, 자동화에 사람 토큰을 쓰는 것 자체가 안티패턴이다. Global Analysis Token은 모든 프로젝트의 분석 업로드 권한을 가지므로 한 토큰의 폭발 반경이 가장 크다. 한 빌드 머신이 여러 프로젝트를 다뤄야 한다면 Global Analysis Token으로 합치는 것이 가능하지만, 그 경우 토큰 회전 정책을 더 엄격하게 운영해야 한다.

상세: `01-02.md` §5.

### 심화

CI/CD 파이프라인에 SonarQube 토큰과 Nexus 자격 증명, 컨테이너 레지스트리 자격 증명이 함께 주입되는 흔한 환경에서, 이 토큰들의 회전 주기를 동기화하는 게 좋은가, 비동기로 두는 게 좋은가? 어느 쪽이 사고 시 회복 시간을 짧게 하는가?
