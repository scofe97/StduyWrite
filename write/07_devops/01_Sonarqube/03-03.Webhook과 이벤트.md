# Webhook과 이벤트

---

> SonarQube가 외부로 보내는 비동기 이벤트의 구조를 본다. Webhook 페이로드, 등록 모델, 시그니처 검증, 운영 함정을 정리한다.



## 1. Webhook이 발사되는 시점

> SonarQube의 Webhook은 한 종류의 이벤트만 발사한다. 분석 종료(Background Task 완료) 시점이다.

다른 SaaS의 Webhook과 비교하면 SonarQube는 단순한 편이다. 이슈 생성, 사용자 추가, Profile 변경 같은 다양한 이벤트는 Webhook이 아니라 Audit Log API(Enterprise+)를 통해 노출된다. Webhook은 한 가지 이벤트만 다룬다 — 분석이 끝났다는 것.

발사 트리거는 다음 흐름의 끝이다.

1. Scanner가 분석 리포트를 Web Server에 업로드.
2. Compute Engine이 Background Task 큐에서 리포트를 꺼내 처리.
3. 이슈 매칭, 측정값 계산, Quality Gate 평가 완료.
4. 등록된 Webhook URL 목록으로 페이로드 POST.

Background Task 처리가 실패한 경우에도 Webhook이 발사된다. 페이로드의 `status` 필드가 `SUCCESS` 또는 `FAILED` 중 하나로 들어온다.



## 2. 페이로드 구조

> 페이로드의 골격은 단순하지만 그 안에 분석 한 회의 모든 핵심 정보가 들어 있다.

표준 페이로드는 다음과 같다.

```json
{
  "serverUrl": "http://sonarqube.internal",
  "taskId": "AYx_qN-LKzNz...",
  "status": "SUCCESS",
  "analysedAt": "2025-12-31T15:30:45+0900",
  "revision": "a1b2c3d4...",
  "changedAt": "2025-12-31T15:30:50+0900",
  "project": {
    "key": "org.okestro:pipeline-api",
    "name": "pipeline-api",
    "url": "http://sonarqube.internal/dashboard?id=org.okestro:pipeline-api"
  },
  "branch": {
    "name": "main",
    "type": "LONG",
    "isMain": true,
    "url": "http://sonarqube.internal/dashboard?id=org.okestro:pipeline-api&branch=main"
  },
  "qualityGate": {
    "name": "Sonar way",
    "status": "OK",
    "conditions": [
      {
        "metric": "new_coverage",
        "operator": "LESS_THAN",
        "errorThreshold": "80",
        "value": "85.3",
        "status": "OK"
      }
    ]
  },
  "properties": {
    "sonar.analysis.tcktNo": "TPS-1234",
    "sonar.analysis.pplnNo": "5"
  }
}
```

세 영역으로 나뉜다.

- **메타**: `serverUrl`, `taskId`, `status`, `analysedAt`, `revision`. 어느 인스턴스의 어느 분석인지를 식별.
- **컨텍스트**: `project`, `branch`. 분석 대상 식별.
- **결과**: `qualityGate`, `properties`. 게이트 판정과 사용자 정의 메타.

`properties`가 가장 흥미롭다. SonarScanner 호출 시 `-Dsonar.analysis.<key>=<value>` 형태로 임의 메타데이터를 주입할 수 있고, 이 메타가 그대로 페이로드에 전달된다. 04-03에서 본 TPS 사례가 이 메커니즘을 쓴다 — `tcktNo`, `pplnNo`로 SonarQube 페이로드를 TPS 도메인 키로 매핑한다.

### 2.1 PR 분석의 페이로드

PR 분석에서 발사되는 페이로드는 `branch` 대신 `pullRequest` 필드를 가진다.

```json
{
  ...,
  "pullRequest": {
    "key": "1234",
    "branch": "feature/login",
    "base": "main",
    "url": "..."
  },
  ...
}
```

수신 측은 `branch`가 있는지 `pullRequest`가 있는지로 분석 종류를 구분한다. 두 필드가 동시에 있는 경우는 없다.



## 3. Webhook 등록의 두 범위

> Webhook은 글로벌 또는 프로젝트 단위로 등록한다. 같은 URL을 두 번 등록하지 않는 게 첫 번째 운영 사실이다.

등록 범위는 둘이다.

- **Global Webhook**: 인스턴스의 모든 프로젝트 분석에 발사. Administer System 권한 필요.
- **Project Webhook**: 한 프로젝트의 분석에만 발사. Administer 권한(프로젝트 단위) 필요.

operator/cicd 사례에서 자동 등록한 것이 Global Webhook이었다. 모든 프로젝트의 분석 결과를 executor가 수신해 ppln-logging-api로 흘려보내는 패턴이 Global 등록과 자연스럽게 맞는다.

### 3.1 등록 한도

SonarQube는 Webhook 등록에 한도를 둔다.

- Global Webhook: 인스턴스당 최대 10개
- Project Webhook: 프로젝트당 최대 10개

10개 한도는 일반 운영에서 충분하지만, 외부 시스템 통합이 많은 인스턴스에서는 도달할 수 있다. 한도를 넘으면 등록 자체가 거부된다(400). 한도를 우회하는 표준 패턴은 한 Webhook 수신자가 여러 외부 시스템으로 fan-out 하는 것이다.

### 3.2 Webhook URL 거부 정책

SonarQube는 보안상 임의의 URL을 Webhook으로 받지 않는다. 04-02에서 본 사실 — 다음 URL은 거부된다.

- 루프백(`localhost`, `127.0.0.1`, `::1`)
- 와일드카드(`0.0.0.0`)
- 비표준 포트의 일부 (정책에 따라)
- 프로토콜이 http/https가 아닌 URL

거부 응답은 400이며 본문에 `loopback` 또는 `wildcard` 키워드가 들어 있다. operator의 `isInvalidWebhookUrl` 휴리스틱이 이 키워드를 검사한다. SonarQube 메이저 업그레이드 시 메시지 문구가 바뀌면 회귀가 일어날 수 있는 부분이다.



## 4. 시그니처 검증

> Webhook 수신자는 페이로드가 진짜 SonarQube에서 온 것인지 검증해야 한다. SonarQube는 HMAC-SHA256 시그니처를 제공한다.

Webhook 등록 시 secret을 함께 등록할 수 있다. 그 secret이 있으면 SonarQube는 페이로드의 HMAC-SHA256을 헤더에 실어 보낸다.

```
POST /webhook-receiver HTTP/1.1
X-Sonar-Webhook-HMAC-SHA256: <hex-encoded HMAC>
Content-Type: application/json

{ "serverUrl": "...", ... }
```

수신 측에서는 다음 검증을 수행한다.

```python
import hmac, hashlib

def verify(payload_bytes: bytes, header_signature: str, secret: str) -> bool:
    expected = hmac.new(secret.encode(), payload_bytes, hashlib.sha256).hexdigest()
    return hmac.compare_digest(expected, header_signature)
```

`hmac.compare_digest`는 timing attack을 방어하는 상수 시간 비교다. 일반 `==`로 비교하면 안 된다.

### 4.1 secret 없이 등록한 Webhook

secret 등록은 선택사항이다. 등록하지 않으면 시그니처 헤더가 들어오지 않는다. secret 없는 Webhook은 수신자가 페이로드의 진위를 검증할 수단이 없으므로 다음 위험을 가진다.

- 누구나 Webhook 수신자 URL을 알면 가짜 페이로드를 보낼 수 있다.
- 가짜 분석 종료 이벤트로 외부 시스템의 워크플로우(예: 빌드 차단 해제)를 트리거할 수 있다.

운영 환경에서는 secret을 등록하는 게 표준이다. 사내 폐쇄망이라도 측면 이동(lateral movement) 방어로 필요하다.



## 5. 재시도와 신뢰성

> Webhook 전달은 best-effort다. SonarQube가 재시도를 보장하지 않는다는 사실이 운영자가 알아야 할 가장 중요한 사실이다.

전달 모델은 다음과 같다.

- SonarQube가 페이로드를 한 번 POST.
- 응답이 2xx면 성공으로 간주.
- 응답이 4xx/5xx거나 timeout이면 한 번의 재시도(인스턴스 버전에 따라).
- 그 후에는 포기.

이 모델이 의미하는 바는 수신 측이 빠르고 안정적이어야 한다는 것이다. 30초 이상 응답하지 않으면 SonarQube가 timeout으로 처리한다. Webhook 수신자가 무거운 작업을 동기로 처리하면 손실이 발생한다.

### 5.1 권장 수신 패턴 — 큐로 분리

수신자의 표준 골격은 다음과 같다.

```
1. POST 수신 → 시그니처 검증 → 페이로드를 큐(예: Kafka, Redis)에 적재 → 200 즉시 반환
2. 별도 워커가 큐에서 페이로드를 꺼내 실제 처리
```

이 분리가 있으면 수신자는 항상 100ms 이내 응답이 가능하고, 워커 처리가 늦어도 SonarQube에 영향이 없다. 워커 처리가 실패해도 큐에서 재시도가 가능하므로 신뢰성도 높아진다.

TPS의 executor가 이 패턴을 쓴다고 추정 가능하다 — Webhook 수신은 즉시 200을 반환하고, ppln-logging-api로의 결과 전달은 별도 처리로 분리됐을 가능성이 크다(코드 베이스에서 직접 확인하지는 않았지만 04-03의 흐름이 이 패턴과 자연스럽게 맞는다).

### 5.2 Webhook 전달 로그

SonarQube는 Webhook 전달 결과를 자체 로그로 남긴다. UI에서는 Project Settings → Webhooks → Recent Deliveries에서 볼 수 있다. Web API로는 다음 엔드포인트다.

```
GET /api/webhooks/deliveries?webhook=<webhookKey>
```

응답 본문에는 시도 시각, 응답 상태 코드, 응답 본문, 페이로드 본문이 포함된다. 운영에서 "Webhook이 안 왔다"는 보고를 받으면 이 로그를 먼저 본다.



## 6. Webhook과 동기 폴링의 선택

> 같은 게이트 결과를 받는 두 방법 — Webhook(비동기 푸시)과 폴링(동기 GET)의 선택 기준을 다시 정리한다.

|  | Webhook | 폴링 |
|--|---------|------|
| 결과 수신 시점 | 분석 종료 직후 | 클라이언트 호출 시점 |
| 빌드 시간 영향 | 없음 (분리된 처리) | 있음 (Compute Engine 처리 대기) |
| 수신 인프라 필요 | 있음 (수신 서비스, 큐) | 없음 (CI 빌드 안에서) |
| 신뢰성 | 재시도 보장 약함 | 명시적 재시도 가능 |
| 게이트 결과 → 빌드 결과 | 비동기 (별도 시스템) | 동기 (빌드 안에서) |

선택의 축은 두 가지다.

- 빌드 결과가 게이트 결과를 반영해야 하나? — 그렇다면 폴링.
- 외부 시스템이 결과를 받아 자체 워크플로우를 트리거하나? — 그렇다면 Webhook.

두 패턴은 배타적이지 않고 동시 사용이 가능하다. 같은 분석에 대해 빌드는 폴링으로 게이트 결과를 받고, 외부 시스템은 Webhook으로 같은 결과를 받는다. SonarQube는 두 클라이언트가 동시에 같은 결과를 가져가도 영향을 주지 않는다.



## 7. 정리

> Webhook은 단순하지만 운영 함정이 셋 있다.

요약한다. SonarQube Webhook은 분석 종료 이벤트 한 종류만 다루며, 페이로드는 메타·컨텍스트·결과 세 영역으로 구성된다. `sonar.analysis.<key>` 임의 메타 주입이 사용자 정의 컨텍스트를 페이로드에 실어 보내는 표준 패턴이다. 등록은 Global/Project 두 범위가 있으며 인스턴스/프로젝트당 10개 한도다. URL 거부 정책(loopback/wildcard) 때문에 자동 등록 시 거부 분류가 필요하다. HMAC-SHA256 시그니처 검증은 secret 등록 시 활성화되며 운영 환경에서 표준이다. 전달은 best-effort이므로 수신자는 큐로 분리해 빠르게 200을 반환하는 패턴이 신뢰성을 보장한다.
