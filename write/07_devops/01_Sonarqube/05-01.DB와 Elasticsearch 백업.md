# DB와 Elasticsearch 백업

---

> SonarQube의 두 저장소를 어떻게 백업하느냐가 운영 신뢰성의 출발이다. DB는 진실의 원본이고 Elasticsearch는 그 인덱스라는 비대칭이 백업 전략을 결정한다.



## 1. 무엇을 백업해야 하는가

> SonarQube의 영속 데이터는 셋이지만 백업이 꼭 필요한 것은 둘이다.

운영 인스턴스가 가진 영속 데이터를 정리한다.

| 저장소 | 내용 | 백업 필요성 |
|--------|------|-----------|
| Database (PostgreSQL/Oracle/SQL Server) | 모든 분석 결과·이슈·사용자·권한·설정 | 필수 |
| Elasticsearch (임베디드) | 검색·집계용 인덱스 | DB로부터 재생성 가능 |
| Filesystem 일부 | Plugin jar, 임시 분석 리포트 | 선택 |

DB가 진실의 원본이라는 것이 핵심이다. ES 인덱스가 손상돼도 DB로부터 재생성할 수 있다. 따라서 백업 전략의 1순위는 DB이며, ES는 운영 시간 단축을 위한 옵션이다.

### 1.1 DB만 백업해도 충분한 이유

다음 시나리오를 가정한다 — DB 백업만 가지고 호스트가 완전히 사라진 상황에서 복구.

1. 새 호스트 준비, 같은 SonarQube 버전 설치.
2. DB 백업 복원.
3. SonarQube 시작 → 자동으로 ES 인덱스 재생성 시도.

3단계에서 ES가 처음부터 다시 채워진다. 인스턴스 크기에 따라 분 단위에서 시간 단위로 걸리지만 데이터 손실은 없다. 그동안 검색·이슈 목록 조회가 비어 보이지만 분석 자체는 가능하다.

ES도 함께 백업하면 3단계가 즉시 끝난다. 차이는 "복구 직후 검색이 비어 있는 시간"이며, 큰 인스턴스에서 이 시간이 운영 영향을 만든다.



## 2. PostgreSQL 백업 패턴

> SonarSource가 권장하는 DB는 PostgreSQL이다. 백업은 PostgreSQL의 표준 도구를 그대로 쓴다.

PostgreSQL의 두 가지 백업 방식이 모두 SonarQube에 사용 가능하다.

### 2.1 논리 백업 — pg_dump

가장 단순한 방식이다.

```bash
pg_dump -h localhost -U sonar -F c -b -v -f sonar_backup_$(date +%Y%m%d).dump sonar
```

옵션 의미는 다음과 같다.

- `-F c` — custom 형식. 압축 + 병렬 복원 가능.
- `-b` — large object 포함.
- `-v` — verbose.
- `-f` — 출력 파일.

복원은 `pg_restore`로 한다.

```bash
pg_restore -h localhost -U sonar -d sonar -v sonar_backup_20260507.dump
```

논리 백업의 장점은 PostgreSQL 메이저 버전이 달라도 복원 가능하다는 것이다. PostgreSQL 14에서 떠서 PostgreSQL 16으로 옮길 수 있다. 단점은 큰 인스턴스에서 시간이 오래 걸린다(GB 단위 분석 이력에서 30분 이상).

### 2.2 물리 백업 — Continuous Archiving + PITR

대규모 운영에서는 물리 백업이 적합하다.

- WAL(Write-Ahead Log) 아카이빙으로 시점 복구(Point-In-Time Recovery) 가능.
- `pg_basebackup`으로 base 스냅샷 + WAL로 시점 복원.
- 백업 자체는 빠르지만 같은 메이저 버전 PostgreSQL에서만 복원 가능.

운영 부담이 더 크지만 RPO(복구 시점 목표)가 분 단위로 짧다. 분석 이력이 큰 인스턴스에서 30분 분량 작업도 잃을 수 없다면 이 방식이 표준이다.

### 2.3 백업 시 SonarQube를 멈춰야 하나

PostgreSQL의 `pg_dump`는 일관된 스냅샷을 보장하므로 SonarQube를 동작 중에 백업해도 안전하다. 다만 백업 중인 시점에 진행 중인 Compute Engine 작업이 있으면, 백업에 그 작업의 중간 상태가 들어갈 수 있다. 복원 후 SonarQube가 그 중간 상태를 어떻게 처리할지가 일관되지 않으므로, 운영 환경에서는 다음 둘 중 하나를 선택한다.

- 활동 시간 외에 백업 (야간 또는 주말)
- 백업 직전 Compute Engine 큐가 비었는지 확인 (`/api/ce/activity` 폴링)



## 3. Elasticsearch 백업의 선택지

> ES 백업은 의무가 아니지만 복구 시간을 줄이기 위해 선택한다.

ES 백업의 두 길이 있다.

- **Filesystem 백업**: ES 데이터 디렉터리(`<sonarqube>/data/es8/`) 통째 복사.
- **Snapshot API**: ES의 표준 스냅샷 메커니즘.

SonarSource의 권장은 두 방법 모두 SonarQube를 정지한 상태에서 수행하는 것이다. 운영 중 ES 데이터 디렉터리를 복사하면 부분 쓰기가 들어가 인덱스 손상 가능성이 있다.

### 3.1 K8s/Helm 환경의 persistence 옵션

Helm 차트는 다음 옵션을 가진다.

```yaml
persistence:
  enabled: true   # Pod 재시작 시 ES 데이터 보존
```

이 옵션을 켜면 ES 인덱스가 PVC에 저장돼 Pod 재시작 후에도 살아 있다. 시작 시간이 단축되지만 트레이드오프가 있다 — Pod 재시작 시 ES 인덱스가 깨끗하지 않은 상태일 가능성이 있어 인덱스 손상 위험이 약간 높아진다. 공식 docs도 이 트레이드오프를 명시한다.

운영 판단:
- 큰 인스턴스(인덱스 재생성 30분 이상) → persistence 켜기
- 작은 인스턴스 → 끄고 매 시작마다 재생성

### 3.2 인덱스 재생성 시간 측정

운영 결정의 근거는 실측이다. SonarQube를 정지하고 ES 데이터 디렉터리를 비운 뒤 시작하는 시뮬레이션을 한 번 해 본다. 시작 시점부터 `/api/system/status`가 `UP`을 반환할 때까지의 시간이 인덱스 재생성 시간이다. 이 시간이 운영 다운타임의 floor를 결정한다.



## 4. 백업의 일관성 — DB와 ES를 따로 떠도 되는가

> DB와 ES가 같은 시점의 스냅샷일 필요는 없다. 그러나 차이가 너무 커지면 복구 후 회복 시간이 늘어난다.

이론적으로는 DB만 떠도 ES는 재생성 가능하므로 두 저장소의 시간 동기화는 의무가 아니다. 그러나 다음을 고려한다.

- DB가 t1 시점이고 ES가 t0 시점(t0 < t1)으로 따로 백업됐다면, 복원 시 ES가 t1까지 따라오기 위해 인덱스 갱신이 필요.
- 갱신 양이 적으면 ES Snapshot의 가치가 살아 있지만, 양이 많으면 처음부터 재생성하는 것과 시간 차이가 없어짐.

운영 패턴은 DB와 ES를 같은 정지 창에서 함께 떠는 것이다. 30분 다운타임 안에 두 백업을 모두 끝내고, 복구 시 둘을 같은 시점으로 복원한다. 이 일관성이 복구 시간을 안정화한다.

### 4.1 PostgreSQL과 ES의 백업 포함 시각 차이

PostgreSQL의 `pg_dump`는 트랜잭션 시점을 기준으로 일관된 스냅샷이지만, ES의 filesystem 복사는 정지된 인덱스의 디스크 상태가 시점이다. 둘이 완벽히 같은 시점이지는 않다 — `pg_dump` 시작 시점과 ES 정지 시점의 차이만큼 어긋난다.

이 어긋남이 의미가 있는 시나리오는 거의 없지만, 분석 결과가 DB에 들어갔는데 ES 인덱스에 반영되기 전에 백업이 시작되면 복구 후 검색에 누락이 생긴다. 다음 분석 시점에 자동 갱신되므로 운영 사고로 이어지지는 않는다.



## 5. 복구 절차

> 백업 정책의 가치는 복구 시연으로 검증된다. 복구를 한 번도 시연해 보지 않은 백업은 사용 가능 여부를 알 수 없다.

복구의 표준 절차는 다음과 같다.

1. 새 호스트 또는 동일 호스트 준비.
2. SonarQube 정지 (실행 중이라면).
3. DB 복원 — `pg_restore` 또는 물리 복원.
4. ES 인덱스 복원 (있으면) — 데이터 디렉터리 복사 또는 Snapshot 복원.
5. SonarQube 설정 파일 (`sonar.properties`) 확인 — DB 접속 정보가 새 환경에 맞는지.
6. SonarQube 시작.
7. `/api/system/status` 폴링 → `UP` 도달 확인.
8. 분석 한 번 시도 → 정상 동작 확인.

복구 시연은 분기 또는 반기마다 한다. 시연하지 않으면 다음 위험을 가진다.

- 백업 파일이 손상됐는데 그것을 모름.
- 복구 절차의 일부 단계가 시간이 지나면서 환경 변화로 작동하지 않음.
- 복구에 걸리는 실제 시간을 모름 — RTO(복구 시간 목표) 산정이 추정에 머묾.



## 6. 클라우드 환경의 자동화

> 클라우드 매니지드 DB를 쓴다면 백업의 상당 부분이 인프라에 위임된다.

AWS RDS, GCP Cloud SQL, Azure Database for PostgreSQL 같은 매니지드 서비스를 쓰면 다음이 자동화된다.

- 일일 자동 백업 (보통 7일 보존)
- WAL 아카이빙으로 시점 복원 (보통 5분 단위)
- 다중 AZ 복제 (선택)

이런 환경에서 운영자가 추가로 해야 할 것은 다음과 같다.

- 백업 보존 기간을 운영 정책과 맞추기 (기본 7일이 짧으면 30일로 연장).
- 정기적으로 백업 스냅샷을 다른 리전으로 복사 (재해 복구).
- 복원 시연을 자체 인프라처럼 분기마다 수행.

ES는 SonarQube에 임베디드이므로 매니지드 서비스의 대상이 아니다. SonarQube 자체의 데이터 디렉터리(또는 K8s PVC)에 대한 스냅샷이 별도로 필요하다.



## 7. 정리

> 백업의 두 사실 — DB가 1순위, ES는 옵션 — 만 잡으면 전략의 90%가 결정된다.

요약한다. SonarQube의 진실의 원본은 DB이며 ES는 그 인덱스이므로 백업 1순위는 DB다. PostgreSQL은 `pg_dump` 논리 백업과 WAL 기반 물리 백업 두 옵션을 가지고, 인스턴스 크기에 따라 선택한다. ES 백업은 의무가 아니지만 복구 시간 단축을 위해 선택하며 SonarQube 정지 상태에서 떠야 안전하다. K8s/Helm은 persistence 옵션으로 Pod 재시작 시간을 줄일 수 있지만 손상 위험과의 트레이드오프가 있다. 백업 일관성은 DB와 ES를 같은 정지 창에서 떠는 것으로 확보한다. 복구는 분기·반기마다 시연해 RTO를 검증한다. 클라우드 매니지드 DB는 백업 자동화의 큰 부분을 인프라에 위임할 수 있다.
