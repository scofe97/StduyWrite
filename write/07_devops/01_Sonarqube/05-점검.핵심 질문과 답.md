# 5단계 점검 — SonarQube 운영 신뢰성 핵심 질문

---

> 다루는 문서 - `05-01.DB와 Elasticsearch 백업`, `05-02.업그레이드와 LTS 정책`, `05-03.모니터링과 트러블슈팅`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다.



## Q1: SonarQube 백업에서 DB는 필수이고 Elasticsearch는 선택인 이유는?

DB가 진실의 원본이고 ES는 그 인덱스이기 때문이다. 모든 분석 결과·이슈·사용자·권한·설정이 DB에 들어 있고, ES는 검색·집계용 인덱스로서 DB로부터 재생성 가능하다. DB 백업만 가지고 새 호스트에서 SonarQube를 시작하면 ES가 처음부터 다시 채워진다 — 인스턴스 크기에 따라 분 단위에서 시간 단위가 걸리지만 데이터 손실은 없다. ES도 함께 백업하면 이 재생성 시간이 사라진다는 차이만 있다. 큰 인스턴스에서는 복구 시간 단축을 위해 ES도 백업하지만, 작은 인스턴스에서는 DB만 떠도 충분하다. 이 비대칭이 백업 전략의 출발이다.

상세: `05-01.md` §1, §1.1.

### 심화

DB와 ES를 같은 정지 창에서 함께 백업하는 패턴과, DB는 운영 중 백업하고 ES는 별도 일정으로 백업하는 패턴 중 어느 쪽이 운영 부담 대비 안전한가? 어떤 인스턴스 규모에서 결정이 갈리는가?



## Q2: PostgreSQL 논리 백업(pg_dump)과 물리 백업(WAL 아카이빙)의 선택 기준은?

인스턴스 크기와 RPO 요구다. 논리 백업은 PostgreSQL 메이저 버전이 달라도 복원 가능한 호환성이 장점이지만, 큰 인스턴스(GB 단위 분석 이력)에서 30분 이상 걸린다는 단점이 있다. 물리 백업은 WAL 아카이빙으로 분 단위 시점 복구(PITR)가 가능하지만 같은 메이저 버전에서만 복원 가능하다. 30분 분량 작업도 잃을 수 없는 큰 인스턴스라면 물리 백업이 표준이고, 매일 야간 백업으로 충분한 작은 인스턴스라면 논리 백업이 운영 부담이 작다. 클라우드 매니지드 DB(RDS, Cloud SQL)를 쓰면 두 메커니즘이 모두 인프라에 위임된다.

상세: `05-01.md` §2.1, §2.2, §6.

### 심화

같은 인스턴스에서 시간이 지나며 분석 이력이 쌓여 논리 백업이 1시간을 넘어가기 시작했다면, 어느 시점에 물리 백업으로 옮길지 판단하는 객관적 기준은 무엇인가?



## Q3: 새 LTS가 출시된 직후 곧바로 갈아타지 않는 게 흔한 이유는?

새 LTS의 첫 빌드는 알려진 이슈가 누적될 시간이 부족하다. 1~2 패치 버전을 기다리면 메이저 회귀가 드러나고 hotfix가 적용된 안정된 상태로 옮길 수 있다. SonarSource의 LTS 트랙은 2년 지원이고 LTS Previous 강등 후에도 6~9개월 병행 지원을 받으므로, 곧바로 갈아타지 않아도 시간 여유가 있다. 표준 패턴은 새 LTS 출시 후 1~2개월 기다려 패치 버전 확인 후 이전, 또는 LTS Previous 지원 종료 1~2개월 전에 마지막 갈아타기 윈도를 쓰는 것이다. 이 윈도들 사이를 활동 시간 외에 잡으면 다운타임 영향이 작다.

상세: `05-02.md` §1, §1.1.

### 심화

조직이 LTS Previous 지원 종료 직전까지 갈아타지 못하는 흔한 이유는 무엇이며, 그 이유들을 미리 줄이려면 운영 차원에서 어떤 준비가 필요한가?



## Q4: SonarQube 메이저 업그레이드의 본질이 DB 마이그레이션이라는 사실이 다운타임 산정에 어떻게 작용하는가?

큰 인스턴스에서 6시간 다운타임이 가능하다는 점이 일정 산정의 출발이다. DB 백업·새 버전 설치·DB 마이그레이션·ES 재인덱싱이 모두 동기적으로 끝나야 다음 단계로 갈 수 있고, 이 시간이 인스턴스 크기(이슈 수, 분석 이력)에 비례한다. 작은 인스턴스(이슈 1만)는 30분 안에 끝나지만 큰 인스턴스(이슈 100만)는 30분~3시간씩 각 단계가 걸린다. 야간/주말 작업창을 잡고, 마이그레이션 직전에 Compute Engine 큐를 비우고, ES는 백업하지 않은 채 깨끗이 재인덱싱하는 게 표준이다. 마이그레이션 실패 시 부분 마이그레이션 상태를 손으로 정리하는 시도는 거의 항상 더 큰 사고로 이어지므로 백업 롤백이 안전하다.

상세: `05-02.md` §2, §2.1.

### 심화

실제 운영 환경에서 업그레이드 시간을 미리 측정하려면 어떤 시뮬레이션이 가능한가? 운영 인스턴스를 멈추지 않고 추정할 수 있는 방법이 있는가?



## Q5: SonarQube의 단일 인스턴스 모델에서 진정한 무중단 업그레이드가 불가능하다는 점이 운영 영향을 작게 만드는 이유는?

SonarQube의 사용 패턴이 다른 운영 시스템과 다르기 때문이다. 분석은 빌드 시점에만 필요하고, 분석 결과 조회 UI는 실시간 트래픽이 적은 보조 인터페이스다. 일시 중단이 사용자에게 직접 영향을 주는 정도가 작다는 뜻이다. 표준 운영 패턴은 CI 파이프라인의 SonarQube 호출이 실패해도 빌드 자체는 실패하지 않게 분리하는 것이며(이게 평소에도 좋은 설계), 이 분리가 있으면 SonarQube 다운타임이 빌드 흐름을 끊지 않는다. Maintenance Window를 1~2주 전 공지하고 야간/주말에 진행하면 정상 운영의 일부로 흡수된다. 이 점이 SonarQube의 다운타임 정책을 다른 시스템보다 관대하게 만든다.

상세: `05-02.md` §5, §5.1.

### 심화

CI 파이프라인이 SonarQube 다운타임에 견고해지려면 어떤 분리 패턴이 필요한가? 분석 호출 실패 시 빌드 결과 결정을 어떻게 다뤄야 하는가?



## Q6: K8s readiness probe가 SonarQube의 STARTING 상태를 어떻게 다뤄야 하는가?

readiness는 false로 두고 liveness는 통과시키는 패턴이 적합하다. STARTING은 SonarQube가 시작 중인 정상 상태이며 ES 인덱싱과 Plugin 로드가 진행 중이다. 이 시간 동안 트래픽을 보내면 사용자에게 에러가 노출되거나 시작 자체가 늦어질 수 있다. readiness false로 두면 K8s가 트래픽을 흐르지 않게 하면서도 Pod를 죽이지 않는다. liveness가 STARTING을 죽이면 시작 시간이 길어지는 큰 인스턴스에서 시작이 영원히 끝나지 않는 사고가 난다. 실용적인 readiness probe 조건은 `/api/system/status`의 `status`가 정확히 `UP`인지 검사하는 것이며, liveness는 응답 자체(어떤 상태든)를 받기만 하면 통과시킨다.

상세: `05-03.md` §1.

### 심화

readiness false 시간이 길어지면 K8s가 다음 동작으로 어떤 트레이드오프를 만드는가? 큰 인스턴스에서 readiness 타임아웃을 어떻게 잡아야 하는가?



## Q7: Compute Engine 큐 적체 진단에서 cancel을 마지막 수단으로 두는 이유는?

cancel은 작업을 잃는 것이며 그 프로젝트의 분석 결과는 다음 분석 호출까지 옛 상태에 머문다. 적체의 원인이 일시적 부하라면 기다리는 게 답이고, OOM이라면 heap을 늘리고 재시도가 답이며, DB 락이라면 락 보유자를 정리하는 게 답이다. cancel을 먼저 시도하면 원인 진단이 어려워지고 같은 적체가 반복될 가능성이 크다. 또한 cancel한 작업은 `/api/ce/activity`에서 사라지므로 사후 분석이 더 어렵다. 운영 표준은 진단 → 원인별 해결 → 마지막에 cancel 순서이며, cancel은 운영 시간을 회복하기 위한 최후 수단이다.

상세: `05-03.md` §3, §3.1.

### 심화

`/api/ce/cancel_all`로 모든 PENDING을 한 번에 cancel하는 호출은 언제 의미가 있고 언제 사고를 만드는가? 안전한 사용 시나리오의 조건은 무엇인가?



## Q8: SonarQube가 Prometheus 호환 메트릭 엔드포인트를 자체 제공하지 않는 결정이 의미하는 바는?

SonarQube의 도메인 메트릭(분석 결과, 이슈 수, Coverage 등)은 Prometheus 시계열 모델보다 SonarQube UI에서 시간 축으로 보는 게 자연스럽다는 판단이다. 이슈 수는 분석 시점마다 한 값이며 초당 측정값이 아니므로 Prometheus의 1초 단위 시계열에 어색하게 들어간다. Prometheus가 의미 있는 영역은 인프라 메트릭(JVM heap, GC, OS)이며, 이건 JMX exporter 같은 일반 도구로 충분하다. 운영자는 이 분리를 받아들이고 SonarQube 헬스/큐 상태는 Web API 폴링 기반 자체 alerting으로 다루며, JVM/OS 메트릭만 Prometheus로 흘려보낸다. 이 분담이 SonarQube의 모니터링 모델을 단순하게 만든다.

상세: `05-03.md` §6, §6.1.

### 심화

SonarQube 도메인 메트릭(예: 큐 길이, 분석 실패율)을 Prometheus alerting과 통합하고 싶을 때, Web API 폴링 방식의 exporter가 가지는 한계는 무엇인가? 폴링 주기를 어떻게 잡아야 하는가?
