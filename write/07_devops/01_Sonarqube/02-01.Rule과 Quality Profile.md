# Rule과 Quality Profile

---

> 분석을 어떤 규칙으로 할지를 결정하는 두 리소스를 본다. 1장에서 모델을 잡았다면 이 문서는 그 모델 안에서 운영자가 손대는 결정 지점을 정리한다.



## 1. Rule이라는 단위

> 한 Rule은 한 검사 규칙이다. SonarQube의 모든 분석 결과는 결국 Rule의 위반으로 환원된다.

Rule 한 개의 식별자는 `<repository>:<key>` 형태다. `java:S1186`이라면 자바 저장소의 `S1186` 규칙이며 "메서드 본문이 비어 있으면 안 된다"는 검사다. SonarQube 인스턴스는 30+ 언어에 걸쳐 수천 개 Rule을 내장하고, 보안 분석 엔진(SonarSource Security Engine)이 SAST 규칙을 추가한다.

Rule이 가지는 메타는 다음과 같다.

| 속성 | 의미 | 예시 |
|------|------|------|
| `key` | Rule 식별자 | `S1186` |
| `name` | 사람이 읽을 이름 | "Methods should not be empty" |
| `language` | 적용 언어 | `java`, `js`, `py` |
| `type` | 결함 종류 (Standard) | BUG / VULNERABILITY / CODE_SMELL / SECURITY_HOTSPOT |
| `severity` | 심각도 (Standard) | BLOCKER / CRITICAL / MAJOR / MINOR / INFO |
| `cleanCodeAttribute` | Clean Code 속성 (MQR) | CONVENTIONAL / FORMATTED / IDENTIFIABLE / ... |
| `impacts` | 품질 차원별 영향 (MQR) | `[{softwareQuality: SECURITY, severity: HIGH}, ...]` |
| `tags` | 분류 태그 | `cwe`, `owasp-a3`, `convention` |

Standard Experience 모드와 MQR 모드에서 같은 Rule이 다른 메타로 보인다는 점이 1장에서 본 사실이다. 두 모드의 차이는 Rule 자체가 바뀐 것이 아니라 분류 축이 바뀐 것이다.

### 1.1 Rule이 어디서 오는가

Rule은 SonarQube가 자체 정의하는 게 아니라 분석기 플러그인이 가지고 들어온다. 자바 분석기(`sonar-java`), JS/TS 분석기(`sonar-javascript`), 파이썬 분석기(`sonar-python`) 등이 별도 SonarSource 프로젝트로 관리되며 SonarQube Server에 플러그인으로 탑재된다. SonarQube 업그레이드 시 분석기도 함께 갱신된다.

조직이 자체 Rule을 추가하고 싶다면 두 길이 있다. 자바라면 SonarQube Java Custom Rules API로 plugin을 만들거나, 단순한 패턴이라면 XPath/RegEx 기반 Rule Template을 사용한다. 운영적으로는 자체 Rule보다 활성/비활성 결정이 더 자주 일어난다.



## 2. Quality Profile — Rule들의 묶음

> Quality Profile은 한 언어에 대한 Rule 활성화 결정의 묶음이다. 프로젝트는 언어마다 하나의 Profile을 사용한다.

같은 SonarQube 인스턴스 안에서도 팀마다 적용하고 싶은 Rule 세트가 다르다. 예를 들어 신규 서비스는 모든 보안 Rule을 켜고 싶지만 레거시 모듈은 일부 Rule을 비활성화한 채 시작하고 싶을 수 있다. 이 커스터마이징의 단위가 Quality Profile이다.

각 Profile은 다음을 가진다.

- `name` — 사람이 보는 이름
- `language` — 한 언어에만 적용
- `parent` — 상속 관계가 있으면 부모 Profile
- 활성화된 Rule들 — 각 Rule에 대해 활성/비활성 + (옵션) 파라미터 오버라이드

한 프로젝트는 언어마다 정확히 하나의 Profile을 쓴다. 자바와 자바스크립트가 섞인 프로젝트라면 자바 Profile 하나, JS Profile 하나가 동시에 적용된다.

### 2.1 Sonar way — 기본 Profile

SonarQube는 모든 언어에 대해 "Sonar way"라는 기본 Profile을 제공한다. SonarSource가 권장하는 Rule 세트이며 새 인스턴스의 모든 프로젝트는 이 Profile을 default로 사용한다. 운영자가 명시적으로 다른 Profile을 default로 지정하지 않는 한 신규 프로젝트는 Sonar way로 분석된다.

Sonar way의 Rule 구성은 SonarQube 버전마다 갱신된다. 새 보안 규칙이 추가되거나 기존 규칙의 권장 severity가 바뀌면 Sonar way도 따라간다. 조직이 Sonar way를 그대로 쓰는 가장 큰 장점이 자동 갱신이다.

### 2.2 커스텀 Profile — 상속과 복제

조직 정책상 일부 Rule을 끄거나 추가 Rule을 켜고 싶다면 커스텀 Profile을 만든다. 두 가지 방법이 있다.

| 방식 | 동작 | 트레이드오프 |
|------|------|-------------|
| 복제 (Copy) | Sonar way의 현재 상태를 한 번 복사. 이후 독립적으로 변경 | 조직 의도가 명확. SonarQube 업그레이드 시 신규 Rule을 자동으로 받지 못함 |
| 상속 (Extend) | Sonar way를 부모로 두고 차이점만 정의 | 부모의 갱신을 자동 상속. 부모 변경이 자식에 미치는 영향이 추적 어려움 |

조직 표준 Profile 하나를 두고 팀별로 그것을 부모로 상속하는 패턴이 흔하다. 부모는 "절대 끄지 못하는 Rule"의 집합을 강제하고, 자식은 팀별 추가 Rule을 정의한다.

### 2.3 Profile은 언어 단위라는 사실

Profile이 언어 단위이므로 멀티 언어 프로젝트에서는 언어마다 별도 Profile 결정을 한다. 이 점이 운영자가 자주 놓치는 함정이다. 자바 Profile은 까다롭게 잡고 JS Profile은 Sonar way 그대로 쓴다면, JS 코드의 분석 품질이 자바보다 낮아진다. 멀티 언어 프로젝트에서는 모든 언어 Profile의 동등한 강도를 유지하는 정책이 필요하다.



## 3. Rule 활성화의 실무 결정

> 어떤 Rule을 켜고 끌지에 대한 일반론이 있다. 운영 경험에서 정착된 패턴을 정리한다.

기본 출발은 "Sonar way 그대로 쓰고 정말 필요한 것만 끈다"다. 대부분의 Rule이 합리적으로 필요하며 끄는 행위 자체에 정당화 비용이 따라온다. Rule을 끌 때는 다음 셋 중 하나의 이유여야 한다.

1. **언어/프레임워크 특성상 부적합**: 예를 들어 Spring 자동 주입이 일반화된 코드 베이스에서 "필드 주입 금지" Rule은 false positive 비율이 너무 높다.
2. **조직 코딩 컨벤션과 충돌**: 메서드 이름 길이, 클래스 line 수 같은 정량 기준이 조직 표준과 다를 때.
3. **개선 시간이 명백히 부족**: 레거시 모듈의 거대한 메서드 길이 Rule을 한 번에 끌 수 없는 경우 일시적 비활성.

세 번째 이유는 임시 처방이며 6개월 단위로 재검토하는 정책을 함께 운영해야 한다. 끈 채로 잊히는 Rule이 가장 나쁜 부채다.

### 3.1 false positive 처리 — Rule 비활성화 vs Issue 분류

특정 Rule이 false positive를 자주 낸다고 Rule 자체를 끄는 것은 마지막 선택지여야 한다. 대안 셋이 있다.

- **이슈 단위 처리**: 개별 Issue를 "False Positive" 또는 "Won't Fix"로 분류. Rule은 살아 있고 다른 위치의 진짜 위반은 계속 잡힌다.
- **Rule 파라미터 조정**: 일부 Rule은 임계값 파라미터를 가진다. 예: "메서드 인지 복잡도 ≥ 15면 위반" 기본값을 20으로 올리면 일부 false positive가 사라진다.
- **scope 제한**: `sonar.exclusions`로 특정 경로를 분석에서 빼거나, Rule별로 `sonar.issue.ignore.multicriteria`로 제외 패턴 정의.

Rule 비활성화는 한 위반 패턴 전체를 잊는 결정이다. 위 셋이 가능하다면 그쪽을 먼저 시도한다.



## 4. MQR 모드에서의 Profile 운영

> MQR 모드에서는 한 Rule이 여러 품질 차원에 영향을 미치므로 Profile 결정의 의미가 달라진다.

Standard 모드에서 "이 Rule은 Code Smell"이라는 분류는 Profile 단계에서 활성/비활성으로 끝난다. MQR 모드에서는 같은 Rule이 security: High, reliability: Medium처럼 두 차원에 영향을 가지므로, 비활성화 결정의 영향이 더 크다. 한 Rule을 끄면 두 차원의 분석 결과가 동시에 약해진다.

MQR Profile 운영의 권장 패턴은 다음과 같다.

1. **품질 차원별로 Rule 묶음을 분리해 검토**: security 차원의 Rule은 따로 모아서 끄지 말고, reliability 차원의 Rule을 끄는 결정과 분리한다.
2. **severity 단계 변경을 활성/비활성보다 먼저 시도**: MQR 모드는 Profile에서 Rule의 severity를 차원별로 오버라이드할 수 있다. 너무 시끄러운 Rule이라면 끄기 전에 severity를 Low로 낮춘다.
3. **품질 차원 Rating과 Profile의 일치 확인**: Maintainability Rating A를 목표로 한다면 maintainability 영향 Rule을 충분히 켜야 한다. Profile에서 Rule을 빼는 결정은 Rating 목표와 충돌하지 않는지 본다.



## 5. Profile 변경의 운영 비용

> Profile을 바꾸면 기존 분석 결과가 어떻게 영향받는지에 대한 사실이 있다.

운영자가 자주 부딪히는 질문 셋이다.

**Q. Profile에 새 Rule을 켜면 과거 분석 결과가 바뀌는가?**
바로 바뀌지는 않는다. Profile 변경은 다음 분석부터 적용된다. 이전 분석 결과의 Issue는 그대로 남고, 새 분석에서 Rule이 활성화돼 있으면 그 시점의 코드에서 위반을 찾아 Issue를 만든다. 결과적으로 같은 코드에 대한 새 Issue가 생긴다.

**Q. Profile에서 Rule을 끄면 기존 Issue는 어떻게 되는가?**
다음 분석에서 그 Rule의 모든 Issue가 CLOSED 상태로 자동 전환된다. 결함이 사라진 게 아니라 검사를 안 하는 것뿐이지만, SonarQube의 데이터는 결함이 사라진 것처럼 보인다. 이 점이 Rule 비활성화의 가장 큰 비용이다.

**Q. Profile을 바꾸면 Quality Gate 결과가 즉시 바뀌는가?**
아니다. Profile 변경 → 다음 분석 → 새 Issue 집합 → 새 측정값 → 새 Gate 결과 순서다. CI 빌드가 다시 돌기 전까지 Gate 결과는 옛날 값이다.

이 셋을 알고 있으면 Profile 변경 시기를 PR 마감 직후 또는 야간 분석 시간으로 잡는 게 자연스럽다. 활동 시간에 Profile을 흔들면 분석 결과 해석이 어려워진다.



## 6. 정리

> Rule과 Profile의 책임 분리만 명확히 잡으면 분석 모델 운영의 핵심이 잡힌다.

이 문서를 한 문단으로 요약한다. Rule은 한 검사 규칙이며 분석기 플러그인이 가지고 들어온다. Quality Profile은 한 언어에 대한 Rule 활성화의 묶음이고 프로젝트는 언어마다 하나를 쓴다. 기본 Profile인 Sonar way는 SonarSource가 권장하는 세트이며 자동 갱신을 받는다. 커스텀 Profile은 복제 또는 상속으로 만들고, Rule 비활성화는 마지막 선택지여야 한다. MQR 모드에서는 한 Rule이 여러 품질 차원에 영향을 미치므로 비활성보다 severity 조정을 먼저 시도한다.
