# 분석 실행 로그 수집

---

> SonarQube가 발사한 Webhook을 받는 쪽의 이야기다. TPS `ppln-logging-api`는 SonarQube를 직접 호출하지 않고 결과만 수신해 자체 로그 모델로 적재한다.



## 1. 책임이 어디까지인가

> ppln-logging-api는 분석을 실행하지도, 결과를 저장하지도 않는다. 환경 코드로 식별된 SonarQube 실행 결과를 다른 모듈로 라우팅한다.

TPS의 모듈 분담을 다시 본다. `pipeline-api`가 분석을 실행하고, `operator/cicd`가 분석 결과를 사용자에게 노출하며, `ppln-logging-api`는 파이프라인 실행 로그를 수집한다. SonarQube 분석은 파이프라인 실행 환경 중 하나로 분류돼 `ppln-logging-api`의 로그 라이터에 들어온다.

`ppln-logging-api`가 SonarQube에 직접 의존하지는 않는다. SonarQube Web API 호출은 모두 `operator/cicd`가 가져갔다. `ppln-logging-api`가 다루는 것은 "이 파이프라인 실행이 SonarQube 환경이었다"는 분류와, 그에 맞는 후속 처리 분기뿐이다.



## 2. 환경 코드로 SonarQube 실행을 식별

> 파이프라인 한 회의 실행 환경을 식별하는 코드 체계가 있다. SonarQube는 그 코드의 한 값일 뿐이다.

`ppln-logging-api`는 파이프라인 실행 환경을 두 글자 코드로 분류한다. 코드는 `JenkinsConstant`에 정의돼 있다.

```java
// ppln-logging-api/src/main/java/.../v3/infrastructure/util/jenkins/vo/JenkinsConstant.java:21
public static final String PPLN_ENVRN_CD_SONARQUBE = "SQA";
```

같은 파일의 다른 환경 코드와 비교하면 분류 의도가 분명해진다.

| 코드 | 환경 | 후속 처리 |
|------|------|----------|
| `BLD` | Build (빌드) | 빌드 로그 수집·분석 |
| `UTT` | Unit Test (단위 테스트) | JUnit 결과 파싱 |
| `SQA` | SonarQube (정적 분석) | 분석 결과 라우팅 |
| `ETC` | 기타 | 기본 처리 |

코드 자체가 두 글자 약어인 점이 한국 SI 환경의 관습을 반영한다. 도메인 명명 규칙은 코드보다 비즈니스 합의가 우선한다는 사실이 그대로 드러난 부분이다.

### 2.1 사용 분기 코드 — 정적 분석 또는 단건 실행

`PPLN_USG_SE_SONARQUBE = "03"` 같은 사용 분기 코드도 함께 정의된다. 이는 한 환경(SonarQube)이 어떤 사용 시나리오로 호출됐는지를 분류한다. 03이라는 번호는 다른 사용 시나리오와 구분만 되면 의미는 환경 외부에서 결정된다.



## 3. 환경 코드별 분기 처리

> SonarQube 환경의 실행 결과는 다른 환경(빌드, 단위 테스트)과 별도 분기를 가진다. 그 분기 안에서 결과 객체가 만들어진다.

분기 처리의 진입은 `LogWriterImpl.java`다.

```java
// ppln-logging-api/src/main/java/.../v3/domain/log/service/impl/LogWriterImpl.java:101
} else if (StringUtils.equals(target.getEnvrnCd(), PPLN_ENVRN_CD_SONARQUBE)) {

    AnalysisResultVo analysisResultVo = new AnalysisResultVo();
    analysisResultVo.setTcktNo(target.getTcktNo());
    analysisResultVo.setCompnInptOrd(target.getCompnInptOrd());
    analysisResultVo.setTrigrSn(target.getTrigrSn());
    analysisResultVo.setPplnNo(target.getPplnNo());
    analysisResultVo.setUsgSe(target.getUsgSe());

    testPplnResponse = pipelineApiFeignClient.endAnalysisTest(analysisResultVo);
    log.debug("===============[ TEST PIPELINE REQUEST ]==============[ POST SONARQUBE RESULT ]===> {}", testPplnResponse);
```

이 블록의 핵심은 다섯 줄이다.

1. 환경 코드가 `SQA`인 경우에만 진입
2. `AnalysisResultVo`로 도메인 결과 객체 구성
3. 식별 키 다섯 개(`tcktNo`, `compnInptOrd`, `trigrSn`, `pplnNo`, `usgSe`) 결합
4. `pipelineApiFeignClient.endAnalysisTest()`로 pipeline-api에 결과 통보
5. 4xx/5xx가 아니면 성공 반환

식별 키 다섯 개의 책임은 다음과 같다.

| 키 | 의미 | 출처 |
|------|------|------|
| `tcktNo` | 티켓 번호 (TPS의 작업 단위) | 파이프라인 트리거 시 결정 |
| `compnInptOrd` | 컴포넌트 입력 순서 | 한 티켓 안에서의 단계 |
| `trigrSn` | 트리거 시퀀스 | 같은 컴포넌트 재실행 시 증가 |
| `pplnNo` | 파이프라인 번호 | 파이프라인 정의 식별자 |
| `usgSe` | 사용 분기 | 정적 분석 / 단건 실행 / ... |

다섯 키의 결합으로 한 분석 실행을 유일하게 식별할 수 있다. SonarQube가 분석 결과 자체를 자기 DB에 저장하지만, TPS는 그 분석이 어느 티켓의 어느 단계에서 어느 트리거로 일어났는지를 자체 컨텍스트로 남기는 것이다.



## 4. ETC 환경의 워크어라운드

> SonarQube 단건 실행이 환경 코드를 우회해 ETC로 들어오는 경우가 있다. 코드 주석이 그 우회의 흔적이다.

다음 분기는 SonarQube 환경이 ETC로 분류된 경우의 처리다.

```java
// LogWriterImpl.java:113
// SONARQUBE 단건 실행이 추가됨에 따라 환경이 ETC(기타)로 고정됨으로 해당 조건일 경우 다음과 같이 응답하도록 구성됨.
} else if (StringUtils.equals(target.getEnvrnCd(), PPLN_ENVRN_CD_ETC)) {

    AnalysisResultVo analysisResultVo = new AnalysisResultVo();
    analysisResultVo.setPplnNo(target.getPplnNo());
    analysisResultVo.setUsgSe(target.getUsgSe());

    testPplnResponse = pipelineApiFeignClient.endManualAnalysisTest(analysisResultVo);
```

주석이 솔직하다. SonarQube를 단건으로 실행하는 기능이 추가됐는데, 그 기능에서는 환경 코드가 `SQA`가 아닌 `ETC`(기타)로 들어온다. 식별 키도 `pplnNo`와 `usgSe` 둘만 채운다. 호출하는 메서드도 `endAnalysisTest`가 아니라 `endManualAnalysisTest`로 다르다.

이 워크어라운드는 도메인 분류 체계가 운영 중에 변형된 흔적이다. 처음에는 "환경 코드 SQA가 SonarQube"라는 단순 매핑이었는데, 단건 실행 기능이 환경 코드 분류와 충돌하는 형태로 들어왔고, 코드 베이스가 이 충돌을 분기로 흡수한 결과다. 도메인 모델이 시간이 지나면서 "왜 ETC가 두 가지를 의미하는가"를 설명해야 하는 상태가 됐다.

### 4.1 어떻게 정리될 수 있는가

이 형태의 부채는 다음 두 방향 중 하나로 정리된다.

- **신규 환경 코드 도입**: SonarQube 단건 실행을 위한 새 코드(예: `SQM` for SonarQube Manual)를 추가하고 ETC 분기에서 떼어 낸다. 호출자(파이프라인 트리거 측)도 함께 수정해야 하므로 변경 범위가 넓다.
- **사용 분기로 재분류**: 환경은 SQA 하나로 통일하되 `usgSe` 사용 분기에서 자동/수동을 구분한다. 코드 변경은 좁지만 기존 데이터의 환경 코드 마이그레이션이 필요하다.

운영 환경에서 첫 번째 정리가 흔히 채택된다. 두 번째는 데이터 마이그레이션 비용이 더 크기 때문이다.



## 5. SonarQube와의 관계 — 직접 호출하지 않는다는 사실

> ppln-logging-api는 SonarQube에 직접 의존하지 않는다. SonarQube가 사라져도 이 모듈의 코드는 그대로 동작한다.

`ppln-logging-api/build.gradle`에는 `org.sonarqube` 플러그인이 없고, 코드 베이스 어디에도 SonarQube Web API를 호출하는 코드가 없다. SonarQube와의 만남은 두 곳뿐이다.

1. 환경 코드 상수(`PPLN_ENVRN_CD_SONARQUBE = "SQA"`) — 단순 식별 라벨
2. 분기 처리 — 그 라벨이 들어왔을 때 분석 결과를 어떻게 다룰지

이 분리는 잘 된 설계의 증거다. 만약 ppln-logging-api가 SonarQube Web API를 직접 호출했다면 SonarQube 인스턴스 주소·토큰을 ppln-logging-api도 가져야 했고, SonarQube 버전 변경 시 ppln-logging-api까지 같이 깨질 위험이 있었다. 지금 구조는 다음을 의미한다.

- SonarQube가 다운돼 분석 결과 Webhook이 안 와도 ppln-logging-api는 멀쩡히 다른 환경 로그를 수집한다.
- SonarQube 응답 형식이 바뀌어도 ppln-logging-api는 영향 없다. 변환 책임은 operator의 Webhook 핸들러에 있다.
- SonarQube를 다른 정적 분석 도구로 교체할 경우, ppln-logging-api는 환경 코드 상수와 분기만 손대면 된다.

### 5.1 분석 결과는 어디로 가는가

`pipelineApiFeignClient.endAnalysisTest(analysisResultVo)` 호출이 결과의 종착지다. ppln-logging-api는 결과를 자기 DB에 저장하지 않고 `pipeline-api`로 다시 보낸다. pipeline-api가 분석 결과를 자기 도메인(파이프라인 실행 이력)에 통합한다.

이런 흐름이 분산 모듈 구성에서 흔한 패턴이다. "수집은 한 곳에서, 의미 부여는 도메인 모듈에서"라는 분담이다.



## 6. SonarQube Webhook 페이로드와의 만남

> ppln-logging-api는 SonarQube Webhook 페이로드를 직접 받지 않지만, 페이로드의 정보가 어떤 경로로 흘러 들어오는지 알 필요가 있다.

SonarQube가 발사하는 Webhook 페이로드는 다음 골격을 가진다.

```json
{
  "serverUrl": "http://sonarqube.internal",
  "taskId": "AYx...",
  "status": "SUCCESS",
  "analysedAt": "2025-12-31T15:30:45+0900",
  "project": {
    "key": "org.okestro:pipeline-api",
    "name": "pipeline-api",
    "url": "http://sonarqube.internal/dashboard?id=org.okestro:pipeline-api"
  },
  "branch": {
    "name": "main",
    "type": "LONG"
  },
  "qualityGate": {
    "name": "Sonar way",
    "status": "OK",
    "conditions": [...]
  },
  "properties": {
    "sonar.analysis.tcktNo": "TPS-1234",
    "sonar.analysis.pplnNo": "5"
  }
}
```

핵심은 마지막의 `properties`다. SonarScanner 호출 시 `-Dsonar.analysis.tcktNo=TPS-1234` 같은 식으로 임의 메타데이터를 주입할 수 있고, 이 메타가 Webhook 페이로드에 그대로 전달된다. TPS는 이 메커니즘으로 "어느 티켓의 분석인가"를 SonarQube로부터 다시 받아온다.

executor 서비스가 이 페이로드를 받고 메타에서 식별 키를 꺼낸다. 그 결과가 ppln-logging-api에 도달하는 시점에는 이미 다섯 키(`tcktNo`, `pplnNo`, `compnInptOrd`, `trigrSn`, `usgSe`)가 채워진 상태다. 즉 SonarQube의 페이로드가 TPS 도메인 키로 변환되는 곳은 ppln-logging-api가 아니라 executor의 webhook 핸들러다.



## 7. 정리 — 수집기의 책임 경계

> ppln-logging-api 사례는 외부 도구의 결과를 수집하는 모듈이 어디까지 책임지면 좋은지에 대한 답을 보여 준다.

이 모듈의 결정을 일반화하면 셋이다.

1. **외부 도구에 직접 의존하지 않는다** — 환경 코드 상수와 분기 처리만 가진다. SonarQube가 사라져도 멀쩡한 모듈이다.
2. **분류 체계의 변경은 분기로 흡수한다** — 단건 실행이 ETC로 들어오는 워크어라운드는 깨끗하지 않지만 일단 작동한다. 이런 부채는 코드 주석으로 흔적을 남기고 별도 정리 작업으로 미룬다.
3. **수집과 의미 부여를 다른 모듈로 분담한다** — `pipelineApiFeignClient.endAnalysisTest`로 결과를 도메인 모듈에 다시 보낸다. 한 모듈이 수집과 도메인 통합을 모두 책임지면 응집도가 떨어진다.

다음 단계로 갈 수 있는 두 갈래가 있다. SonarQube를 Jenkins 파이프라인에서 호출하는 일반론은 [04-04.Jenkins 파이프라인 통합](04-04.Jenkins 파이프라인 통합.md). 통합 활용 4편의 통찰 점검은 [04-점검.핵심 질문과 답](04-점검.핵심 질문과 답.md).
