# 인증과 토큰

---

> SonarQube에 누가 접근하느냐를 결정하는 메커니즘을 본다. 토큰 종류, 권한 모델, 외부 인증(LDAP/SAML/OAuth)까지 운영 어휘로 잡는다.



## 1. 인증 방식 — 두 갈래

> SonarQube는 사용자 인증과 머신 인증을 분리한다. 두 방식이 다른 시나리오에 적합하다.

인증 방식은 둘이다.

- **Basic 인증 (사용자명 + 비밀번호)**: 사람이 UI 또는 외부 도구로 직접 접근할 때.
- **토큰 인증**: API 호출, CI 파이프라인, 자동화 스크립트.

토큰은 비밀번호와 동등한 권한을 가지지만 다음 장점이 있다.

- 사용자 비밀번호를 노출하지 않는다.
- 만료를 설정할 수 있다(SonarQube가 강제하지는 않지만 기능 제공).
- 사용자별로 여러 토큰을 발급해 용도별로 분리할 수 있다.
- 유출 시 토큰 단위로 revoke 가능하다.

자동화 환경에서는 무조건 토큰을 쓴다는 것이 SonarQube의 권장이다.



## 2. 토큰 세 종류

> 1장과 04-02에서 짚었던 토큰 분류를 운영 어휘로 다시 본다.

세 종류 토큰이 있다.

| 토큰 종류 | 권한 범위 | 발급 책임 | 적합한 시나리오 |
|----------|----------|----------|---------------|
| User Token | 사용자의 모든 권한 | 사용자 본인 | 사람이 자신의 도구를 인증할 때 |
| Project Analysis Token | 한 프로젝트의 분석 업로드 | 프로젝트 관리자 | CI 파이프라인의 분석 호출 |
| Global Analysis Token | 모든 프로젝트의 분석 업로드 | SonarQube 관리자 | 여러 프로젝트를 분석하는 공용 빌드 머신 |

세 토큰의 폭발 반경(유출 시 영향)이 결정적으로 다르다.

- **User Token 유출**: 그 사용자가 가진 모든 권한이 통째로 노출. 관리자라면 인스턴스 전체 통제권 손실.
- **Project Analysis Token 유출**: 한 프로젝트의 분석 결과 위변조 가능. 다른 프로젝트는 무관.
- **Global Analysis Token 유출**: 모든 프로젝트의 분석 결과 위변조 가능. 관리 기능은 사용 못함.

CI 시크릿에 어느 토큰을 두느냐에 따라 운영 사고의 범위가 정해진다는 사실이 토큰 선택의 출발점이다.

### 2.1 토큰 발급과 관리

토큰 발급은 다음 경로다.

- **User Token**: 사용자가 자기 계정 페이지에서 발급. UI Account → Security → Generate Tokens.
- **Project Analysis Token**: 프로젝트 관리자가 프로젝트 설정에서 발급. UI Project Settings → Security → Generate Token.
- **Global Analysis Token**: SonarQube 관리자가 인스턴스 설정에서 발급. UI Administration → Security → Tokens.

각 토큰은 발급 시점에만 한 번 평문으로 보인다. 이후는 SonarQube DB에 해시로만 저장되며 다시 볼 수 없다. 분실 시 새로 발급해야 한다.

토큰에 만료일(expiration date)을 설정할 수 있지만 강제는 아니다. 만료 없는 토큰은 영원히 살아 있으므로 운영 정책상 90일 또는 180일 만료를 강제하는 패턴이 흔하다. 만료 강제는 SonarQube 인스턴스 차원이 아니라 운영자의 정책 차원이다.

### 2.2 토큰 회전 패턴

회전(rotation) 패턴은 다음 단계다.

1. 새 토큰 발급 (만료일 = 현재 + 90일).
2. CI 시크릿 매니저에 새 토큰 등록.
3. 다음 빌드부터 새 토큰 사용 확인.
4. 옛 토큰을 SonarQube에서 revoke.

3번과 4번 사이에 시간 간격을 두는 게 안전하다. 옛 토큰을 즉시 revoke하면 캐시된 빌드 잡이 실패할 수 있다. 24시간 정도 옛 토큰을 살려 둔 뒤 revoke하는 게 안전 마진이다.

여러 프로젝트가 같은 Global Token을 공유하면 회전 시 모든 시크릿을 동시에 갱신해야 한다. 동시 갱신이 어려운 환경에서는 Project Analysis Token으로 분리하는 게 운영 비용이 더 작다.



## 3. HTTP 인증 헤더 형식

> 토큰을 어떻게 HTTP 헤더에 실어 보내는지에 두 형식이 있다.

SonarQube는 두 형식을 모두 받는다.

```
# 형식 1: Bearer (권장)
Authorization: Bearer squ_xxxxxxxxxxxxxxxx

# 형식 2: Basic (토큰을 username으로, 비밀번호는 빈 문자열)
Authorization: Basic c3F1X3h4eHh4eHh4Ojo=    # base64(squ_xxxx:)
```

Bearer 형식이 권장이며 SonarQube 공식 docs도 Bearer를 우선 표기한다. Basic 형식은 historical 호환을 위한 것으로, 일부 오래된 클라이언트(예: 구버전 SonarScanner)가 Basic만 지원할 때 사용한다.

operator/cicd의 SonarQubeFeignClient는 다음 형태로 헤더를 받는다.

```java
@RequestHeader("Authorization") String token
```

호출자가 `"Bearer squ_..."` 또는 `"Basic c3F..."` 전체 문자열을 만들어 넘긴다. Feign이 prefix를 자동으로 붙여 주지 않으므로 형식 결정이 호출자에 위임된다.



## 4. 권한 모델 — Permission

> 토큰은 사용자의 권한을 그대로 따라간다. 그 권한이 어떻게 구성되는지가 본 절의 주제다.

SonarQube의 권한은 두 축으로 정의된다.

- **Global Permission**: 인스턴스 전체에 적용. 인스턴스 관리, 사용자 관리, Quality Profile 관리 등.
- **Project Permission**: 한 프로젝트에 적용. Browse, Code View, Analyze, Administer Issues 등.

자주 쓰이는 권한을 정리한다.

| 권한 | 영역 | 의미 |
|------|------|------|
| Administer System | Global | 인스턴스 관리 (설정, 사용자, 라이선스) |
| Administer Quality Profiles | Global | Profile 활성화/비활성화 |
| Administer Quality Gates | Global | Gate 정의/조건 |
| Create Projects | Global | 신규 프로젝트 생성 |
| Browse | Project | 프로젝트 보기 (private 프로젝트 접근) |
| See Source Code | Project | 소스 코드 노출 |
| Execute Analysis | Project | 분석 결과 업로드 |
| Administer | Project | 프로젝트 설정 |
| Administer Issues | Project | Issue 라이프사이클 변경 (assign, transition) |

분석 호출에 필요한 권한은 `Execute Analysis` 단 하나다. Project Analysis Token이 이 권한만 가지므로 토큰 유출 시 분석 결과 위변조까지만 가능하고 프로젝트 설정 변경은 불가능하다.

### 4.1 Permission Template — 신규 프로젝트의 default

매번 프로젝트마다 권한을 손으로 설정하면 운영 부담이 크다. Permission Template은 신규 프로젝트가 자동으로 받을 권한 셋을 정의한다.

- 새 프로젝트가 생성될 때 매칭되는 Template의 권한이 자동 부여.
- Template 매칭은 프로젝트 키 패턴(regex)으로 결정.
- "기본" Template 하나가 모든 프로젝트의 fallback.

조직 구조가 있는 운영 환경의 표준 패턴은 다음과 같다.

- `org.okestro.tps.*` 패턴 → TPS 팀 Template (TPS 팀 그룹에 Browse + Execute Analysis)
- `org.okestro.bok.*` 패턴 → BOK 팀 Template
- 그 외 → 기본 Template (개발자 그룹 일괄 부여)

Template으로 권한을 위임하면 프로젝트별 권한 설정 작업이 사라진다.



## 5. 외부 인증 — LDAP / SAML / OAuth

> 사내 디렉터리 통합은 운영의 빠른 정착을 결정한다. SonarQube는 세 종류 외부 인증을 지원한다.

지원되는 외부 인증은 다음과 같다.

- **LDAP**: Active Directory를 포함한 표준 LDAP 디렉터리. 가장 흔한 사내 통합.
- **SAML 2.0**: SAML 기반 SSO. Okta, Azure AD, Auth0 등.
- **OAuth 2.0**: GitHub, GitLab, Bitbucket Cloud, Microsoft Entra ID(구 Azure AD).

기본 인증(SonarQube 자체 사용자 DB)과 외부 인증은 동시에 켤 수 있다. 외부 인증 실패 시 fallback으로 자체 사용자 DB가 시도된다.

### 5.1 그룹 매핑

외부 인증의 진정한 가치는 사용자 동기화가 아니라 그룹 매핑이다. LDAP 그룹 또는 SAML 클레임을 SonarQube 그룹에 매핑하면 권한 관리가 디렉터리 한 곳에 모인다.

```properties
# LDAP 그룹 매핑 예시
sonar.security.realm=LDAP
ldap.url=ldap://ldap.internal:389
ldap.bindDn=cn=admin,dc=okestro,dc=com
ldap.bindPassword=...

# 사용자 검색
ldap.user.baseDn=ou=People,dc=okestro,dc=com
ldap.user.request=(&(objectClass=inetOrgPerson)(uid={login}))

# 그룹 검색
ldap.group.baseDn=ou=Groups,dc=okestro,dc=com
ldap.group.request=(&(objectClass=groupOfNames)(member={dn}))
```

사용자가 로그인할 때마다 LDAP 그룹이 동기화되고 SonarQube 그룹에 매핑된다. 디렉터리에서 그룹 멤버십을 변경하면 다음 로그인부터 SonarQube 권한이 따라 바뀐다.

### 5.2 외부 인증과 토큰의 관계

외부 인증을 쓰는 환경에서도 토큰은 SonarQube 자체 메커니즘이다. LDAP 사용자가 로그인 후 발급한 User Token은 LDAP의 비밀번호와 무관하게 SonarQube DB에 저장된다. 사용자가 LDAP에서 비활성화되면 다음 로그인에서 차단되지만 발급된 토큰은 별도 revoke 전까지 살아 있다.

이 분리가 운영 위험이다. 퇴사자의 LDAP 계정을 막아도 그 사람의 SonarQube 토큰이 계속 작동한다. 퇴사 프로세스에 "SonarQube 토큰 revoke"가 명시되지 않으면 사고가 잠재된다.

운영 패턴은 정기 토큰 회전(90일)을 강제하는 것이다. 회전이 강제되면 퇴사 90일 후에는 옛 토큰이 자동 만료된다. 즉시 차단이 필요한 보안 사고에서는 SonarQube 관리자가 명시적으로 revoke한다.



## 6. UI 접근과 익명 사용자

> SonarQube는 익명 접근(unauthenticated)을 허용하는 모드와 차단하는 모드가 있다. 두 모드의 트레이드오프를 본다.

기본 설정은 다음 둘 중 하나다.

- **Force user authentication = false (default)**: 익명 사용자도 public 프로젝트의 결과를 볼 수 있음.
- **Force user authentication = true**: 모든 접근에 로그인 필요.

오픈소스 프로젝트의 SonarQube(예: 공개 SonarCloud) 분석 결과는 익명 접근이 의미 있지만, 사내 인스턴스는 익명 차단이 표준이다. 차단하지 않으면 사내 코드 일부가 외부에 노출될 위험이 있다.

설정은 다음 한 줄이다.

```properties
sonar.forceAuthentication=true
```

Property로 적용하거나 Administration → Security → "Force user authentication" 토글로 바꾼다. 이 설정 하나가 인스턴스의 보안 자세를 결정한다.



## 7. 정리

> 토큰 종류와 권한 모델만 잡으면 SonarQube 인증 운영의 80%가 결정된다.

요약한다. SonarQube는 사용자 Basic 인증과 토큰 인증 두 갈래를 가지며 자동화는 토큰이 표준이다. 토큰은 User / Project Analysis / Global Analysis 셋이며 폭발 반경이 다르므로 CI 파이프라인은 Project Analysis Token이 기본이다. 권한은 Global과 Project 두 축으로 정의되며 Permission Template으로 신규 프로젝트의 default를 자동화한다. 외부 인증(LDAP/SAML/OAuth)은 그룹 매핑이 핵심 가치이고, 외부 인증을 써도 토큰은 SonarQube 자체 메커니즘이라 정기 회전이 필요하다.
