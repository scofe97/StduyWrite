# SonarScanner for Gradle 적용 패턴

---

> Gradle 프로젝트가 SonarQube 분석에 들어가는 가장 단순한 길은 `org.sonarqube` 플러그인이다. TPS `pipeline-api` 모듈의 실제 적용을 따라가며 어떤 결정이 어떤 동작을 부르는지 본다.



## 1. 한 줄로 끝나는 적용

> Gradle Scanner의 적용은 플러그인 한 줄이 전부다. 그 한 줄이 무엇을 켜는지가 본 절의 주제다.

TPS `pipeline-api`의 빌드 스크립트는 다음 한 줄로 SonarQube 분석을 켠다.

```groovy
// pipeline-api/build.gradle:8
plugins {
    id 'java'
    id 'org.springframework.boot' version '3.2.3'
    id 'io.spring.dependency-management' version '1.1.4'
    id 'com.google.cloud.tools.jib' version '3.4.0'
    id 'com.github.davidmc24.gradle.plugin.avro' version '1.9.1'
    id 'org.sonarqube' version '4.4.1.3373'
}
```

이 한 줄이 적용되면 Gradle 프로젝트에 `sonar` 태스크(과거 이름 `sonarqube`도 alias)가 추가된다. CI 또는 개발자 머신에서 다음 명령으로 분석을 호출한다.

```bash
./gradlew sonar \
  -Dsonar.host.url=$SONAR_HOST_URL \
  -Dsonar.token=$SONAR_TOKEN \
  -Dsonar.projectKey=org.okestro:pipeline-api
```

플러그인은 위 호출 시점에 다음을 자동 수행한다.

1. Gradle 프로젝트 메타에서 소스/테스트 디렉터리, 언어, 인코딩, 컴파일된 바이너리(`build/classes/java/main` 등) 위치를 추출한다.
2. 의존성 클래스패스를 SonarQube에 전달해 SAST 계열 Rule이 바이트코드 분석을 할 수 있게 한다.
3. JaCoCo 또는 다른 커버리지 플러그인이 적용돼 있으면 리포트 경로를 자동 인식한다.
4. SonarScanner 본체를 다운로드해 분석을 실행하고 결과를 SonarQube Server로 업로드한다.

`pipeline-api`에는 `sonar-project.properties`가 없다. Gradle 메타가 모든 속성을 채워주기 때문이다. 이게 빌드 시스템 통합 Scanner를 쓰는 이유다.



## 2. 플러그인이 손대지 않는 영역

> 플러그인이 자동화하는 부분과 운영자가 결정해야 하는 부분의 경계를 잡는다.

플러그인은 분석 호출의 인터페이스만 제공하며 다음 영역에는 손대지 않는다.

- **SonarQube Server 주소와 토큰**: `-Dsonar.host.url`, `-Dsonar.token` (또는 `SONAR_HOST_URL`, `SONAR_TOKEN` 환경변수)으로 외부 주입.
- **프로젝트 키와 표시 이름**: 자동 추정값을 받지만, 다중 모듈에서는 명시 권장.
- **분기 분석**: PR 분석, 다중 브랜치 분석은 Developer Edition 이상에서만 동작하며 `sonar.pullrequest.*`, `sonar.branch.name` 속성을 별도 전달.
- **Quality Gate 대기 여부**: `sonar.qualitygate.wait=true` 옵션은 운영자가 켜는 것이며 플러그인이 강제하지 않는다.
- **커버리지 리포트 생성**: JaCoCo 등 커버리지 도구의 적용은 별도다. 플러그인은 결과 리포트 경로만 인식한다.

이 분리가 의미 있는 지점은 토큰 관리다. 플러그인이 토큰을 보관하지 않으므로 CI 시크릿 매니저에서만 노출한다. `build.gradle`에 토큰을 박는 것은 안티패턴이다.

### 2.1 다중 모듈 프로젝트에서의 적용

Gradle 멀티 프로젝트에서는 루트 빌드 스크립트에만 플러그인을 적용하는 게 표준이다. 서브 프로젝트는 자동으로 분석 대상에 포함된다. SonarQube는 서브 프로젝트별로 component를 만들고 모듈 트리 형태로 결과를 제공한다.

`pipeline-api`는 단일 모듈이라 이 결정이 단순하지만, `operator` 같은 멀티 모듈(`app`/`cicd`/`core-library`/...)에서는 루트 적용 후 `subprojects { sonar { ... } }` 블록으로 모듈별 속성을 조정하는 패턴을 쓴다.



## 3. Gradle Tooling API 의존성의 의미

> `pipeline-api/build.gradle:148`에 `// sonar gradle` 주석과 함께 추가된 의존성이 있다. 왜 추가됐는지를 알면 플러그인의 동작 모델이 더 분명해진다.

해당 라인은 다음과 같다.

```groovy
// pipeline-api/build.gradle:148
implementation 'org.gradle:gradle-tooling-api:7.3-20210825160000+0000' // sonar gradle
```

Gradle Tooling API는 외부 프로세스가 Gradle 빌드 결과를 프로그래밍 방식으로 조회할 수 있게 해 주는 라이브러리다. `pipeline-api` 자체가 Gradle 빌드 모델을 호출자 쪽에서 다뤄야 하는 도메인을 가지고 있고(파이프라인이 외부 Gradle 프로젝트를 분석/실행), 그 의존성이 SonarQube 도메인 코드(외부 Gradle 프로젝트를 분석한다는 점에서 같은 종류의 일)와 묶여 같은 모듈에 들어 있어서 주석에 `// sonar gradle`이라고 표기된 것이다.

요점은 SonarScanner for Gradle 플러그인 자체가 Tooling API를 요구하는 것은 아니라는 것이다. 이 의존성은 도메인 코드의 필요로 들어왔다. SonarScanner는 자기 plugin 분리 빌드 컨테이너에서 자체 의존성으로 Tooling API를 가져온다.

이 구분이 중요한 이유는 다른 모듈에서 같은 패턴을 복제할 때 "이걸 따라 넣어야 SonarQube가 동작한다"고 오해하지 않기 위해서다. SonarQube 플러그인 적용에 추가 의존성은 필요 없다.



## 4. 커버리지 연동 — JaCoCo 패턴

> SonarQube는 커버리지를 측정하지 않고, JaCoCo 같은 커버리지 도구의 결과를 흡수만 한다. 두 도구의 연동 지점이 분석 품질을 결정한다.

JaCoCo와 SonarQube의 분담은 명확하다.

- **JaCoCo**: 단위 테스트 실행 시 바이트코드 instrumentation으로 line/branch coverage를 측정해 `build/reports/jacoco/test/jacocoTestReport.xml`(또는 .exec 바이너리)을 생성.
- **SonarQube Scanner**: JaCoCo 리포트를 읽어 분석 결과에 통합. 자체 측정은 하지 않음.

연동의 표준 패턴은 다음과 같다.

```groovy
plugins {
    id 'java'
    id 'jacoco'
    id 'org.sonarqube' version '4.4.1.3373'
}

jacocoTestReport {
    reports {
        xml.required = true   // SonarQube가 읽을 수 있는 XML
        html.required = false
    }
}

tasks.named('test') { finalizedBy 'jacocoTestReport' }
tasks.named('sonar') { dependsOn 'jacocoTestReport' }
```

XML 출력이 핵심이다. SonarScanner는 HTML 리포트를 읽지 못하고 XML만 인식한다. `pipeline-api/build.gradle`에는 jacoco 플러그인 자체가 적용돼 있지 않은데, 이는 모듈이 자체 커버리지 측정 부담 없이 SonarQube로 정적 분석만 받는 구성이라는 뜻이다.

### 4.1 커버리지 없는 분석의 한계

커버리지 리포트가 없으면 SonarQube의 Quality Gate에서 "신규 코드 Coverage ≥ 80%" 같은 조건은 0%로 평가돼 게이트가 실패한다. Sonar way 기본 게이트가 신규 코드 Coverage 80%를 요구하기 때문이다. JaCoCo를 빼는 선택은 "정적 분석만 하고 커버리지 게이트는 끈다"는 결정과 짝을 이뤄야 한다.



## 5. CI 환경에서의 호출

> 빌드 시스템에 플러그인을 적용했어도 분석을 실제로 호출하는 곳은 CI다. CI가 토큰·서버 주소·게이트 대기 정책을 모두 결정한다.

CI 환경에서의 권장 호출 형태는 다음과 같다.

```bash
./gradlew build sonar \
  -Dsonar.host.url=$SONAR_HOST_URL \
  -Dsonar.token=$SONAR_TOKEN \
  -Dsonar.projectKey=$PROJECT_KEY \
  -Dsonar.qualitygate.wait=true \
  -Dsonar.qualitygate.timeout=300
```

각 옵션의 역할을 정리한다.

| 옵션 | 의미 | 비고 |
|------|------|------|
| `sonar.host.url` | SonarQube Server 주소 | 환경변수 `SONAR_HOST_URL`로도 받음 |
| `sonar.token` | Project/Global Analysis Token | CI 시크릿에서 주입 |
| `sonar.projectKey` | 분석 결과를 어느 프로젝트로 묶을지 | 미명시 시 Gradle 그룹·이름 기반 자동 추정 |
| `sonar.qualitygate.wait` | Compute Engine 처리 후 게이트 결과까지 대기 | true면 게이트 FAILED 시 빌드 비영 종료 |
| `sonar.qualitygate.timeout` | 대기 최대 초 | 기본 300초 (5분) |

`sonar.qualitygate.wait=true`는 단일 빌드 스텝으로 게이트 결과를 받는 가장 단순한 길이다. Compute Engine이 큐 적체로 늦어지면 빌드 시간이 늘지만, 별도 폴링 스텝을 만들 필요가 없다.

### 5.1 프로젝트 키 명명 규칙

`sonar.projectKey`는 SonarQube 안에서 프로젝트를 식별하는 유일 키다. 다음 규칙으로 잡는 게 운영에 편하다.

- 모노리포 환경: `<repo-name>:<module-name>` (예: `tps-gitlab2:pipeline-api`)
- 프로젝트별 저장소: `<group>:<artifact>` (예: `org.okestro:pipeline-api`)
- 키에 콜론과 영숫자, 하이픈, 언더스코어, 마침표 사용 가능. 공백·한글 사용 금지.

키는 한 번 정해지면 바꾸기 어렵다. 키를 바꾸면 분석 이력이 끊기고 SonarQube에서 새 프로젝트로 인식한다. 첫 분석 전에 명명 규칙을 정한다.



## 6. 자주 부딪히는 함정

> Gradle 적용 자체는 한 줄이지만 운영에서 부딪히는 함정은 같은 패턴으로 반복된다.

다음 셋이 가장 자주 보고된다.

1. **토큰 권한 불일치**: User Token을 CI에 박아 둔 채 그 사용자의 프로젝트 권한이 변경되면 갑자기 분석이 401로 실패한다. Project Analysis Token을 쓰면 권한 변경의 영향이 좁아진다.
2. **동일 키 다중 분석 충돌**: 두 빌드 머신이 같은 `projectKey`로 동시에 분석을 올리면 Compute Engine이 직렬 처리이므로 한 쪽이 이긴다. PR마다 다른 브랜치 키를 쓰지 않으면 결과가 덮어써진다. PR 분석은 `sonar.pullrequest.key/branch/base`로 격리한다.
3. **Java 버전 불일치**: 분석 대상 코드가 Java 21이라도 SonarScanner 자체가 호환되지 않으면 분석이 실패한다. 2025+ Scanner는 Java 17 이상을 요구한다. CI 베이스 이미지의 JDK 버전이 SonarQube Server와 Scanner 양쪽 요구를 모두 만족하는지 확인한다.

이 셋만 피하면 Gradle Scanner 운영은 거의 자율 주행이 가능하다.



## 7. 정리

> 플러그인 한 줄이 자동화하는 부분과 운영자가 결정해야 하는 부분의 경계를 다시 짚는다.

`pipeline-api`의 적용 사례를 한 문단으로 요약하면 이렇다. `id 'org.sonarqube' version '4.4.1.3373'` 한 줄로 Gradle 프로젝트가 SonarQube 분석 대상이 된다. 플러그인은 소스·테스트·바이너리·클래스패스·커버리지 리포트를 Gradle 메타에서 자동 추출하므로 `sonar-project.properties`가 필요 없다. 운영자가 결정하는 것은 Server 주소, 토큰, 프로젝트 키, 게이트 대기 정책이며 이 넷은 CI에서 환경변수로 주입한다.
