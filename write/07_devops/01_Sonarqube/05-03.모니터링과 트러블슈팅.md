# 모니터링과 트러블슈팅

---

> 운영 중 발생하는 흔한 문제를 어떻게 감지하고 진단하느냐를 본다. 시스템 상태 엔드포인트, 로그 위치, 큐 적체·OOM·ES 이상의 흔한 시나리오를 정리한다.



## 1. 헬스체크의 출발점

> 자동화된 모니터링은 두 엔드포인트로 시작한다. `/api/system/status`와 `/api/system/health`다.

두 엔드포인트의 차이는 다음과 같다.

| 엔드포인트 | 인증 필요 | 용도 |
|-----------|----------|------|
| `/api/system/status` | 불필요 | 가동 여부 단순 확인 (UP/DOWN/STARTING/...) |
| `/api/system/health` | 관리자 토큰 | 상세 헬스 (DB·ES·Compute Engine 각각의 상태) |

오케스트레이터(K8s, Docker)의 liveness probe는 `/api/system/status`를 쓴다. 응답의 `status`가 `UP`이면 정상.

```json
GET /api/system/status

{
  "id": "AYz...",
  "version": "25.1.0.123456",
  "status": "UP"
}
```

`status` 가능 값은 다음과 같다.

| 값 | 의미 |
|----|------|
| `UP` | 정상 동작 |
| `DOWN` | 시작 실패 또는 치명적 장애 |
| `STARTING` | 시작 중 (ES 인덱싱, Plugin 로드 등) |
| `RESTARTING` | 재시작 중 |
| `DB_MIGRATION_NEEDED` | DB 스키마 마이그레이션 대기 (UI에서 트리거) |
| `DB_MIGRATION_RUNNING` | 마이그레이션 진행 중 |

`STARTING`은 K8s readiness probe에서 not-ready로 처리해야 한다. liveness는 통과시키되 readiness는 false로 두면 트래픽이 흐르지 않으면서도 Pod가 죽지 않는다.

### 1.1 상세 헬스 — `/api/system/health`

세부 컴포넌트 상태를 받으려면 `/api/system/health`를 쓴다.

```json
GET /api/system/health
Authorization: Bearer <admin-token>

{
  "health": "GREEN",
  "causes": [],
  "nodes": [
    {
      "name": "sonar-app-1",
      "type": "APPLICATION",
      "host": "10.0.0.5",
      "port": 9000,
      "startedAt": "2026-05-07T10:15:00+0000",
      "health": "GREEN",
      "causes": []
    }
  ]
}
```

`health` 값은 GREEN / YELLOW / RED 셋이다. YELLOW는 일부 기능 저하, RED는 가용성 손실. `causes`에 구체적 원인 메시지가 들어 있다.

이 엔드포인트는 관리자 토큰이 필요하므로 Prometheus exporter나 자체 헬스체크 스크립트가 토큰을 보관해야 한다. 토큰 회전 정책의 적용 대상이다.



## 2. 로그 파일의 위치와 분리

> SonarQube는 4개 컴포넌트의 로그를 별도 파일로 분리한다. 어떤 문제든 해당 컴포넌트 로그를 먼저 본다.

로그 파일은 `<sonarqube>/logs/` 또는 K8s에서 `<pod>/opt/sonarqube/logs/`에 있다.

| 파일 | 컴포넌트 | 무엇을 보는가 |
|------|----------|---------------|
| `sonar.log` | 메인 프로세스 | 시작·정지, Plugin 로드, 일반 시스템 로그 |
| `web.log` | Web Server | HTTP 요청·응답, 사용자 인증, API 에러 |
| `ce.log` | Compute Engine | 분석 처리, 큐 상태, OOM 흔적 |
| `es.log` | Search Server | Elasticsearch 시작, 인덱스 갱신, 검색 에러 |

운영 사고 발생 시 이 분리를 모르면 로그를 찾는 데 시간이 걸린다. "분석이 안 끝난다"는 보고는 `ce.log`, "검색이 비어 있다"는 보고는 `es.log`, "로그인이 안 된다"는 `web.log`다.

### 2.1 로그 레벨

기본 로그 레벨은 INFO다. 디버깅이 필요하면 다음 설정으로 DEBUG 또는 TRACE로 올린다.

```properties
# sonar.properties
sonar.log.level=DEBUG
sonar.log.level.app=INFO
sonar.log.level.web=DEBUG
sonar.log.level.ce=DEBUG
sonar.log.level.es=INFO
```

DEBUG는 디스크 사용을 빠르게 늘리므로 진단 후 INFO로 되돌린다. 디스크 회전 정책(logback의 `RollingFileAppender`)이 기본 적용돼 있어 무한 적재는 막혀 있지만, 단기 디스크 압박이 일어날 수 있다.



## 3. 흔한 시나리오 — Compute Engine 큐 적체

> "분석이 끝나지 않는다"의 가장 흔한 원인이다.

증상은 다음과 같다.

- Scanner는 종료됐지만 SonarQube UI에 결과가 보이지 않음.
- `/api/ce/activity`가 PENDING 또는 IN_PROGRESS인 작업을 다수 반환.
- `ce.log`에서 분석 처리 메시지가 한 작업에 머물러 있음.

진단은 다음 순서다.

1. `/api/ce/activity?status=IN_PROGRESS,PENDING&ps=100` — 큐의 현재 상태 조회.
2. 가장 오래된 IN_PROGRESS 작업의 `submittedAt` 확인 — 30분 이상이면 비정상.
3. `ce.log`에서 그 작업 ID로 검색 — 처리 중 어디서 멈췄는지 확인.

해결 패턴은 셋이다.

- **단순 적체**: 큰 분석이 줄을 막은 것. 처리 완료를 기다림.
- **CE OOM**: `ce.log`에 `OutOfMemoryError`. 다음 절에서 다룸.
- **DB 락**: `ce.log`에 트랜잭션 timeout. PostgreSQL의 `pg_stat_activity`로 락 보유자 확인.

### 3.1 큐 강제 비우기

마지막 수단으로 진행 중 작업을 cancel할 수 있다.

```
POST /api/ce/cancel?id=<taskId>
```

또는 모든 PENDING을 한 번에 cancel할 수 있다 (관리자 권한).

```
POST /api/ce/cancel_all
```

cancel은 작업을 잃는 것이므로 다음 분석이 결과를 만들어 낼 때까지 그 프로젝트의 결과는 옛 상태에 머문다. 운영 환경에서는 이 호출 전에 cancel 대상을 한 번 더 확인한다.



## 4. 흔한 시나리오 — Compute Engine OOM

> 큰 분석 리포트를 처리할 때 메모리가 부족하면 일어난다.

증상은 다음과 같다.

- `ce.log`에 `OutOfMemoryError: Java heap space`.
- 해당 시점에 처리 중이던 분석이 FAILED로 종결.
- 큐 적체가 누적될 수 있음 (다음 분석도 같은 패턴이면 반복 실패).

해결은 Compute Engine JVM heap 증가다.

```properties
# sonar.properties
sonar.ce.javaOpts=-Xmx2048m -Xms256m -XX:+HeapDumpOnOutOfMemoryError
```

기본값 `-Xmx512m`이 작은 인스턴스에는 충분하지만 큰 모노리포 분석에는 부족하다. 1GB → 2GB → 4GB 순으로 늘리며 실측한다.

### 4.1 heap dump 분석

`-XX:+HeapDumpOnOutOfMemoryError`가 켜져 있으면 OOM 시점에 `<sonarqube>/data/`에 heap dump가 생성된다. Eclipse MAT 또는 VisualVM으로 분석하면 어느 객체가 메모리를 차지했는지 보인다.

흔한 원인은 Issue 매칭 시 메모리에 펼쳐 놓은 데이터다. 모노리포에 100만 Issue가 있으면 매칭 알고리즘이 한꺼번에 메모리에 올린다. 이 경우 분석을 모듈 단위로 분할(별도 projectKey)하는 게 운영 답이다.



## 5. 흔한 시나리오 — Elasticsearch 헬스 RED

> ES가 정상이 아니면 검색·이슈 목록이 비어 보인다.

증상은 다음과 같다.

- `/api/system/health`의 `health`가 YELLOW 또는 RED.
- `causes`에 ES 관련 메시지.
- `es.log`에 `cluster_health_status` 변경 또는 인덱스 손상 메시지.

원인은 셋이다.

1. **vm.max_map_count 미설정**: ES 시작에 실패. `sysctl vm.max_map_count` 결과가 65530 등 작은 값이면 사고. 524288로 올린다 (01-02 §2.2).
2. **디스크 가득 참**: ES는 디스크 사용률 85%에서 자체 인덱스 쓰기를 차단. `df -h`로 확인.
3. **인덱스 손상**: 비정상 정지 또는 디스크 장애로 인덱스 일부 손상. 재인덱싱이 답.

### 5.1 인덱스 재생성

손상된 인덱스를 깨끗이 다시 만든다.

1. SonarQube 정지.
2. ES 데이터 디렉터리(`<sonarqube>/data/es8/`) 삭제.
3. SonarQube 시작.
4. 시작 중 ES가 DB로부터 인덱스를 새로 채움. 이 시간이 인덱스 크기에 비례.

이 과정 동안 SonarQube의 `/api/system/status`는 `STARTING`이며, `UP` 도달 후에도 검색 결과가 점진적으로 채워진다. 큰 인스턴스에서 30분~2시간 걸린다.



## 6. 메트릭 노출 — Prometheus 통합

> SonarQube 자체의 메트릭(분석 큐 길이, 응답 시간, JVM 상태 등)을 Prometheus로 보내려면 별도 exporter가 필요하다.

SonarQube Server는 Prometheus 호환 메트릭 엔드포인트를 자체 제공하지 않는다. 두 길이 있다.

- **JMX exporter**: SonarQube JVM에 javaagent로 붙여 JMX 메트릭을 Prometheus 형식으로 노출. JVM 상태(heap, GC, thread)를 본다.
- **써드파티 SonarQube exporter**: Web API를 폴링해 SonarQube 도메인 메트릭을 Prometheus 형식으로 변환.

SonarQube 자체의 도메인 메트릭(분석 결과, 이슈 수, Coverage 등)은 Prometheus보다 SonarQube UI에서 보는 게 자연스럽다. Prometheus에 보내야 하는 것은 인프라 메트릭(JVM, OS) 중심이다.

### 6.1 흔한 알람 조건

운영에서 자주 거는 알람 조건은 다음과 같다.

| 신호 | 조건 | 대응 |
|------|------|------|
| `/api/system/status` 응답 != UP | 1분 이상 | 즉시 호출 |
| Compute Engine 큐 길이 | 50 이상 5분 | 적체 진단 |
| CE JVM heap 사용률 | 85% 이상 5분 | OOM 임박 |
| ES 디스크 사용률 | 80% 이상 | 디스크 정리/확장 |
| 분석 실패율 | 10% 이상 5분 | 분석 환경 점검 |

이 다섯이 운영 사고의 90%를 사전 감지한다.



## 7. 정리

> 모니터링의 핵심은 시스템 엔드포인트 둘 + 로그 파일 넷 + 흔한 시나리오 셋이다.

요약한다. `/api/system/status`(인증 불필요)와 `/api/system/health`(관리자 토큰)가 자동화된 모니터링의 출발이다. 로그는 `sonar.log`/`web.log`/`ce.log`/`es.log` 4개 파일로 분리되며 사고 종류에 따라 어느 파일을 볼지 결정한다. Compute Engine 큐 적체, CE OOM, ES 헬스 RED가 가장 자주 부딪히는 시나리오다. 큐 적체는 `/api/ce/activity`로 진단하고 cancel은 마지막 수단이다. CE OOM은 `sonar.ce.javaOpts`의 heap을 늘려 해결하며 큰 모노리포는 모듈 분할이 답이다. ES RED는 `vm.max_map_count`·디스크·인덱스 손상 셋이 원인이며 인덱스 재생성으로 회복한다. Prometheus 통합은 JMX exporter 또는 써드파티로 가능하지만 SonarQube 도메인 메트릭은 자체 UI가 자연스럽다.

이 문서로 5장 본문이 끝난다. 통찰 점검은 [05-점검.핵심 질문과 답](05-점검.핵심 질문과 답.md)으로 간다.
