# 4단계 점검 — SonarQube 통합 활용 핵심 질문

---

> 다루는 문서 - `04-01.SonarScanner for Gradle 적용 패턴`, `04-02.SonarQube Web API·Webhook 통합`, `04-03.분석 실행 로그 수집`, `04-04.Jenkins 파이프라인 통합`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다.



## Q1: Gradle 프로젝트에 `sonar-project.properties`를 두지 않아도 SonarQube 분석이 가능한 이유는?

`org.sonarqube` Gradle 플러그인이 Gradle 프로젝트 메타에서 분석에 필요한 모든 속성을 자동 추출하기 때문이다. 소스 디렉터리, 테스트 디렉터리, 언어, 컴파일된 바이너리 위치, 의존성 클래스패스, 커버리지 리포트 경로가 그것이다. CLI를 쓰면 같은 정보를 `sonar-project.properties`에 손으로 명시해야 하지만, 멀티 모듈 Gradle 빌드에서는 이게 사실상 불가능하다. TPS `pipeline-api`가 이 패턴을 쓴다 — `id 'org.sonarqube' version '4.4.1.3373'` 한 줄이 모든 자동화의 출발이다.

상세: `04-01.md` §1.

### 심화

같은 모듈을 Gradle Scanner와 CLI로 각각 분석했을 때 결과 차이가 가장 크게 나타나는 영역은 어디인가? Rule이 같다고 가정할 때 어느 측면에서 분석 품질이 갈리는가?



## Q2: `pipeline-api/build.gradle`의 `// sonar gradle` 주석이 붙은 의존성은 SonarQube 플러그인이 요구하는가?

요구하지 않는다. `org.gradle:gradle-tooling-api` 의존성은 `pipeline-api` 자체가 외부 Gradle 프로젝트를 분석/실행하는 도메인 코드를 가지고 있어서 들어왔고, 그 도메인 코드가 SonarQube를 호출하는 일과 같은 종류(외부 Gradle 프로젝트를 다룬다)이므로 주석이 `// sonar gradle`이라고 표기됐다. SonarScanner for Gradle 자체는 자기 plugin 컨테이너에서 자체 의존성으로 Tooling API를 가져오므로 추가 의존성 없이 작동한다. 이 구분을 하지 못하면 다른 모듈에 같은 패턴을 복제할 때 불필요한 의존성을 끌고 가게 된다.

상세: `04-01.md` §3.

### 심화

`// sonar gradle`처럼 주석으로만 의도를 남긴 의존성이 시간이 지나면서 의미가 흐려지는 흔한 문제를, 코드 베이스 차원에서 어떻게 막을 수 있는가? 주석 외의 어떤 메커니즘이 가능한가?



## Q3: operator의 SonarQubeFeignClient가 `@FeignClient(url = "sonarqube-placeholder")`로 placeholder만 두고 호출 시 baseUrl을 동적으로 받는 이유는?

여러 SonarQube 인스턴스를 한 운영자 인스턴스에서 다뤄야 하기 때문이다. 사용자가 admin tool로 등록한 SonarQube의 주소가 사용자별 또는 환경별로 다를 수 있고, 그 주소가 호출 시점에 결정된다. 일반적인 `@FeignClient(url = "https://...")`처럼 정적 URL을 박으면 인스턴스가 하나로 고정되는데, operator는 그 가정 자체를 거부한다. URI를 첫 번째 메서드 파라미터로 받는 패턴은 멀티 인스턴스 통합을 가능하게 하는 작은 트릭이지만 효과가 크다.

상세: `04-02.md` §2.

### 심화

같은 패턴을 다른 외부 도구(예: Jenkins, GitLab)에 적용할 때 주의할 점은 무엇인가? Feign 외에 RestTemplate이나 WebClient를 쓰는 경우와 비교했을 때의 트레이드오프는?



## Q4: SonarQube의 401 응답을 일반 4xx와 분리해 `SONARQUBE_UNAUTHORIZED`로 매핑하는 게 왜 가치 있는가?

사용자에게 노출되는 에러 메시지가 구체적이 되기 때문이다. "외부 시스템 오류" 한 가지로 모든 4xx를 뭉치면 사용자는 무엇을 고쳐야 할지 모른다. 401이라면 토큰이 만료됐거나 잘못 입력됐다는 의미이고, 403이라면 토큰은 유효한데 권한이 부족하다는 의미이며, 404라면 프로젝트 키가 잘못됐다는 의미다. 각각의 원인이 다른 행동을 요구한다. operator의 `resolveErrorCode` 매핑이 이 분리를 코드로 구현했다 — switch 한 표현식이 4xx를 도메인 어휘로 바꿔 준다.

상세: `04-02.md` §2.2.

### 심화

같은 401이라도 "토큰 자체가 잘못됨"과 "토큰은 유효하지만 SonarQube가 토큰 캐시를 막 무효화함"은 사용자 행동이 다르다. 운영 시점에서 이 둘을 구분하려면 어떤 추가 신호가 필요한가?



## Q5: operator의 webhook 자동 등록이 `toolCntnId == "admin"`인 경우에만 진입하는 이유는?

SonarQube의 글로벌 webhook 생성 권한은 관리자만 가지므로, 일반 사용자 토큰으로 등록된 도구에서는 어차피 401이 떨어진다. 그러나 더 중요한 이유는 **중복 등록 방지**다. 같은 SonarQube 서버에 여러 사용자가 도구를 등록하면 webhook 생성도 사용자 수만큼 시도된다. 등록 자체는 멱등하게 짜였지만(name+url 검사), 시도 횟수가 늘어날수록 SonarQube에 불필요한 부하가 가고 로그가 어지러워진다. admin tool 단일 진입은 시스템 차원의 게이트키퍼다.

상세: `04-02.md` §3.2, §3.3.

### 심화

admin tool이 단 하나라는 가정이 깨지는 시나리오(예: 두 admin이 동시에 등록 트리거)에서, 멱등성 검사만으로 충분한가? race condition을 막기 위한 추가 메커니즘이 필요한가?



## Q6: ppln-logging-api가 SonarQube에 직접 의존하지 않는 설계가 왜 좋은가?

세 가지 측면에서 모듈이 견고해진다. 첫째, SonarQube가 다운돼 분석 결과 Webhook이 안 와도 ppln-logging-api는 다른 환경의 로그 수집을 멀쩡히 계속한다. 둘째, SonarQube 응답 형식이 바뀌어도 ppln-logging-api는 영향을 받지 않는다 — 변환 책임이 operator의 webhook 핸들러에 격리돼 있다. 셋째, SonarQube를 다른 정적 분석 도구로 교체할 경우 ppln-logging-api는 환경 코드 상수와 분기만 손대면 된다. 외부 도구와의 결합도를 환경 코드 상수 한 줄로 좁힌 결과다.

상세: `04-03.md` §5.

### 심화

같은 분리 원칙을 적용할 수 있는 다른 외부 도구 통합은 무엇이며, 그때 "환경 코드"에 해당하는 추상은 무엇으로 표현되는가?



## Q7: SonarQube 단건 실행이 환경 코드 ETC로 들어오는 워크어라운드는 어떤 종류의 부채인가?

도메인 분류 체계가 운영 중에 변형된 흔적이다. 처음에는 "환경 코드 SQA가 SonarQube"라는 단순 매핑이었는데, 단건 실행 기능이 추가되면서 환경 코드 분류와 충돌했다. ETC 분기에 SonarQube 단건 실행을 함께 흡수한 형태가 작동은 하지만 코드 베이스가 "왜 ETC가 두 가지를 의미하는가"를 매번 설명해야 한다. 정리는 신규 환경 코드 도입 또는 사용 분기로의 재분류 둘 중 하나이며, 후자는 데이터 마이그레이션 비용이 더 크다. 이런 부채는 코드 주석으로 흔적을 남기고 별도 작업으로 미루는 게 현실이다.

상세: `04-03.md` §4, §4.1.

### 심화

이런 부채가 누적되는 코드 베이스에서 "지금 청산할지, 미룰지"를 판단하는 객관적 기준은 무엇인가? 청산 비용과 미루기 비용을 어떻게 비교하는가?



## Q8: Jenkins 동기 모드(`waitForQualityGate`)와 비동기 모드(Webhook 외부 처리)의 핵심 차이는?

빌드 결과가 게이트 결과를 반영하느냐가 차이다. 동기 모드는 게이트 FAILED가 빌드 실패로 직결되므로 "이 빌드는 게이트 통과한 빌드"라는 명제가 강해진다. 비동기 모드는 Scanner 종료 시점에 빌드가 끝나므로 빌드 시간이 짧고 일정해지지만, 게이트 결과가 빌드와 분리된다. TPS는 두 모드를 섞어 쓴다 — 자체 모듈 분석은 동기로 게이트를 만들 수 있고, 사용자 파이프라인의 SonarQube 실행은 Webhook → executor → ppln-logging-api로 흘러가는 비동기 경로다. 두 모드의 선택은 "게이트 실패가 무엇을 차단해야 하는가"라는 정책 결정이다.

상세: `04-04.md` §1, §7, §8.
