# 점검 — CI·CD 핵심 질문

---

> 이 점검 문서는 05 CI·CD 그룹을 다 읽은 뒤 스스로를 시험하기 위한 자가 점검입니다. 먼저 §면접 질문만 보고 답을 떠올린 뒤, §정답 절에서 같은 번호로 대조하세요.



## 진입 — 왜 이 점검이 필요한가

> CI·CD 개념은 하나씩 보면 그럴듯하지만 실제 설계에서는 맞물려 결정을 만듭니다. 점검은 흩어진 개념을 한 시나리오 위에 다시 세워 빈틈을 드러냅니다.

이 그룹의 본문을 다 읽었다면 아래 질문에 막힘없이 답할 수 있어야 합니다. 답이 막히는 질문은 §관련 문서의 해당 본문으로 돌아가 다시 확인합니다.



## 면접 질문

> 답을 떠올린 뒤 §정답 절에서 같은 번호로 대조하세요. 각 질문 뒤의 *심화*까지 답할 수 있으면 충분합니다.

1. CI 파이프라인의 스테이지 순서(Clone→Test→SCA→Quality Gate→Build→Publish)에서 SCA·Quality Gate가 Build 앞에 오는 이유는 무엇이며, Artifactory를 Docker 레지스트리로 쓸 때 package type은 무엇입니까? *(심화: Kaniko 인증용 docker-registry Secret을 jenkins namespace에 두는 이유는?)*
2. 한 Pod에 node·sonar-scanner-cli·kaniko 세 컨테이너를 두고 stage마다 `container()`로 전환할 때, Kaniko 컨테이너에 `sleep 99d`를 두는 이유는 무엇입니까? Multibranch Pipeline의 주 기능과 Blue Ocean의 역할은 각각 무엇입니까? *(심화: SonarQube가 측정하지 않는 metric은?)*
3. Continuous Deployment와 Continuous Delivery의 핵심 차이는 무엇이며, GitOps에서 "단일 진실"과 "self-healing"은 각각 무엇을 뜻합니까? *(심화: 이 저장소의 JCasC GitOps와 챕터10의 앱 배포 GitOps는 무엇이 다릅니까?)*
4. GitOps CI/CD에서 Jenkins와 Argo CD는 각각 무엇을 맡으며, 누가 Git을 감시합니까? Argo CD가 배포를 트리거하려고 GitHub에서 감시하는 파일은 무엇입니까? *(심화: staging 배포 후 production 승급 전에 무엇을 확인합니까?)*
5. CD Jenkinsfile에서 `disableConcurrentBuilds()`를 main/master 브랜치에만 두는 이유는 무엇이며, 애플리케이션의 sync 상태와 health 상태는 왜 같지 않습니까? Argo CD의 health 상태를 프로그래매틱으로 가져오는 방법은 무엇입니까? *(심화: git·alpine 두 컨테이너를 CD용으로 추가하면서 `sleep 99d`를 두는 이유는?)*
6. SonarQube와 JFrog Xray는 둘 다 보안 스캔 도구인데 검사 대상이 어떻게 다르며, CI 파이프라인에서 `xrayScan(failBuild: true)`는 무엇을 보장합니까? *(심화: GitHub Copilot과 ChatGPT 같은 LLM의 1차 목적이 다른 이유와, 오래된 빌드를 `logRotator`로 폐기하는 이유는?)*



## 정답

> 위 질문을 스스로 설명해 본 뒤에 펼치세요.

### 정답 1 — CI 스테이지 순서와 Docker 레지스트리

SCA(정적 분석)와 Quality Gate가 Build 앞에 오는 이유는, 품질 기준을 통과하지 못한 코드가 Docker 이미지로 패키징되는 것을 막기 위해서입니다. 이미지를 먼저 빌드한 뒤 게이트를 두면, 차단되어야 할 코드가 이미 레지스트리에 올라갈 수 있습니다. 검색대(Gate)는 탑승구(Build) 앞에 있어야 의미가 있습니다. Artifactory를 Docker 레지스트리로 쓸 때 package type은 **Docker**입니다. Generic은 파일을 그냥 올리는 범용 저장소라 Docker 클라이언트의 push/pull 프로토콜을 지원하지 않습니다. *심화*: K8s Secret은 namespace 범위 리소스라 다른 namespace의 Pod가 접근할 수 없습니다. Kaniko Pod가 jenkins namespace에서 돌므로 `artifactory-credentials` Secret도 jenkins namespace에 두어야 마운트할 수 있습니다.

### 정답 2 — 멀티컨테이너 Pod·Multibranch·Blue Ocean

Kaniko 컨테이너에 `sleep 99d`(또는 `cat`)를 두는 이유는, Kaniko 이미지를 기본 entrypoint로 실행하면 즉시 종료되어 stage에서 재사용할 수 없기 때문입니다. 컨테이너를 살려 두어야 `container('kaniko')` step이 그 안에서 명령을 exec 방식으로 실행할 수 있습니다. Multibranch Pipeline의 주 기능은 저장소의 브랜치마다 Jenkinsfile을 스캔해 브랜치별 파이프라인을 자동으로 만들고 빌드·테스트하는 것입니다(병렬 실행이나 코드 머지가 아닙니다). Blue Ocean은 파이프라인을 stage 단위로 시각화하는 사용자 친화 UI로, 머신러닝·보안·속도 최적화 도구가 아닙니다. *심화*: SonarQube는 정적 분석 도구라 메모리 사용량 같은 런타임 metric은 측정하지 않습니다(코드 커버리지·복잡도·중복은 측정).

### 정답 3 — CD vs Delivery·GitOps

Continuous Delivery와 Continuous Deployment의 차이는 프로덕션 배포 직전 수동 승인 게이트의 유무입니다. Delivery는 staging까지 자동이지만 prod 배포는 사람이 승인 후 실행하고, Deployment는 검증 통과 즉시 prod까지 자동 배포합니다. 둘 다 CI의 확장이며 extensive testing을 포함합니다. GitOps의 "단일 진실"은 Git 저장소를 시스템의 desired state가 담긴 유일한 출처로 삼는다는 뜻이고, "self-healing"은 실제 클러스터 상태가 Git과 어긋나면 자동으로 Git 상태로 복귀시키는 성질입니다. *심화*: 이 저장소의 JCasC GitOps(02_security/02-02)는 *Jenkins 자신의 설정*을 Git으로 관리해 reload하는 것이고, 챕터10의 앱 배포 GitOps는 *배포되는 애플리케이션*을 Argo CD가 Helm values 변경을 감지해 K8s에 동기화하는 것입니다. 대상이 "Jenkins 설정"이냐 "배포되는 앱"이냐가 다릅니다.

### 정답 4 — Jenkins·Argo CD 역할분담

Jenkins는 CI 엔진으로 빌드·테스트·이미지 push·Helm values 업데이트를 맡고, Argo CD는 GitOps 배포 도구로 Helm chart 저장소를 감시해 K8s에 desired state를 동기화합니다. Git을 감시하는 쪽은 Argo CD입니다(Jenkins가 아닙니다). Argo CD가 배포를 트리거하려고 감시하는 파일은 `values.yaml`입니다. Chart.yaml(메타데이터)이나 Jenkinsfile(CI 정의)·Dockerfile(이미지 빌드)이 아니라, 환경별 설정과 이미지 태그가 담긴 values 파일이 바뀔 때 배포가 일어납니다. *심화*: staging 배포 후 production 승급 전에 Jenkins가 Argo CD API로 sync·health 상태를 확인하고, k6로 성능 테스트를 돌려 임계치를 통과하는지 검증합니다.

### 정답 5 — disableConcurrentBuilds·sync vs health·Argo CD 상태 조회

`disableConcurrentBuilds()`는 같은 파이프라인의 동시 실행을 막아 이전 빌드가 끝날 때까지 새 빌드를 큐에 대기시킵니다. main/master에만 두는 이유는 이 브랜치가 staging·production 같은 공유 환경에 실제 배포를 수행하기 때문입니다. 동시에 배포가 겹치면 배포 충돌·DB 마이그레이션 경합·부하 테스트 자원 경합으로 결과를 신뢰할 수 없고, 순차 적용이 깨지며 트러블슈팅도 어려워집니다. sync 상태와 health 상태는 같지 않습니다(책 Q&A 정답: False). sync는 Git의 desired state와 K8s 실제 state가 일치하는지를 보고, health는 앱이 올바로 동작하는지를 봅니다. 둘은 독립이라 sync인데 unhealthy(예: Pod 하나가 죽음)일 수 있고, out-of-sync인데 healthy(수동 변경으로 desired와 달라짐)일 수도 있습니다. 그래서 승급 게이트는 health를 기준으로 둡니다. Argo CD health를 프로그래매틱으로 가져오는 방법은 CLI(`argocd app status <app>`)와 API(`/api/v1/applications/<app>`의 `status.health.status` 필드)입니다(책 Q&A 정답: A·B). kubectl은 Pod 상태만 보여 앱 리소스 전체를 못 보고, UI는 수동이라 필터·자동화가 안 됩니다. *심화*: git·alpine 컨테이너의 `sleep 99d`·`sleep 86400` 무한 루프는 컨테이너를 오래 살려 두어 파이프라인이 Git 명령이나 shell 명령을 호출할 때 컨테이너가 조기 종료되지 않게 하는 장치이지, 파이프라인이 99일 동안 도는 것을 뜻하지 않습니다.

### 정답 6 — SonarQube vs Xray·xrayScan 보안 게이트

SonarQube와 JFrog Xray는 둘 다 보안 스캔 도구이나 검사 대상이 다릅니다. SonarQube는 소스 코드를 정적분석해 코드 품질과 코드 자체의 취약점을 봅니다(06-05). Xray는 빌드 결과물인 아티팩트·Docker 이미지·의존성 패키지에서 알려진 취약점과 라이선스 컴플라이언스 문제를 찾습니다(책 Q&A 정답: 코드베이스와 의존성의 알려진 보안 취약점 스캔). 같은 SCA 계열로 불려도 한쪽은 소스 코드, 다른 쪽은 패키지·이미지를 대상으로 둡니다. `xrayScan(failBuild: true)`는 스캔에서 취약점이 발견되면 파이프라인을 실패시켜, 취약한 이미지가 다음 단계로 진행하거나 배포되는 것을 차단하는 보안 게이트를 보장합니다(책 Q&A 정답: Artifactory 플러그인으로 스캔 실패 시 파이프라인 fail 설정 가능, True). `failBuild: false`로 두면 취약점을 보고만 하고 파이프라인은 계속 진행합니다. *심화*: GitHub Copilot은 IDE에 통합된 코드 완성·생성 도구이고 ChatGPT 같은 LLM은 범용 대화·텍스트 생성 도구라 1차 목적이 다릅니다(책 Q&A 정답: False). 다만 Copilot이 생성한 파이프라인 코드도 반드시 리뷰·테스트해야 한다는 점은 06-09의 AI 코드 검증 의무와 같습니다. 오래된 빌드를 `buildDiscarder(logRotator(numToKeepStr: '10'))`로 폐기하는 이유는 디스크 공간 절약과 Jenkins 성능 유지입니다(책 Q&A 정답: D, 디스크 절약·성능·logRotator 모두).
