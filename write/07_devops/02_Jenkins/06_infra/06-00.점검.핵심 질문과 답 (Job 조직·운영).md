# 점검 — Job 조직·운영 핵심 질문

---

> 이 점검 문서는 06 Job 조직·운영 그룹을 다 읽은 뒤 스스로를 시험하기 위한 자가 점검입니다. 먼저 §면접 질문만 보고 답을 떠올린 뒤, §정답 절에서 같은 번호로 대조하세요.



## 진입 — 왜 이 점검이 필요한가

> Folder·SSH 배포·MinIO·인라인 Job은 책 정석 밖의 실무 패턴이라, 개념을 따로 외우면 실제 운영 파이프라인을 읽을 때 판단이 어긋납니다.

이 그룹은 실제 운영 Jenkins의 config.xml을 분석해 메운 묶음입니다. Job을 폴더로 조직하고, K8s가 아닌 VM에 SSH로 배포하며, 산출물을 MinIO로 주고받고, 파이프라인을 Job 설정에 인라인으로 박는 패턴이 한 흐름으로 이어집니다. 답이 막히는 질문은 §관련 문서의 본문으로 돌아가 확인합니다.



## 면접 질문

> 답을 떠올린 뒤 §정답 절에서 같은 번호로 대조하세요.

1. 기본 Jenkins에서 두 팀이 똑같이 `deploy`라는 Job을 만들 수 없는 이유는? Folder는 full name으로 이걸 어떻게 풉니까?
2. 폴더가 단순 디렉토리와 다른 점을 클래스 구조로 설명하면? 폴더만 설치하면 RBAC가 됩니까?
3. Multibranch Pipeline과 Folder 플러그인의 관계는?
4. K8s에 Argo CD 배포 흐름이 있는데도 SSH 배포가 필요한 이유는? `sshPut`과 `sshCommand`의 역할 차이는?
5. SSH 배포에서 비밀번호 인증과 키 인증은 코드에서 어떻게 갈립니까? `parallel` 브랜치 키를 일련번호로 두는 이유는?
6. 컨테이너 이미지는 Harbor에 두는데 VM 배포용 tar는 왜 MinIO에 둡니까? `mc cp --version-id`가 재현 가능한 배포에 중요한 이유는?
7. 같은 파이프라인 코드가 인라인(`CpsFlowDefinition`)과 SCM(`CpsScmFlowDefinition`)으로 갈리는 지점과 운영 차이는? `<sandbox>true`는 무엇을 보장합니까?



## 정답

> 위 질문을 스스로 설명해 본 뒤에 펼치세요.

### 정답 1 — Folder의 namespace 격리

평면 구조에서 Job 이름은 전역 유일해야 하기 때문입니다. Folder는 각 폴더를 독립 이름공간으로 만들어, 식별자를 짧은 이름이 아니라 폴더 경로를 포함한 full name(`TRB305/STG/deploy`)으로 바꿉니다. 짧은 이름이 같아도 full name이 다르면 충돌하지 않습니다.

### 정답 2 — AbstractFolder 구조와 RBAC

`AbstractFolder`는 `TopLevelItem`(스스로도 하나의 항목)이면서 `ItemGroup`(자식을 담는 컨테이너)을 동시에 구현합니다. 컨테이너인 동시에 내용물이라 폴더 안에 폴더를 넣는 재귀 중첩이 되고, 이 구조가 full name 계층을 만듭니다. 폴더만 깔아서는 RBAC가 되지 않습니다 — Folder는 권한이 걸릴 경계(full name 경로)만 제공하고, 실제 권한 판정은 Role-based Authorization Strategy 같은 별도 플러그인이 합니다. 반면 folder-scoped credential은 Folder 플러그인이 직접 여는 기능입니다.

### 정답 3 — Multibranch와 Folder

Multibranch Pipeline은 `AbstractFolder → ComputedFolder → MultiBranchProject` 상속 체인 위에 있습니다. 브랜치별 파이프라인은 사실 폴더 안에 브랜치마다 자동 생성된 자식 Job입니다. Folder는 부가 기능이 아니라 현대 Job 모델의 토대입니다.

### 정답 4 — SSH 배포가 필요한 이유와 두 step

배포 대상이 전부 K8s는 아니기 때문입니다. 레거시·온프레미스·망분리 VM처럼 오케스트레이터가 없는 곳에는 "파일을 올리고 명령을 실행한다"는 SSH 방식이 여전히 필요합니다. `sshPut`은 로컬 산출물을 원격으로 올리고, `sshCommand`는 원격에서 명령을 실행해 출력을 받습니다. 배포는 보통 mkdir(command) → 업로드(put) → 후속 명령(command) 순서로 둘을 엮습니다.

### 정답 5 — 인증 분기와 parallel 키

`withCredentials`의 바인딩 타입이 다릅니다. 비밀번호는 `usernamePassword`로 user·password를, 키는 `sshUserPrivateKey`로 키 파일·user·passphrase를 꺼냅니다. 꺼낸 값은 반드시 `withCredentials` 블록 안에서 remote 맵에 채우고 그 안에서 배포를 끝냅니다. `parallel` 브랜치 키를 일련번호(`dply-0`, `dply-1`)로 두는 이유는, 같은 식별자가 두 번 나오면 맵 키가 충돌해 한쪽 배포가 사라지기 때문입니다. 표시용 이름은 별도 필드에 둡니다.

### 정답 6 — MinIO와 버전 핀

이미지가 아닌 일반 파일은 이미지 레지스트리에 넣을 수 없습니다. MinIO는 S3 호환 오브젝트 스토리지라 임의 파일을 버킷에 보관했다가 배포 단계에서 꺼낼 수 있습니다. `--version-id`가 중요한 이유는, 버저닝 버킷은 같은 키에 버전이 쌓여 버전을 지정하지 않으면 항상 최신을 받기 때문입니다. 빌드와 배포 사이에 다른 빌드가 끼면 엉뚱한 산출물이 옵니다. 빌드가 만든 그 버전을 핀하면 같은 입력에 같은 산출물이 보장됩니다.

### 정답 7 — 인라인 vs SCM, sandbox

`<definition>`의 class가 가릅니다. `CpsFlowDefinition`이면 코드가 config.xml의 `<script>`에 인라인으로, `CpsScmFlowDefinition`이면 코드는 Git의 Jenkinsfile에 있고 config는 위치만 가리킵니다. 운영 차이는 SCM이 Git 히스토리로 버전·리뷰가 되지만 저장소가 먼저 있어야 하고, 인라인은 버전 관리가 약하지만 config.xml을 API로 찍어 Job을 양산하기 쉽다는 점입니다. `<sandbox>true`는 인라인 Groovy 스크립트를 허용된 API만 쓰는 제한 환경에서 실행해, 위험한 호출은 관리자 승인을 거치게 함으로써 임의 코드 실행 위험을 막습니다.
