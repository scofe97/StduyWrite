# 인라인 Pipeline Script형 Job — config.xml 형상·CpsFlowDefinition·파라미터화

---

> 이 문서를 읽고 나면 파이프라인을 SCM Jenkinsfile로 두는 방식과 Job 안에 인라인으로 박는 방식을 **구분하고**, Job이 저장되는 `config.xml`의 뼈대를 **읽으며**, `CpsFlowDefinition`과 `sandbox`가 무엇인지 **설명**하고, 파라미터화 빌드를 `params.*`로 어떻게 받는지 **확인**할 수 있습니다.



## 사전 지식

[`06-11.첫 CI Jenkinsfile 구현`](05-02.첫%20CI%20Jenkinsfile%20구현%20%E2%80%94%20완성%20코드%C2%B7Multibranch%C2%B7Blue%20Ocean.md)에서 Jenkinsfile을 SCM에 두는 Multibranch 방식을 본 것이 전제입니다. [`06-17.Folder 플러그인`](06-01.Folder%20플러그인%20%E2%80%94%20namespace%20격리%C2%B7full%20name%20식별%C2%B7폴더%20스코핑.md)에서 Job·폴더가 `config.xml`로 저장된다는 점을 봤다면 이 편이 그 파일의 속을 여는 후속입니다.



## 진입 — 같은 파이프라인, 두 가지 사는 곳

> Declarative Pipeline 코드는 같아도, 그 코드가 어디 저장되느냐는 두 갈래입니다. SCM의 Jenkinsfile이거나, Job 설정 안에 직접 박힌 인라인 스크립트이거나.

[`06-11`](05-02.첫%20CI%20Jenkinsfile%20구현%20%E2%80%94%20완성%20코드%C2%B7Multibranch%C2%B7Blue%20Ocean.md)에서 만든 파이프라인은 Git 저장소의 `Jenkinsfile`에 있었고, Multibranch가 그걸 읽어 실행했습니다. 그런데 현장에는 다른 방식도 흔합니다. 파이프라인 코드를 Git에 두지 않고 **Job 설정 자체에 직접 써 넣는** 방식입니다. Jenkins UI의 Pipeline 설정에서 "Pipeline script"를 고르고 코드를 붙여넣는 그 형태입니다.

이 둘은 코드 내용은 같아도 운영 성질이 다릅니다. 그리고 인라인 방식을 이해하려면 Job이 디스크에 어떻게 저장되는지, 즉 `config.xml`의 형상을 읽을 줄 알아야 합니다. [`04_api`](../04_api/04-02.파이프라인%20CRUD%20모델과%20TPS%20패턴%20%282.222%2B%29.md)에서 "파이프라인 CRUD는 config.xml 전체 교체"라고 했던 그 파일의 속을 이 편에서 엽니다.



## 1. config.xml의 뼈대 — flow-definition

> Pipeline Job 하나는 `<flow-definition>` 루트를 가진 XML 한 장으로 저장됩니다. 이 뼈대를 읽으면 Job의 모든 설정이 보입니다.

Pipeline Job의 `config.xml`은 이런 골격입니다.

```xml
<flow-definition plugin="workflow-job@...">
  <actions>
    <...DeclarativeJobAction .../>
    <...DeclarativeJobPropertyTrackerAction>...</...>
  </actions>
  <description>작업 이름 : 빌드Job-01 ...</description>
  <properties>
    <jenkins.model.BuildDiscarderProperty>...</jenkins.model.BuildDiscarderProperty>
    <hudson.model.ParametersDefinitionProperty>...</hudson.model.ParametersDefinitionProperty>
  </properties>
  <definition class="org.jenkinsci.plugins.workflow.cps.CpsFlowDefinition" plugin="workflow-cps@...">
    <script>pipeline { ... }</script>
    <sandbox>true</sandbox>
  </definition>
  <triggers/>
  <disabled>false</disabled>
</flow-definition>
```

각 부분의 역할은 이렇습니다.

| 요소 | 역할 |
|------|------|
| `<flow-definition>` | Pipeline Job의 루트. plugin 속성에 workflow-job 버전이 박힘 |
| `<actions>` | Declarative 모델이 추적용으로 붙이는 메타데이터(아래 §3) |
| `<description>` | Job 설명. UI의 설명란 그대로 |
| `<properties>` | Job 단위 설정 — 빌드 보관 정책, 파라미터 정의 등 |
| `<definition>` | 파이프라인 코드가 어디서 오는지. 인라인이냐 SCM이냐가 여기서 갈림 |
| `<triggers>` | 트리거(주기 실행·webhook 등) |
| `<disabled>` | 비활성 여부 |

`BuildDiscarderProperty`는 빌드 기록 보관 정책입니다. 예컨대 `numToKeep`을 20으로 두면 최근 20개 빌드만 남기고 오래된 것을 지워 디스크를 아낍니다. 이 형상을 읽을 수 있으면 UI 없이 config.xml만 보고도 Job이 무엇을 하는지 파악할 수 있습니다.



## 2. 인라인 vs SCM — definition이 가른다

> 파이프라인 코드의 출처는 `<definition>`의 class가 결정합니다. `CpsFlowDefinition`이면 인라인, `CpsScmFlowDefinition`이면 SCM Jenkinsfile입니다.

같은 위치(`<definition>`)에 들어가는 class가 두 가지입니다.

```xml
<!-- (A) 인라인 — 코드가 config.xml 안에 직접 -->
<definition class="org.jenkinsci.plugins.workflow.cps.CpsFlowDefinition" ...>
    <script>pipeline { ... }</script>
    <sandbox>true</sandbox>
</definition>

<!-- (B) SCM — 코드는 Git의 Jenkinsfile, config는 위치만 가리킴 -->
<definition class="org.jenkinsci.plugins.workflow.cps.CpsScmFlowDefinition" ...>
    <scm>... 저장소 URL, 브랜치 ...</scm>
    <scriptPath>Jenkinsfile</scriptPath>
</definition>
```

(A) 인라인(`CpsFlowDefinition`)은 파이프라인 전문이 `<script>` 안에 통째로 들어갑니다. (B) SCM(`CpsScmFlowDefinition`)은 코드 대신 저장소 위치와 `Jenkinsfile` 경로만 둡니다. 둘의 운영 차이가 큽니다.

| 축 | 인라인 (CpsFlowDefinition) | SCM (CpsScmFlowDefinition) |
|----|---------------------------|----------------------------|
| 코드 위치 | Job 설정(config.xml) 안 | Git 저장소의 Jenkinsfile |
| 버전 관리 | Jenkins 설정 백업에 의존 | Git 히스토리로 추적 |
| 코드 리뷰 | 어려움(UI/XML 편집) | PR로 가능 |
| 자동 생성 | API로 config.xml 찍어 양산하기 쉬움 | 저장소가 먼저 있어야 함 |

[`06-11`](05-02.첫%20CI%20Jenkinsfile%20구현%20%E2%80%94%20완성%20코드%C2%B7Multibranch%C2%B7Blue%20Ocean.md)의 Multibranch는 (B) SCM 방식입니다. 반면 외부 시스템이 Jenkins Job을 **프로그램으로 대량 생성**할 때는 (A) 인라인이 유리합니다. config.xml에 스크립트를 끼워 넣어 API로 찍어내면, 저장소를 만들 필요 없이 Job이 생기기 때문입니다. 같은 파이프라인이라도 "사람이 Git으로 관리"하면 SCM, "시스템이 양산"하면 인라인이 자연스럽습니다.



## 3. sandbox와 DeclarativeJobPropertyTracker

> `<sandbox>true`는 Groovy 스크립트를 제한된 환경에서 돌린다는 뜻이고, `<actions>`의 추적 액션은 Declarative 모델이 관리용으로 붙이는 메타데이터입니다.

`<sandbox>true</sandbox>`는 Script Security의 핵심입니다. 인라인 스크립트는 Jenkins 안에서 임의 Groovy를 실행하는 셈이라, 그대로 두면 시스템 파일 접근 같은 위험한 호출까지 가능합니다. sandbox는 이 스크립트를 허용된 API만 쓸 수 있는 우리에 가둡니다. 위험한 호출이 있으면 관리자 승인을 거치게 합니다. 인라인 Job이 `sandbox=true`인 건 이 안전장치가 켜졌다는 표시입니다.

`<actions>` 안의 `DeclarativeJobAction`·`DeclarativeJobPropertyTrackerAction`은 사람이 건드리는 설정이 아닙니다. Declarative Pipeline 모델이 "이 Job이 선언한 properties·triggers·parameters가 무엇이었는지" 추적하려고 자동으로 붙이는 메타데이터입니다. 파이프라인의 `options`·`triggers` 블록과 UI 설정이 어긋나지 않게 관리하는 용도라, 읽을 때는 "Declarative가 남긴 흔적" 정도로 이해하면 됩니다.



## 4. 파라미터화 빌드 — params로 받는다

> 같은 Job을 입력만 바꿔 재사용하려면 파라미터를 정의하고, 파이프라인에서 `params.*`로 받습니다.

Job을 매번 복제하지 않고 입력만 바꿔 돌리려면 파라미터를 씁니다. config.xml에서는 `<properties>` 안에 정의됩니다.

```xml
<hudson.model.ParametersDefinitionProperty>
  <parameterDefinitions>
    <hudson.model.StringParameterDefinition>
      <name>GIT_REF_TYPE</name>
      <description>BRANCH | TAG | COMMIT_HASH</description>
      <defaultValue>BRANCH</defaultValue>
    </hudson.model.StringParameterDefinition>
    <hudson.model.StringParameterDefinition>
      <name>GIT_REF_VALUE</name>
      <defaultValue>dev</defaultValue>
    </hudson.model.StringParameterDefinition>
  </parameterDefinitions>
</hudson.model.ParametersDefinitionProperty>
```

파이프라인 코드에서는 `params.이름`으로 그 값을 읽습니다.

```groovy
environment {
    MINIO_BUCKET = "${params.TPS_MINIO_BUCKET}"   // 파라미터를 환경변수로
}
stages {
    stage('Checkout') {
        steps { script {
            def refVal = (params.GIT_REF_VALUE ?: '').toString().trim()
            if (!refVal) { error('GIT_REF_VALUE is required') }   // 필수값 검증
        } }
    }
}
```

핵심 습관 두 가지가 보입니다. 첫째, 파라미터를 그대로 믿지 않고 `?:`로 기본값을 주고 `trim()`으로 다듬은 뒤 비어 있으면 `error()`로 빌드를 멈춥니다. 빈 입력이 조용히 통과해 엉뚱한 곳에 배포되는 사고를 막습니다. 둘째, 이렇게 받은 파라미터(레지스트리 호스트·크레덴셜 ID·버전 등)를 외부 시스템이 실행 시점에 주입하면, **같은 인라인 Job 하나를 여러 환경에 재사용**할 수 있습니다. 파라미터화가 Job 양산과 짝을 이루는 이유입니다.



## 5. 비유 — 붙박이 레시피와 요리책 링크

> 인라인은 메뉴판에 레시피 전문을 인쇄해 둔 식당, SCM은 "레시피는 요리책 12쪽 참조"라 적어 둔 식당입니다.

인라인 Job(`CpsFlowDefinition`)은 메뉴판마다 레시피 전문을 인쇄한 식당입니다. 메뉴판만 보면 바로 만들 수 있지만, 레시피를 고치려면 모든 메뉴판을 다시 인쇄해야 합니다. SCM Job(`CpsScmFlowDefinition`)은 "레시피는 공용 요리책 12쪽"이라고만 적어 둔 식당입니다. 요리책 한 권만 고치면 모든 지점에 반영되고, 누가 언제 무엇을 바꿨는지(Git 히스토리) 남습니다.

이 비유는 "그럼 무조건 요리책(SCM)이 낫다"는 데서 깨집니다. 식당을 기계로 하루에 수백 개 찍어내야 한다면, 요리책을 먼저 비치하는 절차가 오히려 병목입니다. 그때는 메뉴판에 레시피를 박아 통째로 복제하는 인라인이 빠릅니다. 사람이 관리하면 SCM, 시스템이 양산하면 인라인 — 상황이 방식을 정합니다.



## 6. 점검 — 면접 대비

> 이 편을 다 읽었으면 다음 질문에 답할 수 있어야 합니다.

1. **같은 파이프라인 코드인데 인라인과 SCM 방식이 갈리는 지점은?**
   `<definition>`의 class입니다. `CpsFlowDefinition`이면 코드가 config.xml의 `<script>`에 인라인으로, `CpsScmFlowDefinition`이면 코드는 Git의 Jenkinsfile에 있고 config는 위치만 가리킵니다.

2. **인라인과 SCM은 운영상 무엇이 다릅니까?**
   SCM은 Git 히스토리로 버전·리뷰가 되지만 저장소가 먼저 있어야 합니다. 인라인은 버전 관리가 약하지만 config.xml을 API로 찍어 Job을 양산하기 쉽습니다. 사람 관리=SCM, 시스템 양산=인라인.

3. **`<sandbox>true`는 무엇을 보장합니까?**
   인라인 Groovy 스크립트를 허용된 API만 쓰는 제한 환경에서 실행합니다. 위험한 호출은 관리자 승인을 거치게 해, 임의 코드 실행 위험을 막습니다.

4. **`<actions>`의 DeclarativeJobPropertyTrackerAction은 사람이 편집하는 설정입니까?**
   아닙니다. Declarative 모델이 선언된 properties·triggers·parameters를 추적하려고 자동으로 붙이는 메타데이터입니다.

5. **파라미터를 받을 때 `error()`로 검증하는 이유는?**
   빈 입력이 조용히 통과하면 엉뚱한 브랜치·대상에 배포될 수 있습니다. `?:` 기본값과 `trim()` 후 비어 있으면 `error()`로 빌드를 멈춰 사고를 막습니다.
