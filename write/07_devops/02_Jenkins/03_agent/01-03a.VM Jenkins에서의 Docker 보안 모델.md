# VM Jenkins에서의 Docker 보안 모델
---
> Docker 자체가 위험한 것이 아니라, Jenkins 작업자가 Docker daemon에 접근할 수 있으면 그 작업자가 사실상 VM root에 가까운 권한을 가진다는 점이 핵심이다. 위험의 모양과 최소 방어선을 정리한다.



## 1. 흔한 구조 — VM 한 대에 Jenkins와 Docker

> VM에 Jenkins와 Docker가 같이 설치된 구조는 지금도 흔하다. 구조 자체는 단순하지만, Jenkins가 Docker daemon을 조작할 수 있다는 사실이 모든 위험의 출발점이다.

VM 기반 Jenkins에서 Docker를 빌드 도구로 쓰는 구성은 아래와 같은 모양으로 자주 등장한다. Jenkins가 호스트의 Docker CLI와 daemon을 그대로 사용해 이미지를 만든다.

```text
VM
├── Jenkins
│   └── pipeline 실행
├── Docker CLI
└── Docker daemon
```

Jenkinsfile 안에서는 다음과 같이 평범하게 호출한다.

```bash
docker build -t my-app .
docker run my-app
docker push harbor.example.com/my-app:1.0.0
```

이 자체는 흔한 방식이고, VM 기반 Jenkins에서는 지금도 자주 사용한다. 다만 이 구성에서는 Jenkins 사용자가 Docker daemon을 조작할 수 있다는 사실 한 가지가 보안 관점에서 결정적이다.



## 2. 왜 위험한가 — Docker daemon은 root에 가까운 권한이다

> Docker daemon은 보통 root 권한으로 실행된다. Jenkins가 `docker` 명령을 쓸 수 있다는 말은 곧 호스트 파일시스템 전체에 닿을 수 있다는 뜻이다.

Jenkins 사용자가 `docker` 명령을 실행할 수 있다는 것은 다음과 같은 일이 가능하다는 의미다.

```bash
docker run -v /:/host -it ubuntu bash
```

이 한 줄이면 컨테이너 내부에서 VM의 루트 파일시스템이 그대로 보인다.

```text
컨테이너 내부
   ↓
/host 로 VM 전체 파일시스템 접근
```

위험이 실제로 발현되는 경로는 크게 셋이다.

- 악의적인 사용자가 Jenkins Job 또는 Jenkinsfile을 수정할 수 있는 경우
- Jenkins에서 외부 PR을 그대로 빌드하는 경우
- 빌드 스크립트가 의존성 오염이나 사회 공학으로 변조된 경우

세 경로 모두 한 가지 결론으로 수렴한다.

```text
Jenkins가 docker 그룹에 있다
≈ Jenkins가 VM root에 가까운 힘을 가진다
```



## 3. VM과 Kubernetes의 위험 비교

> 같은 Docker 사용이라도 VM과 K8s는 피해 범위가 다르다. VM은 위험을 한 머신 안에 가두기 쉽고, K8s는 노드 단위로 번지기 쉽다.

VM Jenkins는 보통 하나의 빌드 머신 안에서 Docker daemon을 쓰기 때문에 구조가 명확하다.

```text
Jenkins VM
   └── Docker daemon
```

반면 Kubernetes에서는 Pod가 노드의 Docker socket이나 containerd socket에 접근하는 순간, 그 Pod가 노드 전체에 영향을 준다. 이 부분은 [04-03.컨테이너 이미지 빌드](04-03.컨테이너%20이미지%20빌드.md)의 DinD/DooD 논의와 짝을 이룬다.

```text
Kubernetes Pod
   └── node runtime socket 접근
          └── Node 전체 위험
```

VM Jenkins가 안전하다는 뜻은 아니다. 다만 피해 범위를 "해당 Jenkins VM"으로 묶어 두기는 더 쉽다. 그래서 VM 환경에서는 격리 전략이 곧 "Jenkins VM을 어떻게 분리하고 어떻게 폐기할 수 있게 만드는가"의 문제로 좁혀진다.



## 4. 실무적 판단 — 어디까지 허용할 수 있나

> "내부망 + 신뢰된 Jenkinsfile만 실행"하는 경우와 "외부 코드까지 빌드"하는 경우는 위험도가 다르다. 같은 docker build라도 환경에 따라 받아들일 수 있는 수준이 갈린다.

내부망에서 신뢰된 개발자만 Jenkinsfile을 수정할 수 있고 외부 PR 빌드도 없는 환경이라면, VM Jenkins에서 Docker를 쓰는 것이 현실적인 선택이 된다.

```text
사내망
신뢰된 개발자만 Jenkinsfile 수정 가능
외부 PR 빌드 없음
Jenkins VM 권한 분리됨
```

이 환경에서도 Jenkins VM은 "깨끗하게 유지해야 하는 서버"가 아니라 **소모품**으로 취급해야 한다. 즉 언제든 폐기하고 재생성할 수 있는 빌드 워커로 본다.

```text
Jenkins VM은 깨끗해야 하는 서버가 아니라
언제든 폐기하고 재생성할 수 있는 빌드 워커
```

반대로 외부 코드나 불특정 브랜치를 빌드해야 하는 환경에서는 위험도가 크게 올라간다. 다음과 같은 조건이 겹치면 Docker 접근을 그대로 허용해서는 안 된다.

- GitHub/GitLab 외부 PR을 그대로 빌드
- 여러 팀이 공용 Jenkins를 사용
- 각 프로젝트가 Jenkinsfile을 자유롭게 수정
- 빌드 스크립트에 secret 주입
- `docker.sock`을 빌드에 노출

이런 환경에서는 누군가 Jenkinsfile에 다음 한 줄을 넣으면 곧바로 호스트 정보가 노출된다.

```groovy
sh 'docker run -v /:/host alpine cat /host/etc/passwd'
```

더 위험하게는 Jenkins credential, kubeconfig, registry token, SSH key처럼 빌드 과정에서 잠시라도 노출되는 자산이 모두 사정권에 들어온다.



## 5. 최소 방어선 다섯 가지

> VM Jenkins에서 Docker를 쓰겠다면, 최소한 다섯 가지는 지켜야 한다. 핵심은 "Controller에서 docker build를 돌리지 않고, 빌드 권한을 폐기 가능한 Agent로 옮기는 것"이다.

### 1. Jenkins Controller에서 직접 빌드하지 않기

가장 중요한 한 가지다. Jenkins Controller에는 설정, 플러그인, 모든 Credential이 모여 있어, 여기서 Docker 빌드를 돌리는 순간 사고 시 피해 범위가 가장 커진다.

```text
나쁜 구조:
Jenkins Controller VM에서 docker build

좋은 구조:
Jenkins Controller는 조율만
별도 Agent VM에서 docker build
```

### 2. Docker 빌드 전용 Agent를 분리하기

Controller와 빌드 Agent를 물리적으로 나누고, Docker daemon은 Agent VM에만 둔다. 사고가 나도 피해 범위가 Agent VM 안으로 묶인다.

```text
Jenkins Controller
   │
   └── Docker Build Agent VM
          ├── Docker daemon
          └── 빌드 후 폐기 또는 초기화
```

### 3. 외부 PR에는 Docker 권한을 주지 않기

신뢰되지 않은 코드에 Docker socket, Docker daemon, registry push 권한을 한꺼번에 주면 위험이 한 번에 터진다. 외부 PR은 lint와 test처럼 사이드 이펙트가 작은 단계로 제한한다.

```text
trusted branch: docker build 가능
external PR: 제한된 lint/test만 수행
```

### 4. Credential 주입 범위를 좁히기

빌드에 필요한 Secret만 그 단계에서만 살아 있도록 제한한다. `withCredentials` 블록을 짧게 잡고, 블록 바깥으로 Secret이 흐르지 않게 한다.

```groovy
withCredentials([
        usernamePassword(
                credentialsId: 'harbor-creds'
                , usernameVariable: 'REGISTRY_USER'
                , passwordVariable: 'REGISTRY_PASSWORD'
        )
]) {
    sh '''
        docker login harbor.example.com -u "$REGISTRY_USER" -p "$REGISTRY_PASSWORD"
        docker build -t harbor.example.com/app/my-service:${BUILD_NUMBER} .
        docker push harbor.example.com/app/my-service:${BUILD_NUMBER}
    '''
}
```

### 5. 빌드 Agent를 주기적으로 정리하기

Docker 빌드는 캐시, 레이어, 임시 이미지, 컨테이너를 계속 남긴다. 자원 누수뿐 아니라 옛 이미지에 박힌 비밀이 다음 빌드로 새는 경로가 되기도 한다.

```bash
docker system prune -af
```

운영에서는 무작정 prune하면 다른 빌드 캐시까지 같이 날아가므로, 빌드 전용 Agent 단위로 보존 정책과 청소 주기를 따로 잡는 편이 안전하다.



## 6. 결론

> VM Jenkins에서 Docker는 쓸 수 있다. 다만 "Jenkins가 Docker daemon에 접근하면 그 작업은 VM root에 가까운 힘을 가진다"는 사실을 전제로 구조를 짠다.

요약하면 이렇다.

```text
VM Jenkins에서 docker build는 가능하다.
하지만 Jenkins가 Docker daemon에 접근하는 순간,
그 Jenkins 작업은 VM root에 가까운 힘을 가진다.
```

권장 구조는 Controller가 직접 빌드하지 않고, 전용 Build Agent VM에 권한을 위임한 뒤 그 Agent를 자주 폐기·재생성하는 형태다.

```text
Jenkins Controller
   ↓
전용 Docker Build Agent VM
   ↓
docker build / docker push
   ↓
Agent 정리 또는 폐기
```

한 줄로 정리하면, VM Jenkins에서는 Docker 사용이 Kubernetes보다 운영하기 단순할 수는 있지만, Jenkins Controller에서 직접 Docker를 돌리는 구조는 피하고 Docker 빌드 전용 Agent로 격리하는 것이 안전하다.
