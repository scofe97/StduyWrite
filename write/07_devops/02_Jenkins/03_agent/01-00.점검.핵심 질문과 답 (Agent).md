> 다루는 문서: `04-01.실행환경으로서의 Agent` ~ `04-04.빌드 도구 비교와 선택`

## Q1: DinD와 DooD의 차이와 트레이드오프는?

DinD는 컨테이너 안에서 별도 Docker daemon을 띄우므로 `--privileged`가 필수이며 컨테이너 탈출 위험이 있고 레이어 캐시도 독립이라 느리다. DooD는 호스트 socket을 마운트해 `--privileged`는 불필요하지만 socket 접근 = 호스트 root 접근과 사실상 동일하므로 보안 측면은 동등하게 위험하다. 그래서 Kubernetes 보안 정책 환경에서는 두 방식 모두 부적합하고 Kaniko·Buildah처럼 daemon-free 도구로 우회한다.

비교 표: `04-03.md` §1. VM 환경에서의 위험 모델: `04-03a.md`.

## Q2: Docker Agent를 사용하면 어떤 문제가 해결되는가?

호스트 직접 빌드는 Agent에 설치된 도구 버전에 의존하므로 "이 Agent에서만 실패"하는 비결정성이 발생한다. Docker Agent는 이미지 태그로 버전을 고정해 동일 결과를 보장하고, 빌드마다 독립 컨테이너를 써서 빌드 간 간섭을 원천 차단한다. 핵심 오해: Docker Agent는 빌드 "환경"을 컨테이너화하는 것이지 Docker 이미지를 빌드하는 게 아니다 — 안에서 `docker build`를 돌리려면 DinD/DooD/Kaniko 같은 별도 전략이 추가로 필요하다.

상세: `04-02.md` §1.

## Q3: `latest` 태그를 프로덕션에서 사용하면 안 되는 이유는?

세 가지가 동시에 깨진다. 비결정성(여러 브랜치 동시 빌드 시 어느 커밋이 `latest`인지 보장 불가), 롤백 불가("이전 latest" 개념이 없음), Kubernetes 노드 캐시 불일치(`imagePullPolicy: IfNotPresent`가 기본이라 노드마다 다른 버전이 실행됨). 권장 전략은 Git SHA(`myapp:a1b2c3d`)나 SemVer + SHA(`myapp:2.1.0-a1b2c3d`) 조합으로 추적성과 의미를 함께 확보하는 것이다.

태깅 전략 표: `04-02.md` §5.

## Q4: Kaniko가 Docker socket 마운트보다 안전한 이유는?

`/var/run/docker.sock`에 접근할 수 있는 프로세스는 `docker run -v /:/host alpine chroot /host`로 호스트 root 쉘을 얻을 수 있다. Kubernetes Pod Security Standards `restricted` 프로파일은 호스트 path 마운트와 privileged 컨테이너를 모두 금지하므로 Docker daemon 의존이 정책 위반이 된다. Kaniko는 Dockerfile 명령을 사용자 공간에서 실행하고 파일시스템 스냅샷으로 레이어를 만들어 daemon 없이 일반 권한으로 동작한다. 트레이드오프는 레이어 캐싱 효율이 Docker만 못해 빌드 속도가 떨어지는 것.

상세: `04-03.md` §2. 단, **Kaniko는 2026-06부로 archive 상태**이므로 신규 도입 시 `04-04.md` 결정 트리를 먼저 본다.

## Q5: Multi-stage Dockerfile이 CI/CD에서 중요한 이유는?

이미지 크기와 보안을 한 번에 개선한다. Node.js Single-stage 1GB+ 이미지가 nginx 서빙 30~50MB로 떨어지면 레지스트리 전송 시간과 컨테이너 시작 시간이 줄고, 최종 이미지에서 컴파일러·패키지 매니저가 빠지면 공격 표면과 CVE 탐지 수가 극적으로 감소한다. 추가 효과로 `COPY package*.json ./` + `RUN npm ci`를 소스 복사보다 먼저 두면 의존성 레이어가 캐시되어 재빌드 속도까지 빨라진다.

상세: `04-03.md` §3.

## Q6: 정적 Agent와 동적 Agent의 차이는?

정적은 미리 등록되어 항상 온라인이라 빌드가 즉시 시작되지만 빌드 없는 시간에도 리소스를 점유하고 환경이 누적 오염된다. 동적은 빌드 요청 시 생성·완료 후 삭제라 매번 깨끗한 환경을 보장하지만 Agent 생성 시간만큼 지연이 붙는다. 대규모 CI에서는 Kubernetes Plugin으로 Pod를 동적 생성하는 방식이 표준이며, Agent Pool 크기는 피크 동시 빌드 수와 생성 시간의 균형으로 결정한다.

비교 표와 Pod 생성 흐름: `04-01.md` §3.
