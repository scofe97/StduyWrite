# 젠킨스 인증 모델과 TPS 패턴 (2.222+)

> **본 문서는 spec(`05-02.md`)을 읽었다고 가정한 코드 패턴 모음**이다. 인증 API의 호출 순서, crumb·cookie의 발급 절차, 응답 필드 구조는 spec에 있다. 이 문서는 그 위에서 "왜 인증 모델이 단순해졌는지"와 "TPS 코드에서 어떤 패턴이 자리 잡았는지"를 정리한다.

## 1. Jenkins 2.222+에서 바뀐 점 — 짧게

Jenkins 2.222부터 **API Token 인증 요청은 CSRF crumb 검증 대상에서 제외**된다(공식 문구: "Requests authenticating with an API token are exempt from CSRF protection in Jenkins"). `/crumbIssuer/api/json` 자체가 사라진 것이 아니라, API Token 기반 호출은 crumb 발급 단계를 건너뛸 수 있다는 뜻이다.

| 인증 방식 | POST에서 crumb | cookie | 특징 |
|-----------|----------------|--------|------|
| ID/Password | 필요 | 필요 | 현재 TPS 검증 환경 (`05-02` 흐름 유지) |
| API Token | 불필요 | 불필요 | 2.222+ 권장 — GET/POST 인증 헤더 통일 가능 |

이 차이는 코드 구조에 직접 영향을 준다. ID/Password는 GET/POST 분기, crumb 캐시, cookie 보존 로직을 모두 가져야 하지만, API Token은 `Authorization: Basic <user:token>` 한 줄로 통일된다.

> 주의: 운영 환경이 ID/Password 인증이면 2.462.3·2.504 같은 최신 LTS여도 `05-02` crumb + cookie 흐름이 여전히 맞다. "최신 버전 = 자동 단순화"가 아니라 "**인증 수단**과 **`useCrumbs` 정책**"이 결정한다.

## 2. JenkinsAuthVo / JenkinsSession 패턴

핵심은 인증 재료를 호출마다 흩뿌리지 않고 하나의 객체로 캡슐화하는 것이다. ID/Password 환경에서는 `JenkinsSession` 형태로 세션 문맥을 통째로 다룬다.

```java
public record JenkinsSession(
    URI baseUrl,
    String basicAuth,
    String crumbRequestField,
    String crumb,
    String cookie
) {}
```

발급 흐름은 응답 본문에서 `crumb`/`crumbRequestField`를, 응답 헤더 `Set-Cookie`에서 `JSESSIONID`를 추출해 한 번에 묶는다.

```java
ResponseEntity<CrumbResponse> response =
    jenkinsFeignClient.getCrumb(baseUrl, basicAuth);

String cookie = response.getHeaders()
    .getValuesAsList(HttpHeaders.SET_COOKIE)
    .stream()
    .map(value -> value.split(";", 2)[0])
    .collect(Collectors.joining("; "));

JenkinsSession session = new JenkinsSession(
    baseUrl,
    basicAuth,
    response.getBody().crumbRequestField(),
    response.getBody().crumb(),
    cookie
);
```

이후 POST는 `Authorization` + `session.crumbRequestField(): session.crumb()` + `Cookie: session.cookie()` 세 헤더만 다시 꺼내 쓴다. crumb·cookie 발급 메커니즘 자체는 `05-02` §3을 참조한다.

API Token 환경으로 전환되면 객체는 두 필드로 줄어든다.

```java
public record JenkinsAuth(String basicAuth, URI jenkinsUrl) {}
```

## 3. Feign URI Override 패턴

TPS는 하나의 Feign Client로 다수의 Jenkins 서버를 호출해야 하므로, 첫 번째 파라미터로 `URI baseUrl`을 받아 런타임에 실제 Jenkins URL을 덮어쓴다.

```java
@FeignClient(
    name = "jenkins",
    url = "https://",
    configuration = FeignJenkinsConfig.class
)
public interface JenkinsFeignClient {

    @GetMapping("/crumbIssuer/api/json")
    ResponseEntity<JenkinsAuthVo> getCrumb(
        URI baseUrl,
        @RequestHeader("Authorization") String basicAuth
    );
}
```

이 패턴 덕분에 DB에 저장된 Jenkins URL을 그대로 활용할 수 있고, 인증 정보와 대상 서버를 한 요청 문맥 안에서 다룰 수 있다.

`FeignJenkinsConfig`는 단순 HTTP 설정이 아니라 Jenkins 연동 규칙(요청·응답 로깅 통일, 자체 서명 인증서 대응, 공통 헤더 정책)의 집합이다. 검증 환경에서 `curl -k`가 필요한 것과 같은 이유로, 코드에서도 개발/운영의 SSL 처리 정책을 분리하는 것이 핵심이다.

## 4. 현대화 시 줄일 수 있는 것

API Token 환경으로 통일되면 다음 요소를 모두 제거할 수 있다 — `crumb`·`cookie`·`crumbRequestField` 필드, crumb 조회 메서드, crumb 캐시, POST 전용 헤더 분기.

```java
// Before — ID/Password
private HttpHeaders buildPostHeaders(long jenkinsInstanceId) {
    var headers = buildGetHeaders(jenkinsInstanceId);
    var crumb = getCrumb(jenkinsInstanceId);
    if (crumb != null) {
        headers.set(crumb.crumbRequestField, ...);
        headers.set("Cookie", crumb.cookie);
    }
    return headers;
}

// After — API Token
private HttpHeaders buildHeaders(long jenkinsInstanceId) {
    var headers = new HttpHeaders();
    headers.set("Authorization", "Basic " + getBasicAuth(jenkinsInstanceId));
    return headers;
}
```

"현대화"의 실질적 효과는 인증 개념 변화가 아니라 **코드 복잡도 감소**다.

## 5. crumb·cookie 만료와 갱신 전략

crumb은 "고정 토큰"이 아니라 사용자·세션 ID·IP·instance salt에 묶인 CSRF 값이라, 세션이 바뀌거나 IP가 변하면(프록시 환경) 무효화될 수 있다. cookie는 일반 세션 만료(`--sessionTimeout`, 미설정 시 기본 60분)로 끊긴다. `Enable proxy compatibility`를 켜면 IP 조건은 제외할 수 있다.

운영 코드의 현실적인 전략은 다음과 같다.

- 매 POST마다 crumb을 새로 받지 않는다 — `JenkinsSession`을 메모리에 캐시한다.
- POST에서 `403 No valid crumb was included in the request`가 나오면 crumb·cookie를 함께 재발급하고 1회만 재시도한다.
- Jenkins 세션 타임아웃을 알고 있다면 그보다 조금 이르게 선제 갱신한다.

매 요청마다 `GET /crumbIssuer/api/json`을 선행하거나, crumb·cookie를 파일에 장기 보관해 재사용하는 방식은 비효율적이다.

## 6. 버전별 기억할 것

| Jenkins 버전 | 인증 영향 |
|-------------|----------|
| 2.222 | API Token 인증 시 crumb 면제 도입 |
| 2.462.3 | 현재 검증 환경. 비밀번호 인증이면 crumb 흐름 유지 |
| 2.504 | 최근 LTS — crumb 면제 원칙 유지 |

버전 자체보다 중요한 건 **지금 무엇으로 인증하는가**와 **현재 Jenkins의 보안 정책**이다.

## 7. 문서 사용 가이드

- `05-02`: 현재 환경에서 인증 API를 어떻게 호출하는가 (spec)
- `05-02a` (이 문서): 인증 모델이 왜 달라지는가, TPS 코드에서 무엇이 패턴인가
- `05-02b`: API Token을 실제로 어떻게 발급하고 회전할 것인가
- `05-04`: 인증이 끝난 뒤 빌드 실행 POST를 어떻게 보내는가
- `05-05`: 빌드 이후 상태를 어떻게 추적하는가

## 8. 핵심 정리

1. Jenkins 2.222+에서 API Token 인증은 crumb 검증에서 면제된다 — 그래서 GET/POST 인증 규칙을 통합할 수 있다.
2. 다만 ID/Password 환경에서는 spec의 crumb + cookie 흐름이 여전히 맞다. crumb·cookie는 인증 대체재가 아니라 비밀번호 POST의 세션·CSRF 검증 보조 재료다.
3. TPS 코드는 `JenkinsSession`(또는 `JenkinsAuth`) + Feign URI override + 공통 `FeignJenkinsConfig` 세 패턴을 축으로 인증 복잡도를 격리한다.
