# Jenkins 엔진 심화 학습 MOC

---

> [`04_api`](../04_api/README.md)가 "Jenkins를 밖에서 호출하는 법"을 다뤘다면, 이 묶음은 그 호출이 들어온 뒤 **엔진 안에서 무슨 일이 벌어지는가**, 그리고 그 엔진을 **어떻게 분해·개조·원격조작·외부연결하는가**를 다룹니다. 본 묶음을 끝내면 "REST로 빌드를 던졌을 때 Jenkins가 URL을 어느 메서드로 보내고, 큐에 어떻게 쌓고, 어느 실행기에 배정하는지"를 소스 레벨로 설명하고, 직접 플러그인을 만들어 끼우며, REST로 닿지 않는 제어를 Script Console로 수행할 수 있습니다.

## 왜 한 묶음으로 분리했는가

`04_api` 27편은 curl로 Jenkins를 호출하는 관점을 거의 전부 다뤘습니다. 인증·Job CRUD·빌드 트리거·큐 추적·로그·크레덴셜·배포 승인까지, "어떤 엔드포인트를 어떻게 호출하는가"는 그 묶음에서 답이 끝납니다. 그래서 같은 폴더에 "엔진 내부 동작"을 더 쌓으면 호출자 관점과 구현자 관점이 한데 섞여 진입점이 흐려집니다.

이 묶음은 관점을 한 단계 내립니다. `04_api/05-01`이 `POST /job/x/build`를 *보내는 법*이라면, 여기 `02-01`은 그 요청이 `BuildableItem`을 거쳐 `doBuild()`에 *도달하는 경로*입니다. `04_api/05-04`가 큐 상태 전이를 *조회 API로 관찰*했다면, 여기 `03-02`는 그 전이를 *디버거 브레이크포인트로 직접 멈춰* 보고 실행기 배정 알고리즘까지 내려갑니다. 호출자 문서를 다 본 사람이 "그 안은 어떻게 돌아가나"로 넘어오는 자리입니다.

엔진을 들여다보는 능력은 곧 확장하는 능력으로 이어집니다. 내부 구조를 알면 REST에 없는 기능을 플러그인으로 만들거나(`05`), Script Console로 모델 객체를 직접 조작할(`06`) 수 있습니다. 그래서 내부 동작(`02`~`04`)·확장(`05`)·제어(`06`)·통합(`07`)을 한 묶음으로 묶었습니다. 네 축은 "엔진을 안다"는 같은 뿌리에서 갈라집니다.

## 우선순위 — 무엇부터 보는가

학습 동기에 따라 진입 순서가 갈립니다:

1. **이력서·면접이 목표면** `01-01` → `02` → `03` 순으로 봅니다. 큐 내부 동작과 실행기 배정(`03`)은 분산 시스템의 작업 스케줄링·직렬화 주제로 직결되어 면접에서 깊이를 보여 줄 수 있습니다.
2. **Jenkins를 확장해 쓰는 게 목표면** `01-01` → `05` → `06`을 먼저 봅니다. 플러그인 제작과 Script Console 제어는 REST로 안 되는 일을 해내는 실무 능력입니다.
3. **외부 시스템과 통합이 목표면** `04_api`를 먼저 끝낸 뒤 `07-01`로 옵니다. SDK·역방향 인증·모니터링은 호출 패턴을 안 상태에서 다뤄야 맥락이 잡힙니다.

## 사전 지식

> 이 묶음은 다음을 안다고 가정합니다.

1. [`../04_api/`](../04_api/README.md)의 REST 호출 — 어떤 엔드포인트가 무엇을 하는지를 압니다. 이 묶음은 그 호출의 *내부*를 봅니다.
2. [`../05_operations/02-04`](../05_operations/README.md)의 Groovy 기본 문법 — `06-01` Script Console 심화의 전제입니다.
3. Java와 JVM 기본 — 리플렉션·직렬화·클래스로더 개념이 `02`·`04`·`05`에서 반복 등장합니다.
4. IDE의 원격 디버그(remote attach) 경험 — `01-01`에서 처음부터 다루지만, 한 번이라도 attach해 본 경험이 있으면 빠릅니다.

위 항목 중 막히는 부분이 있으면 [`../04_api/`](../04_api/README.md), [`../05_operations/`](../05_operations/README.md)를 먼저 본 뒤 돌아옵니다.

## 면접 대비 체크리스트

> 묶음을 다 읽은 뒤 다음 질문에 답할 수 있어야 합니다. 답을 보지 말고 먼저 스스로 설명해 보십시오.

1. `POST /job/myjob/build` 요청이 들어왔을 때 Jenkins는 이 URL을 어떻게 자바 메서드 호출로 바꿉니까? Stapler가 객체 그래프를 어떻게 순회합니까?
2. 큐에 들어온 작업이 실행기에 배정되기까지 거치는 상태 네 가지는 무엇이며, `maintain()` 루프는 어떤 주기로 무엇을 합니까?
3. Jenkins 큐가 단일 컨트롤러에서 직렬로 dispatch하는 구조를, 여러 인스턴스가 같은 DB를 보는 분산 환경에서 재현하려면 무엇으로 직렬화해야 합니까? (`FOR UPDATE` 비관 락 관점)
4. Extension Point로 기능을 추가하는 것과 `RunListener` 같은 Listener로 후킹하는 것은 무엇이 다릅니까? 각각 언제 씁니까?
5. 플러그인이 다른 플러그인의 클래스를 함부로 보지 못하는 이유는 무엇입니까? Jenkins의 `PluginClassLoader` 격리가 OSGi와 다른 점은?

각 질문에 막히면 해당 절로 돌아갑니다. 특히 3번은 이 묶음의 핵심으로, `03-02`가 Jenkins 큐 내부 동작과 분산 환경 직렬화를 나란히 놓고 답합니다.
