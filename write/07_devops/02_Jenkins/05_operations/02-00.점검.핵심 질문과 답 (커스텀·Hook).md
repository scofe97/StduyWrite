## Q1: `vars/`와 `src/`의 차이는? 어떤 코드를 어디에 넣는가?

| 디렉토리 | 역할 | 특징 |
|----------|------|------|
| `vars/` | Pipeline에서 함수처럼 직접 호출되는 전역 변수 | `call()` 메서드로 진입. Pipeline DSL(`sh`, `stage`, `docker`) 자유롭게 사용 가능 |
| `src/` | 일반 Groovy 클래스 | Java와 동일한 패키지 구조. Pipeline DSL 직접 접근 불가, `this`를 넘겨받아야 함. 반드시 `Serializable` 구현 필요 |

실용 가이드:

- `vars/`: Stage 정의, sh 호출, 알림 같은 Pipeline 흐름
- `src/`: 설정 객체, 조건 판단, 문자열 변환 같은 순수 비즈니스 로직. Jenkins 없이도 단위 테스트가 가능하다.

## Q2: `@Library` 버전 고정이 왜 중요한가? 고정하지 않으면 어떤 일이 발생하는가?

버전을 고정하지 않으면 전역 설정의 기본 브랜치 최신 코드를 항상 사용한다.

- **위험**: 라이브러리에 breaking change가 머지되는 순간, 이 라이브러리를 사용하는 모든 Pipeline이 동시에 깨진다.
- **해결**: `@Library('my-lib@v1.2.0')`처럼 태그를 고정하면 라이브러리가 아무리 변경되어도 해당 Pipeline은 동일한 코드를 사용한다.

| 환경 | 권장 방식 |
|------|----------|
| 프로덕션 | 태그 또는 커밋 해시로 고정 |
| 스테이징/개발 | `@main`으로 최신 변경을 빠르게 검증 |

### 심화

라이브러리 버전 고정 후 보안 패치가 나왔을 때 전체 Pipeline을 일괄 업데이트하는 전략은?



## Q3: Shared Library의 `call()` 메서드는 어떤 역할을 하는가?

`vars/buildDocker.groovy`에서 `call()` 메서드는 Jenkinsfile에서 `buildDocker()`로 직접 호출될 때 실행되는 진입점이다.

- **Map config 패턴** 권장: `buildDocker(imageName: 'app', tag: 'v1')` 형태로 이름 기반 파라미터 호출
  - 가독성이 높고, 새 파라미터를 추가해도 기존 호출부가 깨지지 않는다.
  - 필수 파라미터 즉시 검증: `config.imageName ?: error("imageName is required")`

## Q4: CPS 변환과 Shared Library의 호환성 문제는 무엇인가? `@NonCPS`는 언제 사용하는가?

CPS 변환은 직렬화할 수 없는 객체를 다룰 때 문제를 일으킨다:

- 대표 케이스: `java.util.regex.Matcher`, 클로저 내 람다, `Closure` 직접 저장
- 증상: `NotSerializableException`

| 구분 | CPS 메서드 | `@NonCPS` 메서드 |
|------|-----------|----------------|
| 직렬화 | 가능 | 불가 |
| Pipeline DSL(`sh`, `stage`) 호출 | 가능 | 불가 |
| 직렬화 불가 객체 사용 | 불가 | 가능 |

- 직렬화 불가능한 객체를 다루는 헬퍼 로직은 `@NonCPS`로 분리하고, Pipeline step 호출은 CPS 메서드에서 수행하는 패턴이 일반적이다.

## Q5: Implicit Loading과 Explicit Loading은 어떤 상황에 어느 것을 선택하는가?

| 방식 | 동작 | 특징 |
|------|------|------|
| Explicit | Jenkinsfile에 `@Library('my-lib@v1.2.0') _` 명시 | 추적성 높음. 버전 고정을 Jenkinsfile에서 직접 제어 |
| Implicit | Jenkins 관리 화면에서 "Load implicitly" 활성화 | 모든 Pipeline에 자동 로딩. Jenkinsfile만 봐서는 어떤 라이브러리가 로딩되는지 알 수 없어 디버깅 어려움 |

- 권장 조합(계층화): 보안 스캔, 표준 알림처럼 모든 빌드에 강제해야 하는 거버넌스 라이브러리는 Implicit, 팀별 도메인 라이브러리는 Explicit으로 선언한다.

## Q6: `init.groovy.d`와 JCasC의 차이는? 선택 기준은?

| 구분 | JCasC | `init.groovy.d` |
|------|-------|----------------|
| 패러다임 | 선언적 YAML | 명령적 Groovy 코드 |
| 재현성 | 동일 YAML → 동일 결과 보장 | 코드 실행 순서에 따라 결과 달라질 수 있음 |
| Git 추적 | YAML diff 명확 | Groovy 코드 diff |
| 접근 가능 범위 | JCasC가 지원하는 설정만 | Jenkins 전체 Internal API |
| 업그레이드 위험 | 낮음 | Internal API 변경으로 스크립트가 깨질 수 있음 |

- 선택 원칙: "JCasC로 할 수 있으면 JCasC, 안 되면 `init.groovy.d`"

## Q7: Script Console의 보안 위험은 무엇이고, 어떻게 제한해야 하는가?

Script Console은 Sandbox 없이 Jenkins 프로세스 내부에서 Groovy 코드를 직접 실행한다.

가능한 위험 행위:

- **크레덴셜 평문 노출**: Jenkins는 크레덴셜을 암호화 저장하지만 복호화 키가 프로세스 메모리에 있어 Script Console에서 모든 시크릿을 읽을 수 있다.
- 모든 Job 삭제, 보안 설정 변경, 파일 시스템 접근

이것은 설계 결함이 아니다. Jenkins가 빌드 중 크레덴셜을 사용해야 하므로 프로세스가 복호화 능력을 가져야 하고, 그 능력이 Script Console을 통해 노출되는 구조적 특성이다.

완화 방법:

- Matrix-based Security에서 "Administer" 권한과 "Run Scripts" 권한을 분리
- Script Console 접근을 극소수 관리자에게만 부여
- 감사 로그 활성화
