# Groovy 커스터마이징의 범위와 한계

---

> 이 문서를 읽고 나면 Groovy로 Jenkins를 어디까지 확장할 수 있는지(Pipeline 확장·전역 Hook·동적 설정·모니터링·보안 자동화)를 **설명하고**, 크레덴셜 평문 노출·외부 코드 실행·Sandbox 우회 같은 위험을 **식별하며**, JCasC와 init.groovy.d를 **비교**해 "선언적이 가능하면 선언적, 안 되면 명령적"이라는 우선순위로 수단을 **선택**할 수 있습니다. 나아가 CPS 변환이 무엇을 깨뜨리는지(`@NonCPS` 내부에서 step 호출 불가 등)를 **예측**할 수 있습니다.



## 사전 지식

`02-02`에서 본 Groovy의 세 실행 영역(Pipeline DSL·Script Console·init.groovy.d)과 Script Security Plugin의 Sandbox 개념을 알고 있어야 합니다. JCasC가 선언적 설정 도구라는 점(`02-01`)도 전제입니다.



## 진입 — 왜 "할 수 있다"의 범위를 먼저 그어야 하는가

> Groovy를 통한 Jenkins 커스터마이징은 사실상 무한에 가깝습니다. Jenkins의 모든 Java API에 접근할 수 있으므로, UI에서 할 수 있는 모든 것과 UI에서 할 수 없는 것까지 Groovy로 가능합니다.

문제는 이 무한한 권한이 그대로 운영 위험으로 직결된다는 점입니다. 무엇이든 할 수 있다는 말은, 한 줄의 스크립트가 전체 빌드 인프라를 멈추거나 모든 크레덴셜을 외부로 흘릴 수 있다는 뜻이기도 합니다. 그래서 "어디까지 가능한가"를 그리는 동시에 "어디서부터는 하지 말아야 하는가"의 경계를 함께 그어야 합니다. 이 문서는 가능 범위와 위험 범위를 나란히 놓고, 그 사이에서 수단을 고르는 우선순위를 세웁니다.



## 1. 가능한 것과 위험한 것 — 권한의 양면

> 이미 아는 "관리자 셸 접근"의 자동화 측면입니다. 셸에 들어가면 무엇이든 할 수 있듯, Groovy는 Jenkins 내부에 들어가 무엇이든 할 수 있습니다.

Groovy의 접근 권한은 셋방을 빌린 세입자가 아니라 **건물 마스터키를 가진 관리인**에 가깝습니다. 관리인은 모든 방문을 열어 청소도 하고 수리도 하지만, 그 키가 분실되면 건물 전체가 위험해집니다. 다만 이 비유는 *권한 범위*까지만 들어맞고, **실행 시점**에서 깨집니다. 마스터키는 사람이 손에 쥐어야 쓰지만, init.groovy.d 스크립트는 Jenkins가 기동될 때마다 *자동으로* 키를 돌립니다. 그래서 외부 코드 한 줄이 박혀 있으면 사람의 개입 없이 매 기동 시 실행됩니다.

| 영역              | 가능한 작업                 | 예시                                  |
| ----------------- | --------------------------- | ------------------------------------- |
| **Pipeline 확장** | 커스텀 DSL step 정의        | `deployToK8s()` 같은 도메인 전용 step |
| **전역 Hook**     | 모든 빌드에 전/후 동작 삽입 | 빌드 시작/완료 시 Slack 알림          |
| **동적 설정**     | 런타임에 설정 변경          | 특정 시간대에 Agent 수 조절           |
| **모니터링**      | 시스템 상태 수집            | 빌드 큐 길이, Agent 활용률            |
| **보안 자동화**   | 보안 정책 프로그래매틱 적용 | 사용자 권한 일괄 변경, 감사 로그      |

### 위험한 것

> "할 수 있다"와 "해야 한다"는 다릅니다. Groovy의 무제한 접근 권한이 오히려 보안 사고와 운영 장애의 원인이 될 수 있습니다.

- **크레덴셜 평문 노출**: Script Console에서 모든 비밀값을 읽을 수 있습니다. 누군가 악의적으로 또는 실수로 크레덴셜을 로그에 출력하면, 그 로그가 빌드 아티팩트나 모니터링 시스템으로 전파될 수 있습니다.

- **init.groovy에서 외부 URL 호출**: init.groovy.d 스크립트에서 외부 URL의 코드를 다운로드하여 실행하는 패턴이 있습니다. 이것은 **공급망 공격(Supply Chain Attack)의 벡터**입니다. 외부 서버가 해킹당하면 Jenkins 시작 시마다 악성 코드가 실행됩니다.

```groovy
// ⚠️ 절대 하지 말아야 할 패턴
// 외부 URL에서 받은 텍스트를 GroovyShell로 즉시 평가 — 외부 서버가 탈취되면
// 그 코드가 Master 권한으로 실행됩니다(why: 출처를 이미지·저장소에 고정해야 하는 이유).
new GroovyShell().evaluate(
    new URL("https://example.com/init-script.groovy").text
)
```

- **시스템 프로퍼티 무분별 변경**: `System.setProperty()`로 JVM 레벨의 설정을 변경하면 Jenkins의 동작이 예측 불가능해질 수 있습니다. 예를 들어 HTTP 프록시 설정을 잘못 변경하면 플러그인 업데이트나 Git clone이 모두 실패합니다.

- **Groovy Sandbox 우회 시도**: Pipeline에서 Sandbox를 우회하려는 시도는 Script Security Plugin이 차단하지만, 우회 방법이 발견될 때마다 보안 취약점으로 보고됩니다. Jenkins 보안 어드바이저리에서 가장 빈번하게 등장하는 유형 중 하나가 Sandbox 우회입니다.



## 2. Groovy 커스터마이징 권장 여부 -- 현실적 가이드

> 모든 커스터마이징이 Groovy로 가능하다고 해서, 모두 Groovy로 구현해야 하는 것은 아닙니다. 선언적 방법이 존재한다면 선언적 방법을 우선해야 유지보수 부채를 최소화할 수 있습니다.

선택의 원칙은 한 갈래로 정리됩니다. 선언적으로 풀리면 선언적으로 풀고, 막힐 때만 한 단계씩 명령적 수단으로 내려갑니다.

```mermaid
flowchart TD
    NEED["Jenkins 커스터마이징 필요"] --> Q1{"JCasC로 가능?"}
    Q1 -->|"예"| JC["JCasC (선언적 YAML)"]
    Q1 -->|"아니오"| Q2{"공통 로직?<br/>(Pipeline 재사용)"}
    Q2 -->|"예"| SL["Shared Library"]
    Q2 -->|"아니오"| Q3{"기동 시 자동 설정?"}
    Q3 -->|"예"| IG["init.groovy.d (명령적)"]
    Q3 -->|"아니오 (일회성)"| SC["Script Console<br/>+ 감사 로그"]
    style NEED fill:#ADD8E6,stroke:#333,color:#000
    style Q1 fill:#FFF2CC,stroke:#333,color:#000
    style Q2 fill:#FFF2CC,stroke:#333,color:#000
    style Q3 fill:#FFF2CC,stroke:#333,color:#000
    style JC fill:#90EE90,stroke:#333,color:#000
    style SL fill:#90EE90,stroke:#333,color:#000
    style IG fill:#FFF2CC,stroke:#333,color:#000
    style SC fill:#FFB6C1,stroke:#333,color:#000
```

### 권장 판단 매트릭스

| 용도                 | 대안                          | 결론                                                   |
| -------------------- | ----------------------------- | ------------------------------------------------------ |
| 초기 설정 자동화     | JCasC (Configuration as Code) | **JCasC 우선**, 불가능할 때만 init.groovy.d            |
| Pipeline 공통 로직   | Shared Library                | **Shared Library 사용**                                |
| 일회성 관리 작업     | Script Console                | **주의하여 사용**, 감사 로그 필수                      |
| 빌드 전/후 전역 동작 | Global Pipeline Libraries     | **Library 우선**, 불가능하면 init hook                 |
| 크레덴셜 등록        | JCasC + 외부 Secret Manager   | **JCasC + Vault/K8s Secret**, 불가능하면 init.groovy.d |
| 플러그인 설정 초기화 | JCasC                         | **JCasC가 거의 모든 경우 가능**                        |

- 핵심 원칙은 명확합니다. **"선언적 방법이 있으면 선언적 방법을 쓰고, 없을 때만 명령적 방법을 쓴다."** JCasC는 선언적이고, init.groovy.d는 명령적입니다.

### JCasC vs init.groovy.d 비교

> **JCasC (Jenkins Configuration as Code)**는 Jenkins 설정을 YAML 파일로 선언하는 플러그인입니다. `jenkins.yaml` 파일 하나로 보안, 크레덴셜, 도구, 에이전트 설정을 모두 정의할 수 있습니다.

```yaml
# JCasC 예시: jenkins.yaml
jenkins:
  securityRealm:
    local:
      allowsSignup: false   # why: 자가 가입을 막아 무단 계정 생성 차단
      users:
        - id: "admin"
          # 평문 대신 환경변수 치환 — 비밀번호를 YAML에 박지 않습니다.
          password: "${JENKINS_ADMIN_PASSWORD}"
  authorizationStrategy:
    loggedInUsersCanDoAnything:
      allowAnonymousRead: false   # why: 익명 읽기를 닫아 빌드 로그 노출 방지
credentials:
  system:
    domainCredentials:
      - credentials:
          - usernamePassword:
              id: "git-credentials"
              # 토큰도 환경변수로 — Secret Manager에서 주입(why: diff에 비밀값이 남지 않음)
              username: "${GIT_USER}"
              password: "${GIT_TOKEN}"
tool:
  jdk:
    installations:
      - name: "JDK17"
        home: "/usr/lib/jvm/java-17-openjdk"
```

**init.groovy.d**는 동일한 설정을 Groovy 코드로 명령적으로 작성합니다.

| 비교 항목       | JCasC                       | init.groovy.d                              |
| --------------- | --------------------------- | ------------------------------------------ |
| **패러다임**    | 선언적 (YAML)               | 명령적 (Groovy 코드)                       |
| **버전 관리**   | 쉬움 (YAML diff가 명확)     | 어려움 (코드 diff는 의도 파악이 힘듦)      |
| **디버깅**      | 명확한 에러 메시지          | println 디버깅, 스택 트레이스 해석 필요    |
| **재현성**      | 동일 YAML = 동일 결과       | 실행 순서, 타이밍에 따라 결과 다를 수 있음 |
| **유연성**      | 플러그인 지원 범위 내에서만 | 무제한 (Jenkins 전체 API 접근)             |
| **조건부 로직** | 제한적 (환경변수 치환 정도) | 완전한 프로그래밍 가능                     |
| **학습 곡선**   | 낮음 (YAML 작성)            | 높음 (Jenkins Internal API 이해 필요)      |

- **JCasC로 할 수 있으면 JCasC를 쓰고, JCasC로 불가능한 경우에만 init.groovy.d를 사용하며, Script Console은 긴급 상황이나 일회성 조사에만 사용합니다.**
- 이 우선순위를 어기면 유지보수 부채가 빠르게 쌓입니다.

부채는 숫자로 드러납니다. 예를 들어 30개 항목짜리 설정을 init.groovy.d 스크립트 5개(각 약 200줄)로 운영하다 JCasC로 옮기면, 같은 설정이 `jenkins.yaml` 한 파일 약 120줄로 줄어듭니다. 더 중요한 것은 변경 검토 시간으로, Groovy 5개 파일의 의도를 코드 diff로 파악하려면 항목당 수 분이 걸리던 것이, YAML diff에서는 바뀐 키 한두 줄만 보면 끝납니다(출처: jenkins.io/doc/book/managing/casc).



## 3. Script Security Plugin과 Sandbox

> Script Security Plugin은 Pipeline에서 실행되는 Groovy 코드에 보안 경계를 설정하는 핵심 플러그인입니다. 이 플러그인이 없으면, 개발자가 Jenkinsfile에서 `System.exit(0)`을 실행하여 Jenkins를 종료하거나, 크레덴셜을 평문으로 읽어서 외부로 전송할 수 있습니다.

```mermaid
flowchart TB
    Dev["개발자가 Pipeline<br/>Groovy 코드 작성"]
    Dev --> Sandbox{"Sandbox<br/>Check"}

    Sandbox -->|"허용된 API<br/>(sh, echo, stage)"| Execute["즉시 실행"]
    Sandbox -->|"비허용 API<br/>(System.exit, Runtime.exec)"| Block["실행 차단"]
    Block --> Approval{"관리자<br/>Script Approval"}
    Approval -->|"승인"| AllowList["허용 목록에 추가"]
    AllowList --> Execute
    Approval -->|"거부"| Reject["실행 거부<br/>Pipeline 실패"]

    Execute --> Result["빌드 결과"]
    Reject --> Fail["빌드 실패"]

    subgraph legend["보안 계층"]
        L1["1차: Sandbox 자동 검사"]
        L2["2차: 관리자 수동 승인"]
        L3["3차: 허용 목록 관리"]
    end

    style Sandbox fill:#ffc107,color:#000
    style Block fill:#dc3545,color:#fff
    style Execute fill:#28a745,color:#fff
    style Reject fill:#dc3545,color:#fff
```

- **Sandbox 모드의 동작 원리**: Pipeline의 Groovy 코드는 기본적으로 Sandbox 안에서 실행됩니다. Sandbox는 허용된 API 목록(whitelist)을 유지하고, 코드가 목록에 없는 API를 호출하면 실행을 차단합니다. `sh`, `echo`, `stage`, `parallel` 같은 Pipeline DSL 기본 기능은 허용되지만, `java.lang.Runtime.exec()`, `System.exit()`, `jenkins.model.Jenkins.getInstance()` 같은 시스템 레벨 API는 차단됩니다.

- **Script Approval 프로세스**: 차단된 API가 정당한 용도로 필요한 경우, 관리자가 "Manage Jenkins > In-process Script Approval"에서 해당 메서드 시그니처를 승인할 수 있습니다. 승인된 메서드는 허용 목록에 추가되어 이후 모든 Pipeline에서 사용 가능해집니다. 이것이 왜 위험할 수 있느냐면, 한 번 승인된 메서드는 모든 Pipeline에서 사용 가능하므로, 특정 팀만을 위해 승인한 메서드가 다른 팀에 의해 악용될 수 있기 때문입니다.

- **Sandbox가 없다면 어떤 일이 벌어지는가**: 개발자가 Jenkinsfile에 악의적인 코드를 넣으면 즉시 실행됩니다. Jenkins Master의 파일 시스템에 접근하여 크레덴셜 파일을 외부로 전송하거나, 내부 네트워크를 스캔하거나, 암호화폐 채굴 프로세스를 실행할 수 있습니다. Multibranch Pipeline에서 외부 기여자의 Pull Request를 자동 빌드하는 경우 이 위험은 더욱 커집니다.



## 4. CPS 변환 — Pipeline Groovy가 일반 Groovy와 다른 지점

> 이미 아는 "일반 함수 실행"의 *중단·재개 가능* 측면입니다. 보통 함수는 한 번에 끝까지 달리지만, Pipeline 함수는 중간에 멈췄다 이어 달릴 수 있어야 합니다.

Pipeline Groovy는 CPS(Continuation-Passing Style) 변환을 거쳐 실행 상태를 `program.dat`에 직렬화합니다. 이렇게 해야 Jenkins가 재기동되어도 빌드를 멈춘 지점부터 이어갈 수 있습니다(출처: jenkins.io/doc/book/pipeline/cps-method-mismatches).

CPS를 **이어달리기 경주**에 비유하면 이해가 빠릅니다. 한 선수가 트랙 전체를 도는 대신, 구간마다 다음 주자에게 바통을 넘깁니다. 바통(=continuation)은 "지금까지 어디를 달렸고 다음에 무엇을 할지"를 담은 상태이고, Jenkins는 그 바통을 디스크에 적어 두었다가 재기동 후 다음 주자에게 다시 건넵니다. 이 비유는 *상태를 쪼개 넘긴다*는 지점까지만 들어맞고, **`@NonCPS`에서 깨집니다**. `@NonCPS`로 표시한 메서드는 바통을 쪼개지 않고 단숨에 네이티브 Groovy로 달리므로 성능은 좋지만, 그 안에서는 바통을 넘겨받아야 동작하는 `node`·`sh` 같은 Pipeline step을 호출할 수 없습니다(`expected to call WorkflowScript.X` 경고).

CPS 변환이 깨뜨리는 대표 사례는 다음과 같습니다(출처: jenkins.io/doc/book/pipeline/cps-method-mismatches).

| 깨지는 패턴 | 증상 | 회피 |
| ----------- | ---- | ---- |
| `@NonCPS` 메서드 내부에서 `sh`/`node` 호출 | `expected to call WorkflowScript.X` | step 호출은 CPS 메서드로 빼냄 |
| 클래스 생성자에서 CPS 코드 실행 | 생성자는 CPS 변환 불가 | 로직을 별도 메서드로 분리 |
| CPS closure를 `Iterable.toSorted` 등 non-CPS 메서드에 전달 | 예상과 다른 정렬 결과 | `@NonCPS` 메서드에서 일반 closure만 사용 |
| `"${-> var}"` 지연 평가 GString | closure가 CPS라 직렬화 실패 | 즉시 평가 `"${var}"` 사용 |

```groovy
// ❌ @NonCPS 안에서 Pipeline step 호출 — 직렬화 경계를 넘으려다 실패
@NonCPS
def listFiles() {
    sh 'ls -al'   // why: @NonCPS는 네이티브 실행이라 step 바통을 못 받습니다
}

// ✅ 순수 계산만 @NonCPS로, step 호출은 CPS 본문에 둡니다
@NonCPS
def parse(String json) {
    // 정규식·문자열 가공처럼 직렬화 지점이 없는 계산만 — 성능 이득만 취함
    return new groovy.json.JsonSlurper().parseText(json)
}
```

직렬화 불가 객체도 같은 뿌리의 문제입니다. CPS는 step 사이마다 지역 변수 상태를 직렬화하므로, `Matcher`·`InputStream`처럼 직렬화할 수 없는 객체를 step 호출을 가로질러 변수에 들고 있으면 `NotSerializableException`이 납니다. 정규식 매칭 결과는 변수에 담아 두지 말고 그 자리에서 소비하거나, 매칭 블록 전체를 `@NonCPS` 메서드로 감싸 직렬화 경계를 넘지 않게 합니다.

## 정답

> 위 질문을 스스로 설명해 본 뒤에 펼치세요.

### 정답 1 — "할 수 있다"와 "해야 한다"의 구분

명령적 Groovy는 무제한 권한 탓에 보안 사고·운영 장애의 원인이 되고, 코드 diff는 의도 파악이 어려워 유지보수 부채가 쌓입니다. 그래서 선언적으로 풀리면 선언적으로 푸는 것이 원칙입니다. 우선순위는 JCasC → (Pipeline 공통 로직이면) Shared Library → init.groovy.d → Script Console 순으로, 위에서 막힐 때만 한 단계씩 명령적 수단으로 내려갑니다.

### 정답 2 — 공급망 공격 벡터

`new GroovyShell().evaluate(new URL("...").text)`처럼 외부 URL의 코드를 받아 실행하면, 그 외부 서버가 해킹당하는 순간 Jenkins가 시작될 때마다 공격자가 심은 악성 코드가 그대로 실행됩니다. 이는 전형적인 공급망 공격(Supply Chain Attack) 벡터로, init.groovy.d는 Master 권한으로 무제한 실행되므로 피해가 시스템 전체에 미칩니다. 스크립트는 외부에서 받지 말고 이미지·저장소에 고정해 버전 관리해야 합니다.

### 정답 3 — Sandbox와 Script Approval

Sandbox는 허용 목록(whitelist)에 없는 API 호출을 차단합니다. `sh`·`echo`·`stage`는 허용하지만 `Runtime.exec()`·`System.exit()`·`Jenkins.getInstance()` 같은 시스템 레벨 API는 막습니다. 차단된 API가 정당하게 필요하면 관리자가 "In-process Script Approval"에서 그 메서드 시그니처를 승인합니다. 부작용은, 한 번 승인된 메서드는 **모든 Pipeline**에서 쓸 수 있게 되므로 특정 팀을 위해 승인한 것을 다른 팀이 악용할 수 있다는 점입니다.

### 정답 4 — CPS 변환과 @NonCPS의 제약

Pipeline Groovy는 CPS 변환으로 실행 상태를 `program.dat`에 직렬화하여, Jenkins가 재기동되어도 멈춘 지점부터 빌드를 이어 갈 수 있게 합니다. `@NonCPS`는 이 변환을 끄고 네이티브 Groovy로 실행하므로 성능 이득이 있지만, 직렬화 경계를 넘나드는 `sh`·`node` 같은 step을 그 안에서 호출하면 `expected to call WorkflowScript.X` 경고와 함께 실패합니다. 그래서 `@NonCPS`에는 정규식·문자열 가공처럼 직렬화 지점이 없는 순수 계산만 두고, step 호출은 CPS 본문에 남겨야 합니다(출처: jenkins.io/doc/book/pipeline/cps-method-mismatches).

### 빈칸 정답 — CPS와 선택 우선순위

1. CPS(Continuation-Passing Style) / 직렬화
2. `sh`(또는 `node` 등 Pipeline step)
3. Shared Library / Script Console
4. 직렬화(Serializable)
