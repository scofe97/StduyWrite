# Jenkins Controller JVM Tuning Lab — 기획·로드맵

---

> 이 문서는 코드가 아니라 **계획**입니다. Jenkins Controller의 JVM을 직접 튜닝하는 실습 랩을 만들기 전에, 무엇을 왜 만들고 어떤 순서로 이어갈지를 한 장에 정리합니다. 실제 docker-compose·스크립트·측정 결과는 이 로드맵의 다음 단계에서 채웁니다.



## 한 줄 정의

부하를 걸어 Jenkins Controller의 JVM 병목을 **재현**하고, GC 로그와 thread·heap dump로 **진단**하고, 플래그를 바꿔 같은 부하의 **before/after를 측정**하는 재현 가능한 실험 랩입니다. 목표는 "힙을 키웠더니 빨라졌다" 같은 무용담이 아니라, 같은 부하를 두 JVM 설정에 걸어 숫자로 비교하는 **재현 가능한 측정 루프**를 갖추는 것입니다. 측정 없는 튜닝은 감이고, 한 번만 재본 튜닝은 운입니다.



## 왜 이 랩인가

이 학습 묶음의 가장 큰 공백이 JVM 튜닝입니다. 도메인 허브 README의 격차 분석에서 JVM 튜닝(#12)은 △로 남아 있고, 그 안에서도 Thread dump·Metaspace·Safepoint는 전수 grep 기준 0편으로 확인됐습니다. 단어조차 등장하지 않는다는 뜻입니다. 키워드를 스쳐 지나가는 게 아니라 *주제로* 다루는 편이 아직 없습니다.

기존 두 편이 이 자리를 비워 둔 채 서로를 가리키고 있다는 점이 결정적입니다. `06_infra/05-07`은 성능 지표를 수집하고 부하를 실측하지만, 본문에서 스스로 "JVM *튜닝*은 06-01 §4가 다루고, 이 편은 그 튜닝이 효과가 있었는지 확인하는 *측정* 을 다룬다"고 선을 긋습니다. `06_infra/01-01 §4`는 용량을 산정하면서 어떤 플래그가 어떤 증상을 막는지 나열하는 데서 멈춥니다. 그 사이, **증상 재현 → 덤프 채취 → 판독 → 플래그 변경 → 재측정 → before/after** 라는 튜닝 루프 자체를 손으로 도는 편이 없습니다. 이 랩이 정확히 그 빈자리에 들어갑니다.

이력서 측면에서도 들어맞습니다. Jenkins는 현재 자기평가에서 가장 약한 항목(■■□□□)인데, "Controller JVM을 부하로 재현해 GC pause와 처리량을 N% 개선했다"는 측정 가능한 한 줄은 약점을 성과로 뒤집습니다. 재현 가능한 docker-compose 랩을 공개 저장소로 함께 두면 주장에 근거가 붙습니다.



## 다룰 키워드 체크리스트

도메인 허브 README에서 0편 또는 1편으로 확인된 항목을 직격합니다. 각 항목을 "왜 보는가"와 함께 둡니다 — 키워드를 채우는 게 목적이 아니라, 그 신호가 어떤 판단을 가능하게 하는지가 핵심이기 때문입니다.

| 키워드 | 왜 보는가 |
|--------|----------|
| GC 로그 판독 | pause 길이와 빈도로 힙 부족·Full GC·승격 실패를 구분합니다 |
| Thread dump 채취·해석 | 빌드 행·데드락·블로킹 지점을 스레드 상태로 직접 봅니다 |
| Heap dump histogram | 어떤 객체가 힙을 점유하는지로 누수·과적재를 짚습니다 |
| Metaspace 상한 | 플러그인·클래스 증가가 메타스페이스를 어떻게 압박하는지 봅니다 |
| Safepoint | pause가 GC 때문인지 safepoint 동기화 때문인지 가립니다 |
| G1 파라미터 튜닝 | 힙 크기·MaxGCPauseMillis 같은 손잡이로 pause를 조율합니다 |
| before/after 측정 | 같은 부하 두 회차의 수치 차이로 튜닝의 효과를 증명합니다 |



## 이어갈 작업 로드맵

처음부터 크게 만들지 않고 세 단계로 자릅니다. 코드가 먼저 나오고, 그 측정 결과가 학습편을 낳는 순서입니다. 이번 세션은 이 표를 정리하는 데서 멈추고, 실행은 다음 세션이 1차부터 이어갑니다.

| 단계 | 할 일 | 산출 | 재사용 |
|:--:|------|------|--------|
| 1차 | 랩 골격을 세웁니다. docker-compose에 GC 로깅 플래그(`-Xlog:gc*,safepoint`)와 baseline/tuned 프로파일 토글을 넣고, prometheus·grafana로 관측 스택을 붙이고, 부하·채취·비교 스크립트를 둡니다 | 동작하는 랩 | 부하원은 `05-07 §6-3`의 crumb+200건 트리거 스크립트 |
| 2차 | 진단·튜닝 루프를 한 번 완주합니다. baseline 프로파일로 Full GC·긴 pause·Metaspace 압박을 재현하고, tuned 프로파일로 같은 부하를 재측정합니다 | `RESULTS.md`의 before/after 수치 | `05-07 §6` 부하 하네스 |
| 3차 | 실측을 학습편으로 정리합니다. `06_infra/01-04.…JVM 튜닝·진단….md` 신규 1편을 저작합니다 | 학습 `.md` 1편 | 본문 근거는 `RESULTS.md` |

> 1차의 baseline 프로파일은 힙을 일부러 작게(예: `-Xmx512m`) 두어 병목을 끌어내고, tuned 프로파일은 힙 상향과 G1 파라미터, Metaspace 상한으로 대응합니다. 두 프로파일에 *같은* 부하를 거는 게 비교의 전제입니다.



## 중복 회피 메모

이 랩은 *튜닝·진단 루프*만 다룹니다. 측정 토폴로지·지표 카탈로그·부하 하네스는 이미 `05-07`이 다루므로 그 편으로 링크하고, 용량 추정식은 `01-01 §4`로 링크합니다. 같은 내용을 다시 쓰지 않는 것이 격차를 메우는 신규 편의 조건입니다. 3차 학습편을 쓸 때 이 원칙을 게이트로 둡니다 — 측정·산정을 또 설명하고 있으면 범위를 벗어난 것입니다.



## 이력서 연결

2차에서 before/after 수치가 확보되면 이력서 한 줄로 직결됩니다.

> Jenkins Controller JVM을 부하로 재현하고 GC 로그·thread dump로 진단해, G1 튜닝으로 GC pause와 처리량을 N% 개선 (재현 가능한 docker-compose 랩 + GitHub 공개)

수치 N은 2차 `RESULTS.md`가 채웁니다. 공개 저장소 링크가 주장의 근거가 되므로, 랩은 처음부터 남이 그대로 돌려볼 수 있게 만듭니다.
