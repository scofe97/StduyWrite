# 개발자에게 인프라가 다가온 시대

---

> **"인프라는 인프라 팀이 하는 것"이라는 경계가 왜 무너졌는지를 DevOps 문화·클라우드 네이티브 전환·풀스택 확장 세 축으로 설명할 수 있고, 그 개발자가 인프라를 배우는 출발점으로 왜 쿠버네티스가 적합한지(특정 클라우드에 묶이지 않는 공통 언어)를 답할 수 있으며, 그 넓은 생태계를 혼자 감당하기 어려울 때 터미널형 AI가 새로운 동료가 된다는 흐름까지 말할 수 있다.**

몇 년 전까지 인프라는 인프라 팀의 일이었습니다. 개발자는 코드를 짜고, 인프라 팀이 서버를 만들고, 운영 팀이 배포했습니다. 각자의 영역이 명확했고 경계를 넘을 일이 거의 없었습니다. 지금은 다릅니다. 클라우드와 컨테이너가 보편화되면서 개발자가 직접 인프라를 다루는 일이 늘었고(§1.1), 그 출발점으로 쿠버네티스가 자리를 잡았으며(§1.2), 넓은 생태계를 혼자 감당하기 어려운 자리에 AI 도구가 들어옵니다(§1.2.3).



## DevOps 문화의 확산

> "만든 사람이 운영한다"는 원칙이 조직의 기본이 되면서, 개발자의 책임이 코드 작성에서 배포·운영까지 늘어났습니다.

"You build it, you run it" — 만든 사람이 운영한다는 원칙이 많은 조직의 기본이 되었습니다. 개발자가 코드를 작성하고 끝나는 것이 아니라, 그 코드가 어떻게 배포되고 운영되는지까지 책임지는 흐름입니다.

이 변화가 개발자에게 남기는 실질적인 부담은 단순합니다. 배포 자동화와 CI/CD 파이프라인이 일상이 되면, 개발자는 자기 코드가 어떤 경로로 배포되는지를 직접 파악하고 있어야 합니다. 장애가 났을 때 "운영 팀이 알아서 하겠지"로 넘길 수 없고, 로그와 모니터링을 직접 들여다보며 원인을 좁혀야 합니다. 책임이 한쪽으로 몰리지 않고 만든 사람에게 따라붙는 구조이기 때문입니다.



## 클라우드 네이티브 전환

> 인프라를 콘솔에서 클릭하던 시대에서, 코드로 선언하고 깃에 커밋하면 파이프라인이 적용하는 시대로 바뀌었습니다.

AWS·GCP·Azure 위에서 인프라를 코드로 정의하는 것이 표준이 되었습니다. 서버를 수동으로 세팅하는 대신 YAML이나 테라폼(Terraform)으로 선언하고, 깃(Git)에 커밋하면 자동화된 파이프라인이 그 정의를 실제 환경에 적용합니다. 콘솔에 로그인해 버튼을 클릭하던 방식에서, 인프라를 코드로 적어 두는 방식으로 옮겨 온 셈입니다.

이 전환이 주는 이점은 두 가지로 정리됩니다. 첫째, 누가 언제 무엇을 배포했는지 히스토리가 깃에 남습니다. 둘째, 같은 설정을 다른 환경에 복제하기 쉽습니다. 그리고 코드로 관리되는 범위가 한 군데에 그치지 않습니다. 쿠버네티스 매니페스트(Kubernetes Manifest), 헬름(Helm) 차트, CI/CD 파이프라인, 모니터링 설정, 정책까지 모두 '코드로' 관리됩니다. 인프라의 거의 모든 면이 텍스트 파일로 선언되고 버전 관리된다는 뜻입니다.



## 풀스택의 범위 확장

> '풀스택'의 의미가 프런트·백엔드를 넘어 배포 파이프라인과 인프라까지 이해하는 것으로 넓어졌습니다.

'풀스택 개발자'라는 말의 범위가 달라졌습니다. 프런트엔드에서 백엔드를 넘어, 배포 파이프라인과 인프라까지 이해하는 시대입니다. API를 만드는 것으로 끝나지 않고, 그것이 어떻게 배포되고 모니터링되는지까지 시야에 들어와야 합니다.

그래서 개발자가 마주하는 질문이 달라집니다.

- "이 서비스를 어떻게 배포하지?"
- "장애가 나면 어떻게 알지?"
- "트래픽이 늘면 어떻게 대응하지?"

결국 개발자도 인프라를 알아야 합니다. 문제는 어디서부터 시작하느냐입니다. 이 세 질문이 자연스럽게 다음 주제로 이어집니다 — 배포·장애 대응·트래픽 라우팅이라는 공통 문제를, 어느 클라우드에서든 같은 언어로 다루게 해 주는 도구가 쿠버네티스이기 때문입니다.



## 왜 쿠버네티스가 출발점인가

> 쿠버네티스는 특정 클라우드에 묶이지 않습니다. EKS·GKE·AKS·온프레미스 어디서나 같은 쿠버네티스라, 한 번 익히면 어디서든 통합니다.

개발자가 인프라를 배우려 할 때 쿠버네티스가 좋은 출발점인 첫 번째 이유는 이식성입니다. AWS EKS·GCP GKE·Azure AKS는 모두 같은 쿠버네티스이고, 온프레미스 데이터센터에서도 동작합니다. 클라우드와 온프레미스를 막론하고, 쿠버네티스 위에 올린 워크로드는 동일한 방식으로 운영됩니다. 회사가 AWS에서 GCP로 옮기거나 온프레미스로 내려와도, 쿠버네티스를 알고 있다면 운영 방식이 크게 달라지지 않습니다.

리눅스를 이해하면 어떤 서버든 다룰 수 있는 것과 같은 이치입니다. 쿠버네티스를 이해하면 어떤 클라우드든 다룰 수 있습니다. 더 중요한 것은, 쿠버네티스에서 다루는 개념이 인프라의 공통 어휘라는 점입니다. 배포·라우팅·설정·확장·관찰이라는 공통 문제를 쿠버네티스는 오브젝트라는 같은 단어로 다룹니다.

| 오브젝트 | 답하는 질문 |
|----------|------------|
| Deployment·Pod | 애플리케이션을 어떻게 배포하고 실행하는가 |
| Service·Ingress | 트래픽을 어떻게 라우팅하는가 |
| ConfigMap·Secret | 설정과 민감 정보를 어떻게 관리하는가 |
| HPA·리소스 제한 | 규모를 어떻게 조절하는가 |
| 모니터링·로깅 | 운영 중인 시스템을 어떻게 관찰하는가 |

AWS ECS도, GCP Cloud Run도, Azure Container Apps도 컨테이너 기반 인프라라면 동일한 문제를 다룹니다. 쿠버네티스에서 "Deployment의 replicas를 3으로 설정하고, 롤링 업데이트로 배포하고, HPA로 트래픽을 받아 오토스케일"한 경험은 다른 플랫폼에서도 같은 개념을 빠르게 파악하게 해 줍니다. 쿠버네티스를 이해하면 클라우드 인프라 전체를 보는 눈이 생긴다는 말은 이런 의미입니다.



## 가까운 학습 곡선과 AI라는 새로운 동료

> 쿠버네티스 생태계는 넓어 "무엇을 고를 것인가"가 학습의 부담이 됩니다. 그 빈자리를 터미널형 AI가 채웁니다.

다만 쿠버네티스는 넓습니다. CNCF(Cloud Native Computing Foundation)의 Landscape를 한 번이라도 펼쳐 본 분이라면 알 것입니다. 수백 개의 프로젝트가 빼곡히 들어차 있고, 영역마다 선택지가 넘쳐납니다. 실제로 클라우드를 운영하려면 결정해야 할 것이 한두 가지가 아닙니다.

- 모니터링: Prometheus vs Datadog vs New Relic vs Grafana Cloud
- 로그 수집: ELK 스택(Elasticsearch + Logstash + Kibana) vs Loki + Fluent Bit
- 배포 전략: Rolling Update vs Blue/Green vs Canary
- GitOps: ArgoCD vs Flux
- 시크릿 관리: Sealed Secrets vs External Secrets vs CSI Driver
- 메시징: Kafka vs RabbitMQ vs NATS
- 트래픽 관리: Ingress vs Gateway API

도구 수를 아는 것도 일이지만, 그 사이에서 트레이드오프를 따져 고르는 게 더 어렵습니다. 1~2명이 감당하기엔 벅찬 선택·설치·설정·검증의 연속입니다.

이 질문에 답하기 어려울 때, 터미널에서 직접 동작하는 에이전트형 AI(클로드 코드 같은)가 새로운 동료가 됩니다. CNCF Landscape의 수백 개 프로젝트를 직접 조사할 필요 없이 "우리 상황에는 어떤 게 맞아?"라고 묻고 "그걸로 진행해줘"라고 하면, 1~2명이 감당하기 어려웠던 도구 선택·설치·설정·검증의 과정을 AI가 대화하면서 빠르게 풀어 갑니다. 그래서 개발자가 인프라를 '공부한 다음에 하는 것'이 아니라, 하면서 배우는 것으로 바뀝니다. 이 책 전체가 그 방식 — 클로드 코드와 대화하며 쿠버네티스 인프라를 처음부터 끝까지 구축하는 흐름 — 을 따릅니다.



## 면접 관점 정리

> 변화의 배경(세 축) → 학습 출발점(쿠버네티스) → 그 부담을 더는 도구(AI) 순으로 한 호흡에 엮으면 됩니다.

| 단계 | 핵심 | 한 줄 답 |
|------|------|---------|
| 왜 개발자가 인프라를 다루나 | DevOps·클라우드 네이티브·풀스택 확장 | 책임이 만든 사람에게 따라붙고, 인프라가 코드로 선언되며, 풀스택의 정의가 넓어졌다 |
| 왜 쿠버네티스부터인가 | 이식성 + 공통 어휘 | EKS·GKE·AKS·온프레미스가 같은 쿠버네티스라, 한 번 익힌 오브젝트 개념이 어디서든 통한다 |
| 그 넓이를 어떻게 감당하나 | CNCF Landscape의 선택지 폭발 | 터미널형 AI에게 묻고 검증하며, 공부한 뒤가 아니라 하면서 배운다 |

세 단계는 따로 떨어진 트렌드가 아니라 같은 변화의 연속입니다. 개발자에게 인프라 책임이 따라붙고(§1.1), 그 책임을 감당할 공통 언어가 쿠버네티스이며(§1.2), 그 언어의 넓은 생태계를 AI와 함께 다루게 됩니다(§1.2.3). 이 마지막 조각, 즉 "AI와 함께 인프라를 선언적으로 관리한다"는 발상이 다음 절 [§1.3 GitOps에서 GitAIOps로](./01-02.GitOps에서%20GitAIOps로.md)의 출발점입니다.
