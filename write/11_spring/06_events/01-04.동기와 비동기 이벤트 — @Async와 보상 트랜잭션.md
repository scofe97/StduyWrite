# 동기와 비동기 이벤트 — @Async와 보상 트랜잭션
---

> 앞 세 문서가 "리스너가 *언제* 실행되는가"(Phase)를 다뤘다면, 이 문서는 "리스너가 *어느 스레드에서* 실행되는가"를 다룹니다. 기본 이벤트 처리는 동기라 발행자와 같은 스레드에서 돕니다. `@Async` 를 붙이면 별도 스레드로 떨어져 응답이 빨라지지만, 그 대가로 트랜잭션과 예외 전파가 끊깁니다. 이 문서를 다 읽으면 "후속 처리를 동기로 둘까 비동기로 둘까"를 트랜잭션 경계와 보상 필요성으로 판단할 수 있게 됩니다.

## 1. 동기 처리 — 하나의 트랜잭션, 전부 아니면 전무

기본 이벤트 처리는 발행자와 같은 스레드·같은 트랜잭션에서 동기로 실행됩니다. 모든 처리가 하나의 트랜잭션에 묶이므로, 중간에 하나라도 실패하면 전부 롤백됩니다.

```text
[Client 요청]
     │
     ▼
[Service: createTicketWithWorkflow]
     │
     ├── Ticket 생성 (ticketComponent.create)
     │
     ├── Workflow 생성 (workflowComponent.createInitial)
     │
     ▼
[DB 커밋]
     │
     └─> 응답 반환
```

모든 처리가 성공해야 하나의 트랜잭션으로 커밋되고, 하나라도 실패하면 전체가 롤백됩니다. 티켓과 워크플로우가 *반드시 함께 존재해야 하는* 관계라면 이 방식이 맞습니다. 데이터 정합성이 가장 강하게 보장됩니다.

외부 시스템 연동을 동기로 두면, 연동 결과까지 기다린 뒤에 응답이 나갑니다.

```text
[HTTP Request: /workflow/reject]
      │
      ▼
[TicketFlowModule.rejectWorkflow()]
  ├─ workflowComponent.reject()
  └─ publishEvent(new WorkflowRejectedEvent)
                      │
                      ▼
         [@EventListener (동기)]
             onRejected(event)
                 ├─ ResetHandler.handle()
                 │     ├─ resetByWorkflow()
                 │     └─ sendResetNotification()  ← ★ 외부 연동 결과까지 대기
                 └─ auditLogger.log(...)
                      │
                      ▼
       전체 흐름 완료 후 HTTP 응답 반환
```

이 방식은 외부 연동까지 모두 끝난 뒤 클라이언트에 응답하므로, 외부 시스템 오류가 나면 예외를 던져 트랜잭션을 롤백할 수 있습니다. 외부 응답값을 받아 후속 분기에 바로 쓸 수도 있습니다.

```java
boolean result = notificationService.send(...);
if (!result) throw new RuntimeException("알림 실패");
```

대신 외부 연동이 느리면 그만큼 응답도 느려집니다. 정합성을 위해 성능을 내준 셈입니다.

## 2. 비동기 처리 — @Async로 트랜잭션 분리

핵심 로직만 먼저 커밋하고, 후속 로직을 별도 스레드에서 처리하려면 `@Async` 를 붙입니다. 발행자 트랜잭션은 이벤트 발행 직후 종료되고, 리스너는 새 스레드에서 따로 돕니다.

```text
[Client 요청]
     │
     ▼
[Service: createTicket]
     │
     ├── Ticket 생성 (ticketComponent.create)
     │
     ├── publishEvent(TicketCreatedEvent)
     │
     ▼
[DB 커밋] ← (여기까지만 트랜잭션)
     │
     └─> 응답 반환

(비동기)
     ▼
@Async EventListener
  handleTicketCreated
     │
     ├── workflowComponent.createInitial
     ▼
   별도 트랜잭션
```

티켓은 반드시 생성되고, 워크플로우 생성이 실패해도 티켓은 남습니다. 느슨한 결합과 빠른 응답을 얻는 대신, 워크플로우 실패를 직접 감지해 보상 처리를 해야 합니다.

> 중요한 함정: `@Async` 로 스레드가 바뀌면 [01-03](01-03.@TransactionalEventListener%20내부%20동작%20원리.md)에서 본 `ThreadLocal` 기반 트랜잭션 컨텍스트가 *끊깁니다*. 발행자 트랜잭션의 동기화 정보는 발행 스레드의 ThreadLocal에 있으므로, 새 스레드의 리스너는 그것을 보지 못합니다. 그래서 `@Async` 리스너는 발행자 트랜잭션에 참여할 수 없고, 필요하면 자기 `@Transactional` 로 새 트랜잭션을 시작해야 합니다.

`@Async` 와 `@EventListener` 를 함께 쓰면 비동기 리스너가 됩니다. 별도 스레드라 발행자에 예외가 전파되지 않으므로, 실패는 리스너 안에서 잡아 직접 처리해야 합니다.

```java
@Async
@EventListener
public void handleTicketCreated(TicketCreatedEvent event) {
    try {
        workflowComponent.createInitial(event.ticketId());
        historyLogger.log("WORKFLOW_CREATED", event.ticketId(), "SUCCESS");
    } catch (Exception ex) {
        // 실패 이력 기록
        historyLogger.log("WORKFLOW_CREATE_FAILED", event.ticketId(), ex.getMessage());

        // 보상 처리 등록 (e.g. 재시도 큐, 관리자 알림 등)
        compensationService.enqueueCompensation(event.ticketId());
    }
}
```

## 3. 보상 트랜잭션 — 비동기 실패를 메우는 장치

비동기로 분리하면 발행자 트랜잭션이 후속 실패를 책임지지 않습니다. 그래서 후속 처리가 실패했을 때 시스템을 일관된 상태로 되돌리거나 다시 시도하는 *보상*이 필요합니다. 가장 단순한 보상은 주기적 재시도입니다.

```java
@Component
@RequiredArgsConstructor
public class CompensationService {
    private final WorkflowComponent workflowComponent;

    // 단순 재시도 처리
    public void enqueueCompensation(Long ticketId) {
        // DB나 Queue에 저장해놓고 배치 혹은 트리거로 처리
    }

    @Scheduled(fixedDelay = 30000)
    public void retryFailedWorkflows() {
        List<Long> failedTicketIds = findFailedTicketIds();
        for (Long id : failedTicketIds) {
            try {
                workflowComponent.createInitial(id);
                log.info("보상 완료 - 워크플로우 생성됨: {}", id);
                markAsRecovered(id);
            } catch (Exception e) {
                log.warn("보상 실패 - 다음 주기로 재시도 예정: {}", id);
            }
        }
    }
}
```

`@Scheduled` 재시도가 AOP 프록시 기반으로 동작하는 메커니즘은 [`../05_aop/01-02.스프링 스케줄링`](../05_aop/01-02.스프링%20스케줄링%20—%20@Scheduled에서%20Quartz까지.md)에서 다룹니다.

보상 전략은 상황에 따라 네 가지로 나뉩니다.

| 패턴 | 설명 | 예 |
| --- | --- | --- |
| 재시도 | 실패한 작업을 나중에 다시 시도 | 워크플로우 생성 실패 → 주기적 재시도 |
| 반대작업 | 실패 시 기존 작업을 취소 | 워크플로우 실패 → 티켓 상태 롤백 |
| 관리자 개입 | 실패 이력을 관리자에게 알림 | 알림 시스템 연동 후 Slack, 이메일 등 |
| Dead Letter | 실패 이벤트를 DLQ로 보내기 | Kafka, SQS 기반 |

## 4. 동기와 비동기, 무엇을 고를까

선택의 기준은 "후속 처리가 핵심 데이터와 운명을 같이해야 하는가"입니다.

| 상황 | 처리 방식 | 보상 필요 여부 |
| --- | --- | --- |
| 티켓과 워크플로우 모두 반드시 있어야 함 | 동기 | ❌ |
| 티켓만 필수, 워크플로우는 후속 처리 | 비동기 이벤트 | ✅ 보상 처리 필요 |
| 후속 작업은 실패 가능성 존재, 유연성 필요 | 이벤트 + DLQ or Alert | ✅ 강하게 필요 |

정리하면, 두 작업이 원자적으로 묶여야 하면 동기(또는 `@TransactionalEventListener(BEFORE_COMMIT)`)로 한 트랜잭션에 담고, 핵심만 먼저 확정하고 후속을 분리하려면 비동기 + 보상으로 갑니다. 그리고 외부 연동처럼 "성공 확정 후 한 번만, 실패해도 핵심은 안전"이어야 하는 처리는 [01-01](01-01.스프링%20이벤트와%20두%20리스너%20—%20@EventListener%20vs%20@TransactionalEventListener.md)에서 본 `@TransactionalEventListener(AFTER_COMMIT)` 가 동기·비동기 사이의 안전한 절충이 됩니다.

## 5. 정리

| 축 | 동기 | 비동기 (@Async) |
|----|------|----------------|
| 스레드 | 발행자와 동일 | 별도 스레드 |
| 트랜잭션 | 발행자와 한 트랜잭션 | 끊김 (ThreadLocal 컨텍스트 분리) |
| 예외 전파 | 발행자까지 전파, 롤백 가능 | 전파 안 됨, 리스너 내부 처리 |
| 응답 속도 | 후속 완료까지 대기 (느림) | 즉시 응답 (빠름) |
| 실패 대응 | 트랜잭션 롤백 | 보상 트랜잭션 필요 |
| 적합한 경우 | 원자적으로 묶여야 하는 작업 | 핵심만 확정, 후속 분리 |
