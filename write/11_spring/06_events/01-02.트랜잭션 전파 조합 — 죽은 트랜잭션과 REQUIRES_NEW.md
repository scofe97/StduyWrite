# 트랜잭션 전파 조합 — 죽은 트랜잭션과 REQUIRES_NEW
---

> `@TransactionalEventListener(AFTER_COMMIT)` 안에서 DB에 무언가를 저장했는데 *조용히 사라지는* 함정이 있습니다. 이 문서는 그 "죽은 트랜잭션" 문제가 왜 일어나는지를 트랜잭션 전파 동작으로 설명하고, `REQUIRES_NEW` 조합이 어떻게 푸는지, 그리고 스프링이 최신 버전에서 이 함정을 어떻게 fail-fast로 막았는지를 정리합니다. 한 번 겪으면 디버깅에 한참을 쏟는 사고라, 원리를 미리 알아 두는 값이 큽니다.

## 1. 죽은 트랜잭션 문제

다음 두 코드의 차이는 `propagation` 한 줄뿐인데, 위는 DB 저장이 사라지고 아래는 정상 저장됩니다.

```java
// ❌ DANGER: DB 쓰기가 저장되지 않음!
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional(propagation = Propagation.REQUIRED)  // 기본값
public void saveLog(UserCreatedEvent event) {
    logRepository.save(new Log());  // 저장 안됨!
}

// ✅ CORRECT: 새 트랜잭션 생성으로 정상 저장
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveLog(UserCreatedEvent event) {
    logRepository.save(new Log());  // 정상 저장됨!
}
```

왜 위쪽이 실패하는지를 단계별로 펼치면 이렇습니다.

```text
1. UserService.saveUser() → @Transactional
   ├─ INSERT user
   └─ 메서드 종료 → 트랜잭션 커밋 시작

2. PlatformTransactionManager.commit()
   ├─ JDBC Connection.commit() 호출
   ├─ Connection.setAutoCommit(true)
   └─ 트랜잭션 상태 → COMMITTED

3. AFTER_COMMIT 리스너 실행
   └─ saveLog() 호출

4. saveLog() 내부 @Transactional(propagation = REQUIRED)
   ├─ TransactionSynchronizationManager.isActualTransactionActive() → true
   ├─ 기존 TransactionStatus 재사용 시도
   └─ 하지만 Connection은 이미 커밋됨 → 쓰기 무시됨
```

핵심은 타이밍입니다. `AFTER_COMMIT` 리스너는 트랜잭션이 커밋된 직후, 그러나 `cleanupAfterCompletion()` 으로 트랜잭션 컨텍스트가 *정리되기 전에* 실행됩니다. 그래서 `TransactionSynchronizationManager.isActualTransactionActive()` 가 아직 `true` 를 반환하고, `Propagation.REQUIRED` 는 "활성 트랜잭션이 있으니 거기 참여하자"고 판단합니다. 그런데 그 트랜잭션의 `Connection` 은 이미 커밋돼 버린 "죽은 트랜잭션"입니다. DB 쓰기는 실행되지만 커밋될 트랜잭션이 없어 그대로 사라집니다.

## 2. REQUIRES_NEW — 새 트랜잭션으로 탈출

`AFTER_COMMIT` 에서 DB 쓰기가 필요하다면 `REQUIRES_NEW` 를 조합해야 합니다. `REQUIRES_NEW` 는 기존 트랜잭션에 참여하지 않고 *항상 새 트랜잭션을 시작*하므로, 죽은 트랜잭션과 무관하게 자기 커밋을 갖습니다.

```java
@Service
public class UserService {
    @Transactional  // ← 이전 트랜잭션
    public User createUser(String name) {
        User user = userRepository.save(new User(name));
        eventPublisher.publishEvent(new UserCreatedEvent(user));
        return user;
        // ← 여기서 트랜잭션 커밋
    }
}

@Component
public class UserEventListener {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void afterCommit(UserCreatedEvent event) {
        // ← 여기는 이전 트랜잭션이 이미 커밋된 후
        // 이전 트랜잭션과 완전히 독립적
    }
}
```

일반적인 `@Transactional` 의 기본 전파는 `REQUIRED` 이고, 대부분의 경우 기존 트랜잭션에 참여하는 것이 올바른 동작입니다. 그러나 `@TransactionalEventListener(AFTER_COMMIT)` 는 "이미 커밋이 끝난" 특수한 시점이라, 같은 기본값(`REQUIRED`)을 그대로 쓰면 죽은 트랜잭션에 참여하는 사고가 납니다. 이 자리에서만큼은 `REQUIRES_NEW` 가 옳습니다.

## 3. 스프링의 fail-fast 검증 (6.2 / Boot 3.5)

이 함정은 *조용한 실패*라 발견이 어렵습니다. 스프링 프레임워크 팀은 문제를 인식하고([PR #33470](https://github.com/spring-projects/spring-framework/pull/33470)) 최신 버전에서 검증 로직을 추가했습니다. 이제 위험한 조합은 애플리케이션 시작 시점에 에러로 막힙니다.

```java
// ❌ 이제 런타임(스타트업) 에러 발생!
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional  // REQUIRED가 기본값
public void saveLog(Event e) {
    // 에러: "@TransactionalEventListener method must not be annotated with
    //       @Transactional unless when declared as REQUIRES_NEW or NOT_SUPPORTED"
}

// ✅ 명시적으로 REQUIRES_NEW 지정 - 정상 동작
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveLog(Event e) {
    // 정상 실행
}

// ✅ NOT_SUPPORTED도 허용 - 트랜잭션 없이 실행
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional(propagation = Propagation.NOT_SUPPORTED)
public void readOnly(Event e) {
    // 정상 실행 (읽기 전용)
}
```

검증에 걸리면 다음과 같은 메시지로 시작 자체가 실패합니다.

```text
Failed to process @EventListener annotation on bean with name 'dangerousListener':
@TransactionalEventListener method must not be annotated with @Transactional unless when declared as REQUIRES_NEW or NOT_SUPPORTED: public void com.example.transaction.listener.DangerousListener.handleUserEvent(com.example.transaction.event.UserEvent)
```

이 검증은 스프링 부트 3.5.0 이상, 스프링 프레임워크 6.2 이상에서 동작합니다. 변경의 의미를 정리하면 이렇습니다.

| 항목 | 변경 전 | 변경 후 |
| --- | --- | --- |
| 검증 범위 | 메서드 레벨 `@Transactional`만 | 클래스 레벨 `@Transactional`도 포함 (전체 커버) |
| 오류 처리 | Silent failure (런타임 버그) | Fail-fast (스타트업 시 에러, 조기 발견) |
| 트랜잭션 격리 | 깨질 수 있음 (이벤트 조기 처리) | 보장 (새 트랜잭션이나 비트랜잭션 실행) |
| 개발자 경험 | 디버깅 어려움 | 명확한 에러 메시지로 미스컨피그레이션 즉시 알림 |

조용히 데이터가 사라지던 버그가 시작 시점의 명확한 에러로 바뀌었다는 점이 핵심입니다. 옛 버전을 쓴다면 이 검증이 없으므로, 아래 조합 표를 직접 기억해 두어야 합니다.

## 4. Phase × 전파 조합 표

`@TransactionalEventListener` 와 `@Transactional` 을 함께 쓸 때, 전파 레벨별로 DB 쓰기가 되는지 정리하면 다음과 같습니다.

| 전파 레벨 | 트랜잭션 상태 | isActive | DB 쓰기 가능 여부 | 권장 사용 |
| --- | --- | --- | --- | --- |
| 없음 | 커밋된 TX 컨텍스트 활성 | true | ❌ 불가 (죽은 트랜잭션) | 읽기 전용 작업 |
| REQUIRED | 커밋된 TX에 참여 | true | ❌ 불가 (죽은 트랜잭션) | 사용 금지 |
| REQUIRES_NEW | 새 독립 TX 생성 | true | ✅ 가능 | DB 쓰기 필수 |
| NOT_SUPPORTED | TX 없음 | false | ❌ 불가 | 캐시/외부 API |

### REQUIRED (기본값) — 사용 금지

```java
// ❌ DANGER: 죽은 트랜잭션에 참여 - DB 쓰기가 저장되지 않음!
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional(propagation = Propagation.REQUIRED)
public void afterCommitWithRequired(UserCreatedEvent event) {
    // ⚠️ 이미 커밋된 트랜잭션 컨텍스트에 "참여"하게 됨
    // ⚠️ 이 저장은 실제로 커밋되지 않습니다!
    userLogRepository.save(new UserLog(event.getUser().getId()));  // 저장 안됨!
}
```

```text
[Service Transaction]
  └─ 커밋 완료 ★
      [TransactionSynchronizationManager는 아직 활성]

[AFTER_COMMIT Listener - REQUIRED]
  ├─ 기존 커밋된 트랜잭션 컨텍스트에 참여 시도
  ├─ userLogRepository.save()  ← 실행되지만 커밋 안됨!
  └─ ❌ DB 쓰기 손실!
```

### REQUIRES_NEW — DB 쓰기의 정답

```java
// ✅ CORRECT: 새로운 독립 트랜잭션 생성 - DB 쓰기가 정상적으로 저장됨
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void afterCommitWithRequiresNew(UserCreatedEvent event) {
    // 완전히 새로운 독립 트랜잭션 생성
    auditLogRepository.save(new AuditLog(event.getUser().getId()));  // 정상 저장!
}
```

`REQUIRES_NEW` 는 항상 새 트랜잭션을 사용하므로 DB 쓰기가 정상 커밋됩니다.

### BEFORE_COMMIT — REQUIRED가 정상

```java
@TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
@Transactional(propagation = Propagation.REQUIRED)
public void beforeCommitWithRequired(UserCreatedEvent event) {
    // 발행자의 트랜잭션에 참여
    statisticsRepository.increment("user_count");
}
```

`BEFORE_COMMIT` 은 *아직 커밋되지 않은* 활성 트랜잭션 안에서 실행되므로, `REQUIRED` 로 발행자 트랜잭션에 참여하는 것이 올바릅니다. 죽은 트랜잭션 문제는 `AFTER_COMMIT` 에서만 생긴다는 점을 기억하면 됩니다.

## 5. 실전 케이스

### 외부 API 호출 — 트랜잭션 불필요

외부 API 호출은 DB 쓰기가 없으므로 `@Transactional` 없이 `AFTER_COMMIT` 만으로 충분합니다. 발행자 트랜잭션과 독립이라, 외부 API가 실패해도 사용자 데이터는 안전합니다.

```java
// Service
@Transactional
public User createUser(String name) {
    User user = userRepository.save(new User(name));
    eventPublisher.publishEvent(new UserCreatedEvent(user));
    return user;
    // ← TX1 커밋 완료
}

// Listener
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void callExternalApi(UserCreatedEvent event) {
    // TX1과 완전히 독립
    // 외부 API 실패해도 User 데이터는 안전
    externalApi.notify(event.getUser());
}
```

### 로그 저장 — 새 트랜잭션 필요

후속 로그를 DB에 남겨야 한다면 `REQUIRES_NEW` 로 별도 트랜잭션(TX2)을 엽니다. TX1과 독립이므로 TX2가 실패해도 사용자 생성은 유지됩니다.

```java
// Service
@Transactional
public User createUser(String name) {
    User user = userRepository.save(new User(name));
    eventPublisher.publishEvent(new UserCreatedEvent(user));
    return user;
    // ← TX1 커밋 완료
}

// Listener
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveLog(UserCreatedEvent event) {
    // TX2 시작 (TX1과 독립)
    logRepository.save(new Log(event.getUser().getId()));
    // TX2 커밋
}
```

## 6. 정리

| 상황 | 올바른 선택 | 이유 |
|------|------------|------|
| AFTER_COMMIT + DB 쓰기 | `REQUIRES_NEW` | 죽은 트랜잭션 회피, 새 커밋 확보 |
| AFTER_COMMIT + 외부 API | 전파 지정 없음 | DB 쓰기 없어 트랜잭션 불필요 |
| AFTER_COMMIT + REQUIRED | 금지 | 죽은 트랜잭션 참여 → DB 쓰기 손실 |
| BEFORE_COMMIT + DB 쓰기 | `REQUIRED` | 아직 활성 트랜잭션이라 참여가 정상 |
| 위험 조합 방지 | Spring 6.2 / Boot 3.5+ | 시작 시점 fail-fast 검증 |
