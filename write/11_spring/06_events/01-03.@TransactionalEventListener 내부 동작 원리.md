# @TransactionalEventListener 내부 동작 원리
---

> 앞 문서들에서 `@TransactionalEventListener` 가 "발행 시점이 아니라 커밋 시점에" 실행된다고 했습니다. 어떻게 발행은 트랜잭션 한가운데서 일어나는데 실행은 커밋 뒤로 미뤄질까요? 이 문서는 그 뒤에서 도는 `TransactionSynchronization` 메커니즘을 스프링 내부 코드 수준으로 따라갑니다. 여기까지 이해하면 [01-02](01-02.트랜잭션%20전파%20조합%20—%20죽은%20트랜잭션과%20REQUIRES_NEW.md)의 "죽은 트랜잭션이 왜 아직 활성으로 보이는가"도 한 줄로 답할 수 있게 됩니다.

## 1. 전체 흐름 6단계

이벤트 발행부터 리스너 실행까지 스프링이 자동으로 처리하는 여섯 단계입니다. 핵심은 2~3단계(발행 시 *등록만*)와 5~6단계(커밋 시 *실행*)가 시간적으로 떨어져 있다는 점입니다.

```text
┌─────────────────────────────────────────────────────────────────┐
│ 1. 이벤트 발행                                                     │
│    applicationEventPublisher.publishEvent(event)                │
└─────────────────────┬───────────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. 이벤트 가로채기(Spring이 자동으로)                                  │
│    EventPublicationInterceptor catches event                    │
│    → Checks if transaction is active                            │
└─────────────────────┬───────────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. 트랜잭션용 리스너 등록(트랜잭션이 끝날 때 Spring이 불러준다고 약속          │
│    ApplicationListenerMethodTransactionalAdapter                │
│    → Wraps @TransactionalEventListener method                   │
│    → Registers with TransactionSynchronizationManager           │
└─────────────────────┬───────────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────────────┐
│ 4. 비즈니스 로직 실행 + 트랜잭션 종료                                   │
│    Business logic executes                                      │
│    → Commit or Rollback occurs                                  │
└─────────────────────┬───────────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────────────┐
│ 5. 트랜잭션 콜백 실행                                                │
│    AbstractPlatformTransactionManager                           │
│    → triggerBeforeCommit() for BEFORE_COMMIT                    │
│    → triggerAfterCommit() for AFTER_COMMIT                      │
│    → triggerAfterCompletion() for AFTER_ROLLBACK/COMPLETION     │
└─────────────────────┬───────────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────────────┐
│ 6. 실제 리스너 메서드 실행                                            │
│    ApplicationListenerMethodTransactionalAdapter                │
│    → Filters by Phase                                           │
│    → Invokes actual listener method                             │
└─────────────────────────────────────────────────────────────────┘
```

## 2. ThreadLocal — 스레드마다 독립된 트랜잭션 컨텍스트

`ThreadLocal` 은 각 스레드가 자신만의 독립적인 변수 사본을 가지는 저장소입니다.

```java
ThreadLocal<String> name = new ThreadLocal<>();
name.set("스레드A");  // 스레드A에서만 보임
```

`static` 변수라도 다른 스레드와 공유되지 않습니다. 트랜잭션은 본질적으로 스레드에 묶인 개념이라(한 요청 = 한 스레드 = 한 트랜잭션), 트랜잭션 이벤트 리스너도 스레드별로 독립적인 트랜잭션 컨텍스트를 가져야 합니다. 그래서 스프링은 트랜잭션 동기화 정보를 `ThreadLocal` 에 담아 스레드 간 간섭을 차단합니다. `ThreadLocal` 자체의 누수 위험과 사용 패턴은 [`../05_aop/01-03.템플릿·콜백과 ThreadLocal`](../05_aop/01-03.템플릿·콜백과%20ThreadLocal%20—%20AOP%20등장%20직전의%20두%20시도.md)에서 다룹니다.

## 3. TransactionSynchronization — 콜백 규약

`TransactionSynchronization` 은 "트랜잭션의 특정 시점에 불러 달라"는 콜백 인터페이스입니다. 각 메서드가 트랜잭션 Phase에 1:1로 대응합니다.

```java
// org.springframework.transaction.support.TransactionSynchronization
public interface TransactionSynchronization extends Ordered, Flushable {

    /** 완료 상태 코드 */
    int STATUS_COMMITTED = 0;
    int STATUS_ROLLED_BACK = 1;
    int STATUS_UNKNOWN = 2;

    /**
     * 트랜잭션 일시 중지 시 호출
     */
    default void suspend() {
    }

    /**
     * 트랜잭션 재개 시 호출
     */
    default void resume() {
    }

    /**
     * 트랜잭션 커밋 전 호출
     * - BEFORE_COMMIT Phase가 이 시점에 실행됨
     * - 예외 발생 시 전체 트랜잭션 롤백
     */
    default void beforeCommit(boolean readOnly) {
    }

    /**
     * 트랜잭션 완료 전 호출 (커밋 또는 롤백 전)
     */
    default void beforeCompletion() {
    }

    /**
     * 트랜잭션 커밋 후 호출
     * - AFTER_COMMIT Phase가 이 시점에 실행됨
     * - 예외 발생해도 트랜잭션은 이미 커밋됨
     */
    default void afterCommit() {
    }

    /**
     * 트랜잭션 완료 후 호출 (커밋 또는 롤백 후)
     * - AFTER_COMPLETION Phase가 이 시점에 실행됨
     * - AFTER_ROLLBACK Phase도 이 시점에 실행됨
     *
     * @param status 완료 상태 (STATUS_COMMITTED, STATUS_ROLLED_BACK, STATUS_UNKNOWN)
     */
    default void afterCompletion(int status) {
    }
}
```

01-01의 Phase 네 종류가 결국 이 인터페이스 메서드 호출로 환원됩니다. `BEFORE_COMMIT` 은 `beforeCommit()`, `AFTER_COMMIT` 은 `afterCommit()`, `AFTER_ROLLBACK` 과 `AFTER_COMPLETION` 은 `afterCompletion(status)` 안에서 `status` 값으로 갈립니다.

## 4. TransactionSynchronizationManager — ThreadLocal 등록소

`TransactionSynchronizationManager` 는 위 콜백들을 `ThreadLocal` 에 모아 두는 관리자입니다. 발행 시점에 리스너가 여기 등록되고, 커밋 시점에 여기서 꺼내 실행됩니다.

스레드 하나의 생애를 따라가면 이렇게 흐릅니다.

```text
[스레드 A 시작]
   │
   ├── @Transactional → initSynchronization()
   │     → synchronizations.set(new LinkedHashSet<>())
   │
   ├── publishEvent(...) → registerSynchronization(MyListener)
   │     → synchronizations.get().add(MyListener)
   │
   ├── DB 작업 완료 → commit()
   │     → getSynchronizations() → [MyListener]
   │     → triggerAfterCommit() → MyListener 실행
   │
   └── clearSynchronization() → ThreadLocal 제거
```

각 스레드가 자기 `Set` 을 가지므로 서로 간섭하지 않습니다.

```text
// 각 스레드마다 독립적인 트랜잭션 컨텍스트
Thread-1: synchronizations = [Listener1, Listener2, Listener3]
Thread-2: synchronizations = [Listener4, Listener5]
Thread-3: synchronizations = [Listener6]

// 스레드 간 간섭 없음
```

실제 구현은 `ThreadLocal<Set<TransactionSynchronization>>` 을 중심으로 등록·조회·초기화·정리 메서드를 제공합니다.

```java
// org.springframework.transaction.support.TransactionSynchronizationManager
public abstract class TransactionSynchronizationManager {

    // ThreadLocal 저장소: 현재 스레드의 트랜잭션 동기화 리스너 목록
    private static final ThreadLocal<Set<TransactionSynchronization>> synchronizations =
        new NamedThreadLocal<>("Transaction synchronizations");

    // 현재 트랜잭션 이름
    private static final ThreadLocal<String> currentTransactionName =
        new NamedThreadLocal<>("Current transaction name");

    // 트랜잭션 활성화 여부
    private static final ThreadLocal<Boolean> actualTransactionActive =
        new NamedThreadLocal<>("Actual transaction active");

    /**
     * 트랜잭션 동기화 등록
     */
    public static void registerSynchronization(TransactionSynchronization synchronization) {
        Assert.notNull(synchronization, "TransactionSynchronization must not be null");
        Set<TransactionSynchronization> synchs = synchronizations.get();

        if (synchs == null) {
            throw new IllegalStateException("Transaction synchronization is not active");
        }
        synchs.add(synchronization);
    }

    /**
     * 등록된 모든 동기화 리스너 조회 (정렬됨)
     */
    public static List<TransactionSynchronization> getSynchronizations() {
        Set<TransactionSynchronization> synchs = synchronizations.get();

        if (synchs == null) {
            throw new IllegalStateException("Transaction synchronization is not active");
        }

        // @Order를 고려한 정렬
        List<TransactionSynchronization> sortedSynchs = new ArrayList<>(synchs);
        OrderComparator.sort(sortedSynchs);
        return Collections.unmodifiableList(sortedSynchs);
    }

    /**
     * 트랜잭션 동기화 초기화
     */
    public static void initSynchronization() {
        if (synchronizations.get() != null) {
            throw new IllegalStateException("Cannot activate transaction synchronization - already active");
        }
        synchronizations.set(new LinkedHashSet<>());
    }

    /**
     * 트랜잭션 동기화 정리
     */
    public static void clearSynchronization() {
        synchronizations.remove();
    }
}
```

여기서 세 가지가 드러납니다. `ThreadLocal` 로 스레드마다 독립 컨텍스트를 유지하고, `Set` 자료구조라 한 트랜잭션에 여러 리스너를 등록할 수 있으며, `@Order` 기반으로 실행 순서를 제어합니다. 그리고 [01-02](01-02.트랜잭션%20전파%20조합%20—%20죽은%20트랜잭션과%20REQUIRES_NEW.md)에서 본 "죽은 트랜잭션이 아직 활성으로 보이는" 현상의 정체가 바로 이 `actualTransactionActive` ThreadLocal 이 `clearSynchronization()` 전까지 `true` 로 남아 있기 때문입니다.

## 5. ApplicationListenerMethodTransactionalAdapter — 어노테이션을 콜백으로 변환

`@TransactionalEventListener` 가 붙은 메서드는 그대로는 `TransactionSynchronization` 이 아닙니다. 스프링은 이 어댑터로 어노테이션 메서드를 콜백 인터페이스 구현체로 감쌉니다. 어댑터가 `phase` 를 읽어 두고, 각 콜백 메서드에서 자기 Phase일 때만 실제 리스너를 호출합니다.

```java
// org.springframework.transaction.event.ApplicationListenerMethodTransactionalAdapter
class ApplicationListenerMethodTransactionalAdapter
        extends ApplicationListenerMethodAdapter
        implements TransactionSynchronization {

		// 트랜잭션 단계
    private final TransactionPhase phase;

		// Phase 추출
    public ApplicationListenerMethodTransactionalAdapter(
            String beanName,
            Class<?> targetClass,
            Method method) {
        super(beanName, targetClass, method);

        // @TransactionalEventListener 어노테이션에서 Phase 추출
        TransactionalEventListener annotation =
            method.getAnnotation(TransactionalEventListener.class);
        this.phase = annotation.phase();
    }

    @Override
    public void onApplicationEvent(ApplicationEvent event) {
        // 트랜잭션이 활성화되어 있는지 확인
        if (TransactionSynchronizationManager.isSynchronizationActive()) {
            // TransactionSynchronization으로 등록
            TransactionSynchronizationManager.registerSynchronization(
                new TransactionSynchronizationEventAdapter(this, event)
            );
        }
        else if (this.annotation.fallbackExecution()) {
            // fallbackExecution=true면 트랜잭션 없이도 실행
            processEvent(event);
        }
    }

    /**
     * BEFORE_COMMIT Phase 처리
     */
    @Override
    public void beforeCommit(boolean readOnly) {
        if (this.phase == TransactionPhase.BEFORE_COMMIT) {
            processEvent(this.event);
        }
    }

    /**
     * AFTER_COMMIT Phase 처리
     */
    @Override
    public void afterCommit() {
        if (this.phase == TransactionPhase.AFTER_COMMIT) {
            processEvent(this.event);
        }
    }

    /**
     * AFTER_ROLLBACK 및 AFTER_COMPLETION Phase 처리
     */
    @Override
    public void afterCompletion(int status) {
        // 롤백된 경우
        if (status == STATUS_ROLLED_BACK) {
            if (this.phase == TransactionPhase.AFTER_ROLLBACK) {
                processEvent(this.event);
            }
            else if (this.phase == TransactionPhase.AFTER_COMPLETION) {
                processEvent(this.event);
            }
        }
        // 커밋된 경우
        else if (status == STATUS_COMMITTED) {
            if (this.phase == TransactionPhase.AFTER_COMPLETION) {
                processEvent(this.event);
            }
        }
    }

    /**
     * 실제 리스너 메서드 호출
     */
    protected void processEvent(ApplicationEvent event) {
        // condition 평가
        if (shouldHandle(event)) {
            // 실제 @TransactionalEventListener 메서드 호출
            invokeListener(event);
        }
    }
}
```

`onApplicationEvent()` 가 01-01에서 말한 "발행 시 등록"의 실체입니다. 트랜잭션 동기화가 활성이면 자기를 `TransactionSynchronizationManager` 에 등록하고, 트랜잭션이 없는데 `fallbackExecution=true` 면 그 자리에서 바로 실행합니다. `fallbackExecution` 기본값 `false` 의 의미가 여기서 드러납니다 — 트랜잭션이 없으면 등록도 실행도 안 하고 조용히 넘어갑니다.

## 6. AbstractPlatformTransactionManager — 콜백을 트리거하는 자리

마지막 퍼즐은 "누가 등록된 콜백을 실제로 부르는가" 입니다. 트랜잭션 매니저가 커밋·롤백을 처리하면서 적절한 시점에 콜백을 트리거합니다.

```java
// org.springframework.transaction.support.AbstractPlatformTransactionManager
public abstract class AbstractPlatformTransactionManager
        implements PlatformTransactionManager {

    /**
     * 커밋 처리
     */
    private void processCommit(DefaultTransactionStatus status) {
        try {
            boolean beforeCompletionInvoked = false;

            try {
                boolean unexpectedRollback = false;

                // 1. beforeCommit 콜백 호출
                prepareForCommit(status);
                triggerBeforeCommit(status);  // ← BEFORE_COMMIT 실행

                // 2. beforeCompletion 콜백 호출
                triggerBeforeCompletion(status);
                beforeCompletionInvoked = true;

                // 3. 실제 커밋 수행
                if (status.hasSavepoint()) {
                    status.releaseHeldSavepoint();
                }
                else if (status.isNewTransaction()) {
                    doCommit(status);  // ← 실제 DB 커밋
                }

                // 4. afterCommit 콜백 호출
                triggerAfterCommit(status);  // ← AFTER_COMMIT 실행
            }
            catch (UnexpectedRollbackException ex) {
                // 예기치 않은 롤백 발생
                triggerAfterCompletion(status, TransactionSynchronization.STATUS_ROLLED_BACK);
                throw ex;
            }
            catch (TransactionException ex) {
                // 커밋 실패 시 롤백
                if (isRollbackOnCommitFailure()) {
                    doRollbackOnCommitException(status, ex);
                }
                else {
                    triggerAfterCompletion(status, TransactionSynchronization.STATUS_UNKNOWN);
                }
                throw ex;
            }
            catch (RuntimeException | Error ex) {
                if (!beforeCompletionInvoked) {
                    triggerBeforeCompletion(status);
                }
                doRollbackOnCommitException(status, ex);
                throw ex;
            }

            // 5. afterCompletion 콜백 호출
            try {
                triggerAfterCompletion(status, TransactionSynchronization.STATUS_COMMITTED);
                // ← AFTER_COMPLETION 실행 (커밋 성공 시)
            }
            finally {
                cleanupAfterCompletion(status);
            }
        }
        finally {
            cleanupAfterCompletion(status);
        }
    }

    /**
     * 롤백 처리
     */
    private void processRollback(DefaultTransactionStatus status, boolean unexpected) {
        try {
            boolean unexpectedRollback = unexpected;

            try {
                // 1. beforeCompletion 콜백 호출
                triggerBeforeCompletion(status);

                // 2. 실제 롤백 수행
                if (status.hasSavepoint()) {
                    status.rollbackToHeldSavepoint();
                }
                else if (status.isNewTransaction()) {
                    doRollback(status);  // ← 실제 DB 롤백
                }
                else {
                    // 중첩 트랜잭션의 경우 rollback-only 마킹
                    if (status.hasTransaction()) {
                        doSetRollbackOnly(status);
                    }
                }
            }
            catch (RuntimeException | Error ex) {
                triggerAfterCompletion(status, TransactionSynchronization.STATUS_UNKNOWN);
                throw ex;
            }

            // 3. afterCompletion 콜백 호출
            triggerAfterCompletion(status, TransactionSynchronization.STATUS_ROLLED_BACK);
            // ← AFTER_ROLLBACK 및 AFTER_COMPLETION 실행 (롤백 시)
        }
        finally {
            cleanupAfterCompletion(status);
        }
    }
}
```

`processCommit()` 의 호출 순서가 곧 01-01에서 본 "커밋 흐름 Phase 순서"입니다. `triggerBeforeCommit` → `doCommit`(실제 DB 커밋) → `triggerAfterCommit` → `triggerAfterCompletion(STATUS_COMMITTED)` 순으로 돌고, 각 trigger가 ThreadLocal에 등록된 콜백들을 꺼내 해당 메서드를 호출합니다. 롤백 경로(`processRollback`)는 `triggerAfterCompletion(STATUS_ROLLED_BACK)` 만 돌아 `AFTER_ROLLBACK`/`AFTER_COMPLETION` 만 실행됩니다.

## 7. 직접 구현으로 확인하기

`@TransactionalEventListener` 가 내부에서 하는 일을 `TransactionSynchronizationManager` 로 직접 재현하면 메커니즘이 손에 잡힙니다. 아래 서비스는 어노테이션 없이 콜백을 직접 등록하고, `@Order` 순서대로 실행되는지 확인합니다.

```java
/**
 * TransactionSynchronizationManager 직접 사용 예제
 *
 * 이 서비스는 @TransactionalEventListener가 내부적으로 어떻게 동작하는지
 * TransactionSynchronizationManager를 직접 사용하여 보여줍니다.
 */
@Slf4j
@Service
@RequiredArgsConstructor
public class TransactionSyncDemoService {

    private final UserRepository userRepository;

    @Transactional
    public User createUserWithCustomSync(String name, String email) {
        // 1. 트랜잭션 동기화 활성화 확인
        boolean isActive = TransactionSynchronizationManager.isSynchronizationActive();

        // 2. Custom TransactionSynchronization 등록
        //    이것이 @TransactionalEventListener가 내부적으로 하는 일입니다!
        if (isActive) {
            // 여러 순서로 등록하여 @Order 동작 확인
            TransactionSynchronizationManager.registerSynchronization(
                new CustomTransactionSynchronization("Sync-1 (Order=100)", 100)
            );
            TransactionSynchronizationManager.registerSynchronization(
                new CustomTransactionSynchronization("Sync-2 (Order=1)", 1)
            );
            TransactionSynchronizationManager.registerSynchronization(
                new CustomTransactionSynchronization("Sync-3 (Order=50)", 50)
            );
        }

        // 3. 비즈니스 로직 수행
        User user = User.builder()
            .name(name)
            .email(email)
            .status(User.UserStatus.ACTIVE)
            .build();
        User savedUser = userRepository.save(user);
        return savedUser;

        // 메서드 종료 시 트랜잭션 커밋됨
        // → TransactionSynchronization 콜백들이 @Order 순서대로 실행됨:
        //   1. Sync-2 (Order=1)
        //   2. Sync-3 (Order=50)
        //   3. Sync-1 (Order=100)
    }
}
```

등록 순서와 무관하게 `@Order` 값이 작은 것부터 실행된다는 점이 4절의 `OrderComparator.sort()` 가 하는 일입니다.

콜백 구현체는 `TransactionSynchronization` 의 각 메서드에 로그를 심어 어느 Phase에서 무엇이 호출되는지 눈으로 확인합니다.

```java
/**
 * TransactionSynchronization 인터페이스 직접 구현 예제
 *
 * @TransactionalEventListener는 내부적으로 이 인터페이스를 구현한
 * ApplicationListenerMethodTransactionalAdapter를 사용합니다.
 */
@Slf4j
public class CustomTransactionSynchronization implements TransactionSynchronization, Ordered {

    private final String name;
    private final int order;

    public CustomTransactionSynchronization(String name, int order) {
        this.name = name;
        this.order = order;
    }

    @Override
    public int getOrder() {
        return this.order;
    }

    @Override
    public void beforeCommit(boolean readOnly) {
        log.info("[{}] beforeCommit(readOnly={}) - 커밋 직전 (BEFORE_COMMIT Phase)", name, readOnly);
        log.info("[{}]   → 이 시점에 예외가 발생하면 전체 트랜잭션이 롤백됩니다", name);
    }

    @Override
    public void afterCommit() {
        log.info("[{}] afterCommit() - 커밋 완료 (AFTER_COMMIT Phase)", name);
        log.info("[{}]   → 이 시점에 예외가 발생해도 데이터는 이미 커밋되었습니다", name);
    }

    @Override
    public void afterCompletion(int status) {
        String statusStr = switch (status) {
            case STATUS_COMMITTED -> "COMMITTED";
            case STATUS_ROLLED_BACK -> "ROLLED_BACK";
            case STATUS_UNKNOWN -> "UNKNOWN";
            default -> "UNDEFINED";
        };
        log.info("[{}] afterCompletion(status={}) - 트랜잭션 완료", name, statusStr);
    }
}
```

`REQUIRES_NEW` 로 중첩 트랜잭션을 만들면 외부 트랜잭션의 콜백에 `suspend()` → `resume()` 이 호출되는 것도 같은 인터페이스로 관찰할 수 있습니다. 외부 트랜잭션이 일시 중지되고 새 트랜잭션이 시작·커밋된 뒤 외부가 재개되는 흐름입니다.

## 8. 정리

| 구성 요소 | 역할 |
|----------|------|
| `ThreadLocal` | 스레드마다 독립된 트랜잭션 컨텍스트 격리 |
| `TransactionSynchronization` | Phase별 콜백 규약 (beforeCommit/afterCommit/afterCompletion) |
| `TransactionSynchronizationManager` | 콜백을 ThreadLocal Set에 등록·조회·정리, @Order 정렬 |
| `ApplicationListenerMethodTransactionalAdapter` | 어노테이션 메서드를 콜백 구현체로 변환, 발행 시 등록 |
| `AbstractPlatformTransactionManager` | 커밋/롤백 처리 중 trigger로 콜백 호출 |
| 죽은 트랜잭션의 정체 | `actualTransactionActive` ThreadLocal이 cleanup 전까지 true |

발행은 어댑터가 콜백을 ThreadLocal에 *등록*하는 일이고, 실행은 트랜잭션 매니저가 커밋·롤백 시점에 그 콜백을 *트리거*하는 일입니다. 이 등록과 트리거의 분리가 "발행 따로, 실행 따로"의 전부입니다.
