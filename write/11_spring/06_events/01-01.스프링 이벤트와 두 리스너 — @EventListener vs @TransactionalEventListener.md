# 스프링 이벤트와 두 리스너 — @EventListener vs @TransactionalEventListener
---

> 스프링의 이벤트 기능은 한 객체가 "무슨 일이 일어났다"를 알리고, 그 일에 관심 있는 다른 객체가 받아 처리하도록 둘을 느슨하게 떼어 놓는 장치입니다. 이 문서를 다 읽으면 `@EventListener` 와 `@TransactionalEventListener` 가 *언제* 실행되는지를 트랜잭션 흐름 위에 정확히 그릴 수 있고, "이메일 전송은 왜 `@TransactionalEventListener(AFTER_COMMIT)` 에 둬야 안전한가" 같은 실무 판단을 근거를 들어 답할 수 있게 됩니다.

## 1. 이벤트 기반으로 푸는 이유

이벤트는 시스템에서 발생한 의미 있는 상태 변화를 나타냅니다. 사용자가 생성됐다, 주문이 결제됐다 같은 "사실"을 하나의 객체로 표현하고, 그 사실에 반응해야 하는 후속 처리를 발행자에게서 떼어 냅니다. 이렇게 분리하면 발행자(예: 회원 가입 로직)는 "누가 이 사실에 반응하는지"를 몰라도 되고, 새 후속 처리(환영 이메일, 통계 집계)를 추가할 때 발행자 코드를 건드릴 필요가 없습니다.

```java
// 이벤트 클래스 — "사용자가 생성됐다"는 사실을 표현
public class UserCreatedEvent {
    private final User user;

    public UserCreatedEvent(User user) {
        this.user = user;
    }
}
```

발행자는 `ApplicationEventPublisher` 로 이벤트를 던지고, 리스너는 `@EventListener` 로 받습니다. 둘 사이에 직접 호출이 없다는 점이 핵심입니다.

```java
// 이벤트 발행 (Publisher)
@Service
public class UserService {
    private final ApplicationEventPublisher eventPublisher;

    public User createUser(String name) {
        User user = userRepository.save(new User(name));
        eventPublisher.publishEvent(new UserCreatedEvent(user));
        return user;
    }
}

// 이벤트 리스닝 (Listener)
@Component
public class UserEventListener {
    @EventListener
    public void handleUserCreated(UserCreatedEvent event) {
        System.out.println("User created: " + event.getUser().getName());
    }
}
```

`ApplicationEventPublisher` 는 스프링 컨테이너(`ApplicationContext`)가 기본으로 제공하는 빈입니다. 이벤트 발행/구독이 `ApplicationContext` 의 부가 기능 중 하나라는 사실은 [`../01_core/01-01.객체지향 원리 적용 — DI와 IoC`](../01_core/01-01.객체지향%20원리%20적용%20—%20DI와%20IoC.md) §4 에서 본 컨테이너의 네 가지 부가 기능과 이어집니다.

## 2. 두 리스너의 결정적 차이 — 실행 시점

스프링은 같은 이벤트를 받는 두 가지 리스너를 제공합니다. 둘을 가르는 축은 단 하나, *언제 실행되는가* 입니다.

| 항목 | @EventListener | @TransactionalEventListener |
| --- | --- | --- |
| 실행 시점 | 이벤트 발행 즉시 | 트랜잭션 Phase에 따라 |
| 트랜잭션 컨텍스트 | 발행자와 동일 | 독립적 |
| 롤백 영향 | 이미 실행됨 | Phase에 따라 다름 |
| 주요 사용 사례 | 로깅, 모니터링 | 외부 API, 이메일 전송 |

이 표 한 장이 두 리스너 선택의 거의 전부입니다. "발행자 트랜잭션이 롤백되면 이 후속 처리도 취소돼야 하는가"를 자문하면 어느 쪽인지 갈립니다.

### @EventListener — 즉시 실행

`@EventListener` 는 이벤트가 발행되는 즉시 동기적으로 실행됩니다. 발행자와 같은 트랜잭션 컨텍스트 안에서, 같은 스레드로 돌기 때문에 *발행 코드 한가운데서 끼어드는* 모양이 됩니다.

```java
@EventListener
public void handleEvent(UserCreatedEvent event) {
    // 이벤트 발행 즉시 실행
    // 트랜잭션이 커밋되기 전에 실행됨
    // 트랜잭션이 롤백되어도 이미 실행된 상태
}
```

호출 흐름을 펼치면 이렇게 됩니다. 리스너가 *커밋 전에* 실행된다는 점이 핵심입니다.

```text
Service.createUser()
  ├─ userRepository.save()
  ├─ eventPublisher.publishEvent()  ← 이벤트 발행
  │   └─ @EventListener 실행 ★      ← 즉시 실행 (트랜잭션 내부)
  └─ 트랜잭션 커밋
```

그래서 `@EventListener` 는 즉시 실행이 필요하고, 트랜잭션 커밋 여부와 무관해도 되는 로직에 맞습니다. 로깅이나 모니터링처럼 "발행됐다는 사실 자체"에 반응하는 처리가 대표적입니다. 반대로 외부 이메일 전송을 여기 두면, 트랜잭션이 뒤에서 롤백돼 사용자 데이터가 사라졌는데도 환영 이메일은 이미 나가 버리는 사고가 납니다.

### @TransactionalEventListener — 트랜잭션 Phase에 맞춰 실행

`@TransactionalEventListener` 는 트랜잭션의 특정 Phase에서 실행되며, 트랜잭션 상태에 따라 조건부로 동작합니다. 발행자 트랜잭션과 독립적으로 실행될 수 있다는 점이 `@EventListener` 와 갈리는 자리입니다.

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handleEvent(UserCreatedEvent event) {
    // 트랜잭션이 성공적으로 커밋된 후에 실행
    // 외부 API 호출, 이메일 전송 등에 안전
}
```

흐름을 보면 발행 시점에는 리스너가 *등록만* 되고, 실제 실행은 커밋 이후로 미뤄집니다.

```text
Service.createUser()
  ├─ userRepository.save()
  ├─ eventPublisher.publishEvent()  ← 이벤트 발행
  │   (리스너는 아직 실행되지 않음)
  └─ 트랜잭션 커밋
      └─ @TransactionalEventListener 실행 ★  ← 커밋 후 실행
```

데이터가 안전하게 커밋된 뒤에만 실행되므로, 외부 시스템 연동(이메일·알림·API 호출)처럼 "성공이 확정된 후에만 해야 하는" 처리에 안전합니다. 이 "발행 시 등록 → 커밋 시 실행"이 어떻게 가능한지는 [01-03 내부 동작 원리](01-03.@TransactionalEventListener%20내부%20동작%20원리.md)에서 다룹니다.

## 3. 트랜잭션 Phase 네 종류와 실행 순서

`@TransactionalEventListener` 의 `phase` 속성은 트랜잭션 생명주기의 어느 지점에서 리스너를 실행할지 정합니다. 정상 커밋과 롤백 두 경로에서 어떤 Phase가 도는지 먼저 전체 그림을 봅니다.

커밋이 성공하는 경우입니다.

```text
1. Service 메서드 시작
2. DB 작업 (save/update/delete)
3. eventPublisher.publishEvent()
   └─ @EventListener 실행 ★ (즉시)
4. @TransactionalEventListener(BEFORE_COMMIT) 실행 ★
5. 트랜잭션 커밋
6. @TransactionalEventListener(AFTER_COMMIT) 실행 ★
7. @TransactionalEventListener(AFTER_COMPLETION) 실행 ★
```

예외로 롤백되는 경우입니다. `@EventListener` 는 이미 실행됐고, 커밋 관련 Phase는 건너뛰며, 롤백·완료 Phase만 돕니다.

```text
1. Service 메서드 시작
2. DB 작업 (save/update/delete)
3. eventPublisher.publishEvent()
   └─ @EventListener 실행 ★ (즉시)
4. 예외 발생
5. 트랜잭션 롤백
6. @TransactionalEventListener(AFTER_ROLLBACK) 실행 ★
7. @TransactionalEventListener(AFTER_COMPLETION) 실행 ★
```

각 Phase의 의미와 쓰임은 다음과 같습니다.

### BEFORE_COMMIT

```java
@TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
public void beforeCommit(UserCreatedEvent event) {
    // 트랜잭션 커밋 직전에 실행
    // 아직 트랜잭션이 활성화된 상태
}
```

트랜잭션이 커밋되기 직전, 아직 트랜잭션이 살아 있는 상태에서 실행됩니다. DB 변경사항이 아직 커밋되지 않았으므로, 이 리스너에서 예외가 나면 전체 트랜잭션이 함께 롤백됩니다. "커밋 직전 마지막 검증"처럼 실패 시 전체를 되돌려야 하는 처리에 씁니다.

### AFTER_COMMIT (기본값)

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void afterCommit(UserCreatedEvent event) {
    // 트랜잭션 커밋 후 실행
    // 데이터가 안전하게 저장된 상태
}
```

`phase` 를 생략하면 이 값이 기본입니다. 트랜잭션이 성공적으로 커밋된 뒤 실행되므로, 리스너를 호출한 트랜잭션은 이미 종료됐고 데이터는 DB에 반영된 상태입니다. 여기서 예외가 발생해도 이미 커밋된 트랜잭션에는 영향이 없습니다. 가장 흔히 쓰는 Phase이며, 외부 연동의 표준 자리입니다.

### AFTER_ROLLBACK

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_ROLLBACK)
public void afterRollback(UserCreatedEvent event) {
    // 트랜잭션 롤백 후 실행
    // 보상 트랜잭션 또는 에러 처리
}
```

트랜잭션이 롤백된 뒤, DB 변경이 모두 취소된 상태에서 실행됩니다. 예외 발생 시 보상 로직이나 실패 알림에 씁니다.

### AFTER_COMPLETION

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMPLETION)
public void afterCompletion(UserCreatedEvent event) {
    // 트랜잭션 완료 후 실행 (커밋/롤백 무관)
}
```

커밋이든 롤백이든 트랜잭션이 완료된 뒤 *항상* 실행됩니다. 결과와 무관하게 정리해야 하는 처리(리소스 해제, 종료 로그)에 적합합니다.

## 4. 동작을 세밀하게 다듬는 두 속성

### fallbackExecution — 트랜잭션이 없을 때

`@TransactionalEventListener` 는 이름 그대로 *트랜잭션을 전제*합니다. 그렇다면 트랜잭션 없이 이벤트가 발행되면 어떻게 될까요? `fallbackExecution` 속성이 그 동작을 정합니다.

| 값 | 동작 | 권장 여부 |
| --- | --- | --- |
| `false` (기본값) | 트랜잭션 있을 때만 실행 | 권장 |
| `true` | 트랜잭션 없으면 즉시 실행 | 제한적 사용 |

```java
@Service
public class UserService {

    @Transactional
    public void createUser() {
        publish(new UserEvent()); // AFTER_COMMIT에 실행
    }

    public void nonTxMethod() {
        publish(new UserEvent()); // fallbackExecution=false → 무시됨
    }
}
```

기본값 `false` 가 권장인 이유는, 트랜잭션 없이 발행된 이벤트를 조용히 실행해 버리면 "커밋 후에만 돈다"는 보장이 깨지기 때문입니다. 트랜잭션 경계가 불확실한 곳에서 리스너가 의도치 않게 실행되는 사고를 막으려면 기본값을 그대로 두는 편이 안전합니다.

### condition — SpEL로 실행 여부 결정

`condition` 은 Spring Expression Language(SpEL)로 이벤트 처리 여부를 동적으로 결정합니다. 이벤트 객체의 속성을 보고 "이 조건일 때만 리스너를 돌려라"를 선언적으로 표현합니다.

```java
condition = "#event.속성 == 값"
```

`#event` 는 수신한 이벤트 객체를 가리킵니다. 자주 쓰는 표현식은 다음과 같습니다.

| 목적 | SpEL 표현식 | 예시 |
| --- | --- | --- |
| 숫자 비교 | `#event.amount > 1000000` | 고액 주문 |
| 객체 속성 | `#event.user.age >= 18` | 성인 사용자 |
| Enum 비교 | `#event.status == T(OrderStatus).COMPLETED` | 완료 상태 |
| 논리 연산 | `#event.amount > 100000 && #event.paymentMethod == 'CARD'` | 복합 조건 |
| 컬렉션 | `#event.items.size() > 5` | 대량 주문 |
| 포함 여부 | `#event.tags.contains('VIP')` | VIP 고객 |
| 메서드 호출 | `#event.isEligibleForDiscount()` | 비즈니스 로직 위임 |
| 정규식 | `#event.email matches '.*@company\\.com'` | 회사 이메일 |
| Null 체크 | `#event.oldValue != null` | 변경 전후 비교 |
| Bean 참조 | `@env.getProperty('feature.enabled') == 'true'` | 기능 플래그 |

조건을 리스너 메서드 안의 `if` 로 거르지 않고 `condition` 으로 빼면, "이 리스너는 이런 이벤트에만 반응한다"는 의도가 시그니처에 드러나고 불필요한 메서드 호출 자체가 일어나지 않습니다.

```java
@TransactionalEventListener(
    phase = TransactionPhase.AFTER_COMMIT,
    condition = "#event.customer.vip && #event.totalAmount > 1000000"
)
public void notifyVipLargeOrder(OrderCreatedEvent event) {
    // VIP + 100만원 초과
}
```

```java
@TransactionalEventListener(
    phase = TransactionPhase.AFTER_ROLLBACK,
    condition = "#event.retryCount < 3 && #event.errorType != 'CARD_INVALID'",
    fallbackExecution = false
)
public void scheduleRetry(PaymentFailedEvent event) {
    // 재시도 가능 조건
}
```

## 5. 정리

| 주제 | 핵심 |
|------|------|
| 이벤트 기반 | 발행자와 후속 처리를 느슨하게 분리. 발행자는 누가 반응하는지 모른다. |
| @EventListener | 발행 즉시·동기·발행자 트랜잭션 내부 실행. 로깅·모니터링용. |
| @TransactionalEventListener | 트랜잭션 Phase에 맞춰 실행. 외부 연동·이메일용. |
| BEFORE_COMMIT | 커밋 직전, 트랜잭션 활성. 예외 시 전체 롤백. |
| AFTER_COMMIT (기본) | 커밋 후. 데이터 안전. 외부 연동의 표준 자리. |
| AFTER_ROLLBACK | 롤백 후. 보상 로직·실패 알림. |
| AFTER_COMPLETION | 커밋/롤백 무관 항상 실행. 정리 작업. |
| fallbackExecution | 트랜잭션 없을 때 동작. 기본 false(무시)가 안전. |
| condition | SpEL로 실행 여부 선언. 불필요한 호출 차단. |
