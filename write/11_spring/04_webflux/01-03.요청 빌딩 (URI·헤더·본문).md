# 요청 빌딩 (URI·헤더·본문)

---

> 요청을 만드는 단계는 짧지만 흔히 함정이 박혀 있다. URI 인코딩이 두 번 되거나, 헤더 대소문자가 제대로 동작 안 하거나, `bodyValue`와 `body(BodyInserters)` 둘 중 무엇을 써야 할지가 헷갈린다. 본 챕터는 그 셋을 한 자리에 정리한다.



## 메서드 선택 — `get()/post()/...` vs `method()`

> 정적 메서드 호출이 자연스러우면 `get()`/`post()`를 쓰고, 메서드를 런타임에 결정해야 한다면 `method()`를 쓴다.

가장 흔한 형태는 다음과 같다.

```java
client.get().uri("/users/{id}", 42L);
client.post().uri("/users").bodyValue(newUser);
client.delete().uri("/users/{id}", 42L);
```

위는 컴파일 타임에 메서드가 정해진다. HTTP 메서드를 DB나 설정에서 가져와 동적으로 결정하려면 `method(HttpMethod)`를 쓴다.

```java
HttpMethod httpMethod = HttpMethod.valueOf(rule.method().toUpperCase());
client.method(httpMethod).uri(rule.url());
```

TPS `ApprovalUrlAdapter`가 그 형태다. 결재 완료 후 호출할 외부 API의 메서드·URL·헤더·본문이 모두 DB에 저장돼 있어 런타임에 조립해야 한다.

```java
// ApprovalUrlAdapter.java:76, 102-108
HttpMethod httpMethod = HttpMethod.valueOf(item.httpMethod().toUpperCase());
// ...
WebClient.RequestBodySpec request = webClient.method(httpMethod)
        .uri(trbServicesProperties.resolveUrlByRequestPath(item.apiUrl()))
        .headers(headers -> headerMap.forEach((key, value) -> {
            if (!"content-type".equalsIgnoreCase(key)) {
                headers.add(key, value);
            }
        }));
```

`method()`로 출발해 `uri()`와 `headers()`로 이어지는 흐름이 보인다. URL과 헤더 조립 패턴은 다음 두 절에서 풀이한다.



## URI 빌더 — 두 가지 형태

> 고정 경로에 변수만 채우는 호출은 한 줄로 끝난다. 쿼리 파라미터 인코딩이나 조건부 변수가 들어오면 `Function<UriBuilder, URI>` 형태로 옮긴다.

### 1) 변수 치환만 필요한 경우

```java
// 위치 기반 치환
client.get().uri("/users/{id}/orders/{orderId}", 42L, 7L);

// 이름 기반 치환
client.get().uri("/users/{tenant}/{id}", Map.of("tenant", "default", "id", 42L));
```

위 형태는 가장 가독성이 좋다. 변수에 들어가는 값은 자동으로 percent-encoding된다. 한글이나 공백이 들어가도 정상 인코딩이다.

⚠️ **이중 인코딩 함정**: URL 문자열 자체에 이미 `%20` 같은 인코딩이 들어가 있고 `{var}`가 그 안에 끼어 있다면, 변수 치환이 한 번 더 인코딩하면서 `%2520`이 된다. 이런 케이스는 `UriBuilderFactory.EncodingMode`를 손대거나 `URI` 객체로 직접 만들어 넘긴다.

### 2) 쿼리 파라미터·조건부 변수가 필요한 경우

`uri(Function<UriBuilder, URI>)` 형태로 옮긴다.

```java
client.get().uri(uriBuilder -> uriBuilder
        .path("/users")
        .queryParam("status", "ACTIVE")
        .queryParamIfPresent("name", Optional.ofNullable(name))
        .queryParam("limit", 20)
        .build());
```

`queryParamIfPresent(name, Optional)`는 값이 비어 있으면 파라미터 자체를 생략한다. 동적 검색 화면에서 흔히 쓴다. 같은 이름으로 여러 값을 넣고 싶다면 `queryParam(name, values...)`에 가변 인자를 넘기거나 `queryParam(name, Collection)` 오버로드를 쓴다.

### 3) 인코딩 모드 — `EncodingMode`

`DefaultUriBuilderFactory.EncodingMode`는 네 가지 값을 가진다.

| 모드 | 의미 |
|------|------|
| `TEMPLATE_AND_VALUES` (기본) | 템플릿과 변수 모두 인코딩 |
| `VALUES_ONLY` | 변수만 인코딩, 템플릿은 그대로 |
| `URI_COMPONENT` | RFC 3986 컴포넌트 단위 인코딩 |
| `NONE` | 인코딩 없음 |

기본값(`TEMPLATE_AND_VALUES`)이 대부분의 경우 안전하다. 외부 API가 이미 인코딩된 URL을 통째로 넘겨주는 경우(예: 검색 토큰을 base64로 인코딩한 값을 path에 박아 보내는 API)는 `VALUES_ONLY`로 옮긴다.

```java
DefaultUriBuilderFactory factory = new DefaultUriBuilderFactory();
factory.setEncodingMode(DefaultUriBuilderFactory.EncodingMode.VALUES_ONLY);

WebClient client = WebClient.builder()
        .uriBuilderFactory(factory)
        .baseUrl("https://api.example.com")
        .build();
```



## 헤더 추가 — 세 가지 자리

> 빌더 단계의 기본 헤더, 호출 단계의 개별 헤더, 그리고 일괄 처리 Consumer가 있다.

### 1) 빌더 단계 — `defaultHeader`

모든 요청에 자동으로 붙는 헤더는 빌더에서 설정한다.

```java
WebClient client = WebClient.builder()
        .defaultHeader("Accept", "application/json")
        .defaultHeader("X-Service", "tps-operator")
        .build();
```

### 2) 호출 단계 — `header(name, values...)`

요청별로 추가하는 헤더는 호출 체인에서 직접 붙인다.

```java
client.get()
        .uri("/users/{id}", 42L)
        .header("X-Trace-Id", traceId)
        .header("Authorization", "Bearer " + token)
        .retrieve();
```

같은 이름을 여러 번 호출하면 헤더가 덧붙는다. `header()`는 추가이지 덮어쓰기가 아니다. 한 번에 덮어쓰려면 다음 패턴을 쓴다.

### 3) 일괄 처리 — `headers(Consumer<HttpHeaders>)`

여러 헤더를 한 번에 다루거나, 조건부로 추가·제거하거나, 헤더를 Map에서 변환해 붙일 때 쓴다.

```java
Map<String, String> headerMap = Map.of(
        "X-Trace-Id", traceId,
        "X-Tenant", tenantId,
        "Authorization", "Bearer " + token);

client.get().uri("/users")
        .headers(h -> headerMap.forEach(h::add))
        .retrieve();
```

`HttpHeaders`의 `add(name, value)`는 누적, `set(name, value)`는 덮어쓰기다. `Consumer`에서 무엇을 호출하는지에 따라 동작이 다르다.

TPS 어댑터의 헤더 처리는 다음과 같다.

```java
// ApprovalUrlAdapter.java:104-108
.headers(headers -> headerMap.forEach((key, value) -> {
    if (!"content-type".equalsIgnoreCase(key)) {
        headers.add(key, value);
    }
}));
```

여기서 주목할 부분은 `Content-Type`만 헤더 추가에서 제외한 점이다. 이유는 본문 형식을 결정하는 `Content-Type`이 multipart인지 JSON인지에 따라 본문 빌드 방식이 달라지고, multipart의 경우 boundary 토큰을 WebClient가 자동 생성해 붙여야 하기 때문이다. 사용자가 미리 박은 `Content-Type` 값(boundary 없음)을 그대로 쓰면 서버가 본문을 파싱하지 못한다. 02-01 multipart 챕터에서 깊이 풀이한다.



## 본문 빌드 — `bodyValue` vs `body(BodyInserters)`

> 단순한 객체 한 개라면 `bodyValue` 한 줄, 형식·스트리밍·multipart가 끼면 `body(BodyInserters...)` 형태로 옮긴다.

### 1) `bodyValue(Object)`

가장 단순한 형태. 객체를 그대로 본문에 직렬화한다. JSON Jackson 기본 코덱이 동작한다.

```java
NewUser user = new NewUser("Alice", "alice@example.com");
client.post().uri("/users")
        .contentType(MediaType.APPLICATION_JSON)
        .bodyValue(user)
        .retrieve();
```

`Map<String, Object>`도 그대로 넣을 수 있다. TPS 어댑터의 JSON 본문 분기가 그 형태다.

```java
// ApprovalUrlAdapter.java:160-167
if (item.dmndInfo() != null && !item.dmndInfo().isBlank()) {
    Map<String, Object> dmndInfoMap = om.readValue(item.dmndInfo(), new TypeReference<>() {});
    request.bodyValue(dmndInfoMap);
}
```

DB에서 JSON 문자열을 꺼내 `Map`으로 파싱한 뒤 `bodyValue`에 그대로 넘긴다. 별도 DTO 클래스를 만들 필요가 없다. `Content-Type`은 호출자가 미리 헤더에 넣어 둔 값이 적용된다(어댑터에서는 `headers()`에서 명시적으로 제외했으므로, 별도 처리 필요 — 02-01에서 풀이).

### 2) `body(BodyInserter)`

multipart, 스트리밍, 폼 인코딩처럼 단순 객체 직렬화로 표현 안 되는 본문은 `body(BodyInserters.from...)`로 옮긴다.

```java
// 멀티파트
MultipartBodyBuilder builder = new MultipartBodyBuilder();
builder.part("file", resource).contentType(MediaType.IMAGE_PNG);
builder.part("metadata", metaJson, MediaType.APPLICATION_JSON);

client.post().uri("/upload")
        .body(BodyInserters.fromMultipartData(builder.build()))
        .retrieve();
```

```java
// form-urlencoded
MultiValueMap<String, String> form = new LinkedMultiValueMap<>();
form.add("grant_type", "password");
form.add("username", "alice");
form.add("password", "secret");

client.post().uri("/oauth/token")
        .body(BodyInserters.fromFormData(form))
        .retrieve();
```

```java
// 스트리밍 (Flux)
Flux<DataBuffer> stream = ...; // 어딘가에서 읽고 있는 데이터 스트림
client.post().uri("/upload-stream")
        .body(BodyInserters.fromDataBuffers(stream))
        .retrieve();
```

`BodyInserters`의 자주 쓰는 정적 메서드는 다음과 같다.

| 메서드 | 용도 |
|--------|------|
| `fromValue(Object)` | `bodyValue`와 동등. 단일 객체 직렬화 |
| `fromPublisher(Publisher, Class)` | `Mono`/`Flux`를 본문으로 |
| `fromMultipartData(...)` | multipart/form-data |
| `fromFormData(MultiValueMap)` | application/x-www-form-urlencoded |
| `fromDataBuffers(Publisher)` | 청크 단위 바이너리 스트림 |
| `fromResource(Resource)` | 파일/리소스 본문 |

### 3) `body(Publisher, Class)` 단축

`Mono<T>`나 `Flux<T>`를 본문으로 넣을 때 `body(Publisher, Class)` 단축 형태가 있다.

```java
Mono<NewUser> userMono = userService.create();
client.post().uri("/users")
        .body(userMono, NewUser.class)
        .retrieve();
```

응답 직전에 `userService.create()`가 비동기로 동작하고, 결과가 도착하면 본문으로 직렬화된다. RestTemplate으로는 만들기 어려운 패턴이다.



## `accept`와 `contentType` — 헤더의 단축

> 두 헤더는 자주 쓰여서 단축 메서드가 있다.

```java
client.get().uri("/users/{id}", 42L)
        .accept(MediaType.APPLICATION_JSON)            // Accept 헤더
        .acceptCharset(StandardCharsets.UTF_8)
        .retrieve();

client.post().uri("/users")
        .contentType(MediaType.APPLICATION_JSON)       // Content-Type 헤더
        .bodyValue(user)
        .retrieve();
```

`bodyValue` + JSON 객체 조합은 `Content-Type`이 자동으로 `application/json`이 된다. 굳이 명시할 필요는 없지만, 명시해 두면 코드 리뷰에서 의도가 분명하다.

multipart 본문은 `Content-Type`이 자동으로 `multipart/form-data; boundary=...`가 된다. 절대 호출자가 직접 박지 않는다(boundary 토큰을 매번 다르게 생성해야 하기 때문이다).



## `attribute()` — 필터에 컨텍스트 전달

> 필터(01-06)에서 요청별 메타데이터를 읽고 싶을 때 쓴다.

```java
client.get().uri("/users")
        .attribute("requestId", UUID.randomUUID().toString())
        .attribute("retryable", true)
        .retrieve();
```

`ExchangeFilterFunction`은 `ClientRequest`에서 `attribute(name)`으로 값을 읽을 수 있다. 헤더로 보낼 만한 정보는 아니지만 필터 동작을 분기해야 할 때(예: "이 요청은 재시도하지 마라" 같은 플래그) 깔끔하다.

01-06에서 필터와 함께 다룬다.



## 함정 — 자주 만나는 두 가지

> 처음 만나면 디버깅에 시간이 가는 항목들이다.

### 1. URI에 percent-encoding이 두 번 들어간다

이미 인코딩된 URL을 `uri(String)`에 넣으면서 `{var}` 변수까지 같이 들어 있는 경우가 대표적이다. 해결은 두 가지.

- 변수 치환을 빼고 `URI` 객체를 직접 만들어 `uri(URI)`로 넘긴다.
- `EncodingMode.VALUES_ONLY`로 빌더를 바꿔 템플릿은 인코딩하지 않는다.

### 2. `Content-Type`을 직접 박았더니 본문이 안 보낸다

multipart의 boundary 토큰은 WebClient가 자동 생성한다. `Content-Type: multipart/form-data` 헤더를 호출자가 박으면 boundary 없이 들어가 서버가 파싱을 실패한다. multipart는 `Content-Type` 헤더를 직접 추가하지 않고 `body(BodyInserters.fromMultipartData(...))`만 호출한다. TPS 어댑터가 `Content-Type` 헤더를 명시적으로 제외한 이유가 이것이다.



## 면접에서 받을 만한 질문

> 챕터 마무리 점검.

1. `uri(String, Object...)`와 `uri(Function<UriBuilder, URI>)`는 언제 갈리는가?
   - 답 요지: 변수 치환만 필요한 단순 호출은 첫 번째, 쿼리 파라미터·조건부 변수가 들어오면 두 번째.
2. `header()`와 `headers(Consumer)`는 언제 갈리는가?
   - 답 요지: 한두 개 직접 붙이는 호출은 첫 번째, 여러 헤더를 일괄 처리하거나 조건부로 추가·제거하는 호출은 두 번째.
3. `bodyValue(Object)`로 `Map`을 넘길 수 있는가?
   - 답 요지: 가능. 별도 DTO 클래스 없이 `Map<String, Object>`를 그대로 JSON으로 직렬화한다.
4. multipart 요청에서 `Content-Type` 헤더를 직접 박으면 왜 안 되는가?
   - 답 요지: boundary 토큰이 매 요청마다 다르게 생성되어야 하고, 그 생성을 WebClient가 담당한다. 직접 박은 헤더는 boundary가 없어 서버가 본문을 파싱 못한다.
5. `EncodingMode.VALUES_ONLY`는 언제 쓰는가?
   - 답 요지: 외부 API가 이미 인코딩된 URL을 통째로 넘겨주는 경우. 변수 치환만 추가 인코딩하고 템플릿 자체는 그대로 둔다.
