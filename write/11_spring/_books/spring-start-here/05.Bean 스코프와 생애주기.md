# Bean 스코프와 생애주기
---
> 지금까지는 Spring이 bean을 *언제* 어떻게 만드는지를 기본값에 맡겼습니다. 이 장은 그 생성·관리 방식인 스코프를 다룹니다. 기본이자 가장 흔한 singleton, 그리고 매번 새 인스턴스를 주는 prototype 둘을 비교하고, singleton의 eager·lazy 생성 시점과 동시성(race condition) 주의점까지 정리합니다.


## 핵심 요약

스코프는 Spring이 bean 인스턴스를 만들고 관리하는 방식입니다. 기본은 **singleton**으로, 이름 하나에 인스턴스 하나를 묶어 같은 이름으로 참조하면 항상 같은 인스턴스를 돌려줍니다(이름당 유일이지 타입당 유일이 아님 — 디자인 패턴의 싱글턴과 다름). singleton은 여러 스레드가 공유하므로 **불변(immutable)이어야** 하고, 가변이면 race condition에 빠집니다. singleton은 Context 로드 시 모두 생성하는 **eager**가 기본이며, 첫 참조 시점에 만드는 **lazy**(`@Lazy`)도 있습니다. 가변 객체가 필요하면 **prototype** 스코프(`@Scope(SCOPE_PROTOTYPE)`)를 쓰는데, 참조할 때마다 새 인스턴스를 주므로 동시성 문제가 없습니다. 단, prototype을 singleton에 직접 주입하면 한 번만 주입돼 prototype 효과가 사라지므로, Context에서 매번 꺼내야 합니다.


## 학습 목표

> 이 내용을 읽고 나면 다음을 할 수 있습니다.

1. Spring의 singleton이 "이름당 유일"임을 디자인 패턴 싱글턴과 구분해 설명할 수 있습니다.
2. singleton bean이 불변이어야 하는 이유를 race condition으로 설명할 수 있습니다.
3. eager와 lazy 생성의 차이와 각각의 권장 상황을 말할 수 있습니다.
4. prototype 스코프를 적용하고 singleton과 동작 차이를 보일 수 있습니다.
5. prototype을 singleton에 주입할 때의 함정과 올바른 사용법을 설명할 수 있습니다.


## 본문 정리


### 1. singleton 스코프 — 이름당 유일

Spring은 Context를 로드할 때 singleton bean을 만들고 이름(bean ID)을 붙입니다. 같은 이름으로 참조하면 늘 같은 인스턴스를 돌려주기에 singleton이라 부릅니다.

> ⚠️ **디자인 패턴의 싱글턴과 다릅니다.** 디자인 패턴 싱글턴은 *앱 전체에 인스턴스 하나*를 뜻하지만, Spring의 singleton은 *이름당 유일*입니다. 같은 타입이라도 이름이 다르면 인스턴스를 여럿 가질 수 있습니다. "이름당 유일이지 앱당 유일이 아니다"가 핵심입니다.

`@Bean`이든 `@Component`든 동작은 같습니다. 같은 이름으로 두 번 꺼내면 같은 참조입니다.

```java
@Configuration
public class ProjectConfig {
  @Bean public CommentService commentService() { return new CommentService(); }
}

// Main
var cs1 = c.getBean("commentService", CommentService.class);
var cs2 = c.getBean("commentService", CommentService.class);
System.out.println(cs1 == cs2);   // true — 항상 같은 인스턴스
```

스테레오타입으로 만들어도, 두 서비스가 같은 리포지토리를 `@Autowired`로 주입받으면 **둘 다 같은 인스턴스**를 받습니다.

```java
@Service class CommentService { @Autowired CommentRepository commentRepository; /* getter */ }
@Service class UserService    { @Autowired CommentRepository commentRepository; /* getter */ }
// s1.getCommentRepository() == s2.getCommentRepository()  → true (같은 singleton)
```


### 2. singleton과 동시성 — 불변이어야 하는 이유

singleton은 여러 컴포넌트가 한 인스턴스를 공유한다는 전제입니다. 웹 앱처럼 여러 스레드가 도는 환경에서, 스레드들이 이 공유 인스턴스를 동시에 바꾸면 **race condition**에 빠져 예기치 못한 결과나 예외가 납니다.

가변 singleton을 쓰려면 직접 스레드 동기화를 해야 하는데, singleton은 동기화를 전제로 설계된 게 아닙니다. 동기화는 성능을 크게 떨어뜨려 좋은 선택이 아닙니다. 그래서 3장에서 권장한 **생성자 주입**이 여기서도 빛납니다 — 필드를 `final`로 만들어 불변을 강제할 수 있기 때문입니다.

```java
@Service
public class CommentService {
  private final CommentRepository commentRepository;   // final → 불변
  public CommentService(CommentRepository r) { this.commentRepository = r; }
}
```

> **bean 사용 3원칙**: ① Spring이 관리할 이유(기능 부여)가 있을 때만 bean으로 만듭니다. ② bean으로 만든다면 *불변일 때만* singleton으로 둡니다. ③ 가변이어야 하면 prototype을 고려합니다.


### 3. eager vs lazy 생성

singleton의 기본은 **eager** — Context 초기화 때 모든 singleton을 미리 만듭니다. `@Lazy`를 붙이면 **lazy** — 누군가 처음 참조할 때 만듭니다.

```java
@Service
@Lazy   // 처음 참조될 때 생성. 아무도 안 쓰면 생성 안 됨
public class CommentService {
  public CommentService() { System.out.println("CommentService instance created!"); }
}
```

| | eager (기본) | lazy (@Lazy) |
|---|------------|--------------|
| 생성 시점 | Context 초기화 시 전부 | 첫 참조 시 |
| 위임 시 | 대상 bean이 이미 존재 | 존재 확인 + 없으면 생성 |
| 오류 발견 | 기동 시 즉시 | 실행 중 그 bean 도달 시 |
| 메모리 | 다 올림 | 쓰는 것만 |

> 저자의 권고는 **기본값(eager)을 쓰라**는 것입니다. 위임 시 대상이 이미 있어 검사 비용이 없고, bean 생성 오류를 기동 시점에 바로 잡을 수 있기 때문입니다. lazy가 필요한 상황(거대 모놀리스에서 일부 기능만 쓰는 클라이언트)은 대개 *설계에 문제가 있다는 신호*입니다. 모듈러·마이크로서비스로 설계됐다면 필요한 것만 배포해 lazy가 불필요했을 것입니다. 근본 원인을 못 고칠 때 증상 완화책으로 lazy를 씁니다.


### 4. prototype 스코프 — 참조할 때마다 새 인스턴스

prototype은 참조할 때마다 Spring이 **새 인스턴스**를 만듭니다. Spring은 인스턴스가 아니라 *타입*을 관리하다가, 요청이 올 때마다 새로 찍어 냅니다. `@Scope(BeanDefinition.SCOPE_PROTOTYPE)`로 지정합니다.

```java
@Configuration
public class ProjectConfig {
  @Bean
  @Scope(BeanDefinition.SCOPE_PROTOTYPE)   // 매 요청마다 새 인스턴스
  public CommentService commentService() { return new CommentService(); }
}

// var cs1 = c.getBean(...); var cs2 = c.getBean(...);
// cs1 == cs2  → false (항상 다른 인스턴스)
```

prototype은 스레드마다 다른 인스턴스를 받으므로 **race condition이 없고, 가변이어도 안전**합니다. 스테레오타입은 클래스 위에 `@Scope`를 함께 붙입니다.

```java
@Repository
@Scope(BeanDefinition.SCOPE_PROTOTYPE)
public class CommentRepository { }
// 두 서비스가 @Autowired로 받으면 각자 다른 CommentRepository 인스턴스
```


### 5. prototype을 언제 쓰나 — 그리고 함정

흔하지 않지만 좋은 판단 기준이 있습니다. **가변 객체**가 후보입니다. 댓글을 속성으로 들고 메서드가 그 속성을 바꾸는 `CommentProcessor`를 생각해 봅시다. 이 객체가 단순 가변이기만 하면 bean일 필요도 없습니다. 하지만 이 객체가 `CommentRepository`(bean) 주입이 필요해지면, DI를 받기 위해 bean이 되어야 합니다. 그런데 가변이라 singleton은 안 됩니다 → **prototype**이 답입니다.

여기서 핵심 함정이 나옵니다. prototype을 singleton에 직접 주입하면 안 됩니다.

```java
// ❌ 잘못된 방법 — singleton에 prototype 직접 주입
@Service
public class CommentService {
  @Autowired private CommentProcessor p;   // CommentService 생성 시 단 한 번 주입됨
  public void sendComment(Comment c) { p.setComment(c); /* ... */ }
}
```

`CommentService`가 singleton이라 Spring은 이 클래스를 한 번만 만들고, 의존성도 *그때 한 번만* 주입합니다. 그래서 `CommentProcessor`도 하나뿐이 되어, 매 호출이 같은 인스턴스를 써 prototype의 의미가 사라지고 race condition으로 돌아갑니다.

```java
// ✅ 올바른 방법 — 메서드 안에서 Context로 매번 꺼냄
@Service
public class CommentService {
  @Autowired private ApplicationContext context;
  public void sendComment(Comment c) {
    CommentProcessor p = context.getBean(CommentProcessor.class);  // 호출마다 새 인스턴스
    p.setComment(c); p.processComment(c); p.validateComment(c);
    c = p.getComment();
  }
}
```

매 호출마다 새 인스턴스가 필요하므로, bean 요청을 *메서드 안*에 두어야 합니다.

#### singleton vs prototype 정리

| 항목 | singleton | prototype |
|------|-----------|-----------|
| 연결 단위 | 이름 ↔ 실제 인스턴스 | 이름 ↔ 타입 |
| 참조 시 | 항상 같은 인스턴스 | 매번 새 인스턴스 |
| 생성 시점 | eager(기본) 또는 lazy 선택 | 참조할 때마다 |
| 기본 여부 | 기본 스코프 | 명시(`@Scope`) 필요 |
| 가변 속성 | 비권장(불변이어야) | 허용 |


## 심화 학습

> 책은 Spring 5 기준입니다. 실무 맥락과 이후 동향을 보강합니다.

- **prototype 주입의 더 깔끔한 해법**: 책은 `ApplicationContext.getBean()`을 메서드 안에서 부르는 방식을 보여 주지만, 이는 코드를 Spring API에 묶는 단점이 있습니다. 실무에서는 `ObjectProvider<CommentProcessor>`를 주입받아 `provider.getObject()`를 호출하거나, JSR-330의 `Provider<T>`를 쓰면 같은 효과를 덜 침습적으로 얻습니다. lookup method injection(`@Lookup`)도 같은 문제의 표준 해법입니다.
- **웹 스코프(9장 예고)**: 이 장의 singleton·prototype 외에 웹 앱에는 `request`·`session`·`application` 스코프가 있습니다. 요청 단위·세션 단위로 인스턴스를 나눠, prototype과 비슷하게 상태를 격리하되 수명을 HTTP 생애주기에 맞춥니다.
- **생애주기 콜백과의 연결**: 스코프는 *몇 개를 언제 만드나*를 정하고, 2장에서 본 `@PostConstruct`·`@PreDestroy`는 *각 인스턴스의 생성 직후·소멸 직전*에 무엇을 하나를 정합니다. 단, prototype bean은 Spring이 소멸을 관리하지 않아 `@PreDestroy`가 호출되지 않는 점에 주의해야 합니다.


## 실무 적용 포인트

### 이런 상황에서 사용하세요

- 무상태 서비스·리포지토리(앱 골격) → singleton + 생성자 주입으로 불변 유지
- 호출마다 상태를 들고 처리하는 가변 헬퍼(`~Processor`) → prototype + 메서드 내 `getBean`/`ObjectProvider`
- 일부 기능만 쓰는 거대 앱에서 메모리 절약이 급할 때 → `@Lazy`(단, 설계 재검토 먼저)

### 주의할 점

- ⚠️ 가변 상태를 가진 singleton은 race condition의 원천입니다. 불변으로 설계하거나 prototype으로 바꿉니다.
- ⚠️ prototype을 singleton 필드에 직접 주입하면 한 번만 주입돼 효과가 사라집니다.
- ⚠️ prototype은 `@PreDestroy`가 호출되지 않으므로 자원 해제를 의존하지 않습니다.


## 면접 대비

### 한 줄 정의

"bean 스코프란 Spring이 인스턴스를 몇 개 언제 만들지 정하는 방식이며, 기본 singleton은 이름당 하나를 공유하고 prototype은 참조할 때마다 새로 만듭니다."

### 핵심 포인트 3가지

1. Spring의 singleton은 *이름당 유일*이지 디자인 패턴 싱글턴(앱당 유일)과 다릅니다.
2. singleton은 공유되므로 불변이어야 하며, 가변이 필요하면 prototype을 씁니다.
3. prototype을 singleton에 직접 주입하면 한 번만 주입돼 의미가 사라지니, Context에서 매번 꺼냅니다.

### 자주 묻는 질문

Q: Spring의 singleton과 디자인 패턴 싱글턴은 같은가요?
A: 다릅니다. 디자인 패턴은 앱 전체에 인스턴스 하나지만, Spring singleton은 이름당 하나라 같은 타입이라도 이름이 다르면 여럿 존재할 수 있습니다.

Q: singleton bean은 왜 불변이어야 하나요?
A: 여러 스레드가 같은 인스턴스를 공유하는데, 동시에 상태를 바꾸면 race condition으로 결과가 깨지기 때문입니다. 동기화는 성능을 해쳐 권장되지 않습니다.

Q: prototype bean을 singleton에 주입했더니 매번 같은 인스턴스가 나옵니다. 왜죠?
A: singleton은 한 번만 생성되며 의존성도 그때 한 번만 주입되기 때문입니다. 매 호출 새 인스턴스가 필요하면 메서드 안에서 `getBean`이나 `ObjectProvider`로 꺼내야 합니다.
