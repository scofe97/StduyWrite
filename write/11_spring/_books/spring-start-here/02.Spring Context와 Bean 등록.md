# Spring Context와 Bean 등록
---
> Spring을 다루는 첫걸음은 Context에 객체를 올리는 일입니다. Spring은 기본적으로 우리가 만든 객체를 전혀 모르며, Context에 올려야만 비로소 관리 대상으로 인식합니다. 이 장은 Context가 무엇인지, 그리고 객체(bean)를 올리는 세 가지 방법(@Bean·스테레오타입 애너테이션·프로그래밍 방식)을 각각 언제 쓰는지 정리합니다.


## 핵심 요약

Spring Context는 프레임워크가 관리할 객체 인스턴스를 모아 두는 앱 메모리상의 공간입니다. Spring은 Context에 올라온 객체만 볼 수 있고, 트랜잭션·테스트 같은 기능도 이 Context에 객체를 올리고 객체 사이 관계를 맺는 방식으로 연결됩니다. 그래서 Context 관리법을 모르면 Spring으로 할 수 있는 일이 거의 없습니다. Context에 올린 객체를 **bean**이라 부르며, bean을 추가하는 방법은 `@Bean` 애너테이션, 스테레오타입 애너테이션(`@Component` 등), 프로그래밍 방식(`registerBean()`) 셋입니다. 실무에서는 내 클래스에는 코드가 적은 스테레오타입을 우선 쓰고, 내가 수정할 수 없는 라이브러리 클래스에는 `@Bean`을 씁니다.


## 학습 목표

> 이 내용을 읽고 나면 다음을 할 수 있습니다.

1. Spring Context가 왜 필요하고 무엇을 담는지 설명할 수 있습니다.
2. `@Bean` 애너테이션으로 bean을 등록하는 세 단계를 구현할 수 있습니다.
3. 같은 타입 bean이 여러 개일 때 이름·`@Primary`로 구분해 꺼낼 수 있습니다.
4. 스테레오타입 애너테이션(`@Component` + `@ComponentScan`)으로 bean을 등록할 수 있습니다.
5. 세 가지 등록 방식의 장단점을 비교하고 상황별로 고를 수 있습니다.


## 본문 정리


### 1. Spring Context란 무엇인가

Context(애플리케이션 컨텍스트)는 Spring이 관리할 객체를 담아 두는 메모리 공간입니다. Spring은 기본적으로 우리 객체를 하나도 모릅니다. 객체를 Context에 올려야 Spring이 그 객체를 보고, 자신이 제공하는 기능(트랜잭션·AOP 등)을 거기에 끼워 넣습니다.

> 💬 **비유**: Context는 양동이(bucket)와 같습니다. Spring이 관리해 주길 바라는 객체를 이 양동이에 넣습니다. 양동이 안에 든 것만 Spring이 볼 수 있고, 밖에 둔 객체는 아무리 잘 만들어도 Spring 입장에서는 존재하지 않는 것과 같습니다.

여기서 중요한 점이 하나 있습니다. 앱의 *모든* 객체를 Context에 올릴 필요는 없습니다. Spring이 관리해야 할 객체만 골라 올립니다. 이 장에서는 우선 "어떻게 올리는가"에 집중하고, "무엇을 올릴 것인가"는 4장부터 실전 예제에서 다룹니다.

### 2. 방법 1 — @Bean 애너테이션

`@Bean`은 셋 중 가장 이해하기 쉬운 방식이며, 내 클래스든 라이브러리 클래스든 어떤 객체도 올릴 수 있습니다. 세 단계로 진행합니다.

```java
// Step 1: @Configuration 설정 클래스 정의
@Configuration
public class ProjectConfig {

  // Step 2: bean을 반환하는 메서드에 @Bean 부착
  @Bean
  Parrot parrot() {
    var p = new Parrot();
    p.setName("Koko");
    return p;     // Spring이 이 반환값을 Context에 올림
  }
}

// Step 3: 설정 클래스를 넘겨 Context 초기화
public class Main {
  public static void main(String[] args) {
    var context =
      new AnnotationConfigApplicationContext(ProjectConfig.class);

    Parrot p = context.getBean(Parrot.class);   // 타입으로 꺼냄
    System.out.println(p.getName());            // Koko
  }
}
```

> `@Bean` 메서드 이름에는 동사를 쓰지 않습니다. 자바 관례는 메서드에 동사를 넣는 것이지만, bean 등록 메서드는 *행위*가 아니라 *반환하는 객체*를 나타냅니다. 그래서 명사(보통 클래스명과 같게)를 쓰고, **메서드 이름이 곧 bean의 이름**이 됩니다.

`@Bean`은 어떤 타입이든 올릴 수 있습니다. `String`·`Integer`처럼 내가 만들지 않은 타입도 메서드로 감싸 등록할 수 있습니다.

```java
@Bean String hello() { return "Hello"; }
@Bean Integer ten()  { return 10; }
```

#### 같은 타입 bean이 여러 개일 때

`@Bean` 메서드를 여러 개 선언하면 같은 타입 bean을 여럿 올릴 수 있습니다. 이때 타입만으로 꺼내면 Spring이 어느 것인지 못 골라 예외를 던집니다.

```java
@Bean Parrot parrot1() { var p = new Parrot(); p.setName("Koko"); return p; }
@Bean Parrot parrot2() { var p = new Parrot(); p.setName("Miki"); return p; }
@Bean Parrot parrot3() { var p = new Parrot(); p.setName("Riki"); return p; }
```

```
// context.getBean(Parrot.class) 실행 시:
NoUniqueBeanDefinitionException: No qualifying bean of type 'Parrot' available:
    expected single matching bean but found 3: parrot1,parrot2,parrot3
```

해결책은 두 가지입니다.

| 방법 | 코드 | 설명 |
|------|------|------|
| 이름으로 지정 | `context.getBean("parrot2", Parrot.class)` | bean 이름(기본값은 메서드명)으로 정확히 지목 |
| `@Primary` 지정 | `@Bean @Primary Parrot parrot2() {...}` | 이름 없이 꺼낼 때 Spring의 기본 선택. 타입당 하나만 가능 |

bean 이름을 바꾸려면 `@Bean(name = "miki")`, `@Bean(value = "miki")`, `@Bean("miki")` 중 아무 형태나 씁니다.


### 3. 방법 2 — 스테레오타입 애너테이션

스테레오타입 애너테이션은 클래스 위에 애너테이션을 붙여 "이 클래스의 인스턴스를 Context에 올려라"라고 지시하는 방식입니다. 가장 기본인 `@Component`를 씁니다. 코드가 적다는 점이 장점입니다. 두 단계로 진행합니다.

```java
// Step 1: 올릴 클래스에 @Component
@Component
public class Parrot {
  private String name;
  // getter/setter
}

// Step 2: 설정 클래스에 @ComponentScan으로 탐색 위치 지정
@Configuration
@ComponentScan(basePackages = "main")
public class ProjectConfig { }
```

`@Component`만 붙이고 `@ComponentScan`을 빠뜨리면 Spring이 클래스를 찾지 않아 bean이 생기지 않습니다. 둘은 짝입니다.

스테레오타입은 코드가 적지만 제약이 있습니다. Spring이 생성자만 호출하므로, 생성 *직후* 추가 작업(예: 이름 할당)을 하려면 `@PostConstruct`를 씁니다.

```java
@Component
public class Parrot {
  private String name;

  @PostConstruct
  public void init() {
    this.name = "Kiki";   // 생성자 실행 후 Spring이 호출
  }
}
```

> `@PostConstruct`는 Java EE에서 빌려온 애너테이션입니다. Java 11부터 JDK에서 Java EE API가 빠졌으므로, 쓰려면 `javax.annotation:javax.annotation-api` 의존성을 추가해야 합니다. 반대로 Context 종료 직전 호출되는 `@PreDestroy`도 있지만, Spring이 Context 정리에 실패하면 호출되지 않을 수 있어 DB 커넥션 닫기처럼 민감한 작업에는 권하지 않습니다.


### 4. 방법 3 — 프로그래밍 방식 (registerBean)

Spring 5부터 `registerBean()`으로 Context 인스턴스의 메서드를 직접 호출해 bean을 올릴 수 있습니다. `@Bean`이나 스테레오타입으로는 표현하기 어려운 **조건부 등록**이 필요할 때 씁니다.

> 💬 **시나리오**: 앵무새 컬렉션을 읽어, 그중 *초록색*만 Context에 올리고 싶을 때처럼 "조건에 따라 다른 bean을 등록"하는 커스텀 로직이 필요한 경우입니다.

```java
// registerBean(이름, 클래스, Supplier, BeanDefinitionCustomizer...)
Parrot x = new Parrot();
x.setName("Kiki");
Supplier<Parrot> parrotSupplier = () -> x;

context.registerBean("parrot1", Parrot.class, parrotSupplier);

// 조건부 등록 예
for (Parrot p : parrots) {
  if (p.isGreen()) {
    context.registerBean(null, Parrot.class, () -> p);
  }
}
```

네 번째 파라미터 `BeanDefinitionCustomizer`는 가변인자라 생략 가능하며, `bc -> bc.setPrimary(true)`로 primary 지정 같은 특성을 줄 수 있습니다.


### 5. 세 방식 비교 — 언제 무엇을 쓰나

| 기준 | @Bean | 스테레오타입(@Component) | registerBean |
|------|-------|--------------------------|--------------|
| 인스턴스 생성 제어 | 메서드 본문에서 완전 제어 | 생성 *후*에만 제어(`@PostConstruct`) | 완전 제어 |
| 같은 타입 다중 등록 | 가능(메서드 여러 개) | 클래스당 1개 | 가능(호출 반복) |
| 내가 안 만든 클래스 | 가능(String·Integer 등) | 불가(애너테이션을 못 붙임) | 가능 |
| 코드량 | bean마다 메서드 → 보일러플레이트 | 적음 | 로직형 |
| 주 용도 | 라이브러리 클래스 등록 | 내 앱 클래스 (1순위) | 조건부 커스텀 등록 |

실무 기준은 단순합니다. 내가 만든 클래스에는 코드가 적은 **스테레오타입을 우선** 쓰고, 수정할 수 없는 라이브러리 클래스에는 **`@Bean`**을 쓰며, 조건 분기가 필요한 특수 상황에만 **`registerBean()`**을 씁니다.


## 심화 학습

> 책은 Spring 5(`spring-context` 5.2.x) 기준입니다. 이후 변화를 보강합니다.

- **`@PostConstruct` 의존성 변화**: Spring 6 / Boot 3는 Jakarta EE 9+ 기반이라 애너테이션이 `javax.annotation`에서 `jakarta.annotation`으로 바뀌었습니다. 의존성도 `jakarta.annotation:jakarta.annotation-api`를 씁니다. 책의 `javax` 코드를 최신에서 돌리면 import가 달라집니다.
- **Spring Boot의 자동 등록**: 실무에서 가장 흔한 Spring Boot는 `@SpringBootApplication`이 `@ComponentScan`을 포함해, 메인 클래스 패키지 하위를 자동 스캔합니다. 이 장의 수동 `@ComponentScan`·`AnnotationConfigApplicationContext` 생성은 Boot가 가려 주는 동작을 손으로 드러내 보여 주는 셈입니다.
- **`@Bean` vs `@Component`의 본질**: `@Component`는 클래스 자체를 Spring이 생성하므로 단순 위임에 맞고, `@Bean`은 메서드 본문에서 객체를 직접 조립하므로 외부 라이브러리 객체나 복잡한 초기화에 맞습니다. 자바 설정(`@Configuration`)과 컴포넌트 스캔을 함께 쓰는 것이 현대 Spring의 표준 조합입니다.


## 실무 적용 포인트

### 이런 상황에서 사용하세요

- 내가 정의한 서비스·리포지토리 클래스를 bean으로 올릴 때 → `@Component` 계열(`@Service`·`@Repository`)
- `RestTemplate`·`ObjectMapper`처럼 라이브러리가 제공하는 객체를 설정해 올릴 때 → `@Bean`
- 설정값·환경에 따라 다른 구현체를 골라 등록해야 할 때 → `registerBean()` 또는 `@Conditional`

### 주의할 점

- ⚠️ 같은 타입 bean이 여럿이면 타입만으로 꺼낼 때 `NoUniqueBeanDefinitionException`이 납니다. 이름이나 `@Primary`로 모호함을 없애야 합니다.
- ⚠️ `@Component`만 붙이고 `@ComponentScan` 범위를 지정하지 않으면 bean이 생기지 않습니다.
- ⚠️ `@PreDestroy`는 Context 정리 실패 시 호출되지 않을 수 있어, 자원 해제 같은 필수 작업에는 의존하지 않는 편이 안전합니다.


## 면접 대비

### 한 줄 정의

"Spring Context란 프레임워크가 관리할 객체(bean)를 담아 두는 메모리 공간이며, Context에 올린 객체만 Spring이 보고 관리합니다."

### 핵심 포인트 3가지

1. bean 등록 방법은 `@Bean`·스테레오타입·`registerBean()` 셋이며, 각각 제어 범위와 코드량이 다릅니다.
2. 내 클래스에는 스테레오타입, 라이브러리 클래스에는 `@Bean`, 조건부 등록에는 `registerBean()`이 기본 선택입니다.
3. 같은 타입 bean이 여럿이면 이름이나 `@Primary`로 구분해 꺼냅니다.

### 자주 묻는 질문

Q: `@Bean`과 `@Component`의 차이는 무엇인가요?
A: `@Component`는 클래스 위에 붙여 Spring이 그 클래스를 직접 생성·등록하므로 코드가 적지만 내 클래스에만 쓸 수 있습니다. `@Bean`은 설정 클래스 메서드에서 객체를 직접 만들어 반환하므로 라이브러리 객체도 올릴 수 있고 생성 과정을 완전히 제어합니다.

Q: `@Bean` 메서드 이름에 동사를 안 쓰는 이유는?
A: 그 메서드는 행위가 아니라 *반환하는 bean*을 나타내고, 메서드 이름이 곧 bean의 이름이 되기 때문입니다.

Q: 같은 타입 bean이 여러 개일 때 어떻게 특정 bean을 꺼내나요?
A: `getBean("이름", 타입.class)`로 이름을 지정하거나, 하나를 `@Primary`로 지정해 기본 선택이 되게 합니다.
