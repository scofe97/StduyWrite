# 테스트 피라미드와 Spring 테스트 종류

## 학습 목표

이 문서를 읽고 나면 다음을 할 수 있습니다.

1. 단위·슬라이스·통합·E2E 네 층의 책임 분장을 비용·속도·신뢰도 세 축에서 비교하고 어느 결함이 어느 층에서 잡히는지 설명할 수 있습니다.
2. 새 테스트를 추가할 때 결정 트리 3 질문으로 어느 카테고리에 둘지 빠르게 결정할 수 있습니다.
3. Gradle `test` / `integrationTest` task 분리 패턴(이중 차단·빌드 산출물 재사용·환경 가드) 을 적용해 빌드 시간 폭증을 차단할 수 있습니다.
4. 테스트 작성이 어렵다는 신호(Mock 비대·결정성 깨짐·셋업 비대) 를 *설계 개선* 신호로 해석하고 어느 리팩토링을 트리거할지 판단할 수 있습니다.



---

> Spring Boot 프로젝트는 단위·슬라이스·통합·E2E 네 층의 테스트가 한 그릇에 섞이면 빌드가 빠르게 분 단위로 무거워집니다. 본 챕터는 어느 층이 어떤 결함을 잡는지 비용·속도·신뢰도 축에서 정리하고, Spring 의 어노테이션·네이밍·Gradle task 분리를 함께 묶습니다.



## 한 줄 정의

테스트 피라미드는 단위→슬라이스→통합→E2E 의 네 층으로 검증 책임을 나누는 사고 도구이며, Spring Boot 에서는 각 층이 명확한 도구·어노테이션·실행 task 와 일대일로 매핑됩니다.



## 테스트는 구현의 첫 사용자

피라미드 그림을 그리기 전에 한 가지 관점을 먼저 박습니다. 테스트 코드는 *검증 도구이기 전에 구현 코드의 첫 사용자*다. 카카오페이 Mock Part 2 는 다음과 같이 정리합니다.

> "테스트 코드는 단순한 검증 도구가 아니라 구현 코드의 *첫 사용자 역할*을 한다. 테스트 코드 작성이 어렵다면 구현 코드 품질에 문제가 있다는 신호다."

이 관점이 시리즈 전체의 출발점입니다. 테스트가 어려운 이유를 분해하면 보통 셋 중 하나입니다.

- 의존성이 많아 *Mock 셋업이 거대*해진다 → 책임 분리 신호
- 외부 시스템에 직접 닿아 *결정성이 깨집니다* → 어댑터 분리 신호
- 데이터 셋업이 길어 *핵심 검증이 가려집니다* → 픽스처 추출 신호

테스트가 어려울 때 가장 위험한 대응은 *테스트 편의를 위해 운영 코드를 바꾸는 것*입니다. 카카오페이 Mock Part 1 이 명시한 안티패턴입니다.

> "테스트를 편리하게 하기 위해 운영 코드 설계를 변경하는 것은 바람직하지 않다."

대신 *테스트가 어렵게 만드는 구조 자체*를 다시 봅니다. 책임을 가르고, 외부 의존성을 인터페이스로 추출하고, 도메인 로직을 POJO 로 뽑아냅니다. 그러면 테스트는 자연스럽게 쉬워집니다. 피라미드의 네 층은 이 피드백 루프 위에 얹는 *운영 도구*다 — 어느 층이 어느 결함을 잡는지를 알아야 신호를 어디로 보낼지 결정됩니다.



## 왜 필요한가

> 모든 테스트를 한 층에 몰면 비용·속도·신뢰도 중 둘이 동시에 무너집니다. 층을 의식적으로 나누는 이유가 여기에 있습니다.

빌드가 5분에서 30분으로 늘어나는 사고는 보통 통합 테스트가 단위 테스트와 같은 task 에서 도는 시점에 시작됩니다. PR 한 건당 Testcontainers·EmbeddedKafka 가 매번 뜨고, IT 한 건이 깨지면 PR 빌드가 컨테이너 기동 실패로 몇 분 헛돌며, 결국 누군가가 `@Disabled` 를 붙여 회로를 끊어 놓습니다. 회피가 한 번 시작되면 빠르게 퍼집니다.

반대 방향의 사고도 흔합니다. 모든 결함을 단위 테스트로 잡으려 하면 mock 이 누적되어 가짜 신뢰가 만들어집니다. JPA 매핑·트랜잭션 경계·자동 설정 충돌 같은 결합 결함은 컨텍스트가 떠야만 드러나는데, 그것을 mock 으로 흉내 내면 실제 운영과 다른 신호를 받습니다.

피라미드는 이 두 방향의 사고를 동시에 막습니다. 어느 층이 어떤 종류의 결함을 잡는지를 알면, 새 테스트를 추가할 때 어디에 두어야 빌드가 무겁지 않고 가짜 신뢰가 만들어지지 않는지가 빠르게 결정됩니다.

층별로 잡는 결함을 짚으면 다음과 같습니다.

1. **도메인 객체의 산술·정책 분기**는 단위 테스트가 가장 정확합니다. 입력에서 출력까지가 순수한 함수면 1초 미만에 100 케이스를 돌릴 수 있습니다.
2. **컨트롤러의 검증 어노테이션·ControllerAdvice 결합**은 슬라이스에서 잡습니다. 단위 테스트로는 검증 어노테이션이 동작하지 않습니다.
3. **JPA 매핑의 컬럼 타입·트랜잭션 경계**는 통합에서 잡습니다. H2 호환 모드로는 실제 운영 DB 와의 갭이 가려집니다.
4. **외부 API 응답 형식 변화·카오스·복구**는 E2E 에서 잡습니다. mock 으로 흉내 내면 실제 변화를 발견할 길이 없습니다.



## 아키텍처

### 피라미드 4층 구조

Mike Cohn 의 원형은 아래로 갈수록 격리가 좋아 빠르고 안정적이며, 위로 갈수록 실제 사용자 경로에 가까워 신뢰도가 올라가는 모양입니다.

```
        /\
       /E2\        외부 시스템 결합 — 실제 환경 또는 stub 서버
      /----\       비용↑ 속도↓ 신뢰도↑
     / IT   \      Spring 컨텍스트 + DB/Kafka 등 인프라 기동
    /--------\     비용·속도·신뢰도 중간
   /  Slice   \    웹/JPA/메시지 슬라이스 — 컨텍스트 일부만
  /------------\   비용↓ 속도↑ 부분 신뢰도
 /  Unit / POJO \  컨텍스트 없음 — 도메인 로직, 정책 객체
/----------------\ 비용↓↓ 속도↑↑ 좁은 신뢰도
```

피라미드 모양이 강조되는 이유는 **위층이 무거워질수록 아래층이 지지대 역할을 하지 못합니다**는 점 때문입니다. 단위 테스트가 빈약하면 통합 테스트가 도메인 검증까지 떠안고, 그러면 통합 테스트는 한 건당 수십 초로 늘어나며 깨질 때마다 디버깅 비용이 급등합니다.

다음 다이어그램은 같은 4 층 구조를 *어느 결함을 잡는지* 와 함께 한 장에 옮긴 것입니다. 위 ASCII 박스가 *비용·속도·신뢰도* 축을 보여 준다면, 이 그림은 *각 층이 책임지는 결함 종류* 를 명시합니다.

```mermaid
flowchart TB
    e2e["E2E<br>외부 응답 변화·카오스·복구"]
    it["Integration<br>트랜잭션 경계·자동 설정 결합"]
    sl["Slice<br>컨트롤러 검증 어노테이션·ControllerAdvice"]
    unit["Unit / POJO<br>정책 분기·도메인 산술"]

    e2e --> it --> sl --> unit

    note_e2e["mock 으로 흉내 불가:<br>실제 외부 호출 또는 stub"]:::note
    note_it["H2 호환 모드 불가:<br>실 DB·실 브로커"]:::note
    note_sl["단위로 불가:<br>검증 어노테이션·필터 활성화"]:::note
    note_unit["수십 ms / 100 케이스:<br>컨텍스트 없음"]:::note

    e2e -.-> note_e2e
    it -.-> note_it
    sl -.-> note_sl
    unit -.-> note_unit

    classDef topBox fill:#FFB6C1,color:#000,stroke:#8B0000,stroke-width:1px
    classDef midUpBox fill:#FFA500,color:#000,stroke:#8B4513,stroke-width:1px
    classDef midDnBox fill:#ADD8E6,color:#000,stroke:#00008B,stroke-width:1px
    classDef botBox fill:#90EE90,color:#000,stroke:#006400,stroke-width:1px
    classDef note fill:#FFFFE0,color:#000,stroke:#8B8000,stroke-width:1px
    class e2e topBox
    class it midUpBox
    class sl midDnBox
    class unit botBox
```

이 그림이 보여 주는 것은 *각 층이 잡는 결함이 다른 층으로 옮겨질 수 없다* 는 점입니다. JPA 매핑 결함을 단위 테스트로 흉내 내려면 mock 이 누적되어 가짜 신뢰가 만들어지고, 도메인 정책 분기를 E2E 로 검증하면 한 분기당 수십 초가 걸려 회귀 행렬이 폭발합니다. 결함 종류에 맞는 층을 고르는 것이 *피라미드 도구의 본질* 입니다.

### Spring 도구 매핑

Spring Boot 가 제공하는 테스트 도구를 4 층에 매핑하면 다음과 같습니다.

| 층 | Spring 도구 | 컨텍스트 기동 | 외부 의존 | 잡는 결함 예시 |
|----|-------------|--------------|----------|---------------|
| 단위 | JUnit 5 + Mockito + AssertJ. Spring 의존 없음 | 없음 | 없음 | 정책 객체의 분기, 도메인 산술, 입력 검증 헬퍼 |
| 슬라이스 | `@WebMvcTest`, `@DataJpaTest`, `@JsonTest`, `@RestClientTest`, `MockMvcBuilders.standaloneSetup` | 부분 컨텍스트 | 없음(Mock) | 컨트롤러 검증 어노테이션, JPA 매핑/쿼리, JSON (역)직렬화 |
| 통합 | `@SpringBootTest` + Testcontainers / `@EmbeddedKafka` / `ApplicationContextRunner` | 전체 또는 다대 슬라이스 | 진짜 DB/Kafka | 트랜잭션 경계, 자동 설정 충돌, 다중 빈 결합 |
| E2E | `@SpringBootTest(webEnvironment=RANDOM_PORT)` + 실제/stub 서버 (WireMock·MockServer) | 전체 | 외부 API stub 또는 실제 | 외부 응답 변화, 큐·재시도·복구 흐름, 카오스 시나리오 |

Spring 5/Boot 2.x 시절의 `@RunWith(SpringRunner.class)` 는 JUnit 5 에서 `@ExtendWith(SpringExtension.class)` 로 바뀌었고, `@SpringBootTest` 가 이를 메타 어노테이션으로 흡수하므로 Boot 사용자는 직접 쓸 일이 거의 없습니다. 슬라이스 어노테이션의 핵심 차이는 어떤 자동 설정만 제한적으로 켜는가이며, 자세한 내용은 01-04 에서 다룹니다.



## 핵심 개념

### 네이밍 컨벤션 — `*Test` / `*IT` / `*E2ETest`

같은 폴더에 모든 테스트를 두면 IDE 단축키 한 번에 단위와 통합이 같이 돌고, CI 는 PR 마다 인프라를 띄웁니다. 이를 분리하는 가장 가벼운 도구가 파일명 접미사입니다.

| 접미사 | 의미 | 위치 | 비용 |
|--------|------|------|------|
| `*Test.java` | 단위 또는 컨텍스트 없는 슬라이스 | `src/test/java/` | 1초 미만 |
| `*IT.java` | 통합 — Spring 컨텍스트 + 인프라(Testcontainers/EmbeddedKafka) | `src/test/java/` | 수 초 ~ 수십 초 |
| `*E2ETest.java` | 외부 시스템 결합 또는 풀 컨텍스트 시나리오 | `src/test/java/` | 수십 초 ~ 분 |

Maven Failsafe Plugin 의 기본 패턴(`*IT.java`/`IT*.java`/`*ITCase.java`)을 따르는 편이 호환성 면에서 가장 안전합니다. Gradle 에서는 자체 task 정의로 분리하므로 패턴은 자유롭지만, 표준을 따르면 IDE 와 빌드 도구 양쪽에서 자동 인식됩니다.

JUnit 5 `@Tag` 는 파일명과 별개의 분류 축을 제공합니다. 같은 클래스 내 일부 테스트만 통합으로 돌리고 싶을 때 유용하지만, 가장 단순한 운영은 **클래스 단위로 접미사 + `@Tag` 를 이중으로 표시**하는 것입니다. 이중 표시의 가치는 빌드 도구가 파일명 패턴으로 1차 필터하고 JUnit Platform 이 태그로 2차 필터해, 한쪽 설정이 빠져도 다른 쪽이 안전망이 된다는 점에 있습니다.

### 컨텍스트 캐시 — 양날의 검

`@SpringBootTest` 는 같은 어노테이션 조합을 가진 테스트끼리 컨텍스트를 공유합니다. 100 개 테스트가 같은 설정이면 컨텍스트가 1 개만 떠 빌드가 빠릅니다. 그러나 `@MockBean` 이나 `@DirtiesContext` 가 섞이면 캐시 키가 갈라지면서 컨텍스트가 여러 개 떠 OOM 으로 이어질 수 있습니다.

통합 테스트가 늘어나기 시작하면 빌드 로그에서 `Application context starts/stops` 횟수를 한 번씩 확인하는 습관이 도움이 됩니다. 한 빌드에 컨텍스트가 10 개 이상 뜨면 `@MockBean` 사용처를 줄일 후보입니다. 자세한 내용은 01-04 에서 다룹니다.

### 결정 트리 — 어느 카테고리를 고를지

새 테스트를 추가할 때 어느 층에 둘지가 매번 결정 비용입니다. 다음 결정 트리는 비용·신뢰도 트레이드오프를 *세 가지 질문*으로 압축합니다.

```mermaid
flowchart TD
    Q1{외부 의존성<br/>(DB/HTTP/Kafka)에<br/>닿는가?}
    Q1 -- "아니오" --> UNIT["Unit 테스트<br/>(POJO + Mockito mock)"]
    Q1 -- "예" --> Q2{흐름이<br/>다수 빈에<br/>걸쳐 있는가?}
    Q2 -- "아니오" --> SLICE["Slice 테스트<br/>(@WebMvcTest 등)"]
    Q2 -- "예" --> Q3{외부 시스템을<br/>실제로 띄워야<br/>하는가?}
    Q3 -- "아니오" --> INTEG["Integration<br/>(@SpringBootTest + Mock)"]
    Q3 -- "예" --> E2E["E2E<br/>(Testcontainers + WireMock)"]

    style UNIT fill:#e8f6e8,stroke:#3a8a3a,color:#000
    style SLICE fill:#e8f4fd,stroke:#1f78b4,color:#000
    style INTEG fill:#fef9e7,stroke:#b07d12,color:#000
    style E2E fill:#fce8e8,stroke:#a83232,color:#000
```

기본은 *가장 가벼운 카테고리*에서 시작합니다. 검증이 부족하면 한 단계씩 올립니다. 거꾸로, Integration·E2E 로 짠 테스트 중 *Unit 으로 내려갈 수 있는 것*이 있으면 분리합니다. 가벼운 테스트가 많을수록 피드백 루프가 빨라집니다.

원칙을 한 줄로 압축하면 "*확신을 얻기에 충분한 최소 비용*"입니다. 대체로 위에서 아래(가벼운 쪽)로 미는 것이 옳지만, *외부 시스템과의 계약*이 본질인 코드는 처음부터 Integration 이 옳습니다. Outbox 폴러의 `FOR UPDATE SKIP LOCKED` 동작은 실 DB 가 아니면 검증이 안 되고, Kafka 의 transactional commit 동작은 단위 테스트로 잡히지 않습니다. *그 최소가 Integration 이라면 그게 가장 가벼운 카테고리*다.



## 실습 — Gradle test / integrationTest task 분리

`./gradlew build` 했을 때 단위 테스트만 실행되고 통합은 별도 명령에서만 도는 구조가 표준입니다. TPS operator/ticket 모듈의 `build.gradle` 이 이 분리를 명시적으로 보여 줍니다.

```groovy
tasks.named('test', Test).configure {
    useJUnitPlatform {
        excludeTags 'integration'
    }
    exclude '**/*IT.class'
}

tasks.register('integrationTest', Test) {
    description = 'Runs ticket module integration tests.'
    group = 'verification'

    testClassesDirs = sourceSets.test.output.classesDirs
    classpath = sourceSets.test.runtimeClasspath
    shouldRunAfter tasks.named('test')

    useJUnitPlatform {
        includeTags 'integration'
    }
    include '**/*IT.class'

    environment 'TESTCONTAINERS_DOCKER_SOCKET_OVERRIDE', testcontainersDockerSocketOverride.get()

    doFirst {
        if (testcontainersRyukDisabled.get().equalsIgnoreCase('true')) {
            throw new GradleException('TESTCONTAINERS_RYUK_DISABLED must not be true for integrationTest.')
        }
    }
}
```

핵심은 세 가집니다.

1. **이중 차단**. `test` task 는 `excludeTags 'integration'` + `exclude '**/*IT.class'` 로 통합 테스트를 두 방식으로 동시에 막습니다. 한쪽 설정이 빠져도 다른 쪽이 안전망입니다.
2. **빌드 산출물 재사용**. `integrationTest` 는 별도 task 로 등록하되 `testClassesDirs` 와 `classpath` 는 `test` 와 공유해, 컴파일을 두 번 하지 않습니다.
3. **환경 가드**. `doFirst` 블록에 `TESTCONTAINERS_RYUK_DISABLED=true` 같은 위험 설정을 빌드 시작 시점에 차단합니다. Ryuk 가 꺼지면 컨테이너가 시스템에 남아 다음 빌드가 포트 충돌로 깨지는 일이 잦습니다. 빌드 스크립트가 코드라는 사실을 이용해 운영 사고를 가장 값싼 단계에서 잡습니다.

### TPS 모듈 분장 사례

TPS 코드베이스는 세 모듈에 걸쳐 총 311 개 테스트 파일을 운영합니다. 모듈별 비중과 주된 카테고리가 *모듈 책임의 거울*이 되는 분포입니다.

| 모듈 | 테스트 파일 수 | 주된 카테고리 | 특이 사항 |
|------|--------------|-------------|----------|
| **executor** | 76 | Unit + E2E | Jenkins E2E + WireMock 중심. Slice 거의 없음 |
| **message-lib** | 10 | Unit + Integration(IT) | Outbox·Kafka 설정 검증. 작지만 인프라 핵심 |
| **operator** | 225 | Integration | ticket 도메인 중심, Testcontainers MariaDB 다수 |
| **합계** | **311** | | |

이 분포가 만들어진 이유는 모듈의 *책임*과 일치합니다. `executor` 는 *외부 시스템(Jenkins) 과의 통신*이 본질이라 E2E + WireMock 으로 흐름을 검증합니다. `message-lib` 는 *인프라 라이브러리*라 단위 테스트로 동작을 박고, IT 로 Kafka·DB 결합을 검증합니다. `operator` 는 *도메인 로직과 영속성*이 본질이라 Integration 이 다수입니다. ticket 도메인이 가장 무겁습니다.

Slice 가 거의 없는 이유는 TPS 가 *애플리케이션 부팅 시간이 짧고* `@SpringBootTest` 전체 로드 비용이 *허용 범위 안*이기 때문입니다. Slice 는 Context 로드 시간이 *분 단위로 커진 큰 모놀리스*에서 가치를 발휘합니다. 모듈별로 빈 수가 크게 늘거나 한 PR 이 100 개 이상 테스트를 실행해 *부팅 시간이 누적 문제*가 되면 그때 도입합니다.

operator/ticket 모듈은 약 70 개 테스트를 다음 비율로 운영합니다.

- **단위·슬라이스** (`*Test.java`): `TcktMngControllerTest`(standalone MockMvc + ControllerAdvice), `WebLoggingFilterTest`(필터 + Logback ListAppender) 같은 컨텍스트 없는 빠른 테스트가 다수.
- **통합** (`*IT.java`): `WorkflowApiCreateIT` 가 `WorkflowApiTestSupport` 를 거쳐 `AbstractMariaDbIntegrationTest`(Testcontainers MariaDB + `@ServiceConnection`) 를 상속하며, `@Tag("integration")` 으로 이중 표시.
- **스모크** (`MariaDbContainerSmokeIT.java`): 컨테이너 기동 자체만 검증해, 인프라 깨짐을 빠르게 분리.

executor/engine 은 같은 컨벤션 위에 추가 접미사 `*E2ETest.java` 를 둡니다. `JenkinsConnectionE2ETest`(외부 Jenkins 가용성 ping), `SubmittedRecoveryE2ETest`(WireMock 으로 Jenkins API stub + 스케줄러 수동 호출), `JenkinsDownChaosE2ETest`(다운 시뮬레이션) 처럼 외부 시스템 결합 또는 카오스 시나리오를 분리합니다. E2E 라는 이름이 과도하게 보일 수 있지만, "외부 시스템 응답 형식 변화"는 실제로 단위·통합으로는 잡히지 않는 결함이며, 카오스·복구 시나리오는 별도 라이프사이클로 관리될 가치가 있습니다.

한국어 `@DisplayName` + `ReplaceUnderscores` + `[Green]/[Red]` 접두사는 TPS 컨벤션이지만, 다른 팀에도 추천할 만합니다. 메서드명이 길어지더라도 한국어 DisplayName 이 IDE 트리에 그대로 보이고, `[Green]`/`[Red]` 가 happy/sad path 를 색깔로 인식하게 합니다.

```java
@DisplayName("[Red] 선택 업무코드 워크플로우에 업무코드 목록이 비어 있으면 생성을 거부하고 저장소를 변경하지 않는다")
void createWorkflowRejectsSelectedTaskWorkflowWithoutTaskCodes() throws Exception { ... }
```



## 함정과 회피

> 처음 운영하면 디버깅에 시간이 가는 항목들입니다.

### 1. 빌드 시간 폭증

통합 테스트가 단위 테스트와 같은 task 에서 돌 때 가장 흔히 발생합니다. PR 한 건당 컨테이너가 매번 뜨면 5 분이던 빌드가 30 분이 되며, 개발자는 PR 을 작은 단위로 쪼개기를 포기합니다. 위 task 분리가 이 사고를 막습니다.

### 2. Testcontainers 컨테이너 누수

Ryuk(`testcontainers/ryuk`) 사이드카가 꺼졌을 때 발생합니다. JVM 이 비정상 종료하면 컨테이너가 시스템에 남고, 다음 빌드가 포트 충돌로 깨집니다. CI 환경에서 Ryuk 을 끄고 싶은 유혹이 있어도, 위 가드처럼 명시적으로 차단해 두는 편이 운영을 안정화합니다.

### 3. 컨텍스트 캐시 OOM

`@SpringBootTest` 의 컨텍스트 캐시는 양날의 검입니다. `@MockBean` 이나 `@DirtiesContext` 가 섞이면 캐시 키가 갈라지면서 컨텍스트가 여러 개 떠 OOM 으로 이어질 수 있습니다. 빌드 로그에서 `Application context starts/stops` 횟수를 모니터링합니다.

1. 테스트 피라미드의 네 층은 어떤 트레이드오프 축에서 갈리는가?
2. `@SpringBootTest` 의 컨텍스트 캐시가 양날의 검인 이유는 무엇인가?
3. Gradle `test` task 와 `integrationTest` task 를 분리하는 이유와 핵심 설정 두 가지는?
4. Testcontainers Ryuk 을 끄면 안 되는 이유는 무엇인가?
5. `*Test`/`*IT`/`*E2ETest` 네이밍 컨벤션이 단순한 접미사 이상의 의미를 갖는 이유는?
6. 테스트가 어렵다는 신호를 어떻게 *설계 개선*으로 연결하는가?
7. TPS 에 Slice 테스트가 거의 없는 이유는?
8. "가장 가벼운 카테고리에서 시작하라"는 원칙이 항상 옳은가?



## 자가 점검 — 정답

1. 비용·속도·신뢰도 세 축에서 갈립니다. 아래층은 외부 의존이 없고 격리가 좋아 빠르고 결정적이며, 위층은 실제 사용자 경로에 가까워 신뢰도가 올라가는 대신 비용과 시간이 누적됩니다. 한 층이 다른 층을 대체하지 못하므로 *층별 분장* 자체가 핵심이며, 어느 결함을 어느 층에서 잡을지를 결정하는 사고 도구로 쓰입니다.
2. 같은 어노테이션 조합을 가진 테스트끼리 컨텍스트를 공유해 빌드를 빠르게 만들지만, `@MockBean`·`@DirtiesContext`·`@TestPropertySource` 같은 요소가 섞이면 캐시 키가 갈라져 여러 컨텍스트가 동시에 뜹니다. 그러면 메모리가 누적되어 OOM 으로 이어지므로, "빌드가 빨라지는 도구" 가 곧 "OOM 의 원인" 이 되는 양면성이 있습니다.
3. 통합 테스트가 PR 빌드에 같이 돌면 컨테이너 기동 비용이 누적되어 빌드가 분 단위로 늘기 때문입니다. 핵심 두 설정은 (1) `excludeTags 'integration'` + `exclude '**/*IT.class'` *이중 차단* — 한쪽 설정이 빠져도 다른 쪽이 안전망이 되도록, (2) `shouldRunAfter tasks.named('test')` 로 *순서 강제* — 단위 실패 시 통합이 이어 도는 비용을 막습니다.
4. JVM 이 비정상 종료하면 Ryuk sidecar 가 남은 컨테이너를 청소하는데, 이걸 끄면 컨테이너가 시스템에 그대로 남아 다음 빌드가 같은 포트 점유로 깨집니다. 가장 값싼 방어는 빌드 스크립트의 `doFirst` 블록에서 `TESTCONTAINERS_RYUK_DISABLED=true` 환경변수가 들어오면 `GradleException` 을 던져 시작 시점에 차단하는 것입니다.
5. Maven Failsafe Plugin 의 기본 패턴(`*IT.java`/`IT*.java`/`*ITCase.java`) 과 Gradle `useJUnitPlatform` 의 include/exclude 패턴이 이 접미사를 그대로 인식해 빌드 도구가 자동 분리하기 때문입니다. JUnit 5 `@Tag` 와 *이중 표시* 하면 빌드 도구가 파일명으로 1차 필터하고 JUnit Platform 이 태그로 2차 필터해, 한쪽 설정이 빠져도 다른 쪽이 안전망이 됩니다.
6. 세 갈래로 분해해 *운영 코드 구조* 의 신호로 읽습니다. Mock 셋업이 거대해지면 책임이 너무 많다는 신호이므로 클래스를 가르고, 외부 시스템에 직접 닿아 결정성이 깨지면 port-adapter 가 없다는 신호이므로 인터페이스로 추출하고, 데이터 셋업이 길어 검증이 가려지면 fixture 가 없다는 신호이므로 DomainFixture 를 도입합니다. 테스트를 쉽게 만들려고 운영 코드 설계를 바꾸는 것이 아니라, 테스트의 어려움을 *설계 개선 트리거* 로 쓰는 방향입니다.
7. ApplicationContext 로드 시간이 허용 범위 안이고 `@SpringBootTest` 를 그대로 써도 빠르기 때문입니다. Slice 어노테이션(`@WebMvcTest` 등) 은 Context 부팅이 분 단위로 커진 큰 모놀리스에서 가치를 내는데, 지금 규모에서는 도구 학습 곡선과 어노테이션 조합 다양화의 캐시 갈라짐 비용이 절감 효과보다 큽니다. 부팅 시간이 누적 문제가 되면 그때 도입을 검토합니다.
8. 대체로 옳지만 *외부 시스템과의 계약* 이 본질인 코드는 예외입니다. Outbox 폴러의 `FOR UPDATE SKIP LOCKED` 같은 락 동작은 실 DB 가 아니면 검증이 안 되고, Kafka 의 transactional commit 은 실 브로커가 아니면 흉내 낼 수 없습니다. 원칙을 더 정확히 표현하면 *"확신을 얻기에 충분한 최소 비용"* 이며, 그 최소가 Integration 이라면 그게 그 코드의 가장 가벼운 카테고리입니다.
