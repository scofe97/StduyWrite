# Retry — exponential backoff·jitter·재시도 폭주 방지

---

> Retry 는 *일시적 실패* (네트워크 일시 단절, 일시 5xx, 데드락) 를 *N 번 다시 시도* 하는 패턴입니다. 단순해 보이지만 *백오프 알고리즘*, *jitter*, *Circuit Breaker 와의 조합* 까지 가야 *재시도 폭주 (retry storm)* 같은 사고를 막을 수 있습니다. 본 편은 *4가지 백오프 모델* 과 *jitter 가 푸는 thundering herd 문제*, *Retry × Circuit Breaker 의 데코레이터 순서가 결과를 어떻게 바꾸는지* 까지 봅니다.



## Retry 가 답이 되는 자리

> 모든 실패에 Retry 가 답은 아닙니다. *일시적이고 재시도 시 다른 결과가 기대되는* 자리에 한정됩니다.

**Retry 가 답인 자리** — 네트워크 일시 끊김 (TCP reset), 일시 503 (다운스트림 일시 과부하), 데드락 (DB), 일시 타임아웃.

**Retry 가 답이 아닌 자리** — 400 Bad Request (입력이 잘못된 것이라 재시도해도 같음), 401 Unauthorized (인증 만료), 422 Unprocessable Entity (비즈니스 거절). *재시도해도 같은 결과* 인 실패에 Retry 를 걸면 *자원만 낭비* 됩니다.

`retryExceptions` 와 `ignoreExceptions` 가 이 구분을 표현합니다.

```java
RetryConfig config = RetryConfig.custom()
        .maxAttempts(3)
        .retryExceptions(IOException.class, TimeoutException.class)
        .ignoreExceptions(BusinessException.class)
        .build();
```

`retryExceptions` 에 명시한 예외만 재시도 대상. `ignoreExceptions` 의 예외는 *바로 던지고 끝*. 둘 다 명시 안 된 예외는 *디폴트 재시도 대상* 입니다.



## maxAttempts 와 waitDuration — 가장 단순한 설정

> 재시도 *횟수* 와 *간격* 두 값이 기본입니다.

```yaml
resilience4j:
  retry:
    instances:
      backendA:
        maxAttempts: 3
        waitDuration: 500ms
```

`maxAttempts: 3` 은 *최초 호출 1번 + 재시도 2번 = 총 3번* 시도입니다 (Resilience4j 의 정의). 디폴트 3.

`waitDuration: 500ms` 는 *각 재시도 사이의 고정 간격* 입니다. *fixed wait* 모델.

이 단순 설정의 한계는 두 가지입니다.

**1. 짧은 일시 장애에도 긴 대기** — 다운스트림이 100ms 만에 회복했는데 500ms 대기는 *낭비*.
**2. 긴 일시 장애에는 부족** — 다운스트림이 5초 안 돌아오면 *3번 다 실패*. 시간이 더 필요한 경우.

이 한계를 *exponential backoff* 가 받칩니다.



## Exponential Backoff — 지수적 증가

> 재시도마다 *간격을 배수로 늘립니다*. 첫 시도는 짧게, 후속 시도는 점점 길게.

```java
RetryConfig config = RetryConfig.custom()
        .maxAttempts(5)
        .intervalFunction(IntervalFunction.ofExponentialBackoff(500, 2))
        .build();
```

`ofExponentialBackoff(500, 2)` 가 *초기 500ms, 배수 2* 의 설정입니다. 간격이 다음과 같이 진행됩니다.

| 재시도 | 대기 시간 |
|--------|---------|
| 1차 | 500ms |
| 2차 | 1000ms |
| 3차 | 2000ms |
| 4차 | 4000ms |

총 대기 7.5초 + 호출 시간. 초기 빠른 재시도로 *짧은 일시 장애* 를 잡고, 후속 긴 대기로 *복구 시간이 필요한* 장애도 받칩니다.

YAML 도 동일하게 지원합니다.

```yaml
resilience4j:
  retry:
    instances:
      backendA:
        maxAttempts: 5
        waitDuration: 500ms
        exponentialBackoffMultiplier: 2
```



## Jitter — Thundering Herd 방지

> Exponential Backoff 에 *결정적* (deterministic) 간격을 쓰면 *같은 시점에 실패한 N 개 클라이언트가 같은 시점에 재시도* 합니다. *Thundering Herd* 문제입니다.

상황을 그림으로 봅니다. 다운스트림이 *T 시점에 다운* → *100 개 클라이언트가 동시에 실패* → *모두 500ms 후 재시도* → *T+500ms 시점에 100 개 호출이 동시 도달* → 다운스트림이 *복구 중이지만 100 개 동시 호출에 다시 다운*.

이 패턴이 *재시도 폭주 (retry storm)* 의 핵심입니다. 다운스트림 복구를 *재시도 자체가 방해* 합니다.

답은 *jitter* — 각 재시도 간격에 *무작위 노이즈* 를 추가합니다.

```java
RetryConfig config = RetryConfig.custom()
        .maxAttempts(5)
        .intervalFunction(IntervalFunction.ofExponentialRandomBackoff(
                500,    // 초기 간격
                2,      // 배수
                0.5     // randomization factor (50% 변동)
        ))
        .build();
```

`ofExponentialRandomBackoff(500, 2, 0.5)` 가 *지수 백오프 + ±50% 무작위 노이즈* 를 의미합니다. 1차 재시도가 *250ms ~ 750ms* 사이 무작위. 100 개 클라이언트의 재시도가 *250ms ~ 750ms 사이 균등 분포* 되어 *동시 도달* 이 분산됩니다.

YAML 표현:

```yaml
resilience4j:
  retry:
    instances:
      backendA:
        maxAttempts: 5
        waitDuration: 500ms
        exponentialBackoffMultiplier: 2
        randomizedWaitFactor: 0.5
```

> *jitter 없는 Retry 는 운영에서 거의 권장 안 됩니다*. 1~2 개 클라이언트 환경이라면 차이 없지만, 트래픽이 있는 운영 환경에서는 *thundering herd* 가 다운스트림 복구를 방해합니다.



## 4가지 백오프 알고리즘 비교

| 알고리즘 | 의미 | 적합한 자리 |
|---------|------|----------|
| Fixed | 매 재시도 같은 간격 | 단순 테스트, 클라이언트 1개 |
| Exponential | 간격이 배수로 증가 | 단일 클라이언트, 짧은+긴 장애 모두 |
| Exponential + Jitter | 지수 + 무작위 노이즈 | *대부분의 운영 환경의 표준* |
| Custom IntervalFunction | 임의의 함수 | 정해진 시간표 따르는 특수 케이스 |

운영 환경의 *디폴트 선택* 은 *Exponential + Jitter* 입니다. 다른 알고리즘이 명확한 이유로 답이 되는 자리만 별도 선택합니다.



## Retry × Circuit Breaker — 데코레이터 순서

> *Retry 가 안쪽, Circuit Breaker 가 바깥쪽* 이라는 [`./01-01`](01-01.Resilience4j%20개요%20—%205가지%20모듈과%20도입%20결정.md) 의 원칙이 여기서 구체화됩니다.

```java
Supplier<String> decoratedSupplier = Decorators.ofSupplier(supplier)
        .withCircuitBreaker(circuitBreaker)
        .withBulkhead(bulkhead)
        .withRetry(retry)               // 가장 안쪽
        .decorate();
```

위 데코레이터에서 호출 흐름은 다음과 같습니다.

1. Circuit Breaker 가 *호출 시도를 허용* (서킷이 CLOSED 또는 HALF_OPEN 이면)
2. Bulkhead 가 *동시 호출 수* 확인 후 허용
3. Retry 가 *3번 시도* — 각 시도가 실패하면 *Circuit Breaker 에 1번씩 실패 카운트*
4. 3번 모두 실패하면 마지막 예외 던짐

핵심은 *각 재시도가 Circuit Breaker 에 카운트* 된다는 점입니다. 3번 재시도가 모두 실패하면 *서킷 입장에서는 3번의 실패* 입니다. `failureRateThreshold: 50` 에 빠르게 도달합니다.

반대로 *Retry 가 바깥쪽* 이면:

```java
.withRetry(retry)
.withCircuitBreaker(circuitBreaker)
.withBulkhead(bulkhead)
.decorate();
```

이 경우 *Retry 가 3번 시도하는데, 각 시도가 Circuit Breaker 안에서 일어남*. 하지만 *Retry 가 바깥에서 3번을 묶어 1번의 시도로 본다면* Circuit Breaker 가 *한 번의 실패만 카운트* 합니다. 서킷이 *느리게 반응* 합니다.

운영에서 *Retry 가 안쪽* 이 디폴트인 이유입니다. Spring Boot 의 어노테이션 기반에서도 같은 순서가 자동 적용됩니다.



## 비동기 Retry

> CompletableFuture·WebClient 같은 비동기 호출도 동일하게 Retry 가 적용됩니다.

```java
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(3);

Supplier<CompletionStage<String>> asyncSupplier =
        () -> CompletableFuture.supplyAsync(backendService::doSomething);

Supplier<CompletionStage<String>> decoratedAsync = Retry
        .decorateCompletionStage(retry, scheduler, asyncSupplier);
```

`ScheduledExecutorService` 가 *대기 시간 후 다음 시도를 트리거* 합니다. *호출 스레드를 점유하지 않고* 대기하므로 자원 효율적.

Reactive (Reactor) 통합:

```java
Mono<String> result = Mono.fromCallable(backendService::doSomething)
        .transformDeferred(RetryOperator.of(retry));
```

`RetryOperator` 가 Reactor 연산자 체인에 *재시도 로직* 을 끼워 넣습니다. WebClient 와의 결합이 자연스럽습니다.



## retryOnResult — 결과 기반 재시도

> *예외가 아닌* *응답 자체* 가 재시도 신호일 수 있습니다. 예: *외부 API 가 5xx 대신 200 + body 에 "TRY_LATER"* 를 리턴하는 경우.

```java
RetryConfig config = RetryConfig.custom()
        .maxAttempts(3)
        .retryOnResult(response -> {
            return response instanceof ApiResponse
                    && ((ApiResponse) response).getStatus().equals("TRY_LATER");
        })
        .build();
```

`retryOnResult` 가 *true 를 리턴하면 재시도* 합니다. 비표준 API 의 응답 패턴을 *재시도 로직에 흡수* 할 수 있습니다.

이 자리에서 자주 막히는 함정은 *false 가 디폴트* 라는 점입니다. 명시 안 하면 *결과 기반 재시도는 일어나지 않습니다*. 예외만으로 부족할 때 명시적으로 켭니다.



## 이벤트 리스너 — 운영 관측

```java
retry.getEventPublisher()
        .onRetry(event -> logger.warn("Retry attempt {} after {}ms",
                event.getNumberOfRetryAttempts(),
                event.getWaitInterval()))
        .onSuccess(event -> logger.info("Success after {} attempts",
                event.getNumberOfRetryAttempts()))
        .onError(event -> logger.error("All retries exhausted after {} attempts",
                event.getNumberOfRetryAttempts()));
```

세 이벤트가 운영 신호입니다.

- `onRetry` — *매 재시도 시점* 호출. 빈도가 높아지면 *다운스트림 불안정* 신호.
- `onSuccess` — *최종 성공* 시 호출. `numberOfRetryAttempts > 0` 이면 *재시도로 회복* 된 케이스.
- `onError` — *모든 재시도 실패* 시 호출. 알람 대상.

Micrometer 메트릭은 `resilience4j_retry_calls` 의 `kind` 라벨로 노출됩니다 (`successful_without_retry`, `successful_with_retry`, `failed_with_retry`, `failed_without_retry`).



## 함정 — 자주 막히는 자리들

**1. Idempotent 하지 않은 호출을 Retry 함** — POST 가 가장 위험. 첫 호출이 *실제로는 성공* 했지만 *응답이 안 돌아와* 재시도하면 *두 번 처리* 됩니다. 결제·주문 같은 도메인에서 *Idempotency-Key 헤더* 가 함께 가야 안전. [`../../05_data/batch/01-04.재시작과 Checkpoint — ExecutionContext·Restartable·idempotent Step.md`](../../05_data/batch/01-04.재시작과%20Checkpoint%20—%20ExecutionContext·Restartable·idempotent%20Step.md) 에서 본 *멱등 설계 원칙* 과 동일.

**2. Retry 안에 *서킷 OPEN 예외도 재시도* 하려고 시도** — `CallNotPermittedException` (서킷 OPEN 신호) 까지 재시도 대상으로 잡으면 *fail-fast 의도 자체가 무너집니다*. `retryExceptions` 에서 명시적으로 제외하거나 `ignoreExceptions` 에 포함.

**3. maxAttempts 가 너무 큼** — 5 이상은 보통 *재시도 폭주 가능성* 증가. 일반적으로 3 이 안전한 디폴트. *느린 백오프 + 적은 횟수* 가 *빠른 백오프 + 많은 횟수* 보다 다운스트림에 자비롭습니다.

**4. jitter 없이 운영 도입** — 위에서 다룬 thundering herd. 운영에서 거의 항상 jitter 가 답.

**5. Retry 와 Spring Retry 를 동시 적용** — `@Retryable` (Spring Retry) 과 `@Retry` (Resilience4j) 를 같은 메서드에 같이 붙이면 *재시도가 곱해집니다*. 둘 중 하나만.
