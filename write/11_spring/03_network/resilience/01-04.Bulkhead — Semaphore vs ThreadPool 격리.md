# Bulkhead — Semaphore vs ThreadPool 격리

---

> Bulkhead 는 *선체의 격벽* 에서 유래합니다. 배가 한 구획 침수돼도 *격벽으로 다른 구획을 차단* 해 *전체 침몰* 을 막는 발상을 *호출 격리* 에 옮겼습니다. 한 다운스트림 호출이 *내 시스템의 모든 스레드를 점유* 하지 못하도록 *동시 호출 수에 한도* 를 둡니다. 본 편은 *Semaphore Bulkhead 와 ThreadPool Bulkhead 두 구현체* 의 자원 모델과 *각각이 답이 되는 자리* 를 정리합니다.



## Bulkhead 가 답이 되는 자리

> 한 다운스트림이 *느려졌을 때* *내 시스템의 모든 호출 자원을 점유* 하지 못하게 합니다.

상황 예시입니다. 서비스 A 가 *3 개 외부 API* (Payment, Inventory, Notification) 를 호출합니다. Tomcat 의 스레드 풀이 200 입니다. *Payment API 가 갑자기 30초로 느려지면* — 단순 호출 모델에서는 *Payment 호출이 차츰 200 스레드를 다 점유* 해 *Inventory·Notification 호출도 같이 멈춥니다*. 한 다운스트림의 장애가 *전체 서비스로 전파* 됩니다.

Bulkhead 가 *각 다운스트림에 동시 호출 한도* 를 설정합니다.

| 다운스트림 | Bulkhead 한도 |
|----------|--------------|
| Payment | 50 |
| Inventory | 30 |
| Notification | 20 |
| 여유 | 100 |

Payment 가 느려져도 *최대 50 스레드만 점유*. 나머지 150 은 다른 호출이 쓸 수 있습니다. *장애 전파 격리* 입니다.



## 두 구현체 — Semaphore vs ThreadPool

> Resilience4j 는 *Semaphore Bulkhead* (디폴트) 와 *ThreadPool Bulkhead* 두 구현체를 제공합니다. 자원 모델이 다릅니다.

| 측면 | Semaphore Bulkhead | ThreadPool Bulkhead |
|------|-------------------|-------------------|
| 호출 실행 스레드 | *호출한 스레드* (caller thread) | *별도 ThreadPool 의 스레드* |
| 동기·비동기 | 둘 다 | *비동기만* (`CompletionStage` 리턴) |
| 자원 한정 | *동시 점유 permit 수* | *Pool 크기 + Queue 크기* |
| 호출 지연 | 짧음 (permit 획득만) | 길음 (스레드 컨텍스트 스위치) |
| 사용처 | WebClient·동기 API 호출 | *느린 + 외부 시스템 호출 + Bulkhead 격리가 필요* |



## Semaphore Bulkhead — 가벼운 카운터

> 내부적으로 *Semaphore* (java.util.concurrent.Semaphore) 를 씁니다. *호출 시 permit 획득, 종료 시 반환*. 호출 자체는 *원래 스레드에서* 실행됩니다.

```java
BulkheadConfig config = BulkheadConfig.custom()
        .maxConcurrentCalls(150)
        .maxWaitDuration(Duration.ofMillis(500))
        .build();

Bulkhead bulkhead = Bulkhead.of("backendName", config);

Supplier<String> decoratedSupplier = Bulkhead
        .decorateSupplier(bulkhead, backendService::doSomething);
```

`maxConcurrentCalls: 150` 이 *동시 점유 가능한 permit 수* 입니다. 150 개 호출이 진행 중이면 *151번째 호출은 기다리거나 거부* 됩니다.

`maxWaitDuration: 500ms` 가 *대기 시간* 입니다. 500ms 안에 permit 이 안 나오면 `BulkheadFullException` 을 던집니다.

이 모델의 장점은 *극도로 가볍다* 는 점입니다. Semaphore 의 `tryAcquire`/`release` 가 *수십 나노초* 수준 오버헤드. 동기 호출에 적합.

YAML:

```yaml
resilience4j:
  bulkhead:
    instances:
      backendA:
        maxConcurrentCalls: 150
        maxWaitDuration: 500ms
```



## ThreadPool Bulkhead — 별도 스레드 풀

> *별도 ThreadPoolExecutor* 를 만들어 호출을 그 풀에서 실행합니다. *호출 스레드는 즉시 반환* 되어 다른 일을 할 수 있습니다.

```java
ThreadPoolBulkheadConfig config = ThreadPoolBulkheadConfig.custom()
        .maxThreadPoolSize(10)
        .coreThreadPoolSize(2)
        .queueCapacity(20)
        .keepAliveDuration(Duration.ofMillis(1000))
        .build();

ThreadPoolBulkhead threadPoolBulkhead =
        ThreadPoolBulkhead.of("backendName", config);

Supplier<CompletionStage<String>> asyncSupplier = ThreadPoolBulkhead
        .decorateSupplier(threadPoolBulkhead, backendService::doSomething);
```

설정 4가지의 의미입니다.

| 설정 | 의미 |
|------|------|
| `coreThreadPoolSize` | 항상 살아있는 스레드 수 |
| `maxThreadPoolSize` | 최대 스레드 수 (queue 가 차면 여기까지 증가) |
| `queueCapacity` | 대기 큐 크기 |
| `keepAliveDuration` | core 이상의 스레드가 idle 시 유지 시간 |

호출 흐름은 *core 스레드 사용 → 가득 차면 queue 에 적재 → queue 가득 차면 max 까지 스레드 증가 → 그것도 가득 차면 `BulkheadFullException`*.

이 모델은 *오로지 비동기 호출* 만 가능합니다. `decorateSupplier` 가 `CompletionStage` 를 리턴합니다.

YAML:

```yaml
resilience4j:
  thread-pool-bulkhead:
    instances:
      backendA:
        maxThreadPoolSize: 10
        coreThreadPoolSize: 2
        queueCapacity: 20
        keepAliveDuration: 1000ms
```



## 어느 쪽을 선택하는가

### Semaphore 가 답인 경우

- *WebClient 같은 비동기 클라이언트* 가 *이미 자체 풀* 을 가지고 있음. Bulkhead 는 *호출 수 제한만* 책임지면 충분.
- *Spring MVC + RestClient + 동기 호출*. 호출 자체가 이미 동기 스레드에서 일어남.
- *오버헤드 최소화* 가 중요.

대부분의 운영 환경에서 *디폴트가 Semaphore* 입니다.

### ThreadPool 이 답인 경우

- *호출이 blocking* 인데 *호출 스레드를 빠르게 반환* 하고 싶음 (예: 비동기 API 의 응답).
- *각 다운스트림에 명시적 스레드 격리* 가 필요 (장애 격리가 *스레드 풀 수준* 까지).
- *외부 호출이 느림* (수 초 단위) 이고 *호출 스레드가 점유되면 안 됨*.

비용: 컨텍스트 스위치·메모리·디버깅 어려움.

> *디폴트는 Semaphore.* ThreadPool 은 *명확한 이유가 있을 때만*.



## 비동기 + TimeLimiter 조합

> ThreadPool Bulkhead 와 TimeLimiter 의 조합이 *느린 외부 호출* 에 흔한 패턴입니다.

```java
ThreadPoolBulkhead threadPoolBulkhead = ThreadPoolBulkhead.ofDefaults("backendService");
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(3);
TimeLimiter timeLimiter = TimeLimiter.of(Duration.ofSeconds(1));

CompletableFuture<String> future = Decorators.ofSupplier(supplier)
        .withThreadPoolBulkhead(threadPoolBulkhead)
        .withTimeLimiter(timeLimiter, scheduler)
        .withCircuitBreaker(circuitBreaker)
        .withFallback(asList(
                TimeoutException.class,
                CallNotPermittedException.class,
                BulkheadFullException.class),
                throwable -> "Hello from Recovery")
        .get().toCompletableFuture();
```

이 조합의 의미는 다음과 같습니다.

1. *ThreadPool Bulkhead* 가 *별도 스레드에서 실행*
2. *TimeLimiter* 가 *1초 안에 끝나지 않으면 강제 종료*
3. *Circuit Breaker* 가 *실패 메트릭 누적*
4. *Fallback* 이 *3가지 예외* (Timeout·서킷 OPEN·Bulkhead 가득) 를 받아 *기본 응답*

이 4단 조합이 *느린 외부 호출의 표준 회복탄력성 패턴* 입니다.



## Spring Boot Annotation 통합

```java
@Service
public class BackendService {

    @Bulkhead(name = "backendA")          // Semaphore 디폴트
    public String process(String data) {
        return externalService.call(data);
    }

    @Bulkhead(name = "backendA", type = Bulkhead.Type.THREADPOOL)
    public CompletableFuture<String> processAsync(String data) {
        return CompletableFuture.supplyAsync(() ->
                externalService.call(data));
    }
}
```

`type = Bulkhead.Type.THREADPOOL` 명시가 ThreadPool 모드. 디폴트는 Semaphore.

ThreadPool 모드는 *리턴 타입이 `CompletionStage` 또는 `CompletableFuture`* 여야 합니다. 동기 메서드에 ThreadPool 모드를 붙이면 *컴파일은 되지만 런타임 오류* 입니다.



## 메트릭과 이벤트

```java
Bulkhead.Metrics metrics = bulkhead.getMetrics();
int availableConcurrentCalls = metrics.getAvailableConcurrentCalls();
int maxAllowedConcurrentCalls = metrics.getMaxAllowedConcurrentCalls();

bulkhead.getEventPublisher()
        .onCallPermitted(event -> logger.debug("Call permitted"))
        .onCallRejected(event -> logger.warn("Call rejected - bulkhead full"))
        .onCallFinished(event -> logger.debug("Call finished"));
```

`getAvailableConcurrentCalls()` 가 *현재 남은 permit 수*. 0 이 되면 *다음 호출은 대기 또는 거부* 입니다.

Micrometer 메트릭:

| 메트릭 | 의미 |
|--------|------|
| `resilience4j_bulkhead_available_concurrent_calls` | 현재 남은 permit |
| `resilience4j_bulkhead_max_allowed_concurrent_calls` | 최대 permit |

운영 알람은 *available == 0 이 5분 이상 지속* 같은 룰입니다. *Bulkhead 가 자주 가득 참* = *해당 다운스트림 호출이 한도 근처* = *한도 재조정* 또는 *다운스트림 성능 점검* 신호.



## 함정 — 자주 막히는 자리들

**1. *모든 호출에 같은 Bulkhead*** — *공통 Bulkhead 한 개* 를 모든 다운스트림에 적용하면 *격리 의미가 사라집니다*. 각 다운스트림마다 *독립된 instance* 가 답.

**2. *Bulkhead 한도가 너무 작음*** — Tomcat 스레드 200 인데 Bulkhead 가 10 이면 *대부분 호출이 거부* 됩니다. 한도는 *해당 다운스트림의 P99 응답 시간 × 초당 호출 수* 로 추정.

**3. ThreadPool 모드에 *동기 리턴 타입*** — `String` 리턴 메서드에 `@Bulkhead(type = THREADPOOL)` 을 붙이면 런타임 오류.

**4. Virtual Thread 와 결합 시 *Semaphore 의 가치 변함*** — Java 21 의 Virtual Thread 환경에서는 *Bulkhead 의 스레드 격리 의미가 약해집니다*. Virtual Thread 가 *수십만 개* 가능하므로 *스레드 점유* 자체가 부족 자원이 아닐 수 있음. 다만 *호출 자체의 속도 제한* 이라는 의미는 유지.

**5. *Bulkhead 와 동기 Circuit Breaker 만 사용 → 호출 지연이 길 때 큰 효과 없음*** — 호출이 *오래 걸리는데* Bulkhead 한도가 차면 *대기 + BulkheadFullException* 만 늘어남. *TimeLimiter 와 결합* 이 답.
