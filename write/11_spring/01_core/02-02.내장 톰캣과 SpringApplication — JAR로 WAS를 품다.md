# 내장 톰캣과 SpringApplication — JAR로 WAS를 품다
---

> [02-01](02-01.WAS와%20서블릿%20—%20HTTP%20처리의%20토대.md)에서 WAS(톰캣)가 서블릿을 실행하는 환경이라고 했습니다. 그런데 스프링 부트 프로젝트는 톰캣을 따로 설치한 적이 없는데도 `main()` 만 실행하면 서버가 뜹니다. 이 문서는 그 비결인 *내장 톰캣*을 다룹니다. 톰캣을 외부에 설치하는 대신 라이브러리로 품고, `SpringApplication.run()` 이 자바 코드로 직접 띄우며, 그 모든 것을 실행 가능 JAR 하나로 배포하는 흐름입니다. 끝까지 읽으면 "WAR에서 JAR로 바뀐 것이 단순 확장자 변경이 아니라 배포 모델의 전환"임을 알게 됩니다.

## 1. 과거의 WAR 배포와 그 단점

전통적인 자바 웹 배포는 이랬습니다. 서버에 톰캣을 *미리 설치*해 두고, 애플리케이션은 WAR(Web Application Archive) 파일로 빌드해 그 톰캣에 올렸습니다. 톰캣(실행 환경)과 WAR(애플리케이션)가 분리된 모델입니다.

이 방식에는 단점이 있습니다. 서버마다 톰캣을 설치하고 버전을 맞춰야 하고, 톰캣 설정과 애플리케이션이 따로 관리되어 "이 코드는 어떤 톰캣 버전에서 검증됐는가"가 흐려집니다. 개발자 PC의 톰캣과 운영 서버의 톰캣이 미묘하게 달라 "내 PC에선 됐는데" 문제가 생기기도 합니다. WAS 설치라는 *외부 의존*이 배포를 무겁게 만듭니다.

## 2. 내장 톰캣 — 톰캣을 라이브러리로 품기

스프링 부트는 발상을 뒤집습니다. 톰캣을 *외부에 설치*하는 대신 *애플리케이션 안에 라이브러리로 포함*합니다. 내장 톰캣은 쉽게 말해 톰캣을 라이브러리로 포함하고 자바 코드로 직접 띄우는 것입니다.

```java
// 내장 톰캣을 자바 코드로 직접 띄우는 개념 (스프링 부트가 내부에서 하는 일)
public static void main(String[] args) throws Exception {
    Tomcat tomcat = new Tomcat();
    Connector connector = new Connector();
    connector.setPort(8080);
    tomcat.setConnector(connector);

    // 서블릿 등록
    Context context = tomcat.addContext("", "/");
    tomcat.addServlet("", "helloServlet", new HelloServlet());
    context.addServletMappingDecoded("/hello-servlet", "helloServlet");

    tomcat.start();  // 톰캣 시작
}
```

핵심은 톰캣이 더 이상 "내가 설치한 외부 프로그램"이 아니라 "내 코드가 `new` 해서 띄우는 객체"가 됐다는 점입니다. 톰캣 버전이 빌드 의존성(`spring-boot-starter-web`)에 묶이므로, 애플리케이션과 톰캣이 *한 묶음으로 버전이 관리*됩니다. [01-01 스타터](../07_autoconfig/01-01.스타터와%20라이브러리%20버전%20관리.md)가 내장 톰캣을 딸려 오는 것이 이 토대입니다.

내장 톰캣 덕분에 IDE에 별도의 복잡한 톰캣 설정 없이 `main()` 메서드만 실행하면 톰캣까지 함께 뜹니다. 외부 WAS 설치라는 단계가 통째로 사라집니다.

## 3. SpringApplication.run() — 스프링과 내장 톰캣의 결합

2절은 순수 내장 톰캣에 서블릿을 직접 등록하는 코드였습니다. 스프링 부트는 여기에 스프링 컨테이너를 얹습니다. `SpringApplication.run()` 한 줄이 내장 톰캣을 띄우고, 스프링 컨테이너를 만들고, [03-01](03-01.Spring%20MVC%20—%20FrontController에서%20DispatcherServlet까지.md)의 `DispatcherServlet` 을 그 톰캣에 등록하는 일을 한꺼번에 합니다.

```java
@SpringBootApplication
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}
```

이 한 줄이 내부에서 하는 일을 풀면 이렇습니다.

```text
SpringApplication.run()
  ├─ 스프링 컨테이너(ApplicationContext) 생성
  ├─ 자동 구성으로 ServletWebServerFactory 빈 등록 (내장 톰캣 팩토리)
  ├─ 내장 톰캣 생성 + 시작
  └─ DispatcherServlet 을 내장 톰캣에 등록
        → 이제 모든 HTTP 요청이 DispatcherServlet 으로 들어온다
```

`spring-boot-starter-web` 을 쓰면 내부에서 내장 톰캣을 사용하고, 자동 구성이 `ServletWebServerFactory`(내장 톰캣을 만드는 팩토리 빈)를 등록합니다. [02-01 §1](02-01.WAS와%20서블릿%20—%20HTTP%20처리의%20토대.md)에서 "DispatcherServlet 한 개를 톰캣에 등록해 모든 요청을 받는다"고 한 그 등록이, 외부 톰캣이 아니라 내장 톰캣 위에서 자동으로 일어납니다. 02-01의 서블릿 표준 위에 스프링 부트의 자동화가 얹힌 모습입니다.

## 4. 실행 가능 JAR — 배포 단위의 전환

내장 톰캣으로 톰캣이 애플리케이션 안에 들어왔으니, 배포 산출물도 바뀝니다. 더 이상 WAR를 외부 톰캣에 올리지 않고, *톰캣까지 포함한 JAR 하나*를 그냥 실행합니다.

문제는 일반 JAR가 다른 JAR(라이브러리)를 내부에 포함해 실행하는 것을 표준으로 지원하지 않는다는 점입니다. 그래서 등장한 것이 fat jar(또는 uber jar)와 스프링 부트의 *실행 가능 JAR(Executable Jar)* 입니다. 스프링 부트는 내부 JAR들을 포함한 채로 실행할 수 있는 구조로 JAR를 만듭니다.

```text
my-app.jar (실행 가능 JAR)
├── META-INF/MANIFEST.MF        ← Main-Class: ...JarLauncher
├── org/springframework/boot/loader/   ← JarLauncher 등 부트 로더
├── BOOT-INF/classes/           ← 내 애플리케이션 클래스
└── BOOT-INF/lib/               ← 내장 톰캣 포함 모든 의존 라이브러리(.jar)
```

`MANIFEST.MF` 의 `Main-Class` 가 스프링 부트의 `JarLauncher` 를 가리킵니다. `java -jar my-app.jar` 를 실행하면 `JarLauncher` 가 `BOOT-INF/lib` 안의 라이브러리들을 읽어 클래스패스를 구성한 뒤 우리 `main()` 을 호출합니다. 이 JAR 하나에 애플리케이션·라이브러리·내장 톰캣이 모두 들어 있으므로, 서버에는 자바(JVM)만 있으면 됩니다.

```bash
# 서버에 톰캣 설치 불필요 — JVM만 있으면 실행
java -jar my-app.jar
```

WAR 시절 "톰캣 설치 → WAR 배포"의 두 단계가 "JAR 하나 실행"으로 줄었습니다. 이것이 컨테이너(Docker) 시대에 스프링 부트가 잘 맞는 이유이기도 합니다. 베이스 이미지에 JVM만 두고 JAR 하나를 얹으면 끝이기 때문입니다.

## 5. WAR vs JAR 배포 모델

| 기준 | WAR (전통) | 실행 가능 JAR (스프링 부트) |
|------|-----------|---------------------------|
| 톰캣 | 서버에 외부 설치 | 애플리케이션에 내장 |
| 배포 단위 | WAR + 외부 톰캣 | JAR 하나 |
| 톰캣 버전 관리 | 서버마다 별도 | 빌드 의존성에 묶임 |
| 실행 | 톰캣에 WAR 올림 | `java -jar` |
| 서버 요구사항 | 톰캣 설치 | JVM만 |
| 개발 편의 | 톰캣 설정 필요 | `main()` 실행이면 끝 |

## 6. 정리

| 주제 | 핵심 |
|------|------|
| WAR 배포 단점 | 외부 톰캣 설치·버전 관리·환경 차이 |
| 내장 톰캣 | 톰캣을 라이브러리로 품어 자바 코드로 직접 띄움 |
| SpringApplication.run() | 컨테이너 생성 + 내장 톰캣 시작 + DispatcherServlet 등록 |
| ServletWebServerFactory | 자동 구성이 등록하는 내장 톰캣 팩토리 빈 |
| 실행 가능 JAR | 내장 톰캣 포함, JarLauncher로 `java -jar` 실행 |
| 배포 전환 | "톰캣 설치 + WAR" → "JAR 하나", JVM만 필요 |
