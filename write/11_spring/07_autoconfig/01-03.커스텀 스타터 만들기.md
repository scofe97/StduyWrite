# 커스텀 스타터 만들기
---

> 여러 프로젝트에서 같은 빈 설정을 복사·붙여넣기 하고 있다면, 그것을 스타터로 묶을 때가 된 것입니다. 이 문서는 [앞 문서](01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md)에서 본 자동 구성 원리를 *내가 만드는 라이브러리*에 적용해, 의존성 한 줄만 추가하면 빈이 자동 등록되는 스타터를 직접 만드는 과정을 따라갑니다. 끝까지 만들어 보면 스프링 부트가 제공하는 스타터가 특별한 마법이 아니라 우리도 똑같이 쓸 수 있는 규약임을 알게 됩니다.

## 1. 왜 직접 만드는가

라이브러리를 만드는 개발자 입장에서 생각해 봅니다. 내 라이브러리 안에 어떤 빈을 등록해야 동작하는데, 그 등록 설정을 *사용하는 쪽*에 매번 떠넘기면 사용자는 라이브러리 내부를 알아야 합니다. 설정이 복잡할수록 사용자에게는 귀찮은 작업이 됩니다.

스프링 부트의 자동 구성은 이 문제를 라이브러리 제공자 쪽으로 옮깁니다. 라이브러리 안에 "이 라이브러리가 클래스패스에 있으면 이런 빈을 등록하라"는 자동 구성을 넣어 두면, 사용자는 의존성만 추가하고 빈 설정은 신경 쓰지 않아도 됩니다. 직접 만드는 스타터의 목적이 바로 이 *설정 위임의 역전*입니다.

## 2. 자동 구성 클래스 작성

먼저 라이브러리가 제공할 빈을 자동 구성 클래스로 정의합니다. [01-02](01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md)에서 본 `@AutoConfiguration` 과 `@Conditional` 을 그대로 씁니다.

```java
@AutoConfiguration
@ConditionalOnClass(MyService.class)  // 라이브러리 클래스가 있을 때만
public class MyServiceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean  // 사용자가 직접 등록 안 했을 때만
    public MyService myService() {
        return new MyService();
    }
}
```

두 조건의 의미는 자동 구성과 동일합니다. `@ConditionalOnClass` 는 내 라이브러리가 클래스패스에 있을 때만 작동하게 하고, `@ConditionalOnMissingBean` 은 사용자가 직접 `MyService` 를 등록하면 물러나 사용자 설정을 존중합니다. 라이브러리 제공자가 *기본값만 제공하고 최종 결정권은 사용자에게 남기는* 예의입니다.

## 3. AutoConfiguration.imports에 등록

자동 구성 클래스를 만들었다고 스프링 부트가 알아서 찾아 주지는 않습니다. [01-02 §3](01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md)에서 본 것처럼 `ImportSelector` 가 읽는 목록 파일에 등록해야 합니다.

```text
# 위치: src/main/resources/META-INF/spring/
#       org.springframework.boot.autoconfigure.AutoConfiguration.imports

com.example.mystarter.MyServiceAutoConfiguration
```

이 파일에 자동 구성 클래스의 전체 이름(FQCN)을 한 줄 적습니다. 사용자 프로젝트가 시작될 때 `@EnableAutoConfiguration` 이 트리거하는 `ImportSelector` 가 이 파일을 읽어 `MyServiceAutoConfiguration` 을 후보로 올리고, 2절의 조건이 통과하면 `MyService` 빈이 등록됩니다.

> 참고: 스프링 부트 2.7 이전에는 `spring.factories` 파일에 `EnableAutoConfiguration` 키로 등록했습니다. 2.7부터 `AutoConfiguration.imports` 방식이 도입됐고 3.x에서는 이것이 표준입니다. 옛 라이브러리를 읽을 때 `spring.factories` 가 보이면 같은 역할의 구버전 방식이라고 이해하면 됩니다.

## 4. 설정값을 받게 하려면 — @ConfigurationProperties 연동

빈만 등록하는 데서 한 걸음 더 나아가, 사용자가 `application.yml` 로 라이브러리 동작을 조정하게 하려면 `@ConfigurationProperties` 를 함께 제공합니다.

```java
@ConfigurationProperties(prefix = "my.service")
public class MyServiceProperties {
    private String mode = "default";
    private int timeout = 3000;
    // getter / setter
}
```

```java
@AutoConfiguration
@ConditionalOnClass(MyService.class)
@EnableConfigurationProperties(MyServiceProperties.class)
public class MyServiceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public MyService myService(MyServiceProperties props) {
        return new MyService(props.getMode(), props.getTimeout());
    }
}
```

이제 사용자는 의존성만 추가하면 빈을 얻고, 동작을 바꾸고 싶을 때는 설정 파일만 건드립니다.

```yaml
my:
  service:
    mode: production
    timeout: 5000
```

라이브러리 내부 코드를 전혀 모르고도 `application.yml` 몇 줄로 동작을 조정하는 경험이 완성됩니다. `@ConfigurationProperties` 의 타입 안전한 설정 바인딩은 [02-02 문서](02-02.@ConfigurationProperties와%20타입%20안전%20설정.md)에서 따로 다룹니다.

## 5. 사용하는 쪽 — 의존성 한 줄

스타터가 완성되면 사용자 프로젝트는 의존성 한 줄만 추가하면 됩니다.

```groovy
dependencies {
    implementation 'com.example:my-spring-boot-starter'
}
```

설정 클래스를 import하거나 `@Bean` 을 적을 필요가 없습니다. 의존성이 클래스패스를 채우면 `@ConditionalOnClass` 가 참이 되고, 자동 구성이 `MyService` 를 등록합니다. 스프링 부트 공식 스타터를 쓸 때와 똑같은 경험이며, 그 경험을 우리가 만든 라이브러리에서 재현한 것입니다.

## 6. 정리 — 커스텀 스타터 체크리스트

| 단계 | 할 일 |
|------|------|
| 1 | 라이브러리가 제공할 빈을 결정 |
| 2 | `@AutoConfiguration` + `@ConditionalOnClass` + `@ConditionalOnMissingBean` 으로 자동 구성 클래스 작성 |
| 3 | `META-INF/spring/...AutoConfiguration.imports` 에 FQCN 등록 |
| 4 | (선택) `@ConfigurationProperties` 로 설정값 바인딩 추가 |
| 5 | 사용자는 의존성 한 줄만 추가 |

핵심은 [01-02](01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md)에서 본 자동 구성 원리를 *소비자가 아니라 제공자 입장에서* 적용한다는 점입니다. 스프링 부트 스타터를 받아 쓰던 경험을, 이제 내가 만들어 남에게 제공할 수 있습니다.
