# 외부 설정 — 커맨드라인부터 application.yml까지
---

> 같은 애플리케이션을 개발 서버에서는 개발 DB에, 운영 서버에서는 운영 DB에 연결해야 합니다. 코드를 환경마다 다시 빌드하지 않고 이 차이를 흡수하는 것이 외부 설정입니다. 이 문서는 외부 설정을 주입하는 여러 통로(OS 환경변수·자바 시스템 속성·커맨드 라인·설정 파일)를 훑고, 스프링이 그것들을 `Environment` 라는 하나의 창구로 통일하는 방식을 정리합니다. 끝까지 읽으면 "설정값이 여러 곳에 있을 때 무엇이 이기는가"를 우선순위로 답할 수 있게 됩니다.

## 1. 왜 설정을 밖으로 빼는가

유지보수하기 좋은 애플리케이션의 기본 원칙은 *변하는 것과 변하지 않는 것을 분리*하는 것입니다. 비즈니스 로직(변하지 않는 것)과 환경별 설정(변하는 것)을 한 덩어리로 빌드하면, DB 주소 하나 바꾸려고 전체를 다시 빌드해야 합니다.

이 문제를 푸는 가장 단순한 방법은 환경마다 따로 빌드하는 것입니다. 그러나 같은 코드를 개발용·운영용으로 두 번 빌드하면 "운영에 올라간 것이 정말 테스트한 그 코드인가"를 보장하기 어려워집니다. 더 나은 방법은 *한 번 빌드한 산출물*을 두고, 환경 차이는 실행 시점에 외부에서 주입하는 것입니다. 이것이 외부 설정의 동기입니다.

## 2. 설정을 주입하는 네 가지 통로

스프링 부트는 외부 설정을 여러 통로로 받습니다. 통로마다 사용 범위와 적용 방법이 다릅니다.

| 통로 | 지정 방법 | 특징 |
|------|----------|------|
| OS 환경 변수 | `export DB_URL=...` | 해당 OS의 모든 프로세스가 공유, 사용 범위 가장 넓음 |
| 자바 시스템 속성 | `java -Durl=... -jar app.jar` | 해당 JVM 안에서 유효 |
| 커맨드 라인 인수 | `java -jar app.jar url=...` | `main(String[] args)` 로 전달 |
| 설정 파일 | `application.properties` / `application.yml` | jar 내부에 내장하거나 외부 파일로 분리 |

각 통로는 사용 범위가 다릅니다. OS 환경 변수는 그 OS의 모든 프로세스에 보이므로 범위가 가장 넓고, 자바 시스템 속성은 특정 JVM 안에서만, 커맨드 라인 인수는 그 실행 한 번에만 유효합니다. 범위가 넓을수록 영향이 크다는 점을 기억하면 어디에 무엇을 둘지 판단이 섭니다.

### 커맨드 라인 옵션 인수

커맨드 라인 인수 중에서도 `--key=value` 형태를 *옵션 인수*라 부릅니다. 스프링 부트는 이 형식을 특별히 인식해 키-값으로 파싱합니다.

```bash
# 일반 인수 — 단순 문자열 배열로 전달
java -jar app.jar url=dev secret=123

# 옵션 인수 — -- 접두사, 스프링 부트가 키-값으로 파싱
java -jar app.jar --url=dev --secret=123
```

스프링 부트는 커맨드 라인 옵션 인수를 받아 스프링 빈으로 등록해 둡니다. 그 빈을 주입받으면 입력된 옵션 인수에 키로 접근할 수 있어, `main` 의 `String[] args` 를 직접 파싱하는 수고가 사라집니다.

## 3. 스프링 통합 — Environment라는 단일 창구

통로가 네 가지나 되면 코드에서 일일이 분기해 읽어야 할까요? 그렇지 않습니다. 스프링은 이 모든 통로를 `Environment` 라는 하나의 추상화로 통일합니다. 각 통로는 내부적으로 `PropertySource` 가 되고, `Environment` 가 그 `PropertySource` 들을 모아 단일 조회 창구를 제공합니다.

```java
@Component
@RequiredArgsConstructor
public class MyBean {
    private final Environment env;

    public void logic() {
        // OS 환경변수든 커맨드라인이든 설정파일이든, 같은 방식으로 조회
        String url = env.getProperty("url");
        int timeout = env.getProperty("timeout", Integer.class);
    }
}
```

`env.getProperty("url")` 한 줄이면 그 값이 OS 환경 변수에서 왔는지 설정 파일에서 왔는지 신경 쓸 필요가 없습니다. 코드는 "url이라는 설정이 필요하다"만 표현하고, *어디서 왔는가*는 `Environment` 가 흡수합니다. [01-02 자동 구성](01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md)에서 `@ConditionalOnProperty` 가 설정값을 조건으로 쓸 수 있던 것도 이 `Environment` 통합 덕분입니다.

### 우선순위 — 같은 키가 여러 곳에 있을 때

여러 `PropertySource` 에 같은 키가 있으면 *우선순위가 높은 통로*가 이깁니다. 대략의 순서는 다음과 같습니다(높을수록 우선).

```text
커맨드 라인 옵션 인수
  > 자바 시스템 속성 (-D)
  > OS 환경 변수
  > application.properties / application.yml (jar 내부)
```

이 우선순위가 외부 설정의 실용성을 만듭니다. 설정 파일에 기본값을 박아 두고(낮은 우선순위), 특정 실행에서만 커맨드 라인으로 덮어쓰는(높은 우선순위) 패턴이 가능해집니다. 기본은 파일, 예외는 실행 인수라는 구도입니다.

라이브러리가 이 합성에 *기본값* 으로 끼어들고 싶다면, `EnvironmentPostProcessor` 로 `PropertySource` 를 가장 낮은 우선순위로 추가합니다. ApplicationContext 가 만들어지기 *전* 단계에 끼어들어야 하므로 `@Configuration` 빈으로는 늦어 못 풉니다. `addLast` 로 등록하면 사용자 `application.yml` 이 항상 이기므로 라이브러리 안전 기본값과 사용자 오버라이드가 자연스럽게 공존합니다. 상세는 [01-04 §4](01-04.자동%20구성%20심화%20—%20순서·게이트·기본값%20주입.md) 에서 다룹니다.

## 4. 설정 파일 — application.properties와 application.yml

실무에서 가장 많이 쓰는 통로는 설정 파일입니다. 커맨드 라인이나 환경 변수는 값이 많아지면 관리가 어렵지만, 파일은 구조적으로 묶어 둘 수 있습니다. 스프링 부트는 `application.properties` 를 jar 내부에 기본 내장하고, 특별한 환경에서는 외부 파일로 덮어쓰게 합니다.

`application.properties` 는 평면적인 키-값입니다.

```properties
my.datasource.url=jdbc:h2:mem:test
my.datasource.username=sa
my.datasource.max-connection=10
```

`application.yml` 은 같은 내용을 계층 구조로 표현합니다. 키의 공통 접두사가 반복되지 않아 가독성이 좋습니다.

```yaml
my:
  datasource:
    url: jdbc:h2:mem:test
    username: sa
    max-connection: 10
```

둘은 표현 방식만 다를 뿐 같은 설정입니다. 계층이 깊어질수록 YAML이 읽기 편해, 최근 프로젝트는 대부분 YAML을 씁니다. 다만 YAML은 들여쓰기로 계층을 표현하므로 들여쓰기 실수가 곧 구조 오류가 된다는 점은 주의해야 합니다.

## 5. 정리

| 주제 | 핵심 |
|------|------|
| 외부 설정의 동기 | 한 번 빌드, 환경 차이는 실행 시점에 주입 |
| 네 통로 | OS 환경변수 · 자바 시스템 속성 · 커맨드 라인 · 설정 파일 |
| 옵션 인수 | `--key=value` 형식, 스프링 부트가 키-값으로 파싱 |
| Environment | 모든 통로를 PropertySource로 모아 단일 조회 창구 제공 |
| 우선순위 | 커맨드 라인 > 시스템 속성 > 환경 변수 > 설정 파일 |
| 설정 파일 | properties(평면) / yml(계층), 기본값 보관에 적합 |
