# @ConfigurationProperties와 타입 안전 설정
---

> `env.getProperty("my.timeout")` 은 동작하지만 위험합니다. 키를 오타 내도, 타입을 잘못 캐스팅해도 컴파일러가 잡아 주지 못하고 런타임에야 터집니다. `@ConfigurationProperties` 는 외부 설정을 *객체로 묶어* 이 약점을 메웁니다. 이 문서를 다 읽으면 `@Value` 와 `@ConfigurationProperties` 의 차이를 설명하고, 설정을 불변 객체로 받아 시작 시점에 검증하는 안전한 패턴을 쓸 수 있게 됩니다.

## 1. @Value — 한 값씩 꽂기

가장 단순한 주입은 `@Value` 입니다. 설정 키 하나를 필드 하나에 직접 꽂습니다.

```java
@Component
public class MyBean {
    @Value("${my.datasource.url}")
    private String url;

    @Value("${my.datasource.max-connection}")
    private int maxConnection;

    @Value("${my.datasource.timeout:3000}")  // 콜론 뒤는 기본값
    private int timeout;
}
```

`@Value("${...}")` 는 [02-01](02-01.외부%20설정%20—%20커맨드라인부터%20application.yml까지.md)에서 본 `Environment` 에서 값을 찾아 꽂고, `${key:default}` 형식으로 기본값도 줄 수 있습니다. 설정이 한두 개면 충분합니다.

문제는 설정이 늘어날 때입니다. 관련된 설정(`my.datasource.*`)이 여러 빈에 흩어져 `@Value` 로 박히면, "데이터소스 설정이 무엇무엇인가"가 한곳에 모이지 않습니다. 키 문자열을 직접 쓰므로 오타가 나도 컴파일러가 모르고, `max-connection` 을 `String` 필드에 꽂는 식의 타입 실수도 런타임에야 드러납니다.

## 2. @ConfigurationProperties — 묶어서 객체로

`@ConfigurationProperties` 는 공통 접두사를 가진 설정들을 *하나의 객체*로 묶어 받습니다. `my.datasource.*` 전체가 한 클래스의 필드로 바인딩됩니다.

```java
@ConfigurationProperties(prefix = "my.datasource")
public class MyDataSourceProperties {
    private String url;
    private String username;
    private int maxConnection;
    private Duration timeout;
    // getter / setter
}
```

```yaml
my:
  datasource:
    url: jdbc:h2:mem:test
    username: sa
    max-connection: 10
    timeout: 3s
```

`prefix` 아래 키들이 필드 이름으로 자동 매핑됩니다(`max-connection` → `maxConnection` 처럼 케밥-케이스가 카멜-케이스로 변환). 이렇게 묶으면 세 가지가 좋아집니다. 관련 설정이 한 클래스에 모여 *무엇을 설정할 수 있는지*가 한눈에 보이고, 타입이 필드 선언으로 고정되어 `Duration timeout` 처럼 문자열이 아닌 도메인 타입으로 받으며, 이 객체를 빈으로 주입받으면 키 문자열이 코드에서 사라집니다.

타입 변환은 `Duration` 외에도 자주 쓰는 raw 타입이 자동으로 처리됩니다. `long pollIntervalMs` 는 yml 의 숫자 `500` 을 그대로 받고, `String cleanupCron` 은 `"0 0 3 * * *"` 같은 cron 표현식을 문자열로 받습니다. 사내 `org.okestro.tps.messaging.property.OutboxProperties` 가 이 세 가지(`int batchSize`·`long pollIntervalMs`·`String cleanupCron`)를 한 클래스에 묶어 두는 사례입니다 — 타입 결정이 곧 *그 필드를 어떤 단위로 다룰지* 의 선언이기도 합니다.

등록은 두 방법이 있습니다. 설정 클래스에서 `@EnableConfigurationProperties(MyDataSourceProperties.class)` 로 켜거나, properties 클래스 자체에 `@ConfigurationPropertiesScan` 이 잡도록 두는 방식입니다.

자동 구성 클래스가 properties 객체를 *주입받아 빈을 만드는* 패턴이라면 보통 첫 번째 방법을 씁니다. 사내 `OutboxAutoConfiguration` 이 `@EnableConfigurationProperties(OutboxProperties.class)` 를 클래스에 직접 박아 두는 이유가 이것입니다. 라이브러리는 *컴포넌트 스캔 경로 밖* 에서 동작하므로 `@ConfigurationPropertiesScan` 에 의존하면 사용자 프로젝트마다 스캔 범위 설정이 달라 활성화가 불안정해집니다. 자동 구성에 묶어 두면 properties 등록과 빈 등록이 한 자리에서 보장됩니다.

## 3. 생성자 바인딩 — 불변 설정

2절의 예제는 setter로 값을 받습니다. 그런데 설정값은 애플리케이션 구동 중 바뀌어선 안 되는 *불변값*입니다. setter가 열려 있으면 누군가 런타임에 바꿀 수 있어 위험합니다. 생성자 바인딩을 쓰면 필드를 `final` 로 못 박아 불변을 보장합니다.

```java
@ConfigurationProperties(prefix = "my.datasource")
public class MyDataSourceProperties {
    private final String url;
    private final String username;
    private final int maxConnection;
    private final Duration timeout;

    public MyDataSourceProperties(String url, String username,
                                  int maxConnection, Duration timeout) {
        this.url = url;
        this.username = username;
        this.maxConnection = maxConnection;
        this.timeout = timeout;
    }
    // getter만
}
```

setter가 없으니 외부에서 값을 바꿀 통로가 막힙니다. 생성자 바인딩은 [`../01_core/01-01.객체지향 원리 적용 — DI와 IoC`](../01_core/01-01.객체지향%20원리%20적용%20—%20DI와%20IoC.md) §6 에서 생성자 주입을 권장한 이유(불변성·누락 컴파일 차단)와 같은 발상을 설정 객체에 적용한 것입니다.

## 4. 검증 — 잘못된 설정을 시작 시점에 막기

타입을 객체로 묶었어도, 값 자체가 비정상일 수 있습니다. `maxConnection` 에 음수가 들어오거나 `url` 이 비어 있으면 애플리케이션은 한참 돌다가 엉뚱한 곳에서 실패합니다. `@ConfigurationProperties` 는 Bean Validation 어노테이션과 결합해 *시작 시점에* 설정값을 검증할 수 있습니다.

```java
@ConfigurationProperties(prefix = "my.datasource")
@Validated
public class MyDataSourceProperties {

    @NotEmpty
    private final String url;

    @Min(1) @Max(100)
    private final int maxConnection;

    private final Duration timeout;
    // 생성자, getter
}
```

`@Validated` 를 붙이고 필드에 `@NotEmpty`, `@Min`, `@Max` 같은 제약을 걸면, 설정값이 제약을 어길 때 애플리케이션 *시작 자체가 실패*합니다. "잘못된 설정으로 일단 떠서 운영 중에 터지는" 사고를, "잘못된 설정이면 아예 안 뜨는" 안전한 실패로 바꿉니다. [02-02 — 자동 구성의 fail-fast 검증](../06_events/01-02.트랜잭션%20전파%20조합%20—%20죽은%20트랜잭션과%20REQUIRES_NEW.md) §3 에서 본 "조용한 실패보다 시작 시점 명확한 실패"와 같은 철학입니다.

## 5. @Value vs @ConfigurationProperties

| 기준 | @Value | @ConfigurationProperties |
|------|--------|--------------------------|
| 단위 | 키 하나씩 | 접두사 묶음 객체 |
| 타입 안전 | 약함 (키 문자열·캐스팅 런타임) | 강함 (필드 타입 고정) |
| 오타 감지 | 런타임 | 바인딩 시점·검증으로 조기 |
| 불변성 | 어려움 | 생성자 바인딩으로 보장 |
| 검증 | 없음 | `@Validated` 로 시작 시점 검증 |
| 적합한 경우 | 설정 1~2개, 단순 값 | 관련 설정 묶음, 라이브러리 설정 |

설정이 단순하고 적으면 `@Value` 로 충분하지만, 관련 설정이 묶여 있거나(특히 [01-03 커스텀 스타터](01-03.커스텀%20스타터%20만들기.md)처럼 라이브러리가 설정을 노출할 때) `@ConfigurationProperties` 가 타입 안전·불변·검증을 한꺼번에 줍니다.

## 6. 정리

| 주제 | 핵심 |
|------|------|
| @Value | 키 하나를 필드에 직접 주입, 단순하지만 타입 안전 약함 |
| @ConfigurationProperties | 접두사 묶음을 객체로 바인딩, 키 문자열 제거 |
| 생성자 바인딩 | final 필드로 설정 불변 보장 |
| @Validated | 잘못된 설정을 시작 시점에 fail-fast |
| 선택 기준 | 단순 값은 @Value, 묶음·라이브러리는 @ConfigurationProperties |
