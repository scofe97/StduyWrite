# 프로필 — 환경별 설정 분리
---

> 개발 서버는 개발 DB에, 운영 서버는 운영 DB에 연결해야 합니다. [02-01](02-01.외부%20설정%20—%20커맨드라인부터%20application.yml까지.md)에서 본 외부 설정이 "값을 밖에서 주입"하는 장치라면, 프로필은 "환경별로 *어떤 설정 묶음을 켤지*" 고르는 스위치입니다. 이 문서를 다 읽으면 한 파일 안에서 dev/prod 설정을 나누는 방법과, 특정 환경에서만 빈을 등록하는 `@Profile` 의 쓰임을 설명할 수 있게 됩니다.

## 1. 프로필이 푸는 문제

외부 설정만으로도 환경 차이를 흡수할 수 있지만, 환경마다 *바꿔야 할 설정이 여러 개*면 그것들을 일일이 따로 주입하기 번거롭습니다. DB 주소·계정·풀 크기·로그 레벨이 환경마다 다르다면, 매 실행마다 이 값들을 전부 커맨드 라인으로 넘길 수는 없습니다.

프로필은 이 설정 묶음에 *이름*을 붙입니다. `dev` 프로필, `prod` 프로필처럼 환경별 설정 한 벌을 묶어 두고, 실행 시 "어떤 프로필로 띄울지"만 고르면 그 묶음이 통째로 적용됩니다. 개별 값을 주입하는 대신 묶음을 선택하는 방식입니다.

## 2. 프로필별 설정 파일

가장 직관적인 방법은 프로필마다 설정 파일을 두는 것입니다. `application-{프로필}.properties` (또는 `.yml`) 규칙을 따릅니다.

```text
application.properties          # 공통 설정 (항상 적용)
application-dev.properties      # dev 프로필에서만 적용
application-prod.properties     # prod 프로필에서만 적용
```

```properties
# application-dev.properties
my.datasource.url=jdbc:h2:mem:dev
my.datasource.username=sa

# application-prod.properties
my.datasource.url=jdbc:mysql://prod-db:3306/app
my.datasource.username=app_user
```

활성 프로필은 `spring.profiles.active` 로 지정합니다. [02-01 §3](02-01.외부%20설정%20—%20커맨드라인부터%20application.yml까지.md)의 우선순위가 그대로 적용되므로, 설정 파일에 기본 프로필을 두고 실행 시 커맨드 라인으로 덮어쓸 수 있습니다.

```bash
java -jar app.jar --spring.profiles.active=prod
```

프로필을 지정하지 않으면 `default` 프로필로 동작합니다. 공통 `application.properties` 는 프로필과 무관하게 항상 적용되고, 그 위에 활성 프로필의 파일이 덧씌워집니다.

## 3. 한 파일 안에서 나누기 — YAML 문서 분리

파일을 여러 개로 쪼개는 대신, YAML 한 파일 안에서 `---` 구분자로 프로필별 섹션을 나눌 수도 있습니다. 설정 전체를 한눈에 보고 싶을 때 편합니다.

```yaml
# application.yml — 공통 설정
my:
  app-name: my-service

---
spring:
  config:
    activate:
      on-profile: dev
my:
  datasource:
    url: jdbc:h2:mem:dev

---
spring:
  config:
    activate:
      on-profile: prod
my:
  datasource:
    url: jdbc:mysql://prod-db:3306/app
```

`---` 가 YAML 문서를 나누고, `spring.config.activate.on-profile` 이 그 섹션이 어느 프로필에서 활성화될지 지정합니다. 첫 섹션(프로필 지정 없음)은 공통으로 항상 적용되고, `dev` 로 띄우면 첫 섹션 + dev 섹션이 합쳐집니다. 파일 하나로 모든 환경 설정을 관리하므로 환경 간 차이를 한 화면에서 비교할 수 있습니다.

## 4. @Profile — 환경별 빈 등록

프로필은 설정값뿐 아니라 *빈 등록*도 가릅니다. 특정 환경에서만 필요한 빈은 `@Profile` 로 조건을 겁니다.

```java
@Configuration
public class PayConfig {

    @Bean
    @Profile("dev")
    public PayClient testPayClient() {
        // 개발 환경 — 실제 결제 없이 항상 성공하는 가짜 클라이언트
        return new MockPayClient();
    }

    @Bean
    @Profile("prod")
    public PayClient realPayClient() {
        // 운영 환경 — 실제 결제 게이트웨이 연동
        return new RealPayClient();
    }
}
```

`dev` 로 띄우면 `MockPayClient` 가, `prod` 로 띄우면 `RealPayClient` 가 같은 `PayClient` 타입으로 주입됩니다. 사용하는 쪽 코드는 `PayClient` 인터페이스에만 의존하므로 환경에 따라 구현이 바뀌어도 그대로입니다. [01-02 자동 구성](01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md)의 `@Conditional` 이 클래스패스·빈 존재를 조건으로 빈을 가렸다면, `@Profile` 은 *활성 프로필*을 조건으로 빈을 가르는 같은 계열의 장치입니다.

## 5. 정리

| 주제 | 핵심 |
|------|------|
| 프로필의 역할 | 환경별 설정 묶음에 이름을 붙여 통째로 전환 |
| 프로필별 파일 | `application-{프로필}.yml`, 공통 + 활성 프로필 합성 |
| 활성 지정 | `spring.profiles.active`, 미지정 시 `default` |
| YAML 문서 분리 | `---` + `on-profile` 로 한 파일 안에서 환경 분리 |
| @Profile | 활성 프로필을 조건으로 빈 등록 (Mock vs Real) |
| @Conditional 과의 관계 | 같은 "조건부 빈" 계열, 조건이 프로필인 것 |
