# 스타터와 라이브러리 버전 관리
---

> `spring-boot-starter-web` 한 줄이면 웹 애플리케이션에 필요한 수십 개 라이브러리가 호환되는 버전으로 한꺼번에 딸려 옵니다. 이 문서를 다 읽으면 그 한 줄 뒤에서 스프링 부트가 *버전 관리*와 *묶음 제공*이라는 두 가지 일을 어떻게 나눠서 하는지 설명할 수 있고, "버전을 명시하지 않았는데 어떻게 정해지는가"라는 질문에 BOM이라는 단어로 답할 수 있게 됩니다.

## 1. 라이브러리를 직접 고르던 시절의 고통

스프링으로 웹 애플리케이션 하나를 띄우려면 생각보다 많은 라이브러리가 필요합니다. 스프링 웹 MVC, 내장 톰캣, JSON 처리기, 로거, YAML 파서가 전부 있어야 비로소 `@RestController` 하나가 응답합니다. 스프링 부트가 없던 시절에는 이것을 전부 직접 골라야 했습니다.

```groovy
dependencies {
    //1. 라이브러리 직접 지정
    //스프링 웹 MVC
    implementation 'org.springframework:spring-webmvc:6.0.4'
    //내장 톰캣
    implementation 'org.apache.tomcat.embed:tomcat-embed-core:10.1.5'
    //JSON 처리
    implementation 'com.fasterxml.jackson.core:jackson-databind:2.14.1'
    //스프링 부트 관련
    implementation 'org.springframework.boot:spring-boot:3.0.2'
    implementation 'org.springframework.boot:spring-boot-autoconfigure:3.0.2'
    //LOG 관련
    implementation 'ch.qos.logback:logback-classic:1.4.5'
    implementation 'org.apache.logging.log4j:log4j-to-slf4j:2.19.0'
    implementation 'org.slf4j:jul-to-slf4j:2.0.6'
    //YML 관련
    implementation 'org.yaml:snakeyaml:1.33'
}
```

여기에 더 어려운 문제가 숨어 있습니다. 라이브러리를 *아는 것*도 일이지만, 각 라이브러리의 *버전*까지 골라야 하고, 무엇보다 *라이브러리끼리 호환되는 버전 조합*을 맞춰야 합니다. 어떤 조합은 잘 돌고 어떤 조합은 미묘하게 깨지는데, 그 깨짐이 컴파일 에러가 아니라 런타임 동작 이상으로 나타나면 원인 추적이 길어집니다. 과거에는 이 버전 조합 맞추기에 프로젝트 초기 세팅 시간의 상당 부분을 썼습니다.

스프링 부트는 이 고통을 두 단계로 나눠 풉니다. *버전 관리*와 *스타터 제공*입니다.

## 2. 버전 관리 — BOM이 호환 조합을 책임진다

스프링 부트는 자신이 검증한 라이브러리 버전 조합을 한 곳에 모아 둔 BOM(Bill of Materials)을 제공합니다. BOM은 "이 버전들끼리는 호환을 보장한다"는 *버전 명세서*입니다. 개발자가 버전을 적지 않으면 BOM에 정의된 버전이 적용됩니다.

```groovy
plugins {
    id 'org.springframework.boot' version '3.0.2'
    id 'io.spring.dependency-management' version '1.1.0'
    id 'java'
}

dependencies {
    // 버전을 적지 않는다 — BOM이 정한 버전이 적용된다
    implementation 'org.springframework:spring-webmvc'
    implementation 'org.apache.tomcat.embed:tomcat-embed-core'
    implementation 'com.fasterxml.jackson.core:jackson-databind'
}
```

`io.spring.dependency-management` 플러그인이 BOM을 읽어 버전을 채워 넣는 역할을 합니다. 버전 번호가 사라진 의존성 목록은 단순히 짧아진 것이 아니라, *버전 충돌의 책임을 개발자에게서 스프링 부트로 넘긴* 것입니다. 스프링 부트 버전을 `3.0.2` 에서 올리면 딸린 라이브러리 버전들이 검증된 조합으로 함께 올라갑니다.

특정 라이브러리만 버전을 다르게 써야 한다면 그 의존성에만 버전을 명시해 BOM 값을 덮어쓸 수 있습니다. 기본은 BOM에 맡기고, 예외만 직접 지정하는 방식입니다.

## 3. 스타터 제공 — 묶음으로 한 번에

버전 관리가 "버전을 정해 준다"라면, 스타터는 "필요한 라이브러리 묶음을 통째로 가져온다"입니다. `spring-boot-starter-web` 의존성 하나가 웹 개발에 필요한 라이브러리 전체를 끌어옵니다.

```groovy
dependencies {
    // 이 한 줄이 spring-webmvc, 내장 톰캣, jackson, validation 등을 한꺼번에 가져온다
    implementation 'org.springframework.boot:spring-boot-starter-web'
}
```

스타터는 일정한 이름 규칙을 따릅니다. `spring-boot-starter-*` 형태이며, `*` 자리에 용도가 들어갑니다. 자주 쓰는 스타터는 다음과 같습니다.

| 스타터 | 용도 |
|--------|------|
| `spring-boot-starter` | 코어 — 자동 구성, 로깅, YAML (모든 스타터의 기반) |
| `spring-boot-starter-web` | Spring MVC + 내장 톰캣 + Jackson |
| `spring-boot-starter-data-jpa` | Spring Data JPA + Hibernate |
| `spring-boot-starter-test` | JUnit·Mockito·AssertJ 등 테스트 도구 묶음 |
| `spring-boot-starter-actuator` | 운영 모니터링 엔드포인트 |

스타터가 단순한 "라이브러리 alias"가 아니라는 점이 중요합니다. 스타터를 추가하면 그 안의 라이브러리들이 클래스패스에 올라가고, 그것을 신호로 [다음 문서](01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md)에서 다룰 *자동 구성*이 작동합니다. "라이브러리가 클래스패스에 있으면 알아서 빈을 등록한다"는 자동 구성의 전제가 바로 이 스타터로 채워집니다.

## 4. 두 기능이 맞물리는 자리

버전 관리와 스타터는 따로 동작하지만 함께 쓰일 때 효과가 완성됩니다. 스타터가 라이브러리 묶음을 *결정*하고, BOM이 그 묶음 안 각 라이브러리의 버전을 *호환되게 채웁니다*. 그래서 `spring-boot-starter-web` 한 줄에 버전조차 안 적어도 수십 개 라이브러리가 검증된 조합으로 들어옵니다.

```text
implementation 'org.springframework.boot:spring-boot-starter-web'
        │
        ├─ (스타터) 웹에 필요한 라이브러리 묶음 결정
        │     spring-webmvc, tomcat-embed, jackson, validation ...
        │
        └─ (BOM) 각 라이브러리의 호환 버전 자동 적용
              spring-webmvc:6.x, tomcat:10.x, jackson:2.x ...
```

정리하면, 1절의 길고 버전이 빼곡하던 의존성 목록이 스타터 한 줄로 줄어든 것은 *타이핑이 줄어든 것*이 아니라 *라이브러리 선택과 버전 호환이라는 두 가지 의사결정을 스프링 부트에 위임한 것*입니다.

## 5. 정리

| 기능 | 하는 일 | 효과 |
|------|--------|------|
| BOM + dependency-management | 호환되는 라이브러리 버전 조합 제공 | 버전 충돌 책임을 스프링 부트로 이전 |
| 스타터 (`spring-boot-starter-*`) | 용도별 라이브러리 묶음 제공 | 라이브러리 선택 고민 제거 |
| 둘의 결합 | 스타터가 묶음 결정, BOM이 버전 채움 | 한 줄로 검증된 조합 확보 |
| 자동 구성과의 연결 | 스타터가 클래스패스를 채움 | 자동 구성 작동의 전제 |
