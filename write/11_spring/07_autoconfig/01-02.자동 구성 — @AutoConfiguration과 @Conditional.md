# 자동 구성 — @AutoConfiguration과 @Conditional
---

> 스프링 부트 프로젝트에서 `DataSource` 와 `JdbcTemplate` 을 빈으로 등록한 적이 없는데도 주입받아 쓸 수 있습니다. 누가 등록했을까요? 스프링 부트의 *자동 구성*입니다. 이 문서는 그 "마법"을 세 겹으로 벗겨 냅니다. 클래스패스 상태에 따라 빈을 조건부로 등록하는 `@Conditional`, 자동 구성 클래스를 무더기로 가져오는 `ImportSelector`, 그리고 그 둘을 묶은 `@AutoConfiguration` 입니다. 끝까지 읽으면 "라이브러리만 추가했는데 빈이 생긴다"가 마법이 아니라 명확한 조건 평가의 결과임을 알게 됩니다.

## 1. 등록하지 않은 빈이 존재하는 미스터리

`JdbcTemplate`, `DataSource`, `TransactionManager` 를 직접 등록하는 설정 클래스가 있다고 합시다.

```java
//@Configuration  // 주석처리
public class DbConfig {
    @Bean
    public DataSource dataSource() {
        HikariDataSource dataSource = new HikariDataSource();
        dataSource.setDriverClassName("org.h2.Driver");
        dataSource.setJdbcUrl("jdbc:h2:mem:test");
        dataSource.setUsername("sa");
        dataSource.setPassword("");
        return dataSource;
    }
    @Bean
    public TransactionManager transactionManager() {
        return new JdbcTransactionManager(dataSource());
    }
    @Bean
    public JdbcTemplate jdbcTemplate() {
        return new JdbcTemplate(dataSource());
    }
}
```

`@Configuration` 을 주석 처리하면 이 설정 파일은 컴포넌트 스캔 대상에서 빠지므로, 위 세 빈은 등록되지 않아야 합니다. 그런데 빈을 조회하는 테스트를 돌리면 `DataSource`, `TransactionManager`, `JdbcTemplate` 이 *여전히 존재*합니다.

```text
..DbConfigTest: dataSource = HikariDataSource (null)
..DbConfigTest: transactionManager = org.springframework.jdbc.support.JdbcTransactionManager@5e99e2cb
..DbConfigTest: jdbcTemplate = org.springframework.jdbc.core.JdbcTemplate@76ac68b0
```

우리가 등록하지 않았는데도 빈이 있습니다. 이 빈들은 모두 스프링 부트가 *자동으로* 등록해 준 것입니다. `spring-boot-starter-jdbc` 같은 의존성이 클래스패스에 있다는 사실만으로 스프링 부트가 알아서 등록했습니다. 어떻게 가능할까요?

## 2. @Conditional — 조건이 맞을 때만 등록

자동 구성의 가장 작은 부품은 `@Conditional` 입니다. "이 조건이 참일 때만 이 설정을 적용하라"는 선언입니다. `@Conditional` 자체는 스프링 부트가 아니라 스프링 프레임워크의 기능이고, 스프링 부트는 이것을 확장해 다양한 조건 어노테이션을 제공합니다.

| 어노테이션 | 조건 |
|----------|------|
| `@ConditionalOnClass` | 특정 클래스가 클래스패스에 있을 때 |
| `@ConditionalOnBean` | 해당 타입의 빈이 이미 등록돼 있을 때 (대칭 동작) |
| `@ConditionalOnMissingBean` | 해당 타입의 빈이 아직 없을 때 |
| `@ConditionalOnProperty` (값 일치) | 특정 설정값이 지정된 값일 때 (`havingValue` 지정) |
| `@ConditionalOnProperty` (키 존재) | 키 자체가 *어떤 값으로든* 설정돼 있을 때 (`value`/`havingValue` 생략) |
| `@ConditionalOnWebApplication` | 웹 애플리케이션일 때 |
| `@ConditionalOnSingleCandidate` | 해당 타입 빈이 유일하거나 primary일 때 |
| `@ConditionalOnResource` | 특정 리소스가 존재할 때 |

`@ConditionalOnBean` 과 `@ConditionalOnProperty` 키-존재 형태는 본 표에서는 한 줄로만 짚고 넘어갑니다. *다른 자동 구성이 만든 빈* 을 조건으로 거는 자리와 *환경마다 값이 다른 키* 의 존재만 게이트로 쓰는 자리는 자동 구성 심화에서 자주 만나는 두 패턴이며, 사내 사례와 함께 [01-04 §2~§3](01-04.자동%20구성%20심화%20—%20순서·게이트·기본값%20주입.md) 에서 따로 다룹니다.

자동 구성 클래스는 이 조건들을 조합해 "이 라이브러리가 있고, 사용자가 직접 등록한 빈이 없을 때만 기본 빈을 등록한다"를 표현합니다.

```java
@AutoConfiguration
@ConditionalOnClass(JdbcTemplate.class)  // JdbcTemplate 클래스가 클래스패스에 있을 때만
public class JdbcTemplateAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean  // 사용자가 직접 등록한 JdbcTemplate이 없을 때만
    public JdbcTemplate jdbcTemplate(DataSource dataSource) {
        return new JdbcTemplate(dataSource);
    }
}
```

여기서 `@ConditionalOnClass(JdbcTemplate.class)` 가 1절의 미스터리를 풉니다. `spring-boot-starter-jdbc` 를 추가하면 `JdbcTemplate` 클래스가 클래스패스에 올라가고, 그 순간 이 조건이 참이 되어 자동 구성이 빈을 등록합니다. 라이브러리 추가가 곧 조건 충족입니다.

`@ConditionalOnMissingBean` 의 역할도 중요합니다. 사용자가 `DbConfig` 로 직접 `JdbcTemplate` 을 등록하면 이 조건이 거짓이 되어 자동 구성은 *물러납니다*. 그래서 1절에서 `@Configuration` 을 살려 두면 우리 빈이 쓰이고, 주석 처리하면 자동 구성 빈이 대신 들어옵니다. "사용자 설정이 항상 우선, 없으면 기본값 제공"이라는 자동 구성의 기본 예의가 이 한 어노테이션에 담겨 있습니다.

## 3. ImportSelector — 자동 구성 클래스를 무더기로 가져오기

조건부 설정 클래스 하나하나는 이해됐습니다. 그런데 스프링 부트에는 이런 자동 구성 클래스가 수백 개입니다. 이것들을 어떻게 한꺼번에 등록할까요? `@Import` 로 일일이 적을 수는 없습니다.

여기서 `ImportSelector` 가 등장합니다. `@Import` 는 보통 클래스를 직접 지정하지만, `ImportSelector` 구현체를 지정하면 *문자열 배열로 가져올 클래스 이름들을 동적으로 결정*할 수 있습니다.

```java
public interface ImportSelector {
    // 등록할 설정 클래스 이름들을 동적으로 반환
    String[] selectImports(AnnotationMetadata importingClassMetadata);
}
```

정적 `@Import(AppConfig.class)` 가 "이 클래스를 등록하라"라면, `ImportSelector` 는 "어떤 클래스들을 등록할지 *런타임에 계산해서* 알려 주겠다"입니다. 이 동적 계산 덕분에 "클래스패스에 있는 자동 구성 클래스 목록을 읽어 전부 등록"하는 일이 가능해집니다.

스프링 부트는 자동 구성 클래스 목록을 정해진 파일에 적어 둡니다.

```text
# META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
org.springframework.boot.autoconfigure.jdbc.JdbcTemplateAutoConfiguration
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
org.springframework.boot.autoconfigure.transaction.TransactionAutoConfiguration
... (수백 개)
```

`ImportSelector` 구현체가 이 `AutoConfiguration.imports` 파일을 읽어 전체 자동 구성 클래스 이름을 반환하고, 스프링은 그 클래스들을 모두 후보로 등록합니다. 후보가 *실제로 빈을 만드는지*는 2절의 `@Conditional` 이 각자 판단합니다. 즉 "전부 후보로 올리고, 조건 맞는 것만 살아남는다"는 2단 구조입니다.

## 4. @EnableAutoConfiguration — 시작 스위치

이 전체 흐름의 시작 스위치는 `@EnableAutoConfiguration` 입니다. 그리고 이것은 우리가 매일 쓰는 `@SpringBootApplication` 안에 이미 들어 있습니다.

```java
@SpringBootApplication
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}
```

`@SpringBootApplication` 은 세 어노테이션의 묶음입니다.

| 포함 어노테이션 | 역할 |
|---------------|------|
| `@SpringBootConfiguration` | 설정 클래스임을 표시 (`@Configuration` 의 부트 버전) |
| `@ComponentScan` | 현재 패키지 이하 컴포넌트 스캔 |
| `@EnableAutoConfiguration` | 자동 구성 활성화 — `ImportSelector` 를 통해 자동 구성 클래스 등록 |

그래서 메인 클래스에 `@SpringBootApplication` 한 줄만 붙이면 자동 구성이 켜집니다. `@EnableAutoConfiguration` 이 `ImportSelector` 를 트리거하고(3절), 그것이 가져온 자동 구성 클래스들이 `@Conditional` 로 자기 등록 여부를 판단하며(2절), 그 결과로 우리가 등록하지 않은 `JdbcTemplate` 이 존재하게 됩니다(1절).

## 5. 전체 흐름

세 겹을 한 줄로 꿰면 이렇게 됩니다.

```text
@SpringBootApplication
  └─ @EnableAutoConfiguration            ← 시작 스위치
       └─ ImportSelector                 ← AutoConfiguration.imports 파일을 읽어
            └─ 자동 구성 클래스 수백 개를 후보로 등록
                 └─ 각 클래스의 @Conditional 평가
                      ├─ @ConditionalOnClass: 라이브러리가 클래스패스에 있는가?
                      ├─ @ConditionalOnMissingBean: 사용자가 직접 등록 안 했는가?
                      └─ 조건 통과한 것만 실제 빈 등록 ★
```

"라이브러리만 추가했는데 빈이 생긴다"의 정체는 이것입니다. 스타터가 클래스패스를 채우고(01-01), `@ConditionalOnClass` 가 그 변화를 감지해 자동 구성을 깨우며, `@ConditionalOnMissingBean` 이 사용자 설정을 존중합니다. 마법이 아니라 조건 평가의 연쇄입니다.

## 6. 정리

| 부품 | 역할 |
|------|------|
| `@Conditional` 계열 | 클래스패스·빈·설정값 조건으로 등록 여부 결정 |
| `@ConditionalOnClass` | 라이브러리 존재가 자동 구성을 깨우는 신호 |
| `@ConditionalOnMissingBean` | 사용자 직접 등록 빈을 우선, 없을 때만 기본 제공 |
| `ImportSelector` | 등록할 설정 클래스 이름을 런타임에 동적 결정 |
| `AutoConfiguration.imports` | 자동 구성 클래스 전체 목록 파일 |
| `@AutoConfiguration` | 조건과 빈 정의를 묶은 자동 구성 클래스 표시 |
| `@EnableAutoConfiguration` | 전체 흐름의 시작 스위치 (`@SpringBootApplication` 내부) |
