# 스프링 이벤트 소싱

주제: Spring Study

# 📚 Event Sourcing 심층 가이드

## 0️⃣ 왜 이 문서를 읽어야 하는가?

- *상태를 “행”이 아닌 “이벤트”**로 저장하는 방식을 고민한다.
- 메시지 큐·카프카를 못 쓰는 환경에서 **“스프링 기본 이벤트 + RDB”** 만으로 실전 적용하고 싶다.
- “DDD(도메인 주도 설계)”란 말을 들어 보긴 했지만, CRUD 프로젝트와 무엇이 다른지 모른다.

---

## 1️⃣ Event Sourcing이란 무엇인가?

| 전통 CRUD | **Event Sourcing** |
| --- | --- |
| 테이블에 *현재* 상태 1 행 보관 | 상태 변화 **이벤트(사건)** 를 시간순으로 영속화 |
| `UPDATE ticket SET status='DONE'` | `TicketStatusChanged{from:PROGRESS,to:DONE}` Append |
| 마지막 행 = 진실 | 이벤트 스트림 **재생** = 진실 |
| 상태변경 이력은 로그 테이블 별도 관리(종종 누락) | 이력이 곧 데이터. 모든 변경이 불변 기록 |

> 핵심 정의 한 줄
> 
> 
> “**현재 상태 = 이벤트들의 합**(Σ).”
> 

---

## 2️⃣ CRUD ↔ Event Sourcing 비교 분석

| 항목 | CRUD | **Event Sourcing** |
| --- | --- | --- |
| **저장 단위** | 객체/행 Snapshot | Domain Event(Stream) |
| **읽기 모델** | 테이블 그대로 | Projection(조회용 테이블) 필요 |
| **Rollback** | 과거 상태 값 백업 필요 | n 번째 이벤트까지 Replay |
| **감사/추적** | 별도 로그 or Audit Trigger | 본 데이터가 감사 기록 |
| **성능** | 단순 Join/인덱스 튜닝 | 쓰기는 Append-Only, 읽기는 Projection Index |
| **복잡도** | 낮음 | 아키텍처·코드 난이도 ↑ |
| **장점** | 배우기 쉽다, 쿼리 간편 | 이력 100 % 보존, CQRS·시간여행, 불변데이터 |
| **단점** | 이력 누락 위험, 변경 추적 어려움 | Projection + Snapshot 설계 필요 |

## 3️⃣ Spring Boot(내부 Event) + RDB 로만 구축하려면?

1. **EventStore 테이블** (`stored_event`)
    
    ```
    id | aggregate_id | event_type | payload(json) | occurred_at
    ```
    
2. **`EventStoreService.append(evt)`**
    - `@Transactional` 내부에서 insert → Commit.
3. **`ApplicationEventPublisher.publishEvent(evt)`**
    - 같은 JVM 내 모듈들이 `@EventListener` 로 후속 작업.
4. **Projection 테이블** (예: `ticket_read`)
    - 리스너에서 insert/update.
    - 이 테이블은 *조회 전용*, 쓰기는 금지.

> 메시지 브로커가 없어도 단일 애플리케이션 모놀리식이라면 충분히 동작한다.
> 
> 
> 여러 JVM 분리(마이크로서비스)는 불가 → 그땐 Kafka 등 필요.
>

## 4️⃣ Event Sourcing 도입 장단점 정리

### ✅ 장점

1. **모든 이력** 보존 → 규제·감사 용이.
2. **시간여행(Time-Travel)**
    - “2024-12-01 오전 10시 티켓 상태?” → 해당 시점까지 Replay.
3. **CQRS** (쓰기 모델·읽기 모델 분리) 성능 최적.
4. **불변 데이터 → 동시성 충돌↓** (Append-Only).
5. **캡슐화**: Aggregate 로직 + 이벤트만 공개 → 도메인 규칙 보호.

### ❌ 단점

1. **학습 곡선**: 개발·DBA 모두 새로운 사고방식 필요.
2. **Projection 복잡도**: 조회 테이블 유지·정합성 이슈.
3. **데이터 량 증가**: 이벤트가 많아지면 Storage 폭증 → Snapshot 전략 필요.
4. **쿼리 디버깅**: 즉석 Join 불가 → Projection or Replay 필요.
5. **테스트 난이도**: Stub EventStore, Replay 유닛테스트 필요.

## 5️⃣ DDD 기본문법 vs. “일반 Layered CRUD”

| 개념 | 레이어드 CRUD | **DDD + Event Sourcing** |
| --- | --- | --- |
| **Aggregate** | Entity & Service 섞임 | Aggregate Root 가 자신 안에서 상태 변경·이벤트 생성 |
| **Invariant** | Service 단에서 if-else 검증 | Aggregate 내부 메서드로만 변경 가능 → 규칙 보장 |
| **Repository** | JPA Repository = Table mapping | EventStore Repository = `append` / `load` |
| **Service** | 트랜잭션·비즈니스 혼재 | CommandHandler(Write), Projection(Read) 분리 |
| **이벤트** | 주로 통합(Integration) 이벤트 | Domain Event = 모델 핵심, 상태 변경 원본 |
| **UML 용어** | Entity-DTO-DAO | Aggregate-Command-Event-Projection |

> DDD를 깊이 몰라도, “Aggregate = 불변 룰 + 이벤트 발생 책임자” 정도만 이해하면 첫 단계 설계 가능.

## 6️⃣ Snapshot & Projection 관리 전략

1. **Snapshot 이벤트**
    - 1000 개 이벤트마다 `TicketSnapshotted{snapshotJson}` 저장.
    - Rebuild 시 Snapshot + 이후 이벤트만 Replay → 속도 향상.
2. **Async Projection 리빌드**
    - 새 필드 추가 시 **전체 로그 Replay → Projection 테이블 리빌드** 배치.
    - RDB Space 충분하면 Projection 테이블 여러개 유지(CQRS).

---

## 7️⃣ 보상 트랜잭션(Compensation) 패턴

- EventListener 실패 → `stored_event` 의 “pending” 플래그 사용 OR
    
    `compensation_queue` 테이블에 적재 → `@Scheduled` 재시도.
    
- 실패가 치명적이면 **Saga** 패턴 고려 (Ticket CREATED → Workflow CREATION_FAILED → Ticket CANCELLED 이벤트).

## 8️⃣ 예제 코드 스케치 (요약)

> 세부 구현은 앞서 제공한 event-sourcing-guide 코드 참조.
> 

```java
// 1. Command API
POST /tickets             →  CreateTicketCommand
POST /tickets/{id}/reject →  RejectWorkflowCommand

// 2. CommandHandler
Ticket agg = Ticket.create(title);
store.appendAll(agg.changes());

// 3. Projection Listener
@EventListener TicketCreated → INSERT ticket_read
@EventListener WorkflowRejected → UPDATE ticket_read

// 4. Query API
GET /tickets/{id} → SELECT * FROM ticket_read WHERE id=?

```

## 9️⃣ 도입 체크리스트

1. **이력 요구사항** 높다 👉 Event Sourcing 적합.
2. **마이크로서비스**? 없다 👉 Spring Event + RDB OK.
3. 개발·운영팀 **학습 의지**? 필수.
4. **읽기 성능**: Projection 설계·인덱스 설계 예산 확보.
5. **스토리지**: 이벤트 로그 증가 대비 Snapshot 전략 준비.
