# 카프카 클러스터

```java
Producer
   |
   v
Bootstrap Broker (모든 메타데이터 인지)
   |
   v
Leader Broker (Partition 별 leader)
   |
   +--> Follower1 Broker (복제)
   |
   +--> Follower2 Broker (복제)
```

## Cluster Metadata Topic

- Kafka 내부 topic
- Partition/Leader/Follower 할당 정보 저장 및 전파
- 모든 broker는 구독(본인이 Leader가 아니여도 클러스터 상태 파악 가능)

## 비동기/논블로킹 + Flux 관점

- 클라이언트는 Leader broker에게만 write/read 요청
- Replication은 broker 내부적으로 비동기 수행 → Flux 논블로킹 흐름 유지
- Cluster Metadata Topic 기반으로 새로운 leader로 자동 failover

## 실무 적용

- Replication Factor Best Practice
    - 최소 3(Leader + 2 replica) → 1대 장애 시 데이터 보장
    - Replica sync 설정
- Grafana + Prometheus
    - Replica lag, leader election 이벤트, offline partition 시각화

## Kafka 클러스터 내 3가지 주요 통신

| 통신 종류 | 참여 주체 | 역할 |
| --- | --- | --- |
| **Control Plane** | Controller ↔ Controller | 클러스터 메타데이터 관리, leader 선출, 파티션 재할당 등 |
| **Data Plane (Internal)** | Broker ↔ Broker | 데이터 복제, follower → leader 데이터 동기화 |
| **Client Communication** | Broker ↔ Producer/Consumer (외부 애플리케이션) | 데이터 생산/소비 |

## Kafka Listener

Kafka는 여러 포트와 IP에서 동시에 대기 가능 ⇒ 통신 종류에 따라 포트와 네트워크를 분리하기 위함

| Listener Name | IP / DNS | Port | 용도 |
| --- | --- | --- | --- |
| INTERNAL_LISTENER | 10.0.0.1 | 9092 | 브로커 내부 복제, Controller ↔ Broker |
| EXTERNAL_LISTENER | kafka.mydomain.com | 29092 | 클라이언트 (Producer/Consumer)용 |
| CONTROLLER_LISTENER | 10.0.0.2 | 9093 | Controller 간 통신 |
- INTERNAL_LISTENER는 private IP를 사용해 외부 접근 차단
- EXTERNAL_LISTENER는 public IP 또는 DNS 기반으로 외부 접속

Client는 각 Kafka Broker에서 자신의 Listener 정보를 메타데이터에 등록하므로, Bootstrap Server에 접속할 때 Metadata를 받아서 Broker Listener를 알 수 있다.

```yaml
Docker Compose
 ├── Kafka1 (8081 / kafka1:9092 / 9093)
 ├── Kafka2 (8082 / kafka2:9092 / 9093)
 └── Kafka3 (8083 / kafka3:9092 / 9093)

Producer / Consumer → localhost:8081, 8082, 8083 (Host ↔ External)

Kafka Internal:
 ├── kafka1:9092 <--> kafka2:9092 <--> kafka3:9092
 ├── kafka1:9093 <--> kafka2:9093 <--> kafka3:9093
```

## 중요 개념

| 개념 | 설명 |
| --- | --- |
| **단일 파티션 + replication 3** | 파티션 하나로 데이터 순서를 철저히 유지하며 고가용성 확보 |
| **Leader failover** | ISR 내 다른 노드가 자동으로 leader로 승격 |
| **Bootstrap server 다운 영향 없음** | 초기 메타데이터 수신 후 다른 노드로 직접 연결 |
| **ISR (In-Sync Replicas)** | 리더와 동일한 데이터를 가진 노드. 리더 장애 시 ISR 중 한 노드가 리더가 됨 |
| **데이터 손실 없음** | 리더 장애, failover 시에도 소비자 데이터 연속성 유지 (ISR 내 동기화 덕분) |
