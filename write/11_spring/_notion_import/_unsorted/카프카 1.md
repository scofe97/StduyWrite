# 카프카 로컬 개발환경

Kafka는 Java + Scala기반으로 설계되어 있기 때문에 아래 순서를 따라야 합니다.

1. Java 설치
2. Kafka 바이너리 다운로드 → 압축해제 → bin 디렉토리 PATH에 추가
3. Kafka 실행 스크립트 사용 가능

## 도커 이미지

| 이미지 | 장점 | 단점 |
| --- | --- | --- |
| Confluent Kafka | 공식, 기능 풍부 | Docker 이미지 크기 큼 |
| Bitnami Kafka | 경량, 유명 | M1 Mac 지원 부족 (CPU 아키텍처 이슈) |

## 카프카 디렉토리 구조

```go
kafka_*/ 
 ├── bin/           # 실행 스크립트
 ├── config/        # 설정 파일
 ├── libs/          # 라이브러리
 └── ...
```

```bash
# 카프카 서버 실행
kafka-server-start.sh config/server.properties

# 토픽 생성
**kafka-server-start.sh config/server.properties**

# 메시지 생산
kafka-console-producer.sh --topic my-topic --bootstrap-server localhost:9092

# 메시지 소비
kafka-console-consumer.sh --topic my-topic --from-beginning --bootstrap-server localhost:9092
```

## KRaft 모드(Kafka Raft Metadata mode)

[From ZooKeeper to KRaft: How the Kafka migration works](https://strimzi.io/blog/2024/03/21/kraft-migration/)

[Apache Kafka](https://kafka.apache.org/documentation/#kraft)

[[OpenSource] Apache Kafka 이해하기 -2 : Zookeeper, KRaft(Kafka Raft), 비교](https://adjh54.tistory.com/639)

![image.png](%EC%B9%B4%ED%94%84%EC%B9%B4%201/image%203.png)

아파치 카프카의 메타데이터 관리를 위한 새로운 방식으로, ZooKeeper의 족쇄를 끊고 카프카 자체에 메타데이터 관리 기능을 내장하도록 설계되었습니다.

- 카프카 2.8.0에서 preview로 등장하였고, 3.x버전에서 production-ready로 다듬어지며, 4.x에서는 ZooKeeper 제거가 목표 입니다.

### 장점

- ZooKeeper 종속성 제거를 통해 시스템 복잡성을 줄입니다.
- Kafka 내부에 메타데이터 로그를 두어 단일 시스템으로 구성합니다.
- Raft Consensus Protocol 채택: 안정적이고 일관된 메타데이터 복제 가능

### 핵심 구조 / 동작 흐름

```bash
+----------------+          +----------------+          +----------------+
|  KRaft Broker  | <-------> |  KRaft Broker  | <-------> |  KRaft Broker  |
|  (Controller)  |           | (Follower)     |           | (Follower)     |
+----------------+          +----------------+          +----------------+
        |                           |                           |
        +-------------------------------------------------------+
                            Raft Log Replication
```

- Controller: ZooKeeper가 담당하던 메타데이터와 클러스터 상태를 관리
- Follower: Controller의 메타데이터 로그를 Raft를 통해 복제
- Log: Kafka 내부 주제로 메타데이터 이벤트를 기록, Raft 합의 기반 복제

```bash
[클러스터 초기화]
        │
        ▼
(1) 첫 KRaft Broker가 Controller로 선출됨
        │
        ▼
(2) Controller가 메타데이터 변경 관리 시작
        │
        ▼
(3) 변경 사항은 Raft Log에 기록
        │
        ▼
(4) Follower Broker가 Raft Log 복제
        │
        ▼
(5) 모든 Broker가 메타데이터 일관성 유지
```

# 카프카 기본 개념

[[OpenSource] Apache Kafka 이해하기 -1: 주요 모델 및 구성요소](https://adjh54.tistory.com/636?category=1229201)

Kafka는 이벤트 스트리밍 플랫폼 입니다. 

애플리케이션에서 발생하는 이벤트(데이터)를 실시간으로 수집하고, 안전하게 저장하며, 필요할 때 다른 애플리케이션으로 전달하는 시스템입니다.

- 대규모 사용자 이벤트(주문, 클릭, 로그인 등) 수집
- 데이터 파이프라인(데이터를 여러 시스템으로 전달)
- 실시간 데이터 처리(알림, 추천)

## 이벤트

Kafka에서 말하는 이벤트는 비즈니스에서 의미 있는 한 건의 데이터 입니다.

- 이벤트 = 메시지 = 레코드
- ex) SAM이 트윗에 “좋아요”를 눌렸다.
- ex) 사용자가 특정 링크를 클릭했다.

## 이벤트 저장 방식

Kafka는 Topic이라는 논리적 공간에 이벤트를 저장합니다. Topic은 DB의 Table과 유사합니다.

- 특정 주제나 카테고리의 데이터를 저장
- 애플리케이션의 데이터를 구분해서 보관
- 하나의 Kafka 클러스터에는 수천개의 Topic이 생성 가능하다.

```bash
user-clicks
order_events
taxi-location
```

- Topic 이름은 자유롭게 조합 가능하지만, 일괄된 규칙을 지키는게 좋습니다.

## 이벤트 흐름

![image.png](%EC%B9%B4%ED%94%84%EC%B9%B4%201/image%204.png)

```bash
[Order Service] → Kafka Topic "order-events" → [Inventory Service, Payment Service]
```

- Order Service(생산자): 주문 발생 시 Kafka에 이벤트를 보낸다.
- Kafka Topic: order-events라는 이름의 Topic에 이벤트 저장
- 소비자 서비스들: 다양한 서비스들이 Kafka에서 이벤트를 가져가서 처리한다.

# Kafka 클러스터

![image.png](%EC%B9%B4%ED%94%84%EC%B9%B4%201/image%205.png)

Kafka는 하나의 서버만으로 구성되지 않으며, 여러 개의 Kafka 서버(인스턴스)를 하나의 클러스터로 구성합니다.

| 개념 | 설명 |
| --- | --- |
| **Kafka 서버** | Kafka 소프트웨어가 구동되는 하나의 머신 (물리 or 가상) |
| **Kafka 브로커 (Broker)** | 클러스터 내에서 **데이터 저장/관리/클라이언트 통신** 을 담당하는 Kafka 서버 |
| **Kafka 클러스터 (Cluster)** | 여러 Kafka 브로커들의 집합 |
- 클러스터(한 팀)
- 브로커(팀원)

## 카프카 서버의 역할

| 용어 | 역할 | 설명 |
| --- | --- | --- |
| **Broker** | 데이터 관리자 | 토픽 데이터 저장, 클라이언트 통신 |
| **Controller** | 클러스터 관리자 | 클러스터 메타데이터, 노드 상태 관리 |
| **Leader** | 파티션 리더 | 클라이언트의 읽기/쓰기 처리, 데이터 기록 |
| **Follower** | 파티션 팔로워 | 리더 데이터 복제, 리더 장애 시 승격 |

```bash
Kafka 클러스터
   ↓
Controller 선정 (클러스터 리더)
   ↓
각 토픽 + 파티션에 대해:
   - Broker에 리더/팔로워 역할 지정
   - 데이터 저장 책임 배정
   ↓
Producer → Leader Broker에 이벤트 전송
Leader → Follower에 데이터 복제
Consumer → Leader/Follower에서 데이터 수신
```

Kafka 클러스터 내 각 브로커(서버)는 다음과 같은 기능을 수행합니다.

- 서로 통신하며 클러스터 상태를 공유
- 컨트롤러라는 리더 역할을 가진 서버가 존재
- 컨트롤러가 각 브로커에게 역할/책임을 분담
- 하나의 서버가 Broker + Controller를 동시에 수행할수도 있다.

**컨트롤러**

- 토픽 생성/삭제/변경 시 브로커에게 책임 변경
- 클러스터 상태/메타데이터 관리

**브로커 = 서버**

- 데이터 저장
- 클라이언트(Producer/Consumer)와 통신
- 토픽 데이터 관리

Kafka에서는 각 토픽의 파티션마다, 하나의 리더와 여러 팔로워가 존재한다.

```bash
Order Event topic → Broker 1: Leader, Broker 2/3: Follower
Taxi Location topic → Broker 2: Leader, Broker 1/3: Follower
```

- 리더 브로커: 데이터 기록을 담당
- 팔로워 브로커: 리더 데이터를 복제하며 백업

## 장애 발생 시 동작

컨트롤러 장애

- 클러스터 내 다른 브로커가 자동으로 컨트롤러로 승격

리더 브로커 장애

- 팔로워 브로커 중 하나가 자동으로 리더 승격
- 클라이언트는 계속 이벤트를 전송/수신 가능

# Bootstrap Server

Kafka는 수백~수천대의 브로커가 클러스터를 구성합니다. 이때 애플리케이션은 어떤 브로커에 접속해야하는가?

| 유형 | 예 | 특징 |
| --- | --- | --- |
| **Stateless** | Order Service, Web API | 어느 인스턴스가 요청을 처리해도 무관, Load Balancer 통해 분산 |
| **Stateful** | Kafka, DB | 데이터가 특정 노드에 저장 → 해당 노드와 통신해야 함 |
- Kafka는 stateful → 데이터 가진 브로커와 통신 필요
- Bootstrap Server는 여러 서버 주소를 지정해서 하나 이상 접속을 성공하도록 하는게 좋다.

## 클러스터 접속 로직

```bash
Kafka Client → Bootstrap Server 접속
        ↓
Bootstrap Server → 클러스터 메타데이터 제공
        ↓
Client → 리더 브로커 식별, 직접 통신
        ↓
Client → 이벤트 송신, 수신
```

Kafka의 각 브로커는 클러스터 내의 다른 브로커 정보를 모두 알고있으며, 메타데이터 정보를 공유합니다.

- 수천 대 브로커가 있어도, 하나의 브로커와 연결하면 전체를 알 수 있습니다.
- 첫 연결을 위한 서버를 Bootstrap Server라고 부릅니다.

### Kafka Clinet 라이브러리

- 애플리케이션이 Bootstrap Server에 최초 접속 시 해당 브로커에게 클러스터 메타데이터 요청
- 어떤 토픽의 리더 브로커가 누구인지 즉시 파악, 이후 클라이언트가 해당 리더 브로커와 직접 통신

# Producer / Consumer

![image.png](%EC%B9%B4%ED%94%84%EC%B9%B4%201/image%204.png)

## Consumer

Consumer는 Pull 기반으로 동작합니다.

| 구분 | 설명 |
| --- | --- |
| **Push 모델** | Kafka가 Consumer에게 메시지를 강제로 보냄 |
| **Pull 모델** | Consumer가 Kafka에게 *"메시지 주세요"* 요청 |

```bash
Consumer → Kafka: 메시지 요청 (poll)
Kafka → Consumer: 최대 N건의 메시지 전달
Consumer: 처리 완료 후 다시 poll 요청
```

- Producer는 초당 수천~수백만 건 메시지 전송 가능(고처리량, 저지연)
- Consumer가 같은 속도로 처리하지 못하면 부하, 장애가 발생 가능
- Consumer가 필요할 때만 요청하도록 함으로써 부하 제어

## Producer

Producer는 메시지를 받자마자 즉시 전송하지 않는다.

- 배치(batch)로 묶어 전송해 네트워크 효율성을 높입니다.

### 배치 관련 속성

| 속성 | 설명 | 기본값 |
| --- | --- | --- |
| `linger.ms` | 메시지를 얼마나 모았다가 전송할지 (밀리초) | 0 (console producer: 1000ms) |
| `batch.size` | 배치 크기 (바이트) | 16KB |

```bash
조건1: linger.ms 시간이 경과
OR
조건2: batch.size 만큼 메시지 모임
→ Kafka에 전송
```

## 메시지 직렬화/역직렬화

Kafka는 데이터를 바이트 배열 형태로 저장한다.

| 단계 | Producer | Kafka | Consumer |
| --- | --- | --- | --- |
| 데이터 입력 | String / Number / JSON / 객체 | byte[] 저장 | byte[] 제공 |
| 동작 | 직렬화 (serialize) | 저장 | 역직렬화 (deserialize) |
| 책임 | Producer Library | - | Consumer Library |

```bash
Producer:
String → 직렬화 → byte[]
Kafka:
byte[] 저장
Consumer:
byte[] → 역직렬화 → String
```

# Kafka Offset, Consumer Group

## Offset

Kafka의 토픽은 내부적으로 append-only immutalbe log 구조로 메시지를 저장합니다.

- Kafka는 메시지를 수신한 순서대로 저장하고, 각 메시지에 Offset을 부여합니다.

| 개념 | 설명 |
| --- | --- |
| **Offset** | 토픽 내 각 메시지의 고유 번호 (0부터 시작, 순차 증가) |
| **특징** | Kafka가 메시지를 받은 순서대로 오프셋 부여 |

```bash
Topic: hello-world
------------------------------
| offset:0 | offset:1 | offset:2 | ...
|  "Hi"    | "Hello"  | "Kafka"  | ...
```

Offset은 Consuer Group이 해당 Partitino에서 마지막으로 읽은 메시지 위치

- offset을 수정하는 것 또한 가능하며, 특정 스텝으로 이동할 수 있다.

## Consumer Group

여러개의 서비스들이 하나의 이벤트 수신이 필요한 경우가 있다.

- 각 서비스는 여러 인스턴스로 스케일 아웃이 될 수 있으므로, 각 서비스의 모든 인스턴스가 같은 메시지를 받으면 중복 처리가 발생할 수 있다.
- Consuer Group을 통해 논리적으로 하나의 소비자로 간주되는 Consumer 집합을 만든다. Kafka는 Group 별로 Kafka는 메시지를 1회 전달 보장한다.

```bash
Order Event Topic
       ↓
Kafka: Order Event → Inventory Group → 1개 인스턴스 전달
                    → Payment Group → 1개 인스턴스 전달
```

```bash
Order Event
  ↓
Kafka → Inventory Group → 3 인스턴스 중 1곳에만 전달
      → Payment Group → 2 인스턴스 중 1곳에만 전달
```

## Partition

Consumer Group에 2개의 Consumer가 있는 경우, Kafka는 1곳의 Consumer에만 메시지를 전달합니다.

- 나머지 Consumer는 대기 상태(idle)
- Topic내 메시지 순서를 엄격히 보장하기 떄문에, Group에서 동시에 처리하면 데이터 정합성 이슈가 발생할 수 있습니다.

하지만 순서가 필요없는 경우에는 병목이 발생한 것 처럼 보일 수 있습니다.

```bash
Order Event Topic (Partition 2개)
 → Partition 0 → Consumer A 처리
 → Partition 1 → Consumer B 처리
```

- Partition으로 메시지 단위를 나누고, 병렬 처리 가능
- Partition 별 순서 보장은 유지하며, Consumer 수에 맞게 분배가 가능해진다.

## Key

- Key 기반 해시를 통해 Partition 결정

| --- | --- |
| A1 | Partition 0 |

데이터 고유성이 높은 PK를 활용해서 Parttion이 고르게 분산되도록 해야 합니다.

## Rebalnace

Kafka는 Group내 Consumer 변동 시 Partition 할당을 재조정합니다.

```bash
[Consumer Group: inventory-group]
Initial:
  Partition 0 -> Consumer A
  Partition 1 -> Consumer A

Consumer B join → Rebalance
  Partition 0 -> Consumer A
  Partition 1 -> Consumer B

Consumer B leave → Rebalance
  Partition 0 -> Consumer A
  Partition 1 -> Consumer A
```

- Consumer Join → Rebanlance
- Consuer Leave 0> Rebalance

## Partition 확장

> 한 Consumer Group이 가질 수 있는 **최대 Consumer 수** = **Partition 수**
> 

트래픽이 증가해 기존 Partition을 증가할 때 고려해야할 상황이 있다.

- 확장 후 리밸런싱 되면서, 새로운 Partitino에 메시지가 전송되고 순서처리가 꺠질 수 있다.

| 방법 | 장단점 |
| --- | --- |
| **처음부터 Partition 수를 잘 설계** | ⚠️ 처음 설계가 중요✔️ 순서 문제 없음 |
| **일시적 순서 깨짐 허용** | ✔️ 간단⚠️ 애플리케이션에서 순서 깨짐 수용해야 함 |
| **Producer 중지 → 기존 메시지 소진 → Producer 재시작** | ✔️ 순서 유지 가능⚠️ 다운타임 발생 |
| **새 Topic 생성 + 전환** | ✔️ 무중단✔️ 순서 유지⚠️ 코드 수정 + 관리 복잡성 증가 |

## Consumer의 메시지 반복 전달 & Offset Commit

Kafka는 Consumer Group 별로 offset을 트래킹합니다.

```java
// 메시지 처리 후 offset commit
record.receiverOffset().acknowledge();
```

- consumer가 메시지를 받아 처리했다면, 처리 완료했음을 Kafka에 알려야 Kafka가 해당 메시지를 다시 주지 않습니다.
- Reactor Kafka에서는 명시적으로 commit을 하지 않으면 Kafka는 이 consumer가 메시지를 소비하지 않았다고 가정하고, 같은 메시지를 줍니다.

acknowledge한 offset까지의 메시지는 다시 전달하지 않는다.

```java
1️⃣ broker가 1, 2, 3, 4 번 메시지를 consumer에게 전달
2️⃣ consumer가 1, 2, 3은 처리했지만 acknowledge는 안함
3️⃣ 4만 acknowledge
4️⃣ 서버 다운 → 재시작
5️⃣ broker는 4까지 처리 완료된 것으로 보고 5부터 메시지 제공
```

### enable.auto.commit

- enable.auto.commit = true로 설정하면 카프카에서 주기적으로 offset을 자동 commit 합니다.
- 개발자가 직접 acknowledge()를 호출할 필요는 없지만, 메시지 처리 실패에 대해 처리하거나, 트랜잭션 및 비즈니스 보장이 힘들어 집니다.

### acknowledge 흐름

```java
receiver.receive()
    .doOnNext(record -> {
        // 비즈니스 로직 처리 (예: 결제 처리)
        log.info("Key: {}, Value: {}", record.key(), record.value());

        // 처리 완료 후 offset commit
        record.receiverOffset().acknowledge();
    })
    .subscribe();
```

### Kafka의 중복방지

Kafka는 메시지를 최소 한 번은 소비자에게 보낸다는 철칙을 지키지만, 2번이상 보낼수도 있습니다.

- 중복방지는 Consumer에서 idempotent 처리를 해야한다.
- idempotent(멱등 설계) 처리는 같은 메시지를 여러번 보내도 결과가 같아야 한다는 의미입니다.

## Backpressure

```java
// SendOptions.create()
var option = SenderOptions.<String, String>create(producerConfig)
        .maxInFlight(10_000);

// flux: 100개의 메시지를 100ms 간격으로 생성
var flux = Flux.range(1, 1_000_000)
        .map(i -> new ProducerRecord<>("order-events", i.toString(), "order-" + i))
        .map(pr -> SenderRecord.create(pr, pr.key()));
```

Backpressure는 Consumer의 처리 속도에 맞춰 Producer의 데이터 발행 속도를 조절하는 개념입니다.

- maxInFlight를 수정하면, publisher가 몇 개까지 버퍼링할지 결정됩니다.

```java
───────────── KafkaSender ─────────────
        |                      |
        | 256개의 메시지 준비     |
        ▼                      ▼
┌──────────────────────────────┐
│ KafkaSender 내부 큐           │
│ (최대 256개 in-flight 허용)   │
└──────────────────────────────┘
        │
        │ (1) 메시지 #1 ~ #256 전송
        ▼
───────────── Kafka Broker ─────────────
        │
        │ (2) 각 메시지 leader 파티션에 기록
        │
        │ (3) leader 파티션이 producer 에 ack 보냄 (각 메시지 별 개별 ack)
        ▼
───────────── KafkaSender ─────────────
        │
        │ (4) ack 받음 → in-flight count 1 감소
        │
        │ (5) 새로운 메시지 1개 추가 송신
        │
        │ (6) 나머지 ack 도 순차 도착 → 그만큼 slot 해제 → 새 메시지 추가 송신

```

| 값 | 의미 | 주의점 |
| --- | --- | --- |
| 작음 | 전송 안정성 높음, 메모리 사용 낮음 | 느림 |
| 큼 | 전송 속도 빠름 | OOM 위험 |

## Partition 동작

| 상황 | Kafka 동작 |
| --- | --- |
| 새 Consumer 추가 | Partition 재할당 (Rebalance) |
| Consumer 죽음 | Session timeout 후 Partition 재할당 |
| Producer는? | Partition 할당과 무관, 단순 메시지 전송 |
| Partition은? | 한 Partition → 한 Consumer에만 할당 |

## Assignment Strategy

| 전략 | 동작 | 장점 | 단점 |
| --- | --- | --- | --- |
| Default (Range/Sticky 등) | 새 Consumer 추가 시 **모든 Consumer Revoke 후 재할당** | 단순하고 빠름 | Consumer 전체 잠깐 중단 |
| Cooperative Sticky | 새 Consumer 추가 시 **필요한 Partition만 이동** | 서비스 중단 최소화, Rebalancing 부드러움 | 조금 더 복잡한 관리 |

# Seek Offset

Kafka Consumer가 특정 Offset부터 데이터를 읽도록 제어합니다.

```java
receiverOptions
    .addAssignListener(partitions -> {
        // Partition 할당될 때마다 콜백 실행
        partitions.forEach(partition -> {
            log.info("Assigned offset position: {}", partition.position());
            
            // 특정 offset으로 seek: (현재 offset - 2) 지점부터 시작
            long seekOffset = Math.max(0, partition.position() - 2);
            partition.seek(seekOffset);
        });
    });
```

- addAssignListener
    - Partition 할당 시점에 콜백 등록 가능
    - ReceivePartition을 통해 현재 offset, partition 정보 제공
- seek
    - 해당 Partition의 offset을 원하는 위치로 이동
