# 스프링 이벤트  동기/비동기

주제: Spring Study

# 

## 📊 1. 동기 vs 이벤트 기반 트랜잭션 흐름도 (텍스트 다이어그램)

### ✅ [1] **동기 처리 (Transactional 포함)**

모든 처리가 하나의 트랜잭션에서 실행되며, 중간에 실패하면 전부 롤백됩니다.

```
scss
복사편집
[Client 요청]
     │
     ▼
[Service: createTicketWithWorkflow]
     │
     ├── Ticket 생성 (ticketComponent.create)
     │
     ├── Workflow 생성 (workflowComponent.createInitial)
     │
     ▼
[DB 커밋]
     │
     └─> 응답 반환

```

> 💡 모든 처리 성공 시 하나의 트랜잭션으로 commit
> 
> 
> ❗ 하나라도 실패하면 전체 rollback
> 

---

### ✅ [2] **이벤트 기반 처리 (비동기, 트랜잭션 분리)**

핵심 로직(티켓)은 우선 커밋되고, 후속 로직(워크플로우)은 이벤트 리스너에서 별도로 처리됩니다.

```
scss
복사편집
[Client 요청]
     │
     ▼
[Service: createTicket]
     │
     ├── Ticket 생성 (ticketComponent.create)
     │
     ├── publishEvent(TicketCreatedEvent)
     │
     ▼
[DB 커밋] ← (여기까지만 트랜잭션)
     │
     └─> 응답 반환

(비동기)
     ▼
@Async EventListener
  handleTicketCreated
     │
     ├── workflowComponent.createInitial
     ▼
   별도 트랜잭션
```

> 💡 티켓은 반드시 생성됨, 워크플로우 생성 실패 시에도 티켓은 남음
> 
> 
> ✅ 느슨한 결합, 확장성 탁월
> 
> ❗ 워크플로우 실패는 직접 감지하고 보상처리 필요
> 

---

## 🔁 2. 보상 트랜잭션 패턴 예제 (실패 대응 및 자동 복구)

### 🎯 시나리오:

- 티켓은 정상 생성됨
- 워크플로우 생성 실패 시 **로깅 및 재시도** or **보상작업** 수행

---

### 📄 예제 코드

### 1. 이벤트 핸들러 (비동기)

```java
java
복사편집
@Async
@EventListener
public void handleTicketCreated(TicketCreatedEvent event) {
    try {
        workflowComponent.createInitial(event.ticketId());
        historyLogger.log("WORKFLOW_CREATED", event.ticketId(), "SUCCESS");
    } catch (Exception ex) {
        // 실패 이력 기록
        historyLogger.log("WORKFLOW_CREATE_FAILED", event.ticketId(), ex.getMessage());

        // 보상 처리 등록 (e.g. 재시도 큐, 관리자 알림 등)
        compensationService.enqueueCompensation(event.ticketId());
    }
}

```

### 2. 보상 처리 예시

```java
java
복사편집
@Component
@RequiredArgsConstructor
public class CompensationService {
    private final WorkflowComponent workflowComponent;

    // 단순 재시도 처리
    public void enqueueCompensation(Long ticketId) {
        // DB나 Queue에 저장해놓고 배치 혹은 트리거로 처리
    }

    @Scheduled(fixedDelay = 30000)
    public void retryFailedWorkflows() {
        List<Long> failedTicketIds = findFailedTicketIds();
        for (Long id : failedTicketIds) {
            try {
                workflowComponent.createInitial(id);
                log.info("보상 완료 - 워크플로우 생성됨: {}", id);
                markAsRecovered(id);
            } catch (Exception e) {
                log.warn("보상 실패 - 다음 주기로 재시도 예정: {}", id);
            }
        }
    }
}

```

### 🧠 보상 트랜잭션 전략 요약

| 패턴 | 설명 | 예 |
| --- | --- | --- |
| 재시도 | 실패한 작업을 나중에 다시 시도 | 워크플로우 생성 실패 → 주기적 재시도 |
| 반대작업 | 실패 시 기존 작업을 취소 | 워크플로우 실패 → 티켓 상태 롤백 |
| 관리자 개입 | 실패 이력을 관리자에게 알림 | 알림 시스템 연동 후 Slack, 이메일 등 |
| Dead Letter | 실패 이벤트를 DLQ로 보내기 | Kafka, SQS 기반 |

## 🧭 마무리: 어떤 경우에 보상을 적용할 것인가?

| 상황 | 트랜잭션 처리 방식 | 보상 필요 여부 |
| --- | --- | --- |
| 티켓과 워크플로우 모두 반드시 있어야 함 | 동기 | ❌ |
| 티켓만 필수, 워크플로우는 후속 처리 | 비동기 이벤트 | ✅ 보상 처리 필요 |
| 후속 작업은 실패 가능성 존재, 유연성 필요 | 이벤트 + DLQ or Alert | ✅ ✅ ✅ |

## ✅ A. 이벤트 스토어 연동

- `EventStore` 컴포넌트를 통해 이벤트 내역을 DB에 저장
- `EventStoreLogger`를 통해 도메인 이벤트 로깅과 아카이빙 가능

```java
@Component
public class EventStore {
    public void storeEvent(String eventType, Long refId, String detail) {
        // JDBC 기반 DB 저장 로직 포함
    }
}
```

## ✅ B. 실패 보상 전략 확장

- `AfterEventHandler` 인터페이스에 `rollback()` 메서드 추가
- 각 분기 전략별로 보상 로직을 구현 가능

```java
public interface AfterEventHandler {
    void handle(Long workflowId);
    default void rollback(Long workflowId) { }
}

```

## ✅ C. 상태머신(Spring StateMachine)

- `TicketState`, `TicketEvent`로 상태 및 전이 이벤트를 명확히 정의
- `TicketStateMachineExecutor`를 통해 상태 전이 실행

```java
stateMachine.sendEvent(TicketEvent.START);
TicketState state = stateMachine.getState().getId();
```

모든 코드는 새 문서 [`Event-store-recovery-statemachine`]에 정리해 두었습니다.

## ✅ 예시 시나리오

**워크플로우 반려 후 정책에 따라 티켓 초기화가 된 이후 → 알림 메시지를 외부 시스템에 전송해야 하는 상황**

```java
// Handler 내부에서 다음과 같이 작성했다고 가정
public class ResetHandler implements AfterEventHandler {
    public void handle(Long workflowId) {
        ticketComponent.resetByWorkflow(workflowId);

        // → 이후 작업: 알림 전송
        notificationService.sendResetNotification(workflowId); // 응답 또는 결과가 필요한 작업
    }
}

```

## 🟦 1. 동기 이벤트 처리 흐름

```
[HTTP Request: /workflow/reject]
      │
      ▼
[TicketFlowModule.rejectWorkflow()]
  ├─ workflowComponent.reject()
  └─ publishEvent(new WorkflowRejectedEvent)
                      │
                      ▼
         [@EventListener (동기)]
             onRejected(event)
                 ├─ ResetHandler.handle()
                 │     ├─ resetByWorkflow()
                 │     └─ sendResetNotification()  ← ★ 외부 연동 결과까지 대기
                 └─ auditLogger.log(...)
                      │
                      ▼
       전체 흐름 완료 후 HTTP 응답 반환

✅ 결과: 외부 연동까지 모두 완료된 후 클라이언트에 응답됨
✅ 외부 시스템 오류 발생 시 예외로 롤백 가능
❌ 느려질 수 있음 (성능 저하)

```

### ✅ 처리 방법 (동기)

- 외부 시스템 응답값을 `ResetHandler` 안에서 바로 받아서 사용
- 결과를 이용해 상태를 바꾸거나 예외 처리 가능

```java
boolean result = notificationService.send(...);
if (!result) throw new RuntimeException("알림 실패");

```

## 🟧 2. 비동기 이벤트 처리 흐름 (`@Async`)

```
[HTTP Request: /workflow/reject]
      │
      ▼
[TicketFlowModule.rejectWorkflow()]
  ├─ workflowComponent.reject()
  └─ publishEvent(new WorkflowRejectedEvent)
                      │
                      └─ (메인 트랜잭션 종료)
                             ▼
                   클라이언트에게 응답 반환 완료 ✅

           [Async Thread 시작]
                      │
                      ▼
          @Async EventListener → onRejected()
                  ├─ ResetHandler.handle()
                  │     ├─ resetByWorkflow()
                  │     └─ sendResetNotification() ← ★ 외부 연동 비동기 진행
                  └─ auditLogger.log(...)

✅ 결과: 클라이언트는 즉시 응답을 받고, 알림 전송은 백그라운드에서 진행
❌ 알림 전송 실패 여부를 응답에 반영 불가
❗ 실패 시 로그 또는 DLQ 처리 필요

```

### ✅ 처리 방법 (비동기)

- `sendResetNotification()` 결과를 응답으로 사용할 수 없음
- 대신 로그 기록 또는 실패 큐 저장 등의 방식으로 처리해야 함

```java
try {
    boolean result = notificationService.send(...);
    if (!result) {
        log.warn("알림 실패: {}", workflowId);
    }
} catch (Exception e) {
    log.error("알림 예외 발생", e);
}

```

## 🔁 동기 vs 비동기 예제 코드 비교

### 🔵 동기 핸들러

```java
@EventListener
public void onRejected(WorkflowRejectedEvent event) {
    var success = resetHandler.handleWithResult(event.workflowId());
    if (!success) {
        throw new IllegalStateException("알림 전송 실패");
    }
}

```

### 🟠 비동기 핸들러

```java
@Async
@EventListener
public void onRejected(WorkflowRejectedEvent event) {
    try {
        resetHandler.handle(event.workflowId());
    } catch (Exception e) {
        log.error("비동기 후처리 실패", e);
        // 필요시 실패 이벤트 저장
    }
}

```

## ✅ 언제 무엇을 써야 할까?

| 조건 | 동기 추천 | 비동기 추천 |
| --- | --- | --- |
| 외부 API 응답 결과가 중요 | ✔️ | ❌ |
| 트랜잭션 내에서 오류 처리 필요 | ✔️ | ❌ |
| 성능 중요 / 응답 빨라야 함 | ❌ | ✔️ |
| 알림/로깅/모니터링 등 후처리 | ❌ | ✔️ |
| 이벤트 실패 감지 및 복구 처리 예정 | ✔️ | ✔️ (로그 기반 DLQ 필요) |

## 💡 정리 요약

- **동기 이벤트**는 완전한 결과 일관성이 필요할 때 사용
- **비동기 이벤트**는 알림, 이메일, 로그 등 클라이언트와 무관한 처리를 위해 사용
- 실무에서는 보통 `주요 상태 변경은 동기`, `후처리 및 외부 연동은 비동기`로 조합합니다

---

이 흐름을 기반으로, 다음 단계로 다음을 도입할 수도 있습니다:

- `@TransactionalEventListener(phase = AFTER_COMMIT)`
- `CompletableFuture<T>` 응답을 반환하는 비동기 핸들러
- `RetryTemplate`, `DLQ`를 통한 실패 복구 처리

필요하시면 실무적인 리트라이 및 이벤트 실패 보장 설계도 제공드릴 수 있습니다. 원하시나요?

매우 훌륭한 질문입니다.

비동기 이벤트 처리에서 **예외 발생, 완료, 특정 조건**에 따라 후속 작업을 실행하고자 할 때는

단순 `@Async` + `@EventListener` 조합만으로는 부족합니다.

그래서 이 경우엔 **`CompletableFuture`** 또는 **`ExecutorService`**를 활용한 **비동기 작업 흐름 제어**를 도입하는 것이 적합합니다.

# ✅ 실무 비동기 흐름 설계: 완료 / 예외 / 조건 분기 처리

## ✨ 목표 흐름

```
비동기 작업 실행
  ├── 예외 발생 시 → 예외 핸들링 함수 실행
  ├── 완료 시 → 후처리 함수 실행
  └── 특정 결과 조건 만족 시 → 추가 로직 실행

```

---

## ✅ 예시 1: `CompletableFuture` 기반 구성

```java
@Component
@RequiredArgsConstructor
public class WorkflowEventHandler {

    private final NotificationService notificationService;

    @Async
    @EventListener
    public void onRejected(WorkflowRejectedEvent event) {
        CompletableFuture
            .supplyAsync(() -> notificationService.sendResetNotification(event.workflowId()))

            // ✅ 1. 예외 발생 시
            .exceptionally(ex -> {
                log.error("알림 전송 실패", ex);
                handleNotificationFailure(event.workflowId());
                return false; // 실패 처리
            })

            // ✅ 2. 완료 시
            .thenAccept(result -> {
                log.info("알림 전송 완료: result={}", result);
            })

            // ✅ 3. 특정 값인 경우
            .thenAccept(result -> {
                if (Boolean.TRUE.equals(result)) {
                    triggerWebhook(event.workflowId());
                }
            });
    }

    private void handleNotificationFailure(Long id) {
        // 예외 시 실행
        log.warn("워크플로우 {}에 대한 알림 실패 보상 로직 실행", id);
    }

    private void triggerWebhook(Long id) {
        // 성공 조건 만족 시 추가 동작
        log.info("웹훅 트리거: 워크플로우 {}", id);
    }
}

```

---

## 🧠 설명 요약

| 흐름 | 설명 |
| --- | --- |
| `.supplyAsync()` | 비동기 작업 시작 (`Callable`) |
| `.exceptionally()` | 예외 발생 시 처리 |
| `.thenAccept()` | 결과값을 받아 후속 로직 실행 |
| `.thenApply()` | 변환이 필요한 경우 사용 |
| `.thenCompose()` | 또 다른 `CompletableFuture` 반환할 때 사용 |

---

## ✅ 예시 2: 특정 정책 결과 분기

```java
.thenAccept(result -> {
    switch (result) {
        case "SUCCESS" -> sendSuccessMetric();
        case "TEMP_FAIL" -> retryLater();
        case "PERM_FAIL" -> alertAdmin();
    }
});

```

## 🔁 확장 아이디어

- 여러 비동기 작업을 `CompletableFuture.allOf(...)`로 병렬 처리 가능
- `ScheduledExecutorService`와 조합하여 지연 재시도
- `Result<T>` 도입하여 성공/실패 결과를 명시적으로 분리

---

## ✅ 실무 팁

| 항목 | 팁 |
| --- | --- |
| 스레드풀 사용 | `@Async`는 기본 executor에 연결됨. 용도별 executor 설정 권장 |
| 트랜잭션 경계 | 비동기 실행 시 트랜잭션이 분리됨에 주의 |
| 테스트 시 | `CompletableFuture.get()` 또는 `.join()` 사용하여 동기화 테스트 가능 |
| 실패 저장 | 실패한 워크플로우는 DB 또는 Redis에 저장하여 재처리 설계 필요 |

## 🔚 마무리

비동기 이벤트 처리에서 **완료 후 동작 / 예외 발생시 처리 / 특정 결과값 조건 분기**가 필요한 경우,

단순한 `@Async` 메서드가 아닌 **`CompletableFuture` 기반 처리**를 도입함으로써

정밀한 흐름 제어가 가능합니다.

필요하다면 다음과 같은 흐름도 제공합니다:

- ✅ 비동기 트랜잭션 보장 전략
- ✅ 비동기 작업 재시도 설계 (`RetryTemplate`, `ScheduledExecutor`)
- ✅ Result Wrapper 설계 (`Result<T>` with success/failure model)

어떤 흐름이 더 궁금하신가요?

훌륭한 선택입니다.

`@Async`와 `CompletableFuture`는 Spring Boot + Java 8~17 환경에서

**가장 실무적으로 강력한 비동기 처리 조합**입니다.

아래는 이 조합이 **가장 효과적인 유스케이스(사용 시나리오)**들을

정리한 것입니다. 각각의 예시는 실무에서 자주 등장하는 상황을 바탕으로 작성되었습니다.

# ✅ `@Async` + `CompletableFuture` 유스케이스 총정리

---

## 1. ✅ 외부 시스템 호출 (REST, GraphQL, gRPC 등)

**목표:** 사용자 요청 응답과 별도로 외부 서비스와 비동기 통신

**예시:**

- Slack 알림 전송
- 외부 시스템에 상태 업데이트
- 통합 테스트 결과 전송

```java
@Async
public CompletableFuture<Boolean> sendSlackNotification(String msg) {
    slackClient.send(msg);
    return CompletableFuture.completedFuture(true);
}

```

---

## 2. ✅ 후처리 작업 (감사 로그, 통계 업데이트, 캐시 갱신 등)

**목표:** 메인 트랜잭션 종료 후 실행

**예시:**

- 감사 로그 기록
- 사용자 활동 기록
- Redis 캐시 갱신

```java
@Async
public CompletableFuture<Void> updateAuditLog(AuditInfo info) {
    auditRepository.save(info);
    return CompletableFuture.completedFuture(null);
}

```

---

## 3. ✅ 병렬 API 호출 (멀티 조회 → 통합 응답)

**목표:** 여러 API를 병렬 호출 후 결과를 모아서 처리

**예시:**

- 사용자 기본정보 + 포인트 + 알림 설정
- 병렬 검색 (쿠팡/11번가/위메프 동시에 요청)

```java
CompletableFuture<User> userFuture = service.getUser();
CompletableFuture<Point> pointFuture = service.getPoints();
CompletableFuture<AlarmSetting> alarmFuture = service.getAlarms();

CompletableFuture.allOf(userFuture, pointFuture, alarmFuture)
    .thenApply(v -> new UserProfile(
        userFuture.join(), pointFuture.join(), alarmFuture.join()
    ));

```

## 4. ✅ 비동기 작업 흐름 제어 (후속 처리 / 예외 / 조건 분기)

**목표:** 결과에 따라 다른 작업 실행

**예시:**

- 알림 전송 성공 시 웹훅 호출
- 실패 시 관리자에게 메일 발송

```java
@Async
public CompletableFuture<Boolean> sendEmail() {
    ...
}

public void process() {
    sendEmail()
        .thenAccept(result -> {
            if (result) webhookTrigger();
            else alertAdmin();
        })
        .exceptionally(ex -> {
            log.error("이메일 실패", ex);
            return false;
        });
}

```

## 5. ✅ 리모컨 설계 (명령 큐 스타일)

**목표:** 어떤 명령을 큐에 담고 실행만 위임하는 스타일

**예시:**

- 알림 → 전송 대기 큐
- 실패 작업을 나중에 수동 처리

```java
@Async
public CompletableFuture<Void> enqueueWork(String workId) {
    queue.add(workId);
    return CompletableFuture.completedFuture(null);
}

```

## 6. ✅ 지연 작업 / 스케줄 재시도 (Retry Queue)

**목표:** 실패 시 나중에 다시 시도

**예시:**

- 이메일 전송 실패 시 5분 후 재시도
- 이벤트 누락 시 재전송

public CompletableFuture<Void> delayedRetry() {
    Thread.sleep(5000);
    return send();
}

# ✅ 유스케이스 종합 요약

| 유형 | 예시 | 처리 방식 |
| --- | --- | --- |
| 🔁 비동기 후처리 | 알림, 감사 로그 | `.thenAccept()` |
| 🚀 병렬 실행 | 멀티 API 요청 | `CompletableFuture.allOf()` |
| 🎯 조건 분기 | 알림 성공/실패 | `.thenApply()`, `.exceptionally()` |
| 🔄 재시도 | 실패 후 재시도 | `.handle()`, `.thenCompose()` |
| 🔒 응답 필요 없음 | 로그, 큐, 트리거 | `CompletableFuture<Void>` |
| 📬 응답 필요 | 결과 기반 후속 처리 | `CompletableFuture<T>` |

---

## ✳️ 실무 설계 팁

- ✅ `@Async`는 반환형이 `CompletableFuture<T>`일 때 Spring이 스레드 풀로 실행
- ✅ 반환 없는 `void`보단 항상 `CompletableFuture<Void>` 권장
- ✅ `CompletableFuture.completedFuture(...)`로 즉시 반환 가능
- ✅ 예외 처리용 `.exceptionally()` 반드시 구성
- ❗ `join()`은 예외를 래핑하므로 주의 필요 (실패 분석 어려움)

---

## 🔚 정리

> @Async + CompletableFuture는
> 
> 
> “응답과 무관한 비동기 처리”와
> 
> “병렬 처리 + 예외 흐름 제어”를 동시에 해결하는
> 
> **Spring 기반 실무에서 가장 실용적인 비동기 조합**입니다.
>

> "티켓 생성 시 워크플로우도 같이 생성해야 한다면, 워크플로우 생성을 이벤트로 처리해야 하는가?"

## 📌 핵심 결론부터

| 시나리오 | 워크플로우 생성 처리 위치 | 이유 |
| --- | --- | --- |
| 🎯 **둘 다 반드시 성공해야 하고, 하나라도 실패 시 롤백되어야 함** | `동기 메서드 호출 (같은 트랜잭션)` | 원자성 보장 |
| ⏳ **티켓 생성이 핵심이고, 워크플로우는 나중에 생성되어도 무관** | `이벤트 (비동기)` | 티켓 처리 속도 우선, 워크플로우는 나중에 처리해도 됨 |
| 💥 **워크플로우 생성이 실패할 수 있지만 티켓은 생성돼야 함** | `이벤트 (비동기 + 예외 보상)` | 워크플로우 생성 실패 감지 후 별도 보상 수행 |

---

## 🔍 각각의 경우 살펴보기

### 1. 🎯 동기 방식 (같은 트랜잭션 내 처리)

```java
@Transactional
public void createTicketWithWorkflow(TicketCreateRequest request) {
    Long ticketId = ticketComponent.create(request);
    workflowComponent.createInitial(ticketId); // 같은 트랜잭션 안에서 처리
}

```

- **장점**: 원자성 보장, rollback 쉬움
- **단점**: 성능 저하 가능성, 구조적 결합 강해짐

✅ **권장 시점**: 티켓과 워크플로우의 **도메인 결합이 강하고**, 둘 다 **항상 함께 존재해야 하는 관계**일 때

---

### 2. ⏳ 비동기 이벤트 기반 처리

```java
@Transactional
public void createTicket(TicketCreateRequest request) {
    Long ticketId = ticketComponent.create(request);
    eventPublisher.publishEvent(new TicketCreatedEvent(ticketId));
}

// 비동기 워크플로우 생성
@Async
@EventListener
public void handleTicketCreated(TicketCreatedEvent event) {
    workflowComponent.createInitial(event.ticketId());
}

```

- **장점**: 느슨한 결합, 확장성 높음 (추가 이벤트 핸들러 쉽게 붙일 수 있음)
- **단점**: 워크플로우 생성 실패 시 rollback 불가 → **보상처리 필요**

✅ **권장 시점**: 워크플로우는 **옵션이거나**, **실패 시 보상으로 처리 가능한 부가적 요소**일 때

---

### 3. 💥 비동기 + 실패 보상 처리

```java
@Async
@EventListener
public void handleTicketCreated(TicketCreatedEvent event) {
    try {
        workflowComponent.createInitial(event.ticketId());
    } catch (Exception ex) {
        historyLogger.log("WORKFLOW_CREATE_FAILED", event.ticketId(), ex.getMessage());
        // 혹은 보상 트리거 이벤트 발행
    }
}

```

- **실무에서 많이 사용**되는 패턴
- 워크플로우 생성 실패 시, 티켓은 그대로 두고 실패 이력을 남기거나, 관리자 알림

## 🧭 의사 결정 기준 요약

| 조건 | 이벤트로 처리할까? |
| --- | --- |
| 둘 다 반드시 성공해야 함 | ❌ (X) 동기로 묶어야 함 |
| 후속작업으로 처리해도 됨 | ✅ 가능 |
| 실패 시 별도 보상 로직 존재 | ✅ 가능 |
| 향후 워크플로우 생성을 다른 방식으로 바꾸고 싶음 | ✅ 느슨한 결합 필요하므로 이벤트가 유리 |

---

## 📦 도메인 관계에 따라 권장 방식

| 도메인 관계 | 워크플로우 처리 방식 |
| --- | --- |
| Ticket ↔ Workflow (1:1, 항상 동시 존재) | 동기 (같은 트랜잭션) |
| Ticket ↔ Workflow (1:0~1, 유연함) | 비동기 이벤트 |
| Ticket 생성 후 다양한 후속 처리가 예정됨 | 이벤트 기반 처리 + 전략 패턴 |

> **워크플로우가 “후속작업”이라면 이벤트로 처리**하는 것이 바람직합니다.
