# 2단계 점검 — Pipeline 핵심 질문

> 다루는 문서: 02-01.코드로 파이프라인 정의하기, 02-02.Declarative Pipeline 핵심 구조, 02-03.Pipeline 패턴, 02-04.실패 대응과 파이프라인 원칙

## Q1: Pipeline as Code가 Freestyle Job보다 나은 이유는?

Jenkinsfile은 Git 히스토리로 모든 변경의 저자·시점·이유를 추적할 수 있다. Freestyle Job은 이 기록이 없다.

- **변경 추적**: 파이프라인 변경도 PR 프로세스에 포함되므로 팀원이 위험한 설정 변경을 사전에 발견할 수 있다.
- **복구 용이**: Jenkins 서버 장애 시 Freestyle은 수동 복구가 필요하지만, Jenkinsfile은 Git에서 clone하면 즉시 복원된다.
- **파편화 제거**: 하나의 Jenkinsfile이 모든 브랜치에 적용되므로 브랜치마다 Job을 개별 관리하는 문제가 사라진다.

### 심화

Jenkinsfile을 팀에서 공유할 때 Shared Library와의 관계는 무엇인가? 공통 로직을 어떻게 중앙화하는가?



## Q2: Declarative와 Scripted Pipeline의 차이는?

| 구분 | Declarative | Scripted |
|------|-------------|----------|
| 구조 | `pipeline {}` 블록, 정해진 DSL | `node {}` 안에 순수 Groovy |
| 오류 감지 | 실행 전 문법 구조 검증 | 실행 시점에만 오류 발견 |
| 재시작 | 실패한 특정 stage부터 재시작 가능 | 전체를 처음부터 다시 실행 |
| 실무 비율 | 90%는 Declarative로 충분 | 복잡한 로직은 `script {}` 블록으로 삽입 |

전체를 Scripted로 작성할 필요는 거의 없다.

### 심화

Declarative Pipeline 안에서 `script {}` 블록을 과도하게 사용하면 사실상 Scripted Pipeline과 다를 바 없어진다. 이 경계를 어떻게 관리할 것인가?



## Q3: post actions 종류와 실무 활용 패턴은?

| 조건 | 실행 시점 | 실무 활용 |
|------|----------|----------|
| `always` | 성공·실패 무관 | 워크스페이스 정리(`cleanWs()`), 테스트 결과 수집, 아티팩트 아카이브 |
| `success` | 성공 시 | Slack 성공 알림, CD 연계 트리거 |
| `failure` | 실패 시 | @here 멘션 알림, Jira 이슈 자동 생성 |
| `changed` | 이전 결과와 달라졌을 때 | 상태 전환 알림("이전까지 성공하던 빌드가 실패하기 시작했습니다") |

실행 순서: 결과별 조건(success, failure 등) → `always` 순으로 실행된다.

### 심화

`post` 블록은 파이프라인 레벨뿐 아니라 개별 stage 레벨에서도 정의할 수 있다. 각각 어떤 상황에서 사용하는 것이 적절한가?



## Q4: when {} 조건을 Git Flow에 매핑하는 방법은?

`when { branch }` 조건으로 브랜치 유형마다 다른 동작을 구현한다:

| 브랜치 | 동작 |
|--------|------|
| `feature/*` | 빌드 + 테스트 |
| `develop` | Dev 환경 자동 배포 |
| `release/*` | Staging 배포 + 승인 |
| `main` | Production 배포 |
| `hotfix/*` | 긴급 패치 경로 |

- **`beforeAgent true`**: 에이전트 할당 전에 조건을 평가하여 비싼 리소스(GPU 노드 등)의 불필요한 점유를 방지한다.
- **`changeset "backend/**"`**: 모노레포에서 변경된 부분만 빌드하여 CI 시간과 리소스를 절약한다.

### 심화

`when { changeset }` 사용 시 첫 빌드에서는 이전 커밋이 없어 변경 파일을 감지할 수 없는 문제를 어떻게 해결하는가?



## Q5: 병렬 실행의 이점과 주의사항은?

독립적인 작업(단위 테스트, 통합 테스트, 보안 스캔, 린트)을 동시에 실행하면 전체 소요 시간이 가장 오래 걸리는 작업의 시간으로 수렴한다.

주의사항:

- **워크스페이스 충돌**: 병렬 스테이지가 동일한 워크스페이스를 공유하면 파일 쓰기 레이스 컨디션이 발생한다. 별도 `agent`를 지정하거나 `dir()` 블록으로 작업 디렉토리를 분리해야 한다.
- **리소스 점유**: 4개의 병렬 스테이지는 에이전트 풀에서 4개의 슬롯을 동시에 점유한다. 에이전트 수가 제한된 환경에서는 다른 Job의 실행이 지연될 수 있다.
- **`failFast true`**: 하나의 실패 시 나머지를 즉시 중단한다. 기본값(`false`)은 모든 결과를 한 번에 확인할 수 있다.

### 심화

병렬 테스트의 결과를 하나의 리포트로 합치려면 어떤 접근이 필요한가?



## Q6: 수동 승인 게이트 구현 시 에이전트 점유 문제는?

`input` 스텝을 사용하여 파이프라인을 일시 중지하고 지정된 사용자의 승인을 기다린다.

- **핵심 함정**: `input`이 `steps` 블록 안에 있으면 대기 시간 동안 에이전트가 점유되어 리소스가 낭비된다.
- **해결책**: 해당 stage에 `agent none`을 지정하여 에이전트를 해제한다.
- **타임아웃 필수**: 승인자가 부재중일 때 무한 대기하지 않도록 `timeout(time: 2, unit: 'HOURS')`를 설정해야 한다.
- **감사 기록**: `parameters` 옵션으로 승인 사유를 입력받으면 누가 언제 왜 배포를 승인했는지 기록이 남아 감사(audit) 요구사항을 충족한다.

### 심화

승인 게이트 없이 완전 자동화된 프로덕션 배포(Continuous Deployment)로 전환하려면 어떤 전제 조건이 필요한가?



## Q7: Multibranch Pipeline이 일반 Pipeline Job과 다른 점은?

| 구분 | 일반 Pipeline Job | Multibranch Pipeline |
|------|-----------------|----------------------|
| Job 생성 | 하나의 브랜치에 대해 수동 생성 | Jenkinsfile이 있는 모든 브랜치를 자동 탐지·생성 |
| 브랜치 삭제 시 | 수동 정리 필요 | Orphaned Item Strategy에 따라 자동 정리 |
| PR 빌드 | 직접 설정 필요 | GitHub/GitLab Branch Source Plugin으로 PR을 머지 상태로 자동 빌드 |

Branch Source Plugin이 주기적으로 SCM을 스캔하여 `Jenkinsfile`이 존재하는 모든 브랜치를 탐지하고, 각 브랜치마다 독립적인 하위 Job을 자동으로 생성한다.
