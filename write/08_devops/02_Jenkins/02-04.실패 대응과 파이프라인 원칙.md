# Pipeline 제어와 Best Practices

---

> 실패 대응, 라이브러리 분리, 파이프라인 작성 원칙을 다룬다.

## retry, timeout, input

> 파이프라인 제어 블록 세 가지는 각각 다른 문제를 해결한다. 
>
> - `retry`는 일시적 실패를 재시도로 극복하고, `timeout`은 무한 대기를 차단하며, `input`은 자동화와 수동 검증의 접점을 만든다.

| 블록 | 위치 | 용도 | 주의점 |
|------|------|------|--------|
| `retry(n)` | `steps {}` 내부 | 일시적 네트워크 오류, 외부 서비스 불안정 재시도 | `timeout` 없이 쓰면 한 번 시도가 무한 대기 가능 |
| `timeout(time, unit)` | `steps {}` 또는 `stage {}` | 무한 대기 방지, 전체 실행 시간 상한 | 단위 기본값 없음, `unit` 명시 필수 |
| `input` | `stage {}` 레벨 권장 | 수동 승인, 배포 전 인간 검증 | `agent none` 없이 쓰면 에이전트 점유 |

`retry`와 `timeout`은 반드시 함께 사용해야 한다. `retry(3)` 안에서 외부 API를 호출하는데 타임아웃이 없으면, 한 번의 시도가 무한히 걸릴 수 있어 retry가 의미 없어진다. 안전한 조합은 전체 재시도에 상한을 두는 `timeout` 안에 `retry`를 넣는 것이다.

```groovy
stages {
    stage('Deploy') {
        steps {
            timeout(time: 10, unit: 'MINUTES') {
                retry(3) {
                    sh 'curl --max-time 30 -f http://deploy-api/deploy'
                }
            }
        }
    }
    stage('Production Approval') {
        agent none
        steps {
            timeout(time: 2, unit: 'HOURS') {
                input(
                    message: '프로덕션 배포를 승인하시겠습니까?',
                    ok: '배포 승인',
                    submitter: 'devops-team,tech-lead'
                )
            }
        }
    }
}
```

- `input` 스테이지에 `agent none`을 지정하는 이유는 대기 시간 동안 에이전트 슬롯을 점유하지 않기 위해서다. 
- `agent none` 없이 `steps {}` 안에 `input`을 넣으면, 승인 대기 시간(수 시간이 될 수 있다) 동안 빌드 에이전트가 아무것도 하지 않으면서 다른 빌드를 블로킹한다.



## Shared Library 개요

> 마이크로서비스 100개가 각각 Jenkinsfile을 가지고 있다면, Docker 빌드 단계나 Slack 알림 코드가 100개 파일에 중복된다. 
>
> - 하나를 수정하면 100개를 모두 업데이트해야 한다. **Shared Library**는 이 중복을 해결하는 Jenkins의 공유 코드 메커니즘이다.

Shared Library는 별도의 Git 저장소로 관리하며, 디렉토리 구조가 정해져 있다.

```
jenkins-shared-library/
├── vars/               # 전역 함수 (Jenkinsfile에서 직접 호출)
│   ├── buildDocker.groovy
│   └── notifySlack.groovy
├── src/                # 클래스 (vars보다 복잡한 로직)
│   └── com/example/
│       └── PipelineUtils.groovy
└── resources/          # 설정 파일, 스크립트 리소스
    └── deploy-template.sh
```

Jenkins 시스템 설정(Manage Jenkins → Configure System → Global Pipeline Libraries)에 저장소를 등록하면, Jenkinsfile에서 `@Library`로 불러올 수 있다.

```groovy
@Library('jenkins-shared-library') _

pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                buildDocker(image: 'myapp', tag: BUILD_NUMBER)
            }
        }
    }
    post {
        failure {
            notifySlack(channel: '#alerts', result: 'FAILURE')
        }
    }
}
```

- Shared Library를 도입하면 개별 Jenkinsfile이 비즈니스 로직(무엇을 빌드하고 어디에 배포하는가)에만 집중할 수 있다. 공통 인프라 코드(어떻게 빌드하고 알리는가)는 라이브러리에서 한 번만 관리한다. 상세 구현과 vars/ 작성법은 08 시리즈에서 다룬다.



## Docker with Pipeline 개요

> `agent { docker }` 패턴은 빌드 환경을 컨테이너로 격리한다. Agent 머신에 Java, Node.js, Go 등을 직접 설치하지 않아도 되고, 빌드마다 다른 버전의 도구를 사용할 수 있다.

```groovy
pipeline {
    agent {
        docker {
            image 'maven:3.9-eclipse-temurin-17'
            args '-v $HOME/.m2:/root/.m2'   // 의존성 캐시 마운트
        }
    }
    stages {
        stage('Build') {
            steps { sh 'mvn clean package' }
        }
    }
}
```

- 이 패턴의 핵심 이점은 재현성이다. 팀원 모두가 동일한 이미지를 사용하므로 "내 로컬에서는 됐는데 CI에서는 안 된다"는 상황이 줄어든다. 
- Docker in Docker 구성이나 멀티 컨테이너 Agent 패턴은 04 시리즈에서 상세히 다룬다.



## Jenkinsfile Best Practices

기존 파이프라인 문서에 산재된 실수 패턴을 통합 정리한다.

| Practice | Bad | Good |
|----------|-----|------|
| **시크릿 관리** | `PASSWORD = 'hardcoded123'` | `credentials('my-secret')` 사용 |
| **타임아웃** | `retry(3) { sh 'curl ...' }` | `timeout` 안에 `retry` 중첩 |
| **에이전트 점유** | `steps { input '승인?' }` | `agent none` + `input` 조합 |
| **병렬화** | 테스트 3종을 순차 실행 | 독립적인 테스트는 `parallel {}` |
| **스테이지 이름** | `stage('s1')` | `stage('Unit Test')` (의미 있는 이름) |
| **워크스페이스** | post 없이 종료 | `post { always { cleanWs() } }` |
| **빌드 보존** | 기본값 (무한 보존) | `buildDiscarder(logRotator(numToKeepStr: '10'))` |
| **동시 실행** | 같은 브랜치 빌드 중복 허용 | `disableConcurrentBuilds()` 설정 |
| **조건부 실행** | `if (env.BRANCH_NAME == 'main')` in steps | `when { branch 'main' }` 선언적 사용 |
| **변수 선언** | `steps {}` 안에서 `script {}` 남발 | `environment {}` 블록에서 선언 |

- 이 중 타임아웃과 워크스페이스 정리는 누락되기 가장 쉬우면서 영향이 크다. 
- 타임아웃이 없으면 외부 서비스 응답 지연 시 빌드가 무기한 대기하고, 워크스페이스를 정리하지 않으면 에이전트 디스크가 가득 차서 갑작스럽게 빌드가 실패한다. 
- 새 Jenkinsfile을 작성할 때는 `options { timeout ... }` 과 `post { always { cleanWs() } }`를 먼저 선언하는 것을 습관으로 삼는 것이 좋다.

파이프라인이 길어질수록 `script {}` 블록이 늘어나는 경향이 있다. 같은 `script {}` 로직이 두 개 이상의 Jenkinsfile에 나타나기 시작하면 Shared Library로 추출할 타이밍이다. Declarative Pipeline의 가독성을 유지하는 선에서 `script {}` 블록은 최소화한다.
