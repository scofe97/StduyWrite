# 젠킨스 인증 API 스펙 (ID-Password + Crumb)
---
> 이 문서는 Jenkins 인증 API 자체를 설명하는 스펙 문서다.
>
> - 현재 검증 환경인 `ID/Password + crumb + cookie` 기준으로 정리한다.
> - TPS 패턴, 현대화 이론, 코드 구조는 `05-02a`에서 별도로 다룬다.



## 1. 이 문서에서 검증한 환경

>  실제 검증 기준은 다음과 같다
>
> - Jenkins URL: `https://jenkins.bok.trombone.okestro.cloud`
> - 인증 방식: 사용자 ID/비밀번호 Basic Auth
> - Jenkins 버전: `2.462.3`
> - CSRF 설정: `useCrumbs: true`

루트 API 확인 요청은 다음과 같다:

```bash
curl -k -sSI -u "${JENKINS_USER}:${JENKINS_PASS}" \
  "${JENKINS_URL}/api/json"
```

확인한 핵심 응답 헤더는 다음과 같다:

```http
HTTP/2 200
x-jenkins: 2.462.3
x-jenkins-session: bf2aa909
content-type: application/json;charset=utf-8
server: istio-envoy
```

`-k`는 현재 서버가 자체 서명 인증서를 사용하므로 붙였다. 신뢰된 인증서를 사용한다면 제거한다.

### 1-1. 다루는 인증 API

> 이 문서는 인증에 직접 필요한 두 API만 다룬다.

| 메서드 | 경로 | 목적 | 성공 상태 코드 |
|------|------|------|---------------|
| GET | `/api/json` | Basic Auth 확인, Jenkins 루트 정보 확인 | `200 OK` |
| GET | `/crumbIssuer/api/json` | CSRF crumb과 세션 cookie 확보 | `200 OK` |

이 두 API로 확인하는 것은 다음과 같다:

- 계정이 Jenkins에서 유효한지
- 현재 Jenkins가 crumb 검증을 사용하는지
- 상태 변경 POST 전에 어떤 헤더와 cookie가 필요한지

빌드 실행, 큐 조회, 상태 추적은 이 문서 범위가 아니다. 그 내용은 별도 문서에서 다룬다:

- `05-04. 젠킨스 빌드 실행·큐 API 스펙`
- `05-05. 젠킨스 빌드 상태 추적 API 스펙`
- `05-02a. 젠킨스 인증 모델과 TPS 패턴 (2.222+)`

### 1-2. 실습 준비

문서 전체에서 같은 환경 변수를 재사용한다. 다른 Jenkins 환경에서도 값만 바꾸면 그대로 테스트할 수 있다.

macOS/Linux용 예시는 다음과 같다:

```bash
# 회사 환경
export JENKINS_URL='https://jenkins.bok.trombone.okestro.cloud'
export JENKINS_USER='admin'
export JENKINS_PASS='cloud1234'

# K8s Jenkins:
export JENKINS_URL='http://34.47.74.0:31080'
export JENKINS_USER='admin'
export JENKINS_PASS='admin'

# Docker Jenkins-2:
export JENKINS_URL='http://34.22.78.240:29080'
export JENKINS_USER='admin'
export JENKINS_PASS='admin'
```

Windows PowerShell용 예시는 다음과 같다:

```powershell
$env:JENKINS_URL = 'https://jenkins.bok.trombone.okestro.cloud'
$env:JENKINS_USER = '<YOUR_JENKINS_USER>'
$env:JENKINS_PASS = '<YOUR_JENKINS_PASSWORD>'
```

환경 변수를 쓰지 않고 1회성으로 확인하려면 다음처럼 직접 넣어도 된다:

```bash
curl -k -sSI -u '<YOUR_JENKINS_USER>:<YOUR_JENKINS_PASSWORD>' \
  'https://jenkins.example.com/api/json'
```

문제가 생기면 먼저 아래 값을 확인한다:

```bash
echo "$JENKINS_URL"
echo "$JENKINS_USER"
printf 'PASS_LEN=%s\n' "${#JENKINS_PASS}"
```

응답 전체를 확인하고 싶다면 다음 형식을 사용한다:

```bash
curl -k -sS -D headers.txt -o body.txt \
  -u "${JENKINS_USER}:${JENKINS_PASS}" \
  "${JENKINS_URL}/api/json"
```

판단 기준은 다음과 같다:

- `401 Unauthorized`면 계정 또는 비밀번호를 다시 확인한다.
- `curl: (3)`이면 `JENKINS_URL` 값이 비어 있거나 잘못됐다.
- `content-type: application/json`이면 JSON 파이프 처리로 넘어간다.

이 문서에서는 다음 파일명을 기준으로 설명한다:

- `headers.txt`: 응답 헤더 저장 파일
- `body.txt`: 응답 본문 저장 파일
- `crumb.json`: crumb API 응답 본문
- `cookies.txt`: crumb 발급 시 저장한 세션 cookie 파일



## 2. API 스펙

> 인증 API를 읽을 때는 아래 세 가지를 나눠서 보면 된다.
>
> 1. 요청 시 어떤 헤더가 필요한지
> 2. 응답에서 어떤 필드를 읽어야 하는지
> 3. 실패 시 어떤 상태 코드를 먼저 해석해야 하는지

### 2-1. `GET /api/json`

Jenkins 루트 정보를 조회하는 API다. 인증 문맥에서는 "Basic Auth가 통과하는지"와 "crumb 정책이 켜져 있는지"를 확인하는 용도로 쓴다.

요청 형식은 다음과 같다:

```http
GET /api/json HTTP/1.1
Authorization: Basic <base64(username:password)>
Accept: application/json
```

요청 헤더는 다음과 같다:

| 헤더 | 필수 | 값 예시 | 설명 |
|------|------|---------|------|
| `Authorization` | 필수 | `Basic <base64(username:password)>` | Basic Auth 인증 정보 |
| `Accept` | 선택 | `application/json` | JSON 응답 기대 형식 |

`curl` 예시는 다음과 같다:

```bash
curl -k -sS -u "${JENKINS_USER}:${JENKINS_PASS}" \
  "${JENKINS_URL}/api/json"
```

실제 검증 시 자주 보는 응답 필드는 다음과 같다:

| 필드 | 타입 | 필수 여부 | 의미 |
|------|------|-----------|------|
| `mode` | String | 보통 있음 | Jenkins 동작 모드 |
| `nodeName` | String | 보통 있음 | 현재 노드 이름 |
| `useCrumbs` | Boolean | 중요 | CSRF crumb 사용 여부 |
| `primaryView` | Object | 보통 있음 | 기본 View 정보 |
| `jobs` | Array | 보통 있음 | 루트에 노출된 Job 목록 |

실제 응답 예시는 다음과 같다:

```json
{
  "mode": "NORMAL",
  "nodeName": "",
  "useCrumbs": true,
  "primaryView": {
    "name": "all"
  },
  "jobs": []
}
```

실제 테스트에서는 아래처럼 필요한 필드만 추려서 확인했다:

```bash
curl -k -sS -u "${JENKINS_USER}:${JENKINS_PASS}" \
  "${JENKINS_URL}/api/json" \
  | jq '{
      mode,
      nodeName,
      useCrumbs: .useCrumbs,
      primaryView: .primaryView.name,
      numJobs: (.jobs | length),
      jobs: [.jobs[]? | {name, url, color, _class}]
    }'
```

실제 응답은 다음과 같았다:

```json
{
  "mode": "NORMAL",
  "nodeName": "",
  "useCrumbs": true,
  "primaryView": "all",
  "numJobs": 3,
  "jobs": [
    {
      "name": "test",
      "url": "http://jenkins.bok.trombone.okestro.cloud/job/test/",
      "color": "blue",
      "_class": "org.jenkinsci.plugins.workflow.job.WorkflowJob"
    }
  ]
}
```

해석 포인트는 다음과 같다:

- `200`이면 Basic Auth는 통과했다.
- `useCrumbs: true`이면 비밀번호 기반 POST 전에 crumb 흐름을 준비해야 한다.
- 이 API는 세션 cookie를 별도로 저장하지 않아도 된다.
- `jobs[].url`은 Jenkins가 인식하는 절대 URL이다. 프록시나 Jenkins Location 설정에 따라 요청한 도메인과 다를 수 있다.
- `jobs[].color`는 최근 상태 색상이다. 예를 들어 `blue`는 성공, `red`는 실패, `blue_anime`는 실행 중을 뜻한다.

요청한 도메인이 같은 Jenkins인지 더 확인하고 싶다면 응답 헤더도 같이 본다:

```bash
curl -k -sSI -u "${JENKINS_USER}:${JENKINS_PASS}" \
  "${JENKINS_URL}/api/json"
```

확인 포인트는 다음과 같다:

| 항목 | 의미 | 비고 |
|------|------|------|
| `jobs[].url` | Jenkins가 생성하는 절대 URL | 설정된 Jenkins URL 추정 가능 |
| `X-Jenkins` | Jenkins 버전 | 같은 인스턴스의 증거로는 약함 |
| `X-Instance-Identity` | Jenkins 인스턴스 고유 공개키 | 있으면 같은 Jenkins 식별에 가장 유용 |

- 현재 검증 서버에서는 `X-Jenkins: 2.462.3` 헤더는 확인됐지만 `X-Instance-Identity` 헤더는 노출되지 않았다. 
- 따라서 이 환경에서는 `jobs[].url`과 응답 내용으로 추정은 가능하지만, 헤더만으로 동일 인스턴스를 강하게 단정할 수는 없다.

에러 케이스는 다음과 같다:

| 상태 코드 | 원인 | 확인 포인트 |
|-----------|------|-------------|
| `401` | 계정 또는 비밀번호 불일치 | `Authorization` 값 확인 |
| `403` | 권한 제한 | Jenkins 사용자 권한 확인 |
| `404` | 경로 오타 | `/api/json` 경로 확인 |
| `500` | Jenkins 내부 오류 | Jenkins 서버 로그 확인 |

### 2-2. `GET /crumbIssuer/api/json`

CSRF crumb을 발급하는 API다. 비밀번호 인증 환경에서 상태 변경 POST를 보내기 전에 이 API를 호출한다.

요청 형식은 다음과 같다:

```http
GET /crumbIssuer/api/json HTTP/1.1
Authorization: Basic <base64(username:password)>
Accept: application/json
```

요청 헤더는 다음과 같다:

| 헤더 | 필수 | 값 예시 | 설명 |
|------|------|---------|------|
| `Authorization` | 필수 | `Basic <base64(username:password)>` | Basic Auth 인증 정보 |
| `Accept` | 선택 | `application/json` | JSON 응답 기대 형식 |

`curl` 예시는 다음과 같다:

```bash
curl -k -sS -c cookies.txt \
  -u "${JENKINS_USER}:${JENKINS_PASS}" \
  "${JENKINS_URL}/crumbIssuer/api/json" \
  -o crumb.json
```

핵심 응답 필드는 다음과 같다:

| 필드 | 타입 | 필수 여부 | 의미 |
|------|------|-----------|------|
| `_class` | String | 보통 있음 | Jenkins crumb issuer 구현체 |
| `crumb` | String | 필수 | POST 요청에 넣을 CSRF 토큰 |
| `crumbRequestField` | String | 필수 | crumb을 담아야 하는 헤더 이름 |

실제 응답 예시는 다음과 같다:

```json
{
  "_class": "hudson.security.csrf.DefaultCrumbIssuer",
  "crumb": "95eb3e349d233bb4769df9743f7539c1fef9853c9fa925cd70b6674f4a109163",
  "crumbRequestField": "Jenkins-Crumb"
}
```

해석 포인트는 다음과 같다:

- `crumb` 값은 이후 POST 요청 헤더에 넣는다.
- `crumbRequestField`는 헤더 키 이름이다. 보통 `Jenkins-Crumb`다.
- `-c cookies.txt`를 같이 써서 세션 cookie도 저장해야 한다.

응답 외에 함께 확보해야 하는 값은 다음과 같다:

| 항목 | 저장 위치 | 용도 |
|------|-----------|------|
| crumb 값 | `crumb.json` | POST 헤더에 전송 |
| crumb 헤더 이름 | `crumb.json` | 헤더 키 결정 |
| session cookie | `cookies.txt` | 같은 세션 문맥 유지 |

PowerShell 예시는 다음과 같다:

```powershell
curl.exe -k -sS -c cookies.txt `
  -u "${env:JENKINS_USER}:${env:JENKINS_PASS}" `
  "$env:JENKINS_URL/crumbIssuer/api/json" `
  -o crumb.json

$crumb = Get-Content .\crumb.json | ConvertFrom-Json
$crumb
```

에러 케이스는 다음과 같다:

| 상태 코드 | 원인 | 확인 포인트 |
|-----------|------|-------------|
| `401` | 계정 또는 비밀번호 불일치 | Basic Auth 값 확인 |
| `403` | crumb issuer 접근 권한 제한 | Jenkins 보안 정책 확인 |
| `404` | crumb issuer 비활성화 또는 경로 문제 | `/crumbIssuer/api/json` 경로 확인 |
| `500` | Jenkins 내부 오류 | Jenkins 서버 로그 확인 |



## 3. 인증 절차 요약

현재 환경에서 인증 확인 절차는 두 단계다.

1. `GET /api/json`으로 Basic Auth가 통과하는지 확인한다.
2. `GET /crumbIssuer/api/json`으로 crumb과 cookie를 확보한다.

그다음 상태 변경 POST 요청을 보낼 때는 다음 세 가지를 함께 보낸다:

1. `Authorization: Basic ...`

2.  `crumbRequestField`가 가리키는 crumb 헤더

3. crumb 발급 시 저장한 session cookie

실제 POST endpoint 적용 예시는 이 문서 범위 밖이며 `05-04`에서 이어서 다룬다.

현재 환경 기준으로 인증 재료를 정리하면 다음과 같다:

| 항목 | GET `/api/json` | GET `/crumbIssuer/api/json` | 이후 상태 변경 POST |
|------|------------------|-----------------------------|----------------------|
| Basic Auth | 필요 | 필요 | 필요 |
| crumb 헤더 | 불필요 | 불필요 | 필요 |
| session cookie | 불필요 | 저장 필요 | 필요 |



## 4. 에러 응답

인증 단계에서 자주 보는 상태 코드는 다음과 같다:

| 상태 코드 | 의미 | 대응 |
|-----------|------|------|
| `401` | 인증 실패 | 계정, 비밀번호, 토큰 확인 |
| `403` | crumb 누락 또는 권한 부족 | 응답 본문 메시지 확인 |
| `404` | 경로 오류 | URL 확인 |
| `500` | Jenkins 내부 오류 | Jenkins 서버 로그 확인 |

`403`은 특히 두 종류를 구분해야 한다:

- crumb 관련: `No valid crumb was included in the request`
- 권한 관련: `Access Denied`, `anonymous is missing the`



## 5. API 응답의 URL은 어디서 결정되는가

> Jenkins API 응답의 `url` 필드(예: `task.url`, `executable.url`)는 클라이언트가 보낸 Host가 아니라, Jenkins 내부 설정값을 따른다.

### 5-1. getRootUrl() 우선순위

Jenkins 소스 코드(`Jenkins.java`)에서 모든 절대 URL은 `getRootUrl()` 메서드로 생성된다. 이 메서드는 다음 우선순위로 base URL을 결정한다:

| 우선순위 | 소스 | 설정 위치 |
|---------|------|----------|
| 1순위 | `JenkinsLocationConfiguration.getUrl()` | Manage Jenkins → System → Jenkins Location → Jenkins URL |
| 2순위 (fallback) | HTTP 요청에서 추론 | `X-Forwarded-Proto`/`X-Forwarded-Host` 헤더 또는 요청의 scheme/host |

**1순위 설정이 있으면 클라이언트의 Host 헤더는 무시된다.** 예를 들어 Jenkins URL이 `https://domain-b.com`으로 설정된 상태에서 클라이언트가 `https://domain-a.com`으로 접속해도, JSON 응답의 `url` 필드에는 `domain-b.com`이 나온다.

### 5-2. 현재 Jenkins URL 확인

Jenkins에 설정된 URL은 다음 API로 조회할 수 있다:

```bash
curl -k -sS -u "${JENKINS_USER}:${JENKINS_PASS}" \
  "${JENKINS_URL}/api/json" \
  | jq '{
      url: .primaryView.url,
      useCrumbs: .useCrumbs,
      nodeDescription: .nodeDescription
    }'
```

`primaryView.url`의 도메인이 Jenkins에 설정된 base URL이다.

Jenkins 버전과 기본 정보도 HTTP 헤더로 확인할 수 있다:

```bash
curl -k -sSI -u "${JENKINS_USER}:${JENKINS_PASS}" \
  "${JENKINS_URL}/api/json"
```

```text
HTTP/2 200
x-jenkins: 2.462.3
x-jenkins-session: bf2aa909
content-type: application/json;charset=utf-8
```

| 헤더 | 의미 |
|------|------|
| `x-jenkins` | Jenkins 버전 |
| `x-jenkins-session` | controller 세션 ID (재시작하면 바뀜) |

### 5-3. Location 헤더 vs JSON url 필드

빌드 트리거(`POST /build`) 응답의 `Location` 헤더와 JSON 응답의 `url` 필드는 둘 다 `getRootUrl()`을 사용하지만, 실무에서 차이가 있다:

| 구분 | 소스 | reverse proxy 영향 |
|------|------|-------------------|
| `Location` 헤더 | `getRootUrl()` | proxy rewrite 규칙이 적용될 수 있음 |
| JSON `url` 필드 | `getRootUrl()` | proxy를 **통과해도 그대로** 전달됨 |

즉 reverse proxy가 `Location` 헤더를 고쳐주더라도, JSON 안의 `url` 필드는 Jenkins 설정값 그대로 나온다. Jenkins URL 설정이 잘못되어 있으면 `Location`은 맞지만 JSON `url`은 틀린 상황이 생길 수 있다.

### 5-4. 실무 체크포인트

API 응답의 URL이 예상과 다르면 다음 순서로 확인한다:

1. `Manage Jenkins → System → Jenkins Location → Jenkins URL` 값 확인
2. 이 값이 클라이언트가 접속하는 도메인과 일치하는지 확인
3. 일치하지 않으면 이 설정을 수정 — JSON `url` 필드를 바꾸는 유일한 방법이다
