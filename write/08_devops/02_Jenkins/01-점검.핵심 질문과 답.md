## Q1: Controller와 Agent를 분리해야 하는 이유는?

Controller에서 빌드를 직접 실행하면 세 가지 문제가 발생한다:

- **리소스 경쟁**: 빌드 프로세스가 Controller JVM과 메모리를 공유하므로, 대규모 빌드가 Controller를 OOM으로 죽일 수 있다.
- **보안 위험**: 빌드 스크립트가 `JENKINS_HOME/secrets/`에 접근하여 모든 크레덴셜을 탈취할 수 있다.
- **확장성 한계**: 단일 서버의 수직 확장 한계를 Agent 수평 확장으로 극복한다. Kubernetes에서는 빌드마다 Pod를 동적 생성·삭제하여 리소스를 효율적으로 사용한다.

## Q2: Controller Executor를 0으로 설정하면 어떻게 되는가?

빌드를 실행할 Executor가 없으므로, Agent가 연결되어 있지 않으면 모든 빌드가 Queue에서 무한 대기한다.

- Jenkins는 기본적으로 Controller Executor를 **2개**로 설정한다. 그래서 Agent 없이도 빌드가 돌아간다.
- 운영 환경에서 보안을 위해 0으로 바꾸는 순간, **반드시 Agent를 먼저 연결**해야 한다.
- Controller Executor가 0이어도 Pipeline은 **시작된다**. Flyweight 실행(구조 해석)은 Executor를 소비하지 않기 때문이다. 실제 `sh` step에 도달하면 그때 Agent Executor가 필요하다.

| Controller Executor | Agent 연결 | 결과 |
|---------------------|-----------|------|
| 2 (기본값) | 없음 | Controller에서 직접 실행 (보안 위험) |
| 0 (권장) | 있음 | Agent에서 실행 (안전) |
| 0 (권장) | 없음 | Queue 무한 대기 |



## Q3: Queue에 있는 빌드와 실행 중인 빌드를 어떻게 구분하는가?

Jenkins UI에서 "실행 중"으로 보여도 실제로는 아직 Queue 대기 상태일 수 있다. 정확한 구분은 API로 해야 한다:

- **Queue 상태**: `/queue/api/json` — `why` 필드가 대기 이유를 알려준다.
- **Build 상태**: `/job/.../42/api/json` — `building=true`면 실제 Agent에서 실행 중이다.
- 빌드를 트리거한 직후 받는 값은 **build number가 아니라 queue item ID**다. `executable.number`가 나타날 때까지 queue item을 폴링해야 실제 build number를 얻는다.

## Q4: `agent any`, `agent { label }`, `agent none`의 차이는?

| 지시문 | 동작 | 사용 시나리오 |
|--------|------|-------------|
| `agent any` | 아무 Agent에서 실행 | 환경 무관한 간단한 빌드 |
| `agent { label '...' }` | 라벨 조건 일치 Agent에서만 | 특정 도구/환경이 필요한 빌드 |
| `agent none` | Pipeline 레벨에서 지정 안 함 | stage마다 다른 Agent 사용 시 |

- `agent none`을 쓰면 Controller는 Flyweight(오케스트레이션)만 실행하고, 실제 빌드는 각 stage에서 지정한 Agent가 처리한다.
- Label 조건이 맞는 Agent가 없으면 빌드는 Queue에서 무한 대기하므로, Agent 라벨 관리가 중요하다.



## Q5: Flyweight 실행과 Heavyweight 실행의 차이는?

Pipeline의 실행에는 두 레벨이 공존한다:

- **Flyweight**: `pipeline {}`, `stage()`, `when {}`, `input`, `parallel`, 변수 할당 — Controller에서 실행, Executor를 소비하지 않는다.
- **Heavyweight**: `sh`, `bat`, `git checkout`, `docker.build` — Agent의 Executor에서 실행, Executor 1개를 점유한다.

이 구분을 모르면 "Controller Executor를 0으로 설정했는데 왜 Pipeline이 시작되는가?"를 설명할 수 없다. Flyweight 코드가 과도하면 Controller 메모리를 잡아먹으므로, 복잡한 로직은 `sh` step 안에서 셸 스크립트로 처리하는 것이 좋다.

## Q6: JENKINS_HOME은 어디에 존재하고, 무엇을 백업해야 하는가?

`JENKINS_HOME`은 **Controller에만 존재**한다. Agent에는 `workspace/`만 생기고, 잡 정의·크레덴셜·빌드 기록은 모두 Controller에 저장된다.

필수 백업 대상:

- `config.xml` — 전역 시스템 설정. 잃으면 처음부터 수동 재구성해야 한다.
- `jobs/` — 잡 정의 + 빌드 히스토리. 잡 정의를 잃으면 모든 파이프라인을 재생성해야 한다.
- `secrets/` — `master.key`와 암호화 키. 잃으면 모든 크레덴셜을 복호화할 수 없어 재등록해야 한다.

백업 불필요: `workspace/`(Git에서 다시 checkout), `plugins/`(재설치 가능).
