# 1단계 점검 — Jenkins 구조 핵심 질문

> 다루는 문서: `01-01.Jenkins가 제어하는 것`, `01-02.빌드 요청에서 실행까지`
> 답을 떠올린 뒤 본문 해당 섹션과 비교한다. 표·다이어그램은 본문에 있으므로 여기서는 핵심 통찰만 정리한다.

## Q1: Controller와 Agent를 분리해야 하는 이유는?

리소스 경쟁(빌드가 Controller JVM과 메모리를 공유하다 OOM), 보안(빌드 스크립트가 `JENKINS_HOME/secrets/`에 접근), 확장성(단일 서버의 수직 확장 한계) 세 가지 문제를 한꺼번에 해결하기 위함이다. Kubernetes에서 빌드마다 Pod를 동적 생성·삭제하는 방식이 임시 에이전트의 보안적 이점을 가장 잘 보여준다.

상세: `01-01.md` §2.

## Q2: Controller Executor를 0으로 설정하면 어떻게 되는가?

Agent가 연결되지 않은 상태에서 0으로 두면 모든 빌드가 Queue에서 무한 대기한다. 단, Pipeline은 **시작은 된다** — 구조 해석(Flyweight)은 Executor를 소비하지 않기 때문이다. 실제 `sh` step에 도달하는 순간 Agent Executor가 필요해진다. 운영 환경에서는 보안을 위해 0으로 두되 반드시 Agent를 먼저 연결한다.

조합별 결과 표: `01-01.md` §3.

## Q3: Queue에 있는 빌드와 실행 중인 빌드를 어떻게 구분하는가?

UI의 "실행 중" 표시는 신뢰할 수 없다. `/queue/api/json`의 `why` 필드와 `/job/.../{N}/api/json`의 `building` 필드를 함께 봐야 한다. 빌드를 트리거한 직후 받는 값은 buildNumber가 아니라 **queue item ID**이므로, `executable.number`가 채워질 때까지 폴링해야 실제 buildNumber를 얻는다.

상세 흐름: `01-02.md` §3 (queueId vs buildNumber).

## Q4: `agent any`, `agent { label }`, `agent none`의 차이는?

세 지시문은 모두 "**어디서 실행할 것인가**"를 정하지만, `agent none`은 Pipeline 레벨에서 실행 위치를 비워두고 stage마다 다르게 지정하기 위한 선택이다. Controller는 어떤 지시문이든 Flyweight(오케스트레이션)만 담당한다. Label 조건이 맞는 Agent가 없으면 Queue에서 무한 대기하므로 Agent 라벨 관리가 가용성과 직결된다.

지시문별 표: `01-02.md` §4.

## Q5: Flyweight 실행과 Heavyweight 실행의 차이는?

Flyweight는 `pipeline {}`/`stage()`/`when {}`/변수 할당 등 Controller에서 도는 오케스트레이션이고 Executor를 소비하지 않는다. Heavyweight는 `sh`/`bat`/`docker.build` 등 실제 작업으로 Agent Executor 1개를 점유한다. 이 구분을 모르면 "Controller Executor가 0인데 왜 Pipeline이 시작되는가"를 설명할 수 없다. 복잡한 로직을 Pipeline Groovy에 넣으면 Controller 메모리를 잡아먹으므로 셸 스크립트로 밀어낸다.

분류 표와 실습 코드: `01-02.md` §5.

## Q6: JENKINS_HOME은 어디에 존재하고, 무엇을 백업해야 하는가?

`JENKINS_HOME`은 **Controller에만 존재**한다. Agent에는 `workspace/`만 생기므로 잡 정의·크레덴셜·빌드 기록은 모두 Controller 백업 대상이다. 핵심은 `config.xml`(전역 설정), `jobs/`(잡 정의 + 빌드 히스토리), `secrets/`(`master.key` 포함 — 잃으면 모든 크레덴셜이 복호화 불가)이며 `workspace/`와 `plugins/`는 백업하지 않아도 복구 가능하다.

디렉토리 구조와 항목별 설명: `01-01.md` §4.
