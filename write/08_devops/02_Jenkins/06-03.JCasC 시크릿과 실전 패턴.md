# JCasC 시크릿과 실전 패턴

---

> JCasC에서 비밀값을 안전하게 주입하는 다섯 가지 방법과, 설정 파일을 분할·검증·자동화하는 실전 패턴을 다룬다.



## 1. Secret Source 전체 구조

> 비밀값을 주입하는 경로는 환경변수, Docker Secrets, K8s Secrets, HashiCorp Vault, AWS Secrets Manager 다섯 가지이며, 각각 보안 수준과 운영 복잡도가 다르다.

JCasC는 `${VAR}` 문법으로 비밀값을 주입한다. YAML 파일에 평문 비밀값을 직접 쓰는 대신, 실행 시점에 외부 소스에서 값을 읽어 치환하는 방식이다. 기본값 문법은 `${VALUE:-defaultvalue}`이며, 변수가 해결되지 않을 때 대신 사용할 값을 지정할 수 있다. 리터럴 `${`를 출력해야 할 때는 `^${VAR}` 형태로 이스케이프한다.

비밀값 소스는 운영 환경과 보안 요구 수준에 따라 선택한다:

| Source | 보안 수준 | 운영 복잡도 | 적합 환경 | 비고 |
|--------|----------|-----------|---------|------|
| 환경변수 | 낮음 | 낮음 | 로컬/PoC | Jenkins 관리자에게 노출 가능 |
| Docker Secrets | 중간 | 낮음 | Docker Compose | `/run/secrets/` 자동 매핑 |
| K8s Secrets | 중간 | 중간 | Kubernetes | 볼륨 마운트 필요 |
| HashiCorp Vault | 높음 | 높음 | 엔터프라이즈 | AppRole/Token/UserPass 인증 |
| AWS Secrets Manager | 높음 | 중간 | AWS 환경 | 플러그인 필요 |

JCasC는 변수 값을 변환하는 헬퍼 함수도 제공한다. 파일에서 읽은 값을 인코딩하거나, JSON 구조에서 특정 필드만 추출하는 용도로 사용한다:

| 헬퍼 | 기능 | 예시 |
|------|------|------|
| `base64` | Base64 인코딩 | `${base64:HELLO}` |
| `decodeBase64` | Base64 디코딩 | `${decodeBase64:SEVMTE8=}` |
| `readFile` / `file` | 파일 내용 읽기 | `${readFile:/secret/token.txt}` |
| `readFileBase64` / `fileBase64` | 바이너리 파일을 Base64로 | `${fileBase64:/secret/cert.p12}` |
| `json` | JSON에서 값 추출 | `${json:username:${ENV_VAR}}` |
| `trim` | 공백 제거 | `${trim:${readFile:/secret/pass.txt}}` |

헬퍼는 중첩하여 사용할 수 있다. 파일 내용을 읽은 뒤 Base64로 인코딩하는 예시는 다음과 같다:

```
${base64:${readFile:/secret/file.txt}}
```



## 2. 실습: Docker Secrets 기반 JCasC

> Docker Compose 환경에서 secrets 디렉토리에 파일을 마운트하고, JCasC가 파일명으로 자동 치환하는 과정을 실습한다.

Docker Secrets를 사용하면 `/run/secrets/${KEY}` 경로에 마운트된 파일이 JCasC의 `${KEY}` 변수로 자동으로 치환된다. 별도 플러그인 없이 파일명이 변수명이 되는 자동 매핑 방식이다. 기본 경로를 변경하려면 `SECRETS` 환경변수로 다른 디렉토리를 지정할 수 있다.

Docker Compose 파일 예시는 다음과 같다:

```yaml
version: "3.8"
services:
  jenkins:
    image: jenkins/jenkins:2.440.3-lts-jdk17
    environment:
      CASC_JENKINS_CONFIG: /var/jenkins_home/casc.d/
    secrets:
      - jenkins_admin_password
      - github_token
    volumes:
      - ./casc.d:/var/jenkins_home/casc.d:ro
    ports:
      - "8080:8080"

secrets:
  jenkins_admin_password:
    file: ./secrets/admin_password.txt
  github_token:
    file: ./secrets/github_token.txt
```

`jenkins.yaml`에서 Docker Secrets를 참조하는 방식은 다음과 같다:

```yaml
jenkins:
  securityRealm:
    local:
      allowsSignup: false
      users:
        - id: "admin"
          password: ${jenkins_admin_password}
```

실습 절차는 다음 순서로 진행한다:

1. `echo "mypassword" > secrets/admin_password.txt`로 시크릿 파일을 생성한다.
2. `docker compose up -d`로 Jenkins를 시작한다.
3. `admin/mypassword`로 로그인하여 정상 동작을 확인한다.
4. 시크릿 파일 내용을 새 비밀번호로 변경한다.
5. reload API를 호출하여 변경 사항을 반영한다.
6. 새 비밀번호로 로그인이 되는지 확인한다.



## 3. 실습: 다중 YAML 설정 소스

> `CASC_JENKINS_CONFIG`에 디렉토리를 지정하면 JCasC가 모든 YAML 파일을 재귀적으로 수집하여 병합한다. 단, 파일 간 같은 키를 선언하면 `ConfiguratorException`이 발생한다.

`CASC_JENKINS_CONFIG`는 단일 파일 경로, 디렉토리 경로, URL을 모두 허용한다. 디렉토리를 지정하면 하위 YAML 파일 전체를 읽어 하나의 설정으로 병합한다. 환경변수를 지정하지 않으면 `$JENKINS_HOME/jenkins.yaml`을 기본값으로 사용한다.

설정을 역할별로 분리하면 관리가 편해진다. 아래 구조처럼 루트 키 단위로 파일을 나누는 것이 충돌을 막는 가장 확실한 방법이다:

```
casc.d/
├── jenkins-core.yaml    # jenkins: 섹션
├── credentials.yaml     # credentials: 섹션
├── tools.yaml           # tool: 섹션
└── plugins.yaml         # unclassified: 섹션
```

에이전트 템플릿처럼 반복되는 설정은 YAML 앵커로 중복을 제거할 수 있다:

```yaml
x-agent-template: &agent_template
  remoteFS: "/home/jenkins"
  launcher:
    inbound:
      workDirSettings:
        disabled: true

jenkins:
  nodes:
    - permanent:
        name: "agent-1"
        <<: *agent_template
    - permanent:
        name: "agent-2"
        <<: *agent_template
```

주의할 점은 다중 파일이 "덮어쓰기"가 아닌 "보완" 방식으로 병합된다는 것이다. 두 파일이 같은 `jenkins.numExecutors` 키를 선언하면 `ConfiguratorException`이 발생하여 reload가 중단된다. 충돌을 피하려면 루트 키(`jenkins`, `tool`, `unclassified`, `credentials`) 단위로 파일을 나눠 각 파일이 서로 다른 루트 키를 담당하도록 분리한다.



## 4. 실습: Reload API 4가지 방식

> 설정 변경 후 Jenkins를 재시작하지 않고 반영하는 네 가지 경로를 비교한다.

| 방식 | 인증 | 용도 | 명령어 |
|------|------|------|------|
| Token 기반 | `CASC_RELOAD_TOKEN` 환경변수 | Webhook/CI 자동화 | `curl -X POST "JENKINS_URL/reload-configuration-as-code/?casc-reload-token=TOKEN"` |
| API Token 기반 | 사용자 인증 + CRUMB | 수동 reload | `curl -X POST -u admin:API_TOKEN "JENKINS_URL/configuration-as-code/reload"` |
| YAML POST | 사용자 인증 | 외부 YAML 직접 주입 | `curl -X POST -u admin:TOKEN --data-binary @jenkins.yaml "JENKINS_URL/configuration-as-code/configure"` |
| CLI | Jenkins CLI | 스크립트 자동화 | `java -jar jenkins-cli.jar -s JENKINS_URL reload-jcasc-configuration` |

Token 기반 방식이 CI/webhook 자동화에 가장 적합한 이유는 별도 사용자 인증 없이 토큰만으로 reload를 호출할 수 있기 때문이다. CRUMB 처리나 API 토큰 발급 없이 `CASC_RELOAD_TOKEN` 환경변수 하나로 webhook을 연결할 수 있어 구성이 단순하다.

YAML POST 방식은 다른 세 가지와 성격이 다르다. Git에 저장된 YAML을 reload하는 것이 아니라, 호출 시점에 직접 전달한 YAML 내용을 Jenkins에 밀어넣는다. 외부 시스템에서 동적으로 생성한 설정을 주입할 때 유용하지만, Git과 Jenkins 상태가 달라질 수 있으므로 주의가 필요하다.



## 5. 실습: export → diff 드리프트 감지

> Jenkins의 현재 상태를 주기적으로 export하고 Git의 `jenkins.yaml`과 비교하면 UI 변경에 의한 드리프트를 조기에 발견할 수 있다.

드리프트 감지의 기본 개념은 간단하다. Jenkins 현재 상태를 export API로 추출한 뒤, Git에 커밋된 `jenkins.yaml`과 diff를 비교하여 차이가 있으면 알림을 보낸다. 이를 매일 새벽에 자동 실행하는 Jenkinsfile 예시는 다음과 같다:

```groovy
pipeline {
    agent any
    triggers { cron('H 3 * * *') }

    environment {
        JENKINS_API_TOKEN = credentials('jenkins-api-token')
    }

    stages {
        stage('Export') {
            steps {
                sh '''
                    curl -sSf -u "admin:${JENKINS_API_TOKEN}" \
                      "${JENKINS_URL}/configuration-as-code/export" \
                      > exported.yaml
                '''
            }
        }
        stage('Diff') {
            steps {
                sh '''
                    DIFF=$(diff jenkins.yaml exported.yaml || true)
                    if [ -n "$DIFF" ]; then
                        echo "⚠️ 설정 드리프트 감지"
                        echo "$DIFF"
                        exit 1
                    fi
                '''
            }
        }
    }

    post {
        failure {
            echo '설정 드리프트가 감지되었다. jenkins.yaml과 실제 상태를 확인할 것.'
        }
    }
}
```

주의할 점은 export 결과에 `jenkins.yaml`에 명시하지 않은 기본값들이 포함된다는 것이다. 플러그인이 기본값으로 사용하는 필드까지 모두 출력되므로, diff에 노이즈가 섞인다. 현실적인 대응 방법은 자신이 명시적으로 관리하는 섹션에서만 드리프트를 확인하는 것이다. 예를 들어 `jenkins.securityRealm`, `jenkins.authorizationStrategy`, `credentials` 섹션만 추출하여 비교하면 노이즈를 줄일 수 있다.



## 6. Strict Secret Resolution

> 미해결 시크릿을 빈 문자열로 대체하는 기본 동작은 운영 사고를 유발할 수 있다. strict 모드를 활성화하면 시크릿이 하나라도 해결되지 않으면 reload가 중단된다.

JCasC의 기본 동작은 `${VAR}` 변수를 해결하지 못하면 빈 문자열로 조용히 대체하는 것이다. 이 동작은 위험하다. 예를 들어 `JENKINS_ADMIN_PASSWORD` 환경변수가 설정되지 않은 상태에서 reload하면 관리자 비밀번호가 빈 문자열이 되어 누구나 로그인할 수 있는 상태가 된다.

strict 모드는 다음 두 가지 방법으로 활성화한다:

```bash
# 환경변수로 활성화
CASC_STRICT_SECRET_RESOLUTION=true

# Java 시스템 속성으로 활성화
-Dcasc.strict.secret.resolution=true
```

strict 모드가 활성화되면 기본값 없이 선언된 변수 중 하나라도 해결되지 않으면 reload 전체가 중단된다. 비밀값 누락이 조용히 반영되는 대신, 명시적인 오류로 즉시 드러난다. **운영 환경에서는 항상 strict 모드를 활성화**해야 한다.

기본값 문법은 strict 모드에서도 그대로 동작한다. `${VAR:-fallback_value}` 형태로 선언한 변수는 `VAR`가 해결되지 않아도 `fallback_value`를 사용하므로 strict 모드 위반에 해당하지 않는다. 필수 시크릿은 기본값 없이, 선택적 시크릿은 기본값과 함께 선언하는 방식으로 두 요구사항을 함께 충족할 수 있다.
