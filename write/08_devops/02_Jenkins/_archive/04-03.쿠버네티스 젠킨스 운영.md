# 쿠버네티스 Jenkins 운영

---

> Jenkins를 Kubernetes에 설치하는 것은 시작일 뿐이다. SSL 구성, 백업, 모니터링, 보안 경화까지 갖춰야 프로덕션에서 안정적으로 운영할 수 있다.

## 1. SSL/TLS 구성

### 1-1. Ingress 기반 TLS 종료

Kubernetes에서 Jenkins에 HTTPS를 적용하는 가장 일반적인 방법은 Ingress Controller에서 TLS를 종료(terminate)하는 것이다. Jenkins 자체는 HTTP로 동작하고, 앞단의 Ingress가 인증서를 처리한다.

nginx-ingress + cert-manager + Let's Encrypt 조합이 사실상 표준이다:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: jenkins
  namespace: jenkins
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - jenkins.example.com
    secretName: jenkins-tls
  rules:
  - host: jenkins.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: jenkins
            port:
              number: 8080
```

`proxy-read-timeout: 3600`은 빌드 콘솔 로그의 장시간 스트리밍을 위해 필요하다. 기본값(60초)이면 장시간 빌드의 로그가 중간에 끊긴다.

Jenkins 측에서는 리버스 프록시 뒤에 있음을 인식해야 한다. `values.yaml`에 다음을 추가한다:

```yaml
controller:
  jenkinsUrl: "https://jenkins.example.com"
  jenkinsOpts: "--httpPort=8080"
  containerEnv:
  - name: JENKINS_OPTS
    value: >-
      --httpPort=8080
  ingress:
    enabled: true
    hostName: jenkins.example.com
```

이렇게 해야 Jenkins가 생성하는 URL(빌드 링크, Webhook 콜백 등)이 `https://`로 시작한다. 이 설정이 빠지면 Jenkins가 내부 Pod IP로 URL을 생성하여 외부에서 접근할 수 없는 링크가 만들어진다.

### 1-2. 에이전트-컨트롤러 통신 보안

에이전트 Pod는 JNLP(Java Network Launch Protocol) 또는 WebSocket으로 컨트롤러에 연결한다. 이 통신은 클러스터 내부에서 일어나므로 별도의 TLS 설정이 필요 없다.

Helm Chart 기본 설정에서 JNLP 포트(`50000`)는 ClusterIP 서비스로 노출된다. 클러스터 외부에서 접근할 수 없으므로 보안상 안전하다. 다만 에이전트가 다른 클러스터에 있는 멀티 클러스터 환경이라면 이 포트를 외부에 노출해야 하며, 이 경우 TLS를 반드시 설정해야 한다.

WebSocket 모드(`-webSocket` 플래그)를 사용하면 `50000` 포트 없이 HTTP/HTTPS 포트만으로 통신이 가능하다. Ingress가 WebSocket을 지원해야 하지만 nginx-ingress는 기본 지원한다.

### 1-3. 서비스 메시 환경에서의 mTLS

Istio나 Linkerd 같은 서비스 메시를 사용하는 클러스터에서는 Pod 간 통신이 자동으로 mTLS로 암호화된다. Jenkins 컨트롤러와 에이전트 Pod가 같은 메시 안에 있으면 추가 TLS 설정 없이 암호화가 적용된다.

주의할 점은 Istio의 사이드카 프록시가 JNLP 포트를 인터셉트할 때 연결 문제가 발생할 수 있다는 것이다. 이 경우 JNLP 포트를 사이드카에서 제외하거나 WebSocket 모드로 전환한다:

```yaml
# Pod 어노테이션으로 JNLP 포트를 사이드카에서 제외
annotations:
  traffic.sidecar.istio.io/excludeInboundPorts: "50000"
```

## 2. 백업 전략

Jenkins의 백업 대상은 크게 세 가지로 나뉜다. 각각에 맞는 도구를 사용해야 한다.

| 백업 대상 | 방법 | 복원 시간 | 자동화 수준 |
|----------|------|----------|------------|
| 전체 클러스터 리소스 | Velero 스냅샷 | 10~30분 | 높음 (스케줄 지원) |
| Jenkins 설정 (Job, 플러그인) | JCasC + Git | 즉시 (Helm install) | 높음 (GitOps) |
| 빌드 히스토리 + 아티팩트 | PVC 스냅샷 | 5~15분 | 중간 (CronJob) |

**JCasC + Git 기반 백업**이 가장 중요하다. Job 정의, 플러그인 목록, 보안 설정을 모두 코드로 관리하면 Jenkins를 언제든 `helm install` 한 번으로 재구성할 수 있다. 빌드 히스토리는 복원할 수 없지만, 설정 복원이 가장 시급한 상황(장애 복구)에서 가장 빠르다.

**Velero**는 PVC 스냅샷과 Kubernetes 리소스를 함께 백업한다. 일일 스케줄로 설정하면 전체 Jenkins 상태를 정기적으로 보관할 수 있다:

```bash
# Velero 백업 스케줄 생성
velero schedule create jenkins-daily \
  --schedule="0 2 * * *" \
  --include-namespaces jenkins \
  --ttl 168h
```

**PVC 스냅샷**은 CSI 드라이버가 지원하는 환경에서 VolumeSnapshot을 사용한다. CronJob으로 주기적 스냅샷을 생성하면 빌드 히스토리까지 보존된다. 클라우드 프로바이더별로 스냅샷 비용이 발생하므로 보존 기간을 적절히 설정한다.

## 3. 고가용성 (HA)

OSS Jenkins는 단일 마스터 아키텍처다. 컨트롤러가 하나뿐이므로 진정한 Active-Active HA는 불가능하다. 하지만 다운타임을 최소화하는 전략은 존재한다.

**StatefulSet + PDB 패턴**이 기본이다. Helm Chart는 StatefulSet replica=1로 컨트롤러를 배포한다. PodDisruptionBudget(PDB)을 설정하면 노드 드레인 시에도 Jenkins Pod가 먼저 다른 노드에 재스케줄링된 후 기존 Pod가 종료된다:

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: jenkins-pdb
  namespace: jenkins
spec:
  minAvailable: 0
  selector:
    matchLabels:
      app.kubernetes.io/name: jenkins
```

`minAvailable: 0`인 이유는 replica가 1이므로 업그레이드 시 불가피하게 0이 되기 때문이다. PDB의 역할은 의도하지 않은 동시 종료를 방지하는 것이지, 다운타임 자체를 없애는 것이 아니다.

**DR(Disaster Recovery) 패턴**은 Velero 백업 + 다른 클러스터에서 복원하는 방식이다. 주 클러스터가 완전히 실패했을 때 대체 클러스터에서 Jenkins를 복구한다. RTO(복구 목표 시간)는 보통 15~30분이다.

**CloudBees CI**는 Jenkins의 상용 버전으로 Active-Active HA를 지원한다. 여러 컨트롤러를 Operations Center로 관리하며, 하나가 실패해도 나머지가 빌드를 처리한다. 대규모 조직(100명 이상 개발자)에서 다운타임이 비용에 직결되는 경우에만 검토할 가치가 있다.

## 4. 플러그인 관리

Jenkins의 최대 약점 중 하나가 플러그인 관리다. 수백 개의 플러그인이 서로 의존하며, 하나를 업데이트하면 연쇄적으로 호환성 문제가 발생할 수 있다.

**버전 고정**이 첫 번째 원칙이다. `values.yaml`의 `installPlugins`에서 `latest` 대신 특정 버전을 명시한다:

```yaml
controller:
  installPlugins:
    - kubernetes:4246.v5a_12b_1fe120e
    - workflow-aggregator:600.vb_57cdd26fdd7
    - git:5.6.0
    - configuration-as-code:1850.va_a_8c31d3158b_
```

이렇게 하면 Chart를 재설치해도 동일한 플러그인 버전이 설치된다. `latest`를 사용하면 재설치 시점에 따라 다른 버전이 들어와 "어제까지 되던 게 오늘 안 됨" 현상이 발생한다.

**커스텀 Docker 이미지 프리베이킹**은 에어갭 환경에서 필수다. 플러그인을 이미지 빌드 시점에 포함시키면 설치 시 인터넷 접근이 불필요하다:

```dockerfile
FROM jenkins/jenkins:lts
COPY plugins.txt /usr/share/jenkins/ref/plugins.txt
RUN jenkins-plugin-cli --plugin-file /usr/share/jenkins/ref/plugins.txt
```

**보안 어드바이저리**를 정기적으로 확인해야 한다. Jenkins는 매주 보안 공지를 발행하며, 취약한 플러그인을 자동 비활성화하는 기능이 있다. Jenkins 관리 페이지의 "Manage Plugins > Security Warnings" 탭에서 확인할 수 있다.

## 5. 리소스 사이징

컨트롤러와 에이전트의 리소스는 동시 빌드 수와 빌드 복잡도에 따라 달라진다. 아래는 규모별 권장 사양이다.

| 규모 | 동시 빌드 | 컨트롤러 CPU/메모리 | 에이전트 CPU/메모리 | PVC |
|------|----------|-------------------|-------------------|-----|
| 소규모 (1~5명) | 1~3 | 500m / 1Gi | 250m / 512Mi | 10Gi |
| 중규모 (5~20명) | 5~15 | 1 / 2Gi | 500m / 1Gi | 30Gi |
| 대규모 (20~100명) | 15~50 | 2 / 4Gi | 1 / 2Gi | 100Gi |

컨트롤러의 메모리는 플러그인 수에 비례하여 증가한다. 플러그인 50개 이상이면 메모리를 4Gi 이상으로 잡는 것이 안전하다. `javaOpts`의 `-Xmx`는 컨테이너 limits의 60~70%로 설정한다. 나머지 30~40%는 JVM 메타스페이스와 OS 캐시가 사용한다.

에이전트의 리소스는 빌드 유형에 따라 크게 다르다. Java/Maven 빌드는 메모리를 많이 사용하므로 2Gi 이상이 필요하지만, 단순 스크립트 실행은 256Mi로도 충분하다. Pod 템플릿을 빌드 유형별로 분리하여 리소스를 차등 할당하는 것이 효율적이다.

## 6. 모니터링

### 6-1. Prometheus 플러그인

Jenkins Prometheus Metrics 플러그인을 설치하면 `/prometheus` 엔드포인트에서 메트릭을 수집할 수 있다. Helm Chart에서 ServiceMonitor를 자동 생성하도록 설정한다:

```yaml
controller:
  prometheus:
    enabled: true
    serviceMonitorEnabled: true
    serviceMonitorNamespace: monitoring
```

### 6-2. 핵심 메트릭과 알림 임계값

모든 메트릭을 모니터링할 필요는 없다. 운영에 실질적으로 영향을 주는 핵심 메트릭과 그 임계값은 다음과 같다:

| 메트릭 | 설명 | 경고 임계값 | 위험 임계값 |
|--------|------|-----------|-----------|
| `jenkins_queue_size_value` | 빌드 대기열 크기 | > 10 (5분 지속) | > 30 (5분 지속) |
| `jenkins_node_offline_value` | 오프라인 노드 수 | > 0 (10분 지속) | > 2 |
| `jenkins_job_duration_seconds` | 빌드 소요 시간 | p95 > 기준의 2배 | p95 > 기준의 5배 |
| `jvm_memory_used_bytes` | JVM 메모리 사용량 | > 80% heap | > 90% heap |
| `jenkins_health_check_score` | 전체 상태 점수 | < 80% | < 50% |

### 6-3. Grafana 대시보드

Grafana 대시보드 ID `9964`(Jenkins Performance and Health Overview)가 커뮤니티에서 가장 널리 쓰인다. `helm install`로 Grafana를 배포한 후 대시보드를 임포트하면 빌드 현황, 큐 상태, JVM 메트릭을 한눈에 볼 수 있다.

커스텀 대시보드를 만들 때는 다음 패널을 포함하는 것을 권장한다:

- 빌드 큐 길이 추이 (빌드 병목 감지)
- 에이전트 Pod 수 추이 (스케일링 패턴 파악)
- 빌드 성공/실패율 (품질 트렌드)
- JVM 힙 사용량 + GC 빈도 (메모리 누수 감지)

## 7. 보안 경화

### 7-1. RBAC 최소 권한

Kubernetes Plugin이 에이전트 Pod를 관리하려면 Kubernetes API 권한이 필요하다. Helm Chart는 기본적으로 ServiceAccount와 Role을 생성하지만, 권한 범위를 확인하고 최소화해야 한다.

Jenkins 컨트롤러에는 `jenkins` namespace 내에서 Pod 생성/삭제/조회, PVC 생성, Secret 조회 권한만 부여한다. 클러스터 전체에 대한 권한(ClusterRole)은 부여하지 않는다:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: jenkins
  namespace: jenkins
rules:
- apiGroups: [""]
  resources: ["pods", "pods/exec", "pods/log"]
  verbs: ["get", "list", "watch", "create", "delete"]
- apiGroups: [""]
  resources: ["persistentvolumeclaims"]
  verbs: ["get", "list", "create", "delete"]
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get"]
```

### 7-2. Pod Security Standards

Kubernetes 1.25+의 Pod Security Standards(PSS)를 적용하여 에이전트 Pod의 보안 수준을 강제한다. `baseline` 프로필을 namespace에 적용하면 특권 컨테이너, 호스트 네트워크 접근 등 위험한 설정이 차단된다:

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: jenkins
  labels:
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/warn: restricted
```

Docker-in-Docker(DinD)를 사용하는 에이전트는 `privileged: true`가 필요하므로 PSS와 충돌한다. 이 경우 Kaniko나 Buildah 같은 rootless 빌드 도구로 대체하는 것이 보안상 바람직하다.

### 7-3. NetworkPolicy

Jenkins 컨트롤러와 에이전트 Pod의 네트워크 통신을 최소 범위로 제한한다:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: jenkins-controller
  namespace: jenkins
spec:
  podSelector:
    matchLabels:
      app.kubernetes.io/name: jenkins
  policyTypes: ["Ingress", "Egress"]
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: ingress-nginx
    ports:
    - port: 8080
  - from:
    - podSelector:
        matchLabels:
          jenkins/agent: "true"
    ports:
    - port: 50000
  egress:
  - to: []
```

### 7-4. Vault 연동

Jenkins 시크릿(API 키, 레지스트리 인증 등)을 Jenkins 자체에 저장하면 PVC 백업에 시크릿이 포함되어 유출 위험이 있다. HashiCorp Vault나 Kubernetes External Secrets Operator로 시크릿을 외부에서 주입하는 것이 안전하다.

Jenkins HashiCorp Vault 플러그인을 사용하면 Jenkinsfile에서 Vault 경로로 시크릿을 참조할 수 있다:

```groovy
withVault(
    configuration: [vaultUrl: 'https://vault.example.com'],
    vaultSecrets: [
        [path: 'secret/jenkins/docker',
         secretValues: [[envVar: 'DOCKER_PASS', vaultKey: 'password']]]
    ]
) {
    sh 'docker login -u admin -p $DOCKER_PASS registry.example.com'
}
```

## 8. 업그레이드 전략

### 8-1. Blue-Green 업그레이드

Jenkins를 무중단으로 업그레이드하는 가장 안전한 방법은 PVC 스냅샷을 이용한 Blue-Green 전환이다:

1. 기존 Jenkins PVC의 스냅샷을 생성한다.
2. 스냅샷에서 새 PVC를 복원한다.
3. 새 PVC를 사용하는 Jenkins를 별도 namespace(또는 별도 릴리스명)에 배포한다.
4. 새 Jenkins에서 플러그인 호환성, 빌드 실행을 검증한다.
5. Ingress를 새 Jenkins로 전환한다.
6. 문제 발생 시 Ingress를 원래 Jenkins로 즉시 롤백한다.

### 8-2. Helm rollback

문제가 단순하면 Helm 롤백이 더 빠르다:

```bash
# 이전 리비전으로 롤백
helm rollback jenkins 1 -n jenkins

# 롤백 히스토리 확인
helm history jenkins -n jenkins
```

Helm 롤백은 Kubernetes 리소스(Deployment, ConfigMap 등)만 되돌린다. PVC 데이터(플러그인 파일, Job 설정)는 되돌리지 않으므로, 플러그인 업그레이드로 인한 데이터 마이그레이션이 발생한 경우에는 PVC 스냅샷 기반 롤백이 필요하다.

### 8-3. 업그레이드 전 체크리스트

업그레이드 실패의 대부분은 사전 확인 부족에서 비롯된다. 다음 항목을 매번 점검한다:

- 현재 실행 중인 빌드가 없는지 확인한다 (빌드 중 업그레이드하면 빌드가 유실된다).
- 타겟 Jenkins 버전의 릴리스 노트에서 Breaking Change를 확인한다.
- 사용 중인 플러그인이 새 Jenkins 버전과 호환되는지 Plugin Compatibility Matrix를 확인한다.
- PVC 스냅샷 또는 Velero 백업이 최신 상태인지 확인한다.
- 스테이징 환경에서 먼저 업그레이드를 시도한다.

### 8-4. 로그 관리

Jenkins 컨트롤러와 에이전트의 로그는 Kubernetes 환경에서 stdout/stderr로 출력되므로 클러스터의 로그 수집 파이프라인에 자동으로 포함된다.

Loki + Promtail 조합이 경량이면서도 Grafana와 자연스럽게 통합된다. EFK(Elasticsearch + Fluentd + Kibana)는 더 강력한 검색과 분석을 제공하지만 운영 비용이 크다. 소규모 팀에서는 Loki, 대규모 조직에서는 EFK를 선택하는 것이 일반적이다.

빌드 로그의 보존 기간은 Jenkins의 "Discard old builds" 설정으로 관리한다. Kubernetes 로그는 기본적으로 노드의 `/var/log/pods/`에 저장되며, logrotate에 의해 자동 삭제되므로 장기 보존이 필요하면 반드시 외부 저장소로 전송해야 한다.
