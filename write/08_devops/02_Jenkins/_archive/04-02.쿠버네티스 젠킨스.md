# 쿠버네티스 Jenkins

---

> Docker Compose로 Jenkins를 운영하면 단일 호스트의 자원 한계에 묶인다. Kubernetes 위에 Jenkins를 배포하면 빌드 에이전트를 Pod 단위로 동적 생성·삭제할 수 있어 자원 효율과 확장성이 근본적으로 달라진다.

## 1. 왜 쿠버네티스인가

> 03-01에서 다룬 Docker 기반 Jenkins는 `docker run`이나 Compose로 컨트롤러와 에이전트를 띄운다. 
>
> - 이 방식은 소규모 팀에서 빠르게 시작하기 좋지만, 동시 빌드가 늘어나면 한 호스트의 CPU·메모리를 넘어설 수 없다. 
> - 에이전트를 추가하려면 새 VM을 프로비저닝하고 SSH 연결을 수동으로 설정해야 한다.

Kubernetes 환경에서는 이 제약이 사라진다. Docker Compose가 "단일 서버의 편의"라면, Kubernetes는 "클러스터 수준의 탄력성"을 제공하는 것이다.

- **수평 확장**: 빌드 요청이 들어오면 Pod가 자동 생성되고, 끝나면 삭제된다. Cluster Autoscaler와 연동하면 노드 자체도 자동 확장된다.
- **자원 격리**: 각 빌드가 독립 Pod에서 실행되므로 의존성 충돌이 없다. Maven 빌드와 Node 빌드가 서로 다른 컨테이너 이미지를 사용할 수 있다.
- **선언적 관리**: Helm Chart와 JCasC(Jenkins Configuration as Code)로 Jenkins 전체 설정을 Git에 보관하고 재현 가능하게 관리한다.
- **자가 치유**: 컨트롤러 Pod가 죽으면 Kubernetes가 자동으로 재시작한다. PVC에 데이터를 보관하므로 상태가 유지된다.



## 2. Helm Chart로 설치하기

> Jenkins 공식 Helm Chart는 `jenkins/jenkins`로 배포된다. 
>
> 2026년 3월 기준 Chart 버전 5.x, Jenkins LTS 2.479.x가 최신이다. OCI 레지스트리(`oci://`)를 통한 설치도 지원한다.

Jenkins를 Kubernetes에 설치하는 방법은 세 가지가 있다:

| 방법 | 장점 | 단점 | 권장 상황 |
|------|------|------|----------|
| Helm Chart | 커뮤니티 검증, 풍부한 설정, 업그레이드 용이 | values.yaml 학습 필요 | 대부분의 환경 |
| 수동 YAML | 완전한 통제 | 유지보수 부담, 업그레이드 수동 | 특수 요구사항 |
| Jenkins Operator | CRD 기반 선언적 관리 | 성숙도 낮음, 커뮤니티 작음 | Operator 패턴 선호 시 |

- Helm Chart가 사실상 표준이다. 공식 문서, Stack Overflow 답변, 블로그 대부분이 Helm 기반이므로 트러블슈팅 자료도 가장 풍부하다.

### values.yaml 핵심 설정

Helm Chart의 `values.yaml`은 크게 네 섹션으로 나뉜다. 아래는 Playground 프로젝트에서 실제 사용 중인 설정이다:

```yaml
# --
controller:
  # NodePort로 UI 접근
  serviceType: NodePort
  nodePort: 30880

  resources:
    requests:
      cpu: 250m
      memory: 1Gi
    limits:
      cpu: 1000m
      memory: 2Gi

  # JVM 힙 설정
  javaOpts: "-Xms512m -Xmx1024m"

  # 초기 플러그인
  installPlugins:
    - kubernetes:latest
    - workflow-aggregator:latest
    - git:latest
    - configuration-as-code:latest

# --
persistence:
  enabled: true
  storageClass: nfs
  size: 5Gi

# --
agent:
  enabled: true
  resources:
    requests:
      cpu: 100m
      memory: 256Mi
    limits:
      cpu: 500m
      memory: 512Mi
```

- **controller**
  - Jenkins 마스터 Pod의 설정을 담당한다. 
  - `serviceType: NodePort`는 클러스터 외부에서 `30880` 포트로 직접 접근하게 한다. 
  - 프로덕션에서는 Ingress + LoadBalancer를 사용하지만, 개발 환경에서는 NodePort가 간단하다. 
  - `javaOpts`로 JVM 힙을 컨테이너 메모리의 50~70% 수준으로 설정하는 것이 중요한데, **컨테이너 limits와 JVM 힙이 맞지 않으면 OOMKilled가 발생한다.**

- **persistence** 
  - Jenkins 홈 디렉토리(`/var/jenkins_home`)를 PVC로 영속화한다. 
  - PVC에는 Job 설정, 빌드 히스토리, 플러그인, 시크릿이 저장된다. 
  - `storageClass: nfs`는 NFS 기반 스토리지를 사용하는 것으로, 클라우드 환경에서는 `gp3`(AWS)나 `standard`(GKE)를 사용한다.

- **agent**
  - 동적 에이전트 Pod의 기본 리소스를 정의한다. 
  - 이 설정은 Pod 템플릿에서 개별적으로 오버라이드할 수 있다.

- **installPlugins** 
  - 목록에서 `kubernetes` 플러그인이 핵심이다. 
  - 이 플러그인이 Jenkins와 Kubernetes API를 연결하여 동적 에이전트 Pod를 관리한다.

### 설치 및 업그레이드 명령어

Helm 저장소 추가부터 설치까지의 전체 흐름이다

```bash
# 저장소 추가
helm repo add jenkins https://charts.jenkins.io
helm repo update

# 네임스페이스 생성 + 설치
helm install jenkins jenkins/jenkins \
  -n jenkins --create-namespace \
  -f values.yaml

# 초기 admin 비밀번호 확인
kubectl get secret jenkins -n jenkins \
  -o jsonpath="{.data.jenkins-admin-password}" | base64 -d
```

업그레이드 시에는 `helm upgrade`를 사용한다

```bash
# 설정 변경 후 업그레이드
helm upgrade jenkins jenkins/jenkins \
  -n jenkins -f values.yaml

# 특정 Chart 버전으로 업그레이드
helm upgrade jenkins jenkins/jenkins \
  -n jenkins -f values.yaml --version 5.8.0
```

### 2-4. 자주 겪는 함정

Helm 설치 자체는 간단하지만, 설치 직후 마주치는 문제들이 있다.

1. **JCasC URL 충돌**이 가장 흔하다. 
   - `controller.jenkinsUrl`을 설정하지 않으면 Jenkins가 Pod IP를 자신의 URL로 인식한다. 
   - Pod가 재시작되면 IP가 바뀌므로 Webhook 콜백, 빌드 링크가 모두 깨진다. 
   - `values.yaml`에 `jenkinsUrl`을 Service 이름이나 Ingress 도메인으로 명시해야 한다.

2. **PVC 마운트 지연**도 자주 발생한다. 
   - 특히 `fsGroup`이 설정된 경우 대용량 PVC(50Gi 이상)의 초기 권한 변경에 수 분이 걸릴 수 있다. 
   - Kubernetes 1.23+에서는 `fsGroupChangePolicy: OnRootMismatch`를 설정하면 변경된 파일만 권한을 조정하여 시간을 줄인다.

3. **플러그인 다운로드 실패**는 에어갭 환경에서 나타난다. 
   - `installPlugins`에 `latest`를 지정하면 인터넷 접근이 필요하다. 
   - 폐쇄망에서는 플러그인을 미리 포함한 커스텀 Docker 이미지를 빌드하거나, Nexus 같은 내부 미러를 프록시로 설정한다.

4. **리소스 부족**도 주의해야 한다. 
   - Controller Pod에 `requests`를 너무 낮게 잡으면 스케줄러가 작은 노드에 배치하고, 플러그인 로딩 시 OOMKilled가 발생한다. 
   - 플러그인 30개 이상이면 메모리 `requests`를 최소 1Gi로 잡는 것이 안전하다.

## 3. 동적 에이전트 Pod (Kubernetes Plugin)

### 3-1. 동작 원리

Kubernetes Plugin은 Jenkins에서 가장 널리 쓰이는 에이전트 관리 방식이다. 빌드가 요청되면 Kubernetes API를 호출하여 에이전트 Pod를 생성하고, 빌드가 끝나면 Pod를 삭제한다. 전체 흐름은 다음과 같다:

1. 개발자가 빌드를 트리거한다 (Push, PR, 수동 등).
2. Jenkins 컨트롤러가 빌드 큐에 작업을 넣는다.
3. Kubernetes Plugin이 Pod 템플릿을 기반으로 에이전트 Pod를 생성한다.
4. 에이전트 Pod 내의 JNLP 컨테이너가 컨트롤러에 WebSocket으로 연결한다.
5. 컨트롤러가 빌드 작업을 에이전트에 전달하고, 에이전트가 실행한다.
6. 빌드 완료 후 Pod가 자동 삭제된다.

Pod 템플릿은 JCasC나 Jenkinsfile에서 정의한다. Jenkinsfile에서 인라인으로 정의하는 방식이 가장 유연하다:

```groovy
pipeline {
    agent {
        kubernetes {
            yaml '''
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: maven
    image: maven:3.9-eclipse-temurin-17
    command: ['sleep']
    args: ['infinity']
    resources:
      requests:
        cpu: 500m
        memory: 1Gi
      limits:
        cpu: 2
        memory: 2Gi
  - name: docker
    image: docker:27-dind
    securityContext:
      privileged: true
'''
        }
    }
    stages {
        stage('Build') {
            steps {
                container('maven') {
                    sh 'mvn clean package -DskipTests'
                }
            }
        }
        stage('Docker Build') {
            steps {
                container('docker') {
                    sh 'docker build -t myapp:latest .'
                }
            }
        }
    }
}
```

`container()` 블록으로 각 스테이지가 사용할 컨테이너를 지정한다. 하나의 Pod 안에 여러 컨테이너(sidecar 패턴)를 넣어 Maven 빌드와 Docker 빌드를 같은 워크스페이스에서 처리할 수 있다.

### 3-2. 장점

동적 에이전트의 가장 큰 장점은 **자원 효율**이다. 빌드가 없으면 에이전트 Pod가 0개이므로 클러스터 자원을 소비하지 않는다. 야간이나 주말에 빌드가 없으면 Cluster Autoscaler가 노드까지 축소한다.

**격리** 측면에서도 우수하다. 각 빌드가 깨끗한 Pod에서 시작하므로 이전 빌드의 잔여 파일이나 환경 변수가 영향을 주지 않는다. "내 로컬에서는 되는데 CI에서 안 돼요"의 원인 중 하나인 상태 오염(State Pollution)을 원천 차단한다.

**멀티테넌시**도 자연스럽게 지원된다. 팀별로 다른 Pod 템플릿과 namespace를 사용하면 리소스 쿼터와 네트워크 정책으로 팀 간 격리가 가능하다.

### 3-3. 단점과 콜드 스타트 완화

동적 에이전트의 최대 약점은 **콜드 스타트**다. Pod 생성부터 빌드 시작까지 지연이 발생하며, 이 지연은 세 단계로 나뉜다:

- **이미지 풀**: 노드에 이미지가 없으면 레지스트리에서 다운로드한다. `maven:3.9` 이미지가 ~500MB이므로 첫 풀에 30초~2분이 걸린다.
- **Pod 스케줄링 + 컨테이너 기동**: Kubernetes 스케줄러가 노드를 선택하고 컨테이너를 시작한다. 보통 5~15초.
- **JNLP 연결**: 에이전트가 컨트롤러에 연결하고 작업을 수신한다. 보통 3~10초.

합산하면 첫 빌드 시 30초~3분의 오버헤드가 생길 수 있다. 다음 전략으로 완화한다:

**이미지 프리풀(Pre-pull)**: DaemonSet으로 자주 쓰는 이미지를 모든 노드에 미리 다운로드해둔다. 가장 효과적인 방법이다:

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: jenkins-image-prepull
spec:
  selector:
    matchLabels:
      app: image-prepull
  template:
    metadata:
      labels:
        app: image-prepull
    spec:
      initContainers:
      - name: maven
        image: maven:3.9-eclipse-temurin-17
        command: ['echo', 'pulled']
      - name: node
        image: node:20-slim
        command: ['echo', 'pulled']
      containers:
      - name: pause
        image: registry.k8s.io/pause:3.9
```

**imagePullPolicy 설정**: `IfNotPresent`를 사용하면 이미 존재하는 이미지를 다시 풀하지 않는다. `latest` 태그 대신 특정 버전 태그를 사용해야 이 정책이 효과적으로 작동한다.

**PVC 캐시**: Maven의 `.m2`, Gradle의 `.gradle`, npm의 `node_modules` 캐시를 PVC로 공유하면 의존성 다운로드 시간을 줄인다:

```yaml
volumes:
- name: maven-cache
  persistentVolumeClaim:
    claimName: jenkins-maven-cache
containers:
- name: maven
  volumeMounts:
  - name: maven-cache
    mountPath: /root/.m2/repository
```

**경량 이미지**: `maven:3.9-eclipse-temurin-17-alpine` 같은 Alpine 기반 이미지를 사용하면 풀 시간이 절반 이하로 줄어든다. 다만 glibc 의존성이 있는 도구는 Alpine에서 동작하지 않을 수 있으므로 사전 검증이 필요하다.

### 3-4. Playground 실제 설정 분석

Playground 프로젝트의 `values.yaml`에서 에이전트 섹션을 살펴보면:

```yaml
agent:
  enabled: true
  resources:
    requests:
      cpu: 100m
      memory: 256Mi
    limits:
      cpu: 500m
      memory: 512Mi
```

이 설정은 개발 환경에 맞춰 보수적으로 잡은 것이다. `requests`가 100m/256Mi로 낮은 이유는 개발 클러스터의 노드 사양이 작기 때문이다. 프로덕션에서 Maven 빌드를 돌린다면 `requests`를 500m/1Gi, `limits`를 2/2Gi 정도로 올려야 빌드 중 OOMKilled를 피할 수 있다.

`controller.installPlugins`에 `kubernetes:latest`가 포함되어 있으므로 동적 에이전트 기능이 기본 활성화된다. 개발 환경에서는 이 최소 설정으로 충분하지만, 프로덕션에서는 Pod 템플릿을 JCasC로 세분화하여 빌드 유형별 리소스를 관리하는 것이 좋다.

## 4. 정적 에이전트 Pod (Always-On)

### 4-1. 동작 원리

정적 에이전트는 빌드 유무와 관계없이 항상 실행 중인 에이전트 Pod다. StatefulSet이나 Deployment로 배포하고, 영구 PVC를 마운트하며, JNLP로 컨트롤러에 상시 연결된다.

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: jenkins-agent-gpu
  namespace: jenkins
spec:
  serviceName: jenkins-agent-gpu
  replicas: 1
  selector:
    matchLabels:
      app: jenkins-agent-gpu
  template:
    metadata:
      labels:
        app: jenkins-agent-gpu
    spec:
      containers:
      - name: jnlp
        image: jenkins/inbound-agent:latest
        env:
        - name: JENKINS_URL
          value: "http://jenkins.jenkins.svc:8080"
        - name: JENKINS_AGENT_NAME
          value: "gpu-agent-0"
        - name: JENKINS_SECRET
          valueFrom:
            secretKeyRef:
              name: jenkins-agent-gpu
              key: secret
        resources:
          requests:
            cpu: 2
            memory: 8Gi
            nvidia.com/gpu: 1
          limits:
            cpu: 4
            memory: 16Gi
            nvidia.com/gpu: 1
        volumeMounts:
        - name: workspace
          mountPath: /home/jenkins/agent
  volumeClaimTemplates:
  - metadata:
      name: workspace
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 50Gi
```

StatefulSet을 사용하는 이유는 Pod 이름이 안정적이어야 Jenkins에서 에이전트를 식별할 수 있기 때문이다. `gpu-agent-0`이라는 이름이 Pod 재시작 후에도 유지된다.

### 4-2. 장점

정적 에이전트의 첫 번째 장점은 **제로 콜드 스타트**다. 이미 실행 중이므로 빌드 요청이 들어오면 즉시 작업을 시작한다. 지연에 민감한 파이프라인(예: 핫픽스 배포)에서 유용하다.

**영구 워크스페이스**도 장점이다. PVC에 워크스페이스가 유지되므로 증분 빌드(Incremental Build)가 가능하다. Maven이나 Gradle의 증분 컴파일을 활용하면 전체 빌드 대비 50~80% 시간을 절약할 수 있다.

**특수 하드웨어 접근**이 필요한 경우에도 정적 에이전트가 적합하다. GPU, FPGA, 라이선스 서버 연결 같은 자원은 Pod가 생성될 때마다 연결을 설정하기 어렵다. 상시 연결된 에이전트가 이런 자원을 점유하고 빌드에 제공하는 것이 안정적이다.

### 4-3. 단점

**유휴 자원 낭비**가 가장 큰 문제다. 빌드가 없는 야간에도 GPU Pod가 리소스를 점유한다. 클라우드 환경에서는 이것이 직접적인 비용으로 이어진다.

**상태 오염** 위험도 존재한다. 이전 빌드가 남긴 파일이 다음 빌드에 영향을 줄 수 있다. 워크스페이스 정리(`deleteDir()`)를 빌드 전후에 넣어야 하지만, 이를 빠뜨리면 "어제는 됐는데 오늘은 안 돼요" 같은 문제가 발생한다.

**확장 한계**도 있다. StatefulSet의 replica 수를 수동으로 조정해야 하므로 동적 확장이 어렵다. HPA를 붙일 수 있지만, Jenkins 빌드 큐와 HPA 메트릭을 연동하는 것은 복잡하다.

## 5. 하이브리드 전략 (실무 권장)

실무에서는 동적 에이전트와 정적 에이전트를 함께 사용하는 하이브리드 전략이 가장 효과적이다. 대부분의 빌드는 동적 에이전트로 처리하고, 특수한 요구사항이 있는 빌드만 정적 에이전트로 라우팅한다.

라우팅은 **라벨 기반**으로 구현한다. Jenkinsfile에서 `agent { label 'dynamic-maven' }` 또는 `agent { label 'gpu-static' }`으로 에이전트를 선택한다. Kubernetes Plugin은 라벨에 매칭되는 Pod 템플릿을 사용하여 동적 Pod를 생성하고, 정적 에이전트는 미리 등록된 노드 라벨로 매칭된다.

실무 패턴의 구성은 보통 다음과 같다:

- **동적 에이전트 0~50 Pod**: Maven, Gradle, Node.js, Python 등 일반 빌드 용도. 빌드 큐에 따라 자동 확장/축소된다.
- **정적 에이전트 1~3 Pod**: GPU 학습, 라이선스 도구(SonarQube Enterprise, Fortify), 대용량 증분 빌드 용도.

```groovy
// Jenkinsfile 라벨 기반 라우팅 예시
pipeline {
    agent none
    stages {
        stage('Build & Test') {
            agent { label 'dynamic-maven' }
            steps {
                container('maven') {
                    sh 'mvn clean verify'
                }
            }
        }
        stage('ML Model Validation') {
            agent { label 'gpu-static' }
            steps {
                sh 'python validate_model.py'
            }
        }
    }
}
```

Cluster Autoscaler와 연동하면 동적 에이전트가 쏟아질 때 노드가 자동으로 추가된다. 단, Autoscaler의 스케일업에는 2~5분이 걸리므로 급격한 빌드 증가(예: 월요일 아침 전체 PR 빌드)에는 **오버프로비저닝(Over-Provisioning)** 전략이 필요하다. 빈 Pod(Pause 컨테이너)를 낮은 Priority로 미리 배포해두면 새 에이전트 Pod가 빈 Pod를 밀어내고 즉시 스케줄링된다.

## 6. 비교 요약

| 속성 | 동적 에이전트 | 정적 에이전트 | 하이브리드 |
|------|-------------|-------------|----------|
| **콜드 스타트** | 30초~3분 | 없음 | 유형별 상이 |
| **자원 효율** | 높음 (사용 시만 소비) | 낮음 (상시 점유) | 높음 |
| **빌드 격리** | 완벽 (매번 새 Pod) | 약함 (워크스페이스 공유) | 유형별 선택 |
| **확장성** | 자동 (0~N Pod) | 수동 (replica 조정) | 자동 + 수동 |
| **증분 빌드** | 불가 (PVC 캐시로 부분 대체) | 가능 | 선택적 |
| **특수 하드웨어** | 어려움 | 적합 | GPU만 정적 |
| **운영 복잡도** | 낮음 | 중간 | 중간 |

대부분의 팀에서는 동적 에이전트만으로 충분하다. 정적 에이전트는 GPU, 라이선스 도구, 지연 민감 파이프라인 같은 명확한 이유가 있을 때만 추가한다. 하이브리드 전략은 이 두 가지를 조합하여 비용과 성능의 균형을 잡는 실무적 선택이다.
