# Kaniko 대체 검토
---
> **Jenkins on Kubernetes** 환경을 기준으로, Kaniko를 쓸지, 아니면 다른 빌드 체인으로 바꾸는 것이 맞는지를 판단하기 위한 비교 문서
>
> 비교 대상은 다음 4가지다.
>
> - **BuildKit**
> - **kpack**
> - **Buildah**
> - **언어 특화 빌더**: Java는 **Jib**, Go는 **ko**

## 요약

결론부터 짧게 쓰면 다음과 같다.

- **당장 Kaniko를 급히 걷어낼 필요가 있는 팀은 아니다**.
- 하지만 **2025년 6월 3일 Kaniko 원본 저장소가 archive**되었기 때문에, 새로 표준을 잡는 팀이라면 Kaniko를 신규 표준으로 채택하는 판단은 보수적으로 해야 한다.
- **Dockerfile 중심 Jenkins 파이프라인을 유지하고 싶다면 BuildKit이 가장 유력한 대체 후보**다.
- **플랫폼 차원에서 선언적으로 이미지 빌드를 운영하고 싶다면 kpack**이 더 적합하다.
- **OpenShift/Red Hat 친화성과 rootless 이미지 빌드를 중시하면 Buildah**를 검토할 수 있다.
- **Java 또는 Go처럼 언어가 명확하다면 Jib/ko가 오히려 가장 단순한 답**이 될 수 있다.

즉, Kaniko 대체 필요 여부는 "Kaniko가 지금 당장 고장났는가?"가 아니라 아래 질문으로 판단해야 한다.

1. 앞으로도 Dockerfile 기반 Jenkins 파이프라인을 유지할 것인가
2. 클러스터 안에서 빌드 엔진을 별도 운영할 수 있는가
3. 언어별 최적화 도구를 받아들일 수 있는가
4. 장기 유지보수 리스크를 감수할 것인가



## 2. 최신 유지보수 상태

2026년 4월 1일 기준으로 확인한 공식 소스의 상태는 다음과 같다.

| 도구 | 상태 요약 | 근거 |
| --- | --- | --- |
| **Kaniko** | 원본 저장소 archive | `GoogleContainerTools/kaniko` 저장소 archive, 2025-06-03 |
| **BuildKit** | 활발히 유지보수 중 | `moby/buildkit` 최신 릴리스 `v0.26.3`, 2025-12-16 |
| **kpack** | 활발히 유지보수 중 | `buildpacks-community/kpack` 최신 릴리스 `v0.17.1`, 2025-12-09 |

- **유지보수 활력만 놓고 보면 Kaniko가 가장 약한 위치**다.
- 다만 이것이 곧바로 "즉시 폐기"를 뜻하지는 않는다. 이미 안정적으로 돌아가는 Jenkins 파이프라인이라면 당장 급변환보다 계획적인 마이그레이션이 더 중요하다.



## 3. Jenkins on Kubernetes에서 무엇이 중요한가

로컬 개발 머신과 달리, Jenkins on Kubernetes에서는 다음 항목이 중요하다.

- `privileged` 없이 동작하는가
- `docker.sock` 마운트가 필요한가
- Dockerfile을 그대로 유지할 수 있는가
- 캐시 성능이 충분한가
- 멀티 아키텍처 빌드가 쉬운가
- 레지스트리 인증을 Pod 보안 모델에 맞게 넣을 수 있는가
- Jenkinsfile 변경 폭이 큰가
- 장기 유지보수 리스크가 낮은가

결국 이 문서는 "누가 더 멋진 도구인가"가 아니라, **Kubernetes Jenkins에서 누가 가장 덜 아프게 운영되는가**를 비교하는것이 중요합니다.

### 비교 테이블

| 항목 | Kaniko | BuildKit | kpack | Buildah | Jib / ko |
| --- | --- | --- | --- | --- | --- |
| 주 용도 | Dockerfile 빌드 | Dockerfile/LLB 빌드 | Buildpacks 기반 빌드 서비스 | OCI 이미지 빌드 | 언어 전용 이미지 빌드 |
| Kubernetes 친화성 | 높음 | 높음 | 매우 높음 | 중간~높음 | 중간 |
| Docker daemon 필요 | 없음 | 없음 | 없음 | 없음 | 없음 |
| Dockerfile 유지 | 가능 | 가능 | 보통 불가 | 가능 | 보통 불필요 |
| 캐시 성능 | 보통 | 강함 | 좋음 | 좋음 | 매우 좋음 |
| 멀티 아키텍처 | 별도 설계 필요 | 강함 | 빌드팩 체인 의존 | 가능하나 운영 난이도 있음 | Jib/ko 모두 강점 있음 |
| Jenkinsfile 친화성 | 높음 | 높음 | 낮음 | 중간 | 높음 |
| 레지스트리 인증 | secret/IAM | daemon 또는 pod 단 auth | serviceAccount + secret | secret/auth 파일 | Maven/Gradle auth 또는 Docker config |
| 장기 유지보수 | 낮음 | 높음 | 높음 | 높음 | 높음 |

- **Kaniko -> BuildKit**: 가장 자연스러운 교체
- **Kaniko -> kpack**: 운영 철학 자체가 바뀌는 교체
- **Kaniko -> Buildah**: 가능하지만 Jenkins Kubernetes 기준 주류 전환은 아님
- **Kaniko -> Jib/ko**: 특정 언어에서는 가장 좋은 교체



## 4. 도구별 상세 비교

## 5-1. BuildKit

> BuildKit은 현재 Docker 생태계에서 가장 현실적인 차세대 기본 빌더에 가깝다.

BuildKit은 `buildkitd` 데몬과 `buildctl` 클라이언트 구조를 가지며, Dockerfile 빌드 외에도 캐시, 멀티 플랫폼, rootless 실행, 다양한 출력 형식을 강하게 지원한다.

Jenkins on Kubernetes에서의 전형적인 구조는 다음 두 가지다.

1. 클러스터 안에 `buildkitd`를 별도 서비스/Pod로 띄우고 Jenkins 에이전트 Pod에서 `buildctl`로 원격 호출
2. rootless 또는 daemonless 형태를 Jenkins Pod 안에서 실행

### 장점

- Dockerfile을 유지할 수 있다.
- 캐시 import/export가 강력하다.
- 멀티 플랫폼 빌드 지원이 좋다.
- BuildKit 생태계가 활발하고, Docker/Buildx와도 자연스럽게 이어진다.
- 장기적으로 Kaniko보다 유지보수 리스크가 낮다.

### 단점

- Kaniko보다 구조가 한 단계 더 복잡하다.
- `buildkitd`를 어디서 어떻게 운영할지 결정해야 한다.
- 인증 정보가 **클라이언트가 아니라 daemon 쪽에 필요할 수 있어** 초기 설계가 헷갈릴 수 있다.
- 팀이 Buildx/BuildKit 개념에 익숙하지 않으면 학습 비용이 있다.

### Jenkins on Kubernetes 적합도

매우 높다. 특히 **기존 Dockerfile + Jenkinsfile 흐름을 유지하면서 Kaniko만 교체하고 싶은 팀**에 가장 잘 맞는다.

### Kaniko 대체 관점 판단

- **대체 1순위**
- Kaniko에서 가장 덜 아프게 옮길 수 있는 방향

## 5-2. kpack

> kpack은 "빌드 도구"라기보다 **Kubernetes 네이티브 빌드 서비스**에 가깝다.

kpack은 Cloud Native Buildpacks를 기반으로, `Image`, `Builder`, `Build` 같은 Kubernetes 리소스를 통해 이미지를 선언적으로 빌드한다. Jenkins는 셸 스크립트로 빌드하기보다, kpack 리소스를 생성하거나 갱신하는 트리거 역할에 가까워진다.

### 장점

- Kubernetes 네이티브 운영에 매우 잘 맞는다.
- 빌드, 재빌드, builder 업데이트 반영이 선언적으로 관리된다.
- 비-privileged Kubernetes primitives 중심으로 설계되어 보안 모델이 좋다.
- 플랫폼 팀 관점에서는 CI 빌드를 표준화하기 쉽다.

### 단점

- Dockerfile 중심 사고를 유지하기 어렵다.
- Jenkinsfile에서 `sh 'docker build ...'`를 `sh '/kaniko/executor ...'`로 바꾸는 정도의 단순 전환이 아니다.
- Buildpacks 개념, builder image, stack, lifecycle을 이해해야 한다.
- 애플리케이션별 세밀한 Dockerfile 제어가 필요한 팀에는 답답할 수 있다.

### Jenkins on Kubernetes 적합도

높지만, 전제 조건이 있다. **Jenkins를 빌드 엔진으로 쓰는 팀이 아니라, Jenkins를 오케스트레이터로 낮추고 플랫폼 빌드 서비스를 도입하려는 팀**에 적합하다.

### Kaniko 대체 관점 판단

- **전략 전환형 대체**
- "Kaniko를 다른 Dockerfile 빌더로 바꾸는" 것이 아니라 "빌드 체계 자체를 Kubernetes 리소스 중심으로 바꾸는" 선택

## 5-3. Buildah

> Buildah는 OCI 이미지 빌드 도구로서 강력하지만, Jenkins on Kubernetes에서 기본 선택지로 보기는 다소 무겁다.

Buildah는 Dockerfile 빌드(`buildah bud`)도 가능하고, 더 낮은 수준에서 이미지 레이어와 컨테이너 작업을 조작할 수도 있다. Rootless와 Podman 생태계, OpenShift 쪽과의 친화성이 강하다.

### 장점

- Docker daemon이 필요 없다.
- OCI 이미지 중심으로 유연하게 제어할 수 있다.
- Rootless 운영 시나리오와 잘 맞는다.
- OpenShift/Red Hat 계열 환경에서는 친숙하다.

### 단점

- Jenkins on Kubernetes에서 팀 공통 표준으로 쓰기엔 BuildKit보다 덜 직관적일 수 있다.
- Dockerfile만 쓰는 팀에게는 기능이 과하다.
- 캐시와 멀티 플랫폼 운영 경험은 BuildKit 쪽이 더 자연스럽다.
- Podman/containers 계열 이해가 필요하다.

### Jenkins on Kubernetes 적합도

중간 이상이다. 특히 OpenShift 또는 Red Hat 계열 기반이면 후보가 될 수 있지만, 일반적인 K8s Jenkins 환경에서는 BuildKit보다 우선순위가 낮은 경우가 많다.

### Kaniko 대체 관점 판단

- **조건부 대체**
- OpenShift 친화성, rootless 강박, Podman 계열 표준화 요구가 있으면 좋다
- 아니면 일반적으로는 BuildKit이 더 자연스럽다

## 5-4. 언어 특화 빌더: Jib / ko

> 이 범주는 "범용 빌더"가 아니라 **언어별 최적화 빌더**다.

- **Jib**: Java 애플리케이션 이미지를 Docker daemon 없이 Maven/Gradle에서 직접 빌드하고 push
- **ko**: Go 애플리케이션 이미지를 `go build` 기반으로 빠르게 이미지화하고 push

### 장점

- Jenkins Kubernetes에서 별도 Dockerfile 빌더를 운영하지 않아도 된다.
- 해당 언어에서는 빌드 속도와 캐시 효율이 좋다.
- Docker daemon이 필요 없다.
- 해당 언어 생태계에 맞는 빌드 모델을 제공한다.

### 단점

- 범용성이 없다.
- Java/Go 이외 프로젝트에는 그대로 적용할 수 없다.
- Dockerfile 세부 제어가 중요한 프로젝트에는 부적합할 수 있다.
- 팀 전체 표준을 하나로 통일하기 어려워질 수 있다.

### Jenkins on Kubernetes 적합도

특정 언어에서는 매우 높다. 특히:

- Java Spring Boot 서비스 -> **Jib**
- Go 마이크로서비스 -> **ko**

다만 여러 언어가 섞인 플랫폼에서는 공통 표준으로 쓰기 어렵다.

### Kaniko 대체 관점 판단

- **서비스 단위 대체**
- 플랫폼 전체 대체재보다는 특정 언어 서비스에서 Kaniko를 걷어내는 데 매우 효과적



## 6. 레지스트리 보안: Jenkins Credentials만으로 충분한가

여기가 실무에서 가장 자주 놓치는 부분이다.

결론부터 말하면:

> **아니다. Jenkins Credentials만 등록해 두는 것만으로는 충분하지 않은 경우가 많다.**

이유는 실제 이미지 push를 수행하는 주체가 Jenkins Controller가 아니라, **Kubernetes 위에서 실행 중인 빌드 Pod 또는 그 안의 빌드 엔진**이기 때문이다.

즉, "Jenkins UI에 credential이 있다"와 "빌드 Pod가 registry push 권한을 가진다"는 같은 말이 아니다.

### 왜 별도 설계가 필요한가

Jenkins on Kubernetes에서는 아래 계층이 분리된다.

1. Jenkins Controller
2. Jenkins 에이전트 Pod
3. 빌드 컨테이너 또는 빌드 daemon
4. 레지스트리

레지스트리 인증은 **실제로 push/pull을 수행하는 실행 주체**가 가져야 한다.

그래서 보통은 다음 중 하나로 설계한다.

- Kubernetes Secret을 Pod에 마운트
- Docker config를 컨테이너 안에 마운트
- ServiceAccount + imagePullSecrets 사용
- 클라우드 IAM 연동
  - GKE: Workload Identity
  - EKS: IRSA
  - 기타: workload identity federation 또는 노드/워크로드 IAM

### Jenkins Credentials의 역할

Jenkins Credentials는 여전히 유용하다. 다만 역할이 바뀐다.

- Kubernetes Secret 생성 파이프라인의 입력값으로 사용
- 외부 비밀 관리자에서 비밀을 받아오는 초기 진입점으로 사용
- Jenkinsfile에서 직접 registry 비밀번호를 노출하지 않도록 하는 래퍼로 사용

하지만 이것만으로 끝나지 않는다. 최종적으로는 **빌드 Pod가 해당 자격증명을 실제로 읽을 수 있어야 한다.**

### 도구별로 보면

| 도구 | Registry auth가 실제로 필요한 위치 | 보안 권장 패턴 |
| --- | --- | --- |
| Kaniko | `kaniko` 컨테이너 | `/kaniko/.docker/config.json` secret 마운트 또는 IAM |
| BuildKit | `buildkitd` 또는 build pod | daemon 수준 auth, secret, IAM |
| kpack | `serviceAccount` + secret | kpack `Image.spec.serviceAccount`와 연결 |
| Buildah | build pod 내부 auth 파일 | secret 마운트, containers auth config, IAM |
| Jib | Maven/Gradle 실행 환경 | Maven/Gradle 설정, Jib auth, secret 주입 |
| ko | `ko` 실행 환경 | Docker config, `ko login`, 클라우드 자격증명 |

### 실무 권장

- **가능하면 Jenkins Credentials 직접 주입보다 Kubernetes-native secret/IAM 구조로 정리하는 편이 낫다.**
- 클라우드 레지스트리라면 가능한 한 **정적 비밀번호 대신 워크로드 아이덴티티**를 우선 고려한다.
- Jenkins Controller가 아니라 **에이전트 Pod 권한**을 기준으로 인증 구조를 설계해야 한다.

즉, "Jenkins에 credential 넣었으니 끝"이라고 생각하면 안 되고, **어느 Pod의 어느 프로세스가 이미지를 push하느냐**까지 추적해야 한다.



## 7. 언제 Kaniko를 그대로 써도 되는가

다음 조건이면 Kaniko를 당장 교체하지 않아도 된다.

- 이미 안정적으로 운영 중이다
- Dockerfile 기반 파이프라인이 단순하다
- 멀티 아키텍처 요구가 강하지 않다
- 현재 캐시/성능 문제가 크지 않다
- 내부 표준 문서와 운영 경험이 Kaniko에 맞춰져 있다
- 단기적으로는 기능 추가보다 안정성이 더 중요하다

즉, **지금 잘 돌아가는데 archive만 보고 즉시 교체하는 것은 과할 수 있다**.



## 8. 언제 Kaniko 대체를 진지하게 추진해야 하는가

다음 조건이면 대체 검토가 실질적으로 필요하다.

- 신규 표준을 지금 정해야 한다
- 장기 유지보수 리스크를 줄이고 싶다
- 멀티 아키텍처 빌드가 중요하다
- 캐시 성능이 병목이다
- Dockerfile 고급 기능이나 BuildKit 기능이 필요하다
- 플랫폼 팀 차원의 빌드 서비스 표준화가 필요하다
- 언어별 빌더(Jib/ko)로 빌드 시간을 크게 줄일 수 있다



## 9. Jenkins on Kubernetes 기준 추천 시나리오

### 9-1. 가장 일반적인 추천

Dockerfile 기반 Jenkins 파이프라인을 유지해야 한다면:

- **Kaniko 유지 -> 단기**
- **BuildKit 전환 -> 중기**

이 경로가 가장 현실적이다.

### 9-2. 플랫폼 팀이 강한 조직

플랫폼 팀이 있고, 애플리케이션 팀이 Dockerfile 세부 제어보다 표준 빌드 서비스를 받아들이는 조직이라면:

- **kpack**이 가장 전략적으로 맞다.

### 9-3. OpenShift / Red Hat 친화 환경

- **Buildah** 검토 가치가 높다.

### 9-4. Java / Go 서비스

- Java 서비스는 **Jib**
- Go 서비스는 **ko**

이렇게 서비스별로 Kaniko를 제거하는 편이 전체 플랫폼 전환보다 더 실용적일 수 있다.



## 10. 최종 판단

Jenkins on Kubernetes 기준으로 정리하면:

1. **Kaniko를 급히 제거해야 할 정도는 아니다.**
2. 다만 **신규 표준으로는 BuildKit 쪽이 더 유리하다.**
3. **플랫폼화가 목적이면 kpack**이 맞다.
4. **특정 언어 서비스는 Jib/ko가 최적**일 수 있다.
5. **레지스트리 보안은 Jenkins Credentials만으로 끝나지 않으며, 빌드 Pod 권한 기준으로 다시 설계해야 한다.**

즉, 가장 현실적인 권고안은 아래와 같다.

- 현재 Kaniko 운영 중: 당장 유지
- 신규 서비스/신규 표준: BuildKit 우선 검토
- Java/Go 특화 서비스: Jib/ko 우선 검토
- 플랫폼 빌드 서비스화: kpack 검토
