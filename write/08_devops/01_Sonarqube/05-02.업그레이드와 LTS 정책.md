# 업그레이드와 LTS 정책

---

> SonarQube 업그레이드의 본질은 DB 마이그레이션이다. LTS 트랙을 알면 일정 잡기가 단순해지고, 마이그레이션 절차를 이해하면 다운타임이 예측 가능해진다.



## 1. LTS 트랙의 의미

> 2024년부터 SonarSource는 LTS 트랙을 명시적으로 운영한다. 운영 인스턴스는 LTS Active를 따라가는 게 표준이다.

세 트랙의 정의는 다음과 같다.

| 트랙 | 지원 기간 | 패치 정책 | 적합 운영 |
|------|----------|----------|----------|
| LTS Active | 2년 | 보안 + 버그 수정 | 안정성 우선의 운영 인스턴스 |
| LTS Previous | 직전 LTS, 6~9개월 병행 | 보안 패치만 | 다음 LTS 갈아타기 전까지 유지 |
| Latest | 매월 또는 분기 | 신기능 + 버그 수정 | 신기능 검증, SaaS, 서브 인스턴스 |

LTS Active 채택의 이유는 단순하다 — 신기능에 의존하지 않는 분석 도구라서 안정성이 압도적인 가치다. SonarQube에서 신기능이 운영 차단 요소를 만드는 사례는 드물고, 보통은 새 보안 Rule, 새 언어 분석기 같은 점진 개선이다. LTS Active도 이런 개선을 받는다.

### 1.1 LTS 갈아타기 시점

새 LTS가 출시되면 기존 LTS는 LTS Previous로 강등되고 6~9개월 병행 지원을 받는다. 이 기간 안에 갈아타는 게 표준이다. 강등 후에도 보안 패치는 계속 받지만 새 버그 수정은 끊기므로, 시간이 갈수록 운영 위험이 누적된다.

조직의 정책에 따라 다음 패턴이 흔하다.

- 새 LTS 출시 직후 — 알려진 이슈가 많아 곧바로 옮기지 않음. 1~2 패치 버전 기다림.
- 새 LTS 출시 후 1~2개월 — 패치 버전 확인 후 이전 시작.
- LTS Previous 지원 종료 1~2개월 전 — 마지막 갈아타기 윈도.

이 윈도들 사이를 활동 시간 외에 잡으면 다운타임 영향이 작다.



## 2. 메이저 업그레이드의 본질 — DB 마이그레이션

> 모든 SonarQube 업그레이드는 DB 스키마 변경을 동반한다. 마이그레이션 시간이 다운타임 산정의 출발이다.

업그레이드 절차의 표준은 다음과 같다.

1. SonarQube 정지.
2. DB 백업 (이게 빠지면 회복 불가능 사고).
3. ES 인덱스 백업 또는 삭제 (DB로부터 재생성 가능).
4. 새 버전 설치 (압축 해제 또는 새 이미지 적용).
5. SonarQube 재시작 — DB 스키마 마이그레이션 자동 시작.
6. UI에 "Setup is required" 화면이 뜨면 `/setup`에서 마이그레이션 트리거.
7. 마이그레이션 완료 후 ES 재인덱싱 (인덱스를 지웠을 경우).
8. `/api/system/status` 폴링 → `UP` 도달 확인.

각 단계의 시간 분포는 인스턴스 크기에 따라 다르다.

| 단계 | 작은 인스턴스 (이슈 1만) | 큰 인스턴스 (이슈 100만) |
|------|------------------------|----------------------|
| DB 백업 | 1분 | 30분~2시간 |
| 새 버전 설치 | 5분 | 5분 |
| DB 마이그레이션 | 1~5분 | 30분~3시간 |
| ES 재인덱싱 | 1~5분 | 30분~2시간 |

큰 인스턴스에서 6시간 다운타임이 가능한 작업이라는 것이 운영 일정의 핵심 사실이다. 주말 또는 야간 작업창이 필수다.

### 2.1 마이그레이션 동기성

DB 스키마 마이그레이션은 동기적으로 끝나야 다음 단계로 갈 수 있다. SonarQube는 마이그레이션 도중에 분석 요청을 받지 않는다. UI도 "Setup is required" 화면이 뜬다. 이 화면이 사라질 때까지 어떤 사용 시도도 의미가 없다.

마이그레이션 실패 시 SonarQube는 시작을 거부하고 로그에 실패 원인을 남긴다. 이 시점에서 DB는 부분 마이그레이션 상태일 수 있어 백업으로 롤백하는 게 안전하다. 부분 마이그레이션을 손으로 정리하는 시도는 거의 항상 더 큰 사고를 만든다.



## 3. 메이저 버전 건너뛰기

> SonarQube가 어떤 버전 조합에서 직접 업그레이드를 지원하는지가 일정 산정의 변수다.

버전 건너뛰기 정책은 SonarQube 버전마다 달라진다. 일반적인 규칙은 다음과 같다.

- LTS → 다음 LTS는 직접 업그레이드 가능 (예: 9.9 LTS → 10.x → 25.x LTS).
- 비-LTS 버전 사이는 인접 버전 단위로만 가능.

작은 인스턴스라면 한 번에 멀리 점프하는 게 유리하지만, 큰 인스턴스에서는 중간 버전을 거치는 게 안전하다. 한 번의 큰 마이그레이션이 실패하면 진단이 어렵고, 중간 버전들의 마이그레이션 결과를 차례로 검증하는 게 회귀 추적에 좋다.

### 3.1 호환성 매트릭스 확인

업그레이드 직전에 다음 매트릭스를 확인한다.

- 출발 버전 → 도착 버전이 직접 지원되는가?
- 출발 버전의 분석 결과(이슈, Profile, Gate)가 도착 버전에서 호환되는가?
- 사용 중인 Plugin이 도착 버전에서 동작하는가?

세 번째가 자주 부딪히는 함정이다. 사내 자체 Plugin이나 community Plugin은 SonarQube 메이저 버전 변경 시 호환되지 않는 경우가 흔하다. 업그레이드 전 모든 Plugin의 호환성을 확인한다.



## 4. 호환성 절벽 — 자주 부딪히는 셋

> 메이저 업그레이드에서 가장 자주 발생하는 호환성 문제 셋을 정리한다.

### 4.1 Java 런타임

SonarQube 자체가 요구하는 Java 버전이 메이저 버전마다 올라간다. 2025+ 버전은 Java 17 또는 21을 요구한다. 호스트의 JDK가 옛날 버전이면 SonarQube가 시작에 실패한다.

K8s/Helm 환경에서는 SonarSource 공식 이미지가 적절한 JDK를 동봉하므로 이 절벽이 거의 없다. 직접 설치 환경에서는 OS 패키지 매니저로 새 JDK를 설치하고 `JAVA_HOME`을 갱신해야 한다.

### 4.2 DB 종류 변경

가장 큰 절벽이다.

- H2 → PostgreSQL: 마이그레이션 도구가 따로 있고 한 번에 옮긴다. 업그레이드 시점에 함께 진행하는 게 흔하다.
- PostgreSQL → Oracle, Oracle → SQL Server 같은 횡 이동: 공식 지원 없음. 새 인스턴스를 띄우고 분석 이력을 포기하는 게 현실적.

DB 종류 변경은 업그레이드와 분리해서 단일 작업으로 진행한다. 두 변경을 동시에 시도하면 회귀 진단이 어렵다.

### 4.3 Scanner 호환성

신버전 SonarQube는 구버전 Scanner를 어느 정도 지원하지만, 신규 분석 기능은 신버전 Scanner를 요구한다. 업그레이드 후 다음 두 가지를 확인한다.

- CI 파이프라인의 Scanner 버전이 SonarQube 신버전 호환 범위 안인가.
- 새 분석에서 결과가 정상으로 나오는가 (이슈 추적이 깨지지 않았나).

Scanner 호환 매트릭스도 SonarSource 공식 docs가 제공한다. 업그레이드 직후 첫 분석 결과를 면밀히 본다.



## 5. 무중단 업그레이드 — 가능한가?

> SonarQube의 단일 인스턴스 모델에서는 진정한 무중단 업그레이드가 불가능하다. Data Center Edition이 부분적 답을 가진다.

Community/Developer/Enterprise Edition은 단일 호스트에서 동작하므로 업그레이드는 다운타임을 동반한다. 다음 두 길이 부분적 답이다.

- **Blue/Green 배포**: 새 인스턴스를 다른 DB 복사본으로 띄우고 충분히 검증한 뒤 트래픽 전환. DB 복사본을 만들고 동기화하는 비용이 크다.
- **Data Center Edition**: 분산 노드 구성. Application 노드와 Search 노드를 따로 업그레이드 가능. 그러나 메이저 버전 마이그레이션은 여전히 모든 노드를 정지해야 한다.

대부분의 운영 환경에서 SonarQube는 "분석은 빌드 시점에만 필요한 도구"이므로 일시 중단이 사용자에게 직접 영향을 주지 않는다. 분석 결과 조회 UI가 잠시 안 보이는 것이 가장 큰 영향이다. 이 점이 SonarQube의 다운타임 정책을 다른 운영 시스템보다 관대하게 만든다.

### 5.1 Maintenance Window 운영

표준 패턴은 다음과 같다.

1. 업그레이드 1~2주 전 사용자 공지 — "X월 X일 22:00~02:00 업그레이드".
2. CI 파이프라인의 SonarQube 호출이 실패해도 빌드 자체가 실패하지 않게 분리 (이게 평소에도 좋은 설계).
3. 작업 시간에 분석 큐를 비우고 시작.
4. 새 버전 시작 후 1~2일 모니터링 — 분석 실패율, 큐 적체, 로그 에러.

3번이 운영 위험을 줄이는 핵심이다. 마이그레이션 직전에 큐가 비어 있으면 부분 마이그레이션 상태가 만들 수 있는 사고가 줄어든다.



## 6. Plugin 업그레이드와 호환성

> SonarQube 본체보다 Plugin이 호환성을 자주 깬다.

Plugin은 두 종류로 나뉜다.

- **공식 분석기 (sonar-java, sonar-javascript 등)** — SonarQube에 동봉되며 본체 업그레이드와 함께 갱신.
- **써드파티 Plugin** — Marketplace 또는 자체 빌드. 별도 호환성 관리 필요.

운영 환경에서 자주 발견되는 호환성 함정을 정리한다.

| Plugin 유형 | 흔한 호환성 문제 |
|-----------|----------------|
| 인증 (LDAP/SAML) | API 시그니처 변경으로 일부 메서드 누락 |
| 언어 분석기 (community) | 메이저 버전 차이로 Rule API 깨짐 |
| 자체 Plugin | SonarQube SDK 의존성 충돌 |

업그레이드 전에 다음 절차로 호환성을 확인한다.

1. SonarQube Marketplace에서 사용 중인 Plugin들의 호환 매트릭스 확인.
2. 매트릭스가 도착 버전에 OK인지 확인.
3. 자체 Plugin은 새 SonarQube SDK로 빌드 시도.

자체 Plugin이 호환되지 않으면 업그레이드 일정이 미뤄진다. 자체 Plugin 의존도가 높은 인스턴스에서 LTS 갈아타기가 늦어지는 게 흔한 패턴이다.



## 7. 정리

> 업그레이드의 핵심 사실은 다섯이다.

요약한다. SonarSource의 LTS Active 트랙(2년 지원)을 따라가는 게 운영 표준이다. 업그레이드의 본질은 DB 마이그레이션이며 큰 인스턴스에서 6시간 다운타임도 가능하므로 야간/주말 작업창이 필수다. 호환성 절벽은 Java 런타임, DB 종류 변경, Scanner 호환성 셋이 가장 자주 부딪힌다. 무중단 업그레이드는 단일 인스턴스 모델에서는 불가능하지만 SonarQube의 사용 패턴상 분 단위 다운타임이 사용자에게 직접 영향을 주지 않는다. 자체 Plugin 호환성은 LTS 갈아타기 일정의 변수가 자주 된다.
