# Coverage와 Duplication

---

> SonarQube가 Issue 외에 다루는 두 측정값을 본다. Coverage는 외부 도구의 결과를 흡수만 하고, Duplication은 자체 알고리즘으로 탐지한다는 비대칭이 본 문서의 출발점이다.



## 1. Coverage의 책임 분담

> SonarQube는 Coverage를 측정하지 않는다. 측정은 JaCoCo 같은 외부 도구가 하고, SonarQube는 결과 리포트를 흡수만 한다.

이 분담은 명확하다.

| 책임 | 도구 | 산출물 |
|------|------|-------|
| Coverage 측정 | JaCoCo, Cobertura, Istanbul, Coverage.py 등 | XML 또는 LCOV 리포트 |
| Coverage 통합/시각화 | SonarQube | 측정값(metric)과 라인별 색상 표시 |
| Coverage 게이트 | SonarQube Quality Gate | "신규 코드 Coverage ≥ 80%" 조건 |

SonarQube가 Coverage를 직접 측정하지 않는 이유는 효율이다. 단위 테스트 시 바이트코드 instrumentation이 이미 들어가 있는데, SonarScanner가 따로 instrumentation해서 다시 측정하는 것은 중복이고 시간 낭비다. 빌드 단계에서 한 번 측정하고 SonarQube가 그 결과를 받아오는 분담이 합리적이다.

### 1.1 흡수되는 리포트 형식

언어와 도구에 따라 SonarQube가 인식하는 리포트가 다르다.

- 자바 — JaCoCo XML (`build/reports/jacoco/test/jacocoTestReport.xml`)
- 자바스크립트/타입스크립트 — LCOV (`coverage/lcov.info`)
- 파이썬 — Coverage.py XML
- C#/.NET — VSTest TRX 또는 dotCover XML
- C/C++ — gcovr XML, Cobertura XML

리포트 경로는 분석 호출 시 다음 속성으로 알린다.

```bash
./gradlew sonar \
  -Dsonar.coverage.jacoco.xmlReportPaths=build/reports/jacoco/test/jacocoTestReport.xml
```

Gradle Scanner는 JaCoCo 플러그인이 적용돼 있으면 위 경로를 자동 추정한다. 04-01에서 본 사례다.

### 1.2 HTML 리포트는 인식하지 못한다

자주 부딪히는 함정이 HTML 리포트만 생성하는 구성이다. JaCoCo의 기본 리포트는 HTML이고, SonarQube는 HTML을 읽지 않는다. 다음 설정이 필수다.

```groovy
jacocoTestReport {
    reports {
        xml.required = true   // SonarQube가 읽을 수 있는 XML
        html.required = false // 또는 true (XML과 동시 가능)
    }
}
```

이 한 줄을 빠뜨리면 분석 결과의 Coverage가 0%로 나타난다. Gate가 통과하지 못하고 빌드가 실패한다. 첫 도입 시 흔한 실수다.



## 2. Coverage 측정값의 구조

> SonarQube는 흡수한 리포트를 여러 측정값으로 분해한다. 같은 데이터를 다른 각도로 보는 것이다.

SonarQube가 노출하는 Coverage 관련 측정값은 다음과 같다.

| 측정값 | 의미 | Quality Gate 사용 빈도 |
|--------|------|-----------------------|
| `coverage` | 라인 + 분기 결합 커버리지 (%) | 가장 흔함 — Sonar way 기본 |
| `line_coverage` | 라인 커버리지만 (%) | 흔함 |
| `branch_coverage` | 분기 커버리지만 (%) | 가끔 |
| `lines_to_cover` | 커버 가능한 라인 수 (실행 가능 라인) | 거의 안 씀 |
| `uncovered_lines` | 미커버 라인 수 | 가끔 |
| `new_coverage` | 신규 코드의 결합 커버리지 (%) | Sonar way 기본 |
| `new_lines_to_cover` | 신규 코드의 커버 가능 라인 수 | 가끔 |
| `new_uncovered_lines` | 신규 코드의 미커버 라인 수 | 흔함 |

`coverage`는 라인 커버리지와 분기 커버리지를 결합한 지표다. 공식은 다음과 같다.

```
coverage = (covered_lines + covered_branches) / (total_lines + total_branches) × 100
```

이 결합 지표가 기본인 이유는 분기 커버리지가 라인 커버리지보다 엄격해서다. 한 줄에 `if`가 두 개 있으면 라인 커버리지는 50%만 채워도 라인이 커버됐다고 보지만, 분기 커버리지는 두 분기를 모두 실행해야 한다. 결합 지표는 둘의 평균에 가까운 값을 준다.

### 2.1 Coverage 0%로 보이는 흔한 원인 다섯 가지

운영에서 가장 자주 보고되는 함정이다.

1. **JaCoCo XML 리포트 미생성** — `xml.required = true` 누락.
2. **리포트 경로 오류** — Scanner가 기대하는 경로와 실제 생성 경로가 다름.
3. **테스트 미실행** — `gradle sonar` 호출 전에 `gradle test`가 안 돌았음.
4. **분석 시점 문제** — 빌드 캐시로 인해 테스트가 스킵됐는데 리포트는 옛날 것.
5. **multi-module 경로 누락** — 멀티 모듈에서 모듈별 리포트 경로를 모두 명시하지 않음.

5번이 멀티 모듈 프로젝트에서 자주 일어난다. 다음과 같이 모듈별 경로를 모두 알려야 한다.

```groovy
sonar {
    properties {
        property "sonar.coverage.jacoco.xmlReportPaths",
            subprojects.collect { "${it.buildDir}/reports/jacoco/test/jacocoTestReport.xml" }.join(",")
    }
}
```



## 3. Coverage Gate의 운영적 의미

> "신규 코드 Coverage ≥ 80%"라는 게이트가 어떤 행동을 강제하는지를 본다.

Sonar way 기본 게이트의 Coverage 조건은 "new_coverage ≥ 80%"다. 신규 코드의 결합 커버리지가 80% 미만이면 게이트 실패다. 이 조건이 강제하는 행동은 다음과 같다.

- 새 코드를 추가하는 PR은 그 코드의 80% 이상에 대한 단위 테스트를 함께 추가해야 한다.
- 테스트 없는 PR은 머지가 차단된다.
- 단순 리팩터링(기존 코드를 옮기는 PR)은 새 코드가 적으므로 영향이 작다.

이 조건이 의미 있는 이유는 1장에서 본 Clean as You Code 정책과 일치한다. 기존 코드의 Coverage는 무관심하지만 새 코드는 깨끗하게 시작한다는 정책이다.

### 3.1 80%가 적절한가

80%라는 수치는 SonarSource의 기본 권장이지만 조직마다 조정한다. 결정의 축은 두 가지다.

- 너무 높으면 (90%+): 개발자가 의미 없는 테스트(getter 호출 같은)를 추가해 수치만 채우는 부작용.
- 너무 낮으면 (50%): 게이트가 거의 통과되므로 게이트 자체의 의미가 약해짐.

70~80% 범위가 일반적이다. 단, "신규 코드 미커버 라인 수 ≤ 5" 같은 절대값 조건을 추가하면 작은 PR에서 비율 게임이 깨지는 것을 막을 수 있다 (5줄 PR에서 1줄 미커버는 80% 통과지만 5줄 미커버는 게이트 실패).



## 4. Duplication — SonarQube가 직접 탐지

> Coverage와 달리 Duplication은 SonarQube 자체 알고리즘으로 탐지한다. 외부 도구가 끼지 않는다.

Duplication은 같은 코드 블록이 여러 위치에 반복된 것을 찾는다. SonarQube의 탐지 알고리즘은 다음 단계다.

1. 소스 코드를 토큰 단위로 분해 (식별자, 키워드, 리터럴).
2. 변수 이름·문자열 리터럴을 정규화 (이름 차이는 무시).
3. 일정 길이 이상의 토큰 시퀀스가 같으면 중복으로 등록.

기본 임계값은 언어별로 다르며 자바는 100토큰이다. 100토큰 이상의 동일 시퀀스가 두 곳 이상에 있으면 중복이다.

### 4.1 토큰 기반의 의미

토큰 기반이라는 사실이 의미하는 바가 있다. 다음 둘은 같은 중복으로 잡힌다.

```java
// File A
String userName = request.getName();
if (userName == null) {
    throw new ValidationException("name required");
}

// File B
String customerName = req.getName();
if (customerName == null) {
    throw new ValidationException("name required");
}
```

식별자(`userName`/`customerName`)와 변수 출처(`request`/`req`)가 다르지만 토큰 시퀀스 구조가 같으면 중복으로 잡힌다. 변수 이름만 바꾼 복사-붙여넣기를 잡는 데 효과적이다.

반대로 의미적으로 같지만 표현이 다른 코드는 잡지 못한다. `for` 루프와 `stream()` 체인이 같은 일을 해도 토큰 구조가 달라 중복으로 인식되지 않는다. Duplication 탐지는 의미 분석이 아닌 형태 분석이라는 한계를 가진다.

### 4.2 측정값과 게이트 조건

Duplication 관련 측정값은 다음과 같다.

| 측정값 | 의미 |
|--------|------|
| `duplicated_lines` | 중복으로 식별된 라인 수 |
| `duplicated_lines_density` | 중복 라인 비율 (%) |
| `duplicated_blocks` | 중복 블록 개수 |
| `new_duplicated_lines_density` | 신규 코드의 중복 라인 비율 (%) |

Sonar way 기본 게이트의 Duplication 조건은 "new_duplicated_lines_density ≤ 3%"다. 신규 코드의 중복 비율이 3% 이하여야 한다. 3%라는 수치는 합리적인 default로 간주되지만, 조직에 따라 5%까지 완화하는 경우도 있다.

### 4.3 Duplication 탐지가 잡지 못하는 진짜 중복

토큰 기반의 한계를 보완하는 운영 패턴은 코드 리뷰다. SonarQube가 잡는 중복은 "복사-붙여넣기 흔적"이며, 의미적 중복(추상화 누락, 유사 로직)은 사람이 잡아야 한다. 이 분리를 인정하면 Duplication 게이트는 형태 중복만 막는 자동 장치이고, 의미 중복은 PR 리뷰가 책임진다.



## 5. 다른 측정값들

> Issue·Hotspot·Coverage·Duplication 외에도 SonarQube는 다양한 측정값을 노출한다. 자주 쓰이는 것을 정리한다.

운영에서 자주 보는 측정값은 다음과 같다.

| 측정값 | 영역 | 의미 |
|--------|------|------|
| `cognitive_complexity` | Maintainability | 인지 복잡도. 사람이 코드를 읽기 어려운 정도 |
| `complexity` | Maintainability | 순환 복잡도 (McCabe) |
| `ncloc` | 규모 | 주석/공백 제외 라인 수 |
| `comment_lines_density` | 문서화 | 주석 비율 (%) |
| `sqale_rating` | Maintainability | 유지보수성 등급 (A~E) |
| `reliability_rating` | Reliability | 신뢰성 등급 (A~E) |
| `security_rating` | Security | 보안 등급 (A~E) |

A~E 등급은 절대 수치를 등급으로 환산한 것이다. 예를 들어 Maintainability Rating은 코드 부채 비율로 결정된다 — 부채가 5% 미만이면 A, 5~10%면 B, 그 이상은 C/D/E. Quality Gate 조건은 흔히 "신규 코드의 등급 = A"로 표현되며, 이는 신규 코드에 새 부채를 거의 만들지 말라는 정책이다.

### 5.1 측정값 조회 — Web API

특정 프로젝트의 측정값은 `/api/measures/component`로 조회한다. operator/cicd 사례에서 이미 봤다.

```
GET /api/measures/component?component=org.okestro:pipeline-api&metricKeys=coverage,bugs,vulnerabilities
```

쉼표 구분으로 여러 측정값을 한 번에 받을 수 있다. 멀티 모듈에서 모듈 트리를 한 번에 보고 싶으면 `/api/measures/component_tree`를 쓴다. 04-02에서 본 13개 엔드포인트 중 둘이 이 역할이다.



## 6. 정리

> Coverage와 Duplication은 같은 분석 결과의 두 측면이지만 책임 모델이 다르다.

요약한다. SonarQube는 Coverage를 측정하지 않고 외부 도구(JaCoCo 등)의 XML/LCOV 리포트를 흡수한다. 가장 흔한 함정은 HTML만 생성하고 XML을 빠뜨리는 것이다. Coverage 게이트는 "신규 코드 ≥ 80%"가 표준이며 Clean as You Code 정책의 구현이다. Duplication은 SonarQube가 직접 탐지하며 토큰 시퀀스 기반 알고리즘이다. 의미적 중복은 잡지 못한다는 한계를 코드 리뷰가 보완한다. Maintainability/Reliability/Security Rating은 측정값을 등급으로 환산한 것이며 Gate 조건에 자주 쓰인다.

이 문서로 2장이 끝난다. 분석 모델 운영의 핵심 통찰을 빠르게 점검하려면 [02-점검.핵심 질문과 답](02-점검.핵심 질문과 답.md)으로 간다. 분석 결과를 외부에서 조회·푸시하는 인터페이스가 궁금하다면 3장(Web API와 Webhook)으로 간다.
