# 3단계 점검 — SonarQube 운영 인터페이스 핵심 질문

---

> 다루는 문서 - `03-01.Web API 구조`, `03-02.인증과 토큰`, `03-03.Webhook과 이벤트`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다.



## Q1: SonarQube Web API의 검색 응답이 paging + 데이터 + 참조 데이터 세 부분으로 구성되는 이유는?

N+1 호출을 막기 위해서다. 예를 들어 100건의 Issue가 50개 파일에 흩어져 있을 때, 클라이언트가 components 정보를 위해 50번 추가 호출을 한다면 응답 시간과 인스턴스 부하가 폭증한다. SonarQube는 데이터 배열에서 참조하는 component·rule 같은 엔티티의 메타를 같은 응답 안에 동봉해 클라이언트가 한 번의 호출로 모든 정보를 받게 한다. paging 구조는 페이지 인덱스(1부터 시작)·페이지 크기·전체 개수를 제공해 추가 페이지 호출 여부를 클라이언트가 결정할 수 있게 한다. 이 구조는 RESTful 설계의 표준 중 하나이며 외부 통합 코드의 매핑 비용을 크게 줄인다.

상세: `03-01.md` §2.

### 심화

응답에 참조 데이터를 동봉하는 패턴이 작은 응답에서는 오버헤드를 만든다. 페이지 사이즈와 참조 데이터 동봉의 효율적 균형을 어떻게 잡을 수 있는가?



## Q2: SonarQube `/api/issues/search`의 10000개 결과 한계를 우회하는 방법은?

검색 조건을 좁혀 다중 호출로 분할한다. severity·type·componentKeys·createdBefore 같은 필터로 한 번의 검색 결과를 10000개 이내로 줄인 뒤, 분할된 결과를 클라이언트가 합친다. 예를 들어 severity별로 5번 호출하면 한 번에 받을 수 있는 누적량이 5배가 된다. 또는 일부 엔드포인트가 지원하는 cursor 페이지네이션(SCM 시각으로 정렬 + `createdBefore`)을 쓸 수도 있지만, 인스턴스 버전별로 동작이 다르므로 docs 확인이 필요하다. 한계 자체는 인스턴스 부하 보호용이라 우회 불가능하다 — 받아들이고 분할하는 게 안전한 답이다.

상세: `03-01.md` §3, §3.1.

### 심화

검색 조건 분할로 결과를 합치는 클라이언트 코드는 같은 Issue가 두 번 등장할 위험을 가진다. 어떤 조건 조합이 중복을 만들고 어떻게 막을 수 있는가?



## Q3: SonarQube의 변경 엔드포인트가 form-urlencoded 본문을 받고 JSON을 받지 않는 이유는?

GET의 query string과 POST의 form body 형식을 일치시키기 위해서다. SonarQube Web API는 같은 엔드포인트를 GET이나 POST 둘 다로 받을 수 있는 경우가 있고(SonarQube 자체 UI도 두 형식을 모두 사용), 두 메소드가 같은 파라미터 형식을 공유하는 게 클라이언트 구현을 단순하게 한다. 라이브러리 선택의 영향이 있다 — Spring `RestTemplate`의 `@RequestParam`이나 OpenFeign은 자연스럽게 매칭되지만, JSON 우선 클라이언트(예: 기본 설정의 `WebClient`)는 form-urlencoded 인코딩을 명시해야 한다. operator/cicd가 OpenFeign을 쓴 이유 중 하나가 이 자연스러운 매칭이다.

상세: `03-01.md` §5.

### 심화

같은 파라미터를 GET query string과 POST form body 둘 다로 받는 SonarQube 엔드포인트에서, 캐싱 정책이나 보안(파라미터 노출) 측면에서 어느 메소드를 선택해야 하는 기준은 무엇인가?



## Q4: User Token, Project Analysis Token, Global Analysis Token의 폭발 반경 차이가 CI 시크릿 결정에 어떻게 작용하는가?

토큰 종류 선택은 사고 시 노출 범위를 미리 결정하는 일이다. User Token이 유출되면 그 사용자의 모든 권한이 통째로 노출되며, 관리자 사용자라면 인스턴스 전체 통제권이 사라진다. Global Analysis Token이 유출되면 모든 프로젝트의 분석 결과를 위변조할 수 있다. Project Analysis Token이 유출되면 한 프로젝트만 영향받는다. CI 파이프라인에 박을 토큰은 Project Analysis Token이 기본이며, 한 빌드 머신이 여러 프로젝트를 다룰 때만 Global Analysis Token으로 합친다. 그 경우 토큰 회전 정책을 더 엄격하게 운영한다. User Token을 자동화에 쓰는 것은 안티패턴 — 사람과 자동화의 신원을 분리하지 못한 결과다.

상세: `03-02.md` §2.

### 심화

CI 시크릿 매니저가 단일 SonarQube 토큰으로 모든 빌드를 처리하는 단순 모델과, 프로젝트별 다중 토큰으로 분리하는 모델 중 어느 쪽이 운영 비용 대비 안전한가? 어떤 임계점에서 모델이 바뀌는가?



## Q5: 외부 인증(LDAP/SAML)을 쓰는 환경에서도 SonarQube 토큰은 자체 메커니즘으로 살아 있는다는 사실이 만드는 위험은?

퇴사자의 외부 인증 계정이 막혀도 그 사람이 발급한 SonarQube 토큰은 별도 revoke 전까지 계속 작동한다. 토큰은 SonarQube DB에 저장되며 LDAP/SAML 비밀번호와 무관하다. 퇴사 프로세스에 "SonarQube 토큰 revoke"가 명시되지 않으면 사고가 잠재된다. 운영 패턴은 정기 토큰 회전(90일 또는 180일)을 강제하는 것이다 — 회전이 강제되면 퇴사 90일 후에는 옛 토큰이 자동 만료된다. 즉시 차단이 필요한 보안 사고에서는 SonarQube 관리자가 명시적으로 revoke하고, 일반 퇴사 케이스는 회전 주기로 흡수한다.

상세: `03-02.md` §5.2.

### 심화

조직의 LDAP/SAML 통합 도구가 토큰 revoke API를 자동 호출할 수 있다면 어떤 형태가 가장 안전한가? 퇴사 트리거가 토큰 revoke를 직접 부르는 게 좋은가, 아니면 회전 주기에 맡기는 게 좋은가?



## Q6: SonarQube Webhook이 분석 종료 이벤트 한 종류만 다루는 이유는?

Webhook은 외부 시스템이 분석 결과를 알아야 하는 시나리오에 특화돼 있고, 다른 이벤트(이슈 생성, 사용자 추가, Profile 변경 등)는 운영 감사나 자체 UI에서 다뤄진다. 분석 종료가 가장 자주 외부 트리거를 만들어 내는 이벤트이며 — CI 빌드 결과 결정, 외부 알림, 다른 시스템의 워크플로우 진행 — Webhook의 의미가 거기 집중된다. 다른 이벤트가 필요한 경우 Audit Log API(Enterprise+)나 Web API 폴링을 쓴다. 이 단순화가 Webhook 모델을 강하게 만든다 — 한 종류 이벤트만 다루므로 페이로드 골격이 단순해지고 수신자 코드가 분기 없이 깔끔해진다.

상세: `03-03.md` §1.

### 심화

같은 SonarQube 인스턴스가 여러 외부 시스템(예: Slack, GitLab, Jira)과 통합돼 모두 분석 종료 이벤트를 받아야 한다면, Webhook을 시스템마다 등록하는 게 좋은가, 한 수신자가 fan-out하는 게 좋은가?



## Q7: HMAC-SHA256 시그니처 검증을 일반 `==` 비교가 아닌 `compare_digest`로 해야 하는 이유는?

타이밍 공격(timing attack)을 방어하기 위해서다. 일반 문자열 비교는 첫 다른 문자에서 즉시 false를 반환하므로 비교 시간이 입력값에 따라 달라진다. 공격자는 비교 시간을 측정해 시그니처를 한 글자씩 맞춰 갈 수 있다. `compare_digest`는 두 문자열의 모든 위치를 항상 비교하는 상수 시간 비교이므로 비교 시간이 입력값과 무관하다. 시그니처 검증은 보안이 본질인 코드라 이런 미세한 차이가 의미 있는 위험을 만든다. 사내 폐쇄망이라도 측면 이동(lateral movement) 공격 모델에서는 이런 디테일이 차이를 만든다.

상세: `03-03.md` §4.

### 심화

같은 원칙이 적용되는 다른 보안 비교 시나리오는 무엇이며, 라이브러리가 제공하는 상수 시간 비교 함수의 이름 패턴(예: `compare_digest`, `MessageDigest.isEqual`)에 어떤 공통점이 있는가?



## Q8: SonarQube Webhook의 best-effort 전달 모델이 수신자 설계에 미치는 영향은?

수신자가 빠르고 안정적이어야 한다는 제약을 만든다. 30초 이상 응답하지 않으면 SonarQube가 timeout으로 처리하고, 재시도가 한 번 또는 0번 시도된 뒤 포기한다. 수신자가 무거운 작업을 동기로 처리하면 페이로드 손실이 발생한다. 표준 골격은 수신과 처리를 분리하는 것이다 — 수신 측이 시그니처 검증 후 페이로드를 큐(Kafka, Redis 등)에 즉시 적재하고 200을 반환하며, 별도 워커가 큐에서 꺼내 실제 처리한다. 이 분리가 있으면 100ms 이내 응답이 가능하고, 워커 처리가 늦어도 SonarQube에 영향이 없으며, 큐의 재시도로 처리 신뢰성도 별도로 확보된다.

상세: `03-03.md` §5, §5.1.

### 심화

큐로 분리하는 패턴이 표준이지만, 작은 운영 환경에서는 큐 자체가 새 운영 부담이 된다. 큐 없이 동기 처리를 유지하면서 신뢰성을 확보하는 절충 패턴이 가능한가?
