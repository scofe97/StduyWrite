# 2단계 점검 — SonarQube 분석 모델 핵심 질문

---

> 다루는 문서 - `02-01.Rule과 Quality Profile`, `02-02.Issue와 Security Hotspot`, `02-03.Coverage와 Duplication`. 답을 떠올린 뒤 본문 해당 섹션과 비교한다.



## Q1: Rule을 비활성화하기 전에 시도해 볼 수 있는 세 가지 대안은?

이슈 단위 처리, Rule 파라미터 조정, scope 제한이다. 이슈 단위 처리는 개별 Issue를 "False Positive" 또는 "Won't Fix"로 분류하는 것으로, Rule은 살아 있고 다른 위치의 진짜 위반은 계속 잡힌다. Rule 파라미터 조정은 임계값을 가진 Rule(예: 인지 복잡도 임계값)의 default 값을 조직에 맞게 조정한다. scope 제한은 `sonar.exclusions` 또는 Rule별 `sonar.issue.ignore.multicriteria`로 특정 경로/패턴을 분석에서 빼는 방식이다. Rule 비활성화는 한 위반 패턴 전체를 잊는 결정이며 셋 중 어느 것도 안 될 때의 마지막 선택지여야 한다.

상세: `02-01.md` §3.1.

### 심화

`WONTFIX`로 종결한 Issue가 시간이 지나면서 의미를 잃는 흔한 문제를, 코드 베이스 차원에서 어떻게 모니터링할 수 있는가? 정기 검토 외의 자동화 가능성은?



## Q2: Sonar way Profile을 그대로 쓰는 것과 복제 후 커스터마이징하는 것의 트레이드오프는?

자동 갱신 vs 통제권의 대비다. Sonar way를 그대로 쓰면 SonarQube 업그레이드 시 새 보안 규칙이나 권장 severity 변경을 자동으로 받는다. 손대지 않은 만큼 운영 비용이 작다. 반대로 복제 후 커스터마이징하면 조직 의도가 명확해지고 의도하지 않은 게이트 변경이 일어나지 않지만, 새 SonarQube 버전이 추가하는 신규 Rule을 자동으로 받지 못한다. 정기적으로 Sonar way와의 차이를 검토해 신규 Rule을 옮겨오는 운영 작업이 필요하다. 절충은 상속(extend) 패턴이다 — Sonar way를 부모로 두고 차이점만 정의하면 부모 갱신을 자동 상속하면서 자체 정책을 함께 표현할 수 있다.

상세: `02-01.md` §2.1, §2.2.

### 심화

상속 Profile에서 부모의 Rule이 비활성화되면 자식이 그 Rule을 다시 활성화할 수 있는가? 이런 부모-자식 충돌이 운영에서 어떻게 추적·해결되는가?



## Q3: SonarQube가 같은 결함을 같은 Issue로 추적하는 알고리즘이 깨지는 흔한 시나리오는?

대규모 리포맷팅, 메서드 이름 변경 + 위치 이동, 언어 분석기 메이저 업그레이드의 셋이 가장 흔하다. 추적 알고리즘은 `(rule, message, line)` 1차 매칭 후 코드 라인 내용까지 동원하지만, 라인 정보와 코드 내용이 동시에 바뀌면 매칭이 끊긴다. 추적이 깨지면 기존 Issue가 CLOSED로 전환되고 새 Issue가 OPEN으로 생기며 `creationDate`가 새 시점으로 리셋된다. "30일 추적"이 0일로 돌아간다는 부작용이 있다. 대규모 리포맷팅을 단일 PR로 분리하고 메시지에 명시하면 운영자가 추적 끊김을 사전 인지할 수 있다.

상세: `02-02.md` §2, §2.1.

### 심화

추적이 깨진 Issue를 SonarQube가 사후에 다시 묶을 수는 없다. 그럼에도 운영 보고서에서 "옛 결함과 새 결함을 한 묶음으로 보고 싶다"는 요구가 있을 때, Web API 차원에서 어떤 보완이 가능한가?



## Q4: RESOLVED의 세 사유(FIXED / FALSE-POSITIVE / WONTFIX) 중 사용에 가장 신중해야 하는 것은 무엇이며 왜인가?

`FALSE-POSITIVE`와 `WONTFIX` 둘 다 신중하다. 두 사유는 "같은 결함이 다시 발견되도 무시"라는 의미를 가지므로 한 번 분류하면 그 결함은 영원히 무시된다. `FIXED`는 다음 분석에서 검증된다 — 결함이 사라지면 CLOSED로, 다시 발견되면 REOPENED로 자동 전환된다. 그래서 `FIXED`는 잘못 써도 자동 회복이 되지만 `FALSE-POSITIVE/WONTFIX`는 한 번의 잘못된 결정이 영구 적용된다. 특히 `WONTFIX`는 Quality Gate가 그 Issue를 카운트하지 않으므로 빨간불을 끄기 위해 남용될 위험을 가진다. 정기적인 "WONTFIX Issue 재검토" 워크플로우가 필요하다.

상세: `02-02.md` §3.1.

### 심화

`WONTFIX` 남용을 막기 위한 게이트 조건을 추가로 만들 수 있는가? 예를 들어 "신규 코드에서 WONTFIX 비율 ≤ 5%" 같은 제약이 의미가 있는가?



## Q5: Security Hotspot이 일반 Vulnerability와 다른 점은?

Vulnerability는 도구가 "확실히 위험하다"고 판정한 결함이고, Hotspot은 "위험할 수 있지만 컨텍스트를 봐야 안다"는 검토 요청이다. 예를 들어 약한 해시 알고리즘 사용은 보안 목적이면 위험하지만 빠른 체크섬 용도라면 정상이다. Hotspot은 자동 OPEN/CLOSED 전환이 아닌 사람의 명시적 검토를 요구한다. 라이프사이클이 TO_REVIEW → REVIEWED with SAFE/FIXED/ACKNOWLEDGED로 전환되며 Quality Gate 조건은 흔히 "신규 코드의 Hotspot 검토율 = 100%"다. 이 분리를 섞으면 도구가 자동화하는 영역과 사람이 결정하는 영역의 경계가 흐려진다.

상세: `02-02.md` §5, §5.1, §5.2.

### 심화

같은 SonarQube 인스턴스의 다른 두 프로젝트가 같은 Hotspot 패턴(예: 약한 해시 사용)을 발견했고 한 프로젝트에서는 SAFE, 다른 프로젝트에서는 FIXED로 검토했다면, 운영 차원에서 이 차이를 어떻게 해석해야 하는가?



## Q6: SonarQube가 Coverage를 직접 측정하지 않고 외부 도구의 리포트를 흡수만 하는 이유는?

효율과 책임 분담이다. 단위 테스트 시 JaCoCo 같은 도구가 이미 바이트코드 instrumentation으로 측정을 마쳤고, SonarScanner가 따로 instrumentation해서 다시 측정하는 것은 중복이고 시간 낭비다. 빌드 단계에서 한 번 측정하고 SonarQube가 결과를 받아오는 분담이 합리적이다. 이 분담의 가장 흔한 함정은 JaCoCo가 HTML 리포트만 생성하고 XML을 빠뜨리는 것이다 — SonarQube는 XML만 인식하므로 Coverage가 0%로 표시된다. `xml.required = true` 한 줄을 잊으면 Quality Gate가 빨간불로 막히며, 첫 도입 시 가장 자주 보고되는 사고다.

상세: `02-03.md` §1, §1.2, §2.1.

### 심화

멀티 모듈 Gradle 빌드에서 모듈별 JaCoCo 리포트를 모두 SonarQube가 인식하게 하려면 어떤 추가 설정이 필요한가? 단일 모듈과 비교했을 때의 운영 비용 차이는?



## Q7: SonarQube의 Duplication 탐지 알고리즘이 잡는 것과 잡지 못하는 것의 경계는?

토큰 기반이라는 점이 경계다. 알고리즘은 소스를 토큰으로 분해하고 변수 이름·문자열 리터럴을 정규화한 뒤 일정 길이(자바 100토큰) 이상의 동일 시퀀스를 중복으로 본다. 변수 이름만 바꾼 복사-붙여넣기는 잘 잡히지만, 의미는 같은데 표현이 다른 코드(예: `for` 루프 vs `stream()` 체인)는 잡지 못한다. Duplication 탐지는 의미 분석이 아닌 형태 분석이라는 한계를 가진다. 이 한계를 보완하는 운영 패턴은 코드 리뷰다 — SonarQube가 형태 중복을 자동으로 막고, 의미 중복은 사람이 PR 리뷰에서 잡는다는 분담이다.

상세: `02-03.md` §4, §4.1, §4.3.

### 심화

토큰 임계값을 100보다 낮게 잡으면 어떤 부작용이 생기는가? 예를 들어 50토큰으로 낮춘다면 false positive가 어떤 종류로 늘어나며 그 비용을 어떻게 평가할 수 있는가?



## Q8: Maintainability/Reliability/Security Rating이 A~E 등급으로 표현되는 이유는?

운영자와 비개발자에게 한눈에 들어오는 라벨을 제공하기 위해서다. 절대 수치(예: 부채 비율 7.3%)는 의미를 추가 해석해야 하지만, A/B/C/D/E는 즉시 좋고 나쁨을 알린다. 등급은 절대 수치를 임계값으로 변환한 것이며 SonarSource가 default를 정해 둔다. 예를 들어 Maintainability Rating은 부채 비율 5% 미만이면 A, 5~10%면 B, 그 이상은 C/D/E다. Quality Gate 조건은 흔히 "신규 코드의 등급 = A"로 표현되며 이는 신규 코드에 새 부채를 거의 만들지 말라는 정책의 정량 표현이다. 등급 임계값 자체는 인스턴스 단위로 조정 가능하지만 거의 default를 그대로 쓴다.

상세: `02-03.md` §5.

### 심화

조직이 등급 임계값을 default와 다르게 잡아야 하는 시나리오가 존재하는가? 예를 들어 레거시가 많은 조직과 그린필드 조직에서 임계값을 다르게 잡는 게 의미 있는가?
