# Web API 구조

---

> SonarQube의 외부 진입점 중 가장 풍부한 것이 Web API다. `/api/*` 네임스페이스의 구조와 응답 모델, 페이지네이션 패턴을 잡는다.



## 1. API 네임스페이스 — 무엇이 어디에 있는가

> SonarQube Web API는 `/api/<resource>/<action>` 형태로 일관된 명명 규칙을 가진다. 리소스 영역만 알면 엔드포인트 추정이 쉬워진다.

자주 쓰이는 리소스 영역을 정리한다.

| 영역 | 책임 | 대표 엔드포인트 |
|------|------|---------------|
| `/api/projects` | 프로젝트 메타 (생성/검색/삭제) | `search`, `create`, `delete` |
| `/api/project_branches` | 브랜치 목록 | `list`, `delete` |
| `/api/components` | 컴포넌트 트리 (모듈/디렉터리/파일) | `show`, `tree`, `search` |
| `/api/measures` | 측정값 조회 | `component`, `component_tree`, `history` |
| `/api/issues` | 이슈 검색·관리 | `search`, `assign`, `do_transition`, `set_severity` |
| `/api/sources` | 코드 스니펫 | `lines`, `issue_snippets`, `raw` |
| `/api/rules` | Rule 메타 | `search`, `show` |
| `/api/qualityprofiles` | Quality Profile 관리 | `search`, `activate_rule`, `deactivate_rule` |
| `/api/qualitygates` | Quality Gate 평가/조회 | `project_status`, `list`, `select` |
| `/api/webhooks` | Webhook 등록·관리 | `list`, `create`, `delete`, `update` |
| `/api/ce` | Compute Engine 작업 큐 | `task`, `activity` |
| `/api/system` | 시스템 상태 | `status`, `health`, `info` |
| `/api/user_tokens` | User Token 관리 | `generate`, `revoke`, `search` |
| `/api/permissions` | 권한 관리 | `add_user`, `remove_user`, `users` |

`/api/web_api`(또는 SonarQube UI의 `Web API` 메뉴)에서 모든 엔드포인트의 자체 문서가 제공된다. 인스턴스 버전에 정확히 맞는 명세이므로 정적 문서보다 신뢰할 수 있다.

### 1.1 endpoint 명명의 규칙성

엔드포인트 동사는 다음 패턴을 따른다.

- `search` — 검색 (페이지네이션 포함)
- `list` — 단순 목록 (페이지네이션 없거나 작음)
- `show` — 단건 상세
- `create` / `delete` / `update` — CRUD
- `do_transition` — 상태 전환 (Issue 라이프사이클 등)
- `activity` / `history` — 시간 축 데이터

이 패턴을 알면 처음 보는 엔드포인트도 대략 추정된다. 예를 들어 `/api/qualityprofiles/changelog`라면 Profile의 시간축 데이터일 가능성이 높다 — 실제로 그렇다.



## 2. 응답 모델 — JSON 한 자리 끝까지

> 모든 응답은 JSON이며 일관된 구조를 가진다. paging, errors, total 같은 메타가 본문 안에 들어 있다.

검색 계열 응답의 표준 골격은 다음과 같다.

```json
{
  "paging": {
    "pageIndex": 1,
    "pageSize": 100,
    "total": 543
  },
  "issues": [
    { "key": "AYz...", "rule": "java:S1186", "severity": "MAJOR", ... },
    ...
  ],
  "components": [
    { "key": "...", "name": "...", "qualifier": "FIL" }
  ],
  "rules": [
    { "key": "java:S1186", "name": "Methods should not be empty", ... }
  ]
}
```

세 부분으로 나뉜다.

- **paging**: 페이지 정보. `pageIndex` (1부터 시작), `pageSize` (한 페이지 항목 수), `total` (전체 항목 수).
- **데이터 배열**: 엔드포인트마다 다른 키 (`issues`, `components`, `rules` 등).
- **참조 데이터 배열**: 데이터에서 참조하는 엔티티의 메타. 예를 들어 issue 응답에 `components`와 `rules`가 함께 들어와 N+1 추가 호출을 막는다.

참조 데이터를 함께 반환하는 패턴이 효율적이다. Issue 검색 결과 100건이 50개 파일에 흩어져 있어도 components 배열은 50건만 들어온다. 클라이언트가 component 별로 추가 호출을 할 필요가 없다.

### 2.1 에러 응답

4xx/5xx 응답은 다음 형태다.

```json
{
  "errors": [
    { "msg": "Project key 'foo' not found" }
  ]
}
```

배열인 이유는 한 요청에 여러 에러가 동시에 발생할 수 있기 때문이다. 예를 들어 `/api/qualityprofiles/activate_rule`에 잘못된 Rule key를 여러 개 보내면 Rule마다 별도 에러가 들어온다.

operator/cicd 사례에서 본 매핑은 이 응답 구조를 도메인 어휘로 옮긴 것이다 — HTTP 상태 코드(400/401/403/404)와 errors 본문 메시지를 결합해 `SONARQUBE_*` 도메인 에러 코드로 분류한다.



## 3. 페이지네이션의 한계

> SonarQube Web API의 페이지네이션은 두 한계를 가진다. 페이지 크기 상한과 결과 개수 상한이다.

페이지 크기 상한은 500이다. `ps=1000`을 전달해도 SonarQube가 500으로 깎는다. operator의 `AnalysisReaderImpl`에서 `ISSUE_PAGE_SIZE = 500`이 고정값으로 들어 있는 이유다(04-02 §6에서 본 사실).

결과 개수 상한은 일부 엔드포인트(특히 `/api/issues/search`)에 10000개로 걸린다. 한 검색에서 10000개를 넘는 결과는 받을 수 없다. 큰 프로젝트에서 모든 Issue를 받고 싶다면 검색 조건(severity, type, component 등)을 좁혀 다중 호출로 나눠야 한다.

### 3.1 페이지네이션 패턴

큰 결과를 모두 받으려면 다음 패턴을 쓴다.

```
GET /api/issues/search?ps=500&p=1   → paging.total로 전체 개수 확인
GET /api/issues/search?ps=500&p=2   → 그 다음 페이지
... p가 10000/500=20에 도달할 때까지
```

`/api/issues/search`의 10000 한계 때문에 20페이지가 사실상 한도다. 그 이상이 필요하면 `componentKeys`나 `severities`로 검색을 분할한다.

대안 — 일부 엔드포인트는 `s` (sort) + 마지막 키를 활용한 cursor 페이지네이션을 지원한다. 검색 결과 10000 한계를 우회하려면 SCM 변경 시각으로 정렬 후 마지막 시각을 다음 호출의 `createdBefore`로 넘기는 방식이 가능하다. 단 운영 코드에서 쓰기 전에 인스턴스 버전의 docs를 한 번 확인한다.



## 4. 자주 쓰는 엔드포인트 — 운영자 관점

> 외부 시스템에서 SonarQube를 다룰 때 가장 자주 부르는 엔드포인트를 정리한다.

### 4.1 프로젝트 상태 조회 — `/api/qualitygates/project_status`

CI 파이프라인이 게이트 결과를 확인할 때 쓰는 1번 엔드포인트다.

```
GET /api/qualitygates/project_status?projectKey=org.okestro:pipeline-api&branch=main
```

응답:

```json
{
  "projectStatus": {
    "status": "OK",
    "ignoredConditions": false,
    "conditions": [
      {
        "status": "OK",
        "metricKey": "new_coverage",
        "comparator": "LT",
        "errorThreshold": "80",
        "actualValue": "85.3"
      }
    ]
  }
}
```

`status` 값은 `OK` / `WARN` / `ERROR` / `NONE` 중 하나다. CI는 `OK` 외의 값을 빌드 실패로 처리한다. 각 condition의 `status`도 확인하면 어느 조건이 실패했는지를 보고할 수 있다.

### 4.2 측정값 조회 — `/api/measures/component`

대시보드나 보고서에 측정값을 끌어다 쓸 때.

```
GET /api/measures/component?component=org.okestro:pipeline-api&branch=main&metricKeys=coverage,bugs,vulnerabilities,sqale_index
```

여러 측정값을 쉼표로 한 번에 받는다. 응답은 `component.measures` 배열로 들어온다.

### 4.3 Compute Engine 작업 상태 — `/api/ce/task`

분석 호출 후 Background Task 완료를 폴링할 때.

```
GET /api/ce/task?id=AYz...
```

응답의 `task.status`가 `IN_PROGRESS` / `PENDING` / `SUCCESS` / `FAILED` / `CANCELED` 중 하나다. 분석 후 게이트 결과를 받기 전 단계 — Compute Engine 처리가 끝났는지 확인이 필요한 경우 이 엔드포인트로 폴링한다.

### 4.4 시스템 상태 — `/api/system/status`

헬스체크용. 인증 없이 호출 가능.

```
GET /api/system/status
```

응답의 `status` 필드가 `UP` / `DOWN` / `STARTING` / `RESTARTING` / `DB_MIGRATION_NEEDED` / `DB_MIGRATION_RUNNING` 중 하나다. 모니터링/오케스트레이션 도구가 SonarQube 가용성을 보는 표준 진입점이다.



## 5. POST·PUT 호출의 형식

> 변경 요청은 form-urlencoded 본문을 받는다. JSON 본문이 아니다.

자주 부딪히는 함정 하나가 본문 형식이다. SonarQube의 변경 엔드포인트는 다음 형식이다.

```
POST /api/projects/create
Content-Type: application/x-www-form-urlencoded

project=org.okestro:new-project&name=New+Project
```

JSON으로 보내면 400이 떨어진다. 같은 Web API라도 GET의 query string과 POST의 form body 형식이 일치한다는 점이 SonarQube의 일관된 결정이다.

이 형식이 의미하는 바는 라이브러리 선택의 영향이다. Spring `RestTemplate`이나 OpenFeign이라면 `@RequestParam`으로 자연스럽게 처리되지만, JSON 기반 클라이언트(예: `WebClient`의 default 설정)를 쓰면 인코딩을 명시해야 한다. operator/cicd가 OpenFeign을 쓴 이유 중 하나가 이 form-urlencoded와의 자연스러운 매칭이다.

### 5.1 동시 변경 요청과 idempotency

생성·수정 엔드포인트의 멱등성은 SonarQube가 명시적으로 보장하지 않는다. 같은 요청을 두 번 보내면 다음 둘 중 하나가 일어난다.

- 첫 호출이 성공, 두 번째 호출은 400 (이미 존재) — `/api/projects/create` 같은 키 기반 생성.
- 두 번 모두 성공해 중복 생성 — `/api/webhooks/create` (이름 중복 허용).

operator가 webhook 등록 시 사전 존재 검사를 한 이유가 이 비일관성 때문이다. SonarQube의 동시성 보장에 의존하지 말고 클라이언트가 멱등성을 책임지는 패턴이 안전하다.



## 6. 정리

> Web API의 구조만 잡으면 외부 통합 코드의 70%가 결정된다.

요약한다. SonarQube Web API는 `/api/<resource>/<action>` 형태로 일관된 명명을 가지고, 응답은 paging + 데이터 + 참조 데이터의 표준 JSON 골격을 따른다. 페이지네이션은 페이지 크기 500과 결과 개수 10000의 두 한계를 가지며, 큰 결과는 검색 조건 분할로 우회한다. 변경 엔드포인트는 form-urlencoded 본문을 받고 멱등성은 명시적으로 보장되지 않는다. CI가 자주 부르는 엔드포인트는 `/api/qualitygates/project_status`, `/api/measures/component`, `/api/ce/task`, `/api/system/status` 넷이다.

다음 단계는 호출의 인증 측면이다. [03-02.인증과 토큰](03-02.인증과 토큰.md)에서 본다.
