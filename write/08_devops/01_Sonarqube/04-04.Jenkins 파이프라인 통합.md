# Jenkins 파이프라인 통합

---

> Jenkins에서 SonarQube를 호출하는 표준 패턴을 본다. SonarQube Scanner for Jenkins 플러그인의 두 가지 도구(`withSonarQubeEnv`, `waitForQualityGate`)를 중심으로 잡고, Webhook 비동기 패턴과의 차이를 정리한다.



## 1. 두 가지 통합 모드

> Jenkins에서 SonarQube를 부르는 방법은 두 가지로 갈린다. 하나는 빌드 안에서 결과를 기다리는 동기 모드, 다른 하나는 결과를 외부 시스템으로 흘려보내는 비동기 모드다.

이 둘의 차이를 한 표로 잡는다.

| 모드 | 결과 수신 | 빌드 결과 결정 시점 | 적합한 시나리오 |
|------|----------|-------------------|---------------|
| 동기 (waitForQualityGate) | Jenkins 빌드 내부 | 게이트 결과 도착 후 | "이 빌드는 게이트가 통과한 빌드"라는 명제가 필요 |
| 비동기 (Webhook 외부 처리) | 외부 서비스 | Scanner 종료 시점 | 게이트 결과로 다른 시스템 워크플로우를 트리거 |

TPS는 두 모드를 섞어 쓴다. `pipeline-api` 같은 자체 모듈의 분석은 동기 모드로 빌드 게이트를 만들 수 있고, 사용자 파이프라인의 SonarQube 실행은 Webhook → executor → ppln-logging-api로 흘러가는 비동기 경로를 쓴다.



## 2. Scanner for Jenkins 플러그인의 자리

> Jenkins에는 SonarQube 통합을 돕는 두 종류 자원이 있다. 글로벌 도구 등록과 파이프라인 단계의 헬퍼다.

SonarQube Scanner for Jenkins 플러그인은 다음을 제공한다.

- **글로벌 SonarQube 서버 등록**: Manage Jenkins → System → SonarQube servers에서 서버 주소와 토큰을 한 번 등록. 이름(예: `sonarqube-prod`)으로 파이프라인에서 참조한다.
- **`withSonarQubeEnv("이름")` 스텝**: 등록된 서버의 주소와 토큰을 환경변수(`SONAR_HOST_URL`, `SONAR_TOKEN`)로 자동 주입.
- **`waitForQualityGate()` 스텝**: SonarQube의 Webhook을 받아 게이트 결과를 빌드 안으로 가져온다. Webhook URL은 Jenkins가 자동 등록.

플러그인이 없으면 토큰을 파이프라인 시크릿에서 직접 꺼내 환경변수에 넣고, 게이트 결과는 `sonar.qualitygate.wait=true`나 `/api/qualitygates/project_status` 폴링으로 받아야 한다. 플러그인이 있으면 같은 일을 한 줄로 끝낸다.



## 3. 동기 모드의 표준 골격

> Declarative Pipeline에서 SonarQube 분석과 게이트 대기를 표현하는 표준 형태를 본다.

다음이 가장 단순한 동기 모드 골격이다.

```groovy
pipeline {
    agent any
    tools { jdk 'jdk-17' }
    stages {
        stage('Build') {
            steps {
                sh './gradlew clean build -x test'
            }
        }
        stage('Test') {
            steps {
                sh './gradlew test jacocoTestReport'
            }
        }
        stage('SonarQube Analysis') {
            steps {
                withSonarQubeEnv('sonarqube-prod') {
                    sh """
                        ./gradlew sonar \
                            -Dsonar.projectKey=org.okestro:pipeline-api \
                            -Dsonar.qualitygate.wait=true
                    """
                }
            }
        }
        stage('Quality Gate') {
            steps {
                timeout(time: 5, unit: 'MINUTES') {
                    waitForQualityGate abortPipeline: true
                }
            }
        }
    }
}
```

각 단계의 역할을 정리한다.

| 단계 | 책임 | 실패 시 영향 |
|------|------|-------------|
| Build | 컴파일 산출물 생성 | 컴파일 에러 — 빌드 즉시 실패 |
| Test | 단위 테스트 실행 + JaCoCo 리포트 생성 | 테스트 실패 — 빌드 실패 |
| SonarQube Analysis | Scanner 호출, SonarQube로 결과 업로드 | 업로드 실패 — 빌드 실패 |
| Quality Gate | Webhook 또는 폴링으로 게이트 결과 대기 | FAILED + abortPipeline → 빌드 실패 |

`sonar.qualitygate.wait=true`와 `waitForQualityGate()`가 둘 다 있는 이유는 무엇인가. 같은 게이트 결과를 두 번 기다리는 것 같지만, 차이가 있다.

- `sonar.qualitygate.wait=true`는 Scanner 자체가 결과를 기다린 뒤 종료한다. Scanner의 종료 코드가 게이트 결과를 반영한다.
- `waitForQualityGate()`는 Jenkins 플러그인이 Webhook을 받아 결과를 보고한다. Scanner는 이미 종료된 뒤다.

둘 중 하나만 써도 동작한다. 보통은 `waitForQualityGate()` 한 가지로 통일하는 게 깔끔하다.

### 3.1 timeout이 필요한 이유

Compute Engine 큐 적체 또는 SonarQube 다운으로 게이트 결과가 영영 안 오면 빌드가 무한 대기에 빠진다. `timeout(time: 5, unit: 'MINUTES')`로 외곽을 두르는 게 표준이다. SonarQube 자체가 `sonar.qualitygate.timeout`을 가지지만 Jenkins의 timeout이 더 강제력 있다.



## 4. Webhook 등록의 수동 vs 자동

> Jenkins 플러그인은 Webhook을 자동 등록하지만, 이 자동화가 모든 환경에서 작동하지는 않는다.

`waitForQualityGate()`가 작동하려면 SonarQube → Jenkins 방향의 Webhook이 등록돼야 한다. 플러그인은 글로벌 SonarQube 서버 등록 시 자동 등록을 시도하지만, 다음 경우에는 수동 등록이 필요하다.

- Jenkins URL이 SonarQube 입장에서 `localhost`나 와일드카드로 보일 때 (SonarQube가 거부)
- SonarQube 토큰이 webhook 생성 권한을 갖지 않을 때 (관리자 토큰이 필요)
- 사내 보안 정책상 자동 등록이 막혀 있을 때

수동 등록의 표준 형태는 다음과 같다.

```
Webhook 이름: jenkins-prod
URL: https://jenkins.internal/sonarqube-webhook/
Secret: (선택) HMAC 검증용 시크릿
```

URL의 끝 슬래시가 빠지면 플러그인이 인식하지 못하는 케이스가 보고된 적이 있다. 등록 후에는 SonarQube에서 테스트 페이로드를 발사해 Jenkins가 수신하는지 한 번 확인하는 게 안전하다.



## 5. TPS의 자격 증명 시드 패턴

> TPS Jenkins(트롬본 dev)는 자격 증명을 Groovy 스크립트로 시드한다. 같은 패턴으로 SonarQube 토큰을 등록하는 형태가 자연스럽다.

TPS는 Jenkins 자격 증명을 Manage Jenkins → Script Console에서 다음 스크립트로 시드한다.

```groovy
// operator/cicd/scripts/jenkins-seed-trb-dev-credentials.groovy
import com.cloudbees.plugins.credentials.CredentialsScope
import com.cloudbees.plugins.credentials.SystemCredentialsProvider
import com.cloudbees.plugins.credentials.domains.Domain
import com.cloudbees.plugins.credentials.impl.UsernamePasswordCredentialsImpl
import jenkins.model.Jenkins

def credStore = Jenkins.get().getExtensionList(SystemCredentialsProvider.class).first().getStore()
def domain = Domain.global()

void upsertUsernamePassword(def credStore, Domain domain, String id, String description, String username, String secret) {
    credStore.getCredentials(domain).findAll { it instanceof UsernamePasswordCredentialsImpl && id == it.id }.each {
        credStore.removeCredentials(domain, it)
    }
    credStore.addCredentials(
        domain,
        new UsernamePasswordCredentialsImpl(CredentialsScope.GLOBAL, id, description, username, secret)
    )
}

upsertUsernamePassword(credStore, domain, 'trb-gitlab', 'Trombone dev — GitLab HTTP', 'root', 'cloud1234')
upsertUsernamePassword(credStore, domain, 'trb-harbor', 'Trombone dev — Harbor OCI', 'admin', 'cloud1234')
upsertUsernamePassword(credStore, domain, 'trb-minio', 'Trombone dev — MinIO', 'minioadmin', 'minioadmin')
```

이 시드 스크립트가 SonarQube를 다루지는 않지만 패턴은 동일하다. SonarQube 토큰은 사용자명/비밀번호가 아니라 단일 토큰이므로 `StringCredentialsImpl`을 쓴다.

```groovy
import org.jenkinsci.plugins.plaincredentials.impl.StringCredentialsImpl
import hudson.util.Secret

void upsertSonarQubeToken(def credStore, Domain domain, String id, String description, String token) {
    credStore.getCredentials(domain).findAll { it instanceof StringCredentialsImpl && id == it.id }.each {
        credStore.removeCredentials(domain, it)
    }
    credStore.addCredentials(
        domain,
        new StringCredentialsImpl(CredentialsScope.GLOBAL, id, description, Secret.fromString(token))
    )
}

upsertSonarQubeToken(credStore, domain, 'trb-sonarqube', 'Trombone dev — SonarQube analysis token', 'squ_xxxxxxxxxxxxxxxx')
```

TPS의 시드 스크립트가 자격 증명만 다루고 SonarQube 서버 자체 등록(주소·SonarQube Plugin 설정)은 따로 한다는 점이 운영 사실이다. 서버 등록은 JCasC(Jenkins Configuration as Code) yaml에서 다루는 게 더 깔끔하다.

### 5.1 JCasC에서의 SonarQube 서버 등록

JCasC가 적용된 Jenkins라면 다음 yaml 한 묶음으로 끝난다.

```yaml
unclassified:
  sonarGlobalConfiguration:
    buildWrapperEnabled: true
    installations:
      - name: sonarqube-prod
        serverUrl: http://sonarqube.internal
        credentialsId: trb-sonarqube
        webhookSecretId: ""
        triggers:
          skipScmCause: false
          skipUpstreamCause: false
```

`name`이 파이프라인의 `withSonarQubeEnv('sonarqube-prod')`에 그대로 들어간다. credential ID는 위에서 시드한 `trb-sonarqube`를 참조한다. 환경 간 차이는 yaml만 바꾸면 끝나고, Jenkins UI에 손으로 입력할 일이 없어진다.



## 6. PR 분석 패턴

> 메인 브랜치 분석과 PR 분석은 SonarQube 옵션이 다르다. Jenkins 파이프라인이 어느 트리거로 실행됐는지에 따라 분석 호출이 갈라진다.

PR 분석은 다음 옵션을 추가로 전달해야 한다.

- `sonar.pullrequest.key` — PR 번호
- `sonar.pullrequest.branch` — PR 소스 브랜치
- `sonar.pullrequest.base` — PR 대상 브랜치(보통 `main`)

GitLab이나 GitHub 환경에서는 Multibranch Pipeline 또는 GitLab Branch Source 플러그인이 위 변수를 자동 주입한다. 파이프라인은 다음과 같은 형태로 분기를 거른다.

```groovy
stage('SonarQube Analysis') {
    steps {
        withSonarQubeEnv('sonarqube-prod') {
            script {
                def sonarOpts = [
                    "-Dsonar.projectKey=org.okestro:pipeline-api"
                ]
                if (env.CHANGE_ID) {  // PR 빌드일 때만 채워짐
                    sonarOpts << "-Dsonar.pullrequest.key=${env.CHANGE_ID}"
                    sonarOpts << "-Dsonar.pullrequest.branch=${env.CHANGE_BRANCH}"
                    sonarOpts << "-Dsonar.pullrequest.base=${env.CHANGE_TARGET}"
                } else {
                    sonarOpts << "-Dsonar.branch.name=${env.BRANCH_NAME}"
                }
                sh "./gradlew sonar ${sonarOpts.join(' ')}"
            }
        }
    }
}
```

PR 분석은 Developer Edition 이상에서만 작동한다는 점이 중요하다. Community Build에서는 `sonar.pullrequest.*`를 무시하므로 PR과 메인이 같은 결과로 합쳐진다.



## 7. 비동기 모드 — Webhook 외부 처리

> 빌드 안에서 게이트 결과를 기다리지 않고 외부 시스템으로 흘려보내는 패턴이다. TPS의 사용자 파이프라인이 이 모드를 쓴다.

비동기 모드의 골격은 단순하다.

```groovy
stage('SonarQube Analysis') {
    steps {
        withSonarQubeEnv('sonarqube-prod') {
            sh "./gradlew sonar -Dsonar.projectKey=${PROJECT_KEY}"
        }
    }
    // waitForQualityGate 단계 없음
}
```

빌드는 Scanner 종료 시점에 함께 종료된다. 게이트 결과는 SonarQube가 발사하는 Webhook을 외부 서비스가 수신해 처리한다. TPS의 경우 그 외부 서비스가 executor이며, executor는 결과를 ppln-logging-api로 흘려보내 자체 로그 모델에 적재한다([04-03.분석 실행 로그 수집](04-03.분석 실행 로그 수집.md) 참조).

이 모드의 트레이드오프는 명확하다.

- 빌드 시간이 짧다 — Compute Engine 처리 시간을 기다리지 않는다.
- "이 빌드는 게이트 통과 빌드"라는 명제가 약하다 — 빌드 결과와 게이트 결과의 일치를 사용자가 보장해야 한다.
- 게이트 실패가 빌드 실패로 직결되지 않는다 — 외부 시스템이 빌드를 다시 실패시키지는 못한다.

사용자 파이프라인에서 이 모드가 자연스러운 이유는 게이트 실패가 빌드 차단이 아니라 사용자 알림으로 흐르기 때문이다. 자체 모듈의 메인 브랜치 분석에는 동기 모드가 더 적합하다.



## 8. 정리 — 두 모드의 선택 기준

> Jenkins와 SonarQube의 통합은 동기·비동기 두 모드 중 어느 것을 고르느냐로 거의 결정된다.

선택의 기준 셋이다.

1. **빌드 결과의 의미**: "게이트 통과한 빌드만 머지 가능"이라는 정책이라면 동기 모드. "분석 결과는 외부 시스템에서 다시 다룬다"면 비동기 모드.
2. **빌드 시간 예측 가능성**: 동기 모드는 SonarQube 큐 적체에 빌드 시간이 영향받는다. 비동기 모드는 빌드 시간이 일정하다.
3. **Webhook 인프라**: 비동기 모드는 Webhook을 받는 서비스(executor 같은 것)가 있어야 한다. 그 서비스가 없으면 동기 모드로 가는 게 단순하다.

다음 단계는 4장 통합 활용의 통찰을 점검하는 [04-점검.핵심 질문과 답](04-점검.핵심 질문과 답.md)이다.
