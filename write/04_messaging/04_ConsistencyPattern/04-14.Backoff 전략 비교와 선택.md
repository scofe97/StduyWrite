# Backoff 전략 비교와 선택

---

> 재시도 자체보다 *얼마나 기다렸다가 재시도하는가*가 운영 안전성을 결정한다. 너무 짧으면 장애 시스템에 같은 폭격을 다시 가하고, 너무 길면 head-of-line blocking으로 정상 메시지가 막힌다. 본 글은 Spring Kafka의 `BackOff` 구현 4가지를 정리하고, TPS가 `ExponentialBackOff(1초, 2.0)` + 최대 누적 7초를 고른 근거를 본다.



## 1. 왜 백오프인가

> 재시도의 핵심 가설은 "시간이 지나면 결과가 달라진다"이다. 외부 API 일시 단절, DB deadlock, 네트워크 retry 같은 *transient* 실패가 그 자리에 해당한다. 백오프 없이 즉시 재시도하면 *원인을 만든 부하*가 그대로 반복되어 회복이 더 느려진다.

백오프가 의미 있는 실패 유형:

- 외부 API 일시 단절 — 보통 수초 내 자가 회복
- DB deadlock — 다른 트랜잭션이 release할 시간 필요
- 네트워크 jitter — 다음 packet은 다른 경로
- 리더 재선출 — Kafka broker / DB primary 교체 직후 수초

백오프가 *의미 없는* 실패 유형:

- Avro 스키마 위반 — 시간 지나도 같은 바이트는 같은 결과
- validation 실패 — 메시지가 자체로 잘못됨
- 권한 부족 — 인프라 변경 없이는 변하지 않음

이 분류가 `addNotRetryableExceptions`의 등록 기준이다 (`03-03 §결정3`, `03-04 §2-6`).



## 2. Spring Kafka의 BackOff 구현 4가지

> Spring Kafka는 Spring `org.springframework.util.backoff.BackOff` 인터페이스를 따른다. `FixedBackOff`, `ExponentialBackOff`, `ExponentialBackOffWithMaxRetries`가 표준 구현이고, 직접 구현도 가능하다.

### 2-1. FixedBackOff — 일정 간격, 횟수 한도

```java
new FixedBackOff(1000L, 3);  // 1초 간격, 최대 3회
```

가장 단순하다. 그러나 *장애 시 같은 부하를 같은 빈도로 다시 가하는* 위험이 있다. 외부 API가 부하로 죽어 있다면 1초마다 100개 컨슈머가 동시에 1초마다 재시도하면 회복이 늦어진다. *분산 부하 회복(thundering herd)이 운영 관심사*인 도메인에서는 부적합.

### 2-2. ExponentialBackOff — 지수 증가, 최대 누적 시간

```java
ExponentialBackOff backOff = new ExponentialBackOff(1000L, 2.0);
backOff.setMaxInterval(30000L);     // 한 번 대기 최대 30초
backOff.setMaxElapsedTime(60000L);  // 누적 60초 후 중단
```

초기 1초, 배수 2.0이면 간격이 1, 2, 4, 8, 16, 30(`maxInterval` 캡), 30, 30... 식으로 늘어난다. `maxElapsedTime`이 *누적 시간 상한*으로 동작하며, 누적이 그 값을 넘으면 다음 호출은 `BackOff.STOP`을 반환해 retry가 종료되고 recoverer로 진입한다.

이 구현이 *대부분의 운영에서 표준*인 이유는 두 가지다. 첫째, *장애 시스템에 회복할 시간을 점점 더 주는* 곡선이라 폭격이 줄어든다. 둘째, *최악의 경우 자동 종료*가 보장되어 무한 retry로 컨슈머 lag이 폭증할 일이 없다.

### 2-3. ExponentialBackOffWithMaxRetries — 누적 시간 대신 횟수 한도

```java
new ExponentialBackOffWithMaxRetries(5);  // 최대 5회
```

`ExponentialBackOff`를 상속하지만 *횟수*로 종료를 결정한다. 시간 예측보다 횟수 예측이 직관적인 경우에 쓴다. 시간 한도와 섞어 쓰지 않는다 — `maxElapsedTime`과 `maxRetries`가 충돌하면 동작 예측이 어렵다.

### 2-4. 직접 구현

`BackOff` 인터페이스는 `BackOffExecution start()`를 반환하고, `BackOffExecution.nextBackOff()`가 다음 대기 ms를 돌려준다. *Jitter 추가*(예: ±20% 랜덤)나 *시간대별 다른 곡선*(밤에는 길게)을 원하면 직접 구현한다.



## 3. 곡선 비교 — 같은 5회 재시도일 때

> 같은 횟수라도 곡선에 따라 *외부 시스템이 받는 부하*와 *내부 컨슈머 lag*가 크게 다르다.

| 정책 | 1차 | 2차 | 3차 | 4차 | 5차 | 누적 |
|------|-----|-----|-----|-----|-----|------|
| Fixed 1초 × 5 | 1초 | 1초 | 1초 | 1초 | 1초 | 5초 |
| Exponential(1초, 2.0) | 1초 | 2초 | 4초 | 8초 | 16초 | 31초 |
| Exponential(1초, 2.0, cap 5초) | 1초 | 2초 | 4초 | 5초 | 5초 | 17초 |
| TPS 현재 — Exponential(1초, 2.0) + maxElapsedTime 7초 | 1초 | 2초 | 4초 | (누적 7초 초과로 종료) | — | 7초 |

TPS는 누적 7초로 *짧게* 잡았다. 외부 호출(Jenkins/ArgoCD/GitLab)이 *수초 내에 자가 회복하는 경우*만 retry로 흡수하고, 그 외 실패는 *재시도해도 무의미*라는 판단이다. 30초 이상 늘리면 컨슈머가 한 메시지에 매달려 lag이 누적된다.



## 4. 같이 결정해야 할 셋 — interval, max, jitter

> 백오프 정책을 결정할 때 단순히 "exponential 쓴다"로 끝나면 운영에서 자주 막힌다. 세 가지를 함께 결정한다.

### 4-1. 초기 interval — 외부 시스템의 회복 시간 추정

외부 의존 시스템이 *얼마 만에 자가 회복하는지*를 보고 초기 interval을 잡는다. 1초가 일반적이지만, *느린 외부 시스템*(예: 외부 결제 게이트웨이가 5초 timeout으로 응답)이라면 초기 interval 자체를 2~3초로 늘리는 게 합리적이다.

### 4-2. max — 컨슈머 lag 허용 한도

`maxElapsedTime` 또는 `maxRetries`를 어디서 끊을지는 *한 메시지에 묶일 수 있는 최대 시간*이다. 단건 listener에서 한 메시지가 30초 retry 중이면 같은 partition의 다음 메시지는 30초 동안 대기한다 (head-of-line blocking).

이 한도가 너무 길면 lag이 폭증하고, 너무 짧으면 retry가 의미 없어진다. *외부 시스템의 평균 자가 회복 시간 × 2~3배*가 출발선이다.

### 4-3. Jitter — Thundering herd 방지

여러 컨슈머 인스턴스가 *같은 시각에 같은 외부 API 실패를 경험*하면, 같은 백오프 곡선을 따라 *동시에* 재시도한다. 외부 API가 부하로 죽었다면 회복 직후 다시 동시 폭격을 받는다.

Jitter는 각 인스턴스가 *베이스 대기 시간에 랜덤 비율*을 추가해 폭격 시점을 분산시키는 기법이다.

```java
public class JitteredExponentialBackOff implements BackOff {
    private final ExponentialBackOff delegate;
    public BackOffExecution start() {
        BackOffExecution inner = delegate.start();
        return () -> {
            long base = inner.nextBackOff();
            return (long) (base * (0.8 + Math.random() * 0.4));  // ±20%
        };
    }
}
```

Spring Kafka 표준 구현은 jitter를 내장하지 않으므로 *thundering herd가 운영 관심사*라면 직접 구현하거나 AWS SDK의 `FullJitterBackoffStrategy` 같은 외부 구현을 참고한다.



## 5. @RetryableTopic의 백오프 — 별도 경로

> 글로벌 `DefaultErrorHandler`의 백오프는 *in-place* 재호출이라 같은 파티션이 점유된다. `@RetryableTopic`은 *별도 retry 토픽*으로 메시지를 옮겨 백오프 동안 원본 파티션을 비운다.

```java
@RetryableTopic(
        attempts = "3",
        backoff = @Backoff(delay = 1000, multiplier = 2.0),
        kafkaTemplate = "retryKafkaTemplate"
)
@KafkaListener(topics = "deploy")
public void onDeploy(DeployCommandAvro cmd) { ... }
```

`@Backoff`의 `delay` + `multiplier`가 ExponentialBackOff와 동일한 의미다. 그러나 동작은 다르다 — retry-N 토픽으로 메시지가 이동하고, 별도 컨슈머가 *백오프 시간만큼 늦게* 그 토픽을 polling한다. 원본 토픽의 다음 메시지는 *즉시* 진행 가능.

이 비대칭이 `03-04`이 다루는 사고의 출발점이다. `@RetryableTopic`이 만든 내부 recoverer는 글로벌 `KafkaErrorConfig`의 `excludeHeader` 정책과 무관하므로, *백오프 정책뿐 아니라 헤더 정책도 별도 인스턴스*다.



## 6. 운영 체크리스트

백오프 정책을 도입·변경할 때:

- *외부 시스템 평균 자가 회복 시간*을 측정했는가 — 추측 금지
- `maxElapsedTime` × 평균 처리 시간이 *컨슈머 lag 허용치*를 넘지 않는가
- 같은 토픽의 다른 컨슈머가 *같은 백오프 정책*을 공유하는지 — 다르면 운영 예측이 어렵다
- Thundering herd 위험이 있는 외부 시스템이면 jitter를 추가
- `@RetryableTopic` 컨슈머와 글로벌 컨슈머가 *동시에 살아 있는 경우*, 두 정책이 일관된 곡선인지

모니터링:

- Kafka consumer lag per partition — 백오프가 너무 길면 lag으로 드러난다
- DLQ 메시지 도착률 — 백오프 후에도 격리되는 비율이 외부 시스템 SLA에 맞는지
- Retry 횟수 분포 — 1차에서 끝나는 비율이 높으면 초기 interval이 길 가능성, 마지막에 끝나는 비율이 높으면 max가 짧을 가능성



## 7. 정리

백오프는 *시간을 사는 것*이다. 외부 시스템이 회복할 시간, 부하가 분산될 시간, 같은 메시지를 다시 보낼 때 결과가 달라질 시간이다. 무한 인내(긴 max)는 lag을 만들고, 즉시 포기(짧은 max)는 재시도가 의미 없어진다. TPS는 외부 시스템 자가 회복이 보통 수초 내라는 도메인 특성에 맞춰 *초기 1초, 배수 2.0, 누적 7초*를 골랐다. 도메인이 다르면 곡선도 달라야 한다 — 곡선을 정할 때 SLA와 lag 허용치를 먼저 정의하고 거꾸로 푼다.
