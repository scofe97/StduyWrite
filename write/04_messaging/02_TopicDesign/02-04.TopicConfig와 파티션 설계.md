# TopicConfig와 파티션 수, 소비 병렬성

---

> ⚠️ **현재 코드 상태와의 차이** — 본 문서는 `DlqConsumer` 가 살아있던 시점 자료다. 현재 `message-lib` 에 `DlqConsumer.java` 는 **존재하지 않는다** ([03-02](../04_ConsistencyPattern/04-11.DlqConsumer.md)). DLQ 토픽(`tps.v305p.dlq`) 자체와 파티션 구성·소비 모델 해석은 그대로 유효하지만, "한 노드가 active consumer 가 된다" 같은 서술은 *DlqConsumer 가 다시 도입됐을 때를 가정한* 시나리오로 읽는다.

> 대상 코드
>
> - `message-lib/src/main/java/org/okestro/tps/messaging/topic/TopicConfig.java`
> - `message-lib/src/main/java/org/okestro/tps/messaging/topic/Topics.java`
> - `message-lib/src/main/java/org/okestro/tps/messaging/dlq/DlqConsumer.java`
>
> 확인 포인트
>
> - 왜 어떤 토픽은 3파티션이고 DLQ는 1파티션인지
> - 1파티션 토픽을 여러 노드가 띄우면 각자 소비하는지
> - consumer group과 partition 수의 관계

이 문서는 현재 message-lib의 토픽 설정을 기준으로, Kafka에서 토픽/파티션 수가 소비 병렬성과 어떻게 연결되는지 정리한다. 특히 `tps.v305p.dlq`가 1파티션일 때 여러 노드에서 어떻게 소비되는지를 명확히 설명한다.


## 1. 현재 프로젝트의 토픽 정의

`TopicConfig`를 보면 현재 기본 토픽은 세 개다.

- `tps.example.events`: 3파티션
- `tps.jenkins.build`: 3파티션
- `tps.v305p.dlq`: 1파티션

의도는 꽤 명확하다.

- 일반 이벤트 토픽은 어느 정도 병렬 소비를 염두에 두고 3파티션
- DLQ는 우선 단순 운영을 목표로 1파티션

즉 지금 설계는 "정상 이벤트는 병렬성, DLQ는 단순성" 쪽으로 기울어 있다.


## 2. Kafka에서 파티션 수가 의미하는 것

Kafka에서 병렬 소비의 기본 단위는 토픽이 아니라 **파티션**이다.

중요한 규칙은 이것이다.

> 같은 consumer group 안에서는 **각 파티션이 정확히 한 consumer instance에만 할당**된다.

즉 병렬성은 "consumer 수"만으로 결정되지 않고, **partition 수와 consumer group 구성의 조합**으로 결정된다.

예를 들어:

- 토픽이 3파티션이면 같은 group 안에서 최대 3개의 active consumer가 병렬로 읽을 수 있다.
- 토픽이 1파티션이면 같은 group 안에서는 최대 1개만 active consumer가 된다.


## 3. 질문의 핵심: 1파티션이어도 여러 노드가 각자 소비하는가

답은 **경우에 따라 다르다**다.

핵심 기준은 `group.id`다.

### 경우 1. 여러 노드가 같은 `group.id`를 쓰는 경우

현재 `DlqConsumer`는 이렇게 선언되어 있다.

```java
@KafkaListener(topics = Topics.DLQ, groupId = "tps-dlq-handler")
```

즉 여러 노드에 같은 애플리케이션이 배포되면, 그 노드들은 전부 **같은 consumer group**에 들어간다.

이 상태에서 `tps.v305p.dlq`가 1파티션이면:

- partition 0은 group 안의 딱 한 consumer instance에만 할당된다.
- 나머지 노드들은 idle 상태가 된다.
- 메시지 하나를 모든 노드가 각각 받는 것이 아니다.

즉:

- 여러 노드가 떠 있어도
- 같은 group이면
- **오직 한 노드만 실제로 소비**

한다.

### 경우 2. 여러 노드가 서로 다른 `group.id`를 쓰는 경우

만약 각 노드가 다른 group id를 쓰면 상황이 달라진다.

예:

- node A: `group.id=dlq-handler-a`
- node B: `group.id=dlq-handler-b`

이 경우는 Kafka에서 각 group이 독립 구독자처럼 동작한다.

그러면 1파티션 토픽이어도:

- A group이 한 번 읽고
- B group도 한 번 읽는다

즉 **각 group마다 한 번씩** 소비된다.

다만 이건 같은 애플리케이션 인스턴스들이 로드밸런싱하는 구조가 아니라, publish-subscribe처럼 "그룹별 복수 구독"을 만드는 구조다.


## 4. 현재 `tps.v305p.dlq` 기준으로 정확히 해석하면

현재 코드 기준으로는 다음이 사실이다.

### 사실 1. DLQ 토픽은 1파티션이다

`TopicConfig`:

```java
new NewTopic(Topics.DLQ, 1, (short) 1)
```

### 사실 2. DLQ consumer group은 고정이다

`DlqConsumer`:

```java
groupId = "tps-dlq-handler"
```

### 사실 3. 따라서 여러 노드를 띄워도 동시에 각자 소비하지 않는다

현재 구조에서 여러 노드가 뜨더라도:

- 모두 `tps-dlq-handler` group에 참여
- `tps.v305p.dlq`의 partition 0 하나를 서로 경쟁
- 최종적으로 한 노드만 active consumer

즉 질문에 대한 직접 답은 이렇다.

> 현재 코드 그대로라면, `tps.v305p.dlq`가 1파티션일 때 여러 노드가 "각자 다 소비"하지는 않는다. 같은 group이면 한 노드만 소비하고, 나머지는 대기한다.


## 5. 왜 이런 동작을 하는가

Kafka 공식 문서 기준으로:

- 각 레코드는 각 consumer group 안의 한 consumer instance에만 전달된다.
- 하나의 partition은 같은 group 안에서 한 consumer만 소유한다.
- consumer 수가 partition 수보다 많으면 일부 consumer는 데이터를 받지 못한다.

즉 Kafka의 consumer group은 "브로드캐스트"보다 "로드밸런싱된 queue"에 가깝다.

그래서:

- 같은 group = load balancing
- 다른 group = broadcast-like fan-out

으로 이해하면 된다.


## 6. 1파티션 DLQ의 장점

현재처럼 DLQ를 1파티션으로 두는 건 나쁜 설계는 아니다. 오히려 의도가 분명하면 꽤 실용적이다.

### 1. 순서가 단순하다

모든 DLQ 메시지가 한 파티션에 쌓이면 소비 순서를 이해하기 쉽다.

### 2. 운영이 단순하다

파티션이 하나면:

- 어떤 consumer가 먹는지 단순하고
- 관찰 포인트도 단순하고
- DLQ 메시지 규모가 작을 때 다루기 쉽다

### 3. DLQ는 보통 고처리량보다 격리가 중요하다

정상 토픽은 처리량이 중요하지만, DLQ는 보통 "실패를 격리하고 나중에 분석"하는 쪽이 중요하다. 이 경우 1파티션이 충분할 수 있다.


## 7. 1파티션 DLQ의 한계

### 1. 병렬 소비가 안 된다

같은 group 기준으로 active consumer는 최대 1개다.

### 2. 장애량이 많아지면 병목이 된다

DLQ 메시지가 급증하면 재처리나 분석 속도가 느려질 수 있다.

### 3. 운영 유연성이 떨어진다

토픽별 분리, 파티션별 재처리, 병렬 복구 같은 전략이 어려워진다.


## 8. 언제 1파티션이 괜찮고, 언제 늘려야 하나

### 1파티션이 괜찮은 경우

- DLQ 발생량이 낮다
- 우선 로그 확인만 하면 된다
- 학습/PoC 단계다
- 순차 분석이 더 중요하다

### 파티션을 늘려야 하는 경우

- DLQ 적재량이 많다
- 재처리 throughput이 필요하다
- 여러 인스턴스로 병렬 복구해야 한다
- 토픽별/업무별 DLQ 분리 전략이 필요하다


## 9. 실무에서 자주 헷갈리는 포인트

### 파티션 1개여도 노드가 여러 개면 HA가 되지 않나

된다. 다만 **동시 병렬 처리**가 아니라 **대기 후 takeover**다.

예:

- node A가 partition 0을 맡고 있음
- node B, C는 같은 group에서 대기 중
- node A가 죽으면 rebalance 후 B 또는 C가 partition 0을 이어받음

즉:

- 가용성은 생긴다
- 병렬성은 생기지 않는다

### replication factor 1은 이 문제와 같은가

아니다.

- partition 수: 소비 병렬성
- replication factor: broker 장애 내성

현재 `tps.v305p.dlq`는 1파티션, replication 1이다.

즉:

- 소비 병렬성도 낮고
- broker 장애 내성도 낮다

학습용/PoC로는 괜찮지만 운영 환경에서는 별도 판단이 필요하다.


## 10. 현재 프로젝트 기준 권장 해석

현재 `tps.v305p.dlq` 설계는 "운영 최적화"보다 "단순한 관측과 기본 실패 격리"에 가깝다.

따라서 지금 구조를 이렇게 보는 게 맞다.

- 여러 노드 배포 시: HA는 가능
- 같은 group 기준 병렬 소비: 불가
- DLQ 운영 복잡도: 낮음
- DLQ 처리량 확장성: 낮음


## 11. 한 줄 정리

> 현재 `tps.v305p.dlq`는 1파티션이고 `DlqConsumer`는 같은 `groupId`를 쓰므로, 여러 노드를 띄워도 각 노드가 각각 소비하지 않는다. 같은 group에서는 한 노드만 active consumer가 되고, 나머지는 장애 대비 대기 노드가 된다.
