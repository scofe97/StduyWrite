# 다중 직렬화 컨슈머 구성 3방식

---

> 한 애플리케이션이 Avro 토픽과 JSON 토픽을 동시에 소비해야 하는 상황은 운영에서 흔합니다. 우리 팀은 Avro로 발행하는데 옆 팀은 JSON으로 발행하는 식으로 직렬화 포맷이 토픽마다 갈리기 때문입니다. 이때 컨슈머마다 key/value deserializer가 달라야 해서, 단순히 `@KafkaListener` 하나로는 끝나지 않습니다. ING 발표(*Spring for Apache Kafka — the advanced features*, Spring I/O Barcelona 2025)가 이 문제를 세 가지 방법으로 풀어 보였고, 이 글은 그 셋을 비교합니다.

## 1. 문제 — 토픽마다 다른 직렬화 포맷

Spring Kafka는 `application.yml`의 `spring.kafka.consumer.value-deserializer`로 *애플리케이션 전체의 기본 deserializer*를 정합니다. 그런데 한 앱이 Avro 토픽과 JSON 토픽을 같이 들어야 하면, 이 단일 기본값으로는 둘을 모두 처리할 수 없습니다. Avro 컨슈머는 `KafkaAvroDeserializer`가 필요하고 JSON 컨슈머는 `ByteArrayDeserializer`(또는 `JsonDeserializer`)가 필요하기 때문입니다.

해결의 핵심은 *컨슈머별로 다른 `ConsumerFactory`/`ListenerContainerFactory`를 쓰게 만드는 것*입니다. 이걸 어떻게 구성하느냐에서 세 방법이 갈립니다. 보일러플레이트 양과 유연성이 서로 다릅니다.

## 2. 방법 ① — 빈을 직접 정의

가장 기본적인 방법은 포맷마다 `ConsumerFactory`와 `ConcurrentKafkaListenerContainerFactory` 빈을 손으로 만드는 것입니다. 그리고 `@KafkaListener`에서 `containerFactory`로 어느 팩토리를 쓸지 지정합니다.

```java
@Bean
public ConsumerFactory<String, byte[]> jsonConsumerFactory(KafkaProperties props) {
    Map<String, Object> config = props.buildConsumerProperties(null);
    config.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, ByteArrayDeserializer.class);
    return new DefaultKafkaConsumerFactory<>(config);
}

@Bean
public ConcurrentKafkaListenerContainerFactory<String, byte[]> jsonListenerFactory(
        ConsumerFactory<String, byte[]> jsonConsumerFactory) {
    var factory = new ConcurrentKafkaListenerContainerFactory<String, byte[]>();
    factory.setConsumerFactory(jsonConsumerFactory);
    return factory;
}
```

```java
@KafkaListener(topics = "external.json", containerFactory = "jsonListenerFactory")
public void onJson(ConsumerRecord<String, byte[]> record) { ... }
```

포맷이 둘이면 이 빈 묶음이 두 벌, 셋이면 세 벌이 됩니다. 명시적이라 동작이 투명하지만, 포맷이 늘수록 설정 코드가 선형으로 불어나는 게 단점입니다.

## 3. 방법 ② — 어노테이션 + 프로퍼티 오버라이드

빈을 새로 만드는 대신, `@KafkaListener`의 `properties` 속성으로 *그 리스너에만* deserializer를 덮어쓰는 방법입니다. 기본 컨슈머 설정은 YAML 한 곳에 두고, 포맷이 다른 리스너만 국소적으로 오버라이드합니다.

```java
@KafkaListener(
        topics = "stock-quotes-avro"
        , groupId = "avro-group"
)
public void onAvro(StockQuote quote) { ... }   // 기본(Avro) deserializer 사용

@KafkaListener(
        topics = "stock-quotes-json"
        , groupId = "json-group"
        , properties = {
            "value.deserializer=org.springframework.kafka.support.serializer.JsonDeserializer"
        }
)
public void onJson(StockQuoteDto dto) { ... }   // 이 리스너만 JSON 으로 오버라이드
```

방법 ①보다 빈 정의가 사라져 가볍습니다. 직렬화 차이가 "리스너 한두 개"수준일 때 가장 깔끔합니다. 다만 오버라이드가 어노테이션 곳곳에 흩어지므로, 컨슈머가 많아지면 어디가 어떤 포맷인지 한눈에 보기 어려워집니다.

## 4. 방법 ③ — 설정만으로 런타임 컨슈머 생성

ING 발표에서 *가장 유용한 부분*으로 강조한 방법입니다. 컨슈머를 코드가 아니라 *설정 블록*으로 선언하고, 약 100줄짜리 프레임워크 코드가 그 설정을 읽어 `MessageListenerContainer`를 런타임에 만들어 띄웁니다. 새 컨슈머를 추가할 때 설정 블록만 복사하면 되고, 새 코드는 한 줄도 필요 없습니다.

설정은 다음과 같은 커스텀 섹션으로 둡니다. `enabled` 플래그로 개별 컨슈머를 켜고 끌 수 있어, 설정에서 지우지 않고 잠시 비활성화하는 운영도 가능합니다.

```yaml
app:
  kafka:
    consumers:
      - id: avro-consumer
        enabled: true
        topics: stock-quotes-avro
        group-id: avro-group
        # value-deserializer 미지정 → 기본(Avro) 사용
      - id: json-consumer
        enabled: true
        topics: stock-quotes-json
        group-id: json-group
        value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
```

프레임워크 코드의 골자는 "활성화된 설정만 골라 각각 컨테이너를 만들고 시작"하는 한 메서드입니다. 포맷별 비즈니스 로직은 `MessageListener` 구현(`onMessage`)으로 따로 둡니다.

```java
for (ConsumerConfig cfg : props.getConsumers()) {
    if (!cfg.isEnabled()) continue;                 // enabled=false 는 건너뜀
    var container = containerFactory.createContainer(cfg.getTopics());
    container.getContainerProperties().setGroupId(cfg.getGroupId());
    container.setupMessageListener(listenerFor(cfg));  // 포맷별 MessageListener
    container.start();                              // 런타임에 직접 start
}
```

이 방식이 §2([`03-02` Runtime Configurable Consumer](03-02.Spring%20Kafka%20운영%20고급.md))의 `KafkaListenerEndpointRegistry` 제어와 같은 뿌리에 있다는 점이 중요합니다. 둘 다 컨테이너를 *런타임 객체*로 다룹니다. 차이는 방법 ③이 컨테이너를 *설정에서 새로 만드는* 쪽이고, §3은 *이미 만들어진* 컨테이너를 start/stop하는 쪽이라는 점입니다.

## 5. 세 방법 비교

> 보일러플레이트가 적을수록 좋은 게 아니라, *컨슈머 추가 빈도*와 *동적 제어 필요성*에 맞춰 고릅니다.

| 항목 | ① 빈 직접 정의 | ② 어노테이션 오버라이드 | ③ 설정 런타임 생성 |
| --- | --- | --- | --- |
| 새 포맷 추가 비용 | 빈 묶음 한 벌 | 어노테이션 `properties` 한 줄 | 설정 블록 복사(코드 0) |
| 동작 투명성 | 높음(다 보임) | 중간 | 낮음(프레임워크에 가려짐) |
| 동적 on/off | 어려움 | 어려움 | `enabled` 플래그로 쉬움 |
| 적합한 규모 | 포맷 2~3개 고정 | 리스너 소수 | 컨슈머가 자주 늘거나 운영 토글 필요 |

방법 ③은 강력하지만 프레임워크 코드를 직접 유지해야 하므로, 컨슈머가 자주 추가·폐기되는 환경(발표의 ING 사례처럼 토픽이 매주 바뀌는 곳)에서 값을 합니다. 그렇지 않으면 ① 또는 ②가 더 단순합니다.
