# message-lib config 학습 검증

---

> [01-12](04-01.message-lib%20config%205개%20클래스%20종합.md)에서 다섯 클래스의 구조와 역할을 이해했다면, 이 문서로 *말로 설명할 수 있는지*와 *손으로 만들 수 있는지*를 검증한다. 면접 Q&A 10개와 실습 과제 3개로 구성된다.



## 0. 사용 방법

> 이 문서는 06-01을 다 읽은 뒤 *반드시 한 번은* 거쳐야 학습이 완료된다. 답을 보고 외우는 것이 아니라 먼저 말로 답해 보고 나서 답을 본다.

먼저 §1 면접 Q&A를 한 문제씩 *입으로 답한 뒤* 모범 답안과 비교한다. 막히는 문제가 있으면 해당 답안의 06-01 참조 절로 돌아가 다시 읽는다. Q&A가 끝나면 §2 실습 과제로 넘어가 빈 Spring Boot 프로젝트에서 세 패턴을 직접 재현한다. 검증 명령과 기대 결과까지 모두 한 번씩 눈으로 확인하면 학습이 닫힌다.

06-01에서 정의한 학습 목표 다섯 가지(§0-3)를 옆에 펴 두면 좋다. 본 문서의 Q&A와 실습이 각각 어떤 목표를 검증하는지 의식하면서 풀면 빈틈이 더 잘 드러난다.



## 1. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 10개. 각 답은 한 문장 결론 + 한 단락 설명으로 작성했다. 답을 보지 않고 먼저 본인이 답해 본 뒤 비교한다.

### Q1. `message-lib`의 config 클래스는 왜 다섯 개로 나뉘어 있나요?

부팅 시점이 *두 단계*로 갈리고, 책임이 *다섯 영역*으로 갈리기 때문이다. `EnvironmentPostProcessor` 단계에 들어가야 할 것(KafkaDefaultsEPP)과 `@AutoConfiguration` 단계에 들어갈 것(나머지 넷)이 먼저 나뉘고, 그 안에서 글로벌 에러·재시도 전용 producer·헤더 디코딩·외부 JSON 격리라는 *직교 책임*으로 다시 나뉜다. 한 파일에 합치면 책임이 섞여 변경 영향도가 추적 불가능해진다.

### Q2. `EnvironmentPostProcessor`와 `@Configuration`의 차이를 한 문장으로 설명해 보세요.

EPP는 ApplicationContext가 만들어지기 *전*에 environment의 properties를 조작하고, `@Configuration`은 ApplicationContext가 만들어진 *후*에 빈을 등록한다. Spring Boot의 `KafkaAutoConfiguration`이 properties를 읽어 빈을 만들기 때문에, properties 기본값을 깔려면 EPP 단계가 아니면 늦다.

### Q3. EOS 트리오(`acks=all` + `retries>0` + `enable.idempotence=true`) 중 하나라도 빠지면 어떻게 되나요?

`acks=all`이 빠지면 리더 브로커만 ack해 팔로워 동기화 전에 리더 장애 시 메시지 유실이 생길 수 있다. `retries=0`이면 일시적 네트워크 오류에서 메시지가 즉시 손실된다. `enable.idempotence=false`면 retry가 같은 메시지를 두 번 발행하는 중복 위험이 생긴다. 셋이 합쳐져야 단일 파티션 EOS가 성립한다.

### Q4. `ErrorHandlingDeserializer` 위임 패턴은 컨슈머 thread를 어떻게 보호하나요?

역직렬화 호출을 try/catch로 감싸 예외를 *헤더에 박아서* listener까지 정상 흐름으로 전달한다. 만약 위임 없이 `KafkaAvroDeserializer`를 직접 두면 역직렬화 실패가 컨슈머 thread에서 unchecked로 튀어 thread 자체가 죽고, 컨슈머 그룹 리밸런스가 트리거된다. 위임 패턴이 *thread 안전망* 역할을 한다.

### Q5. `@RetryableTopic`을 쓰는데 `retryKafkaTemplate` 빈이 없으면 어떻게 되나요?

런타임에 `ClassCastException`이 난다. Spring Kafka가 기본 `KafkaTemplate<String, byte[]>`을 쓰려 시도하는데, 컨슈머가 받은 value는 `SpecificRecord`(Java 객체)라 byte[] 직렬화기에 캐스팅이 실패한다. 본 시스템은 `RetryableTopicConfigurationGuardTest`가 이 누락을 빌드 단계에서 잡는다.

### Q6. `KafkaHeaderMapperConfig`의 `addTrustedPackages("*")`는 보안상 안전한가요?

본 시스템에서는 안전하다. 시스템 내부 헤더가 CloudEvents 표준 헤더(`ce_id`, `ce_source` 등)로 *단순 문자열*만 다루고, 외부에서 임의 객체가 헤더로 들어오는 경로가 없기 때문이다. 외부 시스템이 객체 헤더를 보낼 수 있는 환경이라면 `*` 대신 신뢰 패키지를 명시해야 한다.

### Q7. Avro 토픽과 JSON 토픽을 한 애플리케이션에서 소비하려면 어떻게 분리하나요?

`KafkaJsonConsumerConfig`처럼 별도 `ConsumerFactory`와 `ConcurrentKafkaListenerContainerFactory`를 등록하고, JSON 토픽 컨슈머에서 `@KafkaListener(containerFactory = "jsonListenerFactory")`로 명시 지정한다. 컨슈머가 명시하지 않으면 자동으로 기본(Avro) listener factory를 받는 *whitelist 방식*이라 운영상 안전하다.

### Q8. `KafkaErrorConfig`의 정책이 `@RetryableTopic` 경로에 왜 적용되지 않나요?

`@RetryableTopic`은 Spring Kafka의 `RetryTopicConfigurer`가 *별도 Recoverer 인스턴스*를 만들어 동작한다. KafkaErrorConfig가 `DefaultErrorHandler`에 박은 `excludeHeader`·`StripPreviousExceptionHeaders` 설정은 그 별도 인스턴스에 적용되지 않는다. 둘은 *고속도로 안전망과 별도 톨게이트의 안전망*처럼 분리돼 있다.

### Q9. Redpanda v25.3에서 본 다섯 클래스 중 별도 점검이 필요한 자리는 어디인가요?

두 곳이다.

1. `KafkaRetryTemplateConfig`가 활성화되려면 Schema Registry URL이 yaml에 명시돼야 한다. Redpanda는 v22.x부터 Schema Registry를 내장하지만 URL은 환경마다 다르므로 운영·테스트 양쪽에서 누락 여부를 따로 확인한다.
2. `KafkaDefaultsEPP`의 `enable.idempotence=true`가 의존 모듈에서 의도치 않게 비활성화되지 않았는지 actuator로 확인한다. idempotence가 꺼지면 outbox 폴러 retry가 중복 메시지를 만든다.

### Q10. `addLast`로 PropertySource를 등록하는 설계 의도는 무엇인가요?

라이브러리가 *강요하지 않는 기본값*을 깔기 위함이다. `addLast`는 가장 낮은 우선순위 자리라 의존 모듈이 자기 yaml로 같은 키를 정의하면 자연스럽게 덮어쓴다. 반대로 `addFirst`로 등록하면 의존 모듈이 라이브러리 정책을 바꿀 수 없게 되어 운영 자유도가 떨어진다.



## 2. 실습 과제 — 다섯 클래스를 손으로 만나기

> 개념을 머리로 안다고 손이 아는 것은 아니다. 다음 세 과제를 빈 Spring Boot 프로젝트에서 직접 돌려 본다. 각 과제 끝에 *검증 명령*과 *기대 결과*를 명시했다.

### 2-1. 실습 1 — KafkaDefaultsEPP 한 줄 이식

**목표**: EPP 패턴이 actuator에 어떻게 노출되는지 직접 확인한다.

**준비**:

1. Spring Boot 3.x + spring-kafka + spring-boot-starter-actuator 의존성을 가진 빈 프로젝트.
2. `application.yml`에 `management.endpoints.web.exposure.include: env` 추가.

**과제**:

1. `KafkaDefaultsEnvironmentPostProcessor`를 06-01 §3-2/§3-3에서 인용한 11개 키로 작성한다.
2. `src/main/resources/META-INF/spring.factories`에 다음 한 줄을 추가한다.
   ```properties
   org.springframework.boot.env.EnvironmentPostProcessor=your.pkg.KafkaDefaultsEnvironmentPostProcessor
   ```
3. 부팅 후 `curl localhost:8080/actuator/env | jq '.propertySources[] | select(.name=="kafkaMessagingDefaults")'`로 PropertySource 등록 확인.

**검증**: 응답에 `kafkaMessagingDefaults`가 보이고, 그 안에 `spring.kafka.producer.properties.enable.idempotence=true`가 있어야 한다.

**보너스**: `application.yml`에 같은 키를 `false`로 명시한 뒤 actuator를 다시 본다. EPP 값은 그대로지만 *유효 값*은 yaml이 이긴다는 사실을 actuator의 `activeProfiles`/우선순위로 확인한다.

### 2-2. 실습 2 — @ConditionalOnProperty 조건부 빈 재현

**목표**: `KafkaRetryTemplateConfig`처럼 *환경 변수 하나로 빈 등록 자체를 끄는* 패턴을 손으로 만든다.

**준비**: 위 실습 1의 프로젝트를 재사용.

**과제**:

1. 다음 두 빈을 가진 `@AutoConfiguration` 클래스를 작성한다.
   ```java
   @AutoConfiguration
   @ConditionalOnProperty(name = "myapp.feature.x.enabled", havingValue = "true")
   public class FeatureXAutoConfiguration {
       @Bean
       public FeatureXService featureXService() { return new FeatureXService(); }
   }
   ```
2. `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`에 등록한다.
3. `application.yml`에 `myapp.feature.x.enabled` 키를 *빼고* 부팅 후 `getBeansOfType(FeatureXService.class)`이 빈 맵을 반환하는지 확인.
4. 같은 키를 `true`로 추가 후 재기동, 빈이 생기는지 확인.

**검증**: 조건 미설정 시 빈 부재, `true` 설정 시 빈 존재.

**보너스**: `havingValue` 없이 `@ConditionalOnProperty(name = "...", matchIfMissing = false)`로 바꿔 두 경우 동작 차이를 비교한다. 본 시스템의 `KafkaRetryTemplateConfig`가 `havingValue`를 생략한 채 *URL 존재 자체만* 검사하는 의도를 이해한다.

### 2-3. 실습 3 — 두 listener factory 공존 재현

**목표**: `KafkaJsonConsumerConfig`처럼 *기본 factory + 명시 factory*가 한 애플리케이션 안에 공존하는 패턴을 손으로 만든다.

**준비**: 로컬 Kafka 또는 Redpanda 한 노드(`docker run -p 9092:9092 redpandadata/redpanda:v25.3.1 redpanda start --smp 1 --overprovisioned`).

**과제**:

1. 두 토픽을 만든다: `topic-a` (Avro로 발행), `topic-b` (JSON으로 발행).
2. 메인 `@Configuration`에 기본 `ConsumerFactory<String, SpecificRecord>` 한 개와, 별도 `ConsumerFactory<String, byte[]>` + `jsonListenerFactory` 한 쌍을 등록.
3. 두 listener를 작성한다.
   ```java
   @KafkaListener(topics = "topic-a")
   public void onA(MyAvroRecord r) { log.info("avro: {}", r); }

   @KafkaListener(topics = "topic-b", containerFactory = "jsonListenerFactory")
   public void onB(ConsumerRecord<String, byte[]> r) { log.info("json: {}", new String(r.value())); }
   ```
4. `rpk topic produce topic-a`로 Avro 메시지(`kafka-avro-console-producer` 또는 직접 Producer), `rpk topic produce topic-b`로 JSON 텍스트를 각각 발행한다.
5. 두 listener가 각각 자기 토픽만 처리하는지 로그로 확인.

**검증**: `onA`는 Avro 토픽의 메시지만, `onB`는 JSON 토픽의 메시지만 처리한다. `containerFactory`를 빼면 어떤 일이 일어나는지 한 번 의도적으로 깨 본다(JSON 토픽이 Avro deserializer를 만나면 wire format magic byte 부재로 즉시 실패).

**보너스**: 두 listener의 컨슈머 그룹을 같은 ID로 묶었을 때와 다른 ID로 분리했을 때의 차이를 관찰한다 (`../04-03 Consumer Group` 참조).



## 3. 자기 점검 체크리스트

> Q&A와 실습을 마쳤다면 다음 일곱 항목을 스스로 점검한다. 모두 ✓이면 학습 완료다.

1. Q1~Q10을 답안 없이 *입으로* 답할 수 있다.
2. 06-01 §0-3의 학습 목표 다섯 가지에 각각 한 단락으로 답할 수 있다.
3. 실습 1의 `kafkaMessagingDefaults` PropertySource를 actuator로 직접 확인했다.
4. 실습 2에서 `@ConditionalOnProperty` 키 유무에 따른 빈 부재·존재를 직접 관찰했다.
5. 실습 3에서 Avro·JSON 두 listener가 각자 자기 토픽만 처리하는 로그를 봤다.
6. 실습 3 보너스에서 `containerFactory`를 일부러 빠뜨려 wire format 충돌이 나는 모습을 봤다.
7. 06-01 §4의 "건물 5층 비유"를 자기 말로 한 단락 다시 풀어 쓸 수 있다.

체크리스트의 하나라도 막히면 06-01의 해당 절로 돌아가 다시 읽는다. 학습은 *빈틈을 찾는 것*이 절반이다.
