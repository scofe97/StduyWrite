# Kafka 공통 정책 스타터 패턴

---

> 조직에서 Kafka 운영이 어려워지는 진짜 원인은 *각 팀이 같은 결정을 따로 하기* 때문이다. A 팀은 retry 3회, B 팀은 무한 retry, C 팀은 manual ack, D 팀은 auto commit. 장애 시 *각자 다른 모드로 부서지는* 시스템을 누구도 한눈에 진단할 수 없다. 이 글은 이 문제를 *공통 starter*로 풀어내는 패턴을 정리한다. TPS `message-lib`이 그 한 사례다.



## 학습 목표

> *공통 starter*가 Kafka 운영 거버넌스를 어떻게 코드 한 곳으로 집약하는지 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. 팀별 결정의 분산이 만드는 *진단 불가능성*과 운영 비용을 설명할 수 있다.
2. Spring Boot autoconfiguration이 starter 패턴에서 어떤 역할을 하는지 설명할 수 있다.
3. `EnvironmentPostProcessor`로 디폴트를 *부팅 초기에* 강제 주입하는 패턴의 의의를 설명할 수 있다.
4. 공통 starter가 강제해야 하는 정책(retry/ack/DLT/idempotency)의 우선순위를 말할 수 있다.
5. 공통 starter의 *오버라이드 허용 범위*를 어떻게 정의하는지 설명할 수 있다.



## 1. 왜 공통 정책이 필요한가

> 같은 외부 시스템과 같은 도메인 위에서 일하는 컨슈머들이 *각자 다른 안전망*을 들고 있으면, 사고 시 디버깅 비용이 컨슈머 수에 비례해 늘어난다.

증상 예시:

- 서비스마다 다른 retry 곡선 탓에 같은 외부 API 장애가 서비스마다 다른 시간에 다른 패턴으로 회복
- 서비스마다 다른 DLT 이름 (`topic.dlt`, `topic-dlt`, `topic.dead`) 탓에 운영 알람 룰이 서비스 수만큼 늘어남
- 서비스마다 다른 로그 포맷 탓에 한 trace ID로 여러 서비스를 가로지를 수 없음
- producer 측 `enable.idempotence`가 일부 서비스만 켜져 있어 중복 메시지 원인 추적이 한 단계 더 어려워짐

이 모두를 *코드 리뷰 가이드라인*으로 해결하려고 하면 새 서비스가 생길 때마다 같은 이야기를 반복하게 된다. 공통 starter로 *기본값 자체를 갖춰* 두면, 의존 모듈은 *벗어날 때만 명시*하면 된다.



## 2. 공통 starter가 갖춰야 할 5가지

> Spring Boot starter 형태로 라이브러리를 만든다고 가정한다. 패키지 이름 예시는 `company-kafka-spring-boot-starter`.

| 항목 | 기본값 (예시) | 강제 vs 권고 |
|------|-----------|--------------|
| Producer EOS 트리오 | `acks=all`, `retries=3`, `enable.idempotence=true` | 강제 |
| Consumer deserializer 위임 | `ErrorHandlingDeserializer` + delegate=Avro/Json | 강제 |
| Offset commit | `enable.auto.commit=false`, ack-mode=BATCH | 강제 |
| 글로벌 에러 핸들러 | `DefaultErrorHandler` + DLT recoverer + 백오프 | 강제 (오버라이드 가능) |
| DLT 토픽 네이밍 | `<원본>.dlt` 또는 `tps.<env>.dlq` 단일 | 권고 (조직 정책) |
| 공통 로그 포맷 | topic/partition/offset/key/groupId/traceId/retry/exception | 권고 |
| Metrics 표준 라벨 | `kafka_consumer_failures_total{topic, exception_class}` | 권고 |

"강제"는 *기본값으로 자동 설정*되고, 의존 모듈이 yaml로 명시 오버라이드한 경우에만 변경된다. "권고"는 starter가 *도구를 제공*하고, 도입 여부는 모듈이 결정한다.



## 3. 기본값 주입의 두 가지 경로

> 같은 "기본값 깔기"라도 *언제 깔리느냐*에 따라 패턴이 둘로 갈린다. 둘 다 알아야 새 항목을 어디에 두는지 결정할 수 있다.

### 3-1. EnvironmentPostProcessor — 프로퍼티 단계

`spring.kafka.*` 프로퍼티가 *Spring Boot의 `KafkaAutoConfiguration`이 읽기 전*에 깔린다. application.yml에서 같은 키를 선언하면 자연스럽게 오버라이드된다.

```java
public class CompanyKafkaDefaultsEPP implements EnvironmentPostProcessor {
    public void postProcessEnvironment(ConfigurableEnvironment env, SpringApplication app) {
        Map<String, Object> defaults = new LinkedHashMap<>();
        defaults.put("spring.kafka.producer.acks", "all");
        defaults.put("spring.kafka.producer.retries", "3");
        defaults.put("spring.kafka.producer.properties.enable.idempotence", "true");
        defaults.put("spring.kafka.consumer.enable-auto-commit", "false");
        env.getPropertySources().addLast(new MapPropertySource("companyKafkaDefaults", defaults));
    }
}
```

`META-INF/spring.factories`에 등록하면 의존 모듈이 import한 것만으로 자동 적용된다. *프로퍼티로 표현 가능한 모든 설정*은 이 경로가 가장 자연스럽다.

TPS의 `KafkaDefaultsEnvironmentPostProcessor`(`04-01 §3`)가 이 패턴의 실례다.

### 3-2. AutoConfiguration — 빈 단계

빈(예: `DefaultErrorHandler`, `KafkaTemplate`, `RetryTopicConfigurer`)으로 표현해야 하는 정책은 `@AutoConfiguration`으로 등록한다.

```java
@AutoConfiguration(after = KafkaAutoConfiguration.class)
public class CompanyKafkaErrorConfig {
    @Bean
    public CommonErrorHandler kafkaErrorHandler(KafkaTemplate<String, byte[]> template) {
        DeadLetterPublishingRecoverer recoverer = ...;
        ExponentialBackOff backOff = new ExponentialBackOff(1000L, 2.0);
        backOff.setMaxElapsedTime(7000L);
        DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backOff);
        handler.addNotRetryableExceptions(DeserializationException.class, ValidationException.class);
        return handler;
    }
}
```

`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`에 한 줄 등록. 의존 모듈은 `@Import`나 `@EnableXxx` 없이 자동으로 받는다.

TPS의 `KafkaErrorConfig`(`05-04`)와 `KafkaRetryTemplateConfig`(`04-01 §5`)가 이 패턴의 실례다.



## 4. 정책 오버라이드를 허용하되 *벗어남*은 명시적으로

> 강제 정책이 의존 모듈의 *예외 사정*을 막으면 운영이 경직된다. 그러나 *몰래 벗어나는*는 사고로 이어진다. 두 요구를 동시에 만족시키려면 오버라이드를 *허용하되 흔적이 남도록* 설계한다.

기법 셋:

### 4-1. `@ConditionalOnMissingBean`

starter의 기본 빈에 `@ConditionalOnMissingBean`을 붙여 *의존 모듈이 자기 빈을 등록하면 자동으로 양보*한다. 이 경우 모듈의 빈이 *명시적 등록*이므로 git diff에 흔적이 남는다.

### 4-2. `addLast` 프로퍼티 주입

EnvironmentPostProcessor에서 `addLast`로 등록하면 application.yml의 키가 우선한다. *프로퍼티 오버라이드*는 actuator `/env`에서 출처가 그대로 보이므로 운영자가 *어디서 어떻게 바뀌었는지* 추적 가능하다.

### 4-3. ConfigurationProperties로 노출

대표 정책은 ConfigurationProperties 클래스로 노출해 *yaml로 통제할 수 있게* 한다.

```yaml
company:
  kafka:
    retry:
      max-attempts: 5
      initial-interval-ms: 2000
    dlt:
      naming-strategy: SUFFIX  # SUFFIX | SINGLE_DLQ
```

```java
@ConfigurationProperties(prefix = "company.kafka")
public class CompanyKafkaProperties {
    private Retry retry = new Retry();
    private Dlt dlt = new Dlt();
    // ...
}
```

이 패턴은 *오버라이드를 starter가 알고 있는 항목으로 제한*하는 효과가 있다. 무엇이 바뀔 수 있고 무엇이 강제인지 yaml 스키마로 한눈에 드러난다.



## 5. 도입 단계 — 0에서 운영까지

> 새 starter를 처음 도입할 때 *모든 정책을 한 번에 박지 않는다*. 단계로 도입하면 각 단계에서 만나는 문제를 흡수할 시간이 생긴다.

1단계 — *Producer EOS 트리오만 강제*. 가장 부작용 없고 효과 큰 항목. 모듈은 yaml 변경 없이 자동 수혜.

2단계 — *Consumer ErrorHandlingDeserializer 위임 + auto-commit=false*. 역직렬화 실패가 컨슈머를 죽이지 않게 안전망부터.

3단계 — *글로벌 DefaultErrorHandler + DLT 단일 토픽*. 여기서 DLT 운영(알람, 재처리 워커) 합의가 함께 가야 한다.

4단계 — *공통 로그 포맷 + Metrics 라벨*. 관측성은 정책이 안정된 뒤 얹는다. 라벨 변경은 메트릭 백업과 알람 룰 영향이 크다.

5단계 — *@RetryableTopic 전용 KafkaTemplate*. retry topic 패턴이 필요한 컨슈머가 등장한 시점에 추가. 동시에 *글로벌 핸들러와의 정책 불일치 위험*(01-13)을 인지한 가드 테스트도 함께.

이 순서는 *부작용 위험 낮은 것부터 점진적*이라는 원칙을 따른다. 1·2·3은 거의 모든 도메인에 안전하고, 4·5는 도메인별 합의 비용이 따른다.



## 6. 안티패턴 — 자주 보이는 실수

> 공통 starter를 만들거나 도입할 때 자주 부서지는 자리.

### 6-1. starter가 KafkaTemplate을 *하나만* 강제

byte[] 직렬화기 KafkaTemplate만 제공하면 Avro `SpecificRecord` 발행이 안 된다(TPS `KafkaRetryTemplateConfig`의 출발점). *기본 1개 + 특수 용도용 명시 빈*이 안전한 모델이다.

### 6-2. 강제 정책을 `@Configuration`에 박음

`@Configuration`은 의존 모듈이 *명시 import*해야 적용된다. starter는 `@AutoConfiguration` + AutoConfiguration.imports로 가야 의존 모듈이 *import만* 하면 자동 수혜를 받는다.

### 6-3. 모듈 yaml 오버라이드 막힘

starter가 `addFirst`로 프로퍼티를 등록하거나 `@ConditionalOnMissingBean` 없이 빈을 강제하면 의존 모듈이 *합리적인 예외 사정*을 표현할 수 없다. 운영 변경이 starter 릴리스에 묶이게 되어 의존 모듈 팀이 starter를 *우회*하는 코드를 짠다.

### 6-4. 강제·권고 구분 없음

"이 starter를 쓰면 이건 강제고 저건 권고"라는 문서가 없으면, 의존 모듈은 *모든 게 강제*라고 가정해 운영 자율성을 잃거나, *모든 게 권고*라고 가정해 안전망을 우회한다. README에 *항목별 등급 표*를 명시한다(§2 표가 그 예).



## 7. TPS의 message-lib을 이 관점에서 다시 보기

> TPS `message-lib`은 *완성된 공통 starter*는 아니지만 이 패턴의 핵심 부품을 갖춰 두고 있다.

| 항목 | `message-lib` 위치 | 상태 |
|------|---------------------|------|
| EOS 트리오 | `KafkaDefaultsEnvironmentPostProcessor` | ✅ 자동 적용 |
| `ErrorHandlingDeserializer` 위임 | `KafkaDefaultsEnvironmentPostProcessor` | ✅ 자동 적용 |
| `auto-commit=false` | `KafkaDefaultsEnvironmentPostProcessor` | ✅ 자동 적용 |
| 글로벌 `DefaultErrorHandler` + DLT | `KafkaErrorConfig` | ✅ 자동 등록 |
| DLT 토픽 — 단일 `tps.v305p.dlq` | `Topics.DLQ` + `KafkaErrorConfig` | ✅ |
| `@RetryableTopic` 전용 Avro KafkaTemplate | `KafkaRetryTemplateConfig` | ✅ 자동 등록 (Schema Registry 조건부) |
| 헤더 byte[]에서 String 자동 디코딩 | `KafkaHeaderMapperConfig` | ✅ 자동 등록 |
| 외부 JSON 토픽용 listener factory | `KafkaJsonConsumerConfig` | ✅ 자동 등록 |
| Outbox + Inbox 패턴 | `application/outbox/` + `application/inbox/` | ✅ |
| 공통 로그 포맷 | (의존 모듈 책임) | ❌ 아직 미통합 |
| Metrics 표준 라벨 | (의존 모듈 책임) | ❌ 아직 미통합 |
| ConfigurationProperties로 정책 노출 | 일부 (`OutboxProperties`, `TopicProperties`) | ⚠️ 부분 — 백오프/DLT 정책은 코드에 박혀 있음 |

`message-lib`은 *공통 starter의 강제 항목은 거의 갖췄고, 권고 항목과 ConfigurationProperties 노출이 부분적*인 상태로 볼 수 있다. 공통 로그 포맷과 Metrics 라벨을 starter 레벨로 끌어올리면 더 완성된 공통 정책 인프라가 된다.



## 8. 정리

공통 starter의 본질은 *각 팀이 같은 결정을 따로 내리지 않게 하는 것*이다. 결정 자체를 starter에 박고, 의존 모듈은 *벗어날 때만 명시*하면 된다. 두 경로(EnvironmentPostProcessor와 AutoConfiguration)로 기본값을 깔고, 세 가지 오버라이드 기법(`@ConditionalOnMissingBean`, `addLast`, ConfigurationProperties)으로 자율성을 보장한다. 도입은 *부작용 낮은 단계부터 점진적*이며, 강제와 권고의 경계를 README로 명시한다.

다음 학습 단계는 [04-01.message-lib config 5개 클래스 종합](04-01.message-lib%20config%205개%20클래스%20종합.md)에서 본 패턴의 *실제 구현 디테일*을, [05-03.Kafka 예외 처리 통합](../05_ConsistencyPattern/05-03.Kafka%20예외%20처리%20통합.md)에서 *글로벌 정책의 운영 의사결정*을 본다.

## 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. "팀별 결정의 분산"이 만드는 진짜 운영 비용은?

장애 시 *진단 불가능성*이다. A 팀은 retry 3회·B 팀은 무한 retry·C 팀은 manual ack·D 팀은 auto commit이라면, 같은 메시지 큐에 같은 종류 사고가 나도 *팀마다 다른 모드로 부서지므로* 표준 플레이북이 안 만들어진다. 운영자는 매번 그 팀이 어떻게 설정했는지부터 알아야 하고, 그 사이 사고가 더 커진다. 공통 starter가 풀어주는 본질은 *결정의 통일*이고, 그 부산물이 *진단·복구의 표준화*다.

### Q2. Spring Boot autoconfiguration이 공통 starter에서 하는 역할은?

라이브러리를 의존만 추가하면 *별도 설정 없이* 디폴트가 적용되게 만든다. `@ConditionalOnProperty`·`@ConditionalOnMissingBean`으로 *애플리케이션이 명시적으로 오버라이드하지 않은 자리*에만 starter 빈이 자동 등록된다. 즉 정책을 강제하면서도 *특수 케이스 회피 가능한 출구*를 남긴다. 단순한 라이브러리 API와의 차이가 바로 이 자동성이다.

### Q3. `EnvironmentPostProcessor`로 디폴트를 박는 이유는?

`application.yml` 로딩보다도 *더 일찍* 환경변수에 디폴트를 주입해야 다른 자동설정이 그 값을 본 뒤 자기 빈을 구성한다. 예를 들어 `auto.offset.reset=latest`나 `specific.avro.reader=true` 같은 디폴트는 `KafkaProperties` 바인딩이 일어나기 전에 박혀야 효과가 있다. 일반 `@Configuration` 클래스에서 set하면 늦거나 외부 설정에 덮인다.

### Q4. 공통 starter가 강제할 정책의 우선순위는?

(1) 데이터 *유실*을 막는 것 → Producer idempotent + acks=all + max.in.flight=5·5, ErrorHandlingDeserializer. (2) 데이터 *중복*의 부작용을 막는 것 → Inbox 패턴, manual ack(필요 시), `addNotRetryableExceptions`. (3) 운영 *가시성* → 단일 DLT·표준 헤더, traceparent 보존. (4) 그 외 편의(직렬화 래퍼, Topic enum 등). 같은 순서로 적용되어야 정책 충돌 시 *유실 방지*가 항상 이긴다.

### Q5. 공통 starter의 오버라이드 허용 범위는 어떻게 정해야 하나?

기준은 "정책의 안전 경계를 *약화*시키는 오버라이드는 금지, *강화*는 허용". 예를 들어 retry 횟수를 늘리는 것은 허용해도, `addNotRetryableExceptions`에서 `AvroSerializationException`을 빼는 것은 금지다. 이걸 코드로 강제하려면 `ConfigurationProperties`로 일부만 열고, 그 외는 starter 안에 *상수*로 박는다. 후자는 *PR 리뷰 게이트*가 같이 따라와야 우회 변경을 잡을 수 있다.
