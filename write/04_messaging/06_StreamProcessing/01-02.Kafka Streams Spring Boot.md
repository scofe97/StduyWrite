# Kafka Streams — Spring Boot 통합

---

> Spring Boot에서 Kafka Streams는 `@EnableKafkaStreams`와 `StreamsBuilder` 빈으로 시작한다. 자동설정이 application·broker·serde 설정을 모아 `KafkaStreams` 인스턴스를 만들고, 라이프사이클(start/stop/clean-up)을 빈 컨테이너에 위임한다. 이 글은 그 통합의 *키 빈·키 속성*을 정리한다.



## 학습 목표

> Spring Boot에서 Kafka Streams를 *언제 어떤 빈으로* 구성해야 하는지 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. `@EnableKafkaStreams`가 자동 등록하는 빈(`StreamsBuilder`, `StreamsBuilderFactoryBean`)을 설명할 수 있다.
2. `application.id`·`bootstrap-servers`·`default-key-serde`/`value-serde` 핵심 설정을 말할 수 있다.
3. `processing.guarantee=exactly_once_v2`를 Spring 설정으로 표현하는 방법을 설명할 수 있다.
4. `StreamsBuilderFactoryBean.setStateListener()`로 상태 전이를 감지하는 패턴을 설명할 수 있다.
5. clean-up(로컬 상태 삭제)을 *언제 켜고 언제 꺼야* 하는지 설명할 수 있다.

# Spring Boot 설정

----

> Kafka Streams 애플리케이션의 본질은 **read → process → write** 3단계로 압축됩니다. 입력 토픽을 구독하고, DSL 연산(`filter`, `map`, `groupBy`, `aggregate`, `join` 등)으로 변환하고, 다른 토픽 또는 같은 토픽의 다른 파티션으로 출력합니다. 이후 등장하는 모든 복잡도(state store, windowing, exactly-once)는 이 골격 위에 얹히는 추가 책임입니다. Spring Boot 설정도 이 3단계 골격을 받쳐주는 인프라(`StreamsBuilder` 라이프사이클·Serde·토폴로지 등록 방식)부터 시작합니다.

## @EnableKafkaStreams와 StreamsBuilderFactoryBean

@EnableKafkaStreams를 선언하면 2가지가 자동으로 등록됩니다.

1. StreamsBuilderFactoryBean: StreamsBuilder의 라이프사이클(생성, 시작, 종료)을 Spring이 관리합니다.
2. StreamsBuilder 빈: @Autowired로 주입받아 토폴로지를 정의할 수 있습니다.

순수 Kafka Streams API에서는 StreamsBuilder로 토폴로지를 정의한 뒤, KafkaStreams를 직접 생성하고 start/close를 호출해야 합니다. Spring Boot에서는 이 작업을 자동화 해줍니다.

```java
// 순수 API: 직접 생성, 시작, 종료
StreamsBuilder builder = new StreamsBuilder();

// ... 토폴로지 정의 ...
KafkaStreams streams = new KafkaStreams(builder.build(), props);
streams.start();
Runtime.getRuntime().addShutdownHook(new Thread(streams::close));
```

```java
// Spring Boot: @EnableKafkaStreams 선언만으로 위 과정이 자동화됨
@Configuration
@EnableKafkaStreams
public class StreamsAppConfig {
    // StreamsBuilder는 Spring이 자동 생성하여 빈으로 등록
}
```

## KafkaStreamsConfiguration 빈 등록

@EnableKafkaStreams를 사용하려면 defaultKafkaStreamConfig라는 이름의 빈을 등록해야 합니다. 이 이름이 정확히 일치해야 Spring Kafka가 KafkaStreams 인스턴스를 생성합니다.

```java
@Bean(name = KafkaStreamsDefaultConfiguration.DEFAULT_STREAMS_CONFIG_BEAN_NAME)
public KafkaStreamsConfiguration kStreamsConfig() {
    Map<String, Object> props = new HashMap<>();
    props.put(StreamsConfig.APPLICATION_ID_CONFIG, "my-streams-app");
    props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapServers);
    props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, Serdes.StringSerde.class);
    props.put("schema.registry.url", schemaRegistryUrl);
    props.put("specific.avro.reader", true);
    props.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "earliest");
    return new KafkaStreamsConfiguration(props);
}
```

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:19092
    streams:
      application-id: my-streams-app
      properties:
        schema.registry.url: http://localhost:18081
        specific.avro.reader: true
        default.key.serde: org.apache.kafka.common.serialization.Serdes$StringSerde
        default.value.serde: io.confluent.kafka.streams.serdes.avro.SpecificAvroSerde
```

| 속성                             | 값 예시                    | 역할                                                         |
| -------------------------------- | -------------------------- | ------------------------------------------------------------ |
| `APPLICATION_ID_CONFIG`          | `"my-streams-app"`         | Kafka Streams 앱의 고유 식별자. Consumer Group ID로도 사용되며, 내부 토픽(changelog, repartition)의 접두사가 된다. |
| `BOOTSTRAP_SERVERS_CONFIG`       | `"localhost:19092"`        | Kafka/Redpanda 브로커 주소                                   |
| `DEFAULT_KEY_SERDE_CLASS_CONFIG` | `Serdes.StringSerde.class` | 키의 기본 직렬화 방식                                        |
| `schema.registry.url`            | `"http://localhost:18081"` | Avro Schema Registry 주소. SpecificAvroSerde가 스키마를 조회하는 데 사용한다. |
| `specific.avro.reader`           | `true`                     | `true`면 SpecificRecord(생성 클래스), `false`면 GenericRecord(Map 형태)로 역직렬화한다. |
| `AUTO_OFFSET_RESET_CONFIG`       | `"earliest"`               | `earliest` 필수: 처음 기동 시 모든 이벤트를 읽어야 State Store를 완전히 재구성할 수 있다. |

## 토폴로지 등록 @Bean vs @Component 패턴

토폴로지를 등록하는 방법은 2가지 입니다.

```java
// 패턴 A: @Bean 리턴
@Configuration
public class OrderStreamsConfig {
    @Bean
    public KStream<String, Event> topology(StreamsBuilder builder) {
        // 토폴로지 정의 + KStream 반환
        return builder.stream("events");
    }
}

// 패턴 B: @Component + 생성자 주입 (토폴로지를 독립 클래스로 분리)
@Component
public class OrderStreamTopology {
    public OrderStreamTopology(StreamsBuilder builder) {
        // 생성자에서 토폴로지 정의 (반환값 없음)
        builder.stream("orders")
               .groupByKey()
               .aggregate(/* ... */);
    }
}
```

- 패턴 B의 장점은 토폴로지를 독립된 클래스로 구분할 수 있습니다.
- StreamsBuilder를 생성자 주입으로 받으면 Spring Bean이 빈 초기화 과정에서 자동으로 주입하므로, 모든 토폴로지가 하나의 StreamBuilder에 등록됩니다.

# Serde(Serializer/Deserializer)

---

> Kafka Stream에서 Serde는 토폴로지의 각 단계에서 직렬화/역직렬화 방식을 지정하는 역할을 수행합니다. 기본 타입은 내장 Serde를 사용하고, 도메인 객체는 직접 수행합니다.

```java
// 기본 타입
Serdes.String()   // String
Serdes.Long()     // Long
Serdes.Integer()  // Integer
Serdes.ByteArray() // byte[]
```

## Json Serde 커스텀 구현

프로젝트에서 도메인 객체를 JSON으로 직렬화하면 Serde<T> 인터페이스를 구현합니다.

```java
public class JsonSerde<T> implements Serde<T> {
    private final ObjectMapper mapper = new ObjectMapper();
    private final Class<T> type;

    public JsonSerde(Class<T> type) { this.type = type; }

  	// 직렬화
    @Override
    public Serializer<T> serializer() {
        return (topic, data) -> {
            try { return mapper.writeValueAsBytes(data); }
            catch (Exception e) { throw new SerializationException(e); }
        };
    }

  	// 역직렬화
    @Override
    public Deserializer<T> deserializer() {
        return (topic, data) -> {
            try { return mapper.readValue(data, type); }
            catch (Exception e) { throw new SerializationException(e); }
        };
    }
}

// 사용 예시
Materialized.<String, Post, KeyValueStore<Bytes, byte[]>>as("posts-store")
    .withKeySerde(Serdes.String())
    .withValueSerde(new JsonSerde<>(Post.class));
```

## SpecificAvroSerde(Avro + Schema Registry)

Avro를 사용하면 Confluent의 SpecificAvroSerde를 사용한다. Schema Registry와 통합되어 호환성 검증까지 자동으로 수행한다는 점에서 JsonSerde와 근본적으로 다릅니다.

| 항목            | JsonSerde                              | SpecificAvroSerde                                 |
| --------------- | -------------------------------------- | ------------------------------------------------- |
| **직렬화 포맷** | JSON (텍스트)                          | Avro (바이너리)                                   |
| **스키마 검증** | 없음 (POJO 필드 불일치 시 런타임 에러) | Schema Registry에서 호환성 자동 검증              |
| **클래스 생성** | 수동 POJO 작성                         | `.avsc`에서 빌드 플러그인이 자동 생성             |
| **크기**        | 필드명 포함으로 큼                     | 바이너리 인코딩으로 작음                          |
| **의존성**      | Spring Kafka 내장                      | `io.confluent:kafka-streams-avro-serde` 별도 추가 |

### configure() 메서드 

SpecificAvroSerde는 생성 직후 configure()를 반드시 호출해야 합니다. 이는  SchemaRegistry URL과 역직렬화 옵션을 주입하는 과정이며 이 호출을 하지 않으면 Registry와 연결할 수 없어 직렬화/역직렬화가 실패하게됩니다.

```java
SpecificAvroSerde<SpecificRecord> avroSerde = new SpecificAvroSerde<>();

// 설정 호출 필수
avroSerde.configure(
    Map.of(
        "schema.registry.url", schemaRegistryUrl,
        "specific.avro.reader", true
    ),
    false  // isKey: false면 value serde, true면 key serde
);
```

### 입력 Serde != 출력 Serde  혼합 패턴

토픽에서 읽을 때는 Avro, Sate Store에 쓸 때는 JSON을 사용할수 있는데, 입력 이벤트는 Producer가 Avro로 발행했고, State Store 값은 Avro 생성 클래스가 아닌 직접 작성한 POJO일 때는 아래처럼 사용하면 됩니다.

```java
// 입력: Avro (SpecificAvroSerde)
KStream<String, SpecificRecord> stream = builder.stream(
    "order.events",
    Consumed.with(Serdes.String(), avroSerde)
);

// 출력: JSON (JsonSerde) → State Store
stream.groupByKey().aggregate(
    () -> null,
    (key, event, view) -> { /* Avro 이벤트 → OrderView 변환 */ },
    Materialized.<String, OrderView, KeyValueStore<Bytes, byte[]>>as("orders-store")
        .withKeySerde(Serdes.String())
        .withValueSerde(new JsonSerde<>(OrderView.class))
);
```

## 하나의 토픽이 다중 이벤트를 지니는 경우

TopicRecordNameStrategy를 사용하면 하나의 토픽에 여러 Avro 타입이 공존하게 됩니다. 이 경우 Consumer 측에서는 SpecificRecord로 선언하고 instanceof로 분기한다.

```java
// stream avro 생성
KStream<String, SpecificRecord> stream = builder.stream(
    "order.events",
    Consumed.with(Serdes.String(), avroSerde)
);

stream.groupByKey().aggregate(
    () -> null,
    (orderId, event, currentView) -> {
      	// 타입 1
        if (event instanceof OrderCreated created) {
            return new OrderView(created.getOrderId(), created.getAmount());
        } 
      	// 타입 2
      	else if (event instanceof OrderPaid paid) {
            if (currentView == null) return null;  // 순서 역전 방어
            currentView.setPaid(true);
            return currentView;
        }
        return currentView;  // 알 수 없는 타입은 무시
    },
    Materialized.as("orders-store")
);
```

# DSL 연산

---

## Stateless vs Stateful 처리

스트림 DSL 처리는 상태 유무에 따라 2가지로 나뉩니다. Stateless는 이전 이벤트의 상태 없이 현재 이벤트만 처리하고, Stateful은 State Store에 중간 결과를 유지하며 처리합니다.

```java
Stateless:  입력 레코드 → 변환 → 출력 레코드  (1:1 매핑, 상태 불필요)
Stateful:   입력 레코드 + State Store → 갱신된 결과 → 출력  (누적 연산)
```

## Stateless 연산

| 연산            | 용도              | Repartition | State Store |
| --------------- | ----------------- | ----------- | ----------- |
| `filter`        | 조건 필터링       | 없음        | 없음        |
| `mapValues`     | 값 변환 (키 유지) | 없음        | 없음        |
| `map`           | 키+값 변환        | **발생**    | 없음        |
| `flatMapValues` | 1:N 값 분리       | 없음        | 없음        |
| `selectKey`     | 키 변경           | **발생**    | 없음        |
| `peek`          | 사이드 이펙트     | 없음        | 없음        |
| `merge`         | 스트림 합치기     | 없음        |             |

```java
// 조건 필터링
KStream<String, Order> highValue = orders
    .filter((key, order) -> order.getAmount() >= 10_000);

// 값만 변경
KStream<String, OrderSummary> summaries = orders
    .mapValues(order -> new OrderSummary(order.getId(), order.getAmount()));

// 사이드 이펙트
KStream<String, Order> logged = orders
    .peek((key, order) -> log.info("Processing order: {}", order.getId()));
```



## Stateful 연산

|             | 용도                  | Repartition | State Store |
| ----------- | --------------------- | ----------- | ----------- |
| `groupByKey` | 현재 키 그룹핑        | 없음        | 필요        |
| `groupBy`   | 새 키 그룹핑          | **발생**    | 필요        |
| `count`     | 건수 집계             | -           | 필요        |
| `reduce`    | 값 축소               | -           | 필요        |
| `aggregate` | 범용 집계 (타입 변환) | -           | 필요        |

```java
// 현재 키 기준으로 그룹핑
KGroupedStream<String, Order> grouped = orders.groupByKey();

// 새로운 키로 그룹핑
KGroupedStream<String, Order> byRegion = orders
    .groupBy(
        (key, order) -> order.getRegion(),
        Grouped.with(Serdes.String(), orderSerde)
    );

// 두 값을 하나로 축소
KTable<String, Order> latestOrders = orders
    .groupByKey()
    .reduce(
        (oldOrder, newOrder) -> newOrder,  // 항상 최신 값
        Materialized.as("latest-order-store")
    );

//초기값 + 집계 함수
@Bean
public KStream<String, SpecificRecord> orderAggregation(StreamsBuilder builder) {
    SpecificAvroSerde<SpecificRecord> avroSerde = new SpecificAvroSerde<>();
    avroSerde.configure(
        Map.of("schema.registry.url", schemaRegistryUrl, "specific.avro.reader", true),
        false
    );

    KStream<String, SpecificRecord> stream = builder.stream(
        "order.events",
        Consumed.with(Serdes.String(), avroSerde)
    );

    stream.groupByKey()
        .aggregate(
            OrderStats::new,                                    // 초기값
            (userId, event, stats) -> stats.apply(event),       // 집계 함수
            Materialized.<String, OrderStats, KeyValueStore<Bytes, byte[]>>as("order-stats-store")
                .withKeySerde(Serdes.String())
                .withValueSerde(new JsonSerde<>(OrderStats.class))
        );

    return stream;
}
```

# 조인 연산

----

| 항목                | Stream-Stream      | Stream-Table       | Stream-GlobalKTable  |
| ------------------- | ------------------ | ------------------ | -------------------- |
| **왼쪽**            | KStream            | KStream            | KStream              |
| **오른쪽**          | KStream            | KTable             | GlobalKTable         |
| **시간 윈도우**     | 필수 (JoinWindows) | 없음               | 없음                 |
| **Co-partitioning** | 필수               | 필수               | **불필요**           |
| **오른쪽 데이터**   | 윈도우 내 이벤트   | 최신 상태          | 전체 데이터 복제     |
| **적합한 사례**     | 주문+결제 매칭     | 이벤트 보강        | 참조 데이터 조회     |
| **메모리 사용**     | 윈도우 크기에 비례 | 파티션 크기에 비례 | **전체 데이터 크기** |

## 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. `@EnableKafkaStreams`가 등록하는 빈은?

`StreamsBuilder`, `StreamsBuilderFactoryBean`, 그리고 그 안에서 만들어지는 `KafkaStreams` 인스턴스다. 개발자는 `@Bean public KStream<...>` 메서드를 등록하면, `StreamsBuilderFactoryBean`이 그 빈을 모아 토폴로지를 만들고 KafkaStreams를 부팅한다. 라이프사이클은 Spring 컨테이너가 관리하므로 명시적 start/stop이 거의 필요 없다.

### Q2. `application.id`가 단순 이름이 아니라 *애플리케이션 정체성*인 이유는?

`application.id`가 Consumer Group ID, changelog 토픽 prefix, repartition 토픽 prefix를 모두 결정한다. 변경하면 *다른 application.id가 되어* 새 Consumer Group으로 시작하고, State Store도 처음부터 재구축된다. 그래서 운영 중인 streams 앱의 application.id를 바꾸는 것은 *데이터 마이그레이션 수준*의 작업이다. 환경별 prefix는 application.id에 박지 말고 토픽 prefix에 둔다.

### Q3. `processing.guarantee=exactly_once_v2`를 Spring에서 어떻게 설정하나?

`application.yml`의 `spring.kafka.streams.properties` 아래 `processing.guarantee=exactly_once_v2`를 둔다. v1과 v2의 차이는 *트랜잭션 producer 수*다. v1은 task별 producer를 두지만, v2는 thread별 하나로 압축해 처리량을 크게 끌어올렸다. 운영에서는 항상 v2를 쓴다.

### Q4. `setStateListener`로 어떤 상태 전이를 감지해야 하나?

`KafkaStreams.State`는 `CREATED → REBALANCING → RUNNING → PENDING_SHUTDOWN`을 따라간다. 가장 중요한 전이는 `REBALANCING ↔ RUNNING`이다. 리밸런스가 자주 또는 오래 일어나면 *State Store 재구축 비용*과 *처리 지연*이 발생한다. 메트릭으로 `state-rebalance-rate`나 listener 측 로그를 노출해 *리밸런스가 비정상적으로 잦은* 패턴을 감지한다. ERROR 상태로 떨어졌다면 즉시 알람을 띄워야 한다.

### Q5. clean-up(`KafkaStreams.cleanUp()`)을 언제 켜고 언제 꺼야 하나?

clean-up은 *로컬 State Store 디렉터리를 삭제*한다. application.id를 변경하거나 *처음부터 재처리*가 필요할 때만 켠다. 운영 환경에서 매 부팅마다 켜면 *State Store 재구축이 매번 일어나* 리밸런스가 지나치게 길어지고 changelog 토픽에서 데이터를 다시 페치하는 트래픽이 폭발한다. Spring에서는 `setCleanupConfig`로 `cleanupOnStart=false`, `cleanupOnStop=false`가 기본이고, 그대로 두는 게 안전하다.
