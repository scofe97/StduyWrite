# Temporal Spring Boot 통합

---

> Spring Boot 애플리케이션이 Temporal Server와 통신하려면 네 개의 Bean(`WorkflowServiceStubs`, `WorkflowClient`, `WorkerFactory`, `Worker`)을 구성한다. Kafka `@KafkaListener`가 받은 이벤트는 두 가지 방식(Start Workflow / Signal)으로 Temporal에 전달되고, 멱등성은 Workflow ID 설계로 확보한다. 본 문서는 강의 §7.22, §7.24의 Config 구성과 ConsumerListener 통합 코드를 표준 패턴으로 정리한다.



## 학습 목표

> Spring Boot가 Temporal Server와 *어떻게 연결되고*, Kafka 이벤트가 *어떻게 Workflow로 흘러 들어가는지* 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. `WorkflowServiceStubs`, `WorkflowClient`, `WorkerFactory`, `Worker` 네 Bean의 책임을 설명할 수 있다.
2. `@KafkaListener`가 이벤트를 받은 뒤 Workflow를 시작하는 방식과 Signal을 보내는 방식을 구분할 수 있다.
3. Activity 구현 빈을 Spring에 등록하고 의존성 주입을 받는 방법을 설명할 수 있다.
4. Workflow ID 설계가 멱등성과 중복 방지를 어떻게 만드는지 설명할 수 있다.
5. Local Activity와 일반 Activity의 차이와 사용 시점을 설명할 수 있다.



## 1. 빌드 구성

> 의존성과 버전을 먼저 잡는다. 강의는 Kotlin 기반이지만 Java도 동일하다.

Gradle Kotlin DSL 기준 `build.gradle.kts` 예시는 다음과 같다.

```kotlin
dependencies {
    implementation("org.springframework.boot:spring-boot-starter")
    implementation("org.springframework.boot:spring-boot-starter-web")
    implementation("org.springframework.kafka:spring-kafka")

    implementation("io.temporal:temporal-sdk:1.24.2")
    implementation("io.temporal:temporal-kotlin:1.24.2")

    testImplementation("io.temporal:temporal-testing:1.24.2")
}
```

`temporal-sdk`가 핵심이다. `temporal-kotlin`은 코틀린 코루틴 지원과 함수형 API를 제공한다. 자바만 쓰면 `temporal-sdk`만 있어도 충분하다.

`application.yml`에 Temporal 접속 정보를 잡는다.

```yaml
temporal:
  target: 127.0.0.1:7233       # Temporal Server frontend gRPC
  namespace: default
  task-queue: order-task-queue
  worker:
    max-concurrent-activity-execution-size: 100
    max-concurrent-workflow-task-execution-size: 50
```

`target`은 Temporal Server의 frontend 서비스 주소다. `namespace`는 Temporal 안의 논리적 분리 단위로, 환경(dev/staging/prod)이나 팀 단위로 가른다. `task-queue`는 이 애플리케이션의 Worker가 처리할 큐 이름이다.



## 2. Temporal Config — Bean 4종

> 네 Bean의 책임 분리가 명확하다. 한 번 잡아 두면 그 뒤로 손댈 일이 거의 없다.

```java
@Configuration
@ConfigurationProperties(prefix = "temporal")
public class TemporalConfig {

    private String target;
    private String namespace;
    private String taskQueue;
    // getter / setter 생략

    @Bean(destroyMethod = "shutdown")
    public WorkflowServiceStubs workflowServiceStubs() {
        return WorkflowServiceStubs.newServiceStubs(
            WorkflowServiceStubsOptions.newBuilder()
                .setTarget(target)
                .build()
        );
    }

    @Bean
    public WorkflowClient workflowClient(WorkflowServiceStubs stubs) {
        return WorkflowClient.newInstance(
            stubs,
            WorkflowClientOptions.newBuilder()
                .setNamespace(namespace)
                .build()
        );
    }

    @Bean(destroyMethod = "shutdown")
    public WorkerFactory workerFactory(WorkflowClient client) {
        return WorkerFactory.newInstance(client);
    }

    @Bean
    public Worker orderWorker(
        WorkerFactory factory,
        OrderActivityImpl orderActivity,
        ShippingActivityImpl shippingActivity
    ) {
        Worker worker = factory.newWorker(taskQueue);
        worker.registerWorkflowImplementationTypes(OrderWorkflowImpl.class);
        worker.registerActivitiesImplementations(orderActivity, shippingActivity);
        return worker;
    }

    @Bean
    public ApplicationRunner startWorker(WorkerFactory factory) {
        return args -> factory.start();
    }
}
```

각 Bean의 책임:

| Bean | 책임 |
|------|------|
| `WorkflowServiceStubs` | Temporal Server와의 gRPC 연결 풀. 가장 비싼 자원이며 애플리케이션당 1개 |
| `WorkflowClient` | Workflow 시작·Signal·Query를 발행하는 API. namespace 단위로 분리 |
| `WorkerFactory` | Worker들을 만들고 관리하는 팩토리. JVM당 1개 |
| `Worker` | 특정 Task Queue를 polling하며 Workflow/Activity 코드를 실행 |

`registerWorkflowImplementationTypes`는 *Workflow 클래스 자체*를 등록한다. Temporal이 매 Workflow Execution마다 새 인스턴스를 만들어야 하므로 빈이 아닌 *클래스*를 넘긴다. 반면 `registerActivitiesImplementations`는 *빈 인스턴스*를 받는다. Activity는 Spring 빈으로 만들어 DB·외부 클라이언트를 주입받기 좋다.

`ApplicationRunner`로 `factory.start()`를 호출하는 이유는 Bean 등록만으로는 Worker가 polling을 시작하지 않기 때문이다. 명시적 `start()`가 있어야 Task Queue를 polling 한다.



## 3. Activity 구현 빈

> Activity는 외부 부수 효과의 영역이므로 Spring 빈으로 만들어 DB 클라이언트나 외부 API 클라이언트를 주입받는다.

```java
@ActivityInterface
public interface OrderActivity {
    PaymentResult charge(String orderId, BigDecimal amount);
    void reserveInventory(String orderId, List<String> skus);
    void notify(String orderId, NotificationType type);
}

@Component
public class OrderActivityImpl implements OrderActivity {

    private final PgClient pgClient;
    private final InventoryClient inventoryClient;
    private final NotificationClient notificationClient;

    public OrderActivityImpl(
        PgClient pgClient,
        InventoryClient inventoryClient,
        NotificationClient notificationClient
    ) {
        this.pgClient = pgClient;
        this.inventoryClient = inventoryClient;
        this.notificationClient = notificationClient;
    }

    @Override
    public PaymentResult charge(String orderId, BigDecimal amount) {
        String idempotencyKey = "charge-" + orderId;
        return pgClient.requestCharge(orderId, amount, idempotencyKey);
    }

    @Override
    public void reserveInventory(String orderId, List<String> skus) {
        inventoryClient.reserve(orderId, skus);
    }

    @Override
    public void notify(String orderId, NotificationType type) {
        notificationClient.send(orderId, type);
    }
}
```

Activity 메서드는 Workflow 인터페이스 메서드처럼 *비즈니스 의미*로 명명한다. `chargePayment` 같은 도메인 이름이 좋고, `callPgApiStep1` 같은 구현 디테일 이름은 피한다.

`idempotencyKey`가 중요하다. Temporal이 Activity를 자동 재시도할 때 PG사가 같은 키로 두 번 호출되는 상황이 정상이다. 키가 없으면 두 번 결제가 일어날 수 있다.



## 4. Workflow 구현

> Workflow 코드는 *빈이 아니다*. `Workflow.newActivityStub`로 Activity의 *프록시*를 얻어 호출한다.

```java
public class OrderWorkflowImpl implements OrderWorkflow {

    private final OrderActivity orderActivity = Workflow.newActivityStub(
        OrderActivity.class,
        ActivityOptions.newBuilder()
            .setStartToCloseTimeout(Duration.ofSeconds(30))
            .setRetryOptions(RetryOptions.newBuilder()
                .setInitialInterval(Duration.ofSeconds(1))
                .setBackoffCoefficient(2.0)
                .setMaximumAttempts(3)
                .build())
            .build()
    );

    private OrderStatus status = OrderStatus.STARTED;
    private boolean cancelled = false;

    @Override
    public void process(OrderRequest req) {
        status = OrderStatus.PAYMENT_PENDING;
        PaymentResult pr = orderActivity.charge(req.orderId(), req.amount());
        if (!pr.success()) {
            status = OrderStatus.FAILED;
            return;
        }

        if (cancelled) {
            // 취소 Signal이 도착했으면 환불 후 종료
            orderActivity.refund(req.orderId(), pr.transactionId());
            status = OrderStatus.CANCELLED;
            return;
        }

        status = OrderStatus.INVENTORY_PENDING;
        orderActivity.reserveInventory(req.orderId(), req.skus());

        status = OrderStatus.NOTIFY_PENDING;
        orderActivity.notify(req.orderId(), NotificationType.ORDER_CONFIRMED);

        status = OrderStatus.COMPLETED;
    }

    @Override
    public void cancel(String reason) {
        cancelled = true;
    }

    @Override
    public OrderStatus currentStatus() {
        return status;
    }
}
```

`Workflow.newActivityStub`이 반환하는 객체는 평범한 `OrderActivity` 구현처럼 보이지만, 실제로는 메서드 호출 시 Activity Task를 만들어 Server로 보내고 응답을 결정적으로 대기한다. `ActivityOptions`의 `RetryOptions`가 그 호출의 재시도 정책을 결정한다.

Workflow는 Spring 빈이 아니므로 `@Autowired`를 쓸 수 없다. DB·외부 API에 직접 접근하면 *결정성 위반*이 된다. 모든 외부 작업은 Activity로 위임한다.



## 5. Kafka 이벤트에서 Workflow로 — 두 패턴

> 이 부분이 강의 §7.24의 핵심이다. 같은 이벤트라도 *새 흐름 시작*인지 *기존 흐름에 정보 전달*인지에 따라 호출 방식이 갈린다.

### 5-1. 패턴 A — Start Workflow

새 비즈니스 흐름의 시작점(예: `order-created` 이벤트)을 받으면 Workflow를 *새로 시작*한다.

```java
@Component
public class OrderEventConsumer {

    private final WorkflowClient workflowClient;
    private final String taskQueue;

    public OrderEventConsumer(
        WorkflowClient workflowClient,
        @Value("${temporal.task-queue}") String taskQueue
    ) {
        this.workflowClient = workflowClient;
        this.taskQueue = taskQueue;
    }

    @KafkaListener(topics = "order-created", groupId = "order-workflow-starter")
    public void onOrderCreated(OrderCreatedEvent event) {
        OrderWorkflow workflow = workflowClient.newWorkflowStub(
            OrderWorkflow.class,
            WorkflowOptions.newBuilder()
                .setWorkflowId("order-" + event.orderId())
                .setTaskQueue(taskQueue)
                .setWorkflowIdReusePolicy(
                    WorkflowIdReusePolicy.WORKFLOW_ID_REUSE_POLICY_ALLOW_DUPLICATE_FAILED_ONLY)
                .build()
        );
        WorkflowClient.start(workflow::process, event.toRequest());
    }
}
```

`WorkflowClient.start`는 비동기로 Workflow를 시작하고 즉시 반환한다. 컨슈머는 빠르게 ack를 보낼 수 있어 Kafka 측 부담이 사라진다.

`setWorkflowId("order-" + event.orderId())`가 중복 방지의 핵심이다. 같은 `orderId`에 대해 두 번째 이벤트가 들어오면 Temporal이 `WorkflowExecutionAlreadyStartedException`을 던진다. `WorkflowIdReusePolicy`는 그 충돌 시 동작을 결정한다.

| Policy | 의미 |
|--------|------|
| `ALLOW_DUPLICATE` | 기본. 같은 ID로 새 Execution 허용 (이전 Execution 종료 후) |
| `ALLOW_DUPLICATE_FAILED_ONLY` | 이전이 실패로 종료된 경우에만 새로 허용 |
| `REJECT_DUPLICATE` | 같은 ID는 절대 재사용 안 함 |

비즈니스 이벤트가 중복 도착할 가능성이 있다면 `ALLOW_DUPLICATE_FAILED_ONLY`가 안전하다. 정상 종료된 주문에 다시 같은 이벤트가 와도 새 Workflow가 시작되지 않는다.

### 5-2. 패턴 B — Signal Workflow

이미 돌고 있는 Workflow에 외부 이벤트를 *주입*하려면 Signal을 쓴다.

```java
@KafkaListener(topics = "order-cancelled", groupId = "order-cancel-signaler")
public void onOrderCancelled(OrderCancelledEvent event) {
    String workflowId = "order-" + event.orderId();
    OrderWorkflow workflow = workflowClient.newWorkflowStub(
        OrderWorkflow.class, workflowId);
    try {
        workflow.cancel(event.reason());
    } catch (WorkflowNotFoundException e) {
        // 이미 끝났거나 시작 안 된 Workflow — 도메인 정책으로 처리
    }
}
```

`workflowClient.newWorkflowStub(클래스, workflowId)` 형태(옵션 없음)는 *기존 Workflow에 대한 핸들*을 만든다. 거기에 `@SignalMethod`로 선언된 메서드(`cancel`)를 호출하면 Signal이 Server로 보내진다.

Signal은 fire-and-forget이다. Workflow가 다음 결정 지점에서 그 Signal을 인식한다. `WorkflowNotFoundException`은 같은 ID의 Workflow가 없다는 의미로, 비즈니스 정책에 따라 무시·재시도·DLT 처리 중 선택한다.

### 5-3. 패턴 선택 기준

| 상황 | 패턴 | Workflow ID 의도 |
|------|------|------------------|
| `order-created` (새 주문) | Start | `order-{id}`로 중복 방지 |
| `order-cancelled` (취소 요청) | Signal | 같은 `order-{id}` 핸들에 cancel 전달 |
| `payment-confirmed` (외부 콜백) | Signal | 같은 `order-{id}`에 결제 결과 주입 |
| `order-refund-requested` (별도 흐름) | Start | `refund-{id}` 새 Workflow |

기준은 단순하다. *이벤트가 새 비즈니스 흐름의 시작인가, 진행 중인 흐름에 정보를 주는가*. 같은 토픽이라도 도착한 단계에 따라 Start와 Signal 둘 다 가능한 경우가 있으므로, 컨슈머가 Workflow의 존재 여부를 확인하고 분기하기도 한다.



## 6. Workflow ID 설계 — 멱등성의 기반

> 이 한 가지 결정이 전체 시스템의 멱등성 수준을 결정한다.

Workflow ID는 *비즈니스 식별자*여야 한다. 단순 UUID를 쓰면 같은 이벤트가 두 번 와도 두 Workflow가 생성되어 중복 처리된다.

### 6-1. ID 설계 규칙

| 규칙 | 이유 |
|------|------|
| 비즈니스 키 포함 | `order-{orderId}` 형태로 같은 이벤트 = 같은 ID |
| 흐름 종류 prefix | `order-{id}`, `refund-{id}`, `shipment-{id}`로 흐름 구분 |
| 환경/네임스페이스 미포함 | 환경 분리는 namespace로, ID는 비즈니스 의미만 |
| 길이 제한 1000자 | Temporal 제한. UUID 여러 개 이어붙이지 않기 |

### 6-2. 중복 이벤트 시나리오

Kafka가 at-least-once 전달을 보장하므로 같은 `OrderCreatedEvent`가 두 번 들어올 수 있다. 두 시나리오를 본다.

**시나리오 1**: 첫 이벤트로 Workflow가 정상 시작. 두 번째 이벤트가 도착하면 같은 ID로 Start 시도 → `WorkflowExecutionAlreadyStartedException` → 컨슈머는 예외를 잡고 ack. 두 번째 처리는 일어나지 않는다.

**시나리오 2**: 첫 이벤트로 Workflow가 실패로 종료. 두 번째 이벤트가 도착했을 때 정책이 `ALLOW_DUPLICATE_FAILED_ONLY`라면 새 Execution이 시작되어 재시도 효과를 얻는다.

이 구조는 [03-01.Inbox](../../05_ConsistencyPattern/03-01.Inbox.md)의 멱등 INSERT 패턴을 Workflow 차원에서 자동화한 것과 같다. Workflow ID가 자연스러운 멱등 키 역할을 한다.



## 7. Local Activity와 일반 Activity

> 둘의 차이는 *Server를 거치는가*다.

| 측면 | Activity | Local Activity |
|------|----------|---------------|
| 호출 경로 | Worker → Server → Worker | 같은 Worker 안에서 직접 호출 |
| Event History 기록 | 호출/응답 별도 이벤트 | 단일 이벤트(`MarkerRecorded`) |
| 재시도 위치 | Server가 분배 재시도 | Worker 내부에서 즉시 재시도 |
| 적합 자리 | 시간 들고 외부 의존 | 짧고 결정적 가까운 작업 |
| 부적합 자리 | 짧고 단순한 작업 | 외부 시스템 호출, 장기 작업 |

Local Activity는 Server 왕복이 없어 빠르지만, *Worker가 죽으면 그 호출 결과가 사라진다*. 짧고 idempotent하지 않은 작업이면 일반 Activity가 안전하다.

```java
private final ShortCalcActivity shortCalc = Workflow.newLocalActivityStub(
    ShortCalcActivity.class,
    LocalActivityOptions.newBuilder()
        .setStartToCloseTimeout(Duration.ofSeconds(5))
        .build()
);
```

대표 사용처: 단순 검증, 환율 계산, 캐시 조회처럼 *결과 손실이 큰 비용이 아니면서 외부 RTT가 짧은* 작업.



## 8. 실습 흐름 — 로컬 환경 구축에서 첫 Workflow 트리거까지

> 강의 §8 (실습 과정)을 표준 절차로 정리한다.

### 8-1. Temporal Server 로컬 실행

```yaml
# docker-compose.yml (Temporal 부분)
services:
  temporal:
    image: temporalio/auto-setup:1.24
    depends_on: [temporal-db]
    environment:
      - DB=postgresql
      - DB_PORT=5432
      - POSTGRES_USER=temporal
      - POSTGRES_PWD=temporal
      - POSTGRES_SEEDS=temporal-db
    ports:
      - "7233:7233"

  temporal-ui:
    image: temporalio/ui:2.27
    depends_on: [temporal]
    environment:
      - TEMPORAL_ADDRESS=temporal:7233
    ports:
      - "8088:8080"

  temporal-db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: temporal
      POSTGRES_PASSWORD: temporal
```

`docker compose up`이 끝나면 `localhost:7233`이 gRPC 엔드포인트, `localhost:8088`이 Web UI다.

### 8-2. 첫 실행 체크리스트

1. `application.yml`의 `temporal.target`이 `127.0.0.1:7233`을 가리키는지 확인.
2. Spring Boot 기동 후 콘솔에 Worker가 Task Queue를 polling 시작한다는 로그 확인.
3. Web UI(`localhost:8088`)에서 namespace `default` 클릭 → Workers 탭에서 본인 애플리케이션이 등록된 것 확인.
4. Kafka에 `order-created` 이벤트 발행 → Web UI의 Workflows 탭에서 새 Execution 확인.
5. Execution 클릭 → Event History 탭에서 단계 진행 상황 확인.

### 8-3. 디버깅 시작점

문제가 생기면 다음 순서로 본다.

| 증상 | 첫 확인 지점 |
|------|-------------|
| Workflow가 시작 안 됨 | Web UI의 Pending Workflows, 컨슈머 ack 로그 |
| Activity가 진행 안 됨 | Event History에서 마지막 이벤트, Activity Task의 Worker 할당 여부 |
| Workflow가 stuck | Event History의 `NonDeterministicError`, 결정성 위반 의심 |
| 같은 이벤트로 두 Workflow 시작 | Workflow ID 설계 점검, `WorkflowIdReusePolicy` 확인 |

상세 운영 함정과 통합 디버깅은 `[02-01.EDA + CDC + Temporal 통합 아키텍처](02-01.EDA%20%2B%20CDC%20%2B%20Temporal%20%ED%86%B5%ED%95%A9%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98.md)`의 §8에서 다룬다.



## 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. Temporal Spring 통합에 필요한 Bean 4종과 각 책임은?

`WorkflowServiceStubs`는 Temporal Server와의 gRPC 연결 풀로 가장 비싼 자원이며 애플리케이션당 1개. `WorkflowClient`는 Workflow 시작·Signal·Query를 발행하는 API로 namespace 단위. `WorkerFactory`는 Worker를 만드는 팩토리. `Worker`는 특정 Task Queue를 polling하며 Workflow/Activity 코드를 실행. Worker 등록 시 Workflow는 *클래스*로, Activity는 *빈 인스턴스*로 등록한다.

### Q2. Kafka 이벤트로 Workflow를 트리거하는 두 패턴의 선택 기준은?

새 비즈니스 흐름의 시작 이벤트라면 `WorkflowClient.start`로 새 Workflow를 시작한다. 진행 중인 흐름에 외부 정보를 주입하는 이벤트라면 `workflow.signalMethod()`로 Signal을 보낸다. 같은 토픽이라도 비즈니스 단계에 따라 두 패턴이 갈리므로, 일부 컨슈머는 Workflow 존재 여부를 확인하고 분기한다.

### Q3. Workflow ID를 비즈니스 키로 잡아야 하는 이유는?

Kafka가 at-least-once를 보장하므로 같은 이벤트가 두 번 도착할 수 있다. Workflow ID가 `order-{orderId}` 같은 비즈니스 키면 두 번째 시도가 `WorkflowExecutionAlreadyStartedException`으로 거부되어 중복 처리가 자동 차단된다. UUID 같은 임의 ID를 쓰면 같은 비즈니스 이벤트로 두 Workflow가 생긴다. 이 설계가 Inbox 멱등 패턴의 Workflow 버전이다.

### Q4. Activity가 Spring 빈이고 Workflow가 빈이 아닌 이유는?

Activity는 외부 부수 효과의 영역으로 DB 클라이언트나 외부 API 클라이언트를 주입받기에 빈으로 두는 것이 자연스럽다. Workflow는 *결정적 코드*여야 하므로 빈으로 두면 `@Autowired`로 비결정적 의존성이 들어가 결정성 위반이 발생할 위험이 크다. Temporal은 매 Workflow Execution마다 새 인스턴스를 만들어야 하므로 빈 라이프사이클과도 맞지 않는다. 그래서 Workflow는 *클래스*로 등록하고 Activity 호출은 `Workflow.newActivityStub`로 받는다.

### Q5. Local Activity와 일반 Activity의 차이는?

일반 Activity는 Worker → Server → Worker 경로로 호출이 분배되고 Event History에 호출과 응답이 별도 이벤트로 기록된다. Local Activity는 같은 Worker 안에서 직접 호출되고 단일 marker 이벤트만 기록된다. Local은 빠르지만 *Worker가 죽으면 호출 결과가 사라진다*. 짧고 idempotent하지 않거나 외부 호출이 있는 작업은 일반 Activity, 짧고 결정적 가까운 작업(검증·계산·캐시)은 Local Activity가 적합하다.
