# Inbox 트랜잭션 오염과 멱등 어댑터

---

> Inbox 패턴의 멱등 INSERT를 *INSERT 후 unique 위반 catch*로 구현하면 JPA 트랜잭션이 *rollback-only*로 마킹되는 부작용이 생긴다. 이 문서는 그 메커니즘을 분해하고, 라이브러리로서 안전한 어댑터를 어떻게 짜야 하는지를 정리한다.



## 학습 목표

> Inbox INSERT의 *예외 처리 방식*이 트랜잭션 전체 동작에 어떻게 영향을 주는지 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. JPA에서 unique violation 예외가 *왜 rollback-only로 마킹*되는지 설명할 수 있다.
2. select-후-insert vs INSERT-후-catch의 race condition 차이를 설명할 수 있다.
3. MariaDB `INSERT IGNORE`가 race condition과 트랜잭션 오염을 동시에 푸는 방식을 설명할 수 있다.
4. PROPAGATION_REQUIRES_NEW로 별도 트랜잭션을 띄우는 패턴의 비용을 설명할 수 있다.
5. 멱등 어댑터를 라이브러리로 묶을 때 *공통 인터페이스*가 가져야 할 책임을 말할 수 있다.


---

> Inbox 패턴의 멱등 INSERT 를 "select-후-insert" 가 아닌 "INSERT 후 unique 위반 catch" 로 구현하면 race condition 은 회피되지만, JPA 레벨에서는 **트랜잭션이 rollback-only 로 마킹** 되는 부작용이 따라붙는다. 이 문서는 그 메커니즘을 분해하고, 라이브러리로서 안전한 어댑터를 어떻게 짜야 하는지를 정리한다.

[03-01.Inbox.md](03-01.Inbox.md) 가 패턴 자체와 outbox 와의 대칭성을 다룬다면, 이 문서는 그 패턴을 **JPA 환경에서 구현할 때 부닥치는 트랜잭션 의미 충돌** 에 초점을 맞춘다.



## 1. 트랜잭션 오염이란 정확히 무엇인가

Spring + JPA 에서 트랜잭션이 살아 있는 동안 `PersistenceException` 류 예외가 던져지면, 우리가 그 예외를 catch 해서 메서드를 정상 return 하더라도 **트랜잭션 매니저는 해당 트랜잭션을 `rollback-only` 로 마킹** 한다. 이 마킹은 자바 객체 안의 단순한 플래그가 아니라 트랜잭션 매니저의 동기화 레지스트리에 박히는 영구 상태이며, 같은 트랜잭션 안에서 catch 후에 무엇을 해도 commit 시점에는 강제로 rollback 된다.

JPA 명세 (JSR-338 §3.7.1) 에 따르면 영속성 프로바이더가 다음 예외를 제외한 `PersistenceException` 을 던지면 활성 트랜잭션은 rollback 대상으로 마크된다.

- `NoResultException`
- `NonUniqueResultException`
- `LockTimeoutException`
- `QueryTimeoutException`

이유는 영속성 컨텍스트(persistence context, 1차 캐시)의 일관성 보증이 깨졌을 가능성이 있기 때문이다. 한 번 깨진 컨텍스트로 다른 entity 를 조회하거나 수정하면 무엇이 진실인지 알 수 없으니, 영속성 프로바이더는 "이 트랜잭션은 더 신뢰할 수 없다" 는 신호로 rollback-only 를 박는다.

Spring 의 `DataIntegrityViolationException` 은 Hibernate 의 `ConstraintViolationException` 을 변환한 것이고, 그 안에는 원본 `PersistenceException` 이 들어 있으므로 위 규칙이 그대로 적용된다.

증상은 두 가지 형태로 나타난다.

1. `UnexpectedRollbackException` — catch 후 비즈니스 로직을 더 진행하고 commit 을 시도하면, 트랜잭션 매니저가 "이미 rollback-only 인데 commit 요청이 왔다" 며 거절한다. 호출자 입장에서는 멀쩡히 catch 했는데 트랜잭션 경계에서 갑자기 예외가 튀어나오는 것처럼 보인다.
2. 조용한 일관성 깨짐 — 1차 캐시에 부분적으로 더러운 entity 가 남아 다음 조회/수정이 이상하게 동작한다. 이쪽은 예외도 안 나서 디버깅이 훨씬 어렵다.



## 2. Inbox 패턴이 이 함정에 정면으로 들어가는 이유

Inbox 멱등 INSERT 는 본질적으로 "한 번 박힌 키를 두 번 박지 못하게 한다" 는 약속이다. 이 약속을 동시성 안전하게 구현하는 가장 단순한 방법은 select-후-insert 가 아니라 **즉시 INSERT 시도 후 unique constraint 위반을 잡는 것** 이다. SELECT → 없으면 INSERT 는 두 컨슈머 인스턴스가 동시에 SELECT 시점에 모두 "없음" 을 보면 둘 다 INSERT 로 진입해 race 가 난다.

```java
// JpaInboxRepository.tryInsert(...) 의 초기 구현
try {
    entityManager.persist(InboxRecordEntity.fromDomain(record));
    entityManager.flush();
    return true;
} catch (DataIntegrityViolationException | PersistenceException e) {
    return false;
}
```

이 코드의 의도는 깔끔하다. DB 의 unique constraint 가 동시성 직렬화 도구 역할을 하고, 우리는 affected-row 를 보지 않아도 예외만 잡으면 "중복" 을 알 수 있다. 그러나 동시에 1번 절에서 본 부작용을 그대로 떠안는다. catch 가 끝난 시점에 이미 트랜잭션은 더럽혀져 있다.



## 3. 두 호출자, 두 가지 방어 전략

`message-lib` 의 `IdempotencyGuard` 를 호출하는 두 사례 (`ExampleMessageService`, `MultiRecordExampleService`) 가 이 함정을 정반대 방식으로 회피하고 있어 비교가 흥미롭다.

### 3-1. 사례 A — 단일 트랜잭션 + 즉시 return

```java
@Transactional
public void handle(...) {
    if (ceId != null && !idempotencyGuard.tryClaim(...)) {
        return;            // 중복이면 그냥 빠진다
    }
    log.info(...);          // 비즈니스 로직 (지금은 로그뿐)
}
```

핸들러가 단일 트랜잭션을 열고 `tryClaim` 이 false 를 반환하면 곧바로 return 한다. 카펫 위에 함정이 깔린 상태다. 트랜잭션은 분명 rollback-only 로 마킹되어 있지만, catch 후 추가 작업이 0 이라 commit 단계까지 가는 코드 경로가 거의 없다. 함정이 표면화되지 않을 뿐이다.

이 패턴의 위험은 **미래에 누가 `handle()` 안에 "중복이면 통계 카운터 +1" 같은 후속 작업을 넣는 순간** 드러난다. 그 한 줄이 추가되는 순간 commit 시점에 `UnexpectedRollbackException` 이 터지고, 추가한 사람은 자기 한 줄과 예외의 인과를 잇기 어렵다.

### 3-2. 사례 B — `REQUIRES_NEW` 로 별도 트랜잭션

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public boolean claimAlpha(...) {
    return idempotencyGuard.tryClaim(...);
}
```

호출자(`MultiRecordExampleConsumer.onAlpha`) 는 `@Transactional` 자체를 두지 않고, claim 만 자기 트랜잭션 T2 에서 즉시 commit 한다. 이 선택의 1차 동기는 트랜잭션 오염 회피가 아니라 다른 데 있다 — `@RetryableTopic` 시나리오에서 비즈니스 예외가 나도 inbox row 는 살려두고 retry 시 멱등 차단이 작동하게 하기 위함이다. 만약 claimAlpha 가 onAlpha 와 같은 트랜잭션이라면, 비즈니스 예외가 나는 순간 inbox row 도 함께 롤백되어 retry 가 매번 "첫 처리" 가 되어버린다.

부수적으로 트랜잭션 오염도 격리되는 효과가 따라온다. T2 가 더럽혀져도 외부 onAlpha 의 트랜잭션은 (어차피 없으니) 영향을 받지 않는다. 다만 T2 자체의 commit 단계에서 `UnexpectedRollbackException` 이 나는지는 별도 의심 거리로 남는다 — claimAlpha 본문이 `tryClaim` 한 줄뿐이라 catch 후 추가 작업이 없어 표면적으로 안전해 보일 뿐이다.

추가 부작용은 비즈니스 로직이 트랜잭션 없이 실행된다는 점이다. 핸들러가 진짜 DB 쓰기 작업을 가지게 되면 각 작업이 auto-commit 되어 "성공하든 실패하든 inbox 는 박혀있다" 는 비대칭이 만들어진다. 비즈니스 실패가 retry 로 되돌릴 기회를 잃는다는 의미다.



## 4. 시나리오별 위험도 정리

| 시나리오 | 현재 동작 | 위험도 | 비고 |
|----------|---------|--------|------|
| 사례 A, ce_id 없음 | 멱등 우회, 정상 처리 | ⚪ 낮음 | 의도된 우회 |
| 사례 A, 첫 메시지 | INSERT 성공, 비즈니스 정상 | ⚪ 낮음 | |
| 사례 A, 중복 메시지 | INSERT 실패 → false → return | 🟡 중간 | 카펫 위. handle() 에 비즈니스 추가 시 폭발 |
| 사례 A, 중복 + 비즈니스 추가 | UnexpectedRollbackException 가능 | 🔴 높음 | 잠재적, 표면화 시 디버깅 난해 |
| 사례 B, 첫 메시지 + 성공 | T2 commit, 비즈니스 정상 | ⚪ 낮음 | |
| 사례 B, 첫 메시지 + 비즈니스 실패 | T2 commit (row 박힘) → 예외 → retry → 멱등 차단 | ⚪ 낮음 | 의도된 동작 |
| 사례 B, 중복 메시지 | T2: INSERT 실패 → false 반환 | 🟡 중간 | T2 commit 시 UnexpectedRollback 의심 |
| 사례 B, 중복 + 비즈니스 실패 | T2 false → 비즈니스로 못 감 → return | ⚪ 낮음 | OK |



## 5. 세 가지 해결 옵션과 트레이드오프

### 5-1. 옵션 1 — `INSERT IGNORE` 로 예외 자체를 없앤다 (권장)

JPA `persist` 대신 vendor native query 로 갈아탄다. MariaDB/MySQL 의 `INSERT IGNORE` 는 unique 위반 시 affected-row 를 0 으로 반환할 뿐 예외를 던지지 않는다. PostgreSQL 이라면 `INSERT ... ON CONFLICT (cols) DO NOTHING` 이 같은 의미를 한다.

```java
private static final String INSERT_IGNORE_SQL = """
        INSERT IGNORE INTO TB_TRB_OX_002
            (MSG_ID, CONSUMER_GROUP, TOPIC, PARTITION_NO, OFFSET_NO, REG_DT, RGTR_ID)
        VALUES (?, ?, ?, ?, ?, NOW(6), 'SYSTEM')
        """;

@Override
public boolean tryInsert(InboxRecord record) {
    int rows = entityManager.createNativeQuery(INSERT_IGNORE_SQL)
            .setParameter(1, record.getMsgId())
            .setParameter(2, record.getConsumerGroup())
            .setParameter(3, record.getTopic())
            .setParameter(4, record.getPartitionNo())
            .setParameter(5, record.getOffsetNo())
            .executeUpdate();
    return rows == 1;
}
```

장점은 명확하다. 예외 자체가 발생하지 않으므로 트랜잭션 오염 0, 1차 캐시 오염 0. 호출자는 트랜잭션 의미를 신경 쓰지 않고 같은 트랜잭션에서 비즈니스 작업을 자유롭게 이어갈 수 있다. JPA Entity 매핑은 그대로 둔 채 어댑터(Repository 구현)만 native 로 바꾸므로 도메인 영향도 0 이다.

단점은 vendor 종속이다. 운영 DB 가 단일이라면 의미 있는 단점이 아니지만, 라이브러리가 다중 DB 를 지원해야 한다면 dialect 별 분기가 필요해진다. 또한 `INSERT IGNORE` 는 PK/Unique 위반 외에 NOT NULL 위반·FK 위반까지 silent 하게 무시할 수 있어 위험할 수 있다 — 본 테이블처럼 FK 가 없고 NOT NULL 컬럼을 SQL 에서 모두 채우는 경우만 안전하게 채택할 수 있다는 전제가 따라붙는다.

### 5-2. 옵션 2 — SAVEPOINT 로 inner 트랜잭션 격리

JDBC 표준 SAVEPOINT 를 활용해 INSERT 시도를 하나의 inner 트랜잭션처럼 다룬다. unique 위반이 나면 savepoint 까지만 롤백하고 외부 트랜잭션은 살린다.

장점은 vendor 무관성. 단점은 코드 복잡도와 환경 의존성이다. `JpaTransactionManager` 가 `setNestedTransactionAllowed(true)` 로 설정되어야 하고, EntityManager 의 1차 캐시가 savepoint 롤백 후 어떤 상태인지 별도로 관리해야 한다. 실무에서는 옵션 1 만큼 깔끔하게 떨어지지 않는다.

### 5-3. 옵션 3 — 예외를 좁히고 호출자에 책임 전가

catch 를 PK 위반만으로 좁힌다 (예: SQL state 가 `23000` 또는 `23505` 일 때만 false 반환). 그리고 호출자가 `@Transactional(noRollbackFor = DataIntegrityViolationException.class)` 를 명시하도록 강제한다.

라이브러리로서 매우 나쁜 선택이다. 호출자 모두에게 옵션을 강제하면 사용성이 크게 떨어지고, 누군가 옵션을 빠뜨리는 순간 사일런트로 함정에 떨어진다. 라이브러리는 "올바른 사용이 기본 경로" 가 되도록 설계되어야 한다.



## 6. 검증 — 위험을 수치화하는 IT 테스트

옵션 1 로 전환한 뒤, 트랜잭션 오염이 정말 사라졌는지 확인하려면 다음 형태의 IT 가 필요하다. 핵심은 "중복 claim 후에도 같은 트랜잭션에서 다른 INSERT 가 commit 되는가" 를 직접 본다는 점이다.

```java
@Test
@DisplayName("R4: 중복 claim 이후에도 같은 트랜잭션에서 후속 작업이 정상 commit")
void tryClaim_duplicateWithinTransaction_doesNotPoisonTransaction() {
    String ceId = "OX_TEST_R4";
    String group = "alpha-group";
    String topic = "tps.v305p.executor.evt.example-multi";

    transactionTemplate.executeWithoutResult(s ->
            guard.tryClaim(ceId, group, topic, 0, 100L));

    transactionTemplate.executeWithoutResult(s -> {
        boolean second = guard.tryClaim(ceId, group, topic, 0, 100L);
        assertThat(second).isFalse();

        entityManager.createNativeQuery("""
                        INSERT INTO TB_TRB_OX_002
                            (MSG_ID, CONSUMER_GROUP, TOPIC, PARTITION_NO, OFFSET_NO, REG_DT, RGTR_ID)
                        VALUES (?, ?, ?, ?, ?, NOW(6), 'SYSTEM')
                        """)
                .setParameter(1, "OX_TEST_R4_OTHER")
                .setParameter(2, group)
                .setParameter(3, topic)
                .setParameter(4, 0)
                .setParameter(5, 999L)
                .executeUpdate();
    });

    Number count = (Number) entityManager.createNativeQuery(
            "SELECT COUNT(*) FROM TB_TRB_OX_002 WHERE MSG_ID = 'OX_TEST_R4_OTHER'"
    ).getSingleResult();
    assertThat(count.longValue()).isEqualTo(1L);
}
```

이 테스트를 옵션 1 적용 전 코드 (persist + flush + catch) 로 돌리면 후속 INSERT 가 commit 단계에서 실패해 카운트가 0 이 나오거나, 트랜잭션 경계에서 `UnexpectedRollbackException` 이 발생한다. 옵션 1 적용 후에는 카운트가 1 로 나온다. 즉 이 한 테스트가 트랜잭션 오염 위험의 유무를 객관적으로 증명한다.

기존 R1/R2/R3 (멱등 차단, fan-out, retry-topic) 시나리오는 그대로 통과해야 한다 — 어댑터만 갈아탔을 뿐 시맨틱은 동일하다.



## 7. 한 줄 결론

> Inbox 의 멱등 INSERT 는 "예외 catch" 로 구현하면 라이브러리의 사용 패턴이 한 번 깨질 때마다 디버깅 어려운 트랜잭션 오염 장애를 만든다. **`INSERT IGNORE` 기반 native insert 로 전환해 예외 자체를 발생시키지 않는 것** 이 라이브러리로서 가장 안전한 선택이다. 호출자가 트랜잭션 의미를 신경 쓰지 않아도 되는 어댑터가 좋은 어댑터다.



## 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. JPA에서 unique violation 예외가 트랜잭션을 rollback-only로 만드는 이유는?

Hibernate가 `PersistenceException`을 던지면서 *session을 일관성 없는 상태*로 본다. JTA·Spring 트랜잭션 매니저가 이 예외를 받으면 현재 트랜잭션을 *rollback-only*로 마크해서 commit 시도 시에도 강제 롤백한다. 그래서 catch만 한다고 끝나지 않고, 트랜잭션 안의 *이후 모든 작업이 함께 롤백*된다. Inbox INSERT 후 비즈니스 로직을 수행하려면 이 마킹을 회피해야 한다.

### Q2. select-후-insert로 race condition을 막을 수 없는 이유는?

같은 message_id를 두 컨슈머가 *같은 순간*에 select하면 둘 다 "없음"을 보고 INSERT를 시도한다. PK 위반이 한쪽에서만 일어나는 게 아니라 *둘 다 commit 단계*에서 충돌하게 된다. select와 insert 사이에 *gap*이 있는 한 race가 본질적으로 막히지 않는다. INSERT-후-catch가 자연스러운 이유는 *데이터베이스의 PK 제약*이 단일 원자 단계로 갈등을 해소하기 때문이다.

### Q3. MariaDB `INSERT IGNORE`가 두 문제를 동시에 푸는 방식은?

INSERT가 PK 위반에 부딪히면 *예외를 던지지 않고 0 rows affected*로 끝난다. JDBC 측에서 예외가 발생하지 않으므로 JPA가 rollback-only로 마킹하지 않는다. race condition도 데이터베이스 측 원자 INSERT 한 단계로 처리되어 따로 락이 필요 없다. 단점은 *어떤 컬럼이 충돌했는지*가 명시적이지 않다는 것과, 다른 종류의 INSERT 실패(외래 키 위반, NOT NULL 위반)도 함께 무시될 수 있다는 것이다.

### Q4. PROPAGATION_REQUIRES_NEW로 별도 트랜잭션을 띄우는 패턴의 비용은?

비즈니스 트랜잭션과 *완전히 분리된* 새 트랜잭션이라 unique violation이 비즈니스 측을 오염시키지 않는다. 하지만 비용이 크다. 첫째, *별도 DB connection*이 필요하므로 connection pool 압박이 두 배가 된다. 둘째, *Inbox INSERT와 비즈니스 로직이 다른 트랜잭션*이라 부분 실패가 다시 생길 수 있다(INSERT 성공·비즈니스 실패 시 영영 미처리). 그래서 별도 트랜잭션은 Inbox INSERT 자체가 *멱등 마커 역할만 할 때*에만 안전하다.

### Q5. 멱등 어댑터 공통 인터페이스가 가져야 할 책임은?

세 가지다. (1) *멱등 키 추출* — 메시지에서 `(message_id, consumer_group)` 같은 키를 만드는 책임. (2) *INSERT 시도* — DB 종속적 방식(`INSERT IGNORE`, `ON CONFLICT DO NOTHING`)을 캡슐화. (3) *결과 반환* — `boolean isFirstSeen()` 같은 명확한 시그널을 호출자 핸들러에 전달. 핸들러는 isFirstSeen=true에서만 비즈니스 로직을 수행한다. 이 인터페이스가 라이브러리화되면 도메인 코드는 DB 방언과 무관해진다.
