# Batch Listener와 부분 실패 처리

---

> Spring Kafka의 batch listener는 한 poll에서 받은 record 묶음을 *통째로* 핸들러에 전달한다. 단건 listener보다 처리량이 높지만, *100건 중 1건이 실패하면 어떻게 할 것인가*라는 새로운 문제를 만든다. 본 글은 batch listener의 동작, 부분 실패 처리 패턴 세 가지, TPS가 batch listener를 채택하지 않은 이유를 정리한다.



## 1. 단건 vs Batch — 두 모드의 본질

> 같은 `@KafkaListener` 어노테이션이지만 *메서드 시그니처 + 컨테이너 factory 설정*에 따라 모드가 갈린다.

```java
// 단건 모드
@KafkaListener(topics = "order")
public void onSingle(OrderEvent event) { ... }

// Batch 모드
@KafkaListener(topics = "order", containerFactory = "batchFactory")
public void onBatch(List<OrderEvent> events) { ... }
```

Batch 모드는 컨테이너 factory에 `factory.setBatchListener(true)`가 필요하다. 컨슈머의 `max.poll.records`(기본 500)만큼 한 번에 들어온다.

| 측면 | 단건 listener | Batch listener |
|------|---------------|----------------|
| 핸들러 인자 | `EventAvro` 또는 `ConsumerRecord<K,V>` | `List<EventAvro>` 또는 `List<ConsumerRecord<K,V>>` |
| 처리량 | 낮음 | 높음 |
| 에러 처리 단위 | record 한 건 | 묶음 통째 (부분 실패 시 까다로움) |
| 오프셋 커밋 | record당 또는 batch당 | batch당 |
| 메모리 점유 | 작음 | `max.poll.records` × 평균 record 크기 |

처리량 차이는 무시할 수 없다. 단건 listener는 record마다 메서드 호출·트랜잭션 시작/커밋·로그·헤더 추출 등의 오버헤드가 누적되지만, batch는 그 비용이 묶음당 한 번이다. 다만 *그 처리량을 얻는 대신 부분 실패 처리가 복잡해진다*.



## 2. Batch 모드의 부분 실패 문제

> 100개 record를 batch로 받았는데 50번째가 실패하면 어떻게 할 것인가. 단순한 답은 없다.

### 2-1. 통째 retry — 가장 단순하지만 위험

핸들러가 throw하면 Spring Kafka의 `DefaultErrorHandler`(batch 모드)가 *전체 batch를 재시도*한다. 결과:

- 1~49번 record는 *이미 처리 성공했는데* 재처리됨
- 멱등 가드(`04-08` Inbox 테이블)가 없으면 중복 효과
- 49번까지가 외부 API 호출이라면 49번 모두 다시 호출

이 모드의 안전 전제는 *모든 record가 멱등하게 처리됨*이다.

### 2-2. 실패 record만 분리 — `BatchListenerFailedException`

Spring Kafka 2.5+는 `BatchListenerFailedException`을 통해 *어느 인덱스에서 실패했는지*를 컨테이너에 알릴 수 있다.

```java
@KafkaListener(topics = "order", containerFactory = "batchFactory")
public void onBatch(List<ConsumerRecord<String, OrderEvent>> records) {
    for (int i = 0; i < records.size(); i++) {
        try {
            process(records.get(i).value());
        } catch (Exception e) {
            throw new BatchListenerFailedException("failed at " + i, e, i);
        }
    }
}
```

컨테이너는 *해당 인덱스 이전 record까지의 오프셋을 커밋*하고, 실패한 record부터 retry(또는 DLQ 격리) 흐름을 탄다. 결과적으로 1~49는 정상 처리, 50번이 격리, 51 이후는 다음 poll로 진행.

문제는 *for 루프 안에서 한 건씩 처리하는 패턴은 batch의 처리량 이점을 거의 잃는다*는 점이다. JDBC bulk insert, gRPC streaming 같은 *진짜 batch 처리*를 하지 않으면 batch 모드의 이득이 사라진다.

### 2-3. 내부 batch 수집 + 실패만 별도 적재

처리량과 안전성을 모두 잡고 싶을 때 쓰는 패턴이다.

```java
@KafkaListener(topics = "order", containerFactory = "batchFactory")
public void onBatch(List<OrderEvent> events) {
    BulkResult result = repository.bulkInsert(events);  // 일부 실패 가능

    for (FailedRow row : result.failures()) {
        failureRepository.save(row);  // 별도 테이블에 격리
    }
    // throw 없음 → 전체 batch 커밋
}
```

bulk insert가 부분 실패해도 *실패 row만 별도 테이블*에 적재하고 batch 자체는 커밋한다. 운영자가 실패 테이블을 보고 수동 재처리한다. *Kafka DLQ 대신 DB 격리*를 택하는 변형이며, 도메인이 트랜잭션 강제 일관성을 요구하지 않을 때 유효하다.



## 3. Batch 모드의 ack 정책

> Batch 모드는 *묶음 전체*가 처리 완료된 시점에 오프셋이 커밋된다. ack-mode `MANUAL`을 함께 쓰면 묶음 내 부분 ack는 불가능하다.

`Acknowledgment` 객체는 batch 단위로 주입되며, `acknowledge()` 호출은 *batch 전체*의 마지막 오프셋을 커밋한다. *batch 안의 30번째만 ack*는 지원되지 않는다.

```java
@KafkaListener(topics = "order", containerFactory = "batchManualFactory")
public void onBatch(List<OrderEvent> events, Acknowledgment ack) {
    process(events);
    ack.acknowledge();  // batch 전체의 마지막 offset commit
}
```

부분 ack가 필요하면 `BatchListenerFailedException` 방식(§2-2)으로 가야 한다.



## 4. Batch 모드를 채택할지 결정하는 기준

> 처리량 향상이 *얼마나 클 것인가*와 *부분 실패가 얼마나 자주 발생할 것인가*를 곱한 값이 도입 비용을 넘어야 의미 있다.

도입 가치가 큰 경우:

- *진짜 batch 처리*가 가능 — bulk INSERT, bulk API 호출, 메모리 누적 후 일괄 처리
- 메시지 *전부 또는 전무*가 자연스러움 — 분석 토픽, 메트릭 적재
- 멱등성이 이미 보장됨 — 통째 retry가 안전

도입 가치가 작은 경우:

- 한 건씩 외부 API를 호출 — for 루프로 풀면 처리량 이득이 사라짐
- 부분 실패 처리 로직이 도메인에 맞물려 복잡 — `BatchListenerFailedException` 인덱스 관리가 핸들러 본문을 어지럽힘
- 메시지 순서가 비즈니스적으로 중요 — 부분 실패 처리 중 순서가 흐트러질 위험



## 5. 함정 — 동일 토픽에 batch와 단건 컨슈머 공존

같은 토픽을 한쪽은 batch로, 다른 쪽은 단건으로 듣는 *동일 컨슈머 그룹* 구성은 작동하지 않는다. partition assignment가 한 컨슈머에 가면 그 인스턴스의 listener가 *자기 모드*로만 처리한다. 모드가 섞이면 메서드 시그니처 불일치로 NPE 또는 분실.

다른 컨슈머 그룹이라면 각자 독립이라 안전하지만, *같은 비즈니스 의미의 핸들러를 두 모드로 운영*하는 것 자체가 보통 잘못된 설계 신호다.



## 6. TPS가 batch listener를 채택하지 않은 이유

TPS의 거의 모든 핸들러는 *한 record에 대해 하나의 useCase*를 호출하는 패턴이다. 한 record가 한 job 또는 한 명령이고, 처리 시간은 외부 시스템(Jenkins, ArgoCD, GitLab) 응답 시간이 지배한다. *외부 호출 시간이 record 처리 시간의 99%*인 도메인에서 batch는 처리량을 거의 늘리지 못한다.

또한 부분 실패 처리는 `KafkaErrorConfig`의 단건 retry·DLQ 흐름으로 충분하다 (`03-03`). batch를 도입해도 부분 실패를 풀기 위해 결국 단건처럼 처리하게 된다.

향후 도입 후보 영역:

- 대용량 분석 토픽 (메트릭 집계, 로그 적재)
- 알림 batch 전송 (한 번에 1000건)
- 외부 API의 bulk 엔드포인트가 있을 때



## 7. 정리

Batch listener는 *진짜 batch 처리가 가능한 도메인*에서만 의미 있다. 한 건씩 외부 API를 호출하는 패턴이라면 batch 모드는 메서드 시그니처만 다른 단건 처리이며, 부분 실패 복잡도만 늘어난다. TPS는 *외부 호출 지배적 도메인*이라 단건 listener + `DefaultErrorHandler` 정책을 그대로 유지한다.

부분 실패가 자주 발생하는 batch 모드라면 `BatchListenerFailedException`(§2-2) 패턴이 표준이며, 도메인이 트랜잭션을 강제하지 않으면 *실패만 별도 적재*(§2-3)가 운영 부담이 낮다.
