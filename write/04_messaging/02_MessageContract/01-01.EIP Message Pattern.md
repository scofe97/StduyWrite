# EIP Message Pattern

---

> EIP에 수록된 메시지 설계 패턴 중 Kafka/Redpanda 환경에서 의도와 형식을 어떻게 인프라 수준으로 강제하는지를 정리한다. 메시지에 담는 *행동·데이터·사실*의 구분, 형식 변경에 견디는 Format Indicator, 만료·시퀀스·표준화까지 다룬다.



## 학습 목표

> EIP의 메시지 패턴을 Kafka 토픽·헤더·Schema Registry 위에 어떻게 *얹는지* 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. Command·Document·Event Message의 차이를 "실패 책임자"로 설명할 수 있다.
2. CloudEvents type 컨벤션이 Schema Registry 토픽 네이밍과 어떻게 정렬되는지 설명할 수 있다.
3. Kafka 헤더 기반 Format Indicator와 Schema Registry 기반 Format Indicator를 비교할 수 있다.
4. Kafka가 토픽 단위 보존만 제공할 때 메시지별 만료를 어떻게 시뮬레이션하는지 설명할 수 있다.
5. Event Standardizer를 진입점에 둘지 소비자별로 둘지 결정하는 기준을 설명할 수 있다.



## 1. Command vs Document vs Event

> 메시지는 *행동 요청·데이터 전달·사실 통보* 세 가지 의도 중 하나다. 의도가 명시되지 않으면 수신자가 그때마다 해석해야 하고, 같은 토픽에 의도가 섞이면 토픽 운영이 망가진다.

### 1.1 Command Message

"X를 해줘"라는 행동 요청이다. 발신자는 수신자가 특정 작업을 수행하기를 기대하며, 수신자는 정확히 하나여야 한다.

- 두 명이 동시에 같은 명령을 처리하면 중복 실행이 발생한다.
- Command Message는 단일 Consumer Group(Point to Point)과 결합해야 의미가 완성된다.

### 1.2 Document Message

"이 데이터를 전달할게"다. 수신자가 그 데이터로 무엇을 할지는 수신자의 결정이고, 발신자는 수신자의 반응에 관심이 없다.

- 주기적 보고서, 설정 파일 배포, 마스터 데이터 동기화 등이 해당한다.
- 여러 수신자가 동시에 받아도 상관없다.

### 1.3 Event Message

"이런 일이 일어났어"다. 과거 사실의 통보이며 발신자는 이 사실에 누가 관심을 갖는지 몰라도 된다.

- OrderCreated, PaymentProcessed, InventoryReserved가 전형적인 예다.
- Pub/Sub 채널과 자연스럽게 결합되며, 여러 컨슈머 그룹이 각자 독립적으로 처리할 수 있다.

### 1.4 CQRS와의 관계

CQRS 아키텍처와의 관계도 이 분류에서 명확해진다. Command Message는 Write Model로 들어가고, 처리 결과로 발생한 Event Message는 Read Model을 갱신하는 데 사용된다. Document Message는 두 모델 중 어느 쪽과도 독립적으로 존재할 수 있다.

세 가지 유형을 구분하는 실용적인 판단 기준은 "메시지가 실패했을 때 누가 책임지는가?"를 물어보는 것이다.

- Command Message가 실패하면 발신자가 재시도를 책임진다.
- Event Message가 실패하면 컨슈머가 재처리를 책임진다.
- Document Message가 실패하면 다음 발행 주기를 기다리거나 별도의 보상 메커니즘이 필요하다.

### 1.5 토픽 네이밍과 스키마

토픽 네이밍 컨벤션은 의도를 코드 외부에서도 읽을 수 있게 만든다.

```bash
# Command Message 토픽 — 행동 요청
commands.order.create
commands.payment.process
commands.inventory.reserve

# Document Message 토픽 — 데이터 전달
documents.product.catalog
documents.config.feature-flags

# Event Message 토픽 — 사실 통보
events.order.created
events.payment.processed
events.inventory.reserved
```

Avro 스키마 수준에서도 의도를 명시할 수 있다. CloudEvents Spec의 type 필드 컨벤션을 따르면 스키마만 보고도 메시지 유형을 판단할 수 있다.

```json
// Command Message 스키마 예시
{
  "type": "record",
  "name": "CreateOrderCommand",
  "namespace": "com.example.commands.order",
  "fields": [
    {"name": "commandId",   "type": "string", "doc": "멱등성을 위한 커맨드 식별자"},
    {"name": "issuedAt",    "type": "long",   "doc": "epoch millis"},
    {"name": "issuedBy",    "type": "string", "doc": "커맨드 발행 주체"},
    {"name": "customerId",  "type": "string"},
    {"name": "items",       "type": {"type": "array", "items": "OrderItem"}}
  ]
}

// Event Message 스키마 예시
{
  "type": "record",
  "name": "OrderCreatedEvent",
  "namespace": "com.example.events.order",
  "fields": [
    {"name": "eventId",     "type": "string", "doc": "이벤트 식별자 (재처리 감지용)"},
    {"name": "occurredAt",  "type": "long",   "doc": "사건 발생 시각 (epoch millis)"},
    {"name": "orderId",     "type": "string"},
    {"name": "customerId",  "type": "string"},
    {"name": "totalAmount", "type": "long"}
  ]
}
```



## 2. Format Indicator

> 시스템이 진화하면 메시지 형식도 변한다. 변경이 일어날 때 발신자/수신자가 항상 동시에 배포되지 않으므로, 메시지 자체에 *"나는 어떤 형식이야"* 표시를 박아 둔다.

Kafka에서 Format Indicator는 두 계층에서 동작한다.

### 2.1 Kafka Record Header

헤더는 페이로드와 별도로 키-값 쌍을 저장할 수 있는 공간이다. Content-type 헤더를 사용하면 역직렬화 전에 형식을 판단할 수 있다.

```java
// Producer: 형식 정보를 헤더에 포함
ProducerRecord<String, byte[]> record = new ProducerRecord<>(
    "events.order.created",
    orderId,
    avroBytes
);
record.headers()
    .add("content-type", "application/avro".getBytes())
    .add("schema-version", "2".getBytes())
    .add("ce-specversion", "1.0".getBytes());  // CloudEvents 호환 헤더

// Consumer: 헤더로 형식 판별 후 분기 처리
ConsumerRecord<String, byte[]> record = ...;
Header contentType = record.headers().lastHeader("content-type");
String format = new String(contentType.value());

if ("application/avro".equals(format)) {
    processAvro(record.value());
} else if ("application/json".equals(format)) {
    processJson(record.value());
}
```

### 2.2 Schema Registry

Avro/Protobuf를 Schema Registry와 함께 사용할 때, 모든 직렬화된 바이트의 앞 5바이트는 스키마 조회를 위한 구조를 지닌다. Schema Registry의 subject 네이밍 전략은 그 자체로 암묵적인 Format Indicator다.

```bash
# TopicNameStrategy (기본): 토픽 이름 = 형식 계약
events.order.created-value → OrderCreatedEvent v1~vN

# TopicRecordNameStrategy: 토픽 내 다중 형식 허용
events.order.all-value:com.example.events.OrderCreatedEvent
events.order.all-value:com.example.events.OrderCancelledEvent
```

CloudEvents 표준은 Format Indicator를 `datacontenttype` 필드로 표준화한다. 이를 따르면 Kafka 헤더, HTTP 헤더, gRPC 메타데이터를 가로질러 같은 키로 형식을 표시할 수 있다.



## 3. Message Expiration

> 일부 메시지는 시간이 지나면 의미를 잃는다. "지금 당장 재고를 확인해"라는 요청은 5분 뒤에는 무의미해진다. JMS는 메시지별 TTL을 지원했지만, Kafka는 토픽 단위의 보존 정책만 지원하며 메시지별 만료를 네이티브로 제공하지 않는다.

메시지별 만료가 필요하다면 애플리케이션 레이어에서 시뮬레이션한다. 구현은 두 단계로 이뤄진다. 만료 시각을 헤더 또는 페이로드에 포함시키고, 컨슈머가 처리 전에 만료 여부를 확인한다.

```java
// Producer: 만료 시각을 헤더에 포함 (5분 유효)
long expiresAt = System.currentTimeMillis() + Duration.ofMinutes(5).toMillis();
ProducerRecord<String, InventoryCheckCommand> record = new ProducerRecord<>(
    "commands.inventory.check",
    inventoryCheckCommand
);
record.headers().add(
    "expires-at",
    String.valueOf(expiresAt).getBytes()
);
producer.send(record);

// Consumer: 처리 전 만료 확인
ConsumerRecord<String, InventoryCheckCommand> record = ...;
Header expiresAtHeader = record.headers().lastHeader("expires-at");

if (expiresAtHeader != null) {
    long expiresAt = Long.parseLong(new String(expiresAtHeader.value()));
    if (System.currentTimeMillis() > expiresAt) {
        log.warn("만료된 메시지 건너뜀: key={}, expiresAt={}", record.key(), expiresAt);
        expiredMessageCounter.increment();
        return; // 처리 건너뜀
    }
}

// 만료되지 않은 경우에만 처리
processInventoryCheck(record.value());
```



## 4. Message Sequence

> 하나의 논리적 메시지가 너무 커서 단일 Kafka Record에 담기 어려울 때, 또는 분산 처리를 위해 여러 파티션에 나눠야 할 때 Message Sequence 패턴이 필요하다. EIP에서 Message Sequence는 "N개의 연속된 메시지가 모여 하나의 완전한 메시지를 구성한다"는 개념이다.

Message Sequence는 순서 보장이 핵심이다. 여러 메시지가 특정 순서로 처리되어야 할 때 순서를 명시적으로 지정한다.

```json
// Message Sequence: 순서 정보 포함 스키마
{
  "type": "record",
  "name": "OrderLineItemEvent",
  "fields": [
    {"name": "correlationId",   "type": "string", "doc": "전체 시퀀스 식별자"},
    {"name": "sequenceNumber",  "type": "int",    "doc": "현재 순서 (1-based)"},
    {"name": "totalInSequence", "type": "int",    "doc": "전체 메시지 수"},
    {"name": "isLast",          "type": "boolean","doc": "마지막 메시지 여부"},
    {"name": "payload",         "type": "OrderLineItem"}
  ]
}
```

```java
// 파티션 키를 correlationId로 설정하여 순서 보장
for (int i = 0; i < lineItems.size(); i++) {
    OrderLineItemEvent event = OrderLineItemEvent.newBuilder()
        .setCorrelationId(orderId)
        .setSequenceNumber(i + 1)
        .setTotalInSequence(lineItems.size())
        .setIsLast(i == lineItems.size() - 1)
        .setPayload(lineItems.get(i))
        .build();

    producer.send(new ProducerRecord<>(
        "events.order.line-items",
        orderId,           // 파티션 키 = correlationId
        event
    ));
}
```

컨슈머 측에서는 Kafka Streams State Store로 시퀀스를 조립한다. correlationId를 키로 부분 메시지를 상태 저장소에 누적하다가, `isLast == true`인 메시지가 도착하면 완전한 메시지를 조립해서 사용한다.



## 5. Event Standardizer

> 실무에서 이벤트를 발행하는 시스템은 단일하지 않다. 사내 레거시 시스템, 외부 API, 파트너사 연동 등 각각이 자신만의 이벤트 형식을 가진다. Event Standardizer는 이러한 다양한 형식을 *Canonical Data Model(정규 데이터 모델)*로 변환하는 패턴이다.

예를 들어 주문 이벤트는 자사 시스템이 `orderId`, 레거시 시스템이 `order_no`, 외부 API가 `transactionRef`를 쓰는 식으로 모두 다를 수 있다.

```bash
[Canonical Data Model 없이: N×M 변환기 필요]

레거시시스템 ---> 배송서비스 (변환기 1)
레거시시스템 ---> 통계서비스 (변환기 2)
레거시시스템 ---> 알림서비스 (변환기 3)
외부결제API  ---> 배송서비스 (변환기 4)
외부결제API  ---> 통계서비스 (변환기 5)
외부결제API  ---> 알림서비스 (변환기 6)
파트너사API  ---> 배송서비스 (변환기 7)
파트너사API  ---> 통계서비스 (변환기 8)
파트너사API  ---> 알림서비스 (변환기 9)
                              ↑ 3소스 × 3소비자 = 9개 변환기

[Canonical Data Model 사용: N+M 변환기만 필요]

레거시시스템 ---> [Standardizer] --+
외부결제API  ---> [Standardizer] --+--> [canonical.orders] ---> 배송서비스
파트너사API  ---> [Standardizer] --+                       ---> 통계서비스
                                                         ---> 알림서비스
                              ↑ 3개 표준화기 + 0개 소비자측 변환 = 3개
```

### 5.1 구현 전략

진입점에서 표준화하느냐 소비자별로 변환하느냐가 핵심 분기점이다.

1. **진입점(Gateway)에서 표준화**: 이벤트가 시스템에 진입하는 순간 정규 형식으로 변환한다. Kafka의 경우 소스 토픽을 별도로 두고, Standardizer가 소비하여 정규화 토픽으로 발행한다.
2. **소비자별 변환**: 소비자가 각자 원하는 형식으로 변환한다. 단기적으로 쉽지만 같은 변환 로직이 여러 소비자에 중복 구현되어 유지보수 비용이 증가한다.

실무에서는 전략 1이 권장된다. 변환 로직이 한 곳에 집중되므로 소스 시스템의 형식이 바뀌어도 Standardizer 하나만 수정하면 된다.

한 가지 주의점은 Canonical Data Model이 지나치게 범용적으로 설계되어, 모든 시스템의 특수한 요구사항을 수용하려다 거대하고 복잡한 공통 스키마로 만들어지는 경우다. 도메인이 정말 공유하는 핵심 필드만 정규 스키마에 두고, 소스 고유 필드는 `extensions` 같은 우회 필드로 우회한다.

```java
// Kafka Streams: 다양한 소스 이벤트를 정규 형식으로 변환
KStream<String, LegacyOrderEvent> legacyStream = builder.stream("legacy.orders");

KStream<String, CanonicalOrderEvent> canonicalStream = legacyStream.mapValues(legacy -> {
    return CanonicalOrderEvent.newBuilder()
        .setOrderId(legacy.getOrderNo())          // order_no → orderId
        .setCustomerId(legacy.getCustId())         // cust_id → customerId
        .setTotalAmount(legacy.getAmt())           // amt → totalAmount
        .setCreatedAt(parseDate(legacy.getDate())) // 날짜 형식 통일
        .build();
});

canonicalStream.to("canonical.orders");
```



## 6. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. Command·Document·Event Message의 가장 단순한 구분 기준은?

"실패 시 누가 책임지는가"다. Command는 발신자가 재시도를, Event는 컨슈머가 재처리를, Document는 다음 발행 주기 또는 별도 보상 로직이 책임진다. 같은 토픽에 의도가 섞이면 책임이 모호해져 재시도·재처리 로직이 충돌한다. 그래서 토픽 네이밍을 `commands.*`, `events.*`, `documents.*`로 갈라 두는 컨벤션이 운영적으로 의미가 있다.

### Q2. CloudEvents 컨벤션이 Schema Registry와 어떻게 정렬되는가?

CloudEvents는 `type`, `source`, `datacontenttype` 같은 메타 필드를 표준화하고, 그 안의 `type` 값을 도메인이 정한다. Schema Registry의 subject 네이밍 전략(예: TopicRecordNameStrategy)이 이 `type`과 자연스럽게 결합된다. 같은 토픽이 OrderCreated와 OrderCancelled를 동시에 운반할 때, CloudEvents `type`이 인간 친화적 분류를, Schema Registry subject가 머신 친화적 호환성 경계를 제공한다.

### Q3. Kafka 헤더 기반 Format Indicator와 Schema Registry 기반의 차이는?

헤더는 *역직렬화 전*에 형식을 분기할 때 유용하다. Avro인지 JSON인지를 5바이트 wire format을 깨지 않고 알 수 있다. Schema Registry 기반은 *역직렬화 중*에 Writer/Reader 스키마를 매칭해 호환성을 검사한다. 둘은 충돌하지 않고 같이 쓴다. 헤더로 "어느 직렬화 계열인지", Schema Registry로 "어느 버전인지"를 나누는 식이다.

### Q4. Kafka에 메시지별 TTL이 없는데 만료 메시지를 어떻게 처리하나?

만료 시각을 헤더(예: `expires-at`)로 박아 두고, 컨슈머가 처리 시작 전에 `System.currentTimeMillis()`와 비교해 건너뛴다. 동시에 토픽 보존 기간을 만료 시각보다 길게 잡아야 한다. 모니터링 측면에서는 "만료로 스킵된 메시지 수"를 카운터로 노출해야 실제 트래픽에서 만료 정책이 너무 짧지 않은지 검증할 수 있다.

### Q5. Event Standardizer를 진입점에 둘지 소비자별로 둘지 결정하는 기준은?

소비자 수와 변환 로직 안정성이 기준이다. 소비자가 3개 이상이고 변환 로직이 거의 동일하다면 진입점에 두는 게 압도적으로 싸다(N+M vs N×M). 반대로 소비자마다 변환이 *근본적으로* 다르거나, 소스 측이 자주 바뀌어서 표준 스키마가 자주 깨질 위험이 크다면 소비자별 변환이 단기적으로 안전하다. 단 이 경우에도 정규 스키마 일부(공통 식별자, occurredAt 등)는 진입점에서 통일시키는 하이브리드가 보통 정답이다.
