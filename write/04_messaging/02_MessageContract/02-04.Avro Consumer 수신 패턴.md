# Avro 직렬화와 Consumer 수신 패턴 두 가지

---

> Kafka에서 Avro 메시지를 발행하고 소비할 때, 발신자가 누구냐에 따라 Consumer의 수신 방식이 달라진다. 앱 내부에서 Avro로 직렬화한 메시지는 SpecificRecord로 직접 받을 수 있지만, 외부 시스템이 JSON으로 보낸 메시지는 byte[]로 받아서 수동 파싱해야 한다. 이 문서는 두 케이스를 분리하게 된 배경과 구현 패턴을 정리한다.


## 1. 왜 두 가지 패턴이 필요한가

Confluent Schema Registry 기반 Avro 직렬화는 단순한 바이너리 변환이 아니다. wire format에 magic byte(0x00) + 4바이트 schema ID + Avro 바이너리가 포함되며, 이 형식을 생성하려면 Schema Registry와 통신하는 전용 직렬화기(`KafkaAvroSerializer`)가 필요하다.

문제는 `rpk topic produce`나 Jenkins 파이프라인처럼 외부 시스템이 메시지를 보내는 경우다. 이들은 Schema Registry 클라이언트를 내장하고 있지 않으므로, Confluent wire format을 생성할 수 없다. 보낼 수 있는 건 plain JSON 뿐이다.

이 JSON 메시지를 Avro Consumer(`KafkaAvroDeserializer`)로 받으면 magic byte가 없어서 역직렬화가 실패한다. 따라서 발신자의 직렬화 능력에 따라 토픽과 Consumer를 분리해야 한다.


## 2. 토픽 분리 원칙

같은 토픽에 Avro와 JSON이 섞이면 Consumer가 메시지 형식을 추측해야 하므로, 토픽 단위로 직렬화 형식을 고정한다:

```java
/** 앱 내부 Avro 직렬화 토픽 — EventPublisher → Avro Consumer */
public static final String JENKINS_BUILD = "tps.jenkins.build";

/** 외부 시스템(rpk, Jenkins 등) JSON 토픽 — rpk produce → JSON Consumer */
public static final String JENKINS_BUILD_EXTERNAL = "tps.jenkins.build.external";
```

네이밍 규칙은 `{도메인}.{이벤트}` 토픽에 `.external` 접미사를 붙여서 외부 입력 토픽임을 명시하는 방식이다.


## 3. 케이스 1: 앱 내부 Avro 발행 → Avro 직접 수신

### 3-1. 발행

`EventPublisher`는 `SpecificRecord`를 받아 내부에서 Avro 직렬화 후 아웃박스에 저장한다. 호출자는 byte[]를 다루지 않는다:

```java
JenkinsBuildRequest request = JenkinsBuildRequest.newBuilder()
        .setJobName(jobName)
        .setMessage("deploy")
        .setCorrelationId(correlationId)
        .setEnv("dev")
        .build();

eventPublisher.publish(
        "JENKINS_BUILD_AVRO", jobName
        , "jenkins.build.avro.trigger", request
        , Topics.JENKINS_BUILD, correlationId
);
```

`EventPublisher.publish()`의 시그니처가 `SpecificRecord`만 받으므로, 직렬화 책임이 라이브러리 내부로 캡슐화된다. 토픽 정보도 함께 전달되어 `TopicRecordNameStrategy` 같은 토픽 기반 subject 전략으로의 전환이 가능하다.

### 3-2. Avro 전용 ConsumerFactory

기본 Consumer는 `ByteArrayDeserializer`를 사용하므로, Avro SpecificRecord를 직접 수신하려면 별도 `ContainerFactory`가 필요하다:

```java
@Configuration
public class AvroKafkaConsumerConfig {

    @Bean
    public ConsumerFactory<String, SpecificRecord> avroConsumerFactory(
            KafkaProperties kafkaProperties
            , @Value("${spring.kafka.properties.schema.registry.url}") String schemaRegistryUrl
    ) {
        Map<String, Object> props = kafkaProperties.buildConsumerProperties(null);
        props.put("key.deserializer", StringDeserializer.class);
        props.put("value.deserializer", KafkaAvroDeserializer.class);
        props.put("schema.registry.url", schemaRegistryUrl);
        props.put(KafkaAvroDeserializerConfig.SPECIFIC_AVRO_READER_CONFIG, true);
        props.put("value.subject.name.strategy"
                , "io.confluent.kafka.serializers.subject.RecordNameStrategy");
        return new DefaultKafkaConsumerFactory<>(props);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, SpecificRecord>
            avroKafkaListenerContainerFactory(
                    ConsumerFactory<String, SpecificRecord> avroConsumerFactory
    ) {
        var factory = new ConcurrentKafkaListenerContainerFactory<String, SpecificRecord>();
        factory.setConsumerFactory(avroConsumerFactory);
        return factory;
    }
}
```

이 설정은 message-lib 라이브러리에 포함되므로, 라이브러리를 의존하는 모든 앱에서 `containerFactory = "avroKafkaListenerContainerFactory"`를 지정하면 사용할 수 있다.

### 3-3. 수신

`@Payload`로 Avro 클래스를 직접 받고, `@Header`로 CloudEvents 메타데이터를 개별 추출한다:

```java
@KafkaListener(
        topics = Topics.JENKINS_BUILD
        , groupId = "jenkins-avro-build-consumer"
        , containerFactory = "avroKafkaListenerContainerFactory"
)
public void onMessage(
        @Payload JenkinsBuildRequest request
        , @Header(value = "ce_id", required = false) String ceId
        , @Header(value = "ce_correlationid", required = false) String correlationId
) {
    log.info("이벤트 수신 → jobName={}, ce_id={}", request.getJobName(), ceId);
    jenkinsClient.triggerBuildWithParameters(
            request.getJobName().toString(), params);
}
```

`ConsumerRecord`를 다루거나 `AvroSerializer.deserialize()`를 호출할 필요가 없다. 역직렬화는 `KafkaAvroDeserializer`가 리스너 호출 전에 처리한다.


## 4. 케이스 2: 외부 시스템 JSON 발행 → 수동 파싱 수신

### 4-1. 발행 (외부)

외부 시스템은 `rpk`나 HTTP webhook으로 JSON 문자열을 보낸다:

```bash
rpk topic produce tps.jenkins.build.external --key "my-job" \
  <<< '{"jobName":"TF-TEST/my-job","message":"hello","env":"dev"}'
```

이 메시지에는 Confluent wire format 헤더가 없으므로, Avro 역직렬화기로 처리할 수 없다.

### 4-2. 수신

기본 `ByteArrayDeserializer` Consumer를 사용하여 `ConsumerRecord<String, byte[]>`로 받고, `ObjectMapper`로 DTO에 매핑한다. `JsonNode`로 필드를 하나씩 꺼내는 대신 전용 record를 정의하면 타입 안전성과 가독성이 올라간다:

```java
@JsonIgnoreProperties(ignoreUnknown = true)
public record JenkinsExternalBuildRequest(
        String jobName
        , String message
        , String env
        , Map<String, String> parameters
) {}
```

Consumer에서는 `objectMapper.readValue()`로 바로 매핑한다:

```java
@KafkaListener(
        topics = Topics.JENKINS_BUILD_EXTERNAL
        , groupId = "jenkins-json-build-consumer"
)
public void onMessage(ConsumerRecord<String, byte[]> record) {
    JenkinsExternalBuildRequest request = objectMapper.readValue(
            record.value(), JenkinsExternalBuildRequest.class);

    jenkinsClient.triggerBuildWithParameters(request.jobName(), params);
}
```

`@JsonIgnoreProperties(ignoreUnknown = true)`를 붙이면 외부 시스템이 예상치 못한 필드를 보내도 역직렬화가 실패하지 않는다. `containerFactory`를 지정하지 않으면 Spring Boot 자동 설정의 기본 Consumer(`ByteArrayDeserializer`)가 적용된다.


## 5. @KafkaListener 파라미터 어노테이션

Avro Consumer에서 `ConsumerRecord` 대신 개별 파라미터로 메시지를 분해할 수 있는 이유는 Spring Kafka의 메시지 핸들러 어노테이션 덕분이다.

### @Payload

메시지의 값(value) 부분을 지정한 타입으로 변환하여 주입한다. `containerFactory`에 설정된 역직렬화기가 변환을 담당하므로, `avroKafkaListenerContainerFactory`를 사용하면 `KafkaAvroDeserializer`가 Avro 바이너리를 SpecificRecord로 변환한 결과가 들어온다:

```java
// Avro 역직렬화 결과가 JenkinsBuildRequest 타입으로 주입된다
public void onMessage(@Payload JenkinsBuildRequest request) { ... }
```

`@Payload`를 생략해도 동작하지만, 명시적으로 붙이면 "이 파라미터가 메시지 본문"이라는 의도가 드러난다. 검증(`@Valid`)과 함께 쓸 수도 있다.

### @Header

Kafka 레코드의 헤더에서 특정 키의 값을 추출하여 주입한다. CloudEvents 헤더처럼 메타데이터를 개별 변수로 받을 때 유용하다:

```java
public void onMessage(
        @Payload JenkinsBuildRequest request
        , @Header(value = "ce_id", required = false) String ceId
        , @Header(value = "ce_type", required = false) String ceType
        , @Header(value = "ce_correlationid", required = false) String correlationId
) { ... }
```

`required = false`를 지정하면 해당 헤더가 없어도 예외가 발생하지 않고 `null`이 들어온다. 외부 시스템이 보낸 메시지에는 CloudEvents 헤더가 없을 수 있으므로, `required = false`가 안전하다.

### @Headers

개별 헤더가 아닌 전체 헤더 맵을 한 번에 받을 때 사용한다:

```java
public void onMessage(
        @Payload JenkinsBuildRequest request
        , @Headers MessageHeaders headers
) {
    String ceId = (String) headers.get("ce_id");
}
```

헤더 종류가 많거나 동적으로 접근해야 할 때 `@Header`를 하나씩 나열하는 것보다 간결하다.

### ConsumerRecord와의 비교

`ConsumerRecord<String, byte[]>`로 받으면 토픽, 파티션, 오프셋, 키, 값, 헤더를 모두 하나의 객체에서 꺼낸다. 어노테이션 방식은 이것을 분해하여 필요한 것만 파라미터로 선언하는 패턴이다:

```java
// ConsumerRecord 방식 — 모든 정보를 하나에서 꺼냄
public void onMessage(ConsumerRecord<String, byte[]> record) {
    byte[] value = record.value();
    String ceId = headerValue(record, "ce_id");
    int partition = record.partition();
}

// 어노테이션 방식 — 필요한 것만 선언
public void onMessage(
        @Payload JenkinsBuildRequest request          // value (역직렬화 완료)
        , @Header("ce_id") String ceId                // 헤더 하나
        , @Header(KafkaHeaders.RECEIVED_PARTITION) int partition  // 파티션
) { ... }
```

두 방식은 섞어 쓸 수 없다. `ConsumerRecord`를 파라미터로 선언하면 다른 어노테이션 파라미터는 무시된다.


## 6. 두 패턴의 비교

세 가지 측면에서 비교하면 각 패턴의 적용 범위가 명확해진다.

**직렬화 측면에서**, Avro 패턴은 Confluent wire format(magic byte + schema ID + 바이너리)을 사용하므로 Schema Registry를 통한 스키마 진화가 가능하다. JSON 패턴은 plain text이므로 스키마 관리가 불가능하고, 필드 추가/삭제 시 Consumer 코드를 직접 수정해야 한다.

**타입 안전성 측면에서**, Avro Consumer는 `@Payload JenkinsBuildRequest`로 컴파일 타임에 타입이 보장된다. JSON Consumer는 `JsonNode.path("jobName")`처럼 문자열 키로 접근하므로, 오타가 런타임에서야 발견된다.

**적용 대상 측면에서**, Avro 패턴은 앱 내부 이벤트(EventPublisher → Outbox → Kafka)에 사용한다. JSON 패턴은 Schema Registry 클라이언트가 없는 외부 시스템(`rpk`, Jenkins 파이프라인, curl 등)이 보내는 메시지에 사용한다.


## 7. 커스텀 Deserializer를 직접 만들지 않는 이유

여기서 한 가지 짚어둘 것이 있다. Kafka는 `Deserializer<T>` 인터페이스를 구현해 객체별 역직렬화기를 직접 만드는 길도 열어 두지만, *권장하지 않는다*. Producer 측 커스텀 Serializer와 마찬가지로, 커스텀 역직렬화는 producer와 consumer를 강하게 결합시키고 취약하며 에러를 부르기 쉽다.

이유는 직렬화기와 역직렬화기가 *반드시 짝이 맞아야* 하기 때문이다. `IntegerSerializer`로 쓰고 `StringDeserializer`로 읽으면 결과가 깨진다. 그래서 개발자는 각 토픽에 어떤 직렬화기로 썼는지 일일이 추적하고, 토픽마다 그 역직렬화기로 해석 가능한 데이터만 들어가도록 보장해야 한다. 조직에 producer·consumer가 많아질수록 이 약속을 손으로 지키기가 어려워진다.

이것이 바로 Avro와 Schema Registry를 쓰는 이점이다. `KafkaAvroDeserializer`는 토픽에 쓰인 모든 데이터가 토픽 스키마와 호환됨을 보장하므로, 짝이 맞는 역직렬화기와 스키마로 읽힌다. producer든 consumer든 호환성 오류는 적절한 에러 메시지로 즉시 드러나, raw byte 배열을 들여다보며 직렬화 오류를 디버깅할 필요가 없다. 커스텀 역직렬화기를 쓰는 대신 JSON·Thrift·Protobuf·Avro 같은 표준 형식을 쓰는 편이 낫고, 이 문서가 Avro 패턴을 다루는 이유도 여기에 있다. Producer 측 커스텀 Serializer의 같은 논지는 [01-03.Avro](01-03.Avro.md)에서 다룬다.


## 8. 기본 Consumer를 Avro로 바꾸면 안 되는 이유

"Avro만 쓸 거면 기본 역직렬화기를 `KafkaAvroDeserializer`로 바꾸면 되지 않나?"라는 질문이 나올 수 있다. 안 되는 이유는 DLQ 때문이다.

`DlqConsumer`는 처리 실패한 메시지를 원본 그대로 수신해야 한다. 실패한 메시지가 Avro일 수도 있고 아닐 수도 있으므로, DLQ Consumer는 `byte[]`로 받아야 안전하다. 기본 역직렬화기를 Avro로 바꾸면 DLQ Consumer가 non-Avro 메시지를 수신할 때 역직렬화 실패가 발생한다.

따라서 기본은 `ByteArrayDeserializer`로 유지하고, Avro 수신이 필요한 리스너만 `containerFactory = "avroKafkaListenerContainerFactory"`를 지정하는 구조가 안전하다.
