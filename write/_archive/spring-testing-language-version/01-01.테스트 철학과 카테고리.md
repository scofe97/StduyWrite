# 테스트 철학과 카테고리

---

> 테스트 코드는 검증 도구이기 전에 *구현 코드의 첫 사용자*다. 테스트를 짜면서 어려움을 느끼면 그 어려움 자체가 *설계 개선 신호*가 된다. 이 글은 TPS 코드베이스(311개 테스트 파일)를 출발점으로 삼아, 테스트 카테고리 네 개와 그것을 가르는 결정 근거를 정리한다.



## 학습 목표

> 테스트가 *구현의 첫 사용자*라는 관점과 카테고리별 적합 시점을 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. "테스트 작성이 어렵다 = 설계 개선 신호"라는 피드백 루프를 설명할 수 있다.
2. Unit·Slice·Integration·E2E 네 카테고리의 정의와 경계를 설명할 수 있다.
3. 각 카테고리가 검증하는 *책임 범위*와 *실행 비용*의 트레이드오프를 비교할 수 있다.
4. TPS 모듈별 테스트 분포가 *왜 그 모양인지* 설명할 수 있다.
5. 어느 시점에 어떤 카테고리를 고를지 결정 트리를 그릴 수 있다.



## 1. 테스트는 구현의 첫 사용자

카카오페이 Mock Part 2는 다음과 같이 정리한다.

> "테스트 코드는 단순한 검증 도구가 아니라 구현 코드의 *첫 사용자 역할*을 한다. 테스트 코드 작성이 어렵다면 구현 코드 품질에 문제가 있다는 신호다."

이 관점이 시리즈 전체의 출발점이다. 테스트가 어려운 이유를 분해하면 보통 다음 셋 중 하나다.

- 의존성이 많아 *Mock 셋업이 거대*해진다 → 책임 분리 신호
- 외부 시스템에 직접 닿아 *결정성이 깨진다* → 어댑터 분리 신호
- 데이터 셋업이 길어 *핵심 검증이 가려진다* → 픽스처 추출 신호

테스트가 어려울 때 가장 위험한 대응은 *테스트 편의를 위해 운영 코드를 바꾸는 것*이다. 카카오페이 Mock Part 1이 명시한 안티패턴이다.

> "테스트를 편리하게 하기 위해 운영 코드 설계를 변경하는 것은 바람직하지 않다."

대신 *테스트가 어렵게 만드는 구조 자체*를 다시 본다. 책임을 가르고, 외부 의존성을 인터페이스로 추출하고, 도메인 로직을 POJO로 뽑아낸다. 그러면 테스트는 자연스럽게 쉬워진다.



## 2. 카테고리 네 개

테스트는 *검증 범위와 실행 비용*에 따라 네 가지로 갈린다. 모든 분류 체계가 그렇듯 경계가 흐릿한 자리가 있지만, 일관된 어휘를 가지는 것이 운영적으로 중요하다.

| 카테고리 | 범위 | 실행 비용 | Spring Context |
|---------|------|----------|---------------|
| **Unit** | 한 클래스(또는 두세 객체) | 수 ms | 없음 |
| **Slice** | 한 계층(`@WebMvcTest`, `@DataJpaTest`) | 수백 ms | 부분 로드 |
| **Integration** | 다수 빈 결합(`@SpringBootTest`) | 수 초 | 전체 로드 |
| **E2E** | 실제 외부 의존성(DB·Kafka·HTTP) | 수십 초 이상 | 전체 로드 + 컨테이너 |

### 2.1 Unit

도메인 로직을 *외부 의존성 없이* 검증한다. Mockito mock이 있어도 *순수 객체 단위*에서 동작한다.

TPS 예: `OutboxRetryPolicyTest`(`message-lib/src/test/java/.../domain/outbox/OutboxRetryPolicyTest.java`).

```java
private static final LocalDateTime NOW = LocalDateTime.of(2026, 5, 9, 12, 0, 0);
private final Clock fixedClock = Clock.fixed(NOW.toInstant(ZoneOffset.UTC), ZoneOffset.UTC);
private final OutboxRetryPolicy policy = new OutboxRetryPolicy(fixedClock);

@Test
@DisplayName("retryCount=3 → 다음 재시도 8초 후 (2^3 = 8)")
void nextRetryAt_threeRetries_eightSeconds() {
    OutboxEvent event = newEvent(3);
    assertThat(policy.nextRetryAt(event)).isEqualTo(NOW.plusSeconds(8));
}
```

`Clock.fixed`로 시계를 박아 *결정성*을 보장한다. Spring Context 없음, mock 없음, 의존성 한 줄. 가장 가벼운 카테고리다.

### 2.2 Slice

Spring의 한 계층만 로드한다. `@WebMvcTest`는 web layer만, `@DataJpaTest`는 JPA layer만 로드해 *주변 빈을 격리*한다. TPS에는 *거의 없다* — 03-01에서 그 이유를 따로 다룬다.

### 2.3 Integration

`@SpringBootTest`로 전체 ApplicationContext를 로드한다. 대표 예가 `AbstractMariaDbIntegrationTest`(`operator/ticket/.../testsupport/AbstractMariaDbIntegrationTest.java`)다.

```java
@Tag("integration")
@ActiveProfiles("test")
@SpringBootTest
public abstract class AbstractMariaDbIntegrationTest {

    @ServiceConnection
    @SuppressWarnings("resource")
    protected static final MariaDBContainer<?> MARIADB =
            new MariaDBContainer<>(DockerImageName.parse("mariadb:10.11"))
                    .withDatabaseName("TPS")
                    .withUrlParam("characterEncoding", "utf8")
                    .withUrlParam("useUnicode", "true");

    static { MARIADB.start(); }
}
```

`@ServiceConnection`이 컨테이너의 datasource를 자동 주입하고, `static { start(); }` 한 번으로 JVM 전체에서 컨테이너를 재사용한다. 03-02에서 본격적으로 다룬다.

### 2.4 E2E

외부 의존성을 *실제로* 띄워 흐름 전체를 검증한다. TPS executor 모듈의 `e2e/lifecycle`·`e2e/wiremock`·`e2e/recovery` 디렉터리가 대표 예다. WireMock으로 Jenkins HTTP를 흉내내고, Testcontainers로 DB·Kafka를 띄워 *제어된 외부 환경*을 만든다.

```
executor/engine/src/test/java/org/okestro/tps/jenkins/e2e/
├── lifecycle/   - 정상 라이프사이클
├── recovery/    - 장애 복구
├── wiremock/    - Jenkins HTTP fault injection
├── concurrency/ - 동시성 검증
└── scheduler/   - 시간 기반 트리거
```

04-02에서 본격적으로 다룬다.



## 3. TPS 모듈별 분포

| 모듈 | 테스트 파일 수 | 주된 카테고리 | 특이 사항 |
|------|--------------|-------------|----------|
| **executor** | 76 | Unit + E2E | Jenkins E2E + WireMock 중심. Slice 거의 없음 |
| **message-lib** | 10 | Unit + Integration(IT) | Outbox·Kafka 설정 검증. 작지만 인프라 핵심 |
| **operator** | 225 | Integration | ticket 도메인 중심, Testcontainers MariaDB 다수 |
| **합계** | **311** | | |

이 분포가 만들어진 이유는 모듈의 *책임*과 일치한다.

- `executor`는 *외부 시스템(Jenkins)과의 통신*이 본질이라 E2E + WireMock으로 흐름을 검증한다.
- `message-lib`는 *인프라 라이브러리*라 단위 테스트로 동작을 박고, IT로 Kafka·DB 결합을 검증한다.
- `operator`는 *도메인 로직과 영속성*이 본질이라 Integration이 다수다. ticket 도메인이 가장 무겁다.

Slice가 거의 없는 이유는 TPS가 *애플리케이션 부팅 시간이 짧고* `@SpringBootTest` 전체 로드 비용이 *허용 범위 안*이기 때문이다. Slice는 Context 로드 시간이 *문제가 되기 시작할 때* 가치를 발휘한다. 모듈별로 빈 수가 크게 늘면 그때 도입한다.



## 4. 어느 카테고리를 고를지 — 결정 트리

```mermaid
flowchart TD
    Q1{외부 의존성<br/>(DB/HTTP/Kafka)에<br/>닿는가?}
    Q1 -- "아니오" --> UNIT["Unit 테스트<br/>(POJO + Mockito mock)"]
    Q1 -- "예" --> Q2{흐름이<br/>다수 빈에<br/>걸쳐 있는가?}
    Q2 -- "아니오" --> SLICE["Slice 테스트<br/>(@WebMvcTest 등)"]
    Q2 -- "예" --> Q3{외부 시스템을<br/>실제로 띄워야<br/>하는가?}
    Q3 -- "아니오" --> INTEG["Integration<br/>(@SpringBootTest + Mock)"]
    Q3 -- "예" --> E2E["E2E<br/>(Testcontainers + WireMock)"]

    style UNIT fill:#e8f6e8,stroke:#3a8a3a,color:#000
    style SLICE fill:#e8f4fd,stroke:#1f78b4,color:#000
    style INTEG fill:#fef9e7,stroke:#b07d12,color:#000
    style E2E fill:#fce8e8,stroke:#a83232,color:#000
```

기본은 *가장 가벼운 카테고리*에서 시작한다. 검증이 부족하면 한 단계씩 올린다. 거꾸로, Integration·E2E로 짠 테스트 중 *Unit으로 내려갈 수 있는 것*이 있으면 분리한다. 가벼운 테스트가 많을수록 피드백 루프가 빨라진다.

이 결정 트리는 *지침*이지 *철칙*이 아니다. 도메인 특성에 따라 같은 분기에서도 다른 결론이 나올 수 있다. 중요한 건 *결정을 의식적으로* 한다는 점이다.



## 5. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. 테스트가 어렵다는 신호를 어떻게 *설계 개선*으로 연결하나?

세 갈래 신호를 분해한다. (1) *Mock 셋업이 거대*하다 → 책임이 너무 많다는 신호, 클래스를 가른다. (2) *외부 시스템에 직접 닿아 결정성이 깨진다* → 어댑터 인터페이스가 없다는 신호, port-adapter로 추출한다. (3) *데이터 셋업이 길어 검증이 가려진다* → 픽스처가 없거나 잘못됐다는 신호, DomainFixture를 도입한다. 테스트 코드를 고치는 게 아니라 *운영 코드 구조*를 다시 본다.

### Q2. Unit·Slice·Integration·E2E의 결정적 경계는?

*Spring Context 로드 여부와 실제 외부 시스템 사용 여부*다. Unit은 Context 없음. Slice는 한 계층만 로드(`@WebMvcTest`). Integration은 전체 Context 로드(`@SpringBootTest`)에 *mock 외부 시스템*. E2E는 *실제 외부 시스템*(Testcontainers·WireMock). 경계가 흐릿한 자리는 *비용*으로 자른다. 같은 분기를 위해 매번 다른 어휘를 쓰면 운영에서 일관성이 깨진다.

### Q3. TPS에 Slice 테스트가 거의 없는 이유는?

ApplicationContext 로드 시간이 *허용 범위 안*이고, `@SpringBootTest`를 그대로 써도 빠르기 때문이다. Slice는 Context 부팅이 *분 단위로 커진* 큰 모놀리스에서 가치를 낸다. 모듈별 빈 수가 더 커지거나, 한 PR이 100개 이상 테스트를 실행해 *부팅 시간이 누적 문제*가 되면 그때 도입한다. 지금은 비용보다 학습 곡선이 더 크다.

### Q4. operator에 테스트가 가장 많은 이유는?

ticket 도메인 자체가 가장 *상태 변환이 복잡*하고 *영속성 비중이 큰* 도메인이라서다. 도메인 규칙·승인 워크플로우·쿼리 변형이 많아 Integration 테스트로 *실 DB와의 결합*을 검증해야 한다. 반대로 executor는 *외부 시스템 호출*이 본질이라 단위 테스트보다 E2E + WireMock에 무게가 쏠린다. 분포 자체가 *모듈 책임의 거울*이다.

### Q5. 가장 가벼운 카테고리에서 시작하라는 원칙이 항상 옳은가?

대체로 옳지만 *외부 시스템과의 계약*이 본질인 코드는 처음부터 Integration이 옳을 때가 있다. Outbox 폴러의 `FOR UPDATE SKIP LOCKED` 동작은 *실 DB가 아니면 검증이 안 된다*. 같은 이유로 Kafka의 transactional commit 동작은 단위 테스트로 잡히지 않는다. 원칙은 "*확신을 얻기에 충분한 최소 비용*". 그 최소가 Integration이라면 그게 가장 가벼운 카테고리다.
