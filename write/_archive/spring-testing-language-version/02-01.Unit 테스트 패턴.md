# Unit 테스트 패턴

---

> Unit 테스트는 *외부 의존성 없는 단일 책임*을 검증한다. POJO 도메인 객체, 값 객체, 정책 클래스가 주된 대상이다. 이 글은 TPS의 `OutboxRetryPolicyTest`·`BuildLifecycleEventTest`·`ManifestWriterTest`를 가져와 *AssertJ·JUnit 5·`Clock.fixed`* 세 도구가 어떻게 결합되는지 정리한다.



## 학습 목표

> Unit 테스트가 *결정성·가독성·속도*를 동시에 만드는 패턴을 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. AssertJ의 fluent API가 hamcrest·JUnit assertion보다 가독성에서 유리한 이유를 설명할 수 있다.
2. `Clock.fixed`로 시계를 박는 패턴이 *왜 결정성을 위한 표준*인지 설명할 수 있다.
3. `@DisplayName`과 `@Nested`로 시나리오를 구성하는 방법을 설명할 수 있다.
4. POJO 단위 테스트가 *Mock 단위 테스트*와 어떻게 다른지 설명할 수 있다.
5. `assertThatThrownBy`로 예외 시나리오를 표현하는 패턴을 설명할 수 있다.



## 1. POJO 도메인 단위 테스트

가장 가벼운 형태다. 의존성이 *생성자에 1~2개*뿐인 도메인 객체를 직접 만들고 메서드를 부른다.

TPS `OutboxRetryPolicyTest`(`message-lib/src/test/java/.../domain/outbox/OutboxRetryPolicyTest.java`).

```java
class OutboxRetryPolicyTest {

    private static final LocalDateTime NOW = LocalDateTime.of(2026, 5, 9, 12, 0, 0);
    private final Clock fixedClock = Clock.fixed(NOW.toInstant(ZoneOffset.UTC), ZoneOffset.UTC);
    private final OutboxRetryPolicy policy = new OutboxRetryPolicy(fixedClock);

    @Test
    @DisplayName("retryCount=3 → 다음 재시도 8초 후 (2^3 = 8)")
    void nextRetryAt_threeRetries_eightSeconds() {
        OutboxEvent event = newEvent(3);
        assertThat(policy.nextRetryAt(event)).isEqualTo(NOW.plusSeconds(8));
    }

    private static OutboxEvent newEvent(Integer retryCount) {
        return OutboxEvent.builder()
                .aggregateId("agg-1")
                .payload(new byte[]{1})
                .topic("topic-a")
                .status(OutboxStatus.PENDING)
                .retryCount(retryCount)
                .build();
    }
}
```

네 가지 좋은 패턴이 결합돼 있다.

1. **`Clock.fixed`** — 시계를 박아 시간 의존 로직의 결정성을 보장한다.
2. **`@DisplayName` 시나리오** — 시나리오 본문이 한 문장으로 표현된다.
3. **`AssertJ` fluent API** — `assertThat(...).isEqualTo(...)`로 *주어-동사-목적어* 순서가 자연 영어와 같다.
4. **로컬 fixture 메서드** — `newEvent(Integer)`로 호출자 부담을 줄였다.



## 2. AssertJ 사용 패턴

JUnit 기본 assertion(`assertEquals`, `assertTrue`)보다 AssertJ가 *대부분의 자리에서* 더 읽힌다.

### 2.1 기본 단정

```java
// JUnit 기본
assertEquals(8, policy.nextRetryAt(event).getSecond());
assertTrue(result.isPresent());

// AssertJ fluent
assertThat(policy.nextRetryAt(event)).isEqualTo(NOW.plusSeconds(8));
assertThat(result).isPresent();
```

AssertJ는 *기대값이 메서드 체인 끝*에 온다. 영어 어순(`assert that X is equal to Y`)과 일치해 코드가 *그대로 시나리오*로 읽힌다.

### 2.2 컬렉션·optional

```java
assertThat(events)
    .hasSize(3)
    .extracting(OutboxEvent::getStatus)
    .containsExactly(PENDING, PROCESSING, SENT);

assertThat(optionalOrder)
    .isPresent()
    .get()
    .extracting(Order::getAmount)
    .isEqualTo(1500L);
```

체이닝 한 번으로 *컬렉션 필터·매핑·검증*이 한 줄에 모인다. 같은 검증을 JUnit 기본으로 쓰면 5~6줄로 늘어진다.

### 2.3 예외 단정

```java
assertThatThrownBy(() -> publisher.publish(null, new byte[]{1}, "topic-a"))
    .isInstanceOf(NullPointerException.class)
    .hasMessageContaining("aggregateId");
```

`assertThatThrownBy`가 람다를 받아 *던져진 예외를 fluent하게* 검증한다. JUnit의 `assertThrows`보다 메시지 검증 체이닝이 자연스럽다.

TPS `EventPublisherTest`(`message-lib/src/test/java/.../application/outbox/EventPublisherTest.java`)의 패턴.

```java
@Test
@DisplayName("AvroSerializer 빈 미주입 상태에서 SpecificRecord 발행 시 IllegalStateException")
void publish_specificRecordWithoutAvroSerializer_throws() {
    EventPublisher publisher = new EventPublisher(repository, null);

    ExampleMessageAvro record = ExampleMessageAvro.newBuilder()
            .setMessage("hi").setCorrelationId(null).build();

    assertThatThrownBy(() -> publisher.publish("agg-1", record, "topic-a"))
            .isInstanceOf(IllegalStateException.class)
            .hasMessageContaining("AvroSerializer is not configured");
    verify(repository, never()).save(any());
}
```

예외 검증 다음에 `verify(repository, never()).save(any())`로 *부작용도 일어나지 않았음*을 확인한다. 예외 단정과 mock verification이 같은 시나리오에 묶인다.



## 3. Clock.fixed로 시간 박기

시간 의존 로직은 *결정성*이 가장 깨지기 쉽다. `Instant.now()`나 `LocalDateTime.now()`를 코드 안에서 직접 호출하면 테스트가 *실행 순간*에 따라 결과가 달라진다.

해결책은 `java.time.Clock`을 *주입*하고 테스트에서 `Clock.fixed`로 박는 것이다.

```java
// 운영 코드
public class OutboxRetryPolicy {
    private final Clock clock;

    public OutboxRetryPolicy(Clock clock) {
        this.clock = clock;
    }

    public LocalDateTime nextRetryAt(OutboxEvent event) {
        long delaySeconds = Math.min(1L << event.getRetryCount(), 1L << 30);
        return LocalDateTime.now(clock).plusSeconds(delaySeconds);
    }
}

// 테스트
private final Clock fixedClock = Clock.fixed(NOW.toInstant(ZoneOffset.UTC), ZoneOffset.UTC);
private final OutboxRetryPolicy policy = new OutboxRetryPolicy(fixedClock);
```

운영 환경에서는 `Clock.systemDefaultZone()`이 주입되고, 테스트에서는 고정 시계가 주입된다. 코드는 *시간 의존성을 명시*하고 테스트는 *그 시간을 직접 통제*한다.

같은 패턴이 `Random`, `UUID.randomUUID()`, *환경변수* 같은 비결정 자원에 모두 적용된다. *의존성을 명시*하면 테스트가 자연스럽게 결정적이 된다.



## 4. @Nested로 시나리오 묶기

같은 SUT(System Under Test)의 여러 조건을 *카테고리로* 묶을 때 `@Nested`가 강하다.

```java
class OutboxRetryPolicyTest {

    @Nested
    @DisplayName("shouldMarkDead 판정")
    class ShouldMarkDead {
        @Test @DisplayName("retryCount < maxRetries 면 DEAD 마킹 대상 아님")
        void belowMax_returnsFalse() { ... }

        @Test @DisplayName("retryCount == maxRetries 면 DEAD 마킹 대상")
        void atMax_returnsTrue() { ... }
    }

    @Nested
    @DisplayName("nextRetryAt 지수 백오프")
    class NextRetryAt {
        @Test @DisplayName("retryCount=0 → 1초 후 (2^0)")
        void zeroRetries_oneSecond() { ... }
    }
}
```

테스트 보고서가 *카테고리 → 시나리오* 트리로 정리된다. 새 시나리오 추가가 *어느 카테고리 안인지* 자연스럽게 강제된다.

TPS는 `@Nested`를 일관되게 쓰지는 않는다. 단순 케이스에서는 *flat한 메서드 나열*이 더 읽힌다. `@Nested`는 시나리오 카테고리가 *3개 이상* 명확히 갈리는 자리에 도입한다.



## 5. POJO 단위 테스트 vs Mock 단위 테스트

같은 "단위 테스트"라도 두 변종이 있다.

| 변종 | 의존성 처리 | 적합 시점 |
|------|------------|----------|
| **POJO 단위** | 의존성 없음 또는 *실제 객체* 직접 생성 | 도메인 객체, 값 객체, 정책 클래스 |
| **Mock 단위** | 의존성을 `@Mock`으로 주입 | 어플리케이션 서비스, 어댑터 |

POJO 단위가 *더 가볍고 더 빠르다*. 하지만 모든 클래스를 POJO로 만들 수 있는 건 아니다. 외부 시스템 호출을 감싸는 어플리케이션 서비스는 *Mock 단위*가 자연스럽다. 결정은 *클래스의 책임*이 가른다.

TPS `OutboxRetryPolicyTest`는 POJO 단위(`Clock`과 `OutboxEvent`만 직접 생성). `EventPublisherTest`는 Mock 단위(`OutboxEventRepository`, `AvroSerializer`를 mock). 같은 모듈 안에서도 책임에 따라 변종이 갈린다.

02-02에서 Mock 단위를 본격적으로 다룬다.



## 6. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. AssertJ를 JUnit 기본 단정보다 선호하는 결정적 이유는?

*어순과 체이닝*이다. JUnit `assertEquals(expected, actual)`은 인자 순서를 헷갈리기 쉽고, 비교 외 검증을 같이 묶기 어렵다. AssertJ `assertThat(actual).isEqualTo(expected)`는 영어 어순과 일치하고, `.extracting(...)`·`.hasSize(...)`·`.contains(...)`로 *한 줄에 여러 검증*을 묶는다. 같은 코드를 JUnit으로 쓰면 5줄이 AssertJ에서 1줄이 되는 자리가 흔하다.

### Q2. `Clock.fixed`로 시간을 박는 패턴이 단지 시간 검증을 위한 것 외에 가치가 있나?

*의존성 명시화*가 본질적 가치다. `LocalDateTime.now()`를 직접 호출하는 코드는 *시간에 의존한다는 사실을 숨긴다*. `Clock`을 생성자로 주입하면 그 의존이 *시그니처에 박힌다*. 테스트에서 시간을 통제할 수 있는 건 결과적 부산물이고, 진짜 가치는 *코드가 자기 의존을 정직하게 드러내는 것*이다. 같은 원칙이 `Random`, `UUID`, `Path` 등에 모두 적용된다.

### Q3. `@Nested`를 언제 도입하고 언제 피하나?

시나리오 카테고리가 *3개 이상 명확히 갈릴 때* 도입한다. 메서드 4~5개가 flat하게 나열되어도 문제 없으면 `@Nested`는 *과한 구조*다. 도입했다가 시나리오가 줄면 다시 flat으로 돌리는 게 좋다. `@Nested`는 *카테고리가 시나리오를 분류하는* 보조 도구지, *카테고리 자체를 만드는* 도구가 아니다.

### Q4. `assertThatThrownBy`가 `assertThrows`보다 좋은 자리는?

*메시지·원인 검증 체이닝*이다. `assertThatThrownBy(...).hasMessageContaining(...).hasCauseInstanceOf(...).rootCause().hasMessage(...)`처럼 *예외의 여러 측면을 한 체인*에 묶을 수 있다. `assertThrows`는 예외 객체를 반환받아 *별도 변수에 담아 추가 검증*해야 한다. 예외 검증이 1~2개로 끝나면 둘 다 비슷하지만, *3개 이상*이면 `assertThatThrownBy`가 압도적으로 읽힌다.

### Q5. POJO 단위 테스트와 Mock 단위 테스트 중 *우선* 도전할 것은?

POJO 단위다. 같은 검증을 POJO로 짤 수 있다면 *항상 POJO가 더 가볍다*. Mock 셋업 코드가 사라지고, *실제 동작*이 검증된다. POJO로 못 짜는 자리(예: 어플리케이션 서비스)는 Mock 단위가 자연스럽지만, *POJO로 못 짜는 게 정말 책임 분리 때문인지*를 먼저 점검한다. "Mock이 너무 많다 = POJO로 빼낼 도메인 객체가 있다"는 신호일 수 있다.
