# GitOps 성숙도와 미래 고려사항
---
> ArgoCD를 설치했다고 GitOps가 끝나는 것은 아니다. 조직이 커질수록 저장소 구조, 환경 프로모션, 렌더링 방식, 승인 절차가 더 큰 설계 문제가 된다.



## 학습 목표
> 장기 운영 관점에서 GitOps 체계를 돌아본다.

이 장에서 확인할 목표는 다음과 같다:

1. monorepo와 polyrepo의 차이를 GitOps 관점에서 설명할 수 있다.
2. rendered manifests 패턴의 장단점을 설명할 수 있다.
3. 환경 프로모션과 승인 구조를 어떻게 설계할지 판단할 수 있다.



## 1. 저장소 구조
> GitOps의 품질은 저장소 구조에서 이미 절반이 결정된다.

monorepo는 전반 구조를 보기 쉽고 cross-cutting 변경이 편하다. 반면 팀과 서비스가 많아질수록 충돌이 잦아질 수 있다. polyrepo는 팀 자율성은 높지만 공통 규칙 강제가 어려워질 수 있다.

정답은 조직 구조와 배포 빈도에 따라 달라진다. 중요한 것은 “어디가 진실 공급원인가”와 “환경별 값이 어디서 갈라지는가”를 명확히 하는 것이다.



## 2. Rendered Manifests 패턴
> Git에 최종 렌더링 결과를 둘지, 소스만 둘지에 대한 선택이다.

Helm과 Kustomize 원본만 Git에 두고 ArgoCD가 렌더링하는 방식은 일반적이다. 반면 렌더링된 최종 manifest를 따로 커밋하는 방식은 감사와 diff 관점에서 장점이 있다.

하지만 rendered manifests는 저장소 중복과 동기화 복잡도를 만든다. 그래서 강한 감사 요구나 별도 승인 체계가 없다면 무조건 기본값으로 둘 필요는 없다.



## 3. 환경 프로모션
> dev에서 prod로 가는 길은 브랜치 전략과 리뷰 전략이 같이 결정한다.

브랜치 기반 프로모션, 디렉토리 기반 프로모션, PR 승인 기반 promotion 등 여러 방식이 있다. 여기서 중요한 것은 “이미지 태그나 values 변경이 어느 시점에 누가 승인한 상태인가”를 명확히 남기는 것이다.

Image Updater를 함께 쓴다면 이 경계가 더 중요해진다. 자동 태그 갱신이 dev에는 편하지만 prod까지 그대로 이어지면 통제력을 잃기 쉽다.



## 4. monorepo vs polyrepo — 비교 표
> “팀 자율성 vs 공통 규칙 강제”의 트레이드오프를 6개 축으로 정리한다.

| 축 | monorepo | polyrepo |
|----|---------|---------|
| 변경 가시성 | 한 PR로 cross-cutting 보임 | repo별 분산, 합산 어려움 |
| 공통 규칙 강제 | 중앙 규칙·CI 일괄 적용 | repo별 표류 가능 |
| 팀 자율성 | 충돌 잦음 | 높음 |
| 권한 분리 | 디렉토리·CODEOWNERS 의존 | repo 권한이 자연 경계 |
| 빌드 시간 | 큰 repo는 길어짐 | 작은 단위로 빠름 |
| ArgoCD source 매핑 | path별 Application | repo별 Application |

305P처럼 “같은 매니페스트 저장소 한 곳 + 환경별 values”는 monorepo의 변형이다. 환경 분기는 path가 아니라 values 파일로 잡혀 있어 cross-cutting 가시성을 유지하면서 환경별 사고 반경을 분리하는 절충안에 가깝다.



## 5. 305P 환경 프로모션 패턴
> 305P 매니페스트 저장소(`tps_manifest`)에서 환경 프로모션이 어떻게 표현되는지 짧게 정리한다.

| 환경 | values 파일 | 트리거 | 승인 |
|------|-----------|------|------|
| dev | `values/values-dev.yaml` | Image Updater 자동 | 자동 |
| ppp(305P) | `values/values-ppp.yaml` | Image Updater 자동 | 자동 |
| prd | `values/values-prd.yaml` | PR merge | 사람 승인 |
| bok(폐쇄망) | `values/values-bok.yaml` | PR merge + 별도 이미지 적재 | 사람 승인 + 폐쇄망 절차 |

dev/ppp는 자동 흐름으로, prd/bok는 PR 게이트로 갈라지는 구조다. Image Updater가 어떤 파일을 갱신할지(`write-back-target`)를 환경별로 분리한 점이 이 패턴의 안전 장치다.



## 다음 단계
> 여기까지 오면 ArgoCD는 단일 도구가 아니라 플랫폼 운영 체계 일부가 된다.

이후에는 Argo Rollouts, 정책 엔진, 공급망 보안, 서비스 메시와의 연동처럼 더 넓은 운영 주제로 확장될 수 있다. 하지만 그 전에 저장소 경계, AppProject, 운영 알림, 롤백 절차가 안정됐는지 먼저 확인하는 편이 더 현실적이다.
