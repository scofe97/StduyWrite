# Argo Rollouts와 배포 전략 실습
---
> 본문(04-03)의 개념을 실제 클러스터에 적용해 본다. 컨트롤러 설치 → Canary Rollout 적용 → AnalysisTemplate 붙이기 → Blue-Green 변형 순으로 진행한다.



## 실습 환경
> 개인 GCP K8s 클러스터 또는 305P DEV 환경 둘 다 가능하다. 본 실습은 in-cluster ArgoCD가 이미 설치돼 있다고 가정한다.

| 항목 | 값 |
|------|-----|
| Kubernetes | v1.30+ |
| ArgoCD | v2.10+ |
| Argo Rollouts | v1.7+ |
| 트래픽 라우팅 | Nginx Ingress (또는 Istio) |
| 메트릭 백엔드 | Prometheus (`http://prometheus.<ns>.svc:9090`) |



## Step 1 — Argo Rollouts 컨트롤러 설치
> ArgoCD `Application`으로 컨트롤러 자체를 부트스트랩한다.

```yaml
# application-argo-rollouts.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: argo-rollouts
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://argoproj.github.io/argo-helm
    chart: argo-rollouts
    targetRevision: 2.x.x
    helm:
      values: |
        dashboard:
          enabled: true
  destination:
    server: https://kubernetes.default.svc
    namespace: argo-rollouts
  syncPolicy:
    automated: { prune: false, selfHeal: true }
    syncOptions: [CreateNamespace=true]
```

```bash
kubectl apply -f application-argo-rollouts.yaml
argocd app sync argo-rollouts

# 컨트롤러와 CRD 확인
kubectl -n argo-rollouts get deploy
kubectl get crd | grep argoproj.io | grep rollout

# Rollouts CLI 플러그인
brew install argoproj/tap/kubectl-argo-rollouts
kubectl argo rollouts version
```



## Step 2 — Canary Rollout 적용
> 본문 04-03의 spec을 거의 그대로 적용한다.

```yaml
# rollout-canary.yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: web-canary
  namespace: demo
spec:
  replicas: 5
  strategy:
    canary:
      canaryService: web-canary-svc
      stableService: web-stable-svc
      trafficRouting:
        nginx:
          stableIngress: web-ingress
      steps:
        - setWeight: 20
        - pause: { duration: 2m }
        - setWeight: 50
        - pause: { duration: 2m }
        - setWeight: 100
  selector: { matchLabels: { app: web } }
  template:
    metadata: { labels: { app: web } }
    spec:
      containers:
        - name: web
          image: argoproj/rollouts-demo:blue
          ports: [{ containerPort: 8080 }]
---
apiVersion: v1
kind: Service
metadata: { name: web-stable-svc, namespace: demo }
spec:
  selector: { app: web }
  ports: [{ port: 80, targetPort: 8080 }]
---
apiVersion: v1
kind: Service
metadata: { name: web-canary-svc, namespace: demo }
spec:
  selector: { app: web }
  ports: [{ port: 80, targetPort: 8080 }]
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-ingress
  namespace: demo
  annotations:
    kubernetes.io/ingress.class: nginx
spec:
  rules:
    - host: web.demo.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service: { name: web-stable-svc, port: { number: 80 } }
```

```bash
kubectl apply -f rollout-canary.yaml

# 초기 배포: stable 5개 + canary 0개
kubectl argo rollouts get rollout web-canary -n demo

# 새 버전으로 업데이트
kubectl argo rollouts set image web-canary web=argoproj/rollouts-demo:yellow -n demo

# 단계 진행 확인 (UI 또는 CLI watch)
kubectl argo rollouts get rollout web-canary -n demo --watch

# 첫 pause에서 즉시 promote
kubectl argo rollouts promote web-canary -n demo

# 문제 발생 시 abort → stable 회귀
kubectl argo rollouts abort web-canary -n demo
```



## Step 3 — AnalysisTemplate으로 자동 검증
> Prometheus 쿼리 결과로 단계 통과/롤백을 결정한다.

```yaml
# analysis-success-rate.yaml
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
  namespace: demo
spec:
  args: [{ name: service-name }]
  metrics:
    - name: success-rate
      interval: 30s
      count: 5
      successCondition: result[0] >= 0.99
      failureLimit: 1
      provider:
        prometheus:
          address: http://prometheus.monitoring.svc:9090
          query: |
            sum(rate(http_requests_total{service="{{args.service-name}}",code!~"5.."}[1m]))
              /
            sum(rate(http_requests_total{service="{{args.service-name}}"}[1m]))
```

`rollout-canary.yaml`의 steps 사이에 다음을 끼워 넣는다.

```yaml
- analysis:
    templates: [{ templateName: success-rate }]
    args: [{ name: service-name, value: web-canary-svc }]
```

```bash
kubectl apply -f analysis-success-rate.yaml
kubectl argo rollouts get rollout web-canary -n demo
# AnalysisRun이 단계 사이에 생성되고 Successful일 때만 다음 단계로 진행
```



## Step 4 — Blue-Green으로 변형
> 같은 워크로드를 blue-green strategy로 바꿔 본다.

```yaml
# rollout-bluegreen.yaml (변경 부분만)
spec:
  strategy:
    blueGreen:
      activeService: web-active
      previewService: web-preview
      autoPromotionEnabled: false
      scaleDownDelaySeconds: 300
      prePromotionAnalysis:
        templates: [{ templateName: success-rate }]
        args: [{ name: service-name, value: web-preview }]
```

```bash
kubectl apply -f rollout-bluegreen.yaml
kubectl argo rollouts set image web-bluegreen web=argoproj/rollouts-demo:yellow -n demo

# preview Service에서 새 버전을 직접 확인 후 promote
kubectl port-forward svc/web-preview 8080:80 -n demo
curl http://localhost:8080
kubectl argo rollouts promote web-bluegreen -n demo
```



## 검증 체크리스트
> 실습 후 다음을 모두 확인했다면 본문 04-03 학습 목표가 달성된 셈이다.

1. `kubectl argo rollouts get` 결과에서 stable/canary ReplicaSet이 비율대로 늘고 줄어드는 것을 봤다.
2. AnalysisRun이 단계 사이에 자동 생성되고 Successful/Failed에 따라 progress가 갈라지는 것을 봤다.
3. abort 시 stable로 즉시 회귀하는 것을 봤다.
4. ArgoCD UI에서 Application은 Healthy인데 Rollout이 단계 중간에 멈춰 있는 상태를 동시에 관찰했다.
5. ArgoCD Image Updater가 새 태그를 commit하면 Rollout이 자동으로 단계별 progress를 시작하는 흐름을 확인했다.



## 자주 만나는 함정
> 실습 중 자주 나오는 세 가지 오류와 회피.

| 증상 | 원인 | 회피 |
|------|------|------|
| canary가 0%에서 안 움직임 | trafficRouting 미설정 | nginx/istio/alb 중 하나 명시 |
| AnalysisRun이 항상 Inconclusive | 쿼리 결과 빈 벡터 | `count by()` 또는 `vector(1)` fallback 추가 |
| ArgoCD가 OutOfSync 무한 반복 | controller가 Rollout의 status 필드를 desired로 오인 | `argocd app set <app> --sync-option ApplyOutOfSyncOnly=true` |
