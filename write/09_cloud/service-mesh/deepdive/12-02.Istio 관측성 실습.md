# Istio 관측성 실습

> 본 장의 개념을 손으로 확인한다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`istioctl` 범용 명령으로 쓴다. 환경이 바뀌면 `## 실습 환경` 섹션만 갈아 끼우면 된다.


## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker |
| K8s | kubeadm v1.31.14 | — |
| Istio | istioctl 기준 설치 | 메시 주입 네임스페이스: `default`, `bookinfo` |

접속은 `gcp` 스킬의 SSH 절차를 따른다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1. 관측성 애드온 설치

```bash
# Prometheus, Grafana, Jaeger, Kiali 설치 (Istio 샘플 애드온)
kubectl apply -f https://raw.githubusercontent.com/istio/istio/release-1.22/samples/addons/prometheus.yaml
kubectl apply -f https://raw.githubusercontent.com/istio/istio/release-1.22/samples/addons/grafana.yaml
kubectl apply -f https://raw.githubusercontent.com/istio/istio/release-1.22/samples/addons/jaeger.yaml
kubectl apply -f https://raw.githubusercontent.com/istio/istio/release-1.22/samples/addons/kiali.yaml

# 설치 확인
kubectl get pods -n istio-system | grep -E "prometheus|grafana|jaeger|kiali"

# 대시보드 접근 (각각 별도 터미널에서)
istioctl dashboard prometheus
istioctl dashboard grafana
istioctl dashboard jaeger
istioctl dashboard kiali
```


## 실습 2. 표준 메트릭 확인 (Prometheus)

```bash
# 트래픽 발생
for i in $(seq 1 50); do
  kubectl exec -n bookinfo deploy/productpage -- \
    curl -s http://reviews:9080/reviews/1 > /dev/null
done

# Prometheus에서 메트릭 조회 (포트포워딩 후 브라우저)
kubectl port-forward -n istio-system svc/prometheus 9090:9090

# 브라우저에서 아래 쿼리 실행:
# 1. 요청 수 (reporter 구분)
# istio_requests_total{destination_service="reviews.bookinfo.svc.cluster.local"}
---

# 2. 에러율 (5xx 비율)
# sum(rate(istio_requests_total{reporter="destination",response_code=~"5.*"}[5m]))
# / sum(rate(istio_requests_total{reporter="destination"}[5m]))
---

# 3. p99 레이턴시
# histogram_quantile(0.99, sum(rate(istio_request_duration_milliseconds_bucket{reporter="destination"}[5m])) by (le))
---

# 4. response_flags 분포 (트러블슈팅용)
# sum by (response_flags) (irate(istio_requests_total{response_code=~"5.."}[5m]))
```


## 실습 3. reporter 차이 이해

```bash
# ratings에 10% 확률로 500 주입
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: ratings-fault
  namespace: bookinfo
spec:
  hosts:
  - ratings
  http:
  - fault:
      abort:
        percentage:
          value: 10.0
        httpStatus: 500
    route:
    - destination:
        host: ratings
        subset: v1
EOF

# 트래픽 발생
for i in $(seq 1 100); do
  kubectl exec -n bookinfo deploy/productpage -- \
    curl -s http://ratings:9080/ratings/1 > /dev/null
done

# Prometheus에서 두 관점 비교
# 클라이언트 관점 (source)
# sum(rate(istio_requests_total{reporter="source",destination_service_name="ratings",response_code=~"5.*"}[5m]))
---

# 서버 관점 (destination)
# sum(rate(istio_requests_total{reporter="destination",destination_service_name="ratings",response_code=~"5.*"}[5m]))
---

# 두 값이 일치하면 서버 애플리케이션 문제, 클라이언트만 높으면 프록시 레이어 문제
---

kubectl delete virtualservice ratings-fault -n bookinfo
```


## 실습 4. Grafana 대시보드 탐색

```bash
kubectl port-forward -n istio-system svc/grafana 3000:3000
# 브라우저: http://localhost:3000
```

Grafana에서 확인할 항목:

1. **Istio Mesh Dashboard**: 전체 서비스 목록과 에러율 현황
2. **Istio Service Dashboard**:
   - `reviews.bookinfo` 서비스 선택
   - Incoming Requests Success Rate (서버 관점)
   - Request Duration (p50/p90/p99 레이턴시)
3. **Istio Workload Dashboard**: Pod 수준 메트릭


## 실습 5. Kiali 서비스 그래프

```bash
kubectl port-forward -n istio-system svc/kiali 20001:20001
# 브라우저: http://localhost:20001
```

Kiali에서 확인할 항목:

```bash
# 트래픽 발생 (Kiali 그래프에 표시하기 위해)
for i in $(seq 1 30); do
  kubectl exec -n bookinfo deploy/productpage -- \
    curl -s http://productpage:9080/productpage > /dev/null
  sleep 0.5
done
```

1. **Graph** 메뉴 → `bookinfo` 네임스페이스 선택
2. 엣지 색상과 굵기로 트래픽 흐름 파악
3. 설정 오류가 있는 노드의 경고 삼각형 확인
4. **Istio Config** 메뉴 → VirtualService/DestinationRule 유효성 검사 결과 확인


## 실습 6. 분산 추적 (Jaeger)

```bash
kubectl port-forward -n istio-system svc/jaeger 16686:16686
# 브라우저: http://localhost:16686
```

```bash
# 추적 데이터 생성
for i in $(seq 1 10); do
  kubectl exec -n bookinfo deploy/productpage -- \
    curl -s http://productpage:9080/productpage > /dev/null
done
```

Jaeger에서 확인할 항목:

1. **Service** 드롭다운에서 `productpage.bookinfo` 선택
2. **Find Traces** 클릭
3. 트레이스 하나 클릭해 스팬 트리 확인
4. 헤더 전파가 올바르게 동작하면 productpage → reviews → ratings 스팬이 하나의 트레이스로 연결됨


## 실습 7. Telemetry CRD로 샘플링 설정

```bash
# 현재 샘플링 비율 확인 (기본 1%)
kubectl get meshconfig -n istio-system -o yaml | grep sampling

# 특정 네임스페이스 샘플링 비율 변경
kubectl apply -f - <<EOF
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: bookinfo-tracing
  namespace: bookinfo
spec:
  tracing:
  - providers:
    - name: jaeger
    randomSamplingPercentage: 100.0
EOF

# 변경 후 Jaeger에서 추적 수 증가 확인
for i in $(seq 1 20); do
  kubectl exec -n bookinfo deploy/productpage -- \
    curl -s http://productpage:9080/productpage > /dev/null
done

# 정리
kubectl delete telemetry bookinfo-tracing -n bookinfo
```


## 실습 8. 액세스 로그 활성화

```bash
# bookinfo 네임스페이스에 액세스 로그 활성화
kubectl apply -f - <<EOF
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: access-log
  namespace: bookinfo
spec:
  accessLogging:
  - providers:
    - name: envoy
EOF

# 트래픽 발생 후 로그 확인
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s http://reviews:9080/reviews/1 > /dev/null

# Access Log 확인 (response_flags 포함)
kubectl logs -n bookinfo -l app=reviews -c istio-proxy | tail -5

# 정리
kubectl delete telemetry access-log -n bookinfo
```


## 체크리스트

- [ ] `reporter="source"`와 `reporter="destination"` 메트릭의 차이를 PromQL로 확인했는가
- [ ] Grafana Istio Service Dashboard에서 p99 레이턴시를 시계열로 확인했는가
- [ ] Kiali 그래프에서 서비스 간 트래픽 흐름과 에러율을 시각적으로 파악했는가
- [ ] Jaeger에서 멀티홉 추적(productpage → reviews → ratings)이 하나의 트레이스로 연결되는지 확인했는가
- [ ] `Telemetry` CRD로 샘플링 비율을 변경하고 추적 수 변화를 Jaeger에서 확인했는가
- [ ] 액세스 로그에서 `response_flags` 필드를 확인했는가
- [ ] 카디널리티 폭발 방지를 위해 커스텀 레이블 추가 시 고유 값 수를 먼저 계산해야 함을 이해했는가
