# mTLS와 제로 트러스트 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.


## 점검 질문
> 인증서 교체 실패 시나리오, SPIFFE 채택 장벽, mTLS 성능 오버헤드, 제로 트러스트 충분조건 논의, PQC 마이그레이션 과제를 Q&A로 점검한다.


### Q1. 인증서 교체(certificate rotation) 실패 시나리오는 어떻게 되고, 서비스 전체 장애로 이어지지 않도록 어떤 설계가 필요한가?

mTLS의 가장 큰 운영 위험 중 하나는 인증서 만료다. 단기 인증서(24시간)를 자동으로 교체하는 시스템을 구축했더라도 교체 프로세스 자체가 실패하면 인증서가 만료되어 서비스 간 통신이 전면 차단되는 심각한 장애가 발생한다.

인증서 교체 실패의 원인은 여러 층에 걸쳐 있다:

1. 네트워크 레벨에서 CA 컴포넌트(SPIRE/istiod) 자체의 장애가 전체 메시의 인증서 발급을 막는다. 단일 CA는 단일 장애점(SPOF)이 되므로 HA(High Availability) CA 구성(최소 3개 복제본, 분산 키 저장소)이 필수다.
2. 인증서 캐시와 grace period 문제다. Envoy는 인증서를 교체할 때 기존 연결에서 이미 사용 중인 인증서를 즉시 교체하지 않는다. 연결 수립 시점의 인증서가 해당 연결 전체에 유효하므로, 교체 타이밍과 연결 드레이닝이 맞물려야 무중단 교체가 가능하다.
3. 모니터링 부재다. 자동화가 있더라도 인증서 만료까지 남은 시간, 최근 교체 성공 여부, CA 가용성 이 세 가지 메트릭을 수집하고 알림을 설정하지 않으면 첫 번째 실패를 사후에야 알게 된다.

예방적 설계 원칙은 인증서 유효 기간의 50%가 지났을 때 교체를 시작하는 것이다. 만료 24시간 전 알림, 6시간 전 페이저 알림, 1시간 전 에스컬레이션으로 단계화한다. 롤백 계획도 필요하다. 인증서 교체가 실패했을 때 mTLS를 일시적으로 PERMISSIVE 모드로 전환해 평문 통신을 허용할 수 있는 절차를 미리 작성해 두어야 한다. 이것은 보안을 희생하는 결정이지만 서비스 완전 장애보다는 나은 선택일 수 있다.

### Q2. SPIFFE는 서비스 정체성 문제를 어떻게 해결하는가? 실제 채택에서 어떤 장벽이 존재하는가?

SPIFFE의 핵심은 SVID(SPIFFE Verifiable Identity Document)다. SVID는 X.509 인증서 형태로 발급되며, Subject Alternative Name(SAN) 필드에 `spiffe://trust-domain/path/service-name` 형식의 URI를 담는다. 이 URI가 서비스의 정체성이고, mTLS handshake에서 양쪽이 서로의 SVID를 검증함으로써 신뢰가 성립된다.

실제 채택의 장벽은 세 가지다:

1. trust domain 설계 복잡성이다. 멀티 클러스터 환경에서 trust domain을 어떻게 설계할지, 페더레이션을 어떻게 구성할지가 비자명하다. 잘못된 trust domain 설계는 나중에 변경하기 어렵다. 조직 도메인으로 단순하게 시작하는 것이 좋다(`spiffe://company.com`).
2. 레거시 시스템 통합이다. VM이나 베어메탈에서 실행 중인 서비스도 SPIFFE 정체성을 가져야 하는데, 레거시 서비스마다 통합 작업이 필요하다.
3. 정책 관리 복잡성이다. SPIFFE 정체성을 인증 기반으로 쓰기 위해서는 "서비스 A의 정체성(SVID)을 가진 요청만 서비스 B에 접근 허용"하는 정책을 어딘가에서 관리해야 한다. Istio의 AuthorizationPolicy, OPA, SPIRE 자체 등 여러 레이어에서 이 정책이 중복될 수 있다.

현실적인 채택 경로는 Kubernetes 워크로드만 있다면 Istio의 내장 SPIFFE 구현으로 시작하면 충분하다. SPIRE를 직접 운영할 필요는 없다. 멀티 클러스터나 하이브리드 환경(VM + Kubernetes)이 있을 때 SPIRE의 가치가 증대된다.

### Q3. mTLS 성능 오버헤드는 실제로 얼마이고, TLS 1.3과 세션 재개가 이를 어떻게 완화하는가?

TLS handshake의 비용은 CPU 비용(암호화 연산)과 레이턴시 비용(왕복 횟수) 두 부분으로 나뉜다.

TLS 1.2 전체 handshake는 2-RTT가 필요하다. 그러나 같은 데이터센터 내(1ms RTT)에서는 2ms 수준이다. TLS 1.3은 이를 1-RTT로 줄였고, 세션 티켓(session resumption)을 활용하면 0-RTT도 가능하다(단, 0-RTT는 replay attack에 취약하므로 멱등 요청에만 적용). 마이크로서비스 내부 통신처럼 같은 서비스 인스턴스와 반복적으로 연결하는 패턴에서 세션 재개는 handshake 비용을 90% 이상 줄인다.

CPU 비용은 현대 하드웨어에서 상대적으로 미미하다. AES-NI 명령어 셋이 있는 CPU에서 AES-GCM 암호화는 하드웨어 가속되어 1Gbps 처리에 CPU 코어 하나의 10-15%만 소비한다. 실제 오버헤드의 주범은 암호화 자체가 아니라 단기 연결 패턴이다. HTTP/1.1에서 각 요청마다 새 TCP+TLS 연결을 맺는 패턴이 있다면 handshake가 전체 레이턴시의 50% 이상을 차지할 수 있다. HTTP/2로 전환해 연결을 재사용하면 mTLS의 실효 오버헤드가 극적으로 줄어든다.

현실적인 결론: 데이터센터 내 마이크로서비스 통신에서 mTLS의 레이턴시 오버헤드는 HTTP/2와 연결 재사용이 전제될 때 0.2-0.5ms 수준이다. `envoy/stats`에서 `ssl.handshake`와 `ssl.session_reused` 메트릭을 모니터링해 session_reused 비율이 낮다면(50% 미만) 연결 풀 설정을 검토해야 한다.

### Q4. 제로 트러스트와 심층 방어는 어떻게 다르고, mTLS가 제로 트러스트를 "달성"하는가 "가능하게 하는가"?

심층 방어(Defense in Depth)는 여러 보안 레이어를 쌓는 전략이다. 방화벽, VPN, 네트워크 분리, 엔드포인트 보안 각 레이어가 독립적으로 공격을 방어한다. 이 전략의 전제는 "경계(perimeter)가 있고, 그 안에 있는 것은 상대적으로 신뢰할 수 있다"이다.

제로 트러스트는 이 전제를 거부한다. "경계 내부에 있다는 것만으로는 신뢰를 부여하지 않는다. 모든 요청은 명시적으로 인증되고 권한이 확인되어야 한다." mTLS는 "이 서비스가 유효한 인증서를 가지고 있다"는 것을 증명하지만, 유효한 인증서를 가진 서비스가 다른 서비스의 민감한 데이터에 접근할 권한이 있는지는 별도로 정의되어야 한다.

완전한 제로 트러스트 구현에는 mTLS 외에 서비스 간 최소 권한 접근 정책(Istio AuthorizationPolicy), 지속적인 검증(요청마다 권한 재확인), 비정상 접근 탐지, 폭발 반경 최소화가 필요하다. 제로 트러스트 보안 성숙도를 단계적으로 구축하는 경로: 1단계는 mTLS 활성화(인증). 2단계는 AuthorizationPolicy로 서비스 간 접근 허용 목록 정의(인가). 3단계는 요청 수준 감사 로그. 4단계는 이상 탐지다. mTLS는 제로 트러스트의 필요조건이지 충분조건이 아니다.

### Q5. 포스트 퀀텀 암호화로의 마이그레이션은 서비스 메시 운영자에게 어떤 과제를 제시하는가?

PQC 마이그레이션의 기술적 과제는 알고리즘 교체가 아니라 키 크기와 성능의 변화에 있다. ML-KEM-768의 공개키는 1,184바이트인 반면 X25519(현재 ECDH)의 공개키는 32바이트다. 키 크기가 35배 증가해 TLS handshake의 패킷 수를 증가시키고 단편화(fragmentation)를 유발할 수 있다. 계산 비용도 ML-KEM이 X25519보다 CPU 사용량이 약 2-4배 높지만, 절대값 기준으로 여전히 빠르다.

타임라인 현실: NIST 표준이 2024년 확정되었고, TLS 1.3에서 ML-KEM을 사용하는 hybrid 방식이 RFC로 진행 중이다. 주요 TLS 라이브러리(BoringSSL, OpenSSL 3.5+)가 지원을 추가했고, Linkerd는 2.19에서 이미 X25519+ML-KEM-768 하이브리드를 지원한다. 실제 디폴트 전환은 2030년 전후일 가능성이 높지만, 금융, 의료, 국방 같은 규제 환경에서는 더 빠른 채택이 요구될 수 있다.

지금 할 수 있는 준비로는 민감한 데이터(금융 거래, 의료 기록, 개인정보)를 전송하는 경로를 식별하고 우선 마이그레이션 대상으로 표시하는 것이다. Hybrid 방식(PQC + 기존 알고리즘 모두 사용)이 마이그레이션 초기 단계의 현실적 접근이다. 양자 컴퓨터가 아직 위협이 아니어도 기존 보안이 유지되고, 양자 위협이 현실화되면 PQC가 보호해주는 구조다.

### Q6. mTLS STRICT 모드와 PERMISSIVE 모드를 운영 환경에서 전환할 때 어떤 위험이 있고, 어떻게 무중단으로 전환할 수 있는가?

PERMISSIVE → STRICT 전환의 위험 요소는 다음 세 범주다:

1. 메시에 포함되지 않은 클라이언트다. 사이드카가 없는 Pod, 클러스터 외부의 서비스, Kubernetes의 헬스체크(kubelet이 직접 TCP/HTTP 요청을 보냄)가 여기에 해당한다. liveness/readiness probe는 kubelet이 보내는 요청이므로 사이드카를 통하지 않아 mTLS를 사용할 수 없다. Istio는 probe rewriting을 지원하지만 이것이 활성화되어 있지 않으면 STRICT 모드에서 모든 헬스체크가 실패한다.
2. 전환 시 연결 재수립이다. STRICT를 적용하면 새 정책은 새 연결에 적용된다. 기존 연결은 유지되지만 그 연결의 다음 요청에서 검증이 달라질 수 있다.
3. 외부 로드 밸런서와 서비스 메시 경계다. ALB, NLB 같은 외부 로드 밸런서가 Pod로 직접 트래픽을 보내는 경우 이 트래픽은 mTLS를 사용하지 않는다.

안전한 전환 절차: 1) `istioctl analyze`로 사이드카 없는 트래픽 경로 사전 탐지. 2) 테스트 네임스페이스에 STRICT 적용 후 트래픽 모니터링(최소 24시간). 3) 프로덕션 네임스페이스를 하나씩 순서대로 전환하고 각 단계에서 에러율 확인. 4) 전체 전환 완료 후 1주일 모니터링. 전환 성공의 핵심 메트릭은 Prometheus의 `istio_requests_total{response_code="503"}`이 증가하면 mTLS 차단이 발생하고 있다는 신호다.
