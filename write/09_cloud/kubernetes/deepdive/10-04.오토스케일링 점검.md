# 오토스케일링 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 스케일링 정책과 메트릭 선택을 질문 중심으로 다시 점검한다.

### Q1. HPA가 동작하려면 무엇이 먼저 준비돼야 하는가?

CPU/메모리 기반 HPA라면 최소한 `Metrics Server`가 있어야 한다. 그리고 CPU 사용률 기반 HPA라면 대상 Pod에 `requests.cpu`가 반드시 있어야 한다. 그렇지 않으면 사용률을 계산할 수 없다.

```bash
# Metrics API 확인
kubectl top pod -A

# HPA 상태 확인
kubectl describe hpa <hpa-name> -n <ns>

# 대상 Deployment의 requests 확인
kubectl get deploy <deploy> -n <ns> -o yaml | grep -A5 resources:
```

심화 질문. CPU 기반 HPA가 `unknown` 메트릭 상태로 보이면 무엇을 먼저 보는가? Metrics Server 상태, 대상 Pod의 requests, HPA가 참조하는 리소스 이름과 네임스페이스를 순서대로 본다.

### Q2. HPA 쿨다운이 필요한 이유

HPA는 메트릭을 주기적으로 확인해 replica 수를 조정한다. 쿨다운 없이 즉각 반응하면 "스케일 업 → 메트릭 하락 → 스케일 다운 → 다시 메트릭 상승 → 스케일 업"의 진동이 발생한다. 새 Pod가 기동하는 데 시간이 걸리는데, 그 시간 동안 메트릭이 계속 높은 것을 HPA가 보고 또 스케일 업 명령을 내리는 오버슈팅도 발생한다.

`stabilizationWindowSeconds`가 이 문제를 해결한다. 스케일 다운의 경우 설정된 시간(기본 300초) 동안 최솟값을 추적하고, 그 값이 현재 replica보다 작을 때만 줄인다.

```bash
# HPA 생성 및 상태 확인
kubectl create namespace ch18-smoke

kubectl apply -n ch18-smoke -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: php-apache
spec:
  selector:
    matchLabels:
      run: php-apache
  template:
    metadata:
      labels:
        run: php-apache
    spec:
      containers:
        - name: php-apache
          image: registry.k8s.io/hpa-example
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: 200m
            limits:
              cpu: 500m
EOF

kubectl autoscale deployment php-apache \
  --cpu-percent=50 \
  --min=1 \
  --max=10 \
  -n ch18-smoke

# HPA 상태 모니터링
kubectl get hpa -n ch18-smoke -w

kubectl delete namespace ch18-smoke
```

### Q3. VPA와 HPA의 충돌 방지

CPU 메트릭으로 두 가지를 동시에 사용하면 충돌한다. VPA가 CPU Requests를 올리면 CPU 사용률(사용량/Requests)이 낮아지고, HPA가 replica를 줄이고, 그러면 다시 사용률이 올라가는 진동이 발생한다.

해결은 메트릭을 분리하는 것이다. HPA는 CPU 기반, VPA는 메모리 기반으로 사용한다.

```bash
# VPA Off 모드로 추천값만 확인
kubectl apply -f - <<EOF
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: php-apache-vpa
  namespace: ch18-smoke
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: php-apache
  updatePolicy:
    updateMode: "Off"   # 추천만, 실제 적용 안 함
EOF

# 추천값 확인
kubectl describe vpa php-apache-vpa -n ch18-smoke
# Recommendation:
#   Container Recommendations:
#     Container Name: php-apache
#       Lower Bound: cpu: 25m, memory: 262144k
#       Target:      cpu: 50m, memory: 262144k
#       Upper Bound: cpu: 200m, memory: 1Gi
```

심화 질문. VPA가 Pod를 재시작하지 않고 Requests를 변경할 수 있는가? 현재 VPA는 Pod를 재시작해야 Requests를 변경할 수 있다. In-Place Pod Vertical Scaling(KEP-1287)이 Kubernetes 1.27+에서 알파로 추가됐다. 안정화되면 재시작 없이 Requests를 조정할 수 있게 된다.

### Q4. KEDA ScaledObject 구조

KEDA는 이벤트 소스(Kafka lag, Redis 큐 길이, Prometheus 메트릭)로 Pod를 스케일한다. HPA의 External Metrics API를 통해 구현되므로 기존 HPA 인프라를 그대로 활용한다.

```bash
# KEDA 설치 확인
kubectl get pods -n keda

# Kafka 기반 ScaledObject 예시
kubectl apply -f - <<EOF
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: kafka-consumer
  namespace: production
spec:
  scaleTargetRef:
    name: kafka-consumer-deployment
  minReplicaCount: 0      # Scale to Zero 가능
  maxReplicaCount: 20
  cooldownPeriod: 60
  triggers:
    - type: kafka
      metadata:
        bootstrapServers: redpanda.redpanda:9092
        consumerGroup: orders-group
        topic: orders
        lagThreshold: "100"
EOF

# ScaledObject 상태 확인
kubectl get scaledobject -n production
kubectl describe scaledobject kafka-consumer -n production
```

심화 질문. KEDA가 `minReplicaCount: 0`으로 설정해 Scale to Zero했을 때 첫 이벤트 처리 지연이 발생하는가? 새 Pod가 기동하는 시간(수십 초) 동안 이벤트가 큐에 쌓인다. 이 Cold Start 지연이 허용 가능한지에 따라 `minReplicaCount: 1`과 `0` 중 선택한다. 배치 처리처럼 지연이 허용되면 0이 비용 효율적이다.

### Q5. Cluster Autoscaler 노드 제거 조건

Cluster Autoscaler는 노드 CPU/메모리 요청량이 일정 임계값 이하로 유지되는 시간이 충분히 길어야(기본 10분) 노드를 제거한다. 단순히 리소스가 낮다고 즉시 제거하지 않는다.

```bash
# Cluster Autoscaler 상태 확인
kubectl get configmap cluster-autoscaler-status \
  -n kube-system -o yaml

# 노드 제거 안전 여부 확인용 어노테이션
kubectl annotate pod <critical-pod> \
  cluster-autoscaler.kubernetes.io/safe-to-evict="false"
```

DaemonSet Pod와 로컬 볼륨을 사용하는 Pod는 이동 불가로 판단해 해당 노드를 제거하지 않는다. 이런 Pod가 있는 노드는 Cluster Autoscaler가 건드리지 않는다는 점을 계획에 포함해야 한다.

심화 질문. 노드 제거 시 실행 중인 Pod는 어떻게 처리되는가? Cluster Autoscaler가 노드를 삭제하기 전에 `kubectl drain`과 동일한 동작을 수행한다. Pod에 SIGTERM을 보내고 `terminationGracePeriodSeconds` 동안 대기한 뒤 노드를 제거한다. PodDisruptionBudget이 설정된 경우 최소 가용 Pod 수를 유지하면서 드레인한다.

### Q6. CPU 메트릭의 스케일링 한계

CPU 사용률 기반 HPA는 "현재 CPU가 높다"를 감지하지 못하는 상황이 있다. 요청이 들어오기 전에 미리 스케일 업하지 못한다. 응답 시간 지연이 이미 발생한 뒤에 스케일 업이 시작되므로, 트래픽 급증 시 초기 요청은 느린 응답을 받을 수 있다.

KEDA Prometheus Scaler로 HTTP 요청 수를 메트릭으로 사용하면 이 한계를 보완할 수 있다.

```yaml
triggers:
  - type: prometheus
    metadata:
      serverAddress: http://prometheus-server.monitoring:9090
      metricName: http_requests_total
      threshold: "100"      # 초당 요청 100개 기준
      query: |
        sum(rate(http_requests_total{namespace="production"}[1m]))
```

심화 질문. CPU 메트릭 외에 Kubernetes가 기본으로 지원하는 HPA 메트릭 소스는 무엇인가? `autoscaling/v2`에서 `Resource`(CPU/메모리), `Pods`(Pod별 평균), `Object`(Ingress, 다른 K8s 오브젝트), `External`(외부 메트릭)을 지원한다. `External` 타입을 통해 Datadog, CloudWatch 등 외부 메트릭 시스템과 연동할 수 있다.
