# JSONPath와 kubectl 고급 조회

> 운영에서는 리소스를 "보는 것"보다 "필요한 필드만 빠르게 뽑는 것"이 더 중요하다. JSONPath는 `kubectl` 출력에서 자동화와 점검 루틴을 만드는 핵심 도구다.



## 학습 목표
> 사람이 읽는 출력과 스크립트가 읽는 출력을 분리해 다룬다.

이 장에서 확인할 목표는 다음과 같다:

1. `-o jsonpath=...` 출력 형식을 설명할 수 있다.
2. `range`, 탭, 줄바꿈을 사용해 표 형태 출력과 반복 조회를 만들 수 있다.
3. JSONPath가 정규식을 지원하지 않는다는 점과 `jq`와의 경계를 설명할 수 있다.



## 1. 왜 JSONPath인가
> `kubectl get` 기본 출력만으로는 반복 점검이 어렵다.

예를 들어 "모든 Pod 이름과 시작 시각", "모든 노드의 InternalIP", "restartCount가 큰 컨테이너" 같은 조회는 표준 출력만으로는 불편하다. JSONPath는 필요한 필드만 뽑아 사람이 읽거나 스크립트로 넘길 수 있게 해 준다.

```bash
kubectl get pods -o=jsonpath="{range .items[*]}{.metadata.name}{'\t'}{.status.startTime}{'\n'}{end}"
```



## 2. 자주 쓰는 패턴
> 운영에서 반복되는 패턴 몇 개만 익혀도 생산성이 크게 오른다.

```bash
# 현재 네임스페이스 Pod 이름
kubectl get pods -o=jsonpath="{range .items[*]}{.metadata.name}{'\n'}{end}"

# 노드 이름과 InternalIP
kubectl get nodes -o=jsonpath="{range .items[*]}{.metadata.name}{'\t'}{.status.addresses[?(@.type=='InternalIP')].address}{'\n'}{end}"

# 특정 Pod의 재시작 횟수
kubectl get pod my-pod -o=jsonpath="{.status.containerStatuses[0].restartCount}"
```

공식 문서 기준으로 `kubectl` JSONPath는 `range`, `end`, 음수 인덱스 같은 확장을 지원하지만 정규식은 지원하지 않는다. 정규식 필터링이 필요하면 `-o json | jq ...` 조합이 더 적합하다.



## 3. 실무 포인트
> JSONPath는 시험용 요령보다 운영 자동화 도구로 보는 편이 오래 간다.

- 반복 점검 명령을 짧은 스니펫으로 저장해 둔다.
- 사람이 읽을 출력은 `custom-columns`, 정교한 스크립트는 `jsonpath` 또는 `jq`를 쓴다.
- 필드 구조를 모를 때는 `kubectl explain`과 `-o yaml`을 함께 본다.
