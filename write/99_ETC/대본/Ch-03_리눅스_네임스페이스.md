## 0. 인트로 — 컨테이너의 세 기둥
`[00:06]`

안녕하세요, 플레이빌더 카이입니다. 이번 챕터부터 컨테이너를 이루는 중요한 커널 기능 조합들을 살펴보겠습니다.

1. **네임스페이스(Namespace)** — 프로세스를 격리합니다.
2. **cgroup(컨트롤 그룹)** — 자원을 제한합니다.
3. **오버레이 파일시스템(OverlayFS)** — 파일시스템을 조립합니다.

이 셋이 합쳐져서 우리가 컨테이너라고 부르게 되는 것입니다. 이번 챕터에서는 이 셋 중 첫 번째 기둥인 **네임스페이스 격리**를 본격적으로 파고들어 보겠습니다.

## 1. 리눅스 네임스페이스 vs 쿠버네티스 네임스페이스 + 5가지 질문
`[02:09]`

여기서 말하는 네임스페이스는 여러분이 평소에 쓰시는 `kubectl get namespace`의 **쿠버네티스 네임스페이스가 아닙니다.** 쿠버네티스 네임스페이스는 파드나 서비스 같은 리소스를 그룹으로 묶는 **논리적 폴더**입니다(`default`, `kube-system` 등이 여기 속함).

반면 이번 챕터에서 다루는 **리눅스 네임스페이스**는 커널이 프로세스를 **격리(Isolation)하는 메커니즘**입니다. 훨씬 더 낮은 레벨, 커널 내부의 이야기입니다. 이름만 같을 뿐 완전히 다른 개념이라는 점 꼭 기억하세요.

리눅스 네임스페이스가 왜 그렇게 중요한지 **5가지 질문**으로 시작하겠습니다. (지금 답이 바로 안 떠올라도 괜찮습니다. 이 챕터는 이 질문들의 답을 찾아가는 과정입니다.)

1. 파드 안에 두 개의 컨테이너가 있을 때, 왜 **localhost로 사이드카 컨테이너에 접근**할 수 있을까요?
2. 컨테이너끼리 왜 서로의 프로세스가 보이지 않을까요? (`ps aux`를 써도 옆 컨테이너 프로세스는 전혀 안 보임)
3. 파드 안에서 컨테이너가 죽고 재시작되어도 왜 **IP가 변경되지 않을까요?**
4. 컨테이너 안에서 `ps`를 치면 왜 항상 내 프로세스가 **PID 1번**을 기록할까요?
5. 컨테이너 안에서 `hostname`을 쳐보면 노드 이름이 아니라 **파드 이름**이 출력됩니다. 왜 다르게 출력될까요?

이 다섯 가지 답은 전부 **네임스페이스 하나로 연결**됩니다. 마지막 파트에서 다시 검토하겠습니다.

## 2. 네임스페이스 정의 — 같은 커널, 독립된 시야
`[03:56]`

네임스페이스의 정의는 **같은 커널을 공유하면서도 각 프로세스에게 자신만의 독립된 시야를 제공하는 커널 기능**입니다.

다이어그램을 보면 맨 아래에 리눅스 커널이 하나 있고, 그 위에 프로세스 A와 B가 각각 독립된 영역에서 실행됩니다. 프로세스 A 입장에서는 "이 서버에 나(A)만 존재한다"고 느끼고, B도 "B만 존재한다"고 느낍니다. 실제로는 같은 커널 위에서 둘이 동시에 돌아가는데도 서로의 존재를 모릅니다. 서로가 서로를 못 보는 상황 — 이게 네임스페이스의 핵심 정의입니다.

"쿠버네티스만 배우면 되지, 왜 리눅스 네임스페이스까지 알아야 하나?" 답은 단순합니다. **리눅스 커널에는 `struct container`라는 구조체가 아예 존재하지 않습니다.** 커널은 "컨테이너"라는 개념 자체를 모릅니다. 커널이 아는 건 오직 **프로세스**뿐입니다. 커널 입장에서 nginx 컨테이너든 MySQL 컨테이너든 그냥 각자 따로 실행되는 하나의 프로세스일 뿐입니다.

그럼 우리가 "컨테이너"라고 부르는 건 뭘까요? **커널 기능의 논리적 조합**일 뿐입니다 — ① 네임스페이스로 정리하고, ② cgroup으로 CPU·메모리를 제한하고, ③ OverlayFS로 이미지 레이어를 조립한 것. Docker든 containerd든 CRI-O든 이런 도구들은 이 조합을 편리하게 해 주는 도구일 뿐이고, 도구가 바뀌어도 커널의 메커니즘은 동일합니다. 그래서 파드가 왜 격리되는지, 왜 각자의 IP를 가지는지가 전부 네임스페이스로 설명됩니다.

## 3. 실습 ① 호스트 초기 네임스페이스 공유 확인
`[07:40]`

노드 접속은 `terraform output`으로 확인합니다. 컨트롤 플레인에 접속해, sudo 권한으로 **systemd(PID 1번)**의 네임스페이스 번호를 확인합니다. `/proc`이라는 가상 파일시스템의 1번 네임스페이스죠.

```bash
sudo readlink /proc/1/ns/pid    # → pid:[4026531836]
```

`/proc/1/ns/pid`는 PID 1번 프로세스의 PID 네임스페이스 식별 정보를 담은 심볼릭 링크입니다. 결과는 **`4026531836`**. 이 번호는 커널이 생성한 수많은 가상 네임스페이스를 구분하기 위해 부여하는 고유 식별 번호로, 이를 **inode(아이노드)**라고 합니다. `836`으로 끝나는 이 번호가 호스트의 **초기 PID 네임스페이스** 번호입니다.

다음으로 kubelet의 네임스페이스 번호를 확인합니다.

```bash
pgrep kubelet                   # → 3307
sudo readlink /proc/3307/ns/pid # → 4026531836 (systemd와 동일!)
```

kubelet의 네임스페이스 번호도 systemd와 완전히 동일합니다. **containerd**도 동일하게 `836`으로 끝나는 번호입니다.

이제 호스트에 nginx를 직접 설치해 봅니다(컨테이너가 아니라 호스트 OS의 systemd가 관리하는 일반 서비스).

```bash
sudo apt update && sudo apt install -y nginx
sudo systemctl status nginx     # active 확인
```

호스트에 설치한 nginx의 네임스페이스 번호도 확인하면, systemd·kubelet·containerd와 **정확히 같은 번호**입니다. 네 개 프로세스 모두 호스트의 초기 PID 네임스페이스 번호를 쓰고 있는 거죠. **모두가 같은 방에 들어 있다는 증거**입니다. 같은 방에 있기 때문에 서로가 보이고, 필요하면 `kill`로 강제 종료도 가능합니다.

```bash
ps aux | grep nginx             # 마스터 1 + 워커 2개 프로세스
sudo kill <nginx-master-pid>
sudo systemctl status nginx     # failed — 종료됨 (격리 없음의 증거)
```

정리하면, 호스트에 직접 설치된 서비스는 전부 같은 PID 네임스페이스 `4026531836`을 공유합니다. **컨테이너란 이 공유 공간에서 프로세스 하나하나를 떼어내 별도의 네임스페이스로 분리하는 것**입니다.

## 4. 실습 ② 파드 격리 비교 — hostPID:true vs isolated
`[12:20]`

이제 쿠버네티스 파드를 두 개 띄워서, 하나는 격리를 해제하고 하나는 기본 격리를 유지한 상태로 두 파드의 PID 네임스페이스를 비교합니다.

`kubeadm-cluster` 레포지토리의 `lecture/chapter03/` 폴더에서 두 매니페스트를 봅니다.

- **`nginx-host-pid.yaml`** — `hostPID: true`를 줘서 호스트의 PID 네임스페이스를 공유.
- **`nginx-isolated.yaml`** — `hostPID` 필드가 생략(기본값 `false`)이라 새로운 PID 네임스페이스로 격리.

두 파드 모두 같은 nginx 이미지를 쓰지만 격리 상태가 완전히 다릅니다.

```bash
kubectl apply -f nginx-host-pid.yaml
kubectl apply -f nginx-isolated.yaml
kubectl get pods                 # 둘 다 Running (겉보기엔 동일)
```

두 파드 모두 워커 노드 1번에서 실행되니 워커 노드 1번에 접속해 컨테이너 ID를 확인합니다.

```bash
# host-pid 파드의 호스트 관점 PID
sudo crictl inspect <container-id> | grep pid   # → 23401
sudo ls -l /proc/23401/ns/pid    # → 4026531836 (호스트와 동일!)

# isolated 파드의 호스트 관점 PID
sudo ls -l /proc/23159/ns/pid    # → 4026532249 (완전히 다른 번호!)
```

- **`nginx-host-pid` 파드** → 호스트 초기 PID 네임스페이스(`...836`)에 그대로 합류. systemd·kubelet과 같은 방. **격리가 전혀 안 됨.**
- **`nginx-isolated` 파드** → `4026532249`라는 완전히 다른 번호. 커널이 이 파드를 위해 새 PID 네임스페이스를 만들어 준 것. 이게 쿠버네티스의 정상적인 격리 동작.

YAML에서 작성한 `hostPID: true`가 커널 레벨의 격리를 바꾸는 겁니다. 하지만 `hostPID: true`는 심각한 보안 위험입니다. 이 컨테이너에 공격자가 침투하면 `ps`로 호스트의 모든 프로세스를 보고 `kill`로 kubelet이나 컨테이너까지 강제 종료할 수 있습니다. 그래서 최신 쿠버네티스 파드 보안 표준인 **PSA(Pod Security Admission)**에서는 이 옵션을 강력히 차단합니다. **프로덕션 환경에서는 사용 금지**입니다.

## 5. 컨테이너 = 3가지 격리 기술의 조합
`[18:46]`

컨테이너는 단일 기술이 아니라, 리눅스 커널의 세 가지 격리 기술이 결합되어 만들어진 **유저 스페이스 런타임 환경**입니다.

1. **네임스페이스** — 프로세스 시야를 격리
2. **cgroup** — 자원을 제한
3. **OverlayFS** — 독립된 파일시스템을 만듦

이번 챕터에서 다루는 네임스페이스가 첫 번째 — 프로세스 간 시야 격리 경계를 세우는 역할입니다. 앞서 본 `nginx-isolated` 파드가 받은 `...249` 번호는 정확히 말해 **PID 네임스페이스 격리 경계의 식별자(커널이 부여한 inode 번호)**입니다. 이 경계가 다른 프로세스에 대한 접근과 가시성을 완전히 차단합니다.

같은 네임스페이스에 있으면 `ps`로 보고 `kill`로 강제 종료할 수 있다고 했죠. nginx가 kubelet을 종료시킬 수 있고, 그 반대도 가능합니다. 프로덕션에서 이런 일이 일어나면 끔찍하죠. 해결 방안은 **새로운 네임스페이스를 만드는 것**입니다. 프로세스를 별도 네임스페이스에 넣으면 서로 보이지 않고 건드릴 수 없습니다. 시야가 완전히 차단되기 때문입니다.

구체적으로 어떻게 만들까요? 커널의 **`clone` 시스템 콜에 `CLONE_NEW*` 플래그**를 전달합니다. 별표(`*`) 자리에 PID, NET, UTS 같은 것이 들어갑니다. 쿠버네티스에서는 **runc**가 `clone`을 호출하고, 커널이 새 네임스페이스를 생성해 줍니다. 그런데 우리가 runc를 직접 실행할 일은 거의 없으니, 학습·테스트용으로 똑같은 일을 하는 CLI 명령어가 있는데 그게 바로 **`unshare`**입니다. 이번 챕터 실습에서 `unshare`로 네임스페이스를 다뤄 봅니다.

## 6. kubectl → 커널 clone 전체 경로
`[21:45]`

`kubectl run nginx` 한 줄이 어떻게 커널의 `clone`까지 도달하는지 전체 경로를 봅니다.

```text
kubectl run nginx
  → (HTTPS) kube-apiserver        : etcd에 파드 오브젝트 저장 (아직 노드 미결정)
  → kube-scheduler                : 어느 워커 노드에 배치할지 결정 → nodeName 업데이트
  ──────── 여기까지 컨트롤 플레인 ────────
  → kubelet (워커 노드)           : API 서버를 Watch하다 자신에게 할당된 파드 감지
  → (gRPC/CRI) containerd         : "컨테이너 만들어 줘"
  → runc (OCI 런타임)             : ↓ syscall 경계
  → 커널: clone(CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS | CLONE_NEWUTS | CLONE_NEWIPC)
  → 새 네임스페이스 생성 → 그 안에서 nginx 프로세스 탄생 (PID 1번)
```

핵심은 주황색 **syscall 경계(System Call Boundary)**입니다. runc가 `clone` 시스템 콜을 호출하며 `CLONE_NEWPID`·`CLONE_NEWNET`·`CLONE_NEWNS`·`CLONE_NEWUTS`·`CLONE_NEWIPC` 플래그를 전달하는 순간, 실행 흐름이 유저 스페이스에서 리눅스 커널 영역으로 들어갑니다. 커널이 이걸 받으면 새로운 PID·NET·MNT·UTS·IPC 네임스페이스를 생성하고, 그 안에서 nginx 프로세스가 탄생합니다. 이 nginx의 PID는 당연히 **1번**입니다(새 PID 네임스페이스 안의 첫 프로세스이므로). 이게 바로 컨테이너의 시작입니다.

이 전체 경로를 알아야 컨테이너 생성 과정을 정확하게 설명할 수 있습니다.

## 7. 다른 경로, 같은 도착점
`[25:09]`

이 경로가 유일한 길은 아닙니다. 훨씬 짧은 길도 있습니다. **"Different path, same kernel destination"** — 경로는 다르지만 커널 도착점은 같다는 뜻입니다.

- **오케스트레이션/런타임 레이어**(쿠버네티스 경로): `kubectl run nginx` → API 서버 → 스케줄러 → kubelet → containerd → runc → `clone` 호출. **여섯 단계**를 거쳐 커널에 도달.
- **CLI 유틸리티 레이어**: `unshare ... /bin/bash` **한 줄**이면 끝. `unshare`가 `unshare(2)` 시스템 콜을 호출해 네임스페이스를 분리하고 fork로 자식 프로세스를 생성. **단 하나의 단계**로 커널에 도달.

가운데 **syscall 경계**를 두고, 두 경로 모두 이 경계를 넘어가서 만나는 곳은 같습니다. 커널에 진입하면 도착 지점은 이렇습니다.

```text
clone() → copy_process() : 기존 프로세스를 복제해 새 task_struct 생성
        → copy_namespaces() : 네임스페이스를 새로 만들지 / 물려받을지 판단
                              (CLONE_NEW* 플래그가 여기서 체크됨)
        → create_new_namespaces() : (kernel/nsproxy.c) 새 네임스페이스 구조체 할당
                                    → 새 네임스페이스 탄생
```

Docker를 쓰든, containerd·runc를 쓰든, 심지어 `unshare` 명령어를 쓰든 **결국 커널 내부에서 도달하는 지점은 똑같습니다.** 도구가 달라도 커널의 메커니즘은 하나 — 이게 우리가 리눅스를 먼저 배워야 하는 이유입니다.

## 8. 실습 ③ "격리 없음" 확인 — kill로 kubelet 재시작
`[29:01]`

이번 실습의 목표는 `unshare` 없이, 호스트 상태 그대로 모든 프로세스가 같은 네임스페이스에 들어 있다는 것을 확인하는 겁니다. 컨트롤 플레인에서 `sudo -i`로 루트 전환(네임스페이스 작업은 루트 권한 필요).

```bash
ls -l /proc/$$/ns      # 현재 셸($$)이 속한 모든 격리 방 번호
```

`$$`는 현재 셸의 PID를 뜻하는 bash 변수입니다. 출력을 보면 PID·cgroup·IPC·MNT·NET·USER·UTS 네임스페이스가 전부 `4026531...`으로 시작하는 호스트 초기 네임스페이스 번호입니다.

```bash
lsns -t pid            # PID 네임스페이스만 필터링
```

`lsns`는 list namespaces의 약자, `-t pid`는 PID 네임스페이스만 보여 달라는 뜻입니다. 출력을 보면 `NPROCS`(프로세스 수)가 **130개**로 나옵니다 — 호스트의 130개 프로세스가 전부 `4026531836`이라는 하나의 네임스페이스에 들어 있다는 뜻입니다. 그 아래 두 번째는 pause 컨테이너가 차지하는 네임스페이스로, 이미 다른 파드의 pause가 별도 네임스페이스로 격리되어 있음을 확인할 수 있습니다.

```bash
ls -l /proc/1/exe       # → /usr/lib/systemd/systemd (PID 1번의 진짜 이름)
readlink /proc/$$/ns/pid    # 현재 셸 = systemd와 같은 ...836
pgrep kubelet               # → 3307
readlink /proc/3307/ns/pid  # → ...836 (kubelet도 호스트와 같은 방)
```

kubelet 또한 호스트의 시스템 데몬이라 호스트와 같은 네임스페이스에 있어서 서로 확인이 가능합니다. 반면 **kube-apiserver**는 **Static Pod(스태틱 파드)**라 새 네임스페이스 방에서 격리되어 있어(컨테이너로 실행) PID 네임스페이스가 다르게 나옵니다.

같은 PID 네임스페이스에 있으면 서로 프로세스를 보고 `kill`로 종료시킬 수 있습니다. 터미널 두 개로 확인합니다.

```bash
# 터미널 1: kubelet 상태 1초마다 감시
watch -n1 systemctl show kubelet -p ActiveState,SubState,MainPID
# 터미널 2: kubelet 강제 종료
pgrep kubelet           # → 3307
sleep 5 && kill -9 3307
```

5초 후 터미널 1의 kubelet 상태가 `active` → `activating`(auto-restart)으로 바뀌고 MainPID가 `0`이 됐다가 새 PID가 올라옵니다. kubelet이 실제로 죽었다가 systemd의 `Restart=always` 정책으로 자동 재시작된 거죠. 이게 **"격리 없음"의 실체**입니다. (물론 다른 프로세스에 시그널을 보내려면 적절한 권한이 동반되어야 합니다.)

## 9. 격리 구조 — 이중 매핑·일방통행·PID 1번 생명줄
`[36:06]`

격리가 어떤 구조로 이루어지는지 다이어그램으로 봅니다. 바깥의 큰 상자가 호스트 초기 네임스페이스로, 그 안에 systemd·sshd·containerd·nginx·bash 같은 프로세스가 있습니다. 각 PID는 다르지만 네임스페이스 ID는 **전부 동일**합니다. 같은 방에 있으니 서로 보이고 `kill`로 종료할 수 있는 거죠.

안쪽에 새로 만든 PID 네임스페이스가 있습니다. PID 네임스페이스 안에서 처음 실행된 프로세스는 무조건 **PID 1번**을 부여받으며, 내부에서 발생하는 **좀비 프로세스를 수거**하는 등 시스템의 init 역할을 합니다. 호스트 시점에서 bash의 PID는 `3xxxx`였지만 새 네임스페이스 안에서는 PID가 **1번**입니다.

"어떻게 같은 프로세스가 두 개의 번호를 가질 수 있지?"의 답은 커널의 **`struct pid`** 구조체에 있습니다. 이 구조체가 네임스페이스별로 프로세스 ID를 별도로 관리하기 때문에 같은 프로세스에 두 개 이상의 번호가 붙을 수 있습니다.

이 의문을 풀 세 가지 핵심:

1. **이중 매핑(Dual Mapping)** — 하나의 프로세스에 두 PID. 호스트 PID는 `3xxxx`, 네임스페이스 안 PID는 `1`. 동일 대상인데 어느 시점에서 보느냐에 따라 번호가 달라짐.
2. **일방통행** — 호스트는 네임스페이스 안을 보고 `kill`까지 가능하지만, 내부 시점에서 호스트 프로세스를 `kill`하면 `No such process` 에러. 호스트의 다른 프로세스는 존재 자체를 모름.
3. **PID 1번이 생명줄** (가장 중요) — PID 1번이 종료되면 하나의 프로세스가 죽는 게 아닙니다. 커널이 **`zap_pid_ns_processes`** 함수를 호출해 네임스페이스 내부의 모든 프로세스에 `SIGKILL`을 보냅니다. PID 1번이 죽으면 그 네임스페이스 전체가 같이 종료됩니다. 그래서 **컨테이너에서 메인 프로세스가 죽으면 컨테이너 전체가 죽습니다.** 이 로직은 커널 소스 `kernel/pid_namespace.c`에 구현되어 있습니다.

> 참고: `kill`은 보통 특정 프로세스 하나를 종료하지만, `zap`은 여러 개를 한꺼번에 강하게 정리한다는 느낌입니다. 커널 코드에서 꽤 흔한 표현입니다.

정리하면, 호스트 시점은 모든 것을 볼 수 있는 관리자 시점, 새 네임스페이스는 자기 내부만 보는 격리된 시점. 시야는 일방통행이고, PID 1번이 종료되면 전체 프로세스가 종료됩니다.

## 10. 실습 ④ 새 PID 네임스페이스 만들기
`[42:26]`

새로운 PID 네임스페이스를 만들어 격리하는 실습입니다(Docker나 쿠버네티스 파드가 생성될 때 커널 깊숙한 곳에서 일어나는 과정).

먼저 호스트에서 실행 중인 프로세스 목록을 보고 containerd(`2153`)·kubelet(`32389`)의 PID를 확인해 둡니다. 잠시 후 격리된 네임스페이스 안에서 이들이 `/proc`에서 사라졌음을 확인할 겁니다.

```bash
unshare --pid --mount --fork bash
```

`unshare`는 Docker나 containerd가 컨테이너를 띄울 때 하는 행동과 동일한 명령어입니다. 세 플래그:

- **`--pid`** — 나만의 프로세스 번호 체계를 새로 만들어 1번부터 시작.
- **`--mount`** — 호스트의 폴더 마운트 장부를 쓰지 않고 나만의 격리된 파일시스템 공간을 가짐.
- **`--fork`** — 기존 셸을 덮어쓰지 말고 새 자식을 만들어, 새 방의 1번 주인을 bash로 함. (이 옵션이 없으면 현재 셸이 네임스페이스 안으로 빨려 들어가 호스트와의 연결이 끊김.)

PID는 격리됐지만 `/proc`은 여전히 호스트의 프로세스 정보를 보고 있으므로, 격리된 방에 맞는 새 `/proc`을 마운트해 덮어씌웁니다.

```bash
mount -t proc proc /proc        # 격리된 방 전용 /proc (tmpfs 아님, procfs)
ps aux                          # 이제 bash와 ps 딱 2개만 보임!
echo $$                         # → 1 (이 네임스페이스의 PID 1번 = 창조주)
```

`mount -t proc proc /proc`에서 `-t proc`은 "이 파일시스템은 procfs"라고 커널에 알려주는 타입이고, 앞의 `proc`은 관례적 원본 이름(아무 이름이나 가능), 뒤의 `/proc`은 목적지 폴더입니다.

호스트에서는 수백 개 프로세스가 보였는데 여기서는 bash와 ps 딱 둘만 보입니다. bash가 PID 1번 — 이게 **이중 매핑**의 의미입니다(호스트에선 수만 번대, 이 방에선 1번).

이제 호스트의 containerd PID(`2153`)를 이 격리된 방에서 종료 시도합니다.

```bash
kill -0 2153                    # → No such process (존재만 확인하는 테스트)
readlink /proc/$$/ns/pid        # → 4026532674 (호스트 ...836과 완전히 다름)
exit                            # 셸 종료 → 이 네임스페이스도 자동 소멸
ps aux                          # 호스트로 복귀 → 다시 전체 프로세스 보임
```

`kill -0`은 실제 종료가 아니라 프로세스 존재만 확인하는 테스트 핑입니다. `No such process`가 나옵니다 — 새 네임스페이스 방 안에서는 호스트 프로세스의 존재 자체를 알 수 없기 때문입니다(**일방통행**). `exit`로 PID 1번 셸을 종료하면 네임스페이스도 함께 소멸합니다 — §9의 "PID 1번이 죽으면 전체 소멸"의 실체입니다.

## 11. 8개 네임스페이스 전체 지도
`[48:36]`

컨테이너를 만드는 8개 네임스페이스 전체를 훑어봅니다. 새 프로세스 생성에 쓰는 대표 시스템 콜이 **`clone`**이고, 호출 시 "이 프로세스는 무엇을 격리할까"라는 옵션 값(`CLONE_NEW*` 플래그)을 같이 넘깁니다.

| # | 네임스페이스 | 플래그 | 역할 |
|---|------------|--------|------|
| 1 | **PID** | `CLONE_NEWPID` | 프로세스 ID 격리 — 컨테이너 안 bash가 PID 1번이 되는 이유 |
| 2 | **NET(네트워크)** | `CLONE_NEWNET` | 각 파드가 고유 IP를 가질 수 있는 이유 |
| 3 | **MNT(마운트)** | `CLONE_NEWNS` | 컨테이너가 자기만의 파일시스템을 갖게 함 |
| 4 | **UTS** | `CLONE_NEWUTS` | 호스트네임 격리 — `hostname`이 파드 이름으로 나오는 이유 |
| 5 | **IPC** | `CLONE_NEWIPC` | 공유 메모리·세마포어 격리 — 같은 파드 컨테이너끼리 메모리 고속 통신 |
| 6 | **USER(유저)** | `CLONE_NEWUSER` | UID/GID 매핑 격리 — 컨테이너 루트를 호스트 일반 유저로 매핑(보안 방어선) |
| 7 | **CGROUP** | `CLONE_NEWCGROUP` | cgroup 계층 구조의 **시야** 격리 |
| 8 | **TIME(타임)** | `CLONE_NEWTIME` | 시스템 시계 격리 — 2020년대 추가된 최신, **쿠버네티스 미사용** |

> **MNT 플래그가 왜 `CLONE_NEWNS`?** 2002년 격리 기술이 최초 도입됐을 때 마운트 네임스페이스가 첫 번째이자 유일한 네임스페이스라 `NEWNS`(NEW Namespace)로 만들어졌습니다. 이후 다른 네임스페이스가 추가되며 이름 일관성이 깨졌지만, 하위 호환성 때문에 커널 코드를 바꿀 수 없어 지금도 마운트는 `CLONE_NEWNS`로 남아 있습니다.

> **USER 네임스페이스의 보안** — 컨테이너 내부에서는 루트로 보이지만 호스트에서는 일반 유저로 매핑됩니다. 그래서 컨테이너 탈출이 일어나도 해커가 호스트의 루트를 얻지 못하게 하는 방어선 역할을 합니다.

> **CGROUP 네임스페이스 vs cgroup** — cgroup **자체**는 CPU·메모리 등 자원을 제한하는 기능이고, cgroup **네임스페이스**는 그 cgroup 계층 구조의 시야를 격리하는 기능입니다. 이름이 비슷하지만 역할이 완전히 다릅니다.

TIME을 제외하고 쿠버네티스가 실제 사용하는 건 **7가지**입니다(runc 스펙에 TIME 미포함).

## 12. 2-3-2 그룹 구조
`[54:58]`

7개 네임스페이스를 그냥 외우긴 어려우니 **2-3-2**라는 그룹으로 묶어, 바깥(호스트)에서 안쪽(컨테이너)으로 어느 레벨에서 동작하는지 큰 그림으로 봅니다.

**[2] 호스트(워커 노드) 레벨 — 보안 및 노드 통제**
- **USER 네임스페이스** — 권한 강등. 컨테이너 내부는 루트지만 호스트에서는 일반 유저로 래핑. 컨테이너를 탈출해도 호스트에서는 일반 유저라 루트 권한을 못 가짐.
- **CGROUP 네임스페이스** — 시야 격리. 컨테이너가 자기 cgroup을 봤을 때 자기 cgroup이 루트처럼 보이게 함. (TIME은 쿠버네티스 미사용.)

**[3] 파드 레벨 — 같은 머신처럼 동작**
- **NET 네임스페이스** — 같은 파드의 컨테이너들은 같은 IP를 공유. 컨테이너 A가 `localhost:8080`으로 접근하면 컨테이너 B도 `8080`으로 접근 가능(별도 네트워크 설정 불필요).
- **UTS 네임스페이스** — 파드 내 컨테이너들은 모두 동일한 호스트네임.
- **IPC 네임스페이스** — 같은 파드 컨테이너끼리는 네트워크를 거치지 않고 공유 메모리로 직접 공유. 이를 **Zero-copy 통신**이라고 함.

**[2] 컨테이너 레벨 — 컨테이너별 독립**
- **PID 네임스페이스** — 각 컨테이너는 자기만의 PID 1번. 컨테이너 A의 1번은 nginx, B의 1번은 Redis가 될 수 있음. 서로의 프로세스가 안 보이고 `kill`도 불가.
- **MNT 네임스페이스** — 각 컨테이너는 자기만의 파일시스템 격리. `/etc/nginx`를 수정해도 컨테이너 B의 `/etc`엔 영향 없음. 조립된 이미지 레이어(OverlayFS)가 각자의 루트 파일시스템이 되는 게 이 덕분.

정리: **2(호스트, 보안)** → **3(파드, 공유)** → **2(컨테이너, 격리)**. 바깥에서 안쪽으로 이 구조로 기억하면 됩니다.

## 13. IPC 심화 — 공유 메모리와 세마포어
`[1:01:26]`

IPC가 생소할 수 있어 **공유 메모리(Shared Memory)**와 **세마포어(Semaphore)**를 더 깊이 봅니다.

리눅스의 기본 원칙은 **격리**입니다. 프로세스 A와 B는 각자의 메모리 주소만 보고 서로의 메모리를 절대 못 봅니다. 하지만 A와 B(여기선 컨테이너)가 대용량 데이터를 주고받아야 한다면? 일반적인 방식(네트워크 소켓·파이프)이라면 A가 커널 공간으로 데이터를 복사하고, 커널이 다시 B의 공간으로 복사해야 합니다. 데이터가 클수록 이 복사 과정에서 심각한 성능 저하가 발생합니다.

**공유 메모리**는 이 과정을 생략합니다. 커널이 RAM의 특정 구역을 떼어내, 이 공간을 A와 B가 둘 다 직접 읽고 쓸 수 있게 권한을 엽니다. 양쪽이 동일한 메모리 주소를 직접 보고 있어 **커널 복사가 0회** — 이를 **Zero-copy**라 부릅니다. 커널이 제공하는 가장 빠른 통신 방식입니다.

그런데 공유 메모리는 빠르지만 치명적 문제가 있습니다. A와 B가 `0.001초`만에 오차도 없이 동시에 같은 메모리 위치에 데이터를 쓰려 하면 데이터가 뒤섞이고 깨집니다. 데이터 불일치가 발생하죠. 이것을 **레이스 컨디션(Race Condition, 경쟁 상태)**이라고 합니다.

이 문제를 막는 것이 **세마포어** — 정확히는 **동기화 메커니즘(Synchronization Mechanism)**입니다.

```text
1단계: 커널이 세마포어 값을 1(사용 가능)로 설정
2단계: 프로세스 A가 공유 메모리 쓰기 전 세마포어 확인 → 값이 1이면 접근,
       접근과 동시에 0(사용 중)으로 변경 → 잠금(lock) 활성화
       이때 B가 오면 값이 0이라 접근 불가 → 1로 바뀔 때까지 커널 단에서 대기
3단계: A가 쓰기를 마치고 나가며 값을 다시 1로 변경 → 잠금 반납(unlock)
       그제야 대기하던 B가 값을 0으로 바꾸고 데이터 쓰기 진행
```

세마포어는 다수의 프로세스가 하나의 공유 메모리에 접근할 때 데이터가 깨지지 않도록 **접근 순서를 엄격히 통제하는 동기화 장치**입니다. 메모리 접근 전 반드시 세마포어를 확인해야 하고, 값이 0이면 대기 명령을 프로세스에게 전달합니다.

## 14. 네트워크 네임스페이스와 veth pair
`[1:06:13]`

파드에 네트워크 네임스페이스가 만들어진 이후 통신 과정을 봅니다. 새 네트워크 네임스페이스가 갓 만들어지면, 파드라는 공간은 **밖으로 나가는 길이 완벽히 차단**된 상태입니다. 새 집에 입주했는데 인터넷이 설치 안 돼 통신할 방법이 없는 상황과 같죠.

파드 내부에는 자기 자신만 가리키는 **`lo`(루프백, loopback)** 인터페이스 하나만 있고 이마저도 비활성화되어 있습니다. 루프백은 흔히 부르는 `127.0.0.1`, 즉 localhost입니다. 현재 파드는 호스트와 연결되어 있지 않은 완전 단절 상태입니다.

파드는 호스트와 통신해야 하므로 쿠버네티스는 **CNI 플러그인**(우리 환경에선 Calico)을 사용합니다. CNI 플러그인이 **veth pair(가상 이더넷 페어, Virtual Ethernet Pair)** — 가상의 네트워크 케이블을 만듭니다. `pair`는 하나의 쌍으로, 애초에 만들어질 때부터 양쪽 끝이 무조건 하나로 연결되어 있어야 합니다(하나의 랜선과 같음).

```text
[파드 네임스페이스]  eth0 ──veth pair── veth-host  [호스트 네임스페이스]
                                            │
                                       가상 브릿지(CNI)
                                            │
                                       물리 NIC → 외부 인터넷
```

파드 쪽 끝은 파드 네임스페이스 안에 던져져 파드가 쓰기 쉬운 이름인 `eth0`으로 만들어지고, 반대쪽 끝은 호스트 네임스페이스에 `veth-host`로 만들어집니다. 마지막으로 호스트 쪽 `veth-host`가 CNI가 관리하는 가상 브릿지와 연결됩니다. 그러면 통신망이 개통되어, 파드 내부 트래픽이 `eth0 → veth pair → veth-host → 가상 브릿지 → 물리 NIC → 외부 인터넷`으로 나갈 수 있게 됩니다.

## 15. 실습 ⑤ veth — 네트워크 격리 확인
`[1:09:54]`

새 네트워크 네임스페이스가 생성된 직후의 모습을 실습으로 확인합니다(ubuntu 루트 권한).

```bash
readlink /proc/self/ns/net      # → 4026531833 (호스트 네트워크 네임스페이스)
ip addr show | grep -E "lo|ens5"
# lo(루프백) + ens5(외부 통신 메인 랜카드, 10.0.1.10) — 정상 상태
```

이제 `unshare`로 새 네트워크 네임스페이스를 만들고 그 안에서 ping을 날려 봅니다(Docker 컨테이너 생성 초기 단계와 비슷).

```bash
unshare --net bash
ping -c1 8.8.8.8                # → Network is unreachable
```

`-c1`은 한 번만 보낸다는 옵션입니다. `Network is unreachable` — 밖으로 연결된 랜카드가 아예 없어 웹으로 나가는 길조차 막혀 있습니다. 완벽하게 네트워크가 차단된 상태입니다. 쿠버네티스 파드도 만들어진 직후엔 이렇게 차단 상태이고, CNI(Calico)가 veth pair로 파드와 호스트를 연결해야 통신이 가능해집니다.

```bash
readlink /proc/self/ns/net      # → 4026532666 (호스트 ...833과 완전히 다름)
exit                            # 호스트로 복귀
readlink /proc/self/ns/net      # → 다시 ...833
```

`exit`로 빠져나오면 `Ctrl+C` 없이도 다시 호스트의 네트워크 네임스페이스(`...833`)로 돌아옵니다.

## 16. 실습 ⑥ UTS·마운트 — 수제 컨테이너
`[1:13:15]`

컨테이너 안에서 `hostname`을 확인하면 파드 이름이 나오고, 루트 디렉토리도 호스트와 완전히 다릅니다. 그게 가능한 이유가 **UTS 네임스페이스**(호스트네임 격리)와 **MNT 네임스페이스**(마운트 테이블 격리)입니다. Docker 같은 도구 없이 리눅스 커널 기능만으로 일종의 "수제 컨테이너"를 만들어 확인합니다(컨트롤 플레인, 루트).

```bash
unshare --uts --mount --fork bash
hostname                        # 아직 "control-plane" (초기값 상속)
hostname my-nginx-container     # UTS 네임스페이스 안에서만 유효
hostname                        # → my-nginx-container
exec bash                       # 프롬프트에 즉시 반영
```

`hostname` 변경이 UTS 네임스페이스 안에서만 유효합니다. 이게 `kubectl exec`로 파드에 접속했을 때 프롬프트가 파드 이름으로 바뀌는 메커니즘과 같습니다.

이제 마운트 네임스페이스 안에서 임시 메모리 디스크(**tmpfs**)를 마운트하고 비밀 파일을 만들어 봅니다.

```bash
mount -t tmpfs tmpfs /mnt       # 현재 마운트 네임스페이스 안에서만 유효
ls -al /mnt                     # 텅 빔
echo "DB_PASSWORD=secret1234" > /mnt/secret.txt
cat /mnt/secret.txt             # 격리된 방 안에서 정상 동작
exit                            # 네임스페이스 탈출 → tmpfs도 자동 정리
hostname                        # 다시 control-plane
cat /mnt/secret.txt             # → No such file (호스트엔 존재하지도 않음)
```

`exit`로 빠져나오면 커널이 `my-nginx-container` 네임스페이스와 함께 마운트된 tmpfs도 완전히 정리합니다. 호스트에서는 이 파일이 보이지도, 존재하지도 않습니다. **컨테이너 내부에서 만든 파일이 바깥 호스트로 흘러나오지 않는 것이 보안의 핵심**입니다.

## 17. 실습 ⑦ 4개 네임스페이스 동시 격리
`[1:19:50]`

PID·NET·UTS·MNT 네 개의 격리를 하나의 명령으로 만들어, Docker가 컨테이너를 생성할 때 가장 먼저 수행하는 시야 차단 과정을 봅니다. (완벽한 컨테이너가 되려면 cgroup·OverlayFS가 추가로 필요하다는 점은 기억해 두세요.)

```bash
unshare --pid --net --uts --mount --fork bash
mount -t proc proc /proc        # 새 방 전용 /proc 상태판
hostname my-nginx-container
exec bash
```

각 격리를 검증합니다.

```bash
echo $$                         # → 1 (PID 격리: 호스트 큰 번호 → 새 방 1번)
ip addr show | grep ens5 | wc -l  # → 0 (NET 격리: 호스트 ens5가 사라짐)
ps aux                          # 4줄만 (헤더·bash·ps·wc) — MNT+PID 격리
readlink /proc/self/ns/{pid,net,uts,mnt}
# 끝자리 66/67/68/69 — 커널이 4개를 순서대로 빠르게 생성한 증거
exit
hostname                        # control-plane (호스트 복귀)
```

호스트에선 100개 이상이던 프로세스가 새 네임스페이스에선 딱 4개(헤더·bash·ps·wc)만 보입니다. 마운트 네임스페이스로 격리해 파일시스템이 새로 만들어진 거죠. `exit`로 빠져나오면 호스트의 원래 번호로 복귀합니다.

## 18. 영속 네임스페이스 — ip netns
`[1:24:24]`

`unshare`는 프로세스가 종료되면 네임스페이스도 같이 사라집니다. 실무에서는 명시적으로 네임스페이스를 유지해야 할 때도 있어서, `unshare` 말고 **`ip netns`** 명령으로 영속적으로 만들 수도 있습니다.

```bash
ip netns add eth-ns             # 십진수 번호가 아닌 사람이 읽기 쉬운 이름
ip netns list                   # → eth-ns
ip netns exec eth-ns ip addr    # → lo만 있는 텅 빈 네트워크 상태
ip netns delete eth-ns          # 명시적으로 삭제(안 지우면 계속 남음)
ip netns list                   # eth-ns 사라짐
```

`unshare`와 달리 `ip netns`로 만든 네임스페이스는 수동으로 지우지 않으면 계속 남아 있으므로 `ip netns delete`로 삭제해 줘야 합니다.

## 19. 파드 샌드박스와 pause 컨테이너
`[1:26:04]`

쿠버네티스 쪽으로 시야를 옮겨, 파드가 네임스페이스들을 어떻게 조합해 멀티 컨테이너 구조를 만드는지 봅니다.

`kubectl`로 파드를 만들 때 경로의 후반부는 이렇습니다.

```text
kubelet (API 서버 Watch) → (CRI) containerd → containerd-shim → runc → 커널
```

여기서 **containerd-shim(심)**이 중간에서 컨테이너의 생명 주기를 독립적으로 관리합니다. containerd 메인 데몬이 업그레이드·재시작되어도 이미 떠 있는 컨테이너는 영향받지 않습니다(shim이 각 컨테이너 프로세스의 직접적인 부모로서 생명 주기 관리).

중요한 개념이 등장하는데 바로 **파드 샌드박스(Pod Sandbox)** 환경입니다. 커널 입장에서는 "파드"라는 것을 전혀 모릅니다. 그래서 쿠버네티스는 여러 컨테이너가 IP를 공유하고 서로 메모리 통신을 할 수 있게 묶어 주는 논리적이고 안전한 공간이 필요한데, 그것을 **파드 샌드박스**라고 합니다(CRI 스펙의 공식 아키텍처 용어). 파드 샌드박스는 개념이고, 실제로 이 공간을 물리적으로 유지해 주는 것이 바로 **pause 컨테이너**입니다.

pause 컨테이너는 흔히 **인프라 앵커(Infra Anchor)**라고도 합니다. 선박의 닻처럼 파드가 흔들리지 않게 붙잡아 주는 역할이죠. 여러분이 nginx 파드를 띄울 때, 파드 구동 시 가장 먼저 실행되는 건 여러분의 앱이 아니라 **pause 컨테이너**입니다.

```text
kubelet → containerd: RunPodSandbox 요청
containerd가 두 가지 일 수행:
  ① OCI 런타임(runc)으로 clone 호출 → NET·IPC·UTS·MNT·PID 네임스페이스 생성
     → 그 안에서 pause 실행
  ② CNI 플러그인(Calico) 호출 → 파드 네트워크 네임스페이스에 가상 인터페이스 붙이고 IP 할당
```

중요 포인트: 쿠버네티스 `1.24` 버전부터 **kubelet은 CNI를 직접 호출하지 않습니다.** kubelet은 `RunPodSandbox` 요청만 보내고, pause 생성과 CNI 호출은 전부 containerd가 담당합니다.

## 20. 실습 ⑧ 사이드카 — 같은 파드 네트워크 공유
`[1:30:31]`

파드 샌드박스와 pause가 먼저 생성된다는 사실을, 같은 파드에서 두 컨테이너를 실행해 네트워크 네임스페이스 공유로 확인합니다.

```bash
kubectl apply -f lecture/chapter03/nginx-sidecar.yaml
```

이 매니페스트는 `ns-demo`라는 파드에 컨테이너 두 개를 실행합니다 — ① nginx, ② log-collector(BusyBox 기반, 5초마다 로그 기록하는 간단한 컨테이너).

```bash
kubectl get pods                # 컨테이너 2개 정상 동작

kubectl exec ns-demo -c nginx -- hostname -i        # → 10.244.175.4
# log-collector 컨테이너 안에서 hostname
kubectl exec ns-demo -c log-collector -- hostname -i  # → 10.244.175.4 (동일!)
```

같은 파드의 두 컨테이너가 **같은 IP**를 씁니다. 네트워크 네임스페이스를 공유하고 있어, pause 컨테이너가 잡아 놓은 네트워크 네임스페이스에 nginx와 log-collector가 합류한 결과입니다.


log-collector에서 `localhost`를 쐈는데 nginx에 도달합니다. **이번 챕터 첫 번째 질문의 답**입니다 — 같은 파드가 네트워크 네임스페이스를 공유하므로 동일한 `lo` 인터페이스를 사용해 같은 localhost에 있는 거죠.

## 21. pause 컨테이너 소스 코드
`[1:33:38]`

pause 컨테이너는 CPU도 거의 안 쓰고 화면에 아무것도 출력하지 않는데 내부에서 무슨 일을 할까요? 쿠버네티스 pause 컨테이너의 소스 코드를 봅니다.

핵심은 **`SIGCHLD` 시그널 핸들러**와 **`pause()`** 함수입니다.

- **`SIGCHLD`** — 자식 프로세스가 끝났을 때 커널이 부모에게 보내는 신호. pause는 이 신호를 받으면 `waitpid`(libc)를 실행해 죽은 자식들(좀비 프로세스)을 정리(수거)합니다. 정리하지 않으면 좀비 프로세스가 쌓이겠죠.
- **`pause()`** — pause 컨테이너는 신호가 올 때까지 계속 `pause()`로 기다리기만 합니다. 시스템 콜이 올 때까지 기다리는 동안 CPU를 거의 쓰지 않습니다. 그래서 이름이 **pause**인 거죠.

리눅스는 네임스페이스 안에 살아 있는 프로세스가 하나라도 없으면 그 네임스페이스를 자동 삭제합니다. 그래서 **pause 컨테이너라는 별도 컨테이너가 존재해야 공유 네임스페이스가 유지**됩니다. pause 컨테이너는 인프라 앵커 역할을 하며, 시스템 콜 응답이 올 때까지 무한 대기하고 자원은 거의 소모하지 않습니다.

pause 컨테이너가 죽으면 공유 네임스페이스를 쓰는 프로세스가 사라지고, 커널이 네트워크 네임스페이스를 삭제하면서 파드의 IP도 삭제합니다. **그래서 파드가 죽었다 재시작되면 IP가 무조건 새로 바뀝니다**(이번 챕터 세 번째 질문과 연결).

## 22. 파드 공유 규칙 정리 + 컨테이너별 PID 격리
`[1:36:02]`

파드의 네임스페이스 공유 규칙을 정리합니다. pause 컨테이너 양측에서 아래로 연결된 초록색 화살표는 **소유 및 홀딩(Owning & Holding)** — pause 컨테이너의 핵심 임무로, 파드 수명 주기 동안 공유 영역을 생성하고 그 상태를 안정적으로 유지하는 것입니다.

**공유 영역(pause가 소유·유지)** — 세 가지:
1. **NET 네임스페이스** — 예: `10.244.1.5` 같은 단일 IP를 할당받아 유지. 파드의 모든 컨테이너가 localhost로 직접 통신할 기반 제공.
2. **IPC 네임스페이스** — 컨테이너 간 효율적 통신을 위한 공유 메모리.
3. **UTS 네임스페이스** — 파드의 모든 프로세스가 동일한 호스트네임(예: `ns-demo`) 식별.

애플리케이션 컨테이너들(nginx·log-collector)은 자신만의 네트워크·호스트네임 공간을 새로 만들지 않고, pause가 이미 확보한 공유 영역의 네임스페이스에 **합류(Join)**해 리소스를 함께 씁니다.

**격리 영역(컨테이너별 고유)** — 컨테이너의 본질은 격리성이므로 반드시 격리되어야 합니다.
- **PID 네임스페이스** — nginx 컨테이너는 PID 1번이 nginx 마스터, log-collector는 자신의 1번을 가짐(각자 독립).
- **MNT 네임스페이스** — 각 컨테이너가 자신만의 격리된 루트 파일시스템.

전체 요약: NET·IPC·UTS 같은 **인프라 레벨** 네임스페이스는 pause 컨테이너를 통해 **공유**하고, PID·MNT처럼 **애플리케이션 실행 환경**에 직결되는 네임스페이스는 철저히 **컨테이너별로 격리**됩니다.

실제로 컨테이너별 PID 격리를 확인합니다.

```bash
# log-collector의 PID 1번 커맨드라인
kubectl exec ns-demo -c log-collector -- cat /proc/1/cmdline   # → 셸 스크립트
# nginx의 PID 1번 커맨드라인
kubectl exec ns-demo -c nginx -- cat /proc/1/cmdline           # → nginx master
```

동일한 파드 안 두 컨테이너가 **서로 다른 PID 1번**을 가집니다. 커널 레벨에서 두 컨테이너의 PID 네임스페이스가 완전히 격리되어 있다는 증거이고, 따라서 서로의 프로세스 목록을 절대 볼 수 없습니다.

## 23. 실습 ⑨ 파드 간 통신 + 마무리
`[1:41:06]`

이번엔 서로 다른 파드끼리 어떻게 상호작용하는지 봅니다. MySQL 파드를 배포합니다.

```bash
kubectl apply -f lecture/chapter03/mysql.yaml
kubectl get pods                # ns-demo + mysql 두 파드 공존
```

**다른 파드의 프로세스는 안 보임** (PID 네임스페이스 격리):

```bash
kubectl exec ns-demo -c log-collector -- ps aux | grep mysql   # 출력 없음
```

**다른 파드는 다른 IP** (NET 네임스페이스 격리):

```bash
kubectl get pods -o wide
# ns-demo  → 10.244.175.4
# mysql    → 10.244.175.8
```

**격리돼 있어도 통신은 가능** (Calico CNI의 가상 라우팅):

```bash
# L3 통신 (ICMP ping)
kubectl exec ns-demo -c log-collector -- ping -c3 10.244.175.8
# → 3 packets transmitted, 3 received, 0% loss

# L4 통신 (TCP 3306)
kubectl exec ns-demo -c log-collector -- nc -zv 10.244.175.8 3306
# → open (TCP 연결 성공)
```

`nc -z`는 데이터를 보내지 않고 연결만 하겠다, `-v`는 자세한 출력입니다. ping(L3)도 TCP 연결(L4)도 성공합니다. 네트워크 네임스페이스는 격리돼 있지만, Calico가 뚫어 놓은 가상 라우팅을 통해 통신이 이루어지는 겁니다. 파드끼리 프로세스는 안 보이지만 네트워크(CNI 경유)로는 통신이 가능한 상태입니다.

**마무리** — 실습이 끝나면 반드시 환경을 삭제합니다.

```bash
terraform destroy -auto-approve
terraform state list            # 모든 리소스가 삭제됐는지 최종 확인
```

리소스를 삭제하지 않으면 방치된 리소스만큼 AWS 비용이 청구되므로, 항상 랩 환경이 끝나면 정리해 주시면 되겠습니다.
