## 1. 인프라의 변천사와 궁극의 목적
`[00:20]`

안녕하세요, 플레이빌더 카이입니다. 우리가 쿠버네티스를 본격적으로 배우기 전에 가장 근본적인 질문부터 던져 봐야 합니다. 도대체 인프라가 어떻게 변해 왔길래 쿠버네티스라는 게 필요해졌을까 하는 점이죠.

인프라의 변천사를 아주 짧게 짚어 보겠습니다.

- **물리 서버(Physical Server)**
- **가상 머신(VM, Virtual Machine)**
- **컨테이너(Container)**

보시다시피 형태는 완전히 달라졌습니다. 하지만 여기서 놓치지 말아야 할 게 있어요. 기술은 변했지만 **인프라를 구축하는 목적은 단 하나도 바뀌지 않았다**는 점입니다.

모든 인프라의 궁극적 목적은 바로 **안정적인 서비스 제공**입니다. 결국 핵심은 서비스가 죽지 않게 하는 것, 딱 이겁니다. 그래서 이 강의 전체가 "어떻게 하면 서비스를 죽지 않게 관리할 것인가"에 대한 이야기라고 보시면 되겠습니다.

## 2. 왜 쿠버네티스인가 — MSA와 인프라 현실의 변화
`[02:04]`

그러면 왜 쿠버네티스(K8s)인가? 쿠버네티스는 **K8s**라고도 불립니다.

예전에는 서버 몇 대만 관리하면 됐으니까 사람이 직접 배포하고 모니터링을 해도 충분히 운영할 수 있었습니다. 그런데 지금은 왜 안 될까요? 바로 인프라의 현실이 바뀌었기 때문입니다.

인프라의 현실이 왜 바뀌었냐? 바로 **MSA(마이크로서비스 아키텍처, Microservice Architecture)**가 도입됐기 때문이죠. MSA라는 건 하나의 큰 서비스가 수백, 수천 개의 작은 컨테이너로 쪼개져서 서로 복잡하게 얽혀 돌아가는 상황을 말합니다.

그럼 이 수천 개의 컨테이너를 사람이 일일이 배포하고 모니터링하고, 장애가 날 때마다 직접 들어가서 복구한다 — 어떨 거 같나요? 이건 사실상 물리적으로 불가능한 영역입니다. 그래서 나온 솔루션이 바로 우리가 학습할 쿠버네티스인 거죠.

쿠버네티스는 이 복잡한 관리 과정을 자동화해 줍니다. 그 핵심 가치 두 가지를 기억해 보겠습니다.

1. **고가용성(High Availability)**
2. **자가치유(Self-healing)**

## 3. 핵심 가치 ① 고가용성(High Availability)과 SPOF
`[03:23]`

첫 번째는 **고가용성(High Availability)**입니다. 고가용성이란 **서버 일부가 죽어도 전체 서비스는 계속 살아 있게 만드는 힘**이라고 기억하시면 되겠습니다.

쉽게 말해 고가용성은 어느 한 곳이 고장이 나도 전체 서비스는 안 죽는 구조를 만드는 겁니다. 현업에서는 장애가 났을 때 시스템 전체를 무너뜨리는 그 딱 하나의 지점을 **SPOF(단일 장애점, Single Point of Failure)**라고 합니다.

- **SPOF가 있는 구조**: 서버 한 대만 존재하고 그 서버 한 대가 죽으면 전체가 다운되는 상황. 즉 서비스 전체가 다운됩니다.
- **SPOF가 없는 구조**: 서버가 한 대가 아니라 여러 대(서버 A, B, C) 있는 상황. 서버 A 또는 B 또는 C가 한 대 죽더라도 나머지 두 대가 요청을 처리하니 사용자는 장애를 느끼지 못합니다. 즉 단일 장애점이 제거된 구조죠.

쿠버네티스는 이 단일 장애점 자체를 **설계 단계에서부터 없애 버립니다.**

## 4. 핵심 가치 ② 자가치유(Self-healing) — 선언·감지·조치
`[05:53]`

그럼 돌아와서 **자가치유(Self-healing)**. 어떻게 사람 없이 시스템이 알아서 복구를 하는 걸까요? 원리는 의외로 간단한데요, 쿠버네티스는 **상태(State)**를 관리합니다. 상태는 세 가지로 이루어져 있어요.

1. **선언**: 사용자, 즉 엔지니어가 "나 이 파드는 항상 세 개를 유지해 줘"라고 선언합니다.
2. **감지**: 갑자기 파드 한 개 또는 두 개가 죽더라도 항상 선언된 상태를 유지할 수 있도록 감지를 합니다.
3. **조치**: 쿠버네티스의 컨트롤러(Controller)가 기대하는 상태와 현재 상태의 차이점을 발견하고, 즉시 파드 하나를 더 생성해서 다시 세 개로 맞춥니다.

이게 운영 관점에서는 얼마나 혁명적이냐면, 예전에는 서버가 새벽에 죽으면 엔지니어나 개발자가 그 알람 소리에 깨서 즉시 컴퓨터 앞으로 가서 수동으로 대응했어야 했습니다. 하지만 쿠버네티스를 쓰면 알람이 울려도 시스템이 알아서 먼저 복구를 하는 거죠. 어떻게? 바로 이 자가치유 능력 때문에.

그럼 운영자 입장에서는 다음 날 출근해서 어젯밤에 왜 파드가 다운됐는지, 어떤 문제가 있었는지 **원인 분석만 하면 되는** 거죠. 즉 운영의 패러다임 자체가 바뀌었다는 걸 알 수 있습니다.

## 5. 연쇄 장애(Cascade Failure)와 방어장치
`[07:45]`

그런데 실제 운영 환경은 생각보다 훨씬 가혹합니다. 장애는 항상 예기치 못한 순간에, 가장 아픈 방식으로 찾아옵니다.

상황을 한번 가정해 볼게요. 새벽 3시에 **PagerDuty** 같은 장애 알림 서비스에서 알림이 발생했다. 그런데 그 알림이 `503 에러율 급증`, 그리고 `Service Unavailable` 장애가 계속 발생한다면 어떨까요? 화면에 `503` 에러가 뜬다는 건 단순히 컨테이너 한두 개가 죽은 수준이 아닙니다. **서비스 자체가 완전히 마비됐다**는 뜻이죠.

클러스터 지표를 확인했을 때 평소와는 완전히 다른, 기괴한 패턴으로 장애가 번지는 걸 목격합니다. 이걸 우리는 **연쇄 장애(Cascade Failure)**라고 합니다.

연쇄 장애가 발생한 상황을 살펴보면:

1. **트래픽 급증**: 어떤 이유로 서비스의 트래픽이 갑자기 스파이크쳐 올라왔어요.
2. 파드가 세 개 있던 상황에서 **파드 1번이 메모리 초과(OOM)로 강제 종료**됩니다.
3. 서비스는 계속 운영돼야 하니 파드 1번이 감당하던 요청이 **파드 2번과 3번으로 쏟아집니다.**
4. 파드 2번과 3번도 결국 **메모리 임계점에 도달**합니다. 안 그래도 아슬아슬하게 버티던 두 파드도 갑자기 늘어난 부하를 못 견디고 죽게 되는 거죠.
5. 결국 모든 파드가 줄줄이 죽으면서 **서비스 전체가 다운**됩니다.

마치 도미노 현상 같죠. 파드 하나가 죽으면서 옆에 있는 파드까지 같이 쓰러뜨리는 겁니다.

이런 현상을 막으려면 반드시 **방어장치(Defense Mechanism)**가 필요합니다. 여러 가지가 있을 수 있는데:

- **레이트 리미팅(Rate Limiting)**: 들어오는 요청의 속도를 강제로 제한해서 시스템이 현재 감당할 수 있는 만큼만 요청을 수용하게 만드는 것.
- **서킷 브레이커(Circuit Breaker)**: 주식 시장에도 있는 시스템이죠. 특정 지점에 장애가 감지되면 연결을 즉시 차단해서 다른 곳으로 장애가 번지지 않게 격리합니다.

그럼 위 2번에서 발생한 실제 장애 원인이 무엇인지 로그로 확인해 봤다고 가정해 볼게요. 메모리가 과부하로 죽은 상황이죠. 그래서 **OOM(Out Of Memory)**이 발생하고, **exit code는 137**이라고 뜰 겁니다. OOM이라는 건 메모리 초과 또는 메모리 부족을 의미합니다. 시스템이 메모리를 너무 많이 쓰는 프로세스를 강제로 종료시킨 거죠. 이 OOM Killed가 어떤 경로로 발생하고 또 어떻게 근본적으로 해결해야 하는지는 실습을 통해 재현해 보겠습니다.

## 6. 실습 환경 구성 — Terraform으로 kubeadm 클러스터
`[12:39]`

실습 환경을 진행하겠습니다. 이번 실습에서는 **Terraform**을 사용해서 쿠버네티스 클러스터를 생성하겠습니다. 인프라를 코드로 정의하고 자동으로 생성해 주는 IaC 도구인 Terraform을 통해서 서버를 하나씩 수동으로 만드는 게 아니라 코드로 전체 환경을 한 번에 구성합니다. Terraform을 한 번도 사용하지 않으신 분들도 가이드에 따라 Terraform만 설치하면 쉽게 따라오실 수 있습니다.

먼저 Terraform을 설치하지 않으신 분들은 구글에서 `terraform install`로 검색해서 HashiCorp 사이트에 들어와, 사용하는 OS 환경에 맞게 설치해 주시면 됩니다. 설치가 끝나면 터미널에서 버전을 확인합니다.

```bash
terraform version   # 버전이 나오면 설치 성공
```

다운로드할 레포지토리는 **kubeadm 클러스터**입니다. 링크에 따라 `git clone`으로 다운로드 받으세요.

```bash
git clone <kubeadm-cluster-repo>
cd kubeadm-cluster
```

Terraform에 관한 수업은 아니기 때문에 Terraform 코드 자체는 설명하지 않겠습니다. 이 Terraform을 apply 했을 때의 아키텍처를 보면:

- **컨트롤 플레인(Control Plane)** 1대 — 인스턴스 타입 `t3.medium`
- **워커 노드(Worker Node)** 2대 — 실습 환경이라 가장 작은 `t3.small` 사용

터미널로 돌아와서, README에 적혀 있듯이 `terraform.tfvars` 파일을 복사해서 여러분의 IP 주소를 변경해 주면 됩니다. 클러스터 네임이나 AWS 리전을 바꾸고 싶으면 바꾸고, 디폴트로 쓸 분들은 그대로 두세요. 중요한 건 `my_ip` 부분입니다.

```bash
# 내 IP 확인
curl -s ifconfig.me      # 또는 구글에 "my ip address" 검색
```

전체 환경으로 실습하려면 모든 포트를 열어 준 상태로 파일을 수정·저장하고 빠져나옵니다. 이제 Terraform 초기화부터 진행합니다.

```bash
terraform init                  # 프로바이더·플러그인 자동 다운로드
terraform plan                  # 어떤 리소스가 만들어질지 사전 확인(실제 생성 X)
terraform apply -auto-approve   # 실제 리소스(인프라) 생성
```

> ⚠️ 이 실습은 실제 AWS 리소스를 사용하므로 어느 정도 **비용이 듭니다.** 컨트롤 플레인과 워커 노드 2대를 쓰지만 실습 한 시간 기준으로는 큰 비용은 아닙니다. 리소스 생성은 **1분 이내**에 끝납니다.

만들어지면 컨트롤 플레인 1대, 워커 노드 2대 인스턴스가 생성됩니다. 가이드(README)에 따라 워커 노드를 컨트롤 플레인에 조인시키는 작업을 합니다.

접속은 SSH 키가 아니라 **SSM 세션** 방식으로 합니다. 세션 주소를 복사해서 바로 접속하면 됩니다. 오른쪽 터미널에서 컨트롤 플레인부터 접속하겠습니다.

접속하면 root 환경이나 `ubuntu` 계정으로 접속하게 됩니다. `ubuntu` 계정으로 접속하면 `id`로 확인한 뒤, README대로 `tail -f`로 kubeadm을 부트스트랩하는 로그가 어떻게 진행 중인지 확인합니다. 지금 **Tigera Operator(Calico Operator)**를 설치하고 있고, 설치가 완료되면 Calico Operator 상태가 잘 설치됐는지 확인합니다.

`Bootstrap Complete`라고 뜨면 성공적으로 부트스트랩 준비가 된 것입니다. `Ctrl+C`로 빠져나와서 확인합니다.

```bash
kubectl get tigerastatus
kubectl get pods -n calico-system
```

Calico의 컨테이너가 `ContainerCreating`이다가, Calico API 서버와 컨트롤러가 모두 설치 완료되면 `Running`으로 바뀝니다.

완료되면 홈 폴더에서 `ls`를 치면 `join-command.sh` 파일이 하나 만들어집니다. 이 파일을 워커 노드에 접속해서 실행하는 방식으로 kubeadm을 조인합니다. 아래쪽 터미널을 split 해서 워커 노드 두 대에 접속하겠습니다. 접속 방법은 Terraform 설치 후 출력으로 나와 있어요.

```bash
terraform output   # 워커노드 1·2번, 컨트롤 플레인 접속 방법 확인
```

각각 복사·붙여넣기로 워커 노드 1번과 2번에 접속됩니다. `ubuntu` 계정으로 들어가서 확인하면 `worker1`, `worker2`로 나옵니다. 워커 노드 1·2번에서 컨트롤 플레인의 join 커맨드 셸 스크립트 내용을 복사해서 입력하는데, **sudo**로 명령을 실행해야 합니다.

```bash
sudo kubeadm join ...   # 컨트롤 플레인의 join-command.sh 내용
```

컨트롤 플레인에서 노드를 확인하면 아직 `NotReady` 상태인데, 1분 안쪽으로 워커 노드 1·2번이 컨트롤 플레인과 통신할 수 있는 상태가 됩니다.

```bash
kubectl get nodes   # 워커노드 1·2번 모두 Ready 확인
```

총 노드를 확인하면 컨트롤 플레인 1대 + 워커 2대입니다. 실습할 때는 왼쪽에 컨트롤 플레인, 오른쪽에 워커 노드 1·2번을 사용하겠습니다. 앞으로 시스템은 컨트롤 플레인과 워커 노드, 이 두 환경에서 실습을 진행합니다.

컨트롤 플레인에서 `kubectl` 명령어를 다 쳐도 되지만, 매번 치기 귀찮으니 `k`라는 alias를 쓰면 똑같습니다.

```bash
kubectl get nodes   # == k get no
```

## 7. 실습 ① OOM Killed 재현
`[22:45]`

이번 실습에서 **OOM Killed** 문제를 재현해서 분석을 진행한다고 말씀드렸습니다. 장애 상황을 유도할 파드의 매니페스트를 하나 만들어야 하는데, 매니페스트는 `kubeadm-cluster` 레포지토리를 클론해서 파드를 배포할 수 있습니다. 컨트롤 플레인에서 레포지토리 주소를 복사해 옵니다.

`lecture`(랩처) 폴더가 있어요. 이 폴더로 들어가 보면 두 개의 아주 간단한 파드 매니페스트가 있습니다. 그전에 OOM 파드를 확인해 보면 리소스를 얼마만큼 사용하겠다고 정의돼 있고, 중요한 건 **스트레스 테스트**를 주고 있어요. 리소스보다 더 큰 리소스를 지속적으로 사용해서 에러를 출력하게 만드는 파드입니다.

`memory.limit`에 보면 **128 MiB**로 제한을 주고 있는데, 이 뜻은 쿠버네티스에게 "이 파드가 사용할 수 있는 메모리의 물리적인 최대 상한선"을 제한하는 거예요. 그런데 스트레스로 **200MB**를 주는 거죠. 그러면 상한선보다 더 높은 메모리 사용을 하기 때문에, 시작과 동시에 메모리를 초과해서 점유하는 상황이 만들어집니다.

```bash
kubectl apply -f oom-pod.yaml
kubectl get pods
```

확인해 보면 OOM 파드가 `ContainerCreating`하다가 **`OOMKilled`** 상태로 넘어간 거죠. 제한된 128MB 공간에 그보다 큰 200MB 데이터를 강제로 밀어넣으면서 물리적 한계에 부딪히는 상황입니다. 정상 실행이 불가능해져서 파드 상태가 **`CrashLoopBackOff`**와 OOM 상태가 됩니다.

운영 상황에서 이런 상황을 마주치면, 단순히 메모리 사용량을 더 늘리는 것보다 **왜 이런 문제가 발생했는지 원인을 먼저 찾는 게** 중요합니다. `CrashLoopBackOff`라는 에러 메시지가 무엇인지 살펴보겠습니다.

**CrashLoopBackOff** — 이 에러 메시지가 뜨면:

1. 프로세스가 메모리 초과로 **파드가 죽은** 상태를 의미합니다.
2. kubelet은 이 파드가 어떤 이유로 종료됐는지 모르기 때문에 **다시 재시작 시도**를 합니다. 파드의 `restartPolicy`가 기본값 `Always`로 설정돼 있기 때문에 관리 주체인 kubelet이 즉시 재시작을 시도합니다.
3. 그런데 현재 파드의 문제는 메모리 한도보다 훨씬 높은 한도를 사용하기 때문에 **다시 파드가 죽는** 상태가 만들어지죠.
4. 그러면 kubelet이 다시 파드를 생성해서 살리려고 노력합니다. 이 작업을 **백오프(Backoff)**라고 하는데, 재시작 간격을 **10초, 20초, 40초**처럼 두 배씩 지수함수적으로 늘리다가 **최대 5분**이 됩니다.
5. 재시작 대기 시간이 무한정 길어지므로, 이 상태에서는 더 이상 파드가 온전히 살아서 동작하지 않겠다고 판단해서 루프에 빠진 **CrashLoopBackOff** 상태가 만들어집니다.

구체적인 종료 사유는 당연히 디버깅을 먼저 해야겠죠. 디버깅은 쿠버네티스의 `describe` 명령어로 확인합니다.

```bash
kubectl describe pod <pod-name>
```

그러면 `Back-off restarting failed container`로 컨테이너가 이 문제로 멈췄구나를 알 수 있고, 조금 더 자세히 보면 `Last State: Terminated`, `Reason: OOMKilled`, **`Exit Code: 137`**이라고 뜨는 것을 확인할 수 있습니다.

## 8. exit code 137의 의미 — 죽인 주체는 리눅스 커널
`[28:55]`

**exit code 137**은 그냥 단순한 숫자가 아니라 의미가 있습니다.

- 종료 코드가 **128 이상**이라는 것은, 프로세스가 스스로 종료된 것이 아니라 **어떠한 시그널(Signal)에 의해 강제로 종료**되었음을 뜻합니다.
- `137 = 128 + 9`이고, 리눅스 시스템에서 **9번 시그널은 `SIGKILL`**입니다. `SIGKILL`은 프로세스를 즉시, 강제로 죽이라는 명령입니다.
- 프로세스는 이 시그널을 수신받으면 **어떠한 정리 작업도 하지 못하고** 즉각적으로 종료됩니다.

그렇다면 이 `SIGKILL` 시그널을 보낸 주체가 누구냐? 이게 가장 중요한데, 쿠버네티스냐? K8s의 어떤 리소스냐? 그렇지 않습니다. 실제 시그널을 날린 주체는 바로 **리눅스 커널(Linux Kernel)**입니다. 가장 중요한 부분입니다.

많은 분들이 쿠버네티스가 파드를 죽인다고 오해하지만, 쿠버네티스는 단지 자신이 보는 선에서 한도값을 넘어가면 **커널에게 전달하는 역할만** 수행합니다. 즉 실제 메모리 제어가 일어나는 경로를 다시 살펴보면:

```text
메모리 리밋 발생
  → kubelet
  → containerd
  → runc
  → 리눅스 커널의 cgroup(컨트롤 그룹)에 기록
```

1. **메모리 리밋 발생**: 스트레스 테스트로 한도를 초과한 메모리를 사용하겠다고 부여했기 때문에 메모리 리밋이 발생합니다.
2. 쿠버네티스의 **kubelet**이 **containerd**에게 요청하고, 그다음 실제 실행 프로그램(로우 레벨 런타임)인 **runc**를 거쳐 이 정보가 전달됩니다.
3. runc는 리눅스 커널의 **컨트롤 그룹(cgroup)**에 기록합니다. 최종적으로 커널의 자원 관리 시스템인 cgroup 파일시스템에 "메모리가 초과되었다"는 수치가 기록됩니다.
4. 그러면 커널이 **OOM Killer**를 발동합니다. 커널은 메모리 사용량을 실시간으로 감시하다가, cgroup에 기록된 `memory.max` 값을 초과하는 순간 OOM Killer가 동작해서 해당 프로세스에 위에서 봤던 **`SIGKILL`(9번)** 작업을 합니다.

> kubelet은 지금 몰라도 상관없습니다. 이후에 쿠버네티스의 모든 리소스와 컴포넌트를 하나하나 살펴볼 예정이라, 지금은 "아 이런 게 있구나" 정도로 이해하시면 됩니다.

쿠버네티스는 커널이 보고한 결과를 사용자에게 **"당신의 파드가 OOMKilled 상태로 죽었다"는 텍스트만 보여 주는** 겁니다. 이제 실제 워커 노드의 커널 로그로 어떤 과정을 통해 진행됐는지 살펴보겠습니다.

## 9. 커널 로그로 OOM 추적 — cgroup OOM vs 시스템 OOM
`[33:18]`

다시 터미널로 돌아가서, 파드가 어떤 노드에서 실행 중인지 확인합니다.

```bash
kubectl get pods -o wide   # CrashLoopBackOff 상태 + 워커노드 2번에서 실행 중
```

파드가 워커 노드 2번에서 실행 중인 것을 알 수 있죠. 워커 노드 2번으로 들어가서 커널 메시지를 확인하겠습니다. 커널 작업은 root로 확인하는 게 편리합니다(권한이 필요한 작업이 많기 때문).

```bash
sudo -i              # root 환경 진입
dmesg -T             # 커널 메시지 확인
dmesg -T | grep -i oom | tail -5
```

`dmesg -T`로 메시지를 확인하면 워커 노드 2번에서 **`Memory cgroup out of memory`** 에러 메시지가 출력되는 것을 확인할 수 있습니다. 조금 더 구체적으로 grep으로 보면, `Killed process 6386`처럼 종료된 프로세스 번호를 확인할 수 있죠. 커널 로그의 상세 분석을 통해 정확하게 원인을 파악할 수 있는 거죠.

정리하면, `dmesg` 로그라는 상세 기록을 통해 어떤 프로세스가 종료됐는지 확인할 수 있었습니다. cgroup이 할당된 메모리 한도를 넘겼음을 선언해서 프로세스가 종료되는 상황인 거죠. 여기서는 `6374`, `6386` 같은 프로세스가 있고, 커널이 이 번호를 가지고 스트레스 프로세스를 강제로 종료시켰음을 확인할 수 있습니다.

그리고 아래쪽을 보면 **`oom_score 967`** 같은 숫자가 있는데 이 숫자도 의미가 있습니다. 커널이 프로세스를 종료시킬 때 사용하는 **종료 우선순위 점수**입니다. cgroup(컨트롤 그룹) 챕터에서 자세히 살펴보겠지만, 이 종료 우선순위 점수에 따라 커널이 어떤 프로세스를 종료시킬 것인지 결정합니다.

노드의 메모리가 부족해지면 커널은 어떤 파드를 먼저 희생시킬지 결정합니다. 쿠버네티스는 파드의 **QoS Class(Quality of Service)** 점수에 따라, 방금 본 `oom_score_adj`를 통해 점수를 보고 차등 부여합니다. QoS Class는 리눅스 커널의 별도 챕터에서 자세히 살펴보겠습니다.

정리하면, **OOM은 크게 두 가지 경로**로 발생합니다.

1. **cgroup OOM (현재 상황)**: 컨트롤 그룹 단위로 한도(리밋)를 초과해서 결국 파드가 종료된 경우. **해당 파드만 종료**되므로 시스템 전체에는 영향을 주지 않습니다(자신의 한도를 넘긴 경우).
2. **시스템 OOM (System OOM)**: 노드 전체에 메모리가 고갈된 경우. 이건 훨씬 더 큰 문제로, **노드 전체에 영향**을 줍니다. 이때 리눅스 커널은 앞서 본 `oom_score` 점수를 보고 (`1,000점`부터 `-1,000점`까지 범위에서) **가장 높은 점수를 가진 파드부터 순서대로 제거**하며 시스템을 보호합니다.

그래서 현업에서 OOM Killed 알람이 울렸을 때, 단순히 쿠버네티스의 리밋 수치만 올린다고 해서 해결되지 않는다는 점을 꼭 기억하세요. 수치를 올리는 게 아니라 **먼저 원인을 파악**해서 왜 메모리 부족 현상이 일어났는지 확인하는 작업이 반드시 필요합니다. 단순히 수치만 올리는 건 **임시방편**에 불과합니다.

근본적인 해결을 위해서는 당연히 원인 파악을 먼저 해야 합니다. 특정 요청에서만 메모리가 튀는지, GC(가비지 컬렉션)가 원활하게 작동하지 않는지, 비효율적인 알고리즘을 사용하는지 — 이런 원인을 먼저 파악해서 제거하는 것이 진정한 해결책입니다. 수치를 조정하기 전에 **로그를 통해 왜 프로세스가 죽었는지 파악**하는 것이 아주 중요합니다.

## 10. 실습 ② CPU 스로틀링(Throttling)과 CFS
`[40:54]`

이번에는 이전 사례와는 성격이 완전히 다른 유형을 살펴보겠습니다. 실무를 하다 보면 노드 전체에 CPU 자원은 충분해 보이는데도 **특정 서비스의 응답 속도만 비정상적으로 느려지는** 상황을 종종 겪습니다. 더 까다로운 점은 **애플리케이션 로그에는 아무런 에러도 남지 않을 때**입니다. 이런 유형의 대표적인 원인이 바로 **CPU 스로틀링(Throttling)**입니다.

앞서 살펴본 메모리 부족 문제는 파드가 종료되거나 재시작되기 때문에 단서가 남아 있습니다. 하지만 CPU 스로틀링은 **파드가 죽지도 않고 에러 로그도 특별히 없습니다.** 단지 계속해서 서비스 속도만 느려질 뿐이죠. 엔지니어 입장에서는 "문제는 분명히 있는데 겉으로 드러나는 신호가 거의 없는" 상태를 마주하게 됩니다. 그래서 실무에서 까다로운 케이스 중 하나입니다.

배포 이전에 이 파드가 어떤 설정으로 구성됐는지 터미널에서 YAML 파일을 출력해 봅니다.

```bash
cat throttling-pod.yaml   # kubeadm-cluster/lecture/chapter1/
```

보시면 `args`에 `--cpu 1`이라고 적혀 있고 `limits.cpu`는 `50m`이라고 적혀 있습니다. 이게 무슨 뜻인지 살펴보면:

- **`args: --cpu 1`** → 스트레스 도구에게 **CPU 1코어를 전부 사용하라**고 지시. 프로세스 입장에서는 **1000m코어**, 즉 코어 하나를 온전히 점유하려고 시도합니다.
- **`limits.cpu: 50m`** → 쿠버네티스가 이 컨테이너에게 허용한 CPU 상한선이 **50m코어**, 즉 1코어의 단지 **5%**입니다.

즉 프로세스는 1000m코어를 쓰고 싶은데 허용된 수치는 오직 50m코어뿐입니다. 이 간극이 어떤 결과를 만들어 내는지, 그 메커니즘을 정확히 이해하려면 리눅스 커널의 **스케줄러**를 봐야 합니다.

쿠버네티스의 밀리코어 단위는 결국 리눅스 커널의 **CFS(Completely Fair Scheduler)**가 담당합니다. 이름 그대로 "fair = 공정한" 집행을 합니다. CFS는 쉽게 말해 리눅스 커널의 CPU 스케줄러입니다.

쿠버네티스가 "50m코어를 사용하라"고 선언하면, **kubelet**이 그 값을 **cgroup(컨트롤 그룹)의 CPU 설정 파일**에 기록합니다. 그럼 커널이 이 값을 기반으로 해당 컨테이너의 CPU 사용량을 제한합니다.

이때 중요한 점은, **CFS 자체**는 공정한 스케줄링을 담당할 뿐, 특정 워크로드의 CPU 사용량을 직접 제한하지는 않습니다. 이 제한을 담당하는 것은 바로 **CFS Bandwidth Controller(밴드위드 컨트롤러)**입니다.

CFS Bandwidth Controller는 cgroup 단위로 CPU 사용량의 상한선을 설정하고, 일정 주기 동안 사용할 수 있는 CPU 시간을 **쿼터(Quota, 할당량)**로 제한합니다.

- **기본 주기(Period)**: `100ms`(= 0.1초). 1초의 1/10에 해당.
- 이 주기 동안 사용할 수 있는 CPU 시간을 쿼터로 정의.
- 쿠버네티스가 `limits.cpu: 50m`으로 설정했으므로, `100ms` 중 단 **5%**, 즉 **5ms**만 사용할 수 있습니다.

```text
하나의 주기 = 100ms
├── 5%  (5ms)  → 쿼터: 사용할 수 있는 할당량
└── 95%        → 강제 대기(Throttled) 상태
```

나머지 95%는 **강제 대기 상태**입니다. 아무리 연산이 필요해도 무작정 기다려야 하는 강제 대기 상태이고, 이걸 영어로 **Throttled(스로틀)되었다**고 합니다.

매니페스트의 스트레스 프로세스는 1코어 전체를 쓰려 하지만, **5ms 만에** 자신이 사용할 수 있는 할당량이 바닥나 버립니다. 그러면 나머지 95% 시간 동안은 무작정 기다려야 하는 스로틀링 상황인 거죠.

이제 파드를 배포해 봅니다.

```bash
kubectl apply -f throttling-pod.yaml   # lecture/chapter1
kubectl get pods                        # ContainerCreating → Running
kubectl top pod                         # 파드 상태(메트릭) 확인
```

확인해 보면 CPU 사용량이 **`50m`(밀리코어)**라는 걸 알 수 있죠. 정상적으로 작동하는 것처럼 **보입니다.** 이 상황에서 "인프라는 이상이 없으니 애플리케이션 로직을 확인해 달라"고 무작정 개발팀에게 원인을 넘기면 잘못된 방식을 취하게 됩니다.

`limits.cpu`가 `50m`이면, 내부에서 응답이 아무리 밀려 있어도 겉으로는 `50m코어`까지만 표시됩니다. **메트릭 서버**(`kubectl top`)는 정확한 실값이 아니라 **평균값**을 보여 줍니다.

비유하면, "천장에 머리가 이미 닿은 상태인데, `kubectl top`은 천장의 높이만 보여 줄 뿐, 얼마나 많은 횟수로 머리가 계속 천장에 부딪히는지는 보여주지 않는다"는 거죠. 실제로는 CFS 스케줄러가 한 주기(`100ms`) 동안 쿼터 할당량을 다 쓰면, 다음 주기가 시작될 때까지 실행을 막습니다. 그래서 내부에서는 계속 **스로틀링 현상**이 일어납니다.

정확한 상태를 확인하려면 **커널 레벨의 데이터**를 직접 들여다 봐야 합니다.

## 11. 커널 레벨 확인 — cpu.max·cpu.stat 통계 분석
`[49:41]`

스로틀 파드가 어떤 노드에서 실행 중인지 확인합니다.

```bash
kubectl get pods -o wide   # 워커노드 2번에서 실행 중
```

워커 노드 2번으로 이동해서 해당 파드 컨테이너를 직접 들여다 봐야 합니다. 컨테이너 관련 내부 커널 상태를 확인할 것이기 때문에 root로 접근합니다.

```bash
sudo -i
crictl ps                                # 컨테이너 상태 확인 → stress 파드 확인
crictl ps --name stress -o wide | head    # 컨테이너 ID 추출
CONTAINER=<container-id>                  # 변수에 저장
echo $CONTAINER
```

이때 **워커 노드 입장에서 보는 컨테이너의 프로세스 ID(PID)와, 컨테이너 내부에서 보는 PID는 서로 다릅니다.**

```bash
kubectl exec -it <throttling-pod> -- sh
ps -ef    # 컨테이너 내부: stress(--cpu) 프로세스의 PID가 1번
```

컨테이너 내부에서는 이 stress 프로세스의 PID가 **1번**으로 표시됩니다. 이게 가능한 이유는 **PID 네임스페이스(Namespace)** 때문에 격리 상태가 되어 있기 때문입니다. 반면에 호스트 커널이 바라보는 값은 다릅니다.

```bash
crictl inspect $CONTAINER | jq '.info.pid'   # 호스트에서의 실제 PID
```

확인해 보면 워커 노드에서의 stress 프로세스 PID가 **`13916`** — 즉 서로 다른 번호입니다. 호스트 기준 PID를 변수에 저장합니다.

```bash
PID=13916
echo $PID    # 13916
```

이 PID로 cgroup(컨트롤 그룹) 경로를 찾아야 합니다. `/proc`이라는 가상 파일시스템에서 해당 PID의 cgroup을 확인합니다.

```bash
cat /proc/$PID/cgroup
cat /proc/$PID/cgroup | cut -d: -f3   # 앞부분 제외하고 경로만 추출
```

경로를 보면 `kubepods/burstable/...`로 보입니다. **Burstable**이라는 건 QoS Class의 한 종류로, `requests`와 `limits`가 다르게 설정돼 있기 때문에 이 파드는 현재 **Burstable QoS Class**에 속합니다(별도 챕터에서 다룰 예정). 이 경로도 변수에 저장합니다.

```bash
CGROUP_PATH=<cgroup-path>
echo $CGROUP_PATH
```

이 파드에 설정한 CPU 제한이 커널에 어떻게 전달되는지 확인합니다.

```bash
cat /sys/fs/cgroup/$CGROUP_PATH/cpu.max   # 출력: "5000 100000"
```

`cpu.max`를 확인하면 **`5000`**과 **`100000`** 두 숫자가 보입니다. (cgroup v2에서 CPU 리밋은 이 파일 하나로 표현됩니다.)

- 앞의 **`5000`** = **쿼터(Quota)**
- 뒤의 **`100000`** = **주기(Period)**, 한 주기 `100ms`(= 100,000 마이크로초)

즉 `0.1초` 동안 `50m코어`(5,000 / 100,000 = 5%)를 사용할 수 있다는 뜻으로, `limits.cpu: 50m`(CPU의 5%)과 정확히 일치합니다.

한 주기 `100ms` 기준으로 연속적으로 5%를 쓰는 게 아니라, 짧게 자신에게 할당된 만큼만 몰아서 쓰고 이후에는 강제로 멈추는 방식입니다. 그래서 내부에서는 이미 CPU가 막혀 있을 가능성이 매우 높죠. 그걸 정확히 확인하려면 `cpu.stat` 통계 수치를 봅니다.

```bash
cat /sys/fs/cgroup/$CGROUP_PATH/cpu.stat
```

`cpu.stat` 항목이 여러 개지만, 핵심 두 쌍만 확인합니다.

**(1) `nr_periods`와 `nr_throttled`**

- **`nr_periods`** = 총 주기 횟수. 한 주기가 `100ms`이고 **`7,359`번** 발생.
- **`nr_throttled`** = 스로틀링(강제 대기)이 발생한 횟수 = **`6,449`번**.

```text
6,449 / 7,359 × 100 ≈ 87.6%
```

**`87.6%`**라는 압도적인 비율로 쿼터 할당량을 다 써 버리고 강제로 멈춰야만 했던 것입니다.

**(2) `usage_usec`(마이크로초)와 `throttled_usec`(마이크로초)** — 시간 단위로 환산

- **`usage_usec`** = 프로세스가 실제로 CPU를 할당받아 연산한 시간. 약 `36,800,000μs` → **36.8초**.
- **`throttled_usec`** = 강제 정지된 시간. 약 `603,000,000μs` → 약 **603초**.

```text
총 경과 시간 = 36.8초(실행) + 603초(강제 정지) ≈ 640.7초
실행 비율    = 36.8 / 640.7 ≈ 5.7%
강제 대기    = 약 94.3%
```

CPU를 사용한 시간보다 약 **15배 이상** 더 길게 강제 대기 상태인 것을 수치로 확인할 수 있습니다.

이 데이터가 의미하는 바를 정리하면: 1코어 전체를 사용하려는 프로세스에게 단지 `50m코어`라는 매우 제한된 상한선을 설정했고, 리눅스 커널은 우리가 쿠버네티스 YAML에 선언한 `limits` 명령을 **한 치의 오차도 없이 물리적으로 아주 엄격하게 집행**하고 있다는 겁니다.

> 다만 이 시간은 강제로 차단된 시간뿐 아니라, 애플리케이션이 스스로 sleep한 시간이나 I/O 응답을 기다린 시간(I/O wait)은 포함하지 않습니다. 엄밀히 전체 시간 = **실행 시간 + 스로틀링(멈춤) 시간 + 애초에 실행이 안 된 시간(idle/sleep/대기)**입니다.

`kubectl top`(메트릭)으로 확인했을 때와 리눅스의 `cpu.stat`으로 확인했을 때, **어느 계층에서 데이터를 수집하느냐에 따라 해석하는 결론이 달라진다**는 걸 알 수 있습니다. 쿠버네티스에서는 정상으로 보였지만, **리눅스 커널 계층에서 보면 비정상**이었던 거죠. 앞서 OOM Killed 데모에서 본 구조와 비슷한 형태입니다. **쿠버네티스는 지시자, 리눅스 커널은 실제 집행자**인 거죠. 집행자의 현장에서 데이터를 실시간으로 보지 않으면, 쿠버네티스만 봤을 때는 "문제 없다"고 오판하게 됩니다.

실제 운영 환경에서 방금처럼 워커 노드에 직접 접속해 컨테이너 ID → 프로세스 ID → cgroup 경로 → 통계 절차를 일일이 확인하는 건 매우 까다롭겠죠. 그래서 현업에서는 **Prometheus**나 **Datadog** 같은 모니터링 도구를 사용해서, kubelet에 내장된 **cAdvisor**를 통해 커널의 CPU 통계 파일 값을 주기적으로 수집해 옵니다. 여러분이 Prometheus나 Datadog에서 보는 값들의 실제 출처가 바로 **리눅스 커널의 데이터**입니다. 오늘 터미널에서 직접 확인한 `nr_throttled`, `throttled_usec` 같은 값들이 결국 리눅스 커널의 원천 데이터인 거죠.

## 12. 핵심 명제 — 쿠버네티스는 지휘자, 리눅스 커널은 집행자
`[1:03:55]`

지금까지 살펴본 두 가지 실습 사례(**OOM Killed**와 **CPU 스로틀링**)를 보면 핵심 명제가 하나 있습니다. 쿠버네티스를 단순히 편리한 배포 도구, 오퍼레이터 수준으로 한정 지어서는 절대 안 됩니다. **시스템 전체 아키텍처를 이해하는 것**이 중요하고, **쿠버네티스와 커널을 명확히 구분**하는 것이 중요합니다.

- **쿠버네티스(K8s)는 실행자가 아닙니다.** 쿠버네티스는 **오케스트레이션(Orchestration), 즉 지휘자**입니다. "이 파드는 이런 상태로 유지해야 한다"라고 **의도를 선언**합니다. 하드웨어 자원을 직접 제어하거나 프로세스를 강제 종료시키는 **물리적 행위는 직접 하지 않습니다.**
- 그럼 누가 하느냐? 바로 **리눅스 커널**입니다. 리눅스 커널은 **실제 집행자**입니다. 쿠버네티스의 요청을 받아 실제로 CPU 할당량을 제한하고, 메모리 사용량을 감시하며, 프로세스를 정리합니다. **자원 통제의 최종 집행 권한자는 항상 리눅스 커널**에 있습니다.

비유하면, 쿠버네티스는 지휘자입니다. 지휘자는 각 연주자에게 어떤 악보를 어떻게 연주할지 지시를 내립니다. 그리고 실제로 악기를 들고 현을 켜거나 물리적인 소리를 만들어 내는 주체는 연주자죠. 그게 바로 리눅스 커널입니다.

## 13. 커널 핵심 기능 4가지
`[1:06:08]`

리눅스 커널이 중요하다는 걸 알았으니, 커널이 제공하는 핵심 4가지 기능(**Primitives**)을 간단히 살펴보겠습니다.

1. **네임스페이스(Namespace)**
   프로세스의 시야를 논리적으로 분리하고, 특정 컨테이너가 다른 컨테이너의 파일·네트워크·프로세스를 볼 수 없게 만듭니다. 이걸 **격리(Isolation)**라고 합니다.

2. **cgroup(컨트롤 그룹, Control Group)**
   CPU·메모리·네트워크 대역폭 등 하드웨어 자원의 상한선을 설정합니다. 앞서 살펴본 **OOM Killed**나 **CPU 스로틀링**의 실제 집행 주체가 바로 cgroup입니다.

3. **오버레이 파일시스템(OverlayFS)**
   컨테이너에서 가장 중요한 개념으로, 파일시스템 레이어를 하나로 합쳐서 보여 주는 친구입니다. 컨테이너 이미지는 그냥 하나의 파일시스템이 아니라 **여러 층의 이미지를 레이어 형식으로 쌓아** 하나의 통합된 파일시스템으로 동작하게 만드는 기술인데, 그걸 가능하게 한 게 OverlayFS입니다.

4. **veth(가상 이더넷 페어, Virtual Ethernet Pair)**
   컨테이너와 호스트 네트워크 사이를 잇는 가상의 네트워크입니다. 파드끼리 통신할 때 실제로 이동하는 물리적 경로를 형성하게 해 주는 것이 가상 이더넷 페어죠.

> 이 커널 Primitive들은 너무 중요해서 각 개별 챕터에서 아주 상세히 다룰 예정입니다.

인프라/데브옵스 엔지니어가 YAML 매니페스트만 작성하는 수준에서 머무르면, 문제가 직접 발생했을 때 대응의 한계에 부딪힙니다. `SIGKILL`도 cgroup의 메모리 제어 결과이고, 스로틀링도 cgroup의 스케줄링 결과입니다. 이미지 레이어는 OverlayFS의 동작 방식이고, 파드 네트워크 장애도 veth의 동작·설정 문제로 발생했는지를 알아야 합니다. **결국 모든 것은 커널의 기능(Primitive)들에서 맞닿아 있습니다.**

앞서 `kubectl describe`로 상태를 확인했는데, 여기서 끝나면 단순히 메모리/CPU 리밋 수치를 높여 당장의 장애를 해결하는 수준입니다. 그건 **증상에만 대응**하는 단계죠. 더 중요한 건 **근거를 가지고 분석**하는 것입니다.

문제 해결의 3단계:

1. **증상 대응** — 리밋 상한선을 올려 당장의 장애를 넘김(근본 원인보다 현상에 집중).
2. **기술적 근거 파악** — `dmesg` 같은 커널 로그를 확인해 커널이 어떤 결정을 내렸는지 기술적 근거를 파악.
3. **구조적 해결(Root Cause Analysis)** — `cpu.stat` 같은 원천 데이터를 직접 분석해, 애플리케이션과 인프라 설정 사이의 구조적 불일치를 근본적으로 해결.

장애는 무조건 발생합니다. 발생했을 때 단순히 리밋 상한선을 올리는 수준에서 머무는 게 아니라, **커널의 동작 원리를 정확히 이해**하고 그걸 바탕으로 시스템의 구조적 결함을 조정해야 한다 — 그 방향성이 이 강의가 지향하는 핵심입니다.

## 14. 물리 서버 시대의 4가지 한계
`[1:12:17]`

앞서 쿠버네티스와 커널의 관계로 현대 인프라의 작동 원리를 간략히 살펴봤습니다. 이제 이 기술들이 왜 필연적으로 등장할 수밖에 없었는지, 그 뿌리인 **물리 서버 시대의 한계**부터 짚어 보겠습니다.

`2000년대 이전~초반`까지는 이른바 **하나의 물리 서버 시대**입니다. 기업 내부 서버룸이나 소규모 데이터센터에서 물리 서버를 직접 설치하고, 그 위에 운영체제를 구성한 뒤 애플리케이션과 서비스를 배포하던 시절입니다. 당시에는 당연한 방식이었지만, 현대적 관점에서 보면 치명적인 구조적 한계가 있었습니다.

**물리 서버 시절의 문제 4가지:**

1. **의존성 충돌**
   서비스 A가 `Python 2.4`, 서비스 B가 `Python 2.6` 환경이면, 하나의 운영체제 위에서는 서로 다른 라이브러리 버전이라 공존시키기가 매우 어려웠습니다. 그래서 "내 로컬 PC에서는 잘 돌아가는데 왜 운영 서버에서는 작동하지 않냐"는 개발자 사이의 고전적 갈등이 있었죠.

2. **자원 격리(Isolation) 부재**
   동일 서버에서 실행 중인 프로세스 A가 예기치 못한 메모리 누수를 일으키면, 커널이 시스템 전체를 보호하려고 자원을 과다 점유하는 프로세스를 정리하는데, 그 과정에서 같은 자원을 공유하던 다른 프로세스 B까지 연쇄적으로 중단시켜 영향을 끼치는 문제가 있었습니다. A의 문제가 B에까지 전가되는 거죠.

3. **비효율적인 확장성**
   DB에만 트래픽 부하가 집중되는 상황이면 DB만 확장하면 되는데, 특정 컴포넌트만 확장하고 싶어도 **물리 서버 장비를 통째로 추가**해야 하는 문제가 있어서 자원 효율의 유연성이 극도로 떨어졌습니다. **부분적인 확장이 불가능**했던 거죠.

4. **배포의 어려움** (가장 근본적인 문제)
   새 기능을 배포하려면 기존 바이너리 파일을 직접 덮어쓰거나 수동으로 설정을 변경해야 했고, 장애가 발생했을 때 **즉각적인 롤백이 거의 불가능**했습니다. 그래서 과거에는 "금요일 오후에는 배포하지 말라"는 불문율이 있었죠. 한 번 서비스 오류가 나면 사실상 전면 중단으로 이어지고, 복구하려면 밤새 엔지니어·개발자가 수동으로 해결해야 했습니다.

이런 한계 때문에, 이를 개선하기 위해 `2000년대 초반` 이후에 **가상 머신(VM)**이라는 새로운 개념이 등장합니다.

## 15. 가상 머신(VM)의 등장 — 해결점과 새 문제
`[1:17:15]`

물리 서버의 한계를 극복하기 위해 `2000년대 초반`(`2001년경`)부터 **가상 머신(VM, Virtual Machine)**이 등장합니다.

가상 머신의 논리적 구조는, 물리 서버(Physical Server) 위에 가상화 엔진인 **하이퍼바이저(Hypervisor)**가 등장합니다. 하이퍼바이저는 하드웨어를 추상화해서 여러 대의 가상 서버로 나누는 계층입니다. 그리고 이 가상화 엔진 위에 VM들이 올라갑니다. 하이퍼바이저 위에서 구동되는 가상 머신은 독립적으로 운영됩니다.

**VM 등장으로 해결된 점:**

1. **완벽한 격리** — VM 간의 자원 간섭이 실질적으로 차단됨.
2. **의존성 독립** — 격리되었기 때문에 각 VM마다 다른 버전의 OS·라이브러리 설치가 가능.
3. **유연한 확장** — 이미지를 복제해서 빠르게 서버 대수를 늘릴 수 있게 됨.

**반대로 새로 생겨난 문제:**

1. **무겁다** — 각 VM이 자신만의 커널을 포함한 전체 OS를 통째로 가져야 했기 때문에 무거웠습니다(게스트 OS의 가상 CPU·RAM·디스크·네트워크 + 커널 포함 전체 OS).
2. **느린 속도** — OS 부팅 과정을 매번 거쳐야 했기 때문에 부트스트래핑에 수 분 이상 소요됐습니다.
3. **오버헤드** — 서비스와 무관한 OS 유지 비용이 발생. 가상 CPU·RAM·디스크·네트워크에 필요한 리소스를 수행하기 위한 OS 유지 비용이 들었습니다.

그런데 VM의 가장 큰 아킬레스건은 바로 **커널 중복**입니다. 애플리케이션이 아무런 작업을 하지 않아도, 커널의 리소스는 유휴(idle) 상태로 있어야 하기 때문에 리소스 낭비가 발생합니다.

예를 들어 VM이 10대 필요하다면, 메모리상에 **10개의 독립적인 게스트 커널이 상주**해야 합니다. OS도 10대, 커널도 10개. 애플리케이션 A가 요청을 처리하지 않는 순간에도, 그 VM의 커널·OS 백그라운드 프로세스는 지속적으로 자원을 소모합니다. 인프라 전체에 리소스 낭비가 발생해 자원 활용 효율이 급격히 떨어지는 상태가 만들어지는 거죠.

## 16. 하이퍼바이저 Type 1 vs Type 2
`[1:22:52]`

현업에서 인프라를 설계할 때 가장 중요하게 봐야 하는 것이 "가상화 환경이 어떤 방식으로 이루어졌는지" 그 구조를 이해하는 겁니다. 하이퍼바이저는 크게 두 가지 유형으로 나뉩니다.

**Type 1 — 베어메탈(Bare Metal)**
물리 서버 하드웨어 바로 위에 하이퍼바이저가 직접 설치되는 구조입니다. 중간 단계가 없어서 성능 손실이 최소화됩니다. 별도 소프트웨어가 설치되어 있지 않고 하나의 거대한 물리 서버를 온전히 사용하는 상태죠. 기업용 데이터센터나, 클라우드에서 엄청나게 큰 효율성이 필요한 프로덕션 환경에서 사용합니다.

```text
[ Type 1 ]  하드웨어 → 하이퍼바이저 → VM (2단계)
```

**Type 2 — 호스티드(Hosted)**
기존 OS(Windows·macOS) 위에 소프트웨어적으로 설치됩니다. 호스트 OS를 한 번 더 거치기 때문에 (물리 서버까지 포함하면 3단계) 연산 오버헤드와 성능 손실이 발생합니다. 개발자들이 보통 로컬 테스트 환경이나 개인용 PC에서 주로 사용합니다.

```text
[ Type 2 ]  하드웨어 → 호스트 OS → 하이퍼바이저 → VM (3단계)
```

> 물론 `2026년` 기준 현대 시점에서는 Type 2 하이퍼바이저도 성능적으로 많이 개선됐습니다. VirtualBox를 노트북/Windows 환경에서 써 봤다면 속도가 느리다고 느낄 수 있는데, 그 이유가 바로 이렇게 소프트웨어적으로 한 번 더 설치되기 때문입니다.

반면 우리가 사용하는 클라우드 서비스 인프라는 대부분 **Type 1** 방식을 채택해서, 사실상 물리 서버의 리소스를 거의 100% 가까이 사용할 수 있는 성능을 이끌어냅니다. 하지만 Type 1이라도 **커널 중복**으로 인한 근본적 문제가 남아 있습니다. 이 비효율성을 걷어내기 위해 **커널을 공유**하는 **컨테이너** 기술이 역사적으로 전면에 등장합니다.

---

## 17. 컨테이너 — 격리된 리눅스 프로세스
`[1:26:11]`

당시 가상 머신의 무거운 구조를 경험하면서 엔지니어들은 근본적인 의문이 생겼습니다. "단순히 애플리케이션 하나를 격리해서 실행하고 싶을 뿐인데, 매번 기가바이트 단위의 운영체제를 통째로 띄워야 하는가?" 이 갈등을 해결하기 위해 등장한 것이 바로 **컨테이너(Container)**입니다.

많은 분들이 **도커(Docker)**와 컨테이너를 동일한 개념으로 혼동합니다. 하지만 엄밀히 말하면 **도커는 컨테이너를 다루는 수단**이고, **컨테이너는 커널 기술로 구현된 격리 실행 환경**입니다. 둘은 1:1로 매칭되는 개념이 아니라는 점을 기억하세요.

컨테이너의 기술적 정의를 자세히 보면, 컨테이너는 **격리된 리눅스 프로세스(그룹)**입니다. 세 가지 종류의 커널 기술로 구현됩니다.

1. **네임스페이스(Namespace) — 격리**
   프로세스가 참조할 수 있는 시스템 리소스(네트워크·PID·마운트 등)의 가시성을 논리적으로 격리합니다. 프로세스가 시스템 내 다른 프로세스나 리소스에 접근하는 것을 원천적으로 차단하고 독립적인 실행 환경을 보장합니다.

2. **cgroup(컨트롤 그룹) — 자원 제한**
   프로세스 그룹이 점유할 수 있는 호스트의 물리 하드웨어 리소스(CPU·메모리·Block I/O 등)의 상한선을 통제하고, 자원 사용량을 모니터링·제한합니다.

3. **OverlayFS(파일시스템 레이어)**
   베이스 이미지와 변경 사항을 층층이 쌓아 독립적인 하나의 파일시스템처럼 구성합니다.

컨테이너는 완전히 새로운 발명품이 아닙니다. 이미 리눅스 커널이 수십 년 전부터 가지고 있던 기능들을 조합해서, 특정 프로세스가 마치 독립된 서버에서 실행되는 것처럼 논리적으로 분리해 낸 것뿐입니다. **컨테이너의 본체는 결국 리눅스 프로세스**라는 점을 기억하세요.

**가상 머신 vs 컨테이너 — 결정적 차이는 "커널을 각자 가지느냐 vs 공유하느냐"입니다.**

- **가상 머신**: 하드웨어를 가상화하는 기술. 하나의 애플리케이션을 실행하더라도 게스트 OS 전체(커널 포함)를 함께 실행해야 합니다. (`앱 → 커널 포함 게스트 OS → 하이퍼바이저 → 하드웨어`) 독립적인 OS·커널을 가지므로 완전히 격리되지만, 리소스 사용량이 많고 무겁습니다. 이미지 크기는 GB 단위, 기동 시간은 초~분 단위로 상대적으로 느립니다.
- **컨테이너**: OS 레벨의 격리 기술. 별도 OS를 실행하지 않고 **호스트 OS 커널을 그대로 공유**합니다. (`앱 → 컨테이너 런타임 → 호스트 커널 → 하드웨어`) OS 부팅이 필요 없고 필요한 것만 포함되므로 이미지 크기가 훨씬 작고, 기동 속도는 ms~초 단위로 매우 빠릅니다. 여러 컨테이너가 하나의 커널을 공유하므로 동일 서버에서 훨씬 높은 밀도로 실행할 수 있습니다. 다만 커널을 공유하는 구조라 VM보다 격리 수준은 상대적으로 낮을 수 있습니다.

## 18. 도커(Docker) — 커널 기술의 대중화
`[1:31:52]`

컨테이너를 만드는 핵심 기술들은 **도커가 나오기 훨씬 전부터** 존재했습니다. 리눅스 커널의 마운트 네임스페이스, cgroup 등은 이미 `2013년` 훨씬 이전에 도입됐습니다. PID·User 네임스페이스도 도커 이전부터 존재했죠.

- **도커 등장 이전**: 네임스페이스, cgroup, Union File System 등을 **직접 명령어로 조합**해서 격리 환경을 구축해야 했습니다. 리눅스 커널에 깊은 이해가 있는 고급 엔지니어들만 가능했던 **전문가 영역**이었죠.
- **도커 등장 이후**: 대중화가 시작됩니다. `2013년`에 발표된 도커는 이 복잡한 커널 기술들을 단지 **`docker run`**이라는 간단한 명령어 한 줄로 포장해 버렸습니다. 추상화했기 때문에 누구나 손쉽게 커널 기능을 사용할 수 있게 됐습니다.

도커는 새로운 기술을 발명한 게 아니라, 이미 존재하던 강력한 커널 기술들을 **누구나 사용할 수 있도록 패키징**한 혁신적인 도구였다는 점을 기억하세요. 이 편의성이 전 세계 개발자들을 열광시키고 오늘날 클라우드 네이티브 생태계의 토대가 되었습니다. 도커 덕분에 컨테이너 기술이 대중화되었고, 이를 대규모로 관리하기 위한 **쿠버네티스**가 탄생한 거죠.

## 19. 쿠버네티스 런타임의 변천 — dockershim 제거
`[1:33:47]`

그런데 지난 몇 년간 쿠버네티스 생태계에서는 중요한 변화가 있었습니다. 바로 **도커와의 결별**입니다. 컨테이너 자체가 없어진 게 아니라, **도커라는 제품 자체가 분리**된 거죠.

쿠버네티스의 런타임 변화 과정을 짧게 살펴보면:

1. **초기 단계** (`v1.0`~`v1.4`경)
   kubelet이 도커를 API로 직접 호출하는 단계.

2. **CRI 표준화 시기**
   **CRI(Container Runtime Interface)**가 표준화됩니다. 그런데 도커가 CRI를 지원하지 않았기 때문에, kubelet 내부에 **dockershim**이라는 변환기(중계기)가 내장되었습니다. 쿠버네티스가 도커와 직접 통신할 수 없어, 중간에 dockershim을 두고 도커와 통신했던 거죠.

3. **현재 단계** (`v1.24`부터)
   **dockershim이 완전히 제거**됩니다. 이제 쿠버네티스는 CRI 표준을 따르는 **containerd**와 통신합니다. kubelet → (CRI) → containerd로, dockershim이라는 변환기 없이 CRI 표준에 따라 직접 통신합니다. 중간 변환기 없이 커널 기능을 더 효율적으로 제어할 수 있게 된 거죠.

결국 쿠버네티스는 리눅스 커널이 제공하는 정교한 격리 기술들을, **CRI라는 표준화된 인터페이스**를 통해 수천 대의 서버에 걸쳐 자동화하는 거대한 시스템이라고 기억하시면 됩니다.

## 20. VM 위에 컨테이너 — 멀티테넌시와 컨테이너 탈출
`[1:36:49]`

컨테이너 기술이 비약적으로 발전하면서 많은 분들이 "가상 머신의 시대는 끝났다"고 생각했습니다. 하지만 실제 클라우드 인프라 내부를 들여다 보면 상황은 사뭇 다릅니다. 현재 대부분의 클라우드 환경은 VM을 제거하는 대신, **VM 위에 컨테이너를 올리는 계층 구조**를 따릅니다.

AWS의 인프라 스택(EC2 기준)을 보면:

```text
물리 서버(Physical Server)
  → Nitro 하이퍼바이저 (AWS 자체 개발)
    → EC2(VM) — 내부에 리눅스 커널
      → containerd
        → 컨테이너 → 여러 앱 실행
```

VM을 제거한 게 아니라 오히려 VM 위에 컨테이너를 올리는 계층 구조죠. 물론 과거에 비해 클라우드 환경 자체가 훨씬 가벼워진 것도 사실입니다. 그럼에도 왜 효율적인 컨테이너가 있는데 굳이 VM 레이어를 사용할까요? 두 가지 이유 때문입니다.

1. **보안(Security)**
2. **멀티테넌시(Multi-tenancy)** — 여러 사용자가 하나의 거대한 물리 서버를 안전하게 나누어 사용

컨테이너는 호스트 OS 커널을 공유한다고 했습니다. 이 공유 구조는 효율적이지만, 퍼블릭 클라우드 환경에서는 치명적인 단점(보안 허점)이 될 수 있습니다. 멀티테넌시 환경에서는 수백~수만 명의 고객이 하나의 같은 물리 인프라를 공유할 수 있기 때문이죠.

만약 고객 A가 컨테이너를 해킹당했을 때, 공유 중인 커널까지 탈취당할 가능성이 생깁니다. 그러면 같은 커널을 공유하는 고객 B의 데이터도 위험에 빠지게 되죠. 이렇게 컨테이너 밖으로 위험이 빠져나오는 경우를 **브레이크아웃(Breakout, 컨테이너 탈출)**이라고 합니다.

그래서 고객과 다른 고객 사이에 **VM으로 하드웨어를 프로비저닝해서 하드웨어 레벨의 벽**을 세웁니다. 그러면 고객 A가 해킹을 당해도, VM의 하드웨어적 경계 때문에 고객 B의 데이터·애플리케이션은 안전합니다.

결국 가상 머신과 컨테이너는 서로 대체하는 경쟁 관계가 아니라, **VM이 안정성(보안 경계)을 담보하고 컨테이너가 효율성을 담보하는 상호 보완적 관계**라고 이해하시면 됩니다.

## 21. AWS Nitro 하이퍼바이저 — 하드웨어 오프로딩
`[1:40:33]`

AWS는 **Nitro 하이퍼바이저**라는 별도의 하이퍼바이저를 개발했다고 말씀드렸습니다. AWS Nitro 시스템은 두 개의 Nitro 카드(하나는 Primary, 하나는 Networking 기능)로 구성됩니다.

**일반 하이퍼바이저**는 네트워크·스토리지·보안 등을 **메인 CPU가 처리**합니다. 그래서 가상화 관리에 어쩔 수 없이 CPU의 일정 부분(과거에는 약 `15~20%`)을 소모하는 오버헤드가 발생했습니다.

이 문제를 극복하기 위해 AWS는 **Nitro**, 즉 전용 하드웨어 칩(Nitro 카드)을 개발했습니다. 이걸 **하드웨어 오프로딩(Hardware Offloading)**이라고 하는데, 메인 CPU는 가상화 관리에 자원을 거의 안 뺏기면서 거의 `100%`에 가까운 CPU 에너지를 오로지 애플리케이션에 할당할 수 있게 됐습니다. 별도의 하이퍼바이저(가상화 담당) 부분을 하드웨어가 담당하게 따로 만든 거죠.

이 안에 들어간 소프트웨어(AWS Nitro 하이퍼바이저)는 거의 CPU를 사용하지 않기 때문에, 업계에서는 이를 **니어 베어메탈(Near Bare Metal)**이라고 부릅니다. 거의 100%에 가깝게 CPU 리소스를 애플리케이션이 사용할 수 있어 베어메탈에 근접한 성능을 낸다고 해서 그렇게 불립니다.

## 22. 베어메탈(Bare Metal) — 특수 목적 환경
`[1:43:24]`

가상 머신이나 하이퍼바이저가 아무리 가벼워지고 발전했더라도, 하드웨어 자원을 가상화에서 나누어 주는 과정은 필연적으로 **가상화 오버헤드**가 아주 약간이라도 발생합니다. 자체 데이터센터를 운영하는 **온프레미스(On-premise)** 환경이나 극강의 성능이 필요한 곳에서는 이 VM 레이어를 완전히 제거하기도 합니다. 이를 **베어메탈(Bare Metal)**이라고 합니다.

베어메탈은 말 그대로 아무 소프트웨어도 깔리지 않은 순수한 하드웨어 자체입니다.

```text
[ 클라우드 ]  물리 서버 → Nitro(하이퍼바이저) → VM → OS → 컨테이너
[ 베어메탈 ]  물리 서버 → OS → 컨테이너   (단계 축소)
```

베어메탈 환경은 가상화 레이어가 없기 때문에 지연 시간이 거의 0에 가깝고, 하드웨어 자원을 직접 제어할 수 있습니다. 단계가 축소되어 훨씬 효율적으로 하드웨어 자원을 사용할 수 있죠.

**베어메탈이 쓰이는 특수 목적 환경:**

1. **고성능 연산** — 대규모 데이터 분석, AI·머신러닝 모델 학습 시 어마어마한 GPU 자원에 직접 접근해야 할 때.
2. **초저지연(Low Latency)** — 금융권처럼 마이크로초 단위의 극소 지연 시간 내에 체결이 이루어져야 하는 초저지연 트레이딩 시스템.
3. **대규모 배치(Batch)** — 수천만~수억 건의 데이터를 최단 시간 내에 처리해야 할 때. VM의 아주 조그마한 오버헤드조차 허용되지 않는 경우.

## 23. 실습 정리와 1챕터 마무리
`[1:46:33]`

실습을 이어서 바로 진행하지 않는다면, **항상 실행 중인 쿠버네티스 클러스터를 정리**해야 합니다. 그렇지 않으면 AWS에서 리소스 비용이 계속 청구됩니다. 실습 랩 환경이 끝나면 반드시 파드를 삭제하고, 그다음 쿠버네티스 클러스터를 삭제하세요.

로컬 환경의 `kubeadm-cluster` 폴더에서 Terraform으로 클러스터를 생성했으므로:

```bash
terraform state list      # 현재 Terraform 리소스 확인
terraform destroy         # 더 이상 사용하지 않으면 반드시 삭제
```

다시 실습할 때는 `terraform apply`로 다시 생성하면 됩니다.

**1챕터 정리:**

물리 서버 → VM → 컨테이너 → 그리고 현재의 클라우드 네이티브, 즉 **VM이 보안 경계를 담당하고 컨테이너가 효율을 담당하는 공존 구조**까지 이루어졌음을 알 수 있었습니다. VM은 격리·확장 문제를 해결했고, 컨테이너는 커널을 공유하기 때문에 가볍고 빠르다는 장점이 있었죠.

그리고 이 모든 걸 관통하는 핵심 — 쿠버네티스를 잘 이해하고 실무에서 여러 에러 핸들링을 잘하기 위해서는, **쿠버네티스 자체보다 그 밑단에 있는 리눅스 커널을 이해**해야 합니다. **쿠버네티스 리소스만큼 중요한 것이 리눅스 커널**입니다.

> **K8s는 지시자(지휘자), 커널은 집행자다.**

이제부터는 리눅스 커널의 중요 요소들을 하나하나 더 자세히 살펴볼 예정입니다. 앞서 예시로 본 **OOM Killed(메모리 부족 현상)**나 **CFS 스로틀링**이 커널 레벨에서 정확히 어떤 경로를 거치는지, 그 부분들을 하나하나 자세히 살펴보겠습니다.
