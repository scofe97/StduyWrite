## 0. 인트로 — 지휘자 쿠버네티스, 집행자 리눅스 커널
`[00:06]`

안녕하세요, 플레이빌더 카이입니다. 이전 챕터에서 우리가 확인한 사실이 무엇이었죠? **쿠버네티스는 지시자(지휘자)**라는 사실이었죠. 그리고 **리눅스 커널은 실행자(집행자)**.

우리가 마주했던 **OOM**이나 **CPU 스로틀링(Throttling)** 같은 장애 상황들은 결국 모두 집행자인 리눅스 커널이 내린 결정의 결과물인 것을 확인했습니다.

많은 엔지니어가 커널을 그저 알아서 돌아가는 블랙박스처럼 취급하곤 합니다. 하지만 쿠버네티스를 깊이 있게 다루려면 이 블랙박스 안에서 자원이 어떻게 관리되고 명령이 어떻게 수행되는지, 그 설계 구조를 반드시 짚고 넘어가야 합니다. 그래야만 장애 로그 너머의 진실을 볼 수 있기 때문입니다.

## 1. 오늘의 5가지 핵심 질문
`[01:10]`

오늘 다룰 주제와 관련해서 핵심 질문 다섯 가지를 해보겠습니다.

1. **커널은 메모리를 왜, 그리고 어떻게 나누는가?**
2. **유저 스페이스의 프로그램이 커널에 어떻게 요청을 하는가?**
3. **쿠버네티스의 모든 컴포넌트는 커널 위에서 어떻게 동작하는가?**
4. **컨테이너가 동작하려면 커널이 무엇이 준비되어야 하는가?**
5. **`/proc`이라는 가상 파일시스템은 디스크에 없는데도 왜 파일처럼 보이는가?**

이 다섯 가지 질문을 정확히 이해할수록 **kube-proxy**, **kubelet**, **containerd** 같은 쿠버네티스의 컴포넌트를 구조적으로 이해하고 내부 동작을 보기 시작할 수 있게 됩니다. 오늘 수업은 이 질문들에 대한 해답을 찾아가는 과정입니다. 기술적 팩트를 하나씩 살펴보고 실습으로 동작을 확인한 다음, 마지막 총정리 시간에 이 질문들의 답을 하나씩 꺼내 보겠습니다.

그런데 본격적인 시작에 앞서 "왜 빨리 쿠버네티스 컴포넌트나 리소스 만드는 법을 배우지 않고 리눅스 기반 기술을 먼저 배워야 하느냐"는 의문이 드실 수 있습니다. 이유는 명백합니다. **쿠버네티스는 결국 리눅스에 의존 관계가 성립**하기 때문입니다. 쿠버네티스의 모든 컴포넌트는 결국 리눅스 위에서 동작하죠.

- 컨트롤 플레인과 워커 노드를 관리하는 핵심인 **kubelet**은 사실 리눅스의 시스템 데몬에 불과합니다.
- 가장 많이 쓰는 **파드(Pod)**는 결국 리눅스의 **네임스페이스(Namespace)**와 **cgroup(컨트롤 그룹)**으로 만들어진 격리 공간입니다.
- 트래픽을 분산하는 **서비스 라우팅**은 리눅스 커널의 **iptables**나 **eBPF** 기술로 구현되어 있습니다.

즉 밑바탕인 리눅스를 정확히 모르면 쿠버네티스는 단순히 명령어를 쳐서 사용하는 것에 불과합니다. 장애가 났을 때 그 구조를 이해하고 쿠버네티스를 더 정확하게 이해하는 것이 중요합니다.

## 2. 메모리의 두 영역 — 유저 스페이스 vs 커널 스페이스
`[05:14]`

리눅스 운영체제는 메모리를 두 영역으로 나눕니다. 첫 번째는 **유저 스페이스(User Space)**, 두 번째는 **커널 스페이스(Kernel Space)**입니다.

- **유저 스페이스**: kubelet, containerd, Nginx, 또는 여러분이 직접 만든 앱 등이 전부 여기서 동작합니다.
- **커널 스페이스**: OS의 핵심이 사는 곳입니다. 프로세스 스케줄러, 메모리 관리, 파일시스템, 네트워크 스택 등 중요한 요소들이 존재합니다. 뿐만 아니라 cgroup이나 네임스페이스 같이 앞서 살펴본 요소도 여기에 머무릅니다.

유저 스페이스와 커널 스페이스 사이에는 하나의 경계선이 있습니다. 이것을 **시스템 콜 인터페이스(System Call Interface)**라고 합니다. 즉 유저 스페이스에서 커널 스페이스로 넘어가는 **유일한 문**이 바로 이 시스템 콜 인터페이스입니다. 유저 스페이스의 kubelet·containerd 등은 **시스템 콜(System Call)**을 통해 이 경계선을 넘어 커널 스페이스로 전달됩니다.

## 3. 왜 공간을 나누는가 — 보안과 안정성(커널 패닉)
`[06:37]`

그렇다면 왜 굳이 커널 스페이스와 유저 스페이스로 나눌까요?

**첫 번째 이유는 보안**입니다. 유저 스페이스의 프로그램이 직접 하드웨어를 조작할 수 있다면 어떻게 될까요?
> (항상 질문에 잠시라도 생각을 하고 듣는 것과 그렇지 않은 건 장기 기억 측면에서 큰 차이가 있으니, 잠시 멈춰 생각해 보시면 좋습니다.)

바로 다른 프로세스의 메모리를 읽거나 디스크를 파괴할 수 있지 않을까요? 그래서 중간에 커널이 존재하고, 이 커널이 **허용된 작업만 대행**합니다.

**두 번째 이유는 안정성**입니다. Nginx가 버그 등으로 크래시되면 Nginx 프로세스가 다운되겠죠. 하지만 **커널 자체가 크래시되면 전체 시스템이 다운**됩니다. 이런 상황을 **커널 패닉(Kernel Panic)**이라고 합니다.

쿠버네티스 프로덕션 환경에서 커널 패닉이 발생하면 노드가 통째로 `NotReady` 상태로 빠지고, 그 위의 파드들이 다른 노드로 **재스케줄링(Rescheduling)**됩니다. 커널 패닉 메시지에는 `BUG`나 `Oops` 같은 에러가 반복됩니다. 이렇게 되면 해당 노드는 즉시 스케줄 불가 상태이므로 쿠버네티스는 **Cordon(코든)** 상태로 재배치하는 거죠.

그래서 커널 코드는 매우 신중하게 작성되었고, 일반 프로그램(kubelet·컨테이너·Nginx·여러분의 앱)이 이 커널 영역에 직접 접근하지 못하게 막습니다.

## 4. 쿠버네티스 맥락 — kubelet·containerd·runc·커널
`[08:35]`

이걸 쿠버네티스 맥락에서 살펴볼게요. **kubelet**은 유저 스페이스에 존재하죠. 이 kubelet은 Go 바이너리 형태인데, "파드를 생성해 줘"라고 요청하면 **kubelet이 직접 파드를 생성하는 게 아닙니다.**

> 지금 kubelet 컴포넌트가 무엇인지 전혀 몰라도 괜찮습니다. kubelet이라는 쿠버네티스 컴포넌트가 "파드를 만드는 지시 책임자" 정도로 이해하셔도 충분합니다.

그럼 kubelet이 직접 수행하지 않는다면 누구에게 맡길까요? **시스템 콜**을 통해 커널에게 요청합니다. 그 요청을 받은 커널이 **네임스페이스를 만들고 cgroup을 설정**합니다.

**containerd**도 동일합니다. "컨테이너를 시작해라" 요청을 받으면, 최종적으로 **runc**라는 저수준 런타임이 시스템 콜로 커널에게 **프로세스 생성과 격리**를 요청합니다.

실제 터미널 실습에서 자세히 보겠지만, runc 프로세스를 **strace**라는 추적 명령어로 확인해 보면, 컨테이너 생성 시점에 `clone`이나 `unshare` 같은 플래그의 시스템 콜이 찍히는 것을 확인할 수 있습니다.

결론적으로 **쿠버네티스의 모든 컴포넌트는 유저 스페이스에서 돌아가고, 실제 동작하는 일은 커널 스페이스에서** 이루어집니다. 즉 커널이 시스템 콜을 받아서 처리하는 구조입니다.

## 5. 시스템 콜이란 — 460개·seccomp·컨테이너 탈출
`[10:23]`

시스템 콜의 정확한 정의는 무엇일까요? **유저 스페이스 프로그램이 커널에게 "이거 해 줘"라고 요청하는 공식 인터페이스**입니다.

- 파일을 읽고 싶다 → `open` 시스템 콜을 부른다.
- 네트워크로 데이터를 보내고 싶다 → `sendto` 시스템 콜을 보낸다.
- 새 프로세스를 만들고 싶다 → `clone` 시스템 콜을 부른다.
- 컨테이너의 핵심인 프로세스를 격리하고 싶다 → 아주 중요한 **`unshare`** 시스템 콜을 요청한다.

최신 리눅스 커널 기준으로 시스템 콜은 대략 **460개** 정도 됩니다. 인프라 엔지니어가 이 460개를 다 외울 필요는 없지만, **쿠버네티스 보안 아키텍트**라면 이 수백 개 중 보안 허점이 될 수 있는 빈틈들을 미리 파악하고 대비해야 합니다.

생각해 볼게요. 파드 안의 컨테이너가 이 460개의 시스템 콜을 마음대로 다 부를 수 있다면? 해커가 Nginx 취약점을 뚫고 이 파드에서 커널 모듈을 조작하거나 권한을 상승시키는 위험한 시스템 콜을 날리면, 파드 하나가 털리는 게 아니라 **노드 전체의 통제권이 넘어갑니다.** 이걸 **컨테이너 탈출(Container Escape)**이라고 하죠.

그래서 반드시 쿠버네티스 보안 설정 시 **seccomp** 같은 보안 프로파일을 적용해야 합니다. (seccomp 같은 보안 프로파일은 쿠버네티스 보안 챕터에서 상세히 다룰 예정.)

정리하면, 프로그램 하나하나가 **시스템 콜로 커널에게 요청**하고, 커널이 실행해서 **하드웨어에 반영**합니다. 이 동작이 리눅스의 핵심 동작 원리입니다. 쿠버네티스에서 일어나는 모든 일(파드 생성, 네트워크 설정, 볼륨 마운트 등)이 전부 이 원리와 함께합니다. 이제 가장 중요한 커널 스페이스 내부로 들어가 무엇들이 있는지 살펴보겠습니다.

## 6. 커널 아키텍처 — Core(내장) vs Module(확장)
`[13:45]`

리눅스 커널 아키텍처의 핵심 **내장(Core) 영역**과, 동적으로 로드하는 **확장(Module) 영역**의 디테일을 살펴보겠습니다. 커널은 시스템 부팅에 필수적인 **코어 서브시스템(Core Subsystems)**과, 필요할 때마다 동적으로 붙이는 **커널 모듈(Kernel Module)** 수가 있습니다.

**코어 영역**은 커널 컴파일 시 바이너리에 완전히 내장되는 뼈대입니다. `modprobe` 같은 로드 방식으로는 로드할 수 없고, 이 코어 서브시스템 중 하나라도 누락되면 리눅스는 부팅조차 되지 않습니다. 그렇기 때문에 이 요소들은 커널과 한 몸인 셈입니다. 이 핵심 코어 영역들을 하나씩 구체적으로 살펴보겠습니다.

## 7. 프로세스 스케줄러(CFS) — CPU request/limit
`[14:35]`

**CPU를 누구에게 줄지 결정하는 커널 시스템**이 바로 프로세스 스케줄러입니다. 서버에 프로세스가 100개 있는데 CPU가 딱 두 개뿐이라면? 스케줄러가 프로세스에게 CPU 시간을 잘게 쪼개서 순서대로 나눠줍니다. kubelet·컨테이너·Nginx 등 모든 프로세스가 동시에 돌아가는 이유가 바로 이 프로세스 스케줄러 덕분입니다.

쿠버네티스에서는 파드를 만들 때 CPU를 얼마나 쓸 건지 YAML에 적었죠.

- **`requests.cpu: 500m`** → 이 파드에게 CPU 코어 1개의 절반(50%)만큼을 **보장**해 달라는 뜻. `m`은 밀리 단위로, CPU 한 개를 1,000조각으로 나눈 값입니다. 이 값이 커널의 cgroup **`cpu.weight`** 항목으로 전달되어, 프로세스 스케줄러가 CPU 시간을 나눌 때 **가중치**로 사용합니다. CPU가 여유 있다면 500m 이상도 쓸 수 있고, 여러 파드가 경합하면 이 가중치 비율대로 나눠받습니다.
- **`limits.cpu: 1000m`** → 아무리 CPU가 남아돌아도 이 파드는 **1코어 이상 쓸 수 없다**는 뜻. cgroup의 **`cpu.max`** 항목으로 전달되어, 경합 여부와 상관없이 절대적 상한선을 유지합니다. 초과하면 커널이 해당 프로세스의 CPU 사용을 일시 정지시키는데, 이게 챕터 1에서 다뤘던 **CPU 스로틀링** 현상입니다.

결국 쿠버네티스 YAML에 적는 CPU `requests`/`limits`는 단순한 설정값이 아니라, 리눅스 프로세스 스케줄러에게 "이 컨테이너들에게 CPU 시간을 어떤 비율로 나눠주고 언제 끊어 버릴지" 지시하는 아주 디테일한 작업 지시서입니다.

## 8. 메모리 관리 — 할당·회수·OOM Killer·MiB
`[17:07]`

커널의 가장 중요한 임무 중 하나는 프로세스들에게 한정된 RAM을 적절히 나누어 주고 다 쓴 메모리는 다시 걷어들이는 것입니다(**RAM의 할당과 회수**).

시스템 전체적으로 메모리가 부족해지면 커널 안의 **OOM Killer**가 동작합니다. 챕터 1에서 본 OOM 현상(메모리 부족)이 발생하면, 프로세스를 강제로 종료시켜 메모리를 확보합니다.

쿠버네티스에서 파드를 만들 때 CPU처럼 YAML에 **`limits.memory`**를 적죠. 예를 들어 `512MiB`를 적으면 이 파드는 메모리를 512MiB까지만 쓸 수 있습니다. 이 값이 커널의 cgroup **`memory.max`** 파일에 기록되고, 프로세스가 512MiB를 넘어가는 순간 커널의 OOM Killer가 해당 프로세스를 즉시 종료합니다. 그러면 `kubectl describe pod`로 확인했을 때 **`OOMKilled`** 에러 메시지를 볼 수 있죠. 여기서 중요한 건 **파드를 죽인 건 커널**이라는 점입니다. 커널이 집행했고, 쿠버네티스는 그 결과를 `OOMKilled` 상태로 엔지니어에게 보고할 뿐입니다.

참고로 **메비바이트(MiB)** 단위를 보겠습니다. 흔히 쓰는 **메가바이트(MB)**는 100만 바이트이고, **MiB(메비바이트)**는 MB보다 약 `4.8%` 큽니다. 쿠버네티스처럼 리소스를 관리하는 영역에서는 이 4.8% 오차가 엄청난 차이를 불러일으키기 때문에, 오차를 없애기 위해 MB가 아니라 **이진법 기준인 MiB**를 표기로 사용합니다.

## 9. 가상 파일시스템(VFS) — "모든 것은 파일이다"
`[20:02]`

리눅스 안에는 일반 하드디스크를 포맷할 때 쓰는 **ext4**부터, 컨테이너 이미지 레이어를 겹겹이 쌓아주는 **오버레이 파일시스템(OverlayFS)**, 그리고 메모리 홀로그램인 **`/proc` 파일시스템(procfs)** 등 성격이 완전히 다른 수많은 파일시스템이 공존합니다.

신기한 건, 우리가 프로그래밍을 하거나 셸 명령어를 칠 때 이게 물리 디스크인지 컨테이너인지 메모리인지 전혀 신경 쓰지 않는다는 점이죠. 그저 똑같이 시스템 콜만 던져 주면 됩니다 — 파일을 열 때는 `open`, 읽을 때는 `read`, 쓸 때는 `write`. 서로 다른 언어를 쓰는 복잡한 세상인데 어떻게 이런 통일성이 유지될까요? 바로 커널 한가운데 존재하는 **가상 파일시스템(VFS, Virtual File System)** 덕분입니다.

VFS는 커널 한가운데에 존재하면서 "이 경로는 어떤 파일시스템에 존재하는 거지?"를 본인이 판단해서 적절한 파일시스템에 요청을 전달합니다. 컨테이너의 경우는 OverlayFS, 커널 상태를 보여줄 때는 `/proc`, 장치 정보를 보여줄 때는 `/sys` — 전부 이 VFS 위에서 동작합니다.

**"리눅스에서는 모든 것이 파일이다(Everything is a file)"**라는 유명한 말이 가능한 이유가 바로 이 VFS 덕분입니다.

## 10. 네트워크 스택 — 패킷 흐름과 iptables
`[22:01]`

이제 쿠버네티스 아키텍처에서 가장 복잡도가 높다고 평가받는 **네트워크 영역**으로 넘어갑니다. 파드 간 통신이나 서비스 로드 밸런싱에서, 쿠버네티스는 독자적인 네트워크 프로토콜이나 자체 패킷 처리 엔진을 사용하지 않습니다. **쿠버네티스만을 위한 완전히 새로운 네트워크 기술은 존재하지 않습니다.** 쿠버네티스 네트워킹의 실체는 철저하게 **리눅스 커널이 제공하는 기존 네트워크 스택**에 의존해 동작합니다.

패킷 흐름을 간단히 보면, 앱이 소켓을 열어 데이터를 던지면:

```text
앱(소켓) → 커널 진입 → 전송 계층(TCP/UDP) → 네트워크 계층(IP, 목적지 결정) → 하드웨어 → 물리 네트워크
```

쿠버네티스 역시 이 구조를 그대로 사용합니다. 즉 파드 간 기본 통신은 커널의 네트워크 스택 흐름을 그대로 타고 넘어갑니다.

다만 **서비스(Service)**의 경우는 통신이 조금 다릅니다. 서비스는 흐름 중간에 반드시 **netfilter**가 개입합니다. **kube-proxy**가 심어둔 **iptables** 규칙에 의해 목적지 IP가 변환됩니다. (복잡한 네트워크 구조는 네트워크 별도 챕터에서 자세히 다룰 예정.)

## 11. IPC(프로세스 간 통신) — 파드 IPC 네임스페이스
`[24:08]`

다음으로 살펴볼 커널 기술은 **IPC(프로세스 간 통신, Inter-Process Communication)**입니다. 쿠버네티스를 배우는데 굳이 IPC까지 짚는 이유는 무엇일까요?

같은 물리 서버 안에서 돌아가는 프로세스끼리 데이터를 주고받아야 한다고 가정해 보죠. 바로 옆에 있는데 매번 무거운 네트워크 스택을 다 거쳐 통신하면 너무 비효율적이지 않겠습니까? 그래서 커널은 네트워크 대신 **메모리 단에서 직접 데이터를 주고받는 빠른 통신 방식**을 제공하는데, 그게 바로 **공유 메모리(Shared Memory)** 방식이나 **세마포어(Semaphore)** 같은 기술입니다. 이걸 통틀어 **IPC**라고 부릅니다.

쿠버네티스 관점에서 보면, **같은 파드 안의 컨테이너들은 IPC 네임스페이스를 서로 공유**합니다. 즉 컨테이너 A와 B가 한 파드 안에 묶여 있으면, 굳이 네트워크 망을 타지 않아도 커널의 공유 메모리를 통해 직접 데이터를 주고받을 수 있습니다.

하지만 파드가 다르면? 아무리 같은 워커 노드 서버 안에서 동작해도, 파드 A와 파드 B가 다르면 IPC 공간은 완벽하게 분리(격리)됩니다. 서로의 공유 메모리나 파이프가 전혀 보이지 않습니다.

결론적으로 이 IPC 네임스페이스를 누구와 공유하고 누구와 격리하느냐 하는 규칙을 보면, 쿠버네티스가 **파드라는 논리적 단위**를 리눅스 커널 밑바닥에서 얼마나 확실하게 정의하고 보호하는지 직관적으로 확인할 수 있습니다. (IPC 개념은 IPC 네임스페이스 챕터에서 상세히.)

## 12. 커널 보안 2축 — LSM·seccomp·SecurityContext
`[26:39]`

리눅스 커널의 보안 체계는 두 축으로 나뉩니다.

**첫 번째 축은 LSM(Linux Security Module)**입니다. 실무에서 **SELinux**나 **AppArmor**로 고생하신 분들이 있을 텐데, 이 레이어는 쉽게 말해 **문지기 역할**을 합니다. "지금 이 프로세스가 저 파일이나 리소스에 접근해도 되는지"를 통제하는 거죠.

**두 번째 축은 seccomp**입니다. 이건 LSM과는 결이 좀 다른데, 앞서 배운 **시스템 콜 단위로 제어**하는 방식입니다. "지금 커널한테 이런 위험한 시스템 콜을 보내도 되는지" 같은 프로세스의 행동 자체를 틀어막아 버립니다. 즉 이 두 가지는 서로 완전히 다른 레벨에서 겹겹이 방어망을 치는 별개의 보안 체계입니다.

쿠버네티스에서는 파드를 생성할 때 YAML 파일의 **SecurityContext(시큐리티 컨텍스트)** 부분에서 이 두 가지를 제어할 수 있습니다. 여기서 권한 상승을 막거나, 루트가 아닌 일반 사용자로 실행하도록 강제하거나, 기본 seccomp 프로파일 등을 적용할 수 있습니다. 쿠버네티스 보안 전문 자격증인 **CKS**에서도 이 LSM·seccomp·SecurityContext 작성 방법이 핵심 시험 주제로 나오죠. (쿠버네티스 보안 챕터에서 상세히.)

## 13. 블록 레이어와 디바이스 모델 — CSI·GPU
`[28:30]`

**블록 레이어(Block Layer)**는 디스크에 데이터를 읽고 쓸 때 **I/O 요청을 정리하고 스케줄링**하는 계층입니다. 앞서 본 파일시스템이 "어떤 파일시스템인지"를 판단했다면, 블록 레이어는 그 아래에서 "디스크에 실제로 어떻게 쓸 건지"를 처리합니다. 쿠버네티스에서는 **CSI(컨테이너 스토리지 인터페이스, Container Storage Interface) 드라이버**가 이 블록 레이어를 통해 **PV(Persistent Volume)**를 파드에 마운트합니다.

**디바이스 모델(Device Model)**은 GPU 같은 하드웨어 장치를 소프트웨어적으로 관리하는 체계입니다. `/sys` 파일에 VFS를 통해 장치 정보를 노출해서, 쿠버네티스에서는 **DRA(Dynamic Resource Allocation)**를 통해 GPU 같은 특수 장치를 파드에 할당합니다.

이 두 영역은 별도 챕터에서 상세히 다룰 예정이니, 지금은 블록 레이어와 디바이스 모델이 코어 핵심 내장 영역에 속한다는 것을 먼저 기억해 두시면 됩니다.

## 14. 네임스페이스와 cgroup — 격리 vs 제한
`[29:44]`

지금까지 본 프로세스 스케줄러, 메모리 관리, VFS, 네트워크 스택 등은 리눅스가 원래 가지고 있던 범용 기능입니다. 컨테이너가 등장하기 훨씬 전부터 존재했던 기능이죠. 지금부터 볼 **네임스페이스**와 **cgroup**도 같은 코어 영역에 속하지만 성격이 조금 다릅니다. 컨테이너를 가능하게 만든 핵심 기술이기 때문에 둘 다 별도 챕터에서 상세히 다룰 예정이고, 지금은 개념만 봅니다.

한 줄로 정리하면 **컨테이너는 네임스페이스로 격리된 리눅스 프로세스**입니다.

- **네임스페이스 — "무엇을 볼 수 있는가"(격리)**
  프로세스에게 독립된 세계를 보여 줍니다. 컨테이너 내부에서는 PID(프로세스 ID)가 1번으로 보일지라도, 호스트 관점에서는 이 PID가 수백·수천 대의 평범한 프로세스 번호(예: `13576`)로 보입니다. 네트워크, 파일시스템, 호스트네임까지 전부 따로 보이게 됩니다. 네임스페이스는 커널 소스에 컴파일로 내장되어 있어 `modprobe`로 로드되지 않는, 커널과 한 몸인 기능입니다.
- **cgroup — "얼마나 쓸 수 있는가"(제한)**
  네임스페이스가 "무엇을 볼 수 있는가"를 격리했다면, cgroup은 "얼마나 쓸 수 있는가" 제한에 초점이 맞춰져 있습니다. 앞서 다룬 CPU `requests`, 메모리 `limits` 전부 이 cgroup을 통해 커널에 전달됩니다. "파드를 만들어라" 하면 kubelet → containerd → runc 순서로 전달되어 cgroup에 설정이 내려가고, 최종적으로 커널이 그 제한을 적용합니다. 쿠버네티스에서는 버전 `1.35`부터 **cgroup v2**가 필수적으로 도입되었습니다. cgroup도 네임스페이스처럼 커널 소스에 컴파일 내장된 코어 기능입니다.

## 15. 커널 모듈 — overlay와 이미지 레이어
`[32:32]`

코어가 커널과 한 몸인 뼈대라면, **모듈(Module)**은 필요할 때 붙이고 떼는 확장 기능입니다. 부팅할 때 무조건 올라오지 않고, 어떤 앱이 "이 기능을 꼭 써야 돼"라고 하면 그때 메모리에 올라옵니다.

- **`modprobe`** 명령어 → 모듈을 로드
- **`lsmod`** 명령어 → 지금 어떤 모듈이 올라와 있는지 확인

쿠버네티스를 설치할 때 반드시 먼저 올라와야 하는 모듈이 있습니다. 첫 번째가 바로 **overlay**입니다.

컨테이너 이미지는 하나의 덩어리 파일이 아니죠. Ubuntu 레이어가 있고, 그 위에 라이브러리 레이어, 또 그 위에 앱 — 이렇게 여러 겹의 레이어가 쌓여 있는 구조입니다. **overlay 모듈**이 이 여러 겹의 레이어를 겹쳐서 하나의 파일시스템처럼 보여 줍니다. containerd가 컨테이너를 시작할 때 이 overlay를 사용해 이미지 레이어들을 합성합니다. 그래서 이 모듈이 없으면 레이어를 합칠 수 없어 컨테이너 자체가 만들어지지 않죠. 쿠버네티스 노드를 세팅할 때 가장 먼저 `modprobe overlay`를 실행하는 이유가 바로 여기에 있습니다.

## 16. br_netfilter — 서비스 IP 패킷 흐름(도로 비유)
`[34:08]`

다음은 **br_netfilter(브릿지 넷필터)**, 네트워크 패킷 필터링 이야기입니다. 이 CNI 브릿지 통신 주제는 본래 커널 네트워크 스택에 대한 깊은 이해가 선행되어야 하는 고난이도 영역입니다. 여기서는 복잡한 기술 용어 대신 **자동차 도로 비유**로 구조적 역할에 집중해 직관적으로 이해해 봅니다.

핵심 질문: **왼쪽 파드 A가 서비스 IP로 패킷을 보내면 어떻게 오른쪽의 파드 B/C에 실제로 도달하는가?**

**출발 — 파드 A에서 택배(=패킷) 출발**
택배에 붙은 서비스 IP라는 목적지 주소가 `10.96.0.10`입니다. 그런데 이 주소는 **가상 IP**입니다. 어떤 네트워크 인터페이스와도 바인딩되어 있지 않습니다. 터미널에서 `ip addr` / `ifconfig`로 찾아봐도 어디에도 존재하지 않는, **오로지 iptables 규칙에만 존재하는 주소**입니다.

**첫 관문 — L2(레이어 2) 가상 브릿지**
파드 A에서 출발한 택배가 먼저 만나는 곳이 L2 브릿지입니다. L2 브릿지는 **MAC 주소로만 동작**합니다. 즉 IP 주소를 해석하는 건 L2 브릿지의 역할이 아니라서, 들어온 IP 주소를 그대로 둡니다. 여기서 두 가지 시나리오로 나뉩니다.

**시나리오 ① br_netfilter 비활성화 (통신 실패)**
택배 트럭이 L2라는 "논스톱 지하도로"를 그냥 직진합니다. 서비스 IP는 가상 IP라 어디에도 물리적으로 존재하지 않으므로, 트럭은 영원히 L2 브릿지를 달리다가 결국 **목적지 도달 불가**에 직면합니다. 즉 통신이 실패합니다.

**시나리오 ② br_netfilter 활성화 (통신 성공)**
같은 택배 트럭이 L2 브릿지에 진입하는데, 중간에 **강제로 진출하는 나들목(주황색 라인)**을 만납니다. 여기서 중요한 포인트 — br_netfilter는 이게 서비스 IP인지 다른 IP인지 **구분하지 않고, L2를 통과하는 모든 패킷을 다 검사**해서 위층으로 올려보냅니다.

**위층 — L3(레이어 3) iptables ("우회로 안내 교차로")**
맨 위의 **kube-proxy**가 교통 표지판처럼 iptables 규칙을 미리 정의해 둡니다. kube-proxy는 **API 서버를 Watch**하다가, 서비스 `10.96.0.10`이 만들어지고 그게 파드 B·C와 연결되어 있다는 정보를 보고 iptables 규칙을 미리 작성해 둡니다.

그래서 패킷이 올라오면 iptables가 **DNAT** 규칙을 적용합니다 — 서비스 가상 IP가 **실제 존재하는 파드 IP로 변환**됩니다. 예: `10.96.0.10` → `10.244.2.2` 또는 `10.244.1.5`. (두 개로 표시되는 건 로드 밸런싱 때문입니다.)

**라우팅 → 다시 L2 → 도착**
변환 후 라우팅으로 이동합니다. iptables는 주소 변환만 하고 직접 전달하진 않죠. 라우팅 규칙이 변환된 파드 IP가 어떤 파드인지 찾아냅니다. 같은 노드에 있으면 다시 L2 브릿지로 내려와 전달되고, 다른 노드면 **VXLAN** 터널을 타고 상대 노드로 넘어갑니다. 라우팅 판단이 끝나면 택배 트럭이 다시 L2 브릿지로 복귀하고, 이제 실제 목적지 주소로 변환되었기 때문에 L2의 MAC 주소로 파드 B/C를 찾아 성공적으로 패킷이 전달됩니다.

**정리**
- br_netfilter **비활성화** → 패킷이 직진하다 가상 IP는 실체가 없어 **통신 실패**
- br_netfilter **활성화** → 모든 패킷이 위층 L3 iptables로 올라가 **DNAT(가상 IP→실제 파드 IP) 변환** → 라우팅 → 다시 L2 → **목적지 도착**

(디테일은 상이할 수 있으니, 지금은 br_netfilter 활성화/비활성화 차이로 큰 개념만 잡고 네트워크 챕터에서 다시 다룹니다.)

## 17. 노드 필수 조건 — /proc와 커널 모듈/파라미터
`[44:08]`

쿠버네티스 노드가 정상 동작하려면 가상 파일시스템 **`/proc`**에 **`ip_forward`**와 **iptables 관련 파라미터**가 활성화되어 있어야 합니다. `/proc`은 **RAM에 띄운 가상 파일시스템**입니다. 하드디스크에 저장된 진짜 파일이 아니라는 거죠.

이게 무슨 뜻이냐면, 실제 디렉토리(예: `/etc`)에 `du`(디스크 사용량) 명령을 실행하면 실제 데이터 크기(예: `6.6M`)가 나옵니다. 반면 `/proc`의 디스크 공간을 확인하면 **0으로 표시**됩니다. `/proc`은 실제 파일처럼 보이지만 디스크에는 존재하지 않는 상태입니다.

앞에서 배운 VFS 구조를 떠올려 보면, 일반 파일은 `VFS → ext4 → 블록 레이어 → 디스크` 경로로 디스크 블록을 실제 점유합니다. 하지만 `/proc`은 VFS 계층에서 디스크로 내려오지 않습니다. 커널이 RAM에 있는 내부 데이터 구조를, `read` 시스템 콜이 불리면 **텍스트 형태로 실시간 렌더링**해서 터미널에 출력합니다. 그래서 디스크 블록을 점유하지 않아 크기가 0으로 표시됩니다.

> **CPU 정보 예시** — `cat /proc/cpuinfo`를 실행하면 노드의 CPU 정보가 모두 출력됩니다. 프로세서가 `0`, `1` 두 개(0부터 카운팅), 물리 CPU 코어는 1개, **`siblings`는 2**로 표시됩니다. siblings는 **하이퍼스레딩(Hyper-Threading)**으로 만든 두 개의 논리 CPU를 의미합니다. 하이퍼스레딩은 물리 코어 1개를 OS에게 논리 코어 2개처럼 보이게 만드는 기법으로(요리사 1명이 냄비 2개를 쓰는 비유), 약 `15~30%` 추가 성능 향상을 기대할 수 있습니다. 컨트롤 플레인 `t3.medium`은 2 vCPU(v=논리 가상 CPU)입니다. 마지막 **`flags`**는 CPU가 하드웨어적으로 지원하는 특수 기능·명령어 목록입니다(스마트폰 스펙표처럼). 부팅 시 커널이 CPU에게 "무슨 기능을 할 줄 아니?"라고 물어 응답한 결과물입니다. `ht`=하이퍼스레딩, `aes`=암호화(Advanced Encryption Standard), **`hypervisor`** 플래그가 있으면 이 머신이 물리 서버가 아니라 **가상 머신 위에서 동작**한다는 증거입니다.

> **메모리 정보 예시** — `dmidecode`로 메모리를 확인하면 메인보드에 꽂힌 실제 물리 RAM이 `4GB`입니다(이진수 단위라 사실상 4GiB). 그런데 `/proc/meminfo`에서 `MemTotal`을 보면 약 `3,920,000 KB` ≈ **`3.74GiB`**로 나옵니다. 4GB 스펙인데 약 `0.26GB`가 빠진 거죠. 이는 **커널이 부팅 시 자기 자신이 살아남기 위한 최소 생존 공간을 미리 떼어 놓았기** 때문입니다. `MemTotal`은 물리 RAM 총량이 아니라 OS가 일반 프로그램에게 빌려 줄 수 있는 통제 가능한 RAM 총량입니다(호텔 객실 100개 중 5개를 직원용으로 빼놓는 비유). `SwapTotal`은 스왑이 꺼져 있다는 의미인데, 쿠버네티스는 메모리 관리의 예측 가능성을 위해 **스왑을 항상 OFF**로 둡니다. `MemFree`는 완전히 비어 있는 메모리, `MemAvailable`은 캐시를 회수하면 확보 가능한 메모리까지 포함합니다. kubelet은 메모리 부족 판단 시 `MemAvailable`을 기준으로 사용합니다.

쿠버네티스 클러스터가 정상 동작하려면 세 가지 중요한 커널 파라미터가 준비되어야 합니다. ① `ip_forward`, ② `bridge-nf-call-iptables`, ③ `bridge-nf-call-ip6tables`. 이를 지원하는 두 커널 모듈(**overlay**, **br_netfilter**)이 먼저 정상 로드되어 있어야 합니다.

```bash
lsmod | grep -E "overlay|br_netfilter"   # 출력되면 로드됨, 없으면 미로드
```

- **overlay 미로드** → containerd가 이미지 레이어를 합성할 수 없어, 파드 생성 시 `CreateContainerError` 발생(컨테이너 자체가 안 만들어짐).
- **br_netfilter 미로드** → 이 모듈이 로드돼야 `/proc/sys/net/bridge/` 디렉토리가 생성됩니다. 모듈이 없으면 그 디렉토리 자체가 없어 `bridge-nf-call-iptables` 파라미터를 설정할 수조차 없습니다.

그래서 **모듈 로드가 먼저, 그다음 파라미터 설정** 순서여야 합니다.

## 18. ip_forward — 노드 간 파드 통신 라우팅
`[55:46]`

쿠버네티스 노드 통신을 위해 가장 먼저 확인할 커널 파라미터가 **`ip_forward`**입니다. 이 값은 이름 그대로 리눅스 커널이 **패킷 포워딩을 수행할지 말지**를 결정합니다. 기본적으로 리눅스는 이 값이 `0`으로 설정되어 있습니다. 수신된 패킷의 목적지가 자신의 IP가 아니면 포워딩하지 않고 즉시 폐기(드롭)합니다.

다이어그램으로 보면, 노드 1의 파드 A가 노드 2의 파드 B로 패킷을 보냅니다. 파드 A의 패킷이 노드 1의 커널 네트워크 스택에 도달하면, 커널은 먼저 패킷의 목적지 IP를 검사합니다(여기서는 `10.244.2.x`). 리눅스 커널 자체는 "파드"라는 논리적 개념을 인지하지 못하므로, 이 패킷을 단순히 "자신의 IP가 아닌 외부 목적지를 향하는 트래픽"으로 인식합니다.

- **`ip_forward = 0`** → 노드 1의 IP가 본인이 아니므로 포워딩을 거부하고 패킷을 즉시 드롭(폐기). 결과적으로 파드 A ↔ 파드 B 통신 실패.
- **`ip_forward = 1`** (활성화) → 목적지 IP가 자신이 아니더라도 패킷을 버리지 않고 **호스트의 라우팅 테이블을 참조해 다음 경로를 찾아갑니다.** 즉 리눅스 서버가 트래픽을 중계하는 **라우터 역할**을 하게 됩니다. 사전에 CNI가 각 노드의 라우팅 테이블에 파드 네트워크 대역 규칙을 기록해 두었기 때문에, 노드 1의 커널은 라우팅 테이블을 대조하여 노드 2로 정상 포워딩합니다.

만약 이 값이 `0`이라면, 같은 노드 내부의 파드끼리는 통신할 수 있어도, 물리적으로 다른 노드의 파드 간 통신은 라우팅되지 못하고 완전히 단절됩니다.

## 19. bridge-nf iptables/ip6tables
`[58:30]`

이번에는 **`bridge-nf-call-iptables`**와 **`bridge-nf-call-ip6tables`**를 다이어그램으로 봅니다. 클라이언트 파드에서 출발한 패킷은 밖으로 나가기 위해 노드의 가상 브릿지를 지나갑니다. 기술적으로 이 브릿지를 통과하는 패킷은 **iptables를 거치지 않는 것이 기본 동작**입니다. 여기서 `bridge-nf-call-iptables` 파라미터가 등장하는데, 이 설정은 **브릿지를 지나가는 패킷이 강제로 iptables를 거치도록** 통제하는 역할을 합니다.

- **`= 0`** → 패킷이 브릿지를 통과하지만 iptables를 거치지 않습니다(무시). IP 변환 규칙이 적용되지 않아 목적지가 가상 IP인 상태로 지나갑니다. 가상 IP는 가짜 IP라 라우팅 테이블에서 목적지를 찾을 수 없어 패킷이 버려지고 **통신 실패**. (§16의 "비활성화" 시나리오와 동일.)
- **`= 1`** (활성화) → 패킷이 브릿지를 지날 때 강제로 iptables를 지나칩니다. iptables가 검사해 **DNAT** 규칙을 적용 — 클러스터 IP(가상 IP)가 실제 파드 IP로 변환됩니다. 변환된 실제 IP 기반으로 정상 라우팅이 일어나 목적지에 무사히 도착합니다. (§16의 "활성화" 시나리오와 동일.)

**`bridge-nf-call-ip6tables`**도 동일한 원리로 동작하며, 다만 IPv6 트래픽을 담당할 뿐입니다.

정리하면, br_netfilter 모듈을 로드하면 커널이 `/proc/sys/net/bridge/bridge-nf-call-iptables` 파일을 생성합니다.

```bash
cat /proc/sys/net/bridge/bridge-nf-call-iptables   # 1 이면 활성화(ON)
```

이 파일의 값이 `1`이어야 가상 브릿지를 지나는 패킷이 iptables를 경유합니다. (세부는 네트워크 챕터에서.)

## 20. 실습 — /proc는 디스크에 없다
`[1:02:12]`

리눅스의 핵심 철학 "모든 것은 파일이다"를 염두에 두고 터미널에서 검증해 보겠습니다. 먼저 `/proc`이라는 가상 파일시스템은 물리 하드디스크에 존재하지 않는다는 사실을 확인합니다.

```bash
ls -l /proc          # 파일들이 보이지만 크기가 0바이트
du -sh /etc          # 예: 6.6M — 실제 디스크 공간 점유
du -sh /proc/cpuinfo # 0 — 디스크 점유 안 함
```

다만 `/proc` 중에 **`kcore`** 파일은 무려 **140TB**라는 비정상적으로 큰 크기로 보입니다. 이건 디스크 파일이 아니라 커널이 관리하는 **가상 메모리 전체 주소 공간**을 하나의 파일 형태로 매핑해 둔 것뿐입니다. 당연히 실제 하드디스크가 아니죠. 140TB로 표시되는 파일이 있음에도 `du`로 보면 완벽하게 **0바이트** — 하드디스크를 전혀 사용하지 않는다는 명백한 증거입니다.

`cat /proc/cpuinfo`로 엔터를 치는 순간, 커널이 RAM에 있는 자신의 상태 데이터를 읽어 와 실시간으로 텍스트로 렌더링해 줍니다.

```bash
cat /proc/cpuinfo | grep -E "processor|model|cpu cores|siblings"
# processor 0, 1 (총 2개) / cpu cores 1(물리 1개) / siblings 2(논리 2개)
```

## 21. 실습 — strace로 시스템 콜 추적
`[1:05:01]`

이제 커널 내부 동작을 추적하는 **strace** 명령어를 사용합니다. `cat /proc/cpuinfo`를 실행할 때 백그라운드에서 어떤 시스템 콜이 일어나는지 확인할 수 있습니다.

```bash
strace cat /proc/cpuinfo 2>&1 | grep -E "open|read|write"
```

(초기 경로 탐색 과정의 노이즈는 무시해도 됩니다.) 핵심 세 가지:

1. **`open`** — 프로세스가 커널에게 `/proc/cpuinfo` 파일을 열어 달라고 요청. 커널이 파일을 열고 파일 디스크립터 **`3`**을 반환(이제 이 파일은 "3번 통로"로 지정됨).
2. **`read`** — 3번 통로에서 데이터를 읽어 달라고 요청. 커널은 디스크가 아닌 자신의 RAM 상태를 읽어 텍스트를 생성. 반환값 끝의 **`2360`**은 커널이 실시간으로 만들어낸 텍스트 크기가 **2,360바이트**임을 의미.
3. **`write`** — cat 프로그램이 이 2,360바이트를 "1번 통로(표준출력)"로 출력해 달라고 명령. 그 결과가 방금 터미널에서 본 CPU 상세 정보.
4. 마지막으로 `read`가 한 번 더 호출되고 **반환값 `0`** — 더 이상 읽을 데이터가 없다는 뜻.

그래서 무심코 입력하는 `cat` 명령어 하나도 내부적으로는 이처럼 커널과의 정교한 시스템 콜 통신을 통해 완성됩니다.

## 22. 실습 — Core vs Module(lsmod·modprobe)
`[1:07:30]`

이론에서 다룬 커널 **코어(Core)**와 **모듈(Module)**의 차이도 직접 확인합니다. 구분 기준은 명확합니다 — `lsmod` 목록에 나타나면 탈부착 가능한 모듈, 나타나지 않으면 커널 자체에 내장된 코어 기능입니다.

```bash
lsmod | grep cgroup   # 출력 안 됨 → cgroup은 코어(내장), 로드/해제 불가
mount | grep cgroup   # cgroup v2가 /sys/fs/cgroup 에 마운트된 것 확인
```

**cgroup**은 컨테이너 리소스를 격리하는 핵심 기술인데, `lsmod`에 출력되지 않습니다. 커널 코어에 깊숙이 내장되어 별도 로드·해제가 불가능합니다. 다만 `mount`로 보면 가상 파일시스템으로 `/sys/fs/cgroup`에 마운트되어 있고, 쿠버네티스 최신 환경에서 **cgroup v2**를 사용함을 확인할 수 있습니다.

반대로 쿠버네티스 구동에 필수인 **overlay**와 **br_netfilter**는 모듈입니다.

```bash
lsmod | grep overlay     # 출력되면 로드됨
modinfo overlay          # 모듈의 상세 메타데이터(filename=디스크 원본 경로 등)
modprobe overlay         # 수동 로드(이미 로드돼 있어도 충돌 없음 = 멱등성)
```

`modinfo`의 `filename`을 보면 해당 모듈의 물리적 원본 파일이 디스크 어느 경로에 저장돼 있는지 확인할 수 있습니다. `modprobe`는 이미 활성화된 모듈을 다시 로드해도 충돌을 일으키지 않는 **멱등성(idempotency)**을 가집니다. overlay의 사용 카운트가 예를 들어 `29`로 나오면, 현재 이 노드의 컨테이너들이 OverlayFS 마운트 포인트를 29개나 사용 중이라는 뜻이고, 참조 횟수가 0보다 크면 모듈을 강제로 내릴 수 없는 상태입니다.

## 23. 커널 레벨 필수 조건 5가지 점검 + 마무리
`[1:10:23]`

마지막으로 쿠버네티스 노드의 커널 레벨 필수 조건 다섯 가지가 충족되었는지 점검합니다.

```bash
# ① ② 커널 모듈 2개 로드 확인
lsmod | grep -E "overlay|br_netfilter"

# ③ ip_forward 활성화 (1)
cat /proc/sys/net/ipv4/ip_forward            # → 1

# ④ bridge-nf-call-iptables 활성화 (가상 IP→실제 파드 IP 변환)
cat /proc/sys/net/bridge/bridge-nf-call-iptables   # → 1

# ⑤ bridge-nf-call-ip6tables 활성화 (IPv6)
cat /proc/sys/net/bridge/bridge-nf-call-ip6tables  # → 1
```

- ① overlay, ② br_netfilter 모듈이 올라와 있는지
- ③ `/proc`의 IPv4 `ip_forward`가 `1`(패킷 목적지를 넘겨주는 포워딩 활성화)
- ④ `bridge-nf-call-iptables`가 `1`(가상 IP를 실제 파드 IP로 바꿔주는 브릿지 경유)
- ⑤ `bridge-nf-call-ip6tables`도 `1`(IPv6 환경)

이 다섯 가지 중 **단 하나라도 누락되거나 `1`이 아닌 `0`이면**, 컨테이너 생성 시 `CreateContainerError`가 발생하거나 파드 간 통신이 막히는 치명적 장애가 발생합니다.

**실습 정리** — 모든 실습이 끝나면 로컬 터미널에서 Terraform으로 생성된 인프라를 삭제합니다.

```bash
terraform state list                # 모든 리소스가 삭제 대상인지 확인
terraform destroy -auto-approve     # kubeadm 클러스터 전체 삭제
```

삭제하지 않으면 방치된 리소스만큼 AWS에서 비용이 청구되므로, 항상 랩 환경이 끝나면 리소스를 정리해 주시면 되겠습니다.
