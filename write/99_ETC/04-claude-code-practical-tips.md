# Claude Code 실전 활용 10가지 팁
---
> Claude Code를 단순 코드 생성기가 아닌 개발 파트너로 쓰려면, 컨텍스트 관리와 워크플로우 설계가 핵심이다. 이 문서는 중급 사용자를 위한 10가지 실전 팁을 정리한다.

## 1. Second Brain 구축

CLAUDE.md는 프로젝트의 "두 번째 뇌"다. 코딩 컨벤션, 아키텍처 결정, 자주 쓰는 명령어를 여기에 기록하면 매 세션마다 같은 설명을 반복할 필요가 없다. Claude Code는 세션 시작 시 CLAUDE.md를 자동으로 읽기 때문에, 잘 정리된 CLAUDE.md 하나가 프롬프트 수십 줄을 대체한다.

핵심은 "Claude에게 알려줘야 하는 것"과 "코드에서 파악 가능한 것"을 구분하는 일이다. 프레임워크 버전이나 파일 구조는 코드에서 읽으면 되지만, "왜 이 라이브러리를 선택했는지", "커밋 메시지 형식은 무엇인지" 같은 암묵지는 반드시 문서화해야 한다.

## 2. Lazy Loading으로 CLAUDE.md 경량화

CLAUDE.md에 모든 정보를 넣으면 토큰을 낭비한다. 핵심 규칙만 CLAUDE.md에 두고, 상세 내용은 별도 파일로 분리한 뒤 "필요할 때 참조하라"고 경로만 안내하는 방식이 효과적이다. Claude Code는 지시하면 파일을 직접 읽을 수 있으므로, 매 세션마다 전체 문서를 로드할 이유가 없다.

구체적으로는 다음과 같이 구성한다:

- CLAUDE.md에는 진입점 역할의 요약과 참조 경로만 기록
- 상세 규칙은 `.claude/rules/` 하위 파일로 분리
- 프로젝트별 문서는 `specs/projects/` 등 별도 디렉토리에 배치

이 패턴을 적용하면 CLAUDE.md가 수백 줄에서 수십 줄로 줄어들고, 세션 초기 토큰 소비가 크게 감소한다.

## 3. 폴더별 CLAUDE.md 분리

프로젝트 루트뿐 아니라 하위 폴더에도 CLAUDE.md를 배치할 수 있다. Claude Code는 작업 디렉토리 기준으로 가장 가까운 CLAUDE.md를 우선 참조하므로, 폴더별로 해당 도메인에 특화된 지침을 제공하면 컨텍스트 정밀도가 올라간다.

예를 들어 `frontend/CLAUDE.md`에는 React 컴포넌트 규칙을, `backend/CLAUDE.md`에는 API 설계 규칙을 두면 된다. 프론트엔드 작업 시 백엔드 규칙이 컨텍스트를 차지하지 않으므로 응답 품질이 향상된다.

## 4. 한 세션 = 한 기능

하나의 세션에서 여러 기능을 동시에 작업하면 컨텍스트가 오염된다. "로그인 버그 수정"과 "대시보드 신규 기능"을 한 세션에서 처리하면, Claude가 두 작업의 맥락을 혼동할 위험이 높아진다. 세션을 기능 단위로 분리하면 다음과 같은 이점이 있다:

- 컨텍스트 윈도우를 한 가지 목표에 집중시킬 수 있다
- 커밋 히스토리가 깔끔하게 유지된다
- 문제가 생겼을 때 해당 세션만 되돌리면 된다

## 5. MCP 토큰 관리

MCP(Model Context Protocol) 서버는 강력하지만, 활성화된 MCP마다 도구 설명이 컨텍스트에 포함되어 수천 토큰을 소비한다. Notion, Linear, Jira 같은 MCP를 전부 켜놓으면 정작 코드 작성에 쓸 토큰이 줄어든다.

사용하지 않는 MCP는 `claude mcp disable`로 비활성화하고, 자주 쓰는 기능만 골라 커스텀 MCP로 래핑하면 토큰을 절약하면서 응답 품질도 높일 수 있다. `claude mcp list`로 현재 활성 상태를 주기적으로 점검하는 습관이 필요하다.

## 6. 무거운 작업은 스크립트로 오프로드

린트, 테스트, 빌드 같은 반복 작업을 Claude에게 직접 시키면 토큰과 시간을 낭비한다. 이런 작업은 쉘 스크립트나 Makefile로 오프로드하고, Claude에게는 "이 스크립트를 실행해서 결과를 분석해줘"라고 지시하는 편이 효율적이다.

Claude Code의 Bash 도구는 스크립트 실행 결과를 그대로 읽을 수 있으므로, 무거운 연산은 외부에 맡기고 Claude는 결과 해석과 의사결정에 집중시키는 구조가 이상적이다.

## 7. Plan 모드 먼저

복잡한 기능을 구현할 때 바로 코드 작성에 들어가면 방향을 잃기 쉽다. Plan 모드(`Shift+Tab`)를 먼저 활용해 전체 구조를 설계하고, 합의된 계획을 바탕으로 구현에 들어가면 재작업을 줄일 수 있다.

Plan 모드에서 Claude는 코드를 수정하지 않고 분석과 설계에만 집중한다. 이 단계에서 파일 구조, 의존성, 예외 케이스를 미리 정리하면 구현 단계에서 "아, 이건 고려 안 했네" 같은 상황을 예방할 수 있다.

## 8. TDD 기반 개발

테스트를 먼저 작성하고 Claude에게 구현을 맡기면 두 가지 이점이 있다. 첫째, 테스트가 요구사항 명세서 역할을 하므로 Claude가 의도를 정확히 파악한다. 둘째, 구현 후 즉시 검증할 수 있어 피드백 루프가 짧아진다.

Claude Code는 테스트 실행 결과를 읽고 스스로 수정할 수 있다. "테스트 통과할 때까지 고쳐줘"라는 지시 한 줄이면 Red-Green-Refactor 사이클을 자동으로 돌릴 수 있다.

## 9. 에러 로그 원본 전달

에러가 발생했을 때 "로그인이 안 돼요"라고 설명하는 것보다, 스택트레이스를 그대로 붙여넣는 편이 훨씬 효과적이다. Claude는 스택트레이스 분석에 특히 강하기 때문에, 사용자가 해석하는 과정에서 정보가 누락되거나 왜곡되면 오히려 디버깅이 어려워진다.

에러 보고 시 지켜야 할 원칙은 간단하다:

- 스택트레이스/로그를 해석하지 말고 원본 그대로 전달
- 에러가 발생한 재현 조건(어떤 입력, 어떤 순서)을 함께 제공
- "이것 같다"는 추측보다 팩트(로그 원문)가 우선

## 10. WAT 프레임워크

Claude에게 효과적으로 지시하려면 WAT 프레임워크를 활용한다:

- **What**: 무엇을 원하는지 구체적으로 명시
- **Avoid**: 하지 말아야 할 것을 명확히 제한
- **Target**: 완료 조건을 검증 가능한 형태로 정의

"로그인 기능 만들어줘" 대신 "JWT 기반 로그인 API를 만들어줘(What). 세션 방식은 사용하지 마(Avoid). `/api/auth/login` POST 요청 시 JWT 토큰이 반환되면 완료(Target)"라고 지시하면 Claude가 의도에 맞는 코드를 한 번에 생성할 확률이 높아진다.

## 내 환경 적용 현황

| 팁 | 적용 상태 | 구현 위치 | 비고 |
|----|----------|----------|------|
| Second Brain | 적용 완료 | `CLAUDE.md` | Lazy Loading 구조로 운영 |
| Lazy Loading | 적용 완료 | `CLAUDE.md` + `.claude/rules/` | 규칙 파일 분리 완료 |
| 폴더별 CLAUDE.md | 미적용 | - | 프로젝트 규모에 따라 검토 |
| 한 세션 = 한 기능 | 적용 중 | 워크플로우 습관 | `workflow.md`에 명문화 |
| MCP 토큰 관리 | 신규 적용 | `.claude/rules/workflow.md` | 이번 세션에서 규칙 추가 |
| 스크립트 오프로드 | 부분 적용 | Makefile, 쉘 스크립트 | 프로젝트별 상이 |
| Plan 모드 | 적용 중 | `Shift+Tab` 활용 | 복잡한 작업에 선택적 사용 |
| TDD 기반 개발 | 부분 적용 | 프로젝트별 상이 | 학습 프로젝트에서 활용 |
| 에러 로그 원본 전달 | 신규 적용 | `.claude/rules/workflow.md` | 이번 세션에서 규칙 추가 |
| WAT 프레임워크 | 미적용 | - | 프롬프트 습관으로 도입 예정 |
