# 하네스 엔지니어링 — AI 코딩 에이전트의 마구(馬具)를 설계하는 기술
---
> AI 코딩 에이전트가 볼 수 없는 것은 존재하지 않는다. 하네스 엔지니어링은 에이전트가 보는 세계를 설계하는 엔지니어링이다.

## 1. 하네스란 무엇인가

말의 힘을 제어하고 방향을 잡아주는 도구를 마구(harness)라고 부른다. AI 코딩 에이전트에도 같은 장치가 필요하다. Claude Code, Codex CLI, Gemini CLI 같은 도구는 코드를 읽고 쓰는 능력이 있지만, 그 능력을 올바른 방향으로 유도하지 않으면 쓸모없는 코드를 양산하거나 아키텍처를 망가뜨린다.

**하네스 엔지니어링(Harness Engineering)**은 레포지토리 구조, 설정 파일, 훅, 도구 체인을 통해 에이전트의 컨텍스트와 행동을 제어하는 분야다. 용어의 기원은 OpenAI Codex 팀이 대규모 자동 코딩을 수행하면서 발견한 통찰에 있다. 그들이 정리한 핵심 원칙은 한 문장으로 요약된다:

> "What Codex can't see doesn't exist."

Slack에 있는 아키텍처 결정, 엔지니어 머릿속 컨벤션, Google Docs의 설계 의도 — 에이전트는 이것들을 볼 수 없으므로 없는 것과 같다. 하네스 엔지니어링의 출발점은 이 보이지 않는 지식을 레포지토리 안의 검증 가능한 구조물로 인코딩하는 작업이다.

## 2. 왜 필요한가

LLM 기반 코딩 에이전트에는 본질적 한계가 있다. 컨텍스트 윈도우는 유한하고, 환각(hallucination)이 발생하며, 도메인 지식이 부족하고, 세션 간 상태를 기억하지 못한다. 이런 한계는 프롬프트를 아무리 정교하게 작성해도 해결되지 않는다.

Andrej Karpathy는 context engineering이라는 용어를 만들면서 이 문제를 직접 겪었다고 밝혔다:

> "The agents do not listen to my instructions... no matter how carefully you write your AGENTS.md."

산문으로 쓴 지시사항은 신뢰할 수 없다. 에이전트는 때로 무시하고, 때로 잘못 해석한다. 하네스 엔지니어링은 산문 대신 기계적 강제(mechanical enforcement)로 에이전트의 행동을 제어한다. pre-commit hook이 잘못된 커밋을 차단하고, ast-grep 규칙이 금지된 패턴을 감지하며, import linter가 아키텍처 위반을 빌드 단계에서 잡아낸다.

| LLM 한계 | 하네스 해법 | 구현 예시 |
|----------|-----------|----------|
| 컨텍스트 윈도우 제한 | 메모리 관리 | `MEMORY.md`, 세션 wrap, compact |
| 환각 | 팩트 그라운딩 | `.claude/rules/`, MCP grounding |
| 도메인 지식 부재 | 지식 베이스 | `specs/`, `skills/`, `agents/` |
| 상태 관리 불가 | 세션 오케스트레이션 | TaskCreate, `plans/` |
| 외부 시스템 접근 불가 | 도구 체인 | MCP 서버 (Jira, Chrome 등) |

## 3. 다섯 가지 실천법 (2026년 3월 기준)

### 3-1. 레포지토리가 기록 시스템이다

에이전트가 알아야 할 모든 것이 레포 안에 존재해야 한다. `CLAUDE.md`가 프로젝트 불변 원칙을 담고, `.claude/rules/`가 경로별 규칙을 정의하며, `.claude/skills/`가 도메인별 능력을 확장한다. 팀의 코딩 컨벤션, 아키텍처 결정의 이유, 배포 절차 같은 암묵지가 문서화되지 않으면 에이전트는 매번 다른 판단을 내린다.

### 3-2. 산문보다 기계적 강제

"이렇게 해주세요"보다 "이것 아니면 빌드 실패"가 더 확실하다. 커뮤니티에서 표준화된 Six Gates 모델은 에이전트의 모든 변경이 통과해야 하는 6단계 품질 게이트를 정의한다:

- **Gate A** — Formatting + Lint: 코드 형식 자동 교정
- **Gate B** — Import Boundaries: 모듈 간 의존성 규칙 위반 차단
- **Gate C** — Structural Ratchets: ast-grep으로 금지된 코드 패턴 감지
- **Gate D** — Snapshot Testing: 동작 변경을 스냅샷으로 잠금
- **Gate E** — Golden Outputs: 기대 출력과 실제 출력 비교
- **Gate F** — Numerical Equivalence: 수치 결과의 동등성 검증

이 게이트들은 CI/CD 파이프라인에 통합되어 에이전트가 생성한 코드를 사람이 매번 검토하지 않아도 최소 품질을 보장한다. 핵심은 에이전트에게 "하지 마"라고 말하는 대신 "하면 빌드가 깨진다"는 환경을 만드는 것이다.

### 3-3. 점진적 공개

`CLAUDE.md`는 지도(map)이지, 백과사전이 아니다. Claude Code 공식 문서는 200줄 이하를 권장하며, 그 이상은 준수율이 떨어진다고 명시한다. 핵심 규칙만 최상위 파일에 두고, 상세 내용은 별도 파일로 분리한 뒤 "필요할 때 참조하라"고 경로만 안내하는 구조가 효과적이다.

Claude Code에서 이 패턴은 세 가지 메커니즘으로 구현된다:

- `CLAUDE.md`: 진입점 역할의 요약과 참조 경로 (200줄 이하)
- `.claude/rules/`: YAML frontmatter의 `paths` 필드로 특정 파일 작업 시에만 로드
- `.claude/skills/SKILL.md`: 50줄 수준의 파사드, `references/` 하위에 상세 문서 배치

Vercel이 이 접근법의 효과를 실증했다. 40KB의 수동 컨텍스트 파일 대신, 8KB 인덱스와 검색 기반 로드로 100% 태스크 성공률을 달성했다. 에이전트에게 많이 보여주는 것이 아니라, 필요한 것만 정확히 보여주는 것이 핵심이다.

### 3-4. 증거 기반 작업 단위

변경은 단순 diff가 아니라 **청크(chunk)**로 머지된다. 하나의 청크는 세 가지 요소로 구성된다:

- 코드 변경 자체
- 해당 변경을 보호하는 강제 규칙 (테스트, 린트 규칙 등)
- 변경이 올바르다는 증거 산출물 (테스트 결과, 스냅샷, 골든 diff)

이 세 가지가 함께 커밋되어야 변경이 자기 완결적(self-contained)이 된다. 코드만 커밋하고 테스트를 나중에 추가하면, 그 사이에 에이전트가 다른 변경을 쌓아올릴 수 있다. 증거 없는 변경은 나중에 회귀를 일으킬 때 원인 추적이 어렵다.

### 3-5. 훅이 제어 평면이다

Claude Code는 24개 이상의 라이프사이클 훅 이벤트를 제공한다. `PreToolUse`는 도구 호출 전에 차단하거나 수정할 수 있고, `PostToolUse`는 도구 실행 후 자동 포맷이나 검증을 수행하며, `SessionStart`는 세션별 환경을 초기화한다. 핸들러 타입은 command(쉘 스크립트), http(POST 엔드포인트), prompt(Claude 판단), agent(서브에이전트) 네 가지다.

훅의 존재가 에이전트를 자율적 행위자에서 감독되는 프로세스로 전환한다. 에이전트가 파일을 쓰기 전에 훅이 경로를 검증하고, 커밋 전에 메시지 형식을 확인하며, 세션 종료 시 작업 상태를 자동 저장한다. 이것은 전통적인 미들웨어 체인과 같은 역할을 에이전트 실행 루프에 적용한 것이다.

## 4. 전통 소프트웨어 아키텍처와의 관계

하네스 엔지니어링은 새로운 발명이 아니다. 기존 소프트웨어 설계 원칙을 에이전트 도메인에 재적용한 것에 가깝다.

| 전통 계층 | 에이전트 대응물 | 역할 |
|----------|---------------|------|
| Controller | 슬래시 커맨드 | 사용자 요청 진입점, 의도 라우팅 |
| Service Layer | 서브에이전트 | 여러 스킬을 워크플로우로 오케스트레이션 |
| Domain Component | Skills (`SKILL.md`) | 단일 책임의 재사용 가능한 기능 단위 |
| Infrastructure Adapter | MCP 서버 | 외부 시스템 추상화 (Repository 패턴) |
| `package.json` | `CLAUDE.md` | 프로젝트 설정과 의존성 선언 |

SOLID 원칙도 그대로 적용된다. 하나의 스킬은 하나의 책임만 갖고(SRP), 스킬 간 의존은 추상화를 통해 이루어지며(DIP), 새로운 기능은 기존 스킬을 수정하지 않고 확장으로 추가한다(OCP). 에이전트에게 500줄짜리 God Skill을 주면 전통 코드에서 God Class를 만들었을 때와 같은 문제가 발생한다 — 컨텍스트가 흐려지고, 수정 시 예상치 못한 부작용이 생긴다.

## 5. 회의론과 균형

하네스 엔지니어링이 만능은 아니다. 2026년 2월 ETH Zurich 연구는 컨텍스트 파일이 오히려 태스크 성공률을 낮출 수 있다는 결과를 발표했다. 인간이 작성한 컨텍스트 파일도 약 4% 개선에 그쳤고, 일부 모델에서는 오히려 2% 이상 성능이 하락했다. 추론 비용은 20% 이상 증가했다.

Augment Code는 이 연구를 바탕으로 대부분의 `CLAUDE.md`가 "정크 서랍(junk drawer)"이라고 비판했다. 그들이 제시한 진단 기준은 명확하다. 컨텍스트 파일에 담긴 규칙이 다음 네 가지 중 하나에 해당하지 않으면 제거해야 한다:

- **실패 기반(Failure-backed)**: 실제 발생한 문서화된 실수를 방지하는 규칙
- **도구 강제 가능(Tool-enforceable)**: 린터, 훅, CI로 자동 검증 가능한 규칙
- **의사결정 인코딩(Decision-encoding)**: 여러 선택지 중 하나를 확정한 기록
- **트리거 가능(Triggerable)**: 특정 조건에서만 활성화되는 규칙

코드 구조, 의존성 트리, Git 히스토리처럼 에이전트가 직접 코드를 읽어 파악할 수 있는 정보를 컨텍스트 파일에 중복 기재하면 노이즈가 된다. 에이전트가 볼 수 없는 것만 — 배포 절차, 팀 컨벤션, 환경별 설정, 아키텍처 결정의 이유 — 하네스에 인코딩해야 한다.

## 6. 그래서 어떻게 시작하는가

하네스 엔지니어링을 처음 도입한다면 세 단계로 접근하는 것이 현실적이다.

첫째, `CLAUDE.md`를 200줄 이하로 작성한다. 프로젝트의 목적, 기술 스택, 핵심 컨벤션, 자주 실수하는 패턴만 담는다. 코드에서 읽을 수 있는 정보는 넣지 않는다.

둘째, 반복되는 실수를 규칙으로 전환한다. 코드 리뷰에서 같은 피드백이 3~4번 나오면 린트 규칙이나 `.claude/rules/` 파일로 기계화한다. Boris Cherny(Claude Code 창시자)가 실제로 사용한 방법이기도 하다.

셋째, 훅으로 자동화한다. `PreToolUse`로 위험한 도구 호출을 차단하고, `PostToolUse`로 자동 포맷을 적용하며, `SessionStart`로 세션 환경을 초기화한다. 이 세 가지 훅만으로도 에이전트의 행동 범위를 상당히 제한할 수 있다.

하네스 엔지니어링의 핵심은 결국 에이전트가 성공할 수밖에 없는 환경을 만드는 것이다. 좋은 지시를 내리는 것이 아니라, 잘못된 행동이 물리적으로 불가능한 구조를 설계하는 것. 이것이 2026년 AI 코딩 에이전트를 운영하는 엔지니어에게 요구되는 새로운 역량이다.
