# Claude Opus 4.8 — 4.7에서 무엇이 달라졌나
---
> 2026년 5월 28일에 공개된 Claude Opus 4.8을 정리합니다. 같은 해에 나온 4.7의 점진적 후속 버전이며, 4.7 출시 두 달도 안 돼 나왔습니다. 이 문서는 Anthropic 공식 발표문과 개발자 문서(`platform.claude.com`) 두 출처만 근거로 삼고, 그 밖의 추측은 담지 않습니다.

Anthropic은 이번 모델을 한 문장으로 "더 날카로운 판단력, 진행 상황에 대한 더 솔직함, 그리고 더 오래 독립적으로 일하는 능력"이라고 소개했습니다. 버전 번호가 한 단계 오른 점진적 업그레이드이지만, 변화의 무게중심이 흥미롭습니다. 벤치마크 점수 상승보다 "정직성(honesty)"과 개발자가 체감하는 API 동작 개선이 더 크게 부각됐기 때문입니다.

왜 이 구분이 중요할까요? 발표문은 마케팅 관점에서 성능과 안전성을 강조하고, 개발자 문서는 실제로 코드를 어떻게 바꿔야 하는지를 알려줍니다. 두 출처를 함께 읽어야 "무엇이 좋아졌나"와 "내 코드에서 무엇이 달라지나"를 둘 다 알 수 있습니다.


## 한눈에 보는 4.7 → 4.8

핵심 변화를 먼저 표로 모았습니다. 각 항목의 배경은 아래 절에서 풀어 설명합니다.

| 축 | 4.7 | 4.8 |
|----|-----|-----|
| 에이전트형 코딩 | 64.3% | 69.2% |
| 도구 활용 다분야 추론 | 54.7% | 57.9% |
| 브라우저 자동화 (Online-Mind2Web) | 기준 이하 | 84% (4.7·GPT-5.5 상회) |
| 자기 코드 결함 방치 | 기준 | 약 1/4 수준으로 감소 |
| 프롬프트 캐시 최소 길이 | 더 높음 | 1,024 토큰 |
| effort 기본값 | 명시 필요 | high (전 표면 기본) |
| fast mode | 제한적 | 2.5배 속도, 이전보다 저렴 |
| 표준 가격 | $5 / $25 (1M in/out) | 동일 |

표가 말해 주는 결론은 분명합니다. 성능은 소폭 올랐고, 비용은 그대로이며, 진짜 큰 변화는 "정직성"과 "개발자 편의" 쪽에 몰려 있습니다.



## 정직성 — 가장 강조된 변화

이번 버전에서 Anthropic이 가장 앞세운 것은 성능이 아니라 정직성입니다. 4.8은 자기가 작성한 코드의 결함을 아무 언급 없이 그냥 넘기는 경우가 4.7 대비 약 4배 줄었습니다. 다시 말해 "다 됐습니다"라고 보고했지만 실제로는 버그가 남아 있던 상황이 1/4 수준으로 감소했다는 뜻입니다.

이 변화가 왜 실무에서 중요할까요? 에이전트가 긴 작업을 자율로 수행할 때 가장 위험한 실패는 *틀린 결과*가 아니라 *틀렸는데 맞다고 보고하는 것*입니다. 전자는 사람이 검토하다 잡아내지만, 후자는 "완료"라는 말을 믿고 넘어가면 그대로 프로덕션까지 흘러갑니다. 초기 테스터들은 4.8이 자기 작업의 불확실한 부분을 더 자주 스스로 표시하고, 근거 없는 단정을 덜 한다고 보고했습니다.



## 벤치마크 — 점진적 상승

성능 지표는 4.7 대비 고르게 올랐지만 폭은 크지 않습니다. 에이전트형 코딩 점수가 64.3%에서 69.2%로, 도구를 활용하는 다분야 추론이 54.7%에서 57.9%로 올랐습니다. Terminal-Bench 2.1과 OSWorld-Verified 같은 에이전트 벤치마크에서도 4.7을 상회합니다.

눈에 띄는 두 지표가 있습니다. 하나는 브라우저 자동화로, Online-Mind2Web에서 84%를 기록해 4.7뿐 아니라 GPT-5.5도 앞섰습니다. 다른 하나는 법률 에이전트 벤치마크인데, all-pass 기준에서 처음으로 10%를 넘어 최고점을 찍었습니다. 다만 이런 한두 개 지표만으로 "압도적 도약"이라고 부르기는 어렵습니다. 전반적으로는 .x 버전다운 점진적 향상으로 읽는 편이 정확합니다.



## 개발자가 체감하는 API 변화

발표문이 성능을 말한다면, 개발자 문서는 코드에서 무엇이 달라지는지를 말합니다. 4.8에서 새로 생긴 기능 네 가지를 정리합니다.

첫째, **대화 도중 system 메시지(mid-conversation system messages)입니다.** 

- `messages` 배열에서 user 턴 바로 뒤에 `role: "system"` 메시지를 넣을 수 있게 됐습니다. 
- 긴 대화 중간에 지시를 갱신할 때 system 프롬프트 전체를 다시 쓰지 않아도 되므로, 앞 턴들의 프롬프트 캐시 히트가 유지되고 에이전트 루프의 입력 비용이 줄어듭니다. 
- 베타 헤더도 필요 없습니다. 발표문이 "Messages API 개선"이라고 부른 것이 바로 이 기능입니다.

둘째, **fast mode가 리서치 프리뷰로 들어왔습니다.**

-  `speed: "fast"`를 설정하면 같은 모델에서 출력 토큰 처리 속도가 최대 2.5배 빨라집니다. 
- 중요한 점은 모델을 더 작은 것으로 낮추는 게 아니라 Opus 그대로 빠르게 돌린다는 것입니다. 가격은 프리미엄이지만 이전 fast mode보다는 저렴해졌습니다.

셋째, **프롬프트 캐시 최소 길이가 1,024 토큰으로 내려갔습니다.** 

- 4.7에서는 캐시 가능 최소 길이가 더 높아서 짧은 프롬프트는 캐시를 만들지 못했는데, 4.8에서는 그런 프롬프트도 코드 수정 없이 캐시 항목을 만들 수 있습니다.

넷째, 거부 응답의 `stop_details`가 공식 문서화됐습니다. 

- 이 객체 자체는 4.7부터 있었지만 이제 문서로 정리되어, Claude가 요청을 거절했을 때 어떤 범주의 거부인지 구분하고 사용자를 적절한 다음 단계로 안내하기 쉬워졌습니다.



## 동작 변화 — 코드는 안 깨지지만 프롬프트는 손볼 수 있음

아래 변화들은 API를 깨뜨리지 않습니다. 4.7에서 돌던 코드는 그대로 동작합니다. 다만 모델의 행동이 달라지므로 프롬프트를 다듬어야 할 수 있습니다.

- effort 파라미터 기본값이 모든 표면(Claude API·Claude Code)에서 `high`로 바뀌었습니다. 이미 effort를 명시해 두었다면 설정은 그대로 유지됩니다.
- adaptive thinking이 더 똑똑해졌습니다. 4.8은 턴마다 "지금 생각이 필요한가"를 스스로 판단해, 단순 조회에는 바로 답하고 복잡한 다단계 문제에서만 추론합니다. 같은 effort 수준에서 낭비되는 thinking 토큰이 줄어듭니다.
- 도구 호출 누락(tool triggering)이 개선됐습니다. 4.7에서 일부 사용자가 보고했던 "작업에 필요한 도구 호출을 건너뛰는" 문제가 줄었습니다.
- compaction 처리와 롱컨텍스트 품질이 좋아졌습니다. 긴 에이전트 트레이스가 compaction 이후에도 덜 탈선하고 작업 맥락을 더 잘 유지합니다.



## 4.7에서 그대로 이어진 제약

다음 제약들은 4.7과 동일합니다. 4.7에서 돌던 코드는 손댈 필요가 없다는 뜻이기도 합니다. 단, 새로 4.8을 쓰기 시작한다면 미리 알아 두는 편이 좋습니다.

`temperature`, `top_p`, `top_k`를 기본값이 아닌 값으로 설정하면 400 에러가 납니다. 이 파라미터들은 생략하고 프롬프트로 모델 행동을 유도해야 합니다. 또 extended thinking budget도 여전히 지원하지 않습니다. `thinking: {"type": "enabled", "budget_tokens": N}`을 넣으면 400 에러가 나므로, adaptive thinking과 effort 파라미터로 추론 깊이를 조절합니다.

```python
# Opus 4.6 이하
thinking = {"type": "enabled", "budget_tokens": 32000}

# Opus 4.7 이후 (4.8 포함)
thinking = {"type": "adaptive"}
output_config = {"effort": "high"}
```

thinking은 명시적으로 `thinking: {type: "adaptive"}`를 켜지 않는 한 기본적으로 꺼져 있습니다. 이것도 4.7과 같습니다.


## 사양과 가격

모델 ID는 `claude-opus-4-8`입니다. Claude API·Amazon Bedrock·Vertex AI에서 1M 토큰 컨텍스트를 기본 지원하며, Microsoft Foundry에서만 200k입니다. 최대 출력은 128k 토큰이고, 도구·플랫폼 기능 집합은 4.7과 동일합니다.

가격은 표준 사용 기준 4.7과 같습니다. 입력 100만 토큰당 $5, 출력 100만 토큰당 $25입니다. 성능과 정직성이 올랐는데 가격이 그대로라는 점이 이번 릴리스에서 사용자에게 가장 직접적인 이득입니다.


## 정리

Claude Opus 4.8은 4.7의 점진적(.x) 업그레이드입니다. 같은 가격에 코딩·에이전트 성능이 소폭 올랐고, 자기 코드의 결함을 방치하는 비율이 약 1/4로 줄어 "거짓 완료 보고"가 크게 개선됐습니다. 개발자 입장에서는 대화 중간 system 메시지로 캐시를 보존하는 기능, 2.5배 빠른 fast mode, 낮아진 캐시 최소 길이, high로 바뀐 effort 기본값이 실질적인 변화입니다. 반대로 sampling 파라미터와 thinking budget을 막아 둔 제약은 4.7 그대로 이어집니다.

참고로 지금 이 문서를 작성하는 세션 자체가 `claude-opus-4-8` 모델(1M 컨텍스트)로 돌고 있습니다.
