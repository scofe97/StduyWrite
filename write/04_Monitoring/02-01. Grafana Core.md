# Grafana Core

---

> Grafana는 LGTM 스택의 시각화 계층이다. 메트릭, 로그, 트레이스를 직접 저장하지 않고 Prometheus/Mimir, Loki, Tempo에 쿼리를 보내 대시보드로 통합 표현한다. v12(2025년 기준)에서는 Git Sync, Dynamic Dashboards, Drilldown GA로 observability-as-code 흐름이 완성됐다.

## 1. 관측 스택에서의 역할

Grafana는 데이터를 저장하지 않는다. 이 사실이 Grafana 아키텍처를 이해하는 핵심이다. Prometheus가 메트릭을 저장하고, Loki가 로그를 저장하고, Tempo가 trace를 저장한다. Grafana는 이 세 저장소에 쿼리를 보내고, 결과를 하나의 화면으로 합쳐 보여주는 시각화 계층이다.

LGTM 스택에서 Grafana의 위치는 다음과 같다:

- **Loki**: 로그 저장소 (LogQL로 쿼리)
- **Grafana**: 시각화 및 알림 계층 (data source로 LGTM 연결)
- **Tempo**: 분산 트레이스 저장소 (TraceQL로 쿼리)
- **Mimir/Prometheus**: 메트릭 저장소 (PromQL로 쿼리)

이 구조 덕분에 Grafana 하나로 메트릭, 로그, 트레이스를 단일 화면에서 교차 탐색할 수 있다. 예를 들어 Prometheus 메트릭에서 에러율이 급증한 지점을 클릭하면 그 시점의 Loki 로그로 바로 점프할 수 있다.

## 2. Data Source

Data source는 Grafana가 외부 저장소에서 데이터를 읽는 연결 설정이다. Grafana는 data source를 통해서만 데이터에 접근하며, 자체적으로 데이터를 보유하지 않는다.

주요 data source 유형은 세 가지다:

- **Prometheus/Mimir**: PromQL 기반 메트릭 조회. Exemplar를 활성화하면 메트릭에서 trace로 이동 가능하다.
- **Loki**: LogQL 기반 로그 조회. Derived Fields 설정으로 로그에서 trace로 이동 가능하다.
- **Tempo**: TraceQL 기반 trace 조회. tracesToLogs 설정으로 trace에서 로그로 이동 가능하다.

Data source는 UI에서 직접 추가할 수도 있지만, Helm 배포 환경에서는 YAML 프로비저닝이 표준이다. `values.yaml`에 다음과 같이 선언한다:

```yaml
grafana:
  datasources:
    datasources.yaml:
      apiVersion: 1
      datasources:
        - name: Prometheus
          type: prometheus
          url: http://prometheus-operated:9090
          isDefault: true
          jsonData:
            exemplarTraceIdDestinations:
              - name: traceID
                datasourceUid: tempo

        - name: Loki
          type: loki
          url: http://loki:3100
          jsonData:
            derivedFields:
              - name: TraceID
                matcherRegex: '"traceId":"(\w+)"'
                url: "$${__value.raw}"
                datasourceUid: tempo

        - name: Tempo
          type: tempo
          url: http://tempo:3100
          jsonData:
            tracesToLogs:
              datasourceUid: loki
              tags: ["pod", "namespace"]
              filterByTraceID: true
```

### 교차 신호 탐색

세 data source를 연결하면 신호 간 이동이 가능해진다. 이것을 **교차 신호 탐색**이라 한다:

- **Derived Fields (Loki → Tempo)**: 로그의 `traceId` 필드를 파싱해 Tempo로 이동하는 링크를 자동 생성한다.
- **tracesToLogs (Tempo → Loki)**: trace span에서 동일 시간대의 로그를 Loki에서 조회하는 링크를 제공한다.
- **Exemplar (Prometheus → Tempo)**: 메트릭 데이터 포인트에 `trace_id`를 첨부하여 Tempo로 이동 가능하게 한다.

## 3. Dashboard

대시보드는 여러 패널로 구성된 시각화 단위다. 각 패널은 하나의 쿼리 결과를 특정 시각화 유형으로 표현한다.

주요 패널 유형은 다음과 같다:

| 패널 유형 | 용도 | 주요 data source |
|-----------|------|-----------------|
| Time series | 시간에 따른 메트릭 변화 | Prometheus |
| Stat | 단일 숫자 값 (현재 값, 평균 등) | Prometheus, Loki |
| Gauge | 범위 내 현재 값 (CPU 사용률 등) | Prometheus |
| Table | 다차원 데이터 테이블 표현 | 모든 data source |
| Logs | 로그 스트림 표시 | Loki |
| Node Graph | 서비스 의존성 그래프 | Tempo |
| Traces | trace 타임라인 시각화 | Tempo |
| Heatmap | 분포 히트맵 (레이턴시 p50/p95 등) | Prometheus |

### 변수와 템플릿

**Variables**는 대시보드를 동적으로 만드는 핵심 기능이다. 변수를 선언하면 상단에 드롭다운이 생기고, 선택한 값이 쿼리 파라미터로 삽입된다.

자주 쓰는 변수 패턴은 다음과 같다:

- `$namespace`: Prometheus에서 `label_values(kube_pod_info, namespace)` 쿼리로 namespace 목록을 자동 조회한다.
- `$pod`: `label_values(kube_pod_info{namespace="$namespace"}, pod)` 처럼 다른 변수를 참조할 수 있다.
- `$interval`: 집계 간격을 동적으로 변경한다 (`1m`, `5m`, `1h`).

### Dashboard JSON 모델

Grafana 대시보드는 내부적으로 JSON으로 표현된다. 기본 구조는 다음과 같다:

```json
{
  "title": "Service Overview",
  "uid": "abc123",
  "schemaVersion": 39,
  "templating": {
    "list": [
      {
        "name": "namespace",
        "type": "query",
        "query": "label_values(kube_pod_info, namespace)"
      }
    ]
  },
  "panels": [
    {
      "type": "timeseries",
      "title": "Request Rate",
      "targets": [
        {
          "expr": "rate(http_requests_total{namespace=\"$namespace\"}[5m])"
        }
      ]
    }
  ]
}
```

이 JSON을 Git으로 관리하면 대시보드 변경 이력을 추적하고, 환경별로 동일한 대시보드를 재현할 수 있다.

## 4. Grafana v12 주요 기능

Grafana v12(2025년 출시)는 observability-as-code를 향한 여러 기능을 GA로 제공했다.

**Git Sync (GA)**는 대시보드를 GitHub 리포지토리와 양방향 동기화한다. UI에서 대시보드를 수정하면 자동으로 PR이 생성되고, Git 변경이 자동으로 Grafana에 반영된다. 인프라 변경을 Git으로 관리하는 팀에게 대시보드도 코드로 다루는 경험을 제공한다.

**Dynamic Dashboards**는 탭, 자동 그리드 레이아웃, 조건부 로직을 지원한다. 기존에는 각 서비스마다 별도 대시보드를 만들어야 했지만, 이제 변수 값에 따라 패널 레이아웃이 동적으로 바뀐다.

**Drilldown (GA)**는 쿼리 없이 메트릭, 로그, 트레이스를 탐색하는 인터페이스다. 기존의 Explore Metrics/Logs/Traces를 통합하고 정식 기능으로 승격됐다. PromQL이나 LogQL을 모르는 팀원도 클릭만으로 이상 징후를 탐색할 수 있다.

**SQL Expressions (preview)**는 서로 다른 data source의 쿼리 결과를 SQL로 결합한다. Prometheus 메트릭과 Loki 집계 결과를 JOIN하는 식의 교차 data source 분석이 가능해진다.

**Table Panel 재작성**은 성능 측면에서 가장 실용적인 변화다. 기존 테이블은 대량 데이터에서 느렸지만, `react-data-grid` 기반으로 재작성하면서 40,000행 이상에서 CPU 사용률이 97.8% 개선됐다.

## 5. Unified Alerting

Grafana의 알림 시스템은 v8부터 Unified Alerting으로 통합됐다. 외부 Alertmanager 없이 Grafana 자체에서 alert rule 평가, 라우팅, 발송을 처리한다.

알림 흐름은 다음 단계로 진행된다:

```
Alert Rule → Evaluation → Contact Point → Notification Policy → (Silence)
```

### Alert Rule 구성 요소

Alert Rule 하나는 네 가지 요소로 구성된다:

- **쿼리**: PromQL, LogQL 등 평가할 쿼리 표현식
- **조건**: 쿼리 결과가 임계값을 넘으면 Firing 상태로 전환하는 조건
- **폴더**: alert rule을 관리하는 논리적 그룹
- **Evaluation interval**: 몇 초/분마다 쿼리를 실행할지 결정하는 주기 (예: `1m`)

### Contact Point

Contact Point는 알림을 발송하는 채널이다. 지원하는 유형은 다음과 같다:

- Email: SMTP 설정이 필요하며 기본 제공된다.
- Slack: Webhook URL 또는 Bot Token으로 연결한다.
- PagerDuty: Integration Key로 incident를 자동 생성한다.
- Webhook: 임의의 HTTP endpoint로 JSON payload를 전송한다.

### Notification Policy

Notification Policy는 alert를 어떤 Contact Point로 보낼지 결정하는 라우팅 트리다. Label matchers를 이용해 조건을 정의하고, 조건에 맞는 alert를 특정 Contact Point로 분배한다. 예를 들어 `team=backend` label이 붙은 alert는 백엔드 Slack 채널로, `severity=critical`인 alert는 PagerDuty로 보내는 식이다.

**Silence**는 특정 기간 동안 alert 발송을 억제한다. 정기 유지보수 시간에 alert 폭주를 막을 때 유용하다. Silence는 alert 자체를 끄지 않고 발송만 막으므로, 유지보수가 끝나면 Silence가 자동으로 해제된다.

v12에서는 **Recovering** 상태가 추가됐다. 기존에는 Firing/Normal 두 상태만 있어서, 메트릭이 임계값 근처에서 진동하면 alert가 켜졌다 꺼졌다를 반복하는 flapping이 발생했다. Recovering 상태는 Firing에서 Normal로 전환되는 중간 단계를 표현하며, 안정화될 때까지 발송을 유보한다.

## 6. 프로비저닝

Grafana의 프로비저닝은 대시보드, data source, alert rule을 파일로 선언하면 Grafana가 기동 시 자동으로 로드하는 기능이다. UI에서 수동으로 설정한 내용이 재기동 시 사라지는 문제를 방지하고, GitOps 워크플로우와 자연스럽게 통합된다.

**Dashboard 프로비저닝**은 YAML 설정과 JSON 파일 두 단계로 구성된다. YAML은 "이 경로의 JSON 파일을 이 폴더에 로드하라"는 지시를 담고, JSON이 실제 대시보드 내용이다:

```yaml
# /etc/grafana/provisioning/dashboards/default.yaml
apiVersion: 1
providers:
  - name: default
    type: file
    options:
      path: /var/lib/grafana/dashboards
      foldersFromFilesStructure: true
```

**Alerting 프로비저닝**은 alert rule, contact point, notification policy를 YAML로 선언한다. 이 파일을 Git에서 관리하면 알림 설정 변경도 코드 리뷰를 거칠 수 있다:

```yaml
# /etc/grafana/provisioning/alerting/rules.yaml
apiVersion: 1
groups:
  - orgId: 1
    name: infrastructure
    folder: Alerts
    interval: 1m
    rules:
      - title: HighCPU
        condition: C
        data:
          - refId: A
            queryType: ""
            model:
              expr: avg(rate(node_cpu_seconds_total{mode!="idle"}[5m])) > 0.8
```

**GitOps 연동**은 ArgoCD 또는 Flux를 통해 구현한다. `grafana-operator`를 사용하면 `GrafanaDashboard`, `GrafanaDataSource` 같은 Kubernetes CRD로 Grafana를 선언적으로 관리할 수 있다. Git에 CRD YAML을 푸시하면 ArgoCD가 감지해 Grafana에 자동 적용한다.

## 7. K8s Helm 배포

Kubernetes에 Grafana를 배포하는 방법은 두 가지다. 독립 설치와 kube-prometheus-stack 번들 설치다.

### 독립 설치

`grafana/grafana` Helm 차트로 Grafana만 단독 설치하는 방법이다:

```bash
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update
helm install grafana grafana/grafana \
  -n monitoring \
  --create-namespace \
  -f grafana-values.yaml
```

`grafana-values.yaml`에서 설정할 핵심 항목은 다음과 같다:

```yaml
adminPassword: "your-secure-password"

persistence:
  enabled: true
  size: 10Gi

ingress:
  enabled: true
  hosts:
    - grafana.example.com

datasources:
  datasources.yaml:
    apiVersion: 1
    datasources:
      - name: Prometheus
        type: prometheus
        url: http://prometheus:9090
        isDefault: true

dashboardProviders:
  dashboardproviders.yaml:
    apiVersion: 1
    providers:
      - name: default
        type: file
        options:
          path: /var/lib/grafana/dashboards
```

### kube-prometheus-stack 번들 설치

Prometheus Operator, AlertManager, Grafana를 하나의 Helm 차트로 함께 설치하는 방법이다. 컴포넌트 간 연결 설정이 미리 구성되어 있어 초기 구축 시 편리하다:

```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
helm install monitoring prometheus-community/kube-prometheus-stack \
  -n monitoring \
  --create-namespace \
  -f kube-prometheus-stack-values.yaml
```

이 방식으로 설치하면 서비스 이름이 `monitoring-grafana`가 된다. 포트 포워딩으로 접근하는 방법은 다음과 같다:

```bash
kubectl port-forward svc/monitoring-grafana -n monitoring 3000:80
```

### grafana-operator

`grafana-operator`는 CRD 기반으로 Grafana 리소스를 관리한다. `GrafanaDashboard` CRD를 사용하면 대시보드 JSON을 Kubernetes 리소스로 선언할 수 있어 GitOps 환경에 적합하다:

```yaml
apiVersion: grafana.integreatly.org/v1beta1
kind: GrafanaDashboard
metadata:
  name: service-overview
  namespace: monitoring
spec:
  instanceSelector:
    matchLabels:
      dashboards: grafana
  json: |
    {
      "title": "Service Overview",
      "panels": []
    }
```

## 8. Docker 참고

로컬 개발이나 단일 서버 환경에서는 Docker로 빠르게 Grafana를 실행할 수 있다:

```bash
docker run -d \
  --name grafana \
  -p 3000:3000 \
  -e GF_SECURITY_ADMIN_PASSWORD=admin \
  -v grafana-storage:/var/lib/grafana \
  grafana/grafana:latest
```

Docker Compose에서 data source 프로비저닝을 적용하려면 볼륨 마운트로 설정 파일을 주입한다:

```yaml
services:
  grafana:
    image: grafana/grafana:latest
    ports:
      - "3000:3000"
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=admin
    volumes:
      - grafana-storage:/var/lib/grafana
      - ./provisioning:/etc/grafana/provisioning
    depends_on:
      - prometheus
      - loki
      - tempo

volumes:
  grafana-storage:
```

`./provisioning` 디렉토리에 `datasources/`, `dashboards/`, `alerting/` 하위 폴더를 두고 YAML 파일을 배치하면 Grafana 기동 시 자동으로 로드된다.

K8s와의 차이점은 다음과 같다:

- K8s: ConfigMap 또는 PersistentVolume으로 설정과 데이터를 분리 관리하며, 다중 레플리카 구성이 가능하다.
- Docker: 볼륨 바인드 마운트로 호스트 경로를 직접 주입하며, 단일 인스턴스로만 동작한다.
- 환경변수: `GF_` 접두사로 모든 설정을 환경변수로 오버라이드할 수 있다 (예: `GF_SERVER_HTTP_PORT=8080`).

프로덕션 환경에서는 K8s Helm 배포가 표준이다. Docker는 로컬 PoC나 간단한 모니터링 구성에 적합하다.
