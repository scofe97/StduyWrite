# Grafana Beyla

---

> 코드 변경 없이 HTTP, gRPC, SQL 요청을 커널 수준에서 자동으로 계측하는 eBPF 기반 관측 도구다.

## 1. eBPF와 자동 계측

**eBPF**(extended Berkeley Packet Filter)는 커널 공간에서 실행되는 샌드박스 프로그램이다. 시스템 콜, 네트워크 이벤트, 함수 진입/반환 시점에 프로브를 걸어 데이터를 수집한다. 커널을 수정하거나 모듈을 로드하지 않아도 되므로, 안전하게 운영 환경에 적용할 수 있다.

기존 관측 도구는 애플리케이션 코드에 SDK를 직접 삽입하는 방식을 사용한다. 이 접근은 다음과 같은 한계를 가진다:

- SDK를 코드에 추가하고 재빌드·재배포해야 한다.
- 레거시 시스템이나 서드파티 바이너리는 계측 자체가 불가능하다.
- 계측 범위가 개발자의 SDK 사용 여부에 전적으로 달린다.

Beyla는 커널 수준 후킹으로 이 한계를 우회한다. 애플리케이션 코드를 전혀 건드리지 않고 HTTP 요청, gRPC 호출, SQL 쿼리의 메타데이터를 추출한다. 사이드카 컨테이너나 DaemonSet으로 배포하면 즉시 계측이 시작된다.

## 2. Beyla가 수집하는 것

Beyla는 RED 메트릭과 분산 트레이스를 자동으로 생성한다. RED는 Rate(요청 수/초), Error(오류율), Duration(응답 시간)의 약자로, 서비스 상태를 진단하는 최소한의 세 가지 신호다.

수집하는 주요 메트릭은 다음과 같다:

- `http.server.request.duration` — 서버가 수신한 HTTP 요청의 처리 시간
- `http.client.request.duration` — 클라이언트가 발신한 HTTP 요청의 응답 시간
- `rpc.server.duration`, `rpc.client.duration` — gRPC 서버/클라이언트 호출 시간
- `db.client.operation.duration` — 데이터베이스 쿼리 실행 시간

분산 트레이스 측면에서는 HTTP/gRPC 요청마다 span을 자동 생성하고, 요청 헤더의 trace context를 전파한다. 지원하는 프로토콜 범위는 지속적으로 확장되고 있다:

- **네트워크 프로토콜**: HTTP/1.1, HTTP/2, gRPC
- **데이터베이스**: SQL(PostgreSQL, MySQL), Redis
- **메시지 브로커**: Kafka (v3.1에서 추가)
- **지원 런타임**: Go, Java, Python, Node.js, .NET, Ruby, Rust, C/C++

## 3. Beyla vs OTel SDK

두 접근 방식은 대체 관계가 아니라 보완 관계다. 각 방식의 특성을 비교하면 아래와 같다:

| 기준 | Beyla (eBPF) | OTel SDK |
|------|-------------|---------|
| 설치 방식 | 사이드카/DaemonSet 배포 | 코드에 SDK 추가 |
| 코드 수정 | 불필요 | 필수 |
| HTTP/gRPC/SQL 계측 | 자동 | 자동 (라이브러리 계측) |
| 비즈니스 로직 span | 불가 | 가능 (수동 계측) |
| 런타임 오버헤드 | 극소 (커널 공간) | 소~중 |
| 적합한 시나리오 | 레거시, 서드파티, 빠른 가시성 확보 | 상세한 비즈니스 트레이싱 |

Beyla로 빠르게 기본 가시성을 확보하고, 핵심 서비스에는 OTel SDK로 깊은 계측을 추가하는 **하이브리드 전략**이 권장된다. 예를 들어 결제 서비스라면 Beyla가 HTTP 요청/응답 시간을 자동으로 잡아주고, OTel SDK는 결제 승인 흐름의 세부 단계별 span을 추가로 생성한다.

## 4. Beyla v3.1 업데이트 (2026년 3월)

v3.1.2는 2026년 3월에 출시된 최신 버전이다. 이 릴리스에서 가장 큰 변화는 코드베이스의 OTel 업스트림 기증이다. Beyla의 eBPF 계측 코어가 OpenTelemetry eBPF Instrumentation 프로젝트로 이관되었고, Grafana Beyla는 이 업스트림의 Grafana Labs 공식 배포판으로 재정의되었다. 업스트림 기여로 커뮤니티 유지보수가 강화되고, OTel 생태계와의 호환성이 높아진다.

주요 변경 사항은 다음과 같다:

- **Kafka 자동 감지 추가**: Produce/Consume 작업을 코드 수정 없이 계측한다.
- **Redis 자동 감지 추가**: GET, SET 등 커맨드별 응답 시간을 수집한다.
- **커널 요구사항**: Linux 4.18 이상 필수, 5.8 이상 권장. `CAP_BPF` 권한 또는 privileged 컨테이너가 필요하다.

## 5. K8s 배포

### DaemonSet 모드

DaemonSet 모드에서는 노드당 하나의 Beyla Pod가 해당 노드의 모든 프로세스를 계측한다. 전체 클러스터 가시성을 한 번에 확보할 때 선택하는 방식이다.

Helm으로 설치하는 명령은 다음과 같다:

```bash
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update
helm install beyla grafana/beyla -n monitoring --create-namespace -f values.yaml
```

`values.yaml`의 핵심 설정을 보면 아래와 같다:

```yaml
beyla:
  config:
    data:
      # 계측 대상 프로세스 필터링 (포트 기준)
      discovery:
        services:
          - name: my-app
            namespace: default
            open_ports: 8080-8090

      # OTLP 엔드포인트로 메트릭과 트레이스 전송
      otel_metrics_export:
        endpoint: http://alloy.monitoring:4317
      otel_traces_export:
        endpoint: http://alloy.monitoring:4317

# eBPF에 필요한 권한 설정
podSecurityContext:
  runAsUser: 0

securityContext:
  privileged: true
  # 또는 capabilities로 최소 권한 부여:
  # capabilities:
  #   add: [BPF, PERFMON, SYS_RESOURCE]
```

### Sidecar 모드

전체 노드 계측이 필요 없을 때는 사이드카 모드를 선택한다. 특정 Pod에만 Beyla 컨테이너를 추가하므로, 계측 범위를 서비스 단위로 제한할 수 있다.

사이드카로 배포하는 Pod 명세 예시는 다음과 같다:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  # 호스트 프로세스 네임스페이스 공유 필수
  shareProcessNamespace: true
  containers:
    - name: my-app
      image: my-app:latest
      ports:
        - containerPort: 8080
    - name: beyla
      image: grafana/beyla:3.1.2
      securityContext:
        privileged: true
      env:
        - name: BEYLA_OPEN_PORT
          value: "8080"
        - name: OTEL_EXPORTER_OTLP_ENDPOINT
          value: "http://alloy.monitoring:4317"
```

## 6. Alloy 통합

Grafana Alloy는 Beyla를 외부 프로세스로 실행하지 않고 파이프라인에 직접 임베딩할 수 있다. `beyla.ebpf` 컴포넌트가 그 역할을 담당한다.

Alloy 설정 파일에서 Beyla를 내장하는 구성 예시는 아래와 같다:

```alloy
// eBPF 계측: 8080, 3000 포트의 HTTP/gRPC 요청 자동 수집
beyla.ebpf "auto" {
  open_port = "8080,3000"

  output {
    traces = [otelcol.exporter.otlp.tempo.input]
  }
}

// Tempo로 트레이스 전송
otelcol.exporter.otlp "tempo" {
  client {
    endpoint = "http://tempo.monitoring:4317"
  }
}
```

이 방식의 장점은 Beyla 독립 프로세스를 별도로 관리할 필요가 없다는 점이다. Alloy 단일 바이너리가 eBPF 계측, 로그 수집, 메트릭 스크래핑을 통합 관리한다. LGTM 파이프라인에서 신호 흐름은 다음과 같이 구성된다:

- Beyla(Alloy 내장) → 트레이스 → Tempo
- Beyla(Alloy 내장) → RED 메트릭 → Mimir
- Grafana → Tempo, Mimir 조회 → 상관분석

## 7. Docker 실행 참고

K8s 환경이 아닌 Docker에서 Beyla를 실행할 때는 호스트 프로세스 네임스페이스와 `sys` 파일시스템 마운트가 필수다:

```bash
docker run \
  --privileged \
  --pid=host \
  -v /sys:/sys \
  -e BEYLA_OPEN_PORT=8080 \
  -e OTEL_EXPORTER_OTLP_ENDPOINT=http://host.docker.internal:4317 \
  grafana/beyla:3.1.2
```

주요 플래그의 역할은 다음과 같다:

- `--privileged`: eBPF 프로그램 로드에 필요한 커널 권한 부여
- `--pid=host`: 호스트의 모든 프로세스가 보이도록 PID 네임스페이스 공유
- `-v /sys:/sys`: eBPF 맵과 커널 인터페이스 접근에 필요한 파일시스템 마운트

K8s DaemonSet과 비교하면, K8s는 RBAC와 SecurityContext로 권한을 선언적으로 관리하고 자동 배포가 이루어지지만, Docker는 이 플래그들을 직접 지정해야 한다. 로컬 개발 환경이나 단일 서버 테스트 용도로는 Docker 방식이 빠른 검증 수단이 된다.
