# Facebook OAuth2 Login (provider별 attributes 추상화)

---

> 구글 로그인을 마치고 페이스북을 추가하면 첫 번째 함정이 등장한다. 응답 attributes의 키가 provider마다 다르다는 것이다. 이 글은 `OAuth2UserInfo` 인터페이스로 차이를 흡수하는 전략 패턴을 적용해, 추가 provider를 5분 안에 끼울 수 있는 구조로 만든다.

## 한 줄 정의

`OAuth2UserInfo` 인터페이스와 provider별 구현체(`GoogleUserInfo`, `FacebookUserInfo`)로 attributes의 키 차이를 흡수하면, `loadUser` 메서드는 provider 종류를 모르고도 일관된 코드로 동작한다.

## 왜 provider별 분기가 필요한가

> 구글은 사용자 식별자를 `sub`로 주지만, 페이스북은 `id`로 준다. 단순히 `attributes.get("sub")`로 짠 코드는 페이스북 로그인에서 `null`을 받는다. 동의어를 운영 정책 수준에서 매핑하지 않으면 provider가 늘어날 때마다 `loadUser` 안의 분기가 폭증한다.

세 provider만 비교해도 차이가 명확하다.

| 정보 | 구글 키 | 페이스북 키 | 네이버 키 |
|------|---------|------------|----------|
| 식별자 | `sub` | `id` | `response.id` |
| 이메일 | `email` | `email` | `response.email` |
| 이름 | `name` | `name` | `response.name` |
| 응답 형식 | flat JSON | flat JSON | `response` 하위 객체로 한 단계 감쌈 |

네이버처럼 응답을 별도 객체로 감싸는 경우까지 고려하면 단순 키 매핑으로는 한계가 있다. 다음 절의 인터페이스 방식이 이 모든 차이를 흡수한다.

## 페이스북 개발자 콘솔 설정

> [Meta for Developers](https://developers.facebook.com/)에서 앱을 생성하고 Facebook Login → Settings에서 redirect URI를 등록한다.

1. **App ID / App Secret**: 앱 설정 페이지에서 발급
2. **Valid OAuth Redirect URIs**: `http://localhost:8080/login/oauth2/code/facebook`
3. **App Mode**: Development 모드에서는 등록된 테스트 사용자만 로그인 가능

구글과 동일하게 `redirect_uri`의 마지막 토큰(`facebook`)이 yml의 registration ID와 일치해야 한다.

## application.yml 확장

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          google:
            client-id: ${GOOGLE_CLIENT_ID}
            client-secret: ${GOOGLE_CLIENT_SECRET}
            scope:
              - email
              - profile

          facebook:
            client-id: ${FACEBOOK_APP_ID}
            client-secret: ${FACEBOOK_APP_SECRET}
            scope:
              - public_profile
              - email
```

`scope` 값이 구글과 다른 점에 주의한다. 페이스북은 `profile`이 아니라 `public_profile`이다. provider별 scope 명세가 다르므로 [공식 문서](https://developers.facebook.com/docs/permissions/)에서 정확한 값을 확인해야 한다.

Spring Security는 구글·페이스북·깃허브·옥타 4개 provider를 `CommonOAuth2Provider` enum으로 기본 제공한다. 이들은 `client-id`·`client-secret`·`scope`만 등록하면 token endpoint·user-info endpoint 같은 메타데이터가 자동 채워진다. 다른 provider는 [02-04.Naver OAuth2 Login](02-04.Naver OAuth2 Login.md)에서 다룬다.

## OAuth2UserInfo 인터페이스 — 전략 패턴

```java
public interface OAuth2UserInfo {
    String getProviderId();
    String getProvider();
    String getEmail();
    String getName();
}
```

provider별 구현체가 attributes 키 차이를 흡수한다.

```java
@RequiredArgsConstructor
public class GoogleUserInfo implements OAuth2UserInfo {

    private final Map<String, Object> attributes;

    @Override public String getProviderId() { return (String) attributes.get("sub"); }
    @Override public String getProvider() { return "google"; }
    @Override public String getEmail() { return (String) attributes.get("email"); }
    @Override public String getName() { return (String) attributes.get("name"); }
}

@RequiredArgsConstructor
public class FacebookUserInfo implements OAuth2UserInfo {

    private final Map<String, Object> attributes;

    @Override public String getProviderId() { return (String) attributes.get("id"); }
    @Override public String getProvider() { return "facebook"; }
    @Override public String getEmail() { return (String) attributes.get("email"); }
    @Override public String getName() { return (String) attributes.get("name"); }
}
```

같은 인터페이스를 구현하므로 호출 측은 둘을 구분하지 않는다. GoF 전략 패턴의 교과서적 적용이고, 새 provider가 추가될 때마다 클래스 하나만 추가하면 끝난다.

## loadUser — provider 분기 한 곳

```java
@Service
@RequiredArgsConstructor
public class PrincipalOauth2UserService extends DefaultOAuth2UserService {

    private final UserRepository userRepository;
    private final BCryptPasswordEncoder passwordEncoder;

    @Override
    public OAuth2User loadUser(OAuth2UserRequest userRequest) {
        OAuth2User oauth2User = super.loadUser(userRequest);

        OAuth2UserInfo userInfo = resolveUserInfo(
                userRequest.getClientRegistration().getRegistrationId(),
                oauth2User.getAttributes());

        String username = userInfo.getProvider() + "_" + userInfo.getProviderId();
        User user = userRepository.findByUsername(username)
                .orElseGet(() -> register(userInfo, username));

        return new PrincipalDetails(user, oauth2User.getAttributes());
    }

    private OAuth2UserInfo resolveUserInfo(String registrationId, Map<String, Object> attributes) {
        return switch (registrationId) {
            case "google"   -> new GoogleUserInfo(attributes);
            case "facebook" -> new FacebookUserInfo(attributes);
            case "naver"    -> new NaverUserInfo((Map<String, Object>) attributes.get("response"));
            default -> throw new OAuth2AuthenticationException(
                    new OAuth2Error("unsupported_provider"), "지원하지 않는 provider: " + registrationId);
        };
    }

    private User register(OAuth2UserInfo info, String username) {
        User user = User.builder()
                .username(username)
                .password(passwordEncoder.encode(UUID.randomUUID().toString()))
                .email(info.getEmail())
                .role("ROLE_USER")
                .provider(info.getProvider())
                .providerId(info.getProviderId())
                .build();
        return userRepository.save(user);
    }
}
```

`resolveUserInfo`만 provider를 알고, 나머지 코드(`register`, 호출 측)는 `OAuth2UserInfo` 인터페이스만 본다. provider 추가는 다음 두 단계로 끝난다.

1. `XxxUserInfo implements OAuth2UserInfo` 클래스 추가
2. `resolveUserInfo`의 `switch`에 case 한 줄 추가

## 로그인 페이지 확장

```html
<!-- loginForm.html -->
<a href="/oauth2/authorization/google">Google Login</a>
<a href="/oauth2/authorization/facebook">Facebook Login</a>
```

링크가 하나 늘어났을 뿐 다른 변경이 없다. Spring Security가 `/oauth2/authorization/{provider}` URL을 자동으로 처리해 인가 요청 리다이렉트까지 책임진다.

## 페이스북 응답 attributes 구조

```json
{
  "id": "10227481234567890",
  "name": "홍길동",
  "email": "user@example.com"
}
```

`scope`에 `email`을 포함했음에도 페이스북 사용자가 동의 화면에서 이메일을 거부하면 응답에 `email`이 빠진다. 즉 **`getEmail()`은 항상 null 가능성을 가정**해야 한다. 운영에서는 다음 분기를 추가한다.

```java
if (info.getEmail() == null) {
    throw new OAuth2AuthenticationException(
            new OAuth2Error("email_required"), "이메일 동의가 필요합니다");
}
```

이 검증을 `loadUser` 안에서 던지면 Spring Security가 자동으로 `failureUrl`로 리다이렉트한다.

## 공식 가이드와의 차이 — Spring Authorization Server

Spring Security 5.7부터 `spring-cloud-starter-oauth2` 자리에 `spring-security-oauth2-client`가 자리잡았다. 이전 글이나 코드에서 `WebSecurityConfigurerAdapter` + `@EnableOAuth2Client` 조합을 보면 모두 deprecated된 5.x 시절 패턴이다. 6.x에서는 본 글 코드처럼 람다 DSL과 빈 등록만으로 동작한다.

`spring-authorization-server`는 또 다른 모듈로, **우리가 인가 서버 역할을 수행**할 때 쓴다. 본 글의 주제(클라이언트로서 외부 인가 서버를 사용)와는 정반대 방향이라 혼동하지 않는다.

## 면접 대비 요약

### 한 줄 정의

"`OAuth2UserInfo` 인터페이스로 provider별 attributes 차이를 흡수하면, `loadUser`의 분기를 `resolveUserInfo` 한 곳에 모을 수 있다. 새 provider 추가는 구현체 한 클래스와 switch 한 줄로 끝난다."

### 핵심 포인트 3가지

1. **provider마다 키가 다르다** — 구글은 `sub`, 페이스북은 `id`. `attributes.get("sub")`로 일관 처리하면 페이스북에서 `null`이 나온다.
2. **`scope` 값도 provider마다 다르다** — 구글은 `profile`, 페이스북은 `public_profile`. 공식 문서로 확인하지 않으면 동의 화면에서 거부된다.
3. **`email`은 항상 null 가능** — 사용자가 동의 화면에서 이메일을 거부할 수 있다. `loadUser` 안에서 명시적으로 검증해 `failureUrl`로 보낸다.

### 자주 묻는 질문

Q: Spring Security가 기본 제공하는 4개 provider는?
A: `CommonOAuth2Provider` enum 기준으로 Google, Facebook, GitHub, Okta. 이들은 token·user-info endpoint가 자동 채워져 `client-id`·`client-secret`·`scope`만 등록하면 동작한다.

Q: 페이스북 사용자의 이메일이 항상 같다고 가정해도 되는가?
A: 안 된다. 사용자가 페이스북 계정의 이메일을 바꾸면 응답값도 바뀐다. 영구 키는 `id`(facebook의 user id)이고, 이메일은 표시·연락용으로만 사용한다.

Q: 같은 사용자가 구글과 페이스북에 다 가입하면 계정이 분리되는가?
A: 본 글의 구조(`provider + "_" + providerId`)에서는 분리된다. 합치고 싶다면 이메일을 기준으로 통합하는 별도 로직이 필요하지만, 이는 사용자가 명시적으로 "연결" 동작을 한 경우에만 안전하다.
