# DDD 전략적 설계와 전술적 패턴
---
> 도메인 주도 설계(DDD)는 객체 몇 개를 예쁘게 만드는 방법이 아니라, 복잡한 도메인을 팀과 코드 구조에 맞게 분해하는 접근이다.
>
> - Bounded Context
> - Aggregate
> - Entity
> - Value Object

## 1. DDD가 필요한 이유

복잡한 도메인에서는 클래스 설계보다 경계 설계가 먼저 중요하다. 카드게임 서버에서 런 진행, 전투 규칙, 카드 정의, 계정 해금은 모두 관련이 있지만 같은 속도로 바뀌지 않는다.

DDD는 “무엇이 핵심 도메인이고, 무엇이 지원 도메인인가”, “어디까지를 같은 모델로 볼 것인가”를 정하게 해준다. 이 질문에 답하지 않으면 팀은 계속 같은 단어를 다르게 쓴다.

## 2. 전략적 설계: Bounded Context 나누기

예시 프로젝트에서는 다음 context 구분이 자연스럽다:

| Context | 핵심 책임 | 대표 모델 | 비고 |
|---------|----------|----------|------|
| `run` | 런 시작, 종료, 노드 진행 | `Run`, `ActProgress` | 세션 전체 |
| `battle` | 턴, 카드 사용, 적 행동 | `Battle`, `TurnState` | 규칙 복잡도 가장 높음 |
| `catalog` | 카드, 유물, 적 정의 | `CardDefinition`, `RelicDefinition` | 참조 데이터 |
| `progression` | 메타 해금과 업적 | `Profile`, `Unlock` | 핵심 런과 분리 가능 |

이 구분은 나중에 모듈 분리나 서비스 분리의 기준점이 된다. 처음부터 물리적으로 분리하지 않더라도 개념적 경계를 먼저 세워야 한다.

## 3. 전술적 패턴: Aggregate를 어디까지 둘 것인가

Aggregate는 하나의 트랜잭션 경계다. 모든 것을 `Run` 하나에 넣으면 단순해 보이지만, 전투가 두꺼워질수록 하나의 거대 aggregate가 된다.

이 시리즈에서는 기본적으로 `Run`을 상위 aggregate로 보되, 현재 전투 상태는 `Battle` 하위 구성요소로 모델링하는 방향을 예시로 사용한다. 다만 동시성 요구나 저장 전략이 달라지면 `Battle`을 별도 aggregate로 분리할 수 있다.

## 4. Entity와 Value Object

DDD에서 Entity는 식별성을 가진다. Value Object는 값 자체가 의미다. 이 구분은 카드게임 도메인에서 꽤 유용하다.

예를 들어 다음처럼 볼 수 있다:

- `Run`, `Battle`, `PlayerProfile`은 Entity
- `Hp`, `Gold`, `Energy`, `CardCost`, `ActNumber`는 Value Object
- `CardDefinition`은 catalog context에서는 Entity지만, battle context에서는 읽기 전용 참조 데이터에 가깝다

## 5. Domain Service와 Policy

모든 로직이 Entity 내부에 들어가야 하는 것은 아니다. 여러 객체를 엮는 규칙이나, 순수 계산 로직, 선택 전략은 Domain Service나 Policy로 두는 편이 더 깔끔할 수 있다.

예를 들면 다음이 그렇다:

- 보상 후보군을 뽑는 `RewardPolicy`
- 적 의도 선택을 계산하는 `EnemyIntentPolicy`
- 카드 풀에서 랜덤 선택을 담당하는 `CardDraftPolicy`

## 6. Repository를 어떻게 읽어야 하는가

DDD의 Repository는 단순 DAO와 다르다. Aggregate를 로드하고 저장하는 도메인 진입점이다. 따라서 JPA Entity 자체보다 도메인 aggregate 관점으로 인터페이스를 설계하는 편이 낫다.

이때 중요한 것은 저장 기술보다 계약이다. application은 `RunRepository`가 JPA인지 Redis인지 몰라도 된다.

## 7. 예시 프로젝트에 적용한 결론

이 시리즈에서는 `battle`을 핵심 도메인, `run`을 핵심을 감싸는 흐름 도메인으로 본다. `catalog`는 지원 도메인이고, `progression`은 별도 확장 context로 둔다.

즉 DDD의 목적은 클래스를 늘리는 데 있지 않다. 런 관리 서비스에서 어디까지 같은 모델로 보고, 어디서부터 다른 팀과 다른 저장 전략을 허용할지 정하는 데 있다.
