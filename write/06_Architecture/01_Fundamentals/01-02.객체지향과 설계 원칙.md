# 객체지향과 설계 원칙
---
> 좋은 객체지향 설계는 데이터를 담는 클래스 목록이 아니라 책임과 협력을 분리하는 작업이다.
>
> - 캡슐화
> - 다형성
> - 불변성
> - 명시적 책임

## 1. 객체지향은 상태보다 행위를 먼저 본다

객체지향 설계의 핵심은 “누가 이 일을 책임지는가”를 정하는 것이다. `BattleState`가 단순 DTO로 흩어져 있으면 서비스 계층이 모든 규칙을 떠안게 된다. 반대로 전투 규칙이 도메인 객체 안에 있으면 변경 지점이 줄어든다.

카드게임 도메인에서는 `Run`이 현재 진행 상황을 알고, `Battle`이 턴 흐름을 관리하며, `CardEffect`가 효과 적용 책임을 가진다. 이런 식으로 책임을 객체로 내려보내는 것이 첫 번째 원칙이다.

## 2. 캡슐화는 규칙을 데이터와 같이 둔다

캡슐화는 필드를 `private`로 감추는 것보다 규칙을 객체 내부로 모으는 데 의미가 있다. 예를 들어 에너지 차감과 카드 소모 여부를 service에서 따로 계산하면 규칙이 퍼진다.

다음 예시는 카드 사용 규칙을 `Battle` 내부에 모은 형태다:

```java
public final class Battle {
    private Energy energy;
    private Hand hand;

    public BattleEvent play(CardId cardId, Target target) {
        PlayableCard card = hand.requirePlayable(cardId);
        energy = energy.consume(card.cost());
        hand = hand.remove(cardId);
        return card.effect().apply(target);
    }
}
```

## 3. 다형성은 분기문을 줄이는 도구다

카드 효과가 늘어날수록 `switch(cardType)`는 빠르게 비대해진다. 이때 다형성은 새 규칙을 기존 분기문에 끼워 넣지 않고 독립 객체로 추가하게 해준다.

예를 들면 `DamageEffect`, `DrawEffect`, `ApplyStatusEffect`를 모두 `CardEffect` 구현체로 두면 된다. 새 카드가 추가될 때 기존 서비스 코드를 수정하지 않고 새 타입만 추가할 수 있다.

## 4. 불변성은 리플레이와 테스트를 쉽게 만든다

카드게임은 상태 전이가 많아서 변경 가능한 객체가 많아질수록 디버깅이 어려워진다. 특히 특정 전투 로그를 다시 재현해야 할 때, 불변 객체와 명시적 이벤트가 큰 도움이 된다.

Java 25 기준 설계에서는 `record`와 작은 값 객체를 적극적으로 활용하는 편이 좋다. `Hp`, `Gold`, `Energy`, `CardId` 같은 타입을 값 객체로 두면 잘못된 인자 조합을 줄일 수 있다.

## 5. SOLID는 문구보다 경계 설계에 가깝다

SOLID를 외우는 것보다 어디에 적용할지를 아는 편이 중요하다. 런 관리 서비스에서는 다음처럼 읽는 것이 실용적이다:

- SRP: `RunService` 하나에 전투, 보상, 저장, 이벤트 발행을 몰아넣지 않는다.
- OCP: 새 카드 효과 추가가 기존 전투 흐름 수정으로 이어지지 않게 한다.
- DIP: 외부 저장소와 메시지 브로커는 port 뒤로 숨긴다.

## 6. Spring Boot에서 자주 생기는 객체지향 붕괴

Spring Boot 프로젝트는 `@Service`에 모든 로직을 넣기 쉬운 구조다. 이 방식은 처음엔 빠르지만, 시간이 지나면 서비스 클래스가 도메인 규칙과 인프라 세부사항을 동시에 알게 된다.

다음 신호가 보이면 설계가 무너지고 있다고 보면 된다:

- service가 500줄 이상으로 커진다.
- 테스트가 Mockito 설정으로만 가득 찬다.
- 동일한 유효성 검사가 여러 service에 반복된다.

## 7. 런 관리 서비스에 적용한 결론

이 시리즈에서는 `Run`, `Battle`, `CardEffect`, `RewardPolicy` 같은 도메인 객체를 중심으로 설명한다. Spring Bean은 객체 협력을 조립하는 역할에 집중하고, 카드 규칙과 상태 전이는 가능한 한 도메인 모델 안에 둔다.

이 원칙을 지키면 뒤에서 나올 클린 아키텍처, 헥사고날, DDD 문서가 자연스럽게 연결된다. 객체지향은 별도 주제가 아니라 모든 구조적 선택의 바탕이다.
