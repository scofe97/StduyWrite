# 이벤트 기록과 타임라인 시각화

## 왜 이 단계가 필요한가

부하 테스트 결과를 *그 자리* 에서만 보는 건 충분하지 않다. pytest 콘솔 로그는 휘발성이고, 50건 dispatch 가 동시에 만든 상태 전이 200여 건은 사람 머리에 들어오지 않는다. 표준 pytest-html 리포트도 *한 테스트 = 한 줄* 이라 테스트 *내부* 시간축 정보를 보존하지 못한다. 잡 #7 이 trigger 후 몇 초만에 RUNNING 으로 갔는지, 어떤 잡이 quota gate 에 막혀 대기 후 풀렸는지를 보려면 *시계열 이벤트 기록* + *사후 시각화* 가 필요하다.

`qa/python/suites/load_orchestration/_lib/event_recorder.py` 와 `timeline_renderer.py` 는 이 빈자리를 채운다. 한 줄 한 이벤트의 NDJSON 으로 모든 상태 변화를 적재하고, 세션 종료 시 한 HTML 파일로 swim lane 타임라인을 렌더한다. 같은 ndjson 은 dev Grafana 의 후속 분석에도 그대로 흘릴 수 있어, 한번 적재한 데이터가 *로컬 HTML + 원격 시계열* 양쪽으로 분기된다.



## NDJSON — 한 줄 한 이벤트

NDJSON 은 Newline-Delimited JSON 의 약자. 한 줄에 한 JSON 객체를 두고 `\n` 으로 구분한다. JSON 배열 한 덩어리와 다른 점:
- **append-only**. 줄 끝에 한 줄 추가는 O(1). JSON 배열은 마지막 `]` 를 찾아 그 자리에 끼워야 한다.
- **부분 파싱 가능**. 손상된 줄 하나가 있어도 나머지 줄은 그대로 읽을 수 있다.
- **stream 친화**. `tail -f` 로 실시간 관찰 가능, `head -1000` 으로 일부만 읽기 가능.

부하 테스트는 *thread 여럿이 동시에 이벤트를 발행* 하는 상황이라 append-only 가 본질적이다. 50건 dispatch 가 동시에 trigger 이벤트를 쏠 때 한 줄씩 자물쇠 안에서 적는 게 가장 단순한 정답.

### 이벤트 스키마 — 3 가지 타입

`event_recorder.py` 가 적는 이벤트는 세 가지다 — `trigger`, `state`, `terminal`.

```jsonc
// trigger: dispatch 호출 시점. 한 잡당 1건.
{"ts": 1731700000.123, "event": "trigger", "pattern": "balanced",
 "job_excn_id": "EX-A1B2C3", "tl_id": "TL_E2E_CONSOLE",
 "job_path": "job/SBH/job/SBH-LOAD-01", "http_status": 200, "elapsed_ms": 12.4}

// state: 상태 변화 감지 시. 보통 SUBMITTED, RUNNING 등. 한 잡당 2~3건.
{"ts": ..., "event": "state", "job_excn_id": "EX-...",
 "status": "SUBMITTED", "queue_id": null, "build_no": null,
 "changed": ["status"]}

// terminal: 종료 상태 도달 시. 한 잡당 1건.
{"ts": ..., "event": "terminal", "job_excn_id": "EX-...",
 "status": "SUCCESS", "queue_id": 1234, "build_no": 7,
 "bgng_dt": "2026-05-19 14:23:01", "end_dt": "2026-05-19 14:24:35"}
```

세 타입을 같은 파일에 섞어 적는다. 각 줄의 `event` 필드가 분류 키. `ts` 는 `time.time()` 의 epoch float — 시각 비교가 한 산술 연산.

세 타입을 분리한 이유는 *책임 분리* 가 아니라 *측정 시점 분리* 다. trigger 는 dispatch 호출자가 적고, state/terminal 은 폴러 thread 가 적는다. 한 잡의 trigger.ts 와 terminal.ts 차이가 그 잡의 end-to-end 처리 시간이고, 이 차이가 timeline 막대의 길이로 즉시 변환된다.



## NdjsonEmitter — thread-safe append

```python
class NdjsonEmitter:
    def __init__(self, path: pathlib.Path) -> None:
        self.path = path
        self.path.parent.mkdir(parents=True, exist_ok=True)
        self._lock = threading.Lock()
        self._fp = self.path.open("w", encoding="utf-8")   # truncate 로 새 세션

    def record(self, event_type: str, **fields) -> None:
        evt = {"ts": time.time(), "event": event_type, **fields}
        line = json.dumps(evt, ensure_ascii=False, default=str)
        with self._lock:
            self._fp.write(line + "\n")
            self._fp.flush()
```

세 가지 결정이 박혀 있다.

1. **세션 시작 시 truncate (`"w"` 모드)**. append 가 아니라 새로 시작. 한 세션 = 한 ndjson 파일. 이전 세션 데이터가 섞이면 vis-timeline 이 동시에 그려 버려 분석이 깨진다.
2. **`threading.Lock()` 으로 줄 단위 직렬화**. 50 thread 가 동시에 write 해도 한 줄이 갈라지지 않는다. write+flush 가 락 안에서 원자적으로 일어난다.
3. **`default=str` + `ensure_ascii=False`**. datetime · Decimal 같은 타입은 자동 str 변환, 한글 같은 비-ASCII 는 그대로 UTF-8 인코딩. 이 두 옵션이 빠지면 datetime 에서 TypeError 가 나거나 한글이 `\uXXXX` 로 escape 된다.

flush 매번 호출 비용은 부하 테스트엔 무시 가능 — 한 세션 적재량이 수백 줄. 1000+ 이벤트 적재가 필요하다면 *batch flush* 로 전환 검토. 현재 규모엔 즉시 flush 가 디버깅 친화적이다 (테스트 도중 `tail -f` 로 관찰 가능).



## PrometheusEmitter — stub 으로 두는 미래 인터페이스

```python
class PrometheusEmitter:
    def __init__(self, gateway_url: str | None) -> None:
        self.gateway_url = gateway_url or os.environ.get("PROM_PUSHGW_URL")

    def record(self, event_type: str, **fields) -> None:
        if not self.gateway_url:
            return                                # no-op
        # TODO: prometheus-client + push_to_gateway 로 구현

    def close(self) -> None:
        return
```

지금은 *no-op* 다. 그런데 클래스 자체는 박혀 있다. *왜* 박았는가 — 미래 확장점을 *지금* 인터페이스로 못 박는다. `PROM_PUSHGW_URL` 환경변수가 설정되는 순간 record 본문만 채우면 fanout 의 다른 노드는 손댈 필요가 없다. 호출 측 (테스트 코드) 은 emitter 가 어느 백엔드로 보내는지 모른 채 record 만 한다.

이 결정은 [pytest 의 의존성 주입 철학](01-02.conftest%20계층과%20fixture%20설계.md) 의 연장이다. *함수가 자기가 쓰는 자원의 출처를 모르는 것* 이 핵심. 같은 식으로 emitter 도 자기가 보내는 destination 을 모른 채 record 만 한다.



## FanoutEmitter — 여러 emitter 에 동시에 흘리기

```python
class FanoutEmitter:
    def __init__(self, *emitters: MetricEmitter) -> None:
        self._emitters = list(emitters)

    def record(self, event_type: str, **fields) -> None:
        for em in self._emitters:
            try: em.record(event_type, **fields)
            except Exception: pass               # 한 emitter 실패가 다른 곳을 막지 않음

    def close(self) -> None:
        for em in self._emitters:
            try: em.close()
            except Exception: pass
```

세 가지 설계 결정.

1. **try/except 로 emitter 격리**. Prometheus push gateway 가 잠시 죽어도 ndjson 적재는 계속된다. *부하 테스트 자체가 emitter 한 곳 때문에 깨지면 본말전도*.
2. ***MetricEmitter Protocol* 로 타입 지정**. duck typing 이지만 `Protocol` 이 박혀 있어 IDE 가 record/close 시그니처를 검증. 새 emitter (예: `ClickhouseEmitter`, `S3Emitter`) 를 추가할 때 인터페이스가 명시적.
3. **순서 보장**. emitter 목록의 *순서대로* 호출된다. ndjson 을 먼저 적은 다음 Prometheus 로 흘리는 게 디버깅 친화적이라 그 순서로 fixture 에서 생성한다.

이게 *strategy pattern* 의 가장 단순한 모양이다. 추상 클래스도 인터페이스 선언도 필요 없고, Python 의 duck typing + Protocol 만으로 같은 효과를 본다.



## conftest.py — fixture + sessionfinish 자동 렌더

`conftest.py:26-58` 가 emitter fixture 와 HTML 자동 렌더 hook 을 묶는다.

```python
@pytest.fixture(scope="session")
def metric_emitter():
    path = _session_ndjson()            # artifacts/{ts}.ndjson
    ndjson = NdjsonEmitter(path)
    prom = PrometheusEmitter(os.environ.get("PROM_PUSHGW_URL"))
    emitter = FanoutEmitter(ndjson, prom)
    emitter.ndjson_path = path          # session hook 이 참조 가능하도록
    try: yield emitter
    finally: emitter.close()


def pytest_sessionfinish(session, exitstatus):
    candidates = sorted(ARTIFACTS.glob("*.ndjson"))
    if not candidates: return
    latest = candidates[-1]
    summary = render(latest, REPORT_HTML)
    print(f"\n[load_orchestration] report → {REPORT_HTML}\n"
          f"  pattern={summary.get('pattern')} total={summary.get('total')} "
          f"by_status={summary.get('by_status')}\n")
```

`pytest_sessionfinish` 는 pytest 가 모든 테스트를 마치고 종료하기 *직전* 자동 호출되는 표준 hook 이다. 이 안에서 가장 최신 ndjson 을 찾아 vis-timeline HTML 로 변환. 사람이 별도 명령을 칠 필요가 없다.

`emitter.ndjson_path` 를 동적으로 attach 하는 건 좀 어색한 패턴 — 보통 fixture 가 반환하는 객체에 ad-hoc 속성을 박는 건 권하지 않는다. 더 명시적인 모양은 `(emitter, path)` 튜플 반환 또는 `ndjson_path` 를 별도 fixture 로 분리. 다만 현재 사용처 1곳에선 *비용 대비 가독성* 이 미세한 차이라 그대로 둔다.



## timeline_renderer — ndjson → vis-timeline 단일 HTML

`timeline_renderer.py` 는 ndjson 한 파일을 받아 *vis-timeline 임베드 HTML 한 파일* 을 만든다. 외부 의존은 vis-timeline CSS/JS 두 개의 CDN 로드 뿐.

### 핵심 단계

```python
def render(ndjson_path: pathlib.Path, out_html: pathlib.Path) -> dict:
    events = _load_events(ndjson_path)            # ndjson → list[dict]
    items, groups, summary = _build_items(events) # 이벤트 → timeline DataSet
    payload = {"items": items, "groups": groups, "summary": summary}
    html = HTML_TEMPLATE.replace("__DATA_JSON__", json.dumps(payload, ensure_ascii=False))
    out_html.write_text(html, encoding="utf-8")
    return summary
```

이벤트를 잡(`job_excn_id`) 단위로 그룹화해 *시작 - 끝* 의 막대 하나로 만든다.

```python
# _build_items 핵심
by_excn: dict[str, dict] = {}
for evt in events:
    kind = evt.get("event") or evt.get("kind")
    if kind == "trigger":
        by_excn[eid] = {"trigger_ts": ts, "tl_id": ..., "final_status": None, ...}
    elif kind == "terminal":
        by_excn[eid]["final_status"] = status
        by_excn[eid]["terminal_ts"] = ts

# vis-timeline item 1건 = 잡 1개
items.append({
    "id": eid,
    "group": tl_id,                    # swim lane 분리
    "start": int(trigger_ts * 1000),
    "end":   int(terminal_ts * 1000),
    "content": f"{job_name} ({status})",
    "style": f"background-color:{COLOR_BY_STATUS[status]};color:#fff;",
})
```

`group` 필드가 vis-timeline 의 swim lane 분리를 만든다. `TL_E2E_CONSOLE` 잡들은 위쪽 lane, `TL_E2E_V2` 는 아래 lane. 한 lane 안에서 막대들이 시간순으로 누적 표시되어 *언제 누가 trigger 됐고 언제 끝났는지* 가 한 눈에 들어온다.

### 색상 매핑

```python
COLOR_BY_STATUS = {
    "SUCCESS":       "#3cb371",   # 초록
    "FAILURE":       "#e74c3c",   # 빨강
    "UNSTABLE":      "#f39c12",   # 주황
    "ABORTED":       "#95a5a6",   # 회색
    "REJECTED":      "#e67e22",   # 짙은 주황
    "NOT_BUILT":     "#7f8c8d",
    "NOT_EXECUTED":  "#7f8c8d",
}
DEFAULT_COLOR = "#3498db"          # RUNNING / 미도달
```

색은 신호다. 50건 timeline 에서 *빨강이 한 줄* 있으면 즉시 그 막대 hover 로 jobExcnId 와 jobPath 가 보인다. 짙은 주황(REJECTED) 이 무더기로 보이면 dispatch_gate 회귀 신호. 초록만 가득하면 정상.

### HTML_TEMPLATE — string replace 한 곳

```python
HTML_TEMPLATE = """<!doctype html>
<html lang="ko">
<head>
  <link href="https://unpkg.com/vis-timeline@7.7.2/styles/vis-timeline-graph2d.min.css" rel="stylesheet">
  <script src="https://unpkg.com/vis-timeline@7.7.2/standalone/umd/vis-timeline-graph2d.min.js"></script>
  ...
</head>
<body>
  <div id="timeline" style="height: 360px;"></div>
  <table id="rows">...</table>
  <script>
    const DATA = __DATA_JSON__;            // ← 여기에 json.dumps(payload) 가 박힌다
    ...
  </script>
</body>
</html>
"""

html = HTML_TEMPLATE.replace("__DATA_JSON__", json.dumps(payload, ensure_ascii=False))
```

템플릿 엔진 (Jinja2, Mako) 을 안 쓴다. 한 placeholder (`__DATA_JSON__`) 만 치환하면 끝이라 표준 라이브러리 `str.replace()` 가 충분.

> **함정 — `__DATA_JSON__` 이 본문에 우연히 등장하면 잘못 치환된다.** 현재 코드는 안전한 placeholder 이름이지만, 사용자 입력을 그대로 HTML 에 넣는 시스템이면 *템플릿 엔진의 escape* 가 본질적. 이 모듈은 신뢰된 자체 데이터만 처리하므로 그대로 둔다.

## 같은 패턴을 다른 곳에 — 일반화

NDJSON + emitter + fanout + 단일 HTML 패턴은 *부하 테스트 결과 분석* 외에도 응용 가능하다.

- **integration 테스트의 시계열 디버깅** — 여러 fixture 의 setup/teardown 순서를 ndjson 으로 적어 timeline 으로 보면 "왜 이 fixture 가 저 fixture 보다 늦게 호출됐는가" 가 한 눈에.
- **chaos 테스트의 상태 전이** — `jenkins_blocked` 안에서 일어난 상태 변화를 timeline 으로 보면 회복 경로의 결함을 빨리 잡는다.
- **CI 빌드 단계별 시간 분포** — Jenkins/GitLab CI 의 각 stage 를 emitter 로 흘리면 *어느 stage 가 느린가* 가 막대 길이로 보인다.

핵심은 *측정점 (record 호출)* 만 코드 곳곳에 박으면 *시각화 (renderer)* 는 한 모듈에 격리되어 늘어나는 시각화 형태를 한 곳에서 관리한다는 점.



## 함정과 회피

### 함정 1 — ndjson 누적

매 세션이 `artifacts/{ts}.ndjson` 으로 새 파일을 만든다. 정리 정책이 없으면 *artifacts/* 가 무한 누적. `.gitignore` 에는 들어 있지만 디스크는 차오른다. 30일 이전 파일 자동 삭제 정책 또는 *최근 N 개만 보존* 정책 추가 권고 (현재는 미적용).

### 함정 2 — vis-timeline CDN 차단

unpkg.com 이 사내 망에서 차단되면 HTML 이 빈 페이지로 보인다. 회피는 두 가지:
- vis-timeline 번들을 다운로드해 HTML 에 인라인 박기 (self-contained 변형).
- 사내 npm mirror 또는 자체 호스팅으로 CDN URL 교체.

### 함정 3 — 한 emitter 가 blocking 호출이면 fanout 전체가 느려진다

`PrometheusEmitter` 가 push gateway 와 동기 통신하면 record() 한 번이 수십 ms 걸린다. dispatch 가 50 thread 에서 동시에 record 를 호출하면 락 경합이 일어난다. **회피**: blocking emitter 는 내부에 queue + worker thread 를 두어 record() 자체는 enqueue 만 하고 즉시 반환. 현재 PrometheusEmitter 는 stub 이라 무관하지만 *진짜 구현* 시 명심.

### 함정 4 — emitter.ndjson_path 의 ad-hoc 속성

fixture 가 반환한 객체에 `setattr` 로 ndjson_path 를 박는 패턴. IDE/타입체커가 잡지 못한다. 명시적인 대안은 NamedTuple `EmitterBundle(emitter, ndjson_path)` 반환. 현재는 사용처 1곳뿐이라 그대로 두지만, 사용처가 늘면 리팩터 가치.



## TPS 사례 — 두 모듈의 책임 분담

`qa/python/suites/load_orchestration/_lib/` 의 두 모듈:

| 모듈 | 책임 | 호출 시점 |
|------|-----|----------|
| `event_recorder.py` | 이벤트 적재 (NDJSON + fanout) | 테스트 실행 중 (dispatch + 폴러 thread) |
| `timeline_renderer.py` | ndjson → HTML 변환 | 세션 종료 (`pytest_sessionfinish` hook) |

두 모듈은 *시점이 다르다*. 적재는 hot path 라 thread-safety 와 성능이 본질이고, 렌더는 cold path 라 가독성과 정확성이 본질이다. 같은 파일에 두지 *않은* 결정이 옳다 — 두 책임의 변경 주기가 다르기 때문.

산출물 한 줄로 요약: **`qa/python/report-orchestration.html`** — 외부 의존 2 개(CDN), 한 파일, 브라우저로 직접 열기.
