# Faker corpus 시드로 검색·페이지 검증

## 왜 이 단계가 필요한가

검색·정렬·페이지 API 를 진짜로 검증하려면 *적당한 다양성을 가진 충분한 양의 데이터* 가 필요하다. 한 개 두 개로는 ASC/DESC 결과가 다를 수 없고 page=2 가 비어 있어 페이지네이션 회귀를 잡지 못한다. 그렇다고 사람이 손으로 60건 시드 SQL 을 쓰면 *비슷한 데이터* 만 쌓여 한국어 정렬 같은 collation 함정을 절대 못 잡는다. Faker 같은 라이브러리는 이 사이의 사각지대를 메운다 — **결정적 seed + 도메인 다양성** 을 한 줄 코드로 얻는다.

`qa/python/suites/approval/_lib/seed_corpus.py` 는 결재 도메인의 60건 corpus 를 만들어 정렬·페이지·검색·가시성 4축을 회귀 검증한다. 단순한 Faker 사용 예시처럼 보이지만 실제 결정 밀도는 높다 — *실제 DB 의 활성 사용자/역할만 쓰기*, *free page-component pool 부족 시 즉시 실패*, *직접/역할 결재자 round-robin 혼합*, *전체 중 절반만 demand 처리* 같은 세부 정책이 다층으로 박혀 있다.



## Faker 의 핵심 — seed 와 도메인 인스턴스

```python
from faker import Faker

fk = Faker("ko_KR")          # 한국어 도메인 (이름·주소·전화번호)
fk.seed_instance(42)         # 인스턴스 단위 seed — 글로벌 random 과 격리

# 도메인별 generator
name  = fk.name()                                    # '김민수'
email = fk.email()                                   # 'minsu.kim@example.com'
dt    = fk.date_time_between(start_date="-30d", end_date="now")
text  = fk.text(max_nb_chars=200)
uid   = fk.unique.bothify("결재-####")               # '결재-7392' — unique 보장
```

세 가지 결정.

1. **`Faker("ko_KR")` 도메인 지정**. 영어 이름·주소가 한국어 검색 정책의 함정을 못 잡는다. utf8mb4 collation 정렬 순서, 한글 LIKE 검색 등을 검증하려면 한국어 데이터.
2. **`fk.seed_instance(42)` — 인스턴스 단위 seed**. `Faker.seed(42)` (클래스 메서드) 와 다르다. 인스턴스 seed 는 다른 Faker 인스턴스·글로벌 random 과 격리되어 다른 테스트가 random 을 호출해도 영향이 없다. *결정성* 의 본질.
3. **`fk.unique.bothify(...)` — 중복 방지**. unique proxy 가 이미 생성된 값을 기록해 중복을 피한다. 한 결재명이 두 번 생성되면 unique 제약 위반으로 INSERT 실패하므로 필수.

> **Faker seed 가 결정적이라는 보증의 한계**: Faker 자체 버전이 바뀌면 같은 seed 도 다른 결과를 만들 수 있다. `requirements.txt` 의 `faker==30.8.2` 같은 버전 핀이 결정성을 *시간축* 으로도 보장한다.



## 시드 데이터의 결정성 == 회귀 테스트의 결정성

검색·페이지 API 의 회귀 테스트는 *입력 데이터가 다르면 정답도 다르다*. 매 실행 새 데이터를 만들면:
- 5월 14일엔 통과, 15일엔 실패 → flaky test
- 어제 실패 원인이 *코드 회귀* 인지 *데이터 차이* 인지 구별 불가
- 디버깅이 *상태 복원* 부터 시작해야 한다

Faker 의 결정적 seed 는 이 문제를 정확히 푼다. 같은 코드 + 같은 seed = 같은 60건. 회귀가 보이면 *반드시 코드 변경* 이 원인이다.

`qa/python` 은 한 발 더 나간다 — Faker 의 결정성에 더해 *실제 DB 마스터 데이터* 의 결정성도 활용한다. 사용자 ID 와 역할 ID 는 Faker 로 만들지 않고 *DB 에 실제 시드된 마스터* 에서 뽑는다.

```python
def real_active_users(db) -> list[str]:
    with db.cursor() as cur:
        cur.execute("SELECT USER_ID FROM TB_TPS_CM_001 "
                    "WHERE USE_YN='Y' AND DEL_YN='N' ORDER BY USER_ID")
        return [row[0] for row in cur.fetchall()]
```

이유는 명확하다 — 결재선 결재자 ID 는 FK 로 사용자 마스터를 참조한다. Faker 로 만든 가짜 user_id `simbohyeon-fake-001` 을 박으면 FK 제약으로 INSERT 가 실패하거나, 통과해도 *비활성 사용자* 라 my_todo 가시성 테스트가 무의미해진다. 실 마스터에서 뽑으면 그 사용자에게 my_todo 가 실제로 보일지 검증할 수 있다.



## 60건의 *분포 설계*

단순히 60건이 아니라 *어떤 60건* 인가가 본질이다. `seed_corpus.py` 의 `STEP_COUNT_PLAN` 이 분포를 명시한다.

```python
# 단계 수 분포: 합계 60
STEP_COUNT_PLAN: list[int] = [1] * 15 + [2] * 15 + [3] * 15 + [4] * 10 + [5] * 5
#                              ────────  ────────  ────────  ────────  ───────
#                                 25%       25%       25%       17%       8%
```

1단계 결재 15건, 2단계 15건, 3단계 15건, 4단계 10건, 5단계 5건. 이 분포가 의도하는 검증:
- 1단계만 있는 결재 = my_todo 가시성의 최소 케이스
- 3단계 결재 = 흔한 도메인 흐름의 medium 케이스
- 5단계 결재 = 깊은 진행 흐름의 edge case

단계 수가 *고르게* 분포하면 정렬 컬럼 중 `atrzPrcsRsltCd` 같은 enum 의 검색이 한 값에 쏠리지 않는다. 페이지 size=10 으로 6 페이지가 만들어지고, 각 페이지마다 단계 수 분포가 비슷해 페이지 메타 검증의 신뢰성이 올라간다.

### 직접 vs 역할 결재자 round-robin

```python
def _plan_step_approvers(users, roles, step_count, seq):
    specs = []
    for step_idx in range(step_count):
        token = (seq + step_idx) % 2          # 홀짝 mix
        if token == 0 or not roles:
            user_id = users[(seq + step_idx) % len(users)]
            specs.append(StepSpec(aprvr_se_cd="01", aprvr_info=user_id))
        else:
            role_id, members = roles[(seq + step_idx) % len(roles)]
            specs.append(StepSpec(aprvr_se_cd="03",
                                  aprvr_info=str(role_id),
                                  members=list(members)))
    return specs
```

각 단계의 결재자가 *직접(`01`)* 또는 *역할(`03`)* 로 번갈아 박힌다. 가시성 테스트의 두 갈래 — *직접 결재자가 자기 my_todo 를 보는가*, *역할 멤버가 그 역할 결재선의 my_todo 를 보는가* — 를 둘 다 검증하려면 두 유형 모두 corpus 에 있어야 한다.

round-robin 은 `seq + step_idx` 인덱스로 균등 분포. 한 사용자가 여러 결재선에 반복 등장하는 것도 의도 — 실제 운영에서도 같은 결재자가 여러 결재의 1차/2차 결재자로 동시에 박힌다.



## free pool 사전 검증 — 즉시 실패

60건 결재를 만들려면 60 개의 *(menuCd, pageId, pageCompnId)* 트리거 페어가 비어 있어야 한다. 기존 결재가 점유한 페어를 재사용하면 unique 제약 위반.

```python
def free_page_compn_pool(db, page_ids: list[int]) -> list[tuple[int, str]]:
    sql = f"""
        SELECT c.PAGE_ID, c.PAGE_COMPN_ID
          FROM TB_TPS_AT_003 c
         WHERE c.PAGE_ID IN ({placeholders})
           AND c.USE_YN='Y' AND c.DEL_YN='N'
           AND NOT EXISTS (
               SELECT 1 FROM TB_TPS_AZ_002 t
                 JOIN TB_TPS_AZ_001 b ON t.ATRZ_ID=b.ATRZ_ID AND t.VSRN=b.VSRN
                WHERE t.PAGE_ID=c.PAGE_ID AND t.PAGE_COMPN_ID=c.PAGE_COMPN_ID
                  AND b.DEL_YN='N' AND b.USE_YN='Y'
           )
    """
```

NOT EXISTS 로 *이미 활성 결재가 점유한* 페어를 제외. 시드 시작 전에 이 풀의 크기를 검사한다.

```python
pool = free_page_compn_pool(db, [380, 390, 425])
if len(pool) < target:
    raise RuntimeError(
        f"free page_compn pool({len(pool)}) < target({target}). "
        "purge_qa_atrz_masters() 호출 후 재시도하거나 page_ids 를 확장하세요."
    )
```

**즉시 실패** 정책 — 풀이 부족하면 한 건도 시드하지 않고 RuntimeError. *반쯤 만든 상태로 진행하지 않는다*. 부분 성공이 가장 디버깅하기 힘든 상태다. 60건 중 47건만 만들어지면 page=5 결과가 5건만 나와 페이지 메타 검증이 *예상보다 적다* 는 모호한 메시지로 실패한다.

대신 *명시적* 으로 풀 부족을 알리고 해결법까지 메시지에 박는다. 사용자가 `purge_qa_atrz_masters()` 를 호출하거나 `page_ids` 를 늘려야 한다는 것까지 한 줄에.



## 4축 검증 시나리오

`tests/test_09_corpus_queries.py` 가 corpus 60건으로 검증하는 4축 — 정렬·가시성·검색·페이지.

### 1) 정렬 — 모든 sortable 컬럼 × ASC/DESC

```python
MGMT_SORTABLE = [
    "atrzId", "vsrn", "atrzNm", "atrzSeCd", "atrzSeNm",
    "atrzGdncTypeCd", "atrzGdncTypeNm", "useYn", "useYnLabel",
    "mdfcnDt", "mdfrId", "mdfrNm",
]


@pytest.mark.corpus_sort
@pytest.mark.parametrize("column", MGMT_SORTABLE)
@pytest.mark.parametrize("direction", ["ASC", "DESC"])
def test_mgmt_sort(operator_http, approval_corpus, column, direction):
    body = post_json(... sort={"column": column, "direction": direction})
    items, _ = _envelope_list(body)
    values = [it.get(column) for it in items]
    assert _is_sorted(values, reverse=(direction == "DESC"))
```

parametrize 두 겹으로 **12 컬럼 × 2 방향 = 24 케이스** 가 한 함수로 표현된다. 각 케이스는 한 API 호출 + 응답 순서가 단조 증가/감소인지 검증.

`_is_sorted` 의 한 트릭:

```python
def _is_sorted(values, *, reverse):
    pruned = [v for v in values if v is not None and v != ""]
    if not pruned:
        return True
    key = (lambda v: v.lower()) if isinstance(pruned[0], str) else None
    return pruned == sorted(pruned, reverse=reverse, key=key)
```

- **None / 빈 문자열 제외** — DB null-first vs null-last 정책 차이를 흡수.
- **case-insensitive 비교** — utf8mb4_general_ci collation 과 동일 정책. case-sensitive 비교를 쓰면 'Apple' 과 'banana' 의 순서가 collation 과 어긋난다.

### 2) 가시성 — 직접/역할/외부인 3 케이스

```python
@pytest.mark.corpus_visibility
def test_direct_approver_sees_own_todo(operator_http, approval_corpus):
    item = _pick_direct_corpus_demanded(approval_corpus)
    approver = item.step_approvers[0].aprvr_info
    body = post_json(... user_id=approver)
    assert item.atrz_excn_id in str(body)


def test_outsider_does_not_see_direct_todo(operator_http, approval_corpus):
    item = _pick_direct_corpus_demanded(approval_corpus)
    approver_set = {s.aprvr_info for s in item.step_approvers}
    all_users = {s.aprvr_info for c in approval_corpus
                              for s in c.step_approvers if s.aprvr_se_cd == "01"}
    outsider = next(iter(all_users - approver_set))
    body = post_json(... user_id=outsider)
    assert item.atrz_excn_id not in str(body)


def test_role_member_sees_role_todo(operator_http, approval_corpus):
    item = _pick_role_corpus_demanded(approval_corpus)
    role_step = next(s for s in item.step_approvers
                     if s.aprvr_se_cd == "03" and s.members)
    member = role_step.members[0]
    body = post_json(... user_id=member)
    assert item.atrz_excn_id in str(body)
```

세 케이스의 본질은 **차집합으로 outsider 만들기**. corpus 60건의 전체 결재자 풀에서 *이* 결재의 결재선에 없는 사용자를 1명 뽑는다. 그 사용자가 my_todo 를 호출했을 때 *이* 결재가 안 보여야 한다.

`pytest.skip()` 으로 dynamic skip 도 흥미롭다. *역할 결재 + demand + 멤버 있는 corpus 가 없으면* 스킵. 환경에 따라 corpus 분포가 달라져도 안전.

### 3) 검색 — global / detail T·S·R

```python
@pytest.mark.corpus_search
def test_mgmt_detail_T_atrz_nm(operator_http, approval_corpus):
    """detail T(텍스트) — value='corpus k1' → k1/k10~k14 모두 매칭."""
    body = post_json(... search={
        "column": "DTL",
        "detail": [{"type": "T", "column": "atrzNm", "value": "corpus k1"}],
    })
    items, _ = _envelope_list(body)
    names = [it.get("atrzNm", "") for it in items]
    assert names, "detail T='corpus k1' 결과 비어있음"
    assert all("corpus k1" in n for n in names)
```

검색 type 3 종류:
- **T** (텍스트) — LIKE 검색. `value="k1"` → 'k1', 'k10', 'k11'... 모두 매칭
- **S** (단일 enum) — `value="Y"` → 정확 매칭
- **R** (다중 enum) — `valueList=["EXCN","TODO"]` → IN 절

각 type 의 SQL 변환 규칙이 ListQueryResolver 의 책임이고, corpus 가 *충분히 다양해야* 매칭/비매칭이 둘 다 발생해 회귀를 잡는다. 60건 중 'corpus k1' 매칭은 k1, k10, k11, k12, k13, k14 = 6건. 6건이 모두 보이고 다른 케이스는 안 보이는 게 정답.

### 4) 페이지네이션 — 분할·범위 밖·메타

```python
@pytest.mark.corpus_pagination
def test_mgmt_pagination_split_no_overlap(operator_http, approval_corpus):
    p1 = post_json(... page=1, size=10, sort=...)
    p2 = post_json(... page=2, size=10, sort=...)
    items1, meta1 = _envelope_list(p1)
    items2, _ = _envelope_list(p2)
    ids1 = {it.get("atrzId") for it in items1}
    ids2 = {it.get("atrzId") for it in items2}
    assert len(items1) == 10
    assert len(items2) == 10
    assert not (ids1 & ids2), f"두 페이지 교집합 발생: {ids1 & ids2}"
    assert meta1.get("totalItemSize", 0) >= 50
    assert meta1.get("itemSize") == 10
    assert meta1.get("pageSize", 0) >= 5
```

**두 페이지 교집합이 0** 이 핵심 단언. 페이징 SQL 의 `LIMIT OFFSET` 이 정렬과 함께 결정적인지를 본다. 정렬이 *고유 키* 가 아니면 (예: `mdfcnDt` ASC 만) 같은 timestamp 의 row 가 페이지 경계에 걸쳐 중복 또는 누락이 발생한다. corpus 60건은 *atrzId* (PK) 로 명시 정렬해 이 위험을 통제한다.

메타 3가지:
- `totalItemSize` ≥ 50 — corpus 60건이 모두 카운트됨
- `itemSize` == 10 — 현재 page 의 row 수
- `pageSize` ≥ 5 — `pageSize` 가 *totalPages* 의미라 60/10 = 6 페이지 이상

`page=9999` 범위 밖 케이스도 별도 — 빈 list 반환 + totalItemSize 는 그대로 유지.



## conftest 의 corpus fixture 패턴

`tests/test_09_corpus_queries.py` 가 30+ 테스트인데 시드는 한 번만 만들어야 한다. *session scope* fixture 가 정답.

```python
# approval/conftest.py (개념적 형태)
@pytest.fixture(scope="session")
def approval_corpus(operator_http, db):
    items = seed_corpus(operator_http, db, target=60)
    yield items
    # teardown 은 _purge_qa_session 이 다음 세션 시작 시 'QA ' prefix 일괄 처리
```

한 세션에 60건을 한 번 만들고 30+ 테스트가 그 결과를 공유. 매 테스트가 60건 시드를 다시 만들면 분단위 시간이 폭주한다.

teardown 을 *다음 세션 시작 시* 로 위임한 결정은 디버깅 친화적 — 테스트 실패 후 DB 에 corpus 가 남아 있어 사람이 직접 SELECT 로 검증 가능. 다만 자동화된 CI 에선 이 결정이 누적 부담이 되므로 별도 `purge_qa_atrz_masters()` 같은 명시 cleanup 을 제공해야 한다.



## 함정과 회피

### 함정 1 — Faker seed 의존성

Faker 버전이 바뀌면 같은 seed 가 다른 출력. `requirements.txt` 의 핀이 결정성의 *시간축* 을 잡는다. 버전 업그레이드 시 corpus 의 *모든 검색·정렬 테스트* 가 한 번 깨지므로, Faker 업그레이드는 corpus 회귀 의식 후에만.

### 함정 2 — 실 마스터 의존성

`real_active_users(db)` 가 0명이면 `RuntimeError` 즉시 raise. sbh-local 의 시드 SQL 이 사용자 마스터를 안 만들면 corpus 시드 단계에서 명확히 실패. 그러나 *비활성* 사용자(`USE_YN='N'`) 가 결재선에 들어가면 my_todo 가시성이 미묘하게 어긋난다 — 활성 사용자만 뽑는 SQL 의 WHERE 절이 본질.

### 함정 3 — 부분 성공의 위험

60건 중 47건만 시드되면 page=5 결과가 5건이라 페이지 메타 검증의 정답이 어긋난다. 회피는 *즉시 RuntimeError* — 절반 만들고 raise. 사용자는 *모두 만들거나 아무것도 안 만든다*.

### 함정 4 — 'QA ' prefix 의 implicit contract

corpus 의 atrz_nm 은 `QA corpus k{seq:02d}` 형태로 시작한다. teardown 의 `_purge_qa_session` 이 'QA ' 로 시작하는 모든 결재를 일괄 삭제. *이 prefix 가 contract* 다. 다른 테스트가 'QA' 로 시작하는 결재를 만들면 corpus 와 함께 삭제되어 race condition 이 발생한다. 회피는 prefix 규약을 README 에 명시.

### 함정 5 — corpus 에 의존하는 *다른* 테스트의 격리

같은 session 에 corpus 외 다른 결재 시드도 만들면 `MGMT_SORTABLE` 검색 결과가 corpus 외 항목까지 포함한다. corpus 검증은 *corpus 만* 으로 진행해야 정답이 명확. 회피: corpus 시드 + corpus 테스트가 *같은 session* 안에서만 돌도록 격리.



## 같은 패턴의 다른 응용

Faker + 분포 설계 + 즉시 실패 + session scope fixture 의 조합은 검색·페이지 외 다른 도메인에도 응용 가능.

- **사용자 관리 테스트** — 200명의 한국 이름 사용자, 활성/비활성 비율 8:2, 역할 분포 3종
- **메시징 토픽 테스트** — 500건의 메시지 corpus, eventType 5종, timestamp 분포
- **로그 검색 테스트** — 1000건의 로그 corpus, level 4종, source 10종

핵심은 *충분한 양 + 의도된 분포 + 결정적 seed* 의 3박자. 이게 맞으면 검색·정렬·페이지 같은 *데이터 양이 의미를 만드는* API 의 회귀를 정확히 잡는다.



## TPS 사례 — 두 모듈의 책임 분담

```
suites/approval/_lib/seed_corpus.py    # 60건 시드 생성 (CorpusItem dataclass + StepSpec)
suites/approval/tests/test_09_corpus_queries.py
                                        # 정렬 24 + 가시성 3 + 검색 4 + 페이지 4 = 35 케이스
```

`seed_corpus.py` 의 `seed_corpus()` 함수가 모든 결정을 한 곳에 모은다 — 분포·결재자 round-robin·free pool 검사·즉시 실패·demand 절반. 테스트는 *데이터를 어떻게 만드는가* 를 신경 쓰지 않고 *데이터에 어떤 API 가 작동하는가* 만 검증한다. 이 분리가 시드 정책 변경의 영향을 한 모듈에 가둔다.

`CorpusItem` dataclass 가 모듈 간 contract — 어떤 결재자 구성, 어떤 페이지, 어떤 excnId 가 시드됐는지 모두 *데이터 객체* 로 들고 다닌다. 테스트가 *시드된 결과* 를 한 번 더 SELECT 하지 않고 fixture 반환값을 그대로 활용.

```python
@dataclass
class CorpusItem:
    idx: int
    atrz_id: str
    atrz_nm: str
    page_id: int
    page_compn_id: str
    menu_cd: int
    step_approvers: list[StepSpec]
    atrz_excn_id: str | None = None    # demand 한 경우만 채움
```

`atrz_excn_id` 가 Optional — *demand 된 결재만* 가시성 테스트의 후보. 이 모델링이 테스트의 `_pick_direct_corpus_demanded(corpus)` 같은 헬퍼를 자연스럽게 만든다.
