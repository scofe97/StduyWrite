# chaos fixture — bash subprocess와 인프라 단절

mock으로는 절대 잡을 수 없는 회귀가 있다. Redpanda가 멈췄는데 dispatch가 5xx로 폭발하는 회귀, dev Jenkins로 가는 네트워크가 끊겼는데 SUBMITTED 단계로 정상 진입하지 못하는 회귀. 이런 "인프라가 살아 있다고 가정한 코드"의 깨짐은 진짜 인프라를 잠깐 끊었다 다시 붙여 봐야 드러난다. `qa/python` 은 chaos engineering을 거창한 도구 없이 pytest fixture + bash subprocess + yield/finally 패턴으로 만들었다. 이 챕터는 그 결정과 안전망을 한 그릇에 정리한다.



## 왜 chaos를 통합 테스트에 넣는가

가용성을 흔드는 회귀는 인프라가 정상일 때 안 보인다. dispatch 코드가 Redpanda outbox에 메시지를 넣어 두고 즉시 200을 반환하는 패턴은, Redpanda가 살아 있는 동안에는 outbox에 한 줄이 쌓이고 곧 SENT로 전이된다. Redpanda가 멈춰 있어도 200을 반환해야 — outbox가 PENDING 상태로 쌓이고 복구 후 SENT로 전이되는 — 게 본래 패턴의 가치다. 이걸 검증하지 않으면, 어느 날 outbox 코드를 누가 단순화하면서 broker 직접 publish로 바꿨을 때 회귀가 운영에서 처음 보인다.

같은 이유로 Jenkins 차단 시나리오가 의미 있다. dispatch가 SUBMITTED 단계로 들어왔지만 Jenkins API 호출에서 예외가 나면, executor가 그 예외를 흡수해 SUBMITTED 단계를 유지하는지(=재시도 가능)·곧바로 FAILURE로 떨어뜨리는지가 정책이다. 정책이 깨지면 회귀가 통합 테스트로만 잡힌다.

`qa/python` 의 chaos 그룹 6건이 이런 시나리오를 모은다. 이 챕터는 그중 fixture 패턴이 본질인 두 케이스(`jenkins_blocked`, `redpanda_stopped`)를 풀어 본다.



## subprocess.run — fixture가 외부 명령을 실행하는 표준 모양

pytest fixture가 시스템 인프라를 조작하려면 결국 OS 명령에 도달한다. Python에서 OS 명령을 실행하는 표준 도구는 `subprocess` 모듈이고, 그중 `subprocess.run` 이 한 줄 호출에 가장 자연스럽다.

`executor_jenkins/conftest.py` 의 `_run_chaos` 헬퍼가 정확히 이 패턴이다.

```python
import subprocess

CHAOS = pathlib.Path(__file__).resolve().parent / "chaos"


def _run_chaos(script: str, *args: str) -> None:
    subprocess.run(["bash", str(CHAOS / script), *args], check=True)
```

세 가지를 짚는다. 첫째, **list 인자**. 첫 인자를 list로 넘기면 shell parsing이 없다. `subprocess.run(f"bash {script}", shell=True)` 처럼 shell=True로 실행하면 인자에 공백이 있을 때 SQL injection 비슷한 shell injection이 발생할 수 있다. list 인자는 OS 레벨에서 직접 exec를 호출하므로 인자가 그대로 전달된다. 둘째, **`check=True`**. 명령이 0이 아닌 종료 코드를 내면 `CalledProcessError` 를 던진다. 안 두면 실패한 chaos가 조용히 넘어가서 테스트가 잘못된 가정 위에서 진행된다. 셋째, **절대 경로**. `CHAOS / script` 는 `pathlib.Path` 가 절대 경로를 만든다. 작업 디렉토리에 의존하지 않아 어디서 pytest를 실행해도 같은 스크립트를 찾는다.

대안으로 `subprocess.check_output` 이나 `subprocess.Popen` 이 있다. `check_output` 은 stdout을 캡처해 반환하는 게 다르고, `Popen` 은 비동기 실행에 가깝다. fixture에서 동기적으로 명령을 끝내고 결과만 받는 패턴에는 `subprocess.run(check=True)` 가 가장 단순하다.



## yield + try/finally — 복구를 fixture가 책임진다

chaos fixture의 본질은 *상태를 흔들고, 끝나면 복구한다* 다. yield 패턴이 setup/teardown을 한 함수에 묶고, try/finally가 예외 시에도 복구를 보장한다.

```python
@pytest.fixture
def jenkins_blocked():
    if os.getenv("SKIP_NET_CHAOS") == "1":
        pytest.skip("SKIP_NET_CHAOS=1: 네트워크 chaos 스킵")
    _run_chaos("jenkins_block.sh", "on")
    try:
        yield
    finally:
        _run_chaos("jenkins_block.sh", "off")


@pytest.fixture
def redpanda_stopped():
    _run_chaos("redpanda_stop.sh", "stop")
    try:
        yield
    finally:
        _run_chaos("redpanda_stop.sh", "start")
```

각 fixture가 다음 흐름이다.

1. 사전 가드(skip 조건 등)
2. on/stop으로 인프라 단절
3. yield — 테스트가 단절된 상태에서 동작
4. finally에서 off/start로 복구

`finally` 가 핵심이다. 테스트 함수에서 단언이 실패해 예외가 던져져도 finally가 실행되어 시스템 상태가 원래대로 돌아온다. yield 다음에 단순 `_run_chaos("...", "off")` 만 두면 예외가 나면 복구가 건너뛰어진다.

다만 *프로세스가 강제 종료될 때는 finally가 호출되지 않는다*. SIGKILL, OOM, 사용자가 Ctrl+C 한 뒤 즉시 두 번째 Ctrl+C 같은 경우다. 그래서 chaos 스크립트는 별도로 idempotent하게 만든다 — 사람이 손으로 다시 실행해 정리할 수 있도록.



## /etc/hosts swap — 가장 단순한 네트워크 차단

`chaos/jenkins_block.sh` 는 dev Jenkins 호스트로 가는 트래픽만 골라 차단한다. 도구는 `/etc/hosts` 한 파일이다.

```bash
HOST="${JENKINS_HOST:-jenkins.dev.console.trombone.okestro.cloud}"
MARKER="# tps-qa-jenkins-block"

case "${1:-}" in
  on)
    if grep -q "$MARKER" /etc/hosts; then
      exit 0
    fi
    echo "127.0.0.1 $HOST $MARKER" | sudo tee -a /etc/hosts >/dev/null
    sudo dscacheutil -flushcache 2>/dev/null || true
    sudo killall -HUP mDNSResponder 2>/dev/null || true
    ;;
  off)
    sudo sed -i '' "/$MARKER/d" /etc/hosts
    sudo dscacheutil -flushcache 2>/dev/null || true
    sudo killall -HUP mDNSResponder 2>/dev/null || true
    ;;
esac
```

원리는 단순하다. `/etc/hosts` 에 `127.0.0.1 jenkins.dev.console...` 한 줄을 추가하면 그 호스트로 가는 트래픽이 로컬로 향하고, 로컬에서 listen하지 않으니 connection refused가 된다. off 단계에서 sed로 그 줄만 제거하면 정상 DNS 해석이 돌아온다.

세 가지 디테일이 인상적이다. 첫째, **MARKER 주석**. `# tps-qa-jenkins-block` 라는 고유 마커를 줄 끝에 박아 두면, sed로 제거할 때 다른 entry를 건드리지 않는다. 줄 자체를 식별자로 쓰는 패턴이다. 둘째, **idempotent 체크**. on에서 마커가 이미 있으면 즉시 exit 0. fixture가 두 번 실행되는 사고에서도 안전하다. 셋째, **DNS 캐시 flush**. macOS는 `dscacheutil -flushcache` 와 `killall -HUP mDNSResponder` 두 명령으로 캐시를 비운다. 이걸 안 하면 캐시된 IP로 한동안 정상 호스트에 도달한다.

이 패턴이 `pfctl` 같은 본격 방화벽 도구보다 단순하다. pfctl은 룰셋 변경에 root 권한과 anchor 설정이 필요하고 macOS·Linux 사이에 명령이 다르다. `/etc/hosts` 는 모든 unix 계열에서 동일하게 동작한다. 네트워크 *전체*를 끊지 않고 *특정 호스트만* 끊는 격리 효과도 같이 얻는다.

다만 `/etc/hosts` swap에는 sudo가 필요하다. CI 또는 권한이 없는 환경에서 이 fixture가 무조건 실패하면 다른 chaos까지 막힌다. 그래서 환경변수 가드를 둔다.

```python
if os.getenv("SKIP_NET_CHAOS") == "1":
    pytest.skip("SKIP_NET_CHAOS=1: 네트워크 chaos 스킵")
```

`SKIP_NET_CHAOS=1` 이면 fixture가 시작 단계에서 `pytest.skip` 으로 건너뛴다. 같은 그룹의 다른 chaos(예: redpanda)는 영향받지 않는다. 환경별 회피 안전망의 표준 모양이다.



## docker compose — 컨테이너 단위 단절과 healthy 폴링

Redpanda 차단은 호스트가 아니라 컨테이너를 멈추는 모양이다. `chaos/redpanda_stop.sh` 가 `docker compose stop|start` 를 감싼다.

```bash
COMPOSE_FILE="$REPO_ROOT/message-lib/docker/docker-compose.yml"

case "${1:-}" in
  stop)
    docker compose -f "$COMPOSE_FILE" stop redpanda
    ;;
  start)
    docker compose -f "$COMPOSE_FILE" start redpanda
    for i in $(seq 1 30); do
      if docker compose -f "$COMPOSE_FILE" ps redpanda | grep -qi healthy; then
        exit 0
      fi
      sleep 2
    done
    echo "[redpanda] start 했으나 healthy 도달 실패" >&2
    ;;
esac
```

stop은 단순하다. compose 명령으로 redpanda 컨테이너만 stop. 같은 compose 파일의 다른 서비스(zookeeper·schema-registry 등)는 영향받지 않는다.

start가 흥미롭다. compose가 컨테이너를 start해도 healthy 상태에 도달하기까지 시간이 걸린다. 그동안 fixture가 yield를 마치고 다음 테스트가 시작되면, redpanda가 부팅 중이라 dispatch가 의도와 다르게 실패한다. 그래서 `docker compose ps` 출력에서 `healthy` 문자열을 grep으로 대기한다. 최대 60초(2초 × 30회).

이 healthy 폴링이 *fixture 안*이 아니라 *bash 스크립트 안*에 있는 점이 결정적이다. fixture는 단순히 스크립트가 0으로 끝나길 기다리고, healthy 대기는 OS 레벨에서 처리된다. 책임 경계가 깔끔하다.

healthy를 기다리는 시간이 길어지면 다음 테스트가 시작되기까지 지연된다. `qa/python` 은 chaos 그룹을 `--maxfail=1` 로 돌려 한 번에 한 케이스씩 처리하므로 지연이 누적되지 않는다.



## SKIP_NET_CHAOS — 환경별 부분 비활성

CI 환경, 사내망 외부, 권한이 없는 머신에서 일부 chaos를 건너뛰어야 하는 경우가 있다. `qa/python` 의 결정은 *환경변수 한 개*다.

```python
if os.getenv("SKIP_NET_CHAOS") == "1":
    pytest.skip("SKIP_NET_CHAOS=1: 네트워크 chaos 스킵")
```

`SKIP_NET_CHAOS=1` 환경변수가 있으면 fixture가 pytest.skip을 호출한다. 그 fixture를 받는 테스트만 SKIPPED로 처리되고, 같은 chaos 그룹의 다른 테스트(예: redpanda·DB)는 정상 실행된다.

CI 파이프라인이 chaos 그룹 전체를 돌리되 네트워크 부분만 건너뛰고 싶으면 `SKIP_NET_CHAOS=1 pytest -m chaos` 한 줄이다. 더 세밀한 제어가 필요하면 fixture별로 별도 환경변수를 둘 수 있다(`SKIP_DOCKER_CHAOS=1`).

이 가드가 *fixture 시작 첫 줄*에 있는 점이 중요하다. 단절 명령을 실행하기 전에 skip을 결정하므로, 환경이 맞지 않을 때 시스템 상태를 변경하지 않는다.



## chaos가 잡는 회귀의 종류

chaos 그룹 6건이 노리는 회귀를 정리하면 다음과 같다.

| 케이스 | 노리는 회귀 |
|--------|-------------|
| jenkins_blocked + dispatch | Jenkins API 호출 실패가 5xx로 새 나오는 회귀, SUBMITTED에 머물지 않는 회귀 |
| redpanda_stopped + dispatch | broker 직접 publish로 회귀해 outbox 패턴이 깨진 경우, 5xx 누수 |
| 미존재 tlId | 입력 검증 누락으로 서버 측 NPE/IllegalArgumentException 누수 |
| 종료 jobExcnId 재사용 | idempotent 정책 회귀 |
| 잘못된 jobVsrn (parametrize 3건) | 입력 검증 누락 |
| DB lock 경합 | connection pool 소진으로 5xx 누수 |

각 케이스가 전혀 다른 종류의 회귀를 노리고 있다는 점이 중요하다. chaos는 "한 줄로 모든 회귀를 잡는" 것이 아니라 "운영에서 보일 가능성이 큰 시나리오를 미리 줄 세우는" 도구다.



## 함정 모음

**finally가 안 호출되는 강제 종료**. SIGKILL, OOM 같은 경우 finally는 안 돈다. 결과적으로 `/etc/hosts` 에 차단 entry가 남거나 redpanda가 멈춘 채 유지된다. 사람이 손으로 정리해야 한다. chaos 스크립트의 status 모드(`./jenkins_block.sh status`)가 이때 디버깅 첫 도구다. CI에서는 jobs 종료 후 정리 단계를 별도로 두는 게 안전하다.

**다른 테스트로의 영향**. fixture가 시스템 상태를 흔드는 동안 *같은 chaos 그룹의 다른 테스트가 영향받지 않게* 격리해야 한다. `qa/python` 은 chaos를 직렬 실행(`-n` 안 붙임)으로 격리한다. xdist로 병렬화하면 worker 간 state 충돌이 일어난다.

**docker compose 의존성**. redpanda 스크립트는 `message-lib/docker/docker-compose.yml` 의 정확한 경로에 의존한다. 디렉토리 구조가 바뀌면 스크립트가 깨진다. 스크립트 안의 `REPO_ROOT="$(cd "$(dirname "$0")/../../../.." && pwd)"` 가 위치 기반으로 추적하지만, 깊이가 바뀌면 따라간다.

**sudo 비밀번호 프롬프트**. macOS의 `/etc/hosts` 수정에는 sudo가 필요하고, 프롬프트가 뜨면 fixture가 멈춘다. `sudo -v` 로 미리 sudoers timestamp를 갱신해 두거나, sudoers 설정에 NOPASSWD를 추가하면 프롬프트가 안 뜬다. CI에서는 root 사용자로 돌리는 편이 단순하다.

**스크립트 권한**. 첫 실행에서 `Permission denied: chaos/jenkins_block.sh` 가 나오면 `chmod +x chaos/*.sh` 로 권한을 추가한다. git에 커밋된 파일의 실행 권한은 OS와 git config에 따라 달라질 수 있다.

**호스트 변경에 대한 적응**. `JENKINS_HOST` 환경변수로 차단 대상 호스트를 override할 수 있다. dev 환경 외 다른 stage에서 같은 fixture를 재사용할 때 유용하다.



## cleanup_jenkins.sh — 운영 위생 스크립트의 분리

같은 디렉토리의 `cleanup_jenkins.sh` 는 학습 가치가 다르다. fixture가 아니라 운영 위생 스크립트다 — Jenkins API로 QA 폴더 하위 Job을 일괄 삭제한다. 핵심은 위생 자동화 패턴이다.

```bash
JOBS=$(curl -fsS -u "$AUTH" "$LIST_URL" | python3 -c '
import json, sys
data = json.load(sys.stdin)
for j in data.get("jobs", []):
    print(j["name"])
')
```

curl 출력을 python3 한 줄로 파싱해 이름만 추출하는 패턴이다. jq 같은 외부 도구 의존을 줄이고 어떤 환경에서도 동작한다. 이런 운영 스크립트가 chaos fixture와 함께 살아 있으면, fixture finally가 누락한 부수효과(QA 폴더에 남은 Job들)를 사람이 일괄 청소할 수 있다.



## 비교 — toxiproxy / Pumba 같은 본격 chaos 도구와의 거리

chaos engineering의 본격 도구 두 가지를 짚는다.

**toxiproxy** (Shopify). TCP 프록시 위에서 latency·bandwidth·connection drop 등 다양한 toxic을 주입한다. HTTP API로 toxic을 동적 추가/제거할 수 있다. fixture 안에서 toxiproxy API를 호출해 latency 1초를 주입하고, 끝나면 제거하는 패턴이 자연스럽다.

**Pumba**. docker 컨테이너 단위로 chaos를 주입한다. 컨테이너 kill·pause·network emulation(latency·packet loss)를 명령 한 줄로 만든다. compose 환경에서 자연스럽다.

`qa/python` 이 이 도구들을 안 쓰고 bash 스크립트로 간 이유는 두 가지다. 첫째, **시나리오가 이분법적**. "Jenkins로 가는 트래픽이 끊기는가/안 끊기는가"라 latency 같은 fine-grained 제어가 필요 없다. /etc/hosts on/off로 충분하다. 둘째, **추가 의존성 회피**. toxiproxy를 띄우려면 toxiproxy 서버를 별도로 운영해야 하고, 컨테이너 환경에 의존이 늘어난다.

규모가 커지고 latency·packet loss 같은 fine-grained chaos가 필요해지면 toxiproxy로 가는 게 자연스럽다. 현재의 단순함은 의도된 트레이드오프다.
