# pymysql 직접 검증과 폴링 패턴

HTTP 응답이 200으로 떨어졌다고 해서 시스템이 진짜 의도한 일을 했는지 알 수는 없다. 비동기로 처리되는 작업은 응답 시점과 종료 시점이 분리되고, 도메인 검증은 결국 DB에 무엇이 저장됐는지로 확정된다. `qa/python` 은 ORM을 거치지 않고 raw SQL로 DB를 직접 들여다본다. 이 챕터는 pymysql의 cursor 사용·placeholder 안전성·동적 IN 절·polling 패턴·`FOR UPDATE` 락 경합 시뮬레이션을 한 그릇에 정리한다.



## 왜 ORM이 아니라 raw SQL인가

애플리케이션 코드라면 ORM이 우선이다. SQLAlchemy ORM이나 Django ORM은 객체-테이블 매핑·세션 관리·쿼리 빌더를 한꺼번에 제공해 비즈니스 로직의 표현력을 끌어올린다.

검증 테스트는 반대 방향이다. ORM은 *애플리케이션이 본 세상*을 보여 주지만, 검증의 목적은 *DB에 진짜 무엇이 저장됐는지* 확인하는 것이다. 같은 모델 정의를 검증 코드에서 다시 import하면 매핑이 잘못된 경우 양쪽이 똑같이 잘못 본다. 문자열 SQL은 매핑 위에서 작성되지 않아 이 가짜 신뢰를 끊는다.

```python
# 검증 코드의 의도: 정확히 한 row만 있어야 한다
with db.cursor() as cur:
    cur.execute(
        "SELECT COUNT(*) FROM TB_TRB_EC_001 WHERE JOB_EXCN_ID=%s",
        (body["jobExcnId"],),
    )
    assert cur.fetchone()[0] == 1
```

이 단언은 "이 ID로 진짜 한 row만 있는가"를 묻는다. JPA가 같은 ID로 두 row를 만들었다면 ORM 기반 검증은 한 row만 잘못 보여 줄 수 있지만, 위 SQL은 두 row를 정확히 잡는다.

raw SQL의 두 번째 가치는 *데이터베이스 격리 수준과 락 동작을 그대로 들여다볼 수 있다*는 점이다. `FOR UPDATE` 로 락을 잡는 시나리오는 ORM 추상화로는 표현이 어색하지만 SQL로는 한 줄이다.



## qa/python의 db fixture와 cursor 패턴

루트 conftest의 db fixture는 짧다.

```python
@pytest.fixture
def db():
    conn = pymysql.connect(
        host=DB_HOST, user=DB_USER, password=DB_PASS, database=DB_NAME,
        autocommit=True,
        cursorclass=pymysql.cursors.Cursor,
    )
    try:
        yield conn
    finally:
        conn.close()
```

세 결정이 들어 있다. 첫째, `autocommit=True` 라 매 SQL 문이 곧 트랜잭션이다. 검증 SELECT가 다른 세션의 변경을 즉시 보고, 테스트 코드는 트랜잭션 경계를 신경 쓰지 않아도 된다. 둘째, `cursorclass=pymysql.cursors.Cursor` 는 기본값이지만 명시했다. fetch 결과가 tuple로 떨어지므로 인덱스 접근(`row[0]`)으로 단언한다. dict 형태가 더 편하면 `pymysql.cursors.DictCursor` 로 바꿀 수 있다. 셋째, fixture가 function scope라 매 테스트마다 새 connection이 만들어진다.

테스트에서 cursor는 context manager로 받는다.

```python
with db.cursor() as cur:
    cur.execute("SELECT EXCN_STTS FROM TB_TRB_EC_001 WHERE JOB_EXCN_ID=%s", (excn_id,))
    row = cur.fetchone()
```

`with` 블록을 빠져나오면 cursor가 닫힌다. cursor는 connection의 자식 자원이라 cursor를 닫지 않으면 connection 안에서 자원이 쌓인다. 매 SQL을 `with` 로 감싸는 습관이 누수를 막는다.



## placeholder — SQL injection 방어와 타입 변환

pymysql의 cursor는 SQL 문자열에 `%s` placeholder를 받고, 실제 값은 두 번째 인자(tuple 또는 list)로 받는다.

```python
cur.execute(
    "SELECT EXCN_STTS FROM TB_TRB_EC_001 WHERE JOB_EXCN_ID=%s",
    (excn_id,),
)
```

이 분리는 두 가지를 한 번에 한다. **SQL injection 방지**(값이 절대 문자열 결합으로 들어가지 않으므로 `' OR 1=1 --` 같은 입력이 SQL 구조를 바꾸지 못한다)와 **타입 변환**(Python의 `int`·`datetime`·`bytes` 가 자동으로 SQL 표현으로 변환된다).

문자열 결합으로 SQL을 만들면 두 보호가 동시에 사라진다.

```python
# 절대 금지
cur.execute(f"SELECT ... WHERE JOB_EXCN_ID='{excn_id}'")
```

테스트 코드 안에서 hard-coded 값을 결합하는 건 운영 코드보다 위험이 적지만, 한 번 패턴이 생기면 점차 사용자 입력 영역으로 새어 나간다. 처음부터 placeholder만 쓰는 습관이 일관된 안전망이다.

placeholder 형식이 다른 DB 드라이버 — `psycopg2`(PostgreSQL)도 `%s` 를 쓰지만, sqlite3는 `?` 를 쓴다. SQL Server용 pyodbc는 `?`. **drivar 마다 placeholder 형식이 다르다는 점이 가장 자주 만나는 첫 함정**이다. `qa/python` 은 MariaDB라 `%s` 를 그대로 쓴다.



## 동적 IN 절 — placeholder 개수 맞추기

concurrency 테스트에서는 한 테스트 안에서 30~50개의 jobExcnId 를 생성하고, 끝난 뒤 한 번의 SELECT로 모두 확인한다. IN 절에 list를 그대로 넣을 수는 없으니, placeholder를 동적으로 만든다.

```python
excn_ids = [eid for _, eid in results]   # 30개 ID

placeholders = ",".join(["%s"] * len(excn_ids))
with db.cursor() as cur:
    cur.execute(
        f"SELECT JOB_KND_CD, COUNT(*) FROM TB_TRB_EC_001 "
        f"WHERE JOB_EXCN_ID IN ({placeholders}) GROUP BY JOB_KND_CD",
        excn_ids,
    )
    kind_counts = dict(cur.fetchall())
```

`",".join(["%s"] * 30)` 은 `"%s,%s,%s,...,%s"` 가 된다. 두 번째 인자 `excn_ids` 의 길이와 placeholder 개수가 일치하면 pymysql이 알아서 매칭한다. 길이가 다르면 `pymysql.err.ProgrammingError` 가 던져진다. 두 번째 인자는 list든 tuple이든 받아 준다.

이 패턴이 placeholder 안전성을 깨는 것처럼 보이지만 *placeholder 개수만 동적*이고 *값 자체는 여전히 분리된 인자*라 SQL 구조에는 사용자 입력이 들어가지 않는다. 안전한 동적 SQL의 표준 모양이다.

대안으로 SQLAlchemy Core의 `expanding bindparam` 같은 추상화가 있지만, 의존성을 추가하는 비용 대비 얻는 이득이 작다. 한 줄 join이면 충분하다.



## wait_status — 폴링 패턴의 표준 모양

dispatch는 비동기 작업이다. HTTP 응답이 200으로 떨어진 뒤에도 executor 내부에서 SUBMITTING → SUBMITTED → QUEUED → RUNNING → DONE_SUCCESS 같은 상태 전이가 이어진다. 검증의 본질은 "최종 상태가 terminal 집합에 속하는가"이고, 그 시점은 미리 알 수 없다. 폴링이 답이다.

`_lib/helpers.py` 의 `wait_status` 가 정확히 이 패턴이다.

```python
TERMINAL_OK = {"DONE_SUCCESS", "DONE_FAILURE", "FINALIZED", "ABORTED"}


def wait_status(
    db,
    excn_id: str,
    predicate: Callable[[str], bool],
    timeout: float = 120,
    interval: float = 1.0,
) -> str:
    deadline = time.time() + timeout
    last: str | None = None
    while time.time() < deadline:
        with db.cursor() as cur:
            cur.execute(
                "SELECT EXCN_STTS FROM TB_TRB_EC_001 WHERE JOB_EXCN_ID=%s",
                (excn_id,),
            )
            row = cur.fetchone()
        if row:
            last = row[0]
            if predicate(last):
                return last
        time.sleep(interval)
    raise TimeoutError(f"{excn_id} did not reach predicate (last={last})")
```

다섯 가지를 짚는다. 첫째, **deadline 방식**. `time.time() + timeout` 으로 절대 시각을 만들고 매 반복마다 비교한다. 반복 횟수 기반(`for _ in range(N)`)은 interval 변경 시 timeout이 함께 바뀌어 추적이 어려워진다. 둘째, **predicate 함수**. 호출자가 lambda로 종료 조건을 정의한다. happy 그룹은 `lambda s: s in TERMINAL_OK`, chaos 그룹은 `lambda s: s in {"SUBMITTED", "PENDING"}` 식으로 같은 함수를 다르게 쓴다. 셋째, **last 변수**. 마지막 관측값을 들고 있다가 timeout 발생 시 메시지에 포함시킨다. "어디서 멈췄는지"가 디버깅에 결정적이다. 넷째, **row 누락 처리**. 시드 INSERT 직후 select가 즉시 row를 못 보는 race도 가능하다. `if row:` 가드로 None을 흡수한다. 다섯째, **TimeoutError 명시**. 단순 return으로 끝내면 호출자가 timeout인지 predicate 만족인지 구분 못 한다. 별도 예외 타입으로 갈라 둔다.

호출 예는 한 줄이다.

```python
final = wait_status(
    db,
    body["jobExcnId"],
    lambda s: s in TERMINAL_OK,
    timeout=300,
)
assert final in TERMINAL_OK
```

predicate가 lambda로 분리되어 한 함수가 다양한 종료 조건을 받는다. 이게 함수형 fixture의 강점이다.

폴링 interval은 도메인에 맞춘다. dispatch → terminal까지가 보통 수십 초 단위라 1초가 적당하다. 더 짧게 잡으면 DB 부하가 늘고, 더 길게 잡으면 검출이 느려진다.



## FOR UPDATE — 락 경합을 의도적으로 만들기

chaos 그룹의 `test_db_lock_contention` 은 흥미로운 패턴을 쓴다. 한 connection이 `FOR UPDATE` 로 row를 잡고, 같은 시점에 dispatch HTTP 호출을 보낸다. dispatch가 5xx로 폭발하지 않고 timeout이나 4xx로 정상 종료되는지를 본다.

```python
@pytest.mark.chaos
def test_db_lock_contention(http):
    side = pymysql.connect(
        host=os.getenv("DB_HOST", "10.255.17.102"),
        user=os.getenv("DB_USER", "trbuser"),
        password=os.getenv("DB_PASS", "cloud1234"),
        database=os.getenv("DB_NAME", "TPS305P"),
        autocommit=False,           # ← 여기는 명시적 트랜잭션
    )
    try:
        with side.cursor() as cur:
            cur.execute("SELECT * FROM TB_TRB_EC_001 LIMIT 1 FOR UPDATE")
        body = dispatch_payload(job_path="job/QA/job/CHAOS-DBLOCK-01")
        r = http.post(f"{EXECUTOR_BASE}/jobs/dispatch", json=body, timeout=15)
        assert r.status_code < 500, f"lock 경합에서 5xx: {r.status_code} {r.text}"
    finally:
        side.rollback()
        side.close()
```

세 가지가 결정적이다. 첫째, **별도 connection**. fixture의 `db` 와 다른 connection을 새로 만든다. fixture connection은 autocommit이라 락을 오래 잡을 수 없다. 락은 트랜잭션 안에서만 의미가 있어 별도 connection을 `autocommit=False` 로 띄운다. 둘째, **트랜잭션을 명시적으로 끝낸다**. `finally` 에서 `rollback()` 으로 락을 푼다. commit이 아닌 rollback이면 테스트가 실제 데이터에 자국을 남기지 않는다. 셋째, **timeout=15**. dispatch가 락 때문에 영원히 대기하지 않도록 클라이언트 timeout을 명시한다.

이 패턴이 잡는 회귀는 *"락 대기 중 connection pool 소진 → 5xx"* 같은 시나리오다. 운영에서 보일 수 있는 패턴이라 통합 테스트에서 미리 확인한다.



## autocommit — 켜고 끄기의 의미

pymysql connection을 `autocommit=True` 로 띄우면 INSERT/UPDATE/DELETE가 즉시 commit된다. 검증 SELECT는 항상 최신 상태를 본다.

`autocommit=False` (기본값)이면 명시적 `commit()` 또는 `rollback()` 으로 트랜잭션을 닫아야 한다. 안 닫으면 같은 connection의 다음 SQL은 같은 트랜잭션 안에 있고, **다른 connection은 변경을 보지 못한다**. polling 검증에서 다른 세션의 변경을 즉시 봐야 하므로, fixture connection은 `autocommit=True` 가 맞다.

lock contention 테스트는 반대로 `autocommit=False` 가 필요하다. 트랜잭션을 명시적으로 열고 닫아야 락이 의미를 갖는다. 같은 fixture를 두 가지 모드로 쓰는 건 헷갈리니, 한쪽은 fixture를 쓰고 다른 한쪽은 별도 connection을 새로 만드는 편이 안전하다. `qa/python` 의 결정이 그것이다.



## 함정 모음

**connection 누수**. fixture의 finally가 connection을 닫지만, fixture 밖에서 만든 connection(예: lock contention의 side connection)은 직접 닫아야 한다. `try/finally` 안의 `side.close()` 가 그 역할이다. 누수가 누적되면 DB 측 max_connections에 걸려 다음 테스트가 모두 실패한다.

**기본 격리 수준**. MariaDB/MySQL의 기본 격리 수준은 REPEATABLE READ다. 한 트랜잭션 안에서 같은 SELECT를 두 번 하면 두 번째 결과가 첫 번째와 같음을 보장한다. polling 패턴에서는 이게 *문제*가 된다. autocommit이 켜져 있어 매 SELECT가 독립 트랜잭션이라 외부 변경을 본다. autocommit을 끄고 polling을 쓰면 첫 결과 이후 영원히 같은 값만 본다. **polling = autocommit 권장** 이다.

**fetch 모드와 메모리**. `cursor.fetchall()` 은 모든 row를 메모리에 올린다. 검증 SELECT가 수십~수백 row면 문제없다. 큰 테이블을 처리하려면 `fetchmany(N)` 또는 `for row in cur:` 형태로 스트리밍한다.

**날짜 timezone 혼선**. MySQL 측 `TIMESTAMP` 컬럼은 서버 timezone에 의존한다. Python 측 `datetime` 은 naive(타임존 없음) 가 기본이다. 두 timezone이 다르면 단언이 미묘하게 어긋난다. `qa/python` 은 시간 단언을 쓰지 않아 이 함정을 피했다. 시간 비교가 필요하면 `pytz` 또는 `zoneinfo` 로 양쪽을 같은 timezone에 맞춰 비교한다.

**예외 발생 시 cursor 상태**. cursor 사용 중 예외가 나면 cursor는 미완성 상태로 남을 수 있다. `with db.cursor()` 컨텍스트 매니저가 정리를 책임진다. 이걸 안 쓰고 명시적으로 `cur = db.cursor()` 한 뒤 예외를 만나면 다음 SQL이 실패한다.



## 비교 — pymysql vs SQLAlchemy vs psycopg2

**pymysql**. 순수 Python으로 구현된 MySQL/MariaDB 드라이버. C extension이 없어 설치가 단순하고 어떤 환경에서도 같은 동작을 보인다. 성능은 mysqlclient(C extension) 보다 느리지만 통합 테스트 부하에서는 문제없다. `qa/python` 의 선택.

**SQLAlchemy Core**. raw SQL 위에 query builder와 connection pool을 추상화한 층. 같은 코드를 PostgreSQL·MariaDB·SQLite에서 돌릴 수 있다. 검증 테스트에 쓰면 placeholder 형식 차이 같은 디테일을 SQLAlchemy가 흡수해 준다. 다만 의존성이 무거워지고, 추상화 한 단계가 raw SQL의 직접성을 흐린다.

**SQLAlchemy ORM**. 객체-테이블 매핑까지 포함. 애플리케이션 코드에는 적합하지만 검증 테스트에는 위험하다 — 같은 매핑을 양쪽이 공유하면 매핑 결함이 검증으로 잡히지 않는다.

**psycopg2 (PostgreSQL)**. PostgreSQL 표준 드라이버. placeholder 형식이 `%s` 로 pymysql과 같지만, named placeholder(`%(name)s`)는 다르게 동작한다. PostgreSQL의 `RETURNING` 절·배열 타입처럼 PostgreSQL 고유 문법은 raw SQL이 자연스러우니 검증 테스트에서 자주 보인다.

`qa/python` 이 pymysql + raw SQL을 고른 이유는 두 가지로 모인다. 첫째, MariaDB 단일 환경이라 멀티 DB 추상화의 이득이 없다. 둘째, 검증의 본질이 "DB가 진짜 무엇을 저장했는가"라서 추상화 층이 시야를 가리지 않는 편이 좋다.
