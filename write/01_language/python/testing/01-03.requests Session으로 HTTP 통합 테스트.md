# requests Session으로 HTTP 통합 테스트

통합 테스트는 외부 프로세스의 HTTP 엔드포인트를 진짜로 두드리는 일이 본질이다. 한 케이스마다 `requests.post(...)` 를 호출해도 동작은 하지만, 케이스가 수십 건이 되면 connection pool이 한 번도 재사용되지 않고 헤더 설정이 매번 반복된다. `requests.Session` 은 이 두 비효율을 한 번에 해결한다. 이 챕터는 Session 재사용·basic auth·timeout·상태 코드 단언 정책을 한 그릇에 정리한다.



## 왜 Session인가

`requests.post(url, ...)` 와 같은 모듈 함수도 내부적으로는 매번 임시 Session을 만들어 쓴다. 케이스가 1건이면 차이가 없지만, 30건이면 30번의 SSL 핸드셰이크와 connection 생성이 일어난다. `qa/python` 은 happy 그룹 10건만 해도 한 테스트 함수가 parametrize로 10번 실행되고, concurrency 그룹은 한 테스트 안에서 50번 동시 호출이 일어난다. Session을 session scope fixture로 한 번 만들어 두면 테스트 세션 전체에서 connection pool과 기본 헤더가 공유된다.

Session의 두 번째 가치는 *상태 표현*이다. 헤더, cookie, basic auth는 Session에 한 번 설정하면 그 이후의 모든 요청에 자동으로 따라붙는다. `s.headers.update({"userId": "tester"})` 한 줄이면 모든 요청에 그 헤더가 붙는다. 한 테스트 함수의 단언 코드에서 헤더 노이즈가 사라지면 가독성이 한 단계 올라간다.



## qa/python의 http와 jenkins fixture

루트 conftest에는 두 종류의 Session이 정의되어 있다.

```python
@pytest.fixture(scope="session")
def http() -> requests.Session:
    s = requests.Session()
    s.headers.update({"userId": USER_ID, "Content-Type": "application/json"})
    return s


@pytest.fixture(scope="session")
def jenkins() -> requests.Session:
    s = requests.Session()
    user = os.getenv("JENKINS_USER")
    token = os.getenv("JENKINS_TOKEN")
    if user and token:
        s.auth = (user, token)
    return s
```

두 Session은 역할이 다르다. `http` 는 executor REST(`http://localhost:8092/executor/api`)에 dispatch를 쏠 때 쓴다. JSON 요청이 본질이라 Content-Type을 기본 헤더로 박아 둔다. `userId` 는 사내 게이트웨이가 호출자 식별에 쓰는 값이다. `jenkins` 는 dev Jenkins(`https://jenkins.dev.console.trombone.okestro.cloud`)에 직접 호출할 때 쓴다. basic auth로 자격증명이 필요한데, `JENKINS_USER` 와 `JENKINS_TOKEN` 환경변수가 없으면 auth 없이 만들어 준다(테스트가 jenkins fixture를 받지만 실제로는 안 쓰는 케이스를 위해).

테스트 함수는 인자 이름만 적어 받는다.

```python
def test_happy_dispatch_to_terminal(http, db, kind, job_path):
    body = dispatch_payload(kind=kind, job_path=job_path)
    r = http.post(f"{EXECUTOR_BASE}/jobs/dispatch", json=body)
    assert r.status_code == 200, f"dispatch failed: {r.status_code} {r.text}"
```

`http.post` 호출에는 url과 `json=body` 만 보인다. 헤더·세션·키프얼라이브는 fixture가 다 흡수했다.



## json= 파라미터 — 직렬화와 헤더가 한 번에

requests는 `json=` 파라미터를 받으면 두 가지를 자동으로 한다. 첫째, dict를 `json.dumps` 로 직렬화해 body에 넣는다. 둘째, `Content-Type: application/json` 헤더를 자동으로 설정한다(이미 같은 헤더가 있으면 덮지 않음).

```python
# 동등한 두 호출
r = http.post(url, json={"a": 1})

# 위와 동일
r = http.post(
    url,
    data=json.dumps({"a": 1}),
    headers={"Content-Type": "application/json"},
)
```

`qa/python` 은 Session의 기본 헤더에 `Content-Type: application/json` 이 이미 박혀 있으니 `json=` 파라미터의 자동 헤더는 사실상 중복 설정이다. 그래도 명시적으로 `json=` 을 쓰는 편이 좋은 이유는 직렬화 책임이 호출 시점으로 옮겨와 의도가 분명해지기 때문이다. body를 미리 string으로 만들어 `data=` 로 넘기면 dict 변형이나 직렬화 실수가 호출 직전까지 잠복한다.

직렬화에서 한 가지 함정이 있다. `datetime` 객체나 `Decimal` 같은 타입은 기본 JSON 직렬화기가 처리하지 못한다. `qa/python` 의 `dispatch_payload` 는 `time.strftime("%Y-%m-%dT%H:%M:%S")` 로 미리 string을 만들어 dict에 넣는다. 직렬화 단계에서 타입 에러가 나는 것보다 payload 빌드 단계에서 명시적으로 변환하는 편이 디버깅이 쉽다.



## 상태 코드 단언 — 5xx는 회귀, 4xx는 의도

통합 테스트의 단언 정책에서 가장 미묘한 부분이 상태 코드다. `qa/python` 의 happy 그룹은 단순하다.

```python
assert r.status_code == 200, f"dispatch failed: {r.status_code} {r.text}"
```

happy는 정상 흐름이라 200만 받는다. 200이 아니면 즉시 실패. 메시지에 `r.text` 를 넣어 실패 원인이 응답 본문에 들어 있는 경우 디버깅을 돕는다.

lifecycle 그룹은 더 흥미롭다.

```python
r2 = _post(http, body)
assert r2.status_code in (200, 400, 409, 422), (
    f"duplicate dispatch returned unexpected status: {r2.status_code} {r2.text}"
)
```

같은 `jobExcnId` 로 두 번째 dispatch를 보냈을 때 정책상 두 결과가 모두 받아들여진다 — 200으로 idempotent 응답하거나, 4xx로 거부하거나. 단 5xx는 회귀로 본다. 단언을 `in (200, 400, 409, 422)` 로 두면 정책이 코드에 그대로 표현되고, 만약 503이 떨어지면 즉시 실패한다.

chaos 그룹은 한 단계 더 나아간다.

```python
assert all(code < 500 for code, _ in results), \
    f"5xx 발생: {[r for r in results if r[0] >= 500]}"
```

50 burst 부하에서 모든 응답이 500 미만이어야 한다고 단언한다. 200이 아니어도 좋다. 4xx로 거절하든 200으로 받든 서버가 통제권을 잃지 않으면 통과. 5xx는 명백한 회귀.

이 정책의 핵심은 **5xx와 4xx의 분리**다. 4xx는 도메인 응답으로 본다 — "내가 거절한다"라는 의식적 결정. 5xx는 "내가 통제권을 잃었다"는 의미라 통합 테스트가 잡아야 한다. 단언을 `== 200` 으로 좁게 두면 정상 동작하는 4xx 거절도 회귀로 오해된다. `< 500` 으로 두면 회귀의 본질만 잡는다.



## basic auth — Session.auth로 한 번만

Jenkins API는 basic auth를 받는다. requests는 `Session.auth = (user, token)` 한 줄로 모든 후속 요청에 Authorization 헤더를 자동 추가한다.

```python
s.auth = (user, token)
# 이후 모든 s.get/post 호출에 Authorization: Basic <base64> 자동 추가
```

같은 동작을 매 요청마다 명시하려면 `auth=HTTPBasicAuth(user, token)` 을 인자로 넘겨야 한다. Session 레벨로 한 번만 설정하면 보일러플레이트가 사라진다.

`qa/python` 의 jenkins fixture는 환경변수가 없으면 auth를 설정하지 않는다.

```python
user = os.getenv("JENKINS_USER")
token = os.getenv("JENKINS_TOKEN")
if user and token:
    s.auth = (user, token)
```

테스트가 `jenkins` fixture를 받지만 실제로 jenkins를 호출하지 않는 케이스에서는 자격증명이 없어도 fixture가 실패하지 않게 한다. 단순한 안전망이지만 CI나 사내망 외부에서 일부 그룹만 돌릴 때 도움이 된다.



## timeout — 빠뜨리면 무한 대기

requests의 `timeout=` 파라미터를 안 주면 응답이 안 올 때 영원히 기다린다. 통합 테스트에서 한 케이스가 무한 대기에 빠지면 CI 전체가 막힌다.

```python
r = http.post(f"{EXECUTOR_BASE}/jobs/dispatch", json=body, timeout=15)
```

`qa/python` 의 lock contention 테스트가 `timeout=15` 를 명시한 이유다. 다른 세션이 `FOR UPDATE` 로 row를 잡고 있는 상태에서 dispatch를 보내면 서버 측 트랜잭션이 row를 못 잡고 대기한다. timeout을 명시하지 않으면 그 대기가 그대로 클라이언트로 전파된다.

happy 같이 짧은 응답을 기대하는 테스트에서는 timeout을 안 줘도 dev 서버가 적시에 응답하면 큰 문제는 없다. 그래도 명시적으로 넣는 편이 안전하다. 일반적인 가이드는 *연결 timeout과 읽기 timeout을 분리*하는 것 — `timeout=(3.05, 30)` 처럼 튜플로 주면 첫 값은 connect, 두 번째는 read timeout이다. dispatch는 짧은 응답이 본질이라 단일 값(예: 15초)으로 충분하다.



## connection pool과 thread-safety

`requests.Session` 의 connection pool은 기본 10개 connection을 호스트당 유지한다. 한 테스트 안에서 50개 동시 요청을 보내면(concurrency 그룹) connection pool이 부족해 일부는 새 connection을 만들거나 대기한다.

```python
adapter = HTTPAdapter(pool_connections=50, pool_maxsize=50)
s.mount("http://", adapter)
s.mount("https://", adapter)
```

이 코드를 fixture에 추가하면 호스트당 connection을 50개까지 키운다. `qa/python` 의 현재 코드는 기본값을 그대로 쓰지만, burst 50 테스트에서 connection 대기로 elapsed가 길어진다면 이 옵션을 늘릴 수 있다.

Session은 thread-safe로 알려져 있지만, **공식 문서는 명시적으로 보장하지 않는다**. 실제로는 connection pool과 cookie jar가 thread-safe라 대부분의 사용에서 안전하지만, 같은 Session에서 cookie를 읽고 쓰는 경합이 있다면 확실하지 않다. `qa/python` 의 ThreadPoolExecutor 사용은 cookie 변경 없이 POST만 보내므로 사실상 안전한 영역이다.



## 함정 모음

**redirect 자동 처리**. requests는 기본적으로 3xx redirect를 자동 처리한다. 의도하지 않은 GET으로 변환되거나, body가 사라지는 일이 종종 있다. 명시적으로 끄려면 `allow_redirects=False`. POST 후 redirect 받는 일이 의심되면 `r.history` 를 확인.

**SSL 검증 끄기 유혹**. dev 환경 인증서가 깨진 상태에서 `verify=False` 로 끄고 가는 일이 잦다. 한 번 끄면 prod 코드로 옮겨갈 때 함께 따라간다. dev는 사내 CA 인증서를 추가하거나 `REQUESTS_CA_BUNDLE` 환경변수로 풀어 두는 편이 안전하다.

**multipart 업로드의 파일 핸들 누수**. `files={"file": open("path", "rb")}` 형태로 보내면 호출 후에도 파일 핸들이 열려 있다. `with open(...)` 컨텍스트 매니저로 감싸 닫아야 한다.

**`raise_for_status()` 와 단언의 충돌**. `r.raise_for_status()` 는 4xx·5xx에서 예외를 던진다. 의도한 4xx까지 예외로 만들면 lifecycle·chaos 그룹의 단언 로직과 충돌한다. `qa/python` 은 명시적으로 `r.status_code in (...)` 으로 단언하므로 raise_for_status를 쓰지 않는다.

**stream 응답의 본문 미소비**. 큰 응답을 `stream=True` 로 받으면 본문을 소비할 때까지 connection이 풀로 돌아가지 않는다. `r.close()` 또는 `with r:` 으로 명시 종료가 필요하다. dispatch 응답 같은 짧은 본문에는 문제가 없다.



## 비교 — requests vs httpx vs aiohttp

requests는 동기 HTTP 클라이언트의 사실상 표준이다. 같은 영역에 두 가지 대안이 있다.

**httpx**. requests와 거의 동일한 API에 비동기(async) 지원이 추가됐다. `httpx.Client` 가 `requests.Session` 에 대응하고, `httpx.AsyncClient` 가 비동기 버전이다. HTTP/2 지원과 timeout 모델이 더 정교하다. 새 프로젝트면 httpx를 고려할 만하지만, `qa/python` 처럼 ThreadPoolExecutor로 동시성을 처리하는 모델에서는 requests로도 충분하고 의존성이 더 가볍다.

**aiohttp**. 처음부터 비동기로 설계된 클라이언트. `async with aiohttp.ClientSession() as s:` 형태로 쓴다. 통합 테스트에서는 pytest 자체가 비동기를 자연스럽게 다루지 못해(별도 `pytest-asyncio` 플러그인 필요) 부담이 있다. ThreadPoolExecutor + 동기 requests 조합이 더 단순하고 디버깅이 쉽다.

`qa/python` 이 requests를 고른 이유는 두 가지다. 첫째, ThreadPoolExecutor + 동기 클라이언트 조합으로 충분히 빠르다(burst 50 in 1s). 둘째, fixture와의 결합이 자연스럽고 코드가 짧다. 비동기 모델을 고를 만큼의 동시성 요구가 없으면 requests의 단순함이 이긴다.
