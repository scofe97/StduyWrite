# conftest 계층과 fixture 설계

통합 테스트가 빠르게 늘어나면 매 테스트마다 HTTP Session, DB connection, Jenkins 자격증명을 새로 만드는 보일러플레이트가 폭주한다. pytest의 fixture는 이 반복을 의존성 주입 형태로 바꿔 주고, `conftest.py` 라는 자동 로딩 모듈은 fixture를 어디에 두느냐에 따라 자연스러운 스코프 경계를 만든다. 이 챕터는 `qa/python` 의 두 단 conftest 구조를 따라가며, fixture scope·yield 패턴·이름 충돌 규칙을 한 그릇에 정리한다.



## 왜 conftest인가

테스트 함수가 늘어날수록 같은 setup 코드가 반복된다. requests Session을 매번 생성하면 connection pool이 한 번도 재사용되지 않고, DB connection을 매번 만들면 통합 테스트가 끝날 때까지 수십 개의 connection이 만들어지고 끊어진다. 단순 함수로 빼면 setUp/tearDown 짝을 손으로 호출해야 한다.

pytest는 이 문제를 두 단계로 푼다. 첫째, fixture는 함수의 인자 이름으로 주입돼 명시적 호출이 사라진다. 둘째, `conftest.py` 라는 약속된 파일에 fixture를 두면 같은 디렉토리(와 그 하위)의 모든 테스트가 import 없이 그 fixture를 인자 이름만으로 받는다. 디렉토리 구조가 곧 fixture의 스코프가 된다.

`qa/python` 은 이 구조를 활용해 두 단 계층을 만든다. 루트 conftest에는 모든 주제가 공유하는 fixture(`http`, `jenkins`, `db`)를, 주제별 conftest에는 그 주제에서만 쓰는 fixture(`jenkins_blocked`, `redpanda_stopped`)를 둔다. 새 주제를 추가할 때 루트는 손대지 않아도 되고, 주제 conftest는 자기만의 fixture만 생각하면 된다.



## qa/python의 2단 conftest 구조

```
qa/python/
├── conftest.py                       # 루트 — 전 주제 공통
│   ├── http (Session, scope=session)
│   ├── jenkins (Session + basic auth, scope=session)
│   └── db (pymysql, function scope)
└── executor_jenkins/
    ├── conftest.py                   # 주제 — executor_jenkins 전용
    │   ├── jenkins_blocked (yield + finally)
    │   └── redpanda_stopped (yield + finally)
    └── tests/
        └── test_*.py                 # http, db, jenkins_blocked 등을 인자로 받는다
```

루트 conftest는 다음과 같다.

```python
# qa/python/conftest.py
@pytest.fixture(scope="session")
def http() -> requests.Session:
    s = requests.Session()
    s.headers.update({"userId": USER_ID, "Content-Type": "application/json"})
    return s


@pytest.fixture(scope="session")
def jenkins() -> requests.Session:
    s = requests.Session()
    user = os.getenv("JENKINS_USER")
    token = os.getenv("JENKINS_TOKEN")
    if user and token:
        s.auth = (user, token)
    return s


@pytest.fixture
def db():
    conn = pymysql.connect(
        host=DB_HOST, user=DB_USER, password=DB_PASS, database=DB_NAME,
        autocommit=True, cursorclass=pymysql.cursors.Cursor,
    )
    try:
        yield conn
    finally:
        conn.close()
```

세 가지를 짚는다. 첫째, `http` 와 `jenkins` 는 `scope="session"` 이라 테스트 세션 전체에서 한 번만 만들어진다. Session 객체가 stateless이고(실제로는 connection pool이 상태이지만 의도적으로 공유한다) 테스트 간 영향이 없는 자원이라 가능하다. 둘째, `db` 는 기본 scope(`function`)이라 매 테스트마다 새 connection이 만들어진다. connection 누수와 트랜잭션 격리를 동시에 잡기 위함이다. 셋째, `db` 는 `yield + try/finally` 패턴으로 setup/teardown을 한 함수 안에 묶었다. 테스트 함수에서 예외가 나든 정상 종료되든 connection이 닫힌다.

주제별 conftest는 chaos fixture를 둔다.

```python
# qa/python/executor_jenkins/conftest.py
@pytest.fixture
def jenkins_blocked():
    if os.getenv("SKIP_NET_CHAOS") == "1":
        pytest.skip("SKIP_NET_CHAOS=1: 네트워크 chaos 스킵")
    _run_chaos("jenkins_block.sh", "on")
    try:
        yield
    finally:
        _run_chaos("jenkins_block.sh", "off")
```

이 fixture는 `executor_jenkins/tests/` 하위에서만 쓰이고, 다른 주제(`operator_*` 같은 신규 주제)는 자기 conftest에서 별도 chaos를 정의하면 된다. 책임 경계가 디렉토리와 일치한다.



## fixture scope — 어디서 한 번, 어디서 매번

scope는 fixture 인스턴스가 얼마나 오래 살아 있는지를 결정한다.

| scope | 살아 있는 범위 | 적합한 자원 |
|-------|----------------|-------------|
| `function` (기본) | 한 테스트 함수 | DB connection, 트랜잭션, 임시 파일, 변경 가능한 상태를 가진 모든 것 |
| `class` | 한 클래스 | 거의 안 씀(클래스 단위 테스트가 아니면 의미 없음) |
| `module` | 한 모듈(파일) | 파일 단위 비싼 자원(거의 안 씀) |
| `package` | 디렉토리 단위 | 패키지 단위 인프라(드뭄) |
| `session` | 전 테스트 실행 | HTTP Session, 읽기 전용 설정, 외부 시스템 가용 여부 캐시 |

`qa/python` 의 결정은 단순하다. **mutate되지 않는 자원은 session, mutate되거나 격리가 필요한 자원은 function**. `http` 는 같은 Session 객체에 쓰이는 헤더가 한 번 설정되면 안 바뀌므로 session으로 충분하다. `db` 는 한 테스트가 트랜잭션을 열어 두거나 lock을 잡으면 다음 테스트가 영향을 받기 때문에 function이어야 한다. `test_db_lock_contention` 처럼 한 connection이 `FOR UPDATE` 락을 잡고 다른 connection이 dispatch를 시도하는 시나리오는 이 격리 가정 위에서만 동작한다.

scope를 잘못 골라 생기는 사고가 두 가지 있다. **session scope에서 mutate가 일어나는 경우** — 예를 들어 fixture가 만든 객체에 한 테스트가 헤더를 추가하면 다음 테스트로 그 헤더가 새어 나간다. **function scope로 둔 비싼 자원** — 매 테스트마다 5초씩 드는 컨테이너 기동을 반복하면 전체 시간이 폭주한다. 결정 기준은 "이 자원이 mutate되는가" 하나다.



## yield 패턴 — setup·teardown을 한 함수에 묶는다

`yield` 가 들어간 fixture는 yield 이전이 setup, yield 이후가 teardown이다. JUnit의 `@BeforeEach` + `@AfterEach` 짝을 한 함수 안에 묶은 형태다.

```python
@pytest.fixture
def db():
    conn = pymysql.connect(...)
    try:
        yield conn       # 테스트가 여기서 conn을 사용
    finally:
        conn.close()     # 테스트가 끝나면 정리, 예외 발생해도 실행
```

`try/finally` 가 핵심이다. 테스트 함수에서 `assert` 실패로 예외가 던져져도 finally가 호출되어 자원이 닫힌다. yield만 두고 finally가 없으면 예외 시 자원 누수가 발생한다.

이 패턴은 chaos fixture에서 더 빛난다.

```python
@pytest.fixture
def jenkins_blocked():
    _run_chaos("jenkins_block.sh", "on")     # /etc/hosts 에 차단 entry 추가
    try:
        yield
    finally:
        _run_chaos("jenkins_block.sh", "off")  # 테스트 끝나면 entry 제거
```

`finally` 안에서 정리 스크립트를 호출하므로, 테스트가 어떤 식으로 종료되든 시스템 상태가 원상복구된다. 다만 *프로세스가 강제 종료되면* finally는 호출되지 않는다는 점은 주의해야 한다. 그래서 chaos 스크립트는 별도로 idempotent하게 만들어 사람이 손으로 복구할 수 있게 둔다(상세는 01-06).



## fixture 의존성 — 다른 fixture를 인자로 받는다

fixture가 다른 fixture를 인자로 받으면 의존 그래프가 자동으로 풀린다. 예를 들어 `db_with_seed` 라는 fixture가 `db` fixture에 의존한다면:

```python
@pytest.fixture
def db_with_seed(db):
    with db.cursor() as cur:
        cur.execute("INSERT INTO TB_X (...) VALUES (...)")
    yield db
```

테스트 함수가 `db_with_seed` 만 받아도 pytest가 알아서 `db` 를 먼저 만든 뒤 `db_with_seed` 를 만든다. `qa/python` 의 현재 코드는 이 깊은 의존을 쓰지 않지만, 시드를 fixture로 추가할 때 자연스럽게 적용할 수 있는 패턴이다.

scope 간 의존에는 제약이 있다. **session scope는 function scope를 받지 못한다**. 더 길게 사는 fixture가 더 짧게 사는 fixture를 의존하면, 어느 인스턴스를 잡아 둘지 모순이 생긴다. 반대 방향(function이 session을 의존)은 자연스럽게 동작한다.



## conftest 자동 로딩과 이름 충돌

`conftest.py` 는 import 없이 자동 로딩된다. 같은 이름의 fixture가 여러 conftest에 있으면, **테스트 파일에서 가장 가까운 conftest의 fixture가 우선**한다. `executor_jenkins/conftest.py` 에 `db` 라는 fixture를 정의하면 그 디렉토리 하위에서는 루트의 `db` 를 가린다.

이 규칙은 두 면이 있다. 한쪽으로는 주제별로 fixture를 갈아끼우는 자유를 준다. 예를 들어 한 주제만 PostgreSQL을 쓴다면 그 주제 conftest에 `db` 를 새로 정의해 덮을 수 있다. 다른 쪽으로는 의도하지 않은 가림이 생기기 쉽다. 디렉토리 깊이가 깊어질수록 어느 conftest의 fixture가 살아 있는지 추적하기 어려워진다. `pytest --fixtures <test_path>` 로 어떤 fixture가 보이는지 출력해 확인하는 습관이 도움이 된다.

`qa/python` 의 두 단 구조는 의도적으로 단순하다. 루트와 주제 두 단만 두고, 같은 이름의 fixture를 양쪽에 두지 않는다. 새 주제가 늘어도 같은 규칙을 유지한다.



## helpers 모듈과 import side-effect

`qa/python/conftest.py` 첫 줄에는 미묘한 처리가 하나 있다.

```python
from _lib.helpers import DB_HOST, DB_USER, DB_PASS, DB_NAME, USER_ID  # noqa: F401
```

`# noqa: F401` 은 "이 import는 사용되지 않아도 경고를 띄우지 말라"는 린터 지시다. 왜 사용되지 않는 import를 conftest에 두는가? `_lib/helpers.py` 가 import되는 순간 `load_dotenv(QA_PY / ".env")` 가 실행되어 환경변수가 채워지기 때문이다. conftest가 로드될 때 환경변수도 함께 채우려는 의도다.

```python
# _lib/helpers.py
load_dotenv(QA_PY / ".env")    # import 시점에 .env 로딩

EXECUTOR_BASE = os.getenv("EXECUTOR_BASE", "...")
DB_HOST = os.getenv("DB_HOST", "...")
```

`os.getenv("DB_HOST", "...")` 평가 시점이 모듈 로드 시점이라 미리 `.env` 를 불러야 한다. fixture는 호출 시점이지만 모듈 변수는 로드 시점이라 *시점이 다르다*. 이 디테일을 모르고 `_lib/helpers.py` 를 conftest에서 import하지 않으면, 첫 fixture가 실행될 때까지 환경변수가 비어 있는 상황이 생긴다.

side-effect를 import에 의존시키는 패턴은 일반적으로는 안티패턴에 가깝지만, 통합 테스트의 환경 변수 로딩처럼 "한 번만, 가장 일찍" 일어나야 하는 작업에서는 자주 보인다. 명시적으로 `load_dotenv()` 를 conftest 첫 줄에서 부르는 편이 의도가 더 잘 드러난다는 의견도 있다. 어느 쪽이든 *왜 이 import가 있어야 하는가*를 주석으로 남겨 두면 미래의 유지보수자가 같은 함정을 피한다.



## 함정 모음

**autouse=True의 남발**. fixture에 `autouse=True` 를 붙이면 모든 테스트에 자동 주입된다. 편하지만 의도하지 않은 setup이 모든 테스트에 깔리고, 테스트 격리가 무너지기 쉽다. `qa/python` 은 autouse를 쓰지 않는다. 명시적 주입이 작은 보일러플레이트를 감수하더라도 추적성을 유지한다.

**fixture에서 Exception을 던지는 경우**. fixture setup 단계에서 예외가 나면 그 fixture를 받는 모든 테스트가 ERROR 상태가 된다. PASS도 FAIL도 아닌 ERROR. CI 출력에서 ERROR가 묻히는 일이 많기 때문에, fixture에서 외부 자원(DB·HTTP)에 접속할 때는 명확한 에러 메시지로 wrap하는 편이 좋다.

**scope=session에서 mutate**. 한 테스트가 session-scoped Session에 새 헤더를 추가하면 다음 테스트가 그 헤더로 요청을 보낸다. 격리가 무너졌다는 사실을 발견하기 어렵기 때문에, scope=session fixture는 "절대 mutate하지 않는다"는 규칙을 명시해 두는 편이 안전하다.

**conftest 위치 오류**. `conftest.py` 가 아닌 `conf_test.py` 같은 오타로 두면 자동 로딩이 안 된다. pytest는 경고도 띄우지 않고 그냥 fixture를 못 찾는다고 한다. fixture 미발견 에러가 뜨면 가장 먼저 파일명을 확인한다.

**`pytest --fixtures` 로 가시성 확보**. 어떤 fixture가 어디서 보이는지 헷갈릴 때 `pytest --fixtures executor_jenkins/tests/test_01_happy.py` 를 실행하면 그 파일에서 보이는 모든 fixture가 정의 위치와 함께 출력된다. 디버깅 첫 도구로 편리하다.



## 비교 — unittest의 setUp/tearDown과 fixture 의존성 주입

unittest의 `setUp`/`tearDown` 은 클래스 상속으로 동작한다. 여러 테스트 클래스가 같은 setup을 공유하려면 부모 클래스를 만들고 상속을 펼쳐야 한다. setup 안에서 또 다른 setup을 호출하면 의존 그래프가 사람의 머리에 쌓이고, 테스트가 늘면 추적이 어렵다.

pytest fixture는 이 부담을 의존성 주입으로 푼다. fixture가 다른 fixture를 인자로 받으면 의존이 자동 풀리고, scope 다른 fixture를 자유롭게 조합할 수 있다. setUp/tearDown으로는 자연스럽지 않은 "session scope HTTP Session + function scope DB connection 한 테스트가 동시 사용" 같은 조합이 fixture에서는 함수 시그니처 한 줄이다.

unittest에도 `addCleanup` 같은 teardown 메커니즘이 있어 try/finally 패턴을 흉내낼 수 있다. 하지만 의존성 주입이 없어 setup 코드가 인스턴스 메서드 안에 갇힌다.
