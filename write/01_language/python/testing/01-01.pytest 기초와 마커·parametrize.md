# pytest 기초와 마커·parametrize

통합 테스트의 첫 줄은 "이 파일을 어떻게 모으고, 어떤 그룹으로 갈라 돌릴 것인가" 결정이다. pytest는 작은 규칙(파일명·함수명 컨벤션) 몇 개로 디스커버리를 처리하고, 마커와 parametrize라는 두 도구로 시나리오 행렬을 코드 중복 없이 표현한다. 이 챕터는 `pytest.ini` 한 페이지를 시작으로 마커 등록의 의미와 parametrize의 표현력을 한 그릇에 정리한다.



## 왜 pytest인가

unittest로도 같은 통합 테스트를 쓸 수 있다. 굳이 pytest를 고르는 이유는 세 가지다. 첫째, 함수형 fixture가 의존성 주입 형태로 동작해 setUp/tearDown 상속을 강제하지 않는다. 둘째, parametrize가 데이터 행렬과 if문 분기를 분리해 같은 로직에 다른 입력을 넣는 케이스를 자연스럽게 표현한다. 셋째, 마커·플러그인 생태계가 두텁다. pytest-xdist로 병렬 실행을 한 줄 옵션으로 켤 수 있고, pytest-html로 self-contained 리포트를 만들 수 있다. 이 셋이 통합 테스트의 시나리오 폭주에 그대로 맞물린다.

`qa/python`은 이 세 가지를 모두 쓴다. happy 10건은 한 함수의 parametrize로, 동시성 6건은 별 도 함수 + xdist 병렬로, chaos 6건은 마커로 그룹을 묶어 `--maxfail=1` 안전망을 건다.



## 디스커버리 규칙

pytest는 명시적 등록 없이 다음 규칙으로 테스트를 모은다.

| 대상 | 기본 규칙 |
|------|-----------|
| 파일 | `test_*.py` 또는 `*_test.py` |
| 클래스 | `Test*` (없어도 됨) |
| 함수 | `test_*` |

`qa/python` 은 클래스를 쓰지 않는다. `executor_jenkins/tests/test_01_happy.py` 처럼 파일·함수만으로 시나리오를 표현한다. 클래스는 `@Nested` 흉내가 필요할 때만 도입하면 충분하고, 통합 테스트 단계에서는 함수 평면 구조가 더 읽기 좋다.

테스트 루트는 `pytest.ini` 의 `testpaths`로 정한다. `qa/python/pytest.ini` 는 `testpaths = .` 를 둬서 `qa/python` 자체를 루트로 잡는다. 그러면 `pytest executor_jenkins/` 같은 부분 경로 실행이 자연스럽게 동작한다.



## pytest.ini — 한 페이지에 다 들어가는 설정

`qa/python/pytest.ini` 는 다음과 같이 짧다.

```ini
[pytest]
markers =
    happy: 정상 dispatch
    lifecycle: 중복 dispatch / 재요청 / kind 변경
    concurrency: 동시성·부하
    chaos: 인프라·상태 chaos
testpaths = .
addopts = -v --html=report.html --self-contained-html
log_cli = true
log_cli_level = INFO
```

세 가지를 짚는다. 첫째, `markers` 블록은 *문서*다. 등록되지 않은 마커를 쓰면 `PytestUnknownMarkWarning` 이 떠서 오타를 빠르게 잡는다. `--strict-markers` 옵션을 켜면 경고가 에러로 승급되어 CI에서 무방비 마커를 막는 데 쓸 수 있다. 둘째, `addopts` 의 `--html=report.html --self-contained-html` 조합은 pytest-html이 만든 리포트를 한 파일로 봉인한다. self-contained가 빠지면 CSS·JS가 외부 파일로 흩어져 산출물을 메일이나 채팅으로 공유할 때 깨진다. 셋째, `log_cli = true` 와 `log_cli_level = INFO` 는 표준 라이브러리 logging 출력을 콘솔에 흘려 준다. 통합 테스트는 외부 프로세스·네트워크가 끼어들기 때문에 print 보다 logging이 길게 보면 더 잘 보인다.



## 마커 — 그룹을 만들고 부분 실행을 돕는다

마커는 데코레이터로 테스트에 라벨을 붙여 부분 실행을 돕는다. `qa/python` 은 시나리오를 다음 4그룹으로 나눈다.

```python
@pytest.mark.happy
@pytest.mark.parametrize(("kind", "job_path"), HAPPY_CASES)
def test_happy_dispatch_to_terminal(http, db, kind, job_path):
    ...
```

이렇게 라벨을 달면 그룹별 부분 실행이 한 줄 옵션으로 끝난다.

```bash
pytest executor_jenkins/ -m happy            # 정상 흐름만
pytest executor_jenkins/ -m concurrency -n auto    # 동시성만, xdist 병렬
pytest executor_jenkins/ -m chaos --maxfail=1      # chaos는 첫 실패에서 즉시 중단
```

`--maxfail=1` 이 chaos 그룹에 특히 잘 맞는 이유는, chaos 케이스가 시스템 상태를 흔드는 fixture를 동반하기 때문이다. 한 케이스가 실패해 fixture의 finally 블록이 호출되지 않은 채 다음 케이스로 넘어가면 `/etc/hosts` 에 차단 entry가 남거나 redpanda 컨테이너가 멈춘 채로 유지된다. 첫 실패에서 멈추면 사람이 개입해 상태를 정리하는 편이 더 안전하다.

마커는 그룹 라벨 외에도 `xfail`, `skip`, `skipif` 같은 표준 동작이 있다. `qa/python` 은 fixture 안에서 `pytest.skip("SKIP_NET_CHAOS=1")` 을 호출해 환경변수에 따른 동적 skip을 쓴다(상세는 01-06).



## parametrize — 데이터 행렬을 코드 분기로 만들지 않는다

happy 그룹은 같은 검증 로직에 BUILD 4건·TEST 4건·DEPLOY 2건 합 10건을 돌려야 한다. if 분기로 풀면 가독성이 무너지고 케이스 추가가 비싸진다. parametrize는 입력 행렬과 검증 로직을 분리해 이 비용을 일정하게 만든다.

```python
HAPPY_CASES = [
    ("BUILD",  "job/QA/job/BUILD-01"),
    ("BUILD",  "job/QA/job/BUILD-02"),
    # ... TEST·DEPLOY 합쳐 10건
]


@pytest.mark.happy
@pytest.mark.parametrize(("kind", "job_path"), HAPPY_CASES)
def test_happy_dispatch_to_terminal(http, db, kind, job_path):
    body = dispatch_payload(kind=kind, job_path=job_path)
    r = http.post(f"{EXECUTOR_BASE}/jobs/dispatch", json=body)
    assert r.status_code == 200, f"dispatch failed: {r.status_code} {r.text}"

    final = wait_status(db, body["jobExcnId"], lambda s: s in TERMINAL_OK, timeout=300)
    assert final in TERMINAL_OK, f"unexpected terminal: {final}"
```

이 한 함수가 pytest의 출력에서는 10개의 분리된 케이스로 표시되고, 어떤 케이스가 실패했는지가 파라미터 표기로 즉시 드러난다. 예를 들어 `test_happy_dispatch_to_terminal[BUILD-job/QA/job/BUILD-03]` 같은 ID로 식별된다.

chaos의 `bad_vsrn` 케이스도 같은 패턴이다.

```python
@pytest.mark.chaos
@pytest.mark.parametrize("bad_vsrn", [0, -1, 9999])
def test_invalid_job_vsrn(http, bad_vsrn):
    body = dispatch_payload(job_path="job/QA/job/CHAOS-VSRN-01", job_vsrn=bad_vsrn)
    r = http.post(f"{EXECUTOR_BASE}/jobs/dispatch", json=body)
    assert r.status_code in (400, 422, 404)
```

세 값을 데이터로 빼 두면, 새 입력(예: `Integer.MIN_VALUE`)을 추가해도 한 줄이면 끝난다.

parametrize의 강점은 **fixture와도 결합한다**는 점이다. `indirect=True` 옵션을 쓰면 파라미터 값이 fixture로 흘러가고, fixture 안에서 값에 따라 다른 환경을 준비한다. 이 묶음의 코드 베이스는 indirect 파라미터화를 쓰지 않지만, 알고 있으면 "fixture와 데이터 둘을 동시에 갈아끼우는" 시나리오에 유용하다.



## 함정 모음

**마커 미등록 경고**. `markers` 블록에 등록하지 않은 마커를 쓰면 `PytestUnknownMarkWarning` 이 매 실행마다 출력된다. 한두 번은 무시해도 시간이 지나면 진짜 문제 경고를 묻어 버린다. 새 그룹을 만들 때마다 `pytest.ini` 에 줄을 추가하는 습관이 필요하다. CI에서는 `--strict-markers` 로 경고를 에러로 승급시키면 검출이 빨라진다.

**`addopts` 와 명시 옵션의 관계**. `addopts = -v --html=report.html` 가 있는 상태에서 `pytest -p no:html` 같은 옵션을 추가하면 자동 옵션과 충돌해 의도와 다른 동작이 나온다. `addopts` 는 늘 명시 옵션 *앞*에 붙는다는 점을 기억해 두면 디버깅이 빠르다.

**self-contained 누락**. `--html=report.html` 만 두고 `--self-contained-html` 을 빠뜨리면 리포트 폴더에 CSS·JS가 함께 생성된다. 한 파일로 공유할 때만 깨지고, 로컬에서는 정상으로 보이기 때문에 첫 공유 때서야 발견된다.

**parametrize ID의 가독성**. 파라미터 값에 슬래시·공백이 들어가면 pytest가 만드는 자동 ID가 길어지고 깨진다. `pytest.param("값", id="짧은-라벨")` 형태로 명시 ID를 주면 출력이 단정해진다. `qa/python` 의 happy 케이스는 `job_path` 가 짧아 자동 ID로 충분하지만, 파라미터 수가 늘면 명시 ID가 가독성을 회복시킨다.

**discovery에서 빠지는 파일**. 파일명을 `tests_dispatch.py` 같이 두면 디스커버리에 잡히지 않는다. 규칙은 `test_*.py` 와 `*_test.py` 두 가지뿐이다. 사내 컨벤션에 맞춰 다른 이름을 쓰고 싶으면 `python_files = check_*.py test_*.py` 처럼 ini에서 패턴을 확장한다.



## 비교 — pytest와 unittest

같은 통합 테스트를 unittest로 쓰면 어떻게 달라지는지 짧게 비교한다.

| 항목 | pytest | unittest |
|------|--------|----------|
| 함수 형태 | 평면 함수 + fixture 인자 | `TestCase` 상속 + `setUp/tearDown` |
| 데이터 행렬 | `@pytest.mark.parametrize` 한 줄 | `subTest` 또는 ddt 같은 외부 라이브러리 |
| 그룹화 | `@pytest.mark.<name>` + `pytest -m` | `loadTestsFromName` 으로 수작업 |
| 병렬 실행 | pytest-xdist 한 옵션 | 수작업 또는 외부 러너 |
| 단언 메시지 | 단순 `assert` 가 introspection으로 풍부한 출력 | `assertEqual` 등 메서드 호출 |

unittest는 표준 라이브러리에 들어 있어 별도 설치 없이 쓸 수 있다는 장점이 있고, 복잡한 mock 라이프사이클을 명시적으로 다루기 쉽다는 면도 있다. 하지만 통합 테스트의 시나리오가 늘어나면 parametrize·마커·xdist 가 주는 이득이 커지기 때문에 `qa/python` 은 pytest를 골랐다.
