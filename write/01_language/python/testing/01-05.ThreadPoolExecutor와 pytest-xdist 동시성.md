# ThreadPoolExecutor와 pytest-xdist 동시성

통합 테스트의 동시성에는 두 종류가 있다. 첫째, *한 테스트 안에서* 50개 dispatch를 동시에 쏘아 race condition을 끌어내는 동시성. 둘째, *테스트들끼리* 병렬로 돌려 전체 실행 시간을 줄이는 동시성. `qa/python` 은 둘을 명확히 구분해 쓴다 — 한 테스트 안에서는 `concurrent.futures.ThreadPoolExecutor`, 테스트들끼리는 `pytest-xdist`. 이 챕터는 두 모델의 결정과 GIL·thread-safety·xdist 격리를 한 그릇에 정리한다.



## 왜 동시성을 직접 만드는가

mock 기반 테스트로는 race condition을 잡기 어렵다. mock은 호출 순서가 결정적이라 동시 호출이 만들어 내는 경합이 시뮬레이션되지 않는다. burst 50건이 1초 안에 도착할 때 같은 jobPath로 들어온 dispatch가 QUEUE_ID를 어떻게 매칭하는지, 같은 jobExcnId가 거의 동시에 들어왔을 때 idempotent 보장이 깨지는 지점은 진짜 동시성이 있어야 드러난다.

`qa/python` 의 concurrency 그룹 6건은 의도적으로 서버를 흔든다. 같은 tlId 10/30 병렬, 두 tlId 각 15 병렬, 같은 jobPath 20 병렬, burst 50 in 1s, 혼합 30 병렬. 각 케이스가 다른 종류의 race를 노린다. 이걸 한 테스트 안에서 만들려면 thread가 답이다.



## ThreadPoolExecutor — 한 테스트 안의 동시성

`concurrent.futures.ThreadPoolExecutor` 는 thread pool 추상화다. fixed 크기의 thread를 만들고, 작업을 큐에 넣으면 thread가 받아서 실행한다.

`qa/python` 의 concurrency 패턴은 다음 헬퍼 두 개로 압축된다.

```python
def _shoot(http, kwargs):
    body = dispatch_payload(**kwargs)
    r = http.post(f"{EXECUTOR_BASE}/jobs/dispatch", json=body)
    return r.status_code, body["jobExcnId"]


def _parallel(http, payloads, max_workers):
    with cf.ThreadPoolExecutor(max_workers=max_workers) as ex:
        return list(ex.map(lambda kw: _shoot(http, kw), payloads))
```

호출 시점에 50건 burst를 만드는 코드는 한 줄이다.

```python
results = _parallel(http, payloads, max_workers=50)
```

세 가지가 결정적이다. 첫째, **`max_workers=50`** 으로 thread 50개를 만들고, payloads가 50건이라 각 thread가 한 건씩 동시에 발사한다. 더 큰 max_workers는 의미가 없고(payload 수가 상한), 더 작으면 burst가 아니라 batched 가 된다. 둘째, **`ex.map`** 은 입력 순서대로 결과를 반환한다. 결과를 `(status_code, excn_id)` 튜플로 모아 두면 단언과 후속 SELECT에 그대로 쓴다. 셋째, **`with` 블록**. 컨텍스트 매니저가 풀의 thread를 정리한다. 명시적 shutdown을 잊을 일이 없다.

burst 패턴에는 elapsed 측정이 따라온다.

```python
started = time.time()
results = _parallel(http, payloads, max_workers=50)
elapsed = time.time() - started
assert elapsed < 5.0, f"50건 처리에 {elapsed:.2f}s 소요"
```

50건이 1초 안에 떨어지지 않더라도 5초 안에는 끝나야 한다는 단언이다. burst 의 의도는 "서버가 받아 줄 수 있는가"라 짧은 시간 안에 끝나는지가 부수 단언이 된다.



## GIL과 I/O bound — thread가 빠른 이유

Python의 Global Interpreter Lock(GIL)은 한 시점에 하나의 thread만 Python 바이트코드를 실행하게 만든다. CPU bound 작업(예: 큰 행렬 계산)에서 thread는 거의 효과가 없다 — GIL을 두고 다투기만 한다.

I/O bound 작업은 다르다. socket·파일·subprocess 호출 같은 시스템 콜이 *GIL을 일시 해제*한다. requests의 `http.post(...)` 도 내부적으로 socket I/O라 GIL을 풀고 응답을 기다린다. 한 thread가 응답 대기 중일 때 다른 thread가 GIL을 잡고 실행된다. 결과적으로 thread 50개가 socket 50개에 동시에 응답을 기다리는 구조가 된다 — 50배 가까운 동시 처리.

`qa/python` 의 동시 dispatch는 거의 100% I/O bound라 thread가 자연스럽다. 만약 응답 처리에 무거운 JSON 파싱이나 검증 로직이 들어 있으면 일부 GIL 경합이 보이겠지만, status code만 읽는 가벼운 흐름이라 50 thread가 50 동시 발사로 그대로 떨어진다.

CPU bound면 `multiprocessing.Pool` 이나 `ProcessPoolExecutor` 가 답이다. process는 GIL이 분리돼 있다. 다만 process 간 자원 공유(예: requests Session)가 어려워 통합 테스트의 동시 부하 패턴에는 잘 맞지 않는다.



## thread-safety — Session 공유의 안전 영역

`http` fixture는 session scope라 모든 thread가 같은 Session을 공유한다. `requests.Session` 의 thread-safety는 *공식적으로 보장되지 않지만 사실상 안전*이다. 안전한 사용 영역과 그렇지 않은 영역을 구분해 두면 사고를 막는다.

| 동작 | thread-safety |
|------|----------------|
| GET/POST 같은 요청 메서드 호출 | 안전 (connection pool이 thread-safe하게 동작) |
| 같은 Session에서 cookie 읽기 | 안전 |
| 같은 Session의 cookie를 *변경* | 의심스러움 (race 가능) |
| Session.headers 변경 | 위험 (mutate가 다른 thread로 새 나간다) |
| Session.auth 변경 | 위험 (위와 같은 이유) |

`qa/python` 은 Session 생성 후 헤더와 auth를 *한 번만* 설정하고 이후 mutate하지 않는다. 모든 thread는 POST만 호출한다. 이 영역은 안전하다.

만약 dispatch마다 헤더를 다르게 보내야 한다면 `s.post(url, headers={...})` 처럼 호출 인자로 넘긴다. 이 헤더는 그 호출에만 적용되고 Session에 저장되지 않는다.



## QUEUE_ID 충돌 — race를 검증으로 만드는 패턴

같은 jobPath에 다른 jobExcnId 20건을 동시에 보내는 테스트가 가장 흥미로운 race를 노린다.

```python
@pytest.mark.concurrency
def test_same_jobpath_20_parallel_no_queue_collision(http, db):
    payloads = [
        dict(tl_id="TL_QA_DEV", job_path="job/QA/job/CONC-RACE")
        for _ in range(20)
    ]
    results = _parallel(http, payloads, max_workers=20)
    assert all(code == 200 for code, _ in results)
    excn_ids = [eid for _, eid in results]

    placeholders = ",".join(["%s"] * len(excn_ids))
    with db.cursor() as cur:
        cur.execute(
            f"SELECT QUEUE_ID FROM TB_TRB_EC_001 "
            f"WHERE JOB_EXCN_ID IN ({placeholders}) AND QUEUE_ID IS NOT NULL",
            excn_ids,
        )
        queue_ids = [r[0] for r in cur.fetchall()]
    assert len(queue_ids) == len(set(queue_ids)), f"QUEUE_ID 충돌: {queue_ids}"
```

검증의 본질은 마지막 단언이다. `len(queue_ids) == len(set(queue_ids))` 는 list와 set의 길이가 같다는 단언으로, **중복이 없다**를 의미한다. Jenkins QUEUE_ID는 한 job 트리거마다 유일한 값이라야 한다. 같은 jobPath로 20건이 거의 동시에 들어와도 executor가 각각을 분리된 트리거로 만들어야 한다 — 만약 race로 두 dispatch가 같은 QUEUE_ID에 매칭되면 마지막 단언이 무너진다.

이런 race는 mock으로는 절대 만들 수 없다. 진짜 thread + 진짜 socket + 진짜 Jenkins API 응답이 합쳐져야 race window가 열린다.



## pytest-xdist — 테스트들끼리의 병렬

xdist는 다른 단의 동시성이다. 한 테스트 안의 thread가 아니라, 테스트 *함수들*을 process 단위로 분할 실행한다.

```bash
pytest executor_jenkins/ -m concurrency -n auto
```

`-n auto` 는 CPU 코어 수만큼 worker process를 띄우고, 테스트들을 워커에게 분산한다. 6개 concurrency 테스트를 한 머신에서 직렬 실행하면 burst 50 같은 무거운 케이스에서 실행 시간이 길어진다. 4 worker로 분산하면 평균 실행 시간이 줄어든다.

xdist는 *process*를 만든다. thread가 아니다. 각 worker는 자기만의 Python 인터프리터·메모리 공간을 가지고, conftest와 fixture가 worker마다 다시 로드된다. session-scoped fixture도 worker마다 별도 인스턴스가 만들어진다. **session은 "전 테스트"가 아니라 "한 worker 안의 전 테스트"가 되는 셈**이다.

이 격리가 두 면을 만든다. 한쪽으로는 worker 간 자원 충돌이 없어 안전하다. 다른 쪽으로는 session-scoped 비싼 자원이 worker 수만큼 만들어져 비용이 늘 수 있다. `qa/python` 의 Session·DB connection은 worker당 하나라 비싸지 않지만, Testcontainers 같은 자원은 worker마다 컨테이너를 띄우면 부담이 폭주한다(이 묶음의 코드 베이스에는 해당 없음).



## xdist + DB — 테스트 격리의 함정

xdist의 가장 자주 만나는 함정은 **테스트가 전역 상태에 의존**할 때다. 두 worker가 같은 테이블에 같은 ID로 INSERT하면 PRIMARY KEY 충돌이 난다.

`qa/python` 은 이 함정을 두 가지로 피한다. 첫째, jobExcnId 가 UUID 기반이다.

```python
def make_excn_id(prefix: str = "EX") -> str:
    return f"{prefix}-{uuid.uuid4().hex[:12].upper()}"
```

UUID는 본질적으로 충돌이 없으므로 worker가 몇 개든 ID 충돌이 안 일어난다. 둘째, jobPath는 worker 간 공유되지만 PRIMARY KEY가 아니라 충돌 자체가 의미를 갖지 않는다(같은 jobPath로 여러 dispatch가 들어오는 게 정상 시나리오).

만약 테스트가 PRIMARY KEY를 hard-code 한다면 worker 간 충돌이 발생할 수 있다. 그런 경우 worker ID를 ID에 섞어 격리한다.

```python
import os
worker_id = os.getenv("PYTEST_XDIST_WORKER", "main")
unique_id = f"FIX-{worker_id}-{int(time.time())}"
```

`PYTEST_XDIST_WORKER` 는 xdist가 worker마다 자동 설정하는 환경변수다(`gw0`, `gw1` 형태).



## chaos fixture는 xdist와 충돌한다

chaos fixture는 시스템 글로벌 상태를 변경한다. `/etc/hosts` 에 차단 entry를 추가하거나 docker compose 컨테이너를 stop한다. 두 worker가 동시에 chaos fixture를 실행하면 한쪽이 entry를 제거할 때 다른 쪽이 차단 상태에 있어야 하는 시나리오와 충돌한다.

`qa/python` 의 결정은 단순하다 — chaos 그룹은 xdist 없이 직렬로 돌린다.

```bash
pytest executor_jenkins/ -m chaos --maxfail=1
# -n auto 안 붙임
```

대신 happy·concurrency 그룹은 xdist로 병렬화해도 된다. 그룹별로 동시성 정책을 다르게 가져가는 패턴이 깔끔하다.



## 함정 모음

**ThreadPoolExecutor의 예외 묻힘**. `ex.map(...)` 으로 받은 결과를 list로 변환할 때 한 작업의 예외는 그 결과를 꺼낼 때 던져진다. list 변환에서 일찍 멈추는 일이 잦다. `qa/python` 은 `_shoot` 안에서 예외를 던지지 않고 status code만 반환하므로 영향이 없지만, 작업 안에 raise가 있으면 결과 수집 단계에서 예외가 솟는다.

**fixture가 thread-safe하지 않음**. fixture 함수 자체는 thread-safe가 보장되지 않는다. fixture 안에서 thread를 만들면 fixture 종료 시점이 thread 종료보다 빠를 수 있다. `qa/python` 은 fixture가 단순 자원 생성/해제라 thread를 만들지 않는다.

**xdist에서 fixture 분리 발생**. `pytest-xdist` 는 worker process에서 동작하므로, session-scoped fixture가 worker별로 별도다. 한 worker가 만든 자원을 다른 worker가 보지 못한다. 외부 자원(DB·HTTP) 은 영향 없지만, 메모리 캐시·임시 파일은 분리된다.

**asyncio와의 혼합 금지**. ThreadPoolExecutor와 asyncio는 모델이 다르다. asyncio 코드를 thread 안에서 동기로 호출하려면 `asyncio.run()` 으로 래핑해야 한다. `qa/python` 은 asyncio를 쓰지 않아 충돌이 없다.

**timeout 없는 thread 작업**. `ex.submit(work).result()` 에 timeout을 안 주면 work가 무한 대기 시 thread를 점유한다. requests 호출은 클라이언트 timeout으로 보호되므로 자연스럽게 끝난다. 다른 종류의 I/O는 명시 timeout이 필요하다.

**단언 메시지의 정보 누락**. `assert all(code == 200 for code, _ in results)` 만 두면 어떤 case가 실패했는지 안 보인다. `qa/python` 은 fail 메시지에 실패한 결과만 필터해 출력한다.

```python
assert all(code == 200 for code, _ in results), [r for r in results if r[0] != 200]
```



## 비교 — ThreadPoolExecutor vs asyncio + aiohttp

같은 50 동시 발사를 asyncio + aiohttp로 쓰면 다음 모양이다.

```python
async def shoot(session, payload):
    async with session.post(URL, json=payload) as r:
        return r.status

async def burst():
    async with aiohttp.ClientSession() as s:
        return await asyncio.gather(*(shoot(s, p) for p in payloads))

results = asyncio.run(burst())
```

이론적으로 더 효율적이다 — 단일 thread에서 event loop가 50개 socket을 한꺼번에 감독한다. context switch 비용이 thread보다 낮고 thread-safety 걱정이 없다.

`qa/python` 이 thread를 고른 이유는 두 가지다. 첫째, **동기 requests를 그대로 쓰기 위해**. asyncio는 코루틴 함수를 모든 단계에 전파해야 한다. fixture·헬퍼·테스트 모두 async/await로 바꾸면 변경 폭이 크다. 둘째, **pytest와의 결합**. async 테스트를 쓰려면 `pytest-asyncio` 플러그인이 필요하고, fixture 모델이 약간 다르다. 50 동시성에서 thread의 비용이 견딜 만하면 단순함이 이긴다.

진짜 수천~수만 동시 부하가 필요하면 asyncio가 답이다. 그 단계에서는 별도 부하 도구(예: locust, k6)도 검토 대상이다. `qa/python` 의 50 burst는 thread로 충분히 표현된다.
