# 부하 분포 패턴과 dispatch 순서 검증

## 왜 이 단계가 필요한가

01-05 챕터의 동시성 테스트가 *결함 검출* 도구라면, 이 챕터의 부하 분포 테스트는 *동작 검증* 도구다. 50건을 한 번에 쏘는 일은 같지만 묻는 질문이 다르다. 01-05 는 "QUEUE_ID 가 충돌하지 않는가, REJECTED 가 나오지 않는가" 처럼 **race condition 의 존재 여부**를 본다. 이 챕터는 "두 Jenkins 에 잡이 50:50 으로 갈라지는가, 같은 tlId 안에서 priorityDt 순서가 보존되는가, quota gate 가 한쪽 Jenkins 에만 적용되는가" 처럼 **분배 규칙이 의도대로 동작하는가**를 본다.

`qa/python/suites/load_orchestration/` 은 이 질문을 위해 만들어진 별도 suite 다. executor 의 dispatch_gate 정책이 두 dev Jenkins (`console` + `trombone-v2`) 에 잡을 갈라 보내는 라우팅·quota 정책이 회귀하지 않도록, 50~100건 trigger 순서가 보존되는지를 한 HTML 타임라인으로 본다. Locust 나 k6 같은 전문 부하 도구가 아닌 pytest + ThreadPoolExecutor 자체 구현을 고른 이유는 분명하다 — 측정값이 RPS 가 아니라 *분배·순서* 이고, 같은 conftest 의 `http` / `db` / `kafka_topic` fixture 를 그대로 재써야 한다. 외부 도구로 가져가면 시드·인증·환경 변수 코드가 즉시 두 벌이 된다.



## 검증 대상 4 가지

`suites/load_orchestration/README.md` 가 짚는 검증 목록은 네 줄로 요약된다.

1. **라우팅** — tlId 별로 잡이 각 Jenkins 에 분리되는가. `TL_E2E_CONSOLE` 은 console Jenkins 로, `TL_E2E_V2` 는 trombone-v2 로.
2. **per-Jenkins quota** — dispatch_gate 의 동시 실행 제한이 두 Jenkins *합산* 이 아닌 *각 Jenkins 독립* 으로 적용되는가. 합산으로 잘못 박혀 있으면 한쪽이 노는데 다른 쪽이 막힌다.
3. **순서 보존** — 같은 tlId 안에서 priorityDt 또는 큐 진입 순서가 그대로 큐잉되는가.
4. **5xx · REJECTED 0건** — 50~100건 폭주에서도 종단 도달 전에 깨지지 않는가.

이 4가지는 모두 *동시 발사* + *각각 어디로 갔는지 추적* 이라는 한 가지 측정 방식으로 검증한다. 그래서 트리거 패턴이 3가지로 갈린다 — balanced / skewed / ramp.



## 3가지 부하 분포 패턴

`_lib/load_patterns.py` 에 정의된 세 패턴은 같은 size 의 잡 리스트를 만들지만 *분배 비율* 과 *trigger 순서* 가 다르다.

### balanced — 50:50 인터리브

```python
def balanced(size, tl_a, pool_a, tl_b, pool_b):
    half = size // 2
    a = _zip(tl_a, _take(pool_a, half))
    b = _zip(tl_b, _take(pool_b, size - half))
    out = []
    for x, y in itertools.zip_longest(a, b):
        if x: out.append(x)
        if y: out.append(y)
    return out[:size]
```

A·B 두 Jenkins 로 절반씩, **번갈아 가며** trigger 순서를 만든다. 잡 #0 은 A, #1 은 B, #2 는 A, #3 은 B... 이 인터리브가 깨지지 않고 큐 진입 순서로 보존되면 dispatch_gate 가 라운드로빈에 가까운 fair scheduling 을 하는 것이고, 한쪽에 묶이면 라우팅 자체가 깨진 신호다.

### skewed — 70:30 + 셔플

```python
def skewed(size, tl_a, pool_a, tl_b, pool_b):
    a_n = int(size * 0.7)
    a = _zip(tl_a, _take(pool_a, a_n))
    b = _zip(tl_b, _take(pool_b, size - a_n))
    out = a + b
    random.Random(42).shuffle(out)
    return out
```

A:B = 7:3 비대칭. 그리고 *결정적 seed 42* 로 섞는다. 한쪽 Jenkins 가 70% 부하를 떠안을 때 그쪽 quota gate 가 병목이 되는지(다른 쪽은 30% 만 받으니 빨리 비어야 한다), per-Jenkins 격리가 깨졌으면 30% 쪽도 같이 느려진다.

> Seed 42 가 박혀 있어 **매 실행 trigger 순서가 동일**하다. 회귀 테스트는 결정적 입력이 1순위다. random shuffle 을 쓰면서 비결정성을 들이면 같은 코드가 어떤 날은 통과하고 어떤 날은 실패하는 flaky test 가 생긴다.

### ramp — 단계적 burst (10 → 30 → 60 → 100)

```python
def ramp(size, tl_a, pool_a, tl_b, pool_b):
    stages = [0.1, 0.3, 0.6, 1.0]
    out = []
    prev = 0
    for frac in stages:
        target = int(size * frac)
        chunk = target - prev
        half = chunk // 2
        a = _zip(tl_a, _take(pool_a, half))
        b = _zip(tl_b, _take(pool_b, chunk - half))
        out.extend(a + b)
        prev = target
    return out[:size]
```

10% → 30% → 60% → 100% 단계로 증가하는 burst. 호출 측은 *stage 사이에 짧은 sleep* 을 끼워 dispatch_gate 의 quota 가 차오르는 잡을 어떻게 흡수하는지 본다. 10건은 거의 안 막히고, 60건쯤에서 quota 가 차서 일부 대기, 100건 burst 가 들어오는 시점에는 ENQUEUED 가 정상 누적되는가가 관전 포인트.



## ThreadPoolExecutor + 폴러 thread — 두 단 동시성

`load_orchestration` 의 핵심 트릭은 **dispatch 호출은 ThreadPoolExecutor 가, 폴링은 별도 단일 thread 가** 라는 분리다. 50건을 동시에 쏘면서 그 50건의 상태를 또 동시에 폴링하면 DB 커넥션이 폭주한다. 그래서 `_lib/jenkins_poller.py` 의 `start_poller_thread()` 가 한 thread 만 띄우고 N건을 한 쿼리(`WHERE JOB_EXCN_ID IN (...)`)로 일괄 폴링한다.

```python
def start_poller_thread(excn_ids, emitter, *, timeout, interval=1.5):
    result_box = {}
    def _run():
        result_box.update(poll_until_terminal(
            excn_ids, emitter, timeout=timeout, interval=interval,
        ))
    t = threading.Thread(target=_run, name="load-orch-poller", daemon=True)
    t.start()
    return t, result_box
```

폴러는 1.5초 간격으로 IN 절 쿼리를 1회 실행하고, 상태 변화가 감지된 잡만 emitter 로 *state* 이벤트를 흘려보낸다. 한 잡당 보통 SUBMITTED → RUNNING → terminal 의 3번 변화이므로, 50건 × 3 = 150 이벤트가 한 세션에서 적재된다 (그 외 trigger 50건 + terminal 50건 추가).

```python
# poll_until_terminal — 변화 감지 시만 record
changed_keys = [k for k in ("status", "queue_id", "build_no", "bgng_dt", "end_dt")
                if prev.get(k) != row.get(k)]
if changed_keys:
    emitter.record("state", job_excn_id=excn_id, **row, changed=changed_keys)
    if status in terminal:
        emitter.record("terminal", job_excn_id=excn_id, **row)
```

변화 *없는* 폴링은 이벤트를 만들지 않는다. 그래서 ndjson 파일 크기가 잡 수 × 변화 수 에 선형이고, 1.5초 간격 폴링이라도 같은 상태를 반복 적재하지 않는다.



## job_inventory — 두 Jenkins 의 잡 풀 자동 수집

50건을 어디로 쏠지는 사람이 적지 않는다. `_lib/job_inventory.py` 가 양쪽 Jenkins 의 `SBH/SBH-LOAD-{01..20}` placeholder 잡을 자동 수집한다.

```python
def collect_leaf_jobs(target: JenkinsTarget) -> list[str]:
    s = _session(target.user, target.token)
    tree = "jobs[name,_class,jobs[name,_class,jobs[name,_class,jobs[name,_class]]]]"
    r = s.get(f"{target.base}/api/json", params={"tree": tree, "depth": "3"}, timeout=20)
    data = r.json()
    out = []
    for top in data.get("jobs", []):
        if top.get("name") == "SBH":
            for child in top.get("jobs", []):
                _walk("job/SBH", child, out)
    return out
```

Jenkins API 의 `tree` 파라미터로 *재귀 깊이 3* 까지의 잡 트리를 단일 요청으로 가져온다. `WorkflowJob` 클래스만 leaf 로 인식하고, Folder/MultiBranchProject 는 컨테이너로 통과시킨다. 운영 PJ_\* 폴더 잡은 *기본 OFF* — `LOAD_ORCH_INCLUDE_PJ=1` 일 때만 후보에 추가된다. 부하 테스트가 운영 파이프라인을 건드릴 위험을 환경변수로 명시 opt-in.



## 한 테스트가 만드는 50개의 잡 — 호출 흐름

전형적 load_orchestration 테스트 한 건의 흐름:

```python
# tests/test_01_order.py 가상의 형태
@pytest.mark.order_balanced
def test_balanced_50(http, db, metric_emitter):
    inv_a = filter_for_load(collect_leaf_jobs(TARGET_A))
    inv_b = filter_for_load(collect_leaf_jobs(TARGET_B))
    plan = balanced(LOAD_ORCH_SIZE, "TL_E2E_CONSOLE", inv_a, "TL_E2E_V2", inv_b)

    excn_ids = []
    payloads = []
    for tl_id, job_path, kind in plan:
        body = dispatch_payload(tl_id=tl_id, job_path=job_path, kind=kind)
        excn_ids.append(body["jobExcnId"])
        payloads.append(body)

    # 1) 폴러 먼저 띄우기 (dispatch 시작 전에 폴링 ready)
    t, result_box = start_poller_thread(excn_ids, metric_emitter, timeout=600)

    # 2) ThreadPoolExecutor 50 동시 dispatch
    with cf.ThreadPoolExecutor(max_workers=len(plan)) as ex:
        for body in payloads:
            ex.submit(_shoot_with_record, http, body, metric_emitter, "balanced")

    # 3) 폴러 종료 대기
    t.join(timeout=620)

    # 4) 검증
    statuses = {eid: result_box[eid]["status"] for eid in excn_ids}
    rejected = [e for e, s in statuses.items() if s == "REJECTED"]
    assert not rejected, f"REJECTED 발생: {rejected[:5]}"
    # tlId 별 잡 수 분배 검증
    by_tl = _count_by_tl(plan)
    assert by_tl["TL_E2E_CONSOLE"] == 25 and by_tl["TL_E2E_V2"] == 25
```

순서가 중요하다 — 폴러를 *먼저* 띄워야 SUBMITTED 가 너무 빨리 RUNNING 으로 전환되어 SUBMITTED 이벤트를 놓치는 일이 없다. dispatch 호출은 폴러 thread 가 완전히 ready 된 다음.



## emitter — fanout 으로 ndjson + 후속 Prometheus

상태 변화를 한 곳에만 기록하지 않는다. `conftest.py` 의 `metric_emitter` fixture 는 세 emitter 를 묶어 fanout 한다.

```python
@pytest.fixture(scope="session")
def metric_emitter():
    path = _session_ndjson()
    ndjson = NdjsonEmitter(path)
    prom = PrometheusEmitter(os.environ.get("PROM_PUSHGW_URL"))
    emitter = FanoutEmitter(ndjson, prom)
    emitter.ndjson_path = path
    try: yield emitter
    finally: emitter.close()
```

`NdjsonEmitter` 는 항상 동작. `PrometheusEmitter` 는 `PROM_PUSHGW_URL` 환경변수가 설정됐을 때만 활성 — 그 외엔 no-op. fanout 패턴 덕에 *지금은 로컬 HTML 만, 나중에 dev Grafana 추가* 같은 점진 도입이 한 줄 환경변수로 끝난다. 자세한 설계는 01-10 챕터에서 다룬다.

세션 종료 hook (`pytest_sessionfinish`) 이 가장 최신 ndjson 파일을 자동으로 vis-timeline HTML 로 렌더한다.

```python
def pytest_sessionfinish(session, exitstatus):
    candidates = sorted(ARTIFACTS.glob("*.ndjson"))
    if not candidates: return
    latest = candidates[-1]
    summary = render(latest, REPORT_HTML)
    print(f"\n[load_orchestration] report → {REPORT_HTML}\n"
          f"  pattern={summary.get('pattern')} total={summary.get('total')} "
          f"by_status={summary.get('by_status')}\n")
```

pytest 가 종료될 때 자동으로 `qa/python/report-orchestration.html` 이 갱신된다. 사람이 따로 명령을 칠 필요가 없다.



## 함정과 회피

### 함정 1 — 폴러를 dispatch 후에 띄우면 SUBMITTED 이벤트가 빠진다

dispatch 가 너무 빠르면 SUBMITTED → RUNNING 이 1.5초 안에 일어난다. 폴러가 그 사이에 깨어나지 않으면 SUBMITTED 상태는 ndjson 에 한 번도 적히지 않는다. 결과 timeline 에서 SUBMITTED 가 안 보이는 게 *없음* 인지 *놓침* 인지 구별이 안 된다. 회피는 위 흐름의 step 1·2 순서 그대로 — 폴러 먼저, dispatch 나중.

### 함정 2 — pytest-xdist 와 함께 쓰면 metric_emitter 가 worker 별로 만들어진다

xdist 는 worker 별 별도 프로세스라 fixture scope=session 도 worker 별로 한 번씩 호출된다. ndjson 파일이 worker 수만큼 생기고 마지막 sessionfinish hook 이 *한 worker 의* 파일만 본다. load_orchestration 의 부하 패턴 테스트는 xdist 적용 *밖* 으로 두는 게 옳다. 같은 결과를 두 worker 가 동시에 잡으려 하면 dispatch_gate 통계 자체가 의미를 잃는다.

### 함정 3 — random.shuffle 이 결정성을 깬다

`skewed` 가 `random.Random(42).shuffle()` 을 쓰는 이유. 글로벌 `random.shuffle()` 을 호출하면 다른 테스트의 random seed 와 상호작용해 비결정적이 된다. **로컬 Random 인스턴스 + 고정 seed** 가 정답.

### 함정 4 — vis-timeline 이 외부 CDN 의존

`report-orchestration.html` 은 `unpkg.com` 에서 vis-timeline CSS/JS 를 로드한다. 사내 망에서 CDN 차단되면 HTML 이 빈 페이지로 보인다. README 의 *알려진 한계* 절에 명시. self-contained 가 필요하면 vis-timeline 번들을 HTML 에 인라인으로 박는 변형 필요.



## TPS 사례 — load_orchestration suite 의 결정 요약

`qa/python/suites/load_orchestration/` 트리에서 본 챕터가 인용한 모듈들의 책임 분담:

```
load_orchestration/
├── conftest.py              # metric_emitter fixture + sessionfinish hook
├── _lib/
│   ├── load_patterns.py     # balanced / skewed / ramp 3 패턴
│   ├── job_inventory.py     # 양쪽 Jenkins 잡 트리 자동 수집 + filter_for_load
│   ├── jenkins_poller.py    # 단일 thread 로 N 건 IN 쿼리 폴링 + 변화 감지만 emit
│   ├── event_recorder.py    # NdjsonEmitter + PrometheusEmitter(stub) + Fanout (01-10 참조)
│   └── timeline_renderer.py # ndjson → vis-timeline HTML (01-10 참조)
├── scripts/
│   ├── ensure_load_jobs.py  # 양쪽 Jenkins placeholder 잡 생성/삭제
│   └── render_orchestration_report.py  # 사후 단독 렌더러
├── seeds/
│   └── 26-load-orchestration-fixtures.sql.template
└── tests/
    └── test_01_order.py     # balanced / skewed / ramp 3 케이스
```

전문 부하 도구를 안 쓰는 결정의 비용은 명확하다 — RPS 측정이 필요해지면 결국 k6 같은 도구로 옮겨야 한다. 지금 시점은 *동작 검증* 만 필요해 적절한 자체 구현.
