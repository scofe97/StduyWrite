# confluent-kafka Avro Producer로 운영 경로 검증

## 왜 이 단계가 필요한가

REST 엔드포인트는 디버깅용이고 운영 경로는 Kafka일 때, REST만으로는 닫을 수 없는 결함이 있다. 가장 큰 예가 **이벤트 기반 cancel**이다. dispatch는 REST `/jobs/dispatch`로도 받지만 cancel은 REST 엔드포인트가 아예 없고 `tps.v305p.operator.cmd.build` 토픽에 `BuildCommandAvro(eventType=CANCEL)` 메시지를 발행하는 것이 유일한 진입점이다. 이런 시스템에선 REST 통합 테스트만 쌓아도 cancel 분기는 한 번도 코드 경로를 타지 못한다. 운영 경로와 동일한 방식 — Avro 인코딩된 메시지를 Schema Registry를 거쳐 발행 — 으로 두드려야 진짜 운영 흐름이 검증된다.

이 단계는 앞 챕터들이 쌓아 둔 "외부에서 시스템을 두드리는 시각"의 마지막 조각이다. requests로 HTTP, pymysql로 DB, ThreadPoolExecutor로 동시성, bash subprocess로 인프라 단절을 쥐었다면, 이번엔 운영용 메시징 채널을 직접 잡는다.



## 핵심 API와 결정

### confluent-kafka 설치 — Python 3.14 wheel 함정

confluent-kafka는 `librdkafka` C 라이브러리에 의존하는 wrapper라 OS와 Python 버전마다 미리 컴파일된 wheel이 있어야 한다. requirements 핀이 있어도 새 Python 버전에선 그 버전의 wheel이 없을 수 있다.

```bash
# Python 3.14 + confluent-kafka==2.6.0 → wheel 빌드 실패
pip install -r requirements.txt
# ERROR: Failed building wheel for confluent-kafka

# 해결: 최신 버전 (2.14+)
pip install -U "confluent-kafka[avro]" fastavro
```

`[avro]` extras는 `avro`, `cachetools`, `httpx`, `authlib`, `joserfc` 같은 Schema Registry 클라이언트와 인증 의존성을 묶어서 끌어온다. Avro 직렬화를 쓸 거면 처음부터 같이 설치하는 게 의도가 명확하다.

### 메시지 1건 발행에 필요한 4개 컴포넌트

```python
from confluent_kafka import SerializingProducer
from confluent_kafka.serialization import StringSerializer
from confluent_kafka.schema_registry import SchemaRegistryClient
from confluent_kafka.schema_registry.avro import AvroSerializer

sr_client = SchemaRegistryClient({"url": "http://localhost:8084"})
avro_serializer = AvroSerializer(sr_client, BUILD_COMMAND_SCHEMA)

producer = SerializingProducer({
    "bootstrap.servers": "localhost:9092",
    "key.serializer": StringSerializer("utf_8"),
    "value.serializer": avro_serializer,
})

producer.produce(topic="tps.v305p.operator.cmd.build", key=job_excn_id, value=payload)
producer.flush(10)
```

`SchemaRegistryClient`가 가장 먼저 schema를 registry에 등록(POST `/subjects/<topic>-value/versions`)하고 schema id를 받는다. `AvroSerializer`는 메시지마다 그 id를 헤더에 박아 직렬화하므로 consumer 쪽도 같은 registry에서 schema를 fetch해 자동 역직렬화한다. 이 구조 덕분에 **Producer가 처음 발행하는 토픽은 자동으로 schema가 등록**된다 — 별도 등록 API를 호출할 필요가 없다.

### Schema는 어디서 가져오나 — generated source 그대로 임베드

Avro 스키마는 application 입장에서 한 군데에 정의돼야 한다. JVM 쪽에서 .avsc로 정의했다면 build 단계에서 `BuildCommandAvro.java` 같은 generated 클래스가 만들어지고 그 안에 `SCHEMA$` 상수로 `Schema.parse("...")` 호출에 쓰는 JSON이 그대로 들어 있다. Python 쪽은 그 JSON을 그대로 모듈 상수로 박는 게 가장 안전하다.

```python
# _lib/kafka.py
BUILD_COMMAND_SCHEMA = """
{
  "type": "record",
  "name": "BuildCommandAvro",
  "namespace": "org.okestro.tps.avro.operator",
  "fields": [
    {"name": "eventType", "type": {"type":"enum","name":"CommandEventTypeAvro",
        "symbols":["WAIT","QUEUE","EXCN","CMPTN","FAIL","CANCEL","SKIP"]}},
    {"name": "jobExcnId", "type": {"type":"string","avro.java.string":"String"}},
    {"name": "jobId",     "type": {"type":"string","avro.java.string":"String"}},
    {"name": "jobVsrn",   "type": "int"},
    {"name": "jobPath",   "type": ["null", {"type":"string","avro.java.string":"String"}], "default": null},
    {"name": "tlId",      "type": ["null", {"type":"string","avro.java.string":"String"}], "default": null},
    {"name": "priorityDt","type": ["null", {"type":"long","logicalType":"timestamp-millis"}], "default": null},
    {"name": "userId",    "type": {"type":"string","avro.java.string":"String"}},
    {"name": "handler",   "type": {"type":"enum","name":"BuildHandlerAvro",
        "symbols":["JENKINS"], "default":"JENKINS"}, "default":"JENKINS"}
  ]
}
"""
```

이 방식은 두 가지 이득이 있다. 첫째, Schema Registry에 등록된 schema id가 JVM 프로듀서가 발행하는 것과 같아서 같은 토픽에 양쪽이 모두 발행해도 consumer가 일관되게 처리한다. 둘째, JVM 측 `.avsc`가 변경되면 generated 클래스의 `SCHEMA$`도 바뀌므로 그 diff를 보고 Python 상수도 수동 동기화하면 된다 (자동화하려면 build 단계에 hook이 필요하지만, 변경 빈도가 낮으면 수동이 더 가볍다).

### nullable 필드 — Avro `union ["null", T]`

스키마에 `["null", "string"]` 같은 union이 보이면 그 필드는 null 허용이다. Python dict에서 `None`을 넣으면 정상 직렬화된다.

```python
payload = {
    "eventType": "CANCEL",
    "jobExcnId": "EX-...",
    "jobId": "E2E-LONG-LOG",
    "jobVsrn": 1,
    "jobPath": None,        # CANCEL 분기는 null 허용
    "tlId": None,           # CANCEL 분기는 null 허용
    "priorityDt": None,
    "userId": "tester",
    "handler": "JENKINS",
}
```

여기서 핵심은 **스키마가 nullable이라고 해서 도메인 로직이 null을 받아주지 않을 수 있다**는 점이다. 위 예시의 `jobId`는 nullable이 아니라(`"type": {"type":"string"}`) Java handler가 CANCEL 분기에서 jobId를 검증하지 않더라도 Avro 직렬화 자체가 null을 허용하지 않는다. 스키마와 도메인 정책 둘을 함께 본다.

### 운영 멱등성과 테스트의 합의점

REST `/jobs/dispatch`도 Kafka cmd 토픽도 같은 도메인 진입점인데, 둘 다 **`receiveIfAbsent` 패턴**을 쓰고 있을 가능성이 높다. 즉 같은 `jobExcnId`로 두 번 들어오면 두 번째는 무동작. 테스트는 status_code나 produce 결과만 보면 안 되고 **DB에 새 INSERT가 일어났는지, 새 빌드가 트리거됐는지**까지 확인해야 의미 있는 검증이 된다.

```python
def test_redispatch_after_terminal(http, db, seed_run_param):
    body = dispatch_payload(...)
    seed_run_param(body["jobExcnId"], "E2E-FAILURE", {})
    assert _post(http, body).status_code == 200
    final = wait_status(db, body["jobExcnId"], lambda s: s in TERMINAL_OK, timeout=120)

    # 첫 dispatch 결과 캐시
    with db.cursor() as cur:
        cur.execute("SELECT EXCN_STTS, BUILD_NO FROM TB_TRB_EC_001 WHERE JOB_EXCN_ID=%s", (body["jobExcnId"],))
        before_status, before_build_no = cur.fetchone()

    # 같은 ID 재dispatch — 운영 정책상 새 실행 일어나면 안 됨
    r2 = _post(http, body)
    assert r2.status_code in (200, 400, 409, 422)
    time.sleep(3)

    with db.cursor() as cur:
        cur.execute(
            "SELECT COUNT(*), MAX(EXCN_STTS), MAX(BUILD_NO) FROM TB_TRB_EC_001 WHERE JOB_EXCN_ID=%s",
            (body["jobExcnId"],),
        )
        cnt, after_status, after_build_no = cur.fetchone()
    assert cnt == 1
    assert after_status == before_status
    assert after_build_no == before_build_no
```

REST status_code는 200/4xx 중 어느 쪽이라도 받아들이고, 실제 검증은 **단일 row, 상태 보존, BUILD_NO 보존** 세 가지 불변식으로 한다. 이 패턴은 cancel 테스트에도 그대로 적용된다 (terminal 후 cancel은 Avro 발행 자체는 성공해도 도메인이 무동작이어야 한다).



## 함정과 회피

### Schema Registry가 다른 Avro encoding을 받으면 컨슈머가 깨진다

운영 JVM 측이 `org.okestro.tps.avro.operator.BuildCommandAvro` namespace로 등록해 둔 스키마와 Python에서 임베드한 스키마의 namespace/이름이 다르면 Schema Registry가 새 schema id를 받아 등록한다. 동일 토픽에 두 가지 schema id가 섞이면 consumer가 한쪽을 못 알아먹어 group이 깨질 수 있다. **반드시 generated source의 SCHEMA$ JSON을 그대로 복사해 namespace/이름/필드 순서까지 일치**시켜야 한다.

검증법: 발행 후 Schema Registry에 등록된 subject 버전을 본다.

```bash
curl -s "http://localhost:8084/subjects/tps.v305p.operator.cmd.build-value/versions/latest" \
  | python3 -m json.tool
```

`schema` 필드의 JSON을 generated SCHEMA$와 diff한다.

### `produce()`만 하고 `flush()` 안 하면 메시지가 안 나간다

confluent-kafka는 비동기 큐에 넣고 백그라운드 스레드가 보낸다. test 함수가 `produce()` 직후 종료하면 인터프리터가 producer를 GC하면서 큐가 통째로 버려질 수 있다. **반드시 `producer.flush(timeout)`**으로 디스크/네트워크까지 떨어진 것을 확인하고 끝낸다.

```python
producer.produce(topic=..., key=..., value=payload)
producer.flush(10)  # 10초 안에 broker ack 받지 못하면 예외
```

### timeout이 짧으면 Schema Registry 등록에서 막힌다

Schema Registry가 first publish에서 schema를 등록할 때 HTTP POST 한 번이 더 일어난다. broker는 9092고 SR은 8084로 서로 다른 호스트일 수 있어 네트워크 RTT가 더 든다. `flush(timeout)`을 너무 짧게 잡으면 등록 자체는 성공했는데 ack 회수 전에 timeout이 나서 예외가 나는 어색한 상황이 생긴다. 첫 발행은 10초 이상으로 잡는다.

### 같은 jobExcnId로 중복 발행 시 컨슈머가 두 번 처리할 수 있다

토픽 자체에 멱등성이 없다. producer가 한 번 더 발행하면 같은 메시지가 토픽에 두 개 들어가고 consumer는 두 번 받는다. 멱등성은 **도메인 측(`receiveIfAbsent` 같은 가드)**이 책임진다. 테스트가 같은 jobExcnId로 의도적으로 두 번 발행하는 경우(중복 시나리오 검증)는 첫 번째 발행 후 충분히 기다리지 않으면 두 번째 발행이 첫 번째를 처리 중인 도메인과 race를 만든다. 가드의 의미를 검증할 거면 첫 번째가 terminal에 도달한 뒤 두 번째를 보낸다.

### nullable이 아닌 필드에 None을 넣으면 직렬화 단계에서 죽는다

```python
payload = {"jobExcnId": None, ...}  # ❌ AvroSerializer raises
```

스키마가 `["null","string"]`이 아니라 `"string"`만이면 None 금지. Python 함수 시그니처가 Optional이라 None 기본값을 줘 둔 경우 이 함정에 잘 빠진다. **schema와 함수 시그니처를 같이 본다.**



## TPS 사례

`qa/python/_lib/kafka.py`는 consumer 헬퍼 옆에 producer 헬퍼를 추가하는 형태로 운영 경로 검증을 도입했다. 5분 안에 한 케이스가 끝나도록 cancel 시나리오 4건을 정의하고, RUNNING 분기까지 진짜로 도달하는지를 BUILD_NO와 FAIL_RSN 두 단언으로 잡는다.

```python
# qa/python/_lib/kafka.py
def produce_build_command(*, event_type, job_excn_id, job_id, job_vsrn=1,
                          job_path=None, tl_id=None, user_id="tester",
                          handler="JENKINS", timeout=10.0):
    sr_client = SchemaRegistryClient({"url": SCHEMA_REGISTRY})
    avro_serializer = AvroSerializer(sr_client, BUILD_COMMAND_SCHEMA)
    producer = SerializingProducer({
        "bootstrap.servers": KAFKA_BOOTSTRAP,
        "key.serializer": StringSerializer("utf_8"),
        "value.serializer": avro_serializer,
    })
    payload = {
        "eventType": event_type, "jobExcnId": job_excn_id, "jobId": job_id,
        "jobVsrn": job_vsrn, "jobPath": job_path, "tlId": tl_id,
        "priorityDt": None, "userId": user_id, "handler": handler,
    }
    producer.produce(topic=TOPICS["OPERATOR_CMD_BUILD"], key=job_excn_id, value=payload)
    producer.flush(timeout)
```

cancel 테스트는 RUNNING 진입을 명시적으로 기다린 뒤에야 cancel을 발행한다. SUBMITTED에서 잡히면 Jenkins 큐만 cancel되고 빌드는 시작도 안 한다 — 그러면 Jenkins UI에 ABORTED 빌드 페이지가 안 남고, `CANCELLED_FROM_RUNNING` 분기는 한 번도 안 타게 된다.

```python
@pytest.mark.cancel
def test_cancel_running_results_in_aborted(http, db, seed_run_param):
    body = dispatch_payload(tl_id="TL_E2E", job_path="job/SBH/job/E2E/job/E2E-LONG-LOG")
    seed_run_param(body["jobExcnId"], "E2E-LONG-LOG", {})
    assert _post_dispatch(http, body).status_code == 200

    # RUNNING 단독 진입까지 대기 — SUBMITTED 에서 잡히면 큐 cancel 만 되고 빌드 페이지 안 남음
    in_flight = wait_status(db, body["jobExcnId"], lambda s: s == "RUNNING", timeout=120)
    assert in_flight == "RUNNING"

    _send_cancel(body["jobExcnId"], "E2E-LONG-LOG")
    final = wait_status(db, body["jobExcnId"],
                        lambda s: s in CANCEL_TERMINAL or s in TERMINAL_OK, timeout=120)
    assert final == "ABORTED"

    # Jenkins stopBuild 가 실제로 호출됐다는 두 증거
    with db.cursor() as cur:
        cur.execute("SELECT BUILD_NO FROM TB_TRB_EC_001 WHERE JOB_EXCN_ID=%s", (body["jobExcnId"],))
        build_no = cur.fetchone()[0]
        cur.execute(
            "SELECT FAIL_RSN FROM TB_TRB_EC_002_P "
            "WHERE JOB_EXCN_ID=%s AND STTS_CD='ABORTED' ORDER BY REG_DT DESC LIMIT 1",
            (body["jobExcnId"],),
        )
        fail_rsn = cur.fetchone()[0]
    assert build_no is not None and build_no > 0
    assert "CANCELLED_FROM_RUNNING" in fail_rsn
```

이 테스트가 PASS 한 뒤 Jenkins UI `/job/SBH/job/E2E/job/E2E-LONG-LOG/45/`에 가 보면 ABORTED 빌드 페이지가 실제로 남는다 — 운영 관점에서 `stopBuild`가 호출됐다는 살아 있는 증거다. 같은 시나리오가 wait 조건을 `{SUBMITTING, SUBMITTED, RUNNING}`로 느슨하게 잡으면 SUBMITTED에서 cancel이 잡혀 BUILD_NO=NULL, FAIL_RSN=`CANCELLED_FROM_SUBMITTED`로 끝나고 Jenkins UI에는 흔적이 없다. **wait 조건이 곧 "어느 분기를 검증할 것인가"의 결정**이다.
