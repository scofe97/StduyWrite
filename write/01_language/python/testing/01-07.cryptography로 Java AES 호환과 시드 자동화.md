# cryptography로 Java AES 호환과 시드 자동화

통합 테스트가 진짜 시스템과 같은 데이터로 돌려면 시드(seed) 단계가 필요하다. `qa/python` 의 시드는 Jenkins 자격증명을 DB에 미리 박아 두는 일이 본질이고, 그 과정에서 Java 측 `CryptoService` 와 *비트 단위로 같은* AES 결과를 Python으로 만들어야 한다. 한 글자만 달라도 운영 코드가 복호화에 실패해 dispatch 흐름 전체가 깨진다. 이 챕터는 cryptography 라이브러리의 AES/CBC/PKCS7 사용·키와 IV 파생 정책·시드 템플릿 + render 스크립트 자동화를 한 그릇에 정리한다.



## 왜 Python 측에 암호화가 필요한가

운영 측 자격증명은 Java 애플리케이션이 읽기 전에 *암호화된 상태로* DB에 저장된다. `TB_TPS_TL_001.TL_SGNL` 컬럼이 그 자리이고, Java `CryptoService` 가 부팅 시 `trb.crypto.key` 프로퍼티의 비밀키로 복호화한다. 알고리즘은 AES/CBC/PKCS5Padding이고 결과는 Base64다.

테스트가 dev Jenkins에 진짜 호출을 보내려면 *진짜 토큰*을 평문으로 알아야 한다. 동시에 그 토큰이 *DB에는 암호화돼* 있어야 한다. 두 표현을 잇는 단계가 시드 자동화다. `.env` 에 평문 토큰을 두고, render 스크립트가 그 토큰을 Java와 같은 방식으로 암호화해 SQL 시드에 박는다. `.env` 자체는 git에 올리지 않고, 결과 SQL도 `.gitignore` 로 막는다. 평문 토큰은 로컬에만 머문다.

이 자동화의 핵심은 *Python의 결과가 Java와 비트 단위로 같다*는 점이다. 한 글자만 달라도 Java가 해독에 실패한다.



## AES/CBC/PKCS5Padding을 Python으로 — cryptography 라이브러리

Python에서 AES를 만지는 라이브러리가 두 개 있다. `cryptography` (PyCA 권장 표준)와 `pycryptodome` (오래된 구현의 fork). `qa/python` 은 `cryptography` 를 고른다. 공식 권장이고 의존성이 가볍다.

`cryptography` 는 두 단으로 나뉜다. `cryptography.fernet` 같은 *권장 고수준 API*와 `cryptography.hazmat`(Hazardous Materials) *저수준 API*다. AES/CBC를 직접 다루는 건 `hazmat` 영역에 들어 있다. 이름이 시사하듯 "잘못 쓰면 위험하다"는 표시이고, 일반 비밀번호 저장에는 fernet이나 비밀번호 해시 라이브러리(argon2·bcrypt)가 답이다. 이 코드는 *기존 Java 구현과의 호환을 위해* 저수준을 어쩔 수 없이 쓴다.

`_lib/crypto.py` 의 핵심 함수다.

```python
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
from cryptography.hazmat.primitives.padding import PKCS7


def _key_iv(secret: str) -> tuple[bytes, bytes]:
    raw = secret.encode("utf-8")
    key = raw[:16].ljust(16, b"\x00")
    iv = secret[:16].encode("utf-8")
    if len(iv) != 16:
        raise ValueError("trb.crypto.key 는 UTF-8 기준 16 바이트 이상이어야 합니다.")
    return key, iv


def encrypt(plain: str, secret: str) -> str:
    if plain is None or plain == "":
        return ""
    key, iv = _key_iv(secret)
    padder = PKCS7(128).padder()
    padded = padder.update(plain.encode("utf-8")) + padder.finalize()
    cipher = Cipher(algorithms.AES(key), modes.CBC(iv))
    enc = cipher.encryptor()
    ct = enc.update(padded) + enc.finalize()
    return base64.b64encode(ct).decode("ascii")
```

흐름이 짧다. (1) secret으로 key/IV 파생 → (2) 평문에 PKCS7 패딩 추가 → (3) AES/CBC 암호화 → (4) Base64 인코딩. 각 단계가 Java 측 `CryptoService` 와 정확히 일치해야 한다.



## PKCS5와 PKCS7 — 같은 알고리즘, 다른 이름

Java 측 명세에는 `AES/CBC/PKCS5Padding` 이라고 적혀 있는데, Python 측 코드는 `PKCS7(128).padder()` 를 쓴다. 다른 알고리즘처럼 보이지만 *AES 16바이트 블록에서는 정확히 같은 동작*이다.

PKCS5는 8바이트 블록 암호(DES) 시절의 패딩 표준이다. PKCS7은 일반화된 표준으로 1~255바이트 블록을 지원한다. AES의 블록은 16바이트라 PKCS7로 처리되며, "AES/CBC/PKCS5Padding" 이라는 Java 표기는 역사적 호환성 표기일 뿐이다. 같은 16바이트 블록에서 두 표준은 결과가 똑같다.

PKCS7 패딩의 동작은 직관적이다. 평문을 16바이트 블록에 맞추기 위해 부족한 바이트 수만큼 *그 수의 값*으로 채운다. 평문이 13바이트면 끝에 `0x03 0x03 0x03` 을 붙여 16바이트로 만든다. 평문이 정확히 16바이트면 16바이트짜리 블록(`0x10 0x10 ... 0x10`)을 통째로 추가한다. 복호화 시 마지막 바이트 값을 읽어 패딩 길이를 알아낸다.

이 변환을 코드로 보면 두 줄이다.

```python
padder = PKCS7(128).padder()
padded = padder.update(plain.encode("utf-8")) + padder.finalize()
```

`PKCS7(128)` 의 128은 비트 단위 블록 크기다(16 바이트 = 128 비트). AES만 쓰면 항상 128이다.



## 키와 IV 파생 — Java와 정확히 같아야 한다

암호화 결과를 Java와 일치시키려면 키와 IV가 같아야 한다. Java 측 `CryptoService` 의 정책은 다음이다.

> key: `trb.crypto.key` 의 UTF-8 앞 16 바이트 (부족 시 0x00 패딩, 16 바이트 초과분은 절단)
> iv: 같은 `trb.crypto.key` 의 첫 16 바이트

Python 측 `_key_iv` 함수가 이 정책을 그대로 옮긴다.

```python
def _key_iv(secret: str) -> tuple[bytes, bytes]:
    raw = secret.encode("utf-8")
    key = raw[:16].ljust(16, b"\x00")    # 앞 16바이트, 부족하면 0x00 패딩
    iv = secret[:16].encode("utf-8")     # 처음 16자 → UTF-8 인코딩
    if len(iv) != 16:
        raise ValueError("trb.crypto.key 는 UTF-8 기준 16 바이트 이상이어야 합니다.")
    return key, iv
```

세 디테일이 결정적이다. 첫째, **UTF-8 명시**. Java의 `String.getBytes()` 기본은 플랫폼 의존이라(서버 OS에 따라 UTF-8 또는 다른 인코딩) `getBytes("UTF-8")` 또는 `StandardCharsets.UTF_8` 명시가 필요하다. Python 쪽도 `.encode("utf-8")` 로 같은 인코딩을 박는다. 한쪽이 한글·유니코드 비트를 다르게 처리하면 결과가 달라진다. 둘째, **`ljust(16, b"\x00")`** 패딩. secret이 16바이트보다 짧으면 0x00으로 채운다. Java의 `Arrays.copyOf(...)` 도 부족분을 0으로 채우니 같은 결과다. 셋째, **`raw[:16]`** 슬라이싱. 16바이트 초과분은 절단. Java도 같은 결정이다.

이 정책에는 약점이 있다. **IV가 secret에서 파생되므로 같은 평문이 늘 같은 ciphertext를 만든다**. 일반적으로 IV는 매 암호화마다 random이어야 ciphertext의 동일성을 깨고, 이를 평문과 함께 저장한다. 이 코드는 학습/테스트 환경의 단순성을 위해 deterministic한 IV를 쓴다 — 같은 토큰을 두 번 시드해도 SQL이 같아 비교가 쉽다는 운영 편의가 이유다. 운영 보안 관점에서는 random IV + IV 함께 저장이 정공법이고, 그렇게 가려면 Java 측도 함께 바꿔야 한다.



## render_seed.py — 시드 자동화 파이프라인

`scripts/render_seed.py` 는 위 cryptography를 한 번 호출해 SQL 템플릿의 placeholder를 치환한다. 70줄이 안 되는 짧은 스크립트지만 *환경변수 검증·암호화·파일 IO·결과 기록* 네 단계를 한 흐름으로 묶는다.

```python
load_dotenv(QA_PY / ".env")

REQUIRED = ["JENKINS_USER", "JENKINS_TOKEN", "TRB_CRYPTO_KEY"]
missing = [k for k in REQUIRED if not os.environ.get(k)]
if missing:
    sys.exit(f"qa/python/.env 에 필요한 변수가 비어 있습니다: {', '.join(missing)}")

jenkins_user = os.environ["JENKINS_USER"]
jenkins_token = os.environ["JENKINS_TOKEN"]
crypto_key = os.environ["TRB_CRYPTO_KEY"]
token_cipher = encrypt(jenkins_token, crypto_key)

REPO_ROOT = QA_PY.parent.parent
SRC = QA_PY / "executor_jenkins" / "seeds" / "25-executor-jenkins-fixtures.sql.template"
DST = REPO_ROOT / "operator/app/src/main/resources/db/sbh-local/25-executor-jenkins-fixtures.sql"

content = SRC.read_text(encoding="utf-8")
content = content.replace("__JENKINS_USER__", jenkins_user)
content = content.replace("__JENKINS_TOKEN_CIPHER__", token_cipher)

DST.parent.mkdir(parents=True, exist_ok=True)
DST.write_text(content, encoding="utf-8")
print(f"wrote: {DST}")
```

네 가지를 짚는다. 첫째, **early validation**. 필수 환경변수가 비어 있으면 즉시 sys.exit으로 명확한 메시지와 함께 종료한다. 늦게 발견되면 디버깅이 길어진다. 둘째, **placeholder 두 개**. `__JENKINS_USER__` 는 평문(공개 가능), `__JENKINS_TOKEN_CIPHER__` 는 암호문. 단순 `.replace()` 로 치환한다 — Jinja2 같은 템플릿 엔진은 의존성이 무거워 과한 도구다. 셋째, **출력 경로의 명확성**. 결과 SQL은 운영 코드의 `operator/app/src/main/resources/db/sbh-local/` 디렉토리에 떨어진다. 그곳이 `spring.sql.init` 의 `data-locations` 끝에 등록되어 부팅 시 자동 INSERT된다. SQL이 *실제로 동작하는 위치*에 그대로 떨어지니 추가 복사 단계가 없다. 넷째, **mkdir 안전성**. 첫 실행이면 디렉토리가 없을 수 있어 `parents=True, exist_ok=True` 로 자동 생성한다.



## 시드 템플릿 — placeholder를 박는 SQL

`seeds/25-executor-jenkins-fixtures.sql.template` 의 일부다.

```sql
INSERT INTO TB_TPS_TL_001
  (TL_ID, TL_NM, TL_TYPE, TL_CTGRY, TL_URL, TL_CNTN_ID, TL_SGNL, ...)
VALUES
  ('TL_QA_DEV',
   'QA Jenkins (dev)',
   'CICD',
   'JENKINS',
   'https://jenkins.dev.console.trombone.okestro.cloud',
   '__JENKINS_USER__',
   '__JENKINS_TOKEN_CIPHER__',
   ...);
```

placeholder가 SQL 안의 *문자열 리터럴 자리*에 들어 있다. 단순 `.replace()` 로 치환되므로 토큰에 작은따옴표가 들어 있으면 SQL이 깨진다. 그래서 토큰 명세에 "공백·작은따옴표 금지"를 두거나, render 스크립트에 SQL escape 단계를 추가하는 보강이 필요하다(현재 코드에는 없음). 본인 학습 단계에서는 자기 토큰을 알고 있으니 위험이 작다.

`__JENKINS_TOKEN_CIPHER__` 자리의 결과는 Base64라 영문/숫자/+, /, = 만 들어간다. SQL 안전 문자라 escape 걱정이 없다. 결과적으로 *암호화의 부수효과로 SQL escape 문제도 회피된다*. 의도한 설계는 아니지만 좋은 시너지다.



## .env 격리와 .gitignore — 평문이 새 나가지 않게

평문 비밀(JENKINS_TOKEN)을 다루는 자동화는 *어디에 평문이 머무는가*가 핵심이다. `qa/python` 의 답은 단순하다.

| 위치 | 평문 보유 | 격리 방식 |
|------|-----------|-----------|
| `qa/python/.env` | 평문 | `.gitignore` (`.env.example` 만 커밋) |
| `qa/python/.env.example` | 더미 값 | git에 커밋 (값은 채우지 않은 placeholder) |
| 결과 SQL | 암호문만 | `.gitignore` |
| 운영 DB | 암호문만 | DB ACL |

평문이 살아 있는 곳은 `.env` 파일 한 곳뿐이고, 그 위치가 `.gitignore` 로 격리된다. 결과 SQL도 운영 디렉토리에 떨어지지만 `.gitignore` 에 등록돼 commit 대상이 아니다.

이 격리가 깨지는 사고는 두 가지에서 온다. 하나는 `.env` 를 `.env.bak` 같은 다른 이름으로 복사해 두는 경우 — `.gitignore` 의 패턴이 그 이름까지 포함하지 않으면 백업 파일이 git에 들어간다. 다른 하나는 결과 SQL을 잘못된 경로로 빼내 `.gitignore` 가 적용되지 않는 위치로 옮기는 경우. 자동화 스크립트의 출력 경로를 안전한 위치로 *고정*하는 결정이 안전망이다.



## 함정 모음

**`getBytes()` 인코딩 누락**. Java 코드에서 `secret.getBytes()` 를 쓰고 인코딩을 명시하지 않으면 플랫폼 기본(서버에 따라 UTF-8 또는 EUC-KR 등)으로 변환된다. 한글이나 유니코드 비밀키가 있으면 환경마다 결과가 달라진다. Python에서 같은 결과를 내려고 해도 매칭이 안 된다. Java 측 코드가 `StandardCharsets.UTF_8` 을 명시하는지 먼저 확인.

**키 길이 부족**. AES-128 키는 정확히 16바이트여야 한다. `trb.crypto.key` 가 UTF-8 변환 후 16바이트 미만이면 패딩이 들어간다. Java와 Python의 패딩 정책이 일치해야 결과가 같다. `_key_iv` 의 `ljust(16, b"\x00")` 가 그것이다.

**IV 정책 누락**. CBC 모드는 IV가 필요하다. Java가 IV를 어디서 가져오는지(secret 파생/random/별도 컬럼)를 명확히 알아야 Python 측에서 같은 IV를 만들 수 있다. `qa/python` 은 secret 파생이라 단순하지만, 운영 환경 중 random IV + 함께 저장하는 변형도 흔하다.

**Base64 변형**. 표준 Base64(`+`, `/`, `=`)와 URL-safe Base64(`-`, `_`)는 다르다. Java `Base64.getEncoder()` 는 표준이고, Python `base64.b64encode` 도 표준이다. 둘이 같으니 문제없지만, 다른 측이 URL-safe를 쓰면 변환이 필요하다.

**암호화 결과의 가변성 vs 결정성**. 같은 (key, IV)가 같은 결과를 만드는 결정성은 디버깅에 좋지만 보안에는 불리하다. 운영 보안 관점에서 random IV가 정공법이고, 이 chapter 코드는 학습/테스트의 결정성 편의를 위해 deterministic IV를 쓰고 있다는 점을 기억해 둔다.

**SQL 안의 placeholder가 의외의 이스케이프 처리**. ORM의 placeholder(`%s`)와 단순 `.replace()` placeholder(`__X__`)는 다르다. 후자는 escape이 안 된다. 토큰에 작은따옴표가 들어가면 SQL이 깨진다. 앞에서 짚었듯 Base64 결과는 안전 문자만 갖지만, 평문 user 같은 자리에는 escape를 고려해야 한다.

**`hazmat` 모듈 import 경고**. `cryptography.hazmat` 은 의도된 위험 표시다. 일반 비밀번호 저장이라면 `bcrypt`·`argon2`·`fernet` 으로 가야 한다. 기존 Java 구현과의 호환을 위해 어쩔 수 없이 쓰는 경우만 hazmat이 정당하다.



## 비교 — pycryptodome vs Jasypt 호환

**pycryptodome**. PyCryptodome은 PyCrypto의 fork로 더 활발한 유지보수가 이뤄진다. AES API가 짧고 직관적이다. cryptography보다 의존성이 가볍다.

```python
# pycryptodome 버전
from Crypto.Cipher import AES
from Crypto.Util.Padding import pad

cipher = AES.new(key, AES.MODE_CBC, iv)
ct = cipher.encrypt(pad(plain.encode("utf-8"), AES.block_size))
```

`cryptography` 와 결과는 같다. 큰 차이는 라이브러리 정책 — `cryptography` 는 PyCA(Python Cryptographic Authority) 의 공식 권장이라 보안 패치가 빠르고 hazmat 표시로 잘못된 사용을 경고한다. 새 프로젝트면 `cryptography` 를 고르는 편이 안전하다. `qa/python` 의 선택이 그것이다.

**Jasypt 호환**. Java 측에서 Jasypt(`StandardPBEStringEncryptor`)를 쓰고 있다면 IV 파생이 다르다. Jasypt는 password-based encryption으로 PBKDF2 같은 키 파생 함수를 쓴다. 같은 결과를 Python으로 내려면 PBKDF2 단계까지 매칭해야 하므로 코드가 길어진다. `qa/python` 의 `CryptoService` 는 직접 AES/CBC라 매칭이 단순하다. 운영 측 라이브러리가 무엇인지가 호환 가능성의 첫 변수다.
