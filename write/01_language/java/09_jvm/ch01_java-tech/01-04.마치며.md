# 마치며 — 1장이 다음 장에 거는 토대
---
> 1장의 마지막 §1.7은 짧다. 책 전체에서 1장이 어떤 역할을 하는지 한 번 더 정리하고, 2장 이후로 넘어가기 위한 시야를 잡아 준다. 1장 전체를 한 줄로 요약하면 다음과 같다 — **JDK는 명세와 구현과 도구의 합이며, 그 합이 클라우드 네이티브 환경의 압박에 맞춰 빠르게 재배치되는 중**이다. 2장부터의 모든 깊은 이야기는 이 한 줄 위에서 펼쳐진다.



## 1. 1장이 정의한 좌표

§1.5에서 본 다섯 갈래의 미래(언어 독립, 차세대 JIT, 가상화 친화성, 함수형 흡수, 모던 문법)는 모두 *지금 자바가 어디로 움직이는지*를 보여 주는 좌표다. 책의 나머지 장들은 이 좌표 위에서 JVM 내부를 한 층씩 벗긴다. 2부는 메모리와 GC, 3부는 클래스 로딩과 바이트코드 실행 엔진, 4부는 컴파일러, 5부는 동시성 모델이다. 어느 층을 보든 §1.5의 다섯 갈래 중 하나 이상과 연결된다.

§1.6에서 자기 손으로 OpenJDK를 빌드한 경험은 다음 장부터 가장 중요한 자산이 된다. 2부 GC 알고리즘을 읽다가 *실제로 어떤 옵션을 켜면 어디서 분기되는지* 궁금해질 때, 책 본문만 읽고 넘어가는 것과 직접 빌드한 slowdebug 바이너리에 브레이크포인트를 거는 것 사이에는 깊이의 차이가 크다.



## 2. 2장 이후로 가져갈 세 가지 질문

1장을 마무리하면서 들고 갈 만한 질문 세 가지를 정리한다. 이 질문들은 책 전체를 일관된 시야로 읽게 한다.

1. **이 동작은 클라우드 네이티브 환경에서 어떻게 변했나?** §1.5.3의 가상화 친화성을 떠올린다. 시작 시간, 메모리 풋프린트, 워밍업 비용이 운영의 핵심 지표가 된 시점에서 각 JVM 기능이 어떻게 재설계됐는지 본다.
2. **이 동작은 그랄(Graal)에서 어떻게 다른가?** 클래식 핫스팟의 동작을 익히되, Graal이 같은 문제를 다른 IR과 다른 컴파일 전략으로 풀고 있다는 점을 의식한다.
3. **이 동작을 직접 디버깅한다면 어디부터 보나?** §1.6.5에서 익힌 IDE 디버깅 흐름을 떠올린다. 읽기만 하는 학습과 *멈춰서 보는 학습*은 깊이가 다르다.



## 3. 다음 노트로의 연결

1장이 끝나면 책의 본격적인 내부 해부가 시작된다. 책 차례에 따르면 다음은 2부 자동 메모리 관리의 2장 "자바 메모리 영역과 메모리 오버플로"다. 본 저장소에서는 해당 정독 노트가 [`../ch02_memory-area/`](../ch02_memory-area/)에 이미 작성돼 있으며 [01-01.런타임 데이터 영역](../ch02_memory-area/01-01.런타임%20데이터%20영역.md)부터 시작한다. 실습 코드는 [`../_code/ch02-memory-area/`](../_code/ch02-memory-area/)에 7개 Gradle 서브모듈로 분리돼 있어, OOM 7종을 *한 모듈에 하나씩* 재현한다.

2장으로 넘어갈 때 한 가지를 미리 챙긴다. 1장 §1.6의 slowdebug 빌드가 손에 있다면, 2장 §2.4의 OOM 예제 각각을 그 바이너리로 돌려 보면서 `-XX:+TraceClassLoading` 같은 옵션 효과를 함께 본다. 이것이 1장이 2장에 남긴 가장 큰 자산이다.
