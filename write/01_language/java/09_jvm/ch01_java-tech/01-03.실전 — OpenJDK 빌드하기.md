# 실전 — OpenJDK 빌드하기
---
> 1장의 마지막 실전 절은 "남이 만든 JDK를 쓰는 사용자"에서 "JVM 자체를 들여다보는 개발자"로 한 발 옮기는 과정이다. 책의 §1.6은 OpenJDK 소스를 받고 빌드 도구를 갖춘 뒤 `configure`와 `make`로 빌드하고, 최종적으로 IDE에서 JVM 소스를 디버깅하는 데까지 간다. 이 노트는 책의 절차를 그대로 박제하되, *왜 그 명령을 그 순서로 쓰는지*에 무게를 둔다.

## 1. §1.6 들어가며 — JDK를 직접 빌드해야 하는 이유

> 운영용 JDK는 보통 벤더가 제공한 바이너리를 쓴다. 그런데도 직접 빌드를 해 보는 이유는 분명하다.

세 가지 이유가 책에 적혀 있다. 첫째, **소스에 패치를 가하거나 옵션을 바꿔서 동작 차이를 실험**할 수 있다. JIT 옵션을 바꿔 가며 핫스팟의 컴파일 결과를 비교하는 실습이 그래야 가능하다. 둘째, **JVM 내부 디버깅이 가능해진다**. 자바 코드만 디버깅해서는 보이지 않는 클래스 로딩 흐름, GC 안전점 진입, JIT 컴파일 트리거 같은 핵심 동작을 C++ 레벨에서 따라가야 한다. 셋째, **OpenJDK 기여**를 노린다면 빌드는 필수 진입점이다.

빌드 대상은 책 기준 OpenJDK 12다. 본 노트는 책의 절차를 박제하되, 실습 환경(현재 시점 OpenJDK 21 LTS)에 맞춘 조정점을 별도로 표시한다.

## 2. §1.6.1 소스 코드 가져오기

> 옛 OpenJDK는 Mercurial(`hg`)로 배포됐지만 지금은 Git으로 이관됐다.

책 시점에는 두 갈래의 저장소가 공존했다. `hg.openjdk.java.net`의 Mercurial 저장소와 GitHub의 미러다. 책은 둘 다 받는 방법을 소개하지만, 현재 시점에는 **Git 미러가 단일 소스**다.

```bash
# 책의 방법 — OpenJDK 12 태그
git clone https://github.com/openjdk/jdk.git
cd jdk
git checkout jdk-12-ga

# 현재 시점 — OpenJDK 21 LTS 권장
git clone https://github.com/openjdk/jdk21u.git
cd jdk21u
```

저장소 크기가 1GB가 넘는 큰 모노레포다. 처음 클론은 인터넷 속도에 따라 십 분 이상 걸린다. shallow clone(`--depth=1`)으로 줄일 수 있지만, 변경 이력 추적이 필요한 학습에서는 권장하지 않는다.

## 3. §1.6.2 시스템 사양

> 빌드 환경은 책 기준 Ubuntu 18.04 (또는 동등한 리눅스)다.

책이 명시하는 권장 사양은 다음과 같다.

| 항목 | 책 권장 (OpenJDK 12) | 본 노트 기준 (OpenJDK 21) |
|------|--------------------|--------------------------|
| OS | Ubuntu 18.04 LTS | Ubuntu 22.04 LTS 또는 macOS 14+ |
| CPU | 2코어 이상 | 4코어 이상 권장 |
| RAM | 4GB | 8GB 이상 권장 (빌드 중 메모리 압박) |
| 디스크 | 6GB 이상 여유 | 10GB 이상 |
| 부트스트랩 JDK | OpenJDK 11 | OpenJDK 20 (한 단계 낮은 LTS) |

부트스트랩 JDK는 OpenJDK가 자기 자신을 빌드할 때 사용하는 *이전 버전 JDK*다. OpenJDK N을 빌드하려면 보통 N-1을 미리 깔아 둬야 한다. 자바 자체가 자기 자신을 컴파일하는 부트스트랩 구조라 이 단계가 필요하다.

## 4. §1.6.3 빌드 도구 설치

> 자바 코드뿐 아니라 네이티브 코드(C/C++)도 컴파일해야 하므로 시스템 라이브러리가 다수 필요하다.

책이 제시하는 Ubuntu 기준 명령은 다음과 같다. 본 노트는 이 명령을 실습 코드 폴더의 스크립트(`_code/ch01/scripts/install-build-deps-ubuntu.sh`)에 그대로 박제했다.

```bash
sudo apt-get install -y \
    build-essential \
    libfreetype6-dev \
    libcups2-dev \
    libx11-dev libxext-dev libxrender-dev libxrandr-dev \
    libxtst-dev libxt-dev \
    libasound2-dev \
    libfontconfig1-dev \
    libffi-dev \
    autoconf \
    zip unzip
```

각 라이브러리의 의미는 다음과 같다.

| 라이브러리 | 어디에 쓰이나 |
|------------|--------------|
| `build-essential` | gcc, g++, make 등 기본 C/C++ 빌드 체인 |
| `libfreetype6-dev` | 폰트 렌더링 (AWT/Swing) |
| `libcups2-dev` | 프린팅 지원 (자바 Print API) |
| `libx11-*`, `libxrender-dev`, `libxtst-dev` | X Window 시스템 — AWT/Swing GUI |
| `libasound2-dev` | ALSA 사운드 — 자바 Sound API |
| `libfontconfig1-dev` | 시스템 폰트 검색 |
| `libffi-dev` | FFI — JNI에서 사용 |
| `autoconf`, `zip`, `unzip` | configure 스크립트 생성과 아카이브 처리 |

macOS에서는 Xcode Command Line Tools가 build-essential 역할을 하고, FreeType과 X11은 Homebrew로 따로 깔거나 빌드 시 `--without-x11` 옵션으로 제외한다.

## 5. §1.6.4 컴파일하기 — configure와 make

> 빌드는 두 단계다. `configure`로 빌드 구성을 만들고 `make`로 실제 컴파일한다.

### 5.1 configure 단계

OpenJDK는 `bash configure` 한 줄로 빌드 환경을 자동 점검한다. 이 단계는 부트스트랩 JDK 위치를 확정하고, 시스템 라이브러리가 모두 깔렸는지 확인하고, 결과물을 어디에 둘지 정한다.

```bash
bash configure \
    --with-boot-jdk=/path/to/jdk-20 \
    --with-debug-level=slowdebug \
    --with-jvm-variants=server \
    --enable-ccache
```

옵션별 의미가 중요하다.

| 옵션 | 의미 | 권장 |
|------|------|------|
| `--with-boot-jdk=<path>` | 부트스트랩 JDK 경로 | 빌드 대상보다 한 단계 낮은 LTS |
| `--with-debug-level=<level>` | 디버그 정보 양 — `release`/`fastdebug`/`slowdebug` | 디버깅 목적이면 `slowdebug` |
| `--with-jvm-variants=<variant>` | JVM 변형 — `server`/`client`/`minimal` | 학습은 `server` |
| `--enable-ccache` | 컴파일 캐시 사용 | 재빌드 가속, 권장 |
| `--with-num-cores=<n>` | 빌드 병렬도 | 보통 자동 감지로 충분 |
| `--without-x11` | X11 의존성 제외 (서버형 빌드) | 데스크탑 기능 필요 없으면 |

`--with-debug-level=slowdebug`를 선택하면 빌드 시간이 두 배 가까이 늘지만, 그 대신 *모든 inline이 풀린 디버거 친화적 바이너리*가 만들어진다. JVM 내부 디버깅이 목적이라면 이 레벨이 표준이다.

### 5.2 make 단계

`configure`가 성공하면 다음 명령으로 빌드한다.

```bash
make images          # 권장 — 완전한 JDK 이미지를 생성
# 또는
make all             # 모든 산출물
# 또는
make hotspot         # HotSpot VM만 빠르게 빌드 (개발 사이클에 유용)
```

빌드 결과물 위치는 `build/<configuration>/images/jdk/`다. 예를 들어 위 설정이면 `build/linux-x86_64-server-slowdebug/images/jdk/bin/java`가 새로 빌드한 JDK 실행 파일이다.

```bash
./build/linux-x86_64-server-slowdebug/images/jdk/bin/java -version
# 결과 예:
# openjdk version "21-internal" 2024-04-15
# OpenJDK Runtime Environment (slowdebug build 21-internal-...)
# OpenJDK 64-Bit Server VM (slowdebug build 21-internal-..., mixed mode)
```

`version` 끝에 `slowdebug`가 붙는 점이 일반 배포 바이너리와의 차이다.

## 6. §1.6.5 통합 개발 환경에서 소스 코드 디버깅하기

> JVM 자체를 C++ 레벨에서 디버깅하려면 IDE 도움이 필요하다.

책은 JetBrains **CLion**을 사용한다. 이유는 OpenJDK가 CMake가 아닌 자체 빌드 시스템이지만, CLion이 `compile_commands.json` 같은 외부 빌드 데이터베이스를 받아 *소스 인덱싱만 위탁*할 수 있기 때문이다. 절차는 세 단계다.

1. **프로젝트 임포트**: CLion의 *New Project from Existing Sources* 로 OpenJDK 루트를 연다.
2. **빌드 디렉터리 연결**: `build/<configuration>/` 폴더의 컴파일 명령 데이터베이스를 CLion에 등록한다. 책의 그림 1-13~1-14는 그 화면을 캡처해 두었다.
3. **Run/Debug Configurations 작성**: HotSpot 실행 파일(`java`)을 디버거에 연결하고, 실행 인자로 자바 클래스를 지정한다. JVM이 그 클래스를 로드하기 시작하면 C++ 코드에 설정한 브레이크포인트가 잡힌다.

책은 다음과 같은 실험을 권한다. `ClassFileParser::parse_constant_pool` 같은 핫스팟 내부 함수에 브레이크포인트를 걸고, 위에서 빌드한 JDK로 `HelloWorld.class`를 실행해 보면, JVM이 클래스 파일을 읽어 상수 풀을 채우는 순간을 한 줄씩 따라갈 수 있다. 이 경험이 §3부 클래스 로딩 메커니즘을 읽을 때의 정확도를 비약적으로 끌어올린다.

### 6.1 IntelliJ로 자바 레벨 디버깅도 동시에

> C++ 디버깅과 자바 디버깅을 같이 보고 싶으면 IntelliJ를 함께 띄운다.

CLion이 HotSpot을 잡고 있는 동안, IntelliJ에서 같은 자바 프로세스에 *원격 디버거*로 attach하면 자바 코드와 JVM C++ 코드를 동시에 멈출 수 있다. 책의 그림 1-15가 이 시나리오를 보여 준다 — IntelliJ에서 `HelloWorld.main` 라인을 멈춘 상태로 CLion에서 클래스 로더 C++ 콜스택을 보는 화면이다.

## 7. 핵심 정리

OpenJDK 빌드는 다음 다섯 단계로 정리된다.

1. 소스 받기: `git clone openjdk/jdk21u` (또는 책 기준 `openjdk/jdk`의 `jdk-12-ga` 태그)
2. 부트스트랩 JDK 준비: 빌드 대상보다 한 단계 낮은 LTS
3. 빌드 도구 설치: `build-essential` + 폰트/X11/사운드 dev 패키지
4. `bash configure --with-boot-jdk=... --with-debug-level=slowdebug ...`
5. `make images` → `build/.../images/jdk/bin/java -version`

이 다섯 단계가 끝나면 *나만의 OpenJDK 바이너리*가 생긴다. 이후 §3부 클래스 로딩, §4부 컴파일러를 읽을 때 이 바이너리에 직접 옵션을 걸고 동작을 관찰할 수 있다.

| 실습 | 위치 | 다루는 것 |
|------|------|---------|
| Ubuntu 빌드 도구 설치 스크립트 | `_code/ch01/scripts/install-build-deps-ubuntu.sh` | 책 §1.6.3의 apt-get 명령을 그대로 박제 |
| OpenJDK 클론과 빌드 스크립트 | `_code/ch01/scripts/build-openjdk.sh` | clone → configure → make 자동화 (사용자 확인 후 실행) |
| 현재 JVM 메타정보 확인 | `_code/ch01/src/main/java/org/runners/jvm/ch01/JavaTechSystemDemo.java` | 빌드 전후 JVM이 어떻게 다른지 비교용 |

빌드는 환경 의존도가 커서 *실행은 사용자 명시 요청 시*에 한다. 스크립트는 박제용으로만 둔다.
