# 핫스팟 알고리즘 상세 구현
---
> §3.3이 *어떤 알고리즘*인지 정의했다면, §3.4는 *그 알고리즘을 핫스팟이 어떻게 실제로 돌리느냐*를 다룬다. GC의 정확성과 성능을 동시에 잡기 위한 다섯 가지 부품이 등장한다 — OopMap, 안전 지점, 안전 영역, 기억 집합과 카드 테이블, 쓰기 장벽이다. 다섯 부품은 따로 동작하지 않고 서로를 전제로 한다.

## 1. OopMap — 빠르고 정확한 루트 열거

> *어디에 객체 참조가 있는지* 를 알아야 GC가 그 참조를 따라간다. 그런데 자바 스택의 슬롯 한 칸이 `int`인지 `reference`인지 어떻게 구분하는가?

가장 단순한 방법은 *전수 조사*다. 자바 스택의 모든 슬롯을 *값을 보고* 객체 참조인지 추측하는 것 — 보수적(conservative) GC 라고 부른다. C 언어의 BoehmGC 가 이 방식이다. 단점은 *정확도가 떨어진다* — `int` 값이 우연히 객체 주소처럼 보이면 *살아 있지 않은 객체를 살아 있다*고 잘못 판정해서 못 회수한다.

핫스팟은 정확한 GC를 위해 **OopMap** 자료구조를 쓴다. *컴파일 시점*에 메서드의 각 명령어 위치마다 *어느 스택 슬롯이 객체 참조인가*를 미리 적어 둔다.

```
// 의사 표현 — 메서드 m()의 bytecode offset 12 시점의 OopMap
// "스택 슬롯 1, 3, 7번이 object reference, 나머지는 primitive"
m@12: oop_slots = [1, 3, 7]
```

GC가 스택을 훑을 때 OopMap만 보면 *어디를 따라가야 하는지* 정확히 안다. 보수적 GC의 *추측*이 사라진다.

문제는 OopMap이 *모든 명령어 위치마다 만들어질 수는 없다*는 점이다. 메모리 비용이 비대해진다. 그래서 핫스팟은 *특정 위치*에만 OopMap을 저장하고, 그 위치를 **안전 지점**이라고 부른다.

## 2. 안전 지점 (Safepoint) — *언제* GC를 시작하는가

> 모든 애플리케이션 스레드가 *동시에 멈춰야* GC가 일관된 상태에서 동작한다. 어디서 멈추느냐가 안전 지점의 문제다.

안전 지점은 *OopMap이 정확히 정의된 지점*이다. 핫스팟은 다음 위치에 안전 지점을 둔다.

- **메서드 호출**
- **루프 백 엣지** (긴 루프 안에서 GC가 무한히 기다리지 않게)
- **예외 처리 분기**
- **JIT가 컴파일한 코드의 다른 분기**

스레드가 GC 요청을 받으면 *가까운 안전 지점까지 진행한 뒤* 멈춘다. 이 동기화를 어떻게 구현하느냐가 두 갈래.

### 2.1 선점형 중단 (Preemptive Suspension) — 폐기됨

GC가 모든 스레드를 *강제로 일시 정지*한 뒤, 안전 지점이 아닌 스레드는 *깨워서 안전 지점까지 진행하고 다시 멈추게* 한다. 구현은 단순하지만 *복잡한 동기화*가 필요하고, *모든 사례에서 작동한다고 보장하기 어렵다*. 핫스팟이 일찍 폐기.

### 2.2 자발적 중단 (Voluntary Suspension) — 핫스팟의 채택

각 안전 지점에 *체크 명령*을 컴파일 시점에 박는다. JIT 코드에 *메모리 페이지 한 곳을 읽는 명령*이 들어가는데, GC가 그 페이지를 *접근 불가*로 표시하면 모든 스레드가 *세그멘테이션 폴트로 트랩*에 빠진다. 트랩 핸들러가 스레드를 안전 지점에서 멈춘다.

```
// 의사 코드 — JIT가 안전 지점마다 박는 폴링
movq <safepoint_polling_address>, <scratch>  // 페이지 한 곳 읽기
// GC가 페이지를 접근 불가로 만들면 여기서 SIGSEGV → trap handler
```

페이지 한 번 읽기는 *정상 동작 시 거의 비용이 없고* (CPU 캐시에 있음), GC가 필요할 때만 *trap*으로 멈춤이 발동한다. *Read Barrier의 가장 가벼운 형태*다.

## 3. 안전 영역 (Safe Region) — *기다리는 스레드*를 위한 확장

> 안전 지점은 *실행 중인 스레드*에 적합하지만, *Sleep·Block 중인 스레드*는 안전 지점에 도달할 수 없다.

`Thread.sleep()` 으로 잠든 스레드, `wait()` 으로 대기 중인 스레드, IO 블로킹 중인 스레드는 *명령을 실행하지 않으므로* 안전 지점 폴링도 못 한다.

해결책은 **안전 영역**이라는 개념이다. 한 범위의 코드 전체가 *GC와 호환 가능한 상태*를 유지하면, 그 안에서는 *언제 멈춰도 안전*이라고 선언한다. 스레드는 안전 영역에 들어갈 때 *플래그를 켜고*, 나갈 때 *GC가 끝났는지 확인*한다. GC가 진행 중이면 *대기*. 그 동안 GC는 그 스레드를 *이미 멈춘 것*으로 취급하고 자기 일을 한다.

| 상황 | 안전 지점 vs 안전 영역 |
|------|---------------------|
| 자바 코드 실행 중 | 안전 지점 폴링 |
| Native 메서드 호출 중 | 진입할 때 안전 영역 진입, 나올 때 GC 대기 |
| `Thread.sleep()` / `wait()` | 진입할 때 안전 영역 진입, 깨어날 때 GC 대기 |

## 4. 기억 집합과 카드 테이블 — *부분 GC*를 가능하게 하는 자료구조

> Minor GC가 *신세대만* 보려면, 구세대에서 신세대로 가는 참조를 *별도로 추적*해야 한다.

### 4.1 기억 집합 (Remembered Set)

논리적 개념. *영역 A의 객체 중 영역 B를 참조하는 객체들의 집합*. Minor GC가 신세대를 회수할 때 *구세대의 기억 집합*만 추가로 스캔하면 된다.

### 4.2 카드 테이블 (Card Table) — 기억 집합의 구현

기억 집합을 *객체 단위 리스트*로 들고 있으면 너무 비싸다. 핫스팟은 메모리를 *512바이트 카드*로 나누고, 카드마다 *1바이트 플래그*를 둔다.

```
구세대를 카드로 나눔:
[Card 0][Card 1][Card 2][Card 3]...   (각 512바이트)

카드 테이블:
[0][1][0][1]...   (1 = dirty, 0 = clean)
```

*카드 안의 어떤 객체*가 *어떤 신세대 객체를 참조하면* 그 카드 전체가 *dirty*로 표시된다. Minor GC는 dirty 카드 *전체*를 스캔한다 — 정확도는 떨어지지만(필요 없는 객체도 같이 스캔) *오버헤드가 훨씬 낮다*.

### 4.3 핫스팟의 카드 자료구조

핫스팟은 `BYTE CARD_TABLE[]` 라는 1차원 배열을 쓴다. 메모리 주소를 9비트(2^9 = 512) 오른쪽 시프트하면 카드 인덱스가 된다.

```
card_index = (object_address >> 9)
```

매 참조 대입 때마다 이 인덱스의 바이트를 *0이 아닌 값*(예: 0)으로 *덮어쓴다*. *조건 분기 없이 무조건 쓰기*가 핵심 — CPU 파이프라인이 깨지지 않아 빠르다.

## 5. 쓰기 장벽 (Write Barrier) — 카드 dirty 마크의 유지

> 카드 테이블이 정확하려면 *참조 대입이 일어날 때마다 카드를 dirty로* 표시해야 한다. 그 표시를 *AOP의 'after' 어드바이스*처럼 *대입 명령 후*에 자동으로 끼워 넣는 것이 쓰기 장벽이다.

```
// 의사 코드 — 핫스팟이 참조 대입 후 자동으로 넣는 쓰기 장벽
obj.field = value;            // 원래 대입
*(&CARD_TABLE[address_of(obj) >> 9]) = DIRTY;  // 쓰기 장벽
```

이 코드는 *JIT가 자동으로 넣어 준다*. 자바 소스에서는 보이지 않지만, *모든 참조 대입*에 한 줄씩 더 붙는다. 비용은 *대입 한 번에 메모리 쓰기 하나*다.

### 5.1 위장 공유 (False Sharing) 문제

카드 테이블이 *너무 자주* 쓰여지면, *여러 CPU 코어가 같은 캐시 라인을 두고 다투는* false sharing이 발생한다. 한 코어가 카드 32번을 dirty로 표시할 때, 다른 코어가 카드 33번을 표시하면 *같은 캐시 라인*을 두고 *코히어런시 트래픽*이 일어난다.

JDK 7부터 `-XX:+UseCondCardMark` 옵션이 추가됐다 — *카드가 이미 dirty면 다시 쓰지 않는다*는 조건 분기를 더한다. 분기 비용이 추가되지만 false sharing이 줄어든다. 트레이드오프다.

## 6. 동시 마킹 — *애플리케이션이 도는 중에* 마크가 가능한가

CMS·G1·ZGC·Shenandoah 같은 동시 GC는 *애플리케이션 스레드와 동시에* 마크 단계를 돈다. 그러나 *애플리케이션이 객체 그래프를 바꾸는 중*에 마크하면 *살아 있는 객체를 놓치거나 죽은 객체를 살아 있다고 표시*하는 사고가 난다.

**3색 마킹**(Tri-color Marking)이 해결의 출발점이다.

| 색 | 의미 |
|----|------|
| 흰색 | 아직 보지 않음 (회수 후보) |
| 회색 | 본 적 있지만 *자식*은 아직 보지 않음 |
| 검은색 | 자기 자신과 *모든 자식*을 다 본 |

마크가 진행되는 동안 *검은 객체가 새로 흰 객체를 가리키게 되면* 그 흰 객체가 누락될 수 있다. 두 가지 방어:

- **증분 갱신 (Incremental Update)** — *검은 객체가 흰 객체를 가리키게 되면* 그 검은 객체를 다시 회색으로 표시. CMS의 방식.
- **시작 시 스냅샷 (Snapshot at the Beginning, SATB)** — *마크 시작 시점의 객체 그래프*를 기준으로 회수한다. 마크 도중 *지워진 참조*가 있어도 *그 객체는 살아 있다고 본다*. G1, ZGC, Shenandoah의 방식.

두 방어 모두 *쓰기 장벽*을 통해 구현된다. 5절의 카드 테이블 쓰기 장벽이 *세대 간 참조*를 잡았다면, 동시 마킹의 쓰기 장벽은 *동시 마크의 정확성*을 잡는다.

## 7. 한 줄로 정리

§3.4를 묶으면 *핫스팟이 정확한 GC와 짧은 일시 정지를 동시에 달성하기 위한 다섯 부품의 합*이다.

1. **OopMap** — 어디에 객체 참조가 있는지 미리 알기 (정확성)
2. **안전 지점** — 어디서 멈춰야 OopMap이 정확한지 정의 (정확성 + 빠른 동기화)
3. **안전 영역** — 멈춰 있는 스레드도 GC에 협력하게 만들기 (완전성)
4. **카드 테이블** — Minor GC가 구세대를 전수 조사 안 해도 되게 (속도)
5. **쓰기 장벽** — 위 네 부품을 자동으로 유지 (자동화)

다음 노트(01-04)는 이 부품을 조합한 *클래식 가비지 컬렉터들*(Serial, ParNew, Parallel, CMS, G1)을 다룬다.
