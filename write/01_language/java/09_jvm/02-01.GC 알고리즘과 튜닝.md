# GC 알고리즘과 튜닝
---
> 가비지 컬렉션의 기초 알고리즘부터 현대 저지연 컬렉터까지의 발전 흐름을 이해하고, 실무에서 GC를 튜닝하는 방법을 익힌다. Mark-Sweep부터 ZGC까지 각 알고리즘의 트레이드오프, GC 로그 분석, 주요 JVM 옵션을 다룬다.

## 1. GC 기본 알고리즘

GC는 힙 메모리에서 더 이상 참조되지 않는 객체를 회수하는 자동 메모리 관리 시스템이다. 기초 알고리즘 세 가지가 현대 GC 구현의 바탕이 된다.

### 1-1. Mark-Sweep (표시-제거)

*GC 루트*(스택 프레임의 지역 변수, 정적 변수 등)에서 시작하여 참조 그래프를 탐색하며 도달 가능한 객체를 **표시(Mark)**하고, 표시되지 않은 객체를 **제거(Sweep)**한다. 구현이 단순하지만 두 가지 문제가 있다. 모든 사용자 스레드를 멈추는 **STW**(Stop-The-World)가 발생하며, 제거 후 메모리가 단편화되어 연속 메모리 공간을 요구하는 큰 객체 할당에 실패할 수 있다.

### 1-2. Mark-Compact (표시-정리)

Mark-Sweep의 단편화 문제를 해결하기 위해, 표시 후 살아있는 객체를 메모리 한쪽 끝으로 **압축(Compact)**한다. 압축 덕분에 단편화 없이 연속 공간을 확보하지만, 객체 이동 시 참조를 모두 업데이트해야 하므로 STW가 더 길어진다.

### 1-3. Copying (복사)

힙을 두 영역으로 나누고 한쪽만 사용한다. GC 시 살아있는 객체를 다른 쪽으로 **복사**하면 이동과 동시에 단편화가 해소된다. 복사 속도가 빠르지만, 전체 힙의 절반만 사용할 수 있어 메모리 효율이 낮다.

## 2. 세대별 GC와 힙 구조

현대 GC는 **세대별 가설**(대부분의 객체는 금방 죽는다)을 기반으로 힙을 세대로 나누어 관리한다. 세대별 구분의 목적은 GC 성능 향상이며, 짧게 사는 객체는 Young Generation에서 빠르게 회수하고 오래 사는 객체만 Old Generation으로 승격시킨다.

```
힙 메모리 구조 (G1 이전)
┌──────────────────────────────────────────────────────────┐
│                    Young Generation                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐               │
│  │  Eden    │  │ Survivor │  │ Survivor │               │
│  │  (80%)   │  │   S0     │  │   S1     │               │
│  └──────────┘  └──────────┘  └──────────┘               │
├──────────────────────────────────────────────────────────┤
│                     Old Generation                        │
│  (오래 살아남은 객체, 큰 객체)                             │
└──────────────────────────────────────────────────────────┘
```

**Minor GC**: Eden이 가득 차면 발생한다. Eden과 현재 사용 중인 Survivor(From)의 살아있는 객체를 다른 Survivor(To)로 복사한다. 여러 번의 Minor GC에서 살아남아 일정 나이(기본 15)가 된 객체는 Old Generation으로 **승격(Promotion)**된다.

**Major/Full GC**: Old Generation이 가득 차거나 Minor GC 중 Survivor 공간이 부족할 때 발생한다. Young + Old Generation 전체를 GC하며, STW가 가장 길다.

## 3. GC 구현체 비교

| GC | 목표 | 기본 알고리즘 | STW 특성 | 기본 채택 버전 |
|---|---|---|---|---|
| **Serial GC** | 단순, 단일 스레드 | Mark-Compact | 전체 STW | - |
| **Parallel GC** | 처리량 최대화 | 병렬 Mark-Compact | Young/Old 모두 STW | Java 8 기본 |
| **CMS** | 낮은 지연 | 동시 Mark-Sweep | Old 단계별 일부 동시 | Deprecated |
| **G1 GC** | 예측 가능한 지연 | Region 기반 | 짧고 예측 가능한 STW | Java 9+ 기본 |
| **ZGC** | 서브밀리초 지연 | 컬러드 포인터 + 로드 배리어 | 거의 동시 (< 1ms) | Java 21+ 기본 |
| **Shenandoah** | 낮은 지연 | Brooks 간접 포인터 | 거의 동시 | OpenJDK 12+ |

## 4. G1GC 동작 원리

**G1**(Garbage-First) GC는 Java 9부터 기본 컬렉터로, 예측 가능한 STW 시간을 목표로 설계되었다.

### 4-1. Region 기반 구조

G1은 힙을 동일한 크기의 **Region** 수백~수천 개로 나눈다. 각 Region은 동적으로 Eden, Survivor, Old, Humongous(대형 객체용) 역할을 맡는다. 이 구조 덕분에 고정된 Young/Old 경계 없이 유연한 메모리 관리가 가능하다.

```
G1 힙 구조
┌──────┬──────┬──────┬──────┬──────┬──────┐
│ Eden │ Old  │ Eden │ Surv │ Old  │ Hum  │
├──────┼──────┼──────┼──────┼──────┼──────┤
│ Old  │ Free │ Surv │ Free │ Eden │ Old  │
└──────┴──────┴──────┴──────┴──────┴──────┘
각 Region은 동적으로 역할이 부여됨
```

### 4-2. G1 GC 사이클

G1은 세 단계로 동작한다. **Young GC**는 Eden이 가득 찰 때 STW로 수행되며, 살아있는 객체를 Survivor 또는 Old Region으로 이동한다. **Concurrent Marking**은 애플리케이션과 동시에 실행되어 Old Region의 살아있는 객체를 표시한다. **Mixed GC**는 Young Region과 가비지 비율이 높은 Old Region을 함께 회수한다. Garbage-First라는 이름은 가비지가 많은 Region을 우선 수거한다는 전략에서 유래한다.

G1은 `-XX:MaxGCPauseMillis`로 목표 일시 정지 시간을 설정하면, G1이 이 목표에 맞춰 수거할 Region 수를 자동으로 조정한다.

## 5. ZGC: Java 21 기본 컬렉터

ZGC(Z Garbage Collector)는 Java 11에서 실험적으로 도입되어 Java 21에서 기본 컬렉터로 채택되었다. 힙 크기와 무관하게 **서브밀리초(< 1ms) 일시 정지**를 목표로 설계되었다.

### 5-1. 컬러드 포인터(Colored Pointer)

ZGC의 핵심 혁신은 객체 참조 포인터의 상위 비트를 GC 상태 메타데이터로 활용하는 **컬러드 포인터** 기법이다. 64비트 주소 공간에서 일부 비트를 `Marked0`, `Marked1`, `Remapped`, `Finalizable` 플래그로 사용하여, 별도 자료 구조 없이 객체 상태를 포인터 자체에 인코딩한다.

```
ZGC 포인터 구조 (64비트)
Bit 63-44: Unused
Bit 43   : Finalizable
Bit 42   : Remapped
Bit 41   : Marked1
Bit 40   : Marked0
Bit 39-0 : 실제 객체 주소 (1TB 주소 공간)
```

### 5-2. 로드 배리어(Load Barrier)

ZGC는 객체 참조를 **읽을 때마다** JIT 컴파일러가 삽입한 소량의 코드(로드 배리어)를 실행한다. 로드 배리어는 포인터의 색상을 확인하고, 필요하면 GC 작업(리매핑, 마킹)을 즉시 수행한다. 이 설계 덕분에 GC 작업의 대부분을 애플리케이션 스레드와 동시에 수행할 수 있어 STW를 최소화한다.

```java
// ZGC가 활성화된 경우, 참조 읽기마다 로드 배리어 실행
// Object obj = reference.field; 가 아래처럼 변환됨
// Object obj = loadBarrier(reference.field);
// loadBarrier는 포인터 색상 확인 후 필요시 리매핑 수행
```

## 6. GC 로그 분석

GC 문제를 진단하려면 GC 로그를 분석해야 한다. Java 9+에서는 `-Xlog:gc*` 옵션으로 통합된 로깅을 사용한다.

```bash
# GC 로그 활성화
java -Xlog:gc*:file=gc.log:time,uptime:filecount=10,filesize=10m -jar myapp.jar

# 주요 로그 항목 예시
[2.135s][info][gc] GC(0) Pause Young (Normal) (G1 Evacuation Pause) 128M->64M(512M) 12.345ms
# 의미: 2.135초 시점, Young GC, 128MB → 64MB (전체 힙 512MB), 12.345ms STW
```

GC 로그에서 주목해야 할 지표는 다음과 같다.

- **일시 정지 시간**: STW 지속 시간 (목표치 초과 여부)
- **GC 빈도**: Minor GC와 Full GC의 발생 간격
- **힙 사용량 변화**: GC 전후 힙 사용량으로 회수 효율 파악
- **Promotion Failure**: Old Generation 공간 부족으로 Survivor 객체 이동 실패

```bash
# jstat으로 실시간 GC 통계 확인
jstat -gcutil <pid> 1000 10  # 1초 간격으로 10회 출력

# 출력 예시
S0     S1     E      O      M      CCS    YGC    YGCT    FGC    FGCT    CGC    CGCT    GCT
 0.00  45.23  72.13  68.92  95.12  92.34     15    0.312     1    0.521     3    0.043    0.876
```

## 7. GC 튜닝 파라미터

GC 튜닝의 기본 원칙은 측정 → 분석 → 조정의 순서를 지키는 것이다. 먼저 GC 로그와 프로파일링으로 문제를 정확히 파악한 뒤 파라미터를 조정한다.

### 7-1. 힙 크기 설정

```bash
# 기본 힙 크기 설정
java -Xms2g -Xmx2g -jar myapp.jar
# -Xms: 초기 힙 크기 (Xmx와 동일하게 설정하면 동적 조정 오버헤드 제거)
# -Xmx: 최대 힙 크기 (물리 메모리의 50~75% 권장)
```

`-Xms`와 `-Xmx`를 동일하게 설정하면 힙 크기 조정에 따른 GC 오버헤드를 줄일 수 있다. 컨테이너 환경에서는 `-XX:MaxRAMPercentage`로 JVM이 컨테이너 메모리 제한을 인식하게 하는 것이 중요하다.

### 7-2. GC 선택과 주요 옵션

```bash
# G1GC 사용 (Java 9+ 기본)
java -XX:+UseG1GC \
     -Xms2g -Xmx2g \
     -XX:MaxGCPauseMillis=200 \      # 목표 최대 STW 시간 (ms)
     -XX:G1HeapRegionSize=8m \       # Region 크기 (1~32MB, 2의 거듭제곱)
     -XX:InitiatingHeapOccupancyPercent=45 \ # Concurrent Marking 시작 임계값
     -jar myapp.jar

# ZGC 사용 (저지연 요구 시)
java -XX:+UseZGC \
     -Xms4g -Xmx4g \
     -XX:MaxGCPauseMillis=10 \       # ZGC는 서브밀리초 목표 가능
     -jar myapp.jar

# GC 로그 포함
java -XX:+UseG1GC \
     -Xlog:gc*:file=gc.log:time,uptime \
     -jar myapp.jar
```

### 7-3. 튜닝 체크리스트

- **Full GC 빈도가 높다**: Old Generation 크기 부족 → `-Xmx` 증가 또는 메모리 누수 확인
- **STW가 길다**: `-XX:MaxGCPauseMillis` 목표 낮추기, ZGC/Shenandoah 고려
- **Minor GC가 너무 잦다**: Young Generation이 너무 작음 → `-Xmn` 또는 `-XX:NewRatio` 조정
- **Promotion Failure**: Old Generation 공간 부족, 객체 생존율이 너무 높음 → 힙 크기 조정 또는 코드 레벨 객체 생명주기 검토
