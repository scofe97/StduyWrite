# ThreadLocal과 동시성 라이브러리

---

> 동시성 문제는 스레드 간 공유 상태에서 비롯된다. ThreadLocal은 상태를 스레드별로 격리하고, 동기화 도구들은 스레드 간 협력 시점을 정밀하게 제어한다. 본 노트를 한 줄로 압축하면 — **동시성 버그를 줄이는 두 갈래 길은 *공유를 없애기*(`ThreadLocal`·불변 객체)와 *공유의 시점을 좁히기*(`CountDownLatch`·`Semaphore`·`Phaser`)이며, JDK 21 `ScopedValue`가 그 첫 갈래의 더 안전한 대안**으로 등장했다.

## 동시성 문제의 근원

스프링은 싱글톤으로 빈을 관리하므로, 여러 요청 스레드가 동일한 객체를 공유한다. 객체의 필드에 상태를 저장하면 스레드 A가 저장한 값을 스레드 B가 덮어쓰는 경합 조건이 발생한다. 아래 예시처럼 스레드 A가 저장 후 조회하기 전에 스레드 B가 같은 필드를 덮어쓰면, 스레드 A는 자신이 아닌 스레드 B의 값을 읽게 된다.

```java
// 싱글톤 서비스 필드 - 동시성 문제 발생
public class FieldService {
    private String nameStore; // 여러 스레드가 공유

    public String logic(String name) {
        nameStore = name;        // 스레드 A 저장 후
        Thread.sleep(1000);      // 대기 중 스레드 B가 덮어씀
        return nameStore;        // 스레드 B의 값이 반환됨
    }
}
```

읽기만 하는 필드에서는 문제가 없다. 쓰기가 발생하는 가변 필드가 동시성 문제의 원인이다.

## ThreadLocal 동작 원리

**`ThreadLocal<T>`**은 각 스레드에게 독립된 저장 공간을 제공한다. `Thread` 객체 내부에는 `ThreadLocal.ThreadLocalMap` 타입의 필드가 있다. `ThreadLocal.set(value)`을 호출하면 현재 스레드의 맵에 `(ThreadLocal 인스턴스 → 값)` 쌍이 저장된다. `get()`은 현재 스레드의 맵에서만 값을 조회하므로 다른 스레드의 값과 절대 섞이지 않는다.

```java
public class ThreadLocalService {
    private ThreadLocal<String> nameStore = new ThreadLocal<>();

    public String logic(String name) {
        nameStore.set(name);     // 현재 스레드 전용 공간에 저장
        Thread.sleep(1000);
        return nameStore.get();  // 현재 스레드 값만 반환
    }
}
```

API는 세 가지다. `set(value)`로 저장하고, `get()`으로 조회하며, `remove()`로 삭제한다.

## 사용 사례

`ThreadLocal`이 실무에서 가장 많이 쓰이는 세 곳이다.

- **SecurityContext**: Spring Security는 인증 정보를 `SecurityContextHolder`에 보관한다. 내부적으로 `ThreadLocal`을 사용해 요청 스레드마다 독립된 인증 컨텍스트를 제공한다.
- **TransactionContext**: Spring의 트랜잭션 동기화 매니저는 `ThreadLocal`에 현재 트랜잭션의 커넥션을 저장한다. 같은 스레드 안에서 `@Transactional` 메서드들이 동일한 커넥션을 공유할 수 있는 이유가 여기에 있다.
- **MDC(Mapped Diagnostic Context)**: Logback, Log4j2의 MDC는 요청 ID, 사용자 ID 같은 컨텍스트 정보를 `ThreadLocal`에 저장해 로그 메시지에 자동으로 포함시킨다.

## 메모리 누수 위험

**스레드 풀 환경에서 `ThreadLocal`을 사용한 후 `remove()`를 호출하지 않으면 메모리 누수가 발생한다.** 스레드 풀의 스레드는 재사용되므로, 이전 요청이 저장한 값이 다음 요청에 그대로 남아 있게 된다. 이 문제는 두 가지 결과를 초래한다. 첫째, 이전 요청의 개인 정보나 트랜잭션 컨텍스트가 다른 사용자에게 노출될 수 있다. 둘째, `ThreadLocalMap`의 엔트리가 쌓여 메모리를 계속 점유한다.

```java
// 올바른 사용 패턴 - 반드시 finally에서 remove()
public String logic(String name) {
    try {
        nameStore.set(name);
        return process();
    } finally {
        nameStore.remove(); // 스레드 풀 반납 전 반드시 정리
    }
}
```

## InheritableThreadLocal

`InheritableThreadLocal<T>`은 부모 스레드의 값을 자식 스레드가 상속받는다. 부모 스레드에서 설정한 값이 `new Thread()` 생성 시점에 자식 스레드로 복사된다. 요청 ID나 사용자 컨텍스트를 하위 작업 스레드에 전달할 때 유용하다.

단, 스레드 풀에서 생성된 스레드는 풀 초기화 시점에 생성되므로 이후 부모가 변경한 값을 반영하지 못한다. `InheritableThreadLocal`도 `remove()`로 정리해야 한다는 원칙은 동일하게 적용된다.

## Java 21 ScopedValue

**`ScopedValue`**는 Java 21에서 정식 출시된 `ThreadLocal` 대안이다. 가상 스레드 환경에서 `ThreadLocal`을 사용하면 수십만 개의 가상 스레드 각각이 `ThreadLocalMap`을 보유해 메모리 압박이 커진다. `ScopedValue`는 이 문제를 구조적으로 해결한다.

`ScopedValue`는 불변(immutable)이며, `ScopedValue.where(key, value).run(action)` 블록 안에서만 값이 유효하다. 블록을 벗어나면 자동으로 소멸하므로 `remove()` 호출이 필요 없다.

```java
// Java 21+
private static final ScopedValue<String> USER_ID = ScopedValue.newInstance();

public void handleRequest(String userId) {
    ScopedValue.where(USER_ID, userId).run(() -> {
        processOrder();  // 이 블록 안에서만 USER_ID 접근 가능
    });
    // 블록 종료 후 자동 소멸
}

private void processOrder() {
    String id = USER_ID.get(); // 현재 스코프의 값 조회
}
```

가상 스레드 기반의 신규 코드에서는 `ThreadLocal`보다 `ScopedValue`를 우선 고려하는 것이 좋다.

## CountDownLatch

**`CountDownLatch`**는 하나 이상의 스레드가 다른 스레드들의 작업 완료를 기다릴 때 사용한다. 초기 카운트를 지정하고, 각 스레드가 작업을 마치면 `countDown()`으로 카운트를 줄인다. 카운트가 0이 되면 `await()`에서 대기 중인 스레드가 깨어난다.

```java
CountDownLatch latch = new CountDownLatch(3); // 3개 작업 대기

ExecutorService es = Executors.newFixedThreadPool(3);
for (int i = 0; i < 3; i++) {
    es.submit(() -> {
        try {
            loadCache();         // 캐시 로드
        } finally {
            latch.countDown();   // 완료 신호
        }
    });
}

latch.await(); // 3개 모두 완료될 때까지 대기
startServer(); // 이후 서버 시작
```

카운트가 0이 되면 재사용할 수 없다는 점이 `CyclicBarrier`와의 차이다. 서버 초기화, 병렬 테스트의 준비 완료 신호, 외부 서비스 응답 취합 등에 적합하다.

## CyclicBarrier

**`CyclicBarrier`**는 정해진 수의 스레드가 모두 특정 지점에 도달해야 함께 다음 단계로 진행하는 도구다. `await()`를 호출한 스레드는 지정된 수만큼 모두 `await()`에 도달할 때까지 대기한다. 모든 스레드가 도달하면 동시에 해제된다.

```java
CyclicBarrier barrier = new CyclicBarrier(3, () ->
    System.out.println("모든 스레드 준비 완료, 다음 단계 시작")
);

Runnable worker = () -> {
    prepareData();
    barrier.await(); // 다른 스레드 대기
    processData();   // 모두 준비 후 동시 시작
};
```

`CountDownLatch`와 달리 `reset()`으로 재사용할 수 있어 *Cyclic*이라는 이름이 붙었다. 시뮬레이션의 라운드별 동기화, 분산 정렬 알고리즘의 단계 동기화 등에 쓰인다.

## Semaphore

**`Semaphore`**는 동시에 접근 가능한 스레드 수를 제한하는 도구다. 내부 카운터(허가, permit)를 관리하며 `acquire()`로 허가를 획득하고 `release()`로 반납한다. 카운터가 0이면 `acquire()`를 호출한 스레드는 허가가 반납될 때까지 대기한다.

```java
Semaphore semaphore = new Semaphore(3); // 동시 접근 최대 3개

public void accessDatabase() throws InterruptedException {
    semaphore.acquire();
    try {
        query(); // 최대 3개 스레드만 동시 실행
    } finally {
        semaphore.release(); // 반드시 finally에서 반납
    }
}
```

DB 커넥션 풀 크기 제한, API 호출 속도 제한(rate limiting), 파일 시스템 동시 접근 제어 등에 유용하다. `acquire(int permits)`로 여러 허가를 한 번에 획득할 수도 있다.

## Phaser

**`Phaser`**는 `CountDownLatch`와 `CyclicBarrier`를 결합한 유연한 동기화 도구다. 동적으로 참여자를 등록하거나 해제할 수 있고, 단계(phase)가 자동으로 증가해 여러 단계의 작업을 반복적으로 동기화할 수 있다.

```java
Phaser phaser = new Phaser(1); // 메인 스레드 등록

for (int i = 0; i < 3; i++) {
    phaser.register();           // 동적으로 참여자 추가
    final int taskId = i;
    new Thread(() -> {
        phaseOne(taskId);
        phaser.arriveAndAwaitAdvance(); // 1단계 완료 대기
        phaseTwo(taskId);
        phaser.arriveAndDeregister();   // 참여 해제
    }).start();
}

phaser.arriveAndAwaitAdvance(); // 메인 스레드도 1단계 대기
System.out.println("모든 스레드 1단계 완료");
```

참여자 수가 런타임에 결정되거나 단계별로 달라지는 파이프라인 처리에 적합하다. 복잡하지만 `CountDownLatch`나 `CyclicBarrier`로 표현하기 어려운 시나리오를 깔끔하게 모델링할 수 있다.

## 도구 선택 기준 요약

| 상황 | 적합한 도구 |
|------|------------|
| 스레드별 독립 상태 저장 | `ThreadLocal` |
| 가상 스레드 환경의 컨텍스트 전파 | `ScopedValue` (Java 21+) |
| N개 작업 완료 후 진행 (일회성) | `CountDownLatch` |
| N개 스레드 동시 출발 (재사용) | `CyclicBarrier` |
| 동시 접근 수 제한 | `Semaphore` |
| 동적 참여자, 다단계 동기화 | `Phaser` |
