# Executor 프레임워크

---

> 스레드를 직접 생성하는 코드는 실무에서 거의 쓰지 않는다. Executor 프레임워크가 스레드 생성, 재사용, 종료를 모두 관리해주기 때문이다. 본 노트를 한 줄로 압축하면 — **Executor의 본질은 *작업(Runnable/Callable)과 실행 자원(스레드 풀)의 분리*이며, `Future`·`CompletableFuture`가 그 사이에 *비동기 결과 흐름*을 표현한다**. JDK 21의 가상 스레드는 *풀이라는 추상 자체*를 다시 본다.

## 스레드 직접 사용의 세 가지 문제

`new Thread(task).start()`처럼 스레드를 직접 생성하면 세 가지 문제가 따른다.

- **성능**: 스레드 생성은 커널 수준 작업이라 CPU와 메모리를 소비한다. 스레드마다 독립된 호출 스택이 필요하고, 운영체제가 스케줄링 대상에 추가해야 한다.
- **관리**: CPU와 메모리는 한정되어 있으므로 스레드를 무한히 생성할 수 없다. 트래픽이 폭증했을 때 스레드 수를 제한하지 않으면 서버가 다운된다.
- **불편함**: `Runnable.run()`은 반환값이 없고 체크 예외를 던질 수 없어, 결과를 받으려면 별도의 공유 변수를 써야 한다.

**Executor 프레임워크**는 이 세 문제를 모두 해결한다. `java.util.concurrent` 패키지에 포함되어 있으며, 스레드 풀을 통해 스레드를 재사용하고 작업 큐로 과부하를 제어하며 `Future`로 결과를 반환한다.

## Executor / ExecutorService 인터페이스 계층

인터페이스 계층은 세 단계로 이루어진다.

- **`Executor`**: 가장 단순한 인터페이스로 `execute(Runnable)` 메서드 하나만 정의한다. 작업을 어떤 스레드에서 실행할지 분리하는 것이 목적이다.
- **`ExecutorService`**: `Executor`를 확장해 `submit()`, `invokeAll()`, `invokeAny()`, 생명주기 메서드를 추가한다. 실무에서 주로 사용하는 인터페이스다.
- **`ScheduledExecutorService`**: `ExecutorService`를 확장해 지연 실행(`schedule()`)과 주기적 실행(`scheduleAtFixedRate()`)을 지원한다.

기본 구현체는 `ThreadPoolExecutor`이며, `Executors` 팩토리 클래스가 다양한 설정의 인스턴스를 편리하게 생성해준다.

## 스레드 풀 종류 비교

| 종류 | 코어 스레드 | 최대 스레드 | 큐 | 주요 용도 |
|------|------------|------------|-----|----------|
| `newFixedThreadPool(n)` | n | n | 무제한 LinkedBlockingQueue | CPU 집약 작업, 예측 가능한 부하 |
| `newCachedThreadPool()` | 0 | Integer.MAX_VALUE | SynchronousQueue | 짧고 많은 비동기 작업 |
| `newScheduledThreadPool(n)` | n | Integer.MAX_VALUE | DelayedWorkQueue | 주기적·지연 실행 |
| `newSingleThreadExecutor()` | 1 | 1 | 무제한 LinkedBlockingQueue | 순서 보장이 필요한 직렬 처리 |
| `newWorkStealingPool()` | - | 가용 CPU 수 | - | 재귀적·포크-조인 작업 |
| `newVirtualThreadPerTaskExecutor()` | - | 무제한 가상 스레드 | - | I/O 집약 작업 (Java 21+) |

`newCachedThreadPool()`은 내부적으로 `SynchronousQueue`를 사용한다. 이 큐는 저장 공간이 없고 생산자와 소비자가 직접 핸드오프(handoff)하므로, 유휴 스레드가 없으면 즉시 새 스레드를 생성한다. 60초간 작업이 없으면 초과 스레드를 회수한다.

## submit() vs execute()

`execute(Runnable)`은 `Executor` 인터페이스에 정의된 메서드로 반환값이 없다. 작업 중 예외가 발생하면 해당 스레드가 종료되며 예외가 외부로 전파되지 않는다.

`submit(Callable<T>)`은 `ExecutorService`가 제공하며 `Future<T>`를 즉시 반환한다. 예외는 `Future.get()` 호출 시 `ExecutionException`으로 감싸져 전달된다. 결과가 필요하거나 예외를 호출자에서 처리해야 할 때는 반드시 `submit()`을 사용해야 한다.

```java
ExecutorService es = Executors.newFixedThreadPool(2);

// execute: 반환값 없음, 예외 전파 안 됨
es.execute(() -> System.out.println("작업 실행"));

// submit: Future 반환, 예외는 get()에서 처리
Future<Integer> future = es.submit(() -> {
    Thread.sleep(1000);
    return 42;
});

Integer result = future.get(); // 완료까지 블로킹
```

`Future`를 `submit()` 직후에 즉시 `get()`으로 블로킹하면 단일 스레드와 다를 바 없다. 여러 작업을 먼저 모두 `submit()`하고 나서 `get()`을 호출해야 병렬 실행의 효과를 얻는다.

## Future와 Callable

**`Callable<V>`**은 `Runnable`의 한계를 보완한 인터페이스다. `call()` 메서드는 결과를 반환하고 체크 예외를 던질 수 있다.

**`Future<V>`**는 아직 완료되지 않은 비동기 작업의 결과를 나타내는 핸들이다. `submit()` 호출 즉시 반환되므로 호출 스레드가 블로킹되지 않는다. 결과가 필요한 시점에 `get()`을 호출하면 작업이 완료될 때까지 대기한다.

```java
ExecutorService es = Executors.newFixedThreadPool(2);

Future<Integer> f1 = es.submit(() -> computeSum(1, 50));   // 즉시 반환
Future<Integer> f2 = es.submit(() -> computeSum(51, 100)); // 즉시 반환

// f1, f2 동시에 실행 중
Integer sum1 = f1.get(); // 완료까지 대기
Integer sum2 = f2.get(); // 이미 완료되어 있으면 즉시 반환
System.out.println(sum1 + sum2); // 5050
```

`Future`의 주요 메서드는 다음과 같다.

- `get()`: 완료 대기 후 결과 반환. `ExecutionException`, `InterruptedException` 선언 필요
- `get(long timeout, TimeUnit unit)`: 타임아웃 지정. `TimeoutException` 추가
- `cancel(boolean mayInterruptIfRunning)`: 작업 취소. `true`면 실행 중인 작업을 인터럽트
- `isDone()`: 완료(정상/예외/취소) 여부 확인
- `state()`: Java 19+, `RUNNING`/`SUCCESS`/`FAILED`/`CANCELLED` 반환

## 우아한 종료 (Graceful Shutdown)

서버를 재시작할 때 처리 중인 요청을 갑자기 끊으면 데이터 불일치가 발생할 수 있다. 이미 시작된 작업은 마저 완료하고 새 작업만 받지 않는 방식이 **graceful shutdown**이다.

```java
ExecutorService es = Executors.newFixedThreadPool(4);

// 새 작업 수신 중단, 기존 작업은 완료
es.shutdown();

// 최대 10초 대기
if (!es.awaitTermination(10, TimeUnit.SECONDS)) {
    // 타임아웃: 강제 종료
    es.shutdownNow(); // 실행 중인 스레드에 인터럽트 발생
}
```

`shutdown()`은 논블로킹이므로 `awaitTermination()`을 함께 사용해야 실제 완료를 기다릴 수 있다. `close()`(Java 19+)는 `shutdown()` 후 무기한 대기하다가 인터럽트를 받으면 `shutdownNow()`를 호출하는 편의 메서드다. try-with-resources와 함께 사용할 수 있다.

## ThreadPoolExecutor 커스텀 설정

`Executors` 팩토리 메서드가 제공하지 않는 세밀한 설정이 필요하면 `ThreadPoolExecutor`를 직접 생성한다.

```java
ThreadPoolExecutor es = new ThreadPoolExecutor(
        100                          // corePoolSize: 기본 스레드 수
        , 200                        // maximumPoolSize: 최대 스레드 수
        , 60, TimeUnit.SECONDS       // keepAliveTime: 초과 스레드 유지 시간
        , new ArrayBlockingQueue<>(1000) // 작업 대기 큐
);
```

스레드 풀 작업 처리 흐름은 네 단계다.

- 작업 요청 시 코어 스레드 수까지 스레드를 생성한다.
- 코어 스레드가 모두 사용 중이면 큐에 작업을 쌓는다.
- 큐도 가득 차면 최대 스레드 수까지 임시(초과) 스레드를 생성한다.
- 최대 스레드도 모두 사용 중이면 `RejectedExecutionException`을 던진다.

거절 정책은 `setRejectedExecutionHandler()`로 교체할 수 있다. `AbortPolicy`(기본, 예외 발생), `DiscardPolicy`(무시), `CallerRunsPolicy`(호출 스레드가 직접 실행) 중 상황에 맞게 선택한다.

## Java 21: 가상 스레드

Java 21에서 정식 출시된 **가상 스레드(Virtual Thread)**는 플랫폼 스레드와 달리 JVM이 관리하는 경량 스레드다. I/O 대기 중에는 캐리어 스레드(OS 스레드)를 반납하고, I/O가 완료되면 다시 할당받는다. 수십만 개를 동시에 실행해도 OS 자원을 과도하게 소비하지 않는다.

```java
// 작업마다 가상 스레드 생성 (Java 21+)
ExecutorService es = Executors.newVirtualThreadPerTaskExecutor();

Future<String> future = es.submit(() -> {
    // I/O 블로킹 → 캐리어 스레드 반납 → 다른 가상 스레드 실행
    return fetchFromDatabase();
});
```

가상 스레드는 CPU 집약 작업보다 I/O 집약 작업에 적합하다. 기존 `ThreadPoolExecutor` 기반 코드를 크게 바꾸지 않고 `newVirtualThreadPerTaskExecutor()`로 교체하는 것만으로 I/O 처리량을 크게 향상시킬 수 있다.
