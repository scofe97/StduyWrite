# Java Memory Model 심화
---
> JMM은 멀티스레드 환경에서 메모리 가시성과 실행 순서를 정의하는 규약이다. happens-before 관계를 이해하지 못하면 `volatile`이나 `synchronized`를 잘못 사용해도 버그가 재현되지 않는 위험한 코드를 만들게 된다. 본 노트를 한 줄로 압축하면 — **JMM의 본질은 *재정렬을 허용하면서도 정합성을 잃지 않게 하는 happens-before 부분 순서*이며, `volatile`·`synchronized`·`final`·CAS는 그 순서를 *지점적으로 강화하는* 도구**다. 안전한 게시(safe publication)는 그 순서를 *객체 생성 시점*에 한 번 보장하는 핵심 패턴이다.

## 1. JMM 정의와 목적

*Java Memory Model(JMM)*은 JVM 명세(JLS 17장)에 정의된, 스레드 간 메모리 접근 규칙의 집합이다. JMM이 필요한 이유는 현대 CPU와 컴파일러가 성능 최적화를 위해 명령어 순서를 자유롭게 재배치하기 때문이다. 단일 스레드에서는 재배치 결과가 동일하지만, 멀티스레드에서는 한 스레드의 쓰기 결과가 다른 스레드에 보이지 않는 **가시성(visibility)** 문제와 실행 순서가 뒤바뀌는 **순서 보장(ordering)** 문제가 발생한다.

JMM은 두 가지를 보장한다. 첫째, 어떤 쓰기 작업이 다른 스레드에 가시적이어야 하는지를 정의한다. 둘째, 어떤 연산들이 원자적(atomic)으로 처리되는지를 규정한다.

## 2. happens-before 관계

**happens-before**는 "A가 B보다 먼저 발생한다"는 인과 관계를 의미한다. A happens-before B가 성립하면, A의 결과는 B에서 반드시 가시적이다. 프로그램 순서·모니터 잠금(unlock→lock)·`volatile` 쓰기→읽기·`Thread.start()`·`Thread.join()`·전이성 등에서 성립한다.

규칙 목록 전체(인터럽트·객체 종료까지 여덟 규칙)와 "시간 순서가 아니라 가시성 보장"이라는 핵심 의미는 정독본 [`01-02.volatile·happens-before·원자성`](./01-02.volatile·happens-before·원자성.md)§5가 SSOT다. 이 노트는 그 규칙을 *전제로* 메모리 펜스·안전한 게시·DCL 같은 심화 주제를 다룬다.

## 3. 메모리 순서: 컴파일러/CPU 재배치

컴파일러와 CPU는 성능을 위해 독립적인 명령어 순서를 바꾸는 *재배치(reordering)*를 수행한다. 단일 스레드 입장에서는 결과가 동일하지만, 다른 스레드가 공유 변수를 읽을 때 문제가 생긴다 — "데이터를 준비하고 깃발을 올리는" 발행 패턴에서, 재배치로 깃발이 데이터를 추월하면 읽는 스레드가 준비 안 된 값을 본다.

> 재배치가 *왜* 일어나는지(대기 시간 숨기기), `config`/`initialized` 발행 패턴의 코드·버그 시퀀스·`volatile` 펜스가 막는 그림은 정독본 [`01-02`§4](./01-02.volatile·happens-before·원자성.md)가 SSOT다. 본 노트는 그 펜스가 *어떤 종류로* 삽입되는지(§4)부터 이어 받는다.

## 4. volatile 심화: 메모리 펜스

`volatile` 키워드는 가시성과 순서 보장을 제공하지만 원자성은 보장하지 않는다. JVM은 `volatile` 읽기/쓰기 시점에 *메모리 펜스(memory fence)* 명령어를 삽입해 재배치를 금지한다.

메모리 펜스의 종류는 다음과 같다:

- **LoadLoad**: 이전 Load가 완료된 후에 이후 Load가 실행된다.
- **StoreStore**: 이전 Store가 완료된 후에 이후 Store가 실행된다.
- **LoadStore**: 이전 Load가 완료된 후에 이후 Store가 실행된다.
- **StoreLoad**: 이전 Store가 완료된 후에 이후 Load가 실행된다. 네 가지 중 가장 비싼 연산이다.

`volatile` 쓰기 전에는 StoreStore 펜스가, 쓰기 후에는 StoreLoad 펜스가 삽입된다. `volatile` 읽기 후에는 LoadLoad와 LoadStore 펜스가 삽입된다.

```java
// volatile 쓰기 → StoreStore + StoreLoad 펜스
volatile boolean flag = false;

// 스레드 A
data = 42;           // StoreStore 펜스 이전에 반드시 완료
flag = true;         // volatile 쓰기
                     // StoreLoad 펜스

// 스레드 B
if (flag) {          // volatile 읽기
                     // LoadLoad + LoadStore 펜스
    use(data);       // 반드시 42를 읽는다
}
```

## 5. final 필드의 안전한 공개

*안전한 공개(safe publication)*란 한 스레드에서 생성한 객체를 다른 스레드에서 완전한 상태로 볼 수 있도록 보장하는 것이다. `final` 필드는 JMM에서 특별하게 취급된다.

생성자가 완료되면, 해당 객체의 모든 `final` 필드는 어떤 스레드에서도 올바른 값을 볼 수 있다. 이 보장은 생성자 내에서 `this` 참조가 외부로 누출되지 않는다는 전제를 요구한다.

```java
class ImmutablePoint {
    final int x;
    final int y;

    ImmutablePoint(int x, int y) {
        this.x = x;  // 생성자 완료 후 모든 스레드에서 안전하게 읽힌다
        this.y = y;
    }
}

// 잘못된 예: this 누출
class UnsafePoint {
    final int x;
    static UnsafePoint instance;

    UnsafePoint(int x) {
        instance = this; // ❌ 생성자가 완료되기 전에 this가 외부에 노출됨
        this.x = x;
    }
}
```

## 6. Double-Checked Locking 패턴과 volatile

*Double-Checked Locking(DCL)*은 싱글톤 패턴에서 성능을 위해 잠금을 최소화하는 기법이다. Java 5 이전에는 `volatile` 없이 DCL을 구현하면 재배치 문제로 부분적으로 초기화된 객체가 반환될 수 있었다.

Java 5 이후에는 `volatile`을 사용하면 DCL이 안전하게 동작한다. `volatile`이 `instance` 쓰기와 읽기에 happens-before를 보장하기 때문이다:

```java
class Singleton {
    private static volatile Singleton instance; // volatile 필수

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {              // 첫 번째 체크 (잠금 없음)
            synchronized (Singleton.class) {
                if (instance == null) {      // 두 번째 체크 (잠금 안에서)
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

`volatile` 없이 `instance = new Singleton()`을 실행하면 JVM은 이를 세 단계로 처리한다: 메모리 할당 → 초기화 → 참조 할당. 재배치가 발생하면 초기화 전에 참조가 할당될 수 있고, 다른 스레드가 비어있는 객체를 받게 된다.

## 7. 비교: volatile vs synchronized vs Atomic

세 가지 동기화 수단은 보장하는 속성과 성능이 다르므로 상황에 맞게 선택해야 한다:

| 특성 | volatile | synchronized | AtomicInteger 등 |
|------|----------|--------------|-----------------|
| 가시성 보장 | O | O | O |
| 원자성 보장 | X (64비트 long/double은 부분 보장) | O | O |
| 순서 보장 | O (happens-before) | O (happens-before) | O (CAS 기반) |
| 복합 연산 | X | O | O (incrementAndGet 등) |
| 블로킹 여부 | 비블로킹 | 블로킹 (락 경합 시) | 비블로킹 (스핀 루프) |
| 성능 | 빠름 | 느림 (락 획득/해제 오버헤드) | 빠름 (CAS) |
| 사용 적합 대상 | 단순 플래그, 상태 변수 | 복합 연산, 임계 구역 | 카운터, 누산기 |
