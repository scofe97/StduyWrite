# Virtual Threads 기초
---
> Virtual Thread는 JVM이 스케줄링하는 경량 스레드로, I/O 바운드 작업에서 Platform Thread 대비 수십 배 높은 처리량을 달성한다. OS 스레드와 1:1 매핑에서 벗어나 수백만 개의 동시 스레드를 현실적인 메모리 비용으로 운용할 수 있다. 본 노트를 한 줄로 압축하면 — **가상 스레드의 본질은 *OS 스레드와의 1:1 매핑을 끊고* JVM이 *Carrier Thread 위에서 마운트/언마운트를 직접 관리*하는 모델이며, "비동기 코드를 쓰지 않아도 비동기 처리량을 얻는다"가 그 결과**다.

## 1. 개념 요약 — 원리는 정독본에

가상 스레드는 OS 스레드와의 1:1 매핑을 끊고, JVM이 소수의 **Carrier Thread**(= Platform Thread) 위에서 **M:N**으로 마운트/언마운트를 직접 관리하는 경량 스레드다. 블로킹하면 Carrier에서 내려와(unmount) 힙에 상태를 저장하고, Carrier는 즉시 다른 가상 스레드를 실행한다.

> 왜 1:1 모델로는 안 되는가(C10K), Continuation/Scheduler 구현, mount/unmount가 *블로킹할 때만* 일어나는 까닭(협력적), 그 전환이 커널을 안 거쳐 싼 까닭, IO/CPU 바운드 이득 차이는 정독본 [`01-04.자바와 가상 스레드`](./01-04.자바와%20가상%20스레드%20—%20Virtual%20Threads.md)가 SSOT다. 본 노트는 그 원리를 *전제로* 생성·실행기·Spring 적용·벤치마크·Pinning 같은 **실무**를 다룬다.

| 특성 | Platform Thread | Virtual Thread |
|------|----------------|----------------|
| OS 매핑 | 1:1 | M:N (Carrier 경유) |
| 메모리 | ~1MB 스택 | ~1KB |
| 최대 개수 | 수천 | 수백만 |
| 적합 작업 | CPU 바운드 | I/O 바운드 |
| 스케줄링 | OS | Java 런타임 |

기본 스케줄러는 `ForkJoinPool`(워커 = CPU 코어 수)이며, work-stealing으로 부하를 분산한다. 실행 중인 Carrier 이름은 스레드 이름에 드러난다.

```java
Thread.startVirtualThread(() -> System.out.println(Thread.currentThread()));
// VirtualThread[#28]/runnable@ForkJoinPool-1-worker-1   ← @뒤가 Carrier 이름
```

## 2. 생성 방법 3가지

Virtual Thread를 생성하는 방법은 세 가지가 있다:

```java
// 방법 1: Thread.startVirtualThread() - 즉시 시작, 이름 지정 불가
Thread vt1 = Thread.startVirtualThread(() -> {
    System.out.println("즉시 시작되는 Virtual Thread");
});

// 방법 2: Thread.ofVirtual().start() - 빌더 패턴, 이름 지정 가능
Thread vt2 = Thread.ofVirtual()
    .name("my-virtual-thread")
    .start(() -> {
        System.out.println("이름이 있는 Virtual Thread");
    });

// 방법 3: Thread.ofVirtual().unstarted() - 수동 시작
Thread vt3 = Thread.ofVirtual()
    .name("delayed-thread")
    .unstarted(() -> {
        System.out.println("수동으로 시작되는 Virtual Thread");
    });
vt3.start(); // 원하는 시점에 시작
```

## 3. newVirtualThreadPerTaskExecutor()

`Executors.newVirtualThreadPerTaskExecutor()`는 태스크마다 새 Virtual Thread를 생성하는 `ExecutorService`를 반환한다. Platform Thread 풀과 달리 스레드를 재사용하지 않지만, Virtual Thread의 생성 비용이 극히 낮으므로 문제가 없다.

```java
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 10_000).forEach(i ->
        executor.submit(() -> {
            // 각 태스크마다 새 Virtual Thread 생성
            processRequest(i);
        })
    );
} // try-with-resources로 자동 shutdown 및 모든 태스크 완료 대기
```

`newFixedThreadPool(100)`과 비교하면, 10001번째 태스크가 대기 없이 즉시 새 Virtual Thread에서 실행된다는 점이 핵심 차이다.

## 4. 블로킹 I/O 처리와 자동 Unmount

Virtual Thread가 블로킹 연산을 만나면 JVM이 자동으로 Carrier Thread를 양보하고 힙에 상태를 저장한다. Carrier Thread는 즉시 다른 Virtual Thread를 실행한다. 블로킹이 해제되면 JVM 스케줄러가 Virtual Thread를 가용한 Carrier Thread에 재탑재한다.

```java
Thread.startVirtualThread(() -> {
    System.out.println("시작: " + Thread.currentThread());
    // 블로킹 → Carrier Thread 자동 양보
    Thread.sleep(1000);
    // 재개: 다른 Carrier Thread일 수 있음
    System.out.println("완료: " + Thread.currentThread());
});
```

## 5. 지원하는 블로킹 연산

JDK 21 기준으로 대부분의 표준 블로킹 연산이 Virtual Thread의 자동 unmount를 지원한다:

| 카테고리 | 메서드/연산 | 자동 Unmount | 비고 |
|----------|------------|:---:|------|
| 스레드 슬립 | `Thread.sleep()` | O | 가장 대표적 |
| 파일 I/O | `FileInputStream.read()`, `BufferedReader.readLine()` | O | JDK 21+ 대부분 지원 |
| 네트워크 I/O | `Socket.getInputStream().read()`, `HttpClient.send()` | O | JDK 21+ |
| 동기화 | `Object.wait()`, `Lock.lock()` 대기 | O | - |
| BlockingQueue | `queue.take()`, `queue.put()` | O | - |
| JDBC | `ResultSet.next()`, 쿼리 실행 | 조건부 O | JDBC 드라이버가 지원해야 함 |
| synchronized 블록 내 블로킹 | - | X | Pinning 발생 (§9 참조) |

## 6. I/O 바운드 벤치마크

I/O 바운드 작업에서 Virtual Thread의 성능 우위가 극명하게 드러난다. 1000개의 태스크가 각각 10ms 블로킹되는 상황에서 Platform Thread 풀(100개)은 10개씩 순차 처리하지만, Virtual Thread는 1000개를 동시에 처리한다.

실측 결과 (태스크 1000개, 블로킹 10ms):

- Platform Thread Pool (100개): 약 10,328ms
- Virtual Thread: 약 127ms (약 81배 빠름)

CPU 바운드 작업에서는 Virtual Thread의 이점이 없다. Carrier Thread 수가 CPU 코어 수로 제한되어 있으므로, 컨텍스트 스위칭 오버헤드만 추가될 수 있다.

## 7. 메모리 사용량 비교

스레드 10,000개를 생성하고 5초간 유지하는 실험에서 메모리 사용량은 다음과 같다:

- Platform Thread (cachedThreadPool): 약 6MB
- Virtual Thread: 약 3MB (약 2배 적음)

개수가 늘어날수록 차이는 더 커진다. Platform Thread는 스택 크기가 고정(기본 512KB~1MB)이지만, Virtual Thread의 스택은 실제 사용량에 따라 동적으로 확장/축소된다.

## 8. Spring Boot 설정

Spring Boot 3.2 이상에서는 단 한 줄의 설정으로 Tomcat과 스케줄러가 Virtual Thread를 사용하도록 전환할 수 있다:

```yaml
# application.yml
spring:
  threads:
    virtual:
      enabled: true
```

이 설정을 활성화하면 `TomcatProtocolHandlerCustomizer`가 `newVirtualThreadPerTaskExecutor()`를 사용하도록 자동 구성된다. 기존 코드 변경 없이 I/O 바운드 처리량이 크게 향상된다.

`ThreadLocal`을 대량 사용하는 경우 주의가 필요하다. Virtual Thread 수백만 개가 각자 `ThreadLocal` 데이터를 갖게 되면 메모리가 급증할 수 있다. Java 21의 `ScopedValue`가 대안으로 제공된다.

## 9. Pinning — 가상 스레드의 함정

> §4에서 블로킹은 자동 unmount된다고 했는데, 그 자유가 깨지는 두 지점이 있다. Pinning은 가상 스레드가 Carrier에 고정돼 블로킹해도 내려오지 못하는 현상으로, 이때 Carrier까지 함께 멈춰 Platform Thread와 같은 병목이 생긴다.

*Pinning*은 가상 스레드가 Carrier Thread에서 unmount되지 못하고 고정(pin)된 상태다. 정상 동작에서는 블로킹 시 가상 스레드가 Carrier를 양보하지만, Pinning이 일어나면 Carrier도 함께 블로킹된다. Carrier 수는 CPU 코어 수와 같아서 모든 Carrier가 Pinning에 빠지면 새 가상 스레드가 실행될 수 없다.

원인은 *JVM이 스택을 안전하게 옮길 수 없는 두 컨텍스트*로 한정된다. 첫째는 **`synchronized` 블록 내 블로킹**이다. `synchronized` 모니터는 OS 뮤텍스와 연결되고 소유권이 Platform Thread에 귀속돼, 모니터를 쥔 채 블로킹하면 unmount가 불가능하다. 둘째는 **JNI(native 메서드) 호출**로, 네이티브 실행 중에는 JVM이 Java 스택 상태를 저장할 수 없다.

```java
// ❌ Pinning 발생: synchronized 블록 내 블로킹 I/O
synchronized (lock) {
    httpClient.get("https://...");   // Carrier Thread가 함께 블로킹됨
}

// ✅ Pinning 없음: ReentrantLock은 JVM 모니터를 쓰지 않아 정상 unmount
private final ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    httpClient.get("https://...");   // 가상 스레드가 정상적으로 unmount
} finally {
    lock.unlock();
}
```

감지는 `-Djdk.tracePinnedThreads=full`(콘솔 스택 트레이스)이나 JFR의 `jdk.VirtualThreadPinned` 이벤트로 한다. 해결은 위처럼 `synchronized`를 `ReentrantLock`으로 바꾸는 것이다. Java 24(JEP 491)부터는 모니터 소유권이 가상 스레드 기준으로 바뀌어 `synchronized` Pinning이 근본 해결됐으므로, Java 21 환경에서만 이 교체가 필요하다. CPU 바운드 작업의 `synchronized`는 블로킹 I/O가 없어 Pinning이 생겨도 성능 영향이 없으니 교체할 필요가 없다.

라이브러리 Pinning은 직접 확인이 원칙이다. Hibernate/Spring Data JPA 3.3.0 이하는 발생 보고가 있고, JDBC 드라이버는 드라이버마다 달라 `-Djdk.tracePinnedThreads`로 짚는다. 마이그레이션 시에는 이 옵션으로 Pinning 지점을 찾아 블로킹 I/O가 있는 `synchronized`만 골라 `ReentrantLock`으로 바꾸고, Java 24+로 올리면 `synchronized` 관련 Pinning은 자동으로 풀린다.
