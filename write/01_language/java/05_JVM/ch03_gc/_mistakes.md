# 3장 GC 학습 오답 노트

> §14 자가평가에서 점수 낮은 자리 + §12 회차 반복에서 막히는 자리를 박제합니다. 같은 패턴이 3회 이상 반복되면 그 챕터 자체가 약하다는 신호이므로 본문 보강 트리거로 씁니다.

## 패턴 1 — Concurrent Mode Failure의 컬렉터·영역 혼동

- **실수 날짜**: 2026-06-08 (회차 1, 02-08 마치며 Q&A)
- **상황**: §02-08 마치며 Q3 — "Concurrent Mode Failure가 어떤 컬렉터에서, 왜 발생하는가"
- **잘못 답한 내용**: "마크 스윕 컬렉터에서 발생하고, 생존(Survivor) 영역에 공간이 부족해 Full GC가 발생하기 때문"
- **정답**:
  - 컬렉터는 단순 마크-스윕이 아니라 **CMS(Concurrent Mark Sweep)** 고유 현상입니다. CMS가 *애플리케이션과 동시에* Old 영역을 회수하는 컬렉터라는 점이 핵심입니다.
  - 부족한 공간은 Survivor가 아니라 **Old 영역**입니다. 애플리케이션이 객체를 Old로 승격시키는 속도가 CMS의 동시 회수 속도를 앞지르면, 동시 회수가 끝나기 전에 Old가 가득 찹니다.
  - 그 순간 CMS는 동시 회수를 포기하고 **Serial Old로 폴백해 STW Full GC**를 돌립니다. 이 폴백이 `Concurrent Mode Failure`이며, 동시성의 이점이 사라지고 긴 STW가 발생하는 운영 사고 신호입니다.
- **왜 틀렸나**: "Concurrent"라는 단어를 알고리즘(마크-스윕)에만 연결하고 *컬렉터 이름(CMS)*으로 연결하지 못함. 또 GC 실패를 반사적으로 Survivor 부족(조기 승격)과 묶음 — CMF는 Old 영역 + 동시성 경합이 본질입니다.
- **참고 챕터**: 02-04 클래식 가비지 컬렉터 §5 CMS, 02-08 마치며 §3a
- **재방문 트리거**: 회차 2 (+1일, 2026-06-09)에 같은 질문 재시도. "CMS·Old·Serial Old 폴백" 세 단어가 자동으로 나오는지 확인.
- **혼동 짝**: Survivor 부족 = premature promotion(조기 승격)은 *다른 현상*. 둘을 구분해 설명할 수 있어야 함.
- **본문 보강 (2026-06-08)**: 진단 결과 02-04 §5 본문이 CMF를 "단편화의 결과"로만 설명하고 *발생 조건*(Old 영역 + 승격/회수 타이밍 경합)을 누락하고 있었음 → §15 규칙대로 본문 보강 완료. 단편화 항목 아래에 "CMF는 언제 나는가" 인용 단락 추가(두 경로: 연속 빈 공간 부족 / 승격 속도 > 동시 회수 속도). 즉 이 오답은 *본인 복습 부족 + 본문 부실*이 겹친 케이스.
