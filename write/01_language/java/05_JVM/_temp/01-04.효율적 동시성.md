# 효율적 동시성
---
> JVM이 멀티코어 하드웨어의 메모리 모델 차이를 어떻게 추상화하는지, 그리고 동시성 성능을 높이기 위해 락을 어떻게 최적화하는지 이해한다. 하드웨어 캐시 일관성 문제, JMM 설계 원리, volatile/synchronized 동작, 락 최적화 기법, CAS를 다룬다. 본 요약을 한 줄로 압축하면 — **JMM은 하드웨어 캐시 일관성의 *추상화*이며, `volatile`·`synchronized`·CAS는 그 추상화 위에서 *가시성과 원자성*을 표현하는 세 도구**다. 락 최적화는 그 도구의 *런타임 비용*을 깎는 작업이다.

## 1. 하드웨어 효율성과 일관성

프로세서는 데이터를 읽고 작업 결과를 저장하기 위해 메모리 IO가 반드시 필요하다. 그런데 프로세서와 메모리의 속도 차이가 너무 커서, 멀티프로세서 시스템은 프로세서마다 자신만의 **캐시 계층**을 두어 속도 격차를 해소한다. 이 구조는 처리 속도를 높여주지만, 여러 프로세서가 같은 메인 메모리 영역을 수정할 때 어느 캐시 값을 기준으로 동기화할지 결정해야 하는 **캐시 일관성(Cache Coherence)** 문제를 만든다.

일관성 문제를 해결하기 위해 프로세서는 캐시 접근 시 MESI, MSI 등 프로토콜을 따른다. 캐시 외에도 프로세서는 **비순차 실행 최적화**로 성능을 끌어내는데, 명령어 실행 순서가 입력 코드 순서와 다를 수 있으며 최종 결과만 순차와 동일하게 재구성한다. JVM에서는 JIT 컴파일러가 이와 유사한 **명령어 재정렬** 최적화를 수행한다.

## 2. Java 메모리 모델(JMM)

**JMM**(Java Memory Model)은 Java 프로그램이 다양한 하드웨어와 운영체제의 메모리 모델 차이로부터 보호받도록 정의된 명세다. C/C++은 하드웨어/OS의 메모리 모델을 직접 사용하여 개발자가 이를 직접 처리해야 하지만, JMM은 이를 추상화하여 일관된 동작을 보장한다.

### 2-1. 메인 메모리와 작업 메모리

JMM은 변수를 **메인 메모리**와 각 스레드의 **작업 메모리**로 나누어 관리한다.

- **메인 메모리**: 인스턴스 변수, 정적 변수, 배열 요소 등 모든 스레드가 공유하는 변수를 저장
- **작업 메모리**: 각 스레드에 독립적으로 존재하며, 메인 메모리에서 사용할 변수의 복사본을 저장. 지역 변수와 메서드 파라미터는 스레드 간 경합이 없으므로 제외

스레드는 메인 메모리의 변수를 직접 읽거나 쓸 수 없다. 반드시 작업 메모리를 거쳐야 하며, JMM은 두 메모리 사이의 상호작용을 8가지 원자 연산(잠금, 잠금 해제, 읽기, 적재, 사용, 할당, 저장, 쓰기)으로 정의한다.

### 2-2. 원자성, 가시성, 실행 순서

JMM은 동시성 처리의 세 가지 핵심 특성을 정의한다.

**원자성(Atomicity)**: JMM이 직접 보장하는 원자적 연산은 읽기/적재/할당/사용/저장/쓰기다. `long`과 `double`의 읽기/쓰기는 64비트로 두 번의 32비트 연산으로 나뉠 수 있어 원자적이지 않을 수 있다. `synchronized`(monitorenter/monitorexit)로 잠금/잠금 해제를 추가하면 더 큰 원자성 범위를 보장한다.

**가시성(Visibility)**: 한 스레드가 공유 변수를 수정하면 다른 스레드가 즉시 그 결과를 볼 수 있어야 한다. `volatile`은 수정 즉시 메인 메모리에 동기화하고 사용 때마다 메인 메모리에서 새로 읽는다. `synchronized`는 잠금 해제 전 메인 메모리로 동기화를 보장한다. `final` 필드는 생성자에서 초기화 완료 후 다른 스레드에 동기화 없이 올바르게 보인다.

**실행 순서(Ordering)**: 현재 스레드에서는 모든 연산이 순서대로 보이지만, 다른 스레드에서 보면 명령어 재정렬로 인해 순서가 다를 수 있다. `volatile`은 명령어 재정렬을 금지하고, `synchronized`는 락을 소유한 스레드만이 변수에 접근하도록 보장한다.

## 3. volatile과 synchronized

### 3-1. volatile

`volatile`은 자바에서 가볍고 효율적인 동기화 메커니즘이다. 가시성과 실행 순서를 보장하지만, 복합 연산에 대한 원자성은 보장하지 않는다.

```java
// volatile 가시성 문제 해결 예시
public class StopFlag {
    private volatile boolean stopped = false;  // volatile 없으면 루프가 종료 안 될 수 있음

    public void stop() { stopped = true; }

    public void run() {
        while (!stopped) {  // 항상 메인 메모리에서 최신 값을 읽음
            doWork();
        }
    }
}
```

```java
// volatile로 원자성이 보장되지 않는 예시
private volatile int count = 0;

// i++ 는 내부적으로 읽기 → 증가 → 쓰기 세 단계이므로 비원자적
// 멀티스레드 환경에서 데이터 경합 발생 가능
public void increment() { count++; }
```

| 특성 | 일반 변수 | volatile | synchronized |
|---|---|---|---|
| **가시성** | 보장 안 됨 | 즉시 동기화 | 락 해제 시 동기화 |
| **원자성** | 보장 안 됨 | 단순 읽기/쓰기만 | 블록 전체 보장 |
| **명령어 재정렬 방지** | 안 됨 | 됨 | 됨 |
| **성능 비용** | 없음 | 낮음 | 중간~높음 |

### 3-2. synchronized와 모니터

`synchronized`는 **모니터**(Monitor) 기반의 상호 배제 동기화를 제공한다. 바이트코드 레벨에서는 `monitorenter`와 `monitorexit` 명령어로 구현된다.

```java
public class Counter {
    private int count = 0;

    // synchronized 블록: monitorenter, monitorexit 명령어 삽입
    public void increment() {
        synchronized (this) {
            count++;  // 원자적으로 실행
        }
    }

    // synchronized 메서드: ACC_SYNCHRONIZED 플래그 설정
    public synchronized int getCount() {
        return count;
    }
}
```

락 획득은 커널/사용자 모드 전환이 필요한 네이티브 코드를 사용하므로 비용이 크다. 이 때문에 JVM은 동시성 성능을 높이기 위한 여러 락 최적화 기법을 도입했다.

## 4. 락 최적화 기법

### 4-1. 편향 락(Biased Locking)

편향 락은 락이 항상 특정 스레드 하나에 의해서만 획득된다고 가정하는 최적화다. 처음 락을 획득한 스레드 ID를 객체 헤더 마크 워드에 기록하고, 이후 같은 스레드가 락을 요청하면 CAS 연산 없이 즉시 진입한다. 실제로 대부분의 락은 경합 없이 단일 스레드에 의해 사용되므로 효과가 크다.

편향 락은 Java 15부터 deprecated 되었고 Java 21에서 제거되었다. 현대 애플리케이션에서는 스레드 풀 기반으로 여러 스레드가 락을 공유하는 경우가 많아 오히려 편향 해제 비용이 문제가 되기 때문이다.

### 4-2. 경량 락(Lightweight Locking)

경량 락은 실제로 경합이 없는 동기화에서 커널 모드 전환 없이 **CAS**(Compare-And-Swap) 연산만으로 락을 처리하는 최적화다. JVM은 스택 프레임에 락 레코드를 생성하고, CAS로 객체 마크 워드를 락 레코드 포인터로 교체한다. CAS 성공 시 락 획득이며, 실패 시 경합이 있다는 뜻으로 중량 락으로 팽창한다.

### 4-3. 중량 락(Heavyweight Locking)

경량 락 CAS에 실패하면 OS의 Mutex(뮤텍스)를 사용하는 중량 락으로 전환된다. 중량 락은 커널 모드 전환이 필요하여 비용이 크지만, 실제로 경합이 있는 상황에서는 정확한 동기화를 제공한다. 다른 스레드는 잠긴 동안 블록 상태로 대기한다.

```
편향 락 → (경합 발생) → 경량 락 → (경합 심화) → 중량 락
```

### 4-4. 락 제거(Lock Elimination)

*락 제거*는 JIT 컴파일러가 탈출 분석을 통해 특정 락 객체가 스레드 간 공유되지 않는다고 판단하면 해당 동기화를 완전히 제거하는 최적화다.

```java
// StringBuffer는 synchronized 메서드를 가지지만
// 이 예시에서 sb는 메서드 범위를 벗어나지 않으므로 락 제거 가능
public String concatStrings(String s1, String s2) {
    var sb = new StringBuffer();  // 로컬 변수, 탈출 없음
    sb.append(s1);   // synchronized이지만 JIT가 락 제거
    sb.append(s2);
    return sb.toString();
}
```

### 4-5. 락 거칠게 하기(Lock Coarsening)

락 거칠게 하기는 연속된 동기화 블록을 하나로 합치는 최적화다. 동일 락 객체에 대한 잦은 락 획득/해제 반복을 줄여 오버헤드를 감소시킨다.

```java
// 최적화 전: 락 획득/해제가 반복됨
synchronized (lock) { append(s1); }
synchronized (lock) { append(s2); }
synchronized (lock) { append(s3); }

// 최적화 후: 하나의 동기화 블록으로 통합
synchronized (lock) {
    append(s1);
    append(s2);
    append(s3);
}
```

## 5. CAS와 논블로킹 동기화

**CAS**(Compare-And-Swap)는 하드웨어 명령어 수준에서 원자적으로 "비교 후 교환"을 수행하는 연산이다. "현재 값이 예상 값과 같으면 새 값으로 교체한다"는 하나의 원자적 명령으로 처리된다. 스레드를 블록할 필요 없이 실패 시 재시도하는 방식이어서 **논블로킹 동기화**라 부른다.

```java
import java.util.concurrent.atomic.AtomicInteger;

// AtomicInteger는 내부적으로 CAS(Unsafe.compareAndSwapInt)를 사용
public class NonBlockingCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet();  // CAS 기반, 락 없음
    }

    public int get() {
        return count.get();
    }
}
```

CAS의 약점은 **ABA 문제**다. 값이 A → B → A로 변했을 때 CAS는 변화를 감지하지 못한다. `AtomicStampedReference`처럼 버전 번호를 함께 관리하면 이 문제를 해결할 수 있다. 또한 경합이 심한 상황에서 CAS 재시도가 많아지면 CPU를 낭비하는 스핀 오버헤드가 발생할 수 있다.

## 6. ReentrantLock vs synchronized

`ReentrantLock`은 `synchronized`의 모든 기능을 포괄하면서 추가 기능을 제공한다.

| 특징 | synchronized | ReentrantLock |
|---|---|---|
| **공정 락** | 미지원 | `new ReentrantLock(true)`로 지원 |
| **인터럽트 가능 대기** | 미지원 | `lockInterruptibly()` 지원 |
| **타임아웃 시도** | 미지원 | `tryLock(time, unit)` 지원 |
| **복수 Condition** | 미지원 | `newCondition()`으로 복수 지원 |
| **락 해제** | JVM 자동 보장 | `finally` 블록에서 수동 해제 필요 |
| **JVM 최적화** | 모든 락 최적화 적용 | 제한적 |

Java 6 이후로는 `synchronized`도 경량 락, 편향 락 등의 최적화가 적용되어 단순 시나리오에서는 `ReentrantLock`과 성능 차이가 거의 없다. 단순한 상호 배제가 필요한 경우에는 `synchronized`를, 공정성이나 인터럽트 처리 같은 고급 기능이 필요한 경우에는 `ReentrantLock`을 사용하는 것이 일반적인 지침이다.
