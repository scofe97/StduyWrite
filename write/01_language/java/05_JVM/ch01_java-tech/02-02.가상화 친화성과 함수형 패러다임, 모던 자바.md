# 가상화 친화성과 함수형 패러다임, 모던 자바
---
> 앞 노트([01-01](./02-01.자바%20기술의%20미래%20—%20언어%20독립과%20차세대%20JIT.md))의 그랄(Graal)이 *기술 기반의 변화*였다면, 이 노트가 다루는 세 절은 *환경과 사고방식의 변화*다. 자바가 작아져야 하는 이유(§1.5.3 가상화 친화성), 자바가 함수형을 흡수하는 이유(§1.5.4), 자바 문법이 빠르게 변하는 이유(§1.5.5)를 한 묶음으로 본다. 세 주제를 한 노트에 묶은 까닭은 단순하다. 가상화는 *작은 시작 시간*을, 함수형은 *멀티코어 친화적 스타일*을, 모던 문법은 *그것을 어색하지 않게 표현할 수단*을 각각 제공하기 때문이다. 한 묶음으로 봐야 클라우드 네이티브 환경의 자바가 그리는 큰 그림이 보인다.



## 1. §1.5.3 가상화 친화성 — 자바는 왜 작아져야 하는가

> 마이크로서비스와 컨테이너 환경이 자바에 새 압박을 가했다.

자바는 오랫동안 "서버에 오래 떠 있는 큰 프로세스"를 전제로 최적화돼 왔다. 클래스 로딩과 JIT 워밍업은 처음 몇 초에서 몇 분을 잡아먹지만, 일단 안정 상태에 들어가면 빠르고 안정적이다. 클라우드 네이티브 환경은 이 전제를 깬다. 한 마이크로서비스가 수십 개의 컨테이너로 복제되고, 트래픽이 줄면 수가 줄고 다시 늘면 새 컨테이너가 뜬다. 책의 그림 1-7과 1-8(p.38, 39)이 그 현실을 정량적으로 보여 준다. 자바 마이크로서비스의 시작 시간이 다른 언어보다 길고, 메모리 사용량도 더 크다는 막대 그래프다.

이 압박에 대한 자바 진영의 답은 두 갈래다. 한 갈래는 *런타임을 작고 빠르게* 만드는 일이다. 모듈 시스템(JDK 9)으로 필요 없는 모듈을 떼어내고, `jlink`로 맞춤형 JRE를 만들고, AppCDS로 클래스 메타데이터를 미리 적재해 워밍업을 줄인다. 다른 갈래는 *JIT를 포기하고 AOT로 가는* 길이다. GraalVM Native Image는 자바 코드를 사전에 네이티브 바이너리로 컴파일해 시작 시간을 수십 밀리초로 끌어내린다.

두 갈래의 트레이드오프는 분명하다. AOT는 정적 분석에 기반하므로 리플렉션과 동적 클래스 로딩에 제약이 있고, peak throughput에서는 잘 워밍업된 JIT 빌드가 여전히 우세할 수 있다. 반면 시작 시간과 메모리 풋프린트에서는 AOT가 압도적이다. **언제 어떤 모드를 쓸지가 가상화 친화성의 실무 결정 사항**이다. 함수형 람다 기반 서버리스 워크로드는 거의 무조건 AOT, 장기 가동 API 서버는 JIT 쪽이 자연스러운 출발점이다.


### 1.1 가상 머신 위에 가상 머신을 얹는 비효율

> 컨테이너는 가상 머신을 다시 가상 머신 위에 얹는 구조다.

자바 가상 머신은 운영체제 위에서 가상 머신 역할을 하는 도구다. 컨테이너 환경에서는 그 자바 가상 머신이 다시 컨테이너(가상 환경) 안에서 돈다. 두 층의 가상화가 겹치면 메모리 회계가 어긋나기 쉽다. 책은 컨테이너 cgroup이 할당한 메모리를 자바 가상 머신이 정확히 인식하지 못해 OOM-killer가 컨테이너를 죽이는 사례를 든다. 이런 사고가 누적되며 JVM은 `-XX:+UseContainerSupport` 같은 컨테이너 인식 옵션을 추가했고, 최근 JDK는 이 옵션을 기본값으로 켜 두는 방향으로 갔다.



## 2. §1.5.4 함수형 프로그래밍 — 자바가 다른 패러다임을 흡수하는 방식

> 자바는 객체 지향으로 시작했지만, 동시성과 데이터 처리 흐름이 함수형의 손을 빌리는 쪽이 자연스러워졌다.

자바 8의 람다와 스트림은 "문법적 단축"이 아니라 패러다임 흡수의 신호였다. 컬렉션을 명령형으로 순회하던 코드를 `stream().filter().map().reduce()`처럼 *어떤 변환을 어떤 순서로* 표현하는 방식으로 옮기면, 멀티코어에서의 병렬 처리와 코드 가독성이 동시에 좋아진다. 자바 진영은 Scala와 Kotlin이 보여 준 이점을 그대로 두지 않고 자바 자체에 흡수했다.

흡수의 방향은 세 가지다. **불변성 선호**가 한 축으로, `record` 도입과 함께 불변 데이터 클래스가 표준이 됐다. **타입 추론**은 `var` 키워드로 지역 변수 타입을 생략한다. **패턴 매칭**은 `instanceof` 패턴, `switch` 식, `sealed` 인터페이스로 대수적 데이터 타입을 표현한다. 이 세 변화가 자바를 함수형 언어로 만들지는 않지만, 함수형 스타일을 *어색하지 않게 쓸 수 있는 자바*로 만든다.


### 2.1 Kotlin과 Scala의 영향

> 자바 자체가 변한 배경에는 다른 JVM 언어들이 있다.

Scala는 자바보다 먼저 패턴 매칭, 케이스 클래스, for-comprehension을 정착시켰다. Kotlin은 안드로이드에서 빠르게 표준이 되며 자바보다 짧고 안전한 코드 작성 경험을 보여 줬다. 자바는 두 언어를 따라잡으려 하지 않고, *이미 검증된 좋은 아이디어만 흡수*하는 전략을 택했다. 결과적으로 자바 14의 `record`, 17의 `sealed`, 21의 패턴 매칭은 Kotlin과 Scala의 핵심 기능을 자바 문법으로 옮긴 것이다. 자바 진영의 보수성은 흔히 단점으로 지적되지만, 이 절에서는 오히려 강점으로 작동한다. 큰 실수 없이 흡수할 기능만 흡수한다.



## 3. §1.5.5 모던 자바 문법 — 6개월 출시 주기의 결과

> 자바 9 이후 6개월 출시 주기는 문법 변화의 속도 자체를 바꿔 놓았다.

자바 6에서 7로 가는 데 5년이 걸렸지만, 자바 9 이후로는 매년 두 번 새 기능이 들어온다. preview 단계, second preview, 정식 채택의 3단계 흐름이 정착되며 큰 사고 없이 빠르게 진화하는 문법이 가능해졌다. 책은 다음 기능들을 모던 자바의 핵심으로 짚는다.

- **`var` (JDK 10)**: 지역 변수 타입 추론. 우변 타입이 명확할 때 좌변을 생략한다.
- **`record` (JDK 14)**: `equals`/`hashCode`/`toString`/접근자를 자동 생성하는 불변 데이터 클래스.
- **`sealed` (JDK 17)**: 상속 허용 클래스 목록을 명시적으로 제한해 패턴 매칭 완전성을 보장.
- **`switch` 식과 패턴 매칭 (JDK 14·21)**: `switch`를 표현식으로 쓰고, `case Foo f when ...` 형태의 패턴 가드를 작성.
- **텍스트 블록 (JDK 15)**: 여러 줄 문자열을 `"""`로 감싸 자연스럽게 작성.

문법 변화는 *언어를 가볍게 만든다*. 이전 자바에서 데이터 클래스를 정의하는 데 30줄 가까이 든 자리를 `record`는 한 줄로 끝낸다. 다음은 같은 의미의 두 표현을 나란히 둔 예다.

```java
// 모던 자바 (JDK 21)
public record Point(int x, int y) {}

// 이전 방식: 약 30줄의 보일러플레이트 필요
//   - private final int x, y
//   - 생성자, 두 getter
//   - equals: x와 y를 함께 비교
//   - hashCode: Objects.hash(x, y)
//   - toString: "Point[x=..., y=...]"
```


### 3.1 빠른 출시 주기의 부작용

> 모든 기능이 다 좋은 방향만은 아니다.

빠른 진화에는 비용이 따른다. 라이브러리는 LTS 버전(8/11/17/21)을 기준으로 호환성을 잡지만, 그 사이 비-LTS 버전에 들어간 preview 기능이 정식화되지 못하고 빠지는 일도 있다. 책은 명시적으로 이 점을 경고한다 — **production 코드에서 preview 기능을 쓰지 말 것**. 모던 자바를 따라가되, 안정성 라인은 LTS에 맞춘다는 운영 원칙이 필요하다. 도입 페이스는 "preview에서 한 번 만져 보고, 정식 채택된 LTS에서 production 적용"이 무난하다.



## 4. 핵심 정리

§1.5.3~§1.5.5는 한 문장으로 압축된다. *자바는 클라우드 네이티브 환경에 맞춰 더 작아지고, 다른 언어의 좋은 아이디어를 빠르게 흡수하며, 6개월 주기로 그 흡수를 문법에 반영한다*. 이 세 변화는 [01-01](./02-01.자바%20기술의%20미래%20—%20언어%20독립과%20차세대%20JIT.md)의 언어 독립·차세대 JIT와 한 묶음으로 굴러간다. 다음 노트([01-03](./02-03.실전%20—%20OpenJDK%20빌드하기.md))는 이 모든 변화를 실제로 만지기 위한 발판인 OpenJDK 빌드를 다룬다.



## 5. 실습 연결

본 절은 개념 중심이라 즉시 따라갈 코드가 적다. 출발점은 [`_practice/ch01/`](../_practice/ch01/)의 `JavaTechSystemDemo`로, 현재 가동 중인 JVM이 어느 벤더·버전인지 출력해 본인이 어느 LTS 라인 위에서 실험하는지 확인한다. 그 위에 두 가지 확장이 자연스럽게 따라온다. 첫째는 **Native Image 시작 시간 측정** — 같은 코드를 GraalVM Native Image로 빌드해 JIT 빌드와 시작 시간을 비교한다. 환경 설치 비용이 커서 별도 요청 시 [`_practice/ch01/`](../_practice/ch01/) 안에 추가한다. 둘째는 **`record`/`sealed`/패턴 매칭 데모** — JDK 21 기준으로 §1.5.5의 문법을 한 파일에 모아 본다. 둘 다 책이 "한 번 손으로 만져 두면 §1 전체의 감이 잡힌다"고 권한 패턴이다.
