# 고급 컴파일러 플래그 — threshold·threads·inlining·escape analysis
> 컴파일 트리거·스레드·inlining·escape analysis 플래그는 대부분 바꾸지 말아야 하며, 컴파일러 동작을 이해하기 위한 것입니다

[앞 편](./04-02.code%20cache와%20컴파일%20관찰%20—%20PrintCompilation·deoptimization.md)이 컴파일 관찰과 deopt였다면, 이 편은 컴파일러에 영향을 주는 고급 플래그입니다. 저자는 이 플래그들이 대체로 쓰여선 안 되고, 컴파일러 동작을 더 잘 이해하거나 과거 널리 쓰여 마주쳤을 때 의미를 알려는 목적이라고 분명히 합니다.



## 1. compilation threshold와 counter decay
> 컴파일은 메서드 호출과 루프 back-edge 횟수의 합으로 트리거되며, counter가 시간에 따라 감소해 lukewarm 메서드가 생깁니다

코드 컴파일을 무엇이 트리거하는지 그동안 모호했습니다. 주된 요인은 코드가 얼마나 자주 실행되느냐입니다. 일정 횟수 실행되면 **compilation threshold**에 도달하고 컴파일러가 컴파일에 충분한 정보를 가졌다고 판단합니다. 컴파일은 두 counter에 기반합니다. **메서드가 호출된 횟수**와 **메서드의 루프가 back-branch한 횟수**입니다. back-branch는 루프가 끝에 도달하거나 `continue` 같은 분기로 실행을 완료한 횟수로 볼 수 있습니다. JVM은 Java 메서드를 실행할 때 두 counter의 합을 확인해 컴파일 자격을 정하고, 자격이 되면 큐에 넣습니다(이를 standard compilation이라 부릅니다). 루프가 한 번 실행을 완료할 때마다 back-branch counter가 증가·검사되고, 개별 threshold를 넘으면 (전체 메서드가 아니라) 루프가 컴파일 자격이 됩니다.

tiered가 꺼지면 standard compilation은 `-XX:CompileThreshold=N`(기본 10,000)으로 트리거됩니다. 값을 바꾸면 컴파일러가 평소보다 일찍·늦게 컴파일합니다. **그러나 이 플래그는 tiered가 꺼졌을 때만 동작하고, tiered가 켜진(보통의) 상태에서는 아무 일도 하지 않습니다.** 이 플래그를 바꾸라는 권고와 8,000 같은 값을 쓰는 벤치마크가 있지만, JDK 7 이전의 잔재입니다.

이 플래그가 한때 권장된 두 이유가 있습니다. 첫째, 낮추면 C2 컴파일러를 쓰는 앱의 시작 시간이 개선됩니다(코드가 더 빨리, 보통 같은 효율로 컴파일됨). 둘째, 안 그러면 결코 컴파일 안 됐을 메서드가 컴파일됩니다. 마지막 점이 흥미로운 quirk입니다. 프로그램이 영원히 돌면 결국 모든 코드가 컴파일될 것 같지만 그렇지 않습니다. **counter는 메서드·루프 실행에 따라 증가하지만 시간이 지나며 감소하기 때문**입니다. 주기적으로(JVM이 safepoint에 도달할 때) 각 counter 값이 줄어듭니다. 실용적으로, counter는 메서드·루프의 최근 hotness의 상대 측정치입니다. 부작용으로, 다소 자주 실행되는 코드가 영원히 도는 프로그램에서도 C2로 컴파일 안 될 수 있습니다. 이를 (hot과 대비해) **lukewarm** 메서드라 합니다. tiered 이전엔 이게 threshold를 낮추는 게 이로운 한 경우였습니다. **오늘날은 tiered 덕에 lukewarm 메서드도 컴파일됩니다**(C1로지만). 실용적 이득은 적지만, 정 관심 있으면 `-XX:Tier3InvocationThreshold=N`(기본 200)·`-XX:Tier4InvocationThreshold=N`(기본 5000)을 바꿔 봅니다.



## 2. compilation threads
> 컴파일은 백그라운드 큐를 핫 메서드 우선으로 처리하며, CPU 수 기반 스레드 수를 CICompilerCount로 조정합니다

메서드(나 루프)가 컴파일 자격이 되면 큐에 들어가고, 하나 이상의 백그라운드 스레드가 처리합니다. **큐는 엄격한 FIFO가 아니라 invocation counter가 높은 메서드가 우선**입니다. 그래서 프로그램 시작 시 컴파일할 코드가 많아도 가장 중요한 코드가 먼저 컴파일됩니다(이것도 컴파일 ID가 순서 어긋나 보이는 이유). C1·C2는 별도 큐를 갖고 각각 다른 스레드(잠재적으로 여럿)가 처리합니다. 스레드 수는 CPU 수 기반의 복잡한 로그 공식으로 정해집니다.

| CPU | C1 스레드 | C2 스레드 |
|-----|-----------|-----------|
| 1 | 1 | 1 |
| 2 | 1 | 1 |
| 4 | 1 | 2 |
| 8 | 1 | 2 |
| 16 | 2 | 6 |
| 32 | 3 | 7 |
| 64 | 4 | 8 |
| 128 | 4 | 10 |

`-XX:CICompilerCount=N`으로 총 스레드 수를 조정합니다. tiered면 1/3(최소 1)이 C1 큐를, 나머지(최소 1)가 C2 큐를 처리합니다. **언제 조정할까요?** 기본값이 CPU 수 기반이라, **옛 JDK 8을 Docker 컨테이너 안에서 돌리면 자동 튜닝이 어긋날 수 있습니다.** 그 경우 (컨테이너에 할당된 CPU 수 기준 위 표를 가이드로) 수동 설정해야 합니다. 단일 CPU VM이면 컴파일러 스레드 1개가 약간 이로울 수 있습니다(제한된 CPU 경합 감소). 단 그 이점은 초기 warm-up 기간뿐이고, stock 배치 앱을 단일 CPU에서 스레드 1개로 돌렸을 때 초기 계산이 10% 빨랐지만 반복이 늘수록 효과가 사라졌습니다. 반대로 여러 JVM을 동시에 돌리면 스레드가 시스템을 압도할 수 있어, 줄이면 전체 처리량이 나아질 수 있습니다(warm-up이 길어지는 대가).

`-XX:+BackgroundCompilation`(기본 true)은 큐를 비동기로 처리한다는 뜻입니다. false면 메서드가 컴파일 자격이 될 때 실행하려는 코드가 컴파일될 때까지 기다립니다(인터프리트를 계속하지 않고). `-Xbatch`도 백그라운드 컴파일을 끕니다.



## 3. inlining — 가장 유익한 최적화
> getter/setter 같은 작은 메서드 호출을 제거하며, 자주 호출되면 325바이트, 그 외는 35바이트 미만일 때 인라인됩니다

컴파일러가 하는 가장 중요한 최적화의 하나는 **메서드 inlining**입니다. getter/setter로 속성에 접근하는 객체지향 코드를 생각합니다. 메서드 호출 오버헤드는 메서드 코드량 대비 꽤 높아, Java 초기엔 이런 캡슐화를 성능 때문에 피하라는 조언도 있었습니다. 다행히 JVM은 이제 이런 메서드를 일상적으로 인라인합니다. 그래서 `p.setX(p.getX() * 2)`로 쓴 코드를 컴파일된 코드는 본질적으로 `p.x = p.x * 2`로 실행합니다.

inlining은 기본 활성이고 `-XX:-Inline`으로 끌 수 있지만, **끄면 stock 배치 테스트 성능이 50% 넘게 떨어질 만큼 중요해 실제로 끄지 않습니다.** JVM이 코드를 인라인하는 방식에 대한 기본 가시성은 없습니다(소스에서 빌드한 debug 버전의 `-XX:+PrintInlining`만 정보를 줌). 인라인 여부의 기본 결정은 메서드가 얼마나 hot한지와 크기에 달렸습니다. JVM은 내부 계산으로 hot 여부를 정합니다(튜닝 불가). 자주 호출되어 인라인 자격이 되면 **바이트코드 크기가 325바이트 미만일 때만** 인라인됩니다(`-XX:MaxFreqInlineSize=N`). 그렇지 않으면 **35바이트 미만일 때만** 인라인 자격이 됩니다(`-XX:MaxInlineSize=N`).

`MaxInlineSize`를 35보다 높이라는 권고가 있는데, 자주 간과되는 점이 있습니다. 35보다 높이면 메서드가 **처음 호출될 때** 인라인될 수 있다는 뜻입니다. 그러나 메서드가 자주 호출되면(그래서 성능이 더 중요하면) 어차피 결국 인라인됩니다(325바이트 미만이라면). 그래서 `MaxInlineSize` 튜닝의 순효과는 테스트 warm-up 시간을 줄일 수 있을 뿐, 오래 도는 애플리케이션에 큰 영향을 줄 가능성은 낮습니다. **inlining 튜닝은 거의 불필요하고, 권고들은 normal inlining과 frequent inlining의 관계를 간과하기 쉽습니다.**



## 4. escape analysis — 가장 정교한 최적화
> 루프 안에서만 참조되는 객체의 락을 없애고 필드를 레지스터에 두며 객체 할당 자체를 생략하는, 가장 정교한 최적화입니다

C2 컴파일러는 escape analysis가 켜지면(`-XX:+DoEscapeAnalysis`, 기본 true) 적극적 최적화를 합니다. factorial 예입니다.

```java
public class Factorial {
    private BigInteger factorial;
    private int n;
    public Factorial(int n) {
        this.n = n;
    }
    public synchronized BigInteger getFactorial() {
        if (factorial == null)
            factorial = ...;
        return factorial;
    }
}
```

처음 100개 factorial 값을 배열에 저장하는 코드입니다.

```java
ArrayList<BigInteger> list = new ArrayList<BigInteger>();
for (int i = 0; i < 100; i++) {
    Factorial factorial = new Factorial(i);
    list.add(factorial.getFactorial());
}
```

`factorial` 객체는 루프 안에서만 참조되고 다른 코드가 결코 접근 못 합니다. 그래서 JVM은 이 객체에 최적화를 할 수 있습니다.

1. `getFactorial()` 호출 시 **synchronization 락을 안 잡아도** 됩니다.
2. 필드 `n`을 메모리에 저장하지 않고 **레지스터에 둘** 수 있습니다(객체 참조도 마찬가지).
3. 사실 **factorial 객체를 아예 할당하지 않고** 개별 필드만 추적할 수 있습니다.

이 최적화는 정교합니다. 이 예는 단순하지만 더 복잡한 코드에도 가능하고, escape analysis가 어느 최적화가 가능한지 정해 컴파일 코드를 바꿉니다. 기본 활성이고 드물게 틀리는데, 과거 high-profile 버그 때문에 끄라는 권고를 보지만 지금은 적절치 않을 가능성이 큽니다. **escape analysis는 마이크로벤치마크가 빗나가는 흔한 원인**입니다(2장의 함정과 이어집니다). 끄는 게 더 안정적이라면 그 코드를 단순화하는 게 최선입니다. 단순한 코드가 더 잘 컴파일됩니다.



## 5. CPU별 코드·tiered 트레이드오프·javac
> JIT는 CPU에 맞는 AVX 명령을 선택하고, tiered를 끄면 메모리를 아끼지만 시작이 느려지며, javac는 성능에 거의 영향이 없습니다

**CPU별 코드** — JIT의 한 이점은 도는 곳에 따라 다른 프로세서용 코드를 낼 수 있다는 것입니다(JVM이 그 프로세서 지식으로 빌드됐다면). Intel은 2011년 Sandy Bridge에 AVX2를, 2016년 AVX-512를 도입했습니다. AVX-512는 JDK 8엔 없고 JDK 11엔 있습니다. 보통 신경 쓸 일이 없지만(JVM이 CPU를 감지해 적절한 세트 선택) 가끔 어긋납니다. `-XX:UseAVX=N`으로 선택합니다(0 없음, 1 Sandy Bridge, 2 Haswell, 3 AVX-512). 기본값은 프로세서에 달려 JVM이 지원하는 최고값을 고릅니다. **AVX-512는 JDK 11에서 기본 활성이었다가 11.0.6부터 다시 기본 비활성**입니다(여러 false start를 거친 진행 중 작업). 그래서 신 Intel HW의 일부 프로그램에서 이전 명령 세트가 훨씬 나을 수 있습니다. 구 버전 JDK 11을 최신 Intel에서 써야 하면 `-XX:UseAVX=2`를 시도하면 흔히 성능이 오릅니다. (참고로 `-XX:UseSSE=N`은 SSE 1~4를 지원하지만 지금은 robust해 튜닝할 일이 없습니다.)

**tiered 트레이드오프** — tiered가 성능 이점을 주는데 끌 이유가 있을까요? **메모리 제약 환경**입니다. 작은 메모리 한도의 Docker 컨테이너, 메모리가 빠듯한 클라우드 VM, 큰 머신에 수십 JVM을 돌릴 때입니다. NetBeans 시작 측정에서, tiered ON은 클래스 22,733개 컴파일·code cache 46.5MB·시작 50.1초인데, OFF는 5,609개·10.7MB·68.5초입니다. C1이 4배 많은 클래스를 컴파일해 4배 메모리를 씁니다. 여기서 34MB 절약은 크지 않지만, 20만 클래스를 컴파일하는 프로그램에서 300MB 절약은 플랫폼에 따라 다른 선택일 수 있습니다. 끄면 시작이 느리지만, 오래 도는 프로그램은 충분한 warm-up 후 거의 같습니다(stock REST 서버는 warm-up 0/60/300초에서 tiered OFF 23.72/23.73/24.42, ON 24.23/24.26/24.43 OPS — tiered가 끝까지 작은 이점을 냅니다).

**javac** — 성능 면에서 컴파일은 JVM의 JIT 얘기입니다. javac는 한 예외를 빼면 성능에 거의 영향이 없습니다. `-g`(디버그 정보)·`final` 키워드·신 javac 재컴파일 모두 성능을 바꾸지 않습니다. **예외는 JDK 11의 새 문자열 연결 방식**으로, 활용하려면 재컴파일이 필요합니다(상세는 12장). 일반적으로 새 기능 활용을 위해 바이트코드 재컴파일이 필요한 적은 없습니다.



## 자주 받는 오해
> MaxInlineSize를 높이면 성능이 오른다고 생각하기 쉽지만, 자주 호출되는 메서드는 어차피 인라인됩니다

1. "`MaxInlineSize`를 35보다 높이면 더 많은 메서드가 인라인돼 빨라진다"고 생각하기 쉽지만, 자주 호출되어 성능이 중요한 메서드는 어차피 frequent inlining(325바이트 미만) 대상이라 결국 인라인됩니다. `MaxInlineSize` 튜닝의 순효과는 warm-up 시간 단축뿐이라 오래 도는 앱엔 큰 영향이 없습니다.
2. "`CompileThreshold`를 낮추면 컴파일이 빨라진다"고 생각하기 쉽지만, 이 플래그는 tiered가 꺼졌을 때만 동작하고 tiered가 켜진 보통 상태에서는 아무 일도 하지 않습니다. JDK 7 이전의 잔재입니다.
3. "프로그램이 영원히 돌면 모든 코드가 컴파일된다"고 생각하기 쉽지만, counter가 safepoint마다 감소해 다소 자주 실행되는 lukewarm 메서드는 C2 컴파일이 안 될 수 있습니다(지금은 tiered 덕에 C1으로는 컴파일됨).
4. "escape analysis는 버그가 많아 끄는 게 안전하다"고 생각하기 쉽지만, 지금은 드물게만 틀리고 과거 버그 기반 권고는 대개 부적절합니다. 끄는 게 더 안정적인 코드라면 그 코드를 단순화하는 게 최선입니다.



## 면접에서 받을 만한 질문
1. **메서드는 어떤 조건에서 인라인됩니까?** → JVM이 내부 계산으로 hot 여부를 정하고, 자주 호출되는 메서드는 바이트코드 325바이트 미만(`MaxFreqInlineSize`)일 때, 그 외 메서드는 35바이트 미만(`MaxInlineSize`)일 때 인라인됩니다. inlining은 getter/setter 같은 작은 메서드의 호출 오버헤드를 제거하는 가장 유익한 최적화라, 끄면 성능이 50% 넘게 떨어집니다. 자주 호출되는 메서드는 어차피 인라인되므로 `MaxInlineSize` 튜닝은 대개 불필요합니다.
2. **escape analysis가 무엇이고 어떤 최적화를 합니까?** → 루프 안에서만 참조되어 다른 코드가 접근 못 하는 객체를 분석하는 가장 정교한 최적화입니다. 그런 객체는 synchronization 락을 안 잡고, 필드를 메인 메모리 대신 레지스터에 두며, 심지어 객체 할당 자체를 생략하고 개별 필드만 추적할 수 있습니다. 기본 활성이며, 마이크로벤치마크가 빗나가는 흔한 원인이기도 합니다.
3. **lukewarm 메서드가 무엇입니까?** → 컴파일 counter는 실행에 따라 증가하지만 safepoint마다 감소합니다. 그래서 counter는 최근 hotness의 상대 측정치이고, 다소 자주 실행되지만 충분히 뜨겁진 않은 메서드는 영원히 도는 프로그램에서도 C2 컴파일이 안 될 수 있습니다. 이를 lukewarm이라 합니다. tiered 이전엔 threshold를 낮춰 이들을 컴파일했지만, 지금은 tiered 덕에 C1으로는 컴파일됩니다.
4. **tiered compilation을 끄는 게 합리적인 경우는?** → 메모리 제약 환경입니다. tiered ON은 C1이 많은 클래스를 컴파일해 code cache를 더 쓰는데(NetBeans 예: 46.5MB vs 10.7MB), 작은 메모리 한도의 Docker나 수십 JVM을 돌리는 머신에서는 끄면 메모리를 아낍니다. 대가는 시작이 느려지는 것이고, 오래 도는 프로그램은 충분한 warm-up 후 성능이 거의 같습니다(tiered가 끝까지 작은 이점은 냅니다).
