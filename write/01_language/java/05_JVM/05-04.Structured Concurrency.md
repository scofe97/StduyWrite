# Structured Concurrency
---
> Structured Concurrency는 부모 태스크가 살아있는 동안 자식 태스크도 반드시 완료된다는 생명주기 보장을 동시성 코드에 도입한다. Java 21에서 Preview Feature(JEP 453)로 제공되며, 스레드 누수와 예외 손실이라는 비구조적 동시성의 근본 문제를 해결한다. 본 노트를 한 줄로 압축하면 — **구조적 동시성의 본질은 *호출 그래프와 동시성 그래프를 일치시키는 일*이며, `StructuredTaskScope`는 가상 스레드 위에서 그 일치를 *try-with-resources 한 블록*으로 표현한다**. 비구조적 `Future.get()` 코드의 근본 결함을 한 단계 위에서 본 결과다.

## 1. 구조적 동시성 개념

*구조적 동시성(Structured Concurrency)*은 동시 실행되는 태스크들을 부모-자식 관계의 트리 구조로 관리하는 패러다임이다. 핵심 원칙은 자식 태스크의 생명주기가 부모 태스크의 생명주기에 종속된다는 것이다. 부모가 종료되기 전에 모든 자식이 반드시 완료되거나 취소된다.

이 원칙은 단일 스레드 코드의 구조적 제어 흐름(if문, 루프, try-catch)을 멀티스레드로 확장한 것이다. 단일 스레드에서 if 블록 안에서 시작한 작업이 블록 밖으로 나갈 수 없듯이, 구조적 동시성에서 scope 안에서 시작한 태스크는 scope 밖으로 나갈 수 없다.

## 2. 비구조적 동시성의 문제점

비구조적 동시성에서는 스레드가 독립적으로 생성되고 생명주기 관리가 수동으로 이루어진다. 이로 인해 두 가지 심각한 문제가 발생한다.

첫 번째는 **스레드 누수**다. 부모 스레드가 종료된 후에도 자식 스레드가 계속 실행되어 리소스를 점유한다. 두 번째는 **예외 전파 실패**다. 자식 스레드에서 발생한 예외가 부모 스레드에 전달되지 않아 묵살된다.

```java
// 비구조적 동시성의 문제
Thread t1 = Thread.startVirtualThread(() -> {
    Thread.sleep(5000);
    System.out.println("작업 1 완료");
});

Thread t2 = Thread.startVirtualThread(() -> {
    throw new RuntimeException("작업 2 실패!");
    // 이 예외는 어디로도 전파되지 않는다
});

// main 스레드는 즉시 반환
// t1은 5초 더 실행되며 리소스를 점유 (누수)
// t2의 예외는 JVM 종료 핸들러에만 전달되거나 완전히 무시된다
```

## 3. StructuredTaskScope: ShutdownOnFailure

`StructuredTaskScope`는 Java 21 Preview(JEP 453)에서 제공하는 구조적 동시성의 핵심 API다. `try-with-resources` 블록과 결합하여 scope가 닫힐 때 모든 자식 태스크를 자동으로 정리한다.

**ShutdownOnFailure**는 자식 태스크 중 하나라도 실패하면 나머지 자식을 모두 취소하는 정책이다. 여러 API를 병렬 호출하고 하나라도 실패하면 전체를 중단해야 할 때 사용한다:

```java
// Java 21 Preview: --enable-preview 플래그 필요
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

    Subtask<String> user  = scope.fork(() -> fetchUser(userId));
    Subtask<Order>  order = scope.fork(() -> fetchOrder(orderId));

    scope.join();           // 모든 자식 완료까지 대기 (블로킹)
    scope.throwIfFailed();  // 실패한 자식이 있으면 예외 발생

    // 여기까지 왔다면 두 태스크 모두 성공
    return combine(user.get(), order.get());

} // scope.close() 자동 호출: 아직 실행 중인 자식 태스크 모두 강제 취소
```

`user`나 `order` 중 하나가 예외를 던지면 `ShutdownOnFailure`가 다른 태스크를 즉시 취소하고, `throwIfFailed()`가 원본 예외를 부모 스레드에 전파한다.

## 4. StructuredTaskScope: ShutdownOnSuccess

**ShutdownOnSuccess**는 자식 태스크 중 하나라도 성공하면 나머지를 모두 취소하는 정책이다. 여러 서버에 동시에 요청을 보내고 가장 먼저 응답한 결과를 사용하는 패턴에 적합하다:

```java
try (var scope = new StructuredTaskScope.ShutdownOnSuccess<String>()) {

    scope.fork(() -> fetchFromPrimaryServer());
    scope.fork(() -> fetchFromSecondaryServer());
    scope.fork(() -> fetchFromCacheServer());

    scope.join();

    // 가장 먼저 성공한 태스크의 결과를 반환
    // 나머지 두 태스크는 자동으로 취소됨
    return scope.result();
}
```

모든 자식이 실패하면 `scope.result()`는 `ExecutionException`을 던진다.

## 5. fork/join 패턴

`scope.fork()`는 새 Virtual Thread를 생성하고 태스크를 제출한다. 반환값인 `Subtask`는 태스크의 상태와 결과를 나타내는 핸들이다. `scope.join()`은 scope에 제출된 모든 자식 태스크가 완료될 때까지 현재 스레드를 블로킹한다.

```java
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

    // fork: 즉시 Virtual Thread 생성 및 실행 시작
    Subtask<String> task1 = scope.fork(() -> "결과 1");
    Subtask<Integer> task2 = scope.fork(() -> 42);
    Subtask<List<String>> task3 = scope.fork(() -> List.of("a", "b"));

    scope.join();           // task1, task2, task3 모두 완료까지 대기
    scope.throwIfFailed();  // 실패 시 예외 전파

    // Subtask.get()은 join() 이후에만 호출 가능
    String r1 = task1.get();
    int r2 = task2.get();
    List<String> r3 = task3.get();
}
```

`Subtask.get()`을 `join()` 전에 호출하면 `IllegalStateException`이 발생한다.

## 6. 예외 전파와 취소 메커니즘

구조적 동시성은 예외 전파와 취소를 체계적으로 처리한다. `ShutdownOnFailure`를 사용할 때의 흐름은 다음과 같다:

- 자식 태스크가 예외를 던지면 scope가 `shutdown()`을 호출한다.
- `shutdown()` 이후 fork된 새 태스크는 즉시 취소된다.
- 실행 중인 다른 자식 태스크에 인터럽트 신호가 전달된다.
- `scope.join()`이 반환된 후 `throwIfFailed()`가 원본 예외를 `ExecutionException`으로 감싸 던진다.
- 부모 스레드는 try-catch로 이 예외를 처리할 수 있다.

```java
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    scope.fork(() -> { throw new IOException("DB 연결 실패"); });
    scope.fork(() -> { Thread.sleep(10_000); return "느린 작업"; });

    scope.join();
    scope.throwIfFailed(e -> new RuntimeException("태스크 실패", e));
    // IOException이 RuntimeException으로 래핑되어 전파됨
    // 느린 작업은 IOException 발생 즉시 인터럽트로 취소됨
} catch (RuntimeException e) {
    // 부모 스레드에서 예외를 처리
}
```

## 7. ScopedValue와의 조합

*ScopedValue*는 Java 21 Preview(JEP 446)에서 제공하는 불변 스레드-로컬 변수다. `ThreadLocal`과 달리 값을 명시적인 스코프 안에서만 바인딩하며, 자식 태스크에 자동으로 상속된다.

`ScopedValue`와 `StructuredTaskScope`를 조합하면 컨텍스트 정보(요청 ID, 사용자 정보 등)를 자식 태스크에 안전하게 전달할 수 있다:

```java
static final ScopedValue<String> REQUEST_ID = ScopedValue.newInstance();

public void handleRequest(String requestId) {
    ScopedValue.where(REQUEST_ID, requestId).run(() -> {
        try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
            scope.fork(() -> {
                // 자식 태스크에서 REQUEST_ID를 그대로 읽을 수 있다
                log.info("요청 처리 중: {}", REQUEST_ID.get());
                return fetchData();
            });
            scope.join();
            scope.throwIfFailed();
        }
    });
}
```

`ThreadLocal`은 Virtual Thread 수백만 개가 각자 복사본을 보유하므로 메모리가 폭증할 수 있다. `ScopedValue`는 불변이고 스코프 기반으로 관리되어 Virtual Thread 환경에 더 적합하다.

## 8. 비교: CompletableFuture vs StructuredTaskScope

| 특성 | CompletableFuture | StructuredTaskScope |
|------|-------------------|---------------------|
| Java 버전 | Java 8+ (GA) | Java 21 Preview (JEP 453) |
| 생명주기 보장 | 없음 (스레드 누수 가능) | 보장 (scope 종료 시 자식 모두 정리) |
| 예외 전파 | 수동 처리 필요 (`exceptionally`, `handle`) | 자동 전파 (`throwIfFailed`) |
| 취소 전파 | 수동 처리 필요 (각 Future에 개별 cancel) | 자동 취소 (ShutdownOnFailure/Success) |
| 코드 가독성 | 콜백 체인으로 인해 복잡 | 동기 코드 형태로 직관적 |
| 스레드 모델 | 플랫폼 스레드 풀 기반 | Virtual Thread 기반 |
| 컨텍스트 전파 | 별도 처리 필요 | ScopedValue로 자동 상속 |
| 프로덕션 사용 | 가능 (GA) | 주의 필요 (Preview: API 변경 가능) |

`CompletableFuture`는 복잡한 비동기 파이프라인 구성에 여전히 유용하다. `StructuredTaskScope`는 "여러 태스크를 동시에 실행하고 모두 기다린다"는 단순하고 명확한 패턴에 훨씬 읽기 쉬운 코드를 제공한다.

Preview Feature이므로 컴파일과 실행 시 `--enable-preview` 플래그가 필요하며, Java 버전 간 API가 변경될 수 있다는 점을 감안하고 사용해야 한다.
