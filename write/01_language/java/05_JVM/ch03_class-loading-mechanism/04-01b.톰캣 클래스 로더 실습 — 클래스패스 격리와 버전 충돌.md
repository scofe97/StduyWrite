# 톰캣 클래스 로더 실습 — 클래스패스 격리와 버전 충돌
---
> [04-01a](./04-01a.톰캣%20클래스%20로더%20실습%20—%20로더가%20다르면%20타입이%20다르다.md)가 "로더가 다르면 *타입* 이 다르다"를 보였다면, 이 실습은 한 발 더 나아갑니다. 같은 이름(`com.acme.Greeter`)의 클래스를 *내용이 다른 두 버전* 으로 두 로더에 따로 올려, **버전 격리가 어떻게 가능한지** 와 **그 격리가 깨질 때 나는 `NoSuchMethodError`** 를 눈으로 봅니다.

읽고 나면 톰캣 두 웹앱이 같은 라이브러리의 다른 버전을 충돌 없이 쓰는 원리를 설명합니다. 그리고 "클래스는 있는데 그 메서드만 없다"는 오류가 왜 버전 충돌의 신호인지 말합니다.



## 1. 무엇을 증명하는가

04-01a 에서 본 규칙은 "클래스 동일성 = 이름 + 로더"였습니다. 이 실습은 그 규칙의 *실무적 결과* 두 가지를 증명합니다.

| 국면 | 보여주는 것 |
|------|------------|
| 격리 | 같은 이름 `Greeter` 가 로더별로 *다른 버전* 으로 공존한다(좋은 면) |
| 충돌 | 한 로더는 *한 버전* 에 고정돼, 다른 버전의 메서드를 부르면 `NoSuchMethodError`(나쁜 면) |

이 둘은 같은 동전의 양면입니다. 로더 분리가 버전 공존을 가능하게 하지만, 잘못 섞이면 런타임에 터집니다. 톰캣 두 웹앱이 `gson 2.8` 과 `gson 2.10` 을 각자 쓰는 상황이 정확히 이 실습입니다.



## 2. 준비물 — 같은 이름, 다른 두 버전

핵심 장치는 **같은 FQCN `com.acme.Greeter` 를 내용이 다른 두 버전으로 따로 컴파일** 해 두는 것입니다.

```java
// libsrc/v1 — hello() 만 있다
public class Greeter {
    public String version() { return "v1"; }
    public String hello()   { return "hello from " + version(); }
}
```

```java
// libsrc/v2 — hello() 에 더해 hi() 가 추가됐다
public class Greeter {
    public String version() { return "v2"; }
    public String hello()   { return "hello from " + version(); }
    public String hi()      { return "hi from " + version(); }   // ← v1 에는 없음
}
```

두 `Greeter` 는 이름이 완전히 같지만 v2 에만 `hi()` 가 있습니다. 이 차이가 뒤에서 `NoSuchMethodError` 를 만드는 씨앗입니다. 두 버전은 *서로 다른 디렉터리* 에 컴파일됩니다. 같은 이름 클래스 둘을 한 폴더에 둘 수는 없기 때문입니다(컴파일 충돌).

### 폴더 구조로 보기

말로 된 "별도 디렉터리"를 실제 폴더로 옮기면 이렇습니다. 실험 2 에 관계된 부분만 추렸습니다.

```text
ch03-classloader/
├── src/main/java/com/acme/lab/      ← ① 평범한 소스 (앱 클래스패스에 올라감)
│   ├── Exp2_ClasspathIsolation.java ·   실험 2 드라이버(main)
│   └── LabPaths.java                ·   산출물 경로를 알려주는 도우미
│
├── libsrc/                          ← ② 앱 클래스패스에 올리면 안 되는 소스
│   ├── v1/com/acme/Greeter.java     ·   같은 이름, v1 (hi() 없음)
│   └── v2/com/acme/Greeter.java     ·   같은 이름, v2 (hi() 있음)
│
└── build/labout/                    ← ③ 빌드가 ②를 컴파일해 만드는 "별도 산출물"
    ├── v1/com/acme/Greeter.class    ·   loaderA 가 이걸 로딩
    └── v2/com/acme/Greeter.class    ·   loaderB 가 이걸 로딩
```

### 처음 볼 때 헷갈리기 쉬운 세 가지

이 구조에는 처음 보면 막히는 지점이 셋 있습니다. 미리 풀어 둡니다.

**`src` 와 `libsrc` 는 뭐가 다른가.** 

- 둘 다 소스 폴더지만 빌드가 다루는 방식이 정반대입니다. `src/main/java` 는 Gradle 표준이라 *자동* 으로 컴파일돼 앱 클래스패스에 올라갑니다. `libsrc` 는 Gradle 이 모르는 일반 폴더라 *그냥 파일* 일 뿐이고, 앱 클래스패스에 올라가지 않습니다. 
- `Greeter` 를 `libsrc` 에 둔 이유가 이것입니다. 앱 클래스패스에 올라가면 앱 로더가 먼저 로딩해 버려 격리 실험이 깨지고, 같은 이름 두 버전을 표준 소스 한 곳에 둘 수도 없습니다.

**`LabPaths.dir("v1")` 은 `libsrc` 가 아니라 `build/labout/v1` 을 가리킨다.** 

- `libsrc/v1` 은 `.java` *소스* 이고, `URLClassLoader` 가 읽는 것은 컴파일된 `.class` 입니다. 그래서 실험이 로딩하는 곳은 컴파일 결과인 `build/labout/v1` 입니다.
-  `libsrc` 는 그것을 만들기 위한 *재료* 일 뿐, 실행 시에는 쓰이지 않습니다.

```text
libsrc/v1/Greeter.java  ──(빌드가 컴파일)──>  build/labout/v1/Greeter.class  ──(실험이 로딩)
   재료(소스)                                    실행이 읽는 산출물
```

**`build/labout/<이름>` 은 자동이 아니라 빌드가 지정한 위치다.** Gradle 이 `libsrc` 를 보고 알아서 `labout` 에 넣어 주는 게 아닙니다. `build.gradle.kts` 에 "이 소스(`libsrc/v1`)를 저 위치(`build/labout/v1`)로 컴파일하라"고 *직접 적은* 규칙의 결과입니다. 경로 이름도 임의로 붙인 것이라, 다르게 적었으면 다른 곳에 갔습니다.

> 위 세 가지의 빌드 쪽 자세한 설명(소스셋·커스텀 컴파일 태스크)은 [06_Build 07-01 소스셋과 커스텀 태스크](../../06_Build/07-01.소스셋과%20커스텀%20태스크%20—%20표준%20밖%20소스%20다루기.md)에서 다룹니다. 이 글에서는 "v1·v2 가 각각 `build/labout/v1`·`build/labout/v2` 에 따로 있다"만 알면 됩니다.



## 3. 두 로더로 따로 로딩하기

```java
URL v1 = LabPaths.dir("v1");   // build/labout/v1
URL v2 = LabPaths.dir("v2");

URLClassLoader loaderA = new URLClassLoader(new URL[]{v1}, null);  // v1 만
URLClassLoader loaderB = new URLClassLoader(new URL[]{v2}, null);  // v2 만

Class<?> gA = loaderA.loadClass("com.acme.Greeter");
Class<?> gB = loaderB.loadClass("com.acme.Greeter");
```

04-01a 와 다른 점이 둘 있습니다.

- **평범한 `URLClassLoader` 를 씁니다.** 04-01a 의 `ChildFirstClassLoader`(자식 우선)가 아닙니다. 이 실습은 "위임 순서"가 아니라 "클래스패스가 다르면 격리된다"는 더 단순한 사실을 보이는 것이라, child-first 가 필요 없습니다.
- **둘째 인자가 `null` 입니다.** 부모를 부트스트랩 로더로 둡니다. 보통은 앱 로더를 부모로 주지만, 일부러 끊었습니다. 앱 로더를 부모로 주면 `com.acme.Greeter` 가 앱 클래스패스에도 있을 때 표준 위임이 *부모 것* 을 먼저 집어 v1/v2 격리가 깨질 수 있기 때문입니다. `null` 로 두면 각 로더가 *오직 자기 디렉터리* 에서만 `Greeter` 를 찾습니다.

`loaderA` 는 `v1` 디렉터리만, `loaderB` 는 `v2` 디렉터리만 봅니다. 톰캣 두 웹앱이 각자 `WEB-INF/lib` 만 보는 상황과 같습니다.



## 4. 격리 확인 — 같은 이름, 다른 타입, 다른 동작

실측 출력입니다.

```text
gA == gB ? false                    ← 이름 같아도 다른 타입
gA 출처 = .../build/labout/v1/
gB 출처 = .../build/labout/v2/
A.version() = v1
B.version() = v2
```

세 가지가 격리를 증명합니다.

1. **`gA == gB` 가 `false`** 입니다. `Class` 객체 비교인데, 같은 이름 `com.acme.Greeter` 를 JVM 이 서로 다른 타입으로 봅니다. 04-01a 의 "이름 + 로더" 규칙이 여기서도 그대로입니다.
2. **출처(`getCodeSource()`)가 다릅니다.** `gA` 는 `v1/`, `gB` 는 `v2/` 에서 왔습니다. 클래스패스가 다르니 다른 디렉터리에서 바이트를 읽었습니다.
3. **`version()` 결과가 v1·v2 로 갈립니다.** 타입만 다른 게 아니라 동작도 다릅니다. 진짜 두 버전이 한 JVM 에 공존하는 것입니다.

이것이 톰캣 웹앱 격리의 좋은 면입니다. 로더를 나눴기에 같은 이름의 다른 버전이 충돌 없이 공존합니다.



## 5. 핵심 규칙 — 한 로더는 한 버전에 고정된다

> 다음 국면으로 가기 전에 짚을 규칙이 있습니다. 
>
> **하나의 로더는 같은 FQCN 을 단 한 번만 정의합니다.** 한 로더에게 `com.acme.Greeter` 를 두 번 요청해도, 두 번째부터는 *처음 정의한 그 클래스* 를 그대로 돌려줍니다.

그래서 "한 로더가 같은 이름의 두 버전을 동시에 가진다"는 상황은 만들 수 없습니다. 한 로더의 클래스패스에 v1·v2 를 둘 다 넣어도, 로더는 *먼저 걸리는 하나* 만 정의하고 나머지는 그림자에 가립니다. 버전이 다르든 말든 신경 쓰지 않고 먼저 찾은 것을 씁니다. 이것이 실무의 "JAR 지옥" — 클래스패스에 두 버전이 섞여 어느 것이 이길지 모호한 상황입니다.

정리하면, 실습 4장에서 두 버전을 공존시킬 수 있었던 *유일한* 이유는 **로더를 둘로 나눴기 때문** 입니다. 로더가 하나였다면 공존 자체가 불가능합니다.

```text
loaderA (v1만)  →  Greeter(v1)  ┐
                                ├ 로더가 다르니 둘 다 존재
loaderB (v2만)  →  Greeter(v2)  ┘
```

그리고 이 "로더가 버전을 고정한다"가 다음의 `NoSuchMethodError` 로 바로 이어집니다.



## 6. 버전 충돌 — `NoSuchMethodError`

```java
try {
    Method hi = gA.getMethod("hi");   // gA = v1 Greeter. v1 에는 hi() 가 없다
    hi.invoke(a);
} catch (NoSuchMethodException e) {
    System.out.println("NoSuchMethodException: " + e.getMessage());
}

Method hiB = gB.getMethod("hi");      // gB = v2 Greeter. v2 에는 있다
System.out.println("B.hi() = " + hiB.invoke(b));
```

`gA`(v1 로더가 정의한 Greeter)에서 `hi()` 를 찾으면 없습니다. 5장의 규칙대로 `gA` 는 *영원히 v1* 이라, v2 의 `hi()` 를 빌려올 길이 없습니다. 실측 출력입니다.

```text
NoSuchMethodException: com.acme.Greeter.hi()
→ v1 Greeter 에는 hi() 가 없다. v2 라고 착각하고 부르면 깨진다.
B.hi() = hi from v2  (v2 에는 있음)
```

여기서 오류의 진짜 의미가 드러납니다.

> "클래스는 분명히 있는데(`Greeter` 는 로딩됨), 그 메서드만 없다(`hi()`)."

- 이것이 버전 충돌의 전형적 신호입니다. `ClassNotFoundException`("클래스 자체가 없음")과 다릅니다. 클래스는. 있는데 메서드가 없다면, "내가 의도한 버전과 *다른 버전* 이 그 로더에 잡혔다"를 먼저 의심합니다.

> 실습은 리플렉션이라 `NoSuchMethodException` 이 나지만, 컴파일된 코드가 직접 `greeter.hi()` 를 부르면 `NoSuchMethodError` 가 납니다. 둘은 "그 메서드가 런타임 클래스에 없다"는 같은 상황의 다른 표현입니다.

### 왜 무서운가 — 컴파일은 통과, 런타임에 터진다

`NoSuchMethodError` 는 컴파일 시점에 안 잡힙니다.

```text
컴파일 시점:  v2 헤더로 컴파일 → hi() 존재 → 통과
런타임 시점:  클래스패스에 v1 이 먼저 → hi() 없음 → NoSuchMethodError
```

코드를 v2 기준으로 짜서 컴파일은 통과했는데, 런타임 클래스패스에 v1 이 잡힌 경우입니다. 톰캣에서는 웹앱이 라이브러리 v2 를 `WEB-INF/lib` 에 넣었는데 Common 계층의 v1 이 위임 순서로 먼저 잡힐 때 이렇게 납니다. 04-01 §4 오류 표의 `NoSuchMethodError` 항목이 이 상황입니다.



## 7. 정리 — 격리의 두 얼굴

| 국면 | 무엇을 보였나 | 원인 |
|------|--------------|------|
| 격리 | 같은 이름 `Greeter` 가 v1/v2 다른 타입·동작으로 공존 | 로더를 둘로 나눔 |
| 충돌 | v1 로더에서 v2 메서드 호출 → `NoSuchMethodError` | 한 로더는 한 버전에 고정 |

04-01a 가 "로더가 다르면 타입이 다르다"였다면, 이 실습은 그 사실의 *실무적 그림자* 입니다. 격리는 버전 공존을 가능하게 하지만(좋은 면), 잘못 섞이면 `NoSuchMethodError` 로 터집니다(나쁜 면). 핵심 한 줄은 **"한 로더는 한 버전에 고정된다"** 이고, 버전을 공존시키려면 *반드시 로더를 나눠야* 합니다.

```bash

```
