# 다중 디스패치와 가상 메서드 테이블
---
> §8.3.2 후반을 한 줄로 압축하면 — **디스패치는 정적/동적의 종량과 단일/다중의 종수 두 축으로 나뉘며, 자바는 정적 다중 디스패치 + 동적 단일 디스패치 언어입니다.** 그리고 동적 디스패치를 매번 탐색하면 느리므로, 가상 머신은 클래스마다 *가상 메서드 테이블(vtable)*을 두어 고정 인덱스로 빠르게 호출합니다. 부수적인 함정으로 "필드는 오버라이딩되지 않는다"가 있습니다.

이 글을 읽고 나면 단일/다중 디스패치를 구분하고 자바가 왜 "정적 다중 + 동적 단일"인지 설명하며, vtable이 동적 호출을 어떻게 가속하는지와 필드가 오버라이딩되지 않는 이유를 그림 없이 짚을 수 있습니다.


## 진입 — 디스패치를 두 축으로 나누면

> [앞 글](./03-02.메서드%20호출%20—%20해석과%20정적·동적%20디스패치.md)의 정적/동적 구분은 *언제* 고르느냐의 축이었습니다. 여기에 *몇 개의 타입을 보고* 고르느냐의 축을 더하면 디스패치 전체 지도가 완성됩니다.

앞 글에서 디스패치를 정적(컴파일 때)과 동적(실행 중)으로 나눴습니다. 여기에 또 하나의 축이 있습니다. 호출 대상을 고를 때 *몇 개의 정보(타입)를 기준으로 삼는가*입니다. 하나만 보면 단일 디스패치, 둘 이상을 보면 다중 디스패치입니다. 두 축을 곱하면 디스패치의 전체 분류가 나옵니다.

![단일/다중 디스패치와 vtable](_assets/03-03-multidispatch-vtable.svg)


## 1. 단일 디스패치와 다중 디스패치

> 단일 디스패치는 하나의 정보로, 다중 디스패치는 둘 이상의 정보로 메서드를 고릅니다. 자바는 정적일 때 다중, 동적일 때 단일입니다.

종수(宗數)에 따른 분류는 이렇습니다.

1. *단일 디스패치(single dispatch)*는 *하나의 정보*만으로 호출 대상을 고릅니다.
2. *다중 디스패치(multiple dispatch)*는 *둘 이상의 정보*를 함께 보고 고릅니다.

자바의 정적 디스패치(오버로딩)를 보면, 컴파일러는 *수신자의 타입*과 *인자의 타입* 둘 다를 봐야 어느 오버로딩 버전인지 정할 수 있습니다. 두 가지를 보므로 *다중 디스패치*입니다.

반면 자바의 동적 디스패치(오버라이딩)는 실행 시점에 *수신자 객체의 실제 타입 하나*만 보고 메서드를 고릅니다. 인자의 실제 타입은 고려하지 않습니다. 하나만 보므로 *단일 디스패치*입니다.

결론적으로 **자바는 정적 다중 디스패치 + 동적 단일 디스패치 언어**입니다. 이 한 문장이 §8.3.2가 디스패치를 두 축으로 정리한 결론입니다.


## 2. 가상 메서드 테이블 — 동적 호출의 가속

> 동적 디스패치를 매번 수신자 타입부터 부모로 탐색하면 느립니다. 가상 머신은 클래스마다 vtable을 두어 *고정 인덱스*로 메서드를 한 번에 찾습니다.

[앞 글의 invokevirtual 4단계 탐색](./03-02.메서드%20호출%20—%20해석과%20정적·동적%20디스패치.md)은 수신자의 실제 타입부터 부모로 거슬러 올라가며 메서드를 찾습니다. 이 탐색을 *호출할 때마다* 반복하면 성능이 떨어집니다. 그래서 가상 머신은 *가상 메서드 테이블(virtual method table, vtable)*을 메서드 영역에 두어 이를 가속합니다.

vtable은 클래스마다 하나씩 있으며, 그 클래스가 가진 *모든 가상 메서드의 실제 진입점*을 인덱스 배열로 담습니다. 동작 규칙은 다음과 같습니다.

1. 자식 클래스가 부모의 메서드를 *오버라이드하지 않았으면*, 자식 vtable의 그 자리는 부모 구현을 그대로 가리킵니다.
2. 자식이 *오버라이드했으면*, 그 자리만 자식의 구현으로 바뀝니다.
3. 같은 메서드는 부모·자식 vtable에서 *같은 인덱스*를 가집니다.

같은 메서드가 어느 클래스에서든 같은 인덱스를 갖는다는 점이 핵심입니다. 덕분에 `invokevirtual sayHello`는 매번 탐색하지 않고, *수신자의 vtable에서 고정된 인덱스로 한 번에 점프*합니다. 수신자가 `Man`이면 `Man` vtable의 그 인덱스가 `Man.sayHello`를, `Human`이면 `Human.sayHello`를 가리키므로, 한 번의 인덱싱으로 올바른 메서드에 닿습니다. vtable은 클래스 로딩의 연결 단계에 초기화됩니다.


## 3. 필드는 오버라이딩되지 않는다

> 메서드는 동적 디스패치로 실제 타입을 따르지만, 필드는 *정적 타입*으로 접근됩니다. 같은 이름의 필드는 오버라이딩이 아니라 가려짐(hiding)입니다.

동적 디스패치는 *메서드*에만 적용됩니다. 필드는 다릅니다. 부모와 자식에 같은 이름의 필드가 있으면, 접근하는 변수의 *정적 타입*에 따라 어느 필드가 보일지 정해집니다. 이를 오버라이딩이 아니라 *가려짐(field hiding)*이라 부릅니다.

```java
public class FieldHasNoPolymorphic {
    static class Father {
        public int money = 1;
        public Father() {
            money = 2;
            showMeyMoney();   // 생성자에서 오버라이드된 메서드 호출
        }
        public void showMeyMoney() { System.out.println("I am Father, I have $" + money); }
    }
    static class Son extends Father {
        public int money = 3;
        public Son() {
            money = 4;
            showMeyMoney();
        }
        @Override
        public void showMeyMoney() { System.out.println("I am Son, I have $" + money); }
    }
    public static void main(String[] args) {
        Father guy = new Son();
        System.out.println("This guy has $" + guy.money);   // 어느 money?
    }
}
```

출력은 다음과 같습니다.

```
I am Son, I have $0
I am Son, I have $4
This guy has $2
```

두 가지를 짚어야 합니다. 첫째, 메서드 `showMeyMoney()`는 *동적 디스패치*라, `Father` 생성자에서 호출해도 실제 타입 `Son`의 버전이 실행됩니다. 그래서 `I am Son`이 두 번 찍힙니다. 첫 호출 시점에는 `Son`의 `money`가 아직 초기화 전이라 `0`이 보입니다.

둘째, 마지막 줄 `guy.money`는 *필드*라 정적 타입 `Father`를 따릅니다. `guy`의 정적 타입이 `Father`이므로 `Father.money`인 `2`가 보입니다. 같은 객체인데 변수의 정적 타입에 따라 다른 필드가 보이는 것이 필드 가려짐입니다. 메서드는 실제 타입을, 필드는 정적 타입을 따른다는 비대칭을 기억해야 합니다.


## 4. 면접 대비 요약

> 핵심은 "자바=정적 다중+동적 단일", "vtable 고정 인덱스 가속", "필드는 오버라이딩이 아니라 가려짐"입니다.

### 한 줄 정의

다중 디스패치는 둘 이상의 타입으로 메서드를 고르는 방식이고, 가상 메서드 테이블은 클래스마다 가상 메서드의 진입점을 고정 인덱스로 담아 동적 호출을 가속하는 자료구조를 말합니다.

### 핵심 포인트 3가지

1. 자바는 정적 디스패치(오버로딩)에서 수신자와 인자 타입을 모두 보는 다중 디스패치, 동적 디스패치(오버라이딩)에서 수신자 타입만 보는 단일 디스패치입니다.
2. vtable은 같은 메서드를 부모·자식에서 같은 인덱스로 두어, `invokevirtual`이 탐색 없이 고정 인덱스로 점프하게 합니다.
3. 메서드는 실제 타입을 따라 오버라이딩되지만, 필드는 정적 타입을 따르는 가려짐이라 다형성이 적용되지 않습니다.

### 면접에서 받을 만한 질문

1. 자바가 "정적 다중 + 동적 단일 디스패치 언어"라는 말의 의미는 무엇입니까?
2. 가상 메서드 테이블이 동적 호출을 어떻게 가속합니까?
3. 부모 생성자에서 오버라이드된 메서드를 호출하면 어느 버전이 실행됩니까?

> 세 질문에 *먼저 자답한 뒤* 아래 §정답으로 내려갑니다.


## 정답 (자답 후 펼치기)

> 위 §면접에서 받을 만한 질문의 3개에 *먼저 자답한 뒤* 아래를 읽으세요.

### 정답 1 — 정적 다중 + 동적 단일

정적 디스패치(오버로딩)는 컴파일러가 *수신자 타입과 인자 타입 둘 다*를 보고 버전을 고르므로 다중 디스패치입니다. 동적 디스패치(오버라이딩)는 실행 중 *수신자의 실제 타입 하나*만 보고 고르므로 단일 디스패치입니다. 그래서 자바는 정적 다중 + 동적 단일 디스패치 언어로 분류됩니다.

### 정답 2 — vtable의 가속

vtable은 클래스마다 가상 메서드의 진입점을 담은 배열이며, 같은 메서드를 부모·자식에서 *같은 인덱스*에 둡니다. 자식이 오버라이드한 메서드만 그 자리가 자식 구현으로 바뀝니다. 덕분에 `invokevirtual`은 수신자 타입부터 부모로 탐색하지 않고, 수신자의 vtable에서 고정 인덱스로 한 번에 점프해 올바른 메서드에 닿습니다.

### 정답 3 — 부모 생성자의 오버라이드 메서드 호출

*실제 타입의 버전*이 실행됩니다. 메서드는 동적 디스패치라, `Father` 생성자에서 `showMeyMoney()`를 호출해도 객체의 실제 타입이 `Son`이면 `Son.showMeyMoney()`가 실행됩니다. 다만 그 시점에는 `Son`의 필드가 아직 초기화 전이라, 자식 필드 값이 기본값(`0`)으로 보일 수 있습니다.
