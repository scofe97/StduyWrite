# 벤치마크 실전 — jmh와 공통 예제 코드
> jmh는 마이크로벤치마크의 함정을 막아 주는 프레임워크이고, 이 책의 예제들은 가상 주가를 다루는 공통 메소벤치마크를 씁니다

[앞 두 편](./02-01.무엇을%20측정할까%20—%20벤치마크%20종류와%20성능%20지표.md)이 성능 테스트의 원칙이었다면, 이 편은 실전 도구입니다. 책의 일부 예제는 jmh로 마이크로벤치마크를 만들고, 많은 예제는 메소벤치마크(실제 앱보다 단순하지만 JVM 기능을 충분히 자극하는 테스트)의 변형을 씁니다. 이 편은 jmh 벤치마크를 어떻게 작성하는지와, 뒤 장들에서 반복 쓰이는 공통 예제 코드를 짚습니다.



## 1. jmh — Java Microbenchmark Harness
> 벤치마크 작성 프레임워크로, micro 함정을 줄여 주지만 만능은 아니며 무엇을 재는지 여전히 깊이 이해해야 합니다

jmh는 벤치마크 작성 프레임워크를 제공하는 클래스 집합입니다. jmh의 m은 원래 microbenchmark를 뜻했지만, 지금은 nano/micro/milli/macro 벤치마크에 두루 적합하다고 광고합니다(전형적 용도는 여전히 작은 벤치마크입니다). Java 9와 함께 발표됐지만 특정 Java 릴리스에 묶이지 않고, JDK 안에 jmh를 지원하는 도구는 없습니다. jmh 클래스 라이브러리는 JDK 8 이상과 호환됩니다. jmh는 좋은 벤치마크 작성의 불확실성을 일부 덜어 주지만 은탄환은 아닙니다. 무엇을 벤치마크하는지, 좋은 벤치마크 코드를 어떻게 쓰는지는 여전히 이해해야 합니다.

처음부터 작성할 수도 있지만, jmh가 제공하는 main 클래스로 시작해 벤치마크 코드만 쓰는 편이 쉽습니다. 기본 방법은 Maven으로 프로젝트를 만드는 것입니다.

```
$ mvn archetype:generate \
      -DinteractiveMode=false \
      -DarchetypeGroupId=org.openjdk.jmh \
      -DarchetypeArtifactId=jmh-java-benchmark-archetype \
      -DgroupId=net.sdo \
      -DartifactId=string-intern-benchmark \
      -Dversion=1.0
```

이는 *string-intern-benchmark* 디렉토리에 Maven 프로젝트와 골격 벤치마크 클래스 `MyBenchmark`를 만듭니다. jmh는 `Benchmark`라는 어노테이션을 찾아 어느 클래스를 테스트할지 정하므로, 클래스 이름은 자유입니다. `String.intern()` 성능을 테스트하려면 첫 벤치마크 메서드는 이렇습니다.

```java
import org.openjdk.jmh.annotations.Benchmark;
import org.openjdk.jmh.infra.Blackhole;

public class MyBenchmark {

    @Benchmark
    public void testIntern(Blackhole bh) {
        for (int i = 0; i < 10000; i++) {
            String s = new String("String to intern " + i);
            String t = s.intern();
            bh.consume(t);
        }
    }
}
```

`Blackhole`은 마이크로벤치마크의 핵심 함정 하나를 다룹니다. 연산 결과를 쓰지 않으면 컴파일러가 그 연산을 최적화로 제거해도 되므로, 값을 `Blackhole`의 `consume()`에 넘겨 쓰이게 만듭니다. 이 예에서는 사실 `Blackhole`이 꼭 필요하진 않습니다. `intern()`의 부작용(전역 해시 테이블에 문자열 삽입)에 관심이 있고 그 상태 변경은 반환값을 안 써도 제거되지 않기 때문입니다. 그래도 매번 필요한지 따지기보다 계산값을 소비하는 습관을 들이는 게 낫습니다.



## 2. jmh 실행 — trial, fork, iteration
> 포크된 JVM(trial) 안에 warm-up·measurement iteration이 있고, 이를 여러 fork 반복해 재현성과 신뢰구간을 얻습니다

컴파일·실행은 `mvn package` 후 `java -jar target/benchmarks.jar`로 합니다. 출력에서 jmh가 함정을 어떻게 피하는지 보입니다.

```
$ java -jar target/benchmarks.jar
# Warmup: 5 iterations, 10 s each
# Measurement: 5 iterations, 10 s each
# Threads: 1 thread, will synchronize iterations
# Benchmark mode: Throughput, ops/time
...
Result "net.sdo.MyBenchmark.testIntern":
  177.219 ±(99.9%) 10.140 ops/s [Average]
  (min, avg, max) = (155.286, 177.219, 207.970), stdev = 13.537
  CI (99.9%): [167.078, 187.359] (assumes normal distribution)

Benchmark                Mode  Cnt    Score    Error  Units
MyBenchmark.testIntern  thrpt   25  177.219 ± 10.140  ops/s
```

실행 계층은 다음과 같습니다.

1. **iteration** — 기본 10초간 대상 메서드를 가능한 만큼 반복 실행합니다(위 예는 10초에 평균 1,772회).
2. **warm-up iteration 5회 + measurement iteration 5회** — warm-up 결과는 버리고 측정 결과만 보고합니다. warm-up이 컴파일러에 최적화 시간을 줍니다.
3. **fork(trial) 5회** — warm-up과 measurement를 묶은 한 묶음(trial)을 매번 새로 포크한 JVM에서 돌립니다. 결과 재현성을 확인하기 위해서입니다.

각 테스트는 5 warm-up + 5 measurement = 100초이고, 이를 5번 반복해 총 8분 20초입니다. 요약 출력에서 `testIntern()`은 평균 초당 177회 실행됐고, 99.9% 신뢰구간으로 통계 평균이 초당 167~187회 사이에 있다고 말할 수 있습니다. jmh가 변동이 허용 범위인지 이해하는 데 필요한 통계 분석까지 돕는 것입니다.



## 3. @Param, 구현 비교, @Setup
> @Param으로 입력 범위를 주고 여러 @Benchmark로 구현을 비교하며, @Setup으로 측정 밖 초기화를 해 캐시 히트율 같은 함정을 없앱니다

입력 범위가 필요하면 값을 하드코딩하지 않고 `@Param`을 씁니다.

```java
@Param({"1","10000"})
private int nStrings;

@Benchmark
public void testIntern(Blackhole bh) {
    for (int i = 0; i < nStrings; i++) {
        String s = new String("String to intern " + i);
        String t = s.intern();
        bh.consume(t);
    }
}
```

jmh는 두 값 모두에 대해 결과를 보고합니다. 실행 시 `-p nStrings=1,1000000`으로 코드의 하드코딩 값을 덮어쓸 수도 있습니다. 두 구현을 비교하려면 메서드를 하나 더 만들어 `@Benchmark`를 답니다. 문자열 인터닝 대신 직접 관리하는 해시 맵이 더 나은지 보는 예입니다.

```java
private static ConcurrentHashMap<String,String> map = new ConcurrentHashMap<>();
@Benchmark
public void testMap(Blackhole bh) {
    for (int i = 0; i < nStrings; i++) {
        String s = new String("String to intern " + i);
        String t = map.putIfAbsent(s, s);
        bh.consume(t);
    }
}
```

jmh는 모든 어노테이션 메서드를 같은 warm-up·measurement 과정으로 돌려 비교를 냅니다. 그러나 이대로는 함정이 남습니다. 우리가 재려던 건 1만 문자열 인터닝인데, 실제로는 1만 문자열 생성(연결) 시간 + 인터닝 시간을 재고 있고, 문자열 범위도 좁습니다. 더 깊은 문제는 캐시 히트율입니다. 직접 관리하는 ConcurrentHashMap은 테스트 중 비워지지 않아 첫 warm-up에 삽입된 뒤 측정 사이클에서 **100% 히트율**입니다. 반면 string intern 테이블의 키는 본질적으로 weak reference라, 인터닝된 문자열이 삽입 직후 스코프를 벗어나면 JVM이 항목을 언제든 지울 수 있어 히트율이 100%에 한참 못 미칩니다. 그래서 intern 테스트가 테이블을 더 자주 갱신해 더 많은 일을 합니다.

두 문제는 문자열을 정적 배열에 미리 만들어 두고 인터닝(또는 삽입)하면 사라집니다. 정적 배열이 참조를 유지해 string 테이블의 참조가 안 지워지므로, 두 테스트 모두 측정 사이클에서 100% 히트율이 되고 문자열 범위도 넓어집니다. 이 초기화는 측정 밖에서 해야 하며 `@Setup`으로 합니다.

```java
private static ConcurrentHashMap<String,String> map;
private static String[] strings;

@Setup(Level.Iteration)
public void setup() {
    strings = new String[nStrings];
    for (int i = 0; i < nStrings; i++) {
        strings[i] = makeRandomString();
    }
    map = new ConcurrentHashMap<>();
}

@Benchmark
public void testIntern(Blackhole bh) {
    for (int i = 0; i < nStrings; i++) {
        String t = strings[i].intern();
        bh.consume(t);
    }
}
```

`@Setup`의 `Level`은 메서드가 언제 실행될지 정합니다. 셋 중 하나입니다.

| Level | 실행 시점 |
|-------|-----------|
| `Level.Trial` | 벤치마크 코드 초기화 시 1회 |
| `Level.Iteration` | 각 iteration(측정 사이클) 전 |
| `Level.Invocation` | 테스트 메서드가 실행될 때마다 |

`Level.Invocation`은 jmh가 평균 호출 분석을 하기 어렵게 만들므로 드물게, 오래 걸리는 테스트 메서드에만 권합니다. 상태를 지워야 하면 짝이 되는 `@Teardown`을 씁니다. 이 예의 요지는, 단순한 마이크로벤치마크를 쓰는 데도 이만큼의 복잡함이 따른다는 것입니다.



## 4. 실행과 재현성 제어
> fork·iteration·길이를 명령행으로 조절해 신뢰구간을 좁히며, 두 기법의 차이가 작을수록 더 많은 반복이 필요합니다

올바른 마이크로벤치마크를 얻은 뒤에는, 측정 대상에 통계적으로 의미 있도록 돌려야 합니다. 기본값으로 jmh는 대상 메서드를 10초씩 5 warm-up + 5 measurement iteration 돌리고, 이를 5 trial 반복해 신뢰구간을 계산합니다. 앞 예의 99.9% 신뢰구간 범위는 약 10%였는데, 다른 벤치마크와 비교하기엔 충분할 수도 아닐 수도 있습니다. 측정 iteration과 trial이 적으면 범위가 넓어 어느 쪽이 나은지 단정하기 어렵습니다.

```
Benchmark               (nStrings)   Mode  Cnt    Score     Error  Units
MyBenchmark.testIntern       10000  thrpt    4  233.359 ±  95.304  ops/s
MyBenchmark.testMap          10000  thrpt    4  354.341 ± 166.491  ops/s
```

이를 각 10 iteration으로 늘리면 범위가 분리되어, map 기법이 우월하다고 확신할 수 있습니다(적어도 100% 캐시 히트율과 1만 개 불변 문자열 테스트에 한해서).

```
MyBenchmark.testIntern       10000  thrpt   20  172.738 ± 29.058  ops/s
MyBenchmark.testMap          10000  thrpt   20  305.618 ± 22.754  ops/s
```

몇 번의 iteration·fork·실행 길이가 충분한지에 대한 확고한 규칙은 없습니다. 두 기법의 차이가 작으면 훨씬 많은 iteration·trial이 필요하고, 그렇게 가깝다면 차라리 영향이 더 큰 다른 것을 보는 게 나을 수도 있습니다. 여기서 art가 science에 끼어듭니다. 이 변수들은 명령행 인자로 제어합니다.

| 옵션 | 의미 (기본값) |
|------|---------------|
| `-f 5` | 실행할 fork(trial) 수 (5) |
| `-wi 5` | trial당 warm-up iteration 수 (5) |
| `-i 5` | trial당 measurement iteration 수 (5) |
| `-r 10` | 각 iteration 최소 길이(초) — 대상 메서드 길이에 따라 더 길 수도 |

이 값을 키우면 대개 결과 변동이 줄어 원하는 신뢰 범위에 이르고, 줄이면 테스트 시간이 짧아집니다. 요약하면, jmh는 마이크로벤치마크의 요구사항을 제대로 다루도록 돕는 프레임워크이자 하네스이며, 측정 코드를 깊이 고민하는 일을 대체하지는 않습니다.



## 5. 공통 예제 코드 — 가상 주가 시스템
> 대부분의 예제는 무작위 데이터로 가상 주가를 다루는 메소벤치마크이며, BigDecimal 제곱근(바빌로니아 방법)이 "더 나은 알고리즘"의 예시이기도 합니다

이 책의 많은 예제는 어떤 주식의 날짜 범위별 "역사적" 고가·저가와 그 기간 표준편차를 계산하는 샘플 애플리케이션에 기반합니다. 역사적에 따옴표를 친 이유는 가격과 심볼이 모두 무작위 생성된 가공 데이터이기 때문입니다. 기본 객체는 주어진 날의 가격 범위와 옵션 가격을 담는 `StockPrice`입니다.

```java
public interface StockPrice {
    String getSymbol();
    Date getDate();
    BigDecimal getClosingPrice();
    BigDecimal getHigh();
    BigDecimal getLow();
    BigDecimal getOpeningPrice();
    boolean isYearHigh();
    boolean isYearLow();
    Collection<? extends StockOptionPrice> getOptions();
}
```

샘플은 보통 일정 기간(예: 1년 또는 25년)의 가격 컬렉션을 다루는 `StockPriceHistory`를 씁니다. 구현은 DB에서 가격을 로드하도록 설계됐고 그 기능은 11장 예제에서 쓰입니다. 다만 예제 실행을 쉽게 하려고, 대부분은 시리즈에 무작위 데이터를 만드는 mock entity manager를 씁니다. 즉 대부분은 모듈 수준 메소벤치마크이고, 실제 성능은 전체 앱을 돌릴 때(11장)에야 알 수 있습니다.

이 때문에 여러 예제가 사용하는 난수 생성기 성능에 의존하는데, 이는 의도된 것입니다. 임의의 무언가의 성능을 재는 것이 예제의 목표이고 난수 생성기 성능이 거기 맞기 때문입니다(난수 생성 시간을 포함하면 왜곡되는 마이크로벤치마크와는 다릅니다). 예제는 모든 데이터를 저장하는 `BigDecimal`에도 크게 의존합니다. 통화 데이터를 primitive `double`로 저장하면 반올림이 문제가 되므로 `BigDecimal`이 표준 선택입니다. 이 선택은 표준편차 계산에 "비즈니스 로직" 같은 긴 계산이 일어나게 해 예제 작성에도 유용합니다. 표준편차는 `BigDecimal`의 제곱근을 알아야 하는데, 표준 Java API에는 그런 루틴이 없어 예제는 다음 메서드를 씁니다.

```java
public static BigDecimal sqrtB(BigDecimal bd) {
    BigDecimal initial = bd;
    BigDecimal diff;
    do {
        BigDecimal sDivX = bd.divide(initial, 8, RoundingMode.FLOOR);
        BigDecimal sum = sDivX.add(initial);
        BigDecimal div = sum.divide(TWO, 8, RoundingMode.FLOOR);
        diff = div.subtract(initial).abs();
        diff.setScale(8, RoundingMode.FLOOR);
        initial = div;
    } while (diff.compareTo(error) > 0);
    return initial;
}
```

이것은 제곱근을 추정하는 바빌로니아 방법(Babylonian method) 구현입니다. 가장 효율적인 구현은 아닙니다. 특히 초기 추정값을 더 좋게 잡으면 반복을 줄일 수 있습니다. 이는 의도된 것으로, 계산이 시간을 좀 쓰게 해 비즈니스 로직을 흉내 냅니다. 동시에 1장의 요지, **Java 코드를 빠르게 하는 더 나은 길은 흔히 Java 튜닝·코딩이 아니라 더 나은 알고리즘을 쓰는 것**임을 보여 줍니다.

표준편차·평균·히스토그램은 파생 값이라, 예제마다 eager(엔티티 매니저에서 로드할 때)하게 또는 lazy(메서드 호출 시)하게 계산됩니다. 옵션 값도 eager·lazy로 가져올 수 있습니다. 이 인터페이스들은 Java REST 애플리케이션에 자연스럽게 맞습니다. 사용자가 심볼·날짜 범위를 파라미터로 호출하면, JAX-RS 호출이 입력을 파싱하고 내장 JPA 빈으로 데이터를 가져와 응답을 전달합니다.

## 자주 받는 오해
> jmh에 코드를 넣으면 좋은 벤치마크가 된다고 믿기 쉽지만, 무엇을 재는지는 여전히 직접 검증해야 합니다

1. "jmh에 코드를 넣으면 좋은 벤치마크가 된다"고 생각하기 쉽지만, jmh 실행 시 긴 주의문이 뜨는 이유가 여기 있습니다. jmh는 함정을 줄여 줄 뿐이고, 무엇을 재는지·기대대로 재고 있는지는 직접 확인해야 합니다(예: intern 테스트가 실제로는 문자열 생성+인터닝을 재고 있던 것).
2. "ConcurrentHashMap 비교가 string intern을 공정하게 이긴다"고 생각하기 쉽지만, map은 테스트 중 안 비워져 100% 히트율인 반면 intern 테이블은 weak reference라 히트율이 낮아 더 많은 일을 합니다. `@Setup`으로 정적 배열에 미리 만들어 양쪽을 100% 히트율로 맞춰야 공정합니다.
3. "fork·iteration이 적어도 평균이 나오면 비교할 수 있다"고 생각하기 쉽지만, 4 iteration 같은 적은 표본은 오차 범위(±95~166)가 넓어 우열을 단정할 수 없습니다. 차이가 작을수록 더 많은 iteration·trial이 필요합니다.



## 면접에서 받을 만한 질문
1. **jmh의 Blackhole이 왜 필요합니까?** → 연산 결과를 어디서도 쓰지 않으면 JIT 컴파일러가 그 연산을 dead code로 보고 제거할 수 있습니다. `Blackhole.consume()`에 결과를 넘기면 값이 쓰인 것으로 처리되어 연산이 제거되지 않습니다. 다만 `String.intern()`처럼 전역 테이블을 바꾸는 부작용이 있으면 반환값을 안 써도 제거되지 않지만, 매번 따지기보다 계산값을 소비하는 습관을 들이는 게 안전합니다.
2. **jmh의 trial·fork·iteration은 어떤 관계입니까?** → iteration은 대상 메서드를 일정 시간(기본 10초) 반복 실행하는 단위이고, 한 trial은 warm-up iteration 5회와 measurement iteration 5회로 이뤄집니다. fork는 이 trial을 매번 새 JVM에서 5번 반복하는 것으로, JVM마다 새로 warm-up·측정해 결과 재현성을 확인합니다. 기본 설정이면 한 테스트가 8분 20초 걸리고, 결과로 평균 ops/s와 99.9% 신뢰구간을 줍니다.
3. **@Setup의 Level 세 가지는?** → `Level.Trial`은 벤치마크 초기화 시 1회, `Level.Iteration`은 각 측정 사이클 전, `Level.Invocation`은 테스트 메서드 실행마다 setup을 돌립니다. `Level.Invocation`은 jmh의 평균 호출 분석을 어렵게 하므로 오래 걸리는 메서드에만 드물게 씁니다. string intern 비교에서는 `Level.Iteration`으로 정적 배열을 미리 만들어 캐시 히트율 함정을 없앴습니다.
4. **공통 예제가 BigDecimal과 바빌로니아 제곱근을 쓰는 이유는?** → 통화 데이터를 `double`로 저장하면 반올림 문제가 생겨 `BigDecimal`이 표준입니다. 표준편차 계산에 제곱근이 필요한데 표준 API에 없어 바빌로니아 방법을 구현했고, 일부러 초기 추정을 나쁘게 둬 계산이 비즈니스 로직처럼 시간을 쓰게 했습니다. 이 선택 자체가 "Java 튜닝보다 더 나은 알고리즘이 흔히 더 효과적"이라는 1장 요지의 예시입니다.
