# OS 레벨 도구 — CPU·디스크·네트워크
> 성능 분석의 출발점은 Java가 아니라 OS 모니터링이며, CPU는 짧은 시간 높게 쓰는 것이 목표입니다

3장은 성능 분석 도구를 다룹니다. 저자는 성능 분석이 곧 가시성이고, 가시성은 곧 도구이며, 따라서 성능 튜닝은 곧 도구라고 말합니다. 2장이 측정의 중요성을 다뤘다면, 이 장은 "프로그램이 정확히 무엇을 하고 있는지" 데이터를 얻고 이해하는 법을 다룹니다.

이 노트는 그 첫 단계인 OS 레벨 도구를 봅니다. 분석의 출발점은 Java에 특화된 것이 전혀 아니라, OS에 딸린 기본 모니터링 도구입니다. Unix 계열은 `sar`(System Accounting Report)와 그 구성 도구인 `vmstat`·`iostat`·`prstat` 등이고, Windows는 그래픽 자원 모니터와 `typeperf` 같은 명령행 유틸리티입니다.

성능 테스트를 돌릴 때마다 OS에서 데이터를 모아야 합니다. 최소한 CPU·메모리·디스크 사용 정보를 모으고, 프로그램이 네트워크를 쓰면 네트워크 정보도 모읍니다. 테스트가 자동화돼 있으면 명령행 도구에 의존한다는 뜻이고(Windows에서도), 대화식으로 돌리더라도 GUI 그래프를 눈으로 보며 의미를 추측하기보다 출력을 캡처하는 명령행 도구를 쓰는 게 낫습니다. 출력은 나중에 분석할 때 언제든 그래프로 그릴 수 있습니다.



## 1. CPU 사용률 — 짧게, 높게
> CPU는 user time과 system time으로 나뉘며, 가능한 짧은 시간 동안 100%로 쓰는 것이 목표입니다

CPU 사용률은 보통 두 범주로 나뉩니다. **user time**은 CPU가 애플리케이션 코드를 실행하는 시간 비율이고, **system time**(Windows는 privileged time)은 커널 코드를 실행하는 시간 비율입니다. system time은 애플리케이션과 관련됩니다. 애플리케이션이 I/O를 하면 커널이 디스크에서 파일을 읽거나 버퍼 데이터를 네트워크에 쓰는 코드를 실행합니다. 기반 시스템 자원을 쓰는 무엇이든 애플리케이션의 system time을 늘립니다.

성능의 목표는 **가능한 짧은 시간 동안 CPU 사용률을 가능한 높게** 끌어올리는 것입니다. 다소 반직관적으로 들립니다. CPU가 100%라 책상 앞 컴퓨터가 버벅대는 걸 본 적이 있을 테니까요. 그러나 CPU 사용률 숫자의 의미를 생각해 봐야 합니다. 첫째, CPU 사용률은 구간 평균값입니다(5초·30초, 때로 1초). 한 프로그램의 평균 CPU가 실행 10분 동안 50%라면, CPU가 절반은 놀았다는 뜻입니다. 놀고 있는 부분(과 다른 병목)을 없애게 프로그램을 재구성하면 성능을 두 배로 올려 5분에 CPU 100%로 끝낼 수 있습니다. 알고리즘을 개선해 다시 두 배 빨라지면 2.5분 동안 CPU가 100%입니다. **CPU 사용률은 프로그램이 CPU를 얼마나 효율적으로 쓰는지의 지표라, 높을수록 좋습니다.**

Linux에서 `vmstat 1`을 돌리면 초당 한 줄씩 나옵니다.

```
% vmstat 1
procs -----------memory---------- ---swap-- -----io---- -system-- ----cpu----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa
 2  0      0 1797836 1229068 1508276 0    0     0     9 2250 3634 41  3 55  0
 2  0      0 1801772 1229076 1508284 0    0     0     8 2304 3683 43  3 54  0
 1  0      0 1813552 1229084 1508284 0    0     3    22 2354 3896 42  3 55  0
 1  0      0 1819628 1229092 1508292 0    0     0    84 2418 3998 43  2 55  0
```

이 예는 활성 스레드가 하나인 애플리케이션입니다. 매초 CPU는 450ms 동안 바쁘고(user 42% + system 3%), 550ms 동안 idle합니다. CPU가 idle한 이유는 셋입니다.

1. **동기화 기본 요소에 블록**되어 락이 풀릴 때까지 실행 못 함
2. **무언가를 대기** — DB 호출 응답이 돌아오기를 기다리는 등
3. **할 일이 없음**

앞 두 상황은 늘 손볼 수 있는 무언가를 가리킵니다. 락 경합을 줄이거나 DB를 튜닝해 응답을 빨리 돌려받으면 프로그램이 빨라지고 평균 CPU도 올라갑니다(물론 계속 블록할 다른 이슈가 없다는 전제하에). 셋째가 혼동의 지점입니다. 애플리케이션이 할 일이 있고 락·자원 대기로 막히지 않았다면, CPU는 애플리케이션 코드 실행에 사이클을 씁니다. 이는 Java만의 원리가 아닙니다. 무한 루프 스크립트를 짜면 CPU 한 개를 100% 씁니다. 만약 이 스크립트가 CPU를 100% 쓰지 않는다면, OS가 할 일(또 한 줄 출력)이 있는데도 idle을 택했다는 뜻이고, 유용한 긴 계산이라면 주기적 idle은 답을 얻는 데 더 오래 걸리게 할 뿐입니다.

단일 CPU 머신에서 이 무한 루프를 돌리면 대개 도는 줄도 모르지만, 새 프로그램을 시작하거나 다른 애플리케이션 성능을 재면 효과가 분명히 보입니다. OS는 CPU 사이클을 두고 경쟁하는 프로그램을 잘 time-slice하지만, 새 프로그램에 쓸 CPU가 줄어 더 느려집니다. 이 경험 때문에 "혹시 모르니 idle CPU를 좀 남겨 두는 게 좋겠다"고 생각하기 쉽지만, **OS는 당신이 다음에 뭘 할지 추측할 수 없어, 기본적으로 CPU를 놀리기보다 가능한 모든 것을 실행합니다.**



## 2. Java와 단일/다중 CPU, 그리고 스레드 풀
> 배치 작업은 idle이 없어야 하고, 서버는 요청 대기 시 idle하며, 다중 CPU에서는 스레드 부족으로 idle할 수도 있습니다

Java 애플리케이션에서 주기적 idle CPU가 뜻하는 바는 애플리케이션 유형에 따라 다릅니다. 고정량의 일을 가진 배치형 애플리케이션이면 idle CPU가 보여선 안 됩니다. idle이 곧 할 일이 없다는 뜻이기 때문입니다. 배치 작업은 CPU를 높이는 것이 늘 목표이고, 100%여도 더 빨리 끝낼 최적화를 찾을 수 있습니다(CPU를 100%로 유지하면서).

서버형 애플리케이션이면 일이 없어 idle할 수 있습니다. 웹 서버가 모든 HTTP 요청을 처리하고 다음을 기다릴 때입니다. 여기서 평균값이 등장합니다. 앞 `vmstat` 예는 초당 한 요청을 받는 서버에서 나온 것입니다. 요청 처리에 450ms 걸려 CPU가 450ms 동안 100%, 550ms 동안 0%였고, 이것이 45%로 보고됩니다. 같은 거시 패턴은 0.5초마다 요청이 오고 평균 225ms 걸려도 나타납니다(225ms 바쁘고 275ms idle → 평균 45% 바쁨). 요청당 400ms로 최적화하면 전체 CPU도 40%로 줄어듭니다. **이것이 CPU를 낮추는 게 의미 있는 유일한 경우입니다.** 고정 부하가 들어오고 애플리케이션이 외부 자원에 제약받지 않을 때입니다. 다만 이 최적화는 부하를 더 넣을 여지를 줘 결국 CPU를 올릴 수 있고, 미시적으로는 여전히 짧은 시간(400ms) 동안 CPU를 100%로 만드는 문제입니다. 단지 spike가 너무 짧아 대부분 도구가 100%로 못 잡을 뿐입니다.

다중 CPU의 여러 스레드도 개념은 같습니다. 개별 스레드가 블록되지 않게 해 CPU를 높이거나, 스레드가 일을 마치고 대기 중이라 (긴 구간에서) CPU를 낮추는 것이 목표입니다. 다만 다중 스레드·다중 CPU에는 한 가지가 더해집니다. **할 일이 있는데도 CPU가 idle할 수 있습니다.** 그 일을 처리할 스레드가 없을 때입니다. 전형적 사례는 고정 크기 스레드 풀입니다. 작업이 큐에 쌓이고 idle 스레드가 큐의 작업을 집어 실행하는데, 각 스레드는 한 번에 한 작업만 하고 그 작업이 블록(DB 응답 대기)되면 그동안 새 작업을 못 집습니다. 그래서 실행할 작업은 있는데 실행할 스레드가 없어 idle CPU가 생깁니다.

이 경우 스레드 풀 크기를 늘려야 합니다. 그러나 **idle CPU가 있다고 무작정 스레드 풀을 키우면 안 됩니다.** 프로그램이 앞서 말한 다른 두 이유(락·외부 자원 병목)로 CPU를 못 받는 것일 수 있습니다. 행동을 정하기 전에 왜 CPU를 못 받는지 이해하는 것이 중요합니다(9장). CPU 사용률을 보는 것은 첫 단계일 뿐입니다. 코드가 기대만큼 CPU를 쓰는지, 아니면 동기화·자원 이슈를 가리키는지 확인하는 데 씁니다.



## 3. CPU run queue
> 실행 가능한 스레드 수로, Unix run queue는 CPU 수 이하, Windows processor queue는 0이어야 합니다

Windows와 Unix 모두 실행 가능한(I/O 블록·sleep 등이 아닌) 스레드 수를 모니터할 수 있습니다. Unix는 이를 run queue라 하고, 여러 도구가 출력에 run queue 길이를 포함합니다. 앞 `vmstat` 출력의 각 줄 첫 숫자(`r`)가 run queue 길이입니다. Windows는 이를 processor queue라 하고 `typeperf`로 봅니다.

```
C:> typeperf -si 1 "\System\Processor Queue Length"
"05/11/2019 19:09:42.678","0.000000"
"05/11/2019 19:09:43.678","0.000000"
```

여기 중요한 차이가 있습니다. **Unix run queue 길이는 실행 중이거나 CPU가 있으면 실행할 수 있는 모든 스레드 수**입니다. 앞 예에서는 애플리케이션 작업을 하는 단일 스레드가 늘 있어 run queue가 항상 1 이상이었고, 머신 전체를 나타내므로 별개 프로세스의 스레드 때문에 가끔 2였습니다. 반면 **Windows processor queue 길이는 현재 실행 중인 스레드를 포함하지 않습니다.** 그래서 같은 단일 스레드 애플리케이션이 늘 실행 중이어도 processor queue가 0이었습니다.

실행할 스레드가 가용 CPU보다 많으면 성능이 나빠지기 시작합니다. 따라서 **Windows는 processor queue를 0, Unix는 CPU 수 이하(또는 같게)** 두려 합니다. 확고한 규칙은 아니고, 시스템 프로세스 등이 주기적으로 잠깐 값을 올리는 건 큰 영향이 없습니다. 그러나 run queue 길이가 상당 기간 너무 높으면 머신이 과부하라는 신호이고, 작업을 다른 머신으로 옮기거나 코드를 최적화해 일을 줄여야 합니다. 요약하면, CPU time이 성능을 볼 때 첫 검토 대상이고, 코드 최적화의 목표는 CPU 사용률을 (더 짧은 시간 동안) 높이는 것이지 낮추는 것이 아니며, 튜닝 전에 CPU가 낮은 이유를 먼저 이해해야 합니다.



## 4. 디스크 사용 — 양방향 병목과 swapping
> 비효율적 쓰기(처리량 낮음)와 과다 쓰기(처리량 높음) 둘 다 병목이며, swapping은 절대 일어나지 않게 해야 합니다

디스크 사용 모니터링에는 두 목표가 있습니다. 첫째는 애플리케이션 자체입니다. 디스크 I/O를 많이 하면 그 I/O가 쉽게 병목이 됩니다. 디스크 I/O가 병목인지 아는 건 까다롭습니다. 애플리케이션 거동에 달렸기 때문입니다. 데이터를 효율적으로 버퍼링하지 않으면(12장) 디스크 I/O 통계가 낮고, 디스크가 감당할 수 있는 것보다 많은 I/O를 하면 통계가 높습니다. **두 상황 모두 성능을 개선할 수 있으니 양쪽을 살펴야 합니다.** Linux `iostat`의 출력 예입니다.

```
% iostat -xm 5
avg-cpu:  %user   %nice %system %iowait  %steal   %idle
          23.45    0.00   37.89    0.10    0.00   38.56

          Device:         rrqm/s   wrqm/s     r/s     w/s    rMB/s
          sda               0.00    11.60    0.60   24.20     0.02

          wMB/s avgrq-sz avgqu-sz   await r_await w_await  svctm  %util
          0.14    13.35     0.15    6.06    5.33    6.08   0.42   1.04
```

얼핏 좋아 보입니다. 쓰기 서비스 시간(`w_await`)이 6.08ms로 낮고 디스크 이용률은 1.04%뿐입니다. 그러나 잘못된 단서가 있습니다. 시스템이 커널에 37.89%를 쓰고 있습니다. 다른 프로그램의 I/O면 모르되, 그게 다 테스트 중인 애플리케이션에서 나온 거라면 비효율이 일어나는 중입니다. 초당 24.2회 쓰기를 하는 것도 단서입니다. 고작 0.14MBps를 쓰는데 그건 많습니다. I/O가 병목이 됐고, 다음 단계는 애플리케이션이 쓰기를 어떻게 하는지 보는 것입니다. 반대로 디스크가 I/O 요청을 못 따라가는 경우입니다.

```
% iostat -xm 5
avg-cpu:  %user   %nice %system %iowait  %steal   %idle
          35.05    0.00    7.85   47.89    0.00    9.20

          Device:         rrqm/s   wrqm/s     r/s     w/s    rMB/s
          sda               0.00     0.20    1.00  163.40     0.00

          wMB/s avgrq-sz avgqu-sz   await r_await w_await  svctm  %util
          81.09  1010.19   142.74  866.47   97.60  871.17   6.08 100.00
```

Linux는 디스크가 100% 이용 중이고 프로세스가 47.89%를 iowait(디스크 대기)에 쓴다고 바로 알려 줍니다. raw 데이터만 있어도 이상을 알 수 있습니다. I/O 완료 시간(`w_await`)이 871ms, 큐 크기가 크고, 초당 81MB를 씁니다. 모두 디스크 I/O가 문제이고 애플리케이션(또는 시스템 다른 곳)의 I/O를 줄여야 함을 가리킵니다.

둘째 목표는 애플리케이션이 I/O를 많이 안 해도 **시스템이 swapping하는지** 보는 것입니다. 컴퓨터는 고정된 물리 메모리를 갖지만 훨씬 큰 가상 메모리를 쓰는 애플리케이션들을 돌릴 수 있습니다. 애플리케이션은 필요보다 많은 메모리를 예약하고 보통 일부만 씁니다. OS는 안 쓰는 메모리를 디스크에 두고 필요할 때만 물리 메모리로 page in합니다. 대체로 대화식·GUI 프로그램에는 잘 동작하지만, 서버 기반 애플리케이션은 메모리를 더 많이 써서 덜 잘 동작하고, **Java 애플리케이션에는 Java 힙 때문에 특히 나쁘게 동작합니다**(5장). `vmstat`의 `si`(swap in)·`so`(swap out) 컬럼이 swapping을 알려 주고, 디스크 활동도 지표입니다. **swapping(메인 메모리와 디스크 간 페이지 이동)이 일어나면 성능이 꽤 나빠지므로, 시스템은 swapping이 절대 일어나지 않게 설정해야 합니다.**



## 5. 네트워크 사용 — 이용률과 40% 포화
> 표준 도구는 이용률을 안 주므로 nicstat 같은 도구를 쓰며, 로컬 이더넷은 지속 40% 초과면 포화입니다

네트워크를 쓰는 애플리케이션(REST 서버 등)이면 네트워크 트래픽도 모니터해야 합니다. 디스크와 비슷하게, 비효율적으로 써서 대역폭이 너무 낮거나, 인터페이스가 감당할 수 있는 것보다 많이 써서 병목이 됩니다. 안타깝게도 표준 시스템 도구는 네트워크 모니터링에 이상적이지 않습니다. 보통 특정 인터페이스로 주고받은 패킷 수와 바이트 수만 보여 줘, **네트워크가 과소·과다 이용 중인지를 알려주지 않습니다.**

Unix 기본 도구는 `netstat`(대부분 Linux 배포판에는 포함조차 안 돼 따로 받아야 함)이고, Windows는 `typeperf`를 스크립트에 쓰거나 표준 자원 모니터 GUI가 네트워크 사용 퍼센트 그래프를 보여 줍니다(단 자동화 테스트에는 GUI가 도움 안 됨). 다행히 네트워크 대역폭을 모니터하는 오픈소스·상용 도구가 많습니다. Unix의 인기 명령행 도구 `nicstat`은 각 인터페이스 트래픽 요약과 이용률을 보여 줍니다.

```
% nicstat 5
Time      Int       rKB/s   wKB/s   rPk/s   wPk/s   rAvs    wAvs   %Util  Sat
17:05:17  e1000g1   225.7   176.2   905.0   922.5   255.4   195.6  0.33   0.00
```

`e1000g1`은 1,000MB 인터페이스이고 0.33%만 이용 중입니다. 이 도구의 유용함은 인터페이스 이용률을 계산해 준다는 점입니다. `typeperf`나 `netstat`은 읽고 쓴 데이터양만 보고하므로, 이용률을 알려면 인터페이스 대역폭을 알아내 직접 스크립트로 계산해야 합니다. 중요한 점은 **대역폭은 초당 비트(bps)로 재지만 도구는 보통 초당 바이트(Bps)로 보고**한다는 것입니다. 1,000Mb 네트워크는 초당 125MB입니다. 이 예에서 0.22MBps 읽기 + 0.16MBps 쓰기를 125로 나누면 0.33% 이용률이 나옵니다. `nicstat`에 마법은 없고 단지 더 편할 뿐입니다.

**네트워크는 100% 이용률을 지속할 수 없습니다.** 로컬 이더넷은 지속 이용률 40% 초과면 인터페이스가 포화된 것입니다. 패킷 스위치드거나 다른 매체면 최대 지속률이 달라지니 네트워크 아키텍트에게 적정 목표를 확인합니다. 이 목표는 Java와 무관합니다. Java는 OS의 네트워킹 파라미터·인터페이스를 그대로 쓸 뿐입니다.



## 자주 받는 오해
> CPU 100%는 나쁘다고 생각하기 쉽지만, 배치 작업에서는 100%가 목표입니다

1. "CPU 100%는 나쁜 상태"라고 생각하기 쉽지만, 고정량의 일을 하는 배치 작업에서는 100%가 목표입니다. CPU 사용률은 효율 지표라 높을수록 빨리 끝납니다. idle CPU를 "혹시 모르니" 남기는 건 답을 더 늦게 얻게 할 뿐입니다.
2. "idle CPU가 있으면 스레드 풀을 키워야 한다"고 생각하기 쉽지만, 락 경합이나 외부 자원 병목으로 CPU를 못 받는 것일 수 있습니다. 왜 CPU를 못 받는지 먼저 이해해야 합니다.
3. "디스크 I/O 통계가 낮으면 디스크는 문제없다"고 생각하기 쉽지만, 비효율적 버퍼링으로 통계가 낮은 것일 수 있습니다. 과다·과소 양쪽이 병목이라 둘 다 살펴야 하고, system time이 높으면 비효율 신호입니다.



## 면접에서 받을 만한 질문
1. **CPU 사용률은 왜 높을수록 좋습니까?** → CPU 사용률은 프로그램이 CPU를 얼마나 효율적으로 쓰는지의 지표이고 구간 평균값입니다. 평균 50%면 절반은 놀았다는 뜻이라, 놀던 시간을 없애면 절반 시간에 끝낼 수 있습니다. 고정량의 배치 작업에서는 100%가 목표입니다. 예외는 고정 부하가 들어오고 외부 자원에 제약받지 않는 서버로, 이때만 요청당 시간을 줄여 CPU를 낮추는 것이 의미가 있습니다.
2. **CPU가 idle한 세 가지 이유와 각각의 의미는?** → 첫째 동기화 락에 블록, 둘째 DB 응답 같은 외부 대기, 셋째 할 일 없음입니다. 앞 둘은 락 경합을 줄이거나 외부 자원을 튜닝해 고칠 수 있는 신호이고, 셋째는 서버가 요청을 기다리는 정상 상태입니다. 다중 CPU에서는 넷째로, 할 일은 있지만 스레드 풀에 처리할 스레드가 없어 idle한 경우도 있습니다.
3. **Unix run queue와 Windows processor queue의 차이는?** → Unix run queue는 실행 중이거나 CPU가 있으면 실행 가능한 모든 스레드 수라, 단일 활성 스레드만 있어도 1 이상입니다. Windows processor queue는 현재 실행 중인 스레드를 제외해, 같은 상황에서 0입니다. 권장값은 Unix는 CPU 수 이하, Windows는 0이며, 상당 기간 이를 넘으면 머신 과부하 신호입니다.
4. **swapping을 왜 디스크 모니터링으로 감시합니까?** → 시스템은 안 쓰는 가상 메모리를 디스크에 두는데, Java는 힙 때문에 메모리를 많이 써서 swapping에 특히 취약합니다. swapping(메인 메모리와 디스크 간 페이지 이동)이 일어나면 성능이 크게 나빠집니다. `vmstat`의 `si`/`so` 컬럼과 디스크 활동이 지표이고, 시스템은 swapping이 절대 일어나지 않게 설정해야 합니다.
