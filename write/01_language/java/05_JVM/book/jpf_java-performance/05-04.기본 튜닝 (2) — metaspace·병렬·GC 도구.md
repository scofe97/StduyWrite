# 기본 튜닝 (2) — metaspace·병렬·GC 도구
> metaspace는 클래스 메타데이터를 담는 별도 영역이고, GC 스레드 수는 CPU 기반이며, GC 로그는 진단의 핵심 데이터입니다

[앞 편](./05-03.기본%20튜닝%20(1)%20—%20힙과%20세대%20크기.md)이 힙·세대 크기였다면, 이 편은 나머지 기본 튜닝입니다. 클래스 메타데이터 영역인 metaspace, GC 병렬 스레드 수, 그리고 GC를 들여다보는 로그·도구를 다룹니다.



## 1. metaspace — 클래스 메타데이터 영역
> 클래스 메타데이터(클래스 객체가 아님)를 담는 별도 힙으로, 기본 최대가 무제한이라 튜닝이 드물고, 리사이즈는 full GC를 요구합니다

JVM이 클래스를 로드하면 그 클래스의 메타데이터를 추적해야 합니다. 이는 **metaspace**라는 별도 힙 공간에 담깁니다(옛 JVM은 permgen이라는 다른 구현으로 처리). metaspace는 사용자에게 불투명합니다. 클래스 관련 데이터를 담고, 특정 상황에서 크기를 튜닝해야 한다는 것만 압니다. **metaspace는 클래스의 실제 인스턴스(Class 객체)나 리플렉션 객체(Method 등)를 담지 않습니다. 그것들은 일반 힙에 있습니다.** metaspace의 정보는 컴파일러·JVM 런타임만 쓰고, 담는 데이터를 class metadata라 합니다.

프로그램에 필요한 metaspace 크기를 미리 계산할 좋은 방법은 없습니다. 크기는 쓰는 클래스 수에 비례해 큰 앱은 더 큰 영역이 필요합니다. **permgen 튜닝은 흔했지만 metaspace 튜닝은 드뭅니다.** 기본값이 후하기 때문입니다.

| JVM | 기본 초기 | 기본 최대 |
|-----|-----------|-----------|
| 32비트 client | 12MB | 무제한 |
| 32비트 server | 16MB | 무제한 |
| 64비트 | 20.75MB | 무제한 |

metaspace는 일반 힙의 별도 인스턴스처럼 동작합니다. 초기 크기(`-XX:MetaspaceSize=N`)에서 시작해 필요 시 최대(`-XX:MaxMetaspaceSize=N`)까지 늘어납니다. 기본 최대가 무제한이라 (특히 32비트에서) metaspace를 채워 메모리가 바닥날 수 있습니다(8장 NMT로 진단). `MaxMetaspaceSize`를 낮추면 metaspace가 차면 `OutOfMemoryError`가 나는데, **진짜 해결은 class metadata가 왜 큰지 알아내는 것**입니다.

**metaspace 리사이즈는 full GC를 요구해 비싼 연산입니다.** 프로그램 시작 중(클래스 로딩 시) full GC가 많으면 흔히 permgen/metaspace 리사이즈 때문이라, 초기 크기를 늘리면 시작이 빨라집니다(서버는 보통 초기 128MB·192MB 이상 지정). Java 클래스도 다른 것처럼 GC 대상입니다. 애플리케이션 서버는 배포(재배포)마다 새 classloader를 만드는데, 옛 classloader는 참조가 끊겨 GC 대상이 되고 그것이 정의한 클래스도 마찬가지입니다. 새 클래스는 새 메타데이터를 가져 metaspace에 자리가 있어야 하므로, **흔히 full GC가 일어납니다**(metaspace가 자라거나 옛 메타데이터를 버려야 해서).

metaspace 크기를 제한하는 한 이유는 **classloader leak 방어**입니다. 애플리케이션 서버(나 IDE)가 옛 classloader 참조를 유지하며 계속 새 classloader·클래스를 정의하는 경우입니다. metaspace를 채우고 머신 메모리를 많이 쓸 수 있습니다. 다만 그 경우 실제 classloader·클래스 객체도 메인 힙에 있어, metaspace가 문제 되기 전에 **힙이 먼저 차 OOM이 날 가능성**이 큽니다. 힙 덤프(7장)로 어떤 classloader가 있는지 진단하거나, `jmap -clstats`로 classloader 정보를 출력합니다.



## 2. 병렬 제어 — ParallelGCThreads
> serial을 뺀 모든 GC가 여러 스레드를 쓰며, CPU당 1스레드(8까지) 후 1.6 CPU당 1을 더하고, 여러 JVM이면 줄여야 합니다

serial을 뺀 모든 GC는 여러 스레드를 쓰고, **`-XX:ParallelGCThreads=N`**으로 제어합니다. 이 값은 `UseParallelGC`의 young·old 수집, `UseG1GC`의 young 수집, G1의 STW 단계(full GC 제외)에 쓰는 스레드 수에 영향을 줍니다. 이 GC 연산은 모든 애플리케이션 스레드를 멈추므로, JVM은 pause를 최소화하려 가능한 한 많은 CPU를 씁니다. 기본적으로 **CPU당 1스레드를 8까지** 쓰고, 그 후엔 **1.6 CPU당 1스레드**만 더합니다.

```
ParallelGCThreads = 8 + ((N - 8) * 5 / 8)
```

(N은 CPU 수.) 이 수가 클 때가 있습니다. 8 CPU 머신의 작은 힙(1GB)은 4~6 스레드가 약간 더 효율적이고, 128 CPU의 83 GC 스레드는 가장 큰 힙 외엔 너무 많습니다. **Docker 컨테이너에 CPU 한도가 있으면 그 한도를 계산에 씁니다.** 또 **여러 JVM이 한 머신에 돌면 총 GC 스레드를 제한하는 게 좋습니다.** GC 스레드는 효율적이라 각각 단일 CPU를 100% 소비합니다(throughput의 평균 CPU가 예상보다 높았던 이유). 16 CPU에 4 JVM이면 각 13 GC 스레드라, 동시 GC 시 52개의 CPU-hungry 스레드가 경합합니다. **각 JVM을 4 GC 스레드로 제한하면 더 효율적**입니다. 한 JVM이 13 스레드로 GC하면, 나머지 JVM 애플리케이션 스레드가 16 CPU 중 13개가 100% 바쁜 머신에서 CPU를 두고 경쟁하기 때문입니다. (이 플래그가 G1 백그라운드 스레드 수를 직접 정하지는 않습니다 — 6장 상세.)



## 3. GC 로그 — JDK 8과 JDK 11
> GC 로그는 GC 이슈 진단의 핵심 데이터이며, JDK 8은 PrintGCDetails·Xloggc로, JDK 11은 통합 로깅 Xlog로 켭니다

GC가 Java 성능의 중심이라 많은 도구가 GC를 모니터합니다. GC 효과를 보는 가장 좋은 방법은 **GC 로그**에 익숙해지는 것입니다. 로그 관리는 모든 알고리즘에 같지만, **JDK 8과 이후는 다릅니다.**

**JDK 8** — `-verbose:gc` 또는 `-XX:+PrintGC`는 단순 로그를, **`-XX:+PrintGCDetails`는 훨씬 많은 정보**의 로그를 만듭니다(권장, 기본 false). 함께 `-XX:+PrintGCTimeStamps`나 `-XX:+PrintGCDateStamps`를 넣어 GC 간격을 알 수 있게 합니다(timestamp는 0 기준, date stamp는 실제 날짜라 약간 덜 효율적). 로그는 표준 출력으로 가지만 `-Xloggc:filename`으로 바꿉니다(이러면 단순 로그 자동 활성, PrintGCDetails 없으면). 로그 로테이션은 `-XX:+UseGCLogFileRotation -XX:NumberOfGCLogFiles=N -XX:GCLogFileSize=N`으로 제어합니다(셋 다 값을 줘야 기대대로 동작). 유용한 조합입니다.

```
-Xloggc:gc.log -XX:+PrintGCTimeStamps -XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFile=8 -XX:GCLogFileSize=8m
```

이는 timestamp와 함께 GC 이벤트를 로깅하고 8개 파일 64MB로 제한합니다. **오버헤드가 작아 프로덕션에서도 켤 만합니다.**

**JDK 11** — Java의 새 통합 로깅을 씁니다. 모든 로깅을 `-Xlog`로 켜고 옵션을 붙입니다.

```
-Xlog:gc*:file=gc.log:time:filecount=7,filesize=8M
```

콜론이 넷으로 나눕니다. ① `gc*`는 로깅할 모듈(모든 GC 모듈, `gc+age`처럼 특정 섹션만도 가능, `gc*,gc+age=debug`처럼 레벨도), ② 로그 파일 대상, ③ `time`은 time-of-day 스탬프 decorator, ④ 출력 옵션(8MB마다 로테이션, 8개 유지)입니다. **로그 로테이션이 JDK 8과 조금 다릅니다.** JDK 8은 `gc.log.0.current`에 쓰다 차면 `gc.log.0`으로 rename하고 다음으로 넘어가지만, **JDK 11은 `gc.log`에 쓰다 차면 `gc.log.0`으로 rename**하고 새 `gc.log`를 시작합니다. JDK 11에서 7개를 유지로 지정하면 활성 파일이 8개가 되는 이유입니다(`gc.log` + `.0`~`.6`). 어느 경우든 파일에 붙는 숫자는 생성 순서를 뜻하지 않습니다(순환 재사용).

로그 파싱 오픈소스 도구는 많지 않고, jClarity Censum·GCeasy 같은 상용(GCeasy는 기본 파싱 무료)이 있습니다.



## 4. 실시간 도구 — jstat·jconsole
> jconsole·jvisualvm로 힙을 실시간 그래프로 보고, jstat -gcutil로 GC 시간과 각 영역 점유율을 스크립트로 모니터합니다

실시간 힙 모니터링은 **jvisualvm·jconsole**을 씁니다. jconsole의 Memory 패널은 힙 실시간 그래프를 보여 주고, 전체 대신 eden·survivor·old·permanent generation만 표시할 수도 있습니다. 스크립트 솔루션은 **jstat**입니다. jstat은 힙 정보 출력에 9개 옵션을 주고(`jstat -options`로 전체), 유용한 `-gcutil`은 **GC에 쓴 시간과 각 GC 영역의 현재 점유율**을 보여 줍니다. 반복 인자(ms)를 줘 시간에 따른 GC 효과를 모니터합니다.

```
% jstat -gcutil 23461 1000
  S0     S1     E      O      P     YGC     YGCT    FGC    FGCT     GCT
 51.71   0.00  99.12  60.00  99.93     98    1.985     8    2.397    4.382
  0.00  42.08   5.55  60.98  99.93     99    2.016     8    2.397    4.413
```

모니터링 시작 시 프로그램은 이미 young collection 98회(YGC, 총 1.985초 YGCT)와 full GC 8회(FGC, 2.397초 FGCT)를 했고, 총 GC 시간(GCT)은 4.382초였습니다. young의 세 영역(survivor S0·S1, eden E)이 모두 보입니다. eden이 차던(99.12%) 다음 초에 young collection이 일어나 eden이 5.55%로 줄고, survivor가 자리를 바꾸고, 약간이 old(O)로 승격돼 60.98%가 됐습니다. permanent generation(P)은 필요한 클래스가 이미 다 로드돼 변화가 거의 없습니다(전형적). **GC 로깅을 잊었다면 이것이 GC 동작을 시간에 따라 보는 좋은 대체**입니다.

요약하면, GC 성능은 Java 애플리케이션 전체 성능의 핵심이지만 많은 앱은 **적절한 GC 알고리즘 선택과 필요 시 힙 크기 증가**만으로 충분합니다. 그러면 adaptive sizing이 주어진 힙으로 좋은 성능을 내도록 자동 튜닝합니다. 더 복잡한 앱은 알고리즘별 추가 튜닝이 필요하고, 이 장의 단순 설정으로 부족하면 6장 튜닝을 참고합니다.



## 자주 받는 오해
> classloader leak이 metaspace를 먼저 채운다고 생각하기 쉽지만, 보통 메인 힙이 먼저 OOM이 납니다

1. "metaspace가 Class 객체와 리플렉션 객체를 담는다"고 생각하기 쉽지만, 그것들은 일반 힙에 있습니다. metaspace는 컴파일러·런타임이 쓰는 class metadata만 담습니다.
2. "classloader leak은 metaspace를 채워 OOM을 낸다"고 생각하기 쉽지만, 실제 classloader·클래스 객체도 메인 힙에 있어 metaspace가 문제 되기 전에 힙이 먼저 차 OOM이 날 가능성이 큽니다.
3. "GC 스레드 수는 자동이라 신경 안 써도 된다"고 생각하기 쉽지만, 여러 JVM이 한 머신에 돌면 합이 너무 많아집니다(16 CPU 4 JVM = 52 스레드). 각 JVM의 GC 스레드를 줄여야 효율적입니다.



## 면접에서 받을 만한 질문
1. **metaspace는 무엇을 담고 permgen과 어떻게 다릅니까?** → metaspace는 클래스 메타데이터(컴파일러·JVM 런타임이 쓰는 class metadata)를 담는 별도 힙 영역으로, 옛 JVM의 permgen을 대체합니다. Class 객체나 리플렉션 객체는 일반 힙에 있고 metaspace엔 없습니다. permgen 튜닝은 흔했지만 metaspace는 기본 최대가 무제한이라 튜닝이 드뭅니다. 리사이즈가 full GC를 요구하므로, 시작 중 클래스 로딩으로 full GC가 잦으면 초기 크기를 늘려 시작을 빠르게 합니다.
2. **classloader leak이 무엇이고 어디서 먼저 문제가 됩니까?** → 애플리케이션 서버나 IDE가 옛 classloader 참조를 유지하며 계속 새 classloader·클래스를 정의하는 것입니다. metaspace를 채울 수 있지만, 실제 classloader·클래스 객체도 메인 힙에 있어 보통 힙이 먼저 차 OOM이 납니다. 힙 덤프로 어떤 classloader가 있는지, `jmap -clstats`로 classloader 정보를 진단합니다.
3. **여러 JVM을 한 머신에 돌릴 때 GC 스레드를 왜 줄여야 합니까?** → GC 스레드는 각각 단일 CPU를 100% 소비합니다. 기본 스레드 수는 CPU 기반이라(CPU당 1, 8까지) 16 CPU 머신은 JVM당 13 스레드를 씁니다. 4 JVM이면 동시 GC 시 52개의 CPU-hungry 스레드가 16 CPU를 두고 경합합니다. 한 JVM만 13 스레드로 GC해도 나머지 JVM이 13/16 CPU가 바쁜 상태에서 경쟁하므로, 각 JVM을 4 스레드로 제한하는 게 효율적입니다.
4. **jstat -gcutil로 무엇을 봅니까?** → GC에 쓴 시간과 각 GC 영역의 현재 점유율을 봅니다. survivor(S0·S1)·eden(E)·old(O)·metaspace(M/P) 점유율과 young collection 횟수·시간(YGC·YGCT), full GC 횟수·시간(FGC·FGCT), 총 GC 시간(GCT)이 나옵니다. 반복 인자(ms)를 주면 시간에 따른 추이를 봐서, eden이 차다 young collection으로 줄고 일부가 old로 승격되는 패턴을 관찰할 수 있습니다. GC 로깅을 잊었을 때 좋은 대체입니다.
