# 서문 — 책 소개와 2판 변경점
> Java 성능은 JVM·API 동작 원리를 깊이 알아야 끌어낼 수 있고, 환경은 대형 힙과 단일 CPU로 갈라졌습니다

이 노트는 『Java Performance, 2판』(Scott Oaks, O'Reilly)의 서문을 정리한 메타 노트입니다. 기술적인 정독 노트는 1장부터 시작하므로, 여기서는 책 전체가 어떤 관점으로 쓰였는지, 누구를 위한 책인지, 2판에서 무엇이 바뀌었는지를 짚습니다. 본문 정독에 앞서 저자가 깔아 둔 전제를 먼저 이해하기 위해서입니다.

저자 Scott Oaks는 1990년대 말 Sun에서 Java 에반젤리스트로 일했고, 이후 20여 년간 Sun과 Oracle의 Java Performance Group을 비롯한 엔지니어링 조직에 몸담았습니다. 그래서 이 책은 한 사람의 이론이 아니라 오랜 현장 경험이 쌓인 결과물입니다. 저자 스스로 "처음 O'Reilly가 Java 성능 튜닝 책을 제안했을 때, Java 성능은 이미 끝난 주제 아닌가 싶었다"고 털어놓는데, 그 망설임을 뒤집은 이유들이 곧 이 책의 존재 이유입니다.

서문이 던지는 가장 중요한 메시지는 하나입니다. **Java를 정말 빠르게 만들려면 JVM과 Java API가 실제로 어떻게 동작하는지를 깊이 이해해야 한다**는 것입니다. 수백 개의 튜닝 플래그를 맹목적으로 시도해 보며 되는 걸 찾는 방식이 아니라, 동작 원리를 알아 "왜 이 애플리케이션이 느린가"를 설명할 수 있어야 합니다. 원인을 이해하면 나쁜 동작을 걷어내는 일은 한결 단순해집니다.



## 1. 이 책이 다루는 세 축
> JVM 튜닝, API 모범 사용, 측정 방법론 — 코드를 고치는 일과 안 고치는 일 양쪽을 다룹니다

서문은 Java 성능 작업이 서로 배경이 다른 두 부류의 엔지니어에게 걸쳐 있다고 말합니다. 한쪽은 잘 안 쓰이는 API 메서드 시그니처를 수천 개 외우면서도 `-Xmn` 플래그가 무슨 뜻인지 모르는 개발자이고, 다른 한쪽은 GC 플래그를 능숙하게 다루지만 "Hello, World" 정도밖에 못 짜는 테스트 엔지니어입니다. 이 책은 그 둘을 잇기 위해 다음 세 축을 함께 다룹니다.

1. **JVM 튜닝** — 코드를 한 줄도 바꾸지 않고 JVM의 동작을 바꾸는 일입니다. GC와 JIT 컴파일러가 핵심이고, 그 동작을 조정하는 수백 개의 플래그가 있습니다. 다만 플래그를 무작위로 켜고 끄는 건 튜닝이 아닙니다.
2. **API 모범 사용** — 새 버전마다 추가되는 언어 기능과 API를 가장 좋은 방식으로 쓰는 일입니다. 같은 기능이라도 어떻게 쓰느냐가 "쌩쌩 도는" 애플리케이션과 "터덜터덜 가는" 애플리케이션을 가릅니다.
3. **측정 방법론** — 추측이 아니라 데이터로 판단하는 일입니다. 2판은 마이크로벤치마킹 도구 jmh를 새로 다룹니다.

저자는 만약 JVM 자체의 성능에만 관심이 있고 코딩에는 관심이 없더라도 이 책의 상당 부분이 도움이 된다고 말합니다. 코딩 부분을 건너뛰고 관심 있는 영역에 집중해도 좋다는 것입니다. 다만 책 곳곳의 데이터는 샘플 프로그램을 돌려 얻은 것이라, 프로그램 설명에도 적지 않은 분량을 씁니다.

서문은 API 선택이 곧 트레이드오프임을 한 예로 보여줍니다. 두 프로그램이 정보를 주고받을 때 JSON을 쓰는 편이 고도로 최적화된 독자 프로토콜을 새로 만드는 것보다 훨씬 간단합니다. 개발자의 시간을 아끼는 건 큰 이득입니다. 그런데 진짜 목표는 그 생산성 이득이 성능 이득과 함께 오도록(적어도 본전은 되도록) 만드는 것입니다.



## 2. 누가 읽어야 하고, 누가 읽으면 안 되는가
> 성능 엔지니어와 개발자를 위한 책이며, 급한 처방을 찾는 사람에게는 맞지 않습니다

서문은 대상 독자를 분명히 합니다. 이 책은 JVM과 Java API의 여러 측면이 성능에 어떤 영향을 주는지 이해하려는 성능 엔지니어와 개발자를 위한 책입니다. 저자가 명시적으로 선을 긋는 부분이 있어 그대로 옮깁니다.

> 일요일 밤 늦은 시각, 월요일 아침에 사이트가 오픈하는데 성능 문제의 빠른 처방을 찾고 있다면, 이 책은 당신을 위한 책이 아닙니다.

이 단호한 문장은 책의 성격을 압축합니다. 즉석 처방집이 아니라 원리를 이해시키는 책이라는 뜻입니다. 성능 분석이 처음이라 Java에서 분석을 시작하는 사람도 도울 수 있지만, 시스템 분석 자체는 넓은 분야이므로 일반 시스템 분석 교재를 함께 보는 것이 좋다고 저자는 권합니다.

전제로 두는 지식은 **Java로 프로그램을 작성할 줄 안다**는 것입니다. 저자는 프로그램에 관해 적지 않은 분량을 다루므로, 코딩 능력이 어느 정도 있다고 가정합니다.



## 3. 성능 작업의 본질 — 성숙할수록 큰 개선은 드물다
> 초기 앱은 큰 도약이 쉽지만, 성숙한 앱(JVM 포함)에서는 점진적 개선이 대부분입니다

서문에서 가장 곱씹을 만한 통찰은 성능 작업이 시간이 갈수록 점점 더 어려워진다는 점입니다. 이것은 애플리케이션 일반에 적용되는 전형적인 과정이고, JVM 자신도 또 하나의 애플리케이션일 뿐이라 예외가 아닙니다.

프로젝트 초기에는 아키텍처 변경이나 코드 버그를 찾아 고치는 것만으로 큰 성능 향상을 얻기가 쉽습니다. 그러나 성숙한 애플리케이션에서 그런 개선을 찾기란 드뭅니다. 저자는 JVM의 역사를 예로 듭니다.

1. JIT(just-in-time) 컴파일러를 도입해 Java 1.0 대비 1.1.8에서 **8배** 성능을 얻은 것은, 비교적 단순한 엔지니어링이 만든 도약이었습니다. Java 1.0은 거의 완전히 인터프리터 방식이었고, 1.1.8은 실제 JIT 컴파일러를 가졌기 때문입니다.
2. 가비지 컬렉터를 병렬화한 것도 큰 향상이었습니다.
3. 그러나 그 이후의 변화들은 더 점진적입니다. 컴파일러는 계속 좋아지지만, 8배 같은 도약은 다시 오지 않습니다.

이 인식이 저자가 처음 "Java 성능은 끝난 주제 아닌가"라고 의심했던 배경이었습니다. 생각을 바꾼 이유는 두 가지였습니다. 첫째, JVM이 특정 상황에서 어떻게 동작하는지 묻는 질문이 매일같이 쏟아진다는 점입니다. Java에 새로 들어오는 엔지니어는 늘 있고, JVM의 어떤 영역은 여전히 복잡해 동작 안내서가 유용합니다. 둘째, 컴퓨팅 환경의 변화가 엔지니어가 마주하는 성능 고민 자체를 바꿔 놓았다는 점입니다.



## 4. 환경의 분기 — 대형 힙과 단일 CPU
> 거대한 힙을 돌리는 큰 머신과, 클라우드의 작은 단일 CPU 머신으로 성능 고민이 갈라졌습니다

서문은 최근 몇 년 사이 성능 관심사가 둘로 갈라졌다(bifurcated)고 진단합니다. 이 분기가 2판의 방향을 상당 부분 결정했습니다.

한쪽 끝에는 거대한 힙을 돌리는 큰 머신이 일상이 된 흐름이 있습니다. JVM은 여기에 새 가비지 컬렉터인 G1으로 대응했는데, 신기술인 만큼 전통적인 컬렉터보다 손으로 만지는 튜닝이 조금 더 필요합니다.

다른 한쪽 끝에는 클라우드 컴퓨팅이 되살린 작은 단일 CPU 머신의 중요성이 있습니다. Oracle이든 Amazon이든, 작은 애플리케이션 서버를 돌릴 단일 CPU 머신을 싸게 빌릴 수 있습니다. 저자는 여기에 정확한 설명을 덧붙입니다. 실제로 물리적인 단일 CPU 머신을 받는 것이 아니라, 거대한 머신 위의 가상 OS 이미지를 받되 그 가상 OS가 CPU 하나만 쓰도록 제한된 것입니다. **Java의 관점에서 보면 이것은 단일 CPU 머신과 똑같습니다.** 이런 환경에서는 작은 양의 메모리를 올바르게 관리하는 일이 꽤 중요해집니다.

다음 표는 두 환경의 대비를 정리한 것입니다.

| 축 | (A) 대형 머신 | (B) 단일 CPU(클라우드) |
|----|--------------|------------------------|
| 메모리 | 거대한 힙 | 작은 메모리 |
| JVM 대응 | G1 GC (손튜닝 더 필요) | 작은 메모리의 정확한 관리 |
| 실체 | 물리 대형 머신 | 큰 머신 위 CPU 1개 제한 가상 OS |
| 핵심 고민 | 큰 힙의 GC 일시 정지 | 한정된 자원의 효율 |



## 5. 2판에서 새로워진 것
> 대상은 Java 8과 Java 11이고, G1·JFR·컨테이너·jmh·AppCDS·compact string이 새로 들어왔습니다

1판 이후 Java는 6개월 릴리스 주기를 채택했고, 주기적으로 장기 지원(LTS) 릴리스를 둡니다. 그래서 출판 시점에 지원되는 릴리스는 Java 8과 Java 11입니다. 1판도 Java 8을 다뤘지만 당시에는 Java 8이 막 나온 상태였고, 2판은 한층 성숙한 Java 8과 Java 11을 다룹니다.

2판의 변경점은 다음과 같습니다.

1. **G1 가비지 컬렉터**의 대규모 업데이트
2. **Java Flight Recorder(JFR)**에 대한 비중 있는 갱신
3. Java가 **컨테이너화된 환경**에서 어떻게 동작하는지에 대한 주의 깊은 다룸
4. 새 마이크로벤치마킹 도구 **jmh**
5. 새로운 **JIT 컴파일러들**
6. **애플리케이션 클래스 데이터 공유(AppCDS)** 와 새 성능 도구들
7. Java 11의 새 기능인 **compact string** 과 **string concatenation** 개선

이 목록은 정독 노트의 순서를 미리 보여 주는 지도이기도 합니다. GC(특히 G1), JIT, 측정(jmh·JFR), 컨테이너, 문자열 최적화가 본문의 굵은 줄기가 됩니다.



## 6. 예제 코드와 표기 규약
> 예제는 GitHub에 공개돼 있고, 본문은 이탤릭·고정폭 등으로 코드와 입력을 구분합니다

책의 보충 자료(예제 코드, 연습 등)는 GitHub에서 내려받을 수 있습니다.

```
https://github.com/ScottOaks/JavaPerformanceTuning
```

서문은 본문에서 쓰는 표기 규약도 안내합니다. 정독 노트에서 코드·명령어를 옮길 때 이 구분을 의식하면 원문 의도를 정확히 따를 수 있습니다.

| 표기 | 의미 |
|------|------|
| *이탤릭* | 새 용어, URL, 이메일, 파일명, 확장자 |
| `고정폭` | 프로그램 목록, 변수·함수명, 데이터 타입, 환경변수, 키워드 |
| **`고정폭 볼드`** | 사용자가 그대로 입력해야 할 명령·텍스트 |
| `고정폭 이탤릭` | 사용자 값이나 문맥에 따른 값으로 대체할 자리 |

저자는 책의 코드를 프로그램이나 문서에 가져다 써도 좋고, 상당 분량을 그대로 복제하는 경우가 아니라면 허락을 따로 받을 필요가 없다고 밝힙니다. 출처 표기는 권장하지만 필수는 아니며, 표준 형식은 "제목, 저자, 출판사, ISBN"입니다.
