# Mockito와 테스트 더블
---
> 테스트 더블은 실제 의존성을 대체하여 테스트 대상 코드를 격리시키는 핵심 기법이다. Mockito는 Java 생태계에서 가장 널리 쓰이는 목(Mock) 프레임워크로, 선언적 API로 복잡한 의존성 시나리오를 간결하게 표현한다.

## 1. 테스트 더블 분류

*테스트 더블(Test Double)*은 테스트에서 실제 객체를 대신하는 모든 가짜 객체의 총칭이다. Gerard Meszaros가 정의한 다섯 가지 유형은 목적과 동작 방식이 다르다.

| 유형 | 목적 | 동작 방식 | 상태 검증 | 행위 검증 |
|------|------|-----------|-----------|-----------|
| **Dummy** | 파라미터 채우기 | 호출되지 않음 | 없음 | 없음 |
| **Stub** | 간접 입력 제공 | 미리 정의된 값 반환 | 없음 | 없음 |
| **Spy** | 실제 동작 + 호출 기록 | 실제 메서드 위임 | 선택적 | 있음 |
| **Mock** | 기대 행위 검증 | 미리 정의된 값 반환 | 없음 | 있음 |
| **Fake** | 경량 구현체 대체 | 단순화된 실제 로직 | 있음 | 없음 |

가장 많이 혼동하는 쌍은 Stub과 Mock이다. Stub은 "특정 입력에 특정 출력"을 제공할 뿐 호출 여부를 검증하지 않는다. Mock은 "이 메서드가 이런 방식으로 호출되었는가"를 검증하는 것이 목적이다.

## 2. Mockito 기본 사용

Mockito는 `mock()` 팩토리 메서드나 애노테이션으로 목 객체를 생성한다. 생성된 목 객체의 모든 메서드는 기본적으로 반환 타입의 기본값을 반환한다 (int → 0, Object → null, List → 빈 리스트).

```java
import static org.mockito.Mockito.*;

class OrderServiceTest {

    @Test
    void 재고_확인_후_주문_생성() {
        // mock() 팩토리 메서드로 생성
        InventoryRepository inventoryRepo = mock(InventoryRepository.class);
        OrderRepository orderRepo = mock(OrderRepository.class);

        // when().thenReturn()으로 Stub 정의
        when(inventoryRepo.findByProductId(100L))
            .thenReturn(new Inventory(100L, 50));

        OrderService service = new OrderService(inventoryRepo, orderRepo);
        service.placeOrder(100L, 3);

        // verify()로 메서드 호출 검증
        verify(orderRepo).save(any(Order.class));
        verify(inventoryRepo).findByProductId(100L);
    }
}
```

## 3. 애노테이션 기반 설정

`@Mock`, `@InjectMocks`, `@ExtendWith`를 사용하면 반복적인 목 생성 코드를 제거할 수 있다. `@ExtendWith(MockitoExtension.class)`는 JUnit 5와 통합되어 테스트 시작 전 자동으로 목을 초기화한다.

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    InventoryRepository inventoryRepository;

    @Mock
    OrderRepository orderRepository;

    @InjectMocks
    OrderService orderService;
    // OrderService 생성자/필드에 @Mock 객체를 자동 주입한다

    @Test
    void 주문_생성_테스트() {
        when(inventoryRepository.findByProductId(1L))
            .thenReturn(new Inventory(1L, 100));

        orderService.placeOrder(1L, 5);

        verify(orderRepository).save(any(Order.class));
    }
}
```

`@InjectMocks`는 생성자 주입 → 필드 주입 → Setter 주입 순으로 시도한다. 생성자가 명확하게 정의된 클래스라면 생성자 주입이 선택된다.

## 4. Stubbing 심화

단순한 값 반환 외에도 예외 던지기, 동적 응답, void 메서드 처리 등 다양한 Stubbing 방식을 지원한다.

### 4-1. 예외와 동적 응답

```java
@Test
void 재고_부족_시_예외_던지기() {
    when(inventoryRepository.findByProductId(anyLong()))
        .thenThrow(new OutOfStockException("재고 없음"));

    assertThrows(OutOfStockException.class
        , () -> orderService.placeOrder(1L, 5));
}

@Test
void 요청_파라미터_기반_동적_응답() {
    when(inventoryRepository.findByProductId(anyLong()))
        .thenAnswer(invocation -> {
            Long productId = invocation.getArgument(0);
            return new Inventory(productId, productId * 10);
            // productId가 3이면 재고 30을 반환한다
        });
}

@Test
void void_메서드_아무것도_하지_않기() {
    // void 메서드는 when().then() 대신 doNothing() 사용
    doNothing().when(notificationService).send(any());

    orderService.placeOrder(1L, 5);
    // 알림 전송 없이 주문만 처리된다
}
```

### 4-2. 연속 Stubbing

```java
@Test
void 연속_호출_다른_응답() {
    when(inventoryRepository.findByProductId(1L))
        .thenReturn(new Inventory(1L, 10))
        .thenReturn(new Inventory(1L, 5))
        .thenThrow(new RuntimeException("세 번째 호출은 실패"));
    // 첫 호출 → 재고 10, 두 번째 → 재고 5, 세 번째 → 예외
}
```

## 5. 검증 심화

`verify()`는 목 객체의 메서드 호출 횟수와 순서를 검증한다. 과도한 검증은 테스트를 구현 세부사항에 결합시키므로, 비즈니스 규칙에 핵심적인 호출만 검증하는 것이 좋다.

```java
@Test
void 호출_횟수_검증() {
    orderService.placeOrder(1L, 5);

    verify(inventoryRepository, times(1)).findByProductId(1L);
    verify(notificationService, never()).sendAlert(any());
    verify(orderRepository, atLeastOnce()).save(any());
    verify(cacheService, atMost(3)).update(anyLong());
}

@Test
void 호출_순서_검증() {
    orderService.placeOrder(1L, 5);

    InOrder inOrder = inOrder(inventoryRepository, orderRepository);
    inOrder.verify(inventoryRepository).findByProductId(1L);
    inOrder.verify(orderRepository).save(any(Order.class));
    // 재고 확인 후 주문 저장 순서가 반드시 지켜져야 한다
}
```

## 6. ArgumentCaptor

`ArgumentCaptor`는 목 메서드에 전달된 실제 인자를 캡처하여 검증한다. `any()`로는 "호출되었다"만 확인하지만, `ArgumentCaptor`로는 "어떤 값으로 호출되었는가"를 세밀하게 검증할 수 있다.

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Captor
    ArgumentCaptor<Order> orderCaptor;

    @Test
    void 저장된_주문_내용_검증() {
        when(inventoryRepository.findByProductId(1L))
            .thenReturn(new Inventory(1L, 100));

        orderService.placeOrder(1L, 3);

        verify(orderRepository).save(orderCaptor.capture());
        Order savedOrder = orderCaptor.getValue();

        assertEquals(1L, savedOrder.getProductId());
        assertEquals(3, savedOrder.getQuantity());
        assertEquals("PENDING", savedOrder.getStatus());
    }
}
```

## 7. Spy — 부분 모킹

`@Spy`는 실제 객체를 감싸서 일부 메서드만 Stub으로 대체한다. 실제 구현체의 대부분 동작은 유지하면서 특정 메서드만 제어할 때 사용한다. 단, 테스트가 구현 세부사항에 결합되므로 남용을 피해야 한다.

```java
@ExtendWith(MockitoExtension.class)
class PricingServiceTest {

    @Spy
    PricingService pricingService = new PricingService();

    @Test
    void 할인_계산_실제_로직_사용() {
        // getBasePrice만 Stub, 나머지는 실제 메서드 호출
        doReturn(10000).when(pricingService).getBasePrice(1L);

        int finalPrice = pricingService.calculateFinalPrice(1L, "VIP");

        // VIP 10% 할인 → 9000 (실제 할인 로직이 동작한다)
        assertEquals(9000, finalPrice);
    }
}
```

## 8. BDD 스타일

Mockito는 `BDDMockito`를 통해 **Given-When-Then** 구조에 자연스럽게 맞는 BDD 스타일 API를 제공한다. `when().thenReturn()` 대신 `given().willReturn()`을 사용하면 테스트 구조와 Stubbing 표현이 일치한다.

```java
import static org.mockito.BDDMockito.*;

@Test
void BDD_스타일_주문_테스트() {
    // Given
    given(inventoryRepository.findByProductId(1L))
        .willReturn(new Inventory(1L, 50));
    given(orderRepository.save(any(Order.class)))
        .willAnswer(invocation -> invocation.getArgument(0));

    // When
    Order result = orderService.placeOrder(1L, 3);

    // Then
    then(orderRepository).should(times(1)).save(any(Order.class));
    then(notificationService).should(never()).sendAlert(any());
    assertNotNull(result);
}
```

`then(mock).should()` 패턴은 `verify(mock)`와 동일하지만, Then 블록의 의도가 "이 행위가 발생했어야 한다"로 더 명확하게 전달된다.

## 9. Mockito 한계와 우회 방법

Mockito는 동적 프록시 기반으로 동작하기 때문에 몇 가지 제약이 있다. 이 제약을 만나면 설계를 재검토하거나 `mockito-inline`을 활용할 수 있다.

Mockito가 기본적으로 목킹할 수 없는 대상은 다음과 같다:

- `final` 클래스 및 `final` 메서드
- `static` 메서드
- 생성자(`new` 호출)
- `private` 메서드

`mockito-inline`(Mockito 4.x부터는 기본 통합)을 추가하면 `final` 클래스와 `static` 메서드를 목킹할 수 있다:

```java
// final 클래스 목킹
final class ExternalPaymentClient {
    String charge(long amount) { /* 외부 API 호출 */ return "ok"; }
}

@Test
void finalClass_목킹() {
    ExternalPaymentClient client = mock(ExternalPaymentClient.class);
    when(client.charge(5000L)).thenReturn("success");

    // mockito-inline 의존성이 있으면 final 클래스도 목킹 가능하다
    String result = client.charge(5000L);
    assertThat(result).isEqualTo("success");
}

// static 메서드 목킹 — try-with-resources로 스코프를 제한해야 한다
@Test
void staticMethod_목킹() {
    try (MockedStatic<UuidGenerator> mocked = mockStatic(UuidGenerator.class)) {
        mocked.when(UuidGenerator::generate).thenReturn("fixed-uuid-1234");

        Order order = orderService.placeOrder(1L, 3);

        assertThat(order.getTrackingId()).isEqualTo("fixed-uuid-1234");
    }
    // try 블록 밖에서는 원래 동작이 복원된다
}
```

`mockStatic()`은 반드시 try-with-resources 안에서 사용해야 한다. 블록을 벗어나지 않으면 static mock이 해제되지 않아 다른 테스트에 영향을 준다. static 메서드 의존이 많다면 래퍼 객체(Wrapper)로 감싸 인터페이스를 만드는 방식이 장기적으로 테스트 가능성을 높인다.
