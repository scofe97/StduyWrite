# 해시 알고리즘과 Set
---
> 해시는 임의 길이의 데이터를 고정 크기 숫자로 변환해 O(1) 검색을 가능하게 한다. Set은 이 원리 위에서 중복 없는 집합을 구현한다.

## 1. 해시 함수의 원리

해시 함수(hash function)는 임의 길이의 입력을 받아 고정 크기의 정수(해시 코드)를 반환한다. 중요한 점은 같은 입력에는 항상 같은 출력이 나와야 한다는 것이다. 자바에서 모든 객체는 `Object.hashCode()`를 통해 int 해시 코드를 갖는다.

```java
System.out.println("A".hashCode());   // 65
System.out.println("AB".hashCode());  // 2081
System.out.println("SET".hashCode()); // 83500
```

해시 코드 자체는 저장 위치가 아니다. 배열(버킷 배열)의 인덱스로 변환하려면 나머지 연산을 사용한다. 해시 코드가 음수일 수 있으므로 절댓값을 취한 뒤 배열 크기로 나눈다.

```java
int hashIndex = Math.abs(value.hashCode()) % capacity;
```

## 2. 해시 충돌

서로 다른 두 값이 같은 해시 인덱스를 가리킬 때 해시 충돌(hash collision)이 발생한다. 예를 들어 배열 크기가 10이면 9와 99는 모두 인덱스 9를 가리킨다. 충돌은 피할 수 없으므로 어떻게 처리하느냐가 성능을 결정한다.

### 체이닝(Separate Chaining)

충돌한 항목들을 같은 인덱스의 연결 리스트에 함께 저장한다. 자바 `HashMap`의 기본 방식이다. 삽입은 O(1)이지만 같은 버킷 안에서 선형 탐색이 필요하므로 최악의 경우 O(n)이 된다.

```java
LinkedList<Integer>[] buckets = new LinkedList[CAPACITY];

private void add(LinkedList<Integer>[] buckets, int value) {
    int hashIndex = value % CAPACITY;
    LinkedList<Integer> bucket = buckets[hashIndex];
    if (!bucket.contains(value)) {
        bucket.add(value); // 중복 없이 추가
    }
}
```

### 개방 주소법(Open Addressing)

충돌 시 다른 빈 슬롯을 탐사(probe)해 저장한다. 슬롯을 한 칸씩 이동하는 선형 탐사(Linear Probing), 제곱 간격으로 이동하는 제곱 탐사(Quadratic Probing), 두 번째 해시 함수로 간격을 정하는 이중 해싱(Double Hashing) 세 가지 방식이 있다. 모든 데이터가 배열 안에 저장되므로 캐시 효율이 좋지만, 적재율(load factor)이 높아질수록 탐사 횟수가 급증한다.

## 3. Java HashMap의 내부 구조

`HashMap`은 버킷 배열 위에 체이닝으로 구현된다. 초기 버킷 수는 16이며, 적재율이 기본값 0.75를 초과하면 배열을 2배로 늘리고 모든 항목을 재해싱(rehashing)한다.

자바 8부터는 같은 버킷의 항목이 8개를 넘으면 연결 리스트를 레드-블랙 트리(Red-Black Tree)로 변환한다. 이로써 최악의 경우 탐색이 O(n)에서 O(log n)으로 개선된다. 항목이 다시 6개 이하로 줄면 다시 연결 리스트로 되돌아간다.

```
버킷 배열[0] → null
버킷 배열[1] → [Entry] → [Entry]     ← 연결 리스트 (0~7개)
버킷 배열[2] → [TreeNode]            ← 레드-블랙 트리 (8개 이상)
...
```

## 4. hashCode()와 equals() 계약

해시 자료구조가 올바르게 동작하려면 `hashCode()`와 `equals()`가 반드시 일관성 있게 구현되어야 한다. 이 규칙을 계약(contract)이라 부르며, 두 가지 조건으로 요약된다.

- `a.equals(b)`가 `true`이면 `a.hashCode() == b.hashCode()`여야 한다.
- `hashCode()`가 같아도 `equals()`가 `false`일 수 있다(충돌).

이 중 첫 번째 조건이 깨지면 논리적으로 같은 두 객체가 서로 다른 버킷에 저장되어 검색에 실패한다. `hashCode()`만 재정의하고 `equals()`를 재정의하지 않으면 중복 저장이 발생한다. 반드시 함께 재정의해야 한다.

```java
public class Member {
    private String id;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Member member = (Member) o;
        return Objects.equals(id, member.id);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id);
    }
}
```

`Objects.hash()`는 여러 필드를 조합해 안정적인 해시 코드를 만드는 표준 방법이다. Java record는 모든 컴포넌트를 포함한 `equals()`와 `hashCode()`를 자동으로 생성한다.

## 5. HashSet — HashMap에 위임

`HashSet`은 내부적으로 `HashMap<E, Object>`를 갖는다. 요소를 추가하면 그 요소를 키로, 더미 상수 객체(`PRESENT`)를 값으로 `HashMap`에 저장한다. 즉 `HashSet`은 `HashMap`의 키 집합을 래핑한 자료구조다.

```java
// HashSet.add() 내부 (JDK 소스 단순화)
private static final Object PRESENT = new Object();

public boolean add(E e) {
    return map.put(e, PRESENT) == null;
}
```

삽입 순서를 보장하지 않으며, 특정 순서 없이 요소가 저장된다. `null`을 하나 저장할 수 있다.

## 6. LinkedHashSet — 삽입 순서 유지

`LinkedHashSet`은 `HashSet`을 상속하고, 내부적으로 `LinkedHashMap`을 사용한다. 각 버킷 항목이 이중 연결 리스트로도 연결되어 삽입 순서를 기억한다. 해시 기반 O(1) 검색과 삽입 순서 보장을 동시에 얻지만, 연결 리스트 포인터만큼 메모리를 추가로 사용한다.

```java
Set<String> linked = new LinkedHashSet<>();
linked.add("C");
linked.add("A");
linked.add("B");
System.out.println(linked); // [C, A, B] — 삽입 순서 유지
```

순서가 중요하지 않으면 `HashSet`, 삽입 순서가 필요하면 `LinkedHashSet`, 정렬된 순서가 필요하면 `TreeSet`을 선택한다.
