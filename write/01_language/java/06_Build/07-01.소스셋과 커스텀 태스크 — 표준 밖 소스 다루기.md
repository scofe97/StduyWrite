# 소스셋과 커스텀 태스크 — 표준 밖 소스 다루기
---
> Gradle 프로젝트는 보통 `src/main/java` 에 소스를 두면 알아서 컴파일된다. 그런데 *표준 위치 밖* 에 소스를 두고 *내가 정한 위치* 로 컴파일해야 할 때가 있다. 그 순간 알아야 하는 두 개념이 **소스셋(source set)** 과 **커스텀 태스크(task)** 다. 이 글은 그 둘을 개념부터 정리하고, 왜 "`src/main/java` 는 자동인데 다른 폴더는 손수 태스크를 짜야 하는가"에 답한다.

읽고 나면 소스셋이 무엇을 묶는 단위인지, Gradle 의 빌드가 왜 "태스크 그래프"인지, 그리고 표준 밖 소스를 컴파일·실행하려면 무엇을 직접 선언해야 하는지 설명한다.



## 0. 왜 이 개념이 필요했나

[05_JVM 의 클래스 로더 실습](../05_JVM/ch03_class-loading-mechanism/04-01a.톰캣%20클래스%20로더%20실습%20—%20로더가%20다르면%20타입이%20다르다.md)에서 한 가지 요구가 생겼다. 같은 이름(`com.acme.Greeter`)의 클래스를 *두 버전* 으로 동시에 두고, 어떤 클래스는 *앱 클래스패스 밖* 에서 따로 로딩해야 했다. 표준 `src/main/java` 한 곳에 두면 이게 불가능하다. 같은 이름 클래스 둘은 컴파일 충돌이고, 거기 둔 클래스는 자동으로 앱 클래스패스에 올라가 버린다.

해결은 소스를 표준 밖 폴더(`libsrc/`)에 두고, *내가 만든 태스크* 로 *내가 정한 위치* 에 따로 컴파일하는 것이었다. 그 과정에서 마주친 개념을 여기서 일반화한다. 실습 코드는 예시로만 인용하고, 주인공은 개념이다.



## 1. 빌드는 "태스크의 그래프"다

Gradle 을 이해하는 첫 문장은 이것이다 — **Gradle 의 빌드는 태스크(Task)들의 그래프다.** 컴파일·테스트·패키징이 각각 하나의 태스크이고, 태스크 사이의 "이게 먼저 끝나야 저게 시작된다"는 의존 관계가 그래프의 간선이 된다. 이 그래프는 사이클이 없어 **DAG(방향성 비순환 그래프, Directed Acyclic Graph)** 라 부른다.

```text
:compileJava ──> :classes ──> :jar ──> :assemble
     │                          
     └─ "compileJava 가 끝나야 classes 가 시작" 같은 의존이 간선
```

중요한 사실 하나. Gradle 은 이 그래프를 *실행 전* 에 먼저 통째로 계산한다. 빌드는 크게 세 단계를 거치는데, **초기화(Initialization) → 설정(Configuration) → 실행(Execution)** 순이다. 태스크 그래프는 설정 단계에서 완성되고, 실행 단계는 그 그래프를 순서대로 밟을 뿐이다. 그래서 "어떤 태스크가 어떤 태스크에 의존하는가"는 빌드 스크립트가 *미리 선언* 해야 한다.

### 태스크는 입력과 출력을 가진다

태스크의 또 다른 핵심은 **입력(input)과 출력(output)** 이다. 컴파일 태스크의 입력은 소스 파일과 의존성이고, 출력은 `.class` 파일이다. Gradle 은 이 입력·출력을 추적해서, 입력이 지난번과 같으면 태스크를 다시 실행하지 않고 건너뛴다. 빌드 로그의 `UP-TO-DATE` 가 이 상태다. 이를 **증분 빌드(incremental build)** 라 한다.

이 입력·출력 추적은 뒤에서 두 번 더 등장한다. 커스텀 태스크를 만들 때 입력(`source`)과 출력(`destinationDirectory`)을 *내가 지정* 한다는 점, 그리고 태스크 의존성이 입출력 연결로 *암묵적으로* 생긴다는 점에서다.



## 2. 소스셋 — 소스·클래스패스·출력을 묶는 단위

이제 핵심 개념이다. **소스셋(source set)** 은 Gradle 이 Java 소스를 다루는 기본 단위로, *세 가지를 하나로 묶는다.*

| 소스셋이 묶는 것 | 의미 |
|------------------|------|
| 소스 위치 | 이 소스셋의 `.java` 가 어디 있나 (예: `src/main/java`) |
| 컴파일 클래스패스 | 컴파일할 때 참조할 의존성·라이브러리 |
| 출력 위치 | 컴파일된 `.class` 를 어디 둘까 (예: `build/classes/java/main`) |

> 공식 정의 그대로다 — "Source sets tie together the location of source files, the compilation classpath including dependencies, and the destination for compiled class files." (Gradle 사용자 가이드)

Java 플러그인을 적용하면 기본 소스셋 둘이 자동으로 생긴다. **`main`**(운영 코드)과 **`test`**(테스트 코드)다. `main` 소스셋의 약속된 위치가 바로 `src/main/java` 이고, 출력은 `build/classes/java/main` 이다. 우리가 "`src/main/java` 에 두면 알아서 컴파일된다"고 느끼는 것은, `main` 소스셋이 그 위치를 *기본값(convention)* 으로 들고 있기 때문이다.

### 소스셋마다 컴파일 태스크가 자동 생성된다

여기가 1장의 "태스크 그래프"와 만나는 지점이다. **Java 플러그인은 소스셋마다 컴파일 태스크를 자동으로 만든다.** `main` 소스셋에는 `compileJava`, `test` 소스셋에는 `compileTestJava` 가 붙는다. 일반화하면 소스셋 `xxx` 에 대해 `compileXxxJava`(컴파일), `processXxxResources`(리소스 복사), `xxxClasses`(묶음)가 생긴다.

```text
main 소스셋  →  compileJava       →  build/classes/java/main
test 소스셋  →  compileTestJava   →  build/classes/java/test
```

즉 "소스셋을 정의한다"는 것은 단순히 폴더를 가리키는 게 아니라, **그 폴더를 컴파일하는 태스크와 출력 위치까지 한 묶음으로 등록** 하는 일이다.

### 표준 위치를 바꾸려면 — `srcDirs`

소스셋의 위치는 기본값일 뿐, 바꿀 수 있다. `srcDirs`(Kotlin DSL 은 `setSrcDirs`)로 `main` 의 소스 위치를 `src/main/java` 가 아닌 다른 곳으로 지정할 수 있다.

```kotlin
sourceSets {
    main {
        java {
            setSrcDirs(listOf("src"))   // src/main/java 대신 src 를 main 소스로
        }
    }
}
```

이 방법은 "표준과 *조금* 다른 위치"에 쓴다. 그런데 우리 실습처럼 *완전히 별도* 의 소스를 *여러 벌* 따로 컴파일해야 한다면, 소스셋을 더 만들거나 아예 컴파일 태스크를 직접 짜는 쪽이 깔끔하다. 그 이야기로 넘어간다.



## 3. 표준 밖 소스 — "자동"이 멈추는 지점

핵심 질문에 답할 차례다. **왜 `src/main/java` 는 자동인데 다른 폴더는 손수 태스크를 짜야 하는가?**

답은 2장에 이미 있다. `src/main/java` 가 자동인 이유는 *마법* 이 아니라, **`main` 소스셋이 그 위치를 기본값으로 알고 있어서** 다. Gradle 은 `main`·`test` 소스셋만 자동으로 안다. 그 외의 폴더(예: `libsrc/`)는 Gradle 에게 *그냥 파일이 든 폴더* 일 뿐이다. 어떤 소스셋에도 속하지 않으므로:

- Gradle 이 컴파일하지 않는다 (컴파일 태스크가 안 붙음)
- 출력 위치도 정해지지 않는다 (소스셋이 없으니 목적지도 없음)

그래서 표준 밖 폴더를 컴파일하려면 둘 중 하나를 *직접* 해야 한다.

1. **새 소스셋을 정의** 해서 그 폴더를 등록한다 (그러면 컴파일 태스크가 자동 생성된다)
2. **컴파일 태스크를 직접 만든다** — 입력 소스와 출력 위치를 손으로 지정한다

우리 실습은 2번을 택했다. 같은 이름 클래스를 여러 벌 따로, 그리고 출력 위치를 세밀하게 통제해야 했기 때문이다. 다음 장에서 그 모습을 본다.



## 4. 커스텀 태스크로 컴파일하기 — `source` 와 `destinationDirectory`

`JavaCompile` 은 Gradle 이 제공하는 컴파일 태스크 *타입* 이다. `compileJava` 도 사실 이 타입의 인스턴스다. 이 타입으로 태스크를 직접 만들면, 입력과 출력을 내가 정할 수 있다.

```kotlin
tasks.register<JavaCompile>("compileLabV1") {
    source = fileTree("libsrc/v1") { include("**/*.java") }  // ① 입력: 어느 소스를
    destinationDirectory.set(layout.buildDirectory.dir("labout/v1"))  // ② 출력: 어디로
    options.encoding = "UTF-8"
}
```

두 줄이 전부다.

- **① `source`** — "이 폴더의 `.java` 를 컴파일하라". 표준 소스셋이 없으니 입력을 직접 가리킨다.
- **② `destinationDirectory`** — "결과 `.class` 를 여기 둬라". 이것도 직접 정한다.

여기서 자주 하는 오해를 짚는다. `build/labout/v1` 같은 출력 경로는 **"일반 폴더의 특징"이 아니라 내가 `destinationDirectory` 로 지정한 값** 이다. Gradle 이 표준 밖 폴더를 보고 알아서 `labout` 에 넣어 주는 게 아니다. 경로 이름도 내가 알아보기 쉬우라고 붙인 것일 뿐, `build/foo/bar` 로 적었으면 거기로 갔다. *입력과 출력을 둘 다 손으로 정하는 것* 이 커스텀 컴파일 태스크의 본질이다.

이 그림을 1장의 입력·출력 추적과 겹쳐 보면 일관된다. `source` 가 입력, `destinationDirectory` 가 출력이고, Gradle 은 이 둘을 추적해 증분 빌드(`UP-TO-DATE`)를 적용한다. 표준 소스셋이 하던 일을 내가 손으로 했을 뿐, 태스크의 작동 원리는 똑같다.

### 소스가 컴파일 결과를 참조해야 한다면 — 클래스패스

컴파일하는 소스가 다른 곳의 클래스를 참조하면, 그 위치를 `classpath` 로 줘야 한다. 예를 들어 `libsrc/plugin` 의 구현이 `main` 의 인터페이스를 구현한다면:

```kotlin
tasks.register<JavaCompile>("compileLabPlugin") {
    source = fileTree("libsrc/plugin") { include("**/*.java") }
    classpath = files(tasks.named<JavaCompile>("compileJava").map { it.destinationDirectory })
    destinationDirectory.set(layout.buildDirectory.dir("labout/plugin"))
}
```

`classpath` 가 바로 2장에서 본 "소스셋이 묶는 세 가지" 중 *컴파일 클래스패스* 다. 소스셋 없이 태스크를 직접 만들면, 그 클래스패스도 내가 채워야 한다.



## 5. 태스크 의존성 — `dependsOn` 과 암묵적 연결

커스텀 컴파일 태스크를 만들었으면, 그게 *언제* 실행될지도 정해야 한다. 1장에서 본 대로 Gradle 은 태스크 그래프를 미리 계산하므로, "A 가 B 보다 먼저"를 *선언* 해야 한다. 방법은 둘이다.

**명시적 의존 — `dependsOn`.** "이 태스크 전에 저 태스크를 실행하라"를 직접 적는다.

```kotlin
tasks.register("compileLab") {
    dependsOn("compileLabV1", "compileLabV2", "compileLabPlugin")  // 셋을 먼저
}
```

**암묵적 의존 — 입출력 연결.** 한 태스크의 출력을 다른 태스크의 입력으로 연결하면, Gradle 이 의존을 *자동으로* 추론한다. 위 `compileLabPlugin` 예시에서 `classpath` 에 `compileJava` 의 출력을 넣었는데, 그 한 줄로 "`compileJava` 가 먼저 끝나야 한다"는 의존이 암묵적으로 생긴다. 별도 `dependsOn` 이 없어도 된다.

공식 가이드는 가능하면 *암묵적 연결* 을 권한다. 입출력을 제대로 연결하면 의존이 자동으로 맞춰지고, 누락이 줄기 때문이다. `dependsOn` 은 입출력 관계가 없는 순수한 순서 강제에 쓴다.



## 6. 컴파일한 것을 실행하기 — `JavaExec`

마지막 조각이다. 따로 컴파일한 클래스나 `main` 메서드를 실행하려면 **`JavaExec`** 태스크를 쓴다. 이건 "이 클래스패스로 이 `main` 클래스를 실행하라"는 태스크다.

```kotlin
tasks.register<JavaExec>("exp1") {
    dependsOn("compileLab")                              // ① 준비물 먼저
    classpath = sourceSets["main"].runtimeClasspath     // ② 무슨 클래스패스로
    mainClass.set("com.acme.lab.Exp1_TypeIdentity")     // ③ 어느 main 을
    systemProperty("lab.out", "...")                    // ④ 실행 옵션
}
```

`application` 플러그인을 적용하면 기본 `run` 태스크가 생기지만, 그건 *하나의* `mainClass` 만 가리킨다. 실험처럼 `main` 이 여럿이면 `JavaExec` 태스크를 *실험마다* 따로 등록하는 편이 낫다.

여기서 1·5장이 한 번에 모인다. `exp1` 은 `dependsOn("compileLab")` 으로 "실행 전에 모든 lab 컴파일을 끝내라"를 선언한다. 그래서 IDE 에서 클래스의 `main` 을 *직접* 실행하면(이 태스크를 안 거치면) 컴파일·옵션 주입이 빠져 실패한다. 빌드를 태스크 그래프로 보는 관점이 "왜 Gradle 태스크로만 실행되는가"를 설명한다.



## 7. 정리 — 표준의 편의와 그 바깥

Gradle 의 "`src/main/java` 에 두면 알아서 된다"는 편의는 **`main` 소스셋이 위치·클래스패스·출력을 기본값으로 묶어 두고, 그에 맞는 컴파일 태스크를 자동 생성** 하기에 성립한다. 그 편의가 닿지 않는 표준 밖 소스를 다루려면, 소스셋이 *대신 해주던 일* 을 내가 손으로 선언한다.

| 표준 소스셋이 자동으로 | 표준 밖에서 내가 직접 |
|------------------------|----------------------|
| 소스 위치(`src/main/java`) | `source = fileTree(...)` |
| 출력 위치(`build/classes/java/main`) | `destinationDirectory.set(...)` |
| 컴파일 클래스패스 | `classpath = files(...)` |
| 컴파일 태스크 생성·연결 | `tasks.register<JavaCompile>(...)` + `dependsOn` |

핵심은 빌드를 *태스크의 입력·출력·의존 그래프* 로 보는 시선이다. 그 시선이 잡히면 "자동"과 "수동"이 다른 마법이 아니라 *같은 메커니즘을 누가 선언하느냐의 차이* 임이 보인다.

- [ ] 소스셋이 묶는 세 가지(소스 위치·컴파일 클래스패스·출력 위치)를 말할 수 있는가?
- [ ] `main`/`test` 가 자동인 이유가 "소스셋이 위치를 기본값으로 안다"임을 설명할 수 있는가?
- [ ] 표준 밖 폴더가 컴파일 안 되는 이유(어떤 소스셋에도 안 속함)를 말할 수 있는가?
- [ ] 커스텀 `JavaCompile` 태스크에서 `source`·`destinationDirectory`·`classpath` 가 각각 소스셋의 무엇에 대응하는지 아는가?
- [ ] `dependsOn`(명시)과 입출력 연결(암묵) 두 가지 태스크 의존 선언 방식을 구분할 수 있는가?
- [ ] `JavaExec` 로 컴파일 결과를 실행할 때 `dependsOn` 으로 준비 태스크를 엮는 이유를 설명할 수 있는가?
