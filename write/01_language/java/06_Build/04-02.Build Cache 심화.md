# Build Cache 심화

---

> Build Cache의 활성화·원격 캐시·Java 프로젝트 특이점·디버깅·troubleshooting 일곱 줄을 한 자리에 묶는다. 04-01 §6의 짧은 요약을 받아 깊이 들어가는 짝 문서다.



## 1. 본 문서의 자리

> 의존성 캐시(03-01), configuration cache(04-01 §7)와 별개로 Build Cache는 task 출력 단위 캐시다.

본문 시리즈에는 세 가지 캐시가 등장한다. 의존성 캐시는 jar와 metadata를 보관하고(03-01 §7), configuration cache는 task graph 평가 결과를 보관한다(04-01 §7). Build Cache는 task의 출력(`.class`, 보고서 등)을 입력 sha 기반으로 보관한다. 셋은 디렉터리도 다르고 무효화 트리거도 다르다.

본 문서는 Build Cache 한 종에 집중한다. 활성화·원격 설정 → 어디서 효과 → Java 특이점 → 디버깅 → troubleshooting 순으로 한 번씩 본다.



## 2. 활성화와 원격 캐시 설정

> 옵션 한 줄과 property 한 줄로 켠다. 팀 단위로 공유하려면 원격 캐시를 둔다.

```bash
./gradlew build --build-cache
```

영구로 켜려면 `gradle.properties`에 다음을 둔다.

```properties
org.gradle.caching=true
```

단일 머신에서는 로컬 캐시(`~/.gradle/caches/build-cache-1/`)만으로도 효과가 있다. 같은 작업공간에서 두 번째 빌드부터 task 출력 재사용이 일어난다.

팀 단위로 공유하려면 원격 캐시 서버를 둔다.

```groovy
// settings.gradle
buildCache {
    local {
        enabled = true
    }
    remote(HttpBuildCache) {
        url = 'https://cache.example.com/cache/'
        push = (System.getenv('CI') == 'true')   // CI만 push, 로컬은 pull만
        credentials {
            username = providers.gradleProperty('cacheUser').get()
            password = providers.gradleProperty('cachePass').get()
        }
    }
}
```

`push = (CI == true)` 패턴이 표준이다. CI에서만 push하고 로컬은 읽기만 하면 손상된 cache 엔트리가 공유되는 사고를 막을 수 있다. Develocity(구 Gradle Enterprise) 같은 서버를 도입하면 cache 통계·hit ratio 모니터링·취약 task 식별까지 같이 본다.



## 3. 어디서 효과를 보는가

> CI·브랜치 전환·동료 변경 pull 세 자리에서 가장 큰 폭으로 시간이 줄어든다.

캐시가 효과를 내는 자리는 정해져 있다.

- **CI 깨끗한 작업공간**: 매 빌드마다 `clean`을 부르지만 같은 커밋이라면 캐시 hit으로 컴파일을 건너뛴다. 큰 빌드에서 CI 시간이 가장 큰 폭으로 줄어든다.
- **브랜치 전환**: 로컬에서 다른 브랜치로 갔다가 돌아오면 같은 커밋이므로 컴파일이 즉시 끝난다.
- **개발자가 동료 변경 pull**: 자기 작업과 무관한 변경이면 CI가 push해 둔 출력을 그대로 받아 로컬 컴파일 비용이 사라진다.

같은 빌드를 두 사람이 직접 push 공유하는 패턴(peer-to-peer)은 권장되지 않는다. 누군가 의도치 않게 입력을 바꿔 잘못된 출력을 push하면 다른 사람의 빌드가 깨진다. CI를 통해 검증된 빌드만 push하는 단방향 모델이 안전하다.

효과를 측정하려면 `--scan`을 함께 켠다. Build Scan에 cache hit/miss·각 task별 시간이 모두 노출되므로 어느 task가 캐시 효과를 가장 크게 받는지가 정량적으로 보인다.



## 4. Java 프로젝트 캐싱 특이점

> Java 빌드는 일반적으로 캐시되지만 다섯 가지 함정이 자주 나타난다.

### 4.1 ABI 기반 compile avoidance

의존 라이브러리가 변경되어도 ABI(02-01 §4)가 깨지지 않으면 다운스트림은 재컴파일을 건너뛴다. 메서드 시그니처·필드 타입·extends/implements 관계 같은 ABI 요소만 캐시 키에 영향을 준다. 메서드 본문 변경은 다운스트림 ABI에 보이지 않으므로 다운스트림 캐시는 hit 그대로다.

이 동작이 캐시와 결합해 큰 멀티모듈 빌드에서 가장 큰 효과를 낸다. 핵심 라이브러리 모듈에 한 줄 고치고도 그 라이브러리만 재빌드하면 된다 — 다운스트림은 캐시에서 그대로 가져온다.

### 4.2 annotation processor의 함정

어노테이션 프로세서는 컴파일러 입장에서 runtime classpath처럼 다뤄져, 프로세서 jar의 transitive 의존성이 모두 캐시 키에 들어간다. 한 줄짜리 프로세서 의존성 변경이 모든 모듈의 캐시 hit을 깨뜨릴 수 있다.

권장은 프로세서 classpath를 메인 의존성과 분리해 최소만 등록하는 것이다.

```groovy
dependencies {
    // 일반 컴파일 classpath
    implementation 'org.springframework.boot:spring-boot-starter-web'

    // 프로세서 classpath는 별도 — 최소 의존성만
    annotationProcessor 'org.projectlombok:lombok:1.18.34'
    annotationProcessor 'org.mapstruct:mapstruct-processor:1.6.2'
}
```

`tasks.withType(JavaCompile) { options.annotationProcessorPath = configurations.annotationProcessor }`로 명시적으로 분리하기도 한다. operator의 빌드처럼 `subprojects {}`에서 일괄 적용하면 프로세서 classpath가 모듈마다 일관되게 좁아진다.

### 4.3 테스트 task의 runtime classpath 정규화

jar 순서나 timestamp가 캐시 키에 영향을 주지 않도록 정규화되어 있어, 같은 의존성 그래프면 jar 파일의 순서가 바뀌어도 캐시 hit이 유지된다. 의존성 변경 없이 빌드 환경 차이로만 jar 순서가 흔들리는 케이스는 자동으로 처리된다.

### 4.4 통합 테스트 입력 선언

OS·환경 변수·외부 서비스가 테스트 결과를 바꾸는데 입력으로 선언되지 않으면 잘못된 캐시 hit이 발생한다. 통합 테스트가 외부 DB·메시지 브로커를 띄우는 경우 이 위험이 가장 크다.

```groovy
tasks.named('integrationTest', Test).configure {
    inputs.property('docker.image.version', providers.environmentVariable('DOCKER_IMAGE_VERSION'))
    inputs.property('os', org.gradle.internal.os.OperatingSystem.current().name)
}
```

`inputs.property()` 또는 `inputs.environment()`로 외부 변수를 명시 입력으로 등록한다. 의도치 않은 cache hit이 회귀를 숨기는 사고를 막는 안전장치다.

### 4.5 절대 경로 취약성

task 입력에 절대 경로가 들어가면 다른 머신에서는 캐시 hit이 안 된다. CI 컨테이너의 작업 디렉터리와 로컬 머신의 디렉터리가 다르므로 캐시가 완전히 분리된다.

```groovy
tasks.register('myTask', MyCustomTask) {
    @PathSensitive(PathSensitivity.RELATIVE)
    inputDir = layout.projectDirectory.dir('src/main/templates')
}
```

`@PathSensitive(RELATIVE)`나 `CommandLineArgumentProvider`로 상대 경로를 강제한다. 원격 캐시·distributed CI 환경에서는 사실상 필수다.



## 5. 캐시 디버깅

> 캐시 hit이 기대대로 일어나지 않을 때 진단 순서는 정해져 있다.

```bash
# 1) 디버그 로그로 입력 fingerprint와 캐시 키를 출력
./gradlew build -Dorg.gradle.caching.debug=true --info \
    | grep -iE "cache key|fingerprint"

# 2) incremental 동작 확인 — 같은 빌드를 두 번 부르고 두 번째가 UP-TO-DATE인지
./gradlew clean build
./gradlew build   # 두 번째: 모든 task가 UP-TO-DATE여야 함

# 3) relocatability 확인 — 같은 프로젝트를 두 디렉터리에서 빌드해 캐시가 공유되는지
cp -r project /tmp/project2
./gradlew -p /tmp/project2 build --build-cache

# 4) 캐시 미스의 첫 task 식별 — 의존 task가 미스로 떨어진 경우 그 위로 거슬러 올라간다
./gradlew build --build-cache --info | grep -iE "build-cache|FROM-CACHE|UP-TO-DATE|did not"
```

`-Dorg.gradle.caching.debug=true` 출력은 task별 입력 property와 sha를 모두 찍어 줘서 어느 입력이 변경됐는지 정확히 보인다. 두 빌드의 출력을 diff해서 변한 라인만 골라내면 진단이 빨라진다.

캐시 미스가 cascade로 번지는 경우 첫 미스 task가 핵심이다. 그 task 출력이 다음 task 입력에 들어가므로, 첫 미스를 잡으면 그 아래는 자동으로 풀린다. Build Scan의 timeline에서도 같은 정보를 시각적으로 본다.



## 6. 자주 만나는 troubleshooting

> 캐시가 흔들리는 가장 흔한 원인 일곱 가지다. CI 캐시가 의심스러우면 위에서 아래로 한 줄씩 점검한다.

| 증상 | 원인 | 해결 |
|------|------|------|
| 머신마다 캐시 hit이 다르다 | file encoding 차이 | `gradle.properties`에 `org.gradle.jvmargs=-Dfile.encoding=UTF-8` |
| 환경변수 변경 후 캐시가 안 깨진다 | env var는 자동 추적되지 않음 | `inputs.property()`로 명시 |
| Windows·Mac이 같은 캐시를 못 쓴다 | line ending 차이 (CRLF vs LF) | git의 `autocrlf=false` |
| timestamp 입력으로 매번 미스 | volatile input | task를 분리하거나 timestamp를 출력에서 제외 |
| 두 task가 같은 디렉터리에 쓴다 | overlapping output | `Sync` task로 머지하거나 출력 분리 |
| 같은 입력이 다른 archive를 만든다 | 비결정적 zip/jar | `preserveFileTimestamps = false`, `reproducibleFileOrder = true` |
| 서드 파티 플러그인이 캐시를 깨뜨린다 | 절대 경로·volatile 입력 추가 | 플러그인 입력 declaration 검토 또는 교체 |

각 항목은 [common caching problems 공식 문서](https://docs.gradle.org/current/userguide/common_caching_problems.html)에 더 자세히 기술돼 있다. 본 표는 그 핵심을 한 화면에 모은 형태다.



## 7. 한 사례 — operator의 캐시 운영

> 본 문서가 다룬 도구가 운영 빌드에 어떻게 적용돼 있는지.

operator는 현재 단일 머신 로컬 캐시 단계에 있다. `gradle.properties`에 `org.gradle.caching=true`만 켜져 있고 원격 캐시는 도입되지 않았다.

다음 방향으로 단계적 도입이 가능하다.

1. CI 노드에 로컬 캐시 디렉터리를 컨테이너 외부 볼륨으로 mount해 빌드 간 캐시를 보존한다.
2. Develocity 또는 자체 HTTP cache 서버를 두고 `push = (CI == true)`로 단방향 캐시를 운영한다.
3. annotation processor classpath 분리(§4.2)와 통합 테스트 입력 명시(§4.4)를 일괄 적용해 캐시 hit ratio를 안정화한다.
4. Build Scan을 도입해 캐시 효과를 정량적으로 추적한다.

특히 `subprojects {}`로 모든 모듈에 일괄 적용된 Lombok·MapStruct·QueryDSL 어노테이션 프로세서가 §4.2 함정에 가장 가깝다. 한 프로세서의 transitive 의존성 변화가 전 모듈 캐시를 깰 수 있어 분리 적용을 우선 검토할 가치가 있다.



## 8. 정리 — Build Cache 운영 체크리스트

> 새 캐시 도입이나 캐시 회귀 진단 시 따라가는 한 장.

| 점검 항목 | 위치 | 합격 신호 |
|----------|------|----------|
| 활성화 | `gradle.properties` `org.gradle.caching=true` | 두 번째 빌드에 `FROM-CACHE` 표시 |
| 원격 캐시 push 정책 | `settings.gradle` `push = (CI == true)` | CI만 push, 로컬은 pull |
| ABI compile avoidance | 멀티모듈 빌드에서 단일 모듈 변경 시 다운스트림 캐시 hit | dependent 모듈이 `UP-TO-DATE` |
| annotation processor 분리 | `annotationProcessor` configuration 최소 | 프로세서 변경 시 영향 모듈 한정 |
| 통합 테스트 입력 명시 | `inputs.property()`/`inputs.environment()` 등록 | 환경 변경 시 캐시 미스 발생 |
| 절대 경로 회피 | `@PathSensitive(RELATIVE)` 적용 | 다른 디렉터리에서 빌드 시 캐시 hit |
| 디버깅 도구 | `-Dorg.gradle.caching.debug=true` | 캐시 미스 시 입력 sha 비교 가능 |

체크리스트가 모두 통과하면 단일 머신 캐시는 안정적이다. 다음 단계는 원격 캐시·Build Scan 도입으로 팀 단위 효과를 낸다.
