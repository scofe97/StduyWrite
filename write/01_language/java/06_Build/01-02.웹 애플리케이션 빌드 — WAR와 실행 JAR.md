# 웹 애플리케이션 빌드 — WAR와 실행 JAR

---

> 일반 JAR는 클래스를 한 파일로 묶을 뿐입니다. 웹 애플리케이션은 그 위에 "누가 이 코드를 실행하는가"라는 규약을 더한 패키징, 곧 WAR와 실행 JAR를 씁니다.

[01-01](./01-01.패키지·JAR·클래스패스.md)에서 일반 JAR가 클래스패스 루트를 한 파일로 묶는다는 것을 봤습니다. 그런데 실제 웹 애플리케이션을 배포하려고 산출물을 열어 보면 `WEB-INF`나 `BOOT-INF` 같은 처음 보는 디렉터리가 나옵니다. [톰캣의 클래스 로더 아키텍처](../05_JVM/ch03_class-loading-mechanism/04-01.톰캣의%20클래스%20로더%20아키텍처.md)를 읽다 이 구조에서 막혔다면, 이 글이 그 산출물을 *빌드 관점*에서 풉니다.

이 글은 빌드 도구가 웹 산출물을 **어떻게 만드는가**에 집중합니다. 만들어진 산출물을 JVM이 **어떻게 읽는가**(JarLauncher, WebApp 로더)는 클래스 로딩 챕터가 따로 다루므로, 로더 부분은 개요만 짚고 그쪽으로 연결합니다.

- 빌드 관점(이 글): `bootJar`·`bootWar` 같은 태스크와 의존성 스코프가 산출물 구조를 어떻게 결정하는가.
- 로딩 관점(연결): [04-05](../05_JVM/ch03_class-loading-mechanism/04-05.Spring%20Boot%20실행%20JAR와%20클래스%20로딩.md)가 실행 JAR를, [04-01](../05_JVM/ch03_class-loading-mechanism/04-01.톰캣의%20클래스%20로더%20아키텍처.md)이 WAR를 어떤 로더로 읽는지 설명합니다.



## 1. 일반 JAR로는 부족한 이유

> 웹 애플리케이션은 "혼자 실행되거나" "컨테이너가 실행해 주거나" 둘 중 하나입니다. 일반 JAR는 둘 중 어느 쪽도 규약으로 담지 못합니다.

01-01의 계산기 실행 JAR는 `Main-Class` 하나만 있으면 `java -jar`로 떴습니다. 의존성도 클래스패스에 직접 넣거나 한 JAR에 펼쳐 담으면 됐습니다. 웹 애플리케이션은 두 가지가 더 필요합니다.

1. **첫째, 의존성이 수십~수백 개입니다.** 이를 한 JAR에 클래스 단위로 펼쳐 담으면 같은 경로의 리소스가 충돌하고 서명도 깨집니다. 의존성을 *JAR 그대로* 품는 구조가 필요합니다.
2. **둘째, 실행 주체가 둘로 갈립니다.** 애플리케이션이 톰캣을 내장해 스스로 뜨거나(실행 JAR), 외부에 떠 있는 톰캣이 산출물을 받아 실행하거나(WAR)입니다. 누가 실행하느냐에 따라 의존성을 산출물에 *넣을지 뺄지*가 달라집니다. 이 차이는 [01-01 §7.2](./01-01.패키지·JAR·클래스패스.md)에서 의존성 스코프로 처음 짚었고, 이 글 6절에서 빌드 태스크 관점으로 마무리합니다.

그래서 웹 빌드는 일반 `jar` 대신 Spring Boot의 `bootJar`나 표준 `war` 태스크로 산출물을 만듭니다. 각각의 구조를 차례로 봅니다.



## 2. Spring Boot 실행 JAR의 구조

> Gradle `bootJar` 태스크는 내 코드, 의존성, 부트 로더를 세 영역으로 나눠 담습니다. 진입점은 내 앱이 아니라 부트 로더입니다.

`./gradlew bootJar`로 만든 실행 JAR의 압축을 풀면 세 갈래가 보입니다. Spring Boot 공식 문서가 명시하는 구조입니다.

```text
example.jar
 ├─ META-INF
 │   └─ MANIFEST.MF
 ├─ org
 │   └─ springframework/boot/loader/...   # Spring Boot 로더 클래스
 └─ BOOT-INF
     ├─ classes
     │   └─ com/mycompany/project/...     # 내가 작성한 클래스
     └─ lib
         ├─ dependency1.jar               # 의존성 (JAR 그대로 중첩)
         └─ dependency2.jar
```

세 영역의 역할이 갈립니다.

- `BOOT-INF/classes`: **내가 작성한 클래스가 들어갑니다.** 01-01의 일반 JAR에서 `com/...`이 루트에 있던 자리가 여기로 한 단계 내려온 셈입니다.
- `BOOT-INF/lib`: **의존성이 *풀리지 않고 JAR 그대로* 담깁니다.** 1절에서 말한 "JAR를 품는 구조"가 이 디렉터리입니다.
- `org/springframework/boot/loader`: **실행을 담당하는 부트 로더 클래스입니다.** 내 앱이 아니라 *먼저 실행될* 코드입니다.

`bootJar` 태스크가 일반 `jar`와 다른 점이 여기서 드러납니다. 일반 `jar`는 클래스를 루트에 펼쳐 담고 끝이지만, `bootJar`는 의존성을 `BOOT-INF/lib`에 중첩하고 부트 로더를 함께 넣어 "혼자 뜰 수 있는" 산출물을 만듭니다.

### 2.1 fat JAR와 shaded JAR — 의존성을 담는 두 방식

실행 JAR를 이해하려면 "내 코드와 모든 의존성을 한 파일에 담는다"는 발상부터 짚어야 합니다. 이렇게 의존성까지 한 JAR에 다 담아 `java -jar`만으로 실행되게 만든 산출물을 통틀어 **fat JAR**(또는 uber JAR)라고 부릅니다. 의존성 JAR를 따로 들고 다니며 `-cp`에 일일이 나열하던 불편을 없앤 것입니다.

fat JAR를 *만드는 방식*은 두 갈래입니다. 같은 목표(한 파일로 실행)인데 의존성을 담는 방법이 다릅니다.

| 방식 | 의존성을 담는 법 | 만드는 도구 |
|------|------------------|-------------|
| **shaded(uber) JAR** | 모든 의존성의 `.class`를 풀어 한 JAR 루트에 *펼쳐 섞음* | Maven Shade, Gradle Shadow |
| **중첩(nested) JAR** | 의존성을 *JAR 파일 그대로* `BOOT-INF/lib`에 나란히 둠 | Spring Boot (`bootJar`) |

두 방식의 내부 모습을 차례로 봅니다. 먼저 **shaded JAR**는 모든 라이브러리의 클래스를 한 루트에 펼쳐 섞습니다.

```bash
app.jar  (shaded)
├─ com/mycompany/App.class           # 내 코드
├─ org/springframework/.../*.class   # 의존성 클래스가 전부 루트에 펼쳐져 섞임
├─ com/fasterxml/jackson/.../*.class #
├─ org/apache/tomcat/.../*.class     #
└─ META-INF/services/...             # ← 여러 라이브러리가 같은 경로를 두고 덮어쓰기 충돌
```

구조가 단순하지만 두 가지 문제가 있습니다. 

1. 첫째, 서로 다른 라이브러리가 같은 경로의 리소스(예: `META-INF/services/`의 같은 인터페이스 파일, `application.properties`)를 가지면 하나가 다른 하나를 덮어써 충돌합니다. 
2. 둘째, 산출물만 봐서는 어떤 라이브러리가 몇 버전 들어갔는지 알기 어렵습니다. (이름의 "shade(가리다)"는 충돌하는 패키지 이름을 바꿔 숨기는 relocation 기능에서 왔습니다. 충돌을 *이름 변경으로* 우회하는 방식입니다.)

반면 Spring Boot가 택한 **중첩 JAR**는 의존성을 *JAR 파일 그대로* 담습니다.

```bash
app.jar  (중첩, Spring Boot)
├─ BOOT-INF/
│   ├─ classes/
│   │   └─ com/mycompany/App.class        # 내 코드
│   └─ lib/
│       ├─ spring-webmvc-6.1.x.jar        # 의존성이 JAR 그대로
│       ├─ jackson-databind-2.17.x.jar    #   버전까지 파일명에 보임
│       └─ tomcat-embed-core-10.1.x.jar
└─ org/springframework/boot/loader/...    # 중첩 JAR를 읽는 전용 로더
```

- 각 라이브러리의 리소스가 자기 JAR 안에 격리돼 충돌하지 않고, `BOOT-INF/lib`를 열면 의존성 목록과 버전이 파일명으로 그대로 보입니다. 
- **대신 "JAR 안에 든 JAR"라는 비표준 구조가 되므로, 표준 실행기가 아닌 전용 로더가 있어야 읽힙니다**(4절).

### 2.2 BOOT-INF/lib는 왜 JAR를 풀지 않는가

2.1에서 본 중첩 방식을 `BOOT-INF` 안에서 한 번 더 들여다봅니다. 핵심은 의존성 JAR를 클래스 단위로 풀어 헤치지 않고 **JAR 파일 그대로** 넣고, 그 JAR들을 *어떤 순서로* 클래스패스에 올릴지는 옆에 둔 색인 파일에 적어 둔다는 점입니다.

```bash
BOOT-INF/
 ├─ classes/                      # ① 내 코드 (클래스패스 맨 앞)
 │   └─ com/mycompany/...
 ├─ lib/                          # ② 의존성 JAR (파일째로)
 │   ├─ spring-webmvc-6.1.x.jar
 │   ├─ spring-core-6.1.x.jar
 │   ├─ jackson-databind-2.17.x.jar
 │   └─ tomcat-embed-core-10.1.x.jar
 └─ classpath.idx                 # ③ lib 안 JAR를 올릴 "순서" 목록
```

`BOOT-INF/classpath.idx`는 ②의 `lib` 안 JAR들을 **어떤 순서로 클래스패스에 올릴지** 적은 색인입니다. 빌드 플러그인이 자동으로 만들며, 각 줄은 대시와 공백(`- `)으로 시작하고 경로는 큰따옴표로 감쌉니다(YAML 호환 형식).

```yaml
- "BOOT-INF/lib/spring-webmvc-6.1.x.jar"
- "BOOT-INF/lib/spring-core-6.1.x.jar"
- "BOOT-INF/lib/jackson-databind-2.17.x.jar"
- "BOOT-INF/lib/tomcat-embed-core-10.1.x.jar"
```

이 순서가 중요한 이유는 01-01 §5.4에서 본 "앞쪽 클래스가 뒤쪽을 가린다"는 규칙 때문입니다. 같은 클래스가 두 JAR에 있으면 클래스패스에서 앞선 JAR가 이기므로, 의존성을 올리는 순서가 어떤 버전이 쓰일지를 좌우합니다. 빌드 도구가 이 순서를 `classpath.idx`로 *고정*해 두면, 실행할 때마다 동일한 클래스패스가 재현됩니다. 즉 최종 클래스 탐색 순서는 `BOOT-INF/classes`(내 코드) → `classpath.idx`에 적힌 순서대로의 `BOOT-INF/lib/*.jar`가 됩니다.

이 색인은 `java -jar`로 실행할 때만 쓰입니다. IDE 실행이나 Gradle `bootRun`처럼 클래스패스를 이미 따로 구성한 경우에는 색인 대신 그 클래스패스를 그대로 씁니다. 부트 로더가 이 색인을 읽어 실제 클래스패스를 어떻게 구성하는지는 [04-05 §5](../05_JVM/ch03_class-loading-mechanism/04-05.Spring%20Boot%20실행%20JAR와%20클래스%20로딩.md)가 다룹니다.



## 3. 실행 JAR의 진입점 — Main-Class와 Start-Class

> 실행 JAR의 `Main-Class`는 내 앱이 아니라 부트 로더입니다. 내 앱의 main 클래스는 `Start-Class`에 따로 적힙니다.

01-01 §6에서 `java -jar`가 `MANIFEST.MF`의 `Main-Class`를 진입점으로 읽는다고 했습니다. Spring Boot 실행 JAR의 매니페스트를 열면 그 `Main-Class`가 내 앱이 아닙니다.

```manifest
Main-Class: org.springframework.boot.loader.launch.JarLauncher
Start-Class: com.mycompany.project.MyApplication
```

- `Main-Class`는 `JarLauncher`입니다. `java -jar`가 가장 먼저 실행하는 것은 Spring Boot 런처이지 내 앱이 아닙니다. 
- `Start-Class`가 내가 `@SpringBootApplication`을 붙인 진짜 main 클래스입니다. 런처가 준비를 마친 뒤 이 클래스를 찾아 실행합니다.

빌드 도구가 이 두 줄을 대신 기록합니다. Spring Boot 플러그인이 매니페스트를 다시 써 `Main-Class`와 `Start-Class`를 관리하므로, 일반 `jar` 플러그인 쪽에서 진입점을 건드리면 안 됩니다. 어느 클래스가 `Start-Class`가 되는지는 `@SpringBootApplication`이 붙은 클래스를 플러그인이 찾아 정합니다.

> ⚠️ 런처 패키지 이름은 버전에 따라 다릅니다. Spring Boot 3.2부터 `org.springframework.boot.loader.launch.JarLauncher`이고, 그 이전에는 `org.springframework.boot.loader.JarLauncher`였습니다.



## 4. 실행 JAR는 어떤 로더로 읽히나 (개요)

> 표준 실행기는 JAR 안에 든 JAR를 직접 읽지 못합니다. 그래서 `JarLauncher`가 중첩 JAR를 읽는 사용자 정의 로더를 세워 `Start-Class`를 실행합니다.

2절의 `BOOT-INF/lib/*.jar`는 "JAR 안의 JAR"입니다. 01-01 §6에서 짚었듯 표준 실행기는 이 중첩 JAR를 곧바로 읽지 못합니다. 그래서 `Main-Class`인 `JarLauncher`가 먼저 떠서, 중첩 JAR를 클래스패스처럼 읽을 수 있는 사용자 정의 클래스 로더를 만든 다음 `Start-Class`를 그 로더로 실행합니다.

이 글은 빌드 관점이므로 로더 동작은 여기까지만 짚습니다. `JarLauncher`의 기동 흐름, 중첩 JAR 오프셋 읽기, 사용자 정의 로더 상속 구조는 [04-05 §5](../05_JVM/ch03_class-loading-mechanism/04-05.Spring%20Boot%20실행%20JAR와%20클래스%20로딩.md)가 자세히 다룹니다. embedded Tomcat이 이 구조에서 어디에 올라오는지는 [04-05 §6](../05_JVM/ch03_class-loading-mechanism/04-05.Spring%20Boot%20실행%20JAR와%20클래스%20로딩.md)을 참고하면 됩니다.



## 5. 외부 톰캣 WAR의 구조

> WAR는 외부 서블릿 컨테이너가 실행하는 산출물입니다. 클래스는 `WEB-INF/classes`, 의존성은 `WEB-INF/lib`에 담깁니다.

외부 톰캣에 배포하는 WAR는 Spring Boot 실행 JAR와 영역 이름이 다릅니다. 표준 서블릿 규약을 따르기 때문입니다.

```text
example.war
 ├─ META-INF
 │   └─ MANIFEST.MF
 └─ WEB-INF
     ├─ classes
     │   └─ com/mycompany/project/...   # 내가 작성한 클래스
     ├─ lib
     │   └─ dependency1.jar             # 의존성
     └─ lib-provided
         └─ servlet-api.jar             # 컨테이너가 제공할 의존성 (실행 JAR 모드에서만 필요)
```

- `WEB-INF/classes`와 `WEB-INF/lib`는 톰캣이 그 웹앱 전용 탐색 루트로 삼습니다. 외부 톰캣 WAR에서는 실행 주체가 *이미 떠 있는 톰캣*이므로, Servlet API 같은 컨테이너 제공 API를 `WEB-INF/lib`에 넣으면 충돌이 납니다. 
- 그래서 그런 의존성은 `WEB-INF/lib-provided`로 따로 뺍니다. 이 디렉터리는 같은 WAR를 `java -jar`로 직접 띄울 때만 쓰이고, 외부 톰캣 배포 때는 컨테이너가 제공하므로 무시됩니다. 하나의 산출물로 "외부 배포"와 "단독 실행" 두 길을 모두 지원하려는 장치입니다.

WAR를 톰캣이 어떤 로더(Common·WebApp 로더)로 읽고 왜 격리하는지는 [04-01 §3](../05_JVM/ch03_class-loading-mechanism/04-01.톰캣의%20클래스%20로더%20아키텍처.md)이 다룹니다.



## 6. 의존성 스코프가 패키징 위치를 정합니다

> 같은 의존성이라도 빌드 스코프에 따라 `BOOT-INF/lib`에 들어가거나, `WEB-INF/lib-provided`로 빠지거나, 아예 산출물에서 제외됩니다.

[01-01 §7.2](./01-01.패키지·JAR·클래스패스.md)에서 provided 계열 스코프를 개념으로 짚었습니다. 여기서는 빌드 태스크가 그 스코프를 *어느 디렉터리*로 옮기는지로 마무리합니다. `implementation`·`api`·`compileOnly`·`runtimeOnly` 같은 의존성 스코프(Gradle에서는 configuration) 각각의 의미와 차이는 [02-01. Gradle 의존성 키워드](./02-01.Gradle%20의존성%20키워드.md)가 정본입니다. 이 글은 그 스코프가 *산출물 어디에 패키징되는가*만 다룹니다.

먼저 그림으로 봅니다. 같은 `build.gradle`에 적은 의존성이라도, *어떤 스코프로 선언했느냐*에 따라 빌드가 그 JAR를 산출물의 다른 자리에 놓거나 아예 빼 버립니다.

```text
build.gradle 의 의존성 선언
        │
        ├─ implementation 'jackson-databind'   ─┐  내가 실행에 직접 쓰는 라이브러리
        │                                        ▼
        │                                   ┌──────────────────────┐
        │                                   │  BOOT-INF/lib/        │  → 산출물에 포함
        │                                   │   jackson-databind.jar│
        │                                   └──────────────────────┘
        │
        ├─ providedRuntime 'tomcat-embed'    ─┐  단독 실행엔 필요, 외부 톰캣엔 불필요
        │                                      ▼
        │                                   ┌──────────────────────┐
        │                                   │  WEB-INF/lib-provided/│  → 따로 분리(외부 배포 시 무시)
        │                                   │   tomcat-embed.jar    │
        │                                   └──────────────────────┘
        │
        └─ compileOnly 'lombok'              ─┐  컴파일에만 필요, 런타임엔 불필요
                                               ▼
                                            ┌──────────────────────┐
                                            │  (산출물 어디에도 없음)│  → 완전 제외
                                            └──────────────────────┘
```

판단 기준은 1절에서 본 한 문장입니다 — *운영 시점에 누가 그 의존성을 제공하는가.* 내가 제공해야 하면 산출물에 넣고, 외부(컨테이너)가 제공하면 빼고, 빌드할 때만 필요하면 아예 담지 않습니다.

| 빌드·배포 형태 | 의존성 스코프 | 산출물 위치 | 의미 |
|----------------|---------------|-------------|------|
| Spring Boot 실행 JAR | `implementation` 계열 | `BOOT-INF/lib` | 내장 톰캣까지 애플리케이션 의존성이므로 함께 패키징 |
| Spring Boot 실행·배포 겸용 WAR | `providedRuntime` | `WEB-INF/lib-provided` | `java -jar` 실행엔 쓰되 외부 컨테이너와 충돌하지 않게 분리 |
| 외부 배포 WAR (컨테이너 제공 API) | `providedRuntime` | `WEB-INF/lib-provided` | 외부 톰캣이 런타임을 제공하므로 `WEB-INF/lib`에서 제외 |
| 어노테이션 프로세서·롬복 등 | `compileOnly` | 산출물에서 제외 | 컴파일에만 쓰이고 런타임엔 필요 없음 |

구체적인 예로 보면 이렇습니다. `jackson-databind`는 JSON 직렬화를 *런타임에* 직접 쓰므로 `implementation`으로 선언해 `BOOT-INF/lib`에 함께 담깁니다. 반대로 외부 톰캣에 WAR로 배포할 때 `tomcat-embed-core`는 컨테이너가 제공하므로 `providedRuntime`으로 빼 `WEB-INF/lib-provided`에 둡니다. 롬복처럼 컴파일 시점에만 코드를 생성하고 런타임엔 흔적이 없는 의존성은 `compileOnly`로 선언해 산출물에서 완전히 빠집니다.

Gradle에서 embedded 톰캣을 외부 배포 겸용으로 빼려면 런타임 의존성을 `providedRuntime` 구성으로 옮깁니다.

```gradle
dependencies {
    providedRuntime 'org.springframework.boot:spring-boot-starter-tomcat'
}
```

- 여기서 `compileOnly`가 아니라 `providedRuntime`을 쓰는 이유가 있습니다. `compileOnly` 의존성은 테스트 클래스패스에 올라가지 않아 웹 통합 테스트가 실패할 수 있습니다. 
- `providedRuntime`은 산출물의 `WEB-INF/lib`에서는 빼면서도 테스트 런타임에는 남겨 두므로, 외부 배포와 테스트를 함께 지원합니다.



## 7. 한 장으로 비교합니다

> 일반 JAR, 실행 JAR, WAR는 같은 ZIP 아카이브지만 진입점과 실행 주체가 다릅니다.

| 항목 | 일반 JAR | Spring Boot 실행 JAR | 외부 톰캣 WAR |
|------|----------|----------------------|---------------|
| 빌드 태스크 | `jar` | `bootJar` | `war` (Spring Boot는 `bootWar`) |
| 내 클래스 위치 | 루트 (`com/...`) | `BOOT-INF/classes` | `WEB-INF/classes` |
| 의존성 위치 | 없음 / 펼쳐 담음 | `BOOT-INF/lib` (JAR 중첩) | `WEB-INF/lib` |
| 진입점 (`Main-Class`) | 내 main 클래스 | `JarLauncher` (내 앱은 `Start-Class`) | 없음 (컨테이너가 실행) |
| 실행 주체 | `java -jar` | 부트 로더가 `java -jar`로 자립 | 외부 서블릿 컨테이너 |
| 읽는 로더 | 애플리케이션 로더 | 부트 사용자 정의 로더 ([04-05](../05_JVM/ch03_class-loading-mechanism/04-05.Spring%20Boot%20실행%20JAR와%20클래스%20로딩.md)) | 톰캣 WebApp 로더 ([04-01](../05_JVM/ch03_class-loading-mechanism/04-01.톰캣의%20클래스%20로더%20아키텍처.md)) |

- 세 산출물은 모두 01-01의 "JAR는 ZIP 기반 클래스패스 루트"라는 토대 위에 서 있습니다. 달라지는 것은 내 코드와 의존성을 어느 영역에 두느냐, 그리고 누가 그 영역을 읽어 실행하느냐입니다. 
- 빌드 도구는 선택한 태스크와 의존성 스코프에 맞춰 이 구조를 자동으로 만듭니다.
