# 실습 — 저장소와 캐시

---

> 03-01이 다룬 저장소 우선순위·SNAPSHOT TTL·캐시 디렉터리·`--refresh-dependencies`/`--offline`을 손으로 한 번씩 만난다. 실제 jar가 어디 들어가고 언제 다시 받아 오는지 디렉터리를 열어 가며 확인한다.



## 실습 1. 캐시 디렉터리 구조 직접 탐색

> Gradle이 받은 jar와 metadata가 어디에 어떤 이름으로 저장되는지 한 번 본다.

빈 빌드를 한 번 돌린다.

```bash
./gradlew :app:dependencies > /dev/null
```

이후 사용자 홈의 캐시 디렉터리를 들여다본다.

```bash
ls -la ~/.gradle/caches/modules-2/files-2.1/org.springframework.boot/spring-boot-starter-web/3.2.3/
```

각 버전 디렉터리 아래에는 sha1 해시 디렉터리가 있고, 그 안에 jar와 POM·`.module` 파일이 함께 들어 있다(03-01 §1·§7). metadata는 같은 트리의 `metadata-2.X/`에 별도로 보관된다.

```bash
ls ~/.gradle/caches/modules-2/metadata-2.107/descriptors/ | head -5
```

build cache(`build-cache-1/`)는 의존성 캐시와 디렉터리 자체가 다르다는 점도 함께 확인한다.



## 실습 2. `--refresh-dependencies` 동작 확인

> metadata가 강제로 다시 조회되는지를 빌드 로그로 본다.

먼저 정상 빌드를 한 번 돌리고 다시 같은 명령을 부른다. 두 번째 호출은 metadata 다운로드 없이 즉시 끝나는 모습이 보인다.

```bash
./gradlew :app:dependencies --info | grep -i "Downloading\|Cached"
./gradlew :app:dependencies --info | grep -i "Downloading\|Cached"   # 두 번째: 거의 캐시
```

이제 강제 갱신을 부른다.

```bash
./gradlew :app:dependencies --refresh-dependencies --info | grep -i "Downloading"
```

`Downloading https://...` 로그가 다시 줄줄이 출력되면 metadata 재조회가 동작한 것이다(03-01 §9). jar는 sha1이 같으면 다시 다운로드되지 않으므로 비용은 metadata trip뿐이다.



## 실습 3. `--offline`으로 외부 호출 차단

> 캐시에 없는 의존성이 있을 때 어떻게 실패하는지 본다.

먼저 현재 빌드가 캐시만으로 돌아가는지 검증한다.

```bash
./gradlew :app:build --offline
```

캐시에 없는 좌표가 하나라도 있으면 즉시 실패한다 — 이 동작이 의도다. 실제로 현재 잘 빌드되는 프로젝트에 새 의존성을 한 줄 추가한 뒤 다시 부르면 다음 같은 메시지로 실패한다.

```
> No cached version of org.example:new-lib:1.0 available for offline mode.
```

`--offline`은 비행기·격리 환경 외에도 "이 빌드가 외부 호출 없이 재현되는가?"를 검증하는 도구다(03-01 §9).



## 실습 4. SNAPSHOT 캐시 TTL 동작 확인

> 같은 SNAPSHOT 좌표를 짧은 간격으로 두 번 빌드하면 두 번째 trip이 어떻게 변하는지 본다.

사내 Nexus에 SNAPSHOT 좌표를 publish할 환경이 없다면 mavenLocal을 SNAPSHOT 저장소로 활용한다.

```groovy
// my-snapshot-lib 프로젝트
plugins {
    id 'java-library'
    id 'maven-publish'
}

group = 'org.example'
version = '1.0.0-SNAPSHOT'

publishing {
    publications {
        mavenJava(MavenPublication) {
            from components.java
        }
    }
}
```

```bash
./gradlew publishToMavenLocal
```

다른 빌드(`app`)에서 이를 의존하고 빌드한다.

```groovy
repositories {
    mavenLocal()
}
dependencies {
    implementation 'org.example:my-snapshot-lib:1.0.0-SNAPSHOT'
}
```

```bash
./gradlew :app:build --info | grep "my-snapshot-lib"
```

이제 `my-snapshot-lib` 코드를 수정하고 다시 `publishToMavenLocal`을 부른 뒤 `app`을 빌드한다. 24시간 안에는 metadata가 캐시되어 변경이 잡히지 않을 수 있다(03-01 §8). 다음 옵션으로 즉시 갱신을 강제하면 새 SNAPSHOT이 잡힌다.

```bash
./gradlew :app:build --refresh-dependencies
```

또는 `build.gradle`에 일시적으로 다음을 추가한다.

```groovy
configurations.all {
    resolutionStrategy.cacheChangingModulesFor 0, 'seconds'
}
```



## 실습 5. content 필터 효과 측정

> 사내 그룹만 사내 저장소에 묻게 만드는 필터가 어떻게 작동하는지 로그로 본다.

다음과 같이 두 저장소를 선언한다.

```groovy
repositories {
    mavenCentral()
    maven {
        url 'https://nexus.example.com/repository/maven-snapshots/'
        content {
            includeGroup 'org.okestro'
        }
    }
}
```

`./gradlew :app:dependencies --info`로 빌드하면 외부 라이브러리는 mavenCentral에만 trip하고, `org.okestro:*` 좌표는 사내 Nexus에만 trip한다. 필터를 빼면 외부 라이브러리도 사내 Nexus에 메타데이터를 묻는 trip이 추가되는 모습이 보인다(03-01 §6).

차이는 `--info` 로그의 `Resource missing` / `Cached resource` / `Downloading` 줄을 카운트하면 정량적으로 측정된다.



## 실습 점검 체크리스트

> 다섯 실습이 끝나면 다음 질문에 답할 수 있어야 한다.

```
1. ~/.gradle/caches/modules-2/files-2.1/ 아래의 디렉터리 구조를 그릴 수 있는가?
2. --refresh-dependencies와 --offline의 동작 차이를 한 단락으로 설명할 수 있는가?
3. SNAPSHOT의 24시간 TTL이 어디서 설정되고 어떻게 0초로 만드는지 코드로 보였는가?
4. content { includeGroup ... } 필터가 metadata trip에 어떤 영향을 주는지 로그로 확인했는가?
5. 사고가 났을 때 (--refresh-dependencies → 캐시 디렉터리 일부 삭제 → --offline) 진단 순서를 따를 수 있는가?
```
