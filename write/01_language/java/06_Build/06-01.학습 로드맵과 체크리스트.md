# 학습 로드맵과 체크리스트

---

> Gradle은 처음엔 `build.gradle` 파일 한 개처럼 보이지만 실제로는 의존성·태스크·멀티모듈·캐시·배포 산출물·CI 재현성을 다루는 작은 운영 언어에 가깝다. 본 문서는 본 시리즈(02-01 ~ 05-01)를 어떤 순서로 보고 무엇을 손에 익혀야 하는지를 메타로 정리한다. 실습은 각 본문에 짝으로 붙는 `02-01a` ~ `05-01a`로 분리돼 있다.



## 1. 공식 문서 우선순위 5선

> 개인 블로그보다 공식 문서가 먼저다. Gradle 버전이 빠르게 움직이면서 옛 글의 문법은 자주 무너진다.

| 우선순위 | 문서 | 본 시리즈 매핑 |
|---------|------|---------------|
| 1 | Gradle User Manual — Gradle Basics | 02-01 ~ 05-01 전체의 기반 |
| 2 | Gradle User Manual — Dependency Management | 02-01·03-01 |
| 3 | Spring Boot Gradle Plugin Reference | 04-01 §10·05-01 §1·§3 |
| 4 | Gradle User Manual — Build Cache·Performance | 04-01 §6·§7 |
| 5 | Gradle Blog | 최근 동향(rate limit·Configuration Cache 안정화) |

Gradle 공식 문서는 **10개 핵심 개념**(Build·Project·Task·Build Script·Plugin·Dependency·Gradle Wrapper·Settings File·Source Code·Gradle Directory)을 Gradle Basics 페이지에서 한 번에 정리한다. 본 시리즈는 그 10개 개념을 다음 자리에서 다룬다.

| 공식 개념 | 정의 | 본 시리즈 위치 |
|----------|------|---------------|
| Build | 산출물을 만드는 전체 과정 | 04-01 §1, 05-01 §1 |
| Project | 빌드의 단위(애플리케이션·라이브러리) | 02-01 §0 디렉터리 그림 |
| Task | 빌드의 최소 작업 단위 | 04-01 §2·§3 |
| Build Script | `build.gradle(.kts)` | 02-03 §3 Groovy DSL |
| Plugin | Gradle 기능 확장 | 02-01 §5, 05-01 §5 |
| Dependency | 외부·내부 라이브러리 의존성 | 02-01 §1~§9, 03-01 |
| Gradle Wrapper | 프로젝트 고정 Gradle 실행기 | 04-01 §8, 05-01 §7 |
| Settings File | `settings.gradle` — 모듈 등록 | 02-01 §0·§7, 05-01 §6 |
| Source Code | `src/main`·`src/test` | 02-01 §3, 04-01 §5 SourceSet |
| Gradle Directory | `gradle/wrapper`·`libs.versions.toml` | 02-01 §0, 04-01 §8 |

10개 개념을 통과하면 빌드 스크립트 한 줄을 봤을 때 어느 자리의 도구인지 분류가 된다.

## 3. Spring Boot build.gradle 점검 체크리스트 — 11문항

> 새 모듈을 추가하거나 기존 빌드를 검토할 때 위에서 아래로 묻는다.

```
1. 이 모듈은 실행 모듈인가, 라이브러리 모듈인가?
2. 실행 모듈이면 bootJar { enabled = true } + jar { enabled = false }인가?
3. 라이브러리 모듈이면 java-library 플러그인이 적용됐는가?
4. 의존성이 implementation·api 중 의도한 키워드로 선언됐는가?
5. Lombok/MapStruct/QueryDSL 어노테이션 프로세서가 빠지지 않았는가?
6. 테스트 의존성은 testImplementation에만 있는가?
7. 루트 build.gradle의 subprojects 설정이 이 모듈에도 적용되는가?
8. settings.gradle에 이 모듈이 include 되어 있는가?
9. CI 명령이 정확한 task path(:module:task)를 부르는가?
10. 의존성 캐시·build cache·configuration cache가 켜져 있는가?
11. 산출물 경로가 Jenkins/ArgoCD/Dockerfile/Jib가 기대하는 위치와 일치하는가?
```

`./gradlew :module:dependencies`와 `./gradlew :module:tasks --all`이 1·9·10번을 한 번에 검증해 준다. 7번은 모듈에서 `./gradlew :module:properties`로 적용된 plugin·convention을 확인할 수 있다.

## 5. 자주 빠지는 함정과 신호

> 옛 글·옛 코드에서 자주 만나는 패턴이며 새 코드에는 두지 않는다.

| 함정 | 신호 | 권장 |
|------|------|------|
| `compile`·`testCompile` 키워드 | Gradle 7+에서 빌드 실패 또는 deprecated | `implementation`·`testImplementation` |
| `allprojects {}` 또는 `subprojects {}` 남용 | 모듈마다 다른 설정이 필요한데 일괄 적용으로 충돌 | convention plugin (05-01 §6) |
| 라이브러리 모듈에 `bootJar` 자동 적용 | 다른 모듈이 의존할 일반 jar가 사라짐 | `bootJar { enabled = false }` 또는 boot 플러그인 미적용 |
| `buildscript { dependencies { classpath ... } }` | 옛 plugin 적용 방식, 가독성·재현성 약함 | `plugins {}` 블록 |
| 같은 라이브러리 두 버전 채택을 모르는 채 운영 | startup 시 NoSuchMethodError | `dependencyInsight`로 추적 |
| 잘못된 task path (`./gradlew app:build` 같은 오타) | "Task 'app:build' not found" | `:app:build`처럼 root 기준 경로 |
| BOM 미적용으로 의존성 버전 자동 결정 | 같은 모듈이 다른 빌드에서 다른 버전 채택 | `mavenBom` 또는 `platform()` |
| credentials를 빌드 스크립트에 평문 | 빌드 로그·git에 노출 | `gradle.properties` 또는 환경변수 |
| `mavenLocal()`을 운영 빌드에 추가 | 머신마다 다른 jar로 빌드 결과 흔들림 | 사내 Nexus group repository만 |
| daemon 잔재로 OOM | `--status`에 다수 daemon | CI는 `--no-daemon`, 로컬은 `--stop` 주기 |

함정마다 신호가 분명하다. "빌드가 갑자기 깨졌다", "내 로컬에선 되는데 CI에서 안 된다", "지난주엔 4분이던 빌드가 7분이다" 같은 증상이 위 표 어딘가와 매칭된다.
