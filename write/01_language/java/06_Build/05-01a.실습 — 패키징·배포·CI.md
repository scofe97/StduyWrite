# 실습 — 패키징·배포·CI

---

> 05-01가 다룬 산출물·layered jar·OCI 이미지 빌드·maven-publish·CI 통합을 손으로 한 번씩 만든다. Spring Initializr 앱 한 개로 시작해 컨테이너 이미지·라이브러리 publish·Jenkinsfile까지 여섯 단계로 본다.



## 실습 1. 산출물 경로 확인

> bootJar의 산출물이 정확히 어디 떨어지는지를 본다.

`./gradlew :app:bootJar`를 실행한 뒤 build 디렉터리를 들여다본다.

```bash
ls -la build/libs/
ls -la build/classes/java/main/   # 컴파일된 .class
ls -la build/resources/main/      # application.yml 등 리소스
ls -la build/reports/             # test report 등
```

`build/libs/<name>-<version>.jar`이 표준 위치다(05-01 §1). Jenkinsfile이나 Dockerfile에서 산출물을 참조할 때 이 경로를 그대로 쓴다.

산출물 이름을 바꾸고 싶으면 `bootJar { archiveBaseName = 'app' }`를 추가하면 된다. 같은 디렉터리에 `app-<version>.jar`로 만들어진다.



## 실습 2. Layered jar 추출

> fat jar 안의 layer를 풀어 Dockerfile multi-stage에서 어떻게 쓰이는지 본다.

bootJar는 Spring Boot 2.5+에서 layered가 기본 활성화돼 있다. 산출물을 직접 풀어 layer 구조를 본다.

```bash
mkdir -p /tmp/layers && cd /tmp/layers
java -Djarmode=layertools -jar <project>/build/libs/<name>-<version>.jar list
```

다음 같은 4개 layer가 출력된다.

```
dependencies
spring-boot-loader
snapshot-dependencies
application
```

각 layer를 추출한다.

```bash
java -Djarmode=layertools -jar <project>/build/libs/<name>-<version>.jar extract
ls -la
```

`dependencies/`, `spring-boot-loader/`, `snapshot-dependencies/`, `application/` 네 디렉터리가 풀려 나온다. 변경 빈도가 낮은 layer를 Dockerfile에서 먼저 `COPY`하면 컨테이너 캐시 효율이 산다(05-01 §2).



## 실습 3. `bootBuildImage`로 OCI 이미지 빌드

> Dockerfile 없이 Buildpacks로 컨테이너 이미지를 만든다.

Docker Desktop·colima·podman 중 하나가 띄워져 있어야 한다.

```bash
./gradlew :app:bootBuildImage --imageName=local/practice-app:0.1
docker images | grep practice-app
docker run --rm -p 8080:8080 local/practice-app:0.1
```

8080 포트로 접속해 앱이 동작하면 통과다(04-01 §10.4, 05-01 §3.1). `docker history local/practice-app:0.1`로 layer 분리가 어떻게 됐는지 살펴 보면 Buildpacks가 자동으로 layer를 자르는 모습이 보인다.



## 실습 4. Jib로 daemonless 이미지 빌드

> Docker daemon 없이 같은 이미지를 만드는 흐름을 본다.

`build.gradle`에 jib 플러그인을 추가한다.

```groovy
plugins {
    id 'com.google.cloud.tools.jib' version '3.4.0'
}

jib {
    from {
        image = 'eclipse-temurin:17-jre-alpine'
    }
    to {
        image = 'local/practice-jib:0.1'
    }
}
```

로컬 Docker daemon에 빌드(검증용).

```bash
./gradlew :app:jibDockerBuild
docker run --rm -p 8080:8080 local/practice-jib:0.1
```

또는 Docker daemon 없이 레지스트리에 직접 push (실제 레지스트리가 있을 때).

```bash
./gradlew :app:jib \
    -Djib.to.image=registry.example.com/practice-jib:0.1 \
    -Djib.to.auth.username=$REG_USER \
    -Djib.to.auth.password=$REG_PASS
```

bootBuildImage와 비교해 빌드 시간·이미지 크기·layer 분리를 본다(05-01 §3.3). 같은 앱이라도 도구마다 결과 이미지가 조금씩 다르다.



## 실습 5. `maven-publish`로 mavenLocal에 라이브러리 publish

> 사내 Nexus가 없는 환경에서 publish 흐름을 손으로 본다.

별도 라이브러리 모듈을 만들어 publish한다.

```groovy
// my-lib/build.gradle
plugins {
    id 'java-library'
    id 'maven-publish'
}

group = 'org.example.practice'
version = '0.1.0'

publishing {
    publications {
        mavenJava(MavenPublication) {
            from components.java
        }
    }
}
```

```bash
./gradlew :my-lib:publishToMavenLocal
ls -la ~/.m2/repository/org/example/practice/my-lib/0.1.0/
```

`my-lib-0.1.0.jar`, `my-lib-0.1.0.pom`, `my-lib-0.1.0.module`(Gradle Module Metadata)가 함께 생긴다. 다른 빌드에서 `mavenLocal()` 저장소를 추가한 뒤 `implementation 'org.example.practice:my-lib:0.1.0'`으로 받아 쓰면 publish가 동작한 것이다(05-01 §4).

사내 Nexus URL이 있으면 `repositories { maven { url '...' } }`로 바꾸고 `./gradlew publish`를 부르면 된다.



## 실습 6. Jenkinsfile에서 `:app:build` step 작성

> 멀티모듈 task path가 어떻게 호출되는지 declarative pipeline으로 본다.

```groovy
// Jenkinsfile (실습용 — 실제 사내 환경에 적용하지 않음)
pipeline {
    agent { label 'java17' }

    stages {
        stage('Build') {
            steps {
                sh '''
                    ./gradlew :app:build \
                        --no-daemon --build-cache \
                        -Dci=true
                '''
            }
        }

        stage('Image') {
            steps {
                sh './gradlew :app:jib --no-daemon -Djib.to.image=local/ci-app:${BUILD_NUMBER}'
            }
        }
    }

    post {
        always {
            junit 'app/build/test-results/test/*.xml'
            archiveArtifacts artifacts: 'app/build/libs/*.jar', fingerprint: true
        }
    }
}
```

이 Jenkinsfile을 실제로 돌려 보지 않더라도 다음 사항을 확인한다(05-01 §8).

- task path가 `:app:build`처럼 모듈을 명시했는가
- `--no-daemon`이 켜져 있는가 (CI 컨테이너용)
- credentials가 평문이 아닌 Jenkins credential store에서 받는 형태인가
- 산출물 path(`app/build/libs/*.jar`)가 실제 위치와 일치하는가

