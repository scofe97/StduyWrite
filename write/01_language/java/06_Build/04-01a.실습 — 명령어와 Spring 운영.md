# 실습 — 명령어와 Spring 운영

---

> 04-01에서 다룬 빌드 phase·task 모델·캐시 옵션·Spring Boot 플러그인을 손으로 한 번씩 만난다. Spring Initializr에서 시작해 bootJar/jar 분기와 빌드 캐시 효과까지 다섯 단계로 본다.



## 실습 1. Spring Initializr로 단일 프로젝트 생성, bootJar 실행

> 가장 단순한 Spring Boot Gradle 프로젝트가 무엇을 만드는지 본다.

`https://start.spring.io`에서 다음 옵션으로 프로젝트를 받는다.

- Project: Gradle - Groovy
- Language: Java
- Spring Boot: 3.2.x
- Dependencies: Spring Web

압축을 풀고 `./gradlew bootJar`를 실행한다.

```bash
./gradlew bootJar
ls -la build/libs/
```

`*-0.0.1-SNAPSHOT.jar` 한 개가 생긴다. fat jar이므로 직접 실행된다.

```bash
java -jar build/libs/*-0.0.1-SNAPSHOT.jar
```

8080 포트에서 Tomcat이 뜨는 모습이 보이면 통과다(04-01 §10.1). `jar` 파일을 unzip해서 `BOOT-INF/lib/`에 의존성 jar들이 들어 있는 구조도 한 번 확인한다.



## 실습 2. bootJar/jar 분기 동작 확인

> `bootJar { enabled = false }` + `jar { enabled = true }`로 두면 산출물이 어떻게 바뀌는지 본다.

`build.gradle`에 다음을 추가한다.

```groovy
bootJar { enabled = false }
jar { enabled = true }
```

다시 빌드한다.

```bash
./gradlew clean build
ls -la build/libs/
```

이번에는 일반 jar가 만들어진다. `java -jar`로 실행하면 메인 클래스가 없다고 실패한다 — 이 jar는 라이브러리용이지 실행 산출물이 아니다(04-01 §10.1).

다시 두 옵션을 원래대로 돌려 놓고 빌드해 두 산출물의 차이가 분명하게 손에 잡히도록 한다.



## 실습 3. `--build-cache` 활성화 전후 빌드 시간 비교

> 같은 task의 두 번째 실행이 캐시에서 가져와 즉시 끝나는지 측정한다.

캐시 없이 첫 빌드.

```bash
./gradlew clean build --no-build-cache
time ./gradlew clean build --no-build-cache
```

캐시를 켜고 두 번 빌드한다.

```bash
./gradlew clean build --build-cache
time ./gradlew clean build --build-cache
```

두 번째 호출의 출력에서 task 옆에 `FROM-CACHE`가 나타나는 task가 늘어난 모습이 보인다(04-01 §6). 시간 차이가 분 단위로 줄어들 수도 있다.

영구로 켜려면 `gradle.properties`에 다음을 둔다.

```properties
org.gradle.caching=true
```



## 실습 4. `--configuration-cache` 두 번째 빌드 시간

> 빌드 스크립트 평가가 캐시에서 로드되는 동작을 본다.

```bash
./gradlew tasks --configuration-cache
./gradlew tasks --configuration-cache    # 두 번째 호출
```

두 번째 호출 출력 첫 줄에 `Reusing configuration cache.`가 나오면 적용된 것이다(04-01 §7). configuration phase가 거의 사라져 task 목록이 즉시 출력된다.

호환되지 않는 플러그인이 있으면 첫 시도에 실패할 수 있다. 그때는 `--configuration-cache-problems=warn`을 함께 주어 어떤 플러그인이 문제인지 확인하면 된다.



## 실습 5. `--scan`으로 빌드 트레이스 보기

> 빌드 한 번을 Build Scan으로 올려 어디서 시간이 쓰이는지 시각화한다.

```bash
./gradlew clean build --scan
```

빌드 끝에 다음 같은 안내가 나온다.

```
Publishing build scan...
https://gradle.com/s/abc123def456
```

이 URL을 열면 task별 시간, 의존성 트리, build cache hit/miss, JVM 정보가 한 화면에 나타난다(04-01 §14.1, 05-01 §9). 같은 빌드를 두세 번 올려 회귀 추적의 감을 잡아 둔다.

CI에서 항상 `--scan`을 켜면 빌드가 갑자기 느려질 때 어느 task가 원인인지 즉시 확인할 수 있다.



## 실습 점검 체크리스트

> 다섯 실습이 끝나면 다음 질문에 모두 답할 수 있어야 한다.

```
1. bootJar와 일반 jar의 산출물 차이를 unzip 결과로 보였는가?
2. bootJar { enabled = false }로 두면 어떤 산출물이 사라지는가를 확인했는가?
3. --build-cache 활성화 전후 빌드 시간을 측정해 비교했는가?
4. configuration cache가 두 번째 빌드에서 어떻게 동작하는지 로그로 확인했는가?
5. --scan 출력 URL에서 task별 시간을 읽고 가장 느린 task 한 개를 골라낼 수 있는가?
```
