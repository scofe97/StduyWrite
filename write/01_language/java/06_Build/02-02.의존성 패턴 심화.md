# 의존성 패턴 심화

---

> `exclude`, `transitive`, `resolutionStrategy`, dependency locking, version catalog는 큰 빌드에서 한 번씩 만난다. 02-01 본문이 의존성 키워드의 의미를 다뤘다면 본 문서는 그 위에서 충돌 해소·재현성·중앙 관리에 쓰는 도구를 정리한다.



## 1. transitive 의존성 제외

> 라이브러리가 끌어오는 추이 의존성 중 충돌이나 보안 문제가 있는 것을 빼는 패턴.

```groovy
// 한 의존성에서만 빼기
implementation('org.springframework.boot:spring-boot-starter-web') {
    exclude group: 'org.springframework.boot', module: 'spring-boot-starter-tomcat'
}
implementation 'org.springframework.boot:spring-boot-starter-undertow'

// 모든 configuration에서 일괄 제외
configurations.all {
    exclude group: 'commons-logging', module: 'commons-logging'
}

// 추이 의존성을 통째로 끄기
implementation('org.example:fat-lib:1.0') {
    transitive = false
}
```

`exclude`로 Tomcat을 빼고 Undertow로 갈아 끼우는 패턴은 Spring Boot에서 흔하다. `commons-logging`처럼 SLF4J와 충돌하는 라이브러리를 모든 configuration에서 일괄 제거하는 것도 자주 등장한다.



## 2. ResolutionStrategy

> 같은 라이브러리에 여러 버전이 들어왔을 때 어느 버전을 채택할지, 충돌을 어떻게 다룰지를 정밀하게 통제하는 진입점이다. 네 가지 도구가 자주 만난다.

### 2.1 기본 충돌 해소 — newest wins

Gradle은 같은 `group:artifact`가 두 버전 이상 등장하면 기본 정책으로 더 높은 버전을 채택한다. 예를 들어 모듈 A가 `jackson-databind:2.15`를, 모듈 B가 transitive로 `jackson-databind:2.17`을 끌어오면 최종 채택 버전은 2.17이다. 이 정책 덕분에 라이브러리 호환 범위 안에서 버전을 자동으로 위로 올려 주지만, 메이저 버전 차이가 끼어 있으면 런타임 에러가 발생한다.

```groovy
configurations.all {
    resolutionStrategy {
        // 충돌이 발생하면 즉시 빌드 실패 (CI에서 의도치 않은 자동 승급 방지)
        failOnVersionConflict()
    }
}
```

`failOnVersionConflict()`을 켜면 자동 해소 대신 충돌 좌표 목록과 함께 빌드가 실패한다. 운영 빌드의 변경 사고를 조기에 잡는다.

### 2.2 force vs constraint strictly

특정 버전을 강제하는 도구는 둘이다 — 둘 다 결과는 같은 버전 핀 고정이지만 표현력과 권장 정도가 다르다.

| 도구 | 표기 | 강제력 | 권장 |
|------|------|--------|------|
| `resolutionStrategy.force` | `force 'g:a:v'` | 강제 | 짧지만 이유를 남기기 어려움 |
| Dependency constraint `strictly` | `constraints { implementation('g:a') { version { strictly 'v' } } }` | 강제 | `because '...'`로 이유 표기 가능, 권장 |
| Constraint `require` | `version { require 'v'; reject '[2.0,)' }` | 권장 | 다른 모듈이 더 높은 버전을 요구하면 갈아 끼울 수 있음 |

```groovy
dependencies {
    constraints {
        implementation('com.fasterxml.jackson.core:jackson-databind') {
            version {
                strictly '2.17.0'
            }
            because 'CVE-2023-XXXXX 패치 버전 — 자동 승급 금지'
        }
    }
}
```

`strictly`는 다른 모듈이 더 높은 버전을 요구하더라도 무시하지 않고 빌드를 실패시킨다. 이유(`because`)와 함께 적어 두면 1년 뒤 코드 리뷰에서 "왜 이걸 핀했지?"가 사라진다.

### 2.3 dependencyInsight로 충돌 추적

같은 좌표가 두 버전으로 들어왔을 때 누가 어떤 버전을 끌어왔는지 추적하는 도구다.

```bash
./gradlew :app:dependencyInsight \
    --configuration runtimeClasspath \
    --dependency com.fasterxml.jackson.core:jackson-databind
```

출력은 채택된 버전과 그 결정 이유, 그리고 그 좌표를 끌어온 의존성 경로 트리를 보여 준다.

```
com.fasterxml.jackson.core:jackson-databind:2.17.0
   variant "runtimeElements" [...]
   Selection reasons:
      - By constraint: CVE-2023-XXXXX 패치 버전 — 자동 승급 금지
      - By conflict resolution: between versions 2.15.0 and 2.17.0

com.fasterxml.jackson.core:jackson-databind:2.15.0 -> 2.17.0
\--- com.example:some-lib:1.5
     \--- runtimeClasspath
```

해석 순서: 어떤 버전이 채택됐는가 → 왜 채택됐는가(constraint·강제·충돌 해소) → 누가 더 낮은 버전을 끌어왔는가 → 그 모듈이 본 빌드의 어떤 dependency 경로에 있는가. 이 네 단계로 충돌의 정체가 풀린다.

### 2.4 dependencySubstitution과 변경 가능 모듈 옵션

라이브러리의 group이 바뀌거나(예: `javax.*` → `jakarta.*`) 라이브러리를 통째로 다른 좌표로 갈아 끼울 때는 `dependencySubstitution`을 쓴다.

```groovy
configurations.all {
    resolutionStrategy {
        dependencySubstitution {
            substitute module('javax.servlet:javax.servlet-api') \
                using module('jakarta.servlet:jakarta.servlet-api:6.0.0')
        }

        // SNAPSHOT 등 변경 가능 모듈의 metadata 캐시 TTL (03-01 §8 참조)
        cacheChangingModulesFor 0, 'seconds'
        cacheDynamicVersionsFor 0, 'seconds'
    }
}
```

`substitute`는 좌표 단위 치환이라 상위 모듈이 옛 좌표를 의존해도 자동으로 새 좌표로 흘러간다. transitive 정리에 가장 강력한 도구다. `cacheChangingModulesFor`/`cacheDynamicVersionsFor`는 `resolutionStrategy` 안에 있지만 충돌과 무관한 캐시 TTL 도구다(자세한 설명은 03-01 §8). 두 옵션이 같은 블록 안에 있다는 사실만 기억해 두면 된다.



## 3. 다양한 좌표 표기

> `'group:artifact:version'` 한 줄 외에도 표기 방식이 여러 가지다.

```groovy
// 가장 흔한 한 줄 표기
implementation 'org.springframework:spring-core:6.1.0'

// map 형태 — 동적 조립에 유리
implementation group: 'org.springframework', name: 'spring-core', version: '6.1.0'

// classifier (변종 jar)
implementation 'io.netty:netty-all:4.1.100.Final:linux-x86_64'

// extension (jar 외 산출물)
implementation 'org.example:proto-files:1.0@zip'

// snapshot 좌표
implementation 'org.okestro:core-lib:3.0.5.2-DEV-SNAPSHOT'
```

map 형태는 빌드 스크립트 내에서 좌표를 동적으로 만들 때 유리하다. classifier는 같은 좌표가 OS·CPU·variant별로 다른 jar를 배포할 때 그중 하나를 고르는 용도다.



## 4. Dependency Locking

> 같은 빌드를 다음 주에 다시 돌렸을 때 같은 의존성 버전이 잡히는 것을 보장한다.

동적 버전이나 SNAPSHOT을 쓰면 시간이 지나면서 빌드 결과가 달라진다. lockfile은 한 번 결정된 버전을 고정한다.

```groovy
// settings.gradle 또는 init.gradle
dependencyLocking {
    lockAllConfigurations()
}
```

```bash
./gradlew dependencies --write-locks   # 모든 lockfile 작성
./gradlew dependencies --update-locks com.fasterxml.jackson.core:*  # 특정 좌표만 갱신
```

Lockfile은 `gradle.lockfile`에 저장된다. operator처럼 운영 빌드는 lockfile을 도입할 가치가 있고, 학습 프로젝트는 굳이 안 켜도 된다.



## 5. Version Catalog (libs.versions.toml)

> 여러 모듈이 같은 라이브러리를 의존할 때 버전을 한 파일에 모으는 Gradle 7.4+ 표준이다.

```toml
# gradle/libs.versions.toml
[versions]
spring-boot = "3.2.3"
querydsl = "6.12"

[libraries]
spring-boot-starter-web = { module = "org.springframework.boot:spring-boot-starter-web", version.ref = "spring-boot" }
querydsl-jpa = { module = "io.github.openfeign.querydsl:querydsl-jpa", version.ref = "querydsl" }

[bundles]
spring-web = ["spring-boot-starter-web", "spring-boot-starter-validation"]

[plugins]
spring-boot = { id = "org.springframework.boot", version.ref = "spring-boot" }
```

```groovy
dependencies {
    implementation libs.spring.boot.starter.web
    implementation libs.bundles.spring.web   // 묶음 한 번에
}

plugins {
    alias(libs.plugins.spring.boot)
}
```

`libs.versions.toml`은 `settings.gradle`이 자동으로 인식한다. 별도 설정 없이 catalog 이름이 `libs`가 된다. operator도 이 파일을 가지고 있어 단계적 도입 단계에 있다.



## 6. 정리 — 패턴 결정 트리

> 어떤 패턴이 어떤 상황에 맞는지 한 장.

| 상황 | 도구 |
|------|------|
| 특정 transitive 의존성 하나만 빼고 싶다 | `exclude group: 'g', module: 'm'` |
| 모든 configuration에서 라이브러리 일괄 제거 | `configurations.all { exclude ... }` |
| 충돌 시 자동 승급을 막고 빌드 실패 | `failOnVersionConflict()` |
| 보안 패치 같은 명시적 이유로 버전 핀 | constraint `strictly` + `because` |
| group rename된 라이브러리 치환 | `dependencySubstitution.substitute` |
| 같은 빌드의 재현성 보장 | dependency locking + `gradle.lockfile` |
| 멀티모듈 버전 중앙 관리 | version catalog `libs.versions.toml` |

본 문서의 도구는 큰 빌드에서 한 번씩 만난다. 작은 학습 프로젝트는 §1~§3 정도면 충분하고, lockfile·version catalog는 운영 빌드가 안정화된 뒤 단계적으로 도입한다.
