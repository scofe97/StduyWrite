# SOLID 원칙

---

> 좋은 객체지향 설계는 변경에 유연하게 대응한다. SOLID는 그 유연성을 만들어내는 다섯 가지 원칙의 집합이다.

## SRP — 단일 책임 원칙

*Single Responsibility Principle*: 하나의 클래스는 변경되어야 할 이유가 하나뿐이어야 한다.

"책임"이란 "변경의 이유"와 같다. 어떤 클래스가 여러 이유로 수정되어야 한다면, 그것은 여러 책임이 뭉쳐 있다는 신호다. 책임이 분리되지 않으면 한 기능의 수정이 다른 기능에 예기치 않은 부작용을 일으킨다.

**위반 예제**: `UserController`가 요청 수신, 유효성 검증, 저장까지 모두 담당한다.

```java
// 위반: 컨트롤러가 비즈니스 로직을 직접 포함
public class UserController {
    private Store store = new Store();

    public String createUser(String userJson) throws IOException {
        ObjectMapper mapper = new ObjectMapper();
        User user = mapper.readValue(userJson, User.class);

        // 검증 로직이 컨트롤러 안에 있다
        if (!isValidUser(user)) return "ERROR";

        store.store(user);
        return "SUCCESS";
    }

    private boolean isValidUser(User user) { /* 검증 로직 */ }
}
```

**준수 예제**: 컨트롤러는 요청 수신만, 검증은 `UserValidator`, 저장은 `UserPersistenceService`가 담당한다.

```java
public class UserController {
    private UserPersistenceService persistenceService = new UserPersistenceService();

    public String createUser(String userJson) throws IOException {
        User user = new ObjectMapper().readValue(userJson, User.class);

        UserValidator validator = new UserValidator();
        if (!validator.validateUser(user)) return "ERROR";

        persistenceService.saveUser(user);
        return "SUCCESS";
    }
}

public class UserValidator {
    public boolean validateUser(User user) { /* 검증만 */ }
}

public class UserPersistenceService {
    public void saveUser(User user) { /* 저장만 */ }
}
```

## OCP — 개방-폐쇄 원칙

*Open-Closed Principle*: 소프트웨어 엔티티는 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다.

새로운 요구사항이 생겼을 때 기존 코드를 직접 수정하지 않고 새 코드를 추가하는 방식으로 대응할 수 있어야 한다. 추상화(인터페이스 또는 추상 클래스)가 이 원칙을 가능하게 하는 핵심 수단이다.

**위반 예제**: `PhoneSubscriber`에 ISP 요금 계산 로직을 추가하면 기존 클래스를 직접 수정해야 한다.

```java
// 위반: 새로운 구독자 유형을 추가할 때마다 기존 클래스를 수정해야 한다
public class PhoneSubscriber {
    public double calculateBill() {
        // 전화 요금 계산
    }
}
```

**준수 예제**: `Subscriber`를 추상 클래스로 두고, 각 구독자 유형이 `calculateBill()`을 독립적으로 구현한다.

```java
public abstract class Subscriber {
    protected Long subscriberId;
    protected int baseRate;

    public abstract double calculateBill();
}

public class PhoneSubscriber extends Subscriber {
    @Override
    public double calculateBill() {
        List<CallHistory.Call> sessions = CallHistory.getCurrentCalls(subscriberId);
        long totalDuration = sessions.stream()
            .mapToLong(CallHistory.Call::getDuration)
            .sum();
        return (double) (totalDuration * baseRate) / 100;
    }
}

public class ISPSubscriber extends Subscriber {
    private long freeUsage;

    @Override
    public double calculateBill() {
        long totalData = InternetSessionHistory.getCurrentSessions(subscriberId)
            .stream()
            .mapToLong(InternetSessionHistory.InternetSession::getDataUsed)
            .sum();
        return (double) ((totalData - freeUsage) * baseRate) / 100;
    }
}
```

## LSP — 리스코프 치환 원칙

*Liskov Substitution Principle*: 상위 타입의 객체를 하위 타입의 객체로 교체해도 프로그램의 정확성이 깨지지 않아야 한다.

LSP는 상속의 올바른 사용 기준을 제시한다. `Square extends Rectangle`이 흔히 드는 위반 예시인데, 정사각형은 수학적으로 직사각형의 특수한 경우지만, `setWidth`와 `setHeight`가 독립적으로 동작한다는 `Rectangle`의 행동 계약을 깨뜨리기 때문이다.

**위반 예제**: `Square`를 `Rectangle` 자리에 넣으면 assertion이 실패한다.

```java
// useRectangle(square) 호출 시 assertion 실패
private static void useRectangle(Rectangle rectangle) {
    rectangle.setHeight(20);
    rectangle.setWidth(30);
    assert rectangle.getHeight() == 20; // Square는 width 설정 시 height도 바뀐다
    assert rectangle.getWidth() == 30;
}
```

**준수 예제**: `Square`와 `Rectangle`을 각각 별도의 `Shape` 구현체로 분리한다.

```java
public interface Shape {
    int computeArea();
}

public class Rectangle implements Shape {
    private int width;
    private int height;

    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }

    @Override
    public int computeArea() { return width * height; }
}

public class Square implements Shape {
    private int side;

    public Square(int side) { this.side = side; }

    @Override
    public int computeArea() { return side * side; }
}
```

## ISP — 인터페이스 분리 원칙

*Interface Segregation Principle*: 클라이언트는 자신이 사용하지 않는 인터페이스에 의존하도록 강제되어서는 안 된다.

하나의 거대한 인터페이스는 구현 클래스가 불필요한 메서드까지 구현하게 만든다. 이른바 *Interface Pollution*이다. 인터페이스를 역할 단위로 잘게 나누면 각 구현체는 자신에게 필요한 계약만 맺을 수 있다.

**위반 예제**: `PersistenceService`에 `findByName`이 있어, 이름 검색이 불필요한 `OrderPersistenceService`도 이를 구현해야 한다.

```java
// 위반: 모든 구현체가 findByName을 구현해야 한다
public interface PersistenceService<T extends Entity> {
    void save(T entity);
    void delete(T entity);
    T findById(Long id);
    List<T> findByName(String name); // Order에는 필요 없을 수 있다
}
```

**준수 예제**: `findByName`을 별도 인터페이스로 분리한다.

```java
public interface PersistenceService<T extends Entity> {
    void save(T entity);
    void delete(T entity);
    T findById(Long id);
}

public interface NameSearchable<T> {
    List<T> findByName(String name);
}

// UserPersistenceService만 NameSearchable을 추가로 구현한다
public class UserPersistenceService
    implements PersistenceService<User>, NameSearchable<User> {
    // ...
}
```

## DIP — 의존 역전 원칙

*Dependency Inversion Principle*: 고수준 모듈은 저수준 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다.

구체 클래스를 직접 참조하는 코드는 해당 구현체가 바뀔 때마다 영향을 받는다. 인터페이스나 추상 클래스를 사이에 두면 고수준 모듈과 저수준 모듈이 각각 독립적으로 변경 가능해진다. Spring의 DI 컨테이너가 이 원칙의 대표적 적용 사례다.

**위반 예제**: `PizzaStore`가 구체적인 피자 클래스를 직접 생성한다.

```java
// 위반: 고수준 모듈이 저수준 구현에 직접 의존
public class PizzaStore {
    public Pizza orderPizza(String type) {
        Pizza pizza;
        if (type.equals("cheese")) pizza = new CheesePizza();
        else if (type.equals("pepperoni")) pizza = new PepperoniPizza();
        // 새 피자 추가 시 이 코드를 수정해야 한다
        return pizza;
    }
}
```

**준수 예제**: 추상 팩토리 메서드를 통해 의존성을 역전한다.

```java
public abstract class PizzaStore {
    // 구체적인 피자 생성은 서브클래스에 위임
    abstract Pizza createPizza(String item);

    public final Pizza orderPizza(String type) {
        Pizza pizza = createPizza(type);
        pizza.prepare();
        pizza.bake();
        pizza.cut();
        pizza.box();
        return pizza;
    }
}

public class NYPizzaStore extends PizzaStore {
    @Override
    Pizza createPizza(String type) {
        return switch (type) {
            case "cheese" -> new NYStyleCheesePizza();
            case "veggie" -> new NYStyleVeggiePizza();
            default -> null;
        };
    }
}
```

## 원칙 비교

| 원칙 | 핵심 질문 | 위반 신호 | 주요 적용 대상 |
|------|-----------|-----------|----------------|
| SRP | 이 클래스가 변경되는 이유가 하나인가? | 클래스가 너무 길다, 여러 도메인을 다룬다 | 클래스, 모듈 |
| OCP | 새 기능을 추가할 때 기존 코드를 수정하는가? | `if/else`나 `switch`로 타입을 분기한다 | 클래스 계층, 전략 |
| LSP | 하위 타입을 상위 타입 자리에 넣어도 동작하는가? | 오버라이드 메서드가 예외를 던지거나 계약을 무시한다 | 상속 계층 |
| ISP | 구현체가 사용하지 않는 메서드를 구현하는가? | 빈 구현 메서드, `UnsupportedOperationException` | 인터페이스 설계 |
| DIP | 고수준 모듈이 구체 클래스를 직접 참조하는가? | `new ConcreteClass()` 가 비즈니스 로직 안에 있다 | 의존성 구조 전체 |
