# WireMock과 E2E 테스트

---

> E2E 테스트는 *실제 외부 자원*과 결합된 흐름 전체를 검증한다. TPS executor는 Jenkins HTTP를 WireMock으로 흉내내 *동시성·장애 주입·재시도*를 결정적으로 검증한다. 50개+ E2E 테스트(`lifecycle/recovery/wiremock/concurrency/scheduler`)가 그 패턴을 공유한다.



## 학습 목표

> WireMock + Testcontainers + Spring Boot의 결합이 *현실적인 외부 환경을 결정적으로* 만드는 방법을 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. WireMock이 *Mock Server* 패턴의 표준 구현인 이유를 설명할 수 있다.
2. `@BeforeAll` static WireMockServer 패턴의 의도와 비용을 설명할 수 있다.
3. Fault injection(타임아웃·5xx·연결 거부)으로 *복구 로직*을 결정적으로 검증하는 방법을 설명할 수 있다.
4. Concurrency E2E가 *경쟁 조건*을 어떻게 노출하는지 설명할 수 있다.
5. TPS의 E2E 디렉터리 분류(`lifecycle/recovery/wiremock/concurrency/scheduler`)가 무엇을 가르는지 설명할 수 있다.



## 1. WireMock — Mock Server 패턴의 표준

WireMock은 *HTTP 서버를 흉내내는 라이브러리*다. 미리 등록한 stub에 따라 요청에 응답하고, 받은 요청을 기록해 *검증*에 쓸 수 있다.

```java
WireMockServer wireMock = new WireMockServer(wireMockConfig().port(0));
wireMock.start();

wireMock.stubFor(get(urlEqualTo("/job/SBH/job/E2E/job/E2E-SUCCESS/build"))
    .willReturn(aResponse()
        .withStatus(201)
        .withHeader("Location", "http://localhost:" + wireMock.port() + "/queue/item/100/")));

// SUT를 wireMock.port()로 호출하게 만든 뒤
wireMock.verify(postRequestedFor(urlEqualTo("/job/.../build")));
```

TPS의 `JenkinsWireMockSupport`(`executor/engine/src/test/java/.../jenkins/e2e/wiremock/JenkinsWireMockSupport.java`)가 그 베이스 클래스다.

```java
@SpringBootTest(classes = ExecutorApplication.class)
@AutoConfigureMockMvc
@ActiveProfiles("test")
public abstract class JenkinsWireMockSupport {

    protected static WireMockServer wireMock;

    @BeforeAll
    static void startWireMock() {
        wireMock = new WireMockServer(wireMockConfig().port(0));
        wireMock.start();
        System.setProperty("jenkins.base-url", "http://localhost:" + wireMock.port());
    }

    @AfterAll
    static void stopWireMock() {
        if (wireMock != null) wireMock.stop();
    }

    @BeforeEach
    void resetStubs() {
        wireMock.resetAll();
    }
}
```

여섯 가지 좋은 패턴이 모여 있다.

1. **`@SpringBootTest` + WireMock** — Spring Context와 외부 시스템 mock을 같이 부팅.
2. **`port(0)` 동적 포트** — 포트 충돌 회피.
3. **`System.setProperty("jenkins.base-url", ...)`** — SUT가 *동적 포트로* Jenkins를 보게 만듦.
4. **`@BeforeAll` start** — 클래스 단위로 한 번만 부팅.
5. **`@BeforeEach resetAll`** — 매 테스트마다 stub·기록을 *완전 초기화*해 격리.
6. **`@AfterAll` stop** — 리소스 정리.



## 2. E2E 디렉터리 분류

TPS executor의 E2E 디렉터리가 *시나리오 카테고리*를 보여 준다.

```
executor/engine/src/test/java/org/okestro/tps/jenkins/e2e/
├── lifecycle/         - 정상 흐름 (Receive → Dispatch → Submit → Complete)
├── recovery/          - 장애 후 복구 (재시도, 타임아웃, 인스턴스 죽음)
├── wiremock/          - Jenkins HTTP fault injection
├── concurrency/       - 동시성 경쟁 조건
├── scheduler/         - 시간 기반 트리거 (cron, polling)
├── integration/       - 통신 자체 검증
└── support/           - 공통 fixture (Jenkins305PTestFixtures)
```

이 분류가 *시나리오 의도*를 코드 위치에서 즉시 보여 준다. `lifecycle/`을 깨는 건 *정상 흐름의 회귀*고, `recovery/`를 깨는 건 *장애 복구의 회귀*다. 디버깅 시점에 무엇을 봐야 할지가 명확하다.



## 3. Fault Injection — 결정적 장애 검증

WireMock의 강점은 *원하는 장애를 정확히 재현*할 수 있다는 점이다. 실제 Jenkins에서 fault를 일으키려면 네트워크를 끊거나 서버를 죽여야 하지만, WireMock은 *stub 설정만으로* 같은 동작을 만든다.

```java
// 5xx 응답
wireMock.stubFor(post(urlEqualTo("/job/.../build"))
    .willReturn(aResponse().withStatus(503)));

// 타임아웃 (5초 후 응답)
wireMock.stubFor(get(urlEqualTo("/queue/item/100/"))
    .willReturn(aResponse().withFixedDelay(5000).withStatus(200)));

// 연결 거부
wireMock.stop();  // 다음 요청부터 ConnectException

// 일정 횟수 후 회복
wireMock.stubFor(post(urlEqualTo("/job/.../build"))
    .inScenario("flaky").whenScenarioStateIs("started")
    .willReturn(aResponse().withStatus(503))
    .willSetStateTo("attempt-2"));

wireMock.stubFor(post(urlEqualTo("/job/.../build"))
    .inScenario("flaky").whenScenarioStateIs("attempt-2")
    .willReturn(aResponse().withStatus(201)));
```

마지막 패턴이 특히 강하다. *Scenario state*로 *호출 순서에 따른 다른 응답*을 만들어, "처음엔 실패하다가 N번째에 성공하는" 시나리오를 결정적으로 재현한다. TPS `JenkinsFaultInjectionE2ETest`·`SubmitRetryFlowE2ETest`가 이 패턴을 쓴다.



## 4. Concurrency E2E — 경쟁 조건 노출

`SubmitClaimConcurrencyE2ETest`·`DispatchClaimConcurrencyE2ETest`가 그 예다. *여러 스레드가 동시에 같은 작업을 시도*할 때 *정확히 한 번만 성공*하는지 검증한다.

```java
@Test
void multipleInstances_onlyOneClaims() throws InterruptedException {
    int threadCount = 10;
    CountDownLatch readyLatch = new CountDownLatch(threadCount);
    CountDownLatch startLatch = new CountDownLatch(1);
    AtomicInteger claimedCount = new AtomicInteger();

    for (int i = 0; i < threadCount; i++) {
        new Thread(() -> {
            readyLatch.countDown();
            startLatch.await();  // 모든 스레드가 동시에 출발

            boolean claimed = submitClaimService.tryClaim("JOB-1");
            if (claimed) claimedCount.incrementAndGet();
        }).start();
    }

    readyLatch.await();  // 모든 스레드 준비 완료
    startLatch.countDown();  // 출발

    await().atMost(5, SECONDS)
        .untilAsserted(() -> assertThat(claimedCount.get()).isEqualTo(1));
}
```

핵심은 `readyLatch` + `startLatch`의 *barrier 패턴*이다. 단순히 N개 스레드를 만들면 *시작 시점이 어긋나* 경쟁 조건이 재현되지 않는다. 모든 스레드가 *준비 완료* 후 *동시에 출발*하게 만들어야 *진짜 경쟁*이 일어난다.

같은 시나리오를 단위 테스트로 만들 수 없다. *실제 DB의 `FOR UPDATE SKIP LOCKED`* 또는 *Optimistic Lock의 version 충돌*이 동시 발생해야 검증된다. Concurrency E2E의 가치다.



## 5. Recovery E2E — 인스턴스 죽음 시뮬레이션

`SubmittedRecoveryE2ETest`·`RunningRecoveryE2ETest`가 그 예다. *작업 중간에 executor 인스턴스가 죽으면 다른 인스턴스가 이어받는지* 검증한다.

```java
@Test
void runningJob_otherInstanceRecovers() {
    // Given: 작업이 RUNNING 상태로 DB에 기록됨
    insertRunningJob("JOB-1", staleHeartbeat);

    // When: scheduler가 stale heartbeat 감지
    recoveryScheduler.recoverStaleJobs();

    // Then: 작업이 RECOVERED로 마킹되고 재처리됨
    await().atMost(5, SECONDS).untilAsserted(() -> {
        ExecutionJob job = jobRepository.findById("JOB-1").orElseThrow();
        assertThat(job.getStatus()).isIn(SUBMITTED, RUNNING);
        assertThat(job.getRecoveredAt()).isNotNull();
    });
}
```

여기서 *실제로 인스턴스를 죽일 필요는 없다*. *DB에 stale heartbeat가 기록된 상태*를 직접 시드하는 것으로 *같은 시나리오*를 재현한다. WireMock으로 Jenkins 응답을 통제하면 *복구된 인스턴스가 Jenkins 상태를 올바르게 조회*하는 흐름도 검증된다.



## 6. E2E 비용 관리

E2E 테스트는 *가장 비싼* 카테고리다. TPS executor의 50개+ E2E가 다 함께 빌드 시간을 갉는다. 비용 관리의 세 가지 도구.

### 6.1 `@Tag`로 분류 실행

```java
@Tag("e2e")
class FullLifecycleE2ETest extends JenkinsWireMockSupport { ... }
```

```groovy
// build.gradle
test {
    useJUnitPlatform {
        excludeTags 'e2e'  // 기본 빌드에서 제외
    }
}

task e2eTest(type: Test) {
    useJUnitPlatform {
        includeTags 'e2e'  // 별도 태스크에서만 실행
    }
}
```

일반 PR 빌드에서는 *unit + integration*만 돌리고, *야간 빌드 또는 release 빌드*에서 E2E를 실행하는 패턴이 흔하다.

### 6.2 Context 재사용

같은 `@SpringBootTest` Context를 *모든 E2E가 공유*한다. WireMockServer는 static `@BeforeAll`로 한 번만 시작하고 `@BeforeEach resetAll`로 *stub만 초기화*. 컨테이너도 JVM 단위 `static { start(); }`.

### 6.3 병렬 실행

JUnit 5의 `junit.jupiter.execution.parallel.enabled=true`로 *클래스 단위 병렬*을 켤 수 있다. 다만 *상태 공유*(static WireMock, static DB seed)가 *thread-safe*하지 않으면 위험하다. TPS는 *직렬 실행*을 유지하되 *Context 재사용*으로 비용을 줄이는 전략이다.



## 7. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. WireMock이 `@MockBean`보다 *현실적인* 검증을 만드는 이유는?

`@MockBean`은 *클라이언트 빈*을 mock으로 교체해 *HTTP 통신 자체를 우회*한다. WireMock은 *실제 HTTP 서버*를 띄워 *직렬화·헤더·상태 코드·재시도까지* 검증한다. RestTemplate·WebClient·Connection Pool 같은 *HTTP 계층의 동작*이 실제로 일어난다. 검증의 *현실성*이 다르고, 그래서 mock 서버의 검증 가치가 더 크다.

### Q2. `@BeforeAll` static WireMockServer 패턴의 비용은?

장점은 *부팅을 클래스당 한 번만* 한다는 것. WireMock 자체는 가볍지만 `@SpringBootTest` Context와 결합되면 *Context 재시작 비용*이 더 크다. 단점은 *클래스 간 격리가 깨질 수 있다는 것*. `@BeforeEach resetAll()`로 stub·기록을 초기화해야 다음 클래스가 같은 상태에서 시작한다. 빠뜨리면 *flaky 테스트의 원흉*이 된다.

### Q3. Concurrency E2E의 barrier 패턴이 단순 `new Thread().start()`보다 *왜 더 결정적*인가?

단순 시작은 *스레드 시작 비용*만큼 시점이 어긋난다. 한 스레드가 메서드를 부르기 전에 다른 스레드가 *이미 완료*되어 있을 수 있다. `CountDownLatch` 두 개로 *모든 스레드가 준비될 때까지 대기 → 동시 출발*을 만들면 *진짜 경쟁 조건*이 재현된다. 그래야 `FOR UPDATE SKIP LOCKED`나 Optimistic Lock의 *실제 충돌*이 일어난다.

### Q4. Recovery E2E에서 *실제 인스턴스를 죽이지 않는* 이유는?

검증 대상이 *살아있는 인스턴스의 복구 로직*이기 때문이다. *죽인 인스턴스*는 검증할 게 없다. *DB에 stale heartbeat가 기록된 상태*를 시드하는 것은 *죽음의 결과*를 재현하는 것과 동등하다. 실제로 인스턴스를 죽이려면 *복잡한 인프라*(chaos engineering)가 필요하고 *재현성이 낮다*. 시드 방식이 결정성을 만든다.

### Q5. E2E 비용을 줄이기 위해 *가장 먼저 검토할* 것은?

`@Tag`로 *분류 실행 분리*다. 일반 PR 빌드에서 E2E를 제외하고 *unit + integration*만 돌리면 빌드 시간이 5~10배 줄어든다. E2E는 *야간 빌드, merge 전 release 검증, schedule cron*에서 실행한다. 그래도 부족하면 *Context 재사용 + 직렬 실행 → 병렬 실행*을 단계적으로 검토한다. 병렬은 *공유 상태가 thread-safe하지 않으면 위험*하므로 마지막 수단이다.
