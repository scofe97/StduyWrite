# Given-When-Then 구조

---

> 테스트의 *주요 관심사에 집중*하려면 데이터 셋업이 검증을 가려서는 안 된다. 카카오페이 Given Part 1·3가 던지는 핵심이다. 이 글은 그 관점을 TPS 코드(`DefaultDeployDomainServiceTest`, `Jenkins305PTestFixtures`)에 매핑해 *Given-When-Then 구조의 실전*을 정리한다.



## 학습 목표

> Given/When/Then 세 구간이 *왜* 분리되어야 하는지와 *어떻게* 분리해야 가독성이 살아나는지 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. "데이터 셋업은 검증이 아니다"가 무엇을 의미하는지 설명할 수 있다.
2. BDDMockito `given()`이 `Mockito.when()`보다 BDD 문맥에 맞는 이유를 설명할 수 있다.
3. *세 가지 지옥*(파라미터·멀티모듈·Mocking)을 무엇으로 푸는지 설명할 수 있다.
4. `@DisplayName`이 *왜 시나리오 표현*에 적합한지 설명할 수 있다.
5. Given 단계 외부화(JSON, `@Sql`)를 *언제* 쓸지 판단할 수 있다.



## 1. 세 구간의 책임

```
Given — 검증에 필요한 *상태*를 만든다.
When  — 검증 대상이 되는 *동작* 하나를 실행한다.
Then  — 동작의 *결과*가 기대와 일치하는지 확인한다.
```

세 구간을 분리하는 이유는 *시선의 동선*이다. 독자가 Given에서 상태를 머리에 그리고, When에서 트리거를 인식하고, Then에서 기대를 확인한다. 이 순서가 깨지면 *무엇이 검증되고 있는지* 파악하는 데 비용이 든다.

가장 흔한 깨짐은 Given이 비대해질 때다. 카카오페이 Given Part 1이 명시한다.

> "복잡한 데이터 셋업으로 인해 테스트의 주요 관심사에 집중하기 어려운 문제."

데이터 셋업은 *검증이 아니다*. 검증을 가린다면 외부 자원으로 빼야 한다.



## 2. BDDMockito given vs Mockito when

같은 mock 설정도 두 가지 모양으로 표현된다.

```java
// Mockito.when — "이걸 호출하면 이걸 반환해라"
when(deployJobReader.findLatestContainerManifestId("JOB_1"))
    .thenReturn("MFST_001");

// BDDMockito.given — "이런 상태가 주어졌을 때"
given(deployJobReader.findLatestContainerManifestId("JOB_1"))
    .willReturn("MFST_001");
```

기능은 동일하다. 의미가 다르다. `when().thenReturn()`은 *행위*를 설명하지만, `given().willReturn()`은 *상태*를 설명한다. BDD 시나리오(`Given/When/Then`)에서 Given 구간에 `given()`이 자연스럽게 맞물린다.

TPS `DefaultDeployDomainServiceTest`(`executor/engine/src/test/java/.../argocd/domain/deploy/DefaultDeployDomainServiceTest.java`)가 그 예다.

```java
import static org.mockito.BDDMockito.given;

@ExtendWith(MockitoExtension.class)
class DefaultDeployDomainServiceTest {

    @Mock private ArgoCdLinkPort argoCdLinkPort;
    @Mock private ContainerArtifactPort containerArtifactPort;
    // ...

    @Test
    void deploy_linksManifestAndTriggersArgoCd() {
        // Given
        given(deployJobReader.requireContainerStepByJobVersion("DEPLOY_JOB_001", 1))
                .willReturn(deployStepDetailItem);
        given(applicationInfoQueryPort.requireByApplicationName("app-1"))
                .willReturn(applicationInfo);
        given(manifestReader.requireLatestActiveRow("MFST_001"))
                .willReturn(manifestEntity);

        // When
        deployDomainService.deploy(command);

        // Then
        verify(argoCdLinkPort).link(command, "MFST_001", 2);
    }
}
```

Given 블록의 `given()` 세 줄이 *시나리오의 전제 조건*을 명시한다. When 한 줄이 *트리거*다. Then의 `verify()`가 *기대된 동작*이다. 세 구간이 시선의 동선과 일치한다.

같은 코드를 `Mockito.when`으로 쓰면 *Given 구간이 `when()`로 시작하고 When 구간도 동사인 `when()` 어휘로 적혀* 혼동이 생긴다. BDDMockito는 그 어휘 충돌을 푼다.



## 3. @DisplayName으로 시나리오를 박는다

JUnit 5의 `@DisplayName`은 *테스트 이름을 한 문장으로 표현*한다. 메서드명은 코드 컨벤션에 묶여 있지만, `@DisplayName`은 *사람이 읽는 시나리오*를 적을 수 있다.

```java
@Test
@DisplayName("retryCount < maxRetries 면 DEAD 마킹 대상 아님")
void shouldMarkDead_belowMax_returnsFalse() { ... }

@Test
@DisplayName("retryCount=30 → 2^30 초 (cap 직전)")
void nextRetryAt_thirtyRetries_cap() { ... }
```

TPS `OutboxRetryPolicyTest`(`message-lib/src/test/java/.../domain/outbox/OutboxRetryPolicyTest.java`)의 패턴이다. 시나리오가 *전제 → 기대*로 압축돼 있다. 테스트가 깨지면 보고 결과창에 이 한 문장이 그대로 뜬다. 디버깅이 시작되기 전 *어떤 시나리오가 깨졌는지*를 즉시 알 수 있다.

`@Nested` 클래스로 시나리오를 묶을 수도 있다.

```java
@Nested
@DisplayName("retryCount=null 인 경우")
class WhenRetryCountIsNull {
    @Test @DisplayName("0으로 처리되어 DEAD 마킹 대상 아님")
    void treatedAsZero_notDead() { ... }
}
```

테스트 보고서에서 *카테고리 → 시나리오*가 트리로 보인다.



## 4. 세 가지 지옥

카카오페이 Given Part 3은 Given 단계에서 자주 만나는 *세 가지 지옥*을 명시한다.

### 4.1 파라미터 지옥

> "모든 필드를 수동으로 채우는 번거로움."

도메인 객체에 필드가 20개인데 *검증에 필요한 건 3개*뿐이라면, 나머지 17개도 형식상 채워야 한다. 채울 값을 매번 생각하는 비용이 *검증 의도를 가린다*.

해결책은 `DomainFixture`다. 기본값을 가진 팩토리 메서드를 두고 *필요한 파라미터만 override*하게 만든다.

```java
public static OutboxEvent.OutboxEventBuilder defaultEvent() {
    return OutboxEvent.builder()
            .aggregateId("agg-1")
            .payload(new byte[]{1})
            .topic("topic-a")
            .status(OutboxStatus.PENDING);
}

// 사용처
OutboxEvent event = defaultEvent().retryCount(3).build();
```

테스트 본문에는 *override한 필드만* 남는다. 검증 의도가 즉시 보인다.

### 4.2 멀티모듈 재사용 지옥

> "`src/test`의 헬퍼 함수를 다른 모듈에서 참조 불가."

`src/test/java`는 Gradle 기본 설정에서 다른 모듈에 노출되지 않는다. message-lib의 OutboxEvent fixture를 executor에서 쓰고 싶어도 *복사*해야 한다.

해결책은 `java-test-fixtures` 플러그인이다. `src/testFixtures/java`에 둔 코드는 *다른 모듈에서 의존성으로 가져갈 수 있다*.

```groovy
plugins {
    id 'java-test-fixtures'
}

// 다른 모듈 build.gradle
dependencies {
    testImplementation testFixtures(project(':message-lib'))
}
```

05-01에서 본격적으로 다룬다.

### 4.3 Mocking 지옥

> "외부 요청 객체가 도메인과 중복 필드를 가질 때 반복 작성."

REST 요청 DTO와 도메인 객체가 95% 같은 필드를 가지면 *둘 다 만들어야* 한다. 두 번째 fixture는 첫 번째의 단순 복사처럼 보인다.

해결책은 `DomainIoFixture`다. 외부 요청을 *도메인 객체에서 매핑*하는 fixture를 두면 한 번의 정의가 두 곳에 작동한다.



## 5. Given 외부화 — JSON과 @Sql

객체 셋업이 *코드 안에서 너무 길어진다면* 외부 자원으로 뺀다.

### 5.1 JSON 파일

```java
@Test
void create_orderWithComplexNestedItems() throws Exception {
    String requestJson = readResource("fixtures/order/complex-nested-items.json");

    mockMvc.perform(post("/orders")
            .contentType(APPLICATION_JSON)
            .content(requestJson))
        .andExpect(status().isCreated());
}
```

복잡한 중첩 객체·Enum 비정상 값·API 스펙 변경 시 일관성 유지가 *JSON 한 파일*로 압축된다. 카카오페이 Given Part 1의 핵심 추천이다.

### 5.2 @Sql / @SqlGroup

```java
@Test
@Sql("/sql/order-with-pending-status.sql")
void cancel_pendingOrder_succeeds() {
    // Given이 SQL 한 줄로 대체됨
    cancelService.cancel("ORD-001");

    // Then
    assertThat(orderRepository.findById("ORD-001"))
        .extracting(Order::getStatus)
        .isEqualTo(CANCELLED);
}
```

DB 상태 셋업이 코드보다 *SQL로 표현하는 게 자연스러운* 경우에 쓴다. 영속성 계층 통합 테스트에서 특히 강하다.

TPS는 SQL 외부 파일 대신 `TicketBaseDataSeeder`(`operator/ticket/src/test/java/.../testsupport/TicketBaseDataSeeder.java`) 같은 *Java 코드 시더*를 쓴다. `@Sql` 대비 장점은 *컴파일 타임 검증*이 되는 것, 단점은 *데이터 변경 비용*이 크다는 것이다. 둘 다 같은 목적의 변종이다.



## 6. TPS fixture 예시

`Jenkins305PTestFixtures`(`executor/engine/src/test/java/.../jenkins/e2e/support/Jenkins305PTestFixtures.java`)가 도메인 fixture의 좋은 예다.

```java
public final class Jenkins305PTestFixtures {

    public static final String PROJECT_ID = "SBH";
    public static final String PRESET_ID = "E2E";
    public static final TestJob SUCCESS_JOB = TestJob.SUCCESS;
    public static final TestJob FAILURE_JOB = TestJob.FAILURE;

    public enum TestJob {
        SUCCESS("E2E-SUCCESS", ExecutionJobStatus.SUCCESS),
        FAILURE("E2E-FAILURE", ExecutionJobStatus.FAILURE);

        private final String id;
        private final ExecutionJobStatus expectedTerminal;

        public String jobPath() { ... }
    }
}
```

세 가지 좋은 패턴이 모여 있다.

1. **상수 명명** — `PROJECT_ID = "SBH"` 같은 *의미 있는 상수*가 fixture로 쓰인다. 테스트 본문에서 직접 문자열을 쓰지 않는다.
2. **Enum으로 변형 묶기** — `TestJob.SUCCESS`/`FAILURE`/`COMPLEX` 등 *시나리오 변형*이 enum 하나로 모인다. 새 변형 추가가 단순하다.
3. **헬퍼 메서드** — `jobPath()` 같은 *파생 값*이 fixture 안에 있다. 테스트 본문이 도메인을 모른 채로도 쓸 수 있다.



## 7. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. "데이터 셋업은 검증이 아니다"가 정확히 의미하는 바는?

테스트의 *주요 관심사*는 Then 구간에서 검증되는 동작이다. Given 구간이 그 검증을 *가리는 만큼* 테스트의 가치가 떨어진다. 셋업에 30줄이 들어가고 검증은 2줄이라면, 독자는 그 30줄을 머리에 그리는 비용을 매번 치러야 한다. 셋업을 fixture·JSON·`@Sql`로 외부화하면 본문은 *검증 의도만* 남는다.

### Q2. `Mockito.when` 대신 `BDDMockito.given`을 쓰는 결정적 이유는?

어휘 충돌이다. BDD 시나리오의 세 구간은 Given/When/Then인데, `Mockito.when()`이 Given 구간에 들어가면 "when으로 시작하는 줄이 Given에 있다"는 모순이 코드에 박힌다. `BDDMockito.given()`은 그 충돌을 푼다. Then 구간의 `verify()`도 `then()`으로 대체할 수 있어 세 구간이 *어휘 수준에서* 가지런해진다.

### Q3. 세 가지 지옥(파라미터·멀티모듈·Mocking)을 한 가지로 푸는 도구는?

`DomainFixture` + `java-test-fixtures` 조합이다. (1) 파라미터 지옥은 *기본값 팩토리 메서드*가 푼다. (2) 멀티모듈 재사용 지옥은 `src/testFixtures`로 코드를 *모듈 의존성으로 공유*해 푼다. (3) Mocking 지옥은 외부 요청을 도메인에서 매핑하는 `DomainIoFixture`로 *중복 정의를 한 번으로* 줄인다. 셋이 같은 방향에서 같은 문제를 푼다.

### Q4. `@DisplayName`을 메서드명만으로 표현할 수 없는가?

표현은 가능하지만 *코드 컨벤션*에 묶인다. 자바 메서드명에 공백·괄호·화살표를 못 넣으므로 `nextRetryAt_thirtyRetries_cap` 같은 *압축 표현*이 된다. `@DisplayName("retryCount=30 → 2^30 초 (cap 직전)")`은 *시나리오 자체*를 쓴다. 테스트 보고서에서 이 한 문장이 그대로 뜨고, 디버깅이 시작되기 전 *깨진 시나리오*를 즉시 알 수 있다. 메서드명과 `@DisplayName`은 *코드 식별자*와 *사용자 시나리오*의 두 층을 분리한다.

### Q5. Given을 JSON·SQL로 외부화하면 *왜 무조건 좋지는* 않은가?

세 가지 비용이 따라온다. (1) *컴파일 타임 검증*이 사라진다. JSON 필드가 도메인과 어긋나도 빌드는 통과하고 런타임에야 깨진다. (2) *IDE 점프*가 끊긴다. 도메인 필드 rename 시 JSON·SQL 파일까지 자동 따라가지 않는다. (3) *fixture 변경 비용*이 커진다. 한 시나리오만 미세 변형하고 싶을 때 새 파일을 만들어야 한다. 그래서 외부화는 *복잡도가 임계치를 넘었을 때*만 한다. 단순 셋업은 코드 안 fixture가 더 좋다.
