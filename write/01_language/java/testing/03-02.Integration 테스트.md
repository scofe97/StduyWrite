# Integration 테스트와 @SpringBootTest

---

> Integration 테스트는 `@SpringBootTest`로 전체 ApplicationContext를 부팅하고 *Testcontainers로 실제 외부 자원*과 결합한다. TPS의 `AbstractMariaDbIntegrationTest`가 그 표준 진입점이다. 이 글은 `@ServiceConnection` + `static { start(); }` + `@DirtiesContext 회피`라는 세 축으로 Integration 테스트의 *결정성과 속도*를 잡는 방법을 정리한다.



## 학습 목표

> Integration 테스트가 *외부 자원과 결합되면서도 빌드 시간을 폭증시키지 않는* 패턴을 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. `@SpringBootTest` + Testcontainers MariaDB 조합이 *왜 표준*인지 설명할 수 있다.
2. `@ServiceConnection`이 어떻게 datasource 설정을 *자동 주입*하는지 설명할 수 있다.
3. `static { MARIADB.start(); }` 패턴이 *JVM 전체에서 컨테이너를 공유*하는 메커니즘을 설명할 수 있다.
4. `@DirtiesContext`를 *피해야* 하는 이유와 대안을 설명할 수 있다.
5. `@ActiveProfiles("test")`로 *환경 격리*를 만드는 패턴을 설명할 수 있다.



## 1. AbstractMariaDbIntegrationTest — 표준 진입점

TPS Integration 테스트의 표준 베이스 클래스다(`operator/ticket/src/test/java/.../testsupport/AbstractMariaDbIntegrationTest.java`).

```java
@Tag("integration")
@ActiveProfiles("test")
@SpringBootTest
public abstract class AbstractMariaDbIntegrationTest {

    @ServiceConnection
    @SuppressWarnings("resource")
    protected static final MariaDBContainer<?> MARIADB =
            new MariaDBContainer<>(DockerImageName.parse("mariadb:10.11"))
                    .withDatabaseName("TPS")
                    .withUrlParam("characterEncoding", "utf8")
                    .withUrlParam("useUnicode", "true");

    static { MARIADB.start(); }

    @Autowired private JdbcTemplate jdbcTemplate;

    @BeforeEach
    protected void setUpBaseData() {
        TicketTestUserContext.setDefaultUser();
        TicketBaseDataSeeder.resetAndSeed(jdbcTemplate);
    }

    @AfterEach
    protected void clearUserContext() {
        TicketTestUserContext.clear();
    }
}
```

여섯 가지 좋은 패턴이 결합되어 있다.

1. **`@Tag("integration")`** — `gradle test`가 *unit/integration을 선별 실행*할 수 있게 분류.
2. **`@ActiveProfiles("test")`** — `application-test.yml` 로딩, 환경 격리.
3. **`@SpringBootTest`** — 전체 Context 부팅.
4. **`@ServiceConnection`** — 컨테이너의 jdbc URL·user·password를 Spring datasource에 자동 주입.
5. **`static { MARIADB.start(); }`** — JVM 부팅 시 한 번만 컨테이너를 띄우고 *모든 테스트 클래스가 공유*.
6. **`@BeforeEach` 시더 + `@AfterEach` 정리** — 테스트 격리 보장.



## 2. @ServiceConnection — 설정 자동 주입

Spring Boot 3.1+에서 도입된 어노테이션이다. 컨테이너가 제공하는 URL·user·password를 *Spring datasource에 자동 매핑*한다.

```java
// Before (Boot 3.0 이하)
@DynamicPropertySource
static void datasourceProps(DynamicPropertyRegistry registry) {
    registry.add("spring.datasource.url", MARIADB::getJdbcUrl);
    registry.add("spring.datasource.username", MARIADB::getUsername);
    registry.add("spring.datasource.password", MARIADB::getPassword);
}

// After (Boot 3.1+)
@ServiceConnection
static final MariaDBContainer<?> MARIADB = ...;
```

세 줄의 보일러플레이트가 *어노테이션 한 줄*로 대체된다. 그리고 `@ServiceConnection`은 *컨테이너 종류를 인식*해 적절한 spring property를 자동 매핑한다. Kafka·Redis·PostgreSQL·MongoDB도 같은 방식으로 지원한다.



## 3. static { start(); } — 컨테이너 공유 패턴

Testcontainers는 *기본적으로* 컨테이너의 라이프사이클을 어떻게 관리할지 사용자에게 맡긴다. 두 가지 패턴이 있다.

### 3.1 @Testcontainers + @Container (테스트 클래스별)

```java
@Testcontainers
class OrderRepositoryTest {
    @Container
    static MariaDBContainer<?> MARIADB = new MariaDBContainer<>(...);
}
```

`@Container` static 필드가 *클래스 단위*로 컨테이너를 시작/중지한다. 클래스가 여러 개면 컨테이너가 *클래스 수만큼* 부팅된다.

### 3.2 static initializer (JVM 단위)

```java
static final MariaDBContainer<?> MARIADB = new MariaDBContainer<>(...);
static { MARIADB.start(); }
```

JVM이 클래스를 로드할 때 단 한 번 컨테이너가 시작된다. JVM이 살아있는 동안 *모든 테스트 클래스가 공유*한다. 명시적 `stop()`이 없어도 *JVM 종료 시* Testcontainers가 hook으로 정리한다(*Ryuk* 컨테이너).

TPS는 후자다. 이유는 *컨테이너 부팅 비용이 크기* 때문이다. MariaDB 컨테이너 시작이 보통 5~10초인데, 클래스마다 부팅하면 빌드 시간이 폭증한다. JVM 공유 패턴이 같은 비용을 한 번만 치른다.



## 4. @DirtiesContext를 피하는 이유

`@DirtiesContext`는 *테스트 후 ApplicationContext를 무효화*해 다음 테스트에서 *새로* 부팅시킨다.

```java
@SpringBootTest
@DirtiesContext  // ← 매 테스트 후 Context 재부팅
class OrderServiceTest { ... }
```

문제는 *Context 부팅 비용*이다. `@SpringBootTest`가 *수 초~수십 초*의 부팅을 요구하므로 `@DirtiesContext`가 붙은 테스트가 100개면 *총 비용이 빌드 시간 분 단위*로 누적된다.

TPS는 이 어노테이션을 *거의 쓰지 않는다*. 대신 다음 패턴으로 격리를 만든다.

1. **`@BeforeEach` 시더로 리셋** — `TicketBaseDataSeeder.resetAndSeed(jdbcTemplate)`가 매 테스트 전 DB를 *초기 상태로* 되돌린다. Context는 재사용된다.
2. **`@Transactional` 자동 롤백** — 영속성 변경만 검증한다면 트랜잭션 범위를 테스트와 묶어 *자동 롤백*.
3. **`@AfterEach`로 ThreadLocal 정리** — `TicketTestUserContext.clear()` 같은 *공유 상태*만 명시적으로 정리.

Context는 재사용하고 *데이터·ThreadLocal·외부 상태*만 매 테스트마다 리셋한다.



## 5. @ActiveProfiles로 환경 격리

`@ActiveProfiles("test")`는 *`application-test.yml`을 로딩*한다. 운영 설정을 *테스트용 설정으로 오버라이드*한다.

```yaml
# application-test.yml (테스트 전용)
spring:
  jpa:
    show-sql: true
    hibernate:
      ddl-auto: validate  # 운영에선 none, 테스트에선 schema 검증
logging:
  level:
    org.okestro.tps: DEBUG
```

여러 profile을 동시에 활성화할 수도 있다.

```java
@ActiveProfiles({"test", "local-docker"})  // 두 profile이 합쳐짐
```

TPS는 *test profile*과 *local-docker profile*을 함께 쓰는 자리가 있다. 후자는 *로컬 docker-compose에서만 활성화되는 PoC 컨트롤러*를 켠다. 03-04에서 본 적이 있는 `@Profile("local-docker")` 컨트롤러가 그 짝이다.



## 6. Context 재사용을 깨뜨리는 함정

같은 `@SpringBootTest`라도 *몇 가지 조건*이 Context를 재초기화시킨다.

| 조건 | 영향 |
|------|------|
| `@MockBean` 조합이 다른 테스트 클래스 | 각 조합마다 Context 새로 부팅 |
| `@DirtiesContext` | 명시적 무효화 |
| `properties = {"foo=bar"}` 추가 | 다른 properties 조합마다 Context 분리 |
| `@TestPropertySource` 다름 | 같은 영향 |
| `@ContextConfiguration` 다름 | 같은 영향 |

빌드 시간을 가장 크게 갉는 것은 *`@MockBean` 조합의 다양성*이다. 02-02에서 다룬 대로, `@MockBean` 대신 `@TestConfiguration`을 import하면 *Context 재사용률*이 높아진다.



## 7. PipelineRegistrationAdapterIT 사례

TPS `operator/app/src/test/java/.../PipelineRegistrationAdapterIT.java`가 Integration 테스트의 한 변종이다. *adapter 계층*만 검증한다.

```java
@SpringBootTest
@ActiveProfiles("test")
class PipelineRegistrationAdapterIT {
    @Autowired private PipelineRegistrationAdapter adapter;
    @MockBean private ExternalRegistryClient client;  // ← 외부만 mock

    @Test
    void register_callsClientAndPersists() {
        adapter.register(command);
        verify(client).register(...);
        // DB 검증
    }
}
```

전체 Context를 부팅하지만 *외부 시스템(ExternalRegistryClient)만 mock*. adapter ↔ DB 결합은 *실제*로 검증된다. *통합 테스트의 가장 흔한 패턴*이다.



## 8. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. `@ServiceConnection`이 `@DynamicPropertySource`보다 좋은 점은?

*보일러플레이트 제거와 컨테이너 타입 인식*이다. `@DynamicPropertySource`는 jdbc URL·user·password를 *명시적으로* registry에 등록한다. `@ServiceConnection`은 컨테이너 클래스(`MariaDBContainer`, `KafkaContainer`)를 보고 *적절한 Spring property*를 자동 매핑한다. 코드가 짧아지고 *Spring Boot의 datasource auto-configuration과 잘 맞물린다*. 단 Boot 3.1+ 버전 요구.

### Q2. `static { MARIADB.start(); }`가 *JVM 종료 시 정리*되는 메커니즘은?

Testcontainers는 *Ryuk*이라는 별도 컨테이너를 띄워 *부모 JVM이 죽으면 자식 컨테이너를 강제 종료*한다. JVM 정상 종료 시에는 Testcontainers의 shutdown hook도 동시에 동작한다. 결과적으로 사용자는 `stop()`을 명시적으로 부르지 않아도 *리소스 누수*가 생기지 않는다. 다만 *CI 환경에서 Docker 권한*이 Ryuk을 띄울 수 있어야 한다.

### Q3. `@DirtiesContext`를 *언제 어쩔 수 없이* 써야 하나?

세 자리다. (1) *Spring Cache 같은 글로벌 캐시*를 비워야 할 때 — Context 안에 박힌 캐시 빈을 외부에서 리셋할 방법이 없으면. (2) *@Configuration 변경*을 검증해야 할 때 — Context 부팅 시 평가되는 빈을 다른 설정으로 다시 테스트해야 하면. (3) *외부 라이브러리가 ApplicationContext 상태를 더럽히는* 알 수 없는 자리. 셋 다 *최후의 수단*이다. 가능한 한 시더·ThreadLocal 정리·트랜잭션 롤백으로 해결한다.

### Q4. Context 재사용률을 가장 크게 해치는 요인은?

*`@MockBean` 조합의 다양성*이다. 같은 `@SpringBootTest`라도 `@MockBean` 빈 목록이 다르면 Spring TestContext가 *Context를 별도로 캐시*한다. 100개 테스트가 각자 다른 `@MockBean` 조합을 쓰면 Context가 100번 부팅된다. 대안은 *`@TestConfiguration`*에 공통 Mock 빈을 모으고 모든 테스트가 import하게 만드는 것. 빌드 시간이 분 단위로 차이 난다.

### Q5. TPS가 `@DataJpaTest` 대신 *full `@SpringBootTest` + MariaDB 컨테이너*를 선호하는 이유는?

`@DataJpaTest`는 *내장 H2*로 datasource를 자동 교체하고 JPA layer만 로드한다. TPS는 MariaDB 방언(`UPSERT`, `FOR UPDATE SKIP LOCKED`)을 검증해야 하고, 영속성 외 *서비스·이벤트 발행*까지 같이 검증하는 자리가 많다. `@SpringBootTest` + 컨테이너 조합이 *현실 환경과 가장 가까운* 검증을 만든다. 그 비용은 `static { start(); }` JVM 공유 + Context 재사용으로 *상수 시간*에 가깝게 묶었다.
