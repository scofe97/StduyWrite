# Kafka·Outbox·Inbox 테스트

---

> Kafka 통합 테스트는 *비동기 결과*와 *외부 자원*이 결합되어 단위 테스트보다 결정성이 약하다. `@EmbeddedKafka`로 가벼운 broker를 띄우고, Awaitility로 *결과를 기다리며 검증*하고, ArchUnit으로 *컨벤션 위반*을 빌드 단계에서 막는 세 도구가 결합돼야 *결정적인 Kafka 테스트*가 만들어진다.



## 학습 목표

> Kafka 테스트의 *비동기성·외부 자원·컨벤션 강제* 세 축을 어떻게 도구로 푸는지 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. `@EmbeddedKafka`와 Testcontainers Kafka의 차이를 *부팅 비용·현실성*으로 비교할 수 있다.
2. Awaitility의 polling이 *비동기 검증의 결정성*을 만드는 메커니즘을 설명할 수 있다.
3. `RetryableTopicConfigurationGuardTest` 같은 *가드 테스트*가 빌드 단계에서 무엇을 막는지 설명할 수 있다.
4. ArchUnit이 *아키텍처 규칙*을 빌드 단계로 끌어오는 방식을 설명할 수 있다.
5. Outbox·Inbox 패턴의 IT가 *어디까지 검증하는지*를 설명할 수 있다.



## 1. @EmbeddedKafka vs Testcontainers Kafka

Kafka 테스트의 broker 선택지는 두 가지다.

### 1.1 @EmbeddedKafka

`spring-kafka-test`가 제공하는 *JVM 내장 Kafka broker*. JUnit 라이프사이클에 묶여 *경량으로 시작*한다.

```java
@SpringBootTest
@EmbeddedKafka(partitions = 3, topics = {"orders", "orders.dlt"})
class OrderEventListenerIT {
    @Autowired private KafkaTemplate<String, String> kafkaTemplate;
    @Autowired private OrderRepository orderRepository;

    @Test
    void publish_persistsOrder() throws Exception {
        kafkaTemplate.send("orders", "O-1", payload);

        await().atMost(5, SECONDS)
            .untilAsserted(() -> assertThat(orderRepository.findById("O-1")).isPresent());
    }
}
```

장점: *부팅이 빠르다*. 별도 Docker 컨테이너가 필요 없다.
단점: *실제 운영 환경과 다르다*. KRaft·복제·SASL 같은 운영 설정이 제외된다.

### 1.2 Testcontainers Kafka(또는 Redpanda)

```java
@SpringBootTest
class OrderEventListenerIT {
    @ServiceConnection
    static final KafkaContainer KAFKA = new KafkaContainer(
        DockerImageName.parse("confluentinc/cp-kafka:7.5.0"));

    static { KAFKA.start(); }
}
```

장점: *운영 환경에 가장 가깝다*. 실제 broker가 동작한다.
단점: *부팅 비용이 크다*. Docker 컨테이너 시작이 10~20초.

TPS는 `@EmbeddedKafka`를 우선 쓰고, *Redpanda 특유 동작*(SASL, transactional id)을 검증할 때 Testcontainers로 옮긴다. `message-lib`의 `OutboxPollerIT`가 `@EmbeddedKafka` + Testcontainers MariaDB의 *혼합 패턴*이다.



## 2. Awaitility — 비동기 검증의 표준

Kafka 메시지 발행 후 *컨슈머가 처리할 때까지*는 비결정적이다. `Thread.sleep(500)` 같은 fixed wait는 *flaky 테스트의 원흉*이다. Awaitility가 그 자리를 채운다.

```java
import static org.awaitility.Awaitility.await;
import static java.util.concurrent.TimeUnit.SECONDS;

@Test
void publish_triggersConsumer() {
    kafkaTemplate.send("orders", "O-1", payload);

    await()
        .atMost(5, SECONDS)
        .pollInterval(Duration.ofMillis(100))
        .untilAsserted(() -> {
            Order order = orderRepository.findById("O-1").orElseThrow();
            assertThat(order.getStatus()).isEqualTo(PROCESSED);
        });
}
```

세 가지 좋은 패턴이 결합돼 있다.

1. **`atMost(5, SECONDS)`** — 최대 대기 시간을 *명시적*으로 제한. 영원히 기다리는 일을 막는다.
2. **`pollInterval(100ms)`** — 짧은 간격으로 *반복 확인*. 결과가 빨리 오면 빨리 종료된다.
3. **`untilAsserted(...)`** — 람다 안 단정이 *성공할 때까지* 반복. AssertJ를 그대로 쓸 수 있다.

같은 검증을 `Thread.sleep(5000)`으로 쓰면 *항상 5초를 소모*하고, 결과가 5초 후 도착하면 *우연히 통과*한다. Awaitility는 비결정성을 *결정적인 검증*으로 바꾼다.



## 3. OutboxPollerIT — Outbox 통합 검증

TPS `message-lib/src/test/java/.../application/outbox/OutboxPollerIT.java`가 Outbox 패턴의 *end-to-end* 검증을 담당한다.

```java
@SpringBootTest
@EmbeddedKafka(partitions = 1, topics = {"test.outbox.topic"})
@Testcontainers
class OutboxPollerIT {

    @Container
    static MariaDBContainer<?> MARIADB = new MariaDBContainer<>(...);

    @DynamicPropertySource
    static void datasourceProps(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", MARIADB::getJdbcUrl);
        // ...
    }

    @Autowired private JpaOutboxEventRepository repository;
    @Autowired private KafkaTemplate<String, byte[]> kafkaTemplate;

    @Test
    @DisplayName("PENDING outbox row가 폴러에 의해 Kafka로 발행되고 SENT로 마킹된다")
    void pollAndPublish_marksSent() {
        repository.save(OutboxEvent.builder()
            .aggregateId("A-1").payload(new byte[]{1}).topic("test.outbox.topic")
            .status(PENDING).build());

        await().atMost(5, SECONDS).untilAsserted(() -> {
            OutboxEventEntity entity = repository.findById(id).orElseThrow();
            assertThat(entity.getStatus()).isEqualTo(SENT);
        });

        // Kafka 토픽에도 도달했는지 확인
        ConsumerRecords<String, byte[]> records = consumer.poll(Duration.ofSeconds(2));
        assertThat(records).hasSize(1);
    }
}
```

검증 범위가 *완전한 흐름*이다. (1) outbox INSERT → (2) 폴러가 SELECT → (3) Kafka 발행 → (4) outbox row를 SENT로 마킹. 같은 흐름을 단위 테스트로는 검증할 수 없다. *부분 실패가 발생하지 않는지* 보려면 Integration이 필수다.



## 4. RetryableTopicConfigurationGuardTest — 컨벤션 가드

TPS의 *가드 테스트* 패턴이다. `executor/engine/src/test/java/.../common/messaging/RetryableTopicConfigurationGuardTest.java`.

```java
@Test
@DisplayName("executor/engine 의 @RetryableTopic 사용 메서드는 모두 kafkaTemplate=\"retryKafkaTemplate\" 을 지정해야 한다")
void allRetryableTopicAnnotations_referenceRetryKafkaTemplate() throws Exception {
    List<Class<?>> classes = scanClasses("org.okestro.tps.common.messaging");

    List<String> violations = new ArrayList<>();
    for (Class<?> clazz : classes) {
        for (Method method : clazz.getDeclaredMethods()) {
            RetryableTopic retryable = method.getAnnotation(RetryableTopic.class);
            if (retryable == null) continue;
            if (!"retryKafkaTemplate".equals(retryable.kafkaTemplate())) {
                violations.add(clazz.getName() + "#" + method.getName());
            }
        }
    }
    assertThat(violations).isEmpty();
}
```

이 테스트는 *기능을 검증하지 않는다*. *코드 컨벤션*을 검증한다. `@RetryableTopic` 어노테이션의 `kafkaTemplate` 속성을 빈 채로 두면 *Avro가 아닌 byte[] 직렬화기*가 선택되어 `ClassCastException`이 무한 루프가 된다. 한 번 사고가 났던 자리를 *PR 리뷰가 아니라 빌드 단계*에서 막는다.

가드 테스트의 가치는 *과거 사고가 코드로 박혀 있다는 것*이다. 새 합류자가 같은 실수를 다시 했을 때 *빌드 실패가 즉시* 알려준다.



## 5. ArchUnit — 아키텍처 컨벤션

TPS `executor/engine/src/test/java/.../jenkins/ArchitectureTest.java`가 ArchUnit의 사용 예다. *계층 간 의존성·패키지 가시성·어노테이션 사용 규칙*을 코드로 표현한다.

```java
@AnalyzeClasses(packages = "org.okestro.tps.jenkins")
class ArchitectureTest {

    @ArchTest
    static final ArchRule domain_shouldNotDependOn_infrastructure =
        noClasses().that().resideInAPackage("..domain..")
            .should().dependOnClassesThat().resideInAPackage("..infrastructure..");

    @ArchTest
    static final ArchRule controller_shouldHave_RestController =
        classes().that().haveSimpleNameEndingWith("Controller")
            .should().beAnnotatedWith(RestController.class);
}
```

가드 테스트와 같은 의도지만 *훨씬 표현적*이다. ArchUnit의 fluent API가 *아키텍처 규칙을 영어 문장처럼* 표현한다. *port-adapter 계층 분리*, *외부 라이브러리 의존 제한*, *컨벤션 강제*가 모두 ArchUnit으로 들어간다.



## 6. KafkaErrorIntegrationIT — DLT 흐름 검증

TPS `message-lib/src/test/java/.../config/KafkaErrorIntegrationIT.java`가 *에러 핸들러*의 통합 검증을 담당한다.

```java
@SpringBootTest
@EmbeddedKafka(partitions = 1, topics = {"test.topic", "tps.dlq"})
class KafkaErrorIntegrationIT {

    @Test
    @DisplayName("AvroSerializationException은 재시도 없이 즉시 DLQ로 라우팅된다")
    void notRetryableException_routesToDlqImmediately() {
        // Given: 호환 안 되는 페이로드 발행
        kafkaTemplate.send("test.topic", "K-1", invalidAvroBytes);

        // Then: DLQ 토픽에 도달
        await().atMost(5, SECONDS).untilAsserted(() -> {
            ConsumerRecords<String, byte[]> dlq = consumer.poll(Duration.ofMillis(500));
            assertThat(dlq).hasSize(1);
            // DLT 헤더 검증
            Header exceptionFqcn = dlq.iterator().next().headers()
                .lastHeader("kafka_dlt-exception-fqcn");
            assertThat(new String(exceptionFqcn.value()))
                .contains("AvroSerializationException");
        });
    }
}
```

검증 범위: `DefaultErrorHandler` + `DeadLetterPublishingRecoverer` + `addNotRetryableExceptions` 설정이 *함께 동작해 즉시 DLQ로 라우팅*되는지. 같은 검증을 단위로는 만들 수 없다. Spring Kafka의 *실제 흐름*이 필요하다.



## 7. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. `@EmbeddedKafka`와 Testcontainers Kafka를 *언제* 갈라 쓰나?

기본은 `@EmbeddedKafka`다. 부팅이 1~2초로 빠르고 대부분 검증에 충분하다. Testcontainers는 *운영 broker 특유 동작*(KRaft, SASL, transactional id, schema registry 통합)을 검증해야 할 때 선택한다. 부팅이 10~20초로 무거우므로 *해당 테스트만* Testcontainers를 쓰고 나머지는 `@EmbeddedKafka`를 유지하는 *혼합 전략*이 흔하다.

### Q2. Awaitility 없이 `Thread.sleep`을 쓰면 무엇이 깨지나?

*결정성과 속도가 동시에 깨진다*. (1) sleep이 너무 짧으면 *결과가 도착하기 전에 검증*해 flaky 테스트가 된다. (2) sleep이 충분히 길면 *결과가 빨리 와도 무조건 그 시간을 소비*해 빌드가 느려진다. Awaitility는 *짧은 polling으로 결과를 빨리 잡되 최대 대기를 명시*해 두 문제를 동시에 푼다.

### Q3. 가드 테스트(`RetryableTopicConfigurationGuardTest`)의 본질적 가치는?

*과거 사고를 코드로 박는다*. 한 번 일어난 사고(`@RetryableTopic`의 `kafkaTemplate` 미지정으로 `ClassCastException` 무한 루프)는 *PR 리뷰만으로는 다시 막을 수 없다*. 새 합류자가 같은 실수를 하면 코드 리뷰에서 놓칠 가능성이 크다. 가드 테스트는 *빌드 실패라는 강제 신호*로 그 실수를 잡는다. ArchUnit의 표현이 더 자연스럽지만 단순 가드는 *리플렉션 한 줄*로도 충분하다.

### Q4. ArchUnit 규칙을 *어디까지* 강제할 수 있나?

세 단계가 있다. (1) *계층 의존성* — domain → infrastructure 단방향 같은 hexagonal 규칙. (2) *어노테이션 사용* — Controller에는 `@RestController`, Service에는 `@Service` 같은 마킹 강제. (3) *명명 규칙* — `*Repository` 인터페이스만 `@Repository` 등. 단점은 *반례를 빠짐없이 표현하기 어렵다는 것*. ArchUnit이 표현 못 하는 자리는 가드 테스트(리플렉션)나 `@CodeSmell` 주석으로 보완한다.

### Q5. Outbox IT가 *어디까지* 검증해야 충분한가?

세 자리는 필수다. (1) *transaction 묶음* — 도메인 변경과 outbox INSERT가 같은 transaction에 들어가는지. (2) *폴러의 동작* — PENDING row가 SENT로 마킹되고 Kafka에 도달하는지. (3) *재시도 정책* — 발행 실패 시 retry_count 증가, 임계치 초과 시 DEAD 마킹. 추가로 *동시성*(FOR UPDATE SKIP LOCKED)도 멀티 인스턴스 환경에서 검증하면 좋다. 단위 테스트로는 (1)·(3)이 일부 가능하지만 (2)는 IT가 강제된다.
