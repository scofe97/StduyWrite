# Mock 전략과 설계 피드백

---

> Mock은 *언제 쓰고 언제 피하는지*가 핵심이다. 카카오페이 Mock Part 1·2가 던지는 두 축 — *Mock 도구의 진화*(Mock Server → `@MockBean` → `@TestConfiguration` → `java-test-fixtures`)와 *테스트가 설계 피드백을 만드는 방식* — 을 TPS 코드(`EventPublisherTest`, `DeployServiceTest`)에 매핑한다.



## 학습 목표

> Mock 도구 네 가지의 적합 시점과 *Mock이 어렵게 만드는 코드*를 어떻게 리팩터링 신호로 읽는지 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. Mock Server·`@MockBean`·`@TestConfiguration`·`java-test-fixtures`의 적합 시점을 비교할 수 있다.
2. *main 디렉터리에 Mock Bean을 등록하면 안 되는* 이유를 설명할 수 있다.
3. ArgumentCaptor가 behavior verification에서 어떤 자리를 채우는지 설명할 수 있다.
4. "Mock 셋업이 거대하다 = 설계 개선 신호"라는 피드백 루프를 설명할 수 있다.
5. *Black Box 영역 격리*가 mock 전략에서 중심에 있는 이유를 설명할 수 있다.



## 1. Mock 도구의 진화

카카오페이 Mock Part 1은 Mock 도구의 *역사적 진화*를 정리한다. 각 단계가 *이전 단계의 한계*를 푼다.

### 1.1 Mock Server

HTTP 통신을 실제로 진행하되 *대상 서버를 Mock으로* 띄운다. WireMock, MockWebServer, Mountebank가 대표 예다.

장점: *실제 환경과 가장 유사*. HTTP 직렬화·헤더·상태 코드까지 검증된다.
단점: *의존하는 모든 구간*에 Mock Server를 띄워야 한다. 많아지면 운영 비용이 폭증한다.

TPS의 `JenkinsWireMockSupport`(`executor/engine/src/test/java/.../e2e/wiremock/JenkinsWireMockSupport.java`)가 이 패턴이다. 04-02에서 본격적으로 다룬다.

### 1.2 @MockBean

Spring Test의 `@MockBean`은 *ApplicationContext의 특정 빈을 Mock으로 교체*한다.

```java
@SpringBootTest
class OrderServiceTest {
    @MockBean private PaymentGateway paymentGateway;
    @Autowired private OrderService orderService;

    @Test
    void placeOrder() {
        given(paymentGateway.charge(...)).willReturn(SUCCESS);
        orderService.place(...);
        verify(paymentGateway).charge(...);
    }
}
```

장점: *Mock 셋업이 간단*. Mock Server를 별도로 띄울 필요 없다.
단점: *ApplicationContext 재초기화*가 일어난다. `@MockBean`이 들어간 테스트마다 새 Context가 만들어져 빌드 속도가 급격히 느려진다.

### 1.3 @TestConfiguration

`@TestConfiguration` 클래스에 *Mock 빈을 등록*하고 여러 테스트에서 공유한다.

```java
@TestConfiguration
class TestPaymentConfig {
    @Bean
    PaymentGateway paymentGateway() {
        return mock(PaymentGateway.class);
    }
}

@SpringBootTest
@Import(TestPaymentConfig.class)
class OrderServiceTest { ... }
```

장점: *Context 재사용 가능*. 같은 Mock 설정을 쓰는 테스트들이 Context를 공유한다.
단점: *멀티모듈 환경에서 접근 불가*. `src/test` 안에 있어서 다른 모듈에서 import할 수 없다.

### 1.4 java-test-fixtures

`src/testFixtures/java`에 Mock 설정·fixture를 두고 *모듈 의존성으로 공유*한다.

```groovy
// build.gradle
plugins { id 'java-test-fixtures' }

// 다른 모듈에서
testImplementation testFixtures(project(':message-lib'))
```

장점: *멀티모듈 안전 분리*. 운영 코드는 그대로 두고 *테스트 의존성만* 공유한다.
단점: 학습 곡선과 의존성 그래프가 복잡해진다.

### 1.5 비교

| 도구 | 사용 자리 | Context 재사용 | 멀티모듈 |
|------|----------|--------------|---------|
| Mock Server | 외부 HTTP 의존성 | 무관 | 가능 |
| `@MockBean` | 단일 빈 mock | **불가** (Context 재초기화) | 가능 |
| `@TestConfiguration` | 공유 Mock 빈 | 가능 | **불가** |
| `java-test-fixtures` | 모듈 간 공유 fixture | 가능 | 가능 |

요점: `@MockBean`은 *간단하지만 비쌌다*. `@TestConfiguration`은 *Context 재사용을 해결*했지만 멀티모듈에서 막혔다. `java-test-fixtures`가 *둘 다 푼다*.



## 2. 가장 강하게 비판하는 안티패턴

카카오페이 Mock Part 1은 *테스트 편의를 위해 main에 Mock Bean을 등록하는 관행*을 가장 강하게 비판한다.

```java
// src/main/java/.../config/TestableConfig.java  ← 운영 코드에 있음!
@Configuration
class TestableConfig {
    @Bean
    @ConditionalOnProperty("app.use-mock-payment")
    PaymentGateway paymentGateway() {
        return new MockPaymentGateway();  // ← 테스트용 Mock이 운영 빌드에 포함됨
    }
}
```

문제는 두 가지다.

1. **운영 코드 오염** — 테스트 의도가 main 디렉터리에 박혀 *코드 리뷰가 어려워진다*. "이 빈은 운영용인가 테스트용인가?"가 매번 헷갈린다.
2. **프로덕션 배포 사고** — `app.use-mock-payment=true`가 *실수로 운영 환경에 켜지면* 결제가 가짜로 통과한다. 카카오페이 글이 강조하는 "실수로 프로덕션 배포 시 장애" 시나리오다.

해결책은 *테스트 자원은 항상 `src/test` 또는 `src/testFixtures`에만 둔다*는 원칙이다. main에 들어가는 순간 *런타임 의존성 그래프에 박힌다*.



## 3. ArgumentCaptor — behavior verification의 보조 도구

mock 검증은 두 갈래다.

- **State verification** — mock이 반환한 값으로 *결과 상태*를 검증.
- **Behavior verification** — mock이 *어떻게 호출되었는지*를 검증.

ArgumentCaptor는 behavior verification에서 *호출 인자를 캡처*해 추가 검증할 때 쓴다.

TPS `EventPublisherTest`(`message-lib/src/test/java/.../application/outbox/EventPublisherTest.java`).

```java
@Test
@DisplayName("SpecificRecord 발행 시 AvroSerializer 직렬화 결과가 OutboxEvent.payload 로 저장된다")
void publish_specificRecord_serializesAndSaves() {
    when(avroSerializer.serialize(eq("topic-a"), any(ExampleMessageAvro.class)))
            .thenReturn(new byte[]{1, 2, 3});
    EventPublisher publisher = new EventPublisher(repository, avroSerializer);

    ExampleMessageAvro record = ExampleMessageAvro.newBuilder()
            .setMessage("hi").setCorrelationId("corr-1").build();
    publisher.publish("agg-1", record, "topic-a");

    ArgumentCaptor<OutboxEvent> captor = ArgumentCaptor.forClass(OutboxEvent.class);
    verify(repository).save(captor.capture());

    OutboxEvent saved = captor.getValue();
    assertThat(saved.getPayload()).isEqualTo(new byte[]{1, 2, 3});
    assertThat(saved.getTopic()).isEqualTo("topic-a");
    assertThat(saved.getAggregateId()).isEqualTo("agg-1");
}
```

세 가지 좋은 패턴이 모여 있다.

1. **`when(...).thenReturn(...)`** — AvroSerializer가 *어떤 결과를 반환할지* 셋업.
2. **`verify(repository).save(captor.capture())`** — 저장 *호출이 일어났음*을 검증.
3. **`captor.getValue()`** — 호출 인자(`OutboxEvent`)를 *꺼내 추가 검증*.

검증의 시선은 *EventPublisher가 OutboxEvent를 올바르게 조립했는지*다. ArgumentCaptor가 없으면 OutboxEvent 인스턴스에 *어떤 값이 들어갔는지* 검증할 방법이 없다. mock의 사이드이펙트를 들여다보는 창이다.



## 4. when().thenReturn() vs given().willReturn()

같은 mock 설정의 두 어휘다. 기능은 동일하다.

```java
// Mockito.when — "이 호출에 이걸 반환"
when(deployJobReader.findLatestContainerManifestId("JOB_1"))
    .thenReturn("MFST_001");

// BDDMockito.given — "이런 상태가 주어졌을 때"
given(deployJobReader.findLatestContainerManifestId("JOB_1"))
    .willReturn("MFST_001");
```

01-02에서 다룬 바와 같이 *BDD 시나리오 안에서는* `given()`이 자연스럽다. Mockito `when()`은 *시나리오 외부의 일반 mock 설정*에 적합하다. 한 테스트 안에서 둘을 *섞지 않는 것*이 일관성 측면에서 좋다.

TPS는 두 어휘가 *공존*한다. `DefaultDeployDomainServiceTest`는 `given()`을, `EventPublisherTest`는 `when()`을 쓴다. 같은 테스트 안에서는 통일됐다.



## 5. Mock 셋업이 거대하다 = 설계 개선 신호

카카오페이 Mock Part 2의 핵심이다. 한 테스트에서 mock을 *7~8개* 설정하고 있다면 그 자체가 *책임 분리가 안 된* 신호다.

### 5.1 책임 분리 사례

카카오페이 Part 2의 PartnerClient 사례를 풀어 본다. 한 클래스가 HTTP 통신 *와* 비즈니스 로직을 *둘 다* 가지면 테스트할 때 두 종류 mock을 모두 셋업해야 한다.

```java
// Before: 한 클래스에 두 책임
class PartnerClient {
    public Order fetchAndProcess(String orderId) {
        HttpResponse response = httpClient.get("/orders/" + orderId);  // HTTP
        Order order = parse(response);  // 비즈니스 로직
        order.applyDiscount();  // 비즈니스 로직
        return order;
    }
}

// 테스트: HTTP mock + 비즈니스 검증이 한 자리에 섞임
@Mock HttpClient httpClient;
when(httpClient.get(...)).thenReturn(mockResponse);  // HTTP mock
assertThat(client.fetchAndProcess("O-1").getAmount()).isEqualTo(900);  // 비즈니스
```

```java
// After: 책임 분리
class PartnerHttpClient {
    public HttpResponse fetch(String orderId) { ... }  // HTTP만
}

class OrderProcessor {
    public Order process(HttpResponse response) { ... }  // 비즈니스만
}

// 테스트: 각자 자기 책임만 검증
class PartnerHttpClientTest {
    // Mock Server로 HTTP만 검증
}
class OrderProcessorTest {
    // POJO 단위 테스트로 비즈니스만 검증
}
```

mock 설정 수가 줄고, *각 테스트가 자기 관심사에 집중*한다. Part 2의 메시지는 "테스트가 어렵다면 *그 어려움이 곧 설계 피드백*"이다.

### 5.2 외부 의존성 차단

같은 글은 `ResponseEntity`를 직접 반환하는 패턴도 비판한다. 라이브러리 타입이 *상위 모듈로 전파*되면 라이브러리 변경 시 영향이 모든 호출자에 퍼진다. `Pair<Int, T?>` 같은 *언어 기본 타입*으로 감싸면 외부 의존성이 어댑터 안에 갇힌다.



## 6. Black Box 영역 격리

카카오페이 Mock Part 1의 결론 메시지다.

> "Black Box 영역을 격리하는 것이 가장 중요하다. 테스트 불가능한 영역이 다른 구간으로 전이되지 않도록 설계해야 진정한 테스트 가능성을 확보할 수 있다."

Black Box는 *제어할 수 없는 외부 시스템*(외부 API, 시간, 랜덤, DB, 메시지 큐)이다. 이게 도메인 코드에 *직접 섞이면* 그 자리는 영영 테스트가 어려워진다.

해결책은 *port-adapter 패턴*이다.

```
도메인 ── 포트(인터페이스) ── 어댑터 ── 외부 시스템

도메인은 포트만 안다. 테스트에서 포트는 mock으로 대체한다.
어댑터는 별도 통합 테스트로 외부 시스템과의 결합을 검증한다.
```

이 분리가 만들어지면 도메인 단위 테스트가 *Black Box 없이* 가능해진다. 어댑터 통합 테스트는 *Black Box를 격리한 상태로* 검증된다. 두 자리가 깔끔히 갈린다.

TPS `EventPublisher`가 그 예다. `OutboxEventRepository` 인터페이스를 mock으로 대체하고 *EventPublisher의 도메인 로직만* 검증한다. JPA 결합은 `JpaOutboxEventRepositoryIT`라는 별도 IT에서 검증한다.



## 7. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. `@MockBean`이 *왜 비싸*고 `@TestConfiguration`이 어떻게 그 비용을 푸는가?

`@MockBean`은 *ApplicationContext의 빈을 런타임에 교체*한다. 빈 교체는 Context 캐시를 무효화하므로 `@MockBean` 조합이 다른 테스트마다 *새 Context*가 부팅된다. 100개 테스트가 각자 다른 `@MockBean` 조합을 쓰면 Context 부팅이 100번 일어난다. `@TestConfiguration`은 *Mock 빈을 정적으로* 등록하므로 같은 Configuration을 import하는 테스트들이 *Context를 재사용*한다. 빌드 시간이 분 단위로 차이 난다.

### Q2. main 디렉터리에 Mock Bean을 등록하면 안 되는 *기술적* 이유는?

런타임 의존성 그래프에 *테스트 의도가 박힌다*. 첫째, `@ConditionalOnProperty` 같은 가드가 *실수로 운영 환경에서 켜지면* 가짜 빈이 운영을 처리한다. 둘째, *운영 빌드 아티팩트에 Mock 코드가 포함*되어 보안·라이선스 문제가 생긴다. 셋째, 코드 리뷰에서 *"이 빈은 운영용인가 테스트용인가"*가 매번 헷갈려 의도가 흐려진다. 테스트 자원은 항상 `src/test` 또는 `src/testFixtures`에만 둔다.

### Q3. ArgumentCaptor를 *언제* 쓰고 *언제 피해야* 하나?

쓸 때: mock이 받는 *복잡한 객체*의 내부 값을 검증할 때. 단순 비교(`eq(...)`)로는 표현이 어려운 자리. 피할 때: 검증이 한두 필드면 `argThat(...)` 같은 *matcher*가 더 짧다. 또 *너무 깊은 검증*은 mock 검증이 SUT 내부 구현을 *너무 들여다보는* 신호다. 그 자리는 state verification으로 바꾸는 게 보통 더 깔끔하다.

### Q4. "Mock이 7~8개 셋업되어 있다"는 신호를 어떻게 리팩터링으로 연결하나?

세 단계로 본다. (1) *책임이 한 클래스에 너무 많은지* — HTTP·비즈니스·영속성이 한 자리에 섞여 있다면 가른다. (2) *외부 의존성이 너무 깊은지* — 라이브러리 타입이 도메인까지 전파되면 어댑터로 가둔다. (3) *진짜 mock이 필요한지* — POJO로 만들 수 있는 의존성이 있다면 실제 객체를 직접 생성한다. mock 수가 줄면 테스트가 *검증하는 행위에 집중*한다.

### Q5. Black Box 격리가 mock 전략의 중심에 있는 이유는?

*테스트가 어려운 영역을 한 곳에 가두면* 나머지 코드가 자유로워지기 때문이다. 외부 API·시간·랜덤·DB가 도메인 곳곳에 직접 섞이면 *모든 곳이 테스트 어려운 영역*이 된다. 어댑터로 격리하면 *어댑터 한 곳만* 통합 테스트가 필요하고 도메인은 POJO 단위 테스트로 끝난다. mock 전략은 결국 *Black Box를 어디에 가둘 것인가*의 결정이다.
