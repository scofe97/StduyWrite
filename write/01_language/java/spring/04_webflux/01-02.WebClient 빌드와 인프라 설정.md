# WebClient 빌드와 인프라 설정

---

> WebClient를 만드는 방법은 짧게 `WebClient.create()` 한 줄부터, 코덱 한도와 ConnectionProvider 풀까지 손대는 본격 설정까지 폭이 넓다. 본 챕터는 어디까지 기본값에 맡길 수 있고, 어디부터는 명시적으로 잡아야 하는지를 정리한다.



## 두 가지 출발점 — `create()`와 `builder()`

> 한 줄로 끝내는 길과 옵션을 쌓는 길이 있다. 한 번 정한 결정이 의외로 오래 가니 처음에 의식해서 고른다.

가장 짧은 방법은 정적 팩토리 메서드다.

```java
// 1) 가장 단순. 베이스 URL 없이 매 호출마다 절대 URL을 건넨다
WebClient client = WebClient.create();

// 2) 베이스 URL만 박아 두는 형태
WebClient apiClient = WebClient.create("https://api.example.com");
```

위 형태는 옵션을 더 얹을 수 없다. 헤더 기본값, 코덱 한도, 필터, 커넥터를 손대고 싶다면 `builder()`로 출발해야 한다.

```java
WebClient client = WebClient.builder()
        .baseUrl("https://api.example.com")
        .defaultHeader("Accept", "application/json")
        .defaultUriVariables(Map.of("tenant", "default"))
        .build();
```

`builder()`는 다음 메서드를 체인으로 받는다.

- `baseUrl(String)` — 절대 URL이 아닌 경로를 호출하면 자동으로 prefix.
- `defaultHeader(name, values...)`, `defaultCookie(name, values...)` — 모든 요청에 자동 첨부.
- `defaultUriVariables(Map<String, ?>)` — `uri("/users/{tenant}/{id}", Map.of("id", 1))` 같은 호출에서 빠진 변수를 채운다.
- `defaultRequest(Consumer<RequestHeadersSpec>)` — 호출 직전에 모든 요청에 적용할 후크. MDC 기반 헤더 주입에 자주 쓴다.
- `clientConnector(ClientHttpConnector)` — Netty/JDK/Apache 중 어느 transport를 쓸지 결정.
- `exchangeStrategies(ExchangeStrategies)` — 코덱과 메모리 한도를 잡는 자리.
- `filter(ExchangeFilterFunction)` — 로깅·인증·MDC 같은 횡단 관심사. 01-06에서 다룬다.

빌더는 "한 번 만들고 재사용"이 기본이다. `WebClient` 인스턴스 자체는 thread-safe하므로 한 번 만들어 빈으로 두고 여러 호출이 공유한다.



## `WebClient.Builder`를 빈으로 둘 것인가, 완성된 `WebClient`를 빈으로 둘 것인가

> 두 패턴 모두 흔히 보인다. 무엇이 다른지 짚는다.

Spring Boot는 자동 설정으로 `WebClient.Builder` 빈을 제공한다(웹 애플리케이션 또는 webflux starter가 있을 때). 그 빌더를 호출 코드에 주입받아 매번 `build()`하는 패턴을 자주 본다. 반대로 처음 한 번 완성해서 `WebClient` 빈을 두고 그것을 주입받는 패턴도 자주 본다.

```java
// 패턴 A — 완성된 WebClient를 빈으로
@Bean
WebClient apiClient(WebClient.Builder builder) {
    return builder
            .baseUrl("https://api.example.com")
            .defaultHeader("Accept", "application/json")
            .build();
}

@Service
class UserClient {
    private final WebClient apiClient;
    UserClient(WebClient apiClient) { this.apiClient = apiClient; }
    // ...
}
```

```java
// 패턴 B — Builder만 받아 호출 시점에 build
@Service
class UserClient {
    private final WebClient.Builder builder;
    UserClient(WebClient.Builder builder) { this.builder = builder; }

    public Mono<User> get(long id) {
        WebClient client = builder.codecs(c -> c.defaultCodecs().maxInMemorySize(8 * 1024 * 1024))
                .build();
        return client.get().uri("/users/{id}", id).retrieve().bodyToMono(User.class);
    }
}
```

대부분은 패턴 A로 충분하다. 호출 시점마다 다시 `build()`하는 비용은 작지 않다(코덱 등록·헤더 복사 등). 한 번 만들면 재사용하는 편이 일반적이다.

패턴 B가 정당화되는 경우는 다음 두 가지 정도다.

1. **호출마다 코덱·헤더·필터가 다르다.** 외부 API 메타가 DB에 저장돼 있고 요청별로 다르게 조립해야 한다면 빌더에서 시작하는 편이 깔끔하다.
2. **테스트에서 빌더를 주입받아 mock을 끼워 넣는다.** `WebClient.Builder`를 mock으로 대체하기 쉽다.

TPS `ApprovalUrlAdapter`는 패턴 B를 쓴다.

```java
// ApprovalUrlAdapter.java:47, 72-73
private final WebClient.Builder webClientBuilder;
// ...
WebClient webClient = webClientBuilder.codecs(configurer -> configurer.defaultCodecs()
        .maxInMemorySize(50 * 1024 * 1024)).build();
```

이 코드가 패턴 B를 고른 이유는 명시되지 않지만, 첫째 항목(호출마다 메모리 한도가 다를 수 있고, 같은 어댑터가 첨부 파일을 받는 multipart 요청과 가벼운 JSON 요청을 모두 처리한다)에 가깝다. 다만 호출이 들어올 때마다 매번 같은 50MB 한도의 WebClient를 새로 빌드하는 점은 불필요한 반복이다. 한 번 빌드해 필드로 캐시하거나 패턴 A로 빈을 두는 것이 일반적이다. 02-04 사례 분석에서 더 깊이 본다.



## `ExchangeStrategies`와 코덱 메모리 한도

> WebClient가 응답을 어떤 형식으로 디코딩할지, 한 번에 메모리에 올릴 수 있는 최대 크기를 얼마로 둘지를 결정하는 자리다.

WebClient는 응답 본문을 디코딩할 때 등록된 `Encoder`/`Decoder` 목록을 사용한다. JSON은 Jackson, XML은 Jaxb2, 멀티파트는 `MultipartHttpMessageReader` 같은 식이다. 기본 코덱이 충분한 경우가 대부분이지만, 두 가지는 명시적으로 손대는 사례가 흔하다.

### `maxInMemorySize`

기본값은 256KB다. 응답 본문 한 건이 256KB를 넘으면 `DataBufferLimitException`이 떨어진다. 외부 API가 거대한 JSON 배열을 한 번에 내려주는 경우 또는 첨부 파일을 base64로 박아 보내는 케이스에서 자주 만난다.

```java
WebClient client = WebClient.builder()
        .codecs(configurer -> configurer.defaultCodecs().maxInMemorySize(50 * 1024 * 1024))
        .build();
```

`50 * 1024 * 1024`는 50MB. TPS `ApprovalUrlAdapter`가 쓰는 값과 같다. 첨부 파일을 본문에 담아 보내는 multipart 응답을 한 번에 받을 수 있도록 키운 것이다.

⚠️ 한도를 무한히 키우는 것은 답이 아니다. 거대한 응답을 한 번에 메모리에 올리면 GC 압박과 OOM 위험이 동시에 따라온다. 본문이 정말 큰 경우 `bodyToFlux(DataBuffer)`로 청크 단위 처리를 검토한다. 02-01에서 다룬다.

### Jackson `ObjectMapper` 커스터마이즈

Spring Boot는 자동 설정으로 `ObjectMapper`를 만들고 그것을 WebClient의 Jackson 코덱에 연결한다. JSON 직렬화 옵션(예: `WRITE_DATES_AS_TIMESTAMPS`, `SerializationFeature.INDENT_OUTPUT`)을 호출별로 바꾸려면 코덱을 직접 등록한다.

```java
ObjectMapper apiMapper = new ObjectMapper()
        .registerModule(new JavaTimeModule())
        .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);

WebClient client = WebClient.builder()
        .codecs(configurer -> {
            configurer.defaultCodecs().jackson2JsonEncoder(new Jackson2JsonEncoder(apiMapper));
            configurer.defaultCodecs().jackson2JsonDecoder(new Jackson2JsonDecoder(apiMapper));
        })
        .build();
```

같은 앱에서 두 외부 API와 통신하는데 한쪽은 ISO-8601 문자열을, 다른 쪽은 epoch millis를 요구하는 경우가 대표적이다. WebClient 인스턴스를 두 개 두고 코덱만 다르게 잡는 것이 깔끔하다.



## 트랜스포트 — `ClientHttpConnector` 선택

> WebClient는 추상층이고, 실제 IO는 커넥터가 한다. 6.2 시점에서 선택지는 세 가지다.

| 커넥터 | 클래스 | 기본값 | 특징 |
|--------|--------|--------|------|
| Reactor Netty | `ReactorClientHttpConnector` | 기본 | 비동기·논블로킹, 스트리밍, HTTP/2 |
| JDK HttpClient | `JdkClientHttpConnector` | 6.2부터 안정화 | JDK 11+ 내장, 외부 의존성 0 |
| Jetty | `JettyClientHttpConnector` | 옵션 | Jetty Reactive HttpClient 기반 |
| Apache HttpClient 5 | `HttpComponentsClientHttpConnector` | 옵션 | 기존 Apache 생태계 호환 |

Spring Boot는 클래스패스에 Netty가 있으면 Reactor Netty를 자동 선택한다. spring-boot-starter-webflux를 쓰면 거의 항상 Netty다.

JDK 커넥터를 고르는 경우는 Netty 의존성 자체를 빼고 싶을 때, 또는 보안 정책 때문에 외부 라이브러리를 줄이고 싶을 때다.

```java
WebClient client = WebClient.builder()
        .clientConnector(new JdkClientHttpConnector())
        .build();
```

JDK HttpClient는 풀·타임아웃 API가 Netty보다 단순하고, HTTP/2와 WebSocket을 모두 지원한다. 다만 백프레셔 통합이 Reactor Netty만큼 깊지 않아, 스트리밍 요구가 강한 워크로드에서는 Netty 쪽이 안정적이다.



## 타임아웃 — 네 가지를 같이 잡는다

> 타임아웃을 한 군데서만 잡으면 다른 곳에서 새는 일이 흔하다. 네 가지를 함께 명시한다.

WebClient + Reactor Netty 조합에서 타임아웃은 다음 네 단계가 있다.

1. **Connect timeout** — TCP 핸드셰이크가 실패하기까지 기다리는 시간. `HttpClient.option(ChannelOption.CONNECT_TIMEOUT_MILLIS, n)`.
2. **Response timeout** — 요청을 보낸 뒤 응답 첫 바이트가 도착하기까지 기다리는 시간. `HttpClient.responseTimeout(Duration)`.
3. **Read timeout** — 한 번 응답을 받기 시작한 뒤 다음 청크가 오기까지 기다리는 시간. `ReadTimeoutHandler`.
4. **전체 호출 타임아웃** — 위 셋과 별개로, 전체 파이프라인에 거는 deadline. `Mono.timeout(Duration)`.

```java
HttpClient httpClient = HttpClient.create()
        .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 3000)
        .responseTimeout(Duration.ofSeconds(10))
        .doOnConnected(conn -> conn
                .addHandlerLast(new ReadTimeoutHandler(15, TimeUnit.SECONDS))
                .addHandlerLast(new WriteTimeoutHandler(15, TimeUnit.SECONDS)));

WebClient client = WebClient.builder()
        .clientConnector(new ReactorClientHttpConnector(httpClient))
        .build();

Mono<User> userMono = client.get()
        .uri("/users/{id}", 42L)
        .retrieve()
        .bodyToMono(User.class)
        .timeout(Duration.ofSeconds(20));   // 전체 deadline
```

⚠️ 응답 타임아웃과 전체 타임아웃을 둘 다 잡는 이유는 동작 계층이 다르기 때문이다. `responseTimeout`은 Netty 채널 레벨에서 동작해 정확하지만, Reactor 파이프라인 안의 `flatMap`/`retry` 같은 후처리에는 반영되지 않는다. `Mono.timeout`은 파이프라인 전체에 deadline을 걸지만 채널을 즉시 닫지는 않는다. 둘을 같이 두면 Netty가 먼저 끊고 Reactor가 cleanup하는 흐름이 안정적이다.



## 커넥션 풀 — `ConnectionProvider`

> 동시 호출이 많은 워크로드에서 풀이 작으면 큐잉 타임아웃이 떨어진다. 풀 크기·idle 타임·pending 큐를 같이 본다.

Reactor Netty는 기본적으로 도메인별로 커넥션 풀을 유지한다. `HttpClient.create()`는 글로벌 풀을 공유한다. 호출 워크로드가 많은 서비스라면 명시적으로 풀을 잡는다.

```java
ConnectionProvider provider = ConnectionProvider.builder("api-pool")
        .maxConnections(200)                           // 동시 활성 커넥션 상한
        .pendingAcquireMaxCount(500)                   // 풀이 다 찼을 때 대기 큐 크기
        .pendingAcquireTimeout(Duration.ofSeconds(5))  // 대기 큐에서 기다리는 최대 시간
        .maxIdleTime(Duration.ofSeconds(30))           // idle 커넥션 정리 주기
        .maxLifeTime(Duration.ofMinutes(10))           // 커넥션 최대 수명
        .build();

HttpClient httpClient = HttpClient.create(provider);
```

값 결정 가이드를 한 줄로 줄이면 다음과 같다.

- `maxConnections`: 한 인스턴스가 동시 진행 중인 호출 수의 상한. 일반적으로 100~500.
- `pendingAcquireTimeout`: 풀이 다 찼을 때 호출이 얼마나 기다릴지. 응답 타임아웃과 같거나 짧게.
- `maxIdleTime`: 외부 API의 keep-alive보다 짧게. 그쪽이 먼저 닫으면 `PrematureCloseException`이 뜬다.

⚠️ AWS ALB·NGINX 같은 중간 프록시는 idle 60초가 흔하다. 클라이언트 `maxIdleTime`을 그보다 길게 잡으면 죽은 커넥션을 잡고 첫 요청이 실패한다. 보통 30초 또는 그보다 짧게 둔다.



## 실무 시나리오 — 외부 API 두 곳, 한 앱

> 실제 코드에서 위 옵션을 어떻게 조합하는지 한 사례로 묶는다.

쇼핑몰 백엔드가 결제 게이트웨이와 배송 추적 API 두 곳을 부른다고 하자. 결제는 응답 5초 안에 와야 하고, 배송 추적은 30초까지 기다려도 된다. JSON 본문 크기는 결제가 작고(<10KB) 배송 추적은 첨부 라벨 이미지를 base64로 받아 5MB까지 간다.

```java
@Configuration
class ApiClientConfig {

    @Bean
    WebClient paymentClient(WebClient.Builder builder) {
        HttpClient httpClient = HttpClient.create(payPool())
                .responseTimeout(Duration.ofSeconds(5));
        return builder.clone()
                .baseUrl("https://pay.example.com")
                .clientConnector(new ReactorClientHttpConnector(httpClient))
                .build();
    }

    @Bean
    WebClient trackingClient(WebClient.Builder builder) {
        HttpClient httpClient = HttpClient.create(trackPool())
                .responseTimeout(Duration.ofSeconds(30));
        return builder.clone()
                .baseUrl("https://track.example.com")
                .clientConnector(new ReactorClientHttpConnector(httpClient))
                .codecs(c -> c.defaultCodecs().maxInMemorySize(8 * 1024 * 1024))
                .build();
    }

    private ConnectionProvider payPool() {
        return ConnectionProvider.builder("pay").maxConnections(50)
                .pendingAcquireTimeout(Duration.ofSeconds(2)).build();
    }

    private ConnectionProvider trackPool() {
        return ConnectionProvider.builder("track").maxConnections(20)
                .pendingAcquireTimeout(Duration.ofSeconds(5)).build();
    }
}
```

핵심은 두 가지다. 외부 API마다 빈을 분리해 타임아웃·풀·코덱을 독립적으로 잡는다. 그리고 `Builder.clone()`을 호출해 글로벌 빌더의 기본값(필터 등)을 상속하면서 외부 API별 추가 설정을 얹는다. `clone()` 없이 같은 `Builder`를 두 빈에서 쓰면 한쪽 설정이 다른 쪽에 새는 사고가 흔하다.



## 함정 — 자주 만나는 세 가지

> 처음 만나면 디버깅에 시간이 가는 항목들이다.

### 1. `DataBufferLimitException`이 떨어지는데 한도를 키워도 같다

응답 코덱이 아니라 요청 코덱 한도가 작아 발생하는 경우가 있다. multipart 요청이 클 때다. `defaultCodecs().maxInMemorySize`는 양쪽에 동시 적용되므로 둘 다 키워진다. 다만 multipart 본문은 `multipartCodecs()`도 별도로 한도를 걸 수 있다.

### 2. `PrematureCloseException`이 간헐적으로 뜬다

서버 또는 중간 프록시의 idle 타임아웃이 클라이언트 풀의 `maxIdleTime`보다 짧을 때 흔하다. 풀에서 꺼낸 커넥션이 이미 닫혀 있어 첫 요청이 실패한다. `maxIdleTime`을 30초 또는 더 짧게, 또는 풀에 `evictInBackground`를 추가해 주기적으로 정리한다.

### 3. Spring Boot가 만들어 준 `Builder`가 기대와 다르게 동작한다

Spring Boot는 `WebClient.Builder`에 `WebClientCustomizer` 빈들을 자동 적용한다. Tracing, Metrics 같은 것이 자동으로 들어가 있을 수 있다. 디버깅이 막히면 `WebClientCustomizer`를 어떤 것이 등록돼 있는지부터 본다.



## 면접에서 받을 만한 질문

> 챕터 마무리 점검.

1. `WebClient.create()`와 `WebClient.builder()` 중 무엇을 골라야 하는가?
   - 답 요지: 옵션을 하나라도 손댄다면 `builder()`. `create()`는 옵션 추가가 불가능하다.
2. `WebClient`를 빈으로 둘 때와 `WebClient.Builder`를 빈으로 둘 때의 트레이드오프는?
   - 답 요지: 완성된 WebClient를 빈으로 두면 빌드 비용을 한 번만 낸다. 호출마다 옵션이 다른 워크로드(예: TPS 어댑터)는 빌더 빈이 자연스럽다.
3. `maxInMemorySize`의 기본값과 그것을 키울 때의 위험은?
   - 답 요지: 256KB. 무한 키우기는 GC 압박과 OOM. 정말 큰 본문은 청크 처리(`bodyToFlux(DataBuffer)`)로 검토.
4. `responseTimeout`과 `Mono.timeout`을 둘 다 두는 이유는?
   - 답 요지: 동작 계층이 다르다. responseTimeout은 Netty 채널, Mono.timeout은 Reactor 파이프라인. 둘이 함께 있어야 채널 close와 파이프라인 cleanup이 모두 일어난다.
5. 외부 API의 idle 타임아웃과 클라이언트 풀의 `maxIdleTime` 관계는?
   - 답 요지: 클라이언트 쪽이 짧아야 한다. 그렇지 않으면 서버가 먼저 닫은 커넥션을 풀에서 꺼내 `PrematureCloseException`이 뜬다.
