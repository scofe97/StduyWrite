# 에러 처리와 재시도

---

> WebClient의 에러 처리는 두 계층으로 갈린다. HTTP 코드 기반 처리(`onStatus`/`WebClientResponseException`)와 Reactor 연산자 기반 처리(`onErrorResume`/`onErrorMap`/`retryWhen`). 본 챕터는 두 계층이 어떻게 맞물리는지, 그리고 응답 200 OK 안에 비즈니스 실패가 박혀 있는 API를 어떻게 다루는지를 같이 본다.



## 두 계층의 분업

> HTTP 레벨 실패는 WebClient가 자동으로 예외로 변환한다. 그 예외를 도메인 예외로 다시 매핑하는 책임은 호출 코드에 있다.

`retrieve()`가 자동으로 하는 일을 먼저 본다.

1. 응답 상태 코드가 4xx면 `WebClientResponseException` 하위 예외를 생성.
2. 5xx도 같은 흐름으로 예외 생성.
3. `Mono` 파이프라인에 `onError` 신호로 전파.

호출 코드는 다음 단계로 이어 간다.

1. **`onStatus`** — `retrieve()` 직후. HTTP 코드별로 다른 예외 또는 빈 결과로 매핑.
2. **`onErrorResume` / `onErrorMap`** — Reactor 연산자. 예외가 떴을 때 다른 흐름으로 분기 또는 다른 예외로 변환.
3. **`retryWhen`** — 재시도. 일시적 실패에 한해 자동 복구.



## `WebClientResponseException` 계층

> 4xx/5xx 코드별로 자식 예외 클래스가 있다.

```
WebClientException (추상)
└── WebClientResponseException
    ├── BadRequest                  (400)
    ├── Unauthorized                (401)
    ├── Forbidden                   (403)
    ├── NotFound                    (404)
    ├── MethodNotAllowed            (405)
    ├── NotAcceptable               (406)
    ├── Conflict                    (409)
    ├── Gone                        (410)
    ├── UnsupportedMediaType        (415)
    ├── UnprocessableEntity         (422)
    ├── TooManyRequests             (429)
    ├── InternalServerError         (500)
    ├── NotImplemented              (501)
    ├── BadGateway                  (502)
    ├── ServiceUnavailable          (503)
    └── GatewayTimeout              (504)
```

흔히 쓰는 패턴은 `instanceof`로 분기하거나 `getStatusCode().value()`로 코드 직접 비교다.

```java
client.get().uri("/users/{id}", 42L)
        .retrieve()
        .bodyToMono(User.class)
        .onErrorResume(WebClientResponseException.NotFound.class, ex -> Mono.empty())
        .onErrorResume(WebClientResponseException.Unauthorized.class, ex -> {
            tokenRefresher.refresh();
            return Mono.error(new ReauthRequiredException());
        });
```

위 형태는 응답 본문을 못 읽고 끝나는 경우가 흔하다. `WebClientResponseException`은 `getResponseBodyAsString()`로 본문 미리보기를 제공하지만, 디코딩에 실패한 경우 빈 문자열이다. 본문이 필요하다면 `onStatus`에서 `bodyToMono`로 직접 받는다.



## `onStatus` — 가장 자주 쓰는 자리

> `retrieve()` 직후에 끼워 HTTP 코드별로 다른 예외 또는 본문 처리를 한다.

```java
Mono<User> user = client.get().uri("/users/{id}", 42L)
        .retrieve()
        .onStatus(HttpStatusCode::is4xxClientError, response -> response
                .bodyToMono(ApiError.class)
                .flatMap(err -> Mono.error(new ClientApiException(err.code(), err.message()))))
        .onStatus(HttpStatusCode::is5xxServerError, response -> response
                .bodyToMono(ApiError.class)
                .defaultIfEmpty(new ApiError("UNKNOWN", "no body"))
                .flatMap(err -> Mono.error(new UpstreamUnavailableException(err.message()))))
        .bodyToMono(User.class);
```

`onStatus`는 두 가지 인자를 받는다.

- `Predicate<HttpStatusCode>` — 어느 코드에 반응할지.
- `Function<ClientResponse, Mono<? extends Throwable>>` — 그 코드일 때 어떤 예외를 만들지.

응답 본문을 읽고 그 안의 에러 메시지를 도메인 예외에 담아 던지는 패턴이 가장 흔하다. `onStatus` 안에서 `response.bodyToMono(...)`를 호출하지 않으면 본문 버퍼가 자동 해제된다(WebClient가 처리). 호출하면 디코딩한 뒤 해제된다.

⚠️ 4xx와 5xx에 대해 `onStatus`를 제공하지 않으면 `retrieve()`는 기본 `WebClientResponseException`을 그대로 던진다. 본문 처리가 필요 없다면 그게 가장 깔끔한 형태다.



## `onErrorResume` / `onErrorMap` — Reactor 연산자

> 파이프라인에 흐른 예외를 다시 매핑하거나 폴백 흐름으로 분기한다.

`onErrorResume`은 예외를 받아 `Mono`/`Flux`로 변환한다. 폴백 응답을 만들거나 빈 결과로 끝낼 때 쓴다.

```java
client.get().uri("/users/{id}", 42L).retrieve()
        .bodyToMono(User.class)
        .onErrorResume(WebClientResponseException.NotFound.class, ex -> Mono.just(User.empty()))
        .onErrorResume(IOException.class, ex -> Mono.just(User.fallback()));
```

`onErrorMap`은 예외를 다른 예외로 변환만 한다. 도메인 경계에서 인프라 예외를 도메인 예외로 wrap하는 자리에 자주 쓴다.

```java
client.get().uri("/users/{id}", 42L).retrieve()
        .bodyToMono(User.class)
        .onErrorMap(WebClientResponseException.class,
                ex -> new UserLookupFailedException("외부 사용자 조회 실패", ex));
```

두 연산자의 차이는 다음 한 줄로 줄인다. 결과 흐름을 만들면 `onErrorResume`, 예외 타입만 갈아끼우면 `onErrorMap`.



## TPS 어댑터의 에러 처리 — 200 OK 안의 비즈니스 실패

> 응답 코드는 성공인데 본문이 실패를 알리는 API가 흔하다. WebClient의 자동 변환은 도와주지 못한다.

TPS 어댑터는 외부 API가 항상 HTTP 200으로 응답하면서, 본문 안의 `rsltCd` 필드로 성공·실패를 구분하는 패턴을 처리한다.

```java
// ApprovalUrlAdapter.java:175-190
TpsResponse<?> response = request.retrieve().bodyToMono(TpsResponse.class).block();

if (response == null) {
    log.error("[ApprovalUrlAdapter] API call to {} returned null response.", item.apiUrl());
    throw new TpsException(ErrorCodeGeneral.INTERNAL_ERROR, "[Approval End] API 응답이 null입니다.",
            "Downstream API call returned null.");
}
log.info("[ApprovalUrlAdapter] Response Body: {}", response);

String rsltCd = response.getRsltCd();
if (!"TPS200".equals(rsltCd)) {
    String rsltMsg = response.getRsltMsg();
    log.error("[ApprovalUrlAdapter] API call to {} failed. rsltCd: {}, rsltMsg: {}",
            item.apiUrl(), rsltCd, rsltMsg);
    throw new TpsException(ErrorCodeGeneral.INTERNAL_ERROR, rsltMsg, "[Approval End] API 요청 실패");
}
```

세 단계가 보인다.

1. **`retrieve` + `bodyToMono` + `block`** — HTTP 레벨 실패는 자동으로 `WebClientResponseException`을 던진다(이 코드 위에서 잡지 않음, 외층에서 처리).
2. **`null` 검증** — `block()`이 `null`을 반환하는 경우(빈 본문) 별도 예외.
3. **`rsltCd` 검증** — HTTP 200이지만 비즈니스 실패면 `TpsException`을 던진다.

이 코드를 도메인 경계에서 다시 한 번 wrap한다.

```java
// ApprovalUrlAdapter.java:54-65
public void invokeForApprovalEnd(ApprovalEndDto aprvEndDto) {
    try {
        invokeInternal(aprvEndDto);
    } catch (ApprovalUrlInvocationException e) {
        throw e;
    } catch (Exception e) {
        throw new ApprovalUrlInvocationException(
                aprvEndDto.atrzExcnId(),
                "결재 완료 후 외부 API 호출 실패: " + e.getMessage(),
                e);
    }
}
```

`Exception` 전부를 `ApprovalUrlInvocationException`으로 변환한다. WebClient의 `WebClientResponseException`도, 위에서 던진 `TpsException`도, JSON 파싱 실패도 같은 도메인 예외로 통일된다.

`ApprovalUrlInvocationException` 자체는 다음과 같다.

```java
// ApprovalUrlInvocationException.java
public class ApprovalUrlInvocationException extends TpsException {
    private final String atrzExcnId;

    public ApprovalUrlInvocationException(String atrzExcnId, String detailMsg, Throwable cause) {
        super(ErrorCodeApproval.APRV_URL_INVOKE_FAIL,
                ErrorCodeApproval.APRV_URL_INVOKE_FAIL.getRsltMsg(), detailMsg);
        initCause(cause);
        this.atrzExcnId = atrzExcnId;
    }
}
```

도메인 키(`atrzExcnId`)를 예외에 박아 후처리(상태 갱신·실패 알림)에서 어느 결재 건이 실패했는지 알 수 있게 한다. 인프라 예외(WebClient·IO)를 도메인 컨텍스트와 함께 묶는 패턴이다.

이 흐름을 더 Reactor답게 짠다면 다음 형태가 된다.

```java
return request.retrieve()
        .bodyToMono(TpsResponse.class)
        .switchIfEmpty(Mono.error(new IllegalStateException("API 응답이 null")))
        .flatMap(resp -> {
            if (!"TPS200".equals(resp.getRsltCd())) {
                return Mono.error(new TpsException(...));
            }
            return Mono.just(resp);
        })
        .onErrorMap(ex -> new ApprovalUrlInvocationException(atrzExcnId, ex.getMessage(), ex));
```

`block()`을 쓰는 동기 코드와 위 비동기 형태의 동작은 같다. 어느 쪽을 고를지는 02-02에서 다룬다.



## 재시도 — `Retry.backoff`

> 일시적 실패는 자동으로 다시 시도한다. 단, 안전한 호출에 한해서.

Reactor `Retry`는 다양한 변형을 제공한다. 가장 자주 쓰는 형태는 `Retry.backoff(maxAttempts, minBackoff)`다.

```java
import reactor.util.retry.Retry;

client.get().uri("/users/{id}", 42L)
        .retrieve()
        .bodyToMono(User.class)
        .retryWhen(Retry.backoff(3, Duration.ofMillis(200))
                .maxBackoff(Duration.ofSeconds(2))
                .jitter(0.5)
                .filter(ex -> ex instanceof WebClientRequestException
                        || (ex instanceof WebClientResponseException w
                            && w.getStatusCode().is5xxServerError()))
                .onRetryExhaustedThrow((spec, sig) -> sig.failure()));
```

설명을 풀면 다음과 같다.

| 옵션 | 동작 |
|------|------|
| `backoff(3, 200ms)` | 최대 3회 시도, 첫 backoff 200ms |
| `maxBackoff(2s)` | backoff가 2초를 넘지 않음 |
| `jitter(0.5)` | backoff에 ±50% 랜덤 노이즈 — thundering herd 방지 |
| `filter(...)` | 특정 예외만 재시도. 4xx는 일반적으로 재시도 의미 없음 |
| `onRetryExhaustedThrow(...)` | 모든 시도 실패 시 던질 예외 결정. 기본은 `RetryExhaustedException`로 wrap |

기본 `onRetryExhaustedThrow`를 안 잡으면 원본 예외가 `RetryExhaustedException`으로 감싸인다. 도메인에서 원본 예외 그대로 받고 싶다면 위처럼 `sig.failure()`를 던진다.

### `transientErrors` — 같은 실패가 연속이면 위험

`Retry.backoff(...).transientErrors(true)`는 "성공적인 재시도 사이의 실패는 카운트를 리셋"하는 옵션이다. 한 호출이 여러 번 응답을 내는 스트리밍 시나리오에서 의미가 있다. 단일 응답 호출에는 보통 필요 없다.

### POST 재시도는 안전한가

GET은 멱등이므로 자동 재시도가 안전하다. POST는 본질적으로 멱등이 아니다. 같은 호출이 두 번 적용되면 결과가 두 번 만들어질 수 있다.

POST 재시도를 안전하게 하려면 두 가지 중 하나가 필요하다.

1. **서버가 멱등성 키(Idempotency-Key)를 지원**한다. 같은 키로 재시도하면 한 번만 처리된다는 보장.
2. **클라이언트가 응답 도착 여부를 명확히 알 수 있다**. 네트워크 단절(`WebClientRequestException`)은 응답이 도착했는지 모르는 상태이므로 재시도 위험. 응답 도착 후 5xx(`WebClientResponseException`)는 서버가 거부한 것이 명확하므로 재시도 가능.

위 둘 다 만족 못 하면 POST 재시도는 멈추고 호출자에 실패를 알리는 편이 안전하다.



## 함정 — 자주 만나는 두 가지

### 1. `onStatus`로 본문을 읽었는데 다시 `bodyToMono`에서 또 읽으려 한다

응답 본문은 한 번만 읽을 수 있다. `onStatus` 안에서 `bodyToMono(ApiError.class)`로 읽고 예외로 변환했다면, 이후 `bodyToMono(User.class)`는 도달하지 않는다(예외가 던져졌으므로). 이 흐름은 자연스럽지만, 디버깅 중 `bodyToMono`를 두 번 호출하도록 코드를 잘못 짜면 두 번째 호출에서 `IllegalStateException: Only one connection receive subscriber allowed`가 떨어진다.

### 2. `onStatus`에서 비동기 본문 디코딩을 동기 흐름에 섞는다

`response.bodyToMono(ApiError.class).block()`을 `onStatus` 안에서 호출하면 EventLoop 데드락이 일어난다. `onStatus`의 람다는 이미 비동기 컨텍스트에서 실행되므로 `Mono`로 그대로 반환해야 한다.

```java
// ❌ 데드락 위험
.onStatus(HttpStatusCode::is4xxClientError, response -> {
    ApiError err = response.bodyToMono(ApiError.class).block();
    return Mono.error(new ClientApiException(err.code(), err.message()));
})

// ✅ 비동기 그대로
.onStatus(HttpStatusCode::is4xxClientError, response -> response
        .bodyToMono(ApiError.class)
        .flatMap(err -> Mono.error(new ClientApiException(err.code(), err.message()))))
```



## 면접에서 받을 만한 질문

> 챕터 마무리 점검.

1. `retrieve()`가 4xx/5xx를 자동으로 어떻게 변환하는가?
   - 답 요지: `WebClientResponseException` 하위 예외를 생성해 `Mono` 파이프라인에 `onError` 신호로 전파한다.
2. `onStatus`와 `onErrorResume`의 차이는?
   - 답 요지: 전자는 HTTP 코드를 본 뒤 본문을 디코딩해 도메인 예외로 만드는 자리, 후자는 이미 던져진 예외를 폴백 흐름으로 변환하는 자리. 책임 계층이 다르다.
3. 응답이 200 OK인데 비즈니스 실패를 본문으로 알리는 API는 어떻게 처리하는가?
   - 답 요지: WebClient 자동 변환이 적용되지 않으므로 `bodyToMono` 후 `flatMap`에서 본문 필드를 검증하고 명시적으로 예외를 던진다. TPS 어댑터의 `rsltCd` 검증이 그 패턴.
4. POST에 자동 재시도를 거는 것이 안전한 조건은?
   - 답 요지: 서버가 멱등성 키를 지원하거나, 응답 도착이 확정된 상태(5xx)에서만. 네트워크 단절(`WebClientRequestException`)은 응답이 갔는지 모르므로 재시도 위험.
5. `Retry.backoff`에 `jitter`를 거는 이유는?
   - 답 요지: 같은 시점에 다 같이 실패한 클라이언트가 동시에 재시도해 서버를 다시 무너뜨리는 thundering herd를 막기 위해 backoff에 랜덤성을 더한다.
