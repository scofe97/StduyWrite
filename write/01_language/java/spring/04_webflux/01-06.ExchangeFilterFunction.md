# ExchangeFilterFunction (로깅·인증·MDC)

---

> 모든 호출에 같은 처리(로깅·인증 헤더·MDC 전파·메트릭)를 거는 횡단 관심사를 한 자리에 모은 자리다. 본 챕터는 필터 인터페이스 자체부터, 본문 로깅의 함정, OAuth2 통합 개요까지 다룬다.



## 인터페이스

> 필터는 `ClientRequest`를 받아 `Mono<ClientResponse>`를 반환하는 함수다.

```java
@FunctionalInterface
public interface ExchangeFilterFunction {
    Mono<ClientResponse> filter(ClientRequest request, ExchangeFunction next);
}
```

`next.exchange(request)`를 호출해 다음 필터 또는 실제 호출로 넘긴다. 호출 전에 요청을 가공하거나, 호출 후에 응답을 가공할 수 있다. 여러 필터는 등록 순서대로 체이닝된다.

```java
WebClient client = WebClient.builder()
        .baseUrl("https://api.example.com")
        .filter(authFilter())
        .filter(loggingFilter())
        .filter(metricsFilter())
        .build();
```

위 코드는 매 요청마다 `authFilter` → `loggingFilter` → `metricsFilter` → 실제 호출 순서로 실행된다. 응답은 역순으로 거슬러 올라간다.



## 헬퍼 — `ofRequestProcessor` / `ofResponseProcessor`

> 요청만 가공하거나 응답만 가공한다면 헬퍼 메서드가 짧게 끝낸다.

`ExchangeFilterFunction` 정적 메서드 두 가지가 자주 쓰인다.

### 1) 요청 가공만

```java
ExchangeFilterFunction addTraceHeader = ExchangeFilterFunction.ofRequestProcessor(req -> {
    String traceId = MDC.get("traceId");
    return Mono.just(ClientRequest.from(req)
            .header("X-Trace-Id", traceId)
            .build());
});
```

`ClientRequest`는 불변이므로 `from(req).build()` 형태로 새 객체를 만든다. 헤더 추가, URL 변경, attribute 부착 같은 일이 가능하다.

### 2) 응답 가공만

```java
ExchangeFilterFunction logResponseStatus = ExchangeFilterFunction.ofResponseProcessor(resp -> {
    log.info("status={}", resp.statusCode());
    return Mono.just(resp);
});
```

응답 헤더와 코드는 그대로 보고 본문은 건드리지 않는다. 본문을 가공하려면 직접 인터페이스를 구현하면서 `bodyToMono`/`bodyToFlux`를 호출해야 하는데, 본문은 한 번만 읽을 수 있어 함정이 박혀 있다(아래 절 참조).



## 로깅 필터의 함정

> "요청과 응답 본문을 로그에 남기고 싶다"는 요구가 흔하다. 단순히 본문을 읽으면 두 번째 디코딩이 실패한다.

본문은 한 번만 읽을 수 있다. 필터 안에서 `response.bodyToMono(String.class)`로 로깅한 뒤, 호출 코드에서 `bodyToMono(User.class)`를 다시 호출하면 두 번째에서 `IllegalStateException`이 떨어진다.

해결은 두 가지다.

### 1) 메타데이터만 로깅

```java
ExchangeFilterFunction logRequest = ExchangeFilterFunction.ofRequestProcessor(req -> {
    log.info(">> {} {} headers={}", req.method(), req.url(), req.headers());
    return Mono.just(req);
});

ExchangeFilterFunction logResponse = ExchangeFilterFunction.ofResponseProcessor(resp -> {
    log.info("<< status={} headers={}", resp.statusCode(), resp.headers().asHttpHeaders());
    return Mono.just(resp);
});
```

본문을 건드리지 않는다. 디버깅이 빠른 편은 아니지만 안전하다.

### 2) 본문 캐시 — `cache()` 또는 `release` 후 새로 만들기

본문을 로깅하면서 호출 코드도 디코딩할 수 있게 하려면 `Mono`를 캐시하거나 본문을 미리 받아 다시 응답을 만든다. Spring Boot가 제공하는 `WebClient` Actuator 메트릭은 후자를 쓴다.

```java
// 본문 캐시 — 한 번만 디코딩되고 두 번째 호출은 캐시에서 반환
ExchangeFilterFunction logBody = (request, next) -> next.exchange(request)
        .flatMap(resp -> resp.bodyToMono(String.class)
                .defaultIfEmpty("")
                .doOnNext(body -> log.info("<< body={}", body))
                .map(body -> resp.mutate()
                        .body(Flux.just(resp.bufferFactory().wrap(body.getBytes())))
                        .build()));
```

위 형태는 본문을 일단 String으로 받아 로깅하고, 같은 본문을 다시 응답에 채워 다음 단계로 넘긴다. 코드 부피가 작지 않고 본문이 큰 응답에서는 두 번 메모리에 올라간다.

⚠️ 운영 환경의 일반 패턴은 본문 로깅을 끄거나 샘플링한다. 본문 안에 민감정보(토큰·개인정보)가 들어 있을 위험과 비용이 합쳐지면 디폴트로 켜기에 무리가 있다.



## 인증 헤더 자동 첨부

> 매 요청에 Bearer 토큰을 박는 흔한 패턴이다.

### 1) 단순 정적 토큰

```java
ExchangeFilterFunction bearer = ExchangeFilterFunction.ofRequestProcessor(req ->
        Mono.just(ClientRequest.from(req)
                .header("Authorization", "Bearer " + staticToken)
                .build()));
```

토큰이 변하지 않는다면 `defaultHeader`로도 충분하다. 필터를 쓰는 이유는 토큰이 동적으로 갱신되거나(refresh) 호출별로 다를 때다.

### 2) Basic 인증 — `ExchangeFilterFunctions.basicAuthentication`

```java
import static org.springframework.web.reactive.function.client.ExchangeFilterFunctions.basicAuthentication;

WebClient client = WebClient.builder()
        .filter(basicAuthentication("user", "password"))
        .build();
```

내장 헬퍼다. 호출별로 다른 자격이 필요하면 호출 시점에 `attribute`로 넘기는 패턴이 있다.

```java
client.get().uri("/users")
        .attribute(BASIC_AUTHENTICATION_CREDENTIALS_ATTRIBUTE_NAME,
                new BasicAuthenticationCredentials("user", "password"))
        .retrieve();
```

### 3) OAuth2 — `ServletOAuth2AuthorizedClientExchangeFilterFunction`

Spring Security OAuth2 Client와 통합하면 토큰 발급·갱신을 자동으로 한다.

```java
@Bean
WebClient apiClient(OAuth2AuthorizedClientManager manager) {
    ServletOAuth2AuthorizedClientExchangeFilterFunction oauth2 =
            new ServletOAuth2AuthorizedClientExchangeFilterFunction(manager);
    oauth2.setDefaultClientRegistrationId("api-service");

    return WebClient.builder()
            .baseUrl("https://api.example.com")
            .filter(oauth2)
            .build();
}
```

`api-service`는 `application.yml`의 `spring.security.oauth2.client.registration.api-service`를 가리킨다. 이 필터는 토큰 만료를 감지해 자동으로 갱신하고 새 토큰을 헤더에 박는다. WebFlux 환경이면 `ServerOAuth2AuthorizedClientExchangeFilterFunction`을 쓴다.



## MDC 전파

> 비동기 호출 흐름에서 MDC가 넘어가지 않는다. Reactor `Context`로 따로 옮긴다.

MDC(Mapped Diagnostic Context)는 SLF4J가 스레드 로컬에 두는 키-값 저장소다. 한 요청을 처리하는 동안 `traceId` 같은 값을 박아 두면 모든 로그에 자동으로 따라붙는다. 그러나 Reactor는 호출 스레드와 응답 처리 스레드가 다를 수 있어 MDC가 그대로 따라가지 않는다.

해결은 두 가지 패턴 중 하나다.

### 1) 호출 직전에 헤더로 옮긴다

```java
ExchangeFilterFunction propagateTrace = ExchangeFilterFunction.ofRequestProcessor(req -> {
    String traceId = MDC.get("traceId");
    if (traceId == null) return Mono.just(req);
    return Mono.just(ClientRequest.from(req)
            .header("X-Trace-Id", traceId)
            .build());
});
```

가장 단순하다. MDC 값을 외부 호출 헤더에 옮기는 것이 목표일 때 충분하다.

### 2) Reactor `Context`를 통해 전파

응답 처리 단계에서도 MDC가 필요하다면(예: 응답 본문을 받은 뒤 로깅) Reactor `Context`로 옮긴다.

```java
client.get().uri("/users")
        .retrieve()
        .bodyToMono(User.class)
        .doOnNext(user -> {
            // 이 시점은 응답 처리 스레드. MDC가 비어 있다
            log.info("user={}", user);
        })
        .contextWrite(Context.of("traceId", MDC.get("traceId")));
```

`contextWrite`는 Reactor `Context`에 값을 채운다. `doOnEach` 또는 커스텀 hook으로 `Context`에서 꺼내 MDC에 다시 채우는 형태가 흔하다. Spring Cloud Sleuth/Micrometer Tracing은 이 hook을 자동으로 등록해 준다.



## 메트릭 — Micrometer 통합

> 호출 응답시간·횟수·실패율을 자동으로 메트릭으로 노출한다.

Spring Boot Actuator + Micrometer가 있으면 `WebClient.Builder`에 `MetricsWebClientFilterFunction`이 자동 등록된다. `http.client.requests` 메트릭이 다음 태그로 분류된다.

| 태그 | 의미 |
|------|------|
| `method` | GET/POST/... |
| `uri` | URI 템플릿 (`/users/{id}`) |
| `status` | 상태 코드 |
| `outcome` | SUCCESS/CLIENT_ERROR/SERVER_ERROR |
| `clientName` | 호스트 |

URI 템플릿이 태그에 들어가는 점이 중요하다. `uri("/users/42")` 대신 `uri("/users/{id}", 42)` 형태로 호출해야 같은 엔드포인트가 한 메트릭으로 묶인다. ID마다 태그가 갈리면 메트릭 cardinality가 폭발한다.



## TPS 어댑터 — 필터 미사용 비교

> TPS는 필터를 쓰지 않고 호출부에서 직접 로깅한다. 비교를 통해 필터의 가치를 본다.

```java
// ApprovalUrlAdapter.java:170-173
log.info("[ApprovalUrlAdapter] Request URL: {}", item.apiUrl());
log.info("[ApprovalUrlAdapter] Request Headers: {}", headerMap);
log.info("[ApprovalUrlAdapter] Request Method: {}", item.httpMethod());
log.info("[ApprovalUrlAdapter] Request Body: {}", item.dmndInfo());
```

호출 직전에 메서드·URL·헤더·본문을 그대로 INFO로 남긴다. 응답도 line 182에서 `Response Body`를 INFO로 남긴다. 필터를 쓰지 않은 단순한 형태다.

이 코드를 필터로 옮기면 어떤 변화가 있을까.

### 필터로 옮길 때 얻는 것

1. **재사용**. 다른 어댑터에서도 같은 필터를 등록만 하면 된다.
2. **응답 처리와의 일관성**. 호출 직전과 직후 로깅이 한 자리에 모인다. 호출자가 빠뜨릴 위험이 줄어든다.
3. **민감정보 마스킹 일원화**. Authorization 헤더 마스킹 같은 처리를 한 자리에서 적용 가능.

### 필터로 옮길 때 잃는 것

1. **컨텍스트 손실**. TPS는 `[ApprovalUrlAdapter]` 같은 도메인 레이블을 직접 박아 어느 어댑터에서 호출됐는지 명확히 한다. 공용 필터는 호출 컨텍스트를 모르므로 같은 정보를 만들기 어렵다(`attribute`로 넘길 수는 있다).
2. **분기 처리 단순함이 사라진다**. 호출부에서 직접 로깅하면 호출 안 하고 싶은 케이스(특정 URL은 로그 빼기)를 if 한 줄로 처리하지만, 필터에서는 `attribute`나 URL 패턴 매칭이 필요해진다.

TPS 어댑터의 경우 호출자가 한 곳뿐이고 도메인 레이블이 중요하므로 호출부 직접 로깅이 합리적이다. 호출자가 늘어나면 필터로 옮길 만하다.



## 함정 — 자주 만나는 두 가지

### 1. 필터 등록 순서가 동작에 영향을 준다

`auth → log` 순서라면 로깅 필터가 인증이 적용된 헤더를 본다. `log → auth`라면 인증 전 헤더를 본다. 토큰을 로그에 남기지 않으려면 `auth`를 먼저, 로그를 나중에 등록하지 말아야 한다.

### 2. 필터 안에서 `block()`을 호출

필터 람다는 비동기 컨텍스트에서 실행된다. `block()`을 호출하면 EventLoop 데드락이다. 필터 안에서 외부 자원이 필요하다면 `Mono`/`Flux`로 가져와 합성한다.



## 면접에서 받을 만한 질문

> 챕터 마무리 점검.

1. `ExchangeFilterFunction`의 인터페이스 시그니처를 한 줄로 설명할 수 있는가?
   - 답 요지: `(ClientRequest, ExchangeFunction next) -> Mono<ClientResponse>`. 다음 필터로 넘기기 전후에 요청·응답을 가공한다.
2. 필터 안에서 응답 본문을 로깅한 뒤 호출 코드도 본문을 디코딩하려고 하면 무엇이 깨지는가?
   - 답 요지: 본문은 한 번만 읽을 수 있다. 두 번째 호출에서 `IllegalStateException`이 떨어진다. 본문을 미리 받아 새 응답으로 다시 만들거나, 메타데이터만 로깅한다.
3. OAuth2 토큰 자동 갱신은 어떻게 동작하는가?
   - 답 요지: `ServletOAuth2AuthorizedClientExchangeFilterFunction`이 `OAuth2AuthorizedClientManager`를 통해 토큰 만료를 감지하고 갱신 후 새 토큰을 헤더에 박는다.
4. Reactor 비동기 흐름에서 MDC가 사라지는 이유는?
   - 답 요지: MDC는 스레드 로컬이고, Reactor 호출 스레드와 응답 처리 스레드가 다르기 때문이다. `Context`로 옮기거나 호출 직전에 헤더로 박는 패턴이 필요하다.
5. URI 템플릿을 메트릭 태그로 쓰려면 호출 코드를 어떻게 짜야 하는가?
   - 답 요지: `uri("/users/{id}", 42)`처럼 변수 치환 형태로 호출한다. `uri("/users/42")`로 직접 박으면 ID마다 다른 태그로 분류돼 cardinality가 폭발한다.
