# 테스트 (MockWebServer와 WebTestClient)

---

> WebClient 호출 코드를 테스트하는 길은 세 갈래다. 진짜 HTTP 서버를 띄워(MockWebServer) 검증하는 인테그레이션 형태, 가짜 `ExchangeFunction`을 끼워(WebClient.builder() + filter) 단위로 검증하는 형태, 그리고 컨트롤러 위에서 클라이언트로 진입하는 `WebTestClient`. 본 챕터는 셋의 자리를 정리한다.



## 테스트 도구의 분업

> 같은 코드를 어느 도구로 검증하느냐에 따라 잡히는 결함이 다르다.

| 도구 | 무엇을 검증 | 한계 |
|------|------------|------|
| `MockWebServer` | 실제 HTTP 호출이 정확한 메서드·URL·헤더·본문으로 나가는가, 응답 처리가 맞는가 | TCP 포트를 띄우므로 시작 비용이 있고 병렬 실행 시 포트 관리 필요 |
| `ExchangeFilterFunction` 모킹 | Reactor 파이프라인 자체와 본문 변환 로직 | HTTP 자체는 검증 못함, 직렬화·인코딩 결함을 놓침 |
| `WebTestClient` | 컨트롤러 + 핸들러 통합 동작, MockMvc의 WebFlux 짝꿍 | 외부 호출 코드를 직접 검증하지는 않음 |

WebClient를 쓰는 어댑터·서비스 클래스의 외부 호출 정확성은 `MockWebServer`가 가장 정확하다. 본문 가공·예외 변환 같은 호출 코드 내부 로직만 빠르게 본다면 `ExchangeFilterFunction` 모킹이 유리하다.



## MockWebServer — 가장 자주 쓰는 통합 검증

> OkHttp의 `MockWebServer`는 실제 HTTP 서버를 임의 포트에 띄워 응답을 큐로 미리 등록한다. 호출 후 `takeRequest()`로 어떤 요청이 들어왔는지 검증한다.

### 의존성

```gradle
testImplementation 'com.squareup.okhttp3:mockwebserver:4.12.0'
```

### 기본 패턴

```java
import okhttp3.mockwebserver.MockResponse;
import okhttp3.mockwebserver.MockWebServer;
import okhttp3.mockwebserver.RecordedRequest;

class UserClientTest {
    private MockWebServer server;
    private WebClient client;

    @BeforeEach
    void setUp() throws IOException {
        server = new MockWebServer();
        server.start();
        client = WebClient.builder().baseUrl(server.url("/").toString()).build();
    }

    @AfterEach
    void tearDown() throws IOException {
        server.shutdown();
    }

    @Test
    void 사용자_조회_성공() throws Exception {
        server.enqueue(new MockResponse()
                .setResponseCode(200)
                .addHeader("Content-Type", "application/json")
                .setBody("{\"id\":42,\"name\":\"Alice\"}"));

        User user = client.get().uri("/users/{id}", 42L)
                .retrieve()
                .bodyToMono(User.class)
                .block();

        assertThat(user.id()).isEqualTo(42L);
        assertThat(user.name()).isEqualTo("Alice");

        RecordedRequest request = server.takeRequest();
        assertThat(request.getMethod()).isEqualTo("GET");
        assertThat(request.getPath()).isEqualTo("/users/42");
    }
}
```

세 가지 흐름이 보인다.

1. **`server.enqueue(MockResponse)`** — 다음 요청에 응답할 모의 응답을 큐에 등록.
2. **호출 코드 실행** — WebClient가 `MockWebServer`에 실제 TCP 호출.
3. **`server.takeRequest()`** — 들어온 요청을 꺼내 검증.

### 자주 쓰는 변형

#### 상태 코드별 분기 검증

```java
server.enqueue(new MockResponse().setResponseCode(404));

assertThatThrownBy(() -> client.get().uri("/users/{id}", 999L)
        .retrieve()
        .bodyToMono(User.class)
        .block())
        .isInstanceOf(WebClientResponseException.NotFound.class);
```

#### 응답 지연 시뮬레이션 — 타임아웃 검증

```java
server.enqueue(new MockResponse()
        .setResponseCode(200)
        .setBody("{}")
        .setBodyDelay(2, TimeUnit.SECONDS));

assertThatThrownBy(() -> client.get().uri("/users/{id}", 42L)
        .retrieve()
        .bodyToMono(User.class)
        .timeout(Duration.ofMillis(500))
        .block())
        .isInstanceOf(java.util.concurrent.TimeoutException.class);
```

`setBodyDelay`는 본문 시작 전, `setHeadersDelay`는 헤더 자체를 지연시킨다. 응답 타임아웃 분기를 검증할 때 한 줄로 끝난다.

#### multipart 요청 검증

```java
server.enqueue(new MockResponse().setResponseCode(200));

byte[] data = "hello".getBytes();
MultipartBodyBuilder builder = new MultipartBodyBuilder();
builder.part("file", new ByteArrayResource(data) {
    @Override public String getFilename() { return "hello.txt"; }
}).contentType(MediaType.TEXT_PLAIN);

client.post().uri("/upload")
        .body(BodyInserters.fromMultipartData(builder.build()))
        .retrieve()
        .toBodilessEntity()
        .block();

RecordedRequest request = server.takeRequest();
String body = request.getBody().readUtf8();
assertThat(body).contains("filename=\"hello.txt\"");
assertThat(body).contains("Content-Type: text/plain");
```

`request.getBody().readUtf8()`로 raw 본문을 받아 boundary 토큰과 함께 들어간 multipart 페이로드를 직접 검증한다. 02-01에서 다룬 `getFilename()` override 동작이 정상이라면 `filename=` 속성이 채워진 것이 보인다.

#### 여러 요청 시나리오

```java
server.enqueue(new MockResponse().setResponseCode(401));   // 첫 호출 — 토큰 만료
server.enqueue(new MockResponse().setResponseCode(200).setBody("{\"token\":\"new\"}"));  // 토큰 갱신
server.enqueue(new MockResponse().setResponseCode(200).setBody("{}"));   // 재시도

apiClient.callWithAutoRefresh();

assertThat(server.getRequestCount()).isEqualTo(3);
RecordedRequest first = server.takeRequest();
RecordedRequest refresh = server.takeRequest();
RecordedRequest retry = server.takeRequest();
assertThat(retry.getHeader("Authorization")).isEqualTo("Bearer new");
```

`enqueue`는 큐다. 호출 횟수만큼 미리 등록해 둔다. 호출 순서가 시나리오와 다르면 테스트가 실패한다.



## ExchangeFilterFunction 모킹 — 빠른 단위 테스트

> HTTP 자체는 띄우지 않고 `WebClient.Builder.exchangeFunction(...)`으로 가짜 응답을 주입한다.

```java
@Test
void 응답_변환_로직_단위_테스트() {
    ExchangeFunction stub = request -> {
        ClientResponse mock = ClientResponse.create(HttpStatus.OK)
                .header("Content-Type", "application/json")
                .body("{\"id\":42,\"name\":\"Alice\"}")
                .build();
        return Mono.just(mock);
    };

    WebClient client = WebClient.builder().exchangeFunction(stub).build();

    User user = client.get().uri("/users/42")
            .retrieve()
            .bodyToMono(User.class)
            .block();

    assertThat(user.name()).isEqualTo("Alice");
}
```

이 형태는 시작 비용이 거의 없다. 다만 다음 두 가지가 검증되지 않는다.

1. **실제 HTTP 직렬화·인코딩**. JSON 직렬화에 들어가는 Jackson 코덱 동작은 검증되지만, URL 인코딩이나 multipart boundary 같은 transport 레벨 동작은 우회된다.
2. **요청 헤더·본문이 정확히 나가는가**. `ExchangeFunction` 람다에서 `request`를 직접 검증할 수는 있지만, 그 검증이 MockWebServer만큼 자연스럽지 않다.

본문 가공·응답 매핑 같은 내부 로직 단위 테스트에 적합하다. 외부 호출 정확성은 MockWebServer로 별도 보강.



## ExchangeFilterFunction의 단위 테스트

> 필터 자체를 검증하려면 `ExchangeFunction` 모킹이 가장 가볍다.

```java
@Test
void 인증_필터가_헤더를_추가한다() {
    AtomicReference<ClientRequest> captured = new AtomicReference<>();

    ExchangeFunction stub = request -> {
        captured.set(request);
        return Mono.just(ClientResponse.create(HttpStatus.OK).build());
    };

    ExchangeFilterFunction authFilter = ExchangeFilterFunction.ofRequestProcessor(req ->
            Mono.just(ClientRequest.from(req).header("Authorization", "Bearer t").build()));

    WebClient client = WebClient.builder()
            .exchangeFunction(stub)
            .filter(authFilter)
            .build();

    client.get().uri("/anything").retrieve().toBodilessEntity().block();

    assertThat(captured.get().headers().getFirst("Authorization")).isEqualTo("Bearer t");
}
```

`AtomicReference`로 들어온 요청을 캡처해 헤더 검증. 필터가 여러 개라면 등록 순서대로 캡처해 검증.



## WebTestClient — 컨트롤러 위에서 진입

> WebFlux 컨트롤러를 검증하는 도구. WebClient를 쓰는 어댑터 코드의 단위 테스트는 아니지만, "어댑터를 호출하는 컨트롤러"가 정상 동작하는지 통합 검증할 때 쓴다.

### 컨트롤러 단위 — `bindToController`

```java
@Test
void 컨트롤러가_사용자_조회를_정상_반환() {
    WebTestClient client = WebTestClient.bindToController(new UserController(userService))
            .build();

    client.get().uri("/users/{id}", 42L)
            .exchange()
            .expectStatus().isOk()
            .expectBody(User.class)
            .consumeWith(result -> {
                User u = result.getResponseBody();
                assertThat(u.id()).isEqualTo(42L);
            });
}
```

컨트롤러 빈만 띄우므로 가볍다. 서비스 빈은 mock으로 주입한다.

### 통합 — `bindToServer` + 외부 URL

이미 떠 있는 서버에 진입한다. e2e 테스트에 가깝다.

```java
WebTestClient client = WebTestClient.bindToServer()
        .baseUrl("http://localhost:8080")
        .build();
```

### `@AutoConfigureWebTestClient`

`@SpringBootTest`와 함께 쓰면 자동 설정된 `WebTestClient`를 주입받는다.

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureWebTestClient
class UserApiTest {

    @Autowired
    WebTestClient client;

    @Test
    void 통합_조회() {
        client.get().uri("/users/{id}", 42L)
                .exchange()
                .expectStatus().isOk();
    }
}
```



## 어떤 도구를 어디에 쓰는가 — 결정 가이드

> 한 페이지 요약.

| 검증 대상 | 권장 도구 |
|---------|----------|
| 외부 API 호출이 정확한 메서드·URL·헤더·본문으로 나가는가 | MockWebServer |
| 응답 본문 변환 로직(JSON 디코딩 + 도메인 예외 wrap) | MockWebServer 또는 `ExchangeFunction` 모킹 |
| `ExchangeFilterFunction` 자체 동작 | `ExchangeFunction` 모킹 |
| 재시도·타임아웃·지연 시나리오 | MockWebServer (`setBodyDelay`, 여러 enqueue) |
| WebFlux 컨트롤러 동작 | `WebTestClient.bindToController` |
| 서비스 전체 e2e | `@SpringBootTest` + `WebTestClient.bindToServer` |

대부분의 어댑터·서비스 단위 테스트는 MockWebServer 한 줄로 끝난다. 그 위에 필터·연산자 단위 테스트를 `ExchangeFunction` 모킹으로 보강하는 흐름이 일반적이다.



## 함정 — 자주 만나는 두 가지

### 1. `MockWebServer`를 정적 필드로 두고 테스트마다 큐를 비우지 않음

`enqueue`로 등록한 응답이 누적되어 다음 테스트에서 엉뚱한 응답이 반환된다. `@BeforeEach`/`@AfterEach`에서 매번 `start`/`shutdown`하거나, 한 번 띄워 놓고 매 테스트 시작 전에 큐를 비운다.

### 2. 비동기 호출이 끝나기 전에 검증을 진행

```java
// ❌ subscribe만 호출하고 즉시 검증 — 응답이 아직 안 옴
client.get().uri("/users").retrieve().bodyToMono(User.class).subscribe();
RecordedRequest request = server.takeRequest();   // 호출이 안 도착했을 수 있다
```

`.block()` 또는 `StepVerifier`로 명시적으로 완료를 기다린다. `MockWebServer.takeRequest`는 기본 무한 대기지만, 호출 자체가 일어나지 않으면 영원히 막힌다. `takeRequest(long, TimeUnit)` 형태로 타임아웃을 두는 편이 안전하다.



## 면접에서 받을 만한 질문

> 챕터 마무리 점검.

1. MockWebServer와 `ExchangeFunction` 모킹의 차이는?
   - 답 요지: 전자는 실제 HTTP 서버를 띄우므로 transport 레벨까지 검증되고, 후자는 Reactor 파이프라인만 검증한다. 후자는 시작 비용이 작지만 직렬화·인코딩 결함을 놓친다.
2. multipart 호출의 boundary 토큰을 검증하려면 어느 도구가 필요한가?
   - 답 요지: MockWebServer. raw 본문(`request.getBody().readUtf8()`)을 받아 boundary와 파트 헤더를 직접 검증.
3. 토큰 자동 갱신 시나리오를 검증하려면?
   - 답 요지: MockWebServer에 `enqueue`를 세 번(401 → 토큰 갱신 200 → 재시도 200) 등록하고 `takeRequest`를 세 번 꺼내 마지막 호출의 `Authorization` 헤더가 새 토큰인지 검증.
4. WebTestClient는 WebClient 호출 코드 단위 테스트에 적합한가?
   - 답 요지: 적합하지 않다. WebTestClient는 컨트롤러/핸들러 진입을 검증하는 도구다. 외부 호출 코드는 MockWebServer가 자연스럽다.
5. `MockWebServer.takeRequest()`가 무한 대기로 막히는 케이스는?
   - 답 요지: 호출 자체가 일어나지 않았을 때. `.subscribe()`만 호출하고 완료를 기다리지 않은 경우, 또는 호출 코드에서 예외가 떨어져 호출이 발생 안 한 경우. `takeRequest(timeout, unit)` 형태로 타임아웃을 두면 디버깅이 빠르다.
