# 테스트 피라미드와 Spring 테스트 종류

Spring Boot 프로젝트에서 테스트를 처음부터 한 그릇에 섞으면, 어느 순간 빌드가 5분에서 30분으로 늘어나면서 팀 전체가 테스트를 회피하기 시작한다. 단위 테스트가 통합 테스트와 같은 task에서 돌고, IT 한 건이 깨지면 PR 빌드가 컨테이너 기동 실패로 몇 분 헛돌고, 결국 누군가가 `@Disabled` 를 붙여 회로를 끊어 놓는다. 이 챕터의 목적은 그 사고가 발생하기 전에, 어느 층 테스트가 어떤 결함을 잡는지를 비용·속도·신뢰도 축에서 정리하고 Spring에서의 어노테이션·실행 도구·네이밍 규약을 함께 묶는 것이다.



## 왜 층을 분류하는가

모든 테스트를 단위 테스트로 작성하면 빠르지만 통합 결함을 못 잡고, 모든 테스트를 E2E로 작성하면 신뢰도는 높지만 한 번 돌리는 데 분 단위가 든다. Mike Cohn 의 테스트 피라미드는 이 트레이드오프를 한 그림으로 정리했다. 아래로 갈수록 격리가 좋아 빠르고 안정적이며, 위로 갈수록 실제 사용자 경로에 가까워 신뢰도가 올라간다. 한 층의 테스트가 다른 층을 대체하지 못한다는 점이 핵심이다. 도메인 객체의 산술 오류는 단위 테스트에서, 컨트롤러의 검증 어노테이션 누락은 슬라이스에서, JPA 매핑의 컬럼 타입 충돌은 통합에서, 외부 API 응답 형식 변화는 E2E에서 잡는다.

```
        /\
       /E2\        외부 시스템 결합 — 실제 환경 또는 stub 서버
      /----\       비용↑ 속도↓ 신뢰도↑
     / IT   \      Spring 컨텍스트 + DB/Kafka 등 인프라 기동
    /--------\     비용·속도·신뢰도 중간
   /  Slice   \    웹/JPA/메시지 슬라이스 — 컨텍스트 일부만
  /------------\   비용↓ 속도↑ 부분 신뢰도
 /  Unit / POJO \  컨텍스트 없음 — 도메인 로직, 정책 객체
/----------------\ 비용↓↓ 속도↑↑ 좁은 신뢰도
```

피라미드 모양을 강조하는 이유는 위층이 무거워질수록 아래층이 지지대 역할을 하지 못한다는 점 때문이다. 단위 테스트가 빈약하면 통합 테스트가 도메인 검증까지 떠안고, 그러면 통합 테스트는 한 건당 수십 초로 늘어나며 깨질 때마다 디버깅 비용이 급등한다.



## Spring에서의 4개 층 매핑

Spring Boot가 제공하는 테스트 도구를 피라미드 층에 매핑하면 다음과 같다.

| 층 | Spring 도구 | 컨텍스트 기동 | 외부 의존 | 잡는 결함 예시 |
|----|-------------|--------------|----------|---------------|
| 단위 | JUnit 5 + Mockito + AssertJ. Spring 의존 없음 | 없음 | 없음 | 정책 객체의 분기, 도메인 산술, 입력 검증 헬퍼 |
| 슬라이스 | `@WebMvcTest`, `@DataJpaTest`, `@JsonTest`, `@RestClientTest`, `MockMvcBuilders.standaloneSetup` | 부분 컨텍스트 | 없음(Mock) | 컨트롤러 검증 어노테이션, JPA 매핑/쿼리, JSON (역)직렬화 |
| 통합 | `@SpringBootTest` + Testcontainers / `@EmbeddedKafka` / `ApplicationContextRunner` | 전체 또는 다대 슬라이스 | 진짜 DB/Kafka | 트랜잭션 경계, 자동 설정 충돌, 다중 빈 결합 |
| E2E | `@SpringBootTest(webEnvironment=RANDOM_PORT)` + 실제/stub 서버 (WireMock·MockServer) | 전체 | 외부 API stub 또는 실제 | 외부 응답 변화, 큐·재시도·복구 흐름, 카오스 시나리오 |

Spring 5/Boot 2.x 시절의 `@RunWith(SpringRunner.class)` 는 JUnit 5에서 `@ExtendWith(SpringExtension.class)` 로 바뀌었고, `@SpringBootTest` 가 이를 메타 어노테이션으로 흡수하므로 Boot 사용자는 직접 쓸 일이 거의 없다. 슬라이스 어노테이션의 핵심 차이는 어떤 자동 설정만 제한적으로 켜는가이며, 자세한 내용은 01-04에서 다룬다.



## 네이밍 컨벤션 — *Test / *IT / *E2ETest

같은 폴더에 모든 테스트를 두면 IDE 단축키 한 번에 단위와 통합이 같이 돌고, CI는 PR마다 인프라를 띄운다. 이를 분리하기 위한 가장 가벼운 도구가 파일명 접미사다. 표준은 다음과 같다.

| 접미사 | 의미 | 위치 | 비용 |
|--------|------|------|------|
| `*Test.java` | 단위 또는 컨텍스트 없는 슬라이스 | `src/test/java/` | 1초 미만 |
| `*IT.java` | 통합 — Spring 컨텍스트 + 인프라(Testcontainers/EmbeddedKafka) | `src/test/java/` | 수 초 ~ 수십 초 |
| `*E2ETest.java` | 외부 시스템 결합 또는 풀 컨텍스트 시나리오 | `src/test/java/` | 수십 초 ~ 분 |

`Maven Failsafe Plugin` 의 기본 패턴(`*IT.java`/`IT*.java`/`*ITCase.java`)을 따르는 것이 호환성 면에서 가장 안전하다. Gradle 에서는 자체 task 정의로 분리하므로 패턴은 자유롭지만 표준을 따르면 IDE 와 빌드 도구 양쪽에서 자동 인식된다.

JUnit 5 `@Tag` 는 파일명과 별개의 분류 축을 제공한다. 같은 클래스 내 일부 테스트만 통합으로 돌리고 싶을 때 유용하지만, 가장 단순한 운영은 **클래스 단위로 접미사 + `@Tag` 를 이중으로 표시**하는 것이다. 이중 표시의 가치는 빌드 도구가 파일명 패턴으로 1차 필터하고 JUnit Platform 이 태그로 2차 필터해, 한쪽 설정이 빠져도 다른 쪽이 안전망이 된다는 점에 있다.



## Gradle test / integrationTest task 분리

`./gradlew build` 를 했을 때 단위 테스트만 실행되고 통합은 별도 명령에서만 도는 구조가 표준이다. TPS operator/ticket 모듈의 `build.gradle` 이 이 분리를 명시적으로 보여 준다.

```groovy
tasks.named('test', Test).configure {
    useJUnitPlatform {
        excludeTags 'integration'
    }
    exclude '**/*IT.class'
}

tasks.register('integrationTest', Test) {
    description = 'Runs ticket module integration tests.'
    group = 'verification'

    testClassesDirs = sourceSets.test.output.classesDirs
    classpath = sourceSets.test.runtimeClasspath
    shouldRunAfter tasks.named('test')

    useJUnitPlatform {
        includeTags 'integration'
    }
    include '**/*IT.class'

    environment 'TESTCONTAINERS_DOCKER_SOCKET_OVERRIDE', testcontainersDockerSocketOverride.get()

    doFirst {
        if (testcontainersRyukDisabled.get().equalsIgnoreCase('true')) {
            throw new GradleException('TESTCONTAINERS_RYUK_DISABLED must not be true for integrationTest.')
        }
    }
}
```

핵심은 세 가지다. 첫째, `test` task 는 `excludeTags 'integration'` + `exclude '**/*IT.class'` 로 통합 테스트를 이중으로 차단한다. 둘째, `integrationTest` 는 별도 task 로 등록하되 `testClassesDirs` 와 `classpath` 는 `test` 와 공유해 빌드 산출물을 재사용한다. 셋째, `doFirst` 블록에 환경 가드를 둬 `TESTCONTAINERS_RYUK_DISABLED=true` 같은 위험한 설정을 빌드 시작 시점에 차단한다. Ryuk 가 꺼지면 테스트가 실패하거나 강제 종료된 뒤 컨테이너가 시스템에 남아 다음 빌드가 포트 충돌로 깨지는 일이 잦다. 이런 가드는 빌드 스크립트가 코드라는 사실을 활용해, 운영 사고를 빌드 시점에 잡는 가장 값싼 방법이다.



## 함정과 회피

빌드 시간 폭증은 통합 테스트가 단위 테스트와 같은 task 에서 돌 때 가장 흔히 발생한다. PR 한 건당 컨테이너가 매번 뜨면 5분이던 빌드가 30분이 되며, 개발자는 PR 을 작은 단위로 쪼개기를 포기한다. 위 task 분리가 이 사고를 막는다.

Testcontainers 컨테이너 누수는 Ryuk(`testcontainers/ryuk`) 사이드카가 꺼졌을 때 발생한다. JVM 이 비정상 종료하면 컨테이너가 시스템에 남고, 다음 빌드가 포트 충돌로 깨진다. CI 환경에서 Ryuk 을 끄고 싶은 유혹이 있어도, 위 가드처럼 명시적으로 차단해 두는 편이 운영을 안정화한다.

`@SpringBootTest` 의 컨텍스트 캐시는 양날의 검이다. 같은 어노테이션 조합을 가진 테스트끼리 컨텍스트를 공유해 빠르지만, `@MockBean` 이나 `@DirtiesContext` 가 섞이면 캐시 키가 갈라지면서 컨텍스트가 여러 개 떠 OOM 으로 이어질 수 있다. 통합 테스트가 늘어나기 시작하면 한 번씩 빌드 로그에서 `Application context starts/stops` 횟수를 확인하는 습관이 도움이 된다.



## TPS 사례 — Test / IT / E2ETest 컨벤션

operator/ticket 모듈은 약 70개 테스트를 다음 비율로 운영한다.

- 단위·슬라이스 (`*Test.java`): `TcktMngControllerTest`(standalone MockMvc + ControllerAdvice), `WebLoggingFilterTest`(필터 + Logback ListAppender) 같은 컨텍스트 없는 빠른 테스트가 다수.
- 통합 (`*IT.java`): `WorkflowApiCreateIT` 가 `WorkflowApiTestSupport` 를 거쳐 `AbstractMariaDbIntegrationTest`(Testcontainers MariaDB + `@ServiceConnection`) 를 상속하며, `@Tag("integration")` 으로 이중 표시.
- 스모크 (`MariaDbContainerSmokeIT.java`): 컨테이너 기동 자체만 검증해, 인프라 깨짐을 빠르게 분리.

executor/engine 은 같은 컨벤션 위에 추가 접미사 `*E2ETest.java` 를 둔다. `JenkinsConnectionE2ETest`(외부 Jenkins 가용성 ping), `SubmittedRecoveryE2ETest`(WireMock 으로 Jenkins API stub + 스케줄러 수동 호출), `JenkinsDownChaosE2ETest`(다운 시뮬레이션) 처럼 외부 시스템 결합 또는 카오스 시나리오를 분리한다. E2E 라는 이름이 과도하게 보일 수 있지만, "외부 시스템 응답 형식 변화"는 실제로 단위·통합으로는 잡히지 않는 결함이며, 카오스·복구 시나리오는 별도 라이프사이클로 관리될 가치가 있다.

한국어 `@DisplayName` + `ReplaceUnderscores` + `[Green]/[Red]` 접두사는 TPS 컨벤션이지만, 다른 팀에도 추천할 만하다. 메서드명이 `createWorkflowRejectsSelectedTaskWorkflowWithoutTaskCodes` 처럼 길어지더라도 한국어 DisplayName 이 IDE 트리에 그대로 보이고, `[Green]`/`[Red]` 가 happy/sad path 를 색깔로 인식하게 한다.

```java
@DisplayName("[Red] 선택 업무코드 워크플로우에 업무코드 목록이 비어 있으면 생성을 거부하고 저장소를 변경하지 않는다")
void createWorkflowRejectsSelectedTaskWorkflowWithoutTaskCodes() throws Exception { ... }
```
