# Virtual Threads Pinning
---
> Pinning은 Virtual Thread가 Carrier Thread에 고정되어 블로킹 시에도 unmount되지 못하는 현상이다. `synchronized` 블록 안에서 I/O가 발생하면 Virtual Thread의 핵심 이점이 사라지고 Platform Thread와 동일한 성능 병목이 생긴다.

## 1. Pinning 정의

*Pinning*은 Virtual Thread가 Carrier Thread에서 분리(unmount)되지 못하고 고정(pin)된 상태를 말한다. 정상 동작에서는 블로킹 연산이 발생하면 Virtual Thread가 Carrier Thread를 양보하고 힙에 저장되지만, Pinning 상태에서는 Carrier Thread도 함께 블로킹된다.

```
정상 동작:
  Virtual Thread 블로킹 → Carrier Thread 양보 → 다른 Virtual Thread 실행 가능

Pinning 발생:
  Virtual Thread 블로킹 → Carrier Thread에 고정 → Carrier Thread도 함께 블로킹
  → Platform Thread와 동일한 문제 발생
```

Carrier Thread 수는 CPU 코어 수와 동일하므로, 모든 Carrier Thread가 Pinning 상태에 빠지면 새로운 Virtual Thread가 실행될 수 없게 된다.

## 2. Pinning 발생 조건

Pinning이 발생하는 조건은 두 가지다.

### `synchronized` 블록 내 블로킹 I/O

`synchronized` 블록은 JVM 내부의 모니터(Monitor) 객체를 사용한다. 모니터는 OS 레벨 뮤텍스와 연결되어 있고, 모니터 소유권이 Platform Thread에 귀속된다. Virtual Thread가 모니터를 보유한 채 블로킹 연산에 진입하면 모니터를 유지해야 하므로 unmount가 불가능해진다.

```java
// ❌ Pinning 발생: synchronized 블록 내 블로킹 I/O
Object lock = new Object();
synchronized (lock) {
    Thread.sleep(1000);          // Carrier Thread가 1초 동안 블로킹됨
    httpClient.get("https://..."); // 네트워크 I/O도 동일
    file.read();                   // 파일 I/O도 동일
}
```

### JNI(native 메서드) 호출

네이티브 코드 실행 중에는 JVM이 Java 스택 상태를 변경할 수 없다. 스택 상태를 저장할 수 없으므로 Virtual Thread를 unmount할 수 없다.

```java
// ❌ Pinning 발생: JNI 호출
public native void nativeMethod();

Thread.startVirtualThread(() -> {
    nativeMethod(); // 네이티브 코드 실행 중 Carrier Thread에 고정됨
});
```

## 3. Pinning 감지 방법

### JVM 옵션으로 콘솔 경고 출력

가장 빠르게 확인하는 방법이다. Pinning이 발생할 때마다 스택 트레이스를 출력한다:

```bash
# full: 전체 스택 트레이스 출력
java -Djdk.tracePinnedThreads=full -jar app.jar

# short: 첫 번째 프레임만 출력
java -Djdk.tracePinnedThreads=short -jar app.jar
```

### JFR(Java Flight Recorder) 이벤트

프로덕션 환경에서 지속적으로 모니터링할 때 사용한다. `jdk.VirtualThreadPinned` 이벤트를 수집하면 Pinning이 발생한 시간, 스레드, 스택 트레이스를 기록할 수 있다:

```bash
# JFR 기록 시작
java -XX:StartFlightRecording:filename=recording.jfr,settings=profile \
     -jar app.jar

# JDK Mission Control(JMC)로 recording.jfr 파일을 열어
# jdk.VirtualThreadPinned 이벤트를 확인한다
```

## 4. 해결 방법: synchronized → ReentrantLock 전환

`synchronized` 대신 `java.util.concurrent.locks.ReentrantLock`을 사용하면 Pinning이 발생하지 않는다. `ReentrantLock`은 JVM 모니터를 사용하지 않으므로 블로킹 시 unmount가 정상적으로 이루어진다:

```java
// ❌ Pinning 발생
private final Object lock = new Object();

public void doWork() {
    synchronized (lock) {
        blockingIO(); // Carrier Thread 고정
    }
}

// ✅ Pinning 없음
private final ReentrantLock lock = new ReentrantLock();

public void doWork() {
    lock.lock();
    try {
        blockingIO(); // Virtual Thread가 정상적으로 unmount됨
    } finally {
        lock.unlock(); // finally 블록에서 반드시 해제
    }
}
```

`ReentrantLock`은 `synchronized`와 동일한 상호 배제를 제공하면서, Virtual Thread 환경에서 훨씬 효율적으로 동작한다.

## 5. Java 버전별 Pinning 상태

Java 24(JEP 491)부터 `synchronized`의 Pinning 문제가 근본적으로 해결되었다. 모니터 소유권이 Platform Thread에서 Virtual Thread 기준으로 전환되었기 때문이다:

| Java 버전 | synchronized + 블로킹 I/O | 비고 |
|-----------|:---:|------|
| Java 21 | Pinning 발생 | 실제 처리 시간: 3,451ms (5000태스크, 5ms 블로킹) |
| Java 24+ | Pinning 없음 | 실제 처리 시간: 81ms (동일 조건) |

Java 21 환경에서는 `synchronized` 블록 내 블로킹 I/O를 반드시 `ReentrantLock`으로 교체해야 한다.

## 6. 라이브러리별 Pinning 상태

주요 라이브러리의 Pinning 상태와 대응 방안은 다음과 같다:

| 라이브러리 | Pinning 여부 | 대응 방안 |
|-----------|:---:|---------|
| Hibernate/Spring Data JPA 3.3.0 이하 | 발생 (이슈 보고됨) | 버전 업그레이드 또는 Java 24+ 사용 |
| HikariCP | 발생 | 유지 (성능 개선 효과 미미하다고 판단, 공식 입장) |
| JDBC 드라이버 (PostgreSQL, MySQL) | 드라이버마다 다름 | `-Djdk.tracePinnedThreads`로 직접 확인 |
| 파일 I/O (java.io) | Java 21+에서 대부분 없음 | - |

## 7. 실무 마이그레이션 체크리스트

기존 Platform Thread 기반 코드를 Virtual Thread로 전환할 때 확인해야 할 항목이다:

- `-Djdk.tracePinnedThreads=full` 옵션으로 애플리케이션을 실행하고 Pinning 발생 지점을 찾는다.
- Pinning이 발생한 `synchronized` 블록 내부에 블로킹 I/O가 있는지 확인한다.
- 블로킹 I/O가 있는 `synchronized` 블록을 `ReentrantLock`으로 교체한다.
- CPU 바운드 작업을 처리하는 `synchronized` 블록은 교체할 필요가 없다. Pinning이 발생해도 블로킹 I/O가 없으면 성능 영향이 미미하다.
- 사용 중인 서드파티 라이브러리의 Virtual Thread 지원 여부를 공식 문서에서 확인한다.
- Java 24 이상으로 업그레이드하면 `synchronized` 관련 Pinning이 자동으로 해결된다.
