# 실습 — 의존성 키워드

---

> 02-01에서 다룬 configuration 모델·api/implementation·project 의존·BOM을 손으로 한 번씩 만든다. 빈 프로젝트에서 시작해 다섯 단계로 멀티모듈 빌드의 의존성 그래프가 손에 붙는다.

준비물은 JDK 17 이상과 Gradle Wrapper만 있으면 된다. 새 디렉터리를 만들고 `gradle init`으로 빈 멀티모듈 프로젝트를 생성한 뒤 시작한다.



## 실습 1. implementation vs api로 다운스트림 노출 차이 관찰

> 같은 라이브러리를 `implementation`으로 두는 모듈과 `api`로 두는 모듈의 다운스트림 컴파일 결과가 어떻게 다른지 본다.

`core/build.gradle`에서 `commons-lang3`을 `implementation`으로 등록하고, `app` 모듈이 `core`를 의존하게 만든다. `app`의 코드에서 `org.apache.commons.lang3.StringUtils`를 import해 보면 컴파일 에러가 난다 — `implementation`이 다운스트림에 노출되지 않기 때문이다(02-01 §3·§4).

다음으로 `core`의 선언을 `api 'org.apache.commons:commons-lang3:3.14.0'`으로 바꾸고 `core/build.gradle`의 plugin을 `java-library`로 교체한다. 다시 빌드하면 `app`이 `StringUtils`를 직접 import할 수 있다. ABI가 어디까지 전파되는지 코드로 확인하는 단계다.

확인 명령:

```bash
./gradlew :app:compileJava
```

`api`로 두었을 때만 성공하면 실습이 통과한 것이다.



## 실습 2. core 모듈을 분리하고 `project(":core")`로 참조

> `settings.gradle`의 `include`와 `project(...)` 의존이 멀티모듈을 어떻게 묶는지 본다.

기존 단일 모듈 프로젝트에 `core` 디렉터리를 만들고 `settings.gradle`에 한 줄을 추가한다.

```groovy
include 'core'
include 'app'
```

`core/build.gradle`에 `id 'java-library'` 플러그인을 적용하고 도메인 클래스 한 개(`User.java`)를 만든다. `app/build.gradle`에 `implementation project(':core')`로 의존을 선언하면 같은 빌드 안에서 `core`의 코드를 외부 publish 없이 그대로 쓸 수 있다(02-01 §7).

확인 명령:

```bash
./gradlew :app:dependencies --configuration compileClasspath
```

출력 트리에 `project :core`가 나오면 통과다.



## 실습 3. `dependencies` 트리 출력으로 transitive 의존성 보기

> Gradle이 좌표를 어떻게 펼쳐 classpath를 만드는지 한 화면에 본다.

`app/build.gradle`에 `implementation 'org.springframework.boot:spring-boot-starter-web:3.2.3'`을 추가한 뒤 다음 명령을 실행한다.

```bash
./gradlew :app:dependencies --configuration runtimeClasspath
```

출력은 의존성 트리다. `spring-boot-starter-web`이 `spring-web`·`spring-webmvc`·`tomcat-embed-core` 등을 transitive로 끌어오는 모습이 보인다. 화살표 표기(`->`)가 있는 줄은 충돌 해소(02-02 §2.1) 결과 다른 버전으로 갈아탄 의존성이다.

`--configuration` 인자를 `compileClasspath`로 바꾸면 컴파일에만 보이는 의존성, `testRuntimeClasspath`로 바꾸면 테스트 실행에 들어가는 의존성을 볼 수 있다(02-01 §3 표 참조).



## 실습 4. `dependencyInsight`로 충돌 추적

> 같은 좌표가 두 버전 등장할 때 누가 어느 버전을 끌어왔는지 추적한다.

`app/build.gradle`에 의도적으로 충돌을 만든다.

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web:3.2.3'
    implementation 'com.fasterxml.jackson.core:jackson-databind:2.15.0'
}
```

Spring Boot 3.2.3이 끌어오는 jackson 버전과 직접 선언한 2.15.0이 다르므로 Gradle은 한 버전을 채택한다. 어느 버전이 채택됐고 왜 그렇게 됐는지를 본다.

```bash
./gradlew :app:dependencyInsight \
    --configuration runtimeClasspath \
    --dependency com.fasterxml.jackson.core:jackson-databind
```

출력 마지막에 `Selection reasons:` 항목과 함께 채택된 버전, 선택 이유, 그 버전을 끌어온 의존 경로가 트리로 나온다. Selection reasons에 "By conflict resolution: between versions ..."이 보이면 newest wins 동작을, "By constraint"가 보이면 강제 핀이 작동한 것이다(02-02 §2.3).



## 실습 5. BOM 적용 전후 비교

> 같은 의존성에 버전을 명시하지 않았을 때 BOM이 어떻게 채워 주는지 본다.

`app/build.gradle`에 `dependency-management` 플러그인을 적용하지 않은 상태에서 다음을 시도한다.

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'
}
```

빌드는 실패한다 — Gradle이 버전을 알 수 없다.

이제 BOM을 임포트한다.

```groovy
plugins {
    id 'io.spring.dependency-management' version '1.1.7'
}

dependencyManagement {
    imports {
        mavenBom 'org.springframework.boot:spring-boot-dependencies:3.2.3'
    }
}

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'  // 버전 생략
}
```

빌드가 성공하고 `./gradlew :app:dependencies`에 jackson·tomcat 같은 transitive 의존성이 모두 BOM이 정한 버전으로 묶여 들어간 모습이 보인다(02-03 §2). 같은 효과를 Gradle 네이티브로 내려면 `dependencies { implementation platform(...) }`로 대체할 수 있다.



## 실습 점검 체크리스트

> 다섯 실습이 끝난 뒤 다음 질문에 모두 "그렇다"고 답할 수 있으면 의존성 키워드의 손에 붙음 단계가 통과된 것이다.

```
1. implementation과 api의 다운스트림 노출 차이를 코드로 보였는가?
2. project(":core") 참조와 외부 좌표 의존의 차이를 설명할 수 있는가?
3. ./gradlew :app:dependencies 트리에서 transitive 의존성과 충돌(->)을 식별할 수 있는가?
4. dependencyInsight 출력에서 Selection reasons를 해석할 수 있는가?
5. BOM이 어떻게 버전을 자동으로 채워 주는지 한 단락으로 설명할 수 있는가?
```
