# 패키징·배포·CI

---

> 빌드 산출물을 어떻게 만들고, 어떤 컨테이너 이미지 도구를 고르고, 사내 라이브러리를 어떻게 publish하고, CI에서 재현 가능한 빌드를 어떻게 돌리는가. 본 문서는 02-01·03-01·04-01에서 다룬 의존성·캐시·태스크 모델 위에서 운영·배포·자동화 측면을 묶는다.

본 문서는 다음 순서로 진행한다.

## 1. 산출물의 종류 — 5가지

> Java 빌드가 만들 수 있는 산출물은 jar·bootJar·bootWar·OCI 이미지·distribution zip 다섯 종류다.

각 산출물은 빌드 시점이 다르고 소비자가 다르다.

| 산출물 | 만드는 task | 소비자 | 언제 고르나 |
|--------|------------|-------|-----------|
| 일반 jar | `jar` | 다른 모듈, Nexus | 라이브러리 모듈 |
| bootJar | `bootJar` | `java -jar` 실행 | Spring Boot 애플리케이션 |
| bootWar | `bootWar` | servlet 컨테이너 | 외부 Tomcat/Jetty 배포 |
| OCI 이미지 | `bootBuildImage` 또는 `jib` | 컨테이너 런타임 | Kubernetes·Docker 배포 |
| distribution zip | `distZip`/`distTar` | 클라이언트 OS 직접 실행 | CLI 도구·전통 배포 |

흐름의 결정 기준은 "이 산출물을 누가 어떻게 받아 가는가"다. 같은 모듈이 jar와 OCI 이미지를 동시에 publish할 수도 있고, 라이브러리 모듈이라면 jar 하나로 충분하다. 가장 흔한 함정은 라이브러리 모듈에 Spring Boot 플러그인이 붙어 자동으로 `bootJar`를 만들어 일반 jar가 비활성화되는 경우다(04-01 §10.1).



## 2. Layered jar 깊이 — custom layer 정의

> 기본 4계층(`dependencies`·`spring-boot-loader`·`snapshot-dependencies`·`application`)으로 부족하면 `application.layers.toml`로 직접 자른다.

기본 계층은 변경 빈도 기준으로 잘려 있다(04-01 §10.2). 그러나 사내 라이브러리가 많은 환경에선 "사내 SNAPSHOT"과 "외부 SNAPSHOT"을 다른 layer로 두는 편이 캐시 hit rate가 더 높다. 사내 라이브러리는 매일 갱신되지만 외부 라이브러리는 거의 변하지 않기 때문이다.

```toml
# src/main/resources/application.layers.toml
[[layers]]
name = "dependencies"
contents = [
    "BOOT-INF/lib/*",
]

[[layers]]
name = "spring-boot-loader"
contents = [
    "org/springframework/boot/loader/**",
]

[[layers]]
name = "internal-snapshot-dependencies"
contents = [
    "BOOT-INF/lib/org.okestro-*-SNAPSHOT.jar",
]

[[layers]]
name = "external-snapshot-dependencies"
contents = [
    "BOOT-INF/lib/*-SNAPSHOT.jar",
]

[[layers]]
name = "application"
contents = [
    "BOOT-INF/classes/**",
    "META-INF/**",
]
```

```groovy
bootJar {
    layered {
        enabled = true
        configuration {
            file = file('src/main/resources/application.layers.toml')
        }
    }
}
```

layer 정의는 위에서 아래로 순서대로 매칭된다 — 한 파일은 처음 매칭된 layer에 들어간다. 그래서 더 구체적인 패턴(`org.okestro-*-SNAPSHOT.jar`)을 먼저 두고, 더 일반적인 패턴(`*-SNAPSHOT.jar`)을 뒤에 두어야 한다. 순서를 바꾸면 모든 SNAPSHOT이 첫 layer로 흘러가서 의도가 깨진다.

layer를 너무 잘게 자르면 Dockerfile `COPY` 줄이 길어지고 image manifest가 무거워진다. 4~6개가 보통 좋은 균형이다.



## 3. 컨테이너 이미지 빌드 3가지 비교

> bootBuildImage·Dockerfile+layertools·Jib는 모두 같은 OCI 이미지를 만들지만, 빌드 환경 요구·자동화·세부 통제 측면에서 절충점이 다르다.

| 도구 | 빌더 | Docker daemon 필요 | multi-arch | 사내 권장 케이스 |
|------|------|-------------------|-----------|----------------|
| `bootBuildImage` | Cloud Native Buildpacks (paketo) | 필요 | 제한적 (빌더 의존) | 로컬 시제품·간단 배포 |
| Dockerfile + layertools | Docker Engine | 필요 | `docker buildx`로 가능 | 세밀한 OS·런타임 통제가 필요할 때 |
| Jib | Java | 불필요 | 기본 제공 | CI에서 Docker socket 없이 빌드 |

### 3.1 bootBuildImage — 가장 단순한 경로

Spring Boot가 모든 결정을 자동화한다 — JRE 선택, layer 분리, base image 선택까지. 5분 안에 동작하는 이미지를 만들 수 있다는 점이 가장 큰 장점이다. 단점은 빌더가 정한 형식을 벗어나기 어렵다는 점이다. 사내 보안 정책에 맞춘 base image를 강제하려면 별도 builder를 만들어야 한다.

### 3.2 Dockerfile + layertools — 가장 자유로운 경로

OS 패키지 추가, JVM 인자 세밀 조정, 사내 base image 사용 등 모든 선택권을 가진다. 단점은 직접 작성해야 한다는 점이다 — Dockerfile 한 줄을 잘못 두면 layer 캐시 효율이 크게 달라진다(04-01 §10.3). multi-arch 빌드는 `docker buildx build --platform linux/amd64,linux/arm64`로 가능하지만 buildx 설정이 추가로 필요하다.

### 3.3 Jib — daemonless·automated의 균형

Java로 작성된 빌드 도구라 Docker daemon이 필요 없다. CI 컨테이너 안에서 Docker socket을 마운트하지 않고 이미지를 만들 수 있어 보안·운영 부담이 가장 작다. layer 분리·multi-arch가 자동이고, base image 변경도 `from { image = '...' }` 한 줄이다. 단점은 Dockerfile 수준의 OS 통제는 불가능하다 — JVM 옵션·entrypoint·환경변수 정도까지가 한계다.

세 도구는 배타적이지 않다. operator는 로컬 시제품에 `bootBuildImage`, CI 표준 빌드에 `jib`를 함께 둔다. 한 도구만 골라야 한다면 CI 환경에서 Docker socket을 다룰 수 있는지가 첫 분기점이다.



## 4. maven-publish — 사내 Nexus에 라이브러리 publish

> 사내 공통 라이브러리를 Nexus에 올려 다른 빌드가 받게 만드는 메커니즘.

`maven-publish` 플러그인이 표준 도구다.

```groovy
plugins {
    id 'java-library'
    id 'maven-publish'
}

publishing {
    publications {
        mavenJava(MavenPublication) {
            from components.java

            // 좌표
            groupId = 'org.okestro'
            artifactId = 'core-lib'
            version = '3.0.5.2-SNAPSHOT'

            // POM 메타데이터
            pom {
                name = 'Operator Core Library'
                description = '사내 공통 도메인 모델·유틸'
                licenses {
                    license {
                        name = 'Proprietary'
                    }
                }
            }
        }
    }

    repositories {
        maven {
            def releaseUrl = 'https://nexus.example.com/repository/maven-releases/'
            def snapshotUrl = 'https://nexus.example.com/repository/maven-snapshots/'
            url = version.endsWith('SNAPSHOT') ? snapshotUrl : releaseUrl
            credentials {
                username = project.findProperty('nexusUsername') ?: ''
                password = project.findProperty('nexusPassword') ?: ''
            }
        }
    }
}
```

`./gradlew publish`로 모든 publication을 모든 저장소에 올린다. `publishToMavenLocal`은 `~/.m2/repository`에 publish해 다른 사내 빌드가 `mavenLocal()`로 받게 한다(03-01 §10).

서명(`signing` 플러그인)은 외부 저장소(maven Central) publish에 필수이고 사내 Nexus엔 보통 생략한다. 사내 Nexus가 좌표 단위 권한을 강제하는 환경이라면 publish-only 권한을 가진 별도 계정을 만들어 credentials를 분리한다.



## 5. 보조 플러그인 — 품질·커버리지·정적 분석

> jacoco·checkstyle·pmd·sonarqube 네 플러그인이 자주 묶여 들어간다.

| 플러그인 | 역할 | 주요 task | 결과 위치 |
|---------|------|----------|----------|
| `jacoco` | 테스트 커버리지 측정 | `jacocoTestReport` | `build/reports/jacoco/` |
| `checkstyle` | 코딩 스타일 검사 | `checkstyleMain`, `checkstyleTest` | `build/reports/checkstyle/` |
| `pmd` | 정적 분석 | `pmdMain`, `pmdTest` | `build/reports/pmd/` |
| `org.sonarqube` | SonarQube 서버 결과 업로드 | `sonar` | (서버에 업로드) |

```groovy
plugins {
    id 'jacoco'
    id 'checkstyle'
    id 'pmd'
    id 'org.sonarqube' version '5.1.0.4882'
}

jacoco {
    toolVersion = '0.8.12'
}

tasks.named('test') {
    finalizedBy tasks.named('jacocoTestReport')
}

jacocoTestReport {
    dependsOn tasks.named('test')
    reports {
        xml.required = true   // SonarQube가 xml 리포트를 읽음
        html.required = true
    }
}

checkstyle {
    toolVersion = '10.18.2'
    configFile = rootProject.file('config/checkstyle/checkstyle.xml')
    maxWarnings = 0
}

sonar {
    properties {
        property 'sonar.host.url', 'https://sonar.example.com'
        property 'sonar.projectKey', 'okestro:operator'
        property 'sonar.coverage.jacoco.xmlReportPaths',
            "${layout.buildDirectory}/reports/jacoco/test/jacocoTestReport.xml"
    }
}
```

세 가지 운영 팁이 있다. `jacocoTestReport`는 `test`의 `finalizedBy`로 묶어 테스트 실패 여부와 무관하게 리포트가 생성되게 한다. checkstyle/pmd의 `maxWarnings = 0`은 처음 도입할 때 점진적으로 적용한다 — 한 번에 0으로 두면 기존 코드가 모두 실패한다. SonarQube 업로드는 CI 마지막 단계에 두고 PR 단위 분석은 `sonar.pullrequest.*` 속성을 함께 넘긴다.



## 6. buildSrc · convention plugin · included build — 공통 설정 분리

> 멀티모듈에서 같은 빌드 로직이 여러 모듈에 반복되면 별도 위치로 빼낸다.

세 가지 도구의 차이는 적용 범위와 발견 비용이다.

| 도구 | 위치 | 적용 범위 | 언제 |
|------|------|---------|-----|
| `subprojects {}` 블록 | 루트 `build.gradle` | 모든 서브 모듈 | 짧은 공통 설정 |
| `buildSrc/` | `buildSrc/src/main/groovy` | 같은 빌드 안의 모든 모듈 | 중간 규모 공통 로직 |
| Convention plugin (included build) | 별도 디렉터리, `includeBuild('build-logic')` | 여러 빌드가 공유 | 큰 조직, 멀티 저장소 |

### 6.1 buildSrc/ — 같은 빌드 안의 공통 로직

`buildSrc/`는 `settings.gradle` 옆에 두는 특수 디렉터리다. Gradle이 자동으로 인식해 같은 빌드의 plugin classpath에 올린다.

```
operator-api/
├── settings.gradle
├── build.gradle
├── buildSrc/
│   ├── build.gradle
│   └── src/main/groovy/
│       └── org/okestro/build/JavaConventionPlugin.groovy
└── core-library/
    └── build.gradle
```

```groovy
// buildSrc/build.gradle
plugins {
    id 'groovy-gradle-plugin'
}
```

```groovy
// buildSrc/src/main/groovy/org/okestro/build/java-conventions.gradle
plugins {
    id 'java'
}

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(17)
    }
}

tasks.withType(JavaCompile).configureEach {
    options.compilerArgs.add('-parameters')
}

tasks.withType(Test).configureEach {
    useJUnitPlatform()
}
```

```groovy
// core-library/build.gradle
plugins {
    id 'org.okestro.build.java-conventions'
}
```

`subprojects {}` 블록과 같은 효과를 내지만 모듈별로 명시적으로 적용해야 한다는 점이 다르다. 어떤 모듈이 어떤 convention을 따르는지 모듈 자체가 선언하는 게 더 추적 가능하다.

### 6.2 Included build — 여러 저장소가 공유하는 build logic

조직이 커서 build logic을 별도 git 저장소로 분리해야 하면 `includeBuild`를 쓴다.

```groovy
// settings.gradle
includeBuild('../okestro-build-logic')
```

`okestro-build-logic` 저장소를 한 번 가져와 두면 여러 프로젝트가 같은 convention plugin을 공유한다. CI에서 build logic 변경이 있을 때 한 곳에서 관리하고 모든 빌드에 일괄 적용된다.

`buildSrc/`와 included build를 함께 쓰지 않는 게 보통이다. 하나만 골라 일관되게 가져간다.



## 7. Gradle Wrapper와 CI 재현성

> CI에서 빌드가 매번 같은 결과를 내려면 Wrapper·daemon·캐시 옵션을 명시적으로 통제한다.

CI에서 권장하는 옵션 묶음은 다음과 같다.

```bash
./gradlew clean build \
    --no-daemon \
    --build-cache \
    --configuration-cache \
    --parallel \
    -Dci=true
```

각 옵션의 이유:

| 옵션 | 이유 |
|------|------|
| `--no-daemon` | CI 컨테이너는 빌드 후 사라지므로 daemon 유지 비용이 무의미 |
| `--build-cache` | 같은 입력의 task 출력 재사용 (04-01 §6) |
| `--configuration-cache` | 빌드 스크립트 평가 결과 재사용 (04-01 §7) |
| `--parallel` | 모듈 간 독립 task 동시 실행 |
| `-Dci=true` | operator의 SNAPSHOT 캐시 TTL 0초 강제 트리거 (03-01 §8) |

`./gradlew` 자체가 `gradle/wrapper/gradle-wrapper.properties`의 distributionUrl로 정해진 Gradle 버전을 받는다(04-01 §8). CI에서 시스템 Gradle을 별도 설치하지 않는다 — 항상 Wrapper를 통해 실행해 빌드 도구 버전을 프로젝트가 통제한다.

`gradle.properties`에 영구 설정을 두면 `--build-cache --configuration-cache --parallel`을 매번 명시할 필요가 없다.

```properties
# gradle.properties
org.gradle.caching=true
org.gradle.configuration-cache=true
org.gradle.parallel=true
org.gradle.jvmargs=-Xmx2g -Dfile.encoding=UTF-8
org.gradle.daemon=false
```

CI에서 매 빌드를 깨끗하게 시작하려면 GRADLE_USER_HOME을 컨테이너별 별도 경로로 두고 캐시는 외부 볼륨으로 마운트한다. 캐시 디렉터리를 통째로 비우는 행위는 캐시의 의미를 무너뜨리므로 마지막 수단이다.



## 8. Jenkinsfile / GitHub Actions — 멀티모듈 task path

> CI 파이프라인이 호출하는 task path가 정확해야 산출물이 기대 위치에 생긴다.

### 8.1 Jenkinsfile (declarative pipeline)

```groovy
pipeline {
    agent { label 'java17' }

    options {
        timeout(time: 30, unit: 'MINUTES')
    }

    environment {
        NEXUS_CRED = credentials('<credential-id>')
        REGISTRY_CRED = credentials('<registry-credential-id>')
    }

    stages {
        stage('Build') {
            steps {
                sh '''
                    ./gradlew clean :app:build \
                        --no-daemon --build-cache \
                        -PnexusUsername=$NEXUS_CRED_USR \
                        -PnexusPassword=$NEXUS_CRED_PSW \
                        -Dci=true
                '''
            }
        }

        stage('Image') {
            steps {
                sh '''
                    ./gradlew :app:jib \
                        --no-daemon \
                        -Djib.to.auth.username=$REGISTRY_CRED_USR \
                        -Djib.to.auth.password=$REGISTRY_CRED_PSW
                '''
            }
        }

        stage('Publish library') {
            when { branch 'main' }
            steps {
                sh '''
                    ./gradlew :core-lib:publish \
                        --no-daemon \
                        -PnexusUsername=$NEXUS_CRED_USR \
                        -PnexusPassword=$NEXUS_CRED_PSW
                '''
            }
        }
    }

    post {
        always {
            junit 'build/test-results/test/*.xml'
            archiveArtifacts artifacts: 'app/build/libs/*.jar', fingerprint: true
        }
    }
}
```

핵심 두 가지. `:app:build`처럼 모듈 한정 task path를 쓰는 것이 빌드 시간을 절반으로 줄인다 — 다른 모듈의 test까지 매번 돌릴 필요가 없다. credentials는 항상 Jenkins credential store(`<credential-id>`)에서 받아 환경변수로 노출한다 — 빌드 스크립트나 Jenkinsfile에 평문으로 두지 않는다.

### 8.2 GitHub Actions

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: 17

      - name: Setup Gradle (wrapper + cache)
        uses: gradle/actions/setup-gradle@v4

      - name: Build
        run: |
          ./gradlew :app:build \
            --no-daemon --build-cache \
            -PnexusUsername=$NEXUS_USERNAME \
            -PnexusPassword=$NEXUS_PASSWORD \
            -Dci=true
        env:
          NEXUS_USERNAME: ${{ secrets.NEXUS_USERNAME }}
          NEXUS_PASSWORD: ${{ secrets.NEXUS_PASSWORD }}

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: '**/build/test-results/**/*.xml'
```

`gradle/actions/setup-gradle@v4`는 Gradle build cache·configuration cache·dependency cache를 자동으로 GitHub Actions cache에 mount한다. 첫 빌드는 cold start지만 두 번째부터 mavenCentral 다운로드·컴파일이 모두 캐시 hit으로 끝난다.



## 9. Build Scan과 빌드 가시성

> 빌드가 느릴 때, 회귀가 발생할 때 가장 빠른 진단 도구.

`./gradlew build --scan`을 부르면 빌드 끝에 URL이 출력되고, Gradle Enterprise(또는 무료 scans 서버)에 빌드 트레이스가 업로드된다. 화면에서 다음을 한 번에 본다.

- 어떤 task가 얼마나 걸렸는가 (timeline)
- 어떤 의존성을 어디서 받았는가 (network)
- build cache hit/miss 비율
- configuration cache 적용 여부
- JVM 메모리 사용량
- 빌드 환경(JDK·OS·Gradle 버전)

```bash
./gradlew build --scan
```

CI에서 항상 `--scan`을 켜 두면 매 빌드마다 같은 정보가 누적되어 회귀 추적이 가능해진다 — "지난주엔 4분 걸리던 빌드가 7분으로 늘었다"의 원인을 task별 시간 변화로 좁힐 수 있다.

영구 활성화는 `settings.gradle`에 두는 방식이 표준이다.

```groovy
// settings.gradle
plugins {
    id 'com.gradle.develocity' version '3.18'
}

develocity {
    server = 'https://ge.example.com'   // 사내 Develocity 서버 주소
    buildScan {
        publishing.onlyIf { it.authenticated }
        uploadInBackground = !System.getenv('CI')
    }
}
```

사내 Gradle Enterprise/Develocity 서버가 없으면 `https://scans.gradle.com`을 그대로 쓸 수 있지만, 빌드 정보가 외부에 업로드되니 사내 정책에 맞춰 결정한다.



## 10. 한 사례 — operator의 패키징·배포 흐름

> 본 문서가 다룬 도구가 실제 빌드에서 어떻게 결합돼 있는지.

operator는 Spring Boot 플러그인 + Jib + Jenkins를 한 묶음으로 사용한다.

`app/build.gradle`은 `org.springframework.boot`와 `com.google.cloud.tools.jib`를 함께 적용한다. `bootJar { enabled = true }` + `jar { enabled = false }`로 실행 산출물만 만들고, 일반 jar는 끈다(04-01 §10.1). 라이브러리 모듈(`core-library`/`core-web`/`core-db`/`cicd`/`ticket`)은 Spring Boot 플러그인을 적용하지 않아 자동으로 일반 jar만 만든다.

이미지 빌드는 Jib로 통일했다. Jenkins는 컨테이너 안에서 빌드를 돌리는데, Docker socket을 마운트하지 않으면서 OCI 이미지를 만들려면 daemonless 도구가 필요하다(§3.3). Jib는 Java로 작성돼 daemon 없이 동작하고, multi-arch도 기본 지원한다.

Nexus credentials와 registry credentials는 Jenkins credential store에서 받아 `-PnexusUsername`/`-PnexusPassword`로 빌드에 주입한다. 빌드 스크립트는 `project.findProperty('nexusUsername')` 패턴으로 받아 평문이 빌드 결과에 새지 않는다.

`subprojects {}` 블록에서 모든 모듈에 `-parameters` 컴파일 옵션과 Lombok·`spring-boot-starter-test`를 일괄 적용한다(02-01 §6). `buildSrc/`나 별도 convention plugin은 아직 도입되지 않았다 — 모듈 수가 늘어나거나 build logic이 더 복잡해지면 검토할 자리다.

Build Scan과 보조 플러그인(jacoco·checkstyle·pmd·sonarqube) 도입도 아직 진행 중인 단계다. 현재는 `./gradlew :app:bootRun`으로 로컬 실행하고 Jenkins가 `:app:build` + `:app:jib`로 산출물·이미지를 함께 만드는 가장 단순한 흐름이다.



## 11. 정리 — CI 빌드 점검 한 장

> 새 모듈을 CI에 올리거나 빌드가 느릴 때 같은 순서로 점검한다.

| 점검 항목 | 위치 | 실패 신호 |
|----------|------|----------|
| Wrapper 버전 | `gradle/wrapper/gradle-wrapper.properties` | CI와 로컬이 다른 Gradle 버전 |
| 모듈 한정 task path | Jenkinsfile/GHA | 모든 모듈 빌드로 시간 낭비 |
| `--no-daemon` | CI 명령 | 컨테이너 안 daemon 잔재 |
| build cache·configuration cache | `gradle.properties` | 캐시 미적용으로 매 빌드 colds |
| credentials 외부화 | property/환경변수 | 평문 secret이 로그에 노출 |
| publish 좌표 분기 | `maven-publish` repositories | release/SNAPSHOT 잘못된 저장소 |
| 이미지 빌드 도구 | `bootBuildImage`/`jib`/Dockerfile | Docker socket 의존 여부와 어긋남 |
| Build Scan | `--scan` 또는 develocity 플러그인 | 회귀 발생 시 추적 불가 |

빌드가 느릴 때는 `--scan`이 가장 빠른 진단 도구다. CI 컨테이너 안에서 빌드가 끊길 때는 Docker socket·credentials·외부 저장소 응답 세 자리를 차례로 의심한다. 새 모듈을 추가할 때는 본 표를 위에서 아래로 한 번 훑는 것이 가장 짧은 점검 경로다.
