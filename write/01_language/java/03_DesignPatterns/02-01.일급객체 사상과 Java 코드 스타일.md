# 일급객체 사상과 Java 코드 스타일

---

> 일급객체는 단순한 언어 스펙 용어가 아니다. "어떤 엔티티를 값처럼 자유롭게 다룰 수 있는가"라는 질문은 그대로 코드 스타일의 표현력과 유지보수성으로 이어진다. Java/Spring 백엔드 코드에 이 사상을 적용하면 분기는 줄고, 도메인은 살아나며, 테스트는 가벼워진다.

## 1. 일급객체란 무엇인가

일급객체(First-Class Object 또는 First-Class Citizen)는 프로그래밍 언어에서 다른 값들과 동등하게 취급되는 엔티티를 가리킨다. 1960년대 영국 컴퓨터 과학자 Christopher Strachey가 ALGOL에서 함수의 지위를 논하면서 처음 사용한 개념이다.

어떤 엔티티 X가 일급객체이려면 다음 조건을 만족해야 한다.

1. 변수에 할당할 수 있다.
2. 함수의 인자로 전달할 수 있다.
3. 함수의 반환값이 될 수 있다.
4. 자료구조(리스트, 맵 등)에 저장할 수 있다.
5. 런타임에 생성할 수 있다.
6. 고유한 식별성(identity)을 가진다.

이 여섯 가지를 모두 만족하면 일급, 일부만 만족하면 이급(Second-Class), 컴파일 타임에만 존재하면 삼급(Third-Class)으로 분류한다.

## 2. 자주 혼동되는 이웃 개념

일급객체는 다른 여러 개념과 묶여 등장한다. 각각이 어떻게 다른지 짚어두면 이후 적용 단계에서 혼란이 줄어든다.

**고차함수(Higher-Order Function)**는 일급함수가 가능해야 비로소 성립하는 패턴이다. 함수를 인자로 받거나 반환하는 함수를 가리킨다. `map`, `filter`, `reduce`가 대표 예시다. 일급함수가 조건이라면 고차함수는 그 조건이 열어주는 활용 형태다.

**클로저(Closure)**는 일급함수가 자신이 정의된 렉시컬 환경의 변수를 함께 캡처하는 구조다. "일급함수 + 렉시컬 스코프"의 조합으로 성립한다. Java의 람다도 effectively final 변수를 캡처하는 형태로 클로저성을 일부 가진다.

**함수형 인터페이스(Functional Interface)**는 Java처럼 함수가 진정한 일급 시민이 아닌 언어에서, 단일 추상 메서드를 가진 인터페이스로 함수를 흉내내는 우회 장치다. `Runnable`, `Function<T, R>`, `Comparator`가 여기에 속한다. JVM 바이트코드 수준에서는 여전히 객체이지만, 사용 시점의 표현력은 진짜 일급함수에 근접한다.

**일급 타입/모듈/컨티뉴에이션**처럼 일급 개념이 함수를 넘어 다른 엔티티로 확장된 사례도 있다. Python은 타입을 런타임 값으로 다룰 수 있고(`type()`), OCaml은 모듈을 값으로 전달할 수 있으며, Scheme은 제어 흐름 자체(`call/cc`)를 값으로 다룬다. Java는 이 영역에서는 보수적이다.

### 일급 객체와 일급 컬렉션 — 같은 "일급"의 두 얼굴

가장 큰 용어 혼동이 여기서 생긴다. *일급 객체*에서 "일급"은 *어떠한 연산도 가능한 시민*이라는 *언어 차원*의 정의를 가리키고, *일급 컬렉션*에서 "일급"은 *컬렉션을 클래스로 감싸 재사용성과 의미를 부여한다*는 *설계 패턴 차원*의 정의를 가리킨다. 둘은 같은 단어를 쓸 뿐 출발점이 다르다.

이 혼동이 만들어내는 함정이 있다. 감싸지 않은 평범한 `Map<K, V>`도 *언어 차원*에서는 이미 일급 객체다(변수에 담기고 인자로 넘어가고 반환된다). 그러나 *설계 차원*에서는 일급 컬렉션이 아니다. 두 정의의 스케일이 다르다는 점을 짚지 않으면, "일급 객체이므로 그대로 써도 된다"는 잘못된 결론에 도달한다. 본 문서가 [3.2 일급 컬렉션](#32-일급-컬렉션--컬렉션-자체를-타입으로-끌어올리기)에서 다루는 패턴은 *후자*의 일급, 즉 컬렉션에 도메인 의미를 부여하는 캡슐화 결정이다.

## 3. Java/Spring 코드에 적용하는 여섯 가지 패턴

일급객체 사상을 Java/Spring 백엔드 코드에 녹여 넣는 방법을 여섯 개의 패턴으로 정리한다. 각 패턴은 "흔히 보는 코드"와 "사상을 적용한 코드"의 Before/After로 보여준다.

### 3.1 일급함수 — 전략 패턴의 람다화

핵심 사상은 단순하다. 함수가 일급이라면, 분기는 if-else가 아니라 함수 선택으로 표현될 수 있다.

**Before**: 등급별 할인 정책이 if-else 분기로 메서드 안에 묻혀 있다.

```java
@Service
public class DiscountService {
    public BigDecimal calculate(Order order, String memberGrade) {
        if ("VIP".equals(memberGrade)) {
            return order.getAmount().multiply(new BigDecimal("0.2"));
        } else if ("GOLD".equals(memberGrade)) {
            return order.getAmount().multiply(new BigDecimal("0.1"));
        } else if ("SILVER".equals(memberGrade)) {
            return order.getAmount().multiply(new BigDecimal("0.05"));
        }
        return BigDecimal.ZERO;
    }
}
```

이 코드의 문제는 셋이다. 새 등급을 추가할 때마다 메서드를 수정해야 하므로 개방-폐쇄 원칙을 위반한다. 정책이 메서드 안에 묻혀 있어 정책 단위로 테스트하기 어렵다. 등급을 `String`으로 받기 때문에 오타가 컴파일러를 통과한다.

**After**: 정책 자체를 일급 시민으로 끌어올린다.

```java
public enum MemberGrade {
    VIP(amount -> amount.multiply(new BigDecimal("0.2")))
    , GOLD(amount -> amount.multiply(new BigDecimal("0.1")))
    , SILVER(amount -> amount.multiply(new BigDecimal("0.05")))
    , BASIC(amount -> BigDecimal.ZERO);

    private final UnaryOperator<BigDecimal> discountPolicy;

    MemberGrade(UnaryOperator<BigDecimal> discountPolicy) {
        this.discountPolicy = discountPolicy;
    }

    public BigDecimal applyDiscount(BigDecimal amount) {
        return discountPolicy.apply(amount);
    }
}

@Service
public class DiscountService {
    public BigDecimal calculate(Order order, MemberGrade grade) {
        return grade.applyDiscount(order.getAmount());
    }
}
```

정책은 enum 상수에 결합된 일급함수로 표현된다. 등급 추가는 enum에 한 줄을 더하는 작업으로 끝나고, 호출부는 변경되지 않는다. `MemberGrade.VIP.applyDiscount(new BigDecimal("100"))` 형태로 정책을 직접 단위 테스트할 수도 있다.

**Spring 맥락의 확장**: 전략이 Bean 의존성을 필요로 한다면 enum 대신 `List<Interface>` 주입 패턴이 적합하다.

```java
public interface DiscountPolicy {
    boolean supports(MemberGrade grade);
    BigDecimal apply(BigDecimal amount);
}

@Service
@RequiredArgsConstructor
public class DiscountService {
    private final List<DiscountPolicy> policies;

    public BigDecimal calculate(Order order, MemberGrade grade) {
        return policies.stream()
                .filter(p -> p.supports(grade))
                .findFirst()
                .orElseThrow()
                .apply(order.getAmount());
    }
}
```

Spring은 `List<Interface>` 주입 시 컨테이너에 등록된 모든 구현체를 모아 넘긴다. 이 관용구는 "전략을 일급 시민으로 다루는" Spring적 표현이다.

**극단적 적용: if-else 0개로 사칙연산**

dkswnkk가 NextStep TDD 과정에서 인용한 사례를 그대로 가져온다. 일급함수와 enum을 결합하면 "문자열 연산자에 따라 분기"라는 흔한 문제를 *조건문 0개*로 풀 수 있다.

```java
public enum ArithmeticOperator {
    PLUS("+", (a, b) -> a + b)
    , MINUS("-", (a, b) -> a - b)
    , MULTIPLY("*", (a, b) -> a * b)
    , DIVIDE("/", (a, b) -> {
        if (b == 0) throw new IllegalArgumentException("/0");
        return a / b;
    });

    private final String symbol;
    private final BiFunction<Integer, Integer, Integer> op;

    ArithmeticOperator(String symbol, BiFunction<Integer, Integer, Integer> op) {
        this.symbol = symbol;
        this.op = op;
    }

    public int apply(int a, int b) {
        return op.apply(a, b);
    }

    public static ArithmeticOperator from(String symbol) {
        return Arrays.stream(values())
                .filter(o -> o.symbol.equals(symbol))
                .findFirst()
                .orElseThrow(() -> new IllegalArgumentException("unknown: " + symbol));
    }
}

// 사용
int result = ArithmeticOperator.from("+").apply(3, 5);  // 8
```

`+`/`-`/`*`/`/` 분기를 `BiFunction`이라는 *함수 타입*에 위임하면, enum 상수가 *행위까지* 담는다. 새 연산자가 추가될 때 enum에 한 줄을 더하면 끝이고, 호출처에는 어떤 if도 들어가지 않는다. 일급함수가 없는 언어에서는 클래스 4개와 팩토리를 추가로 만들어야 같은 효과를 낼 수 있다.

### 3.2 일급 컬렉션 — 컬렉션 자체를 타입으로 끌어올리기

`List<X>`라는 자료구조는 그 자체로 의미를 가질 때가 많다. 의미가 있는 컬렉션은 원시 타입으로 노출하지 말고, 그 의미를 이름으로 가진 타입으로 감싸야 한다.

**Before**: `Order`가 `List<OrderLine>`을 직접 들고 있고, 라인 관련 로직이 Order 안에 누적된다.

```java
public class Order {
    private List<OrderLine> lines;

    public BigDecimal getTotalAmount() {
        return lines.stream()
                .map(OrderLine::getAmount)
                .reduce(BigDecimal.ZERO, BigDecimal::add);
    }

    public boolean hasOverLimitLine() {
        return lines.stream().anyMatch(l -> l.getAmount().compareTo(LIMIT) > 0);
    }

    public List<OrderLine> getLines() {
        return lines;  // 외부에서 add/remove 가능 — 캡슐화가 깨진다
    }
}
```

라인 처리 메서드가 늘어날수록 Order가 비대해진다. `getLines()`로 내부 컬렉션을 노출하는 순간 불변성도 함께 무너진다.

**After**: 컬렉션을 감싸는 일급 타입을 도입한다.

```java
public class OrderLines {
    private final List<OrderLine> lines;

    public OrderLines(List<OrderLine> lines) {
        this.lines = List.copyOf(lines);
    }

    public BigDecimal totalAmount() {
        return lines.stream()
                .map(OrderLine::getAmount)
                .reduce(BigDecimal.ZERO, BigDecimal::add);
    }

    public boolean hasOverLimitLine(BigDecimal limit) {
        return lines.stream().anyMatch(l -> l.getAmount().compareTo(limit) > 0);
    }

    public OrderLines filterBy(Predicate<OrderLine> predicate) {
        return new OrderLines(lines.stream().filter(predicate).toList());
    }

    public Stream<OrderLine> stream() {
        return lines.stream();
    }
}

public class Order {
    private final OrderLines orderLines;

    public BigDecimal getTotalAmount() {
        return orderLines.totalAmount();
    }
}
```

"주문 라인들"이라는 도메인 개념이 타입으로 승격되면서 Order의 책임은 가벼워진다. `List.copyOf()`로 받은 시점에 불변 복사를 만들기 때문에 외부에서 변형할 통로가 없다. 라인 관련 단위 테스트는 OrderLines를 직접 대상으로 진행할 수 있다.

적용 신호는 간단하다. `List<X>` 필드를 가진 클래스에 그 List를 다루는 메서드가 둘 이상 생기면, 일급 컬렉션으로 분리할 시점이다.

#### 일급 컬렉션의 세 가지 장점

일급 컬렉션이 만들어내는 가치는 세 갈래로 나뉜다. 각 장점이 다른 결합도/유효성/안정성 축에서 기여한다.

**가독성·유지보수성**: `List<OrderLine>`은 의미가 흐릿하지만 `OrderLines`는 도메인 어휘 그대로다. 컬렉션과 관련된 로직(`totalAmount`, `hasOverLimitLine`, `filterBy`)이 한 클래스에 응집되어 변경이 한 곳에서 끝난다. 호출처는 List 자료구조의 세부를 알 필요가 없다.

**유효성 검증의 한 곳 응집**: 생성자 안에서 `null`이나 빈 컬렉션, 도메인 규칙 위반(예: 라인 수가 1000개 초과) 같은 조건을 한 번에 검증하면, 시스템 어디에도 "검증 안 된 컬렉션"이 떠다닐 수 없다. 호출처마다 반복하던 검증 코드가 사라진다.

**불변성 보장**: 생성자에서 `List.copyOf(lines)`로 받고, 외부 노출이 필요할 때만 `Collections.unmodifiableList(lines)` 또는 `Stream<X>`로 내준다. 외부에서 `add`/`remove`로 내부를 변형할 통로가 없다. 동시성 환경에서도 *외부 변경에 의한* 문제는 발생하지 않는다.

> **실무 적용 예시 — TPS approval 도메인**
>
> `ApprovalExecution`(결재 실행 Aggregate Root)은 `List<ApprovalExecutionStep> steps`와 `List<ApprovalExecutionDetail> details` 둘을 보유한다. Aggregate가 직접 일급 컬렉션 역할을 겸하는 형태다.
>
> ```java
> public class ApprovalExecution {
>     private List<ApprovalExecutionStep> steps;
>
>     public List<ApprovalExecutionStep> getSteps() {
>         return steps == null ? Collections.emptyList()
>                              : Collections.unmodifiableList(steps);
>     }
>
>     public void addStep(ApprovalExecutionStep step) {
>         Objects.requireNonNull(step, "step must not be null");
>         validateMutableStatus();         // ← 상태 검증 응집
>         if (steps == null) steps = new ArrayList<>();
>         steps.add(step);
>     }
>
>     private void validateMutableStatus() {
>         if (prgrsSttsCd != WAIT && prgrsSttsCd != EXCN) {
>             throw new TpsException(EXECUTION_ALREADY_TERMINAL, ...);
>         }
>     }
> }
> ```
>
> 위 세 장점이 동시에 적용된 사례다. `addStep`은 *왜 추가가 가능한 상태인지*를 도메인 안에서 검증하고(검증), getter는 `unmodifiableList`로 내준다(불변성). 호출처는 Aggregate Root가 제공하는 메서드만 통하므로 "step 처리는 결재 실행이 안다"는 도메인 어휘가 살아난다(가독성). 다음 단계 진화는 `steps`만 따로 분리한 `ApprovalSteps` VO로 끌어올리는 것인데, 현재 Aggregate가 충분히 응집돼 있어 굳이 분리하지 않은 결정이다.

### 3.3 Value Object — 원시 타입 집착 제거

Primitive Obsession은 의미가 있는 값을 `String`, `Long`, `int` 같은 원시 타입으로 표현하는 경향을 말한다. 도메인 개념이 타입으로 드러나지 않으면, 컴파일러는 그 의미를 보호해주지 못한다.

**Before**: 사용자 정보를 모두 원시 타입으로 다룬다.

```java
public class User {
    private Long id;
    private String email;
    private String phoneNumber;
}

@Service
public class UserService {
    public void sendNotification(String email, String phoneNumber) {
        if (email == null || !email.contains("@")) {
            throw new IllegalArgumentException();
        }
        // 검증 로직이 호출처마다 흩어진다
    }
}
```

`sendNotification(phoneNumber, email)`처럼 인자 순서를 바꿔 호출해도 컴파일러가 잡지 못한다. 이메일 검증 로직은 모든 호출처에 반복된다. "이 String이 진짜 유효한 이메일인가?"를 어디서도 확신할 수 없다.

**After**: Java 14+ `record`로 Value Object를 도입한다.

```java
public record Email(String value) {
    private static final Pattern PATTERN =
            Pattern.compile("^[\\w.+-]+@[\\w.-]+\\.[A-Za-z]{2,}$");

    public Email {
        Objects.requireNonNull(value, "email must not be null");
        if (!PATTERN.matcher(value).matches()) {
            throw new IllegalArgumentException("invalid email: " + value);
        }
    }

    public String domain() {
        return value.substring(value.indexOf('@') + 1);
    }
}

public record PhoneNumber(String value) {
    public PhoneNumber {
        if (!value.matches("^\\d{2,3}-\\d{3,4}-\\d{4}$")) {
            throw new IllegalArgumentException("invalid phone: " + value);
        }
    }
}

public class User {
    private UserId id;
    private Email email;
    private PhoneNumber phoneNumber;
}

@Service
public class UserService {
    public void sendNotification(Email email, PhoneNumber phone) {
        // 컴파일 타임에 인자 순서가 보호된다
        // 이미 검증된 객체이므로 재검증이 필요 없다
    }
}
```

불가능한 상태가 표현 자체로 불가능해진다. invalid Email은 생성되는 순간 예외가 터지므로, 시스템 어디에도 "검증 안 된 이메일"이 떠다닐 수 없다. 검증 로직은 한 곳에 응집되고, `email.domain()` 같은 도메인 메서드를 붙일 자연스러운 자리도 확보된다.

**JPA 호환 주의사항**: JPA에서 `record`를 직접 `@Embeddable`로 쓰는 것은 스펙상 아직 매끄럽지 않다. 두 가지 선택이 있다. 첫째, 도메인 모델은 record로 유지하고 JPA 엔티티는 별도의 `@Embeddable` 클래스로 분리하는 DDD 스타일. 둘째, Hibernate 6.x의 `@JdbcTypeCode`나 `AttributeConverter`로 우회. TPS v305p Domain-first 컨벤션이라면 전자가 정합적이다.

> **실무 적용 예시 — TPS approval 도메인의 ID 4종**
>
> `ApprovalExecution` Aggregate Root는 식별자 4종을 모두 VO로 분리한다.
>
> ```java
> private ApprovalExecutionId atrzExcnId;     // 결재 실행 ID
> private ApprovalKey atrzKey;                // 결재 정의 키 (atrzId + vsrn)
> private PageComponentKey pageCompKey;       // 페이지 컴포넌트 키
>
> // step 식별자는 또 다른 복합 키
> ApprovalExecutionStepKey.of(this.atrzExcnId, ApprovalKey.of(...), atrzSn);
> ```
>
> Primitive Obsession을 ID부터 깨는 게 가장 ROI 높다는 사실을 그대로 보여주는 사례다. 만약 `ApprovalExecutionId`와 `ApprovalKey`가 둘 다 `String`이었다면, `findById(atrzKey)` 같은 매개변수 순서 오타를 컴파일러가 잡아주지 못한다. VO로 분리되면 잘못된 ID 타입을 넘기는 즉시 컴파일 에러가 난다. 도메인이 "다른 의미의 식별자는 다른 타입"이라는 원칙을 컴파일러에 위임한 결정이다.

### 3.4 함수형 인터페이스 — 가벼운 의존성 주입

함수형 인터페이스는 "메서드 한 개짜리 함수 타입"이다. 이 관점을 의존성 주입에 끌어오면, 테스트 더블도 람다 한 줄로 끝낼 수 있다.

**Before**: 단순 조회에도 Mockito 보일러플레이트가 따라붙는다.

```java
public interface UserRepository {
    Optional<User> findById(Long id);
}

@Service
@RequiredArgsConstructor
public class UserService {
    private final UserRepository userRepository;

    public String getUserName(Long id) {
        return userRepository.findById(id)
                .map(User::getName)
                .orElse("Unknown");
    }
}

@ExtendWith(MockitoExtension.class)
class UserServiceTest {
    @Mock UserRepository userRepository;
    @InjectMocks UserService userService;

    @Test
    void getUserName() {
        when(userRepository.findById(1L)).thenReturn(Optional.of(new User(1L, "Alice")));
        assertEquals("Alice", userService.getUserName(1L));
    }
}
```

**After**: 의존성을 인터페이스 전체가 아니라 필요한 함수 하나로 좁힌다.

```java
@Service
public class UserService {
    private final Function<Long, Optional<User>> userFinder;

    public UserService(UserRepository userRepository) {
        this(userRepository::findById);
    }

    UserService(Function<Long, Optional<User>> userFinder) {  // 패키지-프라이빗, 테스트용
        this.userFinder = userFinder;
    }

    public String getUserName(Long id) {
        return userFinder.apply(id)
                .map(User::getName)
                .orElse("Unknown");
    }
}

class UserServiceTest {
    @Test
    void getUserName() {
        var service = new UserService(id -> Optional.of(new User(id, "Alice")));
        assertEquals("Alice", service.getUserName(1L));
    }

    @Test
    void getUserName_notFound() {
        var service = new UserService(id -> Optional.empty());
        assertEquals("Unknown", service.getUserName(1L));
    }
}
```

테스트는 순수 Java로 끝난다. Mockito 환경 이슈(예: Java 25 + inline mock 호환 문제)에서 자유로워진다. 의존성이 "Repository 전체"가 아니라 "함수 하나"로 좁혀지면서 Interface Segregation의 극단적 적용이 자연스럽게 따라온다.

다만 이 패턴을 모든 의존성에 적용하면 과잉이 된다. 순수 조회나 계산 의존성, 테스트 빈도가 높은 핵심 로직에 선별 적용하는 것이 좋다. 일반 CRUD Repository는 기존 인터페이스 주입이 적절하다.

### 3.5 Optional/Stream — 분기를 데이터 흐름으로

`null` 체크와 if-else는 제어 흐름이다. `Optional`과 `Stream`은 같은 작업을 데이터의 변환 파이프라인으로 표현한다. 표현 형태가 바뀌면 의도가 더 잘 드러난다.

**Before**: null 체크와 중첩 if가 쌓인다.

```java
public String getUserCity(Long userId) {
    User user = userRepository.findById(userId).orElse(null);
    if (user == null) return "UNKNOWN";

    Address address = user.getAddress();
    if (address == null) return "UNKNOWN";

    City city = address.getCity();
    if (city == null) return "UNKNOWN";

    return city.getName().toUpperCase();
}
```

**After**: 함수 합성으로 데이터 흐름을 그린다.

```java
public String getUserCity(Long userId) {
    return userRepository.findById(userId)
            .map(User::getAddress)
            .map(Address::getCity)
            .map(City::getName)
            .map(String::toUpperCase)
            .orElse("UNKNOWN");
}
```

의도가 한 문장으로 읽힌다: "유저에서 시작해 주소, 도시, 이름을 거쳐 대문자로 변환하고, 어디서든 끊기면 UNKNOWN을 돌려준다."

**안티패턴 경고**: `Optional`을 필드나 메서드 인자로 쓰지 않는다. 반환 타입에만 사용하는 것이 Java 설계 가이드(Brian Goetz)다. `Optional<User>` 필드는 `null` 필드보다 더 큰 부담을 만든다. 직렬화, equals/hashCode, JPA 매핑이 모두 어그러지기 때문이다.

### 3.6 고차함수 — 횡단 관심사 추상화

같은 트랜잭션 경계, 같은 측정 로직, 같은 재시도 패턴이 메서드마다 반복된다면, 함수를 받는 함수로 추출할 신호다.

**Before**: 같은 패턴이 메서드마다 반복된다.

```java
@Service
public class OrderService {
    public Order createOrder(OrderRequest req) {
        long start = System.currentTimeMillis();
        try {
            Order order = ... ;
            log.info("createOrder took {}ms", System.currentTimeMillis() - start);
            return order;
        } catch (Exception e) {
            log.error("createOrder failed", e);
            throw e;
        }
    }

    public void cancelOrder(Long id) {
        long start = System.currentTimeMillis();
        try {
            // 실제 로직
            log.info("cancelOrder took {}ms", System.currentTimeMillis() - start);
        } catch (Exception e) {
            log.error("cancelOrder failed", e);
            throw e;
        }
    }
}
```

**After**: 횡단 관심사를 고차함수로 추출한다.

```java
public class Measured {
    public static <T> T run(String name, Supplier<T> action) {
        long start = System.currentTimeMillis();
        try {
            T result = action.get();
            log.info("{} took {}ms", name, System.currentTimeMillis() - start);
            return result;
        } catch (Exception e) {
            log.error("{} failed", name, e);
            throw e;
        }
    }
}

@Service
public class OrderService {
    public Order createOrder(OrderRequest req) {
        return Measured.run("createOrder", () -> {
            // 순수 비즈니스 로직만
            return ...;
        });
    }
}
```

측정과 로깅이 한 곳에 응집되고, 비즈니스 메서드는 본질에 집중한다.

**Spring AOP와의 비교**: Spring AOP의 `@Around`도 같은 문제를 푼다. 차이는 표현 위치다. AOP는 어노테이션으로 선언적이지만 동적 프록시 한계(같은 클래스 내 self-invocation 미적용)가 있다. 고차함수는 명시적 호출이라 코드에 노출되지만, 컴파일 타임 안전성과 디버깅 가시성을 얻는다. 도메인 핵심 로직 측정이라면 고차함수, 광범위한 횡단 관심사라면 AOP가 어울린다.

## 4. 적용 우선순위

여섯 가지 패턴을 한꺼번에 도입하려 들면 코드베이스가 흔들린다. ROI가 큰 순서로 단계적으로 적용하는 것이 안전하다.

| 순서 | 패턴 | 도입 효과 |
|------|------|----------|
| 1 | Value Object (Primitive Obsession 제거) | 컴파일러가 즉시 도메인 버그를 잡기 시작한다. ROI가 가장 높다 |
| 2 | 일급 컬렉션 | VO와 짝을 이루어 도메인 모델이 본격적으로 살아난다 |
| 3 | 일급함수/전략 패턴 | if-else가 줄고 OCP가 자연스럽게 충족된다 |
| 4 | 함수형 인터페이스 DI | 테스트 더블 보일러플레이트가 사라진다 |
| 5 | Optional/Stream 점검 | 이미 쓰고 있다면 안티패턴(필드 사용 등)을 정비한다 |
| 6 | 고차함수 횡단 관심사 | 위 다섯이 익숙해진 후. 남용하면 가독성이 떨어진다 |

## 5. 일급객체 사상이 중요한 이유

일급객체 여부는 언어의 추상화 능력을 결정한다. 함수가 일급이면 콜백, 함수 합성, 함수형 프로그래밍이 자연스러워진다. 타입이 일급이면 메타프로그래밍과 동적 의존성 처리가 가능해진다. 컨티뉴에이션이 일급이면 코루틴과 비동기 제어를 라이브러리 수준에서 구현할 수 있다.

Java 8 이전에 람다가 없어 `Comparator`를 익명 클래스로 길게 써야 했던 불편이 곧 "함수가 비일급"의 비용이었다. Java 8 이후 람다와 메서드 레퍼런스가 도입되면서 같은 작업이 한 줄로 끝나는 변화는, 일급 시민성이 코드 스타일에 미치는 영향을 직접 보여준 사례다.

Java는 함수에 대해서는 부분적 일급(함수형 인터페이스 우회), 타입과 모듈에 대해서는 보수적 입장을 유지한다. 이 한계를 알고 우회 도구(`record`, sealed type, 함수형 인터페이스, Spring의 컨테이너 기능)를 적절히 조합하는 것이 Java/Spring 개발자의 표현력을 결정한다.
