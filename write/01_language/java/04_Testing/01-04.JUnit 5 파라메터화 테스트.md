# JUnit 5 파라메터화 테스트
---
> 같은 검증 로직을 여러 입력에 반복할 때 `@Test` 를 복사하지 않고 한 메서드에 데이터셋을 주입하는 기능이 파라메터화 테스트다. 입력 표가 곧 명세가 되며, 케이스가 늘어나도 함수 본문은 그대로다. 어떤 어노테이션을 어떤 데이터 모양에 쓰는지가 학습 포인트다.



## 1. 왜 파라메터화 테스트인가

값만 다르고 검증 로직이 같은 테스트를 여러 개 만들면, 케이스가 추가될 때마다 메서드 하나를 복사하고 어디 한 곳을 빠뜨리는 버그가 끼기 쉽다. 파라메터화 테스트는 입력 데이터를 외부 소스로 분리해 한 메서드에 주입하므로, 케이스 추가가 "표에 한 줄 더 적기" 로 끝난다.

다음은 같은 변환을 6개 입력에 대해 검증하는 코드다. `@ParameterizedTest` 가 `@Test` 자리를 대신하고, 입력 소스 어노테이션이 함께 붙는다.

```java
@ParameterizedTest
@CsvSource({
    "SUCCESS, SUCCESS"
    , "failure, FAILURE"
    , "Unstable, UNSTABLE"
    , "ABORTED, ABORTED"
    , "NOT_BUILT, NOT_BUILT"
    , "GARBAGE, FAILURE"
})
@DisplayName("Jenkins 결과 문자열 → 도메인 상태 매핑")
void mapsJenkinsResult(String input, ExecutionJobStatus expected) {
    assertThat(ExecutionJobStatus.fromJenkinsResult(input)).isEqualTo(expected);
}
```

`@DisplayName` 은 메서드 단위 이름이고, 각 케이스 별 이름은 `@ParameterizedTest(name = "...")` 의 패턴으로 따로 줄 수 있다. 패턴에서는 `{0}`, `{1}` 이 인자 위치, `{index}` 가 케이스 순서를 가리킨다.



## 2. 입력 소스 종류

JUnit 5 는 입력 데이터의 모양에 따라 여러 소스 어노테이션을 제공한다. 자주 쓰는 것은 `@ValueSource`, `@CsvSource`, `@EnumSource`, `@MethodSource`, 그리고 null·empty 보강용 `@NullSource` / `@EmptySource` 다.

### 2-1. `@ValueSource` — 단일 인자 1차원 배열

인자가 하나일 때 가장 단순한 소스. `strings`, `ints`, `longs`, `doubles`, `booleans`, `classes` 등 자료형별 속성을 가진다.

```java
@ParameterizedTest
@ValueSource(strings = {"SUCCESS", "FAILURE", "UNSTABLE"})
void terminal_strings(String s) {
    assertThat(ExecutionJobStatus.fromJenkinsResult(s).isTerminal()).isTrue();
}
```

### 2-2. `@CsvSource` — 다중 인자 표

콤마로 컬럼을 구분한 한 줄이 한 케이스가 된다. 인자가 2개 이상일 때 가장 자주 쓴다. 본 작업에서 사용한 형태가 이 소스다.

```java
@ParameterizedTest
@CsvSource({
    "PENDING,QUEUED", "PENDING,REJECTED"
    , "QUEUED,SUBMITTING", "QUEUED,PENDING", "QUEUED,REJECTED"
    , "SUBMITTING,SUBMITTED", "SUBMITTING,QUEUED", "SUBMITTING,PENDING"
    , "SUBMITTING,REJECTED", "SUBMITTING,ABORTED"
    , "SUBMITTED,RUNNING", "SUBMITTED,PENDING", "SUBMITTED,FAILURE", "SUBMITTED,ABORTED"
    , "RUNNING,SUCCESS", "RUNNING,FAILURE", "RUNNING,UNSTABLE", "RUNNING,ABORTED"
    , "RUNNING,NOT_BUILT", "RUNNING,NOT_EXECUTED", "RUNNING,PENDING"
})
@DisplayName("허용된 상태 전이는 모두 통과한다")
void allowedTransitions(ExecutionJobStatus from, ExecutionJobStatus to) {
    assertThat(from.canTransitionTo(to)).isTrue();
}
```

표 한 줄이 곧 도메인 명세 한 줄이다. 상태 전이표를 따로 그릴 필요 없이 테스트가 표 자체로 동작한다.

자주 쓰는 옵션은 두 가지다.

- `nullValues = "null"`: 표 안의 토큰 `null` 을 자바 `null` 로 해석한다. 기본값은 빈 배열이라 명시하지 않으면 `null` 토큰이 그냥 문자열 `"null"` 로 들어간다.
- `emptyValue = ""`: 비어있는 토큰이 `null` 이 아닌 빈 문자열을 의미함을 명시한다. 빈 토큰과 누락 토큰의 의미를 분리할 때 쓴다.

```java
@ParameterizedTest(name = "url={0}, username={1}, password={2} → invalid")
@CsvSource(value = {
        "null,u,p"
        , "http://x,null,p"
        , "http://x,u,null"
        , ",u,p"
        , "http://x, ,p"
        , "http://x,u, "
        , ", , "
}, nullValues = "null", emptyValue = "")
void anyMissingField_isInvalid(String url, String username, String password) {
    assertThat(new JenkinsConnectionInfo(url, username, password).isValid()).isFalse();
}
```

쉼표를 줄 끝이 아닌 다음 줄 앞에 배치하는 comma-leading 스타일을 쓰면 줄 추가/삭제 시 diff 가 깨끗하고, 이번 학습 작업처럼 라인 사이 콤마 누락 같은 실수를 시각적으로 잡기 쉽다.

### 2-3. `@EnumSource` — enum 값 일괄

대상 enum 의 일부 또는 전체를 자동 주입한다. `names` 로 화이트리스트를, `mode = EXCLUDE` 로 블랙리스트를 줄 수 있다. 같은 enum 의 부분집합에 같은 검증을 돌릴 때 가장 깔끔하다.

```java
@ParameterizedTest
@EnumSource(value = ExecutionJobStatus.class
        , names = {"SUCCESS", "FAILURE", "UNSTABLE", "ABORTED", "NOT_BUILT", "NOT_EXECUTED", "REJECTED"})
@DisplayName("terminal 7개 → true")
void terminal_isTrue(ExecutionJobStatus s) {
    assertThat(s.isTerminal()).isTrue();
}

@ParameterizedTest
@EnumSource(value = ExecutionJobStatus.class
        , names = {"PENDING", "QUEUED", "SUBMITTING", "SUBMITTED", "RUNNING"})
@DisplayName("non-terminal 5개 → false")
void nonTerminal_isFalse(ExecutionJobStatus s) {
    assertThat(s.isTerminal()).isFalse();
}
```

전수 검사가 필요하면 `@EnumSource(FailReason.class)` 처럼 `names` 를 생략해 enum 전체를 주입한다.

### 2-4. `@MethodSource` — 임의 자바 객체 스트림

값이 단순 리터럴을 넘는 객체일 때 쓴다. 같은 클래스의 `static` 메서드가 `Stream<Arguments>` 또는 `Stream<T>` 를 반환하면, 그 결과가 케이스로 들어간다. 픽스처 객체나 빌더로 만든 복잡 입력을 넘길 때 유용하다.

```java
private static Stream<TestJob> successJobs() {
    return Stream.of(TestJob.SUCCESS, TestJob.LONG_LOG, TestJob.COMPLEX);
}

@ParameterizedTest(name = "[{index}] {0} → SUCCESS")
@MethodSource("successJobs")
@DisplayName("UC01~UC04 풀 라이프사이클 - BUILD/SUCCESS 마감 Job")
void fullLifecycle_buildSuccess(TestJob target) throws Exception {
    runFullLifecycle(target, JobKind.BUILD);
}
```

소스 메서드는 같은 클래스 안에 두고 이름만 문자열로 참조한다. 다른 클래스의 메서드를 쓰려면 `@MethodSource("com.example.Fixtures#cases")` 처럼 FQN 을 쓴다.

### 2-5. `@NullSource` / `@EmptySource` — null·빈 입력 보강

다른 소스와 함께 붙여 null 케이스나 empty 케이스 한 건을 보충하는 용도. 메인 데이터셋과 직교하므로 본문은 그대로 두고 케이스 1건만 더한다.

```java
@ParameterizedTest
@NullSource
@DisplayName("null → FAILURE")
void nullInput_mapsToFailure(String input) {
    assertThat(ExecutionJobStatus.fromJenkinsResult(input)).isEqualTo(ExecutionJobStatus.FAILURE);
}
```

문자열 인자에 한해 `@NullAndEmptySource` 가 두 케이스를 한 번에 붙여준다.



## 3. 케이스 표시 이름 패턴

`@ParameterizedTest(name = "...")` 의 표시 이름은 IDE/리포트에서 케이스를 구분하는 핵심 단서다. 사용 가능한 토큰은 다음과 같다.

- `{index}`: 1부터 시작하는 케이스 순번.
- `{0}`, `{1}`, ...: 메서드 인자의 위치별 값.
- `{argumentsWithNames}`: `name=value` 형식으로 모든 인자.

```java
@ParameterizedTest(name = "[{index}] {0} → terminal")
@EnumSource(value = ExecutionJobStatus.class, names = {"SUCCESS", "FAILURE"})
void terminalSubset(ExecutionJobStatus s) { ... }
```

표시 이름이 좋으면 실패 시 곧장 어떤 케이스가 깨졌는지 보인다. 인자가 객체일 때는 객체의 `toString()` 이 호출되므로, 의미 있는 toString 을 가진 도메인 enum/VO 를 인자로 넘기는 패턴이 잘 맞는다.



## 4. 자주 빠지는 함정

### 4-1. `@Test` 와 혼용 금지

`@ParameterizedTest` 가 붙은 메서드에 `@Test` 를 같이 붙이지 않는다. 두 어노테이션은 서로 다른 실행기를 트리거하므로 동시 사용 시 동작이 정의되지 않는다.

### 4-2. `@CsvSource` 의 콤마 — 자바 배열 콤마와 CSV 콤마는 다르다

`@CsvSource(value = { ... })` 안에서 각 줄은 자바 문자열 리터럴이고, 줄 사이는 자바 배열 원소 구분이다. 즉 줄 사이에 `,` 가 없으면 컴파일 자체가 실패한다.

```java
// ❌ 컴파일 에러: "http://x,u,null" 와 ",u,p" 사이에 자바 콤마가 없음
@CsvSource(value = {
        "null,u,p"
        , "http://x,null,p"
        , "http://x,u,null"
        ",u,p"          // ← 콤마 누락 → "} expected"
})

// ✅ comma-leading 스타일이면 누락이 시각적으로 즉시 보임
@CsvSource(value = {
        "null,u,p"
        , "http://x,null,p"
        , "http://x,u,null"
        , ",u,p"
})
```

CSV 한 줄 안의 컬럼 구분은 그 문자열 안의 `,` 다. 이 둘을 구분해 의식하면 실수가 줄어든다.

### 4-3. `@MethodSource` 메서드는 `static` 이어야 한다

기본적으로 소스 메서드는 `static` 이어야 한다. 인스턴스 메서드를 쓰려면 클래스에 `@TestInstance(Lifecycle.PER_CLASS)` 를 줘야 하는데, 테스트 사이 상태 격리를 깨뜨리므로 가능한 정적 메서드를 권장한다.

### 4-4. `@CsvSource` 의 enum 자동 변환은 이름 매칭

`@CsvSource` 의 토큰이 enum 타입 인자로 들어가면 `EnumType.valueOf(token)` 으로 변환된다. 따라서 토큰은 enum 상수명과 정확히 일치해야 하고, 대소문자도 그대로다. `"success"` 라고 쓰면 `SUCCESS` 로 자동 변환되지 않으니 정확히 `"SUCCESS"` 로 적어야 한다.



## 5. 정리

파라메터화 테스트의 본질은 "검증 로직 한 벌 + 입력 표 N 줄" 이다. 표가 명세가 되도록 입력 소스를 고르는 것이 핵심이다. 단일 인자면 `@ValueSource`, 다중 인자면 `@CsvSource`, enum 전수/부분이면 `@EnumSource`, 객체 입력이면 `@MethodSource`, null/empty 보강은 `@NullSource` 계열을 쓴다. 표시 이름 패턴과 comma-leading 줄바꿈을 같이 쓰면, 케이스 추가가 한 줄로 끝나고 실패 시 어느 케이스가 깨졌는지 한눈에 보인다.
