# Foreign Function and Memory API

---

> FFM API는 JNI의 복잡성과 안전성 문제를 해결하기 위해 설계된 현대적 네이티브 상호운용 API로, Java 22에서 GA가 되었으며 Java 21에서는 Third Preview(JEP 442)로 사용할 수 있다.

## 등장 배경

Java에서 네이티브 코드(C, C++ 라이브러리)를 호출하는 전통적인 방법은 **JNI(Java Native Interface)**였다. JNI는 1997년부터 존재했지만 사용하기 매우 불편하다. 네이티브 헤더 파일을 생성하고, C 코드를 작성하고, 공유 라이브러리로 컴파일한 후, JVM이 로드할 수 있는 위치에 배치해야 한다. 타입 매핑도 수동으로 처리해야 하고, 잘못된 메모리 접근은 JVM 크래시로 이어진다.

또한 JNI를 통해 직접 메모리를 관리하려면 `ByteBuffer.allocateDirect()`나 `sun.misc.Unsafe`에 의존해야 했다. `Unsafe`는 이름 그대로 안전을 보장하지 않는 내부 API로, JPMS에서 캡슐화 대상이 되었다. **FFM(Foreign Function and Memory) API**는 이 문제들을 해결하기 위해 Project Panama에서 개발되었다.

FFM API의 릴리스 히스토리는 다음과 같다. Java 14-15에서 Foreign-Memory Access API 인큐베이터, Java 16에서 Foreign Linker API 인큐베이터로 시작했고, Java 19-20 Preview, Java 21 Third Preview(JEP 442)를 거쳐 **Java 22에서 GA(JEP 454)**로 확정되었다.

## MemorySegment와 Arena

*MemorySegment*는 연속된 메모리 영역을 추상화한 핵심 타입이다. 힙 메모리, 오프힙 네이티브 메모리, 메모리 맵 파일을 동일한 인터페이스로 다룬다. 각 `MemorySegment`는 크기와 주소를 가지며, 범위를 벗어난 접근 시 `IndexOutOfBoundsException`을 던진다. 이는 JNI나 `Unsafe`와 달리 메모리 안전성을 보장한다.

*Arena*는 네이티브 메모리의 생명주기를 관리하는 컨텍스트다. `Arena`가 닫히면 그 안에서 할당된 모든 `MemorySegment`가 해제된다. 이 방식은 수동 `free()` 호출 없이 구조적으로 메모리 누수를 방지한다.

```java
// Arena와 MemorySegment를 사용한 네이티브 메모리 관리
import java.lang.foreign.*;
import java.lang.foreign.MemoryLayout.PathElement;

// try-with-resources로 Arena 생명주기 관리
try (Arena arena = Arena.ofConfined()) {
    // C의 int[10] 배열에 해당하는 네이티브 메모리 할당
    MemorySegment intArray = arena.allocate(
            ValueLayout.JAVA_INT
            , 10
    );

    // 값 쓰기
    for (int i = 0; i < 10; i++) {
        intArray.setAtIndex(ValueLayout.JAVA_INT, i, i * i);
    }

    // 값 읽기
    for (int i = 0; i < 10; i++) {
        System.out.println(intArray.getAtIndex(ValueLayout.JAVA_INT, i));
    }
} // arena 종료 시 자동으로 메모리 해제
```

Arena의 종류는 네 가지다.

- `Arena.ofConfined()`: 생성한 스레드에서만 접근 가능, 가장 안전
- `Arena.ofShared()`: 여러 스레드에서 접근 가능, 스레드 안전
- `Arena.ofAuto()`: GC가 관리, 명시적 종료 불필요
- `Arena.global()`: 프로그램 종료까지 유지

## Linker와 SymbolLookup

**Linker**는 Java 메서드 핸들을 네이티브 함수에 연결하는 역할을 한다. `Linker.nativeLinker()`로 현재 플랫폼의 C ABI에 맞는 링커를 가져온다.

*SymbolLookup*은 공유 라이브러리에서 함수 심볼(주소)을 찾는다. 시스템 라이브러리는 `Linker.nativeLinker().defaultLookup()`으로, 특정 라이브러리는 `SymbolLookup.libraryLookup()`으로 로드한다.

```java
// C의 strlen 함수 호출 예시
import java.lang.foreign.*;
import java.lang.invoke.MethodHandle;

Linker linker = Linker.nativeLinker();

// 시스템 라이브러리에서 strlen 심볼 찾기
SymbolLookup stdlib = linker.defaultLookup();
MemorySegment strlenSymbol = stdlib.find("strlen")
        .orElseThrow(() -> new RuntimeException("strlen not found"));

// FunctionDescriptor로 시그니처 기술
// strlen(const char *s) -> size_t
FunctionDescriptor strlenDescriptor = FunctionDescriptor.of(
        ValueLayout.JAVA_LONG     // 반환 타입: size_t
        , ValueLayout.ADDRESS     // 파라미터: const char *
);

// MethodHandle 생성
MethodHandle strlen = linker.downcallHandle(strlenSymbol, strlenDescriptor);

// 호출
try (Arena arena = Arena.ofConfined()) {
    MemorySegment cString = arena.allocateFrom("Hello, World!");
    long length = (long) strlen.invoke(cString);
    System.out.println("Length: " + length); // 13
}
```

## FunctionDescriptor

*FunctionDescriptor*는 네이티브 함수의 파라미터와 반환 타입을 Java 타입으로 기술한다. `ValueLayout`이 C 기본 타입에 대응한다.

| C 타입 | ValueLayout | 크기 |
|--------|-------------|------|
| `int`, `int32_t` | `JAVA_INT` | 4 bytes |
| `long`, `int64_t` | `JAVA_LONG` | 8 bytes |
| `float` | `JAVA_FLOAT` | 4 bytes |
| `double` | `JAVA_DOUBLE` | 8 bytes |
| `char` | `JAVA_BYTE` | 1 byte |
| 포인터 (`void *`, `char *`) | `ADDRESS` | 플랫폼 의존 |

반환값이 없는(`void`) 함수는 `FunctionDescriptor.ofVoid(파라미터...)`를 사용한다.

## JNI vs FFM API 비교

| 항목 | JNI | FFM API |
|------|-----|---------|
| 안전성 | 메모리 오류 시 JVM 크래시 가능 | 범위 외 접근 시 예외 발생 |
| 코드 복잡도 | C 헤더 생성, JNI 래퍼 작성 필요 | 순수 Java 코드로 작성 |
| 타입 안전성 | 수동 타입 변환, 런타임 오류 위험 | `FunctionDescriptor`로 컴파일 타임 기술 |
| 성능 | 오버헤드 최소화 가능 | JNI와 동등하거나 더 나음 (인라인 최적화) |
| 메모리 관리 | 수동 `free()` 또는 `DeleteLocalRef` | Arena 기반 구조적 해제 |
| 디버깅 | 어려움 (크래시 덤프 분석) | Java 스택트레이스 + 예외 |

FFM API의 성능은 JNI와 동등하거나 더 나을 수 있다. JVM이 downcall 경로를 인식하고 인트린식(intrinsic) 최적화를 적용할 수 있기 때문이다.

## 실무 활용 사례

FFM API가 유용한 상황은 크게 두 가지다. 첫째는 기존 C/C++ 라이브러리를 활용해야 할 때다. 이미지 처리(libpng, libjpeg), 암호화(OpenSSL), 데이터 압축(zlib, lz4), 벡터 연산 라이브러리 등은 Java 에코시스템에 동등한 성능의 구현이 없거나, 기존 시스템과의 인터페이스가 필요한 경우가 있다.

둘째는 성능이 중요한 메모리 관리다. GC 대상이 되지 않는 오프힙 메모리를 직접 관리하여 GC 압박을 줄이거나, 대용량 데이터를 Zero-Copy로 처리할 때 유용하다.

```java
// 실용 예시: libz (zlib)를 사용한 데이터 압축
import java.lang.foreign.*;
import java.lang.invoke.MethodHandle;

class ZlibCompressor {
    private static final Linker LINKER = Linker.nativeLinker();
    private static final SymbolLookup ZLIB;
    private static final MethodHandle COMPRESS;

    static {
        ZLIB = SymbolLookup.libraryLookup("libz.so.1", Arena.global());

        // int compress(Bytef *dest, uLongf *destLen, const Bytef *source, uLong sourceLen)
        COMPRESS = LINKER.downcallHandle(
                ZLIB.find("compress").orElseThrow()
                , FunctionDescriptor.of(
                        ValueLayout.JAVA_INT   // return: int
                        , ValueLayout.ADDRESS  // dest
                        , ValueLayout.ADDRESS  // destLen
                        , ValueLayout.ADDRESS  // source
                        , ValueLayout.JAVA_LONG // sourceLen
                )
        );
    }

    byte[] compress(byte[] input) throws Throwable {
        try (Arena arena = Arena.ofConfined()) {
            MemorySegment src = arena.allocateFrom(ValueLayout.JAVA_BYTE, input);
            long destSize = input.length + (input.length / 10) + 12;
            MemorySegment dest = arena.allocate(ValueLayout.JAVA_BYTE, destSize);
            MemorySegment destLenPtr = arena.allocate(ValueLayout.JAVA_LONG);
            destLenPtr.set(ValueLayout.JAVA_LONG, 0, destSize);

            int result = (int) COMPRESS.invoke(dest, destLenPtr, src, (long) input.length);
            if (result != 0) throw new RuntimeException("Compression failed: " + result);

            long compressedLen = destLenPtr.get(ValueLayout.JAVA_LONG, 0);
            return dest.asSlice(0, compressedLen).toArray(ValueLayout.JAVA_BYTE);
        }
    }
}
```

**jextract** 도구를 사용하면 C 헤더 파일에서 FFM API 바인딩 코드를 자동 생성할 수 있어, 복잡한 라이브러리 통합을 크게 단순화할 수 있다.
