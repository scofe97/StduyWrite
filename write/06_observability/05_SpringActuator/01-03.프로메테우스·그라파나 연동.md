# 프로메테우스·그라파나 연동
---

> [01-02](01-02.마이크로미터와%20메트릭%20—%20Counter·Gauge·Timer.md)에서 마이크로미터로 메트릭을 등록했습니다. 그런데 그 메트릭은 애플리케이션 메모리 안에 있을 뿐, 시간에 따라 저장되거나 그래프로 보이지 않습니다. 이 문서는 그 메트릭을 *프로메테우스가 수집해 저장*하고 *그라파나가 시각화*하는 연동을 다룹니다. 끝까지 읽으면 "애플리케이션이 메트릭을 노출하고, 프로메테우스가 끌어가고, 그라파나가 보여 주는" 3단 역할 분담을 그릴 수 있게 됩니다.

> **이 묶음과 LGTM 스택의 경계**: 본 문서는 *스프링 앱 쪽 설정*(메트릭을 프로메테우스 형식으로 노출)에 집중합니다. 프로메테우스·그라파나 자체의 설치·운영·대시보드 설계는 [`../02_LGTMStack/`](../02_LGTMStack/)과 [`../03_Project/`](../03_Project/)에서 인프라 관점으로 다룹니다.

## 1. 세 역할의 분담

모니터링은 세 주체가 역할을 나눠 맡습니다. 이 분담을 먼저 잡으면 설정이 어디에 왜 들어가는지 명확해집니다.

```text
[스프링 애플리케이션]          [프로메테우스]              [그라파나]
 마이크로미터로 측정     →    주기적으로 긁어가(scrape)   →   저장된 메트릭을
 /actuator/prometheus        시계열 DB에 저장              그래프로 시각화
 형식으로 노출
```

애플리케이션은 메트릭을 *노출*만 합니다. 프로메테우스가 그 엔드포인트를 *주기적으로 긁어*(pull/scrape) 시계열 데이터베이스에 저장하고, 그라파나는 프로메테우스에 질의해 *그래프로 그립니다*. 중요한 특징은 프로메테우스가 *끌어가는*(pull) 방식이라는 점입니다. 애플리케이션이 프로메테우스로 보내는 게 아니라, 프로메테우스가 애플리케이션에 와서 가져갑니다.

## 2. 애플리케이션 — 프로메테우스 형식으로 노출

프로메테우스는 자신만의 메트릭 텍스트 형식을 요구합니다. [01-02 §1](01-02.마이크로미터와%20메트릭%20—%20Counter·Gauge·Timer.md)에서 본 마이크로미터의 추상화 덕분에, 애플리케이션은 측정 코드를 바꾸지 않고 *의존성만 추가*해 프로메테우스 형식을 얻습니다.

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-actuator'
    implementation 'io.micrometer:micrometer-registry-prometheus'  // 프로메테우스 구현
}
```

`micrometer-registry-prometheus` 가 클래스패스에 올라가면 [자동 구성](../../11_spring/07_autoconfig/01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md)이 프로메테우스용 엔드포인트를 추가합니다. 액츄에이터 노출 설정에 `prometheus` 를 포함시킵니다.

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics, prometheus
```

이제 `/actuator/prometheus` 가 프로메테우스 형식의 텍스트를 내놓습니다.

```text
# GET /actuator/prometheus
# HELP my_order_count_total 주문 건수
# TYPE my_order_count_total counter
my_order_count_total{type="normal"} 42.0
my_order_count_total{type="cancel"} 3.0
```

[01-02](01-02.마이크로미터와%20메트릭%20—%20Counter·Gauge·Timer.md)에서 단 `tag("type", ...)` 가 여기서 `{type="normal"}` 라벨로 나타납니다. 애플리케이션이 할 일은 여기까지입니다 — 형식에 맞게 노출만 하면 됩니다.

## 3. 프로메테우스 — 긁어가기 설정

프로메테우스 쪽에는 "어느 애플리케이션의 어느 경로를 얼마나 자주 긁을지"를 설정합니다. 이것은 프로메테우스 설정 파일(`prometheus.yml`)에 들어갑니다.

```yaml
# prometheus.yml (프로메테우스 측 설정)
scrape_configs:
  - job_name: 'my-spring-app'
    metrics_path: '/actuator/prometheus'   # 2절에서 노출한 경로
    scrape_interval: 15s                    # 15초마다 긁어감
    static_configs:
      - targets: ['localhost:8080']
```

`metrics_path` 가 2절에서 연 엔드포인트를 가리키고, `scrape_interval` 이 수집 주기입니다. 프로메테우스는 이 설정대로 15초마다 `localhost:8080/actuator/prometheus` 를 호출해 그 시점의 메트릭을 가져가 시계열로 쌓습니다. 그래서 누적 Counter는 시간에 따라 증가하는 곡선이 되고, Gauge는 오르내리는 곡선이 됩니다.

> 이 부분은 *프로메테우스 측 운영*이므로 상세는 인프라 관점의 [`../03_Project/`](../03_Project/) 문서들과 연결됩니다. 본 묶음에서는 "애플리케이션 엔드포인트를 프로메테우스가 어떻게 지정하는가"까지만 봅니다.

## 4. 그라파나 — 시각화

그라파나는 프로메테우스를 *데이터 소스*로 등록하고, 저장된 메트릭을 쿼리해 대시보드로 그립니다. 그라파나 자체는 데이터를 저장하지 않습니다 — 프로메테우스에 질의할 뿐입니다.

```text
[그라파나 대시보드]
  데이터 소스: Prometheus
  쿼리(PromQL): rate(my_order_count_total[1m])   ← 분당 주문 증가율
       │
       ▼
  [프로메테우스에 질의] → 시계열 데이터 반환 → 그래프 렌더링
```

PromQL은 프로메테우스의 질의 언어입니다. 예를 들어 누적 Counter인 `my_order_count_total` 에 `rate(...[1m])` 을 씌우면 "분당 증가율"이 되어, 누적값이 아니라 *주문이 얼마나 빠르게 들어오는가*를 볼 수 있습니다. 그라파나는 이런 쿼리를 패널로 묶어 대시보드를 구성합니다. 그라파나 자체의 대시보드 설계·운영은 [`../02_LGTMStack/02-01.Grafana Core`](../02_LGTMStack/02-01.Grafana%20Core.md)에서 다룹니다.

## 5. 전체 흐름 다시 보기

[01-01](01-01.액츄에이터%20—%20운영%20엔드포인트.md)부터 여기까지를 한 줄로 꿰면 이렇습니다.

```text
1. 마이크로미터로 측정          (01-02) — Counter/Timer/Gauge 등록
2. 액츄에이터 + 프로메테우스 레지스트리로 노출  (본문 §2) — /actuator/prometheus
3. 프로메테우스가 주기적으로 긁어 저장        (본문 §3) — scrape
4. 그라파나가 질의해 시각화                  (본문 §4) — PromQL 대시보드
```

애플리케이션이 책임지는 것은 1~2단계뿐입니다. 측정하고 노출하면, 나머지 수집·저장·시각화는 인프라(프로메테우스·그라파나)가 맡습니다. 이 역할 분담 덕분에 애플리케이션 코드는 모니터링 인프라가 무엇이든 마이크로미터 API로만 측정하면 됩니다.

## 6. 정리

| 주체 | 역할 | 본 묶음에서 다루는 범위 |
|------|------|----------------------|
| 애플리케이션 | 마이크로미터로 측정, 프로메테우스 형식 노출 | 전체 (§2) |
| 프로메테우스 | 엔드포인트를 주기적으로 긁어 시계열 저장 | scrape 지정까지 (§3) |
| 그라파나 | 프로메테우스에 질의해 대시보드 시각화 | 연동 개념까지 (§4) |
| micrometer-registry-prometheus | 마이크로미터를 프로메테우스 형식으로 변환 | §2 |
| PromQL `rate()` | 누적 Counter를 증가율로 변환 | §4 |
