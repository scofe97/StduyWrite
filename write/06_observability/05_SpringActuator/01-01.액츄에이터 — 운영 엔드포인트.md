# 액츄에이터 — 운영 엔드포인트
---

> 서비스를 운영하면 장애는 언제든 일어납니다. 중요한 것은 장애를 *빨리 감지하고 대응*하는 것이고, 그러려면 애플리케이션이 자기 상태를 밖으로 드러내야 합니다. 스프링 부트 액츄에이터는 헬스 체크·메트릭·로그 레벨 같은 운영 정보를 HTTP 엔드포인트로 노출하는 기능입니다. 이 문서는 액츄에이터가 무엇을 제공하고 어떻게 켜는지, 그리고 운영 정보를 노출할 때의 보안 주의점을 정리합니다.

> **관점 구분**: 이 묶음(`05_SpringActuator/`)은 *스프링 앱이 운영 정보를 내보내는* 쪽을 다룹니다. 그 정보를 *수집·저장·시각화하는 인프라*(Grafana·Loki·Tempo)는 [`../02_LGTMStack/`](../02_LGTMStack/)이 담당합니다. 액츄에이터가 메트릭을 노출하고, LGTM 스택이 그것을 긁어 갑니다.

## 1. 프로덕션 준비 기능이란

개발이 끝난 애플리케이션을 운영에 올리면 새로운 요구가 생깁니다. 지금 살아 있는가(헬스), 메모리·CPU·요청 처리량은 어떤가(메트릭), 어떤 빈과 설정으로 떠 있는가, 로그 레벨을 운영 중에 바꿀 수 있는가. 이런 *운영을 위한 정보와 조작*을 묶어 프로덕션 준비 기능이라 부릅니다.

이 정보들을 직접 구현하려면 헬스 체크 컨트롤러를 만들고, 메트릭을 집계하고, 설정 조회 API를 짜야 합니다. 액츄에이터는 이 공통 운영 기능을 *엔드포인트*라는 형태로 미리 제공합니다. 의존성 하나만 추가하면 표준적인 운영 정보 창구가 열립니다.

## 2. 시작 — 의존성 한 줄

액츄에이터는 스타터로 제공됩니다.

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-actuator'
}
```

[01-01 스타터](../../11_spring/07_autoconfig/01-01.스타터와%20라이브러리%20버전%20관리.md)와 [자동 구성](../../11_spring/07_autoconfig/01-02.자동%20구성%20—%20@AutoConfiguration과%20@Conditional.md) 덕분에, 이 스타터가 클래스패스에 올라가면 액츄에이터 엔드포인트가 자동으로 등록됩니다. 기본 경로는 `/actuator` 이고, 그 아래 각 엔드포인트가 붙습니다.

```text
GET /actuator          ← 사용 가능한 엔드포인트 목록
GET /actuator/health   ← 헬스 상태
GET /actuator/info     ← 애플리케이션 정보
```

## 3. 다양한 엔드포인트

액츄에이터는 운영에 필요한 여러 엔드포인트를 제공합니다. 자주 쓰는 것들입니다.

| 엔드포인트 | 제공 정보 |
|-----------|----------|
| `health` | 애플리케이션·DB·디스크 등 구성요소 상태 (UP/DOWN) |
| `info` | 빌드 버전·git 커밋 등 애플리케이션 메타 정보 |
| `metrics` | JVM 메모리·GC·요청 수 등 메트릭 |
| `loggers` | 로거별 로그 레벨 조회 및 *런타임 변경* |
| `beans` | 등록된 스프링 빈 전체 목록 |
| `env` | `Environment` 의 설정값 (프로필·프로퍼티) |
| `mappings` | `@RequestMapping` 경로 매핑 전체 |
| `threaddump` | 스레드 덤프 |

이 중 `loggers` 는 조회뿐 아니라 *운영 중 로그 레벨 변경*까지 됩니다. 장애 조사 시 재배포 없이 특정 패키지 로그를 `DEBUG` 로 올렸다가 다시 내릴 수 있어, 운영 디버깅에서 특히 유용합니다.

## 4. 엔드포인트 설정 — 노출 범위 제어

기본적으로 대부분의 엔드포인트는 *비활성*이거나 *노출되지 않습니다*. 운영 정보를 함부로 열면 보안 위험이 되기 때문입니다. 어떤 엔드포인트를 HTTP로 노출할지는 설정으로 명시합니다.

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics, loggers
        # exclude: env, beans   ← 특정 엔드포인트 제외도 가능
```

`include: "*"` 로 전부 열 수도 있지만 운영에서는 권장하지 않습니다. `env`, `beans`, `mappings` 같은 엔드포인트는 내부 구조와 설정값(때로 민감정보)을 그대로 드러내므로, 필요한 것만 골라 여는 화이트리스트 방식이 안전합니다.

## 5. 헬스 정보

`health` 엔드포인트는 가장 많이 쓰입니다. 로드밸런서나 쿠버네티스가 "이 인스턴스가 트래픽을 받을 수 있는가"를 판단하는 신호로 씁니다.

```json
// GET /actuator/health
{
  "status": "UP",
  "components": {
    "db": { "status": "UP" },
    "diskSpace": { "status": "UP" },
    "ping": { "status": "UP" }
  }
}
```

전체 `status` 는 구성요소 중 하나라도 `DOWN` 이면 `DOWN` 이 됩니다. DB 연결이 끊기면 `db` 가 `DOWN` 이 되고 전체가 `DOWN` 으로 바뀌어, 로드밸런서가 그 인스턴스로 트래픽을 보내지 않게 됩니다. 기본 헬스 인디케이터(DB·디스크 등) 외에 커스텀 `HealthIndicator` 를 구현해 외부 시스템 연결 상태 같은 것도 추가할 수 있습니다.

상세 정보 노출 수준도 설정으로 조절합니다. 기본은 `status` 만 보이고, 구성요소 상세는 인증된 사용자에게만 보이게 하는 식입니다.

```yaml
management:
  endpoint:
    health:
      show-details: when-authorized   # never | when-authorized | always
```

## 6. 보안 — 운영 정보는 함부로 열지 않는다

액츄에이터 엔드포인트는 애플리케이션 내부를 들여다보는 창입니다. 그래서 노출에 신중해야 합니다. 실무에서 쓰는 보호 방법은 세 가지입니다.

첫째, 4절처럼 *필요한 엔드포인트만* 노출합니다. 둘째, 액츄에이터를 일반 서비스 포트와 *다른 포트*로 분리해 외부에서 직접 닿지 못하게 합니다.

```yaml
management:
  server:
    port: 9090   # 서비스는 8080, 액츄에이터는 9090 (방화벽으로 내부만 허용)
```

셋째, 스프링 시큐리티로 `/actuator/**` 경로에 인증을 겁니다. `health` 같은 가벼운 것은 열고, `env`·`beans` 같은 민감한 것은 인증을 요구하는 식으로 엔드포인트별 차등을 둘 수 있습니다. 운영 정보가 곧 공격 표면이라는 점을 잊지 않는 것이 핵심입니다.

## 7. 정리

| 주제 | 핵심 |
|------|------|
| 프로덕션 준비 기능 | 헬스·메트릭·설정 조회 등 운영용 정보/조작 |
| 시작 | `spring-boot-starter-actuator` 의존성, `/actuator` 경로 자동 등록 |
| 주요 엔드포인트 | health·info·metrics·loggers(런타임 변경)·beans·env |
| 노출 제어 | `management.endpoints.web.exposure.include` 화이트리스트 |
| health | 구성요소 상태 집계, 로드밸런서·k8s의 트래픽 판단 신호 |
| 보안 | 필요한 것만 노출 · 포트 분리 · 시큐리티 인증 |
