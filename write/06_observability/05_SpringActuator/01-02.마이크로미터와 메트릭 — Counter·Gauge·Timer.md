# 마이크로미터와 메트릭 — Counter·Gauge·Timer
---

> 모니터링 도구는 프로메테우스·그라파나·데이터독 등 여러 가지인데, 각 도구가 요구하는 메트릭 형식이 다 다릅니다. 애플리케이션 코드가 특정 도구 형식에 묶이면 도구를 바꿀 때마다 측정 코드를 다시 짜야 합니다. 마이크로미터는 이 문제를 *측정 추상화*로 풉니다. 이 문서는 마이크로미터가 무엇을 추상화하는지, 그리고 메트릭의 세 기본 타입(Counter·Gauge·Timer)을 언제 쓰는지를 정리합니다.

## 1. 마이크로미터 — 측정의 추상화

마이크로미터는 메트릭 측정을 위한 *표준 인터페이스*입니다. [`../../11_spring/01_core/01-01.객체지향 원리 적용 — DI와 IoC`](../../11_spring/01_core/01-01.객체지향%20원리%20적용%20—%20DI와%20IoC.md) §3 에서 본 "추상에 의존하라"가 모니터링에 적용된 모습입니다. SLF4J가 로깅 구현(Logback·Log4j)을 추상화하듯, 마이크로미터는 모니터링 구현(프로메테우스·데이터독 등)을 추상화합니다.

```text
애플리케이션 코드
     │  (마이크로미터 표준 API로만 측정)
     ▼
  Micrometer (측정 추상화)
     │
     ├─ Prometheus 구현 → 프로메테우스 형식으로 노출
     ├─ Datadog 구현    → 데이터독 형식으로 전송
     └─ ... 다른 백엔드
```

애플리케이션은 마이크로미터 API로만 측정하고, 어떤 모니터링 도구로 내보낼지는 *의존성과 설정*으로 정합니다. 프로메테우스에서 데이터독으로 갈아타도 측정 코드는 그대로입니다. 스프링 부트 액츄에이터는 마이크로미터를 기본 내장해, [01-01](01-01.액츄에이터%20—%20운영%20엔드포인트.md)의 `metrics` 엔드포인트로 측정값을 노출합니다.

JVM 메모리·GC·CPU·HTTP 요청 같은 기본 메트릭은 스프링 부트가 자동으로 수집합니다. 여기에 *비즈니스 메트릭*(주문 수, 결제 금액 등)을 직접 등록하려면 메트릭 세 타입을 알아야 합니다.

## 2. Counter — 단조 증가하는 누적값

Counter는 *증가만 하는* 누적 카운터입니다. 주문 건수, 처리한 요청 수, 발생한 에러 수처럼 "지금까지 몇 번 일어났는가"를 셉니다. 줄어들지 않습니다.

```java
@Service
public class OrderService {
    private final Counter orderCounter;

    public OrderService(MeterRegistry registry) {
        this.orderCounter = Counter.builder("my.order.count")
            .description("주문 건수")
            .tag("type", "normal")     // 태그로 차원 구분
            .register(registry);
    }

    public void order() {
        orderCounter.increment();   // 1 증가
        // 주문 로직
    }
}
```

`MeterRegistry` 는 마이크로미터가 제공하는 메트릭 등록소 빈입니다. 주입받아 `Counter` 를 등록합니다. `tag` 는 메트릭에 차원을 더합니다. 같은 `my.order.count` 라도 `type=normal` 과 `type=cancel` 로 나눠 세면, 나중에 그라파나에서 정상/취소 주문을 분리해 볼 수 있습니다.

### @Counted — 어노테이션으로 간결하게

매번 `MeterRegistry` 를 주입해 빌드하는 대신, `@Counted` 어노테이션으로 메서드 호출 횟수를 셀 수 있습니다.

```java
@Counted("my.order.count")
public void order() {
    // 메서드가 호출될 때마다 카운터 증가 (AOP 프록시)
}
```

`@Counted` 는 [`../../11_spring/05_aop/`](../../11_spring/05_aop/)의 AOP 프록시로 동작합니다. 메서드를 가로채 호출 횟수를 센 뒤 원래 로직을 실행합니다. 측정 코드가 비즈니스 코드에 섞이지 않는다는 장점이 있지만, 세밀한 제어(조건부 증가 등)는 안 되므로 단순 호출 카운트에 적합합니다.

## 3. Timer — 시간과 횟수를 함께

Timer는 *실행 시간*을 측정합니다. 그런데 단순히 시간만 재는 게 아니라 호출 횟수, 총 소요 시간, 최대 시간을 함께 기록합니다. API 응답 시간, 외부 호출 지연 같은 "얼마나 걸렸는가"에 씁니다.

```java
@Service
public class OrderService {
    private final Timer orderTimer;

    public OrderService(MeterRegistry registry) {
        this.orderTimer = Timer.builder("my.order.timer")
            .description("주문 처리 시간")
            .register(registry);
    }

    public void order() {
        orderTimer.record(() -> {
            // 이 블록의 실행 시간이 측정된다
            processOrder();
        });
    }
}
```

`record()` 가 감싼 블록의 실행 시간을 재고, 동시에 호출 횟수도 1 증가시킵니다. 그래서 Timer 하나로 "평균 처리 시간"(총 시간 ÷ 호출 수)을 얻을 수 있습니다. Counter가 횟수만 센다면 Timer는 *횟수 + 시간*을 함께 잡는 셈입니다.

```java
@Timed("my.order.timer")   // 어노테이션 방식
public void order() {
    // 메서드 실행 시간 자동 측정
}
```

`@Timed` 역시 `@Counted` 처럼 AOP로 동작합니다.

## 4. Gauge — 오르내리는 현재값

Gauge는 *현재 상태값*을 측정합니다. Counter가 누적이라면 Gauge는 스냅샷입니다. 큐에 쌓인 작업 수, 현재 활성 커넥션 수, 캐시 크기처럼 오르락내리락하는 값에 씁니다.

```java
@Configuration
public class GaugeConfig {

    @Bean
    public MyQueue myQueue(MeterRegistry registry) {
        MyQueue queue = new MyQueue();
        // 큐의 현재 크기를 게이지로 관측
        Gauge.builder("my.queue.size", queue, MyQueue::getSize)
            .description("대기 큐 크기")
            .register(registry);
        return queue;
    }
}
```

Gauge는 `increment()` 로 직접 올리지 않습니다. 대신 *측정 대상 객체와 그 값을 읽는 함수*를 등록해 두면, 메트릭을 조회하는 순간 마이크로미터가 그 함수를 호출해 현재값을 읽습니다. 그래서 큐 크기가 늘든 줄든 항상 *그 시점의 실제 값*이 잡힙니다.

## 5. 세 타입 선택 기준

| 타입 | 측정 대상 | 특징 | 예 |
|------|----------|------|-----|
| Counter | 누적 횟수 | 증가만, 감소 없음 | 주문 수·에러 수·요청 수 |
| Timer | 실행 시간 + 횟수 | 시간과 호출 수 동시 | API 응답 시간·외부 호출 지연 |
| Gauge | 현재 상태값 | 오르내림, 조회 시점 값 | 큐 크기·활성 커넥션·캐시 크기 |

선택의 기준은 한 문장으로 정리됩니다. "지금까지 몇 번?"이면 Counter, "얼마나 걸렸나?"면 Timer, "지금 얼마인가?"면 Gauge입니다. 이 세 타입으로 대부분의 비즈니스 메트릭을 표현할 수 있습니다.

## 6. 정리

| 주제 | 핵심 |
|------|------|
| 마이크로미터 | 측정 추상화, 모니터링 도구 교체에도 측정 코드 불변 |
| MeterRegistry | 메트릭 등록소 빈, 주입받아 메트릭 생성 |
| Counter | 단조 증가 누적값, `@Counted` 로 간결화 |
| Timer | 실행 시간 + 호출 횟수, `@Timed` 로 간결화 |
| Gauge | 현재 상태값, 값 읽는 함수 등록 |
| 태그 | 같은 메트릭을 차원으로 분리 (type=normal/cancel) |
