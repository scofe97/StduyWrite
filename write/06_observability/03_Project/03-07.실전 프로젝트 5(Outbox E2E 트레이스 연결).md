# Outbox E2E 트레이스 연결 — message-lib · executor · operator

---

> Outbox 패턴 위에 OpenTelemetry W3C `traceparent`를 흘려 “HTTP 요청 → DB 저장 → Kafka 발행 → 다른 컨슈머의 처리”까지 단일 trace ID로 묶는 작업이다.

TPS 프로젝트의 `message-lib`는 outbox 발행 시점에 현재 span의 traceparent를 캡처해 `TB_TRB_OX_001.TRACE_PARNTS` 컬럼에 저장한다. `OutboxPoller`는 발행 시 그 값을 다시 복원해 Kafka record header에 `traceparent`를 실어 보낸다. 컨슈머 측은 `CorrelationIdRecordInterceptor`가 `ce_correlationid`를 ThreadLocal에 set 하고, OTel Kafka instrumentation이 `traceparent`를 읽어 새 span의 부모로 잇는다.

이 그림은 message-lib 코드 자체로는 완결되지 않는다. 라이브러리는 “현재 span을 캡처”까지 할 수 있어도 “현재 span을 만든다”는 책임을 지지 않는다. 본 문서는 그 경계를 어디서 어떻게 메우는지를 정리한다.



## 1. message-lib 단독으로 부족한 이유

> `compileOnly`의 비전이성과 OTel API/SDK 분리 구조가 두 단계로 침묵 실패를 만든다.

`message-lib/build.gradle`는 OTel API를 다음과 같이 선언했다.

```groovy
compileOnly 'io.opentelemetry:opentelemetry-api:1.40.0'
```

`compileOnly`로 잡은 의존성은 해당 모듈의 `compileClasspath`에만 들어간다. `runtimeClasspath`에 빠지고 다운스트림 모듈에도 전혀 노출되지 않는다. 즉 message-lib을 의존하는 executor, operator 같은 소비 모듈의 런타임 classpath에는 `opentelemetry-api` JAR이 한 줄도 들어가지 않는다.

`TraceContextUtil`은 정적 초기화에서 `Class.forName("io.opentelemetry.api.trace.Span")`로 API 존재를 직접 검사한다. 클래스가 없으면 `OTEL_AVAILABLE=false`로 고정되어 `captureTraceParent()`는 영구적으로 null을 반환한다. 이게 1차 침묵 실패다 — 빌드는 통과하고 런타임 예외도 안 나며, 그저 `TRACE_PARNTS` 컬럼만 영원히 빈다.

소비 모듈에서 `runtimeOnly 'io.opentelemetry:opentelemetry-api'`를 추가해 1차 함정을 피해도 두 번째가 기다린다. OpenTelemetry는 인터페이스(API)와 구현(SDK)을 분리한다. API만 있으면 `Span.current()`는 NoOp 구현인 `Span.getInvalid()`를 반환하고, 그 `SpanContext.isValid()`는 항상 false라 캡처 코드는 또 null을 반환한다. 결국 두 조건이 모두 충족되어야 traceparent가 채워진다.

- API JAR이 소비 모듈 런타임 classpath에 보일 것
- 진입점에서 누군가가 실제 span을 시작해 현재 컨텍스트에 넣어줄 것 (자동 계측)

라이브러리가 `implementation`으로 SDK를 강제 전이하는 방법도 있으나, OTel을 안 쓰는 소비자에게도 의존성을 떠안기게 되어 “선택 의존성” 설계 의도가 깨진다. 또한 SDK만 있다고 자동 계측이 켜지지도 않는다 — 진입점 instrumentation은 본질적으로 애플리케이션 모듈의 책임이다.

> 같은 `compileOnly` 함정은 [01_language/java/06_Build/02-01.Gradle 의존성 키워드.md](../01_language/java/06_Build/02-01.Gradle%20의존성%20키워드.md) §9에 Gradle 키워드 관점에서 별도로 다뤘다.



## 2. 소비 모듈에 starter 부착 — 실제 적용 내용

> executor/engine, operator/app 두 boot 모듈에 OTel starter와 logging exporter를 추가한다.

```groovy
// executor/engine/build.gradle, operator/app/build.gradle 공통

def otelInstrumentationVersion = '2.10.0'

// Spring Boot 3.2.3은 OTel API/SDK를 1.31.0으로 핀해 두므로, instrumentation 2.10.0이 기대하는
// 1.44.x 라인을 쓰려면 BOM을 dependencyManagement 로 import 해 Spring의 핀을 덮어써야 한다.
dependencyManagement {
    imports {
        mavenBom "io.opentelemetry.instrumentation:opentelemetry-instrumentation-bom:${otelInstrumentationVersion}"
    }
}

dependencies {
    implementation 'io.opentelemetry.instrumentation:opentelemetry-spring-boot-starter'
    runtimeOnly 'io.opentelemetry:opentelemetry-exporter-logging'
}
```

`io.spring.dependency-management` 플러그인은 Spring Boot가 관리하는 의존성 버전을 우선 적용한다. Spring Boot 3.2.3 BOM은 OTel 1.31.0을 핀해 두므로, `implementation platform(...)` 만으로는 1.44.x 라인을 가져오지 못한다. `dependencyManagement.imports.mavenBom`으로 OTel BOM을 Spring BOM 뒤에 import 해야 OTel 1.44.1이 적용된다. 이 점이 일반 멀티모듈에서 Spring Boot + OTel 조합의 함정이다.

Tempo Collector가 아직 없는 단계라 exporter는 logging으로 시작한다.

```yaml
# executor/engine/src/main/resources/application.yml
# operator/app/src/main/resources/application.yml
otel:
  service:
    name: ${spring.application.name}
  traces:
    exporter: logging
  metrics:
    exporter: none
  logs:
    exporter: none
```

검증은 `./gradlew :engine:dependencies --configuration runtimeClasspath`로 실행한다. `opentelemetry-api:1.44.1`, `opentelemetry-sdk:1.44.1`, `opentelemetry-exporter-logging:1.44.1`이 보이면 BOM 덮어쓰기가 작동한 것이다.



## 3. logging exporter는 “그냥 로그만 남기는 것”인가

> 답은 맞기도 하고 틀리기도 하다 — 로그로 남는 건 사실인데, 그 안에 trace 데이터의 구조가 그대로 들어간다.

`opentelemetry-exporter-logging`은 SDK의 `SpanExporter` 인터페이스 구현체 중 하나다. 다른 exporter들과 동일한 데이터 흐름을 따른다 — `BatchSpanProcessor`가 끝난 span을 모아 5초마다 flush 시도하고, OTLP exporter가 받으면 gRPC로 Tempo에 보낸다. logging exporter가 받으면 SLF4J logger(`io.opentelemetry.exporter.logging.LoggingSpanExporter`)에 INFO 레벨로 한 줄씩 찍는다.

출력 예시:

```text
INFO  i.o.e.l.LoggingSpanExporter -- 'OutboxPoller.publish' :
  7c3a8f1b2e4d5a6c0a1b2c3d4e5f6789  9f8e7d6c5b4a3210
  INTERNAL [tracer: outbox-poller:]
  AttributesMap{data={outbox.event.id=OX_20260507_00029, outbox.aggregate.id=JEX_20260507_00010}}
```

여기에 담긴 정보를 풀어 보면 다음과 같다.

- `'OutboxPoller.publish'` — span 이름
- `7c3a8f1b...6789` — 32자리 traceId (16바이트 hex)
- `9f8e7d6c...3210` — 16자리 spanId (8바이트 hex)
- `INTERNAL` — span kind (SERVER, CLIENT, PRODUCER 등도 같은 자리에)
- `AttributesMap{...}` — `setAttribute`로 붙인 모든 key-value

즉 trace 데이터의 **구조 전체가 stdout 로그에 직렬화**된다. 다만 Grafana Tempo 같은 백엔드처럼 검색·시각화·집계는 못 한다. 한 trace 안의 여러 span을 시간순으로 잇거나, 부모-자식 관계를 그래프로 그리거나, “지난 1시간 p95 latency”를 계산할 수가 없다 — 그건 Tempo의 일이다.

logging exporter의 실제 용도는 두 가지다.

- 개발 환경에서 “자동 계측이 실제로 span을 만드는지”를 빠르게 확인. Tempo 없이도 코드 경로 검증 가능.
- 운영에서 OTLP 장애 시 임시 fallback. 환경변수 한 줄로 전환.

`message-lib`의 `TRACE_PARNTS` 컬럼이 채워지는지 여부는 logging이든 OTLP든 무관하다. exporter는 “다 만들어진 span을 어디로 내보내느냐”의 마지막 단계만 담당한다. span 생성과 컨텍스트 전파는 그 앞 단계에서 SDK + 자동 계측이 이미 끝낸 일이고, `message-lib`는 그 “현재 span”을 캡처해 DB에 적는 별개의 흐름이다.

그래서 현재 단계에서도 다음은 모두 동작한다.

- HTTP 요청이 오면 Servlet instrumentation이 root span을 만든다.
- `EventPublisher.publish()`가 호출되면 `captureTraceParent()`가 그 span의 traceparent를 잡아 outbox row에 저장한다.
- `OutboxPoller`가 row를 읽어 traceparent를 복원하고, 자기 자신도 새 child span(`OutboxPoller.publish`)을 만들어 그 안에서 `kafkaTemplate.send`를 호출한다.
- Kafka record header에 `traceparent`가 실리고, 컨슈머 측은 OTel Kafka instrumentation이 그걸 읽어 다음 span을 잇는다.
- 모든 span이 stdout에 한 줄씩 찍힌다.

차이는 단 하나 — **여러 서비스의 로그를 traceId로 묶어 보는 화면이 없다**. Tempo가 그 빈자리다.



## 4. Tempo 도입 시 변경 사항

> 핵심은 exporter 전환과 엔드포인트 주입 두 가지뿐. 코드는 손대지 않는다.

### 4.1 application.yml 또는 환경변수

yml 방식:

```yaml
otel:
  traces:
    exporter: otlp
  exporter:
    otlp:
      endpoint: http://tempo.observability.svc.cluster.local:4317
      protocol: grpc            # http/protobuf 사용 시 4318 포트
      timeout: 10s
      compression: gzip
```

환경변수 방식이 환경별 분리에 유리하다.

```bash
OTEL_TRACES_EXPORTER=otlp
OTEL_EXPORTER_OTLP_ENDPOINT=http://tempo.observability.svc.cluster.local:4317
OTEL_EXPORTER_OTLP_PROTOCOL=grpc
```

OTel SDK는 환경변수를 yml보다 우선 적용하므로, 같은 이미지를 dev/staging/prod에 배포하면서 환경변수만 바꿔 exporter를 갈아끼울 수 있다. logging exporter 의존성은 제거하지 않고 그대로 두는 편이 안전하다 — 운영 중 OTLP가 막히면 `OTEL_TRACES_EXPORTER=logging`으로 임시 fallback 가능하다.

### 4.2 의존성 — 추가할 것 없음

`opentelemetry-spring-boot-starter`는 OTLP exporter(`opentelemetry-exporter-otlp`)를 BOM transitive로 이미 가져온다. `dependencies` 보고서에 1.44.1로 잡혀 있는 것을 확인했다. 별도 의존성 추가 없이 yml/env 변경만으로 OTLP가 활성화된다.

### 4.3 sampling 전략

기본값은 `parentbased_always_on`이다. 부모가 샘플링되면 따라가고, root는 100% 샘플링한다. 트래픽이 많아지면 root 비율을 낮춘다.

```bash
OTEL_TRACES_SAMPLER=parentbased_traceidratio
OTEL_TRACES_SAMPLER_ARG=0.1
```

10%로 낮춰도 한 trace 안의 모든 span은 함께 샘플링되거나 함께 드롭되므로 추적의 일관성은 유지된다. (`parentbased_*` 정책의 본질)

### 4.4 Tempo 측 구성 — 인프라 영역

본 코드 변경 범위 밖이지만 짚어 둔다.

- Tempo가 OTLP receiver를 4317(gRPC) / 4318(HTTP) 포트로 노출
- Helm chart 또는 ArgoCD manifest에서 `receivers.otlp.protocols.grpc.endpoint: 0.0.0.0:4317` 설정
- 일반적으로 Tempo 단독보다 **OTel Collector를 앞단에 두고** Tempo로 forward하는 패턴이 권장된다 — batch, retry, attribute 가공, 라우팅을 분리할 수 있다.

상세는 [02-04.Grafana Tempo.md](02-04.Grafana%20Tempo.md)와 [05-01.Tempo 분산 트레이싱 시각화.md](05-01.Tempo%20분산%20트레이싱%20시각화.md)에 분리되어 있다.



## 5. Tempo 없이도 앱이 정상 동작하는가

> 동작한다. 추적 데이터가 외부로 안 나가고 stdout 로그로만 남을 뿐이다.

logging exporter는 외부 네트워크 호출을 일으키지 않는다. `OTEL_EXPORTER_OTLP_ENDPOINT` 환경변수가 비어 있어도 `otel.traces.exporter=logging`이면 OTLP exporter는 시도조차 하지 않는다. 따라서 Tempo가 없어도 앱은 타임아웃, 재시도, 연결 실패 같은 부작용이 일체 없다.

OTLP로 전환한 상태에서 Tempo가 죽으면 별도 시나리오가 된다. `BatchSpanProcessor`가 span을 메모리 버퍼(기본 2048개)에 모아 5초마다 flush 시도하고, export 실패 시 span을 드롭한다(재시도 횟수 제한, 디스크 저장 없음). 핵심은 **exporter가 비동기 처리라 hot path를 차단하지 않는다**는 점이다 — 앱의 HTTP 응답, Kafka 발행, DB 트랜잭션은 영향을 받지 않는다. 실패 로그가 주기적으로 찍히므로 모니터링 알림 트리거로 쓸 수 있다.

이건 OTel의 핵심 설계 원칙이다. **Observability shouldn't impact application** — 관측 인프라가 죽어도 앱은 살아 있어야 한다. 그래서 Tempo 도입을 “장애 허용 여유 있게” 진행할 수 있다. dev에서 OTLP로 먼저 켜고, 한동안 로그 부피와 메트릭을 본 뒤, prod로 옮기는 순서가 자연스럽다.

부팅 단계 실패는 한 가지 주의할 점이다. `otel.traces.exporter=otlp`인데 endpoint URL 형식이 잘못되어 있으면(스킴 누락, 포트 누락) Spring Boot 시작 시 `OpenTelemetrySdk` 빈 생성 단계에서 예외가 날 수 있다. dev 환경에서 한 번 부팅을 검증하면 전환 리스크가 사실상 없다.



## 6. 단계별 운영 흐름

> 한 줄 코드 변경 없이 yml/env로 모든 단계를 옮길 수 있다.

| 단계 | 상태 | 설정 |
|------|------|------|
| 현재 (PoC) | logging exporter | yml `otel.traces.exporter: logging` |
| Tempo 도입 후 dev | OTLP, 100% 샘플링 | env `OTEL_TRACES_EXPORTER=otlp`, `OTEL_EXPORTER_OTLP_ENDPOINT=http://tempo:4317` |
| 운영 전환 | OTLP, 비율 샘플링 | 위 + `OTEL_TRACES_SAMPLER=parentbased_traceidratio`, `OTEL_TRACES_SAMPLER_ARG=0.1` |
| Tempo 장애 시 fallback | logging | env `OTEL_TRACES_EXPORTER=logging` (재기동 또는 actuator refresh) |

이 흐름은 OTel “collector-first” 설계의 직접적인 결과다. exporter는 끝단의 어댑터일 뿐이고, 그 위 단의 SDK·자동 계측·`BatchSpanProcessor`는 동일하게 돌아간다. 한 번 깔아두면 백엔드를 바꾸는 일은 환경변수 작업이다.
