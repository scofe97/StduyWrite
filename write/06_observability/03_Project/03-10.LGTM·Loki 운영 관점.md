# LGTM·Loki 운영 관점 (305P 사례 보강)

---

> 같은 LGTM 스택이라도 어떻게 설정했느냐에 따라 운영 함정이 다르게 노출된다. 본 문서는 일반론 ([02-03.Grafana Loki.md](02-03.Grafana%20Loki.md) 등)이 다루지 않는 305P 특화 발견을 사실로 적는다. 다음 클러스터에 LGTM을 새로 올릴 때 같은 함정에 다시 빠지지 않도록 한다.

## 1. Loki 라벨 폭발(label explosion)

> traceId를 라벨로 넣으면 Loki가 죽는다. 305P에서 실제로 일어난 일은 아니지만, 본 절을 박제해 미래에 일어나지 않도록 한다.

Loki는 라벨 조합마다 별도 스트림(chunk 세트)을 만든다. 라벨 조합 수가 만 단위로 늘어나면 인덱스가 메모리를 잠식하고, 어느 순간 ingester가 OOM으로 죽는다. 이 현상을 라벨 카디널리티 폭발이라고 부른다.

305P Loki는 현재 안전 구간에 있다. `namespace × app × container × pod` 조합이 수십 개 수준이고, `traceId`나 `request_id` 같은 고-cardinality 값을 라벨로 노출하지 않았기 때문이다. 다만 운영자가 새 dashboard나 알람을 짤 때 "이 값을 라벨로 빼면 쿼리가 빨라질 텐데"라는 유혹에 빠지기 쉽다. 그 유혹이 들어왔을 때 본 절을 다시 본다.

라벨로 둘 만한 값과 라벨에 두면 안 되는 값을 구분하는 기준은 "값의 종류가 유한한가"다. 다음 표는 305P 환경에서 안전한 라벨과 위험한 라벨의 예시다.

| 안전 | 위험 |
|------|------|
| `namespace=trb-app` (4종) | `traceId=abcd...` (요청마다 다름) |
| `app=operator-api` (10여 종) | `userId=12345` (사용자마다 다름) |
| `container=istio-proxy` (제한적) | `pipelineExecutionId=...` (실행마다 다름) |
| `level=ERROR` (4~5종) | `request_id=...` (요청마다 다름) |

[02-03.Grafana Loki.md](02-03.Grafana%20Loki.md)에 라벨 cardinality 관리 원칙이 더 정리돼 있다.

## 2. 7일 보존 정책의 의미

> 305P Loki의 retention은 7일이다. 알람에는 영향이 없지만 사후 조사에는 한계가 있다는 점을 운영자가 알고 있어야 한다.

`loki-stack/values-ppp.yaml`의 `loki.config.limits_config.retention_period`가 `168h`로 설정돼 있다. 수집 시점 기준 7일이 지난 청크는 MinIO에서 삭제되고 더 이상 조회되지 않는다.

이 정책이 운영에 미치는 영향은 다음 두 갈래다. 알람은 실시간 평가라 5분 윈도우가 7일을 넘지 않으므로 영향이 없다. 반면 사후 장애 분석은 7일 안에 끝나야 한다. 8일 전에 발생한 ERROR 라인의 본문은 더 이상 Loki에서 볼 수 없고, MinIO 버킷에 백업 정책이 따로 없으면 영구 손실이다.

장기 보관이 필요하면 두 가지 선택지가 있다. `retention_period`를 늘려 Loki 자체에 더 오래 두는 안은 MinIO 비용이 함께 증가한다. 별도 콜드 스토리지로 옮기는 안은 운영 부담이 크다. 305P 본 시점에는 7일을 그대로 두고, "장애 보고서를 7일 안에 작성한다"는 운영 규약으로 운영한다.

## 3. k8s-sidecar autoload 메커니즘

> ConfigMap 라벨 한 줄로 dashboard가 자동 등록되는 마법은 sidecar 컨테이너 두 개가 만든 결과다. 305P에는 이 가운데 일부만 켜져 있다는 사실이 본 시리즈 작업 방향을 바꿨다.

`prometheus-grafana` Pod에는 세 개의 컨테이너가 있다. `grafana` 본체와 `grafana-sc-dashboard`, `grafana-sc-datasources` 두 sidecar다. 두 sidecar는 모두 `kiwigrid/k8s-sidecar` 이미지에서 출발하고, 환경변수로 어떤 라벨의 ConfigMap을 감시할지, 어디에 파일을 쓸지, Grafana에 reload 요청을 어떻게 보낼지를 결정한다.

```
LABEL=grafana_dashboard      LABEL_VALUE="1"   FOLDER=/tmp/dashboards          REQ_URL=http://localhost:3000/api/admin/provisioning/dashboards/reload
LABEL=grafana_datasource     LABEL_VALUE="1"   FOLDER=/etc/grafana/.../datasources  REQ_URL=.../datasources/reload
```

여기서 본 시리즈가 활용한 경로는 `grafana_dashboard=1` 한 줄이다. 이 라벨이 붙은 ConfigMap의 `data` 필드 안 JSON 본문을 sidecar가 파일로 떨구면, Grafana provisioning이 폴더 변경을 감지해 dashboard로 임포트한다.

여기서 빠진 sidecar가 두 종이다. `grafana-sc-alerting` 또는 `grafana-sc-notifications`이 활성화돼 있어야 alert rule이나 contact point ConfigMap이 같은 방식으로 자동 주입된다. 305P Grafana는 두 sidecar가 없어, 알람을 Grafana Unified Alerting으로 가져가려면 Helm release values를 수정해 sidecar를 추가해야 한다. 이 결정의 비용은 [06-01.305P 관측 운영기록.md](06-01.305P%20관측%20운영기록.md) §8에서 다룬다.

## 4. Grafana datasource UID 비일관

> dashboard JSON에 datasource를 어떻게 적느냐가 환경 간 portability를 결정한다.

305P Loki datasource의 UID는 `P8E80F9AEF21F6940`이다. Grafana가 datasource를 처음 생성할 때 자동으로 부여하는 무작위 값이라, 다른 클러스터에 같은 datasource를 만들어도 UID가 다르다. dashboard JSON에 이 UID를 박아 두면 환경 이관 시 매번 검색·치환을 해야 한다.

본 시리즈 dashboard는 datasource를 `"Loki"` legacy string으로 지정해서 이 문제를 우회했다. Grafana 11 기준 `"datasource": "Loki"` 표기는 name 기반 lookup으로 동작하고, datasource 이름은 helm values에서 결정한다. 305P loki-stack 차트는 `name: Loki`로 datasource를 등록하므로 UID 의존성을 끊을 수 있다.

UID 표기가 권장되는 경우도 있다. 같은 환경 안에서 Grafana datasource를 두 개(예: Loki, Loki-RW) 운영하면 이름이 같은 두 datasource가 생길 수 있고, 그때는 UID로 명시해야 모호함이 사라진다. 305P 본 시점에는 datasource가 종류당 하나라 legacy string이 안전하다.

## 5. Alloy → Loki 파이프라인

> 305P Alloy DaemonSet은 컨테이너 stdout을 그대로 Loki로 보낸다. ANSI escape 같은 노이즈를 라벨 직전에 정리할 여지가 있지만 본 시점에는 손대지 않았다.

Alloy는 305P 모든 워커 노드에 1 Pod씩 떠 있다(8 Pod). 노드의 컨테이너 stdout 로그 파일을 직접 tail하고, K8s 메타데이터로 라벨을 붙인 뒤 Loki HTTP push로 전송한다. 이 흐름의 중간에 `loki.process` stage를 끼우면 정규식 치환·라벨 추가·drop 같은 가벼운 변환을 할 수 있다.

가능한 개선 가운데 305P에 가장 어울리는 후보는 ANSI escape 제거다. operator-api stream에서 `[1;31m`, `[0;39m` 같은 sequence를 미리 stripping하면 dashboard의 logs 패널 가독성이 올라가고 LogQL이 `mERROR` 같은 우회 패턴 없이 평이한 `ERROR` 매칭으로 단순해진다. 그러나 이 변환은 Alloy 설정 변경이라 재기동이 동반되고, 본 시리즈 범위에서는 의도적으로 분리했다.

## 6. 알람 인프라 선택지 비교

> 305P에서 ERROR 알람을 보내려면 어떤 길이 있는가. 결정 흐름을 한 곳에 모은다.

세 가지 길이 있다. 본 시리즈 작업 시점에 사용자와 함께 결정한 결과는 두 번째 길(Alertmanager + Loki Ruler)이다.

**Grafana Unified Alerting (Grafana Managed Alert)**. Grafana 자체에 alert rule을 만들고 contact point로 Slack을 연결하는 길이다. LogQL을 그대로 평가하고 Grafana UI에서 룰을 보고 관리하는 일관성이 매력이다. 그러나 305P Grafana는 GitOps 비관리 상태고 alerting sidecar도 비활성이라 "룰을 ConfigMap으로 자동 주입"하는 경로가 막힌다. UI로 수동 입력하면 자동화 요구에서 어긋나고, sidecar를 켜려면 Grafana Helm release 재배포가 필요하다.

**Alertmanager + Loki Ruler**. Loki 자체에 rule을 두고(`loki.config.ruler`) 평가 결과를 Alertmanager로 보내 Slack receiver까지 라우팅한다. 305P에 Alertmanager가 없으므로 새 chart로 띄워야 하지만, Grafana 차트는 건드리지 않는다. 위험이 격리되는 길이다.

**Prometheus Operator의 Alertmanager subchart 활성화**. `kube-prometheus-stack`이 이미 떠 있고 alertmanager subchart는 내부에 있다. 켜기만 하면 된다. 다만 이 차트가 GitOps 비관리라 손대는 자체가 부담이다. 차트 GitOps 이관이 별 티켓에서 끝나면 다시 검토할 가치는 있다.

세 길의 핵심 차이는 "어떤 컴포넌트의 재배포를 받아들이느냐"다. 305P 본 시점은 Loki의 ConfigMap 변경(rolling update)과 Alertmanager 신규 배포만 허용하기로 결정했다.

## 7. 운영자 체크리스트

> 새 클러스터에 LGTM을 올리는 운영자가 한눈에 점검할 항목.

| 점검 | 통과 기준 |
|------|----------|
| Loki ConfigMap 이름 | `helm list -A`로 release name 확인 (305P는 `loki-loki-stack`) |
| Grafana sidecar | `kubectl -n trb-mgm get deploy prometheus-grafana -o yaml | grep "name: LABEL"`로 활성 라벨 카탈로그 확인 |
| Alertmanager 존재 | `kubectl get svc -A | grep alertmanager` 결과 1건 이상 |
| GitOps 관리 범위 | `kubectl get applications -A`로 ArgoCD가 보는 차트와 helm release 목록 비교 |
| datasource UID | dashboard JSON에 박힌 UID가 환경마다 다른지 확인. 다르다면 name 기반으로 교체 |
| retention | `limits_config.retention_period`와 실제 보관 정책이 일치하는지 |

305P 본 시점에는 위 표의 1, 3, 4번이 "예상과 달랐던" 발견이었다. 본 문서는 그 발견을 박제하는 게 목적이다.
