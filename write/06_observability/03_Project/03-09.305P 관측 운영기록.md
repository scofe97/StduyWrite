# 305P 관측 운영기록

---

> 305P 개발계(`dev.trombone-v2.okestro.cloud`)에서 operator/executor ERROR 로그를 보기 위해 실제로 한 일과 발견한 사실을 한 곳에 정리한다. 다음 세션 또는 다른 사람이 이어갈 때 "지금까지 뭐가 됐고 뭐가 안 됐는지"를 PR/Wiki를 뒤지지 않고 이 문서에서 확인할 수 있게 한다.

## 1. 클러스터 현황

> 운영자가 가장 먼저 알아야 하는 사실들. 어떤 네임스페이스에 무엇이 떠 있는지, 어떤 차트로 관리되는지를 본 절에 박제한다.

305P는 Trombone-v2 위에 별도 구축된 K8s 클러스터다. 모니터링 스택은 `trb-mgm` 네임스페이스에 모여 있고, 애플리케이션은 `trb-app`에 있다. DevOps 미들웨어는 `trb-oss`에 있고 ArgoCD도 같은 네임스페이스에서 동작한다.

| 컴포넌트 | 위치 | 출처 |
|---------|------|------|
| Loki 3.1.0 | `trb-mgm/loki-0` | helm release `loki` (`loki-stack-2.10.0`), ConfigMap `loki-loki-stack` |
| Grafana 11.4.0 | `trb-mgm/prometheus-grafana-*` | helm release `prometheus` (`kube-prometheus-stack-68.1.0`) |
| Alloy | `trb-mgm/alloy-*` (DaemonSet, 8 Pod) | helm release `alloy` |
| Tempo | `trb-mgm/tempo-*` | helm release `tempo` |
| MinIO | `trb-oss/minio-*` | Loki·Tempo의 S3 backend |
| Alertmanager | **부재** | `kube-prometheus-stack`의 alertmanager subchart는 비활성, 별도 chart 없음 |
| Prometheus | `trb-mgm/prometheus-prometheus-kube-prometheus-prometheus-0` | helm release `prometheus`(같은 차트) |

이 사실 가운데 운영에 가장 영향이 큰 한 줄은 **Alertmanager가 클러스터에 없다**는 점이다. 로그 ERROR 알람을 보내고 싶다면 알람 라우팅 인프라부터 구축해야 한다. 이 문서를 읽는 사람이 "그냥 룰만 짜면 되겠지"라고 가정하지 않도록 가장 앞에 둔다.

GitOps 관점에서도 비대칭이 있다. `loki/alloy/tempo`는 `argocd-apps/app-of-apps/charts/trb-mgm`에서 자식 Application으로 sync된다. 반면 `kube-prometheus-stack`은 ArgoCD가 관리하지 않고 수동 `helm install`로 떠 있다. 차트 변경이 곧 운영 위험으로 직결되므로, Grafana 자체에 손을 대는 작업은 본 시리즈 범위에서 의도적으로 분리했다.

## 2. Loki 라벨 카탈로그

> dashboard와 알람 LogQL을 정확하게 짜려면 어떤 라벨이 살아 있는지부터 알아야 한다. 305P Loki에서 실측한 결과를 박제한다.

`/loki/api/v1/labels`로 받은 라벨 목록은 다음과 같다.

```
app, component, container, filename, instance, job, namespace, node_name, pod, service_name, stream
```

이 가운데 매처에 자주 쓰는 항목은 `namespace`, `app`, `container`, `pod`다. `traceparent`나 `traceId`를 라벨로 노출해서는 안 된다. 이유와 함정은 [06-02.LGTM·Loki 운영 관점.md](06-02.LGTM·Loki%20운영%20관점.md) §1에서 다룬다.

`detected_level` 라벨은 결과 stream의 메타로만 나오고 stream selector에서 직접 매처로 쓰지 못한다. 본 세션에서 시도해 본 `{namespace="trb-app",detected_level="error"}`는 0건을 반환한다. 따라서 ERROR 매칭은 line filter 정규식으로 해결해야 한다.

## 3. 로그 포맷 실측

> 정규식 LogQL을 짜기 전, operator와 executor가 실제로 무엇을 stdout에 흘리는지를 직접 봐야 한다. 두 서비스가 같은 회사 같은 팀에서 만들었지만 로그 포맷이 다르다.

operator-api는 ANSI 컬러 코드가 포함된 평문이다. 같은 라인 안에 `[1;31mERROR[0;39m` 같은 escape sequence가 들어 있다. logback `[%X{traceId:-}]` MDC slot은 정의돼 있지만 채우는 코드가 없어 `[]`로 비어 출력된다.

```
2026-05-15 15:06:52.369 [1;31mERROR[0;39m [] [operator-scheduler-4] [36mo.s.s.s.TaskUtils$LoggingErrorHandler[0;39m - Unexpected error ...
```

executor는 ANSI 컬러가 없고 대신 대괄호 포맷을 쓴다. `traceId`와 `spanId` MDC slot이 둘 다 존재하지만 역시 비어 있다.

```
2026-05-15 16:00:50 [ERROR] [org.springframework.kafka.KafkaListenerEndpointContainer#1-0-C-1] [traceId=] [spanId=] o.o.t.a.a.deploy.DeployService - [ArgoDeploy] queue processing failed. published FAIL ...
```

JSON 구조화 로그는 둘 다 적용되지 않았다. `| json` parser를 LogQL에 쓰지 못하므로 line filter 정규식으로 가야 한다. JSON 도입 시 LogQL이 얼마나 단순해지는지는 [06-03.305P OpenTelemetry 도입 고려사항.md](06-03.305P%20OpenTelemetry%20도입%20고려사항.md) §8에서 다룬다.

## 4. ERROR 매칭 LogQL 확정

> 두 포맷을 한 정규식으로 잡되 logger 이름의 "Error" 단어로 인한 false positive는 피하는 균형점.

확정된 LogQL은 다음과 같다.

```logql
{namespace="trb-app", app=~"operator-api|executor", container=~"operator-api|executor"} |~ `mERROR|\[ERROR\]`
```

세 가지 결정이 들어 있다.

`container=~"operator-api|executor"`로 istio-proxy 사이드카 컨테이너를 제외한다. 305P는 `trb-app` 네임스페이스의 모든 Pod에 istio-proxy가 사이드카로 붙기 때문에 같은 `app` 라벨로 두 스트림이 잡힌다. 컨테이너 이름이 같은 매처를 통과하는지를 한 번 더 검증해야 한다.

`mERROR`는 operator-api의 ANSI escape에 둘러싸인 `[1;31mERROR[0;39m`를 잡기 위한 패턴이다. 컬러 sequence가 끝나는 `m` 직후 `ERROR`라는 위치적 단서를 활용한다. logger 이름에 `Error`가 들어가도 (`LoggingErrorHandler` 같은 클래스명) 그 앞에 `m`이 붙지 않아 false positive를 피한다.

`\[ERROR\]`는 executor의 `[ERROR]` 대괄호 포맷을 잡는다. operator-api의 stacktrace 라인은 ANSI도 대괄호도 없어 본 매처에서 빠진다. stacktrace 본체는 ERROR 라인 직후에 따라오므로 dashboard의 logs 패널에서 timestamp 순으로 함께 보면 된다.

## 5. 대시보드 chart — `helm-charts/monitoring-error-logs/`

> 본 세션의 가시적 산출물. ConfigMap 한 장으로 dashboard를 자동 주입하는 패턴을 따른다.

신규 chart `helm-charts/monitoring-error-logs/`를 만들고 ConfigMap 한 장을 `trb-mgm` 네임스페이스에 떨군다. ConfigMap label `grafana_dashboard=1`이 붙으면 `kiwigrid/k8s-sidecar`가 자동으로 감지해 Grafana provisioning 폴더에 파일을 쓰고, Grafana가 reload한다. Grafana Helm release 자체에는 손을 대지 않는다.

dashboard uid는 `305p-error-logs`다. 패널 구성은 row 3개 + 데이터 패널 3개로 다음과 같다.

| 패널 | 타입 | 역할 |
|------|------|------|
| ERROR count over time (by app) | timeseries | app별 누적 막대로 spike 한눈에 식별 |
| ERROR distribution by Pod | table | 특정 Pod 몰림 감지, 단일 버그 vs 전체 장애 구분 |
| ERROR log stream | logs | 최신 N건 라이브 스트림, 라인 클릭으로 stacktrace 전체 확장 |

datasource는 `"Loki"` legacy string으로 지정했다. 305P Loki datasource UID는 `P8E80F9AEF21F6940`이지만 이는 Grafana가 자동 생성한 값이라 환경마다 다르다. UID 대신 datasource name으로 참조하면 다른 클러스터로 옮길 때 dashboard JSON을 고칠 필요가 없다.

GitOps 연결은 `argocd-apps/app-of-apps/charts/trb-mgm`에 토글 한 줄과 Application template 한 장으로 끝낸다.

```yaml
# argocd-apps/app-of-apps/charts/trb-mgm/values-ppp.yaml
monitoringErrorLogs:
  enabled: true
  helmPath: helm-charts/monitoring-error-logs
```

`templates/monitoringErrorLogs.yaml`은 `tempo.yaml`을 그대로 베껴 토글 변수만 바꿨다. 본 작업으로 `feature/monitoring` 브랜치에 2개 커밋이 push됐고 Bitbucket PR 머지를 대기 중이다.

## 6. 클러스터 검증

> 매니페스트를 ArgoCD가 사용하기 전, 사람이 같은 ConfigMap을 손으로 한 번 apply해 sidecar→Grafana 파이프라인이 동작하는지 확인했다.

확인 절차는 다음 다섯 단계다.

`helm template`으로 ConfigMap을 로컬에 렌더한다. 첫 시도에서 출력이 1137 bytes로 잘려 Grafana provisioning이 `unexpected EOF`를 냈고, 두 번째 시도(4731 bytes)부터 정상 임포트됐다. 잘림 원인은 `head -15`를 중간에 끼워 미리 본 파이프 결과를 그대로 파일로 저장한 운영자 실수였다. 다음에 같은 검증을 할 때는 `wc -c`로 크기를 먼저 확인하는 게 안전하다.

`kubectl apply -f`로 ConfigMap을 `trb-mgm`에 떨군다. 이때 sidecar 로그가 `Writing /tmp/dashboards/305p-error-logs.json (ascii)`를 찍고 Grafana 본체 로그가 `finished to provision dashboards`로 끝나면 임포트 성공이다.

`/api/dashboards/uid/305p-error-logs`를 admin/cloud1234로 호출해 HTTP 200과 6개 패널, uid·title 일치를 확인한다.

`/api/datasources/proxy/uid/<loki-uid>/loki/api/v1/query_range`로 dashboard와 동일한 LogQL을 Grafana 데이터소스 프록시를 통해 실행해, operator-api와 executor 양쪽 stream이 모두 잡히는지 확인한다. 본 세션에서는 2 stream 매치를 확인했다.

마지막은 사용자 검증이다. 본 작업 시점에는 PR이 머지되지 않았고 클러스터에는 수동 apply한 ConfigMap이 남아 있다. PR 머지 후 ArgoCD가 자식 Application을 생성하면 같은 이름의 ConfigMap이 Helm-managed 상태로 바뀌고 라벨이 patch된다. 사용자 결정으로 수동 apply본은 그대로 둔다.

## 7. 잔여 작업과 차단 요인

> 다음에 무엇을 하면 되는지, 그리고 무엇이 막혀 있는지를 사실 기반으로 적는다.

대시보드 자체는 본 세션에서 완결됐다. 그러나 원 요구사항인 "ERROR가 발생하면 사람이 보지 않아도 Slack에 알림이 와야 한다"는 미완이다. 본 세션에서 사전 조사만 끝낸 알람 인프라 구축이 후속 작업이다.

알람 인프라 구축은 다음 사실에 부딪힌다. Grafana 자체에 alerting sidecar가 없다. 305P Grafana는 `grafana-sc-dashboard`와 `grafana-sc-datasources` sidecar만 띄워서 alert rule ConfigMap을 떨궈도 Grafana가 받아 가지 않는다. 결국 두 갈래가 남는다 — Grafana Helm release를 수정해 alerting sidecar를 켜거나, Alertmanager + Loki Ruler를 별도로 띄우는 길이다. 사용자 결정은 후자다. Grafana는 GitOps 미관리 상태라 차트 손대기가 위험하기 때문이다. 자세한 결정 흐름은 [06-02.LGTM·Loki 운영 관점.md](06-02.LGTM·Loki%20운영%20관점.md) §6에서 다룬다.

이외 잔여 작업은 다음 두 가지다. `kube-prometheus-stack`을 GitOps로 이관하는 작업은 차트 user values가 1517줄이라 별 티켓으로 분리됐다. 305P operator·executor의 로그를 JSON으로 구조화하는 작업도 별건이다. 후자가 끝나면 본 문서의 §3·§4가 단순해진다.

## 8. 의사결정 박제

> "왜 그렇게 결정했지?"는 시간이 지나면 잊는다. 두고두고 뒤집힐 만한 갈림길은 본 절에 사실과 함께 박제한다.

**Grafana Managed Alert를 쓰지 않은 이유**. Grafana Unified Alerting은 LogQL을 자체적으로 평가하므로 가장 자연스러운 후보였다. 그러나 305P Grafana는 GitOps 비관리 상태고, alerting sidecar도 비활성이라 "alert rule을 ConfigMap으로 자동 주입"하는 경로가 끊어진다. UI에서 수동 입력하면 drift가 누적되므로 자동화 요구를 충족하지 못한다.

**kube-prometheus-stack 이관을 본 티켓에서 분리한 이유**. `helm get values prometheus` 결과가 1517줄이다. 운영자가 `helm install` 시 차트 default-equivalent를 통째로 넣어 둔 상태라, 진짜 사용자 override만 골라내는 작업이 본 티켓 범위를 초과한다. 이관 자체는 분명 필요하지만 별 세션이 안전하다.

**정규식 매칭을 받아들인 이유**. JSON 구조화 로그가 정공법이다. 그러나 305P operator·executor는 logback 설정이 평문이고, 변경하려면 두 서비스 코드와 logback dependency가 모두 손에 닿아야 한다. 본 티켓 범위는 manifest 레포로 좁혀져 있었다. 정규식이 false positive를 완전히 없애지 못한다는 한계는 받아들이고 후속 JSON 이관을 위한 자료를 남긴다.

**알람 인프라로 Alertmanager + Loki Ruler를 택한 이유**. Grafana alerting sidecar 활성화 안은 Grafana Helm release 재배포를 동반하고, 그 차트는 GitOps 비관리다. 반면 Alertmanager는 신규 chart로 띄우면 기존 컴포넌트와 결합도가 낮고, Loki Ruler 활성화는 `loki-stack/values-ppp.yaml`의 `loki.config.ruler` 블록 추가만으로 끝난다. 위험이 격리되는 쪽을 택했다.
