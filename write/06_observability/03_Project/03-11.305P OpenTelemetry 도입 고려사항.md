# 305P OpenTelemetry 도입 고려사항

---

> 305P는 Logs(Loki) 위주의 관측만 갖고 있다. Metrics는 부분적이고 Traces는 부재다. 이 상태에서 OpenTelemetry를 도입한다면 무엇을 알아야 하고 어떤 결정을 내려야 하는지를 정리한다. 분량을 두려워하지 않는다. 도입 결정은 한 번이고 운영은 오래간다.

## 1. 관측 가능성 3축 — 305P가 어디 있는가

> Logs, Metrics, Traces 세 축의 의미를 먼저 분리하고, 305P의 현 위치를 사실로 적는다. 분리된 채로 보아야 도입의 우선순위가 보인다.

분산 시스템의 관측 가능성은 보통 세 신호(signal) 축으로 본다.

| 축 | 의미 | 대표 도구 | 305P 현황 |
|----|------|----------|----------|
| Logs | "무엇이 일어났는가"의 사건 기록 | Loki, Elasticsearch | Loki 3.1.0 정상 운영, 7일 보존 |
| Metrics | "얼마나 자주·얼마나 많이"의 수치 | Prometheus, Mimir | kube-prometheus-stack 떠 있으나 GitOps 비관리, 애플리케이션 커스텀 메트릭은 미정의 |
| Traces | "한 요청이 시스템을 어떻게 흘렀나"의 경로 | Tempo, Jaeger | Tempo chart는 있으나 애플리케이션이 trace를 생성하지 않아 사실상 부재 |

운영 초기에는 로그만으로도 어느 정도 버틴다. 그러나 305P처럼 operator → Kafka → executor → operator → DB로 연결되는 EDA 구조에서는 "어느 서비스에서 느려졌는가" 또는 "이 ERROR는 어느 요청에서 시작됐는가"를 로그만으로 추적하기 어렵다. 같은 요청에 속한 라인끼리 묶어 줄 식별자가 없기 때문이다. 305P logback에 `[%X{traceId:-}]` slot이 이미 있고 비어 있는 이유는 이 슬롯을 채울 trace 생성 주체가 없어서다. OTel을 도입하는 가장 큰 동기가 여기에 있다.

## 2. OpenTelemetry의 본질

> OTel은 "또 다른 모니터링 도구"가 아니다. 신호를 만들고 옮기는 표준이다. 이 차이를 놓치면 Tempo/Loki/Prometheus와의 관계를 잘못 이해한다.

OpenTelemetry는 신호(signal)를 생성·수집·전송하는 방법에 대한 공통 규약과 그 규약을 구현한 SDK·Collector 묶음이다. 어디에 저장할지를 결정하지 않는다. 다음 그림이 본 절의 요지다.

```
Application
   ↓ (OTel SDK or Java Agent)
OTel Collector
   ↓ (exporter)
Backend
  ├─ Loki    (logs)
  ├─ Tempo   (traces)
  └─ Mimir/Prometheus (metrics)
```

Kafka에 비유하면 OTel Collector는 메시지 브로커다. 애플리케이션(producer)이 SDK로 telemetry를 만들어 Collector로 보내고, Collector는 처리·필터링·라우팅을 거쳐 백엔드(consumer)로 내보낸다. 305P는 이미 Kafka 운영 경험이 있으므로 이 비유로 Collector의 위치를 빠르게 잡을 수 있다.

305P의 LGTM 스택은 OTel과 충돌하지 않는다. Loki는 OTel logs exporter의 표준 수신처이고, Tempo는 OTel traces exporter의 1차 수신처다. Alloy 자체가 OTel Collector 코드 위에 Grafana 컴포넌트 모델을 얹어 만든 수집기다([02-02.Grafana Alloy.md](02-02.Grafana%20Alloy.md) 참조). 즉 305P는 OTel을 받기 좋은 형태로 이미 준비돼 있다. 빠진 건 신호를 만드는 쪽이다.

## 3. Kafka Trace 전파 — 305P가 풀어야 할 핵심

> HTTP만 쓰는 서비스라면 OTel Java Agent를 부착하는 것만으로 trace가 끝까지 이어진다. Kafka가 사이에 끼면 비동기 경계에서 흐름이 끊긴다. 305P는 이 끊김을 어떻게 다루느냐가 도입의 핵심이다.

HTTP 요청은 header가 응답 흐름과 함께 자연스럽게 전달된다. 클라이언트가 `traceparent: 00-<trace-id>-<span-id>-01` 헤더를 보내면 서버 측 OTel SDK가 받아 자신의 새 span을 같은 trace 안에 잇는다. 같은 trace ID가 끝까지 흘러서 Tempo에서 한 요청의 전체 경로를 본다.

Kafka는 다르다. Producer가 메시지를 보내고 Consumer가 받는 사이에 응답이 없고, 헤더가 자동으로 이어지지도 않는다. 그래서 명시적으로 trace context를 메시지 헤더에 실어 보내야 한다. OTel은 이를 위한 context propagator 표준을 정의한다. Kafka 헤더에 `traceparent` 값을 직접 쓰는 방식이다.

```
Producer (Service A)                       Consumer (Service B)
─────────────────                          ─────────────────
HTTP req → Span A                          poll() → Kafka headers
  ↓ KafkaTemplate.send(msg)                  ↓ extract `traceparent`
  ↓ injector writes `traceparent` header     ↓ new Span B as child of A
Kafka topic                                  ↓ business logic
─────────────────                            ↓ DB call → Span C
                                           DB
```

305P는 이 흐름의 절반이 이미 자체 구현으로 들어와 있다. `message-lib/CorrelationIdRecordInterceptor`는 inbound Kafka record에서 `ce_correlationid` 헤더(CloudEvents 표준 키)를 읽어 ThreadLocal `CorrelationIdContext`에 적재한다. 같은 스레드에서 후속 `EventPublisher`가 발행할 때 다시 헤더로 자동 반사된다. 즉 trace ID에 해당하는 값은 이미 Kafka 경계를 넘어 흐른다.

격차는 두 가지다. 첫째, 헤더 키가 `ce_correlationid`(CloudEvents)와 `traceparent`(W3C/OTel) 둘로 갈린다. OTel SDK는 후자만 읽는다. 둘째, ThreadLocal에 들어간 값을 logback MDC에 채우는 다리가 없어서 로그에 trace ID가 비어 보인다.

도입 시 결정 사항이다. **단순 정렬 안**은 `ce_correlationid`를 유지하되 OTel propagator를 커스텀하여 같은 헤더를 읽고 쓰게 만든다. CloudEvents 호환성을 깨지 않는다. **표준 이행 안**은 `traceparent`를 추가로 쓰고 receiver는 두 헤더를 모두 읽는다. 점진 이행 동안 양립한다. 305P가 후속에서 어느 길을 택할지는 message-lib 오너의 결정이다.

## 4. Span 모델과 Batch Consume 함정

> trace 안의 한 단위 작업을 span이라 부른다. Kafka consumer 환경에서는 "한 poll vs 한 record"를 어떻게 span으로 자르느냐가 분석 가능성을 결정한다.

trace 한 개는 여러 span의 그래프다. 각 span은 시작/종료 시각, 부모 span, 라벨(attribute)을 가진다. HTTP 서비스에서는 보통 1 요청 = 1 root span으로 떨어지지만 Kafka는 다르다.

Spring Kafka는 기본적으로 `poll()` 단위로 컨테이너 스레드가 깨고, 한 번의 poll로 N개 record를 받아 순차 처리한다. span을 어디에 둘지에 따라 분석 가능성이 달라진다.

| 전략 | span 구성 | 트레이드오프 |
|------|----------|------------|
| poll 단위 1 span | poll → process N records | span 수 적음, 개별 record 분석 어려움 |
| record 단위 N span | poll → record1 span → record2 span ... | 개별 분석 가능, span 수 N배 |
| 절충: poll span + record child | poll(상위) → record(자식 N개) | 묶음과 개별 둘 다 보이지만 비용은 N+1 |

305P의 executor는 Kafka 메시지를 받아 운영 파이프라인을 실행한다. 한 record 당 처리 시간이 길고 종류가 다양해 "어느 record가 느렸나"가 중요하다. 따라서 record 단위 span이 정공법이다. 다만 record가 빠르게 흐르는 메시지(예: 상태 ack)라면 batch span으로 묶어 비용을 줄이는 게 합리적이다. 메시지별로 정책이 달라야 한다는 사실이 핵심이다.

함정은 `poll`과 `record processing`의 시간 척도가 다르다는 점이다. poll lag는 컨슈머가 얼마나 뒤처졌는가를 말하고, processing latency는 한 record가 얼마나 오래 잡혔는가를 말한다. trace는 후자만 보여 준다. 즉 trace만으로 "왜 lag가 늘었나"를 답하지 못한다. lag는 Kafka 메트릭이 따로 답해야 한다(§13에서 이어 다룬다).

## 5. K8s ephemeral 철학과 로그 수집

> Pod는 죽는다. 그 전제 위에서 로그 수집 구조를 짜야 한다. 305P가 Alloy DaemonSet을 채택한 이유가 본 절에 들어 있다.

K8s에서는 Pod 수명이 짧다. 노드가 빠지거나 deployment가 갱신되면 Pod가 사라지고 그 안의 파일은 함께 증발한다. 따라서 로그를 Pod 안에 파일로 쓰는 전통적 모델은 위험하다. 표준은 stdout/stderr로 흘려 보내고, 호스트 노드의 컨테이너 런타임이 그 출력을 일정한 위치에 모아 두고, 노드 레벨 에이전트가 그 출력을 외부 스토리지로 옮기는 구조다.

```
container stdout
   ↓
kubelet/containerd writes to /var/log/pods/...
   ↓
Node-level Agent (DaemonSet)
   ↓ remote write
Loki / Tempo / Mimir
```

305P는 이 구조의 표준 위에 있다. Alloy를 DaemonSet으로 띄워 모든 워커 노드(8 Pod)에 한 개씩 배치하고, 노드의 컨테이너 stdout 로그 파일을 tail해 Loki에 push한다. 본 설계가 OTel과 잘 맞는다. Alloy 자체가 OTel Collector 기반이라 같은 DaemonSet에서 logs·metrics·traces를 함께 받을 수 있다. 즉 OTel 도입 시 새 에이전트를 추가할 필요 없이 Alloy 설정 확장만으로 출발할 수 있다.

## 6. Loki 핵심 철학과 라벨 설계

> Loki는 "라벨로 후보를 좁히고 본문을 grep"하는 모델이다. trace ID는 라벨이 아닌 본문에 둬야 한다. 본 절은 OTel 도입과 함께 흔히 어기는 약속을 박제한다.

Loki는 Elasticsearch와 다르다. Elasticsearch가 본문을 다 인덱싱한다면, Loki는 라벨만 인덱싱하고 본문은 압축 청크로 저장한다. 라벨 조합이 곧 chunk 식별자가 되므로 라벨 수가 폭발하면 인덱스 자체가 메모리를 잠식한다([06-02.LGTM·Loki 운영 관점.md](06-02.LGTM·Loki%20운영%20관점.md) §1 참조).

OTel을 도입하면 trace ID·span ID·userId 같은 고-cardinality 값이 함께 들어온다. 이를 라벨로 노출하지 말고 본문의 JSON 필드로 둔다. LogQL `| json | trace_id="abc..."` 같이 파싱·필터를 본문에서 한다. trace ID를 매번 찾는 비용은 라벨 인덱싱이 아닌 chunk 스캔으로 지불해야 안전하다.

이 결정은 OTel logs exporter 설정에 직접 영향을 준다. exporter는 OTel attributes 가운데 어느 것을 Loki 라벨로 승격할지 정한다. 후보 attributes는 다음 표와 같다.

| Attribute | 라벨 승격 | 이유 |
|-----------|----------|------|
| `service.name` | 승격 | 값 종류 유한, 자주 쓰는 매처 |
| `k8s.namespace.name` | 승격 | 종류 유한 |
| `k8s.pod.name` | 승격 가능 | 종류 늘지만 Pod 수 한계 안 |
| `trace_id`, `span_id` | 본문 유지 | 매 요청마다 다름 |
| `user.id`, `tenant.id` | 본문 유지 | 사용자 수만큼 늘어남 |
| `http.target` | 본문 유지 | URL path까지 들어가면 폭발 |

305P 적용 시 기본값은 `service.name = operator-api|executor`만 라벨로 두고 나머지는 본문 JSON으로 두는 것이다.

## 7. 305P stdout → Alloy → Loki 흐름

> 본 절은 305P의 현재 파이프라인을 그림과 함께 박제하고, OTel 도입 후 어디가 어떻게 바뀌는지를 짚는다.

현재 흐름은 다음과 같다.

```
operator-api / executor Pod
   ↓ logback CONSOLE appender (평문, ANSI color 또는 [ERROR] 대괄호)
   ↓ stdout
container runtime → /var/log/pods/<ns>_<pod>_<uid>/<container>/*.log
   ↓
Alloy DaemonSet (tail)
   ↓ K8s metadata enrichment → labels {namespace, app, container, pod, ...}
   ↓ loki.write
Loki ingester
```

OTel logs를 도입하면 두 가지 보강이 자연스럽다. logback이 logback-logstash-encoder로 JSON 출력으로 바뀌면 Alloy `loki.process` stage에서 `| json` parser 없이 본문 검색이 가능해진다. 추가로 OTel SDK가 MDC에 `trace_id`, `span_id`를 자동으로 채워 주므로 logback 패턴에 그대로 흘러 들어간다. 결과적으로 모든 ERROR 라인에 trace ID가 박혀 Tempo에서 같은 trace의 다른 span으로 점프할 수 있다. 이 점프가 Grafana에서 "Trace ↔ Log" 연결의 본질이다.

## 8. JSON 구조화 로그 — 305P가 가야 할 곳

> ERROR 매칭이 정규식 `mERROR|\[ERROR\]`인 이유는 로그가 평문이기 때문이다. JSON으로 가면 LogQL이 단순해지고 동시에 OTel 도입 비용이 줄어든다.

305P operator·executor는 logback 기본 ConsoleAppender를 쓴다. 결과는 사람이 보기 좋은 평문이지만 기계가 파싱하기 까다롭다. logstash-logback-encoder를 적용하면 같은 라인이 다음과 같이 바뀐다.

```json
{
  "@timestamp": "2026-05-15T16:00:50.123Z",
  "level": "ERROR",
  "thread_name": "kafka-listener-1-0-C-1",
  "logger_name": "o.o.t.a.a.deploy.DeployService",
  "message": "[ArgoDeploy] queue processing failed",
  "trace_id": "...",
  "span_id": "...",
  "mdc": { "ce_correlationid": "..." }
}
```

LogQL이 다음 형태로 평이해진다.

```logql
{namespace="trb-app", app=~"operator-api|executor"} | json | level="ERROR"
```

정규식 매칭의 두 문제(operator-api ANSI escape, executor 대괄호 차이)가 모두 해소된다. 또한 OTel SDK 부착 시 `trace_id`·`span_id` MDC가 자동 채워지므로 같은 라인에 곧장 들어간다. 즉 JSON 도입은 OTel 도입과 짝이 잘 맞는다.

비용은 operator·executor 소스 변경이다. `build.gradle`에 dependency 1줄, `logback-spring.xml`에 encoder 교체 5줄 수준이지만, 두 서비스의 ConsoleAppender를 동시에 바꿔야 운영 일관성이 유지된다. 점진 이행 시 한 서비스만 JSON으로 바뀌면 dashboard·LogQL이 둘 다 깨지는 위험이 있다. 한 번에 전환하는 게 정공법이다.

## 9. MDC traceId 채우기 — 305P가 거의 다 와 있는 부분

> logback의 `[%X{traceId}]` slot은 이미 있다. 채울 코드가 없을 뿐이다. OTel Java Agent 부착으로 이 한 줄이 활성화된다.

operator-api와 executor의 logback-spring.xml은 다음 패턴을 이미 갖고 있다.

```
operator-api: %d %level [%X{traceId:-}] [%thread] %logger - %msg
executor:     %d [%level] [%thread] [traceId=%X{traceId:-}] [spanId=%X{spanId:-}] %logger - %msg
```

`%X{traceId:-}`는 logback MDC에서 `traceId` 키를 찾고, 없으면 빈 문자열로 대체한다는 뜻이다. 본 시점에는 MDC에 값을 넣어 주는 코드가 없어 항상 빈 값이다.

OTel Java Agent를 부착하면 다음이 자동으로 일어난다. SDK가 HTTP/Kafka span을 생성할 때 현재 thread의 MDC에 `trace_id`·`span_id`를 set한다(`io.opentelemetry.instrumentation.logback-mdc-1.0` 모듈). 즉 코드 변경 없이 `[%X{traceId}]` slot에 값이 차기 시작한다.

operator-api logback은 `traceId` 키를 쓰는데 OTel SDK 기본은 `trace_id`(스네이크)다. 키 이름을 맞춰야 자동 연동된다. 두 가지 길이다. logback 패턴을 `%X{trace_id:-}`로 바꾸거나, `otel.instrumentation.logback-mdc.add-baggage=true` 같은 옵션으로 매핑한다. 305P에는 operator/executor가 키 표기가 이미 다르므로(`traceId` vs traceId/spanId), 차라리 OTel 표준 키로 양쪽 통일하는 게 깔끔하다.

## 10. OTel Collector 중앙화 vs 직접 export

> 애플리케이션이 곧장 Loki/Tempo로 보낼지, Collector를 통해 보낼지의 결정이다. 305P 폐쇄망 운영 이관 가정 아래에서 답이 정해진다.

초기에는 애플리케이션 SDK가 곧장 백엔드로 보내는 단순한 그림이 매력적이다. 그러나 운영이 커지면 다음 필요가 생긴다.

- 인증 정보를 어디에 둘 것인가 (모든 앱에 토큰 배포 vs 중앙 한 곳)
- sampling 정책을 어디서 결정할 것인가 (앱마다 vs 한 곳에서 일괄)
- 배치·재시도·필터링을 어디서 할 것인가
- 백엔드 장애 시 큐잉을 어디서 할 것인가

답은 Collector를 중간에 두는 것이다. 305P는 Alloy DaemonSet이 이미 노드마다 있으므로 1차 Collector 역할을 그대로 한다. 그 위에 중앙 Collector(Deployment, 2~3 Pod)를 두면 tail sampling·테넌트 분리·재시도 같은 무거운 처리를 모은다.

```
App (OTel SDK)
   ↓ OTLP gRPC
Alloy DaemonSet (1차 Collector, 노드별)
   ↓ OTLP gRPC
Central Collector (Deployment, 2~3 Pod)
   ↓
Backend (Loki/Tempo/Mimir)
```

305P가 운영 이관 대상으로 가정한 BOK 등 폐쇄망 환경에서는 백엔드까지의 네트워크 경로가 제한적이다. 중앙 Collector에서 모든 외부 송신을 통제하면 방화벽 정책이 단순해지고, 백엔드 주소 변경 시 앱을 재배포할 필요가 없다. 폐쇄망일수록 중앙화의 가치가 커진다.

## 11. Sampling 전략

> 모든 trace를 저장하면 비용이 폭증한다. 어떤 trace를 버릴 것인가의 결정이 sampling이다. 305P 트래픽 수준에서 어느 전략이 합리적인지 본 절에서 다룬다.

세 가지 전략이 표준으로 정착돼 있다.

| 전략 | 동작 | 장점 | 단점 |
|------|------|------|------|
| Always On | 모든 trace 저장 | 분석 손실 0 | 비용 폭증, 스토리지 부담 |
| Head Sampling (Probabilistic) | trace 시작 시 N% 확률로 결정 | 단순, 비용 예측 가능 | 장애 trace가 N% 안에 안 들면 영영 못 봄 |
| Tail Sampling | trace 종료 후 평가, 조건부 보관 | 느린·실패한 trace만 보관 | Collector 메모리·복잡도 증가 |

305P 본 시점에는 traces가 0건이므로 첫 단계는 Always On으로 시작해 부담을 보면서 결정하는 게 안전하다. 트래픽이 커지면 Tail Sampling으로 옮긴다. Tail Sampling의 핵심은 다음 두 줄 정책이다.

```
- 오류가 포함된 trace는 항상 저장
- p95 이상 느린 trace는 항상 저장
- 그 외 정상 trace는 일부(5~10%)만 표본 저장
```

이 정책은 "장애는 느린 요청에 숨는다"는 관찰을 코드화한 것이다. Always On에서 Tail Sampling으로 옮길 때 Collector 메모리가 trace 보관 기간(보통 30초~수분) 동안 모든 span을 쥐고 있어야 한다. 305P 트래픽이 분당 수천 trace로 커지면 중앙 Collector의 메모리 예약을 미리 계산해야 한다.

Head Sampling은 305P에 권장하지 않는다. 305P operator의 요청 수가 분당 수십~수백 수준이라 비용 압박이 작은 반면, 장애 발생 시 trace 손실은 치명적이다.

## 12. Kafka Retry/DLT와 Trace 설계

> 305P `KafkaRetryTemplateConfig`는 `@RetryableTopic` 기반이다. retry-1/retry-2/dlt로 재발행되는 흐름을 trace로 어떻게 묶을 것인가가 본 절의 질문이다.

`@RetryableTopic`은 실패한 record를 `topic-retry-0/-1/...` 토픽에 재발행하고, 최대 재시도 후에도 실패하면 `topic-dlt`로 보낸다. trace 관점에서 두 갈래의 선택지가 있다.

**같은 trace로 잇기**. retry 발행 시 원본 record의 `traceparent`를 그대로 헤더에 실어 다음 consumer가 같은 trace의 자식 span으로 받는다. 한 비즈니스 흐름 = 한 trace라는 직관에 맞는다. 그러나 retry 3회를 거치며 같은 trace에 4 span이 줄줄이 매달려 trace 시각화가 길어진다.

**새 trace로 끊기**. retry 발행 시 새 trace ID를 부여하고 원본 trace를 `link`로만 참조한다. 각 trace가 짧아 시각화는 깔끔하지만 "원래 어디서 시작됐나"를 보려면 link를 따라가야 한다.

305P 권장은 첫 번째 길이다. retry 횟수가 보통 2~3회로 한정되고, 한 흐름의 전모를 한 trace에서 보는 운영 편의가 크다. span attribute로 다음 값을 함께 박는다.

```
messaging.kafka.retry_count = 2
messaging.kafka.original_topic = "trb.events.deploy"
messaging.kafka.dlt_topic = "trb.events.deploy-dlt"
```

이 attribute가 있으면 dashboard에서 "DLT로 갔는데 원본 토픽이 무엇이었나"를 한 번에 답할 수 있다.

## 13. Reactor / WebFlux Context Propagation

> 305P가 WebFlux를 쓰지 않는다면 본 절은 미래 대비로 짧게만 안다. 다만 함정의 깊이를 미리 알아 두면 도입 시 비용 추정이 정확해진다.

전통적 Spring MVC는 1 요청 = 1 thread다. 같은 thread의 ThreadLocal에 trace context를 두면 끝까지 따라간다. 305P의 logback MDC도 같은 전제로 동작한다.

Reactor/WebFlux는 다르다. 한 요청이 여러 스케줄러 thread를 갈아탄다. ThreadLocal 기반 context는 thread가 바뀌는 순간 사라진다. OTel은 이 문제를 위해 Context Propagation API와 Reactor용 instrumentation을 별도로 제공한다.

305P가 WebFlux로 옮긴다면 본 절을 다시 봐야 한다. 핵심 결정은 두 줄이다. `Mono`/`Flux` 체인 안에서 trace context를 어떻게 운반할 것인가, 그리고 `subscribeOn`/`publishOn`으로 thread를 바꿀 때 context가 따라가게 할 OTel 헬퍼를 어디에 둘 것인가. 본 시점에는 305P가 MVC 모델이므로 본 절은 표시만 해 둔다.

## 14. 자주 생기는 실무 함정

> 본 절은 도입 후 1년 안에 마주칠 가능성이 높은 함정을 박제한다. 함정을 미리 알면 도입 의사결정의 깊이가 달라진다.

**trace 끊김**. 가장 빈번한 사고. 원인은 보통 다섯 가지다. async executor에서 context가 빠짐, Kafka header 누락, 사용자 정의 thread pool, Reactor 같은 reactive 환경, 외부 라이브러리 직접 호출이다. 도입 직후 trace가 "왜 여기서 끊기지?" 하면 본 다섯 가지부터 점검한다.

**로그 중복 수집**. Alloy DaemonSet과 별도 sidecar agent가 같은 노드의 같은 파일을 둘 다 수집하는 사고. 비용이 2배가 되고 dashboard 카운트가 두 배로 보인다. 새 agent를 추가할 때 항상 "이 노드에 이미 뭐가 뜨는가"를 확인한다.

**high cardinality**. trace_id·user_id를 Loki 라벨로 노출하면 Loki ingester가 죽는다([06-02](06-02.LGTM·Loki%20운영%20관점.md) §1). OTel attributes를 라벨로 승격할 때 화이트리스트로 가는 게 안전하다.

**span 폭증**. Kafka batch consume에서 record 단위 span을 만들면 분당 수만 span이 생긴다. Tail Sampling 없이는 Tempo 저장소가 빠르게 채워진다. §4·§11과 짝지어 정책을 정한다.

**Collector 장애**. 중앙 Collector가 죽으면 telemetry가 통째로 유실된다. memory queue, retry, dead letter exporter 설정이 들어가야 안전망이 생긴다. 305P 도입 시 중앙 Collector를 반드시 2 Pod 이상으로, 그리고 memory_limiter processor를 활성화한다.

## 15. 305P 도입 단계별 로드맵

> 도입은 한 번에 모든 걸 켜는 일이 아니다. 단계마다 비용·위험·이득을 따로 잡는다.

**Phase 0 — 현 logback MDC를 그대로 활용하는 준비 단계**. 코드 변경 0. operator/executor의 logback `traceId` slot 키 표기를 OTel 표준(`trace_id`)으로 통일하고, JSON encoder 적용 PR을 별 티켓으로 분리해 둔다. 이득은 0 → 1 도입 시점에 LogQL이 즉시 단순해진다는 점이다.

**Phase 1 — OTel Java Agent 부착**. 운영 배포 시 `-javaagent:opentelemetry-javaagent.jar`를 jvmOpts에 추가하고, 환경변수로 OTLP endpoint(Alloy DaemonSet의 4317 포트)를 가리킨다. 코드 변경 0. 위험은 agent가 hot path를 우회 instrument해 latency가 1~3% 늘어날 수 있다는 점이다. 이득은 HTTP·Kafka·JDBC trace가 즉시 잡힌다는 점, 그리고 logback MDC가 자동 채워진다는 점이다.

**Phase 2 — OTel Collector 중앙화**. Alloy DaemonSet 외에 중앙 Collector Deployment를 띄운다. tail sampling·테넌트 분리·재시도 같은 처리를 중앙에서 결정한다. 위험은 Collector 자체의 OPS 부담. 이득은 폐쇄망 이관 시 외부 송신 경로가 한 곳으로 모인다는 점.

**Phase 3 — Tempo backend 정상화**. 305P Tempo는 차트만 있고 실 송신이 없다. Phase 1·2 결과를 Tempo로 받기 시작하고, Grafana에서 Trace → Log drill-down을 dashboard에 연결한다. [02-04.Grafana Tempo.md](02-04.Grafana%20Tempo.md), [05-01.Tempo 분산 트레이싱 시각화.md](05-01.Tempo%20분산%20트레이싱%20시각화.md) 참조.

**Phase 4 — Tail Sampling**. 트래픽이 커진 후 중앙 Collector에 tail_sampling processor를 켠다. 정책은 §11 그대로다. 위험은 Collector 메모리. 이득은 비용이 trace 양과 무관해진다는 점.

| Phase | 코드 변경 | 인프라 변경 | 주요 위험 |
|-------|----------|-------------|----------|
| 0 | logback dependency·encoder | 없음 | ConsoleAppender 깨짐 |
| 1 | jvmOpts 추가 | OTel Agent jar 배포 | latency 1~3% 증가 |
| 2 | 없음 | 중앙 Collector 신설 | Collector OPS 부담 |
| 3 | 없음 | Tempo 연결 | Tempo 저장 비용 |
| 4 | 없음 | tail sampling processor | Collector 메모리 |

이 다섯 단계를 한 번에 가지 말고, 단계 사이에 운영 안정화 1~2주를 둔다. 함정 §14의 첫 신호가 보이면 그 단계를 끝내지 않고 멈춰 디버깅한다.
