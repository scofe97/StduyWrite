# 로그 대시보드 설계 — 레벨 구성·분리·레이아웃



---

> Grafana 대시보드를 구성하는 일반 개념(패널 유형·Dashboard JSON 모델·변수·프로비저닝)은 [02-01.Grafana Core.md](../02_LGTMStack/02-01.Grafana%20Core.md) §3·§6이 정본입니다. 이 문서는 그 위에서 **로그를 다루는 대시보드를 실제로 설계할 때 부딪히는 판단** — 로그를 수치로 바꾸는 LogQL 구성, 로그 라인에서 그룹 축을 만드는 법, 대상별로 대시보드를 나눌지 합칠지, 패널 레이아웃과 테이블 정리 — 을 개념으로 정리합니다. 설명을 돕는 예제는 305P operator/executor 로그 대시보드 작업에서 가져옵니다.



## 1. 로그 대시보드가 메트릭 대시보드와 다른 점

> 메트릭은 이미 숫자라 바로 그리면 되지만, 로그는 텍스트 줄이라 "세고·거르고·읽는" 세 관점을 따로 설계해야 합니다.

메트릭 대시보드는 Prometheus가 이미 수치로 만들어 둔 시계열을 그대로 시각화합니다. 반면 로그는 시간이 찍힌 텍스트 줄의 흐름이라, 같은 데이터를 세 가지 다른 방식으로 보게 됩니다. 추세를 보려면 줄 수를 세어 시계열로 바꾸고, 분포를 보려면 대상별로 묶어 집계하며, 원인을 보려면 줄 자체를 그대로 읽어야 합니다.

이 세 관점은 대시보드에서 보통 세 종류의 패널로 나뉩니다. 추세는 timeseries, 분포는 table, 원본 확인은 logs 패널입니다. 같은 LogQL 셀렉터를 공유하되 뒤에 붙이는 집계만 달라집니다.

305P 로그 대시보드도 이 세 구획을 그대로 따릅니다. 레벨별 발생 추세(timeseries), pod·레벨별 분포(table), 라이브 로그 스트림(logs)을 한 화면에 위에서 아래로 쌓아, 추세에서 이상을 발견하면 분포로 범위를 좁히고 스트림에서 실제 줄을 확인하는 동선을 만듭니다.



## 2. LogQL로 로그를 수치화하기

> 로그 줄을 추세 그래프로 만들려면 `count_over_time`으로 줄 수를 세고 `sum by`로 묶습니다.

로그를 시계열로 바꾸는 핵심 함수는 `count_over_time`입니다. 시간 범위를 받아 그 구간에 매칭된 로그 줄 수를 반환하므로, 이를 시간축에 따라 반복하면 발생 추세가 됩니다. 범위 변수로 `[$__interval]`을 쓰면 Grafana가 현재 시간 폭에 맞춰 구간을 자동 조정합니다.

여기에 `sum by (...)`를 씌우면 어떤 축으로 묶어 볼지가 정해집니다. 다음은 305P executor 로그를 레벨별 추세로 만드는 쿼리입니다.

```logql
sum by (level) (
  count_over_time(
    {namespace="trb-app", app=~"executor", container=~"executor"}
      |~ `m(ERROR|WARN|INFO)|\[(ERROR|WARN|INFO)\]`
      | regexp `(?:m|\[)(?P<level>ERROR|WARN|INFO)`
    [$__interval]
  )
)
```

분포 테이블은 같은 골격에서 묶는 축과 범위만 바꿉니다. `sum by (pod, level)`로 pod와 레벨을 함께 묶고, 시간 폭 전체를 한 번에 집계하도록 `[$__range]`와 `instant` 쿼리를 씁니다. 추세는 시간에 따른 변화를, 분포는 현재 범위의 누적을 보기 때문입니다.



## 3. 파생 라벨로 그룹 축 만들기

> 로그 줄 안의 값으로 묶고 싶은데 그게 라벨이 아니라면, `| regexp`로 줄에서 캡처해 임시 라벨로 승격시킵니다.

`sum by (level)`이 동작하려면 `level`이 라벨이어야 합니다. 그런데 로그 레벨은 보통 라벨이 아니라 로그 줄 본문 안에 텍스트로만 들어 있습니다. 이때 `| regexp`로 줄에서 정규식 named capture를 뽑으면, 캡처 그룹 이름이 그대로 파생 라벨이 되어 이후 단계에서 집계 축으로 쓸 수 있습니다.

305P에서는 operator와 executor의 로그 포맷이 달라 이 기법이 특히 필요했습니다. operator-api는 ANSI 컬러 코드에 둘러싸인 `[1;31mERROR[0;39m` 형태이고, executor는 대괄호 `[ERROR]` 형태입니다. 두 포맷에서 레벨 단어를 함께 잡아내는 캡처는 다음과 같습니다.

```logql
| regexp `(?:m|\[)(?P<level>ERROR|WARN|INFO)`
```

`(?:m|\[)`는 레벨 단어 바로 앞에 오는 단서를 비캡처 그룹으로 매칭합니다. 컬러 sequence가 끝나는 `m` 또는 여는 대괄호 `[`입니다. 그 뒤의 `(?P<level>...)`가 실제 레벨을 `level` 라벨로 캡처합니다. 이렇게 하면 포맷이 다른 두 서비스의 로그를 같은 쿼리로 레벨별 집계할 수 있습니다.

라인 필터(`|~`)와 파생 라벨 추출(`| regexp`)을 함께 쓰는 이유가 있습니다. 라인 필터를 먼저 두면 세 레벨에 해당하는 줄만 통과시켜 스캔 대상을 줄이고, 그다음 `| regexp`가 통과한 줄에서만 레벨을 뽑습니다. 순서를 지키면 불필요한 정규식 평가를 줄일 수 있습니다.

파생 라벨과 구분할 개념으로 Loki의 `detected_level`이 있습니다. Loki가 로그를 보고 레벨을 추정해 결과 stream의 메타로 붙여 주지만, 이 값은 stream selector에서 직접 매처로 쓰지 못합니다. 305P에서 `{namespace="trb-app", detected_level="error"}`를 시도하면 0건을 반환하는 이유가 이것입니다. 자세한 라벨 카탈로그는 [03-09.305P 관측 운영기록.md](03-09.305P%20관측%20운영기록.md) §2에서 다룹니다.



## 4. 한 화면에 합칠까, 전용으로 나눌까

> 여러 대상을 변수 하나로 합쳐 한 대시보드에서 전환할지, 대상마다 전용 대시보드로 나눌지는 가독성과 링크 안정성의 트레이드오프입니다.

여러 대상(서비스·환경·인스턴스)을 한 대시보드에서 보는 방법은 두 가지입니다. 하나는 템플릿 변수를 두고 드롭다운으로 대상을 전환하는 통합형이고, 다른 하나는 대상마다 전용 대시보드를 만드는 분리형입니다. 변수 방식의 일반 개념은 [02-01.Grafana Core.md](../02_LGTMStack/02-01.Grafana%20Core.md) §3을 참조합니다.

통합형은 대시보드 수가 적고, 같은 패널 구성을 한 곳에서 관리한다는 장점이 있습니다. 대신 매번 드롭다운을 골라야 하고, 두 대상을 나란히 비교하기 어렵습니다. 분리형은 각 대상의 URL과 즐겨찾기가 고정되어 팀이 특정 서비스 대시보드를 바로 북마크할 수 있고, 대상별로 안내 패널이나 임계값을 다르게 둘 여지가 생깁니다.

305P에서는 operator-api와 executor를 전용 대시보드 두 개로 나눴습니다. 두 서비스가 로그 포맷도 다르고 담당 관점도 달라, 각자의 전용 화면이 운영자에게 더 분명했기 때문입니다. 셀렉터를 `app=~"executor"`와 `app=~"operator-api"`로 각각 고정해 한 화면에는 한 서비스만 들어오게 했습니다.



## 5. 한 차트로 전용 대시보드 여럿 찍기

> 전용 대시보드를 여러 개 만들 때 JSON을 복붙하면 유지보수가 깨집니다. 셀렉터만 변수로 빼면 같은 골격을 대상만 바꿔 재사용합니다.

분리형을 택하면 대시보드가 여러 개로 늘어나는데, JSON을 통째로 복사하면 나중에 패널 하나를 고칠 때 모든 사본을 손대야 합니다. 대신 대상이 다른 부분(셀렉터)만 파라미터로 빼고 골격은 공유하면, 한 번의 수정이 모든 대시보드에 반영됩니다. 305P는 대시보드를 ConfigMap으로 주입하는 Helm 차트 구조라(이 메커니즘은 [03-10.LGTM·Loki 운영 관점.md](03-10.LGTM·Loki%20운영%20관점.md) §3 참조), 셀렉터를 Helm values로 빼는 방식이 자연스러웠습니다.

values에 대상별 셀렉터를 선언합니다.

```yaml
target:
  namespace: trb-app
  executorSelector: "executor"
  operatorSelector: "operator-api"
```

각 대시보드 템플릿은 자기 셀렉터만 참조하므로, executor 대시보드는 `executorSelector`를, operator 대시보드는 `operatorSelector`를 끼워 같은 패널 골격을 렌더합니다. 새 대상이 생기면 values에 한 줄을 더하고 템플릿을 하나 복제하면 됩니다. 패널 구조 변경은 골격 한 곳만 고치면 모든 대상에 전파됩니다.



## 6. 패널 레이아웃과 transformation

> 패널 배치는 `gridPos` 좌표로, 테이블의 자동 컬럼명·불필요한 컬럼 정리는 transformation으로 다듬습니다.

Grafana 패널 위치는 `gridPos`의 네 값으로 정해집니다. 가로 폭 `w`(24분할 기준), 가로 시작 `x`, 세로 시작 `y`, 높이 `h`입니다. 두 패널을 좌우로 두려면 `w`를 12씩 주고 `x`를 0과 12로 나누며, 위아래로 쌓으려면 둘 다 `w`를 24로 주고 `y`를 높이만큼 띄웁니다. 305P에서는 ERROR 전용·WARN 전용 로그 패널을 각각 가로 전체(`w: 24`)로 펼쳐 세로로 적층해, 좁은 반쪽이 아니라 한 줄 전체로 로그를 읽도록 했습니다.

쿼리 결과의 모양을 다듬는 단계가 transformation입니다. 로그 집계 결과는 라벨이 메타로 딸려 오는데, `labelsToFields`로 이 라벨을 테이블 컬럼으로 끌어올립니다. 그러면 `pod`, `level`이 일반 컬럼이 됩니다.

다만 집계 값 컬럼은 Grafana가 쿼리 식별자(refId)를 따 `Value #A` 같은 자동 이름을 붙이고, 의미 없는 `Time` 컬럼도 함께 노출됩니다. `organize` transformation으로 이를 정리합니다.

```json
{ "id": "organize", "options": {
  "excludeByName": { "Time": true },
  "renameByName": { "Value #A": "건수" }
} }
```

`Value #A`는 그 쿼리가 돌려준 값, 즉 해당 `(pod, level)` 묶음의 로그 건수입니다. `renameByName`으로 `건수`처럼 의미 있는 이름을 주고 `excludeByName`으로 `Time`을 숨기면, 테이블이 `pod / level / 건수` 세 컬럼으로 깔끔해집니다. transformation 일반 목록은 Grafana 공식 문서를 참조합니다.



## 7. 레벨 색상 고정

> 레벨별 색을 운영자 직관과 맞추려면 fieldConfig override로 ERROR=red·WARN=yellow·INFO=green을 명시 고정합니다.

레벨별 추세 그래프는 색이 의미와 일치할 때 한눈에 읽힙니다. Grafana는 기본적으로 시리즈에 팔레트 색을 순서대로 배정하므로, 그대로 두면 ERROR가 초록으로 칠해지는 식의 혼란이 생길 수 있습니다. fieldConfig의 `overrides`로 시리즈 이름별 색을 고정하면 이를 막습니다.

```json
"overrides": [
  { "matcher": { "id": "byName", "options": "ERROR" },
    "properties": [{ "id": "color", "value": { "fixedColor": "red", "mode": "fixed" } }] },
  { "matcher": { "id": "byName", "options": "WARN" },
    "properties": [{ "id": "color", "value": { "fixedColor": "yellow", "mode": "fixed" } }] },
  { "matcher": { "id": "byName", "options": "INFO" },
    "properties": [{ "id": "color", "value": { "fixedColor": "green", "mode": "fixed" } }] }
]
```

이 색 선택은 임의가 아니라 로그 원본의 색과 맞춘 것입니다. Spring Boot의 콘솔 로그는 `ColorConverter`가 레벨에 색을 입히는데, 공식 매핑은 ERROR는 빨강, WARN은 노랑, INFO·DEBUG·TRACE는 초록입니다. 세 레벨 모두 ANSI 색이 붙는다는 사실이 §3의 `m(ERROR|WARN|INFO)` 패턴이 성립하는 근거이기도 합니다. 색 escape가 끝나는 `m` 직후에 레벨 단어가 오기 때문입니다.
