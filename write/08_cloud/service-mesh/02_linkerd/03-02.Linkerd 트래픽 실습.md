# Linkerd 트래픽 관리 실습

> 본 장의 개념을 손으로 확인합니다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`linkerd` 범용 명령으로 씁니다.


## 실습 환경
> kubeadm 기반 3노드 GCP K8s 클러스터의 구성과 접속 방법을 안내합니다.


| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| 서비스 메시 | Linkerd (사이드카 주입 네임스페이스) | — |

메시가 주입된 네임스페이스는 `linkerd.io/inject: enabled` 레이블이 붙어 있습니다. 접속은 `gcp` 스킬의 SSH 절차를 따릅니다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1: HTTPRoute 카나리 배포
> emojivoto 앱에서 HTTPRoute로 트래픽을 v1(90%)과 v2(10%)로 분할하는 카나리 배포를 실습합니다.


### 목표

emojivoto 앱에서 emoji-svc의 트래픽을 v1(90%)과 v2(10%)로 분할합니다.

### 사전 준비

```bash
# emojivoto 네임스페이스 생성 및 메시 주입 활성화
kubectl create namespace emojivoto
kubectl annotate namespace emojivoto linkerd.io/inject=enabled

# 앱 배포 (공식 샘플)
curl -sL https://run.linkerd.io/emojivoto.yml | kubectl apply -f -

# 배포 확인
kubectl get pods -n emojivoto
```

### HTTPRoute로 카나리 구성

```bash
# v2 Deployment 생성 (v1 복사본)
kubectl get deployment emoji -n emojivoto -o yaml \
  | sed 's/name: emoji/name: emoji-v2/' \
  | kubectl apply -f -
```

```yaml
# httproute-canary.yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: HTTPRoute
metadata:
  name: emoji-canary
  namespace: emojivoto
spec:
  parentRefs:
    - name: emoji-svc
      kind: Service
      group: ""
      port: 8080
  rules:
    - backendRefs:
        - name: emoji-svc
          port: 8080
          weight: 90
        - name: emoji-svc-v2
          port: 8080
          weight: 10
```

```bash
kubectl apply -f httproute-canary.yaml

# 트래픽 분포 실시간 확인
linkerd viz top -n emojivoto deploy/emoji
```

### 검증 포인트

`linkerd viz top` 출력에서 `emoji-svc`와 `emoji-svc-v2`의 요청 수 비율이 약 9:1인지 확인합니다. 낮은 트래픽 환경에서는 단기 측정값이 설정 비율과 다를 수 있으므로, 충분한 샘플(최소 수백 요청)이 쌓인 후 평가합니다.


## 실습 2: 재시도 예산 확인
> linkerd viz stat으로 Effective RPS와 Actual RPS를 비교해 재시도 발생 여부를 확인합니다.


### 목표

재시도 예산 동작을 메트릭으로 확인합니다.

```bash
# 현재 재시도 관련 메트릭 조회
linkerd viz stat deploy -n emojivoto

# 특정 배포의 상세 메트릭
linkerd viz stat -n emojivoto deploy/web --from deploy/vote-bot
```

`Effective RPS`가 `Actual RPS`보다 높으면 재시도가 발생하고 있다는 신호입니다. 재시도 비율이 20%에 근접하면 재시도 폭풍 위험이 있으므로 원인을 파악해야 합니다.


## 실습 3: EWMA 로드밸런싱 관찰
> 특정 Pod에 지연을 주입하고 EWMA가 다른 Pod로 트래픽을 자동 분산하는지 linkerd viz top으로 확인합니다.


### 목표

특정 Pod의 지연시간이 높을 때 EWMA가 자동으로 다른 Pod로 요청을 분산하는지 확인합니다.

```bash
# emoji Deployment를 3개 replicas로 스케일
kubectl scale deploy emoji -n emojivoto --replicas=3

# Pod별 요청 분포 실시간 관찰
linkerd viz top -n emojivoto deploy/emoji

# 특정 Pod에 인위적 지연 주입 (tc 명령어 사용 예시)
# kubectl exec -it <pod-name> -n emojivoto -- \
#   tc qdisc add dev eth0 root netem delay 200ms
```

EWMA가 올바르게 동작하면 지연이 주입된 Pod로 가는 요청 비율이 감소하고 다른 Pod로 쏠립니다. `linkerd viz top`에서 Pod별 RPS 변화를 관찰하면 확인할 수 있습니다.


## 실습 4: Opaque 포트 설정
> Redis 포트를 Opaque로 지정해 mTLS를 적용하고 L7 메트릭이 수집되지 않음을 확인합니다.


### 목표

비-HTTP 포트(예: Redis 6379)를 Opaque로 지정해 mTLS를 적용합니다.

```yaml
# redis-deployment.yaml (일부)
metadata:
  annotations:
    config.linkerd.io/opaque-ports: "6379"
```

```bash
kubectl apply -f redis-deployment.yaml

# Opaque 설정 확인
linkerd viz edges -n myapp

# TCP 레벨 메트릭 확인 (HTTP 메트릭 없음)
linkerd viz stat -n myapp deploy/redis
```

Opaque 포트로 지정된 서비스는 `linkerd viz stat`에서 HTTP 레벨 메트릭(성공률, 지연시간)이 표시되지 않습니다. TCP 연결 수와 바이트 수만 확인 가능합니다.


## 실습 5: mTLS 상태 확인
> linkerd viz edges로 메시 내 연결의 mTLS 적용 여부를 확인합니다.


```bash
# 네임스페이스 내 모든 연결의 mTLS 상태
linkerd viz edges -n emojivoto

# 특정 배포의 연결 확인
linkerd viz edges -n emojivoto deploy/web
```

출력에서 `SECURED` 컬럼이 `✔`이면 mTLS가 적용된 연결입니다. Ingress Controller 처럼 메시에 속하지 않는 소스에서 오는 연결은 `✘`로 표시됩니다.


## 정리
> 실습에 사용한 리소스를 정리하는 명령어를 제공합니다.


```bash
# 실습 리소스 정리
kubectl delete httproute emoji-canary -n emojivoto
kubectl delete namespace emojivoto
```
