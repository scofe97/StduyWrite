# Linkerd 보안 실습

> 본 장의 개념을 손으로 확인합니다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`linkerd` 범용 명령으로 씁니다.


## 실습 환경
> kubeadm 기반 3노드 GCP K8s 클러스터의 구성과 접속 방법을 안내합니다.


| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| 서비스 메시 | Linkerd (사이드카 주입 네임스페이스) | — |

접속은 `gcp` 스킬의 SSH 절차를 따릅니다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1: 자동 mTLS 확인
> 사이드카 주입만으로 자동 mTLS가 활성화되는지 linkerd viz edges로 확인합니다.


### 목표

사이드카 주입만으로 mTLS가 자동 활성화되는지 확인합니다.

```bash
# 테스트 네임스페이스 생성 (메시 주입 활성화)
kubectl create namespace sec-demo
kubectl annotate namespace sec-demo linkerd.io/inject=enabled

# 간단한 앱 배포
kubectl create deployment web --image=nginx -n sec-demo
kubectl expose deployment web --port=80 -n sec-demo

kubectl create deployment client --image=curlimages/curl \
  -n sec-demo -- sleep infinity

# Pod 준비 대기
kubectl wait --for=condition=ready pod -l app=web -n sec-demo

# mTLS 상태 확인
linkerd viz edges -n sec-demo
```

출력에서 client → web 연결의 `SECURED` 컬럼이 `✔`이면 자동 mTLS가 동작하고 있습니다. 메시 주입 없이 배포했을 때와 비교해보면 차이가 명확합니다.


## 실습 2: Server 리소스와 AuthorizationPolicy
> Server와 MeshTLSAuthentication을 연결해 특정 ServiceAccount에만 접근을 허용하는 정책을 구성합니다.


### 목표

Server를 생성하고 AuthorizationPolicy로 특정 ServiceAccount만 접근을 허용합니다.

### 사전 준비: ServiceAccount 생성

```bash
kubectl create serviceaccount allowed-client -n sec-demo
kubectl create serviceaccount denied-client -n sec-demo
```

### AuthorizationPolicy 먼저 작성 (순서 중요)

```yaml
# mtls-authn.yaml
apiVersion: policy.linkerd.io/v1alpha1
kind: MeshTLSAuthentication
metadata:
  name: allowed-client-authn
  namespace: sec-demo
spec:
  identities:
    - "allowed-client.sec-demo.serviceaccount.identity.linkerd.cluster.local"
---
apiVersion: policy.linkerd.io/v1alpha1
kind: AuthorizationPolicy
metadata:
  name: web-allow-client
  namespace: sec-demo
spec:
  targetRef:
    group: policy.linkerd.io
    kind: Server
    name: web-http
  requiredAuthenticationRefs:
    - name: allowed-client-authn
      kind: MeshTLSAuthentication
      group: policy.linkerd.io
```

```bash
kubectl apply -f mtls-authn.yaml
```

### Server 생성 (AuthorizationPolicy 적용 후)

```yaml
# server.yaml
apiVersion: policy.linkerd.io/v1beta3
kind: Server
metadata:
  name: web-http
  namespace: sec-demo
spec:
  podSelector:
    matchLabels:
      app: web
  port: 80
  proxyProtocol: HTTP/1.1
```

```bash
kubectl apply -f server.yaml

# 접근 테스트: allowed-client ServiceAccount Pod에서
kubectl run allowed-pod \
  --image=curlimages/curl \
  --serviceaccount=allowed-client \
  -n sec-demo -- sleep infinity

kubectl exec -n sec-demo allowed-pod -- curl -s http://web/

# denied-client ServiceAccount Pod에서 (차단 확인)
kubectl run denied-pod \
  --image=curlimages/curl \
  --serviceaccount=denied-client \
  -n sec-demo -- sleep infinity

kubectl exec -n sec-demo denied-pod -- curl -s http://web/
```

`allowed-pod`에서는 정상 응답을 받고, `denied-pod`에서는 연결 거부가 발생해야 합니다.


## 실습 3: default-deny 마이그레이션 시뮬레이션
> 기존 허용 정책에서 default-deny로 안전하게 전환하는 4단계 절차를 따라가며 kubelet 헬스체크 예외 처리를 다룹니다.


### 목표

기존 허용 정책에서 default-deny로 안전하게 전환하는 절차를 따라갑니다.

```bash
# 1단계: 현재 통신 패턴 파악
linkerd viz tap -n sec-demo deploy/web

# 2단계: 정책 위반 메트릭 모니터링 (Server 생성 후)
# Prometheus에서 확인
# policy.linkerd.io/auth_denied_total
---

# 3단계: 헬스 체크 포트 별도 허용 (kubelet은 mTLS 없음)
```

```yaml
# healthcheck-network-authn.yaml
apiVersion: policy.linkerd.io/v1alpha1
kind: NetworkAuthentication
metadata:
  name: kubelet-probe
  namespace: sec-demo
spec:
  networks:
    - cidr: "0.0.0.0/0"   # 실제 환경에서는 노드 IP 범위로 제한
---
apiVersion: policy.linkerd.io/v1alpha1
kind: AuthorizationPolicy
metadata:
  name: web-allow-probe
  namespace: sec-demo
spec:
  targetRef:
    group: policy.linkerd.io
    kind: Server
    name: web-http
  requiredAuthenticationRefs:
    - name: kubelet-probe
      kind: NetworkAuthentication
      group: policy.linkerd.io
```

```bash
kubectl apply -f healthcheck-network-authn.yaml

# Pod의 liveness/readiness probe 상태 확인
kubectl describe pod -l app=web -n sec-demo | grep -A5 "Liveness\|Readiness"
```


## 실습 4: 정책 메트릭 확인
> linkerd viz stat과 tap으로 허용·차단 메트릭을 조회하고 실시간 요청을 관찰합니다.


```bash
# 허용/차단 메트릭 조회 (Prometheus 사용 시)
linkerd viz stat -n sec-demo deploy/web

# tap으로 실시간 요청 관찰
linkerd viz tap -n sec-demo deploy/web --to deploy/client
```
