# Gateway API와 트래픽 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.


## 점검 질문
> Ingress 한계와 Gateway API 복잡성, GAMMA 통일 가능성, 마이그레이션 함정, 역할 분리 적용 규모, 트래픽 관리 안티패턴을 Q&A로 점검한다.


### Q1. Kubernetes Gateway API는 Ingress의 어떤 근본적인 한계를 해결하려 했는가? 그리고 그 해결책이 도입하는 새로운 복잡성은 무엇인가?

Ingress의 근본적 한계는 설계 자체에서 비롯된다. Ingress 스펙은 의도적으로 최소화되었다. 표준 스펙이 HTTP 호스트 기반 라우팅과 경로 기반 라우팅만 정의하다 보니, 모든 컨트롤러는 표현할 수 없는 기능(가중치 기반 라우팅, 헤더 조작, TCP/UDP 라우팅)을 벤더별 어노테이션으로 구현했다. `nginx.ingress.kubernetes.io/canary-weight: "20"`같은 Nginx 전용 어노테이션이 Ingress 리소스를 가득 채우게 됐고, 이 어노테이션은 Traefik이나 Contour로 마이그레이션할 때 전혀 이식되지 않는다. 벤더 락인이 표준 스펙을 통해 이루어지는 아이러니가 발생한 것이다.

Gateway API는 이 문제를 GatewayClass(인프라 공급자 정의), Gateway(리스너와 인증서 설정), HTTPRoute/TCPRoute(라우팅 규칙)의 세 리소스 분리로 해결한다. 이 분리는 역할 기반 관리(클러스터 운영자 vs 애플리케이션 개발자)를 가능하게 하고 라우팅 규칙을 표준 방식으로 표현할 수 있게 한다.

그러나 새로운 복잡성도 따라온다. 세 가지 리소스를 이해하고 올바르게 조합해야 기본 HTTP 라우팅도 동작한다. Ingress에서 단일 리소스로 표현하던 것이 세 리소스로 늘어난다. 소규모 팀이나 단순한 라우팅 요구사항이 있는 경우 이 추가 복잡성은 실질적인 부담이다. 이미 NGINX Ingress나 Traefik Ingress를 안정적으로 운영 중인 팀이라면, 벤더 어노테이션에 의존하는 기능이 3개 이상이거나 여러 팀이 동일 클러스터에서 독립적으로 라우팅 규칙을 관리해야 할 때만 Gateway API 전환이 장기적으로 유리하다.

### Q2. GAMMA 이니셔티브는 어떤 문제를 해결하려 하고, 메시 벤더마다 다른 트래픽 관리 API가 통일될 수 있을까?

GAMMA의 핵심 아이디어는 Gateway API의 HTTPRoute를 동서(East-West, 클러스터 내부 서비스 간) 트래픽 관리에도 사용하는 것이다. Istio는 VirtualService와 DestinationRule, Linkerd는 ServiceProfile과 TrafficSplit, Consul은 ServiceRouter 각자의 독자적인 트래픽 관리 API를 가지고 있어 플랫폼 팀이 특정 메시 벤더에 종속되는 문제가 있었다.

구체적으로 Service를 HTTPRoute의 parentRef로 지정함으로써, 해당 서비스로 향하는 트래픽에 라우팅 규칙을 적용한다. 카나리 배포, 헤더 기반 라우팅, 트래픽 미러링 등이 이 방식으로 표준화될 수 있다. 이것이 성공하면 HTTPRoute 하나로 Istio와 Linkerd 모두를 제어할 수 있게 된다.

그러나 메시마다 지원하는 기능의 범위가 다르다는 도전 과제가 있다. Istio의 VirtualService는 Lua 필터 통합, 복잡한 폴트 인젝션, gRPC 라우팅 등 풍부한 기능을 제공하는 반면 Gateway API의 HTTPRoute는 더 보수적인 스펙을 가진다. 두 시스템을 공통 분모로 수렴시키면 각 메시의 고급 기능은 여전히 벤더 확장으로 남을 수밖에 없다. 현재 시점에서 GAMMA를 프로덕션에 활용하는 것은 이르지만, 새로운 인프라를 설계할 때 HTTPRoute를 기반으로 트래픽 정책을 표현하면 향후 메시 교체 시 마이그레이션 비용을 낮출 수 있다.

### Q3. Ingress에서 Gateway API로, 또는 Istio VirtualService에서 HTTPRoute로 마이그레이션할 때 실제로 어떤 위험과 함정이 존재하는가?

마이그레이션의 주요 함정은 세 가지 범주로 나뉜다.

첫째는 기능 패리티 갭이다. NGINX Ingress의 `nginx.ingress.kubernetes.io/proxy-buffer-size`같은 세밀한 어노테이션은 Gateway API에 직접 대응하는 설정이 없을 수 있다. 마이그레이션 전에 현재 사용 중인 모든 어노테이션을 목록화하고 Gateway API에서 동등한 기능이 있는지 확인해야 한다. 없는 경우 Gateway API 구현체의 커스텀 리소스를 써야 하는데, 그러면 벤더 독립성의 이점이 줄어든다.

둘째는 TLS 인증서 관리 변경이다. Ingress에서는 Secret을 직접 참조했지만 Gateway API에서는 Gateway 리소스가 인증서를 관리한다. 인증서 관리 책임이 클러스터 운영자에게 이동하는데, 기존에 각 팀이 자신의 Ingress에 직접 Secret을 붙이던 패턴은 동작하지 않는다.

셋째는 전환 기간 동작이다. Ingress와 HTTPRoute가 같은 Service를 대상으로 동시에 존재하면, 컨트롤러가 우선순위를 어떻게 결정하는지는 구현체마다 다를 수 있다. 전환 기간에는 트래픽 모니터링을 강화하고, 가능하다면 전용 테스트 도메인을 먼저 마이그레이션해 검증해야 한다. 권장 전략은 "Parallel Run"이다. 새 Gateway API 리소스를 테스트 서브도메인에 먼저 적용하고, 프로덕션 트래픽의 일부를 미러링해 동일한 결과를 내는지 검증한다.

### Q4. Gateway API의 역할 기반 설계는 소규모 팀에서 오히려 부담이 될 수 있는가? 역할 분리가 의미 있는 조직 규모는 어떻게 되는가?

역할 분리 설계의 핵심 이점은 권한 위임과 책임 분리에 있다. 클러스터 운영자는 Gateway를 만들고 관리하되, 애플리케이션 팀이 HTTPRoute를 직접 생성해 해당 Gateway에 트래픽 규칙을 붙일 수 있다. 이 이점이 실현되려면 두 가지 전제가 필요하다:

1. 클러스터 운영팀과 애플리케이션팀이 실제로 분리되어 있어야 한다.
2. 팀 간 권한 위임이 의미 있을 만큼 접근 제어가 세밀하게 관리되어야 한다.

소규모 팀에서는 이 전제가 성립하지 않는다. 모든 개발자가 클러스터 관리자 권한을 갖고 있고 인프라와 애플리케이션 경계가 흐릿한 환경에서 GatewayClass → Gateway → HTTPRoute의 3단계 리소스 계층은 단순히 "Ingress보다 복잡한 것"으로 인식된다. 더 깊은 문제는 3단계 리소스를 모두 이해해야 하는 학습 비용이 팀 전체에 전파된다는 것이다.

팀 규모별 권장 사항을 정리하면, 20명 이하 팀이고 단일 클러스터를 운영하며 모든 엔지니어가 인프라 접근권을 갖고 있다면 Ingress가 여전히 더 실용적인 선택이다. 플랫폼 팀이 별도로 존재하고 여러 팀이 같은 클러스터를 공유하는 조직(50명 이상)에서 Gateway API의 역할 분리 이점이 구체화된다. Gateway API를 채택하더라도 소규모 팀을 위해 Helm chart나 내부 operator로 GatewayClass와 Gateway 생성을 자동화하고 개발자는 HTTPRoute만 작성하도록 하면 복잡성을 숨길 수 있다.

### Q5. 트래픽 관리에서 자주 나타나는 안티패턴은 무엇이고, 그것이 운영 문제로 이어지는 경로는 어떻게 되는가?

트래픽 관리 설정의 실수는 즉각적인 서비스 장애로 나타나기도 하지만, 더 위험한 것은 서서히 문제가 악화되다가 갑자기 임계점을 넘는 패턴이다.

안티패턴 1은 타임아웃 없는 서킷 브레이커다. 서킷 브레이커만 설정하고 타임아웃을 설정하지 않으면 슬로우 응답이 연결 풀을 채우면서 점진적 성능 저하가 발생한다. 서킷 브레이커는 빠른 실패를 탐지하는 데 효과적이지만 슬로우 응답(응답 자체는 오지만 수십 초 걸리는 경우)은 탐지하지 못한다.

안티패턴 2는 재시도 스톰(Retry Storm)이다. 타임아웃을 짧게 설정하고 재시도를 3번으로 설정하면, 업스트림 장애 시 원래 RPS의 3배 트래픽이 이미 과부하 상태인 서비스로 집중된다. 재시도는 지수 백오프와 jitter를 함께 적용해야 하고, 재시도 가능한 상태 코드를 429, 502, 503, 504로 제한해야 한다. 500 오류에 재시도하면 비멱등 요청이 중복 실행될 수 있다.

안티패턴 3은 부적절한 헬스체크와 Outlier Detection 불일치다. Kubernetes의 readiness probe가 5초마다 실행되는데 Envoy의 outlier detection이 10초마다 불량 호스트를 감지하도록 설정되어 있다면, 불량 Pod가 최대 10초간 트래픽을 받는다. 트래픽 관리 설정을 배포할 때 최소한의 검증 체크리스트는 다음과 같다: 타임아웃과 서킷 브레이커가 모두 설정되어 있는가? 재시도에 지수 백오프가 적용되어 있는가? 재시도 대상 상태 코드가 명시적으로 정의되어 있는가?

### Q6. 카나리 배포를 Gateway API 또는 메시로 구현할 때, 단순한 가중치 분할 이상의 고려사항은 무엇인가?

카나리 배포의 핵심 구성 요소는 세 가지다.

첫째는 트래픽 라우팅 전략이다. 단순 가중치 분할(random 10%), 헤더 기반(특정 사용자 세그먼트), 쿠키 기반(세션 고정)으로 나뉜다. 단순 가중치는 구현이 쉽지만 같은 사용자가 매 요청마다 다른 버전을 볼 수 있어 사용자 경험이 불일치한다. 세션 고정이 필요하다면 consistent hashing을 활용한 라우팅이 필요하다.

둘째는 메트릭 기준 정의다. 카나리의 "성공"을 어떻게 정의하는가? 일반적으로 에러율(5xx), p99 레이턴시, 비즈니스 메트릭(결제 완료율)을 기준으로 한다. 기술 메트릭만으로는 비즈니스 영향을 놓칠 수 있다. Flagger, Argo Rollouts 같은 도구가 분석 자동화를 제공하지만, 기준값은 팀이 직접 정의해야 한다.

셋째는 자동화 수준 결정이다. 메트릭이 임계값을 넘으면 자동으로 롤백할 것인가, 아니면 사람이 판단할 것인가? 자동 롤백은 빠른 대응이 가능하지만 일시적인 스파이크로 인한 불필요한 롤백이 발생할 수 있다. Flagger의 분석 윈도우(예: 5분간 3회 연속 임계값 초과 시 롤백)같은 방식이 이 문제를 완화한다.

카나리 배포 성숙도는 "트래픽 분할이 가능한가"가 아니라 "실패를 자동으로 감지하고 롤백할 수 있는가"로 평가해야 한다. 가중치 조정(1% → 5% → 25% → 100%)은 자동화하되, 각 단계에서 최소 5분간 메트릭을 관찰하는 구조를 만드는 것이 최소 카나리 배포 구현이다.
