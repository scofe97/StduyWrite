# Service Mesh 기반 개념 학습 MOC

---

> 특정 구현체(Linkerd·Istio)로 들어가기 전에 알아야 할 공통 토대를 한 폴더로 묶었습니다. 본 묶음을 끝내면 "서비스 메시가 어떤 문제를 푸는가, 프록시를 어디에 두는가, 트래픽을 어떤 API로 다루는가, 통신 신뢰는 어떻게 자동화하는가" 라는 네 가지 질문에 구현체와 무관하게 답할 수 있습니다.

## 왜 한 폴더로 묶었는가

Linkerd든 Istio든 Cilium이든, 모든 서비스 메시는 같은 네 가지 토대 위에 서 있습니다. 마이크로서비스 통신 복잡도를 왜 인프라 계층으로 옮기는가(메시의 존재 이유), 그 인프라를 어디에 배치하는가(프록시 모델), 트래픽 규칙을 무엇으로 선언하는가(Gateway API), 그리고 평문 내부 통신을 어떻게 신뢰 가능하게 만드는가(mTLS) 입니다.

이 네 가지는 어느 구현체를 고르든 다시 등장합니다. 그래서 구현체별 폴더(`02_linkerd`, `03_istio`)에서 매번 반복하지 않고, 개념의 단일 출처(SSOT)로 이 폴더에 모았습니다. 뒤 폴더의 보안·트래픽 문서는 여기 04-01(mTLS)·03-01(Gateway API)을 전제로 하고, 중복되는 개념 설명은 이 폴더를 참조합니다.

## 환경과 버전

> 본 묶음 예제는 다음 기준입니다.

| 항목 | 값 | 비고 |
|------|-----|------|
| Kubernetes | 1.28 이상 | Gateway API CRD 설치 가능 |
| Gateway API | v1.x (Standard Channel) | GAMMA는 v1.1+ GA |
| 실습 클러스터 | 개인 GCP K8s (dev-server 1~3) | 상세는 [`gcp` 스킬](../../../../.claude/skills/gcp/SKILL.md) |

## 사전 지식

> 이 묶음은 다음을 안다고 가정합니다.

1. Kubernetes의 Pod·Service·Namespace 개념을 한 줄로 설명할 수 있습니다.
2. Ingress로 외부 트래픽을 클러스터 안으로 받아 본 경험이 있습니다.
3. TLS가 무엇을 보호하는지 알고 있습니다(암호화·서버 인증).

Pod·Service·Ingress가 막히면 [kubernetes MOC](../../kubernetes/README.md)를 먼저 본 뒤 돌아옵니다.

## 면접 대비 체크리스트

> 네 장을 다 읽은 뒤 다음 질문에 답할 수 있어야 합니다.

1. 서비스 메시가 라이브러리 방식(예: Hystrix) 대비 갖는 구조적 차이는 무엇입니까? 데이터 플레인과 컨트롤 플레인은 각각 무엇을 합니까?
2. 사이드카·앰비언트·에이전트 모델은 각각 어떤 리소스·지연·운영 트레이드오프를 가집니까?
3. Ingress가 풀지 못한 문제를 Gateway API는 어떻게 풀었습니까? 세 페르소나(인프라·클러스터·앱)의 역할 분리란 무엇입니까?
4. mTLS가 일반 TLS와 다른 점은? SPIFFE 신원을 IP 대신 쓰면 무엇이 좋아집니까?

각 질문에 막히면 해당 장 본문으로 돌아갑니다.
