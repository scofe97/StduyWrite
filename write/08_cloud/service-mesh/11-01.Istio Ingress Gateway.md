# Istio Ingress Gateway

> 트래픽 제어 묶음은 외부 진입점부터 보는 편이 자연스럽다.
>
> - Ingress Gateway는 외부 요청이 메시 안으로 들어오는 첫 관문이며, Kubernetes Ingress보다 넓은 L4/L7 표현력을 제공한다.
> - Gateway 리소스로 "어떤 포트와 호스트를 열 것인가"를 정의하고, 다음 문서의 VirtualService로 "어느 백엔드로 보낼 것인가"를 연결하는 이분법적 설계가 핵심이다.


## 학습 목표

> Gateway+VirtualService 역할 분리, TLS 모드 선택, SNI 멀티 도메인, HTTP→HTTPS 리다이렉트, TCP Gateway, Istio vs Kubernetes Gateway API 비교까지 여섯 가지 목표를 다룬다.

학습 목표는 여섯 가지다:

1. Gateway 리소스와 VirtualService의 역할 분리를 설명하고 두 리소스를 연결하는 방법을 작성한다.
2. TLS 모드(SIMPLE, MUTUAL, PASSTHROUGH)를 구별하고 상황에 적합한 모드를 선택한다.
3. SNI 원리를 이해하고 단일 Gateway로 여러 도메인을 서빙하는 설정을 작성한다.
4. HTTP 트래픽을 HTTPS로 리다이렉트하는 VirtualService 규칙을 구성한다.
5. TCP Gateway를 사용해 비-HTTP 프로토콜을 메시에 노출하는 방법을 이해한다.
6. Istio Gateway API와 Kubernetes Gateway API의 차이를 구별하고 선택 기준을 설명한다.




## 1. Ingress Gateway의 역할

> istio-ingressgateway Envoy Pod가 LoadBalancer IP를 통해 외부 트래픽을 받아 메시 내부로 전달하며, Gateway 리소스가 포트·프로토콜을, VirtualService가 실제 라우팅을 담당하는 이분법적 설계를 설명한다.

### 1.1 외부 트래픽 진입점으로서의 Gateway

서비스 메시 내부에서 Pod 간 통신은 사이드카 프록시(Envoy)가 자동으로 처리한다. 그러나 클러스터 외부에서 들어오는 트래픽은 다르다. 외부 요청은 메시 바깥에서 시작하므로 사이드카가 없다. 이 진입 지점을 담당하는 컴포넌트가 Ingress Gateway Envoy 프록시다.

Istio는 `istio-system` 네임스페이스에 `istio-ingressgateway`라는 Deployment를 배포한다. 이 Pod는 사이드카가 아닌 독립 Envoy 프록시로, Kubernetes Service(LoadBalancer 타입)에 연결되어 외부 IP를 가진다. 트래픽 흐름은 LoadBalancer → Gateway Pod → 내부 서비스 Pod 순이다.

Kubernetes Ingress와 비교하면 차이가 명확하다. Ingress는 HTTP/HTTPS만 처리하고 TLS 옵션이 제한적이다. Istio Gateway는 HTTP, HTTPS, TLS, TCP, mTLS를 모두 처리하며, VirtualService와 결합하면 헤더 기반 라우팅, 트래픽 미러링, 장애 주입도 가능하다.

### 1.2 Istio Gateway API vs Kubernetes Gateway API

이름이 비슷해서 혼동하기 쉽지만 완전히 다른 API다.

| 항목 | Istio Gateway API | Kubernetes Gateway API |
|------|-------------------|----------------------|
| API 그룹 | `networking.istio.io/v1` | `gateway.networking.k8s.io` |
| 트래픽 수신 리소스 | `Gateway` | `Gateway` (동명) |
| 라우팅 리소스 | `VirtualService` | `HTTPRoute`, `TCPRoute` |
| 이식성 | Istio 전용 | 멀티 구현체 지원 |
| Fault Injection | 지원 | 미지원 (확장 필요) |

신규 프로젝트이고 Fault Injection·Traffic Mirroring 같은 Istio 고급 기능을 사용하지 않을 계획이라면 Kubernetes Gateway API를 선택해 이식성을 확보하는 것이 합리적이다. 반면 카오스 엔지니어링, 카나리 배포 자동화, 세밀한 재시도 제어가 필요하다면 Istio Gateway API를 선택하고 Istio에 종속되는 결정을 의식적으로 내리는 것이 낫다.

### 1.3 Gateway + VirtualService 리소스 관계

Gateway 리소스만으로는 트래픽이 내부 서비스에 도달하지 않는다. Gateway는 포트와 호스트를 "열어주는" 역할만 하며, 실제 라우팅은 VirtualService가 담당한다.

```
Gateway 리소스                     VirtualService 리소스
─────────────────────────────     ──────────────────────────────────
spec.servers:                     spec.gateways:
  - port: 80                        - my-gateway   (Gateway 이름 참조)
    hosts:                         spec.hosts:
      - "*.example.com"               - "api.example.com"
```

VirtualService의 `hosts`는 Gateway의 `hosts`와 일치하거나 그 하위 집합이어야 한다. 매칭이 맞지 않으면 요청이 라우팅되지 않고 404가 반환된다.




## 2. 기본 HTTP 라우팅

> Gateway 리소스로 포트와 허용 호스트를 선언하고, VirtualService로 URI 매칭과 백엔드 서비스를 연결하는 기본 HTTP 라우팅 구성을 다룬다.

### 2.1 Gateway 리소스 정의

```yaml
apiVersion: networking.istio.io/v1
kind: Gateway
metadata:
  name: my-gateway
  namespace: istio-system
spec:
  selector:
    istio: ingressgateway   # istio-ingressgateway Pod에 적용
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "*.example.com"
```

`selector`는 Gateway를 적용할 Envoy 프록시를 선택한다. `hosts`의 와일드카드는 조심해서 사용해야 한다. `"*"`는 모든 호스트를 허용하므로 보안상 실제 도메인 패턴을 명시하는 것이 낫다.

### 2.2 VirtualService로 서비스 연결

```yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: api-vs
  namespace: default
spec:
  hosts:
  - "api.example.com"
  gateways:
  - istio-system/my-gateway   # namespace/name 형식
  http:
  - match:
    - uri:
        prefix: /v1
    route:
    - destination:
        host: api-service
        port:
          number: 8080
  - route:
    - destination:
        host: api-service
        port:
          number: 8080
```

`gateways` 필드에 `namespace/name` 형식으로 참조한다. `mesh`라는 특수 값을 추가하면 외부 Gateway 트래픽과 메시 내부 트래픽 모두에 이 VirtualService를 적용할 수 있다.

### 2.3 호스트 기반 라우팅과 SNI

단일 IP(LoadBalancer)에서 여러 도메인을 서빙할 때 HTTP에서는 `Host` 헤더를 보고 라우팅한다. HTTPS에서는 TLS 핸드셰이크 시 클라이언트가 접속할 호스트명을 SNI(Server Name Indication)를 통해 먼저 알린다. Gateway는 이 SNI 값을 보고 적절한 인증서와 라우팅을 선택한다.


## 3. TLS 설정

> SIMPLE·MUTUAL·PASSTHROUGH 세 TLS 모드의 차이, HTTPS 인증서 Secret 관리, HTTP→HTTPS 자동 리다이렉트 설정을 다룬다.

### 3.1 TLS 모드 비교

| 모드 | 설명 | 적합한 상황 |
|------|------|------------|
| SIMPLE | Gateway에서 TLS 종료 | 일반 HTTPS 서빙 |
| MUTUAL | 클라이언트 인증서도 검증 | B2B API, 고보안 환경 |
| PASSTHROUGH | TLS를 종료하지 않고 그대로 전달 | 백엔드가 직접 TLS 처리 |

### 3.2 HTTPS 설정 예시

```yaml
spec:
  servers:
  - port:
      number: 443
      name: https
      protocol: HTTPS
    tls:
      mode: SIMPLE
      credentialName: my-tls-secret   # kubernetes.io/tls 타입 Secret
    hosts:
    - "api.example.com"
```

인증서는 Kubernetes Secret으로 관리한다. `credentialName`이 참조하는 Secret은 `istio-system` 네임스페이스에 있어야 한다.

### 3.3 HTTP → HTTPS 리다이렉트

```yaml
# HTTP Gateway: 리다이렉트 설정
spec:
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "api.example.com"
    tls:
      httpsRedirect: true   # 301 리다이렉트 자동 응답
```


## 4. TCP Gateway

> MySQL·Redis 같은 비-HTTP 프로토콜을 메시에 노출할 때 TCP Gateway를 사용하며 L7 기능이 제한된다는 트레이드오프를 설명한다.

비-HTTP 프로토콜(MySQL, Redis, gRPC over non-standard port)은 TCP Gateway를 사용한다.

```yaml
spec:
  servers:
  - port:
      number: 3306
      name: mysql
      protocol: TCP
    hosts:
    - "*"
---
apiVersion: networking.istio.io/v1
kind: VirtualService
spec:
  hosts:
  - "*"
  gateways:
  - my-tcp-gateway
  tcp:
  - match:
    - port: 3306
    route:
    - destination:
        host: mysql-service
        port:
          number: 3306
```

TCP Gateway는 L7 기능(헤더 기반 라우팅, 재시도 정책)을 사용할 수 없다. 연결 수준의 라우팅만 가능하다.


## 5. Kubernetes Gateway API로 마이그레이션

> 기존 VirtualService 기능을 인벤토리화해 HTTPRoute로 표현 불가능한 항목을 먼저 식별하고, 서비스 단위로 점진 전환하는 안전한 마이그레이션 경로를 설명한다.

기존 Istio Gateway API에서 Kubernetes Gateway API로 전환하는 경로는 다음과 같다.

```yaml
# 기존: Istio Gateway API
apiVersion: networking.istio.io/v1
kind: Gateway
metadata:
  name: my-gateway
spec:
  selector:
    istio: ingressgateway
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "example.com"
---
# 변환: Kubernetes Gateway API
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: my-gateway
spec:
  gatewayClassName: istio   # Istio를 구현체로 지정
  listeners:
  - name: http
    port: 80
    protocol: HTTP
    hostname: "example.com"
    allowedRoutes:
      namespaces:
        from: All
```

마이그레이션 전 현재 VirtualService 기능을 인벤토리화한다. `kubectl get virtualservice -A -o yaml`로 전체 VirtualService를 추출하고, Fault Injection·mirrorPercentage·복잡한 retryOn 조건 등 Kubernetes Gateway API로 표현 불가능한 기능을 목록으로 만든다. 표현 불가능한 기능이 없는 서비스부터 순차적으로 `HTTPRoute`로 변환하는 것이 안전하다.


## 6. 게이트웨이 배포 모델

> 공유 게이트웨이·네임스페이스 게이트웨이·자동화 게이트웨이 세 배포 모델의 역할 분리와 비용 트레이드오프를 비교한다.

| 모델 | 설명 | 적합한 상황 |
|------|------|------------|
| 공유 게이트웨이 | 클러스터 전체가 단일 Gateway 사용 | 소규모 팀, 단순한 도메인 구조 |
| 네임스페이스 게이트웨이 | 팀별 전용 Gateway | 멀티테넌트, 독립 배포 필요 |
| 자동화 게이트웨이 | Gateway API + GatewayClass로 자동 프로비저닝 | 대규모 플랫폼 |

공유 게이트웨이는 Gateway를 인프라 팀이 전담하고 애플리케이션 팀은 VirtualService와 DestinationRule만 관리하는 역할 분리가 일반적이다. 네임스페이스 게이트웨이는 팀 간 독립성을 높이지만 LoadBalancer IP가 팀 수만큼 늘어나 비용이 증가한다.



## 면접 대비

> Istio Ingress Gateway 설계에서 자주 받는 네 가지 질문을 답변 형식으로 정리한다.

**Gateway와 VirtualService를 두 리소스로 분리한 이유는?**

운영 책임을 분할하기 위해서다. Gateway는 "어떤 포트·호스트·TLS 인증서를 외부에 연다"는 인프라 결정이고, VirtualService는 "어떤 백엔드로 어떤 가중치로 보낼지"라는 애플리케이션 라우팅이다. 두 책임이 한 리소스에 묶이면 인프라 팀이 인증서를 갱신할 때 애플리케이션 팀의 라우팅 매니페스트까지 건드려야 하는 등 권한 경계가 무너진다.

**TLS 모드 SIMPLE·MUTUAL·PASSTHROUGH는 각각 언제 쓰는가?**

SIMPLE은 게이트웨이가 서버 인증서로 TLS를 종단하는 가장 흔한 외부 진입 구성이다. MUTUAL은 클라이언트 인증서까지 검증하는 B2B API·내부 망 진입에 쓴다. PASSTHROUGH는 게이트웨이가 TLS를 풀지 않고 SNI만 보고 백엔드로 그대로 넘기는 모드라 백엔드가 자체 TLS를 종단해야 하거나 종단간 암호화를 유지해야 하는 시나리오에 적합하다.

**Istio Gateway와 Kubernetes Gateway API HTTPRoute의 차이는?**

Istio Gateway+VirtualService는 메시 특화 기능(가중치·재시도·미러링·Fault Injection)이 풍부하지만 Istio 전용 CRD다. Kubernetes Gateway API는 표준 CRD로 메시를 바꿔도 동일 매니페스트가 다른 컨트롤러(Linkerd·Contour 등)에서 동작한다. 표준 휴대성과 멀티 메시 가능성을 우선하면 Gateway API로 마이그레이션하고, 메시 특화 고급 기능이 필수면 Istio CRD를 유지한다.

**공유 게이트웨이와 네임스페이스 게이트웨이 중 어느 모델을 골라야 하는가?**

팀 수·도메인 구조·LoadBalancer 비용으로 결정한다. 소규모이거나 단일 도메인 위주면 공유 모델이 운영 단순성과 IP 비용 면에서 유리하다. 멀티 테넌트로 팀별 독립 배포 일정이 필요하거나 도메인이 팀별로 분리돼 있다면 네임스페이스 게이트웨이가 변경 격리를 보장한다. 대규모 플랫폼은 GatewayClass 기반 자동 프로비저닝으로 두 모델의 장점을 섞기도 한다.
