# Istio 아키텍처 실습

> 본 장의 개념을 손으로 확인합니다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`istioctl` 범용 명령으로 씁니다.


## 실습 환경

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| 서비스 메시 | Istio (사이드카 주입 네임스페이스) | — |

접속은 `gcp` 스킬의 SSH 절차를 따릅니다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1: istiod 상태와 구성 요소 확인

### 목표

istiod 내부 컴포넌트(Pilot·Citadel·Galley)가 실제로 어떤 포트에서 서비스를 제공하는지 확인합니다.

```bash
# istiod Pod 상태 확인
kubectl get pods -n istio-system -l app=istiod

# istiod Service 포트 확인
kubectl get svc istiod -n istio-system -o yaml

# 주요 포트:
# 15010: xDS (gRPC, 비보안)
# 15012: xDS (gRPC, TLS)
# 15014: 제어 플레인 모니터링
# 15017: Admission Webhook (HTTPS)
# 8080:  헬스 체크
---

# istiod 로그 확인 (xDS 푸시 확인)
kubectl logs -n istio-system \
  -l app=istiod --tail=50 | grep -i "push\|xds"
```

### xDS 푸시 메트릭 확인

```bash
# istiod의 Prometheus 메트릭 포트 포워딩
kubectl port-forward -n istio-system svc/istiod 15014:15014

# 브라우저 또는 curl로 메트릭 확인
curl http://localhost:15014/metrics | grep pilot_xds_push_time
```

`pilot_xds_push_time`의 p99가 5초를 초과하면 istiod가 병목 상태일 수 있습니다.


## 실습 2: Envoy 사이드카 설정 확인

### 목표

iptables 규칙과 Envoy가 xDS로 받은 설정을 실제로 확인합니다.

```bash
# 테스트 앱 배포 (사이드카 자동 주입 네임스페이스)
kubectl create namespace arch-demo
kubectl label namespace arch-demo istio-injection=enabled

kubectl create deployment web --image=nginx -n arch-demo
kubectl wait --for=condition=ready pod -l app=web -n arch-demo

# Pod 내부 컨테이너 목록 확인 (istio-proxy 포함 여부)
kubectl get pod -n arch-demo \
  -l app=web -o jsonpath='{.items[0].spec.containers[*].name}'
# 출력: web istio-proxy
---

# Envoy가 수신한 xDS 설정 확인
kubectl exec -n arch-demo \
  $(kubectl get pod -n arch-demo -l app=web -o name) \
  -c istio-proxy -- pilot-agent request GET /config_dump \
  | python3 -m json.tool | head -100
```

### iptables 규칙 확인

```bash
# Init Container가 설정한 iptables 규칙 확인
kubectl exec -n arch-demo \
  $(kubectl get pod -n arch-demo -l app=web -o name) \
  -c istio-proxy -- iptables -t nat -L -n -v | head -40

# 주요 규칙:
# ISTIO_IN_REDIRECT: 인바운드 트래픽 → 15006 (Envoy)
# ISTIO_OUTPUT: 아웃바운드 트래픽 → 15001 (Envoy)
```


## 실습 3: SPIFFE 인증서 확인

```bash
# Envoy가 사용 중인 TLS 인증서 정보 확인
kubectl exec -n arch-demo \
  $(kubectl get pod -n arch-demo -l app=web -o name) \
  -c istio-proxy -- \
  openssl s_client -connect localhost:15000 2>/dev/null \
  | openssl x509 -noout -text | grep -A2 "Subject Alternative"

# SPIFFE URI 형식 확인:
# spiffe://cluster.local/ns/arch-demo/sa/default
```


## 실습 4: istioctl analyze로 설정 오류 진단

```bash
# 현재 클러스터 설정 전체 분석
istioctl analyze -n arch-demo

# 의도적으로 잘못된 VirtualService 적용
cat <<EOF | kubectl apply -f -
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: broken-vs
  namespace: arch-demo
spec:
  hosts:
  - "nonexistent-service"
  http:
  - route:
    - destination:
        host: nonexistent-service
        port:
          number: 8080
EOF

# 오류 감지 확인
istioctl analyze -n arch-demo

# 정리
kubectl delete virtualservice broken-vs -n arch-demo
```

`istioctl analyze`는 "Referenced host not found" 같은 CRD 간 불일치를 감지합니다.


## 실습 5: xDS 확장성 병목 시뮬레이션

```bash
# xDS 푸시 관련 메트릭 실시간 모니터링
watch -n2 'kubectl exec -n istio-system \
  $(kubectl get pod -n istio-system -l app=istiod -o name | head -1) \
  -- curl -s localhost:15014/metrics \
  | grep "pilot_xds_push_time_count\|pilot_xds_pending"'

# Deployment를 자주 업데이트해 EDS 푸시 부하 생성
for i in $(seq 1 5); do
  kubectl rollout restart deployment/web -n arch-demo
  sleep 5
done
```

롤링 업데이트 중 `pilot_xds_push_time`이 증가하는지 관찰합니다.
