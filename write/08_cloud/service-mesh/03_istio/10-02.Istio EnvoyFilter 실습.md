## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터다. 구성은 다음과 같다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |

| Ingress | nginx-ingress v1.12.0 (NodePort 31292/31726) | — |

접속은 `gcp` 스킬의 SSH 절차를 따른다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1. Lua 필터로 커스텀 응답 헤더 추가

### 1-1. 테스트 서비스 배포

```bash
kubectl create namespace ch19-envoyfilter

kubectl apply -n ch19-envoyfilter -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: httpbin
spec:
  replicas: 1
  selector:
    matchLabels:
      app: httpbin
  template:
    metadata:
      labels:
        app: httpbin
    spec:
      containers:
      - name: httpbin
        image: kennethreitz/httpbin:latest
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: httpbin
spec:
  selector:
    app: httpbin
  ports:
  - port: 80
    targetPort: 80
EOF

kubectl get pods -n ch19-envoyfilter
```

### 1-2. EnvoyFilter 적용 (Lua 응답 헤더 삽입)

```bash
kubectl apply -n ch19-envoyfilter -f - <<EOF
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: add-response-header
spec:
  workloadSelector:
    labels:
      app: httpbin
  configPatches:
    - applyTo: HTTP_FILTER
      match:
        context: SIDECAR_INBOUND
        listener:
          filterChain:
            filter:
              name: envoy.filters.network.http_connection_manager
              subFilter:
                name: envoy.filters.http.router
      patch:
        operation: INSERT_BEFORE
        value:
          name: envoy.filters.http.lua
          typed_config:
            "@type": type.googleapis.com/envoy.extensions.filters.http.lua.v3.LuaPerRoute
            inline_code: |
              function envoy_on_response(response_handle)
                local pod_name = os.getenv("HOSTNAME") or "unknown"
                response_handle:headers():add("x-served-by", pod_name)
              end
EOF
```

### 1-3. 적용 확인

```bash
# config_dump에서 lua 필터 확인
HTTPBIN_POD=$(kubectl get pod -n ch19-envoyfilter -l app=httpbin -o jsonpath='{.items[0].metadata.name}')

istioctl proxy-config listeners $HTTPBIN_POD.ch19-envoyfilter \
  --port 80 -o json | grep -A 5 "envoy.filters.http.lua"

# 실제 요청으로 헤더 확인
kubectl exec -n ch19-envoyfilter $HTTPBIN_POD -- \
  curl -s -I http://localhost/headers | grep x-served-by
```

`x-served-by: <pod-name>` 헤더가 응답에 포함되면 성공이다.


## 실습 2. 커스텀 재시도 상태 코드 설정

### 2-1. EnvoyFilter 적용

```bash
kubectl apply -n ch19-envoyfilter -f - <<EOF
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: retriable-408
spec:
  workloadSelector:
    labels:
      app: httpbin
  configPatches:
    - applyTo: HTTP_ROUTE
      match:
        context: SIDECAR_OUTBOUND
        routeConfiguration:
          vhost:
            name: httpbin.ch19-envoyfilter.svc.cluster.local:80
            route:
              action: ANY
      patch:
        operation: MERGE
        value:
          route:
            retry_policy:
              retry_on: "retriable-status-codes"
              retriable_status_codes: [408, 503]
              num_retries: 3
              per_try_timeout: 2s
EOF
```

### 2-2. 적용 확인

```bash
# route 설정에서 retry_policy 확인
istioctl proxy-config routes $HTTPBIN_POD.ch19-envoyfilter -o json | \
  grep -A 10 "retriable_status_codes"
```


## 실습 3. WasmPlugin vs EnvoyFilter 선택 기준 확인

### 3-1. Istio 버전 확인

```bash
istioctl version
```

WasmPlugin CRD는 Istio 1.12+에서 지원된다.

### 3-2. WasmPlugin 등록 가능 여부 확인

```bash
kubectl get crd wasmplugins.extensions.istio.io
```

CRD가 존재하면 WasmPlugin을 사용할 수 있다. 단순 헤더 조작은 Lua(EnvoyFilter)로, 복잡한 보안 로직이나 100줄 초과 커스텀 로직은 WasmPlugin으로 분리한다.


## 실습 4. config_dump로 디버깅

### 4-1. EnvoyFilter 적용 여부 확인

```bash
# 전체 config_dump 저장
kubectl exec -n ch19-envoyfilter $HTTPBIN_POD -c istio-proxy -- \
  curl -s localhost:15000/config_dump > /tmp/config_dump.json

# lua 필터 존재 여부 확인
grep -c "envoy.filters.http.lua" /tmp/config_dump.json
```

0이 나오면 match 조건이 잘못됐다. context 값이 실제 Listener 방향과 맞는지, filter.name이 현재 Istio 버전에서 사용하는 이름과 일치하는지 확인한다.

### 4-2. 구문 오류 사전 감지

```bash
istioctl analyze -n ch19-envoyfilter
```

### 4-3. 실습 정리

```bash
kubectl delete namespace ch19-envoyfilter
```
