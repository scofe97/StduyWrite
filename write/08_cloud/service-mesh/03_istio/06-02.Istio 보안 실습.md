# Istio 보안 실습

> 본 장의 개념을 손으로 확인한다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`istioctl` 범용 명령으로 쓴다. 환경이 바뀌면 `## 실습 환경` 섹션만 갈아 끼우면 된다.


## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker |
| K8s | kubeadm v1.31.14 | — |
| Istio | istioctl 기준 설치 | 메시 주입 네임스페이스: `default`, `bookinfo` |

접속은 `gcp` 스킬의 SSH 절차를 따른다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1. mTLS 상태 확인 (PERMISSIVE 모드)

```bash
# 현재 mTLS 상태 확인
istioctl x describe pod $(kubectl get pod -n bookinfo -l app=productpage \
  -o jsonpath='{.items[0].metadata.name}') -n bookinfo

# 평문 트래픽이 존재하는지 확인 (PERMISSIVE 상태에서 검색)
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s http://reviews:9080/reviews/1

# connection_security_policy 메트릭으로 mTLS 여부 확인
# Prometheus 쿼리: istio_requests_total{connection_security_policy="none"}
# 결과가 있으면 평문 트래픽이 존재함
```


## 실습 2. STRICT mTLS 활성화

```bash
# 단계 1: bookinfo 네임스페이스에 STRICT 적용
kubectl apply -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: bookinfo-strict
  namespace: bookinfo
spec:
  mtls:
    mode: STRICT
EOF

# 사이드카 없는 클라이언트에서 접근 시도 (실패 예상)
kubectl run test-client --image=curlimages/curl --restart=Never \
  -n default -- curl -s http://reviews.bookinfo:9080/reviews/1
# 사이드카 없는 Pod는 mTLS를 수행하지 못해 연결 거부
---

# 사이드카 있는 Pod에서는 정상 접근
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s http://reviews:9080/reviews/1

# 정리 (이후 실습을 위해 PERMISSIVE로 되돌림)
kubectl delete peerauthentication bookinfo-strict -n bookinfo
```


## 실습 3. 포트 레벨 mTLS 예외 설정

```bash
# 헬스체크 포트만 평문 허용 (나머지는 STRICT)
kubectl apply -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: partial-strict
  namespace: bookinfo
spec:
  selector:
    matchLabels:
      app: reviews
  mtls:
    mode: STRICT
  portLevelMtls:
    9080:              # 메인 포트: STRICT
      mode: STRICT
EOF

# 적용된 정책 확인
istioctl x describe pod $(kubectl get pod -n bookinfo -l app=reviews \
  -o jsonpath='{.items[0].metadata.name}') -n bookinfo

kubectl delete peerauthentication partial-strict -n bookinfo
```


## 실습 4. RequestAuthentication (JWT 검증)

```bash
# JWT 검증 설정 (테스트용 jwksUri 사용)
kubectl apply -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: RequestAuthentication
metadata:
  name: jwt-example
  namespace: bookinfo
spec:
  selector:
    matchLabels:
      app: productpage
  jwtRules:
  - issuer: "testing@secure.istio.io"
    jwksUri: "https://raw.githubusercontent.com/istio/istio/release-1.22/security/tools/jwt/samples/jwks.json"
    outputClaimToHeaders:
    - header: x-user-sub
      claim: sub
EOF

# 유효한 JWT 없이 접근 (RequestAuthentication 단독으로는 차단 안 됨)
kubectl exec -n bookinfo deploy/ratings -- \
  curl -s -o /dev/null -w "%{http_code}" http://productpage:9080/productpage
# 200 반환 (JWT 없어도 통과)
---

# 유효하지 않은 JWT로 접근 (401 반환)
kubectl exec -n bookinfo deploy/ratings -- \
  curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer invalid-token" \
  http://productpage:9080/productpage
# 401 반환 (잘못된 JWT는 거부)
```


## 실습 5. AuthorizationPolicy — Default Deny 후 선택적 허용

```bash
# 기본 거부 적용 (빈 spec)
kubectl apply -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: bookinfo
spec: {}
EOF

# 모든 접근 거부됨 확인
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s -o /dev/null -w "%{http_code}" http://reviews:9080/reviews/1
# 403 반환
---

# productpage → reviews 명시적 허용
kubectl apply -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-productpage-to-reviews
  namespace: bookinfo
spec:
  selector:
    matchLabels:
      app: reviews
  action: ALLOW
  rules:
  - from:
    - source:
        principals:
        - "cluster.local/ns/bookinfo/sa/bookinfo-productpage"
    to:
    - operation:
        methods: ["GET"]
        paths: ["/reviews/*"]
EOF

# 허용됨 확인
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s -o /dev/null -w "%{http_code}" http://reviews:9080/reviews/1
# 200 반환
---

# 정리
kubectl delete authorizationpolicy deny-all allow-productpage-to-reviews -n bookinfo
```


## 실습 6. JWT 클레임 기반 접근 제어

```bash
# Istio 공식 테스트 JWT 토큰 설정
TOKEN=$(curl -s https://raw.githubusercontent.com/istio/istio/release-1.22/security/tools/jwt/samples/demo.jwt)

# RequestAuthentication + AuthorizationPolicy 조합
kubectl apply -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: require-jwt
  namespace: bookinfo
spec:
  selector:
    matchLabels:
      app: productpage
  action: ALLOW
  rules:
  - from:
    - source:
        requestPrincipals: ["testing@secure.istio.io/testing@secure.istio.io"]
EOF

# JWT 없이 접근 → 403
kubectl exec -n bookinfo deploy/ratings -- \
  curl -s -o /dev/null -w "%{http_code}" http://productpage:9080/productpage

# 유효한 JWT로 접근 → 200
kubectl exec -n bookinfo deploy/ratings -- \
  curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer ${TOKEN}" \
  http://productpage:9080/productpage

# 정리
kubectl delete authorizationpolicy require-jwt -n bookinfo
kubectl delete requestauthentication jwt-example -n bookinfo
```


## 실습 7. 정책 검증 명령어

```bash
# 네임스페이스의 인가 정책 확인
kubectl get authorizationpolicy,peerauthentication,requestauthentication -n bookinfo

# 특정 Pod에 적용된 정책 확인
REVIEWS_POD=$(kubectl get pod -n bookinfo -l app=reviews \
  -o jsonpath='{.items[0].metadata.name}')

istioctl x authz check ${REVIEWS_POD} -n bookinfo

# 정책 오류 분석
istioctl analyze -n bookinfo

# mTLS 상태 상세 확인
istioctl x describe pod ${REVIEWS_POD} -n bookinfo
```


## STRICT 마이그레이션 체크리스트

전환 전 준비:

```bash
# 사이드카 없는 Pod 확인
kubectl get pods -A -o json | \
  jq '.items[] | select(.spec.containers | length == 1) | .metadata | {namespace, name}'

# 평문 트래픽 존재 여부 확인 (Prometheus 쿼리)
# istio_requests_total{connection_security_policy="none"}
---

# 헬스체크 포트 확인 (포트 레벨 예외 필요 여부)
kubectl get pods -n bookinfo -o json | \
  jq '.items[].spec.containers[].livenessProbe'
```


## 체크리스트

- [ ] PERMISSIVE 모드에서 평문 트래픽이 존재하는지 메트릭으로 확인했는가
- [ ] STRICT 전환 후 사이드카 없는 Pod에서 연결이 거부되는 것을 확인했는가
- [ ] 포트 레벨 mTLS 예외가 필요한 경우를 파악하고 설정했는가
- [ ] `RequestAuthentication` 단독으로는 JWT 없는 요청이 통과되는 것을 확인했는가
- [ ] 빈 spec의 `AuthorizationPolicy`로 기본 거부가 동작하는 것을 확인했는가
- [ ] `requestPrincipals` 조건으로 JWT 없는 요청이 403이 되는 것을 확인했는가
- [ ] `istioctl x authz check`로 특정 Pod에 적용된 정책을 검증했는가
