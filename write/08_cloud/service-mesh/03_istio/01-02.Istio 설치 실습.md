# Istio 설치 실습

> 본 장의 개념을 손으로 확인합니다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`istioctl` 범용 명령으로 씁니다.


## 실습 환경

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| 서비스 메시 | Istio (사이드카 주입 네임스페이스) | — |

접속은 `gcp` 스킬의 SSH 절차를 따릅니다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1: istioctl로 설치 (demo 프로파일)

### 목표

가장 빠른 방법으로 Istio를 설치하고 구성 요소를 확인합니다.

```bash
# istioctl 다운로드 (최신 버전)
curl -L https://istio.io/downloadIstio | sh -
export PATH=$PWD/istio-*/bin:$PATH

# 사전 확인
istioctl x precheck

# demo 프로파일로 설치 (학습용)
istioctl install --set profile=demo -y

# 설치 확인
istioctl verify-install

# 구성 요소 확인
kubectl get pods -n istio-system
# 출력: istiod, istio-ingressgateway, istio-egressgateway
```

### 설치 결과 검토

```bash
# istiod ConfigMap 확인 (메시 설정)
kubectl get configmap istio -n istio-system -o yaml

# MutatingWebhookConfiguration 확인 (사이드카 인젝션 훅)
kubectl get mutatingwebhookconfigurations \
  istio-sidecar-injector -o yaml | head -50
```


## 실습 2: Helm으로 설치 (로컬 → GCP)

### 목표

로컬 맥에서 준비한 Helm 차트와 values 파일로 원격 GCP K8s 클러스터에 Istio를 설치합니다. 프로덕션·폐쇄망 운영에서 쓰는 패턴을 직접 체험하는 실습입니다. `istioctl install`과 달리 Helm은 차트와 values를 Git으로 선언적으로 관리할 수 있어 GitOps 파이프라인으로 이어집니다.

이후 실습(사이드카 인젝션, analyze, 카나리 업그레이드, 앰비언트)은 istioctl 기반으로 진행하므로 본 실습을 끝낸 뒤에는 `helm uninstall`로 정리하고 실습 1의 demo 프로파일 상태로 되돌려도 되고, 그대로 유지해도 됩니다. 남겨 두는 경우 다음 실습에서 네임스페이스 레이블과 리비전 이름이 충돌하지 않도록만 주의합니다.

### 사전 준비

Helm 3는 순수 클라이언트 도구라 GCP 서버 쪽에는 아무것도 설치하지 않습니다. 로컬 맥에 Helm 바이너리와 GCP 클러스터 kubeconfig만 준비하면 됩니다.

```bash
# 1) Helm 설치 확인 (macOS)
brew install helm   # 이미 있으면 skip
helm version        # v3.x 확인

# 2) kubeconfig 등록 (kubeadm 직구축 클러스터 기준)
scp dev-server:/etc/kubernetes/admin.conf ~/.kube/gcp-config
export KUBECONFIG=~/.kube/gcp-config

# 3) 접속 확인 - 로컬에서 원격 GCP API에 붙었는지 검증
kubectl config current-context
kubectl get nodes
# dev-server, dev-server-2, dev-server-3 Ready 상태
```

> GKE라면 `gcloud container clusters get-credentials <cluster> --zone asia-northeast3-a --project <proj>` 로 대체합니다.

### 차트 준비 (원격 레포 vs 로컬 디렉토리)

로컬에 차트 소스를 내려받아 두면 폐쇄망/오프라인 상황에서도 재현 가능하고 values와 함께 Git으로 버전 관리할 수 있습니다.

```bash
# 작업 디렉토리
mkdir -p ~/istio-helm && cd ~/istio-helm

# 방법 A: 원격 레포에서 pull
helm repo add istio https://istio-release.storage.googleapis.com/charts
helm repo update
helm pull istio/base    --untar
helm pull istio/istiod  --untar
helm pull istio/gateway --untar
# → base/  istiod/  gateway/ 디렉토리 생성

# 방법 B: 릴리스 아카이브를 통째로 사용
curl -L https://istio.io/downloadIstio | sh -
ls istio-*/manifests/charts/
# base/  gateway/  istiod/  ... (동일한 차트 소스)
```

### 로컬 values 파일 작성

환경별 오버라이드는 차트 소스와 분리해 둡니다. 305p 저장소가 `helm-charts/istio-source/`(원본)와 `helm-charts/istio/`(values)를 분리한 이유와 같습니다.

```bash
cat > istiod-values.yaml <<'EOF'
pilot:
  replicaCount: 1            # 학습용 최소화 (프로덕션은 3 이상)
  resources:
    requests:
      cpu: 100m
      memory: 256Mi
meshConfig:
  defaultConfig:
    holdApplicationUntilProxyStarts: true
EOF

cat > gateway-values.yaml <<'EOF'
replicaCount: 1
service:
  type: NodePort             # GCP 방화벽 규칙에 맞춰 NodePort 사용
EOF
```

### 설치 순서 (base → istiod → gateway)

`--kube-context` 를 명시해 다른 클러스터에 실수로 배포하는 사고를 막습니다.

```bash
CTX=$(kubectl config current-context)

# Step 1: CRD 먼저 - 순서를 지키지 않으면 istiod 설치가 CRD 미존재로 실패
helm install istio-base ./base \
  -n istio-system --create-namespace \
  --kube-context "$CTX"

# Step 2: 컨트롤 플레인
helm install istiod ./istiod \
  -n istio-system \
  -f istiod-values.yaml \
  --wait \
  --kube-context "$CTX"

# Step 3: 인그레스 게이트웨이
kubectl create namespace istio-ingress --dry-run=client -o yaml | kubectl apply -f -
helm install istio-ingress ./gateway \
  -n istio-ingress \
  -f gateway-values.yaml \
  --kube-context "$CTX"
```

### 배포 검증

Helm 3는 릴리스 메타데이터를 네임스페이스 내 `Secret` 으로 저장합니다. 클러스터에 상주하는 Helm 서버는 없습니다.

```bash
# 릴리스 목록
helm list -A
# NAME          NAMESPACE      REVISION  STATUS    CHART
# istio-base    istio-system   1         deployed  base-1.x
# istiod        istio-system   1         deployed  istiod-1.x
# istio-ingress istio-ingress  1         deployed  gateway-1.x

# 릴리스 메타데이터 실체 확인
kubectl get secret -n istio-system -l owner=helm
# sh.helm.release.v1.istiod.v1 ...

# 렌더링 결과만 사전 확인하고 싶을 때 (설치 없이)
helm template istiod ./istiod -f istiod-values.yaml | less

# Pod 상태
kubectl get pods -n istio-system
kubectl get pods -n istio-ingress
```

### 업그레이드와 롤백

values를 수정해 `helm upgrade` 만 실행하면 됩니다. 실패 시 `helm rollback` 으로 직전 리비전으로 즉시 돌아갑니다.

```bash
# replicaCount 변경 후 업그레이드
sed -i '' 's/replicaCount: 1/replicaCount: 2/' istiod-values.yaml
helm upgrade istiod ./istiod -n istio-system -f istiod-values.yaml

# 히스토리 확인
helm history istiod -n istio-system

# 롤백 (REVISION 1로)
helm rollback istiod 1 -n istio-system
```

### 자주 막히는 지점

| 증상 | 원인 | 해결 |
|------|------|------|
| `no matches for kind "Telemetry"` | base 차트 미설치로 CRD 부재 | `istio-base` 를 먼저 설치 |
| 로컬에서 `kubectl get nodes` 타임아웃 | 로컬→GCP API 서버(6443) 방화벽 차단 | GCP 방화벽 규칙에 내 공인 IP 허용, 또는 SSH 터널 `ssh -L 6443:localhost:6443 dev-server` |
| Pod `ImagePullBackOff` | 노드가 `docker.io/istio/*` 이미지를 못 받음 (폐쇄망) | `global.hub` 를 Artifact Registry/Harbor 주소로 오버라이드 + 이미지 사전 push |
| `helm list` 가 비어 있음 | kubeconfig 컨텍스트가 다른 클러스터 | `kubectl config current-context` 로 확인 후 `--kube-context` 재지정 |

### 제거

```bash
helm uninstall istio-ingress -n istio-ingress
helm uninstall istiod        -n istio-system
helm uninstall istio-base    -n istio-system
kubectl delete namespace istio-ingress
```


## 실습 3: 사이드카 인젝션 확인

```bash
# 네임스페이스에 인젝션 활성화
kubectl create namespace install-demo
kubectl label namespace install-demo istio-injection=enabled

# 앱 배포
kubectl create deployment web --image=nginx -n install-demo
kubectl wait --for=condition=ready pod -l app=web -n install-demo

# 사이드카 컨테이너 확인
kubectl get pod -n install-demo \
  -l app=web -o jsonpath='{.items[0].spec.containers[*].name}'
# 출력: web istio-proxy
---

# 인젝션 비활성화 (Pod 레벨)
kubectl run no-sidecar \
  --image=nginx \
  -n install-demo \
  --annotations="sidecar.istio.io/inject=false"

kubectl get pod no-sidecar -n install-demo \
  -o jsonpath='{.spec.containers[*].name}'
# 출력: no-sidecar (istio-proxy 없음)
```


## 실습 4: istioctl analyze로 설정 검증

```bash
# 현재 설정 분석
istioctl analyze -n install-demo

# 파일 기반 분석 (CI 파이프라인용)
cat <<EOF > test-vs.yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: test-vs
  namespace: install-demo
spec:
  hosts:
  - "missing-service"   # 존재하지 않는 서비스
  http:
  - route:
    - destination:
        host: missing-service
        port:
          number: 8080
EOF

istioctl analyze test-vs.yaml
# 출력: Warning: VirtualService references host not found
rm test-vs.yaml
```


## 실습 5: 리비전 기반 카나리 업그레이드 시뮬레이션

```bash
# 현재 버전으로 리비전 레이블 부착 설치
istioctl install --set profile=default \
  --set revision=stable -y

# 확인
kubectl get pods -n istio-system | grep istiod

# 새 리비전 설치 (버전 업그레이드 시뮬레이션)
istioctl install --set profile=default \
  --set revision=canary -y

# 두 istiod 공존 확인
kubectl get pods -n istio-system | grep istiod
# istiod-stable-xxx   Running
# istiod-canary-xxx   Running
---

# 테스트 네임스페이스를 새 리비전으로 전환
kubectl label namespace install-demo \
  istio.io/rev=canary --overwrite

# Pod 롤링 재시작으로 새 사이드카 적용
kubectl rollout restart deployment/web -n install-demo

# 새 사이드카 버전 확인
kubectl get pod -n install-demo -l app=web \
  -o jsonpath='{.items[0].metadata.annotations}'

# 이전 리비전 제거 (모든 Pod 전환 후)
istioctl uninstall --revision=stable -y
```

### 롤백 절차

```bash
# 문제 발생 시 이전 리비전으로 롤백
kubectl label namespace install-demo \
  istio.io/rev=stable --overwrite

kubectl rollout restart deployment/web -n install-demo

# 이전 리비전 istiod가 살아있는지 확인 (너무 빨리 삭제하지 않도록)
kubectl get pods -n istio-system | grep istiod-stable
```


## 실습 6: 앰비언트 모드 설치

```bash
# 기존 사이드카 Istio 제거 후 앰비언트 모드 설치
istioctl uninstall --purge -y

# 앰비언트 프로파일로 설치
istioctl install --set profile=ambient -y

# 앰비언트 구성 요소 확인
kubectl get pods -n istio-system
# ztunnel (DaemonSet), istiod, istio-cni
---

# 네임스페이스를 앰비언트 메시에 등록 (Pod 재시작 불필요)
kubectl label namespace install-demo \
  istio.io/dataplane-mode=ambient

# 메시 참여 확인
kubectl get pods -n install-demo \
  -o jsonpath='{.items[*].metadata.annotations}' \
  | python3 -m json.tool | grep ambient
```


## 정리

```bash
kubectl delete namespace install-demo
istioctl uninstall --purge -y
kubectl delete namespace istio-system
```
