# Istio Ingress Gateway 실습

> 본 장의 개념을 손으로 확인합니다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`istioctl` 범용 명령으로 씁니다.


## 실습 환경

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| 서비스 메시 | Istio (사이드카 주입 네임스페이스) | — |

접속은 `gcp` 스킬의 SSH 절차를 따릅니다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1: 기본 HTTP Gateway 설정

### 목표

외부 트래픽을 Gateway → VirtualService → 내부 서비스로 연결합니다.

```bash
# 테스트 네임스페이스 및 앱 준비
kubectl create namespace gw-demo
kubectl label namespace gw-demo istio-injection=enabled

kubectl create deployment httpbin --image=kennethreitz/httpbin -n gw-demo
kubectl expose deployment httpbin --port=80 -n gw-demo

kubectl wait --for=condition=ready pod -l app=httpbin -n gw-demo
```

### Gateway + VirtualService 적용

```yaml
# gateway-basic.yaml
apiVersion: networking.istio.io/v1
kind: Gateway
metadata:
  name: demo-gateway
  namespace: istio-system
spec:
  selector:
    istio: ingressgateway
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "httpbin.example.com"
---
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: httpbin-vs
  namespace: gw-demo
spec:
  hosts:
  - "httpbin.example.com"
  gateways:
  - istio-system/demo-gateway
  http:
  - route:
    - destination:
        host: httpbin
        port:
          number: 80
```

```bash
kubectl apply -f gateway-basic.yaml

# Ingress Gateway 외부 IP 확인
kubectl get svc istio-ingressgateway -n istio-system

# 테스트 (GCP 환경에서 NodePort 사용 시)
INGRESS_IP=$(kubectl get svc istio-ingressgateway -n istio-system \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

curl -H "Host: httpbin.example.com" \
  http://${INGRESS_IP}/get
```


## 실습 2: TLS 설정 (SIMPLE 모드)

```bash
# 자체 서명 인증서 생성
openssl req -x509 -newkey rsa:4096 \
  -keyout tls.key -out tls.crt \
  -days 365 -nodes \
  -subj "/CN=httpbin.example.com"

# Secret 생성 (istio-system 네임스페이스)
kubectl create secret tls httpbin-tls \
  --cert=tls.crt --key=tls.key \
  -n istio-system
```

```yaml
# gateway-tls.yaml
apiVersion: networking.istio.io/v1
kind: Gateway
metadata:
  name: demo-gateway-tls
  namespace: istio-system
spec:
  selector:
    istio: ingressgateway
  servers:
  - port:
      number: 443
      name: https
      protocol: HTTPS
    tls:
      mode: SIMPLE
      credentialName: httpbin-tls
    hosts:
    - "httpbin.example.com"
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "httpbin.example.com"
    tls:
      httpsRedirect: true
```

```bash
kubectl apply -f gateway-tls.yaml

# HTTPS 테스트
curl -k -H "Host: httpbin.example.com" \
  https://${INGRESS_IP}/get

# HTTP → HTTPS 리다이렉트 확인
curl -v -H "Host: httpbin.example.com" \
  http://${INGRESS_IP}/get 2>&1 | grep -i "location\|301"
```


## 실습 3: 멀티 호스트 라우팅 (SNI)

```bash
# 두 번째 앱 배포
kubectl create deployment nginx --image=nginx -n gw-demo
kubectl expose deployment nginx --port=80 -n gw-demo
```

```yaml
# multi-host-vs.yaml
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: nginx-vs
  namespace: gw-demo
spec:
  hosts:
  - "nginx.example.com"
  gateways:
  - istio-system/demo-gateway
  http:
  - route:
    - destination:
        host: nginx
        port:
          number: 80
```

```bash
kubectl apply -f multi-host-vs.yaml

# 두 호스트 모두 테스트
curl -H "Host: httpbin.example.com" http://${INGRESS_IP}/get
curl -H "Host: nginx.example.com" http://${INGRESS_IP}/
```

단일 LoadBalancer IP에서 `Host` 헤더로 다른 백엔드에 라우팅되는 것을 확인합니다.


## 실습 4: Kubernetes Gateway API로 변환

```bash
# Gateway API CRD 설치 (없다면)
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.0.0/standard-install.yaml
```

```yaml
# k8s-gateway-api.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: k8s-gateway
  namespace: gw-demo
spec:
  gatewayClassName: istio
  listeners:
  - name: http
    port: 80
    protocol: HTTP
    hostname: "httpbin-k8s.example.com"
    allowedRoutes:
      namespaces:
        from: Same
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: httpbin-route
  namespace: gw-demo
spec:
  parentRefs:
  - name: k8s-gateway
    namespace: gw-demo
  hostnames:
  - "httpbin-k8s.example.com"
  rules:
  - backendRefs:
    - name: httpbin
      port: 80
```

```bash
kubectl apply -f k8s-gateway-api.yaml

# Gateway API로 생성된 Gateway Pod 확인
kubectl get gateway -n gw-demo
kubectl get httproute -n gw-demo

# 테스트
curl -H "Host: httpbin-k8s.example.com" http://${INGRESS_IP}/get
```

Istio Gateway API와 Kubernetes Gateway API가 동일한 클러스터에서 공존할 수 있다는 점을 확인합니다.


## 실습 5: istioctl analyze로 Gateway 설정 검증

```bash
# 현재 Gateway 설정 분석
istioctl analyze -n gw-demo
istioctl analyze -n istio-system

# VirtualService가 Gateway와 불일치할 때
cat <<EOF | kubectl apply -f -
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: broken-gateway-vs
  namespace: gw-demo
spec:
  hosts:
  - "notingateway.example.com"  # Gateway에 없는 호스트
  gateways:
  - istio-system/demo-gateway
  http:
  - route:
    - destination:
        host: httpbin
        port:
          number: 80
EOF

istioctl analyze -n gw-demo
# 출력: VirtualService hosts not in Gateway
---

kubectl delete virtualservice broken-gateway-vs -n gw-demo
```


## 정리

```bash
kubectl delete namespace gw-demo
kubectl delete gateway demo-gateway demo-gateway-tls -n istio-system
kubectl delete secret httpbin-tls -n istio-system
rm -f tls.key tls.crt gateway-basic.yaml gateway-tls.yaml \
  multi-host-vs.yaml k8s-gateway-api.yaml
```
