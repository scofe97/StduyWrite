# Istio 레질리언스

> 트래픽을 나누는 것만으로는 운영이 끝나지 않는다. 다음 단계는 실패를 어떻게 가두고, 지연을 어디서 끊고, 재시도를 얼마나 허용할지 정하는 레질리언스 설계이며, Istio는 이를 VirtualService와 DestinationRule 위에서 조합한다.


## 학습 목표

> 로드 밸런싱 알고리즘 선택부터 타임아웃·재시도·서킷 브레이킹 조합까지, Istio 레질리언스 메커니즘의 동작 원리와 상호작용을 다룬다.

1. Envoy의 로드 밸런싱 알고리즘 차이를 설명하고 Fortio로 실제 성능을 비교한다.
2. VirtualService로 타임아웃을 설정하고 서비스 체인에서 타임아웃 전파 원칙을 적용한다.
3. 재시도 기본 동작과 `retryOn` 조건을 이해하고 EnvoyFilter로 고급 재시도를 구성한다.
4. 서킷 브레이커의 connectionPool과 Outlier Detection 설정을 작성하고 차이를 구별한다.
5. 타임아웃·재시도·서킷 브레이킹 조합 시 발생하는 상호작용을 설명한다.
6. `x-envoy-overloaded` 헤더와 `503 UO` 응답 플래그로 서킷 브레이커 동작을 식별한다.


## 1. 클라이언트 측 로드 밸런싱

> 서비스 메시에서 로드 밸런싱은 kube-proxy가 아닌 Envoy가 담당하며, 요청 처리 시간이 균일하지 않은 환경에서는 LEAST_REQUEST가 ROUND_ROBIN보다 효과적이다.

서비스 메시에서 로드 밸런싱은 kube-proxy가 아닌 사이드카 Envoy가 담당한다. Envoy는 Pilot로부터 EDS를 통해 엔드포인트 목록을 실시간으로 수신하고 직접 연결을 선택한다.

**ROUND_ROBIN**은 기본값이다. 모든 엔드포인트의 처리 속도가 비슷할 때 공정한 분산을 제공하지만, 하나가 느려지면 그 엔드포인트로 보내진 요청이 큐에 쌓이는 동안 다른 빠른 엔드포인트는 유휴 상태가 된다.

**LEAST_REQUEST**는 현재 활성 요청 수가 가장 적은 엔드포인트를 선택한다. 처리 시간이 균일하지 않은 요청(10ms와 500ms가 섞인 환경)에서 가장 효과적이다. 빠른 엔드포인트는 더 많은 요청을 받고, 느린 엔드포인트는 처리가 줄어들 때까지 새 요청을 적게 받는다.

Istio가 Round Robin을 기본값으로 채택한 이유는 상태 관리 비용 때문이다. Least Request는 각 Envoy가 업스트림 엔드포인트의 활성 요청 수를 실시간으로 추적해야 하므로, 대규모 클러스터에서 이 오버헤드가 무시할 수 없다.


## 2. 지역 인식 로드 밸런싱

> 같은 AZ 내 엔드포인트를 우선 선택해 크로스 AZ 레이턴시와 비용을 줄이며, 노드 레이블과 outlierDetection이 모두 갖춰져야 페일오버가 동작한다.

지역 인식 로드 밸런싱은 같은 가용 영역(AZ)의 엔드포인트를 우선 선택해 크로스 AZ 레이턴시와 비용을 줄인다. 동작하려면 두 전제 조건이 충족되어야 한다.

1. 노드에 `topology.kubernetes.io/region`과 `topology.kubernetes.io/zone` 레이블이 있어야 한다.
2. DestinationRule에 `outlierDetection`이 설정되어야 로컬 파드 장애 시 다른 AZ로 페일오버된다.


## 3. 타임아웃

> 타임아웃은 P99 레이턴시의 1.5~2배를 기준으로 설정하며, 서비스 체인에서는 바깥쪽에서 안쪽으로 갈수록 값이 짧아야 불필요한 연산 낭비를 막는다.

타임아웃이 없으면 느린 서비스가 Envoy의 요청 큐와 연결 풀을 가득 채워 연쇄 장애를 일으킬 수 있다. 타임아웃 값 설정 기준은 P99 레이턴시의 1.5~2배가 일반적인 시작점이다.

서비스 체인(A → B → C)에서 타임아웃은 바깥쪽에서 안쪽으로 갈수록 짧아야 한다. A의 타임아웃 내에서 B의 처리(B 자체 로직 + C 호출 + 마진)가 완료되어야 하기 때문이다. 안쪽 서비스의 타임아웃이 바깥쪽보다 길면 A는 이미 클라이언트에게 에러를 반환했는데 B와 C는 불필요한 작업을 계속하며 리소스를 소비한다.

```
클라이언트: 2000ms → 서비스 A(200ms): 1500ms → 서비스 B(300ms): 1000ms → 서비스 C
```


## 4. 재시도

> Istio 기본 재시도는 2회이며, 멱등성 없는 POST 경로는 반드시 비활성화해야 하고, 재시도 폭주를 막으려면 서킷 브레이커와 함께 사용해야 한다.

Istio는 기본적으로 시도 횟수 2회, 조건 `5xx,retriable-4xx,connect-failure,reset`으로 재시도를 적용한다. 멱등성이 없는 POST 요청에 자동 재시도가 적용되면 동일 요청이 두 번 처리될 수 있으므로, 해당 경로는 `attempts: 0`으로 재시도를 비활성화해야 한다.

```yaml
retries:
  attempts: 3
  perTryTimeout: 2s
  retryOn: 5xx,reset,connect-failure,retriable-4xx
```

`perTryTimeout`은 전체 타임아웃 안에서 각 시도에 허용하는 최대 시간이다. `timeout >= perTryTimeout * attempts`를 만족하도록 설계해야 재시도가 의미 있다.

재시도 폭주(retry storm)를 막으려면 `attempts`를 낮게 유지하고 서킷 브레이커와 함께 사용해야 한다. 업스트림 에러율이 높아지면 서킷 브레이커가 요청 자체를 차단해 재시도가 발생하지 않게 한다. 또한 `retryOn` 조건에서 `503`을 제외하면 서킷 브레이커가 차단한 응답에 대한 재시도를 방지할 수 있다.


## 5. 서킷 브레이킹

> connectionPool은 요청 수 자체를 제한하고, Outlier Detection은 에러를 반환하는 엔드포인트를 격리하며, 두 방어선을 함께 써야 서킷 브레이킹이 완성된다.

### 5.1 연결 풀 제어

서킷 브레이킹의 첫 번째 방어선은 연결 풀 제한이다. `http1MaxPendingRequests`를 초과하면 새 요청은 즉시 `503 Upstream Overflow`로 반환된다. 대기 큐를 너무 크게 잡으면 장애가 전파되는 시간만 늘어난다.

### 5.2 Outlier Detection

Outlier Detection은 실제로 에러를 반환하는 엔드포인트를 자동으로 로드 밸런싱 대상에서 제외하는 서킷 브레이커다. connectionPool이 "요청 수 제한"이라면, Outlier Detection은 "건강하지 않은 Pod 격리"다.

eject된 엔드포인트는 `baseEjectionTime * eject_횟수` 시간이 지나면 다시 풀에 추가된다. 처음 eject는 30초, 두 번째는 60초, 세 번째는 90초처럼 점점 길어진다. 이 백오프 방식은 간헐적으로 회복하는 엔드포인트가 너무 빨리 복귀해 또 에러를 일으키는 상황을 방지한다.

`consecutive5xxErrors`는 HTTP 5xx 응답과 로컬 오류(타임아웃, 연결 거부)를 모두 카운팅한다. `consecutiveGatewayErrors`는 502·503·504와 로컬 오류만 카운팅한다. 500이 주로 비즈니스 로직 오류에서 발생하는 서비스라면 `consecutiveGatewayErrors`를 쓰는 것이 오탐을 줄인다.

### 5.3 서킷 브레이커 식별

Envoy가 서킷 브레이커 때문에 요청을 차단하면 응답에 `x-envoy-overloaded: true` 헤더를 포함하고, Access Log에 `503 UO`(Upstream Overflow) 플래그를 기록한다.


## 6. 복원력 전략 조합

> 타임아웃·재시도·서킷 브레이킹은 각각 독립적으로 동작하지만 잘못 조합하면 서로 상쇄되므로, `timeout >= perTryTimeout * attempts` 등 수치 관계를 반드시 검증해야 한다.

**타임아웃은 재시도 총 시간보다 커야 한다.** `timeout >= perTryTimeout * attempts`를 만족하지 않으면 재시도가 의미 없어진다.

**connectionPool과 재시도는 곱셈 관계다.** 동시 요청 100개에 재시도 3회가 설정되어 있다면, 최악의 경우 업스트림은 300개 요청을 처리해야 한다. connectionPool 설정 시 재시도 횟수를 고려해야 한다.

```yaml
# VirtualService: 타임아웃 + 재시도
timeout: 6s
retries:
  attempts: 3
  perTryTimeout: 2s
  retryOn: connect-failure,reset,retriable-4xx  # 503 제외

# DestinationRule: 서킷 브레이커
connectionPool:
  http:
    http1MaxPendingRequests: 50
outlierDetection:
  consecutive5xxErrors: 5
  interval: 10s
  baseEjectionTime: 30s
  maxEjectionPercent: 50
```

연쇄 장애 방지를 위해 체인의 끝으로 갈수록 타임아웃이 짧아지고 재시도 횟수가 줄어들어야 한다. Outlier Detection은 체인의 끝, 가장 불안정한 외부 의존성 서비스에 배치하면 효과가 크다.


## 면접 대비

> 재시도와 서킷 브레이커의 상호작용, LEAST_REQUEST를 기본값으로 쓰지 않는 이유 등 Istio 레질리언스 메커니즘의 트레이드오프를 설명할 수 있어야 한다.

**Least Request가 기본값이 아닌 이유는?** Least Request는 각 Envoy가 업스트림 엔드포인트의 활성 요청 수를 실시간으로 추적해야 한다. 수백 개 인스턴스 환경에서 이 상태 정보를 유지하고 동기화하는 오버헤드가 무시할 수 없어, 상태가 없는 Round Robin이 기본값으로 채택됐다.

**재시도가 서킷 브레이커를 트리거할 수 있는가?** 가능하다. 재시도도 동일한 업스트림 인스턴스로 전달되므로 Outlier Detection의 오류 카운터에 합산된다. 재시도 횟수가 3이라면 실제 요청 1건이 인스턴스 오류 카운터를 최대 3까지 올릴 수 있다. 이를 방지하려면 `consecutive5xxErrors`를 `retries.attempts + 2` 이상으로 설정하거나, `retryOn`에서 `5xx`를 세밀하게 조정한다.
