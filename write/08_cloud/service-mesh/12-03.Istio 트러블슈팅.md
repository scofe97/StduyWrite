# Istio 트러블슈팅

> 관측성이 상태를 보여 준다면, 트러블슈팅은 그 신호를 원인으로 연결하는 단계다. Istio에서는 먼저 데이터 플레인 문제인지 컨트롤 플레인 문제인지 구분하고, `istioctl proxy-status`, `proxy-config`, Envoy Admin Interface, Response Flag를 순서대로 좁혀 가는 접근이 가장 효율적이다.



## 학습 목표

> 데이터/컨트롤 플레인 구분부터 Listener→Route→Cluster→Endpoint 추적, Response Flag 해석까지 Istio 트러블슈팅의 체계적 접근법을 다룬다.

1. 데이터 플레인과 컨트롤 플레인 문제를 `istioctl proxy-status`로 구분한다.
2. `istioctl analyze`와 `istioctl describe`로 설정 오류를 정적·런타임 두 단계에서 탐지한다.
3. Envoy Admin Interface와 `istioctl proxy-config` 명령어 체계를 통해 Listener → Route → Cluster → Endpoint 흐름으로 문제를 추적한다.
4. Envoy Response Flag(UT, UO, UC, NR, DC)의 의미를 해석하고 대응 방향을 결정한다.
5. Grafana 클라이언트/서버 관점 실패율 비교로 문제 범위를 좁힌다.



## 1. 트러블슈팅 접근 방법론

> "데이터 플레인 문제인가, 컨트롤 플레인 문제인가"를 먼저 구분해야 하며, `istioctl proxy-status`의 SYNCED/STALE/NOT SENT 상태가 그 첫 번째 판단 기준이다.

### 1.1 데이터 플레인 vs 컨트롤 플레인 구분

Istio 트러블슈팅에서 가장 먼저 해야 할 질문은 "문제가 데이터 플레인(Envoy 프록시)에 있는가, 컨트롤 플레인(istiod)에 있는가"다. 이 구분 없이 바로 로그를 뒤지기 시작하면 수십 분을 엉뚱한 곳에서 보내게 된다.

데이터 플레인 문제는 주로 트래픽 처리 단계에서 드러난다. 특정 서비스로의 요청이 503이나 504를 반환하거나, 연결이 갑자기 끊기거나, 타임아웃이 빈번하게 발생한다. 컨트롤 플레인 문제는 설정 변경 이후에 나타난다. VirtualService나 DestinationRule을 배포했는데 반영이 안 되거나, `istioctl proxy-status`에서 STALE 상태가 여러 프록시에 걸쳐 나타날 때다.

### 1.2 proxy-status로 동기화 상태 확인

`istioctl proxy-status`는 컨트롤 플레인이 각 Envoy 프록시에 설정을 성공적으로 배포했는지 확인하는 첫 번째 진단 도구다.

```bash
istioctl proxy-status
```

상태값은 세 가지다. `SYNCED`는 최신 설정이 프록시에 정상 반영된 상태다. `STALE`은 istiod가 설정을 전송했지만 프록시가 아직 ACK를 돌려보내지 않은 상태로, 오래 지속되면 네트워크 문제나 프록시 과부하를 의심한다. `NOT SENT`는 istiod가 아직 해당 프록시에 설정을 전송하지 않은 상태다.



## 2. 설정 오류 탐지

> `istioctl analyze`는 배포 전 정적 분석으로 참조 무결성 오류를 잡고, `istioctl describe`는 런타임에 특정 Pod에 실제로 적용 중인 정책을 확인한다.

### 2.1 istioctl analyze — 정적 분석

`istioctl analyze`는 클러스터에 배포된 Istio 설정 리소스를 정적으로 분석해 오류와 경고를 보고한다. CI/CD 파이프라인에 통합하기 좋다.

```bash
# 현재 클러스터 전체 분석
istioctl analyze

# 특정 네임스페이스
istioctl analyze -n production

# 로컬 파일 분석 (배포 전 검증)
istioctl analyze my-virtualservice.yaml --use-kube=false
```

가장 자주 마주치는 오류는 `IST0101`이다. VirtualService가 참조하는 DestinationRule subset이 실제로 존재하지 않을 때 발생하며, 이 오류가 배포되면 해당 라우트로 보내지는 모든 트래픽이 503 NC로 실패한다. `analyze`가 탐지하는 오류 클래스는 세 가지다. 참조 무결성 오류(`IST0101`), 포트 프로토콜 오류(`IST0118`, 접두사 없는 포트 이름), 게이트웨이 오류(`IST0132`, 존재하지 않는 Gateway 참조)다.

단, `analyze`가 탐지하지 못하는 클래스도 있다. DestinationRule에 subset을 정의했지만 해당 레이블을 가진 Pod가 0개인 경우는 배포 후 `istioctl proxy-config endpoint`로 확인해야 한다.

### 2.2 istioctl describe — 런타임 상태 확인

`istioctl describe`는 특정 Pod가 실제로 어떤 Istio 정책 아래에 놓여 있는지를 런타임 관점에서 보여준다.

```bash
istioctl describe pod <pod-name> -n <namespace>
```

"This pod is NOT on the mesh" 메시지가 나오면 사이드카가 주입되지 않은 상태다. 적용된 VirtualService 라우팅 규칙 요약, mTLS 모드, 적용 중인 AuthorizationPolicy도 함께 표시된다.



## 3. Envoy 프록시 디버깅

> Listener→Route→Cluster→Endpoint 순서로 추적하면 요청이 어느 레이어에서 막히는지 정확히 찾을 수 있으며, 각 레이어에서 문제를 발견하면 그에 해당하는 Istio 리소스만 수정하면 된다.

### 3.1 proxy-config 명령어 체계 (Listener → Route → Cluster → Endpoint)

`istioctl proxy-config`는 요청이 처리되는 실제 흐름 순서로 문제를 추적한다. 이 순서를 따르면 "어느 레이어에서 요청이 막히는지"를 정확히 찾을 수 있다.

```bash
# 1. Listener: 특정 포트에 리스너가 존재하는지 확인
istioctl proxy-config listener <pod-name>.<namespace> --port 9080

# 2. Route: VirtualService 라우팅 규칙이 반영됐는지 확인
istioctl proxy-config route <pod-name>.<namespace> --name 9080

# 3. Cluster: DestinationRule subset 설정이 반영됐는지 확인
istioctl proxy-config cluster <pod-name>.<namespace> \
  --fqdn reviews.default.svc.cluster.local

# 4. Endpoint: 실제 Pod IP가 클러스터에 등록됐는지 확인
istioctl proxy-config endpoint <pod-name>.<namespace> \
  --cluster "outbound|9080|v2|reviews.default.svc.cluster.local"
```

리스너가 없다면 서비스 포트 설정이나 사이드카 주입 문제다. Cluster는 있는데 Endpoint가 비어있다면 해당 subset 레이블을 가진 Pod가 없거나 Ready 상태가 아닌 것이다. 한 단계에서 문제를 발견하면 그 레이어에 해당하는 Istio 리소스를 타겟으로 수정한다.

### 3.2 Envoy Admin Interface (포트 15000)

각 Envoy 프록시는 포트 15000에 Admin Interface를 노출한다.

```bash
# 포트 포워딩으로 접근
kubectl port-forward <pod-name> 15000:15000 -n <namespace>
```

주요 엔드포인트는 `/config_dump`(전체 설정 JSON), `/stats/prometheus`(메트릭), `/clusters`(업스트림 클러스터 헬스), `/listeners`(바인딩된 리스너), `/logging`(로그 레벨 조회/변경)이다. Admin Interface는 인증 메커니즘이 없으므로 프로덕션에서는 상시 노출하지 않고 `kubectl port-forward`를 통한 on-demand 접근이 안전하다.

### 3.3 로그 레벨 조정

기본 로그 레벨에서는 상세 원인 파악이 어렵다. Envoy Admin Interface를 통해 동적으로 `debug`로 올릴 수 있다.

```bash
# debug 레벨로 변경
kubectl exec <pod-name> -c istio-proxy -- curl -X POST \
  "localhost:15000/logging?connection=debug&http=debug&router=debug&pool=debug"

# 문제 재현 후 info로 복원
kubectl exec <pod-name> -c istio-proxy -- curl -X POST \
  "localhost:15000/logging?level=info"
```



## 4. Envoy Response Flags

> 같은 503이라도 NR이면 VirtualService 라우팅을 수정해야 하고 UO이면 업스트림 용량을 늘려야 하므로, Response Flag가 HTTP 상태 코드보다 훨씬 정확한 원인 진단 정보를 제공한다.

Response Flag는 Access Log에 기록되는 짧은 코드로, 실패 원인을 즉시 알 수 있다. 같은 503이라도 `NR`이면 VirtualService를 고쳐야 하고, `UO`이면 업스트림 처리 용량을 늘려야 한다.

| Flag | 의미 | 주요 원인 |
|------|------|----------|
| `UO` | Upstream Overflow (서킷 브레이커 트리거) | connectionPool 임계값 초과 |
| `UC` | Upstream Connection termination | 업스트림이 연결을 먼저 종료 |
| `UT` | Upstream request Timeout | VirtualService timeout 초과 |
| `NR` | No Route found | VirtualService 라우팅 규칙에 매칭 없음 |
| `DC` | Downstream Connection termination | 클라이언트가 연결을 먼저 종료 |
| `NC` | No Cluster found | DestinationRule subset이 존재하지 않음 |
| `URX` | Upstream retry limit exceeded | 재시도 횟수 초과 |
| `UH` | No healthy Upstream host | 헬스체크를 통과한 업스트림 없음 |

`DC`는 클라이언트 타임아웃이 서버 타임아웃보다 짧거나 사용자가 요청을 취소한 경우다. DC 자체는 서버 측 문제가 아니므로 에러율 계산에서 `response_flags!="DC"` 필터로 제외하는 것이 일반적이다. NR이나 NC는 설정 오류를 의미하므로 발생 즉시 알림을 보내야 한다.



## 5. 텔레메트리 기반 진단

> 클라이언트 실패율이 서버 실패율보다 높으면 클라이언트 사이드카가 요청을 자체 거부한 것이고, 두 값이 같으면 서버 애플리케이션 문제로 범위를 좁힌다.

### 5.1 클라이언트/서버 관점 실패율 비교

같은 요청에 대해 두 관점의 성공률이 다르면 문제의 위치를 알 수 있다.

```
클라이언트 실패 > 서버 실패: 클라이언트와 서버 사이 네트워크/프록시 문제 (타임아웃, 서킷 브레이커 등)
클라이언트 실패 = 서버 실패: 서버 애플리케이션 자체 문제
```

클라이언트 관점에서만 실패로 기록된다면 클라이언트 Pod의 Envoy Access Log를 먼저 확인한다. 서버 Envoy도 아무 기록이 없다면 클라이언트 사이드카가 요청을 서버까지 전달하지 않고 자체 거부한 것이다.

### 5.2 Prometheus 쿼리로 원인 분류

```promql
# Response Flag별 에러 분포 (원인 자동 분류)
sum by (response_flags) (
  irate(
    istio_requests_total{
      response_code=~"5..",
      destination_service_namespace="production"
    }[5m]
  )
)

# UO (서킷 브레이커) 에러 비율
rate(istio_requests_total{response_flags="UO"}[5m])

# DC 제외 실제 서버측 에러율
rate(istio_requests_total{
  response_code=~"5..",
  response_flags!="DC"
}[5m])
```

`UO`(서킷 브레이커)가 지배적이라면 업스트림 과부하 문제, `UT`(타임아웃)가 지배적이라면 성능 저하 문제로 방향을 잡는다.



## 6. 실전 트러블슈팅 시나리오

> NC·UT·UO 세 가지 대표 장애 패턴은 각각 설정 오류·타임아웃 과소설정·connectionPool 임계값 초과로 원인과 해결 방향이 명확히 구분된다.

### 시나리오 1. DestinationRule 누락으로 503 NC

**증상**: Access Log에 `response_flags: NC` 기록.

**원인**: VirtualService가 `subset: v2`로 라우팅하도록 설정됐는데, DestinationRule에 `v2` subset이 정의되지 않았다.

**해결 절차**: (1) `istioctl analyze`로 `IST0101` 경고 확인, (2) DestinationRule에 누락된 subset 추가, (3) 배포 후 `istioctl proxy-status`로 동기화 확인.

### 시나리오 2. 간헐적 지연으로 504 UT

**증상**: Access Log에 `response_flags: UT` 기록.

**원인**: VirtualService에 500ms 타임아웃이 설정되어 있는데, 서비스의 p99 레이턴시가 600ms를 넘는 순간에 타임아웃이 발생한다.

**해결 절차**: (1) Prometheus에서 실제 p99 레이턴시 측정, (2) VirtualService 타임아웃 값과 비교, (3) 타임아웃을 p99 × 1.5로 조정하거나 업스트림 처리 속도 개선.

### 시나리오 3. 서킷 브레이커로 503 UO

**증상**: 부하 시 Access Log에 `response_flags: UO` 기록.

**원인**: DestinationRule의 `connectionPool` 임계값이 현재 트래픽을 감당하지 못해 서킷 브레이커가 트리거된다.

**해결 절차**: (1) Prometheus에서 `UO` 플래그 발생 빈도 확인, (2) Envoy Admin `/stats`에서 `overflow` 카운터 확인, (3) `connectionPool` 임계값 상향 조정 + 업스트림 스케일 아웃 검토.



## 7. 컨트롤 플레인 디버깅

> istiod 디버그 엔드포인트는 xDS 동기화 상태와 인가 정책 파싱 결과를 직접 확인할 수 있으나, `/debug/adsz` 반복 호출은 istiod 부하를 높이므로 가벼운 도구부터 순서대로 사용해야 한다.

istiod는 포트 8080에서 디버그 엔드포인트를 제공한다.

```bash
kubectl port-forward deployment/istiod 8080:8080 -n istio-system
```

주요 엔드포인트는 다음과 같다. `/debug/adsz`는 현재 연결된 모든 프록시와 배포된 xDS 리소스 목록이다. `/debug/syncz`는 각 프록시의 동기화 상태 요약으로 `proxy-status`보다 상세한 타임스탬프 정보가 포함된다. `/debug/authorizationz`는 현재 적용 중인 AuthorizationPolicy를 파싱된 형태로 보여준다.

대규모 클러스터에서 `/debug/adsz`를 반복 호출하면 istiod 부하가 증가한다. 진단 우선순위는 `istioctl proxy-status` → `istioctl proxy-config` → `/debug/adsz` 순서로 가볍고 범위가 좁은 것부터 사용한다.



## 면접 대비

> proxy-config 추적 순서의 이유, 클라이언트/서버 실패율 비교 분석법, `istioctl analyze`의 탐지 범위와 한계가 주요 면접 주제다.

**proxy-config의 Listener → Route → Cluster → Endpoint 추적 순서가 왜 중요한가?** 요청이 프록시를 통과하는 물리적 순서와 일치하기 때문이다. 순서를 건너뛰면 실제로는 Listener가 없어서 요청이 거부되는 경우를 놓칠 수 있다. 각 레이어에서 문제를 발견하면 그 레이어에 해당하는 Istio 리소스만 타겟으로 수정하면 된다.

**클라이언트 실패율과 서버 실패율이 다를 때 원인을 어떻게 좁히는가?** 클라이언트 실패율이 서버보다 높다면 클라이언트 사이드카가 요청을 서버까지 전달하지 않고 자체 거부한 것이다. 타임아웃(UT), 서킷 브레이커(UO), 라우팅 실패(NR, NC)가 여기 해당한다. 두 관점이 동일하다면 서버 애플리케이션이나 서버 사이드 프록시에서 문제가 발생하고 있다.

**`istioctl analyze`를 CI/CD에 통합하면 어떤 클래스의 오류를 잡을 수 있는가?** 참조 무결성 오류(존재하지 않는 subset 참조), 포트 프로토콜 오류(접두사 없는 포트 이름), 게이트웨이 오류(연결되지 않은 Gateway)를 사전에 탐지한다. 단, 런타임 엔드포인트 상태(Pod 레이블 매칭 여부)는 탐지하지 못한다.
