# eBPF와 Cilium

> eBPF는 커널을 재컴파일하지 않고 커널 안에서 프로그램을 안전하게 실행하는 기술입니다. Cilium은 이 eBPF를 CNI와 서비스 메시 양쪽에 활용해 iptables 없는 고성능 네트워킹, 사이드카 없는 mTLS, L7 관측성을 단일 에이전트로 제공합니다. 전통적 서비스 메시와 CNI의 경계를 허물고 있다는 점에서 Cilium은 단순한 도구가 아니라 Kubernetes 네트워킹의 패러다임 전환을 대표합니다.


## 학습 목표

> eBPF 핵심 개념(검증기·어태치 포인트·맵), iptables 대체로 얻는 성능, Cilium 아키텍처(에이전트·오퍼레이터·Hubble), L4/L7 처리 계층 분리까지 다섯 가지 목표를 다룹니다.

학습 목표는 다섯 가지입니다:

1. eBPF의 핵심 개념(검증기, 어태치 포인트, 맵)을 이해하고 설명할 수 있습니다.
2. eBPF가 iptables를 대체해 성능을 높이는 원리를 설명합니다.
3. Cilium 아키텍처(에이전트, 오퍼레이터, Hubble)의 구성 요소별 역할을 압니다.
4. Cilium의 L4(eBPF)와 L7(Envoy) 처리 계층 분리 방식을 이해합니다.
5. Hubble로 클러스터 트래픽 흐름을 관측하는 방법을 파악합니다.


## 1. eBPF란 무엇인가

> 커널 재컴파일 없이 안전하게 커널 안에서 프로그램을 실행하는 eBPF의 등장 배경, 역사, 검증기·JIT·맵 세 가지 핵심 요소를 설명합니다.

### 1.1 커널 확장의 딜레마

운영체제 커널의 동작을 바꾸려면 전통적으로 두 가지 방법이 있었습니다. 하나는 커널 소스를 수정하고 재컴파일하는 방법으로 배포까지 수개월이 걸렸습니다. 다른 하나는 커널 모듈을 작성하는 방법으로 버그가 커널 전체를 패닉시킬 수 있어 위험했습니다.

eBPF(extended Berkeley Packet Filter)는 이 딜레마를 풀어냅니다. 커널 소스 수정 없이, 모듈 특유의 위험 없이, 검증된 프로그램을 커널 이벤트에 붙여 실행합니다. 비유하자면, 커널은 도시 전체의 교통 시스템이고, 커널 모듈은 도로 설계를 바꾸는 공사입니다. eBPF는 신호등에 새 규칙을 동적으로 프로그래밍하는 것에 해당합니다. 공사 없이 교통 흐름을 실시간으로 제어할 수 있습니다.

### 1.2 역사: BPF에서 eBPF로

1992년 Steven McCanne과 Van Jacobson이 발표한 BPF는 tcpdump 같은 도구가 커널에서 패킷 필터링을 효율적으로 하기 위해 탄생했습니다. 2014년 Alexei Starovoitov가 eBPF를 Linux 커널 3.18에 병합했습니다. 레지스터 수가 2개에서 11개로 늘고, 64비트 지원, 맵(Map) 자료구조, 다양한 어태치 포인트가 추가되면서 eBPF는 패킷 필터를 넘어 범용 커널 프로그래밍 환경이 됐습니다.

### 1.3 eBPF의 세 가지 핵심 요소

검증기(Verifier)는 eBPF 프로그램을 커널에 로드하기 전 정적 분석을 수행합니다. 무한 루프, 범위 밖 메모리 접근, 초기화되지 않은 변수 사용을 거부합니다. 검증기를 통과한 프로그램은 커널을 다운시킬 수 없다고 보장됩니다.

JIT 컴파일러(Just-In-Time Compiler)는 검증을 통과한 eBPF 바이트코드를 네이티브 기계어로 컴파일합니다. 인터프리터 방식과 달리 네이티브 코드 실행 속도를 달성하므로 성능 오버헤드가 미미합니다.

맵(Map)은 eBPF 프로그램이 데이터를 저장하고 유저스페이스와 공유하는 자료구조입니다. 해시 맵, 배열, LRU 해시 등 여러 타입이 있으며, 커널과 유저스페이스 양쪽에서 읽고 쓸 수 있습니다. Cilium은 이 맵을 서비스 엔드포인트 테이블, 정책 테이블로 활용합니다.


## 2. eBPF 어태치 포인트

> XDP, TC ingress/egress, cgroup, kprobe/tracepoint 등 eBPF 프로그램이 커널 이벤트에 부착되는 위치와 각 포인트의 특성을 다이어그램으로 설명합니다.

eBPF 프로그램은 커널의 다양한 이벤트에 부착됩니다. 어디에 붙이느냐에 따라 처리할 수 있는 데이터와 행동이 달라집니다.

XDP(eXpress Data Path)는 네트워크 드라이버 레벨에서 패킷을 처리합니다. 커널 네트워크 스택에 진입하기 전이라 가장 빠릅니다. DDoS 차단처럼 단순하고 고속인 처리에 적합합니다.

TC(Traffic Control) ingress/egress는 네트워크 인터페이스의 입출력 트래픽을 제어합니다. XDP보다 더 많은 컨텍스트에 접근할 수 있어 Cilium의 주요 네트워킹 처리가 여기서 이루어집니다.

cgroup hooks는 특정 cgroup(컨테이너)에 속한 소켓 연결을 제어합니다. 파드 레벨의 트래픽 정책 적용에 활용됩니다.

kprobe/tracepoint는 커널 함수 호출 시점에 붙어 시스템 동작을 추적합니다. Hubble이 L7 프로토콜 정보를 수집할 때 활용합니다.


## 3. eBPF로 iptables를 대체하는 원리

> iptables의 O(N) 선형 탐색 한계를 eBPF 해시 맵의 O(1) 조회와 소켓 레벨 단축 경로로 극복하는 원리를 설명합니다.

### 3.1 iptables의 한계

전통적인 Kubernetes에서 kube-proxy는 iptables 규칙으로 서비스 로드 밸런싱을 구현합니다. 서비스가 1개 추가될 때마다 수십 개의 iptables 체인 규칙이 추가됩니다. 서비스 1,000개 환경에서 iptables 규칙은 수만 개가 되며, 규칙 평가는 선형 탐색이어서 패킷 하나가 들어올 때마다 체인의 맨 위부터 순서대로 비교합니다. 규칙 수가 많아질수록 평가 시간도 비례해 늘어납니다.

### 3.2 eBPF 맵의 O(1) 처리

Cilium은 iptables 대신 eBPF 해시 맵을 사용합니다. 서비스 IP와 포트를 키로, 백엔드 엔드포인트 목록을 값으로 저장합니다. 패킷이 들어오면 TC 훅의 eBPF 프로그램이 맵을 O(1) 해시 조회로 백엔드를 찾아 직접 DNAT합니다. 서비스가 1,000개든 10,000개든 조회 시간은 동일합니다.

엔드포인트 업데이트도 빠릅니다. Cilium 에이전트가 맵 엔트리 하나를 갱신하면 즉시 반영되어, 수천 개 iptables 규칙을 재작성하는 방식과 달리 원자적 업데이트가 가능합니다.

### 3.3 소켓 레벨 가속

같은 노드 안의 파드 간 통신에서는 eBPF의 소켓 레벨 단축 경로를 사용합니다. 패킷이 네트워크 스택을 올라갔다 내려오는 과정 없이 소켓 버퍼에서 직접 다른 소켓 버퍼로 전달되어 노드 내 파드 간 통신 레이턴시를 크게 줄입니다.

### 4.1 Cilium Agent (DaemonSet)

노드마다 하나씩 실행되는 Cilium Agent는 Cilium의 두뇌입니다. Kubernetes API 서버를 감시해 파드, 서비스, 엔드포인트, NetworkPolicy 변경을 감지합니다. 변경이 감지되면 대응하는 eBPF 프로그램을 컴파일하고 커널에 로드하며 eBPF 맵을 업데이트합니다. 또한 각 파드의 네트워크 네임스페이스에 veth 인터페이스를 설정하고, 해당 인터페이스의 TC 훅에 eBPF 프로그램을 부착합니다. 이 과정이 CNI 플러그인으로서의 역할입니다.

### 4.2 Cilium Operator

클러스터 전체 수준의 작업을 담당합니다. 노드 IP 주소 관리(IPAM), CiliumNetworkPolicy의 클러스터 전파, Kubernetes 오브젝트와 Cilium 내부 상태 동기화를 처리합니다. 클러스터당 하나만 실행되며 DaemonSet이 아닌 Deployment로 배포됩니다.

### 4.3 Hubble: 관측성 계층

Hubble은 Cilium의 관측성 컴포넌트입니다. eBPF 훅에서 수집한 네트워크 플로우 데이터를 처리하며, 사이드카 없이도 L3/L4/L7 트래픽 정보를 수집할 수 있다는 점이 핵심 강점입니다.

Hubble은 세 컴포넌트로 구성됩니다. 각 노드의 Hubble Observer는 로컬 eBPF 이벤트를 수집합니다. Hubble Relay는 모든 노드의 Observer 데이터를 집계합니다. Hubble UI는 서비스 의존성 그래프와 플로우 필터링 인터페이스를 제공합니다.


## 5. Cilium Service Mesh

> L4는 eBPF 커널 처리, L7은 노드 공유 Envoy로 분리하는 Cilium Service Mesh의 두 계층 설계와 CiliumNetworkPolicy 예시를 다룹니다.

### 5.1 두 계층의 분리

Cilium의 서비스 메시는 L4와 L7을 명확히 분리해 처리합니다. 이 설계가 Cilium의 성능 특성을 결정짓는 핵심입니다.

L4 계층(eBPF)에서는 TCP/UDP 연결, 포트 기반 정책, 로드 밸런싱, WireGuard/IPsec 기반 암호화를 처리합니다. 유저스페이스 프록시를 거치지 않으므로 오버헤드가 거의 없습니다.

L7 계층(노드 공유 Envoy)에서는 HTTP 헤더 기반 라우팅, gRPC 트랜스코딩, JWT 검증 같은 L7 기능을 처리합니다. Cilium은 노드에 하나의 Envoy 인스턴스를 실행해 L7 처리를 위임합니다. 사이드카와의 차이는 이 Envoy가 파드별이 아닌 노드 공유라는 점입니다.

### 5.2 CiliumNetworkPolicy

Cilium의 정책 CRD는 Kubernetes 기본 NetworkPolicy보다 표현력이 높습니다. L3(IP/CIDR), L4(포트/프로토콜), L7(HTTP 경로, 메서드, 헤더) 규칙을 하나의 CRD에 선언합니다.

```yaml
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: allow-payment-api
spec:
  endpointSelector:
    matchLabels:
      app: payment
  ingress:
    - fromEndpoints:
        - matchLabels:
            app: checkout
      toPorts:
        - ports:
            - port: "8080"
              protocol: TCP
          rules:
            http:
              - method: POST
                path: /api/v1/pay
```

이 정책은 `checkout` 파드만 `payment` 파드의 `/api/v1/pay` POST 엔드포인트를 호출할 수 있도록 제한합니다. L7 규칙이 포함되어 있으므로 해당 파드 쌍의 트래픽은 노드 Envoy를 경유합니다. L7 규칙이 없다면 순수 eBPF로 처리됩니다.


## 6. Hubble 관측성

> 사이드카 없이 eBPF 훅으로 L3~L7 트래픽 정보를 수집하는 Hubble의 동작 원리와 주요 CLI 명령을 설명합니다.

Hubble은 eBPF의 관측성 능력을 사용자 친화적 인터페이스로 제공합니다. 사이드카 없이도 다음 정보를 수집합니다. L3에서는 소스/목적지 IP와 프로토콜, L4에서는 포트와 TCP 상태, L7에서는 HTTP 메서드·URL·상태 코드·gRPC 서비스/메서드·DNS 쿼리를 수집합니다.

```bash
# 특정 파드의 트래픽 관측
hubble observe --pod payment --follow

# HTTP 5xx 에러만 필터링
hubble observe --http-status 5xx --last 100

# 서비스 간 흐름 요약
hubble observe --from-service checkout --to-service payment

# Hubble UI 포트 포워딩

```

Cilium과 Hubble은 Prometheus 메트릭을 기본 제공합니다. `hubble_flows_processed_total`, `cilium_drop_count_total`, `cilium_endpoint_state` 등의 메트릭으로 클러스터 네트워크 상태를 실시간으로 파악할 수 있습니다.


## 7. Cilium vs 전통적 서비스 메시 비교

> 메모리 절약·운영 단순화 등 Cilium의 장점과 커널 버전 제약·L7 기능 성숙도·CNI 종속 등 한계를 균형 있게 정리합니다.

### 7.1 장점

사이드카가 없어 파드 시작 속도가 빠르고 메모리 오버헤드가 낮습니다. 1,000개 파드 클러스터에서 사이드카 대비 수십 GB 메모리를 절약합니다. CNI와 메시를 통합해 운영 포인트가 줄어듭니다. iptables 없는 O(1) 조회로 서비스 수에 무관한 일정한 성능을 제공하며, 사이드카 없이도 L7까지 관측하므로 레거시 파드도 Hubble로 트래픽을 추적할 수 있습니다.

### 7.2 한계

Linux 커널 5.10 이상이 필수여서 커널 버전을 제어할 수 없는 일부 환경에서는 도입이 어렵습니다. Istio에 비해 L7 트래픽 관리 기능이 덜 성숙합니다. 서비스 메시로 Cilium을 선택하면 CNI도 Cilium이어야 하므로 기존 CNI 마이그레이션 비용이 발생합니다. eBPF 레벨 문제는 `bpftool`로 디버깅해야 해서 전문 지식이 필요합니다.


## 8. Cilium의 시장 위치

> CNI 채택률 50% 돌파가 서비스 메시 시장에 미치는 영향과 CNI·메시 경계 해체 흐름을 분석합니다.

CNCF가 2025년 발표한 연간 설문조사에 따르면 Cilium CNI의 채택률이 50%를 넘어섰습니다. 이 채택률은 서비스 메시 시장에도 직접적인 영향을 줍니다. 이미 Cilium CNI를 사용 중인 조직은 별도 서비스 메시 없이 Cilium의 L7 기능을 활성화하는 경로가 자연스럽기 때문입니다. 이 흐름은 CNI와 서비스 메시의 경계를 허무는 방향을 가리킵니다. Cilium이 성공적으로 자리잡으면 서비스 메시는 별도 설치하는 소프트웨어가 아니라 Kubernetes 네트워크 플러그인에 내장되는 기능으로 변화할 수 있습니다.

Cilium의 eBPF·Hubble을 그대로 쓰면서 Istio Ambient의 ztunnel·waypoint를 위에 얹어 책임을 분리하는 통합 패턴은 [Cilium과 Istio Ambient 통합 전략](./05-01.Cilium%EA%B3%BC%20Istio%20Ambient%20%ED%86%B5%ED%95%A9%20%EC%A0%84%EB%9E%B5.md)에서 따로 다룹니다. Istio의 Ambient Mesh는 2024년 11월 1.24에서 GA로 승격되었으며, ztunnel이 노드 L4 mTLS를, waypoint 프록시가 L7을 맡습니다.


## 면접 대비

> eBPF·Cilium 챕터의 핵심 질문을 면접 답변 형식으로 정리합니다.

**eBPF 검증기(Verifier)가 보장하는 것과 보장하지 않는 것은?**

검증기는 무한 루프 없음, 범위 밖 메모리 접근 없음, 초기화되지 않은 포인터 역참조 없음을 정적 분석으로 보장합니다. 덕분에 eBPF 프로그램이 커널을 패닉시키거나 무한 대기 상태로 만들 수 없습니다. 그러나 프로그램의 논리적 정확성은 보장하지 않습니다. 잘못된 패킷 드롭 결정이나 틀린 정책 적용은 감지하지 못합니다.

**Cilium이 iptables보다 고성능인 이유를 데이터 구조 관점에서 설명하면?**

iptables는 체인 형태의 선형 자료구조로 규칙 N개 환경에서 O(N) 시간 복잡도입니다. Cilium의 eBPF 해시 맵은 목적지 IP와 포트를 해시 키로 백엔드를 즉시 조회해 O(1)에 가까운 속도를 냅니다. 서비스 수가 수천 개인 클러스터에서 이 차이는 수 밀리초 규모의 레이턴시 차이로 나타납니다.

**Cilium의 L4 처리와 L7 처리를 어떻게 분리하며, 언제 Envoy가 활성화되는가?**

기본적으로 모든 트래픽은 eBPF TC 훅에서 L4 레벨로 처리됩니다. CiliumNetworkPolicy에 L7 규칙(HTTP 경로, 메서드, gRPC 서비스)이 포함되거나, HTTPRoute 같은 Gateway API 규칙이 적용된 파드 쌍의 트래픽만 노드 공유 Envoy로 리디렉션됩니다. Envoy는 파드별 사이드카가 아니라 노드당 하나만 실행되므로 L7 규칙을 사용하는 파드가 50개 있어도 Envoy 프로세스는 하나입니다.

**Hubble이 사이드카 없이 L7 정보를 수집할 수 있는 원리는?**

트래픽이 Cilium의 노드 Envoy를 경유할 때 Envoy가 파싱한 L7 정보를 Hubble Observer가 수신하는 방식입니다. L7 정책이 없어 Envoy를 거치지 않는 트래픽은 L4 정보(IP, 포트, 연결 상태)만 수집됩니다. 결과적으로 L7 정보는 L7 정책이 활성화된 서비스에서만 완전하게 수집됩니다.

**Cilium CNI 도입 시 기존 Calico에서 마이그레이션 절차의 핵심 위험은?**

CNI 교체는 노드의 모든 파드 네트워크 재구성을 의미합니다. 가장 안전한 방법은 새 노드 그룹을 Cilium으로 프로비저닝하고 기존 워크로드를 이동한 뒤 기존 노드를 제거하는 롤링 마이그레이션입니다. 두 CNI가 동시에 활성화된 혼재 상태에서 파드 간 통신이 끊기는 것이 가장 큰 위험이므로, 프로덕션 마이그레이션 전 동일 스펙의 스테이징 클러스터에서 완전한 검증이 필수입니다.
