# 비교 매트릭스 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.


## Q&A

**벤치마크 수치는 얼마나 신뢰할 수 있는가?**

합성 벤치마크(고정 RPS)는 버스티 트래픽 패턴을 반영하지 못한다. Coordinated Omission 문제를 처리하는 `fortio` 같은 도구를 사용하고, 누가 측정했는지(벤더 vs 중립 제3자), 실제 워크로드와 유사한 패턴인지, 측정 환경의 노이즈를 통제했는지 확인해야 한다. 직접 측정이 최선이며, 현재 서비스의 프로덕션 트래픽 패턴을 스테이징에서 재현한 후 각 메시를 비교하는 것이 합성 벤치마크보다 신뢰할 수 있다.

**Cilium CNI를 선택하면 어느 정도의 Lock-In이 발생하는가?**

Lock-In은 두 레이어에서 발생한다. 첫째, `CiliumNetworkPolicy` 같은 Cilium 전용 CRD를 사용하면 다른 CNI로 교체 시 이 CRD가 모두 무효화된다. 둘째, 서비스 메시 레이어에서도 Cilium 전용 설정이 쌓인다. Lock-In을 최소화하려면 표준 `NetworkPolicy`와 Gateway API를 최대한 사용하고, Cilium 전용 기능은 차별화가 명확한 경우에만 사용한다. CNI는 클러스터 생애주기에서 한 번 선택하면 잘 바꾸지 않으므로, 장기적 네트워킹 요구사항을 충분히 검토한 후 결정해야 한다.

**eBPF 기반 메시가 요구하는 커널 버전이 실제로 진입 장벽이 되는가?**

클라우드 환경(EKS, GKE, AKS)에서는 최신 커널을 제공하므로 문제가 적다. 하지만 RHEL 7(커널 3.10 기반) 같은 온프레미스나 하이브리드 환경에서는 실질적 장벽이 된다. 엔터프라이즈 커널 업그레이드는 수개월의 인증 및 검증 과정을 거치고, 보안 규정상 변경 관리 위원회 승인이 필요한 조직도 있다. Cilium 설치 전 `cilium-preflight-check`를 실행해 환경 적합성을 반드시 확인해야 한다.
