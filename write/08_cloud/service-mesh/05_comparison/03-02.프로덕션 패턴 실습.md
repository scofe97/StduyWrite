## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| CNI | Flannel (10.244.0.0/16) | — |

접속은 `gcp` 스킬의 SSH 절차를 따른다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 1. 카나리 배포 메트릭 설계 실습

### 1.1 현재 서비스 베이스라인 측정

카나리 분석에 앞서 현재 서비스의 성능 베이스라인을 수집한다.

```bash
# 테스트 환경 구성
kubectl create namespace ch25-prod

# v1 배포
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-v1
spec:
  replicas: 3
  selector:
    matchLabels:
      app: payment
      version: v1
  template:
    metadata:
      labels:
        app: payment
        version: v1
    spec:
      containers:
      - name: payment
        image: nginx:alpine
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: payment
spec:
  selector:
    app: payment
  ports:
  - port: 80
EOF

# 부하 생성 및 베이스라인 측정
kubectl run load-gen -n ch25-prod --rm -it \
  --image=fortio/fortio:latest -- \
  load -qps 100 -t 60s -c 4 \
  -labels "baseline=true" \
  http://payment/
```

측정 결과에서 p50, p99, 성공률을 기록한다. 이 값이 카나리 분석의 임계값 기준이 된다.

### 1.2 카나리 분석 임계값 설정 원칙

```bash
# Prometheus에서 현재 성공률 쿼리 (Istio 메트릭 기준)
# istio_requests_total{destination_service="payment.ch25-prod.svc.cluster.local", response_code!~"5.*"}
# / istio_requests_total{destination_service="payment.ch25-prod.svc.cluster.local"}
---

# p99 레이턴시 쿼리
# histogram_quantile(0.99, sum(rate(istio_request_duration_milliseconds_bucket{destination_service="payment.ch25-prod.svc.cluster.local"}[1m])) by (le))
```

임계값 설정 시 주의할 점은 새벽 시간대처럼 트래픽이 적은 경우 에러 샘플이 부족해 통계적으로 의미 없는 결과가 나올 수 있다는 것이다. 절대 에러 수 임계치와 에러율을 AND 조건으로 사용하는 것이 낫다.


## 2. Istio 카나리 라우팅 실습

### 2.1 DestinationRule과 VirtualService 구성

```bash
# v2 배포 (카나리 버전 - 의도적으로 약간 느리게 시뮬레이션)
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-v2
spec:
  replicas: 1
  selector:
    matchLabels:
      app: payment
      version: v2
  template:
    metadata:
      labels:
        app: payment
        version: v2
    spec:
      containers:
      - name: payment
        image: nginx:alpine
        ports:
        - containerPort: 80
EOF

# DestinationRule - subset 정의
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment
spec:
  host: payment
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2
EOF

# VirtualService - 5% 카나리 트래픽
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: payment
spec:
  hosts:
    - payment
  http:
    - route:
        - destination:
            host: payment
            subset: v1
          weight: 95
        - destination:
            host: payment
            subset: v2
          weight: 5
EOF
```

### 2.2 트래픽 비율 단계적 증가

```bash
# 트래픽 발생 유지 (별도 터미널에서)
kubectl run traffic-gen -n ch25-prod --rm -it \
  --image=fortio/fortio:latest -- \
  load -qps 50 -t 300s http://payment/

# 10% → 25% → 50% → 100% 단계별 전환
for weight in 10 25 50 100; do
  stable=$((100 - weight))
  echo "카나리 트래픽 ${weight}%로 전환..."
  kubectl patch virtualservice payment -n ch25-prod --type=json \
    -p="[{\"op\":\"replace\",\"path\":\"/spec/http/0/route/0/weight\",\"value\":${stable}},{\"op\":\"replace\",\"path\":\"/spec/http/0/route/1/weight\",\"value\":${weight}}]"
  sleep 60
  # 각 단계에서 에러율 확인
  kubectl exec -n istio-system deploy/prometheus -- \
    wget -qO- 'localhost:9090/api/v1/query?query=sum(rate(istio_requests_total{destination_service="payment.ch25-prod.svc.cluster.local",response_code=~"5.*"}[1m]))/sum(rate(istio_requests_total{destination_service="payment.ch25-prod.svc.cluster.local"}[1m]))' \
    | jq '.data.result[0].value[1]'
done
```


## 3. 서킷 브레이커 실습

### 3.1 outlierDetection 설정

```bash
# 서킷 브레이커 설정 (payment 서비스 대상)
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-cb
spec:
  host: payment
  trafficPolicy:
    outlierDetection:
      consecutiveGatewayErrors: 3
      interval: 10s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
    connectionPool:
      tcp:
        maxConnections: 50
      http:
        http1MaxPendingRequests: 25
        http2MaxRequests: 50
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2
EOF
```

### 3.2 서킷 브레이커 동작 확인

```bash
# 과부하 상황 시뮬레이션 (연결 수 초과)
kubectl run overload-test -n ch25-prod --rm -it \
  --image=fortio/fortio:latest -- \
  load -qps 1000 -c 100 -t 30s http://payment/

# 서킷 브레이커 통계 확인
kubectl exec -n ch25-prod deploy/payment-v1 -c istio-proxy -- \
  pilot-agent request GET stats | grep -E 'ejection|outlier'

# Prometheus에서 ejected 엔드포인트 확인
# pilot_total_xds_rejects{} 또는 envoy_cluster_outlier_detection_ejections_active{}
```

### 3.3 벌크헤드 한도 초과 확인

```bash
# 대기 요청 한도(25개) 초과 시 즉시 거부되는지 확인
kubectl run bulkhead-test -n ch25-prod --rm -it \
  --image=fortio/fortio:latest -- \
  load -qps 200 -c 50 -t 10s http://payment/

# 503 응답이 발생하는지 확인 (overflow로 즉시 거부)
# 결과에서 503 count 확인
```


## 4. 폴트 인젝션 카오스 실습

### 4.1 지연 인젝션

```bash
# 10% 요청에 3초 지연 주입
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: payment-chaos-delay
spec:
  hosts:
    - payment
  http:
    - fault:
        delay:
          percentage:
            value: 10
          fixedDelay: 3s
      route:
        - destination:
            host: payment
            subset: v1
EOF

# 타임아웃 검증 (2초 타임아웃 설정 시 지연이 감지되어야 함)
kubectl run timeout-test -n ch25-prod --rm -it \
  --image=curlimages/curl:latest -- \
  sh -c 'for i in $(seq 1 20); do curl -s -o /dev/null -w "%{http_code} %{time_total}\n" --max-time 2 http://payment/; done'
```

### 4.2 오류 인젝션

```bash
# 5% 요청에 503 오류 주입
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: payment-chaos-abort
spec:
  hosts:
    - payment
  http:
    - fault:
        abort:
          percentage:
            value: 5
          httpStatus: 503
      route:
        - destination:
            host: payment
            subset: v1
EOF

# 재시도 정책이 있는 경우 클라이언트가 자동 복구하는지 확인
kubectl run retry-test -n ch25-prod --rm -it \
  --image=curlimages/curl:latest -- \
  sh -c 'for i in $(seq 1 50); do curl -s -o /dev/null -w "%{http_code}\n" http://payment/; done | sort | uniq -c'
```

### 4.3 헤더 기반 폴트 인젝션 (프로덕션 안전 패턴)

```bash
# 특정 헤더가 있는 요청에만 폴트 주입 (내부 테스트 사용자만 영향)
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: payment-chaos-safe
spec:
  hosts:
    - payment
  http:
    - match:
        - headers:
            x-chaos-test:
              exact: "true"
      fault:
        delay:
          percentage:
            value: 100
          fixedDelay: 5s
      route:
        - destination:
            host: payment
            subset: v1
    - route:
        - destination:
            host: payment
            subset: v1
EOF

# 일반 요청은 정상 (헤더 없음)
kubectl run normal-test -n ch25-prod --rm -it \
  --image=curlimages/curl:latest -- \
  curl -s -w "%{time_total}\n" http://payment/

# 카오스 테스트 요청은 지연 (헤더 있음)
kubectl run chaos-test -n ch25-prod --rm -it \
  --image=curlimages/curl:latest -- \
  curl -s -H "x-chaos-test: true" -w "%{time_total}\n" http://payment/
```


## 5. 타임아웃 계층 설계 실습

### 5.1 서비스 체인 구성

```bash
# A → B → C 서비스 체인 구성
for svc in svc-a svc-b svc-c; do
  kubectl apply -n ch25-prod -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ${svc}
spec:
  replicas: 1
  selector:
    matchLabels:
      app: ${svc}
  template:
    metadata:
      labels:
        app: ${svc}
    spec:
      containers:
      - name: app
        image: nginx:alpine
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: ${svc}
spec:
  selector:
    app: ${svc}
  ports:
  - port: 80
EOF
done
```

### 5.2 타임아웃 계층 적용

```bash
# C 서비스: 타임아웃 2초
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: svc-c
spec:
  hosts:
    - svc-c
  http:
    - timeout: 2s
      route:
        - destination:
            host: svc-c
EOF

# B 서비스: 타임아웃 4초 (C 타임아웃 2s + 여유 2s)
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: svc-b
spec:
  hosts:
    - svc-b
  http:
    - timeout: 4s
      retries:
        attempts: 2
        perTryTimeout: 1s
        retryOn: gateway-error,reset,connect-failure
      route:
        - destination:
            host: svc-b
EOF

# A 서비스: 타임아웃 8초 (B 타임아웃 4s + 재시도 1회 + 여유)
kubectl apply -n ch25-prod -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: svc-a
spec:
  hosts:
    - svc-a
  http:
    - timeout: 8s
      route:
        - destination:
            host: svc-a
EOF

# istioctl analyze로 설정 검증
istioctl analyze -n ch25-prod
```
