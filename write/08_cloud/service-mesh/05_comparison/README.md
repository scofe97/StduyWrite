# Service Mesh 비교·도입 학습 MOC

---

> Linkerd·Istio·Cilium 세 메시를 같은 축으로 비교하고, eBPF라는 다른 패러다임을 이해한 뒤, 프로덕션 패턴과 조직 단위 도입 판단으로 마무리하는 문서를 한 폴더에 모았습니다. 본 묶음을 끝내면 "우리 팀·인프라·요구에 맞는 메시는 무엇이고 왜 그런가" 를 근거를 갖고 답할 수 있습니다.

## 왜 한 폴더로 묶었는가

앞 폴더들이 각 구현체를 따로 깊이 파고들었다면, 이 폴더는 그것들을 한자리에 놓고 **결정을 돕는** 문서들입니다. 비교 매트릭스(01)가 세 메시를 같은 축으로 정렬하고, eBPF·Cilium(02)이 사이드카 없는 커널 계층 대안을 보여주며, 프로덕션 패턴(03)이 현업에서 검증된 구성을 정리하고, 도입 전략(04)이 조직·규모·역량을 기준으로 선택 프레임을 제시합니다. 마지막 통합 전략(05)은 Cilium과 Istio Ambient를 함께 쓸 때 책임을 어디서 가를지 다룹니다.

이 폴더가 비교의 SSOT입니다. 구현체 폴더(02_linkerd·03_istio)의 아키텍처 문서에 나오는 "vs" 비교는 그 문서 관점의 요약이고, 세 메시를 같은 축으로 정렬한 전체 비교는 여기 01-01로 모읍니다.

5. Cilium과 Istio Ambient를 함께 쓸 때 L3/L4는 누가, L7은 누가 책임지는 것이 자연스럽습니까?
