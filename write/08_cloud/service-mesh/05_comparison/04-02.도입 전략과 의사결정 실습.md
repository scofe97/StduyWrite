# 도입 전략과 의사결정 실습

> 본 장의 개념을 손으로 확인합니다. 현재 실습 환경은 개인 GCP K8s 클러스터이며, 환경이 바뀌면 `## 실습 환경` 섹션만 갈아 끼우면 됩니다.


## 실습 환경` 섹션만 갈아 끼우면 됩니다.


## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터입니다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| CNI | Flannel (10.244.0.0/16) | — |

접속은 `gcp` 스킬의 SSH 절차를 따릅니다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md)를 참조합니다.


## 1. 도입 준비도 평가 실습

### 1.1 클러스터 환경 프로파일링

```bash
# 현재 서비스 수 파악 (메시 필요성 판단 기준)
kubectl get svc --all-namespaces --no-headers | wc -l

# 파드 수 파악 (사이드카 오버헤드 계산 기준)
kubectl get pods --all-namespaces --field-selector=status.phase=Running --no-headers | wc -l

# 노드 리소스 현황
kubectl top nodes

# 파드별 리소스 사용량 (사이드카 추가 공간 계산)
kubectl top pods --all-namespaces --sort-by=memory | head -20
```

### 1.2 사이드카 오버헤드 사전 계산

```bash
# 파드 수 기반 사이드카 메모리 예상치
PODS=$(kubectl get pods --all-namespaces --field-selector=status.phase=Running --no-headers | wc -l)
echo "현재 실행 중인 파드: ${PODS}개"
echo "Linkerd 추가 메모리 예상: $((PODS * 10))MB (10MB/pod)"
echo "Istio 추가 메모리 예상: $((PODS * 50))MB (50MB/pod)"
echo "Istio Ambient 추가 메모리 예상: 노드 수 × 10MB"

# 노드 수 확인
kubectl get nodes --no-headers | wc -l
```

### 1.3 도입 준비도 자가 진단

다음 질문에 모두 "예"일 때 도입을 진행합니다.

```
□ 서비스 수가 10개 이상인가?
□ 팀이 Kubernetes를 6개월 이상 운영했는가?
□ 메시로 해결하려는 구체적 문제가 있는가?
□ 그 문제를 메시 없이 해결하는 방법을 먼저 검토했는가?
□ 메시 전담 운영 인원(최소 1명)을 확보할 수 있는가?
```

```bash
# Kubernetes 운영 경험 확인 (클러스터 생성 시점)
kubectl get namespace kube-system -o jsonpath='{.metadata.creationTimestamp}'
```


## 2. PERMISSIVE → STRICT mTLS 전환 실습

### 2.1 PERMISSIVE 모드로 시작

```bash
# 테스트 네임스페이스 생성
kubectl create namespace ch26-migrate

# Istio 사이드카 주입 활성화
kubectl label namespace ch26-migrate istio-injection=enabled

# 테스트 서비스 배포
kubectl apply -n ch26-migrate -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: svc-a
spec:
  replicas: 1
  selector:
    matchLabels:
      app: svc-a
  template:
    metadata:
      labels:
        app: svc-a
    spec:
      containers:
      - name: app
        image: nginx:alpine
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: svc-a
spec:
  selector:
    app: svc-a
  ports:
  - port: 80
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: svc-b
spec:
  replicas: 1
  selector:
    matchLabels:
      app: svc-b
  template:
    metadata:
      labels:
        app: svc-b
    spec:
      containers:
      - name: app
        image: nginx:alpine
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: svc-b
spec:
  selector:
    app: svc-b
  ports:
  - port: 80
EOF

# PERMISSIVE 모드 설정 (기본값이지만 명시적으로 확인)
kubectl apply -n ch26-migrate -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: ch26-migrate
spec:
  mtls:
    mode: PERMISSIVE
EOF
```

### 2.2 mTLS 적용 상태 모니터링

```bash
# 파드 재시작으로 사이드카 주입
kubectl rollout restart deployment -n ch26-migrate

# 사이드카 주입 확인
kubectl get pods -n ch26-migrate -o jsonpath='{range .items[*]}{.metadata.name}: {range .spec.containers[*]}{.name} {end}{"\n"}{end}'
# istio-proxy 컨테이너가 있어야 함
---

# mTLS 통신 상태 확인 (Kiali가 있는 경우)
# kiali를 통해 서비스 그래프에서 자물쇠 아이콘 확인
---

# Prometheus 쿼리로 mTLS 비율 확인
kubectl exec -n istio-system deploy/prometheus -- \
  wget -qO- 'localhost:9090/api/v1/query?query=sum(istio_requests_total{connection_security_policy="mutual_tls",destination_namespace="ch26-migrate"})%2Fsum(istio_requests_total{destination_namespace="ch26-migrate"})' \
  | jq '.data.result[0].value[1] // "메트릭 없음 (트래픽 없는 경우)"'

# 테스트 트래픽 생성
kubectl run traffic -n ch26-migrate --rm -it \
  --image=curlimages/curl:latest -- \
  sh -c 'for i in $(seq 1 20); do curl -s http://svc-b/; sleep 1; done'
```

### 2.3 STRICT 모드로 안전하게 전환

```bash
# PERMISSIVE 상태에서 비-mTLS 트래픽이 없는지 확인
kubectl exec -n istio-system deploy/prometheus -- \
  wget -qO- 'localhost:9090/api/v1/query?query=sum(istio_requests_total{connection_security_policy!="mutual_tls",destination_namespace="ch26-migrate"})' \
  | jq '.data.result[0].value[1] // "0"'
# 결과가 0 또는 메트릭 없음이어야 안전하게 전환 가능
---

# STRICT 모드로 전환
kubectl apply -n ch26-migrate -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: ch26-migrate
spec:
  mtls:
    mode: STRICT
EOF

# mTLS 없는 클라이언트에서 접근 시도 (차단되어야 함)
kubectl run no-mesh-client -n default --rm -it \
  --image=curlimages/curl:latest -- \
  curl -s --connect-timeout 3 http://svc-b.ch26-migrate.svc.cluster.local/
# Connection refused 또는 타임아웃 발생해야 정상
```


## 3. 단계적 마이그레이션 시뮬레이션

### 3.1 Phase 1: 비중요 서비스부터 시작

```bash
# 비중요 서비스 네임스페이스에만 주입 (프로덕션은 아직 제외)
kubectl label namespace ch26-migrate istio-injection=enabled

# 주입 상태 확인
kubectl get namespace ch26-migrate --show-labels

# 파드 재시작 후 메트릭 수집 시작
kubectl rollout restart deployment -n ch26-migrate

# 기본 메트릭 확인 (Prometheus가 있는 경우)
kubectl exec -n istio-system deploy/prometheus -- \
  wget -qO- 'localhost:9090/api/v1/query?query=up{job="kubernetes-pods",namespace="ch26-migrate"}' \
  | jq '.data.result | length'
```

### 3.2 Phase 2: 관찰가능성 베이스라인 수집

```bash
# 서비스 간 트래픽 발생 (2~3분간 유지)
kubectl run baseline-traffic -n ch26-migrate --rm -it \
  --image=fortio/fortio:latest -- \
  load -qps 10 -t 180s http://svc-b/

# Prometheus에서 베이스라인 지표 수집
kubectl exec -n istio-system deploy/prometheus -- \
  wget -qO- 'localhost:9090/api/v1/query?query=histogram_quantile(0.99,sum(rate(istio_request_duration_milliseconds_bucket{destination_service_name="svc-b",destination_namespace="ch26-migrate"}[1m]))by(le))' \
  | jq '.data.result[0].value[1] // "데이터 없음"'
```

### 3.3 Phase 3: 타임아웃 정책 적용

```bash
# 베이스라인 기반 타임아웃 설정
kubectl apply -n ch26-migrate -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: svc-b
spec:
  hosts:
    - svc-b
  http:
    - timeout: 3s
      retries:
        attempts: 2
        perTryTimeout: 1s
        retryOn: gateway-error,reset,connect-failure
      route:
        - destination:
            host: svc-b
EOF

# 설정 오류 없는지 확인
istioctl analyze -n ch26-migrate
```

## 5. AuthorizationPolicy 적용 실습

### 5.1 최소 권한 정책 설정

```bash
# 먼저 모든 접근 허용 정책으로 시작 (PERMISSIVE와 별개)
kubectl apply -n ch26-migrate -f - <<EOF
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: allow-all
  namespace: ch26-migrate
spec:
  action: ALLOW
  rules:
    - {}
EOF

# 트래픽 패턴 관찰 후 최소 권한으로 좁히기
kubectl apply -n ch26-migrate -f - <<EOF
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: svc-b-policy
  namespace: ch26-migrate
spec:
  selector:
    matchLabels:
      app: svc-b
  action: ALLOW
  rules:
    - from:
        - source:
            principals: ["cluster.local/ns/ch26-migrate/sa/svc-a"]
      to:
        - operation:
            methods: ["GET"]
            paths: ["/"]
EOF
```

### 5.2 정책 검증

```bash
# svc-a에서 svc-b 접근 (허용)
kubectl exec -n ch26-migrate deploy/svc-a -c app -- \
  wget -qO- http://svc-b/ 2>&1 | head -5

# 다른 서비스에서 svc-b 접근 (차단)
kubectl run unauthorized -n ch26-migrate --rm -it \
  --image=curlimages/curl:latest -- \
  curl -s --connect-timeout 3 http://svc-b/
# RBAC 403 또는 연결 거부가 발생해야 함
---

# istioctl로 정책 분석
istioctl x authz check -n ch26-migrate deploy/svc-b
```
