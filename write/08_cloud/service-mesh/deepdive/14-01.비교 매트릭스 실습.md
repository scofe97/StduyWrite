## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| CNI | Flannel (10.244.0.0/16) | — |

접속은 `gcp` 스킬의 SSH 절차를 따른다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 1. 벤치마크 측정 방법론 실습

### 1.1 기준선 측정

메시 비교 실습에 앞서 메시 없는 상태의 파드 간 레이턴시를 측정해 기준선을 확보한다.

```bash
# 테스트용 네임스페이스 생성
kubectl create namespace ch23-bench

# fortio 서버 배포
kubectl apply -n ch23-bench -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: fortio-server
spec:
  replicas: 1
  selector:
    matchLabels:
      app: fortio-server
  template:
    metadata:
      labels:
        app: fortio-server
    spec:
      containers:
      - name: fortio
        image: fortio/fortio:latest
        ports:
        - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: fortio-server
spec:
  selector:
    app: fortio-server
  ports:
  - port: 8080
    targetPort: 8080
EOF

# 클라이언트에서 100 RPS, 30초 부하 테스트
kubectl run fortio-client -n ch23-bench --rm -it \
  --image=fortio/fortio:latest -- \
  load -qps 100 -t 30s -c 4 \
  http://fortio-server:8080/echo
```

측정 결과에서 p50, p99, p999 값을 기록해둔다. 이 값이 메시 도입 후 비교 기준이 된다.

### 1.2 Coordinated Omission 문제 이해

`wrk`와 달리 `fortio`는 `-qps` 플래그로 고정 RPS를 유지하므로 Coordinated Omission 문제가 적다. 두 도구의 결과를 같은 기준으로 비교하면 안 된다.

```bash
# wrk 방식 (Coordinated Omission 있음)
# wrk -t4 -c100 -d30s http://fortio-server:8080/echo
---

# fortio 방식 (Coordinated Omission 없음, 권장)
# fortio load -qps 1000 -t 30s http://fortio-server:8080/echo
```


## 2. 세 메시 설치 상태 비교 실습

### 2.1 Linkerd 설치 사전 점검

```bash
# Linkerd CLI 설치 확인
linkerd version

# 클러스터 사전 점검
linkerd check --pre

# 설치 가능 여부 확인
linkerd install --dry-run | head -50
```

`linkerd check --pre` 결과에서 모든 항목이 체크 표시가 되는지 확인한다. `[FAIL]` 항목이 있으면 해당 내용을 먼저 해결해야 한다.

### 2.2 Istio 설치 사전 점검

```bash
# istioctl 설치 확인
istioctl version

# 클러스터 호환성 확인
istioctl x precheck

# 프로파일 목록 확인
istioctl profile list

# 최소 프로파일 dry-run
istioctl install --set profile=minimal --dry-run
```

### 2.3 커널 버전 확인 (Cilium 요구사항)

```bash
# 모든 노드의 커널 버전 조회
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}: {.status.nodeInfo.kernelVersion}{"\n"}{end}'

# 기대 출력 예시:
# dev-server: 5.15.0-1040-gcp
# dev-server-2: 5.15.0-1040-gcp
# dev-server-3: 5.15.0-1040-gcp
```

커널 버전이 5.10 미만이면 Cilium Service Mesh의 고급 기능(WireGuard 암호화 등)을 사용할 수 없다.


## 3. 라이선스 비교 실습

### 3.1 현재 사용 중인 메시 버전 확인

라이선스 비용 계산 전 실제 사용 버전을 확인하는 것이 중요하다.

```bash
# Linkerd 버전 확인 (BEL 적용 범위는 2.13 이후 안정 릴리스)
linkerd version --client

# Istio 버전 확인
istioctl version

# Cilium 버전 확인 (설치된 경우)
cilium version
```

Linkerd 2.13 이후 버전의 공식 안정 바이너리를 직원 50인 이상 기업이 상업적으로 사용하는 경우에만 BEL 비용이 발생한다. 개인 학습 환경에서는 무관하다.


## 4. 메시 선택 의사결정 실습

### 4.1 현재 환경 프로파일링

```bash
# 현재 클러스터의 서비스 수 파악
kubectl get svc --all-namespaces | wc -l

# 현재 파드 수 파악
kubectl get pods --all-namespaces | wc -l

# 사용 중인 CNI 확인
kubectl get pods -n kube-system -o wide | grep -E 'flannel|calico|cilium|weave'

# 노드 리소스 현황 파악
kubectl top nodes
```

이 정보를 바탕으로 23-01 본문의 결정 트리를 따라 어떤 메시가 적합한지 판단한다.

### 4.2 메모리 오버헤드 사전 계산

```bash
# 현재 파드 수 기준 사이드카 메모리 예상치 계산
PODS=$(kubectl get pods --all-namespaces --field-selector=status.phase=Running | wc -l)
echo "현재 실행 중인 파드 수: $PODS"
echo "Linkerd 사이드카 추가 메모리: $(($PODS * 10))MB"
echo "Istio 사이드카 추가 메모리: $(($PODS * 50))MB"

# 노드 가용 메모리 확인
kubectl get nodes -o custom-columns='NAME:.metadata.name,MEMORY:.status.allocatable.memory'
```


## 5. 마이그레이션 시뮬레이션

### 5.1 Linkerd 점진적 도입 시뮬레이션

```bash
# 1단계: 테스트 네임스페이스에 Linkerd 주입 활성화
kubectl annotate namespace ch23-bench linkerd.io/inject=enabled

# 2단계: 파드 재시작으로 사이드카 주입
kubectl rollout restart deployment -n ch23-bench

# 3단계: 주입 확인
kubectl get pods -n ch23-bench -o jsonpath='{range .items[*]}{.metadata.name}: {.spec.containers[*].name}{"\n"}{end}'
# linkerd-proxy 컨테이너가 추가되어 있어야 함
---

# 4단계: 메시 상태 확인 (Linkerd가 설치된 경우)
# linkerd check --namespace ch23-bench
```

### 5.2 메시 간 공존 테스트 (위험 구간 이해)

```bash
# 현재 네임스페이스의 메시 주입 레이블 확인
kubectl get namespace ch23-bench --show-labels

# Linkerd와 Istio 레이블이 동시에 있는 경우 위험
# kubectl label namespace ch23-bench istio-injection=enabled  # 이렇게 하면 안 됨
---

# 안전한 전환: 한 번에 하나만
# 1) Linkerd 비활성화
kubectl annotate namespace ch23-bench linkerd.io/inject=disabled --overwrite

# 2) 파드 재시작 후 Istio 활성화
kubectl rollout restart deployment -n ch23-bench
kubectl label namespace ch23-bench istio-injection=enabled
kubectl rollout restart deployment -n ch23-bench
```
