# Istio 레질리언스 실습

> 본 장의 개념을 손으로 확인한다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`istioctl` 범용 명령으로 쓴다. 환경이 바뀌면 `## 실습 환경` 섹션만 갈아 끼우면 된다.


## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker |
| K8s | kubeadm v1.31.14 | — |
| Istio | istioctl 기준 설치 | 메시 주입 네임스페이스: `default`, `bookinfo` |

접속은 `gcp` 스킬의 SSH 절차를 따른다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1. 로드 밸런싱 알고리즘 비교 (Fortio)

```bash
# Fortio 부하 테스트 도구 배포
kubectl apply -f https://raw.githubusercontent.com/istio/istio/release-1.22/samples/httpbin/httpbin.yaml -n bookinfo

kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: fortio
  namespace: bookinfo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: fortio
  template:
    metadata:
      labels:
        app: fortio
    spec:
      containers:
      - name: fortio
        image: fortio/fortio:latest
        ports:
        - containerPort: 8080
EOF

# 기본 Round Robin (baseline 측정)
kubectl exec -n bookinfo deploy/fortio -- fortio load \
  -qps 100 -c 10 -t 30s \
  http://reviews:9080/reviews/1

# LEAST_REQUEST로 변경
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: reviews-lb
  namespace: bookinfo
spec:
  host: reviews
  trafficPolicy:
    loadBalancer:
      simple: LEAST_REQUEST
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
EOF

# 변경 후 동일 부하 테스트 (p99 레이턴시 비교)
kubectl exec -n bookinfo deploy/fortio -- fortio load \
  -qps 100 -c 10 -t 30s \
  http://reviews:9080/reviews/1
```

두 측정값의 p99를 비교해 처리 시간 편차가 큰 환경에서 LEAST_REQUEST의 효과를 확인한다.


## 실습 2. VirtualService 타임아웃 설정

```bash
# ratings 서비스에 2초 지연 주입 (타임아웃 트리거용)
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: ratings-delay
  namespace: bookinfo
spec:
  hosts:
  - ratings
  http:
  - fault:
      delay:
        percentage:
          value: 100.0
        fixedDelay: 2s
    route:
    - destination:
        host: ratings
        subset: v1
EOF

# reviews에 1초 타임아웃 설정 (2초 지연 → 타임아웃 발생)
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: reviews-timeout
  namespace: bookinfo
spec:
  hosts:
  - reviews
  http:
  - route:
    - destination:
        host: reviews
        subset: v1
    timeout: 1s
EOF

# 타임아웃 발생 확인 (504 예상)
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s -o /dev/null -w "%{http_code}" http://reviews:9080/reviews/1

# Access Log에서 UT 플래그 확인
kubectl logs -n bookinfo -l app=reviews -c istio-proxy | grep '"response_flags":"UT"'

# 정리
kubectl delete virtualservice ratings-delay reviews-timeout -n bookinfo
```


## 실습 3. 재시도 정책 설정

```bash
# ratings에 50% 확률로 500 주입
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: ratings-fault
  namespace: bookinfo
spec:
  hosts:
  - ratings
  http:
  - fault:
      abort:
        percentage:
          value: 50.0
        httpStatus: 500
    route:
    - destination:
        host: ratings
        subset: v1
EOF

# reviews에 재시도 설정 (503 제외하여 서킷 브레이커와 충돌 방지)
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: reviews-retry
  namespace: bookinfo
spec:
  hosts:
  - reviews
  http:
  - route:
    - destination:
        host: reviews
        subset: v1
    timeout: 6s
    retries:
      attempts: 3
      perTryTimeout: 2s
      retryOn: connect-failure,reset,retriable-4xx
EOF

# 재시도 효과 확인 (500 비율이 줄어드는지 관찰)
for i in $(seq 1 10); do
  kubectl exec -n bookinfo deploy/productpage -- \
    curl -s -o /dev/null -w "%{http_code}\n" http://reviews:9080/reviews/1
done

# 정리
kubectl delete virtualservice ratings-fault reviews-retry -n bookinfo
```


## 실습 4. 연결 풀 서킷 브레이킹 (connectionPool)

```bash
# 엄격한 connectionPool 설정
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: reviews-circuit-breaker
  namespace: bookinfo
spec:
  host: reviews
  trafficPolicy:
    connectionPool:
      http:
        http1MaxPendingRequests: 1
        http2MaxRequests: 10
  subsets:
  - name: v1
    labels:
      version: v1
EOF

# Fortio로 동시 요청 20개 → UO 플래그 발생 확인
kubectl exec -n bookinfo deploy/fortio -- fortio load \
  -qps 0 -c 20 -t 10s \
  http://reviews:9080/reviews/1

# UO 플래그 발생 확인
kubectl exec -n bookinfo deploy/productpage -c istio-proxy -- \
  curl -s localhost:15000/stats | grep "overflow"

# Prometheus에서 UO 이벤트 조회
# sum(irate(istio_requests_total{response_flags="UO"}[5m])) by (destination_workload)
---

# x-envoy-overloaded 헤더 확인
kubectl exec -n bookinfo deploy/productpage -- \
  curl -sv http://reviews:9080/reviews/1 2>&1 | grep "x-envoy-overloaded"
```


## 실습 5. Outlier Detection (아웃라이어 감지)

```bash
# connectionPool + outlierDetection 조합 설정
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: reviews-outlier
  namespace: bookinfo
spec:
  host: reviews
  trafficPolicy:
    connectionPool:
      http:
        http1MaxPendingRequests: 50
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 10s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
EOF

# Envoy 아웃라이어 감지 통계 확인
kubectl exec -n bookinfo deploy/productpage -c istio-proxy -- \
  curl -s localhost:15000/stats | grep "outlier_detection"

# 엔드포인트 헬스 상태 확인 (ejected 여부)
kubectl exec -n bookinfo deploy/productpage -c istio-proxy -- \
  curl -s localhost:15000/clusters | grep "reviews" | grep "health_flags"
```


## 실습 6. 타임아웃 + 재시도 + 서킷 브레이킹 조합 검증

```bash
# 통합 복원력 설정 적용
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: reviews-resilience
  namespace: bookinfo
spec:
  hosts:
  - reviews
  http:
  - route:
    - destination:
        host: reviews
        subset: v1
    timeout: 6s
    retries:
      attempts: 3
      perTryTimeout: 2s
      retryOn: connect-failure,reset,retriable-4xx
EOF

# timeout >= perTryTimeout * attempts 조건 확인
# 6s >= 2s * 3 = 6s (경계값, 여유를 두려면 timeout을 7-8s로 설정)
---

# Fortio로 통합 부하 테스트
kubectl exec -n bookinfo deploy/fortio -- fortio load \
  -qps 50 -c 5 -t 30s \
  -labels "resilience_test" \
  http://reviews:9080/reviews/1

# 정리
kubectl delete virtualservice reviews-resilience -n bookinfo
kubectl delete destinationrule reviews-outlier reviews-circuit-breaker reviews-lb -n bookinfo
```


## 실습 7. proxy-config로 설정 검증

```bash
PRODUCTPAGE=$(kubectl get pod -n bookinfo -l app=productpage \
  -o jsonpath='{.items[0].metadata.name}')

# 타임아웃/재시도가 Route에 반영됐는지 확인
istioctl proxy-config route ${PRODUCTPAGE}.bookinfo --name 9080 -o json \
  | jq '.[].virtualHosts[].routes[].route | {timeout, retryPolicy}'

# 서킷 브레이커 설정이 Cluster에 반영됐는지 확인
istioctl proxy-config cluster ${PRODUCTPAGE}.bookinfo \
  --fqdn reviews.bookinfo.svc.cluster.local -o json \
  | jq '.[].circuitBreakers'

# Outlier Detection 설정 확인
istioctl proxy-config cluster ${PRODUCTPAGE}.bookinfo \
  --fqdn reviews.bookinfo.svc.cluster.local -o json \
  | jq '.[].outlierDetection'
```


## 체크리스트

- [ ] LEAST_REQUEST와 ROUND_ROBIN의 p99 레이턴시 차이를 Fortio로 측정했는가
- [ ] 타임아웃이 발생하는 상황에서 Access Log의 `UT` 플래그를 확인했는가
- [ ] `retryOn`에서 `503`을 제외한 이유를 설명할 수 있는가 (서킷 브레이커 차단 응답에 대한 재시도 방지)
- [ ] `timeout >= perTryTimeout * attempts` 조건을 만족하는 설정을 작성했는가
- [ ] `UO` 플래그와 `x-envoy-overloaded` 헤더로 서킷 브레이커 동작을 확인했는가
- [ ] `consecutive5xxErrors`와 `consecutiveGatewayErrors`의 차이를 설명할 수 있는가
- [ ] `proxy-config cluster`로 circuitBreakers 설정이 Envoy에 반영됐는지 검증했는가
