# Istio 트래픽 관리 실습

> 본 장의 개념을 손으로 확인한다. 실습은 개인 GCP K8s 클러스터(kubeadm 기반 3노드) 위에서 진행하지만, 본문 명령은 환경에 무관한 `kubectl`/`istioctl` 범용 명령으로 쓴다. 환경이 바뀌면 `## 실습 환경` 섹션만 갈아 끼우면 된다.


## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker |
| K8s | kubeadm v1.31.14 | — |
| Istio | istioctl 기준 설치 | 메시 주입 네임스페이스: `default`, `bookinfo` |

접속은 `gcp` 스킬의 SSH 절차를 따른다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1. VirtualService로 카나리 배포

```bash
# bookinfo 네임스페이스에 레이블 설정 (사이드카 주입 활성화)
kubectl label namespace bookinfo istio-injection=enabled

# v1, v2 Deployment 배포 확인
kubectl get deployments -n bookinfo

# DestinationRule 먼저 배포 (subset 정의)
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: reviews-dr
  namespace: bookinfo
spec:
  host: reviews
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
EOF

# 카나리 VirtualService (v1=90%, v2=10%)
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: reviews-vs
  namespace: bookinfo
spec:
  hosts:
  - reviews
  http:
  - route:
    - destination:
        host: reviews
        subset: v1
      weight: 90
    - destination:
        host: reviews
        subset: v2
      weight: 10
EOF

# 설정 검증
istioctl analyze -n bookinfo

# 트래픽 발생 후 분산 확인
for i in $(seq 1 20); do
  kubectl exec -n bookinfo deploy/productpage -- \
    curl -s http://reviews:9080/reviews/1 | grep -o '"color":"[^"]*"'
done
```

설정이 적용되면 약 10%의 요청에서 v2 응답(별점 표시)이 확인된다.


## 실습 2. 헤더 기반 라우팅

```bash
# QA 헤더를 가진 요청만 v2로
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: reviews-header
  namespace: bookinfo
spec:
  hosts:
  - reviews
  http:
  - match:
    - headers:
        x-user-group:
          exact: "qa"
    route:
    - destination:
        host: reviews
        subset: v2
  - route:
    - destination:
        host: reviews
        subset: v1
EOF

# QA 헤더 포함 요청 → v2
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s -H "x-user-group: qa" http://reviews:9080/reviews/1

# 일반 요청 → v1
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s http://reviews:9080/reviews/1
```


## 실습 3. 장애 주입

```bash
# ratings 서비스에 10% 확률로 HTTP 500 주입
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: ratings-fault
  namespace: bookinfo
spec:
  hosts:
  - ratings
  http:
  - fault:
      abort:
        percentage:
          value: 10.0
        httpStatus: 500
    route:
    - destination:
        host: ratings
        subset: v1
EOF

# 에러 발생 확인 (10회 호출 중 약 1회 500 예상)
for i in $(seq 1 10); do
  kubectl exec -n bookinfo deploy/productpage -- \
    curl -s -o /dev/null -w "%{http_code}" http://ratings:9080/ratings/1
  echo ""
done

# 장애 주입 제거
kubectl delete virtualservice ratings-fault -n bookinfo
```


## 실습 4. 아웃라이어 감지 (서킷 브레이킹)

```bash
# 연결 풀 + 아웃라이어 감지 설정
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: reviews-circuit-breaker
  namespace: bookinfo
spec:
  host: reviews
  trafficPolicy:
    connectionPool:
      http:
        http1MaxPendingRequests: 10
        http2MaxRequests: 100
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 10s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
EOF

# Envoy 서킷 브레이커 통계 확인
kubectl exec -n bookinfo deploy/productpage -c istio-proxy -- \
  curl -s localhost:15000/stats | grep "overflow"
```


## 실습 5. 트래픽 미러링

```bash
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: reviews-mirror
  namespace: bookinfo
spec:
  hosts:
  - reviews
  http:
  - route:
    - destination:
        host: reviews
        subset: v1
      weight: 100
    mirror:
      host: reviews
      subset: v2
    mirrorPercentage:
      value: 100.0
EOF

# v2 로그에서 미러 요청 확인 (shadow- 접두사 확인)
kubectl logs -n bookinfo -l version=v2 --tail=10
```


## 실습 6. proxy-config 설정 검증

```bash
PRODUCTPAGE=$(kubectl get pod -n bookinfo -l app=productpage \
  -o jsonpath='{.items[0].metadata.name}')

# Listener 확인
istioctl proxy-config listener ${PRODUCTPAGE}.bookinfo --port 9080

# Route 확인 (VirtualService 반영 여부)
istioctl proxy-config route ${PRODUCTPAGE}.bookinfo --name 9080

# Cluster 확인 (subset 포함)
istioctl proxy-config cluster ${PRODUCTPAGE}.bookinfo \
  --fqdn reviews.bookinfo.svc.cluster.local

# Endpoint 확인
istioctl proxy-config endpoint ${PRODUCTPAGE}.bookinfo \
  --cluster "outbound|9080|v2|reviews.bookinfo.svc.cluster.local"
```


## 실습 7. ServiceEntry로 외부 서비스 등록

```bash
# REGISTRY_ONLY 모드에서 외부 서비스 등록
kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: ServiceEntry
metadata:
  name: external-httpbin
  namespace: bookinfo
spec:
  hosts:
  - httpbin.org
  ports:
  - number: 443
    name: https
    protocol: HTTPS
  location: MESH_EXTERNAL
  resolution: DNS
EOF

# 등록 후 접근 테스트
kubectl exec -n bookinfo deploy/productpage -- \
  curl -s https://httpbin.org/get | head -5
```


## 체크리스트

- [ ] DestinationRule에 subset을 먼저 배포한 뒤 VirtualService를 적용했는가
- [ ] `istioctl analyze`로 설정 오류가 없는지 확인했는가
- [ ] 카나리 트래픽 분산이 예상 비율로 동작하는지 실측했는가
- [ ] 장애 주입 후 애플리케이션의 폴백 동작을 확인했는가
- [ ] 미러링에서 v2 로그에 `shadow-` 접두사 요청이 찍히는지 확인했는가
- [ ] `proxy-config endpoint` 명령으로 각 subset에 Pod IP가 올바르게 등록됐는지 확인했는가
