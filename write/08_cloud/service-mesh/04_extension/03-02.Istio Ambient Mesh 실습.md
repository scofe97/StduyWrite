# Istio Ambient Mesh 실습

> 본 장의 개념을 손으로 확인합니다. 현재 실습 환경은 개인 GCP K8s 클러스터이며, 환경이 바뀌면 `## 실습 환경` 섹션만 갈아 끼우면 됩니다.


## 실습 환경` 섹션만 갈아 끼우면 됩니다.


## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터입니다. 구성은 다음과 같습니다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| CNI | Flannel (10.244.0.0/16) | — |

Ambient Mesh는 Istio 1.24+ 및 Linux 커널 5.4+ 환경이 필요합니다. Ambient는 1.24에서 GA로 승격됐습니다. 기존 사이드카 모드 Istio가 설치된 클러스터에서 Ambient 컴포넌트를 추가하는 방식으로 진행합니다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1. Ambient 모드 설치

### 1-1. 사전 조건 확인

```bash
# Istio 버전 확인 (1.24+ 필요)
istioctl version

# 커널 버전 확인 (5.4+ 필요)
uname -r

# 현재 CNI 확인
kubectl get pods -n kube-system | grep -E "flannel|calico|cilium"
```

### 1-2. Ambient 컴포넌트 추가 설치 (Helm 방식)

```bash
helm repo add istio https://istio-release.storage.googleapis.com/charts
helm repo update

# CNI 플러그인 (Ambient 모드 전환)
helm upgrade istio-cni istio/cni -n istio-system \
  --set profile=ambient \
  --wait

# ztunnel DaemonSet
helm install ztunnel istio/ztunnel -n istio-system --wait

# 설치 확인
kubectl get pods -n istio-system
kubectl get daemonset ztunnel -n istio-system
```

### 1-3. ztunnel 상태 확인

```bash
# 각 노드에 ztunnel Pod 1개씩 배포됐는지 확인
kubectl get pods -n istio-system -l app=ztunnel -o wide

# ztunnel 로그 확인
kubectl logs -n istio-system -l app=ztunnel --tail=20
```


## 실습 2. 네임스페이스 Ambient 모드 전환

### 2-1. 테스트 워크로드 배포 (사이드카 모드로 먼저)

```bash
kubectl create namespace ch22-ambient

kubectl apply -n ch22-ambient -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: httpbin
spec:
  replicas: 2
  selector:
    matchLabels:
      app: httpbin
  template:
    metadata:
      labels:
        app: httpbin
    spec:
      containers:
      - name: httpbin
        image: kennethreitz/httpbin:latest
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: httpbin
spec:
  selector:
    app: httpbin
  ports:
  - port: 80
    targetPort: 80
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sleep
spec:
  replicas: 1
  selector:
    matchLabels:
      app: sleep
  template:
    metadata:
      labels:
        app: sleep
    spec:
      containers:
      - name: sleep
        image: curlimages/curl:latest
        command: ["/bin/sleep", "infinity"]
EOF

kubectl get pods -n ch22-ambient
```

### 2-2. Ambient 모드 활성화

```bash
# 사이드카 주입 레이블 제거 (이미 없으면 생략)
kubectl label namespace ch22-ambient istio-injection-

# Ambient 모드 활성화
kubectl label namespace ch22-ambient istio.io/dataplane-mode=ambient

# 파드 재시작 (사이드카 제거 적용)
kubectl rollout restart deployment -n ch22-ambient

# 파드에 istio-proxy 컨테이너가 없는지 확인
kubectl get pods -n ch22-ambient -o jsonpath='{.items[0].spec.containers[*].name}'
# httpbin 만 출력되면 성공 (istio-proxy 없음)
```

### 2-3. Ambient 등록 확인

```bash
# ztunnel이 워크로드를 인식하는지 확인
istioctl ztunnel-config workload -n ch22-ambient

# mTLS 상태 확인
SLEEP_POD=$(kubectl get pod -n ch22-ambient -l app=sleep -o jsonpath='{.items[0].metadata.name}')
istioctl x describe pod $SLEEP_POD -n ch22-ambient
```


## 실습 3. HBONE 터널 통신 검증

### 3-1. 파드 간 통신 테스트

```bash
SLEEP_POD=$(kubectl get pod -n ch22-ambient -l app=sleep -o jsonpath='{.items[0].metadata.name}')

# Ambient 모드에서도 정상 통신 확인
kubectl exec -n ch22-ambient $SLEEP_POD -- \
  curl -s http://httpbin.ch22-ambient.svc.cluster.local/headers

# 응답 헤더에서 mTLS 관련 정보 확인
kubectl exec -n ch22-ambient $SLEEP_POD -- \
  curl -s http://httpbin.ch22-ambient.svc.cluster.local/headers | \
  python3 -m json.tool | grep -i "x-forwarded\|x-envoy\|x-b3"
```

### 3-2. ztunnel 로그로 HBONE 터널 확인

```bash
# 트래픽 발생 후 ztunnel 로그에서 HBONE 터널 확인
kubectl logs -n istio-system -l app=ztunnel --tail=30 | \
  grep -i "hbone\|tunnel\|connection"
```


## 실습 4. waypoint 프록시 배포 (L7 기능 활성화)

### 4-1. waypoint 생성

```bash
# 네임스페이스 단위 waypoint
istioctl waypoint apply -n ch22-ambient --enroll-namespace

# 생성 확인
kubectl get gateway -n ch22-ambient
kubectl get pods -n ch22-ambient | grep waypoint
```

### 4-2. L7 AuthorizationPolicy 적용

```bash
kubectl apply -n ch22-ambient -f - <<EOF
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: httpbin-get-only
spec:
  selector:
    matchLabels:
      app: httpbin
  action: ALLOW
  rules:
  - to:
    - operation:
        methods: ["GET"]
EOF

# GET은 허용, POST는 차단되는지 확인
kubectl exec -n ch22-ambient $SLEEP_POD -- \
  curl -s -o /dev/null -w "%{http_code}" \
  http://httpbin.ch22-ambient.svc.cluster.local/get

kubectl exec -n ch22-ambient $SLEEP_POD -- \
  curl -s -o /dev/null -w "%{http_code}" -X POST \
  http://httpbin.ch22-ambient.svc.cluster.local/post
# 403 응답이 나와야 한다
```


## 실습 5. 롤백: Ambient → 사이드카

```bash
# Ambient 비활성화
kubectl label namespace ch22-ambient istio.io/dataplane-mode-

# 사이드카 재활성화
kubectl label namespace ch22-ambient istio-injection=enabled

# 파드 재시작 (사이드카 재주입)
kubectl rollout restart deployment -n ch22-ambient

# 사이드카 컨테이너 확인
kubectl get pods -n ch22-ambient -o jsonpath='{.items[0].spec.containers[*].name}'
# httpbin istio-proxy 가 출력되면 성공
```


## 실습 6. 정리

```bash
# waypoint 제거
istioctl waypoint delete --all -n ch22-ambient

# 네임스페이스 제거
kubectl delete namespace ch22-ambient
```
