# 멀티클러스터 실습

> 본 장의 개념을 손으로 확인합니다. 현재 실습 환경은 개인 GCP K8s 클러스터이며, 환경이 바뀌면 `## 실습 환경` 섹션만 갈아 끼우면 됩니다.


## 실습 환경` 섹션만 갈아 끼우면 됩니다.


## 실습 환경

현재 실습 대상은 `gcp` 스킬에 등록된 개인 클러스터입니다. 구성은 다음과 같습니다.

| 항목 | 값 | 역할 |
|------|-----|-----|
| dev-server | 4 vCPU / 8GB · asia-northeast3-a | Control Plane |
| dev-server-2 | 4 vCPU / 8GB | Worker |
| dev-server-3 | 4 vCPU / 8GB | Worker (주 워크로드) |
| K8s | kubeadm v1.31.14 | — |
| CNI | Flannel (10.244.0.0/16) | — |

단일 클러스터 환경에서 멀티클러스터 개념을 검증하는 실습입니다. 두 번째 클러스터가 없는 경우, Istio의 ServiceEntry와 지역 인식 라우팅 설정을 시뮬레이션하는 방식으로 진행합니다. 상세는 [`.claude/skills/gcp/SKILL.md`](../../../../.claude/skills/gcp/SKILL.md) 참조.


## 실습 1. 공유 루트 CA 기반 mTLS 검증

### 1-1. 현재 클러스터 CA 확인

```bash
# istiod가 사용하는 CA 인증서 확인
kubectl get secret cacerts -n istio-system -o yaml 2>/dev/null || \
  echo "자체 서명 CA 사용 중 (기본값)"

# Envoy sidecar 인증서 확인
kubectl get pods -n istio-system -l app=istiod

PILOT_POD=$(kubectl get pod -n istio-system -l app=istiod -o jsonpath='{.items[0].metadata.name}')
kubectl exec -n istio-system $PILOT_POD -- \
  openssl x509 -in /var/run/secrets/istio-dns/cert-chain.pem \
  -noout -subject -issuer 2>/dev/null | head -5
```

### 1-2. 워크로드 SVID 확인

```bash
kubectl create namespace ch20-multicluster

kubectl apply -n ch20-multicluster -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sleep
spec:
  replicas: 1
  selector:
    matchLabels:
      app: sleep
  template:
    metadata:
      labels:
        app: sleep
    spec:
      containers:
      - name: sleep
        image: curlimages/curl:latest
        command: ["/bin/sleep", "infinity"]
EOF

SLEEP_POD=$(kubectl get pod -n ch20-multicluster -l app=sleep -o jsonpath='{.items[0].metadata.name}')

# 사이드카가 발급받은 인증서의 SPIFFE ID 확인
istioctl proxy-config secret $SLEEP_POD.ch20-multicluster -o json | \
  python3 -c "
import json, sys, base64, subprocess
data = json.load(sys.stdin)
cert = data['dynamicActiveSecrets'][0]['secret']['tlsCertificate']['certificateChain']['inlineBytes']
cert_pem = base64.b64decode(cert)
print(cert_pem[:500])
"
```


## 실습 2. ServiceEntry로 외부 서비스 등록

멀티클러스터에서 원격 서비스를 등록하는 패턴을 단일 클러스터에서 시뮬레이션합니다. ServiceEntry는 실제 멀티클러스터 환경에서 원격 클러스터 엔드포인트를 등록할 때도 동일하게 사용합니다.

### 2-1. 외부 서비스 등록

```bash
kubectl apply -n ch20-multicluster -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: ServiceEntry
metadata:
  name: external-httpbin
spec:
  hosts:
  - httpbin.org
  ports:
  - number: 443
    name: https
    protocol: HTTPS
  resolution: DNS
  location: MESH_EXTERNAL
EOF
```

### 2-2. 접근 테스트

```bash
kubectl exec -n ch20-multicluster $SLEEP_POD -- \
  curl -s https://httpbin.org/get | head -20
```


## 실습 3. 지역 인식 라우팅 설정

단일 클러스터에서 노드 레이블 기반 locality routing을 시뮬레이션합니다.

### 3-1. 노드 레이블 확인

```bash
kubectl get nodes --show-labels | grep topology
```

GCP에서는 보통 `topology.kubernetes.io/region`과 `topology.kubernetes.io/zone`이 설정돼 있습니다.

### 3-2. DestinationRule로 locality 설정

```bash
kubectl apply -n ch20-multicluster -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: locality-routing-dr
spec:
  host: httpbin.ch20-multicluster.svc.cluster.local
  trafficPolicy:
    outlierDetection:
      consecutive5xxErrors: 3
      interval: 30s
      baseEjectionTime: 30s
    loadBalancer:
      localityLbSetting:
        enabled: true
        failover:
          - from: asia-northeast3
            to: us-central1
EOF
```

### 3-3. 멀티클러스터 체크리스트 점검

실제 멀티클러스터 환경이 있다면 아래 명령으로 상태를 점검합니다.

```bash
# Istio 멀티클러스터 연결 상태
istioctl remote-clusters

# Linkerd 멀티클러스터 상태 (Linkerd 설치 시)
# linkerd multicluster check
# linkerd multicluster gateways
```

### 3-4. 실습 정리

```bash
kubectl delete namespace ch20-multicluster
```
