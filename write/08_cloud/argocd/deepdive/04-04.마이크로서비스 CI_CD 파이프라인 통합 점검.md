# 마이크로서비스 CI/CD 파이프라인 통합 점검
---
> 본문(04-04)이 “어떻게 묶는가”를 다뤘다면, 점검 문서는 “설계가 잘 됐는가”를 묻는다. 12개 마이크로서비스 흐름을 운영에 올리기 전에 이 체크리스트를 한 번 통과한다.



## 1. 책임 경계 점검
> CI와 GitOps 사이의 선이 흐려지면 모든 운영 사고가 한 컴포넌트에 몰린다.

| 점검 항목 | 통과 기준 |
|---------|---------|
| CI에 클러스터 자격증명이 있는가? | 없다 — Harbor push까지만 한다 |
| CI가 매니페스트 repo를 직접 수정하는가? | 안 한다 — Image Updater에 일임 |
| 매니페스트 repo가 단일 진실 공급원인가? | 그렇다 — 클러스터 직접 수정은 OutOfSync로 잡힌다 |
| 자동 sync가 켜진 환경과 수동 환경이 분리돼 있는가? | dev/ppp 자동, prd/bok 수동 |
| Application 단위로 환경 사고 반경이 분리되는가? | env별 Application + values 파일 분리 |



## 2. 저장소·values 구조 점검
> 환경별 sentinel은 “파일 분리”와 “PR 게이트”에서 나온다.

| 점검 항목 | 통과 기준 |
|---------|---------|
| `values-{dev,ppp,prd,bok}.yaml` 4개가 모두 존재하는가? | 그렇다 |
| 같은 차트가 환경별로 다른 image registry를 가리키는가? | 305P=Harbor middleware, BOK=폐쇄망 사전 적재 |
| Image Updater의 `write-back-target`이 환경 단위로 분리돼 있는가? | dev/ppp만 자동 갱신 대상, prd/bok 제외 |
| 외부 레지스트리(`docker.io`, `quay.io`) 직접 참조가 없는가? | `helm template ... | grep image: | sort -u`로 확인 |
| Helm 차트 디렉토리 트리에 “서비스별 templates 폴더”가 있는가? | 그렇다 — 서비스 단위 격리 |



## 3. ApplicationSet 설계 점검
> Matrix는 켜는 순간 폭발한다. 환경 단위 List가 가장 안전하다.

| 점검 항목 | 통과 기준 |
|---------|---------|
| generator 타입을 의도적으로 선택했는가? | List/Cluster/Git/PR 중 사유 포함 |
| Matrix를 쓰는가? | 안 쓴다 — 또는 ApplicationSet 두 개로 분리 |
| RollingSync `maxUpdate`가 prd/bok에서 50% 이하인가? | 그렇다 — 영향도 큰 환경 보수적으로 |
| ApplicationSet 변경이 PR 리뷰 대상인가? | 그렇다 — 자동 머지 금지 |



## 4. 배포 전략 점검
> Rollouts를 도입했다면 “안전 임계치가 정말 안전한가”를 한 번 더 본다.

| 점검 항목 | 통과 기준 |
|---------|---------|
| AnalysisTemplate 임계치가 베이스라인 측정 후 결정됐는가? | 그렇다 |
| `pause: {}` 무한 대기가 운영 중간에 들어가 있지는 않은가? | 없다 — hotfix 전용 |
| canary `setWeight: 1` 직후 곧바로 analysis가 도는가? | 안 된다 — pause로 표본 안정 후 |
| autoPromotionEnabled가 prd에서 꺼져 있는가? | 그렇다 — 사람의 promote 신호 필요 |
| `kubectl edit rollout`으로 단계 비율을 직접 바꾸는 운영 관행이 있는가? | 없다 — 항상 Git 또는 CLI promote/abort |



## 5. 알림·관측성 점검
> “모를 때가 위험한 때”다. 자동 롤백이 일어났는데 누구도 모르는 상황을 막는다.

| 점검 항목 | 통과 기준 |
|---------|---------|
| `on-sync-failed` 트리거가 운영 채널로 가는가? | 그렇다 |
| `RolloutAborted` 이벤트가 별도 알림으로 잡히는가? | 그렇다 |
| 성공 알림이 노이즈가 되지 않게 채널이 분리돼 있는가? | `#deploys-success` / `#deploys-fail` |
| ArgoCD Prometheus 메트릭이 Grafana 대시보드에 묶여 있는가? | argocd-overview, app-controller 대시보드 임포트 |
| 30분 이상 OutOfSync가 지속되면 경고가 발송되는가? | AlertManager 룰 존재 |



## 6. 305P 적용 매핑
> 본 점검을 305P 환경에 매핑할 때 보는 위치.

| 점검 영역 | 305P 위치 | 메모 |
|---------|---------|------|
| 저장소 | `bitbucket.org/okestrolab/tps_manifest` | monorepo |
| 차트 | `helm-charts/tps-helm/` | 12개 서비스 우산 |
| values | `helm-charts/tps-helm/values/values-ppp.yaml` | 환경별 분리 |
| Application/ApplicationSet | `argocd-apps/app-of-apps/<env>/`, `argocd-apps/applicationset/` | 부트스트랩 + 양산 |
| Image Updater 설정 | `argocd-apps/image-updater/` | `trb-oss` 네임스페이스 |
| 알림 | `argocd-notifications-cm` | Slack token Secret 분리 |
| 메트릭 | `prometheus.trb-mgm.svc:9090` | Grafana(`grafana.dev.trombone-v2.okestro.cloud`) |
| Rollouts(도입 시) | `trb-oss/argo-rollouts` | 별도 Application으로 부트스트랩 |

토큰·비밀번호 본문 노출 금지 — 실제 값은 인프라 스킬 문서 `tps/infra/SKILL.md`, `references/14-v305p-environment.md`, `references/07-argocd-image-updater.md`에서만 관리한다.



## 7. 이관·확장 점검
> 폐쇄망(BOK) 또는 멀티클러스터로 이관할 때 추가로 보는 항목.

| 점검 항목 | 통과 기준 |
|---------|---------|
| 폐쇄망 이미지 사전 적재 절차가 문서화돼 있는가? | Harbor `middleware`/Nexus 프록시 미러링 절차 |
| Image Updater가 폐쇄망 환경 values를 건드리지 않게 막혀 있는가? | `write-back-target`이 환경 분리 |
| AppProject가 환경별 destination을 제한하는가? | dev/ppp/prd/bok namespace whitelist |
| Bitbucket repository Secret이 환경별로 분리돼 있는가? | bitbucket-creds-{env} 분리 |
| 폐쇄망 환경에서 외부 webhook이 차단됐을 때 폴링 fallback이 정상 동작하는가? | ArgoCD 폴링 주기 확인 |
