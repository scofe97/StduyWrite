# Pod 네트워크와 Linux 기반 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검한다.

### Q1. Pause 컨테이너가 죽으면 Pod 안의 다른 컨테이너에 무슨 일이 일어나는가?

Pause는 Pod의 네트워크 네임스페이스(netns) 소유자다. 다른 컨테이너는 `--net=container:<pause-id>` 형태로 같은 netns에 들러붙는 구조라, Pause가 사라지면 그 netns의 reference count가 0이 되며 정리된다. 결과적으로 다른 컨테이너는 자기 네트워크 컨텍스트(IP, 포트, listen socket)를 한꺼번에 잃는다.

kubelet은 Pause가 죽었음을 감지하면 Pod 자체를 새로 만들고, 사이드카·메인 컨테이너도 새 netns 위에서 다시 시작된다. 따라서 운영자가 보는 증상은 "Pod 재시작 카운트 증가"와 "잠깐 끊김"이다. Pause 자체의 안정성 문제로 이런 일이 일어나는 경우는 드물고, 대부분은 노드의 메모리 압박이나 OOM이 Pause까지 함께 끌어내릴 때다.

진단할 때는 `kubectl describe pod`의 `Last State Reason`이 `OOMKilled`인 컨테이너가 메인뿐 아니라 sandbox·infra 단위에도 보이는지를 확인한다. 노드 단의 cgroup 압박이 Pause에 닿았다는 신호다.

### Q2. 같은 Pod 안 두 컨테이너가 같은 포트에서 listen하면 어떻게 되는가?

Pod 안 컨테이너는 같은 netns를 공유하므로 IP와 포트도 한 풀에 있다. 두 컨테이너가 같은 포트에서 listen 시도하면 두 번째 컨테이너의 `bind`는 `EADDRINUSE`로 실패한다. 사이드카 패턴에서 Envoy가 9090에 listen하는데 메인 컨테이너도 9090에 메트릭 endpoint를 두려는 경우가 흔한 충돌이다.

해결은 각 컨테이너의 listen 포트를 분리하는 것이다. 옛 K8s에서는 hostNetwork 같은 트릭이 쓰였지만 보안 비용이 크다. 메인은 8080·메트릭은 9090, Envoy admin은 9901처럼 명시적으로 분리한다.

### Q3. Pod CIDR이 노드별로 다른 이유와 충돌 시 증상은?

각 노드가 Pod에 IP를 할당할 권한을 갖게 하려면, 노드별로 겹치지 않는 CIDR 블록을 가져야 한다. controller-manager의 `--allocate-node-cidrs=true`와 `--cluster-cidr=10.244.0.0/16` 같은 설정이 노드마다 `10.244.1.0/24`, `10.244.2.0/24` 식으로 잘라 분배한다.

CIDR이 충돌하거나 잘못 분배되면 두 노드가 같은 Pod IP를 만들 수 있다. 증상은 "두 Pod의 IP가 같은데 어느 노드 응답이 돌아올지 알 수 없는" 라우팅 혼란이다. iptables/ipvs 규칙이 한 IP에 두 backend를 만드는 식의 사고가 따라온다. `kubectl get nodes -o jsonpath='{.items[*].spec.podCIDR}'`로 노드별 CIDR을 확인해 충돌이 없는지 본다.

### Q4. CNI 플러그인 변경(flannel → calico)을 무중단으로 할 수 있는가?

거의 불가능하다. CNI는 노드의 라우팅·iptables 설정을 직접 만들고, Pod의 netns를 그 위에서 구성한다. 플러그인 교체 중에는 두 CNI의 dataplane이 같은 노드에서 충돌하므로 일관성 깨진 상태가 된다.

실무 절차는 노드 단위 cordon·drain → CNI 교체 → 노드 재가입이다. 클러스터 전체로 보면 Rolling 방식이지만, 같은 노드의 Pod는 CNI 교체 동안 잠시 끊기거나 다시 만들어진다. 같은 클러스터 안에서 두 CNI가 공존하는 멀티 CNI 구성은 K8s 1.28+의 Multus 같은 보조 도구로 일부 시나리오에서만 제한적으로 가능하다.

### Q5. kube-proxy의 iptables·IPVS·eBPF 모드는 운영 시 무엇이 다른가?

`iptables` 모드는 가장 호환성이 높고 단순하다. Service 1개당 chain이 추가되므로 Service 수가 1만 단위로 늘면 룰 평가 비용이 선형으로 증가한다. 작은~중간 클러스터에 충분하다.

`IPVS` 모드는 hash 테이블 기반이라 Service 수가 많아져도 평가 비용이 거의 일정하다. 대규모 클러스터에서 latency가 안정적이다. 단, 일부 ConnectionTracking 엣지 케이스가 있어 Calico/Cilium 일부 기능과 상호작용 시 주의가 필요하다.

`eBPF` 모드(Cilium의 kube-proxy 대체)는 노드 커널의 tc/XDP 훅에서 직접 패킷을 라우팅한다. 가장 빠르며 conntrack도 자체 관리한다. 다만 커널 버전과 BPF 프로그램 호환성 의존이 크다. 매니지드 K8s에서 항상 가능한 선택지는 아니다.

### Q6. veth pair 한 쪽이 손상되면 어떻게 발견하는가?

`ip -n <netns> link`로 Pod 안에서 `eth0` 상태를, 호스트 root netns에서 `ip link`로 짝(`veth-xxxx`) 상태를 본다. 한쪽만 `DOWN`이면 짝이 손상된 것이다.

증상은 "Pod 내부에서 외부로 ping이 안 가지만 컨테이너는 살아 있음"이다. `tcpdump -i veth-xxxx`로 호스트 측에서 패킷이 들어오는지 확인하고, 없으면 Pod 내부 발신 자체가 막힌 것이다. 주로 CNI 플러그인 버그나 노드 커널 issue가 원인이라 임시 해결은 Pod를 다른 노드로 옮기는 것이다.

### Q7. Pod IP가 외부에서 직접 접근되지 않는 이유는?

Pod IP는 클러스터 내부 routable subnet 안에서만 유효하다. 노드 간에는 CNI가 라우팅을 만들지만, 외부 인터넷은 그 routing table을 모른다. 외부에서 Pod에 직접 도달하려면 LoadBalancer Service나 NodePort, Ingress를 통해 NAT돼야 한다.

이 모델은 보안 측면의 기본 격리다. 외부에서 Pod IP로 직접 들어오면 NetworkPolicy 같은 클러스터 내 정책이 우회될 위험이 있다. Pod IP를 외부 노출 인터페이스로 쓰지 않고 항상 Service 계층을 거치게 하는 패턴이 권장된다.

### Q8. Kubernetes 네트워킹의 4규칙을 잊었을 때 어떤 사고가 생기는가?

본문에 정리된 4규칙(NAT 없는 Pod-to-Pod, NAT 없는 노드-to-Pod, Pod의 자기 IP 인식, Service VIP를 통한 안정 진입)을 어기면 다양한 사고가 난다. NAT를 끼우면 NetworkPolicy의 source IP 매칭이 깨져 정책이 의도와 다르게 작동하고, Pod가 자기 IP를 모르면 메트릭 라벨링·로깅에서 잘못된 IP가 찍힌다.

특히 hostNetwork: true를 켠 Pod는 노드 IP를 그대로 쓰므로 4규칙의 1·2번이 깨진다. 호스트 IP 접근이 필요한 시스템 Pod 외에는 hostNetwork를 쓰지 않는 편이 안전하다.
