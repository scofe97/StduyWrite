# 네트워킹 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검한다.

### Q1. CNI, CoreDNS, kube-proxy는 무엇을 각각 담당하는가?

CNI는 Pod에 IP를 붙이고 Pod 간 네트워크 경로를 만든다. CoreDNS는 Service와 Pod 이름을 DNS로 해석한다. kube-proxy는 Service의 ClusterIP나 NodePort 요청을 실제 Pod IP로 전달한다.

장애를 나눠 보면 더 명확하다. Pod끼리 IP 직접 통신이 안 되면 CNI 쪽을 먼저 본다. 이름은 안 풀리는데 IP로는 붙으면 CoreDNS 문제일 가능성이 높다. 이름은 풀리는데 Service로만 실패하면 kube-proxy나 EndpointSlice 쪽을 의심한다.

운영 디버깅에서는 이 셋을 섞지 않는 것이 핵심이다. "네트워크가 안 된다"가 아니라 "경로 생성, 이름 해석, Service 포워딩 중 어디가 실패했는가"로 나눠야 한다.

### Q2. kube-proxy의 iptables 모드와 IPVS 모드 차이는?

iptables 모드는 Service와 Endpoint 변경 시 리눅스 iptables 규칙을 생성·삭제한다. 규칙 체인을 O(n) 선형 탐색하므로 Service 수가 많아질수록 지연이 증가한다. Pod 수 1,000개 미만의 소규모 클러스터에는 기본값으로 충분하다.

IPVS 모드는 커널의 IP Virtual Server를 사용해 해시 테이블 기반 O(1) 조회를 제공한다. Service 10,000개에서도 밀리초 수준의 지연을 유지한다. ipvsadm 커널 모듈이 필요하다. kube-proxy CPU 사용률이 높거나 Service 응답 시간이 증가할 때 전환을 검토한다.

IPVS 모드에서도 NodePort, IP Masquerade, NetworkPolicy는 여전히 iptables를 사용한다는 점을 알아야 한다. IPVS의 `sh`(Source Hashing) 알고리즘은 소스 IP를 해싱해 항상 같은 Pod로 라우팅하므로 SessionAffinity와 유사하지만, SessionAffinity는 conntrack 기반이라 작동 방식이 다르다.

### Q3. ClusterIP Service의 내부 로드밸런싱 동작 과정은?

클라이언트 Pod가 `http://backend:80`으로 요청하면 CoreDNS가 `backend.default.svc.cluster.local`을 ClusterIP(10.96.5.10)로 해석한다. iptables PREROUTING 체인의 KUBE-SERVICES 규칙이 매칭되고, 확률 기반으로 KUBE-SEP 체인 중 하나를 선택해 DNAT로 Pod IP로 포워딩한다.

같은 TCP 연결 내 패킷은 conntrack으로 같은 Pod로 유지된다. Service ClusterIP는 실제 네트워크 인터페이스에 할당되지 않고 iptables 규칙으로만 존재한다. Endpoint가 0개일 때 Service로 요청하면 타임아웃이 발생한다.

최신 Kubernetes에서는 백엔드 표현을 `EndpointSlice`로 관리하는 것이 기본이다. 그래서 기존 `kubectl get endpoints`만 보지 말고 `kubectl get endpointslices`도 같이 확인하는 습관이 필요하다.

```bash
kubectl get endpoints <service-name>   # Endpoint 목록 확인
kubectl run dnstest --image=busybox:1.28 -it --rm -- nslookup <service-name>
```

### Q4. Ingress Controller와 Ingress 리소스의 관계는?

Ingress 리소스는 라우팅 규칙을 선언한 YAML이고, Ingress Controller는 규칙을 실제 구현하는 Pod다. Ingress Controller가 API 서버를 watch해 Ingress 리소스를 감지하면 nginx.conf를 재생성하고 graceful reload를 실행한다.

여러 Ingress Controller를 동시에 설치할 수 있으며 `ingressClassName`으로 어떤 Controller가 처리할지 지정한다. Ingress Controller가 다운되면 502/503 오류가 발생하므로 Deployment로 배포해 자동 복구가 되도록 해야 한다.

GCP 클러스터에서 Ingress를 사용할 때는 nginx-ingress의 NodePort(31292)를 Worker 노드 IP와 함께 사용한다.

```bash
# Ingress 상태 확인
kubectl get ingress -A
kubectl describe ingress <name>

# Ingress Controller 로그
kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx
```

### Q5. NetworkPolicy의 기본 동작은?

NetworkPolicy가 없으면 클러스터 내 모든 Pod은 서로 자유롭게 통신할 수 있다. NetworkPolicy를 적용하는 순간 해당 Pod에 대한 기본 동작이 차단으로 전환된다. 이후 규칙에 명시적으로 허용된 트래픽만 통과한다.

`podSelector: {}`는 같은 네임스페이스 내 모든 Pod를 선택한다. policyTypes에 Ingress만 명시하면 Egress는 여전히 모두 허용된다. 여러 정책이 같은 Pod에 적용되면 OR 연산(합집합)으로 처리되므로, 하나의 정책만 허용해도 트래픽이 통과한다.

NetworkPolicy는 CNI 플러그인이 구현해야 작동한다. 개인 GCP 클러스터는 Flannel을 사용하는데 Flannel은 NetworkPolicy를 지원하지 않는다. 실제 NetworkPolicy를 테스트하려면 Calico나 Cilium이 필요하다.

```bash
kubectl get networkpolicy -A
kubectl describe networkpolicy <name>
```

### Q6. Headless Service의 DNS 동작 차이는?

일반 ClusterIP Service는 DNS 조회 시 ClusterIP 하나를 반환한다. kube-proxy가 Pod 선택을 담당한다. 클라이언트는 백엔드 Pod를 알 수 없다.

Headless Service(`clusterIP: None`)는 DNS 조회 시 모든 Ready Pod의 IP 목록을 반환한다. ClusterIP가 없으므로 kube-proxy 규칙이 생성되지 않는다. 클라이언트가 직접 Pod를 선택해야 한다.

StatefulSet과 함께 쓰면 각 Pod에 대한 안정적인 DNS 이름이 생성된다(`pod-name.service-name.namespace.svc.cluster.local`). Pod가 재시작되어 IP가 바뀌어도 DNS 이름은 유지된다.

```bash
# DNS 테스트
kubectl run dnstest --image=busybox:1.28 -it --rm -- nslookup nginx-headless
kubectl run dnstest --image=busybox:1.28 -it --rm -- nslookup web-0.nginx-headless
```

Headless Service에서 SessionAffinity는 의미가 없다. ClusterIP가 없으므로 kube-proxy가 관여하지 않기 때문이다.

### Q7. GCP 클러스터에서 LoadBalancer 타입 Service 접근 방법은?

kubeadm으로 구성한 개인 GCP 클러스터는 클라우드 컨트롤러 매니저가 없어 LoadBalancer 타입 Service의 External IP가 `<pending>` 상태로 유지된다.

외부 접근을 위한 대안은 두 가지다. NodePort Service를 만들고 Worker 노드의 외부 IP와 NodePort를 사용하거나, 설치된 nginx-ingress(NodePort 31292/31726)를 통해 Ingress로 접근하는 방법이다.

```bash
# NodePort Service 생성
kubectl expose deployment nginx --type=NodePort --port=80
kubectl get svc nginx  # NodePort 번호 확인

# Worker 노드 IP 확인
kubectl get nodes -o wide   # EXTERNAL-IP 컬럼

# 접근 테스트 (Worker 노드 IP + NodePort)
curl http://<worker-node-external-ip>:<nodeport>
```

NodePort를 통한 외부 접근 시 기본적으로 SNAT이 발생해 클라이언트 IP가 노드 IP로 바뀐다. 클라이언트 IP를 보존하려면 `externalTrafficPolicy: Local`을 설정하되, 해당 노드에 Pod가 없으면 요청이 실패한다는 점을 유의한다.

### Q8. Pod-to-Pod, Pod-to-Service, External-to-Service 트래픽 경로 차이는?

Pod-to-Pod 직접 통신은 kube-proxy를 거치지 않고 CNI 네트워크를 통해 직접 전달된다. NAT 없이 출발지 IP가 보존된다. 가장 빠른 경로지만 Pod IP는 임시적이므로 권장하지 않는다.

Pod-to-Service 통신은 CoreDNS가 Service 이름을 ClusterIP로 해석하고, iptables가 DNAT로 Pod IP로 포워딩한다. 출발지 IP는 보존된다(클러스터 내부이므로 SNAT 불필요).

External-to-Service(NodePort)는 외부 클라이언트 IP가 SNAT되어 노드 IP로 변경된 뒤 Pod로 전달된다(`externalTrafficPolicy: Cluster` 기본값). Ingress Controller를 통한 접근은 클라이언트 IP가 X-Forwarded-For 헤더에 담겨 전달된다.

### Q9. dev 환경에서 DNS 정책은 어떻게 가져가는가?

대부분의 dev 환경에서는 `dnsPolicy: ClusterFirst`를 그대로 둔다. Service 이름 해석이 바로 되고, CoreDNS 기본 설정을 그대로 활용할 수 있기 때문이다. 학습용 Pod, 일반 API, 배치 Pod는 이 기본값이면 충분하다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: api-dev
spec:
  dnsPolicy: ClusterFirst
  containers:
  - name: api
    image: nginx:1.29
```

노드 네트워크를 직접 쓰는 Pod는 예외다. `hostNetwork: true`를 사용하면 `dnsPolicy: ClusterFirstWithHostNet`으로 맞춰야 클러스터 내부 Service 이름 해석을 유지할 수 있다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-agent
  namespace: kube-system
spec:
  hostNetwork: true
  dnsPolicy: ClusterFirstWithHostNet
  containers:
  - name: agent
    image: busybox:1.36
    command: ["sh", "-c", "sleep 3600"]
```

`dnsPolicy: None`은 정말 필요한 경우에만 쓴다. 예를 들어 특정 외부 DNS 서버, 사내 search domain, `ndots` 조정이 꼭 필요할 때 `dnsConfig`와 함께 사용한다. dev 환경에서 기본값처럼 쓰기에는 운영 부담이 크다.

### Q10. DNS 기반 서비스 디스커버리의 캐싱 문제와 해결은?

CoreDNS의 기본 TTL은 30초다. Service ClusterIP는 변경되지 않으므로 긴 TTL도 안전하다. 반면 Headless Service의 Pod IP 목록은 Pod 재시작 시 변경되는데, 클라이언트 DNS 캐시가 오래된 IP를 반환하면 30초간 연결 실패가 발생한다.

Java 애플리케이션은 JVM 수준의 DNS 캐시가 별도로 존재한다. `networkaddress.cache.ttl`을 10초 이하로 설정해야 CoreDNS의 TTL이 의미를 가진다. Go는 기본적으로 DNS를 캐싱하지 않는다.

근본 해결책은 클라이언트 라이브러리에서 DNS 실패 시 즉시 재조회하는 재시도 로직을 구현하거나, 연결 풀에서 실패한 IP를 제거하고 재연결하는 것이다. CoreDNS TTL을 5초로 단축하면 DNS 쿼리가 증가해 CoreDNS 부하가 높아지므로 권장하지 않는다.

### Q11. Kubernetes 네트워킹과 Service Mesh는 어디서 역할이 갈라지는가?

Kubernetes 네트워킹은 연결성과 기본 노출을 담당한다. CNI는 Pod 네트워크, Service는 안정된 내부 엔드포인트, Ingress와 Gateway API는 외부 진입, NetworkPolicy는 기본 접근 제어를 맡는다.

Service Mesh는 그 위에서 서비스 간 호출 정책을 다룬다. 예를 들어 가중치 기반 트래픽 분할, 재시도와 타임아웃의 공통화, 서비스 간 mTLS, L7 관측성은 기본 Kubernetes 네트워킹만으로는 충분히 풀기 어렵다.

실무에서 헷갈리지 않는 기준은 이렇다. "붙을 수 있는가"는 Kubernetes 네트워킹, "어떤 정책으로 호출을 제어할 것인가"는 Service Mesh다. 따라서 Service로도 연결이 안 되면 메시를 보기 전에 CNI, DNS, kube-proxy부터 확인해야 한다.

### Q12. Pause 컨테이너는 왜 존재하는가?

Pod 안 모든 컨테이너가 같은 네트워크 네임스페이스를 공유하려면 그 네임스페이스를 누군가 만들고 끝까지 쥐고 있어야 한다. Pause 컨테이너가 그 역할을 한다. 가장 먼저 만들어져 네트워크와 IPC 네임스페이스를 생성하고, PID 1 자리에서 시그널만 처리하며 무한 sleep으로 살아 있는다.

이 구조가 사이드카 패턴의 기반이다. 메인 컨테이너가 재시작되어도 Pause는 살아 있으므로 사이드카는 같은 네임스페이스를 잃지 않는다. 같은 Pod 안 컨테이너가 `localhost:8080`으로 서로를 호출할 수 있는 이유도 이 공유된 네임스페이스 때문이다.

확인은 노드 안에서 한다. `crictl ps -a | grep pause`로 Pause 컨테이너가 Pod마다 하나씩 떠 있는 모습이 보이고, `lsns -t net`을 보면 네트워크 네임스페이스가 Pod 단위로 묶여 있다.

### Q13. veth pair와 Linux bridge는 Pod 네트워크에서 어떤 역할인가?

veth pair는 양 끝이 가상 NIC인 케이블 같은 자료구조다. CNI 플러그인이 Pod를 만들 때 veth pair를 만들어 한쪽 끝(`eth0`)을 Pod 네임스페이스에 넣고, 반대쪽 끝을 노드 root 네임스페이스의 Linux bridge(`cni0` 등)에 꽂는다.

이 한 쌍이 Pod 네임스페이스와 노드 root 네임스페이스 사이의 유일한 통로다. 같은 노드의 Pod끼리 통신할 때는 bridge에 꽂힌 veth들이 ARP로 서로를 찾아 직접 패킷을 주고받는다. 노드를 벗어날 때만 노드의 라우팅 테이블과 NIC을 탄다.

운영 디버깅에서는 노드에서 `ip link show type veth`, `bridge link show`, `ip -n <pod-ns> route`를 차례로 보면서 veth → bridge → 라우팅 테이블 순으로 따라간다. veth가 보이지 않으면 CNI 호출이 실패한 상태일 가능성이 크다.

### Q14. Pod CIDR이 노드별로 분할되면 노드 간 통신은 어떻게 되는가?

클러스터에는 보통 큰 Pod 네트워크(`10.244.0.0/16`)가 있고 노드마다 `/24`처럼 잘려 분배된다. 각 노드의 라우팅 테이블에는 자기 /24가 `dev cni0`로 직접 잡히고, 다른 노드의 /24는 그 노드 외부 IP를 next hop으로 잡는 라우팅 항목이 들어간다.

즉 노드가 자기 Pod CIDR의 라우터 역할을 한다. Pod에서 다른 노드의 Pod로 보낸 패킷은 자기 노드 라우팅 테이블에서 next hop을 찾고 underlay(VPC, 물리 NIC, VXLAN 오버레이 등)로 전달된다. 받은 노드는 dst가 자기 /24이므로 bridge로 보내고 결국 veth 너머 Pod에 도착한다.

이 모델이 깨지는 가장 흔한 두 케이스를 알아 둔다.

1. 보안 그룹이나 사내 방화벽이 노드 간 Pod CIDR 트래픽을 막는 경우.
2. Pod CIDR과 사내망 IP가 겹쳐 Pod에서 사내 서비스가 보이지 않는 경우.

둘 다 클러스터 안에서는 정상으로 보이고 외부 패스가 들어가야 드러난다.

### Q15. kube-proxy가 iptables에서 nftables로 전환하는 배경은?

iptables 모드는 오랜 기본값이지만 Service 수가 늘어나면 규칙 체인이 거대해진다. KUBE-SERVICES → KUBE-SVC-* → KUBE-SEP-* 체인을 선형으로 따라 내려가야 매칭되므로 endpoint 수에 따라 첫 패킷 지연이 늘어나고, 규칙 추가·삭제마다 전체 체인을 다시 쓰는 비용이 든다.

nftables 모드는 1.31에서 beta로 들어와 1.33에서 GA가 됐다. 같은 의미를 nft 표현으로 다시 적되 set과 map 자료구조로 매칭을 해시 기반에 가깝게 만든다. Service 수가 늘어도 매칭 시간이 안정적이고, 부분 갱신 효율도 높다. 새 클러스터라면 nftables 모드를 기본으로 두는 편이 자연스럽다.

다만 nftables는 kube-proxy 자체의 dataplane 교체이므로, eBPF 기반 CNI(Cilium 등)가 kube-proxy를 통째로 대체하는 결정과는 의사결정 축이 다르다. 후자는 CNI 교체가 따라붙는다.

운영 확인 명령은 다음과 같다:

```bash
kubectl -n kube-system get pods -l k8s-app=kube-proxy
kubectl -n kube-system logs -l k8s-app=kube-proxy | grep -i 'proxier\|mode'
nft list ruleset | grep -i kube     # 노드에서 직접 확인
```

### Q16. Cilium IPAM 모드 변경이 위험한 이유는?

Cilium은 Pod IP 할당을 두 가지 모드로 할 수 있다. `kubernetes` 모드는 노드의 `spec.podCIDR`(kubeadm/kubespray가 주입한 /24)을 그대로 쓰고, `cluster-pool` 모드는 Cilium 자체가 클러스터 풀을 관리해 별도 CIDR을 노드에 분배한다. 두 모드는 노드별 Pod CIDR의 출처와 갱신 주체가 다르므로 서로 호환되지 않는다.

운영 중 모드를 바꾸면 노드의 Pod CIDR이 즉시 다른 값으로 재할당될 수 있다. 그 순간 기존 Pod IP는 더 이상 새로운 Pod CIDR 안에 있지 않게 되므로 노드 라우팅과 eBPF 맵의 endpoint 항목이 불일치하고, Pod 사이 통신이 광범위하게 끊긴다. 실제 사고 사례에서는 Cilium Helm 업그레이드 도중 `ipam.mode: kubernetes`로 잘못 머지된 한 줄 때문에 Pod 간 통신이 전부 차단됐고, Helm 롤백으로 CNI는 복구됐지만 그 사이 분산 DB 헬스체크 실패가 누적돼 정합성 복구 작업이 추가로 필요했다.

이 사례에서 배울 운영 규칙은 두 가지다. CNI 설정 변경은 Day-2 변경이 아니라 클러스터 재구성에 가까운 신중함이 필요하다. 그리고 IPAM 모드처럼 한 번 정해지면 되돌리기 어려운 옵션은 클러스터 최초 설계 단계에서 못박고, 그 이후에는 Helm values diff를 두 사람 이상이 점검한 다음에야 적용한다. CNI 변경 직후에는 Pod 간 통신, kube-dns, 분산 미들웨어 헬스체크 세 가지를 분리해 모니터링한다.
