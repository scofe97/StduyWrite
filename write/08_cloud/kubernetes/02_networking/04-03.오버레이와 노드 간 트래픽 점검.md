# 오버레이와 노드 간 트래픽 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검한다.

### Q1. VXLAN 오버레이의 MTU는 왜 1450(또는 1430)으로 자주 설정되나?

VXLAN은 원본 이더넷 프레임을 UDP로 캡슐화한다. 외부 IP 헤더(20) + UDP 헤더(8) + VXLAN 헤더(8) + 내부 이더넷 헤더(14) 합 50바이트의 오버헤드가 추가된다. 노드 간 물리 MTU가 1500이면 안쪽 페이로드가 가질 수 있는 최대치는 1450이다.

MTU를 잘못 설정하면 큰 패킷이 fragment되거나 ICMP `Fragmentation Needed`가 차단된 환경에서 PMTUD가 깨져 무음 패킷 손실이 일어난다. 증상은 "ping은 잘 가는데 큰 HTTP 응답에서 끊김"이다. CNI 설정에서 `tunnel.mtu`를 명시하고, `ip link show flannel.1` 같은 명령으로 실제 인터페이스 MTU를 확인한다.

### Q2. 네이티브 라우팅과 VXLAN 오버레이를 어떻게 선택하는가?

네이티브 라우팅은 노드 간 네트워크가 Pod CIDR 라우팅을 자체적으로 알 수 있을 때 가능하다. 클라우드(VPC route) 또는 자체 BGP 라우팅이 있는 환경이 이에 해당한다. 캡슐화 오버헤드가 없어 latency가 가장 낮다.

VXLAN 오버레이는 노드 간 네트워크가 Pod CIDR을 알지 못해도 동작한다. UDP 4789를 통과시키면 그만이다. 클라우드 외부, 멀티 AS, 데이터센터 간 같은 환경에 자연스럽다. 단, 50바이트 오버헤드와 캡슐화 비용이 있다.

선택 기준은 단순하다. VPC 안에서 가능하다면 네이티브, 그 외는 VXLAN. Cilium은 두 모드를 노드별로 섞을 수 있고, 단계적 전환이 가능하다.

### Q3. BGP 컨트롤 플레인이 깨지면 어떤 증상이 보이는가?

Cilium BGP나 Calico BGP에서 인접 노드와의 BGP peering이 끊기면, 그 노드로 가는 Pod CIDR 라우팅이 사라진다. 다른 노드의 라우팅 테이블에서 해당 prefix가 빠지므로, 그 노드의 Pod로 향하는 트래픽이 라우팅 실패한다.

증상은 "특정 노드의 Pod에만 도달 못 함"이다. `kubectl get nodes`는 정상, Pod도 Running, 그러나 `kubectl exec`이나 Service를 통한 접근이 timeout된다. `cilium bgp peers` 또는 `calicoctl node status`로 BGP 세션 상태를 확인하고, `ip route show` / `ip rule list`로 라우팅 테이블에 prefix가 있는지 본다.

복구는 BGP 데몬 재시작 또는 ToR 스위치와의 BGP 설정 재검증이다. 운영에서는 BGP graceful restart 옵션을 켜 두면 데몬 재시작 중에도 라우팅이 일정 시간 유지된다.

### Q4. MetalLB의 두 모드(Layer 2 vs BGP) 중 무엇을 고르는가?

`Layer 2` 모드는 ARP/NDP 기반이다. 한 노드만 LB IP를 ARP 응답으로 광고하고, 모든 트래픽이 그 노드로 들어온 뒤 kube-proxy로 분산된다. 설정이 단순하지만 단일 진입 노드가 SPOF다. 노드 장애 시 다른 노드가 ARP를 가져오는 데 약간의 지연이 있다.

`BGP` 모드는 노드들이 ToR 스위치에 LB IP를 BGP로 광고한다. ECMP로 여러 노드에 트래픽이 분산된다. 진입 분산과 내장 failover가 있어 운영적으로 더 견고하다. 단, ToR 스위치가 BGP를 받을 수 있어야 하므로 인프라 의존이 크다.

작은 dev/lab은 Layer 2가 충분하다. 프로덕션에서는 BGP가 안정적이다. MetalLB가 클라우드 LB의 대체라기보다, on-prem이나 베어메탈 환경에서 LB가 없을 때의 보충재라는 점을 잊지 않는다.

### Q5. tunnel 모드에서 노드 간 통신이 안 될 때 어디서부터 보는가?

세 단계로 좁힌다.

먼저 **물리/가상 네트워크 도달성**. `ping <other-node-ip>`이 가는지 확인하고, UDP 4789(VXLAN 기본 포트)나 사용 중인 포트가 노드 간 방화벽에서 막히지 않는지 본다. 클라우드 보안 그룹이 자주 누락된다.

다음 **VXLAN 인터페이스 상태**. `ip link show <vxlan-iface>`로 UP 여부, `bridge fdb show dev <vxlan-iface>`로 FDB(MAC↔VTEP IP)가 채워졌는지 확인한다. FDB가 비어 있으면 컨트롤 플레인(보통 etcd 기반)이 노드 정보를 전파하지 못한 것이다.

마지막 **양쪽 노드 모두에서 tcpdump**. `tcpdump -i <phys-iface> udp port 4789`로 캡슐화된 패킷이 보이는지, `tcpdump -i <vxlan-iface>`로 디캡슐화된 안쪽 패킷이 보이는지를 비교한다. 한쪽만 보이면 그 단계에서 막혔다.

### Q6. Pod CIDR을 같은 클러스터에서 늘릴 수 있는가?

대부분 클러스터에서 노드별 podCIDR과 클러스터 전체 cidr 변경은 무중단으로 어렵다. controller-manager 설정·CNI ConfigMap·노드별 라우팅·iptables 모두 일관되게 변경돼야 하므로 사실상 클러스터 재구축에 가깝다.

회피 패턴은 두 가지다. 클러스터 cidr을 처음부터 충분히 크게 잡아 두는 것(예: `/14` ~ `/16`)이 가장 안전하다. 또는 멀티 CIDR 지원 CNI를 쓰면 추가 cidr 풀을 후속 도입할 수 있다(Cilium의 multi-pool, Calico의 IPPool). 운영자가 이미 cidr 부족에 직면했다면 클러스터 마이그레이션 외에 깔끔한 길이 거의 없다.

### Q7. 외부 LoadBalancer가 없는 클러스터(예: kubeadm dev)에서 Service 외부 노출은 어떤 선택지가 있나?

`NodePort` 모드는 모든 노드의 30000~32767 포트에 매핑한다. 외부에서 `<node-ip>:<nodeport>`로 직접 접근 가능. dev나 임시 검증에 충분하다.

`MetalLB` 같은 자체 LB는 LoadBalancer 타입을 흉내 내며, 클러스터 IP 대신 LAN IP를 LB IP로 광고한다. 진짜 LB처럼 동작하지만 별도 컴포넌트 운영이 필요하다.

`Ingress` Controller(예: ingress-nginx)는 Service를 NodePort로 두고 그 위에서 호스트/경로 라우팅을 한다. HTTP/HTTPS 단일 진입점이 되며, dev에서 가장 자주 쓰인다.

### Q8. Cilium BGP와 Calico BGP는 어떤 점에서 다른가?

운영 모델은 비슷하지만 컨트롤 플레인 통합 위치가 다르다. Calico는 BGP 데몬(`bird`)을 노드별로 두고 calico-node Pod가 관리한다. Cilium은 자체 agent가 BGP 기능을 내장해 별도 데몬 없이 운영된다. operationally는 Cilium 쪽이 컴포넌트 수가 적어 단순하다.

기능 측면에서는 두 솔루션 모두 ToR 스위치와의 peering, ECMP 분산, BFD 같은 표준을 지원한다. Cilium은 K8s API와의 통합이 더 가까워 BGP peering을 CR(`CiliumBGPPeeringPolicy`)로 표현하고, ServiceLB를 BGP로 광고하는 흐름이 자연스럽다. Calico는 BGP가 더 오래된 만큼 복잡한 멀티 AS 시나리오에서 검증된 케이스가 많다.

선택은 클러스터 전반의 CNI 선택과 묶인다. CNI를 Cilium으로 가는 결정이 우선이고, BGP는 그 결정을 따라가는 식이 일반적이다.
