# 토폴로지 분산과 중단 정책 점검

> 본 장의 심화 점검 질문입니다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했습니다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검합니다.

### Q1. Topology Spread `maxSkew: 1`이 분산을 항상 보장하는가?

`maxSkew`는 분포의 균형 상한을 표현할 뿐 절대 분산을 약속하지 않습니다. `whenUnsatisfiable: ScheduleAnyway`로 두면 위반 노드도 점수에만 영향을 받고 배치 자체는 진행됩니다. 결과적으로 [3, 0, 0] 같은 편향 분포가 나올 수 있습니다. 균형을 강제하려면 `DoNotSchedule`을 써야 하지만, 이때는 후보 노드가 부족하면 Pod가 Pending에 들어간다는 점을 같이 염두에 둡니다.

또 한 가지 함정은 `minDomains`다. zone이 3개 등록돼 있는데 어떤 시점에 한 zone의 모든 노드가 NotReady가 되면 후보 zone이 2개로 줄어듭니다. `minDomains`를 명시하지 않으면 K8s는 살아있는 zone 수를 기준으로 maxSkew를 계산합니다. 결과적으로 균형은 맞지만 zone 분산이 깨집니다. `minDomains: 3`을 두면 "zone 3개가 보이지 않는 한 분산을 만족 못 한 것으로 간주"합니다.

운영에서는 `whenUnsatisfiable: DoNotSchedule`과 `minDomains` 명시를 함께 씁니다. 단, `replicas`가 maxReplicas까지 늘어났을 때 배치 가능한 zone 후보가 충분한지 검산한 뒤에 도입합니다.

### Q2. podAntiAffinity와 Topology Spread Constraints 중 무엇을 먼저 쓸지 어떻게 정하는가?

기본 선택은 Topology Spread다. 표현이 정확하고 Filter 비용이 podAntiAffinity required보다 낮습니다. 같은 라벨 집합의 분포 균형이 목적이라면 Topology Spread 한 줄로 끝납니다.

podAntiAffinity가 필요한 경우는 두 가지입니다.

1. **다른 라벨 Pod와의 격리**다. "결제 서비스 Pod와 배치 작업 Pod는 같은 노드에 두지 마라"처럼 서로 다른 워크로드를 떼어 놓는 경우 podAntiAffinity가 자연스럽습니다.
2. **선택자 표현이 podAntiAffinity 쪽이 더 자연스러운 경우**다. labelSelector에 `matchExpressions`로 복잡한 조건을 넣어야 한다면 podAntiAffinity 쪽이 표현하기 쉽습니다.

두 도구를 함께 쓰는 경우도 흔합니다. zone 분산은 Topology Spread, 노드 단위 동거 금지는 podAntiAffinity preferred로 둡니다. required 변형 두 개를 동시에 강하게 걸면 배치 후보가 빠르게 사라지므로 한쪽은 preferred로 내리는 편이 안전합니다.

### Q3. minAvailable과 maxUnavailable 중 무엇을 쓰는 게 좋은가?

비율 표현(예: `maxUnavailable: 25%`)이 가능한 쪽을 우선합니다. HPA가 동작하는 워크로드는 Pod 수가 변하므로, 절대값은 작은 replica 시 모든 evict를 막는 함정에 빠지기 쉽습니다. 비율은 replica 수를 따라 자동 조정됩니다.

의미 차이도 있습니다. `minAvailable: 2`는 "2개는 항상 살아 있어야"고, `maxUnavailable: 1`은 "동시에 1개만 종료"다. 작은 replica 환경에서는 두 표현이 같은 Pod 수를 의미해도 운영자의 정신 모델이 다릅니다. 가용성 약속을 "최소 N개 보장"으로 표현하고 싶으면 minAvailable, "한 번에 한 개씩 점검"의 의도를 표현하고 싶으면 maxUnavailable이 자연스럽습니다.

`maxUnavailable: 0`은 모든 evict를 차단하므로 점검이 영원히 안 됩니다. 이 값은 거의 항상 잘못된 설정입니다. 점검 자체를 금지하려면 PDB가 아니라 다른 운영 정책으로 표현합니다.

### Q4. PDB의 selector가 Deployment와 일치하지 않으면 어떻게 알 수 있나?

`kubectl get pdb -o wide`의 `ALLOWED DISRUPTIONS` 컬럼이 `0`이거나 `MIN AVAILABLE/EXPECTED PODS`의 EXPECTED 값이 0이면 selector가 어떤 Pod도 잡지 못한다는 신호입니다. `kubectl describe pdb <name>`의 `Status` 섹션에서 `Current Healthy`와 `Expected Pods`를 확인하면 더 명확합니다.

selector 불일치는 라벨 명명 규약을 팀마다 다르게 쓸 때 자주 발생합니다. Deployment의 라벨이 `app: my-app`인데 PDB의 selector가 `app.kubernetes.io/name: my-app`이면 둘은 다른 라벨입니다. Helm 차트로 배포하는 경우 차트의 기본 라벨 컨벤션과 직접 만든 PDB의 라벨이 어긋나는 사례가 흔합니다.

운영 시점에서는 PDB도 차트에 같이 두는 편이 안전합니다. 차트 안에서 같은 라벨 헬퍼를 공유하면 selector 불일치가 차트 단위로 봉쇄됩니다. 별도 매니페스트로 PDB를 두려면 배포 후 `ALLOWED DISRUPTIONS`를 매번 검증합니다.

### Q5. PriorityClass 값을 너무 높게 잡으면 어떤 일이 생기는가?

시스템 클래스(`system-cluster-critical`=2,000,000,000, `system-node-critical`=2,000,001,000)에 가까운 값을 사용자 워크로드에 부여하면 두 가지 사고가 납니다.

1. **시스템 Pod와의 우선순위 경합**입니다. 사용자 Pod가 시스템 Pod와 비슷한 값이면 자원 부족 시 시스템 Pod도 preempt 후보가 될 수 있습니다. CoreDNS·kube-proxy가 사라지면 클러스터 전체가 흔들립니다.

2. **모든 워크로드가 동일하게 중요해진다**. 100만 단위로 균등하게 우선순위를 부여하면 자원이 부족할 때 어느 Pod도 양보하지 않아 preemption이 자주 발생하고, 서비스 진동이 심해집니다. 우선순위는 5단계 정도로 단순하게 묶고, 같은 단계 안에서는 PDB·Eviction이 다른 정책으로 결정하게 둡니다.

`globalDefault: true`인 PriorityClass를 정의하면 이름 명시가 없는 모든 Pod가 그 값을 갖습니다. 0 또는 음수로 둬서 "기본은 가장 낮음"을 코드로 박아두면 운영 안정성이 높아집니다.

### Q6. node-pressure eviction이 자주 발생하는 노드는 어디부터 점검하는가?

`kubectl describe node <name>`의 `Conditions`에 `MemoryPressure`, `DiskPressure` 같은 시그널이 자주 뜬다면 eviction이 시작됩니다. 첫 점검 항목은 노드의 Allocatable 대비 모든 Pod의 Requests 합계와 실제 사용량입니다. Requests 합계가 Allocatable의 80~90% 수준이면 일시적 사용량 급증에 여유가 없습니다.

다음으로 `kubelet`의 eviction 임계값을 확인합니다. 기본은 `memory.available<100Mi`, `nodefs.available<10%` 같은 보수적 값입니다. 노드 자체에 시스템 데몬(예: kubeadm 자체, container runtime, 로그 시스템)이 많이 설치돼 있다면 `system-reserved`·`kube-reserved`로 충분히 자원을 떼어 두지 않으면 eviction이 일찍 발동합니다.

워크로드 측에서는 BestEffort QoS Pod의 분포를 봅니다. node-pressure eviction은 BestEffort부터 종료하므로, BestEffort Pod가 많은 노드는 eviction에 가장 취약합니다. 중요 워크로드는 Burstable 또는 Guaranteed로 끌어올리고, BestEffort가 의도된 워크로드는 별도 노드 그룹(taint로 격리)으로 옮깁니다.

### Q7. Cluster Autoscaler 축소 시 PDB가 어떻게 작동하는가?

CA의 노드 축소는 evict API를 호출하므로 PDB가 검사됩니다. 축소 대상 노드의 Pod 중 하나라도 evict가 PDB로 막히면 그 노드는 축소되지 않습니다. CA는 일정 시간(`--scale-down-unneeded-time`, 기본 10분) 동안 같은 조건이 유지된 노드만 후보로 보고, 그중 evict가 모두 통과하는 노드만 실제로 제거합니다.

`cluster-autoscaler.kubernetes.io/safe-to-evict: "false"` 어노테이션은 PDB와 별개로 CA가 축소를 시도하지 않게 만듭니다. 로컬 디스크를 쓰는 Stateful Pod, evict 시 데이터를 잃는 워크로드에 자주 답니다. 단, 이 어노테이션이 너무 많으면 CA가 어떤 노드도 축소하지 못해 비용이 떨어지지 않습니다.

운영 관점에서 PDB와 safe-to-evict의 역할을 구분합니다. PDB는 "동시 종료 상한"을, safe-to-evict는 "축소 대상에서 완전 제외"를 표현합니다. 자동화 가능한 워크로드는 PDB로 풀고, 자동화 불가능한 워크로드만 safe-to-evict로 잠급니다.

### Q8. drain이 무한 대기에 들어갔을 때 어떻게 푸는가?

먼저 어느 Pod가 막혔는지 확인합니다. `kubectl get events --field-selector=type=Warning --sort-by=.lastTimestamp` 또는 drain 명령의 stderr에 PDB 위반 메시지가 출력됩니다.

해결 경로는 세 가지입니다.

1. **다른 노드의 같은 라벨 Pod 수를 먼저 늘린다**. `kubectl scale deployment <name> --replicas=<현재+1>`로 임시 추가해 PDB의 minAvailable을 충족시킨 뒤 drain을 재시도합니다.
2. **PDB의 `minAvailable`을 일시 완화**합니다. 점검 윈도우 동안만 값을 낮추고 끝나면 원복합니다.
3. **HPA가 replica를 줄이고 있다면 잠깐 멈춘다**. HPA의 minReplicas를 점검 시작 시점에 한두 단계 올리는 패턴입니다.

`--disable-eviction --force`는 마지막 수단입니다. 이 옵션은 evict API 대신 직접 Pod를 삭제하므로 PDB를 우회합니다. 가용성 약속이 깨질 수 있어 비상시에만 씁니다. 비상이 잦다면 PDB·HPA 설정 자체를 다시 봅니다. 점검 가능성을 만들어 두는 것이 PDB 자체보다 우선입니다.
