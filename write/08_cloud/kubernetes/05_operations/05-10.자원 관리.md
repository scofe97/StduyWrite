# 자원 관리

> Requests는 스케줄러의 입력이고, Limits는 kubelet의 강제 집행입니다. 이 두 값의 관계가 QoS 클래스를 결정하고, QoS 클래스가 장애 시 Pod 퇴거 순서를 결정합니다.



## 학습 목표
> 스케줄링 값과 런타임 제어 값이 어떻게 이어지는지 정리합니다.

이 장에서 확인할 목표는 다음과 같다:

1. Requests와 Limits의 역할 차이와 QoS 클래스 결정 규칙을 설명할 수 있습니다.
2. CPU throttling과 OOMKill의 차이와 감지 방법을 구분할 수 있습니다.
3. `LimitRange`와 `ResourceQuota`를 조합해 네임스페이스를 격리하는 방법을 설명할 수 있습니다.
4. `Capacity`와 `Allocatable`의 차이와 스케줄링 영향을 이해할 수 있습니다.
5. 워크로드 유형별 적합한 QoS 전략을 판단할 수 있습니다.
6. `emptyDir`, Pod-level resources 같은 최신 자원 관리 포인트를 설명할 수 있습니다.



## 1. Requests와 Limits
> 같은 숫자가 스케줄링과 실행 제어에서 다르게 해석된다는 점을 먼저 잡습니다.

### 1.1 역할 분리

`Requests`는 스케줄러가 노드 선택에 사용하는 값입니다. 노드의 Allocatable에서 현재 Pod들의 Requests 합계를 빼고 남은 여유 자원이 새 Pod의 Requests를 수용할 수 있을 때 배치됩니다. 실제 사용량과 무관하게 예약량입니다.

`Limits`는 kubelet이 강제하는 상한선입니다. CPU는 cgroup CFS throttle로 제한하고, 메모리는 cgroup 한도를 초과하면 OOM Killer가 컨테이너를 종료합니다.

```yaml
resources:
  requests:
    cpu: "200m"       # 스케줄러: 이 Pod에 200m CPU 예약
    memory: "256Mi"   # 스케줄러: 이 Pod에 256Mi 메모리 예약
  limits:
    cpu: "500m"       # kubelet: CPU 500m 초과 시 throttle
    memory: "512Mi"   # kubelet: 메모리 512Mi 초과 시 OOMKill
```

### 1.2 Limits만 설정하면

Limits만 설정하고 Requests를 생략하면 Kubernetes가 자동으로 `Requests = Limits`로 설정합니다. 개발자가 "최대 500m까지만 사용하게 하자"는 의도로 Limits만 설정했어도, 실제로는 500m을 항상 예약하는 Guaranteed Pod가 됩니다. 의도치 않게 노드 활용률이 낮아집니다.

공식 문서에서 최근 강조되는 포인트는 Pod 단위 자원 지정입니다. 기능 게이트와 버전에 따라 `spec.resources.requests` / `spec.resources.limits`처럼 Pod 전체 예산을 선언할 수 있습니다. 다만 대부분의 실무 문서와 예제는 아직 컨테이너 단위 설정을 기본으로 삼고 있으므로, 팀 표준을 정하지 않으면 같은 클러스터 안에서 두 방식이 섞여 가독성이 떨어질 수 있습니다.



## 2. QoS 클래스
> 자원 요청 패턴이 축출 우선순위로 어떻게 이어지는지 설명합니다.

세 가지 QoS 클래스는 설정 조합에 의해 자동으로 결정됩니다.

| QoS 클래스 | 조건 | 퇴거 우선순위 |
|-----------|------|-------------|
| Guaranteed | 모든 컨테이너의 CPU/메모리 Requests = Limits | 가장 마지막 |
| Burstable | Requests < Limits, 또는 일부만 설정 | 중간 |
| BestEffort | Requests/Limits 미설정 | 가장 먼저 |

노드에 메모리 압박이 생기면 kubelet은 BestEffort → Burstable → Guaranteed 순으로 퇴거합니다. Burstable 간에는 Requests 대비 실제 사용량 비율이 높은 Pod부터 먼저 퇴거합니다. Requests를 크게 초과해 사용하는 Pod가 더 먼저 제거됩니다.

### 2.1 워크로드별 QoS 전략

Guaranteed가 적합한 워크로드는 데이터베이스, 캐시, 메시지 브로커처럼 성능 일관성이 핵심이고 퇴거 시 데이터 유실 위험이 있는 Stateful 워크로드입니다. 항상 동일한 리소스가 보장되므로 응답 시간이 안정적입니다.

Burstable이 적합한 워크로드는 Stateless API 서버, 웹 프론트엔드, 배치 작업입니다. 트래픽에 따라 사용량이 변하고, 퇴거 시 재스케줄링으로 복구 가능합니다. 노드 오버커밋으로 비용 효율을 높일 수 있습니다.

하이브리드 전략도 실용적입니다. CPU는 Burstable(Limits를 Requests보다 높게)로 설정해 throttling 완화 여지를 두고, 메모리는 Guaranteed(Requests = Limits)로 설정해 OOMKill을 방지합니다.



## 3. CPU Throttling
> CPU 제한이 단순 보호 장치가 아니라 성능 병목이 될 수 있음을 다룹니다.

### 3.1 발생 메커니즘

CPU throttling은 CFS(Completely Fair Scheduler) bandwidth control로 발생합니다. 기본 CFS period는 100ms이며, CPU Limits가 `200m`이면 100ms 주기당 20ms의 CPU 시간만 사용할 수 있습니다. 20ms를 소진하면 남은 80ms 동안 CPU를 사용하지 못하고 대기합니다.

throttling의 핵심 영향은 tail latency다. 평균 응답 시간보다 p95, p99가 크게 증가합니다. 대부분의 요청은 정상이지만 throttle 구간에 걸린 요청은 최대 80ms를 추가 대기합니다. Java/Go 같은 GC 언어에서는 GC가 CPU를 집중 사용하는 구간에 throttling이 겹치면 GC 시간이 수 배로 늘어납니다.

### 3.2 감지와 대응

```promql
# throttling 비율 (25% 초과 시 Limits 증가 고려)
rate(container_cpu_cfs_throttled_periods_total[5m])
  / rate(container_cpu_cfs_periods_total[5m])
```

이 `container_*` 메트릭들의 수집 원천은 **cAdvisor(Container Advisor)** 입니다. cAdvisor는 각 워커 노드의 kubelet에 내장된 에이전트로, 컨테이너 런타임(containerd 등)과 통신하며 컨테이너별 CPU·메모리·네트워크·디스크 사용량을 실시간으로 기록합니다. 수집 경로는 **cAdvisor → kubelet 요약 → Metrics Server/Prometheus 수집**입니다. `kubectl top pod`이 보여주는 값도, HPA가 스케일 판단에 쓰는 값도 이 경로를 탑니다. 컨테이너 단위로 쪼개 보여주므로, 한 Pod에 메인 앱과 사이드카가 함께 있을 때 어느 컨테이너가 throttle되는지 분리해 볼 수 있습니다.

대응 전략은 세 가지입니다. CPU Limits를 올리는 것이 가장 직접적입니다. CPU Limits를 제거하고 Requests만 설정하는 방식도 있습니다. 스케줄러가 Requests 기준으로 적절히 배치하되 상한 없이 여유 CPU를 활용합니다. Noisy Neighbor 위험이 있으므로 노드 워크로드 구성을 잘 관리해야 합니다.

메모리 기반 `emptyDir`도 함께 봐야 합니다. 공식 문서 기준으로 메모리를 사용하는 `emptyDir`는 Pod 메모리 한도와 함께 터질 수 있습니다. 애플리케이션 메모리 사용량만 계산하고 `/tmp`나 캐시 디렉터리 사용량을 빼먹으면, 원인을 찾기 어려운 OOM이 발생합니다.



## 4. LimitRange와 ResourceQuota
> 팀 단위 자원 거버넌스를 어떻게 강제할지 정책 관점에서 봅니다.

### 4.1 역할 차이

두 리소스는 제어 범위가 다릅니다. `LimitRange`는 개별 Pod/Container 수준을 제어합니다. Requests/Limits 기본값 주입, 단일 Pod의 최소/최대 범위 강제, Limits/Requests 비율 제한이 주 용도입니다.

`ResourceQuota`는 네임스페이스 전체 총량을 제한합니다. 네임스페이스 내 모든 Pod의 Requests/Limits 합계, 오브젝트 수, PVC 스토리지 크기를 제한합니다.

### 4.2 조합이 필요한 이유

ResourceQuota가 활성화된 네임스페이스에서는 모든 Pod에 Requests/Limits가 필수입니다. 설정 없이 배포하면 거부됩니다. LimitRange로 기본값을 설정해 두면 Requests/Limits를 명시하지 않아도 자동으로 채워져 개발자 편의를 높입니다.

적용 순서: LimitRange 기본값 주입 → ResourceQuota 총량 검사 → Pod 생성. 두 가지를 모두 통과해야 Pod가 생성됩니다.

```yaml
# LimitRange: 기본값 주입
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: team-a
spec:
  limits:
    - type: Container
      default:
        cpu: "200m"
        memory: "256Mi"
      defaultRequest:
        cpu: "100m"
        memory: "128Mi"
      max:
        cpu: "2"
        memory: "4Gi"
---
# ResourceQuota: 네임스페이스 총량
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-a-quota
  namespace: team-a
spec:
  hard:
    requests.cpu: "8"
    requests.memory: "16Gi"
    limits.cpu: "16"
    limits.memory: "32Gi"
    pods: "50"
```



## 5. Capacity와 Allocatable
> 노드 총량과 실제 스케줄 가능한 양을 구분해서 읽는 법을 정리합니다.

`kubectl describe node`에서 두 값이 나옵니다. `Capacity`는 노드의 물리 총 리소스입니다. `Allocatable`은 Pod에 실제로 할당 가능한 리소스로, Capacity에서 kube-reserved, system-reserved, eviction 임계값을 뺀 값입니다.

스케줄러는 Capacity가 아닌 Allocatable 기준으로 Pod를 배치합니다. 16GB 노드에서 kube-reserved 1GB, system-reserved 1GB, eviction 100Mi를 빼면 실제 배치 가능한 메모리는 약 13.9GB다. "16GB 노드에 2GB Pod 8개를 배치할 수 있다"고 계산하면 실제로는 6~7개만 가능할 수 있습니다.

Allocated resources의 Limits 합계가 100%를 초과하는 경우가 있습니다. 오버커밋이며, 모든 Pod가 동시에 Limits까지 사용하면 리소스가 부족해집니다. Requests 합계는 스케줄러가 100%를 넘지 않도록 방지하지만 Limits 합계는 제한이 없습니다.



## 6. 다음 단계
> 자원 관리 값을 고정한 뒤 동적 스케일링 장으로 자연스럽게 이어 갑니다.

Requests/Limits가 단단해야 그 위에 얹는 오토스케일링이 의미를 가집니다. HPA의 사용률 계산은 Requests를 분모로 하고, VPA는 Requests 자체를 조정합니다. 다음 장 [오토스케일링](05-11.%EC%98%A4%ED%86%A0%EC%8A%A4%EC%BC%80%EC%9D%BC%EB%A7%81.md)에서는 이 값들을 신호로 받아 Pod 수와 노드 수를 자동으로 조정하는 세 축(HPA·VPA·Cluster Autoscaler)과 이벤트 기반 KEDA를 정리합니다.
