# 모니터링과 트러블슈팅

> Kubernetes 클러스터를 운영한다는 것은 "무엇이 잘못됐는가"를 빠르게 찾아내는 능력을 갖추는 것이다. 관찰 가능성(observability)의 세 기둥 — 메트릭, 로그, 트레이스 — 과 kubectl 기반 디버깅 루틴이 이 능력의 토대다.



## 학습 목표
> 관측 도구와 디버깅 루틴을 한 흐름으로 익히는 장이다.

이 장에서 확인할 목표는 다음과 같다:

1. 메트릭·로그·트레이스의 역할 차이와 우선 조회 대상을 구분할 수 있다.
2. `kube-prometheus-stack`이 어떤 구성 요소를 묶어 제공하는지 설명할 수 있다.
3. `ServiceMonitor`로 커스텀 메트릭 수집 대상을 연결하는 방식을 이해할 수 있다.
4. `CrashLoopBackOff`, `OOMKilled`, `Pending`, `ImagePullBackOff` 같은 전형적 장애 패턴을 진단 루틴에 맞춰 해석할 수 있다.
5. AlertManager의 라우팅·그룹핑·억제 개념을 운영 알림 설계와 연결할 수 있다.



## 1. 관찰 가능성의 세 기둥
> 메트릭, 로그, 트레이스가 각각 어떤 질문에 답하는지 먼저 구분한다.

### 1.1 메트릭·로그·트레이스

세 가지는 서로 다른 질문에 답한다. 메트릭은 "무엇이 얼마나 나쁜가"를 수치로 나타낸다. CPU 사용률, HTTP 요청 수, 에러율 같은 집계 데이터다. 시간 경과에 따른 추세를 파악하고 임계값 기반 알림을 걸기에 적합하다. 로그는 "무슨 일이 일어났는가"를 이벤트 스트림으로 기록한다. 메트릭이 이상 징후를 포착하면 로그로 구체적인 에러 메시지를 확인한다. 트레이스는 "어떤 경로를 거쳐 느려졌는가"를 서비스 간 요청 흐름으로 시각화한다. 마이크로서비스 환경에서 어느 서비스가 병목인지 찾을 때 쓴다.

세 가지의 비용이 다르다. 메트릭은 집계 데이터라 저장 비용이 낮다. 로그는 양이 많아 비용이 높고, 적절한 레벨(INFO/WARN/ERROR) 관리가 중요하다. 트레이스는 전체 요청을 기록하면 오버헤드가 크므로 샘플링을 조정한다.

### 1.2 Prometheus의 Pull 모델

Prometheus는 Pull 모델을 쓴다. Prometheus가 주기적으로 각 대상의 `/metrics` 엔드포인트를 호출해 데이터를 수집한다. Push 모델(대상이 Prometheus에 직접 전송)과 달리, 대상이 죽으면 Prometheus가 즉시 알 수 있고 수집 실패 자체가 알림 대상이 된다. 단기 배치 작업처럼 존재 기간이 짧아 폴링이 불가한 경우에는 Pushgateway를 중간에 두고 Push 방식을 보완한다.

여기서 헷갈리기 쉬운 점이 있다. `kubectl top`이 보여주는 CPU·메모리 사용량은 보통 `Metrics Server`가 제공하는 간이 메트릭이고, Prometheus는 별도의 모니터링 스택이다. 즉 HPA나 운영자가 빠르게 사용량을 보는 경로와, 장기 저장·알림·대시보드용 관측 스택은 보통 분리돼 있다.



## 2. kube-prometheus-stack
> Kubernetes에서 가장 보편적인 모니터링 조합을 한 묶음으로 설명한다.

### 2.1 구성 요소

kube-prometheus-stack은 Helm 차트 하나로 Kubernetes 모니터링 스택 전체를 배포한다. 포함되는 주요 구성 요소는 네 가지다.

| 구성 요소 | 역할 |
|-----------|------|
| Prometheus | 메트릭 수집·저장·쿼리 |
| Grafana | 메트릭 시각화·대시보드 |
| AlertManager | 알림 라우팅·그룹핑·억제 |
| kube-state-metrics | Kubernetes 오브젝트 상태 메트릭 노출 |

node-exporter는 각 노드에 DaemonSet으로 배포되어 CPU, 메모리, 디스크, 네트워크 같은 노드 수준 메트릭을 수집한다.

### 2.2 ServiceMonitor로 커스텀 메트릭 수집

애플리케이션이 `/metrics` 엔드포인트를 노출하면 `ServiceMonitor` CR을 작성해 Prometheus가 수집하도록 지시한다.

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: my-app
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: my-app
  endpoints:
    - port: http
      path: /metrics
      interval: 30s
  namespaceSelector:
    matchNames:
      - production
```

Prometheus Operator가 ServiceMonitor를 감시하다가 새 리소스가 생기면 Prometheus 설정을 자동으로 갱신한다. 직접 `prometheus.yml`을 수정할 필요가 없어 운영 부담이 크게 줄어든다.

실무에서 자주 놓치는 부분은 레이블 연결이다. `ServiceMonitor`만 만든다고 바로 수집되지는 않는다. Prometheus 인스턴스의 `serviceMonitorSelector`와 `ServiceMonitor.metadata.labels`, 그리고 실제 대상 `Service`의 라벨이 모두 맞아야 수집이 시작된다.

### 2.3 PromQL 핵심 패턴

```promql
# HTTP 5xx 에러율 (5분 비율)
rate(http_requests_total{status=~"5.."}[5m])
  / rate(http_requests_total[5m])

# 99번째 백분위수 응답 시간
histogram_quantile(0.99,
  rate(http_request_duration_seconds_bucket[5m]))

# 노드 메모리 사용률
1 - (node_memory_MemAvailable_bytes
     / node_memory_MemTotal_bytes)
```

`rate()`는 카운터의 초당 증가율을 계산한다. 카운터는 재시작 시 0으로 리셋되므로 직접 비교보다 `rate()`를 쓰는 것이 안전하다. `histogram_quantile()`은 히스토그램 메트릭에서 백분위수를 추출한다. p99 응답 시간은 평균과 달리 꼬리 지연을 잡아낸다.



## 3. 로그 수집: Loki 스택
> 로그를 비용 효율적으로 모으고 메트릭과 연결하는 방식을 정리한다.

Grafana Loki는 Prometheus의 로그 버전이다. 로그의 전체 내용을 인덱싱하지 않고 레이블(네임스페이스, Pod 이름, 컨테이너 이름)만 인덱싱해 저장 비용을 낮춘다. Promtail이 각 노드에서 로그를 수집하고 Loki로 전송하며, Grafana에서 Prometheus 메트릭과 로그를 같은 화면에 연결해 볼 수 있다. "에러율이 올라가는 시점의 로그는 무엇인가"를 클릭 한 번으로 이동할 수 있다.



## 4. AlertManager
> 알림을 많이 보내는 것이 아니라 잘 묶고 잘 막는 것이 핵심임을 설명한다.

### 4.1 라우팅·그룹핑·억제·침묵

AlertManager는 알림의 흐름을 네 가지 메커니즘으로 제어한다.

라우팅은 알림을 적절한 채널로 보낸다. 심각도(critical)나 팀 레이블에 따라 PagerDuty, Slack, 이메일로 분기한다. 그룹핑은 같은 클러스터에서 동시에 발생한 수십 개의 알림을 하나의 메시지로 묶는다. 노드 장애 시 그 노드의 모든 Pod 알림이 "N개 Pod 장애" 한 건으로 도착한다. 억제는 상위 원인 알림이 발생하면 하위 결과 알림을 자동으로 차단한다. `NodeDown` 알림이 오면 그 노드의 `PodCrashLooping` 알림을 억제하는 방식이다. 침묵은 특정 기간 동안 알림을 수동으로 끈다. 정기 점검 시간에 활용한다.



## 5. 트러블슈팅 루틴
> 장애가 났을 때 무엇부터 확인할지 순서를 고정한다.

### 5.1 kubectl 기반 디버깅 흐름

장애 상황에서 체계적인 순서가 중요하다.

```
이벤트 → 로그 → 내부 접속 → 리소스
```

```bash
# 1단계: 이벤트 확인
kubectl describe pod <pod-name> -n <namespace>
# Events 섹션에 스케줄링 실패, 이미지 풀 에러, 프로브 실패 원인이 기록됨

# 2단계: 로그 확인
kubectl logs <pod-name> -n <namespace>
kubectl logs <pod-name> -n <namespace> --previous   # 이전 컨테이너 로그

# 3단계: 내부 접속
kubectl exec -it <pod-name> -n <namespace> -- /bin/sh
# 컨테이너 안에서 curl, nslookup, env 등으로 직접 확인

# 4단계: 리소스 확인
kubectl top pod -n <namespace>
kubectl top node
```

### 5.2 주요 장애 패턴

`CrashLoopBackOff`는 컨테이너가 시작했다가 즉시 종료되는 패턴이다. `--previous` 플래그로 이전 컨테이너 로그를 먼저 확인한다. 애플리케이션 시작 에러, 설정 오류, 의존 서비스 미준비가 주요 원인이다.

`OOMKilled`는 메모리 Limits를 초과해 커널이 컨테이너를 강제 종료한 상태다. `kubectl describe pod`의 `Last State`에 `OOMKilled` reason이 표시된다. Limits를 올리거나 메모리 누수를 찾아야 한다.

`Pending`은 스케줄링되지 못한 상태다. `describe pod`의 Events에 "Insufficient cpu", "0/3 nodes are available" 같은 메시지가 나온다. 노드 리소스 부족, NodeSelector/Affinity 불일치, PVC 미바인딩이 원인인 경우가 많다.

`ImagePullBackOff`는 이미지를 내려받지 못한 상태다. 이미지 이름 오타, 태그 미존재, Private 레지스트리 인증 실패를 순서대로 확인한다.

### 5.3 kubectl debug

ephemeral container를 통해 실행 중인 Pod에 디버깅 도구를 주입할 수 있다.

```bash
# distroless 같이 셸이 없는 컨테이너 디버깅
kubectl debug -it <pod-name> --image=busybox --target=<container-name>
```

Pod 내부뿐 아니라 노드 자체를 진단할 때도 `kubectl debug node/<node-name>` 패턴을 사용할 수 있다. 공식 문서 기준으로 이 방식은 호스트 네임스페이스와 파일시스템에 접근하는 디버깅 Pod를 만들어, kubelet 로그나 디스크 상태처럼 Pod 바깥 문제를 확인할 때 유용하다.
