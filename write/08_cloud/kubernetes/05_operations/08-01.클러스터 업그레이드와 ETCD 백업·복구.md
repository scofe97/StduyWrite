# 클러스터 업그레이드와 ETCD 백업·복구

> 운영에서 가장 위험한 작업은 업그레이드와 복구다. kubeadm 업그레이드는 버전 스큐 정책과 순서를 지키는 작업이고, etcd 복구는 스냅샷과 멤버십 재구성의 문제다.



## 학습 목표
> 평소 잘 안 건드리는 작업일수록 절차 중심으로 이해한다.

이 장에서 확인할 목표는 다음과 같다:

1. kubeadm 기반 클러스터 업그레이드의 순서를 설명할 수 있다.
2. minor 버전 스킵이 왜 위험한지와 버전 스큐 정책을 설명할 수 있다.
3. etcd 스냅샷 백업과 복구의 기본 흐름을 설명할 수 있다.
4. 업그레이드 전 점검 항목과 복구 대비 항목을 분리해서 정리할 수 있다.



## 1. kubeadm 업그레이드 원칙
> 업그레이드는 "한 번에 최신으로"가 아니라 지원되는 순서로 밟아 가는 작업이다.

공식 문서 기준으로 kubeadm 클러스터는 minor 버전을 건너뛰는 업그레이드를 지원하지 않는다. `1.33 -> 1.35`처럼 한 번에 올리는 대신 `1.33 -> 1.34 -> 1.35` 순서를 밟아야 한다. 그리고 control plane 노드는 한 대씩 순차 업그레이드하는 것이 원칙이다.

기본 흐름은 다음과 같다.

1. 현재 상태 확인
2. `kubeadm upgrade plan`
3. control plane 노드 순차 업그레이드
4. kubelet / kubectl 정리
5. worker 노드 drain 후 순차 업그레이드

업그레이드는 단순 패키지 교체가 아니다. static Pod 매니페스트, addon(CoreDNS, kube-proxy), 인증서 자동 갱신 여부까지 함께 엮여 있다.



## 2. etcd 스냅샷과 복구
> etcd는 Kubernetes의 상태 저장소이므로, 최악의 경우를 대비한 백업이 반드시 필요하다.

etcd 복구의 기본은 "정상 시점에 스냅샷을 남겨 두는 것"이다.

```bash
ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  snapshot save backup.db
```

최근 etcd 문서 기준으로 복구는 `etcdutl snapshot restore` 흐름을 중심으로 이해하는 편이 안전하다. 복구는 단순 파일 복사가 아니라 새로운 클러스터 ID와 멤버 구성을 기준으로 다시 시작하는 작업이다.

운영에서 중요한 원칙은 두 가지다.

- 스냅샷은 주기적으로, 그리고 복구 절차까지 검증해야 한다.
- 복구는 기존 멤버가 살아 있는 상태에서 덮어쓰는 작업이 아니라, 새 멤버십 기준으로 클러스터를 다시 여는 작업에 가깝다.



## 3. 실무 체크리스트
> 업그레이드와 복구는 기술 지식보다 체크리스트 준수가 더 중요하다.

- 업그레이드 전: `kubeadm upgrade plan`, 노드 Ready 상태, 백업 확보
- control plane: 한 대씩 업그레이드
- worker: `cordon` / `drain` 후 업그레이드
- etcd: leader 기준 스냅샷 확보, 복구 절차 리허설
- 인증서: kubeadm 관리 인증서 자동 갱신 범위 확인
