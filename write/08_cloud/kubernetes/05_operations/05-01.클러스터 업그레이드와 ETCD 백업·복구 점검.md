# 클러스터 업그레이드와 ETCD 백업·복구 점검

> 본 장의 심화 점검 질문입니다. 업그레이드와 복구 작업에서 순서, 버전 스큐, 스냅샷 운용 포인트를 Q&A 형태로 정리했습니다.



## 점검 질문
> 드물지만 위험한 작업일수록 체크리스트 관점으로 다시 점검합니다.

### Q1. kubeadm 업그레이드에서 가장 먼저 무엇을 확인해야 하는가?

`kubeadm upgrade plan`으로 현재 클러스터가 어떤 버전으로 올라갈 수 있는지, 지원되지 않는 스킵 업그레이드가 아닌지 먼저 봅니다. 업그레이드 자체보다 "지원되는 경로인지 확인하는 단계"가 더 중요합니다.

### Q2. control plane을 왜 한 대씩 올려야 하는가?

공식 문서 기준으로 control plane은 순차 업그레이드가 원칙입니다. 한 번에 여러 대를 건드리면 API 서버, addon 갱신, 인증서, static Pod 재기동이 겹치며 장애 반경이 커집니다.

### Q3. etcd 스냅샷은 어디서 떠야 하는가?

리더 기준 스냅샷을 선호하는 편이 안전합니다. 그리고 스냅샷을 남기는 것만으로 끝내지 말고, 복구 리허설과 보관 정책까지 함께 있어야 합니다. "백업 파일은 있는데 실제로 복구한 적은 없다"가 가장 위험합니다.

### Q4. etcd 복구를 기존 데이터 디렉터리 위에 바로 덮어써도 되는가?

권장되지 않습니다. 복구는 새 클러스터 ID와 멤버십 기준으로 다시 여는 작업에 가깝습니다. 살아 있는 기존 멤버와 섞이면 더 큰 혼란을 만듭니다.

### Q5. etcd 는 여러 노드에 복제하면서 어떻게 항상 같은 값을 보장하는가?

Raft 합의 알고리즘 덕분입니다. 리더가 쓰기 로그를 받아 팔로워에게 복제하고, **과반수(Quorum)가 저장을 확인해야만 Commit** 합니다. 과반수 동의를 얻은 값만 확정되므로, 어느 노드에서 읽어도 최신 값이 돌아오는 강한 일관성이 성립합니다.

### Q6. etcd 를 왜 3·5대 같은 홀수로 구성하는가?

과반수 판정을 명확히 하고 장애 허용을 극대화하기 위해서입니다. Raft 는 리더 선출·쓰기 확정 모두 과반수 동의를 요구하므로, 네트워크 단절로 클러스터가 쪼개져도(Split-brain) 과반수를 넘지 못하는 쪽은 리더를 뽑지도 쓰기를 하지도 못해 데이터가 꼬이지 않습니다. 홀수로 두면 3대 중 1대·5대 중 2대 장애까지 견디면서 과반수가 유지됩니다. 짝수(예: 4대)는 과반수 기준이 3으로 올라가 오히려 장애 허용이 3대 구성과 같으면서 비용만 늘어납니다.
