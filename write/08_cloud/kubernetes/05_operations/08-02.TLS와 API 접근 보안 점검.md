# TLS와 API 접근 보안 점검

> 본 장의 심화 점검 질문이다. kubeconfig, 인증서, API 서버 신뢰 경계를 운영 질문으로 다시 정리했다.



## 점검 질문
> RBAC 전에 인증과 신뢰 체인을 먼저 점검한다.

### Q1. kubeconfig를 왜 신뢰된 출처에서만 받아야 하는가?

공식 문서에도 경고가 있다. kubeconfig는 단순 텍스트 설정처럼 보여도 인증 실행 경로를 포함할 수 있어서, 신뢰할 수 없는 파일을 그대로 쓰면 악성 동작이나 정보 노출로 이어질 수 있다. 셸 스크립트처럼 조심해서 다루는 편이 맞다.

### Q2. TLS와 RBAC은 무엇이 다른가?

TLS는 누구를 신뢰하고 누구로 인증할지를 다루고, RBAC은 인증이 끝난 뒤 무엇을 할 수 있는지를 다룬다. 접속 실패와 권한 실패를 같은 문제로 보면 진단이 오래 걸린다.

### Q3. 인증서 만료는 언제 드러나는가?

평소에는 잘 안 보이지만, API 접속 실패, kubelet 인증 실패, kubeadm 업그레이드 시점에 갑자기 드러난다. 그래서 인증서 관리는 보안 장보다 운영 장과 더 가까운 주제이기도 하다.

### Q4. 서버 인증서와 클라이언트 인증서는 어떻게 다른가?

서버 인증서는 *"이 호스트가 진짜 그 호스트인가"*를 증명한다. API 서버, etcd, kubelet이 각자 서버 인증서를 들고 있으며 SAN(Subject Alternative Name)에 호스트명·IP가 박혀 있다. 클라이언트가 접속할 때 이 인증서의 CN/SAN을 확인하고 신뢰 CA로 서명된 것인지 검사한다.

클라이언트 인증서는 *"내가 누구인가"*를 증명한다. kubeconfig에 들어 있는 `client-certificate`/`client-key`가 그것이며, CN이 사용자명, O(Organization)가 그룹이 된다. API 서버는 이 CN/O를 RBAC 결정의 입력으로 쓴다. 같은 PKI 안에서 발급되더라도 *역할이 정반대*다 — 한쪽은 "내 신원을 상대에게 보여 주고", 한쪽은 "상대가 진짜 그 서버인지 검증한다". 운영 사고는 둘을 같은 의미로 본 자리에서 자주 난다.
