# RBAC과 보안 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> RBAC과 보안 경계를 운영 질문으로 다시 점검한다.

### Q1. kubeconfig의 context와 RBAC은 무엇이 다른가?

`context`는 `cluster`, `user`, `namespace` 조합이다. 즉 "어느 클러스터에 어떤 자격으로 접속할지"를 고르는 설정이다. 반면 RBAC은 그 자격으로 접속한 뒤 "무엇을 할 수 있는지"를 결정한다.

```bash
# 현재 context 확인
kubectl config current-context
kubectl config get-contexts
kubectl config view --minify

# 현재 context가 어떤 사용자 속성으로 인증되는지 확인
kubectl auth whoami

# 실제 권한 확인
kubectl auth can-i get pods -n production
kubectl auth can-i create deployments -n production
```

심화 질문. `kubectl config use-context prod-admin`으로 바꿨는데 권한 부족이 발생하면 무엇을 먼저 봐야 하는가? 먼저 `kubectl auth whoami`로 실제 인증 주체를 확인한다. kubeconfig에 기대한 인증서나 exec plugin이 적용되지 않았을 수 있다. 그다음 `kubectl auth can-i ...`로 RBAC를 점검한다. 즉 context 문제인지, RBAC 문제인지 먼저 분리해야 한다.

### Q2. Role vs ClusterRole 적용 범위

`Role`은 특정 네임스페이스 안에서만 유효하다. `ClusterRole`은 클러스터 전체에서 유효하며 네임스페이스가 없는 리소스(Node, PersistentVolume, Namespace)도 대상으로 삼을 수 있다.

```bash
# ClusterRole 조회
kubectl get clusterrole | grep -v system:

# 특정 ClusterRole 내용
kubectl describe clusterrole view

# 네임스페이스 범위로 바인딩 (ClusterRole → RoleBinding)
kubectl create rolebinding dev-view \
  --clusterrole=view \
  --serviceaccount=dev:default \
  --namespace=dev
```

ClusterRole을 RoleBinding으로 연결하면 그 역할은 해당 네임스페이스에서만 유효하다. ClusterRole을 여러 네임스페이스에서 재사용하면서 범위는 제한하는 표준 패턴이다.

심화 질문. ClusterRole의 `aggregationRule`은 무엇인가? 여러 ClusterRole을 레이블 셀렉터로 자동 병합하는 기능이다. `admin` ClusterRole에 `rbac.authorization.k8s.io/aggregate-to-admin: "true"` 레이블이 붙은 ClusterRole의 권한이 자동으로 포함된다. CRD 설치 시 새 리소스를 기존 역할에 자동 추가할 때 활용한다.

### Q3. default ServiceAccount의 위험

모든 Pod는 지정하지 않으면 같은 네임스페이스의 `default` ServiceAccount를 사용한다. 이 계정에는 권한이 없지만 Kubernetes API 접근용 토큰이 자동으로 마운트된다.

```bash
# default SA 토큰 마운트 확인
kubectl exec -it <pod> -- ls /var/run/secrets/kubernetes.io/serviceaccount/

# 토큰으로 API 호출 가능 여부 확인
kubectl exec -it <pod> -- \
  curl -sk -H "Authorization: Bearer $(cat /var/run/secrets/kubernetes.io/serviceaccount/token)" \
  https://kubernetes.default.svc/api/v1/namespaces/default/pods
```

토큰 마운트 차단이 권장 조치다.

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
  namespace: production
automountServiceAccountToken: false
```

심화 질문. 실수로 `default` ServiceAccount에 ClusterRoleBinding을 연결하면 어떻게 되는가? 클러스터의 모든 네임스페이스의 `default` SA를 사용하는 Pod가 해당 권한을 얻는다. 실수 하나로 클러스터 전체 Pod가 과도한 권한을 갖게 된다. 이를 감지하려면 `kubectl get clusterrolebinding -o yaml | grep "default"`로 확인한다.

### Q4. NetworkPolicy default-deny 설정

NetworkPolicy가 없으면 모든 Pod 간 통신이 허용된다. default-deny를 먼저 적용하고 필요한 트래픽만 명시적으로 허용하는 방식이 권장된다.

```bash
kubectl create namespace ch17-smoke

# default-deny 적용
kubectl apply -n ch17-smoke -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
EOF

# 특정 트래픽만 허용
kubectl apply -n ch17-smoke -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
spec:
  podSelector: {}
  policyTypes:
    - Egress
  egress:
    - ports:
        - port: 53
          protocol: UDP
        - port: 53
          protocol: TCP
EOF

kubectl delete namespace ch17-smoke
```

주의: Flannel은 NetworkPolicy를 지원하지 않는다. 개인 GCP 클러스터가 Flannel CNI를 사용하므로 NetworkPolicy를 적용해도 실제로 차단되지 않는다. Calico나 Cilium으로 CNI를 교체해야 동작한다.

심화 질문. NetworkPolicy로 특정 네임스페이스의 트래픽만 허용하려면? `namespaceSelector`로 네임스페이스 레이블을 지정한다. 네임스페이스에 레이블이 없으면 매칭이 안 되므로 `kubectl label namespace monitoring name=monitoring`으로 레이블을 먼저 추가한다.

### Q5. Secret의 Base64 한계와 보완

```bash
# Secret 생성과 내용 확인
kubectl create secret generic db-creds \
  --from-literal=password=mysecretpassword \
  -n default

# Base64로 저장됨 (암호화 아님)
kubectl get secret db-creds -o jsonpath='{.data.password}' | base64 -d
```

Sealed Secrets 사용 예시다.

```bash
# kubeseal 설치 후
echo -n "mysecretpassword" | kubectl create secret generic db-creds \
  --dry-run=client \
  --from-file=password=/dev/stdin \
  -o yaml | kubeseal > sealed-db-creds.yaml

# SealedSecret은 Git에 안전하게 커밋 가능
git add sealed-db-creds.yaml
```

심화 질문. etcd 암호화(Encryption at Rest)를 활성화하면 Git 커밋 문제가 해결되는가? 아니다. etcd 암호화는 etcd 파일에 직접 접근하는 공격에 대한 방어층이다. `kubectl get secret -o yaml`로 Secret을 읽으면 여전히 Base64로 보인다. Git 커밋 문제는 Sealed Secrets나 External Secrets Operator로 해결해야 한다.

### Q6. Pod Security Standards 적용

```bash
# 네임스페이스에 PSS 레이블 추가
kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/warn=restricted

# 위반 Pod 생성 시도 (root로 실행)
kubectl run root-pod \
  --image=nginx \
  -n production \
  --overrides='{"spec":{"securityContext":{"runAsUser":0}}}'
# Error: pods "root-pod" is forbidden: violates PodSecurity "restricted:latest"

# 준수하는 Pod 스펙
kubectl apply -n production -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: nginx:1.25
      securityContext:
        allowPrivilegeEscalation: false
        capabilities:
          drop: ["ALL"]
EOF
```

심화 질문. PSS `restricted` 레벨이 기존 워크로드를 모두 차단하면 어떻게 단계적으로 적용하는가? `enforce` 대신 `warn`과 `audit`만 먼저 설정한다. 경고만 표시되므로 기존 워크로드는 계속 동작하면서 위반 사항을 확인할 수 있다. 위반 Pod를 모두 수정한 뒤 `enforce`로 전환한다.

### Q7. RBAC 설계 실수 패턴

```bash
# 현재 서비스 계정의 권한 확인
kubectl auth can-i list pods --as=system:serviceaccount:production:my-app

# 특정 리소스에 대한 접근 가능자 확인 (kube-who-can 플러그인)
kubectl who-can create deployments -n production

# RBAC 설정 감사: 와일드카드 사용 탐지
kubectl get clusterrolebindings -o json | \
  jq '.items[] | select(.roleRef.name | contains("cluster-admin")) | .subjects'
```

흔한 실수 세 가지다.

1. `verbs: ["*"]`처럼 와일드카드를 남용한다. 필요한 동사만 명시해야 한다.
2. `resources: ["*"]`로 모든 리소스에 접근을 허용한다.
3. `cluster-admin` ClusterRole을 일반 서비스 계정에 직접 바인딩한다.

심화 질문. Operator가 관리하는 CRD에 대한 RBAC는 어떻게 설계하는가? Operator는 자신이 관리하는 CRD에 대한 CRUD 권한과 서비스 계정에 권한을 부여하는 능력(`rbac.authorization.k8s.io/v1` 리소스에 대한 권한)이 필요하다. Operator가 과도한 ClusterRole을 갖지 않도록 최소 권한 원칙을 적용한다.
