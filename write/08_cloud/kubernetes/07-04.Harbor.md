# Harbor

> Harbor는 단순 Docker 이미지 저장소가 아니라 프로젝트, 권한, 취약점 스캔, 복제, OCI Helm chart를 함께 관리하는 클라우드 네이티브 레지스트리다. GitOps 흐름에서는 이미지와 chart의 공급원 역할을 맡는다.



## 학습 목표
> 이미지 레지스트리와 OCI chart 저장소를 하나의 운영 계층으로 묶어 이해한다.

이 장에서 확인할 목표는 다음과 같다:

1. Harbor가 일반 이미지 레지스트리보다 어떤 운영 기능을 더 제공하는지 설명할 수 있다.
2. 프로젝트, 로봇 계정, 권한 모델이 자동화 배포와 어떻게 연결되는지 이해할 수 있다.
3. Helm chart를 OCI 방식으로 Harbor에 저장하고 사용하는 흐름을 설명할 수 있다.
4. Trivy 스캔, 복제, 외부 저장소 연동이 왜 중요한지 설명할 수 있다.
5. ArgoCD와 Harbor를 함께 사용할 때 공급원 구조를 설명할 수 있다.



## 1. 왜 Harbor인가
> 프로덕션에서는 "푸시/풀 된다"보다 "누가 무엇을 어디까지 쓸 수 있는가"가 더 중요하다.

일반 Docker Registry는 이미지를 저장하고 내려받는 기능에 집중한다. Harbor는 여기에 프로젝트 단위 RBAC, 취약점 스캔, 이미지 서명, 복제, 로봇 계정, 감사 로그를 얹는다. 즉 레지스트리를 보안과 거버넌스가 있는 플랫폼으로 바꿔 준다.

특히 GitOps 관점에서 Harbor의 장점은 두 가지다.

1. 팀별 프로젝트로 이미지를 격리할 수 있어 Jenkins나 GitHub Actions, ArgoCD가 같은 레지스트리를 써도 권한을 분리하기 쉽다.
2. OCI 레지스트리로 Helm chart까지 함께 저장할 수 있어 이미지와 chart 공급원을 한 도메인 아래로 모을 수 있다.



## 2. 핵심 구성 요소
> Harbor는 단일 컨테이너가 아니라 여러 컴포넌트의 조합이다.

Harbor는 `core`, `registry`, `jobservice`, `portal`, `database`, `redis`, `trivy` 같은 구성 요소로 이뤄진다. 공식 문서 기준으로 고가용성 설치에서는 대부분의 애플리케이션 컴포넌트는 무상태에 가깝고, 데이터 계층은 외부 PostgreSQL·Redis·오브젝트 스토리지로 분리하는 구성이 일반적이다.

| 구성 요소 | 역할 |
|----------|------|
| `core` | API, 인증, 권한, 프로젝트 관리 |
| `registry` | OCI 이미지와 chart 저장 |
| `jobservice` | 복제, 스캔 같은 백그라운드 작업 |
| `trivy` | 취약점 스캔 |
| PostgreSQL / Redis | 메타데이터와 작업 상태 저장 |



## 3. Helm과 OCI chart
> 현재 흐름에서는 Harbor를 이미지 저장소이자 Helm chart 저장소로 함께 보는 편이 자연스럽다.

과거 Harbor는 ChartMuseum 방식 Helm 저장소를 지원했지만, 현재 문맥에서는 OCI Helm chart 사용이 더 자연스럽다. Helm 3.8 이후 OCI chart 지원이 일반화되었기 때문이다.

```bash
helm registry login <harbor-host>
helm push my-chart-0.1.0.tgz oci://<harbor-host>/<project>
helm pull oci://<harbor-host>/<project>/my-chart --version 0.1.0
```

이 구조의 장점은 단순하다. 이미지와 chart가 모두 같은 프로젝트, 같은 권한 체계, 같은 복제 정책 아래 들어간다. Helm 전용 저장소와 이미지 저장소를 따로 운영할 때보다 흐름이 짧아진다.



## 4. 자동화 계정과 권한
> CI/CD에서는 사람 계정보다 로봇 계정이 기본이어야 한다.

Harbor는 프로젝트별 Robot Account를 제공한다. Jenkins, GitHub Actions, ArgoCD는 일반 사용자 계정 대신 프로젝트 로봇 계정으로 로그인하는 편이 안전하다. 권한도 `push`, `pull`, `scanner`, `project admin`처럼 필요한 범위만 줄 수 있다.

이 구조를 쓰면 어떤 파이프라인이 어느 프로젝트에 이미지를 올렸는지 추적하기 쉽고, 토큰을 개별적으로 폐기하기도 쉽다.



## 5. ArgoCD와의 연결
> GitOps에서 Harbor는 "배포할 것을 저장하는 곳", ArgoCD는 "배포 상태를 맞추는 곳"이다.

권장 흐름은 다음과 같다.

1. CI가 이미지를 빌드해 Harbor 프로젝트에 push한다.
2. 필요하면 Helm chart도 OCI chart로 Harbor에 push한다.
3. Git에는 Harbor 이미지 태그 또는 chart 버전이 기록된다.
4. ArgoCD가 Git을 보고 Harbor의 artifact를 참조해 클러스터에 반영한다.

즉 Harbor는 artifact 저장소이고, ArgoCD는 상태 동기화 도구다. 둘의 역할을 분리해야 GitOps 흐름이 깔끔해진다.
