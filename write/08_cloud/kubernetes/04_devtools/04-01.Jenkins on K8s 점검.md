# Jenkins on K8s 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> Jenkins를 Kubernetes에 올릴 때의 운영 포인트를 질문 중심으로 다시 점검한다.

### Q1. Jenkins Controller와 Agent를 같은 방식으로 다뤄도 되는가?

아니다. Controller는 PVC와 JCasC, 플러그인 상태를 가진 장기 상태 컴포넌트이고, Agent는 빌드마다 생성됐다가 사라지는 단기 실행 환경이다. 둘을 같은 방식으로 취급하면 Jenkins Home 오염이나 Agent 상태 누적 같은 문제가 생긴다.

```bash
kubectl get pods -n jenkins
kubectl get pvc -n jenkins
kubectl describe pod -n jenkins <agent-pod-name>
```

심화 질문. Agent Pod에도 PVC를 붙여 캐시를 재사용하면 무조건 좋은가? 아니다. 빌드 속도는 빨라질 수 있지만, 캐시 오염과 재현성 저하가 생길 수 있다. 공용 캐시와 깨끗한 실행 환경 사이에서 팀 기준을 먼저 정해야 한다.

### Q2. VM 기반 Jenkins Agent vs K8s Pod Agent의 차이와 장단점

VM 에이전트는 수동으로 생성하거나 VM 이미지로 자동화해도 분 단위 시간이 걸린다. K8s Pod는 이미지가 캐시되어 있으면 초 단위로 생성되므로 빌드 큐가 급증할 때 즉시 스케일 아웃할 수 있다.

리소스 효율성도 다르다. VM 에이전트는 24시간 실행되므로 idle 시간에도 비용이 발생한다. 하루 8시간만 빌드가 발생한다면 16시간은 낭비다. Pod는 빌드 시작 시 생성되고 종료 시 삭제되므로 실제 사용 시간만큼만 비용을 지불한다.

환경 일관성 측면에서 VM 에이전트는 시간이 지나면서 "drift"가 발생한다. 여러 팀이 다른 도구를 설치하면 다음 빌드는 이전 빌드의 잔재물을 상속받는다. Pod는 매번 이미지에서 새로 시작하므로 항상 동일한 초기 상태를 갖는다.

보안과 격리 면에서 VM 에이전트는 프로세스 레벨 격리만 제공한다. Pod는 컨테이너 격리와 Kubernetes RBAC, Network Policy로 훨씬 강력한 경계를 제공한다.

유지보수 부담도 다르다. VM 에이전트는 OS 패치, 보안 업데이트를 수동으로 해야 한다. Pod는 이미지만 업데이트하면 되고, 이미지는 CI/CD로 자동 빌드할 수 있다.

심화. Pod의 이미지 pull 시간이 길면 VM보다 느릴 수 있다. `ImagePullPolicy: IfNotPresent`와 노드 이미지 캐싱, 클러스터 내부 레지스트리 배치로 최적화한다. 빌드 캐시(Maven `.m2`, npm `node_modules`) 재사용은 PVC를 공유하거나 외부 캐시 서버를 사용한다.

### Q3. JCasC(Configuration as Code)가 해결하는 문제

전통적인 Jenkins는 웹 UI에서 클릭으로 설정한다. 이 설정을 다른 Jenkins 인스턴스에 복제하려면 똑같은 클릭을 반복해야 하므로 재현이 불가능하다. Jenkins 설정은 파일 시스템에 XML로 저장되어 버전 관리가 어렵고, 개발/스테이징/프로덕션 Jenkins가 서로 다른 설정을 갖기 쉽다.

JCasC를 사용하면 `jenkins.yaml` 하나에 모든 설정을 선언하고 Git PR 리뷰를 거쳐 변경할 수 있다. Jenkins 서버가 손실되면 새 인스턴스를 띄우고 `jenkins.yaml`만 적용하면 몇 분 만에 원래 상태로 돌아온다. Helm 차트의 `values.yaml`에 JCasC 설정을 넣으면 ArgoCD나 Flux 같은 GitOps 도구로 자동 배포도 가능하다.

JCasC로 관리할 수 없는 것은 플러그인 바이너리와 사용자가 웹 UI에서 직접 생성한 작업의 XML이다. 민감 정보(API 토큰)는 JCasC에 직접 쓰지 않고 환경 변수나 Secret 참조로 주입한다.

심화. 기존 Jenkins를 JCasC로 마이그레이션하려면 JCasC export 플러그인이나 수동 변환을 사용한다. 기존 XML 설정을 분석하여 YAML로 변환하는 작업이 필요하다.

### Q4. Kubernetes Plugin의 Pod Template 동작 원리

Pod Template은 에이전트 Pod를 생성할 때 사용할 "청사진"이다. 어떤 이미지를 사용할지, 리소스는 얼마나 할당할지, 어떤 볼륨을 마운트할지를 정의한다. JCasC에 정적으로 정의하거나 Jenkinsfile에 동적으로 정의하며, Jenkinsfile 정의가 우선한다.

Pod Template에는 `label`이 있다. Jenkinsfile에서 `agent { label 'maven' }`이라고 쓰면 Jenkins가 "maven" 레이블을 가진 Pod Template을 찾아서 Pod를 생성한다. 레이블이 없으면 "There are no nodes with the label 'maven'" 에러가 발생한다.

Pod Template은 기본적으로 "jnlp" 컨테이너를 포함한다. 이 컨테이너는 `jenkins/inbound-agent` 이미지를 사용하고, Jenkins Controller의 50000 포트로 연결한다. 에이전트가 Controller에 먼저 연결을 맺는 inbound 방식이므로, 방화벽 뒤의 에이전트도 연결할 수 있다.

Pod Template에 여러 컨테이너를 정의할 수 있다. 예를 들어, `maven`, `docker`, `kubectl` 컨테이너를 하나의 Pod에 넣으면 Jenkinsfile에서 `container('maven') { ... }`로 특정 컨테이너에서 명령을 실행한다. 모든 컨테이너는 같은 네트워크와 볼륨을 공유하므로 localhost로 통신하고 파일을 주고받을 수 있다.

리소스 할당은 Pod Template의 `resourceRequestCpu`, `resourceRequestMemory` 등으로 설정한다. Kubernetes Scheduler는 requests를 보고 Pod를 배치할 노드를 선택하고, kubelet은 limits를 넘으면 Pod를 kill한다.

심화. 한 빌드에서 서로 다른 Pod Template을 Parallel 단계에서 사용할 수 있다. JNLP 대신 SSH로 에이전트를 연결하는 것도 가능하지만, JNLP가 권장된다.

### Q5. Jenkins Controller의 HA 구성 방법과 한계

Jenkins는 기본적으로 Active-Standby HA를 지원한다. 메인 Controller가 작동하고, 백업 Controller는 대기하다가 메인이 죽으면 PVC를 붙여서 활성화된다. Active-Active(여러 Controller가 동시에 작동)는 공식적으로 지원하지 않는다. Jenkins Home의 파일 잠금과 플러그인 상태 동기화 문제가 있기 때문이다.

PVC를 여러 Pod가 공유하려면 RWX(ReadWriteMany)가 필요하지만, 대부분의 클라우드 스토리지(AWS EBS, GCP PD)는 RWO(ReadWriteOnce)만 지원한다. NFS, EFS, Azure Files는 RWX를 지원하지만 성능이 떨어지고 파일 잠금 문제가 발생할 수 있다.

에이전트는 HA가 필요 없다. 빌드 중에 Pod가 죽으면 Jenkins가 다른 Pod를 생성하고 빌드를 재시작한다. Controller가 죽으면 모든 빌드가 멈추는 것이 핵심 문제다.

심화. Jenkins X나 Tekton 같은 대안은 Controller 없이 빌드를 직접 K8s에 제출하는 방식으로 HA를 해결한다. CloudBees Jenkins Enterprise는 Operations Center로 여러 Controller를 중앙에서 관리하는 상용 솔루션이다.

### Q6. Jenkins on K8s에서 Docker-in-Docker vs Kaniko 차이

DinD(Docker-in-Docker)는 Docker 데몬을 Pod 안에서 실행하는 방식이다. `docker:dind` 이미지를 사용하고, `privileged: true`로 컨테이너를 특권 모드로 실행한다. 빌드 스크립트에서 `docker build`, `docker push`를 그대로 사용할 수 있어서 편하지만, Privileged 컨테이너는 호스트 커널에 접근할 수 있어서 컨테이너 탈출 공격에 취약하다.

Kaniko는 Google이 만든 도구로, Docker 데몬 없이 컨테이너 이미지를 빌드한다. Dockerfile을 읽어서 각 레이어를 userspace에서 생성하고 레지스트리에 직접 푸시한다. Privileged 모드가 필요 없어서 안전하다. 단점은 Dockerfile의 모든 기능을 지원하지 않고 빌드 속도가 약간 느릴 수 있다는 점이다.

Docker Socket 마운트는 호스트의 `/var/run/docker.sock`을 Pod에 마운트하는 방법이다. 간단하지만 모든 Pod가 호스트 Docker를 공유하므로 격리가 없다.

프로덕션 환경에서는 Kaniko를 권장하고, 로컬 개발이나 테스트에서는 DinD를 사용할 수 있다. 또는 이미지 빌드를 전용 빌드 클러스터나 외부 서비스(Google Cloud Build)로 분리하는 방법도 있다.

### Q7. RBAC과 ServiceAccount로 Jenkins 권한을 제한하는 방법

Jenkins Controller는 Pod를 생성하고 삭제할 권한이 필요하다. Deployment를 수정하거나 다른 Namespace에 접근하거나 Cluster Admin 권한을 줘서는 안 된다. RBAC으로 정확히 필요한 권한만 부여한다.

Jenkins Controller Pod는 `jenkins` ServiceAccount로 실행된다. 이 ServiceAccount에 Role을 바인딩하면 Pod 내부에서 Kubernetes API를 호출할 때 해당 권한을 갖는다.

Jenkins는 보통 자신의 Namespace에서만 Pod를 생성하므로 ClusterRole이 아닌 Role로 충분하다. 여러 Namespace에 에이전트를 배포하려면 ClusterRole이 필요하다.

Secret 접근은 `rbac.readSecrets: true`로 추가할 수 있지만, 특정 Secret만 읽을 수 있도록 `resourceNames`로 제한하는 것이 안전하다.

```yaml
- apiGroups: [""]
  resources: ["secrets"]
  resourceNames: ["git-token", "docker-registry"]
  verbs: ["get"]
```

심화. ServiceAccount 토큰이 탈취되면 해당 권한으로 API 호출이 가능하다. 토큰 rotation을 설정하고 Audit Log로 의심스러운 API 호출을 추적하는 것이 중요하다.

### Q8. Jenkins Helm chart의 주요 values 파라미터

`controller.image`와 `controller.tag`로 Jenkins 버전을 선택한다. LTS 버전은 `lts` 태그를 사용하고, 특정 버전은 `2.440-jdk17` 같은 태그를 명시한다. 최신 플러그인은 JDK 11 이상을 요구하므로 JDK 버전도 고려해야 한다.

`controller.resources`로 CPU/메모리 requests와 limits를 지정한다. Jenkins는 메모리를 많이 사용하므로 requests를 너무 낮게 잡으면 OOMKilled가 발생한다. 최소 512Mi requests, 1Gi limits를 권장한다.

`controller.serviceType`은 ClusterIP, NodePort, LoadBalancer 중 선택한다. GCP 클러스터에서는 NodePort가 편하고, 프로덕션에서는 Ingress + ClusterIP를 사용한다.

`controller.installPlugins`에 플러그인 이름과 버전을 넣으면 Jenkins 시작 시 자동 설치된다. 프로덕션에서는 버전을 고정하는 것이 좋다. `controller.JCasC.configScripts`에 YAML로 설정을 넣으면 여러 개의 configScript를 이름별로 정의할 수 있다.

`persistence.enabled: true`로 PVC를 사용하고 `persistence.existingClaim`으로 기존 PVC를 재사용할 수도 있다. Helm 차트 업그레이드 시 PVC 데이터는 유지된다.
