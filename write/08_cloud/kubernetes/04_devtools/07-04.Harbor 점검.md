# Harbor 점검

> 본 장의 심화 점검 질문이다. Harbor를 실제 공급원 레지스트리로 운영할 때 프로젝트, 로봇 계정, OCI chart 관리 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> Harbor를 단순 푸시/풀 저장소가 아니라 운영 거버넌스 계층으로 다시 점검한다.

### Q1. 사람 계정 대신 Robot Account를 왜 써야 하는가?

CI/CD에서는 사람이 퇴사하거나 권한이 바뀌어도 파이프라인이 계속 돌 수 있어야 한다. 그래서 Jenkins, GitHub Actions, ArgoCD는 일반 사용자 계정보다 프로젝트 Robot Account를 쓰는 편이 낫다. 토큰 범위를 프로젝트 단위로 제한할 수 있고, 사고가 나도 해당 계정만 폐기하면 된다.

### Q2. Harbor에 Helm chart를 넣을 때 왜 OCI 방식이 중요한가?

이미지와 chart를 같은 프로젝트, 같은 권한, 같은 복제 정책 아래 넣을 수 있기 때문이다. 전통적인 chart repo와 이미지 레지스트리를 분리하면 공급원과 권한 관리가 이원화된다. 현재 운영 흐름에서는 OCI 방식이 더 자연스럽다.

```bash
helm registry login <harbor-host>
helm push my-chart-0.1.0.tgz oci://<harbor-host>/<project>
```

### Q3. Trivy 스캔 결과는 배포를 막는 기준이 될 수 있는가?

가능하지만 무조건 막는 식으로 설계하면 운영이 경직될 수 있다. 보통은 `Critical`, `High` 기준을 프로젝트 성격에 따라 정하고, CI나 승인 절차와 연결한다. 보안 정책은 레지스트리에서 끝나지 않고 배포 정책과 이어져야 의미가 있다.

### Q4. ArgoCD와 Harbor를 같이 쓸 때 단일 진실 공급원은 어디인가?

여전히 Git이다. Harbor는 artifact 저장소이고, ArgoCD는 Git에 기록된 버전 정보를 보고 Harbor artifact를 참조한다. Harbor 자체가 상태 기준이 되기 시작하면 GitOps 흐름이 흐려진다.

### Q5. Harbor 복제와 외부 저장소 연동은 어떤 운영 가치를 만드는가?

복제(Replication)는 다중 리전·다중 클러스터 운영에서 *이미지 가용성*을 높인다. 메인 Harbor가 빠지더라도 복제 대상 Harbor에서 이미지를 받아 배포가 이어진다. Pull-based 복제와 Push-based 복제 두 모드가 있고, 운영에서는 *원격 사이트가 메인을 polling*하는 Pull 모드가 메인 측 부담이 작다.

외부 저장소 연동(Proxy Cache 또는 Replication endpoint)은 Docker Hub·GHCR·다른 Harbor 등을 신뢰된 원본으로 두고 우리 Harbor가 그 앞단 캐시로 동작하는 패턴이다. 사내 클러스터가 외부 인터넷 의존을 줄이고, 동일 이미지 대량 pull 시 외부 트래픽 비용을 절감한다. Trivy 스캔과 결합하면 외부 이미지가 Harbor를 거쳐 들어올 때 일관된 보안 정책 검사를 받게 된다.
