# OOMKilled 사례 분석 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검한다.

### Q1. JVM heap이 멀쩡한데 cgroup이 OOMKill을 발동시키는 경우의 핵심은?

JVM의 RSS는 heap만이 아니다. metaspace, JIT 코드 캐시, thread stack, native memory, NIO direct buffer, GC 데이터 구조 등이 모두 heap 밖(off-heap) 메모리를 점유한다. cgroup은 RSS 전체를 본다.

heap 사용률 80%는 heap 한도 4GB 중 3.2GB가 쓰였다는 의미일 뿐, off-heap이 1.8GB라면 RSS가 5GB가 된다. cgroup limit이 5GB라면 그 직후 OOMKill이 일어난다. heap 모니터링만 보고 "여유 있다"고 판단하면 cgroup OOM의 원인을 못 찾는다.

처방은 heap 비율을 메모리 한도에 비해 낮추는 것이다. `-XX:MaxRAMPercentage=50` 같은 옵션으로 heap을 컨테이너 메모리의 50%로 잡고, 나머지 50%를 off-heap·OS 페이지캐시에 양보하는 마진을 둔다.

### Q2. NMT(Native Memory Tracking)는 무엇을 보여주고 어떻게 켜는가?

NMT는 JVM 자체의 native 메모리를 카테고리별로 분류한다. heap, class, thread, code, GC, internal, symbol, NMT, other 등으로 나뉜다. heap만으로는 보이지 않는 사용처(JIT 코드 캐시, metaspace, 스레드 스택 합계)를 정량화할 수 있다.

켜는 방법은 `-XX:NativeMemoryTracking=summary` 또는 `=detail`을 JVM 옵션에 추가한다. 기동 후 `jcmd <pid> VM.native_memory summary`로 현재 사용량을 본다. baseline을 기록한 뒤 시간이 지나서 다시 비교하면 메모리 누수가 어떤 카테고리에서 발생하는지 좁힐 수 있다.

오버헤드는 summary 모드에서 약 5%, detail에서 더 크다. 운영 환경에서 켜 두기보다는 의심 시점에 일시 적용하고 분석 후 끄는 패턴이 안전하다.

### Q3. Pod 메모리 사용량이 한계에 가까울 때 어떻게 진단을 시작하는가?

세 단계로 좁힌다.

먼저 **컨테이너 RSS와 limit 비교**. `kubectl top pod` 또는 `kubectl describe pod`의 `Last State`. cgroup 통계로 들어가려면 `kubectl exec -- cat /sys/fs/cgroup/memory.current`(cgroup v2) 또는 `memory.usage_in_bytes`(v1)를 본다.

다음 **JVM heap 사용률**. JMX 메트릭(`jvm_memory_used_bytes{area="heap"}` 등)을 Prometheus에 노출했다면 그래프로 추세를 본다. heap이 한도 가까이 가면 GC 직전 패턴을 보인다.

마지막 **off-heap의 차이 계산**. 컨테이너 RSS - JVM heap 사용 = off-heap. 이 값이 시간에 따라 누적 증가하면 native 누수, 갑자기 커지면 NIO buffer 폭증 같은 burst 사용이다.

이 세 값이 함께 그려지는 그래프가 있으면 OOMKill 직전의 분포를 한눈에 본다. heap만 보는 단일 지표 모니터링이 OOM 디버깅에서 자주 부족한 이유다.

### Q4. heap dump가 OOM 직전에 안 떨어지는 이유는?

`-XX:+HeapDumpOnOutOfMemoryError`는 JVM이 OutOfMemoryError를 throw할 때 heap dump를 만든다. 그러나 cgroup OOMKill은 OS가 SIGKILL로 프로세스를 즉시 종료시키므로 JVM에 OOM error를 throw할 시간이 없다. heap dump가 생성되지 않는다.

대안은 두 가지다.

1. **heap 사용률 임계 도달 시 외부에서 dump를 트리거**. 모니터링 알람이 80~90% 수준에서 heap dump를 자동 수집하는 스크립트를 띄운다.
2. **liveness probe로 OOM 직전을 감지**. 메모리 사용률 임계 도달 시 probe가 실패해 컨테이너가 SIGTERM을 받게 만들면, JVM이 정상 종료 흐름에서 dump를 남길 시간이 있다.

근본은 cgroup OOM 자체를 줄이는 것이다. heap·off-heap의 합이 limit에 가까이 가는 설계 자체가 잘못된 출발점이다.

### Q5. emptyDir의 메모리 모드가 OOM에 어떻게 기여하는가?

`emptyDir`은 기본 노드 디스크를 쓰지만 `medium: Memory`를 지정하면 tmpfs로 메모리에 저장된다. 이 메모리 사용은 컨테이너 cgroup의 메모리 사용량에 합산된다. 즉 `/tmp` 같은 캐시 디렉터리에 큰 파일을 쓰면 그만큼 cgroup 메모리가 차서 OOMKill에 가까워진다.

흔한 사고는 "JVM heap도 멀쩡, off-heap도 그대로인데 OOMKilled". 원인을 추적하다 보면 `/tmp`에 5GB짜리 캐시 파일이 발견되는 식이다.

처방은 두 가지. memory emptyDir에 `sizeLimit`을 두어 의도한 상한을 강제한다. 또는 디스크 기반 emptyDir로 바꿔 메모리에서 분리한다. tmpfs가 정말 필요한 경우(빠른 스크래치 공간)에는 컨테이너 limit을 그만큼 더 크게 잡는다.

### Q6. JVM `-Xmx`와 컨테이너 메모리 limit의 관계는?

옛 JVM은 컨테이너 cgroup을 인식하지 못해 `-Xmx`를 안 주면 호스트 메모리 기준으로 heap을 잡았다. 컨테이너 limit이 4GB인데 JVM이 heap을 12GB로 잡고 사용하다 OOM 당하는 사고가 흔했다.

JDK 8u191+, 11+에서는 `-XX:+UseContainerSupport`(기본 true)로 cgroup을 인식한다. `-XX:MaxRAMPercentage=50.0`처럼 비율로 지정하면 컨테이너 limit의 50%를 heap에 쓴다. 4GB limit이면 2GB가 heap, 나머지 2GB가 off-heap·OS 캐시·tmpfs·여유분이다.

운영에서 권장 패턴은 **MaxRAMPercentage 50~75% + 모니터링으로 실제 RSS 확인 후 조정**이다. off-heap 사용이 큰 워크로드(NIO 헤비, native 라이브러리 사용)는 비율을 낮추고, heap 위주 워크로드는 높인다.

### Q7. 같은 이미지를 다른 limit으로 띄웠는데 한쪽만 OOMKill되는 이유는?

`-XX:MaxRAMPercentage`만 같다면 limit이 작은 쪽의 heap도 작아져야 정상이다. 그런데 native 메모리(JIT 코드, metaspace, thread stack)는 거의 일정하게 든다. heap이 작아져도 native는 그대로인 만큼, 작은 limit Pod는 heap : native 비율이 깨져 native가 한도의 큰 비중을 차지한다.

실무에서 쓰이는 방법은 두 가지다. 컨테이너 limit이 작은 환경에서는 `-XX:MaxRAMPercentage`를 더 낮춘다. 또는 `-Xmx`를 절대값으로 명시한다. 이때 native 사용량을 NMT로 측정해 두면 heap 외 마진을 구체 숫자로 잡을 수 있다.

쉽게 잊기 쉬운 점은 **thread 수**다. JVM의 thread stack 기본 크기는 1MB라, 1000개 thread는 1GB가 된다. 동시 요청이 많은 워크로드의 작은 limit Pod는 thread 수만 증가해도 OOMKill에 가깝다.

### Q8. 사례를 일반화한다면 운영 점검 체크리스트는 어떻게 되는가?

다섯 항목이 핵심이다.

1. **heap : off-heap 분리 모니터링**. heap 단일 지표만 보지 않는다. 컨테이너 RSS 그래프와 JVM heap 그래프를 같은 패널에 그린다.
2. **cgroup OOM 알람을 별도로**. `kube_pod_container_status_last_terminated_reason{reason="OOMKilled"}`로 알람을 둔다. JVM의 OutOfMemoryError와 다른 사건이다.
3. **emptyDir tmpfs 사용량 추적**. memory emptyDir이 있는 Pod는 그 사용량도 메모리 한도 안에 있다.
4. **heap 비율 보수적 시작**. `MaxRAMPercentage` 50~60%부터 시작해, 모니터링 데이터로 점진적으로 조정한다.
5. **NMT를 비상시 도구로 준비**. 제로데이 OOM 발생 시 즉시 NMT를 활성화할 수 있는 운영 절차를 두어, 디버깅이 추측 영역에서 측정 영역으로 옮겨가게 한다.
