# 자원 관리 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 자원 요청과 제한 설계를 운영 질문으로 다시 점검한다.

### Q1. Requests와 Limits 중 어떤 값이 스케줄링을 결정하는가?

스케줄러는 `requests`를 기준으로 판단한다. `limits`는 런타임 상한선이지, 스케줄러의 입력이 아니다. 그래서 실제 사용량이 낮아도 requests 합이 노드 allocatable을 넘으면 Pod는 `Pending`이 될 수 있다.

```bash
kubectl describe node <node-name>
kubectl describe pod <pod-name> -n <ns>
kubectl top pod -n <ns>
```

심화 질문. 실제 사용률은 20%인데 왜 `FailedScheduling`이 발생하는가? 스케줄러는 현재 사용률이 아니라 requests 합계를 보기 때문이다. 운영자가 `kubectl top`만 보고 판단하면 쉽게 오해한다.

### Q2. Limits만 설정했을 때 자동 Requests 설정 확인

```bash
kubectl create namespace ch20-smoke

# Limits만 설정한 Pod
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: limits-only
spec:
  containers:
    - name: app
      image: nginx:1.25
      resources:
        limits:
          cpu: "500m"
          memory: "512Mi"
        # requests를 명시하지 않음
EOF

# Requests가 자동으로 Limits와 동일하게 설정되었는지 확인
kubectl get pod limits-only -n ch20-smoke -o jsonpath=\
'{.spec.containers[0].resources}' | jq .
# 출력:
# {
#   "limits": {"cpu": "500m", "memory": "512Mi"},
#   "requests": {"cpu": "500m", "memory": "512Mi"}  <- 자동 설정
# }

# QoS 클래스 확인
kubectl get pod limits-only -n ch20-smoke -o jsonpath=\
'{.status.qosClass}'
# Guaranteed
```

심화 질문. LimitRange에 `default`(Limits 기본값)만 설정하고 `defaultRequest`를 생략하면 어떻게 되는가? Kubernetes가 먼저 LimitRange의 `default` Limits를 적용하고, 이후 Requests가 없으면 Limits와 동일하게 설정한다. 최종적으로 Requests = Limits = LimitRange의 default 값이 된다.

### Q3. QoS 클래스별 Eviction 순서 확인

```bash
# BestEffort Pod (Requests/Limits 없음)
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: best-effort
spec:
  containers:
    - name: app
      image: nginx:1.25
      # resources 미설정
EOF

# Burstable Pod (Requests < Limits)
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: burstable
spec:
  containers:
    - name: app
      image: nginx:1.25
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "512Mi"
EOF

# Guaranteed Pod (Requests = Limits)
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: guaranteed
spec:
  containers:
    - name: app
      image: nginx:1.25
      resources:
        requests:
          cpu: "200m"
          memory: "256Mi"
        limits:
          cpu: "200m"
          memory: "256Mi"
EOF

# 각 Pod의 QoS 클래스 확인
kubectl get pods -n ch20-smoke \
  -o custom-columns='NAME:.metadata.name,QOS:.status.qosClass'
```

심화 질문. 같은 QoS 클래스 내 Eviction 순서는 무엇인가? Burstable Pod 간에는 Requests 대비 실제 메모리 사용량 비율이 높은 Pod부터 퇴거한다. `kubectl top pod -n ch20-smoke`로 현재 사용량을 확인하고, Requests와 비교해 초과 비율이 높은 Pod가 먼저 제거된다.

### Q4. CPU Throttling 감지

```bash
# CPU throttling 메트릭 확인 (Prometheus 필요)
# container_cpu_cfs_throttled_periods_total / container_cpu_cfs_periods_total

# throttling 발생 시뮬레이션
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: cpu-stress
spec:
  containers:
    - name: stress
      image: polinux/stress
      command: ["stress"]
      args: ["--cpu", "2", "--timeout", "60s"]
      resources:
        requests:
          cpu: "200m"
        limits:
          cpu: "300m"   # 낮은 Limits로 throttling 유발
EOF

# 컨테이너 CPU 사용 확인
kubectl top pod cpu-stress -n ch20-smoke

# cgroup 직접 확인 (노드에서)
# cat /sys/fs/cgroup/cpu/kubepods/.../cpu.stat
# nr_throttled 값이 증가하면 throttling 발생 중
```

심화 질문. CPU throttling과 OOMKill의 차이는 무엇인가? CPU throttling은 Limits 초과 시 CPU를 일시적으로 제한한다. Pod는 계속 실행되지만 응답이 느려진다. OOMKill은 메모리 Limits 초과 시 커널이 컨테이너를 즉시 종료한다. CPU throttling은 성능 저하만 유발하지만, 메모리 초과는 서비스 중단을 일으킨다.

### Q5. LimitRange와 ResourceQuota 조합

```bash
# LimitRange 적용
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
spec:
  limits:
    - type: Container
      default:
        cpu: "200m"
        memory: "256Mi"
      defaultRequest:
        cpu: "100m"
        memory: "128Mi"
      max:
        cpu: "2"
        memory: "4Gi"
      min:
        cpu: "50m"
        memory: "64Mi"
EOF

# ResourceQuota 적용
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: ResourceQuota
metadata:
  name: namespace-quota
spec:
  hard:
    requests.cpu: "2"
    requests.memory: "4Gi"
    limits.cpu: "4"
    limits.memory: "8Gi"
    pods: "10"
EOF

# Quota 사용 현황 확인
kubectl describe resourcequota namespace-quota -n ch20-smoke

# Quota 초과 시도
kubectl run extra-pod --image=nginx \
  --requests='cpu=3' \
  -n ch20-smoke
# Error from server (Forbidden): exceeded quota: namespace-quota
```

심화 질문. ResourceQuota가 있는 네임스페이스에서 Requests/Limits 없이 Pod를 만들면 어떻게 되는가? 거부된다. LimitRange로 기본값을 주입하면 Requests/Limits가 자동으로 채워져 Quota 검사를 통과한다. LimitRange와 ResourceQuota는 반드시 함께 사용해야 한다.

### Q6. Guaranteed vs Burstable 선택 기준

```bash
# 데이터베이스 - Guaranteed 적합
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: database
spec:
  containers:
    - name: postgres
      image: postgres:15
      resources:
        requests:
          cpu: "500m"
          memory: "1Gi"
        limits:
          cpu: "500m"    # Requests = Limits -> Guaranteed
          memory: "1Gi"
      env:
        - name: POSTGRES_PASSWORD
          value: "password"
EOF

# API 서버 - Burstable 적합
kubectl apply -n ch20-smoke -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: api-server
spec:
  containers:
    - name: api
      image: nginx:1.25
      resources:
        requests:
          cpu: "100m"    # 평균 사용량
          memory: "128Mi"
        limits:
          cpu: "500m"    # 피크 사용량 허용
          memory: "256Mi"
EOF

kubectl get pods -n ch20-smoke \
  -o custom-columns='NAME:.metadata.name,QOS:.status.qosClass'

kubectl delete namespace ch20-smoke
```

심화 질문. "CPU는 Burstable, 메모리는 Guaranteed"로 혼합하면 QoS 클래스는 무엇이 되는가? QoS 클래스는 Pod 전체를 기준으로 결정된다. 모든 컨테이너의 모든 리소스(CPU, 메모리)가 Requests = Limits여야 Guaranteed가 된다. CPU가 Burstable이면 QoS 클래스 전체가 Burstable이 된다.

### Q6. Capacity vs Allocatable 확인

```bash
# 노드 Capacity와 Allocatable 비교
kubectl describe node dev-server-3 | grep -A 10 "Capacity:"
kubectl describe node dev-server-3 | grep -A 10 "Allocatable:"

# Allocated resources 상세
kubectl describe node dev-server-3 | grep -A 20 "Allocated resources:"

# 전체 클러스터 리소스 현황 요약
kubectl get nodes \
  -o custom-columns=\
'NAME:.metadata.name,CPU:.status.allocatable.cpu,MEM:.status.allocatable.memory'
```

Allocatable이 Capacity보다 작은 이유는 kube-reserved, system-reserved, eviction threshold가 차감되기 때문이다. "Allocated resources"의 Limits 합계가 100%를 초과해도 스케줄러는 이를 허용한다. Requests 합계만 100% 이하를 강제한다.

심화 질문. 관리형 Kubernetes(GKE, EKS)에서 시스템 예약분은 어떻게 설정되는가? 관리형 서비스는 노드 크기에 따라 자동으로 kube-reserved를 설정한다. GKE는 첫 4GB에 더 높은 비율을 예약하고 이후 슬라이딩 비율을 적용한다. 이 값을 커스터마이징하면 시스템 안정성에 영향을 줄 수 있어 일반적으로 수정하지 않는다.
