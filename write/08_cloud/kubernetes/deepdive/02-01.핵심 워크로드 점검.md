# 핵심 워크로드 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검한다.

### Q1. Pod 안에 여러 컨테이너를 넣는 경우는 언제인가?

Sidecar 패턴은 메인 컨테이너를 보조하는 컨테이너를 같은 Pod에 배치한다. 가장 흔한 예는 로그 수집기(Fluentd, Filebeat)로, nginx가 `/var/log/nginx/access.log`에 로그를 쓰면 sidecar가 같은 emptyDir 볼륨을 마운트해 로그를 읽고 Elasticsearch로 전송한다. Service Mesh(Istio)도 sidecar 패턴으로 envoy-proxy 컨테이너를 자동 주입한다.

Ambassador 패턴은 메인 컨테이너가 외부 서비스에 접근할 때 ambassador가 프록시 역할을 한다. 레거시 앱이 `localhost:3306`으로 MySQL에 접근하는데 실제 DB가 클라우드에 있다면, ambassador가 Cloud SQL Proxy를 실행해 로컬 포트를 원격 DB로 포워딩한다.

Init Container는 메인 컨테이너가 시작되기 전에 초기화 작업을 실행한다. DB 스키마 마이그레이션이나 Git 저장소 클론이 대표적이다. Init Container가 성공적으로 종료되어야 메인 컨테이너가 시작된다.

Sidecar와 DaemonSet의 차이를 묻는다면, Sidecar는 특정 Pod에만 붙는 반면 DaemonSet은 모든 노드에서 하나씩 실행된다. Init Container가 실패하면 Pod는 `Init:CrashLoopBackOff` 상태가 되고 메인 컨테이너는 시작되지 않는다.

### Q2. Deployment가 ReplicaSet을 관리하는 방식과 롤백 메커니즘은?

Pod 템플릿(`spec.template`)이 변경될 때만 새로운 ReplicaSet이 생성된다. `spec.replicas`만 변경하면 기존 ReplicaSet의 replicas 필드만 업데이트되고 새 ReplicaSet은 생성되지 않는다.

RollingUpdate 동작은 새 ReplicaSet의 replicas를 0에서 점진적으로 증가시키고 이전 ReplicaSet의 replicas를 줄이는 방식이다. `maxSurge: 1, maxUnavailable: 0`이면(desired=3 기준) Pod 수가 3 → 4 → 3 → 4 → 3으로 변동된다.

롤백 시 Deployment Controller는 이전 ReplicaSet의 replicas를 desired 값으로 올리고 현재 ReplicaSet의 replicas를 0으로 설정한다. 역방향 RollingUpdate가 실행되므로 Pod를 새로 빌드할 필요가 없다.

`kubectl rollout pause deployment/<name>`으로 업데이트를 일시 정지하면 여러 변경을 누적한 뒤 한 번의 RollingUpdate로 처리할 수 있다. ReplicaSet을 직접 수정·삭제하면 Deployment Controller가 즉시 새로 생성하므로 항상 Deployment를 통해 관리해야 한다.

### Q3. Service의 label selector와 Endpoint 동작 원리는?

Service의 `spec.selector`는 네임스페이스 내에서 label이 일치하는 Pod를 찾는다. Service가 생성되면 Endpoint Controller가 자동으로 Endpoints 객체를 생성해 해당 Pod의 IP와 포트 목록을 저장한다.

각 노드의 kube-proxy는 Endpoints 객체를 감시하고 iptables 규칙을 업데이트한다. Readiness Probe가 실패한 Pod는 Endpoints 목록에서 제거되어 트래픽을 받지 않지만 컨테이너는 재시작되지 않는다.

Headless Service(`clusterIP: None`)는 DNS 조회 시 모든 Pod IP를 직접 반환한다. StatefulSet과 함께 `pod-0.service.default.svc.cluster.local` 형태로 개별 Pod에 접근할 때 사용한다. ExternalName Service는 클러스터 외부 DNS 이름을 Service로 매핑해 마이그레이션 시 유용하다.

### Q4. RollingUpdate 전략의 maxSurge/maxUnavailable 튜닝은?

`maxSurge`는 desired 개수를 초과해 생성할 수 있는 최대 Pod 수다. `replicas: 10, maxSurge: 2`이면 업데이트 중 최대 12개 Pod가 실행된다. 높을수록 업데이트가 빠르지만 리소스 소비가 증가한다.

`maxUnavailable`은 desired 개수 미만으로 줄어들 수 있는 최대 Pod 수다. `replicas: 10, maxUnavailable: 2`이면 업데이트 중 최소 8개 Pod가 실행된다. 높을수록 빠르지만 가용성이 감소한다.

무중단 최우선이면 `maxSurge: 1, maxUnavailable: 0`으로 설정한다. 개발·스테이징은 `maxSurge: 50%, maxUnavailable: 50%`처럼 공격적으로 설정해 속도를 높인다. `maxSurge: 0, maxUnavailable: 1`은 추가 리소스 없이 업데이트하는 절약형으로 리소스 제약 환경에 유용하다.

Readiness Probe 없이 RollingUpdate를 진행하면 Pod가 생성되자마자 Endpoint에 추가되어 초기화 중인 컨테이너로 트래픽이 흘러 장애가 발생한다.

### Q5. ConfigMap 변경 시 Pod에 반영하는 방법은?

환경변수로 주입한 경우 Pod 생성 시점에 고정되므로, ConfigMap을 변경해도 기존 Pod에는 반영되지 않는다. `kubectl rollout restart deployment/<name>`으로 Deployment를 재시작해야 한다.

볼륨으로 마운트한 경우 kubelet이 주기적으로(기본 1분) 동기화해 파일을 업데이트한다. 단, 대부분의 애플리케이션은 파일 변경을 감지하지 못하므로 SIGHUP 시그널을 보내거나 재시작이 필요하다. `subPath`를 사용해 마운트한 파일은 자동 업데이트가 되지 않는다.

Stakater Reloader 같은 도구를 쓰면 ConfigMap 변경 시 Deployment를 자동 재시작한다. `immutable: true`로 설정하면 ConfigMap을 수정할 수 없어 실수로 설정이 바뀌는 것을 방지하고 etcd 부하를 줄인다.

### Q6. Secret은 정말 안전한가?

Secret의 데이터는 base64 인코딩되어 etcd에 저장된다. 이는 단순 인코딩이므로 etcd에 접근하면 누구나 디코딩할 수 있다. etcd 암호화(`--encryption-provider-config`)와 RBAC로 Secret 접근을 제한해야 한다.

환경변수로 주입하면 `kubectl describe pod`, 프로세스 목록(`/proc/<pid>/environ`)에서 볼 수 있다. 볼륨 마운트가 더 안전하지만 코드 변경이 필요하다. 프로덕션에서는 External Secrets Operator로 AWS Secrets Manager나 HashiCorp Vault에서 동적으로 가져오는 것이 권장된다.

`stringData` 필드를 쓰면 평문으로 작성해도 API 서버가 자동으로 인코딩한다. 단, Git에 커밋하면 평문이 노출되므로 SealedSecrets이나 SOPS로 암호화한 뒤 커밋해야 한다.

### Q7. liveness vs readiness vs startup probe 구분과 설정 기준은?

Liveness Probe는 데드락이나 무한 루프를 감지해 컨테이너를 재시작한다. 너무 민감하게 설정하면 일시적 부하로 불필요한 재시작이 발생한다. `failureThreshold: 3, periodSeconds: 10`처럼 보수적으로 설정한다.

Readiness Probe는 트래픽을 받을 준비가 되었는지 확인한다. 실패해도 컨테이너를 재시작하지 않고 Endpoint에서만 제외한다. RollingUpdate 중 새 Pod가 준비되기 전에 트래픽을 받는 것을 방지하므로 필수 설정이다.

Liveness Probe에 DB 연결 상태를 넣으면 DB 장애 시 모든 Pod가 동시에 재시작되어 카스케이드 장애가 발생한다. 의존성 체크는 Readiness에, 애플리케이션 자체 건강은 Liveness에 분리해야 한다.

### Q8. Pod QoS 클래스(Guaranteed, Burstable, BestEffort)는?

모든 컨테이너가 `requests == limits`이면 Guaranteed다. 노드 메모리 압박 시 가장 마지막에 종료된다. 프로덕션의 중요 서비스(결제, 인증)에 사용한다.

최소 1개 컨테이너가 requests나 limits를 가지지만 `requests != limits`이면 Burstable이다. BestEffort 다음으로 종료된다.

requests와 limits가 모두 없으면 BestEffort다. 가장 먼저 종료된다. 메모리는 초과 시 OOMKilled로 즉시 종료되고, CPU는 throttling만 발생한다(종료되지 않는다). QoS 클래스는 `kubectl get pod <name> -o jsonpath='{.status.qosClass}'`로 확인한다.
