# Service와 EndpointSlice 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검한다.

### Q1. Service의 ClusterIP는 어디서 오고, 왜 노드별로 달라 보이지 않는가?

ClusterIP는 API 서버의 service-cluster-ip-range에서 할당되는 가상 IP다. 어느 노드의 네트워크 인터페이스에도 실제로 바인딩되지 않는다. kube-proxy가 각 노드의 iptables/IPVS에 "이 IP·포트로 향하는 패킷은 백엔드 Pod IP·포트로 DNAT" 규칙을 만들어 두기 때문에 모든 노드에서 같은 ClusterIP가 동작하는 것처럼 보인다.

따라서 `ip addr`에 ClusterIP가 안 보여도 정상이고, `ip route`에도 ClusterIP에 대한 직접 라우팅이 없다. iptables의 `KUBE-SERVICES` chain에서 같은 IP를 검사하는 규칙이 보이면 정상이다.

### Q2. Pod가 새로 추가됐는데 Service로 트래픽이 안 갈 때 어디서부터 보는가?

세 단계 차례로 본다.

먼저 **Service의 selector와 Pod 라벨이 일치**하는지. `kubectl get pod -l <selector> -n <ns>`로 라벨 매칭을 확인한다. selector 오타 한 글자가 가장 자주 낚는 함정이다.

다음 **EndpointSlice에 Pod가 들어 있는지**. `kubectl get endpointslice -l kubernetes.io/service-name=<service> -n <ns>`로 본다. `ready` 상태가 아닌 Pod는 EndpointSlice에 들어가지만 `conditions.ready: false`로 표시된다.

마지막 **Readiness Probe**. Pod가 Running이지만 Readiness Probe 실패면 EndpointSlice의 ready=false라 트래픽이 안 간다. `kubectl describe pod`의 `Readiness:` 항목에서 실패 사유를 본다.

### Q3. EndpointSlice는 왜 Endpoints를 대체했는가?

옛 Endpoints 객체는 Service당 1개로, Pod 수가 많아지면 단일 객체가 비대해진다. 1만 개 Pod의 Service라면 1개 객체에 1만 개 IP가 박힌다. 이 객체가 변경될 때마다 etcd 저장 비용과 전체 watch 트래픽이 폭증한다.

EndpointSlice는 Service당 여러 슬라이스로 분할한다(기본 100개 단위). Pod 수 변화 시 변경 슬라이스만 갱신되므로 watch 트래픽이 변경 범위에 비례한다. 대규모 클러스터의 컨트롤 플레인 부하를 크게 낮춘다. K8s 1.21+ 기본이며, 옛 Endpoints는 호환성 위해 자동으로도 유지된다.

### Q4. `externalTrafficPolicy: Local`은 어떤 상황에 쓰는가?

기본값 `Cluster`는 외부 트래픽이 노드에 들어오면 다른 노드의 Pod로도 자유롭게 분산된다. 로드는 균형이 맞지만, 패킷이 다른 노드로 SNAT되면서 클라이언트 IP가 노드 IP로 바뀐다. 애플리케이션이 `X-Forwarded-For`나 클라이언트 IP를 보고 정책 결정을 한다면 잘못 동작한다.

`Local`은 들어온 노드의 같은 노드 Pod로만 트래픽을 보낸다. 클라이언트 IP가 보존된다. 단, 그 노드에 해당 Pod가 없으면 트래픽이 그냥 거부된다. 노드별 Pod 분포가 균형이 맞지 않으면 일부 노드는 traffic 0, 일부는 과부하가 된다.

운영에서는 클라이언트 IP 보존이 필요한 워크로드(보안 정책, 지역 IP 기반 결제·과금)에서만 `Local`을 쓰고, 그 워크로드의 분포를 Topology Spread로 균형 맞춘다.

### Q5. SessionAffinity의 `ClientIP`는 어떻게 동작하고 한계는?

Service에 `sessionAffinity: ClientIP`를 설정하면, kube-proxy가 같은 클라이언트 IP의 트래픽을 같은 백엔드로 보낸다. iptables/IPVS에 IP→백엔드 매핑이 일정 시간(`sessionAffinityConfig.clientIP.timeoutSeconds`, 기본 3시간) 캐시된다.

한계는 두 가지다.

1. **NAT 뒤의 다중 사용자**가 같은 클라이언트 IP로 보인다. 사무실 NAT 뒤 100명이 한 백엔드로 몰릴 수 있다.
2. **자체 timeout만 정해진다**. 애플리케이션 세션과 동기화되지 않으므로, 백엔드가 잠시 비정상이 되어도 같은 IP는 그 백엔드로 계속 라우팅된다.

진짜 세션 친화성이 필요하면 Service Mesh의 L7 로드 밸런싱(쿠키, 헤더 기반)을 쓰는 편이 정확하다. ClusterIP의 `sessionAffinity`는 단순한 보조재로 본다.

### Q6. Headless Service(`clusterIP: None`)의 동작은?

Headless Service는 ClusterIP를 두지 않고 DNS만 제공한다. `nslookup my-svc.default.svc.cluster.local` 결과로 모든 ready Pod IP가 직접 반환된다. 클라이언트가 받은 IP 목록을 자체 로직으로 라우팅한다.

쓰임은 두 가지가 흔하다.

1. **StatefulSet의 개별 Pod 접근**(`pod-0.my-svc...`).
2. **클라이언트가 직접 로드 밸런싱을 하고 싶은 경우**(예: gRPC가 채널마다 모든 백엔드 IP를 알고 자체 LB).

VIP를 거치지 않으므로 kube-proxy의 부하 분산이 적용되지 않는다.

### Q7. Service port·targetPort·nodePort 세 종류 차이는?

| 필드 | 의미 | 누가 listen |
|------|------|-------------|
| `port` | Service 자체 포트 | kube-proxy가 ClusterIP에서 |
| `targetPort` | Pod의 실제 listen 포트 | 컨테이너가 |
| `nodePort` | 외부 노출 포트 | kube-proxy가 모든 노드에서 |

`port: 80, targetPort: 8080`은 "Service 80으로 들어온 트래픽을 Pod 8080으로 보낸다"의 매핑이다. 이름 충돌을 피하려고 Service port와 컨테이너 port가 다른 경우가 많다. NodePort 타입에서 `nodePort` 필드를 명시하지 않으면 30000~32767 범위에서 자동 할당된다.

### Q8. Service `topologyKeys`(Topology Aware Routing)는 어떤 차이를 만드는가?

Topology Aware Routing(TAR, K8s 1.21+ 알파/베타 → 안정화 진행 중)은 Service 트래픽을 같은 zone의 Pod로 우선 보낸다. zone 간 트래픽 비용이 큰 클라우드 환경에서 비용을 줄이고 latency를 낮춘다.

`service.kubernetes.io/topology-mode: Auto` 어노테이션을 달면 EndpointSlice에 zone hint가 포함되고, kube-proxy가 가능한 같은 zone의 endpoint로 라우팅한다. 단, zone 분포가 균형이 맞지 않으면 한 zone의 Pod에 과부하가 생긴다. K8s는 이 균형이 깨지면 hint를 자동 비활성화한다.

운영에서는 zone 비용과 분포 균형 사이의 trade-off를 명시적으로 본다. 트래픽이 균등하지 않은 워크로드에서는 TAR를 끄는 편이 안정적이다.
