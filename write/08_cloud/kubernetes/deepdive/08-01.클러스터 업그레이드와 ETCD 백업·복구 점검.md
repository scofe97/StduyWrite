# 클러스터 업그레이드와 ETCD 백업·복구 점검

> 본 장의 심화 점검 질문이다. 업그레이드와 복구 작업에서 순서, 버전 스큐, 스냅샷 운용 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> 드물지만 위험한 작업일수록 체크리스트 관점으로 다시 점검한다.

### Q1. kubeadm 업그레이드에서 가장 먼저 무엇을 확인해야 하는가?

`kubeadm upgrade plan`으로 현재 클러스터가 어떤 버전으로 올라갈 수 있는지, 지원되지 않는 스킵 업그레이드가 아닌지 먼저 본다. 업그레이드 자체보다 "지원되는 경로인지 확인하는 단계"가 더 중요하다.

### Q2. control plane을 왜 한 대씩 올려야 하는가?

공식 문서 기준으로 control plane은 순차 업그레이드가 원칙이다. 한 번에 여러 대를 건드리면 API 서버, addon 갱신, 인증서, static Pod 재기동이 겹치며 장애 반경이 커진다.

### Q3. etcd 스냅샷은 어디서 떠야 하는가?

리더 기준 스냅샷을 선호하는 편이 안전하다. 그리고 스냅샷을 남기는 것만으로 끝내지 말고, 복구 리허설과 보관 정책까지 함께 있어야 한다. "백업 파일은 있는데 실제로 복구한 적은 없다"가 가장 위험하다.

### Q4. etcd 복구를 기존 데이터 디렉터리 위에 바로 덮어써도 되는가?

권장되지 않는다. 복구는 새 클러스터 ID와 멤버십 기준으로 다시 여는 작업에 가깝다. 살아 있는 기존 멤버와 섞이면 더 큰 혼란을 만든다.
