# Ingress와 Gateway API 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> Ingress와 Gateway API를 실제 운영 판단 질문으로 다시 점검한다.

### Q1. Ingress Controller가 없으면 Ingress가 무용지물인 이유

Ingress 리소스는 선언이다. Controller가 없으면 API에 오브젝트가 저장될 뿐 아무 일도 일어나지 않는다.

```bash
kubectl create namespace ch19-smoke

# Controller 없이 Ingress 생성 시 ADDRESS 확인
kubectl apply -n ch19-smoke -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: test-ingress
spec:
  ingressClassName: nginx
  rules:
    - host: test.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: nginx
                port:
                  number: 80
EOF

# ADDRESS가 비어 있지 않아야 nginx-ingress가 동작 중
kubectl get ingress -n ch19-smoke
# NAME          CLASS   HOSTS              ADDRESS   PORTS   AGE
# test-ingress  nginx   test.example.com   <node-ip> 80      ...
```

심화 질문. Ingress Controller를 나중에 설치했을 때 기존 Ingress가 자동으로 인식되는가? Controller가 시작할 때 Kubernetes API에 List 요청을 보내 모든 기존 Ingress를 가져온다(List-Watch 패턴의 List 단계). 이후 Watch로 변경 사항을 감시한다. Controller 설치 후 기존 Ingress가 자동으로 반영되므로 재적용이 필요 없다.

### Q2. Gateway API 역할 분리 모델

```bash
# Gateway API CRD 설치 확인
kubectl get crd | grep gateway.networking.k8s.io

# 없으면 설치
# 버전 URL을 하드코딩하기보다 공식 릴리스 문서의 최신 standard channel 설치 경로를 확인하는 편이 안전하다.
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/<LATEST_VERSION>/standard-install.yaml

# GatewayClass, Gateway, HTTPRoute 생성
kubectl apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: nginx
spec:
  controllerName: k8s.nginx.org/nginx-gateway-controller
---
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: main-gateway
  namespace: infra
spec:
  gatewayClassName: nginx
  listeners:
    - name: http
      port: 80
      protocol: HTTP
      allowedRoutes:
        namespaces:
          from: All
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-route
  namespace: ch19-smoke
spec:
  parentRefs:
    - name: main-gateway
      namespace: infra
  hostnames:
    - "test.example.com"
  rules:
    - matches:
        - path:
            type: Prefix
            value: /
      backendRefs:
        - name: nginx
          port: 80
EOF
```

심화 질문. HTTPRoute가 존재하지 않는 Gateway를 `parentRefs`에서 참조하면 어떻게 되는가? HTTPRoute의 `status.parents[].conditions`에 `Accepted: False` 조건이 기록된다. `reason`에 "NoMatchingParent" 같은 원인이 명시된다. `kubectl describe httproute`로 확인할 수 있다.

추가 메모. 공식 Kubernetes 문서에서는 Gateway API를 Ingress의 후속 API로 안내하지만, 실제 사용 가능 기능은 설치한 Controller의 구현 범위에 따라 달라진다. 따라서 실습 전에 `kubectl get gatewayclass`와 구현체의 conformance 문서를 함께 확인하는 습관이 필요하다.

공식 문서 기준으로 Gateway는 기본적으로 같은 네임스페이스의 Route만 받는다. 다른 네임스페이스 Route를 연결하려면 `allowedRoutes`를 명시해야 한다. 따라서 "왜 내 HTTPRoute가 안 붙지?"라는 문제는 `parentRefs`뿐 아니라 Gateway listener의 `allowedRoutes`도 함께 봐야 한다.

### Q3. TLS 종단 위치 비교

```bash
# Ingress에서 TLS 종단 (Termination)
kubectl apply -n ch19-smoke -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: tls-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - secure.example.com
      secretName: tls-secret   # kubectl create secret tls로 생성
  rules:
    - host: secure.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-service
                port:
                  number: 80
EOF

# TLS Passthrough (Pod에서 종단)
kubectl apply -n ch19-smoke -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: passthrough-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-passthrough: "true"
spec:
  ingressClassName: nginx
  rules:
    - host: passthrough.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: tls-service
                port:
                  number: 443
EOF
```

TLS Passthrough는 L7 기능(경로 기반 라우팅, 헤더 조작)을 사용할 수 없다. Controller가 TLS 내용을 복호화하지 않으므로 SNI 기반 호스트 라우팅만 가능하다.

### Q4. PathType 동작 차이 확인

```bash
# PathType 별 매칭 테스트
kubectl apply -n ch19-smoke -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: pathtype-test
spec:
  ingressClassName: nginx
  rules:
    - host: pathtest.example.com
      http:
        paths:
          # Exact: /api에만 매칭
          - path: /api
            pathType: Exact
            backend:
              service:
                name: exact-svc
                port:
                  number: 80
          # Prefix: /api로 시작하는 모든 경로 매칭
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: prefix-svc
                port:
                  number: 80
EOF

# 테스트 (노드 IP로 직접 접근, Host 헤더 수동 설정)
NODE_IP=$(kubectl get nodes -o jsonpath='{.items[0].status.addresses[0].address}')
curl -H "Host: pathtest.example.com" http://$NODE_IP:31292/api
curl -H "Host: pathtest.example.com" http://$NODE_IP:31292/api/users
curl -H "Host: pathtest.example.com" http://$NODE_IP:31292/apiV2

kubectl delete namespace ch19-smoke
```

심화 질문. Ingress에서 `/api`를 Exact로, `/api`를 Prefix로 동시에 정의하면 어떤 것이 우선하는가? Exact가 항상 Prefix보다 우선한다. `/api`로 정확히 오는 요청은 Exact 규칙의 백엔드로 가고, `/api/users` 같은 하위 경로는 Prefix 규칙을 따른다.

### Q5. Gateway API 다중 프로토콜 지원

Ingress는 HTTP/HTTPS 전용이다. Gateway API는 프로토콜별 Route 리소스를 표준으로 정의한다.

```bash
# GRPCRoute 예시 (Gateway API v1.1+)
kubectl apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: GRPCRoute
metadata:
  name: grpc-route
  namespace: production
spec:
  parentRefs:
    - name: main-gateway
      namespace: infra
  hostnames:
    - "grpc.example.com"
  rules:
    - matches:
        - method:
            service: UserService
            method: GetUser
      backendRefs:
        - name: user-service
          port: 50051
EOF
```

TCPRoute와 UDPRoute는 아직 실험적(Experimental) 상태다. 사용 전에 설치한 Controller의 Conformance Test 통과 현황을 확인해야 한다.

심화 질문. TCPRoute는 어떤 기준으로 트래픽을 분기하는가? TCP 레벨에서는 HTTP 헤더나 경로 같은 L7 정보가 없다. TCPRoute는 포트 기반으로만 라우팅한다. 같은 포트에서 여러 TCP 서비스를 구분하려면 TLSRoute를 사용하고 SNI(Server Name Indication)로 분기한다.

### Q6. 여러 팀의 동일 호스트 충돌 해결

```bash
# team-a와 team-b가 같은 호스트/다른 경로 사용 (충돌 아님)
kubectl apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: team-a-route
  namespace: team-a
spec:
  parentRefs:
    - name: main-gateway
      namespace: infra
  hostnames:
    - "api.myapp.com"
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /users
      backendRefs:
        - name: user-service
          port: 80
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: team-b-route
  namespace: team-b
spec:
  parentRefs:
    - name: main-gateway
      namespace: infra
  hostnames:
    - "api.myapp.com"
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /orders
      backendRefs:
        - name: order-service
          port: 80
EOF

# 각 Route 상태 확인 (Accepted 조건 확인)
kubectl describe httproute team-a-route -n team-a
kubectl describe httproute team-b-route -n team-b
```

심화 질문. 같은 호스트+경로 충돌 시 어떤 Route가 우선하는가? Gateway API는 결정론적 우선순위 규칙을 따른다. 가장 긴 경로 → Exact가 Prefix보다 우선 → 생성 시각이 빠른 것 → 알파벳 순서(namespace/name). `status.parents[].conditions`에 "패배한" Route의 충돌 원인이 기록된다.

### Q7. dev 환경에서는 Ingress와 Gateway API를 어떤 순서로 가져가면 되나?

대부분의 dev 환경에서는 Ingress부터 시작하는 편이 현실적이다. 이미 `ingress-nginx`가 있고 NodePort로 외부 접근만 열려 있다면 `ingressClassName: nginx` 하나로 바로 검증할 수 있기 때문이다.

```bash
# 현재 ingress-nginx 동작 확인
kubectl get pods -n ingress-nginx
kubectl get svc -n ingress-nginx

# Host 헤더로 로컬 검증
NODE_IP=<worker-node-ip>
curl -H "Host: app.dev.local" http://$NODE_IP:31292/
```

Gateway API는 "대체재"라기보다 "다음 단계 구조화"에 가깝다. CRD 설치 여부, `GatewayClass` 존재, 실제 Controller 지원 범위까지 확인해야 하므로 초기 dev 세팅으로는 비용이 더 든다.

```bash
# Gateway API 최소 확인 루틴
kubectl get crd | grep gateway.networking.k8s.io
kubectl get gatewayclass
kubectl get gateways -A
kubectl get httproutes -A
```

판단 기준은 단순하다. 단일 팀과 단일 ingress-nginx면 Ingress로 충분하다. 여러 팀이 하나의 진입점을 공유하고 `Gateway`와 `HTTPRoute`의 소유권을 분리해야 하면 Gateway API를 검토한다.
