# 스토리지와 상태 점검

> 본 장의 심화 점검 질문입니다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했습니다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검합니다.

### Q1. RWO, RWX, RWOP는 무엇이 다른가?

`RWO`는 한 시점에 하나의 노드에서 읽기/쓰기로 마운트하는 패턴입니다. 대부분의 블록 스토리지 기반 DB에서 기본값처럼 사용됩니다. Pod가 재스케줄되면 같은 볼륨이 다른 노드에 다시 붙을 수는 있지만, 동시에 여러 노드에서 쓰는 형태는 아닙니다.

`RWX`는 여러 노드와 여러 Pod가 동시에 읽기/쓰기를 할 수 있는 패턴입니다. NFS, CephFS, 일부 파일 스토리지가 여기에 해당합니다. 여러 애플리케이션 인스턴스가 같은 파일 집합을 공유해야 할 때 적합합니다.

`RWOP`는 더 강한 단일 writer 제약입니다. 한 노드가 아니라 정확히 하나의 Pod만 읽기/쓰기를 수행하게 강제합니다. 같은 노드 안의 다른 Pod까지 차단하고 싶을 때 의미가 있습니다.

운영 판단 기준은 단순합니다. 공유 파일 시스템이면 `RWX`, 단일 인스턴스 DB면 `RWO` 또는 `RWOP`, 쓰기 충돌을 확실히 막고 싶으면 `RWOP`를 먼저 검토합니다.

### Q2. emptyDir vs hostPath vs PVC 선택 기준은?

선택 기준은 데이터 수명주기와 노드 이동성 두 가지입니다.

emptyDir은 Pod 생명주기와 같습니다. Pod가 삭제되면 데이터가 사라집니다(컨테이너 재시작 시에는 유지됩니다). 같은 Pod 내 컨테이너 간 임시 데이터 공유에 적합합니다. 로그 수집 파이프라인의 버퍼, 임시 캐시가 대표적입니다.

hostPath는 노드 파일시스템에 고정됩니다. Pod가 삭제되어도 데이터는 유지되지만, 다른 노드로 이동하면 접근할 수 없습니다. 노드의 `/proc`, `/sys` 접근이나 시스템 모니터링 에이전트에 사용하며 보안 위험이 높아 프로덕션에서는 신중해야 합니다.

PVC는 Pod와 노드에서 분리된 독립적 생명주기를 가집니다. ReadWriteMany 또는 재마운트를 통해 노드 이동이 가능합니다. 애플리케이션 데이터베이스, 공유 미디어 파일에 사용합니다.

emptyDir의 sizeLimit을 초과하면 Pod Eviction이 트리거됩니다. ReadWriteOnce PVC를 사용하는 Pod가 다른 노드로 이동할 때는 볼륨 재마운트에 수 분이 걸릴 수 있습니다.

### Q3. 동적 프로비저닝의 동작 과정은?

PVC를 생성하면 kube-controller-manager가 이를 감지하고 StorageClass의 provisioner를 확인합니다. 외부 프로비저너(CSI 드라이버)에게 요청을 보내면 클라우드 API를 호출해 볼륨을 생성하고 PV 오브젝트를 자동으로 만들어 PVC와 바인딩합니다.

실패 가능 지점은 네 곳입니다. 클라우드 API 할당량 초과, 가용 영역 불일치(노드는 us-east-1a인데 볼륨은 us-east-1b에 생성되는 경우), StorageClass의 프로비저너가 클러스터에 설치되지 않은 경우, IAM 권한 부족입니다.

```bash
kubectl describe pvc my-pvc      # Events 섹션에서 오류 메시지 확인
kubectl get storageclass -o yaml
```

PVC가 Pending 상태로 멈추면 타임아웃 없이 무한 대기하므로 수동으로 삭제해야 합니다. Immediate 모드에서 생성된 볼륨은 다른 가용 영역의 노드에서 사용할 수 없으므로 PVC를 재생성해야 합니다.

### Q4. reclaimPolicy의 운영 영향은?

`Delete`는 PVC 삭제 시 PV와 클라우드 스토리지를 함께 삭제합니다. 개발·테스트 환경에 적합합니다. 실수로 삭제하면 클라우드 스냅샷이 없으면 복구할 수 없습니다.

`Retain`은 PVC 삭제 후 PV가 Released 상태로 유지되고 클라우드 스토리지도 보존됩니다. 수동으로 PV를 삭제하거나 재사용할 수 있습니다. 프로덕션 DB에 권장됩니다.

Retain 정책의 수동 정리 과정은 PVC 삭제 → PV 상태 확인(Released) → PV 삭제(스토리지는 유지) → 클라우드 콘솔에서 볼륨 수동 삭제 순서입니다.

StatefulSet의 `volumeClaimTemplates`로 생성된 PVC는 StatefulSet 삭제 시 자동 삭제되지 않습니다. orphan PVC를 방지하는 설계이므로 수동으로 삭제해야 합니다.

### Q5. StatefulSet vs Deployment 선택 기준은?

StatefulSet이 필수인 경우는 각 인스턴스가 고유한 영구 스토리지가 필요하거나(Kafka 브로커), 안정적인 네트워크 ID가 필요하거나(Zookeeper 클러스터 멤버십), 순서 보장이 필요하거나(MySQL 마스터-슬레이브), Peer discovery가 필요한(Cassandra gossip 프로토콜) 경우입니다.

Deployment로 충분한 경우는 상태를 외부에 저장하는 경우(세션을 Redis에 저장하는 웹 서버), 모든 인스턴스가 동일한 경우(REST API 서버), 빠른 스케일링이 필요한 경우입니다.

`podManagementPolicy: Parallel`로 설정하면 순서는 무시되지만 고유 네트워크 ID는 유지됩니다. StatefulSet의 replicas를 0으로 줄여도 PVC는 유지되어 스케일 업 시 재연결됩니다.

### Q6. StatefulSet의 Pod 순서 보장이 필요한 이유는?

MySQL 복제에서 마스터(mysql-0)가 먼저 초기화되어야 슬레이브(mysql-1, mysql-2)가 복제를 시작할 수 있습니다. Kafka 브로커는 ID와 Pod 인덱스를 매핑하고, Elasticsearch는 마스터 노드가 먼저 클러스터를 형성해야 데이터 노드가 합류할 수 있습니다.

순서를 어기면 데이터 불일치, 스플릿 브레인, 복제 실패 같은 문제가 발생합니다. Zookeeper에서는 동시 시작 시 리더 선출이 실패할 수 있습니다.

Init Container로 의존성을 구현합니다. mysql-1이 시작될 때 `mysql-0.mysql-headless`가 DNS에서 해석될 때까지 대기하는 방식입니다. Readiness Probe가 실패하면 다음 Pod 생성이 무기한 블록되므로 수동 개입이 필요합니다.

### Q7. volumeClaimTemplates와 일반 PVC의 차이는?

일반 PVC를 StatefulSet에 직접 사용하면 모든 레플리카가 하나의 PVC를 공유해 데이터 충돌이 생깁니다. ReadWriteOnce 볼륨은 하나의 노드만 마운트할 수 있어 스케줄링 제약도 발생합니다.

`volumeClaimTemplates`는 각 Pod마다 고유한 PVC를 자동으로 생성합니다. PVC 이름은 `{template 이름}-{StatefulSet 이름}-{순서번호}` 형식입니다. 스케일 다운해도 PVC는 유지되고, 스케일 업 시 기존 PVC에 재연결되어 데이터가 복구됩니다.

`volumeClaimTemplates`의 storageClassName을 변경해도 기존 PVC는 영향을 받지 않고 새 Pod만 새 StorageClass를 사용합니다.

### Q8. NFS는 언제 고려할 만한가?

여러 Pod가 동시에 같은 디렉터리를 읽고 써야 할 때 NFS를 고려합니다. 정적 업로드 파일, 공용 설정 산출물, 레거시 애플리케이션의 공유 파일 시스템이 대표적입니다. RWX 요구가 핵심 신호입니다.

다만 Pod 스펙에 직접 `nfs:`를 적는 방식보다 PV/PVC를 통한 추상화가 더 낫습니다. 운영에서는 마운트 옵션, 재사용성, 권한 정책, 동적 프로비저닝을 고려해야 하므로 NFS 위에 PV/PVC/StorageClass를 올리는 편이 보통 더 안전합니다.

반대로 데이터베이스처럼 지연과 일관성이 민감한 워크로드에는 NFS가 기본 선택이 아닙니다. 이런 경우는 블록 스토리지 기반 `RWO` 또는 `RWOP`가 더 자연스럽습니다.

### Q9. GCP 클러스터에서 스토리지 프로비저너 동작 방식은?

kubeadm으로 구성한 개인 GCP 클러스터에는 클라우드 관리형 CSI 드라이버가 기본 설치되어 있지 않습니다. 사용 가능한 StorageClass를 먼저 확인해야 합니다.

```bash
kubectl get storageclass
kubectl get pvc -A   # 기존 PVC 상태 확인
```

로컬 스토리지 기반의 StorageClass가 있으면 hostPath 방식으로 동작합니다. 이 경우 데이터는 해당 노드의 특정 디렉토리에 저장되므로 Pod가 다른 노드로 이동하면 데이터에 접근할 수 없습니다.

GCP Persistent Disk를 사용하려면 GCE PD CSI 드라이버를 별도로 설치해야 합니다. 개인 학습 환경에서는 로컬 스토리지로 StatefulSet의 순서 보장과 PVC 재연결 동작을 검증하는 것으로 충분합니다.

```bash
# StatefulSet 배포 후 상태 확인
kubectl get statefulset
kubectl get pods -l app=web
kubectl get pvc

# Pod 삭제 후 재생성 확인
kubectl delete pod web-1
kubectl get pods -w  # web-1이 다시 생성되는지 확인

# PVC 재연결 확인 (데이터 유지)
kubectl exec web-1 -- cat /usr/share/nginx/html/index.html
```
