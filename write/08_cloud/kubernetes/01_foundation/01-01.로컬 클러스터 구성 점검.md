# 로컬 클러스터 구성 점검

> 본 장의 심화 점검 질문입니다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했습니다.



## 점검 질문
> 본문 개념을 운영 판단 질문으로 다시 점검합니다.

### Q1. minikube · kind · k3d 선택 기준은?

학습과 개발이 주 목적이면 minikube입니다. `minikube addons enable ingress` 한 줄로 NGINX Ingress Controller가 뜨고 Dashboard·Metrics Server까지 즉시 활성화됩니다. 인프라 설정에 쓰는 시간을 줄여 K8s 개념에 집중할 수 있습니다.

CI/CD에는 kind가 맞습니다. Docker 컨테이너 안에 클러스터를 만들어 시작 속도가 빠르고, GitHub Actions·GitLab CI에서 매트릭스 테스트를 구성하기 쉽습니다. `kind create cluster --config kind-config.yaml`로 멀티 노드와 HA Control Plane을 한 번에 만들 수 있습니다. kubeadm 기반이라 프로덕션 부트스트랩과 가장 유사하다는 점도 장점입니다.

리소스 제약이 강하면 k3d입니다. k3s 기반이라 512MB 이하로도 동작하고, Raspberry Pi 같은 엣지 환경이나 동시 10클러스터 이상을 띄우는 멀티테넌시 실험에 유리합니다.

심화 질문입니다. Docker Desktop 없이 실행해야 하는 VM·원격 서버에서는 어떤 도구일까요? minikube는 QEMU/KVM 드라이버로 Docker 없이 동작하지만 kind와 k3d는 Docker를 필수로 합니다. CI에서 kind 이미지 로딩을 줄이려면 `kind load docker-image <image>`로 로컬 이미지를 직접 주입하거나 Registry 컨테이너를 클러스터와 네트워크를 공유시킵니다.

### Q2. minikube 드라이버 차이는?

Docker 드라이버는 세 OS 모두에서 기본으로 선택되는 옵션입니다. `docker ps`로 클러스터 컨테이너가 보이고 시작도 30초 안쪽입니다. 단점은 Docker Desktop 자체의 무게(메모리 2GB 이상)와 Nested Virtualization 미지원입니다.

Hyperkit은 macOS 전용 경량 하이퍼바이저였으나 minikube 1.30 이후 Deprecated 상태입니다. QEMU로 넘어가는 게 안전합니다.

QEMU는 크로스 플랫폼 옵션입니다. macOS(Apple Silicon)에서 Docker Desktop의 x86 에뮬레이션 성능 문제가 있을 때 QEMU로 전환하면 ARM64 네이티브 속도를 냅니다. Nested Virtualization도 지원해 EC2·GCE 같은 VM 안에서 minikube를 돌릴 때 필수입니다.

VirtualBox는 라이선스 이슈와 시작 속도, macOS 최신 버전 지원 불안정 때문에 지금은 권장되지 않습니다.

심화 질문입니다. Apple Silicon Mac에서 성능 저하의 원인은 x86 에뮬레이션입니다. ARM64 네이티브 이미지를 쓰거나 드라이버를 QEMU로 바꾸면 해결됩니다. VM 내부에서 다시 K8s를 돌리는 시나리오(KubeVirt 포함)는 Nested Virtualization을 지원하는 QEMU·KVM 조합이 맞습니다.

### Q3. kubectl context와 kubeconfig 구조는?

kubeconfig는 기본 경로 `~/.kube/config`에 저장되고 세 블록으로 이뤄집니다. `clusters`는 API 서버 주소, `users`는 인증 정보, `contexts`는 앞 둘과 기본 네임스페이스를 묶은 조합입니다. `current-context`가 지금 유효한 조합을 가리킵니다.

컨텍스트 전환은 `kubectl config use-context <name>`이고, 네임스페이스 기본값은 `kubectl config set-context --current --namespace=<name>`으로 고정합니다. 여러 kubeconfig가 있을 때는 `KUBECONFIG=~/.kube/config:~/.kube/config-prod`처럼 환경변수로 병합해 씁니다. minikube·GKE·EKS·온프레미스가 한 파일에 모이는 구조도 자연스럽습니다.

심화 질문입니다. kubeconfig 유출은 cluster-admin 인증서가 포함됐다면 클러스터 전체 탈취로 이어집니다. git 커밋 금지가 기본이고, 프로덕션은 OIDC(Google·Azure AD)나 IAM Role(AWS) 기반으로 만료를 1시간 단위로 짧게 둡니다. `kubectl config view`에 비밀번호가 안 보이는 이유는 민감 필드가 `<REDACTED>`로 마스킹되기 때문입니다. 원본이 필요하면 `--raw`를 쓰되, 로그나 공개 채널에 흘리면 안 됩니다. 팀이 한 클러스터를 공유할 때는 네임스페이스로 물리 분리하고 RBAC로 상호 접근을 제한합니다.

### Q4. 로컬 클러스터와 프로덕션의 경계는?

노드 장애 재현은 로컬에서 한계가 명확합니다. 단일 노드가 기본이고, `--nodes=3`으로 늘려도 실제 네트워크 파티션이나 노드 간 지연은 흉내 내기 어렵습니다. 스토리지 내구성도 마찬가지로, 호스트 경로 기반 PV는 클러스터 삭제와 함께 사라져 Zone 장애 검증에는 쓸 수 없습니다.

LoadBalancer 타입 Service는 로컬에서 `minikube tunnel`이나 MetalLB로 대체해야 합니다. 이는 EC2 ELB·GCP LB의 실제 동작과 완전히 일치하지는 않습니다. NetworkPolicy와 Pod Security Standards는 로컬에서도 적용되지만, 실제 DMZ·방화벽·VPC 피어링 같은 클라우드 네트워크 토폴로지는 재현 범위 밖입니다.

심화 질문입니다. 멀티 노드 시나리오는 kind 3노드 + `kubectl drain`·Chaos Mesh 조합으로 근사해 볼 수 있습니다. StatefulSet 내구성은 Longhorn·OpenEBS를 올려 스냅샷·복구 흐름을 봅니다(replication 효과는 단일 노드에서는 없습니다). 로컬에서 프로덕션 배포 전에 검증 가능한 항목은 매니페스트 문법, RollingUpdate 전략, ConfigMap/Secret 주입, Requests/Limits, Liveness/Readiness Probe, HPA 동작까지입니다.

### Q5. 애드온과 프로덕션 대응 컴포넌트는?

Ingress Controller는 외부 HTTP(S) 트래픽을 Service로 분배합니다. minikube 애드온은 NGINX Ingress를 올리고, 프로덕션은 NGINX·Traefik·Istio Gateway·ALB Ingress Controller 중에서 고릅니다. cert-manager로 TLS 자동 갱신, WAF 통합이 따라붙는 게 프로덕션의 차이입니다.

Metrics Server는 단기 리소스 메트릭만 담아 `kubectl top`과 HPA 소스로 동작합니다. 장기 보관과 복합 쿼리(95퍼센타일 레이턴시 같은)는 Prometheus + Grafana 조합이 담당합니다.

Dashboard는 프로덕션에서 보안 문제로 대부분 비활성화합니다. 대신 Lens(데스크톱), k9s(TUI), Grafana 대시보드가 자리를 잡았습니다. Storage Provisioner는 PVC를 PV로 자동 연결하는데, 로컬은 호스트 경로 기반이고 프로덕션은 CSI 드라이버(EBS CSI·GCE PD CSI)가 클라우드 스토리지를 동적으로 만듭니다.

심화 질문입니다. Ingress 없이 외부 트래픽을 받으려면 NodePort나 `kubectl port-forward`지만, 도메인 기반 라우팅은 Ingress 없이 불가능합니다. metrics-server와 Prometheus는 보관 기간과 쿼리 표현력에서 갈립니다. minikube storage-provisioner는 VM 내부 `/tmp/hostpath-provisioner/<pvc>/`에 디렉토리를 만들고 PV로 바인딩하는 구조라 `minikube ssh`로 실물 확인이 가능합니다.

### Q6. 리소스 관리 전략은?

기본 할당(CPU 2·메모리 2GB)은 Ingress와 Monitoring을 함께 올리기에는 부족합니다. 16GB 노트북이면 minikube에 8GB 정도가 실용선입니다. 전체를 넘기면 호스트가 버벅여 오히려 실습이 느려집니다. 디스크는 초기부터 40GB로 잡거나 `minikube ssh && docker system prune -a`로 미사용 이미지를 비웁니다.

여러 프로필을 병행할 때는 쓰지 않는 프로필을 `minikube stop`으로 내려 둡니다. 애드온은 실습에 필요한 것만 켭니다. Dashboard는 상시 활성화보다 필요할 때 `minikube dashboard`로 임시 실행이 낫습니다.

심화 질문입니다. 메모리 압박 Eviction은 `kubectl describe node`의 `Conditions`에 `MemoryPressure=True`로 나타납니다. 증설은 `minikube stop && minikube start --memory=12288` 순서입니다. 이미지 캐싱은 `minikube cache add <image>`로 선반영해 두면 재시작·프로필 간에 보존됩니다. 8GB 노트북처럼 리소스가 빠듯하면 k3d로 넘어가거나 GitHub Codespaces 같은 원격 환경으로 이동하는 쪽이 편합니다.
