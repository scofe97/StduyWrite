# TLS와 API 접근 보안

> Kubernetes 보안의 출발점은 API 서버와의 통신이 어떻게 보호되고 인증되는지 이해하는 것이다. kubeconfig, 인증서, API 서버 CA는 각각 다른 층위의 개념이다. 본 장은 컨트롤 플레인 PKI(API 서버·etcd·kubelet 인증서)에 집중하고, 외부 트래픽 인증서 자동화(cert-manager·Issuer·Certificate)는 [04-06 Ingress와 Gateway API §3](./04-06.Ingress와%20Gateway%20API.md#3-tls-관리)에서 다룬다.



## 학습 목표
> RBAC 이전에 "누구로 접속했고, 무엇을 신뢰하는가"를 먼저 분리한다.

이 장에서 확인할 목표는 다음과 같다:

1. API 서버와 etcd, kubelet이 TLS로 어떻게 연결되는지 설명할 수 있다.
2. 서버 인증서와 클라이언트 인증서의 역할 차이를 설명할 수 있다.
3. kubeconfig의 `cluster`, `user`, `context`가 보안상 어떤 의미를 가지는지 설명할 수 있다.
4. 신뢰할 수 없는 kubeconfig를 왜 조심해야 하는지 설명할 수 있다.



## 1. Kubernetes의 PKI 구조
> Kubernetes는 여러 컴포넌트가 서로 TLS로 통신하는 시스템이다.

공식 문서 기준으로 Kubernetes는 API 서버 엔드포인트, etcd 서버, kubelet 서버, 그리고 각 컴포넌트의 클라이언트 통신에 PKI 인증서를 사용한다. 즉 TLS는 "외부 사용자의 kubectl 접속"만을 위한 것이 아니라, control plane 내부 통신 전체의 기반이다.



## 2. kubeconfig와 신뢰 경계
> kubeconfig는 접속 설정 파일이면서 동시에 인증 실행 경로다.

kubeconfig에는 보통 API 서버 주소, CA 정보, 사용자 인증 방식, context가 들어 있다. 여기서 중요한 경고는 공식 문서에도 나온다. 신뢰할 수 없는 kubeconfig 파일은 악성 코드 실행이나 파일 노출로 이어질 수 있으므로, 셸 스크립트처럼 다뤄야 한다.

```bash
kubectl config get-contexts
kubectl config current-context
kubectl config view --minify
```

즉 kubeconfig는 "어느 클러스터를 볼 것인가"를 정하는 편의 파일이 아니라, 실제 인증과 신뢰 체인의 일부다.



## 3. 실무 포인트
> TLS는 평소엔 안 보이지만, 인증서 만료나 API 접속 실패 때 바로 드러난다.

- 인증서 만료는 업그레이드/운영 작업과 함께 관리해야 한다.
- kubeconfig는 trusted source만 사용한다.
- RBAC은 권한, TLS는 신뢰와 인증 경로라는 점을 분리해서 본다.
