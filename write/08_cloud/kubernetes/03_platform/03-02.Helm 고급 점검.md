# Helm 고급 점검

> 본 장의 심화 점검 질문입니다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했습니다.



## 점검 질문
> Helm 차트 설계와 검증 포인트를 운영 질문으로 다시 점검합니다.

### Q1. _helpers.tpl에서 define vs template vs include 차이는?

`define`은 Named Template을 선언합니다. 이름과 내용을 등록하지만 즉시 실행하지는 않습니다. `_helpers.tpl`에서 재사용 가능한 조각을 만들 때 사용합니다.

`template`은 정의된 Named Template을 호출하는 Go 템플릿의 기본 함수입니다. 파이프라인을 지원하지 않아 들여쓰기 제어가 불가능합니다. YAML은 들여쓰기에 민감하므로 이 방식은 구문 오류로 이어지기 쉽습니다.

`include`는 Helm이 추가한 함수로, Named Template을 호출하되 결과를 문자열로 반환해 파이프라인에서 사용할 수 있습니다. `nindent`와 결합하면 들여쓰기를 정확히 제어할 수 있습니다.

```yaml
# _helpers.tpl
{{- define "myapp.labels" -}}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end }}

# deployment.yaml - include 사용 (올바른 방식)
metadata:
  labels:
    {{- include "myapp.labels" . | nindent 4 }}

# template 사용 시 (YAML 오류 발생)
metadata:
  labels:
{{ template "myapp.labels" . }}   # 들여쓰기 0칸
```

Helm 커뮤니티의 베스트 프랙티스는 모든 Named Template 호출에 `include` + `nindent`를 조합하는 것입니다. `helm create`로 생성되는 스캐폴딩도 이 방식을 기본으로 사용합니다.

심화: `define` 양쪽의 `-` 기호(`{{- define ... -}}`)는 앞뒤 공백과 줄바꿈을 제거해 렌더링 결과를 깔끔하게 유지합니다. `include`의 두 번째 인자 `.`는 현재 컨텍스트를 전달하며, 특정 값만 전달하려면 `.Values.app`처럼 좁혀서 넘길 수 있습니다.

### Q2. values.yaml 설계 원칙(flat vs nested, 기본값 전략)은?

flat 구조는 모든 설정을 최상위에 나열해 오버라이드가 단순(`--set imageTag=1.26.0`)하지만 관련 설정 파악이 어렵고 이름 충돌 가능성이 있습니다.

nested 구조는 관련 설정을 객체로 그룹화합니다. 서브차트와 이름 충돌이 없고 확장이 용이합니다. Bitnami와 Helm 공식 차트가 따르는 방식입니다.

```yaml
# nested 구조 (권장)
image:
  repository: nginx
  tag: ""           # 비어있으면 Chart.appVersion 사용
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 80

ingress:
  enabled: false    # 선택적 기능은 비활성화 기본값

replicaCount: 1     # 자주 변경되는 값은 최상위 배치
```

기본값 전략은 네 가지입니다. 대부분의 경우 수정 없이 사용 가능하도록 합리적인 기본값을 제공하고, 선택적 기능은 비활성화 기본값을 둡니다. `image.tag`를 비워두고 템플릿에서 `{{ .Values.image.tag | default .Chart.AppVersion }}`으로 폴백하면 appVersion 관리와 연동됩니다. 각 필드에 주석으로 의미와 예시를 기술합니다.

환경별로는 values.yaml을 개발 환경 기준으로 유지하고 프로덕션은 별도 values-prod.yaml로 오버라이드합니다.

### Q3. Helm Hook의 weight와 delete-policy 동작은?

`helm.sh/hook-weight`는 Hook 실행 순서를 결정합니다. 숫자가 낮을수록 먼저 실행되고 기본값은 0입니다. 동일 weight는 알파벳 순으로 처리됩니다.

```yaml
# 1단계: DB 백업 (weight: -5, pre-upgrade)
# 2단계: 앱 중지  (weight:  0, pre-upgrade)
# Helm 업그레이드 실행
# 3단계: DB 마이그레이션 (weight: 5, post-upgrade)
```

`helm.sh/hook-delete-policy`는 Hook 리소스를 언제 삭제할지 지정합니다. 정책이 없으면 Hook은 클러스터에 영구 보존됩니다.

| 정책 | 동작 시점 |
|------|----------|
| `before-hook-creation` | 새 Hook 실행 전 이전 Hook 삭제 (가장 많이 사용) |
| `hook-succeeded` | 성공 시 삭제 |
| `hook-failed` | 실패 시 삭제 (디버깅 어려워 비추천) |

`before-hook-creation`이 핵심입니다. 업그레이드 실패 후 재시도할 때 Job 이름 충돌을 방지하고 이전 Hook을 자동 정리합니다. 여러 정책은 쉼표로 조합할 수 있다(`before-hook-creation,hook-succeeded`).

심화: Hook이 실패하면 릴리스 전체가 failed 상태가 되지만 이미 생성된 리소스는 자동 롤백되지 않습니다. `--atomic` 플래그를 사용하면 자동 롤백이 됩니다. Job 무한 대기 방지는 `.spec.activeDeadlineSeconds`와 `helm upgrade --timeout`으로 이중 설정합니다.

### Q4. 차트 의존성의 condition과 alias 활용 시나리오는?

`condition`은 values.yaml의 특정 키를 참조해 서브차트 활성화 여부를 결정합니다. 여러 조건을 쉼표로 연결하면 OR 연산으로 동작합니다.

`alias`는 동일한 차트를 여러 번 포함하거나 values.yaml에서 더 명확한 이름을 쓸 때 사용합니다. alias를 설정하면 리소스 이름 생성 시 Chart.Name 대신 alias가 사용되어 충돌을 방지합니다.

실전 시나리오인 환경별 DB 분리를 보자. 개발 환경은 차트 내 PostgreSQL을 사용하고, 프로덕션은 외부 관리형 DB(RDS 등)를 사용합니다.

```yaml
# Chart.yaml
dependencies:
  - name: postgresql
    version: 12.5.0
    repository: https://charts.bitnami.com/bitnami
    condition: postgresql.enabled

# values-dev.yaml
postgresql:
  enabled: true
  primary:
    persistence:
      enabled: false

# values-prod.yaml
postgresql:
  enabled: false
externalDatabase:
  enabled: true
  host: prod-db.us-west-2.rds.amazonaws.com
```

deployment.yaml에서 `{{ if .Values.postgresql.enabled }}`로 조건 분기해 연결 정보를 동적으로 구성합니다.

`tags`는 여러 서브차트를 그룹으로 제어할 때 쓰지만, condition이 tags보다 우선순위가 높습니다.

### Q5. helm template으로 렌더링 결과 검증하는 워크플로우는?

`helm template`은 클러스터 접근 없이 로컬에서 렌더링하므로 kubeconfig 없이도 실행 가능합니다. CI/CD에서 PR 시점에 렌더링 결과를 자동 검증하는 데 적합합니다.

```bash
# 기본 렌더링
helm template myapp ./chart -f values-prod.yaml > rendered.yaml

# 특정 템플릿만 렌더링
helm template myapp ./chart -s templates/deployment.yaml

# 변경 전후 비교
helm template myapp ./chart -f values-old.yaml > old.yaml
helm template myapp ./chart -f values-new.yaml > new.yaml
diff -u old.yaml new.yaml
```

GitOps와의 통합은 두 방식이 있습니다. ArgoCD가 직접 차트를 렌더링하는 방식은 Git이 가볍고 차트 업데이트가 쉽습니다. 렌더링 결과를 Git에 커밋하는 방식은 diff 리뷰가 쉽지만 저장소가 커집니다. 하이브리드 접근은 차트는 직접 사용하되 PR에서만 렌더링 결과를 코멘트로 추가합니다.

환경별 검증 자동화는 다음처럼 구성합니다.

```bash
for env in dev staging prod; do
  helm template myapp ./chart -f values-$env.yaml | kubeconform -strict
done
```

### Q6. Library Chart vs Application Chart 차이는?

Application Chart는 Deployment, Service 등 배포 가능한 리소스를 포함하며 `helm install`로 설치합니다. 차트의 기본 형태입니다.

Library Chart는 `type: library`로 선언하며 Named Template만 제공합니다. `templates/` 디렉토리에 `_*.tpl` 파일만 있고, `helm install`로 직접 설치할 수 없습니다. 다른 차트의 `dependencies`로 포함해 함수를 재사용합니다.

Library Chart가 유용한 시나리오는 세 가지입니다. 조직 표준 라벨/어노테이션을 모든 차트에서 일관되게 적용할 때, 이름 생성 함수나 리소스 검증 로직을 중앙 관리할 때, 기업 보안 정책(Security Context, Network Policy 템플릿)을 표준화할 때입니다.

_helpers.tpl을 차트마다 복사하는 방식과 비교하면 Library Chart는 버전만 올리면 모든 의존 차트가 최신 함수를 사용할 수 있습니다. Bitnami의 `common` 차트가 이 패턴의 대표적 사례입니다.

심화: Library Chart의 템플릿 이름이 Application Chart와 충돌하면 나중에 정의된 쪽이 우선합니다. 충돌 방지를 위해 `common.labels`처럼 네임스페이스를 이름에 포함합니다.

### Q7. 차트 버전 관리 전략(SemVer, appVersion vs version)은?

`version`은 Helm 차트 자체의 버전으로 템플릿, values.yaml, Chart.yaml 변경 시 증가합니다. `appVersion`은 배포하는 애플리케이션의 버전으로 컨테이너 이미지 태그와 대응됩니다.

SemVer 원칙을 적용하면 다음과 같습니다.

| 버전 | 증가 조건 | 예시 |
|------|----------|------|
| MAJOR | 호환되지 않는 변경 | values 키 이름 변경, 필수 values 추가 |
| MINOR | 하위 호환 기능 추가 | 새 템플릿 추가, 선택적 values 추가 |
| PATCH | 버그 수정, 문서 업데이트 | 템플릿 오타 수정, 기본값 조정 |

appVersion은 차트 버전과 독립적으로 관리됩니다. 앱 이미지만 교체하면 appVersion만 올리고 차트 버전은 PATCH 증가로 반영합니다. 템플릿에서 `{{ .Values.image.tag | default .Chart.AppVersion }}`을 사용하면 values에서 tag를 비워도 appVersion이 기본값으로 사용됩니다.

의존성 버전 범위는 `~1.2.0`(PATCH만 허용), `^1.2.0`(MINOR, PATCH 허용), `>=1.2.0 <2.0.0`(명시적 범위)로 제약할 수 있습니다.
