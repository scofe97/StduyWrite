# Kustomize

> Kustomize는 템플릿 없이 순수 YAML 위에서 환경별 차이를 조합하는 도구다. Helm이 패키지와 버전 관리에 강하다면, Kustomize는 같은 매니페스트를 dev·staging·prod로 안전하게 변형하는 데 강하다.



## 학습 목표
> Helm과 Kustomize를 경쟁 도구가 아니라 다른 계층의 도구로 구분한다.

이 장에서 확인할 목표는 다음과 같다:

1. Kustomize가 템플릿 없이 오브젝트를 조합하는 방식을 설명할 수 있다.
2. `base`와 `overlay` 구조를 통해 환경별 차이를 어떻게 관리하는지 이해할 수 있다.
3. `kubectl apply -k` 흐름과 `kubectl kustomize` 미리보기 사용법을 설명할 수 있다.
4. ConfigMap/Secret generator가 갖는 장점과 주의점을 정리할 수 있다.
5. Helm, Kustomize, ArgoCD를 어떤 순서와 역할로 조합할지 설명할 수 있다.



## 1. 왜 Kustomize인가
> 값 치환보다 "원본 유지 + 차이만 선언"이 중요할 때 Kustomize가 빛난다.

Helm은 강력하지만 템플릿 로직이 복잡해질수록 차트 읽기가 어려워진다. 반대로 Kustomize는 원본 매니페스트를 그대로 두고, 이름 접두사, 레이블, 이미지 태그, 패치만 덧씌운다. 그래서 "같은 앱을 dev와 prod에 배포하되 replica 수와 도메인만 다르다" 같은 문제에 특히 잘 맞는다.

공식 문서 기준으로 Kustomize는 `kubectl`에 내장되어 있어 추가 도구 설치 없이 `kubectl apply -k`로 사용할 수 있다. 즉 CI/CD와 GitOps에서 별도 렌더링 서버 없이도 바로 적용 가능하다.



## 2. base와 overlay
> Kustomize의 핵심 구조는 원본과 환경별 차이를 분리하는 것이다.

```text
my-app/
├── base/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
└── overlays/
    ├── dev/
    │   └── kustomization.yaml
    └── prod/
        └── kustomization.yaml
```

`base`는 공통 원본이고, `overlay`는 환경별 차이다. 예를 들어 `prod` overlay는 replica 수를 늘리고, 리소스 제한을 강화하고, Ingress 호스트를 교체할 수 있다.

```yaml
# overlays/prod/kustomization.yaml
resources:
  - ../../base

images:
  - name: my-app
    newTag: "1.2.3"

replicas:
  - name: my-app
    count: 3

commonLabels:
  env: prod
```



## 3. 자주 쓰는 기능
> 실제로 많이 쓰는 것은 이미지 교체, 패치, 공통 라벨, generator 네 가지다.

| 기능 | 용도 |
|------|------|
| `resources` | 다른 YAML이나 base 포함 |
| `images` | 이미지 태그/레포지토리 치환 |
| `patches` | 특정 필드만 덮어쓰기 |
| `commonLabels` / `commonAnnotations` | 공통 메타데이터 주입 |
| `configMapGenerator` / `secretGenerator` | 외부 파일에서 설정 리소스 생성 |

```yaml
configMapGenerator:
  - name: my-app-config
    files:
      - application.properties
```

generator는 편리하지만 이름에 해시가 붙을 수 있다. 이 해시는 설정 변경이 rollout을 유도하는 데 유리하지만, 이름이 고정돼야 하는 참조 구조에서는 의도와 다를 수 있다. 그래서 팀마다 해시 사용 여부를 먼저 정해 두는 편이 좋다.



## 4. Helm과의 관계
> Helm과 Kustomize는 대체재보다 조합 대상에 가깝다.

Helm은 외부 애플리케이션을 패키지 단위로 설치하고 버전 이력을 관리하는 데 강하다. Kustomize는 이미 존재하는 YAML을 환경별로 조정하는 데 강하다. 실무에서는 두 패턴이 같이 쓰인다.

- 외부 오픈소스 설치: Helm
- 우리 팀 앱의 환경별 배포 차이: Kustomize
- Git 상태를 계속 클러스터에 맞추기: ArgoCD

ArgoCD도 Helm 소스와 Kustomize 소스를 모두 지원한다. 그래서 "공용 플랫폼은 Helm, 애플리케이션 오버레이는 Kustomize" 같은 구조가 자연스럽다.



## 5. 기본 명령
> 로컬에서 항상 먼저 렌더링 결과를 보고 apply하는 습관이 안전하다.

```bash
kubectl kustomize overlays/dev
kubectl apply -k overlays/dev
```

`kubectl kustomize`로 결과를 먼저 보고, 원하는 이름과 라벨, 이미지 태그가 맞는지 확인한 뒤 적용하는 편이 좋다. 템플릿 언어가 없다고 실수가 사라지는 것은 아니기 때문이다.
