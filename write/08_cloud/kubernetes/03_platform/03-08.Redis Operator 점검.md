# Redis Operator 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> Redis Sentinel과 Cluster 운영 차이를 질문 중심으로 다시 점검한다.

### Q1. Redis Sentinel vs Cluster 모드 선택 기준은?

Sentinel은 데이터 크기가 10GB 미만이고 단일 마스터의 쓰기 성능(초당 10만 ops)으로 충분하며 멀티키 연산(MGET, MSET, 트랜잭션)을 자주 사용하는 경우에 적합하다. 고가용성은 필요하지만 샤딩이 불필요한 세션 스토어, 소규모 캐시에 맞다.

Cluster는 데이터 크기가 10GB 이상이거나 쓰기 부하가 단일 노드를 초과하거나 수평 확장이 필요한 경우에 적합하다. 대규모 캐시, 순위표, 분산 카운터에 적합하다.

멀티키 연산 제약이 중요한 판단 기준이다. Cluster 모드에서 `MGET key1 key2`는 두 키가 같은 슬롯에 있어야 실행 가능하다. Hash Tag(`{user:1000}:session`, `{user:1000}:cart`)로 같은 슬롯에 강제 배치할 수 있지만 샤딩 효율이 떨어진다.

클라이언트 복잡도도 고려해야 한다. Sentinel은 대부분의 클라이언트가 지원하지만, Cluster는 클라이언트가 MOVED/ASK 리다이렉션을 처리해야 한다. 일부 언어는 Cluster 클라이언트 성숙도가 낮다.

운영 복잡도는 비슷하다. Sentinel은 총 6개 Pod(Redis 3 + Sentinel 3)이고 Cluster도 최소 6개 Pod(마스터 3 + 리플리카 3)다. Cluster가 노드 추가 시 슬롯 리밸런싱이 필요하여 운영이 더 복잡하다.

심화. Sentinel 마스터가 10GB 메모리 제한에 도달하면 Cluster로 마이그레이션해야 한다. `redis-cli --cluster` 명령으로 데이터를 이동할 수 있지만 다운타임 없이 하려면 Cluster를 새로 구성하고 Replication으로 동기화한 뒤 트래픽을 전환하는 Blue-Green 방식을 사용한다.

### Q2. Sentinel이 마스터 장애를 감지하고 페일오버하는 과정은?

각 Sentinel은 마스터에게 1초마다 PING을 보내고 `down-after-milliseconds`(기본 30초) 동안 응답이 없으면 SDOWN(Subjectively Down)으로 표시한다. 이것은 단일 Sentinel의 주관적 판단이다.

Sentinel은 다른 Sentinel들에게 `SENTINEL is-master-down-by-addr` 명령을 보내 마스터 상태를 물어본다. `quorum` 개수 이상의 Sentinel이 SDOWN에 동의하면 ODOWN(Objectively Down)으로 전환하고 페일오버를 시작한다.

ODOWN 상태가 되면 Sentinel들이 투표하여 페일오버를 수행할 리더 Sentinel을 선출한다. 과반수 이상의 표를 받은 Sentinel이 리더가 된다. 리더 Sentinel이 리플리카 중 하나를 새 마스터로 선택한다. 선택 기준은 마스터와 연결이 끊긴 시간이 짧을수록, 리플리카 우선순위가 높을수록, 복제 오프셋이 클수록 우선된다.

선택된 리플리카에 `SLAVEOF NO ONE` 명령을 보내 마스터로 승격한다. 나머지 리플리카들에게는 `SLAVEOF <새 마스터>` 명령을 보내 새 마스터를 따르도록 설정한다. `parallel-syncs` 설정에 따라 병렬로 수행된다.

심화. Sentinel 2개가 죽으면 quorum=2인 경우 페일오버 시작이 불가능하다. Sentinel은 홀수로, 쿼럼은 과반수 이상으로 설정하는 것이 중요하다.

### Q3. Redis Cluster의 해시 슬롯 분배와 리밸런싱은?

클러스터 초기화 시 `redis-cli --cluster create`가 16384개 슬롯을 마스터 수로 균등 분배한다. 마스터 3개이면 각각 5461, 5462, 5461 슬롯을 담당한다.

키를 CRC16 해시로 변환하고 16384로 나눈 나머지가 슬롯 번호다. 슬롯 12345를 담당하는 마스터가 해당 키를 저장한다.

노드 추가 시 리밸런싱은 `redis-cli --cluster reshard`로 수행한다. 기존 마스터들로부터 슬롯을 이동하는데, 슬롯 이동 중에도 서비스는 계속되며 클라이언트는 ASK 리다이렉션으로 이동 중인 키에 접근한다.

슬롯 이동 과정은 다음과 같다. 대상 마스터에 `CLUSTER SETSLOT <slot> IMPORTING` 전송, 소스 마스터에 `CLUSTER SETSLOT <slot> MIGRATING` 전송, 소스에서 `MIGRATE` 명령으로 키를 대상으로 전송, 이동 완료 시 `CLUSTER SETSLOT <slot> NODE` 명령으로 매핑 업데이트.

이동 중 클라이언트가 소스에 요청하면 `-ASK` 응답을 받고 대상 마스터에 `ASKING` 명령 후 재요청한다. 이동 완료 후에는 `-MOVED` 응답으로 영구 리다이렉션한다.

심화. 슬롯 리밸런싱 중에도 데이터 일관성은 보장된다. 키 이동은 원자적으로 처리되며 소스에서 삭제되기 전에 대상에 복사된다.

### Q4. Kubernetes Service와 Sentinel discovery 통합 방식은?

OpsTree Operator는 `redis-sentinel-sentinel` Service(ClusterIP, 26379)를 생성하여 3개 Sentinel Pod를 로드밸런싱한다. 클라이언트는 이 Service DNS를 Sentinel 주소로 사용한다.

`redis-sentinel-additional` Headless Service는 StatefulSet Pod의 고정 DNS 이름을 제공한다(`redis-sentinel-0.redis-sentinel-additional.default.svc.cluster.local`). Sentinel이 마스터 주소를 반환할 때 이 DNS 이름을 사용한다.

클라이언트 연결 흐름은 이렇다. 클라이언트가 `redis-sentinel-sentinel:26379`에 연결하면 Service가 3개 Sentinel 중 하나로 라우팅한다. Sentinel이 `SENTINEL get-master-addr-by-name mymaster` 응답으로 현재 마스터의 Headless DNS 이름을 반환한다. 클라이언트가 해당 주소로 Redis에 연결한다.

페일오버 시 Service 엔드포인트는 업데이트되지 않는다. Service는 여전히 모든 Pod를 가리킨다. 클라이언트는 Sentinel에게 다시 물어봐서 새 마스터를 발견한다. 이것이 Sentinel discovery를 사용하는 핵심 이유다.

Sentinel Service가 3개 Pod를 로드밸런싱하므로 Sentinel 1개가 죽어도 클라이언트는 계속 연결 가능하다.

심화. Sentinel이 페일오버를 수행한 후 Kubernetes Service Endpoint가 업데이트되지 않아도 문제없다. 클라이언트가 Redis 연결을 Sentinel을 통해 발견하기 때문이다.

### Q5. Redis Operator가 자동화하는 Day-2 Operation은?

리플리카 스케일링은 StatefulSet replicas를 변경하면 Operator가 새 리플리카 Pod를 생성하고 Sentinel이 새 리플리카를 마스터에 연결(`SLAVEOF`)한다.

버전 업그레이드는 `kubernetesConfig.image`를 변경하면 Operator가 StatefulSet Rolling Update를 수행한다. Sentinel 모드는 리플리카 → 마스터 순서로 업그레이드하여 다운타임을 최소화한다.

백업은 OpsTree Operator가 자동화하지 않는다. CronJob으로 `redis-cli --rdb /backup/dump.rdb`를 실행하거나 Velero로 PVC 스냅샷을 생성해야 한다.

장애 복구는 Sentinel 모드에서 마스터 Pod 삭제 시 Sentinel이 자동 페일오버를 수행하고, Cluster 모드에서는 리플리카가 자동으로 승격된다. Operator는 CR 상태를 업데이트하지만 실제 페일오버는 Redis/Sentinel 프로토콜이 수행한다.

심화. Operator가 백업/복원을 자동화하지 않는 이유는 Redis의 백업 방식이 RDB/AOF 두 가지이고 각각 트레이드오프가 다르기 때문이다. 운영 환경에 따라 전략이 달라야 하므로 Operator가 강제하지 않는다.

### Q6. Standalone Redis와 Operator 관리 Redis의 차이는?

StatefulSet으로 Standalone Redis를 배포하면 단일 장애점(SPOF)이 된다. 마스터 Pod가 죽으면 StatefulSet이 자동으로 재시작하지만 수십 초의 서비스 중단이 발생한다.

StatefulSet으로 마스터 1 + 리플리카 2를 배포하고 수동으로 `SLAVEOF`로 복제를 설정할 수 있지만 마스터 장애 시 자동 페일오버가 없다. 운영자가 수동으로 리플리카를 마스터로 승격해야 한다.

Operator는 CR로 선언적 관리를 제공한다. `kind: Redis` 하나로 StatefulSet, Service, ConfigMap이 자동 생성된다. Sentinel/Cluster 프로토콜로 장애 감지와 자동 페일오버가 이루어진다.

Redis는 Deployment가 아닌 StatefulSet으로 배포해야 한다. Pod 이름이 고정되어(`redis-0`, `redis-1`) Headless Service DNS로 접근 가능하고, PVC가 Pod에 바인딩되어 재시작 시에도 데이터가 보존되며, 순차 시작/종료로 복제 관계가 유지된다.

개발/테스트 환경에서 간단한 캐시만 필요하다면 Standalone으로 충분하다. 프로덕션에서는 Operator 또는 관리형 서비스(AWS ElastiCache, GCP Memorystore)를 사용하는 것이 권장된다.

심화. Helm Chart(Bitnami)로 Sentinel을 배포하면 페일오버는 Redis/Sentinel 프로토콜이 자동으로 처리하지만, Day-2 Operation(스케일링, 버전 업그레이드, 모니터링 통합)은 Helm values를 수동으로 수정해야 한다. Operator는 이 과정을 CR 변경으로 추상화한다.
