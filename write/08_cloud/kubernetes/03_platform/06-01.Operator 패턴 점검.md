# Operator 패턴 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> Operator 핵심 개념을 운영 자동화 판단 질문으로 다시 점검한다.

### Q1. CRD와 ConfigMap의 차이 — 왜 CRD를 만들어야 하는가?

ConfigMap은 key-value 데이터를 저장하는 범용 컨테이너다. 어떤 데이터든 자유롭게 넣을 수 있지만 스키마 검증이 없어 잘못된 값이 들어가도 API 서버가 거부하지 않는다.

CRD는 Kubernetes API를 확장해 새로운 리소스 타입을 추가한다. `kubectl get postgrescluster`처럼 타입별 조회가 가능하고, OpenAPI v3 스키마로 필드 타입·필수 여부·범위를 강제한다. 잘못된 값은 API 서버 단계에서 거부된다.

버전 관리도 다르다. ConfigMap은 버전 개념이 없지만 CRD는 v1alpha1, v1beta1, v1 등으로 API 버전을 관리하고 버전 간 변환 webhook도 지원한다. RBAC 통합도 세밀하다. ConfigMap은 "모든 ConfigMap"에 대한 권한만 설정 가능하지만 CRD는 리소스 타입별로 권한을 부여할 수 있다.

Controller 연동 관점에서 ConfigMap은 Controller가 데이터를 직접 파싱해야 하지만, CRD는 API 서버가 watch 이벤트를 제공하고 typed client를 생성할 수 있어 개발이 쉽다. Admission Webhook도 CRD에서만 지원된다.

```bash
# CRD 확인
kubectl get crd
kubectl describe crd postgresclusters.postgres-operator.crunchydata.com

# status subresource 확인 (spec=desired, status=actual)
kubectl get postgrescluster mycluster -o jsonpath='{.status}'
```

심화: CRD를 삭제하면 해당 타입의 모든 Custom Resource도 삭제된다. finalizer가 설정된 리소스가 있으면 Controller가 정리를 완료한 후 삭제된다.

### Q2. Reconciliation Loop의 동작 원리 — Watch → Queue → Reconcile은?

Controller는 API 서버에 HTTP long-polling 또는 WebSocket으로 연결하고 특정 리소스의 변경 이벤트를 실시간으로 받는다. etcd 변경이 감지되면 API 서버가 즉시 이벤트를 보낸다.

이벤트를 받자마자 처리하지 않고 Workqueue에 넣는 이유는 동시성 제어와 재시도 때문이다. 여러 리소스가 동시에 변경되어도 순차 처리하고, 에러 발생 시 exponential backoff(5초 → 10초 → 20초 →...)로 재시도한다.

Reconcile 로직은 세 단계다. etcd에서 desired state를 읽고(`spec.replicas: 3`), 클러스터에서 actual state를 확인(`현재 Pod 2개`), 차이를 해소한다(`Pod 1개 추가`). 멱등성이 핵심이다.

Kubernetes Controller는 Level-driven이다. "Pod가 추가됐다"(edge)가 아니라 "현재 총 몇 개인가"(level)를 확인한다. 이벤트를 놓쳐도 다음 reconcile에서 상태를 맞출 수 있다.

Reconcile 함수는 `(ctrl.Result, error)`를 반환한다. `error != nil`이면 자동 재시도, `Result.Requeue = true`이면 명시적 재시도, `Result.RequeueAfter = 5분`이면 일정 시간 후 재확인이다.

```bash
# Controller 로그로 Reconcile 추적
kubectl logs -n operator-system deploy/my-operator -f | grep Reconcile

# Custom Resource 이벤트 확인
kubectl describe postgrescluster mycluster | grep -A 10 Events
```

심화: 동일 리소스에 이벤트가 빠르게 연속 발생하면 Workqueue가 중복을 제거해 한 번만 처리한다. Reconcile 중 리소스가 삭제되면 `client.IgnoreNotFound(err)`로 처리한다.

### Q3. Operator 성숙도 Level 1~5 — 각 단계에서 어떤 자동화가 추가되는가?

Level 1(Basic Install)은 CRD 적용 시 Controller가 기본 리소스를 생성한다. 사용자는 `spec.replicas`, `spec.image` 정도만 설정하고 나머지는 수동이다. Helm Chart를 CRD로 감싼 형태가 대표적이다.

Level 2(Seamless Upgrades)는 `spec.version`을 변경하면 Controller가 롤링 업데이트를 자동 실행한다. 데이터 마이그레이션이 필요한 경우 Job을 자동 생성한다. 백업/복원은 여전히 수동이다.

Level 3(Full Lifecycle)은 백업, 복원, 모니터링이 자동화된다. `spec.backup.schedule: "0 2 * * *"`로 CronJob이 자동 생성되고, 복원 요청 시 자동 실행된다. Prometheus ServiceMonitor도 자동 생성한다.

Level 4(Deep Insights)는 메트릭을 분석해 이상 징후를 탐지한다. 쿼리 응답 시간이 평소보다 3배 느리면 알림을 보내고 에러 패턴을 인식한다.

Level 5(Auto Pilot)는 완전 자율 운영이다. Primary DB 장애 시 Standby를 자동 승격하고, 디스크 부족 시 PVC를 자동 확장한다.

대부분의 Operator는 Level 2~3에 머문다. Level 5는 예측 불가능한 동작의 위험과 비용 증가 등 트레이드오프가 있어 신중하게 접근한다.

### Q4. Operator SDK vs Kubebuilder — 차이와 선택 기준은?

두 도구 모두 Controller Runtime 라이브러리를 사용한다. CRD 생성, Reconcile 로직, RBAC 설정, Deployment 매니페스트 생성 등 핵심 기능은 같고 프로젝트 구조도 거의 동일하다.

Operator SDK는 Red Hat이 관리하며 Go, Ansible, Helm 세 가지 개발 방식을 제공한다. 기존 Ansible playbook이나 Helm Chart를 빠르게 Operator로 전환할 수 있고 OLM 통합이 긴밀하다.

Kubebuilder는 Kubernetes SIG가 관리하며 순수 Go 개발에 특화됐다. 코드가 더 간결하고 최신 Controller Runtime 기능을 빠르게 반영한다.

Ansible 기반 Operator는 복잡한 조건부 Reconcile이나 status 업데이트가 어렵다. Level 1~2 Operator에 적합하다. Helm 기반 Operator는 배포만 자동화하고 Day-2 Operation은 지원하지 않아 Level 1에만 적합하다.

선택 기준: 기존 Ansible/Helm 자산이 많으면 Operator SDK, 순수 Go 개발이면 Kubebuilder. 복잡한 비즈니스 로직이 필요하면 둘 다 Go 기반을 사용해야 한다.

### Q5. Operator가 해결하는 Day-2 Operation 예시는?

백업 자동화는 CronJob을 자동 생성해 매일 새벽 2시에 pg_dump를 실행하고 S3에 업로드한 뒤 보존 기간이 지난 데이터를 삭제한다. `spec.backup.schedule`과 `spec.backup.retention`으로 선언하면 Controller가 CronJob, PVC, S3 자격증명 Secret을 모두 생성한다.

장애 복구(Failover)는 Primary DB Pod의 liveness probe가 실패하면 Controller가 Standby를 자동 승격시킨다. Service의 selector를 변경해 트래픽을 새 Primary로 라우팅하고, 기존 Primary는 복구 후 Standby로 재구성한다.

버전 업그레이드는 `spec.version`을 변경하면 Controller가 백업 실행 → 마이그레이션 Job → 새 이미지로 Pod 재시작 → 헬스 체크 순서로 처리한다.

스케일링은 `spec.replicas`를 변경하면 StatefulSet이 Pod를 추가하고, Controller는 새 Pod가 Ready 상태가 될 때까지 대기한 후 Replication 설정을 자동 업데이트한다.

인증서 갱신은 Cert Manager Operator가 만료일을 모니터링하고 30일 전에 Let's Encrypt에 자동 갱신을 요청한다. Prometheus Operator는 ServiceMonitor CRD를 자동 생성해 메트릭 수집을 자동 설정한다.

### Q6. OLM 없이 Operator를 설치하는 방법과 OLM의 장점은?

수동 설치는 CRD YAML, Controller Deployment, ServiceAccount/Role/RoleBinding을 순서대로 적용한다. 총 5~10개의 YAML 파일을 관리해야 한다.

```bash
# 수동 설치
kubectl apply -f crd.yaml
kubectl apply -f rbac.yaml
kubectl apply -f deployment.yaml
```

OLM은 Subscription 리소스 하나만 생성하면 CatalogSource에서 CSV(ClusterServiceVersion)를 가져와 나머지를 자동 생성한다.

OLM의 장점은 세 가지다. 업그레이드 순서를 CSV의 `replaces` 필드로 자동 관리하고 롤백도 지원한다. Operator 간 의존성을 자동으로 해결해 필요한 Operator를 먼저 설치한다. `installMode`로 클러스터 전역(AllNamespaces)과 네임스페이스별(OwnNamespace) 설치를 선택할 수 있다.

OLM 없이 설치해야 하는 경우도 있다. 자동 업그레이드를 원하지 않거나, 외부 레지스트리에 접근할 수 없는 Air-gapped 환경이거나, CRD에 커스텀 필드를 추가하는 등 커스터마이징이 많이 필요할 때다.

### Q7. Operator Anti-pattern — 피해야 할 설계 실수는?

너무 많은 책임이 가장 흔한 실수다. DB 배포 + 백업 + 모니터링 + 로깅을 하나의 Operator가 담당하면 복잡도가 폭발한다. 단일 책임 원칙을 지켜야 한다.

상태 관리 실패는 Reconcile 로직에서 외부 API를 호출하거나 외부 DB에 상태를 저장할 때 발생한다. etcd를 진실의 원천(source of truth)으로 삼아야 하며 외부 상태와 불일치하면 복구가 어렵다.

비멱등성 로직은 "Pod 1개 추가"가 아니라 "총 3개여야 함"으로 판단해야 한다. 비멱등성 로직은 재시도 시 중복 생성 문제를 일으킨다.

즉시 실행(Imperative) 로직은 `spec.backup: "now"`처럼 즉시 실행 요청을 담으면 이벤트가 중복 발생할 수 있다. 고유 ID를 사용하고 status에 실행 여부를 기록하는 방식으로 선언적으로 설계해야 한다.

과도한 watch는 API 서버에 부담을 준다. Owner Reference로 내가 생성한 리소스만 watch해야 한다.

에러 처리는 일시적 에러(네트워크 장애)와 영구 에러(잘못된 설정)를 구분해야 한다. 영구 에러는 status에 기록하고 재시도를 중단해야 한다.

```bash
# Owner Reference 확인
kubectl get pod mycluster-0 -o jsonpath='{.metadata.ownerReferences}'

# CR status 에러 정보 확인
kubectl get postgrescluster mycluster -o jsonpath='{.status.conditions}'
```
