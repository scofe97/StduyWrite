# Kustomize 점검

> 본 장의 심화 점검 질문입니다. Kustomize를 실제 운영에 붙일 때 base/overlay, patch, generator에서 자주 생기는 혼동을 Q&A 형태로 정리했습니다.



## 점검 질문
> Kustomize를 단순 치환 도구가 아니라 환경별 구성 조합 도구로 다시 점검합니다.

### Q1. base와 overlay를 어디까지 나눠야 하는가?

`base`는 환경에 상관없이 유지되는 원본 매니페스트입니다. `overlay`는 환경별 차이만 가져가야 합니다. Replica 수, 도메인, 리소스 요청값, 이미지 태그처럼 "환경이 바뀌면 달라지는 값"만 overlay에 두는 편이 좋습니다.

심화 질문. dev와 prod가 너무 많이 달라져서 overlay가 사실상 새 매니페스트가 되면 어떻게 하는가? 그때는 base를 더 작은 단위로 쪼개거나, 아예 애플리케이션 구조를 다시 나누는 편이 낫습니다. Kustomize는 차이를 관리하는 도구지 완전히 다른 시스템 두 개를 같은 base에 우겨 넣는 도구가 아닙니다.

### Q2. patch는 strategic merge와 JSON patch 중 무엇을 써야 하는가?

일반적인 Deployment 수정은 strategic merge가 읽기 쉽습니다. 하지만 리스트 항목을 정확히 지정해 교체하거나 삭제할 때는 JSON patch가 더 명확합니다. 팀 규칙이 없다면 "읽기 쉬우면 strategic merge, 정밀 제어가 필요하면 JSON patch" 정도로 나누는 편이 실용적입니다.

```bash
kubectl kustomize overlays/prod
```

심화 질문. patch가 적용되지 않는 것처럼 보이면 무엇을 먼저 보는가? 대상 리소스의 `kind`, `name`, `namespace`가 정확히 맞는지, 그리고 실제 렌더링 결과를 `kubectl kustomize`로 먼저 확인합니다.

### Q3. generator 해시를 유지할지 끌지 어떻게 정하는가?

ConfigMap/Secret generator의 이름 해시는 설정 변경 시 rollout을 유도하는 장점이 있습니다. 반면 이름이 고정돼야 하는 외부 참조 구조에서는 불편할 수 있습니다. 애플리케이션이 설정 변경 때 자동 재배포돼야 하면 해시 유지가 유리하고, 외부 시스템이 이름을 직접 참조하면 팀 규칙을 더 신중히 잡아야 합니다.

### Q4. Helm과 Kustomize를 같이 쓸 때 충돌하지 않는가?

충돌보다 역할 분리가 더 중요합니다. 외부 플랫폼 설치는 Helm, 우리 앱 환경 차이는 Kustomize, 상태 동기화는 ArgoCD처럼 나누면 오히려 구조가 깔끔해집니다. 한 도구가 모든 문제를 푼다고 생각할 때 구조가 복잡해집니다.

### Q5. `kubectl apply -k`와 `kubectl kustomize`의 운영상 차이는?

`kubectl kustomize <overlay>`는 *렌더링만* 합니다. base와 overlay가 합성된 최종 매니페스트를 stdout으로 출력하고 클러스터에는 손대지 않습니다. PR 리뷰에서 "이 overlay를 적용하면 결과가 뭔가"를 보여 줄 때, 또는 CI에서 `diff`로 변경 영향을 비교할 때 이 명령이 쓰입니다.

`kubectl apply -k <overlay>`는 같은 렌더링 결과를 *클러스터에 적용*합니다. 실수로 prod overlay를 잘못 잡으면 즉시 반영되므로, 운영 흐름에서는 `kustomize`로 한 번 확인하고 `apply -k`로 적용하는 두 단계로 두는 편이 안전합니다. ArgoCD를 쓰는 환경이라면 `apply -k`를 거의 쓰지 않고 Git push가 적용 트리거가 됩니다.
