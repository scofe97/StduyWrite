# PostgreSQL Operator 점검

> 본 장의 심화 점검 질문이다. LEARN에서 다룬 개념의 경계와 운영 환경에서 주의할 판단 포인트를 Q&A 형태로 정리했다.



## 점검 질문
> PostgreSQL Operator 선택과 운영 포인트를 질문 중심으로 다시 점검한다.

### Q1. CloudNativePG vs Zalando Postgres Operator 선택 기준은?

새 프로젝트를 시작하고 간결함을 원한다면 CloudNativePG가 적합하다. 설치가 `kubectl apply` 한 번으로 끝나고 외부 의존성이 없다. CNCF Sandbox 프로젝트라 벤더 중립적이고 커뮤니티가 빠르게 성장하고 있다. Barman 통합으로 백업과 PITR이 간단하며 최신 PostgreSQL 버전을 빠르게 지원한다.

Zalando는 수년간 프로덕션에서 검증된 안정성이 강점이다. Patroni 기반이라 복잡한 페일오버 시나리오(네트워크 파티션, 스플릿 브레인)에 강하고 Postgres Operator UI가 제공된다. 대규모 멀티 테넌트 환경의 사례가 많다. 단, Patroni와 etcd 같은 추가 컴포넌트가 필요해 아키텍처가 복잡하고 CNCF 프로젝트가 아니라 Zalando의 우선순위에 의존한다.

선택 기준을 정리하면 이렇다. 새 프로젝트이고 간결함을 원한다면 CNPG, 이미 Patroni를 사용 중이거나 대규모 검증된 솔루션이 필요하면 Zalando, 엔터프라이즈 기능(감사, 암호화)이 필요하면 CrunchyData PGO.

마이그레이션 관점에서는 두 Operator 모두 표준 PostgreSQL을 사용하므로 `pg_dump/pg_restore`로 데이터를 이동할 수 있지만 HA 설정과 백업 정책은 다시 구성해야 한다.

심화. Patroni의 DCS로 K8s ConfigMap/Endpoints API를 사용할 수 있음에도 Zalando가 etcd를 권장하는 이유는 성능과 안정성 때문이다. CNPG가 CNCF에 편입된 것은 벤더 중립성과 장기 유지보수를 보장한다는 기술적 의미를 갖는다.

### Q2. Streaming Replication과 Logical Replication의 차이는?

Streaming Replication은 WAL 파일을 바이트 단위로 Primary에서 Replica로 전송한다. Replica는 Primary의 완전한 복사본이 되며 모든 데이터베이스와 테이블이 복제된다. 트랜잭션이 아닌 블록 단위로 복제하므로 빠르지만 PostgreSQL 버전이 다르면 호환되지 않는다.

Logical Replication은 트랜잭션의 논리적 변경(INSERT, UPDATE, DELETE)을 전송한다. 특정 테이블이나 특정 데이터베이스만 선택적으로 복제할 수 있고 서로 다른 PostgreSQL 버전 간에도 가능하다. 하지만 느리고 DDL(스키마 변경)이 자동 복제되지 않는다.

CNPG가 Streaming을 사용하는 이유는 HA에 적합하기 때문이다. Replica가 Primary의 완전한 복사본이어야 페일오버 시 즉시 Primary 역할을 할 수 있다.

Logical Replication의 사용 사례는 메이저 버전 업그레이드 중 Zero Downtime 마이그레이션, 멀티 클라우드 복제, 특정 테이블만 분석 환경으로 복제하는 경우다.

심화. Streaming Replication에서 Replica는 읽기 전용이지만 Logical Replication의 Subscriber에는 쓰기가 가능하다. 단 충돌 해결이 필요하다.

### Q3. CNPG의 자동 페일오버 메커니즘은?

Operator는 Liveness Probe를 통해 Primary Pod를 주기적으로 체크한다. 3~5회 연속 실패 후 장애로 확정하면 모든 Replica의 LSN(Log Sequence Number)을 확인하여 가장 앞선 Replica를 새 Primary로 선출한다.

선택된 Replica에서 `pg_ctl promote` 명령을 실행하면 Replica가 Recovery 모드에서 빠져나와 쓰기 가능 모드로 전환된다. 이후 나머지 Replica들은 `primary_conninfo`를 업데이트하고 Replication Slot을 재생성하여 새 Primary를 따라간다. Operator는 `-rw` Service의 Selector를 업데이트하여 트래픽을 새 Primary로 보낸다.

PostgreSQL은 페일오버마다 새 Timeline을 시작한다. 이전 Primary가 복구되면 자신의 Timeline이 구버전임을 알고 새 Primary의 Timeline을 따른다. 이렇게 스플릿 브레인을 방지한다.

심화. 모든 Replica의 LSN이 동일하면 Pod 이름 순서, 네트워크 레이턴시 등 추가 기준으로 선출한다. 동기 복제를 사용하면 데이터 손실이 완전히 방지되지만 레이턴시가 증가하고 Replica 장애 시 쓰기가 중단될 수 있다.

### Q4. postgresql.parameters로 설정 관리 시 재시작이 필요한 경우는?

PostgreSQL 설정은 세 종류로 분류된다. `shared_buffers`, `max_connections` 같은 메모리 관련 설정은 PostgreSQL 재시작이 필요하다. 대부분의 설정은 SIGHUP(pg_reload_conf())으로 즉시 적용된다. `work_mem` 같은 일부 설정은 세션마다 `SET work_mem='16MB'`로 다르게 적용할 수 있다.

우선 조정해야 할 설정은 `shared_buffers`(메모리의 25% 권장), `effective_cache_size`(50~75%), `work_mem`(정렬/조인 작업 메모리), `max_connections`(동시 연결 수), `log_min_duration_statement`(느린 쿼리 로깅)이다.

`postgresql.parameters`에서 설정을 변경하면 Operator는 Pod를 하나씩 재시작한다(Rolling Update). Replica부터 먼저 재시작하고 마지막에 Primary를 재시작한다. Primary 재시작 시에는 Replica 중 하나를 임시로 승격시켜 무중단으로 진행하지만, 10~30초의 쓰기 중단이 발생할 수 있다.

모든 설정을 CR로 관리해야 GitOps가 가능하다. Pod에 직접 접속하여 `ALTER SYSTEM`을 실행하면 CR과 충돌할 수 있어 권장하지 않는다.

심화. `shared_buffers`를 메모리의 50%로 설정하면 안 되는 이유는 OS 페이지 캐시와 이중 캐싱이 발생하기 때문이다. PostgreSQL은 OS 캐시에도 의존하므로 메모리의 25%가 통상적인 권장값이다.

### Q5. WAL 아카이빙과 PITR 개념은?

WAL(Write-Ahead Log)은 PostgreSQL의 트랜잭션 로그다. 모든 변경 사항이 WAL에 먼저 쓰이고 나중에 데이터 파일에 반영된다. WAL 파일을 외부 스토리지로 복사하는 것이 WAL 아카이빙이다.

Base Backup은 데이터 디렉토리 전체의 스냅샷이다. Base Backup을 복원하고 그 이후의 WAL을 순서대로 재생하면 원하는 시점으로 복구할 수 있다. 예를 들어 "오전 10시 Base Backup + 오전 10시~오후 2시 30분 WAL"을 재생하면 오후 2시 30분 상태로 복구된다.

PITR의 주요 사용 사례는 실수로 테이블을 DROP했을 때, 악의적인 데이터 변조가 발견되었을 때, 프로덕션 데이터로 테스트 환경을 구성할 때다.

WAL 파일은 쓰기가 많으면 하루에 수십 GB가 쌓인다. CNPG는 gzip으로 압축하여 저장하고(약 70~80% 절약) `retentionPolicy: "30d"`로 30일 이상 된 WAL과 백업을 자동 삭제한다.

심화. WAL 파일이 아카이빙되기 전에 Primary가 죽으면 마지막 WAL이 손실될 수 있다. `archive_mode = always` 옵션으로 이 위험을 줄일 수 있다.

### Q6. CNPG의 Rolling Update가 Primary를 마지막에 재시작하는 이유는?

Primary는 쓰기를 담당하므로 먼저 재시작하면 쓰기가 중단된다. Replica를 먼저 업데이트하면 새 버전의 Replica가 준비된 상태에서 Primary를 전환할 수 있어 쓰기 중단 시간을 최소화한다.

구체적인 과정은 이렇다. Replica들을 하나씩 재시작하고 각 Replica가 정상 동작하는 것을 확인한 후 다음으로 넘어간다. 모든 Replica가 업데이트되면 Switchover를 수행한다. Replica 중 하나를 새 Primary로 승격시키고 기존 Primary를 재시작한다. 재시작된 Pod는 Replica로 합류한다.

기본적으로 CNPG는 Switchback을 자동으로 하지 않는다. 새 Primary가 그대로 Primary로 남는다. 사용자가 원하면 `kubectl cnpg promote`로 수동 Switchback을 수행할 수 있다.

메이저 버전 업그레이드(15 → 16)는 Streaming Replication이 같은 메이저 버전 간에만 동작하므로 Rolling Update가 불가능하다. Blue-Green 방식으로 새 클러스터를 16으로 생성하고 Logical Replication으로 동기화한 뒤 트래픽을 전환해야 한다.

심화. Rolling Update 중 한 Replica가 실패하면 CNPG는 실패한 Pod를 건너뛰고 계속 진행하면서 사용자에게 알림을 보낸다. PodDisruptionBudget(PDB)이 있으면 동시 재시작 수를 제한한다.

### Q7. 애플리케이션이 Primary/Replica 엔드포인트를 어떻게 사용해야 하는가?

가장 단순한 연결 방식은 `-rw` Service만 사용하는 것이다. 모든 쿼리를 Primary로 보내지만 Primary에 부하가 집중된다.

Read-Write 분리를 적용하면 쓰기와 강한 일관성이 필요한 읽기는 `-rw`로, 최종 일관성이 허용되는 읽기는 `-ro`로 보낸다. 예를 들어 사용자가 방금 작성한 게시글 조회는 `-rw`(Primary)로, 대시보드 통계 조회는 `-ro`(Replica)로 처리한다.

Replication Lag 문제에 주의해야 한다. Replica가 Primary보다 약간 뒤처지므로 사용자가 쓰기 직후 Replica로 조회하면 방금 쓴 데이터가 없을 수 있다(Read Your Writes Consistency). 이를 해결하려면 쓰기를 한 사용자의 다음 읽기는 잠시 Primary로 보내는 세션 스티키니스 전략을 애플리케이션 레벨에서 구현한다.

Spring Boot에서 Read-Write 분리 설정 예시:

```yaml
spring:
  datasource:
    write:
      url: jdbc:postgresql://mycluster-rw.default.svc.cluster.local:5432/myapp
    read:
      url: jdbc:postgresql://mycluster-ro.default.svc.cluster.local:5432/myapp
```

심화. Replica에서 `SELECT FOR UPDATE`를 실행하면 읽기 전용이므로 에러가 발생한다. PgBouncer의 Pool Mode는 Transaction 모드가 재사용률이 높지만 Session 모드보다 호환성이 낮다.
