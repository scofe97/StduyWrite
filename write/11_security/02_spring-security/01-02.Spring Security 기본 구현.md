# Spring Security 기본 구현 (SecurityFilterChain·HttpSecurity)

---

> 이 글은 Spring Boot 3 + Spring Security 6.x 환경에서 `SecurityFilterChain` 빈 하나로 인증·인가·CORS·세션·커스텀 필터를 구성하는 방법을 다룬다. `WebSecurityConfigurerAdapter`를 상속하던 5.x 시절 코드는 모두 6.x 컴포넌트 기반 설정으로 바뀐다.

## 한 줄 정의

`SecurityFilterChain` 빈을 등록하면 그 안에서 어떤 경로를 누구에게 허용할지, 어떤 필터를 어디 끼울지, 비밀번호를 어떻게 해시할지 같은 보안 설정이 한 자리에서 결정된다.

## 왜 컴포넌트 기반으로 바뀌었나

> 5.7 이전에는 `WebSecurityConfigurerAdapter`를 상속해 `configure(HttpSecurity)`를 override하는 방식이 표준이었다. 그런데 한 클래스가 `HttpSecurity`, `WebSecurity`, `AuthenticationManagerBuilder` 세 가지를 동시에 손대다 보니 책임이 뒤섞이고 테스트가 어려웠다.

6.x는 각 책임을 별도 빈으로 노출한다. `SecurityFilterChain`은 필터 체인 정의, `WebSecurityCustomizer`는 보안 무시 경로, `AuthenticationManager`는 인증 처리, `PasswordEncoder`는 해시 알고리즘이다. 빈마다 단위 테스트가 쉬워졌고, 같은 `@Configuration` 안에 여러 `SecurityFilterChain`을 등록해 경로별로 정책을 분리할 수도 있다.

## 의존성과 자동 구성

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-security'
    testImplementation 'org.springframework.security:spring-security-test'
}
```

starter만 추가하면 다음이 자동으로 켜진다.

1. 모든 엔드포인트가 인증 필요 상태가 된다.
2. 기본 `UserDetailsService`가 인메모리 user를 만들고 임시 비밀번호를 로그에 출력한다.
3. 기본 `SecurityFilterChain`이 폼 로그인과 HTTP Basic을 둘 다 활성화한다.

운영 코드에서 이 기본값을 그대로 두는 일은 없다. 다음 두 빈을 등록하는 순간 자동 구성은 비활성화되고 직접 정의한 설정으로 교체된다.

## SecurityFilterChain 기본 골격

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(sm -> sm
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/login", "/join", "/actuator/health").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
                .anyRequest().authenticated())
            .formLogin(form -> form
                .loginPage("/loginForm")
                .loginProcessingUrl("/login")
                .defaultSuccessUrl("/"));
        return http.build();
    }
}
```

핵심은 람다 DSL이다. 5.x의 `http.csrf().disable().and().sessionManagement()...` 체인은 6.x에서 deprecated이며, 람다로 명시한 형태만 유효하다. 람다는 메서드 누락을 컴파일 단계에서 막아 주는 효과도 있다.

## BCryptPasswordEncoder가 사실상 표준인 이유

> 평문 비밀번호 저장은 어떤 경우에도 금지다. 단순 SHA-256 해시도 GPU로 초당 수십억 시도가 가능해 안전하지 않다. BCrypt는 의도적으로 느리게 설계되어 무차별 대입 비용을 높인다.

`BCryptPasswordEncoder`는 다음을 자동 처리한다.

1. **솔트 자동 생성** — 같은 비밀번호라도 매번 다른 해시가 나온다. 레인보우 테이블 공격이 무력화된다.
2. **work factor 조절** — 기본 10(2^10번 반복). 하드웨어가 빨라지면 12~14로 올려서 미래의 무차별 대입까지 대비한다.
3. **자체 검증** — `matches(rawPassword, encodedPassword)`가 솔트·work factor를 해시 문자열에서 추출해 비교한다.

```java
String raw = "1234";
String encoded = encoder.encode(raw);
// $2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy

encoder.matches(raw, encoded); // true
```

Spring Security 5 이후 기본값은 `DelegatingPasswordEncoder`로, `{bcrypt}...` 같은 prefix를 통해 알고리즘을 식별한다. BCrypt 외에 Argon2, scrypt, PBKDF2도 같은 인터페이스로 지원한다.

## HttpSecurity 주요 영역

### CORS·CSRF·세션

```java
http
    .cors(cors -> cors.configurationSource(corsSource()))
    .csrf(csrf -> csrf.disable())
    .sessionManagement(sm -> sm
        .sessionCreationPolicy(SessionCreationPolicy.STATELESS));
```

세션 정책 4가지의 차이는 다음과 같다.

| 정책 | 의미 | 적합 시나리오 |
|------|------|--------------|
| `ALWAYS` | 요청마다 세션 생성 | 거의 안 씀 |
| `IF_REQUIRED` | 필요 시 생성 (기본값) | 전통 SSR + 폼 로그인 |
| `NEVER` | 직접 생성 안 하지만 기존 세션 사용 | 혼합 모드 |
| `STATELESS` | 세션 사용·생성 모두 안 함 | JWT·OAuth2 토큰 기반 API |

JWT를 쓰는데 `STATELESS`가 아니면 `JSESSIONID` 쿠키가 발급되어 무상태 가정이 깨진다. CORS 설정은 다음과 같이 분리해서 등록한다.

```java
@Bean
public CorsConfigurationSource corsSource() {
    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowCredentials(true);
    config.addAllowedOriginPattern("*");
    config.addAllowedHeader("*");
    config.addAllowedMethod("*");
    source.registerCorsConfiguration("/**", config);
    return source;
}
```

운영 환경에서 `addAllowedOriginPattern("*")`은 위험하다. 실제 프론트엔드 origin을 명시해야 한다.

### 경로별 인가 — authorizeHttpRequests

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers(HttpMethod.GET, "/login**", "/web-resources/**").permitAll()
    .requestMatchers(HttpMethod.GET, "/admin/**").hasRole("ADMIN")
    .requestMatchers(HttpMethod.GET, "/order/**").hasRole("USER")
    .anyRequest().authenticated());
```

6.x의 `authorizeHttpRequests`는 5.x의 `authorizeRequests`를 대체한다. 차이는 인가 결정이 `AuthorizationFilter`로 이전된 점이고, 사용 문법은 거의 같다. 매처 메서드는 위에서 아래로 평가되므로 **구체적인 경로를 먼저, 광범위한 경로를 나중에** 둔다. `.anyRequest().authenticated()`는 마지막 catch-all이다.

권한 표현은 두 가지다. `hasRole("ADMIN")`은 내부적으로 `ROLE_ADMIN` 권한을 찾고, `hasAuthority("ADMIN")`은 prefix 없이 정확히 매치한다. Spring Security가 자동으로 `ROLE_` prefix를 붙이는 영역과 그렇지 않은 영역이 섞여 있어 혼란이 잦은데, **DB에 저장할 때 `ROLE_USER`처럼 prefix를 직접 붙이고 `hasRole("USER")`로 검사하는 방식이 가장 일관적**이다.

### 폼 로그인 설정

```java
.formLogin(form -> form
    .loginPage("/loginForm")
    .loginProcessingUrl("/login")
    .usernameParameter("userId")
    .passwordParameter("userPw")
    .defaultSuccessUrl("/")
    .failureUrl("/loginForm?error")
    .permitAll());
```

`loginProcessingUrl`은 폼이 POST로 제출되는 URL이고, 이 URL은 직접 컨트롤러를 만들지 않는다. Spring Security의 `UsernamePasswordAuthenticationFilter`가 가로채서 처리한다. `usernameParameter`·`passwordParameter`는 폼의 `<input name="...">` 값과 일치해야 한다.

### 커스텀 필터 끼우기

JWT 인증 필터처럼 표준 동작 앞에서 동작해야 하는 필터는 다음 세 메서드 중 하나로 등록한다.

| 메서드 | 의미 |
|--------|------|
| `addFilterBefore(filter, X.class)` | X 필터 **앞**에 삽입 |
| `addFilterAfter(filter, X.class)` | X 필터 **뒤**에 삽입 |
| `addFilterAt(filter, X.class)` | X 위치에 **대체** |

JWT 인증 필터는 거의 항상 `UsernamePasswordAuthenticationFilter` 앞에 둔다.

```java
.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
```

이렇게 하면 JWT 헤더가 유효한 요청은 `UsernamePasswordAuthenticationFilter` 단계를 건너뛰고 곧장 `SecurityContext`가 채워진다. 자세한 구현은 [03-01.JWT 인증 구현](03-01.JWT 인증 구현.md)에서 다룬다.

## OncePerRequestFilter — 커스텀 필터 베이스 클래스

```java
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final TokenProvider tokenProvider;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {

        String token = parseBearerToken(request);

        if (token != null && tokenProvider.validToken(token)) {
            Authentication auth = tokenProvider.getAuthentication(token);
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        chain.doFilter(request, response);
    }

    private String parseBearerToken(HttpServletRequest request) {
        return Optional.ofNullable(request.getHeader(HttpHeaders.AUTHORIZATION))
                .filter(t -> t.startsWith("Bearer "))
                .map(t -> t.substring(7))
                .orElse(null);
    }
}
```

`OncePerRequestFilter`는 한 요청당 한 번만 동작이 보장된다. `forward`나 `include`가 일어나도 중복 실행되지 않는다. 일반 `Filter`를 상속하면 dispatch가 발생할 때마다 재호출될 위험이 있다.

## 실습 — 1분짜리 자동 동작 검증

`SecurityConfig` 없이 starter만 추가한 상태에서 다음 명령을 돌려 본다.

```bash
curl -i http://localhost:8080/
# HTTP/1.1 401
# WWW-Authenticate: Basic realm="Realm"
```

기본 임시 비밀번호로 인증을 추가하면 통과한다.

```bash
# 로그에 출력된 password 사용
curl -i -u user:8e557245-... http://localhost:8080/
# HTTP/1.1 200
```

이 두 명령이 401 → 200으로 바뀌면 Spring Security 필터 체인이 정상 동작 중이라는 뜻이다. 이후 `SecurityConfig`를 등록하면 자동 구성은 비활성화되고 직접 정의한 규칙이 적용된다.

## 면접 대비 요약

### 한 줄 정의

"`@EnableWebSecurity`와 `SecurityFilterChain` 빈을 등록하면 그 안의 람다 DSL로 인증·인가·세션·CORS·커스텀 필터를 한 자리에서 설정할 수 있다. 6.x부터는 상속 대신 컴포넌트 기반 설정이 표준이다."

### 핵심 포인트 3가지

1. **`PasswordEncoder` 빈은 항상 별도로 등록** — `SecurityFilterChain`에서 참조하지 않더라도 `DaoAuthenticationProvider`가 컨테이너에서 찾아 쓴다. BCrypt가 사실상 표준이다.
2. **경로 매처 순서가 중요** — `requestMatchers`는 위에서 아래로 평가된다. `/admin/**`를 `/**`보다 먼저 두지 않으면 모든 요청이 catch-all에 잡힌다.
3. **JWT는 `STATELESS` + 커스텀 필터** — 세션 정책을 STATELESS로 설정하고, JWT 검증 필터를 `UsernamePasswordAuthenticationFilter` 앞에 끼우는 조합이 토큰 기반 API의 정석이다.

### 자주 묻는 질문

Q: 5.x의 `http.csrf().disable().and().sessionManagement()...`가 동작하지 않는데?
A: 6.x에서 deprecated. 람다 DSL(`csrf(csrf -> csrf.disable())`)로 다시 작성해야 한다. `and()`는 사라졌다.

Q: `hasRole("ADMIN")`과 `hasAuthority("ADMIN")`의 차이는?
A: 전자는 내부적으로 `ROLE_` prefix를 붙여 `ROLE_ADMIN` 권한을 찾고, 후자는 prefix 없이 정확히 매치한다. DB에 `ROLE_USER`처럼 저장하는 관례를 따른다면 `hasRole("USER")`가 일관적이다.

Q: 여러 `SecurityFilterChain`을 동시에 등록할 수 있는가?
A: 가능하다. `@Order`로 우선순위를 매기고 `securityMatcher("/api/**")` 같은 경로 한정자를 두면 API용·웹용 정책을 분리할 수 있다.
