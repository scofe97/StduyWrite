# Naver OAuth2 Login (provider 직접 등록과 response 래핑)

---

> 네이버·카카오처럼 Spring Security가 기본 제공하지 않는 provider를 추가할 때는 `provider:` 블록에 endpoint들을 직접 등록해야 한다. 게다가 네이버는 응답을 `response` 객체로 한 번 더 감싸는 특이 구조라, 기존 `OAuth2UserInfo` 구현체에 작은 변형이 필요하다.

## 한 줄 정의

네이버 OAuth2는 `application.yml`의 `provider:` 블록에 authorization·token·user-info endpoint를 명시 등록하고, 응답 `attributes` 안의 `response` 객체를 한 번 더 풀어 `NaverUserInfo`로 매핑하면 구글·페이스북과 같은 흐름으로 동작한다.

## 왜 직접 등록이 필요한가

> Spring Security는 `CommonOAuth2Provider` enum으로 Google·GitHub·Facebook·Okta 네 곳의 endpoint 메타데이터를 내장한다. 그 외 provider는 정확한 URL을 명세에 맞춰 직접 등록해야 한다. 잘못 등록하면 인가 요청부터 실패한다.

직접 등록이 필요한 항목은 다음 네 가지다.

1. **authorization-uri** — 사용자를 동의 화면으로 보낼 URL
2. **token-uri** — 인가 코드를 access token으로 교환할 URL
3. **user-info-uri** — access token으로 사용자 프로필을 조회할 URL
4. **user-name-attribute** — 응답 JSON에서 사용자 식별자가 들어 있는 키

이 메타데이터는 각 provider의 공식 개발자 가이드에 명시돼 있다. 네이버의 경우 [Naver Developers - 로그인 API](https://developers.naver.com/docs/login/api/api.md)가 출처다.

## 네이버 개발자 콘솔 사전 작업

[Naver Developers](https://developers.naver.com/main/) → Application → 애플리케이션 등록에서 다음을 채운다.

1. **애플리케이션 이름**: 동의 화면에 노출
2. **사용 API**: 네이버 로그인
3. **서비스 환경**: PC 웹
4. **서비스 URL**: `http://localhost:8080`
5. **Callback URL**: `http://localhost:8080/login/oauth2/code/naver`

등록을 마치면 `Client ID`와 `Client Secret`이 발급된다. 마지막 `naver` 토큰은 [02-02](02-02.Google OAuth2 Login.md)와 마찬가지로 `application.yml`의 registration ID와 일치해야 한다.

## application.yml 설정

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          google:
            client-id: ${GOOGLE_CLIENT_ID}
            client-secret: ${GOOGLE_CLIENT_SECRET}
            scope: [email, profile]

          facebook:
            client-id: ${FACEBOOK_APP_ID}
            client-secret: ${FACEBOOK_APP_SECRET}
            scope: [public_profile, email]

          naver:
            client-id: ${NAVER_CLIENT_ID}
            client-secret: ${NAVER_CLIENT_SECRET}
            scope: [name, email]
            client-name: Naver
            authorization-grant-type: authorization_code
            redirect-uri: "{baseUrl}/login/oauth2/code/{registrationId}"

        provider:
          naver:
            authorization-uri: https://nid.naver.com/oauth2.0/authorize
            token-uri: https://nid.naver.com/oauth2.0/token
            user-info-uri: https://openapi.naver.com/v1/nid/me
            user-name-attribute: response
```

구글·페이스북과 다른 부분 세 가지를 짚는다.

| 항목 | 의미 |
|------|------|
| `authorization-grant-type: authorization_code` | 그랜트 타입 명시. 기본값 사용 시 생략 가능하지만 의도를 드러내는 편이 안전 |
| `redirect-uri: "{baseUrl}/login/oauth2/code/{registrationId}"` | 플레이스홀더 형식. 기본값과 같지만 명시해 두면 환경 변경에 안전 |
| `user-name-attribute: response` | 네이버 응답이 `response`라는 키로 한 번 감싸여 들어오기 때문 |

`user-name-attribute`를 잘못 지정하면 Spring Security가 `OAuth2User.getName()`에서 사용자 식별자를 찾지 못해 `IllegalArgumentException`을 던진다.

## 네이버 응답 구조의 특이성

> 구글·페이스북은 `{ "id": "...", "email": "..." }` 처럼 flat JSON을 반환한다. 네이버는 한 단계 더 감싼다.

```json
{
  "resultcode": "00",
  "message": "success",
  "response": {
    "id": "32742776",
    "email": "user@naver.com",
    "name": "홍길동",
    "nickname": "길동이",
    "profile_image": "https://ssl.pstatic.net/..."
  }
}
```

식별자가 최상위가 아니라 `response.id`에 있다. 위 yml의 `user-name-attribute: response`가 이 점을 알려 주는 설정이고, 우리 코드에서도 attributes를 풀어내는 단계가 한 번 더 필요하다.

## NaverUserInfo — response를 풀어내는 구현체

```java
@RequiredArgsConstructor
public class NaverUserInfo implements OAuth2UserInfo {

    private final Map<String, Object> attributes;

    @Override public String getProviderId() { return (String) attributes.get("id"); }
    @Override public String getProvider() { return "naver"; }
    @Override public String getEmail() { return (String) attributes.get("email"); }
    @Override public String getName() { return (String) attributes.get("name"); }
}
```

`GoogleUserInfo`·`FacebookUserInfo`와 키 이름만 다르고 형식은 동일하다. 차이는 **생성자에 넘기는 `attributes`가 이미 `response`로 풀려 있어야** 한다는 점이다. 풀어내는 책임은 `PrincipalOauth2UserService` 안에 둔다.

```java
private OAuth2UserInfo resolveUserInfo(String registrationId, Map<String, Object> attributes) {
    return switch (registrationId) {
        case "google"   -> new GoogleUserInfo(attributes);
        case "facebook" -> new FacebookUserInfo(attributes);
        case "naver"    -> new NaverUserInfo((Map<String, Object>) attributes.get("response"));
        default -> throw new OAuth2AuthenticationException(
                new OAuth2Error("unsupported_provider"), "지원하지 않는 provider: " + registrationId);
    };
}
```

네이버 case에서 `attributes.get("response")`로 한 번 풀어내면 `NaverUserInfo`는 다른 구현체와 똑같은 키 매핑 코드를 갖는다. 차이를 흡수하는 책임을 `resolveUserInfo`에 집중시켰기 때문에 가능한 단순함이다.

## 로그인 페이지 확장

```html
<!-- loginForm.html -->
<a href="/oauth2/authorization/google">Google Login</a>
<a href="/oauth2/authorization/facebook">Facebook Login</a>
<a href="/oauth2/authorization/naver">Naver Login</a>
```

`SecurityFilterChain` 설정은 그대로다. `OAuth2LoginAuthenticationFilter`가 `registration` 블록 전체를 자동 인식한다.

## 카카오·기타 provider 일반화

> 네이버 사례를 일반화하면 "기본 제공이 아닌 provider 추가"는 다음 네 단계로 끝난다.

1. **개발자 콘솔 등록** — `Client ID`/`Secret`/`Callback URL`
2. **application.yml** — `registration.{provider}` + `provider.{provider}`의 endpoint 4개
3. **`XxxUserInfo` 구현체** — `OAuth2UserInfo` 인터페이스 구현
4. **`resolveUserInfo`에 case 추가** — 응답 래핑이 있다면 풀어내기

카카오도 같은 패턴을 따른다. 응답이 `kakao_account.email`처럼 두 단계로 깊어지면 `KakaoUserInfo` 생성자에서 `attributes.get("kakao_account")`를 한 번 더 풀어 받는다.

## 인가 그랜트 타입 — Spring Security 5에서 6으로

> 5.x의 `authorization-grant-type: implicit`는 6.x에서 제거됐다. 인가 코드 그랜트(`authorization_code`)만 지원한다는 뜻이다. 임플리시트는 [02-01 §그랜트 보안](02-01.OAuth2 개념과 흐름.md#자주-묻는-질문)에서 다룬 이유로 사라졌다.

기본값을 그대로 받아도 되지만, 명시해 두는 편이 코드 리뷰 시 의도가 분명해진다.

## 면접 대비 요약

### 한 줄 정의

"Spring Security 기본 제공이 아닌 provider는 `application.yml`의 `provider:` 블록에 endpoint 4개를 직접 등록하고, 응답 래핑이 있으면 `OAuth2UserInfo` 구현체에 넘기기 전에 한 번 풀어 준다."

### 핵심 포인트 3가지

1. **`provider:` 블록 endpoint 4개** — authorization-uri·token-uri·user-info-uri·user-name-attribute. 잘못 등록하면 인가 요청부터 실패한다.
2. **`user-name-attribute`는 응답 구조에 맞춰** — 네이버는 `response`, 카카오는 `id`, 구글은 `sub`. `OAuth2User.getName()`이 찾을 키를 알려 준다.
3. **응답 래핑은 `resolveUserInfo`에서 풀어낸다** — `NaverUserInfo`가 직접 풀면 다른 구현체와 형태가 어긋난다. 차이를 한 곳에 모으는 것이 유지보수성을 결정한다.

### 자주 묻는 질문

Q: `redirect-uri`에 `{baseUrl}` 같은 플레이스홀더를 쓰는 이유는?
A: 로컬·스테이징·운영의 도메인이 다른 환경에서도 같은 yml로 동작하게 하려는 의도다. `{baseUrl}`은 현재 요청의 schema·host·port를 자동으로 채운다.

Q: 카카오 로그인은 왜 Spring Security가 기본 제공하지 않는가?
A: `CommonOAuth2Provider` enum에 등록된 4개(Google·GitHub·Facebook·Okta)가 글로벌 표준에서 가장 많이 쓰이기 때문이다. 한국 provider(네이버·카카오)는 직접 등록 대상이고, 이 패턴이 일반화된 provider 추가 절차다.

Q: `client-name`은 어디에 노출되는가?
A: Spring Security가 자동 생성하는 `/login` 페이지의 provider 버튼 라벨이다. 직접 만든 로그인 페이지에서는 사용하지 않으므로 생략해도 무방하다.
