# 젠킨스 파이프라인 CRUD 모델과 TPS 패턴 (2.222+)
---
> 이 문서는 Jenkins 파이프라인 CRUD를 어떻게 모델링하고, TPS에서 어떤 패턴으로 구현하는지를 설명하는 이론 문서다.
>
> - 순수 API 스펙은 `05-03`에서 다룬다.
> - 이 문서는 현대 Jenkins 관점, TPS 패턴, 코드 구조에 집중한다.

## 1. 이 문서의 관점

`05-03`이 CRUD endpoint를 설명하는 문서라면, 이 문서는 다음 질문에 답하는 문서다:

- 왜 파이프라인 CRUD가 단순 REST CRUD보다 복잡한가
- TPS는 왜 폴더를 자동 생성하는가
- 왜 create가 아니라 upsert처럼 동작하는가
- Jenkins 2.222+ 이후 어떤 코드를 줄일 수 있는가

즉 이 문서는 endpoint보다 동작 모델과 구현 패턴을 설명한다.



## 2. 파이프라인 CRUD가 일반 CRUD와 다른 이유

Jenkins 파이프라인 CRUD는 단순히 "레코드 하나를 저장"하는 작업이 아니다.

다음 특징이 있다:

- 생성 대상이 파일이 아니라 Job 설정 XML이다.
- 상위 폴더가 없으면 먼저 폴더를 만들어야 한다.
- 수정은 patch가 아니라 전체 config.xml 교체다.
- 삭제는 리다이렉트나 폴더 삭제 부작용을 동반할 수 있다.

즉 CRUD이지만 내부적으로는 구조 관리, XML 생성, 보안 헤더 처리가 같이 들어간다.



## 3. TPS에서 사용한 핵심 패턴

### 3.1 pipelineStruct 경로 모델

TPS는 Jenkins 경로를 문자열 하나로 보지 않고, 도메인 규칙이 반영된 구조로 본다.

기본 규칙은 다음과 같다:

- 일반 파이프라인: `/job/{taskCd}/job/{envrnCd}/job/{bizNm}`
- 트리거 파이프라인: `/job/{taskCd}/job/{taskCd}-{envrnCd}`

이 규칙의 의미는 다음과 같다:

- Jenkins 폴더 계층이 TPS 도메인 구조를 반영한다.
- CRUD API 호출 전에 먼저 경로 모델을 결정해야 한다.
- 같은 비즈니스 파이프라인이라도 종류에 따라 생성 경로가 달라진다.

#### 3.1.1 실제 코드에서는 어떻게 만들었는가

구버전 TPS v3의 `JenkinsService`는 경로 조립을 메서드로 고정했다.

일반 파이프라인은 `returnPipelineFullPath()`에서 다음처럼 조립한다:

```java
return "/job/" + pipelineStructVo.getTaskCd()
        + "/job/" + pipelineStructVo.getEnvrnCd()
        + "/job/" + pipelineStructVo.getBizNm();
```

트리거 파이프라인은 `returnTriggerPipelineFullPath()`에서 다음처럼 조립한다:

```java
return "/job/" + pipelineStructVo.getTaskCd()
        + "/job/" + pipelineStructVo.getTaskCd() + "-" + pipelineStructVo.getEnvrnCd();
```

즉 TPS는 path string을 호출부에서 임시로 만들지 않고, `PipelineStructVo`를 받아 일관된 규칙으로 경로를 계산했다.

최근 `redpanda-playground` 쪽은 이 아이디어를 더 단순화해 `JenkinsJobSpec`에 경로 메서드를 넣었다.

```java
public String toApiPath() {
    return "/job/" + projectId + "/job/" + presetId + "/job/" + jobId;
}
```

즉 핵심은 같다. 경로는 요청 직전 문자열 조합으로 흩어지지 않고, 도메인 객체가 책임진다.

### 3.2 autoCreateFolder 패턴

TPS에서 파이프라인 생성은 "파이프라인만 생성"이 아니다. 상위 폴더가 없으면 폴더를 먼저 순서대로 만들어야 한다.

이 흐름이 `autoCreateFolder` 패턴이다.

핵심 절차는 다음과 같다:

1. 상위 폴더 존재 여부 조회
2. 없으면 Folder XML로 생성
3. 다음 단계 폴더 존재 여부 조회
4. 마지막에 파이프라인 생성

이 패턴을 쓰는 이유는 다음과 같다:

- Jenkins는 중간 폴더를 자동으로 만들지 않는다.
- 경로 모델이 깊을수록 폴더 생성 책임을 애플리케이션이 가져야 한다.
- 사용자는 "파이프라인 하나 생성"을 기대하지만, 내부적으로는 여러 요청이 이어진다.

#### 3.2.1 실제 코드에서는 어떻게 순서화했는가

구버전 TPS v3의 `autoCreateFolder()`는 이 과정을 진짜로 순차 실행했다.

일반 파이프라인이면 다음 순서다:

1. `taskCd`만 채운 `PipelineStructVo`로 존재 여부 확인
2. `envrnCd`까지 채운 `PipelineStructVo`로 다시 존재 여부 확인
3. 둘 다 없으면 `taskCd` 폴더 생성 후 `envrnCd` 폴더 생성
4. `taskCd`만 있으면 그 아래에 `envrnCd` 폴더만 생성
5. 둘 다 있으면 그대로 파이프라인 생성으로 진행

실제 생성 호출도 XML 본문보다 form 파라미터를 썼다:

```java
Map<String, String> bodyData = new HashMap<>();
bodyData.put("mode", "com.cloudbees.hudson.plugins.folder.Folder");
bodyData.put("name", folderName);
bodyData.put("from", "");
```

그리고 `createFolder()`는 결국 `POST {folderPath}/createItem`으로 보냈다.

최근 `redpanda-playground`의 `CreateJenkinsJobService`도 같은 구조를 더 읽기 쉽게 드러낸다:

```java
if (!jenkinsApiPort.exists(spec, spec.projectFolderApiPath())) {
    jenkinsApiPort.createFolder(spec, "", spec.projectId());
}

if (!jenkinsApiPort.exists(spec, spec.presetFolderApiPath())) {
    jenkinsApiPort.createFolder(spec, spec.projectFolderApiPath(), spec.presetId());
}

jenkinsApiPort.createPipelineJob(spec, spec.presetFolderApiPath(), spec.jobId());
```

즉 `autoCreateFolder`는 추상 개념이 아니라, "존재 확인 -> 부모 폴더 생성 -> 자식 폴더 생성 -> Job 생성"을 코드로 고정한 절차다.

### 3.3 upsert 패턴

TPS의 `createPipeline()`은 순수한 create보다 upsert에 가깝다.

핵심 동작은 다음과 같다:

- 먼저 존재 여부를 확인한다.
- 이미 있으면 생성 대신 수정으로 위임한다.
- 실행 직전에도 최신 Jenkinsfile로 갱신할 수 있다.

이 패턴의 장점은 다음과 같다:

- 중복 생성 실패를 줄인다.
- Jenkinsfile 변경을 실행 시점에 자연스럽게 반영한다.
- 마이그레이션 배치 없이 정의를 최신 상태로 유지할 수 있다.

#### 3.3.1 실제 코드에서는 create가 update로 바로 떨어진다

구버전 TPS v3의 `createPipeline()`은 이름 그대로 순수 create가 아니다.

핵심 분기는 다음과 같다:

```java
if (!this.existCheck(jenkinsToolVo, pipelineStructVo, triggerAt)) {
    return jenkinsFeignClient.createPipeline(...);
} else {
    return this.updatePipeline(jenkinsToolVo, jenkinsJobVo);
}
```

반대로 `updatePipeline()`도 대상이 없으면 다시 create로 되돌린다:

```java
if (this.existCheck(jenkinsToolVo, pipelineStructVo, triggerAt)) {
    return jenkinsFeignClient.updatePipeline(...);
} else {
    return this.createPipeline(jenkinsToolVo, jenkinsJobVo);
}
```

이 구조 때문에 외부에서 볼 때 CRUD API는 분리되어 있어도, 내부 구현은 사실상 reconciliation에 가까운 upsert 동작을 한다.

### 3.4 returnBooleanWithOut403 패턴

Jenkins는 삭제나 일부 생성 작업 뒤 `302` 리다이렉트를 반환할 수 있다. 클라이언트가 이를 따라가다 `403`처럼 보이는 응답을 받을 수 있다.

TPS는 이 구간을 그대로 오류로 보지 않고, 작업 특성에 맞게 성공으로 간주하는 보정 패턴을 사용한다.

이 패턴이 필요한 이유는 다음과 같다:

- 실제 작업은 성공했는데 HTTP 후속 처리 때문에 실패처럼 보일 수 있다.
- Jenkins의 브라우저 지향 응답과 API 클라이언트 기대치가 다를 수 있다.
- CRUD 흐름을 안정적으로 이어가려면 어떤 `403`이 실질 성공 뒤의 부산물인지 구분해야 한다.

#### 3.4.1 실제 보정 로직은 매우 단순했다

구버전 TPS v3의 `returnBooleanWithOut403()`는 이름 그대로 `403`을 예외 처리했다.

```java
private boolean returnBooleanWithOut403(Response response) {
    HttpStatus httpStatus = HttpStatus.valueOf(response.status());
    if (httpStatus.isSameCodeAs(HttpStatusCode.valueOf(403))) return true;
    return !httpStatus.is4xxClientError() && !httpStatus.is5xxServerError();
}
```

즉 삭제, 폴더 생성, 중지 같은 일부 요청은 "403이면 무조건 인증 실패"로 보지 않고, Jenkins UI 리다이렉트 부산물일 수 있다고 전제했다.

이 패턴은 위험도 있다. 그래서 현대화 이후에는 "모든 403을 성공으로 본다"보다, 어떤 endpoint에서만 이 보정을 허용할지 더 좁히는 편이 낫다.

### 3.5 Freemarker 기반 config.xml 생성 패턴

v3에서는 config.xml을 문자열 이어붙이기보다 템플릿 엔진으로 렌더링하는 쪽이 더 안정적이다.

Freemarker를 쓰는 이유는 다음과 같다:

- Jenkinsfile, 파라미터, 트리거, 보관 정책을 변수로 주입할 수 있다.
- XML 구조 변경이 필요할 때 템플릿 교체로 대응할 수 있다.
- 문자열 연결보다 구조 누락이나 escaping 실수를 줄일 수 있다.

즉 CRUD의 핵심은 단순 POST가 아니라 유효한 config.xml을 안정적으로 생성하는 것이다.

#### 3.5.1 실제 템플릿은 무엇을 고정했는가

구버전 TPS v3의 `FreemarkerService`는 템플릿 로딩 경로를 `/template/jenkins/v3`로 고정하고, Job 종류에 따라 다른 템플릿을 선택했다.

핵심 분기는 다음과 같다:

```java
return this.processFreemarkerHandler(
    objectMap,
    isDevYn ? "JenkinsJobConfigXml.ftl" : "JenkinsJobConfigXmlWithTicket.ftl"
);
```

일반 Jenkins 기반 Job 템플릿은 다음 요소를 고정했다:

- `<flow-definition>`
- `BuildDiscarderProperty`
- 파라미터가 있으면 `ParametersDefinitionProperty`
- `CpsFlowDefinition`
- `<script><![CDATA[...]]></script>`
- `<sandbox>true</sandbox>`

반대로 Git 기반 Job 템플릿은 `CpsScmFlowDefinition`으로 바뀌고, `cloneUrl`, `branch`, `credentialsId`, `scriptPath`를 주입했다.

즉 TPS가 렌더링한 것은 단순 Jenkinsfile 문자열이 아니다. Job 종류에 따라 Jenkins가 해석할 XML 루트 구조 자체를 바꿨다.

최근 `redpanda-playground`의 `JenkinsJobSpec`은 이 부분을 더 작게 가져간다. Freemarker 없이 `toConfigXml()`에서 최소 `flow-definition` XML을 직접 만든다.

이 차이는 역할 차이에서 나온다. TPS는 다양한 파이프라인 유형과 파라미터 정책을 다뤄야 했고, playground는 create-only 성격이 강해서 최소 XML만으로도 충분했다.



## 4. 현대 Jenkins가 CRUD 코드에 주는 영향

### 4.1 인증 단순화

`05-02a`에서 설명한 것처럼 API Token 중심 환경에서는 POST마다 crumb과 cookie를 따로 관리할 필요가 줄어든다.

이 변화는 CRUD API 코드에 직접 영향을 준다:

- 생성 POST 단순화
- 수정 POST 단순화
- 삭제 POST 단순화
- Jenkinsfile 검증 POST 단순화

즉 CRUD endpoint 자체는 그대로인데, 호출 전 인증 준비 코드가 얇아진다.

### 4.2 403 해석 단순화

이전 방식에서는 `403`이 다음 둘 중 하나일 수 있었다:

- crumb 누락
- 권한 부족

현대 Token 환경에서는 crumb 변수가 줄어들어 `403` 해석이 더 단순해진다. 즉 권한 문제를 더 직접적으로 의심할 수 있다.



## 5. 최근 플러그인과 버전 변화가 주는 영향

### 5.1 Groovy Libraries 플러그인 분리

Jenkins 2.442 이후 `Pipeline: Groovy Libraries`가 코어에서 별도 플러그인으로 분리되었다.

이 변화가 CRUD와 연결되는 지점은 `checkScriptCompile`이다.

영향은 다음과 같다:

- `@Library('my-lib')` 사용 시 플러그인 미설치면 검증 실패 가능
- config.xml 구조보다 검증 단계에서 차이가 먼저 드러남
- Jenkinsfile 템플릿이 공유 라이브러리를 전제한다면 운영 환경 플러그인 상태도 같이 봐야 함

### 5.2 버전보다 더 중요한 것

버전 숫자 자체보다 더 중요한 것은 다음이다:

- 현재 인증 수단이 무엇인가
- 현재 Jenkins에 필요한 플러그인이 설치되어 있는가
- 현재 Jenkins 응답을 클라이언트가 어떻게 해석하는가

즉 CRUD 안정성은 endpoint 스펙보다 운영 환경 조합에 더 크게 영향을 받는다.



## 6. TPS 코드 구조에 주는 시사점

CRUD를 안정적으로 구현하려면 책임을 아래처럼 나누는 편이 낫다:

- 경로 조립 책임
- 인증 컨텍스트 준비 책임
- XML 렌더링 책임
- Jenkins 호출 책임
- 응답 해석 책임

이렇게 나누는 이유는 다음과 같다:

- 경로 규칙이 바뀌어도 전부를 다시 고치지 않게 하기 위해
- 인증 방식이 바뀌어도 CRUD 로직을 그대로 두기 위해
- XML 템플릿이 바뀌어도 호출 코드를 최소 수정하기 위해

### 6.1 실제 클래스 기준으로 보면 더 명확하다

구버전 TPS v3 기준으로 책임은 대략 이렇게 나뉜다:

- `JenkinsService`: CRUD 오케스트레이션, 존재 확인, 폴더 생성 순서 제어, 응답 해석
- `JenkinsFeignClient`: Jenkins endpoint 선언
- `FreemarkerService`: config.xml과 Groovy 스크립트 렌더링
- `PipelineStructVo`: Jenkins 경로 조립의 입력 모델
- `JenkinsAuthVo`: Basic Auth, crumb, cookie 묶음

최근 `redpanda-playground`는 같은 책임을 더 작은 조합으로 재배치했다:

- `CreateJenkinsJobService`: 생성 흐름 오케스트레이션
- `JenkinsRestAdapter`: 실제 HTTP 호출
- `JenkinsJobSpec`: 경로와 XML 생성
- `JenkinsPathBuilder`: 입력 값을 `JenkinsJobSpec`으로 조립

즉 문서에서 말한 책임 분리는 추상 설계가 아니라, 실제 코드에서도 클래스로 나뉘어 있었다.



## 7. Jenkinsfile 검증 3계층 모델

> API 스펙(`05-03` 3-4절)에서 `checkScriptCompile`과 `pipeline-model-converter/validate`의 요청/응답 형식을 다뤘다. 이 절에서는 **두 API가 실제로 무엇을 잡고 무엇을 놓치는지**, 그리고 이를 어떻게 조합해야 하는지를 다룬다.

### 핵심 모델: 3개의 검증 계층

Jenkinsfile 검증은 하나의 API로 완결되지 않는다. 실무에서는 세 개의 계층으로 나눠서 생각해야 한다:

| 계층 | 검증 대상 | 도구 |
|------|----------|------|
| **1. Declarative 껍질** | `pipeline/stages/stage/steps/agent` 배치 규칙 | `pipeline-model-converter/validate` |
| **2. script {} 내부 문법** | Groovy/CPS 컴파일 가능성 | `checkScriptCompile` |
| **3. 실행 가능성** | agent 존재, Docker 설치, 디렉터리/권한/플러그인 | 실제 run (smoke test) |

이 구분이 필요한 이유는, Declarative Pipeline 안에 `script {}` 블록이 들어가면 **두 가지 문법 세계가 공존**하기 때문이다. 

1. Jenkins 공식 Declarative AST 문서는 `script` 블록을 "Declarative subset 검증 없이 실행되는 special step"으로 설명한다([ModelASTScriptBlock Javadoc](https://javadoc.jenkins.io/plugin/pipeline-model-api/org/jenkinsci/plugins/pipeline/modeldefinition/ast/package-summary.html)). 
2. 즉 `pipeline-model-converter/validate`는 `script {}` 안쪽을 깊게 보지 않는다.

### 계층 1: `pipeline-model-converter/validate`가 잘 잡는 것

이 API는 **Declarative 구조 규칙 위반**을 잡는 데 강하다.

**예시 A. `steps` 위치 누락**

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            sh 'echo hi'   // steps {} 없이 바로 step 호출
        }
    }
}
```

- `validate`: **실패** (Declarative 구조 위반),  `checkScriptCompile`: 통과할 수 있음

**예시 B. 잘못된 `environment` 식별자**

```groovy
pipeline {
    agent any
    environment {
        1ABC = 'x'     // 숫자로 시작하는 식별자
    }
    stages {
        stage('Build') { steps { echo 'hi' } }
    }
}
```

- `validate`: **실패** (`ModelValidatorImpl`이 식별자 규칙 검증), `checkScriptCompile`: 결과가 다를 수 있음

**예시 C. Declarative 규칙상 허용되지 않는 배치**

```groovy
pipeline {
    agent any
    stages {
        stage('Parallel') {
            agent { label 'linux' }   // parallel 포함 stage에서 agent 금지
            parallel {
                stage('A') { steps { echo 'a' } }
                stage('B') { steps { echo 'b' } }
            }
        }
    }
}
```

- `validate`: **실패** ("agent is not allowed in stage as it contains parallel stages"), `checkScriptCompile`: 통과할 수 있음

### 계층 2: `checkScriptCompile`이 더 잘 잡는 것

이 API는 **`script {}` 내부의 Groovy/CPS 문법 오류**를 잡는 데 강하다.

**예시 D. 괄호/중괄호 문법 오류**

```groovy
script {
    def services = ['api', 'web', 'worker'
    services.each { svc ->
        builds[svc] = { sh "docker build -t ${svc} ./${svc}" }
    }
}
```

→ `checkScriptCompile`: **실패** (닫는 괄호 누락)
→ `validate`: `script {}` 안쪽을 깊게 보지 않으므로 바깥 구조만 맞으면 통과 가능

**예시 E. 잘못된 Groovy 연산자**

```groovy
script {
    services.each { svc =>     // => 는 Groovy에서 유효하지 않음
        builds[svc] = { sh "echo ${svc}" }
    }
}
```

→ `checkScriptCompile`: **실패** (Groovy 파서 오류)
→ `validate`: 바깥 Declarative 구조가 맞으면 통과 가능

**예시 F. 따옴표 누락**

```groovy
script {
    sh "docker build -t ${svc} ./${svc}
}
```

→ `checkScriptCompile`: **실패** (문자열 리터럴 미종료)

### 계층 3: 두 API 모두 못 잡는 것 (실제 run 필요)

**예시 G. 런타임 null 참조**

```groovy
script {
    def services = null
    services.each { svc ->
        builds[svc] = { sh "docker build -t ${svc} ./${svc}" }
    }
    parallel builds
}
```

→ 문법은 정상. 실행하면 `NullPointerException`.

**예시 H. step 이름 오타**

```groovy
script {
    parllel builds    // parallel 오타
}
```

→ Groovy 문법은 정상 (메서드 호출). Jenkins step resolution은 런타임에 일어나므로 컴파일 단계에서 통과 가능.

**예시 I. CPS 변환 비호환**

Jenkins Pipeline은 Groovy를 CPS(Continuation-Passing Style)로 변환한다. 일부 Groovy idiom(`@NonCPS` 없는 직렬화 불가 객체 등)은 문법상 맞아도 실행 중에 깨진다. `checkScriptCompile`이 컴파일은 통과시키지만 CPS 런타임에서 실패하는 케이스다.

**예시 J. 환경/플러그인/경로 부재**

```groovy
sh "docker build -t ${svc} ./${svc}"
```

`./api` 디렉터리 부재, Docker 미설치, 특정 플러그인 미설치 등은 검증 API가 구조적으로 알 수 없는 영역이다.

### 테스트 케이스 요약표

| # | 수정 내용 | `validate` | `checkScriptCompile` | 실제 run |
|---|----------|------------|----------------------|---------|
| A | `steps {}` 누락 | **실패** | 통과 가능 | 실패 |
| B | 숫자 시작 env 식별자 | **실패** | 결과 다를 수 있음 | 실패 |
| C | parallel stage에 agent 배치 | **실패** | 통과 가능 | 실패 |
| D | 괄호 누락 (script 내부) | 통과 가능 | **실패** | 실패 |
| E | `=>` 잘못된 연산자 | 통과 가능 | **실패** | 실패 |
| F | 따옴표 미종료 | 통과 가능 | **실패** | 실패 |
| G | `null.each {}` | 통과 | 통과 | **실패** |
| H | step 이름 오타 | 통과 | 통과 가능 | **실패** |
| I | CPS 비호환 Groovy | 통과 | 통과 | **실패** |
| J | Docker/디렉터리 부재 | 통과 | 통과 | **실패** |

### 실무 권장: 3단계 검증 파이프라인

```text
1. pipeline-model-converter/validate  →  Declarative 껍질 검증
2. checkScriptCompile                 →  script {} 내부 Groovy/CPS 문법 보조 검증
3. 짧은 smoke run                     →  실제 agent에서 최소 1개 분기 실행 확인
```

이 순서로 구성하면 각 계층이 잡는 영역이 겹치지 않으면서도 빈틈을 줄일 수 있다. 

- 1번과 2번은 API 호출이므로 수 초 내에 끝나고, 3번만 실제 빌드 비용이 든다. 
- TPS에서 Jenkinsfile을 외부 시스템이 생성·주입하는 구조라면, 최소한 1번은 필수로 적용하고 2번은 `script {}` 블록이 포함된 경우에만 추가하는 것이 현실적이다.

## 9. 핵심 정리

이 문서의 핵심은 다음과 같다:

1. Jenkins 파이프라인 CRUD는 단순 REST CRUD가 아니라 경로 관리, XML 렌더링, 보안 처리가 결합된 작업이다.
2. TPS는 이를 `pipelineStruct`, `autoCreateFolder`, `upsert`, `returnBooleanWithOut403`, 템플릿 기반 XML 생성으로 다룬다.
3. 현대 Jenkins 환경에서는 endpoint보다 인증 준비 코드가 더 크게 단순화된다.
4. Jenkinsfile 검증은 단일 API로 완결되지 않는다. Declarative 껍질(`validate`) → script 내부 문법(`checkScriptCompile`) → 실제 run의 3계층으로 구성해야 빈틈이 줄어든다.
