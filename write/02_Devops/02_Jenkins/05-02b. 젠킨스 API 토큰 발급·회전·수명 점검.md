# 젠킨스 API 토큰 발급·회전·수명 점검
---
> 이 문서는 Jenkins API Token을 발급하고, 이름을 바꾸고, 폐기하고, 현재 버전에서 수명을 어떻게 점검할지 설명하는 실습형 문서다.
>
> - 현재 검증 기준은 Jenkins `2.462.3`이다.
> - 현재 버전에서는 토큰 "만료 시각"을 다루는 표준 REST 흐름보다 발급·회전 중심으로 보는 편이 맞다.

## 1. 이 문서의 결론

현재 Jenkins `2.462.3` 기준으로 먼저 결론부터 정리하면 다음과 같다:

- 토큰 발급, 이름 변경, 폐기용 endpoint는 있다.
- 현재 버전 기준으로 토큰 발급 요청은 `newTokenName`만 받는다.
- 현재 버전 기준으로 "토큰 만료 시각"을 직접 지정하거나 조회하는 표준 REST 흐름은 문서화하기 어렵다.
- 대신 Jenkins는 토큰의 생성 시점과 마지막 사용 시점 기반으로 회전을 유도한다.
- 공식 안내 기준으로 토큰은 6개월이 지나면 회전을 고려하고, 12개월이 지나면 더 강하게 교체를 권장한다.

즉 현재 문서에서 말하는 "만료 체크"는 보통 아래 의미로 본다:

- 이 토큰이 언제 만들어졌는가
- 마지막으로 언제 사용됐는가
- 내부 정책상 언제 교체할 것인가

자동 만료 시간이 박혀 있는 형태와는 다르다.



## 2. 이 문서의 범위

> 이 문서는 API Token 관련 아래 흐름만 설명한다.
>
> - 토큰 발급
> - 토큰 이름 변경
> - 특정 토큰 폐기
> - 전체 토큰 폐기
> - 현재 버전에서 수명 점검을 어떻게 해석할지

기본 인증 확인과 crumb/cookie 준비는 별도 문서에서 다룬다:

- `05-02. 젠킨스 인증 API 스펙 (ID-Password + Crumb).md`
- `05-02a. 젠킨스 인증 모델과 TPS 패턴 (2.222+).md`

이 문서의 endpoint 경로는 Jenkins 공식 `DescriptorImpl` 메서드명과 Stapler URL 규칙을 바탕으로 정리했다. 즉 아래 경로 구성은 공식 Java API에서 `doGenerateNewToken`, `doRename`, `doRevoke`, `doRevokeAll` 메서드가 존재한다는 점을 근거로 한 실무형 정리다.



## 3. 공통 요청 규칙

> 현재 환경처럼 아직 토큰이 주력이 아니고 `ID/Password + crumb + cookie`가 기본이면, 토큰 발급 요청도 같은 인증 재료로 보낸다.

먼저 변수부터 준비한다:

```bash
export TOKEN_API_PATH="/user/${JENKINS_USER}/descriptorByName/jenkins.security.ApiTokenProperty"
export TOKEN_NAME='api-doc-token-1'
export TOKEN_UUID=''
```

`05-02`를 먼저 수행했다면 아래 값은 이미 준비된 상태다:

- `JENKINS_URL`
- `JENKINS_USER`
- `JENKINS_PASS`
- `cookies.txt`
- `crumb.json`
- `CRUMB`
- `CRUMB_FIELD`

현재 문서에서 의미하는 값은 다음과 같다:

| 변수 | 의미 | 예시 |
|------|------|------|
| `TOKEN_API_PATH` | 현재 사용자 기준 API Token 관리 경로 | `/user/admin/descriptorByName/jenkins.security.ApiTokenProperty` |
| `TOKEN_NAME` | 새로 만들 토큰 이름 | `api-doc-token-1` |
| `TOKEN_UUID` | 기존 토큰 식별자 | `3b1d...` |

응답 확인 원칙은 다음과 같다:

- 토큰 발급은 body를 파일로 저장해서 확인한다.
- 이름 변경/폐기는 본문보다 상태 코드가 중요하다.
- 새 plain token 값은 발급 직후 응답에서만 확인하고, 이후 다시 조회되지 않는다고 보는 편이 안전하다.
- 발급 직후에는 실제 읽기 또는 쓰기 API 하나로 토큰이 바로 동작하는지 확인하는 편이 안전하다.



## 4. 토큰 발급 API

### 4-1. `POST /user/{userId}/descriptorByName/jenkins.security.ApiTokenProperty/generateNewToken`

> 현재 사용자 기준 새 API Token을 발급한다.

요청 형식은 다음과 같다:

```http
POST /user/{userId}/descriptorByName/jenkins.security.ApiTokenProperty/generateNewToken HTTP/1.1
Authorization: Basic <...>
Jenkins-Crumb: <crumb>
Cookie: <session-cookie>
Content-Type: application/x-www-form-urlencoded
```

현재 버전 기준으로 가장 중요한 입력값은 `newTokenName`이다.

```bash
curl -k -sS -D headers.txt -o token-create-response.json -w 'HTTP_STATUS=%{http_code}\n' \
  -X POST -b cookies.txt \
  -u "${JENKINS_USER}:${JENKINS_PASS}" \
  -H "${CRUMB_FIELD}: ${CRUMB}" \
  --data-urlencode "newTokenName=${TOKEN_NAME}" \
  "${JENKINS_URL}${TOKEN_API_PATH}/generateNewToken"

cat headers.txt
cat token-create-response.json
```

응답이 JSON이면 다음처럼 볼 수 있다:

```bash
jq . token-create-response.json
```

현재 Jenkins `2.462.3` 계열 환경에서는 성공 응답이 다음처럼 내려올 수 있다:

```json
{
  "status": "ok",
  "data": {
    "tokenName": "api-doc-token-1",
    "tokenUuid": "0b7f3d2a-4c67-4c2d-9c0e-123456789abc",
    "tokenValue": "11b6d5d2b4f84f5f9a0c8f2b9d1e3a4c56"
  }
}
```

환경이나 래핑 방식에 따라 JSON 최상위에 `status`, `data`가 한 번 더 감싸질 수 있다. 따라서 실무에서는 최소한 아래 3개 값을 기준으로 응답을 읽는 편이 안전하다:

- `tokenUuid`: 이후 rename/revoke 시 식별자로 사용
- `tokenName`: 사람이 보는 표시 이름
- `tokenValue`: 발급 직후 한 번만 확인 가능한 plain token 값

현재 Jenkins 구현상 plain token 값은 발급 직후에만 확인 가능한 것으로 이해하는 편이 맞다. 따라서 발급 직후 응답 body를 바로 확인해야 한다.

### 4-2. 발급한 토큰으로 실제 API 호출 예시

토큰이 실제 호출에 바로 쓸 수 있는지 확인하려면, 발급 직후 가벼운 읽기 API나 단순한 설정 변경 API를 한 번 호출해 보는 편이 안전하다.

먼저 발급 응답에서 `tokenValue`를 꺼내 shell 변수로 저장한다:

```bash
export JENKINS_TOKEN=$(jq -r '.tokenValue // .data.tokenValue' token-create-response.json)
```

`401 Unauthorized`가 나오면 가장 먼저 이 값을 확인한다:

```bash
printf 'TOKEN_LEN=%s\n' "${#JENKINS_TOKEN}"
printf 'TOKEN_PREVIEW=%s\n' "${JENKINS_TOKEN:0:8}"
```

길이가 `0`이거나 `null` 문자열이면 `jq` 경로를 잘못 읽은 것이다. 실제 응답이 `{"status":"ok","data":{...}}` 형태라면 반드시 `.data.tokenValue`를 꺼내야 한다.

예를 들어 기존 파이프라인의 `config.xml`을 토큰으로 바로 덮어쓰는 예시는 다음과 같다:

```bash
export PIPELINE_STRUCT='/job/SBH/job/TEST'

curl -k -sS -i -w '\nHTTP_STATUS=%{http_code}\n' -X POST \
  -u "${JENKINS_USER}:${JENKINS_TOKEN}" \
  -H "Content-Type: application/xml" \
  --data-binary @05-03.sample-pipeline-config.xml \
  "${JENKINS_URL}${PIPELINE_STRUCT}/config.xml"
```

- Jenkins 공식 안내 기준으로 API 토큰 인증 요청은 보통 CSRF crumb 예외 대상이다. 따라서 일반적인 구성에서는 이 요청에 crumb과 session cookie를 따로 붙이지 않는다.
- 즉 비밀번호 인증 POST는 `Authorization + crumb + cookie` 조합으로 보고, API 토큰 POST는 보통 `Authorization + Content-Type`만 있으면 된다고 이해하면 된다.

에러 케이스는 다음과 같다:

| 상태 코드 | 의미 | 대응 |
|-----------|------|------|
| `200` | 발급 요청 처리 성공 | 응답 body에서 토큰 값 확인 |
| `400` | 요청 형식 오류 | `newTokenName` 확인 |
| `403` | 권한 부족 또는 crumb 문제 | 인증/권한 확인 |
| `404` | 경로 오류 | `TOKEN_API_PATH` 확인 |



## 5. 토큰 이름 변경 API

### 5-1. `POST /user/{userId}/descriptorByName/jenkins.security.ApiTokenProperty/rename`

> 기존 토큰의 표시 이름을 바꾼다.

요청 형식은 다음과 같다:

```http
POST /user/{userId}/descriptorByName/jenkins.security.ApiTokenProperty/rename HTTP/1.1
Authorization: Basic <...>
Jenkins-Crumb: <crumb>
Cookie: <session-cookie>
Content-Type: application/x-www-form-urlencoded
```

예시는 다음과 같다:

```bash
export TOKEN_UUID='PUT_TOKEN_UUID_HERE'
export TOKEN_NAME='api-doc-token-renamed'

curl -k -sS -D headers.txt -o /dev/null -w 'HTTP_STATUS=%{http_code}\n' \
  -X POST -b cookies.txt \
  -u "${JENKINS_USER}:${JENKINS_PASS}" \
  -H "${CRUMB_FIELD}: ${CRUMB}" \
  --data-urlencode "tokenUuid=${TOKEN_UUID}" \
  --data-urlencode "newName=${TOKEN_NAME}" \
  "${JENKINS_URL}${TOKEN_API_PATH}/rename"

cat headers.txt
```



## 6. 토큰 폐기 API

### 6-1. `POST /user/{userId}/descriptorByName/jenkins.security.ApiTokenProperty/revoke`

> 특정 토큰 하나를 폐기한다.

예시는 다음과 같다:

```bash
export TOKEN_UUID='PUT_TOKEN_UUID_HERE'

curl -k -sS -D headers.txt -o /dev/null -w 'HTTP_STATUS=%{http_code}\n' \
  -X POST -b cookies.txt \
  -u "${JENKINS_USER}:${JENKINS_PASS}" \
  -H "${CRUMB_FIELD}: ${CRUMB}" \
  --data-urlencode "tokenUuid=${TOKEN_UUID}" \
  "${JENKINS_URL}${TOKEN_API_PATH}/revoke"

cat headers.txt
```

### 6-2. `POST /user/{userId}/descriptorByName/jenkins.security.ApiTokenProperty/revokeAll`

> 현재 사용자의 토큰을 전부 폐기한다.

예시는 다음과 같다:

```bash
curl -k -sS -D headers.txt -o /dev/null -w 'HTTP_STATUS=%{http_code}\n' \
  -X POST -b cookies.txt \
  -u "${JENKINS_USER}:${JENKINS_PASS}" \
  -H "${CRUMB_FIELD}: ${CRUMB}" \
  "${JENKINS_URL}${TOKEN_API_PATH}/revokeAll"

cat headers.txt
```

`revokeAll`은 영향이 크므로 실습 환경에서만 실행하는 편이 맞다.



## 7. 만료 시간 체크는 어떻게 보나

### 7-1. 현재 Jenkins 2.462.3 기준

현재 버전 기준으로는 토큰이 "몇 시 몇 분에 자동 만료된다"는 식의 표준 REST 흐름으로 보기 어렵다.

공식 근거는 두 가지다:

- Jenkins `2.462.3`의 `ApiTokenProperty` Java API는 `generateNewToken(String name)`까지만 보인다.
- 같은 버전의 `DescriptorImpl`도 `doGenerateNewToken(User u, String newTokenName)`까지만 보인다.

즉 현재 버전 기준으로는 발급 시점에 `expirationDate`를 같이 넣는 인터페이스가 보이지 않는다.

### 7-2. 현재 버전에서 실제로 점검할 것

현재 버전에서 수명 점검은 보통 아래 기준으로 해석한다:

- 생성된 지 얼마나 지났는가
- 마지막으로 언제 사용됐는가
- 조직 정책상 몇 개월 주기로 교체할 것인가

Jenkins 공식 안내는 다음과 같다:

- 6개월이 지나면 회전 필요성을 더 강하게 드러낸다.
- 12개월이 지나면 더 강하게 교체를 권장한다.

즉 현재 환경에서는 "절대 만료 시각"보다는 "회전 권장 시점"으로 이해하는 편이 정확하다.

### 7-3. 운영에서 권장하는 방법

현재 버전 기준으로는 다음 방식이 가장 실용적이다:

- 토큰 이름에 용도와 발급월을 넣는다.
  - 예: `tps-api-202604`
- 발급 시점을 별도 문서나 시스템 메타데이터에 기록한다.
- 6개월 주기로 재발급 후 이전 토큰을 폐기한다.
- 장기 미사용 토큰은 last-use 기준으로 정리한다.

### 7-4. 최신 버전과의 차이

최신 Jenkins Java API 문서에는 `generateNewToken(String name, LocalDate expirationDate)` 형태가 보인다. 따라서 더 최신 버전에서는 만료일 개념이 코어 레벨에 들어간 것으로 읽힌다.

다만 현재 검증 환경은 `2.462.3`이므로, 이 문서에서는 최신 코어 시그니처를 바로 실사용 전제로 쓰지 않는다. 버전 업그레이드 후 실제 endpoint와 응답 형식을 다시 검증하는 편이 맞다.



## 8. 운영 설계 의견

> 여러 Jenkins 인스턴스를 운영하면서 토큰 기반 호출로 전환하려면, 토큰 발급 API 자체보다 저장 구조와 회전 전략을 먼저 분리해서 설계하는 편이 맞다.

최종 판단은 다음처럼 정리할 수 있다:

- 여러 Jenkins + DB 메타관리: 좋다.
- 헬스 체크 별도 관리: 좋다.
- 토큰으로 crumb 제거 시도: 좋다.
- 토큰 원문을 DB 저장: 비추천이다.
- 헬스 체크와 토큰 회전 결합: 비추천이다.
- 비밀번호 완전 제거: 가능하지만 부트스트랩/복구 수단 설계 후 진행하는 편이 맞다.

### 8-1. 왜 DB 메타관리는 괜찮은가

여러 Jenkins를 운영하면 인스턴스별 URL, 계정, 버전, 인증 방식, capability를 DB 메타데이터로 관리하는 것은 자연스럽다.

예를 들어 DB에는 아래 같은 정보가 들어갈 수 있다:

- Jenkins base URL
- username
- 현재 인증 모드
- 버전
- 마지막 헬스 체크 시각
- 마지막 토큰 회전 시각

이 정보는 운영 메타데이터라서 DB 저장이 적절하다.

### 8-2. 왜 토큰 원문 DB 저장은 비추천인가

토큰 원문은 사실상 비밀번호와 비슷한 비밀값이다. 따라서 DB 평문 저장은 비밀번호를 DB에 넣는 것과 보안 성격이 크게 다르지 않다.

더 안전한 방식은 다음과 같다:

- DB에는 `secret_ref`만 둔다.
- 실제 토큰 원문은 Vault, KMS, Secret Manager 같은 비밀 저장소에 둔다.
- 애플리케이션은 `secret_ref`를 통해 실제 토큰을 읽는다.

즉 DB는 "관리 정보", 비밀 저장소는 "실제 비밀값"으로 분리하는 편이 맞다.

### 8-3. 왜 헬스 체크와 회전을 분리해야 하는가

헬스 체크와 토큰 회전은 주기와 실패 성격이 다르다.

- 헬스 체크: 짧은 주기로 계속 확인하는 운영 상태 점검
- 토큰 회전: 월 단위나 정책 단위로 수행하는 보안 작업

이 둘을 같은 스케줄러나 같은 실패 정책으로 묶으면 다음 문제가 생길 수 있다:

- 일시적인 네트워크 장애가 토큰 회전 실패로 이어진다.
- 회전 실패가 헬스 상태 이상으로 과도하게 전파된다.
- 운영 상태 수집 로직과 보안 변경 로직이 한 곳에 섞인다.

따라서 보통은 다음처럼 나누는 편이 낫다:

- 헬스 체크 작업: `/api/json` 같은 읽기 API로 생존 확인
- 토큰 회전 작업: 발급 -> 검증 -> 전환 -> 이전 토큰 폐기

### 8-4. 토큰 회전은 어떻게 보는가

토큰 회전은 "새 토큰을 만들자마자 기존 토큰을 바로 지우는 방식"보다 중첩 전환 방식이 안전하다.

권장 순서는 다음과 같다:

1. 기존 인증 수단으로 새 토큰을 발급한다.
2. 새 토큰으로 `/api/json` 같은 가벼운 호출을 검증한다.
3. 애플리케이션의 active secret을 새 토큰으로 전환한다.
4. 짧은 유예 후 이전 토큰을 폐기한다.

이렇게 해야 발급 성공 이후 저장 실패, 저장 성공 이후 인증 실패 같은 중간 장애를 견딜 수 있다.

### 8-5. 왜 비밀번호 완전 제거 전에 부트스트랩 설계가 필요한가

토큰 기반으로 운영 호출을 단순화하는 것은 좋다. 하지만 토큰 발급 자체는 이미 인증 가능한 수단이 있어야 가능하다.

즉 다음 질문이 반드시 남는다:

- 새 토큰을 누가 발급하는가
- 현재 active token이 망가지면 무엇으로 복구하는가
- 전체 토큰이 폐기되면 어떤 자격증명으로 다시 들어가는가

이 문제를 해결하는 최소 설계가 부트스트랩/복구 수단이다.

실무적으로는 다음 중 하나가 필요하다:

- Jenkins 관리자 비밀번호를 별도 비밀 저장소에 보관
- 회전 전용 관리자 토큰을 따로 두고 일반 호출 토큰과 분리
- 수동 복구 절차와 운영 문서 마련

즉 운영 호출은 토큰으로 단순화하더라도, 복구 경로까지 토큰 하나에 전부 의존하는 구조는 피하는 편이 안전하다.
