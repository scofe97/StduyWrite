## Q1: DinD와 DooD의 차이와 트레이드오프는?

| 구분 | DinD (Docker-in-Docker) | DooD (Docker-outside-of-Docker) |
|------|------------------------|--------------------------------|
| 동작 방식 | 컨테이너 내부에 별도 Docker daemon 실행 | 호스트의 `/var/run/docker.sock` 마운트 |
| 필요 권한 | `--privileged` 필수 | `--privileged` 불필요 |
| 보안 위험 | 컨테이너 탈출(container escape) 공격에 취약 | socket 접근 = 사실상 호스트 root 접근과 동등 |
| 레이어 캐시 | 독립 관리 → 반복 빌드 느림 | 호스트 캐시 공유 → 반복 빌드 빠름 |

보안이 최우선인 Kubernetes 환경에서는 두 방식 모두 부족하므로 Kaniko나 Buildah를 사용한다.

## Q2: Docker Agent를 사용하면 어떤 문제가 해결되는가?

호스트 직접 빌드는 Agent에 설치된 도구 버전에 의존하므로 "이 Agent에서만 실패"하는 비결정적 빌드가 발생한다.

- **버전 고정**: 이미지 태그로 도구 버전을 고정하므로 어디서 실행해도 동일한 결과를 보장한다.
- **빌드 간 격리**: 각 빌드가 독립된 컨테이너에서 실행되므로 빌드 간 간섭이 원천 차단된다.
- **동작 원리**: 지정된 이미지로 컨테이너를 생성하면서 워크스페이스 디렉토리를 볼륨 마운트 → 스텝들이 컨테이너 내부에서 실행 → 완료 후 컨테이너 삭제.

주의: Docker Agent는 빌드 "환경"을 컨테이너화하는 것이지, Docker 이미지를 빌드하는 것이 아니다. Docker Agent 안에서 `docker build`를 실행하려면 DinD/DooD/Kaniko 같은 별도 전략이 필요하다.

## Q3: `latest` 태그를 프로덕션에서 사용하면 안 되는 이유는?

- **비결정성**: 여러 브랜치에서 동시에 빌드가 발생하면 어떤 커밋의 이미지가 `latest`인지 보장할 수 없다.
- **롤백 불가**: "이전 latest"라는 개념이 없으므로 배포 후 롤백이 필요할 때 어떤 이미지로 돌아가야 하는지 알 수 없다.
- **Kubernetes 캐시 문제**: `imagePullPolicy`가 `IfNotPresent`(기본값)일 때 노드에 `latest`가 캐시되어 있으면 새 버전을 pull하지 않아, 노드마다 다른 버전이 실행되는 불일치가 생긴다.

권장 전략:

| 전략 | 예시 | 장점 |
|------|------|------|
| Git SHA 기반 | `myapp:a1b2c3d` | 실행 중 이미지에서 정확한 소스 버전 즉시 확인 |
| SemVer + Git SHA | `myapp:2.1.0-a1b2c3d` | 버전 의미 + 추적 가능성 동시 확보 |

## Q4: Kaniko가 Docker socket 마운트보다 안전한 이유는?

`/var/run/docker.sock`에 접근할 수 있는 프로세스는 `docker run -v /:/host alpine chroot /host`를 실행하여 호스트의 root 쉘을 얻는 것과 동일한 효과를 낼 수 있다.

- **Kubernetes 제약**: Pod Security Standards `restricted` 프로파일은 호스트 path 마운트와 privileged 컨테이너를 금지한다. 보안 정책을 준수하면서 이미지를 빌드하려면 Docker daemon에 의존하지 않는 도구가 필요하다.
- **Kaniko 동작**: Dockerfile의 각 명령을 사용자 공간에서 실행하여 파일시스템 스냅샷 기반으로 레이어를 생성한다. `--privileged`도 socket 마운트도 필요 없이 일반 컨테이너 권한으로 동작한다.
- **트레이드오프**: 레이어 캐싱이 Docker만큼 효율적이지 않아 빌드 속도가 느릴 수 있다.

## Q5: Multi-stage Dockerfile이 CI/CD에서 중요한 이유는?

이미지 크기와 보안을 동시에 개선한다.

- **크기 절감**: Node.js Single-stage 빌드 이미지는 1GB를 넘을 수 있지만, Multi-stage로 빌드된 nginx 서빙 이미지는 30~50MB 수준이다. 이미지가 작을수록 레지스트리 전송 시간과 컨테이너 시작 시간이 줄어든다.
- **보안 강화**: 최종 이미지에 컴파일러·패키지 매니저·개발 도구가 없으면 공격자가 이용할 수 있는 도구가 줄어들고, CVE 탐지 수도 극적으로 감소한다.
- **빌드 캐시 활용**: `COPY package*.json ./` + `RUN npm ci`를 소스 코드 복사보다 먼저 실행하면 `package.json`이 변경되지 않는 한 의존성 설치 레이어가 캐시에서 재사용된다.
- **파이프라인 단순화**: CI/CD에서 `docker build`와 `docker push`만 실행하면 되므로 빌드 도구 설치 단계를 파이프라인에서 관리할 필요가 없다.

## Q6: 정적 Agent와 동적 Agent의 차이는?

| 구분 | 정적 Agent | 동적 Agent |
|------|-----------|-----------|
| 생성 방식 | 미리 등록, 항상 온라인 유지 | 빌드 요청 시 생성, 완료 후 삭제 |
| 빌드 시작 | 즉시 시작 | Agent 생성 시간만큼 지연 |
| 리소스 효율 | 빌드 없는 시간에도 리소스 점유 | 필요 시에만 사용 |
| 환경 청결도 | 빌드 환경 누적 오염 가능(잔여 파일, 캐시 충돌) | 매 빌드마다 깨끗한 환경 보장 |
| 동적 구현 예시 | — | EC2 Plugin(인스턴스 생성), Kubernetes Plugin(Pod 생성) |

대규모 CI 환경에서는 동적 Agent에 Kubernetes를 사용하는 것이 표준이다. Agent Pool 크기 계획 시 피크 동시 빌드 수와 Agent 생성 시간 사이의 균형을 고려해야 한다.
