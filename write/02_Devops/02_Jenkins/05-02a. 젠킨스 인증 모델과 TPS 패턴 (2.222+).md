# 젠킨스 인증 모델과 TPS 패턴 (2.222+)
---
> 이 문서는 Jenkins 인증 방식의 변화와 TPS에서 사용한 패턴을 설명하는 이론 문서다.
>
> - 순수 API 스펙은 `05-02`에서 다룬다.
> - 이 문서는 "왜 이렇게 바뀌는지", "TPS 코드를 어떻게 단순화할 수 있는지"에 집중한다.

## 1. 이 문서의 관점

`05-02`가 현재 환경의 인증 API 사용법을 설명하는 문서라면, 이 문서는 다음 세 가지를 설명한다:

- Jenkins 2.222+에서 API Token 인증이 왜 단순해졌는지
- 비밀번호 인증과 토큰 인증의 설계 차이가 무엇인지
- TPS와 Playground 코드에서 어떤 패턴이 생기고, 어떤 코드를 줄일 수 있는지

즉 이 문서는 API 호출 순서보다 "인증 모델"과 "코드 구조"를 이해하기 위한 보조 문서다.



## 2. Jenkins 2.222+에서 바뀐 점

Jenkins 2.222부터 API Token 인증 요청은 CSRF crumb 검증 대상에서 제외된다.

Jenkins 공식 문구는 다음과 같다:

> Requests authenticating with an API token are exempt from CSRF protection in Jenkins.

핵심 의미는 다음과 같다:

- 브라우저 세션 기반 인증은 CSRF 공격 표면이 있다.
- API Token 인증은 세션 기반 브라우저 요청이 아니므로 Jenkins가 crumb 검증을 생략한다.
- 결과적으로 API Token 기반 scripted client는 인증 흐름이 더 단순해진다.

### 2.1 crumb API가 사라진 것은 아니다

중요한 점은 `/crumbIssuer/api/json` 자체가 없어지는 것이 아니라, API Token 인증 요청이 crumb 검증에서 면제된다는 것이다.

즉 다음 두 문장은 다 참이다:

- crumb API는 여전히 존재한다.
- API Token 인증 POST에서는 crumb이 불필요할 수 있다.

### 2.2 인증 방식별 차이

정리하면 다음과 같다

| 인증 방식 | POST에서 crumb 필요 여부 | cookie 필요 여부 | 특징 |
|-----------|--------------------------|------------------|------|
| ID/Password | 필요 | 필요할 수 있음 | 현재 05-02 검증 환경 |
| API Token | 보통 불필요 | 불필요 | 2.222+ 현대 방식 |

- 현재 당신의 운영 환경은 첫 번째다. 그래서 `05-02` 문서는 여전히 `ID/Password + crumb + cookie` 기준으로 유지하는 것이 맞다.



## 3. 이전 방식과 현대 방식 비교

### 3.1 이전 방식

비밀번호 인증 중심의 이전 방식은 보통 다음 흐름을 가진다:

1. `GET /api/json`으로 Basic Auth 확인
2. `GET /crumbIssuer/api/json`으로 crumb과 cookie 확보
3. 이후 POST에서 `Authorization + crumb + cookie` 전송

이 구조에서는 GET과 POST의 인증 재료가 다르다.

### 3.1.1 crumb과 cookie는 왜 필요한가

현재 비밀번호 인증 환경에서 Jenkins POST가 까다로운 이유는, Jenkins가 "누가 요청했는가"와 "이 POST가 브라우저 세션 문맥 안에서 정상적으로 만들어졌는가"를 분리해서 보기 때문이다.

각 재료의 의미는 다음과 같다:

- `Authorization: Basic <id:password>`
  - 사용자가 누구인지 확인하는 인증 정보다.
  - 즉 "이 요청을 보낸 주체"를 식별한다.
- `crumb`
  - CSRF 방어용 토큰이다.
  - 즉 "이 POST가 Jenkins가 발급한 정상 세션 문맥에서 나온 요청인가"를 추가로 본다.
- `cookie`
  - Jenkins가 방금 발급한 세션 식별자다.
  - 즉 crumb이 어떤 세션에서 발급됐는지 연결하는 역할을 한다.

정리하면 다음처럼 볼 수 있다:

| 재료 | 역할 | 없으면 어떤 문제가 생기는가 |
|------|------|------|
| Basic Auth | 사용자 인증 | `401 Unauthorized` 가능 |
| crumb | CSRF 검증 통과 | `403 No valid crumb` 가능 |
| cookie | crumb과 세션 연결 | crumb만 보내도 `403` 가능 |

- 즉 현재 환경의 POST는 단순히 "로그인만 하면 되는가"의 문제가 아니다. 
- Jenkins는 비밀번호 기반 POST에서 "인증"과 "CSRF 방어"를 둘 다 만족하길 요구한다.

### 3.1.2 왜 crumb과 cookie가 있어도 ID/Password가 필요한가

여기서 자주 헷갈리는 점은 다음이다:

- crumb이 있으니 인증도 끝난 것처럼 보인다.
- cookie가 있으니 세션도 있으니 POST가 될 것처럼 보인다.

하지만 Jenkins 관점에서는 다르다.

- crumb은 인증 정보가 아니다.
- cookie도 사용자의 최종 권한을 대체하는 인증 자격증명으로 보지 않는다.
- 두 값은 "이미 인증된 요청 문맥"을 보조하는 값일 뿐이다.

즉 Jenkins는 다음처럼 해석한다:

- `Authorization`
  - 이 사용자가 누구이며, 이 Job에 대해 POST 권한이 있는가
- `crumb + cookie`
  - 이 POST가 Jenkins가 허용한 세션 문맥 안에서 만들어진 정상 요청인가

그래서 crumb과 cookie가 있어도 Basic Auth가 빠지면 보통 다음 문제가 생긴다:

- 사용자를 식별할 수 없다.
- 권한 검사를 할 수 없다.
- 어떤 계정의 Job 실행/수정 요청인지 결정할 수 없다.

실무적으로는 세 값을 이렇게 나눠서 기억하면 된다:

- Basic Auth: "누가 요청했는가"
- cookie: "어떤 세션인가"
- crumb: "그 세션에서 만들어진 POST가 맞는가"

즉 crumb과 cookie는 인증을 대체하는 값이 아니라, 비밀번호 기반 POST에서 인증에 덧붙는 세션/CSRF 검증 재료다.

### 3.1.3 crumb과 cookie에도 유효시간이 있는가

있다. 다만 두 값의 만료 방식은 조금 다르다.

먼저 cookie는 일반적인 세션 만료 개념으로 보면 된다.

- Jenkins의 `JSESSIONID` cookie는 웹 세션 식별자다.
- 세션 타임아웃이 지나면 더 이상 유효하지 않을 수 있다.
- Jenkins 문서 기준으로 `--sessionTimeout`으로 분 단위 설정이 가능하고, 명시하지 않으면 웹애플리케이션 기본값, 그다음 기본 `60분`을 사용한다.

반면 crumb은 "몇 분짜리 토큰"이라기보다 "현재 세션 문맥에 묶인 CSRF 토큰"에 가깝다.

- Jenkins 공식 문서 기준으로 crumb은 사용자 이름, web session ID, IP address, instance salt 같은 값에 묶여 생성된다.
- 그래서 세션이 바뀌거나 만료되면 crumb도 함께 무효화될 수 있다.
- 프록시 환경에서 클라이언트 IP가 바뀌면 crumb 검증이 깨질 수 있다.
- `Enable proxy compatibility`를 켜면 IP 조건은 제외할 수 있다.

실무적으로는 다음처럼 이해하면 된다:

| 항목 | 만료/무효화 기준 | 실무 해석 |
|------|------|------|
| `cookie` | 세션 타임아웃 | 오래되면 새 세션 필요 |
| `crumb` | 세션 변경, 세션 만료, IP 변화 등 | 오래된 crumb 재사용은 실패 가능 |

즉 `crumb.json`과 `cookies.txt`는 장기 보관용 인증 재료가 아니다.

- 짧은 실습이나 한 번의 연속 호출에서는 재사용 가능하다.
- 세션이 끊겼거나 시간이 많이 지나면 다시 발급하는 편이 안전하다.
- `403 invalid crumb`, `403 No valid crumb`, `session expired` 류 오류가 나면 보통 둘을 함께 재발급하는 편이 빠르다.

### 3.2 현대 방식

API Token 중심의 현대 방식은 다음처럼 단순해진다:

1. `Authorization: Basic <username:apiToken>` 준비
2. GET과 POST 모두 동일한 인증 헤더 사용

즉 현대 방식의 핵심은 "POST라서 인증 로직이 하나 더 필요하지 않다"는 점이다.

### 3.3 코드에 주는 영향

이 차이는 코드 구조에 직접 영향을 준다:

| 구분 | 이전 방식 | 현대 방식 |
|------|----------|----------|
| GET 인증 | Basic Auth | Basic Auth |
| POST 인증 | Basic Auth + crumb + cookie | Basic Auth |
| 세션 관리 | 필요 | 불필요 |
| crumb 캐시 | 필요할 수 있음 | 불필요 |
| 헤더 분기 | GET/POST 분기 필요 | 분기 최소화 가능 |



## 4. TPS에서 사용한 패턴

이 섹션은 API 스펙이 아니라, TPS가 Jenkins 인증을 코드로 다루는 방식에 대한 설명이다.

### 4.1 JenkinsAuthVo 패턴

TPS에서 중요한 포인트는 Jenkins 인증 재료를 호출마다 흩뿌리지 않고 하나의 객체로 캡슐화하는 것이다.

예시 구조는 다음과 같다:

```java
public class JenkinsAuthVo {
    private String crumb;
    private String cookie;
    private String crumbRequestField;
    private String basicAuth;
    private URI jenkinsUrl;
}
```

이 구조가 의미하는 바는 단순하다:

- 현재 인증에 필요한 데이터를 한 번에 운반한다.
- 호출부는 헤더 조립 세부사항을 몰라도 된다.
- Jenkins 서버가 여러 개여도 같은 추상화로 다룰 수 있다.

#### 4.1.1 코드에서는 어떻게 처리하는가

`curl -c cookies.txt` 때문에 cookie가 파일 기반 인증 재료처럼 보일 수 있지만, 실제로는 그렇지 않다.

실제 의미는 다음과 같다:

- 응답 본문에서 `crumb` 값을 받는다.
- 응답 본문에서 `crumbRequestField` 값을 받는다.
- 응답 헤더의 `Set-Cookie`에서 `JSESSIONID=...`를 받는다.

즉 코드가 다뤄야 하는 최소 단위는 "crumb 하나"가 아니라 "POST 검증에 필요한 세션 문맥"이다.

예시 구조는 다음과 같다:

```java
public record JenkinsSession(
    URI baseUrl,
    String basicAuth,
    String crumbRequestField,
    String crumb,
    String cookie
) {}
```

실제 처리 흐름은 다음과 같이 잡는 편이 단순하다:

```java
ResponseEntity<CrumbResponse> response =
    jenkinsFeignClient.getCrumb(baseUrl, basicAuth);

String cookie = response.getHeaders()
    .getValuesAsList(HttpHeaders.SET_COOKIE)
    .stream()
    .map(value -> value.split(";", 2)[0])
    .collect(Collectors.joining("; "));

JenkinsSession session = new JenkinsSession(
    baseUrl,
    basicAuth,
    response.getBody().crumbRequestField(),
    response.getBody().crumb(),
    cookie
);
```

이후 POST에서는 다음 세 값만 다시 꺼내 쓰면 된다:

- `Authorization`
- `session.crumbRequestField(): session.crumb()`
- `Cookie: session.cookie()`

현대화 이후에는 이 객체가 더 얇아질 수 있다.

```java
public record JenkinsAuth(
    String basicAuth,
    URI jenkinsUrl
) {}
```

이 단순화는 "인증 방식이 API Token인지"에 따라 가능 여부가 갈린다.

### 4.2 Feign URI Override 패턴

TPS의 Jenkins 연동에서 중요한 또 하나의 패턴은 URI override다.

예시는 다음과 같다:

```java
@FeignClient(
    name = "jenkins",
    url = "https://",
    configuration = FeignJenkinsConfig.class
)
public interface JenkinsFeignClient {

    @GetMapping("/crumbIssuer/api/json")
    ResponseEntity<JenkinsAuthVo> getCrumb(
        URI baseUrl,
        @RequestHeader("Authorization") String basicAuth
    );
}
```

핵심은 첫 번째 파라미터로 `URI baseUrl`을 받아 실제 Jenkins URL을 런타임에 덮어쓰는 것이다.

이 패턴의 장점은 다음과 같다:

- 하나의 Feign Client로 여러 Jenkins 서버를 호출할 수 있다.
- DB에 저장된 Jenkins URL을 그대로 활용할 수 있다.
- 인증 정보와 대상 서버를 한 요청 문맥 안에서 같이 다룰 수 있다.

### 4.3 FeignJenkinsConfig 패턴

Feign 설정은 단순 HTTP 설정이 아니라 Jenkins 연동 규칙의 집합이다.

기존 문서에서 중요한 역할은 다음과 같았다:

- 요청/응답 로깅 통일
- 자체 서명 인증서 대응
- Jenkins 공통 헤더 정책 모음

현재 검증에서도 `curl -k`가 필요했다. 같은 이유로 코드에서도 개발 환경과 운영 환경의 SSL 처리 정책을 분리하는 것이 중요하다.



## 5. Playground와 TPS 코드에서 줄일 수 있는 것

> API Token 인증으로 통일되는 환경이라면 다음 요소를 제거할 수 있다.
>
> - `crumb` 필드
> - `cookie` 필드
> - `crumbRequestField` 필드
> - crumb 조회 메서드
> - crumb 캐시
> - POST 전용 헤더 조립 분기

예를 들어 기존 구조가 다음과 같다면:

```java
private HttpHeaders buildPostHeaders(long jenkinsInstanceId) {
    var headers = buildGetHeaders(jenkinsInstanceId);
    var crumb = getCrumb(jenkinsInstanceId);
    if (crumb != null) {
        headers.set(crumb.crumbRequestField, ...);
        headers.set("Cookie", crumb.cookie);
    }
    return headers;
}
```

현대화된 환경에서는 다음처럼 단순화할 수 있다:

```java
private HttpHeaders buildHeaders(long jenkinsInstanceId) {
    var headers = new HttpHeaders();
    headers.set("Authorization", "Basic " + getBasicAuth(jenkinsInstanceId));
    return headers;
}
```

- 이 차이가 바로 "현대화"의 실질적인 효과다. 인증 개념이 아니라 코드 복잡도가 줄어든다.



## 6. 하지만 crumb이 여전히 필요한 경우

이 문서를 읽을 때 가장 중요한 주의점은 다음이다.

API Token이 단순해졌다고 해서 모든 Jenkins 환경이 단순해진 것은 아니다.

다음 조건이면 crumb 흐름이 여전히 필요하다:

- 인증 수단이 ID/Password다.
- Jenkins가 `useCrumbs: true`다.
- 서버 또는 프록시 구성이 세션 cookie를 같이 요구한다.

### 6.1 crumb과 cookie를 같이 봐야 하는 이유

이 부분은 Jenkins를 "토큰 값 하나를 받는 API"로 보면 오해하기 쉽다.

현재 기본 Crumb Issuer는 crumb을 고정 문자열처럼 발급하는 것이 아니라, 사용자와 세션 문맥에 묶인 값으로 다룬다. 그래서 같은 사용자라도 세션이 바뀌면 이전 crumb이 그대로 통하지 않을 수 있다.

실무에서는 다음처럼 이해하는 편이 정확하다:

- crumb은 단독 비밀값이 아니다.
- cookie는 단순 부가 헤더가 아니다.
- 둘은 같은 세션 문맥의 일부다.

이번 실습에서 "crumb만 보내면 403, crumb+cookie를 같이 보내면 통과"한 이유도 이 관점으로 설명된다.

### 6.2 만료와 갱신 전략

이 질문에서 중요한 포인트는 "crumb 자체의 TTL"보다 "세션이 언제 끊기느냐"다.

Jenkins 공식 문서 기준으로 crumb은 웹 세션과 연결된다. 따라서 별도의 장기 캐시 키처럼 오래 들고 가기보다는, 세션이 살아 있는 동안만 유효한 값으로 보는 편이 맞다.

운영 코드는 다음 전략이 가장 현실적이다:

- 매 POST마다 crumb을 새로 받지 않는다.
- `JenkinsSession`을 메모리에 캐시한다.
- POST에서 `403 No valid crumb was included in the request`가 나오면 crumb과 cookie를 다시 발급받고 1회 재시도한다.
- Jenkins 세션 타임아웃을 알고 있다면 그보다 조금 이르게 선제 갱신할 수 있다.

반대로 다음 방식은 비효율적일 수 있다:

- 매 요청마다 `GET /crumbIssuer/api/json`을 먼저 호출하는 방식
- crumb과 cookie를 장기간 파일에 저장해 재사용하는 방식

즉 현재 환경에서는 "세션 캐시 + 실패 시 재발급" 패턴이 기본 전략이고, "항상 새로 발급"은 단순하지만 불필요하게 왕복이 많다.

즉 현재 검증 환경은 현대 Jenkins 버전이더라도, 인증 수단이 비밀번호이기 때문에 `05-02` 방식이 그대로 맞다.



## 7. 버전별로 기억할 것

| Jenkins 버전 | 의미 | 인증 영향 |
|-------------|------|----------|
| 2.222 | API Token 인증 시 crumb 면제 도입 | 토큰 기반 scripted client 단순화 |
| 2.462.3 | 현재 검증 환경 버전 | 비밀번호 인증이면 crumb 흐름 유지 가능 |
| 2.504 | 최근 LTS 계열 | crumb 면제 원칙 유지 |

버전 자체보다 더 중요한 것은 다음 두 가지다:

- 지금 무엇으로 인증하는가
- 현재 Jenkins가 어떤 보안 정책으로 동작하는가



## 8. 언제 어떤 문서를 봐야 하는가

문서 구분은 다음처럼 보면 된다:

- `05-02`: 현재 환경에서 인증 API를 어떻게 호출하는가
- `05-02a`: 인증 모델이 왜 달라지는가, TPS 코드에서 무엇이 패턴인가
- `05-02b`: API Token을 실제로 어떻게 발급하고 회전할 것인가
- `05-04`: 인증이 끝난 뒤 빌드 실행 POST를 어떻게 보내는가
- `05-05`: 빌드 이후 상태를 어떻게 추적하는가



## 9. 핵심 정리

이 문서의 핵심은 세 줄로 요약된다:

1. Jenkins 2.222+에서는 API Token 인증이 crumb 검증에서 면제될 수 있다.
2. 그래서 현대 Jenkins에서는 GET/POST 인증 규칙을 통합할 수 있다.
3. 하지만 현재 운영 환경처럼 ID/Password를 쓰면 `05-02`의 crumb + cookie 흐름이 여전히 맞다.
