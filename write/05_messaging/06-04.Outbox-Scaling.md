# Outbox 스케일링

---

> Polling Relay의 처리량을 1,350 MPS에서 32,500 MPS로 끌어올리는 단계별 최적화 과정을 벤치마크와 함께 정리한다. 기본 구현과 프로덕션 고려사항은 [Outbox와 Inbox 패턴](06-3.Outbox.md)을 참고한다.



## 1. 베이스라인 측정

> 최적화의 출발점은 현재 병목이 어디인지 측정하는 것이다. Polling Relay의 한 사이클은 세 단계로 구성된다.
>
> 1. PENDING 레코드 조회(Query)
> 2. 브로커 발행(Publish)
> 3. 상태 업데이트(Update)

베이스라인에서 각 단계의 소요 시간은 다음과 같다:

- Query: 70ms (인덱스 없는 전체 테이블 스캔)
- Publish: 320ms (순차 전송, Publisher Confirm 포함)
- Update: 300ms (건별 UPDATE)

배치 크기 100건 기준으로 한 사이클에 약 690ms가 소요되며, 초당 처리량은 약 1,350 MPS다. 일일 기준으로 환산하면 약 1억 1,600만 건이다. 여기서 각 단계를 독립적으로 최적화하면 처리량이 누적으로 증가한다.



## 2. 쿼리 최적화

> 가장 극적인 개선이 나오는 구간이다. PENDING 레코드를 조회할 때 테이블 전체를 스캔하면 outbox 테이블이 커질수록 쿼리 시간이 선형으로 증가한다. 두 가지 인덱스 기법으로 70ms를 1ms 이하로 줄일 수 있다.

### 2-1. Partial Index

 outbox 테이블의 대부분은 이미 처리된 SENT 레코드이고, 폴링이 관심 있는 것은 PENDING 레코드뿐이다. `WHERE status = 'PENDING'` 조건의 partial index를 생성하면 인덱스 크기가 PENDING 레코드 수에 비례하여 작아진다:

```sql
CREATE INDEX idx_outbox_pending_created
    ON outbox_event (created_at)
    WHERE status = 'PENDING';
```

### 2-2 **Covered Index**

PostgreSQL의 `INCLUDE` 절로 SELECT 대상 컬럼을 인덱스에 포함시키면, 테이블 heap 접근 없이 인덱스만으로. 쿼리를 처리할 수 있다:

```sql
CREATE INDEX idx_outbox_pending_covered
    ON outbox_event (created_at)
    INCLUDE (id, aggregate_type, aggregate_id, event_type, payload, topic)
    WHERE status = 'PENDING';
```

- 주의할 점은 PostgreSQL의 인덱스 행 크기 제한이 2712바이트라는 것이다. 
- `payload` 컬럼에 큰 JSON을 담고 있다면 이 제한에 걸릴 수 있다. 메시지 크기가 클 경우 `payload`를 INCLUDE에서 제외하고, 쿼리에서 ID만 가져온 뒤 별도 조회로 payload를 읽는 2단계 접근이 필요하다.

### 2-3. MariaDB/MySQL에서의 대안

Partial Index(`WHERE` 조건부 인덱스)와 Covered Index의 `INCLUDE` 절은 PostgreSQL 전용 기능이다. MariaDB에서는 두 가지를 조합한 대안을 사용한다.

Partial Index 대신, 조회 쿼리의 `WHERE status = 'PENDING'` 조건에 맞는 **복합 인덱스**를 생성한다. `status`를 선두 컬럼에 두면 PENDING 레코드만 인덱스 범위 스캔으로 빠르게 찾을 수 있다.

```sql
CREATE INDEX idx_outbox_status_created
    ON outbox_event (status, created_at);
```

- PostgreSQL의 Partial Index와 달리 SENT 레코드도 인덱스에 포함되므로 인덱스 크기가 더 크지만, `status = 'PENDING'` 조건으로 범위를 좁히기 때문에 실제 스캔 범위는 비슷하다. 
- Covered Index의 `INCLUDE` 절도 MariaDB에서 지원하지 않으므로, 필요한 컬럼을 모두 복합 인덱스에 포함시키는 방식으로 대체한다. 다만 `payload`처럼 큰 컬럼은 인덱스에 넣기 어려우므로, ID만 인덱스에서 가져온 뒤 본 테이블에서 payload를 조회하는 2단계 접근이 현실적이다.

- 이 최적화만으로 쿼리 시간이 70ms → 1ms 이하로 줄어들고, 처리량이 약 2,700 MPS로 두 배 가까이 증가한다.



## 3. 역직렬화 최적화

>  Polling Relay가 outbox 레코드를 읽어서 브로커에 발행할 때, `event_type` 문자열을 실제 클래스로 변환하는 역직렬화 과정이 매 메시지마다 반복된다. Java/Spring 환경에서 `Class.forName()` 호출이나 `ObjectMapper`의 타입 해석은 리플렉션 비용이 발생한다.

**타입 캐싱**은 이 비용을 제거하는 기법이다. `event_type` 문자열과 대응하는 클래스 매핑을 `ConcurrentHashMap`에 캐싱하면 두 번째 조회부터는 리플렉션 없이 즉시 반환된다:

```java
private final Map<String, Class<?>> typeCache = new ConcurrentHashMap<>();

private Class<?> resolveType(String eventType) {
    return typeCache.computeIfAbsent(eventType, key -> {
        try {
            return Class.forName(key);
        } catch (ClassNotFoundException e) {
            throw new IllegalArgumentException("Unknown event type: " + key);
        }
    });
}
```

- `ObjectMapper` 인스턴스도 스레드 안전하므로 매번 생성하지 않고 싱글턴으로 재사용해야 한다. 
- Spring Boot의 자동 구성된 `ObjectMapper` 빈을 주입받으면 이 조건은 자연스럽게 충족된다. 역직렬화 최적화 단독으로는 벤치마크 수치에 극적인 변화를 주지 않지만, 배치 크기가 클수록 누적 효과가 커진다.



## 4. 발행 최적화

>  베이스라인에서 Publish 단계가 320ms로 전체 사이클의 절반 가까이를 차지한다. 이 병목은 두 가지 원인에서 비롯된다
>
> 1. 메시지를 순차적으로 전송하는 것
> 2. 메시지마다 브로커의 확인(Publisher Confirm)을 동기적으로 대기하는 것

### 4-1. 병렬 발행

순차 전송을 비동기 병렬 전송으로 바꾸면 발행 지연이 크게 줄어든다. Kafka의 `KafkaTemplate.send()`는 기본적으로 `CompletableFuture`를 반환하므로, 배치 내 모든 메시지를 먼저 전송하고 마지막에 결과를 일괄 대기하는 패턴이 가능하다:

```java
List<CompletableFuture<SendResult<String, byte[]>>> futures = new ArrayList<>();
for (var event : events) {
    futures.add(kafkaTemplate.send(buildRecord(event)));
}
// 모든 발행 완료 대기
CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
    .get(30, TimeUnit.SECONDS);
```

- Kafka 프로듀서 설정에서 `batch.size`(기본 16KB)와 `linger.ms`(기본 0)를 함께 조정하면, 프로듀서가 메시지를 모아서 한 번의 네트워크 요청으로 전송한다. 
- `linger.ms=5`로 설정하면 5ms 동안 메시지를 모아서 배치 전송하므로 네트워크 라운드트립이 줄어든다.

### 4-2. Publisher Confirm 비활성화

RabbitMQ 환경에서는 Publisher Confirm을 비활성화하면 발행 속도가 추가로 상승한다. Confirm이 없으면 브로커가 메시지를 디스크에 쓰기 전에 프로듀서가 다음 메시지를 전송할 수 있기 때문이다. Milan Jovanović의 벤치마크에서는 이 설정만으로 32,500 MPS → 37,000 MPS까지 올라갔다.

그러나 이 방식은 **권장하지 않는다**. 브로커 장애 시 메시지가 유실되어도 프로듀서가 알 수 없다. Outbox 패턴의 존재 이유가 "메시지 유실 방지"인데, 발행 확인을 끄는 것은 그 전제를 스스로 무너뜨리는 선택이다. Confirm을 끈 상태에서 유실을 보상하려면 Consumer 측 재요청 메커니즘이 필수적이며, 이는 시스템 복잡도를 다른 곳으로 이전할 뿐이다.



## 5. 업데이트 최적화

>  베이스라인에서 100건 배치의 상태 업데이트에 300ms가 걸리는 이유는 건별 UPDATE 때문이다. 
>
> - 이벤트마다 `UPDATE outbox_event SET status = 'SENT' WHERE id = ?`를 실행하면 100회의 DB 라운드트립이 발생한다.

성공한 이벤트 ID를 리스트로 모아 `WHERE id IN (...)` 한 번으로 처리하면 라운드트립이 1회로 줄어든다:

```sql
UPDATE outbox_event
SET status = 'SENT', published_at = NOW()
WHERE id IN (?, ?, ?, ...);
```

Java에서는 MyBatis의 `<foreach>` 태그나 JPA의 `@Modifying` 쿼리로 구현한다:

```java
@Modifying
@Query("UPDATE OutboxEvent e SET e.status = 'SENT', e.publishedAt = :now "
     + "WHERE e.id IN :ids")
int markAsSentBatch(@Param("ids") List<UUID> ids, @Param("now") Instant now);
```

- 이 최적화로 300ms → 52ms를 달성했다. 단순한 변경이지만 효과가 크므로 가장 먼저 적용할 만한 최적화다.



## 6. 다중 워커 병렬화

>  단일 프로세서의 한계에 도달하면 여러 워커가 동시에 outbox 테이블을 폴링하는 방식으로 수평 확장한다. 
>
> - 이때 `SELECT ... FOR UPDATE SKIP LOCKED`가 워커 간 row 분리를 보장한다:

```sql
SELECT id, aggregate_type, aggregate_id, event_type, payload, topic
FROM outbox_event
WHERE status = 'PENDING'
ORDER BY created_at
LIMIT :batchSize
FOR UPDATE SKIP LOCKED;
```

- 워커 A가 row 1~100을 `FOR UPDATE`로 잠그면, 워커 B는 `SKIP LOCKED` 덕분에 해당 row를 건너뛰고 row 101~200을 가져간다. 처리량은 워커 수에 거의 비례하여 증가하며, 5개 워커로 32,500 MPS(일일 28억 건)를 달성한 것이 Milan Jovanović의 벤치마크 결과다.

- `FOR UPDATE SKIP LOCKED`는 PostgreSQL 9.5+, MySQL 8.0+, MariaDB 10.6+에서 지원한다. MariaDB 10.5 이하를 사용한다면 `SKIP LOCKED`가 없으므로, 일반 `FOR UPDATE`를 사용하되 워커 간 lock 대기가 발생할 수 있다. 
- 이 경우 `NOWAIT` 옵션(MariaDB 10.3+)으로 lock 획득 실패 시 즉시 에러를 반환받고, 애플리케이션에서 다음 폴링까지 대기하는 방식으로 우회할 수 있다.

### **순서 보장 trade-off**

`SKIP LOCKED`를 사용하면 같은 aggregate의 이벤트가 서로 다른 워커에 분배될 수 있다. 워커 A가 `order-123`의 `CREATED` 이벤트를, 워커 B가 같은 주문의 `PAID` 이벤트를 동시에 처리하면 브로커 도착 순서가 뒤바뀔 수 있다. 순서가 중요한 도메인에서는 세 가지 대안이 있다:

- 단일 프로세서를 유지하고, 배치 크기와 병렬 발행으로 처리량을 확보한다.
- aggregate_id 기반 파티셔닝으로 같은 aggregate의 이벤트가 같은 워커에 할당되도록 한다.
- Consumer 측에서 Inbox 패턴으로 시퀀스 번호를 검사하여 순서를 재정렬한다.



## 7. 벤치마크 요약

각 최적화 단계의 누적 효과를 정리하면 다음과 같다:

| 단계 | 최적화 | MPS | 일일 처리량 | 비고 |
|------|--------|-----|------------|------|
| 0 | 베이스라인 | 1,350 | 116M | Query 70ms, Publish 320ms, Update 300ms |
| 1 | Covered Index | 2,700 | 233M | Query 70ms → 1ms |
| 2 | 배치 UPDATE | ~3,500 | 302M | Update 300ms → 52ms |
| 3 | 병렬 발행 | ~4,000 | 345M | Publish 순차 → 비동기 병렬 |
| 4 | 5 병렬 워커 | 32,500 | 2.8B | `SKIP LOCKED` 기반 수평 확장 |
| 5 | Publisher Confirm 끔 | 37,000 | 3.2B | 비권장 — 유실 위험 |

단계 1~3은 단일 프로세서 내에서 적용 가능한 최적화이고, 단계 4에서 다중 워커로 전환하면서 처리량이 비약적으로 상승한다. 단계 5는 신뢰성을 희생하는 선택이므로 특수한 요구사항이 아니면 적용하지 않는다.

주목할 점은 가장 큰 도약이 인덱스 최적화(단계 1)와 다중 워커(단계 4)에서 발생한다는 것이다. 인덱스는 비용 없이 적용 가능하므로 가장 먼저 해야 하고, 다중 워커는 순서 보장 trade-off를 수반하므로 단일 프로세서의 한계가 확인된 후에 도입하는 것이 적절하다.



## 8. Consumer 수용량

생산자 측 처리량만 높이면 Consumer가 소화하지 못하는 **백로그**가 누적된다. 

- 32,500 MPS로 발행하는데 Consumer가 5,000 MPS만 처리할 수 있다면, 매초 27,500건의 메시지가 브로커에 쌓인다. 
- 하루면 약 24억 건이 밀리는 셈이다.

Consumer 스케일링은 Kafka의 파티션 모델에 종속된다. 

- 같은 Consumer Group 내에서 파티션은 하나의 Consumer에만 할당되므로, Consumer 인스턴스를 늘리려면 파티션 수도 함께 증가시켜야 한다. 
- 파티션 수를 늘리면 리밸런싱 시간이 길어지는 trade-off가 있으므로, 처음부터 예상 처리량에 맞는 파티션 수를 설계하는 것이 중요하다.

다중 워커로 순서가 깨진 메시지를 Consumer가 올바르게 처리하려면 **Inbox 패턴**이 필요하다. 

- Consumer가 수신한 메시지를 Inbox 테이블에 저장하고, 시퀀스 번호를 기준으로 정렬한 뒤 순서대로 처리하는 방식이다. 
- Inbox 테이블은 동시에 멱등성 키 역할도 하므로 중복 처리를 자연스럽게 방지한다. Outbox와 Inbox의 관계와 멱등성 처리 상세는 [Outbox와 Inbox 패턴](06-3.Outbox.md)의 Consumer 멱등성 섹션을 참고한다.
