# Spring Kafka 운영 고급 — ING 사례에서 배우는 패턴

---

> 이 문서는 같은 [`01-04`](01-04.Spring%20Kafka%20DLT와%20Producer%20Config.md)가 다루지 않는 두 축, 즉 **컨슈머 스케일링 결정**과 **운영 관측성**을 정리한다. `01-04`가 "DLT/retry는 어떻게 동작하는가"를 본다면, 여기서는 "운영 환경에서 어느 옵션을 왜 선택하는가"를 본다.
>
> 1차 자료는 Spring I/O 발표 *Spring for Apache Kafka — the advanced features* (Tim van Baarsen, Kosta — ING). 발표자들의 운영 컨텍스트는 10년간 Kafka 운영, 초당 2M+ 메시지, 8,000+ 토픽, 1,000+ 개발팀이 공유하는 클러스터다. 이 규모에서 작은 디폴트 선택이 어떻게 운영 비용으로 누적되는지를 살펴보고, 우리 규모에서 어디까지 미리 잡아둘 수 있는지 정리한다.



## 1. 왜 "고급"이 필요한가 — 디폴트가 깨지는 지점

Spring Kafka는 디폴트가 잘 잡혀 있어 단일 컨슈머 + 단일 파티션 정도까지는 `@KafkaListener` 한 줄로 끝난다. 문제는 **세 가지 축이 동시에 성장할 때**다.

| 성장 축 | 디폴트가 깨지는 시점 | 본 문서의 대응 절 |
| --- | --- | --- |
| 처리량 | 컨슈머 인스턴스 수를 늘려도 파티션 수에 묶여 더 못 늘어남, 또는 인스턴스를 더 띄우는 비용이 부담스러움 | §2 Vertical Scaling |
| 운영 동적성 | 토픽이 매주 추가되거나, 일부 컨슈머만 잠깐 멈췄다 다시 켜야 하는 운영 상황이 생김 | §3 Runtime Configurable Consumer |
| 장애 격리 | 한 메시지의 처리 실패가 같은 파티션의 뒤쪽 메시지까지 막아버림 | §4 Blocking vs Non-Blocking Retry |
| 추적 | 분산된 컨슈머 사이에서 한 비즈니스 트랜잭션이 어떻게 흘러갔는지 보이지 않음 | §5 Observability |

ING 사례에서 위 네 가지가 어떻게 운영 패턴으로 굳어졌는지가 발표의 본질이다. 우리 규모(TPS 기준 수십~수백 토픽)에서는 모든 패턴을 처음부터 도입할 필요는 없지만, **언제 도입을 고려해야 하는지**의 시그널은 미리 알아두면 좋다.



## 2. Vertical Scaling — `concurrency` 프로퍼티

### 2-1. 두 가지 스케일 축

Kafka 컨슈머를 늘리는 방법은 본질적으로 둘이다.

- **Horizontal scaling**: 같은 컨슈머 그룹에 컨슈머 인스턴스(프로세스/Pod)를 더 추가한다. 그룹 코디네이터가 파티션을 재할당해 새 인스턴스에 일을 나눠준다.
- **Vertical scaling**: 한 컨슈머 인스턴스 안에서 **여러 스레드**가 각자 다른 파티션을 받아 동시에 처리한다.

Spring Kafka에서 vertical scaling은 `concurrency` 프로퍼티로 표현된다. 컨테이너가 내부에 N개의 스레드를 띄우고 그룹 멤버십도 N개 분량으로 잡는다(컨테이너 1개 = 그룹 멤버 N개).

### 2-2. 적용하는 두 위치

팩토리 레벨에 디폴트를 박아두는 방법:

```java
@Bean
ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(...) {
    var factory = new ConcurrentKafkaListenerContainerFactory<String, String>();
    factory.setConcurrency(3);
    // ... consumer factory 등
    return factory;
}
```

리스너 단위로 오버라이드하는 방법:

```java
@KafkaListener(
        topics = "high-throughput-topic"
        , groupId = "my-group"
        , concurrency = "10"
)
public void listen(ConsumerRecord<String, String> record) { ... }
```

운영에서 자주 쓰는 패턴은 SpEL/property placeholder로 환경마다 다르게 두는 것이다. 같은 코드가 dev에서는 1, prod에서는 10으로 뜨게 하려면 다음과 같다.

```java
@KafkaListener(
        topics = "${kafka.topic.name}"
        , groupId = "#{someBean.groupId}"
        , autoStartup = "${listen.auto.start:true}"
        , concurrency = "${listen.concurrency:3}"
)
public void dynamicListener(String data) { ... }
```

### 2-3. 결정 기준 — Vertical을 언제 고르나

Horizontal과 Vertical은 결과적으로는 둘 다 컨슈머 수를 늘리는 행위지만, 운영에서의 무게가 다르다.

| 비교 항목 | Horizontal (인스턴스 추가) | Vertical (`concurrency` 증가) |
| --- | --- | --- |
| 배포/관리 단위 | Pod·컨테이너 1개 추가 | 단일 Pod 안의 스레드 수만 변경 |
| 비용 | 컨테이너 오버헤드(JVM 메모리, 사이드카, 네트워크) 곱하기 | 같은 JVM의 스레드만 추가 — 메모리 효율 |
| 장애 격리 | 인스턴스가 죽어도 다른 인스턴스가 살아 있음 | 한 JVM이 죽으면 N개 스레드가 함께 죽음 |
| 파티션 한계 | 그룹 전체 컨슈머 수가 파티션 수를 넘으면 idle 컨슈머가 생김 | 같은 한계 — 인스턴스 수 × concurrency가 파티션 수를 넘으면 idle 스레드 |
| 변경 시점 | 배포 또는 오토스케일러 트리거 | 같은 배포 사이클 안 (또는 §3의 런타임 제어) |

> **파티션이 ordering·소비 병렬성의 단단한 상한이라는 점**은 이 둘의 공통 제약이다(상위 [`04-03.Consumer Group`](../04-03.Consumer%20Group.md) §2). Vertical을 5로 잡든 horizontal로 5 인스턴스를 띄우든, 파티션이 4개면 한 자리는 비게 된다.

따라서 vertical은 **"컨테이너 비용이 더 중요한 환경"**에서 horizontal보다 우선 검토된다. ING처럼 클러스터 안에서 토픽·팀이 같이 빠르게 늘어나는 환경은 인스턴스를 더 띄우는 것보다 같은 인스턴스 안에서 스레드를 늘리는 게 운영상 단순하다. 우리 규모에서는 horizontal이 더 직관적이지만, 한 서비스가 여러 토픽을 묶어 들어야 할 때(예: TPS의 결과 수신 컨슈머 묶음)는 vertical 쪽이 자연스럽다.



## 3. Runtime Configurable Consumer

### 3-1. 빈 등록만으로는 부족한 순간

`@KafkaListener`를 빈으로 등록하면 컨테이너가 자동 시작·정지된다. 정지하려면 `autoStartup = "false"`를 박고 시작·중지를 외부에서 제어해야 하는 시나리오가 운영에서는 의외로 자주 생긴다.

- **부분 정비**: 한 토픽의 컨슈머만 잠깐 멈추고 백로그를 정리한 뒤 재개해야 할 때
- **카나리 배포**: 새 컨슈머 인스턴스에 일부 트래픽만 흘리고 싶을 때 (혹은 반대로 새 코드 검증이 끝날 때까지 컨슈머 그룹 가입을 미루고 싶을 때)
- **Self-service 토픽 관리**: 토픽 생성·구독 매핑을 코드 외부 설정에서 받아 런타임에 컨슈머를 동적으로 켜고 끄는 패턴

### 3-2. `KafkaListenerEndpointRegistry` 직접 제어

Spring Kafka는 등록된 모든 `@KafkaListener` 컨테이너를 `KafkaListenerEndpointRegistry`에 모아둔다. 컨테이너에 `id` 속성을 부여하면 그 ID로 직접 가져올 수 있다.

```java
@KafkaListener(id = "ordersListener", topics = "orders", autoStartup = "false")
public void listen(ConsumerRecord<String, String> record) { ... }
```

```java
@Autowired
private KafkaListenerEndpointRegistry registry;

public void resumeOrdersConsumer() {
    registry.getListenerContainer("ordersListener").start();
}

public void pauseOrdersConsumer() {
    registry.getListenerContainer("ordersListener").stop();
}
```

`isRunning()`으로 상태 확인이 가능하고, `MessageListenerContainer` 인터페이스의 다른 메서드(파티션 일시정지 등)도 같은 경로로 사용한다.

### 3-3. Self-service 패턴이 의미하는 것

ING의 사례에서 이 기능이 두드러지는 이유는 **1,000+ 팀이 공유하는 클러스터**에서 토픽이 매주 새로 등록·폐기되기 때문이다. 새 토픽이 등록되면 그 토픽을 구독해야 하는 컨슈머 컨테이너를 런타임에 만들어 띄우고, 폐기되면 정지·해제한다. 코드 재배포 없이.

우리 규모에서 굳이 같은 수준까지 일반화할 필요는 없지만, **"운영 중 컨슈머를 멈췄다 켤 일이 한 번이라도 있다면"** 이 인터페이스를 알고 있는 것 자체가 의사결정의 폭을 넓혀 준다. 다음 두 가지가 빠르게 가능해진다.

1. 운영 화면에서 토픽별 컨슈머 on/off 토글 노출
2. 배포 시 새 인스턴스의 컨슈머 그룹 가입을 헬스체크 통과 후로 미루기 (`autoStartup=false` + 헬스체크 핸들러에서 start)



## 4. Blocking vs Non-Blocking Retry — 결정 기준

> 이 절은 두 옵션의 **선택 기준**만 다룬다. 메커니즘 자체와 코드 비교는 [`01-04` §패턴 비교](01-04.Spring%20Kafka%20DLT와%20Producer%20Config.md)에 정리되어 있다.

### 4-1. 두 모드의 본질적 차이

| 모드 | 처리 위치 | 같은 파티션의 다음 메시지는? | 메시지 순서 |
| --- | --- | --- | --- |
| Blocking retry (`DefaultErrorHandler` + `BackOff`) | 같은 컨슈머 스레드, 같은 파티션 헤드에서 재시도 | 재시도 동안 차단 (헤드가 비지 않음) | 보존 |
| Non-blocking retry (`@RetryableTopic`) | 별도 retry 토픽으로 메시지를 옮긴 뒤, retry 토픽의 컨슈머가 시간 지나 다시 시도 | 즉시 진행 (헤드가 비워짐) | 깨짐 (메시지가 재정렬됨) |

코드 형태는 다음과 같다.

```java
@RetryableTopic(
        attempts = "4"
        , backoff = @BackOff(delay = 1000, multiplier = 2.0, maxDelay = 10000)
        , kafkaTemplate = "kafkaTemplate"
)
@KafkaListener(topics = "orders", groupId = "ordersGroup")
public void processOrder(OrderEvent event) {
    orderService.handle(event);
}

@DltHandler
public void handleDlt(OrderEvent event, @Header(KafkaHeaders.RECEIVED_TOPIC) String topic) {
    log.error("DLT 도착 — topic={} payload={}", topic, event);
}
```

자동 생성되는 토픽의 기본 suffix는 `-retry`(여러 단계인 경우 `-retry-0`, `-retry-1`처럼)와 `-dlt`다. 필요하면 `retryTopicSuffix` / `dltTopicSuffix`로 바꾸거나, `RetryTopicConfigurationBuilder`로 코드에서 빌드할 수 있다.

### 4-2. 어느 쪽을 고르나

선택 기준은 **"한 메시지의 재시도가 같은 파티션의 다른 메시지를 막아도 되는가"** 한 줄로 압축된다.

| 상황 | 추천 | 이유 |
| --- | --- | --- |
| 같은 키의 메시지 순서가 비즈니스적으로 중요 (예: 같은 주문의 상태 변경) | Blocking | 순서가 깨지면 비즈니스 로직이 깨짐. 재시도 동안 헤드가 막히는 건 의도된 동작 |
| 메시지가 서로 독립적이고, 한 건의 실패가 전체 처리량을 막으면 안 됨 (예: 알림 이벤트, 메트릭) | Non-blocking | 헤드를 비워 처리량 유지 |
| 외부 API 호출이 일시적으로 실패할 가능성이 높고, 짧은 backoff로 대부분 복구되는 시나리오 | Non-blocking + exponential backoff | 같은 파티션을 막지 않으면서 단계적으로 retry 토픽이 자연스럽게 분산 |
| 정합성·순서가 모두 중요하지만 "재시도 시간이 매우 짧을 것"이 보장됨 | Blocking | 짧은 재시도라면 헤드 차단 비용이 작음. 운영 단순성 우선 |

ING 발표에서 Non-blocking을 선호한 사용자들은 "as long as it's retrying, partition keeps moving"이 핵심 가치라고 정리했다. 큰 클러스터일수록 한 컨슈머 그룹의 lag이 운영 알람을 흔드는 비용이 커서, 재시도가 다른 메시지를 막지 않는 모드가 운영 친화적이다.

### 4-3. 두 모드를 섞을 때 주의

토픽별로 모드가 다를 수 있다. 같은 서비스에서 어떤 토픽은 Blocking, 어떤 토픽은 Non-blocking을 쓰는 게 자연스럽다. 다만 한 리스너 위에 두 모드를 같이 얹지는 못한다(`@RetryableTopic`이 컨테이너 에러 핸들러를 자체 정책으로 대체).

`@DltHandler`로 DLT를 같은 클래스 안에서 처리할 수 있는데, 이 경우 별도 DLQ 컨슈머 클래스([`01-05.DlqConsumer`](01-05.DlqConsumer.md))가 필요 없어진다. 반대로 DLT를 외부로 보내 알림·티켓팅을 연동해야 하면 별도 컨슈머가 더 깨끗하다.



## 5. Observability — Trace 전파

### 5-1. 분산 메시징의 추적 문제

REST 호출은 한 요청의 trace id가 헤더로 따라가니 추적이 비교적 쉽다. Kafka는 producer와 consumer가 시간상으로도 분리되어 있어, 같은 비즈니스 트랜잭션이 어떤 컨슈머에서 어디까지 갔는지 추적하려면 trace 컨텍스트를 메시지 자체에 실어야 한다.

W3C Trace Context 표준은 `traceparent`라는 헤더 한 줄에 trace id·span id·flags를 넣도록 규정한다(상세는 [`01-06.trace-id와 traceparent`](01-06.trace-id와%20traceparent.md)).

### 5-2. Spring Boot 3 + Micrometer Observation 표준 통합

Spring Boot 3 이후 Micrometer Observation API가 표준 추적 진입점이 되었고, Spring Kafka는 producer/consumer 양쪽에 자동으로 Observation을 끼워 넣는다. 활성화는 다음 세 가지가 갖춰지면 거의 자동이다.

- `spring-boot-starter-actuator` 의존성
- Micrometer Tracing 구현(예: `micrometer-tracing-bridge-otel`) + 익스포터(OTLP, Jaeger, Zipkin 등)
- `application.yml`에 sampling 비율 설정 (`management.tracing.sampling.probability`)

위가 갖춰지면 producer가 메시지를 보낼 때 현재 trace 컨텍스트를 자동으로 `traceparent` Kafka 헤더에 직렬화한다. 컨슈머는 받은 메시지의 헤더에서 컨텍스트를 복원해 같은 trace에 이어 붙인다.

### 5-3. Non-blocking retry에서도 trace가 이어지는가

`@RetryableTopic`이 메시지를 retry 토픽으로 옮길 때도 헤더는 그대로 복사된다. 따라서 원본 → retry-0 → retry-1 → DLT까지 같은 trace로 한 줄로 이어진 흐름이 분산 추적 시스템에 보인다. 이건 운영 환경에서 "왜 이 메시지가 DLT까지 갔나"를 추적할 때 결정적이다.

### 5-4. 메트릭·로그·트레이스 세 축 정리

| 축 | Spring Kafka에서의 진입점 | 보는 화면 |
| --- | --- | --- |
| 메트릭 | Micrometer가 자동 수집하는 producer/consumer client metrics + Spring Kafka container metrics | Grafana 등 |
| 로그 | MDC에 trace id·span id가 자동 주입되어 로그에 함께 기록 | ELK·Loki |
| 트레이스 | Observation으로 발행된 span (kafka.send / kafka.receive 류) | Jaeger·Zipkin·OTLP |

세 축이 같은 trace id로 묶여 있어야 비로소 "한 메시지의 일생"을 한 화면에서 따라갈 수 있다. 가장 흔한 운영 함정은 "세 축 중 하나만 도입하고 끝내는" 경우로, 메트릭만 있으면 lag만 보일 뿐 어떤 메시지가 막혔는지 모르고, 트레이스만 있으면 전체 처리량이 안 보인다.



## 6. TPS 적용 가능성 (현재 시점)

| 항목 | 현재 TPS message-lib | 도입 시그널 |
| --- | --- | --- |
| Vertical scaling (`concurrency`) | 개별 컨슈머 설정에서 사용 가능 (Spring Kafka 디폴트). 명시적으로 운영 기준은 아직 없음 | 한 인스턴스가 다중 토픽을 들어야 할 때, 또는 컨테이너 비용이 부담스러울 때 |
| Runtime configurable consumer | 현재는 모두 `autoStartup` 기본값 | 운영 화면에서 컨슈머를 on/off해야 하는 요구 발생 시 |
| Non-blocking retry (`@RetryableTopic`) | `01-04` 정리 기준으로 현재는 `DefaultErrorHandler + DLT` 사용 (Blocking + DLT) | 한 메시지 실패가 같은 파티션의 다른 메시지를 막는 게 비즈니스적으로 곤란해지는 시점 |
| Observability (Micrometer + trace) | trace id·traceparent 인프라는 [`01-06`](01-06.trace-id와%20traceparent.md)에 정리. 메트릭·트레이스 풀스택 통합은 도입 단계 | 분산 추적 시스템(예: OTel collector + Jaeger)과의 정식 연동이 결정될 때 |

지금 모두 도입할 필요는 없다. 다만 **`@RetryableTopic` 도입 여부**는 운영에서 한 번쯤 명시적으로 결정해 둘 가치가 있는 주제다. 같은 파티션의 다른 메시지를 막아도 되느냐는 질문은 토픽별로 답이 다를 수 있고, "디폴트는 blocking, 일부 토픽만 non-blocking"으로 정리해 두면 운영 알람·재시도 정책의 일관성이 잡힌다.



## 한 줄 정리

| 키워드 | 한 줄 |
| --- | --- |
| `concurrency` | 한 인스턴스 안의 컨슈머 스레드 수. 파티션 수가 상한 |
| `KafkaListenerEndpointRegistry` | 빈 등록 후에도 런타임에 컨슈머를 시작·중지할 수 있는 진입점 |
| Blocking retry | 같은 파티션 헤드에서 재시도 — 순서 보존, 헤드 차단 |
| Non-blocking retry | 별도 retry 토픽으로 옮겨 재시도 — 헤드 비움, 순서 깨짐 |
| Trace 전파 | Spring Boot 3 + Micrometer Observation이 `traceparent`를 Kafka 헤더로 자동 운반 |
