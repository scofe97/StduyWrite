# DlqConsumer와 DLQ 소비 흐름

---

> ⚠️ **현재 코드 상태와의 차이** — 본 문서가 가리키는 `DlqConsumer.java`는 현재 `message-lib`(또는 다른 모듈)에 **존재하지 않는다**. DLT 토픽 발행은 `KafkaErrorConfig`의 `DefaultErrorHandler + DeadLetterPublishingRecoverer` 흐름이 그대로 처리하지만, 이를 소비하는 끝단 클래스는 운영 정책상 미배포 상태다. 본 문서는 "DlqConsumer가 있던 시점의 설계 의도"로 읽고, 도입을 검토한다면 미존재 패턴이 어떻게 구현될지를 함께 본다.

> 대상 코드 (당시)
>
> - `message-lib/src/main/java/org/okestro/tps/messaging/dlq/DlqConsumer.java` *(현재 미존재)*
> - `message-lib/src/main/java/org/okestro/tps/messaging/config/KafkaErrorConfig.java`
> - `message-lib/src/main/java/org/okestro/tps/messaging/topic/TopicConfig.java`

이 문서는 `DlqConsumer`가 현재 프로젝트에서 어떤 역할을 하는지, DLQ 흐름 안에서 어디에 위치하는지, 그리고 왜 "지금은 최소 구현"이라고 볼 수 있는지를 정리한다.


## 1. `DlqConsumer`는 무엇인가

`DlqConsumer`는 이름 그대로 **Dead Letter Queue 토픽을 소비하는 Kafka 리스너**다.

현재 구현은 매우 단순하다.

- `Topics.DLQ` 토픽을 구독한다.
- group id는 `tps-dlq-handler`다.
- 메시지를 받으면 토픽, 키, 크기, 헤더를 에러 로그로 남긴다.

즉 이 클래스는 "실패 메시지를 최종적으로 받아서 기록하는 끝단 소비자"다.


## 2. 전체 흐름에서 어디에 위치하는가

`DlqConsumer`는 혼자 동작하는 클래스가 아니라, `KafkaErrorConfig`와 함께 하나의 실패 처리 흐름을 완성한다.

흐름은 다음과 같다.

```mermaid
flowchart LR
    A["@KafkaListener"] -->|"예외 발생"| B["DefaultErrorHandler"]
    B -->|"1s → 2s → 4s 재시도"| A
    B -->|"재시도 소진 / non-retryable"| C["DeadLetterPublishingRecoverer"]
    C -->|"Topics.DLQ / partition 0"| D["tps.dlq"]
    D --> E["DlqConsumer"]
    E --> F["에러 로그 기록"]
```

즉:

1. 일반 consumer가 실패한다.
2. `KafkaErrorConfig`의 `DefaultErrorHandler`가 재시도한다.
3. 그래도 실패하면 `DeadLetterPublishingRecoverer`가 `Topics.DLQ`로 보낸다.
4. `DlqConsumer`가 그 DLQ 레코드를 읽는다.

그래서 `DlqConsumer`는 DLQ 체인의 마지막 수신자다.


## 3. 현재 코드가 실제로 하는 일

코드를 기준으로 보면 핵심은 한 줄이다.

```java
@KafkaListener(topics = Topics.DLQ, groupId = "tps-dlq-handler")
```

이 설정의 의미는 다음과 같다.

- 이 리스너는 DLQ 토픽만 소비한다.
- 일반 비즈니스 consumer group과 분리된 별도 group을 쓴다.
- 따라서 DLQ 메시지는 정상 consumer가 아니라 DLQ 전용 소비자가 처리한다.

메서드 본문은 다음 정보만 로그로 남긴다.

- topic
- key
- payload 크기
- headers

즉 **메시지 내용을 재처리하거나 저장하지는 않는다**.


## 4. 왜 이런 클래스가 필요한가

DLQ 토픽에 메시지를 보내는 것만으로는 운영적으로 충분하지 않다. 실패 메시지가 실제로 들어왔는지, 어떤 메타데이터를 갖고 있는지 확인할 수 있어야 한다.

`DlqConsumer`는 최소한 다음 역할을 한다.

- DLQ 메시지가 실제로 도착했는지 확인
- 헤더 기반 메타데이터 확인
- 운영자가 로그에서 빠르게 실패 흔적을 찾을 수 있게 함

즉 "DLQ로 보냈다"를 눈에 보이는 형태로 만드는 가장 단순한 소비자다.


## 5. 현재 구조의 장점

### 1. 단순하다

DLQ 처리를 별도 시스템 없이 바로 확인할 수 있다.

### 2. 전역 정책과 잘 맞는다

현재 프로젝트는 `KafkaErrorConfig`로 전역 retry/DLQ 정책을 쓰기 때문에, DLQ 소비자도 하나로 단순하게 두는 게 잘 맞는다.

### 3. 메타데이터 확인에 충분하다

헤더를 그대로 로그에 남기므로 다음을 볼 수 있다.

- 어떤 key였는지
- 어떤 CloudEvents 헤더가 붙었는지
- trace 관련 헤더가 있는지


## 6. 현재 구조의 한계

이 클래스는 작동은 하지만, 운영 관점에서는 **최소 구현**에 가깝다.

### 1. 로그만 남기고 끝난다

지금은 실패 메시지를 다시 처리하지 않는다.

즉 다음이 없다.

- 수동 재처리
- 자동 재처리
- DB 저장
- 장애 티켓 생성
- Slack/메일 알림

### 2. payload 본문을 직접 해석하지 않는다

현재는 payload 크기만 찍는다. 그래서 로그만 보고는 실패 메시지의 실제 비즈니스 의미를 바로 알기 어렵다.

### 3. DLQ 자체가 1파티션이다

`TopicConfig`를 보면 DLQ 토픽은 1파티션으로 생성된다.

그리고 `KafkaErrorConfig`도 DLQ 전송 시 partition 0으로 고정한다.

이 구조는 단순하지만:

- 실패량이 많아지면 병목이 될 수 있고
- 토픽별 격리 전략을 쓰기 어렵고
- 원본 파티션 정보를 적극 활용하지 못한다

### 4. DLQ 소비 실패에 대한 전략이 없다

`DlqConsumer` 자체에서 또 예외가 발생하면 어떻게 할지 별도 전략이 없다. 지금은 로그만 남기므로 위험이 작지만, 이후 저장/재처리 로직이 붙으면 이 문제를 고려해야 한다.


## 7. 실무적으로 어떻게 확장할 수 있나

현재 구조를 실무형으로 키우려면 보통 아래 방향으로 확장한다.

### 방향 1. 로그 + 저장

DLQ 메시지를 로그만 남기지 말고 DB나 별도 저장소에 적재한다.

예:

- 원본 topic
- 원본 key
- headers
- payload
- 실패 시각
- 예외 분류

이렇게 남기면 재처리 도구를 만들기 쉬워진다.

### 방향 2. 알림 연계

특정 예외나 특정 토픽의 DLQ 메시지는 Slack, 메일, PagerDuty 등으로 알림을 줄 수 있다.

### 방향 3. 수동 재처리 엔드포인트

운영자가 DLQ 저장소에서 메시지를 골라 다시 publish하도록 만들 수 있다.

### 방향 4. 자동 재처리와 영구 실패 분리

모든 DLQ를 바로 사람이 처리하게 두지 않고,

- 일시 장애형 실패
- 영구 데이터 오류

를 구분해 자동 재처리와 영구 격리를 나눌 수 있다.


## 8. 현재 프로젝트 기준 해석

현재 `DlqConsumer`는 "운영 완성형 DLQ 처리기"가 아니라, **DLQ 파이프라인이 살아 있는지 보여주는 최소 관측 지점**으로 보는 게 맞다.

즉 이 클래스의 가치는 다음에 있다.

- DLT/DLQ 라우팅이 실제로 되는지 확인 가능
- 실패 메시지의 헤더를 로그로 즉시 확인 가능
- 이후 더 고도화된 DLQ 운영 로직을 붙일 위치가 명확함

반대로 이 클래스만으로는 DLQ 운영이 끝났다고 보기 어렵다.


## 9. 지금 구조가 적절한 경우

현재 같은 단순 DLQ 소비가 적절한 경우:

- 학습/PoC 단계
- 공통 라이브러리의 기본 기능 검증
- 아직 재처리 요구가 크지 않은 서비스
- "우선 DLQ로 모으고 나중에 운영 자동화" 전략

부족해지는 경우:

- DLQ 메시지가 자주 쌓임
- 장애 대응 SLA가 있음
- 재처리와 분석이 중요함
- 여러 토픽/여러 서비스의 실패를 장기적으로 관리해야 함


## 10. 한 줄 정리

> `DlqConsumer`는 현재 프로젝트에서 DLQ 메시지를 실제로 받아 로그로 관측하게 해 주는 최소 구현 소비자다. 즉 DLQ 체인의 마지막 수신자이지만, 운영형 재처리 시스템이라기보다는 "실패가 여기까지 왔다"를 보여주는 시작점에 가깝다.
