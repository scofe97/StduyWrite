# 관측성과 장애 복구

---

> 분산 워크플로우에서 관측성(observability)과 장애 복구는 구현만큼 중요합니다. 묶음이 어느 스테이지에서 실패했는지, 어느 Worker가 처리했는지 추적할 수 없으면 운영 상황에서 디버깅이 사실상 불가능합니다.

## MDC 기반 분산 추적

각 로그에 `groupId`, `stageType`, `pipelineId`를 포함시키면 ELK Stack이나 Grafana Loki에서 특정 묶음의 전체 처리 흐름을 한눈에 조회할 수 있습니다. 기존 `SagaMdc` 패턴을 파이프라인 컨텍스트에 재활용합니다:

```java
/**
 * 파이프라인 MDC 유틸리티 — Logback 패턴에 %X{pipeline.groupId} 등을 포함하면
 * 모든 로그에 묶음/스테이지/파이프라인 컨텍스트가 자동으로 첨부된다.
 */
@Component
public class PipelineMdc {

    public static final String GROUP_ID    = "pipeline.groupId";     // 묶음 고유 ID
    public static final String STAGE_TYPE  = "pipeline.stageType";   // build | test | deploy
    public static final String PIPELINE_ID = "pipeline.pipelineId";  // 개별 파이프라인 ID

    /** Worker.onTask() 진입 시 호출하여 MDC에 컨텍스트를 설정한다. */
    public static void set(String groupId, String stageType, String pipelineId) {
        MDC.put(GROUP_ID,    groupId);
        MDC.put(STAGE_TYPE,  stageType);
        MDC.put(PIPELINE_ID, pipelineId);
    }

    /** finally 블록에서 호출하여 MDC 컨텍스트를 제거한다. */
    public static void clear() {
        MDC.remove(GROUP_ID);
        MDC.remove(STAGE_TYPE);
        MDC.remove(PIPELINE_ID);
    }
}
```

`PipelineWorker`의 `onTask` 메서드 시작 시 `PipelineMdc.set()`을 호출하고, 처리 완료 후 `finally` 블록에서 `PipelineMdc.clear()`를 호출합니다. Logback 패턴에 `%X{pipeline.groupId}` 같은 MDC 키를 포함시키면 모든 로그에 컨텍스트가 자동으로 첨부됩니다.



## Dispatcher SPOF 대응

`PipelineDispatcher`는 단일 Consumer이므로 단일 장애점(SPOF, Single Point of Failure)이 될 수 있습니다. Dispatcher가 다운되면 `control` 토픽 소비가 중단됩니다. 다음 두 가지 방법으로 완화할 수 있습니다:

- **Consumer Group 리밸런싱 활용**: Standby Dispatcher 인스턴스를 동일한 `pipeline-dispatcher` Consumer Group으로 기동해 둡니다. 활성 인스턴스가 다운되면 Redpanda가 자동으로 파티션을 Standby 인스턴스로 재할당합니다. 재할당 후 마지막 커밋 오프셋부터 소비를 재개하므로 메시지 유실이 없습니다.
- **이벤트 소싱으로 상태 복구**: `GroupStateTracker`의 인메모리 상태는 `control` 토픽의 이벤트 히스토리를 재생(replay)하여 복구할 수 있습니다. Dispatcher 재시작 시 `EARLIEST` 오프셋부터 읽어 완료된 묶음과 진행 중인 묶음 상태를 재구성합니다.



## 멱등성 보장

Jenkins API 호출은 네트워크 오류 시 동일 빌드를 중복 트리거할 수 있습니다. 이를 방지하기 위해 `ProcessedEvent` 테이블을 재활용합니다. `(correlationId, eventType)` 복합 키를 기본 키로 설정하면 중복 처리를 데이터베이스 레벨에서 차단합니다:

```java
/**
 * 멱등성 보장용 처리 이력 엔티티 — (correlationId, eventType) 복합 유니크 제약으로
 * 동일 요청의 중복 처리를 데이터베이스 레벨에서 차단한다.
 */
@Entity
@Table(name = "pipeline_processed_event",
       uniqueConstraints = @UniqueConstraint(columnNames = {"correlation_id", "event_type"}))
public class PipelineProcessedEvent {

    @Id @GeneratedValue
    private Long id;

    // 형식: {groupId}:{stageType}:{instanceIndex} — 요청을 고유하게 식별한다
    @Column(name = "correlation_id", nullable = false)
    private String correlationId;

    // JENKINS_TRIGGERED: Jenkins 호출 전, RESULT_REPORTED: 결과 발행 전
    @Column(name = "event_type", nullable = false)
    private String eventType;

    @Column(name = "processed_at")
    private Instant processedAt;
}
```

`correlationId`는 `{groupId}:{stageType}:{instanceIndex}` 형식으로 구성합니다. Worker가 Jenkins API를 호출하기 전에 이 레코드 삽입을 시도하고, `DataIntegrityViolationException`이 발생하면 이미 처리된 요청으로 판단하여 Jenkins 호출을 건너뜁니다.



## 실패 전략

스테이지 내 인스턴스 하나가 실패했을 때 두 가지 전략 중 하나를 선택한다. **fail-fast**는 실패 즉시 나머지 인스턴스를 중단하고 묶음 전체를 실패로 처리한다. **continue**는 나머지 인스턴스를 모두 실행한 뒤 묶음 최종 상태를 `PARTIAL_FAILURE`로 표시한다. 이 전략은 `StageDefinition`의 `onFailure` 필드로 스테이지별로 지정한다:

```java
/** 인스턴스 실패 시 onFailure 전략에 따라 묶음 상태를 갱신한다. */
private void handleFailure(String groupId, StageDefinition stage, String failedInstanceId) {
    // 실패한 인스턴스를 DLQ로 전송하여 사후 분석/재처리를 가능하게 한다
    publisher.publish(
            PipelineTopics.TASKS_DLQ, groupId
            , buildDlqRecord(groupId, stage, failedInstanceId)
    );

    if ("fail-fast".equals(stage.getOnFailure())) {
        // fail-fast: 실패 즉시 묶음 전체를 FAILED로 마킹하고 나머지 발행 중단
        stateTracker.markGroupFailed(groupId, failedInstanceId);
    } else {
        // continue: 실패를 기록하되 나머지 인스턴스는 계속 진행
        stateTracker.recordPartialFailure(groupId, failedInstanceId);
    }
}
```

Jenkins의 `parallel { failFast true }` 옵션과 같은 개념을 묶음 레벨로 확장한 것이다. 스테이지마다 전략을 다르게 지정할 수 있으므로, Build 스테이지는 `fail-fast`, Deploy 스테이지는 `continue`처럼 운영 정책에 맞게 조합할 수 있다.

두 전략 모두 실패한 인스턴스를 DLQ로 전송하는 점은 동일하다. DLQ에 쌓인 메시지는 운영자가 원인을 분석한 뒤 수동으로 `pipeline.tasks` 토픽에 재발행(requeue)하여 재처리할 수 있다. 재발행 시 동일한 `correlationId`를 사용하므로 멱등성 테이블에서 기존 레코드를 삭제해야 재처리가 가능하다.
