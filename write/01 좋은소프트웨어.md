# 좋은 소프트웨어 설계란 무엇이고 왜 필요한가?



다양한 소프트웨어 아키텍쳐 방법론이 있지만 각 방법론의 세부 사항부터 파악하기보다는 먼저 아키텍쳐 본질의 중요성을 먼저 이해해야함

- 클란 아키텍쳐를 깊이 탐구하기에 앞서 소프트웨어 아키텍쳐 정의, 소프트웨어 아키텍쳐가 비즈니스와 개발 과정에 미치는 영향, 아키텍쳐를 형성하는 주요 요소를 살펴본다.



## 소프트웨어 아키텍쳐의 정의

소프트웨어 아키텍쳐는 소프트웨어 시스템의 전체 구조와 설계 원칙을 정의하는 개념

- 소프트웨어 아키텍쳐의 주요 컴포넌트 / 컴포넌트간의 관계 / 상호작용 방식 / 시스템 전체에 적용되는 가이드와 제약
- 아키텍쳐는 일관된 방향으로 개발하기 위한 청사진 제공 및 코드의 뼈대를 세워준다.

구조의 결정

- 아키텍쳐 구현 단계 이전에 이뤄지는 고수준 결정으로, 초기 단계에서 올바른 서택이 중요
- 아키텍쳐가 영원히 불변하는것은 아니고 일부 요소는 변형 가능해야하지만 핵심 구조의 변경은 시스템에 큰 영향 미침
- 핵심 구조를 안정적으로 하기위해서는 소프트웨어의 목표와 잠재적 확장 방향을 파악해야한다.
- 가장 중요한 비즈니스 요구사항을 중심으로 구조를 설계하고, 부수적인 요소들을 그로부터 파생되도록 하는것이 핵심이다.

지식 공유

- 아키텍쳐는 개발 팀 내에서 시스템 지식을 공유하는 수단이기도 하다.
- 성공적인 프로젝트에서는 팀원들이 소프트웨어의 철한, 구조, 관련 지식을 공유할 때 지속된다.

두가지 측면 (구조의 결정 / 지식 공유)을 염두에 두고 아키텍쳐를 탐구하면 좋은 아키텍쳐를 만들기 위한 고려사항을 더 깊이 이해가능 



## 좋은 소프트웨어 아키텍쳐의 중요성

좋은 소프트웨어 아키텍쳐를 설계한다는건  소프트웨어 품질이 비즈니스 성공과 밀접하게 연관되는걸 인식 하는것이다.

소프트웨어 상품적 가치

- 소프트웨어는 고객에게 제공되는 상품이다. -> 상품의 가치는 기능뿐 아니라 품질에 의해 결정된다.
  - ex) 동일한 디자인의 옷이 2개(5만원, 6만원)존재 -> 기능적으로 유사해보이지만 6만원 짜리 옷이 더 많이 팔린다면 대게 품질 차이
  - 소프트웨어 품질도 동일 -> 품질(소프트웨어 구조, 아키텍쳐)
- 
- 좋은 아키텍쳐는 안정성, 확장성, 유지보수성을 보장한다. -> 고객 경험과 비즈니스 가치로 직결

구조의 우선순위

로버트 마틴은 '기능보다 구조가 더 중요하다 강조했다.'

- 기능이 작동하지 않아도 된다가 아니라, 구조가 잘 설계되면 소프트웨어는 기능적 문제를 신속히 수정/개선이 가능하다는 의미다.
- 디자인 스태미나 가설은 이를 시각적으로 보여준다. (마틴 파울러의 디자인 스태미나 그래프 이미지)
- 초기에는 구조를 무시하는게 더 효율적이지만 이후에는 점점 좋은 아키텍쳐가 기능 추가나 수정속도가 빨라진다.



변경 가능성의 본질

소프트웨어의 본질은 변경 가능성에 있다.

- 지속 가능한 서비스는 1번의 개발로 끝나지 않으며 새로운 요구사항이 추가되고 기능이 수정/확장되며, 버그 수정과 성능 개선이 반복된다.
- 개발자의 주요 활동은 코드를 읽고, 이해하고, 수정하거나 추가하는 것이다.
- 다른 개발자 및. 과거의 자신이 작성한 코드도 이에 해당

구조가 부실한 소프트웨어는 악순환을 겪으며 새로운 기능추가에도 어디를 수정해야할지 어렵고, 한부분의 변경이 예상치 못한 다른 부분에 영향을 미친다.

좋은 아키텍쳐는 선순환을 만들어 컴포넌트가 명확히 분리되고 의존성이 최소화된 구조는 변경 지점을 쉽게 파악할 수 있게 해준다.



비즈니스 관점에서의 가치

좋은 아키텍쳐는 비즈니스 관점에서도 명확한 이점이 있다.

- 비즈니스 환경은 끊임없이 변화하며 고객의 요ㅛ구사항에 신속히 대응하는 것이 성공의 열쇠다.
- 소프트웨어가 부실한 구조로 인해 필요한 기능을 적시에 제공하지 못한다면 지속 가능성을 위협바든ㄴ다.

개발자는 자신의 작업이 비즈니스에 미치는 영향을 인식해야 한다.

- 구조가 부실하면 코드 수정이 어렵고, 시간이 지나면 점점 이해하기 어려워진다.
- 좋은 아키텍쳐가 이러한 문제를 완화해줌



민첩성을 위한 아키텍쳐의 진화

비즈니스 민첩성을 극대화하려면 좋은 아키텍쳐뿐 아니라 이를 뒷받침하는 프로세스가 필요ㅛㅛ하다.

- CI/CD는 코드 변경을 자동으로 테스트하고 배포하는 파이프라인 제공 이를 통해 새로운 기능, 버그 수정을 빠르게 반영
- 컨테이너화는 배포환경의 일관성을 보장한다.

- MSA 아키텍쳐는 민첩성을 극대화해 각 팀이 독립적으로 개발하고 배포가능해야한다. (서로간의 의존성을 최소화하고, 독립적 배포가 가능해야함)
- 좋은 아키텍쳐에서 Ci/CD, 컨테이너화, 마이크로서비스로 이어지는 흐름은 독립적인 유행이 아니라, 비즈니스 민첩성을 위한 통합된 발전과정



미래를 대비한 설계

좋은 아키텍쳐는 즉각적인 요구사항 뿐 아니라 미래의 확장성을 고려해야한다.

- 마이크로서비스의 전환을 염두에두고 기능 분리, 의존성 최소화

좋은 소프트웨어 아키텍쳐는 단순히 기술적 설계를 넘어 비즈니스 성공과 개발 효율성을 만들어냄 



## 아키텍쳐에 영향을 주는 요소

소프트웨어 아키텍쳐 설계에서 변경이 용이한 구조를 만드는 것은 중요하지만 어떤 방향으로 변경 가능성을 확보하고, 어떤 목표를 기준으로 구조를 설계해야하는지 어떻게 알 수 있는가?

실무적으로는 아키텍쳐는 이해관계자의 요구사항을 반영하는 과정에서 형성된다.

- 이해관계자는 고객, 마케팅, 혹은 다른 부서일 수 있으며 이들의 요구사항은 아키텍쳐 설계의 방향을 결정하는 핵심 요인이다.
- 크게 3가지 있음 (1. 기능 요구사항, 2. 제한 요소, 3. 품질 속성)

기능 요구사항

기능 요구사항은 시스템이 제공해야하는 구체적 기능을 정의한다.

- ex) 2숫자의 합을 계산해 반환한다. -> 시스템이 무엇을 해야하는지를 다루며, 어떻게 구현할지의 세부사항은 포함하지 않음
- 따라서 기능 요구사항 자체는 아키텍쳐에 직접적인 영향을 미치지 않을 수 있다.
- 동일한 기능을 구현하더라도 함수 하나로 처리하거나 객체지향으로 분리된 구조로 설계할 수 있다.

제한 요소

제한 요소는 시스템 설계와 구현에 영향을 미치는 제약조건을 의미한다.

- 개발 일정 / 예산 / 하드웨어 자원 / 기술 스택의 제한이 될 수 있다.
- ex) 메모리가 제한적이라면 메모리 효율적인 코드구조 설계, 짧은 개발 일정은 단순화된 아키텍쳐 요구를 할 수 있다.
- 제한요소는 기능 요구사항보다 아키텍쳐에 더 직접적인 영향을 미치며, 설계 시 현실적인 타협을 찾아야한다.

품질속성

품질 속성은 소프트웨어가 충족해야 하는 비기능적 요구사항으로, 시스템의 성능, 확장성, 유지보수성 등을 정의한다.

소프트웨어의 목적과 비즈니스 목표에 따라 어떤 품질 속성을 우선순위로 둘지 결정되며, 이는 아키텍쳐 설계의 핵심 방향성을 형성한다.

- ex) 도서관 시스템을 개발하며 "새로운 책을 추가할 때 기존 코드를 수정하지 않아야 한다."는 품질 속성이 있다면 이를 충족하기 위해 확장 가능한 구조를 도입해야 한다.
- 일반적인 품질 속성의 예시
  - 가용성: 시스템 다운타임이 연간 1분 미만
  - 성능: API 평균 응답 시간이 100ms
  - 확장성: 100만 대의 디바이스를 성능 저하 없이 지원
  - 변경 용이성: 수정에 필요한 노력을 최소화
  - 유지보수성: 장애 발생 시. 시간 내 인지, 3시간 내 복구
  - 테스트 용이성: 비즈니스 로직 코드의 단위 테스트 커버리지 90% 이상

아키텍쳐 설계의 기준

아키텍쳐는 기능 요구사항, 제한 요소, 품질 속성을 종합적으로 고려해서 설계된다.

- 이 과정에서 이해관계자의 요구를 명확히 파악하고

## 의도를 드러내는 아키텍쳐

좋은 아키텍쳐를 실제로 설계하고 유지하려면 무엇이 필요한가? -> 핵심은 아키텍쳐가 시스템의 의도를 명확히 드러내야한다./

아키텍쳐의 의도가 명확하다는 것은 시스템의 구조를 통해 그 목적과 비즈니스 가치를 직관적으로 파악할 수 있음을 나타낸다. 단순 문서나 주석만으로 달성되지 않는다.

- 코드와 구조 자체가 비즈니스 도메인을 명확히 표현해야 한다. 
- ex) 온라인 쇼핑몰 시스템의 아키텍쳐는 "주문", "결재", "배송", "상품 관리"와 같은 비즈니스 개념을 중심으로 구성 DatabaseManager, HttpClient, CacheHandler 같은 기술적 세부사항은 보조적이고 교체 가능한 요소로 다뤄줘야 한다.
- 비즈니스 로직이 우선순위를 가지며, 기술적 구현은 이를 뒷받침 해야함

단순성과 지속성

아키텍쳐는 단순하고 이해하기 쉬워야 지속성을 띤다. 복잡한 구조는 초기에는 정교해 보일지라도 시간이 지나면서 원래 의도를 잃기 쉽다.

새로운 팀원이 합류하거나 기존 개발자가 오래된 코드를 다시 볼 때 복잡한 구조는 이해하기 어렵게 만든다.

- 이를 방지하려면 시스템의 구성 요소를 성격에 따라 명확히 분리하고, 각 요소 간의 관계와 의존성을 직관적으로 정의해야 한다.
- 비즈니스 로직은 비즈니스 로직끼리, 인프라 코드는 인프라끼리, UI는 UI끼리 묶되 경계가 명확해야 한다.

복잡한 구조는 혼란을 초래한다.

- ex) 비즈니스 로직과 db접근이 뒤섞인다면 코드 수정마다 어디를 수정해야하는지 파악하기 어렵다.
- 유지보수 비용 및 변경속도 늦춤

일관성의 중요성

- 단기적으로 동작하더라도 아키텍쳐의 일관성을 무시하고 임시방편으로 코드를 작성하면 시스템은 점차 스파케티 코드로 변질된다.
- 변경 용이성과 유지보수성을 해치며, 장기적으로 비즈니스 민첩성을 저해한다.

격리와 변경 관리

의도가 명확한 아키텍쳐는 변경, 장애, 보안사고 발생 시. 그 영향 범위를 효과적으로 관리할 수 있게 한다.

인터페이스를 통해 컴포넌트 간의 경계를 명확히 구분하면 특정 부분의 변경이 다른 영역에 불필요한 영향을 미치지 않는다.

- ex) 결제 시스템을 외부 서비스에서 내부 구현으로 전환할때 잘 설계된 아키텍쳐라면 결제 인터페이스를 구현하는 부분만 수정한다.
- 주문이나 배송같은 다른 비즈니스 로직은 변경없이 최소화한다.

살아있는 아키텍쳐

아키텍쳐는 정적인 설계도가 아니다. 비즈니스 요구사항과 환경이 변화함에 따라 아키텍쳐도 진화해야 한다.

- 그 과정에서 원래의 의도와 일관성을 잃지 않도록 주의해야한다.
- 새로운 기능을 추가하거나 기존 기능 수정할 때마다 "이 변경이 아키텍쳐의 원칙과 부합하는가?"를 검토해야 한다.

변경에 대한 용기

TDD가 테스트를 통해 코드 변경에 대한 자신감을 제공하듯, 명확한 아키텍쳐는 개발자가 구조적 변경을 자신있게 수행할 수 있게 한다.

- 시스템의 규칙과 경계가 명확할 때 개발자는 필요한 수정이 전체 시스템에 미치는 영향을 예측하고 필요한 부분만 정확히 수정
- 의도를 드러내는 아키텍쳐는 비즈니스의 본질을 코드 구조에 투영하며, 변경에 유연하게 대응할 수 있게한다.



## 아키텍쳐가 개발자에게 주는 영향

좋은 소프트웨어는 시스템의 품질과 비즈니스 가치를 높일 뿐 아니라, 코드를 작성하고 유지하는 개발자의 업무 경험에 직접적인 영향을 미친다.

변경의 명확성과 효율성

잘 설계된 아키텍쳐는 새로운 요구사항이나 수정 요청이 발생했을 때 변경이 필요한 지점을 직관적으로 파악할 수 있게 한다.

- 컴포넌트간 경계가 명확하고 의존성이 체계적으로 관리되기 때문에 변경의 영향 범위를 쉽게 예측하고 필요한 부분만 수정 이는 개발 속도를 높이고 변경으로 인한 버그 발생 가능성을 줄인다.
- 반면 구조가 부실하면 시스템에서는 수정해야할 지점을 찾는 데 시간이 걸리고, 의도치 않은 파급 효과로 인해 추가적인 디버깅 필요

개발자의 자신감과 만족도

깔끔하고 직관적인 코드베이스에서 작업하는건 개발자에게 자신감을 심어준다.

- 명확한 구조는 코드 작성과 리팩토링을 두려움없이 수행하게하며, 업무만족도와 직결된다.
- 반대로 복잡하게 얽힌 코드베이스는 스트레스를 유발하고, 개발자가 자신의 작업에 확신을 갖기 어렵게 만든다.

팀 협업의 향상

아키텍쳐가 명확하면 팀원간 의사소통과 협업이 원할해진다.

- 공통된 구조와 설계 원칙을 공유하면 코드 리뷰나 기술 논의가 더 생산적으로 이루어진다.
- Ex) 특정 기능의 구현 방식을 논의할 때 모두가 동일한 아키텍쳐 맥락을 이해하고 있으면 불필요한 오해가 줄어든다.

장기적 생산성

좋은 아키테쳐는 단기적인 개발 속도 뿐만 아니라 장기적인 생산성 보장



좋은 소프트웨어 아키텍쳐는 단순히 기술적 설계의 우수성을 넘어 개발자의 업무 환경과 생산성을 근본적으로 개선한다.

- 명확한 구조 -> 효율적인 작업, 원할한 협업, 높은 만족도
