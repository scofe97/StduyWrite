# Logback 학습 및 실습 가이드
---
> 작성일: 2026-04-17
> 대상: Spring Boot 기반 Java 서비스 운영 로그를 직접 설계하고 유지보수해야 하는 개발자
> 학습 목표: Logback의 핵심 개념을 이해하고, TPS 계열 서비스의 실제 설정을 읽고 수정하고 검증할 수 있다.

## 1. 이 문서를 왜 읽어야 하는가

실무에서 로그는 단순 출력이 아니다. 장애를 빠르게 좁히기 위한 운영 인터페이스이고, 코드 변경의 부작용을 가장 먼저 드러내는 관측 신호다. 그런데 많은 경우 개발자는 `log.info()`를 쓰는 법만 알고, 그 로그가 어떤 규칙으로 파일에 쓰이고, 왜 특정 logger만 보이고, 왜 어떤 서비스는 `o.o.t...`처럼 축약되어 보이는지까지는 이해하지 못한다.

Logback을 알아야 하는 이유는 세 가지다.

- 로그 포맷을 설계할 수 있다.
- 서비스별 로그 볼륨과 가독성을 제어할 수 있다.
- tracing, MDC, 샘플링, rolling policy 같은 운영 요구를 코드와 설정 양쪽에서 판단할 수 있다.

이 문서는 개념 설명에서 끝나지 않는다. TPS의 `workflow-api`, `common-api`, `pipeline-api`, `executor`, `message-lib`를 사례로 삼아 "왜 이 설정이 이렇게 되어 있는가"를 읽는 힘까지 목표로 한다.



## 2. Logback을 한 문장으로 정의하면

Logback은 Java 애플리케이션에서 로그를 어떤 형식으로, 어떤 대상에, 어떤 조건으로 출력할지를 제어하는 로깅 백엔드다.

조금 더 정확히 말하면 역할은 다음과 같이 나뉜다.

- 애플리케이션 코드는 보통 `SLF4J` API로 `log.info()`를 호출한다.
- Logback은 그 호출을 실제 콘솔 출력, 파일 저장, 패턴 포맷팅, 레벨 필터링으로 실행한다.

즉, 코드가 "무엇을 기록할지"를 말한다면, Logback은 "어떻게 기록될지"를 결정한다.



## 3. 가장 먼저 잡아야 하는 구조: SLF4J와 Logback의 관계

초보자가 가장 자주 헷갈리는 지점은 "나는 Logback을 쓰는 건가, SLF4J를 쓰는 건가"이다. 정답은 둘 다 쓴다.

관계는 이렇다.

| 계층 | 역할 | 예시 |
|------|------|------|
| 로깅 API | 애플리케이션 코드가 호출하는 인터페이스 | `SLF4J`, `LoggerFactory`, `@Slf4j` |
| 로깅 구현체 | 실제 출력과 설정을 담당하는 백엔드 | `Logback` |
| 설정 파일 | 포맷, appender, rolling, logger level 정의 | `logback-spring.xml` |

실무에서는 대부분 `lombok @Slf4j`로 logger를 만들고, 애플리케이션은 SLF4J에만 의존한다. Logback을 교체할 수 있는 이유도 여기에 있다. 다만 TPS 계열 Spring Boot 서비스는 사실상 Logback을 표준처럼 사용하고 있으므로, 운영 관점에서는 Logback 설정을 읽는 능력이 필요하다.



## 4. Logback의 핵심 구성요소 4개

Logback을 이해하려면 아래 네 개만 먼저 잡으면 된다.

### 4.1 Logger

Logger는 로그를 찍는 주체다. 보통 클래스 단위로 하나씩 생기며 이름은 패키지 경로를 따른다.

예를 들어 다음 클래스가 있다고 하자.

```java
package org.okestro.tps.execution.infrastructure.scheduler;

@Slf4j
public class DispatchDefenseScheduler {
}
```

이 클래스의 logger 이름은 기본적으로 `org.okestro.tps.execution.infrastructure.scheduler.DispatchDefenseScheduler`다.

Logback 설정에서 `logger name="org.okestro"`를 선언하면 그 하위 패키지 전체에 규칙이 적용된다. 이것이 logger hierarchy다.

### 4.2 Level

가장 흔한 로그 레벨은 다음 다섯 개다.

- `DEBUG`: 개발 중 내부 상태 확인
- `INFO`: 정상적인 주요 상태 전이, 마일스톤
- `WARN`: 회복 가능 실패
- `ERROR`: 운영 개입이 필요한 실패
- `OFF`: 출력 금지

중요한 점은 level은 단순 "심각도"가 아니라 "운영 의미"다. 자주 발생한다고 INFO로 낮추면 안 되고, 알람 가치가 없는데 ERROR로 남발해도 안 된다.

### 4.3 Appender

Appender는 로그를 어디에 쓸지를 결정한다.

- `ConsoleAppender`: 콘솔
- `RollingFileAppender`: 롤링 파일
- 기타: Socket, Async, DB 등

TPS API 계열 서비스는 주로 `STDOUT` + `RollingFile` 조합을 쓴다.

### 4.4 Encoder / Pattern

Encoder는 로그 한 줄을 문자열로 바꾼다. 여기서 자주 보는 것이 pattern이다.

예:

```xml
<pattern>%d{yyyy-MM-dd HH:mm:ss} [%highlight(%level)] [%thread] %logger{36} - %msg%n</pattern>
```

이 패턴이 한 줄 로그의 모양을 결정한다.



## 5. 패턴 문자열을 읽는 법

실무에서 가장 자주 보게 되는 건 pattern이다. 아래 토큰은 거의 외워도 된다.

| 토큰 | 의미 |
|------|------|
| `%d{...}` | 시간 |
| `%level` | 로그 레벨 |
| `%thread` | 스레드 이름 |
| `%logger{36}` | logger 이름, 길이 제한 축약 |
| `%msg` | 실제 메시지 |
| `%n` | 줄바꿈 |
| `%X{key}` | MDC 값 |

예를 들어

```xml
%d{yyyy-MM-dd HH:mm:ss} [%highlight(%level)] [%thread] %logger{36} - %msg%n
```

는 다음과 같은 로그를 만든다.

```text
2026-04-17 17:22:41 [INFO] [scheduling-1] o.o.t.t.e.i.s.DispatchDefenseScheduler - [DispatchDefense] Aged PENDING: candidates=12, approved=7
```

여기서 `o.o.t.t.e.i.s.`처럼 보이는 이유는 `%logger{36}`이 logger 이름을 36자 안으로 축약하기 때문이다. `workflow-api`, `common-api`, `pipeline-api` 모두 이 패턴을 사용하고 있어 같은 현상이 발생한다.



## 6. Spring Boot에서 왜 `logback-spring.xml`을 쓰는가

> Logback 기본 파일명은 `logback.xml`이지만 Spring Boot 프로젝트에서는 `logback-spring.xml`을 더 많이 쓴다. 이유는 Spring 확장 기능을 쓸 수 있기 때문이다.

대표 예시는 두 가지다.

- `<SpringProfile name="local">`
- `<springProperty name="APP_NAME" source="spring.application.name" ... />`

즉, profile별 appender/level 분기나 Spring Environment 값을 끌어오고 싶으면 `logback-spring.xml`이 더 적합하다.

TPS 서비스들이 이 방식을 쓰는 이유도 같다. 로컬과 비로컬 환경에서 출력 전략이 다르기 때문이다.



## 7. TPS 계열 설정을 읽는 법

### 7.1 공통 구조

`workflow-api`, `common-api`, `pipeline-api`의 Logback 설정은 거의 같은 골격을 가진다.

1. 공통 framework logger level 선언
2. `LOG_PATTERN` property 선언
3. `local` profile에서는 콘솔만 출력
4. `!local` profile에서는 롤링 파일 + 콘솔 출력
5. `root level="OFF"`로 두고 필요한 logger만 개별 활성화

이 구조는 "필요한 logger만 열어두고, 나머지는 조용히 둔다"는 철학이다.

### 7.2 executor의 현재 방향

executor도 이 공통 패턴에 맞추되, 여기에 trace/span 노출만 추가한 상태다.

핵심 차이는 이것뿐이다.

```xml
[traceId=%X{traceId:-}] [spanId=%X{spanId:-}]
```

즉, formatting 철학은 공통 API 모듈과 같고, 관측성 요구 때문에 tracing 칸만 추가한 것이다.

### 7.3 message-lib의 로그는 누구 설정을 따르는가

`message-lib`는 라이브러리이므로 별도의 Logback 정책을 스스로 집행하지 않는다. `@Slf4j`로 SLF4J 호출만 하고, 실제 출력은 실행 애플리케이션이 가진 Logback 설정을 따른다.

그래서 executor에 붙으면 `message-lib` 로그도 executor의 `logback-spring.xml` 패턴으로 출력된다. 이 점은 실무에서 중요하다. 라이브러리의 로그 정책은 라이브러리 내부보다 소비 애플리케이션의 logger 설정이 더 큰 영향을 준다.



## 8. root logger와 package logger를 어떻게 이해해야 하는가

다음 설정을 보자.

```xml
<logger name="org.okestro" level="DEBUG" />

<root level="OFF">
    <appender-ref ref="STDOUT" />
</root>
```

이 설정의 의미는 "전체 기본값은 끄고, `org.okestro` 하위만 DEBUG로 켠다"에 가깝다.

실무적으로 장점은 분명하다.

- 외부 라이브러리 로그가 과도하게 쏟아지지 않는다.
- 우리 코드 로그만 통제하기 쉽다.
- 장애 대응 시 logger 범위를 의도적으로 좁히기 좋다.

단점도 있다.

- 필요한 외부 라이브러리 logger를 깜빡하면 로그가 안 보일 수 있다.
- 설정을 잘못 이해하면 "왜 내 로그가 안 찍히지?"라는 상황이 생긴다.

즉, root를 OFF로 두는 전략은 강력하지만 보수적이다. 그래서 framework logger들을 별도 선언하는 패턴이 같이 붙는다.

## 9. RollingFileAppender는 왜 필요한가

운영 환경에서 파일 로그는 무한히 커지면 안 된다. 그래서 rolling policy를 둔다.

예:

```xml
<rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
    <fileNamePattern>${LOGS}/api-%d{yyyy-MM-dd}.%i.log</fileNamePattern>
    <timeBasedFileNamingAndTriggeringPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
        <maxFileSize>100MB</maxFileSize>
    </timeBasedFileNamingAndTriggeringPolicy>
</rollingPolicy>
```

이 의미는 날짜 단위 파일명에 size 조건을 함께 걸겠다는 뜻이다. 즉 하루가 지나거나, 하루 안에서도 파일이 너무 커지면 분할된다.

운영 관점에서 이 구성이 필요한 이유는 다음과 같다.

- 디스크 사용량 폭주 방지
- 로그 수집기(Filebeat, Fluent Bit 등)와의 궁합 유지
- 장애 시 특정 날짜 로그 추적 용이

## 10. MDC와 tracing은 어떻게 구분해야 하는가

이 부분은 실무에서 자주 혼동된다.

### 10.1 MDC

MDC는 현재 스레드의 로그 문맥에 값을 붙이는 방식이다.

예:

```xml
[jobExcnId=%X{jobExcnId:-}]
```

장점은 간단하고 즉시 효과가 있다는 점이다. 단점은 스레드 경계, 비동기 경계, 누수 위험, 도메인 결합이 있다.

### 10.2 tracing

tracing은 request/span 단위의 추적 컨텍스트를 별도 시스템(OpenTelemetry 등)으로 관리한다.

장점은 분산 시스템에 강하다는 점이다. HTTP, Kafka, OutboxPoller처럼 경계를 넘어도 trace를 이어갈 수 있다. 단점은 provider, exporter, agent 같은 배포 구성이 필요하다.

### 10.3 현재 TPS 방향에서의 판단

지금 방향은 이렇다.

- 비즈니스 식별자는 로그 본문 `key={}` 위주
- trace/span은 tracing 시스템으로 처리
- MDC 강제 주입은 최소화 또는 금지

이 판단은 적절하다. MDC를 남발하면 잠깐은 편하지만 구조가 빨리 꼬인다.

## 11. 로그 prefix 정책을 왜 별도로 두는가

패턴은 로그 줄의 외형을 정하지만, prefix는 메시지의 의미 체계를 정한다.

예:

- `[BuildLog]`
- `[DispatchDefense]`
- `[OutboxPublish]`

이 prefix의 장점은 grep 가능성이다.

- `grep "\[BuildLog\]"`
- `grep "\[.*Defense\]"`
- `grep "\[OutboxPublish\]"`

즉 pattern은 공통 껍데기이고, prefix는 도메인별 운영 필터링 키다. 둘은 역할이 다르다.

## 12. 성공 로그 샘플링은 왜 필요한가

라이브러리나 인프라 계층 로그는 성공 이벤트 수가 많아질수록 운영 가치를 잃기 쉽다. 특히 outbox poller처럼 고주기/대량 처리 루프에서는 성공 `INFO` 로그가 전체 로그를 덮어버릴 수 있다.

그래서 `message-lib`에는 성공 발행 로그 샘플링을 추가했다.

핵심 전략은 다음과 같다.

- `success-log-sample-rate: 0~100`
- `0`: 성공 로그 없음
- `100`: 전부 출력
- `1~99`: `aggregateId` 우선, 없으면 `eventId` 기준 결정적 샘플링

결정적 샘플링을 쓰는 이유는 랜덤 샘플링보다 재현성이 높기 때문이다. 같은 aggregate는 항상 찍히거나 항상 안 찍히므로 추적성이 유지된다.

반대로 실패(`ERROR`)와 DEAD(`WARN`)는 샘플링하지 않는다. 이런 로그는 운영 의미가 크기 때문이다.

## 13. 실습 1: logger 축약을 직접 확인하기

### 목표

`%logger{36}`가 왜 `o.o.t...`처럼 보이는지 눈으로 확인한다.

### 절차

1. `workflow-api`, `common-api`, `pipeline-api`, `executor`의 `logback-spring.xml`에서 `%logger{36}`를 찾는다.
2. 긴 패키지명을 가진 클래스를 하나 고른다.
3. 실제 로그에서 해당 클래스가 어떻게 축약되는지 본다.
4. `%logger{0}` 또는 `%logger`로 바꿨을 때 어떤 차이가 날지 예상해본다.

### 체크 질문

- `%logger{36}`은 왜 가독성을 해칠 수 있는가?
- 그럼에도 여러 서비스가 이 패턴을 유지하는 이유는 무엇인가?

## 14. 실습 2: trace/span 칸을 추가해보기

### 목표

기존 공통 패턴을 유지하면서 trace/span만 추가하는 감각을 익힌다.

### 실습 패턴

```xml
<property name="LOG_PATTERN"
          value="%d{yyyy-MM-dd HH:mm:ss} [%highlight(%level)] [%thread] [traceId=%X{traceId:-}] [spanId=%X{spanId:-}] %logger{36} - %msg%n"/>
```

### 체크 포인트

- trace/span 값이 없을 때는 빈칸으로 보인다.
- 이 값은 Logback이 생성하는 게 아니라 외부 tracing 시스템 또는 bridge가 채운다.
- `message-lib`만으로는 provider가 생기지 않는다.

## 16. 실습 4: Outbox 성공 로그 샘플링 읽기

### 목표

샘플링이 logger level과 다른 문제라는 점을 이해한다.

### 확인 파일

- `message-lib/src/main/java/.../OutboxPoller.java`
- `message-lib/src/main/java/.../OutboxProperties.java`

### 관찰 포인트

1. 실패 로그는 항상 남는다.
2. 성공 로그는 `success-log-sample-rate`가 0이면 안 남는다.
3. 10% 샘플링이어도 같은 `aggregateId`는 일관된 결과를 가진다.

### 예시 로그

```text
2026-04-17 18:02:11 [INFO] [scheduling-1] [traceId=...] [spanId=...] o.o.t.t.m.a.o.OutboxPoller - [OutboxPublish] Published event: id=101, aggregateId=JOB-1, type=ExecutionRequested, topic=tps.execution.requested, retryCount=0
2026-04-17 18:02:12 [ERROR] [scheduling-1] [traceId=...] [spanId=...] o.o.t.t.m.a.o.OutboxPoller - [OutboxPublish] Failed to publish outbox event: id=102, aggregateId=JOB-2, type=ExecutionRequested, topic=tps.execution.requested, retryCount=2
2026-04-17 18:02:13 [WARN] [scheduling-1] [traceId=...] [spanId=...] o.o.t.t.m.a.o.OutboxPoller - [OutboxPublish] Outbox event exceeded max retries and was marked DEAD: id=103, aggregateId=JOB-3, type=ExecutionRequested
```

## 17. 실습 5: 서비스별 logger를 따로 둘지 판단하기

### 목표

logger 분리가 진짜 필요한 상황을 구분한다.

### 판단 기준

다음 질문에 모두 "예"가 아니면 보통 분리하지 않는 편이 낫다.

- 특정 라이브러리 로그 볼륨이 과도한가?
- 라이브러리 로그의 운영 중요도가 도메인 로그와 확실히 다른가?
- 샘플링만으로는 볼륨 제어가 충분하지 않은가?

### 현재 TPS 문맥의 해석

지금은 `message-lib` logger를 따로 분리할 필요가 크지 않다. 먼저 샘플링과 공통 패턴 통일을 유지하는 쪽이 더 단순하고 일관된다.

## 18. 권장 학습 순서

Logback을 처음 공부한다면 아래 순서를 권장한다.

1. `SLF4J`와 `Logback` 역할 분리 이해
2. `logger`, `level`, `appender`, `pattern` 4요소 이해
3. `%logger{36}`, `%X{key}`, rolling policy 읽기
4. Spring Boot의 `logback-spring.xml`과 `SpringProfile` 이해
5. TPS 실제 설정을 읽으며 공통 구조 파악
6. tracing, MDC, prefix, sampling을 분리해서 사고

이 순서를 건너뛰고 바로 tracing이나 샘플링으로 들어가면 설정의 본질보다 기법만 남기 쉽다.

## 19. 이 문서를 읽고 나서 할 수 있어야 하는 것

이 문서를 제대로 이해했다면 다음 질문에 답할 수 있어야 한다.

- 왜 `workflow-api` 로그는 `[INFO]`인데 logger는 `o.o.t...`처럼 보이는가?
- 왜 `message-lib` 로그는 executor의 Logback 설정을 따르는가?
- 왜 tracing provider는 `message-lib`가 아니라 실행 환경이 책임지는가?
- 왜 성공 로그 샘플링은 logger level 조정과 다른 문제인가?
- 언제 `org.okestro.tps.messaging` logger를 별도로 분리해야 하는가?

## 20. 다음 학습 주제

Logback을 이해한 뒤 바로 이어서 공부하면 좋은 주제는 다음과 같다.

- OpenTelemetry Java agent와 애플리케이션 내부 SDK 방식 비교
- MDC와 tracing context propagation 차이
- Kafka 메시지 전파에서 `traceparent`가 어떻게 이어지는가
- 로그 수집기(Fluent Bit, Filebeat)와 rolling policy의 궁합
- Grafana Loki에서 prefix와 trace ID를 같이 활용하는 검색 전략

## 21. 결론

Logback은 단순 XML 설정 파일이 아니다. 운영에서 무엇을 볼 수 있고, 무엇을 놓치며, 로그 비용을 어떻게 통제할지를 결정하는 관측성의 첫 계층이다.

TPS 서비스들을 보면 이미 공통 철학이 드러난다. root는 보수적으로 닫고, 필요한 logger만 열고, 콘솔과 롤링 파일을 profile별로 나누고, 패턴은 통일한다. 여기에 최근 요구사항으로 trace/span, prefix 정책, 샘플링 같은 관측성 설계가 덧붙고 있다.

실무적으로 중요한 것은 "Logback 문법을 외우는 것"이 아니라 "이 설정이 왜 이렇게 존재하는지"를 설명할 수 있는 상태가 되는 것이다. 그 수준에 도달하면 새로운 서비스의 로그 정책도 더 일관되고 덜 임기응변적으로 설계할 수 있다.
