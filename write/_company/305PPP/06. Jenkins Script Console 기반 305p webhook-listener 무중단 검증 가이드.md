# Jenkins Script Console 기반 305p webhook-listener 무중단 검증 가이드
---
> Jenkins를 재기동하지 않고 305p용 `webhook-listener` 복제본을 메모리에 동적 등록해 검증하는 절차를 정리한 가이드다.
> 작성일: 2026-04-21
> 대상: `305p Jenkins`, `webhook-listener.groovy`, `Script Console`

## 1. 이 가이드의 목적

> 이 문서는 "정식 배포"가 아니라 "무중단 검증"을 위한 절차서다.

이번 방식의 목표는 Jenkins를 껐다 켜지 않고 305p용 리스너를 먼저 검증하는 것이다. 따라서 이 문서는 다음 상황에만 쓴다:

- 새 리스너 로직이 실제 Jenkins 이벤트를 정상 수집하는지 확인하고 싶을 때
- 운영 시간 중 재기동 없이 빠르게 동작 여부를 검증하고 싶을 때
- 이후 GitOps 정식 반영 전에 리스크를 줄이고 싶을 때

반대로 이 문서는 다음 목적에는 쓰지 않는다:

- 영구 배포
- 재기동 후에도 유지되는 최종 상태 관리
- ArgoCD 기준 배포 정본 관리

즉 이 방식은 "검증용"이고, "정식 반영용"은 아니다.



## 2. 이 방식이 가능한 이유와 한계

> Jenkins는 실행 중에도 Groovy를 통해 `RunListener`를 메모리에 등록할 수 있다. 하지만 메모리 등록은 영속적이지 않다.

Jenkins Script Console은 현재 실행 중인 JVM 안에서 Groovy를 바로 실행한다. 따라서 `RunListener`를 찾아 제거하고 새 리스너를 등록하는 작업을 재기동 없이 수행할 수 있다.

하지만 이 방식에는 중요한 한계가 있다:

- Jenkins 재기동 후 사라질 수 있다.
- 누가 언제 등록했는지 Git diff만으로는 추적되지 않는다.
- 잘못 작성하면 리스너 중복 등록 또는 이벤트 중복 발행이 생길 수 있다.

따라서 이 방식의 원칙은 하나다. "검증이 끝나면 정식 GitOps 반영으로 넘어간다."



## 3. 현재 기준 접속 정보

> 작업 전에 접속 대상과 인증 정보를 정확히 확인해야 한다.

현재 확인된 305p Jenkins 기본 정보는 다음과 같다:

| 항목 | 값 |
|------|----|
| 네임스페이스 | `trb-oss` |
| Service | `jenkins` |
| Service 주소 | `http://jenkins.trb-oss.svc.cluster.local:8080` |
| 계정 | `admin` |
| 비밀번호 | `cloud1234` |

로컬 브라우저에서 UI로 접근할 경우에는 현재 포털 도메인 또는 사내 접근 경로를 따른다. 클러스터 내부 점검만 필요하면 `kubectl port-forward`로 충분하다.

예시는 다음과 같다:

```bash
kubectl --context tps-v305p -n trb-oss port-forward svc/jenkins 18080:8080
```

이후 브라우저에서 `http://127.0.0.1:18080`으로 접속해 Script Console을 사용할 수 있다.



## 4. 검증 전에 반드시 지켜야 할 원칙

> 무중단 검증이라도 운영 안전장치는 필요하다.

적용 전에 반드시 아래 원칙을 지킨다:

1. 기존 리스너와 구분되는 고유 클래스명 또는 logger명을 쓴다.
2. 등록 전에 같은 이름의 기존 리스너를 제거한다.
3. 적용 직후 테스트 빌드 1건만 흘려서 이벤트를 확인한다.
4. 실제 운영 Job 전체에 바로 확산하지 않는다.
5. 실패 시 즉시 제거 가능한 롤백 스크립트를 같이 준비한다.

이 원칙이 없으면 "무중단 검증"이 아니라 "무중단 장애 주입"이 된다.



## 5. 이번 검증의 권장 설계

> 305p는 Redpanda Connect 내부 DNS 의존보다 NodePort 직행을 기본으로 잡는 편이 안전하다.

이번 검증에서는 다음 기본 정책을 쓴다:

- 원본 `executor/webhook-listener.groovy`는 건드리지 않는다.
- 305p용 복제본만 Script Console에 등록한다.
- 브로커 기본값은 `10.255.37.171:31092`로 둔다.
- 토픽 기본값은 `tps.v305p.jenkins.evt.job-lifecycle`로 둔다.
- 리스너명과 로그 prefix는 305p 전용으로 분리한다.

이 정책을 쓰는 이유는 단순하다. 현재 확인된 305p Redpanda Connect는 외부 Ingress가 없고 `ClusterIP`만 보였기 때문이다. "동일 K8s가 아닐 수 있다"는 요구사항까지 고려하면, 검증 기본값은 Kafka NodePort가 더 안전하다.



## 6. 적용 절차

> 절차는 준비, 등록, 검증, 정리의 네 단계로 진행한다.

### 6-1. 준비

먼저 해야 할 일은 두 가지다:

- 305p용 복제본 Groovy 스크립트를 준비한다.
- 제거용 롤백 스크립트를 같이 준비한다.

복제본은 최소한 아래 항목을 원본과 다르게 둔다:

- 클래스명
- logger명
- 기본 브로커 값
- 로그 prefix

예를 들어 로그 prefix는 `[WEBHOOK-RPK-305P]`처럼 두는 편이 좋다. 그래야 기존 리스너와 로그가 섞여도 구분이 된다.

### 6-2. Jenkins Script Console 접속

Jenkins UI에서 다음 메뉴로 이동한다:

- `Manage Jenkins`
- `Script Console`

운영 권한이 필요한 기능이므로 일반 사용자 권한으로는 접근하면 안 된다.

### 6-3. 리스너 등록 스크립트 실행

등록 스크립트는 아래 흐름을 가져야 한다:

1. `RunListener` 목록 조회
2. 305p용 클래스명 또는 logger명으로 기존 항목 제거
3. 새 리스너 인스턴스 등록
4. 등록 결과를 `println` 또는 logger로 남김

핵심은 "제거 후 등록"이다. 그냥 추가만 하면 중복 전송 위험이 생긴다.

### 6-4. 테스트 빌드 실행

리스너 등록 직후에는 실제 빌드 1건만 실행해서 검증한다. 이때 확인해야 할 것은 세 가지다:

- Jenkins 로그에 305p용 로그 prefix가 찍히는가
- Redpanda 토픽에 STARTED / FINALIZED 이벤트가 들어가는가
- 같은 빌드에 대해 중복 이벤트가 발생하지 않는가

이 단계에서 바로 실운영 파이프라인 전체를 흘리는 것은 금지한다.



## 7. 검증 방법

> 검증은 Jenkins 쪽과 Redpanda 쪽을 동시에 봐야 한다.

검증 포인트는 다음과 같다:

| 검증 항목 | 확인 방법 | 기대 결과 |
|----------|----------|-----------|
| 리스너 등록 성공 | Script Console 출력 또는 Jenkins 로그 | 등록 완료 메시지 확인 |
| STARTED 이벤트 발행 | Jenkins 로그 + 토픽 consume | 빌드 시작 시 1건 발행 |
| FINALIZED 이벤트 발행 | Jenkins 로그 + 토픽 consume | 빌드 종료 시 1건 발행 |
| 중복 발행 여부 | 같은 build key 비교 | STARTED/FINALIZED 각각 1건 |
| 브로커 연결 정상 여부 | 에러 로그 확인 | 연결 실패 없음 |

305p Redpanda 쪽 확인 예시는 다음과 같다:

```bash
kubectl --context tps-v305p -n trb-oss exec redpanda-0 -c redpanda -- \
  rpk topic consume tps.v305p.jenkins.evt.job-lifecycle -n 10
```

필요하면 Admin API 헬스도 같이 점검한다:

```bash
curl http://10.255.37.171:31644/v1/status/ready
```



## 8. 롤백 절차

> 무중단 검증은 적용보다 롤백이 더 중요하다.

롤백은 단순해야 한다. 권장 절차는 다음과 같다:

1. Script Console에서 305p용 리스너를 식별한다.
2. 해당 리스너만 제거한다.
3. 제거 완료 메시지를 출력한다.
4. 테스트 빌드 1건으로 더 이상 이벤트가 안 나가는지 확인한다.

중요한 점은 "전체 RunListener를 건드리지 않는 것"이다. 이름 기반 또는 클래스 기반으로 정확히 305p 리스너만 제거해야 한다.



## 9. 이 방식으로 하지 말아야 할 것

> 아래 항목은 실제 사고로 이어질 가능성이 높다.

다음은 금지한다:

- 원본 리스너와 같은 클래스명으로 등록
- 제거 없이 새 리스너만 추가
- STARTED와 FINALIZED를 둘 다 여러 경로에서 쏘도록 중복 구성
- 테스트 없이 바로 운영 파이프라인 전체 적용
- Script Console 반영만 해 두고 정식 배포가 끝난 것처럼 취급

특히 마지막 항목이 가장 위험하다. Script Console 등록은 "검증 성공"이지 "배포 완료"가 아니다.



## 10. 검증 후 다음 단계

> 검증이 끝나면 반드시 정식 반영으로 넘어가야 한다.

검증이 성공하면 다음 단계는 정해져 있다:

1. 305p 복제본을 `tps_manifest/helm-charts/jenkins`에 배포 자산으로 반영
2. `rpk` 설치 자동화를 Jenkins chart에 포함
3. 필요 시 ArgoCD Jenkins 앱 auto-sync 정책 설계
4. 정식 반영 후 Script Console 등록본 제거 또는 무력화

즉, Script Console 방식은 "운영 전 검증 단계"에서만 머물러야 한다.



## 11. 급할 때 UI에서 바로 적용하는 방법

> 운영 이슈가 급해서 바로 적용해야 할 때는 Jenkins UI의 Script Console을 사용한다. 다만 이 방법은 어디까지나 임시 대응이다.

가장 빠른 적용 경로는 Jenkins UI에서 Script Console로 바로 들어가 Groovy를 실행하는 것이다. 이 방법은 Helm, ArgoCD, Git 반영을 기다리지 않아도 되기 때문에 즉시 효과를 확인할 수 있다.

적용 순서는 다음과 같다:

1. Jenkins UI에 관리자 권한으로 로그인한다.
2. `Manage Jenkins`로 이동한다.
3. `Script Console`을 연다.
4. 305p용 복제본 스크립트를 붙여 넣는다.
5. 기존 305p 리스너 제거 후 신규 등록 로직이 포함되어 있는지 다시 확인한다.
6. `Run`을 실행한다.
7. 테스트 빌드 1건으로 STARTED / FINALIZED 이벤트를 확인한다.

이 방식의 장점은 분명하다:

- Jenkins 재기동이 필요 없다.
- 적용 속도가 가장 빠르다.
- 장애 상황에서 즉시 우회 검증이 가능하다.

하지만 다음 단점도 반드시 같이 기억해야 한다:

- Git에 남지 않는다.
- Jenkins 재기동 시 사라질 수 있다.
- 누가 어떤 스크립트를 실행했는지 관리가 약하다.
- 중복 등록 실수가 나면 이벤트 중복 발행이 생길 수 있다.

그래서 급한 적용 시에도 아래 원칙은 지켜야 한다:

- 적용 전 스크립트를 로컬 파일로 저장해 둔다.
- 적용 직후 Jenkins 로그와 Redpanda 토픽을 함께 확인한다.
- 끝난 뒤에는 반드시 `tps_manifest` 정식 반영 작업을 이어간다.
- 정식 반영이 끝나면 UI에서 올린 임시 리스너는 제거하거나 무력화한다.

즉, UI 적용은 "장애 대응용 빠른 처치"로는 유효하지만 "최종 상태"로 남기면 안 된다.



## 12. 최종 판단 기준

> 아래 질문에 모두 "예"라고 답할 수 있어야 다음 단계로 넘어간다.

다음 네 가지를 만족하면 검증 성공으로 본다:

1. Jenkins 재기동 없이 리스너 등록과 제거가 가능했다.
2. STARTED와 FINALIZED 이벤트가 각각 1회씩만 발행됐다.
3. 305p Redpanda 브로커 연결이 안정적으로 동작했다.
4. 실패 시 롤백 스크립트로 즉시 원복할 수 있었다.

이 기준을 만족하면 그때 GitOps 정식 반영으로 넘어가면 된다.
