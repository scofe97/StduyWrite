# TPS executor 적용 판단 — 현재 구조 분석
---
> Consumer Group per Developer 전략을 TPS executor 모듈에 적용할 수 있는지 판단한다. 실제 코드 증거를 바탕으로 현재 구조를 분석하고, 설계 의도와 구현이 어긋난 지점을 짚은 뒤, 단독 적용이 부적합하다는 결론에 이르는 근거를 정리한다. 08-4에서 제시할 대안의 전제가 된다.
> 작성일: 2026-04-24
> 대상: `executor/engine`, 305P Redpanda, 개발계 TPS 공유 인프라



## 1. executor 모듈의 Kafka 구조 요약

> executor는 Kafka/Redpanda를 통해 두 가지 흐름을 소비한다. 두 흐름 모두 공유 인프라 위에 놓여 있다는 점이 판단의 시작점이다.

### 1-1. 두 개의 Consumer

engine 모듈 안에는 `@KafkaListener` 기반 consumer가 두 개 존재한다.

```text
engine/src/main/java/org/okestro/tps/execution/infrastructure/messaging/
├── BuildLifecycleConsumer.java   — Jenkins 빌드 STARTED/FINALIZED 이벤트 소비
└── JobCommandConsumer.java       — operator가 발행한 RECEIVE/CANCEL 커맨드 소비
```

두 consumer가 각각 어떤 토픽을 바라보며 무엇을 하는지가 판단의 전부다. 부수효과의 성격이 전혀 다르기 때문에 함께 두고 보기 전에 하나씩 해부한다.

### 1-2. BuildLifecycleConsumer

`engine/src/main/java/org/okestro/tps/execution/infrastructure/messaging/BuildLifecycleConsumer.java:28-33`:

```java
@KafkaListener(
        topics = "#{T(org.okestro.tps.messaging.topic.Topics).JENKINS_EVT_JOB_LIFECYCLE.getValue()}"
        , groupId = "${executor.build-event.consumer-group:executor-build-event}"
        , containerFactory = "jsonListenerFactory"
        , autoStartup = "${executor.build-event.listener-auto-startup:true}"
)
public void onBuildEvent(@Payload byte[] payload) { ... }
```

Jenkins Groovy RunListener가 `rpk`로 발행한 JSON 이벤트를 받아, `CompletionUseCase.handleStarted` 또는 `handleCompletion`으로 분기한다. 처리 결과는 TPS DB의 `JOB_EXCN_*` 상태 전이 + operator 측으로의 완료 통보 producing이다. 외형은 "이벤트 소비자"이지만 내부는 **상태 기계를 전이시키는 작업자**다.

눈에 띄는 특징은 `autoStartup` property가 외부화되어 있어, 프로파일 설정만으로 리스너 자체를 꺼둘 수 있다는 점이다. 로컬에서 consumer를 끄는 전략(08-4 케이스 A)의 기반이 된다.

### 1-3. JobCommandConsumer

`engine/src/main/java/org/okestro/tps/execution/infrastructure/messaging/JobCommandConsumer.java:40-48`:

```java
@RetryableTopic(
        attempts = "3"
        , backoff = @Backoff(delay = 1000, multiplier = 2.0)
        , topicSuffixingStrategy = TopicSuffixingStrategy.SUFFIX_WITH_INDEX_VALUE
)
@KafkaListener(
        topics = "#{T(org.okestro.tps.messaging.topic.Topics).OPERATOR_CMD_EXECUTION.getValue()}"
        , groupId = "${executor.operator-cmd.consumer-group:executor-operator-cmd}"
)
public void onCommand(@Payload ExecutionCommandAvro command, ...) { ... }
```

operator가 발행한 `OPERATOR_CMD_EXECUTION` 토픽을 받아, `eventType`에 따라 `DispatchUseCase.receive`(Jenkins 빌드 트리거) 또는 `CancelUseCase.cancel`(Jenkins 빌드 취소)로 분기한다. 재시도 토픽과 DLT가 함께 선언되어 있다.

이 consumer는 **명백한 command 소비자**다. 받으면 실제로 Jenkins에 HTTP 호출을 보내 빌드를 시작시키거나 취소한다. 토픽 한 번에 메시지 한 개가 들어오면 Jenkins 빌드도 한 번만 일어나야 한다.

### 1-4. 공유 인프라

`engine/src/main/resources/application-local.yml:14`:

```yaml
spring:
  datasource:
    hikari:
      jdbc-url: jdbc:log4jdbc:mariadb://10.255.37.247:3306/TPS?...
      username: trbuser
      password: cloud1234
  kafka:
    bootstrap-servers: 10.255.37.171:31092
    properties:
      schema.registry.url: http://10.255.37.171:30081
```

로컬 프로파일은 브로커뿐 아니라 **DB·Schema Registry까지 개발계를 그대로 공유**한다. 즉 consumer group만 분리해도 여전히 같은 TPS DB에 쓰고 있고, 같은 Schema Registry에 붙고 있다. 이 전제가 뒤에서 결정적으로 작용한다.



## 2. 설계 의도와 구현의 불일치

> executor 코드는 env-id 기반 group 분리를 설계 단계에서 명시적으로 도입했다. 하지만 그 메커니즘이 실제 consumer 선언까지 이어지지 않아, 현재는 "의도만 있고 동작하지 않는" 상태에 놓여 있다.

### 2-1. 설계 의도 — executor-defaults.yml

`engine/src/main/resources/executor-defaults.yml:4-18`:

```yaml
executor:
  # 환경 식별자 — Kafka consumer group 에 박히는 접미사이자 환경 간 격리 단위.
  # 305P 는 Redpanda 클러스터가 1개(trb-oss)라 local/compose/k8s 가 같은 토픽을 공유한다.
  # group 이름이 겹치면 다른 환경 메시지를 뺏어가므로, 환경별로 EXECUTOR_ENV_ID 를 반드시 다르게 준다.
  # local 기본값은 개발자 계정(${user.name})을 포함해 팀원끼리도 자동 분리되게 둔다.
  env-id: ${EXECUTOR_ENV_ID:local-${user.name}}

  kafka:
    # Kafka consumer group 공통 prefix. 각 @KafkaListener 는 여기에 suffix 만 붙여 사용한다.
    # (예) executor-local-simbohyeon-build-event / executor-k8s-prod-operator-cmd
    group-prefix: executor-${executor.env-id}
```

주석이 상황을 명확히 적어두었다. "305P는 Redpanda 클러스터가 1개라 local/compose/k8s가 같은 토픽을 공유한다. group 이름이 겹치면 다른 환경 메시지를 뺏어간다"는 진단이 실제 문제의 거의 교과서적인 기술이다. 해결책도 같이 적혀 있다. env-id를 접미사로 박고, 각 리스너는 prefix에 suffix만 붙여 유일 group 이름을 만든다는 방침이다.

### 2-2. 구현이 따라오지 않았다

그러나 앞 절에서 본 두 consumer는 prefix를 **사용하지 않는다**.

```java
// BuildLifecycleConsumer.java:30
groupId = "${executor.build-event.consumer-group:executor-build-event}"

// JobCommandConsumer.java:47
groupId = "${executor.operator-cmd.consumer-group:executor-operator-cmd}"
```

두 경우 모두 `${executor.build-event.consumer-group}` / `${executor.operator-cmd.consumer-group}` 키를 보지만, **application.yml 계열 어디에도 이 키를 정의하지 않았다**. 그 결과 Spring은 `:` 뒤의 기본값을 쓴다. 기본값은 env-id가 빠진 평문 `executor-build-event` / `executor-operator-cmd`다.

결론적으로 로컬에서 executor를 띄우든, compose로 띄우든, k8s-dev Pod이 띄우든 **전부 같은 group**으로 합류한다. 설계 의도는 `executor-local-simbohyeon-build-event`였지만, 실제 결과는 `executor-build-event` 하나뿐이다.

### 2-3. 결과로 발생하는 현상

이 불일치의 실질적 결과는 08-1에서 정의한 "메시지 뺏기"가 완벽히 재현되는 상태다.

- 로컬이 먼저 파티션을 잡으면 개발계 Pod은 그 파티션의 메시지를 못 본다.
- 개발계가 먼저 잡으면 로컬은 자기 이벤트를 못 본다.
- 파티션 3개 중 일부만 뺏어가는 반쪽 상태가 가장 흔하다.

주석에서 명시적으로 경고한 바로 그 현상이 구현 누락으로 재현되고 있는 셈이다.



## 3. 메시지 의미 판정

> 구현 불일치를 바로잡으면 문제가 풀릴까. 답은 "아니오"다. 두 consumer의 메시지 의미 자체가 group 분리만으로 안전해지지 않는다.

### 3-1. OPERATOR_CMD_EXECUTION은 명백한 Work Queue

operator가 executor에게 보내는 RECEIVE/CANCEL 커맨드는 "특정 Job 하나를 Jenkins에서 실행하라/취소하라"는 지시다. 이 지시가 **둘 이상의 executor 인스턴스에 동시 전달**되면 어떤 일이 벌어지는가.

- RECEIVE: 두 executor가 각자 같은 Jenkins 작업을 HTTP로 트리거한다. Jenkins 큐에 같은 빌드가 두 번 들어간다. `queueId`도 서로 다르게 발급된다. 이중 빌드가 실제로 실행되어 외부 자원을 두 번 쓴다.
- CANCEL: 두 executor가 각자 취소 API를 호출한다. 두 번째 호출은 404 또는 idempotent 응답으로 실패하거나, 운이 나쁘면 이미 삭제된 리소스 상태에서 예외를 던진다.

파티션 키가 `jobExcnId`라 같은 Job의 RECEIVE→CANCEL 순서는 브로커가 보장하지만, 그 순서가 **어느 environment**에 도착하는지는 group 구성이 결정한다. 현 상태에서는 운에 맡긴 셈이다.

### 3-2. JENKINS_EVT_JOB_LIFECYCLE은 외형만 이벤트인 Work

Jenkins Groovy RunListener는 `STARTED`와 `FINALIZED`를 토픽에 흘린다. 이름은 이벤트지만 `CompletionUseCase`가 하는 일은 다음과 같다.

- `handleStarted`: DB 상태를 `QUEUED → RUNNING`으로 전이.
- `handleCompletion`: DB 상태를 `RUNNING → SUCCESS/FAIL`로 전이 + operator에게 완료 통보 producing.

한 JobExcn에 대해 이 전이가 **두 번** 시도되면 다음이 발생한다.

- 두 번째 상태 전이는 "이미 terminal"이라는 invariant에 걸려 예외를 던진다. 로그에 `Invalid state transition` 류 에러가 쌓인다.
- 혹은 낙관적 락(`@Version`) 경쟁으로 둘 중 한쪽이 `OptimisticLockException`으로 실패한다.
- 운이 나쁘면 두 번째 핸들러가 먼저 커밋한 결과를 덮어쓰는 race가 발생한다.

또한 operator 측으로의 완료 통보도 두 번 발행된다. 이것이 operator에서 어떻게 처리되는가는 executor 코드만 봐서는 확정할 수 없지만, 중복 통보를 전제로 한 방어 코드가 operator 측에 있다는 보증은 없다.

### 3-3. 요약표

| 토픽 | Producer | Consumer 동작 | 부수효과 | 의미 |
|------|----------|---------------|----------|------|
| `OPERATOR_CMD_EXECUTION` | operator | `DispatchUseCase.receive` (Jenkins 빌드 트리거) 또는 `CancelUseCase.cancel` (Jenkins 빌드 취소) | Jenkins 실 빌드 트리거/취소, DB write, downstream event | **Work Queue / Command** |
| `JENKINS_EVT_JOB_LIFECYCLE` | Jenkins Groovy RunListener | `CompletionUseCase.handleStarted` / `handleCompletion` | DB 상태 전이, operator 측 완료 통보 producing | **외형은 Event, 실질은 Work** |

두 토픽 모두 "한 메시지는 한 환경에서만 처리되어야 한다"는 제약을 갖는다. Group을 분리해 여러 환경이 각자 처리하는 순간 제약이 깨진다.



## 4. 공유 DB·Jenkins가 만드는 추가 리스크

> 설령 메시지 의미 문제가 없다 하더라도, executor가 공유하는 외부 자원이 별도의 간섭 경로를 만든다.

### 4-1. 공유 TPS DB

로컬 프로파일은 `10.255.37.247:3306/TPS` 개발계 DB를 그대로 쓴다. 로컬 consumer가 상태 전이를 시도하면 개발계 Pod이 방금 쓴 레코드를 덮어쓰거나, 낙관적 락 충돌로 실패한다. 이 충돌은 Kafka 계층에서 group을 아무리 잘 분리해도 해결되지 않는다.

### 4-2. 공유 Jenkins

executor가 공유하는 Jenkins 서버는 실제 빌드를 실행한다. 로컬 consumer가 Jenkins를 트리거하는 순간, 개발계 Pod은 모르는 빌드가 Jenkins 큐에 올라가고 실제 워커가 이를 처리한다. 빌드 로그·산출물이 실제로 만들어진다.

"테스트 목적의 로컬 실행"이 외부적으로는 "진짜 빌드 1건"과 구별되지 않는다. 빌드 수, 워커 점유 시간, Jenkins 메트릭이 모두 오염된다.

### 4-3. 공유 Operator

completion 통보는 operator 측 토픽으로 흐른다. 로컬 executor가 만든 통보가 개발계 operator에 도달하면, operator는 자신이 기억하지 못하는 JobExcn에 대한 완료를 처리하려 시도한다. 이때의 동작은 operator의 방어 코드에 달려 있고, 현재 증거로는 "방어가 없다"고 가정하는 것이 안전하다.



## 5. 결론 — 단독 적용 부적합

> Consumer Group per Developer 전략을 executor에 단독 적용하는 것은 권장하지 않는다. 세 가지 이유를 정리한다.

**첫째, 메시지 의미가 work queue다.** 08-2에서 반복해 강조한 원칙대로, work queue 토픽에 group 분리만 적용하면 중복 실행이 일어난다. `OPERATOR_CMD_EXECUTION`은 명백한 command이고, `JENKINS_EVT_JOB_LIFECYCLE`도 실질적으로 work다. 두 토픽 모두 단일 소비가 필요하다.

**둘째, 공유 외부 자원이 group 분리 너머에서 충돌한다.** Kafka 계층을 완벽히 정돈해도 공유 TPS DB, 공유 Jenkins, 공유 Operator가 각각 독립적인 충돌 경로를 만든다. Group 분리는 이 세 경로를 하나도 해결하지 않는다.

**셋째, 설계는 이미 group 분리 메커니즘을 갖고 있지만 실현되어 있지 않다.** `executor-defaults.yml`의 env-id/group-prefix는 올바른 방향이다. 그러나 실제 consumer가 prefix를 사용하도록 연결되지 않아, 지금은 로컬과 개발계가 같은 group에 들어가는 상태다. 이 불일치를 바로잡는 것은 "안전장치" 역할로는 의미가 있지만 — 실수로 개발계에 붙어도 메시지를 뺏지 않게 한다 — **문제를 근본적으로 해결하는 해법이 아니다**.



## 6. 다음 문서 안내

부적합 판정이 끝났으니 다음 질문은 "그럼 무엇을 할 것인가"다. 08-4에서는 executor에 실제 적용 가능한 세 가지 대안을 구체적으로 정리한다.

- 케이스 A: 로컬에서 consumer를 꺼두기. `autoStartup=false` 스위치가 이미 BuildLifecycleConsumer에 있다. 가장 작은 변경이다.
- 케이스 B: 로컬 전용 스택을 만들기. Docker Compose로 Redpanda + MariaDB + Schema Registry를 띄우고 개발계 의존을 완전히 끊는다.
- 케이스 C: Group prefix 불일치를 해소해 "안전장치"로만 쓰기. A/B와 조합해 실수 대비 역할.

각 케이스별로 수정해야 하는 파일과 라인, 검증 방법까지 다음 문서에서 다룬다.
