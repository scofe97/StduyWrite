# Consumer Group per Developer 전략 (Redpanda 기준)
---
> 개발자별로 consumer group id를 분리해 로컬과 개발계가 공유 브로커에서 메시지를 뺏지 않도록 하는 전략을 Redpanda 기준으로 깊게 본다. 08 묶음의 두 번째 문서이며, 앞 문서(08-1)의 해법 B를 상세화한다. 언제 이 전략이 안전하고 언제 덫이 되는지 판단할 수 있는 수준까지 다룬다.
> 작성일: 2026-04-24
> 대상: Redpanda 305P, Spring Kafka, 로컬 개발 환경 설계



## 1. 동작 원리

> Kafka와 Redpanda의 consumer group 모델이 메시지를 분배하는 규칙을 먼저 정리한다. 이 규칙을 이해하지 않으면 group 분리가 왜 특정 상황에서 덫이 되는지 보이지 않는다.

### 1-1. Group이라는 구독 단위

Consumer group은 "같은 group id를 선언한 consumer들의 묶음"이다. 브로커는 group을 하나의 논리 구독자로 간주한다. 파티션이 3개인 토픽에 group 하나가 붙으면, 그 group 안에서 최대 3명의 consumer가 각각 한 파티션씩 맡는다. Group 내부에서는 **한 메시지가 한 명에게만** 간다.

반면 **다른 group** 사이에서는 완전히 독립적이다. group A가 오프셋 100까지 읽었다고 해서 group B가 영향받지 않는다. 같은 메시지를 group A도 받고 group B도 받는다. 이것이 pub/sub 시맨틱의 기반이다.

### 1-2. 시각화

파티션 3개짜리 토픽에 세 group이 참여하는 상황을 그림으로 표현하면 다음과 같다.

```text
Topic: orders.created (partition 0, 1, 2; 메시지 100건)

                group=myapp-dev           ← Alice와 Bob의 공용 개발계 consumer
┌──────────────────────────────────────┐
│ offset 저장 위치: __consumer_offsets │  100건 모두 처리
│  - partition 0 offset = 33           │
│  - partition 1 offset = 34           │
│  - partition 2 offset = 33           │
└──────────────────────────────────────┘

                group=myapp-local-alice  ← Alice 로컬 consumer
┌──────────────────────────────────────┐
│ 독립 offset (위 group과 무관)         │  100건 모두 처리
└──────────────────────────────────────┘

                group=myapp-local-bob    ← Bob 로컬 consumer
┌──────────────────────────────────────┐
│ 독립 offset                           │  100건 모두 처리
└──────────────────────────────────────┘
```

핵심은 **offset 저장소가 group 단위로 나뉜다**는 점이다. Alice의 로컬이 언제 시작하든, 어디까지 읽었든, 개발계 group의 진행에는 영향을 주지 않는다. 셋 모두 100건을 각자 한 번씩 처리한다.

### 1-3. 이 구조가 풀어주는 문제와 못 푸는 문제

이 구조는 08-1의 "메시지 뺏기" 문제를 pub/sub 의미 토픽에 한해 정확히 풀어준다. 개발자별 group을 열면 개발계 consumer는 그대로 일하고, 로컬 consumer도 자기 몫을 받는다. 둘 사이에 경쟁이 없다.

단, 이 구조는 **부수효과**에는 손대지 않는다. 로컬 consumer가 메시지를 받아 외부 API를 호출하면, 개발계 consumer도 같은 메시지를 받아 같은 API를 호출한다. 브로커 관점에서는 정상 동작이지만, 외부 API 관점에서는 이중 호출이다. 이 한계가 다음 절들의 함정 대부분을 낳는다.



## 2. Redpanda가 이 전략에 유리한 이유

> 같은 Kafka API 위에서도 Redpanda는 group 운영 부담이 Apache Kafka보다 낮다. 로컬 개발용으로 group을 늘리는 것이 현실적인 선택이 되는 배경을 정리한다.

### 2-1. Group 생성·삭제 비용이 낮다

Redpanda는 Raft 기반 메타데이터 저장소를 쓰고, group 정보를 Kafka의 `__consumer_offsets` 토픽이 아니라 내부 엔진에서 직접 관리한다. 그 결과 수천 개의 group이 존재하는 상황에서도 메타데이터 조회가 빠르다. 개발자 10명이 각자 앱 3~5개를 띄워 수십 개 group을 만들어도 브로커 부담이 거의 없다.

### 2-2. rpk CLI가 group 관리에 간결하다

Kafka의 `kafka-consumer-groups.sh`는 장황한 인자를 받는다. Redpanda의 `rpk`는 같은 작업을 더 짧게 표현한다. 자주 쓰는 명령을 아래에 정리한다.

```bash
# 현재 등록된 group 목록
rpk group list

# 특정 group의 멤버, lag, partition assignment 확인
rpk group describe myapp-local-alice

# 특정 시점부터 다시 소비하도록 offset 조정
rpk group seek myapp-local-alice \
  --to timestamp@2026-04-24T10:00:00+09:00 \
  --topics orders.created

# 처음부터 다시 소비
rpk group seek myapp-local-alice --to start --topics orders.created

# group 삭제 (작업 종료 시 정리)
rpk group delete myapp-local-alice

# group 만들지 않고 단순 tail 관찰
rpk topic consume orders.created --num 10 --offset end
```

`rpk topic consume --offset end`는 **새 group을 만들지 않고** 특정 토픽을 엿볼 때 유용하다. 디버깅 단계에서 offset을 오염시키지 않아야 할 때 자주 쓴다.

### 2-3. 내장 Schema Registry와 Console

Redpanda는 Confluent 스택 없이도 Schema Registry(`:8081`)와 Console UI를 자체 제공한다. Console에서 group 이름으로 검색·필터가 가능해 "내 group의 lag만 보기"가 쉽다. 팀 개발계에 Console 하나만 띄워두면 모든 개발자가 자기 group 상태를 브라우저에서 관찰할 수 있다.



## 3. Spring Boot 구체 설정

> 실제 프로젝트에 적용할 때의 설정 단위를 예시로 둔다. 핵심은 "group id에 개발자별 식별자를 자동 주입"이다.

### 3-1. 가장 작은 설정

로컬 프로파일 하나만 수정하면 된다.

```yaml
# application-local.yml
spring:
  kafka:
    bootstrap-servers: dev-redpanda:9092
    consumer:
      group-id: myapp-local-${user.name}
      auto-offset-reset: latest
      enable-auto-commit: false
    properties:
      schema.registry.url: http://dev-redpanda:8081
      auto.register.schemas: false
      use.latest.version: true
```

`${user.name}`은 Spring이 자동으로 주입하는 시스템 속성이다. Alice가 띄우면 `myapp-local-alice`, Bob이 띄우면 `myapp-local-bob`이 된다. 환경변수를 쓰고 싶다면 `${DEV_USERNAME:unknown}` 같은 형태로 바꾼다.

### 3-2. 여러 consumer가 있는 앱

`@KafkaListener`를 여러 개 쓰는 앱에서는 각 리스너의 group id를 따로 property로 외부화해 둔다.

```yaml
# application-defaults.yml
myapp:
  env-id: ${APP_ENV_ID:local-${user.name}}
  kafka:
    group-prefix: myapp-${myapp.env-id}

  order-event:
    consumer-group: ${myapp.kafka.group-prefix}-order-event
  payment-cmd:
    consumer-group: ${myapp.kafka.group-prefix}-payment-cmd
```

```java
@KafkaListener(
        topics = "orders.created"
        , groupId = "${myapp.order-event.consumer-group}"
)
public void onOrderCreated(...) { ... }
```

이 패턴의 핵심은 **prefix 단일 변경으로 모든 consumer가 함께 바뀐다**는 점이다. 로컬은 `myapp-local-alice-order-event`, 개발계는 `myapp-dev-order-event`, 운영은 `myapp-prod-order-event`처럼 환경 접두어가 자동으로 붙는다.

### 3-3. Transactional Producer id

Kafka exactly-once를 쓴다면 `transactional.id`도 같이 유일화해야 한다. 같은 id를 두 인스턴스가 동시에 쓰면 fencing 메커니즘이 한쪽을 강제 종료시킨다.

```yaml
spring:
  kafka:
    producer:
      transaction-id-prefix: myapp-local-${user.name}-tx-
```

이 prefix는 group id와는 다른 네임스페이스지만, 동일한 환경-사용자 정보를 재사용하면 운영상 일관성이 높아진다.



## 4. 놓치기 쉬운 함정 6가지

> 전략 자체는 단순하지만, 실제 도입 시 예상 외 지점에서 실패한다. 실무에서 반복적으로 터지는 여섯 가지를 상황과 함께 정리한다.

### 4-1. Offset reset 정책

새로 만든 group은 "어디서부터 읽을까"를 결정해야 한다. `earliest`로 두면 토픽에 쌓인 과거 메시지 전체를 재생한다. 개발계 토픽에 수십만 건이 있으면, 로컬 consumer가 시작하자마자 과거 이벤트 폭탄을 맞고 정상 개발이 불가능해진다.

기본값으로 `latest`를 박는다. 특정 시점 재현이 필요하면 `rpk group seek --to timestamp@...`로 한 번만 수동 조정한다. Spring 설정:

```yaml
spring:
  kafka:
    consumer:
      auto-offset-reset: latest
```

### 4-2. 부수효과 이중 실행

3-1의 설정만으로는 "메시지를 받는다"까지만 해결된다. 받은 메시지를 처리하는 로직이 외부 API 호출, DB 쓰기, 이메일 발송 같은 **멱등하지 않은 행위**를 포함하면, 로컬과 개발계가 각자 실행해 이중 부작용이 생긴다.

완화 전략은 두 가지다.

1. 로컬 프로파일에서 외부 어댑터를 Mock/No-op 구현으로 대체한다. `@Profile("local")` 어댑터를 쓰거나 Spring Bean 재정의를 활용한다.
2. Producer가 메시지 헤더에 대상 환경 힌트(`X-Target-Env: dev`)를 넣고, consumer 진입점에서 현재 환경과 맞지 않으면 스킵한다. Producer 측 코드 변경이 필요해 무게감은 있지만, 공유 브로커 체제에서 가장 깔끔하다.

### 4-3. Schema Registry 공유 오염

Redpanda Schema Registry는 cluster 단위다. 로컬에서 잘못된 Avro 스키마를 등록하면 같은 브로커를 쓰는 개발계 consumer까지 역직렬화 실패로 깨진다. 특히 backward/forward compatibility 검증이 켜져 있으면 한 번의 실수로 스키마 등록이 락되기도 한다.

로컬은 스키마를 "읽기 전용"으로 제한한다.

```yaml
spring:
  kafka:
    properties:
      auto.register.schemas: false
      use.latest.version: true
```

스키마 등록은 CI/CD 파이프라인에서만 허용한다는 규약이 같이 있어야 규약이 유지된다.

### 4-4. DLQ 공유로 인한 알람 피로

Spring Kafka의 `@RetryableTopic`이나 DLT 패턴을 쓰면 실패 메시지가 `*.dlt` 같은 별도 토픽에 적재된다. 이 DLQ 토픽이 모든 환경에서 공유되면, 로컬의 버그성 실패가 개발계 DLQ 알람을 울린다.

해법은 DLQ 토픽명도 env-id로 분리하거나, 로컬 프로파일에서는 DLT publish를 꺼두는 것이다. 후자가 구현은 쉽지만 "로컬에서 실패한 메시지가 어디로 가는가"를 별도로 관리해야 한다.

### 4-5. Dangling group 누적

퇴사, 머신 교체, 프로젝트 종료로 더 이상 쓰이지 않는 group이 브로커에 남는다. Redpanda는 idle group을 자동 삭제하지 않는다(기본값 기준). `rpk group list`가 점점 길어진다.

정리 방법은 두 가지다.

1. 주 1회 청소 스크립트를 운영: `rpk group delete` 수동 호출. 팀 규모가 작을 때 충분하다.
2. Cluster config로 idle 만료 단축:

```bash
rpk cluster config set group_offset_retention_sec 86400  # 1일
```

이 값은 group이 일정 시간 이상 offset을 커밋하지 않으면 자동 만료시킨다. 단, 운영 group에는 영향이 없어야 하므로 값 변경 전 prod 워크로드 패턴을 확인한다.

### 4-6. Transactional id 충돌

3-3에서 prefix를 유일화했지만, prefix가 같은 로컬 인스턴스 두 개를 실수로 띄우는 경우가 있다(노트북과 데스크톱 동시 가동). Kafka의 fencing이 한쪽을 `ProducerFencedException`으로 강제 종료시킨다.

예방은 간단하다. `transaction-id-prefix`에 hostname까지 넣는다.

```yaml
spring:
  kafka:
    producer:
      transaction-id-prefix: myapp-local-${user.name}-${HOSTNAME:unknown}-tx-
```



## 5. 관찰·디버깅 팁

### 5-1. Redpanda Console

Console UI에서 다음 흐름이 가장 자주 쓰인다.

- Consumer Groups 탭 → 검색창에 내 group id 입력 → 내 group만 필터링해 lag·assignment 확인.
- Topics 탭 → 특정 토픽 선택 → Messages 탭에서 파티션·오프셋 지정 조회. Group과 무관하게 원본 메시지를 직접 볼 수 있어 "내 consumer가 뭘 받았어야 하는지" 역추적할 때 쓴다.

### 5-2. rpk로 빠르게 보기

CLI가 손에 익으면 다음이 가장 빠르다.

```bash
rpk group describe myapp-local-alice
# MEMBER ID | PARTITIONS | LAG 출력
```

MEMBER ID에 현재 실행 중인 프로세스의 client id가 찍힌다. "내 local이 실제로 group에 붙었는지"를 확인하는 1차 수단이다.

### 5-3. MDC 활용

Spring 앱 로그에 MDC로 group id를 주입해 두면, ELK/Loki에서 필터가 간단하다.

```java
MDC.put("kafka.group", "${myapp.order-event.consumer-group}");
```

통합 로그 시스템에서 `kafka.group: myapp-local-alice-*`로 필터하면 내 consumer의 흐름만 쉽게 추려낼 수 있다.



## 6. 결정 체크리스트

> 이 전략을 특정 프로젝트에 도입해도 되는지 5분 안에 판단하도록 항목을 추렸다.

다음 질문에 모두 "예"로 답할 수 있어야 안전하다.

- 대상 토픽의 메시지 의미가 pub/sub(도메인 이벤트)인가?
- Consumer가 외부 API를 호출하거나 DB 상태를 전이시키는 부수효과를 갖지 않는가? (또는 Mock/No-op로 안전하게 대체 가능한가?)
- Schema Registry에 로컬이 스키마를 쓰지 않도록 설정할 수 있는가?
- DLQ 토픽이 공유되어도 알람이 난리나지 않는가? (또는 로컬은 DLT 발행을 끌 수 있는가?)
- 퇴사·머신 교체 시 dangling group을 정리할 책임자/스크립트가 있는가?
- Transactional producer를 쓴다면 id가 유일화되는가?

하나라도 "아니오"가 나오면 단독 도입은 위험하다. 구체적으로는 다음 조합이 자주 필요하다.

- pub/sub이지만 외부 부수효과 존재 → **해법 A(로컬 브로커 분리)가 더 안전**.
- pub/sub + 부수효과를 Mock로 분리 가능 → **B 단독으로 충분**.
- Work queue 의미 포함 → **B 단독 절대 금지**. A 또는 하이브리드로 가야 한다.

우리 프로젝트(TPS executor)는 마지막 항목에 해당한다. 상세 증거는 다음 문서(08-3)에서 다룬다.
