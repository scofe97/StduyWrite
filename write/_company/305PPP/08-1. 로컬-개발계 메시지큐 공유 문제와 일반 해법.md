# 로컬-개발계 메시지큐 공유 문제와 일반 해법
---
> 로컬 개발 환경과 개발계가 같은 Redpanda/Kafka 브로커를 공유할 때 발생하는 "메시지 뺏기" 문제를 정의하고, 업계에서 널리 쓰이는 세 가지 해법을 비교한다. 08 묶음 네 문서의 진입점이며, 이후 각 해법의 심화(08-2), TPS executor 적용 판단(08-3), 권장 대안(08-4)으로 이어진다.
> 작성일: 2026-04-24
> 대상: `executor`, `operator`, 305P Redpanda, 팀 내 개발 환경 표준



## 1. 문제 정의

> 로컬에서 실행한 모듈의 메시지가 개발계 Pod로 흘러가 로그 추적이 끊기는 현상과, 반대로 로컬이 개발계 메시지를 가로채는 현상을 동시에 다룬다.

### 1-1. 현상

개발자가 자기 PC에서 `executor` 같은 Kafka consumer 모듈을 띄우는 순간, 그 consumer는 브로커에 "나도 이 토픽을 구독한다"고 등록한다. 그런데 로컬 consumer와 개발계 Pod의 consumer가 **같은 consumer group**으로 참여한다면, Kafka는 두 인스턴스를 한 팀으로 간주해 파티션을 나눠준다. 결과는 두 가지 중 하나다.

1. 로컬이 이긴다: 개발계 Pod에 가야 할 메시지가 내 PC로 와서 처리되고, 개발계 로그에는 아무 흔적도 남지 않는다. 다른 팀원이 개발계 로그만 보며 디버깅할 때 원인 불명 상태가 된다.
2. 개발계가 이긴다: 내 PC에서 프로듀서로 쏜 메시지가 개발계 Pod에서 처리되어, 내 로컬 로그에는 "보낸 기록"만 있고 "처리 결과"가 전혀 없다.

실제로 두 상황은 파티션 할당 순간의 운에 따라 뒤섞인다. **한 토픽의 파티션 일부는 내 로컬이, 나머지는 개발계가 가져가는 반쪽 분배**가 가장 흔하고 가장 디버깅하기 어렵다.

### 1-2. 왜 이 문제가 생기나

Kafka/Redpanda의 consumer group 모델은 "같은 group id를 선언한 consumer끼리는 일을 나눠 한다"는 단순한 규칙을 지킨다. 이 규칙은 운영 환경에서는 수평 확장을 자연스럽게 해주지만, 개발 환경에서는 다음 두 가지 조건이 겹칠 때 함정이 된다.

- 조건 A: 로컬과 개발계가 **동일한 브로커**에 접속한다.
- 조건 B: 두 환경이 **동일한 group id**를 쓴다.

조건 B는 사소해 보여도 기본값이 하드코딩된 Spring Boot 앱에서 자주 발생한다. `@KafkaListener(groupId = "executor-build-event")`처럼 상수로 박혀 있으면 누가 어디에서 띄우든 같은 group이 된다. 환경별 group 분리가 설계된 프로젝트라도, 구현 단계에서 property 주입이 누락되면 실제로는 이 상태로 떨어진다(→ 08-3에서 executor 실제 사례).

### 1-3. 범위 밖

이 문서는 **운영계(prod) 간섭**은 다루지 않는다. 운영계는 일반적으로 네트워크·인증으로 완전히 차단되므로, 같은 문제가 "로컬이 prod를 건드린다"는 형태로는 거의 발생하지 않는다. 이 묶음의 관심사는 **개발자 로컬 ↔ 개발계** 경계다.



## 2. 메시지 의미를 먼저 구분한다

> 해법 선택의 출발점은 "이 메시지가 누구에게 가야 하는가"다. 토픽마다 답이 다르므로 토픽 단위로 분류한다.

### 2-1. Pub/Sub 이벤트

도메인 이벤트 성격이다. 생산자는 "이런 일이 일어났다"고 브로커에 알릴 뿐, 누가 처리하는지는 관심이 없다. 여러 소비자가 각자 다른 목적으로 같은 이벤트를 받는 것이 정상이다. `OrderCreated`, `UserRegistered` 같은 토픽이 대표적이다.

이 경우 로컬 소비자가 같은 이벤트를 **한 벌 더 받는 것**은 설계 의도 안에 있다. 문제는 "메시지를 뺏긴다"가 아니라 "내 소비자가 이벤트를 못 본다"인데, 이는 group을 분리하면 해결된다.

### 2-2. Work Queue / Command

명령을 큐에 넣고 한 명의 일꾼이 꺼내 실행하는 구조다. `SendEmail`, `ProcessPayment`, `TriggerBuild` 같은 토픽이 해당된다. 같은 명령을 두 명이 실행하면 **이메일이 두 번 발송**되거나 **결제가 두 번** 일어난다.

이 경우 group을 분리하는 순간 상황이 더 나빠진다. 로컬·개발계가 각자 자기 group으로 참여하면, 둘 다 "나에게 온 명령"이라고 해석해 각자 실행한다. Kafka 관점에서는 의도대로 동작한 것이지만, 도메인 관점에서는 **중복 실행**이다.

### 2-3. 혼합형 — 이벤트처럼 보이는 command

실무에서 가장 까다로운 부류다. 토픽명은 `*.evt.*`이고 이벤트 문법으로 선언되어 있지만, 실제로 소비자가 **DB 상태를 전이**시키거나 **외부 시스템에 행위를 촉발**하면 의미상 work queue다. TPS executor의 `JENKINS_EVT_JOB_LIFECYCLE`이 이 경우에 가깝다(→ 08-3).

혼합형을 이벤트로 오해하면 group 분리가 해결책처럼 보이지만, 실제로는 DB invariant 위반을 유발한다. **토픽명이 아니라 소비자의 부수효과**로 의미를 판정해야 한다.



## 3. 일반 해법 3종

> 업계에서 주로 채택하는 세 가지 전략을 요약한다. 각 전략의 상세는 2장(메시지 의미) 분류와 연결해서 읽어야 한다.

### 3-1. 해법 A — 로컬 전용 브로커 분리

개발자 PC에 Docker Compose로 Redpanda(또는 Kafka) 단독 브로커를 띄우고, 로컬 앱은 **오직 이 브로커만** 바라보게 한다. 개발계 브로커와는 네트워크 차원에서 완전히 분리된다.

구성 요소는 보통 Redpanda 1-node + Schema Registry + 필요 시 Redpanda Console이다. 로컬 DB까지 함께 띄우면 완전한 "개인 스택"이 된다.

- 장점: 메시지 의미가 pub/sub이든 work queue든 안전하다. 로컬 로그만 보면 되고, 누구의 메시지와도 섞이지 않는다. Schema 실험이나 토픽 생성도 자유롭다.
- 단점: 초기 설정 비용이 있다. 개발계에서만 흐르는 실제 데이터(예: 한국은행 연동 테스트 이벤트)를 로컬에서 재현하려면 데이터 시딩 전략이 필요하다. 다른 환경과 단절된 만큼 "환경 간 호환성 이슈"를 로컬에서 잡기 어렵다.
- 적합한 상황: consumer 로직 자체를 포함한 E2E를 로컬에서 돌려보고 싶을 때. 사이드 이펙트가 강한 모듈(결제·Jenkins 트리거·이메일 발송 등)을 건드릴 때.

### 3-2. 해법 B — Consumer Group per Developer

브로커는 공유하되, 개발자별로 consumer group id를 다르게 둔다. 예를 들어 로컬은 `myapp-local-alice`, 개발계 Pod은 `myapp-dev`처럼 구분한다. 브로커 입장에서는 "서로 다른 구독자"가 되어 같은 메시지를 각자 받는다.

구현은 `spring.kafka.consumer.group-id: myapp-local-${user.name}` 한 줄이면 시작된다. 환경변수(`USER`, `HOSTNAME`) 또는 OS 속성(`${user.name}`)을 활용해 자동 유일화한다.

- 장점: 인프라 추가 없이 설정 한두 줄로 시작할 수 있다. 개발계 실제 이벤트 흐름을 로컬에서 **같이** 관찰할 수 있어 통합 디버깅에 강하다.
- 단점: **pub/sub 의미에서만 안전**하다. work queue 의미 토픽에 적용하면 중복 실행이 발생한다(2-2). 또한 로컬 소비자가 실행하는 외부 부수효과(DB 쓰기, HTTP 호출)는 여전히 개발계와 충돌할 수 있다. 공유 Schema Registry, 공유 DLQ 같은 보조 자원도 같이 고민해야 한다.
- 적합한 상황: 도메인 이벤트 중심 아키텍처에서 개발자가 실제 이벤트 흐름을 관찰하며 새 consumer를 작성할 때. 외부 부수효과가 약하거나 Mock 주입이 쉬울 때.

상세는 [08-2. Consumer Group per Developer 전략 (Redpanda 기준)](08-2.%20Consumer%20Group%20per%20Developer%20전략%20%28Redpanda%20기준%29.md)에서 다룬다.

### 3-3. 해법 C — 토픽 네임스페이스 분리

토픽명 자체에 환경 접두사를 넣는다. `orders.created` 대신 `dev.orders.created`, `local-alice.orders.created` 식으로 구분한다. 프로듀서·소비자 양쪽이 모두 같은 규칙을 따라야 동작한다.

- 장점: 완벽한 격리. 메시지 의미와 무관하게 로컬 메시지가 개발계로 새지 않는다.
- 단점: **프로듀서 측 코드/설정 변경이 필수**다. 도메인 이벤트 버스가 환경을 인지해야 하는 구조가 되어 코드 복잡도가 올라간다. 팀 전체가 이 규칙을 지켜야 효과가 난다.
- 적합한 상황: 프로듀서·컨슈머가 같은 팀 손에 있고, 토픽 라우팅을 코드에서 통제할 수 있는 신규 프로젝트. 운영에서 멀티테넌시(고객별 토픽 분리)를 이미 하고 있어 같은 메커니즘을 재사용할 수 있는 경우.



## 4. 비교표

아래 비교는 네 가지 축으로 요약한다. 축을 고른 이유는 해법 선택 시 실제로 의사결정을 가르는 항목들이기 때문이다.

| 축 | A. 로컬 브로커 분리 | B. Group per Developer | C. 토픽 네임스페이스 |
|----|---------------------|------------------------|-----------------------|
| 도입 비용 | 높음 (docker-compose·시딩) | 낮음 (설정 한두 줄) | 중간 (프로듀서 규약 합의) |
| Pub/Sub 안전성 | 안전 | 안전 | 안전 |
| Work queue 안전성 | 안전 | **위험** (중복 실행) | 안전 |
| 외부 부수효과 격리 | 강함 | 약함 | 중간 |
| 개발계 실데이터 관찰 | 불가 | 가능 | 불가 |
| 스키마 실험 자유도 | 높음 | 낮음 (공유 SR) | 중간 |
| 팀 합의 필요성 | 낮음 | 낮음 | 높음 |

"Work queue 안전성"이 B에서만 빨간불이라는 점이 핵심이다. 로컬-개발계 간섭 문제를 해결하러 group 분리를 도입했는데 오히려 Jenkins 빌드가 두 번 트리거되는 식의 회귀가 B의 전형적인 실패 패턴이다.



## 5. 선택 가이드

> 완벽한 정답은 없고, 팀이 다루는 메시지의 의미와 테스트 목표에 따라 답이 달라진다.

### 5-1. 결정 트리

다음 질문 순서로 판단하면 대부분의 상황이 걸린다.

1. **Work queue 의미의 토픽이 하나라도 있는가?**
   - 예 → 해법 A를 기본으로 고려한다. 해법 B는 단독으로 쓰지 않는다.
   - 아니오 → 다음 질문.
2. **개발계 실데이터 흐름을 로컬에서 관찰해야 하는가?**
   - 예 → 해법 B가 가장 가볍다.
   - 아니오 → 해법 A가 단순하다.
3. **프로듀서 코드를 우리 팀이 통제하는가? 그리고 멀티테넌시 같은 네임스페이스 규약을 이미 쓰고 있는가?**
   - 예 → 해법 C도 후보가 된다.
   - 아니오 → C는 비용이 크므로 보류.

### 5-2. 하이브리드가 현실적

순수 단일 해법으로 끝나는 경우는 드물다. 실무에서 자주 보이는 조합은 다음과 같다.

- A + consumer 일부를 로컬에서 꺼두기: 로컬은 Mock/Stub 프로듀서로만 테스트하고, 실제 consumer 로직은 통합 테스트 단계에서만 가동한다.
- B + 외부 어댑터 Mock: 이벤트는 공유 브로커에서 관찰하되, 외부 호출(HTTP, DB 쓰기)은 로컬 프로파일에서 No-op 구현으로 대체한다.
- B + 일부 토픽만 C: 대부분 토픽은 group 분리로 충분하지만, 이메일/결제처럼 부수효과가 강한 토픽만 `local.*` 네임스페이스로 분리한다.

### 5-3. 안티패턴

다음 상황은 피한다.

- Work queue 토픽에 group 분리만 적용하고 "문제 해결됐다"고 선언하기. 표면상 로그가 보이기 시작해 해결된 것처럼 느껴지지만, 개발계와 로컬이 같은 명령을 **둘 다** 실행하고 있다.
- 네임스페이스 규약을 일부 토픽에만 도입하고 문서화하지 않기. 신규 개발자가 규약을 모르고 기본 토픽명을 쓰면 그 순간 격리가 깨진다.
- 공유 DB를 그대로 두고 브로커만 격리하기. Kafka 경로는 막혔지만 DB 레코드 충돌로 같은 부작용이 다시 올라온다.



## 6. 다음 문서 안내

이 장에서 제시한 세 해법 중 실제로 많이 논의되는 것은 B(Consumer Group per Developer)다. 설정 비용이 가장 낮아 접근이 쉽고, 도입 직후 "뭐가 바뀐지 즉시 보인다"는 체감이 있어 실무자가 우선 시도하는 경향이 있다. 그러나 2-2에서 본 것처럼 work queue 의미 토픽이 섞여 있으면 덫이 된다.

이어지는 세 문서는 다음 순서로 진행한다.

- **08-2**: 해법 B를 Redpanda 기준으로 깊게 본다. 동작 원리, 구체 설정, 함정 6가지, 결정 체크리스트를 다룬다.
- **08-3**: 우리 TPS executor 모듈에 B를 적용할 수 있는지 판단한다. env-id 기반 group 분리 설계는 이미 들어갔는데 왜 실패 상태인지, 그리고 왜 단독 적용이 부적합한지 증거와 함께 정리한다.
- **08-4**: 그래서 실제로 무엇을 할 수 있는가. A/B/C를 executor 맥락에 맞춰 재해석한 세 가지 대안을 제시한다.
