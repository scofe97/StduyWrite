# TPS executor 권장 대안 3가지 (A/B/C)
---
> 08-3에서 Consumer Group per Developer 전략의 단독 적용이 부적합하다는 결론을 내린 뒤, executor에 실제로 적용 가능한 세 가지 대안 A/B/C를 정리한다. 각 대안의 적용 조건, 수정 대상 파일과 라인, 검증 방법을 함께 담아 이후 실행 단계에서 바로 참고할 수 있게 한다.
> 작성일: 2026-04-24
> 대상: `executor/engine`, 305P Redpanda, TPS 개발 환경 표준



## 1. 세 대안의 포지셔닝

> 세 대안은 경쟁 관계가 아니라 **테스트 목표**에 따라 선택되는 스펙트럼이다. 먼저 각 대안이 커버하는 범위와 비용을 짧게 정리한다.

- **케이스 A — 로컬 Consumer 끄기**: 로컬에서는 consumer를 아예 가동하지 않는다. REST/도메인/프로듀서 경로만 테스트한다. 변경은 프로파일 설정 한두 줄.
- **케이스 B — 로컬 전용 스택**: 로컬에 Redpanda + MariaDB + Schema Registry를 띄우고 개발계 의존을 완전히 끊는다. Consumer 포함 E2E를 로컬에서 돌린다.
- **케이스 C — Group Prefix 안전장치**: env-id 기반 group prefix를 실제로 연결해 "실수로 개발계에 붙어도 피해 없게" 만든다. 단독 해법이 아니라 A 또는 B와 조합하는 보조 장치다.

비용·범위를 표로 요약하면 다음과 같다.

| 항목 | A. Consumer 끄기 | B. 로컬 전용 스택 | C. Group Prefix 안전장치 |
|------|------------------|---------------------|--------------------------|
| 초기 도입 비용 | 매우 낮음 (설정 2~3줄) | 높음 (compose 구성·시딩) | 낮음 (설정 파일 한 개) |
| 로컬에서 consumer 로직 테스트 | 불가 | 가능 | 해당 없음 |
| Jenkins 실 빌드 트리거 방지 | 완벽 | 완벽 (로컬 Jenkins 없으면) | 불완전 (group만 분리) |
| DB invariant 보호 | 완벽 | 완벽 | 불완전 |
| 실수 방어력 (단독 기준) | 강함 | 강함 | 약함 (단독 사용 부적절) |
| 권장 조합 | C와 함께 | C와 함께 | A 또는 B의 보조 |

단독 적용 시 안전한 것은 A와 B다. C는 반드시 A 또는 B와 함께 쓴다.



## 2. 케이스 A — 로컬 Consumer 끄기

> 가장 작은 변경으로 메시지 간섭 문제를 제거한다. 로컬 개발자는 REST API·도메인 서비스·프로듀서 경로만 가동하고, Kafka consumer는 자동 시작을 막아둔다.

### 2-1. 적용 조건

이 방법이 맞는 상황은 다음과 같다.

- 이번 작업이 REST 엔드포인트, 도메인 서비스 레이어, DB 쿼리, 아웃바운드 프로듀서(operator 측으로 통보)만 건드릴 때.
- consumer 내부 로직(`CompletionUseCase`, `DispatchUseCase`, `CancelUseCase`)을 직접 돌려봐야 할 이유가 없을 때.
- 개발계 메시지를 관찰하는 대신 unit/slice 테스트로 consumer 동작을 검증할 때.

반대로 다음 상황에서는 A만으로 부족하다.

- Jenkins 빌드 라이프사이클을 E2E로 재현해야 할 때 (로컬 Jenkins가 없고, 개발계 Jenkins를 쓰려면 consumer가 필요하다).
- RetryableTopic 재시도 흐름이나 DLT 동작을 확인해야 할 때.

### 2-2. 변경 내용

`BuildLifecycleConsumer`는 이미 `autoStartup` property가 외부화되어 있다. `JobCommandConsumer`에는 아직 없으므로 한 줄을 추가한다. 그 다음 `application-local.yml`에 두 스위치를 꺼둔다.

```java
// engine/src/main/java/org/okestro/tps/execution/infrastructure/messaging/JobCommandConsumer.java:45-48
@KafkaListener(
        topics = "#{T(org.okestro.tps.messaging.topic.Topics).OPERATOR_CMD_EXECUTION.getValue()}"
        , groupId = "${executor.operator-cmd.consumer-group:executor-operator-cmd}"
        , autoStartup = "${executor.operator-cmd.listener-auto-startup:true}"  // 신규
)
```

```yaml
# engine/src/main/resources/application-local.yml
executor:
  build-event:
    listener-auto-startup: false
  operator-cmd:
    listener-auto-startup: false
```

이 설정이 적용된 로컬 인스턴스는 Kafka 리스너 컨테이너 자체가 `started=false` 상태로 머문다. 브로커에 join 요청을 보내지 않기 때문에 `rpk group describe executor-build-event`의 member 목록에 등장하지 않는다.

### 2-3. 검증 방법

로컬을 띄운 직후 다음 세 가지를 확인한다.

1. 애플리케이션 로그에서 `KafkaMessageListenerContainer`가 `started` 상태가 아닌 것 확인. `autoStartup=false` 로그 라인이 출력되거나, 최소한 partition assignment 로그가 나오지 않는다.
2. 개발계 Redpanda에 `rpk -X brokers=10.255.37.171:31092 group describe executor-build-event`로 조회했을 때 member 수가 로컬 실행 전후로 변화 없음.
3. 실제 Jenkins 이벤트를 생성(개발계 Pod을 통해)했을 때 내 로컬 로그에 아무런 수신 기록이 없음. 반대로 개발계 Pod 로그에는 정상 수신 기록이 있음.

세 조건이 모두 충족되면 로컬이 더 이상 브로커에서 메시지를 가져가지 않는다는 것이 입증된다.

### 2-4. 한계

이 방법은 "로컬이 소비하지 않는다"만 보장한다. 로컬에서 프로듀서로 `OPERATOR_CMD_EXECUTION`에 메시지를 쏘면 **개발계 executor Pod이 그 메시지를 처리**한다. 이 동작이 의도된 흐름인지 아닌지는 작업마다 다르다. 실수로 프로듀서 경로를 테스트하다 개발계 Jenkins를 트리거하는 일이 없도록, 프로듀서 측 코드에도 로컬 프로파일 가드가 필요할 수 있다.



## 3. 케이스 B — 로컬 전용 스택

> 로컬에 필요한 인프라를 전부 띄워 개발계 의존을 완전히 끊는다. Consumer를 포함한 E2E가 가능한 대신, 초기 구성과 데이터 시딩 비용이 든다.

### 3-1. 적용 조건

다음 요구가 하나라도 있으면 B가 맞다.

- Consumer 로직(`CompletionUseCase` 등)을 실제로 돌려 디버깅해야 한다.
- RetryableTopic 재시도와 DLT 흐름을 관찰해야 한다.
- 개발계에 영향을 주지 않고 Jenkins까지 포함한 E2E 시나리오를 자유롭게 돌려보고 싶다.
- Schema Registry에 실험적 스키마를 등록해야 한다.

### 3-2. 구성 요소

로컬 스택에는 다음이 포함된다.

- **Redpanda 1-node**: bootstrap `localhost:9092`, admin `localhost:9644`, proxy `localhost:8082`.
- **Redpanda Schema Registry**: `localhost:8081`.
- **MariaDB**: `localhost:3306`, `TPS_LOCAL` 스키마, 초기 DDL/DML은 `engine/src/main/resources/db/`를 참고해 시딩.
- **선택: Redpanda Console**: `localhost:8079`, group·토픽 관찰용.
- **선택: 로컬 Jenkins**: Jenkins 자체가 로컬 개발까지 커버해야 하는지에 따라 결정. 없으면 `DispatchUseCase`가 외부 HTTP 호출에서 실패하므로 Feign Mock으로 대체하는 전략이 별도로 필요하다.

### 3-3. 변경 내용

먼저 `application-local.yml`을 로컬 스택 기준으로 재작성한다.

```yaml
# engine/src/main/resources/application-local.yml (재작성 후)
spring:
  cloud:
    config:
      enabled: false
  main:
    allow-bean-definition-overriding: true
  datasource:
    hikari:
      driver-class-name: net.sf.log4jdbc.sql.jdbcapi.DriverSpy
      jdbc-url: jdbc:log4jdbc:mariadb://localhost:3306/TPS_LOCAL?createDatabaseIfNotExist=true&allowMultiQueries=true
      username: trbuser
      password: cloud1234
  kafka:
    bootstrap-servers: localhost:9092
    properties:
      schema.registry.url: http://localhost:8081
```

그 다음 executor 모듈 루트에 `docker-compose.local.yml`을 신설한다.

```yaml
# docker-compose.local.yml (executor 모듈 루트)
services:
  redpanda:
    image: redpandadata/redpanda:latest
    command:
      - redpanda
      - start
      - --overprovisioned
      - --smp 1
      - --memory 1G
      - --reserve-memory 0M
      - --node-id 0
      - --check=false
      - --kafka-addr PLAINTEXT://0.0.0.0:9092
      - --advertise-kafka-addr PLAINTEXT://localhost:9092
      - --pandaproxy-addr 0.0.0.0:8082
      - --schema-registry-addr 0.0.0.0:8081
    ports:
      - "9092:9092"
      - "8081:8081"
      - "8082:8082"
      - "9644:9644"

  mariadb:
    image: mariadb:10.11
    environment:
      MARIADB_ROOT_PASSWORD: root
      MARIADB_DATABASE: TPS_LOCAL
      MARIADB_USER: trbuser
      MARIADB_PASSWORD: cloud1234
    ports:
      - "3306:3306"
    volumes:
      - ./engine/src/main/resources/db:/docker-entrypoint-initdb.d:ro

  console:
    image: redpandadata/console:latest
    environment:
      KAFKA_BROKERS: redpanda:9092
      KAFKA_SCHEMAREGISTRY_ENABLED: "true"
      KAFKA_SCHEMAREGISTRY_URLS: http://redpanda:8081
    ports:
      - "8079:8080"
    depends_on:
      - redpanda
```

### 3-4. 검증 방법

1. `docker compose -f docker-compose.local.yml up -d`로 스택을 띄운다.
2. `rpk -X brokers=localhost:9092 cluster info`가 정상 응답하는 것 확인.
3. executor를 `SPRING_PROFILES_ACTIVE=local`로 부팅. `application-local.yml`이 localhost를 가리키는지 부팅 로그에서 확인.
4. `rpk -X brokers=localhost:9092 topic list`에 `tps.v305p.operator.cmd.execution` 등 executor가 구독하는 토픽이 자동 생성되었는지 확인 (없다면 `rpk topic create`로 수동 생성).
5. 테스트용 operator-stub이나 `rpk topic produce`로 `OPERATOR_CMD_EXECUTION`에 메시지 발행 → 로컬 executor 로그에 수신·처리 흔적 확인.
6. 방화벽 또는 `/etc/hosts` 장치로 개발계 Redpanda(`10.255.37.171:31092`)에 접근이 실제로 불가하도록 검증. 설정만 바꾸고 실수로 접근 가능한 상태로 두면 B의 의미가 반감된다.

### 3-5. 운영상 주의

- `TPS_LOCAL` 스키마는 개발계 DB와 버전이 틀어질 수 있으므로 주기적으로 DDL 싱크 정책이 필요하다. `engine/src/main/resources/db/INIT_INSERT.sql` 변경 시 로컬 스택도 rebuild 한다.
- 로컬 Jenkins가 없다면 `DispatchUseCase`가 Feign 호출에서 실패한다. Mock 어댑터를 별도로 두거나, Consumer 일부(JobCommand)만 꺼두는 하이브리드를 검토한다.
- Schema Registry는 로컬이므로 자유롭게 실험 가능하지만, 팀이 합의한 스키마와 다르게 등록하고 그 상태로 개발계 이벤트를 재현하면 "내 로컬에서는 되는데 개발계에서는 깨지는" 혼선이 생긴다. 주기적으로 개발계 스키마와 동기화한다.



## 4. 케이스 C — Group Prefix 안전장치

> env-id 기반 group prefix 메커니즘이 이미 설계되어 있지만, 실제 consumer가 이를 사용하지 않아 사실상 무력한 상태다(08-3 참조). 이 연결을 복구해 "실수로 개발계에 붙어도 메시지를 뺏지 않게" 만든다. 단독으로는 work queue 중복 실행 문제를 풀지 못하므로 반드시 A 또는 B와 함께 쓴다.

### 4-1. 적용 조건

다음 중 하나 이상에 해당하면 C를 추가 적용한다.

- A 또는 B를 적용했지만, 팀원이 실수로 `application-local.yml`을 건드려 개발계 브로커에 붙는 위험을 줄이고 싶다.
- 과거에 로컬이 개발계 메시지를 뺏어간 사고가 있었고 재발 방지 계층이 필요하다.
- 향후 환경(compose, k8s-dev, k8s-prod)이 늘어날 때도 동일한 규칙으로 group이 자동 분리되기를 원한다.

### 4-2. 변경 내용

두 consumer가 명시적으로 `executor.kafka.group-prefix`를 참조하도록 `application.yml` 또는 `executor-defaults.yml`에 기본값을 재정의한다. 가장 적은 위험으로는 `application.yml`에 프로파일 무관 공통 설정으로 배치한다.

```yaml
# engine/src/main/resources/application.yml (추가)
executor:
  build-event:
    consumer-group: ${executor.kafka.group-prefix}-build-event
  operator-cmd:
    consumer-group: ${executor.kafka.group-prefix}-operator-cmd
```

`executor.kafka.group-prefix`는 `executor-defaults.yml`에서 이미 정의되어 있으므로 자동으로 아래처럼 조립된다.

```text
env-id=local-simbohyeon  → group=executor-local-simbohyeon-build-event
env-id=compose-dev        → group=executor-compose-dev-build-event
env-id=k8s-prod           → group=executor-k8s-prod-operator-cmd
```

### 4-3. 검증 방법

1. 부팅 로그에서 consumer group id가 예상 형태(`executor-local-<username>-build-event` 등)로 출력되는지 확인. Spring은 `[KafkaMessageListenerContainer]`나 `AbstractMessageListenerContainer` 로그에 group id를 찍는다.
2. `rpk -X brokers=10.255.37.171:31092 group list`에 새로운 group 이름이 등장하는지 확인.
3. `rpk group describe executor-build-event`(기존 공용 이름)의 member 수가 **증가하지 않는 것** 확인. 내 로컬이 이 group에 합류하지 않아야 의미가 있다.
4. 실수 방어 테스트: 일부러 프로덕션 Redpanda 주소를 로컬 설정에 넣고 띄웠을 때, 내가 개발계 메시지를 뺏지 않는지 확인(group이 분리되어 있어야 한다).

### 4-4. 한계

반복하지만 C 단독으로는 충분하지 않다. Group을 분리해도 다음 문제는 그대로 남는다.

- `OPERATOR_CMD_EXECUTION` 메시지를 로컬과 개발계가 **둘 다** 수신한다. 각자 Jenkins를 트리거해 **빌드가 두 번 실행**된다.
- `JENKINS_EVT_JOB_LIFECYCLE`도 둘 다 수신한다. 공유 DB에서 같은 JobExcn에 대한 상태 전이가 경쟁한다.

이 문제들은 A(consumer 끄기) 또는 B(로컬 스택)만 풀 수 있다. C는 "A/B가 어떤 이유로 깨졌을 때의 2차 방어"라는 역할로 이해한다.



## 5. 결정 매트릭스

> 케이스를 고르는 출발점은 "이번 로컬 개발에서 무엇을 확인하려 하는가"다. 주요 축을 기준으로 권장 조합을 정리한다.

| 로컬 개발 목적 | 권장 조합 | 이유 |
|----------------|-----------|------|
| REST API/도메인 서비스/DB 쿼리만 작업 | A + C | 인프라 비용 없이 consumer 간섭만 차단. C가 실수 방어. |
| 프로듀서 로직 테스트 (operator로 메시지 발행) | A + C + 프로듀서 측 환경 가드 | 프로듀서가 개발계 Jenkins까지 깨우는 일을 별도로 막는다. |
| Consumer 로직 디버깅 (CompletionUseCase 등) | B + C | Consumer를 돌려야 하므로 로컬 스택 필요. |
| RetryableTopic / DLT 흐름 검증 | B + C | DLT까지 로컬에서 재현해야 깊은 검증이 가능. |
| Jenkins 빌드 트리거까지 E2E | B + 로컬 Jenkins 또는 Feign Mock | DispatchUseCase 경로 전체를 격리해야 실제 개발계 Jenkins에 영향이 없다. |
| 이벤트 순서·파티션 키 동작 확인 | B | 로컬 Redpanda에서 파티션 수 조절·seek을 자유롭게. |

결정 매트릭스의 공통 원칙은 **"단독 C는 선택하지 않는다"**이다. C는 항상 A 또는 B에 덧붙이는 보조 계층이다.

### 6-2. 케이스 B만

| 파일 | 라인 | 변경 유형 | 내용 |
|------|------|-----------|------|
| `engine/src/main/resources/application-local.yml` | 전체 | 재작성 | bootstrap-servers / jdbc-url / schema.registry.url을 `localhost` 기반으로 |
| `docker-compose.local.yml` (executor 모듈 루트) | 신규 | 추가 | Redpanda + MariaDB + Console |
| `engine/src/main/resources/db/INIT_INSERT.sql` | 참조만 | 변경 없음 | compose가 볼륨 마운트로 참조 |

### 6-3. 케이스 C만

| 파일 | 라인 | 변경 유형 | 내용 |
|------|------|-----------|------|
| `engine/src/main/resources/application.yml` | 신규 블록 | 추가 | `executor.build-event.consumer-group: ${executor.kafka.group-prefix}-build-event`, `executor.operator-cmd.consumer-group: ${executor.kafka.group-prefix}-operator-cmd` |

### 6-4. A + C (권장 조합 1)

6-1과 6-3의 변경을 모두 적용한다.

### 6-5. B + C (권장 조합 2)

6-2와 6-3의 변경을 모두 적용한다. B만으로도 격리는 완성되지만, C가 있으면 개발자가 실수로 설정을 되돌렸을 때도 방어된다.



## 7. 전체 검증 체크리스트

> 실제 적용 후 회귀를 방지하기 위한 공통 체크리스트다.

- [ ] 부팅 로그에 현재 env-id, group prefix, 각 리스너 group id가 찍혀 있어 육안 확인 가능.
- [ ] `rpk group list`에 예상된 group 이름(환경별 구분)만 존재.
- [ ] 로컬 실행 중 `rpk group describe executor-build-event`의 member 수가 변하지 않음 (C 적용 시).
- [ ] 로컬 DB와 개발계 DB가 분리되어 있는지(`jdbc-url` 확인) — B 적용 시.
- [ ] `application-local.yml`이 공유 인프라 주소를 갖지 않음 — B 적용 시.
- [ ] `autoStartup=false`가 적용된 리스너는 partition assignment 로그가 출력되지 않음 — A 적용 시.
- [ ] 개발계 Jenkins 큐에 로컬 작업 흔적이 남지 않는지 — A 또는 B가 정상 동작하면 보장됨.

체크리스트를 통과하면 08-1이 정의한 "메시지 뺏기" 현상이 실제로 발생하지 않음을 근거와 함께 확인한 셈이다.



## 8. 묶음 문서 맺음말

08 묶음 네 문서는 다음과 같은 흐름으로 이어진다.

- **08-4**: 실제 적용 가능한 A/B/C 대안 — 변경 포인트와 검증까지.

실제 적용에 들어갈 때는 팀 안에서 "로컬 개발의 기본 전략"을 A+C 또는 B+C 중 하나로 합의한 뒤, 공유 설정 파일에 녹여두는 것이 좋다. 개인마다 로컬 구성이 다르면 장기적으로 "내 로컬에서는 되는데"라는 재현 불가 버그의 원인이 된다.
