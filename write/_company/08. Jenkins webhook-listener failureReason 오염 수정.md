# Jenkins webhook-listener failureReason 오염 수정

> **대상 파일**: `~/okestro/tps_manifest/helm-charts/jenkins/files/init-scripts/webhook-listener-305p.groovy`
> **작업일**: 2026-04-23
> **연관 스킬**: `tps/v305p/references/manifest-guide.md` §305P Jenkins webhook-listener 반영 메모

## 1. 현상

305P Jenkins 에서 빌드 결과가 `UNSTABLE` 로 끝나면 Redpanda 토픽 `tps.v305p.jenkins.evt.job-lifecycle` 로 발행되는 lifecycle 메시지의 `failureReason` 에 읽을 수 없는 문자열이 들어갔다.

```
result=UNSTABLE, failureReason=[8mha:////4Pg7/kETXL7zyNkdhDron1Xi7pqNRyIZsy48ojAvVmyuAAAA
...ox+LCAAAAAAAAP9tjTEOwjAQBDdBFLSUPMIRSFSIitZKwwtMYowT6y7YF5KKF/E1/kBEJCq22plmXm8sU8SRo1ONpdZTqrzqQu+mpwaO7TXwoBq+qIopcbCqtEPJtT3NWLJYzMtyLDRWlqrAyZMTrHVjHqYIhlxxlji5g0bu6zueyPRUFhPlC2PE5tbXienX+RvA2HWCfLsXZLsPwAHG870AAAA=
[0m [Pipeline] End of Pipeline
```

downstream 소비자 입장에서는 "왜 실패했는지" 를 전혀 알 수 없는 상태로 메시지가 쌓였다.

## 2. 원인

문자열은 암호화된 값이 아니라 Jenkins workflow-cps 플러그인이 콘솔 로그에 삽입하는 `hudson.console.ConsoleNote` 가 ANSI hidden escape 로 감싸진 채 직렬화된 것이었다. 구조는 이렇다.

```
[8m  ha:////<base64(gzip(serialized Java object))>  [0m
```

- `[8m` ~ `[0m` 은 "터미널에서 숨김" 이스케이프다. 보통 사용자는 이걸 못 본다.
- base64 로 싸인 gzip 페이로드를 풀면 `NewNodeConsoleNote` 같은 Pipeline 스테이지 노드 참조 객체가 나온다.
- Blue Ocean / classic console HTML 렌더러가 이 블록을 파싱해서 "[Pipeline] stage" 같은 링크 UI 로 바꿔 보여준다.

리스너 쪽 `extractFailureReason` 이 `run.getRootDir()/log` 파일을 **raw byte 그대로** 읽고, `readLines().reverse().find { ... }` 로 "마지막 의미 있어 보이는 라인" 을 고르고 있었다. ConsoleNote 를 제거하는 단계가 없었기 때문에 스테이지 노드 블록이 마지막 의미 라인으로 선택되어 `failureReason` 에 박혔다.

추가로 `extractFullLog` 가 `logFile.getText(UTF_8)` 로 로그 파일 전체를 한 번에 힙에 올리고 있었다. 빌드 로그가 수백 MB 에 이르는 Job 에서는 Jenkins controller 힙 압박 원인이 될 수 있는 구조였다. FINALIZED 콜백은 Jenkins controller 스레드에서 실행되므로 여기서 큰 객체를 만드는 건 controller 전체 지연으로 번진다.

## 3. 수정 내용

수정은 "로그를 최대한 적게 읽는다" 방향으로 구조를 뒤집었다. 크게 여섯 가지.

### 3.1 실패 사유 추출 — ErrorAction 우선

Pipeline 빌드는 실패한 FlowNode 에 `org.jenkinsci.plugins.workflow.actions.ErrorAction` 이 붙고, 이 액션은 실제 `Throwable` 객체를 들고 있다. 로그 파싱이 필요 없다.

```groovy
Class errorActionCls = Class.forName('org.jenkinsci.plugins.workflow.actions.ErrorAction')
for (def action : (run?.getActions(errorActionCls) ?: [])) {
    String msg = action?.error?.message?.trim()
    if (msg) return msg
}
```

305P Jenkins 의 실사용 Job 은 대부분 Pipeline 이므로 여기서 거의 끝난다. 로그 I/O 0회, 메모리 수백 byte.

### 3.2 tail 로그 — 32 KB 상한 + ConsoleNote 제거

ErrorAction 으로 못 찾았을 때만 fallback. 전체 로그가 아닌 파일 끝 32 KB 만 `RandomAccessFile` 로 읽는다.

```groovy
private static final int FAILURE_LOG_TAIL_BYTES = 32 * 1024

File f = new File(run?.getRootDir(), "log")
long len = f.length()
long from = Math.max(0L, len - FAILURE_LOG_TAIL_BYTES)
byte[] buf = new byte[(int)(len - from)]
new RandomAccessFile(f, "r").withCloseable { raf ->
    raf.seek(from)
    raf.readFully(buf)
}
byte[] clean = hudson.console.ConsoleNote.removeNotes(buf)
```

`hudson.console.ConsoleNote.removeNotes(byte[])` 는 Jenkins 공식 유틸로, `^[[8m ha:/// ... ^[[0m` 블록을 원천 제거한다. `getLogReader()` 가 내부적으로 쓰는 것과 같은 로직이지만 RandomAccessFile 과 함께 쓰면 "끝만 잘라 노트 제거" 를 한 번에 할 수 있다.

메모리 사용량은 어떤 빌드에서도 32 KB 이하로 고정된다.

### 3.3 tail 필터 일반화

기존 필터는 `[Pipeline] End of Pipeline`, `[Pipeline] }`, `[Pipeline] //` 세 종류만 걸러냈다. 새 Pipeline 단계 라인이 추가될 때마다 필터가 뒤처질 위험이 있어 접두사 매칭으로 묶었다.

```groovy
!trimmed.startsWith("Finished:") &&
!trimmed.startsWith("[Pipeline]") &&
!trimmed.startsWith("[WEBHOOK-RPK-305P]")
```

### 3.4 리스너 자기 오염 제거

기존 `logMessage` 는 `TaskListener#logger.println` 으로 **빌드 콘솔에도** 리스너 로그를 찍고 있었다. 예를 들어 rpk produce 가 실패하면 `[WEBHOOK-RPK-305P] produce-fail attempt=2 ...` 라인이 해당 빌드 콘솔에 들어가, 그 빌드가 FAILURE 일 때 `extractFailureReason` 이 이 라인을 "마지막 의미 있는 라인" 으로 뽑을 가능성이 있었다. 리스너가 자기 로그에 오염되는 루프다.

근본 해결은 리스너 로그를 Jenkins system log 전용으로 고정하는 것이다.

```groovy
private static void logMessage(String message) {
    LOGGER.info(message)
}
```

빌드 콘솔은 더 이상 오염되지 않는다. 운영자는 Jenkins Manage → System Log 에서 리스너 로그를 본다. 방어선으로 tail 필터에도 `[WEBHOOK-RPK-305P]` 접두를 배제해, 수정 전 빌드 로그에 섞여 들어간 라인이 남아 있더라도 `failureReason` 에 잡히지 않게 했다.

### 3.5 rpk 프로세스 타임아웃

기존 `proc.waitFor()` 는 타임아웃이 없었다. rpk 가 브로커에 못 붙어 hang 되면 FINALIZED 콜백 스레드가 무한 대기에 빠진다. Jenkins controller executor 풀이 그만큼 마른다는 의미다.

```groovy
private static final long RPK_TIMEOUT_SECONDS = 5L

boolean finished = proc.waitFor(RPK_TIMEOUT_SECONDS, TimeUnit.SECONDS)
if (!finished) {
    try { proc.destroyForcibly() } catch (Exception ignored) {}
    // produce-timeout 로그 후 재시도 또는 실패
    continue
}
```

retry 2회 × 5 초 = FINALIZED 콜백 최대 지연 10 초로 상한이 고정된다.

### 3.6 부수적 정리

- `extractFullLog` 함수 전면 삭제. `readLogTail` 이 역할을 대체한다.
- `readStream` 에 `withCloseable` 추가해 프로세스 스트림을 명시적으로 닫는다.
- 상수 세 개(`FAILURE_LOG_TAIL_BYTES`, `RPK_TIMEOUT_SECONDS`, `MAX_RETRIES`) 를 클래스 상단에 모아 나중에 env 튜닝 가능한 형태로 정비.

## 4. 수정 후 메모리/지연 특성

| 상황 | 수정 전 | 수정 후 |
|------|---------|---------|
| SUCCESS 빌드 | 로그 파일 전체 힙 로드 | 0 byte — `SUCCESS` 분기에서 즉시 return |
| Pipeline FAILURE (ErrorAction 있음) | 로그 파일 전체 힙 로드 | 0 byte — 메시지 문자열만 |
| Pipeline FAILURE (ErrorAction 없음) / FreeStyle FAILURE | 로그 파일 전체 힙 로드 | 최대 32 KB |
| 로그 파일 없음 | 전체 로드 시도 후 null | 0 byte, null |
| rpk hang | 무한 대기 (controller 스레드 붙들림) | 최대 10 초 후 실패 처리 |

## 5. 재발 방지를 위한 불변규약

다음 규칙을 깨면 이번 문제 중 하나가 재발한다. 스킬 문서 `tps/v305p/references/manifest-guide.md` 에 동일한 내용을 구현 불변규약으로 올려두었다.

1. **실패 사유 추출은 로그를 최소한으로 읽는다** — ErrorAction → tail 32 KB → 기본 문구 순. 전체 로그 `getText()` 금지.
2. **ConsoleNote 는 반드시 제거한다** — `hudson.console.ConsoleNote.removeNotes(byte[])` 를 통과시킨다. 그렇지 않으면 `failureReason` 이 사람이 읽을 수 없는 블록이 된다.
3. **리스너 로그는 빌드 콘솔에 찍지 않는다** — `LOGGER.info` 전용. 빌드 콘솔에 섞으면 자기 오염 루프가 생긴다.
4. **tail 필터는 접두사 기반으로 유지한다** — `Finished:`, `[Pipeline]`, `[WEBHOOK-RPK-305P]` 세 종. 개별 라인 매칭 금지.
5. **rpk 호출에 타임아웃을 건다** — `waitFor(seconds, TimeUnit)` 로 상한. FINALIZED 콜백은 Jenkins controller 스레드라 여기가 막히면 controller 가 막힌다.
6. **클래스명은 고정한다** — `WebhookListener305pConservative`. 파일 하단의 "기존 리스너 제거 후 재등록" 로직이 클래스명 동일성에 의존한다. 바꾸려면 "이전 이름 + 새 이름" 둘 다 제거 대상에 포함해야 한다.

## 6. 검증 방법

1. 개발계 Jenkins Script Console 에 수정된 groovy 를 붙여 재적용하고
   `registered listener=WebhookListener305pConservative removedExisting=1` 이 찍히는지 확인.
2. 의도적 실패 Pipeline Job (`error("intentional failure")`) 실행 → 토픽 `tps.v305p.jenkins.evt.job-lifecycle` 에 발행된 메시지의 `failureReason` 이 `intentional failure` 로 기록되는지 확인. ErrorAction 경로 검증.
3. 테스트 실패 유도 UNSTABLE Job 실행 → `failureReason` 이 `^[[8mha:////` 블록이나 `[Pipeline] ...` 라인이 아닌 실제 경고/에러 라인인지 확인. ConsoleNote 제거 + tail 필터 검증.
4. SUCCESS Job → `failureReason: null`.
5. 대용량 로그 Job (100 MB 이상) 실행 후 `jstat -gc` 또는 모니터링 대시보드로 수정 전 대비 controller heap/GC 특성이 변하지 않는지 확인.
6. rpk 가 붙을 수 없도록 `RPK_BROKERS_305P` 를 잘못된 주소로 바꾸고 테스트 Job 실행 → FINALIZED 콜백이 10 초 이내 반환되는지 system log 로 확인. 타임아웃 경로 검증.

## 7. 후속 과제

이번 범위에서 의도적으로 제외한 항목. 별도 티켓/커밋으로 진행.

- **`FAILURE_LOG_TAIL_BYTES` env 노출** — 대규모 로그에서 32 KB 가 부족하면 `LIFECYCLE_LOG_TAIL_BYTES_305P` env 로 튜닝 가능하게 확장.
- **`BROKERS` fallback 하드코딩 제거** — 현재 env 미설정 시 개발계 주소(`10.255.37.171:31092`) 로 떨어진다. 운영 환경에 env 누락으로 배포되면 잘못된 브로커로 발행되는 리스크가 있다. fallback 을 null 로 두고 `isConfigured` 에서 차단하는 게 안전.
- **`MAX_RETRIES` 네이밍 정정** — 실제 의미는 최대 시도 횟수이므로 `MAX_ATTEMPTS` 가 정확.
- **`queueId` 모호성** — 조회 실패와 "실제 0" 을 구분 못 한다. `queueId: null` 허용 또는 `queueIdAvailable` 플래그 추가.
- **payload 스키마 버전 / correlationId** — consumer 호환성과 분산 추적을 위한 `schemaVersion`, `correlationId` 필드 부재.
- **DLQ / disk spool** — retry 2회 모두 실패 시 이벤트 유실. 파일 기반 DLQ 로 재발행 경로 확보.
