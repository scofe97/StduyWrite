# GitLab executor·message-lib 이관과 Git Mirror 원리

**작성일**: 2026-04-21
**대상**: TPS `executor`, `message-lib` 저장소
**From**: `gitlab.dev.console.trombone.okestro.cloud/CMPART`
**To**: `gitlab.dev.trombone-v2.okestro.cloud/PPP`

## 왜 이 작업을 했나

TPS 의 `executor` 와 `message-lib` 두 모듈이 `CMPART` 그룹(구 GitLab)에 있었는데, 신규 인프라인 `trombone-v2` GitLab 의 `PPP` 그룹으로 이관해야 했다. 단순히 최신 코드만 복사하면 되는 수준이 아니라 **전체 커밋 이력**을 그대로 보존해야 했다. 이력이 잘리면 `git blame` 으로 변경 맥락을 추적할 수 없고, 배포 태그·릴리스 브랜치의 기준점이 무너진다.

동시에 **개인 브랜치 `test-sbh`** 처럼 정리 대상 ref 는 이관 과정에서 잘라내야 했다. 즉 "모두 보존하되 일부만 선택적으로 배제" 가 요구사항이었다.

## 핵심 개념: Git Mirror 란 무엇인가

### 일반 clone vs mirror clone

`git clone <url>` 로 받는 일반 클론은 "개발자가 작업하기 위한 사본" 이다. 기본값으로 **하나의 브랜치(HEAD 가 가리키는 기본 브랜치)** 만 로컬 브랜치로 만들고, 나머지는 `origin/*` 원격 추적 브랜치 형태로만 둔다. 작업 트리(파일 실체)가 존재하고, refspec 도 `refs/remotes/origin/*` 에 국한된다.

`git clone --mirror <url>` 은 전혀 다른 의도의 클론이다. **저장소 자체를 그대로 복제한 bare 저장소** 를 만든다. 구체적으로:

- 작업 트리가 없다(bare). `.git` 디렉토리 내용물이 루트에 바로 풀린 형태다.
- 로컬 ref 가 원격의 ref 와 **1:1 로 매핑**된다. 원격의 `refs/heads/dev` 는 로컬에서도 `refs/heads/dev` 로 잡힌다. `origin/dev` 같은 원격 추적 프리픽스가 붙지 않는다.
- tags, notes, 다른 종류의 ref 까지 전부 포함된다.
- `.git/config` 의 refspec 이 `+refs/*:refs/*` 로 설정된다. 별표 하나로 "모든 ref" 를 의미한다.

`--mirror` 가 `--bare` 와 자주 혼동되지만 둘은 다르다. `--bare` 는 단지 작업 트리 없는 저장소를 만들 뿐, fetch/push 의 동작 범위는 기본 클론과 같다. `--mirror` 는 여기에 더해 "양방향 복제 의도" 까지 명시한다.

### push --mirror 가 강력한 이유

`git push --mirror <url>` 은 로컬의 모든 ref 를 원격에 그대로 밀어넣는다. 원격에만 있고 로컬에 없는 ref 는 **삭제**된다(이 삭제 동작이 `--mirror` 의 본질이다). 요약하면 "로컬 ref 공간을 원격에 덮어쓴다" 가 된다.

이 특성이 저장소 이관에서 결정적이다. 브랜치별로 `push origin dev`, `push origin prd` 를 순차 실행하면 스크립트가 길어지고, 중간에 실패하면 부분 이관 상태로 멈춘다. `--mirror` 한 번이면 dev/prd/stg 와 태그까지 일관된 스냅샷으로 옮긴다.

단, 덮어쓰기이므로 **대상 저장소가 비어있어야 한다**. README 초기화 옵션으로 만들어진 GitLab 프로젝트에 mirror push 하면 빈 저장소의 초기 커밋이 우리 이력과 연결되지 않아 non-fast-forward 로 거절된다. 신규 프로젝트는 "Initialize repository with a README" 를 끄고 비워 두어야 한다.

### 대안은 왜 기각했나

후보 | 기각 이유
--- | ---
`git push --all && git push --tags` | 로컬 ref 기준으로 동작. 기본 클론은 로컬 브랜치가 `dev` 하나뿐이라 `prd`/`stg` 를 별도로 체크아웃하거나 refspec 을 바꿔야 한다. 누락 위험이 크다.
`git bundle` 후 수동 import | GitLab 웹 UI 기반 import 를 요구. HTTP 로 직접 push 가 훨씬 단순하다.
GitLab 의 Repository Mirroring 기능 | Premium 기능. 장기 양방향 동기화용이지 1회 이관용으로는 과하다.
수동 브랜치 체크아웃 + push 반복 | 스크립트 길어지고 실패 시 부분 상태가 남는다. 이관 원자성을 잃는다.

## 이번 이관에서 신경 쓴 포인트

### 1. test-sbh 브랜치 선택 제외

요구사항이 "dev/prd/stg 만 이관, test-sbh 제거" 였으므로, mirror clone 결과물에서 해당 ref 를 **삭제한 뒤** push 했다.

```bash
git clone --mirror http://.../CMPART/executor.git executor-mirror.git
cd executor-mirror.git
git update-ref -d refs/heads/test-sbh   # bare 저장소에서 ref 삭제
git for-each-ref refs/heads/            # dev/prd/stg 만 남았는지 확인
git push --mirror https://.../PPP/executor.git
```

`git update-ref -d` 는 ref 포인터만 지운다. 해당 커밋들은 여전히 object storage 에 남아있지만, push 시점에는 ref 가 없으므로 원격에도 생기지 않는다. mirror 의 "로컬 ref 공간을 원격에 덮어쓴다" 원칙이 그대로 작동하여 `test-sbh` 는 신 저장소에 존재하지 않게 된다.

### 2. 자체 서명 인증서 처리

신 GitLab `trombone-v2` 는 HTTPS 지만 인증서가 자체 서명이라 기본 git 검증을 통과하지 못한다. curl 로 `-k` 없이 요청하면 exit 60(SSL cert verification failed) 이 떴다. 대응은 **해당 저장소 한정** 으로 검증을 끄는 것이다.

```bash
git config http.sslVerify false   # 현재 저장소의 .git/config 에만 쓴다
```

전역(`--global`) 으로 끄면 다른 HTTPS 원격까지 검증을 잃어 위험하다. 로컬 레포 단위로 제한해야 부작용이 없다. 장기적으로는 인증서 체인을 시스템 keychain 에 추가하는 것이 옳지만, 1 회 이관 목적에는 로컬 옵션이 가볍다.

### 3. 워킹트리 더티 상태 보존

`executor` 디렉토리에는 커밋하지 않은 수정 11 개가, `message-lib` 에는 1 개가 있었다. mirror push 는 **원격 저장소** 를 건드리므로 워킹트리와 무관하다. 따라서 bare mirror 클론을 `/tmp` 에 따로 만들어 push 하고, 원래 작업 디렉토리는 `git remote set-url` 로 원격만 교체했다. 교체 후 `git status` 로 확인했을 때 수정 파일들이 그대로 남아있었다.

```bash
cd /Users/simbohyeon/okestro/tps-gitlab2/executor
git remote set-url origin https://.../PPP/executor.git
git config http.sslVerify false
git fetch origin
git branch --set-upstream-to=origin/dev dev
```

`set-url` 은 `.git/config` 의 `[remote "origin"]` 섹션만 갱신한다. 워킹트리, 스테이징, 로컬 브랜치 포인터는 건드리지 않는다.

## 실제 실행 순서 요약

1. `/tmp` 에 `executor-mirror.git`, `message-lib-mirror.git` 을 `--mirror` 로 클론.
2. `executor` 쪽에서 `refs/heads/test-sbh` 삭제.
3. 각 mirror 에서 `http.sslVerify=false` 설정 후 `git push --mirror` 로 신 GitLab 에 전송.
4. `git ls-remote --heads` 로 구·신 저장소 양쪽의 ref 를 비교. **브랜치 tip SHA 가 완전 일치**하는 것을 확인.
5. 작업 클론(`tps-gitlab2/executor`, `tps-gitlab2/message-lib`) 의 origin 주소를 신 GitLab 으로 교체.
6. bare mirror `/tmp/*.git` 을 제거.
7. `tps-gitlab2/README.md` 에 신 URL, 이관 이력 섹션 반영.

### 검증 증거

아래가 이관 직후 구·신 저장소 tip SHA 비교 결과다. 완전 일치한다.

```
executor
  OLD dev 8b2a96c5... | NEW dev 8b2a96c5...  ✓
  OLD prd 7ae22bb2... | NEW prd 7ae22bb2...  ✓
  OLD stg 7ae22bb2... | NEW stg 7ae22bb2...  ✓
  (test-sbh: OLD 에만 존재, 의도적 제외)

message-lib
  OLD dev d3bb35ff... | NEW dev d3bb35ff...  ✓
  OLD prd 5e226bae... | NEW prd 5e226bae...  ✓
  OLD stg 5e226bae... | NEW stg 5e226bae...  ✓
```

SHA 가 같다는 건 트리·부모·메시지·저자·시각까지 모두 동일하다는 뜻이다. Git 의 SHA-1(또는 SHA-256) 은 커밋 내용의 해시이므로, 한 글자라도 바뀌면 다른 해시가 나온다. 따라서 SHA 일치는 **이력 보존의 가장 강한 증거**다.

## 배운 것

**Mirror 는 "ref 공간의 원자적 복제" 추상이다.** 여러 브랜치를 옮긴다는 문제를 "ref 세트를 통째로 옮긴다" 로 재정의하면, 명령 하나로 해결된다. 개별 브랜치 loop 를 짤 때와 비교했을 때 코드 길이뿐 아니라 실패 모드까지 단순해진다. 병렬 push 가 중간에 끊기는 상황이 없어 "반쯤 옮겨진 상태" 가 원천적으로 사라진다.

**`--mirror` 의 삭제 시맨틱은 양날의 검이다.** 원격에만 있고 로컬에 없는 ref 를 지우는 특성 덕분에 `test-sbh` 제외가 쉬웠다. 반대로, 대상 저장소에 이미 다른 누군가가 push 한 브랜치가 있다면 그것도 지운다. 빈 저장소를 대상으로만 쓰는 규칙을 반드시 지켜야 한다.

**자체 서명 SSL 은 글로벌 설정으로 풀지 말 것.** `git config --global http.sslVerify false` 로 검증을 꺼 버리면 이후 모든 HTTPS 원격이 보호를 잃는다. 레포 단위 옵션, 또는 환경 변수(`GIT_SSL_NO_VERIFY=true`) 를 일회성 세션에만 쓰는 쪽이 안전하다.

**원격 교체와 이력 이관은 분리된 관심사다.** push --mirror 로 원격에 이력을 옮겨 둔 다음, 로컬 작업 디렉토리는 `remote set-url` 로 포인터만 바꾼다. 두 단계를 섞지 않으면 더티 워킹트리가 있어도 안전하게 전환할 수 있다.

## 참고 명령어 모음

```bash
# mirror 클론의 refspec 확인
git config --get remote.origin.fetch          # +refs/*:refs/*

# ref 한 개 삭제 (bare 저장소에서도 동작)
git update-ref -d refs/heads/<branch>

# 원격 브랜치·태그 한 번에 비교
git ls-remote --heads <url>
git ls-remote --tags  <url>

# 원격 전체 교체
git remote set-url origin <new-url>

# SSL 검증 끄기 (repo 범위)
git config http.sslVerify false

# 환경 변수로 1 회성 검증 끄기
GIT_SSL_NO_VERIFY=true git ls-remote <url>
```
