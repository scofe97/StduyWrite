# message-lib build.gradle과 Nexus-Avro 동작 원리
---
> `message-lib/build.gradle`이 실제로 무엇을 만들고, 어떤 값이 어디에서 결정되며, 결과물이 어디에 저장되는지를 정리한 문서다. 특히 Nexus 배포와 Avro 코드 생성 흐름을 중심으로 설명한다.

## 문서 목적

이 문서는 `message-lib` 모듈의 `build.gradle`을 읽을 때 가장 먼저 헷갈리는 세 가지를 정리한다.

- Gradle `publish`가 실제로 무엇을 Nexus에 올리는가
- `nexusUrl`, `version`, `credentials` 같은 값이 어떤 우선순위로 결정되는가
- `.avsc` 파일이 어떻게 Java 클래스로 바뀌고, 그 생성물이 어디에 저장되는가

핵심은 "`build.gradle`은 단순 설정 파일이 아니라 빌드 그래프와 배포 경로를 선언하는 실행 규칙"이라는 점이다. 값 하나가 곧바로 JAR 이름, POM 좌표, Nexus 업로드 주소, 생성 코드 경로에 연결된다.

## build.gradle 전체 구조

`message-lib/build.gradle`은 크게 네 층으로 읽으면 된다.

1. 플러그인 계층
2. 좌표와 버전 계층
3. 컴파일 입력 계층
4. 배포 계층

실제 파일 기준 핵심 구조는 다음과 같다.

```groovy
plugins {
    id 'java-library'
    id 'io.spring.dependency-management' version '1.1.7'
    id 'com.github.davidmc24.gradle.plugin.avro' version '1.9.1'
    id 'maven-publish'
}

group = 'org.okestro.tps'
version = findProperty('version') ?: '0.0.1-SNAPSHOT'
```

여기서 이미 두 가지가 결정된다.

- 이 모듈은 Java 라이브러리로 빌드된다.
- Maven 좌표는 기본적으로 `org.okestro.tps:message-lib:0.0.1-SNAPSHOT`가 된다.

즉 다른 모듈이 `implementation "org.okestro.tps:message-lib:0.0.1-SNAPSHOT"`로 가져다 쓰는 근거는 바로 여기서 나온다.

## 1. 플러그인이 각각 맡는 역할

### `java-library`

이 플러그인은 Java 소스 컴파일, JAR 생성, `api`/`implementation` 의존성 분리를 담당한다.

이 플러그인이 들어오면 다음이 가능해진다.

- `components.java`로 publication 구성
- `api`, `implementation`, `compileOnly`, `testImplementation` 같은 configuration 사용
- `jar`, `sourcesJar`, `compileJava`, `test` 같은 표준 태스크 생성

여기서 중요한 점은 `maven-publish`가 `from components.java`를 쓸 수 있는 이유도 결국 `java-library`가 `java component`를 만들기 때문이다.

### `io.spring.dependency-management`

이 플러그인은 Spring BOM을 가져와 버전 정합성을 맞춘다. 예를 들어 `org.springframework.kafka:spring-kafka`처럼 버전을 직접 쓰지 않은 의존성은 BOM에서 버전이 결정된다.

실제 선언은 다음과 같다.

```groovy
dependencyManagement {
    imports {
        mavenBom "org.springframework.boot:spring-boot-dependencies:${springBootVersion}"
    }
}
```

즉 `springBootVersion = 3.2.3`이 바뀌면 연쇄적으로 여러 Spring 계열 라이브러리 해석 결과도 바뀐다.

### `com.github.davidmc24.gradle.plugin.avro`

이 플러그인이 `generateAvroJava` 태스크를 만든다. `src/main/avro/**/*.avsc`를 입력으로 읽어서 Java `SpecificRecord` 클래스를 생성한다.

이 플러그인이 없으면 `.avsc`는 단순 JSON 스키마 파일일 뿐이고, 컴파일 대상 Java 코드로 연결되지 않는다.

### `maven-publish`

이 플러그인은 `publishing { ... }` 블록을 해석해서 publication과 repository를 만든다. `./gradlew publish` 실행 시 실제 업로드를 수행하는 건 이 플러그인이다.



## 2. 값은 어디서 결정되는가

### `group`

```groovy
group = 'org.okestro.tps'
```

이 값은 Maven Group ID가 된다. Nexus에 올라가는 경로에도 영향을 준다.

예를 들어:

- group: `org.okestro.tps`
- artifact: `message-lib`
- version: `0.0.1-SNAPSHOT`

이면 Maven 좌표는 다음이 된다.

```text
org.okestro.tps:message-lib:0.0.1-SNAPSHOT
```

Nexus 내부 저장 경로도 Maven 표준 규칙에 따라 대체로 다음 형태를 따른다.

```text
org/okestro/tps/message-lib/0.0.1-SNAPSHOT/
```

### `version`

```groovy
version = findProperty('version') ?: '0.0.1-SNAPSHOT'
```

해석 순서는 이렇다.

1. Gradle project property에 `version`이 들어오면 그 값을 사용
2. 없으면 기본값 `0.0.1-SNAPSHOT` 사용

즉 아래처럼 실행하면 버전이 바뀐다.

```bash
./gradlew publish -Pversion=0.0.2-SNAPSHOT
```

이 값은 동시에 여러 군데에 영향을 준다.

- 생성되는 JAR 파일명
- 생성되는 POM의 `<version>`
- Nexus 업로드 대상 리포지토리 선택

특히 아래 로직이 중요하다.

```groovy
def repositoryName = version.toString().endsWith('-SNAPSHOT') ?
        'maven-snapshots' :
        'maven-releases'
```

즉 `-SNAPSHOT`이면 snapshot 저장소, 아니면 release 저장소로 간다.

### `nexusUrl`

```groovy
ext {
    set('nexusUrl', project.findProperty('nexusUrl') ?: 'http://nexus.dev.console.trombone.okestro.cloud/repository')
}
```

이 값의 동작 원리는 "기본 DEV Nexus를 쓰되, 외부에서 override 가능"이다.

우선순위는 다음과 같다.

1. `-PnexusUrl=...`로 넘긴 값
2. `gradle.properties`에 들어 있는 `nexusUrl`
3. 기본값 `http://nexus.dev.console.trombone.okestro.cloud/repository`

즉 운영계로 배포하려면 보통 이렇게 바꾼다.

```bash
./gradlew publish -PnexusUrl=http://nexus.prd.console.trombone.okestro.cloud/repository
```

중요한 점은 `nexusUrl` 자체에는 repository 이름이 아직 붙지 않았다는 것이다. 실제 업로드 주소는 뒤에서 `maven-snapshots` 또는 `maven-releases`가 더 붙는다.

### `credentials`

```groovy
credentials {
    username = project.findProperty('nexusUsername') ?: System.getenv('NEXUS_USERNAME')
    password = project.findProperty('nexusPassword') ?: System.getenv('NEXUS_PASSWORD')
}
```

이 값은 다음 우선순위로 해석된다.

1. Gradle property `nexusUsername`, `nexusPassword`
2. 환경 변수 `NEXUS_USERNAME`, `NEXUS_PASSWORD`

즉 저장 위치는 두 가지다.

- 로컬 `gradle.properties`
- 현재 셸의 환경 변수

반대로 말하면, 이 값은 `build.gradle` 안에 하드코딩되어 저장되지 않는다. `build.gradle`은 "어디서 읽어올지"만 정의하고, 실제 비밀값은 외부에 둔다.

## 3. Nexus publish가 실제로 하는 일

### publication 생성

`publishing` 블록의 핵심은 아래다.

```groovy
publishing {
    publications {
        maven(MavenPublication) {
            from components.java
        }
    }
}
```

`from components.java`는 Java 컴포넌트에서 publication에 필요한 산출물을 자동으로 모은다.

이때 포함되는 주요 결과물은 다음과 같다.

- 메인 JAR
- sources JAR
- POM

`withSourcesJar()`가 선언되어 있으므로 sources JAR도 자동 생성된다.

```groovy
java {
    withSourcesJar()
}
```

즉 publish 전에 사실상 아래 산출물이 준비된다.

- `message-lib-0.0.1-SNAPSHOT.jar`
- `message-lib-0.0.1-SNAPSHOT-sources.jar`
- `pom-default.xml` 또는 publication용 POM 메타데이터

### POM에는 무엇이 들어가나

`pom { ... }` 블록은 메타데이터를 채운다.

```groovy
pom {
    name = 'TPS Message Library'
    description = 'Trombone Platform Services - Message Library Module'
    url = 'http://gitlab.prd.console.trombone.okestro.cloud/TRB305P/message-lib'
}
```

이 값들은 Nexus UI나 downstream 소비자가 POM을 볼 때 설명 정보로 쓰인다. 실제 의존성 좌표를 바꾸지는 않지만, 아티팩트의 정체성을 문서화한다.

### 실제 업로드 주소

실제 주소는 아래 조합으로 결정된다.

```groovy
url = "${nexusUrl}/${repositoryName}"
```

예를 들어:

- `nexusUrl = http://nexus.dev.console.trombone.okestro.cloud/repository`
- `repositoryName = maven-snapshots`

이면 최종 배포 주소는 다음이다.

```text
http://nexus.dev.console.trombone.okestro.cloud/repository/maven-snapshots
```

운영계라면 보통 다음처럼 된다.

```text
http://nexus.prd.console.trombone.okestro.cloud/repository/maven-snapshots
```

또는 release 버전이면:

```text
http://nexus.prd.console.trombone.okestro.cloud/repository/maven-releases
```

### `allowInsecureProtocol = true`

```groovy
allowInsecureProtocol = true
```

이 값은 Gradle이 `http://` 저장소 업로드를 막지 않도록 허용하는 스위치다. 기본적으로 최신 Gradle은 보안상 plain HTTP 저장소에 민감하다. 그런데 현재 DEV Nexus가 HTTP이기 때문에 이 옵션이 없으면 publish가 차단될 수 있다.

즉 이 값은 “Nexus가 어디에 저장되는가”를 바꾸는 값이 아니라, “HTTP 주소로 publish를 허용할 것인가”를 결정하는 값이다.



## 4. publish 결과물은 어디에 저장되는가

Gradle publish는 결과물을 세 군데에 남긴다.

### 1. 로컬 빌드 산출물

먼저 로컬 `build/` 아래에 산출물이 생긴다.

대표적으로:

```text
build/libs/message-lib-0.0.1-SNAPSHOT.jar
build/libs/message-lib-0.0.1-SNAPSHOT-sources.jar
build/publications/maven/...
```

즉 publish는 "바로 Nexus에만 보내는" 동작이 아니라, 먼저 로컬 publication 산출물을 만든 뒤 그걸 업로드한다.

### 2. Nexus 저장소

그 다음 Nexus가 Maven repository 규칙으로 저장한다.

대체로 이런 구조다.

```text
<repositoryName>/
  org/
    okestro/
      tps/
        message-lib/
          0.0.1-SNAPSHOT/
            message-lib-0.0.1-SNAPSHOT.jar
            message-lib-0.0.1-SNAPSHOT-sources.jar
            message-lib-0.0.1-SNAPSHOT.pom
            maven-metadata.xml
```

snapshot이면 timestamped artifact와 metadata가 추가로 생길 수 있다. release면 고정 버전 artifact가 저장된다.

### 3. 소비 모듈의 Gradle 캐시

다른 모듈이 이 아티팩트를 가져오면, 실제 JAR와 POM은 각 개발자 머신의 Gradle cache에도 저장된다.

대표 경로는 대략 다음과 같다.

```text
~/.gradle/caches/modules-2/files-2.1/org.okestro.tps/message-lib/0.0.1-SNAPSHOT/...
```

즉 “Nexus에 저장된다”와 “내 로컬에 내려받아 캐시된다”는 별개다. publisher는 Nexus에 올리고, consumer는 Nexus에서 받아 local Gradle cache에 저장한다.



## 5. 다른 모듈이 이 아티팩트를 어떻게 가져다 쓰는가

사용 방식은 일반 Maven/Gradle 의존성과 같다.

```groovy
implementation "org.okestro.tps:message-lib:0.0.1-SNAPSHOT"
```

이 선언이 동작하려면 두 가지가 맞아야 한다.

- 해당 프로젝트의 `repositories { ... }`가 같은 Nexus를 바라봐야 한다.
- Nexus에 그 버전의 `message-lib`가 실제로 올라가 있어야 한다.

즉 `publish`는 producer 쪽 작업이고, `implementation`은 consumer 쪽 작업이다. 둘을 연결하는 공통 키가 `group`, `artifact`, `version` 세 값이다.



## 6. Avro 생성은 어떻게 돌아가는가

### 생성 디렉터리 선언

```groovy
def generatedAvroMainDir = layout.buildDirectory.dir("generated/sources/avro/java/main")
```

이 값은 Avro 생성 Java 코드의 출력 위치를 의미한다.

실제 경로는 다음처럼 된다.

```text
build/generated/sources/avro/java/main
```

즉 생성 코드는 `src/main/java`가 아니라 `build/` 아래에만 만들어진다. 이건 중요한 설계다. 생성물은 소스가 아니라 산출물이기 때문이다.

### 생성물을 컴파일 대상에 포함

```groovy
sourceSets {
    main {
        java {
            srcDir generatedAvroMainDir
        }
    }
}
```

이 설정의 의미는 "`build/generated/...` 아래 생성된 Java 파일도 main source set의 일부로 취급하라"는 것이다.

즉 컴파일러 관점에서는 다음 두 디렉터리가 함께 main source가 된다.

- `src/main/java`
- `build/generated/sources/avro/java/main`

그래서 개발자는 `.java` 파일을 직접 커밋하지 않아도, 빌드 시 자동 생성된 클래스가 정상적으로 컴파일된다.

### `generateAvroJava` 출력 위치 고정

```groovy
tasks.named('generateAvroJava', GenerateAvroJavaTask) {
    outputDir = generatedAvroMainDir.get().asFile
}
```

이 설정은 Avro 플러그인의 기본 출력 경로를 우리가 지정한 디렉터리로 고정한다.

즉 입력과 출력 관계는 다음과 같다.

- 입력: `src/main/avro/**/*.avsc`
- 출력: `build/generated/sources/avro/java/main/**/*.java`

### Avro 코드 생성 옵션

```groovy
avro {
    stringType = 'String'
    fieldVisibility = 'PRIVATE'
    createSetters = true
}
```

각 옵션은 다음 뜻이다.

| 옵션 | 의미 | 결과 |
|---|---|---|
| `stringType = 'String'` | Avro string을 `CharSequence` 대신 `java.lang.String`으로 생성 | 사용처에서 캐스팅 부담 감소 |
| `fieldVisibility = 'PRIVATE'` | 생성 필드를 private으로 둠 | 캡슐화 유지 |
| `createSetters = true` | setter 생성 | builder 외 setter 접근 가능 |

즉 이 옵션들은 스키마 의미를 바꾸는 게 아니라, 생성 Java 코드의 모양을 바꾼다.

## 7. Avro 생성 과정의 실제 예

예를 들어 아래 스키마가 있다고 하자.

```text
src/main/avro/executor/JobResultEventAvro.avsc
namespace = org.okestro.tps.avro.executor
name      = JobResultEventAvro
```

그러면 생성 결과는 대략 다음 경로로 간다.

```text
build/generated/sources/avro/java/main/org/okestro/tps/avro/executor/JobResultEventAvro.java
```

내부 enum이 있으면 별도 Java 타입도 생성된다.

예:

```text
build/generated/sources/avro/java/main/org/okestro/tps/avro/executor/JobResultEventAvroType.java
```

생성된 클래스는 Avro `SpecificRecord` 구현체이므로 다음이 가능하다.

- `newBuilder()`
- `getJobExcnId()`
- `put(int, Object)`, `get(int)`
- schema 기반 바이너리 직렬화

즉 이 클래스는 단순 DTO가 아니라, Avro serializer와 직접 연결되는 스키마-구현체다.

## 8. generateAvroJava는 언제 실행되는가

실제로는 개발자가 매번 `generateAvroJava`만 따로 실행할 필요가 없다. 빌드 그래프상 `compileJava` 전에 필요한 태스크로 연결되기 때문이다.

즉 일반적으로 다음 명령만 쳐도 된다.

```bash
./gradlew build
```

그러면 내부적으로는 다음 흐름이 생긴다.

```text
generateAvroJava
  -> compileJava
  -> jar
  -> publish (실행 시)
```

`.avsc`가 바뀌면 Gradle incremental build가 입력 변경을 감지하고 생성 태스크를 다시 수행한다.

## 9. 생성 Java 코드는 왜 git에 커밋하지 않는가

생성된 Avro Java 코드는 `build/` 아래에만 존재한다. 일반적으로 `build/`는 `.gitignore` 대상이므로 Git에 커밋되지 않는다.

이 원칙이 중요한 이유는 다음과 같다.

- 진실의 원본은 `.avsc`다.
- 생성 Java는 파생 산출물이다.
- 원본과 생성물을 동시에 버전 관리하면 drift가 생기기 쉽다.

즉 저장되는 진실은 다음처럼 나뉜다.

| 종류 | 저장 위치 | 버전 관리 여부 |
|---|---|---|
| Avro 원본 스키마 | `src/main/avro/**/*.avsc` | Git에 커밋 |
| 생성 Java 코드 | `build/generated/sources/avro/java/main` | Git에 커밋하지 않음 |
| 빌드 산출물 JAR | `build/libs` | Git에 커밋하지 않음 |
| 배포 아티팩트 | Nexus | Git 바깥 |

## 10. Avro 접미사 규약이 왜 필요한가

현재 스키마 이름에 `Avro` 접미사를 붙이는 이유는 “기계적으로 구분하기 위해서”다.

예를 들어:

- 도메인 enum: `JobResultEventType`
- 생성 Avro enum: `JobResultEventAvroType`

이렇게 이름을 분리해두면 같은 파일에서 두 타입을 함께 import할 때도 역할이 바로 드러난다. 컴파일만 놓고 보면 접미사가 없어도 되지만, 유지보수 시점에는 도메인 타입과 직렬화 DTO 타입이 섞여 혼선이 생기기 쉽다.

즉 이 규약은 문법 규칙이 아니라 "코드 독해 비용을 줄이기 위한 팀 규약"이다.

## 11. 실제로 기억해야 할 핵심 요약

가장 중요한 연결 고리는 다음 네 줄이다.

```groovy
group = 'org.okestro.tps'
version = findProperty('version') ?: '0.0.1-SNAPSHOT'
url = "${nexusUrl}/${repositoryName}"
from components.java
```

이 네 줄이 사실상 아래를 결정한다.

- 어떤 좌표로
- 어떤 버전으로
- 어느 Nexus 저장소에
- 어떤 산출물을 올릴지

Avro 쪽은 아래 세 줄이 핵심이다.

```groovy
def generatedAvroMainDir = layout.buildDirectory.dir("generated/sources/avro/java/main")
sourceSets.main.java.srcDir generatedAvroMainDir
tasks.named('generateAvroJava', GenerateAvroJavaTask) { ... }
```

이 세 줄이 사실상 아래를 보장한다.

- `.avsc`를 Java로 생성하고
- 그 생성물을 컴파일에 포함하고
- Git에는 커밋하지 않는다

## 결론

`message-lib/build.gradle`은 두 가지 역할을 동시에 한다. 하나는 `Avro 스키마 -> Java 라이브러리`로 바꾸는 빌드 정의이고, 다른 하나는 그 결과물을 `Nexus에 배포 가능한 Maven 아티팩트`로 바꾸는 출판 정의다.

정리하면 다음과 같다.

- `group`, `version`은 라이브러리의 좌표를 만든다.
- `publishing`은 그 좌표의 산출물을 어느 Nexus에 올릴지 결정한다.
- `nexusUrl`, `nexusUsername`, `nexusPassword`는 외부 값으로 주입된다.
- `generateAvroJava`는 `.avsc`를 `build/generated/...` 아래 Java 코드로 만든다.
- 생성 Java와 JAR은 로컬 `build/`에 저장되고, publish 시 Nexus에도 저장된다.
- consumer 모듈은 같은 좌표를 사용해 Nexus에서 받아 local Gradle cache에 저장한다.

이 구조를 이해하면 `message-lib`는 더 이상 “스키마 모듈”이 아니라, “스키마를 중심으로 컴파일·배포·재사용까지 연결하는 배포 가능한 계약 모듈”로 보이게 된다.
