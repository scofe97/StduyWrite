# Redpanda Playground 관측성 검증 가이드

> 작성일: 2026-03-13
> 대상: redpanda-playground 프로젝트 관측 스택(Prometheus, Tempo, Loki, Grafana, Alloy) 검증

---

## 1. 사전 준비

관측 스택을 검증하기 전에 세 가지 프로세스가 순서대로 올라와 있어야 한다. 순서가 중요한 이유는 Spring Boot 백엔드가 Kafka 브로커에 의존하고, 모니터링 스택이 수집 대상(Spring Boot, Redpanda)이 이미 실행 중임을 전제로 scrape 설정을 구성하기 때문이다.

터미널 세 개를 열고 각각 다음 명령어를 실행한다.

```bash
# Terminal 1: 핵심 인프라 (PostgreSQL, Redpanda, Kafka Connect)
make infra

# Terminal 2: 모니터링 스택 (Prometheus, Tempo, Loki, Grafana, Alloy)
make monitoring

# Terminal 3: Spring Boot 백엔드
make backend
```

세 프로세스가 모두 올라오면 다음 URL에서 각 컴포넌트의 상태를 확인한다.

- `http://localhost:8080/actuator/health` — Spring Boot 헬스체크. `{"status":"UP"}` 응답이 와야 정상이다.
- `http://localhost:23000` — Grafana. 익명 접근(anonymous admin)으로 로그인 없이 진입 가능하도록 구성되어 있다.
- `http://localhost:29090/targets` — Prometheus 스크레이프 타깃 목록. `spring-boot`, `redpanda`, `connect-command`, `connect-webhook` 네 개의 타깃이 모두 `UP` 상태여야 한다.
- `http://localhost:24312` — Grafana Alloy 파이프라인 UI. 수집 파이프라인의 컴포넌트별 상태와 처리량을 확인할 수 있다.

Prometheus 타깃 중 하나라도 `DOWN` 상태라면 메트릭 수집이 부분적으로 실패하는 것이므로, 이 단계에서 반드시 확인하고 넘어가야 한다.

## 2. 텔레메트리 생성

관측 데이터는 실제 트래픽이 발생해야 쌓인다. 아무것도 없는 상태에서 Grafana를 열어봐야 빈 그래프만 보이기 때문에, 먼저 API를 호출해서 HTTP 트레이스, DB 스팬, Outbox 이벤트, Kafka 메시지를 발생시킨다.

```bash
# 1. 티켓 생성: HTTP 트레이스 + DB INSERT + Outbox 이벤트 발행
curl -X POST http://localhost:8080/api/tickets \
  -H "Content-Type: application/json" \
  -d '{"title":"OTel Starter 검증","sources":[{"type":"GIT","url":"https://github.com/test/repo"}]}'

# 2. 파이프라인 실행: Outbox → Kafka → Connect 체인 전체 동작 확인
curl -X POST http://localhost:8080/api/tickets/1/pipeline/start

# 3. 단순 조회: HTTP 트레이스 + DB SELECT 스팬
curl http://localhost:8080/api/tickets
```

세 API 호출이 각각 어떤 텔레메트리를 만들어내는지 파악해두면 나중에 Grafana에서 데이터를 찾을 때 기준이 된다.

| API 호출 | HTTP Trace | DB Span | Outbox Event | Kafka Message |
|----------|-----------|---------|-------------|--------------|
| POST /api/tickets | 생성됨 | INSERT 스팬 생성됨 | 생성됨 | 생성됨 (poller 경유) |
| POST .../pipeline/start | 생성됨 | INSERT/UPDATE 스팬 생성됨 | 생성됨 | 생성됨 (poller 경유) |
| GET /api/tickets | 생성됨 | SELECT 스팬 생성됨 | 생성 안 됨 | 생성 안 됨 |

GET 요청은 조회만 하기 때문에 Outbox 이벤트나 Kafka 메시지가 발생하지 않는다. 반면 POST 요청은 DB 변경과 Outbox 폴링 체인을 통해 Kafka까지 데이터가 흐르므로 트레이스 검증 시 POST 요청의 waterfall이 훨씬 풍부하게 나타난다.

## 3. 메트릭 검증 (Prometheus)

Grafana(`http://localhost:23000`)에 접속한 뒤 왼쪽 메뉴에서 Explore를 선택하고, 데이터소스를 Prometheus로 지정한다. 그런 다음 아래 여섯 개 쿼리를 하나씩 실행해서 메트릭이 수집되고 있는지 확인한다.

| 쿼리 | 의미 | 기대 결과 |
|------|------|----------|
| `http_server_requests_seconds_count{uri="/api/tickets"}` | `/api/tickets` 엔드포인트의 HTTP 요청 누적 횟수 | API 호출 횟수만큼 카운터가 증가해 있어야 한다 |
| `histogram_quantile(0.95, rate(http_server_requests_seconds_bucket{uri=~"/api/.*"}[5m]))` | API 엔드포인트 전체의 HTTP P95 응답 시간 | 0이 아닌 값이 반환되어야 한다 |
| `jvm_memory_used_bytes{area="heap"}` | JVM 힙 메모리 사용량 | 수십~수백 MB 범위의 값이 반환되어야 한다 |
| `kafka_producer_record_send_total` | Kafka 프로듀서의 레코드 발행 누적 건수 | 파이프라인 실행 후 0 이상의 값이어야 한다 |
| `hikaricp_connections_active` | HikariCP DB 커넥션 풀에서 활성 상태인 커넥션 수 | 0~10 범위 안에 있으면 정상이다 |
| `up{job="spring-boot"}` | Spring Boot 스크레이프 타깃의 가동 상태 | 1이어야 한다. 0이면 Prometheus가 메트릭을 수집하지 못하는 것이다 |

이 메트릭들이 별도의 코드 작성 없이 수집되는 이유는 `micrometer-registry-prometheus` 의존성이 Spring Boot Actuator와 통합되어 `/actuator/prometheus` 엔드포인트를 자동으로 등록하기 때문이다. Prometheus는 이 엔드포인트를 주기적으로 스크레이프(scrape)해서 시계열 데이터로 저장한다. 개발자가 해야 할 일은 의존성 추가와 `application.yml` 설정뿐이고, JVM 메모리, 스레드, HTTP 요청, HikariCP, Kafka 프로듀서 메트릭은 모두 자동으로 수집된다.

`up{job="spring-boot"}` 쿼리의 결과가 1이 아니라면 Prometheus가 Spring Boot 메트릭 엔드포인트에 도달하지 못하는 것이다. 이 경우에는 7절의 트러블슈팅 항목을 먼저 확인한다.

## 4. 트레이스 검증 (Tempo)

Grafana Explore에서 데이터소스를 Tempo로 전환한다. 트레이스를 찾는 방법은 세 가지가 있다.

- 서비스 이름으로 검색: `service.name = redpanda-playground`
- 스팬 이름으로 검색: `POST /api/tickets`
- TraceQL 직접 입력: `{resource.service.name="redpanda-playground" && name=~"POST.*"}`

트레이스를 찾아서 클릭하면 waterfall 뷰가 열린다. 이 뷰에서 확인해야 할 네 가지 포인트가 있다.

**첫째**, `POST /api/tickets/{id}/pipeline/start` HTTP 진입 스팬이 최상위에 표시되어야 한다. 이 스팬은 OTel Java Agent 또는 Spring Boot OTel Starter가 자동으로 계측한 것이다.

**둘째**, 그 하위에 `SELECT`, `INSERT` 같은 JDBC 스팬이 붙어 있어야 한다. MyBatis가 실행하는 SQL 구문이 개별 스팬으로 잡힌다는 뜻이다. 이 역시 자동 계측 결과다.

**셋째**, `OutboxPoller.publish` 스팬이 존재해야 한다. 이 스팬은 `OutboxPoller.java`에서 직접 작성한 수동 계측(manual instrumentation) 코드로 생성된다. 자동 계측이 잡아주지 못하는 비즈니스 로직 경계를 명시적으로 표시하기 위해 추가한 것이다.

**넷째**, `OutboxPoller.publish` 스팬 하위에 Kafka 프로듀서 `send` 스팬이 연결되어 있어야 한다. Kafka 클라이언트 자동 계측이 context propagation을 통해 스팬을 이어붙이기 때문이다.

가장 중요한 검증 포인트는 `OutboxPoller.publish` 스팬이 HTTP 진입 스팬의 **자식 스팬**으로 표시되는지 여부다. HTTP 요청이 들어올 때 생성된 trace context(`traceparent`)가 Outbox 폴러까지 전파되었다는 증거이기 때문이다. 만약 `OutboxPoller.publish`가 독립적인 루트 스팬으로 나타난다면 context propagation이 제대로 이루어지지 않은 것이다.

### 노이즈 필터링 확인

Outbox 폴링 SQL 스팬과 `GET /actuator/prometheus` 스팬이 트레이스에 나타나지 않아야 한다. 이 두 종류의 스팬은 운영상 의미 없는 노이즈로, Alloy 파이프라인에서 필터링되도록 설정되어 있다. 만약 이 스팬들이 트레이스에 포함되어 있다면 Alloy 필터 설정을 점검해야 한다.

### Agent vs Starter 차이 확인

OTel Java Agent 방식과 Spring Boot OTel Starter 방식은 계측 범위가 조금 다르다. Starter 방식에서는 순수 JDBC 드라이버 레벨 스팬이 줄어들 수 있고, MyBatis 매퍼 호출이 개별 스팬으로 잡히는지도 구현 방식에 따라 달라질 수 있다. Kafka Producer와 Consumer 스팬이 존재하는지도 함께 확인한다.

## 5. 로그 검증 (Loki)

Grafana Explore에서 데이터소스를 Loki로 전환한다. 아래 네 개 쿼리로 로그 수집 상태를 확인한다.

| LogQL | 용도 |
|-------|------|
| `{container=~"playground-.*"}` | `playground-` 접두사를 가진 모든 컨테이너의 전체 로그 |
| `{container=~"playground-.*"} \|= "ERROR"` | 에러 레벨 로그만 필터링 |
| `{container=~"playground-connect.*"}` | Kafka Connect 파이프라인 관련 로그 |
| `{container=~"playground-.*"} \|= "Published outbox event"` | Outbox 이벤트 발행 성공 여부 확인 |

한 가지 제약 사항이 있다. Spring Boot는 호스트 머신에서 직접 실행되기 때문에 Docker 소켓을 통한 로그 수집 대상에 포함되지 않는다. Loki에 로그가 쌓이는 컨테이너는 Kafka Connect, Redpanda, PostgreSQL 등 `docker-compose`로 띄운 컨테이너들뿐이다. Spring Boot 애플리케이션 로그를 Loki로 보내려면 Spring Boot 내부에서 OTLP 로그 익스포터를 직접 구성해야 한다.

Connect 파이프라인 로그(`playground-connect.*`)는 Webhook 전달 성공/실패 여부를 추적할 때 특히 유용하다. `"Published outbox event"` 필터는 Outbox 폴러가 이벤트를 정상적으로 Kafka에 발행했는지 로그 레벨에서 교차 확인하는 용도다.

## 6. 신호 간 연결 검증

세 가지 신호(메트릭, 트레이스, 로그)가 개별적으로 동작하는 것에서 한 단계 더 나아가, 신호 간 이동이 가능한지 검증한다. 이것이 Grafana의 Correlated Observability 기능이다.

### Tempo → Loki 이동

Tempo waterfall 뷰에서 특정 스팬을 선택하면 스팬 상세 정보 옆에 "Logs for this span" 버튼이나 로그 아이콘이 나타난다. 이 버튼을 클릭하면 해당 스팬의 시간대를 기준으로 Loki 쿼리가 자동으로 구성되어 Loki Explore 화면으로 이동한다.

### Loki → Tempo 이동

Loki 로그 중 Kafka Connect가 남긴 로그에 `traceId` 필드가 포함된 항목이 있을 수 있다. 해당 줄에서 traceId 값이 링크로 표시되면 클릭했을 때 Tempo waterfall 뷰로 바로 이동한다.

이 연결이 동작하지 않으면 `datasources.yaml`의 `derivedFields` 설정에서 traceId를 추출하는 정규식과 Tempo URL 설정을 확인해야 한다. `derivedFields` 설정이 없으면 Loki 로그에서 Tempo로의 이동 링크가 표시되지 않는다.

연결이 정상적으로 동작하면 "로그에서 에러 발생 확인 → 해당 요청의 트레이스 찾기 → DB 어느 쿼리에서 지연 발생했는지 특정"과 같은 흐름으로 장애 원인을 추적할 수 있다. 이것이 세 신호를 연결하는 이유다.

## 7. 트러블슈팅

Grafana를 열었을 때 데이터가 보이지 않는 상황은 네 개 계층 중 하나 이상에서 문제가 발생한 것이다. 계층 순서대로 확인하면 원인을 빠르게 좁힐 수 있다.

| 계층 | 확인 항목 | 확인 방법 |
|------|----------|----------|
| 앱 (App) | Spring Boot가 정상 가동 중인가? OTel Starter가 로드되었는가? | `http://localhost:8080/actuator/health` 응답 확인 + 앱 시작 로그에서 "OpenTelemetry" 키워드 검색 |
| 수집 (Collection) | Alloy 파이프라인이 정상 동작 중인가? | `http://localhost:24312` UI에서 컴포넌트별 에러 여부 확인 |
| 저장 (Storage) | Tempo, Loki, Prometheus 컨테이너가 올라와 있는가? | `docker ps` + 각 컨테이너 헬스체크 |
| UI | Grafana 데이터소스 연결이 정상인가? | Grafana → Settings → Data Sources → 각 데이터소스 → Test 버튼 클릭 |

### Tempo에 트레이스가 없을 때

가장 흔한 원인 네 가지를 순서대로 확인한다.

첫째, `application.yml`에 OTel 설정이 있는지 확인한다. `otel.exporter.otlp.endpoint` 또는 `OTEL_EXPORTER_OTLP_ENDPOINT` 환경변수가 설정되어 있어야 한다.

둘째, OTLP 엔드포인트가 Alloy의 OTLP 수신 포트인 `24318`을 가리키는지 확인한다. Alloy가 트레이스를 받아서 Tempo에 전달하는 구조이므로, Spring Boot가 Tempo에 직접 데이터를 보내도록 설정되어 있으면 Alloy 필터링을 거치지 않는다.

셋째, Alloy 로그에서 OTLP 데이터 수신 여부를 확인한다. Alloy가 데이터를 받고 있다면 로그에 수신 기록이 남는다.

넷째, Spring Boot OTel Starter가 실제로 로드되었는지 확인한다. 앱 시작 로그에서 OTel 관련 Bean 등록 메시지가 있는지 살펴본다. 의존성은 추가했지만 auto-configuration이 비활성화되어 있거나 조건부 설정이 맞지 않아 Starter가 동작하지 않는 경우가 있다.

### Prometheus 타깃이 DOWN일 때

세 가지를 확인한다.

첫째, `docker-compose.monitoring.yml`에서 `host.docker.internal` 설정을 확인한다. Docker 컨테이너 안에서 실행 중인 Prometheus가 호스트의 Spring Boot 메트릭 엔드포인트에 접근하려면 `host.docker.internal` 호스트명이 올바르게 설정되어 있어야 한다. macOS에서는 Docker Desktop이 이 호스트명을 자동으로 제공하지만 Linux에서는 별도 설정이 필요하다.

둘째, Spring Boot의 `management.endpoints.web.exposure.include` 설정에 `prometheus`가 포함되어 있는지 확인한다. 이 설정이 없으면 `/actuator/prometheus` 엔드포인트 자체가 노출되지 않는다.

셋째, 직접 curl로 메트릭 엔드포인트를 호출해서 응답이 오는지 확인한다.

```bash
curl http://localhost:8080/actuator/prometheus
```

응답이 오면 Spring Boot 측 설정은 정상이고, Prometheus → Spring Boot 네트워크 경로 문제일 가능성이 높다. 응답이 오지 않으면 Spring Boot 설정을 먼저 수정해야 한다.

이 가이드에서 다룬 검증 절차를 모두 통과하면 세 가지 신호(메트릭, 트레이스, 로그)가 수집되고 있으며 신호 간 연결도 동작한다는 것을 확인한 셈이다. 이후 새로운 기능을 추가하거나 계측 범위를 확장할 때 이 검증 절차를 기준으로 삼으면 된다.
