# 유비쿼터스 언어와 도메인 모델
---
> 유비쿼터스 언어(Ubiquitous Language)는 문서용 용어집이 아니라 코드, 테스트, API, 회의에서 같은 단어를 같은 의미로 쓰기 위한 계약이다.
>
> - Run
> - Battle
> - Reward
> - Exhaust

## 1. 왜 용어를 먼저 맞춰야 하는가

도메인이 복잡해질수록 같은 단어를 다르게 쓰는 순간 설계가 흔들린다. 카드게임에서는 “턴 종료”, “전투 종료”, “런 종료”가 모두 다른 사건인데, 이를 모두 `complete`라고 부르면 경계가 흐려진다.

유비쿼터스 언어의 목표는 기획 문서와 코드 사이의 번역 비용을 줄이는 것이다. 개발자와 도메인 전문가가 같은 단어를 반복해서 쓰면 모델이 자연스럽게 정제된다.

## 2. 예시 프로젝트의 핵심 용어

예시 프로젝트에서는 다음 용어를 기본 어휘로 고정한다:

| 용어 | 의미 | 코드 표현 | 주의점 |
|------|------|----------|--------|
| Run | 플레이어의 단일 세션 | `Run` aggregate | 계정 전체 진행도와 혼동하지 않음 |
| Battle | 전투 한 판 | `Battle` entity | 런 전체와 같은 범위로 취급하지 않음 |
| Reward | 전투 후 선택 보상 | `RewardOption` | 상점 구매와 구분 |
| Exhaust | 카드를 영구적으로 턴 덱에서 제거 | `ExhaustPile` 또는 이벤트 | 단순 discard와 다름 |

이 표는 단순 용어 사전이 아니라 코드 구조의 기준점이다. API 이름, 이벤트 이름, 클래스 이름도 이 언어를 따라야 한다.

## 3. 도메인 모델은 용어의 관계를 드러낸다

유비쿼터스 언어가 단어를 정한다면, 도메인 모델은 그 단어들의 관계를 정한다. 예를 들어 `Run`이 `Battle`을 포함하는지, 아니면 `Battle`이 별도 Aggregate인지에 따라 트랜잭션 경계가 달라진다.

이 시리즈에서는 다음처럼 읽는다:

- `Run`은 세션 전체의 진행 상태를 관리한다.
- `Battle`은 현재 전투 규칙과 턴 흐름을 관리한다.
- `Reward`는 전투 완료 후 발생하는 선택 결과다.

## 4. 잘못된 언어가 만드는 문제

유비쿼터스 언어가 없으면 코드베이스에 같은 의미의 다른 이름이 동시에 생긴다. `match`, `session`, `run`, `game`이 섞이면 어떤 객체가 무엇을 대표하는지 파악하기 어려워진다.

다음 세 가지는 특히 피해야 한다:

- UI 용어를 그대로 도메인 용어로 가져오는 것
- DB 테이블명을 도메인 이름으로 착각하는 것
- 팀마다 다른 별칭을 허용하는 것

## 5. Spring Boot 프로젝트에서 언어를 고정하는 방법

유비쿼터스 언어는 말로만 합의하면 오래가지 않는다. 코드와 테스트 구조로 고정해야 한다.

실무에서는 다음 방식이 효과적이다:

- 패키지명과 클래스명을 도메인 용어에 맞춘다.
- API 요청/응답 DTO도 같은 어휘를 사용한다.
- 테스트 시나리오 이름을 도메인 이벤트 문장으로 작성한다.

다음 패키지 예시는 용어를 구조에 반영한 형태다:

```text
com.example.spire.run
com.example.spire.battle
com.example.spire.reward
com.example.spire.catalog.card
```

## 6. Bounded Context의 초안

같은 단어라도 context가 달라지면 의미가 달라질 수 있다. 예를 들어 `Reward`는 런 도메인에서는 전투 후 선택지지만, 메타 진행 도메인에서는 해금 보상일 수 있다.

그래서 이 시리즈는 다음 context 초안을 사용한다:

- `run`: 세션 시작, 종료, 맵 진행
- `battle`: 턴, 카드 사용, 적 행동
- `catalog`: 카드, 유물, 적 정의
- `progression`: 계정 해금과 메타 진행

## 7. 런 관리 서비스에 적용한 결론

이 시리즈의 모든 문서는 같은 용어를 유지한다. 특히 `Run`, `Battle`, `Reward`, `ActProgress`, `CardPlayed` 같은 이름은 문서와 코드 예시 전반에 반복해서 사용한다.

아키텍처 문서를 여러 편으로 나눌수록 용어 일관성이 더 중요해진다. 유비쿼터스 언어는 DDD 문서에서만 필요한 개념이 아니라, 전체 시리즈를 엮는 접착제다.
