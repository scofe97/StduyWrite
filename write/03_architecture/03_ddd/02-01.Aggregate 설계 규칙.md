# Aggregate 설계 규칙
---
> Aggregate 는 객체 묶음이 아니라 "한 번의 트랜잭션 안에서 어떤 불변식을 함께 지킬 것인가" 의 결정이다. 이 결정이 어긋나면 동시성 비용이 폭발하거나, 반대로 일관성이 새는 구멍이 생긴다.

`01-02 §3` 에서 "런 관리 예시에서 Run 을 상위 Aggregate 로 보되, Battle 을 하위 또는 별도 Aggregate 로 둘 수 있다" 는 결정 사례를 다뤘다. 본 문서는 그 결정을 가능하게 만드는 규칙 자체만 다룬다.

## 1. Aggregate Root 가 책임지는 것

> Root 는 진입점·불변식·변경 묶음의 세 책임을 한 객체로 통합한다.

Aggregate 는 Entity 와 Value Object 의 그래프지만 외부로 노출되는 진입점은 단 하나 — Aggregate Root 다. 외부 코드가 내부 Entity 에 직접 손을 뻗으면 Root 의 불변식 검증 지점이 사라진다. Root 가 책임지는 것은 다음 세 가지다.

1. 유일한 진입점 — 외부 코드는 Root 의 메서드만 호출하며, 내부 Entity 의 참조를 외부로 흘려 보내지 않는다.
2. 불변식 강제 — "주문 라인 합계 ≤ 신용 한도" 같은 비즈니스 규칙을 Root 메서드 안에서 검증한다.
3. 변경의 묶음 — 내부 Entity 가 바뀌었다면, 그것은 Root 가 그 변경을 받아들였다는 뜻이다.

여기서 질문 하나 — 그렇다면 캡슐화는 문법적 `private` 의 문제일까? 그렇지 않다. 캡슐화는 호출 경로의 정책이다. 패키지 가시성으로 노출된 Entity 라도, 호출 흐름이 항상 Root 를 통과한다면 캡슐화는 살아 있다.

런 관리 예시로 옮기면 `Run` 이 Root 일 때 외부는 `run.completeBattle(battleId, result)` 만 호출한다. `Battle` 인스턴스를 직접 들고 와 `battle.applyDamage(...)` 를 부르면, `Run` 의 "전투 횟수 ≤ 3" 같은 불변식을 검증할 자리가 사라진다.

## 2. 트랜잭션 경계와 일관성

> 하나의 트랜잭션은 하나의 Aggregate — 이 휴리스틱이 강한 일관성과 결과적 일관성의 분기점이다.

DDD 가 제시하는 가장 강한 휴리스틱은 "하나의 트랜잭션은 하나의 Aggregate" 다. 이 규칙은 두 가지를 동시에 말한다. Aggregate 내부는 강한 일관성으로, Aggregate 사이는 결과적 일관성으로 다룬다는 뜻이다.

내부 강한 일관성의 예시는 다음과 같다. `Run.completeBattle()` 안에서 `battle.markFinished()` 와 `run.advanceNode()` 가 함께 일어난다면 둘은 같은 트랜잭션에서 커밋되거나 같이 롤백된다. 부분 갱신은 허용하지 않는다.

Aggregate 사이는 다르다. `Run` 이 끝났다는 사실로 `Progression.unlock(...)` 을 해야 한다면, 그건 두 번째 트랜잭션이다. Domain Event + Outbox + 비동기 핸들러로 처리한다. 구현 패턴은 `../04_edd/03-01.Saga, Outbox, Request-Response Bridge.md` 와 `../../04_messaging/05_ConsistencyPattern/01-03.Outbox.md` 가 다룬다.

이 휴리스틱이 절대 법이 아닌 이유는 분명하다. 한 트랜잭션에 두 Aggregate 를 묶어야 풀리는 사례도 있다. 그렇게 결정했다면 왜 휴리스틱을 깼는지를 코드 주석이 아니라 ADR(`../01-05.ADR과 Spring Boot 아키텍처 의사결정.md`) 에 남긴다. 다음 사람이 다시 풀려고 시간 쓰지 않게.

### 2-1. 동시성과 잠금

같은 Aggregate 인스턴스에 두 요청이 동시에 들어오면 일관성이 깨질 수 있다. 흔한 대응은 두 가지다.

| 방식 | 동작 | 적합 도메인 |
|------|------|-------------|
| 낙관적 잠금 | Root 에 `@Version` 컬럼을 두고 커밋 시점에 버전 충돌이면 재시도 | 충돌 빈도가 낮은 일반 도메인 |
| 비관적 잠금 | `SELECT ... FOR UPDATE` 로 행 잠금 | 결제·재고처럼 충돌이 잦거나 재시도 비용이 큰 도메인 |

Aggregate 가 크면 클수록 잠금 범위도 함께 커진다. 이 부담이 `§4` 의 비용 논의로 이어진다.

## 3. ID 참조 vs 객체 참조

> Aggregate 끼리는 ID 로 잇고, 같은 Aggregate 내부에서만 객체 참조를 쓴다.

Aggregate 끼리는 객체 참조가 아니라 ID 참조로 잇는다. 이유는 두 가지다.

1. 로딩 범위가 명확해진다 — `Run` 을 메모리에 올릴 때 연결된 모든 `Profile` 까지 끌고 오지 않는다.
2. 트랜잭션 경계가 보호된다 — 객체 참조가 있으면 한 메서드 안에서 두 Aggregate 의 상태를 동시에 변경하기 쉬워진다.

`Run` 의 필드 구성은 다음 형태가 된다.

```java
// Run.java
public final class Run {
    private final RunId id;
    private final ProfileId profileId;        // 다른 Aggregate 는 ID 로
    private final List<BattleId> battleIds;   // 다른 Aggregate 는 ID 로
    private ActProgress actProgress;          // 같은 Aggregate 내부 VO 는 객체 참조
    // ...
}
```

`ProfileId` 는 Value Object 다. 단순 `Long` 보다 의미가 명확하고 다른 ID 와 혼동될 일이 없다. record · sealed 로 강제하는 방식은 `02-02.Entity 와 Value Object.md` 가 다룬다.

같은 Aggregate 내부에서는 객체 참조가 자연스럽다. `Run` 이 자기 안의 `ActProgress` 를 직접 들고 있는 것은 문제가 없다. 어차피 같은 트랜잭션에서 같이 로드되고 같이 저장된다.

### 3-1. 외부 Aggregate 가 필요할 때

`Run.completeBattle()` 안에서 `Profile.unlockReward(...)` 를 호출해야 한다고 느꼈다면, 두 가지 중 하나로 풀어야 한다.

1. 두 트랜잭션 + Domain Event — `Run` 이 `BattleCompleted` 이벤트를 발행하고 `Profile` 측 핸들러가 별도 트랜잭션에서 처리한다.
2. Domain Service — 두 Aggregate 를 함께 다뤄야 하는 비즈니스 규칙이라면 `RewardGrantService` 같은 Domain Service 가 두 Root 를 받아 조율한다. 다만 그 안에서도 트랜잭션은 분리한다.

`Run` 메서드가 `Profile` 인스턴스를 직접 잡고 변형하기 시작하면 둘은 사실상 한 Aggregate 다. 합치든가, 합칠 수 없다면 위 두 방식 중 하나를 골라야 한다.

## 4. 큰 Aggregate 의 비용

> "관련 있어 보이니 다 묶자" 는 결정은 로딩·잠금·동시성·변경 비용을 동시에 부풀린다.

초보 설계는 한 Aggregate 가 너무 크다. "관련 있어 보이니 다 묶자" 의 결과다. 비용은 다음 네 지점에서 누적된다.

1. 로딩 비용 — Root 를 불러올 때 그래프 전체가 따라온다. 100 개의 라인이 달린 주문은 매 조회마다 100 개를 가져온다.
2. 잠금 범위 — 비관적이든 낙관적이든 잠금 단위는 Aggregate 다. 한 라인을 바꾸려고 주문 전체를 잠근다.
3. 동시 변경 충돌 — 동일 Aggregate 의 다른 부분을 두 사용자가 동시에 변경할 때 낙관적 잠금에서는 한쪽이 무조건 재시도다.
4. 변경 비용 — Aggregate 가 크면 한 메서드의 책임이 부풀고 테스트도 따라 부푼다.

### 4-1. 분리의 신호

다음 신호 중 둘 이상이 보이면 Aggregate 를 쪼개는 것을 검토한다.

| 신호 | 해석 |
|------|------|
| 서로 다른 라이프사이클 | `Run` 은 세션 단위, `Profile` 은 영구. 같이 묶을 이유가 약하다 |
| 서로 다른 변경 빈도 | 카탈로그 `CardDefinition` 은 거의 안 바뀌고 `Run` 은 매 턴 바뀐다 |
| 서로 다른 트랜잭션 요구 | `Battle` 은 매 턴 강한 일관성이 필요하지만 `Progression` 은 결과적 일관성이면 충분 |
| 서로 다른 저장 전략 | 한쪽은 JPA, 한쪽은 Redis, 한쪽은 Event Sourcing |

### 4-2. 분리의 안전망

쪼개기로 결정했다면 다음 네 단계를 준비한다.

1. 새 Aggregate Root 의 식별자(Value Object) 를 정의한다.
2. 기존 객체 참조를 ID 참조로 교체한다. 호출자에서 `findById` 한 단계가 늘어난다.
3. 두 Aggregate 가 함께 변해야 하는 규칙은 Domain Event 로 전환한다 (`../04_edd/03-01.Saga, Outbox, Request-Response Bridge.md`).
4. 분리 전후의 불변식을 테스트로 박제한다. 분리 과정에서 가장 깨지기 쉬운 지점이다.

런 관리 예시에서 `Battle` 을 별도 Aggregate 로 빼는 결정은 `§2-1` 의 동시성 요구 차이에서 출발한다. 전투는 매 턴 강한 일관성이 필요하지만 런 전체는 그렇지 않다. 분리 비용은 위 네 단계이고, 얻는 것은 잠금 범위 축소와 저장 전략 자유도다.
