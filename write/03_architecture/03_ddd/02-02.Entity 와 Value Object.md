# Entity 와 Value Object
---
> 객체를 식별자로 구분할 것인가, 값 자체로 구분할 것인가 — 이 한 가지 결정이 동등성·변경 가능성·생명주기를 동시에 정한다.

`02-01 §3` 에서 Aggregate 내부에는 객체 참조를 쓰고 외부 Aggregate 는 ID 참조로 잇는다고 했다. 본 문서는 그 "객체" 의 두 종류를 다룬다.

## 1. Entity — 식별자가 본질

> Entity 는 속성이 같아도 ID 가 다르면 다른 것 — 이 단순한 규칙이 생명주기를 정의한다.

Entity 는 고유 식별자로 정의된다. 두 `Run` 인스턴스의 모든 속성이 동일하더라도 `RunId` 가 다르면 다른 런이다. 반대로 `RunId` 가 같으면 속성이 다르더라도 같은 런의 다른 시점이다.

이 규칙이 의미하는 것은 다음 세 가지다.

1. 생명주기를 가진다 — 생성·변경·삭제의 흐름이 ID 단위로 추적된다.
2. 속성은 가변 — ID 는 불변이지만 다른 속성은 메서드를 통해 바뀐다.
3. 영속성이 자연스럽다 — DB 행 하나가 하나의 Entity 에 대응되는 모델이 흔하다.

여기서 질문 하나 — `Card` 는 Entity 일까 Value Object 일까? `01-04 §2` 에서 봤듯 답은 Context 에 따라 다르다. catalog Context 의 `CardDefinition` 은 Entity(고유한 카드 정의)이지만, battle Context 의 손에 들린 한 장의 카드는 Value Object 에 가깝다. 같은 이름의 객체라도 Context 가 다르면 분류도 다르다.

## 2. Value Object — 값 자체가 정체성

> Value Object 는 속성이 같으면 같은 것 — 그래서 불변이고, 교체 가능하고, ID 가 필요 없다.

Value Object 는 식별자 없이 속성의 조합으로 동등성을 판단한다. 같은 화폐와 같은 금액의 두 `Price` 는 같은 객체로 간주한다. 이 특성에서 네 가지 성질이 따라온다.

1. 속성으로 식별 — `equals()` 와 `hashCode()` 가 모든 속성을 기준으로 동작한다.
2. 불변(immutable) — 변경이 필요하면 새 객체를 만든다. `price.addTax()` 는 새 `Price` 를 반환한다.
3. 단명(short-lived) 가능 — 생명주기 추적 부담이 없다.
4. 교체 가능 — 주문의 배송지가 바뀌면 기존 `Address` 를 폐기하고 새 `Address` 를 끼운다.

런 관리 예시에서 `Hp(int current, int max)`, `Energy(int amount)`, `CardCost(int value)`, `ActNumber(int value)` 가 모두 VO 다. 단순 `int` 보다 의미가 명확하고, 다른 정수 값과 혼동될 일이 없다.

### 2-1. Java record 와 sealed 로 강제하기

Java 17+ 에서 VO 는 record 로 자연스럽게 표현된다. record 는 자동으로 `equals` · `hashCode` · `toString` 을 생성하고, 모든 필드가 `final` 이라 불변이 강제된다.

```java
// Hp.java
public record Hp(int current, int max) {
    public Hp {
        if (current < 0 || current > max) {
            throw new IllegalArgumentException("invalid hp: " + current + "/" + max);
        }
    }

    public Hp reduce(int damage) {
        return new Hp(Math.max(0, current - damage), max);
    }
}
```

sealed interface 는 VO 의 변형이 닫힌 집합임을 컴파일 타임에 보장한다. 예를 들어 카드 효과의 종류가 `Damage` · `Block` · `Draw` 세 가지로 한정된다면 다음처럼 잠근다.

```java
// CardEffect.java
public sealed interface CardEffect permits Damage, Block, Draw {}
public record Damage(int amount) implements CardEffect {}
public record Block(int amount) implements CardEffect {}
public record Draw(int count) implements CardEffect {}
```

새 효과 종류가 추가되면 컴파일러가 `switch` 의 패턴 매칭에서 누락된 분기를 잡아준다. 이 안정성이 VO 를 단순 데이터 묶음이 아닌 도메인 어휘로 끌어올린다.

## 3. Entity 와 VO 의 선택 기준

> "이것의 정체는 ID 인가, 값인가?" — 이 한 질문이 분류를 결정한다.

분류가 헷갈릴 때는 다음 표를 본다.

| 질문 | Entity | Value Object |
|------|--------|--------------|
| 같은 속성의 두 인스턴스가 같은가? | 아니다, ID 가 같아야 같다 | 그렇다, 속성이 같으면 같다 |
| 시간에 따라 속성이 바뀌나? | 그렇다 | 아니다, 바뀌면 새 객체다 |
| 생성·삭제를 추적해야 하나? | 그렇다 | 아니다 |
| DB 행으로 저장되어야 하나? | 보통 그렇다 | 부모 Entity 의 컬럼이나 임베디드로 충분 |

잘못 분류된 사례를 보자. `EmailAddress` 를 Entity 로 만들면 같은 이메일 두 번 등록 시 두 행이 생기고, 동등성 비교가 ID 로만 가능해진다. VO 로 두면 문자열만 같으면 같은 객체로 인식한다. 반대로 `Order` 를 VO 로 두면 같은 상품·수량의 두 주문이 하나로 합쳐져 버린다.

## 4. ID 자체도 Value Object 다

> `RunId`, `ProfileId` 같은 식별자도 VO 의 한 종류 — 단순 `Long` 이 아닌 의미 있는 타입이어야 한다.

`02-01 §3` 의 ID 참조 규칙은 ID 가 VO 임을 전제한다. 단순 `Long` 을 ID 로 쓰면 `RunId`, `ProfileId`, `BattleId` 가 모두 `Long` 으로 보이고, 메서드 시그니처에서 혼동된다.

```java
// 나쁨
void grantReward(Long runId, Long profileId);  // 순서를 바꿔도 컴파일 통과

// 좋음
void grantReward(RunId runId, ProfileId profileId);  // 순서가 틀리면 컴파일 에러
```

record 로 정의된 ID 는 자동으로 동등성을 갖고, 추가 검증(예: UUID 포맷)을 생성자 컴팩트 구문에 박을 수 있다. 도메인 어휘를 타입 시스템에 박는 첫 단추가 ID 의 VO 화다.
