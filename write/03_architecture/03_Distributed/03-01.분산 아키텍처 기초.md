# 분산 아키텍처 기초
---
> 분산 아키텍처는 시스템을 여러 프로세스로 쪼개는 기술이 아니라, 네트워크 실패와 일관성 지연을 감수하고서라도 얻을 가치가 있을 때 선택하는 구조다.
>
> - 독립 배포
> - 독립 확장
> - 장애 격리
> - 운영 복잡도 증가

## 1. 무엇이 분산 시스템을 어렵게 만드는가

단일 프로세스 안에서는 함수 호출이 성공하거나 예외가 난다. 분산 시스템에서는 네트워크 타임아웃, 중복 요청, 메시지 재전송, 부분 실패가 항상 가능하다. 즉 “성공인지 실패인지 즉시 판단하기 어려운 상태”가 늘어난다.

그래서 분산 아키텍처를 도입하면 기술 스택보다 먼저 사고방식이 바뀌어야 한다. 호출은 실패할 수 있고, 응답은 늦을 수 있고, 동일 이벤트가 두 번 올 수 있다고 가정해야 한다.

## 2. 카드게임 런 관리 서비스에서 분산이 필요한 영역

예시 프로젝트의 핵심 전투 진행은 강한 일관성과 빠른 응답이 중요하다. 이런 흐름은 보통 하나의 애플리케이션 내부에 두는 것이 낫다.

반면 다음 영역은 분산 확장 후보가 된다:

- 전투 로그 적재와 리플레이 저장
- 밸런스 분석용 이벤트 수집
- 랭킹, 통계, 추천 같은 파생 읽기 모델

## 3. 서비스 분리의 기준

서비스를 분리할지 말지는 기능 개수보다 경계의 성격으로 판단해야 한다. 다음 질문이 핵심이다:

| 질문 | 예 | 분리 신호 |
|------|----|----------|
| 배포 주기가 다른가 | 분석 모듈만 자주 바뀜 | 강함 |
| 부하 특성이 다른가 | 리플레이 조회가 대용량 읽기 중심 | 강함 |
| 일관성 요구가 다른가 | 랭킹은 지연 허용 가능 | 중간 |
| 팀 경계가 다른가 | 운영 데이터 팀이 별도 관리 | 중간 |

## 4. 공유 DB는 왜 위험한가

초기에는 “서비스는 나눴지만 DB는 같이 쓰자”가 쉬워 보인다. 그러나 이 구조는 서비스 경계를 무력화한다. 한 서비스의 스키마 변경이 다른 서비스에 직접 영향을 준다.

카드게임 예시에서도 `battle`과 `analytics`가 같은 테이블을 직접 읽기 시작하면, 분석 요구가 전투 스키마 설계를 끌고 가게 된다. 이런 구조는 서비스 분리의 장점을 대부분 잃는다.

## 5. 분산 아키텍처의 기본 원칙

분산 환경에서는 다음 네 가지가 특히 중요하다:

- 각 서비스는 자기 데이터를 소유한다.
- 동기 호출은 최소화하고, 비동기 흐름을 적극 활용한다.
- 멱등성(idempotency)을 기본 가정으로 설계한다.
- 관측 가능성을 초기부터 포함한다.

이 원칙은 뒤에서 다룰 서비스 간 연동 방식, 트랜잭션 경계, 이벤트 기반 설계의 공통 배경이 된다.

## 6. 런 관리 서비스에 적용한 결론

초기 구조는 모듈러 모놀리스로 두고, 전투 로그 적재나 분석 파이프라인처럼 핵심 규칙 바깥의 기능부터 비동기 분리를 고려하는 편이 좋다. 즉 “핵심 도메인은 가까이, 파생 기능은 멀리”가 기본 전략이다.

분산 아키텍처는 목표가 아니라 비용 있는 수단이다. 이 도메인에서는 전투 규칙 엔진을 먼저 안정화한 뒤, 외곽 기능을 분리하는 순서가 가장 안전하다.
