# REST, gRPC, Messaging 선택 기준
---
> 어떤 프로토콜을 쓸지는 유행보다 호출의 성격으로 판단해야 한다. “즉시 결과가 필요한가”, “재시도가 허용되는가”, “소비자가 여러 개인가”가 핵심 질문이다.
>
> - request-response
> - streaming
> - event fan-out
> - eventual consistency

## 1. 세 가지 방식의 질문 프레임

REST, gRPC, Messaging은 서로 대체재가 아니라 다른 상황에 최적화된 도구다. 판단 기준을 정하지 않으면 팀은 익숙한 기술을 모든 문제에 끼워 맞추게 된다.

우선 다음 질문을 해보면 좋다:

- 호출자는 지금 응답을 기다리고 있는가
- 실패 시 같은 요청을 다시 보내도 되는가
- 같은 결과를 여러 소비자가 동시에 받아야 하는가

## 2. REST를 선택할 때

REST는 브라우저와 가장 잘 맞고, 사람에게 읽히는 디버깅도 쉽다. 공개 API, 백오피스 API, 단순 조회/명령에는 여전히 가장 실용적이다.

런 관리 서비스에서는 다음이 REST에 잘 맞는다:

- 런 시작 요청
- 현재 런 상태 조회
- 보상 선택 제출

## 3. gRPC를 선택할 때

gRPC는 타입 안전성과 양방향 스트리밍이 강점이다. 내부 서비스 간 호출이 많고, 낮은 지연과 계약 엄격성이 중요할 때 효과가 있다.

다만 카드게임 예시의 초기 구조에서는 gRPC가 반드시 필요한 것은 아니다. 핵심 도메인을 먼저 모놀리스 안에 두는 편이 합리적이기 때문이다. gRPC는 분리 이후 내부 RPC가 잦아질 때 고려하면 된다.

## 4. Messaging을 선택할 때

메시징은 결과를 즉시 돌려줄 필요가 없고, 같은 이벤트를 여러 소비자가 구독해야 할 때 강력하다. 생산자와 소비자의 생명주기를 느슨하게 만들고, 장애 격리에 유리하다.

다음 흐름은 메시징에 잘 맞는다:

- `BattleWon` 이벤트를 분석 파이프라인에 전달
- 리플레이 저장용 이벤트 적재
- 이상 패턴 탐지용 비동기 소비

## 5. 비교 기준 요약

| 기준 | REST | gRPC | Messaging |
|------|------|------|----------|
| 사용자 직접 요청 | 강함 | 약함 | 약함 |
| 타입 엄격성 | 중간 | 강함 | 스키마 설계에 좌우 |
| 다중 소비자 | 약함 | 약함 | 강함 |
| 장애 격리 | 약함 | 약함 | 강함 |
| 즉시 응답 | 강함 | 강함 | 약함 |

## 6. 게임 도메인 적용 예시

플레이어가 카드 한 장을 사용하면 결과를 바로 알아야 한다. 이 흐름은 REST API 또는 내부 직접 호출에 가깝다. 반면 “이번 런의 카드 선택 패턴을 데이터 웨어하우스에 쌓는다”는 메시징에 가깝다.

즉 핵심 게임플레이는 동기 흐름, 파생 데이터 처리와 운영 이벤트는 비동기 흐름으로 보는 것이 자연스럽다.

## 7. 실무 결론

이 시리즈의 기본 선택은 다음과 같다:

- 사용자 요청 경계: REST
- 초기 모듈 간 내부 호출: 직접 호출
- 분리 이후 고빈도 내부 RPC: 필요 시 gRPC
- 파생 데이터와 후처리: Messaging

중요한 것은 한 기술로 통일하는 것이 아니라, 호출의 성격에 맞는 기술을 고르는 것이다. 그 판단이 아키텍처 품질을 좌우한다.
