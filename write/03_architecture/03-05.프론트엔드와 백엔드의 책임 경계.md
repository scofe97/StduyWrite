# 프론트엔드와 백엔드의 책임 경계
---
> "비즈니스 규칙은 백엔드가, 화면은 프론트엔드가"라는 문장은 단순하지만 실전에서 자주 깨진다. 유효성 검증, 권한 표시, 상태 머신, 합계 계산 같은 회색 지대가 양쪽에 동시에 살면서 진실 공급원이 둘이 된다.

## 1. 진실 공급원은 한 곳이어야 한다

> 같은 규칙이 프론트엔드와 백엔드에 동시에 박히면 둘 중 하나가 거짓이다.

프론트엔드에 "결재 금액 1억 초과 시 추가 결재선 필요"가 박혀 있고 백엔드에도 같은 규칙이 박혀 있다면, 규칙이 바뀔 때 두 곳을 동시에 배포해야 한다. 배포 시점이 어긋나면 화면은 추가 결재를 안 받지만 서버가 거부하거나 그 반대가 된다.

원칙은 단순하다. **모든 비즈니스 규칙은 백엔드가 단독 보유한다.** 프론트엔드는 사용자 경험을 위해 같은 규칙을 표시할 수 있지만, 실제 검증·결정은 백엔드가 한다. 프론트엔드 검증은 "UX 최적화"이지 "안전망"이 아니다.

## 2. 회색 지대 다섯 가지

실전에서 책임이 흔들리는 다섯 자리를 정리한다.

| 회색 지대 | 잘못된 분담 | 올바른 분담 |
|----------|-------------|------------|
| 입력 유효성 검증 | 프론트에만 (백엔드 우회 가능) | 백엔드가 최종, 프론트는 UX 보조 |
| 권한별 버튼 표시 | 프론트가 자기 권한 판단 | 백엔드가 권한 결과를 응답에 포함 |
| 상태 머신 (다음 가능 액션) | 프론트가 if-else로 추론 | 백엔드가 `availableActions` 배열로 응답 |
| 합계·집계 계산 | 양쪽에서 각자 계산 | 백엔드 응답에 미리 계산된 값 포함 |
| 정렬·필터 옵션 | 프론트가 클라이언트 정렬 | 백엔드가 정렬·필터를 받아 응답 |

다섯 자리 모두 공통 원리가 있다. **백엔드가 "사실"을 응답에 포함시키고, 프론트엔드는 그 사실을 표시한다.** 프론트엔드가 추론을 시작하는 순간 진실 공급원이 둘이 된다.

## 3. 응답에 액션을 포함하는 패턴

상태 머신 응답을 예로 보자. 잘못된 형태는 다음과 같다.

```json
{ "id": "A-1", "status": "PENDING" }
```

프론트엔드는 status가 `PENDING`이면 "승인" 버튼을 보여줘야 하나 같은 사용자가 결재자인지, 이미 한 번 승인한 적이 있는지를 클라이언트가 다시 판단해야 한다. 결국 권한 규칙이 프론트엔드에 박힌다.

올바른 형태는 사용 가능한 액션을 백엔드가 계산해 응답에 포함한다.

```json
{
  "id": "A-1",
  "status": "PENDING",
  "availableActions": ["approve", "reject", "delegate"],
  "blockedActions": [
    { "action": "rollback", "reason": "이미 다음 결재자가 처리 시작" }
  ]
}
```

프론트엔드는 `availableActions`을 그대로 버튼으로 그리고 `blockedActions`은 비활성 + 사유 툴팁으로 표시한다. 규칙이 바뀌면 백엔드 한 곳만 바꾸면 된다.

## 4. UX 최적화는 어디까지

> 프론트엔드 검증을 완전히 없애면 사용자 경험이 나빠진다. 검증을 두되 권위를 두지 않는다는 게 핵심이다.

다음 두 가지는 프론트엔드가 해도 된다.

- 명백한 형식 오류 즉시 표시 (이메일 형식, 필수 입력 누락)
- 입력 길이 제한, 숫자 범위 같은 정적 제약 표시

다음 두 가지는 프론트엔드가 하면 안 된다.

- 비즈니스 규칙 판단 ("이 사용자는 이 결재선에 끼면 안 됨")
- 권한 결정 ("이 사용자는 이 버튼을 볼 수 있음")

경계는 단순하다. **데이터·사용자·시점에 따라 결과가 달라지는 규칙은 백엔드.** 입력 자체의 형태에만 의존하는 규칙은 프론트엔드도 가능하다.

## 5. 책임 분담이 흔들리는 신호

다음 신호가 보이면 책임 경계가 깨지고 있다는 뜻이다.

- 같은 if문이 프론트엔드 코드와 백엔드 코드에 동시에 있다
- 백엔드 변경 PR과 프론트엔드 변경 PR이 항상 같이 올라온다
- 프론트엔드가 "이 화면을 위해 백엔드에 이 필드를 추가해 달라"는 요청을 자주 한다
- 백엔드 API 응답이 화면 구조를 그대로 따라간다 (DB join 결과를 그대로 노출)

마지막 신호는 특히 위험하다. API 응답이 화면 구조를 따라가면 화면이 바뀔 때마다 API가 바뀌고, 백엔드가 사실상 프론트엔드의 코드 생성기로 격하된다. **API는 도메인을, 화면은 UX를 표현한다.** 둘을 잇는 것이 02-10에서 다룬 API Composition·BFF·GraphQL이다.

## 6. 팀 구조와의 정렬

Conway's Law는 시스템 구조가 팀 구조를 닮는다고 말한다. 프론트엔드·백엔드의 책임 경계도 팀 구조와 같이 봐야 한다.

| 팀 구조 | 권장 경계 모델 |
|---------|---------------|
| FE·BE 한 팀, 풀스택 | API 한 묶음, 응답 형태 자유롭게 협상 |
| FE 팀과 BE 팀 분리, 화면 종류 1~2개 | BFF 또는 API Composition으로 책임 명시 |
| FE 팀 여러 개 (웹·모바일), BE 도메인 팀 다수 | 클라이언트별 BFF, 도메인은 BFF 뒤로 |

가장 자주 보이는 안티패턴은 **팀은 셋인데 API는 하나로 두는 것**이다. 백엔드는 화면 요구사항을 다 받아주려다 응답이 비대해지고, 프론트엔드는 자기 화면 외 필드를 무시한다. BFF 또는 API Composition으로 클라이언트 책임을 명시하면 양쪽이 자기 일에만 집중할 수 있다.

책임 경계는 코드의 문제가 아니라 **누가 무엇을 알고 있어야 하는가**의 문제다. 결정이 어디서 일어나는지를 명확히 하면 코드는 따라 정리된다.
