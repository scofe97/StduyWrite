# 도구 레이어 — Implement Layer와 Reader Writer 패턴
---
> 한국 백엔드 진영에서 자주 보이는 "Implement Layer", "Reader/Writer 패턴" 같은 명명은 결국 PoEAA의 Service Layer + Repository를 다른 이름으로 다시 한 번 쪼갠 것이다. 추가 분할이 정당화되려면 그 분할이 새 책임을 표현해야 하는데, 대개는 그렇지 못하다.

## 1. 무엇이 "Implement Layer"인가

> 같은 패키지 안에 `XxxService` 인터페이스와 `XxxServiceImpl` 구현 클래스를 두는 관행이 "Implement Layer"라 불린다.

전형적인 모습은 다음과 같다.

```
application/
  approval/
    ApprovalService.java         (interface)
    ApprovalServiceImpl.java     (class implements ApprovalService)
```

의도는 좋다. 인터페이스로 추상화해두면 구현체를 갈아끼울 수 있다. 그러나 실전에서 갈아끼우는 일은 거의 일어나지 않고, 인터페이스는 구현체와 1:1로 묶여 있다. Martin Fowler가 PoEAA에서 말한 Service Layer는 "응용 외부와의 경계"이지 "모든 클래스에 인터페이스를 붙이는 관행"이 아니다.

## 2. Reader/Writer 패턴의 정체

비슷한 결로 등장하는 것이 Reader/Writer 분할이다. 하나의 `ApprovalRepository`를 `ApprovalReader`와 `ApprovalWriter`로 쪼개고, 서비스가 둘을 각각 주입받는 식이다.

```java
@Service
class ApprovalService {
    private final ApprovalReader reader;
    private final ApprovalWriter writer;
    ...
}
```

이 분할이 정당화되는 경우는 분명히 있다. **Read 모델과 Write 모델이 실제로 다른 저장소에서 다른 형태로 다뤄질 때**다. CQRS(03-07)에서 Read는 별도 read view를 사용하고 Write는 도메인 aggregate를 통과한다. 이때 두 인터페이스는 서로 다른 책임을 표현하므로 분할에 의미가 있다.

문제는 **CQRS가 아닌데도 Reader/Writer를 도입하는 경우**다. 둘 다 같은 JPA Repository를 호출하고, 같은 엔티티를 다루며, 같은 트랜잭션 안에서 쓰인다. 이때 분할은 "이름표 두 개"일 뿐이고 호출자가 메서드를 찾을 때 어디를 봐야 하는지만 헷갈리게 한다.

## 3. 분할이 정당화되는 조건

추가 분할이 가치를 가지려면 다음 중 하나가 성립해야 한다.

| 조건 | 예시 | 분할 이름 |
|------|------|----------|
| 응답 모델이 도메인 모델과 다르다 | API 응답용 DTO 빌드 | Read view / Projection |
| 트랜잭션 정책이 다르다 | 읽기는 readonly, 쓰기는 직렬화 | `@Transactional(readOnly=true)` 분리 |
| 저장소가 다르다 | 쓰기는 RDB, 읽기는 캐시 | CQRS Read model |
| 권한 정책이 다르다 | 조회는 누구나, 변경은 본인만 | Authorization layer |

이 중 하나도 성립하지 않으면 분할은 도구 레이어(tool layer)에 그친다. **도구 레이어는 책임을 표현하지 않고 패턴을 표현한다.** 코드 리뷰에서 "왜 나눴어요?" 물었을 때 "원래 그렇게 해요"라는 답이 돌아오면 도구 레이어다.

## 4. 안티패턴이 만드는 비용

도구 레이어를 도입하면 다음 비용이 생긴다.

- **클래스 수 증가**: 모듈 하나당 인터페이스 + 구현 + Reader + Writer로 4개가 기본. 모듈 5개면 20개 파일이 된다.
- **변경 비용 증가**: 메서드 시그니처 하나 바꾸면 인터페이스·구현·테스트 모두 손봐야 한다.
- **이름 충돌**: `ApprovalReader.findById`와 `ApprovalService.getApproval`이 같은 일을 하는데 이름이 다르다.
- **추적 비용**: 호출 스택을 따라가면 의미 있는 코드까지 가는 점프 수가 늘어난다.

PoEAA의 Service Layer 챕터는 명확하다. "Service Layer는 응용 외부와의 경계에 위치하며, 그 안의 분할은 응용의 use case 구조를 따른다." use case가 아닌 도구로 나누는 것은 패턴 적용이 아니라 패턴 흉내다.

## 5. 권장 기본형

특별한 사유가 없는 한 다음 형태를 기본으로 한다.

```
application/
  approval/
    RequestApprovalUseCase.java     (interface, inbound port)
    RequestApprovalService.java     (class implements UseCase)

domain/
  approval/
    Approval.java                   (Aggregate)
    ApprovalRepository.java         (port)

infrastructure/
  approval/
    JpaApprovalRepository.java      (adapter)
```

여기서 인터페이스가 의미를 갖는 곳은 **inbound port와 outbound port** 두 곳뿐이다. 헥사고날(02-03)이 정확히 이 위치에 인터페이스를 둔다. 그 외 위치에는 인터페이스를 강제하지 않는다.

## 6. 도구 레이어가 이미 박혀 있다면

레거시에 도구 레이어가 박혀 있어도 전면 리팩토링은 잘 일어나지 않는다. 단계적으로 줄이는 방법은 다음과 같다.

1. 새로 만드는 모듈에는 도구 레이어를 적용하지 않는다.
2. 기존 모듈 중 클래스 변경이 잦은 곳부터 인터페이스를 인라인한다 (IntelliJ `Inline Interface`).
3. Reader/Writer 분할은 CQRS로 분리할 곳만 남기고 통합한다.
4. 새 패턴은 ADR(01-05)로 기록해 팀에 공유한다.

핵심은 패턴의 이름이 아니라 **그 분할이 어떤 책임을 표현하는가**다. 표현되는 책임이 없으면 분할도 없어야 한다.
