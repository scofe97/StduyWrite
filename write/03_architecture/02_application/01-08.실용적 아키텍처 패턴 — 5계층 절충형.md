# 실용적 아키텍처 패턴 — 5계층 절충형
---
> 아키텍처는 비즈니스 문제를 푸는 수단이지 그 자체가 목적은 아닙니다. 이 문서는 계층형·클린·헥사고날의 장점만 골라 인지 부하를 최소화한 5계층 절충형 패턴을 정리합니다. 핵심은 유스케이스를 1급 시민으로 끌어올리고, 나머지 계층은 필요할 때만 늘리는 데 있습니다.



## 1. 왜 또 다른 아키텍처인가

레이어드·헥사고날·벌티컬 슬라이스 같은 패턴이 이미 많은데 또 하나를 더하는 이유는, 기존 패턴마다 실무에서 불편한 지점이 분명하기 때문입니다. 레이어드 아키텍처는 출발점으로 쉽지만 확장의 단위가 *요구사항*이 아니라 *기술 계층*이라, 기능 하나를 추가할 때 controller·service·repository를 모두 헤집어야 합니다. 헥사고날은 경계가 명확한 대신 포트·어댑터·도메인 매핑을 전부 갖추느라 공수가 크고, 작은 서비스에서는 추상화 비용이 가치를 넘어섭니다.

이 패턴이 출발하는 전제는 단순합니다. 개발자는 클린 코드를 작성하는 사람이 아니라 비즈니스 가치를 만드는 사람이라는 입장입니다. 그래서 모든 계층을 처음부터 갖추지 않고, 각 계층이 *실제로 책임을 표현할 때만* 도입합니다. 'Vertical Slice'는 원문에서 이름만 거론될 뿐 별도로 설명되지 않으므로, 본 문서도 그 개념은 다루지 않습니다.

다섯 계층은 안쪽부터 도메인 객체 → 포트 → 유스케이스 → 프로세서 → 컨트롤러 순서로 쌓입니다. 가장 안쪽이 가장 순수하고, 바깥으로 갈수록 외부 입출력에 가까워집니다.



## 2. 도메인 객체 계층

> 비즈니스 데이터를 바탕으로 핵심 로직을 수행하는 객체가 사는, 시스템의 가장 안쪽 계층입니다.

도메인 모델 패턴을 따라 로직을 객체 안에 둡니다. 상태를 꺼내 서비스가 계산하는 대신, 객체가 스스로 판단하게 만들면 가독성과 재사용성이 같이 올라갑니다. 예를 들어 활성 여부 판단은 `Member`가 직접 답합니다.

```java
@Getter
@Builder
@NoArgsConstructor
@AllArgsConstructor
@Entity
public class Member {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    private LocalDate birthday;
    @Enumerated(EnumType.STRING)
    private MemberStatus status;

    public boolean isActive() {
        return status == MemberStatus.ACTIVE;
    }
}
```

여러 객체를 묶어 다룰 때는 일급 컬렉션을 씁니다. 컬렉션을 감싸면 필터링 같은 규칙이 흩어지지 않고 한곳에 모입니다.

```java
@Getter
@Builder
public class Members {
    private final List<Member> members;

    public Members getActiveMembers() {
        List<Member> activeMembers = members.stream()
                .filter(Member::isActive)
                .toList();
        return new Members(activeMembers);
    }
}
```

이 패턴이 클린 아키텍처([01-02](01-02.%ED%81%B4%EB%A6%B0%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98.md))와 갈라지는 지점이 여기 있습니다. 클린 아키텍처는 도메인 엔티티와 영속성 엔티티를 분리해 안쪽을 프레임워크로부터 지킵니다. 반면 이 패턴은 둘을 합치고 도메인 객체에 JPA 애너테이션을 그대로 답니다. 분리가 주는 순수함보다, 분리를 유지하는 매핑·변환 코드가 개발 비용과 유지보수 비용을 더 키운다고 보기 때문입니다. 어느 쪽이 옳다기보다, *순수함*과 *비용* 사이의 트레이드오프를 어디서 끊느냐의 문제입니다.

도메인 정책 검증도 이 계층의 몫입니다. 이메일 길이 제한처럼 비즈니스가 정한 규칙은 도메인 객체 안에서 막습니다. 입력 형식 검증과는 책임이 다릅니다(§6에서 다시 다룹니다).

```java
public Member {
    if (email.length() > 200) {
        throw new IllegalArgumentException("이메일 길이는 200을 넘을 수 없습니다.");
    }
}
```



## 3. 포트 계층

> 데이터베이스·캐시·외부 API 같은 외부 인프라와의 통신을 담당하는 계층입니다.

외부 기술과의 결합을 한 곳으로 모아, 도메인과 유스케이스가 인프라 세부를 모르게 합니다. 네이밍은 통신 대상으로 구분합니다. 저장소는 `Repository`, 캐시는 `Cache`, API 호출은 `Client`로 끝맺습니다.

```java
public interface MemberRepository extends JpaRepository<Member, Long> {
    boolean existsByEmail(String email);
}
```

파라미터가 많아지면 `Request`·`Response` 객체로 묶습니다. 단일 값이나 도메인 엔티티를 그대로 넘기는 것도 막지 않습니다.

```java
public class OrderClient {
    private final RestClient restClient;

    public Order order(OrderRequest orderRequest) {
        return restClient.post()
                .body(orderRequest)
                .retrieve()
                .body(Order.class);
    }
}
```

여기서도 실용주의가 드러납니다. 인터페이스는 필요할 때만 도입합니다. 현실에서는 구현체 없이 인터페이스만 갈아끼우는 일이 드물어, 모든 포트에 인터페이스를 붙이면 추상화가 가치를 만들기 전에 비용부터 발생합니다. 이 판단은 도구 레이어([01-06](01-06.%EB%8F%84%EA%B5%AC%20%EB%A0%88%EC%9D%B4%EC%96%B4%20%E2%80%94%20Implement%20Layer%EC%99%80%20Reader%20Writer%20%ED%8C%A8%ED%84%B4.md))가 말한 "추가 분할은 책임을 표현할 때만 정당하다"와 같은 줄기입니다. 네이밍 자체가 무엇이냐보다, 팀이 일관되게 쓰느냐가 더 중요합니다.



## 4. 유스케이스 계층

> 사용자가 시스템으로 목표를 달성하는 일련의 활동을 구현하는, 이 패턴의 핵심 계층입니다.

유스케이스를 1급 클래스로 끌어올립니다. "회원 가입", "상품 주문"처럼 기획자와 대화하는 단위가 곧 클래스 하나가 됩니다. 입력과 출력은 유스케이스 *내부* record로 두는데, 유스케이스가 서비스의 핵심이라 입출력을 가까이 둬야 빠르게 이해하고 바꿀 수 있기 때문입니다.

```java
@Service
@RequiredArgsConstructor
public class RegisterMemberUseCase {
    private final MemberRepository memberRepository;

    public record Input(
            @NotBlank String name,
            @NotBlank String email,
            @NotBlank String password
    ) {}

    public record Output(Member member) {}

    public Output execute(Input input) {
        if (memberRepository.existsByEmail(input.email())) {
            throw new IllegalStateException("중복된 이메일입니다.");
        }

        Member savedMember = memberRepository.save(
                Member.builder()
                        .name(input.name())
                        .email(input.email())
                        .password(input.password())
                        .build()
        );
        savedMember.activate();
        return new Output(savedMember);
    }
}
```

기존 레이어드 아키텍처([01-01](01-01.Spring%20Boot%20%EB%A0%88%EC%9D%B4%EC%96%B4%EB%93%9C%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%EC%99%80%20%ED%8C%A8%ED%82%A4%EC%A7%80%20%EA%B5%AC%EC%A1%B0.md))에서 application 계층은 유스케이스를 "조합"하는 곳이었습니다. 이 패턴은 한 발 더 나아가 유스케이스 *하나하나를 독립 클래스*로 만듭니다. 모든 기능을 단일 서비스에 몰면 클래스가 비대해져 원하는 로직을 찾기 어려운데, 요구사항 단위로 쪼개면 파일 이름만 봐도 어디를 열어야 할지 압니다.

```text
member/
├── RegisterMemberUseCase
├── GetMemberListUseCase
├── GetMemberDetailUseCase
├── UpdateMemberUseCase
└── DeleteMemberUseCase
```

이 분할의 기준은 SRP인데, 여기서 SRP는 "하나의 모듈은 오직 하나의 액터에 대해서만 책임진다"로 읽습니다. 그래서 일반 사용자의 회원 조회와 관리자의 회원 조회는 코드가 비슷해 보여도 *다른 액터*를 섬기므로 별도 유스케이스로 둡니다.

여기서 *진짜 중복*과 *가짜 중복*을 가릅니다. 코드 텍스트가 같다고 모두 중복은 아닙니다. 섬기는 액터가 다르거나 변경 주기가 다르면 가짜 중복이고, 이때 억지로 합치면 한쪽 요구사항이 바뀔 때 다른 쪽까지 흔들립니다. 반대로 같은 액터의 같은 규칙이 여러 곳에 흩어져 있으면 진짜 중복이고, 이건 다음 계층에서 모읍니다.



## 5. 프로세서 / 애플리케이션 서비스 계층

> 여러 유스케이스에 흩어진 진짜 중복 로직만 공통으로 모으는 계층입니다.

이 계층의 가장 중요한 성격은 *필요에 의해서만 탄생한다*는 점입니다. 비즈니스에 따라 아예 존재하지 않을 수도 있습니다. 필요하지도 않은데 계층부터 만들면 인지 부하와 복잡도만 올라가므로, 임의로 파생시키지 않습니다. 진짜 중복(§4)이 실제로 쌓였을 때 비로소 추출합니다.

유스케이스끼리는 서로를 의존하지 않습니다. 한 유스케이스가 다른 유스케이스를 호출하기 시작하면 호출 그래프가 얽혀 추적이 어려워지기 때문입니다. 공통이 필요하면 유스케이스를 부르는 게 아니라, 공통 로직을 이 프로세서 계층으로 내려 양쪽이 함께 의존하게 만듭니다.

이 "필요할 때만 분리한다"는 규칙은 도구 레이어([01-06](01-06.%EB%8F%84%EA%B5%AC%20%EB%A0%88%EC%9D%B4%EC%96%B4%20%E2%80%94%20Implement%20Layer%EC%99%80%20Reader%20Writer%20%ED%8C%A8%ED%84%B4.md))가 Implement Layer·Reader/Writer 남용을 경계한 논리와 정확히 맞물립니다. 분할이 새 책임을 표현하지 못하면 그 분할은 패턴을 위한 패턴일 뿐입니다.



## 6. 컨트롤러 계층

> API 요청을 받아 유스케이스로 위임하는, 핵심에서 가장 먼 바깥 계층입니다.

컨트롤러는 여러 유스케이스를 동시에 의존해도 됩니다. 이 패턴에서 가장 중요한 계층은 유스케이스이고 컨트롤러는 입력을 위임만 하므로, 여러 유스케이스를 들고 있어도 SRP를 깨지 않습니다.

```java
@RestController
@RequiredArgsConstructor
public class MemberController {
    private final RegisterMemberUseCase registerMemberUseCase;
    private final GetMemberDetailUseCase getMemberDetailUseCase;
    private final UpdateMemberUseCase updateMemberUseCase;

    @PostMapping("/members")
    public ResponseEntity<RegisterMemberUseCase.Output> registerMember(
            @RequestBody @Valid RegisterMemberUseCase.Input input
    ) {
        return ResponseEntity.ok(registerMemberUseCase.execute(input));
    }
}
```

유스케이스의 Input을 컨트롤러에서 그대로 받습니다. 별도 요청 DTO를 또 만들지 않아 변환 코드가 줄어듭니다. 다만 PathVariable과 RequestBody가 섞이는 등 입력이 복잡하면 그때만 컨트롤러 전용 DTO를 둡니다.

```java
record UpdateMemberBody(String name, String email, String password) {}
```

검증은 위치를 셋으로 나눕니다. 입력 형식 검증은 Input의 `@NotBlank` 같은 애너테이션으로 컨트롤러 진입에서 막고, 비즈니스 정책 검증은 도메인 객체(§2)가 맡으며, 흐름 차원의 사전 조건은 유스케이스(§4)가 확인합니다. 컨트롤러에서 유효하지 않은 입력을 먼저 걸러내면, 그 아래 모든 계층은 입력이 유효하다고 확신하고 로직을 작성할 수 있습니다. 이 확신이 하위 코드의 방어 코드를 줄여 줍니다.



## 7. 기존 패턴과의 관계

이 패턴은 새 발명이라기보다, 이미 정리한 네 패턴에서 쓸 만한 부분을 골라 절충한 결과입니다. 어디서 같고 어디서 갈라지는지 정리하면 다음과 같습니다.

| 비교 대상 | 같은 점 | 이 패턴의 절충 |
|-----------|---------|----------------|
| 레이어드 ([01-01](01-01.Spring%20Boot%20%EB%A0%88%EC%9D%B4%EC%96%B4%EB%93%9C%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%EC%99%80%20%ED%8C%A8%ED%82%A4%EC%A7%80%20%EA%B5%AC%EC%A1%B0.md)) | 의존 방향을 안쪽으로 단순화 | 확장 단위를 기술 계층이 아니라 요구사항(유스케이스)으로 |
| 클린 ([01-02](01-02.%ED%81%B4%EB%A6%B0%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98.md)) | 핵심 로직을 안쪽에 격리 | 도메인=영속성 엔티티를 합쳐 매핑 비용을 회피 |
| 헥사고날 ([01-03](01-03.%ED%97%A5%EC%82%AC%EA%B3%A0%EB%82%A0%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98.md)) | 외부 인프라를 포트로 경계 | 포트 인터페이스는 필요할 때만, 추상화 최소화 |
| 도구 레이어 ([01-06](01-06.%EB%8F%84%EA%B5%AC%20%EB%A0%88%EC%9D%B4%EC%96%B4%20%E2%80%94%20Implement%20Layer%EC%99%80%20Reader%20Writer%20%ED%8C%A8%ED%84%B4.md)) | 분할은 책임을 표현할 때만 | 프로세서 계층은 진짜 중복이 쌓였을 때만 탄생 |

한 줄로 요약하면, 이 패턴은 클린·헥사고날이 추구한 *경계*는 받아들이되 그 경계를 *비용이 정당화될 때만* 긋습니다. 추상화를 미덕이 아니라 비용으로 보고, 비즈니스 가치를 우선에 둔 결정입니다.
