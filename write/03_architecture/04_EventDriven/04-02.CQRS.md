# CQRS
---
> CQRS(Command Query Responsibility Segregation)는 쓰기 모델과 읽기 모델을 같은 구조로 강제하지 않는 접근이다.
>
> - command
> - query
> - 읽기 최적화
> - 쓰기 모델 보호

## 1. 왜 읽기와 쓰기를 분리하는가

전형적인 CRUD 구조는 읽기와 쓰기가 같은 테이블, 같은 모델을 공유한다. 이 방식은 단순하지만, 도메인 규칙이 복잡해지고 읽기 요구가 다양해질수록 모델이 뒤엉킨다.

카드게임 런 관리 서비스도 비슷하다. 카드 사용과 턴 종료는 강한 규칙 검증이 필요하지만, 현재 런 상태 조회나 리플레이 조회는 읽기 편의성이 더 중요하다.

## 2. command 모델과 query 모델

command 모델은 상태를 안전하게 바꾸는 데 초점을 둔다. 따라서 aggregate, 정책, 유효성 검사가 중요하다. query 모델은 빠르고 읽기 쉬운 형태가 중요하므로, join된 뷰나 문서 형태가 더 적합할 수 있다.

예시로 보면 다음과 같다:

- command: `PlayCard`, `EndTurn`, `SelectReward`
- query: `GetRunSummary`, `GetBattleReplay`, `GetRunHistory`

## 3. 언제 CQRS가 가치가 있는가

읽기 패턴이 쓰기 모델과 현저히 다를 때 CQRS의 가치가 커진다. 특히 도메인 모델은 복잡하지만, 사용자에게 보여주는 화면은 단순 요약이 필요한 경우 효과가 크다.

런 관리 서비스에서는 전투 계산을 위한 모델과, UI에 보여줄 현재 손패/에너지/적 상태 요약 모델이 꼭 같을 필요가 없다. 이 차이가 CQRS 도입의 근거다.

## 4. 과도한 CQRS는 피해야 한다

모든 프로젝트가 물리적 DB 분리까지 갈 필요는 없다. 초기에는 코드 수준에서 command와 query를 구분하는 것만으로도 큰 효과가 있다.

예를 들어 다음 정도로 시작할 수 있다:

- command는 domain aggregate를 통해 처리한다.
- query는 전용 read repository나 projection을 사용한다.
- 저장소는 같더라도 코드 경계는 분리한다.

## 5. 이벤트 기반 구조와의 관계

CQRS는 이벤트 기반 구조와 잘 어울린다. command 쪽에서 발생한 도메인 이벤트를 받아 query 쪽 projection을 갱신하면 읽기 모델을 자연스럽게 구성할 수 있기 때문이다.

예를 들면 `CardPlayed` 이벤트를 받아 리플레이 뷰를 갱신하고, `BattleWon` 이벤트를 받아 런 요약 뷰를 갱신하는 식이다.

## 6. 카드게임 도메인 적용 예시

다음 분리는 현실적이다:

| 모델 | 목적 | 저장 형태 | 예시 |
|------|------|----------|------|
| command model | 규칙 검증과 상태 변경 | aggregate 중심 | `Run`, `Battle` |
| query model | 빠른 조회와 UI 표시 | projection 중심 | `RunSummaryView`, `ReplayView` |

이 구조를 쓰면 전투 규칙이 복잡해져도 UI 조회 모델이 같이 더러워지지 않는다.

## 7. 실무 결론

이 시리즈에서는 CQRS를 “무조건 쓰기 DB와 읽기 DB를 분리하는 패턴”으로 보지 않는다. 먼저 코드 경계부터 나누고, 필요할 때 projection과 저장소를 분리하는 점진적 전략으로 본다.

카드게임 런 관리 서비스에서는 리플레이, 런 요약, 운영 분석 화면이 늘어날수록 CQRS의 가치가 커진다. 핵심은 쓰기 모델을 조회 요구로 오염시키지 않는 것이다.
