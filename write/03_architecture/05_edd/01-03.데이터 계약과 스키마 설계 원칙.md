# 데이터 계약과 스키마 설계 원칙
---
> 이벤트는 메시지가 아니라 계약이다 — 한 번 발행된 이벤트의 형식은 그 토픽을 구독하는 모든 서비스에 대한 약속이고, 약속의 변경에는 호환성이라는 비용이 따른다.

## 1. Event-Driven Data Contracts

> 이벤트는 발행 시점에 다음과 같이 선언한다 — "지금부터 이 형식으로 이런 사실을 보낼 것이다. 너는 이 형식을 기준으로 코드를 짤 수 있다."

SSOT §1 의 핵심 메시지는 이벤트가 명시적 계약이라는 점이다. 명시적이지 않으면 다음 문제가 생긴다.

1. 새 Consumer 가 어떤 형식이 올지 추측해야 한다.
2. Producer 가 형식을 바꿔도 Consumer 에게 알리지 않는다.
3. 호환성 검증이 사고가 난 뒤에야 가능하다.

계약을 명시적으로 만드는 도구가 스키마다. 스키마는 발행 시점에 형식을 박제하고, 변경 시 호환성을 자동 검증하게 만든다.

## 2. Explicit Schema vs Implicit Schema

> JSON 의 "필드가 있을 수도 있고 없을 수도 있다" 는 implicit schema 다 — 같은 키워드라도 명시적 스키마와는 다른 종류의 안정성을 갖는다.

SSOT §2 의 비교는 다음과 같다.

| 측면 | Implicit Schema | Explicit Schema |
|------|-----------------|------------------|
| 형식 | JSON · 자유 구조 | Avro · Protobuf · JSON Schema |
| 검증 시점 | 런타임 | 컴파일 또는 발행 시점 |
| 변경 감지 | 사고 후 | 변경 시 즉시 |
| 코드 생성 | 수동 | 자동 |
| 호환성 검증 | 불가 | 자동 (Forward · Backward · Full) |

Implicit Schema 가 짧은 PoC 에서는 빠르지만, 운영 단계에서는 호환성 검증을 사람의 기억에 맡기는 셈이다. 사람의 기억은 결국 깨진다.

## 3. Schema Definition Best Practices

> 스키마 설계의 결정이 향후 6 개월~수 년의 호환성 부담을 정한다.

SSOT §3 의 베스트 프랙티스는 다음 다섯 가지로 요약된다.

1. 필드 이름은 도메인 어휘로 — `data1`, `payload` 같은 추상 이름 금지.
2. Optional · Default 의 의미를 명확히 — Optional 은 "있을 수도 없을 수도", Default 는 "없으면 이 값".
3. Enum 은 신중히 — 미래에 추가될 값을 호환성 정책에 포함시킨다.
4. Nested 구조 vs Flatten — Consumer 의 사용 패턴에 따라.
5. Timestamp 는 항상 ISO 8601 또는 epoch millis 로 통일.

여기서 질문 하나 — Enum 에 새 값을 추가하면 왜 호환성이 깨지나? Backward Compatibility 를 적용하면 옛 Consumer 가 새 Enum 값을 모르고 unknown 으로 처리할 수 있다. Forward Compatibility 라면 새 Consumer 가 옛 데이터의 Enum 값을 처리해야 한다. 두 방향의 호환성 정책을 미리 정하지 않으면 추가가 위험해진다.

## 4. Breaking Schema Changes

> "이건 깨는 변경이다" 의 정의가 합의되어 있어야, 그 합의 위에서 변경 절차가 가능하다.

SSOT §5 가 정의하는 breaking change 의 분류는 다음과 같다.

| 변경 | Forward 호환 | Backward 호환 | 비고 |
|------|--------------|---------------|------|
| 필드 추가 (Optional) | ✅ | ✅ | 안전 |
| 필드 추가 (Required) | ❌ | ❌ | 깨는 변경 |
| 필드 제거 | ❌ | ❌ | 깨는 변경 |
| 필드 타입 변경 | ❌ | ❌ | 깨는 변경 |
| 필드 이름 변경 | ❌ | ❌ | 깨는 변경 — 추가 + 제거 두 단계로 풀어야 |

깨는 변경을 해야 한다면 다음 둘 중 하나로 푼다.

1. Schema 버전 분기 — v1 토픽과 v2 토픽을 일정 기간 병행, 모든 Consumer 가 v2 로 이전한 뒤 v1 제거.
2. Compatibility Mode 완화 + 마이그레이션 — 모든 Consumer 와 사전 합의 후 정해진 시점에 동시 전환.

상세 구현 패턴은 `./02-05.이벤트 진화 다루기.md` 와 `../../04_messaging/02_MessageContract/01-06.Avro 스키마 진화 패턴.md` 가 다룬다.

## 5. Best Practice — Forward · Backward 호환

> "이 토픽의 호환성 정책은 무엇인가" 가 토픽마다 박제되어 있어야 한다.

Schema Registry 의 호환성 모드는 다음 다섯 가지다.

| 모드 | Producer 가 새 스키마로 바꿀 수 있는 변경 |
|------|-------------------------------------------|
| NONE | 무엇이든 |
| BACKWARD | 옛 Consumer 가 새 데이터를 읽을 수 있는 변경만 |
| FORWARD | 새 Consumer 가 옛 데이터를 읽을 수 있는 변경만 |
| FULL | BACKWARD + FORWARD 둘 다 |
| BACKWARD_TRANSITIVE / FORWARD_TRANSITIVE / FULL_TRANSITIVE | 한 단계 전이 아닌 모든 과거 버전과의 호환 |

기본 권장은 BACKWARD 다. 새 Producer 와 옛 Consumer 가 공존할 수 있어 점진적 배포가 가능하다. FULL 은 안전하지만 변경 자유도가 낮다. Trial-and-error 보다 토픽 신설 시 호환성 모드를 명시적으로 박제하는 것이 안전하다.
