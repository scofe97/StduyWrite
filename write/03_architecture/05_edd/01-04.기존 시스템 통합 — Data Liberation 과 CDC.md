# 기존 시스템 통합 — Data Liberation 과 CDC
---
> 레거시 DB 안에 갇힌 사실을 이벤트 스트림으로 풀어내는 결정 — 어떻게 풀 것인가의 선택이 향후 마이그레이션 자유도를 결정한다.

EDA 로의 전환은 그린필드에서 시작하지 않는다. 보통 이미 운영 중인 모놀리스 · 레거시 시스템의 데이터를 이벤트 스트림으로 옮기는 작업부터 시작한다. 본 문서는 그 옮김의 세 가지 방식과 선택 기준을 정리한다.

## 1. Data Liberation 의 동기

> 데이터가 한 시스템 안에 갇혀 있으면, 그 시스템 없이는 어떤 새 기능도 만들 수 없다.

SSOT §1 의 메시지는 단순하다. 레거시 DB 에 핵심 데이터가 갇혀 있으면 다음이 발생한다.

1. 새 서비스는 항상 레거시를 직접 조회해야 한다 — 결합도 증가.
2. 레거시의 부하가 새 기능의 성능 천장이 된다.
3. 레거시를 못 끄고 영구히 유지해야 한다.

Data Liberation 은 이 갇힘을 푼다. 레거시의 데이터 변경을 이벤트로 흘려보내, 다른 서비스가 자기 모델로 받아 쓸 수 있게 만든다. 결국 레거시는 한 토픽의 작성자일 뿐이고, 새 서비스들은 그 토픽의 소비자가 된다.

## 2. Outbox vs Trigger vs Log-based CDC

> 같은 "DB 변경을 이벤트로 흘려보낸다" 라도 세 방식의 비용·신뢰성·침투도가 다르다.

SSOT §2 가 다루는 세 패턴을 표로 정리하면 다음과 같다.

| 방식 | 동작 | 침투도 | 신뢰성 | 비용 |
|------|------|--------|--------|------|
| Outbox | 비즈니스 트랜잭션에서 outbox 테이블에 함께 기록, 별도 publisher 가 발행 | 코드 수정 필요 | 높음 (트랜잭션 보장) | 코드 변경 + 폴링 |
| Trigger 기반 CDC | DB 트리거로 변경 감지 후 별도 테이블에 기록 | DB 수정 필요 | 중간 | 트리거 부하 |
| Log-based CDC | DB 의 트랜잭션 로그를 직접 읽음 (Debezium 등) | 거의 없음 | 높음 | 인프라 추가 |

세 방식의 선택 기준은 다음과 같다.

1. 레거시 코드를 수정할 수 있고, 새 코드와의 일관성이 중요하면 — Outbox.
2. 코드 수정이 불가능하고, DB 변경은 가능하면 — Trigger 기반.
3. 코드·DB 수정이 모두 불가능하거나, 최소 침투를 원하면 — Log-based CDC.

여기서 질문 하나 — Outbox 가 가장 안전한데 왜 다른 방식이 존재하나? 침투도 때문이다. Outbox 는 모든 트랜잭션 코드에 outbox 쓰기를 추가해야 한다. 레거시가 수십만 줄이면 이 작업이 비현실적이다. Log-based CDC 는 그 비용을 0 에 가깝게 만든다.

## 3. 내부 데이터 모델 격리

> CDC 로 흘려보낸 이벤트가 곧 외부 계약이 되어서는 안 된다 — 내부 모델 변경이 외부를 깨뜨린다.

SSOT §4 가 강조하는 함정은 다음이다. CDC 가 DB 의 행 변경을 그대로 이벤트로 흘려보내면, 외부 Consumer 는 *DB 스키마* 를 계약으로 받는다. DB 스키마가 바뀌면 외부가 깨진다.

해결은 두 단계 변환이다.

```
[Legacy DB]  →  CDC raw event  →  [Translator]  →  도메인 이벤트  →  [Topic]
                                       ↑
                              내부 → 외부 모델 매핑
```

Translator 는 안티커럽션 레이어(`../04_ddd/01-04 §3`)의 EDA 판이다. 내부 변경을 외부 계약으로부터 격리한다.

## 4. Outbox 패턴의 스키마 호환성

> Outbox 가 가장 신뢰성 높은 이유는 비즈니스 트랜잭션과 이벤트 기록이 같은 트랜잭션에 묶이기 때문이다.

SSOT §5 의 Outbox 흐름은 다음과 같다.

1. 비즈니스 메서드가 도메인 변경을 한다.
2. 같은 트랜잭션에서 outbox 테이블에 이벤트를 기록한다.
3. 트랜잭션이 커밋되면 둘 다 영속화된다.
4. 별도 publisher 가 outbox 를 폴링해 메시지 브로커로 발행한다.
5. 발행 후 outbox row 를 표시(또는 삭제).

이 구조의 핵심은 "비즈니스 변경 없이 이벤트만 발행되거나, 이벤트 발행 없이 비즈니스만 변경되는 일이 없다" 는 점이다. Dual write 문제가 사라진다.

상세 구현은 `../../04_messaging/05_ConsistencyPattern/01-03.Outbox.md` 가 다룬다.

## 5. DDL 변경 처리

> CDC 의 가장 까다로운 부분은 DDL — 컬럼 추가·이름 변경·타입 변경 — 이다.

SSOT §7 의 DDL 처리 전략은 다음 셋이다.

1. Expand/Contract (`../04_ddd/03-03 §3`) 으로 점진 변경 — 새 컬럼을 추가하고 옛 컬럼을 한동안 유지, 모든 Consumer 가 새 컬럼을 보면 옛 컬럼 제거.
2. Translator 의 변환 규칙 수정 — DB 변경이 외부 계약을 깨지 않도록 변환을 조정.
3. Schema Registry 의 호환성 정책 활용 — 변경이 BACKWARD 호환되는지 자동 검증.

DDL 변경 시점이 항상 이벤트 스트림과 동기화되어야 한다. DDL 이 먼저 가고 CDC 가 따라가지 못하면, 일정 기간 발행되는 이벤트의 의미가 모호해진다.
