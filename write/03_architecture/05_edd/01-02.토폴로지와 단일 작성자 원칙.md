# 토폴로지와 단일 작성자 원칙
---
> 한 토픽에 한 작성자라는 원칙은 단순한 컨벤션이 아니라 "이 사실의 진실 공급원이 어디인가" 를 못박는 결정이다. 이 결정이 흔들리면 같은 사실이 여러 출처에서 나와 합의가 깨진다.

`01-01` 이 EDA 가 푸는 문제를 다뤘다면, 본 문서는 EDA 의 운영 가능성을 만드는 두 기본기 — 토폴로지 설계와 단일 작성자 원칙 — 를 정리한다.

## 1. 토폴로지란 무엇인가

> 토폴로지는 "이벤트가 어디서 만들어져 어디로 흘러 어떻게 변환되는가" 의 전체 그래프다.

SSOT §2 의 토폴로지는 다음 세 요소로 구성된다.

| 요소 | 역할 |
|------|------|
| Producer | 사실을 이벤트로 발행하는 주체 |
| Topic / Stream | 이벤트가 흐르는 경로 |
| Processor / Consumer | 이벤트를 받아 처리·변환·저장하는 주체 |

토폴로지를 그리지 않고 EDA 를 시작하면 발생하는 흔한 사고는 두 가지다. 한 토픽을 여러 출처가 동시에 쓰는 경합(§3 의 단일 작성자 위반), 그리고 같은 사실이 여러 토픽에 중복 발행되는 분기 — 둘 다 진실 공급원의 위치를 불분명하게 만든다.

## 2. Single Writer Principle

> 한 토픽은 한 마이크로서비스만 쓴다 — 이 규칙이 진실 공급원을 단일화한다.

SSOT §6 의 핵심 규칙은 다음과 같다. 한 토픽에 두 서비스가 동시에 쓰면 다음이 발생한다.

1. 충돌 — 같은 키에 두 출처가 다른 값을 보내면 무엇이 정답인지 알 수 없다.
2. 스키마 표류 — 두 출처가 독립적으로 스키마를 진화시키면 호환이 깨진다.
3. 책임 모호 — 사고 시 누가 원인 제공자인지 추적이 어렵다.

여기서 질문 하나 — 같은 사실을 여러 서비스가 알아야 한다면 어떻게 푸나? 답은 "한 서비스가 쓰고, 다른 서비스들이 읽는다" 다. 토픽은 입력에는 단일 작성자, 출력은 다중 소비자 구조를 갖는다.

## 3. The Contents of an Event

> 이벤트의 페이로드는 의도가 분명해야 한다 — Entity 의 현재 상태인가, 일어난 사건인가, 명령인가.

SSOT §3 이 다루는 이벤트 페이로드 종류는 다음과 같다.

| 종류 | 페이로드 | 의도 |
|------|----------|------|
| Entity Event | Entity 의 현재 상태 전체 | "이 Entity 의 현재 모습은 이렇다" |
| Event-Stream Event | 일어난 사건 + 변경된 부분 | "이런 일이 일어났다" |
| Command Event | 외부 서비스에 요청 | "이걸 해줘" — 일반적으로 권장하지 않음 |

Entity Event 는 materialized state 패턴(`§4`)과 잘 맞는다. Event-Stream Event 는 Event Sourcing(`./02-02`)과 잘 맞는다. Command Event 는 두 서비스 사이의 결합을 만들기 쉬워 신중히 쓴다.

## 4. Materialized State from Entity Events

> Entity Event 를 받는 쪽이 자기 모델로 상태를 재구성해 두면, 매번 원본을 묻지 않고도 일관된 뷰를 갖는다.

SSOT §4 의 materialized state 패턴은 다음 흐름이다.

```
[Producer 서비스]  →  Entity Event 발행  →  [Topic]
                                              ↓
                                    [Consumer A] 자기 DB 에 materialized view
                                    [Consumer B] 자기 DB 에 다른 형태 view
```

이 구조의 효과는 다음 셋이다.

1. Consumer 가 Producer 의 가용성에 의존하지 않는다.
2. Consumer 별로 자기 도메인에 맞는 형태로 뷰를 만들 수 있다.
3. 새 Consumer 추가가 기존 시스템에 영향을 주지 않는다.

비용은 결과적 일관성과 저장 중복이다. 이 비용을 받아들일 수 있는 도메인에서 EDA 의 자율성이 실재한다.

## 5. Event Data Definitions and Schemas

> 토폴로지의 안정성은 스키마의 안정성이다 — 스키마가 자주 깨지면 토폴로지의 모든 노드가 함께 흔들린다.

스키마의 안정성 확보는 다음 두 차원이다. 첫째, 명시적 스키마(Explicit Schema) — JSON 의 약속이 아닌 Avro · Protobuf 같은 정의된 형식. 둘째, 호환성 규칙 — 새 버전이 옛 Consumer 를 깨지 않도록.
