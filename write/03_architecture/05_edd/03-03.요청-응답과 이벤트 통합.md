# 요청-응답과 이벤트 통합
---
> 외부에는 동기 UX, 내부에는 비동기 이벤트 — 둘을 한 시스템에서 잇는 패턴들. 사용자는 즉시 응답을 기대하지만, 그 응답을 만드는 내부 작업은 비동기로 흐른다.

## 1. 외부 이벤트 처리

> 사용자 요청이 시스템에 들어오면, 그것을 이벤트로 변환해 내부 워크플로우로 흘려보낸다.

SSOT §1 의 외부 이벤트 처리 흐름은 다음과 같다.

```
[Browser]  →  HTTP POST  →  [API Gateway]  →  이벤트 발행  →  [Internal Topic]
                                  ↓
                          202 Accepted + 처리 ID 반환
```

API Gateway 는 즉시 202 Accepted 와 처리 ID 를 반환한다. 사용자는 후속 조회(`§2`)로 결과를 본다. 내부에서는 그 사이 워크플로우가 비동기로 진행된다.

이 분리의 효과는 셋이다.

1. API 가 내부 처리 시간에 묶이지 않는다 — 응답이 항상 빠르다.
2. 내부 워크플로우의 실패가 API 의 timeout 으로 노출되지 않는다.
3. 같은 요청이 여러 내부 모듈로 자연스럽게 흩어진다.

## 2. 상태 서빙 패턴

> 비동기 처리의 결과를 사용자에게 어떻게 보여줄 것인가 — 폴링·콜백·웹소켓의 셋이 있다.

SSOT §2 의 상태 서빙 패턴은 다음과 같다.

| 패턴 | 동작 | 적합한 상황 |
|------|------|------------|
| Polling | 클라이언트가 주기적으로 상태 조회 | 처리 시간이 짧음, 단순 구현 |
| Callback | 처리 완료 시 클라이언트에 알림 (Webhook) | 외부 시스템 간 통합 |
| WebSocket / SSE | 서버 → 클라이언트 푸시 | 실시간 UX 요구 |

가장 단순한 것은 Polling 이다. `GET /jobs/{id}` 로 상태를 조회하고, 완료되면 결과를 반환한다. 상세 패턴은 `05-03.202 Accepted + Polling 패턴.md` 가 다룬다.

WebSocket 은 사용자 경험이 가장 좋지만 운영 복잡도가 높다. 연결 유지·재연결·상태 동기화의 부담이 추가된다.

## 3. 워크플로우 내 요청 처리

> 비동기 워크플로우 중간에 동기 요청이 들어왔을 때 — 그 워크플로우의 상태를 동기적으로 반환할 수 있어야 한다.

SSOT §3 의 흐름 예시는 다음과 같다.

```
[User]  →  GET /workflow/{id}  →  [Query Service]
                                       ↓ 읽기 view 조회
                                  현재 상태 반환
```

Query Service 는 워크플로우의 read view (`./02-01.CQRS.md`)를 본다. 비동기 워크플로우의 각 단계가 이벤트로 발행되고, Query Service 는 그 이벤트를 받아 자기 view 를 업데이트한다.

이 구조에서 사용자는 동기 응답을 받지만, 응답의 데이터는 비동기로 모은 결과다.

## 4. 신문 출판 워크플로우 예시 (승인 패턴)

> 사용자가 액션을 일으키고, 다른 사용자의 승인을 기다리는 패턴 — 비동기와 동기의 자연스러운 만남.

SSOT §4 의 신문 출판 예시는 다음 흐름이다.

1. 기자가 기사 제출 → `ArticleSubmitted` 이벤트.
2. 편집자에게 알림 → 편집자가 검토.
3. 편집자 승인 → `ArticleApproved` 이벤트.
4. 출판 시스템이 자동 발행 → `ArticlePublished` 이벤트.

각 단계가 비동기지만, 각 사용자(기자·편집자)는 자기 화면에서 동기적으로 작업한다. 승인 패턴은 이 두 흐름을 자연스럽게 잇는 구조다.

여기서 질문 하나 — 승인 도중 기사가 수정되면? `ArticleModified` 이벤트가 발행되고, 워크플로우는 다시 처음부터 시작하거나 적절한 단계로 되돌아간다. 이 로직은 Orchestrator(`./03-02 §3`) 가 관리한다.

## 5. 마이크로프론트엔드와 BFF

> 프론트엔드도 도메인 경계를 따라 나뉘면 — 마이크로서비스의 자율성이 사용자 인터페이스까지 닿는다.

SSOT §5 의 마이크로프론트엔드는 다음 개념이다. 한 페이지가 여러 팀이 만든 컴포넌트의 조합이고, 각 컴포넌트가 자기 백엔드 서비스와 직접 통신한다.

이 구조의 효과는 셋이다.

1. 팀이 프론트엔드까지 자율적으로 개발·배포.
2. 한 컴포넌트의 장애가 페이지 전체를 죽이지 않음.
3. 도메인 경계가 코드뿐 아니라 사용자 경험에까지 일관되게 박힘.

BFF(Backend for Frontend)는 마이크로프론트엔드의 자연스러운 동반자다. 각 프론트엔드 컴포넌트가 자기 BFF 와 통신하고, BFF 가 백엔드 마이크로서비스들과의 통신을 조율한다.
