# Bounded Context 와 Context Map
---
> 같은 단어가 두 팀에서 다른 의미일 때, 그 차이를 모델 안에서 인정하고 경계와 관계를 그려 두지 않으면 통합 시점에 모든 가정이 깨진다. Bounded Context 는 "여기서 이 단어는 이 뜻이다" 의 선언이고, Context Map 은 그 선언들 사이의 정치적 관계를 가시화한다.

`01-02 §2` 에서 런 관리 예시의 4 개 Context(run·battle·catalog·progression)를 나눴다. 본 문서는 그 분할을 가능하게 만드는 두 가지 도구 — Bounded Context 의 정의와 Context Map 의 7 가지 통신 패턴 — 을 정리한다.

## 1. Bounded Context 가 푸는 문제

> 하나의 거대 모델 대신 비즈니스 의도에 맞춰 여러 작은 모델을 두는 결정의 근거.

DDD 가 "거대한 Customer 객체에 청구 데이터·개인 데이터·배송 주소·본사 주소를 모두 넣지 마라" 고 말하는 이유는 단순하다. 물류 영역을 수정할 때 판매와 구매 영역도 함께 흔들리기 때문이다. 같은 단어가 여러 의미를 동시에 가지면 변경 비용이 그래프의 차수만큼 증가한다.

Bounded Context 는 "이 경계 안에서 Customer 는 이 뜻이다" 를 명시한다. 다른 Context 에서 같은 이름이 다른 뜻이면 그건 다른 객체다. 단어를 통일하는 것이 아니라 단어의 사용 범위를 통일한다.

여기서 질문 하나 — 그렇다면 경계는 어떻게 정하는가? 비즈니스 능력(business capability)·팀 구성·언어 변화의 지점 세 가지를 함께 본다. 자세한 분류 기준은 `01-03 §2~3` 의 sub-domain 신호와 같다.

## 2. Bounded Context 의 경계 식별

> 언어 차이·팀 경계·트랜잭션 요구의 세 신호가 한 자리에 모이는 지점이 자연스러운 경계다.

`01-03` 이 sub-domain 분류 신호를 다뤘다면, 본 절은 그 분류를 Context 경계로 옮기는 절차다. 다음 네 단계로 진행한다.

1. 같은 단어가 여러 의미로 쓰이는 지점을 도메인 전문가와 함께 식별한다.
2. 각 의미별로 책임이 분명한 모델을 그린다. 같은 이름이라도 모델이 다르면 다른 Context 다.
3. 트랜잭션 요구·변경 빈도·저장 전략이 함께 갈리면 경계 후보를 확정한다.
4. Context Map(`§3`) 에서 다른 Context 와의 관계를 정의한다.

런 관리 예시에서 `Card` 는 catalog Context 에서는 정의 데이터(불변 카탈로그)고, battle Context 에서는 매 턴 상태가 바뀌는 인스턴스다. 두 의미를 한 모델에 담으면 catalog 의 영구 저장 요구와 battle 의 메모리 처리 요구가 충돌한다. 경계를 가르는 것이 두 책임을 자유롭게 만든다.

## 3. Context Map 의 7 가지 통신 패턴

> 두 Context 가 만나는 방식은 "누가 누구의 모델을 따르는가" 에 따라 일곱 가지로 갈린다.

Evans 가 제시한 통신 패턴은 권력 관계와 모델 결합도의 두 축으로 정리된다. 같은 통합이라도 어느 패턴을 선택하느냐에 따라 향후 변경 비용이 결정된다.

| 패턴 | 권력 관계 | 모델 결합도 | 적용 상황 |
|------|-----------|------------|-----------|
| Partnership | 대등 | 높음 | 두 팀이 함께 진화시키는 공동 책임 영역 |
| Shared Kernel | 대등 | 매우 높음 | 두 Context 가 공유하는 작은 공통 모델 |
| Customer-Supplier | 상하 | 중간 | Upstream 이 Downstream 요구를 받아들임 |
| Conformist | 상하 | 높음 | Downstream 이 Upstream 모델을 그대로 따름 |
| Anticorruption Layer | 분리 | 낮음 | Downstream 이 자기 모델로 번역하는 방어층 |
| Open Host Service | 일대다 | 낮음 | Upstream 이 공개 프로토콜로 다수 Downstream 지원 |
| Published Language | 일대다 | 낮음 | OHS 위에서 표준화된 데이터 형식 합의 |

여기서 질문 하나 — 외부 결제 시스템과 통합할 때 어느 패턴이 맞을까? Anticorruption Layer 다. 그쪽 모델을 그대로 따라가면(Conformist) 결제 시스템의 도메인 어휘가 우리 도메인을 오염시킨다. 번역층을 둬서 외부 모델의 변경이 우리 도메인까지 닿지 않게 막는다.

## 4. 런 관리 예시에 적용

> 4 개 Context 가 어떤 관계로 만나는지 표로 박제해 두면, 새 기능이 어느 경계를 침범하는지 즉시 보인다.

`01-02 §2` 의 4 개 Context 를 `§3` 의 패턴으로 매핑하면 다음 관계가 된다.

| Upstream → Downstream | 관계 | 패턴 | 이유 |
|------------------------|------|------|------|
| catalog → battle | 데이터 공급 | Open Host Service | catalog 가 카드·유물 정의를 공개 형식으로 노출, battle 은 읽기 전용 |
| catalog → run | 데이터 공급 | Open Host Service | 동일. 런 보상 후보 선택용 |
| run → battle | 흐름 조정 | Customer-Supplier | run 이 battle 시작·종료의 요구를 결정 |
| battle → run | 결과 통지 | Domain Event | battle 결과를 run 이 비동기로 수신 |
| run → progression | 결과 통지 | Domain Event | run 종료가 progression 의 해금 트리거 |
| 외부 결제 → progression | 통합 | Anticorruption Layer | 결제 시스템 모델을 progression 모델로 번역 |

이 표가 박제되면 새 기능이 들어올 때마다 "이 통합은 어느 행에 해당하는가" 만 답하면 된다. 답이 없으면 새 행을 추가하고 패턴을 정한다. 패턴 없이 통합부터 시작하면 결합도가 자라난다.
