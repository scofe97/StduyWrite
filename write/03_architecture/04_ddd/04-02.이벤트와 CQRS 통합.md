# 이벤트와 CQRS 통합 — DDD 시점에서
---
> CQRS · Event Sourcing 의 구현 디테일은 `../05_edd/` 가 다룬다. 본 문서는 한 단계 위 — "왜 DDD 가 이 둘을 자연스럽게 요구하는가" 의 모델적 근거만 다룬다.

`02-01 §2` 에서 Aggregate 사이는 결과적 일관성으로 다룬다고 했다. 그 결과적 일관성을 표현하는 도구가 Domain Event 이고, 그 위에서 읽기·쓰기 모델을 분리하는 패턴이 CQRS 다.

## 1. Aggregate 가 이벤트를 만드는 이유

> "이 Aggregate 에 어떤 일이 일어났는가" 의 사실을 외부와 공유할 때, 이벤트는 그 사실의 표준 표현이다.

DDD 의 Aggregate 는 외부에 자기 내부 상태를 노출하지 않는다 (`02-01 §1`). 대신 Aggregate 의 메서드가 호출된 결과 "무슨 일이 일어났는가" 를 Domain Event 로 발행한다.

```java
// Run.java
public final class Run {
    private final List<DomainEvent> events = new ArrayList<>();

    public List<CardId> completeBattle(BattleId battleId, BattleResult result) {
        // ... 도메인 로직 ...
        events.add(new BattleCompleted(this.id, battleId, result));
        if (result.isVictory() && actProgress.isBossDefeated()) {
            events.add(new RunCleared(this.id, actProgress));
        }
        return rewardCandidates;
    }
}
```

이 구조의 효과는 셋이다.

1. Aggregate 내부 상태가 새지 않는다 — 외부는 이벤트의 페이로드만 본다.
2. 한 메서드 호출이 여러 사실을 발생시킬 수 있다 — 위 코드처럼 전투 완료와 런 클리어가 함께.
3. 이벤트가 외부 모듈·서비스의 진입점이 된다 — `Profile`, `Analytics`, `Notification` 모듈이 같은 이벤트를 구독한다.

## 2. Domain Event vs Integration Event

> 같은 "이벤트" 라는 단어지만, Bounded Context 안을 도는 이벤트와 Context 를 넘는 이벤트는 다른 종류다.

SSOT §7.4 의 구분은 다음과 같다.

| 측면 | Domain Event | Integration Event |
|------|--------------|---------------------|
| 범위 | 같은 Bounded Context 내 | Context 또는 서비스 간 |
| 페이로드 | 도메인 내부 어휘 그대로 | 안정적 공개 계약(`../05_edd/01-03`) |
| 진화 자유도 | 높음 (내부) | 낮음 (호환성 유지 필수) |
| 전달 매체 | 인메모리 publisher | 메시지 브로커(Kafka 등) |

흔한 실수는 둘을 같은 객체로 만드는 것이다. Domain Event 를 그대로 Kafka 로 흘려보내면, 내부 모델 변경이 외부 계약을 깨뜨린다. Domain Event 를 받아 Integration Event 로 *변환* 하는 어댑터(`../05_edd/03-01.Saga, Outbox, Request-Response Bridge.md` 의 Outbox 패턴)가 그래서 필요하다.

여기서 질문 하나 — 둘 다 만들면 코드 양이 늘어나는데 그만한 가치가 있나? Context 의 진화 속도를 외부 계약 변경 비용으로부터 분리한다는 가치다. 내부는 매일 바뀌어도 외부 계약은 분기 단위로만 바뀐다.

## 3. CQRS — 명령과 질의의 모델 분리

> 쓰기 모델은 불변식을 지키고, 읽기 모델은 화면을 만든다 — 두 책임이 같은 모델 안에서 충돌할 때 분리의 정당성이 생긴다.

SSOT §7.5 의 CQRS 는 다음 두 모델을 분리한다.

| 모델 | 책임 | 형태 |
|------|------|------|
| Command Model | 불변식 검증·상태 변경 | Aggregate (정규화·작은 단위) |
| Query Model | 화면·집계·검색 | View · Projection (역정규화·집계) |

같은 Aggregate 가 두 책임을 모두 지면 다음 충돌이 생긴다. 쓰기는 Aggregate 의 작은 경계가 좋지만, 읽기는 여러 Aggregate 의 정보를 한 화면에 모아 보여야 한다. 한 모델로 둘을 만족시키려면 Aggregate 가 부풀거나 N+1 쿼리가 폭증한다.

CQRS 의 핵심은 두 모델이 *다른 저장 구조* 를 가질 수 있다는 점이다. Command 는 JPA Aggregate, Query 는 별도 read DB 의 역정규화된 뷰 — 둘 사이는 이벤트로 동기화한다.

## 4. 읽기·쓰기 동기화의 비용

> CQRS 를 도입하는 순간 일관성은 결과적 일관성이 된다 — 이 비용을 받아들일 수 있는 도메인에서만 의미가 있다.

SSOT §7.6 이 다루는 동기화 전략은 다음 세 가지다.

1. 동기 갱신 — 쓰기 트랜잭션 안에서 read view 도 함께 갱신. 트랜잭션 비용이 커지지만 일관성 강함.
2. 비동기 갱신 (이벤트) — 쓰기 후 이벤트 발행, 비동기 핸들러가 view 갱신. 일관성 약하지만 확장성 높음.
3. 주문형 재구성 — view 를 미리 만들지 않고 조회 시점에 이벤트를 재생해 만든다. Event Sourcing 과 결합.

런 관리 예시에서 "전투 결과 화면" 은 1 번이 자연스럽다 (즉시 보여야 함). "월간 통계" 는 2 번이 자연스럽다 (몇 분 지연 허용). "전체 리플레이" 는 3 번이 맞다 (모든 이벤트를 시간순으로 재생).

## 5. Event Sourcing 으로 확장

> 상태가 아닌 이벤트의 시퀀스를 진실로 삼는 결정 — DDD 의 자연스러운 종착지지만 모든 도메인에 필요하지는 않다.

SSOT §7.7 의 Event Sourcing 은 §1 의 "Aggregate 가 이벤트를 발행한다" 를 끝까지 밀어붙인 형태다. Aggregate 의 현재 상태를 저장하지 않고, 그 Aggregate 에 일어난 모든 이벤트를 저장한다. 현재 상태는 이벤트들을 재생해 복원한다.

장점은 셋이다.

1. 완전한 감사 로그 — 모든 변경의 이유가 이벤트로 남는다.
2. 시간 여행 — 과거 임의 시점의 상태를 복원할 수 있다.
3. 새 read view 생성이 자유롭다 — 이벤트를 재생해 만든다.

비용은 셋이다.

1. 이벤트 스키마 진화의 부담 — 과거 이벤트와의 호환을 평생 유지해야 한다.
2. 현재 상태 조회의 추가 비용 — 스냅샷 전략이 필요해진다.
3. 학습 곡선 — 팀이 익숙해지는 데 시간이 든다.

런 관리 예시에서 Event Sourcing 이 자연스러운 영역은 `battle` 이다. 매 턴이 곧 이벤트이고, 리플레이가 자연스러운 도메인 요구다. 반대로 `progression` 같은 상태 중심 영역에는 과한 선택이다. 모든 영역에 Event Sourcing 을 도입하는 것이 아니라, 도메인별로 가치-비용을 따져 부분 적용한다.
