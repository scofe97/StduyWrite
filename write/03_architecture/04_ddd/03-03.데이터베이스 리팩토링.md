# 데이터베이스 리팩토링
---
> 모델은 진화하는데 스키마는 굳어 있다 — DB 를 도메인 변화에 맞춰 옮기는 작업은 코드 리팩토링보다 위험하다. 잘못된 단계는 운영 중단이나 데이터 손실로 직결된다.

`03-01` 이 코드 리팩토링의 원칙을 다뤘다면, 본 문서는 그 원칙을 스키마에 적용할 때의 추가 안전망을 다룬다.

## 1. 왜 DB 리팩토링이 필요한가

> 코드는 매주 바뀌어도 스키마가 고정되면 도메인 모델은 결국 스키마를 따라가게 된다 — 꼬리가 개를 흔드는 상황이다.

DB 리팩토링이 미뤄지는 흔한 이유는 두 가지다.

1. 한 번에 큰 마이그레이션을 하려고 한다 — 위험이 커서 미뤄진다.
2. 운영 중단 없이 바꿀 방법을 모른다 — 시도조차 하지 않는다.

두 이유 모두 "점진적·무중단 리팩토링" 의 패턴을 알면 풀린다. SSOT §8.2 가 제시하는 진화적 데이터베이스 개발(Evolutionary Database Development)이 그것이다.

## 2. 테이블 소유권

> 한 테이블은 한 모듈만 쓴다 — 이 단순한 규칙이 마이그레이션의 자유도를 결정한다.

SSOT §8.3 의 핵심은 "테이블에 명시적 소유자를 둔다" 다. 여러 모듈이 같은 테이블을 직접 읽고 쓰면 한쪽의 스키마 변경이 다른 쪽을 깨뜨린다. 다음 절차로 소유권을 정리한다.

1. 각 테이블의 현재 사용자를 모두 찾아낸다 — `grep` · 정적 분석 · 로그 분석을 함께 쓴다.
2. 그 테이블의 자연스러운 소유 모듈을 정한다 (도메인 의미상).
3. 비소유 모듈은 직접 접근 대신 소유 모듈의 인터페이스를 거치도록 바꾼다.

소유권이 정해진 뒤에야 §3 의 Expand/Contract 같은 무중단 리팩토링이 가능해진다. 공유 테이블 위에서 무중단 변경은 거의 불가능하다.

## 3. Expand/Contract 패턴

> 한 번에 바꾸지 않는다 — 새 컬럼·새 테이블을 만들어 옆에서 채운 뒤 옛 것을 제거한다.

SSOT §8.4 의 Expand/Contract 패턴은 무중단 마이그레이션의 표준이다. 다섯 단계로 진행한다.

1. Expand — 새 컬럼·테이블을 만든다. 기존 것과 공존한다.
2. Dual Write — 쓰기는 두 곳 모두에 한다. 읽기는 여전히 옛 것에서.
3. Backfill — 기존 데이터를 새 위치로 복사한다.
4. Switch Read — 읽기를 새 위치로 옮긴다. 쓰기는 여전히 양쪽.
5. Contract — 옛 위치에 대한 쓰기 중단 후 제거.

각 단계는 독립적으로 롤백 가능하다. 4 단계에서 문제가 발견되면 읽기를 다시 옛 위치로 돌리고 5 단계로 가지 않는다.

여기서 질문 하나 — Dual Write 중에 두 위치의 데이터가 어긋나면? 두 가지 방어가 필요하다. 첫째, 쓰기를 한 트랜잭션 안에서 묶거나 Outbox(`../05_edd/01-04.기존 시스템 통합 — Data Liberation 과 CDC.md`)로 이중 쓰기를 풀어낸다. 둘째, 검증 쿼리를 주기적으로 돌려 일치 여부를 확인한다.

## 4. 이벤트 기반 데이터 동기화

> 모듈 간 데이터 공유를 직접 조인이 아닌 이벤트로 풀면, DB 리팩토링의 자유도가 한 차원 올라간다.

SSOT §8.6 이 권하는 방향은 다음과 같다. 모듈 A 의 데이터가 모듈 B 에 필요하면 직접 테이블을 읽지 말고 이벤트로 받아 B 가 자기 모델로 복제·캐시한다. 이 구조의 장점은 셋이다.

1. A 의 스키마가 바뀌어도 B 는 영향받지 않는다 — 이벤트 계약만 유지하면 된다.
2. B 의 읽기가 A 의 부하에 영향을 주지 않는다.
3. A 와 B 가 다른 저장 기술을 선택할 수 있다.

비용은 결과적 일관성이다. B 의 데이터는 A 의 변경 후 약간의 지연을 갖는다. 이 비용을 받아들일 수 있는 도메인에서는 이 구조가 DB 리팩토링을 훨씬 쉽게 만든다.

## 5. 분산 데이터 접근

> 마이크로서비스로 가기 전에, 모놀리스 안에서 이미 분산 접근 패턴을 연습해야 한다.

SSOT §8.7 의 분산 데이터 접근 패턴은 §4 의 이벤트 동기화를 일반화한 것이다. CQRS(`../05_edd/02-01.CQRS.md`)·Materialized View · API Composition 이 모두 같은 줄기다.

모놀리스 단계에서 다음 셋을 미리 적용해 두면 마이크로서비스 전환 시점에 데이터 분리가 훨씬 가볍다.

1. 모듈 간 직접 조인 금지 — 인터페이스나 이벤트로만 접근.
2. 모듈별 스키마 네임스페이스 분리 — 같은 DB 라도 스키마를 가른다.
3. 결과적 일관성에 대한 사용자·운영자 이해 확보 — 일관성 대시보드·복구 절차 준비.
