# AI 코딩 시대의 DDD
---
> 생성형 코딩 환경에서 DDD 의 가치는 더 커진다. AI 는 컨텍스트 안의 패턴을 모방해 코드를 만든다 — 그 패턴이 도메인을 잘 표현하면 AI 의 출력도 도메인을 잘 표현하고, 패턴이 어긋나 있으면 AI 가 그 어긋남을 증폭한다.

SSOT 13장이 제시하는 네 원칙은 다음과 같다. 각 원칙이 DDD 의 기존 도구를 AI 워크플로우에 어떻게 옮기는지 정리한다.

## 1. 구조가 패턴이 된다

> AI 는 "지금 보고 있는 코드와 비슷한 코드" 를 만든다 — 기준 패턴 하나가 그 카테고리 전체의 품질을 결정한다.

AI 에게 "Order 도메인을 만들어줘" 라고 부탁하면, AI 는 자기 학습 데이터의 평균적인 Order 코드를 만든다. 평균은 보통 진흙 덩어리에 가깝다. 반면 AI 에게 "이 프로젝트의 Run Aggregate 를 참고해서 Battle Aggregate 를 만들어줘" 라고 하면, AI 는 Run 의 구조를 모방해 Battle 을 만든다.

이게 의미하는 것은 "잘 만든 Aggregate 하나" 가 그 프로젝트 전체의 Aggregate 품질의 천장이 된다는 점이다. 첫 한 개를 SKILL · DDD 원칙대로 만들어 두면, 나머지는 AI 가 그 패턴을 복제한다.

런 관리 예시에서 기준 패턴이 되는 파일은 다음과 같다.

| 카테고리 | 기준 파일 | 다음 추가 시 참조 |
|----------|----------|-------------------|
| Aggregate Root | `Run.java` | `Battle`, `Profile` |
| Value Object (record) | `Hp.java` | `Energy`, `Gold`, `CardCost` |
| Domain Service | `RewardPolicy.java` | `EnemyIntentPolicy`, `CardDraftPolicy` |
| Repository 인터페이스 | `RunRepository.java` | `BattleRepository`, `ProfileRepository` |

## 2. 도메인 단위로 나누면 AI 품질이 올라간다

> AI 의 컨텍스트 윈도우는 유한하다 — 한 번에 전체 코드베이스를 못 본다. 그래서 잘 나뉜 Bounded Context 가 AI 의 작업 단위가 된다.

`01-04` 의 Bounded Context 가 AI 시대에 갖는 추가 가치는 컨텍스트 윈도우의 자연스러운 단위가 된다는 점이다. AI 에게 "battle 모듈에서 새 카드 효과를 추가해줘" 라고 하면, AI 는 battle 디렉토리만 보면 된다. catalog 의 사소한 변경이 AI 의 추론에 노이즈로 끼지 않는다.

여기서 질문 하나 — 그러면 모놀리스에서도 디렉토리 분리만으로 충분한가? 그렇다. 마이크로서비스 단위까지 갈 필요는 없다. 디렉토리·패키지 경계만으로 AI 의 작업 단위를 만들 수 있다. 다만 그 경계가 도메인 의미를 정확히 따라야 한다.

## 3. 코드 이름 = 비즈니스 언어

> 유비쿼터스 언어가 코드에 박혀 있을수록 AI 가 도메인 어휘로 일한다.

`01-01` 의 유비쿼터스 언어가 AI 시대에 더 중요해지는 이유는 단순하다. AI 가 코드를 만들 때 변수명·메서드명을 기준 패턴에서 모방한다. 기존 코드가 `data1`, `proc`, `handleStuff` 같은 추상적 이름을 쓰면, AI 도 비슷한 추상적 이름을 만든다. `Run`, `BattleResult`, `RewardCandidates` 같은 도메인 이름을 쓰면 AI 의 출력도 도메인 이름을 갖는다.

검증 방법도 간단하다. AI 가 만든 코드를 도메인 전문가에게 보여줬을 때 그가 단어를 알아보면 합격이다. 모르면 유비쿼터스 언어가 코드에 박히지 않았다는 신호다.

## 4. Hook · Trigger + 도메인 = 자동 컨텍스트 로딩

> AI 에이전트가 도메인 작업을 시작할 때 자동으로 관련 도메인 문서·코드를 로드하게 만든다.

SSOT §13 의 4 번째 원칙은 도구 차원이다. AI 에이전트가 "Battle 관련 작업" 을 시작하면 자동으로 다음을 로드하게 한다.

1. `Battle.java` Aggregate 와 그 안의 메서드 시그니처.
2. `BattleRepository` 인터페이스.
3. 그 Context 의 Domain Event 목록.
4. 관련 학습 문서 (`02-01.Aggregate 설계 규칙.md`).

Claude Code · GitHub Copilot 같은 도구에서 이 자동 로딩은 hook · settings · skill 정의로 구현된다. 도메인 단위로 hook 을 묶으면, AI 가 그 도메인의 어휘·패턴·제약을 매번 처음부터 추론할 필요가 없어진다.

## 5. 네 원칙의 통합 효과

> 잘 만든 기준 패턴 하나 + 명확한 Context 경계 + 유비쿼터스 언어 + 자동 컨텍스트 로딩 = AI 가 도메인 일관성을 유지하며 일한다.

네 원칙이 따로 적용되면 효과가 약하다. 함께 적용될 때 다음과 같은 흐름이 만들어진다.

1. AI 가 새 기능을 받는다.
2. Hook 이 해당 Context 의 기준 파일·문서를 자동 로드.
3. AI 가 기준 패턴을 모방해 새 코드를 만든다.
4. 새 코드는 유비쿼터스 언어를 그대로 사용.
5. Context 경계 안에서만 작동 — 다른 Context 를 깨지 않는다.

이 흐름이 안정화되면 도메인 일관성이 사람의 코드 리뷰에 의존하지 않게 된다. AI 의 출력이 처음부터 일관성을 갖는다.
