# ADR과 Spring Boot 아키텍처 의사결정
---
> Architecture Decision Record(ADR)는 설계 결론보다 설계 이유를 남기는 문서다. 팀이 시간이 지나도 같은 맥락을 재구성할 수 있게 해준다.
>
> - context
> - decision
> - consequences
> - revisit trigger

## 1. 왜 ADR이 필요한가

아키텍처 결정은 대부분 트레이드오프다. 시간이 지나면 팀은 “왜 서비스로 안 쪼갰지”, “왜 JPA를 유지했지”, “왜 Outbox를 넣었지”를 다시 묻게 된다.

이때 코드만으로는 이유를 복원하기 어렵다. ADR은 특정 시점의 제약과 판단 근거를 남겨서, 같은 논쟁을 반복하지 않게 도와준다.

## 2. ADR의 기본 형식

좋은 ADR은 길 필요가 없다. 다음 네 가지면 충분하다:

- Context: 어떤 문제를 풀고 있었는가
- Decision: 무엇을 선택했는가
- Consequences: 무엇을 얻고 무엇을 감수하는가
- Revisit Trigger: 어떤 조건이 생기면 다시 보나

## 3. 예시 프로젝트의 ADR 후보

카드게임 런 관리 서비스에서는 다음 주제가 ADR 대상이 되기 쉽다:

- 핵심 게임 로직을 모듈러 모놀리스로 유지한다
- 전투 결과 외부 전파는 Outbox 패턴을 사용한다
- 리플레이 요구가 커지기 전까지는 이벤트 소싱을 전면 도입하지 않는다
- `run`과 `battle`은 같은 배포 단위에 둔다

이런 결정은 모두 시간이 지나면 재논의될 수 있으므로, 초기 근거를 남길 가치가 있다.

## 4. Spring Boot 프로젝트에서 ADR을 어디에 연결할까

ADR은 별도 문서 폴더에 모아도 되고, 모듈 README와 함께 둘 수도 있다. 중요한 것은 코드 구조와 문서가 서로 어긋나지 않게 유지하는 것이다.

예를 들어 `battle` 모듈을 왜 독립 서비스로 분리하지 않았는지 ADR이 있다면, 새로운 팀원이 구조를 보고도 납득할 수 있다.

## 5. 간단한 ADR 예시

다음은 이 시리즈의 기본 결정을 ADR로 옮긴 예시다:

```markdown
# ADR-001: 핵심 게임플레이는 모듈러 모놀리스로 유지한다

## Context
전투 규칙은 변경이 잦고, run과 battle은 강하게 결합되어 있다.

## Decision
Spring Boot 4.x 단일 애플리케이션 안에서 `run`과 `battle` 모듈을 유지한다.

## Consequences
배포는 단순해지지만, 독립 배포성은 얻지 못한다.

## Revisit Trigger
팀이 분리되고, battle 부하가 독립적으로 커질 때 재검토한다.
```

## 6. ADR은 결정의 묘비가 아니라 살아 있는 기록이다

ADR은 한 번 쓰고 끝내는 문서가 아니다. 조건이 바뀌면 새 ADR로 대체하거나, 이전 결정을 superseded 상태로 남겨야 한다.

즉 ADR의 목적은 “우리가 항상 옳았다”가 아니라 “그때는 왜 그렇게 판단했는가”를 남기는 것이다.

## 7. 실무 결론

이 시리즈에서 다루는 아키텍처 선택은 대부분 ADR 대상으로 삼을 수 있다. 특히 모놀리스 유지, 이벤트 기반 확장 범위, CQRS 도입 시점, 이벤트 소싱 도입 여부는 나중에 다시 검토될 가능성이 높다.

Spring Boot 프로젝트가 커질수록 기술 선택 자체보다 선택 근거를 남기는 습관이 중요해진다. ADR은 그 습관을 가장 저렴하게 만드는 도구다.
