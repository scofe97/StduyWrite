# 소프트웨어 아키텍처와 품질 속성
---
> 아키텍처는 단순한 디렉토리 구조가 아니라 시스템의 품질 속성을 지키기 위한 제약의 집합이다.
>
> - 성능
> - 변경 용이성
> - 테스트 가능성
> - 운영 가능성



## 1. 아키텍처를 왜 먼저 정하는가

> 아키텍처는 “코드를 어디에 둘까”보다 “어떤 변경을 싸게 만들까”를 결정한다. 같은 기능이라도 어떤 구조를 택하느냐에 따라 배포 속도, 장애 영향 범위, 테스트 비용이 크게 달라진다.

런 관리 서비스 예시에서는 전투 규칙 변경이 잦고, 카드 효과가 계속 추가된다. 이때 아키텍처의 핵심 목표는 새 카드와 새 적을 추가할 때 기존 코드를 최소한만 건드리게 만드는 것이다.

## 2. 품질 속성은 무엇을 의미하는가

품질 속성은 기능 요구사항 뒤에 숨어 있는 비기능 목표다. “전투를 진행한다”가 기능이라면, “전투 규칙을 빠르게 바꿀 수 있다”와 “재현 가능한 전투 로그를 남긴다”는 품질 속성이다.

런 관리 서비스에서 중요한 품질 속성은 다음과 같다:

| 품질 속성 | 의미 | 게임 도메인 예시 | 설계 영향 |
|-----------|------|------------------|----------|
| 변경 용이성 | 규칙을 쉽게 고친다 | 카드 효과 밸런스 조정 | 도메인 로직을 외부 입출력과 분리 |
| 테스트 가능성 | 규칙을 자동 검증한다 | 특정 전투 결과 재현 | 순수 도메인 객체와 Fixture 중심 테스트 |
| 관측 가능성 | 운영 중 상태를 파악한다 | 특정 런에서 오류 재현 | 이벤트 로그, trace, metric 설계 |
| 성능 | 응답 시간을 유지한다 | 턴 종료 후 빠른 결과 반환 | 동기/비동기 경계와 캐시 전략 |

## 3. 구조는 품질 속성의 우선순위를 반영한다

모든 품질 속성을 동시에 최대로 만족시키기는 어렵다. 응답 시간 최적화와 단순성은 잘 맞을 때도 있지만, 완전한 재현 가능성과 단순성은 종종 충돌한다.

예를 들어 이벤트 소싱을 도입하면 리플레이와 감사 추적은 쉬워진다. 반면 단순 CRUD보다 학습 비용과 운영 복잡도는 높아진다. 그래서 아키텍처는 “무엇을 더 중요하게 볼 것인가”를 명시적으로 선택해야 한다.

## 4. 모놀리스부터 시작하는 이유

Spring Boot 프로젝트에서 기본 선택은 모듈러 모놀리스가 합리적이다. 런 관리, 전투, 카드 카탈로그, 분석 적재를 한 저장소와 한 배포 단위 안에 두되, 코드 수준의 경계를 먼저 명확히 한다.

초기부터 서비스를 쪼개면 네트워크 장애, 분산 트랜잭션, 배포 파이프라인 복잡도가 먼저 들어온다. 반대로 모듈러 모놀리스는 도메인 경계를 훈련하기 좋고, 나중에 분리할 때도 기준점이 남는다.

## 5. 아키텍처를 문서로 남기는 법

좋은 아키텍처는 머릿속 감이 아니라 반복 가능한 규칙으로 남아야 한다. 특히 팀이 커질수록 “왜 이렇게 나눴는가”를 설명할 문서가 중요해진다.

아래처럼 간단한 규칙 세트만 있어도 효과가 크다:

- 전투 규칙 엔진은 Spring 의존성 없이 순수 Java로 유지한다.
- 외부 시스템 호출은 application 계층이 직접 하지 않고 port를 통해 수행한다.
- 상태 변경의 근거가 중요한 흐름은 도메인 이벤트를 남긴다.

## 6. 런 관리 서비스에 적용한 결론

이 시리즈의 예시 프로젝트에서는 변경 용이성과 테스트 가능성을 가장 우선한다. 그래서 카드 규칙, 적 의도, 보상 계산은 도메인 중심으로 두고, REST API와 DB 매핑은 바깥 계층으로 밀어낸다.

이 결정은 모든 후속 문서의 출발점이다. 클린 아키텍처, 헥사고날, DDD, 이벤트 소싱은 모두 같은 목표를 다른 방식으로 정리하는 도구로 읽어야 한다.
