# 품질 속성

소프트웨어 아키텍쳐의 가치를 평가하는 핵심 기준은 바로 품질 속성(quality attributes)이다.

1장에서 소프트웨어 아키텍쳐가 시스템의 구조와 비즈니스 목표를 연결하는 역할을 한다는 점을 살펴봤다. 특히 품질 속성은 소프트웨어가 단순히 기능을 수행하는 것을 넘어 안전성, 확장성, 유지보수성 같은 비기능적 요구사항을 충족하며 비즈니스 성공에 기여한다.

- 이러한 품질 속성은 개발, 배포, 유지보수, 확장에 이르는 소프트웨어의 생명 주기 전반에 걸쳐 중요한 역할을 한다.



## 품질 속성의 정의와 역할

품질 속성은 소프트웨어 시스템의 비기능적 특성을 정의하며, 시스템이 이해관계자의 요구를 얼마나 효과적으로 충족하는지를 평가하는 기준이다.

- 1장에서 언급했듯, 품질 속성은 기능 요구사항(시스템이 무엇을 해야하는지?)와 달리, 시스템이 어떻게 동작해야하는지를 나타낸다.
- ex) 온라인 도서 대출 시스템에서 '도서 대출 요청을 처리한다'가 기능 요구사항이라면 '요청 처리 시간이 100Ms 이하여야한다.'는 품질 속성에 해당한다.



품질 속성의 정의

품질 속성은 소프트웨어 성능, 확장성, 유지보수성, 테스트 용이성, 보안성 등. 시스템의 전반적인 품질을 결정하는 비기능적 요구사항이다.

ISO/IEC 25010은 소프트웨어 품질을 평가하기 위한 대표적 프레임워크로, 다음과 같은 주요 품질 속성을 포함한다.

- 기능 적합성: 시스템이 사용자의 업무 요구를 얼마나 정확하게 충족하는가?
- 성능 효율성: 시스템이 자원을 효율적으로 사용해 요구된 작업 수행
- 신뢰성: 장애 없이 안정적으로 동작하는 능력
- 유지보수성: 코드 수정과 유지보수가 얼마나 쉬운가
- 사용성: 사용자 경험이 직관적이고 효율적인가
- 보안성: 데이터와 시스템을 보호하는 능력
- 호환성 및 이식성: 다른 시스템이나 환경과의 상호작용 및 이식 용이성

그러나 이는 포괄적 가이드일 뿐, 모든 시스템에 반드시 적용해야하는 규범은 아니다. 개발하려는 시스템의 맥락에 따라 표준에 없는 품질 속성을 정의할수도 있다.

- ex) 실시간 채팅 애플리케이션에서는 '메시지 지연 시간 최소화' 같은 특정 품질 속성이 더 중요할 수 있다.



품질 속성의 역할

품질 속성은 소프트웨어 아키텍쳐 설계의 방향성을 결정하는 핵심 요ㅛ소다.

선택한 품질 속성에 따라 아키텍쳐의 구조가 달라진다. 

- ex) 테스트 기능성을 우선시하면 모듈화의 의존성 분리가 강조되고, 확장성을 중시하면 마이크로 서비스같은 구조가 적합할 수 있다.
- 이러한 결정은 소프트웨어 생명 주기 전반에 걸쳐 영향을 미친다.
  - 개발단계: 품질 속성은 설계 초기부터 코드 구조와 기술 스택 선택에 영향을 준다.
  - 유지보수 단계: 유지보수성이 높은 시스템은 새로운 요구사항이나 버그 수정 시 변경 비용을 줄인다.
  - 확장 단계: 확장성을 고려한 아키텍쳐는 사용자 증가나 기능 추가에 유연하게 대응한다.



정량적 요구사항의 중요성

품질 속성을 효과적으로 관리하려면 요구사항을 정량적으로 정의해야 한다.

- ex) 시스템이 안정적이여야한다는 모호한 목표 대신 '연간 다운타임이 1분 미만이여야 한다.'
- 개발 과정에서 테스트와 검증을 통해 목표 달성 여부를 명확히 판단할 수. 있게 하며 이해관계자의 협의를 용이하게 한다.



## 품질 속성 시나리오와 예시

품질 속성을 시스템에 효과적으로 반영하려면 정량적이고 구체적인 기준이 필요하다. "시스템이 빠르거나 변경이 쉬워야 한다" 같은 모호한 표현은 주관적으로 검증할 수 밖에없다.

이를 해결하기 위해 품질 속성 시나리오를 사용한다.

- 품질 속성 시나리오는 품질 요구사항을 체계적으로 정의하고 측정 가능한 형태로 표현하는 도구다.



품질 속성 시나리오의 구성

품질 속성 시나리오는 6개 구성 요소로 이뤄져 있으며, 자극이 발생하는 맥락부터 시스템의 응답과 측정 기준까지를 명확히 정읳나다.

이 구조는 추상적인 품질 속성을 실행 가능한 요구사항으로 변환하는데 핵심적이다.



자극원 ---자극--> 대상체(환경) ---응답--> 응답 측정

다음은 각 구성 요소의 설명이다.

- 환경
  - 자극이 발생하는 시스템의 상태를 명시한다.
  - 정상 운영, 과부하, 또는 특정 시간대 등이며, 환경에 따라 동일한 자극에 대한 응답이 달라질 수 있다.
  - ex) 월말 결산 시간대, 서버 유지보수
- 자극원
  - 자극을 유발하는 주체로, 사용자, 외부 시스템, 내부 컴포넌트 또는 이벤트 등이 여기에 해당한다.
  - 자극원을 명확히 하면 시스템이 처리해야 할 입력의 성격을 파악할 수 있다.
- 자극
  - 시스템이 반응해야 하는 구체적인 이벤트다.
  - 사용자 요청, 오류, 또는 부하 증가 등이며, 발생 빈도나 패턴을 포함하면 더 정밀하다.
  - Ex) 도서 검색 요청 서버 다운 이벤트
- 대상체
  - 자극의 영향을 받는 시스템의 특정 부분으로, 전체 시스템이나 서브시스템 또는 컴포넌트일 수 있다.
  - 범위를 한정해 품질 요구사항의 적용 대상을 명확히 한다.
  - ex) 도서 검색 API, 데이터베이스 모듈
- 응답
  - 자극에 대한 시스템의 행동을 기술한다. 오류 처리, 데이터 반환, 부하 분산 등 품질 속성과 관련된 동작을 포함한다.
  - ex) 검색 결과 표시, 오류 로그 기록 및 사용자 알림
- 응답 측정
  - 응답의 성공 여부를 평가하는 정량적 기준으로, 시간, 처리량, 백분율 등으로 표현된다.
  - ex) 응답 시간 1초 이내, 가용성 99%



주요 품질 속성 시나리오 예시

품질 속성 시나리오는 여러 품질 속성에 적용 가능하다. 아래 내용은 가용성/성능/확장성/유지보수성에 대한 시나리오 예시를 요약하며, 온라인 도서 대출 시스템을 기반으로 아키텍쳐 영향을 설명한다.

가용성

- 시나리오 예시
  - 환경: 학기 초 등록 기간
  - 자극원: 서버 장애
  - 자극: 데이터베이스 연결 실패
  - 대상체: 도서 대출 서비스
  - 응답: 자동 페일오버로 백업 서버 전환
  - 응답 측정: 다운타임 20초 이내
- 아키텍쳐 영향
  - 리던던시(중복 서버)와 자동 복구 메커니즘을 설계해야 하며, 비용을 증가시키지만 서비스 연속성 보장

성능

- 시나리오 예시
  - 환경: 일반 운영 시간
  - 자극원: 웹 사용자
  - 자극: 도서 검색 요청
  - 대상체: 검색 API
  - 응답: 검색 결과 반환
  - 응답 측정: 평균 1초 이내, 95% 백분위수 1.5초 이내
- 아키텍쳐 영향
  - 인덱싱과 캐싱을 위한 데이터베이스 최적화 및 계층 분리가 필요하며, 이는 지연을 줄이지만 설계 복잡성을 높일 수 있다.

확장성

- 시나리오 예시
  - 환경: 신규 사용자 급증
  - 자극원: 신학기 캠페인
  - 자극: 대출 요청 증가
  - 대상체: 전체 시스템
  - 응답: 부하 분산 및 스케일 아웃
  - 응답 측정: 500 TPS 처리 가능
- 아키텍쳐 영향
  - 모듈화와 마이크로서비스 구조를 도입, 의존성 최소화가 필수적이다.

유지보수성

- 시나리오 예시
  - 환경: 개발 중
  - 자극원: 개발자
  - 자극: 새로운 예약 기능 추가
  - 대상체: 비즈니스 로직 계층
  - 응답: 기존 코드 수정 없이 통합
  - 응답 측정: 변경 코드 50줄 이내

- 아키텍쳐 영향
  - 인터페이스 기반 계층 분리와 의존성 역전 원칙을 적용으로 변경 비용을 최소화한다.



## 실무적용: 온라인 도서 대출 시스템의 응답 시간 시나리오

도서 검색 기능은 사용자 경험에 직접적인 영향을 미치며, 지연이 길어지면 사용자 만족도가 급격히 떨어진다.

- 시나리오 예시: 학기초 일반 운영시간에 웹 사용자가 도서 제목으로 검색을 요청하면 시스템은 평균 1초 이내에 검색 결과를 표시해야 한다.
- 전체 요청의 95%는 1.5초 이내에 처리되어야 하며 최대 2초를 초과하지 않아야한다.

이 시나리오를 6가지 구성 요소로 나누면 다음과 같다.

- 환경: 학기 초 일반 운영 시간, 동시 접속자 300명, 분당 검색 요청 100건
- 자극원: 웹 애플리케이션 사용자
- 자극: 도서 제목 검색 요청
- 대상체: 도서 검색 API 및 데이터베이스 계층
- 응답: 검색어 검증, 데이터베이스 조회, 검색 결과 정렬 및 반환
- 응답 측정: 평균 응답시간 1초 이내, 95번째 백분위수 1.5초 이내 최대 허용시간 2초, 1.5초 이내 처리율 95%이상



이 시나리오는 개발 생명 주기 전반에 걸쳐 활용된다.

- 설계 단계에서는 데이터베이스 인덱싱, 캐싱, 비동기 처리 같은 기법검토
- 구현 단계에서는 성능 테스트를 통해 요구사항 충족 여부를 확인
- 배포 후. 지속적 모니터링 도구로 지속적 검증을 수행한다.



## 클린 아키텍쳐에서 중시하는 품질 속성

클린 아키텍쳐는 소프트웨어 아키텍쳐 설계에서 다양한 품질 속성을 고려하지만 특히 테스트 용이성과 변경 용이성을 중심적으로 고려한다.

이러한 속성은 소프트웨어의 장기적인 유지보수성과 유연성을 보장하며, 클린 아키텍쳐에서의 핵심 원리인 계층 분리와 의존성 역전을 통해 달성된다.

- 여기서는 각 품질 속성의 정의와 중요성을 설명하고 품질 속성 시나리오 예시를 구체적으로 살펴본다.
- 설계 및 검증 과정에서 고려할 관점을 논의하며, 클린 아키텍쳐가 어떻게 이러한 속성을 지원하는지 힌트를 제시한다.



테스트 용이성

테스트 용이성은 소프트웨어 시스템이 결함을 식별하고 기능 및 비기능적 요구사항을 검증하기 위해 얼마나 쉽게 수행할 수 있는지를 나타내는 품질 속성이다.

- 시스템의 동작 관찰, 입력 제어, 출력 검증 능력과 직접적으로 연결되며, 테스트 설계 / 실행 / 분석의 효율성을 높인다.

테스트 용이성은 복잡한 시스템이나 요구사항이 자주 변하는 환경에서 특히 중요하다.

- 잘설계된 테스트는 코드 변경 사이드 이펙트를 빠르게 탐지해서 아키텍쳐의 무결성을 유지한다.
- 반대로 테스트 용이성이 낮으면 변경이 위험해져 스파게티 코드로 이어질 수 있다.
- 소프트웨어 개발 비용 중 테스트가 15~40%를 차지한다고 알려져있다 따라서 테스트를 쉽게 작성/유지할 수 있는 구조는 더 높은 커버리지를 가능하게 해준다.



테스트 용이성 품질 속성의 구성요소와 예시

요소 / 설명 / 가능한 예시

자극원 / 테스트를 유발하는 주체 / 테스트 엔지니어, 개발자, QA팀

자극 / 테스트 실행 요청 / 단위 테스트, 통합 테스트, 회귀 테스트, 성능 테스트

환경 / 시스템 상태 / 개발, 테스트, 운영 환경에서 동작 중

대상체 / 테스트 대상 / 소프트웨어 코드, 모듈 API, 데이터베이스, 로그 파일

응답 / 시스템의 테스트 처리 및 결과 제공 / 테스트 입력 처리 명확하고 재현 가능한 결과 출력

### 테스트 용이성 시나리오의 설계 및 검증

테스트 용이성을 보장하면 아키텍쳐 구조가 모듈화되고 의존성이 명확해져 전체 시스템의 안전성과 확장성이 향상된다.

- ex) 도서 검색 모듈을 테스트할 때 외부 데이터베이스 의존성을 모킹하면 비즈니스 로직에 집중할 수 있으며, 변경시 버그 최소화한다.
- 클린 아키텍쳐는 비즈니스 로직을 외부 시스템과 분리해서 독립적인 테스트를 가능하게 하며, 계층 분리와 의존성 역전 원리를 통해 테스트 환경 간소화한다.

설계 원칙

1. 모듈화와 인터페이스: 시스템을 독립 모듈로 분리하고 인터페이스로 입력/출력을 제어한다. (테스트 시나리오 단순화 해줌)
2. 모킹과 스텁: 외부 의존성을 모킹해 비즈니스 로직 테스트에 집중한다. ex) 데이터베이스를 스텁으로 대체
3. 관찰 가능성: 상세 로깅과 메트릭으로 내부 상태 관찰 (쿼리 실행 로그 기록)
4. 테스트 자동화

검증 과정에서는 시나리오의 응답 측정 기준을 평가한다. 테스트 보고서는 차트와 통계로 이해관계자에게 전달되어야 하며, 아키텍쳐 유지에 기여한다.



변경 용이성

변경 용이성은 소프트웨어가 새로운 요구사항, 환경 변화, 또는 결함 수정에 따라 수정/확장 재구성될 수 있는 능력을 의미한다.

이는 유지보수성과 직접 연결되며, 기능 추가, 버그 수정, 성능 최적화, 또는 외부 변화에 대한 적응성을 포함한다. 변경 용이성이 높으면 수정 비용과 시간이 줄고 시스템 안전성이 유지된다. 변경 용이성의 경우 빠르게 변화하는 비즈니스와 기술 환경에서 필수적이다.

- 클라우드 서비스, 오픈소스, LLM같은 외부 요소를 활용하는 시대에 특정 기술에 종속되지 않고 유연하게 전환할 수 있도록 설계하는게 중요하다.

### 변경 용이성 시나리오의 설계 및 검증

변경 용이성은 아키텍쳐 구조를 유연하게 만들어 새로운 기능을 추가할 때 전체 시스템 재설계없이 대응할 수 있게 해줍니다.

- Ex) 도서 대출 시스템에서 새로운 검색 알고리즘 추가할 때 모듈화된 구조라면 비즈니스 로직만 수정
- 클린 아키텍쳐는 비즈니스 로직을 외부와 분리해서 계층화하고, 의존성 역전으로 변경의 영향을 최소화한다.

구체적인 설계 원칙

1. 모듈화와 캡슐화: 독립 모듈로 분리해서 변경의 파급 최소화
2. 의존성 관리: 의존성 주입과 인터페이스로 결합도를 낮춘다.
3. 테스트 자동화: 변경 후 안전성 확일을 위한 자동 테스트, 사이드 이펙트 탐지에 필수적
4. 로깅과 추적: 변경 영향을 모니터링 하기 위한 상세 로그, Ex) 예약 모듈 변경 후 로그로 동작



## 품질 속성 간 트레이드 오프

소프트웨어 아키텍쳐 설계에서 모든 품질 속성을 동시에 최적화 하기란 불가능하다. 각 속성은 자원을 공유하거나 상충하는 요구를 가지기 때문에 하나의 속성을 강화하면 다른 속성이 약화될 수 있다. 이러한 트레이드오프는 설계 과정의 핵심 도전으로 균형을 맞춰야한다.

품질 속성간 트레이드는 자원의 제한으로 발생

- Ex) 성능 극대화를 위해 복잡한 알고리즘이나 캐싱을 도입하면 유지보수성 저하 / 단순한 구조를 선택하면 성능이 희생된다.

가용성과 비용의 트레이드 오프

- ex) 높은 가용성을 위해 리던던시를 추가하면 시스템이 좀 더 안전성을 띠지만, 그만큼 추가 시스템을 구축하고 운영하기 위한 비용 추가됨

보안과 사용자 편의성 또한 트레이드 오프의 대상이다.

모든 품질 속성을 최적화 할 수 없으므로 아키텍트는 우선순위를 정해야한다.



## 정리

품질 속성이 소프트웨어 아키텍쳐 설계의 핵심 기준

품질 속성은 소프트웨어가 단순히 기능을 수행하는데 그치지 않고, 성능, 유지보수성, 테스트 용이성 같은 비기능적 요구사항을 충족하며 비즈니스 가치를 창줄한다.

- 이를 통해 개발, 배포, 유지보수, 확장에 이르는 소프트웨어 생명 주기 전반에 지속 가능성을 보장한다.

품질 속성에 대한 이해는 아키텍쳐 설계의 근거를 제공한다. 유행에 따라 클린 아키텍쳐나 특정 기술을 무작정 적용하면 오버엔지니어링이나 비효율적 구조로 이어질 수 있다.
