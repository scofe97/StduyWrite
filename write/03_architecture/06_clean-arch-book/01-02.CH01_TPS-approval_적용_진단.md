# CH01 TPS approval 적용 진단
---
> 1장은 *원칙·관점* 의 장입니다. 코드 한 줄을 고치는 진단보다, 모듈 *주변의 공유 산출물* 이 1장의 정의("구조의 결정 + 지식 공유")를 충족하는지 점검합니다. 따라서 본 회차의 진단은 메타 수준에 집중합니다.

## 1. 진단 대상 범위

1장의 다섯 절은 *코드 변경* 보다 *결정의 공유* 를 다룹니다. 따라서 진단 범위는 좁게 잡습니다.

| 진단 축 | 책 절 | 진단 대상 |
|--------|------|---------|
| 구조의 결정 | §1.1 | `approval/` 패키지 구조 자체 |
| 지식의 공유 | §1.1 | package-info.java, README, ADR 같은 공유 산출물 |
| 의도 드러내기 | §1.4 | 패키지 이름·인터페이스 이름이 *왜 그렇게 지었는지* 추론 가능한가 |
| 살아있는 아키텍처 | §1.4 후반 | 지난 결정의 *왜* 가 어디에 남아 있는가 |

진단에서 *명시적으로 제외* 한 축:

- 의존성 규칙(5장) — 1장 범위 밖
- 트랜잭션 경계(3·9장) — 1장 범위 밖
- 유스케이스 인터페이스 설계의 적정성(6장) — 1장 범위 밖

이 좁힘은 의도된 것입니다. 1장 진단으로 모듈 전체를 훑으면, 책의 진도와 진단의 깊이가 어긋납니다.



## 2. 현 상태 스냅샷 (read-only)

> 어시스턴트가 코드 변경 없이 `find`/`ls` 만으로 확인한 결과입니다.

| 산출물 | 위치 | 결과 | 메모 |
|--------|------|------|------|
| `package-info.java` (approval 하위 전체) | `operator/ticket/.../approval/**` | **0건** | 패키지 의도가 코드 *주석으로도* 남아있지 않습니다 |
| `README.md` (approval 모듈 안) | `operator/ticket/.../approval/` | **0건** | 모듈 진입점에서 "이 모듈이 무엇을 하는가" 를 알려주는 문서가 없습니다 |
| 모듈별 architecture 가이드 | `operator/docs/architecture/` | 3건 존재 | `conditional-bean-guidelines.md`, `core-module-guidelines.md`, `module-dependencies.md`. approval 전용은 없음 |
| ADR (결정 기록) | `operator/docs/architecture/` | **0건** | architecture 가이드는 *현재 규약* 만 기술. *결정의 이유·대안·트레이드오프* 는 어디에도 없음 |
| 자매 문서 (외부) | `runners-high/write/03_architecture/04_ddd/04-01` | 존재 | TPS 결재 헥사고날 변형 사례 — *학습 노트* 라 운영 저장소 내부에서는 보이지 않음 |
| 패키지 1계 구조 | `approval/` | api, application, common, domain, infrastructure, presentation | 6개 |
| 도메인 하위 구조 | `approval/domain/` | component, dto, enums, event, exception, model, port, vo | 8개 |

스냅샷에서 가장 눈에 띄는 사실은 두 가지입니다. 첫째, *패키지가 잘 나뉘어 있다는 점* — `domain/port/in,out` 구조는 헥사고날의 의도가 명확히 들어가 있습니다. 둘째, *그 의도가 코드 바깥에는 어디에도 안 남아 있다는 점* — package-info, README, ADR 셋 다 0건입니다.



## 3. 책 개념과의 거리

책의 §1.1 은 아키텍처를 *구조의 결정 + 지식의 공유* 두 면으로 정의합니다. TPS approval 의 *구조 결정* 은 충실합니다. 그러나 *지식 공유* 면이 비어 있습니다. 새 사람이 approval 패키지를 처음 열었을 때, *왜 domain 과 application 이 나뉘었는지*, *왜 port 가 in 과 out 으로 갈라졌는지*, *왜 component 라는 폴더가 또 있는지* 를 *코드만으로* 추론해야 합니다. 1장의 §1.4 "일관성" 과 "의도 드러내기" 가 절반만 충족된 상태입니다.

특히 §1.4 의 *변경에 대한 용기* 는 *왜 그 결정이 내려졌는지* 가 남아있을 때만 생깁니다. 지금 approval 모듈에는 결정의 *결과* 만 코드에 남았고 *이유* 가 어디에도 없습니다. 그래서 5년 뒤 누군가가 `domain/component/execution/command,query` 의 분리가 부자연스럽다고 느낄 때, 그 사람은 *왜 이렇게 했는지* 를 모른 채 *되돌릴지 유지할지* 만 결정해야 합니다. 1장이 가장 경계하는 상황이 정확히 이 자리입니다.

자매 문서 `04_ddd/04-01.헥사고날 변형 — TPS 결재 도메인 적용 사례.md` 는 *외부 학습 노트* 입니다. 운영 저장소를 열어보는 개발자에게는 보이지 않습니다. *지식이 회사 안에 있긴 하지만 코드 옆에는 없는* 상태입니다.



## 4. 개선 후보와 영향 범위

1장의 관점에서 도출한 후보는 두 개입니다. 4개 이상이 나오면 범위를 잘못 잡은 신호라는 자기 규약에 따라, 1장 범위 안에 분명하게 머무는 두 가지만 남겼습니다.

### 후보 A — `package-info.java` 도입 (8개 패키지)

**의도**: `domain/port/in,out` 의 의도를 *코드 옆* 에 한 줄씩 남깁니다. 1장 §1.4 "단순성과 지속성" 과 "의도 드러내기" 의 가장 가벼운 실천입니다.

**영향 패키지/파일**:

```
approval/domain/package-info.java            (신규)
approval/domain/port/in/package-info.java    (신규)
approval/domain/port/out/package-info.java   (신규)
approval/domain/component/package-info.java  (신규)
approval/application/service/package-info.java (신규)
approval/infrastructure/persistence/adapter/package-info.java (신규)
approval/api/package-info.java               (신규)
approval/presentation/package-info.java      (신규)
```

각 파일에는 5~10줄 정도의 의도만 적습니다. 한 패키지의 책임이 한 단락으로 안 적히면, 그 패키지가 사실은 *두 가지 일* 을 하고 있다는 신호입니다.

**예상 트레이드오프**:

- 이득 — IDE 가 패키지 hover 시 의도를 보여줍니다. Javadoc 빌드에도 자동 포함됩니다. 운영 코드와 같은 PR 에서 결정·이유가 같이 움직입니다.
- 비용 — *처음 한 번* 작성이 부담입니다. 그러나 본 시리즈가 챕터를 진행하며 자연스럽게 채워질 수 있습니다.
- 리스크 — package-info 가 *현재 코드와 어긋난 채로 방치* 되면 오히려 학습 비용이 늘어납니다. ArchUnit 같은 sensor 로 *패키지 안의 클래스 종류가 package-info 에 선언된 의도와 일치하는지* 검증할 수 있습니다 (이 검증은 9장 이후로 미룹니다).

**사용자 적용 난이도**: **S** — 작성과 빌드만 필요합니다. 컴파일·테스트에 영향 없습니다.

### 후보 B — `approval/README.md` 추가 (모듈 진입점 문서)

**의도**: 모듈을 처음 여는 사람이 *3분 안에* 모듈의 책임·경계·확장 지점을 파악할 수 있게 합니다. 1장 §1.1 "지식의 공유" 와 §1.4 "협업의 향상" 의 가장 직접적인 실천입니다.

**영향 파일**:

```
operator/ticket/src/main/java/org/okestro/tps/operator/ticket/approval/README.md (신규)
```

README 안에 들어갈 항목 (제안):

- 모듈의 책임 한 단락 — "결재 요청·승인·반려·위임의 도메인" 같은 한 문장이 아닌, *왜 ticket 모듈 안에 들어 있는지* 까지 포함된 단락
- 패키지 1계 구조 표 — api / application / common / domain / infrastructure / presentation 각각의 역할 한 줄
- 의존 방향 도식 (Mermaid 1개) — 외부 → 어댑터 → port → 도메인
- 진입점 — `ApprovalPrcsCommandUseCase`, `ApprovalBscQueryUseCase` 같은 핵심 인터페이스 3~5개에 cross-link
- 자매 문서 링크 — `runners-high/write/03_architecture/04_ddd/04-01` 와 본 시리즈 (단, *운영 저장소 안에서* Drive 외부 링크가 의미가 있는지는 사용자 판단)

**예상 트레이드오프**:

- 이득 — 신규 합류자의 onboarding 비용이 떨어집니다. PR 리뷰 시 *모듈의 책임 범위* 합의가 빨라집니다.
- 비용 — README 와 코드가 어긋날 위험. 그러나 README 가 *모듈의 책임* 만 다루고 *세부 구현* 을 다루지 않으면, 어긋날 빈도는 낮습니다.
- 리스크 — README 가 너무 길어지면 *읽히지 않는 산출물* 로 굳어지므로, 1페이지 이내라는 길이 상한을 같이 박아두는 게 안전 장치죠.

**사용자 적용 난이도**: **S~M** — 작성은 쉽지만, *어디까지 README 에 담고 어디부터는 별도 문서로 분리할지* 의 결정이 약간의 판단을 필요로 합니다.

### 진단에서 *후보로 올리지 않은 것*

다음 항목들은 1장의 범위를 벗어나 의도적으로 후보에서 제외했습니다.

- ADR(Architecture Decision Record) 도입 — 1장 §1.4 후반의 "살아있는 아키텍처" 와 관계는 있지만, *개별 결정의 형식* 은 본 책의 8장(애자일) 과 9장(MSA 진화) 이후에서 더 자연스럽게 다뤄집니다. 1장 진단 단계에서 ADR 까지 시작하면 *형식만 갖추고 내용은 비는* 산출물이 만들어지기 쉽습니다.
- 의존성 검증(ArchUnit) — 5장 의존성 규칙 학습 이후에 진단합니다.
- 패키지 이름 변경 — 1장은 *왜 좋은 설계가 필요한가* 의 장입니다. 패키지 rename 은 본 시리즈의 어느 챕터에서도 권하지 않을 가능성이 높습니다 (운영 코드의 안정성 > 학습용 개선).
