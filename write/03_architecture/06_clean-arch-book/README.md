# 『클린 아키텍처 핵심 가이드』 학습 시리즈
---
> 한승희 저 『클린 아키텍처 핵심 가이드』를 1장씩 읽으면서, 같은 장의 개념으로 운영 중인 TPS operator-api 의 `approval` 모듈을 진단하고 개선해 보는 시리즈입니다. 책은 파이썬 FastAPI 예제로 설명하지만, 본 시리즈는 책의 원리를 Java/Spring Boot 헥사고날 코드에 *번역해* 적용합니다.

## 1. 이 시리즈가 풀고 싶은 문제

운영 중인 TPS approval 모듈은 이미 헥사고날 비슷한 구조(`domain/port/in,out` + `infrastructure/adapter`)를 갖고 있습니다. 그러나 "왜 이렇게 나뉘어 있는가" 를 제 입으로 설명하기는 어렵습니다. 책 한 권을 챕터별로 따라가면서, 매 장의 개념으로 같은 코드를 다시 들여다보는 방식이 가장 효율적이라고 판단했습니다. 책의 원리를 읽고 끝내면 휘발됩니다. 실제 운영 코드를 진단의 대상으로 두면, 책에서 "왜?" 라고 던지는 질문이 곧 제 코드에 대한 질문이 됩니다.

진단 대상이 `approval` 인 이유는 두 가지입니다. 도메인 자체가 복잡합니다 — 결재 요청·승인·반려·위임이 얽혀 있어 한 장 한 장의 개념이 닿을 자리가 많습니다. 또한 이미 자매 문서가 두 편 있어, 챕터 진단이 *완전히 새 그림을 그리는 작업* 이 아니라 *기존 그림에 한 겹 더 입히는 작업* 이 됩니다.

> **함께 읽으면 좋은 자매 문서**
>
> - [`../02_application/01-02.클린 아키텍처.md`](../02_application/01-02.클린%20아키텍처.md) — 카드게임 도메인으로 정리한 클린 아키텍처 *교과서 요약*. 본 시리즈가 책의 진도를 따라가는 동안 이 문서는 *개념 도식 SSOT* 역할을 합니다.
> - [`../04_ddd/04-01.헥사고날 변형 — TPS 결재 도메인 적용 사례.md`](../04_ddd/04-01.%ED%97%A5%EC%82%AC%EA%B3%A0%EB%82%A0%20%EB%B3%80%ED%98%95%20%E2%80%94%20TPS%20%EA%B2%B0%EC%9E%AC%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A0%81%EC%9A%A9%20%EC%82%AC%EB%A1%80.md) — TPS 결재 도메인을 헥사고날 변형 관점으로 한 번 정리한 사례. 본 시리즈의 진단이 이 문서와 충돌하지 않는지 매 챕터 cross-check 합니다.



## 2. 진행 방식 (1챕터 / 세션)

한 챕터당 두 개의 파일을 만듭니다. 한쪽은 책 요약, 다른 쪽은 TPS approval 진단입니다.

```
NN-01.책_CH{n}_{장 제목}.md            ← 책 요약 (어시스턴트 작성)
NN-02.CH{n}_TPS-approval_적용_진단.md  ← TPS 진단 + 사용자 적용 결과
```

진단 파일은 6개 절로 나뉘는데, 앞 네 개(§1~§4)는 어시스턴트가 read-only 로 작성하고, 뒤 두 개(§5~§6)는 사용자가 실제 코드에 적용한 뒤 직접 채웁니다. 이 분리가 본 시리즈의 핵심입니다.

세션 한 번에 한 챕터씩만 진행합니다. 책의 한 장을 읽고, 그 장을 어시스턴트와 정리하고, 진단을 받고, 다음 세션 전까지 코드에 적용해 보고, 다음 세션 처음에 결과를 평가합니다.

### 세션 6단계

| 단계 | 누가 | 무엇 |
|------|------|------|
| 1 | 사용자 | 책의 N장을 읽고 "CH{n} 시작" 으로 진입 |
| 2 | 사용자 | 그 장의 **핵심 주장·인상적이었던 문장·예시·모호한 지점** 을 어시스턴트에게 전달 |
| 3 | 어시스턴트 | 전달받은 핵심을 두 갈래로 분류 — *자매 문서에 이미 있는 부분 → cross-link*, *없는 부분 → `NN-01` 에 정리* |
| 4 | 어시스턴트 | `NN-02.CH{n}_*.md` §1~§4 진단 (read-only) |
| 5 | 사용자 | TPS 저장소에 직접 적용 (다음 세션 전까지) |
| 6 | 어시스턴트+사용자 | 다음 세션 초입에 `NN-02` §5~§6 채움 |

> ⚠️ **중요**: 어시스턴트는 *책의 절 제목만 보고 본문을 추측해 채우지 않는다.* 사용자 전달이 없으면 `NN-01` 은 "⏳ 사용자 입력 대기" 상태로 둔다. 절 제목 기반 추측은 환각 위험이 있고, 책의 실제 강조점과 어긋날 수 있다.

### 자매 문서 우선 cross-link 정책

본 시리즈는 *책의 진도* 가 축이고, *개념의 정의* 는 기존 자료를 재활용합니다. 사용자가 전달한 핵심이 자매 문서(`01_foundation/`, `02_application/`, `04_ddd/` 등)와 *같은 통찰의 다른 표현* 이면, `NN-01` 에는 자매 문서 링크만 박고 *왜 이 장에서도 같은 결론에 도달했는지* 만 한 단락 적습니다. 본 시리즈에서 본문을 *처음으로* 정리하는 경우는 자매 문서에 없는 책 고유의 강조점일 때뿐입니다.

## 4. 진단 파일의 6개 절 골격

`NN-02.CH{n}_*.md` 는 매 챕터 동일한 구조입니다.

| 절 | 누가 | 무엇 |
|---|------|------|
| §1 진단 대상 범위 | 어시스턴트 | 그 장 개념이 닿는 패키지만 한정 (모듈 전체 진단 금지) |
| §2 현 상태 스냅샷 | 어시스턴트 | 표로 정리. 파일·라인·책 개념 매핑·메모. read-only |
| §3 책 개념과의 거리 | 어시스턴트 | 책이 권하는 모습 vs 현재 코드. "왜 거리가 있는가" 한 문장 |
| §4 개선 후보와 영향 범위 | 어시스턴트 | 1~3개. 의도·영향 파일·트레이드오프·적용 난이도(S/M/L) |
| §5 사용자 적용 결과 | 사용자 | 다음 세션 전 채움. diff, 빌드/테스트 결과, 본인 평가 |
| §6 사용자 평가 | 사용자 | "책 개념이 실제로 도움이 됐는가" 한 단락 |

후보가 4개 이상 나오면 그 장의 진단 범위를 잘못 잡은 신호입니다. 범위를 좁히고 다시 잡습니다.



## 5. 글쓰기 규약

본 시리즈는 `~/claude/.claude/skills/content/writing/` 의 writing 스킬을 따릅니다. 핵심만 옮기면:

- 합니다체 통일. 인접 폴더(`04_ddd/` 일부)가 한다체여도 따라가지 않습니다.
- AI 강조어("매우", "굉장히", "정말로", "획기적인", "혁신적인") 사용 금지.
- 모든 주장에 "왜?" 한 문장 동반. 키워드 나열 금지.
- 같은 어미 3회 연속 금지. 어미 다양화(~다 / ~이다 / ~합니다 / ~일까요).
- 코드 인용은 *실제 파일·라인 그대로* 옮기고, 인용 시 `org.okestro.tps.operator.ticket.approval.XXX:NN` 형식으로 추적성을 남깁니다.

작성 직후 grep 5종(어미·강조어·Mermaid·줄수·코드블록 외부 산문) 검증을 거칩니다. 자세한 절차는 writing 스킬 §8.



## 6. 무엇을 *하지 않는가*

- 한 세션에 여러 챕터 진행 — 본 시리즈는 1챕터/세션 cadence.
- approval 운영 코드의 어시스턴트 직접 수정 — 진단·권고까지만. 적용은 사용자.
- 자매 문서(`02_application/01-02`, `04_ddd/04-01`) 본문을 본 시리즈에 복제 — cross-link 만.
- 책 외부 자료의 무차별 가져옴 — 책의 진도가 시리즈의 진도입니다. 책에 없는 개념은 §"심화" 같은 별도 절로 분리합니다.
