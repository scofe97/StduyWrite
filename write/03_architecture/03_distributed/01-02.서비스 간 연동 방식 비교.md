# 서비스 간 연동 방식 비교
---
> 서비스 간 연동 방식은 기술 취향의 문제가 아니라 응답 시간, 결합도, 데이터 전달 보장 수준을 맞추는 선택이다.
>
> - REST
> - gRPC
> - Messaging
> - CDC

## 1. 어떤 방식들이 있는가

분산 시스템에서 가장 흔한 연동 방식은 네 가지다. 동기 HTTP 기반 호출, gRPC 기반 RPC, 메시지 브로커 기반 비동기 이벤트 전달, 그리고 CDC 기반 데이터 전파다.

각 방식은 해결하는 문제가 다르다. 따라서 “우리 팀 표준” 하나만으로 밀어붙이기보다, 어떤 정보가 어떤 시간 제약 아래 전달되어야 하는지 먼저 봐야 한다.

## 2. 방식별 특성

| 방식 | 강점 | 약점 | 예시 적용 |
|------|------|------|----------|
| REST | 단순하고 범용적 | 채팅형 호출 증가, 스키마 느슨함 | 프로필 조회, 런 시작 API |
| gRPC | 타입 안전성, 저지연 | 브라우저 친화성 낮음 | 실시간 내부 서비스 호출 |
| Messaging | 느슨한 결합, 버퍼링 | 최종 일관성, 운영 복잡도 | 전투 로그 발행, 분석 적재 |
| CDC | 기존 DB 변경 전파 | 소스 스키마에 종속 | 운영 데이터 파생 적재 |

## 3. 카드게임 예시에 대입하면

런 관리 서비스에서 계정 검증이나 세션 확인은 REST나 HTTP client로 충분하다. 호출 결과를 바로 받아야 하고, 사용자가 요청-응답 흐름 안에 있기 때문이다.

반면 전투 로그를 분석 시스템으로 넘기는 흐름은 메시징이 더 적합하다. 플레이어 응답을 기다리게 할 이유가 없고, 분석 시스템 장애가 핵심 전투 흐름을 막으면 안 되기 때문이다.

## 4. 공유 라이브러리와 원격 호출을 혼동하지 말아야 한다

마이크로서비스 환경에서는 DTO 공유 라이브러리를 너무 넓게 쓰면 결합도가 다시 생긴다. 같은 Java 생태계라고 해서 모든 서비스가 같은 도메인 모델 클래스를 공유하면 버전 호환 문제가 커진다.

계약은 스키마나 API 명세로 공유하고, 내부 도메인 객체는 각 서비스가 자기 방식으로 유지하는 편이 낫다. 특히 카드 정의나 이벤트 페이로드는 외부 계약과 내부 모델을 분리해두는 것이 좋다.

## 5. 장애 관점에서 본 비교

연동 방식은 기능보다 장애 전파 양상으로 비교해야 한다. 동기 호출은 단순하지만, 상대 서비스가 느려지면 호출 체인이 함께 느려진다. 비동기 메시징은 격리에 유리하지만 지연과 재처리를 감수해야 한다.

따라서 다음 기준이 실용적이다:

- 지금 당장 결과가 필요하면 동기 호출
- 후처리와 파생 데이터라면 비동기 이벤트
- 기존 DB 변경 전파가 목적이면 CDC

## 6. 런 관리 서비스에 적용한 권장안

이 시리즈의 권장안은 다음과 같다:

- 핵심 게임 진행: 내부 모듈 직접 호출
- 외부 계정/세션 확인: REST
- 전투 로그, 밸런스 분석, 리플레이 적재: Messaging
- 분석용 적재 파이프라인: 필요 시 CDC 보조 사용

이 선택은 응답 시간과 도메인 경계를 동시에 고려한 결과다. 연동 방식은 한 번 정하면 오래 남으므로, 기능 요구보다 품질 속성 기준으로 선택하는 편이 안전하다.
