# Why Event-Driven Architecture

---

> 동기 직접 호출 구조의 한계를 짚고, 이벤트 기반 통신이 이를 어떻게 푸는지 정리한다. Bounded Context, 이벤트 유형, Table-Stream Duality, Single Writer Principle, Event Broker와 Message Broker의 차이까지가 한 묶음이다.



## 학습 목표

> EDA를 *서비스 간 직접 호출을 제거하고, 이벤트 브로커를 단일 진실 공급원으로 두는 통신 모델*로 이해한다.

이 장을 다 읽고 다음 다섯 가지에 자신 있게 답할 수 있으면 학습이 완료된다.

1. 동기 직접 호출의 두 가지 결합 문제(Point-to-Point 결합, 장애 전파)를 설명할 수 있다.
2. Event와 Command와 Query의 차이를 *시제·거부 가능성·결합도* 세 축에서 비교할 수 있다.
3. Message Broker와 Event Broker의 *소비 후 처리·다중 소비·재생*이 왜 다른지 설명할 수 있다.
4. Single Writer Principle이 데이터 소유권을 어떻게 명확히 하는지 설명할 수 있다.
5. EDA 4가지 유형(Internal·Ephemeral·Queue·Pub/Sub Streaming)을 사용 시점별로 구분할 수 있다.



## 1. 전통적인 아키텍처의 문제

모놀리스 또는 동기 마이크로서비스에서 발생하는 근본 문제는 서비스 간 직접 호출 중심의 통신 구조가 결합도를 높인다는 점이다. 각 서비스가 다른 서비스를 직접 호출하는 방식은 서비스 간 강한 결합을 만들고, 이는 마이크로서비스 아키텍처의 핵심 가치인 독립성과 확장성을 훼손한다.

```mermaid
graph LR
    subgraph "동기식 마이크로서비스의 문제"
        S1[Service 1] --> S2[Service 2]
        S2 --> S3[Service 3]
        S2 --> S4[Service 4]
        S3 --> S5[Service 5]
    end
```

- Point-to-Point 결합: 서비스 A가 서비스 B를 직접 호출하면 B의 API 변경이 A에 즉시 영향을 주기 때문에, 독립적인 배포와 버전 관리가 불가능해진다.
- **장애 전파**: 결제 서비스가 다운되면 결제를 호출하는 주문 서비스도 타임아웃이나 에러로 멈추고, 주문을 호출하는 프론트엔드도 응답하지 못한다.



## 2. 이벤트 기반 접근의 해결

이벤트 기반 아키텍처는 서비스 간 직접 호출을 제거하고, 이벤트 브로커라는 중앙 허브를 통해 통신한다. 서비스들은 서로를 몰라도 되며 이벤트 발행/구독만 생각하면 된다.

```mermaid
graph TB
    subgraph "Event Streams (Single Source of Truth)"
        ES[Event Broker]
    end

    P1[Producer 1] -->|publish| ES
    P2[Producer 2] -->|publish| ES

    ES -->|subscribe| C1[Consumer 1]
    ES -->|subscribe| C2[Consumer 2]
    ES -->|subscribe| C3[Consumer 3]
```

- 이벤트 = 데이터: 이벤트는 단순한 알람이 아니라 비즈니스 데이터 자체를 포함한다. 컨슈머가 추가 동기 호출 없이 처리할 수 있을 만큼의 맥락을 담는 것이 좋다.
- 비동기 통신: 프로듀서는 이벤트를 발행한 후 즉시 다음 작업을 진행할 수 있다. 컨슈머가 언제 이벤트를 처리하는지, 컨슈머가 몇 개인지도 알 필요가 없기 때문에 진정한 느슨한 결합이 달성된다.



## 3. 핵심 개념

### 3-1. Bounded Context와 DDD

이벤트 기반 마이크로서비스는 DDD의 Bounded Context 개념과 자연스럽게 매핑된다. Bounded Context는 특정 도메인 모델이 유효한 범위를 정의하며, 이는 마이크로서비스의 경계를 결정하는 핵심 기준이 된다.

| 개념                | 영역        | 왜 중요한가?                                                 |
| ------------------- | ----------- | ------------------------------------------------------------ |
| **Domain**          | 문제 공간   | 비즈니스가 해결하려는 전체 문제 영역이다. 이커머스라면 '온라인 쇼핑 경험 제공'이 도메인이다. 도메인을 이해해야 어떤 서브도메인으로 나눌지 판단할 수 있다. |
| **Subdomain**       | 문제 공간   | 도메인을 구성하는 개별 비즈니스 영역이다. 주문, 재고, 배송, 결제는 각각 독립적인 비즈니스 로직과 규칙을 가지므로 서브도메인으로 분리하면 팀이 각자의 영역에 집중할 수 있다. |
| **Bounded Context** | 솔루션 공간 | 입력·출력·이벤트·데이터 모델이 일관된 의미를 갖는 논리적 경계다. '상품'이라는 단어가 재고 컨텍스트에서는 '재고 수량'을, 카탈로그 컨텍스트에서는 '상품 설명과 이미지'를 의미할 수 있다. 이 경계가 명확해야 서비스 간 오해 없이 통신한다. |

- 높은 응집도: 서로 관련된 기능은 한 곳에 모아야 한다. 주문 생성·취소·조회는 모두 주문 서비스에 있어야 편리하다.
- 느슨한 결합: 한 컨텍스트 내부 변경이 다른 컨텍스트에 영향을 주지 않아야 한다. 이벤트 기반 통신을 쓰면 주문 서비스가 내부 DB 스키마를 바꿔도 발행 이벤트 형식만 유지되면 외부에 영향이 없다.
- 비즈니스 정렬: 기술이 아닌 비즈니스 요구사항으로 경계를 결정한다.

### 3-2. 이벤트 유형

**Key 파티셔닝 기준 분류**

| 유형              | Key         | 왜 이렇게 설계하는가?                                        | 예시                                                         |
| ----------------- | ----------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **Unkeyed Event** | 없음        | 순서가 중요하지 않거나 각 이벤트가 독립적인 사실을 기록할 때 쓴다. 키가 없으면 파티션에 라운드로빈으로 분배되어 처리량이 높아지지만 이벤트 간 순서는 보장되지 않는다. | 사용자가 특정 페이지를 조회한 로그 (조회 순서가 비즈니스 로직에 영향 없음) |
| **Entity Event**  | 엔티티 ID   | 같은 엔티티에 대한 이벤트들이 순서대로 처리돼야 할 때 쓴다. 주문 ID를 키로 잡으면 같은 주문의 '생성→승인→배송' 이벤트가 같은 파티션에 들어가 순서가 보장된다. 이벤트 스트림을 테이블로 구체화(materialize)할 때 키별로 최신 상태만 유지된다. | 주문 ID를 키로 하는 주문 생성·수정 이벤트 (주문 123의 상태 변화 순서 보장 필요) |
| **Keyed Event**   | 비엔티티 키 | 집계·그룹화를 위해 쓴다. 사용자 ID를 키로 잡으면 같은 사용자의 모든 주문이 같은 파티션에 들어가 사용자별 총 구매액 같은 통계를 효율적으로 계산한다. 엔티티 상태 전이가 아닌 분석·집계 목적으로 이벤트를 그룹화할 때 비엔티티 키를 쓴다. | 사용자 ID를 키로 하는 주문 이벤트 (사용자별 주문 패턴 분석 시 유용) |

**이벤트 의도(Command) 분류**

| 구분                | Event                         | Command                       | Query                   |
| ------------------- | ----------------------------- | ----------------------------- | ----------------------- |
| **시제**            | 과거형 — OrderCreated         | 명령형 — CreateOrder          | 요청형 — GetOrderStatus |
| **의미**            | 이미 일어난 사실의 기록       | 특정 수신자에게 행동을 요청   | 데이터 조회 요청        |
| **거부 가능?**      | 불가 (사실이므로)             | 가능 (수신자가 검증 후 거부)  | 해당 없음               |
| **프로듀서:컨슈머** | 1:N (브로드캐스트)            | 1:1 (지정된 수신자)           | 1:1 (응답 필요)         |
| **결합도**          | 낮음 (발행자는 구독자를 모름) | 높음 (발행자가 수신자를 의식) | 높음                    |

- SAGA는 Command를 써서 오케스트레이터가 "배송을 시작(Command)" 명령을 서비스에 직접 보내고 성공·실패 응답을 기다린다.

### 3-3. Table-Stream Duality

이벤트 스트림과 테이블은 동일한 데이터를 다른 관점에서 표현한 것이다. 이 개념을 이해하면 상태 관리와 이벤트 처리를 통합된 시각으로 볼 수 있다.

```mermaid
graph LR
    subgraph "Event Stream → Table"
        E1["AA: v1"] --> T["AA: v2 (최신)"]
        E2["BB: v1"] --> T2["BB: v1"]
        E3["AA: v2"] --> T
    end
```

- Materialization(구체화): 이벤트 스트림을 테이블로 만드는 과정.
- CDC(Change Data Capture): 테이블의 변경 사항을 이벤트 스트림으로 변환하는 과정 (Debezium 등).
- Tombstone: Value가 null인 이벤트로 삭제를 표현.

### 3-4. Message, Command, Event의 관계

**Message**는 시스템 간에 전달되는 모든 데이터의 총칭이다. Command와 Event는 모두 Message의 하위 유형이며, 의도와 시제가 다르다.

```
Message (시스템 간 전달되는 모든 데이터)
├── Command  — "이걸 해라" (명령, 미래형, 거부 가능)
├── Event    — "이게 일어났다" (사실 기록, 과거형, 거부 불가)
└── Query    — "이걸 알려달라" (조회 요청)
```

이 구분이 중요한 이유는 브로커의 설계 철학과 직결되기 때문이다. Message Broker는 Command 성격의 메시지를 1:1로 분배하는 데 최적화되어 있고, Event Broker는 Event 성격의 메시지를 1:N으로 공유하는 데 최적화되어 있다.

| 브로커 유형 | 주로 다루는 Message 유형 | 대표적 패턴 |
|-------------|------------------------|-------------|
| Message Broker (RabbitMQ) | Command — "이메일 보내라", "재고 차감해라" | 작업 분배, 1:1 소비 후 삭제 |
| Event Broker (Kafka/Redpanda) | Event — "주문이 생성됐다", "결제가 완료됐다" | 사실 공유, 1:N 독립 소비 |

즉 "Message Broker"의 "Message"는 모든 메시지를 뜻하는 게 아니라, 실질적으로 Command 성격의 작업 지시를 주로 처리하는 브로커라는 의미에 가깝다. 반대로 Event Broker는 이미 발생한 사실(Event)을 영속 로그로 보존하고 여러 서비스가 독립적으로 소비하는 구조다.

### 3-5. Event Broker vs Message Broker 상세 비교

Message Broker와 Event Broker는 비슷해 보이지만, 근본적으로 다른 철학을 가지고 있다.

| 특성          | Message Broker (RabbitMQ)                  | Event Broker (Kafka/Redpanda)                | 왜 이런 차이가 있는가?                                       |
| ------------- | ------------------------------------------ | -------------------------------------------- | ------------------------------------------------------------ |
| **소비 후**   | 메시지 삭제                                | 이벤트 보존                                  | Message Broker는 '작업 분배(task distribution)'를 위해 설계되었다. 작업이 한 번 처리되면 끝이므로 메시지를 삭제한다. Event Broker는 '상태 공유(state sharing)'를 위해 설계되어, 여러 컨슈머가 같은 데이터를 다른 시간에 읽을 수 있어야 하므로 보존한다. |
| **다중 소비** | 각 컨슈머가 일부만 받음 (경쟁 컨슈머 패턴) | 모든 컨슈머가 전체 접근 (각자 독립적 오프셋) | Message Broker는 메시지를 여러 워커에게 분배해서 병렬 처리한다. 메시지 1은 워커 A가, 메시지 2는 워커 B가 받는다. Event Broker는 각 컨슈머 그룹이 전체 스트림을 독립적으로 읽는다. 결제 서비스도, 재고 서비스도, 분석 서비스도 같은 주문 이벤트를 본다. |
| **재생**      | 기본 모델에서 제한적                       | 가능 (오프셋 리셋)                           | Message Broker는 기본 모델에서 제한적이고, Event Broker는 오프셋 기반 재처리가 기본 기능이다. Event Broker는 이벤트를 보존하므로, 새 서비스를 추가하거나 버그를 수정한 후 과거 데이터를 다시 읽어서 상태를 재구성할 수 있다. |
| **순서 보장** | 단일 큐/단일 컨슈머에서 유지               | 파티션 레벨에서 엄격                         | Message Broker도 단일 큐/단일 컨슈머 조건에서는 순서를 유지할 수 있지만, 병렬 소비 시 순서가 깨질 수 있다. Event Broker는 같은 키의 이벤트는 항상 같은 파티션에 들어가고, 파티션 내에서 순서가 보장되므로, 엔티티의 상태 변화 순서를 정확히 추적할 수 있다. |
| **상태 공유** | 부적합                                     | 적합                                         | Message Broker는 일회성 작업 처리에 적합하다 (이메일 전송, 알림). Event Broker는 여러 서비스가 같은 비즈니스 데이터를 공유하고 각자의 뷰를 유지해야 할 때 적합하다 (주문 상태를 주문 서비스, 배송 서비스, 분석 서비스가 각자의 방식으로 저장). |

### 3-6. Single Writer Principle

이벤트 스트림의 핵심 도메인 이벤트는 해당 애그리게잇을 소유한 서비스가 단독으로 발행하는 것이 바람직하다. 이는 이벤트 기반 아키텍쳐에서 데이터 소유권과 책임을 명확히 하는 핵심 원칙이다.

단일 Write를 권장하는 이유

- 권위 있는 진실의 원천 보장: 여러 서비스가 같은 토픽에 이벤트 쓰면 정보 충돌가능
- 데이터 계보 추적 가능: 이벤트를 보면 어떤 서비스가 생성했는지 알 수 있으니, 데이터 품질을 위해 단일 생성
- 소유권 경계 강제: ACL(Access Control List) 정책으로 토픽 쓰기 권한을 특정 서비스에만 부여하면, 다른 서비스가 변조하는걸 방지 가능

## 4. 이벤트란 무엇인가

> 소프트웨어에서 "이벤트"라는 단어는 UI 클릭, 네트워크 패킷, 로그 항목 등 다양한 의미로 쓰인다. 하지만 Kafka 기반에서 말하는 이벤트는 이보다 훨씬 구체적인 정의를 가진다. 이벤트는 **"비즈니스 도메인에서 실제로 발생한 사실(fact)의 기록"**이다.
>
> - 예: 주문이 접수되었다, 사용자가 구독을 해지했다.
> - 이벤트의 핵심 속성은 "불변성"이며, 과거에 일어난 일은 되돌릴 수 없으므로 한 번 기록되면 수정하지 않는다.

### 4-1. Command vs Event

Command(명령)은 미래에 어떤 일이 일어나길 요청하는 메시지다. "주문을 처리해라", "재고를 차감해라" 같은 형태이다. Command는 수신자가 거부할 수 있다.

Event는 이미 일어난 일의 기록이다. "주문이 처리됐다", "재고가 차감됐다" 형태이며, 거부할 수 없다. 이미 발행한 사실이기 때문이다.

```bash
Command: PlaceOrder     → (처리 성공하면) → Event: OrderPlaced
Command: CancelOrder    → (처리 성공하면) → Event: OrderCancelled
Command: DeductStock    → (처리 성공하면) → Event: StockDeducted
```

이 구분이 중요한 이유는 토픽 설계 방향이 달라지기 때문이다.

- Command 토픽은 특정 서비스가 요청해야 할 요청을 담고, 소비자는 하나다.
- Event 토픽은 도메인에서 발생한 사실을 담고, 여러 소비자가 각자 관심사에 따라 구독한다.

### 4-2. 이벤트 설계의 중요성

이벤트 설계는 단순히 "JSON에 어떤 데이터를 넣을까" 수준의 결정이 아니다. 이벤트 구조는 토픽 수, 소비자 결합도, 스키마 진화 비용, 시스템 전체 유연성에 직접 영향을 준다.

- **나쁜 설계**: 이름이 바뀌면 `UserNameChanged`, 이메일이 바뀌면 `UserEmailChanged` 식으로 각각 별도 토픽에 저장하면 깔끔해 보이지만, 전체 상태를 알려면 세 가지 토픽을 모두 구독·조회해야 한다.
- **좋은 설계**: 어떤 필드가 바뀌든 `user-profile-updated` 단일 토픽에 발행하고, 이벤트 안에 변경 내용을 담는다.

이러한 이벤트 설계를 위해 반드시 결정해야 하는 네 차원을 Confluent 학습 코스가 제공한다.

#### 4-2-1. Facts vs Deltas (전체 상태 vs 변경분)

- Fact 이벤트는 엔티티의 현재 전체 상태를 담는다. 소비자는 이 이벤트 하나만 보면 해당 엔티티의 현재 상태를 안다.
- Delta 이벤트는 변경된 부분만 담는다.

#### 4-2-2. Normalized vs Denormalized (정규화 vs 비정규화)

- Normalized는 엔티티의 고유 정보만 포함한다 (예: 이벤트에 customerID만 있고 고객 이름·이메일은 없음).
- Denormalized는 소비자가 필요한 정보를 모두 포함한다 (예: 이벤트에 customerID + 고객 이름·이메일을 함께 포함).

#### 4-2-3. Single vs Multiple Event Types per Stream (단일 vs 다중 이벤트 타입)

- Single Event Type은 토픽-이벤트가 1:1이다.
- Multiple Event Type은 토픽-이벤트가 1:다다.

#### 4-2-4. Discrete vs Continuous (이산 vs 연속 이벤트 흐름)

- Discrete는 명확한 비즈니스 사건을 나타낸다 (주문 접수, 배송 시작 같은 뚜렷한 트리거).
- Continuous는 시간에 따라 지속적으로 생성되는 이벤트다 (센서 온도, 주식 가격 등).

### 4-3. 도메인 매핑 (DDD → 이벤트 스트림)

좋은 이벤트를 설계하려면 도메인을 먼저 이해해야 한다. DDD의 Bounded Context 개념이 출발점이다.

- 이커머스 주문 시스템이라면 주문·재고·결제·배송이 각각 Bounded Context가 될 수 있다.

#### 이벤트 스토밍 접근

이벤트 스토밍은 도메인 전문가와 개발자가 함께 도메인 이벤트를 발견하는 기법이다. 포스트잇으로 비즈니스에서 발생하는 모든 사건을 시간 순으로 나열한다.

```bash
[주문 접수됨] → [재고 확인됨] → [결제 승인됨] → [포장됨] → [배송 시작됨] → [배송 완료됨]
```

이렇게 나열한 이벤트에서 Bounded Context의 경계가 드러나며, 이를 Kafka 토픽으로 매핑한다.

```mermaid
graph LR
    A[Order Service] -->|OrderPlaced<br>OrderCancelled| B[order-events]
    C[Payment Service] -->|PaymentApproved<br>PaymentFailed| D[payment-events]
    E[Inventory Service] -->|StockReserved<br>StockReleased| F[inventory-events]

    B --> G[Shipping Service]
    B --> H[Analytics Service]
    D --> G
    D --> I[Notification Service]
    F --> G
```

- Bounded Context당 하나 이상의 토픽
- 토픽 이름은 도메인 언어를 반영
- 같은 엔티티의 이벤트는 같은 토픽-파티션에 넣어서 순서 보장

## 5. EDA의 4가지 유형

> 모든 EDA가 같은 방식으로 동작하지는 않는다. 이벤트가 전달되는 범위·영속성·소비 모델에 따라 크게 네 유형으로 갈린다.

| 구분          | Internal                  | Ephemeral                | Queue                        | Pub/Sub Streaming          |
| ------------- | ------------------------- | ------------------------ | ---------------------------- | -------------------------- |
| **범위**      | 프로세스 내부             | 네트워크 (비영속)        | 네트워크 (영속)              | 네트워크 (영속 로그)       |
| **영속성**    | 없음 (메모리)             | 없음 (전달 후 삭제)      | 있음 (소비 시 삭제)          | 있음 (보존 정책까지 유지)  |
| **소비 모델** | Observer/Listener         | 현재 구독자만            | Point-to-point (round-robin) | Consumer Group (독립 소비) |
| **재처리**    | 불가                      | 불가                     | 불가 (소비 시 삭제)          | 가능 (오프셋 리셋)         |
| **대표 도구** | Go channel, Spring Events | Redis Pub/Sub, WebSocket | RabbitMQ, SQS, ActiveMQ      | Kafka, Redpanda, Pulsar    |
| **적합 사례** | 모듈 간 디커플링          | 실시간 알림, 채팅        | 작업 분배, 백그라운드 잡     | 이벤트 소싱, MSA, 분석     |

```mermaid
sequenceDiagram
    participant I as Internal
    participant E as Ephemeral
    participant Q as Queue
    participant S as Pub/Sub Streaming

    Note over I: 유형 1 - 프로세스 내부
    Note over I: Module A -> Module B (메모리)
    Note over I: 프로세스 종료 시 소멸

    Note over E: 유형 2 - 비영속 메시징
    Note over E: Producer -> Broker -> 온라인 Consumer만
    Note over E: 오프라인 Consumer = 유실

    Note over Q: 유형 3 - 메시지 큐
    Note over Q: Producer -> Queue -> Worker 1개만 소비
    Note over Q: 소비 후 삭제 (round-robin)

    Note over S: 유형 4 - 영속 스트리밍
    Note over S: Producer -> Topic Log -> 독립 소비
    Note over S: 재처리 가능, 삭제 안 됨
```



### 5-1. Application Internal (프로세스 내부 이벤트)

단일 애플리케이션 내부에서 컴포넌트 간 이벤트를 전달하는 패턴이다. 네트워크를 거치지 않으므로 지연이 극히 낮고, 별도 인프라가 필요 없다.

```mermaid
graph LR
    subgraph "단일 프로세스"
        A[주문 모듈] -->|OrderCreated 이벤트| B[재고 모듈]
        A -->|OrderCreated 이벤트| C[알림 모듈]
    end
    style A fill:#4a9,stroke:#333
    style B fill:#49a,stroke:#333
    style C fill:#49a,stroke:#333
```



- 예: Go의 채널, Java의 `ApplicationEventPublisher`. 이벤트는 프로세스 메모리에만 존재하므로 종료되면 사라진다.
- 단일 서비스 내부에서 모듈 간 결합도를 낮추고 싶을 때 쓴다.
- 한계: 프로세스 경계를 넘지 못해 다른 서비스와 이벤트를 공유할 수 없다.

### 5-2. Ephemeral Messaging (비영속 메시징)

네트워크를 통해 이벤트를 전달하되 메시지를 디스크에 저장하지 않는 방식이다. 수신자가 온라인이어야 메시지를 받을 수 있고, 연결이 끊긴 동안 발생한 메시지는 유실된다.

```mermaid
sequenceDiagram
    participant P as Producer
    participant B as Broker<br>(메모리만)
    participant C1 as Consumer A<br>(온라인 ✓)
    participant C2 as Consumer B<br>(오프라인 ✗)

    P->>B: 이벤트 발행
    B->>C1: 전달 ✓
    B--xC2: 전달 실패 (유실)
    Note over B: 디스크 저장 없음<br>전달 후 즉시 삭제
```



- 예: WebSocket 브로드캐스터, Redis Pub/Sub(영속 모드 비활성화). 발행자가 메시지를 보내면 브로커가 현재 연결된 구독자에게만 전달하고 메시지를 버린다.
- 한계: 메시지 유실 가능성을 감수해야 한다. 소비자가 잠시 다운되면 그동안의 이벤트를 놓친다.

### 5-3. Queue (메시지 큐)

메시지를 디스크에 영속 저장하고, 소비자가 메시지를 가져가면 큐에서 제거하는 방식이다. 하나의 메시지는 하나의 소비자만 처리한다.

```mermaid
sequenceDiagram
    participant P as Producer
    participant Q as Queue<br>(디스크 저장)
    participant W1 as Worker A
    participant W2 as Worker B

    P->>Q: msg-1 발행
    P->>Q: msg-2 발행
    P->>Q: msg-3 발행
    Q->>W1: msg-1 전달 (삭제)
    Q->>W2: msg-2 전달 (삭제)
    Q->>W1: msg-3 전달 (삭제)
    Note over Q: Round-robin 분배<br>소비 후 큐에서 제거<br>→ 다른 서비스는 같은 메시지를 못 봄
```



- 예: RabbitMQ, ActiveMQ. 여러 소비자가 같은 큐를 구독하면 Round-robin으로 분배되어 작업 부하를 분산한다.
- 한계: 메시지가 소비되면 사라지므로, 같은 이벤트를 여러 서비스가 독립적으로 소비할 수 없다.

### 5-4. Pub/Sub Streaming (영속 스트리밍)

메시지를 영속 로그에 저장하고 소비자가 읽어도 삭제되지 않는 방식이다. 여러 Consumer Group이 동일한 토픽을 독립적으로 읽을 수 있고, 과거 메시지를 오프셋 리셋으로 재처리한다.

```mermaid
sequenceDiagram
    participant P as Producer
    participant T as Topic Log<br>(영속 보존)
    participant G1 as Consumer Group A<br>(재고 서비스)
    participant G2 as Consumer Group B<br>(알림 서비스)
    participant G3 as Consumer Group C<br>(분석 서비스, 신규)

    P->>T: msg-1, msg-2, msg-3 발행
    T->>G1: msg-1,2,3 독립 소비 (offset 0→3)
    T->>G2: msg-1,2,3 독립 소비 (offset 0→3)
    Note over T: 메시지 삭제 안 됨 (보존 정책까지 유지)
    G3->>T: 신규 서비스: offset 0부터 재처리
    T->>G3: msg-1,2,3 전체 재생
```



- 예: Kafka, Redpanda, Kinesis. topic 기반 pub/sub 모델과 Consumer Group을 통한 병렬 처리를 결합해 유연한 소비 패턴을 지원한다.
- 한계: 운영 복잡성이 높아 파티셔닝·리밸런싱·스키마 관리의 설정과 모니터링 비용이 크다.



## 6. Simple vs Complex Event Processing

> 이벤트를 소비하는 방식은 크게 두 가지로 나뉜다. 이벤트에 즉각 반응하는 단순 이벤트 처리(SEP)와, 여러 이벤트를 시간 창(window)으로 묶어 패턴을 감지하는 복잡 이벤트 처리(CEP)다.

| 구분       | SEP                          | CEP                                  |
| ---------- | ---------------------------- | ------------------------------------ |
| **입력**   | 단일 이벤트                  | 복수 이벤트 (시간 창)                |
| **처리**   | 1:1 반응                     | 패턴 감지, 집계                      |
| **지연**   | 낮음 (ms 단위)               | 상대적으로 높음 (윈도우 크기에 비례) |
| **복잡도** | 낮음                         | 높음 (상태 관리, 윈도우 전략)        |
| **도구**   | Kafka Consumer, Spring Kafka | Kafka Streams, Flink, Esper          |
| **사례**   | 주문→재고 차감               | 사기 탐지, surge pricing, 이상 감지  |

### 6-1. SEP (Simple Event Processing)

단일 이벤트가 도착하면 단일 반응을 트리거하는 패턴이다. 대부분의 EDA 시스템이 이 방식으로 동작한다.

- 예: 재고 차감(`order.placed`), 배송 시작(`payment.completed`)
- SEP는 구현이 간단하고 지연이 낮다. Kafka Consumer가 메시지를 읽고 비즈니스 로직을 실행하는 일반적인 패턴이 SEP에 해당한다.

### 6-2. CEP (Complex Event Processing)

여러 이벤트를 시간 기반 창으로 집계하여 패턴을 감지·처리하는 방식이다. 개별 이벤트 하나하나는 의미 없지만 조합하면 의미 있는 신호가 된다.

```bash
# 입력 이벤트
- driver.location (GPS 좌표, 수천 건/초)
- ride.requested (탑승 요청)
- traffic.update(교통 정보)

# CEP 처리(5분 윈도우)
- IF 특정 지역의 ride.request 수 > driver.available 수 * 1.5
- AND traffic.update가 "혼잡"
- THEN surge_pricing 이벤트 발행 (배율 개선)

# 출력 이벤트
- surge.activated { zone: "강남", multiplier: 2.3 }
```



## 7. 면접 대비 Q&A

> 면접에서 자주 나오는 형태로 5개. 답을 보지 않고 먼저 입으로 답해 본 뒤 비교한다.

### Q1. 동기 직접 호출 구조의 두 가지 결합 문제를 한 문장씩 설명해 보세요.

Point-to-Point 결합과 장애 전파 두 가지다. Point-to-Point 결합은 *서비스 A가 B의 API를 직접 호출*하므로 B의 API 변경이 A를 즉시 깨뜨려 독립 배포가 막힌다. 장애 전파는 *호출 체인에서 한 서비스가 다운되면 그 위 모든 호출자가 타임아웃·에러로 줄줄이 멈추는* 현상이다. 두 문제 모두 "호출 시점에 두 서비스가 동시에 살아 있어야 한다"는 동기 모델의 본질에서 비롯된다.

### Q2. Event·Command·Query를 시제·거부 가능성·결합도 세 축에서 비교해 보세요.

| 축 | Event | Command | Query |
|----|-------|---------|-------|
| 시제 | 과거형 (OrderCreated) | 명령형 (CreateOrder) | 요청형 (GetOrderStatus) |
| 거부 가능 | 불가 (이미 일어난 사실) | 가능 (수신자가 검증 후 거부) | 해당 없음 |
| 결합도 | 낮음 (1:N, 발행자가 구독자 모름) | 높음 (1:1, 수신자 지정) | 높음 (응답 필요) |

이 구분이 토픽 설계의 방향을 갈라놓는다. Command 토픽은 수신자가 하나이고, Event 토픽은 여러 소비자가 각자 관심사에 따라 구독한다.

### Q3. Message Broker와 Event Broker의 소비 후 처리·다중 소비·재생은 왜 다른가요?

Message Broker는 *작업 분배(task distribution)*를 위해 설계됐고 Event Broker는 *상태 공유(state sharing)*를 위해 설계됐다. 작업은 한 번 처리되면 끝이므로 Message Broker는 메시지를 삭제한다. 상태는 여러 서비스가 다른 시간에 같은 데이터를 봐야 하므로 Event Broker는 보존한다. 같은 이유로 다중 소비도 다르다 — Message Broker는 워커들이 분담해 받지만, Event Broker는 각 Consumer Group이 *전체 스트림*을 독립적으로 읽는다. 재생도 보존 여부의 직접 결과로, Event Broker만 오프셋 리셋으로 과거 데이터를 다시 읽을 수 있다.

### Q4. Single Writer Principle은 운영에서 어떻게 강제하나요?

세 단계로 강제한다.

1. 토픽 단위로 *권위 있는 소유 서비스*를 한 곳으로 정한다 — 주문 도메인 이벤트는 주문 서비스만 발행한다.
2. ACL(Access Control List)로 토픽 쓰기 권한을 그 서비스의 인증 정보에만 부여한다.
3. 코드 리뷰·아키텍처 가이드에 "이벤트는 도메인 소유 서비스가 발행한다"는 규칙을 명문화한다. 1·2는 인프라 강제, 3은 문화적 강제이며 둘이 함께 있어야 사고가 줄어든다.

### Q5. EDA 4가지 유형(Internal·Ephemeral·Queue·Pub/Sub Streaming)을 사용 시점별로 어떻게 구분하나요?

선택 기준은 *프로세스 경계*, *영속성*, *다중 소비 필요성* 세 가지다.

- **Internal**: 프로세스 경계를 안 넘는다. 단일 서비스 안의 모듈 결합도 낮추기 (Spring Events, Go 채널).
- **Ephemeral**: 프로세스 경계 넘지만 영속성 없음. 실시간 알림·채팅처럼 *현재 연결자만 보면 충분*한 경우.
- **Queue**: 영속성 있지만 1:1 소비. 작업 분배·백그라운드 잡 (이메일 전송, 이미지 변환).
- **Pub/Sub Streaming**: 영속성 + 다중 독립 소비 + 재생 가능. 이벤트 소싱·MSA·분석처럼 *상태 공유*가 본질인 경우.

세 축 중 *다중 독립 소비가 필요한지*가 Queue와 Pub/Sub Streaming을 가르는 결정적 신호다.
