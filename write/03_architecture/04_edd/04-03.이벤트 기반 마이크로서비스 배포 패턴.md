# 이벤트 기반 마이크로서비스 배포 패턴
---
> EDA 의 배포는 단순 코드 교체가 아니다 — 이벤트 처리 위치·Consumer Group offset·스키마 호환성이 모두 함께 움직인다. 잘못된 배포는 이벤트 손실 또는 중복 처리로 이어진다.

## 1. 마이크로서비스 배포 원칙

> EDA 배포의 다섯 가지 원칙 — 이 원칙이 깨지면 사고로 직결된다.

SSOT §1 이 제시하는 원칙은 다음 다섯이다.

1. 무중단 — 배포 중에도 이벤트 처리가 계속되어야 한다.
2. 멱등성 — 같은 이벤트가 두 번 처리되어도 결과가 같아야 한다.
3. Offset 보존 — 배포 후 어디서부터 처리할지가 명확해야 한다.
4. 스키마 호환성 — 새 버전이 옛 이벤트를 처리할 수 있어야 한다.
5. 롤백 가능 — 사고 시 옛 버전으로 즉시 되돌릴 수 있어야 한다.

이 다섯이 모두 만족되어야 배포가 안전하다. 하나라도 깨지면 다음 절의 패턴들이 의미를 잃는다.

## 2. 배포 아키텍처 구성요소

> 배포 자동화의 단위 — 컨테이너 · Consumer Group · Schema · 모니터링.

SSOT §2 의 구성요소는 다음과 같다.

| 요소 | 역할 |
|------|------|
| 컨테이너 이미지 | 새 코드의 패키지 |
| Consumer Group ID | 동일하게 유지 (offset 보존) |
| Schema 등록 | 배포 전 Schema Registry 에 새 스키마 등록 |
| Health Check | 새 인스턴스가 처리 시작 전 검증 |
| 메트릭·알림 | 배포 중 이상 감지 |

다섯 중 하나라도 빠지면 배포 자동화가 깨진다. 특히 Schema 등록이 코드 배포보다 *먼저* 일어나야 한다 — 그렇지 않으면 새 코드가 발행한 이벤트를 옛 Consumer 가 못 읽는다.

## 3. Basic Full-Stop / Rolling Update

> 두 패턴은 EDA 가 아닌 일반 배포 패턴 — EDA 에서 추가로 고려할 점이 무엇인가가 중요하다.

SSOT §3~4 의 두 패턴은 다음과 같다.

| 패턴 | 동작 | EDA 특유 고려 |
|------|------|---------------|
| Full-Stop | 모든 인스턴스 중지 → 새 버전 시작 | 중지 동안 Consumer lag 누적 |
| Rolling Update | 한 인스턴스씩 교체 | Rebalance 가 매 교체마다 발생 |

Rolling Update 가 흔히 선호되지만, EDA 에서는 매번의 rebalance 가 처리 중단을 만든다. Static membership(Kafka 의 Consumer Group 옵션) 으로 rebalance 를 줄일 수 있다.

여기서 질문 하나 — Full-Stop 의 lag 누적은 어떻게 풀까? 두 방법이 있다. 배포 시간을 최소화(이미지 캐싱·빠른 health check), 또는 §6 의 Blue-Green 으로 대체.

## 4. Breaking Schema Change 패턴

> 깨는 스키마 변경을 배포할 때 — Producer 와 Consumer 의 배포 순서가 결정적이다.

SSOT §5 의 Breaking Change 배포 순서는 다음과 같다.

```
1. Schema Registry 에 새 스키마 등록 (호환성 NONE 으로 설정)
2. Consumer 를 새 버전으로 배포 — 옛 스키마와 새 스키마 모두 처리 가능
3. 모든 Consumer 가 새 버전인지 확인
4. Producer 를 새 버전으로 배포 — 새 스키마로만 발행
5. 일정 기간 모니터링 후 옛 스키마 처리 코드 제거
```

이 순서가 어긋나면 다음이 발생한다. Producer 가 먼저 새 스키마를 발행하고 Consumer 가 아직 옛 버전이면, 옛 Consumer 가 새 이벤트를 못 읽는다. 그래서 항상 Consumer 가 먼저, Producer 가 나중이다.

상세는 `./02-05.이벤트 진화 다루기.md §3` 의 버전 공존 전략과 같은 줄기다.

## 5. Blue-Green 과 선택 가이드

> Blue-Green 은 두 환경을 병행하다 한 번에 전환 — 가장 안전하지만 가장 비싸다.

SSOT §6 의 Blue-Green 흐름은 다음과 같다.

```
[Blue 환경 (현재 운영)]  ← 트래픽 100%
[Green 환경 (새 버전)]   ← 트래픽 0%
                ↓ Green 검증 완료
                ↓ 트래픽 전환 (DNS · Load Balancer)
[Blue 환경]  ← 트래픽 0% (롤백 대기)
[Green 환경] ← 트래픽 100%
```

EDA 에서 Blue-Green 의 까다로움은 Consumer Group 의 offset 처리다. 두 환경이 같은 Consumer Group 을 쓰면 두 환경이 같은 토픽을 두 번 처리한다. 별도 Consumer Group 을 쓰면 Green 이 처음부터 처리해 중복이 발생한다.

해결은 다음 셋이다.

1. Consumer 는 Blue-Green 하지 않고 Rolling Update — 항상 같은 Group ID.
2. Producer 만 Blue-Green — 발행 토픽은 영향받지 않음.
3. 멱등성 강제 — 중복 처리가 안전하면 Green 이 처음부터 재처리해도 됨.

### 5-1. 패턴 선택 가이드

| 상황 | 권장 패턴 |
|------|----------|
| Stateless Producer | Rolling Update |
| Stateful Consumer | Rolling Update + Static Membership |
| 깨는 스키마 변경 | Breaking Schema Change Pattern |
| 큰 리스크 변경 | Blue-Green (Producer 만) |
| 빠른 롤백 필요 | Blue-Green |

선택의 기준은 변경의 리스크와 롤백 속도 요구다. 일상 배포는 Rolling, 큰 변경·낮은 리스크 허용은 Blue-Green.
