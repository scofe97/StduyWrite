# 이벤트 진화 다루기
---
> 이벤트는 한 번 발행되면 영원히 산다. 발행 후 일주일 만에 형식을 바꾸면, 그 토픽을 구독하는 모든 Consumer 와 토픽에 보관된 옛 이벤트들이 모두 영향을 받는다. 진화는 *변경* 이 아니라 *공존* 의 문제다.

`01-03` 이 데이터 계약의 정의를 다뤘다면, 본 문서는 그 계약이 시간이 지나며 어떻게 변하는가의 운영 비용을 다룬다.

## 1. CQRS · Event Sourcing · Event Streaming 의 관계

> 셋 다 이벤트를 진실로 삼지만, 진실의 보존 기간과 재처리 자유도가 다르다.

SSOT §11.1 의 비교는 다음과 같다.

| 패턴 | 이벤트 보존 | 재처리 가능 시점 |
|------|-------------|------------------|
| CQRS | 짧음 (read view 업데이트 후 폐기 가능) | 보존된 이벤트 범위 내 |
| Event Sourcing | 영구 | 처음부터 끝까지 |
| Event Streaming | 정책에 따름 (보통 일~주 단위) | 보존 기간 내 |

진화의 부담은 보존 기간에 비례한다. Event Sourcing 은 영구라 진화 정책이 가장 엄격해야 한다. CQRS 는 짧으면 다소 자유롭다.

## 2. 스키마 진화와 의미 진화

> 같은 단어가 시간이 지나며 다른 뜻이 되는 것 — 의미 진화는 스키마 진화보다 더 까다롭다.

진화에는 두 종류가 있다.

| 종류 | 예 | 해결 도구 |
|------|-----|----------|
| 스키마 진화 | `email` 필드 추가, `name` 을 `firstName`/`lastName` 으로 분리 | Schema Registry · 호환성 정책 |
| 의미 진화 | `total` 이 "할인 전 금액" → "할인 후 금액" 으로 의미 변경 | 이벤트 이름·필드명 변경 (Breaking) |

스키마 진화는 도구로 해결된다. 의미 진화는 도구로 해결되지 않는다. 같은 이름·같은 타입이라도 의미가 바뀌면 옛 Consumer 가 잘못된 해석을 하기 때문이다. 의미가 바뀌면 새 필드명·새 이벤트로 분리해야 한다.

여기서 질문 하나 — 의미 진화를 어떻게 미연에 방지하나? 필드 이름을 *의도가 분명한* 어휘로 짓고, 의도가 흐려질 가능성이 보이면 처음부터 더 구체적인 이름을 쓴다. `total` 보다 `totalBeforeDiscount`, `totalAfterTax` 가 안전하다.

## 3. 버전 공존 전략

> 새 버전이 나왔다고 옛 버전이 즉시 사라지지 않는다 — 일정 기간 둘이 공존하는 정책이 필요하다.

SSOT 가 제시하는 공존 전략은 셋이다.

1. Single Topic, 호환 가능한 변경 — 같은 토픽에 여러 버전이 섞이고, 호환성 정책으로 처리.
2. Topic Per Version — `events.v1`, `events.v2` 로 토픽 분리, Producer 가 둘 다 발행 일정 기간, 모든 Consumer 가 v2 로 이동 후 v1 폐기.
3. Migrating Service — 한 서비스가 v1 을 받아 v2 로 변환해 재발행.

선택 기준은 변경의 깨짐 정도와 Consumer 수다. BACKWARD 호환 가능한 변경은 1 번이 자연스럽다. 깨는 변경 + Consumer 가 많으면 2 번이 안전하다. 일부 Consumer 만 자율적으로 마이그레이션해야 하면 3 번을 검토한다.

## 4. 재처리와 마이그레이션

> 새 read view 가 필요하면 토픽을 처음부터 재처리한다 — 이게 가능하려면 이벤트가 충분히 보존되어 있어야 한다.

SSOT 의 재처리 시나리오는 다음과 같다.

1. 새 비즈니스 질문이 생겨 새 read view 가 필요 — 기존 이벤트를 처음부터 재생.
2. 옛 view 의 버그 발견 — 새 코드로 재처리.
3. 새 서비스 추가 — 기존 이벤트를 처음부터 학습.

이 재처리가 가능하려면 다음 조건이 필요하다. 이벤트가 충분히 보존되어 있고(Event Sourcing 또는 긴 retention), 재처리가 결정적이며(`./02-03`), 재처리 중에도 라이브 처리가 영향받지 않아야 한다.

## 5. Event Sourcing 컨텍스트의 특수성

> Event Sourcing 에서는 모든 옛 이벤트가 살아 있다 — 과거 형식과의 호환성이 영원히 부담된다.

SSOT 가 강조하는 Event Sourcing 의 진화 부담은 다음과 같다.

1. 한 번 발행된 이벤트는 절대 수정할 수 없다 — 모든 진화는 새 버전 발행으로.
2. 과거 이벤트를 재생할 때 옛 형식을 해석할 수 있어야 한다 — Upcaster 패턴.
3. 코드가 옛 이벤트 핸들러를 영구히 유지해야 한다.

Upcaster 는 옛 이벤트를 새 형식으로 변환하는 함수다. 재생 시 자동으로 적용된다. 이 도구가 있으면 코드는 새 형식만 다루지만, Upcaster 의 누적이 시간이 지나며 부담이 된다.
