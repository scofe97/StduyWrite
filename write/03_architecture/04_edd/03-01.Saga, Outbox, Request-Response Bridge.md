# Saga, Outbox, Request-Response Bridge
---
> 분산 시스템에서는 하나의 문제를 하나의 패턴으로 풀지 않는다. Saga, Outbox, Request-Response Bridge는 각각 다른 종류의 경계 문제를 해결한다.
>
> - Saga
> - Transactional Outbox
> - Request-Response Bridge
> - 비동기 연계

## 1. 세 패턴은 목적이 다르다

Saga는 여러 로컬 트랜잭션을 조정하는 패턴이다. Outbox는 로컬 DB 변경과 이벤트 발행 사이의 간극을 메운다. Request-Response Bridge는 외부에는 동기처럼 보이지만 내부에는 비동기 처리를 연결한다.

이 셋은 모두 분산 환경에서 자주 등장하지만, 같은 문제를 푸는 것은 아니다. 먼저 “무엇이 깨질 수 있는가”를 분명히 해야 한다.

## 2. Outbox: 가장 먼저 고려할 패턴

전투 결과 저장과 이벤트 발행을 같이 처리해야 한다면 Outbox가 가장 실용적이다. DB 트랜잭션 안에서 도메인 변경과 outbox 레코드를 함께 저장하고, 별도 발행기가 메시지 브로커로 내보낸다.

카드게임 예시에서는 `BattleWon`이나 `RunEnded`를 외부 분석 시스템으로 안전하게 전달할 때 적합하다.

## 3. Saga: 여러 서비스의 상태 변경을 조정할 때

Saga는 한 서비스의 성공이 다른 서비스 작업을 연쇄적으로 유도할 때 쓴다. 각 단계는 독립 트랜잭션이고, 실패하면 보상 동작이나 재시도로 복구한다.

예시로는 다음 흐름이 있다:

- `RunEnded`
- 업적 서비스 반영
- 시즌 랭킹 갱신
- 보상 지급 시스템 반영

이 경우 전체를 하나의 ACID 트랜잭션으로 묶기보다 각 단계의 성공/실패를 관리하는 편이 현실적이다.

## 4. Request-Response Bridge: 동기 UX와 비동기 내부를 잇는다

사용자는 즉시 응답을 원하지만, 내부 처리는 시간이 걸릴 수 있다. 이때 Request-Response Bridge를 사용하면 외부에는 요청-응답 인터페이스를 제공하고, 내부에서는 메시지 기반 비동기 처리를 연결할 수 있다.

예를 들면 “런 종료 후 상세 통계 생성”이 몇 초 걸릴 수 있다. 이 경우 API는 `202 Accepted`와 조회 토큰을 돌려주고, 내부에서는 비동기 작업을 진행하는 방식이 가능하다.

## 5. 카드게임 도메인에 적용한 우선순위

이 도메인에서는 Outbox가 가장 먼저 필요해질 가능성이 높다. 그 다음은 외부 운영 시스템이나 메타 진행이 분리됐을 때의 Saga다. Request-Response Bridge는 시간이 오래 걸리는 파생 처리에 한정해 선택적으로 쓰면 된다.

즉 세 패턴을 모두 도입하는 것이 목표가 아니라, 어떤 경계 문제가 나타났을 때 어떤 도구를 쓰는지가 중요하다.

## 6. PoC와의 연결

제공된 `01-event-driven` PoC는 Outbox, Saga, Request-Response Bridge를 이벤트 기반 설계의 중요한 확장 패턴으로 다룬다. 이 문서에서는 그 개념을 Spring Boot 서비스 구조와 운영 관점으로 옮겨온다.

특히 핵심 게임플레이와 파생 처리를 분리하는 방향은 PoC의 메시지 기반 사고방식과 잘 맞는다.

## 7. 실무 결론

이 시리즈의 권장 순서는 다음과 같다:

- 먼저 로컬 트랜잭션과 모듈 경계를 잘 설계한다.
- 외부 이벤트 전달이 필요해지면 Outbox를 도입한다.
- 여러 독립 시스템 상태 조정이 생기면 Saga를 고려한다.
- 장시간 작업에 동기 UX가 필요하면 Request-Response Bridge를 적용한다.

패턴 이름보다 중요한 것은 문제의 종류를 올바르게 구분하는 것이다. 분산 아키텍처는 패턴 컬렉션이 아니라, 실패를 다루는 방식의 설계다.
