# 상태 기반 스트리밍
---
> 스트림 처리에 상태를 더하는 순간 시스템의 복잡도가 한 단계 올라간다 — 어디에 상태를 둘 것인가, 장애 시 어떻게 복구할 것인가의 결정이 그 복잡도의 분기점이다.

`02-03` 이 결정성의 기반을 다뤘다면, 본 문서는 그 위에서 상태를 어떻게 다룰 것인가를 정리한다.

## 1. 상태 저장소 기본 개념

> 스트림 처리의 상태는 "지금까지 본 이벤트들의 누적 요약" — 다음 이벤트의 처리가 이 요약에 의존한다.

SSOT §1 의 상태 예시는 다음과 같다.

1. 윈도우 집계 — "최근 1 분간 이벤트 수".
2. 조인 — "이 키의 가장 최근 사용자 정보".
3. 패턴 매칭 — "A 이벤트 후 5 분 내 B 이벤트 발생 여부".

이 상태들은 모두 "처리 도중 잃으면 다시 만들기 어렵다" 는 공통점을 갖는다. 그래서 상태 저장소의 위치·복구·확장이 핵심 결정이 된다.

## 2. Internal vs External State Store

> 처리 노드 안에 상태를 두느냐, 외부 시스템에 두느냐의 결정이 성능·운영성·확장성을 모두 정한다.

SSOT §2 의 비교는 다음과 같다.

| 측면 | Internal State Store | External State Store |
|------|----------------------|----------------------|
| 위치 | 처리 노드 로컬 (RocksDB 등) | 외부 DB (Redis · Cassandra) |
| 읽기 속도 | 매우 빠름 (디스크 또는 메모리) | 네트워크 비용 |
| 운영성 | 처리 노드와 한 묶음 | 별도 운영 |
| 복구 | Changelog 재생 | 외부 DB 백업 |
| 확장성 | 파티션과 같이 분할 | 외부 DB 의 확장에 의존 |

Kafka Streams 는 Internal State Store 가 기본이다 (RocksDB). 처리 노드와 상태가 한 묶음이라 운영이 단순하고 성능이 높다. 비용은 다음에서 발생한다. 노드가 죽으면 다른 노드가 그 파티션을 받아야 하고, 받은 노드는 상태를 다시 만들어야 한다.

## 3. Changelog Event Stream

> Internal State Store 의 복구 메커니즘 — 모든 상태 변경을 별도 토픽에 흘려보내, 노드 교체 시 그 토픽을 재생한다.

SSOT §3 의 Changelog 흐름은 다음과 같다.

```
[Processing Node]  →  state update  →  [State Store]
                                            ↓
                                    [Changelog Topic]  (Kafka)
```

새 노드가 그 파티션을 인수받으면 Changelog Topic 을 처음부터 재생해 State Store 를 복원한다. 이 메커니즘이 Internal State Store 의 운영 안전성을 만든다.

Changelog 는 log compaction 모드로 운영된다. 같은 키의 옛 메시지는 지워지고 최신 값만 남는다. 재생 시간이 무한히 늘어나지 않는다.

## 4. Global State Store

> 모든 처리 노드가 같은 상태를 봐야 하는 경우 — 작은 참조 데이터에 적합.

SSOT §4 의 Global State Store 는 모든 파티션에 같은 데이터를 복제한다. 카탈로그 같은 작은 참조 데이터(카드 정의·유물 효과)에 적합하다. 큰 데이터에 적용하면 메모리 비용이 노드 수에 비례해 폭증한다.

여기서 질문 하나 — 그러면 큰 참조 데이터는 어떻게 다루나? 두 선택지가 있다. 데이터를 파티션해 각 노드가 자기 파티션의 상태만 가지거나, External State Store 로 분리한다. 후자는 §2 의 비교 표에서 그 비용을 받아들이는 결정이다.

## 5. 스케일링·복구 — Hot Replicas

> 노드 교체 시 Changelog 재생에 걸리는 시간을 줄이려면 미리 복제본을 만들어 둔다.

SSOT §7 의 Hot Replicas 패턴은 다음 흐름이다.

1. 각 파티션의 active 노드 외에 다른 노드에 standby replica 를 둔다.
2. Standby 도 Changelog 를 지속적으로 따라간다.
3. Active 가 죽으면 Standby 가 즉시 인수받는다 — 재생 시간 없음.

비용은 메모리·저장소·CPU 의 중복 사용이다. 가용성이 중요한 도메인에서 이 비용을 받아들인다.
