# 지원 도구 — 스키마 레지스트리·할당량·오프셋 관리
---
> EDA 를 운영 가능하게 만드는 부가 도구들 — 이 도구들이 없으면 EDA 는 코드 차원에서는 작동하지만 운영 차원에서 무너진다.

## 1. 마이크로서비스–팀 할당 시스템

> 어느 팀이 어느 토픽·서비스의 책임자인가 — 이게 명시되지 않으면 사고 시 책임자를 찾는 데 시간이 든다.

SSOT §1 의 할당 시스템은 다음 메타데이터를 박제한다.

1. 서비스 ↔ 팀 매핑.
2. 토픽 ↔ 작성자 팀 매핑 (`./01-02 §2` 의 단일 작성자 원칙과 연결).
3. 토픽 ↔ 소비자 팀 매핑 (다수 가능).

이 정보가 GitOps · 사내 카탈로그에 박제되면 다음이 가능해진다.

1. 사고 시 책임 팀에 즉시 알림.
2. 스키마 변경 시 영향받는 팀들에 자동 통지.
3. 토픽 신설 권한 통제.

## 2. 이벤트 스트림 생성 및 수정

> 토픽 생성·삭제·파티션 변경이 어드미니스트레이터의 손작업이면 운영이 병목이 된다.

SSOT §2 가 권하는 자동화는 다음과 같다.

1. GitOps 로 토픽 정의 관리 — YAML 로 토픽·파티션·retention 정의.
2. CI/CD 가 변경을 적용 — PR 머지 시 자동 반영.
3. 변경 이력 추적 — Git log 가 곧 변경 이력.

여기서 질문 하나 — 모든 변경을 GitOps 로 통제하면 응급 변경이 막히지 않나? 응급 변경 경로(`break-glass`)를 별도로 두되, 그 사용을 사후 감사한다. 평상시는 GitOps, 응급 시만 직접 변경이다.

## 3. 이벤트 스트림 메타데이터 태깅

> 토픽에 도메인 의미·민감도·SLA 같은 메타데이터를 태깅하면, 거버넌스·접근 제어·모니터링이 자동화된다.

SSOT §3 의 태깅 예시는 다음과 같다.

| 태그 종류 | 예 | 용도 |
|----------|-----|------|
| 도메인 | `domain=run`, `domain=catalog` | 도메인별 모니터링·접근 제어 |
| 민감도 | `pii=true`, `pii=false` | PII 데이터 처리 정책 |
| SLA | `tier=critical`, `tier=best-effort` | 알림 우선순위 |
| 보존 정책 | `retention=7d`, `retention=permanent` | 자동 정리 |

태그가 박혀 있으면 "PII 가 포함된 토픽 목록" 같은 쿼리가 자동으로 답변 가능해진다. 거버넌스가 코드로 표현된다.

## 4. Quota — 보호망

> 한 팀의 폭주가 전체 클러스터를 마비시키지 않게 — 토픽·서비스별로 처리량 한도를 둔다.

SSOT §4 의 Quota 는 다음 차원에서 적용된다.

1. Producer Quota — 초당 발행 바이트 / 메시지 수 제한.
2. Consumer Quota — 초당 소비 바이트 제한.
3. Connection Quota — 클라이언트별 연결 수 제한.

Quota 가 없으면 한 팀의 버그(무한 루프 발행 등)가 클러스터 전체를 마비시킨다. Quota 초과 시 즉시 거부보다는 throttling 으로 처리해 정상 워크로드가 영향받지 않게 한다.

## 5. Schema Registry · 변경 알림 · Offset 관리

> 셋 다 "토픽의 상태를 가시화·통제" 하는 도구다.

SSOT §5~7 가 다루는 셋의 역할은 다음과 같다.

| 도구 | 역할 |
|------|------|
| Schema Registry | 스키마 등록·검증·호환성 강제 (`./01-03 §5`) |
| Schema 변경 알림 | 호환성 검증 통과해도 영향받는 Consumer 팀에 알림 |
| Offset 관리 | Consumer Group 의 위치 모니터링·재설정 |

Offset 관리는 다음 운영 작업의 도구다.

1. Consumer lag 모니터링 — Producer 가 발행한 위치와 Consumer 가 처리한 위치의 차이.
2. Offset 재설정 — 재처리 · 버그 후 복구 시 특정 시점으로 되돌리기.
3. Consumer Group 상태 진단 — 어느 파티션이 어느 Consumer 에 할당됐는지.

상세 구현은 `../../04_messaging/01_MessageContract/01-02.Schema Registry.md` 와 4 장(인프라) 의 Consumer Group 문서들이 다룬다.
