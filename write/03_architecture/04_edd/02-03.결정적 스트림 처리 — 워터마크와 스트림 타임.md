# 결정적 스트림 처리 — 워터마크와 스트림 타임
---
> "같은 입력에 같은 출력" 이라는 결정성은 스트림 처리에서 당연하지 않다. 분산 환경에서 이벤트의 시간 순서가 흐트러질 수 있고, 그 흐트러짐을 어떻게 정렬할 것인가가 결정성의 핵심이다.

## 1. 결정적 처리란

> 결정성은 재실행 가능성을 만든다 — 같은 토픽을 다시 처리하면 같은 결과가 나와야, 장애 복구·테스트·재처리가 가능하다.

SSOT §1 의 결정성은 다음을 의미한다.

1. 입력 순서가 같으면 출력도 같다.
2. 같은 시간 구간의 이벤트들은 항상 같은 윈도우로 묶인다.
3. 외부 상태(현재 시간·랜덤·외부 호출)에 의존하지 않는다.

결정성이 없으면 다음이 깨진다. 장애 후 재처리 시 결과가 달라지고, 두 환경(개발·운영)에서 같은 입력이 다른 출력을 만든다. EDA 의 운영 가능성이 사라진다.

## 2. Event Time vs Processing Time

> 같은 "시간" 이라는 단어로 두 가지 시간을 부르는 혼동이 결정성을 깨뜨린다.

이벤트에는 두 종류의 시간이 있다.

| 종류 | 의미 |
|------|------|
| Event Time | 이벤트가 *실제로 발생한* 시각. Producer 가 박는다 |
| Processing Time | Consumer 가 이벤트를 *처리한* 시각. Consumer 가 현재 시간으로 박는다 |

결정성을 원하면 Event Time 기준으로 처리해야 한다. Processing Time 기준이면 같은 입력을 두 번 처리할 때 결과가 다르다.

여기서 질문 하나 — Event Time 으로만 처리하면 무한히 늦게 도착한 이벤트도 기다려야 하나? 그래서 §3 의 워터마크가 필요하다.

## 3. 분산 타임스탬프 동기화

> 분산 환경의 시계는 항상 미세하게 어긋난다 — 그 어긋남을 인정하고 처리하는 도구가 필요하다.

SSOT §3 의 동기화 문제는 다음과 같다. 여러 Producer 가 각자의 시계로 Event Time 을 박으면, 시계 차이만큼 순서가 흐트러진다. 완벽한 시계 동기화(예: NTP)는 ms 수준의 정밀도만 보장한다.

해결은 다음 셋이다.

1. 시계 동기화 (NTP·PTP) — 어긋남을 줄인다.
2. Producer 별 시계 오프셋 메타데이터 — 어긋남을 명시한다.
3. 워터마크 — 어긋남을 받아들이고 일정 기간 기다린다.

## 4. 워터마크의 의미

> "이 시점 이전의 이벤트는 더 이상 안 온다고 가정한다" — 워터마크는 무한 대기를 끊는 도구다.

SSOT §5 의 워터마크는 다음 약속이다. "워터마크 W 가 도달했다는 것은, Event Time 이 W 이전인 모든 이벤트가 이미 도착했다고 가정한다."

이 약속이 가능하게 만드는 것은 두 가지다.

1. 윈도우 닫기 — 워터마크가 윈도우 끝을 지나면 그 윈도우를 닫고 결과를 발행할 수 있다.
2. 늦게 도착한 이벤트 처리 — 워터마크 이후 도착한 이벤트는 정해진 정책(드롭·별도 처리·재계산)에 따라 처리.

워터마크가 너무 빠르면 늦은 이벤트가 자주 누락된다. 너무 느리면 결과 발행이 지연된다. 도메인의 지연 분포에 맞춰 워터마크 정책을 정한다.

## 5. 스트림 타임과 차이점

> 스트림 타임은 워터마크보다 더 보수적인 개념 — "지금까지 처리한 이벤트의 최대 Event Time" 이다.

SSOT §6~7 의 구분은 다음과 같다.

| 개념 | 정의 |
|------|------|
| Watermark | "이 시점까지 모든 이벤트가 도착했다" 의 약속 (낙관적) |
| Stream Time | "지금까지 처리한 이벤트의 최대 Event Time" (보수적) |

Stream Time 은 약속을 하지 않는다. 단지 관찰된 사실이다. 그래서 늦게 도착한 이벤트가 있을 때 Stream Time 은 갱신되지 않는다.

Kafka Streams 는 Stream Time 을 사용한다. 처리 결정이 사실 기반이라 결정성이 강하다. Flink 는 Watermark 를 사용한다. 처리 결정이 약속 기반이라 지연이 작지만 늦은 이벤트 처리에 정책이 필요하다. 도메인에 따라 둘 중 하나가 더 맞는다.
