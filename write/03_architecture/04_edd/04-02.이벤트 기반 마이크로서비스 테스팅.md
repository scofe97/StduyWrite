# 이벤트 기반 마이크로서비스 테스팅
---
> 모놀리스의 테스트 전략은 EDA 에서 통하지 않는다 — 한 서비스의 정상 동작이 다른 서비스의 입력 가정에 의존하고, 그 가정은 토픽 위의 이벤트 형식이다. 테스트의 단위와 경계가 다시 정의되어야 한다.

## 1. 일반 테스팅 원칙

> 테스트 피라미드는 EDA 에서도 유효하다 — 단위·통합·E2E 의 비율과 책임이 명확해야 한다.

SSOT §1 의 EDA 테스트 피라미드는 다음과 같다.

| 레벨 | 비중 | 책임 | 비용 |
|------|------|------|------|
| 단위 (Topology Function) | 70% | 순수 변환 로직 검증 | 매우 낮음 |
| 스키마 진화 | 5% | 호환성 보장 | 낮음 (자동화 가능) |
| 통합 (Local) | 20% | 토픽 + 서비스 결합 | 중간 |
| 통합 (Remote) | 5% | 실제 인프라 위 E2E | 높음 |

각 레벨의 신뢰 범위가 다르다. 단위는 한 함수의 로직, 통합은 서비스 간 결합, E2E 는 운영 환경의 동작. 한 레벨이 다른 레벨을 대신할 수 없다.

## 2. 토폴로지 함수 단위 테스트

> 스트림 처리의 가장 작은 단위 — "이 이벤트가 들어오면 이 이벤트가 나간다" 의 매핑 함수.

SSOT §2 의 단위 테스트 예시는 다음과 같다.

```java
@Test
void mapsBattleCompletedToReward() {
    BattleCompleted input = new BattleCompleted(runId, battleId, victory());
    RewardCandidates output = rewardMapper.map(input);
    assertThat(output.candidates()).hasSize(3);
}
```

단위 테스트의 조건은 다음 셋이다.

1. 입력이 명시적 — 토픽·DB 같은 외부 의존성을 모의(mock)하지 않고, 순수 함수의 입력만.
2. 출력이 명시적 — 사이드 이펙트가 아닌 반환값.
3. 빠름 — 한 테스트가 ms 단위.

이 단위가 잘 분리되어 있으면 통합 테스트의 부담이 줄어든다. 통합은 *결합* 만 검증하고, *로직* 은 단위가 검증한다.

## 3. 스키마 진화 호환성 테스트

> 새 스키마가 옛 Consumer 를 깨지 않는지, 옛 스키마가 새 Consumer 에서 처리되는지 — CI 에서 자동 검증.

SSOT §3 의 호환성 테스트는 다음 시나리오를 자동 실행한다.

1. 새 스키마로 발행 + 옛 Consumer 로 소비 — BACKWARD 호환 검증.
2. 옛 스키마로 발행 + 새 Consumer 로 소비 — FORWARD 호환 검증.
3. 임의 버전 조합 — TRANSITIVE 호환 검증.

이 테스트가 CI 에 박혀 있으면 깨는 변경이 머지되기 전에 잡힌다. Schema Registry 의 호환성 검증과 함께 두 겹 안전망을 만든다.

상세 호환성 정책은 `./01-03 §5` 가, 구현은 `../../04_messaging/01_MessageContract/01-06.Avro 스키마 진화 패턴.md` 가 다룬다.

## 4. 통합 테스트

> 한 서비스의 진짜 동작 — 토픽에 발행하고, 자기 로직이 돌고, 다른 토픽에 결과를 내는 — 을 끝에서 끝까지 검증.

SSOT §4 의 통합 테스트는 다음 형태다.

```
[Test]  →  토픽 A 에 입력 이벤트 발행
[Test]  →  대상 서비스 시작
[Test]  →  토픽 B 에서 출력 이벤트 소비 (timeout 내)
[Test]  →  출력 검증
```

이 테스트의 가치는 단위 테스트가 놓치는 결합 문제를 잡는 것이다. 직렬화·역직렬화·토픽 설정·Consumer Group 구성 같은 통합 지점의 버그.

비용은 셋이다. 실행 시간(초~분), 인프라(Kafka 인스턴스), 환경 격리(병렬 실행 시 토픽 이름 충돌 방지).

## 5. 로컬 vs 원격 통합 테스트

> 로컬은 빠르고 격리되지만 현실과 다르다 — 원격은 현실에 가깝지만 느리고 격리가 어렵다.

SSOT §5~6 의 비교는 다음과 같다.

| 측면 | Local (Testcontainers) | Remote (실제 클러스터) |
|------|------------------------|-------------------------|
| 속도 | 빠름 (~초) | 느림 (~분) |
| 격리 | 완벽 | 토픽 이름 격리 필요 |
| 현실성 | 단일 노드 Kafka | 실제 클러스터 동작 |
| 비용 | 무료 | 인프라 비용 |
| CI 적합성 | 매우 좋음 | 별도 환경 필요 |

일반 PR 검증에는 Local 이 적합하다. 매일 한 번 또는 릴리스 전에 Remote 를 돌려 클러스터 특이 동작(rebalance · failover) 을 검증한다.

여기서 질문 하나 — Local 에서만 검증한 코드가 운영에서 실패하면 어떻게 진단하나? 운영 로그·메트릭의 트레이스 ID 를 따라가고, 실패 시나리오를 Remote 환경에서 재현한다. 그 재현이 가능하도록 Remote 환경의 카오스 엔지니어링(인위적 장애 주입) 도구를 함께 운영한다.
