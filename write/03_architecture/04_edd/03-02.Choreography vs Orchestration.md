# Choreography vs Orchestration
---
> 두 패턴은 같은 문제의 다른 답이다 — 워크플로우의 통제권을 분산시킬 것인가, 중앙화할 것인가. 어느 쪽이 옳다가 아니라, 어느 쪽이 이 도메인에 맞느냐의 문제다.

## 1. 두 패턴의 본질적 차이

> Choreography 는 "이벤트를 보고 알아서 한다", Orchestration 은 "지휘자가 시킨다".

| 측면 | Choreography | Orchestration |
|------|--------------|---------------|
| 통제권 | 분산 — 각 서비스가 자기 행동 결정 | 중앙 — Orchestrator 가 흐름 결정 |
| 결합도 | 이벤트 계약에만 의존 | Orchestrator 가 모든 참여자를 안다 |
| 변경 | 새 서비스가 기존 이벤트 구독하면 자동 통합 | Orchestrator 에 흐름 추가 필요 |
| 가시성 | 흐름이 코드 곳곳에 흩어짐 | 흐름이 한 곳에 모임 |
| 디버깅 | 분산 추적 필수 | Orchestrator 로그 중심 |

두 패턴은 트레이드오프의 양 끝이다. 결합도·자율성을 우선하면 Choreography, 가시성·중앙 통제를 우선하면 Orchestration.

## 2. Choreography 가 맞는 도메인

> 워크플로우가 단순하고, 각 참여자의 책임이 분명하며, 흐름의 변경보다 새 참여자의 추가가 잦은 도메인.

SSOT §2 의 Choreography 흐름 예시는 다음과 같다.

```
[Order Service]  →  OrderPlaced  →  [Inventory Service]
                                   →  [Payment Service]
                                   →  [Notification Service]
```

각 서비스는 `OrderPlaced` 이벤트만 알면 된다. Order Service 는 누가 듣고 있는지 모른다. 새 서비스(`Analytics`)가 추가되면 같은 이벤트를 구독하면 된다 — 기존 코드 수정 없음.

적합한 도메인의 신호는 다음 셋이다.

1. 워크플로우 단계가 3~4 개 이하로 단순.
2. 각 단계가 다음 단계를 알 필요 없이 자기 일만 할 수 있음.
3. 새 참여자가 자주 추가됨.

런 관리 예시의 `BattleCompleted` 이벤트가 이 카테고리다. `Run` 모듈이 결과를 받고, `Profile` 모듈이 경험치를 갱신하고, `Analytics` 모듈이 통계를 누적한다 — 셋이 서로를 모르고 같은 이벤트만 본다.

## 3. Orchestration 이 맞는 도메인

> 워크플로우가 복잡하고, 단계 간 결정이 있으며, 흐름 자체가 비즈니스 자산인 도메인.

SSOT §3 의 Orchestration 흐름 예시는 다음과 같다.

```
[Orchestrator]  →  step 1: 재고 확인  →  성공?
       ↓ 성공
       →  step 2: 결제  →  성공?
       ↓ 성공
       →  step 3: 배송 예약  →  실패?
       ↓ 실패 — 보상 트랜잭션 실행
```

Orchestrator 가 각 단계의 결과를 보고 다음 행동을 결정한다. 실패 시 보상 트랜잭션(앞 단계들을 되돌림)을 수행한다.

적합한 도메인의 신호는 다음 셋이다.

1. 단계가 5 개 이상으로 많고, 분기·조건이 있음.
2. 실패 시 명확한 보상 절차 필요.
3. 워크플로우 자체가 자주 변경되고 가시성이 필요.

결제·예약·승인 같은 비즈니스 프로세스가 이 카테고리다. 흐름이 비즈니스 자산이라 한 곳에서 관리·변경되어야 한다.

## 4. Direct-Call vs Event-Driven Orchestration

> Orchestration 도 두 종류 — 동기 호출 기반과 이벤트 기반.

SSOT §4 의 구분은 다음과 같다.

| 측면 | Direct-Call (동기) | Event-Driven (비동기) |
|------|--------------------|------------------------|
| 통신 | REST · gRPC 동기 호출 | 메시지 브로커 |
| 응답 시간 | 빠름 | 느림 (큐 대기) |
| 가용성 | 모든 참여자 살아 있어야 함 | 일부 다운 가능 |
| 추적 | 호출 체인 명확 | 메시지 ID 추적 |
| 보상 트랜잭션 | 호출 실패 즉시 인식 | 타임아웃 또는 명시적 응답 대기 |

Direct-Call 은 짧은 트랜잭션에 적합. Event-Driven 은 긴 트랜잭션·높은 가용성 요구에 적합.

여기서 질문 하나 — 그러면 어떤 기준으로 둘 사이를 정할까? 워크플로우 전체 완료 시간 SLA 와 참여자 가용성이다. 응답이 초 단위여야 하면 Direct-Call, 분 단위가 허용되고 가용성이 중요하면 Event-Driven.

## 5. 두 패턴의 혼합

> 실제 시스템은 둘을 혼합한다 — 상위 흐름은 Orchestration, 하위 단계는 Choreography.

SSOT 가 시사하는 실용적 결론은 다음과 같다. 큰 비즈니스 프로세스는 Orchestrator 가 단계를 조율하지만, 각 단계 내부의 부수 작업은 Choreography 로 풀어낸다.

예를 들어 "주문 처리" 의 큰 흐름은 Orchestrator 가 관리하되, "주문 완료" 라는 단계가 끝나면 그 사실을 이벤트로 발행하고 `Analytics`, `Recommendation`, `Email` 같은 부수 서비스가 자율적으로 받아 처리한다. 핵심 흐름의 가시성·통제와 부수 처리의 자율성을 동시에 얻는다.
