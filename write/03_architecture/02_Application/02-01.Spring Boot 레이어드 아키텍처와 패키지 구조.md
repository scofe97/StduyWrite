# Spring Boot 레이어드 아키텍처와 패키지 구조
---
> 레이어드 아키텍처는 가장 흔한 출발점이지만, 규칙을 명시하지 않으면 곧 “service에 모든 로직이 몰린 구조”로 붕괴한다.
>
> - presentation
> - application
> - domain
> - infrastructure

## 1. 레이어드 아키텍처의 기본 역할

레이어드 아키텍처는 의존성 방향을 단순하게 유지하는 데 강점이 있다. 요청은 위에서 아래로 내려가고, 세부 구현은 아래에 모인다. 팀이 Spring Boot를 처음 시작할 때 가장 이해하기 쉬운 구조이기도 하다.

다만 레이어를 나누는 것만으로는 부족하다. 각 레이어가 무엇을 가져도 되고 무엇을 가져오면 안 되는지를 함께 정해야 한다.

## 2. 런 관리 서비스에 적용한 레이어

이 시리즈에서는 다음 네 레이어를 기본형으로 둔다:

| 레이어 | 책임 | 예시 타입 | 금지 사항 |
|--------|------|----------|----------|
| presentation | HTTP 입출력과 검증 | Controller, Request DTO | 도메인 규칙 계산 |
| application | 유스케이스 조합 | UseCase, Facade | SQL 세부 구현 의존 |
| domain | 핵심 규칙과 상태 전이 | Run, Battle, Policy | Spring 기술 의존 |
| infrastructure | DB, 메시징, 외부 API | JPA adapter, Kafka adapter | 유스케이스 결정 |

이 구조에서 카드 사용 규칙은 domain에 둔다. 요청 파싱과 응답 직렬화는 presentation이 담당한다. 저장과 이벤트 발행은 infrastructure가 처리한다.

## 3. 패키지 구조는 도메인 우선이 더 낫다

레이어를 그대로 폴더로 만들면 처음에는 보기 쉬워도 기능이 커질수록 응집도가 떨어진다. `controller`, `service`, `repository` 폴더가 비대해지면 런 관련 코드를 한 번에 파악하기 어렵다.

Spring Boot에서는 “도메인별 패키지 안에 레이어를 넣는 방식”이 더 오래 버틴다:

```text
com.example.spire.run
├── presentation
├── application
├── domain
└── infrastructure

com.example.spire.battle
├── presentation
├── application
├── domain
└── infrastructure
```

## 4. application 계층은 유스케이스를 조합한다

application 계층은 도메인 규칙을 새로 만들지 않는다. 이미 domain에 있는 규칙을 묶어서 “플레이어가 카드 한 장을 사용한다” 같은 유스케이스를 실행한다.

예시는 다음처럼 잡을 수 있다:

```java
public final class PlayCardUseCase {
    private final LoadRunPort loadRunPort;
    private final SaveRunPort saveRunPort;

    public PlayCardResult handle(PlayCardCommand command) {
        Run run = loadRunPort.load(command.runId());
        run.playCard(command.cardId(), command.target());
        saveRunPort.save(run);
        return PlayCardResult.from(run);
    }
}
```

## 5. 자주 무너지는 지점

레이어드 아키텍처는 단순한 대신 경계가 흐려지기 쉽다. 특히 다음 세 가지는 자주 생기는 문제다:

- `@Service`가 사실상 domain 역할까지 떠안는 것
- JPA Entity를 곧바로 API 응답으로 노출하는 것
- infrastructure 구현을 application이 직접 new 하거나 import 하는 것

이 세 문제는 처음엔 개발 속도를 올려 보이지만, 테스트와 리팩터링 비용을 크게 키운다.

## 6. 언제 충분하고 언제 부족한가

규칙이 단순하고 CRUD 비중이 높다면 레이어드 아키텍처만으로 충분하다. 그러나 전투 규칙이 복잡하고 카드 효과가 늘어나는 도메인에서는 domain을 더 명확히 드러내는 구조가 필요해진다.

그 지점에서 다음 단계로 나오는 것이 클린 아키텍처와 헥사고날 아키텍처다. 두 구조는 레이어드 아키텍처의 취지를 더 엄격한 의존성 규칙으로 발전시킨 형태로 볼 수 있다.
