# 모듈러 모놀리스와 Spring Modulith
---
> 서비스를 쪼개기 전에 먼저 모듈을 쪼개는 편이 대체로 낫다. 모듈러 모놀리스는 “한 덩어리 배포”와 “명확한 내부 경계”를 동시에 추구한다.
>
> - 단일 배포 단위
> - 명확한 모듈 경계
> - 점진적 분리 가능성
> - Spring Modulith

## 1. 왜 모듈러 모놀리스가 출발점인가

분산 시스템의 복잡도는 생각보다 빠르게 커진다. 네트워크 장애, 분산 추적, 배포 독립성, 스키마 호환성까지 모두 감당하려면 도메인 경계가 먼저 명확해야 한다.

그래서 카드게임 런 관리 서비스의 초기 구조는 모듈러 모놀리스가 적합하다. 런, 전투, 카탈로그, 진행도, 분석 적재를 한 애플리케이션에 두되, 코드 경계를 엄격하게 관리한다.

## 2. 모듈 후보

예시 프로젝트에서는 다음 모듈 구성이 현실적이다:

| 모듈 | 책임 | 공개 인터페이스 | 내부 구현 |
|------|------|----------------|----------|
| `run` | 런 시작, 종료, 진행 | `StartRunUseCase`, `AdvanceNodeUseCase` | 상태 저장, 이벤트 발행 |
| `battle` | 턴과 카드 규칙 | `PlayCardUseCase`, `EndTurnUseCase` | 전투 엔진 |
| `catalog` | 카드/유물 정의 조회 | `CardCatalogQuery` | 정의 데이터 저장 |
| `progression` | 메타 해금 | `UnlockQuery` | 계정별 상태 |
| `analytics` | 로그 적재와 조회 | 이벤트 구독 | 리포팅 모델 |

이 구조는 이후 서비스를 분리하더라도 경계를 재사용할 수 있다.

## 3. Spring Modulith가 주는 이점

Spring Modulith는 Spring Boot 애플리케이션 안에서 모듈 경계를 명시적으로 다루게 도와준다. 패키지 구조와 이벤트, 문서화, 검증을 통해 “모놀리스지만 아무나 아무 데나 import하지 못하게” 만드는 데 강점이 있다.

특히 다음 세 가지가 유용하다:

- 모듈별 공개 API 패키지를 정할 수 있다.
- 모듈 간 이벤트 흐름을 명확히 표현할 수 있다.
- 테스트에서 모듈 경계 위반을 검출할 수 있다.

## 4. 패키지 구조 예시

다음처럼 각 모듈이 자기 하위 구조를 가지는 형태가 읽기 좋다:

```text
com.example.spire
├── run
│   ├── api
│   ├── application
│   ├── domain
│   └── infrastructure
├── battle
├── catalog
├── progression
└── analytics
```

여기서 다른 모듈이 참조할 수 있는 것은 기본적으로 `api`만 허용하는 식으로 운영하는 편이 좋다.

## 5. 내부 이벤트를 어떻게 쓸 것인가

모듈러 모놀리스에서도 이벤트는 유용하다. 예를 들어 `battle` 모듈이 `BattleWon` 이벤트를 발행하고, `run` 모듈이 이를 받아 보상 단계를 여는 구조는 결합도를 낮춘다.

다만 모든 호출을 이벤트로 바꾸면 흐름 파악이 어려워진다. 즉시 일관성이 필요한 핵심 진행은 직접 호출을 유지하고, 후처리나 파생 데이터 생성에는 내부 이벤트를 쓰는 편이 균형이 좋다.

## 6. 언제 서비스 분리를 고려할까

다음과 같은 신호가 보이면 분리를 고민할 시점이다:

- 팀 경계가 모듈 경계와 일치한다.
- 배포 주기가 모듈마다 크게 다르다.
- 부하 특성이 완전히 다르다.
- 저장 전략이 본질적으로 다르다.

카드게임 예시에서는 `analytics`나 리플레이 적재는 먼저 분리 후보가 될 수 있다. 반면 `run`과 `battle`은 강하게 결합돼 있으므로 초기에 분리할 이유가 약하다.

## 7. 실무 결론

Spring Boot 4.x 기준 설계에서는 모듈러 모놀리스가 가장 실용적인 출발점이다. Spring Modulith는 이 선택을 문서 수준이 아니라 코드 검증 수준으로 끌어올리는 도구다.

핵심은 “서비스로 나누지 않았으니 한 모듈이다”가 아니라, 한 프로세스 안에서도 경계를 엄격히 유지하는 것이다. 그래야 나중에 분리 여부를 설계적으로 판단할 수 있다.
